codegetter say:

Supplementary to my Request for assistance.

After reflecting on the unhelpful error message, I focused on the word “Data” and made the assumption, as this was the final stage of the upgrade process, there must be some connection with data.

Steps I took to resolve the problem.

  1. Executed sfc /scannow that exited with no reported problems.

  2. Checked the ‘System’ partition C:\ for files that could be flagged as potential problem candidates. Other than $Windows.~BT and $Windows.~WS that are used by the Upgrade process, nothing suspicious including zero length files.

  3. Also checked Program Files and Program Files (x86) for legacy specific applications that could have compatibility problems with W10. Note. Setup did flag known applications that had to be removed before continuing.
    Also executed the command drivers.exe to provide a list of installed drivers. Again this scan did not highlight the use of a legacy driver(s) that could be troublesome. Note: Saving the output using >> to a text file would be useful if a driver(s) are uninstalled and the driver(s) is required after W10 is upgraded.

  4. Executed a Register Scanner that validates string value entries to ensure disk locations exist. Found several ‘orphan’ entries, specifically UBCD4WIN that had to be removed during the failed Upgrade attempt.

  5. Downloaded the W10 Setup ISO and burnt it to DVD.

  6. Configured a shortcut to Setup.exe using the Advanced properties option to Run as Administrator and with a parameter of /copylogs. e.g L:\setup.exe /copylogs D:\W10UpgradeLogs

  7. Did not implement the advice I had read about disconnecting from the Internet and disabling USB devices. Couldn’t figure out why this could be problematic as Setup will interrogate attached devices to install drivers.

  8. Executed Setup and it completed successfully. The generated logs were now redundant but hopefully would have proved useful if the Upgrade had failed again.

Upgrade Points of Interest

  1. When W10 was ready for use, the Desktop and Start menu were almost empty. I assumed correctly that the application Start Menu Reviver used with W8.1 was the issue. After an uninstall and reboot the state of the Desktop and Start menu was restored.
    Important remove Start Menu Reviver if installed, before starting the Setup Upgrade process.

  2. Subsequently after installing KB3081424, the WiFi connection was flagged as limited. Checked all adapter settings with no issues found. Ran the Troubleshoot problem option. After several resolution attempts, an Intel adapter driver was reinstalled. The WiFi connection is once again operational.

  3. The browser Edge after previous successful use also failed after the installation of KB3081424. Edge was unable to make a network connection even though Windows Update, Chrome and Firefox were functioning. After going through the Network & Internet settings realised that the previous inherited W8.1 HomeGroup configuration no longer existed. Re-selected the Libraries and Shares and Edge is functioning again.

  4. Reran sfc /scannow that exited with 3 repaired language DLLs.

  5. Reran the Register Scanner that found 1,370 ‘orphan’ entries. Must have been redundant files no longer used by W10 and removed during the upgrade process.

  6. There is a compressed folder called C:\Windows\Panther that has the same content as the folder Setup would have used when configured with the /copylogs parameter. Panther will not exist if Setup fails as C:\Windows rolls back to the previous Windows installation.

  7. Lastly plan to use “disk cleanup” to reclaim gigabytes of used disk. The candidates are $Windows.~BT, $Windows.~WS and Windows.old

Windows 10 has been running without issue since rectifying the above issues. My upgrade experience could be specific to my rig configuration, but I am still hopeful that this feedback could be useful in providing those who are still experiencing issues, a ‘toolkit’ of ideas that may also help achieve a successful upgrade outcome.

One final comment.

The Setup process within it’s Upgrade process must have hit a severe issue or issues that forced Setup to terminate. Other that displaying the “0x80070004-0x4000D”error code and [snip] “ERROR_MIGRATE_DATA operation” message, throw an informative Exception message that notifies the user what the severe issue was that caused Setup to terminate.

If there are too many message permutations, then Setup should by default always create an effective execution audit log. If the Setup terminates then inform the user of this log using an error message. Otherwise with a successful Setup outcome, the log would be redundant and should be removed.

Armed with this information the user would then hopefully resolve the problem(s) without having to jump through so many frustrating time consuming loops searching for a solution.
