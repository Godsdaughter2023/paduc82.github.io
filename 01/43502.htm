Yes, Terry, that's it! You've identified the trigger that results in the anomalous default action.

My customary window arrangement during on-line bill payment has the source window in the foreground at bottom left and the target window at upper right. Behind both of them, a small corner of the Edge browser window (in which I'm paying my bills) is visible between them. And indeed, if the path I happen to take along the way from the source window to the target window touches that exposed corner of Edge, the default action changes to Move and doesn't change back! This is in contrast to what happens if the drag path passes over the background desktop; the action changes to Move while it is over the desktop and then changes back to Copy when it touches the target window.

Once the drag operation touches the Edge window, the file selection becomes tainted with a Move default so long as the same selection remains in effect. Even if I do not drop the files into the target window but return them to the source window and let go, then drag them a second time along a path that doesn't touch Edge, the default remains set to Move. Only by selecting some other file, and then reselecting the desired files, can I shake off the tainted behavior and get them to default to Copy along a clean drag path.

All of the random inconsistency that seemed so inexplicable is now clearly explained by the semi-randomness of my drag path. Sometimes it passed over the visible corner of the Edge window; sometimes it did not, and I now realize that this was entirely responsible for what was happening.

This evening, after confirming your findings, I rearranged my windows such that the Edge window does not show anywhere along the potential drag path, and then I proceeded to make my monthly bill payments without ever experiencing the anomalous default action.

So, now that we know what triggers this anomaly, maybe Microsoft will be in a better position to identify the faulty code and fix it! Dare I hope for that?
