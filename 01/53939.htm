Understanding downgrade rights




Consumer versions of Windows 8 Pro that come preinstalled on PCs have downgrade rights to a previous version of Windows (example: Windows 7, Windows Vista).


If your copy came pre-installed on your system, you may downgrade your copy to Windows 7.


If not you need purchase a copy of Windows 7 and install the same on your computer.


• Downgrade rights are only available for licenses obtained through OEM or Volume Licensing (VL) and not apply to Full version (FPP).


• OEM downgrade rights apply to Windows 8 Pro and allows for downgrades up to two prior version


• Software Assurance through VL (Volume licensing) provides the greatest flexibility, allowing for downgrades to all prior versions and editions besides Windows 7 Ultimate and Vista Ultimate editions.Without Software Assurance coverage, VL has downgrade rights to prior versions of Windows with the exception of Enterprise.


• It is not possible for a Windows 8 copy with multi-language function to downgrade to Windows 7 Pro or Windows XP Pro as multi language function is exclusive to Windows 7 Enterprise edition and not available for Professional edition.

Lets look at those idea bullets...

A system image would be good if your OS is on a small primary partition (so creating/restoring the image doesn't take hours and if you have a good place to keep your image and if your system still boots you can restore your image from Control Panel (which is what MS recommends) or if your system has preinstalled recovery options (Repair your Computer is not always available) or if you have a bootable installation CD/Repair CD - not everyone has that either.

So a System Image would be good under all the perfect environments that we know all visitors here maintain.

If you use the MS method to backup the registry and after your modification the system no longer boots there are no instructions that tell you how to restore the registry backup you made.  S now what?  Or even if the system does boot there are still no instructions on how to restore the backup you made.

The MS method would work if you can still boot and somebody tells you how to import your backup on a running and live system.

The SevenForums method suggests making a Restore Point and restoring the registry from that so if your system doesn't boot but you can still get to restore your system that would work if you can still get to the Advanced Boot Options or from a CD to restore the system.

The SF method is a pretty good idea under the right conditions.

Now I know Frederick Long is pretty sharp so what follows is not an attack on his demonstrated brilliance but let's take a look at how things work and what could (and does) happen with this method.

First understand how the c:\windows\system32\config\RegBack folder works...

That RegBack folder is where you can find a copy of the 5 registry files that Windows 7 automatically creates:

Some folks that even know that folder exists think that backup copy of the registry files gets refreshed "sometimes", "once in a while", "every so often", whenever certain (unspecified) "important" things happen.

Actually it gets refreshed a Scheduled Task:

So you can see that that task runs every 10 days (unless you change it) so the RegBack folder gets refreshed every 10 days...  not "sometimes", not "once in a while", not "every so often", not whenever certain (unspecified) "important" things happen.  It gets refreshed every 10 days.  Take a look at your task history and RegBack folder.

Let's say the last time the task ran was on a Monday.

The next day is Patch Tuesday and you receive 20 critical MS updates full of new critical system files (and any of the registry changes that go with them.

On Wednesday you install 2 new programs.

On Thursday you uninstall 2 old programs.

On Friday you decide to make your registry changes and then something goes wrong and you use this RegBack method.

Your registry would be restored to the previous Monday (the last time the RegIdleBackup task ran).

But that registry from Monday doesn't match up with the 20 MS updates and critical system files you got from the Patch Tuesday updates.  Now you have 20 critical updates installed but all the registry changes that went with them is gone.  Is that going to be okay?

That registry from Monday would know nothing about the 2 new programs you installed on Wednesday but those programs are still installed.  Is that going to be okay?

That registry from Monday would think the two programs you uninstalled on Thursday are still installed on your system but all the files that go with them are gone.  Is that going to be okay?

Hmmmm.   Sounds like that could be quite a mess.

But, it might work sometimes under all the right conditions.

Bug Check 0x3B: SYSTEM_SERVICE_EXCEPTION

The SYSTEM_SERVICE_EXCEPTION bug check has a value of 0x0000003B. This indicates that an exception happened while executing a routine that transitions from non-privileged code to privileged code.

Important Info If You Have Received a STOP Code

If you received a blue screen error, or stop code, the computer has shut down abruptly to protect itself from data loss. A hardware device, its driver, or related software might have caused this error. If your copy of Windows came with your computer, call the manufacturer of your computer. If you purchased Windows separately from your computer, Microsoft provides support. To find contact info for Microsoft or your computer manufacturer, Contact Support.

If you have experience with computers and want to try to recover from this error, follow the steps provided in the Microsoft article Resolving STOP (Blue Screen) Errors in Windows.

These actions might prevent an error like this from happening again:

  1. Download and install updates and device drivers for your computer from Windows Update.
  2. Scan your computer for computer viruses.
  3. Check your hard disk for errors.
