alfa s say:

Hi,

I would recommend you to check the speeds of both the connections and connect whichever is faster.

1) Have you tried changing the adapters and bindings order in the advanced panel of network connections?

In the start menu, search for 'network connections' and press enter.
In the window that comes up, use Alt-key for menu. Choose Advanced. (alt-n-s) On the Adapters and Bindings tab, re-order the connections, putting your wired connection at the top.

2) When more than one network connection is available, Windows uses the one with the lowest metric value. By default, it automatically assigns a metric value based on the network connection's rated speed. See the link below.
http://support.microsoft.com/?id=299540

To force Windows to use a specific network connection, assign a metric value to each one, giving the lowest value to the desired connection:

1. Open the Network Connections folder (Start > Run > ncpa.cpl)
2. Right click the desired connection.
3. Click Properties > Internet Protocol Version 4.
4. Click Properties > Advanced.
5. Un-check "Automatic metric".
6. Enter a number between 1 and 9999 for the "Interface metric".

Here is an article that would help you with the internet speed.
http://www.microsoft.com/atwork/remotely/connections.aspx#fbid=07DpSsGKhfL

Also, you may delete temporary files that will help you a bit. (applicable to windows 7)
http://windows.microsoft.com/en-US/windows-vista/Delete-temporary-Internet-files

Hope you get your issue fixed soon.

Here is my situation/bug:

I have a multi-honed computer. 

The Ethernet adapter is connected with 192.168.1.113/24 and GW 192.168.1.1.  This connection works just fine and as expected.

All relevant/discussed connections are on-link:  there are no switches currently in this setup.

I've tried various things to resolve this particular issue in varied ways: reset the adapters / reboot the machine, static addressing, removing the default routes (as some will surely suggest, but is a non-issue), adding superfluous routes (route add 10.0.2.1/32 10.0.2.31 --which worked temporarily), changing metrics, exploring registry entries related to TCP/IP, exploring local group policy, and even updating windows to the latest build.  If any worked, it was temporary, and Windows uncorrected the problem within 24 hours, with the "fix" still in place.  There is no error message that I can find - the event logs yield no data.

For those that might complain about the multiple default gateways, I have tested this.  Besides, under the right (intentional) circumstances, I would not consider this a bad practice.  Given that the metrics are different, there is a priority in the default routes, and one will truly be the default - the other is a backup for link failover.  Besides, this situation happens (forcefully) on Windows with multiple DHCP interfaces, but does not happen on other OSes. This is not part of my problem.

Here are my registry findings:

HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\PersistentRoutes is not being populated with static routes as shown in route table.  Manual entries did not help, nor modify the route table in any way.  I could not otherwise find the registry entries for the routes - I don't know where the persistent routes are stored.  After a update of windows to the lastest build, the kernel was reading from this registry location - However, it was not writing routes there.  There is an undocumented change (at least I can't find it) to the Windows 10 kernel for the route table handling.

HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces\<AdaptorID>\IPEnableRouter=1 has no effect.  I found this as an entry in Windows XP, but not in Windows 10.  In Windows 10, it is at HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\IPEnableRouter

HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\EnableDeadGWDetect=0
Created this value in Windows 10 - no effect.

Furthermore, in my methodology to test at the lowest possible layer, I created a VM with a different operating system.  This VM is running on the host in question.  The adapters are connected in bridge mode (layer 2 - Ethernet) to the same adapters in question on the host.  Layer 2 (Ethernet) works: both adapters are able to send broadcast requests and get IP addresses on both the host and the VM.  As can be seen Layer 3 (IP) works as expected on the VM, but not the host.

Notice also the simplicity that UNIX has in its routing table compared to Windows.

As can be seen from these tests:

1. Layer 1 connections are firmly established

2. Layer 2 (Ethernet) connections work as expected.  This includes (as some might suspect from the IP scheme) VLANs

3. Layer 3 (IP) is broken on the host (Windows 10 pro), but when bypassed on the VM works as expected

4. Layers 4-7 protocols, as well as DNS, are not in play here and are not affecting the outcome.  DHCP is also not an issue, as I've tested static address.

5. Internet connectivity, while a corollary, is not part of this problem - my problem is getting the local gateway.

Given these tests, I believe this to be a bug in Windows - IP routing is not being handled as advertised in the route table.  While I hold out for a misconfiguration, I believe I have been thorough enough to rule out such a mistake.  I hold an MCSE with certification in ... (wait for it ) TCP/IP.
