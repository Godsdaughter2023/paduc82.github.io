davidnutting say:

OK - I manage a number of Surface Pros - mainly Surface Pro 3s and a few Surface Pro 2s.

While it's great that Microsoft have now released a tool to update the TPM Firmware in the Surface Pro 3s, the Surface Pro 2s are still out in the cold and no sign that Microsoft intend to do anything about it.

The (possibly) good news however is that Infineon do seem to have released updated firmware via some other manufacturers.

At this point I have to add a HUGE caveat!  The process I went through to update the TPM Firmware in a Surface Pro 2 (and I've only done 1 so far) is completely undocumented and I freely admit that I was taking a gamble - this is far from risk free!  But, as far as I can see, it has worked fine on the one Surface Pro 2 I've tried so far.  If you try anything similar, it is entirely at your own risk!  Read all the notes / linked web pages and come to your own view whether the risk is worth it.  No-one is going to make any promises or give any guarantees that this will work - and, at worst, it might brick the Surface Pro 2 completely!

Note the following regarding the hardware / software of the Surface Pro 2 I was working on.  Yours might not be the same.  And note that, in any event, this really does only apply at all to Surface Pro 2s and not any other model.

The Surface Pro 2, 128GB version, had Windows 10 Education x64 1709 installed and fully patched up to date.  Battery fully charged.  On mains power, with Type Cover keyboard.  Connected to the internet via WiFi.  No other hardware connected.

tpm.msc reported the TPM as IFX / version 4.32 / specification 1.2 - and as ready for use, but with the security vulnerability caveat.

Device Manager reported the TPM as 1.2 under Security Devices.

So this is what I did.

N.B. I did a full restart of the Surface Pro 2 before I started and after EVERY step to try to reduce the risks - I'm not going to keep repeating that, so just take it as read.

1. Turn off Bitlocker if it is enabled and wait for decryption to complete fully - NOT just suspended - we're dealing with the TPM here, so every chance (actually certainty as far as I am aware) of losing the Bitlocker keys (but I did also have the Bitlocker recovery key saved in a file elsewhere just in case).

2. Windows 10 1709 won't, by default, let you save the TPM Owner Password at all (IIRC this is standard behaviour from 1607 onwards), so you need to set a registry key to override that as you'll need the TPM Owner Password in order to update the firmware using this process.  So run regedit to edit the registry to set the OSManagedAuthLevel value to 4 in the HKLM\SOFTWARE\Policies\Microsoft\TPM key - see links below and elsewhere on the web for details of this and the rationale for throwing away the TPM Owner Password whenever the TPM is cleared.

3. Run tpm.msc and use Clear TPM to clear the TPM - you have to reboot and press F12 at the boot screen.  Then run tpm.msc again and wait / check that the TPM has been reinitialised fully and is ready for use.  This should have stored a copy of the TPM Owner Password in the registry.

4. Download the SLB9656 / sp82407.exe file from the HP support site - firmware update to 4.34 - link below.

5. Run the EXE to extract the files to somewhere on the C-drive.  Locate the files and read the README thoroughly.  Only when you have digested it and understood it, decide whether to proceed further.  If you do decide to go ahead, I then simply ran the extracted EXE file and followed the instructions.  It's initial tests for compatibility all passed.  And when I then proceeded it obtained the TPM Owner Password from the registry - if it can't get the password from there it will prompt you to either enter it manually or provide a file containing it - first time through I had to abort here as I didn't have either of these and restart the process after completing steps 2 and 3 above.  The computer then asks for a restart.  My Surface Pro 2 restarted but hung at the Surface boot logo - this didn't feel like good news, but I left it for a full half hour - it was then still at the Surface logo (no change - no spinning dots underneath it) - so I took the plunge with fingers firmly crossed and held down the power button to force a shutdown - waited a minute and again pressed the power button to boot it up - which it then did perfectly (BIG sigh of relief)!

6. Run tpm.msc - check that the TPM initialises OK and reports as ready for use and without the security warning.  The version should have updated to 4.34.  The TPM should be unchanged in Device Manager.  Note that this will have changed the TPM Owner Password, and the new one is now stored in the registry!

7. Change the OSManagedAuthLevel value in the registry back to 2.

8. Run tpm.msc and Clear TPM again - to reset the TPM owner password and this time NOT save it in the registry - the advice elsewhere about not normally needing it is (IMHO) good - you really shouldn't ever need it except in rare circumstances such as going through this process.

9. Run tpm.msc and again check that the TPM initialises OK and reports as ready for use and without the security warning.

10. Re-enable BitLocker as appropriate.

With luck, you are now done!

jj_951 say:

Actually, the original post is simply describing a registry setting analysis using XML pattern matching.  The original poster must be running some type of program that is analyzing his registry to confirm if something is correct/valid/compliant/etc.

This process compares the string value of a registry setting to a pattern (defined in XML) in order to determine if it's compliant.  If you look at the pattern, it's immediately clear that it's trying to determine if the value of that registry entry is similar to this:

%systemroot%|%windir%|C:\Windows\System32\LogFiles\Firewall\*

The part that requires knowledge of XML pattern matching is:  [^\.][^\\]*

The ^ character is called a 'negative character group', which means that whatever follows CANNOT be present.  In this specific example, the file path cannot include either a backslash or a double backslash after "...Firewall\".  This is a way of saying that if there are any more subdirectories listed after the Firewall directory, then it won't match the pattern.

The final answer to the original poster's question could be to set the string value in the registry to

C:\Windows\System32\LogFiles\Firewall\logfilename.txt

An example of a value with a log file location that would fail the comparison is:

C:\Windows\System32\LogFiles\Firewall\2017\logfilename.txt

This would fail because subdirectories are not allowed per the defined pattern.

FYI, per the pattern, any of the letters could be upper or lower case, and the drive letter C: could be anything between C and Z.
