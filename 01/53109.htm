gwyzagain say:

Window update error "80070005" Various Causes & Solutions

There are several ''fixes'' posted on various threads.  For some, these fixes work well.  For others, not at all.

I myself have had this same problem on several Windows 7 computers.  Most have been resolved by using the Microsoft utility titled:

WindowsUpdateDiagnostic.diagcab

However, one of my computers could not be repaired no matter how many ''fixes'' I have attempted.  I had finally reached the point where I was about to do a complete re-installation of Windows 7.  However, I was hesitant to do so because that has _also_ failed to help me in the past.  It seems as if every time I install Windows Update Sp1 for Windows 7, the result is the same error (80070005) after the update is applied... At which point the ''WindowsUpdateDiagnostic.diacab'' fails to make the repair.

For that reason, I repeated a ''fix'' that I had done in the past, only this time I tried something different, which finally worked.  I did as follows:

1) System Restore: Use the ''Restore Point'' date when Windows Sp1 update was installed.

2) After restart, and after system is up and running again, _THEN_ use the ''WindowsUpdateDiagnostic.diagcab''

3) The fix was _then_ able to repair my ''80070005'' error.

4) Restart computer (just to be sure).

5) Run Windows Update to install the Sp1 update.

6) After reboot, and after system is up and running again, then _once again_ run the ''WindowsUpdateDiagnostic.diagcab''

7) If the diagnostic utility says all is well, you're done.  If not, it is at this point where other posted ''fix'' remedies can be tried with (perhaps) better success.

IMPORTANT NOTE: With regard to other ''fixes'' involving the installation of ''subinacl.exe'', please be aware that the file can wind up being installed in locations that will not allow the ''reset.bat'' file to work.  The most common installation location is: Program Files\Windows Resource Kits\Tools\subinacl.exe.  The _easiest_ remedy for this ''incorrect location'' problem is to perform a search for the ''subinacl.exe'' file (without the quotes), and then once you have located it, open the location and copy the file ''subinacl.exe'' and then ''paste'' it to the System32 folder of _any_ version of Windows 7 (both x86 _and_ 64bit).  Then, the ''reset.bat'' file will perform correctly for you.  (In other words, there is no need to alter the commands in the ''reset.bat'' file, if you simply copy and paste the ''subinacl.exe'' to the System32 Folder of _any_ Windows 7 version). 

-- The ''Important Note'' tip above, by itself,  can solve the ''80070005'' for many people who have been unable to _apply_ the ''subinacl.exe'' fix. --

As for myself, I intend to run the ''WindowsUpdateDiagnostic.diagcab'' before and after any ''major'' update and/or when Windows Update is seeking to install a large quantity of updates.

Personal Note:  I no longer run Windows Update in ''Automatic'' mode.  I now have it set to ''Check for updates but let me to choose whether to download and install them.''  This gives me the opportunity to choose whether or not run ''WindowsUpdateDiagnostic.diagcab'' before downloading and installing updates.

--------------------------------------------------------------------------

My Explanation:  Apparently, on some systems, the ''80070005 error'' actually exists _prior_ to installing the Sp1 update.  For some reason Windows Update will install the Sp1 update even though the ''80070005 error'' already exists in the system _prior_ to the update.  Then, after the Sp1 update is installed, it then becomes utterly impossible to remedy the situation.

On other systems, the installation of the Sp1 update _itself_ will always result in the ''80070005'' error.  For those users, the ''subinacl.exe fix'' (used along with with the ''reset.bat'' file), will very likely repair the problem; (if done as I explained above in my ''Important Note'' tip).

CAVAET:  It has been my experience that no single fix works for everyone, and for many there does come a point where the _only_ solution is a totally clean re-installation of Windows.

---------------------------------------------------------------------------

Hope the above helps.

Also, please note that sometimes you will get this error (and/or possibly a different coded error) simply because a particular update has already _previously_ been installed.  If your Windows Update refuses (''failure'') to install _particular_ updates, but is capable of installing other updates, the solution is very simple. Just ''hide'' the offending updates so that Windows Update will no longer attempt to install them.
