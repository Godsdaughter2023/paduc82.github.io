frederik long say:

Instead of stepping laboriously through a number of GUIs, let's cut this Gordian knot with a few commands. Try this:
1. Reboot the machine.
2. Log on under an admin account.
3. Click Start
4. Type the three letters cmd the Search box.
5. Press Ctrl+Shift+Enter
6. Run the command as Administrator.
7. Type the following commands and press Enter after each:
    takeown /F L:\  /a  /r  /d y
    cacls L:\ /t /g Administrators:F System:F
    (Respond with Y to the prompt "Are you sure")
    cacls "L:\Folder1" /e /t /g "User1":F
    cacls "L:\Folder2" /e /t /g "User2":F
    (Repeat the command as many times as you have user-specific folders)
8. Test the permissions
9. Examine the permission structure so that you can see what's different from the way it was before.

daddyman say:

Here is an explanation of my problem...and how I fixed it.

As I surmised from the very beginning, there was something I was doing that was fundamentally wrong.

I don't know why, but Authenticated Users (or Users or Everyone) need to have permissions (currently 'read and execute') on the root of my data partition (L:\) ; otherwise, no one can access anything on the partition...not even Administrators.

The mistake I made was in selecting "Replace all child object permissions with inheritable permissions from this object". By doing that,  each user's folders included permissions for Authenticated Users. That made it possible for each user to see each other user's files.

By removing "Replace all child object permissions..." in the partition root, I was able to assign the correct permissions to each user's folders. Now users can only see their own files and folders.

I don't understand why Authenticated Users needs to have permissions on the folder root to begin with. However, I noticed that Authenticated Users also has permissions on my C: drive (although the specific permissions are different). Authenticated Users also has permissions on the root of my second internal hard drive (with different permissions still.)

Some day I'll understand what the deal is with Authenticated Users. For now, my users can see only their own files and folders, and that was my goal.

Many thanks for your patience and advice throughout this ordeal.

wunders say: My first question is "How did you encrypt these files?".  There are many ways to encrypt, but for the balance of this post, I'll assume that you have XP Pro and used the Encrypting File System (EFS).

Rule #1 when even thinking about using EFS is to read and understand the following KB article, paying particular attention to the paragraph titled "Why you must back up your Certificates"

" Best Practices for the Encrypting File System "
  < http://support.microsoft.com/kb/223316 >

... and so the answer to your question is to simply restore your certificate from the backup that you made.

In order for you to lose access to your files, something must have changed in your system that affected your certificate.  The certificate itself is stored encrypted with several factors combining to form the key that unlocks the certificate.  One is your Secure ID (SID) which will change if you re-install windows or delete and re-create the user.  Another is your user password.  Changing your password the wrong way can invalidate your certificates.  Whatever caused your certificate to become invalidated, unless you can recover it, there is no way you will ever get access to your files again.  EFS has been dubbed by many in this forum as the "Delayed Trashcan".

A last resort would be to perform a System Restore to a point in time when you could access it but this is not guaranteed because, according to documentation, passwords are not restored as a result of a System Restore.  So you would need to change your password back to what it was when you last accessed your files before performing the System Restore.

"How to restore Windows XP to a previous state"
  < http://support.microsoft.com/kb/306084 >

As a personal recommendation looking forward, consider using the freeware Truecrypt for your encrypting needs.  I've been using it and its predecessors since the Windows 98 era without ever losing a file.  It will even work across computers, over the network, and on XP Home.  It also has the advantage that file names are hidden as well.

wunders say: John John hit the nail on the head.

You encrypted your files with EFS encryption by checking that box.
With EFS encryption, even an "administrator" cannot access the data without owning the certificate (encryption key) used to encrypt the data.  So it's not that you aren't being recognized as an administrator (which you likely are), it's just that even an administrator cannot decrypt the files.

Whenever you ever deal with encryption, if you can't answer the question "Where is my encryption key or certificate?",  you are setting yourself up for losing data. With some encryption programs, the key is obvious -- usually in the form of a password or passphrase.  With Microsoft's EFS, the encryption key is automatically created and stored without ever notifying you.  EFS makes it easy -- too easy.  So unless you educate your self beforehand things like this are inevitable.

Hi,

STOP: 0xC000021A  

Can be a difficult problem to solve and you may indeed need technical help from a real computer shop
(not leeks and peeks at a BestBuy or other BigBox stores) or the System maker's Support.


Cause

This error occurs when a user-mode subsystem, such as WinLogon or the Client Server Run-Time Subsystem (CSRSS), has been fatally compromised and security can no longer be guaranteed. In response, the operating system switches to kernel mode. Microsoft Windows cannot run without WinLogon or CSRSS. Therefore, this is one of the few cases where the failure of a user-mode service can shut down the system.

Mismatched system files can also cause this error. This can occur if you have restored your hard disk from a backup. Some backup programs might skip restoring system files that they determine are in use.

Resolving the Problem

Running the kernel debugger is not useful in this situation because the actual error occurred in a user-mode process.

Resolving an error in a user-mode device driver, system service, or third-party application:Because bug check 0xC000021A occurs in a user-mode process, the most common culprits are third-party applications. If the error occurred after the installation of a new or updated device driver, system service, or third-party application, the new software should be removed or disabled. Contact the manufacturer of the software about a possible update.

If the error occurs during system startup, restart your computer, and press F8 at the character-based menu that displays the operating system choices. At the resulting WindowsAdvanced Options menu, choose theLast Known Good Configuration option. This option is most effective when only one driver or service is added at a time. If this does not resolve the error, try manually removing the offending software. If the system partition is formatted with file allocation table (FAT), use an MS-DOS startup disk to gain access to the computer's hard disk. If the system partition is formatted with NTFS file system, you might be able to use Safe Mode to rename or delete the faulty software. If the faulty software is used as part of the system startup process in Safe Mode, you need to start the computer using the Recovery Console in order to access the file. If a newly installed piece if hardware is suspected, remove it to see if this resolves the issue.

Try running the Emergency Recovery Disk (ERD) and allow the system to repair any errors that it detects.

Resolving a mismatched system file problem: If you have recently restored your hard disk from a backup, check if there is an updated version of the Backup/Restore program available from the manufacturer. Make sure the latest Windows Service Pack is installed.
