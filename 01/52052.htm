The most common reason you can't save your project as a movie is that Movie Maker is struggling with one or more of the source files. Whenever I encounter that problem, the workaround is to convert the source files. Movie Maker prefers .wma for audio and .wmv for video.

The other possible cause, as alluded to by Ruth is that the project might be "too complex" for Movie Maker to handle. In that case, splitting the project is your best bet.

To diagnose your issue, I'd try the following steps.

  1. Make multiple copies of your original project file. Only work in the copies.
  2. In the first copy, try removing clips/music from the end of the project. Keep paring down the clips until you get something that will save as a movie. (If nothing works, then go back to my prior suggestion of converting the source clips to .wma and .wmv).
  3. Assuming you were able to save a portion of the original project as a movie, repeat the process.
  4. Open the next copy of the project. Remove the part from the beginning that was already saved in step 2. Now repeat the process of paring down from the end until you get another portion saved.
  5. Repeat for as many parts/copies as needed.

In the end, assuming you have generated lots of individual movies for the parts (say part1.mp4, part2.mp4, part3.mp4, etc.), start up a new project. Add the movies previously saved for each of the parts. Now save that as one final movie.

I'm sorry about the workarounds that are necessary, but hopefully that will help you with your project issue.

Memtest86+:

Download Memtest86+ here:

http://www.memtest.org/

Which should I download?

You can either download the pre-compiled ISO that you would burn to a CD and then boot from the CD, or you can download the auto-installer for the USB key. What this will do is format your USB drive, make it a bootable device, and then install the necessary files. Both do the same job, it's just up to you which you choose, or which you have available (whether it's CD or USB).

How Memtest works:

Memtest86 writes a series of test patterns to most memory addresses, reads back the data written, and compares it for errors.

The default pass does 9 different tests, varying in access patterns and test data. A tenth test, bit fade, is selectable from the menu. It writes all memory with zeroes, then sleeps for 90 minutes before checking to see if bits have changed (perhaps because of refresh problems). This is repeated with all ones for a total time of 3 hours per pass.

Many chipsets can report RAM speeds and timings via SPD (Serial Presence Detect) or EPP (Enhanced Performance Profiles), and some even support changing the expected memory speed. If the expected memory speed is overclocked, Memtest86 can test that memory performance is error-free with these faster settings.

Some hardware is able to report the "PAT status" (PAT: enabled or PAT: disabled). This is a reference to Intel Performance acceleration technology; there may be BIOS settings which affect this aspect of memory timing.

This information, if available to the program, can be displayed via a menu option.

We're experiencing an ongoing issue with outgoing Caller ID being inconsistent for users utilizing Skype for PSTN dialing (US):

  • Each site has a published number (defined as a Service number), that points to an AA scoped to users at each site
  • Each user at a site has a CsCallingLineIdentity policy defined as such for all users within:
  • Identity : Tag:FTC_CID Description : EnableUserOverride : False ServiceNumber : 1970484**** CallingIDSubstitute : Service BlockIncomingPstnCallerID : False
  • We've had multiple users at multiple sites (no rhyme or reason) spontaneously ignore the CsCallingLineIdentity policy, and broadcast their LineURI/direct number instead of the Service number as defined in policy.
  • In testing, we've observed instances where alternating calls will result in call #1 displaying Service Number, call #2 displaying LineURI, etc.  This is to multiple carriers as well (Verizon Wireless/ATT Wireless in this recent case).  When directed by support to obtain Tracing logs for affected users, we'll regularly find that we're unable to replicate the issue by the time we can coordinate testing.  
  • In one edge case, a user broadcasted our lone conference bridge dial-in number to a client (as verified by screenshots of the client's call logs).  This Conference Bridge number is in our tenant, but not defined in any CsCallingLineIdentity policy, and should never be broadcast for outbound CID.

Another thread with others experiencing the same issue can be found at https://techcommunity.microsoft.com/t5/Skype-for-Business-IT-Pro/Some-users-not-obeying-CsCallingLineIdentity-setting/td-p/123193

The vast majority of users (to our knowledge) don't appear to have issues, but a small subset that seems to shift on a nearly daily basis does.  As the called party doesn't know what to expect from CID, we're assuming there's a lot of instances of this happening that aren't being reported.
