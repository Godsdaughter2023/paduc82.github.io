Normally manfacturers put an approx. 10GB partition on the HDD in addition to Windows. Some have a 3rd of around 200MB that is for Windows use.

 

There is usually an included app to burn "Recovery Disks" somewhere with their preinstalled programs.

There is also usually an option that appears during POST when you first power on to let you Recover your computer to "AS NEW".

 

This is mainly the major manufactures that do this...and may not apply to all.

 

 

HP for example, calls the partitions "System" and "Recovery" which is the one that contains the files to return to "AS NEW", and would be burned to disk, and then "OS' for windows.

Windows 10 1709; host sleeps during active RDP session - Microsoft Community

gorkster say:

Windows 10 1709; host sleeps during active RDP session

I am reporting a bug/oversight.

HOST OS:

Windows 10 64-bit Version 1709 (OS Build 16299.192)

HOST SETTING:

Set to sleep after 2 hours of inactivity

PROBLEM:

Host computer sleeps after 2 hours of inactivity even when a client is connected via RDP (Remote Desktop)

DESCRIPTION:

In previous versions of Windows and even in Win10 before the Fall Update an active RDP session has always prevented the host computer from entering a sleep state even when set to do so after X minutes of inactivity.  Since the Fall Update the host computer of an active RDP session will sleep during an active RDP session when there is lack of activity on the client computer.  There should never be a reason for the host of an active RDP session to enter sleep mode when there is a client connected.  If the client computer is set to sleep after X minutes of inactivity it seems natural it should do so even during an active RDP session, but then, and only then, should the inactivity timer start for the host computer.

HISTORY:

There was a bug introduced in Remote Desktop sometime prior to Win10 Fall Update wherein a host computer would not sleep when set to do so after a period of inactivity after a client had connected and subsequently disconnected to the host via RDP.  This was finally fixed in Fall Update, but I have a sneaking suspicion the fix was too aggressive and this is what led to the problem I am reporting.
