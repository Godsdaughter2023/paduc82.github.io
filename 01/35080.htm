mac11b say:

I have a similar problem with 3 of 5 new Surface Pro 3 128. I load them using a Windows Backup system image stored on a central server. The unit will load fine and has all the latest firmware and updates. Also we have sleep and hiberboot disabled. We have hyper-v loaded to give us access to the addition power controls and have all of the power save functions turned off. Within 24 hrs., while sitting idle, but not in sleep or hibernate, for at least one hour the OS will lock up, I will get the unhappy face and then it will warm boot into the UEFI BIOS as if there is no mSATA drive present. This has happened once on 2 of the Surfaces twice on the third.

I have seen this issue on other non-Surface systems with non-mSATA solid state drives. The issue is that as the OS/mobo is continuously polling for the for the SSD/mSATA to perform read/write jobs the drive does not report back during one of those cycles. It is as if the drive goes completely off-line. The OS crashes and proceeds into a warm boot. Unfortunately the drive will not come back online until the system is brought down completely and cold booted. In the case of the Surface 3 you will get the recovery environment on that cold boot. If you select Restart the Surface will warm boot into the OS as if nothing happened, until it happens again. 

The fix we found in non-Surface systems is setting the SATA port for the system drive to be hot pluggable. This function allows drives to be added to an actively running PC so the mobo will poll for the drive when introduced as if you were using a hot swap bay and a backplane and the OS will pick up on the new drive without being rebooted. In this case if the main SSD system drive blips out for a cycle or two the mobo can keep re-polling for the missing drive as long as the OS will hold waiting for it. In the case of the Surface this OS will hang for about 30 sec to 1 min. If the mobo can re-detect the drive the OS moves on as if it was just a minor hic-up. 

Unfortunately the Surface does not have this PCH hot-plug option that can be found on conventional UEFI mobos like the Asus H97M-E or the gigabyte GA-H87N.

In conclusion, there is either something wrong with the Samsung PM851 MZ-MTE128 mSATA, a flaw on the Surface system board or there is an update that has come down the pipe since Feb. that is causing this.

I opened a case with Microsoft support and their only recommendation is to do a refresh because they could not determine what firmware is loaded on the Surface I have. I know it has all of the firmware released through Microsoft updates. I have seen them load and the images we use are progressive builds they we make as new updates are added. Classic Microsoft answer. "If we don't know what it is just reload the OS."

ganesh achar b say:

Hi Rob,

Thank you for the detailed information.

A Code 10 error is generated in Device Manager in one of the following situations:

• Device Manager can't start the device.
• One of the drivers that the device needs does not start.
• Device Manager has insufficient information to recognize the error that is bubbled up by the device driver.

Note In this situation, the Code 10 error can be a very generic message that indicates some kind of unspecified driver or device problem.

Update the device driver from the Device Manager and check. 

To update the device driver follow these steps.

a) Press Windows and X keys together, then click on Device Manager.‌

b) If you are prompted for an administrator password or confirmation, type the password or provide confirmation.

c) Look for the device right-click on it and select Update.

d) Reboot the computer.
