Error: 0x8078002A occurs when backing up to a 2.5 or 3.0 TB drive in Windows 7 Backup and Restore
Answer ID 6618 | Last Updated 05/19/2011    -->

Problem:
When using Windows 7 Backup and Restore with a 2.5 TB or 3.0 TB drive, the backup fails with an error code of (0x8078002A).

Cause:
This problem is caused by an issue with the Windows 7 Backup and Restore utility when using drives with 4K Logical sector sizes, which are found on WD 2.5 TB and larger external drives. This is different from an "Advanced Format" (AFD) drive which use 4K physical sector sizes. AFD drives should not cause this problem.

Solution:
It is still possible to use a 2.5 TB drive or larger with Windows 7 Backup and Restore, as long as you don't need to use the drive with Windows XP. Using the WD Quick Formatter tool, select the Factory Default configuration to reformat the drive.

Warning: This tool will erase all data on your drive, so take the necessary precautions.

After formatting the drive to Factory Default mode, the drive will be usable in Windows Vista and Windows 7. However, the drive will not be recognized in Windows XP. This is the only way to get Windows 7 Backup and Restore to work correctly.

To change the delay before keyboard characters repeat
 1.  Open Keyboard by clicking the Start button , clicking Control Panel, clicking Hardware and Sound, and then clicking Keyboard.
 
2.  Under Character repeat, move the Repeat delay slider to the left to increase the amount of time you must hold down a key before the keyboard character repeats. Move the slider to the right to decrease the amount of time before characters repeat.
 
 
 To change the speed at which keyboard characters repeat
 1.  Open Keyboard by clicking the Start button , clicking Control Panel, clicking Hardware and Sound, and then clicking Keyboard.
 
2.  Under Character repeat, move the Repeat rate slider to the left to make keyboard characters repeat more slowly, or to the right to make them repeat faster.
 

This should resolve your issue.  You are most likely a very fast typist and the keyboard is not responding fast enough to your touch so by playing around with the settings you can get it where it is optimal for your typing skill.

a. user say:

Outbound connections are permitted unless blocked by a rule - so the initial connection request should pass unless there is already a rule about it.  Inbound connections will automatically be blocked unless there is an exception already allowingi it.  Go to Start / Control Panel / Security Options / Firewall Settings / Exceptions and you should see a list of programs checked and unchecked to indicate whether or not they are exceptions.  You can click at the bottom to add programs and/or ports to the exceptions.  The initial request will pass through if it is using an exception port or it will be blocked.  To allow the initial connection unless it is already under one of the existing exceptions, you need to create the exception before the connection is attempted.

If you click on Outbound Rules and then create a new rule, you can block a program from making an outbound connection (and it will be added to the list). You can also change the rules for various inbound and outbound options using the program you posted the link on how to find.

What attempted connection are you talking about - something inbound or outbound?  Are you trying to block it or let it through?  You need to know the port it will be using.

I'm aware that notifications of attempted breaches is the default in Windows Firewall in Vista (and see mine is set that way - confirm yours is as well).  I do not know why you're getting no such notification in your situation unless the connection isn't being blocked because of some exception or some rule (or some flaw in the firewall and/or the notification process).  Come to think of it, I've never received such a notification myself and that's odd considering how much time I spend online.

There's a procedure to audit attempted breaches in Vista through Event Viewer:http://technet.microsoft.com/en-us/library/cc754714(WS.10).aspx.  Enable this and test the connection again and see if the block appears in the audit report (then we'll know it was being blocked but somehow the notification process isn't functioning properly or some other setting somewhere has turned it off).  Then we should know if it is getting through.

I hope this helps.

Good luck!
