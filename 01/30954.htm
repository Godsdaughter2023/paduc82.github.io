Your error message looks like it's complaining that WerFault.exe has an invalid signature and an unknown publisher, but WerFault is a Windows system file.

I'm starting to believe that this is an error generated by WerFault actually complaining about some other file.  One of the several very odd things about your problem, however, is that WerFault shouldn't even be invoked if you've really disabled the Windows Error Reporting service.

Let's try to check the digital signature of your file.  There are several methods for doing this, but most involve download Microsoft development "kits" and finding the correct tool.  An easier way is to download Sigcheck from https://docs.microsoft.com/en-us/sysinternals/downloads/sigcheck

Extract the contents of the zip file.

Open a Command Prompt window.

Drag sigcheck.exe or sigcheck64.exe into the Command Prompt window

In the Command Prompt, type a space after sigcheck.exe and then type -vrs -vt C:\Windows\System32\WerFault.exe

Press Enter

You should see something like this:

If "VT detection" is 0/68 as above, that means that none of the 68 antivirus tools that VirusTotal uses detected your WerFault.exe as being infected.

Copying the Virus Total link is a bit difficult.  You can right-click in the Command Prompt and select "Mark" and then drag over those two lines, like this:

Then press Enter and the "marked" material will be copied to the clipboard.  From there, open a Notepad window and Paste, to get:

You can then copy the link and paste it into your browser to see the report.

I expect that all of the above will just confirm that there is not a problem with WerFault.exe on your computer.

The next step is to determine what is causing the error.  Using "Clean Boot" configuration is a tedious process, but other than using tools like Process Monitor and Process Explorer to trace what is actually executing on your machine, it's the only way I know to find a start-time problem.

Here's how it works:

  1. Click Start and type msconfig in the search box.

  2. Press Enter.

  3. On the General tab, click Selective startup.

  4. Clear the Load startup items check box.

  5. Click the Services tab.

  6. Select the Hide all Microsoft services check box (at the bottom).

  7. Click Disable all.

  8. Click OK.

  9. Click Restart.

At this point, the computer has been started with a minimal set of programs and services.  The assumption is that these programs and services are not what's causing the problem, so you should not see the error.  If you DO see the error at this point, we have to approach troubleshooting from a different direction, so skip directly to the "After completing" steps.

Assuming the error DOES NOT appear, you add startup programs and services in a systematic way until you discover what program or service causes the error.  One way to start the process is to re-run msconfig, go back to the Services tab and with all Microsoft services still hidden, re-enable services that start with letters A through M.  Reboot the computer.  If the error does not arise, re-enable the rest of the services and reboot.  Rinse and repeat.

After completing your troubleshooting, restore your system by doing the following:

  1. Click Start and type msconfig in the search box.

  2. Press Enter.

  3. On the General tab, select Normal startup.

  4. Click OK.

  5. Click Yes when asked to restart your computer.

Background.

Despite what you think it means, "(green check) Outbound connections that do not match a rule are allowed" (which is the default) actually means that, unless you've gone to the extreme trouble to create blocking rules, the firewall is wide open to outbound connections -- essentially, you have no firewall for outbound connections. That means, 1, if you have a trojan in your computer that is looking to download its main-payload virus into your computer (see "A true virus story", below), or 2, if you have a keylogger virus in your computer that wants to "phone home" to report your on-line banking username & password to its criminal masters, there's absolutely nothing to stop them.

That's why I have set my firewall to: "(red no-entry) Outbound connections that do not match a rule are blocked".

A true virus story.

A couple of years ago I visited a well-known web site that had been hijacked -- was it cnet? I think so. The site basically looked okay, but something about it made me shy away. I left it without clicking on anything. But just visiting that site (with javascript enabled) transferred a trojan into my computer (see "Trojan details", below).

Once it was in my computer, the trojan then went through my firewall, outbound -- was my firewall zonealarm at that time? I think so -- and downloaded Bagel, a rootkit key logger.

I got no alert from the firewall because the trojan used a well-known port that was wide open, and I got no antivirus warning because Bagel didn't alter any existing program. It used unallocated file space, patched itself into the boot chain, loaded ahead of windows protected mode on the next boot, and reprogrammed the memory management hardware to hide its presence in memory. It took me 2 months to discover Bagel and another 4 months to get rid of it. It was hell. My keyboard and mouse both continuously acted up. I didn't have an on-line bank at that time. If I had, I'm sure my accounts would have been cleaned out because that's what Bagel does. It's a keylogger monitoring the keyboard and browser waiting to catch a bank login.

Trojan details.

The trojan was a javascript function that automatically ran on page load. It created an array (a mathematical object akin to a matrix) consisting of over a million identical array-members. Each array-member was a short string of numbers. As it looped adding array-members, the trojan eventually (in fractions of a second) overflowed the memory segment that the memory management hardware had allocated to my browser. That overflow partly overwrote the next higher segment in memory -- that's called a "bounds violation", and yes, that's possible if you position the base register just below the top of your segment and then use a large offset. There was a flaw in Windows that failed to detect that type of bounds violation and Bagel was designed to exploit that flaw. But the story doesn't stop there. In order to gain control, the short string of numbers in an array-member on the "wrong side" of that segment boundary had to be run. Now, that's the crapshoot that viruses are faced with, but if the program that was partly overwritten is still loaded in memory and running and if it runs the code at the bottom of its segment, then -- Voila! -- the virus gains control of the timeslice allocated to the program that was partly overwritten. In other words, the program that was partly overwritten is not in control anymore, the virus is in control. So, what did the trojan do? Totally in the background and without any clue to me, it went to a web site in Ukraine and downloaded the main Bagel payload. How do I know all this? I examined the javascript trojan and I disassembled -- disassembling is sort of like decoding -- the Bagel payload (the rootkit) which I saved after I found it some 6 months later. I still have the Bagel payload sequestered in a zip file in quarantine -- in fact, I have a menagerie of dozens of viruses that I use to test antivirus programs.
