    robert aldwinckle on forums say:

    I think I found the error:

    [Event 1014 - Name resolution for the name i2.answers.microsoft.com timed out after none of the configured DNS servers responded.]


    Interesting!  I did not know that the DNS client even bothered trying to use a lookup for a related name server.  It makes sense though.  It is a desperate last gasp attempt to get a lookup done.  I have always seen the problem as the fact that when the host name being looked up is an alias nothing ever caches a lookup for the canonical host name and hence the lookup's IP address is never cached.  We can get that done by doing a  ping -n 1 -w 1 for the canonical host name.  For example, this would be enough to do that but not enough to get the whole lookup cached:

    ping -n 1 -w 1 origin.answers.ms.akadns.net

    where the canonical host name has been found using either nslookup or an initial ping request.  You can then use  ipconfig /displaydns  to see how much time you have to use the cached lookup pieces.

    Heh.  Something has changed.  Now just doing the ping for the desired host alias name

        ipconfig  /flushdns

        ping  -n 1  -w 1  answers.microsoft.com

        ipconfig  /displaydns

    The flush is just being done for convenience to simplify scanning of the display.  The ping switches and values are chosen to try to reduce as much as possible the time lost between the time that the ping does its lookup(s) and the time that the display of them is done.  I.e. only do 1 iteration and don't bother waiting for a result from the ping (which in the case of Microsoft host names is typically is blocked anyway and then takes even more time by timing out.)  The only thing we are interested in is getting a complete lookup cached in order let IE's request be independent of the speed of the real DNS.

    So, that is what I usually advise as a workaround to the problem of a slow lookup.  The change in output for just a simple ping also explains why use of the Fix connection problems button is now often so much more successful than it once was for these cases.  Thanks for making me see this!

    Coincidentally recently I noticed that in the same KB article that we often suggest for helping with rendering problems there is also an implied help for lookup problems.  However, I have never seen its use advised so I don't know what value might be appropriate.

    Ref:

    How Internet Explorer uses the cache for DNS host entries

    http://support2.microsoft.com/kb/263558

    (Microsoft search for
        DNS timeout site:support.microsoft.com
    )

    Value Name: ServerInfoTimeOut
    Data Type: REG_DWORD
    Radix: Decimal
    Value: (time in milliseconds)

    </quote>
