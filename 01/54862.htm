markfilipak.windows say:

I created a workaround for my firewall problem:

I changed the 'Scope', 'remote IP addresses' of the outbound firewall rule from

the subnet notation: '192.168.0.0/12', to

the range notation: '192.168.0.0-192.168.0.255'.

Thus, I was able to create an effective 'Scope' of '192.168.0.0/24' by that indirect method.

I tested it by using the 'ping' command: I was able to ping my gateway '192.168.0.1', and I was able to ping '192.168.0.255' (ping responded: "Request timed out" as expected), but I was not able to ping '192.168.1.1' or beyond (ping responded: "General failure" -- that's good).

Now, regarding the bug:

'192.168.0.0/12' (which is what the firewall rule wanted) is 16 times bigger than the scope of the entire class-C LAN address range. The easiest way to see this is binary, thus:

A. Here is what the rule SHOULD BE:

11111111.11111111.00000000.00000000 // this is the /16 mask (entire class-C LAN)

11000000.10101000.00000000.00000000 // this is '192.168.0.0

11000000.10101000.xxxxxxxx.xxxxxxxx // this is the network part, plus xxx... is the host part

The host part: xxxxxxxx.xxxxxxxx == up to 65,536 hosts, and all of them are in the class-C LAN (none route to the Internet).

B. Here is what the firewall erroneously wants:

11111111.11110000.00000000.00000000 // this is the /12 mask

11000000.10101000.00000000.00000000 // this is '192.168.0.0

11000000.1010xxxx.xxxxxxxx.xxxxxxxx // this is the network part, plus xxx... is the host part

The host part: xxxx.xxxxxxxx.xxxxxxxx == up to 1,048,576 hosts. 983,040 of them route to hosts on the Internet. Thus, what the firewall wants is broken and dangerous.

Conclusion: The 'Scope' of firewall rules defaults to a /12 mask (that's wrong), and it rejects anything that's right.

PS: '192.169.0.0' is psc.com, and '192.170.0.0' is hp.com. Clearly, they are on the Internet, and clearly they are not in my LAN, and clearly the default filewall scope mask (/12) allows them. I didn't test any other of the (bogus) allowed IP addresses.
