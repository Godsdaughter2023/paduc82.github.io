cager say: Chkdsk is part of Windows - it is a utility to analyse and fix HD errors. The /f parameter is telling chkdsk to fix any errors it finds.

It may have been scheduled to run by the defragmenter becuase it found errors when it analysed your HD before defragmenting.

To manually run chkdsk, go to Start, click on Run and type chkdsk /f into the box. Then press ENTER. Chkdsk will scan the current drive and fix any problems it finds.

Once you have done this, you can try defragmenting again.

deb_333 say: I've tried this and it's still not working.  Is there any way to unschedule the chkdsk?

a. user say:

It does not mean that a chkdsk is scheduled.

When trying to defragment a volume and Disk Defragmenter reports:

Disk Defragmenter has detected that Chkdsk is scheduled to run on this volume: (C:)
 
Please run Chkdsk /f.

This means that the volume you are trying to defragment has been marked "dirty" by XP.  It doesn't necessarily mean any data has been lost on the
volume, it just means that XP thinks the volume needs to be checked for errors because of something that happened before. 

You will not be able to defragment the volume until the dirty bit has been cleared.

It does not mean that a check disk is scheduled to run and you cannot unschedule what needs to happen in this case because that is not the problem. 

As long as the dirty bit is not set on a volume, you can still run a disk defragment with a scheduled check disk pending (try it). 

Therefore, entering commands to reset or clear a pending check disk is a waste of time - that is not the problem.

A volume can be marked dirty if the system suffered a power interruption (power failure, plug pulling, battery removal, power button), aborted restart or ungraceful shutdown .  Have any of those events occurred?

When XP starts, it automatically checks the dirty bit on every volume.  You can also tell XP to ignore the dirty bit on any volume, but that is probably not a good idea in case there is a real problem that needs to be fixed. 

If the XP startup finds the volume is dirty, it will automatically run check disk with error correction (chkdsk /r) and try to correct any problems it finds.

If the chkdsk succeeds, it will clear the dirty bit and then you can defragment the volume. 

You can check the status of your dirty bit from a Command Prompt window.

To do that, click Start, Run and in the box enter:

cmd

Click OK to open a Command Prompt window.

For example, to query the dirty bit on drive C, type:

fsutil dirty query C:

Sample output:
Volume C: is dirty
Volume C: is not dirty

You cannot use fsutil or chkntfs from a Command Prompt to clear the dirty bit on a volume.

The only thing that can clear the dirty bit on a volume is chkdsk /r (or chkdsk /p).  You could tell XP not to check the dirty bit on a volume
where the dirty bit just will not clear, but that is probably not a good idea since there is some kind of problem on the volume.

You will also not fix this problem by editing your registry - that is not where the problem is.

Once a volume has been marked dirty, the only way to clear the dirty bit is to run "chkdsk /r" or "chkdsk /p" when the volume is not in use.  The only
way a volume can be not in use is while XP is just starting to load or from the XP Recovery Console.

If XP will not run chkdsk properly when your system reboots or the dirty bit will not clear as it should, you need to look in the Event Viewer for the
chkdsk log and see what is going on and fix it.

If XP still won't clear the dirty bit, you should manually run chkdsk /r from the XP Recovery Console so you can watch it as it runs and see what it does and run it again until it does work. 

Telling XP not to check the dirty bit on a volume would be risky behavior since there would be some kid of problem with your hard disk.
