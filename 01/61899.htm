codemason say: I had this problem too. My problem was caused by the different drive letter assignments for the boot drive and the new user storage in the recovery console vs normal boot.

In a normal boot I had
c:   boot drive
f:    new user storage

In windows recovery I had
e:    boot drive
d:    new user storage

So when running mklink in recovery boot, the command I needed to run was
mklink /J e:/Users f:/Users

Here E: is where the recovery boot finds the Users folder but F: is where the normal windows boot will eventually find the new user storage.

To fix, you need to know what the drive letter for your target will be when Windows boots normally, you might have to explicitly assign a drive letter. If you don't now what the SSD drive letter is, put all your Users folder(s) back on C:\Users so you can boot Windows normally, work out what the drive letter is and then repeat the LifeHacker hack.

I have had some issues that might be attributable to using this link technique
  • Windows Easy Transfer thought that there was not enough space on c: even though the files would end up on f: where there was heaps of space, this may also happen for any operation that tries to estimate disk usage that traverses a link
  • facebook cannot upload photos that traversed the link, I had to use f:\Users\... which defeats the entire purpose of this link. 
  • it might be that Java 7 plugin for firefox is not working in my browser because of the linK
I might be undoing this on my system.
