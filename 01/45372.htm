tryx3 say:

Strange, I don't see how I missed that 20th July post [post dates appear in local timezones so if we are on opposite sides of the world it is possible that you posted it on 19th / 21st].

I did keep reading your progress in TenForums over that period.

1  Built-In Admin

As I said before, the Built-In Admin in Windows 10 is not what it was in previous Windows versions.

In Windows 10, the Built-In Admin cannot run Apps. 

  • Apps cannot run with an Admin token
  • The Built-In Admin does not have a split token only the Admin one.

But an Admin account that you create can run Apps

  • Your own Admin accounts have both Admin & user tokens. 
  • Your own Admin accounts are logged in with just their user tokens invoked.
  • The act of giving Admin permission to a task is actually the act of invoking an Admin account's Admin token.

So your own account is better than the Built-In Admin. 

In Windows 7, by contrast, the Built-In Admin was as powerful as your own Admin accounts but it had, by default, switched UAC to off whilst you would have had to do so manually for your own Admin accounts to achieve the same behaviour.  The Built-In Admin was never more powerful then your own Admins.

In Windows 10, version upgrades can reset the Built-In Admin to disabled.  Some reports of this have been made in this forum.  As long as you also have another Admin account ready then that would not be the end of the world.

In both Windows 7 & Windows 10, the Built-In Admin had, by default, UAC set to off.  It has therefore always been essential to use the computer in isolation when the BiA is in use because there would be not even be any prompt to warn of a hacker trying to install something remotely.

MS did not leave the BiA "as an option", MS left it disabled.  In both Windows 7 & Windows 10, the Built-In Admin exists purely in order for it to be automatically enabled and for it then to appear in the Safe mode login screen if Windows has detected that there are no functioning Admin accounts on the computer.

In addition, the more a user account is logged in with and used, the more opportunities there are for its profile to become corrupted.  If you routinely use the BiA then its profile might be the one that suffers from user profile corruption and you would probably not be able to fix it so that built-in recovery facility would be lost.

If you are trying to ensure that you have an Admin account ready for use in case of user profile corruption in your normal Admin account then you can create an additional password-protected Admin local account now.

2  Guest account

MS did not leave the Guest account "as an option", MS left it disabled.  It has been regarded as a security loophole for so long that I cannot even find any old notes on the subject - I think it was merely that the UI did not provide any means of setting a password for it so hackers could try to re-enable it remotely and log into it to gain a foothold from which to attack the system.

You can add additional standard user accounts as much as you want.  When you configure access permissions, you can give or withhold access permission by account as you see fit. 

If you want to, you can create standard user accounts then use net commands to make them part of the Guest group and to remove them from the Users group. 

net localgroup Guests SomeUserAccountOrOther /add

net localgroup Users SomeUserAccountOrOther /delete

The result of this would be that all such user accounts could have their access permission set collectively if that matters to you.  When you look at permissions you will see Users and Authenticated users as groups for whom you can allocate permissions without having to specify each account name individually. 

  • Authenticated users are all the accounts that have an account on the computer [such as all the Admin & Standard accounts you have created & the BiA] except for any that have been reverted to only the Guest group. 
  • Users are all the accounts that have an account on the computer including members of the Guests group.

So, in short, you can regard people as guests on your computer when you have created Standard user accounts for them but there is only a slight potential advantage in reverting them to just the Guests group because of that possibility of configuring their permissions collectively. 

If you have visitors then you can give them standard user accounts.  If you give them all the same one they will see the files left by all the others unless you delete & recreate the account between their visits [see Guest Account [equivalent] - TenForums]

If you have multiple visitors then you can give them each their own account and revert it to the Guests group so that you can manage their access permissions collectively.  Doing so would avoid having to keep removing & recreating accounts.  It would also avoid any troubleshooting, inconvenience, embarrassment or security breach if one of them linked the account to their own MSAccount by accident or design.

3  Why are you doing this at all?

If you explain what it is you are trying to achieve with these changes then perhaps I can suggest an alternative way of doing it.  I know you want them to appear in the login screen as accounts you can log in with but why do you want to do that?  There is no benefit that I can see.
