Hi,

Are you not using the network sharing feature in Windows Media Player?

 

If you are using that service then this service is necessary to run to check for the network computers or devices with which you are sharing your music. This service also indexes/updates the library that is being shared.

 

If you are not using this service and still it is using the CPU you may change the type of start up to manual (that is has to be started by the user).

 

To disable the service temporarily

 

a.      Open up Services from Control Panel, or type services.msc into the start menu search/run box.

b.      Double click on Windows Media Player Network Sharing Service, and then change the service start up to Manual.

 

This has now temporarily disabled the service unless you manually start the service. You may restart the computer and check if the service has been stopped and the CPU usage had come down.

Hi Abwettar:

Just some hints about further diagnostics [assuming you haven't given up and purchased a new Win 10 laptop already :-)]:

I was re-reading your original post and noticed you mentioned a BSOD.  There is a free utility from Nirsoft called BlueScreenView that can be used to interpret the Bug Check (BC) codes in the Windows mini-dump files collected after your BSODs.   If you'd like to post details from your dump files you can do the following:

  1. Download the .ZIP file for BlueScreenView (BlueScreenView.zip for 32-bit OS; BlueScreenView-x64.zip for 64-bit OS) from http://www.nirsoft.net/utils/blue_screen_view.html.  Scroll to the very bottom of the page to find the download links.
  2. Unzip the downloaded file (no installation required) and double-click BlueScreenView.exe to run.
  3. When the scan is completed, go to Edit | Select All.
  4. Go to File | Save Selected Items and save the report as a text file (e.g., BSOD.txt).
  5. Copy and paste details of any recent BSODS from the .txt file in your next post.

BlueScreenView only lists details (filename, product name, file description, and file version) of the suspected driver / module that was loaded in the crash stack at the time of the crash so it isn't as accurate as a detailed Windows Debugger (WinDbg) analysis of your full dump files, but I've used BlueScreenView to successfully diagnose BSODs caused by outdated graphics drivers and other common issues.

If you are hearing short or long beeps during boot-up that could be your computer generating a diagnostic beep code during the power-on self-test (POST) when it checks your internal hardware. The article Computer POST and Beep Codes lists some common beep codes for different brands of BIOS, and the type of BIOS in your computer should be listed in the summary of your System Information at  Start | All Programs | Accessories | System Tools | System Information as shown in How to Find the BIOS Type, Version, and Date.  MS MVP Bill Smithers posted further information about POST beep codes <here>.

Piriform's free Speccy (https://www.ccleaner.com/speccy) is also useful for running a quick diagnostic of your system.  After Speccy has finished collecting your system information go to File | Publish Snapshot and then copy and paste the URL this generates (e.g., http://speccy.piriform.com/results/xxx... ) in your next post as instructed in the support article Publishing a Speccy Profile to the Web so we can see a snaphot of your system.  If you don't want to install Speccy on your hard drive you can also save the portable version of Speccy from their builds page at https://www.ccleaner.com/speccy/builds onto a USB thumb drive and unzip it there.

I'm not a certified hardware specialist, so if you think it's time to get an expert opinion I'd suggest posting in the Windows Crashes, BSOD, and Hangs Help and Support board of the BleepingComputer forum.  The post Blue Screen of Death (BSOD) Posting Instructions - Windows 10, 8.1, 8, 7 & Vista that is pinned at the top of that board explains how to run and post the required diagnostics in your first post.
