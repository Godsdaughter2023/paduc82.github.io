Media Control Interface (NMCI) not working properly on Windows - Microsoft Community

fritsjansenz say:

Media Control Interface (NMCI) not working properly on Windows 10

Like previous Windows versions, Windows 10 offers application programs a “Media Control Interface” (“MCI” hereinafter) allowing application programs various multimedia functions, like playing an audio file (e.g. .MP3, .WAV or .MID files). 

On Windows 10, this works OK right after the system is started, but if the user types control-break for whatever reason, the MCI interface no longer works properly: notably, a call to play a music file terminates immediately, while normally (if the MCI_WAIT option is specified in the call) it only terminates after the file is played. Remarkably, no error code is given if the file is not played.

This is clearly a bug in the Windows systems software. The functionality of Windows as it is sold offers various application program interfaces need by programs from Microsoft and third parties.

I have been using the MCI API in Windows XP for years without any major problems. 

For demonstration purposes, I wrote a minimal program containing nothing more than needed to demonstrate the bug:

/* PLAYSIM - very simple program to play sound files using the Windows Media Control Interface */
#include <stdio>
#include <windows>
#include <mmsystem>

int main( int argc , char ** argv )
{
   char * fileName ;
   MCI_OPEN_PARMS    openParms    = {0};
   MCI_PLAY_PARMS    playParms    = {0};
   MCI_GENERIC_PARMS genericParms = {0};

   if ( argc < 2 ) { puts("No arg."); return 0 ; }

   openParms.lpstrElementName = *++argv ; /* filename */
   if ( mciSendCommand( 0                   , MCI_OPEN  , MCI_WAIT | MCI_OPEN_ELEMENT , (DWORD) & openParms    )) puts("Open  failed."); else puts("Open  OK.");
   if ( mciSendCommand( openParms.wDeviceID , MCI_PLAY  , MCI_WAIT                    , (DWORD) & playParms    )) puts("Play  failed."); else puts("Play  OK.");
   if ( mciSendCommand( openParms.wDeviceID , MCI_CLOSE , 0                           , (DWORD) & genericParms )) puts("Close failed."); else puts("Close OK.");

   return 0 ;
}

This program can be compiled as a C program (no C++ functionality needed). Invocation: PLAYSIM.EXE <audio file name> (e.g. MP3 file).

This program always works fine under Windows XP. Under Windows 10, it says "Play OK" too, but it does not - after the first ctrl-break issued by the user.

Note that the MCI interface always suffered from the somewhat strange but not really problematic phenomenon that issuing ctrl-break for any other application in the system causes the playing of the sound file to stop. 

For completeness I note that I am aware that there are other, more modern interfaces to achieve the same functionality. But Windows 10 formally supports the MCI interface, and a conversion of applications using MCI to another API is a major effort I would like to avoid.
