hairyfool say:

I am sorry but it is. There are two separate status involved with a share.

Sharing permission when applied to a folder makes all subfolders within that folder accessible but the individual subfolders are not shared directly. You can also share any subfolder individually if you wish to apply separate sharing rights but they will still all be visible through the parent folder

For example folder A is shared with subfolders B, C, and D. If a user has sharing permissions for A they will see B, C, D. If D is also shared a different user may have access to D but not A so also cannot see B or C

You cannot unshare B or C because they are not individually shared.

Completely separate to Sharing Permissions are Security Permissions, one problem is by default some versions of Windows do not show the Security Tab. Security Permissions are defined on a per user basis and in the majority of cases subfolders inherit security rights from its parent folder so whichever users has access to folder A will have access to folders B, C, and D. In this case you can add additional rights to a subfolder as well as the inherited rights but if the inherit right option is in place you cannot remove one.

To remove an inherited right there is an option to remove automatically inherited rights and assign rights specifically to a folder. If folder A (note, not Share A) has user W, X, and Y with permissions to read the folder then inherited rights are applied to B, C, and D. While that is in force you can add user Z to folder D but not remove X, Y, or W. To remove user W you have to remove that inherit flag.

It is this separation of Share Permissions and Security Permissions that is the cause of so many "Access Denied" messages.
