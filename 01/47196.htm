keitarochan say:

Hi Patrick,

Unfortunately some of the methods below do not allow access to the "Guest" account directly and can cause a "PowerShell Policy Failure" upon selecting the "Guest" account, providing it actually shows on the User Accounts Screen.

You'll need to ensure that your version of Windows 10 can support Group Policy edits before continuing any further.

Here's the method I followed for my Spare PC at home which is running Windows 10 Pro and was successful in activation and login to the "Guest" account without having to create a "Visitor" or "Guest Account" profile.

  • Push Windows Key + R (or go to start and type in "Run")
  • Type in the following without quotations "gpedit.msc". (Please exercise caution when making adjustments in the Group Policy editor. Making changes to something you have no knowledge of can cause your computer to become unstable or potentially inoperable.)
  • 1 of 2 things will happen; If successful, you'll see a window called Local Computer Policy Editor show up with the options of "Computer Configuration" or "User Configuration"
    If unsuccessful, you'll see a message pop up saying you cannot make changes to your Group Policy on this machine. You may not be able to continue any further if this is the case and may need to go down the path of "Visitor".
  • If successful, you'll need to navigate to the following 2 areas;
    1. Computer Configuration > Windows Settings > Security Settings > Local Policies > Security Options > "ACCOUNTS: GUEST ACCOUNT STATUS" Default selection is "Disabled". Double click the security option and click on "Enabled". Click "Apply".
    2. Computer Configuration > Windows Settings > Security Settings > Local Policies > User Rights Assignment > "DENY LOG ON LOCALLY" Default selection is "Guest". Double click the security option, click on "Guest" and then click "remove" below. Click "Apply".
  • Restart your system. Upon restart on the user accounts screen, you should be able to see the "Guest" account below any other accounts already on your computer. Login will take some time to prepare Windows 10 for the first time and should grant access directly after.

Again please exercise caution with the Group Policy Changes, but this allowed me to access the Guest account successfully.

Hope this helps,

NIS is conflicting with the updating process by possibly causing corruption of the updating database, transaction logs, or the contents of the Download subfolder where the updates are stored.
You can remove the updating database from Windows\Software Distrbution\Data Store\DataStore.edb <-- this is the database
in order to remove it you will have to stop the Automatic Update (AU) and, if started, the Background Intelligent Transfer service (BITS).
Open a command prompt ( Start > Run > type in cmd > Click OK or press Enter ) and enter the following commands, pressing Enter aftereach command

net stop wuauserv
net stop bits
exit

Move DataStore.edb to the Recycle Bin for now. 
NOTE: The removal of DataStore.edb will also remove the updating history shown on the Microsoft/Windows Update web sites.

Restart the system and see if the CPU issue with wuauclt.exe still exists. If it does, then you can move the transaction logs to the Recycle Bin, too, after stopping the AU and, if started, BITS services.
All of the logs will be recreated by Windows when Automatic Updates kicks in again.
