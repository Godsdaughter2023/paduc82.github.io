I'll be frank:  In short, Windows Search simply does not work.  Most folks find this surprising, but it is what it is.

It's a broken design that even when everything is working perfectly will NEVER find some things in some files.  When everything is NOT working perfectly (which seems to be quite common) you cannot be sure what it will find or whether it will find anything at all.  It sounds as though the latter state is the condition you're in.

Windows Search is tied to indexing, which complicates things tremendously.  Indexing can be disabled (IMO a good idea) but this further complicates things because then the configuration items that describe what types of files can be searched and whether their contents will be searched simply disappear.

Even FURTHER complicating things is that Windows Search expects you to learn a particular syntax to be able to find things effectively.  For example, to find just filenames with the letters xyzzy in them, you'd want to enter filename:*xyzzy* into the search box.  If you just type xyzzy it will try to find the string INSIDE files as well as in their names.  Entering intutive search strings like *.jpg simply do NOT work as you would expect them to.

The best solution (and the one I've implemented) seems to be to avoid using Windows Search to find things inside files, and to use it only minimally to find file names.  This describes what I'm talking about in more detail:

1.  Disable indexing (by Disabling the Windows Search service).  This still allows Windows Search to work, but it actually searches files right now instead of looking in some (possibly corrupted) index.  It also works around another major file system problem (search the forum for "atomic oplocks" some time).

2.  Don't use Windows Search to try to find file contents.  Use it only to find file or folder names (using syntax similar to the above).  There's a good reference for the syntax expected here:  http://www.microsoft.com/windows/products/winfamily/desktopsearch/technicalresources/advquery.mspx

3.  For looking for things inside files, install a good 3rd party tool that does a REAL search, such as the freeware program grepWin:  http://tools.tortoisesvn.net/grepWin

4.  I find that with this setup it still properly DOES locate things in the Start menu, which I find handy.

Hope this helps.

Disabling Windows search appears to have far  too many negatives - so let's forget that.

The reason for using a Windows Search like tool is that the results are presented in Explorer, and you can drag and drop them on things.  The command line may be more deterministic for finding filenames, but it sure isn't helpful regarding integrating the results into the rest of the system.

Disabling Windows Search entirely may be a bad idea.  As one who HAS disabled Windows Search indexing I'm not seeing a downside.  I'm getting what you people are referring to as the "Grep" search every time, and thus more consistent and accurate results than anyone running indexing.

But that term is a misnomer.

I know what grep (derived from Global Regular Expression Print) is and what it's origins are (I have Unix experience as well as Windows).   Microsoft DOES have some semblance of a grep-like command line tool (findstr), but this is the first I've heard of the term being used in conjunction with Windows Search.  Other than findstr, the feature called "Windows Search" MOST CERTAINLY DOES NOT accept regular expressions, unless there's some user interface I'm completely missing.

Windows only indexes/finds things it anticipates you might look for...  What are they?  Plain text strings?  No one knows.  Could I search for "ABC%#XYZ", for example, and expect to find something? 

Then there's the small matter that the method by which Windows Search indexes/finds those things is by recognizing the file type and submitting each recognized file to a filter module.  Trouble is, there's no fallback, so an unrecognized file type is ignored.  With indexing on you can reconfigure it to search an additional file type (e.g., a .log file) but that doesn't cover unexpected file types that the system hasn't seen before.  Worse yet, some file types have insufficient filters - the .doc filter module won't read inside older Word files, for example.

Lastly, under NO CONDITIONS will Windows Search find Unicode text inside any types of files.  This is a Microsoft format!

I stand by my first statement - Windows Search is a broken implementation of an incomplete, invalid design!

-Noel

cmdrkeene say:

That's unfortunate.  Windows 7 and higher don't come with a MAPI-compliant email program (Windows Vista was the last, with it's essentially renamed version of Outlook Express being bundled in as Windows Mail).  But in 7 and later, on a default install there's no MAPI email program at all (and on Windows 7 there's not even a plain email program of any kind whatsoever, 7 was very incapable on a plain install).  Anyway, my point is that Windows 7 and higher don't have any mechanism for sending mail.  That's why you can't right-click a file and "send to mail recipient" or use Explorer toolbar email buttons in Windows 7 and higher. You have to install Outlook/LiveMail/Thunderbird to make that ability "light up", and you'll get that error until then.

But if the error is prompting even if the user never clicks anything, then I'd start by pressing Ctrl+Shift+Escape and when it opens, disable everything on the Startup tab. If the dialog still occurs, keep it open and use Microsoft's Process Explorer tool (download link) to identify it's process.  Task Manager might even be able to show it if you right-click and show details on the process, but Process Explorer will show the chain of execution and show exactly what program tried to call a MAPI email command.

Windows 10 1511 refers to the version of Windows 10 you are running, which corresponds to the version number 10.0.10586.  You can see the version 1511 reported by running winver.exe from the Run dialog (Windows+R) or a command prompt.

For an attack to be successful, this vulnerability requires that a user open a specially crafted Journal file with an affected version of Windows Journal. In an email attack scenario, an attacker could exploit the vulnerability by sending a specially crafted Journal file to the user, and then convincing the user to open the file.
