The symptoms: 

  1. The mouse will pair, though it's detected as a general bluetooth peripheral instead of a mouse;  
  2. If the computer goes to sleep, is rebooted, or even after the mouse will not reconnect automatically despite being paired;
  3. If the mouse is left still for a few minutes, it will also not reconnect; 
  4. The only way to get the mouse to work again, is to re-pair it. 

As far as I can tell, this issue affects every single Microsoft Arc Touch Bluetooth and Microsoft Designer Bluetooth mouse out there, though only with macOS Sierra and future versions (it worked okay with El Capitan). 

Needless to say, this renders the mice into paperweights. Frustrating for home use, but especially unacceptable in a professional environment; they go dead halfway through meetings, for instance. 


My question is, what can we (owners of these incompatible mice) expect?

On one hand, we bought mice (recently) that apparently don't conform fully with the Bluetooth Core Specification Version 2.1 + EDR or higher (which in turn means they don't pass Apple Bluetooth Guidelines Validation either). On the other hand, this is a serious shortcoming, which means Apple won't care either (why should they?). So users that put their faith into these particular Microsoft product just lost the cost of these mice (not exactly inexpensive).  

Is there any hope for some sort of driver (if it's even possible)? Is it neccesary to have all of these
vaseem ahmed say:

Hi Par401,
 
Yes It is necessary to have these to programs files. If you have 64 bit operating system then all your 32 bit files will be stored in Program Files (x86).
 
Program files (x86) provides you with the location for 32bit software, and the Program files folder is the one for your 64bit software. It is a good design decision to separate the location of programs with different architecture types.

The "C:\Program Files" folder and tree is for 64-bit applications, and the "C:\Programs Files (x86)" folder and tree is for 32-bit applications. The separation is necessary the applications will often share the same file names between the 32 and 64-bit editions, but can't use one another's versions of the files.

hairyfool say:

Just some background information for your issue.

Microsoft provide various services on line and control access through the use of a Microsoft Account. Because that account is on the internet and therefore potentially accessible by anyone who knows your email address it must be protected by a password, which is logical. There are ways to use a MA without entering a password but this requires specific steps.

What Microsoft want you to do is to log onto the PC with those same Microsoft credentials which then means you connect to those services seamlessly, so as an example the OneDrive online storage appears as another place to store files on your computer even though it is on line.

What you have opted for is to log on locally (Local Account) which is basically the same as you would have done with Windows XP, Vista, 7. You can still use those on line services but you would need to authenticate yourself when you connect to each of them.

No specific reason why you have to use a Microsoft Account but you can always switch later. One word of warning. If you should connect to any Microsoft services (Onedrive, Windows Store, Xbox etc) read the initial dialogues carefully. As I said Microsoft WANT you to use the MA login and when it sees you connect with a local account it tries to push you to doing that switch. The trouble is the wording of that dialogue can be a bit obscure and like your setup screens the option to decline shielded.
