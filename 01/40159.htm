    I've solved my problem but I have to admit I'm disappointed in Microsoft that I had to waste five days of my life figuring out the solution on my own.  It mystifies me that the solution to these problems by Microsoft as well as the community in general involve running System Restores and/or System Repairs. That’s like trying to trim a hang nail with a hatchet.

    Here is the solution:

    1.    First you must take ownership of the C:\Windows\winsxs\ directory and all subdirectories. Also, I gave my account “Full Control” permissions to the directories so that I could manipulate the files in later steps.

    2.    I needed to ensure that the latest and greatest version of the assembly in question was installed on my PC. To be more accurate I needed to ensure it was on my PC since I mentioned that installing the latest C++ 2005 Redistributable resulted in an Error 1935. To accomplish this I first located another Windows 7 machine that had the latest version of the assembly in question within the C:\Windows\winsxs\ folder. Once found manually copied the folder from that machine and put them on the problem child machine. Specifically that folder was:

    C:\Windows\winsxs\x86_microsoft.vc80.openmp_1fc8b3b9a1e18e3b_8.0.50727.4053_none_3b0e32bdc9afe437\

    Note:  Remember I was having trouble with the OpenMP assembly hence the specific folder listed above. You might be having a problem with another or multiple assemblies in which case you would obviously want to include those folders.

    In addition, I copied the files related to that version located under the Manifests directory and moved them to the same location on the problem child machine. Specifically those files were:

    C:\Windows\winsxs\Manifests\x86_microsoft.vc80.openmp_1fc8b3b9a1e18e3b_8.0.50727.4053_none_3b0e32bdc9afe437.cat
    C:\Windows\winsxs\Manifests\    x86_microsoft.vc80.openmp_1fc8b3b9a1e18e3b_8.0.50727.4053_none_3b0e32bdc9afe437.manifest
    C:\Windows\winsxs\Manifests\    x86_policy.8.0.microsoft.vc80.openmp_1fc8b3b9a1e18e3b_8.0.50727.4053_none_6b86c0e9b0196766.cat
    C:\Windows\winsxs\Manifests\    x86_policy.8.0.microsoft.vc80.openmp_1fc8b3b9a1e18e3b_8.0.50727.4053_none_6b86c0e9b0196766.manifest

    At that point the assembly is on your PC despite not technically being installed through the redistributable.

    Note to Microsoft:  It might be a good idea to look into the 1935 error further. Have you noticed it is like a rash on the internet?  Sure reinstalling the redistributable fixes it in most cases but not all cases.

    3.    Finally I needed to convince Windows not to try to use the version of the assembly it so stubbornly wanted to use (8.0.50727.3079). A version I will repeat was nowhere on my system.  To accomplish this I had to manually create a policy file for the 8.0.50727.3079 version called:

    x86_policy.8.0.microsoft.vc80.openmp_1fc8b3b9a1e18e3b_8.0.50727.3079_none_abac38a907ee8801.manifest

    The contents of the file are as follows:

    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
    <!-- Copyright © 1981-2001 Microsoft Corporation -->
    <assembly xmlns="urn:schemas-microsoft-com:asm.v1" manifestVersion="1.0">
        <assemblyIdentity type="win32-policy" name="policy.8.0.Microsoft.VC80.OpenMP" version="8.0.50727.3079" processorArchitecture="x86" publicKeyToken="1fc8b3b9a1e18e3b"/>
        <dependency>
            <dependentAssembly>
               <assemblyIdentity type="win32" name="Microsoft.VC80.OpenMP" processorArchitecture="x86" publicKeyToken="1fc8b3b9a1e18e3b"/>
               <bindingRedirect oldVersion="8.0.41204.256-8.0.50608.0" newVersion="8.0.50727.3079"/>
                <bindingRedirect oldVersion="8.0.50727.42-8.0.50727.3079" newVersion="8.0.50727.4053"/>
            </dependentAssembly>
        </dependency>

Copying files between PC’s from one PC’s to another might affect the functionality of the application/game because the registry keys are generated for that game to cooperate with the current system and its hardware/software/firmware. Hence you may face issues running the game once the process is complete. However you may try the steps below and check if it works for you.

    2. Locate the folder C:\Program Files\WindowsApps. If you’re not able to find, enable Viewing hidden files and you must have administrative rights to access the contents of this folder.

    3. Look for the folder the game’s files reside in. It should look like

      C:\Program Files\WindowsApps\NameOfTheGame

    4. Copy the folder to the PC you want to move the game to. Copy it to your desktop or to a drive that isn’t the OS drive.

    5. Open the Microsoft store on the device which you wanted to install and look for the game.

    6. Click the download button and allow the game to download. Pause it shortly after it starts and close the Windows store app.

    7. Go to “C:\Program Files\WindowsApps” where the Windows Store app will have created a dedicated folder for the app. Delete it. Copy the folder you brought over from your other PC to “C:\Program Files\WindowsApps”.

    8. Open the Windows Store app again and resume the download. The game will begin to install.
