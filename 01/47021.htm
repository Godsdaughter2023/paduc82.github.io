michael bourke say:

POSSIBLE SOLUTION: WINDOWS 10 UPGRADE DELETES OVER-PROVISIONING ON SSD!

After upgrading from Windows 8.1 to Windows 10 I experienced the same slow boot issues. This wasn't anything to do with what was happening IN Windows -- the delay was happening before Windows started, and continued after I rolled back to 8.1 (my Lenovo X220 was not fully compatible). I realized that it must be something that had changed on my SSD and I finally found out what it is:

The Windows 10 upgrade process deleted the over-provisioning space on my SSDs!! I had set up 10% for over-provisioning with Samsung Magician for both my Samsung SSDs and after the upgrade it was gone from both drives. After re-applying 10% over-provisioning with Samsung Magician everything returned to normal after around 24 hours. It wasn't immediate, probably because the drive firmware needed time to get its references set up again.

If you have a Samsung SSD you just need to reactivate over-provisioning with the Samsung Magician utility. If you have a different brand you will need to re-activate it using the method you used to set it up in the first place. Make sure you have a backup, because any changes to hard disk structure are potentially dangerous. 

Info: Over-provisioning space is free space that an SSD needs to use to replace bad sectors. When sectors go bad they are marked as bad and the firmware uses sectors from the over-provisioning space instead.

¡firedog say:

I have a suspicion that this has been fixed now ...

Rats! Just as I'd spent hours staring at impenetrable message headers looking for the rogue element ...

Copying and pasting into this forum is not the best way of sharing data like these, where the tiniest variation might be crucial. For example, at one stage I noticed that each of the header lines apart from the first started with a single space. I took this as some quirk of the poster's text editor, the forum post editor, my browser or my own text editor and swiftly removed all those superfluous spaces. I then tried tacking the body of the Argos message on to rml's headers, doctored to replace email addresses where they had been removed, and then opening the result in Windows Live Mail. No problem. I thought again about those extra spaces and put them back again, then did the same test. Sure enough, the message displayed properly, but the header bar entries were blank - exactly the symptoms you have described.

Your remarks about the =09 characters appearing in the afflicted messages but not in the others may be related. In Quoted-Printable encoding, non-printable ASCII characters are represented by an equals sign followed by the character's ASCII code point in hex. Because the equals sign has this special significance in QP text, any instances of it in the text to be encoded have themselves to be encoded, so where the text has for example <meta charset="UTF-8">, the QP version has <meta charset=3D"UTF-8">; 3D in hex (0x3D) = 61, and the equals sign is no. 61 in the ASCII character set. ASCII character no. 9 (0x09) is the tab character , so the tabs at the beginning of each of those continuation lines had been QP-encoded, presumably by AOL since they appeared neither in the 'good' AOL messages nor in the same message that hadn't passed through AOL.

I wonder, then whether some change in AOL's mail server software that 'fixed' these missing tab characters (which didn't need fixing as far as Windows Live Mail was concerned) also managed to foul up the line endings so that you were suddenly receiving CR-LF-SPACE instead of whatever you were receiving before the change.

Let's see whether things also improve for the other sufferers.
