d gittins say:

I finally have a logical answer on why Outlook performance tanks when I add a second account as an additional account in Outlook (piggy-backed onto a personal account).

The key issue is that Outlook can only run “Cached mode” for one account. (Cached mode means that all the email is initially downloaded to an .OST file (similar to a .PST file). and changes are made gradually, using few resources.)

When you add a second account, the main account runs in cached mode, and the second account runs in online mode.  In online mode, mail is not cached locally – everything is accessed in real time over the network. This is very slow if the second account is big.

Workable solutions:

One option is to have a second computer running so that you can run two Outlook accounts – not entirely practical though.

Another option is to have two profiles in Outlook – one for each Outlook account. That’s not terrific though, because you have to log off and on to switch back and forth.

The best solution that I’ve found is to run Outlook in a virtual machine that accesses the second account exclusively. Because the virtual Outlook runs in cached mode with an .OST, the performance is actually pretty good.

I then run Outlook from the host machine to access my own account.

I’m running the Outlook virtual machine in “XP mode” so that I only see the Outlook window instead of seeing the entire guest operating system with Outlook running inside a window. This is working very nicely, and it’s pretty cool to see two Outlook windows running side by side monitoring two completely different Exchange accounts!

Windows XP Mode (a form of Virtual PC specifically created for Windows 7) is available here:

http://www.microsoft.com/windows/virtual-pc/download.aspx

What is the complete error message, and what file is indicated as being corrupt?  Although it may in fact be advisable to run chkdsk, you should be aware that sometimes chkdsk can cause data loss.

It's spelled chkdsk.

To use it, click Start, then click Run.  In the "Open" box, type the following and then click OK

          chkdsk /f

A black Command Prompt window will open with the following message:

The type of the file system is NTFS.
Cannot lock current drive.

Chkdsk cannot run because the volume is in use by another
process.  Would you like to schedule this volume to be
checked the next time the system restarts? (Y/N)

Type Y and then reboot the computer.  Chkdsk will run after the reboot, before Windows starts.

The following information is from official XP documentation, but  take the comment about the length of time things may take with a grain of salt (it might be true for "millions of files" but not for a 70 GB drive):


Fixing disk errors

Chkdsk corrects disk errors only if you specify the /f command-line option. Chkdsk must be able to lock the drive to correct errors. Because repairs usually change a disk's file allocation table and sometimes cause a loss of data, chkdsk sends a confirmation message similar to the following:

10 lost allocation units found in 3 chains.

Convert lost chains to files?

If you press Y, Windows saves each lost chain in the root directory as a file with a name in the format Filennnn.chk. When chkdsk finishes, you can check these files to see if they contain any data you need. If you press N, Windows fixes the disk, but it does not save the contents of the lost allocation units.

If you do not use the /f command-line option, chkdsk sends a message if a file needs to be fixed, but it does not fix any errors.

If you use chkdsk /f on a very large disk (for example, 70 gigabytes) or a disk with a very large number of files (for example, millions of files), chkdsk might take a long time (for example, over several days) to complete. The computer is not available during this time because chkdsk does not relinquish control until it is finished.


P.S.  Even if chkdsk finds "lost allocation units" and converts them to files, I have never seen any useful data in the files created by chkdsk.
