How Do I Clone A Dual Boot XP Installation With Only Bootloader Files - Microsoft Community

max53000 say:

How Do I Clone A Dual Boot XP Installation With Only Bootloader Files on C:\

For me one of the most difficult things to do in presenting a problem in a text discussion is to clearly specify which drive and which partition I am talking about.  For those who need a refresher in Microsoft's boot loader nomenclature, here it is.  Others may wish to skip to the next paragraph.  The specification of any given partition is of the form multi(a)disk(b)rdisk(c)partition(d).  But for any ordinary computer, a=b=0, so we can forget about them for the sake of discussion.  The value c is the number of the drive in the boot order, with the boot drive having c=0.  The value d is the partition number with the first partition on the drive having d=1.

For the sake of my discussion below, I will be referring to rdisk(0)partition(1) as r0p1, rdisk(1)partition(3) as r1p3, etc to save space and hopefully maintain some clarity of thought.

My intention was to successfully clone my dual boot Windows XP Pro and Windows XP Home installation from a larger disk to a smaller disk.  Unfortunately Partition Wizard Home's hard drive cloning technique failed because of the fact that there is more data on the larger drive than will fit on the smaller drive.  To be specific, there is an extra data partition on the larger drive on F:\ r0p4.  So I used Partition Wizard Home to clone C:\ r0p1 (the system partition which only has the XP bootloader files), D:\ r0p2 (Windows XP Pro), and E:\ r0p3 (Windows XP Home) one at a time.  Because I am using a Promise TX2 PATA hard drive controller equipped with fixed bios settings, the only way to change the order in which the bios boots the disks is to change the disks data cabling. I made that as simple as possible by using both PATA masters.  Note that the partitions I cloned to were G:\ r1p1, H:\ r1p2, and I:\ r1p3.

To aid the reader in understanding the hard drive partition placement, I am repeating it below.  
C:\ r0p1 Windows Bootloader Files only (System, Active)    G:\ r1p1 Windows Bootloader Files only
D:\ r0p2 Windows XP Pro                                                  H:\ r1p2 Windows XP Pro Clone
E:\ r0p3 Windows XP Home (Boot)                                      I:\ r1p3 Windows XP Home Clone
F:\ r0p4 Data Partition

After finishing the cloning I made the following changes to the XP bootloaders on r0p1 and r1p1.  I added the capability to each boot.ini to boot r0p2,r0p3, r1p2, or r1p3 to eliminate having to change the cabling to investigate how the work was going.  I also created a bootable floppy bootloader with the same exact boot loader files in it, also.

I discovered that booting the original disk, I could successfully start all four XP's with either the r0p1 bootloader or the floppy bootloader.  But when I switched the disk order, only the floppy would boot into all four XP's.  The bootloader on r0p1 never appeared.

I take that to mean that a redirect command for where Windows is to look for the bootloader may be necessary.  I did boot from my slipstreamed Windows XP SP3 bootdisk and accessed the repair console.  First I ran bootcfg /list and to my surprise, there were no entries found.  I took that to mean that windows is not looking for the bootloader files in the right place.  Is it possible I need to redirect windows to their position?  I seem to remember reading something about redirecting the search for the bootloader files.  I also thought about running the command that rewrites the MBR but got a warning that my MBR was specially constructed and serious consequences might occur, so I quit before making an uninformed choice.

While I'm hoping for someone's kind offer of assistance, I will be backing up all the partitions on both disks using Acronis True Image.  Then if worst comes to worst, I can experiment with the recovery commands without fear of losing all my work up until now.

I finally discovered the two related errors I made that prevented me from being able to boot the cloned Windows XP operating systems using the bootloader files on the cloned C:\ partition.  Unfortunately, I had never taken the time to verify the truth of the "bootloader files being on the cloned C:\ partition."  In fact the original C:\ partition was a primary partition, while the so called "cloned" C:\ partition was a logical partition.  That fact made no difference to the floppy XP bootloader I had constructed, because it provided the bootloader files in a separate environment to that of the two hard drives.  But because the "cloned" C:\ drive partition was logical rather than primary, there was no way that I could make the "cloned" C:\ partiton active, because only a primary partition can be active.  And the process of making a primary partition active is critical precisely because it tells the computer where to find Windows XP's bootloader files.

In my case after I discovered that the "cloned" C:\ was logical, I used Partiton Wizard to convert it to primary.  But even then Windows XP couldn't find the XP bootloader.  It was only after I booted Partition Wizard using a floppy and marked the cloned C:\ primary partition as active that the bootloader finally appeared.

Even then I wasn't completely out of the woods.  Because I had left the original source of the cloned operating systems on the computer while troubleshooting the bootloader problem, Windows had retained the previous drive letter assignments after the hard drives boot positions were swapped.  Thus when I booted into what should have been Windows Home on drive E:\, I found out that Windows had retained the partitions previous letter I:\.  And the system drive had likewise changed from drive C:\ to drive G:\.

Finally, I believe it is time for me to confess why I have gone to all this trouble since I will be leaving both drives on the computer.  The 2 year-old 1 TB WDC Green drive is interfaced to the Promise TX2 PATA drive controller using an SATA to IDE converter, while the five-year old 300 GB Maxtor drive is natively PATA.  By using the freeware Disk Bench software with the windows swap file disabled, I have repeatedly timed 480 MB file creation on both hard drives on all the different partitions, and there appears to be a maximum 10 MB/sec speed advantage of the old Maxtor PATA drive over the converted SATA drive.  But the speed of creation for both ranges from 20 MB/sec to 65 MB/sec using the PATA drive, while the SATA drive ranges from about 20 MB/sec to 55 MB/sec.  It might be well to point out, however, that the WDC Green drive is ecologically friendly, but not a 7200 RPM drive.  I am planning on investigating whether the measurements obtained with Disk Bench with the boot drives swapped and the swap file disabled will make any difference in the time it takes for file creation.

This is an interesting article about file deletion and recovery of same:

Why Deleted Files Can Be Recovered, and How You Can Prevent It

When you delete a file, Windows removes the pointer and marks the sectors containing the file’s data as available. From the file system’s point of view, the file is no longer present on your hard drive and the sectors containing its data are considered free space.

However, until Windows actually writes new data over the sectors containing the contents of the file, the file is still recoverable. A file recovery program can scan a hard drive for these deleted files and restore them. If the file has been partially overwritten, the file recovery program can only recover part of the data.
