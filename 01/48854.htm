Instead of saying the time to click the clock, you can also say "system clock."

Jozefh, to answer your earlier question about WSR Macros, here's an example. Right now you're figuring out a sequence of things to say to get the desktop calendar to appear. What if you could create a single saying that would run through that sequence of sayings for you? That's what WSR Macros can do. Here's one:

<command priority="50">
    <listenFor>show calendar</listenFor>
    <emulateRecognition>switch to desktop</emulateRecognition>
    <waitFor seconds="0.25"/>
    <sendKeys>{TAB}</sendKeys>
    <waitFor seconds="1.0"/>
    <emulateRecognition>click system clock</emulateRecognition>
</command>

The above macro allows you to say "show calendar," and then the macro runs through the steps to make it happen. Basically it allows you to make your own speech commands.

Controller Not Working? -- Check Here! - Microsoft Community

mister cerb say:

Controller Not Working? -- Check Here!

How do I connect my controller?
If you are using a wireless controller, try to sync your controller to your console. You can doing this by turning on the console, and then pushing the sync button on both the controller (small circular button near the left bumper) and the console (on the original console, the small circular button by the power button; on the S console, the small square button near the front USB ports). The green lights should flash in a circle, then all flash, and then one light will remain indicating which controller position the controller currently is. 

If you are using a wired controller, try unplugging the controller for a few seconds and then firmly reinserting the USB connector. If it still isn’t working, try a different USB port as that particular port might be bad (if this is the case, you will want to use our Online Service Center to set up a console repair). 

My controller won’t connect (flashing green lights)
Does your controller turn on, but can’t seem to connect properly? Perhaps you already have four controllers connected. Are all four green lights around the power button on the console currently lit up? If so, you need to disconnect one of them. 

To disconnect a wireless controller, press and hold the guide button for a few seconds, then select “Turn off controller” from the menu. To disconnect a wireless headset, press and hold the power button until it turns off. To disconnect a wired controller… well, I’m sure you get the idea. 

My wireless controller won’t stay connected (“Please reconnect controller” error)
Before troubleshooting, make sure any objects between the controller and console are removed, such as doors, shelves, or metal dividers. Also make sure that other wireless devices are kept clear of this are, including wireless routers, cordless phones, or microwave ovens. Sorry, you’ll have to make the walk to the kitchen to zap your ramen! 

Also check to make sure your batteries are fresh. If you use rechargeable AAs or Microsoft brand rechargeable battery packs, make sure they are fully charged. You can try using a different battery source to see if there might be an issue with your current batteries. If you are having trouble charging your Microsoft rechargeable battery pack, please check this support article.

Still having issues with your controller?
If your controller still won’t work, try connecting it to a different console. If it works on this other console, there is probably an issue with your console. If the controller still doesn’t work on the new console, there is probably an issue with the controller itself. If it was recently purchased, try returning it to the retailer for a replacement. If you can’t take it back, but it’s still within the Accessory Warranty period, you can use the Xbox Online Service Center to order a new one. If it’s been longer than the Accessory Warranty period, then you would want to purchase a new controller at your retailer. For more information regarding accessory warranties, check out our Xbox 360 Warranty page.

lemp say:
Svchost.exe is just a "host" for "services" (i.e., programs that run in the background and are often, but not always, *.dll files).

If you have XP Pro, open a Command Prompt window (Start > Run > cmd > OK) and type

    tasklist /svc  {press Enter}

You'll get something like this --> http://winhelp2002.mvps.org/tasklist.jpg

You can look up the services in many places; here's a helpful site --> http://www.blackviper.com/service-configurations/black-vipers-windows-xp-x86-32-bit-service-pack-3-service-configurations/
(the names that are output from the tasklist command are the "Registry service name" (second column) on Black Viper's lists.

If you have XP Home, the tasklist command is not available.  You'll have to use a third-party tool like Process Explorer or Process Monitor.  Both of those are powerful tools and have bit of a learning curve.

lemp say:
  • Open Task Manager (CTRL+ALT+DEL) and click the Processes tab. 
  • If there is no column headed "PID," click View > Select Columns and make sure that there is a check in the box for PID (Process Identifier).
  • Click on the header of the "CPU" column to sort.  You may have to click twice to get the big users at the top.
  • Make sure that the box at the bottom to "Show processes for all users" is checked.

Make a note of the name(s) and PIDs of the processes that are taking a substantial percentage of CPU cycles.  Do not forget that "System Idle Process" does not count!  System Idle Process is a very simple task run by the CPU to make the total of all processes add up to 100%.  Thus, if you are not using very much of the CPU's capacity, "System Idle Process" will have a high number, like 97, 98, or 99%.  On the other hand, if there is an instance of svchost.exe, for example, that is using 55% of the CPU, "System Idle Process" can be no higher than 45% and likely will be a bit lower (because there are other processes running).


There will be several instances of svchost.exe running.  This is normal.  If you notice that one of them seems to be taking a large percentage of the CPU (and does so for more than a brief moment) make a note of its PID.


Then run tasklist /svc as described earlier.  Match the PID of the instance that's using the CPU to the instance with the same PID in the list of tasks output by tasklist and you can then see what services are included in that instance of svchost.exe.


For example, say that Task Manager shows that an instance of svchost.exe with PID 940 is taking 95% of your CPU.  You run tasklist /svc and get the output as shown here --> http://winhelp2002.mvps.org/tasklist.jpg  Examining that output, you see that the process with PID 940 is running 13 separate services.  One of those is wuauserv.  If you then go to http://www.blackviper.com/service-configurations/black-vipers-windows-xp-x86-32-bit-service-pack-3-service-configurations/  you'll learn that wuauserv is the name of the Automatic Update service ... supporting your suspicion that Automatic Update may be part of the problem (of course, your work isn't over yet ... it could be another one of the 13 services running in that instance of svchost.exe ... and once you figure out which service is the problem, you then have to figure out why.


If you don't have XP Pro (or even if you do and choose not to use tasklist), Process Explorer and Process Monitor will give you a great deal more information, but using Process Explorer and Process Monitor is beyond what I can get into in this forum.  You'll have to read up on it yourself.
