tryx3 say:

1 Check that you do not have any PC maker's power management utility running [this is likely to apply to Samsung & Sony Netbooks] by checking the PC manual & the PC itself. Samsung Fast Boot / Fast Start stops users changing the lid close action so I don't think [from your description] that you have this complication].

2 If the settings you put in & saved are not shown correctly in the power plan advanced options when you look at them again later on [as your description seems to confirm] then look at this Fixit first

When you configure power options for the active power plan in Windows Vista or in Windows 7, the changes are not reflected in the Power Options item in Control Panel

3 If you still have the problem then run the power troubleshooter

http://windows.microsoft.com/en-US/windows7/Open-the-Power-troubleshooter

4 Then run both the Maintenance & Performance troubleshooters at Control panel, Troubleshooting.

5 Check the entries in Device manager for any warnings, then try to update drivers - particularly the video card driver as that accounts for many power management problems.

6 Further to the check at item 2 above, if the Power plan Advanced options are showing the correct do nothing action in lid closing but the system is not actually doing that then run powercfg to see what the system believes its power settings actually are. The output file is quite long but you'll find it easy to follow through [I pasted the results text file into Excel to make it easier to work with]. I think the output file contains a sufficient description for you to see what's set up but I've included links to full descriptions below.

In a command prompt, enter

powercfg /qh >c:\users\YourUserName\Desktop\PowerSettings.txt

Note A You can see the almost-full syntax using powercfg /? but, for example, the h in qh is not listed [/q gets you the properties, /qh gets the the full list of properties including hidden properties].

Note B Each power property is described in PMPolicy_Windows.docx, see Power Policy Configuration and Deployment in Windows See also Power Schemes (Windows) for further background info.

[Solved] W10 1803 and 1809 inbox and Intel drivers introduced - Microsoft Community

jmichael2497 say:

[Solved] W10 1803 and 1809 inbox and Intel drivers introduced performance issues such as noticeable audio stutter

[Solved] W10 1803 and 1809 inbox and Intel drivers introduced performance issues such as noticeable audio stutter due to older Intel storage controller (in RAID aka IRRT mode) not being properly supported and solved by manual install of older driver (update edit: or by switching out of IRRT mode to AHCI if you aren't really using those special RAID features).

ignore all the lazy and ignorant directions to clean install, since that will not fix an obvious system level driver issue, (as verified by all the unfortunate folks who went through the hassle of a clean install). for folks stubbornly skeptical like I was, that brush this off because "newer drivers are always better, so older drivers can't be the solution"... well... in this case they are.

Symptoms: after updating from 1709 to 1803 my system performance degraded with the most obvious thing being a strange audio stutter every few seconds (even with usb speakers) with streamed and locally stored content, that I did not realize was there before deleting the prior version of windows, (oops run a 3d game to test everything next time), and 1809 did not fix it automatically either.

Driver Research: by manually searching for "compatible ids" in https://www.catalog.update.microsoft.com/home.aspx got everything updated to final official drivers (often newer versions than windows update for some reason) and still audio stutters were happening.

Further Reading: after seeing some mention of drive related issues and windows timers I wasn't really convinced in any particular direction. After one more post about Intel storage drivers dropping proper support for older chipsets from newer versions of windows inbox drivers or direct Intel bundles... I saw they mentioned ICH10 chipset and using a much older v11 driver fixed the stutter issue, so time to give it a try.

WARNING: changing storage controller drivers could potentially lead to boot problems, so being able to use recovery mode, to rollback to a restore point created before you start changing the driver, is essential. I have a habit of cleaning out drivers from safe mode (not really necessary), and I found I had to restart back into normal mode after the storage driver change, before being able to start safe mode again. I used the restore point a few times before noticing that (maybe unnecessarily, but it seemed pretty well hung in normal mode... so best to be safe).

Final Troubleshooting: after uninstalling newest catalog v16.7.5, was back to inbox v15.44.x drivers with audio stutter, tried latest versions of each series until I found one that worked (v15.9.0, v14.8.16, and finally v13.6.0 worked). Then going back up to test sub-versions in between until it stopped working (v14.0.0 and v14.8.0 and stopped working on v14.8.16 again). Yes it is time consuming (but that is the scientific method) thank wintel for that.
