External Hard drive - Remove write protection - Microsoft Community

bayswater say:

External Hard drive - Remove write protection

Somehow I have write protected my external hard drive. How do I remove it? Thank in advance

carey frisch say:

1. Click Start

2. Type regedit and click OK.

3. Save your registry settings by clicking file > Export > save. This is the bacup taken in case something messes up.

4. Open HKEY_LOCAL_MACHINE by clicking on the + sign next to it.

5. Open SYSTEM by clicking the plus sign next to it.

6. Open CurrentControlSet.

7. Open Control.

8. Open StorageDevicePolicies

9. Right click on WriteProtect and click modify.

10. If the value data is 1 change it 0 and click ok.

11. Repeat steps 7 to 10 for ControlSet001 and ControlSet002 .

12. Close Registry editor and its done. You might have to reboot.

REG QUERY on Windows XP - Microsoft Community

matt d-7613 say:

REG QUERY on Windows XP

So I by chance discovered that I almost did "bad things" to my Windows XP PC's during a software upgrade/ installation.

As an example - Microsoft Silverlight needs to be updated.

I first use WMIC to uninstall currently installed software. 

(wmic product where "name like '%%Microsoft Silverlight%%'" call uninstall)

Then I query HKEY_CLASSES_root\installer\products and send the results to a text file.  I then query the results to look for Silverlight.  If it finds it, it deletes the key.  

(
REG QUERY HKEY_CLASSES_ROOT\Installer\products  > silver.txt
FOR /F "Tokens=1-5 DELIMS=^" %%i IN (silver.txt) DO (
Set RegKey=%%i
CALL :START)

GOTO :OS-CHECK

:START
REG QUERY %RegKey% /s | Find "Microsoft Silverlight"
IF NOT %ERRORLEVEL%==0 GOTO :AlmostDone

:DELETE KEY
REG DELETE %RegKey% /f

)

I repeat this in a few other registry locations to be sure that all old installs of Silverlight (or whatever software I am installing) are removed.

I came across an interesting discovery this week when updating Silverlight.    The installation was corrupted (because I forgot to unlock the file I downloaded) so when WMIC ran, it did not find any software to remove, thus leaving the key in HKEY_CLASSES_root\installer\products.  I kept noticing that all of the subkeys were being deleted, but could never recreate it, until this morning.  

Then I realized my critical error.  

As you can see in the picture below, when I run REG QUERY  HKEY_CLASSES_root\installer\products on Windows XP, the results I get not only include "!REG.EXE VERSION 3.0", but also HKEY_CLASSES_root\installer\products.  This does not happen on Windows 7.  

So what was happening was that the REQ QUERY would run against HKEY_CLASSES_root\installer\products and find Microsoft Silverlight and then promptly delete every sub key under HKEY_CLASSES_root\installer\products.  


So the actual questions here are this:

1.  Any way around this on Windows XP?
2.  Is there a better way to accomplish what I want?

Thanks in advance.

- TDR was introduced with Win7 and was meant to prevent premature BSOD when a game lags. Unfortunately it is a mostly

failed concept, and can be as difficult to troubleshoot as BSOD.

The TDR error always blames the driver, but in reality the driver is more often just the last link in a chain of events that led to the TDR.

[EDIT: Well apparently Intel has changed that theory recently :\ ]

The real source of the problem can be either hardware faults or software conflicts.

- As per your other thread -  dxdiag reports are next to useless, Just post your system specs. 

WindowsEvent Viewer > Application and System reports can be useful at times .

- If the TDR has created a mini-dump file you may want to post that as well. They also tend to blame the gpu drivers,

but can occasionally be more on target.

BSOD Finding and fixing them - Microsoft Community

Hardware:
- RAM. Run  memtest to verify the RAM is not producing errors
- HDD. Run  chkdsk to test the HDD for errors
- Run a temperature monitor on the cpu and graphics card to verify they are not overheating under full load.

GPU-Z Video card GPU Information Utility    has sensors for  temps as well as a number of other gpu parameters.

Real Temp   is a good cpu temp option.

- Inadequate, or faulting power supply  is a somewhat common cause of 'random' reboot,

but more often when under load from gaming or similar system stressing software.

Some examples of software causes -
- When TDR occurs when using the internet it is sometimes due to browser add-ons, or the
- Anti-Virus software, particularly AV suites that run a number of programs in the background
are another possibility. Running more than one AV program in resident (always on).
- Malware, Spyware or Virus are possible. Run a full boot-time scan with your AV.
A virus called iehighutil.dll  that come as an added bonus from some torrent downloads is one example.
- System 'optimising' software is another possibility.
- Games often cause TDR. In some cases this is due to using too high graphics settings.
- Are your BIOS settings at default, or have you used some manual settings,

[EDIT another uncommon BIOS issue discovered last week is Aggressive Link Power Management. Another poster found

disabling ALPM fixed his crashing problem. ]

- Have you checked for a BIOS update for your motherboard
- Did you install anything around the time the crashes started ?
- Conflicts with other drivers [ Intel ? ] can also cause TDR
One such driver comes with Daemon Tools.

You may find something by running in Clean Boot to troubleshoot Services and Startup programs.
Having too many Startup programs running in the background can stress the system and cause lag

- Or, how about this long shot... An unusual power related issue I'd never heard of before this week is the BIOS setting

Aggressive Link Power Management.

Someone who was having a problem with his gpu crashing wasn't having any luck and resorted to trawling his BIOS and found that

ALPM was enabled by default on his motherboard, and disabling it solved his problem.

I think his issue was only when running Overwatch.

Another one of those irregular gaming issues. As it works out I have one mobo that defaults enabled, and another that defaults disabled,

and I've never had a problem with it enabled.

Enabled shouldn't really be a problem for anyone as all ALPM does is lower the power to SSD/HDD when the machine is at idle.

( Maybe it's more of a problem if you use Sleep or Hibernate with ALPM enabled ? )
