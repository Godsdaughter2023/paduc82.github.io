irwin2010 say: A quick search of C:\Program Files and C:\Windows and one installed application relied on an uninstaller .lnk file which was a shortcut back to its installer (presumably with an uninstall command line arg.), also a handful of powershell related progs ran in a similar way.  The start menu search box proved to still be available to run programs after disabling .lnk files (start menu items are all .lnk files, however I figured there would be launcher replacements
that would work otherwise and not act as a replacement for the command shell , e.g., possbily http://lee-soft.com/vistart/ ).

The initial key I thought might disable .lnk files which list executable file types (.exe, .cmd, .lnk, .msi):

HKEY_CLASSES_ROOT\CLSID\{1d27f844-3a1f-4410-85ac-14651078412d}\SupportedTypes

Deleting this key results in an 'Unable to delete all specified values'  error from regedit (even running administrator rights), and from the command line gave an 'ERROR: Access is denied' message.

I then did the same for the following keys (ref. http://windows7forums.com/windows-7-support/71332-cannot-create-shortcuts-desktop.html ):

[HKEY_CLASSES_ROOT\.lnk]

[HKEY_CLASSES_ROOT\lnkfile]

Which duly resulted in the same errors as above -- BUT, much to my amazement when I rebooted the .lnk files were deactivated.

To conclude anyway, the idea of deactivating .lnk files from executing is not particularly satisactory as it destabalises the operating system, though it could probably by the looks be made to work.

It might be worth investigating if some of the public access computer kiosk type programs are able to get around this - and if so how.  There might be some other firewall tpe programs similarly also worth looking at (e.g., http://forums.mydigitallife.info/threads/29837-how-to-disable-running-exe-on-usb-drive ).

Apart from which leaves the Windows SteadyState (now discontinued project) approach which was largely (at a quick glance at least) based on returning a user login to a preset state on logout - Fortres Grand' Clean State will do this for home Windows users http://www.fortresgrand.com/products/cls/cls.htm  There are additional measures that can be taken also (e.g. parental contols, built into Windows 7, see previous para. re. Vista, XP; moving user folders to a separate partition and disabling access to C:, etc), which would help limit the damage a hacker could get up to, but not completely render the problem solved.

It appears then it is not so much myth that Apple Macs are more secure computers than Windows, Unix in the broader sense does not have this weakness as outlined above, so presumably Macs also (I have never owned one).  Having said that Windows 8 from what I can gather does not have shortcuts as a feature on its Metro desktop, and together with its ability to use UEFI secure boot enabled hardware (as Chromebooks do from what I can gather), then Windows could prove (though it remains to be seen) more secure than Apple Macs.

irwin2010 say: Quick update on this.

Fortres101 from http://www.fortresgrand.com/ will actually prevent .lnk files from running (No Access Files, Folders and Executables setting using a wildcard), however all the caveats of this approach still apply, and preventing access to the AppData folder using this method would also involve moving the user folders to another partition, rendering C: not accessible via the Windows shell (a registry setting), and using some sort of parental control to prevent the installation or running otherwise or at least ensuring the sandboxing of any programs that might bypass the Windows shell.  (There are also some side effects from Fortres101 to be expected - e.g., I found for some reason every time I ran Firefox it prompted me to set it as the default browser).

Another approach, and the penny only dropped when I found myself reading the Sandboxie http://www.sandboxie.com/ tagline, 'trust no program' - auto sandbox any program that has command line arguments that might allow the AppData folder contents to be written to.  This is a tall order, it's not possible to just sandbox the contents of C:\Windows in its entirety (nor the program files folder).  Some utility software I am sure would do the trick though.


irwin2010 say: Some more on this topic.

Via Micro Mart (issue 1219), an unofficial gpedit.msc installer (note MM does give a disclaimer with this info., at your own risk), http://fc02.deviantart.net/fs71/f/2011/180/b/0/add_gpedit_msc_by_jwils876-d3kh6vm.zip

Which can then be combined with this technique http://www.mechbgon.com/srp/index.html#setup to disable .lnk files.

I haven't tried it, but would be surprised if you could get full gpedit functionality on Windows Home or Premium.  If it is functional though there may be additional functionality in gpedit that would solve the problem some other way (not being a Pro. user - as most people - I'm not familiar with this particular admin. layer).

The sandboxie approach looks promising and would not entail the loss of shortcuts, ref. this thread on the subject of writing a quick util that would output a sandboxie config file while also setting parental controls to prevent an external program from further accessing the AppData folder.
