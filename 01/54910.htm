anamon say:

Administrator takes ownership from TrustedInstaller and grants full access rights, but still cannot modify files

This is a problem that, from a search here and elsewhere on the web, numerous people are confronted with, but I have yet to find an acceptable solution. Please read the question entirely before replying!

The scenario:
I had a previous installation of Windows 7 Pro 64-bit on a hard disk drive. As part of a recent computer upgrade, I decided to add an SSD to my system, on which to install Windows and certain software I use often. Because I was pressed for time, I chose to not go through an involved system backup/transfer procedure, but simply keep the old hard disk with the previous system around as an additional drive, in case I ever had to transfer something.

The problem:
The old hard drive still contains a complete Windows installation that is now out of service. Since Windows' own system files are not the reason why I keep the old drive around, I would like to get rid of them (close to 30GB in the Windows folder alone) to make room for additional file storage. For most of the drive, I had no problem granting the users of my new Windows installation access: give ownership to the new admin account, and give that account full access rights. I can then modify and delete files at will. As it seems this works for everything except for certain Windows system files/folders that used to be owned by the TrustedInstaller service. (Examples are the ehome subdirectory or certain fonts.)

There is seemingly no problem in changing ownership of these files and folders from the TrustedInstaller to the admin account of my new system. Also, giving the admin full access rights goes over smoothly. However, the admin still cannot exercise these new rights. TrustedInstaller is no longer even mentioned as either owner or a user with access rights. But the entire effect was that while previously, the message said that TrustedInstaller denied the admin access to the files (which is correct and makes sense), it now says that the admin denies themselves access to the files (which does make no sense whatsoever).

The solutions:
The only "solutions" I have read about so far are formatting the partition, or using a Linux boot disc to delete the files. Both of these I consider workarounds, not solutions. I'd like to prevent having to go through all the files on the partition (in order to choose what to back up) before formatting, because the time that takes is exactly the reason why I kept the drive around as-is. Also, I don't yet trust Linux' NTFS support 100%.

The plea:
To prevent the standard responses. Please don't ask why I want to gain these access rights—it's explained above, even though it is off-topic. I'm not asking whether I should delete the files, I'm asking how I can delete them. I'm not stupid and I don't need protecting from myself. The drive is no longer a system drive. The files still somehow protected by TrustedInstaller are protected for no reason because they're decommissioned. The entire system and user configuration they were created under no longer exists. For the same reason, please don't tell me that the files are protected for a reason and that I should not mess with them. And finally, I have taken ownership and given myself complete access rights to the files. This would be the standard, sensible solution, and seems to work fine everywhere except for files owned previously by TrustedInstaller. Windows says my account has all the rights—I just can't execute them.

Sorry if I come off as a bit of a wise guy, but I'd just like to know whether there is a solution to this problem, rather than getting the usual "it doesn't work for a reason".
anamon say: This is what turned out to be the problem—I'm not entirely sure how it happened though, but to anyone who experiences the same problem, this might be something worth checking out.

Giving ownership to the local administrator worked as expected (which is why the message changed to telling me that I need permission from the local administrator, which I am). The problem was with the access rights, even though I did not forget about giving local administrators full read/write access. For whatever reason though, they had full access rights twice, which somehow seems to negate their effect. In particular, this is how the table in Properties » Security » Advanced » Permissions looked like:

Allow | Administrators (Daniel-1\Administrators) | Full access | <not inherited>
Allow | Administrators (Daniel-1\Administrators) | Full access | <not inherited>

rather than just having one of those lines. Apparently, being given full access rights twice, Daniel-1\Administrators reverts back to read-only rights. As I said I don't know how it happened, since manually trying to add a second, identical rule is impossible if one already exists. I suppose it has something to do with the special case of taking ownership from TrustedInstaller, since that's the only thing "special" about these folders. Also, it doesn't show up as two entries in the Security tab of the Properties dialogue, so you have to open the Advanced window to see it.

The solution: Deleting one of the duplicate entries restores access rights to the user(s) in question.

It turned out that every folder I had trouble with had these duplicate access rights entries, and I was able to modify and delete them at will after removing them.

This sounds as if you (a) have an old vinyl plotter that has a serial interface and (b) at some time purchased a USB to serial adapter or cable. 

If the above isn't correct, what is the make and model of your plotter?
