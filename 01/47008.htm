FIX : Windows 10 100 Percent Disk Usage Problem

1. Press W8K + R and type devmgmt.msc in the Run, hit Enter key to open Device Manager snap-in.

2. In the Device Manager window, expand IDE ATA/ATAPI controllers and right click on Standard SATA AHCI Controller, select Properties.



3. In the property sheet, switch to Driver tab and click Driver Details.



4. If the Driver File Details window shows storahci.sys driver listed there, it means you’re running inbox driver and hence it is possible to disable MSI mode. Click OK.



5. Back on the property sheet, switch to Details tab. Against Property, select Device instance path from drop-down and note down the Value mentioned there.



Registry Disclaimer: The further steps will involve registry manipulation. Making mistakes while manipulating registry could affect your system adversely. So be careful while editing registry entries and create a System Restore point first.

6. Press W8K + R and put regedit in Run dialog box to open Registry Editor (if you’re not familiar with Registry Editor, then click here). Click OK.

7. In the Registry Editor window, navigate to following registry key:

HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Enum\PCI\<AHCI Controller Value>\Device Parameters\Interrupt Management\MessageSignaledInterruptProperties

*Substitute <AHCI Controller Value> we got in step 5.



8. In the right pane of MessageSignaledInterruptProperties key, look for the registry DWORD (REG_DWORD) named MSISupported which should be corresponding to Value 1 as its Data. Double click on the DWORD to modify its value:



9. Finally, set the Value data as 0 and click OK. Close Registry Editor and reboot. After restarting your machine, the 100% disk usage problem should disappear.

timstone say:

My apology for not responding sooner.  

Sadly we probably won't get any answers from MSFT.  There are many posts where people may find something to temporarily provide a fix, but then it fails again.

Regarding the ADS issue, the problem is when using "discovery" for the client to find the server.  We found that 1803 took away firewall port access and we had to reset those ( 6262 UDP ).  The problem still persisted to do the initial connection but at least the software did work properly once we established communication.

We did find a reliable solution.  Set the server computer to a STATIC IP.  Then create an ADS.INI file on each workstation ( check the manual ) to specify that IP and PORT ( 6262 ).  Without the need for "discovery", the client will immediately connect and work as it should.

One other note.  ADS uses UDP port 2989 for it's discovery of the server.  It's a good idea to open that on all firewalls, but that may not solve your problem.

Be sure to check all of your networking settings.  1803 set them all back to default ( mostly OFF ) for network discovery processes including the Services that should run automatically.

With all of that said, the issue may also be related to Defender.  Even without the Firewall running, people still had problems.  Some people found that using AVG instead of Defender allowed everything to work properly, but of course that is not really a solution.  It may provide a temporary band aid but it does not actually get to the root of the problem and allow a fix.

Finally, from posts I have seen, this affects a lot of networking software including some very large systems.
