lemp say:

You have multiple issues.  Determining whether they interact with each other, are related to each other, or caused by the same thing(s) can probably not be dealt with using an online forum like this.

Here are some comments, in no particular order.

1.  Your question 4 suggests that you have installed several antivirus programs.  You should have only one of these programs installed and active.  In my opinion, you should uninstall and not reinstall both Norton and McAfee.  I don't know enough about Webroot to comment.

2.  You should not manually terminate svchost.exe processes and expect your system to run normally and/or without odd behaviors occurring thereafter.  The svchost.exe "hosts" from one to many services implement as *.dll files.  Some of these may be critical to proper Windows behavior.  At the very least, before you terminate a svchost.exe process, right-click on it in Task Manager and select "Go to service(s)" where you can see the services being hosted by that particular instance of svchost.exe.  Then go to a site such as http://www.blackviper.com/service-configurations/ and read about the services that you are about to stop.

3.  My first reaction on reading your statement that the "check disc operation takes roughly two and a half days" was to conclude that you have a failing hard drive and to suggest that you download and run the diagnostic utility provided by the drive manufacturer (e.g., Western Digital, Seagate, etc.). 

However, after reading your discovery that the time for a disk check dropped to a reasonable number when you aren't connected to the Internet, the obvious conclusion is that you have another process or processes accessing the disk at the same time that chkdsk.exe is attempting to perform its checks.

Because the point of running chkdsk after a restart is to permit chkdsk to have access to the partition before Windows permits other processes to access the file system, my guess is that the only processes that could be competing with chkdsk for access to the file system at this point would either be malware (e.g., a rootkit) or something similar (e.g., Norton or McAfee antivirus).  

[The following is incomplete -- in the Recovery Environment, the Network Adapters are by default not enabled.  I don't know if it's possible to enable a NIC in the RE.  If you want to try this experiment, you'll have to dig deeper than I have at the moment. ]

What happens if you run chkdsk from the Command Prompt in the Recovery Environment rather than from an ordinary reboot (with your router connected and turned on)?  See http://www.bleepingcomputer.com/tutorials/windows-7-recovery-environment-command-prompt/

Note that when you open a Command Prompt in the Recovery Environment, the drive letters very likely will not correspond to the drive letters in normal operation.  That is, before you run chkdsk in the Recovery Environment, you need to determine your Windows system partition.  See the text below Fig. 8 in the article linked above.  Don't, however, change to that drive.  Just run chkdsk from the X: drive prompt.  For example, if your Windows system partition turns out to be F: (as it is on my system), you would run

 x:/>chkdsk F: /F 

(some sources say that the /F switch is not available in the Recovery Environment and to use /P; YMMV)

or

x:/> chkdsk F: /R
