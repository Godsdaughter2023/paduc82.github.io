¡firedog say:

Without seeing the message source of the involved messages, I can't be certain what's happening. But since attachments to new messages are available at the receiving end while those in replies don't, I'll have to guess that you're sending in a different format in the two cases. For example, new messages are sent using the contacts data for the addressee, but replies simply go to the From: or Reply-To: address regardless of any settings in the contacts data for this particular addressee.

If the message is sent using Outlook's proprietary RTF format - which might well be your default - then attachments will be TNEF-encoded. This encoding is not understood by any mail client other than Outlook (or Exchange Client). Some mail services can decode it, notably Hotmail and Gmail. This format should only be used in the corporate environment, and only if the sender knows that the addressee is using Outlook. Otherwise, plain text or HTML should be used.

I wasn't aware that Outlook 2013 still uses this format, but these notes should explain what is happening and may help you find a way around it, like changing the default sending format to HTML and specifying RTF for corporate contacts and others you know to be using Outlook. It's only necessary when you're taking advantage of the interactive capabilities it offers, like meeting accepts and voting buttons.

¡firedog say:
Yes, that sounds like exactly the same problem. Ask your landlord to read the article I referred to.

Because mail servers only understand plain text characters, any file (other than a plain text file, of course) has to be converted into plain text for transmission as an attachment. There are different ways of doing this, and all mail clients can deal with the commonest varieties. Outlook uses its own encoding/decoding system so that it can do more than just transmit files, like including an automatic 'accept' button with a meeting request. The same file will consist of completely different streams of characters if it's prepared for transmission by Outlook using TNEF encoding and not using TNEF. As soon as Windows Live Mail sees the string Content-Type: application/tnef, it closes its eyes and pretends the attachment just isn't there.

Imagine sending a message in code from the front line back to headquarters. If you're using a different code book from the one they have at the other end, they won't be able to understand your message, even though the message itself hasn't changed at all.
