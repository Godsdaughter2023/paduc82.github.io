trevor j lawrence say:

A local account is the type of logon that existed in Windows 7 and earlier.  It consists of a username and password and is used for authenticating (signing on) to that particular computer only.  It has no other purpose.


In Windows 8 you could do things that way, or you had the alternative of using a Microsoft account to logon.  Since the Microsoft account gets you access to a bunch of other cloud resources, logging on with the Microsoft account gets you access to all these Services automatically.  Also, since the account exists in its own right outside your PC, you can use the same logon to access other machines. (All this means that the Microsoft account needs a stronger password than a local account.)


I suspect from the wording you used in your question that you are using a local account.


What your tech support most likely did was not to disable the password, but rather configure the machine to automatically logon when it was powered up.  Thus, although you do not have to enter a username and password, it is being done for you under the hood.


Things are slightly different with Windows 8.1.  You are even more strongly nudged towards using a Microsoft account to logon.  Some thongs will not work without it.


You CAN still configure the machine to not ask for a username and password, just as you have it now configured.


You say that you have no use for apps (by which I take it that you mean "Modern apps" rather than desktop apps.  If you "go to a website to get what you want" it can only be desktop apps), in which case you don't "need" the Store, but do you use or even might you consider using SkyDrive.  If that is so you must logon with a Microsoft account in 8.1 as it currently stands.


If you currently have anything stored on SkyDrive you must logon with the MS account under which that data is stored.


My recommendation is to use a Microsoft account, but get your tech support to hide it as before.

So I renamed my private shares on my NAS devices to correspond to my windows username.  I also mapped it using the ip address\share instead of the device name\share - I read that somewhere using the device name screws things up.  Things seem to work now across different windows devices.  I had to set static ip addresses in my router for the nas devices.
 
Not sure why the newly changed to public shares would request username / password - you could try disconnecting the original mapping and remapping after removing password protection.
