But what is the principal difference between the two options of creating a system backup?

rdrewd2 say:

Option 1 won't create a "system backup"; it will create a  recovery media that you can use to repair/reinstall a clean version of Windows 10.   This will be a bootable device.  It will wipe the drive on to which it is installed and write to it only the files needed to reload Windows 10.  Nothing about your current system as it stands now is included.

 

Option 2 creates a "system image" that you can use to restore your PC to it's current state.  It creates an exact copy of your hard drive (including all files, programs, etc.) that can be reloaded in the event of a catastrophe.  This method creates a file called "Windows Image Backup" on to the selected drive and leaves everything else on there alone.  This will not be a bootable file, but it will give you the option to create a system repair disk at the end of the procedure (bootable) that will allow you to install the saved system image on to your hard drive.

 

I would recommend doing both; use an 8GB USB stick to create a recovery USB with option 1, and use an external USB-connected hard drive that has at least as much free space as your C: drive currently occupies to create a system image with option 2. Also, create a repair disk at the end of that procedure (by just clicking "yes" to the option it gives you).  If you're looking to create a full system backup with all your files and programs intact, the method you're referring to as the "2nd option" is the way you want to go.

A Windows 10 client accessing a SMB3 file share will in some cases use - Microsoft Community

laurent vonau say:

A Windows 10 client accessing a SMB3 file share will in some cases use a WinRM connection (TCP/5985)

Hi everyone,

Here is a very strange behaviour that happens when from Windows 10 (1607), you try to access a file share with no rights.

For instance, let suppose you share the TEMP folder on a Windows 10 machine (CL99156) with "Full Control" to everyone.

Then, into this folder, you create a "no_access" folder. All accesses on this folder a removed from the security tab (NTFS ACLs), only allowed for instance to a local admin user of the file server.

If you try to access this folder from another Windows 10 machine, for instance by clicking on \\CL99156\TEMP\no_access link, you will first get this Message Box for 15 seconds.

Only 15 seconds later, you are notified that the network folder access is forbidden with the correct "Network Error" message.

Have a look beyond the hood with Wireshark. The trace below shows what happen. It looks like SMB2 protocol informs the client that the resource "no_access" is denied, that is correct. But in parallel we have 3 attenpts to connect to the TCP/5985 port of the server. But the TCP packet is bloqued by the firewall (paquets are dropped).

By disabling the firewall, the "Network Error" message is immediately displayed to the Windows 10 client. A look on the network capture shows that the client tries to connect to the TCP/5985 port but the server immediately responds with a "TCP Reset" packet.

It looks like an issue in the SMB3 implementation of the Windows 10 (1607).

This issue is only reproductible on Windows 10 (1607) clients talking to servers with SBM3 dialect (3.0.2 and 3.1.1). The issue is not present on Windows 10 clients talking SMB2.x dialect.

To work around this problem, you must allow clients to access TCP/5985 port. Or maybe (not tested) implement a REJECT rule on the firewall that will reset the TCP socket initiated by the client.

Problem will also appear on FileShares hosted by Windows 2012 or 2016 server protected by a firewall that will drop TCP/5985 packets.

I did not see any TCP/5985 port required for using SMB3. Is this problem known by Microsoft?

Thanks for your feedback
