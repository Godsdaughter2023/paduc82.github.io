petercon [msft] say:

Sorry, I missed that that was your site.

It's important that the way characters are encoded in the HTML file match the encoding declaration given in the page (or, in the absence of a declaration within the page, reported by the HTTP server in the HTTP header).

If I look at this page,

http://books.richlabonte.net/

I see the meta element declaring charset="windows-1252". And I see the copyright sign on the page. In this case, the copyright sign is encoded using the windows-1252 encoding — as the single byte 0xA9. The encoding of the character matches the charset declaration, so it displays correctly.

On the other page mentioned earlier, the copyright sign was encoded using UTF-8 — a byte sequence 0xC2 0xA9. But without a UTF-8 declaration, the page was being interpreted assuming windows-1252. Because of the encoding assumed, the two bytes are interpreted according to code page 1252, which results in "Â©" being displayed.

If pages are working, there's no need to go changing the charset declaration, unless it would help you in maintaining them to have everything consistent. But if you change a declaration from windows-1252 to utf-8, then the encoded representation of the copyright sign (and possibly other special characters) may need to be changed to match. If you use an HTML editor such as Dreamweaver, it ought to be taking care of such things (but evidently something got missed on that one page).

If you have only a handful of special characters you use on your pages, another alternative to avoid encoding glitches would be to used named character entities, or numeric character entity references. If you're not familiar with these, here's a good starter reference:

http://www.w3schools.com/html/html_entities.asp

This W3C reference page gives a comprehensive list of named character entities:

http://dev.w3.org/html5/html-author/charref

The numeric references can be used to represent any Unicode character; the numbers are always the Unicode code point, or "Unicode scalar value" (which is not the same as the UTF-8 byte sequence), and can be expressed in either decimal or hexadecimal. (Hex is more convenient since Unicode scalar values are always cited as U+xxxx using four to six hex digits.)

So, for the copyright sign, you could use one of these:

&copy;

&#169;

&#xA9;

These will always work regardless of what charset is used to encode the page.

I hope that all makes sense and is helpful.
