cmdrkeene say:
UAC presents itself on an alternative desktop where no application can interfere with it.  Scripts can be written (and in fact I use such all the time at work) that manipulate the keyboard and mouse directly.  When I run some of my tools, for example, it literally moves the mouse pointer and clicks for me (I like to automate my work so I can be lazy).

If UAC didn't block this, viruses would easily be able to hit "Yes" (or just press the keyboard equivalent) to approve themselves and give themselves full access to your computer).  UAC therefore switches to another user account where none of your currently-running apps can even see it.  The first thing UAC does before making this switch is take a quick picture of your desktop as a trick to make it seem like you didn't really switch.  (Nerd trivia: it didn't always do this, but people didn't like their actual desktop seeming to disappear entirely during the switch).

...but you can turn this off.  If you go to UAC settings, you can drag the slider down one level to the "do not dim my desktop" setting.  With this set, you'll be able to screenshot the UAC window.  But remember that you're now less secure.

The error you quoted usually appears when you try to run a 16-bit executable on 64-bit Windows (that may be the game itself or only the installer for the game). In those cases (for a rule of thumb, most games released before Windows XP = before 2001 can offer this problem), your only way is using an emulator for DOS (www.dosbox.com, or D-Fend Reloaded, an easier to use frontend for this program). If the game is using a 16-bit installer but is a Windows - not DOS based - program nevertheless, you're out of luck.
