wunders say:

The obvious problem you are having is that Computer 1 does not authenticate connections from your other computers.  Important information missing from your post is 1) Whether Computer 1 is XP Home or XP Pro (you state the other machines are XP Pro) and 2) Whether Simple file sharing is enabled (or not) on Computer 1.

"How to disable Simple File Sharing and how to set permissions on a Shared Folder in Windows XP"

  < http://support.microsoft.com/kb/307874/en-us >

If Computer 1 is XP Pro and Simple File Sharing is disabled, then IIRC, the authentication sequence is approximately as listed below:

  1. If the user name on the connecting machine matches an existing username on [Computer 1] and the passwords of this user on both machines match, then access is immediately granted.
  2. If user names match but passwords don't, then access is immediately denied.
  3. If user names don't match, then an attempt is made to authenticate as the "Guest" user with no password.  If this succeeds, then you connect.
  4. I believe that if the Guest user has a password, then access is denied
  5. If Guest user is disabled [on Computer 1], then a logon screen pops up asking for your logon credentials.

If Simple file sharing is enabled (or XP Home) on Computer 1, then authentication is *forced* to be the Guest user (#3/#4 above).  If that fails, so does the connection.

Therefore, the first thing that I would check (assuming Simple File Sharing is enabled) is that the "Guest" User is enabled on Computer 1 and that it does not have a password.  The second thing I would check would be that the folder that is being shared has permissions that allow the Guest user to access it.

HTH,

Sorry for the delay...  I think you are correct in that the name resolution is having a problem.  One way to confirm this is to use the IP address of the machine instead of the machine's name.  If this works to connect to the other machine, then it's likely address resolution. Assuming Computer 3 has IP addres of 192.168.0.3, try the following syntax:

     net  use  \\192.168.0.3\sharename

One possible, but not common, cause of this is addressed in the following KB article:

  "You cannot view other workgroup computers on the network on a Windows XP-Based computer"

   < http://support.microsoft.com/kb/903267/en-us >

This conditions happens more often in laptops that move from one location to another and is caused by the DHCP Server which misconfigures the Node Type.  When you type the following in at a command prompt window:

    ipconfig  /all

Look at the top of the printout for the "Node Type" line.  It should be "Mixed", "Hybrid", or "Broadcast".

You say that the computers ping each other OK.  Instead of "Ping", try the Microsoft Networking equivalent of a ping "nbtstat".  Try this from Computer 2 to Computer 3 with the computer name (-a) and again with the IP Address (-A) and see if the response is different:

      nbtstat  -a  computer3            (no "\\" is necessary here)

      nbtstat  -A  192.168.0.3

If successful, it will print a few lines describing the Microsoft Networking parameters of the other machine.

wunders say:

You provided a very nice summary...

It seems that your Computer #2 and #4 are working just fine.  It also appears that Computers #1 and #3 are not allowing incoming connections for some reason.  This could be due to a configuration issue or a Firewall issue.  Some things you may be able to verify on Computers #1 and #3:

Bring up the "Services" window (Start -> Run -> "services.msc").  When the window appears, scroll down and make sure the line labeled "Server" shows that it is "Started" with Startup Type "Automatic".  If not, double-click on it and make it that way.  Do the same for the "Computer Browser" line, but I don't think that is your problem.

Assuming you are using Windows Firewall only on these machines, open up the Firewall settings

  (Control Panel -> Windows Firewall).  Verify that the "Dont allow Exceptions" box under the General tab is command uses UDP 137 only and it appears at least that port is open.  If you are using another firewall, try to temporarily disable it and see if it helps.

Go to Settings -> Network Connections (or Control Panel -> Network).  Right-click on your active network connection and select "Properties".  In the window verify that "File & Printer Sharing" and "Client for Microsoft Networks" is present and checked.  Click on "TCP/IP" to select it and Click "Properties" -> Advanced ->WINS (tab) -> and make sure "Enable NetBIOS over TCP/IP is checked.  Move to "Options" tab -> TCP/IP filtering -> Properties and verify no check beside "enable TCP/IP filtering".

Make sure that you have a folder that a Guest user has permissions to that is shared.  If there are no folders or printers shared, then connections won't work.  Just to make sure, create a new folder in your "My Documents" folder, right-click on it, select "Sharing", give it a share name, and share the folder with all permissions and see if you can now connect to that machine.

I hope something up there helps.

I don't know why the "net use" does not work for you.  There must be a syntax error because you should have no trouble with "net use" to computer 2 or 4 -- being that you can connect normally anyway.  I don't believe you can "net use" to the same computer.  Try just "net use" all by itself.  It should give you a list of current connections.  Maybe it will fail if there is already a connection.   Your "node types" all look good.  Even "unknown" is usually acceptable.  I can get to KB903267 with no difficulty but this is not your problem because you have good node types.  Computer #3's "HOST NOT FOUND" looks like it is because you have a 2nd network connection on that machine that is not currently used.  There should be no difference between wired and wireless connections.

bruce hagen ~ mvp say:

You have some dbx file corruption easily fixed.

Move most of your messages out of the Inbox and then create new Outbox and Sent Items folders after you move any messages you wish to save to a local folder you create.

Tools | Options | Maintenance | Store Folder will reveal the location of your Outlook Express files. Write the location down and navigate to it in Windows Explorer or, copy and paste it into Start | Run.

In WinXP, Win2K & Win2K3, the OE user files (DBX and WAB) are by default marked as hidden. To view these files in Windows Explorer, you must enable Show Hidden Files and Folders under Start | Control Panel | Folder Options Icon | View, or in Windows Explorer | Tools | Folder Options | View.

With OE closed, find the DBX files for the Outbox & Sent Items and delete them.  New ones will be created automatically when you open OE.

After you are done, follow up by compacting your folders manually while working *offline* and do it often.

Click on Outlook Express at the top of the folder tree so no folders are open. Then: File | Work Offline (or double click Working Online in the Status Bar). File | Folder | Compact all folders. Don't touch anything until the compacting is completed.

General precautions for Outlook Express:

Do not archive mail in the Inbox or Sent Items. Create your own user defined folders and move the messages you wish to save to them. Empty Deleted Items folder daily. Although dbx files have a theoretical capacity of 2GB, I recommend about a 300MB max for less chance of corruption.

Information about the maximum file size of the .dbx files that are used by Outlook Express:
http://support.microsoft.com/?kbid=903095

Turn off e-mail scanning in your anti-virus program. It is a redundant layer of protection that eats up CPUs, slows down sending and receiving, and causes a multitude of problems such as time-outs, account setting changes and has even been responsible for loss of messages. Your up-to-date A/V program will continue to protect you sufficiently. For more, see:
http://www.oehelp.com/OETips.aspx#3
