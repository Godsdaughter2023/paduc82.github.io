Please be informed that Recorded date and Date Taken are different metadatas. Photo Gallery only takes information based on the Date Taken that can be viewed on the said column within the Properties section of the photo. If the photo does not have a Date Taken metadata, it will use the Date created information. 

If the original photo has no information as to when it was first taken, Photo Gallery uses the Date created as its basis. You can check this behavior when you right click your photo/video, go to Properties and then Details. A Date taken column will be there. In case it has a date within this column, this will be the basis of Photo Gallery. But, if your photo/video has no information within Date taken, it will create a date as to when you have copied the file in your computer.

iscoed say:

Hello.  I have an XP SP3 computer at work that would not boot this morning.  The only new event the previous day was Firefox updating to version 9.  When I stopped it looping, it was a STOP c0000135 WINSRV not found error.  I have looked at all the various solutions on the web, and have not had any success.  I have used a UBCD4WIN boot CD to run Recovery Console, and have run chkdsk /r.  I have replaced the winsrv.dll and user32.dll files with various versions from the computer, and another similar computer.  I have run fixboot, fixmbr and bootcfg/rebuild.  I tried running registry restore, but it could not find a back up.  Last known good configuration does not work.  So, my last hope is to ask if any of you more expert that I can think of another solution, other than to re-install windows.

Thanks.
iscoed say: Thank you for this.  I swapped out the registry components, and it made no difference.  However, it led me to the real answer...(!)

I was looking the system32\config folder, and realised, of course, that the last time the .evt files had been updated was when the computer was last working properly.  Looking in the system log, I found this:

File replacement was attempted on the protected system file c:\windows\system32\usp10.dll. This file was restored to the original version to maintain system stability. The file version of the system file is 1.420.2600.5969.

There were then similar entries for locale.nls, sortkey.nls, and sorttbls.nls. 

Then

The system file c:\windows\system32\sortkey.nls could not be copied into the DLL cache.  The specific error code is 0x800b0100 [No signature was present in the subject.
].  This file is necessary to maintain system stability.

Then

The protected system file sortkey.nls could not be restored to its original, valid version. The file version of the bad file is unknown  The specific error code is 0x800b0100 [No signature was present in the subject.
].

Then another attempt on locale.nls.

Then event log was stopped.

I looked on the harddrive, and sortkey.nls was missing from the system32 folder.  There was a oldsortkey.nls dated 2004, and two copies of sortkey.nls dated 2005, one in the system32/DLLcache folder, and another in a service pack update folder.  Given the message about the DLL cache, I copied a 2008 dated file from another similar computer to the system32 folder, and the computer has booted up fine several times, with no problem. 

So, thanks to the three of you for your help - I got there in the end.

For the record that error is usually from the RPC Service not running but it should always be running and takes extra effort to stop it so something went haywire somewhere.

If the Service or the system is not afflicted in some other way restarting is a good first move.

Your error may have looked like these examples from my notes:

a. user say:

I'm not sure what any of that means but the only way I can see this problem is for the Remote Procedure Call (RPC) Service to not be running - and it should always be running so you can check yours and make adjustments if needed.

First open the Services applet thusly:

With the Services applet open locate the Remote Procedure Call (RPC) Service and notice the Status and Startup Type columns should say Started and Automatic respectively and if that is not what you see that is at least part of the problem, right click and choose Properties:

Set the Service Startup typ field to be Automatic and click OK enough times to save the changes:

Close the Services applet and restart the system then check the Remote Procedure Call (RPC) Service again to be sure it is running.
