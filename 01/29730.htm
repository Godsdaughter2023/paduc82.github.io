a. user say:

You can rename your win32k.sys to win32k.old and Windows will not "make" a new one, but what Windows File Protection will do is quickly and silently replace the missing file from the backup copy in the dllcache folder (that is where XP keeps backup copies of important files).  You will not "see" anything happen, but the file will be replaced and you will find an event like this in the Event Viewer System log:

Event Type:    Information
Event Source:    Windows File Protection
Event Category:    None
Event ID:    64002
Description:
File replacement was attempted on the protected system file c:\windows\system32\win32k.sys. This file was restored to the original version to maintain system stability. The file version of the system file is 5.1.2600.6379.

That is Windows File Protection in action.  WFP looks after 3498 critical files (yes there is a list) and if messing with the win32k.sys file seems a little scary, use some other less scary file like taskmgr.exe.  If you rename or delete taskmgr.exe it too will be quickly and silently replaced and the event will look like this:

Event Type:    Information
Event Source:    Windows File Protection
Event Category:    None
Event ID:    64002
Description:
File replacement was attempted on the protected system file c:\windows\system32\taskmgr.exe. This file was restored to the original version to maintain system stability. The file version of the system file is 5.1.2600.5512.

Let me assure you with 100% certainty that the suggestion to run the system file checker (SFC) will not replace a missing or corrupt win32k.sys file.  That is not how sfc /scannow works, that is not what it does and that is not what it is for.  That is just another Microsoft engaged Support Engineer wild goose chase.  Suggesting it is akin to saying "I have no idea what the problem is or how to fix it".

IF your win32k.sys file was missing or corrupt, your system would not boot and you will get specific errors that will help you figure out the problem.

When a BSOD implicates an XP file, the XP file displayed in the message is extremely unlikely to be the real cause of the problem and you will have to look at the crash dump files with the Windows debugging tools to see what the real problem is and there is no need for you to install all that stuff since other folks here can help you look at your crash dump files.

You need to follow the directions from forum contributor ZigZag3143 and disregard the directions from the Microsoft engaged Support Engineer "expert".
