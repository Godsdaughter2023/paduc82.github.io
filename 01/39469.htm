    james rshaw say:

    Here is a list of debugging steps I have developed to track down the offending condition(s) that may cause this auto-hide no longer auto-hiding anomaly. There may be more. These steps are taken from my personal experience on a real case that I just finished resolving.

    1. Cause: Window overlapping taskbar
      Sometimes when a window pushes under the taskbar, it prevents the taskbar from auto-hiding. Grabbing the top edge of the offending window, sliding the top edge down a portion of an inch, and then sliding the whole window up to remove the overlap will free up the auto-hiding mechanism.       This is not fixing the problem today.
    2. Cause: Window pushing against bottom of screen
      Sometimes when a window is against the bottom edge of the screen—such as when one does a <Window>+right or left command, it prevents the taskbar from auto-hiding. Grabbing the top edge of the offending window, sliding the top edge down a portion of an inch, and then sliding the whole window up to remove the bumping against the bottom edge of the screen will free up the auto-hiding mechanism. This is not fixing the problem today.
    3. Cause: Windows fullscreen
      Sometimes when a window is full screen—such as when one is watching YouTube videos, it prevents the taskbar from auto-hiding. Clicking the tiling button in the upper right corner of the screen to remove the bumping of the full screen against the bottom edge of the screen will free up the auto-hiding mechanism. This is not fixing the problem today.
    4. Cause: Too many tasks running
      If one has a bunch of ready-to-go tasks pinned to the taskbar and has run more of them than is room on the taskbar--forcing a spill-over condition at the right part of the running taskbar list, the taskbar will lock in place. Start exiting the running/active tasks until you have an empty space in the running tasks list.       This is not fixing the problem today.
    5. Cause: Prompt request pending
      When a process requests input from the user, (Do you want to install ABC program?), a taskbar icon will begin flashing in the taskbar awaiting input. The taskbar will remain unhidden and asserted until the click/input request is satisfied. This is frequently encountered when installing weakly written software from third parties that do not handle screen events well. This is not the problem today.
    6. Cause: Tasks running counter corrupted
      If one has a bunch of tasks running and displaying icons on the taskbar, the taskbar management system loses track and by some kind of hidden default, perpetually asserts the taskbar until the counter for the number of tasks running and displayed on the taskbar is reset. Solution: One by one end/exit each of the active tasks in the taskbar,. Auto-hide may resume ONCE THE LAST ACTIVE task exits. This was the source of my problem today! Problem solved!

    Now for some other ideas I have tried in the past.

    1. Cause: A hidden Icon is preventing hiding
      When a hidden icon is stealthily requesting input from the user, the taskbar will remain unhidden until the click/input request is satisfied. This is frequently encountered with weakly written software from third parties. Debug by carefully starting and stopping each process that is in the Hidden icons pop-up window in the system tray. Right-click each one until you find the offender. Once you find the offending program, you will know how to "fix" the auto-hide malfunction by right-clicking, starting and stopping the offending program, or stopping the offending program altogether.
    2. Cause: Something is corrupted in the taskbar process
      When stealth corruption messes up the taskbar volatile in-memory database, Anything goes. The fix is to either restart the system management program/service (explorer.exe in the task manager window), or reboot to clean things up.

    Hope this helps a bunch of folks. quickly resolve their auto-hide taskbar issues.
