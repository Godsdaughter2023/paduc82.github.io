I set up the task like this:
      Name: Notepad Test
      When running the task use the following user account: admin
      Run only when user is logged on.
      Run with highest privileges. (checked)
      Trigger: at logon of AUSER
      Action: Start a program: C:\Windows\Notepad.exe (with a valid
         start directory specifed).
      Everything else set to defaults.

If I "Run only when user is logged on" the task never triggers because mine is a single user system, I am either logged on as admin or AUSER but not both.  (And even if I was in a multi-user system, the admin is not logged in ALL the time, is he?)  

If I set the "Run As" user to AUSER my program GUI appears but  throws exceptions when it reaches methods that require elevated privileges.  (Apparently the "run with highest privileges" doesn't give AUSER higher privileges, it just elevates them if AUSER already is allowed the privileges in the first place.)

I've seen several posts on the .NET reporting success with this approach, but now I suspect their "AUSER" is is part of the admin group as well as users.  I can't have that and maintain good security.

tom

frederik long say:

I've seen several posts on the .NET reporting success with this approach, but now I suspect their "AUSER" is is part of the admin group as well as users.  I can't have that and maintain good security.

Sorry, you can't have a standard user account run tasks at the highest privilege. AFAIK, running the task with the highest privileges only works if the account used for the scheduled task is a member of the Administrators group. The solution might be to run the task in the background, create a log, then report the results in a separate process in the foreground.
