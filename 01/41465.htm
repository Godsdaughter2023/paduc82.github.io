hairyfool say:

RDP should not be dependant on the resolution of the remote desktop, it is it's own display device on the remote computer so by default will be rendered by the remote computer at your local computers resolution. This is because you don't normally take over the actual remote desktop but create a separate one. If you right click on the remote connection shortcut and go to the Properties > Display tab you can select a resolution less than your local resolution to make the remote desktop a Window.

 

The symptom you describe is more a desktop shadow facility such as provided by VNC or Log Me In.

rb-sight say:

OS doesn't.  The driver does.

You do not list a dock, laptop or full situation. (aka 4 year old dock with new E7470 downgraded to 7 enterprise)

Possible causes:

Chipset driver.  use the one from support.dell.com

Dock driver.  use the one from support.dell.com

Power supply.  If you use the laptop power supply on the dock this can cause the issue.

Power profile.  Many enterprises enable performance power profile instead of default balanced leading to a overload (common on tablets e.g. the Venue can pull 45w but the PS is 25w). 

Make sure to start your load with a clean DVD.  Having other drivers embedded can mess up hardware (e.g. Lenovo USB 3 Intel gen 6 drivers (M900) and HP USB 3 Intel Gen 6 (800dmG2) drivers are catastrophically incompatible it will result in a no boot situation during MDT deployment if slip streamed into the deployment wim and may brick the HP system board).

If you have a old dock, update both the dock firmware and the laptop / mobile workstation BIOS (while being docked).  I had to do this with a new 7710 and a 3y old dock. (note getting this error and no DP in the factory OS before the update.)

rgharper911 say:

In general the answer is yes - you can boot a DOS system on 64-bit hardware. However, as with all things, the devil is in the details.  The various DOS architectures are very simple creatures and don't support a lot of expandability - for example, device drivers.  Yeah, there are some you can load (mouse or memory for example) but if your hard drive controller is not seen as a simple IDE controller by DOS then it won't be recognized and won't work.

Also please do keep in mind that you've oversimplified slightly, as the various DOS flavors are 16-bit architecture and not 32-bit architecture.  I can see the possibility that any given 64-bit platform won't run 16-bit code well or at all.
