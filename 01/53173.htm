a. user say:

This will not resolve your system hang problem (not enough information) but just so you know what is going on with the restart and file system checking...

Using the power button, reset button, pulling the plug/battery to get a hung up system going again are of course and should be the last resort.

As a result sometimes the NTFS (NT File System) will get some slight corruption as a result and the Windows volume becomes "dirty".

When Windows starts it automatically checks the dirty bit on all volumes and if any volume is marked dirty Windows will run a chkdsk with error correction (chkdsk  /r) to see about fixing the problem.  It usually works to clear the dirty bit.

That is why you saw chkdsk running and that could take a long time to complete depending on the speed of your system, the size of the volume, the amount of data on the volume and what the chkdsk finds to do.  It can take a long time but always let it finish.

A clean run of chkdsk with error correction is the only thing that can clear the dirty bit on a volume so if you see chkdsk running on every reboot the dirty bit is not being cleared which could be a sign of a hard drive failure. 

Sometimes the dirty bit will never clear and you can tell Windows to stop checking the afflicted volume (to save your boot time) but if the dirty bit won't clear I would be suspicious of the volume and replace the drive or reassign it to other duties.

You can query the dirty bit on a volume from the command prompt window with Administrator privileges.

Click the Start orb/button and in the box enter the following:

cmd.exe

Right click cmd.exe and choose to Run as Administrator:


To query the dirty bit on drive C, in the Command Prompt window enter the following command:

fsutil  dirty  query  C:

Sample output should be one or the other of:

Volume C: is dirty
Volume C is NOT dirty

You should check each of your volumes or each of your hard drives (C, D, E, etc.).


Enter 'exit' to close the Command Prompt window.

I have recently come across a computer which would not boot properly and would freeze for no apprent reason once it was up and running and bluescreen occasionally. BIOS did appear but required 'F1 to' continue.

 

I reseated everything including the processor, changed out all of the drive cable connections, replaced the CMOS battery and updated BIOS. Then tested RAM and hard drive and still it baulked at BIOS.

 

The event viewer showed a mass of errors all related to being initially brought about by faulty hardware. In particular, there was a constant DHCP error which co-incided with the client accessing mail. Disabling the integrated NIC saw the error disappear. I fitted a 3rd party NIC and now the client computer doesn't freeze when accessing mail. 

 

The faulty integrated NIC signifies issues with the motherboard, and the fact that none of the other errors have gone suggests the same. It still occasionally baulks at BIOS boot but at least the freezing issue has gone. The other motherboard errors relate to the drive controllers, I think. Windows says they are working. Well of course they are otherwise Windows wouldn't be. The drive controller issue is intermittent and not driver induced.

 

The client, who is 84 yrs old by the way, has decided that she can live with the 'F1 to continue' until such time as F1 no longer continues..
