Hi Shehaal,

To index a network folder or drive, you must right-click on it first and select "Always Available offline".

Then it can be indexed, and added to libraries as well.

To add a non-indexed UNC as a library to Windows 7:

1. Create a folder on your hard drive for shares. i.e. c:\share

2. Create another folder in the above share. i.e. c:\share\music

2. Link the Library to this folder.

3. Delete the folder.

4. Use the mklink in an elevated command prompt to make a symbolic link. Name the link the same as the folder you created above.

i.e - mklink /d c:\share\music \\server\music

5. Done. Now you have non-indexed UNC path as a library.

Why is a shared EFS file transmitted in plain-text? - Microsoft Community

foxtwo say:

Why is a shared EFS file transmitted in plain-text?

I am just trying to understand why Microsoft has developed the sharing of EFS files the way it works.

I mean, when a shared efs file is requested remotely, it is first decrypted locally and then it is transmitted in plain-text through the network until it reaches the computer that requested it.

In the webdav folders shares, when someone requests an encrypted file on a web folder, the web server transmits the raw efs encrypted data to the computer that requested the file, the webdav redirector driver on the computer that requested the file imports the raw efs data (i suppose through windows Ntbackup APIs) and “mount” the file as an encrypted efs file on the webdav folder. Finally the file is decrypted only locally on the computer that requested the file, clearly if the user has the correct key.

I wonder why they didn't manage to make file shares through SMB working the same.

In my mind, this could be the possible scenario: If an encrypted file is requested locally it is directly decrypted and sent to ram but if the same encrypted file is requested remotely (through SMB) it is simply transmitted in its raw efs format (using OpenEncryptedFileRaw and ReadEncryptedFileRaw APIs) and then “mounted” in the computer that requested it (through WriteEncryptedFileRaw API) and decrypted only locally.

I tried to imagine the possible reasons:

  1. In a domain network the network communications are yet encrypted through kerberos so the trasmitted plain-text data is secured

  2. My scenario would add an additional overhead that would have made network communications slower (anyway I am really not sure about this)

  3. Implementing the EFS raw data transmission for remote requests would need a deep change of either the EFS drivers or SMB distributed file system structure and they considered that the effort was not worth it considering that communications in a domain are yet secure (not exactly the same in a workgroup)

 

I would really appreciate if someone could give me a technical explanation why they decided to transmit the efs data in plain-text

Thanks a lot for reading me

Michele

imran chand say:

Hi Wolvespete,

This problem can occur due to the following reasons:

a. The file is not installed correctly.

b. The file is corrupted.

c. Other programs, including memory-resident and anti-virus programs, are running and using any version of the file.

Step 1: Register the File Manually

1. Click Start and then click Run.

2. In the Open box, type the following command, and then click OK

RegSvr32 actskn43.ocx

Step 2:
1. Remove the corrupted installation files from Programs and Features and then try to install the PC – TV program (Start>Control Panel>Programs>Programs and Features).

2. Refer the below link to know about Uninstalling a Program:

http://windows.microsoft.com/en-US/windows7/Uninstall-or-change-a-program

Step 3:

I suggest that you disable the Antivirus\Firewall temporarily and then try to install the program.

 Note: Please make sure that you enable the antivirus software after performing the test to keep your computer protected.
