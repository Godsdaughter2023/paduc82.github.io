mjtrader say:

file and folder size

Hi, I am working on getting a new computer up and running including getting my backups up to date before loading on my new computer.  When comparing a few folders on my main computer hard drive, the backups done just last night don't always match in size and number of files, at least in some cases, but not all.

Can you explain this to me and if there is a way to verify that all of my information is actually backed up properly?  Also, would you recommend resetting my back up program to select all files I wanted backed up as a new beginning and then running another backup?  Thanks

lemp say:
There may be several things going on here.

First, make sure that you are comparing the folder's "size" on one drive to the folder's to "size" on the other drive and not to the folder's "size on disk."

"Size on disk" is very often not the same as "size" (it's usually larger).  This is because files are stored in "clusters" of fixed number of bytes.  There is only one file per cluster and thus there is almost always unused space.  This is called "slack." 

Cluster size represents the smallest amount of disk space that can be used to hold a file. When file sizes do not come out to an even multiple of the cluster size, additional space must be used to hold the file (up to the next multiple of the cluster size). On the typical hard disk partition, the average amount of space that is lost in this manner can be calculated by using the equation (cluster size)/2 * (number of files).

As an example, assume that you have 12 ounces of water and only 10 ounce cups.  You need two cups -- 20 ounces of "storage space" -- to contain all the water, and one of them is mostly empty.  On the other hand, if you had 4 ounce cups, you would need three cups -- 12 ounces of "storage space" -- and would have no unused space.

Cluster size differs between the different file systems used by Windows.  Most XP installations use the NTFS file system, which has a default cluster size of 4096 bytes (4 KB) for partitions up to 16 TB.  The FAT file system, on the other hand, uses clusters of 512 bytes and 1, 2, 4, 8, 16, 32, or 64 KB depending on the size of the partition.

Many external drives come preformatted using the FAT file system.  So if you are comparing "size on disk" and the internal drive uses NTFS and the backup drive uses FAT, there almost certainly will be a different "size on disk" between the two.

In your first example (168 GB folder), it looks as if you may not have backed up some hidden or system files.  What software are you using to do the backup?

Most good backup software has a "verify" option.  Exactly what is "verified" may differ for different applications, but if it is available in the software you have, you should use it.

colin carr say:

Finally,  I have a sort of a solution.

I do not know what causes the Task Scheduler to stop running but that is what is happening.   Why that particular error is displayed is anybody's guess.

If you have this problem you can re-start it as follows.

Click on the windows logo at the bottom left.

Click on Settings

In the search box at top right type in 'Services'

There should be several items displayed including 'Local Services' - click on that.

About 2/3 of the way down the list you should see Task Scheduler and a column towards the right which is either blank or has 'running'.

Left click on the line and a screen will appear which will allow you to start the Task Scheduler again.

Don't know how long it will take before it falls over again or what conditions have to occur for that to happen.
