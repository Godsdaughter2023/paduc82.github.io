Welcome to the Microsoft community!

Regardless of whether it is for work or personal use, you can connect to a VPN (virtual private network) on your Windows 10 PC. A VPN connection can help you provide a more secure connection to your company's network and the Internet (for example, when you are working in a cafeteria or similar public place).

In order to connect to a VPN, you need a VPN profile on your PC. You can create a VPN profile on your own or set up a professional account to obtain a company VPN profile.

Create a VPN profile
If you do not have any VPN profile on your Windows 10 PC, you will have to create one on your own. Before starting:

If it's for work, look for the VPN configuration or a VPN application on your company's intranet site when you're at work, or contact the company's technical support staff.
If it is for a VPN service that you subscribe to for personal use, visit the Microsoft Store to see if there is an application for that service, and then go to the VPN service website to see if the VPN connection configuration that is must use appears there.
Select the Start button and then select Settings> Network and Internet> VPN> Add a VPN connection.
In Add a VPN connection, do the following:
In VPN Provider, choose Windows (integrated). In the Connection box, type a descriptive name (for example, my Personal VPN) for the VPN connection profile. This is the name of the VPN connection that you should look for in the server names or in the address box when you want to connect. Enter the address of the VPN server.
In VPN Type, choose the type of VPN connection you want to create. You will have to know what type of VPN connection the company uses or the VPN service.
In Type of login information, choose the type of login information (or credentials) you are going to use. This information can be a username and password, a one-time password, a certificate or a smart card if you connect to a work VPN. Enter your username and password in the corresponding boxes (optional).
Select Save
If you need to edit the VPN connection information or specify additional configuration options, such as the proxy settings that will be used for the VPN connection, choose the VPN connection and then Advanced Options.
Connect to a VPN
When you have a VPN profile, you can connect.

At the far right of the taskbar, select the Network (or) icon.
Select the VPN connection you want to use, and then perform one of the following actions depending on what happens when you select the VPN connection:
If the Connect button is displayed below the VPN connection, select Connect.
If the VPN is opened in Settings, select the VPN connection, and then select Connect.
If asked, enter your username and password or other login information.
Once connected, Connected will appear below the name of the VPN connection.
To quickly check if you are connected to the VPN while doing tasks on the computer, select the Network icon (or) at the far right of the taskbar and then see if the VPN connection indicates "Connected" below.

brendon windmeyer say:

I've seen this, too, and can only speculate that your experience was the same as mine.   For me, I had search indexing turned off, and I don't think Windows likes that too much.  So therefore, when you ask to search something, it takes the opportunity to search like crazy, and that entails searching in places it's already looked. 

My test that proved it to me was that I had it search one folder, which only had a few files, for a file I knew wasn't there.  Being so small, it "finished" the search instantaneously.  While leaving the search window up and untouched, I created a file in the folder that was "done searching".  The file then immediately appeared in the search results, suggesting that either the file creation triggered an event or the search kept searching that folder.  I spread my test to include larger and larger sections of my hard drive and the effect was the same, albeit slower.  Eventually the scope of my search was so large, that although it checked everything once, it immediately started checking everything again and looked like it never was stopping.   When I turned on indexing, however, this behavior stopped. 

Another guess as to why this was happening is that the new search results are triggered as an event, but an event that also triggers a virus scan.  The virus scanner checks all the files that were touched in the file search, which in turn triggers the search window to search all those files again.  The war of attrition continues until you terminate the whole thing.  I think I missed my opportunity to check this, but I think it's a very valid scenario.

This is only one experience, though.  Other reasons this may happen, especially if you're searching multiple drives, is that one piece of hardware isn't responding in a way windows expects, and the search never gets the final acknowledgement that it's done.  Something like an obscure bug in the hardware drivers or bad media could cause this. 

To summerize things to try:

  • turn on indexing (temporarily if just to test, but give it time to do a full index)
  • turn off active virus scanning (definately temporary if there's any virus risk)
  • update virus scanner engine
  • ensure your device drivers are up-to-date
  • run disk checks on the drives you are checking.

Hope one of these wordy shots-in-the-dark help!

lemp say:

No problem -- no time wasted.

Your statement isn't quite clear how your printers are connected, but I assume you are using a router.  If you are not using a router, the basic problem described below is very similar, but some details are different.

In order to work when connected by Ethernet cable, a printer has to be assigned an IP address.  In this situation, the typical default setting of the printer can easily cause a problem such as you describe. 

Here's a description of the problem and how to avoid it.

All home printers with which I am familiar that are capable of being connected over a network default to obtaining an IP address automatically.  This is not a problem in the typical home network, where the printer is connected to a router by using either an Ethernet cable or wi-fi.  The router has a "DHCP server" that automatically assigns IP addresses to each device connected to it (as long as the device is configured to request an address).

When the printer is first installed on a Windows computer, part of the installation is the discovery of the IP address assigned to the printer.  Windows then configures its printing subsystem to direct output for that printer to that IP address.

For example, the image below shows that the "Downstairs HP 6988" printer has been assigned IP address 192.168.1.26.  You can see that there is another network printer in this system, the "Upstairs HP 6988" printer; that printer has been assigned IP address 192.168.1.25.

The problem arises when the printer is turned off and other devices are connected to the local network during that time. 

At some time after the printer is turned off (or disconnected from the network), the router's DHCP server recognizes that the IP address is no longer in use and makes it "available."  If another device (a printer or something else) is then connected to the network, that IP address may be assigned to the new device.  When the printer is subsequently turned on (or re-connected to the network), the DHCP server assigns it a new IP address.  The problem is that Windows still directs print output to the old (original) IP address.  Because the printer isn't there, nothing happens.

Step 6 in my earlier response describes how to see the IP addresses where Windows directs print output for each of your printers.

If those IP addresses were assigned automatically and if the "Upstairs HP 6988" printer was turned off long enough for address 192.168.1.25 to become available, and if I then connected another network device that was assigned that address, when I then turned the printer back on, it would be assigned a different address, e.g., 192.168.1.27.  But Windows would still try to print to 192.168.1.25.

Step 8 in my earlier response describes how to see the IP address actually assigned to the printer.  If the IP address assigned to the printer (as shown in step 8) isn't the same as the IP address where Windows is directing the printing output (as shown in step 6), you can't print.

One way to avoid this problem is to assign a "static" IP address to the printer and then configure Windows to use that address.  A "static" IP address simply means one that doesn't change.  How you do that will be described in the User Guide for your printer. 

In fact, both of the HP 6988 printers in the image above are assigned static IP addresses.  They will always have those addresses, even if they are powered off or disconnected from the network.  Thus, these printers are always where Windows expects to find them when it sends a print job.

Another way is to "reserve" the IP addresses assigned to the printer(s) so that the router will not assign that address to anything else, even if the printer is powered off or disconnected from the network.  How you do that will be described in the User Guide for your router (not all routers can do this, but many newer ones can).

If you want help on the above, please provide the specific models of your Epson and HP printers and the make and model of your router.  For the latter, look at the label on the back or bottom of the device and include any hardware version indicated.
