richman101 say:

This may be a late reply, but here is what I know and will research more and add to this post alter.

The file extension .vhd stand for Virtual Hard Disc. As I believe that may imply, that the system image function possibly creates a clone instead of an image as most commonly defined. The difference would be that a clone, as in what most software that does both cloning and imaging, creates an actual duplicate of your System drive or partition. This is good if you have a hdd failure and need to reproduce your system on another (possibly new) drive. A clone though, is only as current as the day you produced it and will have no updates or changes to the system since it was created. A clone can not be changed or manipulated in any way. Cloning in essence, creates a clone by copying a system bit-by-bit and then writing it to another drive bit-by-bit. It is an exact replica. Any attempt to change the clone will break the integrity and cause it or the resulting clone to not function properly if at all. They also have hardware specific info included which is alright in these situations since it is your system and hardware specific files will pertain to you and your system. This is specifically designed to do this in user situations such as these and Microsoft did not intend to provide an imagine provisioning that would allow a user to duplicate system and deploy them of other similar or dissimilar hardware since the user license agreement normally does not include such provisioning.

The file extension .wim stands for Windows IMage Format. This use of the word image would imply and most factually so, that the software creates an image of a system. An image is a file with embedded files that recreates an already installed system. This allows the image file to be changed and manipulated in ways that a clone of a system can not be manipulated. An image can have parts disassembled, parts added to it such as software, drivers, updates and changes that occurred while using the original system that it was created from, usually called incremental and differential updates, and have hardware specific changes deleted. This method is usually used for deployment purposes where a hardware manufacturer will create an image by installing an OS on a reference system and then image that install for deployment to other machines that they are producing or an IT professional who is setting up 100's of client systems in a corporate office environment and creating an image for deployment onto them.

Will the one be able to use the file extension of the other? Not likely. At least I don't believe they do as they were developed separately for different purposes in mind.

Can other PE's use these files? I think it revolves around whether or not they utilize the same core application or technologies, namely Sysprep and ximage or imagex (as its name is sometimes interchangeable) as these file extensions were created for these specific applications. It is not impossible for an application to be able to use another 's file extension type but I don't believe this to be the case as I think MS wanted to keep them separate for a reason. There is UBCD in the form of .iso which I think has Sysprep but not sure about imagex or ximage. There are other PR creators such as PE3 and BartPE and I believe if you had the right files you could create a WinPE that had imagex included. I am not sure if any of these PE creators by default. create WINPE images that create images utilizing both of these extensions but I am not entirely sure. I will be finding out more about this and edit this later as more info is available

These crashes were related to memory corruption (probably caused by a driver). 

Please run these tests to verify your memory and find which driver is causing the problem.  

If you are overclocking (pushing the components beyond their design) you should revert to default at least until the crashing is solved. If you dont know what it is you probably are not overclocking.

1-Memtest. (You can read more about running memtest here)
Download a copy of Memtest86 and burn the ISO to a CD using Iso Recorder or another ISO burning program. Get it here
Boot from the CD, and leave it running for at least 5 or 6 passes.
Just remember, any time Memtest reports errors, it can be either bad RAM or a bad motherboard slot.
Test the sticks individually, and if you find a good one, test it in all slots.

If a known good stick fails in one motherboard slot but not another it is probably the slot.


2-Driver verifier (for complete directions see our wiki here)
Using Driver Verifier is an iffy proposition.
Most times it will crash and it will tell you what the driver is, but sometimes it will crash and won't tell you the driver.
Other times it'll crash before you can log in to Windows. If you can't get to Safe Mode, then you'll have to resort to offline editing of the registry to disable Driver Verifier.

I'd suggest that you first backup your data and then make sure you've got access to another computer so you can contact us if problems arise.
Then make a System Restore point (so you can restore the system using the Vista/Win7 Startup Repair feature).

Here is the procedure:
Go to Start and type in "verifier" (without the quotes) and press Enter
Select "Create custom settings (for code developers)" and click "Next" 
Select "Select individual settings from a full list" and click "Next"
Select everything EXCEPT FOR "Low Resource Simulation" 
For Win 8 dont check Concurrency stress test, and DDI compliance checking click "Next"
Select "Select driver names from a list" and click "Next"
Then select all drivers NOT provided by Microsoft and click "Next"
Select "Finish" on the next page.

Reboot the system and wait for it to crash to the Blue Screen.
Continue to use your system normally, and if you know what causes the crash, do that repeatedly. 
The objective here is to get the system to crash because Driver Verifier is stressing the drivers out. 
If it doesn't crash for you, then let it run for at least 36 hours of continuous operation.

If you can't get into Windows because it crashes too soon, try it in Safe Mode.
If you can't get into Safe Mode, try using System Restore from your installation DVD to set the system back to the previous restore point that you created.

Hi,

 

Thank you for posting your question in the Microsoft Community!

 

It seems that you need more information to enable deleting the confirmation; I would suggest you to go through the following information.

 

Enable Delete Confirmation Dialog in Windows 7.

 

Every time you delete a file, Windows asks you “Are you sure you want to move this file to the Recycle Bin?” This is typically a good thing, to prevent you from deleting files by accident, but if you are trying to do some cleanup and need to delete a lot of files, you might not want to be prompted every single time.

 

There’s a simple way to turn it on, thankfully.

a.      Right-click on your Recycle Bin and choose Properties from the menu.

b.      Now you’ll see the Trash Properties dialog, and notice the “Display delete confirmation dialog” at the bottom.

c.       Check the box, and close the dialog.
