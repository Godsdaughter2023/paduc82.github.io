jamesmaro say:

Can't add network location to Library

I have two machines, one with Vista Ultimate SP2 32-bit and one with Windows 7 Professional 64-bit.

I am trying to add a folder on a mapped drive shared from the Vista machine to a library on the W7 machine. I get the error message "This network location can't be included because it is not indexed."

Due to the size of the folder I do not wish to make it available offline; the help says "If the folder is already indexed on the device where it is stored, you should be able to include it directly in the library" so that shouldn't be necessary!

I can access the mapped drive and folder directly from the Windows 7 machine with no problem (password protected sharing is disabled, no special credentials are stored).

The network between the two is listed as Private in Vista and as a Home network in Windows 7. I do not have a Homegroup set up. The Workgroup is set to WORKGROUP on both machines.

As I have Vista SP2 installed, I believe I already have Windows Search 4.0 (attempting to install it directly says the update is not applicable).

The disk that is shared is listed under "Included Locations" in "Indexing Options" on the Vista machine and I can perform indexed searches on it locally with no problem.

The mapped drive letter is Z: and the network location is \\VISTA\e

I have the following listed in "Default Indexed Paths" in group policy (on the Windows 7 machine).
otfs://{*}/VISTA/e/*
otfs://{*}/Z/*

(I have tried with none, either, and both present.)

I have tried rebooting after each change and rebuilding the index from scratch on both machines, even waiting for it to say "Indexing Complete."

In "Indexing Options" on the Windows 7 machine, neither the drive or the folder are listed under "Included Locations"; however, if I go to Modify, they are listed and ticked but with the tooltip "This Location is currently unavailable. Uncheck it to stop finding items that used to be here."

There doesn't appear to be any network traffic if I stop and restart the wsearch service and reopen the "Indexing Options" page. What makes it think the location is unavailable?

Any ideas for further investigation or fixes would be much appreciated!

Thanks,

James

jamesmaro say: I've now solved this; I'll list what I've discovered here in case it helps someone else!

I had two connections between the computers (The Vista machine was in fact a VMware VM - I have one bridged connection on it, and one host-only. I set this up after having transfer speed issues with shared folders going between a bridged connection and the host). Even though file and print sharing was disabled in the connection properties of the "real" network connection (which I didn't want the traffic to go over), W7/Vista still did some of the netbios traffic over it over IPv6! I disabled IPv6 in the connection properties on both machines.

The remote query part of Windows Search seems to have permissions issues and not work at all with "Everyone"/password-free access. I set up a proper user account and stored the credentials.

Accessing or mapping the folder with the IP instead of the machine name stops W7 being able to recognize the folder as indexed.

The whole "otfs://" thing seems to be a massive red herring in W7. The folders still don't appear in the "Indexed Locations" list even though they're now in the libraries and indexed search works on them.

Note that it also doesn't work with admin shares, and having the same username on both machines with different passwords can also confuse things (that came up when I still trying to get things working without password protected access).
