Hi cosmingurau,

 

If you go to your family doctor and he thinks you have a serious lung problem, he doesn't try to treat it but instead refers you to a Pulmonologist who specializes in such things so you can get the proper answer from someone who is an expert in that field.  Even if it isn't a true answer, it's the best recommendation because it will in the long run get you to someone more likely to be familiar with the most recent treatments and tests to diagnose and cure your condition.  If it resolves the problem or leads to a resolution in the long run, then it was the best "answer" you could get from that family doctor.  In essence, that's what my recommendation was.  We don't have the skills of these malware-removal experts and so they are who you should be consulting if this is actually an infection-related problem.

 

You indicated you don't want to run troubleshooting tests or anything because you've tried it all before and it didn't help.  Given that, there isn't much we can do because no, it's not a simple of matter of "if XYZ, do ABC."  There's almost never only one cause of any problem (and we don't have the results of "everything" you did before so we don't have the information.  It's like asking the doctor for a diagnosis without allowing blood tests or radiology tests or a physical exam or even providing a historical chart, but expecting a working treatment based only a brief explanation of symptoms by the patient.  The doctor usually can't do that. 

 

You're convinced it is an infection (or the result of an infection).  I'm inclined to agree - so the solution is to determine what infection and then the treatment for it.  Rather than guess and try various options to try to find out (like a family doctor), I referred you to specialists in infections (like the Pulmonologist) who can use the best tools to identify it and the best tools to eradicate it.  That must be done before anything can be fixed or it will just break again almost immediately.  Once that is done, assuming doing that doesn't fix the problem on its own, then we can address whatever problem remains KNOWING FOR CERTAIN that an infection is not involved and anything we do will not be undone by that infection. 

 

And if malware-removal techniques don't fix the problem on their own (but eliminate the cause so fixes may now actually work), there's a good chance we'll need to do some troubleshooting first.  That means that unless the malware-removal experts resolve your problems entirely (and that might work), that no there still isn't someone who can say "if XYZ, do ABC" because the fact is that we don't really know enough about what "XYZ" is to do that.  We just have a superficial explanation and that simply may not be enough.  If we REALLY knew what "XYZ" was it would mean we WOULD know what files or programs or settings or hardware or whatever was affected (like a doctor might who had a complete chart with not only the patient's explanation but all the lab and other detailed tests run in the meantime) and then maybe we would be able to come up with "ABC." 

 

For example, you say SFC /scannow doesn't fix it.  OK, I accept that.  Sometimes it works great, and sometimes it doesn't do the trick.  That may be because of the infection at the moment or it may be because SFC cannot repair the damage.  But even if it doesn't fix the problem, the report may help identify the problem (even if we need to use the detailed report in the log and not the shorter txt version) and that gets us closer to "ABC".

 

Most of  the time, error 0x80004005 means "Access is denied" somewhere which generally means some DLL file or something like that (or more than one something) needs to be re-registered (it could be worse than that or a bigger registry problem too - but I don't know that from the information we have at the moment). 

 

This needs to be done not only as an administrator but using Command Prompt run in elevated mode (meaning right click on Command Prompt and click "Run as Administrator" even though you are logged in as an administrator because despite that, some functions won't work unless you do that).  But that's just an aside. 

 

The point I'm getting at is that it suggests, at least to me, that one or more of your system files is not properly registered and is therefore being denied access and THAT is what is causing the problem.  It could also be permissions-related, but that we should be able to fix that if we discover that to be the case.  You would like me to magically know which files those are so they can be re-registered or have permission problems and fix the problem (or possibly have them replaced from somewhere else on the computer or from the installation disk if you have one or from another similar computer with exactly the same OS and the same hardware if the file is hardware-related - assuming any of that is possible) - but I don't know because I have no access to reports that tell me which files are affected.  If the error had been nice enough to provide which file or files were affected, then we'd know.  But perhaps it was. 

 

There's a fair chance that this error resulted in an Event being recorded in Event Viewer and details of that event (or those events - there may be more than one) may give us a clue about which files are affected or which systems are affected - but you don't want to consider running tests or providing us with results (since we should somehow know this) so we are missing the key facts needed to possibly resolve the problem (no guarantees - but it's a possibility).  Note though that it could be any number of different files (one or more) and without testing or seeing logs, we simply don't know which ones.  Even if we encountered an almost identical problem on another computer, the files affected might be different in each case.  In fact, to be perfectly frank, after the infection is resolved,  this may or may not resolve the problem even if we identify those files as the damage may be too severe or too extensive or the files cannot be simply replaced for a variety of reasons and it may come to a clean install - but I'm not ready to give up on it yet.  But it's also a possibility that can't be determined because we lack enough information about the details of the problem(s).

'mike p say:

Mostly computers that come with factory installed Win8 and newer will have the hard disk layout in UEFI format.  https://en.wikipedia.org/wiki/Unified_Extensible_Firmware_Interface

Windows ten Power Options has a feature called 'Fast Startup'   basically a partial hibernate when you shut down the computer.    When its in any hibernate state you don't see the available options to enter bios or alternate boot device choice.     Win10 you can Restart the computer and then it doesn't use that 'Fast Startup' hibernate and you will see the key press options that will work to enter bios or your F12 boot device selections.

You can disable 'Fast Startup in Power Options / Choose what the power buttons do / Change settings that are currently unavailable ... scroll down and you will see a check box to enable or disable 'Fast Startup'
