rohn007 say:
Probably not.

The thing is, CHK's are not files as such. They are actually file fragments that have become "disassociated" from each other because the file system was interrupted before it could create the links required to link the blocks used by a file together. The chkdsk utlity scans the disk to find these orphan blocks and names them CHK files.

This page goes into more detail:
https://www.howtogeek.com/282798/what-are-the-f...

A CHK file represents "1 block" of storage on the HD. Exactly how big that is depends on your total drive size. On current drives you can expect 4K, 8K, and larger. If your file fits completely in a block/chk, then you stand a chance of being able to recover the file. But files that are larger than the block are a problem. If you can match the CHK files together great. I may be possible with text files. But for files like graphics, very unlikely you will be able to reassemble them.

http://www.alphaurax-computer.com/computer-tips...

You can extract text, then piece the text found in separate CHK files back together, if you are familiar with the file content.

  • On the File menu, click Add/Remove Snap-in
  • Under Available snap-ins, click Certificates, and then click Add
  • Under This snap-in will always manage certificates for, click Computer account, and then click Next
  • Click Local computer, and click Finish
  • If you have no more snap-ins to add to the console, click OK
  • In the console tree, double-click Certificates
  • Double-click the Trusted Root Certification Authorities store and click on Certificates to view all certificates in the store
  • Select the two DigiNotar Root CA certificates. You can confirm the right certificates by checking their thumbprints which should be “c0 60 ed 44 cb d8 81 bd 0e f8 6c 0b a2 87 dd cf 81 67 47 8c” and “43 d9 bc b5 68 e0 39 d0 73 a7 4a 71 d8 51 1f 74 76 08 9c c3”
  • Right-click the certificates and select Delete

To perform the above steps from the command-line, you can use the certutil.exe tools as follows:

  • certutil -delstore authroot “c0 60 ed 44 cb d8 81 bd 0e f8 6c 0b a2 87 dd cf 81 67 47 8c”
  • certutil -delstore authroot “43 d9 bc b5 68 e0 39 d0 73 a7 4a 71 d8 51 1f 74 76 08 9c c3”

If you distribute roots in your enterprise using group policy, follow the instructions to remove a root across an enterprise network via group policy -http://technet.microsoft.com/en-us/library/cc786148(WS.10).aspx. Step 2: Clear the cache to remove any older cached CTL

The simplest way to do so is to use “certutil –urlcache * delete”. This will clean up the cache for the current user. You can find further documentation on this step, including a Microsoft Fix It package to clean up the cache, at http://support.microsoft.com/kb/2328240

tryx3 say:

The default location is at C:\Documents and Settings\YourUserName\Local Settings\Application Data\Microsoft\Outlook and the default name is Outlook.pst

 

You can put Outlook files anywhere you want so you might need to search for *.pst to find them all [assuming you created more than one].

 

If you want to put them in a location that gets included in your backups in Windows 7 then:

1 Close Outlook after having used it at least once [the file is created on first use].

2 Go to C:\Users\YourUserName\AppData\Local\Microsoft\Outlook [you will need to show hidden files to reveal the path to get there or you can just paste the path into the Windows explorer address box to go straight there]

3 Move the Outlook.pst you find there to a convenient location within My Documents.

4 Open Outlook.  It will report that it cannot find Outlook.pst and give you the option to browse for it so point it to where you have put the file.  [When you next open Outlook, it will look at the new location without any further prompting]
