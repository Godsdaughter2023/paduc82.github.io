dogtamer say:

Hello, saberman

     The fact that the details tab of your net.exe's property sheet isn't populated with the correct information is undoubtedly a problem; this is why Support Engineer System File Checker ("SFC"), which is designed to find and replace corrupt or missing system files from the backups in your component store (the Windows Side by Side or WinSXS folder; note that the KB929833 article I've just linked ("System File Checker") has yet to be re-written to reflect the fact that since the advent of Windows Resource Protection in Windows Vista, there is no longer a dllcache folder in System32, and the system file backups are now located @ C:\Windows\winsxs\Backup).  Running SFC should have found the problem with net.exe & repaired it, without the need to resort to a system restore, which can sometimes cause problems such as the one reported by the error message you've posted, i.e., that net.exe "isn't valid for your system".  I'd suggest reading the KB929333 article about SFC and seeing if running the tool doesn't correct the problem; if it does not, a good place to start troubleshooting to resolve the difficulty is the sfcdetail.txt log, which shows the results of each file-verification operation and each repair transaction attempted; instructions for how to extract/transform/load this log from its "home" in the over-arching servicing log (CBS.log) into a useable format are included under the "How to view details of the System File Checker process" section of KB929333, and instructions for how to repair files that SFC cannot are included in the article's "How to manually replace a corrupted system file with a known good copy of the file" section (both sections are available by clicking the circled "+" sign next to "For more information").  It's worth noting that it's not unusual to have to run SFC multiple times to take care of all corruption - 3 times is the recommended minimum, and I've personally seen it require more...the error dialog you're seeing leads me to believe there's more wrong than simply a lack of net.exe functionality, and SFC may be just the thing to get things straightened out.

     That said, it's also true that there is often more than one place a protected system file is stored, other than C:\Windows\winsxs\Backup, on a Windows 7 computer; typical locations for other instances of net.exe include the folders C:\Windows\winsxs\x86_microsoft-windows-net-command-line-tool_31bf3856ad364e35_6.1.7600.16385_none_5208a7a3d3caa54c, while net1.exe can usually be found at both C:\Windows\winsxs\x86_microsoft-windows-net1-command-line-tool_31bf3856ad364e35_6.1.7600.16385_none_86b24994b5e6bfbf and (on a 64-bit machine with an AMD processor) at C:\Windows\winsxs\amd64_microsoft-windows-net1-command-line-tool_31bf3856ad364e35_6.1.7600.16385_none_e2d0e5186e4430f5 (note that the "31bf3856ad364e35" portion of these folder names may vary); there may be other locations in the component store depending on such things as how many times system restore has been run, whether it's an OEM computer  (i.e, one pre-loaded with Windows by an Original Equipment Manufacturer) that has ever been restored to "Factory Spec", what Windows Updates, service packs and other hotfix/patches have been applied, and other factors.  The trick is in knowing which file is the correct one to use in replacing a corrupt version in C:\Windows\System32 - which is not always easy to determine - and in "taking ownership" of the offending corrupt file so it may be deleted or overwritten. 

     The windows component store can usually be used as the source of a replacement file for one that SFC is unable to correct using the information in C:\Windows\winsxs\Backup, but a user must be sure they know what they're doing to prevent catastrophe (which is why Microsoft has included the "automated" option of performing a "Repair Install" to correct system corruption) - and if the user has access to another, working computer running the same version of Windows, or to the original installation media (provided no Service Pack or other kernel update has been applied), or to self-created "slipstream" media (if Windows 7 Service Pack 1 has been applied) crafted specifically for the operating system in use, extracting known-good copies of the needed file from those are typically an "easier" approach.  Step-by-step instructions on how to use any of these forms of installation media with SFC may be found in Wiki Author Try*3's December 6, 2013 post to the Community thread "System File Checker and Access to Files Blocked", while step-by-step instructions on how to create "slipstreamed" media may be found in his August 25, 2012 post to pg 3 of the thread "A guide to recovering from Windows 7 Home Premium User Account lost password, cannot login, login problem" (it's also worth having a look at that whole thread, which has some service OneDrive, which is where one typically places uploaded files (such as the zipped copy of your corrupt net.exe file you mention) to make them available to others; to learn more about how to do that I recommend Community Moderator Gerry C J Cornell's excellent tutorial "Microsoft Community Forums" on his Gerry's Computer Tips site.

     As you've noted, net.exe (and net1.exe - which as I recall has been around in all NTFS operating systems since @ least NT4.0, though scarcely documented; the only official documentation I've ever

seen mentioning it is the Windows Embedded article

protected system files) is "owned", in terms of file/folder permissions security, by the "Trusted Installer" service.  The implementation of Windows Resource Protection from Vista onwards restricts ownership of protected system files to this service, largely to protect against accidental or malicious deletions.  Nevertheless, a user with administrative ("elevated") privileges may always take ownership of most any file and subsequently manipulate it at will.  The TechNet Library article "Take Ownership of a File or Folder" provides guidance on how to do this, but without the expertise in creating a batch file, macro, visual basic program, Power Shell cmdlet, or the like to handle such action on a large-scale basis, the procedure described is basically limited to a one-at-a-time approach.  In my opinion, a better technique is to add the capability to take ownership of any file or folder to the context ("right-click") menu of the operating system; many folks have found the the information and automated "registry hacks" in the Seven Forums tutorial "Windows 7: Take Ownership Shortcut" quite useful in this regard.  Even with such ability, users may occasionally run across a file that stubbornly refuses to behave even after ownership has been re-assigned to said user; in such cases the stand-alone ("Portable Executable", requiring no installation to run) freeware program UnLocker can prove invaluable.

     I'm hopeful that SFC will suffice to take care of this problem for you, but if not, I'd also consider

verifying hard drive/partition integrity with KB2255099 article "How to Perform an In-Place Upgrade on Windows Vista, Windows 7, Windows Server 2008 & Windows Server 2008 R2" and in Seven Forum's tutorial "How to Do a Repair Install to Fix Windows 7".  Good Luck!

MAC Filtering - Microsoft Community

xbr poseidon say:

MAC Filtering

Westell 9100

Login Information

Gateway - 192.168.1.1

Username - admin

Password - <User Created Password> or password1 or  The serial number on the back of the device

MAC Filtering

  1. Plug the computer into a LAN port via an ethernet cable.
  2. Open a web browser.
  3. Log into the router by entering the default gateway into the address bar of the internet browser. Example 192.168.1.1
  4. The router will then prompt you to enter the username and password.the default user name and password are: "admin" and "password"/"admin"/"password1".
  5. Select Wireless Settings from the tabs at the top.
  6. Select Advanced Security from the menu on the left.
  7. Select Wireless MAC Authentication.
  8. Make sure that there is no check in the Enable Access List box.
  9. Click the Apply button.
