Usually we receive this error message if the TS license in the computer is damaged or corrupted. You need to remove it from the TS client to fix this issue. The license is stored in the client's registry.

Its a dell Vostro laptop and they come with Windows 7 Pro. Yes I can always use the Dell OEM disk to reinstall the OS and still make it genuine because it activates itself automatically but is this still legal? No COA sticker?
balaji kundalam say:
The issue is, OEM release Windows 7 installation disk exclusively for their brand range of PCs, which doesn't require activation (meaning, Windows will be activated by default). This legacy feature is applicable for all the models, irrespective of the Windows license that the product holds.

For example, if you purchase a Dell laptop with Windows 7 Ultimate preinstalled (you can find COA with the product key), and creates a recovery disk or an installation disk was provided with the PC, the disks work fine with other models (works with other model range of Dell only and not suitable with other brands like HP) also where Windows 7 Home Premium or no Windows (like DOS or Linux, in which case, COA is not present) was preinstalled. You can't find any issue in using the same Windows 7 Ultimate in these PCs as these PCs also happen to be the same OEM built. But, this step is illegal as per law as the license is for Windows 7 Home Premium and that no payment (for upgrade or a new license) has been made before using Windows 7 Ultimate edition.

To stop this, Microsoft upgraded the licensing technique in Windows 8 where the COA/product key is embedded in the computer's BIOS itself and so, the Windows installation disk provided with that particular PC can't be installed in other PCs even though they are of the same OEM built.

In your case, absence of COA means, either the label must've been damaged/removed or that the product is legally licensed to hold preinstalled Windows 7. In that case, a new license must be purchased. If the label is damaged/removed, you can install the same edition of Windows 7, but kindly confirm this with Dell, using the Express Service Code or Service Tag, before proceeding.

Forced silent auto reconnection

When a server has been unavailable (offline mode) and then becomes available again for connection, Offline Files Client Side Caching tries to transition that server to online mode if all the following conditions are true:

  • There are no offline changes for that server on the local computer.
  • There are no open file handles for that server on the local computer.
  • The server is accessed over a “fast” link. You can adjust the definition of “slow” and “fast” by using theSlowLinkSpeed Offline Files policy.

With this hotfix, you can configure Offline Files Client Side Caching to ignore these conditions and transition the server to online mode regardless of whether these conditions exist. To do this, follow these steps:

  1. Click Start, click Run, type REGEDIT, and then clickOK.
  2. Locate and click the following registry subkey:
    HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\NetCache
  3. Click Edit, point to New, and then click DWORD Value.
  4. Type SilentForcedAutoReconnect, and then press ENTER to name the value.
  5. Double-click SilentForcedAutoReconnect.
  6. In the Value data box, type 1, and then click OK.

Note After you apply this hotfix, the following behavior may occur:

  • Any offline changes will remain unsynchronized and unavailable until the changes are synchronized in the future. This situation causes a dirty cache condition that is announced through a small warning overlay image on the Offline Files icon in the notification area.
  • Any open handles to files that are cached on the associated server will be automatically closed and invalidated. This action may cause problems if the programs that are using those files cannot work gracefully with invalid file handles.
  • If the server is available over a slow network link, such as a satellite link or a telephone line, the server will still be transitioned online. This scenario may cause very slow access to the remote file system on the server from the local computer.
