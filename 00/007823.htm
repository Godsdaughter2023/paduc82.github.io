a. user say:

The only VBA-less, .NET code-free way I can imagine that would let you insert XML data with repeating groups into a Word document would be to use an XSL transform to generate either Word 2003 WordprocessingML or Word 2007 Flat OPC format XML ("Flat OPC" is an uncompressed, single-file encoding of the OOXML format used by .docx).

All I can say is that I covered some of this ground many years ago and the main problems for the kind of use we discussed in the earlier conversation would be
 a. you would really have to define all the WordML you wanted in the XSL file. You could not use this approach to merge XML data with (say) a Word document containing a table that provided the formatting that the user wanted.
 b. at the very least, you would need someone with strong XSL skills to come up with a fixed piece of XSL that could work for an arbitrary XML data source. I do not have the XSL skills to tell you whether or not that can even be done, but my suspicion is that it could not. It would, I think, be slightly easier to generate an XSL for a given data source (e.g. given a sample XML file or perhaps an XSD). But that would typically be done using a procedural language such as VBA or VB.NET, which means you would be back in "you need VBA" territory.

It is possible to insert XML data into a Word document using an INCLUDETEXT field. With XML sources, you can specify some XML namespace mappings, some XPath, and an XSL transform. But I think you would still have the same problems I have described above. The documentation for this approach is sparse - I have never seen it used in any practical application, and although I have looked at it more than once in the past, off the top of my head I cannot remember whether you can actually transform a complete tree of XML nodes this way. AFAICR there is some problem with the way that the XPath behaves - either it can only actually return the text of a single node, or a text that concatenates the text of the selected nodes, i.e. it does not return a list of nodes that the XSL can then transform.

So I would suggest that that puts you back in one of the following scenarios:
 a. you have a mechanism that transforms your XML into a flat data source that you can process using the kind of facilities we have already discussed
 b. as (a), but the XML is a red herring, and you create the data source that you need more directly. I wondered, for example, whether you could dynamically create a Transact*SQL procedure that generated a table that you could then use as the merge data source.
 c. you use VBA or another procedural language to do at least some of your data insertion. On that front, there is one thing I haven't mentioned in any of our previous discussions, which is that the Range object in Word's object model has an InsertXML method that lets you insert a chunk of WordML into a Word document. What that might give you is the ability to examine a user-supplied table layout, generate the WordML code that reflects that layout, and then merge the XML data and WordML data. That's a pretty hefty set of problems, but it may be worth tossing the ideas at your C# guy and seeing if that sparks any ideas.

norman david jones say:

Hi Jeff,

You can use the Worksheet_Change event to obtain your desired results.

As an example, right-click the worksheet's tab | select View Code and paste the following code:

'==========>>
Option Explicit

'---------->>
Private Sub Worksheet_Change(ByVal Target As Range)
    Dim rng As Range, rngDV As Range, rngCondition As Range
    Dim Res As Variant
    Const sDVaddress As String = "A1:A10"                              '<<==== Change
    Const sConditionAddress As String = "B8:B10"                    '<<==== Change
    Const myValue = "X"

    Set rngDV = Me.Range(sDVaddress)
    Set rngCondition = Me.Range(sConditionAddress)
    Set rng = Intersect(rngCondition, Target)

    If Not rng Is Nothing Then
        Res = Application.Match(myValue, rngCondition, 0)
        With rngDV
            If Not IsError(Res) Then
                With .Validation
                    .Delete
                    .Add Type:=xlValidateList, _
                         AlertStyle:=xlValidAlertStop, _
                         Operator:=xlBetween, _
                         Formula1:="=myList"
                    .IgnoreBlank = True
                    .InCellDropdown = True
                    .InputTitle = ""
                    .ErrorTitle = ""
                    .InputMessage = ""
                    .ErrorMessage = ""
                    .ShowInput = True
                    .ShowError = True
                End With
            Else
                .Validation.Delete
            End If
        End With
    End If
End Sub
'<<==========

Alt+Q to return to the worksheet.

In my test workbook, I created a name myList which refers to a single column range elsewhere on the worksheet - in my case it is a list in cells D1:D10.

Now, when any of the cells in the range B8:B10 contain an X, the range A1:A10 will have a data validation list which is a named range myList.

Amend the data validation as required. If you are unsure how to amend the DV part of the above code for a different data validation scenario, either use the macro recorder to obtain the requisite DV code, or post back with your DV requirements.

===

Regards,

Norman

zukorl say:

list of abbreviations

What is the best way to select them from the text? Is there any tool in Word2011 to do it automatically?
Thanks,


daniel g. grau say:
I would apply a highlight to each abbreviation as I read the document, using the Text Highlight Color tool in the Ribbon's Home tab (see screenshot in Tech on the Net).

Once you've read through the whole document and you've highlighted all abbreviations, make a copy of the document and do the following global replacement:

  • Edit > Find > Advanced Find and Replace
  • Click on the Replace tab at the top
  • Place your cursor in the Find what field and make sure it's blank
  • Click on the disclose triangle to reveal the Format dropdown menu at the bottom
  • Select Highlight twice in a row so it becomes Not Highlight. This should add "Format: Not Highlight" under the "Find What" search field.
  • Click on the Replace what field and type ^p (or click on the Special menu at the bottom and select Paragraph mark
  • Click on Replace All

That will replace everything that is not highlighted by a Carriage Return, leaving you with a list of highlighted terms that you can copy and paste as the top of the original document. Finally, do Edit > Select All (or press Command A) and take the highlight off from the whole document.

Using font colors will also work, except that you cannot search for "not this color." Instead, you'll have to look for every other font color used in the document, although most times Auto will suffice.
