Word 2013 Headers/Footers not linking properly - Microsoft Community

yvonnecox1 say:

Word 2013 Headers/Footers not linking properly

In our Word 2013 documents, we need staff members to be able to insert various headers/footers created as "Building Blocks" under the "Quick Part Gallery" as needed.  Section breaks are inserted where needed and the Link to Previous option is set for the second page headers and footers to match the first page however when we insert headers/footers on the first page from the Quick Part Gallery, they are not duplicated on page 2, nor are we able to then repeat the insert header/footer process on page 2 using the same Quick Part Gallery command.  

Thanks in advance for any advice on addressing this issue. 

charles kenyon say:

Your question reflects an understandable lack of understanding of how headers and footers work.

Each section in Word has three headers and three footers. After the first section, these can be linked to previous. The link-to-previous setting is for each header and footer, each section can have six settings for this.

Header/Footer Settings Recap

You cannot link the main header to the first-page header using the link to previous. Previous refers to the previous section.

The insertion of headers/footers using building blocks is a crutch. Learn to actually create and edit them. The thing is that when you insert a header or footer using that, it replaces any existing content.

You may not need the section breaks. If the reason for the section break is to change the header and/or footer, look into using the three already in the section and using the StyleRef Field to change content in the header/footer to reflect contents of the page without actually changing the header or footer itself.


charles kenyon say:
(excerpt from Recap of Header/Footer settings)
  • Every Section will have three different Headers and three different footers even if you never see them. Because of this, and because there can be multiple Sections on a single page, it may be easiest to temporarily expand each Section to at least three pages (to see the First-Page, Even-Page, and Odd-Page Headers and footers. This way you can see what is in each Header/footer and what the settings are. You can temporarily expand the number of pages using manual page breaks (Ctrl+Enter)
  • Section formatting is contained in the Section break that follows the Section. If you delete all Section breaks you can find in a document your formatting will be contained in the last paragraph mark in the document, which is the last Section break.
  • If there is a Section break on a page, changes to the Header/footer that follow that Section break may not have any effect on the Header for the page that contains the Section break. (Those settings or changes are stored in the next Section break, the one governing that Section.) Whether changes to the next header change the previous one depend on the Link-to-previous setting of the next header.
  • If you have a Section break on a page, the header and footer for that page will be governed by the settings for the first Section on that page, even if that Section contains no text (i.e., the page starts with a Section break).
  • The Link-to-previous setting is specific to the type of Header/footer: Primary Header, First-Page Header, and Even-Page Header, Primary Footer, First-Page Footer, Even-Page Footer. Link to previous is the default setting.
  • Link to previous is specific to Headers and footers (can be different). It is also specific to each type of Header/footer. That is, the first-page Header can be linked to previous but not the first-page footer and not the even-page Header. The Link-to-previous setting can be set for each Header/footer in every Section after the first. This setting can be different in each type of Header / footer in each Section.
  • Since each Section has three different Headers and three different footers and the link to previous setting is specific to the type of Header or footer, each Section can have up to six different link-to-previous settings. This is true even if the Header/footer to which it applies is not visible. This is true even if the Section involved is a continuous Section in the middle of a page and has no Headers or footers displayed.
  • The different-first-page setting covers both Headers and footers and is a Section (not document) setting. This setting is carried over into a new Section started from a Section set for different-first-page.
  • Different odd and even covers both Headers and footers and is a document (not Section) setting.
  • See also: Quick Reference Card for Headers, Footers and Page Numbers
  • See also: Using Headers and Footers by Suzanne Barnhill, MVP - excellent and comprehensive web page
  • For anyone having trouble with this, I strongly recommend downloading the Word Toolkit for Advanced Management of Headers and Footers from Lene Fredborg's site. It was developed by Shauna Kelly and implemented by other Word MVPs especially Bill Coan.

why cant i update(reschedule uncompleted work ) activity with - Microsoft Community

rafiaplanning depatment say:

why cant i update(reschedule uncompleted work ) activity with finish no later than constraint ?

why cant i update(reschedule uncompleted work ) activity with finish no later than constraint ? knowing that the update date way before the finish no later than constraint
tom_bpc say:

Dale, I'm surprised you don't see this in 2016 release - it has been default behavior for each version of MSP that I have used, at least since 2003.  Perhaps this is a new feature since the 2010 version that I continue to use.

As I understand it, "Reschedule uncompleted work to start after:" imposes either a SNET constraint or a resume date on the affected tasks (the latter for in-progress tasks.)  If there is already a constraint applied (i.e. NOT ASAP), then Project will detect a conflict and decline to impose the new constraint.  After all, Project presumes that you have applied the existing constraint for a reason and would object to it having it over-written during a typical progress update.  I've observed the same behavior for all constraint types, not just Rafia's example of FNLT.  (Exception: Project WILL replace your SNLT constraint if it is less stringent than the new one.) This is just how Project is designed to work, and I learned to accept it some years ago.

This is one more argument for using a Deadline instead of FNLT constraint.

tom_bpc say:

Well yes, introducing resource leveling does complicate things, so Deadline apparently won't give you what you want. Instead I would suggest introducing a dummy milestone between the start of your constrained (FNLT) task and its predecessors, then have Project assign the updating constraint to the dummy milestone instead of your constrained task.
