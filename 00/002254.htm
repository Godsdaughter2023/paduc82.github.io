Common Excel performance issues and solutions - Microsoft Community

erik davidsson (msft) say:

Common Excel performance issues and solutions

1. Did your file work a lot faster in a previous version of Excel, such as Excel 2010 or 2013?

You may find that after you install Office 2016, your file works slower.

Possible causes and solutions:
-Try Excel Options – Advanced menu – disable Flash Fill.
-See the “Slow opening” section from this article, discussing why it may be slower to open the first time in a new Excel version. Excel always performs a Ctrl+Alt+Shift+F9 (Full Rebuild Recalculation) if you open a file that was last saved in a previous version of Excel. This is the expected behavior. Try saving it and reopening it, and it might be faster the 2nd time.
-Also see this which is quite similar: “The second time that you calculate a workbook is often significantly faster than the first time.” with comments such as this: “With multiple core computers, Excel … tries to optimize the way the calculations are spread across the cores based on the results of the previous calculation.”
-Also, files generally grow, as the normal way of working with Excel is that you add content rather than remove content. It is possible that the file you are looking at has significantly increased in complexity and size since you opened it the last time on an older version of Excel. You may want to verify that the exact same file is still faster in the older Office version.
-If you are specifically having performance issues with VBA code that Protects and Unprotects worksheets thousands of times, and notice a clear performance degradation in newer Office versions, this is the expected behavior. For security reasons, protecting sheets uses stronger encryption in Excel 2013 and forward.
-If it indeed was faster in the older Office version, but none of the above applied, you may want to proceed with this article to clean up the file. If that also did not help, you may want to see the official Excel 2013/2016 performance article, or open a support case.

2. Did your file work faster before you installed updates for Office 2016?

It may be a temporary issue. You may want to roll back to a previous update level for the time being. You may also want to open a support case to check if it is known.



3. Is your file a VLOOKUP database type of file with more than 100 000 rows?
If so, there may be things that can improve the performance, but please note that it is always a possibility that no amount of optimization is going to bring you towards the performance you would find acceptable.


Examples of different issues you may see in this scenario:
-Very often, going from e.g. 40 000 rows to 50 000 rows you may think the performance of the file would be roughly the same. But with lookups, it can sometimes make the file e.g. 4 times slower, since the type of lookup complexity in these files can cause exponential performance decreases. This may trick you, as you think the design of the file is good at first, but as the file grows suddenly reach a file size where it becomes obvious that Excel is not a database application.
-The formula complexity may hit a point where Excel runs out of memory, and either gives an error, freezes, hangs, or takes very long.

What you can do in this scenario:
-Evaluate if 64-bit Excel improves the situation. If you have no 64-bit Office available for a test, you can check for peaks up towards 3 GB memory using Process Explorer, that has a Total Memory column. Please note: Task Manager is not measuring Total Memory!

-Make sure you use the fastest formula possible. See section "VLOOKUP vs. INDEX and MATCH or OFFSET" from this article. As for formulas that can help you evaluate the optimal performance, range.calculate can be useful.
-Limit the lookup range to only the necessary range, read more in the “Formulas” section here.
-Perform the other steps from this article.
-Set the calculation to manual, and limit the amount of calculation done in the sheet to when it is absolutely necessary.
-If nothing helped, you may have to accept that Excel might not be the optimal tool for you. While Excel is well known by most and usually has the technical capabilities, it does not always have the performance capabilities. You may consider if Excel is really the right application for this, or if a proper database application such as Access can be more suited for your needs.

4. Is your performance issue related to running VBA code?
Unless it is covered by Step 1 above (meaning it was faster in a previous version of Office), here are some things you can try:
-Look into ensuring that none of Excel's inner workings are being run while your code is running. Even the smallest performance hit can be devastating if it is repeated 10 000 times in a loop. E.g. certain formatting such as “Wrap Text” or “Merge Cells” will introduce a performance hit that is hardly noticeable, but if the cell with that is looped through thousands of times, it may be enough to render the macro incredibly slow.

-Selecting objects, copy pasting, and referring to worksheet data, are general things you also want to avoid.

-Even better (if you are up for the task of redesigning your code) is of course to avoid code loops altogether.

-A general file cleanup can be a good idea.

5. Does your Excel seem to hang forever, exactly when you Undo a Paste?
There are many performance issues that can arise from having too many Conditional Formats, and this is maybe the most common one.

Solution: Remove the superfluous Conditional Formats in your sheets. Step 6 in this article discusses how to.

6. Does your Excel hang when you try to Insert or Delete a row or a column?
If so, you may first want to check if removing your filters solves this. The reason is that when you have filters applied, some rows are hidden. Excel treats each of them as a "section". When e.g. inserting a row, it needs to move everything one step down. Moving a huge amount of sections down is heavier than moving just a single section down in a single operation.

If this does not help, this can also happen because your sheet's boundaries are much further to the right and down than they need to be. Try pressing Ctrl+End in each sheet, to see what cell it jumps to. If this is very far down or very far to the right, it could explain this. The problem is that insert a row can be a surprisingly heavy operation, as it has to move everything below it one step down/up.

Typically, if you have this problem, you may see that when you press Ctrl+End in the problem sheet, it jumps to column "XFD" (the 16000th and last column) and/or a row number higher than 10000. It can also depend on the amount of content or formulas how heavy this "insert" or "delete" will be. The key factor is how many rows/columns you have after the row you try to insert/delete, until the last row/column of the sheet.

Please note that the last cell Excel jumps to when pressing Ctrl+End may not always be the one that you expect.

Solution: Please reset the last cell in the sheet, save your file and restart Excel, and try pressing Ctrl+End again and verify that it does not jump down so far, before trying to insert the row again. It may not sound like a complex operation to insert a row, but when you do, Excel will need to copy everything one step down, and if it is a huge sheet copying can be a very time consuming operation.
(If you have no luck with resetting the last cell in Excel since you have so much content that Excel hangs, saving the file as .XLS temporarily, restarting Excel and saving it back to .XLSX could be a workaround, since .XLS only supports 65k rows (and 256 columns), so saving in this format will effectively clear row 65k until row 1 million instantly.)

If you are of the opinion that this step applies, but you have reset the last cell to something small already, and even removed everything on the sheet with Home - Clear - Clear all, and it still is slow, you could try removing the other worksheets. There may be very many formulas referring to this sheet that cause the issue, as they need to be re-evaluated and changed when you insert/delete a row, and all formulas must be updated to still point to the correct location.

7. My file is actually only slow to open when I have enabled (legacy) sharing
First of all, please note that shared files can be much slower to open than non-shared files, so expecting the same performance may be unrealistic.
-Change some settings for sharing, such as View tab – Custom views – delete all unneeded ones, and open Review – Share Workbook – Advanced tab, and minimize the sharing settings here.
-Test another server. There may be issues with SMB2/SMB3 on your server. See this.
-Follow Step 1 from this article to save it as .XLSB instead in particular, and then follow the rest too to clean it up.

To go back to the Excel performance article series, click here.

Hi Terrance,

The problem here is that Microsoft Word is a Word Processor, not a page layout program. While word processors break your document into pages, it is done on the fly unless you insert page breaks everywhere (not recommended). Page layout programs have a concept of a page and scale contents to fit.

The problems you describe are problems encountered from one computer to another when using word processors. PC vs Mac is not usually a big factor.

Doug pointed out that Word checks to see what printer is active and formats your document accordingly. When you open the document on your Mac, go to File > Page Setup. Notice what printer the document is formatted for. If the printer is different (brand, model, or both) from the computer that was used to make the file, this is where most of the formatting difference is likely to come from. Choose Format For: Any Printer to reduce problems caused by having different printer drivers in play.

The other likely source of layout difference is your fonts. Fonts come in versions just like other software. On your Mac, use Font Book app to discover the version of your font. Here's the version of Times New Roman that's active on my Mac at the moment:

    Unique name    Monotype:Times New Roman Regular:Version 5.05 (Microsoft)
    Manufacturer    The Monotype Corporation
    Designer    Monotype Type Drawing Office - Stanley Morison, Victor Lardent 1932
    Copyright    © 2008 The Monotype Corporation. All Rights Reserved.

Different versions of fonts having the same name will cause layout differences. Ensure that all machines, whether Mac or PC) are using the same brand and version of Times New Roman (or whatever fonts you are using). Even if you format your document for any printer, you still must ensure that the font is the same on all computers if you want to have the same layout.

Right so we've found what the issue was, for us at least, and it was related to the Microsoft Exchange Add-In as mentioned in my previous post. 

In the ProcMon results the Outlook Exchange Add-In result came up as Access Denied. So I investigated the affected laptops and they've come up with the Exchange Add-in being inactive or disabled. Unfortunately in our case simply enabling the add-in didnt work. So I did the following

  • Run Outlook as Administrator
  • Disable the Exchange Add-In (it would be enabled in admin mode)
  • Close Outlook and then run Outlook as the user
  • Enable the Exchange Add-In
  • Close Outlook and open it again as the user to confirm that the Exchange Add-In is still active
  • Close Outlook and restart the computer
