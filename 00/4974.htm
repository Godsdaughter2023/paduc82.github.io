On-Premises AD Sync with AAD - Microsoft Community

tek-nerd say:

On-Premises AD Sync with AAD

Hi,

We have On-Premises domain "mydomain.pri" and Office 365 E3 licenses with verified domains "mydomain.net" and "mydomain.com (Public DNS) and mailboxes were migrated from 1and1 to Office 365. Now we would like to use AAD Connector to sync On-premises AD accounts to Offcie 365, so that users can use SSO.

Can we sync on-premises ".pri" domain accounts to ".com" and ".net" Office 365 domains? If so, what is the best practice to sync?

Thanks in advance.

adam drayer say:

Whatever you put into the proxyaddresses or UPN attributes on-prem will sync (overwrite) what is in Office 365/Azure.  If any of the addresses are non-routable (not internal only and not verified), they will be transformed into "<tenant>.onmicrosoft.com".  The primary email address will be set to whatever is on-prem in ProxyAddresses that has an uppercase "SMTP".  The secondary addresses have lower-case smtp.

If neither the primary smtp or upn for the on-prem environment match the Office 365 user, then syncing will not work and Azure AD Connect will just provision a new user in office 365.  There's a video on YouTube regarding how Azure AD Connect matches users to Office 365:  https://youtu.be/8m00k3AU-HA.  Disclaimer: this is my video.

And yes, you are correct that if you try to synchronize a user that is already using Exchange, you should make sure their cloud email address is in the on-prem proxyaddresses field.

If you are using Azure AD Connector, there are 2 ways you could test this.  One is to run the server in staging mode.  This will do import and sync only.  That let's you see what will be in the outgoing connector to Azure AD without actually writing to it.  Secondly you can use the "Preview" button to simulate a What-If.  1 Turn off the scheduler, 2. make your on-prem changes, 3. perform manual imports on both connectors.  4. Run a preview without commiting changes to see what would flow through to the AAD connector.  If it is not what you want, you can revert your changes, run another import, and then turn the scheduler back on.  

Hopefully, all that helps a little.
