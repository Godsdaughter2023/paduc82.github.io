jim gordon mvp say:
Phillip gave you half the answer you seek.

As Phillip pointed out, you can set the default colors for shapes within a workbook by right-clicking on a shape that has the formats you desire and choosing Set as Default. 

The other half of the solution is to save your workbook as your personal default workbook. So the whole process is:
  1. Open a new, blank workbook
  2. Add a shape
  3. Adjust the colors you want to use as default
  4. Right-click the formatted shape then choose Set as Default Shape.
  5. Delete the shape
  6. Save the workbook as your personal default workbook

To accomplish step #6 follow these instructions provided by JE McGimpsey:

In your Startup folder (by default:

    HD:Applications:Microsoft Office 2011:Office:Startup:Excel:

but you can change it in Preferences) create a new workbook with all your default specifications (e.g., zoom, # of sheets, fonts/styles, etc). Save the workbook AS A TEMPLATE (.xltm) with the name "Workbook" (no quotes). In the Finder, navigate to that folder, select the file and choose Get Info from the right-/Ctrl-click popup menu, and delete the extension (.xltm) on the file. XL will now use that workbook as a default.

Create a one-worksheet workbook template in the same way and save it with the name "Sheet". Delete the extension in FInder, and that workbook's worksheet will be used for inserted worksheets.

ken sheridan say: You could amend the SQL statements to compare all columns, but the simplest solution would be to replace all rows which match a row in the imported table on ProjectID.  So, having emptied the holding table with the DELETE statement and then having imported the Excel data you'd then delete the matching rows from the operational table with:

DELETE *
FROM [OperationalTable]
WHERE EXISTS
    (SELECT *
     FROM [HoldingTable]
     WHERE [HoldingTable].[ProjectID] = [OperationalTable].[ProjectID]);

Then insert all rows from the holding table with:

INSERT INTO [OperationalTable]
SELECT *
FROM [HoldingTable];

The latter statement above would replace the rows deleted by the first, and add any new row previously not in the operational table.  The code to do all this would be something like this:

Dim strSQL As String

' import rows into holding table:
DoCmd.TransferSpreadsheet _
        TransferType:=acImport, _
        TableName:="HoldingTable, _
        FileName:=F:\Spreadsheets\NameOfSpreadsheet.xlsx", _
        HasFieldNames:=True

' empty the holding table
strSQL = _
"DELETE * " & _
"FROM [HoldingTable]"

CurrentDb.Execute strSQL, dbFailOnError

' delete all matching rows from the operational table
strSQL = _
"DELETE * " & _
"FROM [OperationalTable] " & _
"WHERE EXISTS " & _
    "(SELECT * " & _
     "FROM [HoldingTable] " & _
     "WHERE [HoldingTable].[ProjectID] = [OperationalTable].[ProjectID])"

CurrentDb.Execute strSQL, dbFailOnError

' replace deleted row in operational table and insert new rows
strSQL= _
"INSERT INTO [OperationalTable] " & _
"SELECT * " & _
"FROM [HoldingTable]"

CurrentDb.Execute strSQL, dbFailOnError

It's important that ProjectID is the primary key rather than an autonumber, to preserve the relationships with ant referencing tables, e.g. a ProjectEmployees table.  If you do have such  tables be sure that the relationships with the operational table do not enforce referential integrity.  Normally we'd advise that relationships should be enforced, but in this case it would prevent you deleting rows from the operational table.  Ideally you should drop enforced relationships before running this code, and then recreate and enforce them afterwards, which can be done in code, but I don't want to confuse you with too much at this stage.

To replace the foreign key values in such tables, let's assume a ProjectEmployees table is currently structured like this:

ProjectEmployees
….ProjectID
…EmployeeID

where ProjectID references the current autonumber primary key, ID, of the operational table, and EmployeeID references the primary key of an Employees table.  Firstly you should add a new column named ProjectID_New, of the same data type as ProjectID in the operational table, so the struicture is now:

ProjectEmployees
….ProjectID
….EmployeeID
….ProjectID_New

Then create and execute an UPDATE query like this:

UPDATE [ProjectEmployees] INNER JOIN [OperationalTable]
ON [ProjectEmployees].[ProjectID] = [OperationalTable].ID
SET [ProjectEmployees].[ProjectID_New] = [OperationalTable].[ProjectID];

To create the query open the query designer, switch to SQL view, and paste in the SQL statement in place of what's there.  You'll need to change the table and column names to your actual names of course, in this and all the other SQL statements.  Execute the query with the Run icon on the Design ribbon.  

Do the same for any other tables which reference the operational table.  Once you are satisfied that the [ProjectID_New] columns in the referencing tables have been populated correctly, you can delete the original ProjectID from each of the referencing tables (not the operational table of course), and rename the ProjectID_New column as ProjectID in each case.  You'll then be able execute the import code in a button's Click event procedure without losing the relationships between rows in the operational table and any referencing tables.

The asterisk character in a query does mean 'all columns' BTW.
