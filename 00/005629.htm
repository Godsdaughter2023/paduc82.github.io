ronrosenfeld say: As written:  =MyFunc(A1:B1,D1:E1)  you are passing two arguments to your function which is only set up to accept a single argument.

One solution is to use ParamArray.

Another, if you want to pass a multiple range argument to your function, as written, you need to enter the formula:

=MyFunc((A1:B1,D1:E1))

Note that the multirange itself is enclosed within parentheses.

And take a look at this for some further insight:

Option Explicit
Function MyFunc(DataRng As Range) As String
    Debug.Print DataRng.Areas.Count, _
        DataRng.Areas(1).Address, DataRng.Areas(2).Address
    MyFunc = "Dog"
End Function

delete query - with joined tables - Microsoft Community

trinakropf say:

delete query - with joined tables

I am trying to create a delete query where data is deleted from one table (Last) if it is found in the other table (tblLastExcpetions), using DevStyle to join the two tables.  The below SQL works if I'm viewing that data in Datasheet View, but get an error message of "Could not delete from specified table" when trying to run delete query.

DELETE Last.*
FROM Last INNER JOIN tblLastExceptions ON Last.DevStyle = tblLastExceptions.DevStyle
WHERE (((tblLastExceptions.DevStyle)=[Last].[DevStyle]));

ken sheridan say: Is there a relationship between the tables, and if so, is Last the referenced (one-side) table in the relationship, and is referential integrity enforced?  If so you will be unable to delete rows from Last until the matching rows in tblLastExceptions have been deleted.  This can be done automatically by selecting the 'cascade deletes' option in the relationship dialogue.  This might not be appropriate in the context of your database, however, in which case referential integrity should not be enforced in the relationship.

An alternative to a JOIN, and the approach I'd generally use, is:

DELETE *
FROM Last
WHERE EXISTS
    (SELECT *
      FROM tblLastExceptions
      WHERE tblLastExceptions.DevStyle = Last.DevStyle);

Sender IP (212.212.212.212) is our On premise exchange server external address.

domain.mail.onmicrosoft.com is our exchange online domain.

Is this by design that our On-premise exchange server will be seen as sender?

We are getting more than enough spoofing emails that are directed to our CEO and FInance director and adding SPF record doesn't seem to help.

I know DKIM and DMARC should help better against spoofing, but currently we cannot implement them, since our MX record does not point to EOP.

young yang msft say:

Hi Off2w0rk,

 

When external users send emails to your users, this mail flow has no relationship with your own domain’s Office 365 SPF record. Office 365 will check sender’s SPF record.

 

When your Office 365 users send emails to external recipients, your own SPF record will be checked by external mail system.

Young
