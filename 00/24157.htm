How do I create multiple prices for product tables in access - Microsoft Community

cwm321 say:

How do I create multiple prices for product tables in access

I am new to Access and am developing an Access 2013 database to run my business, and need some assistance.  I sell hundreds of products that routinely have different costs to me, different profit margins for different quantities, and subsequently prices to my customer based upon the quantity of the item that are purchased, and need to set up my product table, and queries and order form.  

I need to: 1) store the item costs based upon the quantity to be purchased; 2) sore the markup % for each quantity range; 3) calculate the sale price based upon the quantity and profit margin; and 4) finally enter the calculated price in an Order Form.

Example:

Widget XYZ costs me $20.00 for quantities of 1 to 50, for which I apply a markup of 50% to get a Sale Price of $30.00 each;  quantities of 51 to 100 are Costing $15.00 each with a markup of 40% and a Sale price of $21.00; 101 to 200 are costing $12.00 each with a markup of 30% and Sale price of $15.60 and so forth.  

Widget ABC has different costs for each with different unit price breaks and different markup percentages that Widget XYZ.

Can anyone give me some guidance as to the best way to set up the table(s) and query(s)?   Thanks in advance?

ken sheridan say:

You need to model this with a related table like this:

MarkUps
....ProductID  (FK)
....LowerQuantity (Integer Number)
....UpperQuantity (Integer number)
....MarkUp  (Single precision floating point number)

The table has two composite candidate keys ProductID and LowerQuantity, and ProductID and UpperQuantity, so one of these can be made the primary key, and in the case of the other the two columns should be contained in a unique index.  The foreign key ProductID will normally be a long integer number referencing the primary key of the products table, usually a autonumber.  The MarkUp   values should be represented as fractional values, e.g. 0.5 for 50%.

You can then join this table to PurchaseOrderDetails and Products tables for instance to return the SalePrice, e.g. for a read-only query:

SELECT PurchaseOrderDetails.OrderID, PurchaseOrderDetails.ProductID,
PurchaseOrderDetails.Quantity,
PurchaseOrderDetails.UnitPrice * (1+MarkUps.MarkUp) AS SaleUnitPrice
FROM PurchaseOrderDetails INNER JOIN MarkUps
ON PurchaseOrderDetails.ProductID = MarkUps.ProductID
AND PurchaseOrderDetails.Quantity >= MarkUps.LowerQuantity
AND PurchaseOrderDetails.Quantity <= MarkUps.UpperQuantity;

Or for a less efficient but updatable query for use as the RecordSource of a PurchaseOrderDetails subform in a PurchaseOrders form, calling the DLookup function:

SELECT OrderID, ProductID, Quantity, UnitPrice AS PurchasePrice,

UnitPrice * (1+DLOOKUP("MarkUp", "MarkUps","ProductID = " & [ProductID] &
" AND " & [Quantity] & " BETWEEN LowerQuantity AND UpperQuantity")) AS SaleUnitPrice
FROM PurchaseOrderDetails;

ken sheridan say: PS:  if the mark-ups per product and/or quantity ranges might be changed over time you should not return the sale price in a computed column in a subform's RecordSource, but assign the value to a SaleUnitPrice column in the PurchaseOrderDetails table by calling the DLookup function in the AfterUpdate event procedure of the Quantity control in the subform.  The selection of a product in the ProductID control should also be enforced before a quantity can be entered by means of code in the Qunatity control's BeforeUpdate event procedure, setting the return value of the procedure's Cancel argument to True if the ProductID control is Null.

scottgem say:

I would go a somewhat different path then Ken's suggestions. As you indicated the cost is tied to the quantities.  So I would have a table like this:

tblPricing

PricingID (PK Autonumber

ProductID (FK)

LowerQuantity (Integer Number)

MarkUp  (Single precision floating point number)

I would then have a query like this:

SELECT ProductID, LowerQuantity, UpperQuantity, UnitPrice, MarkUp, UnitPrice*(1+Markup) AS SalesPrice

FROM tblPricing

ORDER BY ProductID;


I would name this query qryPricing. I would include a field in your Order Detail record for SalesPrice. 

From there you have a few ways of populating SalesPrice when you create an order. One way would be to have a combobox to select the product AFTER you enter the quantity. The RowSource of this combo would look like this:

SELECT ProductID, ProductName, SalesPrice

FROM tblProducts, qryPricing

ON tblProducts.ProductID = qryPricing.Product ID

WHERE LowerQuantity < Forms!formname!txtQty AND UpperQuantity > Forms!formname!txtQty;

The combo would then show all product pricing that matches the entered quantity. In the After Update event of the combo I would have code like:

Me.txtSalesprice = Me.cboProduct.Column(2)

This will populate the control bound to the SalesPrice field in the Order Detail table with the sales price for the selected product.

Another Option would be to use a DLookup to pull the Salesprice from the query after the Product and quantity are selected.

Me.txtSalesPrice = DLookup("[SalesPrice]", "qryPricing,"[ProductID] = " & Me.cboPricing & " AND (LowerQuantity < " & Me.txtQty & " AND UpperQuantity > " & Me.txtQty & ")")
