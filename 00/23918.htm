Susie - you have a nasty dilemma.

There is no way you can import the data and confidently parse it into nice clean access tables.

you have our permission to throttle the excel creator.

it is easy to link the table to access - but that assumes it is already in a table format in excel. I suspect that may not be the case.

Brute force will still need to be employed. If you pre-clean the excel SS and have it in a table format then you can do the link or import it.

I would be inclined to do most of the brute force "fixing" in excel - where you can run formulas over the columns of data and see the erroneous results for - say - the name parsing and fix it manually.

I have a similar problem importing lists of student names for supply of garments. I will often get several people share the same name.

So is Kelly Burke one person or 2 with the same name.

when doing the brute force fixing before importing - you need a way of telling the duplicates.

in the excel - apply an ID column - so that each name listed gets a unique number - then for names that double up - you decide if its the same person. if it is then change one of the ID fields to match the other. If they are different - the ID field will identify as being different people.

in Access have a table to list ALL people - using the ID code as the key. include a status field to denote what they are - eg parent/ adult / girl guide.

Next - have an activities table to hold what each person does - and the year (period) that they do it.

Then at the end or start of the year you can have a cloning routine that allows you to copy the current activity into one for the new year.

Remember - you are dealing with people - very unpredictable behaviour - not suited to clean automatic DB processing . . .

scottgem say:

You can't really convert an Excel workbook to an Access database. A relational database is not a more powerful version of a spreadsheet, its a different entity.

What you need to do is design a relational database from the ground up and then copy your data into it. This can be done by linking to the Excel sheet and running Append queries to get the data into the proper tables.

Fortunately for you, there are a number of examples of membership type databases out there so you don't have to start right from scratch. I would suggest downloading a looking at a few to see how your database should be designed. You may even find one you can modify.

I would also imagine, you aren't the first person with this dilemma. I would suspect there are forums where other girl scout leaders can exchange idea and you may find someone who has already built an Access database for their council.
