That's the whole point of IMAP and Exchange. All your clients are sync'd. You don't have to jump through hoops to leave things on the server when a client grabs it, like you would with PoP3. jllatham say:

That's the classic .Find/.FindNext code right out of Excel 2003 help - and it's a pretty stable routine, I've used it many times.

 

What error is it giving you at the Loop While Not line of code?  The Object Not Set??

 

If that's it, change the code to read:

 

        Do
          c.Value = 25
          Set c = .FindNext(c)
          If c Is Nothing Then Exit Do
        Loop While Not c Is Nothing And c.Address <> firstAddress

 

Not unusual for them to publish something that really doesn't quite work - although as long as you don't actually change the value of c, it works fine.

 

What happens is that it finds "LR", then changes that to 25 and repeats that until it's replaced all LRs with 25, at which point it can no longer find LR and so can't assign a new location for c and can't stop at the firstAddress to indicate it's wrapped around.  So we have to catch the fact that c wasn't set to another cell before this all tries again and fails.

 

 

In theory it should work the way originally written, but oddly enough it no longer does.  How odd...

gregor weihs say:

Hello,

this wrong behavior is another nasty effect of the Windows dpi scaling functionality. The dpi setting is being abused to make screen fonts and elements appear in whatever size people find comfortable. This is complete nonsense, of course.

The dpi value is the pixel density of the screen. In order for an A4 page to show with the correct size (at 100% zoom) it needs to be set to the physical dpi value of the display. For high-resolution displays this results in ridiculously large fonts and window borders.

I have search long and wide for a solution for this problem and would be happy to learn that there is one. I think Windows is really broken that way.
