a. user say:

Re: "Slow? file in Excel 2013 - who can help me out"
  

The file in question crashed xl2010.

I looked at it in xl2003 and found...
  

1. All the cells, in the used range, on each sheet had borders

2. Each sheet had every cell colored white.

3. There were huge merged areas on every sheet.

4. The number of objects on each sheet varied from ~400 to ~21,000. (A total of ~62,000 for the workbook)
 

You can get by with that sort of formatting in xl2003 as it has 65,000 rows x 256 columns.

Ribbon versions of XL have 1,000,000 rows x 16,000 columns - do the math.

Removing the above **** reduced the file size from 13,500 kb to 224 kb and it nows opens in xl 2010.

The VBA code below will clean up the the workbook in a few seconds.

The workbook owner will have to redo a few things - so run the code on a copy of the workbook. Suggestion: the check boxes appear to be cosmetic - there are several fonts that can do that instead.
  

'---

Sub CleanItUp()

'Jim Cone - May 2014

Dim wsSht As Excel.Worksheet

For Each wsSht In ThisWorkbook.Worksheets

wsSht.Cells.UnMerge

wsSht.DrawingObjects.Delete

wsSht.Cells.Interior.ColorIndex = xlColorIndexNone

wsSht.Cells.Borders.LineStyle = xlLineStyleNone

wsSht.Rows.AutoFit

wsSht.Columns.AutoFit

Next

End Sub

'---

Search engines such Bing uses algorithms depending on the following factors:

Ranking Queries - This is based on the relevance and most number of searches users indicate depending on the keywords they use.

Geographic location - Though you might be using a global search engine, results can be tailored to your geographic location. The search engine can find out the IP address and personalize the result based on location.

Web crawler and indexing - Web crawlers discovers public web page and goes into every link and bring data it gathers back to Bing servers. You're going to need to be much clearer about what exact help you are seeking. You're going to a typesetter? I'm having a flash back moment to pre-computer days. Are you referring to a Desktop Publisher or a Print Shop?

Pages... there is no such thing in Word. At least not from a static physical sense. bob jones aka: cybertaz say:

I agree with Rich -- What you need really isn't clear...

As for point 1: That's a Print operation, Word really doesn't have much to do with it. Take a look in the Print dialog at the option pertaining to Page Range. Follow the example there to indicate what individual pages & page ranges are to be included in any given print job. The procedure is the same regardless of output device. If you've used Word's page numbering features in Headers or Footers they will be included in the output.

There really isn't much you can do to have page numbering selectively imposed as a part of the body content while the print job is being executed. The only alternative would be to write a very complex procedure which would insert PAGE fields on each page being printed, but inserting them into the text would change the text flow of the output... And trying to have them added as a part of the print process quite possibly cannot be done.

As for point 2: Rich is referring to use of Word's Caption feature to number illustrations, figures, etc. Used correctly they automatically update when new Captions are inserted. See the Word 2011 Help article Use captions . If the numbering has been typed manually, trying to get it to update programmatically is a challenge even an experienced coder would find challenging.

Also, the keystroke Rich mentioned is valid, but the UpdateFields command can also be executed by using the F9 key. You may have to use fn+F9 depending on your keyboard type & OS X keyboard preferences.

jllatham say:

IF (big word) I understand you correctly, you want to get some information through a UserForm and save that to be used in other portions of the VBA code to affect how that code works or it's results.

 

I'm going to assume that this information from the UserForm are values - either numbers, dates, text, perhaps references to cell locations.  That type of thing, but not actual instructions.

 

Here's one way to handle it:  Add a worksheet to your workbook, when I do this kind of thing, I usually name it SysUse - eventually you will make that sheet VeryHidden through the VBA Editor and properties section in it.

 

You get your data from the user into the UserForm and on it you have a "change the data" button and what it does is take the information from the UserForm and stores it in cells on that SysUse sheet.  Then the other parts of the program can look on that sheet in the proper cell location for the value(s) it needs.

To move the value from the UserForm to the worksheet, the code would look something like this:

Worksheets("SysUse").Range("A1") = Me.TextBox1.Text

or for a number, perhaps

Worksheets("SysUse").Range("A2")= Val(Me.TextBox1.Text)

 

To clarify things you might consider declaring a bunch of Public Const values in a code module to refer to those cells on the SysUse sheet so you can better understand things in your code later.  Like this

Public Const UserAgeCell = "A1"

Then back in the UserForm process you could do something like this:

Worksheets("SysUse").Range(UserAgeCell) = Val(Me.TextBox1.Text)

 

When you need to use it in other code, you can pull it from the SysUse sheet and put it into a variable for use:

Dim CurrentAge As Integer

 

CurrentAge = Worksheets("SysUse").Range(UserAgeCell)

'calculate something based on their age

FudgeFactor = CurrentAge * 1.324

 

Does this seem to even be close to what you're looking for?

Remember that whatever you put onto the SysUse sheet is going to be 'remembered' when you save the workbook.
