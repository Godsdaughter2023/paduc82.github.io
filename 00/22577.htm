a. user say:

I have never quite got to the bottom of this, but 
 a. Just to create even more confusion, Word (and possibly Windows) refers to the digits used in Arabic as "Hindi", and sometimes refers to the digits used in Hindi as "HindiArabic"). This all stems from the unfortunate fact that the digits in Latin scripts (as in English) came from Arabic, and so were named Arabic even though their shapes were changed. This leaves us with no good Word for the digits actually used in Arabic. I think the use of the term "Hindi" to describe Arabic digits is recent and very unhelpful.
 b. There are also other digit systems used in Arabic, known in Word as ARABICABJAD and ARABICALPHA - you can apply an \*ARABICABJAD switch to a field such as the { PAGE } field and get that formatting. But you can't apply a formatting switch that applies the formatting you want.
 c. So what you have to do is go to Word Options->Advanced->Show Document Content and set the "Numeral" option to "Hindi", "Context", or perhaps "System". 
 d. What you should then see is that some numerals in your document appear as Arabic (i.e. as used in Arabic). If you use "Context" then Word seems to base which digits it uses on a set of rules that I do not fully understand. FOr example, if you have a { PAGE } field in a header, it will probably be a "Latin" page number if it is in a Left-to-Right paragraph, but an Arabic (as in Arabic writing) if it is in a Right-to-Left paragraph. But some other field types are only displayed with Arabic (as in Arabic writing) need to be entered while the keyboard is set to Arabic.
 e. Also, this setting is not made in the document. So if you send your document to a person who has the setting as "Arabic" rather than "Hindi", they wil see the "Latin" numbers in many cases.
 f. The "System" option refers to the settings in the Regional and Language Settings in the Control Panel - it is possible to specify a set of digits in there and specify how they are used. I'd have to research that further to understand how it really works and how it interacts with Word, but again, the setting will be stored in Word, not the Word document.

I hope that helps you to make some progress, but if you do find a better way, or even more info., please post back.

Oh yes, in Word 2007, the settings for "proofing and editing languages" can be accessed via Word Options->Popular->Language Settings... button. There is also a separate program that you can start via the Windows Start button. I don't think you will need to use either of those features to do what you need. Please not that changing some things in there can destroy your Normal.dotm template and any customisations and macros in there. I think that only happens if you change the "Primary Editing Language", but I prefer to locate and back up my Normal.dotm file before changiing anything in there.

Info only

Greg Wilson years ago wrote some event code that handled autofit on merged cells with wrap text and line breaks, although cells with line breaks get extra height when autofit.

For your perusal. . . . .

Private Sub Worksheet_Change(ByVal Target As Range)

'merged cells must be preset to Wrap Text

    Dim NewRwHt As Single
    Dim cWdth As Single, MrgeWdth As Single
    Dim c As Range, cc As Range
    Dim ma As Range
    Set ma = Range("C3:F20")
    With Target
        If .MergeCells And .WrapText Then
            Set c = Target.Cells(1, 1)
            cWdth = c.ColumnWidth
            Set ma = c.MergeArea
            For Each cc In ma.Cells
                MrgeWdth = MrgeWdth + cc.ColumnWidth
            Next
            Application.ScreenUpdating = False
            ma.MergeCells = False
            c.ColumnWidth = MrgeWdth
            c.EntireRow.AutoFit
            NewRwHt = c.RowHeight
            c.ColumnWidth = cWdth
            ma.MergeCells = True
            ma.RowHeight = NewRwHt
            cWdth = 0: MrgeWdth = 0
        End If
    End With
End Sub
