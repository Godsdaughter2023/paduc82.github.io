A single quote in a csv filename lets Excel repair the XLSX file after - Microsoft Community

maninweb say:

A single quote in a csv filename lets Excel repair the XLSX file after saving and reopening

Hi there,

may be the following has already been reported and/or is already in your databases as I could reproduce this also in older versions of Excel down to Excel 2010. I searched on MS Anwers, if someone has already posted it, but I could not find matching threads. I hope I did not oversee something. For this test, I used Excel 2016 Version 1702 Build 7870.2020.

I created a simple textfile and saved it as CSV file. The file includes just 3 rows with some dummy data. After then, I copied my file within the Windows Explorer and renamed the copies to the names shown in the following screenshot. You see, the filenames include a single quote in their names, one file is 31 characters long and another file has more than 31 characters in its filename.

After creating the CSV files, I opened them all in Excel. Excel recognizes my list separator and automatically creates a sheet name based on the filename of the CSV file. This is how it looks in Excel 2016.

I saved each file as XLSX file and also closed the files. After then I reopened the XLSX files in Excel. For some files, Excel had to repair them, as the sheet names were invalid. The files affected are 'ABC.csv and ABC'.csv.

So the problem is that, if a CSV file includes a single quote at the beginning or at the end of its filename, Excel does not recognize this on opening the CSV file and sets the sheet name to the full filename. When saving as XSLX, Excel does also not inform the user about the invalid sheet name. Logically, when reopening the XLSX file, Excel then correctly recognizes the faulty sheet names.

When opening the CSV file with a filename with more that 31 characters in it, the sheet name is shortened up to 31 characters. So, no problem there after saving as XLSX and reopening the file. However, if you rename for example 123...789_XYZ.csv to 123...789_'YZ.csv you will have the same problem as a single quote then becomes the last character.

My suggestion would be to inform the user about the invalid sheet name on saving the file as XLSX or remove the single quote(s) from the generated sheet name in a manner that the sheet name becomes valid.

If this is a model of your code, you need to call .Update ...

    rsRX.Edit

    rsRX("something") = GetCost()

    rsRX.Update

    rsRX.movenext

Loop

 

If that is not a model of your code, it is often more helpful to the volunteers helping you troubleshoot to see your actual code.

---

Also, when updating data like this, you accumulate your locks quicker than you can release them.  The default MaxLocksPerFile is 9500, so you can adjust that to a higher number for your session with ....

 

DBEngine.SetOption dbMaxLocksPerFile, 50000

 

But, you may still run into the same issue with such a large recordset.  So, I would suggest that in addition to increasing the locks for your session of Access, put a "pause" in your loop in order to let the release of the locks catch up.

 

Dim dtTimerExpired As Date

Do While Not rsRX.EOF
    rsRX.Edit

    rsRX("something") = GetCost()

    rsRX.Update

    If rsRX.AbsolutePosition Mod 9000 = 0  Then 'Set the value for less than your MaxLocksPerFile

        dtTimerExpired = DateAdd("s", 2, Now())

        Do Until Now() > dtTimerExpired

            DoEvents

        Loop

    End If

    rsRX.movenext

Loop

 

Also, you may want to adjust the MaxBufferSize.  Jet (and ACE I presume) dynamically allocates buffer size with the following expression IF the registry setting for MAXBUFFERSIZE is 0 (I *think* that is the default, but I have seen it at different values before, not knowing if it had been manually altered)...

((Total RAM on machine in MB - 12MB) / 4) + 512K

 

You can change that value to a fixed value for your DBEngine session with ...

DbEngine.SetOpen dbMaxBufferSize, <some value in KB>

 

So ... before you run your code, try setting the values for maxlocks and buffersize in order to accommodate your situation, in *theory* that will (may) allow you to control your bloat ...

 

DBEngine.SetOption dbMaxLocksPerFile, 25000

DBEngine.SetOption dbMaxBufferSize, 1048576 'KB

Dim dtTimerExpired As Date

Do While Not rsRX.EOF
    rsRX.Edit

    rsRX("something") = GetCost()

    rsRX.Update

    If rsRX.AbsolutePosition Mod 9000 = 0  Then 'Set the value for less than your MaxLocksPerFile

        dtTimerExpired = DateAdd("s", 2, Now())

        Do Until Now() > dtTimerExpired

            DoEvents

        Loop

    End If

    rsRX.movenext

Loop

datadrenaline say:

Row Level locking ... definately something to consider and expand upon :). 

DAO does not support invoking row-level locking on a Jet/ACE datafile.  Row-level locking can only be invoked/set/enabled by ADO or the Access UI. In addition, a Jet/ACE datafile can only have one row-locking mode (enabled or not) while its opened.  That mode is applied during the initial opening process of the file, subsequent connections to the file force the connecting db engine to interact with the data in the mode mode that was set by the initial opening entity.  So, if a database file is initially opened with row-level locking enabled, DAO operations from all subsequent connections will be done in row-level locking mode.

What this means is that, if row-level locking is enabled in your Access UI environment, and your Access UI is entity that initially opens your database file, row-level locking will be in effect for local table objects.  With linked table objects, the database file that houses them rarely has UI objects, so typically the initial opening of that file is done by a linked table object of a "Front End".  Unfortunately, the connection mechanism a linked table object uses cannot invoke row-level locking when initially opening the source database file.  However, using startup code in your "Front End", you can create a persistent ADO connection object which opens your "Back End" database file in row-locking mode before your linked tables are used to access the data of your back end.  By using the ADO "trick", you can then accomplish row-level locking with linked table objects.

So, the moral of the story is this ... understand the effects of the poison your are drinking! :)

In the scenario illustrated by the question starter, it is unclear if the tables are local or linked, if they are linked, its likely row-level locking is not in effect.  If they are local, then it is likely that the row-level locking setting is in effect during the processing.

Hope that was helpful!
