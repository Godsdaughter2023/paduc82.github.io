A more efficient Append if not exist - Microsoft Community

jeeped say:

A more efficient Append if not exist

I would like to perform a db.Execute in Access VBA that inserts a record if the record does not exist and I would like to do this without creating a queried recordset and examining the count before deciding to execute an append query. The RFQ table only has ~5K records.

In T-SQL I can use this reasonably efficient SQL statement,

INSERT INTO [dbo].[RFQ]
   ([custid],[rfq],[mod])
SELECT 'CUST001', 'RFQ999', 0
  WHERE 'RFQ999' NOT IN (SELECT rfq FROM [dbo].[RFQ] WHERE custid LIKE 'CUST001' AND rfq LIKE 'RFQ999' AND [mod] = 0)

In Access 2010 64-bit, I kludged together something similar with this,

INSERT INTO RFQ

   ([custid],[rfq],[mod])

SELECT DISTINCT 'CUST001', 'RFQ999', 0 FROM RFQ

  WHERE 'RFQ999' NOT IN (SELECT rfq FROM RFQ WHERE (((custid)='CUST001') AND ((rfq)='RFQ999') AND (([mod])=0)))

The access query is literally slogging through the command; taking 2-3 seconds per execution. I've got to loop through a few hundred of these on a regular basis, INSERTing as necessary. Can anyone offer some improvement on that latter Access query or suggest an alternative method? I'm in the development stage and I know some judicious indexing could improve performance but I'd like to see it get better than it currently is before I tune it with indexes.

I need the Access VBA for other things or it would be abandoned. The machine if a PowerEdge server running Server 2008 R2 with MS SQL 2008. The Access is 64-bit and the tables are only linked to their SQL originals, not local Access tables.

jeeped say:

I realized my logic was way more complex than it needed to be. I gained some ground with the following.

INSERT INTO RFQ
   ([custid],[rfq])
SELECT DISTINCT 'CUST001' AS [custid], 'RFQ999.0' AS [rfq] FROM RFQ
  WHERE ((('CUST001') Not In (SELECT [custid] FROM RFQ WHERE [rfq]='RFQ999.0')));

This runs a respectable 3-5 per second. It'll slow down as the table builds but indexing will improve overall performance.

Still open to any suggestions.

jeeped say:
I realized² that I used a reserved word (mod is a function in Access) as a field name. My bad. This has been rectified.

This only appears when I have the document sitting on my desktop and I open it from there. If you have any questions about the situation to get more clarification, please do ask

Is there any way I can keep this from happening?

Is there something wrong with my Microsoft Office - Word?

jay freedman say:

Those files are one type of temporary file that Word creates whenever a document is open. Specifically, this is an "owner file" that indicates that the main document file is in use, in case some other program or network user tries to open it at the same time. The temporary files are usually deleted automatically when you close the document, unless Word crashes.

Word creates these and other kinds of temporary files where the document is stored, no matter where that is. You're seeing them here only because the document is on the desktop. That's generally a poor practice anyway, because files on the desktop are easily deleted by mistake.

If you want the documents immediately available from the desktop, store them in some other folder and create shortcuts to them. Right-click the document file, click Send To, and click Desktop (create shortcut).

Then you won't see the temporary file unless you go to the actual document location while the document is open.
