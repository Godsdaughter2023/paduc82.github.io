Upon my analysis, you do not need to enable modern authentication right now. 

1. About “It appears that by default Outlook uses basic authentication and therefore does not support SSO unless you implement modern authentication” you said,

Outlook client always supports SSO accounts. That is to say, even without modern authentication, we can always configure ADFS/SSO-federated Office 365 accounts in Outlook client.

2. Office 2016 clients support modern authentication by default. But modern authentication is turned off by default for Exchange Online: https://support.office.com/en-us/article/Enable-Exchange-Online-for-modern-authentication-58018196-f918-49cd-8238-56f57f38d662. To use modern authentication in Outlook client with Exchange Online, we need to manually enable modern authentication in Exchange Online.

As scenario #3 is not yet supported for public preview, I don’t recommend you enable modern authentication for Exchange Online now.

You can just configure ADFS Client Access Policies to block external access, as Vasil mentioned.

Thank you

ken sheridan say:
HAVING (((tblClasses.class) Not Like "1*"))

That will work, but you should really change the HAVING to WHERE.  A HAVING clause is intended to operate on a value after grouping in an aggregating query, and is used for restrictions such as 'all customers who have made orders totalling 100000 GBP or more'.  To restrict before grouping as in this case a WHERE clause should be used.

ken sheridan say: You need to edit the RowSource property of the combo box so that it is an SQL statement whose WHERE clause restricts the rows returned.  Without know in the current RowSource property I can't say more than that, but the WHERE clause would be along these lines:

WHERE LEFT([TheFieldNameGoesHere],1) <> "1"

If the field in question can be Null (empty) and you want those rows returned it would be:

WHERE LEFT([TheFieldNameGoesHere],1) <> "1" OR [TheFieldNameGoesHere] IS NULL
