jay freedman say:

It's unlikely that this problem was caused by a Microsoft update, although that can't be ruled out. The immediate cause of the lack of entries in the menu is the absence of (or perhaps damage to) the file "Built-In Building Blocks.dotx" in your Windows profile.

With Word shut down, open Windows Explorer (the file manager program). Paste this into the program's address bar and press Enter:

%appdata%\Microsoft\Document Building Blocks

 If you use the English (US) version of Windows and Office, there will be a folder there named "1033"; for other locales the number will be different. Double-click that folder. You may find the Built-In Building Blocks file there, or you may find a subfolder named "14" (because Office 2010 is version 14) that contains the file. If you find the file in either place (or in both), delete it.

When you restart Word, it will copy the file from the folder C:\Program Files (x86)\Microsoft Office\Office14\Document Parts\1033\14 (or the equivalent for your locale) into your profile.

If that doesn't fix the problem, go to Control Panel > Programs & Features, right-click on Office 2010, click Change, and choose Repair.

daniel g. grau say:
I would use an AutoCorrect entry instead. Just select the field and assign to it a unique text sequence, like tablenum or table\ or /tn/. Then, just typing that keyword will automatically produce { SEQ "Table{ STYLEREF "chapter name" \n }" \* ARABIC }.

daniel g. grau say:
[Peter's reply was not showing when I posted this!]


Just use the "reset" switch in the first Table in each chapter that you need to renumber. That is:

          { SEQ Table \r 1 \* ARABIC }

will cause your named sequence "Table" to restart from 1 (or from any other specified number).

You can also do it through independent sequences named Table1, Table2... You obtain these self-numbered sequence names by pasting a STYLEREF field inside the SEQ field:

           { SEQ "Table{ STYLEREF "chapter name" \n }" \* ARABIC }

Note that you do need the quotes here. This has the advantage that you can always use the same field (as above) throughout your document. The above combo is equivalent to using:

  • { SEQ "Table1" \* ARABIC } in Chapter 1
  • { SEQ "Table 2" \* ARABIC } in Chapter 2
  • etc.

except that the names are built automatically, according to the value of each { STYLEREF "chapter name" \n }" as it appears each time.

Your post is accurate except for the quoted piece here.  Nothing has to be activated at the time a re-installation of Windows 10 is done.  The computer will already be a registered Windows 10 device and that is all that is required.  When the software licensing program polls Microsoft's licensing servers it will send the device's unique identifiers and the servers will find the registration and return an affirmative.  Windows then activates.  This is true even if the re-installation is a parallel installation (multi-boot) of the same edition.  The installation methodology is unimportant as well.  The user can reinstall the same edition of Windows 10 by clean or upgrade installation or by transferring a hard drive on which Windows 10 is already installed.  Image copy from another device will also work.  The only requirement is that the computer already be a registered Windows 10 device for the edition transferred.  The user just needs to remember to click the metalink on the Product Key page that says "I don't have a product key."  The metalink could better say "I don't need a product key," because that is really what it means.

Usually if the app will open on some computers, but not others, the problem is with the local database that Access uses to cache the server objects (so it doesn't have to download them every time you open the application).  If something has gone wrong with the cache database, you may see this error.

If that is the problem, you can resolve this issue by deleting the local cached database for the app.  It will get recreated when you 'Customize in Access' again.  You can find the cache database at:

  • C:\users\username\AppData\Local\Microsoft\Access\DatabaseCache\15.0

    The file will be named "appname on server.accdb" (italics replaced with your user/app/server name
