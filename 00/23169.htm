a. user say:

I think MSE is meant to work this way in case the update is unsuccessful. If it's unsuccessful it can simply restore your previous definitions.

Here is a batch script you can use to automatically remove all files starting with the word Temp in that folder.

Open notepad put this in it:

echo y | del %1\Temp*.*

Save it & rename it to DELALLTEMP.bat, and place it in the folder where the temp files are. Right-click the file & create a shortcut. Now place that shortcut in your Startup folder. Now everytime your PC boots up all the temp files will automatically be deleted.

I use this all the time:

 

Function LastRow(sh As Worksheet)    'Finds last cell with data in the last used row.
   On Error Resume Next
   LastRow = sh.Cells.Find(What:="*", After:=sh.Range("A1"), _
   Lookat:=xlPart, LookIn:=xlFormulas, SearchOrder:=xlByRows, _
   SearchDirection:=xlPrevious, MatchCase:=False).Row
    On Error GoTo 0
End Function

 

So add that to your code module, and adjust your code thusly:

 

Private Sub CommandButton1_Click()

 

FirstEmptyRow = LastRow(Sheet1)+1

ken sheridan say: To return the distinct count in a query you need to return the distinct values in a subquery, and then the count of rows returned by the subquery.  Let's say you wanted to return the number of distinct last names for a table of contacts:

SELECT COUNT(*) AS NameCount
FROM (SELECT DISTINCT LastName
            FROM Contacts)  AS DistinctName;

If you save this query as qryDistinctNames say, you can then reference it in the ControlSource property of a text box in a report with:

=DLookup("NameCount", "qryDistinctNames")

Alternatively you could simply make the query this to return a distinct set of names:

SELECT DISTINCT LastName
FROM Contacts;

and reference it in the report with:

=DCount("*", "qryDistinctNames")

You've mixed up the two alternative approaches I previously suggested.  The query should be:

and the ControlSource of the VillCount control in the report should be:

However, I should out that your Timothy Table is not normalized to Third Normal Form (3NF) as District is functionally determined by District so is transitively dependent on the key of the table.  Consequently it is open to the risk of update anomalies.  It should be decomposed by creating separate Villages and District tables, with a foreign key column in Timothy Table referencing the primary key of Villages, and a foreign key column in Villages referencing the primary key of Districts.  So the model would be:

Timothy Table>---Villages>---Districts

The relationships should be enforced of course.  Note that if you ever have occasion to insert a row into Timothy Table where you need to record the District but not the Village, there should be rows in Villages for each District with a Village value of N/A or similar.  This then allows the row in Timothy Table to map to a row in Districts via the relevant N/A row in Villages.  This does of course mean that the primary key of Villages must be a 'surrogate' key, normally an autonumber, which would be referenced by a long integer foreign key column in Timothy Table.
