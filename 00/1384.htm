hansv mvp say:

Yes. Setting the default value of the text box bound to the ID field to


=Nz(DMax("ID","Table1"),0)+1


should work. It is risky in a multi-user environment, though: the value is assigned to the ID field when the user creates a new record. If the user leaves this new record open without saving it, and if another user then creates a new record, the same value will be assigned to the ID field. So when they try to save the record, one of them will get an error message because of a duplicate primary key value.


Using the Before Update event procedure doesn't prevent this entirely, but it makes the probability of a conflict much smaller.

tomheagy say:

In very simple terms, this is what you want:

Public Sub Test()
    Range("Table1").AutoFilter Field:=1, criteria1:=Range("B7").value, Operator:=xlFilterValues
    Range("Table1").SpecialCells(xlCellTypeVisible).Copy
    Range("A9").PasteSpecial (xlPasteValues)
End Sub

In the first row you are performing a filter operation on a table:

"Table1" is the name of the table containing your original data.

"1" in the field parameter indicates which column you want to filter on.

"Range("B7").value" is referring to a cell that has the criteria you want to filter by. It would be better to use a named range for a cell like this.

In the second row, you are copying only VISIBLE cells to the clipboard:

"Table1" is the name of the table containing your original data.

In the third row, you are pasting it to another location:

"A9" refers to the first destination cell. You should probably change this to a table or at least a named cell reference.

    rob koch say:

    Is NIS still truly in use with MSE at this point?  Or at least is there any indication that it's been invoked with a recent NIS definition since I last recall discussing the Network Inspection Service several years ago?

    As I recall, at some point in the past Microsoft had determined that the NIS services hadn't been used in something like a year or more, so they had considered removing it completely from at least one version of the client on a particular version of Windows, though at this point I've lost track of any of that historical information.

    The reason for this was that the NIS service in MSE was never more than a limited detection/blocker for certain very specific network based attacks, which was only invoked if at startup the service detected that various critical operating system updates weren't yet in place.  The NIS module would only then truly initialize in order to load and operate only the specific definition(s) required to protect that individual machine until the required operating system patches had been applied.

    I don't know if there were later attacks that had occurred which caused Microsoft to retain the NIS services or they simply left it in place in case a need arose, but I've seen little to no forum or other activity related to this module in at least a few years.

    During my search today, I did find a similarly named Windows Defender Antivirus Network Inspection Service module, but that appears to have evolved as a portion of the behavioral monitoring for newly discovered vulnerabilities in network protocols.  So whether this retains the same abilities as the parallel module within Windows 7 is anyone's guess.

    The key here is whether the MSE NIS module is truly in use, since unless it is, it wouldn't be surprising if portions of its definition or other contents became out of sync with others such as the signature definitions which most assuredly are.
