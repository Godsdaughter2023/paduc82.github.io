ken sheridan say: For an overview of the use of the Hijri calendar in Access see:

http://office.microsoft.com/en-us/access-help/about-using-different-calendars-HP005257922.aspx

If the database calendar is set to use the Hijri calendar you can evaluate a date string as a date/time value using the CDate function e.g CDate("1/25/1420"), but a literal date must use the Gregorian format so if the database calendar is set to use the Hijri calendar, #2012-02-07#  will still evaluate as a date/time value representing today.  I've used the ISO standard of YYYY-MM-DD her as this is internationally unambiguous, but a date literal can also be in US short date format, mm/dd/yyyy e.g. #02/07/2012#  or any other internationally unambiguous format.

So while I have not been able to test this I think it can be assumed that if you compare the value returned by the CDate function with your Hijri calendar dates against a date literal for the Gregorian calendar dates you should get correct results.  You may need to write a little function which accepts both dates as string expressions for this.  You should be able to use the Eval function to evaluate the string for the latter, e.g. the following works in the debug window:

strDate = "2012-02-07"
? Eval("#" & strDate & "#")
07/02/2012

rob koch say: Lots of it all over this forum, why not read one of those you apparently found already?

Basically, Email scanning is simply doubling up on the same process, since all it really does is insert itself as a 'Proxy' between your email server and email client program.  This is known to cause stability and performance issues and so has been recommended against by virtually anyone on the net with a decent reputation, including the major AV vendors themselves, for years.

The reason it exists is historical, since in the early days there wasn't any way for an AV program to get control of a file before another program did, which created a 'race' condition and the danger that an opened attachment could be executed before it was scanned.  This has changed since at least the late 1990's, when Microsoft provided the OfficeAntiVirus API which allowed the AV programs to 'hook' the filing system and gain access to the files before being released to Office programs.  Over time this was expanded to operate on all files, regardless of the application(s) involved.

So today there's simply no reason to provide a proxy, since when you open an attachment it will immediately be checked by MSE (or any modern AV) before another application is allowed to access it.  The proxy is simply redundant, since it uses exactly the same detections anyway and thus provides no real additional protection.

So yes, MSE does protect from email borne viruses along with any other type of application, it just doesn't contain a pointless 'email scanner'.

Rob
