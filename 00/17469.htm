That would return the same value unless the string expression is one character long, e.g.

? StrComp("V","V",0)

 1

? StrComp("AVG","V",0)
-1

-1

or:

? StrComp("VAG","V",0)
 1
? StrComp("vAG","V",0)
 1

doesn't.

The character could be extracted from the string expression:

? StrComp(Mid("AVG",Instr("AVG","V"),1),"V",0)
 0
? StrComp(Mid("AvG",Instr("AVG","V"),1),"V",0)
 1

However, this would raise an error arising from the use of the Mid function if called in a query in this way where there is one or more rows with a value in the column in which there is no 'V' of either upper or lower case.  This could be avoided with more complex parsing, but the expression would be cumbersome.  The simplest solution is to create a module in which comparisons are binary, and put a little function in it:

' basBinaryCompare
' module starts
Option Compare Binary
Option Explicit


Public Function CaseLike(varFindIn As Variant, strFind As String) As Boolean

    If Not IsNull(varFindIn) Then
        CaseLike = varFindIn Like "*" & strFind & "*"
    End If

End Function
' module ends

It can then be called it in a query:

SELECT *
FROM [SomeTable]
WHERE  CASELIKE([SomeField],"V");

ken sheridan say: To return rows where the value in the package column includes either of the substrings 'MB' or 'MV'

WHERE Nz(InStr(1,Package,"MB",0),0) > 0 OR Nz(InStr(1,Package,"MV",0),0) > 0

To return rows where the value includes both substrings:

WHERE Nz(InStr(1,Package,"MB",0),0) > 0 AND Nz(InStr(1,Package,"MV",0),0) > 0

To return rows where the value includes 'M', but not 'BM':

WHERE Nz(InStr(1,Package,"M",0),0) > 0 AND Nz(InStr(1,Package,"BM",0),0) = 0

In essence, the expression Nz(InStr(1,Package,<String Expression>,0),0) returns a zero if the string expression is not present in upper case, a non-zero value if it is.

However, the values in the package column appear to be what is known in the jargon as an 'encoding system', i.e. the column is being used as a data structure.  This is not a good design in relational database terms as each row is storing multiple values of the one attribute, tour code.  Consequently the table is not normalized to First Normal Form.  The correct way to model this would be to have each tour code recorded in a column in a separate row of a related table which also includes a foreign key column referencing the primary key of the current table.  Querying the database on the tour codes would then be very simple.  Whether you wish to normalize the table by decomposing it in this way is for you to decide of course, but even if you don't, then you should be aware of this design flaw.  Sometimes you'll be able to work around it, as we've seen in the current context, but it could give rise to more intractable problems, particularly if you ever find the need to relate tables on the codes.

If Not IsNull(Me.[Date] Then

Else

Also, in the form's Current event procedure put the following to enable/disable the control on the basis of existing values:

Me.OT1.Enabled = (Nz(Weekday(Me.[Date]),6) <> 6)

The OT1 control will, when the form is at a new record be disabled until a date is entered.  If the date entered is a Friday it will remain disabled.  In an existing record the control will be disabled if the date in that record is a Friday.  If the user changes the date to another day of the week  will be enabled.  If the user deletes the entry from the date control the OT1 control's existing value will be removed and it will be disabled until a date other than a Friday is entered in the row.
