2017 02 00-  ACCELERATING YOUR WINDOWS 10 PROJECT WITH VIRTUALIZATION
VirtualizationAdmin.com - Monthly Newsletter - February 2017
Over the last couple years, I’ve been advising many different customers on their Windows 10 rollout plans and helping them devise a deployment strategy. The two biggest issues that emerge in these deployment strategies are application and hardware incompatibilities with Windows 10. Large organizations have considerable investments in proprietary or mission-critical software and hardware that may not have migration options. 
On the application side, issues tend to cluster into two groups. In one group, the issues stem from applications that are compatible with Windows 10, but they fail functionality testing for some reason. In the second group, issues stem from applications that pass functionality testing, but the software vendor doesn’t support it on Windows 10. 
The second group is more prevalent in large IT environments because they tend to have a greater number of specialized (and often long-lived) applications in Production that require special consideration when changes are made to the client operating system.
The takeaway message is that “application compatibility” makes for good marketing hype, but the reality is that migrating to Windows 10 in large IT environments comes with many challenges, and the lack of vendor support can be a significant blocking issue when moving applications to a Windows 10 environment.
Virtualization can play a role in your deployment strategy. There are three approaches for using virtualization to help migrate (or at least carry forward) Windows 10 incompatible applications:
1) local virtual machines
2) application virtualization
3) complete virtual desktops
As a simple example, consider the situation of migrating a Windows 95 application to Windows 7 using a local virtual machine. Within the Windows 7 framework, we had technologies such as Microsoft’s MED-V, Citrix’s XenClient, and VMware ACE, which allowed us to deploy a Windows XP virtual machine on top of Windows 7. These technologies gave us the ability to “carry forward” applications that did not work with Windows 7.
The downsides of running a local virtual machine are mainly poor user experience and overly complex machine management. Loading two operating systems (operating in parallel) on the same device means that we have to duplicate management efforts for things like Group Policies and patching. 
Other issues emerge around resource management, especially in laptops running on battery power, where we tend to see significant decreases in battery performance. Resource sharing also becomes an issue with hard drives that have slow rotational disks, which are not well suited to running a virtual machine. 
The side effect of these issues inadvertently increases management time in an attempt to ensure that the user experience is not degraded as the hardware copes with a load of an additional operating system.
The big advantage of using a local virtual machine (i.e., local execution of the application) rarely outweighs the costs that this approach incurs. 
The unfortunate result of these initial trials of using local virtual machines to solve application compatibility issues was a bad rap, and hence it didn’t get much use in the real world. 
As the industry and technology have evolved, organizations have started to house their “bad” applications in Remote Desktop Application Host environments. In the application virtualization approach, applications run in the data center and are delivered (both the UI and the data) to the end=user. 
One advantage of hosting applications in a data center is that they are in proximity to the file and database servers that support them. Applications also run on multi-user servers that optimize the resources being used in the data center. When I say, optimization of data center resources, I’m talking about the ability to have multiple users running on one operating system without having a dedicated Windows operating system for each user.
Sharing Windows is ideal, especially if the user only has one or two applications that need to be in the environment. I strongly believe that using a dedicated virtual desktop for a single application is the poorest approach to achieve application compatibility. 
The downside of moving an application’s UI into the data center is the dependency on network connectivity - when the network is down, local execution of the application is not possible. Having said that, network connectivity isn’t usually that big of a deal for most users that are on-premises. The story changes for those users who are either mobile or very remote (with limited to poor network access).
The final approach for using virtualization to migrate incompatible Windows 10 applications is to create a complete virtual desktop, which has a number of advantages:
• the ability to move the entire workspace to the data center
• recycle the user’s existing PC as a thin client or evergreen the hardware with a thin or zero client device
• significant security improvements 
• enable corporate application access for BYOD or mobile users
The takeaway message from the above is that migrating applications is a complicated decision and, in a larger organizations, picking only one approach is unlikely to fulfill all use cases within the organization. 
I tend to adopt a pragmatic stance regarding these different hosting options and look to leverage on-premises solution where it makes sense, otherwise I recommend hosting the VDI environment in the cloud. Moving applications to the cloud might seem risky, but the reality is that the underlying virtualization layer is the same as the on-premises solution.
The key differences are around IT management practices and underlying hardware; although, in the world of virtualization, I believe the underlying hardware is less relevant as long as you have the uptime and performance to support the workloads you are putting on top of it. New management practices will be needed because not all the management tools and concepts are mirrored identically with traditional on-premises management tools.
The biggest advantage when implementing a cloud-based solution (to host Windows 10 incompatible applications) is the lack of capital investment. Already, many organizations are moving disaster recovery, storage and application servers to the cloud as their aging equipment enter the end of life. When we look at VDI for application compatibility, the need for additional VDI capacity is temporary as applications are modernized and moved back to the desktop and laptop devices. The savings can be further optimized by using VDI infrastructures that are elastic, which will automatically deallocate unused resources and further reduce the consumption charges of using the cloud.
The challenge with all these approaches is understanding the drivers of change within the organization and who the decision-makers are. Who decides when the cost of remaining on Windows 7 past its end-of-life-date (and the compounding support costs) justifies a migration? Is the cost of modernizing a blocking application so massive that leaving it on an older operating system makes sense to the business in the short term? There is always this battle between modernizing the operating system and the application portfolio without creating an excessive burden on the business. 
In an ideal world, we would be able to upgrade everything at once, and even if that were the case, the reality is that some of the more complicated applications can take months or even years to modernize once the decision is made to upgrade them. Virtualization is going to play an important role in Windows 10 migrations for years to come given the bulk of organizations have yet to begin preparations for the retirement of Windows 7 and the machines that run that it.

Follow these methods.

Method 1: The RPC (Remote Procedure Call) service in Services should be started and the startup type should be set to Automatic. Follow the steps and see if the service is running properly:

b)    Right click on Services.msc and then click "Run as administrator".

c)    Double click the service Remote Procedure Call (RPC).

d)    Click on the General tab; make sure the Startup Type is Automatic.

e)    Then check if the Service Status is set to started.
