tvk65 say:

WLAN connection problems on S60

Hello,

I had an E60 phone for about one year and could not understand what problems with connectivity people have on this forum. I have read a lot of messages about problems with connection to different WLANs - open, WEP/WPA, TLS protected and so on. My E60 worked well at my home (Linksys WRT54G WPA2-TKIP+AES), at work (Cisco 802.1x with Windows authentication) and had no any problems at airports/hotels open netwworks (I travel a lot)
Yes sometimes I've seen errors like "No gateway reply", but normally refreshing the page I could get what I wanted.

Some other reasons (for example, unavailability of v3 firmware for my E60 model code for a long time) forced me to buy E65


And I've got problems! Actually problems were not related to CONNECT to the WLANs, but to connect to Internet over it. My problems were getting premanently "No gateway reply" in majority of networks I tried to connect. And funny thing that two phones laying side by side old E60 and new E65 with same one-to-one settings had shown different results: E60 worked perfectly while E65 was showing only "No gateway reply"

Being intrigued I started investigation why it's so. First, I installed IPInfo utility http://tinyhack.com/freewarelist/s603rd/2007/03/09/ifinfo/
which had shown one difference between phones: E60 had a valid IP address with valid DNS names for that WLAN and E65 had not.Of course, if E65 had not proper DNS servers it couldn't find any web page.

The multiple advices on this topic on this forum to setup static IP worked, and I understand why. But you know which static IP address, mask, gateway and DNS servers only for your home and work networks. For public hotspots they are not available and DHCP should work for them to get connected.

So first my finding was - E65 do not receive IP address from DHCP server.
I started dig in that direction. I've installed MS Network monitor on my laptop and captured packets on WLAN from E65 and DHCP server.
My second finding was that proper sequence of packets between E65 and DHCP server was not ended properly. Honestly, it was ended properly several times (when E65 could connect properly and get IP address) but the ratio was about 1/10 (one good try from ten)
And this was not only for my home network, but for work network too, with another ratio (about 1/4)
Normally the DHCP sequence should consist of 4 packets
1 Client broadcast packet to find DHCP server
2 Server responses broadcast with its address
3 Client asks specifically server to assign IP addres
4 Server acknowledges requested address or if it is used already assign one from a pool

After this sequence client sets assigned IP address and continues to work. Together with IP address DHCP server sends network mask, default gateway, DNS servers and possibly additional information asked by client.

I've looked into packets, everythings looked OK. But 4th packet from DHCP server was not sent to E65 resulted in problems.

I was stucked a little bit and started to thing that problem comes from my LinkSys router. But the same situation in a powerfull corporate environment.

Then I discovered my third finding: WLAN driver on the phone has TWO IP Addresses on one MAC address!You can see it via IPInfo or Ipconfig utility on the phone. E65 and E60 both had a second IP address in their MAC interface. It was from 169.254.x.x network. People working with MS Windows auto IP adresses knows what it is ;-)

Nothing criminal in that. All computers can have two IP addresses on the same network interface. One only thing - they should be set manually, not using DHCP.

Looking through catched packets I saw that phone periodically sends ARP (Address resolution protocol) packets from its 169.254.x.x. This is simply broadcast packet showing IP and MAC addresses and other hosts on the network can build ARP table where IP address corresponds MAC address. This is basis of Ethernet networks.

Catching that I looked more carefully on the tracing and found, that if after sending DHCP request packet (part 3) and before receiving DHCP ACK packet (part 4) phone sends ARP packet from 169.254.x.x address DHCP do not provide last DHCP ACK packet. In my understanding DHCP server when receiving ARP packet which show that particular MAC addres alreay has 169.254.x.x address do not send last packet with acknoledge.

If no ARP packet in between - phone is getting proper IP address from DHCP server and works properly

I beleive, that due to nature of working on shared media (radiochannel) and due to Ethernet standards (if collisions happened packet resent after delay) in busy WLAN networks such as corporate, public hotspots, ARP packets are sent less often and chances to get proper IP address increase. That's explain why success ratio in my case is higher in corporate network.

The fact that older E60 get IP address more often that E65 can be explained that E60 is a little bit slower and therefore ARP packets are generated rarely. Or maybe something was changed in FP1 (E65)

And my last finding was that despite of DHCP server sends every IP address details according to the standard phone do not set proper default gateway, leaving it as 0.0.0.0.
It is working in principle, but better to follow DHCP standard.

So, summary (apart from problems with connectin to secured networks - WEP/WPA preshred keys, certificftes so on):

1. WLAN on S60 (checked on E60,E65) has default address 169.254.x.x. (not clear for me why it is)
2 WLAN driver on S60 tries to get dinamic IP after connecting to the WLAN. It sends proper sequence of DHCP packets
3 Phone periodically broadcast ARP packets. When such packets wedged in between DHCP REQ and DHCP ACK packets, DHCP do not send last DHCP ACK packet. Phone do not get proper IP addres, DNS server and other settings.

I consider it as obvious bug in Symbian.

Questions to Nokia:
1. Why they need two IP addresses?
2. Why ARP packets are sent so often?
3. Why WLAN driver do not set gateway address received from DHCP?

Guys, if you could read all of this - could you please make additional tests? I mean for all S60 phones with WiFi check what I've found?

It would be nice to get this confirmed and then to send mutual request to Nokia to refresh their firmware ASAP.

I send this info to local Nokia, they responded that somebody will get back to me. Two weeks passed - no reaction.

Hi Krunal,

 

I'd like to explain that it is an expected behavior to get an threshold error when your filtered result is more than 5000 in SharePoint Online.

 

The List View Threshold is approximately is 5000 items by default and is set to allow users to work with large lists, but keep good performance. And for SharePoint Online, this limit can't be changed.
