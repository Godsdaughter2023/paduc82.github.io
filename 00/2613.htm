Since the issue is related to the mail from Exchange Online to Exchange On-premises in the Hybrid mode, my suggestion is 

1. Check the mail route. 

The simplest route is Exchange Online to outbound connector to Exchange Online Protection (EOP) to the Exchange Cas server to mailbox server. Make sure that there are no unnecessary network devices such as anti-spam gateway devices between the Exchange Cas server and EOP. Those devices could remove the necessary header. 

2. Check the remote domain in Office 365. 

a. Connect to Exchange Online by using remote PowerShell. For more info about how to do this, see Connect to Exchange Online using remote PowerShell.

b. Run the following PowerShell command:

Get-RemoteDomain <FQDNOfOnPremisesEndConnector> | FL  
c. In the output, make sure that the TrustedMailnboundEnabled attribute is set to True.
d. If the attribute in step 3C isn't set to True, use the Set-RemoteDomain command to change the value to True. 
3. Make sure that Transport Layer Security (TLS) is implemented and enabled in both environments and that the fully qualified domain name (FQDN) is set correctly. Check the on-premises received connector and the EOP outbound connector by using the Exchange Server Deployment Assistant at the following Microsoft website:

https://technet.microsoft.com/en-us/exdeploy2013/Checklist?state=3229-W-AAAAAAAAQAAAAAEAAAAAAAAAAAAAwAMAAAA%7e

By the way, for your mentioned "Transport rules that are supposed to apply only to external emails where "the sender is located" "Outside the organization" are erroneously being applied to emails that Exchange Online users are sending to the On-Premises users.", please try temporarily disable the incorrect transport rule to see if it helps.
