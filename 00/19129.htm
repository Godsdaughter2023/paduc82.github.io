I'm sorry, but I'm not quite following you. Do you want to final result to show both the UPC and SKU or do you want it to show one and only the other if the one doesn't exist (i.e. Show the SKU, except where a SKU doesn't exist, then show the UPC).

ken sheridan say: You can join (not 'link' BTW, that's something completely different) tables (which includes the result table of a query) on a Boolean OR expression, but it can only be done in SQL view, not in design view which does not support this sort of join:

SELECT [Sales Order].[Style], [SKU],
[Qty], [Color], [Size]
FROM [Sales Order] INNER JOIN [Combine]
ON [Sales Order].[UPC SKU] = [Combine].[SKU]
OR [Sales Order].[UPC SKU] = [Combine].[UPC];

However, your having to resort to this stems from a bad design.  Values of separate attribute types, UPC and SKU in this case, should not be stored in one column.   In a relational database each column position in each row in a table should store one value of the single attribute type which the column represents. It would probably be best if the Sales Order table stored the SKU value only, but a data entry form could be set up to show both the SKU and UPC values, the latter in an unbound control, with provision to allow the user to enter either and automatically assign the corresponding value to the other control.
