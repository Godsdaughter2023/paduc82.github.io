albert d. kallal say:

You cannot run update queries directly on data in web applications.

 

The other thing to keep in mind is that the code you write in a web form actually runs inside of that form (on your smartphone, or iPad web browser).

 

Last time I looked, you not find recordset commands inside of a Firefox browser, but that is in fact where your form code runs.

 

However, what you can do is call a data macro from that form.

 

You not mentioned what records you want "filter" to run this code on.

 

However, at the end of the day, you can filter the records you update is need be.

 

The basic concept here is you need to call a data macro to do the update.

 

In this example form code, we have form and an unbound check box called chkSet. What ever value is set in this check box will be applied to every row in the table when you click on the following button code:

 

Thus the calling code from the web form looks like this:

 

RunDataMacro
    Macro Name:  tblCustomers.CheckCollum
Paramters:
pCheckCollumValue =  [chkSet]

 

The above is thus really only one line of code in the form side.

Before you write the above, you first need to place the following code in the table as a named macro (I called it CheckCollum).

 As noted, in this simple example we will update all rows to the passed value (true/false). The code for this macro would look like:

For Each Record In   tblCustomers
    Where Condition
                    Alias
        EditRecord
                     Alias
           SetField
                   Name   tblCustomers.ActiveMember
                   Value = [pCheckCollumValue]
       End EditRecord

So you cannot execute update queries or update records inside of a web browser running on your SmartPhone, but you can write a simple little bit of code that will accomplish the same thing.

HTTP is broken in IE11 Mobile - Microsoft Community

ericwieser say:

HTTP is broken in IE11 Mobile

RFC2616 states:

A "host" without any trailing port information implies the default port for the service requested (e.g., "80" for an HTTP URL)

Therefore:

  • a request to http://mydomain.com/path must give a header of "Host: mydomain.com" or "Host: mydomain.com:80"
  • a request to http://mydomain.com:8080/path must give a header of "Host: mydomain.com:8080"

On the latest developer preview, IE11 Mobile does not correctly handle the second case. This can be verified with a tiny python server:

from bottle import *

@route('/'):

    return repr(request.headers.items())

run(host='mydomain.com', port=8080)

Is this bug known about?
