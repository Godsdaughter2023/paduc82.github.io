Linking one spreadsheet with several over LAN - Microsoft Community

terry parker say:

Linking one spreadsheet with several over LAN

Ok, here's what I want to do.  I am looking to create a distributable, survey type, form-laden spreadsheet with check boxes and such, that will feed the answers chosen by the users filling them out, back to a single "master" or "database" page dynamically over the LAN with running totals per category that I can track.  I'm under the impression this will require some VB, but I can deal with that. 
f z z say: All of the users would have access to the same master workbook as the same network location? And you plan on having this master workbook open yourself in order to see collected results in real time? The only way multiple users could save data in one workbook simultaneously is using a shared workbook and a hacked-together table locking and data entry macro.

You need just one workbook, but it needs to be shared. The table in which user entries would be stored should be hidden for these other users and visible just to you. You could add code to the Workbook_Open macro to check user IR, and it the user ID is yours, unhide the table and any summary worksheets you want.

As for the forms, shared workbooks will allow multiple simultaneous users to make their own separate entries. You'd need a command button on each form worksheet to record user form entries in the master table. The mechanics of how to records the data is up to you.

The difficulty is in the table locking mechanism, that is, the macro used to store form data in the master table needs to check whether another user is in the process of storing data. If so, the first user must wait until the table becomes unlocked, then the first user's session would lock the table, store the user's data, and unlock the table.

Messy to build the locking mechanism into the workbook. Cleaner to use the network file system, with the macro storing form data in the master table checking whether a specific file exists in the same directory as the workbook. If so, the table is locked and the macro would need to wait a second or two and check again. As soon as the file doesn't exist, the macro needs to create it (quickest to use VBA Open and Close statements), then write the form data to the table, then delete the file to unlock the table.

Using local credentials is still possible in Windows 8.1 and 10.
It's just that in Windows 10 preview it is "disabled". I think the main reason for that is the purpose of this preview: When you install it, you agree to allow MS to sample operational diagnostics. Using a local account would complicate this.

By default Microsoft strongly incurrages you to use your Microsoft ID.
Select to use local credentials during Windows Setup when asked for the login account.
Afterwards you can choose to connect a MS Account to the local login account also. Or else you will be asked every time a MS Account ID is expected.

One advise I have for Microsoft is to make sure the general public understands the need and benefits of using Microsoft ID (a.k.a. Windows Live ID) to login to Windows or any other service that supports it.

Most importantly Microsoft ID enables the use of a mechanism called federated authentication (ADFS) in which logon credentials can be "reused" in a secure way when needed. For instance when authenticating the Windows Store, Skype, OneDrive, Outlook, Office 365, or any other App you trust. It can be used for 3th-party authentication at Google, Facebook, a SharePoint site. This is the new standard for authentication that is a much more secure and (almost) impossible to break especially because it is tightly build into Windows 10 itself. Feel assured: you fully stay in control over the use of your credentials!

Some additional benefits:
* Single sign-on, so no more hassle managing your passwords (passwords being one of the main reasons why so many systems get hacked)
* Automatically supports Use of 2-pass authentication (secure even when a hacker gets you password)
* Windows settings sync: your personal settings, IE favorites, saved login credentials are automatically shared with all your devices.
* Optionally authenticate using PIN, picture password, NFC, your phone, a fingerprint reader, etc.
