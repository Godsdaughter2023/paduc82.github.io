ronaldoonenil say:

On sheet 1 and sheet 2 insert a new column at A which will be used as the VLOOKUP value. The formula for this new column will be a concatenation of the Last Name, First Name & DOB.

=C2&D2&E2

Copy this down for as many rows as you have on sheets 1 & 2

I assume on your new sheet you have copied all the last names, first names & DOB from both sheets into column s C, D & E and then removed the duplicates.

Your formula for cell A2 can then be as below and copied down

=VLOOKUP(C2&D2&E2,Sheet1!A:B,2,0)

Your formula for cell B2 can then be as below and copied down

=VLOOKUP(C2&D2&E2,Sheet2!A:B,2,0)

Once done, you could copy columns A & B and paste back as values only to remove the formulas.

macropod say:

You might do better to bookmark the page in question, then simply insert cross-references to the bookmark wherever you need to replicate the page. That will also help keep the contents synchronised if you change anything on the bookmarked page. If you don't want the synchronising, simply select the cross-reference after inserting it, then press Ctrl-Shift F9. The equivalent VBA code is:

 

Sub Demo()
With ActiveDocument.Range
  .Characters.Last.InsertBefore Chr(12)
  .Fields.Add Range:=.Characters.Last, Type:=wdFieldEmpty, Text:="REF MyBookMark", Preserveformatting:=False
  .Fields.Unlink
End With
End Sub

where 'MyBookMark' is your bookmartk name.

 

The .Fields.Unlink line breaks the synchronisation if you want the copies to be independent.

macropod say:

Hi Valeria,

 

Sure you can replicate pages consistently using VBA - I've given you one of many, which will work reliably provided you adhere to its constraints. Another method (that doesn't need a bookmark) is:

 

Sub Demo()
Dim Rng As Range
With ActiveDocument
  Set Rng = .GoTo(What:=wdGoToPage, Name:="1")
  Set Rng = Rng.GoTo(What:=wdGoToBookmark, Name:="\page")
  Rng.Copy
  With .Range.Characters.Last
    .InsertBefore vbCr & Chr(12)
    .Collapse wdCollapseEnd
    .PasteAndFormat (wdFormatOriginalFormatting)
  End With
End With
End Sub

 

However, if all you're aiming to achieve is the sequential numbering of prints, there really isn't any need to replicate the page. What you could do instead is to sequentially number the printouts of the same page. See, for example, the attachment to: http://www.msofficeforums.com/word-vba/12641-sequential-print-numbering.html#post33829

To generate the sequentially-numbered prints, run the FilePrint macro.

ken sheridan say: I would agree with Albert that you can simply use the version of SQL which comes with Access to learn the main points of the language.  The international standards which have been issued over the years define standard SQL, but each product has its own variations.  In general the core of the language is much the same in each, however, so using Access as a learning tools will give you skills which are for the most part transferable.  By the time you've gained a good grounding in Access's flavour of SQL you'll have little difficulty adapting to other products, though bear in mind that high end (and expensive!) products like Oracle include very much more advanced features than you'll encounter in Access SQL.

If you are familiar with using procedural languages like VBA the key point to take on board about SQL is that it is a declarative language.  You state the task in the SQL statement and the database engine does the rest.  To return a set of rows from a table which match certain criteria say, you use a SELECT statement with a WHERE clause,e.g.

SELECT FirstName, LastName
FROM Employees  INNER JOIN Departments
ON Department.DepartmentID = Employees.DepartmentID
WHERE Department = "Sales";

This is a query.  It joins the Employees and Departments tables on the numeric DepartmentID keys and restricts the query's results to those rows where the value in the Department column is "Sales", i.e. it returns the names of all employees in the Sales department.

You'll also find that Access uses what it calls 'action queries'.  Strictly speaking in SQL-speak these are not queries as such, but SQL statements which manipulate data.  They are written in the same way as a true query, however, in the query designer in SQL view.  Let's say we want to give everyone in Sales a 10% pay rise, this would be done with an UPDATE query:

UPDATE Employees  INNER JOIN Departments
ON Department.DepartmentID = Employees.DepartmentID
SET Salary = Salary * 1.1
WHERE Department = "Sales";

In the above, instead of writing procedural code which steps through the rows in a recordset and whenever it find a row where the value in Department is "Sales" updates the Salary column's value by 10%, the task is simply declared in a single statement.  The updates are then done as a single set operation rather than one by one.

One thing I would recommend is that when naming your tables and columns, try to use words which are as close as possible to plain English, and which describe the entity type which the table represents or the attribute which the column represents clearly and unambiguously.  Writing and reading queries then becomes much closer to the writing and reading plain English, and hence easier, particularly when you come to write more advanced complex queries.
