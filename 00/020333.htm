ken sheridan say: This is essentially a question of getting the logical model right and then representing the model in a form/subforms.  You start with tables for the three principle entity types:

Bars
....BarNumber (PK)
....BarName

Events
....EventID (PK)
....EventName
....EventDate

Products
....ProductID (PK)

To model the many-to-many relationship type between bars and products, i.e. the products stocked by default in each bar, you need a table:

BarProducts
....BarID (FK)
....ProductID (FK)

the primary key of this table is a composite one made up of both columns.

To model the many-to-many relationship type between bars and events you need a table:

EventBars
....EventID (FK)
....BarID (FK)

The primary key of this table is again a composite one made up of both columns.

You then need a table to model the relationship type between EventBars and BarProducts:

EventProducts
....EventID (FK)
....BarID (FK)
....ProductID (FK)
....BeginningWeight

The primary key of this table is again a composite one, this time made up of the three columns EventID, BarID and ProductID.  EventID and BarID make up a composite foreign key referencing the composite primary key of EventBars, and BarID and ProductID make up a composite foreign key referencing the composite primary key of BarProducts.

In an events form, in single form view, you'll firstly need a subform, in continuous forms view, based on EventBars, linked to the parent form on EventID and a further subform in continuous forms view, based on EventProducts.  In the first you enter one row per bar used for the event.  The EventProducts subform needs to be correlated with the EventBars subform so that as you navigate from row to row in the latter, the EventProducts subform will be requeried to show the rows for the currently selected bar.  This is done by including a hidden text box in the parent form, txtBarIDHidden say, whose ControlSource property references the BarID control in the EventBars subform, e.g.

=sfcEventBars.Form!BarID

where sfcEventBars is the name of the subform control, i.e. the control in the parent form which houses the subform.  The LinkMasterFields property of the EventProducts subform will then be:

EventID;txtBarIDHidden

and its LinkChildFields property will be:

EventID;BarID

To automatically insert rows into the EventProducts subform you can execute an SQL statement in the AfterInsertEvent of the EventBars subform:

Dim strSQL As String

strSQL = INSERT INTO EventProducts(EventID, BarID, ProductID) " & _
    "SELECT " & Me.EventID & ", BarID, ProductID " & _
    "FROM BarProducts " & _
    "WHERE BarID = " & Me.BarID

CurrentDb.Execute strSQL, dbFailOnError

' requery EventProducts subform to show new rows
Me.Parent.sfcEventProducts.Requery

This will show rows in the subform for all products stocked by default in the bar in question.  If there are any which will not be available for the current event you can delete those rows in the EventProducts subform.

In the relationship between EventBars and EventProducts be sure to enforce cascade deletes so that if a row ids deleted in the EventBars subform the rows in EventProducts for the bar in question will automatically be deleted.

ken sheridan say: Where the model as you've now described it differs from the revision I made in my last reply to Scott is that BarType appears to be an attribute of Events not of Bars.  So there should be a BarTypes table referenced by a foreign key BarTypeID column in Events, not by one in Bars.  There should also be a BarTypeProducts table with foreign keys referencing the primary keys of BarTypes and Products tables.  However, I'm now inclined to think that there should be no Bars table.  My original assumption was that Bars represented a fixed set of physical bars within a facility of some sort, but from your last post it looks like they might well be ephemeral sets of a varying number in relation to each event, in which case no Bars table would be required.  However there is no attribute of Events NumberOfBars as this would introduce redundancy and the possibility of update anomalies.  This value is represented by the number of rows inserted into EventBars, in which a set of serial BarNumber values would be inserted for each EventNum value.  This can be automated when inserting a row into Events via the parent form by having an unbound control, txtNumberOfBars say, and executing an SQL statement in the parent form's AfterInsert Event procedure:

Dim strSQL As String
Dim n As Integer

For n = 1 to Me.txtNumberOfBars
    strSQL = INSERT INTO EventBars (EventID, BarNum) " & _
        "VALUES(" & Me.EventID & "," & n & ")"

    CurrentDb.Execute strSQL, dbFailOnError
Next n

' a further SQL statement would then insert rows into EventProducts

For n = 1 to Me.txtNumberOfBars
    strSQL = INSERT INTO EventProducts(EventID, BarNum, ProductID) " & _
        "SELECT " & Me.EventID & "," & n & ", ProductID " & _
        "FROM BarTypeProducts " & _
        "WHERE BarTypeID = " & Me.BarTypeID

    CurrentDb.Execute strSQL, dbFailOnError
Next n

The subforms can then be requeried to show the new rows.  Whether or not you correlate the subforms is a matter for your own preference.  Normally where two related continuous subforms are located in one parent form they would be correlated, but it is by no means essential to do so.

Other data such as the bartender and beginning weights can then be entered manually in the subforms.

[Edit] a non-macro solution is given in the next posting from me.

The code below should do the trick for you.  The code goes into a Regular Code Module, and you can find instructions on doing that here:

http://www.contextures.com/xlvba01.html#Regular

Once the code is copied, you need to change the sheet name for Const SourceSheetName from "Sheet1" to whatever the name is on the tab of the sheet you import the data to.  I've made that line bold in the code below.  Other things that could have been made 'static' like column numbers and starting rows, I've made Const values also so that if things were to change in the future, you can adapt the code very easily by changing them.

The code will take the data in column A and put it into 2 columns of 36 entries each on as many pages as it takes to get the job done.  It's not limited to only 500 entries, it could be 1 entry or up to 65,535 (need at least one blank row at the very bottom of the sheet) in Excel 2003.

It creates sheets needed to hold the copied information - and each time you run it, those sheets are actually deleted and rebuilt: so if you need to save old information, copy those sheets into a new workbook before running the routine.

To run the code after importing your data, go to the [View] tab and choose [Macros] then pick {View Macros} and click on the one named TransferImportedData and click the [Run] button.  It should only take a few seconds to do the job with 500 entries.

IF YOU USE EXCEL 2007 OR LATER:  Be sure and save the workbook as a "Macro Enabled Workbook", type .xlsm or .xlsb once you have added the code to it.  If you do not, the code will be erased when you close the workbook.  Be sure to Enable Macros/Content when you open the workbook in the future.

  You should not have to worry about this in Excel 2003 - and you'll get a slightly different warning when you open the workbook, but you still have to allow the macros to run.

Here is the code:

Sub TransferImportedData()
'takes data in A1:A# of sheet named SourceSheetName (change)
'to other sheets in 72 row sets divided into 2x36 columns
'per sheet.  Adds sheets as required.
'Deletes old sheets along the way before beginning.
  'change this to the name of the sheet you
  'imported the data into
  Const SourceSheetName = "Sheet1" 'change as needed
  Const srcCol = "A"
  Const srcFirstRow = 1
  Const totalPerSheet = 72
  Const totalPerCol As Integer = totalPerSheet / 2
 
  'this is going to be the start of the name of
  'the sheets created during the data copying process
  Const newWSNamePrefix = "SNs Page "
  'the column to put 1st 36 entries into
  Const destCol1 = "A"
  Const destCol2 = "B"
  Const destFirstRow = 1
'
'working variables
'
  Dim calcState As Integer
  Dim srcWS As Worksheet
  Dim destWS As Worksheet
  Dim pgNumber As Integer
  Dim srcLastRow As Long
  Dim srcRowPtr As Long
  Dim cellsToCopy As Range
 
  Set srcWS = ThisWorkbook.Worksheets(SourceSheetName)
  'find last entered serial number (so it doesn't have to be
  'exactly 500)
  srcLastRow = srcWS.Range(srcCol & Rows.Count).End(xlUp).Row
  If srcLastRow = 1 And IsEmpty(srcWS.Range(srcCol & 1)) Then
    'no work to do, nothing in column A!
    Set srcWS = Nothing
    Exit Sub ' so just quit
  End If
  'delete all old "SNs Page #" worksheets!
  'for no 'sheet has data..." nags
  Application.DisplayAlerts = False
  'to prevent any recalculations from taking place
  calcState = Application.Calculation ' save current setting
  Application.Calculation = xlCalculationManual
  'to generally make things go faster
  Application.ScreenUpdating = False
  'prevent any auto-processes from activating
  Application.EnableEvents = False
  'and finally, if we boo-boo reset things properly
  On Error GoTo CleanupAndExit
  For Each destWS In ThisWorkbook.Worksheets
    If Left(destWS.Name, Len(newWSNamePrefix)) = newWSNamePrefix Then
      destWS.Delete
    End If
  Next
  'initialize the column pointer
  pgNumber = 1 ' initialize
  For srcRowPtr = 1 To srcLastRow Step totalPerSheet
    'try to set destWS = "SNs Page #"
    'if it doesn't exist, create it
    On Error Resume Next
    Set destWS = _
     ThisWorkbook.Worksheets(newWSNamePrefix & _
     Format(pgNumber, "000"))
    If Err <> 0 Then
      'sheet doesn't exist, create it
      ThisWorkbook.Worksheets.Add _
       after:=ThisWorkbook.Worksheets(ThisWorkbook.Worksheets.Count)
      Set destWS = ActiveSheet
      destWS.Name = newWSNamePrefix & Format(pgNumber, "000")
      Err.Clear
    End If
    On Error GoTo CleanupAndExit
    Set cellsToCopy = srcWS.Range("A" & srcRowPtr).Resize(totalPerCol)
    cellsToCopy.Copy
    destWS.Range(destCol1 & destFirstRow).PasteSpecial xlPasteValues
    Set cellsToCopy = cellsToCopy.Offset(totalPerCol) 'get 2nd group of 36
    cellsToCopy.Copy
    destWS.Range(destCol2 & destFirstRow).PasteSpecial xlPasteValues
    pgNumber = pgNumber + 1
  Next ' end srcRowPtr loop
  ' just for neatness
  srcWS.Activate
  Range("A1").Select

  Application.EnableEvents = True
  If calcState <> 0 Then
    Application.Calculation = calcState
  Else
    'make it automatic
    Application.Calculation = xlCalculationAutomatic
  End If
  Set srcWS = Nothing
  Set destWS = Nothing
End Sub

jllatham say:

Shanna, you can do what the code did with worksheet formulas only, it just takes some time and a little effort on your part to do it.  But if your data is always 500 serial numbers (or at most 500 of them) then here is the basics of how to do it.

We will assume that you import the data to a sheet named [Sheet1] on its tab.

On the first sheet that is to hold the 'transferred' data, put this formula into cell A1

=Sheet1!A1

move the mouse cursor down over the lower right corner of that cell until it becomes a small + symbol instead of the big cross.   Click and hold the left mouse button and drag it over to cell B1.

Now you should have formulas in A1 and B1 like these

=Sheet1!A1       =Sheet1!B1

Select both of the cells and again move the mouse pointer to the lower right corner of cell B1, now click and hold the left mouse button and drag all the way down to row 36.

This should fill all of those cells with formulas that each reference a different cell on Sheet1 to get cells A1:A72 over into the new sheet.

Now, copy that sheet into the workbook.  Give it a new name if you want to, doesn't matter.

On the new sheet, change the formula in A1 to become

=Sheet1!A73

and repeat the two 'drag' operations just like before.

Repeat until you have created enough sheets to hold all 500 serial numbers: if you forget or can't easily do the 'public math' to determine what number to put into A1 of any new sheet, simply look at the last cell in column B and the number you need will be 1 more than is in at the start of the new sheet setup.

Hope one of these two methods helps you get the job done easier.

You don't have to do anything special with those extra sheets after you set them up that first time - after that, when you import new numbers into [Sheet1], all of the sheets you created will hold the new set of serial numbers.

If you think your Outlook data file is damaged, use the Inbox Repair Tool (scanpst.exe) to fix it. Repair Outlook Data Files (.pst and .ost). Run it multiple times until it reports no more errors.

If you have a data file left over from the Outlook 2002 (Office XP) days, you should make sure you create a new data file to replace it. The old Outlook 2002 format is obsolete and can't get larger than 2 GB.

The phone number field format depends upon having Windows telephony values configured properly for your country and area code. Click Start and in the Search field, entry Telephone. The "Phone and Modem" control panel should appear. You can set the values there.

If you update something in your Contacts folder, it must, of necessity, be correct in the Address Book interface because the Address Book gets its information from Contacts.

Updating a contact's email address doesn't instantly update any Distribution Lists containing those contacts. If you update a contact's address, open the DL containing that contact and click "Update Now" to make sure their new information is added to the DL.

Seeing addressees appear when you start typing in the To field is not the Address Book and the information in the pop-up list can be different from what's in the Address Book. That pop-up list is just a cache of the addressees you've used in the past, whether or not they're in your Contacts folder. So, if you've sent mail to a DL before, it's membership list at the time you sent that earlier message is retained in the cache and will be used again if you select the DL from the pop-up list. In order to get the new DL definition added to the cache, you need to remove the current value by moving your mouse over it so it turns blue, then pressing delete. Re-add the new DL by specifically selecting it from the Address Book after pressing the To button.
