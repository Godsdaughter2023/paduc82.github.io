rich michaels - mvp (greatcirclelearning.com) say:

Hi Jim,

There are a variety of reasons of why this can occur. The most common is having multiple versions of Word installed on the computer (example Word 2008 and Word 2011) and each point to the same location for the Normal template that each version uses and you regularly change between the versions.

If you do have more than one version loaded and really don't need to, then my recommendation is uninstall the other unnecessary version.

Other possibilities for why this is occurring can be:

  1. The system's cache memory is corrupted because you haven't restarted in quite sometime
  2. The Normal.dotm template has corrupted and needs to be regenerated
  3. You are working from a Flash Drive
  4. The system contains multiple copies of the "Normal.dotm" template and various documents point to the different locations.
  5. Word expects "Normal.dotm" to be located in ~/Library/Application Support/Microsoft Office/User Templates... but it has been moved someplace else and the User Templates File Location has not been properly set in Word > Preferences > File Locations

The list is not meant to be exhaustive... it's just some places to to start looking.

ken sheridan say: If the column to which the combo box is bound represents a legitimate attribute of the Flights entity type, and there can be a values of the attribute other than indicating the presence or absence of matching rows in the referencing table to which the subform is bound, then it would be valid to have such a column in the Flights table.  If, on the other hand, it would merely indicate the presence or absence of matching rows in the referenced table then, as Scott points out, it is redundant.

Redundancy allows scope for update anomalies, which is exactly what you are experiencing when a user selects a value in the combo box which conflicts with the presence or absence of matching rows in the referencing table.  It is a simple task to indicate whether there are or are not any matching rows in the referencing table by means of an expression in an unbound combo box in the parent form.

If the column is not redundant and can have values with significance beyond the presence or absence of matching rows in the referencing table, then the solution illustrated in my Families demo could be used to enforce at least one row in the subform, with the conCRITERIA constant amended in the way I described to make the criterion conditional on the FlightStatus value, along with the other amendments which would be necessary to the rest of the code in the parent form's module.
