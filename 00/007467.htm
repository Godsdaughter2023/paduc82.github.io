albert d. kallal say:

You have a number of options here, first of all when you say 50 MB. If that amount after you split your database, and then do a compact + repair? Then how large is the data file? 

 

I ask the above since while the office live does not support linked access tables for shareing data, Ofifce Live small Business DOES support linked tables, and it still free. (I think the 1st 500 megs is free).

 

So don’t confuse office live and office live small business.  Keep in mind that offive live small business (OLSB) does not work well fo large tables with access 2007. And this solution does not work well if your application has lots of related tables. Because of these limitations that is why I asked for some general numbers such as how large is your data file when you split the application. And, what is the number of rows in the largest table?

 

So, keep in mind that office lives small business does support Access Data as linked tables being shared in different locations or even different cities.

 

However, it does not not work well for very large files. There a great demo here

 

http://office.microsoft.com/en-us/access/HA101672561033.aspx

 

(right now, the office.microsoft.com sites have not been working for a few days, I not sure if they the above site is going to come back, but the above link is a great demo on using office live small business with access). Why those links are down right now is strange, so try the above in a day or so, as the article + demo is worth watching.

 

So one free way to consider is to use OLSB. It does allow multiple uses in different locations to actually share data live with linked tables to the offive live small business site. And this means you don’t have to have a convoluted file transfer type system, you’re simply sharing live data at all times.

 

If your application has a lot of related tables or the table sizes tend to be in excess of 5000 rows, then office live with 2007 probably will not perform very well. For access 2010 performance is REALLY good, but it is un-known at this time if OLSB will support 2010 performance enhancements.

Another great solution to consider is using SQL server for the backend. You can set up one of the many several free editions of sql server.  This solution would also allow users in different locations to connect to that data OVER the internet even when the applications installed on their desktop. You have to setup a VPN to accomplish this, but it works well.

 

Another solution that I’m actually using for several of my clients is using a web hosting package that supports connections to SQL server.  Normally the connections to SQL server are reserved for uploading data that the websites going to use.  However for those web hosting providers that allow this ODBC type of connection, it also means you can get very low cost distributed computing and simply can install the access front end on any computer. With that conneciton, they are sharing data live. These low cost web providers can be had for a few dollars per month. I am doing this now wtih great success for a good number of clients.

I also have an article that outlines some additonal possibilities and ways to work with data over the Internet here:

 

http://www.members.shaw.ca/AlbertKallal//Wan/Wans.html

Figured it Out

Private Sub Worksheet_Change(ByVal Target As Range)
    On Error Resume Next
    Range("C4").Sort Key1:=Range("C467"), _
      Order1:=xlAscending, Header:=xlYes, _
      OrderCustom:=1, MatchCase:=False, _
      Orientation:=xlTopToBottom
End Sub
