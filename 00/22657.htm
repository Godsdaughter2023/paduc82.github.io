a. user say:

FWIW I am not an Excel XML expert so may have got the wrong end of the stick, but in the absence of any other useful suggestions I'll bite:
 1. AFAICS if you have a .xsd that describes your XML and specifies that a particular column has type xsd:string, Excel will honour that type, but unfortunately in its usual way of marking the cell with an "error"
 2. But in order to get that far, you have to have
     a. your xml data file
     b. a .xsd schema for your data file /that Excel can understand/ - and Excel doesn't seem to be able to understand many of the constructs I would usually use in a .xsd
     c. a spreadsheet in which you have already loaded the .xsd
     d. a mapped area within the spreadsheet. Otherwise when you try to import the .xml file, you just get an error saying that the .xml doesn't specify a schema, and Excel will try to create one.

So that's not really a "fire-and-forget" approach. It's possible that (d) can be solved by specifying a schema name in some way that Excel understands, but that seems to be one of the things that all the documentation slithers around.

An alternative might be to generate the entire Excel workbook using Spreadsheet XML of some kind, allowing you to define the cell format you need and so on. But I'm not sure what then happens when you try to insert that into an existing sheet , if that is what you want to do.
