Clipboard (cut-and-paste) broken in Windows 10 Mobile - Microsoft Community

alandye say:

Clipboard (cut-and-paste) broken in Windows 10 Mobile

Reposting this problem for helpful workarounds (or pointers from MS folks to actually FIX the issue); others have experienced similar cut-and-paste problems, but I believe this is a wider issue:

The clipboard functionality appears to be PARTIALLY BROKEN in Windows 10 mobile under certain circumstances, especially with regard to Edge. I have experienced bad behavior using Microsoft's own Authenticator app (which generates codes for 2-factor authentication of Microsoft Accounts such as hotmail.com, outlook.com, etc.). I am unable to copy-and-paste text to and from applications, and the clipboard icon which worked on Windows Phone 8.1 now no longer works reliably.

In the case of Edge, attempting to paste text into the URL bar or page inputs MAY allow the "press-and-hold, select Paste from context menu" technique as a work-around, but the clipboard icon functionality is still broken.

Win Phone 8.1 showed the clipboard icon on the area above the soft onscreen keyboard (where the microphone icon also now appears for vocal input), or you could right-swipe this section to get to the clipboard icon hidden offscreen left so long as content was on it. By pressing this icon, the clipboard content would be pasted into the active input, but this functionality appears to be broken in Win 10 mobile. Even with content on the clipboard, the clipboard icon remains greyed out and non-responsive. This type of failure can be repro-ed using Edge, OneNote, and the Microsoft Authenticator app.

Repro steps for this problem using the Authenticator app:

(Note that you must have a Microsoft Account with 2-factor authentication enabled and configured in Authenticator before starting.)

1) Boot up the phone clean (to ensure you don't have credentials already cached in Edge)

2) Launch Edge and navigate to http://account.live.com

3) When prompted, enter your username and password

4) When challenged for the 2-factor code, switch to the Authenticator app, press-and-hold for the context menu of the given account, and select Copy to Clipboard to place the current code value on the clipboard.

5) Switch back to Edge and select the "code" input box.

Result:

Notice that the clipboard icon is NOT shown, and if swipe-dragging the area about the keyboard to view the icon, it is still in a disabled state.

Note that if (after doing the above) you switch to another app which takes text input (such as OneNote) and click to enter text, the clipboard icon DOES appear and works as expected, so this is likely an issue limited to interactions with Edge or different security contexts across apps.

Additional repro notes:

I have also seen bad behavior going "the other direction" (e.g. copying FROM Edge to something else), for example if you copy the URL from the navigation box in Edge and attempt to paste that into OneNote, the clipboard FAILS this cases also, and THERE IS NO WORKAROUND. I'll repeat that: You cannot copy a URL from Edge to OneNote, which seems like a very common scenario. Press-and-hold does not work in this case, so this definitely looks like a BUG in Win 10 mobile.

EWSMaxConcurrency (Exchange Online): Defines the number of concurrent open connections that a specific user can have against an Exchange server that is using EWS at one time. The default value for Exchange 2010 is 10. The default value for Exchange 2013 and Exchange Online is 27.

Concurrency refers to the number of connections from a specific user. A connection is held from the moment a request is received until a response is sent to the requester. If users try to make more concurrent requests than their policy allows, the new connection attempt fails. However, the existing connections remain valid. Throttling policies can affect concurrency in a number of different ways.

The EWSMaxConcurrency throttling policy parameter sets the number of concurrent connections that a specific user can have against an Exchange server at one time.

Best practice for scaleable management of Office 365 Groups

The "old" model in an Exchange/SharePoint environment was to create security groups and assign administrative permissions to those groups, rather than to individuals.  Then, when personnel changes inevitably occurred, the global admin only had to modify the membership of the security group, and not every instance in which that group was given permissions.

However for the "new" Office 365 Groups, an AD security group can not be assigned ownership permissions.  There doesn't seem to be an option to allow compartmentalized oversight of several groups by a defined set of users.  Best I can tell, the only way owner-level permissions are obtained is by individually assigning specific users at the group level, which is time consuming and not at all scaleable.

Obviously, I'm missing something...

So, how does one go about administering Office 365 Groups?

iry chen msft say:

Hi Kamerone,

For the "old model" you mentioned, you are totally correct.
