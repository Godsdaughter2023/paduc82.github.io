tom van stiphout say:

You got a bit in the weeds :-)

No problem; let's see what we can do.

OpenDatabase is used to open a second database. That is a fairly rare thing for senior programmers, and not needed in your case. I'm not very clear on the purpose of Form_Load. It's not necessary to open subforms, if that's what you were trying to do.

Procedure cmdGoToDetails_Click looks good. It shows a few imperfections that we can wave at this time as not germane to the discussion, but for later:

1. [ID Number] indicates you have field names with spaces and/or other funny characters, which can cause notational difficulties (like having to wrap in square brackets all the time].

2. [ID Number] is apparently text because you're wrapping it in single-quotes for sWhere. It's somewhat unusual to have textual primary key values, and also unusual to call a text field "Number".

Since you were able to remove the Requery action, I would think the subform does not jump to first record anymore. You said you took it out "...without benefit" so my guess is there is another Requery lurking somewhere, because the normal behavior of the subform is to stay at the current record when another form is opened.

ken sheridan say: Can you perhaps clarify a few things here:

1.  You say that the main form contains a subform in datasheet view, but when building the string expression for the WhereCondition argument of the OpenForm method you are referring to a control in the parent form with:

sWHERE = "[ID Number] = '" & Me.txtMasterID & "'"

That suggests that the parent form is unbound and the txtMasterID control is an unbound control which references the ID Number of the subform using the syntax:

    =[SubFrmMaster].Form![ID Number]

Is this so?

2.  The value of the ID Number column is being wrapped in quotes characters in the above whereas, as Tom points out, its name suggests that it is a number data type.  Is it really of text data type?

3.  It is difficult to see why you have so much code in the main form's Load event procedure.  The only line which seems to do anything useful is the line which calls the MoveSize method to set the position and size of the form.

If my assumption in 1 above is correct it seems to me that you can remove the vast majority of the code, in which case the two event procedures would be:

Private Sub Form_Load()

    DoCmd.MoveSize 1400, 2400, 15000, 9000

End Sub

Private Sub cmdGoToDetails_Click()

   Const MESSAGE_TEXT = "No current record selected."
   Dim sWHERE As String

   sWHERE = "[ID Number] = '" & Me.txtMasterID & "'"

   If Not IsNull(Me.txtMasterID) Then
       DoCmd.OpenForm "frmHandsOnDetails", _
           WhereCondition:=sWHERE, _
           WindowMode:=acDialog
   Else
        MsgBox MESSAGE_TEXt, vbExclamation, "Invalid Operation"
   End If

End Sub

If the ID Number column is in fact a number data type the code would be amended to:

   sWHERE = "[ID Number] = " & Me.txtMasterID

Note that in the above code the frmHandsOnDetails form is opened in dialogue mode.  I would recommend this as it prevents the user moving focus back to the main form until they close the frmHandsOnDetails form.  There is no need to establish a recordset or grab a bookmark.  The only situation in which it would be necessary to do this would be if a new row were inserted in the frmHandsOnDetails form and the subform were requeried when code execution is returned to cmdGoToDetails_Click event procedure.  At present you have given no indication that you wish to include that additional functionality, but if you do so my recommendation would be that this be done via a separate 'Add New Item' button, rather than as part of the current routine.  To prevent the user being able to add a new row in the frmHandsOnDetails form you could extend the above code as follows:

       DoCmd.OpenForm "frmHandsOnDetails", _
           WhereCondition:=sWHERE, _
           WindowMode:=acDialog, _
           OpenArgs:="NoAdditions"

and in the frmHandsOnDetails form's Open event procedure put:

    If Not IsNull(Me.OpenArgs) Then
        Me.AllowAdditions = (Me.OpenArgs <> "NoAdditions")
    End If

With this simplification of the code you should be able to open the frmHandsOnDetails form from the button on the main form, and return to the main form without the record pointer in the subform moving from the selected record.  If it still does so then that would suggest there is other code in either the main form's or subform's module which is causing this to happen.
