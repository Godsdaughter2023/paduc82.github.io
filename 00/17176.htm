ken sheridan say: The purpose of this assignment is all-important here.  If this is seriously being put forward as a valid design for a relational database application, then, as Tom has quite rightly stressed, this is not the way to go.  If a club membership number is to be built from values in other columns, then it should be done so by returning a computed column in a query, or an unbound computed control in a form or report.  The values from which the membership number is built must of course be a candidate key, i.e. the values in combination must be distinct within the table.  The columns can be made the composite primary key of the table, but that would mean composite foreign keys of the same columns must exist in referencing tables.  This is possible, and is perfectly valid in the database relational model, but in most situations would be cumbersome.  Generally in a context like this an autonumber surrogate key, MemberID say, would be used, which would then allow it to be referenced by a single column foreign key of long integer
If, on the other hand, the assignment is merely an exercise in the use of check digits, and MS Access is merely being used as the vehicle for the exercise, then I see no objection to this, provided that it is appreciated that this would not be done in this context in an operational relational database application of this nature.

Check digits are used to validate the values which are used as encoding systems, i.e. single values which act as a data structure, encoding individually atomic data in a single value.  We encounter such encoding systems all the time, e.g. ISBN numbers postal codes such as the numeric US Zip Codes, or the alphanumeric UK Post Codes, telephone numbers etc.  Not all will contain a check digit, however.  In your assignment you have been given the mathematical operations necessary for the algorithm to compute the check digit on the basis of the values in the columns in the table.  It's now a question of expressing this algorithm in VBA code.  It's not our job to do this for you as the object of the assignment is to test your abilities not ours, but Mimmo has, unwisely and inappropriately in my view, suggested some code.  I have not looked at it in any detail to see if it fits the bill, but I will say that my approach would have been different.  Rather than using string manipulation functions when parsing a numeric value I would make use of the basic arithmetical operations of division (possibly including integer division), multiplication subtraction and addition, along with the Mod operator.  I'll give you one example as a starter:  2017 Mod 100 returns 17.

Returning to the question of the use of concatenated values in a column in a base table I'll leave the last word with Edgar F Codd, who first introduced the database relational model around nearly 50 years ago:

"Every datum (atomic value) in a relational database is guaranteed to be logically accessible by resorting to a combination of table name, primary key value and column name."

This is the Guaranteed Access Rule, and was Rule #2 in the set of 12 rules he published in 1985 in response to his concerns with the increased use in relational databases of things like that which you have been asked to do, which would violate this rule of course.
