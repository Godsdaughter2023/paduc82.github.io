ken sheridan say: Amanda:

When you select a product type you need to requery the products combo box so that it shows only the products for the selected product type.  Firstly, though, could you want both ordinary and special foods listed as well as being able to list one or the other type?  If so make the RowSource of the product type combo box something this:

SELECT ProductTypeID, ProductTypeName, 1 As SortColumn
FROM ProductTypes
UNION
SELECT 0, "All Products", 0
FROM ProductTypes
ORDER BY SortColumn, ProductTypeName;

Set its other properties as follows:

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first dimension is zero to hide the first column and that the second is at least as wide as the combo box.

This will give you a list starting with 'All Products' and followed by the different product types.

For the RowSource of the other combo box  you'd then use:

SELECT ProductID, Product
FROM Products
WHERE ProductTypeID = Form!cboProductType
OR Form!cboProductType = 0
ORDER BY Product;

Where cboProductType  is the name of the first combo box.  Set its other properties as for the first combo box.

Whether or not you set the controls up as above to allow all products to be listed, you need to requery the second combo box in the AfterUpdate event procedure of the first, cboProductType, with:

Me.cboProduct.Requery

Where cboProduct is the name of the second combo box.

One way to handle the problem, since your "measures" tables all have in common the character of being "measures" of something about about the assessment, would be to abstract that character into a single table of AssessmentMeasures, and combine the 75 tables' records into that one table.  That table would have, maybe, a field structure like this:

    AssessmentMeasureID (autonumber, PK)
    AssessmentID (long integer, FK: Assessments)
    MeasureID (long integer, FK: Measures)
    MeasureValue (text)

The MeasureValue field would have to be of type Text if any of the measures to be stored has a text data type; otherwise it could be a number type such as Double or Long, whatever is appropriate.

There would also be a Measures table with one record for each of the different types of measure.  So its field structure would be like this:

    MeasureID (autonumber PK)
    MeasureName (text)
    MeasureUnits (text) (maybe)

 

This sort of structure is often used in Laboratory Information Management Systems (LIMS), where the results of many different sorts of tests have to be stored and managed.

ken sheridan say: You have confirmed the suspicions I expressed in your other thread.  This is not how data should be stored in a relational database.  In broad outline you should have the following tables:

Stores
….StoreID  (PK)
….StoreName

Products
….ProductID  (PK)
….Product

StoreSales
….StoreID  (FK)
….ProductID  (FK)
….SaleDate
….Quantity

The last table models the many-to-many relationship type between stores and products by resolving it into two one-to-many relationship types.  Its primary key is composite one of the two foreign keys, StoreID and ProductID,along with SaleDate.

You can automate the recasting of your existing data into this model, and you'll find an example as UnencodeColumns..zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file does require some familiarity with VBA and SQL, so you might find it easier to do so by means of a set of 6 'append' queries, each inserting rows for one product into StoreSales  by joining your current table to prepopulated Stores and Products tables.

With a correct structure as above, it is a simple matter of summing the sales per product over a set of rows, rather than adding columns.  This can easily be done in a report grouped appropriately.
