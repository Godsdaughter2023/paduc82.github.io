Set Autonumber in Access - Microsoft Community

kimfisher say:

Set Autonumber in Access

I need to specify the starting number in my Access table primary key field.
john w. vinson say:

I do have to disagree with JackieL about the reliability of autonumbers; there have been bugs, years ago, fixed with service packs, that caused them to duplicate, but they otherwise work fine...AS DESIGNED.

But they're NOT designed to be sequential, gapless, human-meaningful numbers, and they aren't!

They will ALWAYS have gaps. If you delete a record, that autonumber will never be reassigned. It's gone. Even if you just start a record, and hit <ESC> after the first keystroke to cancel the record, an autonumber will be used up and lost forever.

Most developers use autonumbers only "under the covers", as a meaningless unique key and linking field; they should generally not be made visible to users at all, and certainly shouldnot be assigned any meaning. They are emphatically not a record count, as some folks mistakenly assume.

So if you want a meaningless, unique key for a record, use an autonumber. If you want an Invoice Number that a green-eyeshaded auditor will be inspecting; or a number that will count how many widgets are produced in a year - don't use Autonumber; instead use a Long Integer and assign it manually or in code. A Bing or Google search for "Custom Counter" will turn up plenty of examples of how to do so.

John W. Vinson/MVP

graham r seach say:
Hi Kim,
 
There are two ways to do this.
 
Firstly, you can specify the starting value in DML when you create the table. The following SQL creates a tabled called "MyTable" containing two fields; one being an AutoNumber primary key having an initial value of 1, and incrementing by 10; the other being a text field called "test" having a size of 50 characters.
 
    CREATE TABLE MyTable (id COUNTER(1,10) NOT NULL PRIMARY KEY, test TEXT(50))
 
If you just want it to increment it by 1, then use this:
 
    CREATE TABLE MyTable (id COUNTER(1,1) NOT NULL PRIMARY KEY, test TEXT(50))
 
The other method is to insert a record whose primary key value is one less than the value you want as your starting value. Using the above table, the following SQL adds a record where the primary key value is 999. I then delete that record (since we don't actually want it). The very next record's primary key value will be 1000.
 
    INSERT INTO MyTable (id) VALUES (999)
    DELETE * FROM MyTable WHERE id = 999

Are these emails now on my hard drive? Or is Outlook 2011 simply displaying what Outlook.com would be?

talking moose say:
Office 365 (outlook.office365.com) and Outlook.com are not the same service.

If you're using Office 365 then Outlook for Mac is synchronizing message with your server.

If you're using Outlook.com then Outlook for Mac is copying the messages from your server and will by default leave copies of your messages on your server.

The two behaviors aren't the same, though. If you connect another device to an Outlook.com account then its behavior may delete your messages from the server after a certain period of time. You'll still have the downloaded copies.

Hope this helps!
