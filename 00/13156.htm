ken sheridan say: Rather than screenshots of your queries it usually helps us more if you post a  query's SQL statement as text in your reply.  To do this switch from query design view to SQL view and copy what you see in the SQL window.  To post an image file here use the Image icon in the menu bar at the top of the reply window, rather than pasting.

From the information you've provided my guess is that you should be using a LEFT OUTER JOIN rather than an INNER JOIN in the query.  The former will return all rows from the table on the left side of the join, regardless of whether there is a match on the right side.  

Note however that if you want to restrict a query's results on a column in the table on the right side of the join a LEFT OUTER JOIN will not work, in effect becoming an INNER JOIN.  The way to get round this is to LEFT JOIN the referenced table to a subquery in which the restriction is undertaken.  The following is an example form my DatabaseBasics demo:

SELECT FirstName, LastName, EmployerCount
FROM Contacts LEFT JOIN
    (SELECT ContactID, COUNT(*) AS EmployerCount
     FROM ContactEmployers
     GROUP BY ContactID
     HAVING COUNT(*) > 1)  AS CE2
ON Contacts.ContactID = CE2.ContactID
ORDER BY LastName, FirstName;

This returns all contacts, along with the number of employers for any who have more than one employer.  The subquery is given the alias CE2 and joined to the Contacts table in the same way that you'd join two conventional base tables.  You cannot easily do the above in query design view, however, though you could create two separate queries in design view and then join those queries in a further query, all in design view.

macropod say:

Hi searaygator,

When you access the header/footer where you want the page numbering to appear, first unlink it from the previous section. Then:
1. Press Ctrl-F9 to create and empty field, thus '{ }' and type 'SECTION' between the field braces, so that you end up with '{SECTION}'
2. Type '-' after the field
3. Press Ctrl-F9 to create and empty field, thus '{ }' and type 'PAGE' between the field braces, so that you end up with '{PAGE}'
4. Format the PAGE field to start from 1.
5. Select the fields and press F9 to update.
6. Copy the field code.

Repeat the header/footer unlinking for the subsequent sections and format the page numbering there to start from 1 also. If they're already unlinked, paste the copied field code that you copied at (6) above.

Cheers,

macropod MS MVP - Word

tallyman say:

Moving Outlook 2003 Data Filses to Outlook 2010

Will be installing Outlook 2010 and want to keep all data files (.pst).  Can I save the 2003 .pst files from their present location and then after installing Outlook 2010, copy them to the location where the .pst files are kept for this version?  I have a very complex e-mail account tree and do not want to re-build it or loose the data in the file.  I understand that it is best to uninstall Outlook 2003 before installing Outlook 2010. Thanks Much
russ valentine say:

No. Do not just copy files to the default location. Doing so does not connect them to the Outlook profile correctly and may just corrupt your new installation. Correct methods are poasted here daily. Sorry you couldn't search the form.

The file you need is your Personal Folders file (*.pst) It's where all the mail, calendar, contacts etc are stored.

Take a look at these pages for info on Outlook data backup or transfer:
http://www.slipstick.com/config/move_outlook.asp
http://www.howto-outlook.com/Howto/backupandrestore.htm
http://office.microsoft.com/en-us/assistance/HA010771141033.aspx

Note that some of the information on the Microsoft page is outdated and contains errors, particularly when applied to Outlook 2003, 2007 or 2010. Specifically:
1. Ignore any advice to copy your PST file to Outlook's default location to connect it to your profile. Doing so will often corrupt your Outlook profile. Copy it anywhere BUT the default location and then open it within your Outlook profile.
2. Ignore any advice that tells you to use export or import to transfer an entire Outlook data file. That has never been sound advice, but the process has become too deeply flawed to be trusted.

Also, Outlook 2010 has a nice new option that lets you connect to a pre-existing data file when creating a new profile instead of having to jump through all the hoops of creating a new file, connecting to the pre-existing file, then disconnecting the new file.

Microsoft Access - create a comment box? - Microsoft Community

crazy_stats say:

Microsoft Access - create a comment box?

I am VERY new to Access. I have limited knowledge of Access capabilities.

I have a form set up and on that form is a button, when you click the button it will open up a subform that has a few sections, review date, follow-up etc., and a comment box. When I fill in the information in the comment box and close out it does not save any of the information. I would like it to save the information for that specific record associated with the click. I would also like the review date etc. to populate the main form in the locations that have the same labels. I hope that makes sense? Please let me know if you have any words of wisdom!

ken sheridan say: Does the fact that you are using two forms rather than one bound form imply that you are trying to associate multiple comments with each record in your main form?  If so you need two related tables, one with columns for the attributes of the main table, the other with the attributes of the related comments table.  The comments table will contain columns for the attributes of each comment record, along with a foreign key column which references the primary key of the main table, and a relationship should be created between the two tables, with referential integrity enforced.

You are using the term 'subform' incorrectly here.  A subform is a form which is embedded in a parent form, so in your case you could have a comments subform within your main form, linked to it by setting the LinkMasterFields and LinkChildFields properties to the primary and foreign key names respectively.  Normally a subform will be in continuous forms view or datasheet view, but not necessarily so; it can be in single forms view.

What you are describing are 'linked' forms, which is an alternative approach.  In this scenario, to open the linked form to insert a new comment record the code for a button's Click event procedure would be along these lines:

    DoCmd.OpenForm "frmComments",  _
        DataMode:=acFormAdd, _
        WindowMode:=acDialog, _
        OpenArgs:=Me.YourPrimaryKey

where YourPrimaryKey is the name of the main form's table's primary key column.

In the frmComments form's BeforeInsert event procedure, assign the value passed to the form as its OpenArgs property to the foreign key column like this:

    If Not IsNull(Me.OpenArgs) Then
        Me.YourForeignKey = Me.OpenArgs
    End If

You'll find examples of the use of subforms and linked forms in FormsDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file one option uses subforms on separate pages of a tab control, another option uses linked forms in dialogue mode as described above, while a third option uses linked forms in synchronized mode.

However, I'm a little concerned by your statement "I would also like the review date etc. to populate the main form in the locations that have the same labels."  This suggests that you might be storing data redundantly in more than one table.  A fundamental mechanism of the database relational model is that of Normalization, which requires all non-key columns in a table to be functionally determined solely by the key of that table.  For an introduction to Normalization, of which a good understanding is essential to anyone designing relational databases, see Normalization.zip in my same OneDrive folder.  You might also like to take a look at DatabaseBasics.zip, which is a simple introduction to some of the main principles and methodologies involved in the design of relational databases.
