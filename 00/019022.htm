scottgem say:
When you have a field that has a code populated from a lookup table, that field is called a Foreign Key. To include more info from the related table in a report, you first create a query. In that query you include your main table (the one with the foreign key) and the associated lookup table. You can then add columns from both tables. When you create your report you can use whatever columns you want.

john w. vinson say:

For example my table has several fields for types of trees - eg. Dominant species in overstory, dominant species in secondary canopy, dominant species in understory, secondary species in overstory, secondardy species in secondary canopy, etc.

 


That's the source of the problem then: your table is not correctly normalized. These fields are what's called a "repeating group" - you have a one (forest) to many (trees) relationship, embedded in this one table!

 

Consider instead modeling this as a many to many relationship:

 

Forest <your current table basically>
  ForestID
  <other info about the forest as an entity, but NOTHING about trees>

Species
  SpeciesID
  SpeciesName
  <other info about the organism>

SpeciesInForest
  SiFID <autonumber primary key>
  ForestID <link to Forest>
  SpeciesID <link to Species>
  Rank (Dominant, Secondary, Invasive, Disturbances, ...)
  GrowthHabit (Overstory, Understory, Primary Canopy, Secondary Canopy, Groundcover, Liana, ...)

"Fields are expensive, records are cheap" - rather than have a different FIELD for each type, you could have a different RECORD.

john w. vinson say: Well, I would argue - pretty strongly! - that this is a perfectly classic many to many.

Each Stratum can have many Species.

Each Species can appear in zero, one or many Strata.

I'd see your VegStructure table working like:

VegStructure
  VegStructureID (autonumber primary key)

  VegSpeciesID (foreign key to VegSpecies lookup table)
  LevelID (foreign key to Levels table)
  PrimarySecondary (=1 for primary species at this level, =2 for secondary; will there NEVER EVER UNDER ANY CIRCUMSTANCES be only one, or will there NEVER EVER UNDER ANY CIRCUMSTANCES be 3?)

Levels
  LevelID (Long Integer manually maintained foreign key)
  Level

4      Tall strub
5      Short shrub
<room to expand to other levels such as ground cover, lianas, etc.; that should be a scientific decision, not one forced by your initial table design>

As for Table Lookup Fields, see http://www.mvps.org/access/lookupfields.htm for a critique. The Lookup Field is not portable to other software (such as SQL/Server or MySQL), and conceals the actual content of your field behind a combo box, making it much harder to develop queries and causing much confusion.

scottgem say:
John already covered it. But the key here is you are using field names to define your data. That's what I meant by a horizontal structure. Almost ANY application can be modeled better using a vertical structure. I.E.:

PrimaryKey
ParentKey
DataType
DataValue

So, A1_DomSpp1 would be your data type. Given you a record for each type rather than a field for each type.

=INDEX(B1:B100,SMALL(IF(A1:A100=2784,ROW(A1:A100),1E+100),2))
