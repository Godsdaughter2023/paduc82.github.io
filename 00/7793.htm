tom van stiphout say:

Hi Dayvid,

Thank you for posting early in the process. The number one thing rookies do wrong is the database design, and: welcome to the club. "and 20 fields for recipes" is one of those mistakes: it creates a "repeating group" which is great in Excel but not in a relational database. You need to think skinny tables with many rows.

Before we design the tables, we first need to think about the relationships between them. As one example: each menu consists of one or more recipes, and each recipe is used (over time) in many menus. This sets up a classic Many-To-Many relationship, which requires THREE tables to resolve:

tblMenu

MenuID autonumber PK

MenuDate date/time required uniqueindex

'possible other fields

tblRecipe
RecipeID autonumber PK
RecipeName text(50) required uniqueindex
'possible other fields

tblRecipesForMenu

MenuID long int PK
RecipeID long int PK

Then select these tables in the Relationships window and draw the lines between them to create relationships, and in the dialog that comes up check the box to enforce them.

This is just ONE aspect of your database design. You need to get this right to benefit from the relational database engine. Digest my suggestions and learn as much as you can about database design before creating your first query or form.

john w. vinson say: Just to carry on from Tom's good suggestions: just as you would not have 20 fields for recipes, you would not have 10 fields for ingredients in a recipe. It's another many to many relationship: each Recipe has one, two, or many Ingredients; and each ingredient may be used in one, two or many Recipes.  Again, three tables - follow Tom's example! It's a recipe for a tasty database!

scottgem say:

As noted, the basis for any relational database is good "normalized" design. If you search for the term normalization you will find a lot of info about this essential technique for relational design.

Three tables is not going to be near enough. What you need to do here is start from ingredients and work backwards. 

So you need a list of ingredients that you will use. So very simple:

tblIngredients

IngredientID (PK Autonumber)

IngredientName

Note; I didn't use the generic ID name for the autonunmber and I don't use spaces in object names.

You may want to expand on the ingredients by grouping them into categories. In that case, you add a lookup table for categories:

tluCategory

CategoryID (PK Autonumber)

CategoryName

You would then add a foreign key to tblIngredients for CategoryID. Since you may have different categories for ingredients and recipes, you might want to add a field to indicate the type of recipe or have separate lookup tables for the different types.

The next level is your recipes. So you would have a table like this:

tblRecipe

RecipeID (PK Autonumber)

RecipeName

RecipeCategoryID (FK)

other info about the recipe.

Then, as described, you would use a junction table to add the ingredients for that recipe. Now you would probably need some more fields in this junction table:

tjxRecipeIngredient

RecipeIngredientID (PK Autonumber)

RecipeID (FK)

IngredientID (FK)

Quantity

Each combination of recipe and ingredient might have a different quantity. For example you might use more sugar in a cake, then in a pie. 

Anyway, I hope this gives you a better idea on how to structure your tables. Once you get the structure down you will be able to create a shopping list more easily.

Please feel free to ask for advice along the way.

That's not how you go about inserting the cross-reference. The 9-digit # is essentially random and is allocated by Word. The following code shows how you'd insert a cross-reference at the selection point of the page # on which numbered paragraph 1.1 appears:

Sub Demo()

For i = 1 To UBound(xRefs)

    ReferenceType:=wdRefTypeNumberedItem, _

End Sub
