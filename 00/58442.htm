driverz say:

Why does Windows 7 command prompt need full name (must provide suffix) for executables?

Hello,

I recently got a very weird problem on my Windows 7. After opening a command prompt by running 'cmd.exe', if I try to run an executable e.g. "notepad.exe", I have to type the full name "notepad.exe". If I just type the name without extension like "notepad", I simply got the following error message:

'notepad' is not recognized as an internal or external command,
operable program or batch file.

This annoys me a lot, although I don't mind to type more letters. The actual problem is some programs do not run properly any more. E.g. some program or batch file invoke command without using their full names but only the name without extension. This way the programs or the scripts are "cross-platform" because on other OS like Linux an executable does not need to have any name extension. They do this based on the fact that by default executables on Windows can also be invoked by their name without extension. Due to some unknown reason, on my computer I could not call the executable in this way any more, meaning some of programs on my computer that I'm using daily stop working properly!

I have struggled for days to solve this problem. I have tried to check the Windows registry that is related to exefile. Unfortunately I didn't find a solution. However, I found something interesting. If I start the command prompt with "cmd.exe /e:off" , which turns off the command extension, I was able to run the executables without file name extension! This indicates that the issue is somehow related to the setting for the command prompt extension. But this finding is not a solution. As you can imagine, if I always start the command prompt in this way, batch files (*.bat) that need to use the command extension will fail to run. This is just the case for Windows SDK command prompt or Visual Studio command prompt.

What I need is to go back to 'normal life'. I want to be able to type 'notepad' (without .exe) in the command prompt to start the program without disabling the Windows command extension (by adding "/E:OFF" option). This is in fact the default behavior for Windows 7. I do not know exactly how this issue was triggered. But if you could help, I would be very happy to try your suggestion. (Because I use my computer every day for my work, to re-install everything is definitely the worst solution.)

Thank for viewing!

driverz say: Found the reason by searching more. In fact the user "PATHEXT" environment variable was screwed up by some python program installer. This overloaded the system variable with the same name, and all the executable files with name extensions like ".cmd", ".exe" etc stopped being treated as executable. After setting the user "PATHEXT" to "%PATHEXT%;.py;.pyw" the above issue was resolved.
