Ok, Solved it, hang onto your hats! TL;DR; at the bottom.

All my Microsoft things (Azure, VSTS, Office365, etc) are all under my personal account email address (lets call it *** Email address is removed for privacy ***`). I was trying to associate that account with a VSTS tenant for `foo.org`. So, through my *** Email address is removed for privacy ***` login, I added *** Email address is removed for privacy ***` as a user. I then tried to log in with *** Email address is removed for privacy ***` and the UI said that that account doesn't have access, but my "work" one does and suggests I rename my personal account.

So, this is a no go, so I go on a mission to delete my "Work" account, also named *** Email address is removed for privacy ***`. I had not idea what the password was, so I reset it. This causes an Office 365 tenant to be created (not that I realised that at the time). I took ownership of that tenant via DNS records, changed the ownership of the tenant to *** Email address is removed for privacy ***`, disassociated `example.com` from that Office 365 tenant and remove all the `example.com` accounts.

Right, now back to VSTS. I still can't log on, there is still this mysterious "Work" account. I removed *** Email address is removed for privacy ***` from `foo.org` account and then re-added it. I got the invite in email, and it prompted me for a password (strange, but whatever). I was then able to log into `foo.visualstudio.com`. 

However, I could not access my other VSTS tenants, I had to log out and back in... but now I can't get to `foo.visualstudio.com` without using the password I created before and the "Work" account :/. 

Then I stumbled on the problem. My Work account (*** Email address is removed for privacy ***`) was not managed under the Office 365 example.com tenant, but under the `foo.com` Azure Active Directory! That "Work" account had been created when I added *** Email address is removed for privacy ***` to the `foo.visualstudio.com` VSTS account.

I think I went down the Office365 track as the reset password process to the VSTS login led me to that.

TL;DR; The solution was to disconnect `foo.visualstudio.com` from Azure AD. I was then able to add *** Email address is removed for privacy ***` and lo and behold, I could now access all my VSTSs with the one login.

andreas killer say:
That is possible with VBA only, do as follows:

Right-click on the sheet tab
Choose "View Code"
Paste the following code
Close the VBA editor
Save the file as .XLSM, otherwise the code is lost if you close the file.

Be sure your macros are enabled:
https://support.office.com/en-us/article/change...

Try a scan. How does it work?

Andreas. 

Private Sub Worksheet_Change(ByVal Target As Range)
  Dim Where As Range
  'A change in column A?
  Set Target = Intersect(Target, Columns("A"))
  If Target Is Nothing Then Exit Sub
  'Only if one cell is changed
  If Target.Count > 1 Then Exit Sub
  'Search for the bar code
  Set Where = Columns("A").Find(Target.Value, LookIn:=xlValues, LookAt:=xlWhole)
  'Found in a different row?
  If Where.Row <> Target.Row Then
    With Application
      .EnableEvents = False
      'Remove the current barcode
      .Undo
      'Switch to the Quantity
      Me.Range("C" & Where.Row).Select
      MsgBox "Already exists, increase the quantity", vbInformation, Where.Value
      .EnableEvents = True
    End With
  End If
End Sub

steven. s say:

Hi,

 

The problem occurs when Excel is using OLE to interact with another application and that application is not responding in a timely manner.

 

For example, the workbook may contain an embedded object that is tied to another application. If that application is currently running on your computer you may want to try quitting it and test. You can also try closing both the application and Excel and then reopen your workbook. The trick here is to make sure the other application is closed completely (no process running in the background).

 

This can also occur if an add-in is corrupt, you try to send the workbook as an attachment or there is VBA code in the workbook that automates another application.

 

For more detailed information and steps on how to resolve the problem, read this Microsoft article:

 

Excel 2010 OLE error when you open or use a workbook.
