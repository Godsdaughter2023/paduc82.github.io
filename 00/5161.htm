naps1saps say:

I work for an MSP and recently ran across this issue with a client computer.  It appears to be a profile issue.  Logging in as another user, I don't see this problem.  Best way to fix this looks to be to reset your profile.

  1. Make sure to export any favorites from Chrome/FireFox.
  2. Log in as another admin user.  If there isn't one, create a temporary one.
  3. Rename the user folder of the offending login to something like *username*.old
  4. Go to the registry editor to the location: HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList
    Delete or rename the S-#-#-#*etc* folder that contains the offending login name.  You'll see what I'm talking about.
  5. Log out and log into the offending profile.  It will re-create the profile.
  6. Once complete, copy the data back over from the user folder you renamed in step 2.  Do not copy the appdata folder.  If anything is amiss like a setting or configuration, it will be located in that old appdata folder and should be able to be imported by copying it over if needed.

There may be a better way but this is the way I do it.  I haven't confirmed if this works yet but I believe it should.

hansv mvp say:

If you save your code in your personal macro workbook, ThisWorkbook is Personal.xlsb. Use ActiveWorkbook instead, or omit the reference to the workbook altogether.

Also, it isn't necessary to activate each sheet:

Sub unprotectsheet()
    Dim ws As Worksheet
    For Each ws In Worksheets
        ws.Unprotect
    Next ws
End Sub

robert aldwinckle on forums say:

Got the message that I needed updates.  I have been through the updates on the page several times and get the message "updates already installed" or "updates not applicable to this computer".

A common problem is that there is a distinction between updates being installed and applied.  Unfortunately troubleshooters do not do a good job of checking for this or drawing correct inferences from the facts which such checking would provide.

For example, assuming you can open IE and its Help, About, there may be information there which could contradict an assumption about your IE8 being up-to-date.  In later versions I could tell you to try using a link which is provided in the dialog with Update versions: but I can't remember if IE8 presents such a diagnostic.  More precisely, you can look at the versions of modules which are being loaded (e.g. in ResMon or msinfo32) and see if they would be consistent with the assumption that what you have installed is what you have applied.  If not, fixing that problem would be a better approach than trying to install a new version of IE and hoping that that would fix that problem.  As you have found that is unlikely to happen.

C.f.

https://answers.microsoft.com/en-us/ie/forum/ie11-windows_7/internet-explorer-11-windows-7-for-64-wont-install/5209598b-ac87-4ee5-a8f4-cbb7b624dda8?page=3

I keep forgetting about CheckSUR for W7 users.  It allows them to do manually what DISM can do for W8 users automatically.  So using that may be an alternative to doing the kind of analysis that I would normally advise, in case that's not "your cup of tea".   ; )

 

Good luck
