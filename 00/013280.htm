gary's student say: Here is an example of handling failure to add a sheet:

Sub SheetAdder()
TOP:
sName = Application.InputBox(Prompt:="Enter the new sheet name", Type:=2)
On Error Resume Next
Sheets.Add.Name = sName
If Err.Number <> 0 Then
    Err.Number = 0
    Application.DisplayAlerts = False
    ActiveSheet.Delete
    Application.DisplayAlerts = True
    MsgBox "Bad name chosen......choose again"
    GoTo TOP
End If
MsgBox "O.K."
End Sub

This will handle both duplicated tab names as well as illegal tab names.

a. user say:

Are you committed to generating your Word document by generating XML via a transform (e.g., perhaps you are doing it on a server) or could you actually use other approaches, e.g.
 a. generate the document using a .NET language and the Open XML SDK?
 b. Automate Word to do it?

Because doing either of (a) or (b) is likely to be significantly simpler than doing it via a transform. In the case of (b) you would
 - open Word
 - open your saved "skeleton" document in Word
 - set some values (e.g. ActiveDocument.Variables("Client").Value = ' some value that you get from Access, or from your exported XML
 - update your DOCVARIABLE fields
 - save and close

If you really need the transform approach...

To transform xml into a Word document your transform needs all the WordProcessingML code that the document would contain, /plus/ all the XSL code necessary to insert your field values into that code at the correct points.

I don't think you have posted all your transform code, but if it does have that, it can't work.

You mention that you want Word 2010 (for example), and although the process is similar, it's actually more complicated than generating a Word 2003 XML format document because you have to generate "Flat OPC" format, with all its parts and references. It can also become complicated if for example you need to generate a Word document with a variable number of parts (e.g., multiple sections with multiple headers/footers).

Since Word 2010 can (usually) open Word 2003 XML documents, it can be easier to generate that instead. I'd say it's probably easier to start using the Word 2003 format because there are toolkits to help you do that (though whether they all still work with Word 2010 I cannot tell you).

For example,

The Word 2003 XML SDK
http://www.microsoft.com/download/en/details.aspx?id=21923

I cannot remember whether this contains the "Inference Tool" that helps you create an XSLT from a Word 2003 XML file - if not, it is here:
http://www.microsoft.com/download/en/details.aspx?displaylang=en&id=3412

but again, I don't know whether you will be able to get that to run with more recent versions of Windows than XP.

Unfortunately, I couldn't find a starting point example on the web.

As it is, it doesn't make sense, but I'm guessing you want part of the ArborID between, before or after some number of dashes.

If that's a reasonably close guess, then it's a poor table design issue in that there is more than one value in the ArborID field.  Instead you should have two or three fields, each with the individual parts of the ArvorID.  Then you wouldn't need to parse the field and it would be a trivial expression to put the parts together on forms/reports whenever people need to see it.
