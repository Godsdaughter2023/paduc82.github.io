Your structure is all wrong. an IIF statement has only 3 arguments: the expression to evaluate, the return if true, and the return if false. You can nest them for more that one test.

Rank: IIF([CTRank]<5, 1,IIF([CTRank] between 5 and 20, 2, IIF([CTRank]>20,3,NULL)))

If you want to usesome other value other than NULL you can change it.

  2. Choose VIEW>Slide master.
  3. The Slide Layout responsible for the selected slide will be in the main window.
  4. If the text placeholder reads Click to edit Master title style, then the wrong type of placeholder has been used. Replace it with a content or text placeholder by using SLIDE MASTER>Insert Placeholder.

  6. Close the Slide Master to return to your presentation, then save it.

gord dibben say:

If you want a static timestamp you must enter the time manually using the shortcut key combo to do so.

 

CTRL + SHIFT + ;  for time only.    CTRL + ;   for date only

 

To get both   CRL + ;   <space>  CTRL .+ SHIFT + ;

 

Alternative is VBA event code to enter the time in a cell when you type in another cell.

 

Sample worksheet code for time entry.  Type in a cell in B, F or L column and time will be stamped in adjacent cell.

 

Private Sub Worksheet_Change(ByVal Target As Range)
 On Error GoTo ws_exit:
 If Intersect(Range(Target(1).Address), _
        Range("B:B, F:F, L:L")) Is Nothing Then Exit Sub
 Application.EnableEvents = False
 Target.Offset(0, 1).Value = Format(Now, "mm-dd-yyyy hh:mm:ss"
ws_exit:
     Application.EnableEvents = True
 End Sub

 

To enter the code, right-click on the sheet tab and "View Code".  Paste the code in there.  Alt + q to go back to the sheet.

The Excel team has released a change in HTML/XLS file behavior in today’s security update for Excel 2010, 2013, and 2016. Excel will warn about the mismatch between the file extension and HTML content, but will now open the workbook in Protected View as an additional layer of security. If you trust the workbook, you can then enable editing. XLA and XLAM files are not part of this change, they will continue to not open in Excel.

 

How do I get the update? It depends on which version of Officeyou have.

  • Office 365 subscription (Click-to-Run)—install the latest updates

  • Windows Installer version (MSI)—you could wait about a week for the next security update for MSI, and it will update automatically. Or, to get the fix today, use the Download Center to get the KB for your version of Office:

    • Office 2016: https://support.microsoft.com/kb/3115438

    • Office 2013: https://support.microsoft.com/kb/3115455

    • Office 2010: https://support.microsoft.com/kb/3115476

 

Issue recap:

The Excel team has made a change in the behavior of certain file types to increase security. This change came in the security updates KB3115262, KB3170008, and KB3115322. Previously, when you tried to open an HTML, XLA, or XLAM file with an .XLS file extension from an untrusted location, Excel would warn about the mismatch between the file extension and content, but would still open the workbook without Protected View security. After the security updates Excel no longer will open the workbook because these files are not compatible with Protected View and there is no warning or other indication it was not opened.

 

Workarounds for .xla and .xlam files: https://support.microsoft.com/kb/3181507

 

Thank you for your patience and input—community suggestions drove our decision to change this behavior in a way we hope will be much better for your workflows!

What is lacking from that user's message is version information. Most commonly, problems related to printing are the direct result of Office, OS X and/or printer drivers being behind on updates. It's imperative that older software [such as Office 2011] be kept fully updated when running on OS X versions more recent than that for which they originally were designed. From what I've seen, this seems to be particularly true of OS X 10.11.x [El Capitan].

Pursuant to that, my experience suggests that if the content looks as intended on screen but fails to print as displayed, the fault typically lies with the print subsystem. What is shown in any print preview (including the Quick Preview of later OS X versions) is based on feedback from the subsystem. IOW, if the document looks correct on screen, Word is holding up its end. The aberration in the preview display or on paper usually is the result of the print subsystem incorrectly interpreting the information sent from the program.

Because of the variables involved, we can't always rely on Apple pushing out the most current revisions to printer drivers. It's often overlooked, but in many cases more current drivers are available from the manufacturer's web site. I've even had a few instances where the mfr recommended using a driver different from the one supplied for a particular printer.

It's pointless to speculate further, though... There just isn't enough detail in that user's message to enable pinpointing the actual cause. Of course, if the cause isn't known it's virtually impossible to suggest a cure :-)

john korchok say:

I'd recommend that you revise your how-to page. Instead of inserting one section break, converting the rest of the document to landscape, inserting another section break, then converting the remainder back to portrait, try this:

Insert 2 section breaks into the portrait document, one at the beginning and one at the end of the area to be changed. Then switch the orientation of just the pages between the 2 breaks. You'll get a more reliable result.

The reason is that Word takes the margins of your portrait page and rotates them 90 degrees when applying them to a new landscape page. (Bottom margin measurement is applied to left margin, left margin to top margin, etc.) Then it rotates them back when switching to portrait, but it doesn't always get them right, especially if you fixed the margins for the landscape area before inserting the second break.
