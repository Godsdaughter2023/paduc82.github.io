In my first example, I used an inner join for each of the table elements.  This works, but if a book hasn't ever been checked out, a record for that book will not be displayed.

So here it is necessary to make an outer left join from Books to BookCheckOut.  An out left join may be desirable for BookInventory as well, depending on whether or not an inventory value should be displayed if the book hasn't ever been added.

SELECT Books.BookID, Books.BookTitle, IIf(IsNumeric([CountOfBookInv_FK_BookID]),[CountOfBookInv_FK_BookID],0) AS Inventory, IIf(IsNumeric([CountOfBookCO_FK_BookID]),[CountOfBookCO_FK_BookID],0) AS ChkdOut, [Inventory]-[ChkdOut] AS Available
FROM (Books LEFT JOIN SELECT_BookInventory ON Books.BookID = SELECT_BookInventory.BookInv_FK_BookID) LEFT JOIN SELECT_BookCheckOut ON Books.BookID = SELECT_BookCheckOut.BookCO_FK_BookID;

sockatume---01 say:

When your employer turned off data for your handset, they blocked A-GPS, a service which speeds up GPS locks by sending the phone approximate satellite positions over the network. Disabling this means it takes longer for the device to get a fix "from cold", although it'll still be relatively quick if it's recently had a lock. (The satellites transmit the same sort of data about the approximate positions of their neighbours.)

Bluetooth GPS receivers generally have very large antennas and so they can get by without A-GPS. It's likely that the old PDAs you had were similar, they would've had a large antenna because they couldn't rely on the A-GPS service to speed up locks.

 

I think you'll get this issue with any modern phone, they're all switching to smaller antennas and a combination of A-GPS, network positioning, and wi-fi positioning. You can try pairing up your bluetooth GPS with your 5800, it should be as fast on that handset as it was on your PDAs.

A-GPS works similar to DGPS (differential GPS) which uses land based beacons in conjunction with the satellites to improve the accuracy of the system In the case of surveyors equipment down to the sub-metre level. djsteele, mvp say:

It's likely an issue with folder permissions. In order to share a Jet database, all users must have read/write permissions on the folder where the .MDB file exists. That's because the first user to connect to the database creates a locking file (.LDB) and subsequent users must be able to write to that locking file. (When the last user disconnects, Access will try to delete the locking file, but it's actually not essential that users have Delete permission on the folder).
