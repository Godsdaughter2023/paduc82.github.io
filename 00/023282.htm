Access 2016 -- pulling multiple fields at once from a related table - Microsoft Community

ersatzdev say:

Access 2016 -- pulling multiple fields at once from a related table

Hello!

I've been attempting to develop a table (and the related form) that allows for someone to pull the information from a dropdown in one column and the data in related columns pops up automatically.

For example, Table1 has the fields Name, Unit, Location.  Table2 has these same fields and the additional fields OrderNo, OrderDate, a Yes/No for OrderComplete, and Notes.  It needs to reference the fields from Table1 multiple times, but not on the same line.  

Is it possible to set up Table2 so that if I select the Name on a dropdown, it will automatically populate the Unit and Location fields from the relevant record in Table2?  That is to say, if I have Alex from the Sales unit in Madgascar in Table 1, and he puts in an order, can I select Alex and the relevant fields in Table2 will autopopulate with Sales and Madagascar, respectively?

Thank you very much!

ken sheridan say:

You can, but you should not.  Table 2 should not have Name, Unit or  Location columns at all.  Instead it should have a column such as EmployeeID referencing the primary key of Table 1.  Name cannot be used as a key as personal names can legitimately be duplicated, so a surrogate numeric key with distinct values is essential.  This would normally be an autonumber in Table 1, and a straightforward long integer data type in the foreign key EmployeeID in Table 2.

The reason for Table 2 not having Unit or  Location columns is that these would be transitively dependant on the primary key via EmployeeID, so the table would not be normalized to Third Normal Form (3NF), and consequently wide open to the risk of update anomalies.

With the single foreign key EmployeeID in Table 2 you can simply base the form on a query which joins the two tables on EmployeeID and bind controls in a form or report to the Unit and Location columns from Table 1.  By setting the Locked property of these controls to True (Yes) and their Enabled property to False they become read-only, so cannot be inadvertently edited by the user.  Bind a combo box to EmployeeID, setting it up to hide the first column and show the employee's name, e.g.

ControlSource:   EmployeeID

RowSource:     SELECT EmployeeID, EmployeeName FROM Employees ORDER BY EmployeeName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.  Note however that if there are two employees of the same name, the above would not enable them to be distinguished in the control's drop down list, so you would need to include other columns in the list, e.g. the Unit and Location columns, to distinguish between them.  Even this is not bulletproof as you could have two employees of the same name in the same unit and location, so in this sort of context some value guaranteed to be distinct, e.g a payroll number, is really required.  You might think this is unnecessary with your current employees, but Murphy's Law* says otherwise.

BTW, do not use Name as a column name.  As the name of a built in property in Access it is a 'reserved' word and should not be used as an object name.  Use EmployeeName or similar.

* Murphy's Law:  'If something can go wrong, it will go wrong.'

scottgem say:

Please see my blog on Displaying Data from Related Tables. 

Notice the word DISPLAYING. As Ken has advised you should NOT be populating redundant data. One of the principles of a Relational Database is that data exists in one place and one place only. 

In fact I see several other potential design flaws in your database.

1) People names. Unless you will be storing just a first name, people names should be broken up into its components. I generally use 5 fields for people names; (Salutation (Mr, Ms, etc.), first, middle, Last and Suffix (Jr, Sr, etc.).

2) Unit and Location. these sound like that should be lookup tables with coded foreign keys. You don't no want to repeat Madagascar for every person at that location. You should have a table with a Autonumber Primary ky and use that as your foreign key.

3) Rather than a Y/N for OrderComplete, what about a CompletedDate field. This way you know the order is complete because the CompletedDate is filled in and you know when it was completed.
