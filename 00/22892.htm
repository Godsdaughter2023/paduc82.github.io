  • Select individual Skydrive documents for downloading and use offline.
  • Install on any Android phone with v4.0 or higher (which would be most sold in the last few years).

What CAN’T it do?

  • Save to anywhere except your Skydrive account, Office 365 or Sharepoint.
  • Save documents to the device folders.
  • Open password protected documents
  • Easily select multiple documents or folders to be synced for offline use.
  • Edit ‘old format’ documents (doc, xls or ppt). They have to be ‘Save As’ to the new (‘docx’ etc) formats
  • Install Office for Android on any Android tablet including ‘mini’ tablets like the Nexus 7.
    • Unlike Office for iPhone which can be installed on an iPad or iPad mini
  • Buy an Office 365 subscription directly in the app (you can on Office for iPhone).

What about tablets?

Microsoft has blocked Office for Android from Android tablets devices.

Microsoft’s recommendation is Office Web Apps on tablet devices. That recommendation has more to do with Microsoft’s corporate strategies and preferences than a practical suggestion for tablet owners. Office Web Apps is fairly good but no replacement for an installed app, particularly when offline.

ken sheridan say: As there are 'mandatory' columns which can legitimately be Null when the row is inserted into the table, but must at some later stage be given values, then you cannot enforce this in the form's module or table definition.  In addition to your '2-stage' approach you might also think about an independent housekeeping exercise whereby you can periodically check for rows with Nulls at any of the 'mandatory' column positions.  A form bound to a simple query which is restricted to WHERE SomeColumn IS NULL OR SomeOtherColumn IS NULL and so on would suffice

dirk goldgar say:

I did not pay too much attention to the data validation.  After about one year of usage, I notice lots of incorrect and missing data.  The users are commonly not self-discipline.  I would like to prevent from missing data with the additional measures.  On the form, the users will fill in about 18 fields from beginning to end.  Thus I cannot put the restrictions in the form and now I plan two levels: add a new record and complete the record.  When a new record is created, some basic fields must be completed before saving.  You are right Main Cat is one of fields must be completed when create the record.  Other fields data may not be available at the beginning and I have allowed them to fill them later.  However, once the project completes, they have to complete all mandatory fields.

So there are two set of requirements:  those fields that must be filled before the record can be saved even the first time, and those fields that must be filled before the project can be marked as complete.  Assuming that the "project complete" data state can be recognized by looking at the fields in the record, I don't see why you can't have two sets of validations in the form's BeforeUpdate event: one set that applies no matter what, and one set that only applies when trying to save a "project complete" record.

You probably should also set the Required property, in the table design, of those fields which are always required, even for non-complete projects.  For the remaining fields, it's possible to create a validation rule for the table to prevent a record from being saved in the "process complete" state with certain fields Null.  However, I prefer to have the form's BeforeUpdate event set up to handle the situation more gracefully, and have the table-level protections as a fallback.
