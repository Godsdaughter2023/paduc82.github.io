a. user say:

If your want the task to run under the XP Administrator account and usually on a new build that account has no password, if you try to create a task with that account and it has no password or you enter the password incorrectly, you will see a message like this:

The new task has been created, but may not run because the account information
could not be set.  The specific error is 0x80070005:  Access is denied.

The task will still be created, but will not run properly.

It is best practice for STs to assign them to run under an account that has a password.  There is a way around it, but that would go against the best practices for STs.

If you are compelled to use the Administrator account for some reason, assign it a password and use it for your Scheduled Task.  The trouble with that might be that someday you might forget that password and could be in a small jam if you need the Administrator account for something. 

What some folks do is create a separate user account with an account type of Computer administrator (with a password) and use that account to run your STs. 

You can also use your user account - if it has a password, but depending on how you set up your ST, when the time comes for the job to run, it may just pop up on your screen while you are doing something else - that can be annoying and a good reason to create a separate user account just to run your STs.
