nrgins say: Your signature says you're an Access MVP. You should know what the advantages are of putting an audit log in a data macro in the back end, instead of using VBA code in the front end. 

First, putting it in the front end only ensures that changes made to the front end would be caught. Sure, you can lock down the back end. But can you guarantee that no one will get into the tables or into the back end or link to the tables from another db and make modifications? Using back end macros catches all.

Second, using VBA code requires you to write VBA code everywhere there's a change. Yes, you can write a single audit log routine and call that routine from each form's BeforeUpdate event. But what about changes made in code, through recordsets or queries? You'd have to manually write to the audit log in that routine, which leaves open much room for error of that either being forgotten or the values not written correctly. By using a back end macro, you don't have to think about it at all. Whenever the data is updated, whether by the form, a query, a recordset, or anything else, it will be logged.

So it's a far, far superior method, even though I realize that people have used VBA-based audit logs for many years and they "work fine." But now that table triggers are available in Access 2010, it's time to realize that a superior method is available, and it should be utilized.

And, yes, you can write it in VBA and call it from a table trigger (even though that's not the intended purpose of the table triggers). But that would require the VBA code to be in both the front end and back end (in case anyone updates from the back end). And if anyone updates from outside the db (either through an Access linked table or ODBC), then the audit log would fail, and the changes would not be records. Using a data macro for everything avoids all of those problems.

Hi Kenny,

Could you let us know how does that user upload the notebook to OneDrive? Is it a local notebook before, and that user directly upload the folder or notebook file to his OneDrive?

OneNote is different from other Office apps. It's not suggested directly upload notebook related file to OneDrive or OneDrive for Business. The best practice is to create a notebook in OneDrive or OneDrive for Business. If we want to move a local notebook to cloud, we should select that notebook in OneNote 2016>click File>Share>Move Notebook>enter the related account for OneDrive/OneDrive for Business or select the existing OneDrive/OneDrive for Business services>Move.

Meanwhile, once we open a notebook in OneNote 2016, we don't have to close or reopen it. We can keep it open in OneNote 2016. Just close the OneNote app after finishing our work.

You could let that user try steps mentioned above to move the local notebook to cloud. If he cannot get the local notebook any more, he could first create a new notebook in his OneDrive/OneDrive for Buisness, and then manually move/copy sections from the problematic one to the new one in OneNote 2016.
