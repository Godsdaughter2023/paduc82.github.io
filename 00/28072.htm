jllatham say:

I'll tell you how the company I work for handles this kind of thing:  Besides an actual hire date, we keep an "adjusted hire date".  It is used both for situations such as you've described where an employee returns to work with the company within a time period that "grandfathers" previous time served (oops, make that time on the job :)).   It is also used for people we hire as part of a contract we may be taking over and we want to honor their time on the job with the previous contractor, so we have an actual hire date (date we put them on our payroll) and an adjusted hire date (one that is, in effect, the date they went to work for the previous contractor).  Then all benefits for all employees that are time based are calculated using the adjusted hire date.  For most employees the actual and adjusted hire dates are the same, but not all.  But having those types of benefits calculated on the adjusted hire date for all employees means one-rule for all.

 

What we do for the particular employee you mention is give her an adjusted hire date that is 1 year after her original hire date.  Let's say that she came to work for you originally on 7/1/2006.  She terminated that employment on 7/1/2012, so she had 6 years tenure at that point.  She stays gone until 7/15/2013 when she comes back to work for you - so her adjusted hire date would be 7/15/2007: 6 years before she actually came back to work for you.

 

Now, the actual dates probably don't work out that pretty for you.  You can use Excel to help determine her adjusted hire date.  Use the undocumented DATEDIF() function.  It takes 3 arguments, as: =DATEDIF(earliest date, later date, measure of difference) where "measure of difference" can be years, months, or days (and some combinations we don't need).  We want days.

You need:  =DATEDIF(date she quit, date she rehired, "d")  That gives you the number of days you need to ADD to her original hire date to get an adjusted hire date.

 

The graphic below shows how to set it up and get an adjusted hire date.  Just change the 3 dates shown to her actual original hire date, quit date, rehire date, and use the formulas I've shown:

vijay a. verma say:

You will have to use MICROTIMER for a very precise measurement. This is provide by Charles Williams (https://fastexcel.wordpress.com). He has also provided other macros to measure many other timings https://msdn.microsoft.com/en-us/library/ff700515%28v=office.14%29.aspx

Below is a piece of code to use MicroTimer (You can omit Result)

     Dim Result As String, Timer As Double
    Timer = MicroTimer()
    ---- Your code -----
    MsgBox "Result=" & Result & vbNewLine & "Time Taken=" & Format(1000 * (MicroTimer() - Timer), "0.00") & " Milliseconds"

Below is the MicroTimer code which you need to copy and paste in a standard module

#If VBA7 And WIN64 Then
    Private Declare PtrSafe Function getFrequency Lib "kernel32" _
    Alias "QueryPerformanceFrequency" (cyFrequency As Currency) As Long
    Private Declare PtrSafe Function getTickCount Lib "kernel32" _
    Alias "QueryPerformanceCounter" (cyTickCount As Currency) As Long
    #Else
    Private Declare Function getFrequency Lib "kernel32" _
    Alias "QueryPerformanceFrequency" (cyFrequency As Currency) As Long
    Private Declare Function getTickCount Lib "kernel32" _
    Alias "QueryPerformanceCounter" (cyTickCount As Currency) As Long
#End If

Function MicroTimer() As Double
' Returns seconds.
    Dim cyTicks1 As Currency
    Static cyFrequency As Currency
    MicroTimer = 0
' Get frequency.
    If cyFrequency = 0 Then getFrequency cyFrequency
' Get ticks.
    getTickCount cyTicks1
' Seconds
    If cyFrequency Then MicroTimer = cyTicks1 / cyFrequency
End Function

tom ogilvy say:

If you want to predict how long your macro will take, you would need to put in varying amounts of data and then run the macro and record the time (using the timer suggested).    You could then graph the data and fit a trend line to the the data.   The more test values you have the better your estimate will be (if the graph appears linear, then it will probably be a pretty good estimate, but I suspect it won't be linear).    Forecasting outside your experience area is always imprecise.   

Also, there are external factors (other things your computer is doing) that will affect these times, but this would be about the best you can do in terms of estimating how long your macro will run in advance of running it.

jay freedman say:
All versions of Word use some of the same registry entries. When you use Word 2010, it sets the entries the way it likes them. Later you run Word 2007, and it says "Hey, the registry isn't set the way I left it. I need to fix that!"

The fix is (surprise!) to add another entry to the registry for each version to say "don't fix that."

Read http://word.herbtyson.com/?p=14 for the details. The version numbers you'll use are 12.0 for Word 2007 and 14.0 for Word 2010.

Using this procedure, I have four different versions of Word on the same computer, not stepping on each other's toes.
