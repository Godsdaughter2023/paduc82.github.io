You can create a query based on the table.

Add the Employee and Manager fields.

In the Criteria row of the Manager Column, enter

 

[Enter Manager]

 

When you switch to datasheet view, you'll be prompted to enter a manager. After clicking OK, the query will display only the records for the specified manager.

 

A more user-friendly solution is to create a form frmParameter with a combo box cboManager that lists the managers.

Change the parameter prompt [Enter Manager] to [Forms]![frmParameter]![cboManager]

The query will now display all records corresponding to the manager selected in the list box.

 

You can use the query as record source for a report.

ken sheridan say: Essentially this is the classic bill of materials  problem.  If you want to return the hierarchy down to a fixed number of levels its quite simple, you simply chain a fixed number instances of the table together in a query.  If on the other hand you want to return an arbitrary number of levels, which a true BoM would involve, then it becomes more complex as it requires recursive querying, which is not possible in Access.  It can however be simulated.

Many years ago I wrote a little demo application for this which you'll find as BoM.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

The zip includes a file Bom.accdb for creating a true BoM by simulating recursive querying.  This works by creating a temporary external database at runtime and building the BoM as a table in it.  It also includes a file PartsTree.accdb which uses the simpler option of a  query down to a fixed number of levels.  The prime purpose of this file is to illustrate how to create a tree-structured report, but the query illustrates how you could approach your problem.  Just think of a manager an employee responsible to the manager as a minor part.  All but the lowliest of employees are both of course.
