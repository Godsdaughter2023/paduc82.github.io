Consider creating a form with a couple text boxes formatted as Short Date. Name them something like txtFromDate and txtToDate. Then use the control wizard to add a command button named "cmdOpenReport" that opens the report. Change the code to something like:

Private Sub cmdOpenReport_Click()
On Error GoTo Err_cmdOpenReport_Click

    Dim strWhere As String
    strWhere = "1 = "
    If Not IsNull(Me.txtFromDate) Then
        strWhere = strWhere & " AND [YourDateFieldName] >=#" & Me.txtFromDate & "# "
    End If
    If Not IsNull(Me.txtToDate) Then
        strWhere = strWhere & " AND [YourDateFieldName] <=#" & Me.txtToDate & "# "
    End If
   
    stDocName = "YourReportName"
    DoCmd.OpenReport stDocName, acPreview, , strWhere

    Exit Sub

    MsgBox Err.Description
    Resume Exit_cmdOpenReport_Click
   
End Sub

dhookom say: Parameter prompts in the criteria of queries is fairly common but have too many limitations. IMO, users should never be subjected to them.
  • They don't allow defaults
  • They don't allow picking from a list
  • They don't allow seeing multiple values at once
  • They aren't easily re-used
  • You can't check their integrity
  • You can't limit one based on the value of another
  • They don't support input masks or formats
  • Their values can't be displayed in a report that returns no records
  • They are annoying like pop-ups
  • They are difficult to query with a foreign key value that has no meaning
  • You rarely if ever find a parameter prompt type interface in any decent software application

scottgem say:

After i posted this i took a break and then came back to searching. I ended up finding a very, VERY close second option that didnt require sql. I ended up putting [Enter Date] in the Criteria cell on the query in design mode. 

And why do you think you weren't using SQL? Query Design mode is just an interface to generate a SQL statement. All Access queries are SQL statements.

I agree with Duane 100% about using parameter prompt queries. To add to his list, if the date doesn't exist or the user enters a wrong date. The report is blank with no explanation to the user. 

You can eliminate these problems without actually dealing with SQL. If you do as Duane suggested and have a form with controls for start and end dates, then all you have to do is enter the following criteria instead of your [Enter Date] parameter:

BETWEEN Forms!formname!StartDate AND Forms!formname!EndDate

use the actual formname and name the 2 textboxes StartDate and EndDate. 

Note: If you ever want to see the SQL generated by Query Design mode. Just change to SQL View. SQL is not a difficult language to learn. It is mostly plain english:

SELECT field list FROM tablename WHERE fieldname = some value.

Is that difficult to understand?

ken sheridan say: Another point here is that parameters of DateTime data type should always be declared.

This does two things:

1.  It prevents any inadvertent misinterpretation of the parameter value as an arithmetic expression rather than a date, which would give the wrong results.  With Scott's example the query would begin like this:

PARAMETERS Forms!formname!StartDate DATETIME,
Forms!formname!EndDate DATETIME;
SELECT etc….

Even with a crude system generated parameter prompt the parameters should be declared in this way.  It can also be done via the Parameters dialogue in query design view.

2.  It caters for regional variations in the date/time format used on the system.  This is also an important factor when building a date literal in code, as in Duane's example.  His code would fail if used on my system with its UK date formatting unless amended in the following way:

strWhere = strWhere & " AND [YourDateFieldName] >=#" & Format(Me.txtFromDate,"yyyy-mm-dd") & "# "

strWhere = strWhere & " AND [YourDateFieldName] <=#" & Format(Me.txtFromDate,"yyyy-mm-dd") & "# "

Note the use of the ISO standard for date notation of YYYY-MM-DD to ensure international unambiguity.

For examples, including restiction by a date range, of how to restrict a report via a dialogue form take a look at MultiSelect.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form definition accordingly.
