Applying the styles was easy after assigning each to a hotkey (Ctrl+Alt+M for Metric, Ctrl+Alt+N for Not :-) ). I wrote a short macro that turns the Hidden property on/off in those two styles, and assigned the macro to a button in a custom toolbar: one for SAE, one for Metric. I also wrote a tiny macro to toggle Show/Hide Text, because though I normally like to see the hidden text (gotta proof that too), having two graphics appearing where there should only be one really messes up pagination, plus moves them out from under their callouts. Here's the VB code for changing the Hidden property and toggling hidden text; the styles are up to you. It's a time-saving, graceful, low-code, one-click solution to conditional text. Sub Metric()
    With ActiveDocument.Styles("Metric Text").Font
        .Hidden = False
    End With
    With ActiveDocument.Styles("SAE text").Font
        .Hidden = True
    End With
End Sub

Sub ToggleHiddenText()
    If ActiveWindow.View.ShowHiddenText = True Then
        ActiveWindow.View.ShowHiddenText = False

john mcghie say:
Everybody has discovered this problem :-)

You have to use tracked changes VERY carefully, or they do indeed cause crashes.  Each change you track keeps the old text and the changed text in the file.  Word has to work out on the fly which bits to use.  The inside of the file becomes extremely complex.  If you do any of a number of things while editing, you can damage the code in the document and Word will crash.

This is a well-known weakness in Microsoft Word.  There are several ways around it:

1)  Always, Always display the non-printing characters while working with tracked changes.  And always display both the insertions and the deletions.  It means you get a fairly complicated display, but makes it easy to avoid corrupting the document code while you are editing.

2)  Don't use tracked changes at all.  This is what I do.  I keep a copy of the document before I start working.  At the end of my editing, just before I deliver the document to the customer, I use Compare Documents (look it up in the help) to mark all the changes by comparing the original with the current version.  Your customer will get exactly the same thing: all the changes in the document will be marked.  But since Word inserts the changes, there is no chance of corruption causing crashes.

3)  Before starting work on a document, ACCEPT (or REJECT) all the changes in the document and SAVE and close the document.  The accept or reject will clear all of the previous changes in the document when you close the document.  The accepted/rejected changes are not actually removed from the file until you close.  WHen you re-open, the internal code of the document will be clean and less likely to suffer corruption.

This forum is part of the "support" that Microsoft provides.  This is not a problem that Microsoft can solve (they have made several attempts: Word 2011 is the best attempt yet).  The real problem is that Word allows users to directly edit in a file that also contains computer code.  It requires a lot of care and knowledge to do much of this without the difficulties you are experiencing.  After 30-odd years of practice, I could probably do this successfully, but I choose not to: the risk of me making a mistake is too great.

My suggestion is Number 2: Use Compare Documents.  That's what I do.  It also makes editing a lot easier and quicker :-)

Cheers

Hi Chris:

Basically "Yes". It applies to every version of Word, on any platform.

The issue is that users are typing in the middle of XML code that the "system" is trying to read.

This was always an unstable design.  Within limits, you would get away with it, for a few changes in a short document.

Complex (changes within changes upon changes overlapping changes...) and complex documents (lotsa pictures and tables...) reduced your mileage proportionately.

The only one I have seen that got this right was Borland Sprint, where each line of text was written to disk at the end of each line, and where all the formatting and tracking etcetera were stored in a "shadow line". For example, in each disk file, the even-numbered lines contained the text, the odd-numbered lines contained only code.

It had other issues: the company going broke being amongst them.

As other users have pointed out, this problem has been around "forever".  Each version of Word is "a little" better.

MS security essentials unreliable schedule running - Microsoft Community

ron_w say:

MS security essentials unreliable schedule running

Related to: Microsoft Security Essentials Consolidated FAQ

Question:    Why doesn't Microsoft fix the scheduling error in MS Security Essentials?  I run Malwarebytes using the scheduler and it never fails. The problem with MS security essentials has been there for years and MS does not bother to fix it. WHY? 

greginmich say:

Real-time protection is the main defense in MSE, so we shouldn't attach too much importance to scheduled scans. In fact, scheduled scans are optional (but recommended) in MSE – just because they don’t normally detect active malware. Nevertheless, most people will still want to run scheduled scans because they’re often able to detect and clean up inactive malware components and they’re also capable of finding malware that’s “hiding” in archive files where real-time protection doesn’t look.

The problem with the MSE scheduled scan isn’t that there’s some kind of “error” or “bug” per se; it’s just that two of the key features that make scheduled tasks run reliably have been disabled for the MSE scheduled scan. The ability to “Run task as soon as possible if a scheduled start is missed” was actually built into MSE (in a watered-down version) as a “catch-up” scan – which is a scan that runs at startup after several scheduled starts have been missed. But this “catch-up” scan is now disabled in the registry because the developers believed that running the scheduled scan at startup might be too intrusive for many people. In most cases, this means that the MSE scheduled scan simply won't run at all unless the user reschedules it for a time when the PC is powered up. So in recent versions, a “Scan now” prompt is presented to the user after a couple of weeks have gone by without the scheduled scan running.

The other thing that’s missing in the MSE scheduled scan is the ability to “Wake the computer to run this task”. Enabling this option would definitely increase the chances of the scheduled scan running for most users, provided that they had already rescheduled the scan for a time when the PC is powered on. The rationale for disabling this option isn’t entirely clear, but the effects of disabling it are perfectly clear – the scheduled scan won’t run if the system has idled to sleep, and a scheduled Full Scan won’t run without being interrupted by sleep transitions.

For many users it’s not too difficult to tweak the MSE scheduled scan into something that will do the job tolerably well:

http://answers.microsoft.com/en-us/protect/forum/mse-protect_start/mse-scan-schedule-off-by-days/f0e68e51-02d6-4fce-bf0b-999b1f0268c8

…and for users who won’t settle for anything less than perfect reliability, we do have the option of scheduling your own scan:

http://answers.microsoft.com/en-us/protect/forum/mse-protect_start/running-mse-full-scan-as-a-scheduled-task/f627f5ce-0b74-4b1d-be64-d6ceeb4d9b02

Once you’ve set up your own scheduled scan according to the instructions, and you’ve enabled the “Wake the computer to run this task” option, your scheduled scan will run anytime the PC is powered up. Now all you have to do to achieve perfect reliability is to make sure that the scheduled scan will run even if the PC isn’t powered up at the scheduled time. That’s not as impossible as it sounds, because all you have to do is add your own “catch-up scan” by enabling the “Run task as soon as possible after a scheduled start is missed” option.

http://answers.microsoft.com/en-us/protect/forum/mse-protect_start/scheduled-scan-wont-start-after-missing-scheduled/059a0c60-6f78-4523-a7a1-dc8ead43aa67

 

Now your scheduled scan will run the next time the PC is powered up if the PC is powered off at the scheduled time. This is how we achieve the perfect reliability that the Windows Task Scheduler was designed to deliver; and this is exactly what they did in Windows 8 to make the Windows Defender Scheduled Scan task perfectly reliable. So there you have it – a free Windows 8 upgrade for your MSE scheduled scan.

GreginMich
