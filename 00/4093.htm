See the snip below. As you can see I expanded the number of decimal places in H73 and it shows that H73 does not equal 2,72 and this is a well documented feature of the way that excel handles floating point numbers. Excel works in binary and often there is no precise binary equivalent of a floating point number. See the link below for a fuller explanation. =ROUND(H1+0.01,2)

jeeped say:
The Nested levels of functions for Excel 2003 is 7 (source: Excel specifications and limits) but there are ways around this.

You can pass over the function to another cell in the 7th IF()'s value_if_false parameter. The second cell can contain another 7 IF()s and can pass over to a third cell if necessary.

Most formula with 7 IF() statements can be rewritten using alternate syntax and/or alternate functions to tighten up the formula and calculate results more efficiently. Consider posting the formula to see if anyone has an alternate solution. Not every formula can be rewritten but it has been my experience that at least 9 out of 10 can benefit from alternative logic.

Phil

saberman say:

When you run the query on the Access side it can determine if the set of keys returned will be unique -- it can't do that with the passthrough query.  Try using a select distinct from the passthrough query so Access will know there is only one update per key.  (Puting in in the passthrough query won't help as Access does not interprete the SQL in a passthrough query.)

john p spencer say:

If I recall correctly PASS THROUGH queries are NEVER updateable.  If you are using the pass through query ONLY in the where clause you might be able to accomplish your goal.

roger carlson access mvp say: It doesn't matter.  If ANY part of the update query uses a non-updateable recordset (even if its not the table being updated), the whole thing is non-updateable.  As John said, pass-through queries are never updateable.

saberman say:
I will try a local query distinct based on my passthrough query to see if that helps.

If that doesn't work replace the join with a WHERE clause:

WHERE [<key field in target table>] IN (SELECT * FROM <back end query that returns keys>)

or just:

WHERE [<key field in target table>] IN (<back end query that returns keys>)
