greginmich say:

The Antimalware Service (MsMpEng.exe) is the core of Windows Defender, and that service is always loaded at startup by a line in a registry Run key – but the Windows Defender user interface (MSASCui.exe) isn’t started automatically.

You can either use the reg file to add a startup line to the registry, or you can just drop a shortcut for the user interface into your Startup folder. The shortcut method gives you the option to run an automatic scan or update when the interface loads. For example, I use the following target in the shortcut to start the interface hidden and run an update. Since I restart this PC several times a day, this always keeps the signatures right up to date:

"C:\Program Files\Windows Defender\MSASCui.exe" -Update -hide

Open PW protected access 2007 from commandline - Microsoft Community

christw1 say:

Open PW protected access 2007 from commandline

I'm doing some research and have yet to get a direct answer; SO...is it possible to open and run a password protected Access 2007 database from the command line? If so, can someone give me the proper syntax and switches.  This will be used in task scheduler to perform some unattended actions.

Please note: I know about the /pwd switch but from my research it does not apply to Access 2007 (Accdb's)

boblarson say:

I'm not sure about the switches but you could create a special Access database to do the work and just have the scheduler open it.

 

1. So, create a new Access file.

 

2., Create an AutoExce macro to kick off the procedure we'll have below.

 

3. Create the procedure to kick off the database

 

Function StartDb(strPath As String)

   Dim objAcc As Object

 

   Set objAcc = CreateObject("Access.Application")

 

   With objAcc

      .OpenCurrentDatabase strPath, , "PasswordHere"

      .UserControl = True

   End With

Set objAcc = Nothing
End Function

 

So in the AutoExcec macro you can call it by using RunCode and select StartDb("PutYourPathAndFileNameHere")

 

Now, I had hard coded the password as you can see.  If you have several you want to deal with you can put a table in there and hide it but it is not all that secure. 

 

Also, the UserControl = True part disconnects this database from that one and allows it to do its thing.  But if you need to run some stuff in the database and then close it, you can execute things directly in this code I gave you if you want.  It would just take some modification. 

 

For example, if you had a query to run you can use the objAcc.DoCmd.OpenQuery "queryNameHere"  but if you are just wanting to run queries then I would not use this code but use DAO instead.
