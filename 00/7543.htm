jay freedman say:
You're correct, a macro would be necessary.

One approach would be an AutoOpen macro that calls this little routine:

Sub ReplaceHomepath()
    Dim HL As Hyperlink
    For Each HL In ActiveDocument.Hyperlinks
        If InStr(HL.Address, "\%homepath%") Then
            HL.Address = Replace(HL.Address, "\%homepath%", Environ("homepath"))
        End If
    Next
End Sub

If any hyperlinks in the document don't include "\%homepath%" then nothing will be changed in those hyperlinks. The unfortunate side effect of the macro is that if there are any replacements, the document will be "dirtied" and the user will be prompted to save it when closing. Also, the replacement is case-sensitive, so you must use homepath and not HOMEPATH or HomePath in the hyperlinks.

rick rothstein, mvp - excel say:
One problem I see with your data is you have mixed methods of specifying your dates. Clearly this number 4182010 must mean April 18, 2010.... the month must be the first digit as there is no 41st or 18th month and the next two digits the day. However, later in the list you have this number 15210 where the 10 must be the year, and the first two numbers the day as there is no 15th month, with the month being the single digit following the day value. So, in the first case the month came first followed by the day whereas in the second case the day came first followed by the month. While this could be easily figured out for the posted values, any code that did so would not know what to do with numbers such as 1232010 (which could be January 23, 2010 or March 12, 2010) or 12310 (which could be January 23, 2010 or March 3, 2010)... as long as there is no fixed order to how the month and day are specified, there will be many, many cases that the code would not be able to decide on. So, you will have to create some kind of rule to handle these indeterminate cases and, once you do, you and your users will have to remember that you have locked down whatever rule you come up with. And I guarantee that once you lock down a rule for these indeterminate cases, you or your users will eventually forget that you did so and end up entering wrong dates down the line. It is probably best not to rely on abbreviated date entries in order to avoid problems in interpretation.

ashish mathur say:
Hi,
 
This is not the complete solution to your problem but will reduce the number of inconsistencies.  Try the following steps
 
1. Sort the entries in ascending order - this will bunch up all the dates and non dates separately;
2. Under custom filter select "Is after" and 1 (This process will show up all dates and numbers)
3. Select the data and go to Data > Text to columns > Delimited > Ensure all delimiters are unchecked > Select Date > DMY > Finish.  This will convert 15210 to 15-02-2010 and 11022010 to 11-02-2010.  Please note that 4182010 will still remain the way it is
4. Under custom filter select "Equals" and * (This process will show up all non dates)
5. Select the data and go to Data > Text to columns > Delimited > Ensure all delimiters are unchecked > Select Date > MDY > Finish.  This will convert 04/18/2010 to 18-04-2010 and 4/18/2010 to 18-04-2010
 
After performing the above mentioned steps on your sample dates, all dates were corrected for except 4182010.
