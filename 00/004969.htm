Sorry if I confused you - these two lines

Set wb = xl.workbooks.Open("Path and filename")

Set ws = wb.worksheets("Sheet Name")

Should actually reference the names of those things, not the phrases, e.g.

Set wb= xl.workbooks.open("C:\foldername\foldername\somefile.xlsx")

where foldername and somefile are the actual names of those things.

Same for the sheet name:

Set ws = wb.worksheets("Sheet1")

This code works for me to open a file from within Access, copy a range of cells from one sheet and paste them into another sheet.  Note that if you actually want to see the workbook, somewhere after creating the xl object, you need a

xl.Visible = True

statement, otherwise it remains invisible.

Sub OpenExcelWB()
  Dim xl As Excel.Application
  Dim wb As Excel.Workbook
  Dim ws As Excel.Worksheet
 
  Set xl = CreateObject("Excel.Application")
  Set wb = xl.Workbooks.Open("C:\users\jlatham\desktop\aug13-nov14.xls")
  Set ws = wb.Worksheets("FNB-399")
  With ws
    .Range("C3:C50").Copy
    'copy to a different sheet into column A at A1
    wb.Worksheets("Sheet1").Range("A1").PasteSpecial xlPasteAll
  End With
  'shut everything down
  Set ws = Nothing
  'close the workbook and save changes
  wb.Close True
  Set wb = Nothing
  'quit excel
  xl.Quit
  Set xl = Nothing
End Sub

1. Start Event Viewer (run eventvwr.exe)

2. In Vista or Win7, navigate to Windows Logs->System.  In XP navigate to System.

3. Look for events with the source "Microsoft Antimalware"

 

You should look for the following events:

Event ID 1000: Scan Start

Event ID 1001: Scan Complete

Scans will definitely not occur while the machine is in Sleep or Hibernate modes.  I believe Standby is the XP equivilant of Sleep.  If that is the case, I would not expect the scan to occur.  The Scheduled Scan configuration described above asks the Task Scheduler to only start the scan when there has been a period of no keyboard/mouse input and low CPU usage.

  • Field stores Customers ID
The problem I'm having is that it won't store the ID while also allowing Name to do the lookup for the field - it will only store the Name. If I sort by ID, then it will store the ID, but then it also uses the ID field for the lookup, and that won't work for our situation.

Would appreciate any ideas for getting around this problem.

Matthew

john w. vinson say:

You might want to read this critique of Microsoft's Lookup field type:

http://www.mvps.org/access/lookufields.htm

I would (very!) strongly suggest that you not use Lookup Fields at all, and that you also avoid using Table Datasheets for data entry or editing. They're very limited, even with misfeatures like Lookup Fields.

Instead, create a Form based on the order table; on the form you can put a Combo Box based on a Query selecting the ID and the concatenation of the name and address. The SQL would be something like

SELECT ID, [Name] & " " & [Address]
FROM Customers
ORDER BY [Name], [Address];

This will do as you describe.

However, note that Name is a reserved word; a field has a Name, a table has a Name, a textbox has a Name - Access can and will get confused. It's usually best to store personal names as (at least) two fields, FirstName and LastName; middle name and suffix are often useful as well.

ken sheridan say: Following on from John's reply you'll also need to set a few other properties of the combo box:

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.

In addition to John's point about not using Name as an object name (use CustomerName instead if these are corporate customers where first and last names aren't appropriate) I'd also suggest that the primary key of Customers be named CustomerID and the corresponding foreign key in Orders be similarly named CustomerID.  This will make the semantics of any code or SQL statements easier to follow than if the generic ID is used as the column name.  In forms or reports you can use a label 'Customer' of course regardless of the column name.  As John says, the user interface with the data should always be via forms and reports rather than raw datasheet view.

If separate FirstName and LastName columns are used the RowSource property of the combo box would be like this:

SELECT CustomerID, (FirstName + " " ) & LastName & ", " & Address FROM Customers
ORDER BY LastName, FirstName, Address;

By using the arithmetical + operator in a parenthesized expression to add a space following FirstName the space will be suppressed if FirstName is Null because Nulls propagate in arithmetical expressions.  This means that you could use the LastName column to store the name of a corporate customer with no first name, as well as storing individuals by their personal names.

Note BTW that none of the object names in the above include spaces.  While permitted, spaces or other special characters are best avoided.  Use CamelCase as above, or an underscore to represent a space  instead, e.g. Customer_Name.
