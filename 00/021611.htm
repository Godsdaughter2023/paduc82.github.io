marshall barton say:

That is a really BAD idea.  You should never use code that modifies the design of a heavy duty object (table, form or report) while your application is being used for normal activities.  Design changes, manual or via VBA code, should only be made by you, the developer, before testing and distribution to users, including yourself, for production use.

OTOH, you can easily create a datasheet form based on the table and use code in its Open event to hide the form's datasheet columns.  In this scenario, I think your code should look something like this untested air code (i.e. you'll probably need to fix it up some):

Dim dbs As DAO.Database
Dim rst As DAO.Recordset
Dim fld As DAO.Field

On Error Goto ErrorHandler

Set dbs = CurrentDb
Set rst = dbs.OpenRecordset("table1")

   For Each fld In rst.Fields

       Me(fld.Name).Properties("ColumnHidden") = Not fld

   Next fld1

ExitHere:

   rst.Close : Set rst = Nothing
   Set dbs = Nothing

ErrorHandler:

   Select Case Err.Number
   Case Else

      MsgBox Err.Number & " - " & Err.Description  

      Resume ExitHere
End Sub

Note that code assumes the name of each bound control on the form is the same as the name of its bound field.

marshall barton say:

Don't be so impatient, I am busy with a lot of other things and occasionally need to eat and sleep.

1.  You should leave the error handling in there so you can see which errors occur.  If you need to ignore an error, you can do that in the error handler.  Using On Error ResumeNext is a very poor practice because it causes all errors to be ignored and you will never know when or why your program breaks.

2.  Because the fields in table1 are all Yes/No fields, their value must be either True or False.  That means
   Not fld

will be either False or True.  Therefore, the column hidden property will be set to the opposite of the value in the Yes/No field.  In other word, if the field is checked, the column will not be hidden and if the field is not checked, the column will be hidden.

Why would you want to change the data type of the Yes/No fields in table1?  Are you changing the problem you originally posted?   If you do change the data type of the fields in table1, the code would have to be redesigned to deal with the different data type and the possible values that might be in those fields.

zhen jiang say:

Hello Techie Bear,

1. You should set your tolerance much higher :) It is very common that System & Compressed Memory (or any modern application) consumes a memory of 300MB. For example, Google Chrome browser can easily take up 700~800MB of memory on normal use. If you receive a low memory warning on your 6GB machine, your target would be something that takes up ~1GB of memory at least.

So please continue to monitor your computer every now and then. Besides, please switch to the "Performance" tab in Task Manager and click "Memory", and take note on the numbers shown in "In Use", "Available", and "Committed". Those numbers will be helpful in further diagnosis.

2. Based on your description, I am pretty sure that the UA string of your browser (Pale Moon) is somehow incorrect. A website cannot directly tell what operating system you are using; instead, it reads the information from your UA string. To verify your UA string, please go to any UA detection website (for example, http://www.useragentstring.com/) and see what your UA is. Mine is:

Mozilla/5.0 (Windows NT 10.0; WOW64; rv:48.0) Gecko/20100101 Firefox/48.0

Note the bold part - this is the part where your browser tells a website what OS your are using. Mine shows Windows NT 10.0, which means Windows 10. If your string shows something else (for example, Windows NT 6.x), that is the problem.

Please let me know whether you need any help in fixing the UA string of your Pale Moon.
