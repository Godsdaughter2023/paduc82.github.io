longhorndkr say:

I have actually determined the root cause and solution for this problem.  It appears that Microsoft issued a cumulative (critical) security update on September 21, 2012.  This update effected IE 9 versions purportedly 9.0.5 and above.  A colleague with IE 9.0.4 who had not allowed Windows Update to update his IE version did not experience the error.  I experienced this issue in IE 9.0.8 - IE 9.0.10 (the latest version).

 

In any case, it appears that previously we were able to Allow a website access via the Trusted Sites or Local Intranet security zones under the Security tab in IE Tools->Internet Options, and to Disable the XSS Filter found under the Custom level button in both of these zones.  Since the cumulative security update, despite disabling the XSS Filter in Trusted Sites and Local intranet, you will HAVE to disable the XSS Filter under the "Internet" security zone or it will continue to prompt you with cross-site scripting errors. 

 

Since IE9 basically works in Windows 7 (i.e. not in XP), you should preferably right-click on IE and run it as an administrator or the "Internet" security zone may appear grayed out and you will not be able to change the Internet level XSS filter setting.  This was an additional complication.

 

Hope that helps.  Thanks,

Going to be honest when it comes to Imap with O'2013 versus O'2010 - have seen different things depending on whether O'2013 was a brand-new (only) install versus an upgrade and will also readily acknowledge that have read reports in the past about Imap related issues - mostly to do with timing so I may (and most likely) not be the best person to ask for any kind of definitive answer at this point in time (given the nature of our products - each account type is tested in its own right across different variations and IMap hasn't made it to the top of the list as yet for all the variations).

The other difference is that I use a "long timeout" versus "short" which may or may not have any kind of impact. (That aside, not all IMap servers are created equal so that's also something to keep in mind)

In a nutshell, simply because I haven't encountered the same problem(s) you have does NOT mean that an issue doesn't exist - only means that I haven't run across it. I should (but don't due to time constraints) keep more meticulous track of all the variations used/tied with all different combinations of Windows OS / Office / OS & Office bit levels / email account types and all else else in between.
