bob jones aka: cybertaz say:

The Developer tab & the tools it provides cannot be used in Publishing Layout or Notebook Layout View. It is normal for that tab not being displayed when in either of those Views, as is the case with several other tabs. Many other features cannot be used when in those views, either.

It sounds like you created the Table in Publishing Layout view, then switched to Print Layout View. If that's the case, the document already is defined as a Pub document which does not support Forms Controls. Just switching from Pub View to Print Layout View does not change the document type.

Everything in Pub View is inserted as an object, including tables. As a result, Forms Controls cannot be inserted into Tables created in Pub View. That's why you're having to click into the main body of the document. Although Forms Controls can be inserted there while in Print Layout View, they will not be usable in Pub View. In fact, if you enable Forms Protection while in Print Layout View the document can't be displayed in Pub Layout while Protection is in effect.

Bottom line: Create your form by starting with a standard Word document in Print Layout View & do not try to convert it to a Pub document.

The video you provided the link to will no display unless yhe viewer is enrolled in lynda.com training, but it's noteworthy that the descriptive info on that page refers to Word 2016, not Word 2011. Since I can't view the video I can't be sure, but my guess is that it also may be specific to the Windows version.

cortig say:
The issue with RichText/HTML e-mails is essentially always the same:
The fonts!
The problem is that fonts often look different on different computers (let alone different platforms) and when fonts are not available on a computer, Font substitution takes place and adds another layer to this mess.
The only somewhat safe way to deal with the issue is to use so-called web-safe fonts (fonts that you know are going to be available on all computers).

MS probably received tons of complaints about this, and so do all e-mail client providers, but the problem is extremely complex and most of the time, not in the hands of the e-mail client companies.

The general rule is:
  • send plain text every time you can
  • use Web-safe fonts
  • use only one font (or as few as possible): at least, the message will be homogeneous: everything a bit too big or a bit too small, but consistent.
