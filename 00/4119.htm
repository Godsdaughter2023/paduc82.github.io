joeu2004 say:

wiens wrote:

Why is this not calculated.

[....]

a = 12543 * 72

From the VBA "* operator" help page:  ``The data type of result is usually the same as that of the most precise expression``.

Since 12543 and 72 are both type Integer, their product is type Integer (16 bits), which has a range -32768 to 32767.

Note that 12543*72 is 903096, which exceeds the range of type Integer.  But it is within the range of type Long (32 bits), which is -2,147,483,648 to 2,147,483,647.

Also note that there is an exception when the operands are type Variant, to wit:  ``[if] the data type of result is an Integer variant that overflows its legal range, [then result is] converted to a Long variant``.

So there are (at least) two alternatives:

Alternative #1:

a = 12543 * 72&

That is a proofreader's nightmare, IMHO.  Note that when "&" is preceded by a space, it is the string concatenation operator, not the implicit type Long character.

Instead, I would opt for:

a = 12543 * CLng(72)

I don't know if VBA is clever enough to coerce 72 to type Long at "compile time" instead of actually calling the conversion function at run-time.

In either case, the arithmetic can still cause an overflow error if the product exceeds the range of type Long.

Alternative #2:

x = 12543

y = 72

a = x * y

Here, x and y are presumed to be type Variant.  IMHO, it is better to have explicit declarations for all variables, to wit:

Dim a As Variant, x As Variant, y As Variant

Aside....  To enforce that programming practice, I set the VBA option "Require Variable Declaration", which automatically creates the Option Explicit statement at the top of every new module.

One advantage of alternative #2:  if the product exceeds the range of type Long, the arithmetic is automatically coerced to type Double, instead of causing an overflow error.

joeu2004 say:

wiens wrote:

If I declare 'a' as Double before the calculation it still gives an overloop error.

I'm not here to have a philosophical discussion, but to help you understand how VBA works.  "Grant me the serenity to accept the things I cannot change, courage to change the things I can, and wisdom to know the difference". ;-)

In VBA, like most procedural programming languages, the data type of the left-hand side of an assignment statement does not affect the data type that is used to evaluate the right-hand side.

And the data type of sub-expressions on the right-hand side is determined left-to-right, taking precedence into account.

Consequently, we get an integer overflow error even in the following statement:

a = 0.123 + 12543 * 72

Even though 0.123 is obviously type Double and the result of the sum will be type Double, the product is still evaluated as type Integer because its operands are both type Integer.

It is prudent to "program defensively" by writing 12543 * 72.0 or 12543.0 * 72 in order to ensure that the product is evaluated using type Double, if that is your intent.
