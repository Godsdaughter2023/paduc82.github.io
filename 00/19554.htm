That answer may not be very helpful, but it's possibly the best answer for you and was given in good faith. It is very well known that lots of HP printers seize to work after IE upgrades and continue to work only once HP provides a driver or utility software update. It's also known that they often don't do this (provide newer software). Many say that HP provides crappy drivers and doesn't support their printers/drivers for very long. I can't comment on this as I'm not using HP printers.

The reason that printing from IE after a major update stops can be that the underlying system of privileges and integrity levels has been revised, another that a newer version of IE uses some Windows printing method that has been available for a while but has not been much used for compatibility reasons. At one point it may nevertheless switch to this method and other applications may follow over the next years. If the supplied hardware driver is then not suitable for this method it will fail.

What you can do to find out more about the problem:

- check if the problem exists only when printing over the network (e.g. connect that printer directly)

- check if the problem exists only with that printer (e.g. try to use another printer via network)

- try different methods of network printing (if available)

- start IE "as Administrator" and try to print (already suggested in that KB article?)

- uninstall security software temporarily

- if there's more than a driver installed, e.g. some utility software like "HP Web Print" try to uninstall it and print without it

- check in the printer/printer driver settings (not the "printing preferences") for methods that change printing methods, e.g. stop queuing of the print or do not use "advanced" features, change print processor and default data type. Before you do this, make sure that you write down the original options! Many of the combinations may not create output or no useful output.

Do not "try to search the HP support site". Ask a question there! There are myriad methods of network printing and I fear your printer uses a legacy method that depends heavily on the driver or utility software installed. They

VBA Code- Insert and resize image with square text wrap - Microsoft Community

spikybytes say:

VBA Code- Insert and resize image with square text wrap

i am new to VBA and don't really understand too much as i have only learnt to record macros. I am looking to insert an image into a template and have it 4cms tall with the width relative to the original sizing. I also need to have the text wrap as square wrap. Can anyone help at all please.
jay freedman say:

Here's a rather minimal macro to do what you asked for:

Sub InsertPictureSpecial()

    Dim strFilename As String

    Dim shpShape As Shape

    

    With Dialogs(wdDialogInsertPicture)

        .Display

        strFilename = .Name  ' full path of selected file

    End With

    

    If Len(strFilename) > 0 Then

        Set shpShape = ActiveDocument.Shapes.AddPicture( _

            FileName:=strFilename, _

            Anchor:=Selection.Range)

            

        With shpShape

            .LockAspectRatio = msoTrue

            .Height = CentimetersToPoints(4#)

            .WrapFormat.Type = wdWrapSquare

        End With

    End If

End Sub

The first part uses the built-in Insert > Picture dialog to let the user select a file. The .Display command shows the dialog, but when the user clicks the Insert button the dialog just closes without inserting the picture in the document. Instead, the next line captures the filename (including the full path) in a String variable.

The next part uses the filename to insert the picture as a floating Shape object (even if the user's Options setting says to insert pictures as In Line with Text). The object is "anchored" to the current location of the cursor (the Selection in VBA-speak). Then the macro locks the aspect ratio (the fraction height/width), sets the height to 4 cm, and sets the wrapping.

This macro could be expanded in several ways, such as setting the picture's position (its .Left and .Top values) relative to the anchor or to the page margins; using a different method of choosing the file to insert; and adding some error-trapping in case something goes wrong.

Sure you can.  But it would be easy enough to modify the code to make an exception for your "prep" form, like this:

'------ start of code ------

Sub RemoveColonsFromLabels()

    Dim ao              As AccessObject
    Dim frm             As Access.Form
    Dim ctl             As Access.Control
    Dim blnModified     As Boolean
   
For Each ao In CurrentProject.AllForms
   

        If ao.Name <> "frmPrep" Then   ' subsitute your own form name for "frmPrep"

            DoCmd.OpenForm ao.Name, acDesign, WindowMode:=acHidden
            blnModified = False
       
            Set frm = Forms(ao.Name)
       
            For Each ctl In frm.Controls
       
                With ctl
               
                    If .ControlType = acLabel Then
                        If .Caption Like "*:" Then
                            .Caption = Left$(.Caption, Len(.Caption) - 1)
                            blnModified = True
                        End If
                    End If
       
                End With
       
            Next ctl
       
            If blnModified Then
                DoCmd.Save acForm, frm.Name
            End If
           
            DoCmd.Close acForm, frm.Name, acSaveNo
            Set frm = Nothing

        End If       

    Next ao

    MsgBox "Done!"
   
End Sub

'------ end of code ------

Rather than having separate fields for the startdate and starttime, or for the enddate and endtime, you should consider having (e.g.) #1/26/2007 07:00# in a startdatetime field. It makes searching and calculating much easier!
