You could perhaps fake a "new record" section with some controls strategically placed above a subform in datasheet view. dirk goldgar say:

The split form feature of Microsoft Access doesn't support having the "new record" row at the top, but you can fake it by using two subforms instead:  one to show only existing records, with its AllowAdditions property set to No, and one to show only the "new record" row, with its DataEntry property set to Yes. 

You would also have to have a little code (or macro) here and there to get the behavior you want:  in the Current event of the "existing records" subform, put code to navigate to the record on the main form, and in the AfterUpdate event of the "new record" subform, requery the "existing records" subform.

It's up to you whether you think this layout, with its additional overhead, is better than the standard layout that is familiar to most users of Access.

A number of issues:

1. your app should be split into front end and back end. Only the
does not work well in the long run with anything other than data
tables (and queries).
2. why do you think you need partial replicas? I've done lots of
replication over the years and never once used them. I like having
all the replicas being identical and having all the data -- it's
automatic backup! With partial replicas, only some of the data is in
all the replicas, and it means you have less in the way of backup.
3. with a proper split architecture, this will synchronize the back
end with another replica:
  Dim strBackEnd As String   Dim db As DAO.Database

  strBackEnd = _
    Mid(CurrentDB.TableDefs("tblReplicatedTable").Connect, 11)    Set db = DBEngine.OpenDatabase(strBackEnd)
  db.Synchronize "\\Server\FolderName\OtherReplica.mdb"
  db.Close
  Set db = Nothing

Now, you'll want error handling and you might want to check for
conflicts:
  If db.TableDefs("MSysConflicts").RecordCount <> 0 Then
    ...

All this assumes you're on a wired LAN, because on WiFi or a WAN,
direct replication is not suitable (too dangerous because the
bandwidth and reliability are such that it can easily drop a
connection and corrupt the remote replica). If you need to synch
across WiFi or a WAN, then you can't use direct synchs and the whole
thing becomes an order of magnitude more complex. It's absolutely
doable and works quite well, but it's much, much more complicated.

pibkac say:

We were having this problem with several users from remote locations accessing Excel files on a file server.  The clients are Windows 7 Pro systems connecting to a 2003 R2 server. I suspect this would also occur with 2008/2008r2 servers, but I have not tested it.  It does not occur with XP clients. The file was getting a lock on it even though it was not being opened.  The file lock would last 10-20 seconds before being released. There are three things that contribute to this problem, all of them Windows Explorer related:

  1. Selecting the file in Explorer -just highlight it without opening
  2. Hovering over the filename so that the pop-up file information box was displaying
  3. Using the Preview Pane in Explorer

To fix the problems:

  1. Turn off the Details Pane shown at the bottom of Windows Explorer.  Go to Organize -> Layout -> Details Pane.  That fixes the first problem.
  2. Turn off “Show pop-up description for folder and desktop items”  Click on Tools -> Folder Options.  In the box that opens up, click on the View tab.  Scroll down the list to “Show pop-up description for folder and desktop items” and clear the checkbox and then click OK.  This fixes the second problem.
  3. Turn off Preview Pane.  Go to Organize -> Layout -> Preview Pane. This fixes number 3. 

The preview pane is not on by default in the "Details" view, but the other two options are on by default.  The file locks are created when Explorer is gathering file information like size, author, modified date, etc..  I suspect that it would also do this with any other file type, but I have not tested that situation either.  For those of you in a managed environment (I guess that would be anyone reading this since the likelihood of home users having this problem would be approaching zero), I am going to use regmon to see what registry keys are modified by the changes.  I will post them so that you can deploy them to your environment if I find them.

 

 

pibkac say:

Here's the keys that you need to set - save the following as a .reg file

Windows Registry Editor Version 5.00

[HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Explorer\Advanced]
"ShowInfoTip"=dword:00000000


[HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Explorer\Modules\GlobalSettings\Sizer]
"PreviewPaneSizer"=hex:35,00,00,00,00,00,00,00,00,00,00,00,bd,02,00,00
"ReadingPaneSizer"=hex:04,01,00,00,00,00,00,00,00,00,00,00,64,02,00,00

It's to do with the frequencies that the phone supports. If you cannot get a 3G signal then the RM-428 doesn't support the 3G bands used in Bangladesh. You can save battery by setting GSM only in Menu > Settings > Connectivity > Network > Network Mode so the phone isn't constantly searching for a 3G signal.

As for release versions, it might be that 52.0.007 for RM-428 has the same features as 60.0.003 for RM-356. The release versions available for the two models aren't the same, it's not as if you're missing the latest release that RM-356 has, it's that the release version numbers are completely different.
