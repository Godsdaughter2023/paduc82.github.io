suzannethomas1 say:

Access Database for Oil and Gas Royalty Income Checks

I am trying to set up a database for client who has oil and gas royalty income checks.  I want to be able to enter details of each check regarding property name, volumes, gross amount, prod tax deductions, etc.

I know I need a Oil Producer table.

Then I need to set up a checks received table showing oil producer (look up field), check number, check date, net amount of check

Then I want to set up a table with check detail associated with that one check.

Can anyone point to me toward someone who has done this.  I've spent hours and I just can't seem to get the lookup tables and the relationships set up correctly!!

Suzanne

john w. vinson say:

Well... I'd avoid "Lookup Fields" like the plague they are: see This Critique. And you should not be using table datasheets to interact with data in any case - Forms are much friendlier, more powerful, and more in your control. 

That said, you need to identify all the relevant "Entities" - Producers, Properties, Wells (I presume), Checks. Each kind of Entity will have its own table, and the relationships will be based on your business rules: for example, "Every Property will have zero, one, or many Wells, every Well will be on one and only one Property"; "Every Producer will have zero, one, or many Properties and every Property may have zero, one, or many Producers". Once you have these logical relationships you can start building the tables and the table relationships. Every Table should (nay, MUST) have its own Primary Key, which uniquely identifies the specific entity in that entity type; for one to many relationships, the "child" table will contain a field - called a "Foreign Key" - as a link to the primary key of the parent table.

Values which can be calculated should be calculated, not stored; I would assume that each well (or property) will have a volume produced over some period of time; but gross production should be calculated by summing individual production amounts, and not stored in any table.

This is probably not a very difficult application, but it's not trivial either; you'll have to do some reading about "Normalization" and the principles of database design. The Utter Access New Users Guide is a good place to start. Lay in a good supply of Midnight Oil; relational databases have a learning curve! But we'll be glad to help!
