ken sheridan say:
...............anyway to get the OTHER RECORDs that Are NOT the duplicates( the ones we want to see that are different???)

It's usually easy enough to determine if rows exist in one table without a match on one or more of a set of columns in another table provided the basis for the correlation can be logically defined.  Let's assume a table x with a key column a, and non-key columns b and c, and another table y with an identical definition.  Let's then assume we need to return those rows in x for which there is either no match on the key a, or there is a match on the key a, but with a non-matching value in b and/or c.  A query for this almost writes itself:

SELECT *
FROM x
WHERE NOT EXISTS
    (SELECT *
     FROM y
     WHERE y.a = x.a)
OR EXISTS
    (SELECT *
     FROM y
     WHERE y.a = x.a
     AND (y.b <> x.b OR y.c <> x.c));

There are other ways in which a query to do this could be written, but the logic of the above SQL statement is readily apparent, and provided that the indexing of the columns in both tables is appropriate performance should be good.

Scott's premise still applies, though.  Data from the original sources should be appended to a single table in Access, adding an additional column to the key to differentiate the rows for each month.  Exactly the same logic as above can then be applied by basing both the outer query and sub-queries on the single table and restricting each to the month in question.
