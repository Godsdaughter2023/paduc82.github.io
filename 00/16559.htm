audreymac say:

Cascading Combo 2nd Box does not hold the value

I have or had a perfect cascading combo box.  The AllService table has two fields Service and SubService.  I removed the ID so that I could use SQL DISTINCT the eliminate the duplicates in Service.  Now it's doing something entirely different.  When I select the Service, I get the SubService for that item but it doesn't stick.  I save the record | Close it and when I open it again SubService is blank. 

Another strange thing, when I select Service then SubService | Save the record | Click next record the value stays | it's like the record changes but the data doesn't.  Only when I close the form does it disappear completely.  I read a Q&A similar and the instruction was that "the bound column of the second combo box is not made up of distinct values".  Any ideas on how I can keep the Service field from repetitive data and bound the second column without adding a ID in AllService table?  Or is this the problem? 

Row Source SELECT DISTINCTROW AllService.Service FROM AllService ORDER BY AllService.Service;

ken sheridan say: Firstly, the SQL statement should use the DISTINCT predicate, not the DISTINCTROW predicate.  The latter has been pretty much redundant for a long time.  It is a hangover from the early days when Access did not support the use of subqueries.

To allow the enforcement of referential integrity the AllService table should really be a SubServices table in which Service is a foreign key referencing the Service primary key column of a Services table.  The RowSource of the first combo box would then be:

SELECT Service FROM Services ORDER BY Service;

And the RowSource of the second combo box would be like this:

SELECT SubService FROM SubServices WHERE Service = Form!cboService ORDER BY SubService;

Note the use of the Form property in the above to return a reference to the current form.  You do not have to reference the form by name as a member of the Forms collection as both controls are in the same form.

The behaviour you describe suggests that the second combo box is unbound rather than being bound to the SubService column in the form's recordset.  Check that its ControlSource property has the appropriate value.

However there is a more fundamental problem here in that, by having both Service and SubService columns in the form's underlying table, the table is not normalized to Third Normal Form (3NF), which requires that all non-key columns be functionally determined solely by the whole of the table's primary key.  As it stands Service is transitively dependent on the key via SubService, so is redundant, leaving the table at risk of update anomalies, and should be deleted from the table.  You can still use correlated combo boxes for data entry, but the service combo box should be unbound.  If the form is in single form view then it is a simple task to set the value of the unbound combo box in the form's Current event procedure with:

Me.cboService = DLookup("Service", "SubServices", "Service = """ & cboSubService & """")

If the form is in continuous forms view it is a little trickier and a 'hybrid' control made up of a bound read-only text box superimposed on an unbound combo box must be used for selecting the service value.  You'll find an example of this in ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the use of correlated combo boxes in various contexts is illustrated, but note that the primary keys of the tables are the more usual surrogate autonumber keys, rather than natural keys as described above.  There is nothing in the database relational model against the use of natural keys, however, and they do have a distinct advantage when using correlated combo boxes.

Hi Gordon,

 

I believe the “Microsoft Text” driver was a windows driver that was included with Jet.  Since these Jet components are all 32 bit components if your application is running as a 64 bit application on a 64 bit OS, it most likely won’t have visibility into these drivers by default.  To ensure it has visibility into the 32 bit drivers on any 64 or 32 bit machine, I would recommend compiling it specifically as a 32 bit application (x86). 

 

With that being said, the 64 bit Ace driver does include a 64 bit text driver.  The problem is this ACE driver is not a replacement for these Jet drivers and is only intended to be used in specific scenarios.  The article below lists these scenarios:

 

Microsoft Access Database Engine 2010 Redistributable - http://www.microsoft.com/download/en/details.aspx?displaylang=en&id=13255
