Rethinking Windows - Microsoft Community

capch say:

Rethinking Windows

Sorry to dump it here, but there really is no any other place to give a general feedback on Windows, anywhere; which kind of shows the real root issues with Windows...

Windows' foundations' time has long passed and nowadays' users want much more than the traditional PC setup with the enormous amount of history baggage. Windows today make almost all the things that users need a pain. The UX is horrible in these areas. It would be time to start doing the right things, thinking outside of the box and really listen to users. Here is my 2 cents.

1. For a long time now Windows has been famous for making installation, uninstallation and backup of programs extremely frustrating. It's time to take an action on that. If Mac can do it, surely Windows too. Let's forget finally the damn shared directories/libraries/registry tools and let each app just be installed with a simple copy into ONE INDEPENDENT location only. It would make app maintenance, development and backups much easier than it is today.

2. Users care a great deal about their data. Sure, it's nice to protected the OS itself from viruses and so on (admin accounts), but please provide a way where we can restrict which app can access to what in an easy way. Having everything dumped in Documents and let every app access that, is not a solution.

3. Microsoft always implement half-solutions only. WPF, OS widgets, and almost all other code bases make it impossible for average developers to develop apps QUICKLY without investing into 3rd party libraries, just because the base OS libraries are so poor. With WPF, there is still no D&D reordering of items in lists, and the list goes on and on. VS express is a good step into the right direction, but the OS libraries need to catch up big time.

4. Licensing. That mess has to be cleaned up and centralized for all products. Anyone who bought Windows, then an upgrade license, should be allowed to get a free license for the updated full OS version and download the ISO image for that. Nowadays HDDs/SSDs die every 1-2 years, so going through a system reinstall is a huge pain, mainly because of the bad licensing solutions and also the nonexistent backup solutions (see point 1).

Added:

5. Attaching meta-data to files. Though this feature works for some files, not all file format can be used, which is nonsense, since the meta-data shoudl be attached to the file on the FS level, not embedding into the file itself. The end result is that this feature sometimes work, sometimes not, which is especially bad for media files, the main use case for file metadata.

6. Office. Well, a long list of UX problems could be added here, so I'm just gonna pick some of them.

a) In Outlook there is no a single global font setting for emails. The way font settings work reflects the Office programmer's view of the topic, not the users. This makes simple tasks like increasing font sizes permanently in all email formats a big pain.

b) Outlook also makes it hard to see which email address is used for a sent email, a trivial task again for a mailer application of an Office suite.

c) Still no row/column move and swap in Excel.

The list of these minor UX annoyances can be continued almost endlessly. All of them is present today because Microsoft not listened to users good enough.

john w. vinson say: You can't, other than with VBA code in the data entry forms or with Before Insert Data Macros in the tables (and I'm not even sure if that will work). Referential integrity itself only applies within one database.

scottgem say:

There are many ways. As an example. Lets say you have a student database with a grades table. Now the student drops out so you want to delete his record and also his grades. So you have a delete student button. Behind the button you have code like:

Dim strSQL as String

strSQL = "DELETE * FROM tblGrades WHERE StudentID = " & Me.StudentID & ";"

CurrentDB.Execute strSQL, dbFailOnError

strSQL = "DELETE * FROM tblStudents WHERE StudentID = & Me.StudentID & ";"

CurrentDB.Execute strSQL, dbFailOnError

Me.Requery

john w. vinson say:

Scott, what you've shown implements Cascade Deletes, which is a good thing and is part of RI. But it's only a piece of the puzzle! You also need to check that deletions from a parent table don't leave orphan records in a child table, and error out the deletion if so; you need to ensure that adding, or changing the foreign key value, in a child record is only allowed with a valid parent record; and so on. It can all be done but the effort required really makes you appreciate a working RI system!

KarenAnne, why the two backends? What benefit are you getting from the extra complication?

tom van stiphout say:

Also, I'm sure Scott's code was just to highlight the main idea; such sets of queries always need to run in a Transaction.

scottgem say:
True, enforcing RI through code is not simple. I just tried to illustrate the concept.

ken sheridan say: There's something of a logical contradiction in your question.  Having a table in a local front end implies that one user can update it independently of other users, in which case an enforced relationship would not in any case be possible as different users' updates might conflict.

Local tables are perfectly valid, e.g. for storing each user's preferences, default values etc., but such tables would not need to be in an enforced relationship with any back end tables.  If, in such a table, there are columns which reference the table in the back end and are constrained by the relationship to that table, whereas other columns reflect the user's local preferences, then such a table should be decomposed, putting the columns which are constrained by the relationship into a separate table in the back end, and enforcing the relationship between that table and the referenced table, leaving only those columns which are not constrained in this way in the local table.

Perhaps if you can explain the exact real world context behind your question, we'd be able to advise further.
