I don't think there's a quick fix to move them all back the way you had sorted them, but to prevent this from happening in the future (because whatever bug causes this with every update MS is unlikely to fix) just do what I do - keep all your image folders anywhere but the Pictures folder, don't add them to any Libraries and don't use the useless in-built image viewers or management tools such as the Photos app (I've uninstalled it).

You can use free apps like IrfanView to view and manage images far better than Windows' own apps can ever hope to do.

Safe Boot Problem - Solved (Kind of) - Microsoft Community

sjseattle say:

Safe Boot Problem - Solved (Kind of)

My problem arose when trying to boot in safe mode, which I was doing for reasons that aren't really relevant.

I set Windows to boot in safe mode by running msconfig and checking the appropriate box at the "Boot" tab.

The problem that immediately arose is that Windows wouldn't reboot any more at all, in safe mode or any mode. Every time I attempted to boot, I got the diagnostic message, then a notation that Windows was trying to fix a problem and finally (often after a verrrry long time) a message saying it was giving up. Looking around on the Internet (which I had to do with my phone, since I didn't have a functional computer), made me worried that the problem wasn't anything to do with safe boot, but a hard drive that happened to go sideways at a very inopportune time. BUT: I was able to determine, by going to the command prompt - the only way to operate the computer at all - that my hard drive was still there, and apparently all intact ... it was just called "F:" (or, if I removed various other things, "D:" or "E:") instead of "C:"

This made me reasonably confident the problem was in the allegedly "safe" boot. Which raised the problem of how to UN-choose the safe boot option without being able to boot Windows. More searching on my phone turned up the answer ... or the approximate answer, as the suggestions I found had the command arguments in the wrong order. In any event, trial and error revealed that the following command-line entry did the trick (the trick being to make the computer boot the next time in the usual way, rather than safely):

bcdedit /deletevalue {default} safeboot

The stabs at a solution that I found online had the "{default}" before "/deletevalue" - or omitted altogether - which doesn't work.

So the problem is, I suppose, solved. Except, of course, I can't boot in safe mode.

frederik long say:

Remote Desktop works in exactly the same way over the Internet as over an internal network. However, when working over the Internet, you must take some extra steps in order to establish a connection:

  • The client must know the host's external IP address.
  • You must create a tunnel for port 3389 on the host's router.
  • The host must have a fixed internal IP address.
  • The host's router must direct port 3389 packets to the host's internal IP address.
  • The hosts's firewall must allow port 3389 packets.

BTW, "Host" is the machine that is being taken over, often a PC at the work place.

"Client" is your own machine, often the PC at your home.

frederik long say:

I can see these reasons why you might not know about port forwarding:

  • You made the connection by means of a VPN (Virtual Private Network)
  • Someone else created the tunnel for you
  • Instead of using a router you use public IP addresses for your PCs (very, very risky!)

Any of your internal PCs would be a potential recipient but only one must be allowed to respond! It's done by creating a tunnel, or in other words, you tell the router to forward all port 3389 packets to the RDP host's internal IP address. frederik long say:

What I didn't know was "packet" (as opposed to "signal").  

With a radio or an old-fashioned phone line you get a signal which consists of a continuous mixture of sine waves of different amplitude and frequency.

The Internet works differently. The signals you get are digital and are highly structured. They consist of packets composed of a header and data. The header contains a lot of information about the packet itself, e.g. data source, destination, port numbers, packet length etc. This is why several PCs in the one household can browse the Internet at the same time, without interfering with each other. With an old-fashioned this was clearly impossible.

When examining a packet's header block, your router "knows" what to do with the packet itself: Accept it, reject it, modify it, pass it on to one or all internal PCs etc. When a port 3389 packet arrives, the router will reject it unless you create a forwarding rule for this port.

If the Laptop/Computer came with the Windows Operating System preinstalled (Dell, HP, Acer, etc) , that Product Key that came with the Laptop/Computer is for the preinstalled OEM License

The OEM License/Product Key (preinstalled Operating Systems on Dell, HP, etc machines) is tied to the first machine it is installed on and dies when that machine dies.

The above rule also applies to OEM System Builders Licenses.

Retail Operating System License, that you buy and install yourself, are allowed to be transferred to another machine.

But Retail Licenses can also only be installed on one machine at any particular time.
