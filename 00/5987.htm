Note that if your problem is in finding and opening the Normal template for modification, here is an easy and foolproof way to do it:

In Word, press Alt+F11; the Visual Basic Editor opens. Press Ctrl+G to activate the Immediate window. In that window, type

     NormalTemplate.OpenAsDocument

and press Enter. Normal opens, but note that it may not be on top of all windows. You can close the Visual Basic Editor. Make the desired changes to the Normal file, and then save and close it. If you have two versions of Office on the same computer, there will be a conflict in the sense that they "compete" over file associations.

The corresponding re-registration process in Excel can be done by running Excel with the /regserver switch.

albert d. kallal say:

I'm pretty sure this behavior is by design. Think of the edit record command as if you're creating a new recordset in VBA. However this is occurring on a row of data that's already being edited and updated by the database engine own recordset. So, now you have two recordsets trying to edit the same row of data.

 

This is very much like having you edit some data in a form, then opening up a NEW recordset based on that form but the record still is still dirty or being edited by the other form or another recordset.

 

Which of the two reocrdsets now should be able to update at the same time? If you check the application log (remember this tip I told you), you will see a message about how the record is currently read only. So, you can view the data for the record, but you can't execute the editrecord command on that row (it will be locked and thus read only).

 

In fact, there is quite a good error message for this issue. This is what I see in the error when I attempt to edit data using the after update

 

 

So, I quite sure jet does not release the locks on that record until the triggers completely finished (otherwise would be a lot of difficult issues to control here in regards to concurrency and your example shows this).

 

The solution this case is simply to use the before change event. In fact, this is not really different then modifying data in a form. You likely would not use the after update event of a form on a record to have further modifying to that same record.

 

Also keep in mind that any record you modify, it will in fact cause triggers attached to that table will fire again. So, in your case you would wind up with a endless loop.

  

Note that in the before change event you don't even have to use edit record command (our kind of recordsets) because jet is already chewing away on that reocrd and there is a open reocrd in the context of the macro. So you can use setfield without the need for editrecord to create a record in the current context.

 

The after update is better for creating records in other tables or maintaining aggregate totals etc. However it's not very suitable for modifying the current record no more so than using the after update event in a form being used to modify data which once again in theory should make the after update event fire again.
