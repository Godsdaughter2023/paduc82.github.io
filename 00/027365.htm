malkeleah say: Since you forgot to tell us what versions of Excel you're using I can't be sure, but if the older Excel is very old indeed:

Information about certain file formats that are blocked after you install Office 2003 Service Pack 3:
http://support.microsoft.com/kb/922849

In order to open files created in Word/Excel 2.0 (really old) in Office 2007 or Office 2003 SP3, you'll need to change (or create) registry entries. The KB article above describes the issue for Office 2003 SP3. You can manually fix the situation by hacking the registry. For Office 2003 you want to use the values Office\11.0 and for 2007 use the values Office\12.0.

Navigate to or create:  HKEY_CURRENT_USER\Software\Microsoft\Office\11.0\Excel\Security\FileOpenBlock
and/or HKCU\Software\Microsoft\Policies\Microsoft\Office\12.0\Word\Security\FileOpenBlock

If this doesn't exist, create new keys until it does and then highlight FileOpenBlock and create a new DWORD called FilesBeforeVersion. Make sure the value is 0.

This will only allow you to open the old files, not save in the old format which is just as well. If it is really necessary to save in the old format, create another new key under Security as above and type: FileSaveBlock and make the Value Data 0.

The error "Microsoft Account does not exist" usually means:

criteria for paragraph mark after merge field? - Microsoft Community

exotericist say:

criteria for paragraph mark after merge field?

it seems the default behaviour for word is not to show a paragraph mark if the directly preceeding merge field is blank.

except sometimes it isn't, and blank lines are left in the document.

can someone point me to a page listing the criteria for this behaviour either way?

many thanks in advance,

e.

a. user say:

I doubt if there is a page anywhere that describes this in detail.

As far as I can remember, the rules are as follows for "traditional" field codes ((forget ADDRESSBLOCK for now) in recent versions of Windows Word at least.

Suppression acts on a paragraph - i.e. if you have a blank line terminated by a hard line end, followed by more lines, the blank line will only be suppressed if the whole paragraph is suppressed. Then...

 a. "blank line suppression" is switched on by default, and there is no option in the standard User Interface to switch it off

 b. blank lines with no MERGEFIELDs on them are not suppressed.
 c. the only lines that are suppressed are lines that would be completely blank when all the MERGEFIELDs etc. are resolved, where there is at least one MERGEFIELD field, and where all the MERGEFIELD fields have an empty result. 
 d. A MERGEFIELD whose result is inserted as the result of an IF field does not count as a MERGEFIELD.
 e. So if MERGEFIELD A has an empty result, B has an empty result and C has a single space as its result, the following lines will not be suppressed. <whitespace> means any combination of spaces, tabs (unless they have dot leaders etc.), hard spaces). I do not know exactly what counts.
{ MERGEFIELD C }
{ MERGEFIELD C }{ MERGEFIELD A }
<whitespace>
{ IF "{ MERGEFIELD A}" = "X" "" "{ MERGEFIELD A }" }

and the following lines will be suppressed:

{ MERGEFIELD A }
{ MERGEFIELD A }<whitespace>
{ MERGEFIELD A }<whitespace>{ IF "{ MERGEFIELD A}" = "X" "" "{ MERGEFIELD A }" }
{ MERGEFIELD A }<whitespace>{ MERGEFIELD B }
{ MERGEFIELD A }{ MERGEFIELD B }

Suppression does not occur when the field is nested inside another field (this was new in Word XP or 2003 I think). That includes IF fields and INCLUDETEXT fields.

ISTR that there can also be a difference in behaviour between preview (you usually see a couple of small dots when the line is going to be suppressed), output to a new document, and output to a printer, but I'm not going to check the details of that right now.

As for ADDRESSBLOCK fields, the overall field may behave that way, but how the field works internally is that text (including paragraph marks etc.) inside an ADDRESSBLOCK "<<>>" field is emitted if the field is non-blank. text outside "<<>>" areas is emitted anyway. The first line is usually treated specially. Modifying the ADDRESSBLOCK field manually is hazardous because any alteration to the properties of an ADDRESSBLOCK field will overwrite your settings. But for example, the following ADDRESSBLOCK fields should result in suppression of the lines for _COMPANY_ and _STREET1_ but would always put a paragraph mark (or line break, depending on what's actually there) before _CITY_

Right now I couldn't tell you whether or not the rules are the same on Mac Word (2011) as in (say) Word 2010, but I think they are. (FWIW, there's no ADDRESSBLOCK on Mac versions of Word). I have not checked any of the above stuff for Word 2013.
