Sub Test()

I can re-enter them by cutting and pasting or dragging, but would like it not to happen in the first place. Any suggestions?
john mcghie say:
Hi Jez:

Three things come to mind:

1)  Tracked Changes

2)  External images

3)  Incompatible file types.

If you are using the Track Changes feature, and you were to delete a paragraph or part of it, and you did not have your "Show non-printing characters" button enabled, it's possible to mark the figure anchor as a deletion.  Which will cause the picture to disappear, even though it is still in the file.  If you "Accept all changes in the document" that will resolve this condition (although be careful: any pictures that have been marked for deletion will then be removed from the file).

If the pictures in the file are stored externally (in a separate drive or folder) they will not arrive on your machine with the download.  When you first open the document, Word may display the "placeable header", which is a copy of the image it makes in its native format.  But if some edit moves or resizes the picture, Word will need to the original picture file, which won't be there, hence the image vanishes.

If the image was made with an incompatible graphics application (Visio or ChemDraw can sometimes do this) you have images that can be displayed by the PC, but unless the application that made the image is on the Mac, if you do anything to the image, it will vanishe because the Mac can't interpret the image file.

My guess is that the cause is most likely to be Tracked Changes.  Make sure Track Changes is OFF and all the "Highlight changes" options are ON.

Hope this helps

boblarson say:

Checkboxes can have triple state - Yes, No, Null.  If the checkbox is unbound then you need to set the value to NO on the open of the form so that it isn't null.  Or you can simply check for null AND NO, You should modify your code to check for NO as well.

Bob Larson, Former Access MVP (2008-2010) http://www.btabdevelopment.com (free Access tools, tutorials, and samples)

marshall barton say:

SInce you do not have an Else block for either If block, it might be difficult to figure out why.

The reason appears to be that chkExactNonARVPresc is not equal to True.  That could happen if its value is Null, False or any number other than -1.  And that can happen if the check box is bound to a table field that is not Yes/No data type, posiibly its the Number data type.

A clumsy way to try to figure out waht's happening is to add one or more MsgBox line that displays the value of the checkboxes:
   MSGBox "chkExactNonARVPresc=";chkExactNonARVPresc, "chkPrescription=";chkPrescription, "chkOTC=";chkOTC

A more versatile way would be to set break points on the If statements and inspect the value of each check box to find out what values they actually have.  If you are not familiar with breakpoints, open the form in design view, switch to the VBA window and click in the left margin of the If statements.  That should highlight the lines in red and display a big red dot in the margin.  Then switch the form back to form view and run your test.  When the code executes, it should stop at a breakpoint and you can view the value of each check box just by moving the mouse over the check box name.

WARNING:  Do NOT edit the code module in any way when the form is not in design view as that is a common cause of code corruption.
