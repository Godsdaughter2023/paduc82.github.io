gina whipp say:

Well, you really don't have to do much for that to happen.  It does sounds like you *care* about the value in the field and are using the autonumber for something other than a unique ID.  If so, that's a mistake. Autonumbers are not guaranteed to be sequential and you can easliy *lose* a number or they can change to negative number, which is what you are seeing now, with no chance of recovery or being *repaired*.

 

I would suggest using DMax on field in your table…

 

DMax("YourIDField","YourTable") + 1

 

You can place that on the Before_Insert of your form and achieve the same results.  You can also take a look at…


http://www.rogersaccesslibrary.com/forum/forum_posts.asp?TID=395

 

Leave your Autonumber field as the PK but display the field that has your sequential numbers.

john w. vinson say:

Perhaps you or someone (intentionally or inadvertantly) changed the autonumber's "New Values" property from Increment to Random.

 

It shouldn't matter much; the purpose of an autonumber is to provide a unique nonrepeating identifier. They will always have gaps. Access "Replication" - now no longer available - depended on random autonumbers to allow disconnected users to add new records without competing for the "next" ID number.

 

I'd suggest Compacting your database; if you don't want the big awkward numbers you might need to create a new table with a sequential autonumber, and run an Append query appending the ID's up to 12015 (excluding the negative numbers - random numbers can be from -214783658 to -1) including the ID, and the negative and "big" numbers EXCLUDING the ID, letting them autonumber. If the sequence of autonumbers is important... well, that's really too bad, since that information is now forever lost.

daniellejones say:

I am getting several different responses. Once again, I typed documents in word 2003 but closed word without saving them. I am told that i can use the AutoRecover feature OR some temporary file to possibly retrieve them. Others are telling me that word 2003 AutoRecover only works if the system crashes and is unlike word 2007 and 2010 where AutoRecover will recover documents if word is closed.  Some one please help in answering this. THANKS!!!

jay freedman say:
In Word 2003, the AutoRecover feature writes the .asd file at intervals of 10 minutes or whatever you set in the Options. As part of the code that runs when you close Word normally (without a crash), that file is erased. If Word crashes, the closing code doesn't get a chance to run, so the .asd file is still present and will be recovered when you restart Word.

If you did close Word normally without saving the file, AutoRecover has no way to help you.

You might be able to find one or more temporary files that haven't been erased, by searching for some word or phrase that you know is in your lost document. The probability of recovering a complete document in this way is fairly low but not zero.

jay freedman say:
According to http://support.microsoft.com/?kbid=211632,

"A temporary file is a file that is created to temporarily store information in order to free memory for other purposes, or to act as a safety net to prevent data loss when a program performs certain functions.... temporary files only exist during the current session of Word. When Word is shut down in a normal fashion, all temporary files are first closed and then deleted."

This is a slight oversimplification, because Word doesn't always delete all its temporary files. Those leftovers are the places where, if you're lucky, you might find some or all of your document. Also, there are programs that can read "deleted" files. When a file is "deleted" in Windows, the part of the disk it occupies isn't immediately erased; it's just marked as available.  But any program may write different data in that area at any time. The more you use your computer, and the longer the time since the file was deleted, the more likely it is that the file's data area will be re-used by something else.

Re-register the application

There are several different ways you can reset the proper file associations. Word itself provides a way to do it. Open a command prompt window and, at the command line, enter the following:

winword /r

Nothing seems to happen when you press Enter, but Word will reregister itself back in Windows. In the process, it resets the file associations for the files it uses, such as DOC and DOT files. You can close the command prompt window and double-click on a document; Word should start right up.

 

Alternate Re-register the application

Close the app. Open the Run Window: <WIN> R, use the following command (Modify the Path to match your specific installation):

"C:\Program Files\Microsoft Office\Office\winword.exe" /unregserver

The quotes are necessary, but you may need to change the path so it reflects the location of the winword.exe file on your system. (You can use Windows’ Search tool to locate the path to the winword.exe file.) The /unregserver switch “unhooks” all the references to Word in the Windows Registry.

You should then, immediately, use the following command:

"C:\Program Files\Microsoft Office\Office\winword.exe" /regserver

This command causes Word to rewrite all its Registry keys and to reassociate itself with document and template files. Now you can double-click a document file and see if Windows starts Word and opens the document.
