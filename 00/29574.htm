andrew schwalbe say:

I have no resolution of any kind on this.  I still experience it myself once in a long while (but not probably for a month) and continue to have users report it in but its so infrequent and unable to reproduce it has become nearly impossible to troubleshoot. 

I did receive some information from Microsoft on troubleshooting, but it got me mostly nowhere because the issue is so hard to reproduce.  I'e summarized the steps here, which is what I send to users having the problem.

This data is the first thing Microsoft will ask of us, so if we submit right away when we open the ticket it will produce a faster turnaround.

  1. Quit all office apps
  2. C:\Users\<Username>\AppData\Local\Microsoft\Office\16.0
  3. Rename the Lync folder to something else
  4. Fire up Skype, reproduce the issue.
  5. Exit Skype.  A new Lync folder will be created at the above location.
  6. Install the Office Configuration Analyzer (OffCAT).  Ask IT for download, or download from: https://www.microsoft.com/en-us/download/details.aspx?id=36852. 
  7. Run OffCat from the stat menu.  Set it up to scan Skype for Business under the More Applications area.  Export/save the resulting file.
  8. Collect the logging info below and send to IT via email or Temp drive. Ask IT for assistance if needed.
    1. Zipped up version of the new Lync folder created in the above steps.
    2. Export your Application and System event logs.
    3. The OffCat resulting file created above.

As you can see, a major part of the issue is needing to get the logs to a decent size so they can be troubleshot.  One of our engineers that experiences this problem wrote a script (below) and shared it with me.  What this script does is clear the Skype logs every morning so when the issue does happen, the logs will be small enough that they can be uploaded to Microsoft O365 support.  He set this powershell script up to run on his computer on every log in.  

Get-Process lync |   Foreach-Object { $_.CloseMainWindow() | Out-Null } | stop-process -force
$currentdate = get-date -format yyyy-MM-dd
$newname = 'c:\users\USERNAME\appdata\local\microsoft\office\16.0\Lync-' + $currentdate
rename-item -path 'c:\users\USERNAME\appdata\local\microsoft\office\16.0\Lync' -NewName $newname -ErrorAction ignore
$count = 0
$item = $null
$item = get-item -path 'C:\Users\USERNAME\AppData\Local\Microsoft\Office\16.0\Lync' -ErrorAction Ignore
 
while (($item) -and ($count -lt 6))
{
    Start-sleep 10
    write-host 'Sleeping - ' $count
    rename-item -path 'c:\users\USERNAME\appdata\local\microsoft\office\16.0\Lync' -NewName $newname -ErrorAction ignore
    $count = $count +1
    $item = get-item -path 'C:\Users\USERNAME\AppData\Local\Microsoft\Office\16.0\Lync' -ErrorAction Ignore    
}

start-process 'C:\Program Files (x86)\Microsoft Office\root\Office16\lync.exe'

Unfortunately, we have yet to see the issue since he started doing this (or at least have yet to get it reported). 

The only other major thing I did recently was move all our users to the Monthly channel for Office 365 product updates.  I haven't had anyone report this issue or experience it myself since doing so, but also because it's so intermittent, I'm not confident in saying that more recent updates from the Monthly channel resolved the problem. 

It might all be a moot point soon, with Microsoft push to move everyone to Teams.  I expect that we'll be migrating our telephony system to Teams by year end if Microsoft makes the migration available for us.
