So obviously my question is, can this be fixed?

In general, the work-around is:  if you expect accuracy to a specific number of decimal places, explicitly round.  For example, =ROUND(STDEV(A1:A6),6).

That applies to all arithemtic with non-integers.

The problem is:  Excel uses binary to represent numbers, specifically 64-bit binary floating-point, an industry standard.  So most non-integers cannot be represented exactly.

In particular, 0.1 cannot be represented exactly by any sum of powers of 2.  Instead, its internal representation is exactly 0.100000000000000,0055511151231257827021181583404541015625.

(I use period for the decimal point and comma to demarcate 15 significant digits, the most that Excel will format.)

Because of the approximate representation, the sum and subsequent division in computing the average and the variance incur varying degrees of numerical error (difference from expectations mathematically).

In particular, AVERAGE(A1:A5) =

0.100000000000000,0055511151231257827021181583404541015625

which is the same representation as 0.1.  But AVERAGE(A1:A6) =

0.0999999999999999,9167332731531132594682276248931884765625

which Excel displays as 0.100000000000000, but which is not the same as the constant 0.1.  So when calculating the variance, 0.1 - average might not be exactly zero.

You might notice that =0.1-AVERAGE(A1:A6) is exactly zero, nevertheless.

But =0.1-AVEARGE(A1:A6)-0 is not exactly zero, as we would expect.  This reflects the true arithmetic result, which Excel uses internally (usually).

Excel returns exactly zero in the first formula because of a dubious heuristic whereby under very specific conditions, Excel replaces a non-zero result with zero if it deems the non-zero result to be "close enough" to zero.

[EDIT] Of course, mathematically, the "- 0" operation at the end should not make any difference.  The difference is due to the capriciousness of the implementation of the heuristic; that is, the "very specific conditions".  Even adding parentheses around the expression -- that is, =(0.1 - AVERAGE(A1:A6)) -- is sufficient to violate the "very specific conditions" and thwart the heuristic.  Klunk!

I've written the code below to create 4 columns of colored circles, where they line up with the data in the rows. As written, the first row can be changed, as can the margin between circles.



Sub TestMacro2()
    Dim dblHt As Double
    Dim rngC As Range
    Dim lngr As Long
    Dim dblMargin As Double
    Dim lngSR As Long
    
    lngSR = 2  'Row where the data starts
    
    dblMargin = 6  ' Distance between shapes
    
    'On Error Resume Next
    ActiveSheet.Shapes.SelectAll
    Selection.Delete
    On Error GoTo 0

    
    dblHt = Rows(lngSR).Height * 4
    
    For lngr = lngSR To Cells(lngSR, "B").End(xlDown).Row
        ActiveSheet.Shapes.AddShape(msoShapeOval, _
            Cells(lngSR, "D").Left + ((lngr - lngSR) Mod 4) * dblHt + dblMargin, _
            Cells(lngSR, "D").Top + Int((lngr - lngSR) / 4) * dblHt + dblMargin, _
            dblHt - 2 * dblMargin, _
            dblHt - 2 * dblMargin).Select
        Selection.Name = "Round" & Cells(lngr, "B").Address
        Selection.ShapeRange.TextFrame2.VerticalAnchor = msoAnchorMiddle
        Selection.ShapeRange(1).TextFrame2.TextRange.Characters.Text = Cells(lngr, "B").Value
        With Selection.ShapeRange(1).TextFrame2.TextRange.Characters(1, 2).ParagraphFormat
            .FirstLineIndent = 0
            .Alignment = msoAlignCenter
        End With
        With Selection.ShapeRange(1).TextFrame2.TextRange.Characters(1, 2).Font
            .Bold = msoTrue
            .Fill.Visible = msoTrue
            .Fill.ForeColor.ObjectThemeColor = msoThemeColorText1
            .Fill.ForeColor.TintAndShade = 0
            .Fill.ForeColor.Brightness = 0
            .Fill.Transparency = 0
            .Fill.Solid
            .Size = 24
        End With
        With Selection.ShapeRange.Fill
            .Visible = msoTrue
            If Cells(lngr, "B").Value > 95 Then
                .ForeColor.RGB = RGB(0, 176, 80)
                ElseIf Cells(lngr, "B").Value >= 90 Then
                .ForeColor.RGB = RGB(255, 255, 0)
            Else
                .ForeColor.RGB = RGB(255, 0, 0)
            End If
            .Transparency = 0
            .Solid
        End With
    Next lngr
    Range("A1").Select
End Sub

albert d. kallal say:

A really simple solution would be to simply specify a custom ribbon for all your reports. Then when any report is launched, you ONLY see the ribbon/report options, and then when you close the report, the previous ribbon will automatic re-display without any coding on your part.

The basic point I am making is you are FAR better off to simply specify a ribbon for the given report or form. If you have several reports or forms open, then simply jumping to that form or report will automatic take care of the ribbon switching for you (Access will do the work).

Attempting to control what tab is set when jumping between multiple forms will over time become like some “dog chasing its tail”. The amount of coding you will encounter over time attempting to just “manage” the simple act of a user switching or flipping between several open forms (or reports) will become a huge challenge. You have to start placing code in each forms “activate” events etc. And you have to determine when focus goes aways.

What happens if the report is not closed, or the user switches to another open form?

The amount of code to “manage” this issue will become too complex over time.

If you simple specify a whole ribbon for the given form, or build a ribbon that can be used for all reports, then no code is required to “switch” the ribbon or tabs.

Here is a calendar booking form I have with a custom ribbon. The “print calendar” simply launches a report that prints the interactive calendar form. And when viewing the  report then user has options to save as PDF or email etc.

So the form looks like this:

When the "print calendar" button is clicked, then a report is launched, then the report has a custom ribbon also. So the user then sees this:

The above ribbon gives all the standard report options such as save to PDF, or print or email etc.

And in place of “close” report, I have a “back” button in all my ribbons – users naturally click on this to “go back”, and that option is to simple close the current object. Users click on this option without thinking – since it looks much like a browser “back” button – this thus reduces training!

So it better to display a WHOLE new ribbon then jump around some existing tabs. I mean what happens if user has a form open, and they click on the other tab but the report not yet open? They now have a bunch of ribbon options for a report, but no report open.

Most simple here is to specify a whole ribbon for a given form (or report). Doing so will thus eliminate most if not all code required to switch to some tab. As this means if more then one form or report is open, then when user changes from several forms or reports open, then ribbon will automatic change for you for the given object (and do so without any code).
