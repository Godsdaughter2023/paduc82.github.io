john w. vinson say:

A "non-equi join" is indeed possible - but I think the logic of your relationship is backward! Typically the "child" (dependent) table would be defined more narrowly than the parent table; in your case it's broader. Knowing the value in MyDate would not constrain the values of periodStartDate and periodEndDate, and you could have many different date ranges in the child table, all of which would match.

 

What you could TRY - no guarantees! - is something like

SELECT <whatever fields>
FROM table1_somestuff INNER JOIN table2_periodDetails
ON table1_somestuff.myDate >= table2_periodDetails.periodStartDate
AND table1_somestuff.myDate <= table2_periodDetails.periodEndDate

Since the join is not (cannot be!) unique, I doubt that this query will be updateable, if that's an issue.

tmscobra say:

Outlook 2007 Import / Export function

I have just noticed that the Import /Export function has disappeared from my File menu. how do I get it back or is there another way to retrieve PST files i have archived on external storage.

brian tillman [outlook mvp 2007-2019] say:

If you have PSTs on external storage and wish to access their contents, if the external storage is a writable device (flash drive, USB disk, etc., but not DVD+-RW), then you can simply open the files from within Outlook.  File>Open>Outlook Data File.  For safetly sake, so as to not modify a file you are keeping for backup purposes, it would probably be better to copy the PST to a subfolder of Documents or some other folder you create and then open them from that location from within Outlook.  Import/Export always loses data and should never be used when transferring PST data into Outlook.  There's simply no need.  Likewise, if you want to keep backups of your PSTs, don't export.  Just close Outlook and copy the PSTs.  Use export/import for transferring data between Outlook and some other program, not between two Outlooks.
