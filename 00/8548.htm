OK, actually your table structure seems fine. You have a many to many relationship between clients and the types of fixes:

"one customer can have many versions/program fixes/datafixes

one program fix can have many customers"

These seem to be the Junction tables that model those relationships:

a datafixes sent which links customer and datafix and a program fixes sent which links customer and program fixes.

However, even if the forms are only used for lookupsI would not base them on multi-table queries. I would have three forms:

1) main form bound to clients table with 2 subforms, one bound to the program fixes junction table, the other bound to the data fixes junction table. Both linked on ClientID and in Continuous form view. This would allow you to see all fixes applied to a client. 

2) main form bound to the program fixes table with a subform bound to the Program fixes junction, linked on ProgramFixID. This would allow you to see all clients that received a fix. 

3) main form bound to database fixes table with a subform bound to database fixes junction table linked on DatabaseFixID. This would allow you to see all clients that got a specific database fix.

This may eliminate the problem you are encountering and make your database work better.
