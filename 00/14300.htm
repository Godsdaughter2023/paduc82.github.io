    tisky zheng msft say:

    Hi Saurav,

     

    To keep good performance, it is not feasible to disable or increase the list view threshold in SharePoint Online. 

     

    To work around the limit, we've outlined some actions you can do to keep your queries within the limit.

     

    1. Index and filter. Planning and, creating indexes and using them in filters can keep the number of items to under the List View Threshold.

     

    2. Using folders to organize. You can use folders to organize data effectively, though you should be careful not to have a query on the folder return more than the List View Threshold.

     

    3. Using Document center site. A Document center is a template that can be used to create a site with features geared for document search, storage, and manipulation. With the right permissions, you can create a site or subsite with this template.

     

    4. Synced and offline data. Taking data offline allows you to use Excel or Access to query your list data without limits. When you sync a document folder, you can work locally on your computer, and changes are updated to the server automatically.

     

    For more information, you can refer to this article: Manage large lists and libraries in SharePoint.

    aykutkurtay say:

    Hi James and All,

    Ok here is my solution at least for now work around. I need to find why this happens on each new created account. 

    We migrated users from Exch 2007 and are using Exch 2013 hybrid servers now. 

    New created users are not getting X500 assigned on premise which supposed to happen as before. In meanwhile we raised AD to 2012. 

    Looking into X500 on cloud and it is same as the legacyExchangeDN for the AD account on premise. I basically copied it over but it didn't fix my issue. (it was not starting with ExchangeLabs indeed.) 

    I started outlook Test Email Auto configuration and on XML I found what exactly Outlook was looking for as LegacyExchangeDN. 

         <LegacyDN>/o=ExchangeLabs/ou=Exchange Administrative Group (FYDIBOHF23SPDLT)/cn=Recipients/cn=Username</LegacyDN>

    I added this as Email Proxy X500 for on premise mailbox account. 5 minutes later outlook started to work and I am able accessing Public Folders without issue now. 

    Hope this helps someone else out there. 

    Regards

    Aykut

    james maccallum say:

    Hi. Yes we did get this resolved. There was a X500 address mis-match between the Cloud Mailbox and the On-premise remote mailbox object.

    X500 address will start something like /o=ExchangeLabs/ou=Exchange Administrative Group.
