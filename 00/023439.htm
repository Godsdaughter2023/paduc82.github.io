shane devenshire say:

Regarding these formulas:  First, in my example I used a test range of 20 rows but you say you had 2000+.  All of my formulas, and the ones you are using, are for that range.  So all three formulas will fail for a range of only 20 rows because:

EM3:EM20  is 8 characters long, the expected range was EM3:EM2224  which is 10 characters long.  so for test purposes the three formulas, for a range of 20 rows should be:
=INDEX(INDIRECT(LEFT($I$1,8)),ROW(A1))
=INDEX(INDIRECT(IF(LEN($I$1)=8,NA(),MID($I$1,9,8))),ROW(A1))
=INDEX(INDIRECT(IF(LEN($I$1)<17,NA(),MID($I$1,19,8))),ROW(A1))

How do they work the original ones work?

You pick one of the choices from your pick list and the VLOOKUP function returns the corresponding ranges to the cell I1.

Left(I1,10) returns the ten left characters from I1.  In the case of a 1 column reference this is the entire reference.  This would beEM3:EM2224.   INDIRECT(LEFT(I1,10)) returns the range EM3:EM2224.  The function ROW(A1) returns the row of A1 which is 1. The function =INDEX(INDIRECT(LEFT($I$1,8)),ROW(A1)) amounts to =INDEX(EM3:EM2224,1)  in other words this returns the first entry in the range.  When copied down the A1 becomes A2 and the formula returns the second row in the range.

jggtn say:

Yes. Solved.

AutoRecovery files are not only hidden but they are saved as an Excel binary files.

Additionally, Microsoft suppresses Excel's ability to see hidden files in the AutoRecovery folder(s) from Menu > File > Open window (unless you hit the Command+Shift+Period keys together while in the Open window.)

The solution I used was:

        (1) :  Locate Auto Recovery folder

        (2) :  Locate target file (modify OS X settings to reveal hidden files & folder)

        (3) :  Reset file permissions & unhide

        (4) : Change file extension to .xlsb (if needed) and test in Excel

1. Locate the AutoRecover folder(s) [yes, Virginia, there can be more than one location, especially if you are migrating from Office 2011 or were part of the beta {which, in itself, may be part of the problem; but I digress.}]

If the AutoRecovery folder isn't in the following location:

file:///Users/<account>/Library/Containers/com.microsoft.Excel/Data/Library/Preferences/AutoRecovery/

you may find it here:

file:///Users/<user account>/Library/Application Support/Microsoft/Office/Office 202011 AutoRecovery/

[file:///Users/gomberg/Library/Application%20Support/Microsoft/Office/Office%202011%20AutoRecovery/]

(Your may prefer to configure Finder to see hidden files. Http://ianlunn.co.uk/articles/quickly-showhide-hidden-files-mac-os-x-mavericks/ is one of many web pages that has walk through instructions demonstrating how you can do this for both Mavericks & Yosemite. (Or you can Google instructions for your version of OS X.)

2. Open the AutoRecovery folder and locate your desired file.


If it is hidden and/or compressed, its icon's color will be muted if Finder is configured correctly. Excel appends a prefix string to the beginning of the file name which reads "AutoSave to <8 character random string>"<original file name> (all without quotes or brackets.)

3. Change the target file's permissions & Unhide. There are a number of ways to do this. From Finder, right click on the target's icon and select "Get Info". At the bottom or the Get Info panel is "Sharing & Permissions". There should be three entries. They should read as follows:

Name                  |  Privilege

--------------------------------------------------------

<user> (Me)        |  Read & Write

 staff                    |  Read only

 everyone            |  Read only

(where <user> is probably your login name)

Click the black triangle to expand the section (if it isn't already expanded.). Click and unlock the padlock icon on the bottom right of the section. Add in any elements missing from the above and set the privileges as shown.

If "staff" is not an entry option (it wasn't in mine) you'll have to execute "chmod 644" (without quotes) from the command line in the Terminal app. So, say your original file "Budget.xlsx" was saved to "AutoSave to 1BADF522Budget.xlsx". Go to Applications, open the Utilities folder and open Terminal (I personally use iTerm). After the command prompt "$", place your cursor and type (or paste [without the quotes]):

"chmod 644 /Users/gomberg/Library/Containers/com.microsoft.Excel/Data/Library/Preferences/AutoRecovery/AutoSave\ to\ 1BADF522Budget.xlsx"

Hit Enter.

Next, unhide the target file. Type (or paste [without the quotes]):

"chflags nohidden /Users/gomberg/Library/Containers/com.microsoft.Excel/Data/Library/Preferences/AutoRecovery/AutoSave\ to\ 1BADF522Budget.xlsx"

(Note: chmod & chflags both work with wildcards if desired.)

If you reopen a new Finder - Get Info panel for the target file your Name and Privileges columns should now be correctly populated and configured.

4. Change file extension: If you are like me and save your workbooks as .xlsx, when you double click to open the target file you will get an Excel popup message that reads:

don't panic. The file is not corrupt. Excel auto-saves as Excel binary files without changing the .xlsx file extension to .xlsb. Simply rename the file extension to .xlsb and you should be good to go.

Alternatively, you can hit the Command+Shift+Period keys together on your keyboard as you navigate from Excel > File > Open window. This will temporarily reveal any hidden folders and files as you navigate to the AutoRecovery folder(s).

I'm not suggesting this is either the best and/or only approach. It was simply the path that I ended up following while hacking together a solution.

ken sheridan say: As Bill says, use a query or queries in the front end to restrict the data available to the users.  You can do this without having to amend your forms, reports, other queries etc by means of a little subterfuge:

1.  Change the names of the linked tables in the front end.  If you have a linked table names Orders say, change its name to Orders_Lnk.

2.  Create a query based on Orders_Lnk which restricts the rows returned to those you want to be available to the users, and save this query with the name Orders.

All your existing form's, reports, queries etc which referenced the Orders table should now work as before but will only show the restricted data.  Do the same with any other linked tables you need to restrict.

Most VBA code which referenced the original tables should still work, but will now be referencing the query, e.g. an expression such as DMax("OrderDate","Orders") would still work, now returning the latest order from the restricted set rather than from the whole table.  Some code, which specifically refers to a table won't work, so you may need to debug this, e.g. code such as

Dim tdf AS DAO.TableDef
Set tdf = CurrentDb.TableDefs("Orders")

would need to be changed to:

Dim qdf AS DAO.QueryDef
Set qdf = CurrentDb.QueryDefs("Orders")

bill mosca say: If you want the same filter all the time change your form's recordsource to a query and put in the criteria you want. That's really all a filter does is change the chriteria to limit the records shown.
