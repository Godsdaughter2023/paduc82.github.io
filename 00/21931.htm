Sub ClearData()

    With ActiveSheet

    End With
End Sub

ken sheridan say: Firstly, I should point out that the example in the guide you are using is a very bad one.  Personal names can legitimately be duplicated, so should never be used as keys.  Even if you were to do so it would then make it very cumbersome to relate this table to other tables in one-to-many relationship types as it would be necessary to have composite foreign keys in other tables.  Composite keys are fine where appropriate, but not in this case.  An autonumber 'surrogate' primary key, MemberID say, would be used in this context and referenced by a corresponding foreign key of long integer data type.

Even if, for some reason, it were necessary to allow only distinct values of first and last name in the table, the way to do it would not be to make them a composite key, but to include them in a unique index and use an autonumber as the key.  But that's another story, which we can go into in more detail later should you wish.

I can't test this in Word 2007 or 2010, and Word 2013 is a bit different, but I think it will work...

After you've assigned $word = New-Object -ComObject Word.Application, you should be able to say

$word.ActiveWindow.View.ShowRevisionsAndComments = $False
$word.ActiveWindow.View.RevisionsView = 0

The RevisionsView property can have values of 0 (the enumeration value wdRevisionsViewFinal) or 1 (wdRevisionsViewOriginal).

Once you assign the COM object, your commands go to the VBA engine, and the PS syntax looks a lot like that of VBA, so the reference you need is
http://msdn.microsoft.com/en-us/library/office/ff841702%28v=office.14%29.aspx
There are a few differences, for example PS uses square brackets around the index into a collection where VBA uses parentheses.

Although you're welcome to add Subscript & Superscript to a toolbar if you wish there really is no need to do so. Those tools are already available in the Font group on the Home tab of the Ribbon. You can also use the built-in keyboard shortcuts:

macropod say:

Alternatively, you could type your equations without the subscripts, then run the following macro:


Sub ChemicalFormatter()
Application.ScreenUpdating = False
Dim oRng As Range, fRng As Range, bState As Boolean
Select Case MsgBox("Do you want to process the whole document?", _
    vbYesNoCancel + vbQuestion, "Chemical Formatter")
  Case vbYes
    bState = True
  Case vbNo
    bState = False
  Case vbCancel
    End
End Select
With Selection
  Set oRng = .Range
  With .Find
    .ClearFormatting
    .Text = "[A-Za-z)][0-9]{1,}"
    .MatchWildcards = True
    .Wrap = wdFindContinue
    .Forward = True
    Do While .Execute = True
      Set fRng = ActiveDocument.Range(Start:=Selection.Start + 1, End:=Selection.End)
      If bState = False Then
        If fRng.Start >= oRng.End Then Exit Do
        If fRng.End >= oRng.End Then fRng.End = oRng.End
      End If
      If fRng.Font.Superscript = False Then fRng.Font.Subscript = True
      fRng.Collapse Direction:=wdCollapseEnd
    Loop
  End With
End With
oRng.Select
Set fRng = Nothing: Set oRng = Nothing
Application.ScreenUpdating = True
End Sub

 

Unless you're working with isotopes, the results should be correct - you'll need to apply the isotope superscripting yourself (if the numbers are already superscripted, they’ll be left alone). The above code will correctly handle chemical equations like 'C5H8(N2S)4' and '3(CaO)•2(SiO2)•4(H2O)(gel) + 3Ca(OH)2'.

 

If your document has other alphanumeric strings in which a non-superscripted number follows a letter (e.g. table cell references), you’ll need to select only the range(s) containing the text to be converted and answer ‘No’ to the macro's 'whole document' prompt.
