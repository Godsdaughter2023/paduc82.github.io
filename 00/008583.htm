ben schorr say:

O.K., so there are a couple of ways you could do it:

1. You could convert their user mailboxes into shared mailboxes and give the monitors permissions to that shared mailbox.

2. You could remove the email addresses from their user mailboxes and create a distribution list for each email address than then redirects that incoming mail to the user(s) you want to monitor those addresses. Then delete their old mailboxes.

Option 1 has the benefit of preserving all their legacy mail and removing their user accounts. But it means an extra silo for the monitors to check.

Option 2 is clean and lets you easily assign multiple monitors to a single former-user email address if you want, but doesn't retain their legacy mail (and calendar, contacts, tasks, etc).

ken sheridan say: As Bill has pointed out, it appears that your table is not normalized to First normal Form (1NF) whose formal definition is:

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple contains exactly one value for each attribute.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).

Consequently the table should be decomposed into two tables in a one-to-many relationship type if each zip code is covered by one technician only, broadly speaking like this:

Technicians
….TechnicianID  (PK)
….FirstName
….LastName
….etc

ZipCodes
….ZipCode (PK)
….TechnicianID  (FK)

Or if one zip code can be covered by more than one technician the relationship type is many-to-many, and should modelled by another table, so the tables would be like this:

Technicians
….TechnicianID  (PK)
….FirstName
….LastName
….etc

ZipCodes
….ZipCode (PK)

and to model the relationship type by resolving it into two one-to-many relationship types:

ZipCodeTechnicians
….ZipCode (FK)
….TechnicianID  (FK)

The primary key of this table is a composite one made up of the two foreign key columns.

Returning the technician or technicians who cover a particular zip code is in either case a simple task of joining two tables on TechnicianID and restricting the query on the ZipCode column.

By default, Internet Explorer resets the Zoom level to 100% for every new window and new tab opened. To preserve your Zoom Level setting, use these steps:

Go to the Tools menu, and choose Internet Options;

In the options dialog that opened, select the Advanced tab the under Accessibility.

The option pertaining to zoom levels is the "Reset zoom level for new windows and tabs" checkbox, which is unchecked by default. This means that, when the checkbox is unchecked, Internet Explorer will remember and apply your zoom level across all tabs currently opened, and any new tab you open. If you check this checkbox, Internet Explorer will apply the custom zoom level to the current tab.
