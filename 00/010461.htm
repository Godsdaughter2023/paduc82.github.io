yebyps say:

John

Let's not get into semantics about whether it's a defect or not.  It is certainly a grave lack in functionality.  I've been using Project since the mid 90s and I've seen this issue come up repeatedly with no change in behaviour. 

There are clear and very basic reasons why one should want to have two instances of an application running at the same time.  One very fundamental case is when one has successive versions of a situation and one wants to review changes which have happened between the two. In the case of project planning this is particularly relevant - seeing how similarly formatted bars have shifted by pattern recognition, compared to the version of the file you saved in a previous month, rather than fiddling around with different baselines shown in a cluttered manner on the same screen, is clearly useful.  Working on multiple screens is a common setup and having different instances running multiplies one's screen area. 

Another windows-related deficit is the display of the task (or task details) window.  I daily use this to make fine adjustments to ongoing tasks.  The layout of this feature is a rectangle whose height is greater than its width - portrait style.  The split screen which displays it is landscape. The bottom right area of the screen is blank.  This is a clear mismatch in screen real estate.  Why not have the task details as a floating window which is interactive with the main Gantt chart?  Ah, you might say, why not use Task Information - firstly because when Information is open it closes off clicking on the main screen. This is inconvenient - one would want to click on a different task on the main screen and have the Information details change without closing it.  Secondly because the representation of the task in the Information screen is illogically (through incremental growth from early versions I guess) set out in tabs rather than presented in one view - for example the task dates are on the General tab, and the Constraint date is on the Advanced tab.  

These kinds of very basic needs are appreciated by other software including Word and Excel, and particularly Outlook. Excel was lacking in this respect until 2013 but they did eventually fix it.  I guess the problem with MS Project is legacy spaghetti code. Developers are able to add more and more sophisticated features on top, like the Timeline (which itself would be much more useful if there was more screen real estate to play with) but are perhaps baffled about how to change root behaviour.  However to declare  "Since by definition it is not a defect, there will be no "fix"" I feel is patronising in the extreme.  It is an insult to practical users who are actually trying to change things in the real world.

One day people will fix these fundamentals and it may well be  people in another organisation if your comment represents the attitude to change within the MS project team. 

Yours

Alan
