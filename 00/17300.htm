howiedickerman [microsoft] say:

Hi,

GETPIVOTDATA is known to be an expensive calculation to perform.  It is intended to be used to pluck specific values out of a PivotTable in a way that will continue to work if the layout of the PivotTable changes.  Adding six columns of GETPIVOTDATA formulas to 156,000 rows will require calculating nearly a million instances of the function.  This will all happen on a single logical processor, so if you have four logical processors it will appear that the CPU is 25% busy, but viewing the breakdown, you'll see one processor at 100% while the others are waiting.

If you want to add columns beside your PivotTable, I would encourage you to use cell references instead, because referring to cell D1234 will be MUCH faster, and in this scenario your formulas are dependent on the PivotTable layout staying the same.

This issue is completely separate from the memory usage issues mentioned earlier in this thread.  For large data sets, 64 bit Excel is clearly the way to go.  32 bit Excel will never use more than 2 GB of memory.  64 bit Excel will use all the memory on the machine if it needs to.  It's always important to avoid hitting memory limits, because when that happens, Windows will start paging memory in and out, and performance will become so terrible that the machine will effectively be unusable, and the computer can take hours to return to a workable state.

Hope this helps,

howiedickerman [microsoft] say:

Hi,

If your screen shot was taken while using 64 bit Office, then you're not running out of memory.  Any hang or freeze that you are encountering is due to something else.  If you're running 32 bit Office in this screen shot, you might be hitting the memory limit for the Excel process.

Note that you want to look at memory usage for each process separately.  In the picture below you can see that I have a workbook open in Excel and Excel is using more than 400 MB of memory.   It doesn't matter how much memory I'm using overall.

Having said that, there are many, many reasons why you might encounter a delay or hang that have nothing whatsoever to do with hitting any memory limits.  In fact the odds are enormous that hitting a memory limit is not the issue for you.  Let me list a number of examples:

1 - Excel might be saving the file to disk so that AutoRecover can have a restore point in the event of a crash or other problem.  (See if changing the AutoRecover settings in Options changes this behavior for you.)  This can feel like a temporary hang if you have a very large workbook that takes some time to save, and it can get even worse when it's being saved to a remote machine over a slow connection.

2 - Sometimes  I encounter delays that are due to other things happening on my computer.  When my IT department decides to deploy new builds to my machine or perform special virus scans, my machine gets crazy busy/slow while that is going on.  If I happen to be working in Excel at the time, it can feel like Excel is hung, when the reality is that my computer is busy with something else.

3 - Sometimes add-ins or VBA code or other apps perform operations that can take a while to perform and can make the app seem busy.

4 - If the workbook has connections to external data, the data provider (or connector or link) will sometimes try to connect to the remote data source, and if there's a connectivity problem, there can be a delay while Excel waits for the attempted connection to "time out" after a set time interval.

5 - Sometimes there can be a formula or calculation that is particularly expensive in terms of the compute power it requires.  I see this most often with some complex DAX measures in the data model that are doing nested scans of very large tables, and performing those calculations each time there is any update to anything in any of the PivotTables.  In these cases, using a more efficient formula can address the issue.

6 - We do sometimes have bugs where a specific feature does not perform as it should and the problem presents itself as a delay.  I doubt this is what you're seeing, but if it is, we will want to identify it and fix it.  In this scenario, you would see the hang behavior each time that you attempt to do something specific.  Please tell us what that something specific is, and we'd be happy to look into it.

These are merely examples of dozens of things that can interfere with the running of any program. I'm not suggesting that you have one of these specific problems, but am saying that more investigation would be needed to more accurately track down what's happening.  You might try things like opening the file on a different computer to see if the behavior is the same.  You might try turning off add-ins to see if that helps.  You might try saving a copy of the file to your local "C:" drive and then work with that copy a bit to see if the delays go away. 

We do fix performance issues (and other bugs) regularly, so I do recommend that you make sure to install the latest updates.  Sometimes we see customers who are running a build that is several years old, and simply updating to the latest version with any recent fixes is all that's needed to address problems that they are encountering.  Again, you may already be on the latest and greatest build, but I wanted to mention this just in case.

Hope this helps,

howiedickerman [microsoft] say:

Hi Santiago,

One possibility to explore would be to move some of this into what we call the Excel Data Model.

This is an Analysis Services Tabular Model running within Excel, which is an in-memory column based database that is stored within the Excel workbook.  This approach might be a good way to handle large volumes of data.  The downside is that you may not be familiar with it.  

The window that comes up where we design the database is called Power Pivot, and the formula language that is used to define measures is called DAX.  It uses Excel syntax, but there are some different functions. There are advantages that include really good of copies of the formula in a worksheet where it's easy for something to accidentally get changed in a way that's hard to identify.

By building a relational database in memory, you eliminate the need for all the vlookup, index, match, etc. formulas.  But I wouldn't try to do something of this size and complexity as my very first "Data Model".  If you've never worked with Power Pivot before, you might want to take a look at a book on Power Pivot written by Rob Collie.  I'd build something smaller and simpler to start with, and then consider building the full model using Power Pivot after getting more experience with it.

I'm not surprised to learn that turning off the automatic saves for auto-recover has dramatically reduced the problem you were seeing.

Hope this helps,

chip pearson say:

Drag a Forms (not ActiveX Control) dropdown control on to your worksheet. Right-click it and choose "Format Control", then the Control tab. There, enter the range of cells that contains the list of values to display. Click OK to get out of the dialog boxes. Create a procedure like

 

Sub ColorRow()
    Dim DD As DropDown
    Set DD = Worksheets("Sheet1").DropDowns("DropDown1")
    Select Case LCase(DD.Value)
        Case 1
            ActiveCell.EntireRow.Interior.ColorIndex = 3 'red
        Case 2
            ActiveCell.EntireRow.Interior.ColorIndex = 6 'yellow
        Case 3
            ActiveCell.EntireRow.Interior.ColorIndex = 23 'blue
        ' and so on
    End Select
End Sub
   
Then, right-click the drop down control and choose Assign Macro an choose ColorRow from the list. When the drop down is clicked, it will execute the procedure ColorRow. This code looks at the control and gets the selected index value (1-based) and applies a color to the entire row of the active cell.

Hi PaulW_912,

It is possible using VBA to set Conditional formatting with color in a word table.

1. Click Macros in View tab > View macros > type in the name, create > replace all the content with the following code > Save.
Example explanation: This can change the color of value < 0 to red, 3 > value ≥ 0 to yellow, value ≥ 3 to green, and blank cells to white.
Note: You can replace the range you want in the following code.

Sub colourSelectedTable()
Dim c As Word.Cell
If Selection.Information(wdWithInTable) Then
  For Each c In Selection.Tables(1).Range.Cells
    If IsNumeric(Left(c.Range.Text, Len(c.Range.Text) - 1)) Then
      If Val(c.Range.Text) < 0 Then
        c.Shading.BackgroundPatternColor = wdColorRed
      ElseIf Val(c.Range.Text) >= 3 Then
        c.Shading.BackgroundPatternColor = wdColorGreen
      Else
        c.Shading.BackgroundPatternColor = wdColorYellow
      End If
    Else ' set non-numeric to White
      c.Shading.BackgroundPatternColor = wdColorWhite
    End If
  Next
End If
End Sub

2. Put the cursor into the table > Click View > Macros > View macros > select the macros you created > click Run to check the outcome:

The Calculate event does not have a Target. 

So the line

If Intersect(Target.Cells, Range("1:50"), Union(Columns("A"), Columns("D"))) Is Nothing Then Exit Sub

will give an error.

You might consider your original code, the Change event, but have it check for changes in the J column or wherever they can occur.  In other words you might expand the Worksheet_Change events range to include some or all of column J in addition to any other cells that you actually change manually.
