macropod say:

Hi Tommy,

You really don't need the ' If oHeader.Exists Then' or ' If oFooter.Exists Then' tests, since the respective For Each loops  already take care of that. Even so, deleting those tests would have little impact on the performance. Of more importance, you shouldn't need a macro to update the fields one by one at print time - at most, you might want a macro that ensures the 'Update fields before printing' option is checked, under Word Options|Display (or, with Word 2010, 'update linked data before printing' under File|Options|Display). As it is, having a macro to update the fields as you have now may only mean that installations of Office with the 'Update fields before printing' option set are actually updating the fields *twice* at print time.

I also note that, although you say the only thing that has changed is an upgrade from Office 2003 to Office 2010, you're also now running Windows 7. If the INCLUDETEXT links are over a network drive, it may be that the network isn't configured correctly and that could explain the performance hit. plus, you'd need to make sure the INCLUDETEXT links are pointing to the corrrect versions of the files (ie change the field's file reference from .doc to .docx if they're now meant to point to a docx file).

{IF {INCLUDEPICTURE {IF TRUE «Image»} \d} {INCLUDEPICTURE {IF TRUE «Image»} \d}}
or:
{IF {INCLUDEPICTURE {IF TRUE {MERGEFIELD Image}} \d} {INCLUDEPICTURE {IF TRUE {MERGEFIELD Image}} \d}}

The above field constructions insert the images and breaks the links to them as the merge executes. That should minimize/eliminate any delays. It also means you don't need to keep the images afterwards.

Note: The field brace pairs (ie '{ }') for the above examples are created in the body of the document via Ctrl-F9 (Cmd-F9 on a Mac) - you can't simply type them or copy & paste them from this message. Likewise, you can't type or copy & paste the chevrons (ie '« »') - they're part of the actual mergefields, which you can insert from the mailmerge toolbar.

As it happens, I can answer that because I recently downloaded and installed Open Sans Condensed and Open Sans Condensed Light (these fonts are specified by Rotary International for Rotary websites, and I'm our club's webmaster, trying to update our site). They seem to display and print correctly in both Word 2003 and Word 2013, so they should be okay in Word 2007 and 2010 as well.

On a PC, fonts are not imported into Word but installed in Windows. Word then has access to all fonts installed in Windows. As described in this TechNet article, font licenses include varying levels of "embeddability": Installable, Editable, Print & Preview, and Restricted. There was a time when an Installable font would be installed automatically when a document using it was opened. This no longer happens (I guess it was a privacy/security issue that users objected to), but in any case the Open Sans font family have Editable embedding, which means that a document containing these fonts can be opened on a system on which the fonts are not installed and edited using the font. To have the font available for other documents, however, it would have to be installed in Windows.

All of which is to say that you can provide a template that uses Open Sans, and you can provide instructions for downloading and installing the fonts, but it would probably be simpler to supply the fonts with the template. In Windows 7 and 8, the fonts can be saved anywhere; installing them just requires that you right-click the font file and choose Install from the context menu or preview the file and click the Install button.
