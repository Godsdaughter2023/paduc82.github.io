net stop msiserver

How to auto create records for each day in Access 2016? - Microsoft Community

evanwsmith say:

How to auto create records for each day in Access 2016?

Hello,

I have a form called "Log1" in Access 2016 that has 24 records, one for each hour of the day to record temperatures, etc for each of those hours. There is a "Hours" field in the "Log1" table that displays the hour in the "Log1" form for each record (example:  hour 1 is record #1, hour 2 is record #2, hour 3 is record #3, etc) until it reaches record # 24 for the 24th hour.

Once I reach record #24 (24 hours), is there a way for record #25 to start a new day automatically (day #2), and repeat with another 24 records for the 24 hours in day 2, and do so for every day after?

If not, is there another way to auto create a new day with 24 records in each day so the user does not have to manually do this when inputting the data?

Any help is greatly appreciated.


Thanks!

ken sheridan say: Your hours table is missing a row for midnight, 0000.  There are then two ways you could approach it:

1.  You can create a table with columns logDate, logHour along with the twenty columns for the attributes Oil Cooler Water Outlet etc and execute the query I gave you:

INSERT INTO NameOfYourLogTable(logDate,logHour)
SELECT calDate, hour
FROM Calendar,Hours
WHERE YEAR(calDate) = 2016;

This will fill the table with rows for each hour of each day in 2016, with values in the logDate and logHour columns ready for values tom be entered into the Oil Cooler Water Outlet etc columns for each hour as and when the readings are available.

You don't have to do it for the whole year at once of course; it could be done on a daily basis with:

INSERT INTO NameOfYourLogTable(logDate,logHour)
SELECT calDate, hour
FROM Calendar,Hours
WHERE calDate = DATE();

This will insert 24 rows into the table, each with the current date and the hours of the day.

2.  The other way to do it would be to dispense withe log table in its current format and have a table such as:

LogDetails
....logDate
....logHour
....itemLogged
....reading

The primary key of this table would be a composite one of its first three columns.  The itemLogged column would have the values Oil Cooler Water Outlet etc in separate rows, and this column would be a foreign key referencing the primary key of an ItemsLogged table with a single column of 20 rows.  In the data entry form you'd have bound combo boxes to select the date and the time with RowSource properties respectively of:


SELECT calDate FROM Calendar ORDER BY calDate;

and:

SELECT Hour FROM Hours ORDER BY Hour;

The logHour column in LogDetails and the combo box bound to it could have a DefaultValue property of Date() to insert the current date automatically by default when a row is inserted via the form.

The ItemLogged would be selected in a combo box with a RowSource property of:

SELECT itemLogged FROM ItemsLogged ORDER BY itemLogged;

Rows for a period could similarly be inserted with a query, e.g. to insert rows for the current day:

INSERT INTO LogDetails(logDate, logHour, itemLogged)
SELECT calDate, hour, itemLogged
FROM Calendar, Hours, LogDetails
WHERE calDate = DATE();

It could be argued that the latter approach is relationally more valid as it regards each item logged as a value of an itemLogged attribute, rather than the item as an attribute per se.  However, I would see no strong objection to the first approach in this context.
