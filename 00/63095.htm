TDR can cause mouse inoperability - Microsoft Community

markermolov say:

TDR can cause mouse inoperability

Hello,

I've encountered with bug in win32k DIT/RIT handover, that occurs when TDR (Timeout Detection and Recovery) mechanism in DXGK causes GDI reset (and DWM/MIL partition restart). In brief, if there are many mouse events during TDR (win32k!InputApc is called often), after TDR has been completed mouse buttons aren't clickable at all, and cursor moves slowly (is jumping with delays).

The problem is in DIT, that during control transferring to RIT, due to race condition between CInputManager::DwmInputThread and QueueMouseEvent, can become having inconsistent state and insensitive to mouse events.

Please see ETW win32k trace log of the bug:

The key moment is that highlighted WakeDIT had not caused ProcessesQueueMouseEvents before following WakeRIT.

Result has been shown in second window: no one of following WakeDIT causes ProcessesQueueMouseEvents.

Please see win32k!WakeDIT function code:

signed __int64 __fastcall WakeDIT(unsigned int a1)
{
  signed __int64 result; // rax@1

  result = 0i64;
  if ( gbDIT )
  {
    __asm { prefetchw byte ptr cs:gdwDITWakeReason }
    if ( !(_InterlockedOr((volatile signed __int32 *)&gdwDITWakeReason, a1) & a1) )
      ZwSetEvent(ghDITEvent, 0i64);
    result = 1i64;
  }
  else
  {
    _InterlockedOr((volatile signed __int32 *)&gdwDitIgnoredWakeReasons, a1);
  }
  return result;
}

As we can see, ghDITEvent is fired only if gdwDITWakeReason doesn't already have specified bit set (0x40 for mouse event).

So, if gdwDITWakeReason isn't zero at DIT startup (UserActivateDwmInputProcessing), the DIT thread will be insensitive to events marked by this bit field, what is our case when WakeDIT called from QueueMouseEvent when DIT thread is between UserProcessDwmInput and  CInputManager::CleanupDwmInputThread.
