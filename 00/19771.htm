dirk goldgar say:

Sorry for the delayed reply; I was busy.

 

I would not do it the way that is suggested at that link, because I think it better not to let the user into the subform at all if the required info on the parent form is not filled out.  This problem boils down to two simple requirements:

 

  1. Don't let a record be saved on main form or subform if required fields are not filled out.
  2. Don't let the user into a subform at all unless the parent form has been filled out.

If you take care of (1), then (2) is a matter of using the Enter event of the subform control (to see if the form that contains that control is on a new record.  If it is, then warn the user and set the focus back to the appropriate control.  You don't have to check the individual controls, because requirement (1) will prevent the record from being saved, and automatically keep the focus from going to the subform if the parent form has only been partially filled out.

 

I have downloaded your database, and find the following modifications handle the situation.  First, put the following utility function into a standard module:

 

'------ start of code ------
Function fncRequiredFieldsMissing(frm As Form) As Boolean

    ' Check form (passed as <frm> for controls marked as  "Required"
    ' in their Tag properties.  If any required control is empty, return
    ' True, display an error message listing all the "missing" fields,
    ' and set the focus to the first missing control (where "first" is
    ' based on the controls' tab order).  In the list of missing controls,
    ' use each control's caption if possible, else use the name of the
    ' control after stripping off the expected object-type prefix (if
    ' present).
    '
    ' Copyright (c) 2013, Dirk Goldgar.  You may use this code freely in your
    '       applications, provided that the copyright notice remains intact.

    On Error Resume Next
   
    Dim ctl As Access.Control
    Dim strErrCtlName As String
    Dim strErrorMessage As String
    Dim strMsgName As String
    Dim lngErrCtlTabIndex As Long
    Dim blnNoValue As Boolean
   
    lngErrCtlTabIndex = 99999999  'more than max #controls
   
    For Each ctl In frm.Controls
        With ctl
            Select Case .ControlType
                Case acTextBox, acComboBox, acListBox, acCheckBox
                    If .Tag = "Required" Then
                        blnNoValue = False
                        If IsNull(.Value) Then
                            blnNoValue = True
                        Else
                            If .ControlType = acTextBox Then
                                If Len(.Value) = 0 Then
                                    blnNoValue = True
                                End If
                            End If
                        End If
                       
                        If blnNoValue Then
                       
                            strMsgName = vbNullString
                            If .Controls.Count = 1 Then
                                strMsgName = .Controls(0).Caption
                                If Right$(strMsgName, 1) = ":" Then
                                    strMsgName = Trim$(Left$(strMsgName, Len(strMsgName) - 1))
                                End If
                            End If
                            If Len(strMsgName) = 0 Then
                                strMsgName = .Name
                                Select Case Left$(strMsgName, 3)
                                    Case "txt", "cbo", "lst", "chk"
                                        strMsgName = Mid(strMsgName, 4)
                                End Select
                            End If
                           
                            strErrorMessage = strErrorMessage & vbCr & _
                                        "   " & strMsgName
                                       
                            If .TabIndex < lngErrCtlTabIndex Then
                                strErrCtlName = .Name
                                lngErrCtlTabIndex = .TabIndex
                            End If
                           
                        End If
                    End If
                Case Else
                    ' Ignore this control
            End Select
        End With
    Next ctl

    If Len(strErrorMessage) > 0 Then
        MsgBox "The following fields are required:" & vbCr & _
                strErrorMessage, _
                vbInformation, "Required Fields Are Missing"
        frm.Controls(strErrCtlName).SetFocus
        fncRequiredFieldsMissing = True
    Else
        fncRequiredFieldsMissing = False
    End If

 

End Function
'------ end of code ------

First, you might be confusing comments about Excel formatting with the VBA function Format.

You are correct that Excel formatting changes only appearance, not the value.

But VBA Format returns a string that appears exactly like its format.  So Format(75.985,"0.00") does indeed return the string "75.99".

-----

Second, I don't see how you determined that Format(75.985, "0.00") - 78.985 returns zero.

Perhaps you are confusing appearance of the result (in some unknown context) with the actual value.  Or perhaps you mistakenly stored the result into an "integer" variable.

We cannot know because you do not provide complete VBA examples, which is essential when discussing how an operation behaves. 

For example, the following displays 4.99999999999545E-03 (about 0.005) in the MsgBox.  And it stores the constant 0.00499999999999545 into A1, which appears as 0.005 when formatted as General.

Sub doit()
Range("a1").NumberFormat = "General"
Range("a1") = Format(75.985, "0.00") - 75.985
MsgBox Format(75.985, "0.00") - 75.985
End Sub

-----

As for why the result is not "exactly" 0.005....  That is due to the usual rounding issues with the native computer arithmetic (64-bit binary floating-point).  If you are unfamiliar with the issue, see my explanation (click here [1]) or MS KB 78113 (click here [2]).

Consequently, =(A1=0.005) returns FALSE(!).

If we wanted the result to be "exact" to 3 decimal places, we should write one of the following in VBA:

Format(Format(75.985, "0.00") - 75.985, "0.000")
or
WorksheetFunction.Round(Format(75.985, "0.00") - 75.985, 3)

Note:  The binary representation is not truly exactly 0.005 because 0.005 cannot be represented exactly with any sum of powers of 2, much less 53 powers of 2.  The exact internal value of 0.005 is 0.00500000000000000,0104083408558608425664715468883514404296875 .

joeu2004 say: Sue wrote:

Yes, we have said as much.

Sue wrote:
but neither produce the required format as well so Worksheet format is required in addition.

That is right.  Usually, when we assign to a cell in VBA code, Excel interprets the right-hand side just as if we had entered it manually.

So if the cell format is not Text and the right-hand side can be interpreted as numeric (numbers, dates, percentages), that is how it is stored in the cell.

For example, if A1 is formatted as General and we execute Range("a1") = "1/3", Excel interprets that as Jan 3 (if our short-date form is MDY).  If A1 is formatted as Number with 2 decimal places, the result is 0.333333333333333.

Sue wrote:

But however I was amazed that subtraction produced an inexact answer.  I had thought that only

when non-terminating division was involved that an answer was "a tiny bit" off.  [....] Is this the flow on from [sic; read: fault of?] the "Rounding"? - or is it the subtraction itself?


As I explained in my aforementioned posting (click here) [1] (TMI?), it is due to the way that Excel and most computer applications (but not all) represent numbers and perform arithmetic, namely relying on the native computer's 64-bit binary floating-point form.

So, for example, IF(10.1 - 10 = 0.1, TRUE) returns FALSE(!).

It affects all arithmetic, not just subtraction and division.  In fact, sometimes differences arise due to the order in which we do a series of additions and/or subtractions.

To make matters worse, Excel tries to "correct" for some (but not all) tiny-bit-off results.  But it is done inconsistently.  So sometimes IF(A1=A2,TRUE) returns TRUE, but IF(A1-A2=0,TRUE) returns FALSE and MATCH(A1,A2,0) returns #N/A (no match) at the same time.

Sue wrote:

Do I need to Worksheet Round or VBA Format any subsequent use of any number produced

by Worksheet Round or VBA Format?


My rule is:  explicitly round whenever we expect the result to be "exact" to a specific number of decimal places.

Bear in mind that "exact" means:  the same internal representation as if we had entered the value manually.

For example, 1.23 cannot be represented exactly internally.  But Excel represents the constant 1.23 only one way, even though there are many internal representations, the result of calculations, that can appear as 1.23000000000000 when rounded to 15 significant digits, as Excel does.

Sue wrote:
Would truncating help or would this cause rounding down?

Probably not.  Depends on what you mean by that.

Certainly rounding and rounding up or down to integers eliminate the binary representation problem, except for integer values larger than 9,007,199,254,740,992 (2^53) and except for division.  But I don't think that is what you mean.

Rounding and rounding up or down to fractional decimal places will not always benefit you.

PS:  Someone might mention the "Precision as displayed" feature (PAD).  First, that will not help with VBA calculations.  Second, it only affects the final result in cells.  It does not help with intermediate calculations.  For example, IF(10.1 - 10 = 0.1, TRUE) still returns FALSE(!).  Finally and most importantly,  PAD has a pervasive effect on the entire workbook.  It can alter constants irreversibly; and it can affect the result other cells unintentionally.  Bottom line:  do not consider using PAD.  IMHO.

joeu2004 say: Clarification....
Sue wrote:
Would truncating help or would this cause rounding down?

I wrote:
Rounding and rounding up or down to fractional decimal places will not always benefit you.

Although that response is technically correct ("will not always benefit you"), rounding or rounding up or down individual terms of an expression will sometimes benefit you.

For example, suppose that A1 displays 123.99 and B1 displays 123.45, but both are unrounded results of calculations.

We might expect ROUND(A1-B1,2) to result in 0.54.  But the result might be 0.55 because the actual values are as much as 123.994999999999 in A1 and as little as 123.445 in B1.

Of course, the ideal solution might be to round each formula in A1 and B1.  But there might be good reasons not.  In that case, the WYSIWYG solution might be ROUND(A1,2)-ROUND(B1,2).

And to avoid those pesky "tiny bit" arithmetic differences:  ROUND(ROUND(A1,2)-ROUND(B1,2),2).
