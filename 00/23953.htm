Formatting Vowels and consonants diff color in a cell - Microsoft Community

staceyv say:

Formatting Vowels and consonants diff color in a cell

separate

 

I am dyslexic and I would like to format vowels and consonants in a cell different colors then apply that formatting to the column. how can that be done.

jeeped say:
You cannot apply formatting like that to a column, but you can apply that formatting to the constant (i.e. typed) value within a cell. Unfortunately, this will not work on the values returned by formulas. The action  could be triggered on an addition or editing change in the values typed into one or more column(s). Due to a lack of specifics offered, I'm going to assume column C.

It took me a while to cover all of the possible scenarios that an event macro will run into but I believe I have a satisfactory resolution for you to try. Right-click the name tab of the worksheet you want to apply the special vowel/consonant formatting and when the VBE opens, paste the following into the pane titled something like Book1 - Sheet1 (Code),

Private Sub Worksheet_Change(ByVal Target As Range)
    If Not Intersect(Target, Columns("C")) Is Nothing Then
        If Not CBool(Application.CountA(Intersect(Target, Columns("C")))) Then Exit Sub
        On Error GoTo ResetNumbers
FormatText:
        If Not Intersect(Target, _
          Columns("C").SpecialCells(xlCellTypeConstants, xlTextValues)) Is Nothing Then
            On Error GoTo FallThrough
            Application.EnableEvents = False
            Dim c As Range, i As Long
            For Each c In Intersect(Target, _
              Columns("C").SpecialCells(xlCellTypeConstants, xlTextValues))
                For i = 1 To Len(c.Value)
                    If CBool(InStr(1, "aeiouy", LCase(Mid(c.Value, i, 1)))) Then
                        c.Characters(Start:=i, Length:=1).Font.FontStyle = "Bold"
                        c.Characters(Start:=i, Length:=1).Font.ColorIndex = 3 'red
                    ElseIf CBool(InStr(1, "bcdfghjklmnpqrstvwxz", LCase(Mid(c.Value, i, 1)))) Then
                        c.Characters(Start:=i, Length:=1).Font.ColorIndex = 5 'blue
                        c.Characters(Start:=i, Length:=1).Font.FontStyle = "Regular"
                    Else
                        c.Characters(Start:=i, Length:=1).Font.ColorIndex = xlAutomatic
                        c.Characters(Start:=i, Length:=1).Font.FontStyle = "Regular"
                    End If
                Next i
            Next c
        End If
ResetNumbers:
        On Error GoTo FallThrough
        If Not Intersect(Target, _
          Columns("C").SpecialCells(xlCellTypeConstants, xlNumbers)) Is Nothing Then
            With Intersect(Target, Columns("C").SpecialCells(xlCellTypeConstants, xlNumbers))
                Application.EnableEvents = False
                .Font.Bold = False
                .Font.ColorIndex = xlAutomatic
            End With
        End If
    End If
FallThrough:
    Application.EnableEvents = True
End Sub

The above is for column C. This will have to be modified to suit your own situation. There are 6 references to column C that would have to be changed. Post back if you have trouble modifying this for your own purposes. I also included y and Y as vowels and you can edit the vowel and consonant strings (noted in bold above) if you prefer those as consonants. 

Tap Alt+Q to return to your worksheet. Anything typed or pasted into column C will receive one of the following processes:

    1) consonants will be formatted in blue
    2) vowels will be formatted as red, bold
    3) numbers will be reverted to non-bold, automatic (e.g. black) color

                                    

I had to use a couple of 'bandaids' to prevent errors on the .SpecialCells([...]) being non-existent. If anyone has some ideas that would make this more efficient, more operable or simply more elegant, I would welcome the input.

I've uploaded that sample workbook to my SkyDrive here for your reference and download.

Looks like they're just beginning their first round of tests ever, so they are apparently quite new to the testing scene.  Though they at least define their methodology and list the specific results for failures, both their methods used and management of the process appear a bit rudimentary at present, since they are questioning their own processes in that same thread.

Another key item to remember is that MSE must do its best to avoid false positive detections at all costs, even if that might delay the detection and removal of some newer malware.  Of course, this must also be balanced with how quickly a new threat is being distributed in the wild and how damaging the payload itself is determined to be.

In the case of MSE, errors in detection are potentially far more dangerous, since the largest portion of its user base will be non-technical users with no real understanding that something might be a false positive.  The general mode of operation for these users is "shoot first and ask questions later, if ever".  This is why you'll notice that OneCare already has one of the lowest false positive rates in the industry based on other tests.

Personally, I've yet to see any testing that really performs anything close to how a user is actually going to experience a malware infection in real life.  The reason is that the setup for this would require not just clients, but also web servers (possibly multiple) and much more comprehensive investigation of what actually remains on the client after an attack.  This process would be almost prohibitively costly in both equipment and time, so much simpler 'static' testing is usually performed, which generally results in flawed assumptions that affect the results for many protection products.

The basic truth with all protection is that nothing will, or even can stop everything.  The best you can hope is that you will always be made aware that an attack has, or is occuring, so that you may take the appropriate action.  In the best case the protection may have already completely dealt with the attack, but it's dangerous to assume this no matter how supposedly good the protection has been rated by anyone's testing.

    Set oSession = Application.Session
    If oSession.ExchangeConnectionMode = olCachedOffline Or _
        oSession.ExchangeConnectionMode = olOffline Then
        ' We are offline, so no can run rules
        MsgBox "Outlook is offline; cannot run rules", vbOKOnly + vbExclamation, "Can't Run Rules"
        GoTo Exit_Rule
    End If

    ruleList = ""   ' Initialize info string
    ' We are online. For each account that is Exchange or iMap, try to run rules
    For acct = 1 To oSession.Accounts.count
        ' iterate all the rules
        Set eachAcct = oSession.Accounts.Item(acct)
        If acct > 1 Then
            ' Blank line between accounts (after the first account)
            ruleList = ruleList & vbCrLf
        End If
        If eachAcct.AccountType = olExchange Or eachAcct.AccountType = olImap Then
            ' There are rules to process
            ruleList = ruleList & "Processing rules for account: " & eachAcct.DisplayName & vbCrLf & "===========================" & vbCrLf
            ' Get the store for this account
            Set st = eachAcct.DeliveryStore

            ' Now iterate through the rules
            Set myRules = st.GetRules
            For count = 1 To myRules.count
                On Error GoTo BadRule_Err
                ' determine if it’s an Inbox rule
                Set rl = myRules.Item(count)
                If rl.RuleType = olRuleReceive And rl.Enabled Then
                    ' if so, run it. See https://msdn.microsoft.com/en-us/library/office/ff864433.aspx for args
                    ' which are supplied as (shows defaults):
                    ' rl.Execute ShowProgress:=False, Folder:=inbox, IncludeSubfolders:=False,
                    '                          RuleExecuteOption:=OlRuleExecuteOption.olRuleExecuteAllMessages
                    rl.Execute RuleExecuteOption:=OlRuleExecuteOption.olRuleExecuteAllMessages
                    ruleList = ruleList & rl.Name & vbCrLf
                ElseIf rl.RuleType = olRuleReceive Then
                    ruleList = ruleList & "[DISABLED - not run] " & rl.Name & vbCrLf
                End If  ' some sort of OlRuleSend rule...
                GoTo Next_Rule

BadRule_Err:
                ' Something went sideways processing this rule
                ruleList = ruleList & "ERROR processing rule Item # " & count & ", Error Info: " & Err.Number & vbCrLf
                Resume Next_Rule
Next_Rule:
            Next ' rule
        ' END of IF block on account type
        Else
            ' Account type doesn't support rules
            ruleList = ruleList & "NO rules for account: " & eachAcct.DisplayName & vbCrLf & "===========================" & vbCrLf
        End If

    Next ' account

    ' tell the user what you did (uncomment next 2 lines to show)
    ' ruleList = "These rules were executed against the Inbox: " & vbCrLf & ruleList
    ' MsgBox ruleList, vbInformation, "Macro: RunAllInboxRules"
Exit_Rule:
    ' Free up memory
    Set rl = Nothing
    Set st = Nothing
    Set eachAcct = Nothing
    Set myRules = Nothing
    Set oSession = Nothing
End Sub
