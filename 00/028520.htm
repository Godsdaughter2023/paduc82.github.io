Well, Excel is an excellent spreadsheet. It is NOT a relational database, and it's not designed to be.

scottgem say:
First, you don't export to ODBC. ODBC is a technology that allows programs to read data from ODBC compliant programs. If the tracking system is ODBC compliant, then it is either using a back end database that you may already have a driver for, or it will provide you one to install on your system. 

From there you can then create an ODBC DSN or use a DSN less connection. I would start with a DSN to test it. Once the DSN is created, you link to their tables by going into Access, select the External Data ribbon and select ODBC. You will be prompted to select your DSN and then you will see a list of tables you can can link to. 

In my opinion this would be the best way as you have no interim steps. 

You can then create Access tables with the CtlRefNum as a foreign Key. Users should then be able to add data to the Access tables, first selecting the CtlRefNum.


john w. vinson say:

As Scott says, that's not how ODBC works. Think of ODBC as a very fancy data-pipe; with a hardware store full of different connectors on each end: you can hook up the ends of the pipe to Excel, or Access, or SQL/Server, or DB2 or any of dozens of other data sources (including your tracking system). Since it's so flexible it's correspondingly complicated to use. But it is basically logical; you should be able to create an ODBC connection - called a "DSN" - to connect your system to your Access database. Once you've done so you should be able to create an ODBC link in Access which works very much like a Table; it might or might not be updateable depending on the system's settings, but you should at least be able to read the data as if it were in an Access table. There might be fields you don't need; if so, just don't include them in your ODBC linked queries!
