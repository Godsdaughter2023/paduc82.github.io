jay freedman say:

You really have to be careful with "cleaning" a PC, because it's too easy to remove things that shouldn't be removed. It should be an absolute requirement that you take a complete backup before you ever run any kind of cleaning program.

In this case, the master citation list should be in a file named Sources.xml. To get to the hidden folder where it should be stored, click in the address box in File Explorer, type %appdata%\Microsoft\Bibliography, and press Enter.

If the file isn't there, try searching the drive for the filename Sources.xml.

If you can't find the file at all, or if it's there but empty, then your only hope is to have a backup from before the cleaning.

For the future, (a) set up the Windows File History feature (https://support.microsoft.com/en-us/help/17143/windows-10-back-up-your-files) and (b) create an off-site backup routine, such as to OneDrive or DropBox, or to an external hard drive.

macropod say:

All is not lost! Your references will still be in whatever documents you attached them to, so they can be copied from there back to Word's master list. That process can even be automated with a macro. The following macro will retrieve all sources referenced in the documents in a selected folder.

Sub RecoverSources()
Application.ScreenUpdating = False
Dim strFolder As String, strFile As String, strDocNm As String, wdDoc As Document, Src As Source
strDocNm = ActiveDocument.FullName
strFolder = GetFolder
If strFolder = "" Then Exit Sub
strFile = Dir(strFolder & "\*.doc", vbNormal)
While strFile <> ""
  If strFolder & "\" & strFile <> strDocNm Then
    Set wdDoc = Documents.Open(FileName:=strFolder & "\" & strFile, AddToRecentFiles:=False, Visible:=False)
    With wdDoc
      For Each Src In .Bibliography.Sources
        On Error Resume Next
        Application.Bibliography.Sources.Add Src.XML
        On Error GoTo 0
      Next
      .Close SaveChanges:=False
    End With
  End If
  strFile = Dir()
Wend
Set wdDoc = Nothing
Application.ScreenUpdating = True
End Sub

Function GetFolder() As String
Dim oFolder As Object
GetFolder = ""
Set oFolder = CreateObject("Shell.Application").BrowseForFolder(0, "Choose a folder", 0)
If (Not oFolder Is Nothing) Then GetFolder = oFolder.Items.Item.Path
Set oFolder = Nothing
End Function

The macro includes a folder browser, so all you need do is select the folder to process.

For PC macro installation & usage instructions, see: http://www.gmayor.com/installing_macro.htm

hoi boinst say:

At the outset, please understand that I’m no computer “geek,” but I did manage to find something that worked for me when I was faced with this problem of not being able to gain access to my Outlook.com email account, getting the rather vague message “outlook.live.com didn’t accept your login certificate, or one may not have been provided.”  So, I thought, well, maybe there’s a problem with my certificate and, if so, how do I fix this?

For those in a hurry, here’s the bottom line first: On my Windows 10 desktop computer, I discovered that my anti-malware program had mysteriously started to block Microsoft certificates, which prevented my computer from accessing my Microsoft email account.  I changed the settings in the anti-malware program from “block” to “allow” Microsoft certificates to work and, in so doing, I was able once again to restore access to my email account.

Now, the details.  Because I use ESET 9 as my computer anti-malware software, my solution to this problem applies specifically to ESET, but in case it might possibly apply in a general sense to other programs as well, I offer it here.

Since ESET monitors all internet traffic on my computer – and that includes certificates – I thought that maybe ESET had something to do with this issue, so I opened ESET and tried an experiment to unblock certificates that might have mysteriously gotten blocked, and I’m pleased to say that I managed to change the “block or allow” settings for certificates that are managed by ESET.  I discovered that several certificates had indeed somehow become blocked (!!), and so I changed specific certificates from being blocked by ESET to allowing ESET to automatically choose if something should be blocked or not.  (You can also force ESET to either block or allow a certificate, but, using the “AUTO” option that comes up in the choice list, I chose the option of letting ESET – in its wisdom—to automatically decide to block or allow a certificate instead of me forcing it to do something.  (And of course, one can always try the brute-force “unblock” choice if the automatic “AUTO” choice doesn’t work, but this may not be the wisest thing to do from a security point of view.)

It’s a little involved to navigate to the window in ESET to make these changes to blocking and allowing certificates, but here’s the sequence.

Open ESET.  (Duh.)  You’re probably on the “Home” window.

Click on “Setup.”  Four green-fonted choices appear on the right side of the window.

Click on “Internet protection.”  (Four more choices appear on a new page.)

Click on the “gear wheel” icon at the extreme right side of “Web access protection.”  An “Advanced setup” page opens, listing many items in a column on the left side.

Click on the blue-fonted WEB AND EMAIL heading.  A column of selections appears on the righthand side that shows two headings:  “PROTOCOL FILTERING” and “SSL/TLS.”

Click on the plus/minus symbol of the SSL/TLS heading; a dropdown list appears, showing a lot of selections to click on, including the entry “List of known certificates.”

Click on the blue-fonted “Edit” link to the right of the “List of known certificates.”

On my computer, a list of certificates appeared, including six certificates related to Microsoft and, for reasons unknown, they were all marked “Block.”

Perhaps this is foolhardy, but since I couldn’t gain access to my Microsoft Outlook/Email account, I decided to unblock anything related to Microsoft.  To do this, I clicked on a Microsoft Certificate issuer entry to select it and edit it.  When I clicked on an entry, the grayed-out “Edit” button became active and so I clicked on the Edit button for that particular highlighted selection.  A new window opened called “Edit certificate.”  Under “Access action,” I found that “Block” was already selected, so I changed that by clicking on “Auto” and then clicking on “OK.”  The window closed and I was back to the “List of Known certificates” window.  The entry had now been changed from “Block” to “Auto.”

I did this for each of the six certificate issuers mentioned above.  After these edits, all six of these issuers were now listed under “Access” as “Auto” -- not “Block”! -- and I clicked “OK.”

This returned me to the “Advanced setup” screen.  {Note that I didn’t do any other changes to the remaining choices under the SSL/TLS heading.)  I clicked on the “OK” button and when I did this, the standard Windows alert message came up, asking me if I wanted to allow this app (“ESET Elevated Client”) to make changes to my computer.  I clicked “Yes” and the changes were made and I was returned to the “Internet Protection” window.

I closed ESET and then directly tried to access my email in my Outlook account through Chrome.  (Note that I didn’t close Chrome first, nor did I reboot the computer.)

I’m pleased to say that I was finally able to regain access to my email account!  (Note:  I typically use Chrome as my browser but I also tried logging into my email account with Microsoft Edge and it worked fine.  Also note that I did all this with Chrome open; I didn’t have to close Chrome and reopen it to make it work – which would have been my next attempt.)

Anyway, I hope that this information might be helpful to someone.

In any case, good luck!

It should be in the SecureTemp folder or the Temp Internet files. Empty your internet cache (IE's Optios). Paste the following line in the address bar of windows explorer then press enter. Securetemp is a folder under this path -  delte the folders and restart outlook.

%USERPROFILE%\Local Settings\Temporary Internet Files\Content.Outlook
