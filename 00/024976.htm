ken sheridan say: As Peter has pointed out, you have (at least) one table too many.  The question is whether you should need to drop one table or two.  This depends on what is the nature of the entity type which the Items table models:

1.  I suspect, on the basis of my own experience as a professional museum curator, that the relationship type between donors and items is one-to-many.  This would be the case if each row in items represents a single physical entity. e.g. taking an example from my own experience, the iron age torque found at Clascote, Staffordshire, rather than iron age torque as a class of artefact.  The model for this would comprise, in broad outline, the following tables:

Donors
….DonorID  (PK)
….FirstName
….LastName
….AddressLine1
….AddressLine2
….etc

Items
….AccessionID  (PK)
….AccessionDate
….ItemDescription
….DonorID  (FK)
….SubcategoryID  (FK)
….etc

ItemSubcaterories
….SubcategoryID  (PK)
….Subcategory
….CategoryID  (FK)

ItemCategories
….CategoryID  (PK)
….Category

2.  Where the Items table models a class of artefact rather than a specific physical artefact then the relationship type between Donors and Items is many-to-many, and modelled like this:

Donors
….DonorID  (PK)
….FirstName
….LastName
….AddressLine1
….AddressLine2
….etc

Items
….ItemD  (PK)
….ItemDescription
….SubcategoryID  (FK)
….etc

Accessions
….AccessionID  (PK)
….DonorID  (FK)
….ItemD  (FK)
….AccessionDate
….etc

ItemSubcaterories
….SubcategoryID  (FK)
….SubCategory
….CategoryID  (FK)

ItemCategories
….CategoryID  (PK)

Note that in both scenarios the Items table does not include a CategoryID foreign key column.  CategoryID is functionally determined by SubcategoryID, so, if included in Items, would be transitively dependant on the key, and the table would consequently not be normalized to Third Normal Form (3NF) and open to the risk of update anomalies.

Whichever model is correct in your case the interface would be the same, a donors table based on Donors and a subform, linked to the parent form on DonorID, for entering accessions.  In scenario 1 the subform would be based on Items; in scenario 2 it would be based on a query which joins Accessions and Items, but note that in the subform data is entered only into the columns from Accessions.  The controls bound to the whatever columns from Items you wish to show in the subform should be made read only by setting their Locked property to True (Yes) and their Enabled property to False (No).  I suspect, however, that the simpler scenario 1 is the correct one.

You will probably have to cater for situations where an item is accessioned other than as a donation, e.g. purchased at auction.  To cater for this you should include a dummy row in Donors with 'N/A' as the value in all columns bar DonorID.  In a fully developed Accessions database other tables  would be included to cater for other methods of accession, referencing Items or Accessions by means of an AccessionID foreign key.

You'll have noted that none of the object names in the above include spaces or other special characters.  Including these in object names is not considered good practice.  I know that you'll find it done in Microsoft templates and sample files, but these unfortunately include many points of bad practice.  You'll also have noticed that I use CamelCase for column names, but if you prefer to use all lower case that's fine.  Spaces can then be represented by the underscore character like_this.

ken sheridan say:

2nd row of Accession subform has a different accession ID.  I want it to be the same until a new accession is made for the same donor.  Do you know how I can accomplish that?

In my museum experience it is more usual for each item to be given a distinct accession number, this being the principal identifier of each object in the collection, whereas you appear to want each set of items given by a donor at any one time to have the same accession number, i.e. you are equating an accession with a donation event.  For that the model would be

Donors---<Accessions----<AccessionItems>----ItemSubcategories>----ItemCategories

Note the symmetry of the model.  To represent this in a donors form would require two subforms. Accessions and AccessionItems.  You have two options for the design:

1.  Nested subforms in which the Accessions subform, linked to the parent form on DonorID, is in single form view, and the AccessionItems subform is in continuous forms view.  This is easy to set up as it is largely code free, apart from any code you might put in the unbound Category combo box's, and the bound Subcategory combo box's NotInList event procedures to allow you to transparently insert rows into the ItemCategories and ItemSubcategories tables by typing the new values into the controls.  This, while desirable in a developed interface, is not essential to the main functionality of the subforms.

2.  Correlated subforms in which both are embedded directly in the parent form, and both are in continuous forms view.  The two subforms are correlated by basing the AccessionItems subform on a query which references the primary key of the Accessions subform's recordset as a parameter.  This is the better interface in terms of the presentation of comprehensive data about each donor's donations in a single view, but does require a small amount of code to achieve the correlation, firstly to Requery the second subform in the Current event procedure of the first subform, and secondly to assign a value to the foreign key column in the BeforeInsert event procedure of the second subform.

You'll find examples of both nested and correlated subforms, using Northwind data, in CorrelatedSubs.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

    julie sheets say:
    Hello deathshead75,

    When you create the link between the sharer files and the pool, there is an option "On conflict with calendar or resource information" and options for "Pool takes precedence" or "Sharer takes precedence."  The default is "Pool takes precedence."  As you are making changes to resource information in the sharer file it keeps getting over written by the pool when you save.

    The challenge is if you set every share to "sharer takers precedence" you may also fun into conflicts between changes in each sharer file.  My recommendation is to stay with the pool taking precedence and then making all changes in the pool file.

    I hope this helps.

    Transferring Word 2003 templates, with their macros - Microsoft Community

    beeorchid say:

    Transferring Word 2003 templates, with their macros

    Word 2003:  I have a number of templates with macros stored in the /Templates folder.  they represent years of development.

    Word 2010:  I want to use these templates as Personal Templates in Word 2010, accessed by pressing the orange "My Templates" button.

    Questions:  

    (1)  Where does Word 2010 store its personal templates?

    (2)  If I open my old templates and save to this folder as *.dotm, will that work, and will the macros still work?

    jay freedman say:

    (1) The default location for templates in Word 2010 is the same as that for Word 2003, %appdata%\Microsoft\Templates. (The Windows environment variable %appdata%, when used in a file path, expands to C:\Users\<your login name>\AppData\Roaming on Windows 7.) It can be changed within Word to any other location, but it's unlikely you've done that if you're asking about it here.

    (2) Yes, resaving a .dot template as a macro-enabled .dotm template will work, and it will preserve the macro code. Whether the macros will still work depends entirely on what code is in them.

    Most macros from 2003 will work as is, but others will need anything from a small tweak to a major rewrite. Typically, if a macro contains statements that deal with opening and saving .doc files, they'll have to be updated to work with .docx (or possibly .docm) extensions. If you've relied on naming macros with the names of built-in commands to intercept the user's clicks or shortcuts, some of those will no longer do the interception -- particularly FilePrint. Macros that deal with AutoText will continue to work, but AutoText is now just one type of "building block," and you might want to revise the macros to use the full feature. There are of course many other new features that you might want to include in new macros.

    A gratuitous tip to save a few clicks: Add the "New Document or Template" button to the Quick Access Toolbar. (It's in the All Commands list in the Options > Quick Access Toolbar dialog.) That will open the same File > New dialog that you had in Word 2003, which is much more efficient than the miserable thing perpetrated in Word 2010 (and made worse in Word 2013).

    grschinon say:
    samflynn wrote:
    thanx but i want to set the phone to save all music to memory card, for exmaple when it is bluetoothed to me

    Files that arrive via bluetooth are handled as messages and therefore end up in your messaging inbox.

    What you have to do is open the message containing the file you want to store. If it's a music file then the music player will start playing it. Press the "stop" button, exit the music player, and you will be told that the piece of music was not saved. Confirm that you do want to save it and then navigate to the location on the memory card where you want to save it.

    Don't forget to delete the "message" from your inbox when you're done as it will take up a lot of space in your phone's C: drive.

    If you want the music to go directly to the memory card without going through the convoluted steps described above then bluetooth is not the method you need to use. It's also very slow for that matter.

    The N95 comes with a miniUSB cable, you should use that. And even if you've lost yours, just about any standard miniUSB cable will do. When you plug the phone in, you will be asked what mode you want the phone to work in. Select "mass storage". Your phone's memory card will then appear under "My Computer" (assuming you're using Windows) as a removable drive. You can then simply drag'n'drop or copy/paste files directly onto it.

    Alternatively, you can invest in a small USB card reader. Remove the card from the phone, put it in the card reader, plug the card reader into a free USB port and the card will appear as above, as a removeable drive onto which you can drag'n'drop or copy/paste files, but it'll be significantly faster than using the phone's USB connection.
