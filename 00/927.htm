ken sheridan say: Perhaps I can illustrate this best by describing a similar situation which I encountered some years ago.  The database in question was one which related to my own area of work as an environmental planning officer and included a Sources table relating to technical publications.  The table included an Author column in which I found three versions of my own name, one correct, two incorrect, along with other incorrect entries for other authors.  When I transferred the database from dBASE to MS Access  the steps with regard to this were:

1.  Create a Sources table in Access with exactly the same data as in the original dBASE file.

2.  Add an AuthorID column of long integer data type to the Sources table.

3.  Create an Authors table with an autonumber AuthorID column as its primary key.  Note that personal names cannot be used as keys as they can legitimately be duplicated.  I know of at least one other author named Ken Sheridan.

4.  Join the Authors and Sources tables on the author name column or columns in an update query and update the AuthorID foreign key column in Sources to the value of the AuthorID primary key column in Authors.

5.  Delete the author name column or columns from sources.

6.  Create an enforced relationship between Authors and Sources on AuthorID

7.  In form bound to the Sources table include a combo box bound to the AuthorID column, set up in the usual way to show the referenced names column or columns from Authors.  Manually go through each record in this form and in the combo box select the correctly spelled name of each author.

8.  In the Authors table delete all rows with incorrectly spelled names.  In our case this was easily done with a simple query:

DELETE *
FROM Authors
WHERE NOT EXISTS
    (SELECT *
     FROM Sources
     WHERE Sources.AuthorID = Authors.AuthorID);

This was possible because the Authors table was only intended to contain authors referenced in sources.  In an Employees table or similar it is unlikely to be possible however, as there will almost certainly be employees with no matching rows in SalesHistory or similar, whom you would not wish to delete, in which case the deletions would need to be done manually.

If your database is correctly structured you'll probably already have an Employees table or similar from which your combo box's RowSource property derives its list, so steps 1 to 6 would be unnecessary.  It would merely be a case of correcting the incorrect records in SalesHistory or similar, and then deleting the redundant rows from the referenced Employees table or similar.

Let me make sure we understand each other. A combobox is what you are referring to as a drop down. The actual name of the control type is combobox. Controls are the things that are on a form that allow you to interact with the form. Controls are things like textboxes, comboboxes, checkboxes, buttons, etc. Every control has properties and you control what Access can do by manipulating those properties. A combobox has a RowSource property that controls what the combobox displays. It also has a Bound column property that determines what value in the RowSource is stored in the table. 

In a properly designed database, you would have a table of people and that table would be the RowSource of the combo. Whenever you chose a name from the combo it would store a code in the table that relates back to the people table.
