ken sheridan say: What is your underlying model?  You have a many-to-many relationship type between Movies and Genres, so this should be modelled by a table which resolves the relationship type into two one-to-many relationship types.  In broad outline the tables would be:

Movies
…MovieID  (PK)
…Title
…etc

Genres
….GenreID  (PK)
….Genre

and to model the relationship type:

MovieGenres
…MovieID  (FK)
….GenreID  FK)

The primary key of the last table is a composite one made up of the two foreign keys.  The interface would be a Movies form within which would be a MovieGenres subform, the former in single form view, the latter in continuous forms view.

With this normalized setup I don't know that there is any way you can filter on multiple genres using the built in filtering functionality.  You'll find examples of filtering on multiple values by means of a multi-select list box in MultiSelect.zip in my public databases folder at:

Every Windows system which installed Microsoft Office products certainly got the C:\MSOCache folder, if the user doesn't like its position which is putted directly under the C:\ root folder, could he or she move it to other place?
Yes, absolutely yes.


For example, want to move it from "C:\MSOCache" to "C:\Windows\MSOCache", just in the registry replace all the "C:\MSOCache" string with the "C:\Windows\MSOCache" string, and then move it into the system root folder, that's all right.


For ease of use when installing or reinstalling Microsoft Office products, can make a .reg file which includes all the settings of the replacement. After installing or reinstalling Microsoft Office products, just can double-click the .reg file to import the settings into the registry.
