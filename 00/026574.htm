CountIF inequalities inconsistent with normal inequalities - Microsoft Community

chase r t say:

CountIF inequalities inconsistent with normal inequalities

Why does CountIF inequalities treat strings and numbers differently than a typical if inequality or inequality?  For example if I have N/A in cell and I check if this is greater than a value, say 1, it will return true, however if I try to count over that cell looking for values that are greater than 1, it no longer includes it as the true condition.  What's causing this?  Did this change or am I mis-remembering something?

joeu2004 say:

It would help if you provided specific formulas and data. But I think this is what you are saying....

Suppose you have text "N/A" in A1, not the Excel error #N/A.

Then yes, =IF(A1>1,TRUE) returns TRUE  because all text values are considered greater than all numeric values when using a comparison operator in a conditional expression.

But =COUNTIF(A1:A1,">1") returns zero because COUNTIF recognizes 1 as numeric, and it compares only with numeric values in the range (A1:A1).

If you write =COUNT(A1:A1,">1*"), COUNTIF treats "1*" as text. But then it compares only with text values in the range.

For compatibility with IF() results, use:

=SUMPRODUCT(--(A1:A1>1))

surfinbirdone say:

Office 2010 license deactivation

I have Office 2010 Pro Plus and I maxed out my licenses. The problem is that 2 weeks ago I used my last license on a brand new computer that I had to get exchanged due to defects. Now I can't install Office in my replacement PC. Can I deactivate my last installation so that instead of wasting that license I can use it in the replacement PC?
brian tillman [outlook mvp 2007-2019] say:

Office itself doesn't keep track.  The Microsoft Activation Center does.  Office gathers information about the PC on which it is being activated.  That information is sent to the Activation Center.  If the data match (meaning it was activated on the same system in the past), the activation is allowed.  If the number of activations is lower that the max, it is allowed.  If the number of activations is at the limit and the data doesn't match, the activation fails.  The hole in this is that the Activation Center isn't notified when you uninstall, so if you're at the max and you deinstall it on one PC and install it on another, the Activation Center must be contacted by phone so you can explain the discrepancy.

dan at it associates say:

I gues I'm assuming Office is smart enough to know how many activations are in effect under the supplied reg key and knows when to quit activating new installations.


Microsoft does build a degree of "wiggle room" into the product activation system under the theory of granting the benefit of the doubt to the consumer.  If there was a lapse of time of more than about 120 days between the 2nd and 3rd activations, that's probably the wiggle room kicking and and why that third activation went thru.

Cascade a delete row

Is there any way I can cascade a delete command?  For instance I have 8 worksheets, the master work sheet (no 1) has the same number of rows (or the same row number) and I want to delete row 6.  Can all the other row 6's on the other sheets be automatically deleted as well?
mike h.. say:

Hi,

 

Right click the sheet tab where you want to initiate this, view code and paste this code in.

 

EDIT..... I should have added the code only executes when you select a single entire row.

 

Private Sub Worksheet_SelectionChange(ByVal Target As Range)
If Target.Address = Target.EntireRow.Address Then
Dim response As String
Dim x As Long, r As Long
    If Target.Rows.Count > 1 Then
        Exit Sub
    End If
    response = MsgBox("Are you sure you want to delete" & vbNewLine & _
    "Row " & Target.Row & " on every sheet?", vbOKCancel)
        If response = vbCancel Then Exit Sub
        r = Target.Row
        For x = 1 To Worksheets.Count
            Sheets(x).Rows(r).EntireRow.Delete
        Next
End If
End Sub

kosh vorlon say:

Hi Sharon,

 

ZigZag provided excellent procedures for how to do it - but let's be clear about what it does and doesn't do after being enabled (and note it isn't outlook.com doing it, but by running outlook.com through Windows Live Mail - a separate email-program that isn't browser-based which is what's doing it).

 

It will prompt the recipient to send you a notification that the email has been received when they open the email you sent with that feature included.  If they say yes, you'll be notified when they send the notification of receipt of the email back to you  But they can say no and still be able to do anything else they want with the email.  It cannot be done secretly or "behind the curtain" without knowledge of the recipient and canot be enforced (even with their knowledge) without 3rd-party add-on software (which typically adds far more than just this one thing and I don't think can be done only from your side but requires them to make some changes too) - at least any I know about.

 

In essence, it works if the recipients let it work and it doesn't work if they choose not to bother or choose not to do so.  There's really not much you can do about enforcing their cooperation - unless you are truly in a position of obtrusively, overtly, (perhaps arrogantly) controlling those who receive your emails (and have enough leverage and use free or provide compatible products that work on the devices (PCs, Macs, Linux, mobile phones,....) in various environments (stand-alone recipients, networks, domain networks, secured or classified-secured systems, foreign languages,...) of any recipeit you may encounter and want to do this.

 

If you want to do that, you need a 3rd party security add-on which not only blocks reading the email until a decryption key is input but also *forces* the recipient to send a receipt before even opening your email.  Then you need policies concerning the forwarding or synchronizing of these emails (are multiple confirmations required, is forwarding permitted,...).

 

If your intent is to verify they received and were able to read the email and/or attachments, this is somewhat defeated because they must acknowledge before even they know if it arrived properly and can be decrypted (and read along with any attachments.  So it may help confirm someone received something, but not that it could be fully and completely read.  An example might be attaching a OneNote 2013 formatted file and sending it to someone who cannot install or read higher than OneNote 2010 - you have an acknowledged receipt but the intended communication did not take place.

 

If your intent is to simply and unabtrusively confirm your emails are not hitting SPAM or Junk Filters (or being ignored) = and not that they were readable when opened, then this too would be defeated as this will work only if they do something complementary on their side to *force* their system to submit confirmation when they get such emails (and unless there's some mutually-acknowledged legal or security reason, most average recipients are going to trash such emails - or at best be suspicious of them - especially the first time if they need to install software and retrieve public decryption keys).

 

So before charging down this path, may I suggest that you be clear on your objectives, the degree of intrusiveness and compatibillity you want, the costs you want (or don't want), the consequences of such a process (on you and on the recipients of your emails), how much is "good enough", and so forth.   Then, unless you want to go beyond what I mentioned in my second paragraph, that is pretty much as much as you'll be able to do here or with any commonly-included service with any email hosting provider and almost all separate email programs without 3rd party add-on features.

 

I hope this helps.

 

Good luck!

 

Kosh
