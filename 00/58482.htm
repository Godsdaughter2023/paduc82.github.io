wunders say:

You appear to have problems authenticating to the other machines when logged in locally to one of them ("this computer").  Assuming that you are logged in locally as user: "DanLocal" with password "DanLocalPW", then, IIRC, authentication will work something like this:

  1. If target computer has local user "DanLocal" with password "DanLocalPW", then you should automatically connect and authenticate to the target computer as user "DanLocal". and be able to access all resources accessible to local user "DanLocal".
  2. Else If target computer has local user: "DanLocal" with a different password than above, then you will be denied access to that computer.
  3. Else If target computer has the "Guest" user enabled with no password, then you will automatically authenticate as the user: "Guest".
  4. Else a login window will pop-up on your screen asking for a username / password to authenticate as.

Since you do not get a pop-up window asking for your credentials, I am guessing that you end up authenticating as the "Guest" user and that the share that you are trying to access (/share) does not have permissions for the user "Guest".  When you are logged in to the computer with your domain credentials, then your domain credentials will get you access to the other computer with the permissions associated with your domain credentials.

 

So what you appear to want to do is to connect to one of the other machines using a different set of credentials than the ones that you are currently using when logged in locally to one of the machines.  So if I am correct above in that you end up authenticating as Guest user, one solution would be to disable the "Guest" account on the other computers.  Then a pop-up window should appear and you can specify another set of credentials to attach to the other machine (use "domainname\user" if you want to use your domain credentials)  One way to disable the Guest account is to go to the other machine(s), log in as an administrator user, bring up a command prompt, (Start -> Run -> "cmd") and enter the command:

    net   user  guest  /active:no

 

Another solution is to connect to the other machine specifying your credentials.  Personally, I always do this by using the command line.  Bring up a command window as above and enter the command:

    net  use   \\computername\share  password  /user:domainname\username

where you replace "computername", "share", "password", "domainname", and "username" appropriately. Take care with slashes (/) vs back-slashes (\).  If you want to use a local username and password to the remote machine, substitute the "domainname" with the computer name.  If you leave out the password, it will prompt for it.  This will authenticate you to the remote machine as you specify.  Once connected, you should have little problems accessing it.  A bonus is that errors received from the command line are more descriptive that those from the GUI.

 

Hope this Helps,

Assuming all these computers are on the same subnet (are they?), a "Master Browser" is elected to maintain the list of computers in the same Workgroup or Domain on a given subnet.  When you ask for a list of computers, the list comes from the Master Browser machine.  The master browser may be identified by bringing up a command prompt window and entering the command:

   nbtstat -a computername

or

   nbtstat -A computerIPAddress

where "computername" is the computer name of any of your 22 machines without the "\\".  The machine that is the Master Browser will return a line with "__MSBROWSE__" in it.  This master browser should return all of the machines in the same group as itself as identified by the name in the "<1E>" line.  A computer not in this list might be having firewall issues or is in a different group.  If more than one computer on the subnet identifies itself as Browse Master that have identical Group names, then a firewall is likely the cause.

  Win7: Double-Click "Computer" - "Map Network Drive"(top) -> then check box "Connect using different credentials"

a. user say:

That sounds like a hardware problem. I'll explain:

When a computer first starts up, long before Windows loads, the BIOS or EFI (whichever your computer has) performs a test to check if the hardware is working properly. If it's not, the computer will make a series of beeps. The number and type of beeps tells you what the problem is. These are called beep codes - really.

Every computer manufacturer has their own beep codes. You can probably look up the beep codes for your computer on the internet, but it's easier to just call the manufacturer and ask them what 5 beeps means for you. Hopefully it's something you can easily fix.
