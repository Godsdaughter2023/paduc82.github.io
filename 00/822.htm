There still is a report menu. You don't say which older Access, but I think you are talking about the Database Window from 2003 or older. This was replaced by the Nav Pane in 2007 and later. 

A report is generally bound to a query. There are two ways to use the same report to display different sets of data. One way is to filter the query, the other to filter the report. 

Within those 2 methodologies, there are multiple ways to set a filter. 

One way is to use a parameter query. A better way is to create a form where you enter criteria (for example a class) that filters the the query. Or you can use the form to create a WHERE clause that can be passed to the Report when you open it.

ken sheridan say: The best way to achieve a form layout like that would be to have a parent form, in single form view, based on a table structured along these lines:

PatientSurveys
....Pat_ID  (FK)
....RoundNumber
....SurveyDt
....EnteredBy
....DateEntered

And a  subform, in continuous forms view, based on a table structured along these lines:

PatientSurveyResponses
....Pat_ID  (FK)
....RoundNumber (FK)
....SurveyID (FK)
....QuestionID  (FK)
....Answer

The primary key of this table would be a composite one of the first four columns.  Of these Pat_ID and RoundNumber would be a composite foreign referencing what you say would be the composite primary key of PatientSurveys, and SurveyID and QuestionID would be a composite foreign key referencing the composite primary key of the following table:

SurveyQuestions
....SurveyID (FK)
....QuestionID  (FK)

The other referenced tables in the model would be Surveys, Questions and Patients.

When a row is inserted into PatientSurveys in the parent form rows would automatically be inserted into PatientSurveyResponses by executing an 'append' query in the parent form's AfterInsert event procedure, followed by requerying the subform to show the newly inserted rows ready for the answers to be entered.

What you don't do is have separate tables to represent the three surveys, or encode data as column headings by having columns Q1, Q2, Q3 etc.  As a design for a relational database that would be about as bad as it could get.  A fundamental principle of the spreadsheet masquerading as a relational database table.  A correctly designed relational database table on the other hand models an entity type, with each column representing an attribute of the entity type.

To achieve the output in a horizontal format with columns per question the best way to do this would be as a report based on a query joining PatientSurveys and PatientSurveyResponses, which returns distinct rows of all columns from PatientSurveys and SurveyID from PatientSurveyResponses.  Within this embed a multi-column subreport based on PatientSurveyResponses with across-then-down column layout.  Group the parent report by SurveyID and put the subreport in the group header, linked to the parent report on Pat_ID, RoundNumber and SurveyID.

scottgem say:

A survey database has a very specific structure and you are not using it. Using fields to stand for each question is not correct or normalized.

The proper structure is:

tblQuestions

QuestionID (Primary Key Autonumber)

Question

GroupID (Foreign Key)

tblRespondent

RespondentID (PK Autonumber)

FiratName

LastName

Other info about respondent (your patient)

tblResponse

ResponseID (PK Autonumber)

RespondentID (FK)

ResponseDate

tblAnswers

AnswerID (PK Autonumber)

ResponseID (FK)

QuestionID (FK)

Answer

The general data entry structure is a main form bound to tblResponse where the user selects the respondent etc. and a subform with a record for each question and its answer.

1.  No, this is a foreign key referencing the primary key of Patients (see 4 below).  You yourself said at the start that the primary key would be a composite one of PatientID and RoundNumber, and I've worked on this basis, i.e. the values in these two columns, in combination, be distinct within the table.  PatientID in this case would not be an autonumber, however.

2.  Again no, for the same reasons.

3.  You can have table exactly like that provided that each question can apply to one survey only, in which case you would not need a separate Questions table.  In fact you could rename this table Questions and QuestionID would be its primary key.  If on the other hand a question could apply to more than one of the surveys you would omit the Question column from the table and have a separate Questions table:

Questions
....QuestionID
....Question

SurveyQuestions would be modelling a many-to-many relationship type between Surveys and Questions by resolving it into two one-to-many relationship types, which is how a many-to-many relationship type is always modelled.

4.  Yes, you would definitely need Patients and Surveys tables like this:

Patients
....PatientID  (PK)
....FirstName
....LastName
....etc

Surveys
....SurveyID  (PK)
....SurveyTitle

Whether you also need a separate Questions table is addressed in 1 above.

Looking at the questions you've given as examples I really do think you should have an Answers table, with a distinct set of answers:

Answers
....AnswerID (PK)
....Answer

and a QuestionAnswers table to model its many-to-many relationship type with Qusstions:

QuestionAnswers
....QuestionID  (PK)
....AnswerID (FK)

Simply allowing freeform answers does not make a lot of sense as it makes meaningful analysis virtually impossible.  Pain levels are normally rated on a scale from 1 to 10 in my experience so for your question What is your pain level?, whose QuestionID is 1 for instance, there would be 10 rows in QuestionAnswers

1    1
1    2
and so on to
1    10

I should emphasise that the model I've outlined should not be regarded as definitive.  I'm working on the basis of what you've told us, but you know what you are trying to achieve in real world terms better than we can, so you might find it's necessary to fine-tune the model when you test it, which you must do thoroughly of course before putting it into production.

Finally, you will find an example of an application for building questionnaire databases as Questionnaire.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This would allow you to build a database which would do what you are attempting, though the interface would differ a little from what you have described.  It allows for questions which have a single answer or allow multi-select answers.  It is illustrated with a very simple customer satisfaction survey, so you can try it out by answering the questionnaire.  New questionnaires can easily be created.  In the Zip file there is also a FreeFormAnswers database which includes alternative objects which can be used to replace the relevant ones in the main Questionnaire database to allow completely freeform answers, but these are rarely appropriate.  They produce a record of people's views, but nothing capable of real analysis.
