rohn007 say:
The builders of Windows, back in the 80's made a conscious decision. Windows Apps would no LONGER store printer information in their data files, like documents and spreadsheets. Windows would define and control "default" printers and printer settings.  This was a regression from what was done in DOS apps. At that time I was using WordPerfect. It stored all of the printer settings in the document. You opened the document and it knew which printer to use and what the required settings were. This was because DOS assumed that files would only rarely be moved to different computers. The printers were plugged directly in to the computer. Windows assumed that computers would be networked and could be printed from any computer on the network to any printer. So it was safer to keep the printer settings with the computer rather than the file...

So, the only way I know of "saving" these settings is using a macro.  Many of the printer settings are specific to the printer driver. The easiest way to get the syntax right is to record a macro, then edit it if you have to. 

Here is an example of the recorder output on my computer:
Sub printertest()
'
' printertest Macro
'
'
    Application.PrintOut FileName:="", Range:=wdPrintAllDocument, Item:= _
        wdPrintDocumentWithMarkup, Copies:=3, Pages:="", PageType:= _
        wdPrintAllPages, Collate:=False, Background:=True, PrintToFile:=False, _
        PrintZoomColumn:=0, PrintZoomRow:=0, PrintZoomPaperWidth:=0, _
        PrintZoomPaperHeight:=0
End Sub

I would suggest you select the printer too, I didn't in this example.

Parenthesis are not the problem. VB uses parenthesis to evaluate the value contained. Since the value is a constant, then the parenthesis have no effect.

Part of the problem is that you are playing loose with your indentation. Here is the code with indentation more appropriate to the logic so you can see visually the structure:

Private Sub Worksheet_SelectionChange(ByVal Target As Range)

    If Target.Count > 1 Then
        If Not Intersect(Target, Range("D2,K2,R2")) Is Nothing Then
            On Error GoTo FallThrough
            Application.EnableEvents = False
            Target.Offset(0, 4).MergeArea.ClearContents ' erase their Entries!
            Target.Offset(4, 0).Resize(40, 1).ClearContents ' erase their Entries!
            Target.Offset(86, 0).Resize(13, 7 - (Target.Column = 18)).ClearContents ' erase their Entries!
            Target.Offset(1, 1).MergeArea.ClearContents ' erase their Entries!
            Target.Offset(1, 5).MergeArea.ClearContents ' erase their Entries!
            Application.EnableEvents = True
            Select Case Target.Address

                 Case ("$D$2")
                     'and so we need to clear these ranges on that sheet
                      Worksheets("ShiftReport").Range("C6:E45").ClearContents  ' erase their Entries!
                      Worksheets("ShiftReport").Range("C6:E45").ClearContents  ' erase their Entries!

                 Case ("$K$2")
                     'and so we need to clear these ranges on that sheet
                      Worksheets("ShiftReport").Range("Q6:S45").ClearContents ' erase their Entries!
                      Worksheets("ShiftReport").Range("Q47:S59").ClearContents ' erase their Entries!

                 Case ("$R$2")
                     'and so we need to clear these ranges on that sheet
                      Worksheets("ShiftReport").Range("AE6:AG45").ClearContents  ' erase their Entries!
                      Worksheets("ShiftReport").Range("AE6:AG45").ClearContents  ' erase their Entries!
            End Select
        End If
    End If
    
FallThrough:

End Sub

As we look at the code, you can see that the entire set of logic is contained in an if clause that will only be positive if more than one cell is selected.

But, if more than one cell is selected (or changed if using the Worksheet_Change event handler), then the constants used in the Select statement will never match as they represent only a single cell.

I been converting some of my larger applications to run under 2013, and not experienced anything less then say 2010.

I do suggest that after “any” type freeze up or crash, that you do a de-compile. (and don’t let any startup code run when you do this – exit, re-enter (again without any startup code running). Now do a C+R (again never let startup code run). Now in the code editor do a debug compile.

Doing the above then, 2013 been VERY stable for me. In fact since VERY few changes (if any) occurred to the Access client side (no new features in 2013 desktop over 2010), then few if any changes have been made to 2013! As a result, few if any change should occur in stability.

The best tips are:

If your database is multi-user, then split the application.

Always distribute a compiled accDE as opposed to accDB.

Always run the same version of Access to consume the accDE. And that includes always running the application with the SAME service pack (SP) update level.

And if you not done already, I would consider creating a new blank database with 2013, and then importing everything from the 2010 application.

From that point on for development, then ONLY open + use the application with the given version you are developing with – say 2013 (and again this means the SAME SP level also)

And for any other users of the application, ensure they are using a compiled accDE version, not your development copy.

With the above given set of rules, I not seen or found or experienced any “reduction” in stability with 2013 as opposed to 2010.
