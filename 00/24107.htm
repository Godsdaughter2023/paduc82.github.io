Yes, we were. 

The web security team had reported that they were not able to implement the control using our Cisco IronPort proxy farms, which led to my initial inquiry to Microsoft.  Microsoft weighed in that it should be achievable but no direction on the approach.

I did identify that Cisco does not use standard regex coding, which ultimately was the issue with the web team being unable to block. So, I created my allow and block rules using regex, then researched Cisco syntax, modified the allow / block rules accordingly and successfully implemented tenant restriction based upon domain. Key consideration when creating the block is understanding the yammer pattern for identifying domains is consistent.

Solution Approach:

This design decrypts all proxy traffic that is routed to Yammer.com, looks at the URL properties of the traffic and then enforces either an allow or a block rule based upon the Yammer domain.

Allow Rule Function: All traffic that matches the allow rule (approved domains) is subsequently encrypted and forwarded, enabling full yammer functionality.

Block Rule Function: All traffic that does not explicitly match the allow rule is then inspected against the block rule and prevented from being routed to yammer. Blocked domains are presented a “This Page Cannot Be Displayed”

Regex Rule Set:

Allow Rule:

www\.yammer\.com/approvedomain\.com/

www\.yammer\.com/approvedomain\.com/.*

www\.yammer\.com/approvedomain2\.com/   

www\.yammer\.com/approvedomain2\.com/.*   

Block Rule:

www\.yammer\.com/.*\..*/

www\.yammer\.com/.*\..*/.*

Why not test the application on your working machine with the full edition as runtime mode?

Simply start the application with the /runtime switch and see if it works.

A typical shortcut will look like this:

"C:\Program Files (x86)\Microsoft Office\Office15\MSACCESS.EXE"
"c:\YouProg\MyApp.accDE" /runtime

(above is on two lines, but the shortcut would be on one line with a space between each above line).

The above will thus run your application as runtime on your dev machine (and I NEVER seen ANY documented example that something runs different using /runtime switch as to deploy).

I assume you are deploying access x32 in all cases?

I been deploying runtime Access applications for OVER 15 years. In those 15 years NEVER have I found any example of code working different in runtime as opposed to full edition when using a compiled accDE/mde and running the same version of runtime as compared to full (and this includes sp level).


>([Prod_Date]=#" & fDay & "#)"

How is fDay being formatted in above? If you don’t format fDay to USA format, then simple difference in regional settings will cause above to produce different results or return no records.

Next up, is the issue of using a compiled accDE (often referred to an access executable).

At this point in time the fact of using the runtime is not the issue until such time you test using the /runtime switch on your development machine. If the code fails on your development machine using /runtime, then you have a starting point to demonstrate this is a runtime issue. However, if your application works fine on your dev computer with /runtime switch, then the application should work just fine with runtime on other machines.

Most important here is issue is of course are you deploying a accDB with source code, or are you using a compiled version (accDE)? If you are using a compiled version, then you need to ensure that the SP level of the runtime is the same as your development version. So either the runtime is at a different SP level then your dev version, or vice versa. Note that WHEN you include source code, then EVEN the runtime system can and will re-compile code on the fly - this can often reduce SP differences.

If your code works fine on your dev machine using runtime switch, then the code can and WILL run as runtime mode on other machines. The issue then becomes broken references or different SP levels.

Moving the right() function to server side is a band aid solution. This means MANY OTHER routines and simple VBA code in your application will likely break. You now have "lurking" bugs. I simply would NOT sign off and run the application until such time that broken functions are made to work.

Attempting to repair ONLY ONE query in this manor solves VERY little and would suggest that MANY OTHER parts of your VBA code etc. will remain broken or has lurking bugs. You cannot thus have any confidance that the application will work fine.

You need to resolve + fix the broken function issue – it can effect many other functions and your VBA code. You can never be sure that other code is now working correctly.

I would first test using the above shortcut idea and /runtime switch on your dev machine. If that works, then this is SP level, or broken reference. (and knowing if you are deploy accDB vs accDE will go a along way to resolve this - you likely best try a compiled accDE on those runtime machines.

I much belive this broken reference issue can be resolved, but more important much suggest you fix this, since right now many other features and functions of the application may be broken. It is in a way a "good" that something reproduciable is not wokring, since once you fix this issue, then all other issues are also fixed for this application.
