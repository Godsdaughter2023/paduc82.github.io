a. user say:

That is not the way to do it.

there is 3 ways to do this, depending on what you want to do and how to control it.

Disabling it will not do much as other tasks can trigger these tasks telling the ASSIGNED USER NAME for that task to run it anyways

Deleting each task is an option, not advised and another task can still trigger the assigned user name to run the task. It all depends on the types of tasks and how many different tasks are involved with a certain task.

But for Windows Update, The best way is to do this.

First, you highlight each task and EXPORT the task and save it to an XML file. So you can later "IMPORT the task back if you need to.

Then for each task as there is 4 for Windows Update, you highlight  them, and for each task in General Tab on bottom you see what USER NAME runs the task. The first one is "Interactive User" and the next 3 are "System User"

Now for each one you select the Actions Tab. This is the location of the file that runs the task. If Actions does not show any file, then if it is a custom handler controlled by another trigger.

Now you can disable the 4 tasks from the right side, but not necessary if you do the following.

So what I do to assure it will not automatically run, but I can manually run it and update my system, remember the file locations I mentioned and the assigned user name that runs each task, go to C:\Windows\system32 folder.

Type in the file name from the actions tab for each task in search box.

Right click on the file and select properties>>security tab>>advanced.

Now take a snippet of the Owner and all users in the list.

Now you select CHANGE next to OWNER.

Advanced>>Find Now>>Select your user account name in the list>>OK

Apply>>Verify OK. Now you are Owner of that file.

Now select disable inheritance.

Now select ADD>>Select a Principle>>Advanced>>Find Now>>Select your user account name in the list>>OK>>OK

Now select full control permissions box>>OK

Now you are owner and on the list with full permissions.

Now delete all the other users in the list except yours.

Now no other user name, no matter what, can start that file to start that task.

The reason I take ownership and delete all other users, is I learned long ago that many tasks can send other tasks requests or demands to run other tasks and even if disabled and even deleted in some cases, that a disabled or deleted task does not mean that another task can not trigger the deleted or disabled task and run it. But if you take ownership of the file that runs the task and with full permissions and delete all other user types such as system, Users, Interactive, etc, then if they are not there in the list, they can not run the task.

And you can run the tasks anytime yourself manually.

If you choose to delete all tasks inside a task, I would still take ownership, BUT ALWAYS EXPORT EACH TASK AND SAVE IN A FOLDER BEFORE DELETING THEM TO RESTORE LATER. You can also view the exported XML files in your web browsers.
