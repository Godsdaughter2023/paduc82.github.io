The Windows Modules Installer Worker is controlled by the TIWorker.exe file.

This is part of the normal System Maintenance Tasks that Windows runs on a regular basis.

What it does is to check the Windows Component Store (C:\WinSxS folder) for files have been replaced by Windows Update and are safe to delete. If any of these outdated files are found it will remove the files. It will also compress some files. This helps to keep the component store optimized and prevents it from growing too large which was a problem in earlier versions of Windows.

It is designed to start when the computer is idle and will continue until the task is finished.

It isn't normally anything to be concerned about.

britcay say:

So it's probably too late for the OP but I had a similar issue with cryptsvc.dll on a windows server 2008 R2 machine, preventing windows updates from running amongst other things. I suspect in our case it was an residual issue with removing antivirus (as the problem stems from around that time and I know we had serious problems removing Avast!)

I simply copied the dll and the registry settings from a good, working identical system

c:\windows\system32\cryptsvc.dll

HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\services\CryptSvc

So in regedit, right click the cryptsvc key, then export. Copy the saved .reg file onto the bad server along with the .dll

Then double-click the .reg file to import it and then register the .dll from an elevated command prompt window.

regsvr32.exe c:\windows\system32\cryptdlg.dll

(NB: also use command line rather than explorer to copy the dll)

The failing service then started immediately and in my case (server 2008 r2) I was able to then successfully start appidsvc (application identity service).
