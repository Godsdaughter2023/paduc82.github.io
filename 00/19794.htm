The root of the problem here is the incorrect table design.  By having multiple shipment cost columns A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

A correct design would decompose the table into two related tables of which the referencing table has a foreign key column referencing the primary key of the current table, and a single ShipmentCost column.  There can then be no Nulls and the concatenated data can be returned as a single string expression by a concatenation function.  You'll find such a function, which uses the highly efficient GetString method of the DAO recordset object, in Concat.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

This little demo file illustrates the use of a GetList function which concatenates values from a related table, using Northwind data as its example.  Allen Brown also provides a concatenation function, using DAO, at:

http://allenbrowne.com/func-concat.html

A reason often given for an incorrectly structured table like this is that data is being imported from another source, often an Excel worksheet.  Where this is the case the data should be decomposed in Access as part of the import routine.  The DecomposerDemo file in my same OneDrive folder illustrates the methodologies for this.

I don't believe that you are aware what you are asking for.

Let us assume we have a FromDate, a ToDate and want to check if a ThisDate is inside that period.
The result can be -1 if ThisDate is before FromDate or +1 if ThisDate is > ToDate and 0 if ThisDate is inside.

Means we have 3 possible results.

The next step is to compare 2 periods, means we have to compare 4 dates and we get 2*3 results if we perform the last step with FromDate/ToDate of period2 with FromDate/ToDate of period1.

But to get the result we don't have 2*3=6 different calcualtions, we need 3^2=9 calcualtions!

If we add another period we don't need 9*3=27 calculations, we need 3^3^2=729 calculations to catch all possible scenarios.

In other words: Forget it, there is no simple formula for that.

The calculation is too complex if you want to compare which date is lower as other dates.

A relative simple solution is to "draw" a Gantt string, that can also be done with formulas, but is also a complex solution:

a) Calculation the number of days for the overall period, you need one cell for each day. The StartDate is accosiated with the first cell.

b) For the first period, calculate the cell offset = FromDate - StartDate and "draw" a 1 into the cell, continue till you reach ToDate in the following cells.

So you'll end up with something like this:

Now you can use a simple COUNT formula combined with OFFSET to get the number of days of any period you like.

It is relative simple to use VBA for such a "Gantt calculation".

And the result of the dates from your picture is 770 and not 761 as you think:

15/11/2008 till 31/12/2010 = 777 days

minus
