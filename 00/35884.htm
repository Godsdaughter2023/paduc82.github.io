lemp say:
You probably don't want to hear this ... but the problem likely is something you're doing rather than "another MS developer problem."

I have a Linksys WPC54G (which I think is probably ver. 1 because it doesn't have any version number on the label) running in an ancient ThinkPad A22m (XP Pro sp3) that has no problem remembering a WPA key.

It's of course up to you what you use as an encryption key, but most people these days use alphanumeric passphrases rather than a string of hex characters.

According to the spec:

This key may be entered either as a string of 64 hexadecimal digits, or as a passphrase of 8 to 63 printable ASCII characters. If ASCII characters are used, the 256 bit key is calculated by applying the PBKDF2 key derivation function to the passphrase, using the SSID as the salt and 4096 iterations of HMAC-SHA1.

See this page --> http://jorisvr.nl/wpapsk.html

Unlike the situation with WEP, user WPA interfaces -- both Linksys and Windows -- use the passphrase only, rather than permitting you to enter the actual encryption key.

You are certainly correct that Linksys explicitly says that its utility only accepts "a Pre-shared key of 8-63 characters in the Pre-shared Key field." Windows WZC is undoubtedly the same, i.e., limited to 8 to 63 characters. 

As far as I have been able to determine, Microsoft has removed the WZC documentation from its web site.  Although Windows may not complain when you type in your 64 hex characters, it's probably either truncating it or just ignoring the entry completely.

As far as modifying the entries in the Alternate Addressing tab, that is not likely to help you.  Here's what that does:

By default, the computer first attempts to contact a DHCP server on the network and dynamically obtain configuration for each installed network connection, as follows:
  • If a DHCP server is reached and leased configuration is successful, TCP/IP configuration is completed.
  • If a DHCP server is not reached after approximately 60 seconds or leased configuration fails, the computer uses the setting on the Alternate Configuration tab to determine whether to use APIPA for automatic TCP/IP configuration or to configure TCP/IP with the alternate configuration. When APIPA is used, Windows TCP/IP uses an address in the APIPA IP address range (169.254.0.1 through 169.254.255.254) and the subnet mask is set to 255.255.0.0.

What that all means as a practical matter is that if you can't connect to the router's DHCP server, Windows will configure the NIC as directed.  But if you can't connect to the DHCP server, you almost certainly can't connect to the router at all, so it doesn't matter what IP address you put into the NIC.

I suggest that you make up a 63-character (or shorter) WPA passphrase and see if that solves your problem.  If you use a non-dictionary set of characters, including digits and non-alpha characters, you are more than adequately safe with a lot fewer than 63 characters.  See this --> http://www.zdnet.com/blog/ou/how-long-does-a-wpa-key-need-to-be/127

the solution is to create the follwing .cmd file and execute it, after execution imnmediately reboot !

 

-------------------------------------

@echo off

:: 32 bit and 64 bit
IF EXIST "%SystemRoot%\System32\actxprxy.dll" "%SystemRoot%\System32\regsvr32.exe" "%SystemRoot%\System32\actxprxy.dll"
IF EXIST "%ProgramFiles%\Internet Explorer\ieproxy.dll" "%SystemRoot%\System32\regsvr32.exe" "%ProgramFiles%\Internet Explorer\ieproxy.dll"

:: 64 bit only (32bit on 64 bit)
IF EXIST "%WinDir%\SysWOW64\actxprxy.dll" "%WinDir%\SysWOW64\regsvr32.exe" "%WinDir%\SysWOW64\actxprxy.dll"
IF EXIST "%ProgramFiles(x86)%\Internet Explorer\ieproxy.dll" "%WinDir%\SysWOW64\regsvr32.exe" "%ProgramFiles(x86)%\Internet Explorer\ieproxy.dll"

-------------------------------------------------------

after execution and immediate reboot, all problems are gone, everything is ok again.

please note the different path under windows 7 x64 (ProgramFilesx86), therefor the x64 part.

 

You must use regsvr32 to re-register the two proxy DLLs (actxprxy.dll and ieprxy.dll), thenreboot

You can use the upper batch file (.cmd file) to do this.

You must run this batch file with administrative privileges (right click on the file and chooseRun as administrato r)

Previously, actxprxy.dll (ActiveX Interface Marshaling Library) was used as the proxy for a multitude of system interfaces, such as IShellFolder and IServiceProvider.
