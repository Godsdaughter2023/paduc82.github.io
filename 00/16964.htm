If the Picture has the same Name on each sheet, then a simple macro can be used.  Suppose we start with a single sheet and we:

  1. put a picture on the sheet
  2. make 30 copies of the sheet
  3. change various cells on the sheets

 

We now have a workbook with the same picture on each tab.  All the pictures have the same Name (for example Picture 1).  The following code:

Sub PicKiller()
Dim s As Shape, w As Worksheet
For Each w In ActiveWorkbook.Sheets
    w.Activate
    w.Shapes("Picture 1").Select
    Selection.Delete
Next
End Sub

will delete that same Picture in each sheet.

 

If the Picture has a different Name on each sheet, the code must be modified

If some sheets do not have the Picture, the code must be modified.

Peanaboo, here is an alternative macro that you can try if the name of the picture is not the same on each sheet :

1. On any sheet, select the picture to be deleted
2. Run the macro

Sub PicKiller2()
   Dim ws As Worksheet
   Dim sh As Shape
   pictop = Selection.Top
   picleft = Selection.Left
   precision = 10   'set to 0 if the pictures are EXACTLY in the same place on each sheet
   For Each ws In Worksheets
     For Each sh In ws.Shapes
       If Abs(sh.Top - pictop) <= precision And Abs(sh.Left - picleft) <= precision Then
         sh.Delete
       End If
     Next sh
   Next ws
End Sub

Warning: This macro will delete all shapes in all worksheets that have their top left corner at the same spot (+/- the value you have for precision) as the picture that you have selected.

Why I returned my Galaxy S7 Edge for a Lumia 950

I believe in giving credit where credit is due, so I wanted to share my story as a token of my appreciation of the mobile environment the fine folks at Microsoft have developed that IMHO doesn't get nearly enough recognition.

After owning 2 iPhones I switched several years ago to my first of many (now 5) Windows phones. After loving my Lumia 800, 920 and 1520 phones I decided it was time to try Android after all the hype I read about the new Galaxy S7 Edge and seeing how slick it looked in person. 14 days later (the maximum allowed by AT&T for returns), however, I finally had to give up on Android and returned it to the store for a Lumia 950 to the bewilderment of every sales tech in the store. Here's why...

My Lumias have always just worked for me in every way I've needed them to without me having to give it a second thought. Being self-employed, I rely heavily on my phone to keep up to date with information online, emails and keeping in touch with customers. It is my main device, and I couldn't run my business nearly as well without it. So when I opened up the email and browser apps on the S7, it was almost painful to see how clunky they were in comparison; browser tabs were handled awfully, email conversations were sorted weirdly, everything required more clicks than I was used to just to get basic functionality. I loathe extra clicks. So I spent an evening finding new apps to handle these basic features and setting them up properly to get me closer to the simplicity I was used to, and to its credit there were a plethora of apps available, understandable considering how poor the included apps are.

Next issue was with notifications. There were hundreds of them. Every app that came with the phone, of which there are more than a few(!), decided I needed to be notified of one thing or another multiple times daily. Into settings I went to disable notifications on dozens of applications. While that cut back on the amount, it was frustrating getting a notification and selecting it hoping to go into the app only to be taken to the phone's home screen more than half the time. I'm not sure if that was a bug or somehow user error, but the extra clicks were again quite unappreciated. While Microsoft may have come late to the notification center party, its solution is simplistically beautiful and has always gotten me exactly where I need to go, and quickly.

Then finally came strike three: OK, Google. I would say it felt like a beta, but I used Cortana during its beta and it was much more functional and accurate. It worked for me about 1 in every 4 attempts. I use Cortana constantly as I spend a lot of time driving and it keeps me focused on the road instead of my phone screen. Cortana will read my texts to me through my car (S7 did too but it required me to push a button on my car's display) AND ask me if I'd like to voice a reply (S7 didn't). When I would attempt to use OK, Google to reply to texts it would recognize I was trying to text, again, about 25% of the time. Upon opening up the messaging app to reply to the text (grr... more clicks) I could still find no way to verbalize a reply. This all came to a head finally on day 14 of ownership, fumbling with my phone while driving trying to reply to a time sensitive text message, with calling back not an option, and I just so happened at that wonderful moment to be nearby the AT&T store. So back it went. When I complained about this feature to a friend of mine afterwards he said that I could probably have found an app that would handle this feature, and perhaps that's true, but in my opinion a phone who's raison d'etre is sending and receiving calls and texts, if nothing else, that requires you to find a third party app to handle these basic tasks well is not a "phone" I wish to have, thank you very much.

So all this to say, "Thank you, Microsoft. I know you haven't had nearly the success you deserve in the phone market, but for this one customer at least, you are the bee's knees, and I couldn't be as successful personally without you!" Thanks for reading.

Yours Truly,

-Michael (who wasn't compensated for his opinion shared here)

vijay a. verma say:

My take -

1. In DSUM, You need to put the criteria in fields itself somewhere where in SUMIFS, you need not put criteria anywhere but criteria will be built inside the function itself. In my view, this is a great flexibility.

EDIT - Rather than fields itself, it is sheet / workbook itself.

2. Making criteria in DSUM is not user friendly whereas making criteria in SUMIFS is intuitive as it follows regular Excel syntax. While building crietria in DSUM, it is easy to commit mistake.

3. Building complex criteria in SUMIFS is very easy.

EDIT - I tried to make criteria using fields and had a tough time in DSUM.

3. DSUM requires column headers for both the range and criteria whereas SUMIFS doesn't require column headers. That is why excel uses the term database in connection with DSUM as database means that column headers should exist. Also it mentions fields which means headers.

4. I also feel that SUMIFS should be faster than DSUM (This is just a conjecture and need to be tested)
