Here's an alternative code solution. I'm not sure when this method was added to Macs so you may need the latest upgrades and 365 to make it work.

Steve's method makes shapes invisible until you manually reset them from the Selection pane and also your presentation will need to have macros. IF this works the shapes will return when the presentation closes and the code is not needed once the animation is added.

Sub triggerMe()
    Dim osld As Slide
    Dim oshp As Shape
    Dim oeff As Effect
    On Error Resume Next
    Set oshp = ActiveWindow.Selection.ShapeRange(1)
    If Not oshp Is Nothing Then
        Set osld = oshp.Parent
        Set oeff = osld.TimeLine.InteractiveSequences.Add.AddEffect(oshp, msoAnimEffectFade, , msoAnimTriggerOnShapeClick)
        With oeff
            .Timing.TriggerShape = oshp
            .Exit = True
        End With
    End If
End Sub

Select each shape in turn and run the macro. Once done the code is no longer needed. If it does not work or errors please say which version you have

john mcghie say:

Hi Peter:

Word's backup file is the "save before" the current version.  With "Always make backup" turned on, Word re-names the previous version to "Backup of..." as part of each Save process.

Because it is a "rename" not a "save", you can't change the location.

So it protects you from accidental deletions, Word crashes and hangs, document corruptions, etc.  Provided you remember NOT to save a file that is having trouble! If you think you might need the backup, discipline yourself to use "Save as..." to a new file name, leaving the previous backup intact.

In Word 2016, due to sandboxing rules, Word creates a subfolder in the original folder with the same name as the document and places the backups in there.

John is correct: you really should be running Time Machine to an external disk.  Remember that it is absolutely guaranteed that your hard disk WILL fail sometime.  And when it does, chances are you will lose the entire contents of it.

Hope this helps

amer abdeen say:

Creating History Records in Microsoft Access 2010

Hi,

 

I have three tables in Microsoft Access 2010:

1. Employee details (file number, name, ...)

2. Training list of 2011 (containing file numbers, course date, course ID, ...)

3. Training list of 2012 (containign file numbers, course date, course ID, ...)

The subform shows all the courses that the employee has attended in chronological order (i.e. first by the courses in 2011 then by the courses in 2012). The subform should be a datasheet where its content changes when moving to the next record of the main form.

 

Actually, I was unable to create this form... any ideas?

ken sheridan say: Unfortunately you are starting with a flawed design.  By having separate tables for each year's training you are 'encoding data as table names'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

What you have is a many-to-many relationship type between employees and courses.  To model this you need three tables, e.g.

Employees
....EmployeeID (PK)
....FirstName
....LastName
....etc

Courses
....CourseID (PK)
....CourseTitle
....CourseDate
....CourseVenue

and to model the many-to-many relationship type between then:

EmployeeCourses
....EmployeeID (FK)
....CourseID (FK)

The primary key of the last table is a composite one made up of both columns.  The table can if necessary have other non-key columns representing other attributes of the employee's attendance at the course in question.

Setting up a form/subform as you describe is now very simple.  Firstly create a form based on Employees, in single form view.  Then create a form based on the following query:

SELECT EmployeeCourses.*
CourseDate, CourseVenue
FROM EmployeeCourses INNER JOIN Courses
ON Courses.CourseID = EmployeeCourses.CourseID
ORDER BY CourseDate;

This form can be in datasheet view or continuous forms view as preferred, though I'd always use the former as it allows more control over the layout.  Embed it as a subform in the parent employees form, linked to it by setting the LinkMasterFields and LinkChildFields properties of the subform control in the parent form to EmployeeID.  In the subform include bound controls for the CourseID column and the CourseDate and CourseVenue columns (or whatever non-key columns you have in the Courses table which you want to show in the subreport), along with any other non-key columns from the EmployeeCourses table.

The control bound to the CourseID  column in the subform should be a combo box set up as follows:

ControlSource:    CourseID

RowSource:     SELECT CourseID, CourseTitle FROM Courses ORDER BYCourseTitle;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.

The other controls will be text boxes.  Those bound to the columns from the Course table, CourseDate and CourseVenue in my example, should have their Locked property set to True (Yes) and their Enabled property to False (No) to make them read-only.

You'll find an example of a basic many-to-many relationship type and how it is represented in a form/subform as ParentActivities.zip in my public databases folder at:
