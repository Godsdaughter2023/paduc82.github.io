dan at it associates say:

"hmm...even if i re-activate through the telephone and speak to a customer service rep, its still not transferable?"

 

Nope.

 

"I thought only OEM version are non-transferable?"

 

Yes, in the past Microsoft's consumer licensed retail-purchased software was always transferable and OEM software was non-transferable.  With Office 2007 Microsoft introduced a new license concept, consumer retail software that is not transferable.  It was called the MediaLess licensing kit (MLK).  For Office 2010 this evolved to the Product Key Card (PKC).  Now, for Office 2013, all consumer license designations are non-transferable.

 

I suspect in the future Microsoft will transition all consumer Office licenses to the pay as you go subscription model, a la Office 365.  With ubiquitous internet connectivity, this is now very doable.

Access relationship with Zip Codes - Microsoft Community

gward99 say:

Access relationship with Zip Codes

I need to build a database that tracks which zip codes doctors are near so I can link patients with the closest doctor.  More specifically, so I can run a query that will match a doctor with a patient based on the patient zip code.  I currently have 3 tables.  A doctors table that consists of: docID, docFirstName, docLastName, docAddress, docCity, docState, docZip, docEmail, docPhone.  A patient table:  patID, patLastName, patFirstName, patAddress, patCity, patState, patZip, patEmail.  My 3rd table is where my trouble is.  Currently it consists of docID and zipsServiced, I made both a primary key and connected the doctor table w/ a 1-many relationship. My thought process was for this third table to have one record for each zip code serviced and the doctor who services it.  What is the best way to connect the patient table to this 3rd table? Is there a better way to do this? Thank you in advance for your help.

ken sheridan say: While having no direct bearing on you current question, you should really have a fourth table, which is simply a list of all zip codes.  Your table of zips serviced by doctors models a many to many relationship type between the doctors table and the table of zip codes, and should be related to both the doctors and zip codes table in many-to-one relationships type, with referential integrity enforced in each case.  This then ensures the integrity of the data.

To find doctors who service the zip code area in which each patient is located is just a case of joining your current three tables in a query; patients is joined to the zips_serviced table on patZip/zipsServiced, and doctors to zips_serviced on docZip/zipsServiced.  Return the relevant columns from the patients and doctors tables in the query, which will return multiple rows per patient, one for each doctor who services their zip code area.

You can base a form/and or report on the query and open it filtered to one or more patients via a simple dialogue form, e.g. you could include a multi-select list box, lstPatients, of patients on a form, setting up the list box as below, to open a report:

For its RowSource property:

SELECT patID, patFirstName & " " & patLastName FROM Patients ORDER BY patLastName, patFirstName;

For other properties:

Name:                  lstPatients
BoundColumn:    1
ColumnCount:     2
ColumnWidths:    0cm;8cm (Access will automatically convert these to inches if you are not using metric measurements)
MultiSelect:          Simple or Extended as preferred.

Add a button to the form to open the report, called rptPatientDoctors in this example, with the following in its Click event procedure:

    Dim varItem As Variant
    Dim strPatIDList As String
    Dim strCriteria As String
    Dim ctrl As Control
    
    Set ctrl = Me.lstPatients
    
    If ctrl.ItemsSelected.Count > 0 Then
        For Each varItem In ctrl.ItemsSelected
            strPatIDList = strPatIDList & "," & ctrl.ItemData(varItem)
        Next varItem
        
        ' remove leading comma
        strPatIDList = Mid(strPatIDList, 2)
        
        strCriteria = "patID In(" & strPatIDList & ")"
        
        DoCmd.OpenReport "rptPatientDoctors", _
            View:=acViewPreview, _
            WhereCondition:=strCriteria
    Else
        MsgBox "No Patients Selected", vbInformation, "Warning"
    End If

The above assumes that patID is a number, not text.

john w. vinson say:

I'd say that your AllZips table should have nothing but the zipcode - that's the POINT. A Zipcode is a valid "entity" in its own right; it's a mail delivery service provided by the Post Office, and as such has nothing intrinsically to do with patients or doctors. So a Zipcode is a valid attribute of a Patient or a Doctor - each person presumably has a mailing address; but a Patient is not an attribute of a Zipcode (there'll be more than one, typically!)

You can load all your Zips from both tables into AllZips by using an Append query. Easiest would be to just create an Append query based on your Patients table; select only the zipcode; append it to allZips, which would have Zipcode as its Primary Key. You could append the city name as well (bearing in mind that one zipcode might have more than one city, this is a bit iffy). You'll get a warning message that "213 records could not be added due to key violations" - but don't worry; that's just the primary key rejecting duplicates, and you WANT to reject duplicates.

Then you can do the same with the Doctors table; there'll be duplicates rejected there too, but you'll add all the new unique zips.

If you wish, set the Unique Values property of these queries so it will only try to append each zipcode once. But for a one-time operation I don't think it makes any difference.
