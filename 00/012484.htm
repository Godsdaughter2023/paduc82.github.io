    nunof32 say:

    Spell Checker options - ignoring certain repeated words

    I have a document where from time to time I need to type relatively long strings of x's (e.g. x x x x x x). Is there a way to tell the spell checker that flagging repeated words is ok but not when they are x's ? Thanks.
    daniel g. grau say:
    Try this:

    • Select the string of Xs
    • Apply Tools > Language > English (US) and place a checkmark in Do not check...
    • Select two Xs in a row
    • Select Word > Preferences > Autocorrect
    • Place a checkmark at the top left
    • Make sure the Autocorrect blue tab is selected
    • Place a checkmark in Replace text as you type
    • Place a checkmark in Formatted text
    • In the Replace field, type two Xs
    • Click on Add

    From now on, every time you type several Xs in a row, they won't be spell checked. The language will resume to English (plus Do spell check…) after typing them, starting with a pair. If you only type one X, it will not have these attributes, but since it's a single letter, Word won't nag you.

    In order to test the results more clearly and get a hang of how that works, apply a color to the Xs before following the above bullets. You'll see the color change as you type and Word autocorrects them, and see it change back to black as you continue typing.

    a. user say:

    Nice bit of detective work, Greg. I also had a look and could replicate your finding (which certainly goes a long way towards explaining some of the odd results I have seen in the past). 

    However, for anyone else looking, I don't think it's just RTL language support that does it because, for example, setting up Hindi as a language also causes the problem, and AFAIK that doesn't come into that category. But it's not just "languages that typically use a non-Latin script," either, because, for example, setting up Japanese doesn't appear to cause the behaviour change. So perhaps there is some other factor in Word's language/script support that makes the difference.

    Historically, the documentation for \*Charformat has been ambiguous (but I'm going back a long way here :-) ). Originally I think it suggested that you formatted the first character in the field. In the very early versions of Windows Word, Word did not by default insert the space character after the opening "{" that it has since around Word 6, I think. So "the first character" would typically be the first character of the field. Since then, I think it's fair to say that most people probably assumed that formatting the first non-whitespace character of the field code was what was needed. Which is interesting, because the ISO standard says this:

    <<

    If this switch is present, the formatting of the run containing the first instrText element after the fldChar element with a fldCharType attribute value of begin is applied to all runs in the field result whenever a new field result is generated

    >>

    In the case where the field is set up as { REF }, this actually implies that the formatting of the space character should determine the formatting of the result.

    Experiment (in Word 2010) suggests that with only English enabled, the result formatting follows the formatting of the "R" regardless of whether it is the first or second character. With (say) an RTL-script language enabled as well, the formatting of the R appears to be ignored, and \*Charformat will only work if the R is preceded by a space, in which case, the formatting of the space is used. OK, I have only used "R" and "<space>R", and not other combinations, so it may be more complicated than that, but
     a. neither the "English only" result nor the "English + RTL" result seems to me to conform to the ISO standard. Not that that is a surprise
     b. to play safe you would at least need to ensure that every field that used \*Charformat had a space and formatted both the space and "R" (for REF fields). That should work in both scenarios, except perhaps when the formatting modifies the R in a way that causes Word not to recognise the field code, e.g. you apply hidden formatting to the R.

    Finally, when I looked at the XML generated after field updates in those two scenarios (English, English+), the split into text runs was sometimes different. It's not unreasonable to assume that Word uses a different algorithm for dividing up text when certain scripts or script types are enabled. But that would be slightly odd, because Word has to be able to do that even when the RTL language (or whatever) is not enabled in Office (because you can still type using an RTL keyboard, for example).

    Or maybe it is an arbitrary consequence of some other setting.

    Talk about ever-decreasing circles... :-)
