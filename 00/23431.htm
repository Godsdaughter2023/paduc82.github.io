ji_chicago say:

I found the answer to my problem, and hopefully this will resolve some of the other people's errors as well.  My problem was related to the date formats.  Outlook 2003 accepts DTSTART/DTEND without any time zone specification.  Outlook 2007 (and later versions) requires a time zone.  In my case, the solution was for the application to convert the date/time to UTC and then set the DTSTART/DTEND to a format of YYYYMMDDTHHMMSSz (note the 'z' appended at the end which tells Outlook that the time zone is UTC).  You could also define a time zone explicitly and include that in the file, but that is much more work ;)

It appears this might be the same problem Pinki64 is having, since the file opens when you tell it to skip the file's dates by drag/drop into a timeslot on the calendar.

Hope this helps other experiencing a similar problem...

What's the problem with this macro, it makes the other ones - Microsoft Community

danimaranelo say:

What's the problem with this macro, it makes the other ones fail?

Hi!

I have this macro: It records every change made on any sheet.

Private Sub Workbook_SheetChange(ByVal Sh As Object, ByVal Target As Range)

If Sh.Name = "Changes" Then Exit Sub

Application.EnableEvents = False

UserName = Environ("USERNAME")

NewVal = Target.Value

Application.Undo


oldVal = Target.Value
With Sheets("Changes")
LR = .Range("A" & Rows.Count).End(xlUp).Row
.Range("A" & LR + 1).Value = Format(Now(), "dd/mm/yyyy")
.Range("B" & LR + 1).Value = Format(Now(), "hh:nn:ss")
.Range("C" & LR + 1).Value = Sh.Name
.Range("D" & LR + 1).Value = Target.Address(False, False)
.Range("G" & LR + 1) = oldVal
.Range("E" & LR + 1) = NewVal
.Range("F" & LR + 1) = UserName

Target = NewVal
End With
Application.EnableEvents = True

End Sub

The problem: Each time I try to run the other macros it throws an error on Application.undo or it says the macro can't be run. This macro is on ThisWorkbook and the others are on standard modules.

jllatham say:

It actually works for me exactly as written.

 

Do any of your sheets have data filters applied to them?  I did a little reading (search for 'Excel Application.Undo' and you'll see a rather prominent option shown as'Excel Application.Undo Error'.  A couple of them mentioned problems when the data on a sheet was filtered.

 

Doing just about anything to the workbook (as writing to a cell) clears the Undo stack.  But in this case, the first thing you're doing is handling a change without changing anything at the point you call Application.Undo.  It should work, as it is doing for me.  If you were calling another routine first, that might clear the .Undo stack and you'd get an error.

 

Do any of the other worksheets have their own individual Worksheet_Change() or other Worksheet_ level event processing going on?  Those would get called before the Workbook_SheetChange() gets called and if it makes changes to the sheet also, then the .Undo stack would get cleared.

 

There are ways to work around that issue if it turns out to be part of the problem.

 

BTW: you could kind of save a bit of work within the code. You set LR = the last row in column A, then you constantly add +1 to that. I'd just use

LR = .Range("A" & Rows.Count).End(xlUp).Row + 1

and then all of my assignments would be without the +1 in them. You're making a calculation 7 times that you could make just once.
