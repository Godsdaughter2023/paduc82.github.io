Conditional Formatting of Symmetric Table

Let's say I have a table that looks something like this:

 

A    B    C    D    E

A    -    3    5    1    4

 

B    2    -    1    3    4

 

C    0    2    -    1    1

 

D    5    3    1    -    4

 

E    0    2    2    1    -

 

The table is symmetric in that the row headers and column headers are the same.  It represents all the times that two players have met.  I want to conditionally format the cells to highlight the cells that have the greater value across the symmetric diagonal.

For example, look at the two cells that represent A versus B.  I want to highlight the (A,B) cell because it has a higher value (3) than the (B,A) cell (2).    Player A beat player B 3 times, while Player B beat Player A only 2 times.

Is there a nifty formula approach to doing this kind of conditional formatting?  =IF( (A,B) > (B,A), 1, 0) = 1 or something line that?

 

Thanks

 

Eric

hansv mvp say:

Select the table (with or without the row/column headers, that doesn't really matter).

Note the address of the active cell within the selection. In the following, I'll assume it is G11, just as an example.

Click Conditional Formatting > New Rule... on the Home tab of the ribbon.

Click "Use a formula to determine which cells to format".

Enter the following formula, using the address of the active cell as noted above:

 

=G11>OFFSET($G$11,COLUMN()-COLUMN($G$11),ROW()-ROW($G$11))

 

It's important to use relative/absolute references exactly as above.

Click Format... and specify the desired highlighting.

OK your way out.

Sub ajunk()

' ajunk Macro

        wdPrintAllPages, Collate:=True, Background:=True, PrintToFile:=False, _
        PrintZoomColumn:=0, PrintZoomRow:=0, PrintZoomPaperWidth:=0, _
        PrintZoomPaperHeight:=0
End Sub

"Sub ajunk()" is the start of the macro, by the name "ajunk" (make yours more descriptive)

ActivePrinter = "PDF-XChange Printer Lite 2012" - this is the command to set PDF as the printer (your printer name will be different)

jllatham say:

I would start off by creating a test folder with a sub-folder or two in it and a few Excel files in each.  That way if I have messed up terribly you won't lose actual data.

 

The workbook this code is in cannot be in any of those folders - it must be outside of them in a separate folder.

 

Download this file:

MDuffy_ProcessAllExcelFiles.xlsm (167 KB)
http://www.mediafire.com/view/g5gv4wee46zeyjg/MDuffy_ProcessAllExcelFiles.xlsm

Given the name of your macro, the code should be all set up to run it.  All you have to do is copy your code into the workbook.  You can put your code into a separate regular code module, or just paste it in at the end of the code that's already in the workbook.

 

Near the beginning of the code module AccessFileSheetsCode, routine Sub ListIndividualFiles()  and just a little way into it you will see this section:

 

'******************************************************************
'******************************************************************
'YOUR PIECE OF THE PIE
        'call your routine here
        'it can be in this module or another
        Run "EtractMaterialData"

'******************************************************************
'******************************************************************

If you ever need to change the name of your routine - that's where it goes. 

 

There are 4 sheets in the workbook.  3 of them are used by the code I have written.  The last one, currently named "Sheet1" is yours to do with as you need, and you may add as  many others as necessary.  Two of the 3 sheets are 'report' sheets: you'll get a list of the files and the sheets within those files that were processed, and if any errors are detected during it, that will be reported on the ErrorReport sheet.  About the only error I can think of (and it actually occurred during testing) is that Excel might not be able to open a file because it has become corrupted.

Starting in the top-level folder, it processes all ".xls*" files in it, they could be .xls, .xlsx, .xlsm or .xlsb type files.  It then looks for sub-folders and processes all Excel files in those, and even in their sub-folders.

 

In processing each Excel workbook, my code "flips through" each worksheet in the workbook, one at a time, and after it flips to a worksheet, that's when the call to ExtractMaterialData is made - so your code can process against the "ActiveSheet" properly.
