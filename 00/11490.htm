ODBC linked table vs Pass-Through query - Microsoft Community

darrend7 say:

ODBC linked table vs Pass-Through query

Earlier today, it was suggested that I do a pass-through query to get data from an ODBC source, instead of linking a table. I have done that, but the query is returning only about 2/3 the records that the linked table pulls. I have checked and rechecked the ODBC connect str and it looks identical to the tables.

Any ideas on why the table would give me over 200K records and I am only getting about 138K in the query?

I don't have anything in the "Max Records" field in the property sheet of the query.

albert d. kallal say:

As noted, I can’t see why the linked table vs that of the PT query does not return the same number of records. However, it possible that some server “time out” is occurring.

It not clear what the additional queries in Access are doing, but I would “attempt” to break down the process and only pull records from the server based on some date range, or invoice number range or some such – I can’t imagine the “need” to process that many records for some kind of often used “daily type” of process.

 

I would also consider moving those large tables to a “temp” database that you re-create each time, since you working with such large data sets, and this approach would get rid of the “huge” bloat (file increase) problem also. (so you link to that external local data store).

 

Another approach would be to ONLY pull new records each day. If the new records have a date, or even some kind of “auto number” ID, then you likely only have to pull a rather small “new” dataset each day to append to the existing large dataset.

 

However, all of these “ideas” assume you have to always work on such a large data set for each report or whatever your process is doing – that as noted by the other poster seems like a rather un-usual process.

 

And also ensure you turn off row locking in the Access client, this can “reduce” bloating by huge amounts.

 

1 million rows is not too large by today’s standards. If each row is say about 120 charters, then you can fit 30 million records in Access. And if each row is 240, then you can fit 15 million. So 1-3 million records should fit rather comfortable in Access tables with room to spare.

 

However, I would break down what you actually doing with the data, and cook up a approach in which you only query the data you need, or as above suggest only pull “new” records on a daily basis (this approach is quite workable especially if you have some kind of auto number “ID” that increments from the server tables. (You simply pull all new records greater than the max last “id” you have in Access.

 

Last but not least, one may well want to write + build some of the processing code you required server side, and then have the server only produce the results you require that are THEN pulled down to Access for reporting/processing. So those "multiple" queries should be moved server side - only the "last" steps should occur in Access.

 

Having to work with such a large data set that resides on the server suggests that some kind of “server side” processing, or at least pre-processing of data should occur on that side before such data reaches Access.

 

At the end of the day, since this is a single table, then you likely not see much increase in using a simple linked table vs that of a PT query – with one table, they usually run the SAME speed.

The gains and fix in speed to your problem is not hardware, or a linked table vs that of a PT query, but a re-think of your overall process and what you are attempting to achieve will be key to your performance issues.

You can spend some time “researching” why the PT query seems to choke, but finding/fixing that issue not going to yield you much of anything in terms of performance – a different approach likely is required here.

arnoldkomitzky say:

Power point, setting up slide timing

Hi,

I'm working on a presentation and am trying to set up the slide timing. I have been able to do some of this but have a question as to procedure.

Do I have to start at the beginning and go all the way through the slides setting the time for each one? I would like to work on some of them in sequence and then continue with new ones later. It seems that I have to start from the beginning each time, is that true or is there a way to do it in segments?

Thanks for any help!

lauramfoleydesign say:

Hi, Arnold!

The simplest way to capture slide timing is to work through the whole presentation from start to finish using the Rehearse Timings in the Set Up toolset in the Slide Show tab:

But using that tool, it's not possible to start anywhere but from the beginning of the presentation. But it's easy to segment the file so that you can work on subsets of the main presentation.

  1. Note the numbers for the slides you want to work on. Then Save As a new file, using the slide count to name it (e.g., slides 23–28.pptx).
  2. In this new file, delete all of the slides before and after the ones you'll be working on. In the above example, you'd delete slides 29–end of show and 1–22. Save your file.
  3. Use the Rehearse Timings tool on this new file.
  4. When you're happy with the timing, copy all of the slides. Open the original file and delete the old versions of the slides you just worked on. Then paste the revised slides into the presentation where they belong.

All of the new timing you just created will become part of the larger file. Ta-da!
