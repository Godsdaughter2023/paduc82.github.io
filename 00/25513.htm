ken sheridan say:
However if I save the query and reopen it the calculated field disappears...?


Why does that happen?



Because it's an UPDATE query, not a SELECT query.  You don't need a separate computed column for this, simply use the expression:

UPDATE [MyTable]
SET [MyField] = [quantity] * [price];

But why are you introducing redundancy, and the consequent risk of update anomalies, by storing a computed value in a column?    If you wish to do this in the table rather than a SELECT query use a 'calculated field' in the table design.  That way it will automatically be kept consistent with the values in the two columns from which the computed value is derived.

alexct say:

I have a complicated nested iif statement that updates my table, part of which uses the product of two columns and wanted to simplify for readability purposes the iif statement by introducing a variable (equal to the product,  etc)

I guess the only other way would be to code the iif statement in vba. 

ken sheridan say:
I guess the only other way would be to code the iif statement in vba. 

I think you are probably right.  I would normally tend to wrap complex conditional operations in a function rather than building an expression of Byzantine complexity using the IIF or SWITCH functions.  As well as reducing the complexity of the expression, the function's code can be appropriately commented.

Another possibility would be to split it into several queries, so that the computed column in the result table of a SELECT query can be referenced in the UPDATE query.
