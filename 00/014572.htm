****Read below regarding the Security Center alert about MSE being off****

There have been some recent reports of this situation. It may be due to a miscommunication between MSE and the WMI database. It may also be due to undetected malware.

Some people report that opening MSE, performing an update check and turning off and back on again the MSE real time protection resolves the issue temporarily or permanently.

 

If the WMI repository is broken, the following may help:

We've seen a number of reports here of this behavior following the recent upgrade of MSE.

I suspect that there are multiple causes.

Some have reported that uninstalling and reinstalling MSE resolves it.

Others have said that forcing an update or running a scan with MSE each time it happens solves it after a few days.

Others have reported that turning real time protection of and back on has resolved it.

And still others indicate nothing fixes it.

The problem can also be due to a broken WMI repository (see below) and perhaps a resource issue -- that is, the PC is low on memory or running slowly due to many programs and processes starting with Windows causing MSE and the Alert Center to not communicate in a timely manner after which it gives up (either reporting status to WMI, or WMI querying the MSE status). Try a clean boot or disabling excess startup programs to see if that's your case. 

Here are procedures to correct the WMI repository that causes the Windows Security Center to report that no antivirus, antispyware, or firewall is active, yet MSE is actively protecting your PC and in green status.

 

When running this in Vista7, you must run the procedures as an administrator, elevated.

 

Start>Run>services.msc [enter]

Scroll down to Windows Management Instrumentation and double-click it.

Now click on the "Pause" button. Leave that window open and double-click

My Computer. Navigate to %systemroot%\Windows\System32\wbem (where

%systemroot% is the drive where XP is installed). Delete the Repository

folder and *only* the Repository folder. Now go back to the WMI service

window you left open and restart the service.

This will rebuild the Repository and hopefully straighten out the

incorrect entries for all your duplicates.

In order to see the Windows files, you may need to unhide them:

Make sure you are able to see all hidden files and extensions (View tab

in Folder Options).

Check "Display the contents of system folders".

Check "Show hidden files and folders".

Uncheck "Hide protected operating system files" and click "OK" to the

dialog box.

 

Method B from MVP Torgeir Bakken (more elegant)

Open a command window (Start/Run --> cmd.exe) and run the following commands:

 

net stop winmgmt

cd /d %windir%\system32\wbem

ren repository repository.old 

net start winmgmt

It may take a minute or so to complete while WMI rebuilds the database.

andreas killer say:

I attached a subset of the list of choices when I go to Tools->References from the Excel VBA editor.  Can a kind soul please say what are the proper things to check off? 

Well, any and all. :-)

You can use either DAO (Data Access Objects), RDO (Remote Data Objects) or ADO (ActiveX Data Objects).

DAO, RDO and ADO are data access interfaces ie. they are object and programming models used to access data. Earlier, DAO was used to interface with local databases (viz. MS Access), RDO was used to interface with large databases such as Oracle and SQL Server. ADO was their replacement to interface with all types of data sources.

For more details have a look here:
http://www.globaliconnect.com/excel/index.php?option=com_content&view=article&id=171:microsoft-access-activex-data-objects-library-ado&catid=79&Itemid=475

So for ADO you need "Microsoft ActiveX Data Objects" and when you have a look at the VBA references, you'll see a lot of that entries:

Microsoft ActiveX Data Objects 2.0 Library

Microsoft ActiveX Data Objects 2.1 Library

Microsoft ActiveX Data Objects 2.5 Library

Microsoft ActiveX Data Objects 2.6 Library

Microsoft ActiveX Data Objects 2.7 Library

Microsoft ActiveX Data Objects 2.8 Library

Microsoft ActiveX Data Objects 6.1 Library

It depends on your system if you see more or less of this entries, resp. which of these DLL are installed. So which one should you use?

That depends on where you want to use your file. Only on your PC or do you want to share it?

The technical question is whether you should use "Early Binding" or "Late Binding".

For you personal use only on your PC you can use "Early Binding", means choose the highest version and set a reference. Then write your code as usual, e.g.:

Function OpenConnection(ByVal Path As String, _
    Optional ByVal UserId As String = "SYSDBA", _
    Optional ByVal Password As String = "masterkey") As ADODB.Connection
  Dim Conn As ADODB.Connection
  Set Conn = New ADODB.Connection
  Conn.Open "DRIVER=Firebird/InterBase(r) driver;UID=" & UserId & _
    ";PWD=" & Password & ";DBNAME=" & Path
  Set OpenConnection = Conn
End Function

But when you copy this file to an other PC that hasn't the referred library (DLL) installed, your file did not work. The references in the VBA editor shows a broken link.

To avoid this problem, you must use "Late binding", means you have to remove the reference. But when you try to compile your code, you get errors, because the compiler did not know "ADODB.Connection" anymore.

To get the code back to work, you have to change all "ADODB.Connection" to the overall compatible "Object" declaration, and you must use CreateObject instead of New. But after that you loose the intellisense functionality, that is bad. The solution is to use a conditional define:

Function OpenConnection(ByVal Path As String, _
    Optional ByVal UserId As String = "SYSDBA", _
    Optional ByVal Password As String = "masterkey") As Object
#If Develop Then
  Dim Conn As ADODB.Connection
#Else
  Dim Conn As Object
#End If
  Set Conn = CreateObject("ADODB.Connection")
  Conn.Open "DRIVER=Firebird/InterBase(r) driver;UID=" & UserId & _
    ";PWD=" & Password & ";DBNAME=" & Path
  Set OpenConnection = Conn
End Function

Right-click a module of your file in the VBA project explorer, open the VBA properties and add this as argument for the conditional define:

  Develop = 0
When you develop your file, change it to
  Develop = 1
and set the reference to the appropriate DLL.

Be sure any module has the line
  Option Explicit

at the top, then compile your file.

You also have to declare all constants from the DLL that your code uses. So add a module for such declarations and add them like this sample:

Option Explicit

#If Not Develop Then
Public Const adOpenDynamic = 2
Public Const adLockOptimistic = 3
Public Const adSchemaTables = 20 '(&H14)
Public Const adSchemaColumns = 4
#End If

Look into the Object Catalog for the values of the constants.
