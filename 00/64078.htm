OK, I finally have an answer for this:

Firstly, my previous 'solution' did not work - I actually ended up contacting Microsoft and they gave me a brand new replacement keyboard and Surface 2 - and guess what - it did the same after a while.

Anyway - the solution is:

DO NOT USE THE POWER BUTTON ON TOP OF THE SURFACE2.

Just close the keyboard and DO NOT PRESS THE BUTTON.

When you want to reopen the keyboard and start Surface2, just pull the keyboard down - again DO NOT TOUCH THE POWER BUTTON ON TOP OF SURFACE2.

If you do this and avoid the power button, you will see that it lasts a long time when sleeping.

(What is happening here is that you press the power button to turn off the machine, which puts it into ConnectedStandby Mode, then you shut the keyboard, which actually brings it out of ConnectedStandby Mode and so it sits awake and drains all your battery - when you believe it's sleeping.  If you don't press the button and just close the keyboard, then the machine goes into ConnectedStandby Mode and stays in that mode, saving battery.

Why Microsoft Support doesn't know this is beyond ridiculous. 

There is hopefully a firmware update coming for this, but for now - AVOID THE POWER BUTTON!)

You can check the battery status by going to desktop, press windowskey+x, select Command Prompt (Admin), and then type: powercfg -batteryreport (then press enter)

Go to C:\windows\system32\batteryreport.html and view the file to see a breakdown of when the Surface2 was sleeping in ConnectedStandby Mode and 'active' (awake).

Hope this helps!

VirtualAlloc fails when using MEM_RESERVE with MEM_LARGE_PAGES. - Microsoft Community

jussi knuuttila say:

VirtualAlloc fails when using MEM_RESERVE with MEM_LARGE_PAGES.

When VirtualAlloc is called with the combination of MEM_RESERVE and MEM_LARGE_PAGES, the call always fails. According to web searches (in particular, http://blogs.msdn.com/b/oldnewthing/archive/2011/01/28/10121300.aspx), the reason for this is that the OS might not be able to guarantee that a 2 MB contiguous physical memory space can be found.

However, it seems undesirable to outright forbid this behavior on modern computers that can often have 8 GB or 12 GB of physical memory, which should mean that there should often be a reasonable amount of 2 MB pages available. Reserving a large amount of virtual space and only committing it as necessary is a highly useful technique, and its overhead is reduced by using large pages (not to mention that the memory itself should perform faster because of fewer TLB misses and associated page walks). Committing all memory up front is very undesirable because only a fraction of that memory might actually be required (normally, the OS only allocates committed memory when it's accessed, but apparently this is also disabled for large pages). On the other hand, using realloc() or similar techniques can invalidate any previous pointers to the allocated area, which makes them inconvenient for some tasks.

Additionally, once VirtualAlloc has been called once with MEM_RESERVE|MEM_LARGE_PAGES, all subsequent VirtualAlloc calls by the same program seem to also fail with an error, even if they would work fine without it. This seems likely to be a bug in the API.

Would it be possible to allow MEM_RESERVE without commit for MEM_LARGE_PAGES (and the associated MEM_COMMIT later) at some point in the future?
