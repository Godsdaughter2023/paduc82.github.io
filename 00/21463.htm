A MakeTable query will indeed fail the second time you try to create a table with the same name. Its intent is to create a new Table - and you can't create a new table if the table already exists!

Fortunately, you don't even need a MakeTable query. It's just exactly as easy to export a Query as it is to export a Table. You can just change your MakeTable query into a basic select query, save it (say as qryExport), and export that query using the TransferSpreadsheet method, just exporting qryExport rather than exporting tableFacThisMonth.

john w. vinson say:

Having SetWarnings off is a getaround that sometimes works - but it's a really dangerous one. You never know WHAT warning is being ignored! And it's never necessary. If you do need to re-make a table, simply delete it first and then make it; or - better - run a Delete query to empty it and then an Append query to repopulate it.

But your "cannot find object" error suggests that the query isn't there, or that it's still an Action query. Did you change it from a MakeTable query to just a SELECT query? It's really preferable to use the query directly; storing the data in a "scratch" table wastes time (since Access must fill in all the system tables defining the table and its fields, update indexes, etc.) and worse hastens the bloating of your database. Deleting or overwriting the table will NOT free the space it formerly occupied. You must regularly Compact the database if you use that technique!
