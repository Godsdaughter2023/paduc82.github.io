ken sheridan say: The term 'junction table' is a colloquialism for a table which models a many-to-many relationship type.  The relationship type can be binary, ternary, quaternary etc.  Scott's example of a relationship type between students, courses and instructors is a ternary relationship type, though in reality it would more likely be a binary relationship type between students and another binary relationship type between courses and instructors, itself modelled by a further table.

In the database relational model a relationship type is really just a special kind of entity type, and can be modelled in the same way by a table, though a unary relationship type is usually not of course, other than in situations where it is considered necessary not to have a Null foreign key.  In addition to the foreign key columns referencing the primary keys of the tables which are party to the relationship type, a table modelling a relationship type can have columns which represent non-key attributes of the relationship type.  A very common example is an OrderDetails table which models a binary relationship type between Orders and Products.  This would have a non-key column Quantity for instance.  Note that it would also have a UnitPrice column, even though the Products table would have a UnitPrice column.  This does not introduce any redundancy because the UnitPrice of a product is not time-independently determined by the key of  Products, i.e. the price will change over time, so the value of the column in Products is the current unit price of the product.  That in OrderDetails is the price at the time the order was made, so is functionally determined solely by the whole of the key of OrderDetails, which is a composite of the two foreign keys OrderID and ProductID.  Both tables are therefore normalized to Third Normal form, and in fact to the higher normal forms to at least Fifth Normal Form.

For an example of a simple binary many-to-many relationship type, and how it is represented in forms/subforms see ParentActivities.zip in the same OneDrive folder.  Note how, in the activities by parent form/subform a new row can be inserted transparently into the Activities table by typing the new activity name into the activities combi box in the subform, which executes code in the NotInList event procedure of the control.  In this case the insertion can be done transparently in this way because no other data than the activity name need be entered.  Where it is necessary to insert other data the NotInList event procedure would open a form bound to the referenced table in dialogue mode, passing the new value to the form via the OpenArgs mechanism.  You'll find an example in NotInList zip in the same folder.  In this the frmContacts form includes a city combo box whose NotInList event procedure opens a frmCities form, passing the new city name to it.  In the cities form the region (e.g. state) in which the city is located is then entered.

For the purposes of the demo it is assumed that all city names are district, which in reality is not the case of course.  A better solution in such a case is illustrated in DatabaseBasics.zip in the same folder.  In the section on 'Entering data via a form/subforms' a contacts form includes three correlated combo boxes in which the country, region and city are selected or entered.  This allows for cities of the same name, e.g. Paris, France and Paris, Texas, to be entered, and for the NotInList event procedure of the city combo box to be used to insert a new city name after entering/selecting the country and region in which the city is located.  The assumption made here for the purpose of the demo is that no 'region' will include two cities of the same name, which in reality is not the case (think Springfield in the USA).  In reality a further sub-regional level of the hierarchy (and a further correlated combo box), such as county in the USA, would need to be included to be reliable.  The problem does not arise in the UK, where, being a relatively small country, county is the 'region' in addressing terms.

ken sheridan say:

Have you looked at the ParentActivities demo file to which I referred you in my last reply?  That illustrates how a simple many-to-many relationship type is reflected in the user interface of a form bound to a query on one of the referenced tables, with a subform bound to a query on the 'junction' table which modes the relationship type.

 

In the subform the only control is a combo box bound to the foreign key which references the primary key of the other referenced table in the relationship type.  The combo box is set up to hide the numeric bound columns, and show the value from a non-key column in the referenced table.

 

If you wish to show other columns from the referenced table in the subform, then it can be based on a query which joins the 'junction' table and the referenced table, and further text box controls included in the subform, binding each to a column from the referenced table.  The Locked property of each of these controls should be set to True (Yes) and the Enabled property to False (No).  This makes them read-only.

 

To insert a new row into the referenced table the value of the relevant non-key column is typed into the combo box in the subform, and code in its NotInList event procedure either inserts the new row transparently, or where further data need to be inserted into the new row in the referenced table, opens a form bound to the referenced table at a new record and passes the value typed into the combo box to it.  Both of these scenarios are illustrated in the NotInList demo to which I referred you earlier.

Issues with high resource consumption can result from various sources, but these issues are most commonly related to the "on-access" scanning performed by real-time protection, rather than the "on-demand" scanning performed by the scheduled scan task. The Windows Defender Scheduled Scan task is actually triggered and controlled by Automatic Maintenance, which only allows it to run when the system is in an idle state, and then cancels it with any user activity (this is by design). Also, the likelihood of the scheduled scan task being responsible for any high resource consumption has now been further reduced by the fact that this scheduled scan no longer runs on its daily schedule, and in fact now tends to run rather sporadically and infrequently.

Since the problem is usually with Defender’s “always-on” scanning, you should first turn off real-time protection and see if that resolves the issue:

https://www.tenforums.com/tutorials/3569-turn-off-windows-defender-real-time-protection-windows-10-a.html

If real-time protection is determined to be the source of the issue, then the next step would be to remove any leftover third-party antimalware components, and also any undetected malware; since these are the most common causes of the high resource consumption issue. If that fails to resolve the issue, then the best course of action is generally to install a third-party AV solution, since this will disable Windows Defender’s real-time protection, but will still provide the option to run Defender in its Limited Periodic Scanning mode:

https://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_scanning/how-to-stop-antimalware-service-executable-from/a3a57d31-4687-43c0-b274-261da7d89245

Using an Out of Office automatic reply on certain days of the week - Microsoft Community

diane poremsky mvp (slipstick.com) say:

Using an Out of Office automatic reply on certain days of the week

I’m frequently asked how to set up an automatic reply (aka Out of Office or OOF) for specific recurring dates, such as every Thursday.  While the perfect solution would be for the calendar to set the automatic reply if you have an appointment whose Show time as is marked as Out of Office, it’s not currently possible. This means most users are limited to manually configuring automatic replies.

In a corporate Exchange environment (including Office 365 business accounts), there are third party utilities that can be used to configure Out of Office in advance for multiple dates and in bulk (for multiple users at once).  Unfortunately, this capability is generally limited to administrators, as end-users don’t have the permissions necessary to do this.

This leaves a lot of users looking for a way to easily configure automatic replies.

The recommended method is to use set the Out of Office autoreply up in advance to start at a specific time and end a couple of hours after you return. When you return to the office, turn off the automatic reply and immediately setup the automatic reply for the next date.

Although far from automated, it’s the safest method since you won’t have to worry about “mail storms” if the recipient also uses an automated reply as each sender will receive just one Out of Office message.  You can set up the automatic reply from Outlook desktop software or Outlook on the web.

These articles have instructions to configure automatic replies:

  • Send automatic out of office replies from Outlook
  • Send automatic (out of office) replies in Outlook on the web
  • Send automatic replies in Outlook.com
  • Send automatic Out of Office replies from Outlook for Mac

Rule to Reply from Server

A more automated method uses rules to send a reply from the server.  There is one problem using rules to reply to the sender:  The rule will reply to every message, which can be a problem if the sender is also using an automatic reply.

See Use rules to create an out of office message for instructions. 

You can limit when the rule runs using one of these methods:

  1. Create a rule to look for specific words in the message header, where the words are the days of the week you want the rule to run. Set exceptions for words that may indicate the message is an automatic reply. 
  2. Create multiple rules using the condition “Received after [date]” and “Received before [date]”. To reduce the number of rules you need, create a rule for each date this month and at the end of each month, update the rules for the next month.

Note that when you use either method, the rule runs on the calendar day, midnight to midnight, not from the end of a workday until the start of the next workday.

To run the rule after you leave the office until you return, you need to enable the rule when you leave the office and turn it off when you return. If you use Outlook desktop software and can use a macro, you can automate this step using task reminders. See Enable or Disable an Outlook Rule using Reminders for instructions.


If you'd like to see better options for configuring Out of Office automatic replies, I recommend voting for and commenting on these Outlook Uservoice requests: 

  • Set up Recurring out of Office Replies for certain dates.
  • Provide an option to automatically set an Out Of Office message in Microsoft Outlook 2016 if a meeting in Calendar is set to Out Of Office
