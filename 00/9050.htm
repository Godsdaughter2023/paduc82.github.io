ken sheridan say:

I'd also recommend that you declare the parameter.  It's always a good idea to do so, but especially so with parameters of date/time data type to (a) avoid the value entered being misinterpreted as an arithmetical expression, and (b) cater for regional variations in the system date format.  So the query would now be like this:

 

PARAMETERS [New date for update:] DATETIME;

UPDATE tblDates

SET dtToBeUpdated = [New date for update:];

 

A more developed solution would be to use a text box control on a dialogue form as the parameter.  By setting the Format property of the text box to 'Short Date' this will show the date picker when the control has focus, so the data can be inserted by selecting in the data picker.  The query can then be executed from a button on the form.  The query would now be like this:

 

PARAMETERS Forms!frmDateDlg!txtDate DATETIME;

UPDATE tblDates

SET dtToBeUpdated = Forms!frmDateDlg!txtDate;

 

With a dialogue form like this you don't really need to have a saved query at all; the SQL statement can be built and executed in the button's Click event procedure, e.g.

 

Dim strSQL As String

 

If Not IsNull(Me.txtDate) Then

    strSQL = "UPDATE tblDates SET dtToBeUpdated = #" & _

        Format(Me.txtDate, "yyyy-mm-dd") & "#"

 

    CurrrentDb.Execute strSQL, dbFailOnError

    MsgBox "Query successfully executed", vbInformation, "Confirmation"

Else

    MsgBox "No date selected.", vbExclamation, "Invalid Operation"

End if

 

Note the use of the ISO standard for date notation of YYYY-MM-DD to ensure international unambiguity.  The dates in the table will still show in your local regional format of course.

When you add a parameter to your query that Access cannot locate (i.e. not a field name or variable), it assumes that the unknown is an input parameter and uses the unknown as the prompt for an input box. In SQL this looks like,

UPDATE tblDates SET dtToBeUpdated = [New date for update:]

... where dtToBeUpdated is an existing field in the tblDates table.
