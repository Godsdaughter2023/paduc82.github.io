No, a trendline does not have to pass through the first and last point. The trendline is drawn in such a way that the total vertical distance from ALL points to the line is minimized.

If the trendline would pass through the 5 and 6 points, the distance from the other three points would be too large.

which is what you have described in the case of the maps table.  However, you also appear to have a column in Entries referencing Maps?  That column is unnecessary.

You said earlier "I've started the entry form and have each region at the top of the form. Now I need to have the subform allow me to pick from the various entries."  I assume from this that you have a form based on Regions in single form view.  Within this include a subform based on Maps.  Link the parent and subform on RegionID.

In the subform include a combo box bound to EntryID in Maps whose RowSource is a query which joins Entries to each of the three referenced tables, with EntryID as the first column returned, and the referenced text columns from each of the other tables as other columns.  Order the query however you want the rows in the combo box's list to be ordered.  Set the combo box up to hide the first column and show the other three in its list. 

The value from the second column in the RowSource query will show in the combo box when a row is selected.  Add two unbound text boxes to the subform, each of which references the other two columns of the combo box as theirControlsource properties, e.g.

=cboEntry.Column(2)
and:
=cboEntry.Column(3)

The Column property is zero-based, so these will show the third and fourth columns from the RowSource.

With a combo box like this which references a table which models a relationship type you cannot use the NotInList event procedure to insert a new row of course, so if you want to be able to insert a new row into Entries from the subform, you'll need a button whose code opens a form based on Entries in dialogue mode (which will pause code execution until the form is closed), and then requeries the combo box in the subform to show the new row in its list.

ken sheridan say: Opening a form to insert a new row into the Entries table is very simple; it only requires a few lines of code.  Where you put the button to do this in the subform is up to you, it could go in its header or footer, but what I've often done is to put a very small button with a +  sign on it at the end of each row in the subform, giving it a suitable tool tip.  The code for the button would be like this:

    DoCmd.OpenForm "frmEntries", DataMode:=acFormAdd, WindowMode:=acDialog

A label merge can use any manufacturer's label format, not just one of the listed ones, provided it uses a table the way Word does. Simply open the manufacturer's label document, or create a new document having a table with the required dimensions, then choose Mailings|Start Mail Merge>Labels, then press Cancel. You're now ready to complete the merge via Finish & Merge.
