THE INSTRUCTIONS I HAVE SEEN POSTED BY PAUL, WHICH WORK PERFECTLY FOR DOCUMENTS TO BE PRINTED, BUT I AM NOT QUITE GETTING TO WORK FOR EMAIL.

To embed an image in a document via mailmerge, you need to enclose the mailmerge field in an INCLUDEPICTURE field. Depending on your data setup, a field construction like the following can be used:
{IF {INCLUDEPICTURE {IF TRUE "C:\Users\My Document Path\Pictures\«Image»"} \d} {INCLUDEPICTURE {IF TRUE "C:\Users\My Document Path\Pictures\«Image»"} \d}}
{IF {INCLUDEPICTURE {IF TRUE "C:\Users\My Document Path\Pictures\«Image»"} \d} {INCLUDEPICTURE {IF TRUE "C:\Users\My Document Path\Pictures\«Image»"} \d}}
or
{IF {INCLUDEPICTURE {IF TRUE "C:\Users\My Document Path\Pictures\{MERGEFIELD Image}"} \d} {INCLUDEPICTURE {IF TRUE "C:\Users\My Document Path\Pictures\{MERGEFIELD Image}"} \d}}
After running a mailmerge coded this way, you'll have the correct, unlinked image for each record.

If the filepath for the images is held in the mailmerge data source, you could use:

{IF {INCLUDEPICTURE {IF TRUE "«FilePath»\«Image»"} \d} {INCLUDEPICTURE {IF TRUE "«FilePath»\«Image»"} \d}}
{IF {INCLUDEPICTURE {IF TRUE "«FilePath»\«Image»"} \d} {INCLUDEPICTURE {IF TRUE "«FilePath»\«Image»"} \d}}
or
{IF {INCLUDEPICTURE {IF TRUE "{MERGEFIELD FilePath}\{MERGEFIELD Image}"} \d} {INCLUDEPICTURE {IF TRUE "{MERGEFIELD FilePath}\{MERGEFIELD Image}"} \d}}

jay freedman say:

The orientation and other formatting of a section is stored in the section break at the so deleting the section break converts everything to landscape.

The solution is to make a third section by inserting a new Next Page section break at the end of the landscape page. Then:

  • Unlink its header(s) and footer(s) from the previous section. Make sure that any text in the third section's headers/footers matches what's in the first section's headers/footers, as this will be applied to the whole document later on.
  • With the cursor in the third section, click the little arrow in the bottom right corner of the Page Layout > Page Setup group to launch the Page Setup dialog.
  • On the Margins tab of the dialog, make sure the Apply To box is set to "This section". Then click the Portrait button and click OK.
  • Delete the section break between the second and third sections -- you should now have only portrait pages in two sections. Then delete the section break between those two sections. Done!

ken sheridan say:
OK, but I'm still trying to figure out why I can't enforce referential integrity for the relationship.

Referential integrity requires that for every row in the referencing table there must exist exactly one row in the referenced table with a value in the referenced key column which matches that in the referencing foreign key column.  So if you insert a value into the foreign key column of the referencing table, i.e. the CustomerNumber column in Orders in your case, then there must already exist a row in the referenced table which has that value as the key, the CustomerNumber column in Customers in your case.

If you for some reason need to include rows in Orders which don't reference an existing customer then the way to do it would be to set the DefaultValue property of the CustomerNumber column in Orders to zero, and to include a row in Customers with a CustomerNumber value of zero, but a Customer value such as 'N/A'.  You would then be able to create an enforced relationship on CustomerNumber.

However, I can't envisage any real-world scenario where you would want to do that, as creating an order with no customer wouldn't make a lot of sense.  This sort of thing is quite commonly done legitimately however, where a linear hierarchy is being modelled and there are gaps in the hierarchy.  Take this model for instance:

 Countries----<Regions----<Cities

Now while large countries will generally have a regional structure, e.g. US states, French départments etc, very small countries might not have any regional structure (Luxembourg?).  In order to maintain the mapping from a city to a country in such cases therefore it is necessary to introduce a row in Regions with a value 'N/A' or similar region for the country in question, whose foreign key CountryID column will reference the primary key of the relevant row in Countries, and whose primary key RegionID will be referenced by the rows in Cities for any cities in that country.
