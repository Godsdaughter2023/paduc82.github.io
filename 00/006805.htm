don cuthbert say:

There is a difference between normal Insider Builds and Preview Editions.  An Insider Build will not activate without a Digital License or Product Key for the VM and Product Keys are no longer provided as a part of the Windows Insider program.  I think the only available Preview of Windows 10 is Enterprise, so you could install Enterprise Preview and then bring it over to Insider.  I wonder if that would retain the Preview License expiry date, rather than it not retaining activation.

Depending on how long your work "Preview Build" has been installed on the same VM, the VM might have attained a Digital License for Windows 10 via the Insider Preview program that existed prior to July, 2015.  I was able to install and activate Windows 10 Technical Preview builds up to October 1st, 2015 and then upgrade and activate the 10240 RTM build on multiple VMs.  These VMs still retain a Digital License to clean install and activate any build of Windows 10 Pro today, even without any MSA login.

Another way to test Insider Builds is to use the Digital License of your physical device for activation, by installing Insider Builds on a small partition or on a VHDX disk which is still booted from your PC's System Partition (Native Boot).  The issue with Native Boot is that it will not upgrade from one build to the next, forcing clean installs to be done with each Insider build release.  But allocating a small partition for just the Windows OS works well as a dual-boot setup and will use your PC's Digital License to activate the Insider Builds.  You can offload all of the Apps and data to another physical (or VHDX) partition, if desired, by using the option in Settings for that.

banoz say:

SOLVED:

 

Change the following Registry Entries:

 

\HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\WebClient\Parameters\ [ServerNotFoundCacheLifeTimeInSec=0]

 

\HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\MRxDAV\Parameters\ [FileNotFoundCacheLifeTimeInSec=0]

 

\HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\MRxDAV\Parameters\ [RequestTimeOutInSec=0]

Yes DoCmd.RunSQL should throw an error if the SQL Statement doesn't go through. If you use a DoCmd.SetWarning False prior to running the RunSQL it will suppress error messages.

CurrentDB.Execute strSQL, dbFailOnError
