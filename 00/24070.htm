joeu2004 say:

I am unable to download the Excel that you reference.  But based on your description, I think I can explain the problem.

Even though I8 and I9 appear to be the same and I8=I9 even returns TRUE, the fact is:  the two values differ to an infinitesimal degree that Excel does not show.

To confirm, try =I8-I9=0 instead.  My guess:  that returns FALSE.  And in fact, =I8-I9-0 formatted as General displays an infinitesimal difference of the form 1.23E-16.  Read:  1.23 times 10^(-16).  (The redundant -0 defeats tricks that Excel might play to give the illusion that the values are "close enough" to being the same.)

We cannot see these differences because Excel formats only the first 15 significant digits.  Calculated values might have many more significant digits.

In general, the remedy is:  when you expect a calculation to be accurate to some number of decimal places,

For example, IF(10.01 - 10 = 0.01, TRUE) returns FALSE(!).  But IF(ROUND(10.01 - 10, 2) = 0.01, TRUE) returns TRUE as expected.

However, you mention that you have a "table".  If you are using a structured table object -- for example, a pivot table -- you might have less control over the calculation, at least by default.  I am not a PT person.  But I believe you might be able to work around the problem by using a "calculated field".  (Is that the correct terminology?)

PS....  It seems odd that a count of votes would have such problems.  But your problem descriptions fits the explanation above.  Please correct the problem with the downloadable example Excel file.  Test the download yourself.  Be sure to log out of the file-sharing website, when you do the test.  And choose a file-sharing website that does not require that we create a login or download some other software in order to download the Excel file.  box.net/files is a good site to use, in that respect.

It would not have mattered how many decimal places you displayed.  Excel formats only the first 15 significant digits (rounded), and the difference in B9 is beyond that.  Although B8 is exactly 12, B9 is 11.9999999999999,982236431605997495353221893310546875.

(I use period for the decimal point and comma to demarcate the first 15 significant digits.)

And yes, the mystery arises because Excel is inconsistent in the tricks that it plays to try to hide such infinitesimal differences. There is no good reason for that. The inconsistency is defective design, IMHO.

B8=B9 returns TRUE because for the "=" operator, Excel compares only the first 15 significant digits (rounded).

But MATCH(B8,B9,0) returns #N/A, indicating that they do not match exactly.  RANK.EQ does a similar exact comparison.

Likewise, =B9-B8 returns exactly zero, suggesting (incorrectly) that they are exactly the same, because

But =B9-B8-0 and SUM(B9,-B8) return about -1.78E-15, demonstrating that they are not really exactly the same.

I cannot help you further because you do not show the formula that B9 is copied from.

The best remedy is probably implemented in the source of the data.

Alternatively, you could create a helper column with formulas of the form =ROUND(B9,2), and use the helper column to calculate RANK.EQ.

(I round to 2 decimal places because you display only 2 decimal places.)
