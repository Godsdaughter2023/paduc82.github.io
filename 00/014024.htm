rob koch say:

I'll defer to Jose for some of this, since he asked the original question and so had some idea what he was looking for with that particular question.  I see that I misunderstood your comment about the missing information, so now you know a bit more about using the Defender interface.

Scans are static detections performed by the program actually looking at files, including the registry for known infection signatures.  A quick scan looks in locations that are known to commonly contain malware items while a full scan looks in all files on the main system disk and possibly other locations such as removable drives if you've included those in settings.

Full scans are long because Microsoft performs complete "deep scanning" during these rather than using the shortcuts many other security applications do to reduce scan time, since full scans aren't expected to be run unless malware is already suspected, so very rarely in most cases.  The 90% pause is commonly seen when malware has been detected, since Defender is spending time figuring out how to handle the potential for false positive detections or the removal process, which may include communicating with the Microsoft servers.  Be patient when this happens since it's just trying to be as effective as possible.

A Real-time detection is detected by various file system, firewall or other monitoring modules on the fly, while the activity is actually occurring which contains the suspicious items.  The firewall is only one of several possible monitoring locations, it's just more common that Real-time might detect something entering from the network while browsing for example.

The way to tell how the detection was made is to closely inspect the path information, since this typically starts with either "file:"; which is a filing system detection or "html:"; which is an [Internet Explorer] browser based detection.  I'm not sure what an actual firewall detection might be offhand, since I don't recall seeing one, but it would more likely be a behavioral type of detection where suspicious activity rather than a file was detected attacking via the network itself.

Microsoft doesn't specifically separate out the firewall as a protection method, since for most consumers the true technical information related to such attacks is highly confusing and so basically useless, causing mistakes when deciding how to react to the notifications.  For this reason, Microsoft instead collects and combines this attack profile information internally within Defender, since this allows a more complete picture of the attack profile to be developed that Defender can compare to its signature definitions to confirm a match to a specific malware.

The entire point when malware detections occur is that they've already "gotten through the firewall".  This is because whatever attack method was used, whether network (e.g. firewall), browser download, email attachment or some other form of socially engineered method that's tricked you into allowing access, it has already succeeded.  Once the malware passes through the network or hits the filing system, it's already gotten past the firewall which only exists to block any data not requested by something inside the PC such as the browser.

The reason Windows 10 is more secure than 8.1 or each successive version before it is other improvements to the security of Windows itself, not necessarily the firewall which is relatively simple and consistent.  Though many of the advertised improvements are more useful to businesses, some provide improvements to those already within Windows 8.1 relating to malware, though few details of these have been discussed in depth.

The key reason for getting Windows 10 for most is the model for maintaining the operating system in the future, since rather than service packs and further upgrades, all new improvements will be added to Windows 10 incrementally.  Think of these as "super service packs" which contain not only the previous security patches and other fixes, but also enhancements to the features as well.  Microsoft wants those with hardware sufficient to run it to move to Windows 10, since they will be focusing all new effort on this and leaving the other older versions behind.

rob koch say:

Since Jose hasn't responded yet I'll add my thoughts as to why these detections may not have occurred as you expected.  I'm still not certain you truly understand how al of this really works, but that's not surprising if you don't have a deep background in computers which most people today really don't.

The Windows firewall is nothing more than that, a program that either allows network protocols to pass through or blocks them based on configuration.  Though it does also contain protocol monitoring which collects information about this traffic, most of the analysis of this occurs elsewhere and there is no such thing as "firewall definitions", since the firewall makes no decisions regarding malware.

Instead, Windows 10 Defender and the associated Network Inspection Service (NIS, not Norton) each contain definitions for known malware or in the case of NIS, helps to protect devices against network-based exploits by using the signatures of known vulnerabilities to detect and block malicious traffic.  So unlike some other security products, the firewall is just that and nothing more and all of the malware and exploit protection are contained within either Defender or NIS.

With this knowledge, let's look at how that specific file detected by the Defender Quick scan may have been "missed" earlier.

file:C:\Users\RobbyStellarSeed\AppData\Local\Microsoft\Windows\Temporary Internet Files\IE\ZTLBY8XB\615[1].exe

First, note that the location of this file is within the user's individual AppData folder containing the Temporary Internet Files for Internet Explorer.  This is where the browser places the files it downloads from a website before display by rendering via the video driver.

This and any other files downloaded by the browser may be checked for malware, but since this is all happening in real time, it's possible they may not either detect or display a notification.  However, since most such items contained within a browser download should not include an executable file, these may be either automatically suppressed by the browser or the exploits they use blocked either by the firewall or NIS if this is a known vulnerability, none of which would generate a notification.

The result is that though a detection notification isn't displayed for the downloaded file, the actual attack attempt fails for any one of several reasons including the firewall, NIS or Defender detections or simply blocking within the IE browser itself.  This is why it's so important to keep both the operating system and browser up to date, since security fixes within these often result in such exploit attempts failing to work once the vulnerability involved has been fixed and patched.

Once the attack has failed though, the file itself may still remain in the Temporary Internet Files indefinitely depending how you have the browser configured.  Thus if you later perform a scan using Defender the file may be detected and flagged, though in truth the malware now has no easy way to be launched, since unless you revisit the same page and it requests the same file again the browser will simply ignore it.

Though this instance seems a bit strange, since typically a Quick scan only detects active malware and I'd expect to see this from a Full scan instead, in either case this file is simply a dead "leftover" from a failed attack attempt.  The reason I say this is because whether it was also detected at the time it was originally downloaded or not, it's not running anywhere else on the system currently so apparently that original attack had failed.

I can't say anything specifically about any other detections unless I have such complete information, but I wonder at those supposed rootkit detections by MBAM, since this is something that Defender would normally have detected itself.  Again, I can't really say without complete info and in that case it's even more difficult since different security applications often name or even label types of items differently depending on their own criteria, so comparison is nearly impossible.  Trojan, for example, is a popular term used by many products today attempting to stay relevant at a time when free products like Defender and a few others are taking away their revenue stream.

So though I can't give you a 100% certain answer, that particular detection itemized above wouldn't especially concern me.

The only thing I'd want to be sure of is that I'm keeping the system updated for both Windows and any applications from 3rd-parties, especially those like browsers or others used while browsing the Internet.  In fact, I specifically avoid all 3rd-party browsers and anything else I don't truly require in order to reduce the risk, since each added program simply increases the chance of successful attack by adding more vulnerabilities.

suzanne s. barnhill say:

In order to be included in a TOC, the style must either have an outline number of (for the default three-level TOC) 1, 2, or 3 or be specified in the TOC field code. There are several ways to accomplish this.

  1. When you create the style, in the Paragraph dialog, select an outline level higher than Body Text.
  2. When you insert the TOC, instead of just selecting one of the Automatic building blocks, use the Custom Table of Contents... menu item. This opens the Table of Contents dialog, in which you can click the Options... button. The Table of Contents Options dialog allows you to type an outline number beside any listed style, and that style will be included in the TOC at that level.

For more on creating and formatting TOCs, see http://wordfaqs.ssbarnhill.com/TOCTips.htm

But I would strongly urge you to avoid these problems by using Word's built-in heading styles. You can modify them to have any appearance you desire, and they have many "magic" qualities that make them preferable for more than just ease in creating a TOC; see https://shaunakelly.com/word/numbering/usebuiltinheadingstyles.html

kergga say:

When you use a GoTo like this, it skips everything inbetween and drops to your error handler. In normal practice, you would have an Exit Sub just before your error handler, otherwise it will also always run after the rest of your code. Your error handler throws a message box, so the message box will appear even if there is no error description (e.g. no error) to show.

Without going into depth on your code or what you are trying to accomplish, I think what you want is:

On Error GoTo Errorcatch

.

.

.

.

.

Exit Sub

Errorcatch:
MsgBox Err.Description
