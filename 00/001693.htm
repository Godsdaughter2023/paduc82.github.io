Unable to view the free busy status of the people to schedule a meeting in outlook 2010.
krishna b. reddy say:

I found the answer

  1. Download the Microsoft Exchange Server Messaging API (MAPI) Editor tool (MfcMapi.exe). To do this, visit the following Microsoft Web site:

    http://MFCMAPI.codeplex.com

    Start the MfcMapi.exe program that you downloaded, and then clickOK.
  2. On the Session menu, click Logon and Display Store Table.
  3. In the Profile Name list, select the online mode profile for the mailbox, and then clickOK.
  4. Double-click the Mailbox - <var>username</var> store.
  5. In the navigation pane, click Root Container.
  6. In the details pane, right-click PR_FREEBUSY_ENTRYIDS, clickDelete Property, and then click OK.
  7. In the navigation pane, expand Root Container, expand Top of Information Store, and then click Inbox.
  8. In the details pane, right-click PR_FREEBUSY_ENTRYIDS, clickDelete Property, and then click OK.
  9. In the Information Store: Inbox dialog box, click Exit on theFile menu.
  10. Exit the Microsoft Exchange Server MAPI Editor.
  11. Click Start, click Run, type outlook.exe /cleanfreebusy, and then press ENTER.

whycu say: The connectivity problem with N95 in power save mode is partly due to the N95 flawed implementation: it sends probe request with power save bit ON. That is, the phone is telling the AP that it is asleep so don't send it anything. But on the other hand, the phone won't try to associate if it doesn't hear back a probe response quickly... Therefore, if you are lucky to get the probe response in time, the phone will connect, otherwise it won't. The behavior will be random depending on the environment and traffic situation of the moment. Turing power save off solves the problem, but turning it back on won't fix the problem. Nokia needs to fix the implementation so we can have a reliable connection all the time.

mel77084 say:

Hi, Steve

The link that Ken suggested might be helpful to you is for a string of posts about a memory issue that could be what your problem is. It concerns the record of corrections that Windows kept from a class of memory known as ECC memory. If your computer uses ECC memory, your problem might well be the same as described in that string of posts. You can try the solution Jason used to fix his problem ... it won't hurt anything, and might allow your Win 10 installation to succeed.

To try it, you'll need to find your Command Prompt, right click on it, and, in the menu that opens, click on "Run as Administrator". Then, copy this command line, which is from Jason's post:  

bcdedit /deletevalue {badmemory} badmemorylist

Note that it has spaces, which are part of the command ... then paste it into the Command Prompt, and hit "Enter".

If you have ECC memory, it will delete the list of previous error corrections, allowing the Win 10 installation to see your memory as being OK, and proceed (if that was your problem). (And don't be surprised if the same problem appears again, later, requiring the same fix.) If you don't have ECC memory, this will have had no effect.
