john w. vinson say:

It absolutely WOULD make sense to move the backend to SQL. Access desktop databases are simply not suitable for use over the Internet; Access is extremely demanding of fast, stable connectivity, and that only happens when you have a wired (not wireless!!!) fast Local Area Network.

For remote access, either migrate the data to SQL/Server (or, sorry Microsoft, another good client-server database such as MySQL, Oracle, etc.) and connect your Access frontend to it, using ODBC for example; or use remote terminal software so that a remote user can log on to a computer in Houston and connect to the database from there. In any case - local, remote, or a combination - the database must be Split into a "backend" containing only the tables and a "frontend" containing the forms, queries, reports and code; each user should have their own private copy of the frontend.

daniel pineault say:

A couple more ideas to add to John's very valid comments.

CITRIX is a very nice solution to such a setup.

Have you though of using a Hybrid Office365 BE with a local FE.  Depending on the database, this could be a potential solution.

Do you use VBA?  If not, or if it could be rewritten to use macros instead, the an MS Access Web App would be ideal.

More complex, but copying data locally to a temp db to work with and then updating the record(s) in the master db could also be an alternative solution.

As for setup, Access over a WAN (including wireless) is asking for trouble and corruption. 

As with any db, Split is critical, as is establishing a persistent BE connection as soon as the FE is launched.

As noted, latency issues will not be solved with drive mapping.

And keep in mind you looking at internet speeds. You very cheap low cost office network runs at 100 (still 10 or 3 times as fast). And today most companies are running 1000. So your internet speed is either 100 or 30 times slower!

So something that takes say 3 seconds can now take 90 or 300 seconds (1.5 minutes or 5 minutes) depending on your work internet or home internet.

So while they are great internet speeds, compared to a typical and cheap local network running in your office, they are both MAJOR slower speeds then what you running local.

So while it is interesting to compare home internet and work internet speeds, the REAL comparison needs to be against how you running the Access application now on that local office network, and then over that VPN.

As noted, WAN/VPN speeds are much better today then say 10 years ago. However, they still tend to not play well with a split database and tend to be quite slow anyway.

So you back to Terminal Services and other technologies choices.

Okay, so about the columns: What I need is for Column Left to not wrap up to Column Right when it hits the end of the page. Instead, it needs to automatically create a new page, and continue in Column Left. Over and over again. Same with Column Right.

When I do footnotes with a table, it puts them outside table itself, and since the table is multiple pages, if the footnote number is on page 2, and the table ends on page 80, the footnote text will appear on page 80.

Also, I don't care if the footnotes get split up between Column Left and Right. I just need them at the bottom of the same page as their corresponding number. They can be inside the table or out.

Is there any way to accomplish this without manually breaking the column after every page?

doug robbins - mvp office apps & services (word) say:

You need to be aware that Word does not like tables with rows that extend over many pages, and I think you would definitely run into problems with a one row table that spanned 80 pages.

As far as the footnotes are concerned, you can achieve something like it if you add the text of the "footnote", including a second "footnote reference"  (index) into the table itself, using a particular style for that text, with the font color set to white. Then use a StyleRef field in the footer that references that particular style.  

You will need to manually number the "footnotes" and the "footnote references"  

It will not be possible to have the "footnote" index in the footer appear as superscript, so it might be best to insert the "footnote" indices as number inside parentheses

If you have multiple "footnotes" on a single page, you will need to define a separate style for each, and to prevent the same footnotes from appearing in the footer of subsequent pages, you will need to have an instance of each of those styles on every page, that just contains a space, if it is not required for a "footnote".

Here's an example of a "footnote" created as detailed above:

You will probably find it easier to start with the color of the font used in the styles set to Automatic, and then when you have finished creating the document, modify the styles to set the color to white.

suzanne s. barnhill say:

To add to what Doug has said, a footnote will appear at the bottom of the page on which the row in which the reference appears ends. Most footnote references fall at the end of a paragraph (or at least at the end of a sentence). If you start a new row following the end of the paragraph containing the footnote reference, the footnote will be on that same page.

What you will find if you extend a single row over many pages and have a lot of footnotes is that Word will try to put all the footnotes on the page where the table (or row) ends. When it runs out of room on that page, the footnotes will just disappear.
