ken sheridan say: When closing the calling form, passing the value to the second form via the OpenArgs mechanism is generally the favoured approach.  Where the calling form remains open you can simply reference it as the ControlSource property of an unbound text box in the second form.  This is commonly done when using a login form, in which case the login form is hidden rather than closed, by setting its visible property to False in the code.  You'll find examples in SecDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates means of implementing 'row level security' and has examples using both auto-login, in which the user's Windows login user name is automatically used, and a simulation, for purposes of the demo, of manual login.

BTW you'll also find an example of more flexible use of the OpenArgs mechanism in the same OneDrive folder, as Args.zip.

The acSaveNo constant is only used if design changes can be made to the form by the user, and you do not wish these to be saved on closure.  It has nothing to do with saving changes to the data, which is automatic when closing a form or moving off the current record.

You most likely have OneDrive installed on your phone, and this solution assumes you do.

3. Toggle the switch labeled "Clear Account Settings"

4. Close all associate OneDrive connected apps, including Office Lens.

5. When you reopen Office Lens, sign back into your OneDrive account within the app

Note that each section may have up to three different headers (and three different footers). This is the case if you have enabled both "Different first page" and "Different odd and even" in Page Setup. In other words, make sure that you are unlinking the correct header. To be clear, the unlinking applies to the particular type of header in the particular section.
