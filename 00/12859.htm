Hyperlinks are sometimes stored as relative to the folder that contains the workbook.
When you move the workbook those relative hyperlinks break.

Try moving the workbook back to where it was before and setting File > Properties > Summary > Hyperlink base to\\NoServer\Nofolder  or some other non-existent location.  Then the hyperlinks you create will be relative to that location - in other words they have to be absolute paths.     Non relational and relational databases? - Microsoft Community

    james lewis kent say:

    Non relational and relational databases?

    I need some help understanding the difference of a non-relational database and a relational database? 
    I need some help understanding the difference of a non-relational database and a relational database? 
    Also, how does it compare to a flat table? 

    marshall barton say:

    A flat table is not relational.  Most spreadsheets are essentially flat files and they are very difficult to search and manipulate.

     

    You can google for Database Normalization and get a ton of hits that explain in more or (lots) less understandable language what the difference is.  I like to sum it all up with something along the lines of --  When you need to edit a value, you should only have to edit one field in one recor in a single table.  Think about it  ;-)

     

    Another important point is that tables and rows are a whole lot less expensive than columns.  This means you never want a table to look like a spreadsheet where you have columns such as Jan, Feb, etc.

     

    Also, calculated values are the cheapest thing around because they should not bi in any column, row or table.  That's not to say that calculated values should not be used, but that they should be (re)calculated anytime you want to display them in a form or report.

    scottgem say:
    A non relational database is basically a flat file. It will have one table with all the fields in. Data is redundant and data entry cumbersome.

    A relational database is one that is designed to eliminate redundancy, make data entry easier and make data analysis easier.

    ken sheridan say: The relational model is a way of representing the underlying reality in terms of relations, tuples and attributes.  Loosely speaking, in the language of the relational model, a relation equates to a table, a tuple to a row (record) and an attribute to a column (field).  Strictly speaking the term relvar (relation variable) should be used in this context rather than relation, as the latter really refers to a relation value, i.e. the sum total of the information content of the relation.

    The model was introduced by E F Codd in 1970 and applied relational mathematics (originally relational algebra, but later also relational calculus) to the model of an underlying reality expressed in the above way.  These mathematical operations were expressed in SQL, which is the language used for querying the data, so we don't have to worry about the underlying maths.  The beauty of the database relational model is that because it represents the reality in a formal theoretical structure which mirrors the real world structure, provided that the model is built correctly in terms of the tables and the relationship types between them, it will act reliably in the same way as the real world mechanisms which it is modelling.  

    In a relational database each table models an entity type, and each column of a table represents an attribute of the entity type.  Each row in a table represents one entity of the entity type in question.  It is important that each attribute is specific to the entity type, e.g. in a table of contact addresses you might have a column for the City, but you should not have a column for the State.  This is because City determines State, so to include the latter in the table would introduce redundancy which leaves the door open to inconstant data, there would be nothing to stop San Francisco being in California in one row and Colorado in another.  From the database per se there is no way of knowing which is correct.

    The column to represent the city in the contacts table would in fact be a numeric CityID foreign key column as city names can legitimately be duplicated.  This would reference the CityID primary key of a Cities table, which would also include City and State column.  The State column would be a foreign key referencing the primary key of a States table.  In this case the state name can be the key as state names are distinct, though there is no reason why a numeric StateID can't be used as the key.  If so, it's important the State column in States be uniquely indexed, as it is a 'candidate key' whose values must be distinct.

    The relationship between the States and Cities tables is one-to-many, as is that between the Cities and Contacts tables.  Relationship types can be many-to-many, however, e.g. An Orders table would be related many-to-many to a Products table, as each order can be for one or more products, and each product can be in one or more orders.  This is known as a binary relationship type as it is 2-way, but a many-to-many relationship type can be 3-way(ternary) , 4-way(quaternary) or more.  The relationship type between Orders and Products would be modelled by an OrderDetails table with two foreign key columns OrderID and ProductID referencing the primary keys of Orders and Products respectively.  The primary key of the OrderDetails table is a composite one made up of both columns, which in combination must have distinct values of course.  The table would also have non-key columns such as UnitPrice and Quantity representing attributes of the relationship type. An relationship type is really just a special kind of entity type.

    To retrieve data from the tables they are joined in queries on the primary/foreign key columns, e.g. a Contacts table would be joined to a Cities table on CityID.  Criteria can be applied to the query to restrict its results, e.g. all contacts in New York.  More complex operations on the data can be carried out in a query by aggregating data, e.g. A query could be based on Customers, Orders, OrderDetails and Ptroducts tables, grouped by Customer and Product, summing the UnitPrice*Quantity and restricted by criteria on the OrderDate column to a particular data range to return the total amount spent per customer on each product ordered within the date range.

    A non relational flat-file database on the other hand would be akin to a spreadsheet, with values such as customer and product names repeated many times.  This allows a lot of scope for errors in data entry, and makes it much more difficult to get the sort of results which a query of the type described above can do.

    The relational model has proved a very robust tool, particularly for business applications, over the 42 years since Codd introduced it.  During that time it has undergone much development, on which there is a vast body of material in the literature.  It's not necessary to concern yourself with this, however, but a reasonable general understanding of the theoretical basis of the model is of great benefit, not least in avoiding the common pitfalls which newcomers to relational database development can easily fall in to.
