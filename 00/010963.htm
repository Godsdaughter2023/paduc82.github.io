The location where the new Microsoft Skype for Windows 10 application saves the chat history is different from what was the case with the classic Skype for Windows Desktop.

 

With the classic Skype (the 7.X version), the chats were saved in the main.db database file.

 

%appdata%\Skype\<Skype Name>\main.db

 

Microsoft Skype for Windows 10 (the 12.X version) saves the data in the skype.db database file:

 

%localappdata%\Packages\Microsoft.SkypeApp_kzf8qxf38zg5c\LocalState\<Skype Name>\skype.db

pamcaswell say:
When Word detects errors that could make it unstable and cause loss of data, it will repair what it can but some data losses may happen.  In recent years,  Word's repairs have been so good that I rarely lose anything, except an occasional custom style. In my experience, though, repairing the errors does not necessarily eliminate their cause. So, since this is happening so often, I suggest you try to remove the problems from the document yourself. 
 
One way is to copy  all but the last paragraph mark in the document and paste  it into a new blank document as unformatted text.  If the document has section breaks, you should do this section by section copying everything in the section except for the break itself.   You need to  reformat the  document., recreate tables, and reinsert all images and objects, cross references and such.   If any of the  images and objects were earlier deleted by Word, insert fresh versions of the files.
 
Another way is to  clean up the document in place--delete unused styles, delete any unlinked (char) styles, clear then redo auto numbering, simplify formatting where possible, replace all images, etc.  Then copy all but the last paragraph mark into a new document.

When you are done, or even along the way, use Open and Repair to make sure the document is stable.

The related threads link under the posts on this page explains how to disable the show repairs message.  But because you are sharing  the document with others, it probably is not useful for you.


Good luck,
Pam

djsteele, mvp say:

So long as your users can enter the date in the format to which they're accustomed, and the application will display it to them in the format to which they're accustomed, why do you care what you need to do in the VBA code? Your users will never see it... Simply change the format in your call to your Format function, and everything will be fine. AFAIK, the CDate function respects regional settings, so you might want to wrap the text box in that function as well:

 

rst.FindFirst "[Date] = #" & Format(CDate(Me.Text1), "mm/dd/yyyy") & "#"

 

or

 

rst.FindFirst "[Date] = #" & Format(CDate(Me.Text1), "yyyy\-mm\-dd") & "#"

Problems in accessing a shared mailbox from 4 different computers - Microsoft Community

jamalbdeir_msft say:

Problems in accessing a shared mailbox from 4 different computers

Hi Team,

One of our customers has 4 employees using 4 different computers but require to access one shared email ID through their desktop outlook application in cached mode.

The mailbox size of this email ID is roughly 30 GB. Whenever they start outlook from any PC, the emails are downloading again into that machine and messing up their indexing.

Can you please advise on the most suitable solution to address this problem?

Should those 4 employees have their own email licenses configured with their outlook application and then assign the shared email ID to those mailboxes to avoid these issues?

Regards,

Jamal

larry yu msft say:

Hi Jamal,

 

Thanks for your reply.

 

Our suggestion is buying 4 individual licenses for those 4 employees and then granting Full Access permission of the shared mailbox to them. After that, the shared mailbox will be auto-mapped to the Outlook client after they configure their primary mailbox on it. Please note it may also need some time to load the shared mailbox since the size of it is roughly 30 GB.

 

Regards,

Larry
