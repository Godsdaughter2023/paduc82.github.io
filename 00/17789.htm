SharePoint Sync and mass file deletion

Hi all, 

I am the Systems Administrator for a company that has recently adopted O365 as its base application and data storage solution.

Multiple users Sync with SharePoint for offline access to libraries and document stores.

The issue I've run into, and after reading some of the other threads in this community others have as well, is that when a user who has an entire document library sync'd with SharePoint moves or deletes a folder, or multiple folders, on their local drive it then replicates that deletion in SharePoint. This is obviously not ideal, as having users of various technical knowledge to deal with some understand that the local stub shown "seemingly as a local file structure" is in fact an image of the directory or directories they have sync'd with on SharePoint.

So some will, and have, begin to restructure their folders to their personal liking not realizing that they are manipulating data in the cloud when they do so.

Worse yet, if they move the data from their sync folder to their My Documents folder for example, it has a catastrophic effect as SharePoint sees this data as having been deleted and will then naturally remove it from the cloud.

This is obviously correctable by restoring from the sites online recycle bin, but I have found that that can be problematic also if the user is actively syncing, the restored files simply get deleted again unless the sync on the local machine is stopped.

The problem there of course is tracking down exactly which machine is responsible in order to stop the sync in the first place.

So, what I have done is to manipulate the retention policies of all of our sites so the recycle bins will maintain deleted files and folders for a long enough time that mistakes like this can be corrected.

I have also created a new permission set called "Contribute-NoDelete" which I assign to the majority of users of a site. This allows users to perform all the actions Contribute permissions would allow but removes the ability to Delete files/folders from a site or library. This has created some issues for some users as the ability to move data from one folder to another, etc. is now curtailed by all but managerial or IT level personnel who have been granted Contribute or Owner level permissions to the site.

This has now, unfortunately, added to the manager and/or IT's work load when directory structure manipulation is required. So this is not overall an ideal solution.

User education has been a main focus for me of course, however in a comapny with hundreds of employees it is a challenge to ensure everyone has received the proper training. Also, as in all things, people are human and mistakes happen, so even educated users can delete entire data stores and/or entire shared document libraries with little more than a press of a button. It seems a serious oversight in design of SharePoint and it's Sync functionality to me at this point.

So my questions are:

Are there others dealing with this issue that have implemented different strategies than my own to maintain the data integrity of your SharePoint sites with as little IT team involvement as possible?

Is Microsoft looking at this issue and developing tools or global settings that will make this less of a workload for IT teams to manage?

rich007a say:

Hi,

As you know, it's always been that way.  Some good explanations here: http://discuss.fogcreek.com/newyork/default.asp?cmd=show&ixPost=3479&ixReplies=4

 

I particularly like this explanation:

 

The official reason is that Excel doesn't really have cut and paste, it has move and copy. That's necessary because Excel automatically does reference fix up. For example, if cell A2 is defined as =A1, and you move cell A1 to A3, cell A2 will be updated to =A3.

If Excel actually cut things to the clipboard you would somehow need to have a reference pointing >into< the clipboard which is bizarre and for which there is no reasonable syntax. In other words, Excel doesn't want to leave you with dangling references during a move operation and isn't confident that it would be able to fix them up correctly when you completed the move by selecting "Paste."

Joel Spolsky
Fog Creek Software
Tuesday, March 09, 2004

 

That said, there are also some pretty good arguments about how it should be changed.

Hope that helps.

Font download Security Risk? - Microsoft Community

pefp say:

Font download Security Risk?

Hi,

I’m just curious why “Font download” is in the Internet Explorer Security Policy settings. Are there any security concerns if Font Download is enabled, if there are, can you please specify the security risks.

intelligent primate say:

In a true-type font, each character is a tiny computer program . Rendering a font requires executing these programs in a special environment and is frequently done in a privileged part of the operating system called the kernel. Because font rendering is so complicated, this part of the kernel contains undetected mistakes by the programmers. Sometimes these mistakes are exploitable in a way that gives the font program the same privileges as the kernel - which is carte-blanche to do anything they want to the computer. This has happened on numerous occasions. For example, in 2011 and in 2015. See this stackoverflow question for a bit more detail.

A couple of points for clarification :-)

  • Objects cannot be inserted into cells in any version of the program. Supported object types are are inserted onto a worksheet, but not into individual cells.
  • Microsoft Office 365 is not software - it's Microsoft's marketing moniker for their subscription service. The Office version being used varies depending on OS version as well as personal preference. On a Mac it currently would be either Office 2016 for Mac or Office 2011. To determine your version go to the program name in the main menu & select About program. Or, select the program in your Applications folder. According to your screen shot you're using Office 2011, but I can't be positive. Accurate version info can make a definite difference.

All that as it is, the short answer to your fundamental question is that Object Linking & Embedding in all versions of Mac Office is significantly less robust than what is offered by its Windows counterparts. Those object types you see listed are the only file types that are supported... and even that has been reduced in Mac Office 2016. [The old Equation Editor & Chart Editor are no longer supported.]

In Excel 2011 you can use Insert> Photo as described in my Answer to a similar question: How do I insert a pdf into excel

As Bob G suggested, you also have the option to insert hyperlinks to external files. In that case, though, the linked files are not stored [embedded] in the workbook, so if a copy of the workbook is forwarded to another user the linked files have to be forwarded as well. That takes some prior planning.

bob jones aka: cybertaz say:

OK, thanks for the version confirmation, but everything remains as stated... PDF is out of the question. As I wrote, PDF is not a supported OLE file type in Mac Office although it is supported by Windows Office. Again, OLE is a Windows technology which Microsoft never has fully implemented in Office for Mac.

Your other option should be doable, though - Using a Word document instead of a PDF. What you haven't clarified is what it is that is preventing you from doing so :-)

If you have an existing Word document the steps are:

  1. Insert> Object
  2. Click the From File button
  3. Navigate to the file's location & select the file
  4. Check the Display as Icon box
  5. Click OK

The icon will appear on the active worksheet bearing the actual name of the document as shown in this screen shot:

If you do not have an existing document but want to create one that is stored only within the Excel file (not as a Word document in a folder):

  1. Insert> Object
  2. Select Microsoft Word Document
  3. Check the Display as Icon button
  4. Click OK

A new document will open in Word. Enter your content then close the file. You will not be prompted to Save nor for a location & it will simply be identified as Document on the worksheet.

If you're incurring any sort of roadblock with either method please provide complete descriptive details about what is happening.

In either case the embedded object (icon) is not inserted into a cell. It floats on the surface of the worksheet. I believe that what Bob G may have been trying to relate is that even though you can do this you wind up with a floating graphic object for each record in the list. If there are more than just a few records that can become a management nightmare because the document icons cannot be made a part of the record data content. That's why he suggested using hyperlinks instead. Hyperlinks are stored as textual data in cells, so they are a part of the record. Also, hyperlinks can point to any type of file & will work as long as an associated program is available.

petrib say: Probably too late now, but did you use the latest version of PC Suite to back up your phone?

Also, when you did the backup, did you check the log file that the PC Suite backup program creates for any errors/problems making the backup? If no errors backing up contacts were reported, then they should be in the backup.

A second way to keep a "backup" of the contacts is to use the PC Suite Contacts app and synchronize the contacts to a PC that way.

A third way is to use the PC Sync app of PC Suite and synchronize the contacts with, e.g., Microsoft Outlook or Lotus Notes.

A fourth way is to use the "Memory" app on the phone to backup the contacts, messages, etc., to memory card (and then use the same app to restore them).

A fifth way is in the "Contacts" app to hit "Options" and then choose "Copy" and then "To memory card" to save/export the contacts to to a file on the memory card (and then use "From memory card" in the same menu to do the reverse).

A sixth way is to use the "Transfer" app on the phone to keep it it synch with another phone.

A seventh way is to use the "Sync" app on the phone with a remote, network based SyncML Server.

I use the PC Sync app as my primary means, but I also use the PC Suite backup (Content Copier) often + just in case also the PC Suite Contacts app.

Less regularly I use the sixth way & the seventh never, really.

I suspect all of these are just turning the knife in the wound now, but maybe they'll help you in the future, even if you can't recover from your lost contacts now.
