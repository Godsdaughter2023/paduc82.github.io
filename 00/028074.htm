shane devenshire say:

Run time error 1004 is a generic error.  When you get the error and click Debug, which line is highlighted in yellow?

Most likely - The macro was already run and all the blank rows have been deleted, therefore there are no rows to select or delete, hence the error.

One way to handle this would be:

    Range("Table16[Tax Lot]").Select
    On Error Resume Next
    Selection.SpecialCells(xlCellTypeBlanks).EntireRow.Delete

This is not the best solution but it should work in your situation.  There are much more sophisticated ways to trap errors, but lets not get too fancy, but here is another:

    

    Range("Table16[Tax Lot]").Select
    On Error Resume Next
    Selection.SpecialCells(xlCellTypeBlanks).Select
    If Err = 0 Then

        Selection.EntireRow.Delete
    End If

ken sheridan say: The only obvious error in your code which I can see is in the final line.  You either need to include an additional comma, i.e.

DoCmd.OpenForm strForm, acFormDS, , "FID=" & Me.FileID

or better still, explicitly name the arguments:

DoCmd.OpenForm strForm, View:=acFormDS, WhereCondition:="FID=" & Me.FileID

The code would be entered in the Click event procedure of a button in the form.  The button can be in the detail section, in which case it will appear in each row of the form, or in the form header or footer, in which case it will appear only once.

If you put the button in the detail section it should open one or other of the forms filtered to the FileID value in the row in which the button is clicked.  The form opened will be dependent on the value of the combo box.  If the combo box is a bound control in the detail section, the value will be that in the current row; if it's an unbound control in the header or footer, the value will be that of the single instance of the control.

If the button is in the header or footer the form will be filtered to the FileID value in whatever is the currently selected row in the form.

Your code does assume that the FID and FileID columns are each of a number data type, not text.  If they are of text data type the value needs to be wrapped in literal quotes characters:

    "FID= """ & Me.FileID & """"

Your code also assumes that the BoundColumn property of the combo box is the visible text column, not a hidden numeric key column.

If you are unfamiliar with entering code into a form's or control's event procedures, this is how it's done in form design view:

1.  Select the form, section or control as appropriate and open its properties sheet if it's not already open.

2.  Select the relevant event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new lines between these.

You need to understand the difference between working time and calendar time. By default the Standard calendar shows a 5 day work week. If you set the duration of a task at 10 days, that task will actually span 12 ,or possibly 14, calendar days depending on how many weekends fall in that duration.

You need to understand the definition of "1 Day". Since "day" is ambiguous MSP always uses a conversion definition to turn it into something which is not ambiguous, ie hours.
