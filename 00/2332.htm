rob koch say:

Since you've confirmed that you have sleep or hibernate disabled then the most common reason for a system shutdown during an antivirus full scan is overheating, resulting in the computer shutting itself down to protect the processor from damage.

This can obviously be due to dust or failure of the cooling system to operate properly, including less common issues such as heatsink compound that's not effective or even configuration of the processor speed-step or other technologies which allow it to reduce performance to lower power consumption as well as heat.

Less obvious issues can be due to badly designed cooling or any of the already mentioned processor protection systems, including things like the BIOS portions which control these functions, though these types of design issues are relatively rare inmost modern devices from major manufacturers.

As for why the scans are taking so long, the most common reasons along with the again obvious quantity of files, include large numbers of compressed files such as app installers or image files, since these must both be exploded and then the individual files scanned.

It's also important to understand that MSE typically uses whatever built-in applications are associated with particular files, such as WinZip for example, to perform the scanning of that file type.  So if that particular app is relatively inefficient it can lead to long scan times for the file type due to high processor utilization, which is one reason among many that I've always avoided 3rd-party applications for major file types where Windows already contains an association for handling them.

There are other more technical reason for long scan times of course, including malware, but if the two systems are otherwise similar in the time spent scanning, it's less likely that any of these other issues exist.

Rob

rob koch say:

Disregarding potential causes doesn't solve your problem John, you need to perform testing to determine what's truly happening if you wish to resolve the problem.

I can tell you that nearly all such shutdown issues I've ever seen in these forums have been due to overheating, though a much smaller number have been related to malware attacks.

It's common for people not to believe that their systems can't possibly be affected by overheating, but quite a few of these have returned later to post that some seemingly strange issue like badly installed processor cooling adhesive or misconfigured BIOS settings were resulting in their problems.

Modern AV programs like MSE are notorious for causing high processor utilization, since they perform their operations like exploding compressed files in memory rather than being disk bound as in the past, placing far more stress on the processors.  Unless the cooling system is properly designed to both dissipate this heat, as well as step the processor(s) down to lower speeds via BIOS configuration, this can quickly cause overheat.

If the folders containing those zip archives are encountered near the end of the scan, it may be worth configuring a manual scan for these, starting the scan within the higher level documents folder if that's where they're contained in order to build a more normal level of heating during the scan before the zip files are encountered.

Monitoring the processor heating during this test might be a good idea, but note that this very act can result in reduced heating due to monitoring delays, so I'd begin by simply testing those personal folders, as they're often the last items added to a system and so the last enumerated and scanned by MSE.

Also note that I've seen a few threads here recently that seem to indicate an increase in scan times for MSE on Windows 7, though not a large enough number for the problem to be considered common.  It's possible that whatever is contributing to this issue is being exacerbated by recent changes within the MSE scanning engines, displaying only on systems with specific potential malware or even past file system damage that contributes to processing overhead.

Rob

davidmorr say:

Mail merge catalog layout

I'm trying to make a catalog using mail merge in Word 2011. I want to have a logo and description at the top of the first page, which I have done by setting up a header specifically for the first page, and a header for subsequent pages which is a single line title and page number.

However, I also have some information I want to be included at the end of the catalog entries, information on how to order, etc. Is there any way to set up the merge so that this can be added automatically after the last entry? Or is that something I have to do manually after creating the new merged document?

Thanks

David

a. user say:

AFAIK there is no completely simple way to do it, e.g. just using fields, unless there is something in your data that indicates that a record is the last record to be merged.

You can do it using VBA, using a macro to execute the merge then insert the text you want (either directly from the VBA, or by inserting it from a file).

If you have some kind of indicator (suppose the merge field "last" is only "Y" in the last record), you can use a nested field like this:

{ IF "{ MERGEFIELD last }" = "Y" "the text you need, with formatting, but avoid putting double-quotes in it" }

or e.g.  
{ IF "{ MERGEFIELD last }" = "Y" "{ INCLUDETEXT "the path name of a file containing your text" }" }

Otherwise, the main problem with using fields is knowing that you have reached the last record. You can test for the first record easily (because you know { MERGESEQ } will be 1) but not the last one, because there is no field for { MERGERECORDCOUNT }. 

If you knew you were only outputting one record per page, you might hope you could compare { PAGE } with { NUMPAGES }  and output some text only if they agree. But unfortunately, as the merge proceeds, { NUMPAGES } and { PAGE } are always the same. But what you can do is rely on the fact that a calculation involving { NUMPAGES } and { PAGE } will survive as a field in the output document. So if you have a field like
{ =SIGN({ NUMPAGES }-{ PAGE }) } 
in the merge, you will end up with that { = } field for every record. You can then select the output document text (cmd-A), and update all the fields, leaving you with a value "1"  on every page except the last, where it will be "0". What that allows us to do is to use a field like

{ INCLUDETEXT "the name of a file containing your text" "bm{ =SIGN({ NUMPAGES }-{ PAGE }) }" }"

In your file, you bookmark the entire body text using "bm0" and you insert a point bookmark called bm1. You then
 a. do the merge
 b. select the output text
 c. update all the fields

But what if you are outputting more than one record per page? Well, then the problem is identifying the final record. I think you can only do it if the number of records per page is known (i.e. in all probability, the same number per page). Then you can modify the { = } field to take that into account.

My own view of the above is that in most situations it would be simpler to use VBA to do the same thing, or to insert the final chunk manually (or perhaps using Applescript, if that's your thing).
