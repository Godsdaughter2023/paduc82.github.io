It is done by setting the colorization color.  The hex value is not as you would expect from COLORREF.  It is 0xaarrggbb where rr = red level, gg = green level, and bb = blue level.  Just the opposite of COLORREF. 


However the aa is not alpha as a reversed COLORREF order would seem to require. 


For aa = 0x1A it is 0% intensity of the color.  For aa = 0xD9 it is 100% intensity of the color. 


The color of the window caption bar and border is computed as follows:


Oops, I made a mistake

red = 0x1A + rr * 100 * aa /0xD9

green = 0x1A + gg * 100 * aa / 0xD9

blue = 0x1A + bb * 100 * aa / 0xD9

Should be

   intensity = 100*(0XD9 - aa)/(0xD9 - 0x1A);
   red = rr + (0XD9 - rr)*(100-intensity)/100;
   green = gg + (0XD9 - gg)*(100-intensity)/100;
   blue = bb + (0XD9 - bb)*(100-intensity)/100;

The color of the menu background and the task bar is set using the rr, gg, bb values.


Once you set the registry value for colorization color, you may need to restart to make it show.


The order of data in the colorization color DWORD does have Microsoft documentation to support it but the rest had to be determined by some good guesses and a lot of experimentation.   I have an experimental program that does this calculation and supports the setting of all kinds of colors an non client metrics.  So I know the calculation works.
