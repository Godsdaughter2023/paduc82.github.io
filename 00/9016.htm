There is a solution that does not require a listbox. Let's start with the database design: you must have 3 tables: FoodItems, Persons, and PersonFoods. The last one is called the "junction table" and has the primary key (PK) fields of the other two tables, and its PK is the combination of those two. Then go into the Relationships form and draw the relationships between the fields, and enforce them (check the checkbox).

Now create a new form and bind it to PersonFoods. Put a single textbox on it and bind it to the PersonID field. Put a single dropdown on it with FoodItemID, FoodItemName from FoodITems, and bind it to the FoodItemID field. Set the form's DefaultView to be Datasheet.

marshall barton say:

Actually that should use three tables and a subform (instead of a list box)

What you have is a many to many relationship between foods and people.  Each person can select multiple foods and each food can be selected by multioke oeioke.  You already have the foods table and a people table, what you need is what's often called a junction table to connect them.

Let's call the junction table the PersonFoods table.  Then fields in this table would be FoodID (the foreign key of a selected food) and PersonID (a foreign key to the person that selected the food).  Those two fields would comprise the (compound) primary key.  You may also have other fields related to the person/food combination suxh as the date selected. cost. or whatever.

With that table design, you can set up a form for the person table and a continuoous subform for the PersonFoods table.  The subform would have a combo box to select the food and other text boxes for any other fields in that table.
