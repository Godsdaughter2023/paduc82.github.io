Although running totals in a query or form are possible, it's difficult and has a negative impact on performance.

Reports have support for running totals built in: a text box on a report has a Running Sum property. ken sheridan say: The most efficient means of computing a balance is by means of a JOIN, but this will render the query non-updatable, so is unsuitable for a form in which you need to enter data.  It is necessary to call the VBA DSum function, but this is far less efficient.  You'll find examples of both methods, in a variety of contexts, in the file Balances.zip in my public databases folder at:

As you are doing this in a subform you would need to use a method which computes the balances per group, in this case the order number or similar.  As the rows are not individually dated, unlike most cases where balances are being computed, you will need a distinct value on which to order the rows, so ensure that the OrderDetails table has an autonumber column and order the subform's query on this column.  Here's an example using the sample Northwind database:

SELECT [Order ID], ID, [Product ID], Quantity, [Unit Price],
Quantity*[Unit Price] AS Total,
DSUM("Quantity*[Unit Price]", "[Order Details]", "[Order ID] = "
& [Order ID] & " AND ID <= " & [ID]) AS Balance
FROM [Order Details]
ORDER BY [Order ID], [ID];

BTW I'd advise against using the generic name ID as the name of a key column as Northwind does.  Use something more explicitly descriptive such as OrderDetailID.  Also the inclusion of spaces or other special characters in column names is not good practice.  Use CamelCase or an underscore character to indicate a space, e.g. Unit_Price.
