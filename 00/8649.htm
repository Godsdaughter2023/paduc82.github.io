bananarepublic say:

So you're looking for ODBC drivers to connect to an Access database by an external application. 

Because Access database engine runs in-process, you must use the same bitness the external application runs in. If it's a 32-bit application, then you must use 32-bit driver. Likewise, if this application is 64-bit, then you must use 64-bit driver. The bitness of Windows is irrelevant in this context. 

With Office, however, you cannot mix 32-bit installation and 64-bit installation and that goes for Access database engine, which is a part of the Office installation. Note that you can install the database engine without the Office by downloading and installing "Access Database Engine redistributable" (please note that this is different from Access runtime -- runtime allows you to run Access applications (think forms & reports) whereas redistributable contains only the engine but no interface.

You can get either 32-bit or 64-bit 2010 redistributable here: 

https://www.microsoft.com/en-us/download/details.aspx?id=13255

As Tom noted earlier, on a 64-bit Windows, you do have two odbcad32 applet and you must open one that is the appropriate bitness. You will likely find that because you've already installed Office, you will be able to see the 2010 Access ODBC drivers in 32-bit odbcad32.exe in sysWOW64 folder. By default Office 2010 installs 32-bit and one has to explicitly go out of their way to install 64-bit version of Office. 

If your external application is 32-bit and the Office installed is 64-bit, then you have a problem because as said at the beginning, the drivers and the application has to be same bitness to be usable. 

HTH.

albert d. kallal say:

You have to install a newer version of the Access database engine.

 

However BY DEFAULT you will NOT see the option to connect to the Access database if you launch the x64 bit version of the ODBC manager (and that is what windows does by default).

 

So from the control panel you are launching (by default) the x64 bit version of the ODBC manager.

 

So, you MUST launch the 32 bit version of the ODBC manger.

 

It usually best to just let Access launch the correct ODBC manager.

 

However, if you not launching the ODBC manager from Access, then launch the 32 bit version of the ODBC manager and it is found in this location:

C:\Windows\SysWOW64\odbcad.exe

 

You can thus type the above into a cmd session you start, or right into the search programs box when you hit the windows key (at least on win 7).

 

So FIRST you need to install the newer version of the Access data engine. BananaRepublic post has a link to this engine. Once done, then you need to ensure you launch the correct bit size of the ODBC manager.

jsheaott say:

lucatrotta1/LT_1 - assuming you are same original poster -

If it keeps happening after updates, you have at least one Exchange or Office 365 type account out of your 3? There's an error with your email account provider's setup or the server that worked at one point is periodically offline. After some types updates - where key files have been altered, Outlook needs to re-establish the connection again and is re-using the information that worked last time. For whatever reason, that info is now incorrect/has been changed or the server is unreachable or the info in that file has become corrupted or a virus or hacker has altered them ... whatever - the last known info is no longer working. 

Short term fix without having to remove the account or profile completely (so you don't lose any cached offline data):

1) Go to C:\Users\%Username%\Appdata\local\Microsoft\Outlook- just delete the xml and tmp files here as they contain the autodiscover information.

2) Go to C:\Users\%Username%\Appdata\local\Microsoft\Outlook\16 and delete the files there

(you need to have explorer set to "show hidden files" to see that '16' folder)

Then open Outlook, the files will get recreated.

Some suggest to set those few involved files to read-only so that they do not get corrupted or changed in future. Making them unwriteable can cause issues down the road, say when something does legit need to be changed, you add another address, etc... (and you may have forgotten you did that by then), so not recommended unless it's a last resort.

Long term Fix is to have your domain email host correct their configuration. Point your IT department to these:

KB 3049615 lower half of the page - test with the "Microsoft Remote Connectivity Analyzer "

https://support.microsoft.com/en-us/kb/3049615

https://blogs.technet.microsoft.com/sarao365/2016/12/09/outlook-crashes-with-multiple-pop-ups-post-migration-to-office-365/
