ken sheridan say: You don't really need a table for this; you can use the report object's own Description property and list the reports by description in an unbound combo or list box whose list is filled in the form's open event procedure.  This is code which does this, listing all reports which have a Description property in a list box named lstReports:

Private Sub Form_Open(Cancel As Integer)

    On Error GoTo Err_Handler

    Const NODESCRIPTION = 3270
    Dim ctrl As Control
    Dim dbs As DAO.Database
    Dim ctr As DAO.Container
    Dim doc As DAO.Document
    Dim strReportname As String
    Dim strDescription As String
     
    Set dbs = CurrentDb
    Set ctr = dbs.Containers("Reports")
    Set ctrl = Me.lstReports
    
    ctrl.RowSourceType = "Value List"
    ctrl.ColumnWidths = "0"
    ctrl.BoundColumn = 1
    ctrl.ColumnCount = 2
     
    For Each doc In ctr.Documents
        strReportname = doc.Name
        On Error Resume Next
        strDescription = doc.Properties("Description")
        Select Case Err.Number
            Case 0
            ' no error
            Case NODESCRIPTION
            ' report has no description
            ' so ignore and go to next report
            GoTo EndLoop
            Case Else
            ' unknown error
            GoTo Err_Handler
        End Select
        Err.Clear
        On Error GoTo Err_Handler
        ctrl.AddItem strReportname & ";" & strDescription
EndLoop:
    Next doc
    
Exit_Here:
    Exit Sub
    
Err_Handler:
    MsgBox Err.Description, vbExclamation, "Error"
    Resume Exit_Here
    
End Sub

To list a subset of reports you can easily amend the code, e.g.

        End Select
        Err.Clear
        On Error GoTo Err_Handler
         If strDescription = "Some Report" Or _
            strDescription = "Some Other Report" Or _
            strDescription = "Yet Another Report" Then

            ctrl.AddItem strReportname & ";" & strDescription

        End If
EndLoop:

or more concisely you can use the method I've usually adopted in cases like this and use some naming convention for your report names which allows for the relevant reports to be identified, and examine the value of the strReportname variable rather than the strDescription  variable.  I have done this by adding a suffix of a fixed number of characters to a report name to classify the report.  To identify reports of a particular class is then simply a matter of examining the last n characters of the strReportname  variable using the Right() operator.

Note that the combo box or list box lists only the descriptions, so the names of the report objects are never exposed to the user.  An additional text box is therefore unnecessary.  The value of the control is the hidden report name, so you simply have to reference the control's value to open a report.

jack rowand say: The solution that worked for me was to remove the Send to Bluetooth COM Add-in.

To do this you need to run excel as Admin. Assuming you have administrator privileges, click the start button, type in Excel in the search box, and then press Shift, Ctrl and Enter at the same time.

You should then tick Send to Bluetooth (Unselect any other Add-Ins if they have automatically been ticked.

Then click remove.

Close and restart excel, and hopefully the problem has been solved.
