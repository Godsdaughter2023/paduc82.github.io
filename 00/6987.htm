joeu2004 say: Paul wrote:

I found the type of output from the above code not too surprising because of all the examples on the web.  The For-Each iterating variable through a range acts like a pointer (or a "reference variable in C++) so that any changes made to the iterating variable affect the cell that it represents.


At the risk of repeating myself:  the difference is a collection of objects v. "collection" of values.

In C++, you can have an array of pointers or an array of values.  Think of a collection of objects, Range("a1:b4"), as an array of pointers.  In contrast, Range("a1:b4").Value, Array(...) and an array variable are "collections" of values.

For Each is treating them both the same:  it simply gets the next element of the "array" (collection).

The difference is how we code the body of the For Each loop.  Just like C++, we need to know that the element (ergo, the For Each control variable) is a pointer or a value, and we need to use the appropriate syntax.

If we write For Each r in Range(...), we get away with writing just r because for a range class object, .Value is the default member.  In other words, r by itself usually is shorthand for r.Value.

You have to understand how the system handles 'by value' references to objects and atomic variables.

Using for each x in... *always* creates a copy in x.  The issue is a copy of what.  For atomic variables it is a copy of the value of the variable.  For objects it is a copy of the pointer (or more accurately the handle) to the variable.  I've discussed this before.  The system *cannot* create a copy of an object.  It is fundamentally impossible.

So, in your case elvar contains a copy of the element.  If you change elvar, only the copy is affected.  For an object, if you do something that affects only the handle in elvar, there will be no impact on the actual object.  See the sample code below and the article on byval and byref arguments.

Option Explicit

Function TestFunc11()
   
    Dim arVar() As Variant 'Array of Variants
    Dim elVar As Variant   'Scans elements in arVar
        
    Dim aColl As Collection: Set aColl = New Collection
   
    ReDim arVar(1 To 1, 1 To 2)
    Set arVar(1, 1) = aColl
    arVar(1, 2) = "a value"
    For Each elVar In arVar
        If TypeOf elVar Is Object  Then Set elVar = Nothing _
        Else elVar = Empty
        Next elVar
    For Each elVar In arVar
        Debug.Print TypeName(elVar)
        Next elVar
    End Function

    Alright!  The fix has been deployed server-side to stop generating content incompatible with OneNote 2010.  New pages/sections created in OneNote for Windows 10 will now sync properly to OneNote 2010.  So it's safe to use OneNote Win10 again.

    Unfortunately, the existing content is already broken in OneNote 2010 :(

    The easiest workaround is to run OneNote 2016 which understands the format change.  We have a free version compatible with OneDrive:

    http://www.onenote.com/download

    The free version has a few limitations, such as no local notebooks, but since you all are using OneDrive already this may not be an issue.  Although some of you have reported these are work devices, so I don't know if you're allowed to install OneNote 2016 onto these devices (might be blocked by IT admins).

    To get this working again in OneNote 2010, unfortunately you need to copy the content within OneNote for Windows 10 to regenerate it.  This can be a cumbersome process...  If it's just a few pages, right-click those pages in OneNote Win10 > Move/Copy > Copy into the same section, then delete the old page.  Page copy + delete will be the most reliable.

    If lots of pages are broken, or the entire section is broken, you can right-click the section > Move/Copy > Copy into the same notebook.  This copy will append a "2" to the name since sections can't have duplicate names ("Quick Notes" copies to "Quick Notes 2").  New section sync can be slow, and rename can be flaky: it's best to right-click the notebook and Sync This Notebook (or Shift+F9) to force it to sync.  Once 2010 sees the new section, then you can delete the old one, and again force it to sync.  Once the delete finishes syncing, it's then safe to rename it to remove the "2" appended to the copy.

    I apologize for this tedious workaround :(
