Message could not be delivered. Please ensure the message is RFC 5322 - Microsoft Community

corry l say:

Message could not be delivered. Please ensure the message is RFC 5322 compliant

Just got this message, Message could not be delivered. Please ensure the message is RFC 5322 compliant

Now, I know that someone from "MS" (more like a contracted support center with non-technical people) is going to reply and tell me to check some web posts, and such, and assume I'm non-technical, and don't understand the message.  Quite the opposite, so you are going to need to pass this one up the chain.

This isn't from something I did, it was a message from a vendor.  Yes, their message was not in fact RFC 5322 compliant.  They send their From: field as Company Name, Inc.  <*** Email address is removed for privacy ***>  Obviously, their email server should have followed RFC 5322 Section 3.2.4 Quoted Strings.  Obviously, as a human its easy to understand that what they meant was "Company name, Inc" <*** Email address is removed for privacy ***>  This isn't about what their server should have done, this is about what the MS Exchange server backing Outlook.com should have done.  In the simplest case, that rejection would have gone back to the original sender, they'd get a message about the misconfiguration of their server, and all would be well in the world.  They'd know their message didn't get delivered, and they had a problem.  However, that's not the case for a lot of outlook.com and Hotmail.com users!  Every post on the forums about this issue seems to be from people who had gmail accounts forwarded to outlook.com.  So now the chain of events is originator (Non RFC 5322 compliant) -> Google (accepted, still non RFC 5322 compliant) -> outlook.com REJECTED -> Google (non forwarded email sent to my gmail address) Email rejected, here's why. END.  Never got back to company.com so they don't know there's a problem, and had I not just routinely (that I'm trying to break) gone to gmail for webmail, I wouldn't have known.  This is a breakdown of the email system! 

Obviously, in a perfect world, we should just be able to reject poorly formed messages.  Unfortunately, the world is far from perfect.  It would be better to let the user know especially if the email is forwarded.  Google uses a non-standard header X-Forwarded-To and X-Forwarded-From, and I guess MS just uses a MAPI property.  Regardless of standards, if the message is obviously a forwarded message, this means an intermediate server has accepted the email, and the rejection is not going to make it back to the originator, and thus, the final recipient should be notified! 

In my case, as an engineer, I'm also going to send a message to google, because again, in a perfect world, gmail would see the bounce from the forwarded address, and return the bounce to the originator, but it seems there's no standard for that just as there's no standard for forwarding messages, thus, a non-standard approach to handling errors of a non-standard email chain must be taken!  However, at the very least, MS should do the right thing and notify the user since the entire chain is non-standard meaning the rejection mechanism is completely broken.

Lastly, again, I'm a software engineer, so I get how difficult it can be to start allowing non-standard stuff.  However, in this case we're talking just a few lines to allow a poorly formatted from field which is the cause of every thread in this chain.  Instead of parsing it as char*address=strtok(string, ","); then if (strcmp(strtok(address, "@"), string) == 0) Reject Message; else {bValidAddressFound=TRUE; HandleValidAddressaddress();  Ok, well, you'd have to make a copy of address since strtok would blow the @ away with a NULL, but you get the picture :) In the case of my rejected email, this would have resulted in CompanyName having been ignored as invalid, then it wuld have found a valid from address named Inc with the address <*** Email address is removed for privacy ***>Pretty simple really, and I suspect this is exactly what gmail is doing, and why it is accepting these poorly formatted emails.  It may be a bug on their end, or it may be, like what I just said, it doesn't really make sense to reject an email based on the poor formatting of the From field, so long as it does include an email address in there somewhere!  Again, in a perfect world, you wouldn't have to care, but as I said before, this isn't a perfect world, and we just have to try and make due! :) 

Hopefully, this will stop the sporadic, but annoying threads on here about this problem, and will help your migration of gmail users.  There are many of us that have been online since nearly the beginning, which means we got into various company's eco-systems before they decided they wanted a full eco-system, before the silicon valley "me too" business model got started, and as such have email from one, a phone from another, and a desktop/laptop from one or more of the others!  This business of trying to make that difficult for people (google) is going to cause users to leave.  I'm glad that so far at least, MS hasn't decided to play this game!

If you’d like to manage your Hotmail account using an email app, you’ll need to set up your POP3 or SMTP server and port settings in your email app. You’ll need the following info:

    POP3 server: pop3.live.com (port 995)
    SMTP server: smtp.live.com (port 25)

If port 25 has been blocked in your network or by your ISP, you can set the SMTP port to 587 with TLS or SSL Encryption (depending on which email app you're using).

Choose the Secure Sockets Layer (SSL) option for both POP3 and SMTP connections.

Choose the SMTP authentication for SMTP connections.

 Choose the outgoing server requires authentication option. (In most email apps, this option isn't checked by default.)

    User name: Enter your full email address.
    Password: Enter your Microsoft account password.

    If you don’t want email messages to be permanently deleted from your inbox after you download them to your mail program, select the Leave a copy of messages on the server option.

Prevent losing email in Hotmail when you've set up POP3

Hotmail has a feature that prevents losing email messages if you accidentally set up POP3 to delete them after reading them on the server. Here’s how it works:

    The first time that Hotmail receives a POP3 instruction to delete email messages after reading from an email app, Hotmail won't delete the messages. Instead, Hotmail moves the email messages that were successfully downloaded by the email app into a POP folder and out of the inbox. The POP folder is available in the Folders list.

    Hotmail keeps the email messages in the POP folder until you sign in to your Hotmail account and verify whether Hotmail should respect or ignore the POP3 delete command.

To verify your POP3 settings:

    In the upper-right of the Hotmail window, click Options, and then click More options.

    Under Managing Your Account, click Pop and deleting downloaded messages, and then select one of the following:

        Don’t let another program delete messages from Hotmail. (If your other program is set to delete messages from the server, we’ll simply move them to a special POP folder. They won’t be deleted.)

        Do what my other program says—if it says to delete messages, then delete them.

    Click Save.

How to hide or unhide rows based on a value enter in a specific cell - Microsoft Community

nathan chew say:

How to hide or unhide rows based on a value enter in a specific cell

Hi, I would like to know if there is any excel vba coding for situation like this, example when I key in 3 at the no of T/C Sensor cell, after I press the OK button, the row from 14 until 19 will auto hide. Then when I key in 5, the row from 16 to 19 will auto hide.

Any idea or suggestion, thanks.

ossiemac say:

Hello Nathan,

Code below. I have used Select Case code so that you can add additional options fairly easily if required.

Deleting or entering zero in the cell will both unhide all rows. Reason is that a blank cell returns zero value in the code.

You can change the name of the sub if required but it is necessary to assign the correct macro to the Forms control button. (Right click the button and select "Assign macro" and then select the macro from the list and click OK).

The code could have unhidden all rows with the Hidden = False command immediately after the line With ws. However, I have placed it in each of the Case statements so if you have a change of mind and want to hide additional rows without unhiding all. If you decide you want this option then remove the unhide statement from each case statement.

Sub Button1_Click()

    Dim ws As Worksheet

    Set ws = Worksheets("Sheet1")       'Edit "Sheet1" to your worksheet name

    Application.ScreenUpdating = False  'Suppress excessive screen flicker as rows unhidden and hidden

    With ws
        Select Case .Cells(3, "C").Value
            Case 3
                .Rows.Hidden = False
                .Rows("14:19").Hidden = True
            Case 5
                .Rows.Hidden = False
                .Rows("16:19").Hidden = True
            Case 0      'Note zero or deleted both return zero value
                .Rows.Hidden = False
        End Select
    End With
    Application.ScreenUpdating = True   'Display final hidden/unhidden result

End Sub

My Symptoms:

  1. Some contacts duplicated  between 0 and over 30 times
  2. Went from 800+ contacts to 2400+ contacts
  3. Not all duplicate contacts were identical - some had new and old info and some had corrupted with info from multiple contacts

  •  Trying to manually edit the corrupted contacts.

  • To fix the problem manage your contacts from the People section of Outlook.com (https://outlook.live.com), ONLY
