john mcghie say:

Hi Francesco:

If you set your default to paste images "Inline with text", they should adopt the size of the table cell.

If you paste as anything else, the image is not actually inside the table cell, and so is unaware of its margins.

Here's a macro I use in Word 2011/2016 to size pictures:

Sub SizePic()
'
' Size Picture Macro
' Macro written 14/09/2004 by John McGhie
'
Dim MarginWidth As Single

With ActiveDocument.Sections(Selection.Information(wdActiveEndSectionNumber)).PageSetup
   MarginWidth = .PageWidth - .LeftMargin - .RightMargin
End With
If Selection.InlineShapes.Count = 1 Then
   With Selection.InlineShapes(1)
      If .ScaleWidth <= 100 Or .Width > MarginWidth Then
         .Width = MarginWidth
         .ScaleHeight = .ScaleWidth
      End If
   End With
ElseIf Selection.ShapeRange.Count = 1 Then
   Selection.ShapeRange(1).ConvertToInlineShape
   With Selection.InlineShapes(1)
      If .ScaleWidth <= 100 Or .Width > MarginWidth Then
         .Width = MarginWidth
         .ScaleHeight = .ScaleWidth
      End If
   End With
End If
End Sub

That macro sets the picture to exactly the width of the text margin on the page.  You can set it to a different size by deleting the first "With... End With" statement (the bold bit), then setting "MarginWidth" to the width you want the picture to become (in points).

The macro is complex because there are two "kinds" of pictures: "Inline" and "Floating". The second half of the code detects that the selected picture is a floating picture and converts it to an inline picture (otherwise you can't re-size it!).

Note that MarginWidth is defined as a "Single".  All of these variables are singles, which is rare in VBA (a single is a spectacularly huge number which is massive over-kill). It is necessary to use singles (4-byte, 32-bit) or Doubles (8-byte, 64-bit) to provide sufficient space to multiply or divide floating point numbers in points.

Just do it, otherwise the code blows up :-)

Hope this helps

Microsoft HD-500 Display Dock appears to not be USB-C Compliant, does not function properly with Google Pixel

Hello all, I do independent verification and testing of USB-C devices and cables alongside Benson Leung, GTrusted, and other reviewers.

I recently bought a HD-500 display dock for use with a Google Pixel phone and discovered this dock is not USB-C compliant.

The dock itself is missing Rd resistors on its input USB-C port. Because of this, only Vbus hot corded chargers (or spec-violating Vbus hot receptacle chargers) will successfully power it. This is the same flaw the Nokia N1 suffers from, and means this dock is USB-C Spec 1.2 noncompliant.

I find this exceptionally odd since the manufacture date of the dock is "30.06.2016" and by no means "first gen". The standard was well established, and Microsoft/Nokia engineers sit on the committee that helped write the spec.

G.Ilex, with due respect I don't know why someone marked your response as an answer. Your answer is demonstrably incorrect.

(1) The dock claims to be compatible with USB-C and Power Delivery specifications. Therefore ANY phone that is USB-C compliant OR USB-PD compliant should function. The only part that may not function is the PD alternate-mode component (DisplayPort output) or PD datarole-swap component (USB hub).

I have demonstrated this with verifiable traffic dumps using a Google PD Analyzer ("Twinkie") device.

> tw trace on
SRC/0 [1161]SRCCAP 5000mV/2000mA

> tw res rd 0
> tw cc
CC1 = 988 mV ; CC2 = 2976 mV (<--- 1.5A advertisement... on a 1A power supply!)
> tw vbus
VBUS = 5118 mV ; 0 mA

> tw trace on
SRC/0 [1161]SRCCAP 5000mV/900mA

(2) This dock has proven engineering flaws that render it unsafe to use with USB-C compliant equipment. These flaws make it vulnerable to overdrawing current from standards-compliant devices.

Translation: Microsoft broke the USB specifications in such a way that if you plug it into 100% safe, tested, USB compliant gear, it will risk damaging a charger or phone.

https://plus.google.com/102612254593917101378/posts/YQSStyFMzKP

https://plus.google.com/102612254593917101378/posts/TLP2B4JDu8G

There is no dodging this by trying to say it "is for Continuum capable devices only". Microsoft used the USB specification and connector, meaning they have to follow the basic rules to make it safe with USB-C compliant equipment. In simpler terms, Microsoft cannot simply decide to run 120 volts AC through USB then claim "it's only intended for our products". That is in blatant violation of USB specification, and Microsoft can get in trouble for it.

The dock design as-is is defective and should be fixed. Even Benson Leung himself replied and noted this behavior appears to be in error.

"It's almost as if for the upstream port they had designed the thing around a dedicated barrel-jack DC and then at the last minute switched it out for Type-C and decided not to do any of the work to make it a proper Type-C receptacle.

Most of the other charge-through hubs I've encountered seem to at least be proper Type-C receptacles... heck, all of the others I've seen talk PD on the upstream port, so I know they are paying attention to CC."

This flaw is EXACTLY the same type of flaw that occurs with "bad" A-to-C cables that Benson has been crusading against. I work with him in doing charger and cable verification, and hence this came up.

Please take this report of an engineering flaw in the product seriously. Users run the risk of seriously damaging their devices or hardware otherwise! Please contact either myself or Benson regarding the flaw and how to correct it.

john mcghie say:
Well...  You can "sort of" do this, but you probably wouldn't offer the result as a commercial solution...

When you Protect Document for Filling Forms, you can nominate which sections to protect.

So if you add multiple section breaks to the document you can protect the ones that contain form fields so the form fields will work, and leave the sections that do not contain form fields unprotected, so they behave like part of a normal document.

Surround the parts containing form fields with continuous section breaks.  You can have as little as one single form field in a protected section.

You can't mix the two modes: form fields won't work in the unprotected sections and the user can do anything they like in those sections.  No editing apart from filling in form fields is possible in the protected sections.

There's no restriction on what the users can do in the unprotected sections, and unskilled users may be completely unaware of the table and seriously wreck your document.

Once you get to this level of complexity, it's time to ask yourself whether Word is really the place to be doing this.  None of this stuff will work on the web in Word: so you may be better off looking at an HTML application now.

Hope this helps
