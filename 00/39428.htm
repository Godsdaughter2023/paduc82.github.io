malkeleah say: Failure to install an operating system is usually caused by (in order of probability):

1. You are attempting to install with other devices connected to the computer. Disconnect them before trying to do an installation.

2. Faulty installation media. Check the installation CD/DVD for scratches, smudges. Test in another machine.

3. Faulty optical drive. Test with another bootable CD/DVD or swap it out for a known-working one.

4. Faulty RAM. Test with Memtest86+ from www.memtest.org . You will download the .iso to make a bootable cd. Burn the file with third-party burning software* - as an image, not as data. Then boot with the media you made (you may need to change the boot priority in your BIOS so that the optical drive is the first boot device). The test will run immediately. Let the test run for an hour or two - unless errors are seen immediately. If you get any errors, replace the RAM.

A good free choice is ImgBurn from www.imgburn.com .

5. Faulty hard drive. Test with a diagnostic utility downloaded from the drive mftr.'s website. You will create a bootable CD with the file you download. Burn the file with third-party burning software - as an image, not as data. Boot with the CD you made and do a thorough test of the drive. If it fails any physical tests, replace it.

(Note: Since you just replaced your hard drive this is probably not the cause. However, you should still test because it is not uncommon for new hard drives to fail out of the box.)

6. Other hardware components are faulty.

I can, however, tell you the following facts, based on my Windows 7 Pro system

  1. The Windows Time Service (W32Time) has a startup type of "Manual" and does not start when Windows starts.
  2. On the other hand, the Task Scheduler service (Schedule) has a startup type of "Automatic" and does start when Windows starts.
  3. There is a system-created scheduled task named "SynchronizeTime" located in the Task Scheduler Library under \Microsoft\Windows\Time Synchronization.  This task runs under the "LOCAL SERVICE" user account, runs with highest privileges, but only runs if the computer is not on battery power (this is a desktop) and there any sort of network connection.   The Task is "Enabled" and its "trigger" is "Weekly ... at 1:00 of every Sunday of every week starting 1/1/2005."  The action taken when the task starts is to run the following program:   %windir%\system32\sc.exe start w32time task_started
  4. The program Sc.exe "Communicates with the Service Controller and installed services," so what that scheduled task does is start the w32time service.
  5. The value of "SpecialPollInterval" in the registry location you mentioned is 604800 -- which happens to be the number of seconds in 7 days.
  6. The Event Log in this system shows that time is, in fact, synchronized every 7 days.

My take from the above is that you don't have to configure the W32Time service to start automatically because the scheduled task will start it whenever it is needed to do the time synchronization.  I don't know what stops the service. 

It appears that somehow the value in the registry at

         HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\services\W32Time\TimeProviders\NtpClient\SpecialPollInterval

gets picked up by the Task Scheduler.  As I said, don't ask me how that happens.

Having written all of that, if your system is working to your satisfaction the way you've got it currently configured, don't worry any more about it.
