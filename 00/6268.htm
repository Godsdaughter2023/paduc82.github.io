Actually these days  ping  -n 1  is only really useful as an addressability check.   Ever since  ping  was used in moronic "Denial of Service"  exploits many sites prevent use of its  ICMP  protocol even close to their servers.  That's probably what you are seeing in the case of  bing.com.   You could check on that idea using a related connectivity test tool called  tracert.

So a better connectivity test would be one which actually tests the protocol that you intend to use, e.g. to see if there is a server listening for it on the other side.  You can do that with  telnet, e.g. specifying port 80 for the site's HTTP server.  You can actually do more than just see that the port is open, e.g. simulate some of the HTTP request headers that IE sends such as (minimally)  GET /  and  Host:  but that may be a bit advanced for this discussion.

Another option would be running Fiddler2 to trace how IE's requests are being treated and Fiddler can even assist you with a more complex minimal simulation of those requests (or those of any other browser) using its Request Builder feature. 

But sometimes the simplest connectivity test is just to use a different browser.  If that works you would know that something is blocking the use of IE, typically a misconfigured software firewall or other security package in the client computer.

    =PERCENTILE(IF(B2:B20="A",A2:A20),0.3)
