First of all, the instructions from Tina Chen will cause trouble sooner or later: A user document saved with the name Normal.dotm will not function properly as a basis for new documents. The Normal.dotm must be generated by the program.

It would have been better to Open the original Normal.dotm & change the Zoom level there. Unfortunately, this has been made difficult due to Apple's decision to make the Library folders for User Accounts Hidden by default [OS X 10.7 & later]. As a result it cannot be accessed from within the program unless you first UnHide that folder.

Hopefully you have not deleted your former Normal.dotm template or did follow the suggestion to create a backup of it. If that's the case I'd suggest that you:

  1. Quit all programs
  2. Return to the Library folder & remove any existing Normal.dotm files
  3. Launch Word so that it generates a new Normal.dotm
  4. Quit the program then launch it again

To retrieve your Macros as well as any User-Defined Styles, & AutoText Items:

If you haven't deleted your former Normal.dotm template, use the Organizer to copy the customizations from one template or document to another. The Organizer can be reached from either of several directions:

  • From the Macros dialog window, Click the Organizer button

ken sheridan say: In Access a carriage return/line break is represented by The Chr(13) & Chr(10) characters.

You should be able to replace the Chr(10) characters in an update query with:

UPDATE [YourTable]
SET [YourField] = REPLACE([YourField], CHR(10), CHR(13) & CHR(10));

Microsoft Access- how to order contacts in alphabetical order - Microsoft Community

sandyw43 say:

Microsoft Access- how to order contacts in alphabetical order

I created an Access document and loaded my contacts. How do I get Microsoft Access to load my contacts in alphabetical order. 

thedbguy say:

Hi,

How exactly are you "loading" your contacts? Assuming that you're storing them in tables, the data is not stored in any particular order. To see them in a specific order, you'll need to use a query to retrieve the data and display them using a sort order.

sandyw43 say:

Thank you so much for responding...I am in the midst of making a 'contact phone list'.  I have 15 entries so far.  I was wanting something that I could click on to alphabetize them after I enter 10 or more.  How would I do that?

Sandy

ken sheridan say: Tables in a relational database are sets.  By definition a set has no order.  To see the rows ordered on the basis of values in one or more columns use a query as the RecordSource of a form, e.g.

SELECT *
FROM Contacts
ORDER BY LastName, FirstName;

 Data should always be entered via a form of course, never in a table's raw datasheet view.

Do not base a report on an ordered query, however.  A report uses its own internal sorting and grouping mechanisms, set via the sorting an grouping option in the ribbon in report design view.

With a simple report based on a single table as above the first sort level would be LastName and the second sort level FirstName.  In a more complex report base on a query which joins multiple tables, account needs to be taken of the fact that personal names can be duplicated.  If it were necessary to group a report by contact and return aggregated data per contact, e.g. the number of interactions with each contact, then the report would be sorted firstly on LastName, then FirstName and then grouped on the primary key ContactID.  The personal data for each contact would be shown in the ContactID group header, the details of each interaction with the contact in the detail section, and the aggregated data in the ContactID group footer.  This would sort the report alphabetically by name, but would return two or more contacts of the same name separately, whereas if the report were sorted on name only their data would be incorrectly intermingled.

Duplication of personal names is more common, even in small populations, than might be expected.  I was once present at a clinic when two female patients arrive within minutes of each other.  Not only did they have the same first and last names, but the same date of birth!  A distinct numeric key should always be used for people therefore.
