ken sheridan say:

...................but It does not list any fields in the nested subform.

Write the expression yourself.  Subforms are not members of the Forms collection, so the syntax to reference a control in a subform is:

Forms![ParentFormName]![SubformControlName].Form![ControlName]

Consequently to reference a nested subform the syntax is:

Forms![ParentFormName]![SubformControlName].Form![NestedSubformControlName].Form![ControlName]

Note that SubformControlName and NestedSubformControlName are the names of the controls in the parent and first subform's Controls collections which house the subforms.  The name might or might not be the same as its source form object.

daniel g. grau say:
It is, in fact, toast.

As you may know, .docx files are kind of a .zip file. So I changed the document's extension to .zip, decompressed it and opened the various files in there. Of particular interest is the file called "document.xml," which contains the bulk of the text.

Just about all it contains is 57,000 copies of this XML code:
  • <w:r w:rsidR="001D1569"><w:pgNum/></w:r>
Every single character in the file has been changed to a page number reference, including those in headers and footers. All that remains from the original are some styles, and the preview icon:

 

Were you using AutoRecover at 1-minute intervals? Too short. I think that was related to the corruption replacement. At any rate, even the AutoRecover files would have faithfully reflected the corruption, as it was designed as a fallback measure when Word crashes and not when the user quits.

Next time, enable Word's backup option whenever you're looking to work long hours in a document. After you're done, disable it, to avoid Word's littering your whole disk with files named "Backup of..." And if you ever see corruption in a document, don't save. If you need to save, do instead a Save As to preserve the last-saved version.
