In this situation, we suggest you create a SharePoint group and give this SharePoint group Edit or Contribute permission, then add these standard users into it. 1. As a SharePoint Online admin or global admin, you can open this site and navigate to the Site Settings-> People and Groups (https://contoso.sharepoint.com/sites/test/siteA/_layouts/15/groups.aspx , replace contoso, test, site A with your own tenant or site names)

Note: If you don’t see the group permission part when creating a group, you maybe in a sub site. 4. After creating a SharePoint group with relate permission, you can open this group and add some users.

5. Then open the site or library you’d like to share, click share button and input the SharePoint group to give these users with Edit permission globally.

Microsoft AutoUpdate Fails to Notice Available Updates

     A relatively recent Microsoft AutoUpdate release* delivered alongside other Office updates via Office 365 Personal's Insider Slow channel has introduced what I suspect could be a bug in the logic the utility uses to check for updates and display them to the end user.  After you have had an Office release with patch level zero (that is, x.y.0 — for example, 16.16.0) installed on your machine for enough time to have passed for a new patch release to have become available for at least one program in the Office suite of applications and you install that patch release, Microsoft AutoUpdate neither fetches nor displays properly any further future updates when they become available any longer.  My experience with this issue hints that it may be narrowed down to when all of your Office applications' versions don't match exactly, even down to the build number, but I could be wrong.  What follows below is what I hope is standard, good, and sufficient enough of a bug report for you to use to reproduce what I'm seeing locally.  

---

Steps to Reproduce:  

     Given that one has already:  

  1. booted their machine,
  2. logged into an administrator account from which they can manipulate their system-wide Office installation,
  3. and ensured they are on all of their main Office applications (Word, OneNote, Excel, PowerPoint, and Outlook) are currently installed at the exact same release, build number included,

perform the following steps:  

  1. Launch Microsoft AutoUpdate.  
  2. Ensure that you have selected the following settings in the utility's main dialog window:  
    1. 'Automatically Check' under 'How would you like updates to be installed?'  
    2. 'Join the Office Insider program to get early access to new releases,' etc.
    3. 'Office Insider Slow' under 'Choose how you get Insider builds.'  
  3. Click 'Check for Updates' if opening Microsoft AutoUpdate didn't automatically start doing so for you (it hasn't for me lately, but that's orthogonal to this issue.)  
  4. Proceeding beyond that point, the bug I'm reporting here will manifest if the next available set of updates is only for an incomplete subset of Office's (main?*) applications; to observe it, continue if this is the case.  
  5. Go through the update process as usual.  

Expected Behavior:  

     For the all Office (main?*) applications' next-released updates to show up in Microsoft AutoUpdate when they become available through the Office Insider Slow release channel.  

Actual Observed Behavior:  

     No further updates announced on your 'What's New for Office Insiders' page (or any other release notes you publish, for that matter) beyond those just installed appear in Microsoft AutoUpdate after you've finished the former, no matter how long you wait.  

Known Workaround:  

  1. Uninstall Office from your machine completely.  
  2. Reinstall Office; this should get you to a non-patch release.  
  3. Run Microsoft AutoUpdate again, and you should see any available non–patch-level updates, thus allowing you to skip them this time around and upgrade Office to the next minor-version release successfully.

vaibhavin say:

Is there any way I can know who opened my excel file last ?

Dear Ms Team/Users,

Regards,

jllatham say:

You could do this with a Workbook.Open() event macro - it runs when the workbook is opened BUT it depends on the user enabling macros when it is opened.  So if someone disables macros, the record is not written.

 

You can find instructions for placing the code below into the ThisWorkbook code module in a workbook here: http://www.contextures.com/xlvba01.html#Workbook

 

Be sure to save the file as a Macro Enabled file, type .xlsm not as a .xlsx file.  If you save it as a .xlsx file, the code gets thrown away and won't be there when you open the file later.

 

This macro would keep a list of who's opened the workbook in the past - so you'd have a picture of who has been looking at it/using it over time.  It is going to print the currently logged on username into a cell in column A of a sheet named Sheet3 - you can change that sheet name in the code.

 

Private Sub Workbook_Open()
  Const wsName = "Sheet3" ' change as desired
  Dim nextCellAddress As String
 
  nextCellAddress = _
   ThisWorkbook.Worksheets(wsName).Range("A" & Rows.Count).End(xlUp).Offset(1, 0).Address
 
  ThisWorkbook.Worksheets(wsName).Range(nextCellAddress) = Application.UserName
End Sub

 

If you just want the name of the last person that opened it - well, that's difficult to do because you would always be the last person to have opened it!  So if you just try writing that information to a single, specific cell, your name would always show up in it.

 

 

graham mayor say:
If you can overcome the difficulty that relates to the fact that macros are required to achieve this then save your workbook as a macro enabled workbook and in the ThisWorkbook module of that workbook put the following code.

Create a four column workbook with a header row to log the opening and closing of your workbook e.g

Open-Close Time Date User
Opened 14.07.2013
09:30:41 Graham Mayor
Opened 14.07.2013 09:31:21 Graham Mayor
Closed 14.07.2013 09:32:27 Graham Mayor
Opened 14.07.2013 09:32:47 Graham Mayor
Closed 14.07.2013 09:32:51 Graham Mayor

Save the workbook and put the path and filename in the macro below where indicated in bold.
When your workbook is opened and closed the times will be recorded.

Option Explicit
Private strOpenClose As String
Private strDate As String
Private strTime As String
Private strUser As String
Private strValues As String
Private Const strlogfile As String = "C:\Path\Log.xlsx"

Private Sub Workbook_Open()
    strOpenClose = "Opened"
    strDate = Format(Date, "dd.mm.yyyy") 'set format to taste
    strTime = Format(Time, "HH:MM:SS")
    strUser = Environ("username")
    strValues = strOpenClose & "', '" & strDate & "', '" & strTime & "', '" & strUser
    WriteToWorksheet strWorkbook:=strlogfile, strRange:="Sheet1", strValues:=strValues
End Sub

Private Sub Workbook_BeforeClose(Cancel As Boolean)
    strOpenClose = "Closed"
    strDate = Format(Date, "dd.mm.yyyy")
    strTime = Format(Time, "HH:MM:SS")
    strUser = Environ("username")
    strValues = strOpenClose & "', '" & strDate & "', '" & strTime & "', '" & strUser
    WriteToWorksheet strWorkbook:=strlogfile, strRange:="Sheet1", strValues:=strValues
End Sub

Private Function WriteToWorksheet(strWorkbook As String, _
                                 strRange As String, _
                                 strValues As String)
Dim ConnectionString As String
Dim strSQL As String
Dim CN As Object
    strRange = strRange & "$]"
    ConnectionString = "Provider=Microsoft.ACE.OLEDB.12.0;" & _
                       "Data Source=" & strWorkbook & ";" & _
                       "Extended Properties=""Excel 12.0 Xml;HDR=YES;"";"
    strSQL = "INSERT INTO [Sheet1$] VALUES('" & strValues & "')"
    Set CN = CreateObject("ADODB.Connection")
    Call CN.Open(ConnectionString)
    Call CN.Execute(strSQL, , 1 Or 128)
    CN.Close
    Set CN = Nothing
End Function
