ken sheridan say: Changing the display controls of the columns to text boxes was a wise move.  As data should never be entered directly in a table's datasheet but only via forms, it's best to see the actual values stored in the columns in the table's datasheet.  Use combo boxes in forms for data entry.  And never use the dreaded 'lookup field' wizard when designing a table.  For reasons why see:

http://www.mvps.org/access/lookupfields.htm

List boxes are generally not ideal as bound controls in forms, where a combo box is the usual type of control to be used for a foreign key column.  Where multiple values from a related table are to be shown in a form a subform is the usual solution.  List boxes are more suitable as unbound controls for querying the database, especially on multiple values from one column where a multi-select list box is a good choice, e.g. to return all customers in London, Paris and Rome (which in the Boolean logic used when querying on a City column would normally be London OR Paris OR Rome of course).

Data integrity in a relational database is enforced by and large through normalized table designs, indexes, and enforced relationships.  This might be augmented in forms, particularly through the use of code in BeforeUpdate event procedures, but the logical model should be as independent of the interface as possible.

Hi Lorraine,

Actually there is not certain relationship between BitLocker, and IRM, they apply to different scopes.

For BitLocker, it blocks unauthorized access to the data on your hard disk.

For IRM, it blocks unauthorized access to the data related to your Office 365.

If a file belongs to both these two scopes, it will be protected by both BitLocker and IRM.
