About the Malicious Software Removal Tool:

...keep in mind that MSRT only scans for and removes a limited number of specific malware families...a small subset of active malicious software so it is not comprehensive. Chances are (99.99%) your HP computer came with a Norton free-trial or a McAfee free-trial preinstalled. Doesn't matter if you never used it. Both are notorious for not uninstalling (or upgrading) cleanly. The "leftovers" may be playing a role here.

ACTIONS

schikora say:

Finally solved it.  Solution appears to be related to installation of the latest version of Windows Update Agent:
http://support.microsoft.com/kb/949104

But let me recount the final set of steps that fixed it in case anyone else should stumble across this thread with a similar problem.

I reviewed my installed updates to see where the winodws updates started failing.  Noted it was after installation of Windows Update Agent 7.2.6001.788.  Searched and found the link above and downloaded the appropriate update.

Based on what I read in other threads I did the following.  Duplicate at your own risk:
- Deleted the MSIServer key from HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\ and rebooted
- Installed Windows Update Agent from above link and rebooted
- Downloaded and installed Windows Installer 4.5 from link found here: http://support.microsoft.com/KB/942288  (prior installations of this update were fruitless).  Rebooted.
- Installation of MSSE still failed but with new error code 0x8004FF01 -- another windows installer error.  Checked registry and noticed MSIServer key still missing.
- Downloaded msiserver.zip package from here: http://www.winhelponline.com/blog/vista-the-windows-installer-service-could-not-be-accessed/.  Had seen this referenced in numerous threads and also tried it earlier with no luck.  But downloaded again and merged the registry data.  MSIServer key back in regsitry.  Rebooted.
- Tried one of critical updates packages I had downloaded some time ago but couldn't install.  Success!
- Installed MSSE with no problems.

Result of an expression in a form is not saving the data result to the - Microsoft Community

a. payne say:

Result of an expression in a form is not saving the data result to the Table

I have been trying to get a calculated field to appear in my table.  I am keeping up with a roster list for our contract agencies.  I started with the Table and created multiple fields, including Last Name, First Name, and Full Name.  I would like to use the Full Name as a Lookup field for future tables.  I have been able to modify the field on the form to build an expression combining the First and Last Names, and that works.  However, the data result is not saving to the table and is only visible on the form itself.  Therefore, is not available to build queries, Combo Boxes, etc.  I have tried to create a calculated field directly on the table and the expression builder will not appear.  It merely saves my expression as the text for that field.

There has to be an automated way of doing this to prevent having to manually key the separate fields.  I have tried using Default Value in Table Design mode and it tells me my expression is invalid (even though the same expression works on the form).

What am I doing wrong?  Is there a simple setting I am overlooking that will fix this?

Thank you!

dirk goldgar say:

Most commonly, you would not want to build a calculated field in the table for this, because Access won't currently let you index the calculated field.  Instead, you'd have only first name, last name, and any other "name-part" fields in your table, and you would base forms & reports on a query that creates a calculated field *in the query* that constructs the name; for example, a field defined like this in the query's design view:

    FullName: [LastName] & (", "+[FirstName]) & (" "+[MiddleName]) & (" "+[NameSuffix])

Anything based on that query would have access to the FullName calculated field.  Combo boxes designed to lookup a specific individual by name would use a similar expression in their RowSource queries, but would also include the ID or other primary key of the record representing that individual, so that any subsequent actions involving the selected record can use the ID to efficiently locate the record.

All that said, if you're using any version of Access from A2010 on, you ought to be able to create a calculated field in the table, that constructs the full name from other fields.  Did you choose the "Calculated" field type for your field?  If so, what expression did you enter for the field?

a. payne say:

Thank you for the information. I had tried to set up a calculated field on the table, but after specifying a calculated field and selecting a text field, it did not bring up the expression builder.  I entered my string as follows : =[First Name] & " " & [Last Name] and it merely inserted the string as the text for that field.  That's when I went to the form and was able to get the expression to work there, but there the data sits.

I went into the query and was successful in entering the expression there.  Thank you so much for your assistance!!

dirk goldgar say:

I don't know why the expression builder didn't come up for you automatically.  Failing that, you ought to have been able to click on the calculated field's Expression property, have the "builder" button (caption "...") appear at the far right of the property line, and click on that button to bring up the expression builder.  If that didn't work, I can only guess that something was wrong with the other properties of the field.

The expression for a calculated field in a table -- as entered in the Expression property -- shouldn't include a leading equals sign ("="), so that was probably why that didn't work.

I'm glad you got it to work defining the calculated field in the query.  For most purposes, that's the better way to do it, anyway.
