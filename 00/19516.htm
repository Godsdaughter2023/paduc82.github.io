In above coding, some of the lines include .cmd commands. I did this because I wanted to give some commands administrator privileges. So the .cmd files contain the admin privilege codes and then the tools code. If you are a standard user, you will be requested to enter admin password each time you run that desktop right-click menu item.

These .cmd files must be made and put into the "Windows\System32" directory. They can be put anywhere but the registry code must be modified so as to point to the new path.

Please also note that the registry keys created must be unique. If they happen to be like one of the keys in the registry, then trouble arises. I chose the keys that way. If you wish to add more items and more keys, you must be careful.

For example: "A2_devmgr.cmd" file includes the following code ( between dashed lines ):

---------------------------------------------------------------------------------------------------------------------------

rem ========= admin privilege codes ===========================================

@echo off

REM  --> Check for permissions
>nul 2>&1 "%SYSTEMROOT%\system32\cacls.exe" "%SYSTEMROOT%\system32\config\system"
REM --> If error flag set, we do not have admin.
if '%errorlevel%' NEQ '0' (
    echo Requesting administrative privileges...

Create Long Lookups, and inserting a field within another field. - Microsoft Community

samedysam say:

Create Long Lookups, and inserting a field within another field.


This is a two part question.

I am trying to write a database to create reports for my class. Now, depending on how the student is doing, they will either get 1 of 5 comments in each section.

I tried creating a lookup for this, but Access gives me the message "The text is too long to be edited". I tried changing the data type to Memo, but then I don't get the option for lookup.

My second question is, is there a way to insert data from one field another field. What I want to happen, some comments, it would look nicer to have the name in the comment section... "Sam is doing very well in the course".. Is there a way to get Access to automatically put that data into the other field?

ken sheridan say: Firstly, never use the Lookup Wizard when setting the data type of a column (field) in table design.  It is, quite rightly, derided by experienced developers.  For reasons why see:

http://www.mvps.org/access/lookupfields.htm

If the text of your comments is too long to be accommodated within a text data type column then you'll need to use a memo field, so what I'd suggest is that in your Comments table you have a separate text column to describe the comment briefly as well as the memo column.  Also use an autonumber column as the tables primary key.  So the table design would be like this:

Comments
.....CommentID (autonumber)
.....ComentTitle (text)
.....CommentText (memo)

As you have multiple sections per report, with a comment for each, you have a many-to-many relationship type between reports, comments and secrtions, so the other tables required for the model would be along these lines:

Students
....StudentID
....FirstName
....LastName

Reports
....ReportID
....StudentID
....ReportDate

ReportSections
....SectionID
....Section

ReportDetails
....ReportID
....SectionID
....CommentID

This last table models the 3-way relationship type between reports, sections and comments by having three foreign key columns, each referencing the primary key of the relevant referenced table.  The tables primary key in this case is a composite one of all three columns.  Notre that while ReportID, SectionID and CommentID are autonumber columns in each of the referenced tables of which they are the primary keys, in ReportDetails they are straightforward long integer number data type, not autonumbers.

For data entry of reports base a form on the R4pots table and within it include a subform based on a query which joins the ReportDetails and Comments tables like so:

SELECT ReportDetails.*, Comments.CommentText
FROM ReportDetails INNER JOIN Comments
ON ReportDetails.CommentID = Coments.CommentID;

Link the subform and the parent form on ReportID.

In the subform you'd use combo boxes bound to the SectionID and CommentID columns, set up so that the text values are shown rather than the underlying numeric ID values, so that for the SectionID for instance would be set up like this:

ControlSource:   SectionID

RowSource:     SELECT SectionID, Section FROM Employees ORDER BY Section;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first dimension is zero to hide the first column and that the second is at least as wide as the combo box.

The combo box for the CommentID would be set up similarly, with its RowSource property being:

SELECT CommentID, CommentTitle FROM Comments ORDER BY CommentTitle;

For the CommentText include a text box bound to this in the subform.  When you select a comment by its title in the combo box the corresponding comment text will show in the text box.  As you would not want to edit this in the subform set the text box's Enabled property to False (No) and its Locked property to True (Yes).

As regards your second question, how you'd do this depends on to what degree you'd want to personalize the comments.  For simply inserting the student's name you might be able to set up the values of the CommentText column in the Comments table so that the name and text can easily be concatenated into a compute control in the final report (by which I mean the Access report object here).  If the personalization is to be more complex, and particularly if you want to be able to manually edit a comment for a student, however, you'd need to assign the personalized text to a CommentText column in the ReportDetails table.  For this, to automate the process, you'd either have to structure the CommentText values in the Comments table so that they allow for easy concatenation with the student's personal data (which would probably mean splitting the comments up into separate columns), or be able to parse the text from a single column so as to build the personalized comments from it and the student's personal data.  The simplest solution, though most labour intensive form the tutor's perspective, would be to simply assign the value from the CommentText column in the selected row in Comments to a CommentText column in ReportDetails and then edit it manually.  Rather than binding a text box in the subform to the CommentText  column from Comments you'd bind one to the CommentText  column in ReportDetails  itself, and assign the value from the selected row in Comments to it using some VBA code in the CommentID combo box's AfterUpdate event procedure.  The value 'pushed' into the bound control can then be edited.  In this case you'd leave the text box's Enabled and Locked properties at the defaults of True and False of course.
Ken Sheridan, Stafford, England
