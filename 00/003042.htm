teylyn say:

Hello,

I don't think this approach will work. You can use a formula on Sheet2 that pulls the data from Sheet1 like

=Sheet1!A1

Then copy that formula across and down.

But, when the data in sheet 1 changes, the formula in sheet 2 will immediately reflect that. You can't use formatting to see any changes, because there is no difference between the two sheets. The changes don't leave a trace. 

You can create a copy of the sheet by right-clicking the sheet tab and selecting "Move or copy". Then select "Create a copy" and hit OK. Now you have an exact copy of the original sheet. You can now apply conditional formatting to highlight cells that differ between the two sheets. 

Select cell A1 the copy and use conditional formatting with a formula along the lines of

=Sheet1!A1<>A1

(Note that there are no $ signs in the formula). Select a color fill for the format and apply that conditional format to all cells in the copied sheet. Now, when you change a value in the original sheet, the copy will highlight the cells that differ from the original. 

erica blair say:

Hi Hitechtoys,

So there are two ways you can do this but they're very "MacGyver"

How you would accomplish this is by doing what was suggested by teylyn, create a sheet that has the following formula in each cell up to the length and width you'd want:

=if(isblank(Sheet1!A1),"",Sheet1!A1)

Simply replace Sheet1 with your front sheet's name and then place this formula in cell A1 on the back page and copy-paste the cell or use the fill handle to drag it to the desired width and height. 

Then do one of the following:

  1. Simple Non-Macro solution:
  • Go to file>options
  • Navigate to the formulas tab
  • Under Calculation Options see Workbook Calculation
  • Select the Manual radio button and check or uncheck the recalculate upon saving box as desired (seems you would want this to be unchecked)

         You will now be able to have your user make changes to the front page and they will only be reflected on the back page when you press F9 to recalculate the entire workbook. This solution of course has a number of limitations including no calculations on the front page updating either.

     2. More complex macro-enabled solution:

  • Save your workbook as a macro enabled workbook (and never again save it without ensuring you're saving as a macro enabled workbook)
  • Unhide the developer tab on the ribbon
  • Go into the macro editor on the developer tab by selecting the visual basic button
  • You should see a new window open up that looks more like a code development environment
  • On the left side of the screen there should be a project browser in which you can see the sheet name of your back page, front page, other tabs, and "ThisWorkbook"
  • Double click ThisWorkbook
  • Paste the following code in the editor:

                    Private Sub Workbook_Open()
                        Application.Calculation = xlCalculationManual

                        Application.CalculateBeforeSave = False

                    End Sub

  • Calculation will be set to manual any time the document is opened and prevent saves from recalculating the workbook
  • Now go back to the macro editor and double click on your front page tab's name, the one that should update as the user interfaces with it
  • Paste the following code in the editor for that tab:

                    Private Sub Worksheet_Change(ByVal Target As Range)
                        Worksheets("Payroll").Calculate
                    End Sub

  • Now when any value is changed on the user-facing tab it will recalculate only itself, preventing the back page from changing while the user is free to work on the front page that will act normally and recalculate on any change made. 
  • All that is necessary to update the back page to match the front page when you're ready is to press F9, which forces a recalculate on the entire workbook.

Please let me know if either of these options ended up helpful to you or if you need more assistance.

Thanks,

EBW

Microsoft Publisher 2013 - Automatically Update Linked Images

This is our biggest problem with Publisher as there doesn't seem to be a way to do this efficiently unlike the standard application in Adobe InDesign. Any input would be greatly appreciated.

kmill097 say:

So I'll answer my own question since I found the solution...

"Linked to file" images are updated through the Graphics Manager in the View tab. Once an image is altered, the Graphics Manager changes the status of the linked image from "linked" to "Original is modified".

The only issue is that you have to check every image by hovering over the thumbnail in the Graphics Manager to see if the original has been modified and needs to be updates. Similar to Adobe InDesign standard, image updates should be highlighted or automatically updated when the program is opened or reloaded. Despite this design flaw, it is comforting to know that Publisher does have the ability to link and "manually" update images. Its a step in the right direction.

lonely in ivory tower_y say:

Any disadvantage or limitation of using MSAccess 2010 Runtime on an MDB created with MSAccess 2003?

I have been reluctant to fully convert my front end/back end application with full workgroup security to accdb file format, because of the loss of ability to restrict access to my design and code.  I have recently been experimenting with opening the .mdb file and distributed .mde file with 2010 runtime, and include the startup options to connect to the secure system.mdw file.

Are there any known problems in continuing to use 2010 runtime this way? I have even made some design changes to the .mdb using the full copy of msaccess 2010, and would also like to know if that will present problems.

Thank you so much.
scottgem say:
A couple of points. If your goal is to restrict access to design and code, then converting to MDE/ACCDE will do that without the need for workgroup security. Using the runtime also prevents uses from getting into design mode. 

Access 2010 is less forgiving of syntax errors than 2003. So its possible that code that runs fine in 2003 may crash in 2010. Anytime you use an mde/accde or runtime versions you need to use a lot of error trapping so errors can be properly identified.

And Albert is right on, do not do any design changes in 2010. Use 2003 for designing.

albert d. kallal say:

I would as a general rule make ALL changes to the database with Access 2003 since that is the version the application was developed with. And if you going to use 2003 or the 2003 runtime in any case then you MOST certainly do not want to use 2010 to modify the application.


If you stick to the above, use late binding, then you should be able quite well to run + use the access 2010 runtime.


However if NO machines are going to use 2003 (full or runtime), and you simply require/use access 2010 runtime in all cases? Then you likely can risk/attempt doing design + modification work in Access 2010. In other words, if Access 2003 (full or runtime) is going to be used at any time then I strongly suggest you do not edit/design with 2010. As noted, if all machines will use/run 2010 runtime then you can risk using access 2010 for development.


> reluctant to fully convert my front end/back end application with full workgroup security to accdb file format, because of the loss of ability to restrict access to my design and code.


Actually, you don't need nor want to use security for the above. Security is to restrict what users can open what tables or forms. To restrict DESIGN changes you should simply distribute a compiled version of your application. When you compile your application (an mde) or now an accDE, then the VBA code and design changes to forms are not possible.


So security never really was designed to prevent changes to code and all these years you really never did have to use security to prevent such changes. To lock down your application then just use a compiled version. A few settings in the startup options and users cannot make changes to the application you distribute.


So don't confuse the security  issues and requirements of that of preventing the user from making changes to forms/code or reports - they are two different goals and the latter does not require Access security to achieve.
