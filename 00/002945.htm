john w. vinson say:

As noted, you want the "Enable design changes" turned OFF. You'll still be able to add and change data in the table, but you won't be able to (nor should you want to!!) change the structure of the table (such as adding new fields) if you're just viewing a datasheet.

The Lookup Fields are working as designed. A lot of us really dislike the Lookup Field feature - see http://www.mvps.org/access/lookupfields.htm for a critique. What a Lookup Field does is actually create a lookup Table in your app, and put a Combo Box control on the table datasheet; this conceals the actual content of the field (the number) behind a looked-up text value.

In any finished application, users should interact with date solely using Forms. They should never even SEE a table datasheet; tables should be kept "under the hood", opened only for design and debugging. It's perfectly standard to have combo boxes - "lookups" - on a Form, but it is not necessary to use a table lookup field in order to do so.

ken sheridan say: As John says, the lookup field wizard should be avoided when designing a table.  That's not to say that you should not have columns (fields) which reference a column in another table.  Quite the contrary in fact as this is the fundamental basis of the tables like this:

Streets
....StreetID  (autonumber PK)
....Street
....CityID (long integer number FK)

Cities
....CityID  (autonumber PK)
....State (text FK)

States
....State  (text PK)

Enforced relationships between Streets and Cities, and between Cities and States should be created to protect the integrity of the database.

Note that both the Streets and Cities tables have autonumber 'surrogate' primary keys.  This is because both street and city names can legitimately be duplicated, so the names are unsuitable as keys.  How many Main Streets or Springfields are there in the USA?  USA state names on the other hand are distinct, so in this case a 'natural' key of the state name can legitimately be used.  Many people would nevertheless use a StateID surrogate key, but unlike streets and cities it's not necessary to do so, and there are situations where the use of a 'natural' key can have significant advantages.

In an Addresses table all you now need is a long integer number foreign key StreetID column and you know which street, city and state the address is in from the value in this single column.  Again an enforced relationship between Addresses and Streets should be created.  For data entry in a form based on an Addresses table you can then use a combo box bound to the unseen StreetID foreign key column, but showing the street name, city and state in its drop down list, and unbound text boxes which show the city and state automatically after selection in the combo box, by referencing the Column property of the combo box.

Colored Cross-references - Microsoft Community

whiterat say:

Colored Cross-references

I write technical documents that have lots of cross-references, to Tables, Figures, Appendices, etc. Lately, many of my clients are asking to have them (the cross-references) be colored blue, I suppose because that makes one realize they are hyperlinks. I can't find a field switch for the REF field that would let me do that automatically as I create them. Any help would be greatly appreciated.

WhiteRat

macropod say:

There isn't a field switch for that, besides which cross-references and hyperlinks are different things (even though cross-references can hyperlink). The essential differences are that cross-references:

1. do not have a blue, underlined, font that changes colour when followed, which makes them better suited to printed content;

2. update automatically to reflect changes in the referenced content, whereas hyperlinks must be edited manually; and

3. can only point to content within the same document, whereas hyperlinks can reference external content.

If your clients want to see where the cross-references are, that's a simple matter of setting Word's 'field shading' option to 'always'.

If you really want to have the cross-references look like hyperlinks:
1. Press Alt-F9 to expose the document's field codes. Each cross-reference will look something like { REF BkMk \h }, where 'BkMk' is a bookmark name.

2. Edit each cross-reference field code so it looks like { REF BkMk \* Charformat \h }

3. Format the 'R' in 'REF' blue, with underlining;

4. Press Alt-F9 to hide the field codes, then Ctrl-A, F9 to update the display.

Note: Even with these mods, the cross-references won't change colour when followed - but all your printouts of the document will now have the blue underlining, which you'll only be able to get rid of by reformatting the 'R's to their original state.
