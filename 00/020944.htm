asbjørn say:

A better solution for backwards compatibility

Microsoft has been doing excellent work with backwards compatibility in Windows for years. It is quite amazing how many old programs just work or only require a compatibility setting.

That being said, at some point it would be beneficial to Windows to shed all this compatibility code. There are simply too many things in Windows that can't be changed for compatibility reasons - things like only having 255 characters in paths, even though NTFS supports much longer paths. Or the common dialogs that haven't been updated for 15+ years. A break is needed at some point.

It think it would be much more useful if a transparent virtualization solution was used for compatibility. Microsoft already has much of the technology needed for this in the form of App-V and Hyper-V. All it would take is to link these up and make it pretty much transparent that this old app is really running on virtualized XP or Windows 7. This would even make it possible to run 16-bit apps on 64-bit Windows.

Sort of like Classic when Mac OS X launched, but a bit more elegant.

HKEY_CURRENT_USER\Software\Microsoft\Windows\Current Version\Explorer\User Shell Folders.

Click Start and in the start search field, type

Regedit

HKEY_CURRENT_USER\Software\Microsoft\Windows\Current Version\Explorer\User Shell Folders

On the right hand pane, check the value data for Cache.

It has to be %USERPROFILE%\AppData\Local\Microsoft\Windows\Temporary Internet Files.

If it is different, double click on Cache and type:

%USERPROFILE%\AppData\Local\Microsoft\Windows\Temporary Internet Files

under value data and click ok. Close the registry and  restart the computer and check if it works.
