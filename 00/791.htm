    ken sheridan say: You cannot amend the system generated message box. To customize it you must call the MsgBox function.  I don't understand your statement 'Yours only had a Yes message'.  All of the MsgBox function calls in my demo pass the vbYesNo constant into the function, which gives you the two buttons, with the Yes button as the default, exactly what you asked for in your original post.

    The message which you have received, 'Do you want to edit the items in the list?' means that a value list is being used as the control's RowSource.  This should only be done where the values are those of an immutable set in the external world, such as days of the week, months of the year etc.  Otherwise the values are data.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.  Consequently to use a value list of data would violate this rule by the encoding of data in the object definition.

    A list of data should always be by means of an SQL statement as the RowSource property, drawing on a referenced table.  The default message for this prompts the user to select a value from the list.  By using the NotInList event procedure as illustrated in my demo this is replaced by the message box called in the event procedure  On the basis of the user's response to the message box, the control is either undone, or a new row is inserted into the referenced table, either transparently by the execution of an INSERT INTO statement, or by opening a form bound to the referenced table in dialogue mode where it is necessary to enter further data in addition to the value of the event procedure's NewData argument, which in my demo is passed to the form as its OpenArgs property, and assigned to the relevant control's DefaultValue property in the form's Open event procedure.

    You say you have opted for the use of a command button to insert a new item.  This is only necessary if the new value is not a candidate key of the referenced table into which a row is to be inserted, as with personal names, whose values can be and often are legitimately duplicated, even in small populations.  I worked with two Maggie Taylors for instance.  Where the new value is a candidate key of the referenced table the use of the NotInList event procedure is simpler and more intuitive to the user.

    A manual page break is actually inside of a paragraph. The formatting applied is that of the first paragraph on the next page. This is counterintuitive.

    Generally, avoid inserting manual page breaks in anything that will be edited in the future. They make for problems. Instead format the paragraph to begin the new page either "page break before" or "keep with next" in the paragraph formatting.
