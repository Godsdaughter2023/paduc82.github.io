daniel g. grau say:
You did exactly what I meant, which actually was "publicly accessible."

What I see in your sample is, those objects are not text boxes but SmartArt shapes. SmartArt is a series of pre-packaged graphics basically meant to be taken "as is." Although some degree of customization is possible, it is not a drawing environment for creating any kind of graphic. Let me quote from this excellent description by Bob Jones:
  • SmartArt is not a feature for creating graphic content of your own design. SmartArt Objects are ready-made diagrams, flow charts, etc. intended to save time & effort. Also, the shapes that comprise a SmartArt graphic are all contained within a single workspace or canvas. Among other characteristics they also interact with each other, so if you add or delete a shape the others scale automatically.

    Although they can be customized to some extent they are not free-form drawing tools. To delete or otherwise modify a specific shape in the diagram click the edge of that shape to select it. You can then press delete to remove the shape, change it to a different shape using alternatives provided by the Shape tool, move & scale it, change its color, etc. Shapes can also be added/removed using the toolbar in the Text Pane of a SmartArt diagram. The Text Pane can facilitate entering text & provides some additional tools for promoting/demoting as well. 
In addition, SmartArt objets contain placeholder text (shown as "[Text]") that appears and disappears from view as you type. If you strip the object to obtain blank frames, as seems to be the case in your file, strange behavior ensues, at least on the Mac, and text disappears for no apparent reason as you move the objects to overlapping positions. Some objects become non-selectable, like the blank frame at the bottom right, although it actually contains a rectangular shape.

What you should have done is create text boxes using the Home tab in the Ribbon. The interaction between text boxes is simpler and easier to handle.

But there may be a simple solution. Save As your file into Word 97-2004 format (.doc). Those SmartArt objects will turn into regular text boxes, some of them grouped. Then, if required, repeat the Save As into .docx format. (You can also use RTF for this intermediate step.)

Note that objects not supported in the older version of Word will become pictures, but they will reconstitute once you save back as .docx. The good thing is that simple SmartArt such as what you are using (containing only rectangular shapes) will remain converted into grouped text boxes. As such, I think you'll find all your problems are easily solved.

Is there any way to catch this error rather than crashing Excel?

Details

I see this error in Excel 2011 for Mac; I haven't tested it on other versions.

The following code fragment illustrates the problem.  Paste this into the VBA code into a code module, and run it.  If you set the constants "bPasteArray" and "bAddError" both True, then running the macro will crash Excel.  If "bPasteArray" is False, and "bAddError" is True, then a MsgBox is displayed ("Application-defined or object-defined error")  If "bAddError" is false, then the macro runs without any problems.


public Sub BugDemo()

    Const bPasteArray As Boolean = false
    Const bAddError As Boolean = True
    Dim dstData(1 To 1, 1 To 2) As Variant
    Dim dstRange As Range, dstTable As ListObject

    On Error GoTo ErrHandler

    dstData(1, 1) = "Sum"

    With ActiveSheet
        .UsedRange.Clear
        If bAddError Then
            dstData(1, 2) = "=1+[@A]"   '  This is a bad formula
        Else
            dstData(1, 2) = "=1+1"
        End If

        If bPasteArray Then
            .Cells(1, 1).Resize(1, 2).Formula = dstData '   Crash happens here
        Else

tucker [msft] say:

Hi, I think I have some suggestions that may help you out.

First, the advice above will let you lock out automatic inking so the pen selects by default, but it won't help force ink on and ignore touch. 

If you're having problems with unwanted touch input, understanding how the system works can help you get better behavior.  When the pen tip is in range for the digitizer to use it, we lock out stray touch input.  Because of this, you're most likely to get those stray inputs when your hand touches the Surface before the pen is in range.  If you get in the habit of "leading with the pen" and bringing the pen tip close to the screen as you move to rest your hand on it, the system will do a much better job of ignoring those random inputs.  The pen doesn't have to actually touch the screen for this to work.  You'll notice that an ink cursor appears when the pen is about a half inch above the screen, which lets you know the pen is detected.

The Surface also does some detection to try to avoid input from your palm and that works best when the contact with the screen is easy to distinguish from a finger.  Even though it may seem unintuitive, it works best if you aren't too cautious about resting your palm on the screen.  When you're trying to be careful and only let a bit of your hand contact the screen it's actually more likely to misinterpret it.  So when your inking letting your had rest fully on the screen is actually best.

Solving "palm rejection" is a tough problem and the Surface team and Office are continually working to improve it, so it will get better over time, but hopefully these suggestions will help you out.

That is not currently possible. If you just want more space between bulleted paragraphs, you can do one of the following:
