the cancel button in the screen of "the following programs are preventing windows from restart" may not work during certain circumstances (forcing closing anyway and losing data)

If you have a program with unsaved data that may prevent a reboot by forcing windows to display the screen "the following programs are preventing windows from restart", you may end up with the CANCEL button not working and act save as the FORCE CLOSE button (instead of halt the restart itself and bring you back to windows where you can save your work)

The steps are the follow:

- have a program in a state that would trigger that screen. From an unsaved word document to a not-responsive application busy in a loop.

- Open settings->Update&Recovery->ADVANCED STARTUP and press RESTART NOW

- the screen "the following programs are preventing windows from restart" will appear with your application that is preventing the operation

- HERE IS THE ERROR: Regardless of pressing FORCE CLOSE or pressing CANCEL, in both cases you will have your applications closed AND you get the ADVANCED STARTUP window. What it should have happened is that, by pressing cancel, the operation is interrupted.

- In the Advanced Startup screen, the only logical option to press (believing to go back to the desktop and save the data)  is "Continue to Windows" BUT it will force a full restart without further question. The unsaved document is lost.

How do I make custom built hyperlinks in word mail merge? - Microsoft Community

schlaefers say:

How do I make custom built hyperlinks in word mail merge?

I'm working on a project that uses mail merges to create reports on users.  However, I want there to be hyperlinks in the reports that get built a certain way.  Lets say one of my fields is <<name>> and another is <<twitter_id>>.  I want the mail merge document to have a field that looks like this:

Name: <<name>>

but <<name>> is actually a hyperlink to "www.twitter.com/<<twitter_id>>".  At the moment I have to do these by hand.

In HTML this would look like 
Name: <a href="www.twitter.com/<<twitter_id>>" > <<name>> </a>

I've been working on this for a while and would really appreciate a solution. 
PS. I'm running Microsoft Word for Mac 2011
daniel g. grau say:
You'd need to enter something like this in the main document:
  • Please visit {MERGEFIELD name}

Then, select the underlined text above, choose Insert > Hyperlink. Type "PLACEHOLDER" in the dialog box. It should look like this:
  • Link to: PLACEHOLDER
  • Display: <<Selection in Document>>

Now go back your document and change:
  • {HYPERLINK PLACEHOLDER}
by adding a merge field so it reads:
  • {HYPERLINK {MERGEFIELD DOMAIN_PLUS_twitter_id}}

Where each of the record fields named DOMAIN_PLUS_twitter_id contain a full path in quotes ("www.twitter.com/twitter_id1", "www.twitter.com/twitter_id2", etc.).

I tried building with just the final portion, as in {HYPERLINK "http://www.twitter.com/"{MERGEFIELD twitter_id}} but it did not work. I had to use the full path between quotes.

After generating the document, you'll need to manually update fields (Select All, F9), or perhaps close and reopen the resulting document. The result will be:

  • Please visit name1 (linking to DOMAIN_PLUS_twitter_id1)
  • Please visit name2 (linking to DOMAIN_PLUS_twitter_id2)

don cuthbert say:

It is not long since build 18362 was an Insider build, but you should not still get Green bugcheck screens. The public build normally would have a BSOD or Blue bugcheck screen during a crash like this.

Check your BCD. At an Administrator Command Prompt, run bcdedit. See if there is any line there that shows flightsigning Yes.  You might see this line under Windows Boot Manager and/or under Windows Boot Loader sections.  If it exsits you can delete it:

bcdedit /deletevalue {bootmgr} flightsigning

bcdedit /deletevalue {current} flightsigning

Also, check under Settings - Update & security - Windows Insider Program. Here you should only see an option to Get started. If it shows that you are in any other state, you can try selecting to leave the Insider Program. If that is not an option, try deleting a registry key, using regedit:

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\WindowsSelfHost

You can delete that entire key.

It is possible that this is just an oversight that is present in the current public build. You have the correct public build number 18362.239.

Of course this does nothing to address what might be the cause of the crash.

There is a utility from Nirsoft called BlueScreenView which might help to diagnose the crash, if it saved a dump file.

Meanwhile, you can try running sfc /scannow. If that finds anything wrong that it cannot fix try DISM /Online /Cleanup-Image /RestoreHealth. Run these from an Administrator Command Prompt.
