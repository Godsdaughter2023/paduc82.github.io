greg_1985 say:

scottgem say:

First a couple of points. Primary keys are used behind the scenes. There is no reason a user will see them, though some developers might expose them as a visible identifier. Normalization rules were first put forth by Dr EJ Codd in the early 70s and have become the standard for relational databases, largely unchanged since then. So they have stood the test of time.

1. If you don't have primary and foreign keys then you don't have related data, end of story. How can you know what customer put in an order if you don't have a FK linking to the Customer record? If you are storing all the Customer info in the Order record then you have a very inefficient database that wastes a lot of space and makes it harder for data entry. You have a relationship when you include a FK in a table linking to a record in another table. You may be thinking about formalizing relationships using the Relationships Window in Access. It is not required to use that. The purpose of formalizing a relationship is to impose referential integrity. While this is a good thing, you can impose RI through your user interface.

Without knowing your  data and table structure, I can't say that you do not have relationships as you seem to think.

2. See above explanation about RI.

3. Again, without knowing your table structure its really impossible to answer.

What I don't understand, however, is if I import two tables -- one table of all transactions, another of just refunds in order to match/reconcile them through queries -- do I really need t normalize them upon uploading them to Access?

scottgem say:

Again, I can't answer that without knowing your table structures and what your queries/reports are supposed to do.

1. If you are not doing data entry, the risks are in importing bad data and having orphan records.

2. I'm not sure what you mean by "behind the scenes". Again, if RI is not imposed you may have orphan records. If the source of your data will prevent that, then you should be fine.

3. If there is a need to normalize data after import and you are consistently importing the same data then you can automate things so after you link to the table a series of queries can append the data to normalized tables.

The process of normalization is to reduce/eliminate redundancy. If you have customer name and address info repeated with each transaction than that is not efficient use of storage.

ken sheridan say: For an example of how to decompose non-normalized imported data into a set of correctly normalized tables see DecomposerDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file takes you through the process step by step, with explanations of the 'append' queries used in each step.  At the end of the process the opportunity is given to add further data, simulating what would happen if you were to import further non-normalized data and repeat the process of decomposition into a set f normalized tables.

Whether you need to decompose your data in this way depends on whether further data will be added or edited in Access, or whether Access is simply being used for presentation of the imported data without amendment.  If the former the data should certainly be decomposed into normalized tables as otherwise it will be open to the risk of update anomalies.

Note that queries do not relate tables, but join them, doing nothing to contribute to data integrity.  Relationships can only be created and enforced via the relationships window (or less commonly in Access, by DDL).  All relationships must be enforced to be of any use in ensuring data integrity.  An unenforced relationship is as useful as a chocolate poker.  

albert d. kallal say:

In your case it not really much of a big deal. The reason of course is you not using Access to edit + maintain the data.

In other words, you are downloading data from some application that likely has the RI (referential integrity) and is already normalized (and referential integrity and normalizing are two different concepts). You can have all the PK and relationships enforced even on a very poorly normalized application.

Since the Access application is NOT creating/editing/deleting the data, then RI and normalizing concepts really only much apply to the ORIGINAL software and programs that create + produce  this data.

While RI likely not important, what can become VERY important is to normalize such data for reporting purposes. It really depends on the data source, and if the data your receive is de-normalized for import purposes. What might occur is then is the “appearance” of a table structure change when in fact no such tables from the source application have been changed.

As long as the tables can be imported and used – then RI and normalize is likely not too big of an issue.

However, if the Access application will be used to edit + update such data over time, then setting up RI and proper relationships between tables becomes far more valuable and warranted.

bob jones aka: cybertaz say:
AutoCorrect does not - cannot - interpret every conceivable misspelling. The algorithms involved work on a branching structure, so once a specific point in that structure is passed the mechanism will not make a replacement based on a "hunch". If you think about it, that's a good thing :-) Otherwise there is great potential for inappropriate substitutions that might go undetected. That's why the error is marked as misspelled but the option to accept the suggestion is at the discretion of the user.

Private Sub PictureName_AfterUpdate()
    If Right(Me.PictureName, 4) <> ".jpg" Then
        Me.PictureName = Me.PictureName & ".jpg"
    End If
    If Dir(Left(CurrentDb.Name, InStrRev(CurrentDb.Name, "\")) & "Pics\" & Me.PictureName) = "" Then

karl donaubauer say:

Hi,

starting with Access 2007 you don't need to code the Picture property any more. Instead it's usually better to use the "new" Controlsource property of the image control. In your example you could replace the Form_Current code by this control source:

=CurrentProject.Path & "\Pics\" & [PictureName]

advantages:

#1: You don't have to code the image setting.

#2: You don't have to code the update. When PictureName changes, the new image will be shown automatically.

#3: You don't get an error message if the picture doesn't exist. Instead the image control just is empty.

#4: It shows different pictures in a continuous form.

Concerning the jpg extension and file existance you could use sth like:

If Not IsNull(Me!PictureName) Then

    If Right(Me!PictureName, 4) <> ".png" Then
        Me!PictureName = Me!PictureName & ".png"
    End If
   
    If Dir(CurrentProject.Path & "\Pics\" & Me!PictureName) = "" Then
        MsgBox "This picture doesn't exist in the picture folder."

        Me!PictureName = "Logo.jpg"

    End If

End If

I would approach it like this:

    SELECT * FROM Table1 WHERE Table1.Date Between Forms!frmReports!txtDateFrom  And  Forms!frmReports!txtDateTo

    DoCmd.TransferSpreadsheet acExport, acSpreadsheetTypeExcel9, "qryExport", CurrentProject.Path & "\ExportedData.xls", True
