rob koch say:

This is a simple to explain problem, since it's completely related to the inability to remove an item from a compressed file contained in the Firefox browser cache.

The issue here is that though a full scan can detect items found in such a GZip packed file via decompression, it's unable to remove it without destroying the integrity of the file that contains it.  For this reason, MSE will not attempt to remove the item itself and instead simply notifies the PC user of its existence.

Since files contained in the browser cache are only normally accessed when the website that contained them is displayed in the browser, in most cases such files can simply be ignored.  Typically, that's why such items aren't detected during the initial download, since unless the malware they contain are successfully decompressed and delivered to the browser's display, it will simply remain in the cache folder in an inactive state until the files this contains are deleted by some other process.

The problem is that many people seem to think that a full scan is somehow "better" than a quick scan, so they run these regularly as if that somehow protects them better.  In truth, this action most often results in exactly the type of issue we're seeing here, with a confused consumer asking just as confused helpers to resolve something that didn't matter in the first place.

The simplest way to remove such an item from the browser cache is to first try the most obvious method of clearing the Firefox cache.  If that is unsuccessful, then its possible the file's attributes have been modified so it can't be deleted, at which point the technical issues that may be involved make uninstalling Firefox typically easier than explaining how to do this manually.

The reason that an MSE's full scan exists is to look for items like malware "droppers" or installers that may remain on the system after a known malware attack.  In this case, the potential issues for false positive or non-active malware detections as is happening in this case, are overweighed by the need to find and remove any potential leftovers from the previous malicious attack.

In most cases though, it's better to use a quick scan for any regular checks both because these are faster, since they look at fewer locations and also because they are designed specifically to look for only active malware in all places they are known to operate on a system.  Only in extremely rare cases might a full scan find something related to malware that's active elsewhere on the system which is somehow hiding its existence from the AV program.

For this reason, most of us who understand this don't even bother to run a full scan except as a sanity check every few months, since unless malware has managed to shut down the AV completely, it's real-time components or quick scan would typically detect any active items first.  If I questioned whether my AV was working properly, I'd also typically use a different, manually operated security program to perform a sanity check in its place.

chip pearson say:

Can you provide an example of the data and the formula you are using? It might be an issue with the displayed value versus the actual value in the cells being averaged. For example, if you have the display rounding to two decimal places, but the number of decimal places in the actual value in the cell is four, the average may appear to be incorrect because AVERAGE (and most other functions) use the actual value of a cell, regardless of how that cell is formatted for display. The displayed value is not necessarily the actual value of the cell. If you compare the results you get in Excel with the results you get with a calculator, you may get inconsistent results if you use the displayed values from Excel in the calculator calculation.

 

For example, suppose you have in cells A1:A3 the values 0.336. If you take the average of these numbers, you'll get 0.336, as expected. However if you change A1:A3 to display only 2 decimal places, you'll get 0.34 in each cell. But the average of the three 0.34 values is 0.34, not 0.336.  It may appear that the 0.336 value is an error, but it is in fact correct.

 

Finally, Excel (and nearly all other software) uses what are called Double Precision Floating Point number to store fractional numbers. This data type is limited to 15 significant digits, and anything beyond that is rounding. This rounding can be compounded in a formula and seemingly erroneous result may appear.

 

Also, it might be that you have a value in a cell that appears to be a number but is in fact a text value, which is ignored by AVERAGE.
