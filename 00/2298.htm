If Right(Me.EmailFieldName, 10) = "lK12.WI.US" Then
    Exit Sub
Else

Save your changes.

The key point in recasting data like this is that rows must be inserted into the referenced table in a one-to-many relationship type before rows are inserted into the referencing table.  Where a referencing table models a many-to-many relationship type, as with ContactEmployers in my file, rows must be inserted into both referenced tables first of course.  My demo uses a simple model, but in a more complex model a referencing table which models a relationship type might reference one or more other tables which themselves model relationship types, in which case rows must be inserted first into the latter of course.

A few comments on your proposed model if I may:

1.  Avoid the generic and semanically maeningless 'ID' as a column name for keys.  I know Microsoft does it in templates and wizards, but I'm afraid this is only one example of bad practice exhibited in some templates.  Always use a term which describes the attribute represented by the column, e.g. CompanyID, ContactID etc, and use the same name for the corresponding foreign key in referencing tables wherever possible.

2.  In ContactPhones and ContactEmailAddr, CompanyID is determined by ContactID so is transitively dependant on the key.  The tables are thus not normalized to Third Normal Form (3NF), and consequently open to the risk of update anomalies.  The CompanyID columns can be deleted from each table with no information loss.

3.  Rather than having CompanyPhones and  CompanyEmailAddr tables, an alternative model would have 'Company' as a Contact value in the Contacts.  You'd then have a simple linear model:

CompanyAddresses>----Companies----<Contacts----<Phones

Public Function MakeWeeklyCalendar(ByVal strTable As String, _
                            ByVal dtmStart As Date, _
                            ByVal dtmEnd As Date, _
                            intStartOfWeek As Integer)
                            
    ' Accepts:  Name of calendar table to be created: String.
    '           Start date for calendar: DateTime.
    '           End date for calendar: DateTime.
    '           Start day of week: Integer.
    
    Dim dbs As DAO.Database, tdf As DAO.TableDef
    Dim strSQL As String
    Dim dtmDate As Date
    
    Set dbs = CurrentDb
    
    ' does table exist? If so get user confirmation to delete it
    On Error Resume Next
    Set tdf = dbs.TableDefs(strTable)
    If Err = 0 Then
        If MsgBox("Replace existing table: " & _
            strTable & "?", vbYesNo + vbQuestion, _
            "Delete Table?") = vbYes Then
            strSQL = "DROP TABLE " & strTable
            dbs.Execute strSQL,dbFailOnError
        Else
            Exit Function
        End If
    End If
    On Error GoTo 0
    
    ' create new table
    strSQL = "CREATE TABLE " & strTable & _
        "(WeekStart DATETIME, WeekEnd DATETIME," & _
        "CONSTRAINT PrimaryKey PRIMARY KEY (WeekStart), " & _
        "CONSTRAINT WeekEnd UNIQUE (WeekEnd))"
    dbs.Execute strSQL,dbFailOnError
    
    ' refresh database window
    Application.RefreshDatabaseWindow
       
    ' fill table with start and end dates of each week
    For dtmDate = dtmStart To dtmEnd
        If Weekday(dtmDate) = intStartOfWeek Then
            strSQL = "INSERT INTO " & strTable & "(WeekStart,WeekEnd) " & _
                "VALUES(#" & Format(dtmDate, "mm/dd/yyyy") & "#,#" & _
                Format(dtmDate + 6, "mm/dd/yyyy") & "#)"
            dbs.Execute strSQL,dbFailOnError
        End If
    Next dtmDate
    
End Function

Your problem lies in the field name. The syntax error comes when you're trying to just input "Total Sales (UnitPrice * Quantity)

The correct field should be the Name: [Field 1] +-*/ [Field 2] so in your case it should look like this;

Total Sales: [UnitPrice]*[Quantity]

Then for your criteria you should have >300 which I see you have.

For the Total Revenue you will probably run into the sub query issue because it wants you to use the "Total Sales" - "Coupon * Quantity) the best work around for this and the quickest way is just to use

Total Revenue: ([Quantity]*[UnitPrice])-([Coupon]*[Quantity])

Hope this helps, just remember that just because your assignment says Total Sales (This * This) that isn't how it's input into the field. Also there is a "builder" under the "Design" Tab in the Query Setup area, don't be afraid to use it. Good luck and hope this helps you.

ken sheridan say: One other important point to note is that the tblEmployee.EmployeeID column should be included in both the SELECT clause and the GROUP BY clause.  Personal names can legitimately be duplicated, so it is essential that they be differentiated by the key in the absence of some other column which can be guaranteed to do so.  As this is an academic assignment the tblEmployees table probably has distinct personal names, but in the real world that might not be the case; I once worked with two Maggie Taylors.

I would also recommend that you get into the habit of using the Boolean constants TRUE and FALSE rather than Yes and No.  Aiming for portability is good practice when writing queries.  Other points of good practice to note are that spaces should be avoided in object names, use CamelCase, e.g. FirstName or represent a space by an underscore character, e.g. First_Name; and avoid special characters like the question mark in object names.  Active is sufficient as a column name; a question mark can be added to a bound control's caption in the data input form if desired.

suzanne s. barnhill say:

Are you referring to the Footnote/Endnote Reference Mark and the Footnote/Endnote text? If so, you may just need to check the formatting of those two styles.

The Footnote/Endnote Reference Mark style is a character style and should be defined as Default Paragraph Font + Superscript. If it contains any other formatting (such as font formatting), you should remove it.

The Footnote/Endnote Text style should be based on Normal, with the font size set to 10 points and the spacing set to Single, with 0 points of Spacing After.

If you have not set the default document font to Arial 12 (which should also be the font of the Normal style), then some styles could reflect other formatting. For more on setting defaults for documents and templates, see http://wordfaqs.ssbarnhill.com/CustomizeNormalTemplate.htm#Word2010Font
