tina chen msft say:

Hi Andrei,

Thanks for posting back. I would like to share with you information below:

Giving the URL of the root folder to users doesn't mean giving permission of the root folder to them.

When they open the URL and try accessing the root folder, they will only see the folders they have permissions.

For the users don't have folder 3's permission, they have limited access permission to the root folder, so they won't see folder 3 when they open root folder's URL.

    albert d. kallal say:

    Ok, this is a really cool and great question.

    Since this is a web database, then we don't have update query, and we really cannot start copying records all over the place for this to work.

    However, can we setup a macro for this?

    Yes.

    The first trick here that we DO in fact need a max function and I show how to do this.

    So, lets break this down.

    First thing we need is to build a query, and that query will be build like this
    (order of columns is important).

    Id,      Project,    EntryDate,     LastRecord
    Sort: Descending, Descending 

    So we sort the query by descending Projet and THEN by descending date.

    The sort of above query results in this setup:

    ID Project EntryDate LastRecord
    6 D 2/22/12 True
    8 B 2/26/12 True
    5 B 2/25/12 False
    9 B 2/21/12 False
    7 A 2/29/12 True
    3 A 2/24/12 False
    4 A 2/23/12 False
    1 A 2/22/12 False
    2 A 2/21/12 False

    Note how the top most project ALWAYS comes first with this query. If we filter this query by any project, the FIRST return value record will thus be the TOP most or "max" project date. We just build our custom max function here.

    We going to use the above cool query in several places and as such it make this task quite easy.

    Ok, the first macro code we will write is to set/test if current record we are just adding (or edit) is the most max. This code is easy. We place this code in the before change event.

    It will look like:

    Look Up A Record In   qryLastP
    Where Condition = [Project]=[tblProjects].[Project] And [EntryDate]>[tblProjects].[EntryDate]
    Setfield
       Name   tblProjects.LastRecord
       Value = False
       StopMacro
    /* if we get here, then we are the max record
    SetField
        Name    tblProjects.LastRecord
       Value  = True

    So the above is quite simple, it simply sets if the record we just added is the max record.

    Next up is we need code to un-check and un-set the previous record that is set as last record. We will place this code in the AFTER update event (and we also need the same code in the after insert event since only one of them fires (update on update, or insert after inset).

    Since we need the code in two places, then we use a named macro.

    And I am gong to toss in the code lodgic for a "change" of project.

    So following code is saved as a named macro, and called from both after insert and after update.

    This first part says:
    Any record "less" than current, and lastrecord = true means we have to change that record. (might be none, but if there is one other record = true, we change it to false since it is not the last record anymore).

    The code is:

    Look Up A Record In qryLastP
    Where condition  =  ([Project]=[tblProjects].[Project]) And ([EntryDate]<[tblProjects].[EntryDate]) And ([LastRecord]=True)
        Alias   rstLast


    Edit Record
         Alias rstLast
       setField(rstLast.LastRecord,False)

    next part is for "change" project

    If Updated("Project") Then
       Look up A record in qryLastP
        Where Condition = ([Project] = [Old].[Project])
       If [rstLast].[LastRecord] = False then
           EditRecord

     

        Alias  rstLast
        SetField (rstLast.LastRecord, True)
       End EditRecord
     End If
    End if


    That is it. You call the above code from both after insert, an after update.

    Once all of the above is setup, then editing project, the date and even changing the project will automatic update the table.

    The result is that the last project will always have the LastProject column set = true.

    Hum, it should not matter if you change an existing record to a new never before used project.

    If it is a new project, then the before change event will STILL set this correctly for you.

    Look CLOSE at the before change event code.

    There is a "stop macro" if it finds + sets a record greater then.  Since there is no project, then that first part will ALWAYS fail.

    If this is not working, then your code indent might be wrong.

    Here is a screen shot:

    https://public.bay.livefilestore.com/y1pB-2pow1hWbZEfiCLdU3QIdn3kFD5tCkvgR8Yj8iS23adjKzLODiQUVFJ8s3LLvU8ju-LBQwQbWjvxAavMjPBLw/mlast.png


    Remember, the "code block" of the lookup record is important here. That code is "indented" inside the lookup record. If the lookup record fails to find anything, then ALL CODE inside of code block does NOT run.  In the above, the code editor is shading the code that will run for the Lookup (I did not do that shading).

    If that lookup fails, then none of the code inside runs and we drop out to the next part that says if we get here, then we are max.

    So the before change event takes care of this setting for you, even for a new record.

    So, double check your code indent (the lookup record is much like a if/then block of code – if the fails, then no code inside runs).
