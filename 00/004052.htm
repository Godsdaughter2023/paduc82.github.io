ken sheridan say:

As, from what you say, the Comment column appears to be functionally determined solely by the key of the OrderLines table, by virtue of some of the comments being specific to an individual order line rather than the order per se, it should be a column in that table, but also having a Comment column in the Orders table is unnecessary.  Instead use an unbound text box, either in the parent form or, and this would be my preferred solution, in a separate form which can be opened  in dialogue mode from an 'Update All Comments For This Order' button in the parent form, passing the current OrderID value to it as its OpenArgs property. 

In the dialogue form's Open event procedure populate the unbound text box with the current comment value:

    Me.txtComment = Forms("YourParentForm").Comment

A 'Confirm' button in the dialogue form can then update all rows in OrderLines with the OrderID value in question by means of code along these lines:

    Dim strSQL As String

    strSQL = "UPDATE OrderLines " & _
        "SET Comment = " "" & Me.txtComment & "" " & _
        "WHERE OrderID = " & Me.OpenArgs

    CurrentDb.Execute strSQL, dbFailOnError

However, I'm not over the moon about the use of a single memo (long text) column in this context.  It sounds to me from your earlier statement, 'I update frequently with additional comments' that you are using it as a data structure to store a set of comments.  String expressions should not be used as data structures.  A better design would be to have a separate Comments table related to OrderLines in which each row would store one comment per order line.  You would need a more developed interface than a simple split form, which is something I'd never use; it's a cheap and cheerful solution, but a very crude one in my view.
