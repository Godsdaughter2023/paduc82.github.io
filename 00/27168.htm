How to set the combo box select the first item by default in an Access Web database

albert d. kallal say:

Local variables in VBA, and in fact for access macro's and in fact for most languages means exactly that concept of local. So a local variable ONLY exists in the current routine. The instant you exit that routine, the variable does not exist.

So, those local variables are only useful for the code that running and the very name and "local" means that the variable will not been seen by any other code or routine. This is usually a good thing as then you free to write many many routines and NEVER have to worry about if some other of the perhaps 100's of routines uses the same variable name by accident. So as  general rule local vars are good as then one can take some code and copy over again for use in other routines.

 

When I run the code the values of the local variables are NEVER effected by any other routine in the whole system (even those that use the same variable names). So the very concept of local variables is that of not stepping on the toes of other routines and thus those other routines will not see or be effected if you happen to use the same variable names over and over in the application. The allows one to build complex applications and not have to worry about some variable be set or used by other routines (because they are all separate and "local" to each routine).

 

If you need to share variable values between routines, then you use what is called global variables or in the case of macros, TempVars. However, data macros run on the web server and the temp vars you set are running on your desktop in the browser. You will notice that data macors DO NOT have SetTemp vars command and this is due to where the data macros are running (they are not on the desktop, but running on the server).

 

Data macros ONLY have local vars and thus the values can't be shared between routines. The only way to thus get values returned or "let out" of a data macro is to use the return vars feature.

 

Also your question as stated simply asks for the FIRST value out of a table. I not sure why you have all kinds of parameters and all kinds of where conditions here when you simply state that you looking to grab the first value from the table?

 

A simple data macro to grab the first value out of a table looks like:

 

Look up a record in   tblMonths

    Where Condition  [id] > 0

              Alias

 

   SetReturnVar

             Name  FirstID

       Expression       [ID]

 

 

Note the use of Set return var. This is how you get values "out" of a data macro, or return values from data macros.

 

So, the UI macro (user interface macro) in the form can thus get the first value by going:

 

RunDataMacro

  Macro Name       tblMonths.GetFirst

 

 

Now that the data macro been called, we can inspect and use the return vars of FirstID and set the combo box. We thus can now set the comb box:

Eg:

 

SetProperty

   Control Name       cboMonth

     Property         Value

       Value          =([ReturnVars]![FirstID])

 

So, the above is all you need to get the first value out of the table.  All your additional stuff such as where clauses and parameters and all that extra stuff you have is not needed to simply grab the first value out of that table. Note that you can also use a query in place of the table name and thus that query can set the sort order. If you base the combo box on that query, then the data macro can also use that query name in place of the table name.

 

I would also suggest that if the form has to be loaded lots of times during program use, then there is a fairly big performance cost to this operation since the database engine has to grab this value on the web server side and THEN the value has to travel down over the internet into the users browser for use with code that runs on your desktop.  If this operation has to occur a lot, then you might want to load up the first value into a global variable (tempvars) once in your application startup as opposed to each time the form loads. This is not necessary, but only a good performance tip and suggestion due to the high cost of pulling that value down over the internet. Setting the value once at startup means you would only have to execute this code once as opposed to over and over.

The most simple solution is to set the combo default value to whatever the first value is. For example, if you have a combo box that lets you select the month, then simply set the default value to 1 in the data tab.

 

Another way is in the forms on-load event, you can set a value by going:

 

SetProperty

  Control Name   cboMonth

      Property   Value

         Value   1

  

In the above examples it is assumed you know the 1 value. If you can't know the value ahead of time, then you could certainly call a data macro that plucks out the 1 value out of the table. I think setting the default value is most easy. (but, does assume you know the 1 value ahead of time).

 

There no collum() option in web forms. As mentioned, if the above two ideas don't work, then the last idea of calling a data macro to get the first value and then setting SetProperity as above is certainly possible if the value has to come from the table and can't be known ahead of time.
