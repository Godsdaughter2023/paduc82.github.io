ken sheridan say: I'm not quite clear from your description whether there are independent relationships between patients and patient medications, and between patients and visits, as in Scott's suggested model, or whether the relationship is between visits and patient medications:

Patients---<Visits----<PatientMedications

i.e. medications prescribed or given to a patient are recorded in relation to each visit.  If this is the case then, with Scott's model, you'd simply remove the PatientID column form his tblPatientMeds and substitute it with a VisitID foreign key column.

With regard to medications prescribed to patients you'll find an example of a simple model for this as Prescriptions.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo also model the drugs in terms of the forms and strengths available in each case.  Take a look at the relationships window to see how this is done.  To extend the model so that it catered for prescriptions per visit would simply be a case of inserting a Visits table between the patients and PatientPrescriptions table, removing the PatientID foreign key column from the latter and substituting a VisitID column.  The PatientPrescriptions table table would no longer require a DatePrescribed column of course as there would be a date in the refernced Visits table.

One thing to note about this demo is that, with the exception of PatientID autonumber keys are not used, but 'natural keys'.  This is done explicitly to allow for easy correlation of the combo boxes in the forms when selecting drugs for prescription in the subform within the patients form, so that when a drug is selected only the strengths and forms in which the drug is available are listed for selection in the combo boxes.  It's this correlation of the controls for ease of data entry which is really the main point of this demo file.

ken sheridan say:
It may be helpful to keep medication data on the visit table.

It would be wrong to include any columns for medication in the Visits table as this would involve the introduction of redundancies into the table.  The prescription or administration of medication to a patient is a separate entity type, not an attribute of the Visits entity type.  The former is related to the latter by means of a foreign key in a PatientMedications table or similar.  This table is in fact modelling a many-to-many relationship type between the Visits and Medications entity types, so would also have a foreign key referencing the key of a Medications table or similar, though in reality the model is likely to be a little more complex, as the Prescriptions demo to which I referred you  illustrates; and moreover, the demo is itself a simplified model which is really only intended as a starting point for a fully developed model.

It might be better to think of the Visits entity type in a more generalised way as PatientEvents or similar, of which 'visit' is just one of a number of possible attribute values.  The word visit has quite a specific connotation, and would suggest that medication is prescribed or administered solely in this context, whereas a more generic PatientEvents entity type, modelled by a table, would allow the context to be 'visit' or one of any other type represented by a different value in an EventType column.
