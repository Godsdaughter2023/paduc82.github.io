graham mayor say:

I don't think a rule would help. You could use the startup event in Outlook to run the code every time Outlook starts, but how are the messages in question identified, when there are clearly other unrelated messages in the inbox that should not be moved?

At its most basic (and taking no account of the type of message)  the code (in the ThisOutlookSession module) would be

Private Sub Application_Startup()
Dim olNS As Outlook.NameSpace
Dim fldr As MAPIFolder
Dim olItem As MailItem
Dim strFolder As String
    Set olNS = GetNamespace("MAPI")
    Set fldr = olNS.GetDefaultFolder(olFolderInbox)
    fldr.Items.Sort "[Received]", True
    For Each olItem In fldr.Items
        If Format(olItem.ReceivedTime, "yyyymmdd") <= _
           Format(Date - 2, "yyyymmdd") Then
            MsgBox olItem.ReceivedTime
            strFolder = "Info" 'The name of the folder to move it to
            olItem.Move Session.GetDefaultFolder(olFolderInbox).folders(strFolder)

    Next olItem
lbl_Exit:
    Set olNS = Nothing
    Set fldr = Nothing
    Set olItem = Nothing
    Exit Sub
End Sub

to which you would have to add the conditions to identify the messages in question and the individual folders to move them to.

Yep:  That was the idea, to make you revisit it from the basics.  And I am pleased you got it sorted the way you want it.

I was constrained in my answer by wanting to avoid suggesting that you start again with unformatted text and re-make the document.  I would probably have found it quicker to do it that way, because because I have all the tools and styles and stuff already set up.  You did not have that option. And I know that in a Word document, "everything depends on everything else" so once you hit something fundamental such as margins or headers and footers, there's a lot of knock-on effects.  I tried to save you from the really fiddly stuff.

I don't think I gave you any 'personal' preferences: most people doing long documents direct to press from Word would do the things I do, because we find it gets you a consistent result faster and easier.

Yes, sometimes you "have" to split paragraphs.  This becomes murderously difficult if you have empty paragraphs in the text: Word then loses control of its widows and orphans (which are not very good to begin with).  If you have empty paragraphs in there, take them out, and add 24 points of space after to your body text style.  That gives you the same spacing, and enables Word to control the paragraph splits properly.


When you turned on "Different Odd and Even", you "revealed" page numbers that were already there.  That's why I avoid Word's Page Number control on the Document Elements tab.  Instead, I teach people to go into the header or footer where they want the page number, then use Insert>Page Numbers... from the menu.  That places a { Page } field at the insertion point location in the header or footer, instead of in a floating text box on headers and footers you can't yet see.

That's a bit convoluted...  There's three headers and three footers in each section.  When you use the Ribbon control, Word creates page numbers in all the headers or footers, including ones you can't see.  When you switched to "Different Odd and Even" for double-sided printing, you then got to see page numbers you didn't realise were there.  It's not a "great" design. :-)

I suggested that you set your body text styles to have ONLY "Space After", and 0 "Space Before" to avoid "Ragged top margins".  That's what is causing your text lines to not line up.  If you make the change to that style, the top line of text will sit on the top margin of both sides, and they will line up.  You would also need to get the empty paragraphs out, to enable that to work.

Following the rule of software that "Any bug that is documented in the help is a 'feature'" the presence of the empty paragraph mark in the footer is a "feature".

This is really arcane so I will skip the detail: but headers or footers do not exist in the document until you ask for them.  When you check "Different Odd and Even" the left and right headers are "created" but they are empty.  In other words, the "container structure" exists in the document code, but so far, there's nothing in it.  When you edit a header or footer, even if you do not put anything in it, it is "instantiated" -- i.e. it now has "content". That content will be the empty paragraph, which is actually not 'empty' to Word, it is a container that Word requires to contain all the formatting of the header or footer.  Once the header or footer has been instantiated, that paragraph remains, even if there is no text in there to format. You can't delete it without breaking the document.

This is really difficult to find in the documentation (note: I am not providing a reference because it would take me ages to find it...).  But it is "the way Word works".  It is not a bug, it's "working as designed".  I can think of a few designs I would have preferred to this, but the chances of getting them to change it now are 0, and any change would break a few hundred million documents out there :-)

Hope this helps

It was the only problem I encountered. I made no attempt to fix the code. I didn't notice anything else right off the bat that would cause troubles.


The most common troubles people have when they want to program cross-platform is when working with files and folders. Ron's site does a great job explaining how to get around most of these differences.


Another common problem is when Excel for Windows users incorporate open source Active-X controls. The open source community never ported Active-X to the Mac, so no aspect of Active-X works on the Mac. Use only built-in Excel controls aka "legacy" controls.


Yet another problem is that the Ribbon on the Mac is not programmable. Use the Toolbars collection, instead.
