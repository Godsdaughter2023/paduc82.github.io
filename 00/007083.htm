ken sheridan say: If you have a referenced table, Computers, the referencing table should have a foreign key column only, referencing the primary key of the referenced table.  The referenced table's other columns should be functionally determined solely by the whole of the table's primary key.  This is not the case with the table you illustrate, in which Make is functionally determined by Model for instance, and is thus transitively dependant on the key, which violates Third Normal Form (3NF),

Moreover, while it is conceivable that each machine is used by one user only, is it really the case that each machine has only one program installed?  A more likely scenario is that there is a many-to-many relationship type between computers and programs, which would be modelled by a table which resolves the relationship type into two one-to-many relationship types.  Consequently there would be no Program column in the Computers table.

As regards buildings, the relationship type between a Buildings table and a Computers table would be one-to-many, so a building number would map to multiple rows in a Computers table.   

ken sheridan say:

.................what I'm trying to do is create a relationship between tables so that when we pull a report and need all the data from multiple tables that relate to each other we can display it all in one place.


For that you firstly need to ensure that every table models a valid entity type, e.g. computers, buildings, users etc, and that each non-key column in each table is functionally determined solely by the whole of the table's key.

Relationship types between tables can be unary (one-to-many) e.g. the relationship type between computers and locations is one-to-many, as a compute can be in one location only, but a location might hold one or more computers.  Or the relationship type can be many-to-many (binary, ternary etc), as in the case of the relationship type between computers and programs referred to in my last reply, which is a binary relationship type.  This would be modelled by a table like this:

ComputerLocations
â€¦.ComputerID  (FK)
â€¦.LocationID  (FK)

The primary key is a composite of the two foreign keys.  The table might contain other non-key columns which represent attributes of the relationship type, but these must be dependant solely on the whole of the key.

With a correctly normalized model, which accurately reflects the reality, the production of a report is a relatively straight forward task of joining the relevant tables on the keys in a query, and returning those columns from each table which are to be output in controls in the report.  The report can be grouped as necessary to eliminate duplication, or subreports can be used to reflect a one-to-many relationship type.

It is important to understand than a relationship type is really just a special kind of entity type, so there can be relationship types between relationship types, each modelled by tables.  You might like to take a look at Relationships.zip in my public databases folder at the link I gave you earlier.  This little demo file illustrates how relationship types are built up across a database, using a simple medical prescriptions database as its example.

john w. vinson say: You may be making the common mistake of assuming that a Report must be based on a Table containing all the information you need to display. Of course this is not correct - all Reports I've ever built are based on multitable Queries! Why do you feel that the Category or Location data must be "in" the Inventory table? The Inventory table should (and probably does) have Foreign Key fields linked to those other tables, so a Query joining the tables on those keys will contain all the data you want to see (and will be fast and efficient, using the Indexes created when the relationship is established).
