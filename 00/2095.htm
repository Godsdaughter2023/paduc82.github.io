rohn007 say:
Maybe not yet.

In general, unless you have a specific need to support someone on an older version you might as well uninstall it when you get a new version. 

Right now there are 3 reasons for keeping the old version.

One is that if you are moving from 2003 / menu to Ribbon it is a steep learning curve. So you may want to keep the old one around until you are comfortable with the new version. That way you can fall back to the older version when you are in a hurry and "just have to get work done ...".

One disadvantage is the the September Windows Updates introduced a problem that reset windows defaults to point away from 365 to the older version if it was present.

Another good reason for removing an older version is if your hard drive is limited on space.

Problem with delete query in access - Microsoft Community

nadia galway say:

Problem with delete query in access

Hi ,

Trying to delete records based and matching field names but when i try to delete it says cannot delete from specified table 

DELETE [tbl_Mc Lernons Final].*, [tbl_Mc Lernons Final].Repeat

FROM [Tbl_Exception Summary form numbers] INNER JOIN [tbl_Mc Lernons Final] ON [Tbl_Exception Summary form numbers].[form number] = [tbl_Mc Lernons Final].Repeat

WHERE ((([tbl_Mc Lernons Final].Repeat)=[Tbl_Exception Summary form numbers].[form number]));

Any ideas

Thanks

ken sheridan say: You appear to be attempting to delete those rows from tbl_Mc Lernons Final which have a matching row in [Tbl_Exception Summary form numbers.  If that is the case, then I would normally uses the EXISTS predicate rather than a JOIN in this context:

DELETE *
FROM [tbl_Mc Lernons Final]
WHERE EXISTS
    (SELECT *
     FROM [Tbl_Exception Summary form numbers]
     WHERE [Tbl_Exception Summary form numbers].[form number] =
     [tbl_Mc Lernons Final].Repeat);

This assumes that tbl_Mc Lernons Final is the referencing table in the relationship.  If it is the referenced table, however, and referential integrity is enforced, as it should be, then you will not be able to do so, as the DELETE operation would violate referential integrity.

BTW, in your original SQL statement the WHERE clause is redundant as an INNER JOIN returns only those rows where there is a match on the columns in the JOIN clause, which in your case are the same as those in the WHERE clause.

add this function in module:

Option Compare Database
Option Explicit

Public Function SimpleCSV_A(myField, myTable, myFieldCond, myFieldVal, myFieldType, Optional strDelim As String = ",") As String

 Dim db As DAO.Database
 Dim rs As DAO.Recordset
 Dim strCSV As String

Set db = CurrentDb()
Set rs = db.OpenRecordset("Select " & myField & " From " & myTable & " Where [" & myFieldCond & "] = " & IIf(myFieldType = "T", "'" & myFieldVal & "'", myFieldVal), dbOpenSnapshot)

 With rs
     Do While Not .EOF
         strCSV = strCSV & strDelim & .Fields(0)
         .MoveNext
     Loop
     .Close
 End With

 SimpleCSV = Mid$(strCSV, Len(strDelim) + 1)

Set rs = Nothing
Set db = Nothing

End Function

melepoyil say:

Hijri to Gregorian convertion

Hi,

I have the following data extracted from a database

341113
330702
340121

It is Hijri date in yymmdd and the data is in text format. The same should look like the following in yyyy/mm/dd format.

1434/11/13
1433/07/02
1434/01/21

I am trying to find a way out to convert the data to yyyy/mm/dd format as above and then get the equivalent Gregorian date.

Thanks and regards, 
jp ronse say: Hi,

To convert 341113 to 1434/11/13 use the formula:

=TEXT(14&A1,"0000\/00\/00")

To convert to a Gregorian date, see the following links:

http://www.access-programmers.co.uk/forums/showthread.php?t=246559

It provides a UDF for this: StDteGregOfStDteHijri

The following linkn is an on-line convertor:

http://www.islamicfinder.org/dateConversion.php?mode=hij-ger&day=21&month=1&year=1434&date_result=1

The UDF can differ 1 day from the on-line convertor.
