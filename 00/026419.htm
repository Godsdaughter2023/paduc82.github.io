How can I right align chart vertical text labels? - Microsoft Community

jc-- say:

How can I right align chart vertical text labels?

I am frustrated by the fact that chart labels on the vertical axis align center, which looking unprofessional. How can they be RIGHT aligned, i.e., when two lines of text, the align right, closest to the graph?
sandy johnson say:

It is a pain, but there is no native way in PowerPoint to change the alignment. Here are a couple of work arounds:

 

  1. Depending on the number of data points in say, a bar chart, don't show a vertical axis at all. After you've entered the data, delete the axis, then show the data points above or inside the bars. If you have a lot of data points this will get too clutter, but is actually my preferred method of showing data when possible.
  2. Delete the vertical numbers then with the chart selected, create your own text box and right align the numbers. Adjust the plot area to move to the right so you get the room to add the box. If you do this within the chart, it will travel with the chart as a chart element (but will not change if you edit the data).

a. user say:

The first thing I would do is 

 a. if the workbook was closed when you started the merge, try opening it in Excel befoe you open the mail merge main document. If it was open, try closing it (I think the first of those is more likely to have a positive outcome). Things work a little differently when the workbook is open.

If that is no help, and if your spreadhseet is fairly simple (no formulas, just simple tables) I would try to reconsitute it by exporting to .csv format, creating a new workbook, and re-importing. Otherwise, I would consider

 b. exporting the sheet to .csv and using that as the data source

 c. select the data in the sheet, Edit->Copy, Edit->Paste into a new Word document, save that, and use it as the data source.

The background is that when Word tries to connect to an Excel workbook, it first tries using an OLE DB provider. If that fails (which it does in your case), it tries the ODBC driver, which behaves the way you describe. If that fails, it tries the old DDE method. 

I don't know why the OLE DB provider fails, but there must be something about the workbook/worksheet that it does not "like". In previous experiments, actions such as creating a database /sometimes/ seem to have caused problems, even after the database/list has been removed. 

If the OLE DB provider fails, I doubt if the ODBC driver will succeed. The only way you can choose another workbook is in the dropdown - if the one you need isn't listed, you can't get any further. If the workbook has a long path name, you won't in any case be able to see the whole pathname and selection becomes a matter of guesswork. Even if you manage to select the right workbook, you have to clcik Options and check all four boxes in order to see the various sheets and named ranges in the workbook.
