xps350 say:

There are two ways to solve this.

First option is to store the actual prices with each transaction. You will have the actual prices in the table with articles. Each time a transaction is stored you duplicate these prices to the transaction table.

In the second option you have a price table related to your table with articles. Fields in this table are: ArticleID, Cost, Price and StartDate. Now you do not store prices in the transaction table. On generating a transaction report you look up the prices valid on the transaction date.

Option one is easy to build, but you won't have full price history there.

scottgem say:

This sounds like a standard order entry system. I don't see any special needs here. 

You are encountering one of the very few exceptions to the rule that data should only exist in one place only. That is for time sensitive data. Pricing is the prime example of that exception. The general way to handle this is by having a field for unit price in your transactions table. So when a product is selected for a transaction the UnitPrice field in the transactions table is automatically filled in from the UnitPrice field in the Products table. 

The way this is commonly done is to use a combobox on the form where you enter the Transaction to select the Product. The RowSource of this combo would include the UnitPrice so it would look like this:

SELECT ProductID, ProductName, UnitPrice

FROM tblProducts

ORDER BY ProductName;

Then you use the After Update event of the Combo to run one line of code like:

Me.txtUnitPrice = Me.cboProduct.Column(2)

This will populate the UnitPrice control (bound to the UnitPrice field in the Transactions table) with the current value of the UnitPrice from the Products table. So it will be stored with the transaction and will not be changed when the UnitPrice is updated in the Products table. 

The other option is to maintain a history of prices and when you create your reports pull the price from the record for the date of the transaction. 

As xps indicated, this is much more complex. The way I described is the more common approach.

ken sheridan say:

For an example of what the other shave described see InvoicePDF.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file is primarily intended to illustrate how to output reports as PDF files, but it is based on a simple hypothetical invoicing application.  Think of each invoice as analogous to a sale in your case.  In the AfterUpdate event procedure of the ProductID combo box in the invoice details subreport the current price (and in my case VAT rate)  derived from the UnitPrice column in the Products table is assigned to the UnitPrice column in the InvoiceDetails table.  Consequently as prices in the Products table change over time, those for each invoice remain static as the price when the invoice was raised.

In relational database terms this is a question of functional dependency, which is a fundamental  concept of the database relational model.  Each non-key column in a valid table is functionally determined solely by the whole of the table's primary key (the table is normalized to Third Normal Form).  In this case the current unit price is determined by the key of Products, while the unit price per sale is functionally determined by the key of InvoiceDetails, so each table is normalized to 3NF, and there is no illegitimate redundancy.  Note BTW that the same is done with customer addresses in the main invoice form, so that if a customer changes address, the original invoice will retain the address current at the time of the invoice.  This is not an absolute rule, however.  It might be considered appropriate for addresses for all invoices past and present to be the same per customer, in which case the address would be referenced, not assigned.

Method 3

If Internet Explorer 9 is still installed, follow these steps:

  1. Log on to the computer as an administrator.
  2. Close all Internet Explorer browser windows.
  3. Start Notepad.
  4. Copy and paste the following HTML text into Notepad to convert the HTML text into plain text: 
    FORFILES /P %WINDIR%\servicing\Packages /M Microsoft-Windows-InternetExplorer-*9.*.mum /c "cmd /c echo Uninstalling package @fname && start /w pkgmgr /up:@fname /norestart"
  5. Click Start and then click Run.
  6. Copy and then paste (or type) the following command in the Open box, and then press Enter:
    cmd
  7. Copy the text from Notepad, paste the text into the Command Prompt window at a command prompt, and then press Enter.
  8. After the command runs, and the Command Prompt window returns, repeat step 7, and then go to step 9.
  9. Restart the computer to complete uninstallation of Internet Explorer 9. When you do this, the previously installed version of Internet Explorer is installed. If the previous version of Internet Explorer is not Internet Explorer 8, you can install and then configure Internet Explorer 8 from the following Microsoft website:
    Download Internet Explorer 8
