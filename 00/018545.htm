ken sheridan say: If using phone numbers as identifiers:

1.  Entering a sequence of numbers is prone to errors, or a visitor might have used a different number last time they signed in.  You therefore need to take account of the visitor inadvertently (or even maliciously) entering another visitor's number, which is then erroneously recognized, or of entering their own number incorrectly.

2.  A visitor might use a corporate number which is shared by others in their organisation, and consequently be recognized incorrectly as another representative of that organisation.  In my own case for instance, while I had a distinct phone number, members of my section did not, and there were many organisations which any of those sharing a single number might visit. Personal names can also be shared of course.  I was once present at a clinic where two patients arrived within minutes of each other, both female, both with exactly the same first and last names and both with the same date of birth.

To cater for this, if a number or name is not recognized the visitor should be given the opportunity to correct the number before being forced to enter their full details via a form.  If a number or name is recognized the further confirmation obtained by displaying some other identifying information which the visitor then can confirm should be kept to the minimum to avoid breaching data confidentiality.  If the visitor's name is used for confirmation, then use a list box to  display this to allow for multiple visitors from a single organisation who share a phone number being in the system.  Extreme care should be exercised in deciding what information to use  to confirm the visitor.  A sign-in system like that you are proposing, where it would be very easy to see another visitor's data, whether by accident of design, even if that is only their name, is full of ethical and legal pitfalls.

Charter,

First, please don't piggyback your question on another thread. This can cause confusion. Please use the Participate link above to add your own question. If you need to you can reference this thread with a link.

The DCount function is used to count the number of records in a table that meet a criteria. I think you would be better off using code like I suggested. 

Sounds like you are trying to do this with macros and macros can be very limiting. Lets look at the code I suggested:

If IsNull(DLookup("[VisitorID]","tblVisitor","[PhoneNo] = '" & Me.txtphone & "'")) Then

      Docmd,OpenForm "frmNewVisitor",,,,acAdd

Else

     DoCmd.OpenForm "frmExistingVisitor",,,"[PhoneNo] = '" & Me.txtphone & "'"

End If

The DLookup looks in your table to see if there is a record in the table that matches the phone number. If there isn't the IsNull function will return True.

So if there is no matching record, the first part will run. This opens the New Visitor form in add mode so its ready to add a new record. If there is match, the IsNull function returns a false whihc will run the Else argument. That argument opens a form for existing visitor filtered for the current visitor. 

Access automatically saves a record when you move the focus away from the form or record. So you don't need to save anything. However, the initial form where the Visitor enters their phone number is an unbound form so there is nothing to save. The only purpose of that form is to enter the phone # so it can be checked.

doug robbins - mvp office apps & services (word) say:

No need for Microsoft to do it.  You can easily do it yourself.

Here is the code:

Dim arev As Revision
With ActiveDocument
   .TrackFormatting = False
   For Each arev In .Revisions
       arev.Range.Shading.BackgroundPatternColor = wdColorYellow
   Next
End With

Needs to be run after you have made the changes.
