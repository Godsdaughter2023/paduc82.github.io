For anyone who had their phone mic stop working after installation of a new screen, I've found out why this happened in my case. I tried all of the solutions in multiple forums and got nowhere. I bought a good quality screen from an OEM supplier but obviously, they missed this quality problem.

ken sheridan say: This is not good relational database design.  Hierarchical structures such as your folder structure, and the underlying topic classifications which this represents, should be modelled as separate tables, one for each level of the hierarchy.  Each table, bar that which represents the topmost level of the hierarchy, includes a foreign key which references the primary key of the next level above it in the hierarchy.  The referencing table, Items in your case, should include a foreign key which references only the primary key of the lowest level in the hierarchy.

You'll find a simple example of this in DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file follow the link to 'Bringing the model together'.  You'll see that in the Contacts table, only a CityID foreign key column is included to map each row to the full location data of cities, regions and counties via the hierarchical chain of relationships.  Consequently there is no redundancy, which would have been the case if all three has been referenced by columns in Contacts.  The risk of update anomalies is thus eliminated.

If you move on a couple of forms you'll find a form which illustrates how the location data is inserted into the Contacts table by selecting first a country, then a region and finally a city in three correlated combo boxes, only the last of which is bound.

In the next form, which illustrates how to retrieve data from the database, you'll see that the report which is opened when a search is made in the form, brings the data from the referenced Cities, Regions and Countries tables together in a query's result table.

Note, incidentally, how the two combo boxes on the right of the form allow the report to be filtered on the combined city and employer values selected in the controls, in addition to being filtered on only one or neither of the two.  This is done by referencing the controls as parameters in the report's query.

ken sheridan say: I've no idea how that reference came to exist in the file.  You can remove it by selecting References in the Tools menu on the VBA menu bar, and then deselecting the reference.  I've uploaded a new copy of the file to OneDrive, without the reference.

As regards the model. It's important to bear in mind that this is what a relational database is; a model of that part of the real world with which the database is concerned, in terms of its entity types, the attributes of those entity types, and the relationships between entity types.  So the first step is to define the referenced entity types which make up the reality to be modelled, and which will need to be related.  In your case these are Users, Folders, ParentFolders, BaseFolders, Files, Levels, Formats, Books.  Each of these is represented by a table whose columns represents attributes of the entity type which are functionally determined solely by the whole of the primary key of the table (see the Normalization demo in my OneDrive folder for more on this).  Note that I've not included Favourites; I'll come back to those.

Next, what are the relationship types between the above entity types? As a file can only be in one folder, there is a one-to-many relationship type between Folders and Files.  Similarly there is a one-to-many relationship type between ParentFolders and Folders, and a one-to-many relationship type between BaseFolders and ParentFolders.  There is consequently a simple liner hierarchy like this:

Files>----Folders>----ParentFolders>---BaseFolders

But what if a file is in a second tier parent folder or a third tier base folder, not in one of the third tier folders?  In a hierarchical structure this is handled by plugging the gaps by the insertion of a N/A row in Folders for each row in ParentFolders, so if there were 10 rows in ParentFolders there would be 10 N/A rows in Folders, each with a different foreign key referencing the key of ParentFolders.  Similarly there would be an N/A row in ParentFolders referencing each row in BaseFolders.  This parallels the geographical hierarchy in my DatabaseBasics demo, so the interface would be similar.  There could be a situation for instance where a contact is recorded only by their region, not by their city, for which a N/A row in Cities for that region would be necessary.

As regards Levels and Formats, I'm assuming there are one-to-many relationship types between each of these and Files, so all that's necessary is to have LevelID and FormatID foreign keys in Files.

With books, however, I would have thought that there would be a many-to-many relationship type between Files and Books, i.e. each file can be associated with one or more books, and each book can be relevant to one or more files.  If so the relationship type should be modelled by a table which resolves it into two one-to-many relationship types, e.g.

Citations
….FileID
….BookID

The key is a composite one of the two foreign key columns.  If you wish to record non-key attributes of the relationship type such as the page in the book number in the book relevant to the file this can be represented by a further column.  If there can be more than one relevant page number per file/book the PageNumber column would also be a component in the primary key.  This part of the model would thus be like this:

Files---<Citations>---Books

If you are recording the authorship of books then be sure to have an Authors table, and include an AuthorID foreign key column in Books, not the name.  I once found three versions of my own name in a badly designed database as author of publications in my own field of work (environmental planning).  If publications can have multiple authors, which is common in my field of work, the relationship type between Books and Authors is many-to-many, so should be modelled by table which resolves it into two one-to-many relationship types.

Finally, how do we model favourites?  This is a many-to-many relationship type between Users and Files, so again is modelled by table which resolves it into two one-to-many relationship types:

UserFavourites
….UserID
….FileID

The table's primary key is a composite one of both foreign key columns.  Other columns can be added to represent non-key attributes of the relationship type, such as your 'ranking' of a user's favourites from 1 to 5.  If you do this then the UserID and Rank column should be included in a single unique index to prevent invalid duplication of the Rank value per user.  Dta entry would be via a UserFavourites subform in a Users parent form.

With a logically constructed model like this the interface would be easy to build, and you would have little difficulty in querying the database in any way you might wish.
