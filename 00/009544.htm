jay freedman say:
Here are the steps for a manual procedure.

  1. Press the ¶ button and verify that each "paragraph" ends with a paragraph mark and not a manual line break (http://www.word.mvps.org/FAQs/Formatting/NonPrintChars.htm).
  2. Select all the paragraphs, then click Insert > Table > Convert Text To Table.
  3. In the bottom section of the dialog, choose Paragraphs, then click OK. Each paragraph will be placed in a separate cell of a 1-column table.
  4. On the Table Tools > Layout tab of the ribbon, click the Insert Left button to create a new first column.
  5. In the first cell of the first column, press Ctrl+F9 to insert field marks, and type   seq a   between them.
  6. Press F9 to update the field, which will show the number 1. Copy the field to the clipboard.
  7. Select the whole first column (move the cursor above the top of the first column so it becomes a downward arrow, and click).
  8. Paste from the clipboard, which will place the field (still showing 1) in each cell of the column. While the column is still selected, press F9 to update all the fields to consecutive numbers.
  9. On the Table Tools > Layout tab of the ribbon, click the Sort button. In the "Sort by" section of the Sort dialog, select the Descending option button and click OK.
  10. Again select the first column. On the Table Tools > Layout tab of the ribbon, click the Delete button and choose Delete Columns. Then at the right end of the same ribbon, click Convert To Text.


If you need to do this very often, you can use this macro (see http://www.gmayor.com/installing_macro.htm) which does the same thing:


Sub ReverseOrderOfParagraphs()
    Dim tmpTable As Table
    Dim rg As Range
   
    Set tmpTable = ActiveDocument.Range.ConvertToTable(Separator:=wdSeparateByParagraphs)
    With tmpTable
        .Columns.Add BeforeColumn:=.Columns(1)
       
        Set rg = .Cell(1, 1).Range
        rg.End = rg.End - 1
        rg.Fields.Add Range:=rg, Type:=wdFieldSequence, Text:="a"
        rg.Cells(1).Range.Copy
       
        .Columns(1).Select
        Selection.Paste
        Selection.Fields.Update
       
        .Sort ExcludeHeader:=False, FieldNumber:=1, SortOrder:=wdSortOrderDescending
       
        .Columns(1).Delete
       
        .ConvertToText
    End With
End Sub

Why Excel date time format "m/d/yyyy hh:mm:ss" - Microsoft Community

giulio8 say:

Why Excel date time format "m/d/yyyy hh:mm:ss" is NOT managed for different Regional Settings (US English, Italian, etc) ?

When formatting an Excel Cell as 

"m/d/yyyy hh:mm"

the Excel file visualization respects the current OS regional settings, so (for example) if one changes them from US English to Italian, the Excel column immediately changes and shows the day before the month.

Why didn't Microsoft manage this functionality also when seconds are shown like below?

"m/d/yyyy hh:mm:ss"

Thanks in advance.

Edit.

Which is the Excel equivalent of the .NET "G" format?

"G"

General date/time pattern (long time).

More information: The General Date Long Time ("G") Format Specifier.

6/15/2009 1:45:30 PM -> 6/15/2009 1:45:30 PM (en-US)

6/15/2009 1:45:30 PM -> 15/06/2009 13:45:30 (es-ES)

6/15/2009 1:45:30 PM -> 2009/6/15 13:45:30 (zh-CN)

Once I've written "6/15/2009 1:45:30 PM" in En-US I don't want to reinvent the wheel, writing a custom code that has to check application.International and then has to change the format of all the cells taking into account what is the corresponding zh-CN version... Isn't that agreeable? Why is Microsoft assuming that we're supposed to lose the visualization of the seconds (selecting the format "m/d/yyyy hh:mm") in order to get a localized solution when we're opening an Excel file? (while such an issue doesn't happen for a normal .net datagrid)

Consider also that a package such as DevExpress automatically exports a Grid to XLSX: the users are losing something if they're spread around the world.

Edit 2

It really looks like an unintentional bug of Excel: is this forum the appropriate place to report a product bug to Microsoft?
As a proof, for instance, just read the Infragistics post: the concept is that, after changing the regional settings (of course such action takes effect even without closing and reopening the excel file and it is not limited to MDY vs DMY changes at all), Microsoft Excel updates a date format such as "m/d/yyyy" also when it is part of an extended "m/d/yyyy hh:mm" and there is no reason and no microsoft documentation saying that  "m/d/yyyy hh:mm:ss" is not going to be updated with regional settings as well.

Another counterproof that it is a bug: given a cell with a format "m/d/yyyy hh:mm:ss", the switch from US English to CH Italian affects the visualization of the date, because the "/" are replaced with "." => so it is clear that the intention was to manage the Internationalization also in that case.
