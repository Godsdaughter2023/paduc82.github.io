There is a problem with Microsoft Publisher 2016 which does not occur in applications such as Word & Excel.

Background: 

Our domain extensively uses a Windows Server 2012R2 unit to provide access to user folders via DFS. Folders use enumeration to prevent knowledge of other folders by non authorised users.

Our client units currently run Windows 7 Pro.

Roaming profiles with redirected folders are used for user accounts.

For example "My Documents" is mapped to \\domain.com\Accounts\person\documents - this is the principle format for all paths for saving/opening files.

Scenario:

Client logs in with their roaming profile

Launches M$ Publisher 2016

Creates a document.

Attempts to save the document

Problem --- An error occurs in Publisher, the Microsoft Office Alerts event log shows the following...

"Microsoft Publisher

   1100285  
   16.0.4300.1000  

   0x80070005"

Repeat exercise but using M$ Word 2016 --- no problem occurs when saving. Clearly a fault within Publisher.

Diagnosis:

Examine the permissions to each folder of the supplied DFS URL that is giving as the path for saving.

The accounts folder is particularly locked down in order to prevent a user from browsing to who else exists on the system (think GDPR - non authorised users do not have the right to even know who else is there)

Workaround:

Adjust the permissions of the accounts share which is used in the DFS URL to allow all 'accounts' to have read access for that folder ONLY.

Testing of Publisher 2016 then shows that the application no longer complains of being unable to save because of a 'read only' target. This keeps Publisher happy but also prevents users from seeing all the actual folders in the 'accounts' folder.

There is obviously a fault within the code of Publisher in that it requires read access to each part of a URL that is to be used for saving.

The FINAL destination folder of the URL; which is where Publisher was actually to save, was the only part that needed to be visible to publisher.

james yan msft say:

Hi Akiko,

Generally, this error "The webpage cannot be found" indicates that the site has been deleted.

To find out all the sites in your tenant, you can use PowerShell commands:

1. Download and install the SharePoint Online Management Shell from this article.

2. Launch SharePoint Online Management Shell (search and right click the "SharePoint Online Management Shell" and choose "Run as Administrator"), connect to SharePoint Online via the following commands using your Office 365 global admin account and list all sites in your tenant:

Connect-SPOService -Url https://contoso-admin.sharepoint.com

(replace contoso with your tenant domain)

get-sposite | Select-Object url

get-spodeletedsite | Select-Object url

For the deleted sites, you can also run this command:

Get-SPODeletedSite | fl

If the DaysRemaining is NOT 0, you can restore the deleted site by running 

Restore-SPODeletedSite -Identity siteURL

About whether you can access those sites with administrator's privilege, that depends: If it is a private group and you're not added into the group, you can't access that group site even if you're the gloal admin.
If it is a public group, by default, everypone except external users has Edit permission level to that group site.

daniel g. grau say:
That's a VBA/Visual Basic error message, indicating that part of the code is missing. Did you try recording a macro and then accessed the code? If not, you may have a macro virus.

Anyway, since you just installed Office, there's no need for you to preserve any customizations to the Normal template. Thus, the easiest thing to do is to trash your current Normal template (which has the erroneous macro code). To do that:

  • Quit Word
  • Go to the Finder
  • Select Go > Go to Folder and type ~/Library
  • Navigate to the following folder: Application Support/Microsoft/Office/User Templates
  • In there you should find a file called Normal.dotm. Trash it.
  • Now restart Word. It will recreate the missing template and possibly ask you to save it upon quitting.

For good measure, and before opening any Word document, ensure Word is protected against macro execution:

  • Open Word by double-clicking on its icon (not on a document)
  • In the main menu bar, select Word > Preferences > Security (in the Personal Settings section)
  • Under Macro Security, make sure there's a checkmark next to "Warn before..." This will let you decide if you want to cancel any macro execution, so you can still open contaminated documents.
