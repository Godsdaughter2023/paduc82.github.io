What you're encountering is the new "simplified" account settings UI. There are now 4 ways to go about changing email account settings.

Option #1

From within Outlook
- Press CTRL+ALT+S (this brings up the Send/Receive Settings window)
- Select the group you want to edit and click on the <Edit> button
- Select the email account you want to modify --> click on <Account Properties>

Option #2

From within Outlook
- go to File --> Options --> Advanced --> click on the Send/Receive button
- click on the applicable <Account Group> (most people will only have one)
- click <Edit>
- select the <Email Account> --> click on <Account Properties

Option #3

With Outlook closed
- open the Control Panel <Mail> app
- select <Email Accounts>
- select the desired <email account> --> select <Change>

*** the Control Panel <Mail> app is not an option if the MS Office "Store" version is installed. If it is the store version, then the alternative in this instance is to click on Windows <Start> and <Run> the following command

Outlook.exe /manageprofiles

Option #4 - To change Server Settings

From within Outlook

- go to File --> Account Settings
- choose the desired email account and click Repair.
- click on Advanced options
- check <Let me repair my account manually>
- click on <Repair>

silentbob say:

Linked Tables (SQL Server 2005) and Data Types - Decimal places being truncated

It has been many years & several versions since I last used MS Access on a regular basis. However, I have now come across a scenario where itshould provide the ideal solution.

I need to use Access to allow our Finance department access to a small number of tables on a SQL Server 2005 database in order that they can change some parameters and allocations used in producing a particular report for them. Quite simple, I thought, link the tables and knock up a copule of forms for them to navigate & modify the data. However, I have come across a bit of a problem...

For fields that are holding monetary or percentage parameters, I have used a datatype of Decimal with the default precision of 18 and a scale of 4 (originally I had used the Money data type, but soon realised that Access didn't like this). This fits quite well and accomodates all of the parameters, including one that is currently set as 0.005 (=0.5%).

After linking the table, when I go into the design view (from within Access) it shows correctly as Decimal,18,4. However, if I open the table (or access via a form) the value which is 0.005 in SQL is shown in Access as 0.01. If I try to edit this I then get a message telling me that:
"this record has been changed by another user since you started editing it. If you save the record, you will overwrite the changes the other user made."
and the "Save Record" button is greyed out.

If (from SQL) I reset the value to something that only uses 2 decimal places or less, then it displays correctly within Access and I can edit it with no problems. I can even enter a value of 0.005 which will be accepted, displayed & is confirmed when I check SQL. If, after entering a value with > 2 decimal places, I refresh the view (by closing the table and reopening it), the value is again rounded to 2 decimal places, and again I get the same message and cannot edit it.

It seems that Access is correctly identifying the datatype of the field and the properties (precision & scale) applied. It also allows me to write values that conform to this. However, when a field already contains a value of more than 2 decimal places, Access seems to think it necessary to change this to 2DP. This change, although not saved, prevents me changing it any further.

I have experimented by creating a table within Access with a Decimal,18,4 field, and adding/editing different values. This works as expected, no matter what values I type.

Why is Access exhibiting this strange behaviour with the linked table?
Is there anything that I can do to make it work properly?
Or is it a limitation that cannot be prevented?

I have thuoght about perhaps typing the actual percent. I.e. for 0.5% I enter 0.5. But this seems a little unnecessary, adds an extra element to any calculation using it and cannot be formatted to display as a percentage.

Any help would be greatly appreciated.

Thanks,
Simon

However, with this problem occurring, it lead me to a much better solution - Access Projects (ADP)...
This uses OLE, instead of ODBC, which is a big plus.
It stores the login credentials, instead of prompting for them each time the database is opened.
It works directly from the SQL data, using the SQL data types. Therefore no problems with unexpected truncating.
Any changes to the tables structure feed straight through. No need to re-link.
Smaller file size, as the table data is not stored in the Access file.

So, no answer needed for this now. My solution was to use ADP instead of linked tables. :)
