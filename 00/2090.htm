ken sheridan say: If you are taking the Filter property from one form, whose recordset includes a column TransDate and applying it to another form whose recordset includes the column InvoiceDate, then rather than changing the RecordSource property of the second form, why not simply replace the substring 'TransDate' in the string expression with the substring 'InvoiceDate'.  Generically code to do this would be along these lines:

Dim strFilter As String

strFilter = Forms("Form1").Filter
strFilter = Replace(strFilter,"TransDate","DateValue(InvoiceDate)")

DoCmd.OpenForm "Form2", WhereCondition:=strFilter

By also calling the DateValue function Form2 will be filtered to those rows where the value of the InvoiceDate column is whatever is the date the first form was filtered to, regardless of the time of day element of the InvoiceDate value.

It's important to remember that dates are stored not in any format, but as a 64 bit floating point number with its origin at 30 December 1899 00:00:00.  Provided that a date literal can be correctly interpreted the format is irrelevant.  You can see this by entering the following in the debug window for instance:

? #27-Nov-14# = #2014-11-27#
True

As you see it evaluates to True.  The first date is in medium date format, the second in the ISO standard format for date notation.  Both represent the same value, which under the skin is 41970, this being the number of days from 30 December 1899.  Similarly you can see how the DateValue function strips off the time of day, or more accurately speaking converts it to zero, by entering the following:

? #27-Nov-14# = DateValue(#2014-11-27 14:30:00#)
True
 
Although the values of the date literals are now different the expression again evaluates to True, because the DateValue function has converted the time of 2:30 PM to zero when it returns a value.  Any 'date' value is not really the day per se, but the point of time at midnight at the start of the day, i.e. time zero on the day in question.

Actually they work quite well together and not to be argumentative with Charles or anyone else I still wonder where this "general rule" came from.

The problem is and as it has been the "protected mode."  Protecting (or restricting a document) from being edited has undersirable consequences and one of those is that it prevents content controls title tags from appearing in documents.  This is true regardless if you use the more robust "Restrict Editing" features or the the less robust content control grouping.  Both allow you to edit the content of a control and both inhibit the display of the control title tab.

 

As a work around and if you can use VBA procedures in your document then you can unprotect the document when you enter a CC and reprotect it when you exit a CC:

Option Explicit
Private Sub Document_ContentControlOnEnter(ByVal ContentControl As ContentControl)
On Error Resume Next
ActiveDocument.Unprotect
End Sub
Private Sub Document_ContentControlOnExit(ByVal ContentControl As ContentControl, Cancel As Boolean)
ActiveDocument.Protect wdAllowOnlyFormFields, True
End Sub

 

or you could modify the placeholder text to provide the CC title.
