ken sheridan say: While you can select multiple items in a multi-select list box, the control cannot be bound to a column in the table.  A multi-select list box does not have a value to which the selection is assigned in the same way as a conventional list box; it has an ItemsSelected collection, which is a collection of variants identifying the selected rows.  It would be necessary to write code in the form's module which (a) when moving to an existing record reads the data from the relevant table and selects the items in the control, and (b) inserts rows into, or deletes rows from the table when selecting/deselecting items in the list.

What you have here is a many-to-many relationship type between lessons and standards.  This type of relationship type is modelled, not by having a foreign key column in the lessons table, but by means of a separate table which resolves the many-to-many relationship type into two one-to-many relationship types.  So, in addition to the Lessons table you'd have the following tables:

Standards
....StandardID (PK)
....StandardDescription

and to model the many-to-many relationship type:

LessonStandards
....LessonID (FK)
....StandardID (FK)

the primary key of this table is a composite one made up of both columns.

In your lessons form one possible approach would be to have a list box and text box in the way that Dirk describes, but the list box would be unbound and would not be a multi-select control.  When the user selects an item the text box would show the full description.  Selecting an item would not in itself commit any data to the LessonStandards table.  For that you could use the DblClick event procedure of the list box, but I would suggest two buttons beside the text box, one with a plus sign to assign the standard to the current lesson, the other with a minus sign to delete it from the current lesson if it is already assigned.

The RowSource property of the lstStandards list box would be:

SELECT StandardID, StandardDescription
FROM Standards
ORDER BY StandardDescription;

Set its ColumnCount property to 2 and its ColumnWidths property to 0 to hide the first column.  The ControlSource property of the text box would be:

=[lstStandards].Column(1)

The Column property is zero-based, so this will show the second column, the StandardDescription.

To show the selected standards in the form include a continuous forms view subform, linked to the parent form on LessonID and based on the following query:

SELECT LessonStandards.*, StandardDescription
FROM LessonStandards INNER JOIN Standards
ON LessonStandards.StandardID = Standards.StandardID
ORDER BY StandardDescription;

In the subform include just one control, a text box bound to the StandardDescription column and set its Locked property to True (Yes) to make it read-only.

The code behind the 'plus' button would be:

Const MESSAGETEXT = "No standard selected."
Dim strSQL As String

strSQL = "INSERT INTO LessonStandards(LessonID, StandardID) " & _
    "VALUES(" & Me.LessonID & "," & Me.lstStandards & ")"

If Not IsNull(Me.LessonID) Then
    If Not IsNull(Me.lstStandards) Then
        Me.Dirty = False
        CurrentDb.Execute strSQL, dbFailOnError
        Me.sfcLessonStandards.Requery
     Else
         MsgBox MESSAGETEXT, vbExclamation, "Invalid Operation"
    End If
End If

where sfcLessonStandards is the name of the subform control which houses the lesson standards subform.  The code behind the 'minus' button would be:

Const MESSAGETEXT = "No standard selected."
Dim strSQL As String

strSQL = "SELECT * FROM LessonStandards " & _
    "WHERE LessonID = " & Me.LessonID & _
    " StandardID = " & Me.lstStandards

If Not IsNull(Me.LessonID) Then
    If Not IsNull(Me.lstStandards) Then
        CurrentDb.Execute strSQL, dbFailOnError
        Me.sfcLessonStandards.Requery
     Else
         MsgBox MESSAGETEXT, vbExclamation, "Invalid Operation"
    End If
End If

When the user adds or deletes a standard to/from a lesson via the 'plus' or 'minus' buttons the subform will be requeried to add or remove the standard.

If, on the other hand, you continue with the use of a multi-select list box then I know of no way you can show the full description of an individual item in a separate control, but you could put code in a separate 'confirm' button's Click event procedure to update the LessonStandards  table and requery the subform:

Const MESSAGETEXT1 = "No standards selected."
Const MESSAGETEXT2 = "No current lesson record."
Dim strSQL As String
Dim strStandardIDList As String
Dim varItem As Variant
Dim ctrl As Control

set ctrl = Me.lstStandards

If Not IsNull(Me.LessonID) Then
    ' ensure current record is saved
    Me.Dirty = False
    If Not IsNull(Me.lstStandards) Then
        ' first delete all rows for current from LessonStandards
        strSQL = "DELETE * FROM LessonStandards " & _
             "WHERE LessonID = " & Me.LessonID
        CurrentDb.Execute strSQL, dbFailOnError

         ' insert new rows into LessonStandards
         For each varItem in ctrl.ItemsSelected
             strStandardIDList = strStandardIDList & "," & ctrl.ItemData(varItem)
         Next varItem

         ' remove leading comma
        strStandardIDList  = Mid(strStandardIDList ,2)

        ' insert new rows into LessonStandards
        strSQL = "INSERT INTO LessonStandards(LessonID, StandardID) " & _
            "SELECT " & Me.LessonID & ", StandardID FROM Standards " & _
            "WHERE StandardID IN (" & strStandardIDList  & ")"
        CurrentDb.Execute strSQL, dbFailOnError
        Me.sfcLessonStandards.Requery
     Else
         MsgBox MESSAGETEXT1, vbExclamation, "Invalid Operation"
    End If
Else
    MsgBox MESSAGETEXT2, vbExclamation, "Invalid Operation"
End If

Not that the above first deletes all the rows in LessonStandards for the current lesson and then inserts the standards currently selected in the list box, so it is cannot be used to select one or two standards and add them to those currently assigned to the lesson; it's all or nothing!  The first method, selecting and adding a standard individually, can be used to add further standards to an existing set for the lesson, however.  Note that it is very important that the two columns in LessonStandards are designated as the composite primary key to avoid the same standard being added more than once.  

The easiest solution of all of course would be to simply enter rows into a subform in which you have a bound combo box to select a standard and an unbound text box to show its full description.  No code whatsoever is then required.

I note that you appear to be using a multi-valued field for the LessonStrand column.  Unless you are interfacing with SharePoint and using its specific features which demand the use of this type of field I would strongly advise against it.  Instead model the many-to-many relationship type between lessons and strands via a third table in the way I described earlier for standards.  In this case a simple continuous subform would be adequate for data entry.  You'll find an example of this sort of basic many-to-many relationship type and how to represent it as a form/subform as ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

In a different context you might also like to take a look at the StudentLog.zip file in the same SkyDrive folder.  Although this little file is only intended as a demo of the use of correlated subforms rather than an real application, I know that at least one school in the USA has used it as a basis for their class registrations and attendances system.

Excel can't cope with times greater than 9999 hours.  You will probably notice that the numbers greater than 10000 hours are left-aligned whereas the 9912:00 is right-aligned as it is recognised as a valid time.

 

If your data were in cells A1:A9 you could use the following array formula to total the hours only:

 

=SUM(IF(ISNUMBER(A1:A9),(A1:A9)/(1/24),VALUE(LEFT(A1:A9,FIND(":",A1:A9,1)-1))))

 

Press Ctrl+Shift+Enter to enter it as an array formula.  Your data does not show any minutes, if you need to total minutes as well then an extra bit of calculation would be needed.

scottgem say:

The order of records in a table is totally arbitrary. Remember that an electronic file is just a bunch of ones and zeros stored somewhere on a disk. When Access retrieves those ones and zeros it converts them to a readable format. So the order of the records may seem odd depending on how the data is stored on disk. Generally, however, when Access displays the records, it applies a sort using the Order By property of the table, for, report or query. So, as Daniel noted, if records are appearing in a different order, you need to look at the Order By or default sort.
