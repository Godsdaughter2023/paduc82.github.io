a. user say:

There are two approaches to solving this problem, but first, a bit of background.

When Word 2010 tries to open an Access data source, by default, it tries to open using a method called "OLE DB". Unfortunately, that method does not work with the type of encryption used by default in Access 2010. 

The other approach you can try is to revert to the old DDE connection method. This is "deprecated" and suffers from some problems, but is in fact the only easy way you can connect to some things, such as Access parameter queries. 
  
 a. in Word, go to File->Options->Advanced->General (near the bottom of the Advanced options)
 b. check "Confirm file format conversion on Open"
 c. re-open your Word Mail Merge Main Document
 d. go through the process of connecting to the Access database again (e.g. Mailings tab->Select Recipients->Use Existing List
 e. Once you have selected the .mdb or .accdb, you should see an additional dialog box titled "Confirm Data Source". For an ACCDB, only one option will be displayed ("OLE DB Database Files"). In that case, check "Show All". A number of other options should appear.
 f. locate and select the option titled "MS Access Databases via DDE (*.mdb;*.mde)", then click OK.
 g. If the database is already open in Access, you should now see a dialog box with a list of tables/queries. Otherwise, Word tries to start Access if it is not already running, then asks Access to open the database. At this point, Access should prompt you for the database password. However, it is possible that the dialog box will appear behind Word, so if you do not see it quickly, alt-tab to the Access application and respond to the dialog box. Then you should see the list of tables/queries and you can proceed.

An advantage of using this method is that when you save/close and re-open your Word document, at most you will have to re-enter the database password again.

If you want to open using OLE DB you will need to do the following:

1. In Access
 a. close your database, if it is open
 b. re-open the database exclusively
 c. remove the password
 d. go to File->Options->Client Settings->Advanced->Encryption method
 e. check "Use legacy encryption"
 f. recreate the password
 g. close your database.

As far as I know, you can then change the default Access encryption method back to what it was before, if you want to use the more powerful encryption for databases that you do not need to use in Mail Merge. But check that!

2. Then, when you try to connect to the Access database from Word,
 a. You will probably see a dialog box that says "Error has occurred: Not a valid password"
 b. click OK.
 c. you should then see a dialog box titled "Data Link Properties"
 d. In the Connection tab, check "Allow saving password"
 e. Click the "All" tab
 f. locate and select the property called "Jet OLEDB:Database Password"
 g. Click Edit->Value..., and type the password in the box that appears
 h. click OK.

You should now be able to select the Query/table that you want. However, when you have done that, you will probably see the Data Link dialog again, and will have to repeat 2(d) to 2(h) before you can continue. Further, when you save/close and re-open your Word document, you will need to go through this entire process again. But at least you will have established that the OLE DB approach can be used.

One way to improve this situation is to create a ".odc" file (Office Database Connection) and use that to connect to the database. The problem with doing that is that it only makes things easier by storing the database password in the .odc file, and since it is stored unencrypted, that undermines security. It would also be possible to open the data source using VBA, but that will also result in storing the password in the Word document (although perhaps a little less visibly). So think about that, and if you need to know how to do either of those things, let us know.

hansv mvp say:

Try this macro:

 

Sub SplitValues()
    Const c = 5 ' column with multiple values
    Dim r As Long
    Dim m As Long
    Dim arrParts
    Dim i As Long
    Application.ScreenUpdating = False
    m = Cells(Rows.Count, c).End(xlUp).Row
    For r = m To 2 Step -1
        arrParts = Split(Cells(r, c), ";")
        If UBound(arrParts) > 0 Then
            For i = UBound(arrParts) To 1 Step -1
                If arrParts(i) <> "" Then
                    With Cells(r, c).EntireRow
                        .Copy
                        .Insert
                    End With
                    Cells(r + 1, c) = arrParts(i)
                End If
            Next i
            Cells(r, c) = arrParts(0)
        End If
    Next r
    Application.CutCopyMode = False
    Application.ScreenUpdating = True
End Sub

you are an expert for Word, Word has also tables and the things are different. But your example code for Excel is nonsense.

ActiveCell refers always to the active cell of the active window, that has nothing to do with Select or Activate.

https://docs.microsoft.com/en-us/office/vba/api/excel.application.activecell

The With statement create a reference to cell C1 in the second sheet, so far okay.

In the next line .Current.Region.Rows.Count is wrong, a Range object do not have a Current property. We can assume a typo and use .CurrentRegion.Rows.Count instead, but what does that mean?

https://docs.microsoft.com/en-us/office/vba/api/excel.range.currentregion

CurrentRegion returns a Range object that include all cells around. So on fact we can get many many rows with .CurrentRegion.Rows.Count and if we loop though all rows and check each cell in column C the code becomes very slow. The OP already use Range.Find and that is much faster.

At next

  If InStr(.Offset(i, 0).Formula, "CID") > 0 Then

looks for the string "CID" inside the formula, but not for the contents of the variable CID as the OP want.

BTW, InStr is case sensitive unless you specify the last argument vbTextCompare.

And if you use a loop from 0 to Rows.Count you hit the cell below, Rows.Count-1 would be correct. You can say "who cares, the cell is empty" and I say yes, usually you are right. But there can be scenarios where the cell is not empty.
