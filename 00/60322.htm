wunders say:

I am assuming that your XP Pro machines have "Simple File Sharing" disabled.

When you attempt to access another machine for File Sharing, Authentication proceeds close to the following sequence:

  1. If the same username and password exists on both computers, then the connection is silently authenticated
  2. If the same username exists on both machines but the passwords don't match, then the connection is disallowed.
  3. If the same usernames don't exist on both machines, then it attempts to authenticate as "Guest" user with a null password.
  4. If it can't authenticate as "Guest" (either because Guest user was not active or the Guest user had a password), then the client machine will put up a window asking for a Username and Password for authentication.

... so if you're falling all the way down to #4, then you need to fix it so that one of the other conditions are met. 

If "Simple File Sharing" is enabled, then all users are forced to authenticate as "Guest" user.  To determine if SFS is enabled, double-click any folder, then Tools -> Folder Options -> View (tab) -> Use Simple File Sharing (last in list)

wunders say:

Your apparent symptom-connection to the new router is baffling since routers do not do very much on the LAN-LAN side in terms of firewalling.  However, I did come up with one possibility that can be router-connected.  The DHCP service on the router usually sets IP address, Gateway, and DNS Server.  However, it also has the potential to set the Node-Type on a Windows machine.  If the Node-Type were to be set to "Peer-to-Peer" (P-P), Then that machine would stop listening to broadcasts and effectively remove itself from the Windows Networking (NetBT) network.  This can be confirmed by bringing up a command prompt window (Start -> Run -> "cmd") and entering the command:

   ipconfig /all

and note the "Node Type" line.  Normally, without a WINS server on the network the type should be "Broadcast" but anything except P-P (P-node) will work.  If this is your problem, the solution is here:

"You cannot view other workgroup computers on the network on a Windows XP-based computer"
   < http://support.microsoft.com/kb/903267 >

In terms of troubleshooting, I find that error messages are much more revealing in the command-prompt environment.  Try connecting to another machine "manually" by bringing up the command-prompt window (Start -> Run -> "cmd") and enter the command:

   net  use  *  \\192.168.1.100\sharename

or, to authenticate as a particular user, try:

   net  use  *  \\192.168.1.100\sharename  /user:joe

Where you replace 192.168.1.100 with the target machine's actual IP address and replace "sharename" with the name of the network share  folder on that machine.  Using the IP address bypasses "Browse Master" problems.  If it works with the IP address, then disconnect and try reconnecting using the computer name instead of the IP address as in:

   net  use  *  \\computer1\sharename

Errors that you get using the command-prompt approach are much more helpful than the GUI.

wunders say:

You have a lot of good information in your post.  When you tried to connect to your "ASSARASZ27SP" machine (which you had previous problems connecting) the "net use" command successfully connected that share to the "W:" drive.  ... and when you connected to the "SOM" machine, it too connected successfully to the V: drive.

The way you read the output from the "nbtstat" command is briefly as follows...  "Local Area Connection" refers to the hard-wired ethernet on the machine you typed the nbtstat connection on.  The "0.0.0.0" that follows indicates that you have nothing connected to that hardwire connection.  "Wireless Network Connection 2" is a wireless port on the machine that you typed the nbtstat command on and the "192.168.1.9" is the IP address on that [local] wireless port.  The results that follow come from the target machine and you either get a [successful] output like you received or you get an error message.  All this indicates that there are no connectivity problems with your machines and file sharing seems to be working properly.

My guess as to where your problem originates is that your computers seem to be playing "musical chairs" with IP addresses.  I downloaded the manual for your router from:

    < ftp://support.dlink.co.in/broadband/DSL-2750U/manual/DSL-2750U_C1_Manual_IN_1.00.pdf >

Not a very good manual as they go, but if you look at the settings on page 47 and assume that they are default, you will notice that the "Lease Time" is set to 24 hours.  That means that if one of your servers goes idle and "sleeps" for more than half that time (12+ hours), it is possible that the IP address will be returned to the pool and when it wakes it will could get a new/different IP address.  You would like your "server" machines to maintain a consistent IP address.  Here are some things you can do to encourage that to happen:

  1. Connect to your router by bringing up a web browser and entering http://192.168.1.1 in the Address window.  In your Router Settings (in manual above, page 47), Change the "Leased time" setting to something in the 2 week range (350 hours or so)
  2. In your Router Settings, assign a "Static IP" lease to your server machines.  On page 47, click the "Add Entries" button.  Then you will need to associate an Ethernet (MAC) address with an IP address.  From your "nbtstat" output, you see that "SOM" Machine's Ethernet address is 44-87-FC-74-FF-4D.  Associate that E-net address with a fixed local address such as 192.168.1.200.  Make the next machine ".201" and so forth.  Your router can apparently accept up to 32 of these fixed associations.  Don't forget to click the "Apply/Save" button after making the changes.
  3. Keep your machines from going to Standby.  On each machine, go to the Control Panel -> Power Options and configure your file server machines never to go into sleep/standby.
  4. Keep your network cards from powering down to save power.  Bring up the device manager (Start -> Run -> "devmgmt"), double-click on your wireless network adapter, look for a "Power Management" tab at the top, and uncheck the box "Allow the computer to turn off this device to save power". (On Windows 7,    Start -> Right click "Computer" -> Manage -> Device Manager)

After doing 1 and 2 above, you will need to restart every affected machine on your network to assure they receive the updated settings.

I hope this helps,
Good Luck,
    JW
