robgagnon say:

I found and fixed the problem... it took days to find the problem, and hours to figure out a solution, so now I wonder how do you get Microsoft to apply the fix I found so others don't have an issue??

The problem was not in the Silicon Image driver file as suspected, but I was close.

The nvraid.inf file is missing a single underscore that prevented it from matching up with the signature in the NVIDIA raid chipset on my motherboard, so Windows 10 was only applying "Generic SCSI Array" drivers to it, and those obviously prevented the system from loading.

The funny thing is that the boot drive had no problem loading its stuff up.  It was during the switch to second boot that it could not read the main partition windows lives on, so this makes me think that the nvraid.inf file in the boot.wim file differs somehow from the one in install.wim.

The short fix if Microsoft can do it, is to add the following line in two places within nvraid.inf, and then they would have to re-sign the driver as the hash for nvraid.inf will change and make the .cat file invalid.

  • "%NVRAID_DESC%=nvraid,SCSI\__NVIDIA_______Raid_Disk_"

nvraid.inf already has "%NVRAID_DESC%=nvraid,SCSI\__NVIDIA_______Raid_Disk", but the chipset on my motherboard shows an extra "_" on the end to the plug-n-play system.

For me to fix it was a pain, and involved re-installing Windows on a new drive temporarily, and also involves a loss of all the data that was originally on the RAID disk.  I had a backup from before the update, so I was OK, other than the time to restore things later.

  • I started by downloading the latest drivers I could find from NVIDIA's driver download site (Legacy, nForce6, Windows7-64bit, v15.58).
  • I updated the .inf file as shown above to include another line with the extra underscore in it, and I set the files aside on a USB stick for now.
  • I added another hard disk to the motherboard connected to a standard SATA (non-raid) connector.
  • Now I installed Windows 10 1607 onto the newly connected hard disk from a bootable USB that was created with the Media Creation Tool.
  • After Windows booted and settled down from its updates, I performed a reboot using the "Advanced startup" feature, so I could disable Driver Signing Verification on the next boot.
    • Windows Key, type "Advanced", click on "Change Advanced startup options"
    • Click on "Restart now" under Advanced startup
    • Select 7 from the menu that appears during reboot
  • Upon reboot, I opened up the device manager, and found the "Generic SCSI Array" device
    • I can't recall, but it might not have been visible until I set the "View" to "Devices by Connection"
  • On that device, I did the Update driver option, with the "Have Disk" version of how to do it, and pointed the source over to the USB memory stick containing the modified v15.58 driver from up above.
  • After proceeding through the warnings about the driver not being signed, windows suddenly was able to see my hard drive.--- This gets you about 3/4 of the way through the entire solution

Now that windows can see the RAID, I still had to get windows ONTO it.  Don't forget, I am running from a temporary hard disk. 

To do this, I used the Acronis utilities that came with the hard drive (and downloadable from Western Digitial) and cloned the temporary hard disk that had my working Windows on it, over to the RAID drive.

Once cloned, I removed the single hard disk, and rebooted.

*POOF* Windows 10 Anniversary Edition Build 1607 on my PC.  All I needed to do now was restore the documents and software from my backup.
