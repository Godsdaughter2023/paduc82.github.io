a. user say:

Just discussing LINK type fields, unfortunately, as things stand there cannot be a complete "fire and forget" work around, where the user somehow specifies a relative link and Word just "does the right thing." That is because even if you could fix up the XML code in the .docx to use a relative link, for a LINK field Word always saves an absolute path to the linked object. So however "relative" the paths are when the user opens the document, they definitely won't be relative by the time the user saves and closes it.

There are then two questions, really:

 a. could Word be changed to allow relative LINKs?

 b. What workaround(s) could I/my organisation/my users live with?

For (a), the accepted place to go these days is word.uservoice.com. Trying to achieve anything in Answers is probably a waste of effort for a problem like this which is actually well known. But nothing is likely to happen until the number of people supporting a uservoice request becomes quite large. I can't see any relevant suggestion that has already gathered a lot of votes so perhaps  https://word.uservoice.com/forums/304924-word-for-windows-desktop-application/suggestions/32684152-relative-paths-for-includepicture-includetext-li is the one to go for. As you can probably see, some of the more "techie-looking" requests just don't get much interest.

For (b), it seems to me that you can work around some of the problems with LINK fields using more or less the same "{ FILENAME \p }/../" mechanism that I came up with many years ago, but

 1. it's still only going to work the first time someone opens the document that contains the links. After they've updated/saved, it's back to absolute links

 2. you're still going to have to work quite hard to set it up, because at some point you have to replace all the paths in LINK fields with fields+text, and that's the thing I think you ended up doing by hand. You might be able to use find/replace, and I would guess it would be fairly easy to adapt macropod's code to help you.

As if that was not enough, it's only going to work

 3. for links represented by LINK fields

 4. on file systems that support whatever path syntax you use in the pathname of the linked object

 5. for clients that actually update LINK fields, or allow the users to do so, and that is probably just Windows and Mac desktop Word. Even Mac Word 2011 may have problems with a path syntax based on "/" separators rather than the AFP ":" separator - I'd have to check.

For (3), there can be linked objects that are represented by a graphic and have no LINK field.

For (4), although the { FILENAME \p } mechanism works more widely than I would expect (considering the field result contains the document name as well as its path), there are some file systems where it won't work. In particular, if you store the linked files on a OneDrive and replicate them to a local file system, the links may work on one file system but not the other. Again, I would have to check the details.

If you want to do that, what has to happen is that you have to be able to get the necessary nested fields into your LINK fields and save the document without Word resolving the fields. Just to try it out for yourself, you may be able to do that for a single LINK field in the body by replacing the path+file name of the linked file with the necessary fields and text, saving the file immediately without updating anything, and closing it. IMO the simplest test is then to modify the linked file (so you can be sure that the link has updated correctly), rename the enclosing folder, re-open the .docx, and update the fields.

Personally, I have occasionally used a slightly different approach which is:

 a. insert a { SET XLINK "Please update these fields" } field at the beginning of the document.

 b. in the LINK fields, change "LINK" to "XLINK" fields

 c. modify the pathname in the field to include the { FILENAME \p } etc.

 d. update the fields

 e. save and close the file

 f. go into the XML, locate the XLINK fields (e.g. in word/document.xml if they are in the main document body) and delete the X.

 g. save the .xml and reconstruct the .docx

 h. open the .docx and update the fields

I haven't written code to do (f) to (h) but, ignoring a few difficulties, I do not think it would be particularly hard in .NET, and probably do-able in VBA.

When I first started looking at this, my first thought was probably the same as yours: what if you could fix the link code in the XML to be relative? That would still have the problems described at the beginning of this response, but would IMO be a lot easier than the { FILENAME \p } approach. Well, in a .docx, a LINK field may be represented in the document.xml (assuming the link is in the main document body), but AFAIK it will be represented in a Relationship element in the relevant .rels file (in this case, typically in word/_rels/document.xml.rels ). So all you would need is for Word to 

 1. construct the correct absolute path/file/subset name from the text in that element

 2. use that to update the LINK field code rather than the path/file/subset text actually specified in the encoding for any LINK field in document.xml

Since none of the rather obvious things I tried to specify a relative path in the Relationship element actually worked, I have to assume that either

 - it can't be done or

 - if it can be done, someone else will manage to work out what syntax makes it work.

Finally, perhaps a short discussion of the "why doesn't it work this way already?" issue. Well, personally I think it should be done as long as it can be done in a secure way (and that is a bit of a moveable feast). I do not know why Microsoft has never done it this way, but I suspect there are several contributing factors, e.g.

 a. The usual commercial concerns (cost/perceived value to customers) etc.

 b. In the case of LINK fields, the path/file/subset names are arguably not really what they seem, because they are in fact the readable form of OLE Monikers, which do not necessarily have a path/file/subset structure. It is possible that there was reluctance within Microsoft to undermine that mechanism, particularly at a time when they were pushing OLE quite hard.

 c. also in the case of LINK fields, Microsoft had their own approach to dealing with files moving around file systems, in the form of the Distributed Link Tracking (DLTS) service. e.g. if I have a linked file called c:\xyz1\abc.xls and I move it to c:\xyz2, when I re-open a Word file with a link to that .xls, the link code will be updated automatically because Word goes through OLE to open the link, and OLE can use the DLTS. 

 d. finally, OLE links are different from HYPERLINKs in the sense that an OLE link update will typically result in material being inserted in the document. But that's really little different from an INCLUDETEXT field (say).

Personally, I believe the behaviour of the DLTS is not actually very helpful, because

 1. it doesn't always work. (it only works on NTFS volumes, not e.g. FAT32, it does not seem to work on all of them, it only works if the DLTS client service is running, and although the docs. I have seen suggest it should only work in a Windows Domain networking environment, that does not seem actually to be true). But the upshot is that th user cannot easily be sure it will work or not. 

 2 .it only works with .doc files that contain links, not .docx (and it does not actually seem to work with linked .docx either - that may be something to do with the fact that a .docx is not actually an OLE object, but I do not know). This may be one of the reasons why many people insist that link behaviour changed somewhere around Word 2007 - because if you moved from .doc to docx, it has changed.

 3. It's not going to work if you copy a file then delete the original (I think).

In other words, the behaviour may seem arbitrary and uncontrollable to a user. IMO it should be clear that when you specify what you think is an absolute link, that's the way it works, and when you specify a relative link, it's clear what absolute path Word will look for. Perhaps some people would like another option, e.g. if you can't find the linked file at the absolute or relative location, look in the same folder as the .docx file, or look along some path that can be defined in the environment. But IMO that probably really is a recipe for security problems.
