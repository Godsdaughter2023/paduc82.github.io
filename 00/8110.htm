Hi,

Thanks for picking up on the typing error (TheRwo)

XLToLeft was correct and the change you have made is making the code select all the columns. Try it like this

Sub Select_Range()
Dim lastrow As Long, LastCol As Long
Dim TheRow As Long, TheCol As Long
Dim StartCell As String
LastCol = ActiveSheet.Cells(1, Columns.Count).End(xlToLeft).Column
lastrow = Cells(Rows.Count, LastCol).End(xlUp).Row
StartCell = InputBox("Enter Start cell")

TheRow = Range(StartCell).Row
TheCol = Range(StartCell).Column

ActiveSheet.Range(Cells(TheRow, TheCol), Cells(lastrow, LastCol)).Select

End Sub

williamtrotter say: The standard Windows 10 installation on a GTP-UEFI disk creates four partitions.  If you're in the Insider program, then at least one more will be created when your os is updated.  When I do a fresh install, the dialogue box gives the option of deleting old partitions and I always delete them all.  Admittedly, this is not the same as a dd wipe or a low level format, but my disks are new and I've never had a problem with the Rufus/USB setup in the past.  This includes probably 100 or more "clean" installs.

shilohbob say:

I also upgraded 4 PC,s, but on one PC the upgrade failed and virtually made the PC unusable, it indicated that it could not locate a bootable devise and no operating system was installed, after much research and trying various solutions, I finally went into UEFI/ BIOS setting and disabled the Secure Boot setting, saved & exit UEFI/BIOS > powered off the PC & after waiting for a while, restarted PC and it restarted with the previous W10 build.

Although not directly related to your question, there have been many report of this occurring with Bld. 16125, personally I feel that the build is causing some conflict with the UEFI settings on some PC,s.

williamtrotter say:

Mystery solved ... at least I have an explanation.  Both machines

are Lenovos (M93/p and T460s).  Both bios' had "secure boot"

enabled so both were "uefi only".  But apparently, Lenovo

doesn't take Windows 10 Build 1703 as an "approved" os.

But all I had to do was to disable Secure Boot, leaving "uefi only"

in place.  Then everything went without a hitch.  Go figure.

don cuthbert say:

Something is being changed by Microsoft with the way they handle Secure Boot. There were many users who found that they could not roll back from builds 16193 or 16199 to the public build 15063. Once that happened, they could not even boot 15063 USB install media created by the Media Creation Tool with the public build on it. Nor could they boot from their hard disk. Even after totally wiping all partitions from the hard disk, they could not boot from Windows 10 build 15063 UEFI install media. The newer builds had caused Secure Boot to reject the current public build as a valid secure OS.

Many reports on Feedback Hub and in these forums. Most have error 0xc0e90002. Feedback Hub example: https://aka.ms/S0jq47

Some devices, such as the Surface Pro 4, had no option in UEFI settings to clear the Secure Boot keys from firmware and had to use extreme measures to clear these keys so that they could reinstall the public build of Windows 10.

Many with this situation resorted to installing Windows 8.1 as the first step to enabling Windows 10 to be installed again, even on devices that had never had anything other than Windows 10 and had no Product Key for Windows 8.1.

One user came up with a method to install an EFI boot manager that had a utility to clear the problem Secure Boot keys.

So being able to simply disable Secure boot to get Windows 10 to install is the best case scenario in comparison to what some are encountering.

jmarco0921 say:

You may Registry problem.

Some insiders that have done rollback /restore from build 10049 have not been able to update to build 10049 again.

See if you this same Registry key setup:

Run "regedit" and navigate to HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\WindowsSelfHost\Applicability

You should remove "RecoveredFrom" from registry.

Before deleting registry key back it up 

  1. In Registry Editor, locate and click the registry key or subkey that you want to back up.
  2. Click File > Export.
  3. In the Export Registry File dialog box, select the location where you want to save the backup copy to, and then type a name for the backup file in the File name field.
  4. Click Save.

  1. In Registry Editor, click File > Import.
  2. In the Import Registry File dialog box, select the location where you saved the backup copy to, click to select the backup file, and then click Open.

After removing RecoveryFrom be sure and reboot your pc before trying your Windows update again.
