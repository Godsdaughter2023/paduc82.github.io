Advanced network functionality causes local DNS lookup to fail - Microsoft Community

ken sentell say:

Advanced network functionality causes local DNS lookup to fail

 

We're running with a Windows Mobile 5 device and having some problems with network communications when activating the advanced network functionality in the USB to PC settings.
 
We have the following scenario:
- WM 5 device (OS 5.1.476, Build 15704.3.5.0), Handheld Products Dolphin 9500 series
- The device is cradled in a docking station attached to a WinXP laptop running ActiveSync 4.5
- A web service exists on a server that is accessable via the internet without a public DNS name (by customer request).
- The request to the web service from the device will be using https
- The https certificate contains the server name which requires us to use a local DNS name lookup on the device (so the names will match). These parameters have been added to the registry in [HKLM\Comm\Tcpip\Hosts] as below
 
    [HKEY_LOCAL_MACHINE\Comm\Tcpip\Hosts\myservername.com]
    "ipaddr"=hex:c2,89,7b,b9
    "ExpireTime"=hex:99,99,99,99,99,99,99
 
- In the Settings | Connections | USB to PC control panel icon on the device we have the "Enable advanced network functionality" check box unchecked
 
 
When using Pocket Internet Explorer I can reach the server using the url (e.g. https://myservername.com/wspath/Mobile.jws). I can see on the IE status bar the status change from "Connecting..." to "Locating..." to "Opening..." before the page is shown.
 
If I then go to the USB to PC control panel function and check the box "Enable advanced network functionality" I can no longer reach the server. On the IE status bar, I see Connecting..., Locating..., and then get the error message "The page cannot be found. Check the name and try again". I suspect that the local DNS lookup is being bypassed for some reason.
 
This same behavior is occurring in our handheld application. This is preventing us from using some utility programs (Entrek CodeSnitch) to analyze our handheld application while it is running. This program will only connect to the handheld when running with the advanced network functionality (PPP?) is enabled. The problem is that when it's enabled we can't reach the server and therefore can't log in using the web service to test the rest of the program.
 
Has anybody had a similar problem or know why this is occurring?
 
Thanks in advance
Ken
ken sentell say:

I ended up solving the problem myself. For any one who is interested the solution is as follows.

 

When the Enable advanced network functionality checkbox is not checked the driver for SerialUSB is used and the Hosts registry entry is used on the handheld.

When the checkbox is checked the driver for USBRndis is used and Host registry entries are not used, rather all such lookups are directed to the host PC running ActiveSync.

 

To solve my problem, I added the server names and ipaddresses to the Hosts file in the \Windows\System32\drivers\etc folder. The names were then translated correctly to the IP-addresses.
