stefan blom say:
Page numbering options, including restarts, are stored at the section level. Also, when a section break is added, the settings will be inherited by the new section. This explains the frequent page number restarts that you are seeing. If you are really sure that you don't need the sections, you can delete the breaks.

However, a quick fix is to run the following macro on the affected document:

Sub SetContinuousPageNumbering()
Dim s As Section
For Each s In ActiveDocument.Sections
s.Headers(wdHeaderFooterPrimary).PageNumbers _
.RestartNumberingAtSection = False
Next s
End Sub

For assistance, see http://www.gmayor.com/installing_macro.htm.

mikek703 say:

A lot of section breaks sometimes messes up the numbers.  View the document in the "Draft" mode and see if there are "Continuous section breaks" between pages 3 and 4.  If they are not necessary (e.g., there are no columns in the text) delete them and try fixing your page numbers then.

Check default mailbox folder permissions - Microsoft Community

alexandru cazacu say:

Check default mailbox folder permissions

Hello again :).  

I am back with another article that involves a bit of scripting this time around (something that I do not excel at, but I do try to improve).

So, let's begin with some background to all this. I have recently handled an issue for a customer, who came to us with the following case description:

We want to check the default permissions group on all folders in all shared mailboxes to see if this is changed. We want the default permission group set on None. We had a problem with a shared mailbox where the user changed the default group, and everyone suddenly has access. Is there a Powershell script available for this?

Initially, the script available here was provided, but it could not be scaled up for the issue we were facing, due to a high number of mailboxes to be checked (hundreds), the high number of folders (ALL folders had to be checked) and the sheer complexity of the script itself which was doing more than what the Exchange Online servers can handle (throttling would kick in).

Given the above history, I've decided to see if I am capable of modifying the existing script, to make it simpler and more efficient. After several hours of testing and busting my head around the concepts of Powershell scripting, I've come up with this:

===================================================================================

#Gathering a list of the User Principal Name attribute for the specified recipient type

Write-Host "======================" -ForegroundColor Gray

Write-Host "SharedMailbox" -ForegroundColor Green

Write-Host "UserMailbox" -ForegroundColor Yellow

Write-Host "RoomMailbox" -ForegroundColor Cyan

Write-Host "EquipmentMailbox" -ForegroundColor Magenta

Write-Host "======================" -ForegroundColor Gray

$Type=Read-Host -Prompt "Please type the RecipientTypeDetails for which you wish to get the information, available values are shown above"

$UPN = Get-Mailbox -RecipientTypeDetails $Type -ResultSize Unlimited | select UserPrincipalName

$UPN=($UPN).UserPrincipalName

foreach ($MBX in $UPN) {

function Get-SharedMailboxFolderPermission {

#Extracting the Name attribute for the mailbox

$alias=(Get-Mailbox $MBX).Name.ToString()

#Extracting the Primary SMTP Address for the mailbox

$SMTP =(Get-Mailbox $MBX).PrimarySMTPAddress.ToString()

#Getting the default mailbox folders list

$folders=get-mailbox $MBX | Get-MailboxFolderStatistics | ? {($_.FolderType -eq "Inbox") -or ($_.FolderType -eq "Archive") -or ($_.FolderType -eq "Drafts") -or ($_.FolderType -eq "Outbox") -or ($_.FolderType -eq "Calendar") -or ($_.FolderType -eq "SentItems") -or ($_.FolderType -eq "Contacts") -or ($_.FolderType -eq "Tasks") -or ($_.FolderType -eq "Notes") -or ($_.FolderType -eq "DeletedItems") -or ($_.FolderType -eq "JunkEmail")} | select Identity

$rights = @()

#Adjusting the folder Identity values obtained by previous command, to comply with the Get-MailboxFolderPermission cmdlet required format. Getting the folder permissions as well.

foreach ($folder in $folders) {

$foldername = $folder.Identity.ToString().Replace([char]63743,"/").Replace($alias,$SMTP + ":")

try

{

$MBrights = Get-MailboxFolderPermission -Identity "$foldername" -ErrorAction Stop

$MBrights =$MBrights | Select FolderName,User, AccessRights, @{Name = 'SMTP'; Expression = {$SMTP}}

#With below 2 command lines I am attempting to get the Top of Information Store folder permission as well in the mailbox.

$MBRightsRoot = Get-MailboxFolderPermission -Identity "$MBX" -ErrorAction Stop

$MBRightsRoot = $MBRightsRoot | Select FolderName,User, AccessRights, @{Name = 'SMTP'; Expression = {$SMTP}}

$rights += $MBrights

}

Catch {}

}

return ($rights + $MBRightsRoot)

}

#Export a CSV file for each mailbox, that contains the default folders, their associated permissions and the Primary SMTP address of the mailbox in question.


Get-SharedMailboxFolderPermission | Export-Csv -Path "$(($MBX.ToString()))_MailboxFolderPermissions.csv" -NoTypeInformation -Encoding UTF8 -UseCulture

}

==================================================================================

IMPORTANT: Please take into consideration the following details:

- The script doesn't handle the Powershell connection to Exchange Online, so it is expected that before running the script, you are already connected to the tenant.

- The script is provided AS IS and in order to be used, it needs to be copy/pasted into a notepad without the "====" seen here above and below the script, then the extension must be replaced from .txt to .ps1.

- The script doesn't modify any existing folder permissions, but it will allow you to identify permissions which shouldn't exist and you will be able to manually modify them after.

- The script will only check the following mailbox folders: Top of Information Store, Inbox, Outbox, Drafts, Sent Items, Tasks, Notes, Calendar, Contacts, Archive, Deleted Items and Junk E-mail.

- If you need to check the folders that have been created by users as well alongside the default ones, you can modify this entry in the script shown below, by adding the FolderType "User Created". Beware though of the caveat, which is that the script will run for a much longer time, depending on the number of  user created folders in each mailbox.

$folders=get-mailbox $MBX | Get-MailboxFolderStatistics | ? {($_.FolderType -eq "User Created") -or ($_.FolderType -eq "Inbox") -or ($_.FolderType -eq "Archive") -or ($_.FolderType -eq "Drafts") -or ($_.FolderType -eq "Outbox") -or ($_.FolderType -eq "Calendar") -or ($_.FolderType -eq "SentItems") -or ($_.FolderType -eq "Contacts") -or ($_.FolderType -eq "Tasks") -or ($_.FolderType -eq "Notes") -or ($_.FolderType -eq "DeletedItems") -or ($_.FolderType -eq "JunkEmail")} | select Identity

That's about it for now. If anyone has ideas on how the script can be improved, please share!

jim gordon mvp say:
Try starting your Mac in "Safe Mode" by holding the Shift key down as the Mac starts. It will take your Mac a long time to boot this way, so be patient. 

When your Mac starts up, open the problem Excel file and try working with it. 

If the workbook is OK in this Safe Mode, then something external to Excel is causing the problem. Most likely, some other application that runs when you start up normally is using up RAM that Excel needs to run right.

If the workbook runs just as bad in Safe Mode, then there's probably something wrong with the workbook, Mac OS X, or Excel. 

While in Safe Mode and with normal startup, run Activity Monitor. On the System Memory tab look at Page-ins, Page Outs, and Swap used. If these are not zero, then you are using the hard drive as RAM, which is very slow. The cure is to add more RAM.

If you look up the hardware ID in Device Manager, sometime the Microsoft Update Catalog can be searched for drivers.  You have to use Internet Explorer for the Catalog website, but Edge will tell you that.

http://catalog.update.microsoft.com/

Check if your Device Manager shows Intel(R) Management Engine Interface when you view by connection.  Windows 10 did not used to find the right driver for mine until Build 10586 and I always had to get one from Dell.  If you double-click the device and choose Details tab, look for Compatible Ids.  On mine, I chose the 2nd one down, right-clicked and copied it.  Then in Microsoft Update Catalog, I searched for that and just now found a new driver dated June 28, 2016.  You can do the same for each device shown with your chipset name, but you may find that updating the Intel Management Engine is enough to get the right driver for the chipset.
