jeeped say:
In the VBE, go to Tools, References. AT the top of the list may be one or more with MISSING concatenated into the library name. These may need to be re-registered or substituted with another library. Also, see:

Missing References In VBA (cpearson)

chip pearson say: In the VBA Editor, go to the Tools menu and choose References. This will display the References dialog box. If there is a missing reference, it will be prominently marked with the word MISSING.

Once you've identified the missing reference, you have to decide what to do with it. If you don't need the reference, just uncheck it in the list and you're done. If you do need it, you'll have to contact the vendor or supplier to get a new version of the file. If you don't know whether you need it or you can't determine who the vendor is, take your workbook to another machine and run your application. If it works on the new machine, open VBA, go to the Tools menu, and choose References. Note the position in the list of the reference in question. Then, open the Immediate window (CTRL G) if it is not already open and enter the following command and press ENTER.

?ThisWorkbook.VBProject.References(N).FullPath

Change the "N" to the position number in the list of references of the reference in question. Copy the file named in the result in the command above to the same location on your machine. Then, on your machine, close Excel, go to the Windows Start Menu and choose Run (or press the Windows key and press R). In the Run dialog, enter

/regsvr32 "C:\Folder\Folder\FileName.dll"

Of course, change the file name to the newly copied file. This will register the reference in the system registry. Now, open your workbook in Excel. Things should work fine.

kergga say:

 

One possible reason references would be unavailable is if your are currently running code (even in break mode). If you have 'trusted' the file in 2010, then it will open with macros enabled, and if you have any macros that run automatically (such as the workbook open event) then maybe that is preventing you from troubleshooting and solving the problem.

 

In Excel 2010 you can simply rename the closed file, and when you try to open it Excel should re-prompt you to see if you want to enable macros. Say no, and then you should be able to get into the VBA (and tools/references) without issues.

 

Troubleshooting your code to know what reference is missing (or what else might be going on) would require a whole lot more information, about your code and potentially the machines where this is or isn't working (for example, are any of these machines 64-bit installs of Windows7/Excel2010). However, I suspect that if you can get into the references, you can find the one that needs to be checked.

 

One other thought, if you are working on 32-bit installs, and all other efforts fail - I have seen (on three different PCs) issues where the installation somehow left the references/add-ins pointing to the wrong location. Everything looks fine (references do not show as missing in Excel add-ins or in the tools/references) but you still get stuck in an endless loop of errors. If all else fails, disable macros, then in Excel go to your add-ins and manually browse to find each one in the appropriate folder, and select them to re-load/re-install them. Same on the VBA side. In all three cases that I've seen, this successfully solved the problem (although there is some voodoo here, I'm not sure of the exact order of reset and reloading excel, so it took me a couple of tries to get it working)

O365 DKIM and Google Domains - Microsoft Community

chris b b say:

O365 DKIM and Google Domains

I had some issues getting DKIM set up for google domains with O365, and after working with google support, this is now possible through the google domains UI using a modified syntax.

*** Email address is removed for privacy ***
10:25 AM (47 minutes ago)

Please try to add the CNAME record using this format:
  • Name: selector1._domainkey
  • Type: CNAME
  • TTL: 3600
  • Data: selector1-contoso-com._domainkey.contoso.onmicrosoft.com.
  • Name: selector2._domainkey
  • Type: CNAME
  • TTL: 3600
  • Data: selector2-contoso-com._domainkey.contoso.onmicrosoft.com.

Note:  there must be a period ('.') at the end of onmicrosoft.com in the data field.  This is what prevents it from adding an extra .contoso.com.

Once these records are created, I had to wait about 15 minutes for caching/propogation, and then you can enable DKIM in the exchange admin portal.  If you'd like to do DKIM validation, you can use an external site, such as mxtoolbox.com
