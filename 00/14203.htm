johndonnelly say:

I have a similar problem often and it is related to VBA. Here's a work-around I use:

 

1) Always have Macros disabled with notification. If you don't then Excel will crash on opening the file because it will encounter the error directly.

 

2) With macros disabled the filw should open and ask you to enable macros. If you click on "Enable Content" Excel will crash. Instead go to the "developer" tab then click on "Visual Basic" on the ribbon. On the left "Project Navigator" window scroll down until you see "Modules". Right Click on "Modules" and select "Insert" -> "Module". This will insert a blank module. I think this resets the VB as "uncompiled" or something.

 

3) Save the workbook and re-open. You can then click on "Enable Macros" and Excel will not crash.

 

4) Before you save a file that tends to crash, insert a blank module. You can always delete them later.

 

I hope this helps someone.

john w. vinson say:

A basic principle of relational databases is that you should store each piece of information ONCE, and once only. If the Ward can be determined based on the value of the Postcode, then the Ward field should exist in the Postcodes table, and noplace else.

 

You should not be storing your ward field in the schools table at all. Instead, you can use a Combo Box to select the postcode on the form bound to the schools table. You can include the Ward as one of the columns in the combo box's RowSource query, and put a textbox on the form with a control source such as

 

=cboPostcode.Column(n)

 

where cboPostcode is the name of the postcode combo box and n is the zero based index of the field containing the ward.

ken sheridan say:

You'll find an example of the sort of arrangement John describes in the file Images.zip in my public databases folder at:

 

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

 

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

 

Open the Images_Simple.acdb file from the zip archive.  Forget about the fact that this demo is intended to show ways of linking image files to a database, it's the way the first form is set up which is relevant here.  You'll see that the City combo box's RowSource query returns the County in a third column and this is shown in the text box below, whose ControlSource is:


=[City].[Column](2). 


The Column property is zero-based so Column(2) is the third column.  If you think of city as analogous to post-code and county as analogous to ward in your database, yours would be similar.

 

You'll notice that my form does have a PostCode control bound to a column in the table.  Ideally I would have set up the table so that it stored only the post-code and this automatically showed the city and county, but not having that information available I've had to omit that.


As you probably know, here in the UK we can buy post-code data which allows not only the city to be shown automatically, but the street name, so that the user then merely selects their house number from a restricted list.  This is done in many on-line forms for ordering goods etc.  Under the skin these work in much the same way as we've described; they just have a lot more data to work from.
