While this has nothing to do with Outlook 2013, I may know what's happening.

When you import a digital certificate with its public and private keys, you are asked to specify the security level of the key access.  The higher value of security requires the entry of the password that protects the key each time you use the key.  I believe this is the value you chose when you imported the key.  You can solve this problem if you have the original CER or PFX file containing the signature by removing the certificate from the Windows crypto store (using certmgr.msc or IE's interface to the crypto store - Tools>Internet Options>Content>Certificates), then reimporting the cert, disabling the higher lever of key security.  Here's a picture:

The circled option, if enabled, will cause Windows to prompt you for the private key password every time you use the certificate.  Make sure this box is unchecked.  However, make sure the second box, the "Mark this key as exportable" box is checked so that you will be able to export your private key for backup purposes or if you ever change PCs and need to reimport the file you save.

How are you trying to access your mailbox. Are you using a web browser? Are you using a mail client? What version of Windows? Did it every work? If so, what changed between then and now? 

¡firedog say:

This will only happen when connecting to a server using POP3/SMTP. Most servers nowadays require authentication before allowing the use of their service. This often used to be achieved by sending a POP3 request for new messages in a particular mailbox, which requires transmitting the username and password. Once the POP session was complete, the SMTP server would accept messages for onward transmission from the same already-authenticated user. That is what is meant by you must check for new mail before sending mail.

You may be able to avoid this by turning on SMTP authentication in the account settings in your mail program. How to do this varies from program to program, and the settings you need depend on which SMTP server you're trying to connect to. All it means is that your username and password are sent to the SMTP server at the beginning of the transaction. 

So:

  • Which version of which program are you using to get your email?
  • Who is the Email Service Provider (usually apparent from the part of your email address after the '@' symbol)?
  • What version of Windows are you using?

    Yes, you and all of the various services and applications running background processes on your PC in order to make it all work are constantly causing such items to be logged in the Security Events logs.

    None of this is truly suspicious and these event logs are really only meant to be viewed by technically knowledgeable IT professionals or others trained in what these specific items mean, usually only when portions of the operating system or software are malfunctioning.

    There are thousands of discussions of these security principals (e.g. Account Domain/Name) and special privileges events found all over forums on the web, including here.  However, few of these will perfectly match the specific events seen on your own PC, so answering such questions in complete detail is rarely possible and is why so many continue to ask, since finding the precise answer to what caused these requires intimate knowledge of your own PC and everything that's installed and operating.

    The general answers are simply more technobabble and complexity, which is why they're typically unsatisfying, but I'll provide some references anyway in hopes you'll understand why it's generally best to simply ignore these logs.  These specific events have existed since the beginnings of Windows NT, the earliest versions of what is now Windows 10, so these are the same for your Windows 7 as well.

    4634(S) An account was logged off. (Windows 10)  Microsoft Docs

    If you're willing to play a little fast and loose with the definitions, NT_AUTHORITY essentially refers to the Windows operating system itself. Or perhaps as "things the OS authorizes on your behalf."

    (At first, NT meant New Technology, a version of the OS generally meant for businesses. It contrasted with the less strict, less secure Windows 9.x kernels used in Windows 98, 98, and earlier versions. Starting with Windows 2000, the various versions were combined on a version based on Windows NT 4. Those eventually grew into Windows Vista, 7, 8.x, and the soon to be released 10.

    The "NT" token is basically a legacy token left from earlier times. You can think of it as a surrogate for Windows itself. More officially, it's the parent for a set of service users that handle background tasks and maintenance operations.

    The tokens on the right side of the slash refer to individual internal service "users" of the OS.For example, NT AUTHORITY\SYSTEM handles system services, NT AUTHORITY\LOCAL SERVICE does local services, NT AUTHORITY\NETWORK SERVICE is network services, and so on.     Essentially the same thing run on behalf of a service, which is (essentially) a utility that runs in the background. (The BITS service, for example, downloads updates in the background.)
