No, each item can be the subject of one or more transactions, so the relationship type is one-to-many.  Normally the relationship type in cases like this is many-to-many in fact, and modelled by a table which resolves the relationship type into two one-to-many relationship types., e.g.  with the following tables:

Orders
….OrderID  (PK)
….OrderDate
….CustomerID  (FK)

Product
….ProductID  (PK)
….Product
….UnitPrice

and to model the relationship type between them:

OrderDetails
….OrderID  (FK)
….ProductID  (FK)
….UnitPrice
….Quantity

The primary key of the last table is a composite of the two foreign keys.  You might like to take a look at DatabaseBasics.zip in my public databases folder at:

hi mike240se2,

if you can confirm the passwords are synced to office 365 correctly, cutover migration should not overwrite them in the incremental sync. however, please keep in mind that the steps you used are not officially support ways to sync the passwords in a cutover migration. the correct steps should be:

  1. wait for the cutover migration is completed.
  2. covert the on-premises mailboxes to mail-enabled users.
  3. deploy dirsync to link the on-premises users to office 365 users so that the on-premises passwords will be synced to office 365.

also, cutover migration won’t re-provision users for those existing office 365 users, it will report errors once the same object is already existed in office 365.

to check if the incremental sync is working, we can perform the following steps:

  1. on the migrations tab in office 365, check the value in the last synced time box for the migration batch.
  2. on the same page, check if any errors show up in the migration batch. if so, we can check the detailed log file so that we can find which mailbox encounter the specific issue.

 

also, if any issues occur during the migration, please feel free to open/create a new thread with the details in our forum and we shall provide further assistance then.

john w. vinson say:

Great suggestion, Hans. You can actually trick the UNION query by having the first SELECT specify the datatypes, but just not return any records!

Assuming that (say) the Stock_Value is a CURRENCY field, you could also explicitly cast the datatype as Currency rather than the default Long Integer number type:

SELECT -1 AS site, -1 As article, -1 As week, -1 as LY_SQ, -1 AS LY_NSV, -1 As TY_SQ, -1 AS TY_NSV, -1 as Current_Stocks, CCur(-1) AS Stock_Value, -1 as Open_PO_Qty FROM PO WHERE 1=0

The WHERE clause will ensure that no "dummy" record gets created, but it will set the datatypes.
