Rather than checkboxes I would use buttons, per best practices of using the appropriate control for the indended action.

In a fancy system which products the buttons would be about is driven by actual sales, e.g. the top 6 products on the 6 buttons. For now we will hard-code with some flexibility in mind.

When clicking a button what will happen is that you run a parameterized Append query to add a row to OrderDetails, then Requery the subform so the new item appears.

(of course I will have to guess at your exact object names)

The append query will be something like:

(this is from Northwind; I don't advocate using spaces or other funny characters in object names - they may cause notational difficulties down the road)

INSERT INTO [Order Details] ( [Order ID], [Product ID], Quantity )
SELECT [parOrderID] AS Expr1, [parProductID] AS Expr2, 1 AS Expr3;

dim qd as dao.querydef

set qd=currentdb.querydefs("myQuery")

qd!parProductId=123   'Whatever number goes with this button

qd.execute dbFailOnError

To refresh the subform is a one-liner:

Me.mySubformControl.Requery

Tom's suggestion is very workable. Not my preference (I would prefer to preserve screen real estate) but workable. A lot of these interface design issues are personal preference. But he makes one point that enhances my suggestion. I was assuming that the most common products were a set grouping. But lets say you want those products to be your top sellers. ken sheridan say: First define 'most frequently ordered'.  If you simply want to select these manually then add a Boolean (Yes/No) column such as  FrequentPurchase to the Orders table.  Using Northwind as an example (note Tom's caveat about Northwind's bad object naming) a query would be like this:

SELECT [Product ID], [Product Name]
FROM [Order Details] INNER JOIN Products
ON [Order Details].[Product ID] = Products.ID
WHERE FrequentPurchase
ORDER BY [Product Name];

If you mean the product with the highest number of orders irrespective of quantity, the you can return these with a query like this:

SELECT TOP 6 [Product ID], [Product Name]
FROM [Order Details] INNER JOIN Products
ON [Order Details].[Product ID] = Products.ID
GROUP BY [Product ID],  [Product Name]
ORDER BY COUNT(*) DESC;

If you mean the products with the highest total quantities ordered, with a query like this:

SELECT TOP 6 [Product ID], [Product Name]
FROM [Order Details] INNER JOIN Products
ON [Order Details].[Product ID] = Products.ID
GROUP BY [Product ID],  [Product Name]
ORDER BY SUM(Quantity) DESC;

In the latter two cases you might want it to be the highest COUNTs or SUMs in relation to the current customer's orders only, in which case you'd restrict the queries by a WHERE clause which references the current CustomerID in the parent form as a parameter.

Whichever of these is appropriate one way of selecting them would be to pop up a dialogue form from a button on the main form.  The dialogue form would contain a multi-select list box based on one of the above queries, in which as few or as many of the products can be selected.  In the code behind a Confirm button on the form you'd loop through the list box's ItemsSelected collection and, at each iteration of the loop, build and execute an INSERT INTO statement to insert a row into OrderDetails, inserting the selected ProductID, the current CustomerID and 1 as the Quantity.  Then requery the subform to show the new rows, and close the dialogue form.

Note BTW that TOP 6 will return more than 10 if the COUNT or SUM operations return two or more rows which tie for 6th place.

gina whipp say:

Well, this query speaks to table or tables that are not normalized.  While it is strange that it will work one day and not the other for that query it is not surprising.

You are using an INNER join which will only show results if the data is in BOTH tables which is why it returns less records than you expect.

Queries open the tables included within when they run and your is opening tables AND *thinking* about each row of data each time it runs which would explain the error.  You are asking it to do too much.  I would strongly recommend you break this up into two separate queries (or more).  If attached to a Form or a Report then consider a From\Subform or Report\Subreport scenario.

suzanne s. barnhill say:
You can't delete the symbol itself (which does not print). You can hide it (by not displaying nonprinting marks), or you can change the formatting of the paragraph not to have the "Page break before," "Keep with next," "Keep lines together," or "Suppress line numbers" property.

jrj1040 say: I guess I'm obtuse, but you say I cannot remove (edit) the symbol, but I can change it????  HOW?

suzanne s. barnhill say:

The symbol (if in fact it is the small black square in the margin that does not print) is just an indication that certain properties have been applied to the paragraph. If you double-click the square, the Paragraph dialog will open to the Line and Page Breaks dialog. The only way to get rid of the square is to disable the properties that cause it to be displayed.
