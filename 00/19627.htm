scottgem say:

I see no reason to have a separate field for date and time. If you want, you can enter the date and time into separate controls, but store them as one field. You can also display them in separate controls. 

If you want to check for a conflicted booking I would do it this way:

DCount("*","tablename","TeacherID = " & Me.cmbTeacher & " AND ClassDate BETWEEN #" & Me.txtFull ClassDate & "# AND #" & DateAdd("h",Me.txtHours,Me.ClassDate) & "#")

txtFull would be a hidden control that has a controlsource of:

=txtDate+txtStart

In your INSERTINTO, I would use txtFull to populate ClassDate and remove the ClassStart field.

ken sheridan say: Try a query like this:

PARAMETERS
Forms!frmClassScheduleDlg!cmbTeacher LONG,
Forms!frmClassScheduleDlg!txtHours SHORT,
Forms!frmClassScheduleDlg!txtDate DATETIME,
Forms!frmClassScheduleDlg!txtStart DATETIME;
SELECT *
FROM ClassSchedule
WHERE TeacherID = Forms!frmClassScheduleDlg!cmbTeacher
AND ClassDate =  Forms!frmClassScheduleDlg!txtDate
AND ClassStart < DateAdd("h", Forms!frmClassScheduleDlg!txtHours, Forms!frmClassScheduleDlg!txtStart)
AND DateAdd("h", ClassHours, ClassStart) > Forms!frmClassScheduleDlg!txtStart;

Where Forms!frmClassScheduleDlg is the name of the dialogue form.

Note that the parameter declarations are important to avoid the possible misinterpretation of the parameter values as arithmetical expressions rather than date/time values.  I've assumed that the class hours value will always be an integer.  If not change the declaration of the Forms!frmClassScheduleDlg!txtHours parameter accordingly.  

If the query works as I expect (hope?) you can then simply test to see if it returns a row  by means of the DLookup function, which will return a Null if no rows are returned.

BTW there is actually no such thing as a time value in Access only a date/time value.  When you enter a time only into a date/time column what you are really entering is a value (under the skin it's a 64 bit floating point number) which represents the point of time on 30 December 1899, which is day-zero in Access's implementation of the date/time data type.

ken sheridan say: The logic for detecting intersecting ranges is quite simple:  The start of one range is before the end of the other range, and the end of one range is after the start of the other range.  This is the logic behind the WHERE clause of the query I posted.  As you now have a single date/time value for the start time you don't need a match on the date, so in your case the logic is:

The start date/time value in the table is less than the start date/time parameter in your form plus the length of the parameter for the session in your form, and start date/time value in the table plus the length of the session in the table is greater than the  start date/time parameter in your form.  

If calling the DCount function against the base table you'll need to build the expression employing the above logic and use it as the criteria for the DCount function call.  Personally I'd use a query as I described, referencing the form's controls as parameters, because you then don't need to worry about wrapping the parameter values in the # date delimiter character or formatting it in an internationally unambiguous format when building the expression; you simply declare the parameters as DateTime in the query.  You can then call the DCount or DLookup function against the query without any criteria argument.

Another approach is to call a little function I wrote once which encapsulates the logic.  This could be called in the query or in an expression in the form itself.  Note that 'date' here really means date/time:

Public Function WithinDateRange(dtmParamRangeStart As Date, _
                       dtmParamRangeEnd As Date, _
                       dtmDataRangeStart As Date, _
                       dtmDataRangeEnd As Date) As Boolean
                       
' Accepts: date at which parameter range starts
'          date at which parameter range ends
'          date at which data range starts
'          date at which data range ends
' Returns: True if data range intersects with parameter range
'          False if whole of data range outside parameter range
 
   WithinDateRange = _
       dtmDataRangeStart <= dtmParamRangeEnd And _
       dtmDataRangeEnd >= dtmParamRangeStart
       
End Function

One thing to note about this function is that it uses the <= and >= operators so will detect contiguous ranges with a common start/end date or vice versa.  In your case I don't think this would be regarded as an intersection, so you'd change the operators to < and > respectively.

albert d. kallal say:

The most simple approch is to NOT use a call back, but the Access expression service.

 

Simply place in your application in a standard code module the following pubic function that you call DIRECT from the ribbon xml.

 

Public Function OpenForm(strForm As String)

 

   DoCmd.OpenForm strForm
  
End Function

 

Now, for any ribbon xml, to open a form use this for a button:

 

          <button id="btnOpen" label="Open Form Test"
               imageMso="FileOpenDatabase"  size="large"
              onAction="=OpenForm('frmtest')"  />

 

That is it. And for more buttons + forms to open, I just cut + paste the above tiny bit of xml over and over.

 

Fast, easy, and note how you enter the form into the xml and you DO NOT have to create a new call back each time.
