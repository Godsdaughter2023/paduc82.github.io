The 360 only recognizes FAT32 and HFS+ volumes. It does not support NTFS. Most Western Digital external drives these days come from the factory partitioned/formatted as one big NTFS partition.



The 360 itself can repartition/reformat a USB-connected HDD into one gigantic FAT32 volume (something Windows hasn't been able to do natively for a long time). Actually, there are three different things of a 'Total available graphics memory'

 

1. Dedicated graphics memory - RAM physically in the graphic card. Every graphic card has it. Built-in graphics solution don't have this.

 

2. Dedicated system memory - RAM in your system but ONLY used by the graphics solution built-in to your system (It is not applicable if you have a separate graphic card).

 

3. Shared system memory - RAM in you system that can by used by the graphic card or built-in graphic solution AND also used by your CPU.


You could change the amount of shared memory, if the BIOS allows it. You cannot change the amount dedicated system memory If you don't have a built-in graphic solution.

It sounds to me as if you've found a genuine bug in the Windows 7 implementation of zip/unzip.  Just as a guess, given the fact that your zip file was created by a non-Windows OS, the problem you're seeing is probably related to the [different] issue discussed here:  https://blogs.msdn.microsoft.com/asklar/2012/05/03/why-do-zip-files-from-mac-os-show-up-as-greenencrypted/)

Given that Windows 7 is now at least 2 generations of Windows old (depending on how you count 8, 8.1, and the various "builds" of 10), Microsoft isn't going to be the least bit interested in fixing it.  That's especially so given your report that the bug appears to have been fixed in Windows 10.

It's difficult but not impossible to report bugs in Microsoft products.  For some products, see https://connect.microsoft.com/

For Windows 10, see http://www.techulator.com/resources/13419-How-to-report-Windows-10-bugs-to-Microsoft.aspx

There's apparently also a feedback mechanism in the current Microsoft Office programs, but I don't know the details.

This post from 2013 explains the state of Microsoft bug reporting (for a more amusing version, see http://www.schveiguy.com/blog/2017/05/how-to-report-a-bug-to-microsoft/ ):

The are many ways to report bugs to Microsoft, but there's only one that will get your bug looked at.

It's to open support case with tech support. The case may be not free; however, if the software indeed not working as expected, or the solution was not documented, most likely you will not be charged. I personally had dozens of such incidents, reported numerous bugs, and the balance is 0.00. So there's certain risk here, make sure you have explored all avenues.

Once the case is opened, you have to follow up patiently. Initially it will be assigned to 1st level support overseas, and that representative will try to resolve it. You have to be patient and allow him to make all the efforts he can think of. Of course, it's possible that he will resolve it, for example by having you install a hotfix. That hotfix may be totally unreleated to your problem, but still fix it. If this is the case, ask to waive support fee since it was not documented.

If there's no resolution, eventually he will give up. At this point, insist that the case is escalated. He will escalate it, and there will be second round of attempts to resolve it, already by someone in US.

When this fails as well, this representative will be able to send it to triage at Microsoft that will decide whether this is indeed something that needs to be fixed. At this time you will be asked to provide business justification, that is, you will have to show how much dollars your company is losing because of this problem.

The triage then will decide. This whole process from the initial opening the incident can take months and even years, keep that in mind when you start it.

I underscore, this is the only way to have your bug in released product looked at individually. Of course, this is not the only process Microsoft is using to decide what to work on, but this is the only process you have direct access to.

Microsoft also runs Connect, however this is about future products, the window to report bugs that actually have a chance to be looked at is narrow, and in general Connect is much more marketing tool than anything else.
