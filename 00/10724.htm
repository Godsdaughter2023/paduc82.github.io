ken sheridan say:

That sounds like a very bad design as, in Table1, you would be storing values of more than one attribute in a single column.  This violates Codd's Rule #2, The Guaranteed Access Rule, which is:

Every datum (atomic value) in a relational database is guaranteed to be logically accessible by resorting to a combination of table name, primary key value and column name.

Simply split the column in Table1 into two.  If the two values, in combination, constitute a candidate key of Table1 the make them the composite primary key of the table.  You can then create an enforced relationship, which at present you cannot.  Concatenating the values together in a computed column in a query, or in a computed control in a form or report, is a trivial task if you need to return them as a single string expression for any reason.

BTW you might see occasionally see it said that storing values of more than one attribute in a single column means the table is not normalized to First Normal Form.  That's not really true.  1NF requires that there be only one value of each attribute in each row of a table.  Formally it is defined as:

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple contains exactly one value for each attribute.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).  However, the reason we use these terms rather than the ones which we are more used to is that the model does not really have any concept of column in the same way that a table does, but of attribute.  Storing two attibutes in one column does not affect the fact thata they are separate attributes, so the table is normalized to 1NF, but the values of those attributes cannot be addressed in accordance with The Guaranteed Access Rule.  What you are doing is a good example of why the rule exists and should be obeyed.

jay freedman say:
Technically, the Now function returns a value of the Date data type, but combining it with a string should give you a string (this is called data type coercion, or automatic conversion). Exactly what error are you getting?

When you get it working, you probably won't like the default format of Now, which includes the date and time. if you want to control exactly how the date (and/or time) looks in a string, use the Format function. For example, Format(Now, "yyyyMMdd") executed today would return the string 20120805.

jay freedman say:
First likely problem is that there's no backslash after the "G:", so no matter what else you append to it is going to be an invalid filename.

The second and more likely problem is that appending just Now to the "Backup" is going to include the current time, with colons after the hour and minute parts -- and colons are not valid characters in any filename. Use the Format function as I advised before; if you want to include the time, then put different separator characters into the time that are valid in a filename.

jay freedman say:
Try it this way, using underscores instead of slashes or colons, both of which are invalid characters for filenames (http://support.microsoft.com/kb/177506).

The meanings of the letters in the format string are shown near the bottom of the page at http://msdn.microsoft.com/en-us/library/office/gg251755.aspx, so you can figure out how to change it to get any form you want.

The correct syntax would be
