timyoungls say:

Ok.  It is nice when a problem consistently returns.

It turned out to be my webroot antivirus.  Occam's Razor is not always correct; sometimes multiple things can happen at once.

I am not entirely sure what is happening behind the scenes, and I will try to post a followup for how to solve it instead of rebooting or turning off and on the virus protection...  There should be a way to disable checking of a certain directory or something.

The symptoms are basically that, when a directory of contents is created and then deleted, webroot would do something to the directory (and probably the contents) so it can finish scanning it.  So the directory remains, even though it is deleted.  You cannot see the permissions on the directory, you cannot delete it, and you cannot see the contents of the directory.  You cannot remove it, rename it, or do anything to it.  It basically blocks your usage of that.

I have not tried it in all cases, but it seems that when a new directory is created and then quickly deleted, the above is what happens.

I have not yet just sat and watched it for an hour or so to see if it disappears on its own.

But, if I disable webroot, then in an instant, the directory disappears, and then I can re-enable the protection.

Some of my initial tests, as posted in my first and second post, were probably misleading.  After a reboot, I would re-run my tests (reinstall, etc) which would create a new temp folder, delete it, and then block it.  I thought the original temp folder was blocking it, but the process recreated the problem using the same directory name, so I thought it was the original directory.

It is nice to be able to say that it is not the fall creator's update, nor ntfs corruption.  It just happened to coincide with when I did the update.  I am not sure if this is something new to webroot, or why it just now showed up.
