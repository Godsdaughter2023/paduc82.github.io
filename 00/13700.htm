fazul---01 say:

I used to be a happy user of Nokia Audiobooks, which, shamefully, never got officially released from Nokia Beta Labs. This program had it all: a lovely interface displaying the image of a book with the right cover and showing the playing position as a bookmark on the pages of the book image, recall of last position played, bookmarks, sleep timer, compression into awb (which was handy in S60 devices, now we can manage with mp3 and a large micro SD card). The companion application that generated the indices and compressed the audio files to awb was clunky and only for Windows, but a number of scripts were available to seamlessly do the same in Linux or elsewhere. There was just one bug with large files that could be circumvented by chopping down the audio files.

 

So as an audiobook fan I didn't need much else. Alas, my E51 eventually cracked, then the same happened  to my E52, and at that point the touchscreen revolution (or nightmare) was in full force. So after a lot of search, and even if Symbian was almost officially "dead", I got a 701 with Symbian Belle at the start of January 2013, which I immediately updated to FP2. What a great device (bar some issues with wireless connections in older routers which I still expect Accenture to fix...). If you are not a snob about Apple's substandard devices and ignore the "app" craze (most apps are tat anyway...) then the 808 pureview and the 701 are among the most useful and featureful mobile devices.

 

BUT: where was the audiobook player for Nokia Belle?! This was bad, especially given that the device comes with an integrated FM transmitter which works extremely well... The old S60 application obviously did not work and had not been ported (THANK YOU NOKIA).

 

First I installed Audiobook Reader 1.0 from the Nokia Store, but this was a terrible program. It was ugly, it was clunky, it connected to the internet to download random covers rather than using a local file, it made the phone freeze, it was unwieldy and not useful to sum up. So I  just scrapped it. Audiowave media player, also in the store, can recall the position of one single audiobook, but was equally awful. I binned it.

 

After a lot of searching, I realised that there is an easy way to handle audiobooks using the podcast capability of Nokia Music Player. Assume that your audiobook is in a folder containing all the files in playing order in mp3 format, with the book cover embedded as an ID3 tag (this last point is not necessary but it is nice to see the cover of the audiobook when you play it...). Just copy the folder to F:/Podcasts (assuming that your player uses the F: drive) and then refresh the library of the music player. Don't copy it to F:/Music or it will be dealt with as a music album (probably not if you alter the ID3 tags to say it is a podcast, but it is simpler to just copy the audiobook to the podcasts folder without further ado).

 

To play the audiobook select Podcasts in the player and then long press the audiobook name and choose "listen". That's it. You can have as many audiobooks as you want, and the last playing position of each of them will be remembered. The integration with Belle's features (drop down menu to see currently playing file, FM transmitter control, volume control) is excellent. If you actually go into the audiobook rather than choose "listen", you will see that unplayed files have a red asterisk, partially played files have a white asterisk and played files have no asterisk. You can change the status of a file by long pressing and choosing "mark as played" or "mark as unplayed" as whished, if you want to replay parts of an audiobook.

 

Caveats: no bookmarks (but I rarely need them) and no sleep timer (but there is a nice application in the store called Music Stop that does the job very well). 

 

I wish that Nokia had explicitly mentioned Podcasts and Audiobooks in their player to make things easier, and that the features above had been implemented. I assume that Nokia diehards are the ones who have to do the work, because Nokia works the bare minimum after selling the device to you.

 

Enjoy!

dirk goldgar say:

If I understand you correctly, you are operating the database in the OneDrive folder.  This not a suitable location for Access because Access doesn't read and write the database file as a whole, the way (for example) Word and Excel write their document files.  Instead, Access writes to individual pages within the database file, and I believe that each time it does so it updates the file's last-modified stamp.  Although I'm not privy to the inner workings of OneDrive, I think it might take each page write as a cue to initiate a synchronization, which would then get involved in lock contention with Access for the whole database file.  And it would be particularly problematic if two or more different people were actually working in the database at the same time, because they might see incompletely updated data pages, leading to a possibility of a corrupt database.

I'm surprised it appeared to be working before the OS update.  Maybe something changed in some OneDrive settings?  I don't know, but I don't recommend this setup.  Others have experienced database corruption when running Access databases on OneDrive.

vanderghast say:

You can add another database as "Reference" to your database, which will expose Forms and Reports (in addition to VBA code in modules), and classes from this "r_database", but while you can refer to these classes (forms and reports) in you code of your "f_database", you won't be able to create any object defined in r while in f.

It is like DAO.Recordset. You just can't :

     Dim  x  As  NEW  Dao.Recordset

neither

    Dim x As Dao.Recordset :  Set x = NEW Dao.Recordset

It is rooted in COM. Access-VBA only allows you to define "Public-Not Creatable" in addition to "Private", class instances.

So, f cannot create an r object.

But like CurrentDb can create a DAO.Recordset for you, in the same manner, r can creates object, for you, and then, you are able to use it in f.

So, you will need a method, in a standard module, of r, which creates and return the required object, like:

==== code in r_database ====

Public Function MakeMeSomeClassInRObject( ) AS SomeClassInR

    Set MakeSomeClassInR = new SomeClassInR

End Function

=======================

And NOW, in f, you can use code like:

==== code in f_database ====

Dim toto AS r_database.SomeClassInR

   Set toto = r_database.MakeMeSomeCallInRObject( )

======================

 

And what you do with the object, define in r, but used in f, is up to you. Note that like CurrentDb.OpenRecordset, the public function that creates the object for you can accept argument to deliver a FULLY INITIALIZED object.  In fact, it is a way to be sure that your code always works with a fully intialized object (or with your object set to Nothing) rahter than having to deal with object 'in the progress' of being usable.

 

Now, for form and report used as sub form/ sub report, I think that this will imply to add the object to an existing form/report, open in design view, by your code (like wizards do), and sincerely, I am not sure if that worths the trouble... not sure at all. But if you really want to explore the possibility, ... you can. And good luck.
