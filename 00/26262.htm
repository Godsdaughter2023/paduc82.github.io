ken sheridan say:

Reading some documentation, I believe MAX() only works on numerical values in Access.


Not so; you can use the MAX operator with string expressions; it will return the value which occurs last in the sort order, sorted alphabetically.  In your case, if there is no more one student per grade per last name, then this is immaterial of course, so you could also use the MIN operator to return the same results.

However, you should call the IIF function rather than the SWITCH function.  The following is a simple example I've tested with a dummy table of contacts in which I've artificially ensured that the CityID values are distinct per last name value:

SELECT LastName,
MAX(IIF(CityID=1,FirstName,NULL)) As City1,
MAX(IIF(CityID=2,FirstName,NULL)) As City2,
MAX(IIF(CityID=3,FirstName,NULL)) As City3,
MAX(IIF(CityID=4,FirstName,NULL)) As City4,
MAX(IIF(CityID=5,FirstName,NULL)) As City5
FROM Contacts
GROUP BY LastName;

If there were more than one row with the same last name and CityID values (analogous to more than one student in the same family in the same grade in your case) the contact(student) whose first name is last in the alphabetic sort order would be returned for the city(grade) in question.

The differentiation is rather simple, determined primarily by why the updating is being done:

  • OTOH, if editing changes have been made to add/delete/modify TOC entries, the entire table option should be used.

If you never again want to see the prompt, assign a keystroke to the UpdateTocFull command or add it as a toolbar button. Perhaps that command can also be addressed programmatically. If so it should bypass the prompt. I can't comment further on that because I'm not a programmer.

I defer to his experience, but from what I've seen deleted headings are instantly expunged & the prompt does not appear regardless of whether page numbering has been affected. AFAICT, it's only new entries or revisions to existing entries that are ignored when the page numbers only option is presented & selected.
