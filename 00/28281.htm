teylyn say:

For the record: the steps above will fail in Excel 2013, so for completeness' sake here's what to do in Excel 2010. 

- create a table somewhere on the template sheet with two columns and two rows. In my example this table starts in cell O2

X Y
=Today() 0
=Today() 1

- Select O2 to P4 and copy

- Select the chart on the chart sheet and use Home Ribbon > Paste Special. Select New series, values in columns, tick Series Names in First Row and tick Categories in first column and hit OK.

- now a tiny new data point is visible stacked on top of the Task 1 and Task 2 data markers. 

- right-click the new data series and select "Change Series Chart Type". This will bring up the new combination chart dialog.

- Find the "y" data series and tick the box next to the drop-down to send the series to the secondary axis.

- then select Scatter with Straight line and markers from the drop-down 

- a line will appear on the Gantt chart and another vertical axis will appear on the left.

- format the secondary vertical axis to have a minimum of 0, maximum of 1, no markers and no labels 

- format the series line to have no markers and make the line red.

willy denoyette say:

Note that this doesn't reflect the size of the windows folder, the real size is lower.

Run this:

dism /online /cleanup-image /analyzecomponentstore

This should return something like this:

Windows Explorer Reported Size of Component Store : 6.81 GB
Actual Size of Component Store : 6.80 GB
    Shared with Windows : 6.15 GB
    Backups and Disabled Features : 621.39 MB
    Cache and Temporary Data : 33.53 MB

Shared with windows means that the real files are stored somewhere else, WinSxS mostly contains hard links and their sizes are counted twice (or more if more links).

So you can safely say that the real size of Windows is the reported size in properties - shared size of the component folder (WinSxS).

You can safely remove the Backups from the CS by running dism with /startcomponentcleanup, and you can further reduce the CS by removing the disabled features (stuff that you never use - see Programs and Features).

scottgem say:
Sorry, Russ, but I suspect you are more comfortable with Excel and less comfortable with databases. The reason why this doesn't work in Access is that data is not stored in Access in any sort of order. Its just a loose jump of bits tied together by a primary key (or a GUID internally). Therefore the order of records in any recordset is based on the sort order applied to that recordset. Which can be changed at any time.

The x of x you see in the Navigation bar is arbitrary. Change the sort or and record 123 becomes record 456.

Data is related between tables by the use of key fields. So if you want record 1 to correspond with record 8 then you want the key fields between those two records to be a relationship. Not some number assigned as the result of a sort.
