e.j. gun say:

In Excel, time is stored as the fractional part of a decimal number.  For example, the value that Excel uses to represent 12:00 noon would be 0.5.  Why?  Because 0.5 is half of a day, which is the same as noon.

Excel also happens to represent the date by using the integer part of a decimal number.  In your case, you are feeding Excel an integer number, so Excel is trying to interpet it as a date.  Hence the time (the fractional part) is zero to Excel, so you get "00:00:00".

 

Try this instead (assuming the variable txtTime is a String, not Integer or Long):

 

    Select Case Len(txtTime)
        Case 6
            txtTime = Left(txtTime, 2) & ":" & Mid(txtTime, 3, 2) & ":" & Right(txtTime, 2)
        Case 5
            txtTime = "0" & Left(txtTime, 1) & ":" & Mid(txtTime, 2, 2) & ":" & Right(txtTime, 2)
        Case 4
            txtTime = "00:" & Left(txtTime, 2) & ":" & Right(txtTime, 2)
        Case 3
            txtTime = "00:0" & Left(txtTime, 1) & ":" & Right(txtTime, 2)
        Case 2
            txtTime = "00:00:" & txtTime
        Case 1
            txtTime = "00:00:0" & txtTime
        Case 0
            txtTime = "00:00:00"
        Case Else
            txtTime = "00:00:00"
    End Select


 

HTH,

Eric

bill manville say:

Generally if you insert a row within a block of rows that have the formatting etc that you desire the new row will inherit for formatting (though not the formulas) of the surrounding rows.

So if you have an unused row formatted in the desired way, but otherwise blank, immediately below your other rows and you insert above this row you should find the formatting is as you wish.

Other options are available:

- use a table - you can convert your data into a table by selecting the data including header row and using Insert > Table.  The table will replicate formatting and formulas on new rows as you add them below the table

- use a macro to insert your new row and copy the formatting and formulas from a reference row.

First, let’s take a quick look at some examples of how not to deal with Windows Defender's Controlled Folder Access blocking events:

https://answers.microsoft.com/en-us/windows/forum/windows_10-files/windows-10-home-unauthorized-changes-blocked/1feba668-4273-4a1b-8cd3-ac6485cb4280

https://support.microsoft.com/en-us/help/4046851/windows-10-controlled-folder-access-windows-defender-security-center

In most cases, Controlled Folder Access (ransomware protection) has been turned on by the user, or by an administrator, for the purpose of providing ransomware protection, and of course tuning it off will simply cancel that protection. But turning off your ransomware protection isn’t really necessary, because allowing an unrecognized app is a lot easier than the third-party documentation is making it out to be, especially if you use the shortcut that I describe below.

So let’s take a look at a real-world example of how this works: The other day I encountered a stream write error when I went to save a VHF station in my SDR app (SDRuno) – and a few seconds later, I got a notification that Windows Defender had blocked SDRuno from writing to a protected folder.

Now there’s a very handy shortcut for allowing an app – just click on the notification, and that immediately brings up the Allow an app through Controlled folder access window. But before you click on the notification, be sure to either jot down the file path provided, or commit it to memory – because clicking on the notification will also erase it. Since I hadn’t expected that to happen, I had to open up Event Viewer and find the blocking event (Event ID 1123) in order to confirm the file path for SDRuno.exe:

Right-click on the Start button and select Event Viewer.

Then navigate to Applications and Services > Microsoft > Windows > Windows Defender > Operational

Filter for (or just look for): Event ID 1123

The file path is all you need to know in order to allow access for an unrecognized app. Here’s a Microsoft article that covers the topic in a way that at least makes sense.

https://docs.microsoft.com/en-us/windows/threat-protection/windows-defender-exploit-guard/customize-controlled-folders-exploit-guard

The Allow an app through Controlled folder access link is right there in the settings under the Controlled folder access heading – and this option uses an open-file dialog to choose apps (executable files) that will be allowed access to protected document files (whitelisted). So isn’t it amazing that the Windows Support document entitled Allow a blocked app in Windows Defender Security Center doesn’t have a word to say about that link, or about the Allow an app through Controlled folder access window:
