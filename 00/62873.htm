.NET Framework 4.5.2 fails to install with error 643. - Microsoft Community

zaknfn say:

.NET Framework 4.5.2 fails to install with error 643.

We have run into this issue:

.NET Framework 4.5.2 fails to install. Initial install error details list the error 643. We then reviewed the logfile created and it showed an error of 0x80070643. When checking the MSI failure log, we searched for keyword 0x80070643. When checking the MSI failure log, we searched for keyword failed and found numerous failures stating 'MsiViewExecute failed', then the next line would state 'Could not reserve disk space for <filename>'. Then we found the error stating 'secrepair: failed to create hash for the install source files'.

After researching further, found this particular issue to be linked to 3 Windows updates (KB3008627, KB3000988 & KB2918614). After uninstalling these updates, we temporary hid the updates to prevent the reinstallation of them, then installed .NET 4.5.2 using Windows Updates and rebooted. Once installed, we un-hid the uninstalled updates and reinstalled them.

I hope this will be able to assist others in resolving this issue if it happens to them. There were other possible resolutions out there, which we tried, but this is the one that ultimately resolved the issue.

Windows 10 did not fully install USB 3.1 asmthub3 driver which caused - Microsoft Community

ducky3280 say:

Windows 10 did not fully install USB 3.1 asmthub3 driver which caused BSOD and unable to reboot or repair

I'm writing this post "after the fact" in hopes that someone else would not have to go through what I went through which is 12 hours of thinking I'd have to wipe out my installation of a brand new build on Windows 10 after spending weeks getting it the way I like it.

I finished building a new computer with the ASUS X99 Deluxe II motherboard.  I had been working with Windows 10 (1607) without problems for about 5-6 weeks.  I went to move the computer into its more permanent location and in the process I plugged in all of my USB devices including using (for the first time) the USB 3.1 port slots on the back of the machine.  This I believe was the fatal flaw.  After the machine went through the motions and before the login screen, I got the BSOD with a message:

Your PC ran into a problem and needs to restart.  We'll restart it for you.  Stop code: SYSTEM THREAD EXCEPTION NOT HANDLED coupled with the cute frowny face.

The machine rebooted a couple of more times with the same results.  Many hours passed where I tried to recover from this situation, none with any success.   A summary of things I tried:

  1. Pulled out everything from the USB 3.1 ports and rebooting, no help.
  2. Pulled out all cards other than video and disconnected all drives other than boot drive, this also did not work
  3. Tried to boot into Safe mode (Cmd Line, network, regular).  None would boot.
  4. Tried to repair the disk, both with the original Win-10 DVD and with a USB stick I created from the media creation tool, utilizing Windows PE.  These attempts also did not work.  The repair would churn away for 5-10 minutes and then exit with "Root cause found: Unspecified changes to system configuration might have caused the problem."
  5. I tried to boot with the "logging" turned on.  The results I got 5 BOOTLOG_LOADED messages until it failed, the last one being werkernel.sys
  6. Tried to look for a system restore point and unfortunately I didn't have one.
  7. Tried to run the SFC command and I got back messages that my install was fine
  8. Then I tried searching for every single log file on Windows and Windows/System32 that I could find.  That's when I started getting closer

I ultimately found setupapi.dev.log and noticed Windows had tried to install a version of asmthub3 that appeared to fail.  The suspicious entries were the ones with an "!":

     flq:                          Copying 'C:\Windows\System32\DriverStore\FileRepository\asmthub3.inf_amd64_d51694ebc3f18205\amd64\asmthub3.sys' to 'C:\Windows\System32\drivers\asmthub3.sys'.
!    flq:                          MoveFile: 'C:\Windows\System32\drivers\SET7E38.tmp' to 'C:\Windows\System32\drivers\asmthub3.sys' FAILED!
!    flq:                          Error 5: Access is denied.
     flq:                          Shuffle-MoveFile: 'C:\Windows\System32\drivers\SET7E38.tmp' to 'C:\Windows\System32\drivers\asmthub3.sys'
     dvi:                          Existing files modified, may need to restart related services.
     sto:                     {Configure Driver Package: exit(0x00000bc3)}
     ndv:                     Restart required for any devices using this driver.
     dvi:                     Install Device: Configuring device (oem35.inf:usb\asmediausbd_hub,Hub_Product2_Install.NTamd64). 06:59:40.204
     dvi:                     Install Device: Configuring device completed. 06:59:40.206
     dvi:                     {Restarting Devices} 06:59:40.206

After that there was a long string of NUL's which I'm not including in the above snippet.  These log entries gave me hope that the USB driver was truly the cause of the problem.  I then spent several hours trying to understand how DISM works since I presumed that if I was going to try to remove a driver, I'd need a program like that to make sure the registry matched.  I had noticed that there already was a version of asmthub3 installed a couple of weeks prior, but it must have been than when I plugged my mouse into the USB 3.1 port, this driver was getting replaced with another driver with the same name.

Of course this asmthub3 is a system driver and I wasn't sure if I was going to commit configuration suicide by trying to mess around with that.  Well, it didn't matter as DISM didn't seem to list the asmthub3 driver in the /Get-Drivers option.  I was getting ready to call MS Tech Support and spent the $99.

But I tried one last thing.  Looking on the disk I saw SET7E38.tmp and what appeared to be the different asmthub3.sys in the Windows/System32/Drivers directory.  Using Windows PE command prompt, I saved the old asmthub3 and renamed SET7E38.tmp to be the new asmthub3.sys.  I rebooted and to my shock, the computer booted.  I'm in the process of making restore points now :)

Hope this helps someone.  It does look like a Windows bug of some sorts.
