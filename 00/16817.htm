rob koch say:
There seems to be a bug in Microsoft Security Essentials in which the text for the buttons (and the green/red notification flag on the top) does not show up if windows is set to use the font MS Sans Serif in message boxes.


Yes, that's correct since that's a simple Raster "fon" file and not a TrueType font, so it isn't as scalable and will have display issues with certain screen display combinations.

 

http://technet.microsoft.com/en-us/library/cc750361.aspx

 

Raster fonts are stored in files as bitmaps and are rendered as an array of dots for display on the screen and printing on paper. Raster fonts cannot be cleanly scaled or rotated.

 

Vector fonts are outline fonts and are rendered from a mathematical model, where each character is defined as a set of lines drawn between points. Vector fonts can be scaled to any size or aspect ratio.

 

TrueType fonts are outline fonts and are rendered from line and curve commands as well as a collection of hints. Windows uses the line and curve commands to define the outline of a character or symbol. Windows uses the hints to adjust the length of the lines and shapes of the curves when scaling or rotating the character. TrueType fonts can be scaled and rotated.

 

Note that you are likely choosing the MS Sans Serif Regular font, so try the MS Reference Sans Serif Regular font instead, which is a TrueType font and so should work with applications that require it like MSE.

john w. vinson say:

I did a Paradox to Access conversion several years ago. I don't recall the versions (it's long enough ago that it was probably Access 2.0). It's not one of my more enjoyable memories!

The two programs have utterly different philosophies. Access is much more "relational'; it's easy (and essential) to define table relationships and construct multitable queries in SQL; you can use VBA code but it's often possible (even easier) to use queries and forms with code only added for enhancements, bells and whistles. Paradox makes less use of relational joins and much more use of code.

In addition, Paradox user interfaces are completely structurally different than Access Forms. They really have very little in common. When I did the conversion, I had to use the Paradox application more as a "specifications design document" than as a resource; I had to create the tables, migrate the data (getting rid of a ton of calculated fields which had to be reintroduced in queries), and build the forms and reports from scratch.

Good luck. This'll be a lot of work but it can be done.

 

John W. Vinson/MVP

scottgem say:

Well, first you shouldn't be using lookup fields. This has been discussed many times in this and other forums. You can refer to this article on lookup fields: http://www.mvps.org/access/lookupfields.htm.

Second, your conversion of the lookup TABLES may not have been necessary. I spoke about this in another thread here. While the usual model is to have a lookup table with two fields; an autonumber ID and a description its not an absolute necessity. If you descriptions are short text values, then you can use a single text field in your lookup table. And, therefore, store that text value in your table.

As to converting lookups, I would have done it this way:

Step 1. Create a MakeTable query to create table from the values in the main table. Something like:
SELECT DISTINCT fieldname FROM table INTO tlulookup ORDER BY feldname;

Step 2. Add an autonumber field to the new table and make it the PK

Step 3. Add a new field to the main table naming it the same as the new Autonumber PK in your lookup table

Step 4. Run an Update query populating the new field with the autonumber based the text value

Step 5. Remove the old text field

Hope this helps, Scott<> P.S. Please post a response to let us know whether our answer helped or not.
