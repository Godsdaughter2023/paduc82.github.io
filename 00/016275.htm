kosh vorlon say:

Hi highstream,

 

I think MSE is fine, but there may be some corruption on your drive.

 

I recommend you check your hard drive for general or system file corruption as follows:

 

Go to Start / All Programs / Accessories / Command prompt and right click on command prompt and click run as Administrator (the last part may not be needed in XP, if there's no run as Administrator option when you right click, then just open the Command Prompt).



Type sfc /scannow and enter and let it run.  It will scan and try to fix some of your system files.  Hopefully it will complete with no corruption it could not repair (if there is such corruption post back here or try to analyze it to find the problem file(s) using http://support.microsoft.com/kb/928228).

 

While in Command Prompt, type chkdsk /f /r and enter and let it run.  It will want to schedule itself to run at the next restart.   Answer yes and then reboot to run the program.  It will scan and try to fix any corruption or bad sectors on your hard drive and mostly remove that as a potential cause.

 

Then let's do a more throrough test of your hard drive. 

 

To test your hard drive, try HD Diagnostic (http://www.carrona.org/hddiag.html) or this test: Hard Drive Diagnostics Tools and Utilities (Storage) - TACKtech Corp  (http://www.tacktech.com/display.cfm?ttid=287) or go to the website of the hard drive manufacturer to see if they have a specific diagnostic utility for your drive.  That will provide better proof the drive is OK.

 

When done, defragment the hard drive as a final step to restore it to its best operating condition.

 

Hopefully that will resolve the problem unless you discover you need to do a repair installation or need to replace the hard drive as you run the tests.  But that's a matter for http://answers.microsoft.com/en-us/windows/forum/windows_xp-system?page=1&tab=all&tm=1311056281858.

 

I hope this helps.

 

Good luck!

doug robbins - mvp office apps & services (word) say:

The following code will copy the range that begins with Background and ends with Discussion

Selection.HomeKey wdStory
Selection.Find.ClearFormatting
With Selection.Find
    .Execute FindText:="Background", MatchWildcards:=False, Forward:=True, Wrap:=wdFindStop
End With
Set rngFind = Selection.Range
With rngFind
    .End = ActiveDocument.Range.End
    .End = .start + InStr(rngFind.Text, "Discussion") + 9
    .Copy
End With

john korchok say:

It's possible that PowerPoint is exporting the gradient incorrectly. But I know that at least  part of the problem is lousy application of color management. It's reliably reproducible every time. Here's the method for anyone caring to verify this for them selves:

  1. Create colored shapes in PowerPoint. I used one of my test files with separate squares, each with a 10% variation in transparency, just because it's easier to measure reliably than a gradient. But this works with any color variation, not just transparency.
  2. Export to a PDF.
  3. Take screenshots of the slide in Edit mode, in Slideshow mode and of the PDF in Preview or Acrobat. These shots will show the effect of color management.
  4. If you have specced a color in PowerPoint and don't know the RGB values (as when you apply transparency), export the slide to a PNG or TIFF. This shows the values specced in the PowerPoint file, unaffected by color management.
  5. Open the PDF in Adobe Illustrator and check the values. This shows the PDF values unaffected by color management.

Here are the values for the green in the upper left that has no transparency:

Export to PNG 146 208 80
PDF File 146 208 80
PowerPoint Edit Mode Screenshot 141 209 82
PowerPoint Slideshow Mode Screenshot 146 208 80

From this, we can see that PowerPoint in Edit Mode and Preview are both using color management, but getting different values, none of which are the original.
