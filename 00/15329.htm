dirk goldgar say:

Just to verify some assumptions:

  • You have a table named "Transactions"?
  • That table has a field named "ID"?
  • The table contains at least one record, with an ID value that is greater than 0?
  • The maximum value in the table is currently <= 32767?

Assuming those assumptions are true, then I can't think of a reason that the statement:

    intSearch = DMax("[ID]", "Transactions")

would assign a value of 0 to intSearch.

However, if the table is empty, then that statement will fail, leaving intSearch with its default value of 0.  So that could be your problem, except that I would expect you to see an "Invalid use of Null" error message letting you know about the problem, unless you have executed an On Error Resume Next statement.

Further, if the ID field is an autonumber, as the logic of your code seems to suggest, then it could have values greater than 32767.  Autonumbers map to the Long data type, so you should declare lngSearch As Long, not intSearch As Integer.  That could be the problem here, I suppose, but again you would get an error, this time an Overflow error, if the value returned from DMax were greater than 32767.

Regardless of those issues, it seems to me that this code is flawed.  First, if you were on a record that is not the last record in the form's recordset, and you use the command button to move to the next record, the code would change the value of TextDate on that record to value you looked up.  But that wouldn't be a new record, so I'm guessing you wouldn't want to do that.

Second, because you are changing the actual value of the textDate field, if you use the command button to go to a new record, but then decide you don't actually want to add a  record, then you have already dirtied the record, and it will be saved unless you explicitly undo it.  Even if you do undo it, you will have used up an autonumber, which you may not want to do.

Better is to set the DefaultValue property of the TextDate control, before you go to a new record, so that the value only takes effect if some field in the record is dirtied.  How you do this depends on your actual intention.  If you really want to use the textDate from the very last record that is in the table, regardless of whether or not that's a record that was entered in this particular user's session with this form, then IF you want to trust the autonumbers to be always ascending, you can use DLookup as above to find that value.

If, on the other hand, you just want to carry the last entered date (in this session with the form) forward as the default value for the next new record, then the easiest thing to do is to use the form's AfterUpdate event to set the DefaultValue property.  For example:

    Private Sub Form_AfterUpdate()

        With Me.TextDate

            If Not IsNull(.Value) Then

                .DefaultValue = """" & .Value & """"

            End If

        End With

    End Sub

https://support.t-mobile.com/docs/DOC-30240

Wipe cache partition
  1. Turn off the device.
  2. Press and hold the Volume Up key and the Home key, then press and hold the Power key.
  3. When the device logo screen displays, release only the Power key.
  4. When the Android logo displays, release all keys ('Installing system update' will show for about 30 - 60 seconds before showing the Android system recovery menu options).
  5. Press the Volume down key several times key to highlight 'wipe cache partition'.
  6. Press the Power key to select.
  7. Press the Volume down key to highlight 'yes' an press the Power key to select.
  8. When the wipe cache partition is complete, 'Reboot system now' is highlighted.
  9. Press the Power key to restart the device.

P.S. Yes wiping the Cache Partition worked.
