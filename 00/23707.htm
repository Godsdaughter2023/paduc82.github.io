Example:

If it is 13 digits long (or has dashes), extract and place in C1

Result: 123-5678901-1

If it is 11 or 12 digits long (only one will exist), extract and place in D1

Result: 123456789011

rick rothstein, mvp - excel say:

Here is my try at it...

Sub ParseCodes()
  Dim x As Long, Cell As Range
  Dim PartText As String, Col As String, Parts() As String
  For Each Cell In Range("B1:B" & Cells(Rows.Count, "B").End(xlUp).Row)
    Parts = Split(Cell.Value)
    For x = 0 To UBound(Parts)
      Col = ""
      PartText = Parts(x)
      If IsDate(PartText) Then
        Col = "E"
      ElseIf Len(PartText) = 13 And Not PartText Like "*[!0-9-]*" Then
        Col = "C"
      ElseIf (Len(PartText) = 11 Or Len(PartText) = 12) _
              And Not PartText Like "*[!0-9]*" Then
        Col = "D"
      ElseIf PartText Like "####" Then
        Col = "F"
      ElseIf PartText Like "#" Then
        Col = "G"
      End If
      If Len(Col) Then Cells(Cell.Row, Col).Value = PartText
    Next
  Next
End Sub

NOTE: Make sure you format Column D as Text so the 13-digit numbers you place in it remain as numbers and do not get changed power-of-ten notation.

a. user say:

I can't answer your questions definitively, but some observations:

PLACEHOLDER fields still function in Mac Word 2016 as far as I can see, but 

 a. they have never been available on Windows Word and

 b. the Ecma and ISO Open Office XML standards have never mentioned this field type, so unless Microsoft decide to include it in the standard, it's unlikely ever to be available cross-platform.

> How are Word Templates created (as they don’t use developer tool or tables, yet work well).

I have only looked at a small number (for example "Paper with cover and TOC"), but  for text, they mostly seem to use Content Controls, with the property "Remove content control when contents are edited" set, and for images, they mostly seem to use an inserted image. 

Since you cannot create content controls using the Word user interface in the current version of Mac Word 2016, either these templates have been created in the Windows version of Word, or perhaps by relying on a stock of content controls that have been created on Windows Word and copied to Mac, or perhaps using code on the Mac that can generate the necessary content controls, either outside Word or using the new Javascript APIs. Looking at theXML of the "Paper with cover and TOC" template, it has clearly been created by a particular group within Microsoft that uses SharePoint, as there are a number of SharePoint related structures in them. [I also notice that they seem to have set up the templates with a particular Bibliographic Style that does not exist on the Mac version).

This kind of text content control also worked on Mac Word 2011 (at least eventually), but the Picture Placeholder content controls never have, which may be one reason why they are not used in Mac templates. This content control type does function (as far as I can see, in a similar way to the Windows version) on Mac Word 2016, but in my view they are rather awkward to use. Perhaps Microsoft people believe that the right-click option to "Change Picture..." is simpler.

> Is there a replacement to the placeholder function that does not involve using the developer tool or inserting pictures within tables?

As far as the text side is concerned, since it appears you have access to Windows Word, using that to develop your Mac templates may be the easiest approach.

As far as the image side is concerned, the main thing I see when using "Change Picture..." is that Word fits the picture in to the existing space, maintaining the *inserted* picture's aspect ratio and adjusting its size to fit as best it can. If what you need is to fit the picture into the existing shape, resizing *and cropping*, then I don't think either that approach or the Picture Placeholder content control approach will work, even assuming "right-click, Change Picture..." is an easy-to-use option for your users.
