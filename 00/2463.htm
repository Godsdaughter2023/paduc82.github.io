jeeped say:
I think I was able to get pretty close to your expectations with a VBA macro. I have run it on your Sheet1 and examination of the first dozen rows of the newly created Sheet3 seems to be correct.

Tap Alt+F11 and when the VBE opens, immediately use the pull-down menus to Insert, Module (Alt+I, M). Paste the following into the new pane titled something like QBA Raw split scratch - Module1 (Code), 

Sub SplitIt()
    Dim c As Long, lr As Long, nr As Long, rw As Long, v As Long
    Dim vy As Variant, vx As Variant, ws1 As Worksheet, ws2 As Worksheet
    Set ws1 = Sheets("Sheet1")
    Sheets.Add after:=Sheets(Sheets.Count)
    Set ws2 = Sheets(Sheets.Count)
    With ws1
        .Cells(1, 1).Resize(1, .Cells(1, Columns.Count).End(xlToLeft).Column).Copy _
          Destination:=ws2.Cells(1, 1)
        lr = .Cells(Rows.Count, 25).End(xlUp).Row  'Y is column 25
        For rw = 2 To lr
            vy = Split(.Cells(rw, 25), Chr(10))
            For v = LBound(vy) To UBound(vy)
                nr = ws2.Cells(Rows.Count, 25).End(xlUp).Offset(1, 0).Row
                ws2.Cells(nr, 1).Resize(1, 24) = .Cells(rw, 1).Resize(1, 24).Value
                ws2.Cells(nr, 25) = vy(v)
                For c = .Columns("Z").Column To .Columns("AI").Column
                    If CBool(Len(.Cells(rw, c))) Then
                        If UBound(Split(.Cells(rw, c), Chr(10))) >= v Then _
                            ws2.Cells(nr, c) = Split(.Cells(rw, c), Chr(10))(v)
                    End If
                Next c
                ws2.Cells(nr, 36).Resize(1, 16) = .Cells(rw, 1).Resize(1, 16).Value
            Next v
            'If rw > 3 Then Exit For
        Next rw
    End With
    Set ws2 = Nothing
    Set ws1 = Nothing
End Sub

Tap Alt+Q to return to your worksheet. Tap Alt+F8 to open the Macros dialog and Run the macro. The macro will not alter your data in any way. Instead it creates a new worksheet at the end of all of the worksheets and puts the split data there. You can delete these extra worksheets if they do not meet your expectations.

I've run the macro and produced 11,087 rows of split data. As mentioned, I've examined the first dozen rows and everything seems to be collecting as you have described. I also reset the row height on Sheet1 to 40px to make it easier to work with.

 

The workbook I have run the macro on with a completed Sheet3 is on my SkyDrive here.

QBA Raw split scratch (Jeeped).xlsb

I turned it into a binary worksheet to save upload/download time (smaller file size). The split data is on the new Sheet3. The macro will not be retained on the uploaded workbook as SkyDrive doesn't allow that, so you will have to put it in (instructions above) if you want to rerun it.

ken sheridan say: With regard to Scott's point concerning OrderDetails and Disposals, in my model Acquisitions and Disposals are all non-sale transactions, so Disposals covers things like write-offs due to damage, promotional gifts etc, i.e. any disposal of stock which does not relate to a sales order.  The Acquisitions table relates to movements into stock so is really something of an oversimplification.  In reality one would be more likely to model this as purchase orders with tables like this:

Suppliers>---PurchaseOrders----PurchaseOrderDetails>----Products

mirroring a sales orders model:

Customers>---SalesOrders----SalesOrderDetails>----Products

There would also be a need to cater for returns, back orders etc.  In my demo orders are really sales orders, but I make no provision for returns or back orders.  As I said earlier, it only covers the basic methodology and, like all my demos, is intended as no more than a starting point from which others can adapt it to their own specific requirements.

In computing the current inventory this is done firstly by retuning all stock movements since the last stock take (if any) by the following query:

SELECT "Acquisition" AS MovementType, ProductID, Quantity
FROM StockAcquisitions
WHERE AcquisitionDate >=
    NZ((SELECT MAX(StockTakeDate)
     FROM StockTakes
     WHERE StockTakes.ProductID = StockAcquisitions.ProductID))
UNION ALL
SELECT "Disposal", ProductID, Quantity*-1
FROM StockDisposals
WHERE DisposalDate >=
    NZ((SELECT MAX(StockTakeDate)
     FROM StockTakes
     WHERE StockTakes.ProductID = StockDisposals.ProductID))
UNION ALL
SELECT "Stocktake", ProductID, Quantity
FROM StockTakes AS ST1
WHERE StockTakeDate=
    (SELECT MAX(StockTakeDate)
      FROM StockTakes AS ST2
      WHERE ST2.ProductID = ST1.ProductID);
UNION ALL
SELECT "Sale", ProductID, Quantity*-1
FROM Orders INNER JOIN OrderDetails
ON Orders.OrderNumber = OrderDetails.OrderNumber
WHERE OrderDate >=
    NZ((SELECT MAX(StockTakeDate)
     FROM StockTakes
     WHERE StockTakes.ProductID = OrderDetails.ProductID));

In this all disposals, whether sales or not, are returned as negative values, and by summing these in the following query based on the above the current stock in hand figures per product are returned:

SELECT ProductID, SUM(Quantity) AS StockInHand
FROM qryStockMovements
GROUP BY ProductID;

Treating all stock movements as instances of a single Transactions entity type as Scott does certainly makes things much simpler.  The additional complexity of my model really stems from its inclusion of a StockTakes entity type, an attribute of which is the actual quantity of a product in stock at the time.  I made the decision to do it this way on the basis of my own experience many years ago in the Purchase and Supply department of a large manufacturing company.  To include stock takes in a single Transactions table would require not the actual quantity to be stored, but the adjustment of this in relation to the currently computed stock in hand, i.e. the stock take quantity less the currently computed stock in hand.  This would give a positive or negative value depending on the direction of the adjustment required, or zero if no adjustment is required.  Whether you adopt an approach similar to mine, or a simpler one similar to Scott's the figures at the end of the day should be the same.  If I were to adopt the simpler approach, though, I would model sales orders and purchase orders as sub-types of transactions by means of tables in one-to-one relationship types with the latter.  To my mind, as each has its own distinct attributes, they are entity types themselves rather than attributes of a transactions entity type.

scottgem say:

I've incorporated the concept of a Stock Take in some of my apps. Clearly the more transactions you have, the longer its going to take to calculate QOH. Allen Browne explains this in his article on Inventory. The way I have handled it is to add an Verified Inventory amount with an effective date. Generally, companies do a physical inventory of all stock a specific times, so the effective date of the physical inventory is the same for all products. I then adjust the QOH queries to start with that number and then include transactions after the effective date,
