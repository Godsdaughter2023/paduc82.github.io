forbiddensas say:

A common issue with WP7 and corporate exchange servers is WP7 doesn't support the full range of EAS (Exchange Active Sync) policies.

If your corp Exchange server has the "allow non-provisionable devices" option unchecked (not enabled) - WP7 devices will not be able to sync if the typical policy settings (i.e device encryption is enabled). See this link : http://social.technet.microsoft.com/wiki/contents/articles/exchange-activesync-considerations-when-using-windows-phone-7-clients.aspx

Your exchange admin can create or exclude you from the default policy and then your phone will (or should) work.

FYI - the upcoming WP8 upgrade supports device encryption.

 

PS - the reason I am suggesting this is if other devices are sync'ing and yours isn't then I would make a large bet this is the culprit.  You can further verify this by checking to see if the Android phones are using a 3rd party email client (the popular one being Touchdown), since a lot of Android phones (vendor specific versions) don't support EAS Policies - in outher words without the 3rd party app their phones wouldn't be able to sync either.

ken sheridan say: It sounds like you've allowed Access to automatically generate a report from the query object rather than designing the report yourself.  While this is a cheap and cheerful way of creating a report its is a very blunt instrument and rarely produces a satisfactory result.  If the number of columns exceeds the page width the extra columns will generally appear on the next page, which is only useful if you are going to print it out and then physically paste the pages together.

For most purposes, however, you'd want everything to fit the width of a physical page, so you need to ensure that the report design is such that the width of the design area plus the margins is within the paper size being used.  Then you need to ensure that all the controls fit within the width of the design area. Changing the page format to landscape might help, and making the header labels narrower and deeper so that their values wrap to two or more lines might be possible.

Sometimes, to achieve a horizontal layout, you can avoid using a crosstab query completely by using a standard query and a multi-column subreport within the parent report.  You'll find an example as ClubMembership.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This file has reports which list members of each household horizontally, and includes two reports which give identical results, one using a subreport, the other which manipulates the report's layout in code at runtime.  Don't pay too much regard to the latter; it's only there because the file was originally produced for a magazine column by a contact of mine which required a code solution in response to a reader's request.

In the same SkyDrive folder is another file Payments.Zip, which gives a result similar to a crosstab query.  In this both the column headings and the data at the intersections are generated by subreports, so the problem of handling varying column headings in 'dynamic' subreports is avoided.
