Is this normal, or do we need to do something differently to acheive the desired results?

nemanja pantelic msft say:

Hi Bill, 

I have tested it on my side, and I have the same results. Analyzing the message header, I couldn't find the alias in case of messages sent from the organization. This is expected behavior in Office 365. 

I'd like to suggest you a workaround solution. Instead of aliases, create a shared mailboxes. This way you will have free address instead of alias. Now you can forward the messages recieved from shared mailboxes to the oreginal mailbox where you recieved emails sent to aliases. You can set a rule in Exchange admin center - https://portal.office.com > Sign in with admin account> Admin> Admin centers> Exchange> Mail flow> Rules> New rule> Apply if... - the reciepient is> Do the following... - Bcc the message to. Now you can set the inbox rules from the first reply.

I hope you can use this suggestion. If you have aditional question, feel free to write.

The suggestion to rename %appdata%\skype folder is totally useless. This folder has nothing to do with the new Skype, and is only used by the older Skype 7.X version.
