Although you may well have a problem with your virtual COM port, that shouldn't have anything to do with your printer.  Virtual COM ports are used to connect a device that expects to use a serial port.

Your printer uses a normal USB connection (which is why "it works fine using USB cable").  Note that almost all printers will not connect via wi-fi while also connected via USB.

lemp say:

The "port" indicated in printer properties means that print jobs directed to this printer will go to the USB port and thence to the printer.  If you disconnect the USB cable and thus enable the printer's wi-fi adapter, Windows will still send the print job to the now-disconnected USB port and nothing will print.

You need to create a second instance of your printer.  You can either retain the current instance if you sometimes want to use the USB connection or you can delete it.  If you don't delete the first instance before you create the new one, Windows will append something like (copy) to the name.  You can right-click on the icon and "Rename" to give it a more useful name, e.g., Lexmark 4690 USB or Lexmark 4690 wifi.

The following should get you connected to your printer by wifi:

1.  Disconnect the USB cable from the printer.  The printer's wi-fi status light should be green.

2.  Go to Devices and Printers and click Add a Printer.

3. Click "Add a network, wireless, or Bluetooth printer"

4.  Wait a few seconds and Windows should detect your Lexmark 4690.  Select it and click Next

5.  On the screen that says "Which version of the driver do you want to use" follow the recommendation to use the existing driver.

6.  Complete the wizard and print a test page

If you have problems, please describe at which step you were unable to follow the directions and what happened.  If you received an error message, please provide the complete text.

I came up with a workaround that sort of accomplishes the original intent of the functionality. The workaround is to, from the Task Schedule task, call an intermediate .bat script which, in turn, calls the target task once the .bat script determines that the system is connected to the desired network.

REM Determine if connected to the network
NETSH WLAN SHOW INTERFACE | FINDSTR /r SSID | FINDSTR /r NetworkName
IF %ERRORLEVEL% == 0 (
    SET "ConnectedToNetwork=True"
) ELSE (
    SET "ConnectedToNetwork=False"
)

REM Run tasks that require a connection to the server
IF %ConnectedToNetwork% == True (
    [Run original task requiring network connection]
)

One advantage to this technique is that you can use it to check for multiple network names without having to create multiple Task Scheduler tasks. For example, if connecting to a wireless network and there are two or more network names that all start with "MyNetwork" such as "MyNetwork" and "MyNetwork 5G", then searching for "MyNetwork" will return an "ERRORLEVEL" of 0 in both cases.

The disadvantage is that we can't tell from the Task Scheduler logs when the target task was run and not run based on the status of the environment.

All in all, Microsoft's testing department needs an interjection of some best practices such as incorporating better and more complete automated regression testing to ensure nothing gets broken when changes are made to the system.
