bob jones aka: cybertaz say:

In addition to the responses from others;

You cannot 'Opt not to use styles" because Word is a style-based program. It's just a matter of which styles you apply, what attributes they include, & how you apply them :-)

The Heading style is being applied because you're pressing return twice after typing one line of text without end punctuation. That tells Word that what you've typed is a Heading to which the style should be applied & triggers the AutoFormat as You Type feature Jacques referred to. Disabling the feature is one option, but his instructions are for Win Word, not Mac Word. In Mac Word go to Word> Preferences> AutoCorrect - AutoFormat as You Type.

Many users do that in an effort to "create space" between paragraphs of text, but what it does is create empty paragraphs separating them. Doing so typically causes a number of other issues which precipitate the use of even more improvised methods of addressing them. The result can become a rats nest of band-aids which make controlling the text flow much more problematic & eventually can corrupt a document.

The preferred solution is to not use empty paragraphs to force text flow in the first place. Instead, modify the styles to adjust the Space Before/After parameters to precisely set the amount of of spacing between paragraphs. Line Spacing may be another option which can be specified in a style as well. In that way, you need press return only once in order to get the preferred spacing every time. The Space Before/After also can be specified manually, but it then has to be set for each instance.

Please understand that I'm not being judgemental... just offering some friendly advice based on 30 years of experience with the program [and others like it]: Rather than trying to avoid the use of styles it's in the user's best interest to learn to use them effectively.

Also, the square symbol is displaying because the nonprinting characters [¶] are turned on in the document. That symbol indicates that at least one of the parameters of the Paragraph Formatting for Line & Page Breaks has been modified for the paragraph.

As for the Page Break, it's unclear what may be causing it. One possibility is that you're holding the Shift key while pressing return, which is the keyboard shortcut for inserting a Manual Page Break. One other possibility is that the Heading Style has been modified to include the Page Break Before attribute, but there just isn't enough descriptive information in your message to enable making a determination.

Adding a calculated field to a report in Access - Microsoft Community

donalo'callaghan say:

Adding a calculated field to a report in Access

I tried to do this two different ways but I cannot get it to work.

e.g. by dragging a field from the Field list pane and editing the Control source as an expression. I remembered to change the field name but this is not the issue as it comes in with a name like text204. My expression in the formula refers to 2 fields in tables, and one of the fields is the same field that I dragged in:

= [UncertaintyParams]![uTemperature] / [DUCTS]![TEMP] *100

When I drag in each of the fields on its own it is displayed in the report, but when I modify ione of them or insert a textbox and build the expression in it, the program asks "Enter a parameter vale" in respect of UncertaintyParams. This table has only one row, so I do not understand the ambiguity. I am baffled as to what to try.

ken sheridan say: In a computed control you do not need to reference the tables, only the columns:

= [uTemperature] / [TEMP] *100

If you are returning more than one column of the same name in a query, i.e. each from a different table, then be sure to alias at least one of them in the query definition so as to avoid ambiguity.

The alternative would be to do the computation in the query, in which case you can reference the table names, though it's not necessary to do so unless the column names are duplicated in the different tables.  So in query design view the expression in the 'field' row of a blank column would be like this:

TempPercentage:[UncertaintyParams].[uTemperature] / [DUCTS].[TEMP] *100

Note that the dot operator is used, not an exclamation mark, when qualifying a column with a table name in a query.

You can then bind a control in the report to the computed TempPercentage column (or whatever you like to name it).
