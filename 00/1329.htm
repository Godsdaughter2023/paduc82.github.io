yaqub k say:

Hi Hill,

 

InPrivate Browsing enables you to surf the web without leaving a trail in Internet Explorer. This helps prevent anyone else who might be using your computer from seeing where you visited and what you looked at on the web. You can start InPrivate Browsing from the new tab page or the Safety button.

 

When you start InPrivate Browsing, Internet Explorer opens a new browser window. The protection that InPrivate Browsing provides is only in effect during the time that you use that window. You can open as many tabs as you want in that window, and they will all be protected by InPrivate Browsing. However, if you open another browser window, that window will not be protected by InPrivate Browsing. To end your InPrivate Browsing session, close the browser window.

 

While you are surfing using InPrivate Browsing, Internet Explorer stores some information—such as cookies and temporary Internet files—so that the webpages you visit will work correctly. However, at the end of your InPrivate Browsing session, this information is discarded.

Access searching multiple columns to display results - Microsoft Community

rob ingles say:

Access searching multiple columns to display results

Hi!

I am doing a study of the psychoactive of various cannabis strains. I have compiled a list of strains listing what are believed to by the primary, secondary and tertiary effects etc. As you can see from my table, an effect such a "Happy" can appear in any column (but never twice in the same row).  I want to set up a search that will result in all of the strains that share that effect no matter where they appear in the primary, secondary, tertiary hierarchy. I only know how to set up a search for results from a single column so I am hoping someone can suggest a way to search across columns.

Thanks!

 

scottgem say:
Your problem is incorrect design. You need to study up on Normalization. these are a set of rules that govern the design or relational databases.

You should have at least five tables:

tblStrains
StrainID (Primary Key AutoNumber)
StrainDescription
StrainCategoryID (Foreign Key)

tluStrainCategory
StrainCategoryID (PK autoNumber)
StrainCategory

tluEffects
EffectID
Effect

tluRanking
RankingID (PK Autonumber)
Ranking (Primary, Secondary, etc.)

tblStrainEffects
StrainEffectID (PK autonumber)
StrainID (FK)
RankingID (FK)
EffectID (FK)

Instead of making each effect a field in a table (which violates the first law of Relational data) each effect should be a record in a table. With this structure you would only need search for the EffectID in tblStrainEffects to find all strains with a specific effect.

tom van stiphout say:

Scott is exactly right. A relational database is not a spreadsheet. The sooner you follow the rules of normalization, the sooner you take advantage of them.

One case in point: what if we wanted to know where the Happy effect occurs? That would require a complex UNION query in your design, and a simple select query in Scott's.

Sorry, I thought you were dealing with text that was already present in the document. If you want to prevent the automatic capitalization as you type, there are two equivalent ways to turn that off:
  • The next time any word is automatically capitalized, move the mouse over the capital letter. When you see a little blue rectangle under the letter, click it. In the menu that appears, click "Stop auto-capitalizing first letter of sentences".
