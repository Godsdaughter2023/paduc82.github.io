rajesh govind say:

 

Hi Ross,

 

I would like to ask few questions regarding this issue:

 

1)       How did you perform a System Recovery? Was it done using the recovery partition created by the computer manufacturer?

2)       Did you turn “ON” the File History on the computer after performing a System Recovery?

3)       Do you receive any error message regarding the File History?

 

By default, the File History feature on Windows 8 is turned “OFF”. So, when you perform a System Recovery using a recovery partition created by the computer manufacturer, it resets all the settings on the computer to the factory default settings. This means that if you had the file history turned “ON” before, it will be turned “OFF” after the system recovery. This might be the reason why you are unable to find any backed up files from the time of System Recovery till now.

 

If the file that you are looking for, existed on the computer even before you performed the System Recovery, then it might also be possible that the File History was set up in such a way that it deleted the older version of the file that were backed up from the backup drive. There are settings in the File History which decides how long the File History should keep the older versions of the files in the backup drive. To check this, you may refer to the section: “Change how long File History keeps files” from the given article:

 

What if something goes wrong in File History?

 

You may also try creating a local copy of the previously backed up data from the backup drive and then re-include it in the File History and check if the file that you are looking for is shown up there.

 

You may also refer to these articles:

 

How to use File History

 

Protecting user files with File History

 

A New Way to Backup: File History in Windows 8

 

Hope this information is helpful.

dbeachy1 say:

I had the same problem a few weeks ago.  After digging around for several hours the problem was that Windows Backup (C:\Windows\System32\sdclt.exe) was being relaunched and immediately failing due to a DCOM permission error 67 times per second, maxing out the CPU on two or three cores.  My event log was full of errors like this:

 

--------

 The machine-default permission settings do not grant Local Activation permission for the COM Server application with CLSID 

  {C2F03A33-21F5-47FA-B4BB-156362A2F239}

   and APPID 

  {316CDED5-E4AE-4B15-9113-7055D84DCC97}

   to the user NT AUTHORITY\LOCAL SERVICE SID (S-1-5-19) from address LocalHost (Using LRPC) running in the application container Unavailable SID (Unavailable). This security permission can be modified using the Component Services administrative tool.

--------

 

If you look in Task Manager you will see the PID for sdclt.exe changing rapidly.  I don’t use Windows Backup anyway, so what I did to work around the problem until Microsoft fixes this rather epic bug is:

 

1. Reboot in Safe Mode so sdclt.exe won't be launched.

2. Take ownership of C:\Windows\System32\sdclt.exe and grant full control to the local administrators group.

3. Rename C:\Windows\System32\sdclt.exe C:\Windows\System32\sdclt.exe.no

4. Reboot.

5. Have a beer; you've earned it.

 

What makes this bug so nasty is that a few minutes after login it pegs 2-3 cores at 100% indefinitely without showing up in Task Manager, so it is very hard to track down: the offending process (sdclt.exe) launches and immediately exits only to be relaunched again immediately in an endless loop.   Renaming sdclt.exe is a hack, though, so if you need Windows Backup to work you could instead dig into granting the DCOM permission it is complaining about.

You can acheive what you want using i.e. a FOR /L command from the command prompt:

  • [Win] + X
  • Command Prompt
  • CD /D Drive:\DirectoryWhereYourFilesAreStored
  • Type FOR /L %G In (1,1,100) Do @If Exist TEXT.File%G Ren TEXT.File%G File%G.txt then press [Enter]
