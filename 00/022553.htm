Programmatically import .txt into Word - Microsoft Community

judyihmiller say:

Programmatically import .txt into Word

Hi Everyone. 

I am not very savvy with Office, or even searching the community for info so I am asking for help! 

I am using a tool that creates a pipe delimited .txt file with Column Headers.  I want to import that Data into a pre-formatted Table(s) in a Word document, programmatically.  This way, each User can create reports with a "common look and feel." 

Is this possible? 

If so, how? 

Happy 2018!

Thank you very much.

a. user say:

I know this does not answer your question as stated, but there are two things you can try that may turn out to be simpler than inserting data into a preformatted table. 

1. Use a DATABASE field and one of its standard table formats.

2. Use Word VBA's InsertDatabase method to insert the database, then format it using code (rather than relying on the document to specify the table format details).

In their simplest form, these will only work because pipe is one of the characters that Word recognises as a field delimiter when it tries to insert a database from a .txt file

The following instructions are for Windows Word 2016. They won't work if your table needs to be nested inside another table. Also, if your .txt file has more columns than a Word table can have (63 or whatever it is) you will not actually get a table this way.

To try (1), click at the point where you want to insert the table, then

 a. use ctrl-F9 to insert a pair of the special field code characters {  } (you cannot just type them on the keyboard)

 b. type the following between the {  }

DATABASE \d "the full path name of your .txt file with backslashes doubled up" \h

e.g. if the .txt file is at d:\mydata\mypipefile.txt you would have

{ DATABASE \d "d:\\mydata\\mypipefile.txt" \h }

Then select the field, press F9 to update it, and Alt-F9 to display the results. The \h option specifies that you want the column headings from the file at the top of the table.

You can then add some standard formatting using the "\l" and \b options - use alt-F9 to show the field code again, and change it to

{ DATABASE \d "d:\\mydata\\mypipefile.txt" \l "8" \b "511" \h }

then re-execute the field. You will see that a particular table style has been applied by the \l option, but the columns are now "autofit" instead of spread across the page.

The \b option controls which parts of the predefined style are applied. So for example, \b 16 applies autofit and nothing else (with one exception). \b 495 applies everything except autofit. 

The DATABASE field has more options, but if you always want to display the entire table, that is really all you need. You can find out more at https://support.office.com/en-us/article/Field-codes-Database-field-04398159-a2c9-463f-bb59-558a87badcbc?ui=en-US&rs=en-US&ad=US

Many examples "out there" suggest that you have to have a "connection string" (a \c option" and a SQL query (the \s option) in a DATABASE field, but in this case you do not need either.

If you need to select particular columns, filter the records, and/or sort, you can add an SQL SELECT statement> In this case, Word gets the data through a "text converter", and lets you specify queries in its own very basic SQL language. The most basic query looks like this:

SELECT * FROM it 

You put that in the DATABASE field using an \s option like this:

{ DATABASE \d "d:\\mydata\\mypipefile.txt" \s "SELECT * FROM it" \l "8" \b "511" \h }

Again, examples "out there" suggest that you need the full path name of the .txt file in the SELECT statement, like this:

{ DATABASE \d "d:\\mydata\\mypipefile.txt" \s "SELECT * FROM it" \l "8" \b "511" \h }

but in this case (where Word gets data via its text converters), although you cannot omit the FROM, pretty much anything after the will do. "it" seems a reasonable choice to me. In fact, if you have other SQL keywords after the FROM, you can omit the it, e.g.

SELECT * FROM WHERE k  > 1

but it does look a bit weird and I think it was probably not the intention of the Microsoft coders to allow that.

In Word's dialect of SQL, you can

 a. specify a list of columns, e.g. SELECT ID, firstname, lastname FROM it
 b. specify simple filters, e.g. SELECT * FROM it WHERE ID > 5 AND lastname > 'Jones'
 c. specify some sort columns, e.g. SELECT * FROM it ORDER BY lastname, ID DESC
 d. all 3, e.g. SELECT ID< firstname, lastname FROM it WHERE ID  5 AND lastname > 'Jones' ORDER BY lastname, ID

(DESC means sort from the largest to the smallest)

A big advantage of a DATABASE field is that it is already in the document, in simple cases you do not need any VBA, and you do not need to have anything else in the document (such as placeholders, bookmarks, etc. to tell your code work out where to insert the data. However,

 a. the users do have to know how to update DATABASE fields

 b. you must be able to put the data in the place where DATABASE will find it

 c. you are constrained by the formatting options available

2. Word's VBA InsertDatabase method does much the same thing as a DATABASE field, but it can either insert the data on its own, or a DATABASE field.

Let's take our first example DATABASE field:

{ DATABASE \d "d:\\mydata\\mypipefile.txt" \h }

e.g. the following code would do the same as that, inserting a database field and the data at the current selection 

Sub insertmydatabase1()
Selection.Range.InsertDatabase _
  LinkToSource:=True, _
  DataSource:="d:\Test\2018010201 pipe delimiter db\pipe1.txt"
End Sub

If you just want the table with no field, omit the LinkToSource parameter or set its value to False, e.g.

Sub insertmydatabase1()
Selection.Range.InsertDatabase _
  LinkToSource:=False, _
  DataSource:="d:\\mydata\\mypipefile.txt"
End Sub

You can also use...
  - the Format parameter to do the same thing as the \l parameter
  - the Style parameter to do the same thing as the \b switch
  - the SQLStatement (and SQLStatement1) parameters to specify an SQL Statement
  - the Includefields parameter to omit the header row
  - From and To parameters for start and record numbers (these correspond to \f and \t options in the DATABASE field that I have not described above)

So to do the same as our final DATABASE field example, which was

{ DATABASE \d "d:\\mydata\\mypipefile.txt" \s "SELECT * FROM it" \l "8" \b "511" \h }

you might use

Sub insertmydatabase3()
Selection.Range.InsertDatabase _
  Format:=8, _
  Style:=511, _
  LinkToSource:=True, _
  SQLStatement:="SELECT * FROM it", _
  DataSource:="d:\mydata\mypipefile.txt", _
  Includefields:=True
End Sub

To use that, you would have to have some way to identify where you needed to put each table - possibly a bookmark, or a rich text content control. Once you have inserted the table, you can use any VBA to format it how you like.

hansv mvp say:

Here is the code from the site I posted a link to:

 

Function ConcatenateIf(CriteriaRange As Range, Condition As Variant, _
        ConcatenateRange As Range, Optional Separator As String = ",") As Variant
    Dim i As Long
    Dim strResult As String
    On Error GoTo ErrHandler
    If CriteriaRange.Count <> ConcatenateRange.Count Then
        ConcatenateIf = CVErr(xlErrRef)
        Exit Function
    End If
    For i = 1 To CriteriaRange.Count
        If CriteriaRange.Cells(i).Value = Condition Then
            strResult = strResult & Separator & ConcatenateRange.Cells(i).Value
        End If
    Next i
    If strResult <> "" Then
        strResult = Mid(strResult, Len(Separator) + 1)
    End If
    ConcatenateIf = strResult
    Exit Function
ErrHandler:
    ConcatenateIf = CVErr(xlErrValue)
End Function

You almost got it right, you only forgot to change ConcatenateIf to Concatenate2Ifs in the body of the function:

Function Concatenate2Ifs(CriteriaRange As Range, Condition As Variant, _
        CriteriaRange2 As Range, Condition2 As Variant, _
        ConcatenateRange As Range, Optional Separator As String = ",") As Variant
    Dim i As Long
    Dim strResult As String
    On Error GoTo ErrHandler
    If CriteriaRange.Count <> ConcatenateRange.Count Or _
            CriteriaRange2.Count <> ConcatenateRange.Count Then
        Concatenate2Ifs = CVErr(xlErrRef)
        Exit Function
    End If
    For i = 1 To CriteriaRange.Count
        If CriteriaRange.Cells(i).Value = Condition And _
                CriteriaRange2.Cells(i).Value = Condition2 Then
            strResult = strResult & Separator & ConcatenateRange.Cells(i).Value
        End If
    Next i
    If strResult <> "" Then
        strResult = Mid(strResult, Len(Separator) + 1)
    End If
    Concatenate2Ifs = strResult
    Exit Function
ErrHandler:
    Concatenate2Ifs = CVErr(xlErrValue)
End Function
