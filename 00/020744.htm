a. user say:

FWIW I went through a similar set of tests and was able to replicate much the same things as you did. The main thing I would add is that after you try to "Change Source..." only the full name (path/file name) in the bottom part of the Links dialog changes. The name in the list above does not change - that's a little harder to spot sometimes because of the way that the names are shortened.

So what we see is that Word contacts Excel, Word gets a new picture from Excel corresponding to the new source that you specified, and inserts it, displays the new details in the bottom half of the dialog, but does not actually change the link itself. So next time you view the Links, the old details are still there.

So my assumption is that your original problem (which AFAICS is all about *Links* rather than *Embedded objects* has not been solved.

I think that your observation that the existing object cannot be found hits the nail on the head. When I tried to open a Word file with a linked Excel sheet in a folder that did not exist on my system, I saw exactly the same problems as you did. When I created the folder and put a suitable Excel file in it, when I updated the link, MacOS's "Grant Access" dialog appeared, and everything worked OK. So it seems to me that when you try to "Change Source...", Word is trying to access the absent file, failing, and as a consequence is not completing the change. Since this used to work in previous versions I wonder whether the code is failing in a new way because MacOS wants to grant access. However, we don't see a dialog for a file that isn't there, so maybe that is not it.

But there is another way in which this seems to me to be failing. The trouble is that the thing that Word is actually trying to open is not necessarily a file. If you created you file in an earlier version of Word (haven't checked Word 2016 yet) If you selected a range or a range name in Excel before Copy/Paste Linking, what Word is actually trying to open is an "OLE Moniker" that specifies the file name and the range info. So for example if your workbook is in /Users/me/sheets/mywb.xlsx and you are inserting range R2C2:R3C3 from Sheet mysheet, Word *used to* make a link to a file called "/Users/me/sheets/mywb.xlsx!mysheet!R2C2:R3C3" (on Mac that appears to be modified to  "/Users/me/sheets/mywb.xlsx!mysheet!R2C2/R3C3" , probably because of problems with invalid characters in pathnames)

When you look at the Links dialog in that case, you see that the "Source File" is "/Users/me/sheets/mywb.xlsx!mysheet!R2C2/R3C3" and the "Item" is blank.

But if you now close that file, re-open it, let Word prompt you about the automatic links, and update the link, *then* look at the Links dialog, you will see that Word now shows the Source File as "/Users/me/sheets/mywb.xlsx" and the "Item" as "mysheet!R2C2/R3C3"

I think the other problem you may be seeing is that in addition to the fact that Word cannot open files that are not there, on Mac it also cannot open files that are actually "OLE Monikers", presumably because MacOS sandbox intercepts the open request and would try to get you to grant access to the file, but it isn't a file.

Unfortunately, even in VBA it is impossible to fix this (except perhaps by removing the link inserting a new one) because although VBA can access and modify the full path name and file name via a property called LinkFormat.SourceFullName, it cannot modify the property that is supposed to hold the "Item" name (OLEFormat.Label). Further, trying to modify SourceFullName when the existing SourceFullName does not exist or is not really a file name does not work. So I think this one is entirely up to Microsoft to sort out.

Phew!

julie fka profjulie say:

You can each use separate Windows Live ID accounts.

In Windows Live on your computer (open up the browser and log into your windows live id account).  You can share your calendar with your wife (and she can share her calendar with you) as co-owners.  Then her calendar will show up on your Windows Live account and your calendar will show up under her Windows Live account.  Then you both can edit each other's calendars by logging in to your windows live accounts on any computer.  You can use the Outlook Connector to sync the calendars to Outlook on your computer.

You won't be able to sync both calendars to a windows mobile phone, but if you upgrade to windows phone 7, then you'll be able to sync both calendars to your phone.
