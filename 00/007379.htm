kelli.w say:

Access 2007 option buttons

Kelli

ken sheridan say: As Scott says, the root of the problem is an incorrect table structure.  How you correct this depends on what type of values you want to insert into the columns, but as you are using an option group my assumption would be that you want each row to be associated with one and one only of the values ME1, ME2, ME3, ME4, or ME5 (I'm guessing MS5 in your post is a typo).

If so you need to decompose the table into two related tables.  Create a table, Options say (or a more suitable name of your choice)  with two columns such as OptionCode (number - primary key) and OptionValue (text - indexed uniquely) and fill it like this:

1    ME1
2    ME2
3    ME3
4    ME4
5    ME5

In your current table have one column, OptionCode (number - indexed to allow duplicates) in place of the five.  Create a relationship between this table and your current table on OptionCode, and enforce referential integrity.

Now create your option group with  values from 1 to 5 for each option button, and set its ControlSource property to OptionCode.  When you select an option a value from 1 to 5 will be inserted into the column, and this will reference the primary key of the Options table, so will map to the row with the appropriate OptionValue.

You can show the option value whenever necessary, e.g. in a report, which joins the two tables on OptionCode and returns the OptionValue column in the query.

scottgem say:

Do you really have a subform are are you referring to the datasheet portion of the Split form? The purpose of a Split form is to display data from a table in BOTH a form view and a datasheet view. The datasheet view is used to find and select the record you want and then it will be displayed in the Form view.

If you have a large number of records, don't just scroll through the datasheet. Use Datasheet filtering to find the record you want. In each column header in Datasheet view there is a pull down that will allow you to filter the data based on the data type of the column. You can use these filters cumulatively, one filter builds on the next. For example, lets say you are trying to find an Order record. You know that the order was placed by x company during the previous quarter. So you can pull down the Customer column and select the customer, then pull down the date column and select Last Quarter. This may filter enough to easily find the order, select it and have it displayed.

I have a method I find works well for many of my forms with similar issues.

First - create a form - continuous records.

in a single line - ad the Important fields - the ones you scan or search for.

keep it to just 1 row of fields - controls.

add the form header and footer sections to the form.

put all the other fields into the footer section.

when this is working - play with selection criteria to narrow down your searching.

BTW - this structure can be put into a subform.

The way a SELECT CASE works is to look at an expression and take several possible actions depending on the value of the expression:

SELECT Case Me.cboAuthors

         Language = "Latin"
