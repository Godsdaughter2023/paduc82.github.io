Access Forms Set Focus in a Subform - Microsoft Community

tukatz say:

Access Forms Set Focus in a Subform

I have built an app for recording plant purchases at my Master Gardener plant sale. Each purchaser gets the next sequential ticket number and plants being purchased get sale records related to that ticket number. I have a form with the ticket number on it and it has a subform for recording each plant being purchased. Plants have a label with a bar code that is scanned. Focus then moves to a quantity field. When you tab out of the quantity field the next sale record comes up with focus in the bar code field. When you have entered the quantity of the last plant there is a button you click on the main form for the next client. That button retrieves the next ticket number into the main form, but I have not been able to get it to set focus in the bar code field in the subform. There are no fields in the main form where data can be entered, so when you first open the main form focus is set in the bar code field of the subform, the first editable field. This is the code for the click event on the Next Client button.

Private Sub Command9_Click()
  DoCmd.GoToRecord acDataForm, Me.Name, acNewRec
  Me.[Ticket_Number] = DMax("[Ticket_Number]", "Ticket_Master_T") + 1
  Me!Bar_Code.SetFocus
  If Me.Dirty Then
    Me.Dirty = False
  End If
End Sub

I have tried everything I can think of or find on the web, but the SetFocus isn't working.

ken sheridan say: First move focus to the subform control, i.e. the control in the parent form's Controls collection which houses the subform.  Focus can then be moved to the Bar control in the subform by referencing it via the subform's Form property.

You do not need to explicitly save the new record in the parent form.  The record is saved automatically when focus moves to the subform.  So the code for the button's Click event procedure would be along theses lines:

    DoCmd.GoToRecord acDataForm, Me.Name, acNewRec
    Me.Ticket_Number = Nz(DMax("Ticket_Number", "Ticket_Master_T"),0) + 1
    Me.sfcPlantsOrdered.SetFocus
    Me.sfcPlantsOrdered.Form.Bar.SetFocus

where sfcPlantsOrdered is the name of the subform control in the parent form's Controls collection.  Note the Nz function call.  This won't be necessary in your case as the Ticket_Master_T table is not empty, but it's good programming practice to include it nevertheless, as it caters for the possibility of the database being used afresh without any orders data.

Another point of good programming practice is that meaningless default object names such as Command9 should be immediately replaced with a meaningful name such as cmdNewClient or similar when first added to the form.  This provides degree of self-documentation.  If you change an existing control's name however, the link with its event procedures will be broken, so these will need to be recreated.

"Type A plugs and receptacles from both USB 3.0 and USB 2.0 (or earlier) are designed to interoperate.
