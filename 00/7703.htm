As you see they evaluate without any problem.

I don't think there is much mileage in your expressions to handle a situation where the range spans the end of the calendar year.  You can't assume that the query is only opened in the first of the years.  While trying to define the ranges solely by month and day of month might at first sight seem more simple, in reality the problems to which it gives rise make it a less efficient solution in my view.  I would recommend the you store the ranges as actual dates in a related table like this:

HolidayDesignRanges
....HolidayID  (FK)
....StartDate
....EndDate

This table has two candidate keys, HolidayID/Startdate  and HolidayID/EndDate.  One can be used as the primary key, the other included in a unique index.

It is a simple task, once you have inserted one set of dates for all the holidays over one year to roll these forward for as many years as you wish with a little bit of code, e.g. to roll the dates forward for 5 years:

    Dim strSQL As String
    Dim n As Integer
    
    For n = 1 To 5
        strSQL = "INSERT INTO HolidayDesignRanges(HolidayID,StartDate,Enddate) " & _
            "SELECT HolidayID, DateAdd(""yyyy"",1,MAX(StartDate)), " & _
            "DateAdd(""yyyy"",1,MAX(EndDate)) " & _
            "FROM HolidayDesignRanges " & _
            "GROUP BY HolidayID"
            
        CurrentDb.Execute strSQL, dbFailOnError
    Next n

The above code can be executed as many times as you wish to roll the rows forward by 5 years each time.

It is then a simple matter to return the row from the current table where the current date falls within the range:

SELECT vrsns.*
FROM vrsns INNER JOIN HolidayDesignRanges
ON vrsns.ID = HolidayDesignRanges.HolidayID
WHERE DATE() BETWEEN StartDate AND EndDate;

An alternative solution would be to store just one year's dates in the table, and include the table and an auxiliary Counters table of sequential numbers in a query to return the Cartesian product of the two tables, adding the Counter value to the year in each row to return dates in sequential years.  However, I again think this is only a superficially more economical solution; I'd still go with the table of dates over a sequence of years into the future.  It is always going to be a small table, given that the number of years ahead to which it would be reasonable to project is limited.
