Although unlikely, checking for numbers within text using such methods is potentially subject to error. It is better to perform the check against each character within the string explicitly. For example (and given appropriate regional date settings) the three formulas given above will return incorrect results for the string:

EKISKISNGNJUL1

or

EKISKISNGNMAR10

Although not appropriate to this question, the same applies when checking for numericals within strings such as:

EKISKISNGN2E+5

or

EKISKISNGN2-5

which, unless accounted for, will allow mixed substrings to be interpreted as numerical (scientific notation and date respectively).

Better to use the suggestion of Mike H., or, as I said, to test explicitly, e.g.:

=MID(A2,MATCH(TRUE,INDEX(ISNUMBER(0+MID(A2,ROW($1:$100),1)),,),0),100)

Regards

Function RemAlpha(str As String) As String
'Remove Alphas from a string. . . . .Ron Rosenfeld
Dim re As Object
Set re = CreateObject("vbscript.regexp")
    re.Global = True
    re.Pattern = "\D"
RemAlpha = re.Replace(str, "")
End Function

Function RemDigits(str As String) As String
'Remove numbers from a string . . . . Ron Rosenfield
Dim re As Object
Set re = CreateObject("vbscript.regexp")
    re.Global = True
    re.Pattern = "\d+"
RemDigits = re.Replace(str, "")
End Function

charles kenyon say:
Not easily doable.
You could write macros to give choices, but it would be about the entire field, not just one or two words in the field unless you put in incredible complexity, which probably defeats the purpose of having a form to start with.

The rich text content control allows formatting, but isn't for use in a protected form in a protected section.

If you want to do this, consider using a userform instead.

Rich text allows formatting. Plain text does not.

graham mayor say:
You cannot mix content controls and legacy form fields in the same document. The latter require the document to be protected, the former does not. It is an essential conflict that cannot be resolved without macros and you cannot force users to run the macros in a document.

Documents that require free editing are almost certainly unsuited to the use of form fields. Forms are essentially vehicles for collecting large amounts of similar information, often for automatic processing into (say) a database or worksheet. It is difficult to see the relevance of italicising or emboldening text in such an application.

Your best plan would be to create a template, from which the user will create new documents, driven from a userform to collect and enter the variable data (probably with docvariable fields), with the fixed text already placed (or you can place it with the macro that drives the userform). This will then result in a document that may freely be edited.

There is little to be gained by protecting such a document. The protection will not deter anyone with malicious intent, but by using protection you make the document far less convenient for ordinary users.

So the document is either a form in which case the free editing is probably superfluous, or it is not a form in which case there are better ways of achieving your aims.
