daniel pineault say:

Firstly, Access will allow multiple concurrent database connections, so in plain English, you can have multiple people working simultaneously and still be able to run your reports at the same time.

 

Now if you want to setup this up through SharePoint, then you are talking about using the 'new' web database format.  Using this format you cannot use any VBA, so you need to become familiar with Access Macros to do whatever you need to accomplish.

 

That said, if everyone is in the same building, using a LAN connection, I personally would opt for a standard, non-SharePoint, database.  This then gives you the full power of VBA to automate your database functionalities.

 

How data is displayed, who has access to what, ....  all of this you control as the developer.  These are issues that need to be defined while developing your database specifications (what is it your are trying to develop,  Your inputs and expected output, how they should work, look, ....).  Specifications, are the most crucial aspect of any development process!!!!

 

As for tutorials,  There are tons of great books, and website, just google a bit.  That said, without more detailed information about the type of database, subjects, data,...it is impossible to guide you better.  Perhaps a good starting point would be the 'Newcomer's Reading List' found at http://www.utteraccess.com/forum/Newcomer-s-Reading-List-t1997455.html

 

Perhaps trying to locate an existing database model might help you get going in the right direction.  You could take a look at http://www.databaseanswers.org/data_models/ and see if you subject has already been covered and start from there.  Then post back here with questions as they come up.

albert d. kallal say:

In your case you kind of lucky, since you already have SharePoint.

 

With Access, what you could do is create an Access application (that you install on each computer, or use the free runtime). This application would have VBA, reports and ALL data would be shared among all users (because your tables reside on SharePoint).

 

So in this use case, Access sounds like an ideal choice and possibility. And the beauty of such a setup is you really just have to send your tables up to SharePoint, you don't need SQL server or networking skills. And you don't need really to learn any new skills here. All your maintains, table design changes etc. will occur INSIDE from Access.  So you can use Access for creating this application.

 

And to be CLEAR:
You do NOT need to use Access Web publishing for this setup at all. So we are talking about creating a standard "classic" desktop application here, but the data tables reside on the server side. (your team site).

 

I suppose if you have SharePoint 2010 (enterprise) you could create a few forms for data entry via web browser and build a few Access web reports. So Access 2010 does allow you to create + publish some web forms. This indeed could also work well and you remove Access requiremtns (it all on your sharepoint site via web).

 

However, the basic idea here is to use Access front ends linked to SharePoint lists. I do recommend that you use Access 2010 or later since the performance upgrades to Access are significant in this area for VBA enabled applications.

 

And also SharePoint 2010 is required since it now has relational database features.

Go to Settings/ System/ Display.  Identify which monitors are 1 or 2 and highlight the one you want to be the main monitor.  Go to the bottom of the page to 'Multiple displays'  and tick to 'make this my main display'.

'Extend these displays' effectively gives you one screen across the two monitors. You do not say which screensaver you use but a picture is normally on only the main monitor. If it also partly shows on the second monitor I would suspect that the aspect ratio of the picture is not the same as the screen. Other screensavers, such as the bubbles, will flow from one monitor to the other.
