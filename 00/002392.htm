scottgem say:

12/30/1899 is the Zero date. Access stores dates as a double precision number where the integer portion is the number of days from 12/30/1899. So if that date is showing up, then the filed value is being set to zero. 

Setting the default value to =Date() should work. As soon as you enter data into a new record, the control bound to the date field should display today's date. If you want to use a button to set today's date, the code behind the button would be:

Me.datecontrol = Date()

So, if this isn't working, then either your default value is not correct or you are doing something else incorrectly. Note that Date is a reserved word in Access. So if you are using that for a field name, it might cause some issues with your codeing.

Do you in fact have textboxes or fields named "1 stWednesday" - with a blank in the name? If so that's the problem: blanks are MEANINGFUL, and Access will see this code as referring to the number 1 and to some OTHER thing named stWednesday. I suspect that the code will not even compile correctly - if you select Debug.... Compile do you not get error messages?

I'd avoid putting blanks, punctuation, or any special characters other than the underscore _ in names of fields or objects; and I'd also avoid starting names with a number. If you do so anyway, then you must enclose the name in [square brackets] in your code: 

Me.[1 stWednesday] = Date

will load today's date into the textbox named [1 stWednesday] on the current form (that's what the Me. means).
