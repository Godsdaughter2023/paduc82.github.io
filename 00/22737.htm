karl timmermans [outlook mvp 2012-2018] say:

First - Pollardie is correct in that many of your questions are Access related (specifically anything related to linking/syncing) and should be posted in:

http://answers.microsoft.com/en-us/office/forum/access?page=1&tab=all 

However, to fast track this a little

I'm trying to link Calendar, Tasks, and Contacts from Outlook 2010 to Access 2010. Problem is, one of the CRITICAL fields--Category--won't show up. (I know other important fields likewise don't show up, such as tons of fields in Contacts.) WHY IS THIS?!

Is there a fix?

It's the way the MS Access addins were designed. No fix (that I'm aware of) and not an Outlook issue since it's not the one establishing the connection or exposing fields.

 

I tried import/export from Outlook 2010 to Access 2010, but it only supports Access 97-2003, not Access 2010 databases. (Likewise with Excel, only Excel 97-2003.)

Why would Microsoft do this? What's the fix?

The fix is to save your Access .accdb databse as an .mdb file and Excel workbooks as an .xls file. The import/export process has not changed since O'2000 (or earlier - in answer to "why would Microsoft do this). While many have asked the same question about the current Access/Excel native formats - this is just how it works today

I thought I read about BCM for Outlook 2010 supports a better import/export, but is that only for data in BCM? What if I install BCM, but "just use Outlook"? Would I get the import/export to Access 2010 and Excel 2010, even if the folders are in Outlook?

Yes BCM employs it's own import/export process for .accdb/xlsx file types but only for BCM item types - would not apply to the standard Outlook folders. You still need to emply the standard import/export wizard.

 

Finally, how does one link from Access to Outlook, or, alternatively, keep importing and exporting between Access and Outlook (and thereby sync between the two by constantly overwriting the data from the first to the second, then vice versa, in order to 'sync' them) if the Access table has a Primary Key? Can one only do this if there's no Primary Key?

 

#1 - You can only "link" to Outlook folders where changes on either side (Outlook/Access) will be visible to each side immediately in 'real-time" since both sides are looking at the exact same dataset.

#2 - You cannot directly associate an Access table to a "linked" Outlook folder. More pointedly, you cannot "map" a field from one table to a linked Outlook table and then have dynamic bi-directional updating. There may be variations available via queries but will leave that to those more involved with day-to-day Access development. That question would need to be posed in the forum noted above - answer (if any) would not likely be something that is "out-of-the-box".

"Sync'ing" generally infers on-going updates at some pre-deternined time(s) where changes can be made on either side of the equation without the knowledge of the other side in between "sync" sessions. That in turn requires designation of primary/secondary directional updating and some form of "conflict resolution". Not a trivial task if it's going to be done right and goes beyond just the simple association of 2 datasets (of any kind).
