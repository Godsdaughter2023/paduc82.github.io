But going back to your countif approach,

Say your countif function is in F and the invoice is in E

in F2 put    =Countif($E$2:$E2,$E2)

then drag fill down the column.  This will mark the duplicates sequentially so there is no need to sort.   Now you can apply a filter (home tab, editing button, sort & filter dropdown, select filter) on this column and filter out all the 1 values (unselect 1).   Those are your uniques and after you filter, only the duplicates will be visible.  Select all your data except for your header row and Delete entire rows.

karl donaubauer say:

Hi,

 

if you have to deal with that kind of data it always comes in handy to have a "date work table", i.e. a table that contains all days you could ever have to handle in your application. You can create such a table with some lines of VBA like this:

 

Public Function fctDateTable(dteStart As Date, dteEnd As Date)

 

Dim db As DAO.Database
Dim rs As DAO.Recordset
Dim dteDay As Date

 

Set db = CurrentDb

 

db.Execute "CREATE TABLE tblDate(myDate DateTime)"

db.TableDefs.Refresh

 

Set rs = db.OpenRecordset("tblDate", dbOpenDynaset)

 

For dteDay = dteStart To dteEnd
    rs.AddNew
        rs!myDate = dteDay
    rs.Update
Next dteDay

 

End Function

 

Call the function e.g. from the immediate window:

 

fctDateTable #1/1/2010#, #12/31/2020#

 

to get all dates from 2010 to 2020.

 

Now you can use this work table whenever you need to unfold a period of dates etc.

The SQL statement for your example could look like this:

 

SELECT Query1.Name, Format(mydate,"mm/yyyy") AS MonthAndYear
FROM Query1, tblDate
WHERE tblDate.myDate>=Query1.StartDate And tblDate.myDate<=Query1.EndDate
ORDER BY Query1.Name, tblDate.myDate

 

The Access query design view can't handle such a non-equi join but you can copy the string into SQL view and adjust it there.

macropod say:

You can find hyphenated numbers easily enough, but not hypenated words. For example, the following field code can be used to suppress all except the last four digits in a mergefield that gets its data from a field named ‘SSN’, where the data are formatted as ‘123-45-6789’.
{QUOTE
{SET ID {MERGEFIELD SSN}}
{SET EXP3{=({ID}*(-1)-ID)/2}}"XXX-XX-"{EXP3 \# 0000}}

 

I have already given you a field code to properly format Zip Codes, including malformed ones, here: http://answers.microsoft.com/en-us/office/forum/office_2010-word/mail-merge-does-not-print-leading-zero-in-zip-code/0071f3a7-c7e1-4c59-9909-3690c5f1d2bc?page=1

With that field code, it really doesn't matter if the data omit the leading 0s either side of the hyphen. Of course, if the hyphen is missing, some very clear rules would be required before any kind of code (field or otherwise) might be able to work out whether any missing 0s belong to the start of the second part of the Zip Code.
