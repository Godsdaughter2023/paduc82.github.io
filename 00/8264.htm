a. user say:

One possibility:

Use Alt-F9 to view the contents of your TOC field.

By default, Word creates a TOC field like this:

{ TOC \o "1-3" }

This collects entries from all paragraphs that have Level 1, 2 or 3, which includes paragraphs with the standard Heading 1, Heading 2, Heading 3 styles (which are level 1, 2, and 3 respectively. The user cannot change those levels). Incidentally, it will also work with user-defined styles and paragraphs that have had their "Outline level" set to 1, 2, and 3.

So this does not rely on the names of the paragraph styles, only on the Level. If you create a document in the English version of Word and create paragraphs with styles Heading 1, Heading 2 etc., when you open it in the German version (say), you will see that Word actually presents these standard styles with their German language names (Überschrift 1 and so on), but this TOC will still work.

However, if for some reason someone has created a ToC like this

{ TOC \t "Heading 1,1,Heading 2,2,Heading 3,3" }

it may work on your English language installations, but (e.g.) the German version of Word does not translate the text  "Heading 1,1,Heading 2,2,Heading 3,3" . So that ToC field won't work on language versions of Word that use different standard style names.

I don't know what the problem with the missing right-click menu is - On a German version of Mac Word it seems to be OK, but the Windows version may behave differently and I can't check that right now. Do you see the problem when you select a single field and right-click? (You may have noticed even in the English language version that if you have selected a field and some text, right-clicking may bring up the spelling context popup if there is a misspelling. So if you have some standard English text even in the German/Spanish/Italian documents the users may see that popup. But I have to say that that doesn't sound very likely to me.)

a. user say:

I don't think that style name would change. The other possible problem is that the list separator will typically be "," on a Windows system in the US or UK, and ";" in the three other countries you mention. These are set in Control Panel->Regional options and apply to everything (i.e. it would probably upset other things on users' systems if you change the value.

So for those countries you would probably need

{ TOC \h \z \t "COMPANYNAMEHead1GoldContents;1" }

If you can insert the correct version via the available options, you may be able to get the document you need in each country, but if you are shipping the documents from one country to another, the field will fail e.g. when one goes from USA to Germany.

If you can determine exactly what goes into this ToC field, it may be possible to define a field that does a "good enough" job of detecting whether the list separator is ";" or "," - e.g. you could use

{ TOC \h \z \t "COMPANYNAMEHead1GoldContents{ IF { =SUM(1,2) } = 3 "," ";" }1" }

In the case where the separator is actually ";" the { = } field generates a syntax error that will cause problems if you happen to use any of these documents as a Mail Merge Main document. But that sounds an unlikely use for these documents.

graham mayor say:
It might be simpler just to test for the separator character. With a little digging on line I came up with a variation on the following function

Option Explicit
Declare Function GetLocaleInfo Lib "kernel32" Alias _
"GetLocaleInfoA" (ByVal Locale As Long, ByVal LCType As Long, _
ByVal lpLCData As String, ByVal cchData As Long) As Long
Declare Function GetUserDefaultLCID% Lib "kernel32" ()
Private iRetVal1 As Long
Private iRetVal2 As Long
Private lpLCDataVar As String
Private Position As Integer
Private Locale As Long
Public Const LOCALE_SLIST = &HC

Public Function GetListSeparator() As String
    Locale = GetUserDefaultLCID()
    iRetVal1 = GetLocaleInfo(Locale, LOCALE_SLIST, lpLCDataVar, 0)
    GetListSeparator = String$(iRetVal1, 0)
    iRetVal2 = GetLocaleInfo(Locale, LOCALE_SLIST, GetListSeparator, iRetVal1)
    Position = InStr(GetListSeparator, Chr$(0))
    If Position > 0 Then
        GetListSeparator = Left$(GetListSeparator, Position - 1)
    End If
End Function

If you are creating the TOC in code you could perhaps test for the separator and write the value as appropriate.

a. user say:

I've posted a list taken from Windows 2010 here:

http://goo.gl/PpsyQ

You will see that the default is either ";" or "," for every locale except for Amharic and Divehi.

These are just defaults set by Windows. People can change them. It's entirely possible that people in some countries/regions typically change the defaults because they are not what is actually used in those places.

But the { =SUM(1,2) } field I've posted is only going to work in some cases. If the decimal point character and list separator character are both "," (which they can be), it will fail, I think. Also, it is possible to define a List Separator character in Windows that won't work at all in { =SUM(1,2) }, but which will work fine in the \t parameter of a TOC field. "p", for example - as long as the list of paragraph styles in the \t parameter do not also contain "p" characters :-) )

So if you need to deal with a lot of locales, using some VBA (cf. Graham's) to set up the separator character is likely to be essential, but you then need to be sure that the machine generating the document is using the same separator as the one being used to edit the document. Either that, or use an AutoOpen type macro to ensure that the character is detected and updated whenever the user opens the document. You could either stuff the character in directly, or perhaps set a Document Variable called (e.g.) LS, then insert the character using { DOCVARIABLE LS }

Alternatively, if you only need one level, you might be able to arrange that the relevant paragraph style had an otherwise unused level (e.g. 9) and generate the TOC using { TOC \o "9-9" } and avoid having to know the List Separator at all.

Msft will dance around this but the real answer is no. Modern and classic are incompatible and there's absolutely no way to "convert" classic pages to modern. The title of the article linked to above has a misleading title:

"Transform classic pages to modern client-side pages"

but then states:

"Classic SharePoint sites.... cannot present themselves using a modern user interface. A classic site however can host modern client side pages and this is the solution here. "

Modern sites offer a minor styling change at the expense of a lot of classic functionality  - and the styling can easily be replicated with css in a classic site. So I'd stick to classic.

jeeped say:
... As long as the "refers to" field in the name manager uses the default absolute address (e.g. =Sheet!$A$4), the names are stable. However if I change to relative addresses, the reference doesn't "stick" ...

If a defined range name is created with a relative range reference, it is created relative to the current active cell. Typically, a defined name is created with the cell or cell range selected. If the Refers to: is relative, then this will change when a different cell or range of cells is selected. Here are some examples:
  • If C3 is selected and you create a defined name as =Sheet1!A1 then select Z10, the defined name refers to Sheet1!X8. 
  • If C3 is selected and you create a defined name as =Sheet1!$A1 then select Z10, the defined name refers to Sheet1!A8.
  • If B2 is selected and you create a defined name as =Sheet1!A1 then select A1, the defined name refers to Sheet1!XFD1048576. If effect, it has wrapped off the worksheet in an upper-left direction and ended up on the last possible cell in the lower-right corner.
Circular references are also a common problem if you are defining a name with both relative columns and rows while in the cell being defined since the reference will follow the currently selected cell around. There really isn't any use for relative addresses in a defined name except in the most exceptional cases, and if used, the results from any formula containing them must be monitored closely to avoid errors.

BTW, this is also how Conditional Formatting formula references work.

... I've forgotten the significance of tick marks on the sheet reference as shown below. What is the difference?

In you example, there is no difference and Excel will remove the ticks are thery are unnecessary. However, if a worksheet name has a space in it, the ticks preserve the name in its entirety. If you worksheet was called Sheet 1 instead of Sheet1, then ticks would be required (e.g. 'Sheet 1'!A1)
