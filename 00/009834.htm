Paddy 

brad[m] say:

You're welcome.

You're on an activated Win10 build, so yes, if you create the media you should be able to do an in-place upgrade to the full release.

Some people have reported that they did *not* do an in-place upgrade first (ie: they reformatted and installed the final build from scratch) and it still activated but that's not supported so doing so would be risky.  Build the media (make sure it's the same version as what you're using (ie: x64/x86 Pro) then boot into your existing install and run setup from there.  If it asks for a key, just skip that and continue the install (it will likely ask you at install and then again at first boot).  If it doesn't activate right away, don't sweat it - it took mine a little over a day to finally activate.

All depends on the type (POP3, Imap, Outlook.com/Exchange) of email account involved

For POP3 & Imap - no need to export/import, just copy the correct PST file to the new computer

scottgem say:

The Format function returns a text string. Surrounding the text strings with octothorpes (#) does not turn it back into a date value. So that's why its returning no values. 

As Gina correctly asks, why use a hard coded date and why format it as text. Either you use:

DLookUp("Slots","tbl_Slots_Driver","VacDate= #8/21/2016#")

or

DLookUp("Slots","tbl_Slots_Driver","VacDate= #" & MyDate & "#")

where MyDate is a DateTime variable

Aushknotes, 

Your solution is incorrect. You are making things worse by using the Format function. If the VacDate field did contain a time portion then, it would have to be considered. But the correct solutions would be:

DLookUp("Slots","tbl_Slots_Driver","DateValue(VacDate)= #8/21/2016#")

ken sheridan say: I laso would ask why are you hard-coding the 21 August 2016 date?  Normally you'd reference the date from a control on a form or some other parameter.

Even if the relevant date is always the same date, it is data, and data should never be encoded in an expression.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.  Consequently the date should be stored in a table and the column in the table referenced in the expression.
