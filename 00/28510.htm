jack stockton say:

DSum cannot cross tables.  Instead, create a query that links the two tables together, then use the query as the domain source of the DSum.

 

Jack

ken sheridan say: The simplest solution would be to join the two tables in a query which returns the Listed Weight column from Weight_Values and the Group column from Weight_Types, and then call the DSum function against this query.

You could do it in one expression, but you'd need to call the Dlookup function to determine the value of the foreign key in the criterion of the DSum function.  If we assume that the tables are related on numeric columns called Weight_TypeID for instance:

DSum("[Listed Weight]", "Weight_Values", "Weight_TypeID = " & DLookup("Weight_TypeID", "Weight_Types", "[Group] = 'Group1'"))

john w. vinson say:

Well, as a rule, it would be bad design to store the sum in ANY table field. If one of the underlying field values were to be edited, your sum would now be wrong, with no easy way to detect that it is wrong or why it is wrong!

Generally one would simply redo the calculation when it's needed. That's what computers are good at - calculations!

That said, the syntax of the DSum would not change at all. You would be (unwisely!!!) using the DSum, just as described above, as one calculated field in an update query; how you're using it doesn't change how it works.

 

ken sheridan say: As John says, it is rarely appropriate to store aggregated values as this opens the door to update anomalies.  Generally the sum would be computed in a query, which means it always correctly reflects the values from which it is derived.  Generally speaking the only situation in which a derived value should be stored is where the value is not time-independently determined by the column(s) from which it is derived, e.g. the price of a product in an invoice is stored as a value in a column in a row in an InvoiceDetails table, not merely looked up from a Products table, as the latter value will change over time, whereas the invoiced amount should remain static as that at the time when the invoice was raised.

Perhaps if you could explain what you are trying to do in real world terms rather than how you are trying to do it we will be able to advise further.

erniekeen say:

My application is a weights database where information about an object can be stored at various levels of granularity.  If weights are input at a fine level of granularity, their sum should also be stored and, preferably, calculated for the user.  Once input, the weights would not change unless a correction is needed.

 

I was planning to launch the update query (containing the DSum) from my weight input form.  So when the user inputs weights at a given granularity, the aggregate levels are populated by running the query.  I also considered using a calculated control to do the summation and let the user input the sums manually.

 

That is my situation, and I'm sure there are probably better ways to attack it.  Thanks for your thoughts,

ken sheridan say: That doesn't tell us a lot I'm afraid, but it still suggests that you are attempting to store aggregated values redundantly.  The precision of each value (which is what I assume you mean by granularity), should not have a bearing on the issue; the sum of a set of values will reflect the precision of the values, just as if I sum the transactions in my bank account solely on the integer GBP values the sum will be less precise (though not necessarily less accurate) than if I summed the full currency values of pounds and pence.

The key point is that once the individual values are stored in a table, you can compute the sum of a subset of those values at any time; you don't need to store the sum.  To do so is not only unnecessary, it potentially undermines the integrity of the data.

Let's assume a simple scenario of the following tables:

Objects
....ObjectID
....Object
....CategoryID
....Weight

Categories
....CategoryID
....Category

You can sum the total weight of all objects per category with:

SELECT Category, SUM(Weight) AS TotalWeight
FROM Categories INNER JOIN Objects
ON Categories,CategoryID = Objects.CategoryID
GROUP BY Category;

The result table of a query such as this can be used in just the same way as you would a base table in which the sums are stored, but unlike in the latter there can never be any discrepancy in the query's result table  between the sums and the values from which they are computed.

The easiest way to do this is: open both the workbook where you want the formula and the workbook that you want to refer to. To make the following easier, activate the appropriate worksheet in each workbook.
