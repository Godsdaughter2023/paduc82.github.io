ken sheridan say:

Opening a form or report is better, but if you do want to open a query as a datasheet the following function will do it:

 

Public Function OpenTempQuery(strSQL As String)

 

    Dim qdf As DAO.QueryDef

    Static n As Integer

   

    n = n + 1

              

    ' delete temporary querydef object if exists

    On Error Resume Next

    CurrentDb.QueryDefs.Delete "Temp" & n

    Select Case Err.Number

        Case 0

        ' no error

        Case 3265

        ' temporary querydef does not exist

        ' ignore error

        Case Else

        ' unknown error

        MsgBox Err.Number, vbExclamation, "Error"

    End Select

    ' create temporary querydef object

    Set qdf = CurrentDb.CreateQueryDef("Temp" & n)

    CurrentDb.QueryDefs("Temp" & n).SQL = strSQL

    ' open query and then delete temporary querydef object

    DoCmd.OpenQuery "Temp" & n

    ' delete temporary querydef object

    CurrentDb.QueryDefs.Delete "Temp" & n

 

End Function

 

In your case call it with:

 

OpenTempQuery  "SELECT * FROM Q_Tbl_Cor_Enr WHERE Emp_Ful_Nme = """ & Emp_Ful_Nme & """"

 

I'd recommend using a literal double quotes character, each represented in the above by a contiguous pair of quotes characters, rather than the single quote character, to delimit the value of the Emp_Ful_Nme control as names can include apostrophes, e.g. my name in its unanglicized form is Cináed O'Siridean.

 

One thing about this function is that if called repeatedly it will open multiple datasheets, so you could call it and open the query restricted to John Brown, and then move to another record in the form and open it restricted to Jane Green, and both datasheets will remain visible.

ken sheridan say: For this post I've changed your object names to CamelCase for greater readability.  Using upper case for object names makes the SQL very difficult to read.  A common convention, which I follow wherever possible, is to use plural or collective nouns for table names, to reflect the fact that a table is asset, single nouns  for column names, to reflect the fact that each columns represents an attribute, and CamelCase for object names.   Some people prefer CamelCase for table names and camelCase for column names, also eschewing tags lie 'tbl'.  Upper case is usually restricted to SQL keywords.

Firstly the query should have a WHERE calsue, not a HAVING clause.  The latter operates on the data after grouping and is used for things like:

HAVING SUM(OrderAmount) >= 10000

e.g. to identify customers who have ordered at least 10,000 USD or GBP worth of products.  A WHERE clause restricts the rows returned on the basis of one or more criteria, which is what you want here. Note that I've defined the range on those dates on or after the start date and before the day following the end date.  This allows for date/time values in the table which might not have a zero time of day element, something you cannot completely rule out unless you have taken specific steps in the table definition to  allow only integral date/time values.  By defining the range as you had done any rows with such date/time values on the final day of the range would not have been returned.

Forget the DMin and DMax functions, I mentioned those originally on the basis of a misunderstanding of what dates you wanted to show.  Try this:

PARAMETERS [Beginning Date:] DATETIME, [End Date:] DATETIME;
SELECT tbl_Tcd_Cor_Cat.Cor_Nme, tbl_Tcd_Cor_Cat.Cor_Dte, tbl_Cor_Enr.Cor_Ttl, COUNT(tbl_Cor_Enr.Emp_Ful_Nme) AS CountOfEmp_Ful_Nme,
[Beginning date:] AS BeginningDate, [End Date:] AS EndDate
FROM tbl_Tcd_Cor_Cat LEFT JOIN tbl_Cor_Enr ON (tbl_Tcd_Cor_Cat.Cor_Dte = tbl_Cor_Enr.Cor_Dte) AND (tbl_Tcd_Cor_Cat.Cor_Nme = tbl_Cor_Enr.Cor_Ttl)
WHERE tbl_Tcd_Cor_Cat.Cor_Dte >= [Beginning Date:]
AND  tbl_Tcd_Cor_Cat.Cor_Dte < [End Date:]+1
GROUP BY tbl_Tcd_Cor_Cat.Cor_Nme, tbl_Tcd_Cor_Cat.Cor_Dte, tbl_Cor_Enr.Cor_Ttl
ORDER BY tbl_Tcd_Cor_Cat.Cor_Dte;
