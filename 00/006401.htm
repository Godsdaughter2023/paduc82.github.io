This particular field I added is a multivalued field, and each option should appear on a separate row.

If you open the query in design view you will see in the table's field list that  the multi-valued field is shown like this:

    NameOfField
          NameOfField.Value

Click and drag the NameOfField.Value into an empty column in the design grid.

When you open the query as a datasheet it will return multiple rows for each value per row in the multi-valued field.  The values for each of the other fields in those rows will consequently be repeated.  If you are expecting it to return only one distinct instance of the other fields' values, then that is not how queries work.  To do that you would need to return the results in a form or report, not a datasheet.

Goal value - the 'gv' Parameter

When using script to dynamically assign values to goals, an additional goal value parameter is used. In the Bing Ads web application a Goal value is assigned a constant value. In code, you can use JavaScript to assign values to the　'gv'parameter at run-time for Destination goals and Event goals. This goal value is a numeric monetary value that represents the revenue associated with the goal, and is used to calculate the "Revenue" of conversions in reports.

Parameter

Description

Explanation

'gv'

Goal value

Used to pass a numeric custom goal value for a Destination or Event goal. This represents the goal’s revenue shown in reports.

　

For more information Regarding the UET tracking methods with a custom event and API. Please review this link:

https://msdn.microsoft.com/en-us/library/bing-ads-universal-event-tracking-uet-tags-analytics-scripts.aspx

Developer Notes

  1. uetq Object Must Exist

    albert d. kallal say:

    Actually, in both cases (using default sql driver, or native 11 driver), the  end result is a DSN less connection (if you use a file dsn).

    So I recommend you attempt to re-link the table (use a file dsn). When you reach the point to choose a SQL driver, then simply choose the native 11 in place of the SQL driver.

    Everything else should then work as before.

    You don’t need or have to worry about DSN-less, since that’s what your first example is, and that’s what Access will use and create by default.

    So when creating + using a file DSN, during the linking process Access creates a dsn-less link for you.

    Your first connection example is thus already DSN-less.

    Keep in mind that both the native 11, and the standard SQL driver BOTH are using ODBC, and both are standard ODBC connections.

    So you still always using ODBC here – it just a question of choosing native 11 or the legacy “older” SQL driver.

    Since Access by default creates a DSN-less connection, once you link your tables you are then free to distribute the linked database to your users and on each workstation you not need to setup anything with the ODBC confit prompts and dialogs.

    The only real downside of using native 11 is that you have to install the native 11 driver on each workstation – it not installed as part of the standard windows build.

    Because of the above issue, I still use the default SQL driver since then no additional install or setup of the native 11 is required on each workstation.

    So when linking, choose a file DSN as opposed to a “system” DSN, and then Access will create DSN less links for you without any additional efforts on your part.

    The native 11 driver is considered better, but then there is additional hassle and work for deployment. The other main issue is native 11 supports the newer SQL datetime2 formats, were as the older SQL driver returns such columns as text. If you had this working all before, then you likely don’t need to adopt native 11 – but doing so means you’re still using ODBC.
