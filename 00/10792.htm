ken sheridan say:

Rather than a single form based on a single query, you will need a form and two subforms.  The parent form will be for entering each order.  You then have a choice of using nested or correlated subforms for entering the order lines and order line details data.

 

Nested subforms are the easiest solution.  The parent form will contain an order lines subform in single form view and within it will be an order line details subform, in continuous forms view, in which one or more shipments can be entered.  This would include a combo box to select a shipment from a Shipments table, which will include columns for the date of the shipment and any other attributes of the shipment as a whole.  A new shipment can be entered by means of the combo box's NotInList event procedure.  The quantity for the current order line item in the first subform would be entered in a text box in the nested subform.

 

Correlated subforms would be similar, but both would be in continuous forms view and placed separately in the Parent orders form.  The order line details subform's RecordSource would be a query which references the primary key of the order lines subform as a parameter.  The order line details subform would be requeried in the Current event procedure of the order lines subform, so that the order line details subform would show only the details related to the current order line item as the user moves from row to row in the order lines subform.

 

You'll find examples of both nested and correlated subforms in my CorrelatedSubs demo, to which I gave you a link earlier in this thread.

ken sheridan say: The unbound ProductID combo box in the first subform should be requeried in the AfterUpdate event procedure of the unbound OrderID combo box in the parent form.  The  unbound OrderID combo box itself should be requeried in the AfterUpdate event procedure of the Customer combo box.

However, the AfterUpdate event procedure  of a control does not execute when the control is updated by code, only as a result of a user updating the control.  Consequently, further code to reproduce this is necessary.  A further point is that requerying a combo box control will not affect its value, only reload its recordset, so it is necessary to also clear the control of any current value by assigning a Null to it.  It is also necessary, where more than two controls are correlated, to cascade these operations down the line

The code for the AfterUpdate event procedure of the Customer combo box would thus be along these:

' requery the correlated controls
Me.cboOrder.Requery
Me.cboOrder = Null
Me.sfcOrderLines.Form.cboProduct.Requery
Me.sfcOrderLines.Form.cboProduct = Null
' requery first subform
Me.sfcOrderLines.Requery

The cboOrder control's AfterUpdate event procedure would be:

Me.sfcOrderLines.Form.cboProduct.Requery
Me.sfcOrderLines.Form.cboProduct = Null
' requery first subform
Me.sfcOrderLines.Requery

Note that sfc prefix used above is because it is the subform control which is referenced here by name, not the name of the controls source form object.

I've assumed here that the first subform's recordset is restricted by means of a parameter which references the unbound cboProduct control in the subform.

As regards the shipments, Scott and I have both pointed out to you that there is a many-to-many relationship type between OrderLines and Shipments.  The Quantity shipped is an attribute of the relationship type, and consequently a column in the table which models the relationship type.  The nested subform will consequently be based on this table, which models the many-to-many relationship type into two one-to-many relationship types, and will be linked to the first subform by setting its LinkMasterFields and LinkChildFields properties to OrderLineID, or whatever is  the name of the primary key in OrderLines and corresponding foreign key in the table which models the relationship type.

The nested subform would contain a combo box to select the shipment, and a text box to enter the quantity of the item being shipped.  The combo box will be restricted to shipments relating to the current order by referencing the unbound cboOrder combo box in the parent form as a parameter.  Consequently the cboProduct control in the first subform, will need to requery the shipment combo box in the nested subform, e.g.

Me.sfcProductShipments.Form.cboShipment.Requery

If the row being entered into the nested subform represents the first shipment relating the current order, then no row will yet exist in the Shipments table of course, so the NotInList event procedure of the shipment combo box in the nested subform will need to be used to insert a new row into shipments when a new value is typed into the combo box.  How each shipment is identified in the combo box's list is for you to decide.  I'd envisage it being by date, in which case the code for the NotInList event procedure would insert values into both the ShipmentDate column and the OrderID column.  The first would be the value of the procedure's NewData argument, the second would be the value of the cboOrder combo box in the parent form.

john mcghie say:

Spelling in Mac Word 2011 works EXACTLY the same as it does in Word on the PC.

 

You should open a blank document and select the language you want to use most of the time.  Then click the "default" button, to store your selection in your Normal.dotm template as your default language.

 

If you omit this step, Word will work in English USA, which may be what you need.

 

When the spelling checker scans your document, it will automatically switch languages if it finds other languages marked in the text of a document you got from somewhere else.  If you open a document with a mixture of English USA, English Australian and English UK in it, the spelling check will automatically switch dictionaries, word-by-word, while it is working.

 

If the spelling checker comes to a language you do not have, you may see a warning message that "Word can not start the spelling checker."  That means that some text in the document (or the whole document) is in a language Mac Word doesn't have: you will then need to set the language, as you have been doing.

 

This is exactly how Word on the PC works also.
