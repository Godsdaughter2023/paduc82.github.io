I assume that you already have a Media table.  From your description I'm assuming that the different media relate to the item in the parent form, rather than to the individual issues per item.  In which case you will need to model the many-to-many relationship type between the parent form's table, which I'll call Items for this example, and the Media Table.  So the tables would, in broad outline be like this:

Items
....ItemID  (PK)
....Title
....Category
....Rating

Note that the above table does not include a MediaID column.

Issues
....IssueID  (PK)
....Issue
....Comment
....ItemID  (FK)

Media
....MediaID  (PK)
....Media

and to model the many-to-many relationship type between Items and Media:

ItemMedia
....ItemID  (PK)
....MediaID  (FK)

The primary key of this last table is a composite one made up of both foreign key columns.

You'll find an example of this sort of basic many-to-many relationship type and how to represent it in forms as StudentCourses.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates a number of ways of interfacing with the data, but in your case a standard form/subform should be used, adding the subform to your existing parent form and basing it on the ItemMedia table and linking it to the parent form on ItemID.  If space on the form is restricted you can place your existing issues subform and the new media subform on separate pages of a tab control.  The FormsDemo file in my same OneDrive folder includes an example in which the main data from the parent form is above the tab control, so stays visible throughout, with other data on its first page, and with  two subforms on the second and third pages of the tab control.

BTW, really you should also have a separate Categories table which is referenced by the Items table in a enforced relationship.

ken sheridan say:

So am I right in thinking that by media you mean the programmes per se, not in the following sense of the OED definition?

Medium

noun. LME.
[ORIGIN Anglo-Norman = Old French nun, num (mod. nom) from Latin nomen name.]

5 An intermediate agency, instrument, or channel; a means; spec. a channel of mass communication, as newspapers, radio, television, etc. See also media noun². E17.

Sloan Wilson: Television is developing into the greatest medium for mass education and entertainment. H. Read: If we have ideas to express, the proper medium is Language.

 - NOTE: As media is the pl. of medium, the sing. uses are traditionally regarded as erroneous; however, use as a collective pl. with a sing. verb (like e.g. government) is now generally accepted. Cf. notes at agenda, data.

What you appear to have is a binary many-to-many relationship type between programmes and issues in that each programme can relate to one or more issue , and each issue can relate to one or more programmes.  A many-to-many relationship type is always modelled by a table which resolves the relationship type into two or more one-to-many relationship types, so in this case you need a table like this:

MediaIssues
....MediaID  (FK)
....IssueID  (FK)

Each of the foreign key columns references the primary key of the media and issues table respectively, and together the two columns constitute the composite foreign key of the table.  You can if you wish add further non-key columns to the table to

I previously gave you the link to my OneDrive folder containing the StudentCourses  demo file which illustrates several ways of interfacing with data in this sort of basic many-to-many relationship type, though I'd recommended the standard form/subform interface.  The subform in my demo includes only a single combo box control bound to the CourseID foreign key column, but you can include read only controls to show other columns from the referenced (issues) table.  There are a number of ways of doing this.  If you take a look at the NotInList demo in my OneDrive folder this contains a frmContacts form in which data from the referenced Regions table is automatically shown in a read-only text box when a city is selected.  This is done by basing the form on a query which joins the relevant tables.  You can do the same by basing your subform on a query which joins the issues table to the MediaIssues table.   My form also illustrates how a new city can be inserted by typing its name directly into the combo box and using code in its NotInList event procedure to open a form to which the new city name is passed as its OpenArgs property.  The region in which the city is located can then be entered in the cities form.  You could do the same to enter a new issue not currently represented in the issues table.

To assign the same group of issues from one programme to another you can execute a simple 'append' query in code to insert rows into the MediaIssues table with the MediaID of the programme to which you wish to assign the group of issues and the IssueID values from those rows in the MediaIssues table with the MediaID of the programmed whose issues you wish to assign to the new programme.  This could be done by opening a dialogue form from a button in the parent form.  In the dialogue form would be a combo box to select the programme from which the issues are to be 'copied' and a button to execute the query and close the dialogue form.  So the code in a button in the dialogue form would be like this, where cboMedia is the name of the combo box in the dialogue form:

Dim strSQL As String
Dim lngMediaID As long

lngMediaID  = Me.OpenArgs

If Not IsNull(Me.cboMedia) Then
    strSQL = "INSERT INTO MediaIssuses(MediaID,IssueID) " & _
        "SELECT " & lngMediaID & ", IssueID " & _
        "FROM MediaIssues " & _
        "WHERE MediaID = " & Me.cboMedia

    CurrentDb.Execute, dbFailOnError
    DoCmd.Close acForm, Me.Name
End If

The code for the button in the parent form would be:

If Not IsNull(Me.MediaID) Then
    DoCmd.OpenForm "YourDialogueForm", _
        WindowMode:=acDialog, _
        OpenArgs:=Me.MediaID

    ' requery issues subform to show inserted issues
    Me.sfcIssues.Requery
End If

Note that sfcIssues is the name of the subform control, i.e. the control in the parent for which houses the subform, not the name of its source form object, unless both have the same name of course.

al edlund say:

 

try this,

al

 


Add this to EventDblClick in the shapesheet (obviously replacing the ip address)
CALLTHIS("ThisDocument.TermServeToTarget",,"127.0.0.1")

Add "Windows Script Host Object Model" to Tools => References in the vba editor


Add this macro to the document
'
'
'
Public Sub TermServeToTarget _
        (ByVal visShape As Visio.Shape, _
        ByVal strIpAddr As String)
       
        On Error GoTo ErrHandler
       
        Dim strCommand As String
        strCommand = "c:\windows\system32\mstsc.exe /v:"
        strCommand = strCommand & strIpAddr
        strCommand = strCommand & " /console"
        ' create the script shell environment
        Dim shellWin As Object
        Set shellWin = CreateObject("shell")
        ' now execute the command that we built
        shellWin.Run strCommand
        Exit Sub

ShellCommand:
        'Debug.Print "shell start"
         Shell strCommand, vbNormalFocus
         'Debug.Print "shell start"
         DoEvents
         Exit Sub

ErrHandler:
    Dim myErr As ErrObject
    Set myErr = Err
   
    If Err.Number = 429 Then
        'Debug.Print "shellwin failed"
        GoTo ShellCommand
    End If

End Sub
