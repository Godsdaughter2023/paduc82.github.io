ken sheridan say: I didn't really mean to propose Bob's last post as an answer; it was an errant mouse-click.  Nevertheless he makes a good point.

From your description of the problem it sounds like the bound column of the second combo box is not made up of distinct values.  As it is invariably going to the first item in the list my guess would be that it is the same value in each row, the most likely culprit being the ClientID, but this is only informed guesswork of course.  Whatever it is, this sort of behaviour occurs when you select an item from the list where the value of the bound column is the same as that of a row earlier in the sort order.  What happens is that the first row in the sort order is selected, not the one you clicked on.

Trying to base all the RowSource properties on a one-size-fits-all query is probably the root of the problem, though there is no reason why that approach should not work if you get the SQL for each RowSource property right, which means ensuring that rows with distinct bound column values are returned.  The fact that the column is the primary key of one of the tables in the query doesn't ensure this, as the value will be repeated for all matching rows in another table or tables in the query.  You can ensure this provided that only a subset of columns with distinct values (in combination, not individually) are returned, e.g. for clients you could have:

SELECT DISTINCT ClientID, FirstName & " " LastName FROM YourQuery ORDER BY LastName, FirstName;

However, the more usual approach would be to return rows solely from the Clients table:

SELECT ClientID, FirstName & " " LastName FROM Clients ORDER BY LastName, FirstName;

The BoundColumn property would be 1 in each case, i.e. the ClientID is the value of the control.

Similarly with the case types combo , you'd base the RowSource of on the cases table if this includes a Case 'natural' foreign key column of text data type, along these lines:

SELECT DISTINCT CaseType FROM Cases WHERE ClientID = Form!cboClient ORDER BY CaseType;

Note that in this you have to use the DISTINCT option because CaseType is not the primary key, so there can be multiple instances of each value per client.  Note that you can use the Form property to reference the current form here rather than referencing the form as a member of the Forms collection.  In fact you can even omit that and simple reference the control by name, but I prefer to qualify it with the Form property to avoid any possible ambiguity.

If one the other hand the cases table includes a numeric CaseID 'surrogate' foreign key you'll need to join the Cases and CaseTypes tables in a query:

SELECT DISTINCT CaseTypes.CaseTypeID, CaseType FROM Cases INNER JOIN CaseTypes ON Cases.CaseTypeID = CaseTypes.CaseTypeID WHERE ClientID = Form!cboClient ORDER BY CaseType;

For cases you'd use something like this in the first scenario:

SELECT CaseID, CaseDescription FROM Cases WHERE ClientID = Form!cboClient AND CaseType = Form!cboCaseType ORDER BY CaseDescription;

or this in the second:

SELECT CaseID, CaseDescription FROM Cases WHERE ClientID = Form!cboClient AND CaseTypeID = Form!cboCaseType ORDER BY CaseType;

Finally for the referrals combo box, something like this:

SELECT ReferralID, ReferralDescription FRO Referrals WHERE CaseID = Form!cboCase ORDER BY Referral

If you've been keeping count you'll have noticed that there are four tables referred to above, Clients, CaseTypes, Cases and Referrals, whereas you say you have three.  At a guess I'd say you are probably missing the CaseTypes table as all others are essential to the model.  While you can get away without it, the danger of bad data is always there as you cannot enforce referential integrity to ensure that only valid case types are entered in the cases table.

The other issue with this is whether the Meeting table into which you are inserting values via the form is correctly normalized.  Of the values selected in the combo boxes only the ReferralID should be inserted into a column in the meetings table.  This is because ReferralID determines CaseID, which determines both CaseType  ( or CaseTypeID) , and ClientID.  Consequently to store those values in columns in a Meetings table would introduce redundancy and leave the table open to the risk of inconsistent data.  I have to say that many people accept this degree of redundancy, but if you do then be aware of the potential dangers to the integrity of the data.  In my line of work, which is partly of a quasi-judicial nature, we could not accept this type of risk as the legal consequences would be too great to contemplate (things did sometimes go wrong as result of simple clerical errors in the pre relational database days, with the result that people found themselves subject to legal constraints which should never have applied to them!).  For others it may not be such a serious issue, however.  With a fully normalized table, with no redundancy, it is still possible to enter data by top-down selection from correlated combo boxes, and if you are interested to see how I've posted a demo at:

http://community.netscape.com/n/pfx/forum.aspx?nav=libraryMessages&tsn=1&tid=23626&webtag=ws-msdevapps

It uses the local administrative areas of County, District and Parish in my area, but the same principles apply to any hierarchically structured data.

Ken Sheridan, Stafford, England

Private Sub Form_Unload(Cancel As Integer)

   Set db = Nothing
