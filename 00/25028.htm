stefan blom say:
By default, Word tries to apply the formatting of styles in the target document. This doesn't work if the pasted data is formatted in styles that do not exist in the target document, of course. Also, if you paste in text using a built-in style which has not yet been used in the target document, Word modifies the target style.

"Keep Text Only" guarantees that no formatting is brought in from the source file, but it will also remove any formatting which isn't text (such as fields, pictures, etc.), so it won't help in this case.

Sometimes, especially if the target is a mix of styles and direct formatting, reversing the process might help. In other words, instead of copying content from document A into document B, copy content from B into (a copy of) A.

Tables are a special case in Word. Formatting from paragraph styles may be ignored unless you make sure that the document defaults match the settings in the Normal style of the target document.

Part of the keyboard is adapted to include local umlauts, such as ä, ö, ü, etc. Some special symbols also have a different place.

Sounds like your firmware has been changed to the German version, you need to buy an E90 with qwerty...

bob jones aka: cybertaz say:
Despite superficial appearances, Word table objects & PPT table objects are quite different... PPT does not have the tools to work with Word tables [Note that there is no Table menu in PPT]. That's why they're pasted as Microsoft Word Document Objects.

They aren't linked to the original Word table but are linked to a Word document embedded in the presentation file. What you see on the slide is just a graphic representation of the pasted table. Any editing must be done using Word.

What you can do is create a table in PPT with the requisite number of columns, then click in the top left cell to paste. The corresponding number of rows will be generated in the PPT table. The result may not retain all formatting but it will be an editable PPT table object.

braytonfisher say:

braytonfisher say:

Amazingly, saving a CSV file in Excel 2016 uses a different format from 2011.  I can't believe such a common format would be changed by Microsoft without an option to use the previous format - such poor testing.  Anyway, I was able to process the data after removing the extra line feed, so you will probably have to do the same, unless either your program is changed, or Microsoft gives a "Save As" option the makes the file compatible to 2011 again.  A great link explaining this is at 

https://gist.github.com/jennybc/0be7717c2b5b30088811

I had to open terminal, and use two commands:

First change to the directory (folder) where my file is located:

cd /Users/brayton/Documents/myfolder

Next run the transliterate (tr) command to remove the extra line feeds (\n).  In this example I am reading the file "file.csv", removing the line feed, and copying to "file2.csv":

tr -d '\n' < file.csv > file2.csv

Try it out and please respond.
