a. user say:

I was finally able to reproduce this problem. (I also deleted my earlier post).

What seems to fix the problem here is changing the default format for date insertion in the Insert Date dialog in Insert tab->Text group->Insert Date and Time. At this point, I do not know whether any change will work or whether it needs to be in a particular language (such as the editing language or the interface language).

I do not know the precise sequence of steps that will cause this, but I had been experimenting with various language settings (in Windows control panel, the Word default, the Display language and the proofing language applied to the DOCPROPERTY fields.

The thing that appeared to cause the change was changing the default editing language (in my case, from Dutch to English). Before that


{ DOCPROPERTY thedate } displayed in the current default format for Dutch, regardless of any proofing language applied to the field

{ DOCPROPERTY thedate \@ "d MMMM" } correctly applied the date format using the proofing language applied to the field.

When you change the editing language you have to restart Word. I changed it to English (UK) then re-opened the document and re-executed all the fields using F9. At this point they all showed the date in the current default format for Dutch. The date format switches were not applied.

I then modified the date format using Insert tab->Text group->Insert Date and Time. As it happens, I modified the format for the English (UK) locale.

WHen I updated the fields, 

{ DOCPROPERTY thedate } displayed in the current default format for English (UK), regardless of any proofing language applied to the field

{ DOCPROPERTY thedate \@ "d MMMM" } correctly applied the date format using the proofing language applied to the field.

I have come across the problem where Word does not apply defaults correctly until you change them before - please see my posts (and others) in the conversation at https://answers.microsoft.com/en-us/msoffice/forum/msoffice_word-mso_other/insert-date-and-time-in-word/17bee601-80b6-4a52-92e2-65c720db1a91?page=2

If it is any help to you, when I use Custom Document Properties, I always use Text format variables, partly because they do not force any particular format on you, and partly because for dates I always prefer to use the standard ISO date format YYYY-MM-DD (well, OK, it is only part of the standard format) because 

 a. there are no language-dependent day or month names

 b. as far as I can tell, it is locale-independent, in the sense that Word never confuses the month with the day when you use that format, whereas it may do so if you use DD/MM/YYYY

 c. it sorts correctly as a text string

Because the Custom Document Properties dialog requires that you enter date fields using the short date format specified in Windows, you cannot use the ISO format unless you also change the format in Control Panel, which I do not want to do. So I use a text value and accept that Word cannot validate the date.

Yes, they can still be connected to Exchange.  They have an on-prem Exchange 2007 server that they're migrating away from.  If an account for this server still exists in your Outlook 2013 profile when Outlook 2013 is upgraded to Outlook 2016, these errors will appear since Outlook 2016 cannot work with an Exchange 2007 server.  The fix would be to completely remove all references to this server from Outlook 2013 before it is upgraded to 2016.  If you have the Office 365 desktop apps (which is, in fact, Office 2016) installed and are connecting to an Office 365 Exchange Online server, use Outlook's Auto Account Setup feature which should use Exchange Server protocol, not EAS.
