john w. vinson say:

I don't think you need the IIF() statement or the text strings at all! A calculated field

InsideDateRange: [Course_Date] >= [PreEffectiveDate]

will contain -1 (a TRUE Boolean value) or 0 (FALSE) depending on the dates. A criterion of True or False (without the quotes) will retrieve the appropriate values.

scottgem say:

True and False are constants in Access that refer to a value of a Yes/No datatype. In a Yes/No datatype, a 0 value represents False and a -1 (or any non zero value) will return True. 

Your expression: 

Inside_Outside Range: IIf([Course_Date]>=[PreEffective Date],"True",IIf([Course_Date]<=[PreExpiration Date],"False"))

Does NOT return a Yes/No datatype, but a string data type. Also it is incorrectly coded since you don't use a False condition for the second IIF (which is also unnecessary). As John pointed out you don't need an IIF at all. But the correct IIF would be:

Inside_Outside Range: IIf([Course_Date]>=[PreEffective Date],True,False)

then you could use True or False (without quotes) as criteria or just 0 or -1.

Is Microsoft abandoning the power users? - Microsoft Community

rantzky say:

Is Microsoft abandoning the power users?

Hi,
As a power user I'm experiencing a significant decline in user experience and work effectiveness in applications, led by MS. I am referring to the elimination of the traditional menu system in favor of more "friendly" alternatives in programs such as Office. Very unfortunately, this trend set by MS is seemingly being adopted by additional companies such as Adobe, Mozilla, Google and others, leading to a frustrating world which I feel is centered around novices.

There is no doubt that the new visual systems may be easier to pick up by a novice user. They might be more efficient in terms of screen real estate. However, for a power user working with a large screen, they are an utter disaster. I will try to demonstrate with two examples.

1. As a power user, I can master a program extremely quickly by browsing its menu bar. The menu system guarantees that all program features are arranged in a systematic way. No other UI system guarantees such visibility of the program features. Even grayed-out options convey a huge amount of information on what the program can do. This listing of all program features in one place is *extremely* important for a power user and accelerates the learning curve. No dynamic system that toggles visibility of UI features, and substitutes symbols for text, can provide a similar level of informativity and conciseness.

2. Keyboard shortcuts: this is probably the most critical disadvantage of all alternatives to the menu system. As a power user and a typist, I achieve dramatically accelerated work pace through utilization of keyboard shortcuts. Thus, over time I tend to learn the shortcuts for the more useful program features. In a menu system, this occurs naturally, as the concept is tightly integrated in the system: each shortcut is attached directly to its menu item. The shortcut is thus straightforward to find, and typically after just a few times the menu item is accessed, the keyboard shortcut is memorized. Every alternative UI system makes it unbelievably cumbersome to determine keyboard shortcuts, and they are treated as an after-thought rather than an important component. On some systems (not MS, admittedly) keyboard shortcuts are becoming essentially extinct, listed only deep in the program manuals. For a power user, this situation leads to much frustration, and substantial loss of effectiveness.

What I am seeing is that these new trends towards intuitiveness and low-power devices, though beneficial to many users, are loosing the power users on the way. In other words, the speed of initially learning a new program is effectively limiting the *maximum* speed that same program can be ultimately used.

I thus wish that these new user interfaces did not come to replace the traditional menus, but to augment them. The menu system should be kept as a secondary means of input, unifying all commands and shortcuts in one place. I would much rather see MS, as the leader of this recent trend, *not* lead a trend of eliminating the menu bar, as much as a trend of reinventing the rest of the interface.

Regards,
Ron
