graham mayor say:
You can delete with a rule, but only after the message has arrived.

The name you quoted appears related to a gaming web site. You should be able to instruct the site not to send you messages, or check the message header and see which domain the messages are coming from.

You can only block an address or a domain.

For the third option, you open a gmail account and get it to fetch your mail from all your accounts (or at least the one you are getting spam from). The sender doesn't need to know that you have gmail. You setup gmail as a pop server and download your mail from gmail rather than your isp address. Any mail that passes through gmail is subject to gmail's spam filtering.  See https://dl.dropbox.com/u/57986755/Use%20Google%20GMail%20to%20remove%20spam%20from%20your%20email.pdf

The above instructions are old, and the dialogs have changed, but the principles remain good.

The filtering is quite aggressive so you will need to check its spam filters for the first few weeks and rescue any legitimate mail. It helps to create a 'white list' of wanted addresses in gmail from your Outlook contacts.

The fourth option uses a third party spam filter - Mailwasher. All your mail is received by Mailwasher and you can setup filtering options that Outlook cannot readily manage.

I'm not clear what you are trying to achieve here, but it looks like you might be trying to return data from two tables in a many-to-many relationship type modelled by a third table, but with all values from both referenced tables returned regardless of whether there is a row in the third table which maps to the relevant rows in both referenced tables.  If so you would need to return the Cartesian product  of the two referenced tables, which is done by including both tables in the query, but without a specific join.  You can then indicate which of the rows represent an actual relationship value, and which do not, by means of the EXISTS predicate and a subquery.  The query below is an example which used data from my StudentCourses demo:

SELECT StudentID, FirstName, LastName, CourseName,
EXISTS(SELECT *
            FROM StudentCourses
            WHERE StudentCourses.StudentID = Students.StudentID
            AND StudentCourses.CourseID = Courses.CourseID) AS Enrolled
FROM Students, Courses
ORDER BY StudentID,CourseName;

ken sheridan say: I said nothing about SubjectID, it's StudyID I'm referring to, which is a key of neither tmpq_bst nor tblEnrollment.  It's nothing to do with Access per se, but a basic property of the  database relational model.  A join on two non-key columns represents a binary relationship type.  A binary relationship type is modelled in one way only, which is by its resolution into two unary relationship types by a separate table (StudentCourses in my earlier example is a case in point).   In a unary relationship type the referenced column must be a candidate key, not necessarily the primary key note, of one table, while the referencing column is its corresponding foreign key.  Note also that 'table' does not necessarily mean a base table, but also the result table of a query.

ken sheridan say: As you've found, the solution is to join two queries, or a table and a subquery.

If we take the tables from my earlier example, and try to use two LEFT JOINs with a view to returning all contacts, whether or not employed, and all employers whether or not they have any contacts as employees:

SELECT *
FROM Employers LEFT JOIN (Contacts LEFT JOIN ContactEmployers
ON Contacts.ContactID = ContactEmployers.ContactID)
ON Employers.EmployerID = ContactEmployers.EmployerID;

This will not work because the 'preserved' tables, Employers and Contacts are in a many-to-many relationship type modelled by ContactEmployers.  The subquery can use a RIGHT JOIN, which would return employers without employees, but not unemployed contacts.  To return both a UNION operation is used:

SELECT FirstName, LastName, Employer
FROM Employers LEFT JOIN (Contacts RIGHT JOIN ContactEmployers
ON Contacts.ContactID = ContactEmployers.ContactID)
ON Employers.EmployerID = ContactEmployers.EmployerID
UNION
SELECT FirstName, LastName, Employer
FROM Employers RIGHT JOIN (Contacts LEFT JOIN ContactEmployers
ON Contacts.ContactID = ContactEmployers.ContactID)
ON Employers.EmployerID = ContactEmployers.EmployerID;

jim gordon mvp say:
Hi.

The purpose of making a new user account in system preferences is to test to find out whether the problem lies with preference files or the Normal (default) template. If everything is OK in a new user account, then we have isolated the problem to preference files or the Normal file. If the problem continues to be evident we know to look to other aspects such as a faulty install of Office, problems with Mac OS X installation, problems with permissions, problems with the file system, etc. 

The order in which things are suggested to try is based on what the crash report indicates and what seems to have worked for others who experienced the same or similar problems. Troubleshooting is trial and error, unfortunately, because the same symptom can have many causes.

Since the problem exhibits in a new user account, we can rule out problems with preferences and the Normal template.

You say that reinstalloing fixes things, but they quickly degrade. That hints at system problems or hardware (disc) problems. So lets focus on these things.

Run Disk Utility to verify the startup volume and repair disk permissions. If you have Drive Genius or DiskWarrior, run the repair utility and rebuild the volume and directory.

Is the problem fixed? If yes, then you're all set. If  not, then I would nex suggest a reinstall of Mac OS X (you don't have to wipe your drive to reinstall Mac OS X).
