greginmich say:

CFA is a new Windows Defender anti-ransomware feature, and if you feel that you don’t need ransomware protection, then you can just follow the steps in this article to turn off CFA:

https://support.microsoft.com/en-ph/help/4046851/windows-10-controlled-folder-access-windows-defender-security-center

However, that Microsoft article is purporting to tell us how to:

“Add an app to the list of safe or allowed apps to prevent them from being blocked.” 

When in fact it's really only telling us how to turn off CFA – which will allow your friendly apps access to protected folders to be sure; but right along with unfriendly apps, malware, ransomware, and data wipers – so it’s important to understand how logically distorted this particular article is. Turning off CFA is not allowing an app through CFA; and it’s not allowing an app through the WDSC; it’s just turning off CFA – which means that nothing will be blocked. If nothing is blocked, then nothing can be “allowed”.

The Microsoft article that actually describes how to allow a blocked app through CFA is right here:

https://support.microsoft.com/en-us/help/4012987/windows-10-virus-threat-protection-windows-defender-security-center

When you see one of these Unauthorized Changes Blocked notifications; follow these simple steps to allow the program through Controlled Folder Access:

1. Click on the notifications icon at the far right in the taskbar.

2. Write down the file path provided in the notification.

3. Click on the notification, and then on the Add button (+).

4. Navigate to the file identified in the notification, and then select and “open” it.

The file path in the Unauthorized Changes Blocked notification tends to be truncated – although there’s usually enough of it there to locate the app without any trouble if you just assume that it’s in the Program Files or Program Files (x86) directory. If the file path is truncated to the point where you can’t locate the blocked app in the file-picker dialog; then open Event Viewer; navigate to the Windows Defender Operational log; and locate the blocking event (Event ID 1123):

1. Right-click on the Start button and select Event Viewer.

2. Navigate to Applications and Services > Microsoft > Windows > Windows Defender > Operational

3. Filter for (or just look for): Event ID 1123

Generally speaking, CFA should be turned off whenever you’re installing any new application, and then turned back on again once the installation is complete.

Additional information can be found here:

https://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_scanning-windows_10/windows-defender-is-blocking-too-much-saying-it/83a1d803-c687-4ddc-800b-5bb53c6151d1 

If CFA starts blocking apps when you haven’t enabled it, this might be the effect of a compatibility bug that seems to be activating CFA in the presence of some third-party AV apps. In this case, the first thing to try would be disabling the third-party AV to see if this activates Windows Defender. If so, then toggle CFA on, and then back off again – and then enable the third-party AV in order to turn Defender back off. If that doesn’t work, then you’ll probably have to uninstall, and then reinstall the third-party AV app.

Additionally, some third-party AV apps that are capable or running in tandem with Defender (in its real-time mode) might cause problems with CFA. For example, we’re not sure if the anti-exploit protection packaged with Malwarebytes Premium is fully compatible with Windows Defender’s CFA.

If CFA is acting erratically in the absence of a third-party AV app then follow these troubleshooting steps

1. Remove any undetected malware by scanning with several third-party malware-removal apps, starting with Malwarebytes Free:

https://answers.microsoft.com/en-us/protect/forum/protect_other-protect_scanning-windows_other/list-of-malware-removal-tools/d824b9af-ebd8-4c47-94e2-8ee6c544c100

2. Remove any antimalware remnants by running the cleanup utilities for any preinstalled or previously installed AV apps:

https://answers.microsoft.com/en-us/protect/forum/protect_other-protect_start-windows_other/list-of-anti-malware-product-removal-tools/2bcb53f7-7ab4-4ef9-ab3a-6aebfa322f75

3. Run the standard Windows 10 system integrity checks:

https://answers.microsoft.com/en-us/windows/forum/windows_10-update/system-file-check-sfc-scan-and-repair-system-files/bc609315-da1f-4775-812c-695b60477a93

john mcghie say:

Hi Sherry:

This problem is very frustrating, and annoying.  Essentially, it is caused by a very poor design of Word's numbering system.  That's not helped by the tutorial you refer to giving you only half the information :-)

The issue is that when you want to send documents to other computers, you need the numbering to be held together with a LIST STYLE.  The "list" you created in that tutorial lives in the local computer.  A List Style is stored in the document and travels with it.  It's a big design bug in Microsoft numbering.

Deleted files reappearing, broken Search Index and File History to - Microsoft Community

keris say:

Deleted files reappearing, broken Search Index and File History to blame

Over the past few preview builds, I've been having issues where files I've deleted have been popping back up as 0 byte stubs. This started around 9/22/2016, judging by the dates on some of the 0kb files. Unlike the usual zombie file (that is usually caused by something with an open handle that refuses to give it up), they would not go away on reboot; I could actually delete them normally. It'd been semi-random which files come back, and not really on any noticeable schedule. I'd hoped it might get fixed in an update, but so far has not. So I sat down to try to figure out what was causing the files to reappear.

Enabling File Object Auditing did not help. As I observed some populate onto my Desktop, the Event Log showed no file write operations on the folder, only reads from Explorer looking at the metadata of the "new" files. I then turned Sysinternals Process Monitor on and filtered for file operations on my Desktop. And then waited. And waited. And then some files popped into existence, and the offending process was revealed: SearchProtocolHost.

Something in the index is out of whack; I don't know why it's CREATING files it thinks are there (this seems like one part of the bug). I first thought to try to let the search indexer rebuild via the Advanced tab in Control Panel. Big mistake. It now churned through and created dozens of 0 byte files, of things that I deleted months and months ago.

At first, I thought the edb database that Windows Search uses had been corrupted over an update. This seemed pretty similar to the Windows Start tile database issue I had in earlier builds (which was much harder to fix without deleting my entire user profile). I admit I didn't try searching for any MS troubleshooters to attempt to repair the corrupt search database, as deleting it and letting it recreate is fairly harmless; the most evil side effect is how long it takes to re-index a large amount of content.

But that didn't fix the problem. Even rebooting and starting from scratch had the 0 files start showing back up after the index finished with all the real files. In fact, every file I'd ever deleted in a folder marked for index AND in my Libraries started showing up. This time, I left Procmon running for an entire search rebuild, stopping things after a bundle of 0 files showed up. Filtering for a single file that shouldn't be there, I found a rather odd pattern: it seems that said file was looked for in my File History first before SearchProtocolHost added it back to the Desktop. That seemed ... odd.

The log looks like this:

  1. CreateFile to open the File History file for Read, getting "SUCCESS"
  2. QueryInformationVolume followed by QueryAllInformationFile to read the attributes on the remote file, then a CloseFile.
  3. CreateFile to open the file locally for Read, getting "PATH NOT FOUND"
  4. QueryDirectory looking for the file, getting "NO SUCH FILE"
  5. CreateFile to open the file for Read Attributes, getting "NAME NOT FOUND"
  6. CreateFile again to Read Attributes, only this time with OPEN_ALWAYS set as it's disposition, creating a 0 byte file.

I have Procmon logs of it doing this for multiple files.

After stopping the search and moving the File History configuration (so it'd be very much off), I tired letting the indexer run again. The flood of "new" old files stopped, but any zombie 0 file already there would come back, over and over. I had to delete the search database and reboot before letting it build from scratch again to get it to stop. With File History off, I've yet to have any old deleted files pop up again.

Reconnecting File History to the shared drive, picking my PC from the list, causes the zombie files to start coming back again (forcing another delete and long search index). If I remove the File History config database on the file share, pointing my PC at the share causes it to make a new PC folder to put backup files into instead of parsing the old one. At this point, the only solution to get File History working without the zombie files is to delete the entire File History archive for this PC and start over; I could keep the giant blob of files around for manually looking for something to restore, but more than doubling my backup size isn't very appetizing.

Steps for a manual fix and cleanup:

Warning, you will lose your File History database and will have to fish files out of the old folder by hand to restore them!

  1. In the Control Panel, open up Indexing Options and note down the folders you have set to index (so you can clean them up).
  2. Stop the Windows Search service. (If it restarts on you, you might also have to Disable it until you're done cleaning up.)
  3. Turn File History Off.
  4. Delete the directory "C:\ProgramData\Microsoft\Search"
  5. Delete the directory "%LOCALAPPDATA%\Microsoft\Windows\FileHistory"
  6. Clean up the 0 byte files. They'll only be in folders you have set to index, so consult the list from step 1. You can use Powershell to make life easier:
    1. $zero =  gci "C:\Users" -Recurse | where { $_.Length -eq 0 -AND $_.LastWriteTime -gt (Get-Date).AddDays(-1) }
    2. "C:\Users" being one path that normally set to index; replace as needed (you can add to it any other paths by making a comma seperated list). Replace the -1 on AddDays with a bigger negative to look further into the past (or just remove the LastWriteTime check altogether if you want).
    3. CHECK THE FILE LIST. Make sure it's all stuff you are sure you want gone.
    4. $zero | Remove-Item
    5. All the files should go poof. (And NOT to the Recycle Bin, so again, CHECK THE LIST FIRST.)
  7. Reboot, to clear out any and all file caches and whatnot.
  8. Start the Windows Search service back up; it's Automatic (Delayed Start) if you Disabled it.
  9. If you want to use File History again, you will need to clean up your backup drive. Either delete your old backups or move the folder to a different place. Then just point File History at the drive again and let it rebuild.
