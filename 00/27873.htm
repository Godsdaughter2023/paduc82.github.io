dbagshaw say:

I had the same issue, running Windows 7 with Office 2007, all updates installed.  The problem would only occur if Excel was already running when I tried to edit the embedded spreadsheet.  My problem was apparently caused by an Excel COM add-in; after I disabled the add-in, the problem went away;  as a sanity check I re-enabled the add-in and the problem came back again.  This was my problem:

1. Create a new Word document, embed an Excel spreadsheet, add a few values, click outside the spreadsheet, save the document, close Word.

2. Make sure Excel is not running.

3. Open the same Word document again.  Double click on the embedded spreadsheet.  You should be able to edit cells.  Close Word.

4. Start Excel. 

5. Open the same Word document again.  Double click on the embedded spreadsheet.  This time you will get the error "The program used to create this object is Excel. That program is not installed on your computer. to edit this object, you must install a program that can open the object."

5. From Excel, disable the COM add-in (Excel Options, Add-Ins, COM Add-ins, Go, deselect the add-in, OK).

6. Open the same Word document again.  Double click on the embedded spreadsheet.  You should once again be able to edit cells.


In my case, the add-in was Excellent Analytics version 1.0.0.71.  I don't know whether the problem is with Excel or the add-in.  Just reporting the facts!

This problem seems to be fairly widespread, and has been happening in older versions of Windows and Office.  Perhaps someone can tell us what the add-in is doing to cause the problem??

mgevans say:

Hey Folks - from Microsoft on this case as of End of April 2015:

"If all goes as planned with design, implementation and testing the current target for release would be the June timeframe."

Detail behind why this occurs with certain versions of office

Office is designed the way it is to be in accordance with RFC 2111 and that there may be scenarios other than our case here where Office does and will continue to rely on CID being globally unique.

The product group wanted to make sure the customer understood that unique CID values is not an Office requirement, it is part of the IETF specification for all MIME encoded email messages.  Per RFC 2111, a Content-ID (CID) value must be globally unique, which is documented in a couple of different places in section 2:

 

“The Content-ID of a MIME body part is required to be globally unique.”
 
“Both message-id and content-id are required to be globally unique. That is, no two different messages will ever have the same Message-ID addr-spec; no different body parts will ever have the same Content-ID addr-spec.”
 

It is not unusual or incorrect for Office to build a performance cache that expects globally unique CIDs for different content.  The issue in this case was with a performance cache.  It is faster to index cache items based on existing CIDs that are already linked through the email  when (per spec) they are already unique, than to generate some other globally unique value (independent of CID) and then have to map every use of CID in email to this other value to then find index in cache.  Since the CID itself should be unique, and is in 99% of all cases, it is faster to just use the CID value.  While one could argue we should forgo performance for robustness, but we have found most customers care more about performance gain for the 99% case over the robust gain for the 1% case.
