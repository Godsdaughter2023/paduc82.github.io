john p spencer say:

If I understand what you want the SQL of the query would look like

SELECT *

FROM SomeTable

WHERE (DateFound is Not Null and DateReturned is Null)

OR Location is Not Null

In query design view enter

  • Is not Null under DateFound on the first criteria line
  • Is null under DateReturned on the first criteria line
  • Is not Null under Location on the second criteria line

That will return record that have a value for location

AND any records where DateFound has a value and DateReturned has no value (regardless if Location has a value or not.  IF you want all three conditions to apply then the criteria for the fields should all be on the same line.

ken sheridan say: Normally entering NOT NULL in the first criteria row of the column in design view would do the trick, with the criteria for the other columns also in the first criteria row (criteria in the same row implements a Boolean AND operation, criteria in different rows a Boolean OR operation).

If this isn't working then two possibilities spring to mind:

1.  If Location is a column of text data type then it could contain a zero-length string rather than be NULL.  In which case I'd suggest amending the table design, so that it accepts NULLs nut not zero-length strings.  You'd need to do this in stages:

1.1 Set the Location column's Required property to False (No) in the table design.

1.2 Replace all zero-length strings in the column with NULL by executing the following 'update' query:

    UPDATE YourTable
    SET Location = NULL
    WHERE Len(Location) = 0;

1.3 Change the columns AllowZeroLength property to False (No) in the table design.

2.  The other possibility is that the Location column's data type has been set with the 'lookup field' wizard when designing the table.  If this is the case then the data type is really a number which references the numeric primary key of a separate locations table.  The DefaultValue property of the column will very likely be 0 (zero).  So when you see 'blank' location in the table the value is in fact zero.  In this case you'd therefore need to set the criterion of the column in the query to 0.  This is one of the many reasons why 'lookup fields' are derided by experienced developers, and should be avoided like the plague.  For the gory details see:

http://www.mvps.org/access/lookupfields.htm

gabriel bratton say:

If you find search is not working for one specific POP account or PST file check the path of the PST data file and make sure it is correct.  In Outlook click File, Account Settings, Account Settings click Data Files.  Look under location and see if any of the paths are UNC based such as \\?\C:\users\username\Documents\Outlook Files\file.pst.  To fix this click Settings and see if you can manually fix the path.  If not, remove the PST and add it back.

Normally after fixing the UNC path and restarting Outlook search will start working again or shortly thereafter.  If the fixed path does not take affect in a half our or so go trigger a re-index and it should in some time, Search, Search Tools, Search Options, Indexing Options, Advanced, Rebuild.

What... there are people who are still using 13 year old software out there?<g> Yes, if someone is using XL97, then I would use the following modification to your 2nd macro (I prefer to use VB's built-in functions whenever possible as opposed to reaching out for a worksheet function) rather than create a Split function equivalent...

Sub getnums()
  Dim p1 As Long, p2 As Long, c As Range
  For Each c In Range("b1:b2")
    p1 = InStr(c, " ")
    p2 = InStr(p1 + 1, c, " ")
    c.Offset(, -1).Value = Replace(Mid(c, p1, p2 - p1), ",", "")
  Next c
End Sub

Note that in the macro I posted, if we assume the number being sought is always surrounded by spaces (as opposed to the assumption I made that the only guaranteed space was in front of the number), then we can eliminate the Val function call...

Sub GetNums()
  Dim C As Range
  For Each C In Range("b1:b2")
    C.Offset(0, -1).Value = Split(C)(1)
  Next
End Sub
