This is possible to a limited extent with XML Hacking. Here's my article on the subject: XML Hacking: Locking Graphics. This article assumes you are locking graphics, which is a more fruitful pursuit than trying to lock placeholders. You should also read Introduction to XML Hacking and Editing XML in OS X as background.

The issue with locks on placeholders is that the lock is not inherited from the Slide Master to the Layout, or from the Layout to the final Slide. Locks on placeholders are only effective when you provide a final slide in the presentation and apply locking to the placeholders on that slide. When someone generates a fresh slide from layout, the placeholders will not have any locks on them.

My article uses the noSelect parameter. This page shows a list of other typical locks that can be applied: a:picLocks This list will vary somewhat depending on the actual placeholder type that a lock is being applied to.

Running a new O365 E3 pilot with a few users. Hybrid connection to on-prem Exch2013 is working for mailbox moves, free/busy, AAD Sync working fine.

We use Mimecast for our inbound/outbound filtering as well as journaling, so I have an oubound connection in O365 pointing to Mimecast's SMTP server (in addition to the "to on-prem" and "from on-prem" that are created by default). Due to this arrangement, our MX records still point to Mimecast as per their instructions (we don't want incoming mail hitting mail.protection.outlook.com, want it it routed through Mimecast).

For my O365 Exchange Online hosted test user, both outbound and inbound messaging for internet email is working fine (i.e. Gmail, yahoo, etc.)

However, when that hosted user tries to send a message to an internal, on-premises Exchange user on my own domain, instead of Exchange Online using the connector for "Office 365 to your organization's email server" the message is being routed out to Mimecast via the connector. Mimecast, of course, detects this as someone spoofing my domain and rejects it.

My question is, shouldn't O365 Exchange know which connector to use for my domain, and route messages directly to my on-premises Exchange 2013 server, which it's been communicating with just fine?

johnm121 say:

Issue resolved -- two steps required:

- Set my domains to Internal Relay, instead of Authoritative

- Change the Connector for O365 -> On Prem to use my server's IP address instead of FQDN. Even if the FQDN of the Exchange server is specified, O365 will still do an MX lookup on the domain portion, and route mail solely based on that. Highly questionable programming decision, but we got it to route mail back on-prem with these two changes.
