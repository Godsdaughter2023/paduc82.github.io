scott mcdaniel say:

You can set the RecordSource of the Subform directly, or you could Filter it.

To set the Recordsource, first build up a valid SQL string that uses a WHERE clause to filter your data. For example:

Dim sql As String

sql = "SELECT * FROM Leads WHERE Field1='" & Me.cboMobile1 & "' AND Field2='" & Me.cboMobile2 & "'"

Me.YourSubformCONTROL.Form.Recordsource = sql

To use the Filter property:

Me.YourSubformCONTROL.Form.Filter = "Field1='" & Me.cboMobile1 & "' AND Field2='" & Me.cboMobile2 & "'"

Me.YourSubformCONTROL.Form.FilterOn = True

Note that YourSubformCONTROL is the name of the Subform Control on the parent form, and may or may not be named the same as the form you're using as a Subform.

If you want to check for the existence of data, you can use the RecordsetClone method:

Dim rst As DAO.Recordset

Set rst = Me.YourSubformControl.RecordsetClone

rst.FindFirst "Field1='" & Me.cboMobile1 & "' AND Field2='" & Me.cboMobile2 & "'"

If rst.NoMatch Then

  '/ the record was not found

  MsgBox "No Data Returned"

Else

  '/ records were found, do something else here.

End If

To Rest the subform:

If you used the Recordsource method, clear the WHERE portion from the SQL statement:

Me.SubformControl.Form.Recordsource = "SELECT * FROM Leads"

If you used the Filter method, just set Me.YourSubformControl.Form.FilterOn = False

scottgem say:

Actually this makes very little sense. 

You are using the Comboboxes to populate a filter for your table, right? So then why would you want to be able to enter something that is not a valid value? Think about it! If I had a US State field in a table and that field was populated from a table of States so it stored the state abbreviation, would I want to be able to type in ZX (not a valid state abbr) since I know there would be no matching records?

One of the purposes of a combobox is to restrict data entry to valid data. 

Now if your data is bad and you do have values that are not in your Rowsource, change your RowSource. Make it lookup against the field rather than the lookup table. For example:

SELECT DISTINCT fieldname

ORDER BY Fieldname;

I did see one possible issue with your Rowsource. You have:

SELECT DISTINCTROW Inquiries.*

Not sure why you are using DISTINCTROW or .*, you want to select a field value to search on, so should only have one or two fields in the Rowsource depending on structure (either the FK value and a description or just the description).

doug robbins - mvp office apps & services (word) say:
You need to go to the Primary Header/Footer of each section, which is the one that will appear on the second page and insert the page number field on that Header/Footer.   The First Page Header/Footers are not related to the Primary Header/Footers and can have completely different content.

suzanne s. barnhill say:
When you enable Different First Page for a given section (and it must be done separately for each existing section), you get a new, separate First Page Header/Footer for each section. By default these will be empty, so you don't actually have to do anything. Don't remove numbering from the primary Header/Footer, which will be used for the remaining pages in the section.
