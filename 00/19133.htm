Lets take for example this scecific form. Basically it is form with a subform. The fields of the main form belong to one table which is called PRF_Table (that is why i named the main form PRF_Table_Main_Form) and all the fields of the sub form belong to a table called Purchase_Table

Well, it's not a question about "efficiency" in terms of execution -- as long as Access accepts the name and it doesn't conflict with any other object, one name will be handled as efficiently as another, from an execution standpoint.  Where naming conventions become important is in making things easier on the developer, including any future developers who may work with this database.  So a good naming convention makes it easier for you, now and in the future, to do development work on this database.

Therefore:

  1. Whatever naming convention you use has to make sense to you, first of all.  In this consideration, you can be as idiosyncratic as you like -- if no one but you will ever work on this database.  If any one else will work on the database, or even if you expect to ask questions about it in online forums <g>, then you would do well to follow, at least loosely, one of the generally accepted conventions.  That saves time.

  2. Since spaces and other punctuation characters in object and field names make you work harder when you refer to them in code -- you have to enclose them in square brackets ([]) -- you should avoid them.  Then you don't have to do the extra work.

  3. The underscore is one common way to avoid using spaces in names, to separate meaningful words.  Instead of naming a field, for example, "Customer Name", this approach would name it "Customer_Name".  There's nothing wrong with that (though it's not what I do -- see below).

  4. Another way to compose names is remove spaces, but capitalize the first letter of each word.  With this approach, "Customer Name" becomes "CustomerName".  This is what I do.  There are a few situations where the resulting name might be unreadable, in which case I resort to using underscores.  For example, while "CustomerID" is a good field name, for a purchase-order ID I might use "PO_ID" instead of "POID".  Or I might not; it depends on whether I think it's going to be an issue.

  5. Aside from the construction of field names, we often have to distinguish between different types of objects.  You are doing this when you have table names ending in "_Table" (e.g., "PRF_Table" and "Purchase_Table"), and form names ending in "_Form" (e.g., "PRF_Table_Main_Form").  I personally find the lengthy suffixes more trouble than they're worth, and user shorter prefixes instead.  There are commonly used prefixes for Access objects:

    • "tbl" - tables
    • "qry" - queries.  I enhance this to "qry" for select queries, "qupd" for update queries, "qapp" for append queries, "qdel" for delete queries, "qmk" for make-table queries, etc.  But that's just me -- I find it helpful to make that distinction.
    • "frm" - forms -- also "sf" for forms intended to serve as subforms
    • "rpt" - reports
    • "mac" - macros
    • "mod" - standard modules
    • "cls" - class modules


  So "tblPurchases", "frmPurchases" (a form based on tblPurchases), "qryPOUnpaid" (maybe), etc.

There are also relatively standard standard prefixes for control names:  "txt" for text boxes, "cbo" for combo boxes, "lst" for list boxes, "cmd" for command buttons, "btn" for radio buttons, and so on.  However, for controls that are bound directly to a field in the form's recordsource, and which are not manipulated in any special way in code, I see no harm in having the control named the same as the field. Some people insist that you shouldn't do this, but since Access will automatically name the controls for the fields if you let it create the form from the table or create the control by dragging it from the field list, and I see no point in going through renaming everything.  I only rename the controls when I need to distinguish between the control and the field.

There's also a point of view that says you don't prefix table names;  just use, e.g., "Purchases" instead of "tblPurchases".  The fact that the name has no prefix means that it's the name of a primary data entity.  Nowadays, I follow this rule, but once I didn't, and not everybody does.  Reasonable people can disagree.  Again, what matters is what works for you, what makes your life easier, and what will be comprehensible to anyone coming along later to work on your database.
