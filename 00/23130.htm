ken sheridan say: I don't see any need for separate queries for this.  If the expiry period is a constant of 8 months, then for each Qual date you can easily compute the number of months from the current date until the expiry date with the following simple expression:

8-DateDiff("m",Qual,Date())

If we enter it with some literal dates in the debug window:

Qual = #2017-05-21#
? 8-DateDiff("m",Qual,Date())
 1

The above returns 1 because the expiry date of 21 January 2018 is 1 month ahead of today, 20 December 2017.

If we advance the qualification date by 3 months:

Qual = #2017-08-21#
? 8-DateDiff("m",Qual,Date())
 4

We now get a value of 4 returned, because the expiry date of 21 April 2018 is 4 months ahead of today, 20 December 2017.

If we move the qualification date back further so that the expiry is imminent:

Qual = #2017-04-21#
? 8-DateDiff("m",Qual,Date())
 0

We get a value of zero, because the expiry date is in the current month.  The person in question is still qualified, however, albeit by only I day.  If we move the qualification date back further so that the person's qualification has recently expired:

Qual = #2017-04-10#
? 8-DateDiff("m",Qual,Date())
 0

We again get a value of  zero, because the expiry date is in the current month, but in this case the qualification has expired by 10 days.  So, using this simple expression it's not possible to distinguish between someone whose qualification has recently expired, and someone whose qualification will imminently expire.  This is because the DateDiff function works on a month-on-month basis, and takes no account of the day of the month.  We can distinguish between the two, however, by calling the following function, which was written to return the age of infants in months, but will serve here.  

Public Function GetAgeMonths(varDoB As Variant, Optional varAgeAt As Variant) As Variant

    Dim intMonths As Integer

    If Not IsNull(varDoB) Then
        If IsMissing(varAgeAt) Or IsNull(varAgeAt) Then varAgeAt = Now()
        
        intMonths = DateDiff("m", varDoB, varAgeAt)
        
        If Day(varDoB) > Day(varAgeAt) Then
            intMonths = intMonths - 1
        End If
        
        GetAgeMonths = intMonths
    End If
    
End Function

So, if we repeat the last two operations, but call the above function rather than the DateDiff function:

Qual = #2017-04-21#
? 8-GetAgeMonths(Qual,Date())
 1

Qual = #2017-04-10#
? 8-GetAgeMonths(Qual,Date())
 0

We now get a value of 1 returned where the qualification will imminently expire, but a value of zero where it has expired.  So in this case to identify rows where the qualification has expired or will expire within two months a single query can be restricted to:

WHERE 8-GetAgeMonths(Qual,Date()) <=2

A further simple function can return the expiry status for a given Qual date and expiry period in months:

Public Function QualStatus(dtmQual As Date, intExpiryPeriod As Integer) As String

    Select Case intExpiryPeriod - GetAgeMonths(dtmQual, VBA.Date)
        Case Is <= 0
        QualStatus = "Expired"
        Case 1
        QualStatus = "Expires within one month"
        Case 2
        QualStatus = "Expires within two months"
    End Select
    
End Function

You can see how it works in the debug window:

? QualStatus(#2017-04-10#,8)
Expired
? QualStatus(#2017-04-25#,8)
Expires within one month
? QualStatus(#2017-05-25#,8)
Expires within two months

So the query would call the function in a computed column like this:

QualStatus(Qual,8) AS QualificationStatus

A report based on the query could then be grouped on the computed QualificationStatus column.

This is possible. Not sure why Microsoft made it a little more difficult in 2013, but that is another discussion.

You can have any number of your own templates & themes displayed in the Personal Tab (once its displayed).

Very useful for organizations with standard themes / who will never use the off the shelf / featured themes.

Taken from a combination of http://www.poremsky.com/office/use-office-2013-local-templates/ & http://www.indezine.com/products/powerpoint/learn/themes/2013/change-default-template-theme.html

1) Set your default personal templates location (PowerPoint Options - Save - Default personal templates location)

  •     Save all of your templates & themes in that folder.
  •     Name the themes as "____ theme" and templates as "____ template" as they aren't distinguishable from each other on the start screen ;)

Once the folder location is set the "Personal" tab will show up on the PowerPoint start screen with all your templates

2) Set the Personal Tab as the default (useful for organizations with Themes/ Branding standards, which I suspect is most organizations)

Open the registry editor and modify (or add) the following key in the registry.

          HKEY_CURRENT_USER\Software\Microsoft\office\15.0\PowerPoint\options
          DWORD: officestartdefaulttab
          Value: 1

Create PPTX / PPT programatically - Microsoft Community

michal golda say:

Create PPTX / PPT programatically

I would like to create programatically presentation based on data which I will get from Excel (PowerPivot)

Is it possible with PowerPoint object model (Microsoft.Office.Interop.PowerPoint).

shyam pillai say:

Basic example of automating it from Excel.

Dim oPPT As Object 'PowerPoint.Application
Dim oPres As Object 'PowerPoint.Presentation
Dim oSld As Object 'PowerPoint.Slide

Range("A3:B25").CopyPicture

Set oPPT = CreateObject("PowerPoint.Application")
If Not oPPT Is Nothing Then
    oPPT.Visible = True 'make the window visible (optional)
    Set oPres = oPPT.presentations.Add(msoTrue)
    With oPres
        Set oSld = .Slides.AddSlide(1, .Designs(1).SlideMaster.CustomLayouts(1))
        With oSld
            With oSld.Shapes.Paste(1)
                .Top = 10
                .Left = 10
            End With
        End With
        .SaveAs "c:\temp\test.pptx", ppSaveAsDefault
    End With

Else
    MsgBox "PowerPoint object could not be instantiated.", vbExclamation
End If
