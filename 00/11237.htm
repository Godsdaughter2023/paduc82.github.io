Note that you can never be sure that field numbers and cross-references are correct in a document where you are tracking changes. macropod say:

Hi Anj J,

Try running the following macro. It accepts any tracked changes affecting fields.

Sub AcceptTrackedFields()
'This sub accepts any tracked changes affecting fields
Dim Story As Range, oFld As Field, oRev As Revision, Rng As Range
With ActiveDocument
  ' Loop through all range objects and accept tracked changes on fields
  For Each Story In .StoryRanges
    For Each oRev In Story.Revisions
      For Each oFld In oRev.Range.Fields
        oFld.ShowCodes = True
        Set Rng = oFld.Code
        With Rng
          .MoveEndUntil cset:=Chr(21), Count:=wdForward
          .MoveEndUntil cset:=Chr(19), Count:=wdBackward
          .End = .End + 1
          .Start = .Start - 1
          oFld.ShowCodes = False
          .Revisions.AcceptAll
        End With
      Next
    Next
  Next
End With
Set Rng = Nothing
End Sub

    SubSite "Limited Access" nightmare. - Microsoft Community

    a. user say:

    SubSite "Limited Access" nightmare.

    Hello,

    I am in the process of creating a subsite for an organization which is geared for issue tracking. The way the subsite is set up, is to have dataviews to different team subsites within the Site Collection as a hub for viewing issues that need to be resolved. The problem I have is, the site owner is set to "Full Control, Limited Access". I don't want the user to be assigned to Limited Access, and I have been going through every library, list, page to find what is causing this to happen but I can't seem to find anything.

    Is there something I am missing here? I need this user to be able to access the pages on the entire Subsite, but this user keeps needing to request permission to access the page and documents. It is really frustrating, due to the whole design of the SharePoint permissions system makes it very difficult to see a full mapping of what causes what to happen.

    I went to the main Team Site and disabled the "Limited Access Lockdown Mode" Collection Feature; but that allows Anonymous external users to view the site. I do not want external users in the site, so I have to keep that one on.. If anyone has a viable solution that does not involve me to pull down the entire subsite and re-design it, that would be much appreciated.

    Thank you.

    a. user say:

    I fixed it.

    After managing permissions and seeing what different directories are giving the "limited access", I have determined that it was the Home Page that was causing the "limited access" to happen for the owner. Even though the dashboard did *not* inherit the Home Page permissions, it was still denying access to the dashboard. After editing the Home Page in SharePoint Designer, I was able to delete the Home Page + .aspx file then create a fresh one which did not have this issue. The site is only accessible to the assigned members and all pages are accessible to them with *no* anonymous access rights.
