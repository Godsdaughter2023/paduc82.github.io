brook zhang msft say:

Hi MenR2004,

 

Usually, if we deploy AD FS, we will not sync local password to Office 365.  The account authentication occurs in local AD. Thus, online passwords will not be involved. That’s why we usually don’t sync passwords. It’s also why a password file will be generated automatically for online account when we change a domain from Federated to Managed. These passwords are assigned to online users directly (not through directory synchronization)

 

Then we need to reconfigure Azure AD Connect to enable the password synchronization feature to sync local passwords to online.

 

It’s not possible to manually set these passwords for online users when changing a domain from Federated to Managed.

 

If you want to manually set passwords, set them in your local AD and then let Azure AD Connect sync them to Office 365.

Brook

ran mo msft support say:

Hi Arjan,

First, what you plan to do (multi ADFS farms, each in every AD forest) is correct as if there is only one way trust between forests, there must exist ADFS farm in every AD forest. You cannot only use one farm in this scenario. 

And for the "set-msoladfscontext" question, this cmdlet will not affect other domains. This cmdlet is used when you need to connect to ADFS server if Azure AD powershell is not install in the ADFS server. Otherwise, you don't need run this cmdlet.

For the second question,  Convert-MsolDomaintoFederated -DomainName ccc.local -SupportMultipleDomain will not affect other domains. For other domains to be completely moved to ADFS, you just need to run the above cmdlet for every domain.

In nearly all of the cases I've seen the Edge popups that display false warnings aren't malware, they're simply a modal dialog (e.g. "announcement box") that's usually easily disabled by selecting the checkbox Microsoft added to block such dialogs from that specific website in the future.

greginmich say:

You don’t need to touch the registry in order to enable PUA Protection; you can just copy, paste, and enter this command at the elevated (administrator) PowerShell prompt:

Set-MpPreference -PUAProtection 1

Or you can use the REG file available here, which might very well predate the Winhelponline write-up:

http://www.tenforums.com/tutorials/32236-windows-defender-pua-protection-enable-disable-windows-10-a.html

Like it or not, Windows Defender is being revamped for the enterprise environment, and its settings are only available via the PowerShell Set-MpPreference cmdlet and/or Group Policy. Since there’s no Local Group Policy Editor in the Home version of Windows 10, Group Policy settings sometimes have to be applied directly via the registry:

https://technet.microsoft.com/en-us/library/dn433291.aspx?f=255&MSPPError=-2147217396

But enabling PUA Protection won’t help even a little bit when it comes to the typical fake security/virus warning webpages that lock up browsers. The Dialog Loop Exploit that’s used by these Tech Support Scam webpages has probably been the most prevalent, disruptive, and costly malware ever, and it’s been with us for several years now – so the question of why this script-based form of malware hasn’t been adequately dealt with is totally legitimate. The overwhelming prevalence and destructive impact of these browser-locking Tech Support Scam pages should be readily apparent to anyone who follows this forum; and savvy web developers were already warning Edge’s developers about the misuse of the showModalDialog method prior to the release of Windows 10 – but Dialog Loop Protection was only implemented for Microsoft Edge (but not for Internet Explorer) belatedly, when it became painfully apparent that Edge’s AutoRecover feature would bring one of these Tech Support Scam pages right back up again when the browser was restarted after being killed with Task Manager. Dialog Loop Protection was something that had already been implemented in other browsers, so it was easily adopted for Edge. But note that this kind of protection doesn’t actually block specific websites from generating modal alerts – it only stops the malicious script that’s currently running on the scam page and repeatedly calling the modal dialog. Stopping this script prevents the dialog from being reloaded, and thus unlocks the browser.

On top of being ever-so-slow to respond to this exploit, there was never any attempt on the part of Microsoft to explain or publicize Edge’s workaround (I don’t consider it a fix) for these malicious scam pages – only a brief changelog announcement for developers. And since I was the only person in this forum who was following this issue closely, getting the word out about Dialog Loop Protection somehow became my own personal responsibility, e.g.:

https://answers.microsoft.com/en-us/protect/forum/protect_other-protect_scanning/system-error/dcccc74e-fb98-43a8-8579-956692a40982

But if we now revisit some of the major threads dealing with this issue, it’s pretty obvious that my efforts were mostly in vain, and that the participants are still completely in the dark about Dialog Loop Protection, despite my pronouncements, e.g.:

https://answers.microsoft.com/en-us/protect/forum/protect_other-protect_scanning/edge-frozen-with-virus-warning-popup/7212a393-90ad-48d0-ab55-4ffe1f21a554

So I can’t help but wonder if maybe some of the blame for this glaring failure might be attributable to the fact that this huge and phenomenally costly scam has somehow never managed to merit even a single sticky thread here in the Answers forum. Anyway, I’m personally still waiting for the Dialog Loop Exploit issue to be fixed in a way that’s completely reliable, and that doesn’t depend on any kind of user interaction whatsoever. The way it stands now, we can look forward to many more years of having to struggle with new variants of the Tech Support Scam, and with the impossible task of educating Windows users about how to properly identify and handle script-based malware.

GreginMich
