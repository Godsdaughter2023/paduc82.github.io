Excel 2013 Countif with grouped worksheets - Microsoft Community

leonard704 say:

Excel 2013 Countif with grouped worksheets

Should countif work with grouped  worksheets range?  My range includes 1 cell from each spreadsheet, but I get a value error each time I try the formula:  All cells in the Formula range contain numbers.

=COUNTIF('Data Sheet:3rd Data Sheet'!J14,1)

The formula is in a summary worksheet in the same workbook.

Thank you.

jeeped say:

No, I'm afraid not. COUNTIF, SUMIF, AVERAGEIF (and their ...IFS companions) do not work in ranges across worksheets.

Your options are,

  • Stacking a series of COUNTIF formulas together with SUM or +
  • Using INDIRECT as an array of worksheet names
  • Using a VBA based User Defined Function (aka UDF)
  • Use COUNTIF.3D from the MoreFunc Excel add-on. If you are interested in this, a safe copy is available from my OneDrive here.

Regarding the second option if you put Sheet1, Sheet2, Sheet3 in Z2:Z4, the COUNTIF for values greater than 0 across column G in those three sheets would look like this.

=SUMPRODUCT(COUNTIF(INDIRECT(Z2:Z4&"!G:G"), ">"&0))

If your worksheet names have spaces, you will have to wrap them in single quotes like 'Sheet One'.

braintrove.com say: In the DOM you are dealing mainly with Elements and Attributes. If you are familiar with HTML, the tags are elements and their properties are attributes. So in <img src="xyz"/> img is the element and src is an attribute of the img element.

In your schema you can create elements and attributes on those elements. If you have an element named "Field", you can add an attribute on it named "user". In the XML it would look like this:

<Field user=""/>

When Field changes, a rule on Field would set the user attribute with the value of the current user.

A formatting rule on Field would disable the control if the user value is not blank and is not the current user. Why both conditions? Because if the value is blank, you want the field editable. And if the value is the current user, you want the field editable.

If you have a lot of fields in your view, this can become quite tedious to go field by field. Code would solve it but introduce a new level of complexity. There might be another way, but I don't have any time now to research whether there is.

You cannot freeze the dimensions of a cell independently of the height of the cells in the same row and independently of the rest of the table.

Setting the row height to an exact figure will freeze the height of all of the cells in that row and removing the check mark from the "Automatically resize to fit contents" in the Table Options dialog, will lock the width of all of the cells in the table.

You access these features via the Properties dialog in the Table section of the Table Tools>Layout tab of the ribbon.  The row height is set via the Row tab of the Table Properties dialog and the Table Options dialog is accessed via the Options button on the Table tab of the Table Properties dialog.

You will need a macro to do that:

Sub RemindMe()
Dim olItem As MailItem
Dim strDate As String
Dim strTime As String
    If Application.ActiveExplorer.Selection.Count > 0 Then
        strDate = InputBox("Enter date", "Reminder Date", Format(Date, "Short Date"))
        strTime = InputBox("Enter time", "Reminder Time (24 hour clock)", Format(Time, "HH:MM"))
        For Each olItem In Application.ActiveExplorer.Selection
            With olItem
                .ReminderSet = True
                .ReminderTime = strDate & Chr(32) & strTime
                .Save
            End With
        Next olItem
    Else
        MsgBox "No mail items selected", vbCritical, "Reminder"
    End If
lbl_Exit:
    Set olItem = Nothing
    Exit Sub
End Sub
