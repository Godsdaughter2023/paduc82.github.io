tom van stiphout say:
Auch, I was afraid of that when I saw the Excel file.
Access is not Excel on steroids.
Rather it is a Relational Database, and you must follow the rules of normalization that come with that. There is no "I'll do that later" or "That does not apply to my simple db".
Unfortunately we are not born with this knowledge. There is no need to be embarrassed about that. I highly recommend that you stop development of your database and first increase your knowledge about relational databases and normalization. There are many books, web pages, videos related to this topic.
You will learn every table must have a PK. Tables are related to each other and those relations must be enforced so the database engine can enforce referential integrity (the ones you have are just notational niceties without teeth). Repeating groups (11 sizes, 11 solids, 11 pack1pack,... pack5pack, and also tblSizeRange, tblSeasons) cannot be allowed and need to be spun off into child tables.
Thinking about orders in particular: in most order entry systems we see both an Order table, and one for the order line items. Check out the Northwind sample database for an example.

groverparkgeorge say:

Tom is quite right. 

While one can, in the first step, get away with importing Excel files into "staging tables" in Access that have a similar structure as the source Excel file, eventually a properly designed relational database will require parsing that temporary, staging table into the required relational tables. 

In the last post in this thread, you mention "...minor changes to that table " and that reveals the source of the problem. Changing a destination table means that the source and destination layouts no longer match and now you'll have to make adjustments to either the source file, the destination table, or both to restore the compatibility. 

And then the important work begins, parsing the data out of that temporary, staging table in Access into the appropriate relational tables. We'll be happy to help you with both phases, getting the source and destination layouts back into alignment, and then correctly moving the data from that staging table into its final home.
