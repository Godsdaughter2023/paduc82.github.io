diane poremsky mvp (slipstick.com) say:
Sounds like you can't use a formula then (unless the subfolders are the recipients name or valuesi n other cells) - a macro using FSO could walk the folders. But it will need the same basic continuity - it needs to be able to build the path and know where to put the filepath.

This example is a basic macro that writes all files in a folder to a column. if the sibfolder name can be picked up from the entry in the workbook.
Sub CreateLinks()
Dim objFSO As Object
Dim objFolder As Object
Dim objFile As Object
Dim i As Integer

Set objFSO = CreateObject("Scripting.FileSystemObject")
Set objFolder = objFSO.GetFolder("D:\Documents\")
i = 1
'loop files in the directory
For Each objFile In objFolder.Files
'create hyperlink in selected cell
Range(Cells(i + 1, 1), Cells(i + 1, 1)).Select
ActiveSheet.Hyperlinks.Add Anchor:=Selection, Address:=objFile.Path, TextToDisplay:=objFile.Name
' next file, next cell
i = i + 1
Next objFile
End Sub

ken sheridan say: Set the combo box up as follows:

Name: cboGoToClient

RowSource:     SELECT ClientID, LastName & ", " & FirstName FROM Clients ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

In the combo box's AfterUpdate event procedure put code along these lines:

Dim rst As DAO.Recordset
Dim strSQL As String

strSQL = "SELECT * FROM Clients WHERE ClientID = " & Me.cboGoToClient

Set rst = CurrentDb.OpenRecordset (strSQL)

With rst
    Me.txtClientID = .Fields("ClientID")
    Me.txtFirstName = .Fields("FirstName")
    Me.txtLastName = .Fields("LastName")
    ' and so on
End With

However, selecting a person purely by name is not really satisfactory, as personal names can be duplicated (I worked with two Maggie Taylors).  To differentiate between them, further details can be shown in another column or columns of the combo box's list, e.g. their address data.  For an example take a look at  NotInList.zip in my public databases folder at:

ken sheridan say: The combo box should have as its bound column the ClientID, and the concatenated name as its second column.  Personal names should never be used as keys.  The search should be undertaken on the basis of the distinct ClientID, not the name, as I described in my last post.

What you show in the form need not correspond to the values at the column positions in the recordset.  You could show the concatenate name in a single text box, e.g.

    Me.txtFullName = .Fields("LastName") & ", " & .Fields("FirstName")

Why are you using an unbound form in the first place?  If the recordsets are based on the same columns from the same table(s), but  differ by some restriction (such as a case being open) then you can simply amend the form's RecordSource property in the code behind your buttons, and use the unbound combo box to navigate to the selected Client by synchronizing the form's bookmark with that of its RecordsetClone in the usual way.

If the recordsets are radically different, then you can use an unbound parent form containing the buttons, the unbound combo box, and a subform control.  The subform control's SoureObject property can then be changed in the code behind the buttons.  To navigate to the selected client in the subform, you can either filter it, e.g.

    Me.SubformControlName.Form.Filter = "ClientID = " & Me.cboGoToClient
    Me.SubformControlName.Form.FilterOn = True

Or navigate to the selected client in the subform, e.g.

    With Me.SubformControlName.Form
        .RecordsetClone.FindFirst "ClientID = " & Me.cboGoToClient
        If Not .RecordsetClone.NoMatch Then
            .Bookmark = .RecordsetClone.Bookmark
        End If
     End With

BoundColumn:    1
