Hi Jetta48,


You've asked an important question, but as is typical in such cases are mixing in other attributes that are less related than they might seem and ignoring others that are far more important.


First, we must break down the differences between the various operating systems you've mentioned and other only slightly related items like the firewall.  Note that your statements are lumping both the Windows 8.x and Windows RT based operating systems together, while these systems are both far different from anything Microsoft has produced in the past.


The Windows 8.x are still a general purpose operating system like those before them, though in terms of security they are also far more inherently secure than these earlier versions.  As with Windows 7 and Vista before it, Windows 8 has taken great strides in securing some of the major attack vectors used against these older versions, as discussed in the following "Protecting you from Malware" article in the Building Windows 8 series.

http://blogs.msdn.com/b/b8/archive/2011/09/15/protecting-you-from-malware.aspx


Though this can't close all of the holes that exist for malware and nothing can, it goes a long way towards shutting down some of the common techniques used by current malware or at least limiting the ability to exploit them effectively.  Also note that this version of Defender can be easily disabled by a properly designed and installed 3rd-party antimalware product, so there's really nothing changed here other than the fact that Defender doesn't require a separate installation.


Addressing the Windows RT (Run-Time) version requires a completely different set of criteria, since this operating system is radically different in several ways.  First, it operates on a completely different hardware platform (ARM) than the original Windows, so none of the native code written for the earlier platforms can operate on it.  Next, it doesn't provide support for either the Win32 or .NET application platforms, so none of this code can be easily ported either.  Finally, it limits the core applications like video and music to those already built-in, so this reduces the chances that 3rd-party applications can create gaping holes that bypass core protections of the operating system.


As for the security in Windows RT, the version of Windows Defender used there is as radically different as the operating system, since it must support an entirely different ARM processor and tiled applications.  Both Windows RT and Defender are stored in firmware/SSD and Windows Update is always on, so nothing including user installed applications is allowed access to or control over either the operating system or Defender.


This limits the ability for malware that can directly attack the operating system, though in theory a vulnerability in a key component like Internet Explorer or Windows RT itself might be used to attack it, though obviously there are similar protections for things like buffer overflows, etc. that Microsoft has ported to Windows RT and doesn't talk about, so these might mitigate many of these attacks as well.


Tiled (were METRO) applications are only installed via the Windows Store, which gives Microsoft a choke point to examine and disable distribution of any rogue applications found to be acting like malware.  I should also mention that these applications are all sandboxed which limits what they may do to each other, though the user can increase these abilities to some extent, so social engineering could still be at least a partial problem for privacy.


The firewall is really a completely unrelated item, since it exists only to block traffic either based on port configurations or possibly upon certain malicious detections by Defender, since both use the Windows Filtering Platform (WFP) in Win32 based Windows 8 to monitor networking traffic as virtually all 3rd-party firewalls do as well.  We know less about the WinRT platform firewall, since these components would only require some basic level of user control but otherwise be under operating system control for most typical purposes, so I won't try to discuss that here.


Do note though that antivirus or antimalware are not the first line of defense, they are the last and only need to act when all other forms of protection; firewall, patching, security features or other methods have failed and the malware has gotten past them.


Finally, as for your primary concerns relating to the diversity of 3rd-party antivirus and the potential for an increased risk as these products disappear, your initial premise seems reasonable.  However, some quick examination of the facts indicates that there are extenuating circumstances for most of these.


First we need to understand that Microsoft initially produced first Windows Live OneCare and then Microsoft Security Essentials to solve the problem that many (approximately 70% was the original estimate) of the consumer and small home business users were not running a functional antivirus for reasons such as cost.  This has been largely resolved by offering a free "basic" protection program in MSE that removed this issue.

http://www.microsoft.com/en-us/news/press/2008/nov08/11-18nocostsecuritypr.aspx


Next, the simple fact is that other than the above users who never paid for their antivirus, many of the consumer and small business users have already moved on to more mobile devices like tablets or smartphones, leaving their PC use much reduced if not non-existent as time wears on.  This is a part of the reduction in PC sales in the US and the world at large, as the decreased cost and increased mobility provided by these devices is offset by their increasing utility for light-duty applications like email, music, video and anything web browser based.


So the reality is that MSE and now Defender on Win8 isn't the only and is likely one of the least of the pressures on the various 3rd-party security vendors.  Anyone smart enough to see this coming would see the necessity for Microsoft to prepare for the eventuality that most if not all of the 3rd-party vendors will exit the business when it becomes unprofitable.  So having a decent antimalware product already in place on the current systems which those wanting alternatives can easily replace only makes sense.


There are likely other factors at play here that I'm missing, but these seem like the most obvious that may have been involved in the current situation.

Your interpretation of the AV-testing lab data is once again in error, Jetta48. There isn’t anything like a “minefield of opinions and subjectivity” in the to a more representative, “real-world”, testing methodology.


Of course there’s always going to be a disparity in the relative rankings of the AV participants because the products themselves are in a constant state of developmental flux, and also because the threat landscape is continuously evolving. Each individual test is just a snapshot of an ever-changing scenario, and the different outcomes are to be expected as a result of the changing environmental factors – although, obviously, using a different camera will also color the results to some extent. But there really is a “minefield of opinions and subjectivity” out there, and it’s always to be found in the opinions of people who reject scientific testing methods, and insist that their own personal observations and/or the firsthand accounts of others (these are “subjective” in the accepted definition of the term) are the best source of information on the subject. Unfortunately, if your opinion of AV products is still dominated by that awful experience that you had with Norton Internet Security 2008, then you’re living in the past.


When we look at an individual data set, we also observe a “clustering factor” due to broad similarities in detection methodologies and mutually shared data. But we always observe a highly significant spread in the effectiveness spectrum (distribution of detection/remediation scores varying from worst to best), just because the differences in detection and remediation strategies are substantial, and are highly significant factors in determining the effectiveness of each individual AV product. Logic and common sense will only tell us that divergent methods will always result in divergent levels of effectiveness, but the empirical test data tell us where those different detection/remediation strategies actually fall on the “effectiveness” scale at any specific point in time. Not with perfect accuracy, to be sure; but with nothing like the “minefield of opinions and subjectivity” that we see in opinions that are dominated by anecdotal accounts from a world that no longer exists.


The key to appeasing the competition, and also to promoting a diverse and sustainable AV marketplace, was Microsoft’s apparent willingness to pull its AV entries out of the competition. Hence the direct relevance of the question of how well Microsoft AV products perform under test: If the built-in AV product performs exceptionally well, then it would be expected to have a disruptive effect on the marketplace. If it performs very badly, then it would be expected to have a disruptive effect on consumer confidence. Maybe performing at a “baseline” level seems to provide the proper balance between these two rather unpleasant alternatives.
