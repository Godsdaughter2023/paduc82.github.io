john korchok say:

Automatic functions in Word can be rigid in their requirements, and this is one example. Word expects endnotes to be at the end.

To work around this, please try these steps:

  1. Go to the end of the document and switch ro Draft View (View>Draft). This hides the endnotes.
  2. Insert a Section Break (Next Page). This new section will eventually become the spot for your additional content.
  3. Using Insert>Footnote, set the endnotes to appear at the end of each section instead of the end of the document.
  4. In each section except the second-last, choose Format>Document and select the Layout tab. Check the box beside Suppress endnotes. This should force all endnotes to appear at the end of the second-last section. 
  5. Add your author's page in the last section.

doug robbins - mvp office apps & services (word) say:

You will need to insert a Section Break to separated the body of the document from where you want the author's page, and then in the Footnote and Endnote dialog, select End of Section as the location for the Endnotes.

If the body of the document contains multiple Sections, you will need to position the selection in each of those Sections (Except for that before the Section which is to contain the author's page) and then on the Layout tab of the Page Setup dialog, check the box for Suppress endnotes

gord dibben say:

Try this macro I found on the 'net

Sub ShadeRows()
    Dim ThisOrder As Long
    Dim PrvOrder As Long
    Dim LastRow As Long
    Dim Clr As Integer
    Dim R As Long

    LastRow = ActiveSheet.Range("D" & Rows.Count).End(xlUp).Row

    ' Enter desired color codes here
    ' (15 is gray, -4142 no color)
    RwColor = Array(15, -4142)

    Clr = 0  ' Used to toggle between the two colors

    For R = 2 To LastRow
        ThisOrder = Cells(R, 4).Value
        PrvOrder = Cells(R - 1, 4).Value
        If ThisOrder <> PrvOrder Then Clr = 1 - Clr

        ' Select only the columns that are used
        Range("A" & R & ":K" & R).Select
        Selection.Interior.ColorIndex = RwColor(Clr)
    Next R
End Sub

Accessing SharePoint files in windows, without storing locally.

Hi all,

I have been searching for a security specific solution to access SharePoint files from a local application (ideally file explorer), while avoiding storing 365 credentials/files locally.  The goal here would be an improved user interface, as opposed to SP online, while still keeping data stored in the cloud.  I realize that while accessing files they may temporarily be stored locally, but the goal is to avoid data persistence on the local machine.  Below are some of the methods I have looked into, and rationale as to why they don't fit:

1) SharePoint in file explorer via OneDrive sync client

This is probably the ideal user experience, as they are able to easily access files, see their formats, navigate folders, open/edit/share/save in a familiar capacity, and seamlessly push changes to the cloud. That being said, this is not a viable option as 1) The risk of files being stored locally is too great 2) Even if files are pushed to the cloud via the "Free up space" option, they can easily be retrieved and stored locally 3) User credentials, once entered, are saved for the user.

2) SharePoint as a mapped network drive via net use/"open in explorer"

Again, the user experience here is ideal for the same reasons mentioned above, but this option has the added benefit of storing the files remotely in the cloud.  I thought this was my golden ticket until i found: 1) In order to use this method you must save Office 365 login credentials on windows 2) Saved credentials stay stored in IE ("Connected with windows") effectively removing the need to authenticate 3) Upon mapping the network drive, you are always authenticated - you do not need to reenter 365 credentials to access that drive.

3) SharePoint in Outlook via STSSYNC protocol

This is not an ideal user experience as it is confusing, clunky, and appears to be a depreciated MS technology.  It also does not allow you to edit files locally, so the essential workflow is [ Find file in Outlook > Navigate to 365 > Work off browser] or [Find file in outlook > right click "copy shortcut" > paste copied SP URL into browser > download file locally].  Obviously neither of these are good solutions as the first option eliminates any benefits of having the local access, while the second is even worse as it has users download a true offline copy that is stored locally/not synced with SP online for version updates.

The one benefit of this method is that files are only stored locally while they are being accessed (C:\Users\XXXXXX\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook\XL7OJL0J), and then disappear after closing.

Again the goal here is to maintain an interface method for SharePoint, other than SP online, in which the user must re-authenticate upon each login, and no files are stored on the local machine (only in the cloud).  I believe method (2) comes closest to this goal, and I had the thought of remapping the drive each login via script to force users to re-authenticate, but it appears that you are not able to do so without the credentials already being stored in IE. 

I realize there may not be a perfect fit to my organization's needs, but would love to hear input into ways to achieve our security goal.

Thanks in advance,

Matt

webbrewers3 say:

To be clear about what I'm talking about: "instead of mapped drives, use network locations. "

This is a standard Windows functionality. In Windows explorer, right click computer/pc and select "add a network location" from the context menu.

Follow the prompts using the url of your root site <domain>.sharepoint.com

If you can overcome the flawed process by which Msft authenticates to a local machine, you'll be able to navigate to all your SPO locations from this shortcut via W explorer. If you do run into errors (generally caused by Msft's inability to resolve how your co has your network configured), post back.
