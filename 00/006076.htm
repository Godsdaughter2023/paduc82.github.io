jay freedman say:
This works in Word 2010 and 2013:

Sub OpenTrustCenter()
    With Dialogs(wdDialogToolsOptions)
        SendKeys "t%t"
        .Show
    End With
End Sub

If you want the dialog to open to a particular page other than the default Macro Settings, you can add more letters to the end of the SendKeys string; for example, to open at the Privacy Settings page, use

        SendKeys "t%tpp"

because Privacy Settings is the second item in the left-hand list that begins with P.

I have MS Access project (ADP) application with MS SQL Server 2008 R2.

And I am experiencing slow performance due to the known issue described below.

When running the VBA code CodeProject.OpenConnection to change the ADP connection in MS Access, it send queries to SQL server as follows (those queries are captured by SQL Server Profiler)

======================================================================================================

select tbl.name, schema_name(tblschema_id), tbl.type, cast(case when tbl.is_ms_shipped = 1 then 1when (select major_id from sys.extended_properties where major_id = tbl.object_id and minor_id = 0 and class = 1 and name = N'microsoft_database_tools_support')is not null then 1 else 0 end as bit)from sys.all_objects AS tbl where tbl.type in (N'U', N'S') and permissions(tbl.object_id) & 4096 <> 0

select v.name, schema_name(v.schema_id), v.type......from sys.all_objects As v where v.type = N'V' and permissions(v.object_id) & 4096 <> 0

select sp.name, schema_name(sp.schema_id), ...........from sys.all_objects AS sp where sp.type = N'P' and permissions(sp.object_id) & 32 <> 0

======================================================================================================

Of course, it depends on the number of SQL objects, it takes too long to finish the queries above unless a user does not have sysadmin role in the SQL server.

Even a user whose role is db_owner takes too much time to run the queries.

And I have found that it is because the SQL system function Permissions().

SQL Server 2008 R2 Help says the following:

This feature (Permissions()) will be removed in a future version of Microsoft SQL Server. Avoid using this feature in new development work, and plan to modify applications that currently use this feature. Continued use of the PERMISSIONS function may result in slower performance.

But there is no way to change the queries that MS Access send to SQL server to replace Permissions() function with the suggested ones.

So I wonder if there is a workaround for the issue.

And since the problem caused by SQL function Permissions() is the known issue and confirmed by Microsoft, I hope the issue will be corrected by the following ways.

1. Modifying the Permissions() function in MS SQL or

2. Modifying the OpenConnection function in MS Access not to use Permission() function or

3. Any other ways.

Thank you very much for your help.

a. user say:

For a QWERTY keyboard you may be able to use the following shortcuts:

for the acute accent (accent aigu in French), type ctrl-' then the letter (e.g. for é, press ctrl and the APOSTROPHE key, release, then press the e key). 

for the grave accent (accent grave), type ctrl-` then the letter, e.g. for à, press ctrl and the ACCENT GRAVE key, release, then press the a key

for the circumflex accent (accent circonflexe), type ctrl-shift-^  then the letter

for the diaeresis (accent tréma), type ctrl-shift-: (ctrl-shift-COLON) then the letter

for the cedilla (cédille), type ctrl-, (ctrl-COMMA), then the letter c

Just bear in mind that if you are typing the first letter in a sentence, Word may change a lowercase letter to an uppercase one, which can be confusing when you are trying this for the first time.
