abi99 (2) say:

Nokia Suite connects to the server account.nokia.com and auth.gfx.ms. For these servers, SSL 3 was disabled on October 15th 2014. As of today, these servers require TLS 1.0. Nokia Suite does not use TLS 1.0 but only SSL 3, because Nokia Suite uses the QT framework 4.7.4. Up to that version, the QtNetwork class QSslSocket used SSL 3 and not TLS 1.0 as default protocol. You can test yourself via a terminal command and OpenSSL:

openssl s_client -connect account.nokia.com:443 -ssl3

which simulates the behaviour of the Nokia Suite. Therefore, the initial request of the Nokia Suite is rejected with a SSL/TLS handshake-failure. Because of that, Nokia Suite gives this wrong error message. Hopefully, Nokia enables SSL 3 again. Edit: added workarounds

1st Workaround – use another tool

If you skip the creation of an account and ignore that wrong error message, you are still able to use the Nokia Suite. Only these features need an account:

  • Software Update:
    A) Recovery Tool (requires Windows 7 or newer; does full reset)
    B) Software Updater (updates; does not work for devices since 2011)
    C) Care Suite…
  • Nokia Music: MixRadio » Settings » Purchase history …
  • Nokia Store: download from its webpage …
  • Purchase History: view on its webpage …
  • Upload to OneDrive (formerly SkyDrive)
    A) Transfer my Data (for contacts),
    B) Microsoft Outlook 2013 (for contacts and calendar), and/or
    C) Micro-USB cable (for photos) 

2nd Workaround – replace a DLL file (3rd Edit: added exit step, reduced to just one DLL)

  1. exit the Nokia Suite via right-click in (bottom, left) Windows Notification Area (system tray) 
  2. download this modified version of ssleay32.dll …
  3. replace this file at C:\Program Files (x86)\Nokia\Nokia Suite\ 

A previous version of this workaround forgot the step of installing the ‘Visual C++ Redistributable Packages for Visual Studio 2013’. Later, this step was added. As of today, this installation step is not required anymore because the DLL was updated and does not need that package anymore. If you still face an issue, report here in this thread! If everything works, please, just mark this message as helpful.

To completely disable lock screen Window 10 needs two registry values.

Begin code:

[HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Microsoft\Windows\Personalization]
"NoLockScreen"=dword:00000001

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\Authentication\LogonUI\TestHooks]
"Threshold"=dword:00000000

:End code

All gone.

graham mayor say:
When Word crashes, if you have autorecovery set (and you appear to have) Word will to recover the information at the time of the crash. However crashing also tends to leave behind temporary files that will make it more likely that Word will crash in future.

Reasons for crashing have been discussed in some of your previous postings, but this post provides further clues.

1. You should not be saving as macro enabled documents. Your documents don't appear to have macros in them. Documents do not need macros to be enabled for picture content. If you need macros then they are better located in the document template or a global template. Macro enabled documents are rarely required. Save as DOCX document format, and set Word's options to ensure that this is the default file format.

2. The desktop does not behave as a normal file location - especially in Windows 7. It is not the ideal place to store documents and may be a contributory factor to the problems you are having. Save your documents in My Documents or a sub folder of My Documents and make that folder the default document folder in Word's options > Advanced > File Locations.

The backup function is optional but advisable, especially when you are having problems. When you save a document, Word creates a backup of the previously saved version and it is useful to have such a backup to help recover from screw-ups. You can delete them when you have no further use for them

Your illustration shows the presence of several lock files, which begin with the characters ~$. If Word is closed and these files remain, delete them as they will cause problems - see again http://www.gmayor.com/what_to_do_when_word_crashes.htm which covers the issue of lock files.

larry yu msft say:

Hi Paul,

The retention actions applied to Deleted Items are “Delete and Allow Recovery” and “Permanently Delete”. If we choose the former when creating the retention policy, then we can recover the delated items in 14 days after deleting them from Deleted Items. If we choose the latter, the deleted items will be deleted permanently after deleting them from Deleted Items.

In addition, if In-place hold or litigation hold has been enabled for this user mailbox, administrators can create an In-place eDiscovery to find those deleted items and then export them to a PST file.

Feel free to let us know if anything is unclear.

Regards,

Larry
