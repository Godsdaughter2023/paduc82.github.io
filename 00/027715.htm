LookUp Field - Multiple Values and Not Limit to List - Microsoft Community

jay_89 say:

LookUp Field - Multiple Values and Not Limit to List

In Access 2010, I want to create a lookup field that allows multiple values to be selected but I want users to be able to add their own entries (not limited only to choices in the list). Is this possible? If so, how do you go about doing this in Access? Thanks.

marshall barton say:

Don't use a lookup field in a table.  It will just confound most everything else you want to do with the field.  Instead, create a form bound to the table (or better a query based on the table) and use a combo box for the lookup selection or data entry.  To allow for data entry in the combo box, make sure its LimitToList property is set to No.

 

Use a table to contain the lookup values and some VBA code in the combo box's NotInList event procedure to add a new value to the table.  How you would do that depends on what else you have going on in the table.  If there are other fields in the table that need to entered, you'll probably want to open a dialog form based on the lookup table for users to enter those other values.  OTOH, if the lookup table only has one data field (plus an autonumber primary key?) then the code could execute an append query to add the new lookup value.

ken sheridan say: How you do this depends on whether, when you enter an item not in the list, you want that added to the list for future selection, or whether you want it inserted as an ad hoc value in the current row in the table to which the form is bound to the, and not to appear in the list.  Let's deal with the first scenario, which is the more usual approach.

Firstly you need three tables to model this.  You'll find an example of such a model as ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

You'll see that this has a Parents table, an Activities table and a ParentActivities table which models the many-to-many relationship type between them.  In the activities subform in the parents form in my demo the NotInList event procedure of the ActivityID combo box has code which inserts a row into the Activities table when a new activity is types in.  The combo box's NotInList property must be True (Yes) for this.  Try entering Hockey and see what happens.  In situations where you need to add other data, however, the code would need to open a form and pass the value to it ready for the user to insert the additional data.  You'll find an example in the NotInList.zip demo file in my same SkyDrive folder, in my case to enter a region in which a city is located when a new city is inserted.

Turning to the second scenario, where the new entry is ad hoc for the current record, if we stick with my ParentActivities example the Activities and ParentActivities tables' definitions would need to change to

Activities
....Activity (PK - text)

ParentActivities
....ParentID (FK - ling integer number)
....Activity (FK - text)

As before the primary key of ParentActivities is a composite one made up of the two columns, but the relationship between Activities and ParentActivities would not now be enforced.

The RowSource property of a combo box bound to the foreign key Activity column in the activities subform would now be:

SELECT Activity FROM Activities ORDER BY Activity;

The combo box's NotInList property must now be False (No), and its ColumnCount and ColumnWidths properties are left as the default.

You state you can connect other devices to your home network, so either the setup on your Tilt is incorrect or it may be malfunctioning.

 

Make sure you are typing the key correctly, many times when you type it incorrectly it connects to the network but has no access.

 

Also in order to ensure that it is not the device's fault, try going over to the neighbours house and ask politely to test your device there or in any other wireless network you can have access to.
