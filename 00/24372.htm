ken sheridan say:

Is there a way that maybe the query button could just open the report?

Yes, opening the query separately serves no purpose.  Set the report's RecordSource property to the name of the query.  Then all you need to do is open the report from the form and it will be restricted by the parameters you've entered in the form.  In fact you don't even need a separate query; you can just paste in the SQL statement as the report's RecorSource property.

One thing you should note, however, is that using an ORDER BY clause in a query used as a report's RecordSource merely slows things down.  You should not order the query, but order the report by means of its internal sorting and grouping mechanism in design view.

ken sheridan say: You can extract the individual elements of a multi-valued field by means of its Value property in a query, so you should be able to create an 'append' query along these lines:

INSERT INTO ProjectAssociates(Associate, ProjectName)
SELECT Associates.Value, [Project Name]
FROM [YourImportedSharePointList];

Note that I've changed the name of the Associates column in your new table to Associate.  Conventionally, in a relational database, column names are singular nouns wherever possible to reflect the fact that a column represents an attribute, whereas table names are conventionally plural or collective nouns wherever possible to reflect the fact that a table represents a set.

You should also insert rows into a referenced table, Projects, with:

INSERT INTO Projects(ProjectName)
SELECT DISTINCT [Project Name]
FROM [YourImportedSharePointList];

and then create an enforced  relationship between the tables.

ken sheridan say: An alternative approach, which I use, is to try and create the folder and trap the error if it exists.  First place the following function in a standard module:

Public Function CreateFolder(strPath As String) As Boolean

    Const FOLDEREXISTS = 75
    
    On Error Resume Next
    MkDir strPath
    Select Case Err.Number
        Case 0
        ' no error
        CreateFolder = True
        Case FOLDEREXISTS
        ' anicipated error
        CreateFolder = True
        Case Else
        ' unknown error
        CreateFolder = False
    End Select
    
End Function

Then call it in your code:

Dim strPath As String
Dim strFolder As String
Dim stDocName As String
Dim strFile As String
Dim i As Integer

stDocName = "Claim Form"
DoCmd.OpenReport stDocName, acPreview
 
strFolder = ((Me.CustomerID - 1) \ 100) * 100 + 1 & " To " & ((Me.CustomerID - 1) \ 100) * 100 + 100 & "\"
strPath = "O:\Open Files\Open " & strFolder

If CreateFolder(strPath) Then
    strPath = "O:\Open Files\Open " & strFolder & Me.CustomerID
    strFile = strPath & "\Claim Form.PDF"
    If CreateFolder(strPath) Then
        If Dir(strFile) <> "" Then
            i = 0
            Do
                i = i + 1
                strFile = strPath & "\Claim Form" & i & ".PDF"
            Loop Until Dir(strFile) = ""
        End If
    
        ' code to add file to folder
    
    End If
End If

Note how the CreateFolder function is called twice to create the main subfolder if it does not exist before creating the CustomerID subfolder if that does not exist.  This caters for Doug's point about the ability of the MkDir command to only create one level of subfolder at a time.  The Windows API CreateDirectory function is much more flexible, but a little trickier to use.

BTW you can create a PDF file from a report without opening it in print preview by using the OutputTo method of the DoCmd object, provided that any restriction on the report is by means of parameters, not by filtering the report, as, unlike the OpenReport method, the OutputTo method does not have a WhereCondition argument.  You'll find an example as InvoicePDF.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169
