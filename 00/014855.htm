andreas killer say:

Specifically, there are workbooks that folks graciously post to various blogs and websites, but in protected mode I can't see the VBA to know if it is safe to exit protected mode; the webpages that describe protected mode (like above) indicate that disabling protected mode exposes the user to viruses, malware, etc... so it feels like a catch-22, where I can't see if the code is safe to unprotect, until after I've unprotected it and exposed myself to the risk?

If a file contains a virus/malware, you can not see it. And if you open such a file it doesn't matter if you open that file in protected mode or whatever, the virus installs itself before Excel opens the file!

This works quite differently than most people think. Therefore, most hackers disguise their Trojans as PDF files.

So if you download a file from anywhere, you should use a virus scanner to check the file.
If the file is clean, never double click it within the Windows Explorer! Do this:

Open Excel

Click File\Open and select your file
Press and hold the SHIFT key
Click Open
Wait for your file to open
Release the SHIFT key

This prevents that code is executed at startup. Now you can view the code.

And if you want to be 100% sure: Do all that inside a virtual machine, not on your main system.

peter burton (pburton) say:

Hey all,

After 2 months of working with the Microsoft product team I have finally solved this and wanted to share the solution.  

In our particular situation this was caused because we had set an IP address restriction on our Sharepoint site (property IPAddressEnforcement).  Setting this through the UI also sets a timeout value for IPAddressWACTokenLifetime to 15 minutes. 

The fix for this is to update the IPAddressWACTokenLifetime value, in my case I used 500 minutes

Steps for this are

  1. Download SharePoint Online Management Shell.

https://www.microsoft.com/en-us/download/details.aspx?id=35588

  1. Connect-SPOService
  2. Input https://YOUR_URL_HERE-admin.sharepoint.com/ as URL and input the credentials.
  3. Get-SPOTenant | fl and check if the IPAddressEnforcement is True and IPAddressWACTokenLifeTime is 15.

  1. If yes, then run command Set-SPOTenant – IPAddressWACTokenLifeTime (A period which is longer than 15 ).

Example Set-SPOTenant – IPAddressWACTokenLifeTime 20

Reference, https://docs.microsoft.com/en-us/powershell/module/sharepoint-online/set-spotenant?view=sharepoint-ps

  1. Then the issue should be resolved

Side-by-side errors are special cases.  Here's some broad overview information about this: http://www.winvistaclub.com/f16.html.  I believe you have a problem within the core OS system files or the folders and storage locations and folders where manifests and other information is stored and nothing we try here or even at MSE Microsoft Support is going to help with this type of problem.  The simplest aspect of SXS problems is reviewing the CBS.LOG from an SFC /scannow (and I mean the entire log, not the sfcrepair.txt summary).  Then it gets more complex reviewing the CheckSur.Log and other things.  There are other more detailed and comprehensive scans to try to find problems - but even when found, fixing them can be a nightmare as you need to somehow get the hashes correct or it will always show as an error if it even resolves the problem - and I mean even when using the exact same file version and date and everything if you can find it.

They may have someone in the Answers Windows Forum - System Repair and Recovery - for your operating system who can do this (I'm not sure and have never seen any thread with a complete analysis and SXS repair, though I haven't seen every thread), but I don't believe we have such a person here.

 

Since mostly everyone who could help in this Forum has posted here or in the above thread (and trying support was already suggested), it may be time to try the other forum instead of starting over here where you're likely to just see the same answers from another set of people - and few exist who haven't replied in one or the other.

I am the administrator but my Word CD'S are Read Only. How can I edit them

In addition to CD's being read only, very often I cannot edit a document on my hard drive.  When I close it and reopen it I can edit it.  What is the problem.
jay freedman say:
Unless you're using a special type of driver that makes a CD act like a big floppy disk, files on CDs will always be read-only. It has nothing to do with being the administrator or not -- it's just that most CDs physically can't be rewritten or erased.

When you copy a file from a CD to the hard drive, Windows keeps the read-only attribute on the file. After copying, you need to right-click the file, click Properties, and uncheck the read-only box.

charles kenyon say:
