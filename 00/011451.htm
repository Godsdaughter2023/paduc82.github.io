emxpert say:

I have had the same problem for years, on WindowsXP, with a Windows Mobile 2003 device (Sprint PPC-6600 - a/k/a/ HTC Harrier), I believe with multiple versions of ActiveSync (I'd have to check which other versions I tried, but at the moment, it's using 4.5.0).  Current laptop is a MacBookPro 17" (Core Duo 2.1 GHz; circa June 2006) running Windows XP SP2 via BootCamp.

 

As described by others, whenever I put the laptop in standby mode (doesn't matter if the PDA is or isn't connected at the time), when I wake up the laptop, ActiveSync no longer works via bluetooth, because the bluetooth COM port that was linked to ActiveSync has mysteriously "disappeared" from the "connection settings" window of ActiveSync, even though the COM port still shows up in Windows Device Manager.  Just as with others in this thread, after I reboot, the problem goes away.  It's not clear to me if it's an ActiveSync problem, or bluetooth driver issue or a Windows power management issue.  Based on my experience (below), I can only guess that ActiveSync regenerates a list of COM ports at reboot, but maybe doesn't do so when ActiveSync is restarted (without rebooting).  Perhaps when waking from standby ActiveSync searches for active bluetooth COM ports too soon, finds none, and rebooting is the only available way to refresh the list of COM ports.  Or maybe after resuming from standby, the bluetooth radio is still in some kind of low power mode that prevents its detection by ActiveSync?  (sorry, just brainstorming)

 

I was glad to find that a somewhat recent (relatively speaking) thread actually exists on this issue, but I'm sorry to report that I don't have a real solution, and disappointed to see that none has yet been found.  I'm fearful that MS has abandoned ActiveSync, now that Vista uses the "Windows Mobile Device Center" in place of ActiveSync.  If it's a problem that requires a new build of ActiveSync, we may never see a solution.

 

However, I thought I should post, since it seems maybe I have some information that hasn't been mentioned yet:

 

If I use the Device Manager to "uninstall" the device before I put the laptop in standby, after I wake up the laptop, Windows can "scan for new plug and play devices", and automatically reinstall the device.  This is *almost* a decent work-around, except for the following problem:  When the device is "uninstalled", the COM port is treated as a disconnected/hidden device, and when it's reinstalled, it's assigned a new (next available) COM port number, instead of reusing the old one.  Therefore:

1. ActiveSync connection settings needs to be set to use the new COM port.

2. After a few times, you'll run out of COM ports that Activesync can use.

 

To get around the second issue, you could:

Option A:

You could try the following approach, (after you uninstall the bluetooth COM port device manager entry, but before you put the laptop in standby would be best I think); haven't tried this yet :

http://www.bisque.com/SC/forums/430/ShowPost.aspx

 

Option B:

 set up Device Manager "View" menu for "show hidden devices", and periodically remove the unused Bluetooth COM ports manually, or you can do the same thing faster by deleting their entries from the registry (though the latter could cause you more problems if you accidentally delete a COM port that's used for something else).  Bluetooh Serial ports are found under:

HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Enum\BTHENUM

scottgem say:
First, it is NOT recommended that on use lookup fields on the table level (http://access.mvps.org/access/lookupfields.htm).  Users should never interact directly with tables, so lookups should be done with list controls on the form level. Second, I wouldn't structure my

You have a many to many relationship here so I would use a junction table to model it.

tblStudentSubject
StudentSubjectID (PK Autonumber
StudentID
SubjectID
DateTaken
DateCompleted

So you would use a Subform on your student mainform to enter the Subjects chosen. When the student is assigned to a course you can enter the date taken

To then display the number of courses selected, you could have a textbox with the ControlSource:

=DCount("*","tblStudentSubject","[StudentID] = " & Me.StudentID)

There is no need to store this value as its a calculated value, the Dcount will display the count accurately.

kevin pyle say:

Flag mail merge so only Excel cells with data are imported

We have annual giving pledge campaigns. When someone pledges, we send a thank you letter to them confirming the amount and frequency of their gift. We track the pledges in an Excel spreadsheet which is used as the data file for the Word mail merge thank you letter.

Is there a way to set a flag so that if a cell has data in it then only merge that record's data into the Word mail merge document?

The fields in the spreadsheet that are merged are:

Name; Street; City; Name2; 2018-Freq; 2018-Amt

(One row per record)

A tracking field that shows the date the pledge was posted to our CMS is "Posted".

What little I remember from my programming days beaucoup years ago says I could use an "IF - THEN" statement, such as IF "Posted" >{Today's Date} THEN Merge fields "Name"; "Street"; "City"; "Name2"; "2018-Freq"; "2018-Amt" to Mail Merge document. At least that's the code my organic computer would use.

We're looking at upwards of 150 letters in each batch, if not (we hope) more.

Thank you.

a. user say:

Put the following nested fields at the beginning of your mail merge main document and see if that is enough for your requirement.

{ SKIPIF { MERGEFIELD Posted \@YYYYMMDD } <= { DATE \@YYYYMMDD } }

All the { } need to be the special field brace pair characters that you can insert in Windows versions of Word using ctrl-F9. You can't just type them on the keyboard. You can use Alt-F9 to toggle between "field code" view and "field results view" so you can see what you are doing. 

It is possible to filter the records using the facilities in "Select Recipients", but as far as I know in there you can only compare a date field against a specific date, not a function whose value changes from day to day such as "today" or some such). So you would have to change the criteria manually each day.

However, you can also set up the criteria using VBA, e.g. if your data is coming from Sheet1, you can connect to the sheet, then set the query in VBE's immediate pane using 

activedocument.MailMerge.DataSource.QueryString="SELECT * FROM [Sheet1$] WHERE d > date()"

Once you have executed that command, the query should "stick, and the two main problems then are (a) if the user is supposed to be able to apply more filters to the data, it may not work, and (b) if the end user needs to reconnect to the data the filter will not be re-applied.
