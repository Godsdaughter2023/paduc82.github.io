This little file includes an option to create a 'Daily Calendar'.  This could be amended to create a separate calendar table for each medication/date/time by adding a txtTime text box and cboMedicationID combo box to the form, the former to enter the time of day when the medication is taken, the latter to select the MedicationID primary key value for the medication in question, looked up via the combo box from a Medications table, hiding the first (MedicationID) column and showing the medication name in the usual way (the combo box wizard can set this up for you if you wish).  You'd then amend the Click event procedure of the Create Table button in the form as follows:

Private Sub cmdCreateTable_Click()

    On Error GoTo Err_Handler

    Const FORMNAME = "frmCalendar"
    Const MESSAGETEXT1 = "At least one day of the week must be selected."
    Const MESSAGETEXT2 = "A table name and both dates must be entered."
    Dim ctrl As Control
    Dim blnNoDays As Boolean
    
    blnNoDays = True
    
    ' loop through day of week checkboxes and
    ' ensure at least one is True
    For Each ctrl In Me.Controls
        If ctrl.Tag = "CheckMe" Then
            If ctrl = True Then
                blnNoDays = False
                Exit For
            End If
        End If
    Next ctrl
    
    ' if all necessary controls are filled create calendar table
    If Not blnNoDays Then
        If Not IsNull(Me.txtTableName) _
            And Not IsNull(Me.txtDateStart) _
            And Not IsNull(Me.txtDateEnd) Then
            
            DoCmd.Hourglass True
            
            MakeCalendar_DAO _
                Me.txtTableName, _
                CDate(Me.txtDateStart), _
                CDate(Me.txtDateEnd), _
                CDate(Me.txtTime), _
                Me.cboMedicationID, _
                IIf(Me.chkSun, 1, Null), _
                IIf(Me.chkMon, 2, Null), _
                IIf(Me.chkTue, 3, Null), _
                IIf(Me.chkWed, 4, Null), _
                IIf(Me.chkThu, 5, Null), _
                IIf(Me.chkFri, 6, Null), _
                IIf(Me.chkSat, 7, Null)
            
            DoCmd.Hourglass False
            ' close calendar form if open and reopen it
            ' using newly created table
            DoCmd.Close acForm, FORMNAME
            DoCmd.OpenForm FORMNAME, OpenArgs:=(Me.txtTableName)
                
        Else
            MsgBox MESSAGETEXT2, vbExclamation, "Invalid Operation"
        End If
    Else
        MsgBox MESSAGETEXT1, vbExclamation, "Invalid Operation"
    End If
    
Exit_Here:
    Exit Sub
    
Err_Handler:
    DoCmd.Hourglass False
    MsgBox Err.Description, vbExclamation, "Error"
    Resume Exit_Here
    

End Sub


You'd also need to amend the MakeCalendar_DAO fuction in the basCalendarmodule as follows:

Public Function MakeCalendar_DAO(ByVal strTable As String, _
                            ByVal dtmStart As Date, _
                            ByVal dtmEnd As Date, _
                            ByVal dtmTime As Date, _
                            ByVal lngMedicationID, _
                            ParamArray varDays() As Variant)
                            
    ' Accepts:  Name of calendar table to be created: String.
    '           Start date for calendar: DateTime.
    '           End date for calendar: DateTime.
    '           Time per day  to be included in date/time value: DateTime.
    '           Days of week to be included in calendar
    '           MedicationID:Long Integer
    '           as value list, e.g. 2,3,4,5,6 for Mon-Fri
    '           (use 0 to include all days of week)

    Dim dbs As DAO.Database, tdf As DAO.TableDef
    Dim strSQL As String
    Dim dtmDate As Date
    Dim varDay As Variant
    Dim lngDayNum As Long
    
    Set dbs = CurrentDb
    
    ' does table exist? If so get user confirmation to delete it
    On Error Resume Next
    Set tdf = dbs.TableDefs(strTable)
    If Err = 0 Then
        If MsgBox("Replace existing table: " & _
            strTable & "?", vbYesNo + vbQuestion, _
            "Delete Table?") = vbYes Then
            strSQL = "DROP TABLE " & strTable
            dbs.Execute strSQL
        Else
            Exit Function
        End If
    End If
    On Error GoTo 0
    
    ' create new table
    strSQL = "CREATE TABLE " & strTable & _
        "(calDate DATETIME, " & _
        "MedicationID LONG," & _
        "CONSTRAINT PrimaryKey PRIMARY KEY (calDate))"
    dbs.Execute strSQL
    
    ' refresh database window
    Application.RefreshDatabaseWindow
       
    If varDays(0) = 0 Then
        ' fill table with all dates
        For dtmDate = dtmStart To dtmEnd
            lngDayNum = lngDayNum + 1
            strSQL = "INSERT INTO " & strTable & "(calDate,MedicationID) " & _
               "VALUES(#" & Format(dtmDate + dtmTime, "mm/dd/yyyy hh:nn:ss") & _
               "#," & lngMedicationID & ")"
               
            dbs.Execute strSQL
        Next dtmDate
    Else
        ' fill table with dates of selected days of week only
        For dtmDate = dtmStart To dtmEnd
            For Each varDay In varDays()
                If Not IsNull(varDay) Then
                    If Weekday(dtmDate) = varDay Then
                        lngDayNum = lngDayNum + 1
                        strSQL = "INSERT INTO " & strTable & "(calDate,MedicationID) " & _
                            "VALUES(#" & Format(dtmDate + dtmTime, "mm/dd/yyyy hh:nn:ss") & _
                            "#," & lngMedicationID & ")"
                        dbs.Execute strSQL
                    End If
                End If
            Next varDay
        Next dtmDate
    End If
    
End Function

For each medication you'd create a calendar table for each time per day it is taken, stretching as far into the future as you wish, giving each calendar table a separate name.  You'd thus end up with a bunch of separate calendar tables.  This is not a good design of course, so you'd next create an empty calendar table for all medications in which the two columns, calDate and MedicationID are the composite primary key.

The final step would be to import each of the tables you created into this final table by a series of simple 'append' queries.  Once you are satisfied that the final table includes all the data you can delete the original tables.  With a little bit of coding the whole process could be semi-automated, but even without, it would take less time to insert all the data than it's taken me to write this.  The final table, joined to the Medications table in a query, can then be used as the RecordSource of a report, grouped by day for instance, to show the medications and their times to be taken each day.  The report could easily be set up to open for one day by means of a parameter, or even to open automatically at the current day by restricting its query to:

WHERE DateValue(calDate) = Date()

I'm sorry, Scott, but IMO this is in fact badly denormalized. A fundamental principle of relational databases is that fields should be "atomic" - one field value should convey one and only one item of information. You're making this field contain three (or seven, I presume), and you have to pick the field apart to find which.

You have a bog-standard Many to Many relationship: each medication will be taken on zero, one, or many days, and on each Day you will take zero, one or many Medications. Am I misunderstanding, or are you intentionally denormalizing for some specific benefit?
