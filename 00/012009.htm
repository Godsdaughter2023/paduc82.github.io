bill mosca say:

First of all, don't store the age. It will change every year which means you will be constantly updating that field. A better way is to display the age in a query.

Here is a user-defined function for calculating age. Use it as a field in your queries.

Public Function CalcAge(DOB As Variant)
'Purpose : Calculate age
'DateTime : 3/18/2000 10:50
'Author : Bill Mosca
On Error Resume Next

CalcAge = DateDiff("yyyy", DOB, Now()) _
+ Int(Format(Now(), "mmdd") < Format(DOB, "mmdd"))

End Function

Open a query in design view. Put this in a field name: CalcAge([YourDOBFieldName])

When you run the query the age will automatically be calculated.

Effect of proofing option "Ignore all" in Word should be - Microsoft Community

birgerbjerkeng say:

Effect of proofing option "Ignore all" in Word should be document specific and persistent between sessions.

Dictionaries are always limited, in particular with regards to technical terms. For this purpose, the "Ignore all" option is useful to remove error flagging throughout a document of a correctly written term not listed in the dictionary, while retaining flags on possible typing errors of the same term. What I do not like, and find somewhat illogical, is that specifications of "Ignore all" are forgotten when MS Word is closed. 

They are remembered across documents as long as Word is kept open, i.e. within the session, but forgotten when Word is closed. I think the option would be much more useful if instead the "Ignore all" specifications were document specific, but were remembered the next time the document was opened by the same user or another user. In that way, it would be a real supplement to document-independent Custom Dictionary feature that is currently implemented.

So what I would like is for "Ignore all" to operate on  an implicit Custom Dictionary built into each document. It should of course also be open for inspection and editing, and should probably be controlled through Options.

The real problem turned out to be that Outlook strongly guides the user into a crippled IMAP property dialog via the File / Account Settings / Account Settings (dup intentional) / (select account ) / Change... command path. Here is the crippled IMAP dialog:

 

The problem is that this crippled dialog does not let you provide a different value for the account email than for the account itself. Using an alias requires the two to be different, and you can do this using the uncrippled IMAP properties via File / Account Settings / Manage Profiles / Email Accounts / (select account) / Change...
