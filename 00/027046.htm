vicelly say:

Hello Bob,

Thank you for your reply. I have done some further analysis on this problem. Here is what I have found. Maybe this will be of some help to others having this issue.

In my spreadsheet, there are no links to other documents. That is, the Edit-Links menu is dimmed. But the reason for the Security Warning is that I had used File-Import ... to import a text file to the spreadsheet. When you do that, Excel thinks you want to establish a data connection to that text file. There doesn't seem to be any way to eliminate that bogus data connection.

The problem can be avoided in the first place, however. When you use the File-Import ... command, Excel first asks what file to import and then allows you to format the import. After that, Excel presents a window asking where the file is to be placed - in the existing sheet or a new sheet. In that window, there is a button called "Properties ...". When you click that button, a window opens which has an item called Query definition - "Save query definition". If you de-select that option, and then import the file as normally, the text is imported, AND YOU WILL NOT SEE THE SECURITY WARNING message for that file when you next open it.

Unfortunately, that does NOT break any existing data connection. It will only prevent the new import from establishing a connection. Any current file with this warning will continue to warn you.

So, whenever you import a text file, always de-select that option, and you won't ever be bothered by the warning for that file. I have found that the warning, while annoying, does no real harm. I just dismiss the warning by clicking the "X", and continue on. The spreadsheet itself appears unaffected.

marshall barton say:

You can not get this to work precisely in all situations.  The first time the report is processed (to calculate the value of Pages), all page footers, even the "last" page, will be invisible.  The second time through, when Pages has a value, the footer where  Page = Pages is made visible.  But, that might not leave enough room for all the details that were on the "last" page the first time, so another page is needed to display all the records.

 

I think what you are doing is the closest you can get unless you leave the page footer visible on all pages and just make the page footer controls invisible until the last page or cancel the event in the page footer as Ken said.

 

Another way that may or may not be acceptable is to set the report footer section's ForceNewPage property to Before Section so the last page has no details and the report footer and page footer are the only things on the last page.

Private Sub PageFooterSection_Format(Cancel As Integer, FormatCount As Integer)

    If Page <> Pages Then
        Cancel = True
        Me.Section(acPageFooter).Height = 0
    Else
        Me.Section(acPageFooter).Height = 2880
    End If

End Sub
