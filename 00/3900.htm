Concurrent tasks with same resources, fastest completion possible - Microsoft Community

jennarj say:

Concurrent tasks with same resources, fastest completion possible

I am trying to schedule out the following scenario and cannot find a way to schedule the tasks concurrently. I'm hoping to find an automated way of doing the following:

- I have 350 explorations to complete with 5 drill rigs. The rigs can all run concurrently, and they are not dependent on each other. I have durations ranging from 1 to 3 days per exploration, and each exploration has a different site access date, ranging from now to March 2020. I want to schedule the rigs so that each rig is running at the same time, and can jump from any exploration to any exploration in the fastest time possible (so, by access date and duration). 


I've tried the following and run into the following issues:

- Assigning the same resources (rigs A, B, C, D, and E) to each task and leveling them, but that schedules them linearly.

- I could manually assign a resource to each and then level them, but then that does not guarantee that fastest drill time (for example, if I assign resource A to 10 explorations, they could all have 3 day durations, whereas resource B could have all 1 day durations. Resource B would get done in 10 days and then be sitting there doing no work while resource A finishes in 30 days). 

- I've considered just making a new project schedule for each drill rig and assigning each one 1/5 of the explorations, but that could result in the same issue as above.

Any suggestions? I am using Project 2010.

trevor rabey say:

I would go with the five separate resources called Rig A, Rig B, Rig C, Rig D, Rig E. Each rig includes a crew which is permanently attached to that rig.

Start by assigning the 5 rigs to the 350 holes, 70 holes each. I am assuming that all of the holes have the same estimated depth in metres and so have the same estimated durations, but it also works for different durations for each hole. I am also assuming that each rig drills holes at the same rate, but it also works if they drill at different rates. I am also assuming that for each hole and a given rig, the rate is determined by the rig, and not the particular hole (soft sand or hard rock). That's a lot a variables to ignore, but press on. I would distribute rigs to holes so that the work of all 5 rigs is about the same, even if some rigs get more holes than others.

Then assign priorities to each hole drilling task and level, with leveling order by priority and options off.

Whatever you get, it's feasible. If all of the durations are equal, all feasible solutions have the same overall duration.

Introduce different durations for each hole and it starts to get more interesting.

You also have to move the rigs from one hole to the next, and the distance and time that it takes from each hole to the next will all be different.

Feasible is a good place to start but it is not optimal (where optimal is defined as shortest overall duration). If you are going for "optimal" you have to define what optimal means. It may mean shortest overall duration, but it may also mean lowest overall cost.

The only control you really have in MSP is the priorities. Try different combinations of priorities.

What you have here is a variation on the classic travelling salesman problem, one of the great unsolved optimisation problems, and if you can solve that you can have a Fields medal and be famous. There are many algorithms which approach a solution but none perfect. Have you considered a linear programming approach (Simplex Method etc)?

tom_bpc say:

Jenna,

You’re almost to the answer.  I.e.:

  1. You have 350 tasks, each with a set duration and a constrained early-start date, and no logic relationships between them.  I.e. given unlimited resources and access, you’d start them all tomorrow.  Each task demands 1 drill rig to be assigned for the duration of the task.
  2. You have 5 drill rigs/crews that are equal in terms of skills, productivity, and cost.  The assignment protocol is essentially “next man up.”

This is what I’d suggest:

  1.  Make the schedule based on the access constraints and the durations.
  2. Create a single work resource – Drill Rig – with maximum capacity of 500%.
  3. Assign the “Drill Rig” resource to each task.  (By default, the assignment will be 100% – i.e. 1 rig – for the duration of the task.)
  4. With Leveling Options set to Manual, Priority/Standard, and no splits, run Level All.  That should result in a resource-achievable schedule.
  5. Use explicit task priorities where needed to over-ride the default leveling rules.  (A task with a priority of 1000 will be scheduled to start on its access date, full stop.  Too many of these at the same time will cause unresolvable resource conflicts.)

This approach presumes that mob/demob/travel-time is irrelevant.  If mob/demob/travel time is not irrelevant, then you may consider dividing the sites into regions or clusters to be blitzed sequentially.  In that case, you’d need to trade off the increased chances of idle time and consequent overall schedule delay (due to insufficient site access in the area being blitzed) against the potential extra time and costs of traveling to (and returning from) an accessible site elsewhere.  

Presuming you’ve avoided unresolvable conflicts, the result of all this will be a site drilling schedule that violates no logic and never requires more than 5 drill rigs at a time.  The assignment of which specific rig drills which specific site is something to be decided using a map, a rig maintenance assessment, and a weather forecast on a week to week basis.  In any case, Project’s leveler will provide a workable but not necessarily fastest schedule.  You may be able to improve it using explicit priorities.

Good luck, tom

jay freedman say:

Probably what you saved is the merge main document (the one where you inserted the merge fields) with the Preview Results button activated.

While that document is open, click the Finish & Merge button and select Print Documents. In the little dialog that pops up, choose All and click OK.

If you want to save the results of the merge, instead of Print Documents you can select Edit Individual Documents. That will open another document, which you can save. In the case of the Letters merge, all the letters will be in the one document, with section breaks between them and with the page numbers restarting at 1 for each section.

You may also find the add-in from Graham Mayor's page less confusing than the built-in merge.

If I kept trying to export to PDF, eventually it would do as it was told and I'd get an exported PDF.
What I found was that if the file already existed, the PDF would save as expected.
To get around the issue, I wrote a macro to firstly create an empty file then output as PDF.
Here's my code:

    Dim fs As Object
    Dim TextFile As Object
    Set fs = CreateObject("Scripting.FileSystemObject")
    Set TextFile = fs.CreateTextFile(DestPDF, True)
    TextFile.Close
    DoCmd.OutputTo acOutputReport, MyReportName, acFormatPDF, DestPDF

Good luck!
