Woohoo!

After multiple hours of trying to find a solution, I've cracked it and thought I'd post it here. It isn't a workaround either so I suggest you try this before entertaining new user profiles or reinstalling Windows.

The issue seems to lie with Microsoft's Azure AD Broker which handles your Office 365 account authentication in Windows 10. Office 2016 will try to use this authentication method by default and all things being well, it's much better than the legacy approach. Unfortunately it looks like the token for your account can become corrupted (unsure what caused it, maybe AV), and no amount of disconnecting and reconnecting your account will refresh the token.

I take no responsibility for software or data loss if you manage to somehow bork up your system.

Open Event Viewer (quick way to do this is Start -> type "eventvwr.msc") and navigate to Windows Logs -> Security.
- If you've just tried to use your account under an Office program you may notice both Audit Failure and Success being logged in quick succession.
- Scroll down on an Audit Success until you find 'Key Name' looking something like this "***_microsoftonline.com". If the entry you've selected doesn't contain this line, find a logged entry which does.
-Under 'Key File Operation Information' you should find 'File Path'. Mine was C:\Users\BobSled\AppData\Local\Packages\Microsoft.AAD.BrokerPlugin_cw5n1h2txyewy\AC\Microsoft\Crypto\TokenBindingKeys\Keys
fb47a8d42be514593f982d4b0d7edd08_90cb05ef-cd90-4066-aafe-3397eb8039a5_4068bc0f75fd0f09b9c2c7507f15f982
- Start -> Type 'Run' -> Type "%localappdata%\Packages" and hit Enter - Scroll until you see a folder starting with "Microsoft.AAD.BrokerPlugin_". Open it. Then open AC\Microsoft\Crypto\TokenBindingKeys\Keys.
- In this folder is the offending file, matching the Event Log entry you found above, so we'll rename the file to something expressive of our frustration. Right click -> Rename. I suggest adding something like "stopwastingmytime_".
- Reboot.

If an Office 365 mailbox is using the default retention policy, and its Inbox folder emails are not deleted, archived or moved, these emails will always be kept in the Inbox folder. However, there is a 2-year archive retention tag in the default retention policy which will automatically archive emails older than 2 years if the mailbox’s Archive feature has been enabled. Also, if your mailbox has been assigned any custom retention policy, Inbox folder emails may be deleted/archived due to it.

So to better understand the situation, I’d like to collect the following information:

1. Does the email disappearing issue only occur to the Inbox folder? How about other mail folders?
2. Has the Archive feature been enabled for the affected mailboxes?
3. Is there any custom retention policy applied to the affected mailboxes? (If you’re not an administrator of the Office 365 tenant, please contact an administrator for checking)
4. Assign eDiscovery permissions in Exchange to an admin account and then create an In-Place eDiscovery search to see if you can find out the problematic user’s emails which are older than 2 years (This also needs to be done by an Office 365 administrator).

Aaron
