ken sheridan say: There are a couple of issues here:

1.  You are using the LIKE operator with the asterisk wildcard character to ignore the parameter where the check box is FALSE.  This is not reliable as it does not allow for a NULL in the column (field) in question.  This is because NULL is not a value, but the absence of a value so nothing can equal NULL, not even NULL.  The result is always NULL.

2.  In the case of the cut and cuas columns you are using the >= operator against the asterisk wildcard, rather than against a zero, but again this would not allow for NULLs.

You can overcome both of these by using a different approach:

WHERE ((forms]![block model]![check_formation] = TRUE
    AND formation = [forms]![block model]![formation])
    OR (forms]![block model]![check_formation] = FALSE)
AND (([forms]![block model]![check_oxide] = TRUE
     AND oxzone = [forms]![block model]![oxzone])
     OR [forms]![block model]![check_oxide] = FALSE)
AND (([Forms]![Block Model]![check_grade] = TRUE
    AND cut >= [Forms]![Block Model]![cut])
    OR [Forms]![Block Model]![check_grade] = FALSE)
AND (([forms]![block model]![check_class] = TRUE
    AND class = [forms]![block model]![class])
    OR [forms]![block model]![check_class] = FALSE)
AND (([Forms]![Block Model]![check_ascu]= TRUE
    AND cuas >= [Forms]![Block Model]![cuas])
    OR [Forms]![Block Model]![check_ascu] = FALSE)

A few points to note:

1.  The positioning of the parentheses in the above is crucial to the correct evaluation of the criteria.

2.  Enter the WHERE clause and save the query in SQL view, not in query design view.   If you do the latter Access will move things around.  The underlying logic should be preserved, but at best will be significantly obscured, or at worst the query won't work at all.

3.  In the Block Model form none of the check boxes must be NULL, so be sure to set them to FALSE by default, either by means of their DefaultValue properties, or by assigning a value of FALSE to each in the form's Load event procedure.

What this does is join the tables on the ID column (field)  from Customers and the Customer ID column from Orders.  Normally when you join in tables in a query the default join type is an INNER JOIN, which means rows will only be returned where there are matches, so any 'unmatched' rows would not be returned.  In this case, however, a LEFT JOIN is used.  This means that all rows from the left side of the join (the Customers table in this case) will be returned even if there is no match on the right side.  What we want to do, however, is exclude all the rows where there is a match and return only those without matches.  This is done by the criterion in the WHERE clause 'WHERE Orders.[Order ID] IS NULL'.  Where there is no match the columns on the right side of the join will all be NULL, i.e. have no value, so by testing for the Order ID column being NULL we can restrict the results to the unmatched rows as the Order ID column is the primary key of Orders, and as such cannot be NULL.  So we can't get rows where there is a match in Orders but with a NULL order ID column, as this column can never be Null.  Consequently we get only genuine unmatched rows, i.e. customers without any orders.

The above query can easily be created in design view by adding both tables.  Then edit the join properties and select: Include ALL records from 'Customers' and only those records from 'Orders' where the joined fields are equal.  Add the asterisk from Customers to the first column in the design grid; this returns all columns from Customers.  Add Order ID from Orders to the second column, uncheck 'show' and in the criteria row of this column enter Is Null.

I don't know where things went wrong with the wizard, but if you 'hand-craft' the query as above you'll get a far better understanding of how such a query works than just letting the wizard do the work for you.  You'll then be in a position to apply this understanding to other similar situations.

A couple of supplementary comments:

1.  Why the designers of the Northwind database chose to call the primary key of Customers ID and the corresponding foreign key in Orders [Customer ID] is a mystery to me.  It would have made a lot more sense to call the primary key of Customers [Customer ID] also.  Not only would this  more specific descriptor of the attribute which the column represents, but it would also be immediately apparent that the tables are related on these columns.

2.  You'll note that the [Order ID] and [Customer ID] columns, along with others in the tables, include spaces, which is why the names have to be enclosed in square brackets.  Most experienced developers do not favour the inclusion of spaces, or other special characters, in column names.  If you want to represent a space, use the underscore character, e.g. Customer_ID.  Some database systems don't allow spaces at all in object names, something which should have been done in the case of Access from the start in my view.

denisec[msft] say:

Hello--

Hope this information helps.

OneNote 2016

Language detection before typing:  OneNote 2016 doesn’t have a language detection feature at this point. It marks text with language information based on the current keyboard layout. If you are typing with US-English keyboard, the text will be marked with English automatically. If you want to type in a different language, then you can install additional keyboard layouts in Windows and switch to that keyboard layout before typing.  So if you switch to a German keyboard layout, it will automatically mark the text with German and spell checker will use German language for that part of the text. This is the simplest way to get the proofing language you want.

Language detection after typing: Users can select and switch the language for part of a text through Review > Language > Set Proofing Language if they want to do it after they typed it. The language list is long and you must scroll the alphabetical list. 

Using OneNote App for Windows

We made some improvements to this experience in our Windows app, which has this feature through Right Click > Set Language submenu. The frequently used languages will show up in this submenu, so you don’t have to go through the language list again. The language list also has a search box to make it easier to find a language there.

Denise, OneNote Team
