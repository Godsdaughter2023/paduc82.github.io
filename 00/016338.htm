BINOM.DIST returns #NUM! for number_s equal to zero when - Microsoft Community

maarten kruijver say:

BINOM.DIST returns #NUM! for number_s equal to zero when probability_s is equal to one

According to the documentation, BINOM.DIST returns the individual term binomial distribution probability.  The function has four arguments:

  1. number_s: number of successes
  2. trials: number of trials
  3. probability_s: success probability
  4. cumulative: logical to indicate whether cumulative distribution should be returned. We set this to FALSE.

BINOM.DIST incorrectly returns #NUM! when the success probability is 1.0 and the number of successes is 0 (and trials >0). The correct output should be 0, since the probability of 0 successes is 0 when the success probability is 1 and the number of trials is positive.

This has not changed in more current versions.

The root cause of the problem is the way that Excel stores numbers and performs arithmetic, called 64-bit binary floating-point.  Consequently, most non-integers (and integers larger than 2^53) cannot be represented exactly.  This causes small anomalies ("numerical errors", but not defects) in calculations.

The necessary work-around is to round calculations that you expect to have a specific degree of precision, e.g. 2 decimal places for dollar-and-cents values.

There are two ways to accomplish that:  (a) use the ROUND function, e.g. ROUND(A1-A2,2); or (b) format cells appropriately (e.g. Number with 2 decimal places) and set the "Precision as displayed" (PAD) calculation option (in XL2003, click on Tools, then Options, then the Calculation tab).

I prefer #a, explicit rounding.  Except for #b (PAD), changing the cell format alone affects only the appearance of the value, not the underlying value.

To illustrate, the following is the actual exact values that Excel uses for you example.

A1,94341.41:
A2,95291.22:
A1-A2:

A1,94341.41:   94341.4100000000,0349245965480804443359375
A2,95291.22:   95291.2200000000,0116415321826934814453125
A3,=A1-A2:      -949.809999999997,6716935634613037109375

You cannot see these exact values because Excel formats only up to the first 15 significant digits to the left of the comma in my presentation.

You could see the inaccurate result in step 3 if you format A3 as Number with 12 decimal places.

You see the difference in the Formula Bar after the paste-special-value in step 4 because Excel formats the Formula Bar according to its own rules, in this case different from the cell format.

jay freedman say:

Most likely what happened is that the Word window is not minimized, but is located outside the screen area. Here's how to get it back:

- Open Word, and open one other program.

- Right-click the task bar and choose either "Tile vertical" or "Tile horizontal". That should bring Word onto the screen, with each window getting half.

- Close both programs. Then re-open Word, and it should appear in the same location and size. Now resize it to your preference.

lucialabanino say:
Hi TWDbag , 
My name is Lucia. I am an Independent Advisor. Thanks for contacting our support.

What is happening is that the rules that are created in Outlook are of the Inbox Rule type, this means that they will only be activated for incoming emails that arrive in the mailbox, not outgoing ones.

That is, it only works for messages received not for messages sent by your own mailbox.
---------------------
If you want the emails that you send to a specific recipient to be ordered in a certain folder, you must do that movement manually by activating the rule assigned to that folder.

It is located in the "sent items" folder and in the Move option in the main menu, it displays and selects the rule, in this way all the mails will be moved to the selected folder.
