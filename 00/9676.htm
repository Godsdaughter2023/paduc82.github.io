a. user say:

Looking deeper, I realise there is a problem with my on-screen keyboard (OSK) suggestion: when you try to enter niqqud (vowel points) using AltGr + the appropriate letter using the OSK, Word actually inserts the wrong character (e.g. a Copyright symbol when you use AltGr-Beth). Using a regular keyboard works OK. I had a look back and realise that this has been a problem for some time, but it only affects some programs (e.g.,you can type OK in NotePad). I suppose what that means is that Word is detecting the keypresses and processing them itself. In Notepad, which probably does not try to do that, typing using the OSK works much better. 

But to answer your question about the OSK, when you choose a different language/input method, by clicking the language icon near the bottom right corner of your screen, the OSK layout should change as well:

I had never really used the Insert Symbol method for inserting more than a few characters, but experiments suggest that when the font is designed for Hebrew, inserting a space from that font causes Word to treat it asan RTL space, whereas if the font is designed for LTR and general use, inserting a space causes Word to treat it as an LTR space. (And that could change if you set your WIndows interface to use an RTL script - I really do not know).

However, Unicode does not use two separate codes for RTL and LTR spaces - there is just one, the 0x0020 character. Unicode actually defines an algorithm that should be used to determine the how LTR, RTL and mixed direction character sequences should appear on screen or paper, so if you have Aleph, Beth Gimel followed by a space, as far as Unicode is concerned that should be treated as an RTL sequence. Unicode has special characters for overriding its usual behavious. You can also insert those characters in Word via Insert->Symbol->SPecial Characters. However, that Unicode algorithm is really about how a stored sequence of characters should be displayed, whereas Windows an/or Word do not appear to use that algorithm to determine how a sequence of characters you type should appear. 

FWIW, when Word stores text internally in a .docx document it does so as a sequence of "runs". Each "run" can have a set of "run parameters", which might for example define that the run should be bolded, or a particular font or size. The run parameters also let you defined the text direction. So when you type aleph, beth, followed by a space on an English QWERTY keyboard, Word actually stored one RTL run with three Hebrew characters and one LTR run with a space, It's like this although I have simplified it a bit):

<w:r>
  <w:rPr>
    <w:rtl/>

  </w:rPr>
  <w:t>אב</w:t>
</w:r>
<w:r>

  <w:t xml:space="preserve"> </w:t></w:r>
</w:r>

THe <w:rtl\> element says that the run should be RTL.

If you actually type the space on a Hebrew keyboard you get this instead.

<w:r>
  <w:rPr>
    <w:rtl/>

  </w:rPr>
  <w:t>אב</w:t>
</w:r>
<w:r>

  <w:rPr>

    <w:rFonts w:hint="cs"/>

    <w:rtl/>

  </w:rPr>

  <w:t xml:space="preserve"> </w:t></w:r>
</w:r>

In this case both runs are RTL, and the w:rFonts element is a hint to use the "Complex Script" font defined for the paragraph or its style.

diane poremsky mvp say:
Hi Eman,
Email calendar sends a copy of your calendar by email. It's a one time thing and only contains info for the included dates. Its static and will never update.

Share calendar gives the person you shared with the ability to view your entire calendar - with the permissions you selected. They can see it until you revoke permissions.

Publish calendar is similar to share but you publish it to a web server and anyone with the link can see the calendar. This calendar is updated hourly.

cookiekantode say:
hi i am an independent advisor and would like to help you resolve your issue

Calendar Publishing
By publishing an Outlook Calendar as an Internet Calendar you create special URLs that you can give to people outside your organization.

One URL (tipically ending with .ics) lets the recipient subscribe to your calendar by using Outlook or Webmail (Outlook Web App or OWA), while the other lets the recipient view your calendar as a webpage in a web browser.

You can control how much detail others can see but you have less control as with Calendar Sharing.

Calendar Publishing is a very practical solution if:

you want to reach a vast audience;
you need to include recipients outside the EUI;
you want recipents to only view your calendar (not to add and/or edit events).

->https://support.office.com/en-us/article/introd...

Calendar Sharing
Calendar Sharing is more powerful as it allows for users to invite people inside their organisation to see their calendar and, if proper access is granted, even to add or edit events. Users who share their calendars can specify how much detail the recipients can see and/or modify.

Please note that even if users don’t share their calendars, others in their organization can still see free or busy information (so-called Free/Busy Time).

Users can give recipients access to more detail when they share their calendar compared to Calendar Publishing.

Calendar sharing is available mainly in Outlook or Webmail (Outlook Web App or OWA) where you can display a shared calendar next to yours.

Please note you can NOT share your calendar with people outside your own organization.

Calendar Sharing should be used if:

you want to share your calendar with a restricted audience (such as your work unti);
you do not need to include recipenjts outside of the EUI;
you want to detail sharing access;
you want others to schedule events on your behalf.
