goranzg say:

Hi,

I just registered here to give a solution which I found for this issue. I have detected what seems to be a rounding error in WLMM, and it manifests itself when you use the Split function to cut out parts of your source clips. It seems that Split will cut the clip not always at a frame boundary, but using floating point representation of seconds. This has the effect of not always "snapping" to exact timepoints of frames, giving choppy playback.

How it manifested in my case: I was editing a series of MP4 format files from my dashcam: it splits the recording to multiple source files, each approximately 1 minute in duration, 1920x1080, 30 fps (not 29.97!). Importantly, all files have identical format and frame rate, and come from the same source and same recording session.

My movie I was editing was basically a highlight reel from a long drive: it consisted of about 50 short edits from the source files pieced together, but importantly practically each one I trimmed specifically using the Split button: I would import the whole 1 minute clip, mark a position, click Split, and then delete the unwanted portion.

When I rendered my driving movie and played it back, some (not all) of these cut sections looked like 20 fps video. More specifically, examining the output frame-by-frame, every 3rd frame was a duplicate of the previous one. However, some sections of the output were perfectly smooth. This ruled out general incompatibilities like source video format, frame rate conversions along the processing path, codec issues or similar - as basically all clips were processed the same way from source to final video. (None of the source video clips were sped up or slowed down.)

I was sure the suggestions given here previously didn't apply: either the entire output video would be choppy, or issues would not be specific to sections of the source files. There was no way that bitrate settings, video driver or codec issues could have caused anything like this. My playback and codecs are set up properly and there was no way this would be the cause.

My assumption on the cause: I had a look in the XML file that is the WLMM project file, and noticed that all timestamps were in floating point representation of seconds with many decimal places showing approximations of fractions. This led me to believe the trimming of the beginning and end of video files might be the root cause: a rounding error in trimming the file would request frames from the source file on nonexistent timestamps which may cause the codec the return the previous frame from the sequence as the "next closest fit" to the requested timestamp.

How I solved it: I went through the project, selecting every clip one by one, and selecting the Trim tool. The Trim tool has two time fields, start and stop time, represented as floating point seconds. But importantly each field has an "up" and "down" button to increase and decrease the value. I simply clicked once "up" and then once "down" for each field and for each clip. This has the effect of "snapping to frames" basically. Occasionally for example the initial value would be for example "10.23". Clicking "up" would bring it to "10.25". Clicking "down" would bring it to "10.22" (but not 10.23!).

Note however that this didn't work fully the first time round. Still several clips were dropping frames: for those, a bit of fiddling with the values, effectively trimming the clip a few frames left or right, fixed it. Basically trial and error and several exports of the final file.

I know this comes quite late after the original post, but I hope this helps someone.

If you want to disable the pop-up notification of Windows Live Family Safety filter. Please follow the steps below:
