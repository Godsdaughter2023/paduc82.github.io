jllatham say:

Well, on your own computer you could require a password just to open the file, but that's no good since you have to give such a password to the users who would presumably pass it along with the filched copy of the file.

Workbook and Worksheet 'protection' are not security measures.  If you read the help topics on them you will find that they explicitely say they are not for security.  There are any number of free code samples and complete tools available on the internet to 'crack' such passwords in mere minutes, if not mere seconds.

Now, the security for the macros section of a workbook is a bit tougher to crack - although there are tools available to do it.  Most of the good ones cost $$.  I have one of them myself.

But if you will go into the VBA editor and use Tools --> VBAProject Properties, then go to the [Protection] tab in the window and check the box next to "Lock project for viewing", and then enter and confirm a good strong password, you will have done as much as you reasonably can.

Be sure and do 1 of 2 things (if not both) if you go this route:

#1 - write the password you use to protect the VBA Project down somewhere, where you can recover it later.

#2 - save a copy of the workbook WITHOUT protecting the VBA Project.

The reason for this is so that you can work on the workbook/code in the future without having to pay $$ to go buy a tool to get you back into the project !!

 

Now - as for the REGKEY or licensing side of things - since Excel's 'security' is so inherently weak, it may not be worth the effort or cost of hooking up with a company to provide such service.  Picture this: your (protected) code does the validation of the license information in some fashion -- as soon as someone manages to remove the lock on the VBA project, they can examine the code and nullify the security efforts very easily.
