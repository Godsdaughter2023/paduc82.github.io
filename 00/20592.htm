a. user say:

re. the thing about keeping the index open in a separate window, ou may be able to use the split document view to do that, although I cannot see how right now.

Alternatively, you could create a completely separate document that contains

 a. an RD field that references your document

 b. a copy of the INDEX field that you have in your document.

Then, you could keep your main document and your "RD" document open, mark up entries in the main document, and update the INDEX field in the "RD" document either after each entry is marke, or each time you wanted to check your index.

So if for example your main document is c:\a\maindocument1.docx" and had an INDEX field like

{ INDEX \c "2" }

your RD document would contain

{ RD "c:\\a\\maindocument1.docx" } 

{ INDEX \c "2" }

(Where all the {} pairs are the special field code braces that you can insert using ctrl-F9 on Windows Word).

    Application.GoBack

    ' document where you are working
End Sub

jay freedman say:

Although that macro does the job, it's inefficient to select things (which moves the cursor and forces Word to redraw the screen) and then have to move the cursor back to its original position. A better plan is to use the properties of the fields themselves to determine where the changes are made. Also, VBA knows where the index is, separately from its membership in the Fields collection, so you don't have to put the index at the beginning of the document unless you need to do it for some other reason.

The following macro does the same work as yours with less fuss:

Sub Highlight_XE_Fields()
    Dim fld As Field
    
    For Each fld In ActiveDocument.Fields
        If fld.Type = wdFieldIndexEntry Then
            fld.Code.Font.ColorIndex = wdDarkRed
        End If
    Next
    
    ActiveDocument.Indexes(1).Update
End Sub

jonasweibel say:

Okay, i could resolve the problem at least for our infopath form.

 

Our form was marked as protected in infopath 2003. This configuration is not available anymore in infopath 2010 (see http://social.microsoft.com/Forums/en-US/Offtopic/thread/2c058000-6db6-4f2c-ade0-3554190eb830) but still in the form's definition. I created a InfoPath 2007 Form and looked for differences in its MSF file after i marked it as protected.

 

It's the allowCustomization attribute which indicates the form as protected and which I removed in the XSF file:

 <xsf:applicationParameters application="InfoPath Design Mode">
  <xsf:solutionProperties fullyEditableNamespace="http://schemas.microsoft.com/office/infopath/2003/myXSD/2013-04-29T06:38:05" lastOpenView="view1.xsl" allowCustomization="no"></xsf:solutionProperties>
 </xsf:applicationParameters>

 

After a republish the form worked again (make sure you delete your local cache: http://blogs.msdn.com/b/michael_yeager/archive/2008/07/09/infopath-cache.aspx)

karl timmermans [outlook mvp 2012-2018] say:

What is the difference between IMAP and POP3 A/c server . While setting up the company email in outlook 2016, Which is the best choice to set up an email in outlook

#1 - To learn about the differences between POP3 and Imap - just do a search using the phrase "difference between pop3 and imap" (or any variation thereof) and you will have no shortage of results providing explanations and suggestions.

#2 - As for which is the "best" choice or which protocol is "better" - comes down to what you need/want to do along with a number of other factors. There are those who are almost religious about Imap being the "only" choice anyone should make - am not in that camp - there is no such things as a "one size fits all" answer to this question. Each has its place depending on your specific individual scenario but as Palcouk said, if you want to access the same email info on multiple devices - POP3 is not an option. If you want to share everything (contacts/calendar etc) across multiple devices - then either using a hybrid approach or using an Exchange based account is going to be the right choice.
