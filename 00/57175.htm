Getting the error was not migrated due to partial or ambiguous match usually appears when an incompatible driver has been installed or trying to be installed on your PC. To work around this problem, please check out the following suggestions:

Suggestion 1: Rollback device driver.

  1. Open Device Manager.
  2. Double-click on the device for which you want to rollback the driver.
  3. Click the Driver tab and then click Roll back driver.
  4. Follow the on-screen instructions.

Suggestion 2: Uninstall device driver.

  1. Open Device Manager.
  2. Right-click the Unknown USB device and click Uninstall. Make sure to check the box for Delete the driver software for this device.
  3. Restart your computer.

The customer has upgraded to pro. The computers are not domain just PC name and workgroup. What would be the easiest way to have to allow them to connect to the Router. I've tried porting on the Router to no Avail. 

al jarvi say:

Since the Remote Desktop [RDC] host computer you want to connect to remotely is behind a DSL modem/router, you need to open/forward TCP Port 3389 through that device to the private LAN IP of the host PC, as noted by Jack. You call that from the remote RDC client PC using the public IP of the DSL modem/router or use a fully qualified domain name [FQDN], ie. yourFQDN.noip.com for example. On the RDC host PC you also need to make sure the Windows Firewall (or any other third party firewall) has a port exception for Remote Desktop or TCP Port 3389.

http://windows.microsoft.com/en-US/windows7/search?prd=Windows7&q=port%20exception

To get your public IP go to the http://www.whatismyip.com site while on the RDC host PC. Use that to call from the remote RDC client. If you want to use a FQDN you can sign up for a free one from organizations like No-IP.com or DynDNS or similar.

http://www.no-ip.com

http://www.dyndns.com

The way that works is you run a small program on your RDC host PC that communicates on a time scheduled basis. The No-IP.com or DynDNS servers then know your current public IP and map that to your custome FQDN. That information is propagated across the public internet to name servers allowing you to call versus the FQND versus the public IP. Note that most residential customers get dynamic IPs from their ISP makeing it much easier to call using a FQDN versus a public IP that may change each time you connect to your ISP.

You can test port forwarding through the DSL modem/router by going to the http://www.canyouseeme.org site while logged on the RDC host PC and test for TCP Port 3389. If the test fails either port forwarding in the DSL modem/router or a firewall is not configured correctly or your ISP is blocking TCP Port 3389 inbound to your DSL modem/router. In most cases this latter case is very unlikely however.

Note there is no need to setup a VPN or open mulitple ports on the DSL modem/router since you only have one RDC host behind it that your trying to remotely control.

What DSL modem/router do you have? If you call that out perhaps someone can help further with port forwarding issues. You can also check the http://www.portforward.com site for help.

Is the RDC client PC on a work network or a home network? If its the former its possible the network admins have blocked TCP Port 3389 outbound or use a proxy.

In all cases make sure the RDC login account uses a strong password...

https://www.microsoft.com/protect/fraud/passwords/checker.aspx?WT.mc_id=Site_Link

...and that both the RDC host PC and the RDC client PC are configured to use Network Level Authentication [NLA].

http://windows.microsoft.com/en-US/windows7/What-types-of-Remote-Desktop-connections-should-I-allow

http://theillustratednetwork.mvps.org/ScreenShots/RDC-NLAandServerAuth/Host-UseNLA.jpg

http://theillustratednetwork.mvps.org/ScreenShots/RDC-NLAandServerAuth/ClienttoServer-UseServerAuth.jpg

MS-MVP Windows Desktop Experience, "When all else fails, read the instructions"

jack mvp say:

Hi

Remote Desktop Connection uses port 3389 to listen to incoming connection.

When done over the Internet you need to forward the port to the computer's IP.

Alas, a Router can forward a specific port only to one computer.

Since you have two computers you need to change the listening port of one of them to a different port number and forward the new port toward the computer's IP.

Leave one computer as is using the default port 3389.

In the second computer, change the port to another port of your choice, http://support.microsoft.com/kb/306759/en-us

Assign static IPs to the computers (make sure that it is within the DHCP range).

In the Router, open port 3389 toward the IP of the computer with the default setting.

Open the port that you used for the second computer toward its IP.

When you connect, if the connection is a default connection, it would go to one computer. When the connection is done with to the second port, it would go to the second computer.

This page describe other Remote Program but it depicts the principle in more details,http://www.ezlan.net/vnc.html   (toward the end of the page).

Method to deal with external IP here, http://www.ezlan.net/myip.html

Windows 7 RDT, http://windows.microsoft.com/en-US/windows7/allow-remote-desktop-connections-from-outside-your-home-network
