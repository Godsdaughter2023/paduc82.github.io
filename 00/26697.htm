jay freedman say:

I discovered a quirk in VBA’s Find function when it’s used on comments: If the comment contains exactly the string you’re searching for but nothing else, then Find doesn’t match it. If there is at least one other character—a space, a letter or digit, or punctuation—then the Find succeeds. I believe this is a bug in the code behind the Find function itself. I’ve tested in Word 2013, 2010, and 2003 with the same results. Typically, when a bug is that old, Microsoft won’t fix it unless it results in data loss, for fear of breaking workarounds that assume the bug is present.

 

Fortunately, there is a completely different approach to your task that avoids the Find function, and I believe it works correctly regardless of the comment’s content. The InStr function tries to find a substring within a larger string, returning the starting index of the match if there is one, or zero if there is no match. The first argument of the function is the index within the larger string at which to start.

 

Sub Find_Scoresc()

Dim Crange As Range

Dim c As Comment

Dim TargetList() As String

Dim TriggerWords As String

Dim E1 As Integer, E2 As Integer

Dim element As Variant

Dim pos As Long

E1 = 0

E2 = 0

 

Application.ScreenUpdating = False

Application.DisplayAlerts = False

Application.DisplayStatusBar = True

TriggerWords = "E1, E2"

TargetList = Split(TriggerWords, ", ")

 

For Each c In ActiveDocument.Comments

For Each element In TargetList

pos = InStr(1, c.Range.Text, element)

While pos > 0

Set Crange = c.Range.Duplicate

Crange.Start = c.Range.Start + pos - 1

Crange.End = Crange.Start + Len(element) - 1

Crange.HighlightColorIndex = wdYellow

Select Case element

Case "E1"

E1 = E1 + 1

Case "E2"

E2 = E2 + 1

End Select

' prepare for next search in same comment

pos = InStr(pos + Len(element), c.Range.Text, element)

Wend

Next element

Next c

MsgBox ("Error Type 1= " & E1 & " and Error Type 2= " & E2)

 

Application.ScreenUpdating = True

Application.StatusBar = ""

Application.DisplayAlerts = True

End Sub

jay freedman say:

Using the Advanced Document Properties dialog will enable you to define a custom document property such as Company Address 2, but it won't add that property to the list at Insert > Quick Parts > Document Property. It also won't enable you to insert a content control in the document to display the value of the property.

You can display the property's value in the document by inserting a DOCPROPERTY field (Insert > Field, then select DocProperty in the field name list and choose the property's name in the Field Properties list).

However, you can't change the value of the property by editing the field result, the way you can change the built-in properties' values by editing the corresponding content control.

To make a custom document property behave like the built-in ones (with the exception of the Insert > Quick Parts > Document Property list), you have to "map" a content control to the property. That can be done with the add-in you can download from http://gregmaxey.mvps.org/word_tip_pages/cc_var_bm_doc_prop_tools_addin.html.
