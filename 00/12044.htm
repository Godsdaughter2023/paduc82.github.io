I believe Exchange provides options for automatic messaging, but you could probably do it with an Outlook script run from a rule that identifies the messages in question. The script would be something like

Sub Acknowledgement(Item As Outlook.MailItem)
Const strReply As String = "This is an automatic reply.<br/>" & _
      "Your message has been received and will be processed shortly"

Dim olOutMail As Outlook.MailItem
    With Item
        Set olOutMail = .Reply
        With olOutMail
            .HTMLBody = strReply & .HTMLBody
            .Display        'Change to .Send after testing
        End With
        Set olOutMail = Nothing
    End With
End Sub

If you select an already received message, you can test the above script using the following macro, to enable you to fine tune the message. You can use html tags to format the text.

Sub Test()

End Sub

The directions vary slightly depending on the version of Windows that's on the computer from which you're copying the font.

Start Windows Explorer (in Windows 7 this is the Computer item on the Start menu; in some other Windows versions it's called "My Computer" or something equally stupid). Go to the folder C:\Windows\Fonts (this is the same in all Windows versions).

SetWarnings False

before the queries you want to run; BE SURE to put a line

Dim qd As DAO.Querydef

Always a good idea to start with believing what the error message is trying to tell you. In this case it appears one of the data points does not fit the table design. I would proceed as follows:

1. Import last week's file. Does it still work correctly? Assuming yes that tells you that this week's data is different.

2. Create new spreadsheet with first half of the rows of this week's data. Does it import correctly? If yes, try second half. The error will occur.

3. Cut the error half in halfs. Repeat until you can locate the offending row.


My guess is a number is overflowing an integer, or the text is longer than a text field allows (this can happen with embedded CrLf).


Alternatively you can LInk to your spreadsheet, and use an Append query to copy the data. That process may point out the offending line more quickly.

john w. vinson say:

A Join is typically between the Primary Key of the parent table (here Properties), or a uniquely indexed field in the parent table, to a foreign key field in the Child table. Your Properties table has ID (an autonumber I guess) as the Primary Key, but you're trying to join from Landlord_ID - which is NOT the primary key. Access is complaining because Properties.Landlord_ID does not have a unique Index.

The design is flawed here. You need to define the real-life relationships: e.g.

Every Property must have One and only one Landlord

Every Landlord can manage zero, one, or several Properties

would be a typical one to many relationship.  

Each table should have its own unique primary key; Access calls it "ID" but I'd change it to be more self documenting - PropertyID for the Properties table, LandlordID for the Landlords table. The "One" side table would be Landlords (since one Landlord may have more than one Property), so the join would be from Landlords.LandlordID to a Long Integer field named LandlordID in the Properties table. There'd be nothing about the Property in the Landlords table, because there might be SEVERAL properties.

If your business rules are different (can one Property have several Landlords, over time perhaps?) you'll need to adjust the model, but this should give you a start.

On another subject, don't expect Autonumbers to be gapless or sequential. That's not their purpose; a primary key must be unique (only one property in the entire database has this specific propertyID) but that's ALL; autonumbers are not designed to be a count, or a sort order.
