Yes, provided that each of the queries references the controls in the form as its parameters.  But, taking up Gina's point, you only need separate queries if they are returning data from different tables in each case, or different columns from the same tables.

Subject to the above caveat I'd make a couple of points:

1.  Open a form or report based on the selected query, not the query itself as a datasheet, i.e. your 'Select Query' combo box should list the forms or reports.  You can make the first column of the combo box's list a hidden one of the form or report name, and the second one a plain English description, e.g. the combo box's RowSource property could be a value list:

"rptIncome";"Income report";"rptExpenditure";"Expenditure report"

Set the combo box's BoundColumn property to 1, its ColumnCount property to 2, and its ColumnWidths property to 0cm.  The list would then show as:

Income report
Expenditure report

The value of the control would be the relevant hidden report name, however.

2.  In the button's Click event procedure the code to open the relevant report in print preview would, assuming the first combo box is named cboReports, be:

Dim varReport As Variant

varReport = Me.cboReports

If Not IsNull(varReport) Then
    DoCmd.OpenReport, varReport, View:=acViewPreview
Else
    MsgBox "No report selected.", vbExclamation, "Invalid operation"
End If

3.  The method you've used for allowing a control to be left empty:

Like "*" & [Forms]![SearchRemitPro]![AccountNumber] & "*"

is not totally reliable as it will, in the event of the control being left empty, not return any rows where the column is Null.  A better method is:

[Forms]![SearchRemitPro]![AccountNumber] OR [Forms]![SearchRemitPro]![AccountNumber] IS NULL

ken sheridan say:

Are you saying that if ANY of the columns in the query (that are tied to this search form) are blank (null), that no records will be returned?



No; using the LIKE operator, if there is a NULL in a row in the table in the column which corresponds to a control you've left blank (NULL), then that row won't be returned as NULL LIKE "" evaluates to NULL, neither TRUE nor FALSE.  Testing for the parameter being NULL in a parenthesized Boolean OR operation on the other hand will return that row because, if the parameter is NULL, the parenthesized expression will evaluate to TRUE for every row, so all rows will be returned.  It is also a more efficient method as it allows the optimizer to make use of the indexes.

ken sheridan say: A subform is not a member of the Forms collection.  To reference it you do so via the Form property of its parent form.  The syntax is:

Forms!ParentFormName!SubformControlName.Form!ControlName

Note that SubformControlName is the name of the subform control in the parent form's Controls collection, i.e. the control which house the subform, which may or may not be the same name as its source form object (Notes_Banking_Search in your case).  ControlName is the name of the referenced control in that form (AccountNumber in your case).

I don't use navigation forms myself, but I'm assuming the syntax is the same as with a conventional form/subform.
