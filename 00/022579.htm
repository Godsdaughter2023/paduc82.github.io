I'm all but certain that your table design needs rethinking. A Title field would typically be Text or even Long Text (Memo). ken sheridan say:

.................I have 7 tables of each with titles, authors, etc. fields...............

Having multiple tables with the same columns (fields) is bad relational database design.  It encodes data as table names.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

I assume each table represents a different attribute, in which case you should have a single table with an additional column representing the attribute.  You can easily coalesce the seven tables into one by adding the additional column to one of them, and then executing six 'append' queries with the value to be inserted into the additional column as a different constant in each case, e.g. if one attribute was 'Foo' and another 'Bar' you would return the constants 'Foo' or 'Bar' as the constant in each of the two queries.

Having coalesced the tables into one table it is then an easy task to identify duplicates with a query GROUPed BY the columns which determine the duplication HAVING COUNT(*) > 1.

ken sheridan say: To find duplicates after inserting a row into a table you don't need a JOIN, you need a UNION ALL operation:

SELECT Title
FROM
    (SELECT Title
     FROM [B&N scottwallstreetmehring]
     UNION ALL
     SELECT Title
     FROM [Book Collection]
     UNION ALL
     SELECT Title
     FROM [B&N swm258])
GROUP BY Title
HAVING COUNT(*) >1;

Alternatively you can check to see if a title exists in any table before inserting a row into any of the tables with:

PARAMETERS [Enter title:] TEXT(255);
SELECT "B&N scottwallstreetmehring" AS TableName, Title
FROM [B&N scottwallstreetmehring]
WHERE Title = [Enter title:]
UNION ALL
SELECT "Book Collection", Title
FROM [Book Collection]
WHERE Title = [Enter title:]
UNION ALL
SELECT "B&N swm258", Title
FROM [B&N swm258]
WHERE Title = [Enter title:];

In either case just extend the UNION ALL operations to bring in all seven tables.

But you would not need to torture the data until it confesses like this if the books entity type were correctly modelled as a single table, in which a column differentiates the different 'categories', or whatever attribute each table's name is currently encoding contrary to the Information Principle.

rudyk2evy say: Thanks for your comment...

I have done some more work on this problem and am certain that this is not a new "infection."  What has happened is that AVG has added this virus definition in a recent upgrade.  If you maintain clone type backups you can scan with AVG before allowing the program to upgrade and the disc will appear clean.  Then allow AVG to download the latest definitions and the "virus" will appear.

Luhe.Fiha.A appears more than once but always in the C:\windows\installer folder (must unhide system files to see this folder).  Since .msi files are storage files the infection can be found more than once within a single file by AVG.  It may not be possible to locate every specific instance but deleting one file in the installer folder will remove more than one instance of the "virus."  Look in the C:windows\installer for any of the files AVG finds and delete.

AVG has trouble removing large files and placing them in the AVG "vault."  Find the file/files in Windows explorer and delete from there.  You will probably have to delete again from the Recycle Bin.

This installer files involved appear to be related to Roxio install.  My install was from a retail box disc but Roxio might have updated on line so I can't be sure of the origin of the problem files.  I don't see any problems after deleting the files and Roxio seems to be running fine.  However, I think it is possible that if the need arose to uninstall Roxio I might not get a completely clean uninstall.

On reflection the best and easiest recourse might be to just tell AVG to "ignore" this "infection."  Is this thing actually a virus?   or an infection?  I have seen no operational problems, nothing in chkdsk, sfc, Registry Mechanic, etc., to concern me.
