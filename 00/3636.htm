The following data validation formula works in 2003 but not 2010. - Microsoft Community

salguod11 say:

The following data validation formula works in 2003 but not 2010. Could someone confirm this?

I am creating a Purchase Order (PO) template for my company and am using ODBC (Data-->Microsoft Query) to import into Excel various pieces of data from our accounting software.  One step in the creation process is to select the vendor who will be receiving the PO.

Currently, we have an adequate solution.  Using the named range Vendor ID (located on "Vendors"!$A:$A) I am using Data Validation (list) to show, alphabetically, all 1205 vendors currently in our master file.

Our users are finding this drop-down list cumbersome, however, due to its length.  Brainstorming with the user and searching the internet, I found an improved solution.  The author, usingExcel 2003, presented a list of alphabetically sorted names in column B and set up in column A the following list:  A-Names, B-Names,...Z-Names.  Two Name Ranges were set up:  AlphaList, defined as =Lists!$A$1:$A$26 and NameList, defined as =Lists!$B$1:INDEX(Lists!$B:$B,MATCH("zzz",Lists!$B:$B)).  On a separate worksheet, a cell was chosen and set up using Data Validation (list) with the following formula in the Source field:

=IF(TRIM(B2)="",AlphaList,INDEX(NameList,IF(ISERROR(MATCH(LEFT(B2)&"*",NameList,0)),MATCH("*",NameList),MATCH(LEFT(B2)&"*",NameList,0))):INDEX(NameList,MATCH(LEFT(B2)&"zzz",NameList)))

What's neat about this formula is that when you click on the arrow in B2 the first time, the AlphaList appears (i.e., A-Names, B-Names, etc.).  When you select an item from that list (say, D-Names)and then click on the same arrow a second time, only the names that start with "D" from the NameList appear.  Managing a long list such as our vendor master in this fashion would significantly improve the efficiency of PO creation.

When I try to replicate this logic, verbatim, using Office 2010, I get the following error message after keystroking the formula noted above in the source field of the Data Validation window:

"You may not use reference operators or array constants (such as unions, intersections or ranges) for Data Validation criteria."

So: 1) is there an error in my keystroking that I just don't see?  2) did some programming logic change in Excel 2010 which makes the formula fail?  3) is there a better formula to use (certainly beyond my skill set) that would accomplish our objective?

Any assistance would be greatly appreciated.  I need to roll out this new PO template as soon as possible.  Thank in advance for your time!

f z z say:

The error message states the problem: Excel 2010 won't accept reference operators. That means Excel 2010 won't accept INDEX(...):INDEX(...) expressions.

Your entry cell appears to be B2 in the current worksheet. Define the following names to make it easier to work with: EFC as =LEFT(TRIM($B$2)), and EFCFO as =MATCH(EFC&"*",NameList,0)-1. Then define NameListSubset as

=IF(COUNT(EFCFO),OFFSET(NameList,EFCFO,0,COUNTIF(NameList,EFC&"*"),1),NameList)

and finally change your validation formula to

=IF(EFC="",AlphaList,NameListSubset)

NB: EFC stands for Entry First Character, EFCFO stands for Entry First Character First Occurrence. I prefer avoiding volatile functions in cell formulas, but validation formulas are only calculated upon entry into cells with validation rules, so they don't affect recalc speed. In this case, OFFSET is hugely better than INDEX(...):INDEX(...).
