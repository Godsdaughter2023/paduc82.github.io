Re-worked the UDF to ignore 0's and/or blanks.

No need to trap for 0's in formulas.

Amended version. . . . . .

Function ConCatRange(CellBlock As Range) As String
'for non-contiguous cells  =ccr((a1:a10,c4,c6,e1:e5))
'note double parens
Dim Cell As Range
Dim sbuf As String
    For Each Cell In CellBlock
    On Error GoTo fred
    If Len(Cell.text) > 0 And Cell.text <> "0" Then sbuf = sbuf & Cell.text & ", "
        Next
    ConCatRange = Left(sbuf, Len(sbuf) - 2)
fred:
End Function

doug robbins - mvp office apps & services (word) say:

The following code will perform the required sort

 

Dim i As Long
With ActiveWorkbook.Worksheets("Sheet1").Range("A1")
    For i = 2 To .CurrentRegion.Rows.Count
        .Offset(i - 1, 2).Value = Replace(.Offset(i - 1, 2), "CST", "")
    Next i
    i = .CurrentRegion.Rows.Count
End With
With ActiveWorkbook.Worksheets("Sheet1").Sort
    .SortFields.Clear
    .SortFields.Add Key:=Range("C2:C" & i), _
        SortOn:=xlSortOnValues, Order:=xlAscending, DataOption:=xlSortNormal
    .SetRange Range("A1:D" & i)
    .Header = xlYes
    .MatchCase = False
    .Orientation = xlTopToBottom
    .SortMethod = xlPinYin
    .Apply
End With
With ActiveWorkbook.Worksheets("Sheet1").Range("A1")
    For i = 2 To .CurrentRegion.Rows.Count
        .Offset(i - 1, 2).Value = .Offset(i - 1, 2).Value & " CST"
    Next i
End With

albert d. kallal say:

If you hold your cursor over the control name, you see this:

 

<quote>
Enter the name of the control that corresponds to the subform or subreport to be filtered or leave the control blank for the current form.
</quote>

 

So, if you trying to filter the current form then leave the control name part blank.

 

So your filter code in the after update of the text box is:

 

SetLocalVar
    Name   strSearch
Expression = [txtBoxPromptOnForm]
SetFilter:
   Where Condition = [Firstname] = [LocalVars]![strSearch]

 

Now, in the above, note how I stuffed the text box value into a local variable. I did not have to do this, but then I would have to use a "nasty" form! reference and the filter command would have to include the form name like this:

 

Where Condition = [FirstName]=[Forms]![WEBSearchCustomers2]![txtSearch]

 

So, while in web we don't have a me! to reference to current form like in VBA, WORSE is the filter command requires a fully qualified forms! name. So the above trick of FIRST stuffing the text box control into a variable means we don't have to hard code the forms! reference. This works due to the fact that setVar, and setLocalVar does NOT need a forms! qualifier to reference a text box - but the filter command does).

 

So, stuff the text box into a variable is a great tip and no hard code forms! references are required. Thus if you now copy the form, rename the form, or even run your code in a sub form, then no change in the code and there are no "hard" coded forms references in your code.

 

In effect your filter code can work with just one line of code, but I tend to use the "extra" line of code to stuff the text box into a variable since then that eliminates the hard coded forms! reference.

 

Also, keep in mind that a filter is a EXPRESISON and NOT a string expression.

http://www.kallal.ca/searchw/WebSoundex.htm

 

I don't recommend the soundex part of the code, but the above example does give an example of how some VBA code would be converted to web code.
