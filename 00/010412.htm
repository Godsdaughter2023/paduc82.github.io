ken sheridan say: You have many-to-many relationship type between modules and parts which should be modelled by three tables, in broad outline like this:

Modules
….ModuleNumber (PK)
….ModuleDescription

Parts
….PartNumber (PK)
….PartDescription

and to model the relationship type:

ModuleParts
….ModuleNumber (FK)
….PartNumber (FK)
….Quantity

The primary key of the last table is a composite of the two foreign keys.  Even if you redundantly add a surrogate autonumber key to this table, the other two foreign key columns still constitute a candidate key, and would have to be defined as such by their inclusion in a single unique index.  Making them a composite primary key is simpler.

The interface for data entry would be a modules form, in single form view, and within it a ModuleParts subform in continuous forms view, linked to the parent form on PartNumber.

The above assumes that a module is composed of 'atomic' parts, none of which are themselves modules (sub-assemblies) made up of parts, which might be 'atomic' or further sub-assemblies.  If this were the case the model would be a 'bill of materials' type of model, which is simple in structure, but in which it is difficult to compute the bill of materials (aka 'parts explosion') as it requires recursive querying, which Access does not support.  There are standard algorithms for this, however, and its possible to simulate recursive querying in Access.  You'll find and example in BoM.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

The zip archive includes two files, BoM.accdb, which simulates recursive querying to an arbitrary number of levels by progressively writing data to a temporary external database created at runtime; and PartsTree.accdb which returns the bill of materials to a fixed number of levels by means of a query which joins multiple instances of the same table.

ken sheridan say: The total number of rows in the ModuleParts table will be the sum of the total number of parts per module.  So if module x is made up of parts a, b and c and module y is made up of parts c,d and e, the rows in the table would be:

Module    Part
x               a
x               b
x               c
y               b
y               c
y              d

As you can see, duplicate parts, b and c in this case, can be included.  What this table does is model the many-to-many relationship type between modules and parts by resolving it into two one-to-many relationship types.  You'll sometime see a table of this type colloquially referred to as a junction table.

Modelling a bill of materials, on the other hand, is done differently.  In this case each assembly, sub-assembly and atomic part is a 'part', and is included in a single Parts table, so you might have:

Part
Assembly A
Sub Assembly B
Sub Assembly C
Sub Assembly D
Atomic Part X
Atomic Part Y
Atomic part Z

The bill of materials for Assembly A is modelled in an adjacency list table like this:

MajorPart                 MinorPart                Quantity
Assembly A              SubAssembly B       2
Assembly A              SubAssembly C       3
Sub Assembly B       Sub Assembly C      1
Sub Assembly B       Atomic part X         4
Sub AssemblyC        Sub Assembly D     2    
Sub AssemblyC        Atomic part X         4
Sub AssemblyC        Atomic part Y         5
Sub Assembly D      Atomic part X         3
Sub Assembly D      Atomic part Z         2

The MajorPart and MinorPart columns are each a foreign key referencing the primary key of the Parts table.  So:

Assembly A comprises 2 x Sub Assembly B and 3 x Sub Assembly C.  
Sub Assembly B comprises 1 x Sub Assembly C and 4 x Atomic part X.
Sub Assembly C comprises 2 x Sub Assembly D, 4 x Atomic part X and 5 x Atomic part Y.
Sub Assembly D comprises 3 x Atomic part X and 2 x Atomic part Z.

An adjacency list table like this is a very simple data structure, but can model the bill of materials of an assembly of any complexity to an arbitrary number of levels.  As I said before, processing the data to build a bill of materials to an arbitrary number of levels is not straightforward due to the need for recursive querying, which Access does not support, but it can be simulated, as my demo illustrates.
