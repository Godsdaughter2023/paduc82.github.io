Is it possible to allow a user edit text alignment in an unprotected - Microsoft Community

xavi-tan say:

Is it possible to allow a user edit text alignment in an unprotected field in a protected Word document?

Hello, Microsoft Community!

I have been asked by one of my clients if there is a manner in which she can edit the text alignment of an unprotected field (a "fillable" text field) in a protected Word document. Is this, at all, possible?

Thank you for any help!,

Xavi-tan

jay freedman say:

There's a distinction in wording here.

If you've used the "Select sections" link in the Restrict Editing pane to make a section of the document unprotected while the rest is protected for "Filling in forms", then yes, text alignment of paragraphs in the unprotected section can be changed.

If you're asking whether the alignment within a legacy text form field can be changed, the answer is no, it cannot.

If you need that capability -- and if the person filling the form is using Word for Windows 2007 or later -- then you can't use legacy form fields, but you must instead use content controls.

  • If you use plain text content controls, you can then set their properties to allow carriage returns, but the entire content is treated as a single paragraph; it will all have the same text alignment and the same paragraph style.
  • If you use rich text content controls, the individual paragraphs can be formatted separately as in regular text.
  • To prevent editing of anything outside the content controls, do not use the protection methods in the Restrict Editing pane. Instead, select the whole document (or whatever part of it you want to protect), click the Group button on the Developer tab, and click Group in the menu.

Build a temporary distribution list to hide addresses - Microsoft Community

song.usa say:

Build a temporary distribution list to hide addresses

My email VBA (see code below) will start Outlook and list addresses in Bcc. Is it possible to build a temporary distribution list to hide those addresses? I do not want the sender to copy all addresses from Bcc to steal it for other purposes.

Private Sub cmdEmailFaculty_Click()
   
    DoCmd.OpenForm "MassEmailCertify", acNormal, , , , acDialog
    Dim blnEval As Boolean
    blnEval = True
   
    If blnEval Then
   
    Dim db As DAO.Database
    Dim MailList As DAO.Recordset
    Dim MyOutlook As Outlook.Application
    Dim MyMail As Outlook.MailItem
    Dim Subjectline As String
    Dim strRecip As String
    Dim lngCount As Long
   
   
    DoCmd.SetWarnings False
    '  Now, we open Outlook for our own device..
    Set MyOutlook = New Outlook.Application
    lblStatus.Caption = ""

    ' Set up the database and query connections
    Set db = CurrentDb()

    DoCmd.OpenQuery "qryEmailInstSem"
    Set MailList = db.OpenRecordset("tblEmailSend")
    MailList.MoveFirst
    Subjectline = MailList("yyyy") & " " & MailList("Semester") & " "
   
    Do Until MailList.EOF
        lngCount = lngCount + 1
        lblStatus.Caption = "Email: " & CStr(lngCount)
        Me.Form.Repaint
   
        ' This addresses it
        strRecip = strRecip & "; " & MailList("email")
       
        'And on to the next one...
        MailList.MoveNext
    Loop
        ' This creates the e-mail
        Set MyMail = MyOutlook.CreateItem(olMailItem)
       
        With MyMail
       
        .BCC = Right$(strRecip, Len(strRecip) - 2)
        .Body = "Dear " & Subjectline & "faculty:" & vbNewLine & vbNewLine
       
        End With
   
    MyMail.Display
    'Cleanup after ourselves
    Set MyMail = Nothing
    Set MyOutlook = Nothing
    MailList.Close
    Set MailList = Nothing
    db.Close
    Set db = Nothing
    Me.lblStatus.Caption = ""
    Else
        MsgBox "This function is pending IT evaluation.", vbInformation, conAppName
    End If
    DoCmd.SetWarnings True
End Sub

bob jones aka: cybertaz say:
What you're witnessing is Widow/Orphan Control in action. It's a paragraph formatting attribute which prevents paragraphs stranding a single line at the beginning/end of a page. The attribute is used by default in virtually all of Word's Paragraph Styles including Normal.

You can modify the style to exclude the attribute or you can remove it directly in much the same way as any other formatting attribute... Format> Paragraph> Line & Page Breaks, clear the check on Widow/Orphan Control.

Just keep in mind that by doing so you will most likely encounter pages which leave a single line [1st line of a paragraph] at the bottom as well as some that start with a single line [last line of a paragraph] at the top.

That may give you the result you want, but other formatting influences 'the bottom line' as well. For more suggestion on the subject see this article by MVP Suzanne Barnhill:

“Improving the bottom line”—tips for achieving more uniform page length

ossiemac say:

In my test example I had the named range on the same worksheet as the combo box. I am now assuming that you have the named range on another worksheet. Is this correct? If so, and the named range is on a different worksheet then the worksheet must be included with the range name like the following. (Replace "Sheet2" with the worksheet name where the named range resides.)

With Worksheets("Sheet2").Range("FullName")

Alternatively you can use the CodeName of the worksheet like the following. This is actually my preferred method of referencing a worksheet.

With Sheet2.Range("FullName")

The advantage of using the CodeName is that it does not change if the User changes the worksheet name. You can find the CodeName in the Project Explorer which is the left column in the VBA editor. The CodeName is NOT in brackets and the name given by the user is in brackets. Initially they default to the same name but try changing a worksheet name and then view the Project Explorer and you will see that the CodeName does not change.

It is possible to change the CodeName. Simply open the VBA editor and select the name in the Project Explorer and press F4 to display properties and you can change the name there. If you protect the project then the User is not able to change the CodeName. If you google Protect Excel VBA code you should find info to password protect the project.
