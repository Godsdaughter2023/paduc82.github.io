Hi All and Microsoft

Here's the solution I found for my setup. Windows Defender is not malfunctioning and its not an update issue.

It's a multi-core usage problem, which was alluded to when I examined the event logs after a WD error. In prior versions of MSE and WD it is my understanding that CPU usage could be controlled via an application setting. Apparently Microsoft has removed this feature.

WD is a very cpu intensive app running as high as 70% during a full scan. I have a 4 core i7 extreme processor running at 4.45ghz so at 70% cpu usage one or more of my cores are spiking above critical temp and thus shutting down.

I have cooling for my cpu so when its turned on I can run indefinitely at 100% cpu usage at 4.45ghz. And when I turn it on during WD full scans I am now able to run full scans to completion over and over again without any errors.

For those who do not have cpu cooling the solution is as follows:

1) Open the task manager by right clicking on the task bar.

2) Click on the Details tab

3) Locate MsMpEng.exe process and right click on it.

3) Select Set Affinity and deselect all virtual cores except for Core 0 it should remain checked.

4) Click ok and then you can close the task manager

Then start WD or MSE and update definitions if necessary and then choose to do a full scan.

It will take about 3-4 times longer to complete a full scan now because if you monitor it within the task manager performance tab it will not exceed roughly 18% cpu usage.

I tested it on my computer several times without any cooling and it has yet to give the error I have been getting previously for several weeks now.

Also depending on your setup you may be able to run on more than one virtual core. That is maybe only Cores 0 and 1 can be checked in Set Affinity.

Also be aware that if the MsMpEng.exe process does terminate such as with a Restart the Affinity Settings will need to be changed back to at least 1 virtual core as explained above.

I'm still looking into if it is possible to save Affinity settings for an app if anyone knows how please let us know.

Anyone viewing this post please pass this fix along in general.

And also Microsoft, please at some point restore the old feature "Limit CPU Usage" in WD and MSE for all of us with the latest multi-core processors and also the majority of us who use them in LAPTOPS.

scott mcdaniel say:

Access will automatically upgrade the reference, so the most likely scenario was that the 2013 user simply opened the database in 2013, and Access upgraded that reference.

You should be able to remove that reference, however. Try a Compact/Repair on the database (make a backup first, of course) and try again.

Also, you'll find this to be a bit tedious if you continue to share the database with users running different versions of Access. In situations like this, you're much better off giving users their own copy of the database FrontEnd (the Forms, Reports, etc - everything but the data tables). This way, when you open your 2007 version, your references will remain the same, and when the 2013 user opens their copy, their references will remain the same as well.

scottgem say:
This is one of the reasons (there are several others) that we strongly recommend that any multi-user app be split between back end (the tables ) and front end (everything else). The back end will be stored on a shared network drive and the front end on each users local machine. This completely eliminates this type of problem (among others).
