ken sheridan say: What are the RowSource, BoundColumn, ColumnCount and ColumnWidths properties of the control?  If the RowSource property is the name of a query, what is the query's SQL statement?

Note that personal names are unsuitable as keys, as they can legitimately be duplicated.  People should always be identified by a distinct numeric key.  The following is an example of how a combo box to select an employee by name might be set up:

ControlSource:   EmployeeID (for a bound control; leave blank if the combo box is an unbound 'navigational' control)

RowSource:     SELECT EmployeeID, FirstName & " " & LastName FROM Employees ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

Alternatively, you can concatenate the names so that the last name is first, which would be better with a large list of names as it allows the user to enter the initial characters of the last name and progressively go to the first match as each character is entered:

RowSource:     SELECT EmployeeID, LastName & ", " & FirstName FROM Employees ORDER BY LastName, FirstName;

However, selecting a person purely by name is not really satisfactory, as personal names can be duplicated (I worked with two Maggie Taylors).  To differentiate between them, further details can be shown in another column or columns of the combo box's list, e.g. job title, department etc.

Well, this looks like a good spreadsheet - but a good spreadsheet is rarely a good table!

It would appear that you have two (probably many more) Rates - MED rates, MAK rates; that suggests that you have a many to many relationship: each File has many Rates, and many Nets; each kind of Rate can apply to many Files.

Also, the MedMakTotal field sets off some mental alarm bells: if it is indeed a total of the values of other fields, then it should simply not exist in your table! You will almost never store a field which can be calculated from other fields. Doing so wastes disk (trivial these days); wastes time (an addition is MUCH faster than a disk fetch); but most importantly risks storing invalid data. If a field MedMAKTotal is a sum of a Med field and a Mak field, then if either one of those fields were to be edited, you would have a total which is wrong, with no easy way to tell that it is wrong or why! Just store the base data and calculate the total in a Query on the fly.

3. Click on GO options beside Manage, uncheck all the add-ins and click Ok.
