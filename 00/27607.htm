I'll guess that your paragraph's style includes "Justified" alignment, or that it's applied as direct formatting. Press Shift+F1 to open the Reveal Formatting pane, and check the box at the bottom labeled "Distinguish style source". Look for "Justified" and determine whether it's in the "From Paragraph Style" section or the "Direct Formatting" section.

davem121 say:
Hi Sze Hung,

Open that document and press Alt + F11

that will open the VBA Editor

In the pane on the left, double click 'ThisDocument'

Copy and Paste the code below into that blank page

Then click somewhere in the middle of that code and at the top, click the Play (run) button

That Code will remove all unused styles from that document

Delete all that code and close the VBA Editor

That's It!

Sub DeleteUnusedStyles()
Dim aStyle As Style

For Each aStyle In ActiveDocument.Styles
If aStyle.BuiltIn = False Then
With ActiveDocument.Content.Find
.ClearFormatting
.Style = aStyle.NameLocal
.Execute FindText:="", Format:=True
If .Found = False Then aStyle.Delete
End With
End If
Next aStyle
End Sub

ken sheridan say: The DefaultValue property is not appropriate to what you are attempting.  It assigns a default value to a column or control in a new row before data is inserted into the row.  What you are attempting is to assign a value when data is inserted.  This should be done in a data entry form, not in the table definition.  All data should be entered via a form of course, never directly into a table in datasheet view.

The first question to be asked, however, is whether the value you are computing is immutably determined by the values in the other columns, or whether the value can then be legitimately amended by the user?  You should only assign the value to control bound to a column in the table in the latter case; in the former a computed control in the form or a computed column in its underlying query should be used as otherwise redundancy would be introduced into the table, with the consequent risk of update anomalies.  As you have used the term 'default', however, I'm assuming the value can legitimately be amended.

You can assign a value to the relevant bound control by means of code in the AfterUpdate event procedure of each of the four other controls as follows:

    Me.[EEE] = (Me.[AAA]/Me.[BBB])+Me.[CCC]-Me.[DDD]

where EEE is the name of the bound control to which the computed value is being assigned.  The value will only be assigned when values have been entered into all of the four controls as Null propagates in arithmetical expressions, so the result of the expression will be Null until all four controls have a value.

scottgem say:

OK, now we can understand the issues. In that case, what I would do is provide a calc button. You will need some code, but its very simple:

Me.Controlname = expression

Controlname will be the name of the control on your form that will hold the calculated value. It will have a Controlsource of the field in your table. Expression is the formula you want to use to calculate the value ( ([AAA]/[BBB])+[CCC]-[DDD]). This single line of code needs to be entered in the On Click event of the button using the Code Builder. 

When you click the button, the control will be populated with the results of the calc. You can override it if need be. If you change any of the components, just click the button again. 

ken sheridan say: Whether you assign the value in code, or return it in an unbound control depends on the answer to the question I originally posed:

Which is it?

As regards the error you are encountering, did you by any chance try entering the code directly in the control's properties sheet?  If you are unfamiliar with entering code into a form's or control's event procedures, this is how it's done in form design view:

1.  Select the AAA control  and open its properties sheet if it's not already open.

2.  Select the After Update event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new lines between these.

5.  Repeat for the BBB, CCC and DDD controls.

To return the computed value in an unbound control on the other hand, if the answer to the above question is that the value cannot then be legitimately amended by the user, you omit the Me keyword, which is only used in VBA, so the ControlSource property would be:

=([AAA]/[BBB])+[CCC]-[DDD]

ken sheridan say: I'm pleased to hear you have it working, but you should understand the difference in modelling terms between what Scott and I said.  They are not alternative solutions, but based on different premises.  Scott's solution will always return a value based on the values of the other four columns, i.e. the value is time independently determined by the four values.  Mine on the other hand assigns a default value to a column in the table (not to be confused with the DefaultValue property), i.e. the value is not time independently determined by the four values, and can be edited.  Can we assume the former is what you intended?

An example of the former would be in a production database where the area of a sheet component is computed by multiplying the length and width of the sheet.  The area can't be anything else but the multiple of the two dimensions, i.e. it is time independently determined by them, so is computed not stored in a column.

An example of the latter would be in a sales database where the unit price of an item is assigned to a column in an OrderDetails table on the basis of the current unit price of the product in a Products table.  The unit price of a product in the Products table will change over time, but the price in the OrderDetails table should remain static as that at the time of the order, i.e. the value is not time independently determined by the current unit price, so must be assigned to a column, not simply referenced from it.  The price in the OrderDetails table can if necessary be edited, e.g. to reflect a discretionary discount.
