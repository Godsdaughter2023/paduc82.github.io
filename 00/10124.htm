mdb file explode - Microsoft Community

wwang72 say:

mdb file explode

Hi,

I am using the access 2016. I have around 200 tables in a .mdb file and those tables are all identical in terms of fields. I developed a program to write to those 200 tables at 10 records/second (2000 records/second in total) using .net and the growing rate of file size is normal. However, if I open the .mdb file while the program is inserting data into it, its size will rapidly grow, for example, 4 times faster than the normal growth rate. Could anyone help me on this issue?

albert d. kallal say:

A most interesting effect.

 

 The reason for this is surprising but will make sense when I explain. 

 

Access has the ability to open a database with what is called “row” locking. This “feature” does not necessary mean you are going to use record locking, but if you open the database in Access, then this feature is “turned on”. And note that the FIRST person in is the way this setting takes effect.

 

And more important this feature is ONLY accessible and can be set from Access (you cannot set this from ANY external program .net or anything else). So the DAO object model does NOT expose this feature – ONLY the access client.

 

Access over its history does not actually have true row/reordlocking. So what the feature in question does is flip a switch that causes access to EXPAND all records you work with to the size of one database page.

Access does have “page” locking, so by expanding all records you edit to the size of one database page, then the result in a round-about way is true row locking, but at a rather HUGE cost in bloat and file expansion rate. Normally “several” records can fit in a database “page” that Access resolves to internally. So this "page" based system is internal to how the database engine for Access works. Access can and does lock these "pages" for its internal operations.

 

So if the first person into the file does NOT set this expansion feature, then all other users will not see or have this feature either. And the reverse is true – and that is your case.

 

So when you open with Access, then you have the row locking or “expand” each record feature turned on.

 

You can set or turn off this feature in Access. You don’t mention what version, but say in Access 2010, go:

 

File->options->client settings.

 

The option to un-check is this one:

 

Note that you MUST exit and then re-enter to see this setting change. Now if you open + enter the database with Access, then additional programs or users that ALSO open the database will of course adopt and respect the row lock setting (which is now we assumed to be turned off)

 

Regards,

30 days late formula - Microsoft Community

working__101 say:

30 days late formula

What is the fomula I use to find out when items are 30 days late and another formula for 60 days late?
joeu2004 say:
What is the fomula I use to find out when items are 30 days late and another formula for 60 days late?

Note that those two conditions are not mutually exclusive.  Anything that is 60 days late is also 30 days late.

"Gary's Student" provided the paradigm if you want two separate formulas, and you do not care that both might be true at the same time.

But if you want the 30-day formula to be true only if it is not 60 days late, you might write:

=IF(AND(TODAY()-A1>=30,TODAY()-A1<60),"30 days late","")

=IF(TODAY()-A1>=60,"60 days late","")

Alterrnatively, if you want a single formula to determine either one, you might write:

=IF(TODAY()-A1>=30,IF(TODAY()-A1>=60,"60 days late","30 days late"),"")

where A1 contains the date that an item was expected.
