I'd think that, previously, you were able to connect the phone to any old USB port on your system, as you should be. The above step of connecting the phone to a totally different USB port is to check that there are no conflicts being caused by the sloppy way in which Windows sometimes tends to deal with USB Device Management. Windows detects each device on each USB port. This means that, should you connect a device to port A at first, it will be detected. If you connect the very same device to port B the next time, then it will be detected AGAIN, as opposed to the way *Nix deals with these situations: the device's connection parameters will be re-used, instead of the device driver being installed twice.
I'd suggest that, if you seem unable to use Nokia PC Suite at all (i.e. your phone is not recognised), you try to use a different port. jllatham say:

No, those references are controlled internally to Excel.  And as you delete a lower numbered shape, the ones with higher index numbers shift to fill the empty space.  That's why the code I showed earlier:

Sub DeleteAllShapes()
  Dim AnyShape As Shape
   Do While ActiveSheet.Shapes.Count > 0
    ActiveSheet.Shapes(1).Delete
  Loop
 End Sub

will work for any number of shapes on a sheet - as you delete .Shapes(1), then (3) moves to (2) and (2) moves to 1, etc, until in the end the very last one, originally with the highest index number, becomes .Shapes(1) and is deleted and .Shapes.Count becomes zero.

This type of thing almost becomes one of those "you have to know the answer before you can ask the question type of things".  When you create a shape, you can assign it a name that you could use to reference it later - but then you have to 'remember' the names, either by having them as part of the code somewhere, or keeping a separate list on a hidden worksheet (I've never done it that way, but can envision doing so).

If you know something unique about the line(s) in question, you can loop through all shapes and test that unique feature - perhaps the .Left and .Top values or start and end points, or height/width - or it's .Name and once you find it, then you deal with it.

This will show you another problem with it all:  Open a new workbook, and put a shape on a sheet, put a couple more.  Notice that Excel may (or may not) assign them numbers in sequential order beginning with 1.  But if you then delete them all and go through the same process without closing Excel, the next "first" shape you add to the sheet is NOT #1 again - in its name, although it is .Shapes(1)
