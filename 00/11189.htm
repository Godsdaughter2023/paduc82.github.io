Hi Diann,

This is expected behavior. If you are using shared mailbox in Office 365 Exchange Online. When one person read an email from shared mailbox, it will marked as read on all machines for everyone this is also applicable to any other action such delete/move emails.

Regarding "It does not show up in deleted files either", if you use Outlook desktop to delete items from a shared mailbox, the deleted items go to your Deleted Items folder rather than to the Deleted Items folder of the shared mailbox. You can use Registry setting to switch the destination of deleted items. See Items that are deleted from a shared mailbox go to the wrong folder in Outlook

Note: Serious problems might occur if you modify the registry incorrectly. Therefore, make sure that you follow these steps carefully. For added protection, back up the registry before you modify it.

However, Litigation hold meaning is different than what you described. If you are using Exchange Online plan 2 Litigation hold/ In-Place Hold is enabled for a mailbox it will preserve all mailbox content in Exchange Online.

You can recover those items by an Exchange Online admin (who is a member of the Discovery Management Role Group) perform a discovery search to recover those items. For more information: In-Place eDiscovery

Note: If you want to put a Litigation Hold on a shared mailbox, then an Exchange Online Plan 2 license or an Exchange Online Plan 1 with Exchange Online Archiving license is required.

ken sheridan say: A design I have used on a number of occasions where rows in a referencing table are differently categorised is to use separate subforms for each category (programme in your case), placing each subform on a separate page of a tab control.  I once did this for instance in a database of company registrations in one Canadian province, where each subform in a parent companies form recorded the directors, officers and shareholders of the company respectively.  The subforms where each based on a query on the same table restricting the query in each case to the relevant category of registered persons.

The advantage of this is that the controls on each subform and their layout can be specific to the category in question, so in your case, each subform can be for one programme (or a group of programmes if preferred) and designed in the context of that programme or programmes.  The user would simply select the appropriate tab when entering or editing reservation data.  For inserting a new row into each subform the DefaultValue property of the relevant foreign key, e.g. ProgramID, would be set the relevant value for the programme in question or, in the case of a subform covering a group of programmes, a combo box bound to the foreign key column would have its RowSource restricted to those programmes in the group.

You might wish to consider whether this sort of solution would or would not be appropriate in your case.  If not, then using conditional formatting, as described, by Duane, would be the obvious solution, though I would also enable/disable and lock/unlock the control in the subform's Current event procedure and the AfterUpdate event procedure of the ProgramID or similar control.  Otherwise, even though the control's contents would be hidden by the conditional formatting a user could still inadvertently edit it without being aware that they are doing so.

al edlund say:

Let's see if we can talk to some of these.

"I have played with the Excel data linking " Many users like excel for dataentry because of the visual feel of columns and rows. A significant (IMHO) problem with that is a worksheet doesn't have columns in the database sense because datatypes are stored at the field level and excel doesn't internally require the user to be consistent, thus you can end up with data integrity issues that the relational databases can protect you from. ODBC raises an issue with excel in that the data import is limited to the

"What database is the best solution in terms of use with Visio?" Visio doesn't care which of the platforms you select. The import mechanism is ODBC which in many cases is what gives us the headaches. For an enterprise solution I prefer the real SQL database. For those on a budget SQL Express is good. I sandbox most of my 'real' database solutions with Access just because it's focused more on solution development. If you sandbox be sure to set your Access options to create SQL queries for ease of sharing. SQL is better for multiple users since Access cannot 'record lock' the last time I checked on this feature. This is only important if your solution intends to roundtrip information or the database is dynamically being updated by other users.

"How do we handle the shapes, as they will come from a single location on the server " This is more of a document management question. If not already using it, you might consider SharePoint to store your stencils. SharePoint can also be used as a datasource for your Visio drawings and should be considered in the discussion of database operations.

"We need to ensure that if the data source changes, that Visio drawings do not self-update, but rather requests an update of its data fields" Good news/Bad news. Visio only sets the fields from a datalinking operation at the first time, after that changing fields can cause issues because if  a critical field moves it may not be remapped correctly.

Your bullet number seven is a solution data design issue and not a visio question. Providing audit trail of database changes can be as simple as time/date stamping and the userid of who made the changes but it goes to building and writing the appropriate record. This of course can also (and should) get into a discussion of span and scope (who is allowed to change a record, and how much can they change).
