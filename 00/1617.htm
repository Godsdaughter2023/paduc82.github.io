jay freedman say:

I can't say it's intentional, but Word is actually being helpful in this case. If you open an Outlook attachment directly into Word, edit it and save, and close Word, you'll probably lose all your work. The reason is that Outlook opens a copy of the attachment in a temporary folder, the edits are saved in that temporary copy, and the copy is deleted when Word closes.

If that happens to you, you might -- maybe, no guarantees -- be able to get the document back as described in http://www.gmayor.com/outlook_attachments.htm.

It's important to save the attachment to a real file on the hard drive, and edit that.

e.j. gun say:

There are two ways to do this.  If you simply want to copy the Values from Sheet1 to Sheet2, you use this line of code:

ActiveWorkbook.Worksheets("Sheet2").Range("A1:A100").Value = ActiveWorkbook.Worksheets("Sheet1").Range("A1:A100").Value

If you want to set it up so that Sheet2 always has the same values as Sheet1, even when Sheet1 changes, use this code, which sets the Formula in each cell of Sheet2 to point to Sheet1:

ActiveWorkbook.Worksheets("Sheet2").Range("A1").Formula = "='Sheet1'!A1"
ActiveWorkbook.Worksheets("Sheet2").Range("A1:A100").FillDown

Hope this helps,

Eric

joeu2004 say: snmuni wrote:
When I have given this ....

Application.ScreenUpdating = False
Application.Calculation = xlCalculationManual
wsTarg1.Range("b2:b12005") = "=large($a$2:$a$12005,row(1:1))"

Then it is becoming very slow. Is there any formula instead  of large formula?

Not surprising:  you are performing 12004^2 = 144+ million operations, searching the entire range A2:A12005 12004 times.

And as I explain in the previous response, you doing that twice:  once in Manual mod, and once when switching to Automatic mode.

In this case, doing the assignment in Automatic mode takes half the time on my computer.  (But I do not have any other formulas.)

In any case, this is poor design, IMHO.

At the very least, ROW(1:1) should be changed to ROW(b1).

The change will not improve the performance of the macro.  But it might significantly improve performance when you make a change somewhere else.  As written, the formula must recalculated because it references an entire row.

But the intent of the formulas is to copy values into column B in descending order.  Why just do that directly?

Try the following code snippet (be very mindful of leading periods within the With construct):

Application.ScreenUpdating = False
Application.Calculation = xlCalculationManual
With Sheets("sheet2")
   .Range("b2:b12005") = .Range("a2:a12005").Value
   .Sort.SortFields.Clear
   .Range("b2:b12005").Sort Key1:=.Range("b2"), Order1:=xlDescending, _
        Header:=xlGuess, OrderCustom:=1, MatchCase:=False, _
        Orientation:=xlTopToBottom, DataOption1:=xlSortNormal
End With
Application.Calculation = xlCalculationAutomatic
Application.ScreenUpdating = True

That takes about 0.1 sec on my computer, compared to nearly 60 seconds for the range of LARGE formulas.

You can execute the code snippet directly within a larger macro.  Or you could use alt+F8 to execute it after modifying A2:A12005 manually.  And you could create a "button" to make that more intuitive.

If you need help with any that, let us know.
