DLNA transcode under windows 7 broken - Microsoft Community

don engineer say:

DLNA transcode under windows 7 broken

Windows 7 DLNA automatically transcodes lossless .wav -> .mp3 when streamed.  Vista didn't do this.  I have a hard wired gigabit network.  My DLNA enabled devices do support .wav!  They worked perfectly under Vista.  I don't want it transcoded.  What device doesn't support .wav?  It's really part of the DLNA standard!

I'm hoping you guys have a workaround as I just spent $500 to set this system up!

ccazang say:

Don,

Thanks a lot for your feedback and I am sorry about the trouble. I had briefly pass through this thread which was pass to me by Barb, and here is the explanation and the possible solution for your trouble. First of all let me see if I got the problem right: MP3 stream that goes to your device is not only white noise. It is the actual song that you try to play but the quality is not the one that satisfies you, right?

The reason for why it worked in Vista was because Vista was limited in transcoding capabilities. Win Vista did no transcode to MP3 and this is why you device could not pick up the MP3 version. Win7 on the other side, wants to enable more devices to consume content from a PC. We want devices not to worry much about codecs. If they support what DLNA mandates (PCM and MPEG2) they will be able to stream any content from a Win7 PC (from WMA to AVCHD). Of course there has to be a side effect: the quality degrades. While for most of the people this is fine, there are people that know exactly what they want and ask for highest quality. This is why I am glad I got your feedback.

Now, you've mentioned to scenarios: PULL (when you pull content from the device) and PUSH (when you PlayTo content to the device). The fundamental difference is that in a PULL scenario, the device has all the control in picking up the right URI for a specific media item. In PlayTo scenario, Win has the first option, but the device can still choose. Here is how it works:

- win7 provides for every supported media item in the library a list of so called RES elements. These elements are XML tags associated with each media item and contains metadata that specify the parameters of each format provided by that specific RES element (e.g.: bitrate, resolution, DLNA Profile Name, mime type, size, duration, etc.). For a M4A file for example we provide alternative RES elements corresponding to WMA, WAV/PCM and MP3 (for each format various RES elements with various parameters)

- a device PULL-ing content, browses the server, finds the item that it needs and checks the RES elements. Each device has its own algorithm for choosing the right RES element, so it is up to them to decide what to play

- when PUSH-ing, PlayTo does the browsing, checks the device capabilities (usually mime types and associated DLNA Profile Names), and decides what RES element can push to the device based on a matching content algorithm. As a rule of thumb, PlayTo prefers MP3 or PCM in this order. (By the time when we did PlayTo there were no PlayTo capable devices, so whatever junk we found to test with, used to support MP3 more than PCM. So this is why we decided this way, but it turned out to be wrong in the last year after we ship and good devices started to show up). However, we check all the RES elements and if there are others matching the device capabilities (like a PCM one) we send those too. The device is free to ignore the MP3 one and choose what is best.

In your case (sorry for the long introduction), the PULL scenario is totally controlled by the device. For some reason your high fidelity audio device prefers MP3 when it finds the MP3 RES element. This seems to be an issue with the device. More than that, the MP3 RES element is coming in the beyond the 4th position in the RES elements list, so it is curious how your device got so deep ignoring all the PCM/WAV RES elements found before.

In the PlayTo scenario, we are in control, and as I said above, we send by default MP3. If the device is not "smart" enough to choose something else, than you get the experience that you get.

In order to workaround the problem, you have to modify some registry key (I would make a back-up or something because I do not know if this will have any side effects). However, this will -remove completely the capability of your WIn7 machine to transcode to MP3. Here is what you need to do:

- Type regedit in an elevated command prompt or in the search box in your Start menu. By doing that you get to the Windows registry

- navigate in that tree to the following path: HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Media Player NSS\3.0\Transcode

- remove the MP3 entry.

- reboot your machine and give it another try to your scenario.

These steps will remove the MP3 transcoding capability from Win7 DMS. Unless you put the keys back, your machine will never transcode to MP3 again. If your device now prefers WMA (cause we transcode WAV to WMA also), do the same thing for the WMA key. I hope this will help you. Let me know if it does not. It worked on my side...

Thanks a lot again for your feedback.
