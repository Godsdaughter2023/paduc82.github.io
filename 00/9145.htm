Patrick, You mentioned the headsets were sold with the 620. Do you mean they were packaged in the box, or just bought at the same time. I fear they may not actually be compatible. The pinouts (which ring does what on the connector) did change between Symbian, WP7 and WP8, over the years. The fact that it works OK on your Symbian phone, and when you push a button, makes me suspicious. I even got an adapter dongle with some Nokia headsets for Lumia, to be used if I were going to use an older Nokia device. It just fixed the wiring and button functionality.

The wiring on headphones, what is left, right, mic and ground, is not standardized across the industry. Worse yet the implementation of the answer/end, pause/play, volume up and down is proprietary by OEM, so you can't go get headsets designed for Apple and have everything work on HTC/Nokia, etc. In some cases even within an OEM (Nokia) different lines and times yielded different implementations.

Vertical Ribbon versus Horizontal Ribbon - Microsoft Community

michaeljcarroll say:

Vertical Ribbon versus Horizontal Ribbon

I am posting as someone who is mostly a computer user.  I am finding more and more -- as we move into the world of Windows 10 and Edge -- that display controls and data are horizontally oriented.  As a PC user, I find this incredibly inefficient and annoying.  But even as a handheld device user, I see no point to it.

First of all, there is the problem of the endless string of text flowing across the screen off the sides of the window so one must scroll from left to right to even read the text.

Second, there is the issue of data orientation.  Consider a list of names followed by phone numbers.  On a traditional vertical scroll screen, one would see something like the following:

Thomas Auyer -- (xxx) xxx-xxxx

Al Bertstein --(xxx) xxx-xxxx

Mick Emmanuel -- (xxx) xxx-xxxx

Adam Firsmon -- (xxx) xxx-xxxx

Frank and Lynn Rosebelt -- (xxx) xxx-xxxx

Ludbert Tovern -- (xxx) xxx-xxxx

Tom Clint Zey -- (xxx) xxx-xxxx

Notice how easy these are to read and scan.

Look at the same data in horizontal orientation:

Thomas Auyer -- (xxx) xxx-xxxx     Al Bertstein --(xxx) xxx-xxxx     Mick Emmanuel -- (xxx) xxx-xxxx     Adam Firsmon -- (xxx) xxx-xxxx     Frank and Lynn Rosebelt -- (xxx) xxx-xxxx     Ludbert Tovern -- (xxx) xxx-xxxx      Tom Clint Zey -- (xxx) xxx-xxxx

Now keep in mind that you are looking at this with word wrap.  Word wrap is disappearing among all the displays in Windows Store.  So imagine that instead of wrapping, the string disappears off to the right of the screen.  Next imagine that instead of seven entries, there are seventy.  Your eye could scan the list and pick out the name you wanted -- even if you had forgotten the last name -- within a few moments ... if the list were vertical.  Horizontally, the list becomes almost useless.

You only have to start scanning apps in the App Store to find out the difficulty of this kind of navigation.

MS may be saying that they aren't coming up with one-size-fits-all for all technology from computers through phones; but you only have to look at everything that's new to see that this is simply not so.  Endless left-to-right movement is not useful to any real work.  How we read is not an accident; it is the result of growth and adaption and recognition of mental needs.

The pattern is symmetrically replicated for cultures that read up-to-down or right-to-left or any other way.  There is a motion in one direction, staying on the visible page, then a perpendicular shift to a parallel line, still staying on the same page.

If someone were to give you a story in three parallel columns, where after getting to the end of a column you had to go to the next page instead of to the next parallel column, and you had to continue this way until you got to the end of the pages, after which you would return to the first page and continue into the second column and so on, you would think they were nuts.

Our display patterns should match our inculcated reading patterns!

violetchen_msft say:

Hi Craig,

Based on my test, you can do the same thing for Public Folders. However, we don't recommend that you use Outlook’s PST export feature to migrate public folders to Office 365 or Exchange Online if your on-premises public folder hierarchy is greater than 30 GB.

Office 365 public folder mailbox growth is managed using an auto-split feature that splits the public folder mailbox when it exceeds size quotas. Auto-split can’t handle the sudden growth of public folder mailboxes when you use PST export to migrate your public folders and you might have to wait for up to two weeks for auto-split to move the data from the primary mailbox.

In addition, please note the following before using Outlook PST to export public folders to Office 365 or Exchange Online:

1) Public folder permissions will be lost during this process. Capture the current permissions before migration and manually add them back once the migration is completed.
2) If you use complex permissions or have many folders to migrate, we recommend that you use the cmdlet method for migration.
3) Any item and folder changes made to the source public folders during the PST export migration will be lost.

If you still want to migrate your public folders by using PST files, follow these steps to ensure a successful migration.

1. Download the migration scripts To download the migration scripts, you only need to download the PublicFolderToMailboxMapGenerator.ps1 file.
2. Generate the .csv files to create the public folder-to-mailbox mapping file. This file is used to calculate the correct number of public folder mailboxes in Exchange Online.
3. Create the public folder mailboxes that you’ll need based on the mapping file. For more information, see Create a public folder mailbox.
4. Use the New-PublicFolder cmdlet to create the top-most public folder in each of the public folder mailboxes by using the Mailbox parameter.
5. Export and import the PST files using Outlook.
6. Set the permissions on the public folders using the EAC. For more information, please see: Assign permissions to the public folder

For detailed information, please see: Migrate Public Folders to Office 365 by using Outlook PST export

For Exchange 2003 users, we strongly recommend that you upgrade your on-premises servers to Exchange 2010 SP3 RU8 or later/Exchange 2007 SP3 RU15 or later, and use batch migration, which is the only supported public folder migration method.

Regards,
Violet

suzanne s. barnhill say:

There are two limitations to the use of AutoCorrect entries:

  1. They are language-specific. To be available, they must have been saved in the .acl file corresponding to the language applied at the insertion point. That is, if you created an English AutoCorrect entry, but the language applied to your text is French, that English entry will not work (this is pretty granular; I believe there are separate .acl files for, say, UK and US English).
  2. Only unformatted AutoCorrect entries are saved to .acl files and available Office-wide. Formatted entries are saved in the Normal template and area available only in Word.
