" that works wonderfully for my needs"

Have you ever worked with a database? A spreadsheet is not a database. And while this may work for you, once you see what a well built database does you will wonder how you ever thought what you have works "wonderfully". The caveat here is that properly setting up a database will not be that simple, but it will pay dividends in time savings and accuracy of data down the road.

For example, to set up a spreadsheet to "automatically add or subtract the inventory" will require some complex calculations to account for the transactions. Entering transactions may require a lot of redundant data entry.

I would suggest you look for and review some inventory templates. For your purposes, you would need at least 2 tables:

tblProducts

ProductID (Primary Key autonumber)

ProductName

UnitPrice

Reorderlevel

tblTransactions

TransactionID (PK Autonumber)

TransDate

ProductID (Foreign Key)

Quantity

UnitPrice

TransType

tblProducts would track all the products you want to track (diapers, wipes, etc.) One record for each product. Add more products as you go along. tblTransactions will track your purchase and usage. The TransType field will let you know whether the transaction was a purchase, usage, etc.

All setup so you just run the report anytime you want.

As I suggested it would pay to look at Inventory templates to see how they are done. But your questions are easily answered.

First let me explain that Access uses object to represent everything. A form is an object a table another object. A field in a record is an object and a control on a form another object. Everything is done in Access by manipulating the properties of Objects. 

The way I would do this is to create a form for entering transactions. In the header of that form I would put 2 unbound (meaning no ControlSource) controls; a textbox and a Combobox. The textbox would have a Default Value property of =Date(). So when you open the form it will show today's date. If you want to enter transactions for a different date, you just change the date. The Combobox would allow you to select a transaction type. Now the question here is how many types do you want to use. You need at least 2 (incoming/Outgoing) but you might want to have more. As for product name, that what comboboxes are for. If you use the Combobox wizard to create a Combobox for ProductID it will, by default create a combo where you select from a list of product names, but it stores the ProductID. You use a similar combo wherever you need to select a ProductID.

As for Brandnames, you can do that in a variety of ways. You would create a table of Brands:

tblBrands

BrandID (PK autonumber)

Brand

From there you either add a BrandID field as a foreign key to tblProducts or as a FK in tblTransactions. If you do it in tblProducts, then you would need a record for each combination of Product and brand. For example:

Diapers Pampers

Diapers Huggies

If you do it that way, then you would need to display that column in the Product combo when you enter transactions. If you have a separate field in tblTransactions, then you would use a synchronized combobox that would only display brands applicable to the Product. that would mean another table to store the combination of ProductID and BrandID. I would be more inclined for you to start off slowly and add a Brand feature later on.

I would leave TransID on the form, but set the Visible property to No so you don't see it.

ken sheridan say: Another way of doing this sort of thing, and one which I find in many cases a better solution than using a crosstab query as a report's RecordSource property, is to base the parent report a query which returns a distinct set of the relevant values for the rows (machine numbers in your case), along with a multicolumn subreport whose column flow is 'across then down'.  This has the advantage of being completely data driven, so there is no need for any code to handle the variable column headings returned by a crosstab query.  I've posted a simple demo of this type of arrangement as Payments.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

You might have to copy and paste the text of the above (not copy the link location) into your browser; for some reason it sometimes doesn't seem to work as a hyperlink.

There are a few things to note about my file:

1.  It includes a Calendar table, which is simply a table of all dates over a period of time, and is easily created by serially filling down a column in Excel and then importing the worksheet into Access as a table.

2.  The calendar table is included with the Customers table in a query to return a complete set of customer/date values regardless of whether the customer has made a payment on a date.  The tables are not joined, so that the query returns the 'Cartesian product' of the two tables, i.e. each row in one is joined to each row in the other.  Your machines are analogous to the customers in my file.

3.  This query is then joined to the CustomerPayments table on the date and customer columns in another query, which is used as the RecordSource of the subreport.  This ensures that a row is returned for each customer/date even if the customer has made no payment on the date.  The payment amounts are Null in such cases so appear blank in the subreport.

4.  The subreport's query references a control in a dialogue form to restrict the result set to one week's payments.  The same control is referenced by unbound text boxes in the parent report's header to provide the column headings for the seven days in the week.

5.  The total payments per customer for the week are shown in an unbound text box which calls the DSum function to sum the values for the current row's customer, using the same query as the subreport's RecordSource property.
