PARAMETERS [Enter start date:] DATETIME,
[Enter end date:] DATETIME;
SELECT RoomNumber
FROM Rooms
WHERE NOT EXISTS
    (SELECT *
     FROM Calendar, RoomOccupations
     WHERE RoomOccupations.RoomNumber = Rooms.RoomNumber
     AND calDate BETWEEN EntryDate AND DepartureDate
     AND DepartureDate >= [Enter start date:]
     AND EntryDate <= [Enter end date:]);

ken sheridan say: If you need to identify what capacity is available at any given date in the future you will need an auxiliary calendar table.  See the calendar demo in my OneDrive folder for how to build one.  You presumably have a figure for the total number of beds in the facility, which, being data, should be stored as a single row in a single column table, not hard-coded in an expression.  This table is the third table to which Scot refers, but rather than having individual rows for  beds (analogous to my Rooms table) stores only the aggregated value of the total number of beds.

You will then need a query which returns the total capacity less the count of reservations on each day within the range in the future.  If you merely want to project ahead n number of days then the query would be restricted with:

WHERE calDate BETWEEN DATE() AND DATE() + n

If you want to define a range at runtime then you will need parameters.  In my example I used simple parameters which prompt for values, but in a developed application the parameters would be references to controls in an unbound dialogue form.

A query to return the available free capacity on each day within the range would be simpler than mine as you do not have to identify intersecting ranges, so once you have  Calendar and TotalCapacity tables, should not be difficult to build by including these in a query with your existing tables.

SELECT calDate, COUNT(ReservationDate) AS Occupancy

ken sheridan say: Yes, if you have two date columns such as AdmissionDate and DischargeDate, where the latter can be NULL, which I'm guessing from what you say is the case, and the join is solely on Admission date it would return the number of admissions on each day.  To return the number of patients in residence on each date you'd need to join the tables on the intersecting ranges like this:

SELECT calDate, COUNT(AdmissionDate )
FROM Calendar LEFT JOIN Reservations
ON Calendar.calDate >= Reservations.AdmissionDate
AND Calendar.calDate <= NZ(Reservations.DischargeDate,#2100-01-01#)
WHERE calDate BETWEEN DATE() And DATE()+30
GROUP BY calDate;

The NZ function is called to return a date unrealistically far in the future if the DischargeDate is NULL, i.e. the patients residence is currently open ended.  If the database is still in use in 2100 then someone would have to amend the query of course!

The query I posted in my last reply would return the occupancy per day if the many-to-many relationship type between patients and dates were modelled in the classic way by having a row in Reservations for each date a patient is anticipated to be in residence, rather than by a range defined by admission and discharge dates in each row.
 

ken sheridan say: Sorry, I forgot to answer your question about where you out the SQL statement.  SQL is the language in which queries are defined.  When you create a query in design view it generates an SQL statement which you can see if you switch to SQL view.  To create a new query in SQL view open the query designer, but don't add any tables.  Then switch to SQL view and paste in the SQL statement I posted, changing the table and column names to your actual names as necessary.  Then save the query and open it.  It will automatically return the dates from the current date to 30 days ahead whenever it is opened.

jay freedman say:
Just like with legacy text form fields, the way to limit the size of content controls is to put each one in a separate cell of a table. Set the AutoFit (on the Table Tools > Layout tab of the ribbon) for the whole table to Fixed Column Width; and in the Table Properties dialog, on the Row tab, set the row height to an "Exactly" value such as 0.2".

This won't prevent users from typing more text into the content control than will fit, but the excess won't be visible.

Even without a table, you can tab from one content control to the next -- except when the cursor enters a Rich Text control, which accepts tab characters.

HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Explorer\Desktop\NameSpace\

8. Close Registry Editor and restart Windows File Explorer.

joeu2004 say: Luv wrote:
do you how to de-annualized a 3 month interest rate?

That depends on how it was annualized in the first place.

I will use "aR" for annual rate and "qR" for the 3-month (quarterly) rate.

1. Simple rate

if:        aR = 4 * qR
then:  qR = aR / 4

2. Compounded rate

if:        aR = (1+qR)^4 - 1
then:  qR = (1 + aR)^(1/4) - 1

3. Square-root of time

if:         aR = qR * SQRT(4)
then:   qR = aR / SQRT(4)

The square-root of time rule applies to the standard deviation of log returns.
