jllatham say:

You can simulate freezing row 6 by getting the sheet set up so that cell A6 is at the very top of the sheet (with rows 1-5 out of sight).  Then select cell A7 and use Freeze Panes. 

Only problem with this is that you can't ever see rows 1-5 without unfreezing the sheet.  You'd need a couple of macros with easy access to both to set it up, unset it and reset it 'on demand'.  Here's some code that would do that for you:

Sub UnfreezeSheet()
  ActiveWindow.FreezePanes = False
  Application.Goto Range("A1"), True ' force rows 1-5 into view

  Application.Goto Range("A6"), True
  Range("A7").Select
  ActiveWindow.FreezePanes = True
End Sub

steve rindsberg say:

I suspect you're running into a long-standing (and as yet uncorrected) bug in PowerPoint.

For pathless links or relative links to work, there needs to be a predictable current directory.  Each app maintains, or can maintain, a current directory.  Faced with a link that isn't fully pathed, PPT appends the link to its current directory and looks for the file there.

For example, if MyPresentation.PPTX includes a link to somefile.txt, PowerPoint will generally (BUT NOT ALWAYS) look for somefile.txt in the same directory ad MyPresentation.PPTX.

Likewise, if MyPresentation.PPTX is in C:\temp\ and contains a link to \TextFiles\somefile.txt, PowerPoint will look for the linked file in C:\temp\TextFiles\somefile.txt

This generally works because when you open a presentation file, PowerPoint sets the current directory to the directory the presentation file resides in.

However, if you open the file from the most recently used list, all that goes out the window.  In that situation, PPT won't change the current directory; it remains whatever Windows last set it to (often but not always the Fonts folder; almost certainly NOT the directory your PPT file's in). 

Try opening the same file by doubleclicking its icon, by using File | Open and by using the most recently used list.  See if there's any consistent pattern to what works and what doesn't.

psychomania---01 say: The limit of SD is 2GB and that cannot be exceeded. It is a limitation of the format.

That is why they created SDHC so that we could have bigger cards.

Older phones like the 6300 are only capable of reading SD, not SDHC.

sidream say:

as a matter of fact - SD cards CAN have capacity up to 4GB using undocumented SD features and such cards are quite popular among owners of older Pocket PCs, but I have never seen such cards in "micro" format size. I've seen only full size SD cards from Transcend and other manufacturers with 4GB capacity and they are not SDHC cards.

I may only guess, that 100% hard boiled DIY enthusiast could possibly make an adapter connecting respective contacts of microSD dongle with externally (not in slot) located full size 4GB SD card, but whether such construction has any practical use is different question. Swapping 2 cards 2GB each seems much easier for everyday use.

It's also possible that Nokia's (contrary to Pocket PCs) may possibly fail to handle such 4GB storage, due to software limitations.

tzsm98 say: From my understanding it has to do with memory addressing. microSDHC cards use an additional bit for memory addressing that was formerly used for something else, I believe identifying the card to the device.  Since the device expecting a microSD card is not looking for memory at that identification bit,  it will never see the memory addressed by it. So the 2 GB limit is a pretty solid limitation in the 6300. the 6301, being newer, knows what to look for and can handle microSDHC. It's a bit better. (you can groan now):smileymad:
