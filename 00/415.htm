ken sheridan say: You cannot use an SQL statement as the ControlSource property of a control.

If the table whose column you wish to bind the control to is a referenced table in a relationship in which the subform's underlying table is the referencing table, and the column in question is not part of the referenced table's primary key, then you can base the subform on a query which joins the tables on the relevant keys, and bind the control to the relevant non-key column from the referenced table.  However, this is an unlikely scenario unless the relationship type is a one-to-one relationship type modelling a type/sub-type.

If you can describe the database's logical model and the role of the two tables in it we would be in a better position to advise further.

About MSDN or TechNet Product Keys

They are genuine Microsoft product keys for retail licenses, but not for Retail sale. They are intended for a particular product channel either the Microsoft Software Developer Network (MSDN) or TechNet for IT Professionals who pay a subscription fee. The main purpose is for evaluation purposes. The great thing about them, unlike trial Microsoft software, MSDN or TechNet keys don't expire. Because the agreement under which the subscription is provided is a single license, none of the software should be distributed outside of it. Microsoft project - Linked Tasks not shifting when one isn't one - Microsoft Community

jakewinstone say:

Microsoft project - Linked Tasks not shifting when one isn't one is moved

I have a programme that I have linked tasks on, I am moving some of the lines that are linked. when I move one of the tasks the rest of the tasks that are linked the rest just stay where there were originally set. how to do set the programme so that linked tasks move when the successor task is moved. 

dale howard [mvp] say:

hakewinstone --

First of all, make sure that all of your tasks are Auto Scheduled.  Secondly, if successor tasks have an Actual Start date they will not move.  If successor tasks have Inflexible constraints, such as Must Finish On or Finish No Later than, they may not move as well.  Hope this helps.

julie sheets say:

When you say you are moving tasks - how are you doing this?  By typing in start or finish dates?  If you have typed in all dates you have constrained the tasks.  If you type in a start date you are setting a Start No Earlier Than constraint.  If you shorten the predecessor's duration and the tasks are linked Finish to Start, the constraint (SNET) will not allow the successor task to move earlier.

Add the Constraint Type field to a task view to view any constraints you may have added.
