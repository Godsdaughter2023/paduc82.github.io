frederik long say: Your problem might be caused by a scheduled task that you created but do not remember. You can find out like so:
  1. Create a System Restore Point. This is important!
  2. Open the registry editor (regedit.exe).
  3. Navigate to this key: HKLM\CurrentControlSet\services\Schedule
  4. The data against the Start value on the right is 2. Make it 4 (=disabled) and press Enter.
  5. Reboot the machine normally.
  6. Open a Command Prompt.
  7. Type this command:
    net start {Enter}
    Can you see a service called "Task Scheduler"?

Does this fix the problem? Regardless of the answer, reverse the action you took above.

frederik long say:

No, you cannot "move" your Windows installation to drive D:. You would have to use a cloning method like so:

  1. Make a clone of partition C: on disk D:.
  2. Disconnect the disk that contains partition C:.
  3. Set your BIOS so that is uses disk D: as its primary boot device.
  4. Boot up with just the former disk D: connected. It will now appear as drive C:.
  5. Reconnect the former drive C: disk.

Windows 10 partitions new drive with GPT table containing an invalid - Microsoft Community

jamesreitz say:

Windows 10 partitions new drive with GPT table containing an invalid protective MBR.

My experience when using Windows 10 to partition a new, bare drive (i.e., a drive with no pre-existing partition table) of 2-terabytes in size or smaller shows that Windows 10 incorrectly denotes the size of the only partition listed in the “protective Master boot record” as 0xffffffff instead of the required value of one less than the total number of sectors on the drive.


The UEFI specification ( http://www.uefi.org/sites/default/files/resources/UEFI%20Spec%202_6%20Errata%20A%20final.pdf#page169  ), section 5.2.3, table 17 denotes how the value of the SizeInLBA field should be “set to the size of the disk minus one” (in sectors), and should only be set to 0xffffffff when the disk is too large for its size to be represented in this field (i.e., drives larger than 2.199 terabytes.)


This error does not present a problem for Windows 10 because Windows 10 uses the GPT partition information, and ignores the protective MBR.  However, the error in the MBR prevents legacy software tools that rely on a valid MBR from working properly.  SpinRite is one example of such a tool.
