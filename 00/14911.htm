Why the MS Access Listbox/Combobox Droplist Row Limit Can Cause - Microsoft Community

mark edwards (mark.edwards) say:

Why the MS Access Listbox/Combobox Droplist Row Limit Can Cause SQL Server Update Queries to Hang

I posted a problem I was having a few days ago concerning why my pass-thru and local Access UPDATE queries were "hanging" and timing out if the table being updated was being displayed in an Access form listbox that was trying to display more rows than the 65,535 row limit.

I finally figured it out.  Using the sp_who stored procedure in SQL Server, I discovered that an Access form listbox whose rowsource was trying to display more that the 65,535 row limit caused the rowsource query to go into a "suspended" state.  As a result, any query trying to make changes will not be able to access the table and timeout.  By simply limiting the rows in the rowsource query to <65,535, the thread does not get suspended, and all update queries are allowed to run as normal.

Just thought I'd share that since I can't seem to get back into the original question and post the solution.

albert d. kallal say:

This may well not fix this issue or problem.

It is a “nasty” problem.

Of course, anything more than a few 1000’s rows kind of starts to break down.

I have seen relative small tables also cause this table lock.

So even tables with far less then 65,000 rows can cause a table lock during the query process. What then occurs is access does ot always pull all records into the combo box – if Access “stops” the flow of records , the query does not finish, and the table lock remains in palce – even for tables with far less the 65,000 row.

The only solution I really have is that you base the combo box on a pass-though query.

And add to the PT query the no lock hint such as:

Select id,FirstName, LastName, Hotel from tblHotels WITH (NOLOCK)

I don’t have a great article that outlines alternative solutions, but the above is a workable fix.
