Is there a difference between Windows Defender full scan and MS - Microsoft Community

family it sup say:

Is there a difference between Windows Defender full scan and MS Windows Malicious Software removal tool full scan in Win10?

Do they scan the same files and use the same definitions, or is one a superset of the other?  Basically is there any reason to run both of them occasionally?
rob koch say:

MSRT is a subset, so anything contained within it should typically also be included in Defender.

However, the answer isn't always that simple, since if Defender weren't updating properly for example, then the copy of MSRT may contain something that is currently missing from Defender, which is the entire reason that the MSRT exists.

I shouldn't say there is a never a reason for downloading the MSRT manually, but there are better tools such as the Microsoft Safety Scanner which contains basically the same full set of definitions as Defender or others from 3rd-party vendors that work in similar ways.  For this reason I imply never bother to download the MSRT manually and instead choose a better tool for the job if I'm gong to spend the time.

As I mentioned above, if you are supporting others using the PC(s) being scanned, determining what the true risks are is your first step.  If these include potential file downloading or browsing by unskilled or younger users, then in most cases the MSRT will be pointless if the PC already has Defender installed.  Something like MalwareBytes Antimalware would likely be far more useful to determine whether PUPs or other less malicious software may have been installed, while this will still provide some overlap for the most malicious software as well for the sanity check you'd typically have used MSRT to do.

MSRT is a valuable automatic sanity check normally performed once a month when it's released, since it's usually not updated again until the next month.  Any other tool that's more complete and updated more often is far more useful when downloading manually to the PC.

Rob

rob koch say:

ThatGirl-007,

I have answered this same basic certificate question in the past in another forum, but don't have that link so here's the "short" version.

These Root Certificates do not directly relate to any specific website and instead are more typically related to the operation of code signing certificates used to sign the code that operates Windows itself as well as other applications.  At least one of the certificates you have listed, the NO LIABILITY ACCEPTED issued by VeriSign, is specifically mentioned in the following Microsoft Support article.

Trusted root certificates that are required by Windows...

https://support.microsoft.com/en-us/kb/293781

Note that these certificates exist for legacy reasons and though they may not actually be required on a specific system, there's no real way to know what the effect of removing them might be.  As for the expired status of some of these certificates, that is specifically mentioned in the article excerpt below and doesn't matter since these certificates typically support Time Stamping, which for Code Signing allows the certificates to remain valid forever unless they are revoked by Microsoft.

"Some certificates that are listed in the previous tables have expired. However, these certificates are necessary for backward compatibility. Even if there is an expired trusted root certificate, anything that was signed by using that certificate before the expiration date requires that the trusted root certificate be validated. As long as expired certificates are not revoked, they can be used to validate anything that was signed before their expiration."

Note also that Microsoft is very diligent about maintaining these Root Certificates and in fact has or is in process of revoking some of these as discussed on the following official page.

http://social.technet.microsoft.com/wiki/contents/articles/31680.microsoft-trusted-root-certificate-program-updates.aspx

The general Microsoft Trusted Root Program Requirements are found here.

http://social.technet.microsoft.com/wiki/contents/articles/31633.microsoft-trusted-root-program-requirements.aspx#C_Revocation_Requirements

Personally, I leave such core security issues up to Microsoft, since if I don't trust them then why would I be using their operating system at all?

Rob
