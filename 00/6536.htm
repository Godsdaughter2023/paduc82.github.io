albert d. kallal say:

I explain why such a WAN (wide area network) it is too slow here:

http://www.kallal.ca/Wan/Wans.html

 

You cannot put the Access accDB file on SharePoint since then this would not allow multi-users of the document (SharePoint would view this database as a single document like word).

However you can upsize the table data from the back end part to what are called SharePoint "lists". You then link to these lists from your Access front end (this works very much like linking the front end part to an access back end file).

However for good performance and support of relational things like cascade deletes on SharePoint, then you quite much need Access 2010. Furthermore, with Access 2010 + SharePoint 2010 you have MUCH better performance with linked tables than that of Access 2007. So you would still install the Access application on each computer with tables linked to the SharePoint site.

You can test/try some of this with 2007, but you not have the relational delete (cascade) delete ability and you have to test performance. Take some of your test data and use the external data option in Access to try sending some data up to SharePoint. After you link, your performance might be ok, but performance is far better in a2010 for this setup (so there is greater possibility that performance will be good enough for your application).

Access 2010 + SharePoint also have web creating ability now (so, no Access is required, just a web browser + the SharePoint site). However, web based might not be required solution here. I share some thoughts on converting Access applications to web based here:

http://www.kallal.ca/Toweb/Index.html
