ken sheridan say:

 I'd suggest writing the queries in SQL, using the EXISTS predicate against a subquery which returns for rows with a different ID primary key value but matches/mismatches in the other columns as appropriate in each case.  To do this open the query designer but don't add any tables, switch to SQL view and paste in the SQL as follows, changing the table and column names to your real ones where necessary, but leaving the aliases PH1 and PH2 as is.  

1.  For a query to return rows with same vendor number, invoice number, invoice date, and invoice amount:

SELECT *
FROM [Payment History] AS  PH1
WHERE EXISTS
    (SELECT *
     FROM [Payment History] AS PH2
     WHERE PH2.[ID] <> PH1.[ID]
     AND PH2.[vendor number] = PH1.[vendor number]
     AND PH2.[invoice number] = PH1.[invoice number]
     AND PH2.[invoice date] = PH1.[invoice date]
     AND PH2.[invoice amount] = PH1.[invoice amount]);

2.  For a query to return rows with same vendor number, invoice number, invoice date, and differing invoice amounts:

SELECT *
FROM [Payment History] AS  PH1
WHERE EXISTS
    (SELECT *
     FROM [Payment History] AS PH2
     WHERE PH2.[ID] <> PH1.[ID]
     AND PH2.[vendor number] = PH1.[vendor number]
     AND PH2.[invoice number] = PH1.[invoice number]
     AND PH2.[invoice date] = PH1.[invoice date]
     AND PH2.[invoice amount] <> PH1.[invoice amount]);

3.  For a query to return rows with same vendor number, invoice number, invoice amount, and differing invoice dates:

SELECT *
FROM [Payment History] AS  PH1
WHERE EXISTS
    (SELECT *
     FROM [Payment History] AS PH2
     WHERE PH2.[ID] <> PH1.[ID]
     AND PH2.[vendor number] = PH1.[vendor number]
     AND PH2.[invoice number] = PH1.[invoice number]
     AND PH2.[invoice date] <> PH1.[invoice date]
     AND PH2.[invoice amount] = PH1.[invoice amount]);

4.  For a query to return rows with same vendor number, invoice date, invoice amount, and differing invoice numbers:

SELECT *
FROM [Payment History] AS  PH1
WHERE EXISTS
    (SELECT *
     FROM [Payment History] AS PH2
     WHERE PH2.[ID] <> PH1.[ID]
     AND PH2.[vendor number] = PH1.[vendor number]
     AND PH2.[invoice number] <> PH1.[invoice number]
     AND PH2.[invoice date] = PH1.[invoice date]
     AND PH2.[invoice amount] = PH1.[invoice amount]);

5.  For a query to return rows with same invoice number, invoice date, invoice amount, and differing vendor numbers:

SELECT *
FROM [Payment History] AS  PH1
WHERE EXISTS
    (SELECT *
     FROM [Payment History] AS PH2
     WHERE PH2.[ID] <> PH1.[ID]
     AND PH2.[vendor number] <> PH1.[vendor number]
     AND PH2.[invoice number] = PH1.[invoice number]
     AND PH2.[invoice date] = PH1.[invoice date]
     AND PH2.[invoice amount] = PH1.[invoice amount]);

The logic behind each query is pretty much self evident from a reading of the SQL.

ken sheridan say: You can return a distinct count of customers per product in a query by means of a subquery:

SELECT Product, COUNT(*) AS CustomerCount
FROM (SELECT DISTINCT CustomerID, Product
            FROM YourTable)  AS DistinctCustomers
GROUP BY Product;

You then you just have to look up the CustomerCount value for the current customer:

=DLookup("CustomerCount", "YourQuery", "Product = """ & [Product] & """")

This assumes that the Product column in your table is of text data type, and that the column is not a so-called look-up field whose real value is a hidden long integer number referencing an autonumber primary key of a Products table.  If the latter is the case omit the literal quotes characters around the value:

=DLookup("CustomerCount", "YourQuery", "Product = " & [Product])

BTW I hope that you do not really have CustomerID and CustomerName columns in your table.  CustomerName would be transitively determined by the key of the table via CustomerID.  The table would consequently not be normalized to Third Normal Form (3NF), whose definition is:

Third Normal Form: A relvar is in 3NF if and only if it is in 2NF and every non-key attribute is nontransitively dependent on the primary key.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, and an attribute to a column (field).

The table would be open to the risk of update anomalies.  The CustomerName column should be therefore be removed from the table.  In the form you can use a combo box bound to the CustomerID column by hiding the bound column, and showing only the second text column from its RowSource.  The combo box wizard can set this up for you, or you can retain your present layout by basing the form on a query which joins the table to the Customers table on CustomerID and return the name in the query.  A text box in the form can then be bound to the CustomerName column from the referenced table.
