hansv mvp say:

IT_Requests_Request_Type=B13 returns an array of TRUE/FALSE values: TRUE for each cell whose value equals that of B13, FALSE for the others.

Similarly, IT_Requests_eNum=D6 returns an array of TRUE/FALSE values: TRUE for each cell whose value equals that of D6, FALSE for the others.

When we multiply them, TRUE is treated as 1 and FALSE as 0. Since 1*1=1, 0*1=0, 1*0=0 and 0*0=0, the resulting array only contains 1 for rows in which both conditions are met.

MATCH(1,...,0) searches for a 1 in the array, and returns the index (position) of the first 1 it encounters.

INDEX(IT_Requests_Req_Num,...) returns the corresponding value in IT_Requests_Req_Num.

ken sheridan say: As always the starting point is the logical model, i.e. the tables, their attributes and the relationships between them.  The list of ingredients is, in relational database terms, a many-to-many relationship type between the entity types recipes and ingredients, so you'd start with two tables to model these, in broad outline as follows:

Recipes
….RecipeID  (PK)
….RecipeName
….RecipeMethod

the last would be a Long Text column, and would be the text description of the 'instructions' for making the dish in question.

Ingredients
….IngredientID  (PK)
….Ingedient

and to model the many-to-many relationship type:

RecipeIngredients
….RecipeID  (FK)
….IngredientID  FK)
….Quantity
….Unit

The Unit column should also be a foreign key referencing the primary key of a Units table with just one column, Unit.  This would have values for all the units used in the recipes, e.g. gms, ml, teaspoonful, etc.

The primary key of RecipeIngredients is a composite one of the two foreign key columns, RecipeID and IngredientID.

For data input you'd use a form, in single form view, based on the Recipes table and, embedded within it, a subform, in continuous forms view, based on the RecipeIngredients table.  The subform would be linked to the parent form on RecipeID, and would contain combo boxes bound to the IngredientID and Unit columns, and a text box bound to the Quantity column.

You'll find an example of such a form/subform in StudentCourses.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates a number of possible interfaces for a basic many-to-many relationship type like this, but you should use the conventional form/subform approach as, unlike the others, this allows non-key attributes of the relationship type, quantity and unit in this case, to be entered.

For output you can use a report which joins the three tables in a query, grouping the report on RecipeID, with the recipe name in the group header, the recipe details in the detail section and the recipe method in the group footer, or for a more flexible layout you could use a report based on the Recipes table and within it a subreport based on a query which joins the RecipeIngredients and Ingredients tables.
