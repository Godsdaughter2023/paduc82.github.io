john w. vinson say:

Well, your proposed change is a bit of an improvement, but neither it nor the current design are properly normalized. 

That said... a Normalizing Union Query may be the ticket here. You need to create this in SQL view, the query grid can't handle it. Create a new Query selecting just FirstName, LastName, Address, and the 1 fields. Then go into SQL view - you should see something like

SELECT FirstName, LastName, Address, Product1, Company1, [Policy#1], [X-Date1] FROM yourtable;

Edit this to:

SELECT FirstName, LastName, Address, Product1 AS Product, Company1 AS Company, [Policy#1] AS Policy, [X-Date1] AS XDate FROM yourtable

UNION ALL

SELECT FirstName, LastName, Address, Product2 AS Product, Company2 AS Company, [Policy#2] AS Policy, [X-Date2] AS XDate FROM yourtable

UNION ALL

SELECT FirstName, LastName, Address, Product3 AS Product, Company3 AS Company, [Policy#3] AS Policy, [X-Date3] AS XDate FROM yourtable

UNION ALL

SELECT FirstName, LastName, Address, Product4 AS Product, Company4 AS Company, [Policy#4] AS Policy, [X-Date4] AS XDate FROM yourtable

UNION ALL

SELECT FirstName, LastName, Address, Product5 AS Product, Company5 AS Company, [Policy#5] AS Policy, [X-Date5] AS XDate FROM yourtable;

using your actual table name of course.

A normalized design will have a table for customers (or whatever entity has the name and address), with NOTHING about Products but with a unique CustomerID Primary Key (such as an Autonumber); a second table just about products, with a unique ProductID; and a third table linking customers to products, using just the ID fields.

scottgem say:

i have what seems to be a common goal, that is to populate a second table from a subset of records from the first table.


Actually this is an UNCOMMON goal. There is only one reason to populate a second table with a subset of another table. And that is for performance improvements if applicable. And even those may be negligible if the primary table changes. 

This is the function of queries. A query is used to produce a subset of records in a table. That query can then be used the same way as any table to populate reports, or join with other data. 

So, I would forget the Append part of your query and just use a SELECT statement.

Hi Barry,

Please be noted that Outlook "Clean Up" is for deleting the redundant messages of an Email conversation. For example, Jeff sends a message to Anne and Tim. Anne replies to both Jeff and Tim and in her Outlook message body, Jeff’s complete original message is included. Now, Tim sees Anne’s message and replies to both Jeff and Anne. Tim’s reply includes all of the previous messages in the Conversation. Conversation Clean Up can delete all of the messages except the last one from Tim because within that message is a copy of all of the previous messages.

So, for your situation, if the 2 emails are the email conversation, the 2 emails should come from different 2 senders. However, the picture shows that they come from the same sender (they look like the same sender sent the email twice). Moreover, if they are the email conversation, Outlook "Clean Up" should move the elder email into the Deleted Items folder (by default) not including the newest email.

    bob jones aka: cybertaz say:

    First consideration is that everything be fully updated: I understand that El Capitan is at 10.11.1 & 14.5.8 is current for Office 2011. Restart your Mac after doing any updates. That may be enough to correct the situation.

    The Word 2011 Normal.dotm template is in the same location as it used to be:

    Macintosh HD:Users:Username:Library:Application Support:Microsoft:Office:User Templates

    The difference -- if you upgraded directly from OS X 10.6.8 -- is that starting with OS X 10.7 Apple saw fit to make the User Account Library folder Hidden. To get to it, go to Finder & hold the Option key while you open the Go menu... You should see Library listed directly below Home.
