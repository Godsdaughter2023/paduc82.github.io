daniel pineault say:

I'm not saying it is typical naming convention, but there was nothing wrong with using underscores.  The important thing was you had, and were using, a convention!  Also, why mess with something that was working in the first place?

Now, taking a complete stab in the dark here, but I am gathering, you would change a field name in the table, then update the control's Control Source to reflect this new name, then go into your code and update the field name there as well.  Is that correct?

I'm thinking while you renamed the Control Source, you omitted to update the actual control Name.  As such, when you rename the control name in VBA, you hadn't actually renamed the control itself and thus it would err.

You need to differentiate, in VBA, working with control values vs. fields.

fieldname.OldValue is wrong

formcontrol.OldValue is correct (see: https://msdn.microsoft.com/en-us/library/office/aa172955(v=office.11).aspx) and thus it is the control name that is critical (as well as the control source...).

As I said, this is merely a stab in the dark though, but I think a good guess.

But I still don't understand messing around with a design that was functional.

john w. vinson say:

If you have a table field named OldValue, that's the problem! Any bound control will have an OldValue property - it contains the value prior to the user's updates. So if you refer to a control .OldValue Access can get confused whether you mean the field named OldValue, or the control Property also named OldValue.

Use your backup and go back to the underscores, I'd suggest! Old_Value is a perfectly good fieldname, and will not cause any compiler errors or changes.

I would also suggest that you turn OFF "Apply Name Autocorrect" in your database - this can be a useful feature if it's used sparingly, intentionally and knowingly, but has earned its nickname "Name Autocorrupt" if you don't realize that Access is using it.

scottgem say:

Another point. You said you deleted the underscores So are your fieldnames now First name or Firstname? That wasn't clear. As Daniel said, nothing wrong with underscores, but spaces ARE a problem. 

Another issue: "store the difference in AccumHours field". As a general rule we do not store calculated values. if you need to display a running balance of accumulated hours you can do that in with a Running Sum query or report. 

runningmanhd say:

Another point. You said you deleted the underscores So are your fieldnames now First name or Firstname? That wasn't clear. As Daniel said, nothing wrong with underscores, but spaces ARE a problem. 

Another issue: "store the difference in AccumHours field". As a general rule we do not store calculated values. if you need to display a running balance of accumulated hours you can do that in with a Running Sum query or report. 

Following up on what Scott is telling you, it sounds like you are storing a single record for each person and trying to add to a cumulative field AccumHours without any history of actual hours entered.  If this is the case, then this is a really bad design and should be changed. 

You should have a table of Staff Members and a table of Staff Hours.  The problem with your current design is that there is no way to tell if the accumulated hours are accurate, and there is no accountability for how you arrived at the current sum.  There are a number of reasons why the value could be off and you have no means to reconcile the figure. 

With a table of Staff Hours, you would have accountability for each person's hours and the ability to validate their entries and totals.  When you need to get a sum of hours, as Scott stated, you just use a query to calculate the sum.  This can be done in total, or for a particular period, etc..

Thank you for your information. If you didn't send emails to those recipients, then the issue could be related with NDR backscatter. Spammers on the Internet sends spoofed emails which pretend as sending from your email address to non-existence recipients, so that the recipients' mail server will bounce back a non-delivery email to your email address. In this case, first please check if the SPF record of your domain is configured correctly. For your reference, see the article below.

https://docs.microsoft.com/en-us/office365/securitycompliance/set-up-spf-in-office-365-to-help-prevent-spoofing

Besides, you can try to enable NDR backscatter in Admin Center to check if it helps stop receiving NDRs.

1. Use an admin account to sign in Exchange admin center.

2. Navigate to protection > spam filter, then double click Default policy to edit.

3. Select advanced options and turn NDR backscatter on.
