daniel pineault say:

I think it is entirely due to some extra spaces in the strLink URL.

The following works for me

Private Sub CmdViewMap_Click()
    On Error GoTo Error_Handler
    Dim strPostalCode         As String
    Dim strURL                As String

    If IsNull(Me.PostalCode) Then 'Ensure we have a postal code to get a map from
        MsgBox "This record does not have a postal code", vbCritical
        GoTo Error_Handler_Exit
    End If

    strPostalCode = Replace(Me.PostalCode, " ", "") 'Remove any spaces from the postal code
    strURL = "http://maps.google.com/maps?hl=en&q=" & strPostalCode 'Build the URL
    Application.FollowHyperlink strURL 'Navigate to the URL

Error_Handler_Exit:
    On Error Resume Next
    Exit Sub

Error_Handler:

    MsgBox "The following error has occured" & vbCrLf & vbCrLf & _
           "Error Number: " & Err.Number & vbCrLf & _
           "Error Source: CmdViewMap_Click" & vbCrLf & _
           "Error Description: " & Err.Description & _
           Switch(Erl = 0, "", Erl <> 0, vbCrLf & "Line No: " & Erl) _
           , vbOKOnly + vbCritical, "An Error has Occured!"
    Resume Error_Handler_Exit
End Sub

joeu2004 say:

Hercules wrote:

Most of them utilise the maximum allowance of 15 digits of integer and decimals. In the following example, if a make a simple formula, viz. = A1= B32, this will return TRUE. However, the formula = MATCH(A1, B1:B42) returns #N/A  ??

That might be your problem:  the "15 digit" maximum is a myth.

A1=B32 returns TRUE because for comparison operators like "=", Excel rounds the left and right operands to 15 significant digits.

But MATCH(A1,B32,0) returns #N/A because, in fact, there is an infinitesimal difference in the internal binary representation, which MATCH compares.  A1-B32=0 should also return FALSE.

Consider this example.  Enter 1234.56789012345 into A1, and enter =A1+2E-12 into B32.  Both appear to be the same when formatted with for the reason explained above.  But A1-B32=0 returns FALSE because, in fact, A1-B32 is not exactly zero.  Likewise, =MATCH(A1,B32,0) returns #N/A.

That said, I am surprised that MATCH(A1,B1:B42) [sic] returns #N/A, if you expect A1 to match B32, because that syntax is shorthand for MATCH(A1,B1:B42,1), which does not require an exact match.  (Except that A1-B1>=0 must be true.)  I can understand that MATCH might return an unexpected index, 31 or 33 instead of 32.  But it should not return #N/A.

Does your posted syntax contain a typo, and your actual formula has a third parameter, which is zero?

If the posted syntax is correct, be sure that B1:B42 is sorted in ascending order.

Generally, if we explicitly round (using ROUND), the value will be the same binary representation as how the value appears (when displayed with 15 significant digits).  But if we only round the

Note that I say "the same binary representation".  Most numbers with decimal fractions cannot be represented exactly in binary, even numbers with fewer than 15 significant digits.

For example, 0.123 is represented internally as exactly 0.122999999999999,998223643160599749535322****. And since that is the best binary representation, rounding to 3 decimal places does not change it.

[EDIT] Digits replaced by "****" (separated by semicolons to thwart auto-edit): 1;8;9;3;3;1;0;5;4;6;8;7;5 .]

(I use period for the decimal point and comma to demarcate the first 15 significant digits, which is all that Excel will format, rounded.)

Also, the binary representation of the same decimal fraction depends on the magnitude of the integer part.

For example, 10.123 is represented internally as exactly 10.1229999999999,99332****31884765625.  Consequently, 10.123-10 is 0.122999999999999,332****31884765625, and =10.123-10=0.123 returns FALSE (!).

[EDIT] Digits replaced by "****" (separated by semicolons to thwart auto-edit):  0;8;9;8;2;8;3;8;5;5;0;5;8;2;5;2;8;1;1;4 .]

As I noted in my previous posting, the "=" operator is tricky because it rounds the left and right operands to 15 significant digits.  Consequently, 10.456-10=0.456 does return TRUE as expected.  But MATCH(10.456-10,{0.456},0) returns #N/A and 10.456-10-0.456=0 returns FALSE (!) because there is indeed a binary difference.

Yeah, without the "****" (bogus automated censorship).

In Excel, numbers are represented internally using 64-bit (double-precision) binary floating-point, per the IEEE 754 standard.  They are the sum of 53 consecutive powers of 2 ("bits") times an expontential factor that determines the power of 2 of the left-most bit.

I evaluate the internal representation exactly.  For 10.123, the least-significant power of 2 is 2^-49 (1/562949953421312), which is 0.0000000000000017763568394002504646778106689453125 .

You might look for a website that does the same.  I'm sure that one exists; but I have not been able to find it at the moment.  Beware of the many websites that convert single-precision binary floating-point, based on the sum of only 24 consecutive powers of 2.  That truncation can lead to a significantly different exact decimal value.  For example, single-precision 10.123 is exactly 10.12300014495849609375 .

Oh, I see - you meant the first and second row instead of the first and second field. A field is the same as a column, not a row...

A simple workaround is to open the text file in Notepad, delete the first line, then save and close it.

Or you could write VBA code that will do this for all text files in a folder.

Let's say your text files are in the folder C:\OldFiles.

Create another folder C:\NewFiles.

You can run the following code from Access (or from any other application that has VBA):

Sub RemoveFirstLine()
    ' Modify as needed but keep the trailing backslash
    Const strOld = "C:\OldFiles\"
    Const strNew = "C:\NewFiles\"
    Dim strFile As String
    Dim strLine As String
    ' Get first filename
    strFile = Dir(strOld & "*.csv")
    ' Loop through the files
    Do While strFile <> ""
        ' Open existing file in input folder
        Open strOld & strFile For Input As #1
        ' Create new file in output folder
        Open strNew & strFile For Output As #2
        ' Read and discard first line
        Line Input #1, strLine
        ' Read and write the rest
        Do While Not EOF(1)
            Line Input #1, strLine
            Print #2, strLine
        Loop
        ' Close the files
        Close #2
        Close #1
        ' On to the next
        strFile = Dir
    Loop
End Sub

After running this code, the folder C:\NewFiles will contain copies of the text files with the first line removed.
