jim thomlinson say:

Here is a function that you can call in your beforesave macro that will select all but a single specified sheet.

 

Public Sub SelectSheets(ByVal wksExcluded As Worksheet)
    Dim wks As Worksheet
   
    For Each wks In ThisWorkbook.Worksheets
        If wks.CodeName <> wksExcluded.CodeName And wks.Visible = xlSheetVisible Then
            wks.Select
            Exit For
        End If
    Next wks
   
    For Each wks In ThisWorkbook.Worksheets
        If wks.CodeName <> wksExcluded.CodeName And wks.Visible = xlSheetVisible Then
            wks.Select False
        End If
    Next wks
End Sub

jllatham say:

The two Private Sub Workbook_...() events must reside in the ThisWorkbook object's code module, not in a regular code module.

Also, for the various Workbook_...() events like _BeforeSave() and _Open() there can only be one of them.  Any code that you want to execute during that specific event must all be included in the one process.

 

So your Private Sub Workbook_BeforeSave() and Private Sub Workbook_SheetChange code must both be in the ThisWorkbook object's code module and you must combine the code from the two _BeforeSave() event code segments into a single one - just decide whether you want that to take place before or after the

Call SelectSheets(Worksheets("Audit Trail"))

in the one short one.

jllatham say:

Gene,

The problem with this code being in the _BeforeClose() event is the apparent dependence on Target to determine what cells to lock and which not to lock.

That is all referenced down in the
For Each ws in ThisWorkbook.Worksheets
loop - where you have references to Target (and PreviousValue)

 

This kind of thing needs to be taking place on the individual sheets in "real time" - that is, through either the individual Worksheet_Change() events for the sheets, or through the Workbook_SheetChange() event processors.

Target and PreviousValue are both going to just hold the most current information about a single cell, not a history of what has happened to those cells during a session with Excel.  So waiting until you are closing the workbook is too late.

If you do need to wait until the workbook is closed to lock down the cells that were changed, then we need to work up a process that keeps a history of what cells on which sheets were changed during the session and need to be locked before saving.  I don't see why we couldn't do that with the contents of the [Audit Trail] sheet since it's recording changed cells in real time!

All of this points to a need for a "requirements definition" - something that says "When this takes place on these sheets, then do this".

Running Macros on a protected sheet - Microsoft Community

judi_d say:

Running Macros on a protected sheet

I have some macros that I want to run on a workbook that has one protected sheet. Is that not possible?

Thanks!

Judi

ossiemac say:

Hello Judi,

Yes. Include code to Unprotect the sheet before changing anything on the worksheet and re-protect after the change is made.

It is a good idea to create a constant at the top of a standard module with the password and then you only need to change the password in one place and the constant is used wherever the password is required.

It is also a good idea to write a sub that is called from the code to Unprotect and Protect the worksheet  as per the following example. The Sub Protection(wSht As Worksheet, ProtAction As Boolean) is called from within the code and the worksheet and a Boolean value is passed to it. (Boolean value is True for Protect and False for unprotect.)

Const strPassword As String = "ossie"   'Place this line at the top of a standard module.

Sub ProcessingCode()    'Example of calling the sub for unprotecting a worksheet and re-protecting
    Dim ws As Worksheet
   
    Set ws = Worksheets("Sheet1")
   
    Call Protection(ws, False)   'This will unprotect the worksheet
   
    'Insert code here in lieu of the MsgBox that needs to write to sheet
    MsgBox "The worksheet is unprotected"
   
    'After the processing code is finished use the following line
    Call Protection(ws, True) 'This will protect the worksheet
    MsgBox "The worksheet is now protected"

End Sub

Sub Protection(wSht As Worksheet, ProtAction As Boolean)
    'This sub is called to Protect and Unprotect worksheets
   
    If ProtAction = True Then
        wSht.Protect Password:=strPassword  'Protect the worksheet
    Else
        wSht.Unprotect Password:=strPassword    'Unprotect the worksheet
    End If
End Sub

ossiemac say:

Another option is to use VBA code like the following to protect the worksheet and it allows macros to make changes but the user cannot make changes directly on the worksheet. It is simply an example of protecting.

While it works quite well with most code, there are some cases that it does not allow the macros to make changes so I generally avoid it.

Sub ProtectAndAllowMacros()
   
    Dim ws As Worksheet
   
    Set ws = Worksheets("Sheet1")
   
    ws.Protect Password:="ossie", UserInterfaceOnly:=True    'Allows macros to change but user cannot change on worksheet
   
End Sub
