When you choose Developer>Restrict Editing, a panel opens on the RHS. Initially, that panel has a greyed-out button with 'Yes, Start Enforcing Protection'. Aside from highlighting (probably a bug), I am unable to replicate a situation in which someone change make such changes when the Style restrictions are applied. With such restrictions, one cannot:

• import new Styles (eg via Copy/Paste or via the Organizer);

• modify an 'allowed' Style;

• create new Styles that are not replicas of the 'allowed' Styles;

• apply character attributes (eg highlighting, bold) that are not themselves applied via 'allowed' Styles.*

Indents, bold, italic, underline, etc are all lost if they're not applied via an 'approved' Style.

 

Depending on what Style restriction options you checked, the user may be able to change themes and/or quick style sets, but that is a different issue and is just as easily managed.

 

None of this means a determined user can't mess up your document. They could use a Style that is meant for one thing for something it wasn't intended for. A really determined user might 'repurpose' a Style with, say, manual tabs & line breaks to give the appearance of an indented, non-justified paragraph with double-line spacing to a Style that defines a non-indented, justified single-spaced aparagraph; they could also apply the 'emphasis' Style to the text to make it bold (assuming you've allowed that Style to remain), and so on. And no amount of Style comparison between the template & document will identify such machinations.

 

* Using 'track changes' may result in different font attributes, pending the acceptance/rejection of the changes, but those are not permanent changes and, in any event, can be overridden by switching to 'Final' or 'Orginal' view.

mark mcw say:

karl timmermans [outlook mvp 2012-2018] say:

The fact that you are able to import any rows means that the column names is not at issue.

Not sure what you mean by "Each time I subsequently got the same error message, I cut and pasted it at the end of the database................"

The only way to determine why you can't import anymore is to identify the cause of the problem which is likely going to be related to one of the two following items

#1 - You have defined a "rule" for a given field which is failing based on the value being imported

or 

#2 - There is a problem with a given field what is occurring in multiple rows

The quickest way to eliminate item #1 as the potential cause is to simply try and add a new contact which has no values entered for any field. If there is a rule that is failing - you will get an Outlook message stating that the contact cannot be added but unfortunately, it will not identify which field is the cause of the problem

If item#1 is not the cause, you need to isolate a contact that fails to import, create an input data source with only that one contact (either as a CSV file or an Excel worksheet with a named range that only includes the header row and that one contact row. After that, it's a process of elimination on a field by field basis to determine which field is triggering the problem

Another alternative to try and speed up isolating the cause is to download/install ContactGenie QuickPort (no purchase is required for this exercise) and import the one record that is causing the problem. If CQP reports an "import failure" - then the problem is related to item #1 - otherwise the data error log should contain information regarding an invalid field value.

One thing as absolutely certain, given the description - the issue is data related.
