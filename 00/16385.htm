Tables in Access, unlike Excel, have no *sort order*, they are more like buckets of water, doesn't matter the order.  That said they do have a tendency when importing to sort on the first row of data but there is no guarantee.  If you want a specific order then you must use Autonumber, however, I have had that even mess up, rare but it has happened.  The only true guarantee is to assign a column to your Excel spreadsheet.

scottgem say:

Can you give more info about what these keywords are used for?

First, having numbered fields in a table like keyword1, keyword2, etc. is not considered good design. It is referred to as a repeating group which violates Normalization rules. MVFs also violate normalizations rules. So while using an MVF has some, very limited usages, using a child table is usually the better choice. 

So if you explain the purpose here we may be able to suggest a solution.

scottgem say:

Well the first key here is that an MVF is not necessary to giving them unlimited keywords. What should have been done in the first place is having 2 tables:

tluKewyords

KeywordID (PK Autonumber)

Keyword

and 

tjxEventKeywords

EventKeywordsID (PK Autonumber)

EventID (FK)

KeywordID (FK)

Then add a keywords subform to your Events form. The subform would have be in continuous form mode. It would have a hidden textbox for EventID and a combobox with the following relevant properties:

RowSource: SELECT KeywordID, Keyword

FROM tluKeywords

WHERE KeywordID NOT IN (SELECT KeywordID FROM tjxEventKeywords WHERE EventID = Forms!mainformname!eventID

ORDER BY Keyword;

Bound Column: 1

Column Count: 2

Column Widths: 0";2"

So, when the user adds a record in the subform the combobox will only list keywords not already assigned. 

If the user insists on using an MVF, then you still need the tluKeywords table. But you have to set the Rowsource on the table level to:

SELECT KeywordID, Keyword 

FROM tluKeywords 

ORDER BY Keyword;

One of the major drawbacks of using an MVF in this situation is that you don't have a NotInList event. So the user can't, easily, add a keyword that isn't already in the table. 

If he still insists on using an MVF after you show him the subform route, you will need to go through some additional hoops if he wants a report by keyword. I'm not even sure how that would be done.
