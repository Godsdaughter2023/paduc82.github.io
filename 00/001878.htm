hansv mvp say:

Excel comes with Visual Basic for Applications as programming language - press Alt+F11 to start the Visual Basic Editor.

But it's easy to create a multiplication table in Excel without programming:

 

Enter 1 in cell A2 and 2 in cell A3.

Select A2:A3.

Now drag the fill handle in the lower right corner of the selection down, for example to cell A21. When you release the mouse button, A2:A21 will be filled with the numbers 1, 2, 3, ..., 20.

Optional: format the selected cells, for example by making the text bold and by applying a fill color.

 

Enter 1 in cell B1 and 2 in cell C1.

Select B1:C1.

Drag the fill handle in the lower right corner of the selection to the right, for example to cell U1. You will now have 1, 2, ..., 20 in B1:U1.

Optional: format the selected cells.

 

Select B2:U21.

Enter the formula =$A2*B$1 (make sure the $ characters are places exactly like that) and confirm it with Ctrl+Enter.

Rather than focus on the cell's color when it changes (an event is not triggered when a cell's color changes,) let's focus on what triggers the cell's color to be set to red. What are the conditional formatting rules that have to be met in order for the cell's color to be set to red? My guess is that they are tied to certain values being set in certain cells. If so, then we can look at the Worksheet_Change event. In that event handler, we can replicate the rules that determine the color of the cell in question. For example, if the cell in question is set to red when cell A1 is greater than 10, then we can monitor for a change to the worksheet with the Worksheet_Change even handler and, when a change is detected in cell A1 and the new value is greater than 10, we have our trigger for whatever else there is that we need to do.

Alternatively, if the conditional formatting rules are complex, we can use that same event handler to detect a change to the worksheet and, as Gord suggested, look directly at the cell's current color as set by it's conditional formatting rules.

ken sheridan say:

How do you join 2 instances of the relevant tables?



As you have only the one table, which I'll call Transactions for this example, a query which joins two instances of this would be:

PARAMETERS [Enter brand:] TEXT(50);
SELECT DISTINCT T1.Customer_ID, T1.Brand AS Purchased,
T2.Brand AS AlsoPurchased
FROM Transactions AS T1 INNER JOIN Transactions AS T2
ON T1.Customer_ID = T2.Customer_ID
WHERE T1.Brand = [Enter brand:] AND T2.Brand <> [Enter brand:]
ORDER BY T1.Customer_ID, T1.Brand;

I've assumed that Brand is a text data type, but if it's a number data type change the parameter declaration in the first line of the query to:

PARAMETERS [Enter brand:] LONG;

LONG here is a longer integer data type, which is the default number data type when you create a table in Access.

Another point to note is that it is important that the Customer_ID column be indexed (duplicates allowed) in the table design to give good performance.

The way the query works is that it joins each customer's row in one instance of the table to every other row of the same customer in a second instance of the table, and returns the customer ID and brand from the first instance.  As the query is restricted on the brand column from this table to whatever brand you enter at the parameter prompt at runtime, each customer's rows will have the same brand in that column.  The brand column returned from the second instance of the table is restricted to not being whatever brand you enter at the parameter prompt, so consequently returns all other brands purchased by the customer.

I've added an ORDER BY clause to the query to sort it by customer, then brand, but if you use it as the RecordSource of a report, omit the ORDER BY clause and use the report's internal sorting and grouping mechanism to order the rows.
