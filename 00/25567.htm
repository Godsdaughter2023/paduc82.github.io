john korchok say:

Preference files and AutoCorrect entries are 2 different things.

The prefs are located at:

Users/YourUserName/Library/Preferences/com.microsoft.Word.plist

Users/YourUserName/Library/Preferences/com.microsoft.office.plist

AutoCorrect entries are at:

Users/YourUserName/Library/Preferences/Microsoft/Office 2008Microsoft Office ACL [YourLanguage]

By default, the Library folder is hidden in recent OS X versions, hold down the Alt key while clicking on the Go menu and choosing Library to show it. I recommend that you move prefs files to your desktop rather than delete them. Then if moving them has no beneficial effect, your can replace the fresh prefs files with your old versions.

Bob's Mar 17 reply is one of the best ways. Just spend time "playing" with the application, including spelunking through the more commands list.

Another good place is reference materials and articles for old, MUCH OLDER, versions of the application, ie versions released BEFORE the mouse became common so keyboard shortcuts were more important. Quite often long time / "old' features get "buried" by newer features and newer gooies (GUI's) like the mouse and now touch.

I have right at hand (actually my left hand, but never mind) "official" MS Winword 1.0 User Guide (1989) and Word 5.0 for DOS User Guide(1989) and couple of additional 3rd party books for later versions. I still use them as reference materials, even for current versions.  The reason I/you can do it is that change in computer programs tends to be "evolutionary" rather than "revolutionary".  So even a "large" apparent change like the switch from the menu to "ribbon" gooey (GUI) is actually quite small. They simply "put new lipstick on the pig", just changing the user interface. But all of the underlying commands remained the same, including most of the old keyboard shortcuts. So that keyboard "fiends" can still rely on their existing "muscle memory", even though the letters used in the keyboard path may no longer relate to the Ribbon UI.  The point is that new features normally are built on top of existing program code rather than replacing old features.
