If you installed the two files you mentioned manually and MSE indicates it is Up to Date, then you should only download a relatively smaller file of up to a few MB in size.  If you update at least once every 24 hours, the size will typically be under 1MB, since only a smaller daily update file is required to keep current.

However, if you use the PC less frequently and the time has been over 24 hours, then the file will typically range between about 1 to several MB, since it must 'catch-up' to the current point in the month since the last Full Engine and Definitions update.

 

The 64MB file you mentioned above is the Full update package I just mentioned and are usually only released once a month around this time.  In fact the Antimalware Engine Notification Blog that tracks these indicates that a new one will release this Wednesday Sept. 19th.

http://blogs.technet.com/b/enginenotifications/

 

So unfortunately you just happened to download and install only a few days before the normal monthly update cycle.  They don't always come each month, but only one month out of the year has typically been skipped in the past, so I'd usually expect one around the early part of the third week.

You may wish to monitor that blog to know when to prepare for a larger download, since MSE will attempt to download this automatically via Windows Updates unless you have these totally disabled.  Then you can either enable them manually or instead go and manually download this complete file again at a time you prefer.

As far as I know, you cannot directly reference the column of a listbox or combobox in a query.

One solution is to add a textbox on your form and sets its source to
=[ComboWhatever].Column(1)
Now you can reference the control in your query.

Another solution is to write a VBA function where you pass in a reference to the control and a column number.  Here is one you could try by calling

fGetComboColumn("MyFormName","MyControlName",1)

Public Function fGetComboColumn(strFormName As String _
                  , strComboName As String _
                  , iColumn As Integer)
' Return the value of a combo or listbox column
' Useful? if you need to do this in a query.

   On Error GoTo ProcError

   fGetComboColumn = Forms(strFormName).Controls(strComboName).Column(iColumn)

EXIT_Proc:
   On Error GoTo 0
   Exit Function

ProcError:
   fGetComboColumn = Null
   'Enable/Disable the following depending on what error behavior you want.
   'MsgBox "Error " & Err.Number & IIf(Erl <> 0, ": Line " & Erl & vbCrLf, vbCrLf) & _
         Err.Description, _
         , "modTestBed : fGetComboColumn"

   Resume EXIT_Proc

End Function

keratl say:

Try this; I've included some notes to help you adapt it to your needs.

Sub EMail_Reports()

    'For the email side
    Dim OutApp As Object
    Dim OutMail As Object

    Set OutApp = CreateObject("Outlook.Application")

            Set OutMail = OutApp.CreateItem(0)
           
            With OutMail
                .Subject = "This is my blank email's subject line"
           
                .Body = "This is an example of" & _
                        Chr(13) & _
                        "building a multi-line email where I " & _
                        Chr(13) & _
                        "want to control where the line breaks are." & _
                        Chr(13) & _
                        Chr(13) & _
                        "Thank you," & _
                        Chr(13) & _
                        "My name goes here"
               
            'If you knew the address you were sending it to in advance, you could add it here
            '    .To = CurrUserID
               
            'optional: set read receipt
            '  .ReadReceiptRequested = True
           
            'optional: set delivery receipt
            '  .OriginatorDeliveryReportRequested = True
               
            End With
   
        'If you wanted to just send the email without editing or reviewing:
        'OutMail.Send
        'If you just want it to remain open as a draft so you can edit or save it:
        OutMail.Display
       
        Set OutMail = Nothing
    
    Set OutApp = Nothing

End Sub

Note: This worked for me in Office2010, I don't have/use 2013, so caveat emptor.

Note: I recall from a previous project that MS introduced a new limit in o2010 on the number of draft emails you can create and have open at the same time, I think I ended up limiting ours to 20 at a time, at which time the user could review and send (or save/close) them, then create the next 20.

The code above just creates a "blank" email, per your original post. If you are trying to load data from the form into the email, you'd want to loop the text fields you created and uniquely named to pull the data and push it into this email. If you are trying to attach a copy of the Word form, that would be different (and probably easier, although I haven't done it- I'd look for the VBA equivalent of 'save and send').

a. user say:

That behaviour reproduces here, and it certainly looks like an error in Word to me.

Unfortunately these are peer-to-peer groups and posting in here is no more likely to result in a fix than what you or others have probably already done. You can always try contacting MS via Word's File->Help->Contact Us option, but those channels tend to be one-way only.

I wonder what method you have used to get around the problem so far, other than perhaps not use this Hebrew numbering scheme.

Although using a field-based approach to numbering has a lot of disadvantages compared to using the built-in numbering, it's the only way I can see that has a chance of working.

I hadn't looked at this format before, and can see that it is complicated by the need to avoid some sequences. Is there a better description with examples than the wikipedia example?

If it would be useful to have a field-based numbering approach, the best way to do it is probably to do the following:
 a. use VBA to create one document variable for every number you envisage using. Yes, you can create 100000 document variables if necessary - it takes a while, and you may find that updating a large number of fields is slow, perhaps slow enough to cause Word to hang. And the data will take space in each document. 

Let's say those variables are named N1, N2, N3, N4,...,N10,N11, etc.

Then to number each item in your list, you insert a nested field like this:
{ DOCPROPERTY "N{ SEQ N }" }

Where each { } is a pair of the special field braces that you can insert using ctrl-F9
 
To insert a following full stop and tab you can probably use something like
{ QUOTE "{ DOCPROPERTY "N{ SEQ N }" }.<tab>" }
where you use an actual tab character instead of <tab>

You can make it easier to insert each list number by creating an Autotext containing the necessary fields.

Or you could consider inserting the ".<tab>" in the document variables.

You might also need to modify that to deal with right-to-left sequence correctly.

For page numbering, you could probably use a page field like this:

{ PAGE \#"'{ DOCPROPERTY "N{ SEQ N }" }'"

However, one problem with doing your own page number formatting is that it probably won't be correctly reflected in page number references of various kinds, including page numbers in Table of Content.

  4. Double-click the notebook folder you want to open.

  5. Select the notebook section backup file you want to open, and then choose Open.
