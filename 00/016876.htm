max meng say:

Hi Lisa,

My suggestion is to use the import function in the Access 2003 to accomplish your goal.

Follow the steps:
1. Open the destination database. Make sure the database is not read-only, and that you have permissions to make changes to the database.
2. On the File menu, point to Get External Data, and then click Import.
3. In the Import dialog box, in the Files of type box, select Microsoft Excel.
4. Click the arrow to the right of the Look in box, select the drive and folder where the worksheet file is located, and then double-click its icon.
5. Follow the directions in the Import Spreadsheet Wizard dialog boxes.
You can select the file in the Wizard, and you also can select which sheet in the file you want to import.
When the import operation is complete, you will see a message that informs you of the status of the operation. If there were errors, Access creates an error log table in the database and displays the name of the table in the message.
6. Review the imported data and the error log table and take corrective action

For more detail steps, please refer to the link, and there are also other ways to bring Excel data to Access:

Hello SAMMIBLINKCHICK

This is not an answer that you want to hear but based on your description, there is no foolproof way to accomplish what you want to do for one very important reason, which that you need at least one rule (or set of rules) to follow as to how things "should be merged", Without that, how would an automated process determine what data to choose.

For instance, you mentioned that in one case you could have two identical email addresses but the fields in the second field are blank and populated in the first record. Take another two records and reverse which of the records have populated and un-populated. Which is the most accurate in each case - the first or second record?

The other key question is "what actually constitutes a "duplicate" contact? Is it jsut the email address or email address/firstname/lastname or account# (if it exists) or email address/company name/last name etc etc (you get the idea).

One the issue of defining a duplicate is resolved, the next issue is "which record" contains the most accurate data. One thing dealing with fields that are blank but what about when all records contain values?

I could go on but I hope you get what I'm trying to convey. If you can't define the rules you want something to follow, there's no way a "bulk process" can be successfully completed. Comes down to how accurate you want the final result to be. Once you have some rules defined and depending on what those rules are, would submit that using Microsoft Access would be a far superior choice to get to where you want to be.
