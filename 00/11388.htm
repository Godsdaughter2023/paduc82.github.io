trevor rabey say:

Short answer is, yes, you are missing something. You need to get the definitions straightened out.

When you say "consistent", I am assuming you mean "equal", ie the same.

Well, they are not, because slack is one thing and variance is a different thing.

And it is obvious that they are not the same thing ever from any simple example.

Make a single 1 day task that starts today and finishes today.

The early and late dates are the same, the slack is 0.

Baseline it.

Change the duration to 2 days.

The early and late dates are the same, the slack is 0.

The difference between the baseline finish and the current early finish is 1 day, finish variance is 1 day.

First, I am assuming that all tasks are scheduled from the start date, all tasks are by default scheduled to start and finish as soon as possible. Scheduling backwards from the finish date wherein all tasks are by default scheduled as late as possible is not considered.

This means that the finish is always the early finish. You can see the early and late dates in the schedule table.

The baseline finish is whatever the early finish was when the task was baselined, and it will not change unless the task is re-baselined.

Second, let's exclude actuals, so everything is in the future and hasn't happened yet.

The baseline finish is not the date that the task "should have" finished.

In fact, there is no such thing as the date the task "should have" finished.

There is only when it was scheduled to have finished when the task was baselined and when it is is scheduled to finish now.

But the current early finish can be the same as, or earlier than, or later than the baseline for any number of reasons.

And if there is no baseline, there will still be a finish variance and there will still be slack.

Hope this helps. Please don't forget to mark as answer if it does.

rupertsland say:

I think I might have solved the problem with the distortions affecting the Hero web part. I tested this out this morning.

Use 4:3 aspect ratio for images:

Any image filling a aspect ratio of 4:3. If the image spans 2 tiles, then the aspect ratio would have to be 8:3.

Microsoft actually mentions this in their section Use the Hero web part (see image scaling). So, the instructions were available, but you had to find this through online Support.

 

As an example, a photo having the 4:3 aspect ratio would be one that is 4 x 3 inches in size (try to use 96 dpi resolution; avoid 72 dpi, because the images look grainy/blurry). You can use a size larger than 4 x 3 inches, but you must maintain the 4:3 ratio exactly. A photo having the 8:3 ratio (i.e., to span 2 hero tiles), would have a size of 8 x 3 inches.

Use the .png file type:

 

Also, saving the image in .png file format reduces the file size and makes the image download much more quickly than a .jpg file.

Of course, it would be easier and faster if one could simply point to an image, and then click publish. But with the Hero web part, it doesn't quite work that way. The correct aspect ratio must be applied beforehand to the custom images you upload to SharePoint for use with the Hero web part. Using the .png file type also improves performance.
