tom van stiphout say:

"I think"???

Please take 0.3 seconds to be sure, before posting.

Lookup fields are evil. They seem oh-so-friendly and easy to use, but put ANY requirements on them and they fall over.

Rather use the standard Relational Model. This also allows you to upsize the database later to SQL Server or other, which don't support Lookup fields but DO support the standard RM.

Each judge can serve one or more counties

Each county is served by one or more judges

This sets up a classic many-to-many relationship, which is resolved in an RDMBS by creating a "junction" table with the PK over of both FK fields:
tblJudgesServingCounties

JudgeID PK

CountyID PK

(maybe a StartDate and EndDate)

(of course you want to enforce referential integrity in the Relationships form)

With regards to your other questions: I am ignoring them until the db design is solid.

ken sheridan say: I strongly endorse what Tom has said about not using 'lookup fields', and multi-valued fields in particular.  Modelling a many-to-many relationship type in the classical way by means of a third table which resolves the relationship type into two one-to-many relationship types will make things far easier for you in the long run.

As regards combo boxes for selecting a county and a judge, what you are referring to are correlated (or 'cascading') combo boxes.  This is achieved by making the RowSource of the second combo box, cboJudge, an SQL statement which references the first combo box, cboCounty, as a parameter, e.g. in your case, with an amended model as Tom describes:

SELECT Judges.JudgeID, LastName & ", " & FirstName AS FullName
FROM  Judges INNER JOIN JudgesServingCounties
ON Judges.JudgeID = JudgesServingCounties.JudgeID
WHERE JudgesServingCounties.CountyID = Form!cboCounty
ORDER BY LastName,FirstName;

In the AfterUpdate event of the cboCounty control the cboJudge control would be requeried with:

    Me.cboJudge.Requery

A few supplementary points:

1.  Note that people, including judges, must always be identified by a distinct numeric primary key column, JudgeID in this case.  Personal names can legitimately be duplicated, so are unsuitable as keys.

2.  Note the use of the Form property in the SQL statement rather than referencing the current form by name as a member of the Forms collection.  This is possible because both controls are in the same form.

3.  I've omitted the 'tbl' tags from the table names.  I dislike them intensely in table or column names as I think they merely interfere with the semantics when writing complex SQL statements, but it's a matter of personal choice if you prefer to use them.  I do use tags for other object names like forms, controls etc.

One way would be to store the date field (but DON'T use the reserved word Date as a fieldname!), but include calculated fields in the query: e.g.

 

SELECT [Date], Year([Date]) AS TheYear, Month([Date]) AS TheMonth, Day([Date]) AS TheDay FROM...

 

You could then use a form with combo boxes for the three numbers (cboYear with values from 2000 through 2100 or whatever years you want), cboMonth with values 1 to 12, cboDay 1 to 31); you could then use

 

=Forms![YourFormName]![cboYear] OR Forms![YourFormName]![cboYear] IS NULL

 

as a criterion on TheYear, and similarly for month and day.

john p spencer say:

A bit more information about the structure of your tables would help.

If I were doing this I would have an auxiliary table that has one field with a date field (primary key) that has all the dates in the range.  Then you can use this Calendar table to force records for every date in the range.  The basic query would look like the following.

The following query can only be designed and worked with in SQL view - query design view cannot handle the non-equi join.

SELECT Calendar.TheDate, Courses.ID, Courses.DelegateCount
FROM Calendar INNER JOIN Courses
ON Calendar.TheDate >= Courses.StartDate
AND Calendate.TheDate<=Courses.EndDate

You could also build a less efficient cartesian query.  The advantage to this being that you can use the query design view instead of SQL view to build and modify the query.  The disadvantage being a big performance hit with large amounts of data

SELECT Calendar.TheDate, Courses.ID, Courses.DelegateCount
FROM Calendar , Courses
WHERE Calendar.TheDate >= Courses.StartDate
AND Calendate.TheDate<=Courses.EndDate
