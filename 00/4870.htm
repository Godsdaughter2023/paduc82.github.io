Multi-SQL statement parser
--------------------------
Option Explicit
Option Compare Database


Function IsPathRel(p As String) As Boolean
    IsPathRel = Not ((Left(p, 3) Like "[A-Z]:\") Or (Left(p, 1) = "\"))
End Function


'Multiple SQL statements separated by semicolons
Sub runMultiSql_file( _
   sqlFname As String, _
   Optional ifLogImmediate As Boolean = False, _
   Optional logFname As String = "NONE" _
)
   'Example invocation from Immediate window
   '  runMultiSql_file _
   '     sqlFname:="SQLfilepath.sql", _
   '     ifLogImmediate:=False, _
   '     logFname:="LOGfilepath.log"


   Dim cdb As Database
   Dim sqlFnum As Integer
   Dim logFnum As Integer
   Dim DataLine As String
   Dim sqlStatement As String
   Dim ifLogFile As Boolean
   Dim ifComment As Boolean


   Set cdb = CurrentDb


   ifLogFile = ( logFname<>"NONE" )
   If IsPathRel(sqlFname) Then _
                       sqlFname = CurrentProject.Path & "\" & sqlFname


   sqlFnum = FreeFile()
   Open sqlFname For Input As #sqlFnum


   If ifLogFile Then
      If IsPathRel(logFname) Then _
                       logFname = CurrentProject.Path & "\" & logFname
      logFnum = FreeFile()
      Open logFname For Output As logFnum
   End If


   If ifLogImmediate Then Debug.Print Now vbCrLf "Starting " sqlFname vbCrLf
   If ifLogFile Then Print #logFnum, Now vbCrLf "Starting " sqlFname vbCrLf


   sqlStatement = ""


   Do While Not EOF(sqlFnum)


      Line Input #sqlFnum, DataLine
      If ifLogImmediate Then Debug.Print "Read: " & DataLine
      If ifLogFile Then Print #logFnum, "Read: " & DataLine


      DataLine = Trim(DataLine)
      ifComment = ( Left(DataLine,2) = "--" )


      If ifComment Then
         DataLine="" 'Unnecessary, but just in case
      Else
         sqlStatement = sqlStatement & " " & DataLine
      End If


      If EOF(sqlFnum) Or ( Right(DataLine,1) = ";" And Not ifComment ) Then


         sqlStatement = Trim( sqlStatement )


         If sqlStatement<>"" Then


            If ifLogImmediate Then Debug.Print "SQL: " & sqlStatement
            If ifLogFile Then Print #logFnum, "SQL: " & sqlStatement


            If UCase(sqlStatement) = "EXIT" Or _
                  UCase(sqlStatement) = "EXIT;" _
               Then Exit Do


            If UCase(Left(sqlStatement, 11)) = "DROP TABLE " _
               Then On Error GoTo tbDropErrHand
            cdb.Execute sqlStatement, dbFailOnError
            On Error GoTo 0


            If Not UCase(Left(sqlStatement, 11)) = "DROP TABLE " Then
               If ifLogImmediate Then Debug.Print "RecordsAffected: ", cdb.RecordsAffected
               If ifLogFile Then Print #logFnum, "RecordsAffected: ", cdb.RecordsAffected
            End If


            If ifLogImmediate Then Debug.Print
            If ifLogFile Then Print #logFnum, ""


            sqlStatement = ""


         End If 'sqlStatement<>""


      End If 'EOF(sqlFnum) Or ( Right(DataLine,1) = ";" And Not ifComment )


   Loop


   If True Or ifLogImmediate Then Debug.Print _
      "Residual SQL statement: """ sqlStatement """" _
      vbCrLf Now vbCrLf "Done " sqlFname
   If ifLogFile Then Print #logFnum, _
      "Residual SQL statement: """ sqlStatement """" _
      vbCrLf Now vbCrLf "Done " sqlFname


   'Clean exit
   Set cdb=Nothing
   Close
   Exit Sub


   tbDropErrHand:
      If Err.Number = 3376 Then Resume Next
         '3376 is the error number when dropping non-existent table
      On Error GoTo 0
      Resume


End Sub

I did run into some noteworthy discrepancies between SQL-92 capability and Access 2010's specific synatx for them.  Speficially, Access 2010's Help for SQL Reference says to add a field using

ALTER TABLE ADD COLUMN myFieldName myFieldDataType DEFAULT myDefaultValue

The default clause is optional, but that was the specific capability that I was after.

I hope this saves someone a bit of time.

AFTERNOTE:  DOH !!!!  Even with the keyword COLUMN removed, the statement is still not accepted by CurrentProject.Connection.Execute.  In fact, after solving the real problem, the word COLUMN could be happily added back without issues.  The real problem seems to be the DEFAULT clause; when present, the statement is rejected regardless of what kind of data type is specified.

john w. vinson say:

Both FindFirst and Seek simply do what they say - find a record in a set of data. Once upon a time the SEEK method was (reputedly) faster and more efficient than FindFirst, but in my experience it's six of one, half a dozen of the other; FindFirst will make good use of indexes on your tables, and won't usually do anything inefficient. In a sense this is "legacy code"; it made sense once and it still works, but in my opinion it is not necessary. If you do a careful and detailed comparison of the performance of the two methods, using your hardware and your data and optimized code in both versions, and demonstrate that the SEEK method is (say) 3.54 seconds faster, and your users really, really need that 3.54 seconds, then use it;

One advantage of findfirst is that the code can be a lot simpler and can more easily filter the recordset rather than just finding a single record:

Dim strSQL As String

strSQL = "SELECT this, that, theother FROM tbl_Employees " _

  & "WHERE [datefield] = #" & dtCurDate & "#"

Forms!SomeForm!Recordsource = strSQL

Your example is presumably "made up" code - as written it would find the employee whose ID is vbThursday (employee 5), not a particularly useful operation!

marshall barton say:

Using Index also sorts the records so the Seek can use a highly optimized search algorithm.  As John said, with a moderate number of records, the speed difference is probably not noticeable, but with a large number of records and lots of searches, Seek can be orders of magnitude faster.

Regardless of whether you use Seek or FindFirst, your point of not wanting to reopen the recordset (and back end?) is important to the performance and you should do everything possible to only open it once.  One way is to create a Function to open the recordset if is not already open and skip that if it is open.

Here's the general idea if you end up using Seek (it's simpler if you use FindFirst):

Public Function OpenBeRecordset(Optional CloseDb As Boolean =False) As Database

Static db As Database

Static rs As Recordset

   If CloseDB = True Then

      rs.Close: Set rs = Nothing

      db.Close: Set Db = Nothing

      Exit Function

   Else

      If db Is Nothing Then

         Set db = DBEngine.Workspaces(0).OpenDatabase("PathToBE\BEFileName.accdb")

      End If

      If rs Is Nothing

         Set rs = db.OpenRecordset("tbl_Employees", dbOpenTable)

         rs.Index = "PrimaryKey"

      End If

      Set OpenBeRecordset = rs

   End If

End Function

Then your code can look like:

Set rs = OpenBeRecordset()    'won't open anything that is already open
rs.Seek "=", EmpID, Weekday(dtCurDate)

When you know you are done doing whatever it is, call the function this way to clean up:

         OpenBeRecordset True

stefan blom say:

The capitalization options on the AutoCorrect tab of the AutoCorrect dialog box only apply when you type, as you have noticed. Capitalization does not change if you move around existing text. This function isn't part of grammar check or spellcheck (although you access the AutoCorrect dialog box via File | Options | Proofing).
