tryx3 say:

Report of an investigation into the use of Admin accounts for day-to-day computer activities on Windows 7 computers

This document is also available in pdf format.  [17 May 2015 - I have amended the pdf file by improving the logic of some of its arguments rather than by introducing any new information.]

1                   Aim

My aim is to answer this question - Is it always correct to advise computer owners to log in for day-to-day computer activities as Standard users rather than as Admin users as a system security measure?

I am keen to find out what ought to be the advice on this subject as 75% of non-network Windows computers run with a single user who is logged in as an Admin user.  Whilst those owners would normally be expected to create Standard user accounts for all other individuals using the computer, this would not necessarily determine their choice of user account for their own use.

I am not questioning the utility of having two additional Admin accounts used purely for providing Admin permission.  The existence of & limited usage of these accounts enhances system reliability by establishing a high degree of confidence that at least one fully-functioning Admin account will exist even if some accounts are afflicted with user profile corruption.

2                   Review of published Microsoft advice

2.1            Microsoft advice

I found six sources of Microsoft advice on the issue of running day-to-day operations in Windows 7 whilst logged in as an Admin.  They are not all consistent with each other & some have been revised in the last month.

1  Windows Help, at Why use a standard user account instead of an administrator account?, has remained constant for quite some time.  It advises that: 

The standard account can help protect your computer by preventing users from making changes that affect everyone who uses the computer, such as deleting files that are required for the computer to work. We recommend creating a standard account for each user.  When you are logged on to Windows with a standard account, you can do almost anything that you can do with an administrator account, but if you want to do something that affects other users of the computer, such as installing software or changing security settings, Windows might ask you to provide a password for an administrator account.

3                   Review of UAC

Windows Help provides an introduction to UAC at What is User Account Control?. In short, UAC prompts the user if an executable will make changes to the computer or to Windows settings. 

In this section, I have defined many UAC terms.  Appreciation of these definitions is essential for studying many of the references I have provided. 

In studying the references I have provided, it is useful to keep in mind that UAC was originally conceived as a convenience rather than as a security measure in its own right.  Whilst UAC does have significant benefits for security, it was conceived as a means of assisting users to log in as Standard users without having to switch over to Admin accounts in order to perform certain tasks.  In other words, it is user account privileges that protect the system whilst UAC merely assists in temporarily switching between accounts.

3.1            Assumed conditions

These assumed conditions set the context within which protective measures, including but not limited to UAC, operate.

The default [recommended] UAC setting is to notify the user using the Secure Desktop mode dialog for both potential changes to the computer [such as installations or running Administrative tools] and for changes to Windows settings.  To proceed, Admin permission must be given in the Secure Desktop mode dialog [also referred to as the Elevation UI].

The Secure Desktop mode dialog [the Elevation UI] needs to be checked.  Unthinking approval of Elevation UIs bypasses its protection against ill-advised user activity, malfunctioning programs or malware.

In addition to UAC, some degree of protection is also achieved by IE’s Protected Mode as this prevents any malware running in an IE session from altering user settings.  IE’s Protected Mode, just for example, stops anything being able to set itself up to start every time a user logs in.  Similarly, some degree of protection is achieved by the IE Smartscreen filter and by Data Execution Prevention (DEP).  Most notably, some degree of protection is also achieved by such real-time anti-malware utilities as Windows Defender & Microsoft Security Essentials as well as their third-party counterparts.

In addition to Standard users & Admin users, Windows 7 also provides for a Guest account that is, by default, disabled.  See Windows Help at What is a guest account?.

The Built-In Administrator account [see section 3.3 below], by default, runs in Admin Approval Mode and UAC does not operate.    The Built-In Administrator account is, by default, disabled and therefore users cannot log in using it.

Windows 7 installs executables, by default, in protected folders such as C:\Program files & C:\Program files (x86).

In Windows Help at What is a password?, as well as in several other Windows Help entries that can be found by searching for password, Microsoft recommends that every account is password-protected with a strong password. 

3.2            UAC operation

A Standard user account has a Standard User Token [also known as a Filtered token] whilst an Admin user has both a Standard User Token & an Admin User Token. 

For both Standard users & Admin users, only Standard User Tokens are invoked when logging in [see Windows Help at What is User Account Control?].  Standard User Tokens are sufficient permission for routine application processes & Windows components to run so that day-to-day computer operations can be undertaken.  An Admin user in this condition is also referred to as a Protected administrator. 

For a Standard user, Admin permission requires the provision of an Admin password or a smart card PIN, fingerprint, etc in the Secure Desktop mode dialog [this Elevation UI is also called the Credential UI].  For an Admin user, Admin permission can be given by clicking Yes in the Secure Desktop mode dialog [this Elevation UI is also called the Consent UI].  After permission is given the Admin user is, in effect, the user in control of the approved process.  An Admin user in this condition is also referred to as a Elevated administrator. 

Admin permission applies only to the executable identified in the Secure Desktop mode dialog [Elevation UI].

3.3            The Built-In Administrator account

However, there is also a Built-In Administrator account that, by default, bypasses UAC & therefore the Secure Desktop mode dialog i.e. no Admin permission is required for changes to the computer or for changes to Windows settings - see the second para of http://msdn.microsoft.com/en-us/library/aa511445.aspx.

It is possible to force the Built-In Admin to use the Secure Desktop mode dialog [the Consent UI] - see Windows Help at User Account Control: Use Admin Approval Mode for the built-in Administrator account. 

3.4            Account privileges

Account privileges can be displayed using this command in each account:-

whoami /user /priv

The results indicate the following:-

1  The privileges of Standard users, Guest users & non-elevated Admin users [Protected administrators] are the same. 

2  The extra privileges of elevated Admin users [Elevated administrators], the Built-In Admin user [in its default condition] & the elevated Built-In Admin user are the same.

3  These extra privileges are

          SeImpersonatePrivilege - Impersonate a client after authentication

          SeCreateGlobalPrivilege - Create global objects

3.5            My assessment of UAC

There is no difference between logging in as a Standard user or a normal Admin user as far as UAC effectiveness is concerned.

The most that can be said for always logging in as a Standard user is that doing so enforces the need for a password in the Elevation UI so, it might be asserted, the user is more likely to pay attention to the Elevation UI than if it merely requires a click on the OK button [as is the case when logged in as an Admin user].  However, I found no evidence to justify such an assertion despite exhaustive searches.

There is a significant difference between logging in as one of these users & logging in as the Built-In Admin since the latter runs with Elevated administrator permissions without any UAC prompts.

4                   Review of malware behaviour

4.1            Malware interactions with Windows 7

Malware that assumes the existence of administrative permissions will fail to run in the absence of these permissions unless the Built-in Admin account is logged in.  The absence of UAC Elevation UIs when the Built-in Admin account is logged in avoids that layer of protection [as already explained, it runs by default with those Admin permissions already granted].

Without Admin permission, malware should only be able to affect the current user's data as it should not be able to make system-wide changes or affect other users’ data.  However, some malware is able to avoid operations that require elevated operations so the absence of Admin permission does not protect against the operation of all malware.

With or without Admin permission being necessary for any given piece of malware to run, Microsoft believe that, with UAC set correctly, malware cannot get onto a system then run without a user’s express consent.  A key design criteria in UAC development has been to try to ensure that express consent is given wittingly i.e. without UAC Elevation UI prompts interrupting the user so much that consent is given unthinkingly.  As stated in the discussion of Microsoft advice in section 2, in early Windows 7 UAC user testing the choice of UAC protection level had no effect on the rate of malware infection.  Subsequent discussion of those results implied, but did not categorically state, that this lack of effectiveness was caused by unthinking decisions made to give Admin permission to an executable after there had been more than two UAC Elevation UI prompts during a logged-in session or within a day.

However, malware can attempt to exploit resources that are common to both the Standard user’s environment & the environment of an operation for which Admin permission has been granted.  For example, an executable that is held in a folder such as C:\Utilities is not given the system protection afforded to executables within C:\Program files or C:\Program files (x86).  Therefore, even if an Elevation UI is correctly generated and correctly identifies the executable concerned, that executable itself might have been hijacked by malware unless the executable is signed [and the signature is witnessed in the Elevation UI]. 

4.2            My assessment of malware interactions with Windows 7

Nothing in these UAC & malware related documents establishes any difference between system vulnerabilities when a Standard user is logged in & when an Admin user is logged in unless any of the assumed conditions in section 3.1 are absent or the Built-In Administrator is logged in. 

Bypassing the protection inherent in the assumed conditions in section 3.1 or logging in as the Built-In Administrator bypasses the significant protection achieved by UAC & other Windows 7 protection measures.

5                   Conclusions

The answer to my initial question is, No, it is not always correct to advise computer owners to log in for day-to-day computer activities as Standard users rather than as Admin users as a system security measure. 

If the assumed conditions at section 3.1 are met & if the Built-In Admin is not logged in, a computer owner can decide whether or not to log in as an Admin user based merely on convenience.
