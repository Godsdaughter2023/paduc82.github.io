techie bear say:

Hello Mike,

It has to do with how the two functions are handled. "Zip" refers to a group of files, a subtype of Windows Explorer, its hardwired to 'protect-files-of-a-given-type' so it renames those files -(1), -(2), etc. ; whereas WindowsImageBackup is a folder - in essence a file -  and single files are hard-wired to 'update-unless-protected'. Its the nature of the beast.

 

For your files "protection", I'd at least designate at least 1 computer with a unique identifier, like a "0".

 

Let me know how you fare.

tryx3 say:

When you restore from an image you are shown this dialog box when the system has analysed the contents of the WindowsImageBackup folder.  Note that the PC concerned is identified & that you can also select a different image [it has to be in the WindowsImageBackup folder though].

 

 

On the face of it, therefore, Windows can handle multiple system's being imaged within that same folder.  You'd just have to look through all the images you can select to find the one that is needed.  This might give you sufficient grounds for feeling that a thorough test was worthwhile.  I have never seen a report of anybody else having done this so I quite understand that you'd want to do several tests before deciding to rely on this arrangement.

System images are not completely reliable but I have only had one that has failed to restore during the eighteen months I have been making weekly images [I rely on them to get back to where I started if I'm recklessly experimenting with the system].

- - - -

As a workaround though [which is what I do]:

If you want to manage the images for complete confidence then you can manually move the whole of the WindowsImageBackup folder into a different folder to have multiple independent images on the same drive.  

I have a folder called Dell7Images that has subfolders for each image so before making yesterday's image I moved last week's WindowsImageBackup folder into H:\Dell7Images\20130519 and it sits there next to H:\Dell7Images\20130512 which has the WindowsImageBackup folder from the week before. 

If I then want to restore an image, I have to move the whole of the current H:\WindowsImageBackup folder out of the way into H:\Dell7Images\20130526 and then move the one I want to restore from, say, H:\Dell7Images\20130519 back into H:\WindowsImageBackup because it is only in that location [off the root folder of any drive] that Windows can work with the image for restoration [which is why the latest one sits in that position for a week where it is ready for action].

In your case, you'd have Computer1\20130526, Computer1\20130526 and so on.

wunders say: How are you getting to Google?  Try typing "http://www.google.com" in the address window of your browser.  If you're still getting the same error, it could be due to malware that has altered your "hosts" file.  Examine your "hosts" file with a text editor such as Notepad.  The hosts file is usually located at C:\Windows\System32\Drivers\etc\hosts.

A default hosts file will have only one line that does not begin with a "#" character and that should be the line:
   127.0.0.1   localhost

If you have other lines in this file, you should clean out your hosts file.
