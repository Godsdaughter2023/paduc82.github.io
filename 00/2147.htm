ossiemac say:

I actually prefer the following code where the source worksheet is referenced by name and assigned to a variable rather than rely on it being the ActiveSheet. The following code can be run when any sheet in the workbook is the ActiveSheet and it still references the correct worksheet as the source containing the original table to be copied.

By assigning the new sheet to a worksheet variable, it can then be given a new name. AFAIK it is not possible to assign the new worksheet to a variable and name it in the one line. These 2 commands must be separated.

Sub CopySheets_2()

    Dim wsSource As Worksheet
    Dim wsNewSht As Worksheet
   
    Set wsSource = Worksheets("Sheet1")     'Edit "Sheet1" to your worksheet name.
   
    Set wsNewSht = Worksheets.Add(After:=Worksheets(2)) 'Assign new sheet to a worksheet variable
    wsNewSht.Name = "NewSheet"      'Rename new sheet
   
    wsSource.Range("AllData[#All]").Copy Destination:=wsNewSht.Range("A1")
   
    wsNewSht.Range("A1").ListObject.Name = "MyNewName"
   
End Sub

ossiemac say:

The reason that the Copy and Paste does not work is because when a new worksheet is added then the new worksheet becomes the ActiveSheet and the "Range("AllData[#All]").Copy "  code is referencing the ActiveSheet (which is "NewSheet" and of course AllData is not on that sheet).

Start by assigning the ActiveSheet to a worksheet variable and then use the worksheet variable to reference the original ActiveSheet.

To rename the table then reference the table from any cell which is known to be within the table. Cell A1 is the best choice because that is where you are pasting the table so it must be part of the new table. The new sheet can now be referenced as the ActiveSheet or see my alternative commented out code by referencing it by its name.

Sub CopySheets()

    Dim wsActive As Worksheet
   
    Set wsActive = ActiveSheet      'Assign the ActiveSheet to a worksheet variable
   
    Worksheets.Add(After:=Worksheets(2)).Name = "NewSheet"
   
    wsActive.Range("AllData[#All]").Copy Destination:=Worksheets("NewSheet").Range("A1")
   
    ActiveSheet.Range("A1").ListObject.Name = "MyNewName"
   
    'Alternative to ActiveSheet because worksheet name is known.
    'Worksheets("NewSheet").Range("A1").ListObject.Name = "MyNewName"
End Sub

Windows 10 updates, UUP and Catch 22! - Microsoft Community

georgeogden say:

Windows 10 updates, UUP and Catch 22!

When the release of Insider build 16215 to the “Fast Ring” caused my test PC running 16199 to fail to update (and even clean install using that or any subsequent build) I discovered that the underlying reason appeared to be missing (corrupted?) drivers for its aging motherboard (c.2007) – and probably those for other components, NIC, graphics etc. but there’s no way to tell without a basic install. Either way, the PC has stopped dead in its tracks as far as Windows 10 was concerned. Annoyingly rolling back to 16193 doesn’t stop the updates being attempted again and again and again although the graphics driver for a newer card (c.2016) were updated without a problem. (I’m running the 64 bit version and, yes, the 32 bit version runs 16241 just fine so the problem is only with 64bit drivers or perhaps changes to WOW64 but this is only tangential to the main topic I’m discussing here.)

This got me to thinking…. Windows is now employing a SAS model and is no longer a “stand alone” OS platform on which to run 3rd party applications. So, how on earth could Windows not only continue to deliver enhanced OS features and extend functionality while, at the same time, continuing to support the underlying hardware components and the drivers necessary for them to function… adequately if at all?

The new features and functions of Windows 10 are to be lauded and, if not all, most will be appreciated by Windows 10 users but naturally, over time, this will place increasing demands on the hardware to a point where the PCs performance becomes unacceptably slow. (It’s then probably time to upgrade the PC and I’ve no problem with that.)

In parallel, the number of hardware components, motherboards, CPUs, NICs, storage devices and graphics cards to name only the more obvious ones, will grow in ever increasing numbers to a point when maintaining the drivers will become unsustainable both in terms of complexity and the cost of doing so – that point has probably already been and gone.

So, it would be unreasonable to expect Microsoft, and OEMs, to maintain and update hardware drivers in perpetuity – particularly as new hardware typically delivers new features and enhanced performance that the OS could employ. But dropping support for a driver should not render a PC totally useless – it should, at worst, as in the past halt progression to newer versions of the OS. I’d suggest that today with the SAS model and UUP future features and functionality updates that are independent of the underlying hardware could still be delivered. More importantly, what about the security updates? Even if a newer build can no longer support a legacy drivers, why shouldn’t they continue?

With Windows versions prior to 10 hardware driver compatibility was clearly obvious – the component just didn’t work properly… if at all. However, to some extent the line between delivering Windows updates, features and functionality, and hardware drivers was blurred when Windows 7 started to take on the responsibility of delivery of some of these drivers. This was no bad thing for the vast majority of Windows users who, in the past, had never even thought to update hardware drivers either because they didn’t know that they could, should, or if they did simply didn’t have the knowledge of how to do it themselves.

Now, to the crux of the matter…  I’d not be unhappy if, at some stage in the future, support for my hardware ceased but I would still like it to function even if I knew that there would be no further updates to the hardware drivers and, consequently, I could, potentially, miss out on future enhancements in Windows 10. I don’t believe that’s an unreasonable expectation!  However, dropping drivers without any warning is totally unacceptable.

Employing the car analogy used elsewhere in this forum about driver updates… My car worked when I drove it out of the saleroom and, even though Ford have released newer models, I still expect to wake up in the morning, start my 10-year-old Ford and drive it from A to B with the same passengers and luggage on board – I’m also pretty certain that if I asked Ford to upgrade or install a new improved digital radio or satnav they’d be only too willing to do so and take my money. My car shouldn’t cease to function simply because the model is no longer in production or spare parts are no longer available from the OEM.

As I said at the start, my test PC was running Windows 10 64bit right up to build 16199 without a problem and, if I understand it correctly, UUP played a part in that by migrating the motherboard’s hardware drivers along with others but why that ceased with build 16215 and subsequent builds I have no idea. Perhaps a bug or possibly an intentional decision not to support the drivers going forward – I don’t know as Microsoft have not responded nor made any public statement (that I can find) either way.

A clean install, on a bare drive, has been possible up to an including 16193 but impossible with builds 16215 and above - 16199 might also work but I have no ISO to test it.

So, what will happen in the future when Windows 10 drops support for the motherboard drivers that were used to install Windows 10 in the first place? This makes the need to have an off-line recovery option ,with the latest drivers included, an absolute necessity.

As it may be already, or possibly soon will be, impossible to include ALL drivers in an ISO the migration to on-line installation of drivers and their updates makes sense BUT without the OEM's motherboard drivers available off-line there’s no way to clean install Windows 10 on a bare drive – Catch 22!

If you read this far I’d appreciate your comments – just keep them clean (pun intended) :) :)
