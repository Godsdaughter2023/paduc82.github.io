You can change it via PowerShell script, but you will have to adjust this for any new user as well. Here's an example:

$calendars = Get-Mailbox -RecipientTypeDetails UserMailbox | Get-MailboxFolderStatistics | ? {$_.FolderType -eq "Calendar"} | select @{n="Identity"; e={$_.Identity.Replace("\",":\")}}

stephentu say: OK, since no one seems to know anything, or the ones that do are on vacation or don't follow this board, I did a lot of digging and experimentation and have found my own answers.

My findings:
1. The problem only exists on IE10 on windows 8 and windows RT machines with touchscreen capabilities.  Non-touch win8 machines do not have this issue.  This was replicated on a number of different machines of each type. The difference?  The user-agent string sent by IE10 to the website; touch machines send roughly (some differences depending on 32 vs 64-bit, ARM vs. x86):
Mozilla/5.0 (compatible; MSIE 10.0; Windows NT 6.2; WOW64; Trident/6.0; Touch)
while non-touch machines omit the "; Touch" at the end. 
In fact one can duplicate this behavior on windows 7 / ie9 by changing the user-agent string to mimic this (via registry hack or UAPick add-on for IE).

So it looks like Amazon deserves a large percentage of the blame here; their website is sending something subtly different when it encounters the MSIE 10.0 / Touch combination that breaks things.  Still, I think MSFT could have done better by forcing IE10 to send the IE9 user-agent string for amazon, see #2 below.

The workaround is to force compatibility mode, which forces IE to send a different user-agent string, basically pretending that it is IE7.

2. The answer to my question about the iecompatdata.xml file is that "emulateIE9" only affects the rendering of the page, not the user-agent string sent to the site, so amazon still thinks it is seeing IE10/touch and is responding with something incompatible.  In order for the user-agent string to be affected, the iecompatdata.xml entry for amazon.com would also require a uaString setting, looking like:
 <domain docMode="EmulateIE9" uaString="9">amazon.com</domain>
instead of simply
<domain docMode="EmulateIE9">amazon.com</domain>
as it does now.
Manually changing the iecompatadata.xml file (under %localappdata%\Microsoft\Internet Explorer\iecompatdata) and deleting browser history would also work, but then you'd have to disable the downloading of updated compatibility lists which is probably a bad idea.  Better to just use compatibility mode for amazon until the site (or official compatibility list) gets fixed.

I will send some note to Amazon to see if they can look into it and change things on their end.
