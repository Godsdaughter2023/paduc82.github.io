john mcghie say:

Hi Jasta:

For this kind of thing, I would certainly use a table to get things lined up.  If you go to the Format menu and set the Borders to NONE, the table will be totally invisible when printed.

If you go to the Table menu and check "Gridlines", you will be able to see the table when you are working, but the lines won't print.

Certainly don't use Publishing View.  PC Word does not have this view, and it can cause major problems (crashing...).

You need to know that Word is designed to FLOW text.  Everything in Word is "stacked" on on top of the other going down the page, beginning with the top left character.  It is very difficult to prevent this: it's the way Word works.  It is important to remember that until you put something in it, a Word document is "empty", it is not "blank".  With a piece of paper, you have the fabric of the paper to hold things apart.  In Word, there's nothing there: anything you put in the document floats up like a balloon until it bumps into the next higher object.

If you want graphics objects to be lined up by the cells of a table, you need to set their "Layout" to be "Inline with text" (right-click the shape and choose Format Shape).

A Word document has a large number of layers, but there is no easy way to manage them.  When an object is set to Inline with Text, the object is in layer 0, on the same level as the text.  If the layout is set to anything else, the object can be on any layer that is in front of the text, or any layer that is behind the text.  I don't know how many layers there are: at least 99 either side of the text.  As you add shapes to a document, each one gets its own layer.  You can then use "Arrange" on the right-click menu to send it back or bring it forward, one layer at a time, to create the layering you want.

By definition, a shape is always visible.  But if you set its Fill to "None" and its Line to "None" it becomes very hard to find! I would set the "Line" to light grey while working, then set it to "None" when I wanted to hide it.

Now, as Rich implied, it would be helpful to know a little more about what you are trying to do, and why you are trying to do this in Word.  I have a horrid feeling you are about to strike some limitations that Word simply can't do.

One of the first is that if you give said template to a user, when they can't get things to fit, they will simply drag their shapes around.  As soon as you drag a shape, you "float" it out of the text layer: it can then go anywhere you like, but never where you want, unconstrained by the table.  You will find the users will cheerfully drag stuff off the print image then complain that their pix are getting cut off.

Note also that Word has no concept of a "page" in the file. Word makes up pages at print time, like an old-fashioned compositor.  There are not "pages" in a document file, and no way to control which page an object will print on.

This is a word-processor.  It was designed to flow text in columns.  They have added a whole lot of junk to try to make it behave like a page layout program, but that was NOT a good idea, and this eye-candy they added to entertain the clueless newbies tends to break very easily and cause all manner of problems in professional production.

Hope this helps

Table sort with two or more header rows - Microsoft Community

honest abe say:

Table sort with two or more header rows

PROBLEM:  We continually face a problem with sorting tables (Excel and Word) if we employ two or more rows for the column headings in the table.

DISCUSSION:  First, it is not uncommon for tables to have headings of two or more rows.  Often the first row includes labels that group the sub-labels in the second row and so on.  However, we cannot create such headings because they disrupt the sort feature used with tables.  Second, the visual or printed headings may not be conducive to the sort feature.

As such, the user often has to use workarounds where multi-row column headings must lie outside of the table in order to not 'disrupt' the sort feature.  The downside is that the column label headers no longer respond to the modifications made within the table, such as table moves or columns size adjustments.

Apparently, Word and Excel tables can only sort by a one row header, and the user has no way to instruct the program which of the header rows include the unique column headings.  Furthermore, the column headings have to be unique, which often conflicts with the user's desired visual implementation of column headings.  For example, the headings may have two sections visually designated in row 1, with the same subsection labels used for both sections shown in row 2.  As such, neither row includes truly unique labels for each column; something necessary for a sort operation.

SOLUTION:  In our opinion, the best solution would be to allow users to create multiple row column headings, but then allow the user to designate which row contains the unique column headings for a sort operation.  This way the user can have the visual headings that they want printed at the top of the table and the top of split pages, but also include a hidden row with unique names used only in the sort operation.

Perhaps we have overlooked some other features, so would appreciate anyone's suggestions in this regard.

suzanne s. barnhill say:

You have received a reply regarding Excel, but, as you say, the Excel forum is a better place for that question. Word and Excel are different because they serve different purposes. Word has "tables"; Excel has "spreadsheets." Tables are a way of arranging text. Spreadsheets are a way to perform mathematical functions. Calculation in Word tables is very limited compared to Excel. Formatting in Excel spreadsheets is limited compared to Word.

I don't know what you mean by a "hidden header row." Do you mean one with text formatted as "Hidden"? You can hide text in a table, but that doesn't hide the row. If it's formatted as a header row, it will be so treated in the sort.

suzanne s. barnhill say:

Okay, I take your point in (1). In such instances, I think your only recourse is to do what I believe you said you're already doing: split the table temporarily.

Actually, table styles do respect header rows. If you have more than one row designated as a header row and have checked the box for "Header row" on the Table Tools | Design tab, any special formatting for the header rows will apply to all rows designated as header rows. In my tests just now, this required that you (re)designate the header rows (by clicking Repeat Header Rows on the Table Tools | Layout tab) after selecting table style on the Design tab, but you do get the desired effect.

As to (3), no, this wouldn't work, so the workarounds you've already described would be necessary.

I won't comment on (4) and (5) since this is a forum for Word, and I have little experience with formatting in Excel, which I use primarily for data and calculations.

d0gknees say:

While some of these differences may seem confusing to the occasional user, they make perfect sense when you have used them more regularly and when you use the more complex functionality in each.

For example, cells Tables in Word can be split and combined in a pretty arbitrary way, where as in Excel it makes no sense to split a cell as the addressing would break. Say you want to split cell A1. What would the address of each new cell be? If you totaled column A, should both halves get added or just one?

The key is that while Tables exist in Word, Ecxel and Access, their behaviour and purposes are very different. It would make a lot of things impossible if they were all the same and some of the behaviours are mutually exclusive.

An example from Access. In a database table every row must be different in some way. It is simply impossible to add data that is identical. This limit makes no sense in Word or Excel but without it Access would not give reliable results.
