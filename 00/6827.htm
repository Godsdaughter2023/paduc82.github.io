Hi,

First of all, I want to warn that Excel protection is not very strong and can be hacked in few minutes. The policy of the forum does not allow to go in depth on this (google is your best friend) but on the other hand, most people do not make the effort to do this.

IMO, checking a website is not a good idea because that requires an internet connection, so your legitimate team members wouldn't be able to use the tool off-line or in case your web site is down.

A "tool" is vague, is it just a file or an add-in? In case of an add-in you could use the add-in install event to write something into the registry. Thinking about the MAC-address (unique) of the PC and a code that the user has to enter.

http://www.myengineeringworld.net/2014/12/get-public-ip-local-ip-mac-address-vba.html

During the installation of the tool, you can combine the MAC with the code and at opening read the MAC and the code and check of this combination is the same in the registry. MAC addresses are not really world wide unique but the chance that an ex-employee gets the same address in another company is almost nil.

For reading and writing the registry see:SaveSetting and GetSetting

m h g say:

Easy to set Excel's Application.EnableEvents to FALSE, then open any workbook, and those workbooks' Open event handlers won't run. That defeats your plan.

If you truly want to make Excel workbooks unusable except within controlled environments, give those workbooks File-Open passwords and use an outside program (even a Windows Script Host/VBScript script) to use Automation to launch Excel application instances, open those files with the necessary File-Open passwords and with events enabled, then make the application instance visible. Even so, once the workbook is open, it's easy to copy everything from the workbook into a new, unprotected workbook.

Simple rule: there's no robust way to protect ANYTHING in an open Excel workbook.

Excel and VBA internal passwords are quite weak. There are hundreds of web sites showing how to defeat both. Don't think you're securing anything using them. They're about as effective as using a piece of rope with a tight knot instead of a padlock. If you don't believe me, send me a copy of your protected file, and I'll send you back a copy with all protections removed.

m h g say: Almost as easy to break into the VBA project in an .XLAM add-in as VBA projects in .XLSMs/.XLSBs, so any MAC address could be by-passed. Compiled .XLL add-ins, OTOH, would be more secure.
