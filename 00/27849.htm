ericvond say: There seems to be no support from Microsoft's side, So I'm gonna place my workaround here. I hope this will be usefull to you and other users encountering the issue in the future.

I have to add that I am not a macro developer and have compiled this using the record macro option in word and a code snippet I found online. This snippet was used to delete all images inside a document. I replaced the delete command with my recorded macro of adding a caption.

Plus, this does not immediately add the captions to your images and running it twice in the same document will add a second caption to the images.
Still I hope this will help some of you. 

First of all, define a style for your images (for example Figure 1-1) and give it a unique name.

Sub AutoCapAllPics()

Dim oIShape As InlineShape
        For Each oIShape In ActiveDocument.InlineShapes
                With CaptionLabels("Figure")
                    .NumberStyle = wdCaptionNumberStyleArabic
                    .IncludeChapterNumber = True
                    .ChapterStyleLevel = 1
                    .Separator = wdSeparatorHyphen
                End With
            oIShape.Select
            Selection.InsertCaption Label:="Figure", TitleAutoText:="InsertCaption1", _
            Title:="", Position:=wdCaptionPositionBelow, ExcludeLabel:=0
        Next oIShape
End Sub

Specifies the number of days for which items are stored in the scan log folder. After this time, Windows Defender deletes the items. Troubleshooting Office 365 Calendar sharing:"Why can't share my Calendar with my colleagues"

Technical Level : Intermediate

Summary

In this article we will walk you though the troubleshooting steps you need to take to troubleshoot the following situations:

  1. User/Delegate unable to view/create/modify items in another user's Calendar.
  2. User unable to share his Calendar folder
  3. User unable to view/grant Calendar permissions
  4. Users unable to share their calendar by using calendar sharing

This covers both cross premises issue as well as cloud to cloud or on-premises to on-premises scenarios.


Details

Start troubleshooting by running SARA on the computer that's having the problem (if the mailbox is hosted in Office365).

    • The SARA logs are available at: %LOCALAPPDATA% in the SaraLogs and SaraResults folders.

Note: If this is a cross-premises scenario where the user is not able to create/modify items then note that cross-Premises Outlook delegation/folder permissions is NOT supported at this time. However, we DO support cross premises full access permissions scenarios. This is documented here.

      1. If the target user is a shared mailbox, user mailbox, room mailbox etc.
      2. If all protocols are enabled.
      3. Type of EXO licenses assigned to the users.

Note: Calendar editing, delegation, access by using Outlook are NOT supported if the delegator has a Kiosk license. MAPI is disabled for Kiosk users and therefore the delegate will only be able to access the delegator's calendar by using OWA protocol. See this articlefor additional details on the subscription plans and what it's included

Collect an OffCAT from the impacted user. Make sure Outlook is running when you collect it.

Note: OffCAT is a Windows tool and cannot be installed on a Mac OS X computer. Some information given by OffCAT may be relevant also if the issue is seen only in OWA as the data is stored on the mailbox (e.g shared calendars, delegate related props etc). See Office Configuration Analyzer Tool (OffCAT)for additional information on how to review the log.

If the issue can be reproduced in Outlook, then enable Outlook logging and collect the Outlook logs while reproducing the issue.

    • Outlook for Windows Logging
    • Outlook for Mac Logging

If you suspect or narrowed down to a possible corruption within the Calendar folder, then run Calcheck with the switches -F -R -A and collect the CalCheck.log and CalCheckerr.csv file on the impacted mailbox and target mailbox (if applicable).

Note: Launching Calcheck.exe as specified above, will create a CalCheck folder, and move flagged error items to the folder, and put a report message in the inbox. Additionally, please note that Calcheck is a Windows tool and cannot be installed on a Mac OS X computer.

Note: The RCA results are useful if the issue can be replicated from external by using Outlook. Additionally, if external access is blocked it is expected to see the below tests fail. Move to the next step if external access is blocked or if the issue cannot be replicated from external.

Outlook autodiscover

  1. Use the Test E-mail AutoConfiguration tool to determine whether Outlook can connect to the Autodiscover service and collect screen shots from Results and Log tabs.

Iv got the below code in a module, then behind a cmd button on a form i have  ExecuteFile Me.DocLink, "Print" - (me.doclink contains a file path).  This works well, but the document opens, prints, then closes.  Can i stop the document actually opening or being visible if it does?

My module code is:

'Source: http://www.pacificdb.com.au/MVP/Code/ExeFile.htm
Public Const SW_HIDE = 0
Public Const SW_MINIMIZE = 7
Public Const SW_RESTORE = 9
Public Const SW_SHOW = 5
Public Const SW_SHOWMAXIMIZED = 3
Public Const SW_SHOWMINIMIZED = 2
Public Const SW_SHOWMINNOACTIVE = 7
Public Const SW_SHOWNA = 8
Public Const SW_SHOWNOACTIVATE = 4
Public Const SW_SHOWNORMAL = 1

Public Declare Function ShellExecute Lib "shell32.dll" Alias "ShellExecuteA" _
    (ByVal hWnd As Long, ByVal lpOperation As String, ByVal lpFile As String, _
    ByVal lpParameters As String, ByVal lpDirectory As String, _
    ByVal nShowCmd As Long) As Long

Public Sub ExecuteFile(sFileName As String, sAction As String)
    Dim vReturn As Long
    'sAction can be either "Open" or "Print".
   
    If ShellExecute(Access.hWndAccessApp, sAction, sFileName, vbNullString, "", SW_SHOWNORMAL) < 33 Then
        DoCmd.Beep
        MsgBox "File not found."
    End If
End Sub

hansv mvp say:

If neither

If ShellExecute(Access.hWndAccessApp, sAction, sFileName, vbNullString, "", SW_SHOWMINNOACTIVE) < 33

nor

If ShellExecute(Access.hWndAccessApp, sAction, sFileName, vbNullString, "", SW_HIDE) < 33

works, you can't hide the window using ShellExecute.

Note to others: the SW_... constants don't correspond to bit positions, they are just enumerated values 0 ... 9, so you can't combine them using OR.
