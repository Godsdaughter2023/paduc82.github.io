APA formatting is incorrect, is there a patch?__

The APA formatting for references is incorrect.  The font, size, spacing, and indent are not correct.  Is there a patch to fix the problem?
yves dhondt say:
There is a problem with the way Word 2007 lists the bibliography / work cited at the end of the document. APA style reference list should be with the first line of the reference flushed to the left, and each subsequent line indented 5 spaces or 1/2". Well when list of references is inserted in Word 2007, every line of the source is flushed to the left, which is incorrect. Is there a fix for this?

Bibliographies are formatted using the MsoBibliography style which by default inherits from the MsoNormal style. If you want a different formatting, like with an indentation, you will have to adjust the style. What you see is therefore normal behaviour and not a bug which needs fixing.

The moment you inserted a bibliography in your document, the 'Bibliography' style became available in the style dialog (CTRL+ALT+SHIFT+S). Right click on the style and select 'Modify...'. In the dialog, click the 'Format' button and choose 'Paragraph...'. On the 'Indents and Spacing' tab, set the 'Special' dropdown to 'Hanging'. Also change the line spacing if required.

Note: the name of the style is localized. So if you were using an English copy of Word, you would see "Bibliography" while if you were using a Dutch copy, you would see "Bibliografie".

bill mosca say: Kathy

Requery the combo right after you assign a new RowSource. Don't do it in the form's current event.

marshall barton say:

Your AfterUpdate and Current events are using two different dependent combo box techniques that do not go together.  If you are going to set the RowSource in the AfterUpdate event, then the Current event does not gain anything by using Requery.  Instead the Current event also needs to set the row source.  In your case the code in the AfterUpdate event does exactly that so you can just call it:

 

   Private Sub Form_Current()
       Call Country_AfterUpdate()
   End Sub

 

As for your question, you need to explain why the combo box is not bound to the field where you want to save the value.  That is the simple, standard way to do things so if you have a special situation where that won't do whatever it is you want, we need to understand what that problem is before trying to suggest a complete solution.

ken sheridan say: In a case like this it's the first combo box (Country) which should be unbound, not the second (State), and the form's underlying table should only include only a State column (field), not a Country column.  Country is determined by State so including a Country column in the table would introduce redundancy into the table, and leave it open to the risk of update anomalies.  Each non-key column in a table should be functionally determined solely by the whole of the primary key of the table.  In your case Country is determined by the non-key column State, so is transitively dependant on the key via State.  The table is therefore not normalized to Third Normal Form (3NF).  It would be possible for mutually inconsistent values to be inserted into Country and State.

You can still enter data in the way you wish in the context of a normalized table, selecting a country first, then a state.  You'll find an example as ComboDemo.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file uses the local administrative areas of county, district ad civil parish in my area as an example, but the principle is exactly the same.  As you are only concerned with two levels of hierarchy, the form illustrating a simple 2-level hierarchy would be the appropriate model in your case.

Tip when using beta testing software - Microsoft Community

joenamco say:

Tip when using beta testing software

When using beta testing software, do a dual boot system first boot is a "clean" install with nothing added, 2nd boot same version installed which you can add delete "stuff", this way when an issues shows (which it will) after an upgrade you can check the 1st boot and see if the clean install/upgrade runs without issues, if it does you will know that it is something you added/removed from the 2nd boot, saves a lot of hair pulling and time, I've been testing systems since the early 60's and found this method helps even with mechanical systems.
