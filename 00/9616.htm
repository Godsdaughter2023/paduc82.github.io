Keith wrote:

I keep getting #DIV/ 0 error for my IRR calculation. The issue lies in the highlighted yellow box in excel sheet one. I want to make the repayment period to be say 15 or 20 years. But when i do so, i get the error. It is only if i keep the repayment period below say 5 years will i get an IRR value. Could you help me with this issue?

When Excel IRR returns #DIV/0, we should interpret it the same as #NUM, namely:  Excel IRR could not find a solution within the limits of the internal algorithm.

The problem is:  you are trying to model the operational net cash flows (costs minus revenues) of the business; and in that context, the "inflow" from a loan should not be counted as operational revenue.  So rows 23:25 should not be part of the operational model.

Ironically, if we delete the values in B23 and C24 (there is nothing in row 25), Excel IRR still returns #NUM.

But in that case, the problem is simply:  Excel IRR needs a reasonable guess.  IRR(B31:CW31,5%) returns about 2.79581573724528% in B34.  And B31+NPV(B34,C31:CW31) returns about -5.36441802978516E-07, which is close enough to zero.

FYI, I chose a guess of 5% because when we create a table of IRRs from -99% to 100% and calculate NPV based on them, as I did in my previous response in this thread [1], we see that the NPV appears to cross zero between 0% and 10%.  So I chose the midpoint for my guess.

[1] [EDIT] That is, the previous response dated 4/24/2016 3:09:45 AM in the thread where this response originated (click here).

(I got lucky.  It would have been prudent to then look at the NPV for IRRs from 0% to 10%, and see that it appears to cross zero between 2% and 3%.  So a better guess would be 2.5%.)

First the simple answer - you cannot install the Office apps on your hard drive - Office can (will) only install to the default system drive (in your case, the SSD drive). Not what you want to hear but unfortunately, that's the reality.

In terms of Outlook data, that would depend on the type (i.e. POP3, Imap, Outlook.com/Exchange) account involved which also brings up the following questions

1) Why are you "importing" a PST file?
2) A PST file typically belongs to a POP3 account so the next question is - are you configuring the account as IMAP instead of POP3? If not, there typically would be no reason to import anything.

Data File locations .............
For POP3 - can be located anywhere on any hard drive so using your HDD would definitely not be an issie

For IMAP/Exchange accounts - the default location is on the Windows system drive (C:\Drive or SSD in your case). There is no option directly within the Outlook UI to change that. However the following instructions describe how to force a change in location of .ost files associated with those account types
