julie sheets say:

Pardon me for bumping in.  A single task with an ALAP constraint will not not delay other tasks. However, if the task has a successor, the successor relationship over rules (for want of a better term) the ASAP constraint of the successor, resulting in the successor aligning with the project finish date.

The two pictures below show the effect of a link relationship on ALAP tasks and their successors. 

Task ID 1 is ALAP, all other tasks are ASAP.  Without a successor, its finish is equal to the project finish date.

Once a successor is added the ALAP constraint still pushed Task ID1 as late a possible, now taking into consideration the successor.  Task ID 2 is ASAP given its link relationship.

If you want a zero slack schedule (and I agree with Trevor - not wise) you can do the following.

Add a milestone with a must finish on constraint as the last task in the project.  The constraint date will be equal to your contractual finish date.  This will show any slack you have.  Now, change the project you show to your client to be scheduled from the finish date.  This will remove all slack from the tasks moving everything to align with the finish date.

hal hostetler - mvp-outlook say:

A "Personal Folder" (.PST) file is a local storage file associated with a POP account.  An IMAP account creates a different storage file called an Offline Store File or .OST that is a mirror image of the server folder structure.  What I suspect may have happened is that you now have two sets of folders, a .PST file from your old POP account and a new .OST file that works with your new IMAP account.  New email should be delivered to this file and, when deleted from here, will also delete from the server.  Anything you do in a .PST file is local and not reflected on the server.

Pst-files vs. pst-archives vs. ost-files
