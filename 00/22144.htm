bill mosca say:

That will probably work providing you don't need the actual dates for each day of the week. I'd start with gathering the requirements for displaying the data in reports. It might prove cumbersome dealing with a table that is not normalized.

a tanve say:

In your opinion table should be: Resouce ID, Project ID, Date, Hrs?

ken sheridan say: That's not a good design as you are doing what's known as 'encoding data as column headings'.  It's more akin to a spreadsheet than a relational database table.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

A correct structure would be:

Resources
....ResourceID (PK)
....ResourceName
....etc

Projects
....ProjectID (PK)
....ProjectName
....StartDate
....EndDate

The table in which the hours per resource per project per day are recorded is modelling a many-to-many relationship type between Resources and Projects:

ProjectResourcesUsed
....ProjectID
....ResourceID
....DateUsed
....HoursUsed

This table's primary key is a composite one made up of the ProjectID, ResourceID and DateUsed columns.

With this structure it is a simple task to aggregate the hours over any period, week, month , whatever by grouping a query.  For data input you can have a resources form, in single form view, with a ProjectResourcesUsed subform embedded in it, linked on ResourceID.  You can also have a projects form, in single form view, with a ProjectResources subform embedded in it, linked on ProjectID.  To validate the DateUsed value entered you can referenced the value of the StartDate column in the related row of the Projects table, and, if the project has a termination date, the EndDate column.

The above model is something of an oversimplification as it allows any resource to be selected per project, or vice versa.  It may well be that only certain resources are used on a specific project, in which case you would ideally introduce another table, ProjectResources say to model this:

ProjectResources
....ProjectID
....ResourceID

This table is all key, i.e. its primary key is a composite one made up of the ProjectID and  ResourceID columns.  The relationship between this table and ProjectResourcesUsed would be on these two columns, so only resources legitimate to a project, or vice versa, can be entered in ProjectResourcesUsed.
