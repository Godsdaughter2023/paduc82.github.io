alexandru cazacu say:

How to purge mailboxes in Exchange Online

I made a video about this a while back, which can be found here: https://channel9.msdn.com/Series/Welcome-to-the-Office-365-Support-Corner/How-To-Video-Purge-Soft-deleted-mailboxes-in-O365.

The scope of this article, is to re-share the above video :) and to provide some sort of a transcript on what is discussed in the video:

Why would we want to do such a thing, you may ask. The most obvious reasons are:

- provisioning of new mailboxes results in conflict
- we have issues with an active mailbox and we discover that the cause is a soft deleted mailbox that has the same details (this can occur during an e-Discovery)
- we are performing a migration from on premise Exchange to Exchange Online, and mailboxes are not being migrated and we have errors

Let's begin by discussing a bit our today's Agenda:

What was the previous behavior of a soft deleted mailbox in Office 365 ?
What is the current behavior of a soft deleted mail here and here 


1. How can a mailbox go into a soft deleted state in Office 365?

- Login to Office 365 Admin Center and go to Active Users
- Select a user that has Exchange license assigned and delete him
- The user will go to Deleted Users in Office 365 Admin Center and his Exchange mailbox will go into a soft deleted state
- Purge the user completely from Office 365, using PS command Remove-MsolUser -UserPrincipalName *** Email address is removed for privacy *** -RemoveFromRecycleBin -Force
- The user will be hard deleted, but his mailbox will still be in a soft deleted state, but it will no longer have the ExternalDirectoryObjectId populated
- ExternalDirectoryObjectId is in fact the ObjectID of the Msol User that we just hard deleted
- Important to keep in mind that only when deleting a user while still having a license assigned, or if the mailbox is placed on Litigation Hold or In-Place Hold, will cause the mailbox User
- The solution here is obviously to first purge the soft deleted Msol User with Remove-MsolUser -UserPrincipalName *** Email address is removed for privacy *** -RemoveFromRecycleBin -Force
- After the user has been hard deleted, we confirm that we no longer have an ExternalDirectoryObjectId populated on the soft deleted mailbox and we run the usual command:

Remove-Mailbox -Identity *** Email address is removed for privacy *** -PermanentlyDelete


3.3 The operation couldn't be performed because it is outside the writing scope of this server 

- This is encountered when we again follow the steps from scenario 1, but the user is synchronized from Active Directory
- Or instead of following the steps from scenario 1, we simply either delete the user from local Active Directory and run a sync, or we just move the user into an OU which is not synced with Office 365, case in which again AAD Connect, will trigger a soft deletion of the user in Office 365. Again, the user is remove while still having the license assigned.
- After the user has been soft deleted, we go further and we hard delete him
- We go directly to scenario 2 and try to purge the remaining soft deleted mailbox, but we receive the above error
- The issue is caused by the IsDirSynced flag setup on the soft deleted mailbox
- To bypass this issue and completely get rid of the soft deleted mailbox, we must first reconnect the mailbox to new, unique test user, by running this command:

New-Mailbox -InactiveMailbox <address of soft deleted mailbox or guid> -Name <new unique name> -MicrosoftOnlineServicesID <a new unique UPN>

- Do not attempt to use in the above command the parameter UserPrincipalName, as the system will not let you, instead go with MicrosoftOnlineServicesID
- After you press Enter on the command, you need to specify a password for this new user mailbox. Keep in mind that you Get-Mailbox -SoftDeletedMailbox | Remove-Mailbox -Permanently
- The solution is to purge mailboxes once at a time, or run these 2 commands in Powershell:

$mbx=Get-Mailbox -SoftDeletedMailbox | select ExchangeGuid
$mbx | % {Remove-Mailbox -Identity $_.ExchangeGuid -PermanentlyDelete -Confirm:$False}

- We use the ExchangeGuid in order to avoid having issues with duplicates


3.5. Mailbox is placed on Litigation Hold or In-Place Hold and cannot be deleted - Litigation Hold

- We create a user mailbox and we place it on Litigation Hold
- After a while we decide to delete the user mailbox that we have created and we begin by removing the Msol User in Office 365 Admin Center (removing the license or not, does not affect us, since the mailbox is placed on hold)
- We also purge the user completely from Office 365 using Powershell command "Remove-MsolUser -UserPrincipalName *** and then Remove-Mailbox -Identity *** Email address is removed for privacy *** -PermanentlyDelete

- Another switch to keep in mind, is the ExcludeFromAllOrgHolds used with the Set-Mailbox cmdlet only for inactive mailboxes.

3.6. Mailbox is placed on Litigation Hold or In-Place Hold and cannot be deleted - In-Place Hold

IMPORTANT: Below instructions apply to an older scenario when In-Place Hold was setup through the Exchange Admin Center e-Discovery feature.

- We create a user mailbox and then we include his mailbox in a mailbox search in Exchange Admin Center/e-Discovery
- During the mailbox search setup, we also check the box "In-Place Hold"
- After a while we decide to delete the user mailbox that we have created and we begin by removing the Msol User in Office 365 Admin Center (removing the license or not, does not affect us, since the mailbox is placed on hold)
Get-MailboxSearch -InPlaceHoldIdentity <In-Place Hold GUID> | FL Name. This command will show us what mailbox search contains this inactive mailbox.
- The next step is to remove the mailbox from the search that has been place on In-Place Hold, and we again use Powershell to do this:

$InPlaceHold = Get-MailboxSearch -InPlaceHoldIdentity <In-Place Hold GUID>

$InPlaceHold.Sources - we determine what mailboxes are included in the search that is place on In-Place Hold

$InPlaceHold.Sources.Remove("<LegacyExchangeDN of the inactive mailbox>") - we remove the inactive mailbox that we need to purge, from this mailbox search

$InPlaceHold.Sources - we check that the mailbox has been removed from the search

Set-MailboxSearch $InPlaceHold.Name -SourceMailboxes $InPlaceHold.Sources - we modify the search with the updated list of mailboxes include Remove-Mailbox -Identity *** Email address is removed for privacy *** -PermanentlyDelete

Tables have no order - they should be viewed as "buckets of records". If you want to see records in a particular order you must use a Query and sort the data there. A Combo Box gets its data from either a Table (in which case the order is whatever Access finds convenient) or a Query (which you can sort yourself).

Click the ... icon by the Row Source property of your combo box, and accept Access' offer to create a Query. Select Ascending or Descending (as desired) under the field that you want sorted and close the query. Access will ask you to verify the change - do so, and your combo will now appear sorted.
