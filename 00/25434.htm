User Security Access 2013 - Microsoft Community

katherine richardson say:

User Security Access 2013

Is there a way in Access 2013 to allow a user the ability to edit only the records assigned to that user?  The supervisor is okay with a user viewing another's records, but wants to limit the ability to edit a record to the user the record is assigned to.

ken sheridan say: You'll find examples of this sort of row level security' in SecDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file simulates such security for demonstration pourposes, using both the user's system login name or by logging in to the database file.   In the demos access to rows is on the basis of groups (or projects) to which one or more user can belong, but to restrict access on the basis of individuals each group could be a group of one of course.  Access to a row is via a many-to-many relationship type between the data table and a users table, modelled by a third table. When a new row is inserted it is a automatically assigned to the current user by means of code in the form's AfterInsert event procedure which inserts a row into this table.

To allow users to have read-only access to other users' rows would mean amending the forms so that rather than giving access to only those rows to which the user's group has permission, all rows would be returned.  You then have a couple of options:

1.  The AllowEdits and AllowDeletions properties of the form could be set to True in the form's Current event procedure to those rows where the user's group has permission and to False otherwise.  Setting the AllowEdits property to False denies access to other controls such as unbound navigational combo boxes, however.

2.  Instead of setting the form's AllowEdits property to True or False in the form's Current event procedure its code can loop through the form's Control collection and set the Locked and Enabled properties of selected bound controls, identified, by a common value in their Tag property, to Tue or False as appropriate.  This retains access full to other controls in the form.

Note the 'public health warning' in my demo files.  To call this sort of thing 'security' in Access is really an oxymoron as it can so easily be subverted by a moderately knowledgeable user.  All it really does is control access in the normal day-to-day operation of the database in its intended way.  If you want true security Access is not a suitable product.

henry huang- [o365] say:

Hi Philip,

Regarding Lync messages, are you referring to the saved entries in the conversation history folder?
If yes, an administrator is able to access users’ mailboxes and delete IMs by assigning the Full Access permission to himself. To grant the Full Access permission:
1. Go to Exchange admin center > recipients > mailboxes.
2. Double click a user and go to mailbox delegation.

After that, the admin can open this user’s mailbox by clicking the avatar on the upper right corner in Outlook Web App > Open another mailbox.

And the deleted IMs are able to be retrieved like emails. If they are soft deleted, you can find them in the Deleted Items folder in 30 days. If they are hard deleted or removed from the Deleted Items folder, it’s possible to get them back in 14 days by accessing the Recoverable Items folder: right click Deleted Items > Recover Deleted Items.
