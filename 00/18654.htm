'------ start of example code ------
If IsNull(Me.NumOrder) Then
     ' Shouldn't happen, but ignore it just in case.
Else
     CurrentDb.Execute _
         "UPDATE tblOrders SET OStatus = 'Billed' WHERE OrderNum = " & Me.NumOrder, _
        dbFailOnError
End If
'------ end of example code ------

If you really want to do it with a recordset, you can, but you have to either open the recordset on a query that returns only the record for the current order number, or else position the recordset first to the record for that order using the recordset's .FindFirst method. rob koch say:

Okay CodeVisio, haven't seen this question in a few months, so let's try to clear up the typical FUD and confusion propagated by those with no basic understanding of Microsoft's security products, nor those of third-parties and why there are simple differences that explain all of this.

First, there's nothing wrong here and the Windows Malicious Software Removal Tool (MSRT) is operating exactly as designed, you as most simply don't understand its true purpose or function.

Release information

"The Malicious Software Removal Tool is released on the second Tuesday of every month. Each release of the tool helps detect and remove current, prevalent malicious software. This malicious software includes viruses, worms, and Trojan horses. Microsoft uses several metrics to determine the prevalence of a malicious software family and the damage that can be associated with it."

Said another way, the MSRT is designed to detect only a limited (small) number of specific malware that Microsoft's criteria consider prevalent, so it inherently detects very few items.  The reason for this is obviously speed, since a deeper scan for all known malware takes significantly longer, partially due to the requirement that a much larger database of malware definitions be initially downloaded to the PC.

In addition, the normal scanning process for the Microsoft security product scan engines is to first perform a detection pass that finds individual items like registry keys/values and files associated with known malware.  That's what is being displayed by the MSRT Files Infected status during the scan in progress phase.

Near the end of this scanning phase, the Microsoft security products use their built-in cloud functions to contact the Microsoft Active Protection Service (MAPS) to verify both that the latest detection definitions are being used, as well as there are no false positive detections.  Once this phase is completed the final Scan results page is displayed with detailed results of any detections of the prevalent malware the MSRT is designed to find.

The inconsistent situation that exists with the MSRT tool is with its use for a "full" scan, since in terms of the tools true purpose, that's an oxymoron.  That's because the tool was initially designed as a quick download and scan tool for only a small number of prevalent threats, specifically for use during the monthly Windows Update process in order to verify that the currently installed anti-malware product(s) were performing effectively.  The MAPS process also allows Microsoft to collect the results of the scan, including any individual registry or file items detected, in order to determine what other malware might need to be added to MSRT detections in the future.

It's this last portion that creates the confusion, since though the MSRT engines include the capability to detect portions of most malware items, the specific malware families it displays in final results are only a small subset of the much larger number of detections the full Microsoft definitions database is capable of detecting.

So in truth, the problem here is the application of the MSRT tool for use in a full scan, since though all files and registry items are scanned, only this subset of final malware definitions will ever be displayed as detections within the scan results.

The proper tool for use in a manually performed on demand scan is mentioned in the second paragraph of the Summary section at the top of that MSRT page previously referenced.

"Microsoft generally releases MSRT monthly as part of Windows Update or as the standalone tool. Use this tool to find and remove specific prevalent threats and reverse the changes they have made (see covered threats). For comprehensive malware detection and removal, consider using Microsoft Safety Scanner."

However, since with Windows 10 you've already got both the Windows Defender security application as well as the ability to run its offline version, there's little need for any of these separate Microsoft tools since they the definitions used are the same and so these scans are mostly redundant.

The recommendation to optionally use the Malwarebytes Anti-Malware product instead is actually a reasonable one, but the idea that this scanner is "most powerful" is a flawed description of reality.  In truth, it's the fact that Malwarebytes' designers decided and declared a few years ago that they would become more aggressive in their product's ability to detect and optionally remove the potentially unwanted program (e.g. PUPs) category of typically less malicious software that makes this tool different from most antivirus products such as Windows Defender.

Malwarebytes gets tougher on PUPs - Malwarebytes Labs
