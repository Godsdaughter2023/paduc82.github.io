Folder

jordan dom say:

Hi David,

There are two ways we can use to create a new folder on your Windows 10 device. The easiest method to resolve your concern can be done by just pressing the key combination "Ctrl + Shift +N" either in File Explorer or just on your desktop screen. For the usual steps you can follow them below:

  1. Right-click inside File Explorer (or on the desktop) and choose the option labeled New.
  2. From the options shown select Folder.

When you choose Folder, shown here, a new folder quickly appears, waiting for you to type a new name.

Type a new name for the folder. When you begin typing, Windows quickly erases the old name and fills in the new name. Save the new name by either pressing Enter or clicking somewhere away from the name you’ve just typed.

If you want to change the name again, right-click the folder, choose Rename, and start over. Also, Take note of the following items below:

  • Certain symbols are banned from folder (and file) names, but you never have trouble when using plain old letters and numbers for names.
  • Different customer may remark that their right-click menu looks different than the one shown. There’s nothing wrong; programs often add their own items to the right-click menus, making the menu look different on different PC's.

After you have successfully created the new folder, all you need to do is drag and drop the folder onto the taskbar and it will be automatically pinned.

scott mcdaniel say:

The first thing you should do is test your application in a runtime environment. To do that, as Albert suggested:

1. Make a COPY of your current .accdb file

2. Rename that copy to have the file extension "accdr". As an example, if my Access file is named "Customers.accdb", I'd rename it to Customers.accdr

3. Now open and run the ACCDR file

Doing this forces Access to simulate Runtime mode, so you can thoroughly test your application with this method.

If you're getting an error about "too many forms", you may need to perform maintenance on your database before you deploy. Here's what I do before I deploy an Access database:

0. MAKE A COPY OF THE DATABASE!!!

1. Compact the Database. In 2016 you click File > Compact and Repair

2. Compile the database. To do that, open the VBA Editor and click debug-Compile. Fix any errors, and continue doing this until the menu item is disabled.

3. Compact again.

4. Create a compiled version. To do that, click File > Save As > Make ACCDE

Once you've done that, you can still test that file in the runtime mode, again by changing the ACCDE extension to ACCDR. If the ACCDR file runs correctly in your environment, it should function in your users environment.

Some go a step further, and actually create a new, blank database and import everything to that new database before going through the steps above. You can certainly do that, and it's a very good idea - but it's not absolutely necessary.

Note that any References you've made must also exist on the target user's machine. In my 2016 Access, the following 4 references are checked by default (and you should leave them there):

Visual Basic for Applications

Microsoft Access 16.0 Object Library

OLE Automation

Microsoft Office 16.0 Access database engine Objects

If you have ANY OTHER REFERENCES, then you must be sure those exist on the target machine. The Runtime will contain those 4, but may or may not contain others. If you can post a screenshot of your references we can review them for you.

Here are a few MSDN articles that might help you to better understand how to prepare for Runtime deployment:

https://msdn.microsoft.com/en-us/library/bb501030(v=office.12).aspx

albert d. kallal say:

Ok, use of the runtime should be “too” much of a jump or hassle.

The basic approach should be:

User downloads and installs the runtime.

Then they take a copy of your application placed say in “my documents” and then they double click on that applcatation and it should run.

The above is the process in a “nut shell”.

However, there are few things and details you want to test + try before you give the user(s) your cool and wonderful application.

The first tip I can give is you “test” how your application will run on YOUR machine AS RUNTIME! The simple way to do this is make a copy of your application, and rename the extension from accDB to accDR. Now when you launch the application it will run EXACTLY if you were a runtime user.

So the above "testing" on your machine is recommend.

The other tip is that the runtime “assumes” that you built the application for end users. That means those users don’t know much about Access and of course they don’t have a full working copy of Access (which exposes all the cool design tools and allows them to modify + create applications). Because the runtime system does NOT provide design and “much” of anything in regards to an interface, then you MUST provide start up forms and a user interface.

So keep in mind for example that the runtime does not have a navigation pane what this means is you the “developer” have to provide some start up form. That start up form can jump right into a form that edits data, but more preferable would be some kind of form that has buttons to launch the forms and reports the user requires.

So just keep in mind that a “bit” of extra work is required to use the runtime system. You can get real fancy, or just provide some plane Jane start up form with a few buttons that allows the user to launch the required forms (and possible reports) from some start up form. We often call such forms a switchboard, or whatever – but you the creator of the application have to provide the means to “move around” and launch the appropriate forms.

So runtime does not provide the user with the ability to “launch” forms – and users likely don’t know which ones to use anyway!

Another tip:
You should consider providing users with a “compiled” application. While this sounds complicated, it really just means you take your existing application and do a save & publish, and choose:

Make accDE.

The reason for the accDE is that any un-handled errors are ignored and when using the runtime un-handled errors in an accDB will cause the application to shut down, but as an accDE, it will NOT – the application will continue to run – even when errors are encounter.

However, ALWAYS before you try to use the "make accDE" option is to first while in code do a debug->compile to ensure no errors exist - if you don't do this, then your runtime version likely will not work.

So the runtime is not hard to use but the above outlines the several additional steps you want (need) to take when working with the runtime.

Also, you want to ensure that users install the correct runtime. There are “different” versions of office, but the main idea here is that two runtimes exist:

An x32 version (for use with x32 office)

An x64 version (for use with x64 office)

So make sure users are installing the correct version.

At the end of the day use of the runtime is not difficult and should not present too many hoops, but the above does outline some of the hoops you do have to jump through.

So, as a “test” first try running a copy of your application on your development computer – and try the application running with the extension accDR – this will “simulate” how your application will run as runtime.

If your application runs fine as accDR, then try deploying a accDE (compiled) version of your application to those user(s) with runtime only – it should work just fine if you tested and followed the above basic outline and steps.

Best of luck!
