albert d. kallal say:

You most certainly can call macros or even VBA DIRECTLY from the ribbon on action.

This gives an idea:

<customUI xmlns="http://schemas.microsoft.com/office/2009/07/customui">

  <ribbon startFromScratch="true">

   <tabs>

      <tab id="Home" label="Home">

         <group id="Foms" label="Open For Business">

            <button id="btnOpen" label="Open Form Products"

             size="large" supertip="Open form using built in Access OpenForm"

              onAction="=OpenForm('products')" />

         <button id="testMacro" label="Run macro1"

              size="large" supertip="runs the macro called Macro1"

              onAction="Macro1" />

         <button id="testVBA" label="Run VBA function"

              size="large" supertip="runs a VBA function called Test1 - function must be public"

              onAction="=Test1()" />

         </group>

    </tab>     

       </tabs>

  </ribbon>

</customUI>

Note in above how we can call VBA functions DIRECTLY from the Ribbon. This means you don’t need complex call-backs or anything.

In summary there are 3 ways to use the on action. Note that you can EVEN call built in docmd options in Access.

#1

Ribbon calling subs (call backs)

onAction = "Some VBA sub name"

The above is a ribbon call back, typically like this:

Public Sub MyRibbonLoadA(ByRef nribbonUI As office.IRibbonUI)

The above MUST be in a standard public module and NOTE how this is a SUB (not a fucntion – MUST be a sub). I in general recommaned you ADVOID call backs – they are a pain and in 99% of cases are NOT required.

#2   Access "expressions" (expression service)

Perfered: Call a VBA function directly.

OnAction = "=Name of VBA function()"

The above is perfered since the VBA code can be in the form that has the current focus “or” the VBA function can be placed in a standard code module. The beauty of this choice is that custom VBA code you call can be placed right into the form’s code module – so both buttons on the form, and as you move buttons to the ribbon then the SAME VBA routines in the SAME form can be called and used freely. This approach means that calling VBA routnines is really much like a button on a form calling simple VBA routines. Note that the above for you tec wiz people is the expression service syntax – so even built in docmand openform as above sample XML shows can be used.

#3

Next up: calling macros

OnAction = "Name of a macro"

This 3rd syntax allows your on-action to call a macro directly – and again without any call backs. Note the lack of “=” and the “() which is required for calling VBA functions.

So in general you don’t need call backs, in general the onAction can call your VBA code or run macros directly and with little hassle.

Unless a REAL pressing need arises, I STRONG suggest you avoid callbacks. Callbacks are only required for image loading, and also for the onLoad ribbon even to grab a ribbon handle – but for the most part you can have the onAction call your macros or VBA code directly without messy ribbon callbacks.

edit:

I would also try just "clicking" on your existing macros to see if they run. If they don't work or run by simply clicking on those macros, then they not going to work from the ribbon either. So your issue may well have nothing to do with the ribbon, but that security permissions don't allow running of macros.

2. I assume by default that the original copy is not updated / deleted / edited if the copy is?

3. I read an item saying that if the copy has the same columns / fields as the original, then these will be updated. I assume this means the sharepoint list data, not the content of the file? e.g. modified date... but then why would these fields be updated?

4. Is there a way to choose if the original document is updated or not if the copy is changed?

jerryduan_msft8021 say:

hi dave,

here are answers for your questions:
1. if the copy is sent to another document library, the permission of the copy is independent from the original file. the copy will inherit from the library where it is. however, you can manually manage the permission for this copy.

2. the original file will not be affected no matter how the copy is modified or the copy is deleted. however, if you edit the copy file and the original file, when you update the copy files, the changes made in the copy file will be wiped out. the copy will be consistent with the original file. to update the copy, we need to go the library where the original file is. in the file ribbon, click manage copies to manually update the copy.
     

3. yes. if the copy has the same columns as the original files such as modified by, created by, these columns of the copy file will be updated if the original file is updated because the copy shares the same column with the original file.

regards,
jerry
