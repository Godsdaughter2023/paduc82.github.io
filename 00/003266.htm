That may be a by-product of your page layout. For example, if you're using multiple page columns with a full-width header & footer, Word would only be able to represent that by creating Section breaks when the PDF is imported. That's because the PDF format knows nothing about headers, footers, footnotes, endnotes, etc. etc. It's little more than an electronic representation of what you'd see if you printed the PDF on paper. Which also explains why the round-tripping can't easily be as seamless as you'd like. You should be able to use Find/Replace to delete the unwanted Section breaks and the per-page headers & footers.

The ConvertFixedNotesToWordNotes macro is unaffected by Section breaks, though you'd need to at least  do something to differentiate the endnote references in the endnotes from those in the document body - something that could easily be done using Find/Replace to enclose them in square brackets, for example. Before running the macro, you would need to delete any endnote continuation separators that have become in-line shapes - but that should be a fairly simple task.

Addendum - Solution

With further searching I found that if I configured the limited user account wherein the problem appeared, as an administrator account - temporarily - and then ran a LanDesk Portal update and then reverted the account to limited user, the needed certificate configuration remain in place as required.

Another option I found was to go to security in internet options and open 'Custom Level', and within the list of options presented, find 'Don't prompt for client certificate selection when no certificate or only one certificate exists', and invoke the ENABLE radio button.

These two options worked and eliminated the pop up window, and the Websites in question began operating normally again.

Cheers, DaveJB.

cmdrkeene say:

That theme is still there in Windows 8 but it can't use hardware acceleration to boost performance. When using that theme, you're using 1980's GDI display rendering (GDI graphics are the #1 all-time cause of blue screen failures).  Plus your graphic card, a free supplemental processor, sits idle while the CPU has to struggle to do computing and graphics.

Every modern operating system (literally: even iOS, Android, Mac OS, and most Linux distributions) have started using GPU acceleration for graphics.  Microsoft did it in Vista wiStilth Aero glass but it used a ton of memory because every texture was separate.  Windows 7 tuned it better and it performed wonderfully.  You can still disable the transparency (if you want), but leaving the actual Aero engine will improve application performance because the GPU is handling all the rendering -- even for simple "2D" like scrolling a webpage or CSS code on a website -- will benefit from using the Aero engine (which is really WDM before the marketing team came up with "Aero").

Further reading: http://cmdrkeene.com/speed-up-your-computer-with-eye-candy

Another complication is that a great very many of the controls (buttons, checkbox, text, icons ... elements that make up the interface) did not exist 24 years ago when the Classic theme was made, so things tend to get shifted or display oddly, text doesn't align quite right.  But that's just nitpicky.

daniel g. grau say:
I tried adding "ExportPictureWithMetafile" as False and 0, both to Word's .plist and Office .plist, with no success.

[ For those in the know:
  • defaults write com.microsoft.office ExportPictureWithMetafile 0
  • defaults write com.microsoft.office ExportPictureWithMetafile False
  • defaults write com.microsoft.Word ExportPictureWithMetafile 0
  • defaults write com.microsoft.Word ExportPictureWithMetafile False
The string is known to Word, at it does appear in file "strman_0002.strings. Tried deleting it from there as well.]

Here's an emergency solution:

  • Make a copy of the .rtf file (not really needed unless it's your original file)
  • Enable Word > Preferences > General > Confirm conversions at Open
  • Open the file as Text only (not as RTF)
  • Select Find > Advanced Find and Replace
  • Enable "Wildcards"
  • Look for this 19-character long string: \{\\nonshppict*\}\}
  • Globally replace it by nothing
  • Save the file, close it (optionally disable "Confirm conversions...") and reopen it as RTF to check

If you need to do that a lot, the whole thing could be placed into a macro.

I'm sure some {nonshppict codes remained in the RTF. Please check the reduced RTF opening it as text and look for that sequence. If you find it, position the cursor there and search or the first } that follows. There will probably be extra text (like a return) right between the ending }}, preventing a match with the original search.

Alternative solutions:

  • Not too viable: Add an AutoOpen macro to the user's Normal.dotm, checking for an RTF format (it could just check for a name or partial name), then have it immediately Save As .doc.
  • You can have a dedicated Word running on the server. An AutoExec macro stored in Normal.dotm would periodically check for files in a given "Files ready" folder. Word would then open all files on the folder, Save As into  a "Converted Files" folder, then perhaps delete the originals. There is an Application.OnTime command that allows relaunching a macro at set intervals from 1 minute up, to prevent the loop from hogging the CPU.
