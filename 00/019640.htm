ken sheridan say:

I think you need one more 'junction' table.  With just the two tables modelling the relationship types you will fall into what is known as the 'the connection trap'.  To avoid this you need to '3-decompose' the relation by adding a table which models the relationship type between projects and employees.  

Starting from square one, firstly you model the relationship type between Projects and ResourceGroups with a table:

ProjectResourceGroups
....ProjectID (FK)
....GroupID  (FK)

The table is all-key, it's key being a composite one of both columns.

Similarly you model the relationship type between Employees and ResourceGroups with a table:

EmployeeResourceGroups
....EmployeeID (FK)
....GroupID  (FK)

Again the table is all-key.

Finally there is the relationship type between projects and employees, modelled by means of the following table:

ProjectEmployees
....ProjectID (FK)
....EmployeeID (FK)

However, this table must not be related to the above tables, only to Projects and Employees as over time employees and resource groups will change.  If you were to enforce relationships with the above tables you would be prevented from removing an employee from a resource group for instance.  The above tables provide the framework for the interface.  

When it comes to the interface this is a little tricky because of the need for correlation on the basis of multiple rows.  Normally you'd have a projects form, in single form view, and within it a resource groups subform in continuous forms view, the subform being based on ProjectResourceGroups and linked to the parent form on ProjectID.  This is straightforward; a combo box in the subform to select resource groups would have the following as its RowSource property:

SELECT GroupID, GroupName
FROM ResourceGroups
ORDER BY GroupName;

You'd also have an employee's subform in continuous forms view, the subform being based on ProjectEmployees and linked to the parent form on ProjectID.  This is a little trickier; a combo box in the subform to select an employee would have the following as its RowSource property:

SELECT DISTINCT EmployeeID, FirstName & " " & LastName
FROM EmployeeResourceGroups INNER JOIN Employees
ON EmployeeResourceGroups = Employees.EmployeeID
WHERE GroupID IN
    (SELECT GroupID
     FROM ProjectResourceGroups
     WHERE ProjectID = Forms!frmProjects.ProjectID)
ORDER BY LastName, FirstName;

Note that the employee name is stored in separate first and last name columns and concatenated into a single value in the query.

The combo box in the employees subform should be requeried when a row is inserted into the resource groups subform.

There is one more fly in the ointment, however.  Because the bound column of the combo box in the employee's subform is a surrogate key, if an employee is removed from a resource group, then any rows for that employee in the subform for existing projects will appear blank.  No data will be lost, merely hidden because there is no row in the combo box's list to which the value of the control can map.  The solution to this is to use a hybrid control in which the combo box is overlaid by a read-only bound text box to show the employee, leaving only the combo box's arrow visible.  This requires the subform to be based on a query which joins the ProjectEmployees  table to the Employees table so that the employee's name can be returned in a computed column which concatenates the first and last names.

PS:  I've assumed that the interface for assigning employees to resource groups has been set up independently of the projects form, so the EmployeeResourceGroups table will already be populated.

Hey, I'm sorry you're running into this problem. The names of sections do occasionally get confused and display the wrong name until you click on them. The easiest way that I know of to fix that problem is to do the following for any section you're having this problem with:

 

1. Rename the section to a name that isn't the name you see before OR after the click. In your case, don't rename the section to "Manage Your Business" or "August 2012".

 

2. Press Ctrl+S to save the new section name and then refresh the browser.

The StyleRef field (see http://office.microsoft.com/en-us/word-help/field-codes-styleref-field-HP005186193.aspx?CTT=5&origin=HA102110133) displays the text of the current (or most recent) paragraph in the referenced style.

suzanne s. barnhill say:
I suspected it would not work. Basically, the only running head you can have is one continued from the last previous non-notes section. So the best you can do is to have a running head that says "Endnotes" or "Notes" with the page number. To accomplish this, you can type the "Endnotes" or "Notes" heading in the last paragraph before the notes begin, then insert a page or section break before the heading. You can then either type the heading in the header, or, if you're already using a StyleRef field in your header to pick up chapter titles, then you can continue that header from the previous section and apply the same heading style to the Endnotes title.
