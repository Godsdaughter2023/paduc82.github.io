Sub DeleteNonBold()

Dim rCell As Range

   For Each rCell In Selection

    If rCell.Font.Bold = False Then
    rCell.ClearContents
    End If
    Next rCell
   

End Sub

Sub FilteringOutUnbold()
   Dim z As Long

There were four messages visible in Mailwasher, but after hitting Send/Receive in Outlook only two appeared initially. The other two showed up after I closed and restarted Outlook, and showed up about ten seconds after the Inbox was first loaded.

It's as if Outlook keeps new messages in a separate holding area while it tries to decide whether they are spam or not (these two messages are definitely not spam). I'd bet a paycheck that Outlook spawns a separate thread to evaluate each new incoming message, and the death of any one of these threads would leave its associated message un-handled while other messages are handled normally. marshall barton say:

The Load event runs after the form's data is loaded.  You should use the Open event instead.

 

Instead of trying to empty the temp table in the Close event, run the delete query right before running the append query.

 

Note that The Execute method never generates a message about about records that were or can't be added so your use of Set Warnings has no effect.

lakesidemom say:

Private Sub Form_Open(Cancel As Integer)

tom van stiphout say:

I think you misunderstand the error message. It says that tableX already exists. No kidding, in Execute1 you were able to delete THE RECORDS from tableX, so obviously the table existed. A delete statement does not delete the table itself (a table is a container of records).

Then on he next line I presume the maketable query is trying to make a new table by that same name, and the table already exists.

 

A much better idea would be to change the MakeTable query to an Append query. This type of query adds (appends) records to an existing table.

lakesidemom say: I designated a primary key in the table and the append query now works perfectly. Thank you for the suggestion, Tom!

tom van stiphout say:
Great!. One more point for the Primary Key. Make sure you have one in every table. It's one of the pillars of good database design.

marshall barton say: Whoa!  You did not have a delete table query, you had a delete the records in the table query.  Deleting the records leaves the table intact, but empty and ready for the append (not make table) query to add a new set of records.  Put the Delete query back and you will not have duplicated record problems and will not have to do anything about the primary keys.
