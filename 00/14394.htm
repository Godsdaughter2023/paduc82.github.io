More often than not, that is a message that is encountered when the current view for the folder encountering the issue has been corrupted.

There are two things you can try to see if it resolved the issue:

Using the folder where the message is encountered

 

Option 1 - Reset view for one folder

#1 - Select the folder via the navigation pane (error will be displayed)

#2 - From Outlook menu --> Views --> Current View --> Customize Current View --> Reset Current View

 

Option 2 - Reset All views for all folders

#1 - Close Outlook

#2 - Re-start Outlook with the following commandline switch (From Start /Programs)
           Outlook.exe /cleanviews

(be sure to have a space between Outlook.exe and /cleanviews

#1 - If this involves a "default" folder then you will not be able to rename or delete it but there is no such restriction to any sub-folders created underneath a default folder - leastwise not if these were manually created. You may want to run ScanPST on this PST file for good measure but experience has been that this seldom has rectified a "folder Full" issue.

Where is Scanpst.exe?

http://www.outlook-tips.net/beginner/scanpst.htm

#1 - Does this folder use a custom form?

#2 - If no custom form in use - are there UDF's created at the folder level - or more specifically, UDF's that have been included in a view?

#3 - Have you changed the settings to the view that was in use in terms of fields displayed/grouping/sort etc

#4 - What is the size of this folder in terms of # of contact items?

#5 - How were the sub-folders created - manually or via the Outlook PST2 PST import process?

#6 - Do you sync with anything that involves these folders?

a. user say:

You may have seen the "aside" Rich and I had concerning controlling this dialog via Applescript.

This message shows how that can be done. In my view, to run the Applescript manually would probably be about as difficult as going into the Page Setup... dialog for each document and choosing the paper size (and printer, if necessary). To do it without intervention, as far I know you would have to have a Word VBA AutoOpen macro that ran the Applescript. Making that work has proved quite difficult.

So this is going to be a long message, with plenty of steps. If you (or anyone else reading this) can make your way through it and either confirm or deny that it works on your system, please post back. I can only easily work with Snow Leopard and Mavericks here so I can't check how things work on Lion/Mountain Lion. FInally, if there is asimple way to avoid the problems described in here, please let us all know!

The summary is that

 a. the Applescript uses Apple's System Events to drive the user interface programmatically.

 b. That requires that you enable "Access for assistive devices" in Mac OS X Preferences. This works differently in Snow Leopard and Mavericks.
c. Running anything that tries to make use of "access for assistive devices" seems to fail when run using VBA.Macscript(), no matter how you do it.

d. The only way I have found to run the script from Word VBA is to use the VBA.Shell command to run it.

e. In order to do that, you have to save the Applescript as a .app. Doing that is also different in Snow Leopard and Mavericks.

Running the Applescript

First, here is the Applescript. It assumes that Word is open, with a document open. If the VB Editor is open, the script has to switch to proper Word to bring up the Page Setup dialog. The only way I found that worked was to close the VBE Editor.

To run this script, you will need to do the following:
 1. copy the text of the script and save it as a .applescript file (just a text file). Let's assume that your username is xyz, that your root user folder is at Macintosh HD:Users:xyz . Let's assume you save the file as Macintosh HD:Users:xyz:Applescripts:WordPageSetup1.applescript

 2. find and run the Applescript editor (typically in Applications/Utilities)

 3. open the .applescript file

 4. compile it and deal with any errors

 5. I'd then save it as a .scpt in the same folder. To do that in the Snow Leopard version of the Applescript editor, you use the File->Save As option, then specify the format you want. IN Mavericks, you will probably need to use the File->Export... menu to achieve the same thing.
 6. Modify the values of the properties at the top as appropriate for your system. You will at least need the name of your printer (I'd copy it from the File Print dialog box) and the name of the paper size.

 7. save again.

 8. with Word open with a document open, try running the applescript. Unless you ave already enabled access for assistive devices, you will probably see an error. To fix that on Snow Leopard, go to System Preferences->Universal Access and check Enable access for assistive devices . On Mavericks, you need to enable access for specific programs. In this case, it should be enough to enable the Applescript Editor. If you need to do that, go to System Preferences->Security and Privacy->Accessibility and see if the Applescript Editor is listed. If it is, check its checkbox (you may have to unlock the padlock first). If it isn't there, find it in the finder and drag it into the list, then check its box.

 9. At this point, you should be able to run the script from the applescript editor and see if actually does what you need. If it doesn't, maybe you will be able to debug it until it does.

Running the script from VBA.

As mentioned before, I have been unable to get anything that uses the support for assistive devices to work when run using VBA.Macscript. The script has no problem invoking System Events, and can get infromation about processes. But it does not seem able to "see" any "UI elements". I have read many articles about this and the main problem in Mavericks seems to be that after you have enabled an app to use asssitive devices, if you then modify the app, Mac OS X will disable the app again. Because many applescripts self-modify, this is a common problem, but other apps may also self modify. I have seen various possible workarounds, including changing the ownership of (say) the Microsoft Word.app, but so far, none of them have actually made any difference.

The only way I have been able to run the Applescript from VBA is to use VBA.Shell instead.

To do that, you first have to save the Applescript as a .app using Save As/Export from the Applescript Editor. On Mavericks, you will also have to drag the App into the System Preferences->Security and Privacy->Accessibility dialog and check its box. What's more, you will probably need to uncheck it/re-check it every time you modify the app, unless you sign the app's code (another story).

Then, you might hope that you could us a VBA Sub like this to invoke the app:

Sub AutoOpen()
VBA.Shell "Macintosh HD:Users:xyz:Applescripts:WordPageSetup1.app"
End Sub

But there is a final twist. The .app is a "package". If you right-click on it in the Finder you should see an option "Show Package Contents". Clicking that should reveal another subfile. We actually have to sspecify one of those subfiles. By default we use

Sub AutoOpen()
VBA.Shell "Macintosh HD:Users:xyz:Applescripts:WordPageSetup1.app:Contents:MacOS:applet"
End Sub


If you only need this for one document, you should put that AutoOPen sub in that document and save it as a .docm. 

If you put it in the Normal.dotm it will affect everything, which you presumably do not want. If you want to use the same settings for a number of files, you will probably be better off creating a template (.dotm), create your files from that, and have an AutoOpen and AutoNew macro to set the printer/paper characteristics.
