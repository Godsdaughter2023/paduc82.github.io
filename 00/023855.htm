Some housekeeping items..............

#1 - Terminoligy - For sake of clarity and accuracy, in Outlook, contact items are stored in a contact "folder".

#2 - The Outlook AddressBook is a specialized "view" of the contact "folders" set to be members of the AddressBook.

#3 - What you have set to "save" in terms of contacts is in reference to the "auto-complete cache" which is used to populate when you start to enter items in the To/CC/Bcc address boxes but this is completely separate from any contact folder, These entries are/cannot be displayed in the Outlook AddressBook. If you want email addresses from an email message to saved to the default contact folder for the Outlook profile, you need to right-click the email address in the email message
--> click on "Add to Outlook contacts"

Also, any recipient of an email message sent gets added to the Auto-Complete cache.

#4 - The fact that the contact folder has the added words "This Computer Only" means that the account is configured as an IMAP email account. Just as an FYI, contact/calendar items do not sync with the server.

So with the above, to address your specific question/issue along with a recommendation

#5 - As already noted, by right-clicking on sender email addresses on messages rec'd, contacts will appear in the default contact folder which in your case appears to be the one labeled as "This Computer Only". The default contact folder is stored in the "default data file" set for the profile. You can see which data file is set as the default by going to Outlook menu --> File --> Account Settings --> Data Files tab (if there is only one data file in the profile, it will always be set as the default.

#6 - It is not recommended storing contacts in a "This Computer Only" folder. If the IMAP account is removed - the underlying .ost data file for the account also gets deleted resulting in all data in "This Computer Only" folders being lost. Also, if Outlook can't find the .ost file for any reason, the ost file gets recreated. The email folders will then be automatically sync'd with the server but the "This Computer Only" folders will be empty. You would need to explicitly and manually backup (export) "This Computer Only" folders to ensure you do not lose that data.

A better more permanent approach to protecting data contact/calendar in "this computer only folders is to do the following

a) create a new PST file in the profile. Done by clicking on <New Items> (in the Outlook ribbon) --> More Items --> Outlook Data file
b) go to File --> Account Settings --> Data FIles tab --> select the newly created PST file --> set it as the default data file

From this point forward, any email addresses added to Outlook contact folder will appear in this folder and since it's the default data file, it will be part of the Outlook AddressBook which, in turn allows you to create your Distribution List (Contact Group).

Just to keep the Outlook AddressBook clean, would suggest removing the contact folder from the AB by right-clicking on the "This Computer Only" contact folder --> select <Properties> --> select <Outlook AddressBook> tab --> uncleck the box for "Show this folder as an email address book"

The following demonstrates how to use Goal Seek to determine B4, the monthly payment.



Explanation....

Row 9 demonstrates the formulas for a normal loan.  B9 is the next monthly date.  C9 is the regular payment.  D9 is the monthly interest:  the previous balance (E8) times the monthly interest rate, B2/12.  E9 is the new balance:  the previous balance (E8) plus interest (D9) minus the payment (C9).

D8 demonstrates the special interest calculation for the first payment.  Normally, the first payment includes interest for one month after the normal funding date, E7*B2/12.  In this case, the actual funding date is less than one month (namely, it is one day) before the normal funding date.  So the additional interest is loan amount (E7) times the additional days, EDATE(B8,-1)-B5, times the daily interest rate, B2/365.

Note:  The calculation of monthly and daily interest rates depends on how the annual interest rate is specified for your region.  The calculations shown are based on "US" rules, which actually applies to some other countries.  For the EU and GB, the monthly rate can be calculated using RATE(12,0,-1,1+$B$2), and the daily rate can be calculated using RATE(365,0,-1,1+$B$2) [1].  The calculation for Canada is very special; let me know if you need the formulas.  Some other countries might have different rules that I am not familiar with.

Also note that the monthly payment in B4 is not rounded.  This is necessary for Goal Seek and Solver.  You can round the derived payment manually or use the ROUND function; but only after using Goal Seek or Solver.  However, you will also need to adjust the last payment in B67.

In any case, it is prudent not to round the interest and balance calculations.  Lenders have different rules for handling fractional amounts, anyway.

Consequently, the final balance in E67 is usually not exactly zero.  It might be infinitesimally more or less than zero.  Again, you can round the final balance using the ROUND function; but only after using Goal Seek or Solver.


-----
[1] For the EU and GB, I prefer to use the following formulas to calculate monthly and daily rates:
monthly:  (1+$B$2)^(1/12) - 1
daily:       (1+$B$2)^(1/365) - 1
The RATE formula should return about the same values.  But in different circumstances, sometimes the RATE function returns a #NUM error, requiring us to enter a "guess" parameter.

[2] Even "payment with uneven cash flows" is vague ambiguous.  Often, it connotes unequal payments that are regularly or irregularly-spaced.  If that is what you meant, the paradigm might be adapatable.  But again, if you need help with that, please provide a concrete example.

I chose to interpret "payment with uneven cash flows" to mean that you wanted equal payments, and only the first payment is irregularly-spaced.  That is consistent with your first (and only :-<) example.

That was my first thought, as well.  But then I realized that is wrong.

In actual practice (e.g. mortgage loan), the interest for any such abnormal delay between funding and first payment is simply added to the closing costs.  Alternatively, it might be added to the first regular payment, making the first payment an irregular amount.

We cannot simply add the odd-period interest to the loan principal.  That would mean we are charging interest on the interest.  And it would mean the lender does not receive the full odd period interest by the time of the first payment.  To see this, consider the interest calculation for the first payment (r = periodic interest rate):

(loan + oddInt)*r
= loan*r + oddInt*r

financial engineer say: Let me state that t-Value is using actual number of days in it's calculation of payment thus working more like a Excel XPMT function rather than a Excel PMT function

With that in mind using Excel you would get slightly different payments for both instances once where payments begin at 09/15/2013 and once where payments begin at 10/01/2013

This is so as t-Value is using actual number of days between payments

Now for the second part

Payments Begin at 10/01/2013

For this I am going to have to explain to you how Excel makes use of a TVM equation to find the payment

Excel uses a sum of initial investment, discounted annuity payments and a discounted terminal value as follows

PV + PMT (1+RATE*type) [ 1 - {(1+RATE)^-NPER}] / RATE + FV(1+RATE)^-NPER = 0

This equation applies when payments are begin either at time period 0 (in this example at 08/15/2013) or at time period 1 (in this example at 09/15/2013)

As we are dealing with payments at time period 1.5 thus you would have discount the periodic payment and the terminal value for an extra t=0.5

Thus our TVM equation now looks like this

PV + PMT (1+RATE*type) (1+RATE)^-0.5 [ 1 - {(1+RATE)^-NPER}] / RATE + FV(1+RATE)^-0.5(1+RATE)^-NPER  = 0

This all looks complicate so let me use financial notations to rewrite this equation as follows

PV + PMT (1+RATE*0) PVIF(RATE, 0.5) PVIFA(RATE, NPER) + FV PVIF(RATE, NPER+0.5)  = 0

Solving for PMT makes the fomula as follows

PMT = [-FV * PVIF(RATE, NPER+0.5) - PV ] / [PVIF(RATE, 0.5) PVIFA(RATE, NPER)]

In Excel we can use PV formula as follows to find PVIF

PVIF(RATE, NPER) = PV(RATE, NPER, 0, -1, 0)

And in Excel we can use PV formula as follows to find PVIFA

PVIFA(RATE, NPER) = PV(RATE, NPER, -1, 0, 0)

So using the following values as input

RATE = 6.25%/12
NPER = 60
PV = -100000
FV = 12000
TYPE = 0

You would have to create your own PMT formula as follows

= (-12000 * PV(RATE, NPER+0.5, 0, -1, 0) + 100000 ) / (PV(RATE, 0.5, 0, -1, 0) * PV(RATE, NPER, -1, 0, 0))

This will give a payment of $1779.09 as compared to the payment of $1,780.25 given by t-Value

The slightly higher payment in t-Value is attributed to the fact that it uses actual number of days in it's calculation

doug robbins - mvp office apps & services (word) say:
The bit-ness of the language pack must match that of the Office installation.  Therefore, if you have 32 bit Office, which is the recommended version unless you need the 64bit version because you are running very large, very complex Excel models, it is the X86 version of the language pack that you should download and install.
