teylyn say:

Hello,

Excel expects the exact password that was entered when the sheet was protected. This is the dialog that comes up when you want to protect a sheet: 

If you want to use a password in order to remove the sheet protection, you can enter the password here. You decide what that password is, but it has to be just one word, no spaces. It can contain numbers and symbols.

You will then be prompted to re-enter the password and then the locked cells on the sheet cannot be changed. 

If you or someone else then unprotects the sheet, this dialog shows up:

It only accepts the password that you set when you protected the sheet.

Note:

  • you don't have to type a password when you protect a sheet
  • sheet passwords are not a security tool, because they are easy to remove
  • sheet protection with passwords is helpful to prevent inadvertent changes, but it is not powerful enough to prevent malicious intent.

davidf2 say:

No, you can only use workarounds.

One possible workaround is to do a Save As and make a copy of your original file. Then break the text flow from the first page and Ctrl+A in the first text box to select it all and delete. Now you have a 90 page publication with empty text boxes that you can copy and paste from the original publication. Tedious I know but it will get you there. I also would suggest that you go to View > check Boundaries or else you won't be able to see the empty text box boundaries unless you mouseover them.

Here is another possible workaround if you are breaking up the 90 pages into chapters or some such and you want some pages to flow together. Once again make your second copy of your file with empty text boxes on 90 pages. This time make a third publication with a blank custom page that is 90 X the length of your pages in your original publication. If it is a 11 inch page make a custom page of 990 or about 1,000 inches long. Create one 1000 inch long text box, go back to your original publication, click in the first paragraph and Ctrl+A and copy to select all the content that is flowed throughout the original 99 page publication. Go to the long text box, paste special, formatted text. Now you can see all the content on one single page and select multiple inches/pages of text...an entire chapter, copy and then paste into the appropriate text box and page of your blank 90 page document, link the overflow to the next text box, page etc. to form flowed chapters.

Perhaps someone else can come up with a better alternative. What you are finding is that Word is a better choice for books and such. In fact you could just Edit Story In Word but you are likely to loose your formatting.

j_i_m_s_t_e_r say:

Take the back of the phone off.

 

The window in the back, which has the cover for the lens, has a rim around it.  Part of this rim is raised up - this is the contact to enable the phone to see the memory card.


What happens over time is that the case on the phone may become compressed and the contact gets flattened down.

 

What you need to do is slide a thin flat blade scredriver under the raised part, and carefully twist the blade to raise the contact upwards, very slightly (~1mm).

 

Put the back of the case back on the phone.

 

Worked fine for me!

ken sheridan say: Do not use the lookup field wizard.  For reasons why see:

http://www.mvps.org/access/lookupfields.htm

Also it won't work with your Duration column.  For that create a table durations with a single column, Duration, designated as the primary key, of integer number data type and insert rows with the values  15, 30, 45, 60, 90  and 120.  In the referencing table include a column Duration of integer number data type and in the data entry form use a combo box as the control bound to this column, with a RowSource property of:

SELECT Duration FROM Durations ORDER BY Duration;

Set its LimitToList property to False (No).  Do not create an enforced relationship between the referencing table and the referenced table.

For employees create a table Employees with columns EmployeeID (automber primary key), FirstName, LastName etc.  In the referencing table include a foreign key column EmployeeID of long integer number data type.  In the data entry form use a combo box bound to the EmployeeID column set up as follows:

RowSource:     SELECT EmployeeID, FirstName & " " & LastName FROM Employees ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.

To enable you to enter a new employee in the format FirstName LastName in the combo box you'll need to put code in the combo box's NotInList event procedure.  You'll find an example as NotInList.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

In this little demo file's opening form you can enter a new name in the above format, which will open a form in which the other data concerning the new person can be inserted.  Note that the other form, frmContacts in my demo, also needs code in its Open event procedure to assign the new first and last names as the DefaultValue properties of the relevant control in the form.

The demo also illustrates the use of the NotInList event procedure in other contexts.
