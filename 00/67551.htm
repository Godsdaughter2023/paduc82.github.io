donald milne (dmilne) say:

Having recently done programming work developing a shell extension on Windows 10, I can tell you that this is how the recycle in behaves, it is not something weird about the OPs machine.

I think the clue is in the mention of TeraCopy. But, if you want to reproduce it using only Microsoft code then here is a way:

1. Delete a file (i.e. cause it to be moved to the recycle bin.

2. Open a zip file using the windows explorer. I.e. using the "Compressed Folder Tools" shell extension.

3. Drag the deleted file from the Recycle Bin to the still open Zip folder.

4. Observe the strange "$Rxxxxxx" filename now given to the file in the zip folder.

The explanation seems to be that Windows assigns a temporary name to the file inside the Recycle Bin meta folder. If you restore this file or drag it to a shell folder then some undocumented magic happens behind the scene to restore the original filename. In some cases (e.g. the Zip tool) this magic trick has been overlooked.

It is also overlooked in all third party tools (including my own shell extension), probably because this behaviour seems to be undocumented. In fact I tripped over this thread while looking for some documentation.

donald milne (dmilne) say: I did some investigating and I've pinpointed exactly why it goes wrong. When a file is dragged from the recycle bin the clipboard also includes a "FileNameMapW" object which tells the target to rename the files as it copies them. My own shell extension wasn't doing the rename, and presumably neither is the Win10 zip tool, nor TeraCopy.

From your screenshot, I am assuming that you are trying to set up a 5.1 Surround sound system. I suggest trying the methods listed below.

Since the issue started when an Windows Update performed. Try to rollback the Realtek driver first and check if it resolves your issue:

If your PC has a Restore Point. Use it to take your PC back to an earlier point in time:

1. Right-click (or press and hold) the Start button, and then select Control Panel.

2. Search Control Panel for Recovery.

3. Select Recovery > Open System Restore > Next.

4. Choose the restore point related to the problematic app, driver, or update, and then select Next > Finish.

Note

If you’re not seeing any restore points, it might be because system protection isn’t turned on. To check, go to the Control Panel, search for Recovery, and then select Recovery > Configure System Restore > Configure and make sure Turn on system protection is selected.

Method 3. Update drivers im Windows 10.

Update drivers in Windows 10
  • Driver updates for Windows 10 and many devices (such as monitors, printers, and video cards) are automatically downloaded and installed through Windows Update. 
  • While it's likely you already have the most recent driver, if you're having trouble with a device, you can try fixing it by updating the driver or reinstalling it. Here's how:

    3. Select Search automatically for updated driver software.
    4. If Windows doesn't find a new driver, you can try looking for one on the device manufacturer's website and follow their instructions.
  • If these steps don't work, try reinstalling the driver: Open Device Manager, right-click (or tap and hold) the name of the device, and select Uninstall. Restart your PC and Windows will attempt to reinstall the driver.

lemp say:
Frankly, if you have too many Event Viewer errors to mention -- and you don't have a Windows install CD -- you probably will be best served by taking your computer to a reputable local professional (not a BigBoxStoreUSA or geeksquad type of place).

However, you may want to see if the following steps help:
  • If you have Spyware Doctor installed, uninstall it.
  • If you have ZoneAlarm installed, open it, click the "Overview" tab, then select "Preferences", and UNcheck the "Protect ZA Client" check box.

Click Start, then Run, type cmd in the Open box and click "Ok". At the prompt in the Command window, type the following commands, pressing "Enter" after each one: (Please note the spaces in the following commands.)

  • chkntfs /d ..... (This will reset autocheck options to default)
  • chkntfs /c C: ..... (This will allow checking drive C:)
  • fsutil dirty query c: ..... (This queries the drive and will most likely tell you that it is dirty)
  • chkntfs /x c: ..... (The x switch tells Windows to NOT check the specified drive on the next boot)

At this point, restart your computer, it will not do a chkdsk and will boot directly to Windows.

This next step is important as this is where the Dirty Bit will be unset.
Click Start, then Run, type cmd in the Open box and click "Ok". At the command prompt, type the following, pressing "Enter" after each one: (Again, note the spaces.)

  • chkdsk /f /r c: ..... (To manually run a full chkdsk operation)
  • Y ..... (To accept having it run on the next boot)

This should take you through 5 stages of the scan and will unset that Dirty Bit. Be patient...this is a very thorough check and will take quite a while...it may even appear to start over during the final two stages...60 to 90 minutes would not be unreasonable for this to complete in some cases.

Finally, when the chkdsk operation has completed, type fsutil dirty query c:, press "Enter", and Windows will confirm that the Dirty Bit is not set on that drive.

Now reboot again and see if chkdsk still runs on bootup.

If chkdsk still runs when you reboot, you'll have to create a bootable CD with the Windows Recovery Console and run chkdsk from there:

Follow the directions here: http://www.bleepingcomputer.com/forums/topic270018.html (scroll down to "How to Create a Bootable XP Recovery Console CD")

Once you boot to the Recover Console, type

    chkdsk c: /r


and press Enter
