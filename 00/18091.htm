rob koch say:

Hi Ricardo,

Looking at the situation I can see multiple reasons that these files may be giving problems, which I'll try to explain here.

First, the proliferation of malware has forced Microsoft to become very aggressive in the ways they handle even non-malicious files, especially executables and packed files.  This has not only created problems for malware purveyors, but also otherwise valid software developers, especially those smaller developers who provide programs to much more limited audiences that are rarely found installed on a PC.

To deal with this problem, Microsoft began recommending and later requiring that software developers digitally sign their software using a certificate provided by a member of the Microsoft Trusted Root Certificate Program.  This not only allows for the creation of a digital signature to insure the integrity of a file, but also provides the ability to identify the true source of any program or file thus signed, since the developer must provide this information when acquiring the certificate initially from the Trusted Root Certificate vendor.

This Authenticode system of identification and file integrity confirmation has become more integrated into the antimalware and newer Windows operating system versions as time has gone on.  This includes not only security applications like Defender, but also the SmartScreen Filter and other file reputation systems Microsoft has integrated into Windows and Internet Explorer.  As Microsoft has moved to using their cloud based automation systems to quickly identify and deal with malware, the importance of digital signatures and the use of certificates has increased rapidly, especially in the Defender version now included in Windows 10.

With the above as a base, here are the details for your application I see as a problem.

The file distribution appears to be a packed file format type of "rar", which isn't commonly used with Windows today since there's no ability included in Windows itself to unpack this proprietary format.  Though unpackers are available via the Windows Store (I checked for my Windows 8.1), these programs themselves are obviously different and so may provide differing amounts of information about the files contained within a specific rar package.

Since Microsoft typically uses the unpacker application currently installed and associated on a specific system to unpack and analyze a file of that format, this can lead to differing results between individual unpacker apps and systems, fragmenting the ability to consistently detect and verify specific files even further.

The lack of a rar unpacker app on my own Windows 8.1 system my be part of the reason I received no warnings myself when I downloaded and examined the latest version of your installation package from the website URL you provided above.

If the executable file(s) contained within the specific archive (e.g. rar) file aren't themselves digitally signed, there's no way for the Microsoft security applications to verify the integrity of the files or the identity of the developer who created them.  This means that the files are inherently untrusted and so there's no way they can be "whitelisted", so Microsoft hasn't done this for their security applications for several years now, as the first item on the following developer FAQ page indicates.

Microsoft Malware Protection Center - Developer FAQs

So your problem appears to me to relate to at least portions of the above, due to the rar archive distribution and possibly a lack of digital signing, though I can't confirm this without acquiring a rar unpacker and at least examining the files contained within if not installing the application, none of which I intend to do.

As an aside, there is currently a relatively common piece of malware called MPC Cleaner, which though clearly not related to your own app, does sound startlingly similar to the name you chose.  Though this shouldn't be directly causing your specific problems, this should show you how difficult it makes such issues for Microsoft, since they have no real way to differentiate anything you've created from such a similar named program without the proper use of a digital signature.

Otherwise, Microsoft has no way to truly identify you personally or trust a thing you say, any more than a typical consumer can insure the identity of some random caller trying to scam them via phone or pop-up message.

So in order to protect their customers, Microsoft has become less responsive over time to developers who aren't following these now standard best practices regarding Authenticode signing of their software.  I can't say exactly what your results would be if you managed to upload your file or talk to someone at Microsoft, but I suspect the response might contain something similar to what I've described above.

I think you're not that far off in your process, you just need to update the inflight tasks before using the "Reschedule" command. 

If tasks are complete - record the Actual Start and Actual Finish.  Project will update the % complete to 100%.

If the tasks are started but not complete, set the Actual Start date, as you have been.  Then record the "Actual Duration" - how long have they worked on it?  Project will calculate the remaining duration based upon the original scheduled duration minus the Actual.  Confirm the remaining duration (it may be more or less) with the performing resource.  Update as needed.  Make sure you capture progress on all inflight tasks. 

After all updates, then use the "Reschedule uncompleted work" to move any remaining work (and duration) to the new start date.

Regarding Duration and Actual Duration.  Before updating tasks, you should save a baseline to capture your "plan".  Saving a baseline copies the following values: Start into Baseline Start, Finish into Baseline Finish, Work into Baseline Work, Duration into Baseline Duration, and Cost into Baseline Cost.  The baseline values are not changed unless you manually change them or save a new baseline.  As you update tasks with Actual Start, Actual and Remaining Duration - Project calculates the variance between the Actual and Baseline Start, Actual and Baseline Finish, Actual and Baseline duration, etc.  This allows the user to view how accurate their plan was and take action if needed to meet the schedule commitments they may have made.

I hope this helps.

Test1, Test2, Test3 and Test4. Each checkbox has the macros below set as entry and exit macros.  Here is a template containing the macros and sample checkboxes that act like radio buttons. I've tested it in Word 2016 for Mac.

Sub CheckEntryMacro()
  CheckBoxName$ = Selection.FormFields(1).Name
  If OldCheckBoxName$ <> "" Then
    OldCheckBoxLetters$ = Left(OldCheckBoxName$, 2)
    CheckBoxLetters$ = Left(CheckBoxName$, 2)
    If BoxFilledb = False And OldCheckBoxLetters$ <> CheckBoxLetters$ Then
      ActiveDocument.Bookmarks(OldCheckBoxName$).Select
      MsgBox "You must check off a selection in this row."
    End If
  End If
  CheckBoxResult$ = Selection.FormFields(1).Result
  If CheckBoxResult$ = 1 Then
    CheckExitMacro
  End If
End Sub

Sub CheckExitMacro()
  Dim CheckBoxLetters$, CheckBoxNumber$
  Dim Box1 As Variant, Box2 As Variant, Box3 As Variant, Box4 As Variant, Box5 As Variant, CurrentBox As Variant
  Dim Box1b As Boolean, Box2b As Boolean, Box3b As Boolean, Box4b As Boolean, Box5b As Boolean

  CheckBoxName$ = Selection.FormFields(1).Name
  CheckBoxLetters$ = Left(CheckBoxName$, (Len(CheckBoxName$) - 1))
  CheckBoxNumber$ = Right(CheckBoxName$, 1)
  Box1 = CheckBoxLetters$ + "0"
  Box2 = CheckBoxLetters$ + "1"
  Box3 = CheckBoxLetters$ + "2"
  Box4 = CheckBoxLetters$ + "3"
  Box5 = CheckBoxLetters$ + "4"
  Box1b = False
  Box2b = False
  Box3b = False
  Box4b = False
