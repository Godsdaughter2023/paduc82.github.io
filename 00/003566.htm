ken sheridan say: If I understand you correctly, what you want to do is:

1.  Enter two dates in text boxes in a form.
2.  Click on a button in the form.
3.  Which then opens your report in print preview filtered to the data range.

Right?  If so then your 'select report form' does not need a RecordSource property at all; it should be an unbound form.  The two text boxes should be unbound, i.e. have empty ControlSource properties.  The report can be filtered by code in the button's Click event procedure as follows:

Const MESSAGETEXT = "Both dates must be entered."
Dim strCriteria As String

If IsNull(Me.txtStartDate) Or IsNull(Me.txtEndDate) Then
    MsgBox MESSAGETEXT, vbExclamation, "Invalid Operation"
Else
    strCriteria = "Date_Occurred >= #" & Format(Me.txtStartDate,"yyyy-mm-dd") & _
        "# And Date_Occurred < #" & Format(Me.txtStartDate,"yyyy-mm-dd") & "#+1"

    DoCmd.OpenReport "YourReportNameGoesHere", _
        View:=acViewPreview, _
        WhereCondition:=strCriteria
End If

A few explanatory points:

1.  The code first checks to see that both dates have been entered (neither is Null) and if not warns the user.

2.  The date range is defined as on or later than the start date, and before the day following the end date.  This is more reliable than a BETWEEN....AND operation as it allows for dates with non-zero times of day, which can easily creep in without you being area of it unless you've made specific provision in the table definition to prevent such values.  A BETWEEN....AND operation would miss any dates on the last day of the range if they have a non-zero time of day element.

3.  When building the criterion to filter the report the dates are formatted using the ISO standard for date notation of YYYY-MM-DD.  This is internationally unambiguous and allows for the dates being entered in the text boxes in any regional format.

If you are unfamiliar with entering code in event procedures, this is how it's done:

1.  Select the button and open its properties sheet if it's not already open.

2.  Select the On Click event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new lines between these.

Restrict footers to just one page within a document

I have Windows 7 and I'm using Office Standard 2007 (and no, I don't intend to "buy" any of the new versions) and I'd like to know, in Word, if there is any way to restrict a footer to simply just one page within a document.  My headers must remain as Different First Page and Different Odd & Even Pages but the footer needs to be with specific pages.

Help will be appreciated.

stefan blom say:

If you want a specific footer for a specific page number, you can make use of an IF field construct in the footer.

For example, to display the text "this is my special footer" on page 7, add the following field to the main body footer:

{ IF { PAGE } = 7 "this is my special footer" }

To add each pair of field delimiters, { }, press Ctrl+F9 (on some keyboards, you have to press Ctrl+FN+F9). Note that you can't type these from the keyboard.

macropod say:
There are numerous ways of achieving unique footers. You can use field coding, as Stefan suggested or, if the footer is to replicate content in the document body, you might apply a unique Style name to that content, using that style name on every page, then reference it via a STYLEREF field in the footer. Employing Section breaks is the least desirable method, as it compromises your ability to ensure edits to the content before them don't muck up the relationship between the text and footer.
