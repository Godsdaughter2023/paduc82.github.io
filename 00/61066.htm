Port Forwarding

Login Information

  • Gateway - 10.0.0.1
  • Username - admin
  • Password - password

Port Forwarding

  1. Open a web browser.
  2. Log into the router by entering the default gateway into the address bar of the internet browser. (Example 10.0.0.1)
  3. The gateway will then prompt for a user name and password.  The default is that user name is “admin” and the password is “password”.
  4. From the menu on the left select Advanced.
  5. From the submenu on the left select Port Forwarding.
  6. Select Enabled next to “Enable Port Forwarding”.
  7. Select “+ Add Service”.
  8. In the following screen enter a Service Name that indicates something to do with Xbox Live (XBL).
  9. Change the Service Type to TCP/UDP.
  10. Enter the IP Address assigned to the Xbox console in Server IP Address.
  11. For Start Public Port, End Public Port and Private Port(s) enter 3074.
  12. Select Save.

Disable Port Forwarding

  1. Open a web browser.
  2. Log into the router by entering the default gateway into the address bar of the internet browser. (Example 10.0.0.1)
  3. The gateway will then prompt for a user name and password.  The default is that user name is “admin” and the password is “password”.
  4. From the menu on the left select Advanced.
  5. From the submenu on the left select Port Forwarding.
  6. Select Disable or select the “X” on the ports you would like to remove.

Windows 10 Hot Key "Heck" – My Journey There and Back - Microsoft Community

jedisql say:

Windows 10 Hot Key "Heck" – My Journey There and Back

I really like using Windows Hot Keys (keyboard shortcuts) to open programs.  I have been using them for over 15 years.  There is a major issue with Hot Keys – when I want to assign a Hot Key to a new program, Windows provides no way of determining which key strokes have already been assigned to existing file shortcuts.  So, I came up with my own way of tracking Hot Key assignments.  In the user Start Menu folder branch I would create a folder named Hot Keys.  Then for each program that I wanted to assign a Hot Key, I would copy the program's main Start Menu shortcut (say "Microsoft Excel 2007.lnk") to my Hot Keys folder.  For Excel I use the Hot Key Ctrl-Alt-X, so I would rename the shortcut to "{X} Microsoft Excel 2007.lnk", then open the Properties and then assign Ctrl-Alt-X as the shortcut key.

By keeping all the shortcuts with Hot Keys in the same folder, and naming them with an indication of the Hot Key assigned, I had no trouble keeping track of which key strokes I had used for Hot Keys.  This worked wonderfully for me from Windows NT 4 through Windows 8.1.  Now, however, Windows 10 has completely undermined my method of Hot Key management.  Here follows my trials and tribulations of attempting to have a working Hot Key management process in Windows 10.1511.

First of all, I tried copying a shortcut (e.g., "Excel 2016.lnk") from the system Start Menu folder (C:\ProgramData\Microsoft\Windows\Start Menu\Programs) to my user start Menu folder (C:\Users\«User»\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Hot Keys).  This results in "Excel 2016" disappearing from the "E" section of the Start Menu and appearing in only my "Hot Keys" Start Menu folder.  I strongly dislike that, and it is not acceptable.  (From what I have read, this probably results from Microsoft’s attempts to limit the Start Menu sprawl that affected earlier versions of Windows.)

Secondly, Hot Keys are supposed to work on shortcuts stored in the Start Menu or on the Desktop.  I tried working with a Hot Keys folder on my Desktop.  I copied a system Start Menu shortcut there, and the item remained in its original Start Menu location (cool).  I assigned the Hot Key, and it worked (cool).  Then I tried the Hot Key later that day, and it did not work (uncool).  Even when they did work, they were much slower to respond than ones in the Start Menu (I'm guessing Windows scans the Start Menu folders before the Desktop folders). 

Then I had an idea.  What if I just left he shortcuts in the system Start Menu folder, and rename them to have the {K} key stroke indicator as a suffix (so the items do not move in the Start Menu).  Then I would create a Saved Search for *{*} on the system Start Menu folder and its subfolders.  That should allow me to readily identify which key strokes I have assigned to Hot Keys.  But, alas, that had a major hiccup that took me a day to work around. It turns out that Windows 10 creates a broken Saved Search for *{*}.  With research I was able to construct a search argument that finds my Hot Key filenames and works correctly as a Saved Search:
System.FileName:~="{" AND System.FileName:~="}"

Then I had an important realization.  If I modify the system Start Menu folder shortcuts with my Hot Keys, then that would force my Hot Key assignments on all other users on the computer, too.  To avoid that I copied the shortcuts from their location in the system Start Menu folder to the same location in my user Start Menu folder.  This way, when my user copy of the shortcut causes the system shortcut to disappear from the Start Menu, my user copy appears in the same place.  Then I renamed my user "Excel 2016" shortcut to "Excel 2016 {X}" and assigned the Ctrl-Alt-X Hot Key, and the Start Menu item then appeared as "Excel 2016 {X}", as expected.  When I logged out and logged back in as a different user, the Start Menu item was, as I hoped it would be, still "Excel 2016".  Back in my account, with the {K} suffixes appearing on the items in their original locations, if I don't use an application for a while and forget it has a Hot Key, when I open it from the Start Menu, I will be reminded it has a Hot Key.

So, finally, I can:

  • Assign Hot Keys to start Applications,
  • Have the Start Menu items stay where they should be (albeit with the {K} suffix),
  • Count on the shortcuts to always respond to their Hot Keys,
  • Quickly open a Saved Search to find which key strokes I have used for Hot Keys (though not totally as easily as when the {K} indicators were prefixes, and the shortcuts sorted in key stroke order),
  • Get reminders that Applications have Hot Keys,
  • Let other users assign their own Hot Keys.

I know there’s probably not 10 other people in the world that care about this, but for the other two or three that might, I hope this helps.

"It's not a true crash, in the sense that the Blue Screen was initiated only because the combination of video driver and video hardware was being unresponsive, and not because of any synchronous processing exception".

Since Vista, the "Timeout Detection and Recovery" (TDR) components of the OS video subsystem have been capable of doing some truly impressive things to try to recover from issues which would have caused earlier OS's like XP to crash.

As a last resort, the TDR subsystem sends the video driver a "please restart yourself now!" command and waits a few seconds.
If there's no response, the OS concludes that the video driver/hardware combo has truly collapsed in a heap, and it fires off that stop 0x116 BSOD.

If playing with video driver versions hasn't helped, make sure the box is not overheating.
 Try removing a side panel and aiming a big mains fan straight at the motherboard and GPU.
 Run it like that for a few hours or days - long enough to ascertain whether cooler temperatures make a difference.

If so, it might be as simple as dust buildup and subsequently inadequate cooling.

I would download cpu-z and gpu-z (both free) and keep an eye on the video temps
