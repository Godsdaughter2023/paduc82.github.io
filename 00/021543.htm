BUG: System Tray and Shell_NotifyIcon API broken in build 14271 - Microsoft Community

techie007l say:

BUG: System Tray and Shell_NotifyIcon API broken in build 14271

    I recently received Windows 10 build 14271, and I immediately noticed the new bug where the system tray no longer sizes itself correctly.  No big deal; I had expected this bug because I saw it in the Known Issues report for this build. 

    However, what I did not expect, and I want to make sure that the Microsoft Windows Development Team is aware of, is that the Shell_NotifyIcon API no longer works as expected either.  I discovered this when I noticed that a couple of my tray icons were missing entirely.  I tried restarting the affected programs, but nothing I could do would make their tray icons appear, while other programs' tray icons appeared fine (or after scrolling the tray to find them).

    As a developer, I dug deeper and did some debugging.  It turns out that the Shell_NotifyIcon API is returning True—indicating success—even when it is called with the NIM_MODIFY flag and the tray icon does not exist, and has not been added to the tray.  This results in the icon never being added to the tray, and the program in question not even being aware of the failure.  The expected behavior is that the API will return False to indicate failure, and that the calling program will try again with the NIM_ADD flag to add the icon to the tray.

    I am aware of three different methods of managing tray icons:

  1. Call Shell_NotifyIcon with the NIM_ADD flag at program start, and use NIM_MODIFY periodically or when something changes.
    • Issue: If Explorer crashes, tray icons added this way will not reappear until the user manually restarts the program so that it uses the NIM_ADD flag.  Usually this means that the tray icons and any notifications from the affected programs will not show until the user signs out or reboots.
    • Few programs use this method because of the "Explorer crash=tray icon gone for rest of session" issue.
  2. Periodically call Shell_NotifyIcon with the NIM_ADD flag and immediately call it again with the NIM_MODIFY flag.
    • Brute force; these programs are currently working fine, although they incur extra inefficiency (lower battery life?) and an internal error every interval because of the NIM_ADD call which fails because the icon has already been added.
    • Most programs use this method.
  3. Periodically call Shell_NotifyIcon with the NIM_MODIFY flag.  If the call fails, call it again with the NIM_ADD flag to re-add the icon to the tray.
    • Proper way to manage tray icons.  If Explorer crashes, the tray icon gets re-added shortly because Shell_NotifyIcon returns false, triggering another call with the needed NIM_ADD flag.  The rest (and most) of the time, only one call is made per-interval.
    • Due to the new bug in the Shell_NotifyIcon API, this method no longer works because the initial Shell_NotifyIcon call always returns True, even when it fails because the icon was not previously added.

    Yes, I have already submitted this bug in the Windows Feedback app, so please vote it up!
