cmdrkeene say:
The standard best practice is for Program Files to be the actual program, and then the data stored in your Documents folder (or at least inside your user profile folder).  That way other users' can use the same program and their info is separate (they can't access your user folder).

There's a few ways around it.

1 - I'd really do anything you can to just store your data separate from the actual application.  But that's just me and I'm the ultimate sheep: I do whatever the Microsoft gods tell me to do, ha ha ha.

2 - You could install the program to a folder where you already have full write access.  For example, install the program to C:\Users\Name\Programs I Like (that is, make your own folder).  Since it's installed to a place that isn't a protected system folder (like Program Files), Windows won't try to protect you. 

3 - You could disable UAC, which I think (I'm not 100% sure) also disables this virtualization stuff where Windows tricks programs into thinking they have write access to a folder that they really are blocked from.  I really don't like this option, because it basically takes away the past decade of security evolution.  UAC has critics, but Linux and Mac have had it for 20 years: I think it's a good thing we finally have it too.




PS.  This is probably one of the biggest things I wish more people knew about: the folder redirection.  Even most programmers aren't aware of it, which is kind of scarey :-)

urielb say:

How can I change the drive assigned to the SSD to a SATA drive instead of the SCSI?

Original Title: SCSI instead of SATA

I built a PC with Windows 7 Ultimate 64bit.  My C: drive is a Samsung 840 Pro SSD.  The system never worked properly and I found out that in Device Manager the SSD which is a SATA 3 drive is configured as a SCSI drive.  Anybody know how I change the driver assigned to the SSD to a SATA driver instead of the SCSI?

a. user say:

Hello Uriel,

Thank you for your response.

SATA controllers will normally show up as "SCSI" controllers within windows. This is normal and nothing to worry about. Often SATA controllers will also be able to run in IDE mode, and then they would show up as IDE controllers, but then it is unlikely that they will be able to function optimally, as the IDE interface misses some features of SATA. So even if you don't have the option to set the cache in device manager, a SATA controller if running in SATA mode should run optimally.

The IDE bus is handled by the SATA3 controller, and if that controller is set, in the BIOS to "IDE" somehow it seems to get reported back to the OS as a SCSI device.  By setting that BIOS entry to "RAID/IDE" the IDE devices get detected correctly.

This can also happen if you change the BIOS settings of the SATA ports from "IDE" to "AHCI" after the Windows 7 installation.

I hope this information helps.

Thank you

The fix that worked on that thread is as follows:

Do a search on your PC and find the file RDIF1148.INF

Back it up sonewhere safe . . .

Open RDIF1148.INF in a text editor
Change the following lines from (You can copy and replace from here):

[Roland.NTx86.6.2]
;; Windows8
%RDID0148DeviceDesc%=RDID0148Install, USB\VID_0582&PID_0187 ; GT-001

[Roland.NTx86.7]
;; not supported

To


[Roland.NTx86.6.2]
;; Windows8
%RDID0148DeviceDesc%=RDID0148Install, USB\VID_0582&PID_0187 ; GT-001

[Roland.NTx86.7]
;; Windows 10
%RDID0148DeviceDesc%=RDID0148Install, USB\VID_0582&PID_0187 ; GT-001
