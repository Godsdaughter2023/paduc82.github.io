john w. vinson say:

This should be a fairly straightforward project, but you do need to study up about "Normalization" - the proper design of databases. For one thing, you need to start with the Tables and their relationships; forms and data entry can only be done once you have a good foundation in the tables. In particular, each Table must represent one type of "Entity" - real-life person, thing, or event; each field in a Table refers to an "Attribute" of that entity - something that you want to know about it.

In particular, an Employee has FirstName, LastName, hire date, etc. as attributes of that person; but a Voucher doesn't have a First Name! The Vouchers table should have an EmployeeID (don't use the # character in fieldnames, it's a date delimiter) as a foreign key, and NOTHING else about the employee; that's what queries, forms, combo boxes and other tools are for.

scottgem say:

Hi Melissa,

First, I agree with John, but there are some specific issues I can comment on.

1) Naming. You should not use spaces in Object names. They can come back to haunt you. Use either camel notation (FirstName) or underscores (First_Name). Also don't use special characters like the octothorpe (#). Also you should keep object names short ExpenseVouchers is sufficient, even better tblExpenseVouchers to indicate its a table object.

2) It appears you used a lookup field for EmployeeID (a better name) and, worse, made it a Multi-Value Field (MVF). It is not recommended that you do lookups on the table level, they should be done on the form. Also, since it would appear that each expense record is associated with a single employee, there is no reason for a multi-value field

3) You are using redundant data. This will be covered in the links John gave you, but since the employee name info is in the EmployeeList table, they should NOT be in the ExpenseVoucher table. Please see my blog on Displaying so the user would only see their records.

Not sure I understood all that.  If I did, then your report is grouping on the event ID and the event's "reports" are listed in the detail section.  (If you are only sorting on event ID, add a group header section to make it a group and make the header section invisible so you won't see it.)

ken sheridan say: You can return the distinct count of events in a query independently of the report by counting the rows returned by a subquery which returns distinct EventID values restricted by the same parameters as your current query:

SELECT COUNT(*) AS EventCount
FROM
    (SELECT DISTINCT EventID
     FROM YourTable
     WHERE SomeField = Forms!SomeForm!SomeControl
      AND SomeOtherField = Forms!SomeForm!SomeOtherControl)
AS DistinctEvents;

As this query returns only one row just add it to the report's current RecordSource query and return the EventCount column in the query, but do not join it to any other table in the query.  You can then bind a text box control to the EventCount column.

Hi

I did find a solution to this problem a while ago, thought I posted it. Anyway ...

I have a a spreadsheet where the tabs are by month. In each tab I have a number of column calculations (sum) based on indirect references. In the beginning I referenced the columns as A:A, to sum all the values in column A, etc. As I progressed thru the months, the calculation time got longer and longer. 

To solve it, I updated the indirect references to include the number of filled rows in a particular tab, using the counta function, so if I have 20 rows filled in a particular tab, the reference will look like A1:A20. For the counta function I used the maximum number of rows I expect in a tab (100), so this was counta(A1:A100)

This solved the problem, as Excel was only performing the calculation (sum) on the actual number of rows filled, rather than every row in the column.

Multiple internal domain how to Sync with One Office 365 Tenant - Microsoft Community

kathyyunli say:

Multiple internal domain how to Sync with One Office 365 Tenant

The company currently have multiple internal domains. The company have an Office 365 Tenant and all of business is configured for that cloud based email and integrated solution. 

The company have other domains that are not sync with the cloud environment with Microsoft yet but They need to allow cross other domain users to access and use the online SharePoint sites documents. 

what is best practices and procedure to the multiple domains users to access the share point online document management site. 

They do want to setup the security management, not open the document for everybody.

neha singh msft say:

Hi Kathyyunli,

If User A (userA@xyz.com) and User B (userB@abc.com) are added in same tenant such as contoso.onmicrosoft.com, then they are considered in the same organization and in this case external sharing is not required to be enabled.

If both the users are part of different organizations then they are considered as external users to each other and external sharing is required to be enabled to share the site.

If your site contain sensitive data and you want to share the site with external users and you are also concern about the security, then you can break the inheritance of the library which contains the sensitive documents and do not give the access of this library to external users or you can remove the data from the site and store it in another site which is not shared.

Once you turn external sharing globally, you can allow external sharing for specific site collections, while keeping it turned off for site collections that will store content that is sensitive for your business. You should consider where you enable external sharing and what level of external sharing you allow.

After successfully migrating users to Office 365 tenant 6 months later, the users with who you have shared the site will still have the access to site.

Since you are planning to migrate users to Office 365/SharePoint online, please check the following articles for help:

SharePoint Online planning guide

Deployment advisors for Office 365 services

Thanks,

Neha
