Windows 10 xHCI driver stack fails when IgnoreHWSerNum set - Microsoft Community

a. user say:

Windows 10 xHCI driver stack fails when IgnoreHWSerNum set

Environment: Windows 10 Home, version 1709

CPU: Intel Celeron N3060

Driver: Intel(R) USB 3.0 eXtensible Host Controller - 1.0 (Microsoft), version 10.0.16299.15

Assume the following: PC with configuration as above running USBview, and a USB device configured with a serial-number descriptor with index correctly set in iSerialNumber in the device descriptor.

  • If HKLM\CurrentControlSet\Control\usbflags\vvvvpppprrrr\IgnoreHWSerNum is set to 0x00 then a USB device as above, will enumerate on the serial number stored in the device.

    So, when IgnoreHWSerNum is set to 0x00, then enumeration works fine when a USB device is attached to a port, and the device is recorded in the registry HKLM\CurrentControlSet\Enum\USB\VID_vvvv&PID_pppp with a serial-number specific key.

    A USB bus-analyser shows that the xHCI driver reads in the serial-number string descriptor from the device during enumeration, and (using USBLyser to sniff traffic between USBview and xHCI driver) when USBview sends down a GetDescriptor(iSerialNumber) request then no such request is seen on the bus which means that the xHCI driver caches the serial-number internally, since the xHCI driver returns a serial-number back to USBview.

  • If HKLM\CurrentControlSet\Control\usbflags\vvvvpppprrrr\IgnoreHWSerNum is set to 0x01 then a USB device as above, will enumerate on the port at which the device was attached.

    So, when IgnoreHWSerNum is set to 0x01, then enumeration works fine when a USB device is attached to a port, and the device is recorded in the registry HKLM\CurrentControlSet\Enum\USB\VID_vvvv&PID_pppp with a port-specific key.

    But a USB bus-analyser shows that the xHCI driver does not read in the serial-number string descriptor from the device during enumeration, and when USBview sends down a GetDescriptor(iSerialNumber) request (as seen using USBLyser sniffer) then no such request  is seen on the bus which means that the xHCI driver still assumes that the serial-number is cached, since the xHCI driver returns a failure back to USBview.

    And on USBview an error is shown saying (in my case)

        iSerialNumber = 0x03

        *!*ERROR: No string descriptor for index 3!

So, the xHCI driver is build on the assumption that it shall always read in the serial number string descriptor during enumeration and cache this internally, so that subsequent request from drivers/apps above won't have to result in a request over the bus. But since IgnoreHWSerNum is set to 0x01 then the enumerator doesn't read in the string descriptor, meaning that there won't be any serial number in the cache. And therefore the upper layer of the driver which assumes that there ALWAYS will be a cached serial number available won't try to request the serial number from the device, and just return a failure back to the requester.

This description should be sufficient for a Microsoft developer to pinpoint the problem and correct it.

If you do a custom install, it will move your old installation to the Windows.old folder:

If you delete or format a partition that contains a version of Windows, all data on the partition is permanently deleted. However, if there is an existing copy of Windows on the partition you selected, but you do not format or delete the partition, user files are saved to a Windows.old folder on the partition, which you can browse to after the installation completes. This is for the Windows Recovery Environment (WinRE), which also shipped with earlier versions of Windows, such as Vista. It was available from the Windows Vista installation disc only, although some OEMs made it available on a separate partition on their factory installations.

In Windows 7, this feature is installed on all computers, and is stored in the 100MB partition if you install using a single partition scheme. However, if you install on a drive that has multiple partitions, the recovery environment is stored in a folder in the root of your installation drive, so if you install to a partition with a label of C:\ it will be located at C:\Recovery.

It's intended purpose is for loading the recovery environment, which can be used to repair and diagnose issues preventing Windows from starting, examples include: memory diagnostics, startup repair, command prompt, which you can use to fix boot issues.
