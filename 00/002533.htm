    IE 8 Internet Options window does not open (Some causes, hints and - Microsoft Community

    mdigno0o say:

    IE 8 Internet Options window does not open (Some causes, hints and solutions)

    Disclaimer: Below is a problem I encountered, a summary of what I learned in Microsoft Answers and MS Knowledge Base for similar symptoms, and how I solved my problem. Hope it will help.
    I am not a power user and I have no specific knowledge of the internals of Windows, although I am not quite a novice. So use the following with care (advanced users only).
    For detailed receipes on some solutions below, see existing threads (search for "inetcpl.cpl" in KB and Microsoft Answers).
    (Also, as my system is in french, I tried here to guess how some names would appear in an english version of Windows.)

    PROBLEM: (IE 8) "Internet Options" window does not open, either from IE, either from the Configuration Panel (CPL). And it may even be absent of the CPL, or be hard to find there (my case: was at the top of the CPL, with no name).

    CAUSE: This may have many different causes, two of them are described in MS documents and have a MS fix:.
    See:  MS Fix it ( http://support.microsoft.com/fixit/ ) Internet Explorer | Parameters and read below.

      (1)  An error message appears after clicking on Internet options (: ".. restrictions .. on this computer"). This is not a bug but a feature, due to a "NoBrowserOptions Windows" key in the Registry (HKCU\Policies\Microsoft\Internet Explorer\Restrictions). There is a fix, if you do have administrative rights. See: http://support.microsoft.com/kb/216583/

      (2) Otherwise, this may be due to any cause: incompatible IE external modules, incompatible options, corruption...There is a fix, for reinitializing all IE parameters (with loss of many preexisting objects), when "Internet Options" can't be reached directly in IE See: http://support.microsoft.com/kb/923737/  But this fix is mainly if your IE works bad, and may be overkill for our specific problem, and potentially useless here also (I did not try). Unfortunately, KB923737 document does'nt tell if "Internet Options" will be restored inside IE by this process. (In my case, Cause-4) it would probably have been useless.)

      (3)  The job of "Internet Options" is done in file "inetcpl.cpl" (in the system folder: e.g. C:\WINDOWS\system32\  ). That file may be corrupt, misinstalled, unreached, ... Below are some things to do for checking and trying to execute that file.

      (4) This may be due to a problem with internationalization (language), as I discovered by chance in my case: inetcpl.cpl does not find the language interface file he is looking for ( inetcpl.cpl.mui ). In that case, if one looks carefully, when clicking on "Internet Options" from IE, a little white window (with seemingly an OK button) does appears in a blink (less than 0.5 sec). My solution for this cause below.

    MY CONTEXT: Windows XP SP3 (Regional option and language: fr-ca, with a fr-ca keyboard) and IE6 SP3 (french version) (all up-to-date in Microsoft Update), and working well. I updated to IE 8 (8.0.6001.18702 / french version) from MU (followed with a 2nd run in MU for any after release security updates). (Account with administrative privilege, of course).
    At first glance just after installation, IE 8 (french interface) seemed to work correctly, but I immediately noticed the "Internet Options" problem (see Problem, and Cause #4 above). Was especially annoying, as I made a lot of options personnalization in my old IE 6 (prompting for ActiveX exec or redirection, file download off  in Internet zone and put to "on" manually only when I needed, etc.), many of these functions are handled differently in IE8 (e.g. no prompts), and nothing could be adapted due to the problem, so that IE 8 became practically unusable on many sites

    SOLUTION-1 (useful for Cause-3 above): Trying to make the file inetcpl.cpl to execute. Below are some (schematic) suggestions I found in MS ANSWERS (thanks to the contributors), although none of them worked in any context for my specific problem before I discovered why (Cause-4).

      - Right click the IE icon on the Desktop, and open with no external modules, to discriminate the case.
      - Try to execute "Internet Options" from the CPL if it is present there (see icon at the top if no name)
      - See if the file exists in (SYS) (e.g. ..\WINDOWS\system32 ). Caution in this folder!! Examine Properties (WinXP: should see: File type: CPL extension, Opens in Common DLL Windows shell, or Opens in Configuration Panel  (approx translations), depending on the location of the file).
      - Examine the file association for ".cpl" file type. (WinXP: In Explorer: Tools | Folder Options | File types). Should see info similar as above (CPL extension / Opens with: ...Windows  DLL)
      - File may be corrupt: try to repair it in case (with System File Check)
      - Ensure the file is registered:  Start menu | Run: regsvr32.exe  /n  /i  inetcpl.cpl
      - Execute the file (.. Run:): inetcpl.cpl
      - Execute the file with more confidence: control.exe inetcpl.cpl
      - Retry the same in Windows safe mode with Network (account "Administrator"). Try also IE there.

    SOLUTION-2 (for Cause-4 above): (Advanced users only).
      1. Locate and copy the 2 related files in a new local folder (e.g. My documents\IE_TMP\ ): inetcpl.cpl  and  inetcpl.cpl.mui  The first is in (SYS), and in my case, I found the other in (SYS)\fr-fr\ 

      2. An alternate method if the first gives no results in step 3. (the one I used before discovering the solution), is to download IE 8 (in the appropriate language) directly from MS Download Center.
     Then open the downloaded file (Self extracting cabinet): this will extract all the files in a temporary folder (mine was in C:\{.....}\ ).
    After that, the welcome window of the installation Wizard will appear (do nothing there). Locate the temporary installation folder just created, and copy the 2 extracted files from there into your local folder.
    Then cancel the installation in the Wizard, as you do not want to reinstall IE... The temporary folder will disappear.
    (Notice with this method that you will possibly get a different version of inetcpl.cpl than in your original installation. Check the file properties.)

     3. Execute (open) inetcpl.cpl from your local folder. In my case, it worked: the Internet Options window now appears and is functional. And I could see most of my old (IE 6) options have been preserved in IE 8, or adapted to the new options. And I now can modify them! (*)

        But (as I checked), the very same file that works in my local folder no more works when copied into the (SYS) folder!!! This was for me the hint that the file itself in (SYS) was OK, but it was just unable to locate its associated MUI needed for presentation of the window.

    One has now the option of creating link on the desktop (e.g.called "Internet Options"), that will point to the local copy of the file, and to execute the link for the "Internet Options" window to appear. So, this is a good workaround.

     4. (Advanced users) In my case, I guessed that inetcpl.cpl expected to find inetcpl.cpl.mui in the (SYS)\fr-ca\ folder (language code of my Windows). But the folder did not exist. So I created the appropriate subfolder, and put there a copy of the MUI file.Then, eureka!  all was working well from IE, as well as from the CPL (except for the fact that the icon is unnamed.).

    In other words, in this specific Case-4, it seems the "subdialect" of the system which was determined during installation of the IE8 build for a given language was not the same as the reel subdialect of the system (e.g. fr-fr vs fr-ca / en-us vs en-uk), so that the MUI file is stored in the "wrong" place by the installation.

    It would be fine if Microsoft produce a Fix it for this specific problem (installing the proper MUI at the right place).

    I must say here that there was only a "Français" (french), no specific regional version of IE8 available when I downloaded from the MS Download Center. (This is probably normal.) And it is probably the same for the version installed through Microsoft Update.

    NOTE (*): If the test in step 3 works, it is because one can also demonstrate that if inetcpl.cpl does'nt find the MUI in the language subfolder (e.g. fr-ca), it fetches it in its own folder.

    REMARK: It is important however to precise here that the problem arised, as I said, after installing IE8 from Windows Update (WU).
    But I did not reinstalled it with the version (IE8-WindowsXP-x86-FRA.exe V:6.3.15.0) downloaded from MS Download Center (DC) that I used for extracting and testing the 2 CPL files.
    So, I cannot confirm that the MUI file would also have been misplaced if I had installed IE8 with that DC version..

    REMARK: Just after IE 8 installation, I saw a "Language bar" appear in the task bar. It disappeared at some point. (Retrospectively, this was probably the first visible sign of the problem.) So, I suspect that the IE installation made some things new on the language management side, and/or Windows detected the appearance of a "new" language-region. Later, I went into the CPL (Regional options), and I saw that the Language bar was in the menus. But I do not dare playing with this, as all have worked well for years on that side in my system.

    EPILOGUE: Hope this will be useful if you get into the same problem. Sorry that I won't be able to help more on my side.

    Great you got it sorted.

    This issue is (briefly) discussed here:
    http://msdn.microsoft.com/en-us/library/aa722523.aspx

    as follows:

    (VBA only) If two documents have the same callback signatures, the callbacks from the active document are called. How do I ensure my UI calls only the callbacks associated with my document?

    This is an issue that was also present in Office 2003. As a workaround, you can make your callback names unique by adding your add-in or solution name to the callback name. You can also put your callbacks in a module, and then refer to your callbacks by using the full name of the procedure. For example, if you put your callbacks in a module named "MyAddInXYZ", you can refer to the callbacks by using "MyAddInXYZ.myCallback".

     

    As I understand it, if two callbacks with the same name are present, the one that gets called depends on the order that the addins were loaded - so that makes sense of your issue since the Other add-in (template) will have been loaded first when you loaded your template by hand.

    Thank you for reminding me of this, as I still have on my to-do list to go through a big ppt add-in I have made and change my rather too generically named rib As IRibbonUI to something "more" [sic] unique, as well as my module names.

    So I think if you wanted to be completely paranoid (sorry, I mean "robust") you can "uniquely" name:

    (a) your ribbon object

    (b) your modules (and include their names in your XML callbacks: modXYZ.getVisible )

    (c) your callbacks (by including the project name and control ID in the name, eg pjojXYZ_btnABC_getVisible, but in that case you loose the ability to have a single VBA procedure for all / a group of your controls, but in a large project I find it easier to group similar controls by VBA module, and have separate callbacks for each so that I can add and remove controls without affecting other controls).
