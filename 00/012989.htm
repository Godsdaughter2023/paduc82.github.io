ken sheridan say: Criteria is just the plural of the word criterion ( –on is the nominative neuter singular ending of a first declension noun in Greek, –a is the nominative neuter plural ending; equivalent to –um and –a in Latin, e.g. datum and data).

So a condition:  WHERE City = "London OR City = "Paris" represents two possible criteria because a row will be returned if the value in the City column is London or it is Paris.  There are thus two possible criteria on which the row can be returned.   For each row only one criterion can be TRUE, however, which is what John is referring to when he says there can be one [actual] criterion only, as distinct here from two or more possible criteria.

On the other hand :  WHERE Country = "France" AND City = "Paris" represents only one possible criterion; the row must have values of France and City, with no other possibilities.  So a row for Paris, France would be returned, but not one for Paris, USA.  In this case the possible criterion and the actual criterion are the same.

A Query can have only ONE criterion. Just one! However that criterion can be quite complicated, with multiple fields being searched, independently or together. My point wasn't that "Access will break", just that the Design view query grid is a somewhat limited tool; its purpose is to build SQL strings. The SQL is the REAL query, and it's possible to type in SQL strings which cannot be represented in the query grid at all (UNION queries for example). 

There are limits - some queries will fail with an error message "Query too complex" - but you're nowhere near that point. When I've gotten the error it involved several fields and many levels of parentheses (or equivalently, many rows on the grid). Two is a breeze.
