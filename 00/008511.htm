john --
 
To piggyback on Joel's excellent answer, here is what you need to know about filters in Microsoft Project 2010:
  • When you select a filter from the Filter pick list, the system applies the filter as aStandard filter. This means that the filter shows only the tasks that meet the filter criteria, and hides all tasks that do not meet the filter criteria.
  • When you select a filter from the Highlight pick list, the system applies the filter as aHighlight filter. This means that the filter displays all tasks, but highlights the tasks that meet the filter criteria.
So, as you can see from the above explanation, the Filter pick list and the Highlight pick list are mutually exclusive because each pick list does something entirely different from the other.  Hope this helps.

macropod say:
By default, Word *doesn't* count content in footnotes, endnotes or textboxes. You can change that by checking/unchecking the appropriate checkbox on the word count popup.
Regardless of which setting you use, Word also doesn't count content in page headers or footers. Note also that, if you have Word's 'print field codes instead of their values' option checked, Word's word count will reflect the # words in the field codes, not what the fields display.

What happens when more than one conditional formatting rule evaluates to true

For a range of cells, you can have more than one conditional formatting rule that evaluates to true. Either the rules don't conflict or they conflict:

jeeped say:
I believe you might be hitting a functional limitation of an array passed back as a value from a UDF. The number (65536) brings suspicion immediately as it is the upper limit of an unsigned two byte integer (aka unsigned short). It also represents the row limit on pre-2007 Excel worksheets.

I played with this and had no luck getting past the 65536 cell limit. The array is fine up to the point that it is assigned back to the function as a return value, then  it becomes an array of non-negotiable #VALUE! errors on the worksheet end.

Couple of things to point out: 

I did notice that there was no finite limit of 65536 cells, only 65536 rows. If you split your values into three columns of roughly 54,138 rows each, the following array formula does not choke,

=AVERAGE(IF(IsVisible(B2:D54138),ABS(B2:D54138)))

Another option is to use the AVERAGE() function's ability to accept split ranges.

/** the formula may not be correct **/

Secondly, since you are converting the values in column B to their ABS() values, blank cells will be averaged as zeroes instead of being discarded from the data set as they normally would be in an AVERAGE() formula. To leave them discarded, their status as a blank or non-blank cell should be added to the criteria for hidden rows and/or columns. Something like,

    For RNdx = 1 To InRange.Rows.Count
        For CNdx = 1 To InRange.Columns.Count
            Arr(RNdx, CNdx) = Not (InRange(RNdx, CNdx).EntireRow.Hidden Or _
                                   InRange(RNdx, CNdx).EntireColumn.Hidden Or _
                                   IsEmpty(InRange(RNdx, CNdx)))
        Next CNdx
    Next RNdx

And finally, to get a functional hidden row/column average from a single column of rows greater than 65536, it would help to have a second form of formula that you can use as a check-and-balance. For this you are going to require a helper column. In an unused column to the right, use this formula in the second row,

=IF(ISBLANK(B2),"",ABS(B2))

... and fill down to row 162412. Now you can use SUBTOTAL() for an average of the ABS() values while discarding hidden row, text and blank cells. Assuming that you used column Z for the helper column then,

=SUBTOTAL(101,Z2:Z162412)
