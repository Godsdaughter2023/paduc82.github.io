Unbound controls, by their very nature, do no update when the record changes. You can use the On Current event to set the value of the control to match the current record.

On a separate note, a name should NOT be used as a "common field". First names are not unique, second, names are subject to change, third, text fields will slow down joins. You should have an autonumber primary key and use that as your linking fields.

 

Hope this helps, Scott<> P.S. Please post a response to let us know whether our answer helped or not. Microsoft Access MVP 2010 Blog: http://scottgem.spaces.live.com/blog Author: Microsoft Office Access 2007 VBA Technical Editor for: Special Edition Using Microsoft Access 2007 and Access 2007 Forms, Reports and Queries

ken sheridan say: What is the unbound combo box for?  If, for instance, it's to navigate to a particular record, then you can set the combo box's value to the value of the relevant column in the current record in the form's Current event procedure.

The following for example is the code for a combo box's AfterUpdate event procedure of a combo box to navigate to a selected contact:

    Dim rst As DAO.Recordset
   
    Set rst = Me.RecordsetClone
   
    With rst
        .FindFirst "ContactID = " & Me.cboFindContact
        If Not .NoMatch Then
            Me.Bookmark = .Bookmark
        End If
    End With

In the form's Current event procedure, the following keeps the combo box in sync with the current record if the user navigates via the navigation bar or keyboard:

    Me.cboFindContact = Me.ContactID

Your combo box's purpose may differ of course, but it will almost certainly be the form's Current event procedure where you'll need to update its value.

Ken Sheridan, Stafford, England

ken sheridan say: As a rider to my previous reply, I notice that you are using client name columns as the keys on which your tables are related.  This is not a good idea as names can legitimately be duplicated.  At the risk of boring those who've heard the story here before, I was once present at a clinic where two patients turned up; both female, both with exactly the same names and both with the same date of birth.  This illustrates how unsuitable names are for keys, even if combined with other attributes.  A distinct numeric identifier should always be used for people.  This can be an autonumber in the referenced table, with a corresponding long integer number column in the referencing table.  ContactID in my example is such.  In the combo box the numeric key is hidden and the name shown by setting up the control as follows:

RowSource:     SELECT ContactID, FirstName & " " & LastName FROM Contacts ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first dimension is zero to hide the first column and that the second is at least as wide as the combo box.

Creating the Link
  1. In Microsoft Access, activate the Database window.
  2. Choose the Query button (or from the View menu choose Queries.)
  3. Select the icon representing the desired query to copy. From the Edit menu, choose Copy. (Note: Do not open the query and choose Select All Records from the Edit menu.)
  4. Switch to Word for Windows.
  5. On the Edit menu, click Paste Special.
  6. In the Data Type box, select Unformatted Text and then choose the Paste Link button.
Question is can we do something similar with an Access report? Or is there a better means to see the "live" result of an Access report directly from SharePoint 2010 without a user having to browse to Access and run the report?

tom van stiphout say:

That is a really old KB article. I would not put too much stock into it.

 

To answer your question: No, that is not possible. Access reports are Access objects that only live within the context of an instance of msaccess.exe. You can export such report to some output formats like PDF or Excel, but it will not be live, but the snapshot at the time of exporting.

 

If you want to see report data live in SP, then SP will have to query the data source and show the results. You would have to read up on how to do that, or ask a question in a SP-related forum.

 

Maybe the data should be stored in a SP list rather than an Access table. SP would then EASILY be able to report on it, and Access can link to that list as well.
