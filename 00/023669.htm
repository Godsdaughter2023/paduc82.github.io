Automatically Normalize Data during Import/Append and Synchronize - Microsoft Community

pmprog say:

Automatically Normalize Data during Import/Append and Synchronize Primary/Foreign Key Values

Scenario:

 

My database’s current design has a single table scheme with one primary key field and several dependent fields. This format is convenient since new data is downloaded from a corporate database to a csv file. An Excel macro is then used to clean-up the data for importing/appending into my database using an Access saved import. Data is imported daily and the volume is such that manual entry/updates are impractical.

 

Unfortunately, the comments field combines multiple comments in a single record so that it is not even possible to acheive 1NF with the single table scheme. The single table could be transformed into 1NF by separating each of the comments into its own record and storing the comments in their own table. Each comment in the comments table can then be linked to its reading in the readings table.

  

 

Example:

Single Table Scheme (Current Design)

 

           tblImport(ImportID,DeviceName,DateStamp,Methane,Comments)

             where ImportID is (PK)AutoNumber

 

Normalized Table Schemes (Design Goal)

 

           tblReadings(ReadingID,DeviceName,DateStamp,Methane)

             where ReadingID is (PK)AutoNumber

 

           tblComments(CommentsID,Comments,ReadingID)

             where CommentsID is (PK)AutoNumber and ReadingID is (FK)Number

 

where the normalized schemes have a one (tblReadings) to many (tblComments) relationship

 

Problems:

 

1)     How can the comments data be distributed automatically to its own table during importing/appending?

2)     How can the primary key/foreign key values of new data be synchronized automatically during importing/appending?

marshall barton say:

Maybe you question is about the steps involed?  You alreadu have a table with the imported data so you can use that as a temp table, then tble readings can be populated with a simple append query.

I think you would need a VBA procedure to read each reord in the temp table, use a loop to parse the comments field into the separate comments and write them to tblComments.  If you have a rigorous syntax that can be used to separate the individual comments, this is relatively straightfoward using two recordsets.  The primary key in ttblComments will be supplied automatically and the foreign key is just a copy from the temp table's PK.

Set rsTemp = db.OpenRecordset("temptable")
Set rsComments = db.OpenRecordset("tblComments")

With rsComments
Do Until rsTemp.EOF
   Do
      strComment = <parse the next comment>
      .AddNew
         !fk = rsTemp!PK
         !comment = strComment
      .Update
  Loop Until <no more comments>
  rsTemp.MoveNext
Loop
End With

macropod say:
The limit is by design. To get that large, your document presumably has many graphics. If you change the file extension to .zip, then open it as a zip archive, you should be able to delete one or more of the embedded graphics; better still would be to reduce their size and leave them in-situ. Once you've done that, restore the original extension and re-open the document. Word will complain about any deleted graphics, but the file should otherwise open OK. If you activate Word's picture compression, that should result in a further-reduced file size.

Use the Manage Add-ons tool to determine which add-on is causing the issue and try to delete that add-on.

g. If issues do not reoccur, repeat steps ‘a’ through ‘c’.

h. Click Enable for a single add-on.

i. Repeat steps ‘f’ through ‘h’ until you determine which add-on causes errors to occur.
