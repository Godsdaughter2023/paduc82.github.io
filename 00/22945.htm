Files with a .doc extension are not templates - they are documents. Documents are created from templates, most commonly from Word's 'Normal' template.

In a business environment, documents normally should not be changed just because a new person comes along or titles change, and so on, as they are meant to represent the state of affairs when they were created. If you are using documents as templates (i.e. you're re-using old documents and saving them with new names and details), which is bad practice and fraught with danger, there is no way a macro will be able to distinguish between them and the other documents.

Application.xlfunc v. WorksheetFunction.xlfunc? - Microsoft Community

joeu2004 say:

Application.xlfunc v. WorksheetFunction.xlfunc?

Consider the following:

Sub doit()
Dim x
x = Application.VLookup(0, Array(1, 2), 1)
If IsError(x) Then MsgBox "error" Else MsgBox x
x = WorksheetFunction.VLookup(0, Array(1, 2), 1)
If IsError(x) Then MsgBox "error" Else MsgBox x
End Sub

Application.VLookup returns an error to "x", whereas WorksheetFunction.VLookup stops in debug mode due to a VBA error.

Obviously I can work around the difference in the latter case by using On Error Resume Next and If Err.Number<>0 Then... around the call.

But that is not my question.  My questions are:

1. Where is Application.VLookup et al documented?  That is, where is it documented that all(?) of the same XL functions that can be used with WorksheetFunction can also be used with Application?  VBA 6.5.1024 in XL2003 does not show those functions as members of Application, at least not in the interactive tips.

2. Does Application.xlfunc invoke exactly the same code as WorksheetFunction.xlfunc, at least for the "meat" of the functionality?  That is, do they operate identically other than the error handling?

3. Is there any reason to use WorksheetFunction.xlfunc instead of Application.xlfunc, assuming that I do not want procedures to stop due to a VBA error?

But that is not my question.  My questions are:

1. Where is Application.VLookup et al documented?  That is, where is it documented that all(?) of the same XL functions that can be used with WorksheetFunction can also be used with Application?  VBA 6.5.1024 in XL2003 does not show those functions as members of Application, at least not in the interactive tips.

2. Does Application.xlfunc invoke exactly the same code as WorksheetFunction.xlfunc, at least for the "meat" of the functionality?  That is, do they operate identically other than the error handling?

3. Is there any reason to use WorksheetFunction.xlfunc instead of Application.xlfunc, assuming that I do not want procedures to stop due to a VBA error?


A long time ago, when Microsoft introduced the Worksheetfunciton object the documentation indicated the direct references to application.xlfunc were depracated.  However, they have never been removed and even new functions (like those introduced with 2010) are available directly through applicatin.newxlfunc!

I don't know if the difference in behavior between application.vlookup and application.worksheetfunction.vlookup is intentional or not (I believe but am not sure there is a similar mismatch for the Match function).  Irrespective, it's probably too late for MS to make the two work the same -- too many instances of code breaking no matter which way they go. ;-) 

I suspect the difference in behavior is intentional because a function like NORM.S.DIST (introduced with 2010) throws an error if used with app.worksheetfunction but returns an error if used as app.Norm_S_Dist()

Why use app.worksheetfunction.xlfunc?  Because the returned value will be as expected (for example, a real for Norm_S_Dist) and not a variant.

 

dim Rslt as double

on error resume next

Rslt=application.worksheetfunction.norm_s_dist("a",false)

if err.number <>0 then ...

...

 

compared to

 

dim Rslt as variant

Rslt=application.norm_s_dist("a",false)

if typename(Rslt)="Error" then ...

 

I prefer worksheetfunction.xlfunc because I can use an error handler in a consistent manner.

Move tables in front end to back end - Microsoft Community

rochesterhousing say:

Move tables in front end to back end

I separated my database into a front end and back end.  Then I (foolishly) created some new tables and created them in the front end instead of the back end. Now my users, who are using an ACCDE front end can't see the info in the new tables.

 

I'm trying to find out if there is a way to move individual tables to the back end.

 

Thank you.

john w. vinson say:

Back up your frontend and backend.

Open the backend (you should open it exclusively), and use External Data... Access... Import to import the tables from your frontend. I'd suggest compacting the database after you do so.

 

Then open the frontend (your accdb master copy). Take a deep breath, and DELETE these tables. (you did back everything up... RIGHT?) Then use External Data.... Access... Link to link to the tables. Compact the frontend as well.

 

Now you can make a new .accde and it will have links to the new tables in the backend.

 

Of course if any users have entered data into their individual frontend copies of the tables, you're in some real trouble, since they will each have their own incompatible copies... good luck!!
