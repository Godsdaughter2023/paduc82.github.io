wulfraem say:

Hi,

I have this issue as well with Microsoft Office 2016 on Windows 10 and at least found a workaround for it.

For some reasons Windows seems to look in the wrong folder and one way to fix this is:

  • rename  the folder "C:\Program Files\Microsoft Office\Office16" to "C:\Program Files\Microsoft Office\Office16.bak" to backup the existing state
  • start a cmd as admin, then
    cd "C:\Program Files\Microsoft Office"
    mklink /J Office16 "C:\Program Files\Microsoft Office\root\Office16"
  • This will create a directory junction to the folder that has the executeables Windows is looking for at the place it is searching for it. Kind of a reverse solution as it does not fix the issue that Windows is searching in the wrong place but it works for me.

If you're using a different version of Office you have to replace the "16"s in the description above with the number matching your version. Should be easy to figure out when you're already in the file system.

Something like this (though you really should rename the command button to a more meaningful name):

'------ start of code ------

Private Sub Command137_Click()

    On Error GoTo Err_Command137_Click

    If MsgBox( _
            "Do you want to update the Verified Date?", _
            vbYesNo, _
            "Update Date?") _
        = vbYes _
    Then
        Me.VerifiedDate = Now()
        Me.Dirty = False   ' save record
    End If

    DoCmd.Close acForm, Me.Name, acSaveNo

Exit_Command137_Click:
     Exit Sub

Err_Command137_Click:
    MsgBox Err.Description

End Sub

john w. vinson say:

The tab order is completely independent of the position of the controls on the form. Select the control in form design view and set its Tab Order property to 5 - Access will renumber the rest. Or you can right click the little square at the upper intersection of the rulers and choose Tab Order; you can manually move controls up or down in the order, or select Auto Order to reorder them left to right, top to bottom.

 

Do note that if you have a Table with 25 fields, one for each question, you're making a pretty serious design mistake! "Fields are expensive, records are cheap": a better design for a questionnaire has THREE tables (at least):

 

Questions

  QuestionNo (Primary Key)

  Question (text)

  <maybe other fields about the question>

 

Questionnaires

  QuestionnaireID (Primary Key)

  AdministeredOn (date/time)

  Answerer (who answered the questionnaire - text or a link to a People table)

 

Answers

  AnswerID (autonumber primary key)

  QuestionnaireID (whose answers are these and when)

  QuestionNo (link to Questions)

  Answer (whatever the datatype of your answers is)

 

Rather than 25 fields for the questions and answers, you would have 25 rows. If you add a new question you don't need to change your form at all!

 

For an excellent worked-out example see

 

http://www.rogersaccesslibrary.com/Otherdownload.asp?SampleName='At%20Your%20Survey%202000'
