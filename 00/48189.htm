xb1r athena say:

DMZ

Login Information

 

  • Gateway: 192.168.1.1
  • Username: admin
  • Password: admin

 

 

DMZ

 

  1. Open a web browser on a computer.
  2. Type the address “192.168.1.1” into the web address bar.
  3. When prompted for a username and password type in “admin” for both the username and the password.
  4. On the left hand side of the web page select the “NAT” option.
  5. Select the “DMZ” subcategory below.
  6. Check the “Enable” box under DMZ settings.
  7. Select the Public IP address in the “Public IP Address” drop down list.
  8. Type in the Xbox IP address in the “IP Address of Virtual DMZ Host” box.
  9. Click on the “Add” button.
  10. Click the “Ok” button.

 

Disable DMZ

 

  1. Open a web browser on a computer.
  2. Type the address “192.168.1.1” into the web address bar.
  3. When prompted for a username and password type in “admin” for both the username and the password.
  4. On the left hand side of the web page select the “NAT” option.
  5. Select the “DMZ” subcategory below.
  6. Uncheck the “Enable” box under DMZ settings.
  7. Click the “Ok” button.

cwh050 say:

File and print sharing stops working after RNDIS devices are connected a number of times

I have a customer, who is supporting a range of devices that use Remote NDIS to establish a data connection between our device and our configuration tool. They are saying that after a number of connections (i.e. after they've set up a number of these devices) file and print sharing stops working. If one tries to click on a shared folder in Windows Explorer, an unspecified error appears with error code 0x80004005.

Only file and print sharing (SMB) stops - email and internet access remain unaffected. They are able to ping network devices (printers; servers and their access router) so network connectivity is still there. This problem is still evident if our device is not connected (IP address conflict).

They say that the only way to restore network access is to perform a system restore to a date some time prior to the error appearing. After doing the system restore, the PCs will work with our hardware for some time (i.e. they are able to connect and set up a number of devices) before the error appears again - after which they have to restore to the same point I mentioned above.

Between the "known good" restore point and present, there were no applications installed or changes made to Windows. Our application and hardware (we've sold over 4 million of our devices) are used on other PCs throughout the world so it's definitely not our application or hardware or driver. However, as the error only occurs after the customer has connected a device x number of times to the PC, it appears to be a problem in our driver; application or hardware.

The mapped (J:\); UNC (\\server\share) and and IP (\\ip.add.res.s\share) paths don't work. The other PCs on the LAN can be seen but cannot be accessed if they have a shared folder.

I've tried the following to no avail:

  • Disabling Windows firewall
  • Stopping the Antivirus service
  • Ensuring network discovery is enabled
  • Deleting the IPv6to4 device
  • Deleting the ISATAP drivers
  • Clearing cached credentials
  • Uninstalled and reinstalled the File and Printer Sharing for Microsoft Networks
  • Disabled IPv6 in the network connection (their LAN is purely IPv4)
  • Confirmed that the mode of authentication and session security to be used for network logons is set to LM and NTLM NTLMv2.
  • Confirmed that the network connection is in order
  • Confirmed that there are no entries in the LMHOSTS file.
The customers network is a flat, layer 2 LAN with file servers; PCs. They do have a domain controller but this problem occurs even when the PC is not connected to the LAN.

When one of their PCs exhibit this problem Wireshark shows no SMB or NetBIOS/IP traffic whatsoever whilst the PC is connected to the LAN.

As a test, the customer tried setting up a simple network consisting of two PCs connected directly to each other via a WiFi AP (i.e. two PCs on the same LAN). They cheated a shared folder on both PCs but they were unable to access the shared folder on either PC. So the assumption is that there is no issue on the network.

Only a system restore works!

What I do notice is that on their PCs, our driver is not being copied to the specific folder defined by our application. On PCs which don't seem to exhibit this problem - specifically mine - the driver files are there.

Any ideas?

Thanks
Wayne
