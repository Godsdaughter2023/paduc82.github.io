ken sheridan say: While server products like SQL Server and Oracle include the means for hierarchy processing I wouldn't advise anybody to roll their own genealogy application with them.  The availability of genealogy software is widespread, and even the free packages include all the functionality most people will need, and if something more powerful is later needed then the easy portability of the data in the GEDCOM format makes upgrading a doddle.

While any genealogy package worth the salt will import or export a GEDCOM file I'm not sure if there are any which will import from data in the adjacency list format described by the OP.  There are many tools for converting GEDCOM data into other formats, including into an Access .mdb file, but the assumption always seems to be that the starting point is the GEDCOM file, not the other way round.

My little 'bill of materials' demo could probably be adapted to produce a family tree of sorts, but it would be pretty crude compared with what can be done with even the simplest genealogy packages.   I'll maybe give it a try some day and see what can be done.

If we take the mathematicians' view of a family tree, where the relational positions in the tree are the nodes rather than the individuals who occupy them, then Joe Celko's nested set model would be worth looking at.  Each node could be identified by his lft and rgt values, along with a foreign key referencing a table of people.  As a person can occupy more than one node, then the values in this foreign key column would not be distinct.

One thing to remember about family trees of courses is that they go both up (to a person's ancestors)  and down (to their descendants).  Most people are looking back to their ancestors, though if you were Alec Guinness in Kind Hearts and Coronets you'd want to work down from one of your ancestors to find out who you need to bump off to inherit the family fortune.
