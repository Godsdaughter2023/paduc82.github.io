hi matt,

after confirming with the related team, this is an expected behavior. we can’t achieve the goal by running powershell cmdlets/user admin tools.
let me make some summaries for this based on my test:
when we set the “editor” permission on the delegate (user a) for user b in the outlook client, the checkbox of “delegate receives copies of meeting-related messages sent to me” is checked by default:
 
there are three permissions on this action:
1. grant the “editor” permission for the calendar folder to user a.
2. grant the “send on behalf of” mailbox permission to user a.
3. receiv the meeting request message for user b.

and then user a can get:
1. edit the calendar items on user b via the “editor” permission.
2. receive the meeting request message.
3. response to the meeting request ofthe meeting organizer (a sends on behalf of b).

if we only remove the “editor” permission via running “get/remove-mailboxfolderpermission”, the “send on behalf of” permission can’t be removed. user a can still receive the meeting request, but can’t reply to it.
 
even though we remove the send on behalf of permission via running the powershell cmdlet (set-mailbox "b" -grantsendonbehalfto @{remove="a"}), in ui of eac or uncheck “delegate receives copies of meeting-related messages sent to me” in the outlook client, the notifications can still be delivered to the inbox of user a.
i have also analyzed the message header of the meeting request email that user a received and noticed the message was forwarded from user b. see the screenshot below:
 
however, i can’t find this inbox rule via the powershell cmdlet (get-inboxrule -mailbox *** Email address is removed for privacy ***|fl) or ui. if we can, just removing this inbox rule should achieve your goal.

given the situation, to completely stop the delegate receiving the meeting requests, the right action is removing the delegate from the outlook client manually.
