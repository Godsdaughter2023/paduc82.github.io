10581: Storage settings and SD card info - Microsoft Community

mike (microsoft) say:

10581: Storage settings and SD card info

Hey folks -

As mentioned in this morning's blog post, there's some details we want you to understand in regards to default save locations for storage and SD cards on your phone:

You will be unable to set your default save locations via Settings > System > Storage and will not correctly reflect the current storage settings. If you are using a SD card in your phone, it will show up as a garbled name in Storage settings. Whatever you had configured for your save locations in the previous build will be the same.

If I hard reset my phone on 10581, what happens to my storage settings?

Phone storage settings go back to default (which is "on phone") and you cannot change on this build.

If I upgrade from an earlier build to 10581, what happens to my storage settings?

The phone will respect your previous storage and default save location configurations. So, for example, if you had configured your SD card as the default save location for apps in a previous build it will still be respected on 10581. 

If I had configured my default save locations to an SD card, will my apps and other data still be accessible in 10581?

Yes, your apps and data should still be accessible. The problem only impacts the ability to configure and change the default save locations in this build.

Position a userform at caret in Word - Microsoft Community

ervit say:

Position a userform at caret in Word

Hi all,

I'm trying to enhance an add-in, and want to be able to place a small userform at the caret position in a document. Since Word doesn't allow inline text manipulations via VBA (long story), the author of the addin created a userform to handle such editing (inline editing happens in a textbox and then final text is placed in the document), but his solution uses a static userform placement, whereas I want to place a smaller version of the userform right where the active editing is, at the caret position (so that it would move with the caret in near-real time).

I have searched high and low and have found some solutions, but they don't work for my intent. The best result has been so far with POINTAPI described here https://groups.google.com/forum/#!topic/microsoft.public.word.vba.general/pKq4PsqD3cM, but it is not precise enough. If I use the coordinates from that VBA code to position the form I have two problems:

1. First of all, while the userform is active, the caret position is that of the active textbox on the userform, not on the document (the userform is not modal, but text input happens there)

2. There is no precision, with the form "overshooting" or "undershooting". I suspect the coordinate systems are different between what POINTAPI provides and what userform.Left and .Top properties use. I have noticed that the application.width/.height properties are significantly less than my screen resolution: 1272x770 versus 1650x1080. I tried adjusting the values for userform.Left/.top by taking a proportion between these measurements, which seems to have helped (almost) with horizontal placement, but not with vertical (still overshooting or undershooting by a variable distance, depending on where on the screen the caret is). Even if I get the precision, I still don't know how to link it to the caret position in the document versus the caret position in the active textbox.

There is also an option to use Selection.Information(wdHorizontalPositionRelativeToPage) I believe, but this only gives the twips coordinates from the edge of the page apparently, and I don't know how to convert it into coordinates for userform.

I have found a pixel to twips conversion routine at http://answers.microsoft.com/en-us/office/forum/office_2007-access/position-form-bottom-left-using-window-size/c8fef52f-d584-4eed-be8b-333d5329a214 but haven't figured out if i can use it for my purpose (theoretically I could, if I could extract the position of the page relative to screen, I suppose).

Basically, the question is this: how to place a user form exactly at the caret position?

Thanks,

Vit

ervit say: ok, one problem is solved: the coordinates for the userform have to be in points, so pixels need to be converted to points using the PixelsToPoints method, as described at https://msdn.microsoft.com/en-us/library/office/aa211293%28v=office.11%29.aspx. I have also figured out that window size is reduced by rulers, title bar and the ribbon, so I can compensate for those with hard-coded pixel values (various panes also affect the positioning, but I won't be looking for a fix for those - it's easier to temporarily hide them). Now, if only i could find a way to reference the caret from the doc while the userform is active.

ervit say:

Found the solution. Turned out very simple.

Dim pLeft As Long
Dim pTop As Long
Dim pWidth As Long
Dim pHeight As Long

        ActiveWindow.GetPoint pLeft, pTop, pWidth, pHeight, _

        userform.Left = PixelsToPoints(pLeft, False)
        userform.Top = PixelsToPoints(pTop, True)

Compilation error message

A protected module contains a compilation error. Because the error is in a protected module it cannot be displayed.

This error commonly occurs when code is incompatible with the version or architecture of this application (for example, code in a document targets 32-bit Microsoft Office applications but it is attempting to run on 64-bit Office).

This error has the following cause and solution:

Cause of the error:

  • The error is raised when a compilation error exists in the VBA code inside a protected (hidden) module. The specific compilation error is not exposed because the module is protected.

Possible solutions:

  • If you have access to the VBA code in the document or project, unprotect the module, and then run the code again to view the specific error.

  • If you do not have access to the VBA code in the document, then contact the document author to have the code in the hidden module updated.

jay freedman say:

A module that is hidden is usually in a template (*.dotm file) stored in the folder at %appdata%\Microsoft\Word\Startup. (Paste that expression into the address bar of the File Explorer and press Enter. It takes you to a hidden folder.)

A typical case is the template installed by the Avery Label Wizard, which contains that kind of compilation error. It's usually named AVWiz12s.dotm.

Move any templates you find in the Startup folder to some other place and restart Word. There should now be no error message. If there are two or more templates, you can try putting them back into Startup one at a time and restarting Word, until you find the one that causes the error.
