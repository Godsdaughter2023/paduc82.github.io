joeu2004 say:

David wrote:

the total gives an odd net figure of 0.0nnnnnnn even though all the amounts only have 2 decimals. Using various methods including trim, clean, etc. did not help. I even try manually entering the amount into another column and it still gives the same result. Can anyone help or is there a bug in 2013?

This is not considered to be defect.  However, it is a common issue with the way that Excel stores numbers and performs arithmetic, namely the native computer's 64-bit binary floating-point.  The issue arises because most decimal fractions cannot be represented exactly in 64-bit binary floating-point.

The best remedy is to explicitly round the sum formula (and any other arithmetic formula) to the precision that you expect.  For example, =ROUND(SUM(A1:A100),2).

Simply formatting the cell to display 2 decimal places rounds the appearance, but not the actual value.

Joeu2004, Thank you for the answer.

joeu2004 say:

David wrote:

The issue is that we need to load this file into another system which then checks if the total net to zero. As such, we need each amount of every record to be exact and using rounding for each field does not achieve the result either. 

I can understand that you might not want to explicitly round every formula (constants do not need to be rounded).  But I don't understand why you think it "does not achieve the result either".  Perhaps you can offer some examples.

PS....  You might also need to explicitly round some expressions.  For example, =IF(10.1-10=0.1,TRUE) returns FALSE(!).  You need to write =IF(ROUND(10.1-10,2)=0.1,TRUE).

Another option:  set the "Precision as displayed" calculation option (PAD), and format every appropriate cell as Number, Currency or Accounting with 2 decimal places.  The operative word is "appropriate".  Continue reading.

Caveat:  Make a copy of the Excel file before you set PAD.  And after you set PAD, review the entire Excel file (all worksheets) to ensure that there are no bad side-effects.

Normally, I dissuade people from setting PAD because its effect is pervasive.  And PAD can alter constants permanently.

For example, suppose you format a cell as Percentage with 2 percentage decimal places, but you enter an interest rate accurate to 4 percentage decimal places; e.g. 4.6875%.

When you set PAD, the constant is changed to 4.69%.  Not only will that adversely affect dependent interest calculations, but also the change is permanent.  If you try to correct the problem later by changing the format to 4 percentage decimal places, the constant will still be 4.6900%.

Moreover, PAD only affects the final value in a cell.  So explicit rounding is still required in the IF() example above.

In either case (use PAD or add explicit rounding), there is no point in grousing about it.  This is simply the way that Excel behaves, as do most computer applications.  Since the problem is important to you, you need to work around it.

scottgem say:
There are several issues here.

First, If you link an Excel sheet from an Access database the link is one way. You can't make changes in Access. What you have to do is make a data connection in Excel that connects to a table or query in your Access file. 

Second, the problem you are experiencing is due to the lookups. It is not recommended that you do lookups on the table level. That's what is causing you to THINK that the value is in your table when it isn't. For example, If you have a lookup table of Fruit:

ID  Fruit
1   Apple
2   Orange
3   Pear

What is generally stored is the ID number, NOT the description. But by using lookup fields on the table you are masking that.

What you need to do is link to a query that joins the main table with the lookup table to pull in the description and link to the query.
