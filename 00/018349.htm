tom van stiphout say:

VBA and most VB lookalikes have a tremendous amount of "implicit datatype coercion" built-in. This means the compiler will do its very best to figure out what you mean, converting data types as needed. This as opposed to C lookalikes like C# or C++ which frown on expressions like "abc" & 4.


You can think of variants as just another aspect of this philosophy: you don't have to worry if some variant is a number or a string or a date: it will all fall into place when your expression is evaluated. Variants also have a few other nice attributes that ONLY WHEN REALLY NEEDED can come in very handy: they can be null and empty (yes, these are two different things).


The conversion game comes at a small cost, so it would behoove you to not overdo it. For example I like to write

"abc" & Left$("def",1)

rather than

"abc" & Left("def",1)

I'm not sure the difference can be measured, but I find it more accurate and explicit.


But I also write:

dim x as integer
x=4
debug.print "x=" & x

I never use:

debug.print "x=" & CStr(x)

figuring the compiler already does that for me.


You'll figure out your own style. My recommendation would be to use the strengths of VBA to your advantage and let it do its thing.

Just one more bit of information...


This is a waste of time:

CStr("111")

CStr converts other data types to a string. Why convert a string to a string?

 

And a + operator will concatenate two strings unless one is null. Then you get a null instead of the non-null part. The + is used in very special circumstances. Most of the time you should use an ampersand as in:

CStr(111) & "1"

-or even-

CStr(111) & 1

 

Both will result in "1111" in VBA but in a query the result is different.

SELECT top 1 Cstr(111) +1 as test
from ErrorLog

returns 112 as adding a number to a numeric string results in a mathamatic operation.

eli qian msft say:

Hi lan,

For the situation, if we open files through Outlook directly, the files will be read-only mode. However, it will quit read-only mode when we save it as to a local path from Outlook.

The reason of defaulted read-only mode is caused that the files only temporary files format in Outlook and only be for viewing temporarily. Considering format of the attachments in Outlook is .temp, it means the files are temporary files and only for viewing temporarily. So if you would like to make modifications on them, you have to save them as to a local path in proper format like .docx for saving these changes. If you have more thoughts about this situation, I encourage and suggest you go to UserVoice to vote for Remove the edit lock on read-only files or to post a new feedback.

Besides, if the files are also locked locally, you could try changing the settings of files by following steps below:

1.Open Finder and navigate to the file on which you want to change permissions. Right-click its icon, and select "Get Info" to enter the file's Info window.

2.Check if the selection "Locked" in General is enabled. If so, disable it and the files will quit Read-only mode.

Regards,
Eli
