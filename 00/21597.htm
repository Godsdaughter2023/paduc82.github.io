ken sheridan say: I see three immediate problems with your current model:

1.  There should be no EmployeeLastname column in tblCaseInfo.  It should instead have an EmployeeID foreign key column of long integer data type, assuming the primary key of tblEmployees is a long integer number, most probably an autonumber.  For data entry of the foreign key EmployeeID column you'd use a combo box set up along these lines:

ControlSource:    EmployeeID

RowSource:     SELECT EmployeeID, FirstName & " " & LastName FROM Employees ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

2.  The relationship type from tbEmployees to tblCaseInfo should not be one-to-one.  As an employee can be assigned to more than one case, the relationship type should be one-to-many.  This would be possible with the keys as described above.

3.  If the relationship type from tblDocumentType to tblCaseInfo is one-to-many this would mean that each case can only have one document, but each document can apply to one or more cases.  That sounds unlikely to me.  I'd have expected the relationship type to either be a one-to-many relationship type from tblCaseInfo to tblDocumentType, i.e. tblDocumentType would contain a CaseID foreign key column, which would mean each document applies to one case only, but each case can have one or more documents; or, if each document can apply to one or more cases, and each case can have one or more documents, the relationship type would be many-to-many, in which case the relationship type would be modelled by a further table which resolves the many-to-many relationship type into two one-to-many relationship types as follows:

tblCaseDocuments
....CaseID   (FK)
....DocumentID   (FK)

The table might also contain other columns which represent other attributes of the case's relationship with the document.  With this latter scenario data input would be via a continuous subform based on the CaseDocuments table, with the subform linked to the parent form on CaseID and containing a combo box bound to DocumentID, with the other properties of the combo box set up similarly to those described above for EmployeeID so that the names of the documents would be listed for selection.

So if there is a one-to-many relationship type from tblCaseInfo to tblDocumentType the model would, diagrammatically, be like this:

tblEmployees----<tblCaseInfo----<tblDocumentType

If on the other hand it is many-to-many it would be like this:

tblEmployees----<tblCaseInfo----<tblCaseDocuments>----tblDocumentType

davidf2 say:

Publisher 2013 does seem to have more problems with large image files than other versions of Publisher. I don't pretend to understand why. I don't have a satisfactory answer for you but here are some thoughts.


1200 dpi is larger than you need. 300 dpi is generally plenty for commercial print purposes.


If you are optimizing the images before you insert them then make sure that you size them such that they can be displayed at 100% scale in Publisher. You can right click any individual image > format picture > Size tab > make sure the height and width are at 100% and that you always 'Lock aspect ratio'. If you resize the picture box to less than 100% scale than compressing the image within Publisher will resample the image to the box dimensions. And though you probably know this if you resize the box to more than 100% scale your picture quality will suffer.


Try using .PNG format for your images. I find that they scale better than .JPG or .GIF, support transparencies, and unlike .TIF files are smaller file size.

Here is another possible workaround to try: Go to View > check Graphics Manager. Go to the bottom of the GM pane and click 'Change Picture display...'. Right before you do your merge, check the option to 'Hide Pictures' and see if that helps.

I assume you are creating a PDF file of your merge for the printer? If so look within the PDF tool you are using to tweak settings that might help with the output. Or perhaps try a different PDF tool? I like the freebie primopdf...just be careful when you install it to opt out of the junkware and other optional stuff that comes with these free utilities. Always do a custom install and read carefully. I also like the free PDF creating tool that comes with the free version of the FoxIt pdf reader. I don't use the Adobe reader at all anymore.

Unfortunately there is not a good mechanism to give feedback to MSFT about these 'bugs' or design issues. The best we can do sometimes is just find workarounds, or if that is not enough then make the difficult decision to find other software that does the job for us. Try one of the freebies I suggested.
