Integrated Windows Authentication was configured in infrastructure. If the customer tried to login with a user, the credentials for logged-on user were automatically sent further. ADFS was giving the error because it was expecting a token for another user.

jeeped say:

Let's say that the Project Type input (e.g. Other) is a merged cell with M4 as the upper-left-cell and that the Notes input is another merged cell with I8 as the upper-left-cell. When Other is chosen from M4 you can push the user into I8 and not let them leave until there is something in I8.

Right-click the worksheet's name tab and choose View Code. When the VBE opens, paste the following into the pane titled something like Book1 - Sheet1 (Code),

Private Sub Worksheet_Change(ByVal Target As Range)
    If Target.Count > 1 Then Exit Sub
    If Not Intersect(Target, Range("M4")) Is Nothing Then
        On Error GoTo FallThrough
        Application.EnableEvents = False
        If Target.Address(0, 0) = "M4" And LCase(Target) = "other" Then
            Range("I8").Select
        End If
    End If
FallThrough:
    Application.EnableEvents = True
End Sub

Private Sub Worksheet_SelectionChange(ByVal Target As Range)
    If Target.Cells(1, 1).Address(0, 0) <> "M4" And LCase(Range("M4")) = "other" _
      And Not CBool(Len(Range("I8"))) Then
        MsgBox "Notes need a response when Project Type is Other."
        Application.EnableEvents = False
        Range("I8").Select
        Application.EnableEvents = True
    End If
End Sub

Tap Alt+Q to return to your worksheet. If Other is selected for the Project Type, then the only options will be to type something into the Notes or change the Project Type.

jllatham say:

Since your goal is to force some entry into I10 when M5 becomes "Other", you'd have to put some type of data validation in every possible cell on the sheet that they might select!  Probably impractical.

But there is a macro (vba code) solution.  The one below checks each time you select one or more cells on a sheet to see if M5 has "Other" in it AND if I10 is empty.  If that's the case then it forces the selected cell to become I10.  No matter where they click on the sheet (other than in M5), they are going to be forced to I10 for its entry.  Selecting M5 is still permitted so that if they didn't mean to choose "Other", they can change it and still leave I10 empty.

Putting the code to work is easy enough.  But just in case, work with a copy of your workbook until you are sure it works the way you want.

Open the (copy) workbook.  Choose the sheet this all has to work on.  Right-click on the sheet's name tab and choose [View Code] from the list that appears.  Copy the code below and paste it into the code module presented to you and close the VBA Editor by using its red-x or pressing [Alt]+[Q].

Give it a trial run.

If this is good for you, and others, save the workbook as a Macro Enabled workbook, type .xlsm or .xlsb.  If you don't do that then the code will be erased when you save it/close it as a regular .xlsx file.

You are also going to have to give notice to all users that for proper use of the workbook that macros must be enabled when opening the book.

Caveat: all of this can be circumvented by the simple act of NOT enabling macros when the workbook is opened.

Here is the code:

Private Sub Worksheet_SelectionChange(ByVal Target As Range)
  If Target.Address(False, False) = "M5" Then
    Exit Sub ' lets them make changes in M5
  End If
  If UCase(Trim(Range("M5"))) = "OTHER" _
   And IsEmpty(Range("I10")) Then
    'could add a message box here
    'MsgBox "You must provide proper note before continuing."
    Range("I10").Select
  End If
End Sub

jay freedman say:

No, you can't rename your .acl file to use with the AutoCorrect template. You would have to run the equivalent program for Word 2003 from http://www.word.mvps.org/FAQs/Customization/ExportAutocorrect.htm in order to get a backup document in the correct format.

You may be able to substitute your 2003 .acl file for the one created by Office 2016. I don't think the file format has changed, but I'm not sure of that. It's worth a try, though.

With all Office programs shut down, start File Explorer. Type or paste into the address bar the path

%appdata%\Microsoft\Office

(The "environment variable" %appdata% automatically expands to the hidden AppData folder in your Windows profile.)

In the Office folder, you should see a file named MSO1033.acl, which is the AutoCorrect file for English (US). If your default language is something else, in the following instructions you must substitute the one with the correct locale ID number.

Move the MSO1033.acl file to a different folder, so you can move it back again if the procedure doesn't work.

Now copy the old .acl file into the Office folder, using the same name as the file you moved out.

Start Word and/or another Office program and look in the AutoCorrect Options dialog. If you see the complete list of entries that you expect, you're in luck. Otherwise, bring the original Office 2016 .acl file back into the Office folder and start adding entries the hard way.
