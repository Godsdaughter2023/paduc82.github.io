ifix of lv say:

Where is the Outlook 2013 ost backup?

Is there another place the ost is stored? Offline cache mode is enabled so this is a real head scratcher.

roady [mvp] say:

Ost-files are not intended to be backed up and recovered. As you are connecting to an Exchange server, the emails can be recovered via the "Recover Deleted Items" feature of Exchange. If the retention time has expired (default is 14 days), then the mailbox needs to be restored from an Exchange database backup.

The reason why the ost-file is within the AppData folder is because it is only a cache and not treated as "original data". Only pst-files will be stored within the Documents folder.

The tmp-files are used to optimize the performance of the actual ost-file. They only contain data that hasn't been committed to the ost-file yet.

The fact that the ost-file itself doesn't show in his backups is either because ost-files have been excluded in general or because Outlook was running at the time of the backup and the backup application couldn't copy it because of the file-lock.

If you do find an ost-file backup, make sure you place Outlook in Offline Mode before replacing the ost-file or the content will be deleted again as soon as Outlook syncs with the Exchange server. Once recovered (check via OWA if the sync completed), rename the ost-file to .old when Outlook is closed and then start Outlook again to build a fresh cache again.

ken sheridan say: You might like to take a look at Prescriptions.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file is primarily intended to illustrate techniques for entering data into a set of related tables by means of forms/subforms, where it is necessary to insert multi-column foreign keys which reference multi-column primary keys in referenced tables.  It might serve as basis for what you need to do, however, if you change patients to students.  You wouldn't really need to change any of the tables for this, just the labels on the forms, in form design view.  To see the structure of the database take a look at the Relationships window from the Database Tools ribbon.

To generate a list of all medications to be administered to each student would just need the creation of the following simple query:

SELECT Patients.PatientID, Patients.FirstName, Patients.LastName,
PatientPrescriptions.DrugName, PatientPrescriptions.DrugForm,
PatientPrescriptions.Strength, PatientPrescriptions.Unit
FROM Patients INNER JOIN PatientPrescriptions
ON Patients.PatientID = PatientPrescriptions.PatientID;

which you can do by opening the query designer and switching to SQL view.  Then paste the above SQL statement into the SQL window in place of what's there by default.  You'd then be able to base a report on the query, for which you can use the built in report wizard.  Again you'd just need to change the labels in the report where necessary, which is easily done in report design view.

As you are relatively inexperienced in relational database design, you might also like to take a look at DatabaseBasics.zip in the same OneDrive folder.  As its name suggests this is a simple introduction to some of the basic principles and methodologies of relational database design.  Other introductory files which you might find useful once you've digested the DatabaseBasics demo are Normalization.zip and Boolean.zip

vincentchoy say:

When the partner signs up with Microsoft as a CSP partner, it is a term and condition that they handle customer billing AND customer first line support. Microsoft will provide tier 2 support to the partner.

From my personal understanding, the CSP license was designed for Microsoft Partners who bundle (or OEM) the license as part of a larger service they provide to the customer, under their own branding. However, there will be partners who simply sell it from a pure license perspective.

It is a license type that is ideal under certain use cases, for example when a customer outsources its desktop computing to a partner, where the partner provides the desktop infra and takes care of desktop support inclusive of Office365. The partner would still rely on the manufacturer for the repair of the desktop, and Microsoft for tier 2 support of O365, but they will handle the helpdesk and be client facing.

From my experience, the problem comes with customers who were on other licensing programs (that can raise direct support tickets), and gets sold a CSP license upon renewal, without understanding what a CSP license entails (loss of direct support tickets, assignment of delegated admin, monthly billing etc). 

You can always have a re-look at your CSP agreement to see if there is any way to move back to the other forms of licenses, or move at the end of the agreement period, since Office365 is a subscription based service.
