jaygourley say:

OneNote files in Windows File Structure

Will someone who understands how OneNote manages files within Windows and within OneDrive file structures please review what follows and make appropriate corrections?  Information below is my guesswork from poking around and is bound to contain errors.  There are many questions in this forum that seek this information, but I have not found any answers dated within the last few years.

OneNote keeps a separate file for each OneNote Section using the extension .one.  By default, copies on Windows and on OneDrive are kept synchronized by OneNote.

Locally, the Windows files are stored in a folder named after the OneNote Notebook that contains those sections.  The default location of that file is C:\Users\<username>\AppData\Local\Microsoft\OneNote\16\Backup\<NotebookName>\Sectionname+datestamp>.one

On OneDrive, copies of the OneNote Section files cannot be found by browsing OneDrive because they have complex web addresses that don't show up under the user's OneDrive file structure.  (One way to find the OneNote Section files in OneDrive on the web is to make a change locally within the section you're looking for, then synchronize the notebook, then browse to the OneDrive "recent" link.  The web location of the OneNote Section file you're looking for should be at or near the top of the list.)

I haven't found a way to control what notebooks are kept locally.  After a while a user may not want to keep local copies of old OneNote Notebooks, but he may want to keep archival copies on OneDrive.

Beware of the following possible sources of confusion:

  • The current local OneNote Section files are referred to in three different ways by OneNote, by OneNote options, and by Windows.  As explained above, OneNote creates a Windows file for each OneNote Section and puts it in a folder named for the OneNote Notebook.
  • However, in the Windows file structure, that local OneNote Notebook folder is under what is called a "Backup" folder (see default location above).  But these are not backups.  These are the current local copies of the OneNote Section files of the OneNote Notebook folders.  Deleting them will remove them from the local system and from OneDrive.
  • Under the OneNote options for "Save & Backup" this inappropriately named "Backup" folder is called the "Cache file location."
  • If you ask OneDrive to change the location of a OneNote Notebook to a location on your local OneDrive, it merely puts a web link in the specified destination folder.  This web link will be useless if you are offline.

I will ignore the case of local OneNote notebooks.

In the case of OneNote notebooks on Onedrive you have theoretically four places where you can see your notebook(s):

  1. On the OneDrive server you see, using your browser to access https://onedrive.live.com the master or original of a notebook.
    It's by default in the Documents folder. You don't see the sections. The notebook has another format than a copy stored on your computer.
  2. In the local OneDrive folder you see a link to the notebook, a .url-file. Its purpose is to prevent the direct access to OneNote files on OneDrive (other format !). So in the context of OneNote you should ignore this folder.

Changes you make in a notebook are at first written to that cache, later to the master on OneDrive, if accessable (and to the backup, depending on your settings).

Places 1 and 4 exist for all variants of OneNote on all platforms.

swischner say:

Caution! Very long post incoming.....

Local OneNote notebooks

This is the „old“ method and format as network or cloud storage was added later (with OneNote 2007 and 2010) but it does still exist in the recent OneNote 2016.
At a given location, by default under the users Document folder (can be set in the OneNote options of OneNote 2010, 2013 and 2016, Office versions only!) each notebook gets a folder with the same name as the notebook. Those folders can have three kinds of contents: .ONE files holding the sections of the notebook, folders for section groups (if there are any) and a open notebook.onetoc2 file which is a kind of index for the sorting of the sections in the notebook. If there are section group folders, they also contain .ONE files or more folders (for nested section groups). If any page or section has ever been deleted a special section group folder is also created for the recycle bin.

That file and folder structure may also live on a network share

Cloud stored notebooks

I am just talking about OneDrive, not OneDrive for business here because that one is slightly different but basically the same.

  • Even if there is a OneDrive client installed on your PC, you can not just copy or save the above folder and file structure to the local sync folder to create a cloud based notebook. It MAY seem to work at first sight but believe me… it won’t (I could list all the symptoms here that you would encounter but I’ll just leave it to that)
  • To create a notebook on OneDrive you HAVE to do it either in the OneNote client program or on the OneDrive- or OneNote-interface with your webbrowser. To move a formerly locally stored notebook to OneDrive DON’T copy any files or folders manually to your local OneDrive folder nor upload it to OneDrive using the browser. Won’t work! Use the option to share a notebook to OneDrive in your OneNote program instead.
  • In your local OneDrive folder AND on the web interface of OneDrive existing notebooks are only represented by a small link file. That one can be used for example to rename or delete an notebook. But it does not contain any OneNote data! (in facts it’s only a few bytes big).
  • The real OneNote data is hidden on OneDrive. It seems to be also organized in .ONE files (you CAN make them visible if you use your web browser and the OneDrive interface to display files by their size. This is a function you find behind the link about your used and remaining OneDrive capacity). But although you can make them visible with some tricks just leave them alone. You can’t do much meaningful things with them.

The backup folder

The desktop versions of OneNote for Windows (OneNote 2010, 2013, 2016) do have a basic internal backup function. This will create backups in given (selectable via Options / Save and Backup) intervals or started manually in a folder of your choice (also selectable in the options). Only notebooks that are currently open in OneNote (2010, 2013, 2016) are being backed up and only while OneNote is running. By default also only locally stored notebooks are backed up. To include notebooks in the backup that are stored on OneDrive (or OneDrive for Business or Sharepoint) you have to check the option “Also backup notebooks stored on Sharepoint”. The backup format is almost the same as the storage format for local notebooks (folders, .ONE-files…) with one exception: no index file (.onetoc2) is being created.

Those backups are not accessed by the sync mechanism, by editing notebooks or by anything else apart from OneNotes restore function in the file menu (again: Only desktop OneNote versions 2010-2016). They are simple copies.

The local cache

As Bernd P. already said there is another copy of every OneNote notebook which is currently being opened (loaded into OneNote). It’s format and location varies between the different OneNote versions but ALL versions (including OneNote for Android, iOS, Mac…) are using such a copy. It’s the OneNote cache. If you open a notebook a cache folder and files are created for them on your local device in a “hidden” place. All edits that you make to the notebook are applied to that cache copy first. The sync mechanism of OneNote (NOT of your OneDrive app; OneNote is using it’s very own sync!) is then applying the changes to the “real” save location, be it a local folder or a cloud location like OneDrive.

As soon as you close a notebook (or some other circumstances like resetting the OneNote app on a mobile device) that cache is being deleted and lost … and so are your edits as long as the cache has not been synchronized to OneDrive or the local “real” location of the OneDrive files first.

The most important information (in my opinion):

Although OneNote is using a kind of file and folder structure and not a real database it’s best to think of a database. Why? Because you would never get the idea to copy, move, backup, rename or otherwise handle database data (like in Evernote or a CMS like Wordpress for example) simply by pure file operations.

In fact OneNote is just “pretending” to work on files like other office programs but things are much more complicated. The reason is that OneNote allows concurrent edits of the same notebook, section, even page by several devices or users at the same time without having to lock and unlock files first. This is not possible with any file based program like Word or Excel, right? So the sync mechanism is doing a complicate file splitting and partial locking. I am not quite sure how that works on local notebooks as they seem to be standard files inside the windows file system (I think it’s handled by the sync and cache mechanism as cache files are only fragments of data).

On OneDrive a special sync protocol named Cobalt or MS-FSSHTTP is used which is the same as in Sharepoint. That one is based on splitting actual files in small chunks of data as well. This is one reason why OneNote files are hidden away and you can't just move files around or copy them into the local OneDrive folder.

To make the file splitting thing a bit clearer: If it would be that easy that just all section content is stored in a ONE file and that file would be synced to OneDrive using the local OneDrive client and folder like it's done with word files, this is what would happen:

  • Even the smallest change to a notebook page would cause the whole ONE file to be uploaded to OneDrive to replace the old version.
  • As OneNote is saving changes automatically this would happen every few seconds producing an enormous amount of bandwith usage and make things really, really slow.
  • If two persons would edit any page in the the same section roughly at the same time, two different copies of that ONE file would be uploaded causing a synchronisation conflict. Which one is to keep? So for security reasons both copies would be kept and one of them renamed, creating lots of duplicate content. By the way... this is one of the effects likely to happen when you try to copy ONE-files to your local OneDrive folder.

So the sync mechanism has to handle those files completely different, only updating changes INSIDE the files and merging those changes to the version on OneDrive. 

End of the longest post I ever made here…..i hope I did not just add to the confusion.

I already published several articles on topics like backups, file handling and OneNote cloud storage on my blog but unfortunately in German language only.

Maybe this is where you are wrong:

I assume you think that OneNote creates and stores notebooks locally on your device and then synchronizes them to a version stored in the cloud / on OneDrive. This is wrong.

There is always only ONE place where a notebook lives. It's EITHER locally (and this is only possible to use and access with an Office OneNote 2016 or earlier) OR on OneDrive.

At time of creation of the notebook you decide the location of that notebook (again only with Office OneNote 2016 as the other versions do only have the cloud option).

This all has NOTHING to do with the backup folder of OneNote 2016!

Perhaps you didn't understand that the suggestion to run without add-ons was intended as a diagnostic to try to isolate the possible cause of a problem symptom to just the set of all add-ons without having to modify any of them?

Anyway, now that you know it changes your problem symptom you can stop using that tool and start trying to figure out which add-on(s) are the ones that you want to avoid.  To do that you can use the Manage Add-ons tool to disable all add-ons individually and start re-enabling and testing them one by one.   However, there is a more efficient testing technique available if you can manage the extra complexity of it.   Re-enable an arbitrary half of the disabled set and re-test with that combination.  If your problem symptom doesn't reappear with a test that previously reliably would reproduce it it may be safe to assume that none of those add-ons was a culprit.  Therefore you wouldn't need to test any of them individually.   So, then you can divide the remaining half in half and re-enable those and re-test.  Eventually a set that you have re-enabled may contain a problem add-on, so then presumably your problem symptom would reproduce, so then you have to re-disable half of those and re-test to try to find another large set to eliminate from consideration and leave enabled.  If your assumption was that you only had one bad add-on to find it would actually be sufficient then to not bother re-testing the other half but I think that I would re-test both just to avoid relying on that assumption of there being only one problem add-on to find.  You can keep going like that until you have reduced the test sets to one or two and thus quickly isolate a problem add-on.

Another useful thing to be aware of is that both of the preceding test procedures have an implicit assumption which if  false could save you a lot of work.   The assumption is that all those add-ons have equal value to you and that you care whether they are re-enabled.  Often that assumption is indeed false, so in that case you could perhaps leave many of the add-ons disabled and only re-enable for your testing ones that you knew you needed or wanted.  E.g. you could always re-enable some only when they are needed again and then be much more aware of new problems symptoms and being able to associate them with that change.
