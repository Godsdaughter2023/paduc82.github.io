And it is.  Excel simply does not display it.  Excel only displays up to 15 significant decimal digits; it simply appends zeros for any additional digits.

However, Excel writes up to 17 significant digits into the XML file because that is the most significant decimal digits that are needed to convert to and from the binary representation accurately, according to the IEEE 754 standard.

Let's review....

Numbers are stored as a sum of 53 consecutive powers of 2 ("bits") times an exponential factor.  Consequently, most non-integers cannot be represented exactly.

For example, 9.8 is 2^3 + 0*2^2 + 0*2^1 + 2^0 + 1/2^1 + 1/2^2 + 0/2^3 + ... + 1/2^48 + 0/2^49.  The exact value is 9.80000000000000,0710542735760100185871124267578125.

(I use period for the decimal place and comma to demarcate the first 15 significant digits.)

It is difficult to prove that 9.8 is really 9.80000000000000,07... because 9.8 is always converted to the correct binary sum internally.

But consider this....

Enter =9.8+2^-49 into A1.  Excel still displays 9.80000000000000.  But the exact value is 9.80000000000000,24868995751603506505489349365234375.

Enter =A1-9.8-0 into A2.  (I'll explain the -0 below.)  Excel displays 1.78E-15.  Enter =2^-49 into A3.  Excel displays 1.77636E-15, which is indeed about 1.78E-15.  In fact, =A2-A3=0 displays TRUE.

The point is:  yes, there are often more decimal digits than Excel displays.  That is often why certain comparisons (notably lookups) do not behave as we expect.  And that is why it is so important to explicitly round arithmetic calculations that you expect to be accurate to a specific number of decimal places.

-----

To add to the confusion, Excel tries to hide these infinitesimal differences; but only under very limited conditions.  This causes a number of inconsistencies.

For example, in the experiment above, if we enter =A1-9.8 (without -0) into A2, the result is exactly zero (0.00E+00).  In that specific condition, Excel arbitrarily decides that the difference is "close enough" to zero, and it substitutes zero for the exact binary arithmetic result.

In contrast, =SUM(A1,-9.8) returns about 1.78E-15.  Likewise, =A1-9.8-0 returns about 1.78E-15 because subtracting zero does not fit the very specific (and arbitrary) conditions for which Excel hides the infinitesimal diffference.

Similarly, in the experiment above, =A1=A2 returns TRUE (a "lie"), but =A1-A2=0 returns FALSE, which is correct.  Moreover, =MATCH(A1,A2,0) returns #N/A, indicating that indeed they are not the same.

The "=" operator compares only the first 15 significant digits (rounding the 16th).  By the way, that is a different criterion than the condition when Excel might substitute exact zero for =A1-A2.  So there are conditions when A1=A2 might return TRUE, but =A1-A2 displays an infinitesimal difference(!).

(Example:  A1 is 0.1234.  A2 is =A1-8*2^-56.  =A1=A2 returns TRUE.  But =A1-A2 returns about 1.11E-16.)

In contrast, lookup functions like MATCH always compare the exact internal binary representation.
