Is the issue reproducible every time, most of the time, some of the time and just on this one system?

The file that gets created when you hibernate (hiberfil.sys) is supposed to be some snapshot of your RAM and one consequence of adding more RAM is the size of the hiberfil.sys will get larger and consequently take longer to load when you resume.  At some point it might take longer to come out of hibernation than a cold boot.

Perhaps somehow Windows is not handling the increase in RAM properly when creating/updating the hiberfil.sys file so let's just create a new one.

Open a Command Prompt as Administrator and issue the following command which will disable hibernation and delete the current and potentially afflicted hiberfil.sys:

powercfg  -h  off

Then restart your system and enter this command to enable hibernation again and create a fresh hiberfil.sys file:

powercfg  -h  on

Then restart your system and test your failure condition and see how that works.

Solution

Problem:  Hibernation suddenly fails to execute and standard methodologies do not restore its functionality.

 

Solution:  Create a new user profile and enable the hibernation sub system with it/from it.

 

  1. In any and all user accounts, Disable Hibernation via Powercfg /hibernate off
  2. Create a brand new user account with admin privileges
  3. Reboot
  4. Log into new account. (If auto login is enabled, log off from the active profile and log into the newly created account)
  5. Enable Hibernate from that new account via Powercfg /hibernate on.
  6. Reboot
  7. Back in the main user account/profile, Hibernation should now be operational.

 

Conclusion: It is unclear as to why hibernation failed without notice for a user profile that has long been used and seemingly in a perfect operational state. 

However, its seems that the issue is linked to a user profile and results in the inability to save the session to the hiberfil.sys - for whatever reason(s).
