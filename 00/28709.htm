rich michaels - mvp (greatcirclelearning.com) say:

Here's what you could do ...

  • In a new blank document insert each image.
  • Add the image name as Alternative Text to each image.
  • Save each image into your Normal template as AutoText, giving the AutoText Name the exact spelling you used in the Alternative Text for each image.

Now when you put these images into a document, always use the AutoText version because it will contain the Alternative Text tag and then you can use the following for your Find and Replacing.

Sub SwapPictures()
    Dim iSplit() As String, iName As String, Loc As Long
    Dim a As word.AutoTextEntry, found As Boolean
    iName = InputBox("Enter name of the note to find, then a comma followed " & _
    "by the name of the replacement" & vbCr & vbCr & "Example: Find Name,Replacmeent Name", "Find and Replace")
    If iName = vbNullString Then Exit Sub
    iSplit = Split(iName, ",")
    
    For Each a In NormalTemplate.AutoTextEntries
        If a.Name = iSplit(1) Then
            found = True
            Exit For
        End If
    Next
    If found = False Then
        MsgBox "Invalid Replacement Name", vbExclamation, "Find and Replace"
        Exit Sub
    End If
        
    Loc = Selection.Range.Start
    Application.ScreenUpdating = False
    Selection.HomeKey wdStory
    On Error GoTo errHandler

    With Selection.Find
        .ClearFormatting
        .Format = False
        .Forward = True
        .Text = "^g"
        .Wrap = wdFindStop
        .Execute
        Do While .found
            Debug.Print Selection.Range.InlineShapes(1).AlternativeText
            If InStr(1, Selection.Range.InlineShapes(1).AlternativeText, iSplit(0)) > 0 Then
                Selection.Delete
                NormalTemplate.AutoTextEntries(iSplit(1)).Insert _
                    Where:=Selection.Range, RichText:=True
            End If
            .Execute
        Loop
    End With
errHandler:
    Selection.SetRange Loc, Loc
    Application.ScreenUpdating = True

End Sub

ken sheridan say: The balances per account should not be saved in a column in a table.  This leaves the database wide open to update anomalies, and as Murphy's Law infallibly states, 'If something can go wrong, it will go wrong.'  The balances can be computed from the transaction amounts.

Computing balances is pretty standard.  Essentially the debit balance following each transaction is the sum of all debits less the sum of all credits per account up to and including the transaction.  The current balance is that following the latest transaction.  There are various ways of computing balances, of which the most efficient is by means of a JOIN of two instances of the transactions (appointments in your case) table.  Less efficient, is the use of a correlated subquery to return the balance.  Neither of these return an updatable result table, however, so if the query is to be used as the basis for an editable form it is necessary to resort to the VBA DSum function, which is the least efficient of all.

You'll find examples in Balances.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates how to compute balances in a variety of scenarios.  Note that the queries bring the TransactionID primary key into play to act as the tie breaker in the event of two or more transactions occurring on the same day.  This might not be necessary in your case if there can be one appointment only per account in any one day.

ken sheridan say: The Requery method of the combo box should be called to reload the recordset for the combo box's drop down list, not the form's Refresh method.

If a single payment can be made in relation to multiple appointments then payments should be modelled by a table which references Clients, i.e. it should have a PaymentID primary key column and a ClientID foreign key column, along with the non-key columns such as the payment date and amount.  The Appointments table should, in addition to a ClientID foreign key column, also have a PaymentID foreign key column.

I'm not clear from your last post whether you are recording the balances in a column in a table or not.  You should not do so.  Whether payments are in cash or by other means is irrelevant to the computation of balances.  Provided that each payment is recorded in a row in the payments table balances can be computed.  My online demo, to which I referred you earlier has examples of queries for simple credit and debit accounts.  In the case of a loan account for instance you'd also have to take account of interest accrued.
