Assuming you are (as I do suspect) certainly infected by the latest and most current version (4.0) of the TeslaCrypt (AKA Tescrypt - Microsoft's detection) ransomware, which has - unfortunately - encrypted your data, please note that the newest TeslaCrypt 4.0 no longer uses an extension appended to the original file names/extensions that have been encrypted, although when first released it used the .mp3 extension as in their prior version 3 (see this related thread/post).

These ransom notes (.txt files) may be located in every folder where files have been encrypted, and they only contain the information for the 'ransom payment' - action which is, of course, NOT recommended!
You should also see an .html file with same or similar name and characteristics and containing same info.

More information in this BC News article: TeslaCrypt 4.0 Released with Bug Fixes and Stops Adding Extensions.

Most security experts will advise against paying the ransom demands of the malware writers because doing so only helps to finance their criminal enterprise and keep them in business. The more people pay the ransom, the more the attackers are encouraged to keep creating ransomware for financial gain.
In addition, there is no guarantee that paying the ransom will actually result in the restoration (decryption) of your files. Remember that you are dealing with cybercriminals.

On the other hand, however, unfortunately - at this time - there is no way of decrypting TeslaCrypt V4.0 (or V3) unless you pay the ransom to obtain your private key, since both these improved versions use a different protection/key exchange algorithm, a different method of key storage, and the key for them cannot (yet) be recovered using the TeslaDecoder tool.

BloodDolly, the author of the TeslaDecoder tool, and other experts at bleepingcomputer.com (BC), are certainly working on this already to find a solution.
If infected by the new TeslaCrypt 4.0, victims should backup all their encrypted files and wait for a future solution.

Currently, there is no way of really marking how possible it is or how long it might take, but the 'good guys' at BC are certainly trying hard.

Please note that, as with most ransomware infections, the best solution for dealing with encrypted data is to restore from backups.

TeslaCrypt shall delete all Shadow Volume Copies with vssadmin.exe (see: Why Everyone Should disable VSSAdmin.exe Now!) so that you cannot restore your files via System Restore or using a program like Shadow Explorer... but it never hurts to try, so as to check in case the ransomware failed to do everything it's supposed to do.

While Restoring from backups may be your only choice at this point ... unless you get lucky with recovery software such as Recuva, ShadowExplorer, R-Studio, or PhotoRec - always at least worth a try ... please note that the encrypted files themselves aren't a threat, since they do not hold the malicious encrypting executable in them. They are just encrypted.

So if you have unsuccessfully tried recovery software suggested above to restore your files, or couldn't recover them all from a recent - prior to infection - external backup (as it's the first and best method to restore your data), or if there wasn't any at all; it is always recommended to backup the encrypted files and save them on an external storage media (like an external HDD) 'as-they-are' and leave them be in hopes of a solution in the future (if one is ever found).

Since all of the public keys and other encryption-related info are stored in the header of every encrypted file (Tesla leaves nothing else of use behind), once you have the encrypted files backed up by whatever means, you are free to reload the OS if you wish. If there are any break-throughs, the backup of the encrypted files is all that is necessary for decryption when/if the expected free solution is released.

pamcaswell say:

In the text, select the level 8 numbers and press Ctrl+spacebar to clear direct character formatting. 

If the bolding goes away, then bold must have been specified in the  numbering dialog. Because the bold of the paragraph style toggles the  bold in the numbering (that is, turns it off), someone must have applied  bold as direct formatting to the numbers. The TOC feature picks up some direct character formatting (which is often not just embellishment), which is why the numbers are bold in the TOC.

The fix is to go into the define new multilevel list or modify list style dialog—depending on which type of list you have—and for number level 8, click the Font button,  and clear the input box for Font style.  Do the same for level 9.

Let us know if this works.

Pam
