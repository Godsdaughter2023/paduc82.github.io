Calculated values should be done in queries or controls. Let say, for example, that you current value uses the formula: Purchase price * (1- (% * Years))  so each year the value decreases by 5%. So your expression might look like this:

=[PurchasePrice] * (1-(.05*DateDiff("yyyy",[PurchaseDate],Date())))

You could use that expression anywhere you want to display the current value.

scottgem say:

My blog address is in my signature below. once there you will see a list of articles. 

My asset database does the same, There are 2 ways to handle that. You start off with your Asset table with information that is common to every asset. This holds ALL assets with a field to indicate category. Then you can go with a more normalized structure and have a properties table like so:

tblAssetProperty

AssetPropertyID (PK Autonumber)

AssetID (FK)

PropertyID (FK)

Detail

with a lookup table:

tluAssetProperty

PropertyID (PK Autonumber)

Property

So for example, you might have a monitor. Monitors have a screen size property that not all asset types would have. So you have a record in tblAssetProperty that has a PropertyID indicating Screen size and the detail is the size. (note, this property could also apply to cellphones). 

The other way would be to have type tables with a one to one relationship to the Asset. These type tables would have fields that are specific to the asset type. So there are only records in each type table for assets of that type.

daniel g. grau say:
It won't be easy, because those automatic connecting lines are obtained through a mixture of rectangles (yes, but zero width), lines and elbow connectors (some are L shaped) that are grouped, but cannot be ungrouped or manually deleted.

You need to variously click close to the connecting lines to select the shape that created it. It can be disconcerting, as you will click on a line and see a square. Then, edit the shape by changing the line color to match the background, so it will disappear from view.

Finally, to add new connectors, do it manually with lines or elbows. In this case you'll need to match the line weight (usually .75 points) and color of the rest of the lines, and perhaps cancel its shadow and glow options as well. Aligning the lines can be hard, even with "Snap to grid" and "Snap to shape" selected, so you'll need to zoom in and move your manually added shapes just a tiny bit using the cursor arrows (press "Control" for fine adjustment).

SmartArt works OK as long as you don't try to move or add more than one or two new shapes using the built-in tools (i.e., clicking on the "notebook shaped" icon at the top left of the SmartArt picture). Otherwise, they won't fit and will become seriously misaligned, and connecting lines will no longer overlap.

jeeped say:

when using vlookup with "false", it always returns with #NA even lookup value is equal to col_index in the same format ...


Your description of the situation (e.g. "... lookup value is equal to col_index in the same format") leads me to believe that you are not setting up the VLOOKUP() formula correctly.

                

Let's assume a grid of data (as above) populated with a simple =ROW()*COLUMN() formula in A1:D5. The syntax of the VLOOKUP() function is:

=VLOOKUP(lookup_value, table_array, col_index_num, [range_lookup])

If we use A1:D5 as the table_array, 3 as the lookup_value, 2 as the col_index_num and FALSE as the optional range_lookup like this:

=VLOOKUP(3, A1:D5,2,FALSE)

then VLOOKUP() will return 6 because 6 is in the 2nd column (e.g. col_index_num=2) of the data grid (e.g. table_array=A1:D5) where the first column matches 3 (e.g. lookup_value=3). If we change the lookup_value to 5 and the col_index_num to 3 then VLOOKUP(5, A1:D5,3,FALSE) will return 15.

... when using vlookup without "false", it works but return will not exactly the same.


If your primary column (i.e. the first in the table_array) is not sorted, expect unreliable results when the range_lookup is TRUE or omitted (defaulted to TRUE).

It is possible that I've misinterpreted your problem and that it is simply a case of leading and/or trailing spaces or non-breaking spaces masquerading as normal spaces, but I think you should have a close look at the construct of your VLOOKUP() formula first. If you are trying to lookup a value in a column and return a value from the left of the lookup column, you should switch to an INDEX(MATCH()) pair. VLOOKUP() always looks up in the first column and returns a value in the first column or to the right depending upon the col_index_num. In other words, the col_index_num cannot be a negative value.
