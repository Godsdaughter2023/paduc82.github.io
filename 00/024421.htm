Outlook's autodiscover feature has a major flaw - Microsoft Community

dlong500 say:

Outlook's autodiscover feature has a major flaw

This has been bugging me for years, and I assumed Microsoft would fix it in newer Outlook releases, but it still acts the same way even in Outlook 2013.

The autodiscover feature, which allows auto-configuration of Outlook settings based on just an email address and password, is a great feature but has been implemented incorrectly.  The problem lies in the order of the autodiscover requests.

The current order of autodiscover methods that Outlook uses is the following:
SCP lookup
HTTPS root domain query
HTTPS Autodiscover domain query
HTTP redirect method
SRV record query
Local XML file
cached URL in the Outlook profile (new for Outlook 2013)

The problem is specifically with the two HTTPS (SSL) methods being early in the order.  Let's look at a simple example that is going to be very common these days: a hosted Exchange scenario.  The user has an email address of *** Email address is removed for privacy ***, but the exchange server is exchangehoster.com.

The first method (SCP) doesn't apply because this is not a domain connected scenario.  No problem, because the method will silently fail and move on to the next method.  Now it tries the "HTTPS root domain query", so it looks for the following URL: https://business.com/autodiscover/autodiscover.xml.  But wait, the user's website (like most sites out there) is on a shared host without a dedicated IP address and the SSL certificate is for websitehoster.com.  And so despite the fact that no autodiscover information exists at that URL it will still give an "invalid certificate" message every time the autodiscover service is checked.  Because it is not a one-time only request at setup, but continually checked for updates, this is a big problem.

The second method, "HTTPS Autodiscover domain query", has the exact same problem when it tries to query https://autodiscover.business.com/autodiscover/autodiscover.xml.

The fix would be extremely simple, and the fact that it hasn't already been implemented is very puzzling and annoying.  All that is needed is to simply put the HTTPS methods behind the "HTTP redirect" and "SRV record query" methods.  That way the user could implement either of those methods by adding the proper DNS records and avoid the SSL mismatch mess.

Yes, the issue can be worked around on a per machine basis by registry hacks, but that is ridiculous and clunky especially given how easy the fix would be.  Can anybody tell me why this was implemented this way to begin with?  How did Microsoft not see the widespread problem it would cause with hosted Exchange accounts.  Getting a certificate for your website could also avoid the error, but involves not only purchasing a certificate on a permanent basis but also upgrading to a web hosting account that supports installing certificates (which still usually involves needing a dedicated IP address even with SNI starting to make headway).

jay freedman say:

Right-click the Quick Access Toolbar and click "Customize Quick Access Toolbar".

In the dialog, open the "Choose commands from" dropdown and click "Developer Tab".

In the list below the dropdown, click the Lock item. Then click the Add button between the two list boxes. You can use the up and down arrow buttons on the right side of the dialog to move the item within the list of toolbar buttons. Finally, click OK.

The Lock button is a toggle (on/off). It's specifically for the "Filling in forms" type of protection, not for the others that are available in the Restrict Editing pane.
