Hi KW

the command is available in Word Options/Customize/Customize button (for keyboard shortcuts). Set the category to "All commands" then locate EditCitation in the list of commands. Click on it. Then click in "Press new shortcut key".  Press the key combination you prefer and check whether it's currently assigned. Note: This will be saved in a document, probably Normal.dotm. Normal.dotm means it will always be available, but the assignment could get lost if Normal.dotm ever has to be replaced.     gustavbrock say:

    First: The title should probably read:

    DateDiff omitting one calendar month in year/month calculation

    Second: Your expression is correct in the sense that it ignores the day of each date, only counting the difference in calendar months. Thus, for example, any day for a start date of Dec. 2000 and any day for an end date of Nov. 2005 will return:

    4 yrs 11 mnths

    So, your issue has nothing to do with the day of the month, time values, nor input formats.

    If the count you want is not the difference in calendar months but the count of full months, you can use this expression:

    =Months([LeaseStartDate],[LeaseEndDate])\12 & " yrs" & " " & Months([LeaseStartDate],[LeaseEndDate]) Mod 12 & " mnths"

    ken sheridan say:

    I store the the LeaseStart and LeaseEnd as dd-mmm-yyyy.

    Actually you don't.  A value of date/time data type in Access is stored as a 64 bit floating point number in which the integer part represents the number of days since 30 December 1899 00:00:00, and the fractional part represents the time of day.  The format in which you enter and see the date merely reflects the regional date/time format.  I would enter noon on 4th July 2019 as 04/07/2019 12:00:00.  In the USA it would be entered as 07/04/2019 12:00:00.  In either case it would be stored as 43650.5.  To avoid ambiguity I normally use the ISO standard for date/time notation of YYYY-MM-DD hh:mm:ss when citing a date here.  The standard uses case do differentiate months and minutes BTW.

    The implementation of date/time data in this way is why there cannot be such a thing as a date value in Access.  the value 43650 would not define 4th July 2019 to a precision of one day, but 4th July 2019 00:00:00 to a precision of one second.  Because in day to day life we think of a date as defining the whole day, however, it is conventional to enter a date into a column in a table with that intention.  Access does not see the value in the same way, however, so when processing date/time data we often have to allow for this.  For instance if we restricted a query with:

    WHERE TransactionDate BETWEEN #2019-01-01# AND #2019-01-31#

    this would not work if any transactions on 2019-01-31 had a value with a non-zero time of day, i.e. with a non zero value after the decimal point in the number as which the date/time is stored.  We can only guarantee this if such values have been specifically prohibited in the table definition.  Otherwise, to be confident of returning all transactions in the month, we'd define the one month range like this:

    WHERE TransactionDate >= #2019-01-01# AND TransactionDate  < #2019-02-01#

    If, as would be more usual, the dates are parameters then the range would be defined as TransactionDate >= [StartDateParameter] AND TransactionDate < [EndDateParameter]+1.

    3. Check the box of the email that you want to archive and click Archive located in between Delete and Junk.
