It is not uncommon for some security scans to take many hours while others finish quickly.

The speed and ability to complete an anti-virus or anti-malware scan depends on a variety of factors.
  • The program itself and how its scanning engine is designed to scan: using a signature database vs heuristic scanning or a combination of both...see How an Anti-virus Program Works.
  • Options to scan for rootkits, adware, riskware and potentially unwanted programs (PUPS).
  • Options to scan memory, boot sectors, registry and alternate data streams (ADS).
  • Type of scan performed: Deep, Quick or Custom scanning.
  • What action has to be performed when malware is detected.
  • A computer's hard drive size.
  • Disk size and used capacity (number of files that have to be scanned).
  • Types of files (.exe, .dll, .sys, .cab, archived, compressed, packed, email, etc) that are scanned.
  • Whether external drives are included in the scan.
  • Competition for and utilization of system/CPU resources by the scanner.
  • Other running processes and programs in the background.
  • Whether the scanning engine stalls, hangs or freezes.
  • Interference from malware. Addison,

Keep in mind that external links are schedule based and not cost based. Therefore baseline dates WILL be reflected in linked structures but baseline cost will NOT. For example, in the screen shot below you can see the baseline dates at summary level reflect the impact of the external task links but the baseline cost at summary level only reflects costs for the active plan, even though the Baseline Cost field shows the external task baseline data.

If the add-in is attempting to add up all the data in the baseline cost field, it will not agree with the real summary line baseline cost and hence an error will be flagged. This is something you need to confirm with forProject.

When using external task links, here's a quick test to see if you have rogue link fragments (i.e. corruption). If you don't already have a dynamic master with ALL linked subproject inserted, create a temporary dynamic master. If you see any ghost tasks in the master then you definitely have a corrupt linked structure. However, just because you do not see any ghost tasks at master level, that is not a guarantee that there is no corruption.

Hi!

I'm sorry I didn't reply earlier. We managed to resolve the issue.

If it helps anyone else: The problem was in the way the calendars were "hierarchialy" organized. Our customer apparently managed to open some of the calendars as "subcalendars" of others already in existance. This "hierarchy" didn't have any effect on the calendars themselves and was in fact invisible when managing the calendars by online Outlook web app (all the calendars there were just listed independently of their "level"), but only the top "root" level of calendars was publishable.

The solution was to manage calendars in desktop Outlook, where their position was visible when trying to use "move" option on any calendar. There the hierarchy was clearly visible and I was able to move all the "subcalendars" to the top level. Then the calendars could be published in online management page and they could be viewed normally.
