suzanne s. barnhill say:

The easiest solution is to create a new document based on the template (in this case Normal.dotm) containing the desired styles and formatting and then paste in the content of the old document, omitting the final paragraph mark. If the old document contains more than one section, you will need to copy/paste a single section at a time, without the section break.

If the old documents contain styles with the same names but different formatting, you can reattach the template with "Automatically update document styles" enabled. Be sure to clear that check box after updating the styles once, or you won't be able to make any manual changes to styles/paragraphs in the document.

stefan blom say:

Styles are not "global" in Word; they are stored in documents and templates, and they are inherited from a document template at the time you create a document based on the template. For more on templates in Word, see https://shaunakelly.com/word/templates/templaterelations.html.

Attaching, or re-attaching, a template to a document and checking the box for "Automatically update document styles" would be a way to force new styles in a template into a document. For more on this approach, have a look at Suzanne's reply.

The alternative is to make use of macros. Basically, you could create an auto macro that adds a particular style, or a particular set of styles, to all documents that you open. You will have to decide if this is really something that you want to do. See also Charles's links.

scottgem say:

First, Don't accept the default ID name for PKs. I always name my PK tablenameID, so your ID field should be CategoryID. Also the corresponding Foreing Key should be named the same, So your Main Inventory fields should be CategoryID and ComponentID. Another tip is to not use spaces in object names. Use Camel Notation (MainInventory) or underscores (Main_Inventory).

I agree with Sandro. You should not be using separate tables for each component category.  You should have one Component table with CategoryID as a foreign key. And therefore CategoryID should NOT be in your Main Inventory table.

What you are asking about is called cascading or synchronized comboboxes. How to use them is well documented if you search on that phrase. 

ken sheridan say: You'll find examples of the use of correlated combo boxes in various contexts in ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file uses the local administrative areas of county, district and civil parish in my neck of the woods as an example, but the methodology applies to any hierarchically structured set of entity types.  Note how the Locations table includes a foreign key referencing only the lowest level of the hierarchy, ParishID in my case.  This illustrates the point made by Scott about not having a CategoryID column in your inventory table.  To do so would introduce redundancy die to the transitive dependency of category on the key via component, resulting in the table not being normalized to Third Normal Form, and consequently open to the risk of update anomalies.
