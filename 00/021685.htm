Hi Herb,

Q: "Is this change to identify the new name of the ADFS server since the domain primary DNS suffix of that server will change?"
A: Yes.

Q: "If the ADFS server is still available at the original DNS name, is the first part necessary?  Where is this done?"
A: It is necessary. 
    In AD FS Management console.

Q: "If so, what is difference between "Convert" and "Update" and which should be preferred? "
A: They are used in different scenarios.
In this article, the goal is to set up a second federated domain on the existing AD FS server, so at step 4, you need to convert additional federated domains using "Convert-MSOLDomainToFederated".
But in this article, if the AD FS configurations has some problem or change the settings, you need to use "Update-MSOLFederatedDomain". Moreover, the Set-MsolADFSContext cmdlet sets the credentials to connect to and to the Active Directory Federation Services (AD FS) server. If you directly run PowerShell command on the AD FS server, you don't run this command "Set-MSOLADFSContext –Computer:<AD FS 2.0 ServerName>".

Finally, since the steps are complicated, it is recommended that you re-configure AD FS function.

In order for the E71 to be able to use the nuevasync.com service with the built-in SHA1 certificates you have to connect through the insecure.nuevasync.com URL

kev sherratt say:

Hi,

I've been suffering from the same issue for the last week or so connecting to a simple IMAP4 mail server and it seemed to connect, download one email and then crash with the exact same error you reported.  I may have just found a solution - at least for me.  Be warned though that this might cause data loss if anything in your files is not sync'd with hotmail so do this at your own risk, but if like me you rely extensively on your email it's worth a shot.

Edit: By doing this, it looks like I actually lost at least 2 emails that had evidently failed to send.  I'm wondering if they were the cause somehow.

I opened Windows Explorer and found two Outlook data files with the .ost extension in

  "%USERPROFILE%\AppData\Local\Microsoft\Outlook"

The files are likely named after your email address (eg, *** Email address is removed for privacy ***) so should be easy to spot.  I moved these out of this folder to a safe place (in case something went wrong) and restarted Outlook - which recreated new files in their place and has been happily sat for the last 30 mins instead of crashing almost immediately.

If it still doesn't work, you should be able to copy the .ost files back to their original place so you don't lose anything.
