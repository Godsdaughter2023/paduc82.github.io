ken sheridan say: The address types (mailing or physical, or indeed any other type you might wish to record) are data, so should be stored as values at column positions in rows in tables and not encoded as column headings, which would violate a basic principle of the database relational model, the Information Principle, Codds Rule #1.   What you really have here in modelling terms are two ternary (3-way) relationship types, between People, Addresses and AddressTypes on the one hand, and Organisations,,Addresses and AddressTypes on the one hand. So Neither the People nor Organisations tables should include any address data, and there should be a table AddressTypes  The two relationship types would be modelled by tables as follows:

PeopleAddresses
....PersonID  (FK)
....AddressID  (FK)
....AddressTypeID  (FK)

OrganisationAddresses
....OrganisationID  (FK)
....AddressID  (FK)
....AddressTypeID  (FK)

An address can now be for both an organisation and one or more people, or for an organisation and people independently.  The mailing or physical address can be the same or different per person or organisation.

As in Scott's model the addresses per organisation and person would be entered via subforms, in which rows would be entered into the OrganisationAddresses and PeopleAddresses tables respectively.  In each subform the address and address type would each be selected in combo box, whose NotInList event procedure would be used to insert a new address not already represented in the combo box's list into the Addresses table.

As, in the case of the organisation address for a person being used for a person would simply involve selecting the same address in each subform, the need for a check box and any code is avoided.

Each subform should be based on a query which joins the OrganisationAddresses or PeopleAddresses to the Addresses table, and via this to the referenced tables described below.  Non-key columns from Addresses and the other referenced tables can be returned in the query and shown in bound text boxes in the subforms.  Each of these text boxes should have its Locked property set to True (Yes) and its Enabled property to False (No) to make them read-only.

The Addresses table should not include any redundancy.  Consequently it should include a CityID foreign key column, but not CountyID or StateID columns as these are determined by CityID, so to include these in the table would introduce transitive dependencies and the table would not be normalized to Third Normal Form (3NF), opening it to the risk of update anomalies.  A Cities table should include a CountyID foreign key column, and a Counties table a StateID foreign key column.  To allow for cities which map directly to a state, without an intervening county (assuming this is possible in the USA as it is with the corresponding geographical entity types in the UK) the Counties table should include N/A or similar rows for each State to plug the gap in the hierarchy.

I assume that zip codes, like UK postal codes, do not reliably map to any single geographical unit, so would be a column in the Addresses table.

For examples of the use of the NotInList event procedure when inserting address data over a hierarchy of hierarchical entities take a look at NotInList.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file includes a simple two-level hierarchy of Cities and Regions, with Cities referenced by a CityID column in a Contacts table.

I think perhaps I've found the solution.

The domain server which emails didn't arrive at hotmail had a spf dns entry with this data:

mydomain.com. 600 TXT v=spf1 ip4:999,999,999,999 a mx a:subdomain.mydomain.com ~all

I've changed the domain name and the ip address. Just use yours.

The thing was that the subdomain named at a:subdomain.... was wrong.

That subdomain is supposed to be the one that mailer uses to send emails, and it was wrong.

The domain has two subdomains available for sending emails, and this one was the one not used. Emails were being sent from the other subdomain.

So, I changed subdomain.mydomain.com and typed subdomain1.mydomain.com

The problem comes from the check that hotmail server does. Because in the "a" parameter you are telling hotmail where it has to admit emails from, and the emails are sent from subdomain that is not  this one. It's a security check for avoiding others to use your identity from another servers.

    john korchok say:

    To add a keyboard shortcut:

    1. Open the template file in Word. If the window title says Document1 or similar, you created a new document from the template rather than opening it. The window title must match the template file name.
    2. Choose Tools>Customize Keyboard.
    3. Under the left-hand Categories list, choose Format.
    4. In the right-hand Commands list, choose Bold.
    5. Click inside the Press new keyboard shortcut: field.
    6. Press the key combination you want to use to apply bold.
    7. Click on the Assign button (this is an easy step to forget).
    8. OK out and test your keyboard shortcut.

    If, for some reason, the Bold command already has the correct shortcut assigned, you can try re-assigning it. If that doesn't fix it, you might assign custom shortcuts like Control + B.
