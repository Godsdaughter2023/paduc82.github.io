ken sheridan say: Rather than using a simple parameter prompt a better option would be to use an unbound combo box from which the ID number or the member's name could be selected or entered.  This way only existing members can be selected or entered.  The combo box could be in the header of a bound form based on the query, in which case the parameter in the query would be along these lines

Forms!frmMembers!cboFindMember

where frmMembers is the form's name and cboFindMember is the combo box's name.  You'd then simply requery the form in the combo box's AfterUpdate event procedure with:

Me.Requery

An alternative approach would be, rather than referencing the combo box as a parameter, to navigate to the record in question, or filter the form to the relevant record with code in the combo box's AfterUpdate event procedure.  You'll find examples of both approaches in FindRecord.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this simple little demo file the combo box in the form's header enables  selection by the person's name, but could easily be adapted to allow selection by their ID number.  

Another approach would be to have the combo box in a separate unbound dialogue form, and open the bound form from this.  The opening form in the NotInList demo file in my same OneDrive folder does this, and also illustrates how to include values additional to the person's name in the drop-down list, to enable people of the same name to be differentiated.  Duplication of names is not uncommon, even in quite small groups of people.  As its name suggests the main purpose of the NotInList demo is to illustrate how to insert a new row into a referenced table when a new value is entered into a combo box, but in your case you could if you wish simply put code in the control's NotInList event procedure to pop up a custom message informing the user that the value they'd entered does not exist in the table.  My demo opens the bound form filtered to the selected person by means of code, but alternatively you could reference the combo box as a parameter in the bound form's query and simply open the form when an item is selected in the combo box.

It sounds like what is happening is that the conditional section is possibly contained within a layout table or another section, and that when hidden the table/section is not collapsing properly. This can be fixed by setting the height of the table cell or containing section to 0 (zero).

Nothing looks right about your code.  For the purpose that you are talking about, it would be better to insert { DOCVARIABLE fieldname } fields in the document where you want the information to appear and use code similar to:

Dim objDoc As Object

Dim objWord As Object

Dim bolOpenedWord As Boolean

Dim i As Long

Dim rs As DAO.Recordset

Set rs = CurrentDb.OpenRecordset("[name of Access table or query]")

If rs.RecordCount = 0 Then

    Exit Sub

End If

On Error Resume Next

Set objWord = GetObject(, "Word.Application")

If Err.Number = 429 Then

    Set objWord = CreateObject("Word.Application")

    bolOpenedWord = True

End If

On Error GoTo 0

rs.MoveFirst

Do Until rs.EOF

    Set objDoc = objWord.Documents.Add("C:\Users\user\Desktop\Access2Word.dotx"), False)

    With objDoc

        For i = 0 To rs.Fields.Count - 1

            If rs.Fields(i).Value <> "" Then

                .variables(rs.Fields(i).Name).Value = rs.Fields(i).Value

            Else

                .variables(rs.Fields(i).Name).Value = " "

            End If

        Next i

        .Range.Fields.Update

        .SaveAs2 "C:\Users\user\Desktop\" & rs.Fields(0).Value & ".docx" 

' Field number starting from 0 of the field that contains the data to be used as the filename

' I WOULD MOST STRONGLY SUGGEST THAT YOU DO NOT SAVE THE DOCUMENTS TO THE DESKTOP!

        .Close

    End With

    rs.MoveNext

Loop

Set objDoc = Nothing

If bolOpenedWord = True Then

    objWord.Quit

End If

Set objWord = Nothing

Set objDoc = Nothing

rs.Close

Set rs = Nothing

The requirements for using the system are:

  1. The mail merge main document must be of the Letter type, though that does not mean that the output cannot be sent as an e-mail message. 

  2. For the Many To One, Merge with Attachments and Merge to Individual Docs utilities, the data source may be either a table or query in an Access database, or in the form of an Excel worksheet and that worksheet must be the first sheet in the Excel workbook. If the data is on some other sheet, you can easily move that sheet so that it is the first sheet in the workbook by clicking on the sheet tab and dragging it to the left.  For the Chart Merge utility, download the Mail Merging with Charts document that is also on that page of my OneDrive for additional requirements of the data source for use with that utility

  3. For a data source in the form of an Excel worksheet, the field names must be in the first row of the worksheet and there must be a field name in all of the cells in that row that are within the range of columns that contain the data.

  4. For both types of data source, the field names must contain only alphanumeric characters (No @,#,$,%,&,(,), etc) and the field names must not start with a numeric character (0-9).

  5. For a data source in the form of an Excel worksheet, the data must start in the second row of the worksheet and there should be no empty rows within the range of data that is to be processed.

  3. the Using the Many to One Facility document that describes how to use that facility.
