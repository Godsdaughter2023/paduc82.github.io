roger carlson access mvp say:

Actually, it's bad practice to store calculated values in most cases.  In your case, I don't see a reason.  You can sum your Item prices for your Order any time you want with a query.  If you store the calculated value and you change one of the prices after the fact, it's possible that the calculated value will not get recalculated and you'll have a data anomally.

This is one of the fundamental principles of database design.  Don't let any of your fields depend on values in other fields for their value.  Since a calculation depends on the values of other fields, you shouldn't store it, but just recalculate it when you need it.

Sorry for the delay.  Tom has responded with good thoughts, but I thought I chip in too.

When a value becomes an historic fact, that is indeed the exception to the rule of storing a calculated value because at that point, it no longer depends on the values of its constituant fields.  Once the order has been fulfilled, it is the amount they've agreed to pay.

However, I don't see that what you've descibed applies.  Since the customer can update the order, it is even more imperative not to store the calculated value.  An expression on a form, query, or report will ALWAYS show the most current calculation, whereas a stored calculated value COULD show an incorrect value.

There are a number of ways to handle this situation, but perhaps the easiest is to ONLY store the order total after the order has been fulfilled.  In which case, you could have a button on your main form that marks the order fulfilled (with a Fulfilled boolean field) and code that copies the calculated value into the order total field:

Private Sub cmdFullfilled_Click()
On Error GoTo Err_cmdFullfilled_Click

Me!Fulfilled = True
Me!OrderPrice = Me![ItemsSubForm].Form![txtOrderTotal]

Exit_cmdFullfilled_Click:
    Exit Sub

Err_cmdFullfilled_Click:
    MsgBox Err.Description
    Resume Exit_cmdFullfilled_Click
   
End Sub

The above assumes you have a textbox called "txtOrderTotal" in the footer of your subform (which I have called "ItemsSubForm") that is unbound and has as its control source the expression =sum([ItemPrice]).

You could also have code that locks the subform if the order is marked fulfilled.  For instance, you could have a subroutine

Sub LockSubform()
    If Me!Fulfilled = True Then
        Me![ItemSubForm].Locked = True
    Else
        Me![ItemSubForm].Locked = False
End Sub

Then you could call this subroutine from the button AND from the OnCurrent event of the form.  This way the subform will lock an unlock depending on the status of the Fulfilled field.

There are a number of different ways to do this.  I already suggested one, that is to sum the value in a textbox in the footer of a subform and reference that value on the main form.  This assumes, however, that you have a form/subform.

As for putting the value of this query in a textbox, there are a number of ways to do that as well. 

First of all, I don't see that you need the JOIN at all.  The only thing you're using from the Orders table is the OrderNum, which you've also got in your Detail table, so you can use this query instead:

SELECT Sum(InventoryDetails.PriceOverRide) AS SumOfPriceOverRide
FROM  InventoryDetails
GROUP BY OrderNum, Deleted
HAVING OrderNum=[Forms]![OrderForm].[OrderNum] AND Deleted=False;

Or even better:

SELECT Sum(InventoryDetails.PriceOverRide) AS SumOfPriceOverRide
FROM  InventoryDetails
WHERE OrderNum=[Forms]![OrderForm].[OrderNum] AND Deleted=False
GROUP BY OrderNum, Deleted;

You might want to look here: for an explanation of The Difference between Where and Having in an Aggregate query

Now, you could save this as a query (called, say, qryOrderTotal) and then get the value into you textbox using DLookup() in the ControlSource property:

=DLookup("SumOfPriceOverRide", "qryOrderTotal")

You don't need a Where clause in your DLookup because you're already restricting it in the saved query.

A more direct method, however, would be to use the DSum() function in the textbox ControlSource:

=DSum("PriceOverRide", "InventoryDetails", "[OrderNum] =" & [Forms]![OrderForm].[OrderNum] & "AND [Deleted] = " & False

In this case, you are putting the criteria in the "where" clause of the DSum.

doug robbins - mvp office apps & services (word) say:

Select the Checkbox and assign a bookmark name to it (you will need to select the checkbox and at least a space alongside it to assign a bookmark)

Then if the bookmark name that you assigned to is was checkMale, you can use the code

With ActiveDocument
    If .Bookmarks("Client_IsMale").Range.Text = 1 then
        .Bookmarks("CheckMale").Range.ContentControls(1).Checked = True
    Else
        .Bookmarks("CheckMale").Range.ContentControls(1).Checked = False
    End If
End With

Note, you can probably dispense with the "Client_IsMale" bookmark and set the Checked attribute of the Content Control to True or False using the same logic that you are using to set the Client_IsMale bookmark to 1 or whatever it is for the other case.

herbie goat say:
Manage aliases on your Microsoft account
  •  Email
  •  Print

You can change the way you sign in and show up by adding aliases to your Microsoft account. An alias is an additional account name that can be an email address, phone number, or Skype name. It uses the same inbox, contact list, and account settings as the primary alias. You can sign in to your account with any alias—they all use the same password.

Some reasons to use aliases on your account include:

  • You're ready for a new email address, but don't want to lose all your settings and data.  
  • You want the flexibility of having several aliases that share the same subscriptions and inbox.
  • You want an alias that can't be used to sign in to your account, so that you can give it out to companies or other people and know they can't use it to access your info.
Note

You cannot add an alias that is already associated with another Microsoft account. Aliases cannot be used to share information between two Microsoft accounts.

Existing @hotmail.com, @live.com, Outlook.com, and @msn.com addresses can't be added as aliases.


By default, the email address or phone number that you use to sign up for your Microsoft account is considered the primary alias. Your primary alias shows up as the main account name when you view your profile information. This can be changed at any time.

Any other aliases are considered secondary aliases. You can control which secondary aliases can be used to sign in to your Microsoft account.
You can have up to ten aliases on your Microsoft account at any given time.
