I cannot recall the last time that office installs were MSI files? Going back all the way to 2003 – a single MSI was never an install???

So, if your installing system can deploy a setup.exe, then you should be just fine.

Keep in mind that MSI is “just” a file extension for the windows installer (msiexec.exe). In other words, you STILL launching msiexec.exe for the install. And the process for installing office is RATHER complex. The installer FIRST installs VBA, ribbon, and a ton of other SHARED code.

The installer THEN installs Word, or Access or whatever. I cannot stress that installing Word, or Access or Access runtime is VERY MUCH the same process – and is not a simple MSI file. (Read this sentence several times if you have to grasp this incredible important concept!).

Note that the Access runtime is really an “identical” install as the full edition of Access. So the process for installing office would be the SAME for installing the runtime. If you can or have been deploying office, then the SAME approach should work for the runtime.

The office install NEVER to my knowledge was “just” a MSI file. If you look at the office CD, you see a bootstapper called setup.exe, and that runs an install script (and msiexe.exe – the windows installer). This then installs all that SHARED library of stuff like VBA, ribbon etc. – common to all of office (and that's several MSI's).

Keep in mind that now armed with the above information, we realize that the issue with the Access runtime is that it ALSO has to install a good number of office parts. (So Ribbon, VBA, and many dependant libraries are installed by the runtime (unless office has been installed anyway).

Because the “several” parts, then the msi files for the runtime is actually “several” installs chained together, then no single MSI is available – and I not aware that such a MSI been available for Access, Access runtime or ANY office program.

However, it certainly possible that you require the office “standard” setup routines – many a “system” deployment tools do require a standard windows install (and not a download setup.exe).

 

So none of the office installs have been a single “msi” to my knowledge. And all of office programs (including Access, and Access runtime) certainly do include MSI files, but they are launched with a setup.exe bootstapper which THEN in turn launches msiexec.exe.

However, no question some deployment and installing tools don’t’ work with the single .exe download and they REQUIRE the standard setup.exe and install system.

You CAN extract the standard install from the setup.exe for most (if not all window download files).

You can extract the standard install (which DOES include the MSI files) with the following command (from the windows command line).

Those setups in near all cases can be extracted OUT to the standard windows install

For office installs, it usually extractPath, but for the access runtime, use /extract:


Thesetup.exe /extract:YouPathNameHere name of output folder

So for Access runtime, you would use this:

AccessRuntimexxxx.exe /extract:c:\myRuntime

The above will extract all of the install files to a folder – including the MSI files.

As noted, the install process for office or word or Excel or in this case Access is the SAME process. And it is a “series” of MSI installs – not just one.

So there no single MSI file for office programs. And to my knowledge there never been one for installing office programs.

Using the above “extract” option above will result in a standard windows install folder - and that should work with system "management" deployment tools.  In fact I use the above trick to build my own custom runtime installs.  I simply modify some of the config files and then re-package up the resulting folder and files with an installer.

Modify alphabetic caption numbering - Microsoft Community

wybrow say:

Modify alphabetic caption numbering

Hi,

I've set up a Caption in Word 2003 with the numbering format as capital alphabetic, so I can insert Cross-references A, B, C etc into my document. But I noticed that it continues after Z as AA, BB, CC etc. How can this be changed to give Captions AA, AB, AC etc instead?

Thanks.

cindy meister say:

Hi wybrow

I'm afraid it's not possible to change the automatic captioning to do this.

The only thing I can imagine would be to switch to a set of user-defined SEQ fields.

Word itself uses SEQ fields, like this, for example:

{SEQ Abbildung \* ARABIC }

You can see these if you press Alt+F9 to toggle on the field code display.

Switches can be added to SEQ fields, so after "Z" you can restart the first field at "A" and add a second field (which will need a different label) to display the running "A,B,C" for the next level. So:

{ SEQ FigI \* ROMAN } Up to Z, then...
{ SEQ FigI \r 1 \* ROMAN }{ SEQ FigII \* ROMAN } For AA
{ SEQ FigI \r 1 \* ROMAN }{ SEQ FigII \* ROMAN } For AB, etc.

Only problem is that if you're generating a Table of Figures (or whatever) it's going to stop showing the values correctly after "Z" (it will then repeat "A" to the end). So you'd have to unlink it and complete the TOF manually.

Cindy Meister, VSTO/Word MVP

ken sheridan say: The first non-zero width column is always the visible one in the control.  You either have to reverse the order of the columns and set the BoundColumn property to 2, but this will change the AutoExpand behaviour, or resort to a hybrid control in which a text box is superimposed on a combo box to give the appearance of a single combo box control, referencing the Column property of the combo box in the text box's ControlSource property.

Reset the global template (Normal.dot) back to default settings to fix this issue. Do the following,

 

1.     Click Start->Run and type %appdata% and click ok.

2.     Click on Microsoft folder-> Templates->Right click on Normal.dot and rename it to Normal.old

3.    Open Word and check if it open’s it using the default theme

 

Note: If you are not able to find Normal.dot, click on Tools at the top->folder options->view and select show hidden files and folders and uncheck Hide extension for known file types
