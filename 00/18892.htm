ken sheridan say:
One table in the db is for members of our organization and the other table is for prospective members.

I think you are probably right, and a single Members table is likely to be the correct option.  At its simplest all you need is a Boolean (Yes/No) column in the table to indelicate that the member is active or prospective.   In the application you can simply filter out the prospective members when you only want to see the active members, or vice versa.  You can easily do this when opening a form or report for example by filtering the form or report by means of the WhereCondition argument of the OpenForm or OpenReport method, or you can base the report on a query which is restricted by a criterion on the Boolean column.

It could be a little more complex than this if active and prospective members have different attributes.  You could of course simply have columns for each attribute in the members table and leave them Null where they are inappropriate to the member in question.  This is a little risky, however, as it would be very easy to assign a value to an attribute for a member where that attribute is inappropriate by virtue of their status as active or prospective.

In this scenario the other option is to model active and prospective members as sub-types of type Members.  This would require three tables: (a) a Members table with an autonumber primary key MemberID and columns for those attributes like names etc which are common to both active and prospective members; (b) an ActiveMembers table which also has a MemberID primary key column, but in this case not an autonumber, simply a straightforward long integer number data type. The other non-key columns of this table will represent attributes which are specific to active members and inappropriate to prospective members; (c) a ProspectiveMembers table which is the converse of the ActiveMembers table, so also has a MemberID primary key column, but again not an autonumber, simply a straightforward long integer number data type. The other non-key columns of this table will represent attributes which are specific to prospective members and inappropriate to prospective members.

The relationship types between Members and ActiveMembers, and between Members and ProspectiveMembers in each case will be one-to-one, so will be on MemberID, i.e the MemberID columns in ActiveMembers and ProspectiveMembers are foreign keys as well primary keys.

Whether you use a single table, or three tables modelling the type and two sub-types depends on the nature of your data.  Either could be correct, and a single table is obviously to be preferred for simplicity, but should not be used at the expense of data integrity if the situation really demands a 3-table model.  Two tables as at present is without doubt incorrect, however.

With a real header that you want to span across the whole page, there is no need to have a Section Break because the text in the body of the document is in columns

On the other hand, if, in the body of the document, you have text that you want to span right across the page as well as text that is to be formatted into newspaper style columns, then you will need a Section Break to separate those portions of the document.   

When you have multiple Continuous Section Breaks on a single page, it can sometimes be difficult to access the Section\Header\Footer that is to appear on that page and the best thing to do is sometimes use a macro to link the header\footers in ALL of the Sections

Dim i As Long, j As Long
With ActiveDocument
    For i = 2 To .Sections.Count
        With .Sections(i)
            For j = 1 To .Headers.Count
                .Headers(j).LinkToPrevious = True
            Next j
            For j = 1 To .Footers.Count
                .Footers(j).LinkToPrevious = True
            Nextj
        End With
    Next i
End With

        wMinute As Integer
        wSecond As Integer

User-defined Type statements must be at the module level, not within a procedure.  For example:

Private Type SYSTEMTIME
            wYear As Integer
            wMonth As Integer
            wDayOfWeek As Integer
            wDay As Integer
            wHour As Integer
            wMinute As Integer
            wSecond As Integer
            wMilliseconds As Integer
End Type

Private Declare Sub GetSystemTime Lib "kernel32" _
        (lpSystemTime As SYSTEMTIME)

Sub doit()
    Dim typTime As SYSTEMTIME
    ' .... your code ....
    GetSystemTime typTime
    Cells(rw + 3, 13) = Cells(rw + 3, 13) & typTime.wMilliseconds
End Sub

However, it is not necessary to use GetSystemTime if all you want is milliseconds.  The VBA Timer function returns time accurate to multiples of 15.625 milliseconds.  So you could do:

Dim t as Double
t = Timer
Cells(rw + 3, 13) = Cells(rw + 3, 13) & Round((t - Int(t))*1000)

If you want exactly the same result as wMilliseconds, change Round to Int.

And depending on what you are trying to do, perhaps you simply need to do:

Cells(rw + 3, 13).NumberFormat = "h:mm:ss.000"

I do not know what is in Cells(rw+3,13); presumably time acquired somehow.  The Excel Now function returns time accurate to multiples of 10 milliseconds.  Good enough?

(But note:  the VBA Now function returns time accurate only to one second.)
