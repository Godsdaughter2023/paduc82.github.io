scottgem say:

A foreign key is not something you locate. It is a field that allows you to join to another table. Usually it is the primary key of the related table. So, for example, you want attribute each Visit Note record to a specific client. To do that you need to include the ClientID as a foreign key. 

Also, I do NOT recommend using lookups on a table level. Lookups should be done only on the form level using List controls.

As to the data types:

tblVisitNotes

VisitNoteID (Primary Key Autonumber)

ClientID (Foreign Key)  - Depends on the datatype of the client primary key. If an autonumber then the FK should be a Number>Long Integer

VisitDate - Date/Time

Notes - Long Text

tblVisitDocs

VisitDocID (PK Autonumber)

ClientID (FK) (see above

DocPath - Short Text

VisitNoteID - Number>Long Integer

Also, these are suggestions. You mean need to include additional fields depending on your needs. For example, you might need a field in tblVisitNotes to indicate who made the visit. Or you may expand Visit to include phone calls, so you might need a field to indicate the type of visit. In tblVisitDocs, you might want a field to briefly describe what the doc is. Just some food for thought.

The rule criteria “If Sender Address Is” has been removed from the Outlook.com option to create rules for sorting new messages. The reason behind this is because the action done by the option “If Sender Address Is” can be easily attained by using the option “If Sender Contains”.

 

For instance, if you want to move all the emails coming from the email address “abc@***.com” to a folder “abc”, then you just need to provide the email address “abc@***.com” for the criteria “If Sender Contains” while creating this rule. turning off rounding on access 2010 - Microsoft Community

laurenlee say:

turning off rounding on access 2010

Just downloaded Access 2010.  When entering a decimal into a number field, Access is automatically rounding, which I never set it up to do so.  I am wondering how I can "turn this off"?

 

Thanks,

 

Lauren

hansv mvp say:

Open the table in design view.

Select the number field.

Look at the field properties in the lower part of the table design window.

If the Field Size property is set to Long Integer, Integer or Byte, it can only contain whole numbers. In order to be able to enter decimals, the Field Size must be Single or (preferably) Double.

If it is already set to Single or Double, look at the Format property. If this is set to Fixed, and Decimal Places is set to 0, the field can contain decimals, but all values will be displayed as whole numbers.

marshall barton say: The same thing applies to the Currency datatype, which always has 4 decimal places.
