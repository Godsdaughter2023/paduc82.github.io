julie fka profjulie say: On first sync, ActiveSync (or WMDC) should recognize that you've got data already on the pocket pc and prompt you with how you wish to proceed with 3 choices:  1) Combine the data on both; 2) Replace the data on the pocket pc with the data from the computer; 3) Do nothing.

If you choose the 1st option, whatever common records that you have on both the computer and the pocket pc will be duplicated on both, and the unique data on both devices will be synced to both devices. So you'll wind up with all your data on both the pocket pc and the computer with some records duplicated.

If you choose the 2nd option, all the data on your pocket pc will be wiped out and all the data on the computer will be synced to the device.  So you'll lose whatever unique information you had on the pocket pc.

If it were me, I would take some time cleaning up the data either on the pocket pc (deleting any data you've already got on the computer) and then sync the computer and the pocket pc, choosing the 1st option.
Prof Julie, Microsoft MVP Mobile Devices | http://frommypocket.pocketprimer.com/

Unless the queries use parameters to filter the records, I think you should use this kind of thing to run the make table queries:

 

   CurrentDb.Execute "User1", dbFailOnError

 

right after deleting the tables.

 

FWIW:  I object to constantly deleting tables and recreating them.  It greatly increases the chances of corruption, will bloat you database and can not work if you should ever try to compile and distribute the database as an accde or mde file.  If the created tables always have the same field list, then it would be better to just delete the records in the tables and append the new set of records.  Best would be for the reports to use Select queries instead of tables as their record source and avoid the temp table scenario altogether.

Resizing multiple images at once - Microsoft Community

uluque say:

Resizing multiple images at once

Hello,

I am trying to print more than 150 images in size 2"x3". Currently the images are all different sizes. Is there any way to select all the images, or at least multiple at once and resize them? I'm on a Mac and using Word 2016.

bob jones aka: cybertaz say:

Do all of the images have some form of Text Wrap applied? IOW, not In Line with Text? If so, the simplest method is to use the Selection Pane which can be launched from the right end of the Layout tab, then

  1. Click the first one in the list
  2. Hold Shift while you click the last one in the list
  3. Use the Size fields on the Picture Format tab to set the Height & Width. (Clear the box that links the 2 fields)

Some considerations:

  • Images which are set to In Line will not be included in the selection. You can't select multiple In Line images at once, nor can an In Line image be included in a selection of Text Wrapped images. In Line would have to be resized individually or by using some other method.
  • Resizing a number of images will cause them to be relocated so they will then have to be repositioned.
  • Any images which are not the same 2x3 aspect ratio may become distorted... how severely is determined by the disparity of the aspect ratio. IOW, an image that is 4x6 will resize well but one that is 5x7 may not.

unable to add table columns to maximum (255)

I've been keeping norms databases in Access for a dozen years or so and continue to have problems creating and entering data into tables containing > about 175 columns. My current proposed table has exactly 222 columns, but when I try to enter a text file with 222 fields, Access gives me an error message saying the table has more than 255 columns. Why won't this work? I could break the table up, but this will create lots of inefficiency and extra work.
john p spencer say:

Access keeps track of the number of fields added to a table and the upper limit is 255 fields.

When you delete a field - the count is not decreased.

When you change the datatype of a field, a new field is added, then populated data converted to the proper data type, then the old field is deleted. So the count is increased by one.

I suggest you compact your database.  This usually resets the field count to the actual number of current fields.

 

That said, I agree with the others that your design is very probably not correct for a relational database.  We can't make any suggestions on what you may or may not need to do - we don't have sufficient information on your current data structure or the purpose of your database.
