Someone might have a more succint answer, but ...

Find the lowest and highest 10% of marks.

In F5 Lowest =PERCENTILE(Marks,0.1) 'in my data 42

in F6 Highest =PERCENTILE(Marks,0.1) ' in my data 82

Average:

=SUMPRODUCT(--(Marks>42)*(Marks<82),(Marks))/SUMPRODUCT(--(Marks>42)*(Marks<82))

Substitute the named range 'Marks' with the the range of your exam marks.

joeu2004 say: Tony Valko wrote:
That's why I said: As a wild guess...
[....]
=TRIMMEAN(A1:A15,2/COUNT(A1:A15))

Well, maybe you should stop making "wild guesses".

First, the OP wants to exclude 20% (upper 10% and lower 10%).  For TRIMMEAN, that means the second parameter should be 20%.  (Of course, 20% and 0.2 are the same thing.)

The problem is:  2/COUNT(A1:A15) is not always close to 20%.  For example, if A1:A15 contains 15 numbers, 2/COUNT(A1:A15) is about 13.33%.  If A1:A15 contains 3 numbers, 2/COUNT(A1:A15) is about 66.67%.

I suspect that you decided that in a range of 3 to 15 numbers, the "right" answer is to exclude 2 -- 1 from each end -- whereas TRIMMEAN excludes 0.  The OP may or may not agree with you.

And yes, TRIMMEAN(A1:A15,2/COUNT(A1:A15)) should always exclude 2, barring floating-point anomalies, which we can hope are unlikely.

But excluding 2 is certainly not the "right" answer for all ranges.  For example, we should exclude 4 (2 from each end) from a range of 20 numbers, since that is 20%.

So at the very least, I think your solution is difficult to generalize without explanation, since the OP said nothing about the size of his/her range.

Second, as for your example (9 numbers in a range of 15), I see nothing in the OP's question that suggests that the range contains anything but numbers.  Nonetheless, as an aside, I think you raise an interesting question:  what does TRIMMEAN do with non-numbers in the range?

I see nothing in the TRIMMEAN help page (for Excel 2003) that answers that question.  Empirically, it appears that TRIMMEAN ignores non-numbers (at least text and empty cells), just as COUNT does.  So TRIMMEAN handles a range of 15 with 9 numbers the same as a range of 9 numbers.

But I don't see any need to complicate the situation.  I think the point is simply:  20% of 9 is 1.8, not 2.

What TRIMMEAN should do in that case is debatable.  That is why I wrote (to the OP):  "You might want to be sure that TRIMMEAN makes the right selection (exclusion) for you when the size of the range times percentage is not an integer".

The TRIMMEAN help page says that 1.8 would be rounded __down__ to the nearest multiple of 2.  That means 0 in this case.

So yes, if A1:A9 contains only the sorted numbers in your example, TRIMMEAN(A1:A9,20%) returns about 58.11, the same as AVERAGE(A1:A9), the entire data set.

Perhaps you feel that TRIMMEAN should round __up__ to the nearest multiple 2.  Thus, TRIMMEAN(A1:A9,20%) would return 61, the same as AVERAGE(A2:A8) is 61, the middle 7 data.

Is TRIMMEAN "wrong"?  Yes, IMHO.  Apparently, you agree.

Are you "right"?  Maybe; maybe not.

But even if you are, 2/COUNT(range) is not the right answer in general because TRIMMEAN(range,2/COUNT(range)) will always exclude 2.

Instead, the following is one way to force TRIMMEAN to round up:

=TRIMMEAN(range,MROUND(pct*COUNT(range),2)/COUNT(range))

Ouch!

Caveat:  I can only hope that TRIMMEAN will not stumble over floating-point arithmetic anomalies.  And note to OP:  MROUND requires the ATP add-ins.  Alternatively, replace MROUND(expression,2) with ROUNDUP((expression)*2,0)/2.

However, rounding to the nearest multiple of 2, be it round up or round down, might not be the "correct" answer statistically.  For example, see how PERCENTILE(range,k) handles the situation when k is not a multiple of 1/n (actually 1/(n-1)).  (And I can think of another alternative off-hand, although I have not given any thought to its efficacy.)

For that reason, perhaps TRIMMEAN is not the right function to use for the OP's purpose in the first place.

Frankly, I have never given it any thought because, generally, I disagree with methodologies that exclude a percentage or number of data arbitrarily in the first place.

Using Index/Match Functions with a Number and Date Lookup - Microsoft Community

gdcross say:

Using Index/Match Functions with a Number and Date Lookup

The Challenge. I need to lookup an employee's hourly rate based on their employee ID and a week ending date. The week ending date of course rarely matches the employee's effective date for an hourly rate and we need to make sure the correct hourly rate is chosen.

I am using the following formula to lookup in Table2 (given below) an hourly rate in Col D (the fourth column of Table2). The lookup is based on (1) an Employee ID (number format) which matches nicely the Employee ID in Table2 and (2) a Week Ending Date (date in which I use the number format).

=INDEX(Table2,MATCH(A2&D2,Employee_ID&Effective_Date_Num,1),4)

In this equation, A2 equals one Employee ID value, D2 equals one Week Ending Date value (in number format), Employee_ID is the name range for the data in Table2's Col A, and Effective_Date_Num is the name range for the data in Table2's Col E.

This equation works well (from what I can tell) until I get to row 9766, At this point the hourly rate seems to not recalcuate until row 10901. Row 10901's value continues to not recalculate to the end of the data, row 11792.

Any help on my situation or maybe an alternative approach would be much appreciated.

 

TABLE2

 
  Col A Col B Col C Col D Col E
Row 1 Employee ID Effective Date Annual Rate Hourly Rate Effective Date (#)
Row 2 109537 04/01/07 $### $11.000 39173.00
Row 3 109538 09/01/07 $### $10.79 39326.00
Row 4 109538 03/01/08 $### $12.91 39508.00
Row ... ... ... ... ... ...

 

 

jeeped say:
You might be able to find an exact match using a combination of a conventional SUMPRODUCT() together with an array formula MAX(IF()).

        

The array formula in H2 is,
=SUMPRODUCT((A$2:A$5=F2)*(B$2:B$5=MAX(IF(A$2:A$5=F2,IF(B$2:B$5<G2,B$2:B$5))))*(D$2:D$5))
... which must be finalized with Ctrl+Shift+Enter↵.
You should note that I have added an entry and intentionally mis-sorted the data in A2:D5 to demonstrate that the SUMPRODUCT() is finding an exact match to the Employee ID together with the Maximum Effective Date that is lower than the date provided.

BTW, there is no need to work with your Effective Date (#) representation of the dates in column B as the underlying number is what will be calculated on in any event. You can prove this by putting =A2+B2*10^-5 in a cell and filling down to show subsequent rows. The formula tacks the Effective Date as a decimal onto the Employee ID as an integer.

jeeped say:
You are using a 1 as the match_type parameter in the MATCH() portion of the INDEX(MATCH()). This requires that the entries be sorted numerically, alphabetically or boolean-wise. From Excel's F1,

Match_type Behavior
  • 1 or omitted MATCH finds the largest value that is less than or equal to lookup_value. The values in the lookup_array argument must be placed in ascending order, for example: ...-2, -1, 0, 1, 2, ..., A-Z, FALSE, TRUE. 
  • 0 MATCH finds the first value that is exactly equal to lookup_value. The values in the lookup_array argument can be in any order.
  • -1 MATCH finds the smallest value that is greater than or equal to lookup_value. The values in the lookup_array argument must be placed in descending order, for example: TRUE, FALSE, Z-A, ...2, 1, 0, -1, -2, ..., and so on.

I believe this may be why you are receiving errors after 9766. Without seeing your data, it is just a stab in the dark, but it seems likely that the data set requires a Sort at this point.
