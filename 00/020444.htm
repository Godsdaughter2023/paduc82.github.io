john w. vinson say:

I'd suggest a more radical solution. Don't delete the data; don't delete the table; instead, delete the database! 

That is, use a split database design with a frontend containing your forms, reports, queries, and code, and not one but two backends containing your tables. One backend would contain your big million row table, and when you need a new table you would simply trash it and create a new .accdb file and import the data. The second backend (which I'm only guessing would be needed) would contain any accessory tables (lookup tables, logs, etc.)

I would suggest creating a "template" database containing a version of your big table, empty of data, but with all the needed fields with appropriate datatypes, indexes, and so on; your import could be done by linking to your data source to a copy of this template and running an Append query to populate it. You'd then run your Powerpivot and any needed reports or display. The copy can then be deleted when you're ready for a new set of data. This avoids bloat, autonumber issues, etc.; the only downside is that you cannot maintain referential integrity between this big table and your other tables, but I'm guessing you're not doing so anyway.

With VBA code it's easy to perform the file operations - KILL "xyz.accdb" will delete the old database, MOVE "template.accdb" "xyz.accdb" will copy the template, etc.

ken sheridan say: You seem to misunderstand a couple of things:

1.  The IIF function returns a value and has three arguments.  The first argument is a Boolean expression which evaluates to TRUE or FALSE; the second argument is the return value if the first argument evaluates to TRUE; the third argument  is the return value if the first argument evaluates to FALSE.  You  appear to be trying to use the function to assign a value to another column in the way that you would in an If….End If construct in a VBA procedure.

2.  You say 'The relationship is set up between FirstNational (FK) table and CommunityRevenueSize table (PK)'.  A table is not a primary key or a foreign key.  The keys are one or more columns in each table.  In an one-to-many relationship type the primary key has distinct values, the foreign key has non-distinct values.  What appears to be the  case from your screen shot is that CommunityRevenueSize  is the referenced table in the relationship type , and FirstNational is the referencing table.  I think that's what you meant, but it's not how your statement reads.

If ttlAnnualRevenue is the sum of the AnnualRevenueExpenditureGov and AnnualRevenueExpenditureOSR columns per row, for each row in FirstNational to map to a row in CommunityRevenueSize on the basis of a the total revenue in a row in FirstNational you cannot have  a relationship (i.e. a relationship object) to CommunityRevenueSize without introducing redundancy into FirstNational, with the consequent risk of update anomalies.  The conceptual relationship can still exist, however, and be used as the basis for a JOIN in a query, but the CommunityRevenueSize table firstly needs to be redesigned as follows:

CommunityRevenueSize
….LowerValue
….UpperValue
….CommunityRevenueCategory

The table would thus contain the following rows:

0                   5000000       Small
5000001      10000000      Medium
10000001    100000000    Large

The 100000000 value in the last row is an artificially high value which will never be reached.  Each column in the table is a candidate key, so one can be made the primary key and the other two each indexed uniquely.

The FirstNational and  CommunityRevenueSize tables can then be joined in a query on the following expression:

((FirstNational.AnnualRevenueExpenditureGov + FirstNational.AnnualRevenueExpenditureOSR) BETWEEN CommunityRevenueSize.LowerValue AND CommunityRevenueSize.UpperValue)
