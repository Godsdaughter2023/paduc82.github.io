paulgailiunas say:

Hi

Like Scott and Tom, I strongly agree that a solid tables definition is needed before you build forms and reports.

They are actually much easier to apply to correctly normalised tables.

Firstly - your table design with fields showing actual metal names as fields is a NO-NO.

So firstly - define your "terms"

Lets say a Holding is a parcel of asset items . . 5 oz gold 10 oz palladium 20 oz silver etc

so the Holding is a simple record that has an identity and an owner

the main data is the list of metals within a holding.

Lets call these "Units"

So for any 1 Holding - there can be 1 or more Units. I would assume a Unit record to have fields like . .

UN_ID . . a unique identity . . autonumber

UN_HLD_ID . . identity of the Holding to which this Unit belongs - FE or Foreign Key

UN_TYPE_ITEM , ,  FE to list of metals in a reference table of metals and there data - like value

UN_QTY_ORIGINAL . .  quantity originally acquired.

UN_VALUE_PURCHASE . . value at time of purchase

UN_QTY_ONHAND . . quantity of metal held now.

UN_MEASURE . . unit of measure . . oz . . gm kg etc . . maybe irrelevant if only 1 measurement is used

UN_DATE_AQUIRED . . Date unit purchased

UN_DATE_SOLD . . Date of last sale (transfer)

UN_VALUE_

etc

note . . I use CAPS for field names and always add a prefix to the field to denote the table where it exists.

The movement of Units becomes the transactions mentioned by Scott

I would be inclined to always create a new Unit when one is sold - that is the seller keeps a record of the unit and the buyer has a record created . the seller unit has quantity reduced and the buyer has quantity increased.

the transactions are then a simple recording of the movement of all or part of a Unit to a NEW unit that belongs to a different Holding.

it would have fields like . .

TRN_ID . . a unique ID . . autonumber

TRN_DATE . . date it occurred

TRN_UN_ID_SELLER. .  selling unit ID

TRN_UN_ID_BUYER . . .

TRN_QTY . . .

TRN_VALUE . .  Value at time of transfer -  varies according to current value.

the Transaction record provides an historical trace of the movements - transactions.

the UNIIT record provides a list of current values

the tricky bit is creating processing the transaction.

identify the seller - select the Unit to be sold - enter the quantity to sell - create a new Unit record for the buyer and set initial quantity - reduce the seller quantity.

in this scenario - partial movements are easy to track.

I assume there can also be acquisitions from outside - just adds a little bit more.

Hope this helps to get you started. Its a bit of a ramble - but it is late at night here . . . .

cheers

PaulG

scottgem say:

First, moving partial holdings has no bearing on the design I suggested. With each transfer being a transaction, it doesn't matter whether a transaction  represents a full or partial transfer.

Second, your table is poorly designed. I won't even go into trying to fix it. It needs to be scrapped and you need to start from scratch.

The attributes of a transaction are as follows: date, metal, quantity, custodian, and type. Where type indicates if it is a purchase or sale. The only other attribute you might need to capture is the unit value, but that depends on your need. If the transaction is a transfer I would automate adding the companion transaction. The way I would envision it having 2 unbound controls on your form. A checkbox to indicate its a transfer and a combobox to select the other custodian. Then after you add the transaction, you create a second transaction for the other side.

Another point I just thought of is whether you need to record external sources. If the transaction is a new purchase do you need to record where it was purchased from? Similarly if there is a sale to an external source, do you need to record that?

I disagree with Paul's suggestions on a few points. Not that he is wrong, just that its not the approach I would take. For one I don't think you need a specific holdings table. A custodian's holdings will be calculated based on the transactions made for that custodian. The holdings table would be redundant daya. The only reason I might see for such a table would be if you needed to create a starting point rather than record all the transactions that created the current holdings. I would be more inclined, however, to create initial transactions to record the current holdings.

I also don't see the need for prefacing field names to indicate the table. I prefer to keep field names short and it is not common to use the same field name in multiple tables. Only where that happens might I use such a prefix.

So I see, at minimum, the following tables: Custodians, Transactions, Metals. The latter being a lookup table to select the metal from. It looks like you have the first in your People table. But your ledger table needs to be scrapped for the Transactions table I described. I'm not sure you need the Trans Types since there are only 2.

Sub TestMe()

Sub MoveAComment(lFromSlide As Long, lCommentNumber As Long, lToSlide As Long)
   
    Dim oSource As Comment
    Dim oTarget As Comment
   
    Set oSource = _
    ActivePresentation.Slides(lFromSlide).Comments(lCommentNumber)

    Set oTarget = _
    ActivePresentation.Slides(lToSlide).Comments.Add( _
        oSource.Left, _
        oSource.Top, _
        oSource.Author, _
        oSource.AuthorInitials, _
        oSource.Text)
   
    ' and optionally, delete the original
    'oSource.Delete
   
End Sub

Option Explicit
Sub ColorTeams()
  Dim Team As Range, Teams As Range
  Dim Wb As Workbook
  Dim Ws As Worksheet
  Dim All As Range
  
  'Refer to all teams in A2:A... in the active sheet
  Set Teams = Range("A2", Range("A" & Rows.Count).End(xlUp))
  
  'Visit all opened files
  For Each Wb In Workbooks
    'Skip our file
    If Wb.Name = ThisWorkbook.Name Then GoTo SkipWb
    'Skip hidden files
    If Wb.Windows(1).Visible = False Then GoTo SkipWb
    'Visit each sheet in that file
    For Each Ws In Wb.Worksheets
      'Skip if protected
      If Ws.ProtectContents Then GoTo SkipWs
      'Now perform the team coloring
      For Each Team In Teams
        'Find all cells for this team
        Set All = FindAll(Ws.UsedRange, Team, LookIn:=xlValues, LookAt:=xlWhole)
        'Found?
        If All Is Nothing Then GoTo SkipTeam
        With All
          'Same background color
          .Interior.Color = Team.Interior.Color
        End With
SkipTeam:
      Next
SkipWs:
    Next
SkipWb:
  Next
End Sub
Private Function FindAll(ByVal Where As Range, ByVal What, _
    Optional ByVal After As Variant, _
    Optional ByVal LookIn As XlFindLookIn = xlValues, _
    Optional ByVal LookAt As XlLookAt = xlWhole, _
    Optional ByVal SearchOrder As XlSearchOrder = xlByRows, _
    Optional ByVal SearchDirection As XlSearchDirection = xlNext, _
    Optional ByVal MatchCase As Boolean = False, _
    Optional ByVal SearchFormat As Boolean = False) As Range
  'Find all occurrences of What in Where (Windows version)
  Dim FirstAddress As String
  Dim C As Range
  'From FastUnion:
  Dim Stack As New Collection
  Dim Temp() As Range, Item
  Dim i As Long, j As Long
  If Where Is Nothing Then Exit Function
  If SearchDirection = xlNext And IsMissing(After) Then
    'Set After to the last cell in Where to return the first cell in Where in front if _
      it match What
    Set C = Where.Areas(Where.Areas.Count)
    'BUG in XL2010: Cells.Count produces a RTE 6 if C is the whole sheet
    'Set After = C.Cells(C.Cells.Count)
    Set After = C.Cells(C.Rows.Count * CDec(C.Columns.Count))
  End If
  Set C = Where.Find(What, After, LookIn, LookAt, SearchOrder, _
    SearchDirection, MatchCase, SearchFormat:=SearchFormat)
  If C Is Nothing Then Exit Function
  FirstAddress = C.Address
  Do
    Stack.Add C
    If SearchFormat Then
      'If you call this function from an UDF and _
        you find only the first cell use this instead
      Set C = Where.Find(What, C, LookIn, LookAt, SearchOrder, _
        SearchDirection, MatchCase, SearchFormat:=SearchFormat)
    Else
      If SearchDirection = xlNext Then
        Set C = Where.FindNext(C)
      Else
        Set C = Where.FindPrevious(C)
      End If
    End If
    'Can happen if we have merged cells
    If C Is Nothing Then Exit Do
  Loop Until FirstAddress = C.Address
  'FastUnion algorithm :copyright: Andreas Killer, 2011:
  'Get all cells as fragments
  ReDim Temp(0 To Stack.Count - 1)
  i = 0
  For Each Item In Stack
    Set Temp(i) = Item
    i = i + 1
  Next
  'Combine each fragment with the next one
  j = 1
  Do
    For i = 0 To UBound(Temp) - j Step j * 2
      Set Temp(i) = Union(Temp(i), Temp(i + j))
    Next
    j = j * 2
  Loop Until j > UBound(Temp)
  'At this point we have all cells in the first fragment
  Set FindAll = Temp(0)
End Function

Option Explicit
'
' SetIndentLevel Macro
'
' Used integer value in Column B to set the indent level in Column C
'
Sub SetIndentLevel()
    Dim i As Long, nRows As Long
'
    nRows = ActiveSheet.Range("C1").End(xlDown).Row - 1 ' minus 1 assumes a header row
'
Application.ScreenUpdating = False
    For i = 1 To nRows
        ActiveSheet.Cells(i + 1, 3).IndentLevel = Application.WorksheetFunction.Max(Application.WorksheetFunction.Min(CInt(ActiveSheet.Cells(i + 1, 2)), 25), 0)
    Next i
Application.ScreenUpdating = True

End Sub

So, I have seen this question on many forums but have not yet located a solution..............

QUESTION: Can I configure Outlook2007 so that it will open 2 Exchance mailboxes in separate windows thus allowing USERA to have his own Outlook window open, and USERB to have his own Outlook window open, at the same time?

sachin shetty say:

As both the accounts are configured in a same profile you are facing this issue.

Usually the first account setup in outlook is considered as default account and all emails sent goes through this account. So in this case, USERA’s account is the default account and all emails sent goes through this account, resulting in reply email being received in USERA’s inbox. 

To avoid this, use USERB’s account while sending the email. Refer the link below on same and check if it helps.
