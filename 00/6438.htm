bob umlas, excel mvp say:

Select the cells to copy, run this code:

 

Sub Copy_Filtered_Cells()
    Set from = Selection
    Set too = Application.InputBox("Select range to copy selected cells to", Type:=8)
    For Each Cell In from
        Cell.Copy
        For Each thing In too
            If thing.EntireRow.RowHeight > 0 Then
                thing.PasteSpecial
                Set too = thing.Offset(1).Resize(too.Rows.Count)
                Exit For
            End If
        Next
    Next
End Sub

guessed say:

I would classify this as 'by design'. The macros run list only shows the Subs which don't take any input parameters. Functions won't show in that list either (whether they have parameters or not).

Adding a parameter to the sub to allow it to be added to the QAT isn't necessary if you add buttons by "Customize Quick Access Toolbar" but if you are doing it by a Ribbon Editor then it may well be (I haven't made QAT edits via a Ribbon editor).

The way I get around your issue on Ribbon buttons is to point each button's onAction to the same generic macro and pass the macro name via the button's ID or Tag property.

Macros in RibbonX Module

Sub RunMacro(control As IRibbonControl)
  'allows the ribbon to run macros which don't have parameters
  On Error GoTo ErrCatch
  If control.Tag = "" Then
    Application.Run control.ID
  Else
    Application.Run control.Tag
  End If
  Exit Sub
ErrCatch:
  MsgBox Err.Number & vbCr & Err.Description & vbCr & control.ID
End Sub

Sub HiMum()

  MsgBox "Hi Mum"

End Sub

Sub HelloWorld()

  MsgBox "Hello world"

End Sub

RibbonX code to run those macros

<button id="HiMum" label="Hi" onAction="RibbonX.RunMacro"  imageMso="BulletsGalleryWord" />

<button id="HelloWorld" label="Hi Again" onAction="RibbonX.RunMacro"  imageMso="BulletsGalleryWord" />

brian tillman [outlook mvp 2007-2019] say:

So, you have an Outlook.com account A and another Outlook.com account B.  You want to combine them.  You have several choices.

  • Make the account B a send/receive account underneath A.  This will cause A to pull all of B's Inbox contents into A and allow you to send messages appearing to come from B while using A. You can also continue to log into B separately.  You will need to log into B at least once each year to keep it operational.

  • Create an alias C in account B and make that alias the primary alias of the account.  This will essentially rename the account from B to C.  You can then delete the alias B and after 30 days, that alias should be available for adding to the A account.

Public Function FractionIt(dblNumIn As Double) As String
'========================================================
 ' Name:  FractionIt
 ' Purpose: Converts a double into a string representing a rounded fraction
 ' Inputs: dblNumIn As Double
 ' Returns: String
 ' Author: Arvin Meyer
 ' Date: Revised - 6/22/2001
 ' Comment: Rounds down from 1/64 over
 '========================================================
On Error GoTo Err_FractionIt

Dim strFrac As String
Dim strSign As String
Dim strWholeNum As String
Dim dblRem As Double

    If dblNumIn < 0 Then
        strSign = "-"
        dblNumIn = dblNumIn * -1
    Else
        strSign = " "
    End If

    strWholeNum = Fix([dblNumIn])

    dblRem = [dblNumIn] - [strWholeNum]

Select Case dblRem
    Case 0
        strFrac = ""
    Case Is < 0.140625
        strFrac = "-1/8"
    Case Is < 0.265625
        strFrac = "-1/4"
    Case Is < 0.390625
        strFrac = "-3/8"
    Case Is < 0.515625
        strFrac = "-1/2"
    Case Is < 0.640625
        strFrac = "-5/8"
    Case Is < 0.765625
        strFrac = "-3/4"
    Case Is < 0.890625
        strFrac = "-7/8"
    Case Is < 1
        strFrac = "1"
End Select

If strFrac = "1" Then
    FractionIt = strSign & (strWholeNum + 1)
Else
    FractionIt = strSign & strWholeNum & "" & strFrac
End If

Exit_FractionIt:
    Exit Function

Err_FractionIt:
    Select Case Err
    Case 0

    Case Else
        MsgBox Err.Description
        Resume Exit_FractionIt
    End Select

End Function

The txtFaceWidthIn and txtFaceWidthFrac controls can remain permanently enabled.

Enabling the length controls is a little tricky.  Ideally one would do so when both the txtFaceWidthIn and txtFaceWidthFrac have been updated, but I'm assuming that it's possible for a width to be an integral number of inches, so the txtFaceWidthFrac could be left Null.  I'm assuming that you won't have any widths of less than 1 inch, so a value will always be entered in txtFaceWidthIn.  The best you can achieve therefore is to enable the length control in the txtFaceWidthIn control's AfterUpdate event procedure:

Private Sub txtFaceWidthIn_AfterUpdate()

    Me.txtFlin.Enabled = Not IsNull(Me.txtFaceWidthIn)
    Me.txtFlFrac.Enabled = Not IsNull(Me.txtFaceWidthIn)

End Sub

If it were possible to have a width of less than 1 inch and for txtFaceWidthIn to be left Null, then you'd also do similarly in the txtFaceWidthFrac control's AfterUpdate event procedure of course.

You'd then compute the price in the AfterUpdate event procedures of both the Me.txtFlin and Me.txtFlFrac controls after assigning a value to the txtFL control:

Private Sub txtFlin _AfterUpdate()

    Me.txtFL = Nz(Me.txtFLin, 0) + Eval(Nz(Me.txtFLFrac,"0"))
    Me.txtPrice = DMin("price", "tblCouilessePrices", "Width>=" & _
        Me.txtFaceWidth & " And Length>=" & Me.txtFL)

End Sub

Private Sub txtFlFrac _AfterUpdate()

    Me.txtFL = Nz(Me.txtFLin, 0) + Eval(Nz(Me.txtFLFrac,"0"))
    Me.txtPrice = DMin("price", "tblCouilessePrices", "Width>=" & _
        Me.txtFaceWidth & " And Length>=" & Me.txtFL)

End Sub

Note that the underscore character at the end of the line is a continuation character and should be followed by a carriage return/line feed, i.e. you press the Enter key after the underscore character.  When entering a non-integral length the price will be computed incrementally of course, first for the inches when you enter a value in the txtFlin  control, and then for the fill width when you enter a value in the txtFlFrac control.  Again the ideal would be for it to be computed only when both are updated, but as, in the event of the width being an integral number of inches, the latter won't be updated, one has to compromise slightly.

One thing you will need to be sure of is that the values in the Width and Length columns in the tblCouilessePrices table are decimal representations of the inches values which exactly match those  returned by the expressions used in the form's module.  To assist in this you might want to set up a little unbound 'calculator'  form with controls txtIn, txtFrac and txtDec, giving the latter a ControlSource property of:

    =Nz([txtIn], 0) + Eval(Nz([txtFrac],"0"))

Since you say you're a novice programmer, you're likely not aware that your variable declarations don't do what you probably think they do. You cannot "short circuit" type declarations: you must include As datatype for each variable. In other words, your declarations need to be

  'Excel object variables

    Dim xlApp As Object, xlWorkbook As Object, xlCurrentSheet As Object

    

    'Access object variables

    Dim wspDefault As Workspace, dbs As Database, tbl As TableDef, fld As Field

    Dim varTableName As Variant, varFieldName As Variant, varFieldDataType As Variant

    Dim intNoOfFields As Integer, intFieldSize As Integer

    Dim idx As Index, fldIndex As Field

 

That being said, why are you declaring varTableName, varFieldName and varFieldDataType as variants when CStr will never pass a Null value? (in fact, CStr cannot accept a Null value either...)

 

Try changing them to String: that should get by the type conversion error.
