Hi Lisa,

 

I would suggest you follow the steps below to troubleshoot this issue.

  1. Navigate to the affected subsite.
  2. Click the gear and select Site settings.
  3. Select Site permissions under Users and Permissions.
  4. Check which default group is missing in this page. (<subsite> Owners, <subsite> Members, <subsite> Visitors)
  5. Modify the URL of this page to <siteURL>/_layouts/permsetup.aspx.
  6. Create a new group for the missing default group. (You can also use an existing group.)
  7. Select OK.
  8. Then the error should disappear.

tasos_k say: try this..

in an empty cell  write:

ALT+F11 to open VB editor.

Insert > module and paste the code below on the right

Function aaa(rng As Range)
Dim r As Long
Dim rr As Range
r = Cells(Rows.Count, "I").End(xlUp).Row
Set rng = Range("I3:I" & r)
t = 0
For Each rr In rng
If rr.Value = "Long" Then
t = t + 1
If t = 2 Then
aaa = "row = " & rr.Row
Exit Function
End If
End If
Next
End Function

here's a cheating way to do it with a helper column. Lets say your data are in A1:A20. Enter 1 in b1 and drag down so your data looks like below i.e. the helper column has the row number in it and you can hide this column

=IFERROR(INDEX($B$1:$B$20,SMALL(IF($A$1:$A$20="Long",ROW($A$1:$A$20)-ROW($A$1)+1),$E$1)),"")

Question: Can someone please verify that the pre-set 30 days assigned to Deleted Items folder in Office 365 is still valid or not anymore?

 

Yes, the default retention policy (named Default MRM Policy )will take effect on delete items folder unless admin changed it. Every email service is operated by a Retention Policy that deletes items automatically from the ‘Delete’ folder after a certain period of time (30 days). bobgreenblatt say: Well, it depends on exactly what you are trying to do. Office 2008 does not support VBA. Office 2011 (the current version) and Excel 2004, the version prior to 2008 both support VBA.

Your add-in will have to "understand" that certain things you do on windows, like send.keys or direct calls to the windows OS are not supported on the Mac. Likewise you will have to allow for changes in path separator : vs. \ in your file strings.

bobgreenblatt say: Macintosh Excel ONLY allows add-ins created in VBA. Therefore it is highly unlikely that conversion of a windows Excel add-in will be successful.

If the Add-in is written in VBA it is perfectly possible to code it in such a way to have it work on both platforms. You can use a combination of run-time platform determination like application.operatingsystem, or conditional compilation to have the proper piece of code executed for each platform.
