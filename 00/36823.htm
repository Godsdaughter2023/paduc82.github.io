lemp say:

If the + and - keys control the volume when NUMLOCK is engaged, you probably have some third-party hotkey program installed.  Laptop manufacturers often do this as do some sound card drivers/software.

You may find a "customize shortcut" option if you right-click on the volume control icon in the Notification Area (near the clock display). 

While you're in this area, investigate ALL of the icons (don't forget to click the left-pointing arrow to view any hidden icons) using both left and right clicks to see if any of them relate to your sound volume.

What is the make and model of your computer?

What is the make and model of your sound card: 

  • Open Device Manager (Start > Run > devmgmt.msc > OK)
  • Double-click "Sound, video and game controllers
  • List all items in that category

Although you can create your own keyboard shortcuts, they generally require you to use one character (a letter, number, or symbol) plus at least two of the following three keys: CTRL, ALT, and SHIFT.  See https://support.microsoft.com/en-us/kb/310417

That reference that Andrew E provided is merely a list of the built-in Windows shortcuts, which is not responsive to your question.  You can find it here --> https://support.microsoft.com/en-us/kb/126449

There are some third-party utilities that claim to be able to modify the built in Windows shortcuts.  I've not used them, but you can experiment for yourself --> http://www.techmalaya.com/2008/05/23/windows-shortcut-keys-hotkeybind/   See also http://www.downloadzen.com/keyboard-shortcut-to-control-volume-globally-windows-7-vista-xp-volume-shortcut-keys.htm

I doubt, however, that the above utility will be able to change the volume control behavior you describe because, as I said, that's not built-in Windows behavior.

"active"/ start/system partition.

On my box the recovery partition is the first partition, HarddiskVolume1 instead of HarddiskVolume2.  You could ask reagentc /info for a second opinion where your recovery partition really is.  You could use diskpart and temporarily mount the 1st and 2nd partition, assign letters R: and S:, and check which partition contains the RAM disk image file \Recovery\WindowsRE\Winre.wim.  If it's S: your BCD is okay, if it's R: you can try to fix the ramdisk entries, and if it's AWOL you can ask the experts here how to continue.

You really have to use diskpart, the normal disk management won't let you mount the hidden recovery partition.

omniplex say: A common layout for MBR-disks is recovery + system + windows (3 primary partitions.)  It's not necessarily a good idea, but also possible, to arrange it in 2 partitions: recovery + a system/windows combo.  I don't know if your layout, recovery/system combo + windows, is also possible.  

Obviously your combo can be started.  The partition boot code finds the BOOTMGR, the BOOTMGR finds the \boot\BCD, and the BCD entry for your windows partition works as expected.  But the entry for the recoverysequence, a RAM disk image on the same partition, apparently fails.  

You can try reagentc /boottore to verify that this really fails.  It's the same idea as using msconfig to select a BCD entry only for the next reboot.  If it fails spectacularly you need a backup of your actual BCD, and a boot medium with a tool allowing you to restore this backup.
