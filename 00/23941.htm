Copying partial records between Access Tables - Microsoft Community

parishpete say:

Copying partial records between Access Tables

I have a club membership database with current member details on a table.

within the same database I also have a separate table with names and addresses of people on the waiting list to join.

I need an easy way of individually copying those name and addresses, but not the entire record, from the waiting list, into the memberhip table.

Because there are a number of other fields in the membership table to be completed manually, straight copy/paste of complete records doesn't work so at present I have to type all the data manually.

 

The fields to be copied are common to both tables.

 

Any help welcomed

marshall barton say:

You can use an Append query to create a new record in the member table with whatever fields you want from the waiting list table.  The important point to do this is to specify (somewhere) the criteria that uniquely idenitfies the waiting list entry you want to add to the members table.

There are lots of ways to specify criteria. the simpleast and least desirable is to just set the identifying field's criteria to something like:
    [Enter waiting list ID]
A better way is to create a form with a text box for users to enter the waiting list ID along with a command button to run the query.  In thins case, the xriteria would look like:
    Forms![name of the form].[name of the text box]

OTOH, if you have already added a partial member record to the members table, you can use an Update query to set other fields in the member table.  Again the query's criteria is critical on selecting the eaiting list record with the fields to copy.

I think that answers your question, but I feel compelled to try to explain that a better table structure would pretty much eliminate this whole issue.  If you had a people table that contained information about people. then your members table would only need a foreign key field to connect a member specific record to the people table's record with the general information and no data would have to be copied.  A similar arrangement could be used for a waiting list table with a foreign key to the people table or maybe even better would be to have a field in the members table that could be used to distinguish a full member from a waiting list person, a previous, but no longer current member, an honary member, etc.

scottgem say:

I agree with Marshall. This shouldn't be an issue. You should have a people table with a flag to indicate member or waiting list. Then all you need to do is change the flag. Have the membership details in a separate table linked in the personID.
