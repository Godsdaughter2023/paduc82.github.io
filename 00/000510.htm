ken sheridan say: You are doing very much as I suspected.  Having each destination, hospital and diagnosis as a column is 'encoding data as column headings'.  Sometimes this is colloquially referred to as 'committing spreadsheet' because it is akin to the way values would b entered in a worksheet.

A relational database works very differently; it models the part of the real world with which the database is concerned in terms of its entity types (each modelled by a table), the attributes of the entity type (each modelled by a column), and the relationship types between the entity types, which might be represented by a foreign key in one table referencing the primary key of another in the case of a simple one-to-many (or more rarely one-to-one) relationship type, or by a table in the case of a many-to-many relationship type, in which the many-to-many relationship type is resolved into two or more one-to-many relationship types.

It is also important that each table contains no redundancies, i.e. that each 'fact' is stated once and once only.  This achieved by the process of normalization, for which there are formal principles.

To be honest I think this is a big task for someone who is, as you say yourself, very new to MS Access.  Your present model is very badly flawed, and I would not even attempt to try and advise you on how to get the results you describe in the context of the model as it stands.  I would not even attempt to suggest what tables and relationship types are required without a detailed analysis of your organisations modus operandi.

The best advice I can give you is that you spend some time, and it will not be inconsiderable time, gaining a good understanding of the database relational model so that you can restructure your database on a sound basis.  Then you are faced with how to return the information you are seeking, and the design of the queries and reports to do this are far from trivial.  

I realize that this is not what you want to hear, but if you are going to achieve a robust and reliable database which delivers the goods I do not see that you have any alternative but to invest time and effort in learning how it's done.  It's a road all of us who use Access as a tool in our day-to-day work have had to travel.  If you are not able or willing to do so, then the other option is to hire someone with the necessary skills.

ken sheridan say: I'm afraid I cannot picture your model.  If we take the  AdmitToID table as an example, you said in respect of this table 'Columns: All the names of the hospitals'.  So, using the hospitals in my immediate area as an example this suggests that you have a table with column headings like this:

Stafford Hospital    Cannock Hospital    North Staffs University Hospital

If so what values would go in each column?  Does the table have other columns?

You now say 'AdmitTo has an automated number primary key, where each hospital is numbered 1, 2, 3, 4, etc.' which suggests a table like this:

HospitalID    Hospital
1                 Stafford Hospital
2                 Cannock Hospital
3                 North Staffs University Hospital

The latter would be the correct design for a table Hospitals, i.e. it is modelling the hospitals entity type. Other columns would represent other attributes such as the hospital's address etc.

The patients entity type would be modelled by a table like this:

Patients
....PatientID (PK)
....FirstName
....Lastname
....DoB
....etc, so a row for me would be like this:

PatientID    FirstName    LastName    DoB
42             Ken              Sheridan     1946-11-01        

When it comes to patient's admission to a hospital this is a many-to-many relationship type between patients and hospitals, so could be modelled by a table like this:

PatientAdmissions
....PatientAdmissionID (PK)
....PatientID (FK)
....HospitalID (FK)
....AdmissionDate
....DischargeDate
....etc

So if I were admitted to North Staffs on 27 May 2013 and discharged today:

PatientAdmissionID    PatientID    HospitalID    AdmissionDate        DischargeDate
187                          42              3                 2013-05-27              2013-06-28

Other columns in this table would represent other attributes of the patient admissions relationship type.  This table would be referenced by other tables such as PatientDiagnoses, which itself is modelling a many-to-many relationship type between patient admissions and diagnoses, which would also be modelled by a table Diagnoses.

So, each table models an entity type or a relationship type (a relationship type is really just a special kind of entity type).  Each column in the table models an attribute of the entity type, and the attribute is determined solely by the key of the table, not by any other non-key attribute, eliminating redundancy and the consequent risk of update anomalies.

When it comes to returning information of the type you wish this is computed in queries from the data in the tables.  For instance the time between a patient being discharged back from a hospital and then readmitted to the hospital is the admission date value in one row less the latest discharge date value in another row for the same patient where the PatientID is the same and the discharge date is prior to the admission date.  This computed by joining two instance of the PatientAdmissions table.  Whether or not the readmission is for the same condition ids determine by the values in the PatientDiagnoses tables for each if the admissions.  Bear in mind that an admission could result in more than one diagnosis, so the query would need to determine a match in any two rows in two sets of rows.

Now all of this can be done, but the point I was trying to get across initially was that it needs the necessary skill base to be able to do it.  Writing the queries for these sort of tasks is not easy, and requires an understanding not only of how to model the data correctly, but how to determine the logic which will give the answers you are looking for, and how to translate that logic into queries, which in turn requires a knowledge of the SQL language as it is not the sort of thing which can be done easily in the point-and-click query design interface.

The reason I quote the great Richard Feynman below my signature is that that learning should come through understanding, not by rote, a point which he stresses very strongly in his writings.  Looking at a few tutorials and seeing that they tell you to do this, that and the other achieves little unless you understand why you have to do this, that and the other, and are able to apply that understanding to other situations.  Understanding does not come easily; it takes time and you have to work at it.

So when you ask 'How should I continue forward?' the answer I would give is gain the necessary understanding so that you know how to continue forward.  It may be possible to install a copy of Word, open a new file and write a perfectly presentable document without any prior knowledge.  Access is not the same, you can download a template a catalogue your DVD collection or whatever with no trouble, but once you go beyond that and need to create a customized database application you are then in a development environment, and need to be comfortable in that environment.

Having said that, it may be possible for you to cobble something together from the data as you have it to give you the results you want.  As some experienced programmers used to say about the original Basic programming language, you can get into your house through the letter box if you push hard enough, but it's not the best method.  There may be others here prepared to help you do so, but not me.  I'm more than happy to help you achieve a good solution, though.

ken sheridan say: To build a data base composed of correctly normalized tables you design those tables in accordance with 'Normal Forms'.  These are a set of rules, governed by formal principles as set out below.

To keep things simple let's look at the first three normal forms.  This is not to say that the higher normal forms are unimportant, and where necessary a database should be normalized to as high form as possible, but in most cases a table which is normalized to Third Normal Form will also be normalized to the higher normal forms.

Before looking at the normal forms themselves, let's look at the terms used in the formal definitions:

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).  

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple contains exactly one value for each attribute.

This means that in each row in a table there should not be multiple values of an attribute, e.g. in a table of Orders there should not be multiple values in one column for the items ordered, Apples,Bananas,Oranges for instance, or in separate columns named Item1, Item2, Item3.  If the table were to have two columns OrderNumber and Item it would be in1NF as each row has one value of each of the two attribyutes.

Each of the two columns is a foreign key referencing the primary keys of Orders and Items tables respectively.  In combination the two columns are the key of the table, so can be designated as its composite primary key.  

Second Normal Form:  A relvar is in second normal form if and only if it is in 1NF and every non-key attribute is irreducibly dependent on the primary key.

This means that, if a table is in 1NF as defined above, each column in it which is not a part of a key must be determined by the whole of the key.  Let's assume we have a table or orders with columns OrderNumber, Item and Customer.  The table is in 1 NF because in each row we have only one value of each of the three attributes.  The tables key, however, is a composite one of OrderNumber and  Item as the values in these two columns in combination uniquely identify each row.  However, Customer is determined by OrderNumber as each order is by one customer only, so this is not determined by the key, but by only one part of it.  If we decompose this table into two tables with columns OrderNumber and Item in one, and OrderNumber and Customer in the other, the two tables would be in 2NF as the first table would be 'all key', i.e. its key is a composite one of its only two columns.  In the second there would be only one row per OrderNumber value as this is the key and Customer is determined by its value and nothing else.

Third Normal Form: A relvar is in 3NF if and only if it is in 2NF and every non-key attribute is nontransitively dependent on the primary key.

For this let's assume our second table from the above example has, in addition to the OrderNumber and Customer columns, a CustomerAddress column.  We've already seen that Customer is determined by the key, OrderNumber.  CustomerAddress  is also determined by the key in that, once we know the order number we know the address of the customer placing the order.  However, CustomerAddress  is also determined by the non-key attribute Customer as, once we know who is the customer we know their address (we are assuming for this example that customers never change their address).  So CustomerAddress is transitively determined by the key, OrderNumber, via Customer, so the table is not in 3NF.  If we decompose the table as follows:

Orders
….OrderNumber (key)
….Customer

Customers
….Customer (key)
….CustomerAddress

then in each table the non-key column is determined solely by the whole of the key, and each table is consequently in 3NF.

In reality we'd use 'surrogate' numeric keys, so the tables would be:

 Orders
….OrderNumber (primary key)
….CustomerID

Customers
….CustomerID  (primary key)
….Customer
….CustomerAddress

CustomerID would be an autonumber in Customers, of which it is the primary key, and a long integer number in Orders, in which it is a foreign key referencing the primary key of Customers.

Orders has a many-to-many relationship type with Items, i.e. each order can be for multiple items, and each item can be ordered in multiple orders.  A many-to-many table is modelled by a further table which resoles it into two or more one-to-many relationship types, so in this case we'd have the following tables:

Items
….ItemID  (primary key)
….Item
….UnitPrice

OrderDetails
….OrderNumber (foreign key)
….ItemID  (foreign key)
….UnitPrice
….Quantity

The primary key of OrderDetails is a composite one of the two foreign keys OrderNumber and ItemID.  Note that both Items and OrderDetails columns have UnitPrice columns.  This might seem to mean that OrderDetails is not in 3NF as it is determined by one part of the key, ItemID rather than solely by the whole of the composite key.  This is not the case, however, as prices change over time.  The unit price in Items is the current price of the item, and will be updated each time the price changes.  The unit price in OrderDetails, on the other hand, is that at the time when the order was made and is obtained from the current price in Items when the row is inserted into the table, but thereafter remains static regardless of any changes to the price of the item.  Consequently UnitPrice in OrderDetails is determined by the whole of the key, and the table is in 3NF.

Why does all this matter?  Well, if tables are not correctly normalized they will contain redundancies, i.e. each 'fact' might be stated multiple times in different rows.  This allows update anomalies where 'facts' in different rows are not consistent with each other, so the data is unreliable.

Method : Use the Select by Type command or the Select Special command
In Microsoft Office Visio 2007, Microsoft Office Visio 2003, and in Microsoft Visio 2002, you can use the Select by Type command. In Microsoft Visio 2000, you can use Select Special command. To use the Select by Type command or the Select Special command, follow these steps:
1. Start Visio, and then open your drawing.
2. On the Edit menu in Visio 2007, Visio 2003 or in Visio 2002, click Select by Type. Alternatively, click Select Special in Visio 2000.
In Visio 2010, on the Home tab, in the Editing group, click Select, and then click Select by Type.

o  Click Shape type. Then, click to select or click to clear one or more of the following check boxes:
§  Shapes
§  Groups
§  Guides
§  OLE objects
§  Metafiles
§  Bitmaps
Additionally, the following shape type is available in Visio 2003:
§  Ink objects
o  Click Layer, and then click to select the check box next to the layer or layers that you want.
4. Click OK.
5. On the Edit menu, click Copy.

In Visio 2010, on the Home tab, in the Clipboard group, click Copy.
6. Switch to the destination file to which you want to paste the Visio object.

For example, if you want to paste the Visio object to a Microsoft Word document, start Word, and then open the document to which you want to paste the Visio object.
7. On the Edit menu, click Paste or Paste Special to insert the Visio object.
