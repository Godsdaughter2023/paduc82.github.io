I reinstalled my old WMC setup again by using an Acronis backup (the setup that is stuck at zap2it), made a backup of the eHome directory,  and started testing which file in the eHome directory is responsible for the guide content, by deleting files and reinstalling them again, a tedious job.

The media guide content turns out to be a file named “mcepgX-X.db” and a directory “mcepgX-X”, all in the c:\program data\microsoft\eHome directory. The “X-X” means it can be different for your PC, for instance for me it is “mcepg2-0” (will always be the same X-X- for the directory name and file name) for you it might be different like “mcepg1-0” depending how many times you have rerun the WMC setup again in the past.

There might be even more than one mcepgx-x file and directory, the one with the highest numbers for x-x is the one that is active.

So I deleted both the mcepgx-x file and directory on the problem PC, and copied these from a PC where the switchover to Rovi DID happen successfully earlier. Then I installed these copied files on the problem PC (make sure you rename the files so the x-x- matches the original file names, as these x-x values might be different on the PC from which you copied the files)

Then I crossed my fingers, rebooted the PC and started WMC again. Upon startup, WMC responds immediately with a message that there is a problem with the database file, and that it will therefore reload program data again to fix the issue (make sure you are connected to the internet at this stage). When WMC was all done with that, I had my guide back (all the way to August 6 as it should) and things had switched over to Rovi!! (check the “about” under “Guide” to verify this)

So all you need is a PC where Rovi already works, and then you copy the mcepgx-x.db file and mcepgx-x directory from that PC to the problem PC. Do NOT delete the entire eHome directory first, leave it intact. Just replace the mcepgx-x file and directory.

john_barnett say:

Hello grandmalg,

 

Are you actually booting from the CD/DVD drive? I can understand the 'boot manager missing' if your system is by-passing the CD/DVD but not if you are booting directly from the CD/DVD drive.

 

What happens when you insert the DVD into the DVD drive? Does the DVD drive start whirring and accessing the DVD drive or does it quickly go to 'boot manager missing' error?

To boot from the DVD the DVD 'must' be the first boot option. If you have a modern PC you should see an option when you first boot to press an F key. This maybe F12 (it maybe another F key) to access the boot menu. If this is the case you need to press the relevant F key and a menu will appear. From the menu you need to select your DVD drive. Your original CD or restore CD should then start installing.

 

If you have an older machine (one without the F key option) you need to access the BIOS and change the boot menu so that the CD/DVD is the first boot option. This link will give you an idea of how to change the boot order: http://www.hiren.info/pages/bios-boot-cdrom There maybe subtle differences depending on which BIOS your PC has.

What I think is happening is that you are actually bypassing the DVD boot up and the system is trying to access the hard drive first, hence the 'boot manager missing' error as the system cannot find the windows boot manager.

Once you have the boot option correct you should be able to use the 'repair' option on your original CD to repair the boot manager. If this doesn't work then use your restore disc to restore the operating system.

milumiu say:

I don't have an answer, but this is a good question. The notions of DLNA and Homegroup are sort of blurred, or muddled into each other. I mean, what is it technically? The question probably wouldn't arise if it always just worked, but it doesn't. Videos don't show up or do show up but are not playable.

DLNA is HTTP transfer of media data, after preliminary setup using UPnP messages. It doesn't need SMB at all.

The Homegroup, on the other hand, is an SMB share for a special account (HomeGroupUser$) authenticated by the homegroup password, which can be read in the homegroup settings. It is just SMB and doesn't have anything to do with UPnP, media, codecs, or HTTP.

Technically, DLNA could function without a homegroup. The WMP network service (which I think handles DLNA) just needs filesystem level read access to the files to be shared, and then streams them over the network as requested. No SMB required at all.

Conversely, SMB obviously doesn't need anything DLNA related.

So what is the relevance of the Homegroup to DLNA on Windows 7 (and 8 and 8.1)?

It would be useful to have some means of verifying which files are actually served up via DLNA, and which ones are not, and why not (container, codec, permissions, whatnot).
