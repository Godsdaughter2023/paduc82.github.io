ken sheridan say: My interpretation of what you have described is that you have two entity types, personal and corporate entities, each of which can have one or more phone numbers, i.e. each is the referenced entity type in a one-to-many relationship type in which phone numbers is the referencing entity type.  Each member of each entity type, personal and corporate, therefore, is also a member of a single entity type, contacts.

What you have, therefore, is a type Contacts, with two subtypes PersonalContacts and CorporateContacts.  A subtype is characterized by sharing all attributes of its (super) type, but not those of other subtypes.  In a relational database this is modelled by having one table to model the type, and multiple tables, each of which models a subtype.  The relationship type between the table modelling the type and each table modelling a subtype is one-to-one, in which the type table is the referenced table, i.e. the primary key of each subtype table is also a foreign key referencing the primary key of the type table. This means of course that, while the primary key of the type table can be an autonumber, that of each subtype table cannot, so their primary keys will be of straightforward long integer data type.

So, you would have a Contacts table in which all contacts, both personal and corporate are represented.  Its columns would represent those attributes which are common to both personal and corporate contacts.  You would then have PersonalContacts and CorporateContacts tables, each referencing Contacts in a one-to-one relationship type.  The columns of each of these tables would, in each case, represent the attributes which are specific to the subtype in question.

Your phone numbers table would thus reference the primary key of Contacts.  Through the one-to-one relationship types between Contacts and PersonalContacts, and Contacts and CorporateContacts, each row in the phone numbers table references a row in one of those tables.

The same would be true of Addresses if there is no relationship type between phone numbers and addresses, i.e. if you are simply storing one or more phone numbers for each contact, and one or more addresses for each contact, but with no indication of a phone number relating to a specific address.  This would be true of mobile phone numbers of course, but landline numbers would normally be related to an address.

To relate one or more phone numbers to an address, therefore, the relationship type would not be between PhoneNumbers and Contacts, but between PhoneNumbers and Addresses.  Addresses would reference Contacts.  However, this leaves mobile phone numbers out in the cold as they do not relate to one specific address.  The solution is to relate a mobile phone number to each of a contact's address, but this means the relationship type between phone numbers and addresses is now a many-to-many relationship type.  This would consequently require an AddressPhoneNumbers table to model the many-to-many relationship type by resolving it into two one-to-many relationship types.  The relationship types would thus be like this:

PhoneNumbers---<AddressPhoneNumbers>----Addresses>----Contacts

    Contacts-----PersonalContacts

    Contacts-----CorporateContacts

If phone numbers are classified as landline or mobile it would not be difficult to write some code which would automatically assign a mobile number to all of a contact's addresses when a mobile number is inserted for any one of a contact's addresses.  Similarly all of a contact's mobile phone numbers could automatically be assigned to a new address when it is inserted for a contact.

The other question which arises is whether there is a relationship type between personal and corporate contacts?  If so, can a personal contact be a contact for more than one corporate contact, in which case the many-to-many relationship type between PersonalContacts and CorporateContacts would be modelled by a further table which resolves it into two one-to-many relationship types, rather than by a CorporateContactID foreign key column in PersonalContacts.

You said earlier that you have never quite grasped many-to-many relationship types.  You might care to take a look at a couple of demo files in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

1.  DatabaseBasics.zip is a simple introduction to relational database design in Access, and illustrates the various relationship types by means of simple examples.

2.  Relationships.zip illustrates how relationships are built up across a database, using a simple medical prescriptions database as its example.

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.
