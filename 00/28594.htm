ken sheridan say:

The way to do this is to model it by a set of correctly related tables.  What you are talking about are many-to-many relationship types, which, in a relational database are modelled by tables which resolve the relationship type into two or more one-to-many relationship types.  You must represent all the data by values in tables, not by value lists.  This is a fundamental principle of the database relational model, The Information Principle (Codd's Rule #1), viz:

 

'The entire information content of the database is represented in one and only one way, namely as explicit values in column positions in rows in tables'.

C J Date - Introduction to Database Systems; 7th Edition; 2000

 

1.  Locations:  This is a straightforward binary (2-way) many-to-many relationship type between tutors and locations, so in addition to a Tutors table and a Locations table you'll have a TutorLocations table.  The tables are related as follows:

 

Tutors---<TutorLocations>----Locations

 

TutorLocations will have foreign keys TutorID and LocationID referencing the primary keys of Tutors and Locations respectively.  In combination the two primary keys are the composite foreign key of the table.  The table might have other non-key columns representing other attributes of the relationship type.

 

2.  TutorAvaialability:  Again you need to model the binary relationship type between Tutors and Sessions table by a TutorSessions table in the same way.

 

3.  Subjects and Age Group:  You will firstly need to model the relationship types between Tutors and Subjects, and between Tutors and Age Groups in the same way as above, with TutorSubjects and TutorAgeGroups table.  If a tutor does not offer all subjects to all age groups, and vice versa, you will also need to model the relationship type between the TutorSubjects and TutorAgeGroups by another table with three columns TutorID, SubjectID and AgeGroupID, all three columns comprising the composite primary key of the table.

 

For the interface forget about using multi-select list boxes, they are unnecessary and you cannot use them without writing code as it is necessary to iterate the list box's ItemsSelected collection.  Within a Tutors form embed three subforms, each in continuous forms view, each based on one of the three tables which model the relationship types.  To save space you can place each subform on a separate page of a tab control if you wish.  Each subform is linked to the parent form on TutorID.  Taking locations as an example, a row is inserted into the subform for each location at which the tutor is available for work.  The location is select from a combo box bound to the LocationID column in TutorLocations, set up as follows:

 

ControlSource:    LocationID

 

ControlSource:   LocationID (for a bound control; leave blank if the combo box is an unbound 'navigational' control)

 

RowSource:     SELECT LocationID, Location FROM Locations ORDER BY Location;

 

BoundColumn:   1

ColumnCount:    2

ColumnWidths:  0cm

 

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

you can nest formulas in other formulas. You don't say what formula you use to pull data from the chart, but let's say it's a VLookup formula.

If you calculate the date with a formula, and if you would like to use the result in a Vlookup, then you can use something like this:

Try to reset all Skype settings.
