jeffkalv_msft say:

Thanks for the posts. I can confirm the envelope badge has been removed from versions 15.33 and higher.

We removed the envelope badge for two reasons:

1) Show unread count only. We were conveying two different pieces of information in one place: unseen message(s) with the envelope, and unread count across inboxes with the red badge. The red badge count is common for apps on Mac and iOS, and we decided to keep this badge and remove the envelope, which is more of a Windows Outlook convention. Windows Outlook also shows badges on the taskbar like the alert exclamation point which Outlook for Mac doesn't show.

2) Appearance. The envelope badge itself was overlapping the “O” in the Outlook icon to create the appearance of a lock icon, prompting feedback around whether Outlook was in some sort of locked state. It also didn’t help that the icon for Outlook 2016 has an envelope within it, too, so showing the envelope for unseen messages created a double envelope appearance.

I have linked a folder in an Outlook pst file.  I need to make a table in Access to add some fields.  The rows in the Access table will correspond 1 to 1 with the records in the pst file. -- I need to know what fields Outlook (2010) uses to serve as primary key so I can use them in the Access table to link to the corresponding records in the linked pst table.  I can probably determine it with a query and the unique values property but was hoping for an easier way.
karl timmermans [outlook mvp 2012-2018] say:
To put this is absolute terms - Outlook data is not a relational database employing the concept of "keys" as the terms is normally used and as such there, other than the EntryID field which is guaranteed to contain a unique value for any item within a PST file regardless of item type, there is no field or combination of fields that would absolutely "guarantee" that no duplicates would exist. The best that you could achieve is the selection of fields that would "minimize" the potential of duplicates being found which is based largely on the source of your original data.

Sign in to your account.

Saving a document on screen to another drive - Microsoft Community

highstream say:

Saving a document on screen to another drive

In Word 2010, is there a way to save a file to another drive - flash drive in this case - without changing the location of the document currently on screen?   Save As changes the drive of the current document.  Given the ubiquity of laptops and the need to save copies for use there, I would think there would be an equivalent of Win Explore's "Send to," but I can't find it.  Am I missing something?  Thanks,
suzanne s. barnhill say:

You can use Send To in Word as well. In either the Open or the Save dialog in Word (either of which is an Explorer window), select the file, right-click, and choose Send To. All active drives, plus any other locations you've added to the Send To folder, will be listed. Select the appropriate one.

Note also that this (using Send or Copy/Paste) is the correct method of getting a Word document from the hard drive to an external drive; saving directly to the drive from Word poses a strong risk of document corruption.

suzanne s. barnhill say:
Btw, for anyone coming across this, with a document already saved that would be the Open or the Save As dialog.

Of course. It would be Save As even if the document weren't saved because that's the dialog you get the first time you save. And you couldn't send a document that hadn't been saved. In fact, of course, there is no Save dialog, and I don't know why I just tend to think of Save As as Save.

After further investigation kindly assisted by a technician at the Spamhaus Project, some light appears to have been shed into the issue and it appears that the problem is not on the end-user side – as many Godaddy's support staff believe – but on Godaddy’s server side.

 

As it seems, Godaddy’s Support staff are being – erroneously – instructed and relaying the information to users that Microsoft’s block to e-mails sent from Godaddy’s hosting servers is due to some customers practicing spam and affecting everyone on the same server. However, last night I spoke once again with Godaddy’s Support and they confirmed what Spamhaus hinted at. Whenever they run an IP search on the Spamhaus lists, Godaddy’s servers don’t come up. If the problem were some customers spamming, then the server should be listed.

 

This was Spamhaus's response:

 

“If so, we'd not only hear instantly from GoDaddy (and all their customers) but GoDaddy themselves would be able to immediately remove all GoDaddy IP ranges from the PBL themselves via their PBL account.”

 

What is happening instead is a completely different matter. Many e-mails sent from Godaddy’s maillservers are not leaving identified with a Godaddy’s Source IP – as they should – but instead with end-users IP addresses as a Source IP, that belong to broadband internet providers, or even with an unknown IP. As Spamhaus also explained:

 

“It's normal for that IP to be on the PBL (all end-user/home broadband IPs in the world are on the PBL).”

 

The server’s behavior is also erratic. I ran several tests sending emails using Godaddy’s webmail – therefore isolating any chance that the problem could be my e-mail program configuration (which is what Spamhaus initially presumes to be the cause – to both my personal Outlook and my domain address and confirmed this myself:

 

  • The vast majority failed to arrive at @outlook.com but all arrived at my own domain's inbox and were either identified with a Source IP corresponding to my home IP or an unknown IP (127.0.0.1)
  • The very small minority that arrived at my Outlook.com inbox was correctly identified with a Source IP corresponding to Goddady's mail server, which is what should happen all the time

 

It would be great if other people like yourself could test this themselves and replicate my experience.

 

I have also relayed this information to Godaddy’s Support Staff (in Brazil) and to their HQ, in the hope that some solution may be found, other than asking their customers to purchase additional Microsoft’s services to work around their own server’s erratic behavior.
