Easily done by use of a couple of macros to insert the sections, one for landscape and one for portrait.

Sub Landscape()
Dim newSec As Section
Selection.InsertBreak Type:=wdSectionBreakNextPage
Set newSec = Selection.Sections(1)
With newSec
    With .PageSetup
        .Orientation = wdOrientLandscape
        .LeftMargin = CentimetersToPoints(2)
        .RightMargin = CentimetersToPoints(2)
        .BottomMargin = CentimetersToPoints(2)
        .TopMargin = CentimetersToPoints(3)
    End With
End With
End Sub
Sub Portrait()
Dim newSec As Section
Selection.InsertBreak Type:=wdSectionBreakNextPage
Set newSec = Selection.Sections(1)
With newSec
    With .PageSetup
        .Orientation = wdOrientPortrait
        .LeftMargin = CentimetersToPoints(2)
        .RightMargin = CentimetersToPoints(2)
        .BottomMargin = CentimetersToPoints(2)
        .TopMargin = CentimetersToPoints(3)
    End With
End With
End Sub

nathan o. - microsoft support say:

By default, when you open Access 2007 or Access 2010 it still joins to an .mdw file.  If Access is unable to create this file or join to the one it is currently joined to, you will receive this error when you open an .mdb file or an .accdb file.

 

During a standard install of Access 2007 or Access 2010, the install will automatically create a default .mdw file and place it in a folder similar to the following:  C:\Users\<your user name here>\AppData\Roaming\Microsoft\Access (this is the path on a Windows 7 machine).  If you don’t have permissions to create a file here, this file won't get created, and you will see an error message similar to the one you are seeing.  If Access is currently working for you, as a test, you could remove your .mdw file from this folder and then deny all permission to the folder this file was in.  The next time you try to open Access, you should see this same error (at least I did).

 

This may not be the only cause of this error, but I would recommend making sure you have permissions to the above mentioned folder and make sure you can create files in this folder.  After you have the proper permissions to this folder, test opening Access again.  If it still fails, then try creating a new system.mdw file.  You can create it in the above mentioned folder or in a folder that you know you have permission to (ex. documents folder).  To create this file, you can do the following:

 

+>Open Access by itself.

+>Press Ctrl + G to open the VBA window.

+>Within VBA window ensure your cursor is in the immediate window and type the following: 

DoCmd.RunCommand acCmdWorkgroupAdministrator

+>Put your cursor at the end of this line and then hit Enter.  This should launch the workgroup administrator.

+>In this window, click the Create button.

+>In the next dialog, just click Ok.

+>When prompted with a message that states you haven’t entered a workgroup id, just click Yes.

+>In the next dialog, click the browse button and browse to a folder that you want to save the file to.  You can give it a name of system.mdw and then click the open button.

+>This should then bring you back to a screen that shows the file path and the file name of the system.mdw file.  Click Ok on this box and then click Ok again.

+>This should then bring you back to the Workgroup Admin screen and it should now show  you are joined to the workgroup file you just created.  Click Ok again.

+>Close Access and try opening your database again.

 

If you receive an error during this process, please provide the exact text of that error.

Azure AD sync with password synchronization and the Exhange Online Cutover migration tool

ynnhoj gnahz say:

Hi 365 Questions,

 

Thanks for the updates.

For your question “I want to deploy AAD Connect with password sync Before cutover migration completes - is that possible with a cutover migration?”, the answer is no. We need to finish the Cutover migration first. The reason is Cutover migration uses the value of the mail/WindowsEmailAddress attribute while DirSync will provision them based on the UPN. If you already have object with matching 'mail' attribute in the cloud, you will either end up with an error or duplicate account during the migration.

For your question “Also is there more info on how and why we will need to convert the on-premises mailboxes to mail enabled users?”, the reason why we will need to convert the on-premises mailboxes to mail enabled users is that when we convert on-premises mailboxes to mail-enabled users (MEUs), the proxy addresses and other information from the Office 365 mailboxes are copied to the MEUs, which reside in Active Directory in your on-premises organization. These MEU properties enable the Directory Synchronization tool, which you activate and install DirSync, to match each MEU with its corresponding cloud mailbox.
