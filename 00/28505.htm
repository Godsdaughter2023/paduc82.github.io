You have used terms incorrectly when speaking of Access relational database.

1- A form can only display information, not store it.

2- A database has tables which in turn has records that have fields.  You would not have separate 'databases' per driver.

3- You would have a table for drivers just containing their personal information.  You would have another table related to the driver for their activities.

4- You would use a form or report that has a query to display the current activity (or past OR future activity) of the driver.

So, how many databases do you have?   What tables do you have (name?) and their field names and datatype - DateTime, text, number, etc.?

ken sheridan say: I'm not sure whether what I take to be Karl's interpretation of your post is correct, i.e. that when referring to separate 'databases' you are in fact referring to separate 'tables'.  It sounds to me like you might actually be proposing to have completely separate databases per driver, i.e. one file for each driver with each file containing the same set of table definitions, but with each file containing data for the one driver only.

If it's the latter which you have in mind this would only be appropriate if the files were to be used remotely from each other, without any means of real-time interconnection, e.g. if the data is to be inserted by drivers while away from base and without any means of accessing a common database.  This is feasible, and I have done this myself in the past, but it is far from trivial to implement, and not something I would recommend for a relatively inexperienced developer.  Access at one time included 'replication' functionality to do this, but it was more trouble than it was worth, and the need has largely been superseded by shared online access.

If there is no need for disconnected input away from the base, having separate databases for each driver would be a big mistake, as you can easily return data per driver from a single database for periodic reports.  This requires that the database's logical model comprises a set of correctly normalized related tables which accurately model the real world situation in terms of its entity types, the relationships between them, and the attributes of each entity type.  Forms and report are merely the means of interfacing with the data; it's the table structure which is the basis of a successful relational database.  You've not provided us with any information on your database's logical model, however, so it's not currently possible for us to comment on its validity, or how you can achieve your aim of returning driver specific information.

Memory card is not recognized

Memory card is not working as expected

Format your memory card on a PC

Important! Before formatting your memory card, make sure you have a backup of the data as it is lost during the formatting:

  1. Connect your memory card to a PC with a USB reader that has a microSD card slot. If your card reader or PC has a regular SD card slot, you can use a microSD card adapter.

  6. Right-click the memory card, select Eject, and wait until it is safe to remove the card.
  7. Insert the memory card into the memory card slot in the phone until it locks into place.

john korchok say:

Steve has the right direction, but there are a couple of precautions you have to take with a Mac:

  • You can't use the OS X Archive utility to unzip or zip the files. Instead, you need to open a Terminal window and use the unzip command.
  • You can't look at the files using a Finder window. Using a program's File>Open dialog or using Terminal to look in folders is fine.

OS X automatically creates hidden .DS_Store files in each folder viewed by the finder, and PowerPoint will not tolerate these in the final file. You can read more detail in this post: XML Hacking: Editing in OS X

After unzipping the PowerPoint files, look in ppt/media for file with names like media1.wav. If you recorded the whole narration in one go, there will be one big audio file.

techbuzzireland say: The screen gets turned off during a phone call when the proximity sensor is covered. This is done to avoid accidental touch screen actions when keeping the phone close to the ear during the phone call. The proximity sensor is located in the top part on the display side of the device (e.g. Nokia N8-00, C6-01, C7-00, E7-00, Nokia Lumia phones).
If the screen gets turned off immediately when dialing a number or receiving a call, check that you are not covering the proximity sensor e.g. with your finger. Check also that there's no protective foil tape on the screen which would cause the proximity sensor to react too easily and turn off the screen when still in the process of setting up the phone call. If using 3rd party screen protectors, the protector may not be of the right size and will cover the proximity sensor, causing blank screen issues during the calls.
