You cannot delete the folder, it is forced to give you by Microsoft, just as the "3D Builder", "Alarms & Clock", "Calculator", "Calendar", "Camera", "Cortana" etc. items in start menu is forced and imposed on you by Microsoft, however you cannot delete it, but you can control it.

To own it, please execute the following batch file:
OwnAppX.bat
takeown /F "%ProgramFiles%\WindowsApps" /R /A /D Y
icacls "%ProgramFiles%\WindowsApps" /grant Administrators:F /T /C


There are two parts of the universal apps you have: one are the Microsoft default universal apps that are forced and imposed on you by Microsoft, one are those universal apps you downloaded and installed from Store, I called them "user's customed universal apps".
You can change and custom the folder of your customed universal apps.

For the first time, you can set the customed universal apps' path or folder by running the Registry Editor (regedit.exe or regedt32.exe), 

for example, I wanna change and custom the folder of my customed universal apps

from the default folder "%ProgramFiles%\WindowsApps" to my customed folder "E:\Software\Win10Apps", so that I could execute the following steps:
rightclick
HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\Appx

click "Advanced" button
click "Change"
click "Advanced..." button
click "Find Now" button
click "Administrators"
click "OK" button
click "OK" button
click "OK" button
click "Administrators"
click and select "Full Control - Allow" checkbox
click "OK" button
"PackageRoot"="E:\Software\Win10Apps"

Later, once after you already had taken the Administrators ownership of this registry hive key, you also can run a PowerShell script to set it.

For example, I change and custom the folder of my customed universal apps

from the default folder "%ProgramFiles%\WindowsApps" to my customed folder "E:\Software\Win10Apps", could run the following PowerShell script:
SetCustomedAppxRoot.ps1
New-Item "E:\Software\Win10Apps" -ItemType Directory
$key = [Microsoft.Win32.Registry]::LocalMachine.OpenSubKey("Software\Microsoft\Windows\CurrentVersion\Appx",[Microsoft.Win32.RegistryKeyPermissionCheck]::ReadWriteSubTree,[System.Security.AccessControl.RegistryRights]::TakeOwnership)
$acl = $key.GetAccessControl([System.Security.AccessControl.AccessControlSections]::None)
$owner = [System.Security.Principal.NTAccount]"Builtin\Administrators"
$acl.SetOwner($owner)
$key.SetAccessControl($acl)
$key = [Microsoft.Win32.Registry]::LocalMachine.OpenSubKey("Software\Microsoft\Windows\CurrentVersion\Appx",[Microsoft.Win32.RegistryKeyPermissionCheck]::ReadWriteSubTree,[System.Security.AccessControl.RegistryRights]::ChangePermissions)
$acl = $key.GetAccessControl()
$rule = New-Object System.Security.AccessControl.RegistryAccessRule("Builtin\Administrators","FullControl","ContainerInherit","None","Allow")
$acl.SetAccessRule($rule)
$key.SetAccessControl($acl)
Set-ItemProperty -Path "HKLM:\Software\Microsoft\Windows\CurrentVersion\Appx" -Name "PackageRoot" -Value "E:\Software\Win10Apps"
$key = [Microsoft.Win32.Registry]::LocalMachine.OpenSubKey("Software\Microsoft\Windows\CurrentVersion\Appx",[Microsoft.Win32.RegistryKeyPermissionCheck]::ReadWriteSubTree,[System.Security.AccessControl.RegistryRights]::ChangePermissions)
$acl = $key.GetAccessControl()
$rule = New-Object System.Security.AccessControl.RegistryAccessRule("Builtin\Administrators","ChangePermissions,ReadKey,QueryValues,EnumerateSubKeys,Notify","ContainerInherit","None","Allow")
$acl.SetAccessRule($rule)
$key.SetAccessControl($acl)

ForeachAppX.ps1
Get-AppXPackage -AllUsers | Foreach {Add-AppxPackage -DisableDevelopmentMode -Register "$($_.InstallLocation)\AppXManifest.xml"}

Ah, wait. Apologies - I do not myself use Structured References very often and so was unaware that you could in fact use the reference to the entire column's values and that Excel will, for certain functions, interpret that in a row-relative manner.

But not for all functions, it would seem. For some, it appears as if Excel makes the assumption that you in fact wanted the reference to refer to all of the values in that column, not just the single value in the same row as the formula.

For example, if you use e.g.:

=SUM([ID])

then, based on your data, you end up with 15 in all rows, since, for the SUM function, Excel interprets the reference [ID] as being one to the entire column.

Whereas, with other functions, e.g. MOD, Excel interprets the same reference in a row-relative manner, so that, rather than referring to all cells within that column, it instead refers to just the value in the same row as the formula.

It would be interesting to see a list of which functions are treated in which manner. My guess would be that, when using certain functions which usually operate over more than a single value, e.g. SUM, Excel makes the assumption that a reference to the entire column was intended. It would appear that - perhaps strangely - QUOTIENT also falls into this category.

Also quite strangely, we can continue to use the same referencing but with a very small coercion, viz:

=QUOTIENT([ID]+0,4)

which now returns, correctly: 0, 0, 0, 1, 1.

So the additional, seemingly superfluous coercion of the values within the range via the addition of zero forces Excel to reinterpret the reference to the ID column this time as being a row-relative one.

Very interesting and seemingly inconsistent behaviour.

Hopefully someone can shed some light!

scott mcdaniel say:

I don't use macros, so can't really help with that, and I'm not even sure you can do this with a macro.

You can do it with VBA, but you have to first create an Export Spec manually. To do that:

  1. Select External Data - Export - Text. Do NOT select any of the "Specify Export Options" boxes
  2. Click OK
  3. Click the Advanced button 
  4. Make sure the Field Delimiter is set to the comma
  5. Click the Save As .. button
  6. Enter a name for your Export Spec. Be sure to remember this, or write it down.
  7. Click OK >> OK
  8. Back in the Export Text Wizard, click Finish

You can now use the Export Spec you saved when exporting via VBA. To do that:

DoCmd.TransferText acExportDelim, "YourSpecName", "YourTableName", "Full File Location"

Obviously you'd have to change YourSpecName and the other arguments to match your own. Note the "Full File Location" is the full path where you want to save your file.

See here for more information on using TransferText:

https://msdn.microsoft.com/en-us/library/office/ff835958.aspx?f=255&MSPPError=-2147217396
