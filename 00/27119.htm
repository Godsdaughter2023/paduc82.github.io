john w. vinson say:

The problem is that you got sucked into using Microsoft's misdesigned, incompletely implemented, frustrating and almost never necessary Multivalue Field misfeature.

People have been developing relational databases for about half a century without multivalue fields. It's very straightforward! In your case you have a Many to Many relationship.  Each Team has zero, one, or many People; each Person can work on zero, one, or many Teams. The relational structure to manage this involves three Tables:

tblTeams

  TeamID (primary key)

  TeamName (what do you call this team)

  <other information about the team as an entity in its own right, nothing about team members)

tblEmployees

  EmployeeID (primary key)

  LastName

  FirstName

  <all the usual information about the person, including seniority)

tblTeamMembership

  TeamMembershipID (autonumber primary key)

  TeamID (which team are you looking at)

  EmployeeID (who is on this team)

  <other info about THIS person's involvement with THIS team, e.g. date started, role in the team, ...)

If Team Blue has eight members, there would be eight records in tblTeamMembership. And if you want to display an alphabetical list of them, you can create a Query joining all three tables, sorted by LastName/FirstName to see them alphabetically, or by HireDate to show them in order of seniority, or in whatever order you choose. This is easy with the relational design; all but impossible with MVF.

john w. vinson say:

You don't!

It would appear that you're trying to store all the information in one table. That's not how databases work! Each Table contains information about one type of "Entity" - real-life person, thing, or event such as an Employee, a Project, a Team, a Company, etc. Your database will have several tables for different kinds of entities.

But the Tables will stay "under the hood". If you're assuming that all of the information must be in one Table, and that you will use that Table to work with it... well, that's not how databases work. Users should never even SEE a table; all interaction would be done using Forms, typically with Subforms, Combo Boxes, listboxes and other tools displaying data from more than one table. You would have a Project form showing data from the Projects table, but it might have a Teams subform showing (multiple records!) of information from the the third table, and employee names from the Employees table. It's neither necessary nor practical to have One Big Master Table containing all the information. 

ken sheridan say:
...............how do I then change my original table that holds most of the data (relating to each project) so that the "Team" field corresponds to the new "Team" table?

Add a column named TeamID to the projects table, of  long integer number data type, and in a form for entering data into the table use a combo box set up as follows:

ControlSource:  TeamID

RowSource:     SELECT TeamID, TeamName FROM tblTeams ORDER BY TeamName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.  In the combo box you can then select a team by name, but the underlying value will be the long integer number.

Once you've done this for all your project's records you can delete the redundant multi-valued field and create a relationship between the projects table and tblTeams on TeamID, being sure to enforce referential integrity.

In a report's RecordSource query join the Projects table to the tblTeams table, this to the tblTeamMembership table, and this to the tblEmployees table.   You can include whatever columns from tblEmployees  you wish to include in the report for the team members, and you can include sort orders on whatever columns you wish to govern the order in which the data is output.  Group the report first by project and include the project data in the group header.  The team membership data can be included in the detail section.  If you wish the first team member to be on the same line as the project data, in the group headers Format event procedure put:

MoveLayout = False

 

In your case a projects data entry form could include a subform, in continuous forms view, based on the tblTeamMembership table and linked to the parent form on TeamID.  This would enable you to enter team members once you've entered a team in the parent form.  If you select an existing team its members will automatically show in the subform.  To create a new team you can type the new team name directly into the combo box on the parent form, and include code in the combo box's NotInList event procedure to insert a new row into tblTeams.  The combo box in the activities subform in my demo illustrates this, allowing a new activity name to be added to the list by typing it into the control, and answering yes when prompted to confirm.


For an example of this sort of many-to-many relationship type see Parentactivities.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file shows how to represent a many-to-many relationship type like this in a form/subform and includes two reports in which each parent's activities are listed alphabetically, or vice versa.
