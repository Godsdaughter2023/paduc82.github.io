The first portion is a generic section displayed in all such messages, that's really just indicating that the value change might be normal, unless you have some reason to suspect that it's not.  No, it's not really a user friendly statement, but as I mentioned in that other thread, it's not typically something the average consumer would be expected to read anyway.

All this message itself means though is that the value itself has changed and thus the information log entry is simply indicting that fact, with no other meaning implied that I'm aware.

Glen knows far more about the operation of these boot time scans themselves, though it became clear to me after watching him troubleshoot a few that these operate within the Windows Recovery Environment (WinRE), that's normally used to troubleshoot and repair common causes of unbootable operating systems.

So from my own observations, it appears that most problems with the operation of the Offline scan are likely due to either a problem passing configuration information into this WinRE environment or simply an issue with the operation of WinRE itself.

That's why as Glen stated a "Repair Upgrade" may be required and is also the reason that little information about the Offline scan results are available within the normal Windows operating system, since these 2 operating systems are completely independent and the normal operating system is in fact not really aware that the WinRE operating environment exists, since it operates outside the limits of it's own environment.

Think of them as 2 worlds where the WinRE environment is only operating at special request and though aware of the normal operating system, typically doesn't write into or change the other's filing system unless something malicious is identified and removed.  However, once booted, the normal operating system isn't aware of what WinRE and the Offline scan has done unless it were to pass something via an additional file, which I'm not certain that it does.

Maybe Glen can shed some light on this from his own observations?  Note though that troubleshooting the Offline scan operation without this knowledge is simply guesswork and experimentation.

Rob

< EDIT > BTW, the boot time scan would obviously be designed to scan the items that couldn't be scanned when the normal Windows operating system is running, such as the Windows kernel and core files, as well as drivers and other files that would get loaded before the built-in Windows Defender or other AV app could launch, including looking for items like rootkits or others that try to hide themselves.

It also likely runs at least a portion of a Quick scan, since this always checks all locations that malware is known to operate.  Full scans are never necessary and only performed in order to insure that "dropper" files like malware installers don't remain on a previously infected system or other leftovers aren't left in places like temporary file folders.  In truth, since these items aren't actually operating, they don't really cause a problem except that they often create confusion when detected and are much more likely to cause a false positive detection as well.

rob koch say:

Though I understand the potential risk to modification that UEFI presents, I think what most forget is that the issues of lack of firmware maintenance to deal with known vulnerabilities had always been a far more real and difficult issue.

A parallel issue had existed with the manual USB based boot process as well, since this was dependent on the availability of boot device software that suited the specific hardware installed on that system.

For most PC users, the above issues were far more prevalent and in most cases only resolved by involving a skilled person, which in many cases was an unavailable resource, so these abilities were never actually nor firmware updates performed.  These were arguably issues of much more concern to the integrity and security of systems than the potential for mostly theoretical and technically difficult attacks.

I don't deny that someone may eventually figure out a way to compromise the systems used to update a popular device, but in reality this had always been possible and in fact much easier when most of the boot and other operating system code was contained on disk, as had always ben true of any general purpose operating system and the PC itself.

It's only with the advent of much more highly secure devices like the Microsoft Surface that contain additional security controls and devices such as TPM, Secure Boot and others that these issues truly are worth considering, since there's finally enough security within the boot chain in order to consider the potential for compromise during the earliest phases of operation.

So not that these issues should be ignored, but I don't see how they're really that important to consider until the devices themselves are already capable of the levels of security just mentioned.  In other words, a Windows PC has always been an insecure device, it's just becoming more evident as the attacks and their mitigations are finally approaching the point were they may actually matter.

I've recently purchased a Surface Go tablet that along with Windows 10 S contains these security features like TPM that used to be limited to systems sold only to businesses, but are now available at reasonable cost to anyone.  The first thing that most consumers might be likely to do wrong however, is to choose to "Upgrade" the device to Windows 10 Pro, which if you understand the technical issues you'd know is reducing the security by returning the very vulnerabilities that this legacy version of Windows contains.

I think that again, the more difficult issues will be the basic education of the public to make the choices that are more appropriate to their security, rather than be concerned about some relatively unlikely scenario affecting the potential compromise of the built-in secure portions of their device.

Rob

Can't be done. You cannot assign the results of a SQL statement to a variable or control. Since a SQL statement can return multiple rows and multiple values, even if its set to return one row and one field, Access will not be able to do it.

That's why you have domain aggregate functions because they return a single value from a query that can be assignable.

diane poremsky mvp (slipstick.com) say:
Do you want to export the recipients from a group email?

There is no option specifically to do that, but you can copy the To or CC field from a message - open the message (expand the header if using Outlook 2013 or 2016) then click in the To row, right-click to Copy then Paste it into excel or notepad to get the recipient names and email addresses. Repeat for the CC row.

You'll also get the names and address if you export a folder to Excel, but you can't easily limit it to one message, other than by moving the message to an empty folder.
