ken sheridan say: The message means what it says, so there must be one or more rows with a Null in one or other, or even both, of the columns in PartSuppliers.  This is invalid in a table which models a many-to-many relationship type.

Check the data in the tables carefully, and when you have corrected the data, set the Required property of each column to True (Yes), and their AllowZeroLength property to False (No).

ken sheridan say: That's not the classical model for a bill of materials, in which assemblies and parts are a single entity type, two instances of which are related by means of an adjacency list table.  The methodologies illustrated in my demo consequently do not apply.  However, if we adopt your model it requires some amendment:

As it stands Company (i.e. the supplier of a part) is functionally determined by the primary key of Parts, PartNumber.  Consequently the Company column in Bom is redundant as it is transitively dependent on the key of Bom via the foreign key Partnumber.  The Company column could be removed from Bom therefore.  It follows from this that the relationship between Suppliers and Bom could also be deleted.  But that's not the solution.

While  the above amendment to the model would normalize the tables it would not be an accurate model of the reality, and would not allow you to return a result table like that you've posted, however, as each part maps to only one supplier.  Consequently, rather than the above, you need to model the many-to-many relationship type between Parts and Suppliers by a further table, PartSuppliers, which resolves the relationship type in two one-to-many relationship types.  This will have columns Company and PartNumber as foreign keys.  The primary key of this table is a composite one of the two foreign keys.

The Bom table can now retain the Company column as, in conjunction with PartNumber, the two columns constitute a composite foreign key referencing the composite primary key of PartSuppliers.  The manufacturer's number is a column in PartSuppliers as it is an attribute of the relationship type between parts and suppliers.  Do not use the # character in the column name, however.  Special characters should be avoided in object names.  Call it MfrNum or similar.  There is no relationship between Bom and Parts directly.

With this amended model it is now a simple task to join the tables in a query to return a result table like that you have described.  The result table of a query will of course repeat the values in the first three columns, once for each supplier of the part, as there will be separate rows for each in Bom where the part is supplied by multiple suppliers for the same assembly.  You can output the data in the format you've used in a report, by grouping the report appropriately.

One further amendment I'd suggest is that you add a SupplierID autonumber column to Suppliers as its primary key and replace the Company columns in PartSuppliers and Bom with long integer number SupplierID columns.

Picture border inset MS Word - Microsoft Community

phillipm2001 say:

Picture border inset MS Word

I would like to inset a picture from a border in Word. I have inserted the picture, but it goes right to the edge of the border. I would like it to be, say, 3pts from the edge of the border. Thanks. 

rich michaels - mvp (greatcirclelearning.com) say:

The Border function won't do that, try using a single table cell and inserting your picture there. You can set the cell padding to the distance you want and color and size the cell border also as you want.

If you are wrapping text around your inserted images, you can do the same with a table cell. If you are planning on captioning your pictures, I recommend adding a row to the table cell with the image and inserting the caption there. The gives you the most flexibility when moving the images around.

Finally, if you are doing a lot of this, take a look at AuthorTec Insert. It's a free add-in to Word. It will make it easier to do all of this at once and not have to click yourself to death.
