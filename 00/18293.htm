scott mcdaniel say:

1. I'm not sure what you mean by "encrypt a file in a manner it can be sold to several companies". You don't "encrypt" an Access program, and in fact you cannot create a true executable file with Access. You can "compile" the file, such that the users could not get to code and design sections, but that is not a true executable. Your users will still require the correct version of Access (the Full or Runtime version).

2. Yes, you can provide filtering capabilities. How you do that depends on how you store the data.

3. A 2gb Access file can contain a LOT of data, but if you're concerned with running out of space you can use SQL Server Express, which has 10gb limit. If that's not enough, you could require your users to have a full version of SQL Server, which is essentially unlimited.

With that said - asking questions such as this brings quite a bit of doubt to mind regarding your readiness to create a commercial Access program. It's hard enough to create a commercial program with a development language designed for such (like .NET, Java, etc). Access is most certainly NOT designed for that, and brings quite a lot of challenges when doing so. I'd encourage you to shelve this idea for a bit, and learn quite a bit more about software development in general, and development with Access in particular (if you stick with Access, of course).

scott mcdaniel say:

You need software licensing/copy protection of some sort, not encryption.  If you search for "software copy protection" you'll find quite a few links.

There are many, many ways to do this, but if you're truly interested in doing this but in my opinion commercial software would provide your best protection. I use CryptoLicensing from http://www.ssware.com/ (they have an ActiveX product, which is what you'd use in Access).

You could roll your own process, of course, but it would not generally be as robust as commercial stuff. For example, you could come up with a scheme where you'd create your own product key, and you could include the Company name in that product key.  When the program is run, your software would "decrypt" the product key and show it on every form, and on every printed report. Most companies would not use software that shows another companies names on their invoices, for example.

There are many, many ways to create software protection, but - again - I'd be much more concerned with learning to properly develop software at this point.

All you need to do for bug reproduction, is install all Additional language support resources onto the phone, i.e.:
  • In Settings - Language add all available* languages to the list, though keep English (or your preferred language) on top of the list, to not cause interface locale change on restart to a language that you may not understand. Tip: instead of constantly selecting "move up" in the context menu, you can remove your language and re-add it - new languages are always added to top of the list (on Windows Phone at least). The language list works much the same way as on Windows 8/8.1 devices, in determining the language order for any setting that uses the language list, such as the order of the keyboard switcher or language preferences for websites in Internet Explorer.
  • Add all keyboards in Settings - Keyboard.
  • Add all speech TTS engines in Settings - Speech. Though it will require you to reboot for each pack installation.

So that's it: all apps that are based on APPX packaging (not XAP packaging) are now crashing on startup.

* - applies mostly to Nokia/Microsoft devices, other OEMs such as HTC often include only several pre-selected languages for a phone that is to be sold in a specific region, and do not have sufficient languages to be able to reproduce the bug.

Now to describe how these bugs behave on a Windows 8/8.1 desktop. This is related, because Windows Phone 8.1 carried the bug over when implementing app universality, APPX packaging and WinRT APIs. The bugs do need to be fixed at the core level of the Windows OS, since Windows Phone 8/8.1 runs on NT kernel and on an underlying system, that is basically the same familiar Windows.

So: if you add many languages (for bug reproduction best is to add all available languages) to the input language list, APPX apps, i.e. all Windows Store apps will start to behave slowly, even to the point of crashing on splash screen if loading for too long (what is the point of loading all language resources, this happens also to apps that don't have the resources for any extra language except default). Even though Windows allows slightly more time tolerance when opening an app, than Windows Phone 8.1 allow, that often does not help. Windows Store itself will still open, but won't be able to connect, so it will show a connection error page instead of Store home page. While logging errors into the verbose winstore.log located in the Store app's sandbox (in the userdata one in the AppData\Local\Packages folder and the Store master package in Program Files\WindowsApps. Not the winstore.log located in %TEMP% folders).

On both systems (phone/desktop) any associated bugs are getting worked around by removing any extra languages from the language list, but that scenario is not a fix, but a simply problem avoidance. For example you may need to visit websites that require many languages on the language list to function properly.

Also related: if you install all Language Packs (MUIs) and all the smaller Language Interface Packs (LIPs) to Windows, up to the point of exceeding the length of the Windows Update connection string (see %systemroot%\windowsupdate.log), you will be unable to use Windows Update, Microsoft Update or any other service, that uses it directly or indirectly. You will not be able to (until you remove some language packages) to check for updates, get definitions for Windows Defender/Security Essentials/System Center Endpoint Protection, install or update any apps from the Windows Store, etc. Windows Update in Control Panel will give connection errors, while logging the following to windowsupdate.log:
