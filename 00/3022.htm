Solution: (for importing from csv)

I was importing individuals from csv files and I kept getting only the email address in the "Display As" field. However, I want the full name in this field. This is professional in the "To:" section (hiding the email) and also in the a "Contact Group" list (the "Name" column actually should show a name).

It's all in the mapping from MS Excel (csv) to Outlook Contacts (vcf)...

MS Excel (csv) File:

Just make sure you have a column that exist that has the Full Name (or what ever you what). 

Outlook 2013:

Step 1:  Go through the wizard up to the window verifying "The following actions will be performed:"...

Go to File > Open & Export > Import/Export > Import from another program of file > Comma Separated Values > (choose file to import and select Options) > (select contact folder) > select Map Custom Fields... 

You should now have the "Map Custom Field..." dialog box up...

Step 2: From the "To:" area at the right under "Microsoft Office Outlook," scroll down to the field "[+] Email" open the tree.

Step 3:  From the "From:" area at the right under "Comma Separated Values," scroll down to the field that has the value you desire, like "Full Name" for example.

Step 4:  Drag that that field onto the "E-Mail Display Name" (it may look like "E-mail Display N...") that is listed under the "[+] E-Mail" tree until it highlights and release. Then you will see your field listed next to it under the "Mapped from" column.

Step 5:   Click OK and Finish.

After completing the steps above, open a contact and you will see the "Display as" field has you desired format/nomenclature.

E71 - Bug in SMTP authentication - Microsoft Community

kreye say:

E71 - Bug in SMTP authentication

If you enter a user name and a password in the settings for the outgoing server, the E71 uses SMTP AUTH (meaning it provides user name and password while sending out mails).

Unfortunately there is a bug in the implementation, at least in firmware version 100-07-76 of the E71.

The phone authenticates itself correctly (AUTH PLAIN), but then it sends a reset command (RSET), which is completely nonsense. With this command the previous authentication gets deleted, and therefore the mail isn't accepted.

This problem already occurred with the E61i, too:
http://skolobov.wordpress.com/2007/08/24/bug-in-nokia-e61i-smtp-client-software/

More than one year ago ....

There is a dirty workaround for people who run their own Linux mail server. The workaround disables the RSET command in the server, so the phone can't delete the authentication any more. It is unknown if the workaround has got any side effects, like producing memory leaks.

If you have Qmail, please see the link above.
If you have Postfix (V2.3.8), please comment out line 2832 in file:
src/smtpd/smtpd.c

So it should look like:
/*
* Restore state to right after HELO/EHLO command.
*/
chat_reset(state, var_smtpd_hist_thrsh);
//mail_reset(state); ****COMMENT OUT THIS LINE****
rcpt_reset(state);
smtpd_chat_reply(state, "250 2.0.0 Ok");
return (0);

Again, this is only a quick and dirty workaround. Though it works for me, the bug should be fixed in the firmware of the phone, so nobody has problems with it any more.

While sending out an Email, the E71 initially authenticates itself correctly to the mail server. But then it resets the authentication, and so the Email isn't accepted.

06 E71 : RSET
07 Server: 250 2.0.0 Ok

kreye say: Well, my workaround deaktivated the execution of the RSET command in the server, and then everything started working fine.

RFC 2821 states clearly:
RSET: "Any stored sender, recipients, and mail data MUST be discarded, and all buffers and state tables cleared."

Please see:
http://www.ietf.org/rfc/rfc2821.txt
(Chapter 4.1.1.5)

Especially the last part of the sentence regarding the buffers and state tables is more than clear, I think. After an RSET any formerly done authentication is deleted.

So an RSET command doesn't make much sense here. This is a bug.

precisium say: "After an RSET any formerly done authentication is deleted.
So an RSET command doesn't make much sense here"

Actually, clearing authentication state with RSET is the action that doesn't make sense.
RFC 4954 states: "After an AUTH command has been successfully completed, no more
AUTH commands may be issued in the same session."

RSET is often issued in between different messages that are submitted in the same session. If RSET cleared the auth state it would stop you sending multiple messages to various recipients in a single authenticated session.

You might argue that an RSET directly after an authentication could reset the auth state - but that would be at odds with the normal behaviour of RSET, and what would it's purpose be anyway?

In short, I think any server resetting auth state on RSET is bogus - and whilst it isn't clearly spelled out in the RFCs - 'all buffers and state tables cleared' should not include the authentication identity.

You're right that an RSET doesn't make much sense there.. but it *should* be harmless.

kreye say: After having read the RFC 4954, I must agree that the RFCs are a bit contradictory in this case.

RFC 2821 states that "all buffers and state tables [are] cleared". RFC 4954 states, that "after an AUTH command [...] no more AUTH commands may be issued [...]".

Taking both statements together, it wouldn't be possible to send two Emails within one authenticated session.

I digged into that a bit deeper, and discovered that some guys already found this contradiction:
http://notabug.com/2002/drums-archive/6812

Unfortunately they simply removed the sentence. Replacing it with a more precise statement would have been much better. Especially the behaviour of the RSET command in authenticated connections should have been descibed.

So the RSET command resets all buffers and state tables of a _transaction_. The buffers and state tables of the _connection_ remain untouched. So the RSET command shouldn't flush an authentication, because it belongs to a _connection_.

Due to this misleading statements, we have some (a lot of) mail servers out there which flush the authentication. So it might be a good idea to take out the RSET command after the AUTH, though Nokia/Symbian implementation is not against the RFCs.

Simply go through the topics, and you will see that a lot of people have problems sending Emails. And they blame their Nokia, because all other mail clients work flawless.

So I agree: An RSET directly after an AUTH shouldn't do any harm. But in reality it does - due to misleading RFCs and therefore incorrect server implementations. And as you say, RSET doesn't make much sense there, so it should be removed.
