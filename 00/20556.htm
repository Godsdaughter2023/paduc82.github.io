ken sheridan say: Tom's DELETE query might not be quite enough.  I assume that you want to specify a date:

PARAMETERS [Enter date:] DATETIME;
DELETE *
FROM TableA
WHERE EXISTS
    (SELECT *
     FROM  TableB
     WHERE TableB.DateField =  TableA.DateField
     AND DateField = [Enter date:]);

This does assume that the tables are related on the date columns, which your post does tend to suggest, but it's a little ambiguous in this regard.  It could also be interpreted as meaning that you want to delete all rows form TableA which have a row in TableB with a match on some other column and the date value in question in another column, i.e. TableA does not contain the date column at all.  In which case:

PARAMETERS [Enter date:] DATETIME;
DELETE *
FROM TableA
WHERE EXISTS
    (SELECT *
     FROM  TableB
     WHERE TableB.SomeField =  TableA.SomeField
     AND DateField = [Enter date:]);

In either case if TableB is the referencing table, i.e. it is on the 'many' side of the relationship, then you'd need to ensure that referential integrity and cascade deletes are enforced in the relationship; otherwise you'd be left with orphaned rows in TableB.  If is the referenced table, i.e. it is on the 'one' side of the relationship, then enforcing cascade deletes is not required.

I'm inclined to think that the latter scenario above is the more likely one as if the relationship was on the date column then you'd simply need:

PARAMETERS [Enter date:] DATETIME;
DELETE *
FROM TableA
WHERE DateField = [Enter date:]);

Enforcing cascade deletes if TableA is the referenced table (on the 'one' side of the relationship).

Whichever is the case, it would be wise to back up the tables first.

john w. vinson say:

Well... I'd recommend "unasking" the question. This kind of calculation should generally NOT be stored in any table, anywhere; just store the underlying data and calculate it (by putting the expression in a Field cell in a query, or preceded by an equals sign in the Control Source of a textbox on a Form or Report) whenever you need it.

Storing calculated values risks update anomalies: if one of the underlying fields is edited, your Gross value will now be Grossly wrong (sorry!) with no easy way to detect the error. Computers love to do calculations; that's what they're built for! 

If you have some GOOD reason (such as wanting to be able to override the calculation for some records) then Hans' suggestion is of course correct; but I don't think your accountants or the Labor Relations Board would approve.

scottgem say:

Correct your flawed table design.

Don't store Gross. Its a calculated value so shouldn't be stored. Also you should not have the name fields in this table. You should have an employees field with an employeeID. Only the EmployeeID should be in this table. And you refer to a Years field, but you appear to be calculating on a weekly basis.

So you table should look like this:

tblWeeklyPay

WeeklyPayID (PK Autonumber)

EmployeeID (FK)

WeekEnding  (or weekBeginning)

Hours

Rate

The create 2 queries:

qryRegOTHours

SELECT EmployeeID, WeekEnding Rate, IIf(Hours>40,40,Hours) As RegHrs, IIF(Hours<=40,0,Hours-40) AS OTHrs

FROM tblWeeklyPay;

qryRegOTPay

SELECT EmployeeID, WeekEnding, RegHrs*Rate as RegPay, OTHrs*Rate AS OTPay

FROM qryRegOTHrs;

You can do it in one query, but this is easier to follow the logic and debug.

ken sheridan say: John and Scott would be right if it can be guaranteed that the standard working week will be 40 hours, and the overtime rate will be 1.5 until the end of time.  I doubt that's a valid assumption, but even if we assume that it is, then encoding the length of the standard working week and the overtime rate in an expression is not good practice.  It is data, and a fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

The length of the standard working week should be a value at a column position in a row in a table.  If the above assumption can be made then the table would have a single row and a single column.  The same is true of the overtime rate of 1.5, which also, as values of 40 and 1.5

On the other hand, if the assumption is not valid (more likely in my experience), then the working week hours table, and the overtime rates tables would each also have DateFrom and DateTo columns.  The two tables would then be joined to the timesheet table on its relevant date column being between the start and end dates in the other two tables respectively.

The fundamental point which John and Scott are making is correct in both scenarios, i.e. that the gross weekly pay should not be stored at a column position in a row in a base table, but computed from the stored data on which it is based.
