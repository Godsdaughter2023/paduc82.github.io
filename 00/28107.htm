Range("D2:D" & Range("B" & Rows.Count).End(xlUp).Row).Formula = "=D2*0.5"

You should not put a formula in the cells of column D that refer to the same cells. That will be a circular reference with unexpected results.

Try this macro instead:

Sub leo()
   first_row = 2
   last_row = Range("B" & Rows.Count).End(xlUp).Row
   For i = first_row To last_row
     If Cells(i, "N") = "X" Then
       Cells(i, "D") = Cells(i, "D") * 0.75
     Else
       Cells(i, "D") = Cells(i, "D") * 0.5
     End If
   Next i
End Sub

Hope this helps / Lars-Åke

myall_blues say:

You could also create a custom function in VBA using the following code.

Function SpaceCap(InputLine As String) As String
     Dim i As Long
     SpaceCap = Mid(InputLine, 1, 1)
     For i = 2 To Len(InputLine)
         If Mid(InputLine, i, 1) = UCase(Mid(InputLine, i, 1)) Then
             SpaceCap = SpaceCap & " " & Mid(InputLine, i, 1)
         Else
             SpaceCap = SpaceCap & Mid(InputLine, i, 1)
         End If
     Next i
 End Function

Then use it like a normal formula in your worksheet like so:

=SpaceCap(A1)   or

=SpaceCap("IAmGoingToSchool")

If I'm interpreting things correctly, my method of accomplishing this is to bookmark the text following the number sequence. Just the text, not the auto-number or the paragraph mark (¶) or you will carry the original formatting with it. Let's call that bookmark Num511.

Now go to where you want the 5.1.1 Heading2 (continued) and tap Ctrl+F9 to insert a set of field braces then type REF Num511 \n and move outside the field braces to the right. Put in a <space> or two and tap Ctrl+F9 again. Type REF Nun511 then move outside the field braces to the right and type (continued). The result should look like this (with formatting marks showing,

{·REF·Num511·\n·}··{·REF·Num511·}··(continued)¶

A quick Ctrl+A (i.e. Select All) and then F9 (i.e. Update Fields) and you should have,
5.1.1 Heading2 (continued)
... in the format of your choice. The bonus is that this will update numbering if 5.1.1 changes for some unforeseen reason.

macropod say:

The underlying problem here is that your data source has what are interpreted as mixed data types in the zip-code field.

By default, Word 2002 & later use the OLE DB provider to get merge data from an Excel workbook. Because the OLE DB provider is designed to return data in a way that is compatible with Access and other relational database packages, it requires a specific the data type for each column. The first 8 records are used for that (the 8 can be changed in the Windows Registry, but it’s not advisable to do so). Records that don’t conform to the determined data type are liable to not be handled correctly. To complicate matters, there are two text data types: "text" (up to 255 characters) and "memo" (can be longer than that).

Some common mailmerge issues arising out of this include:
• Numbers but not text or dates being output for some records; and
• Text data being truncated at 255 characters.

Ideally, one would ensure each field has only one data type. Workarounds include:
• Inserting a dummy first record containing data in the format that is not being output correctly; or
• Reordering the data so the first record has content in the format that is not otherwise being output correctly.
In your case, the solution would be to ensure the first record's zip/postal code is non-numeric.

That it is old does not mean it isn't good. Things that worked in earlier versions still work in Office 2019. Things that were problems probably still are; the workarounds still work.

1. If fields are case-sensitive. Testing for "" does not involve that, but other tests can.
2. The { } braces cannot be typed, they must be inserted by Word, usually using Ctrl+F9.

This is going to involve some massaging of data. I think you would want to have two columns, one for a code to tell which type of postal code you are dealing with and the second to hold the code itself. The alternative would be to have one column for each type of postal code.

Do any of the codes other than US have leading zeros that cause problems?

Are all of your US codes 5-digit?

Suggestion for the code column (pCodeType):

Most frequent type = 0

Next most frequent = 1

Next most frequent = 2

You could (1) use letters instead of numbers but then you run into the case-sensitive problem, (2) have your most frequent type simply be blank and test for "" instead of a number.

You will likely have to hand code that. Still easier than more columns, though, I think.

Assume that 0 = US.

{ IF "{ Mergefield pCodeType }" = 0 "{ Mergefield PostalCode \# "00000"}" "{ Mergefield PostalCode }" }

Pay attention to Paul's warning about mixed data types.

zorvek say:

Use this function to determine if a specific worksheet or tab name already exists:

Public Function ExistingSheet( _
      ByVal SheetName As String, _
      Optional ByVal TargetWorkbook As Workbook _
   ) As Boolean

' Return True if the named sheet exists, False otherwise.
'
' SheetName - The name of the sheet.
'
' TargetWorkbook - The workbook to check. Optional. Of omitted then ThisWorkbook is assumed.

    Dim TargetSheet As Object

    If TargetWorkbook Is Nothing Then
        Set TargetWorkbook = ThisWorkbook
    End If

    For Each TargetSheet In TargetWorkbook.Sheets
        If UCase(TargetSheet.Name) = UCase(SheetName) Then
            ExistingSheet = True
            Exit For
        End If
    Next TargetSheet

End Function
