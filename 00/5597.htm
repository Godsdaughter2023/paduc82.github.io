You need to see this as a string with three elements, separated by Periods. First pick it apart using Split, then convert the element(s) to numeric (using CInt) and increment, then format using Format and put back together using Join.

marshall barton say:

That's because you are packing multiple pieces of information into a single field.  What you have is not a number, so you can not use arithmetic on it.

The answer to this kind of problem is to use a separate field for each value.  In this case, it appears that you have 4 pieces of information, a letter (C), a number (1234), another number (01) and the number you want to increment (01).  Put those into separate fields and you can then use the standard technique to increment it.  Whenever you need to display that stiff to users in a form or report text box, you can put it together using an a query or text box expression like:

[letter field] & [first number field] & "." & Format([second number field], "00") & Format([third number field], "00")

If it does, parse it off to a new CHAR(1) field and use formatting to display the result of it concatenated to an autonum or programmed increment field.

ken sheridan say: You can parse it like so:

CWIP_POS_CTRL_NUM = "C1234.01.02"
? Left(CWIP_POS_CTRL_NUM,Instr(CWIP_POS_CTRL_NUM,".")-1)
C1234
? Mid(CWIP_POS_CTRL_NUM,Instr(CWIP_POS_CTRL_NUM,".")+1,InstrRev(CWIP_POS_CTRL_NUM,".")-Instr(CWIP_POS_CTRL_NUM,".")-1)
01
? Mid(CWIP_POS_CTRL_NUM,InstrRev(CWIP_POS_CTRL_NUM,".")+1)
02

This appears to be an encoding system.  Encoding systems are of course very common; US Zip codes, UK post codes, ISBN numbers, Credit Card Numbers are all examples of encoding systems in which individual elements of the full value represent different data.  However, while its fine for the full value to be used in most day to day situations, e.g. entering a customer's Zip code or post code in a single column in a table, in situations where the individual components need to be processed, as in your case, it makes far more sense to store the individual elements in separate columns.  You can of course create a unique index on all of the constituent columns.

This is a known issue and as a work around, you must create an alias and use that for the mean time.

-Log in to http://www.account.live.com using your account, go to "Your info" tab and click on "Manage how you sign in to Microsoft".

-Enter your password and last 4 digit number of your phone or alternate mail, enter the code once received.

-After that, you should be able to see the page where you can add an alias, either create new or use a 3rd party email. Set the new one as primary.

-Log out of the page, restart your browser and go to http://www.outlook.com and log in using the newly created alias. Use the same password.

*Note: This is just a workaround for this type of issue. Once the issue is fixed, you can set your old account as primary again.
