ken sheridan say: When calling a procedure in a standard module you do not need to qualify the procedure with the modules name.  Nor do you need to enclose the argument in parentheses; that is only required when calling a function which returns a value.  So, you'd call it like this:

    FErrorHanlder DataErr

More to the point,  a one-size-fits-all solution like this is really of very little advantage to the user.  Error handling needs to take account of the specific context in which the error occurs, and either inform the user why, in that context, the error has occurred and advise them accordingly, or take action to rectify the error.  The following is an example in a form's module in one of my own demo files for generating sequential numbers per group (gender in this case).  It uses an extension of the method originally published by Roger Carlson, and the function is called from the form's Error event procedure if an error occurs when an attempt is made to save a record, passing the DataErr value as its argument:

Private Function IncrementField(DataErr)

    Const KEYVIOLATION = 3022
    
    If DataErr = KEYVIOLATION Then
        Me!NameID = DMax("NameID", "NamesByGender", strCriteria) + 1
        IncrementField = acDataErrContinue
    Else
        IncrementField = acDataErrDisplay
    End If

End Function

In this strCriteria is a module level variable to which a value is assigned as follows in another function in the form's module into which the gender entered in the current row is passed as the strGender argument:

    strCriteria = "Gender = """ & strGender & """"

Rather than simply informing the user that a primary key value has been duplicated, in this case as a result of one or more other users having saved a new record prior to the current user while simultaneously inserting a row into the table, the code handles the error and increments the number until no error occurs.  This does require more work on the part of the developer, but it's the developer's role to ensure the smooth running of the database, not to make life easier for him or herself at the expense of better functionality.

helphenry say: I fixed the problem.

Another thread mentioned a conflict with poorly written Add-Ins. So I had a look at active Add-Ins and sure enough there was a bluetooth Add-In there. I removed it and now Word opens and works as it should. I'm happy again. 

The steps
1. Open Word. In Safe mode in this case. 
2. In File menu go to Options
3. Choose Add-Ins to view and manage Microsoft Add-Ins
4. At the bottom click Manage Add-Ins, Choose which group of Add-Ins type might be affecting you then click Go. (I chose Coms Add-Ins as that's where the Bluetooth Add-In was listed.)
5. Remove the Add-In. (you could try deselecting the Add-In if you didn't want to remove it and see if that solves your problem.) 

Easy fix really. Wasted a few hours finding it though.

joeu2004 say:

DanBrown wrote;

I know the Compound Average Growth Rate (CAGR) for a 10 year period.

Which Excel formula can I use to calculate the Average Annual Growth Rate (AAGR)[.] It isn't CAGR/10 because that does not allow for compounding.

The CAGR is the compounded average rate.  The AAGR is not a compounded rate.

The AAGR is simply the arithmetic mean of the series of annual growth rates.

You cannot calculate the AAGR directly from the CAGR.  There is insufficient information.

But if you have the 9 annual growth rates for a 10-year period in B2:B10, the AAGR is simply =AVERAGE(B2:B10).
