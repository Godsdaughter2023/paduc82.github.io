bhringer say:

Antimalware engine update notifications used to be available through a TechNet blog.

https://answers.microsoft.com/en-us/search/search?SearchTerm=antimalware+engine+notifications&IsSuggestedTerm=false&tab=&isFilterExpanded=false&searchFormBtn=&CurrentScope.ForumName=protect&CurrentScope.Filter=&ContentTypeScope=#/protect///1

An announcement of upcoming release would be followed by actual release notice sometimes including a change log but for the most part it would state "bug fixes and improvements" with few other specific details.

In the early stages of MSE development a new antimalware engine would be released to resolve vulnerabilities/bugs within the software. At present such issues are usually resolved with new definition updates. An example might be a false positive detection for a popular third-party software such as CCleaner.

AFAIK the only way to determine when a new antimalware enguine has been released is to check change log for definition updates.

https://www.microsoft.com/en-us/wdsi/definitions/antimalware-definition-release-notes

The change will be indicated by definition version numbering. In the screenshot below you will note the definition version changes from  1.287.803.0 to 1.289.4.0. If you select 1.289.4.0 from drop down menu and click Go you will see that that was released Feb 26, 2019 10:22 PM UTC which would be close to the actual release time for the new antimalware engine.

There was speculation when notifications ceased that it was meant to prevent a heads up for malware creators and other miscreants though more likely it had to do with downsizing staff at MMPC.

Hope this has been informative.

Cheers.

john - project say:
prasadnk,
Let me add a couple things to Jan's response. First, the columns for Start and Finish fields most likely disappeared because you inserted the new cost field in place of the Start and Finish fields.

In Project a view is made up of a screen, a view table, and may have a filter, grouping or sorting applied. The view table determines which Project fields are displayed as columns and in what order. You can see the view table by going to the View tab on the main ribbon. From there click Tables in the Data section. Then click More Tables. In the More Tables window that opens, select one of the tables (e.g. Entry) and click the Edit button. You will now see all the fields that make up the Entry table. You can add and delete fields as desired and change how they are displayed (i.e. column width, alignment, etc.).

One other thing for your reference. In Project a user cannot permanently delete a field. Selecting a column heading as Jan mentioned and hitting delete only removes that field from the view table. The field and all its data still exists and can be added back to the view table at any time by adding the field as a column in the view either through the view table as described above or using the quick method as Jan outlined. However, if you select an entire task or resource row (depending on the view type), and hit Delete, that row is in fact deleted and can only be recovered by using the Undo command.

Hope this helps.

The activation system is limited and sophisticated at the same time. 

The limited part is that for 2010 and earlier, the activation system does not track uninstalls/"deactivations".  Starting with 365/2013 (but not for local installs of 2013, AFIK) USERS do have an online function to "deactivate" installations from specific machines to allow activation on other machines making it easy for users to move their 5 allowed 365 installations around to various machines.

The sophisticated part is that the activation process takes a "snapshot" of specific components of your computer. When you try to activate again, or it "verifies" your activation, it checks the components to confirm the activation is on the same computer previously activated, or not. An added layer of sophistication is that the activation system allows for "limited" (a vague, undefined by MS limit) upgrades to your system before it considers it a "new computer".

So you’re not personally identifying me, but you are still requiring information about the make and model of my PC.

To ensure the end user's privacy, Microsoft uses a one-way mathematical algorithm to create the hardware hash used by Product Activation to create the Installation ID. Once created, the hash information cannot be calculated back to its original values. Hardware information is sent through the algorithm in the software on the PC-not at Microsoft-to create the hash. The raw hardware information is not known or sent to Microsoft. Ensuring end user privacy is a No. 1 design goal for Microsoft with Product Activation.

 

Product Activation keeps users from changing or upgrading their hardware.

Not true at all. Users can change or upgrade their hardware. One of the forms of piracy that Product Activation guards against is hard disk imaging. Not all forms of hard disk imaging are illegal. In the case where a pirate copies data from one PC hard drive to another to illegally run the software on two PCs, Product Activation stops that by forcing the copied software to be reactivated. It does so by comparing the hardware on which it was activated to the hardware on which it is now being booted. If the hardware is substantially different, then reactivation is required. If it is the same or similar, then the software will continue to work. Those who upgrade their PC's hardware substantially may be asked to reactivate. Reactivation for this reason is easy and can be completed by contacting Microsoft to obtain another confirmation ID.
