  1. Create two queries, each with about half the fields you need. Include the field that identifies the student (let's say StudentID) in both queries.
  2. Create a report based on each of these queries. Do not include a page header and footer, nor a report header and footer in these reports, only a Detail section. Make each of the reports fit in half a page.
  3. Create a report based on a table or query that lists the unique students. You can add a page header and footer and/or a report header and footer to this report.
  4. john w. vinson say:

Don't confuse data PRESENTATION with data STORAGE. They are different tasks with different requirements!

It appears that you are assuming that all of the data for a report must be as individual fields in a single query. That's absolutely NOT the case. You will instead want to have a properly normalized set of tables - a table for Courses, a table for Students, multiple other tables; your Report may well have multiple ROWS (not fields!) from some of these tables. But trying to shoehorn all 300 pieces of information into one "spreadsheet" style grid is not the way to go. Trying to design your tables based on an existing paper report is dangerous; relational databases and paper forms are both good, standard ways of managing data, but they are profoundly and fundamentally different!

If you could post some description of your tables and of the kinds of information you want to display we may be able to help come up with a table structure that will help.
