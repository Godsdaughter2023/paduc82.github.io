joeu2004 say:

Apparently, YEARFRAC("1/31/2017","2/28/2017") is correct, according to the https://en.wikipedia.org/wiki/360-day_calendar.

(On the other hand,  wikipages are not authoritative.  They might be wrong.)

But YEARFRAC("2/28/2017","3/31/2017") does not agree with the wikipage.

(I write dates in the form m/d/yyyy.)

In any case, like you, I am surprised that the result is not always 30/360 (about 0.083...3).

-----

According to the wikipage:

duration(A,B) = (YEAR(B)-YEAR(A))*360 + (MONTH(B)-MONTH(A))*30 + (DAY(B)-DAY(A))

assuming A <= B with the following modifications:

1. If both date A and B fall on the last day of February, then date B will be changed to the 30th.
2. If date A falls on the 31st of a month or last day of February, then date A will be changed to the 30th.
3. If date A falls on the 30th of a month after applying #2 above and date B falls on the 31st of a month,
    then date B will be changed to the 30th.
 

(I would expect #3 to also apply if B falls on the last day of Feb.)

So YEARFRAC("1/31/2017","2/28/2017") should be and is:

( (2017-2017)*360 + (2-1)*30 + (28-30) ) / 360 = (30-2) / 360 = 28 / 360

because 1/31/2017 (A) is changed to 1/30/2017.  Rule #3 does not apply(!) because 2/28/2017 (B) does not fall on the 31st.

YEARFRAC("2/28/2017","3/31/2017") should be:

( (2017-2017)*360 + (3-2)*30 + (30-30) ) / 360 = 30 / 360

because 2/28/2017 (A) should be treated as day 30 (rule #2), and 3/31/2017 (B) should be treated as 3/30/2017 (rule #3).

Instead, YEARFRAC returns 31/360, ignoring rule #3, thus:

( (2017-2017)*360 + (3-2)*30 + (31-30) ) / 360 = 31/ 360 (about 0.0861...1)

Note that 2/28/2017 should not be treated as literally DATE(2017,2,30). That would be equivalent to 3/2/2017, and YEARFRAC would return 29/360 (about 0.0805...56).

May I assume the column you call "team" is a renamed text field (e.g. Text1)? What is a "performance task"? A performance task, (often called a subtask), is simply an activity expressed in action verb form that has resources assigned to it. Resources should only be assigned at task level, not at summary level.

Resources can indeed be defined as group resources (or disciplines in your case). Just set the Max Units for group resources at the appropriate level. willy denoyette say:

Long reply ahead.... sorry.

If you say backups, what backup system do you mean? Is it a System Image backup or File History backup or something else?

Image backup is the fastest because it uses the Systems Shadow copy  feature (IO data is not moved to user space, but moves back and forth from the device driver cache ).

History File backup is also using Volume Shadow copy, but is somewhat slower, this is by design as to not interfere with normal user activity. If none of these are used you should check how the source data is read and written.

If the kind of backup program used is based on a simple file read/write mechanism, the throughput will highly depend on the number and sizes of the files. Many small files (< 4 Kbytes) will have a very low throughput while large files (>1 Mb) will have a better throughput.

To test this you should copy a couple of large files (say the install.esd from the hidden $BT folder if you still have it)  from the source to destination (your USB3 attached drive). You must watch the throughput using Task Manager's Resource Monitor. Look at the disk counters Read bytes/sec of the source drive (probably the slowest) and Write Bytes of the destination drive (your USB3 drive).

If the throughput (determined by the slowest drive) is lower than say 60-70 MBytes/sec then you have a problem with this source or destination drive or their driver configurations. Take care the throughput is determined by the slowest (or faulty) drive! In that case you should test each drive individually.

joeu2004 say: JSBdtx wrote:
The principle part was easy enough but this is providing to be harder. The payments are to be paid on the first of each month with a 30 day grace period for interest calculation.  There is no credit for over payments.

Of course, the challenge is not with the computation of simple interest, but with the timing of when interest is charged and paid.

Suppose the rent is $1000 per month, and the daily interest rate is 0.05% [1].  Suppose rent is due on the 1st of each month, and the grace period is 15 days [2].  And suppose the current balance is zero (all caught up).

Suppose rent is due on Jan 1.  If it is paid before Jan 17, no interest is charged.  Right?

But if it is paid on Jan 17, we would charge 16 days interest, which is $8.  Right?

In that case, the $1000 payment would be applied as follows:  $8 would go toward paying off interest, the remaining $992 would go toward rent, leaving an outstanding balance of $8 rent.  Right?

The next rent is due on Feb 1.  Suppose no payment is received in Feb.  The unpaid rent balance would become $1008; and the unpaid interest balance would become  $14.17:  15 days interest on $8 ($0.06 = 15*8*0.05%) plus 28 days interest on $1008 ($14.11 = 28*1008*0.05% rounded).  Right?

The next rent is due Mar 1.  Suppose $2022.17 is received on Mar 16, within the 15-day grace period.  Would you consider the account paid in full?  Or would you charge an additional 15 days interest on the unpaid balance of $1008:  $7.56 = 15*1008*0.05%?

In other words, does the grace period apply only to the current month's rent; or does it also apply to the outstanding balance?

Finally, suppose instead that $2022.17 is received on Feb 20, before the next due date.  Would you refund the undeserved 8 days interest:  $4.03 = 8*1008*0.05%?  Or would you hold the remainder on account to be applied to the next month's rent?

These questions are about the mechanics.  Never mind that the amounts might seem too small to worry about.  If you cannot let go of that thought, multiply all the amounts by 100 -- a very expensive celebrity rental in Manhattan ;-) [3].


------
[1] If the annual rate is 18.25%, the daily rate is 0.05%:  18.25% / 365.

[2] I am choosing a grace period of 15 days instead of 30 days to illustrate some points.

[3] The most expensive Manhattan rental was $165,000 per month in 2011.

jsbdtx say: This problem is more about calculation routines than the process itself. Interest is charged for each day past the 30 day grace using simple interest. If the payment is 500 days past due you have to calculate the number of days from the grade due date to the end of the year, say 150, and then use the rate for that year for those days. I’ve never been very pleased with Outlook’s HTML engine. When Microsoft dumped Internet Explorer for Mac they lost good Mac HTML personnel.  I don’t see that changing any time soon.

Within Office there are some decent work-arounds.

You can use Microsoft Word to compose your email. In Word, go to the View menu and choose Web Layout. Then when you are ready to send, use File > Share Email as HTML. The recipient’s email will usually look like what you sent regardless of their email program.

If you are looking for even more precise rendering, use Microsoft Word’s Print Layout view and then when ready use File > Share > As Attachment. If you stick with common fonts the recipient will receive the message as an attachment that opens in Word and will render correctly.

Even more universal, use Microsoft Word in Print Layout view. When ready, use File > Share > Save to OneDrive. Then switch to OneDrive on the web and then send the link to the file to the recipient. The document will render the same on all computers whether or not they have Word. Even in LINUX .
