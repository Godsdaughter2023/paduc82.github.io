scottgem say:

The technique is the same. The RowSource of the second listbox needs to use as its criteria, the selections in the first listbox. However, with a multi-select listbox, you need get the selections first. 

So you need to do something like this:

Dim frm As Form, ctl As Control
Dim varItem As Variant
Dim strSQL As String, strSelect As String
    Set frm = Form!frmMyForm
    Set ctl = frm!lbMultiSelectListbox
    strSQL = "SELECT fieldlist FROM table WHERE [field] IN("

    For Each varItem In ctl.ItemsSelected
        strSelect = strSelect & ctl.ItemData(varItem) & ", "
    Next varItem
    strSelect=Left(strSelect,len(strSelect)-2))
    strSQL = strSQL & strSELECT & ")"
    Me.lstbox2.RowSource = strSQL

This code would go in the After Update event of the first listbox and will construct a SQL statement that will filter for the selections in the first. Then assign that SQL statement as the RowSource of the second listbox

This little demo file uses data from Northwind and has a form with two list boxes, one of customers, the other of products.  Selecting one or more customers in the first lists the products ordered by those customers in the second.

The RowSource property of the first list box is:

SELECT ID, [First Name] & " " & [Last Name]
FROM Customers
ORDER BY [Last Name], [First Name];

The code in its AfterUpdate event procedure assigns builds a an SQL statement as a string expression and assigns it to the RowSource property of the second list box as follows:

Private Sub lstCustomers_AfterUpdate()

    Dim varItem As Variant
    Dim strCustomerList As String
    Dim strSQL As String
    
    With Me.lstCustomers
        If .ItemsSelected.Count <> 0 Then
            For Each varItem In .ItemsSelected
                strCustomerList = strCustomerList & "," & .ItemData(varItem)
            Next varItem
            ' remove leading comma
            strCustomerList = Mid(strCustomerList, 2)
            strSQL = "SELECT DISTINCT Products.ID,[Product Name] " & _
                "FROM Products INNER JOIN (Orders INNER JOIN [Order Details] " & _
                "ON Orders.[Order ID] = [Order Details].[Order ID]) " & _
                "ON Products.ID = [Order Details].[Product ID] " & _
                "WHERE [Customer ID] IN(" & strCustomerList & ") " & _
                "ORDER BY Products.[Product Name]"
        End If
        Me.lstProducts.RowSource = strSQL
    End With
          
End Sub

Note that the SQL statement draws upon three tables, Products, Orders, and Order Details.  This is because to map a customer to a product it is necessary to join the tables in a query to reflect the underlying model of the database:



BTW I'd strongly recommend that you do not follow the bad example of Northwind of including spaces in table and column names as in the above.  Either use CamelCase or represent a space by an underscore character Like_This.

A short history: What you're using is a legacy convenience which always has been limited in terms of how it can be modified. It is an element available from Format> Borders & Shading - Horizontal Line, which provides selection from several pre-designed .gif graphics "lines". That one, specifically, is called Default Line.

Even that dialog did not enable changing the Color of that particular element, though. Also, changing the height actually revealed the gradient grayscale graphic on which the line is based... IOW, not much good at all :-) Why Microsoft chose to include it on the palette - especially given its crippled state - I have no idea.

Frankly, I'd avoid it unless you want to accept the defaults. Even then, I don't like it because, among other reasons, each one adds an extraneous paragraph to the document. In my experience that adds unnecessary complexity to many tasks.

Instead, my recommendation is to use Bottom (or any of the other Border choices) from that same palette. Paragraph Borders can be formatted using the Borders & Shading dialog mentioned above, they don't add paragraphs, they're far more versatile, and they make for a more stable document as well.

ronrosenfeld say:

I don't think the WEBSERVICE function, as described in your link using the Yahoo API, will be the best way of doing this.  The Yahoo API seems to return a csv output, with all the column headers and relevant data in two lines.  You would then have to parse out the closing price.

This can clearly be done, as the format is delimited and known (e.g. the close will always be the 5th item in the second line).  With multiple stocks, you may have a speed problem.

So if you have multiple stocks, and multiple dates, you might be better off with a Web Query, or, possibly, using YQL (Yahoo Query Language) to return an XML string, and FILTERXML to return the closing price.

I would suggest reading about the Yahoo Finance API's and deciding how you want to approach this.  Personally, I would use YQL and a VBA routine to parse things out for this kind of problem, so as to be able to run this with a single call.

In point of fact, I use Fidelity, but I am not looking for historical pricing.
