Acces 2003, how to calculate age from DOB and record it on a form and - Microsoft Community

nelsrac say:

Acces 2003, how to calculate age from DOB and record it on a form and in a table.

I have used =DateDiff("yyyy",[Birthdate],Now())+Int(Format(Now(),"mmdd")<Format([Birthdate],"mmdd")) in the age box control source but unable to store the age in a table, if I just point to the  table age column in the control source it does store in the table but it is ant an automatic calculation on the form.
ken sheridan say: It is important that you understand why we are labouring this point about not storing the age value unless it is fixed in relation to some event which is not itself dated.

One of the fundamental mechanisms of the database relational model is a process known as wherever the ContactID value appears we know that I am referred to.  The FirstName and LastName columns are functionally determined by the ContactID column, so in a table of Appointments only a ContactID column is need to know that the appointment is with me.

If the Appointments table also included FirstName and LastName columns then it would be perfectly possible for these to be David and Cameron in a row in which the ContactID is 42, i.e. the redundancy has allowed an update anomaly.

The Contacts table might also include a DateOfBirth column, which is fine because it is also determined by the key, ContactID.  If we want to know my current age in real time we can get it from the DateOfBirth value at any time, and be guaranteed that it's always correct and up to date.  If we store it in the table, however, by assigning the return value of the GetAge function to a column in the row for me, then come November it will be wrong because I'll be a year older.  In this case age is not functionally determined solely by the key ContactID, but also by the current date, so its redundant and should not be stored.  This is not a question of preference, it's simply wrong to do so.

Let's say for some reason we need to know the age of each appointee at the time of an appointment, which would be perfectly reasonable in a medical database say.  In this case we would again compute the age from the DateOfBirth value, but this time also from an AppointmentDate column, which we could do in a query like this for instance:

SELECT Contacts.ContactID, FirstName, LastName,
AppointmentDate, GetAge(DateOfBirth, AppointmentDate) AS AgeAtAppointment
FROM Contacts INNER JOIN Appointments
ON Contacts.ContactID = Appointments.ContactID;

Once again to store the age as a separate column in Appointments would be wrong.  In this case it's wrong not because it will go out of date, but because it could be changed to a value which is not the age of the contact at the appointment date, or conversely the Appointment date value could be changed so that the age in the column is incorrect, i.e. there are two possible update anomalies.

So, while you might think it's convenient to store someone's age in a column I find it very difficult to envisage a situation where, if the person's date of birth is stored, it would be right to do so.  It not only offers very little, if any, additional convenience it undermines the integrity of the database by allowing update anomalies to occur.

Scott mentioned Ted Codd who first proposed the database relational model in a paper in 1970 in the journal 'Communications of the Association for Computing Machinery', and the rest, as they say, is history.  One way to remember the basic principles of normalization is:


What this means is that any non-key column in a table should be determined by the whole of the primary key and only the whole of the primary key.  As well as being a single column a table's primary key can be a composite one of two or more columns, which is where the 'whole' comes in.
