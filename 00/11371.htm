This simple little demo file is for room reservations.  In your case, villas would be analogous to rooms.

I imagine you would also wish to include financial data in your application, which the demo does not.  Rental Periods would be stored in a referenced table.  Assuming rental rates differ both between villas and at different periods throughout the year these would be stored in a table which references  RentalPeriods and Villas.  PaymentsDue would be stored in a table which models a ternary many-to-many relationship type between Customers, Villas and RentalRates by resolving the relationship type into three one-to-many relationship types.  An important point to note is that this table should include a RentalRate column to which the current rate is assigned when a row is inserted, rather than simply referencing the RentalRates table.  This is because, while rates will change over time, the values stored in relation to each rental transaction should be those current at the time of the transaction.  In database terms, the RentalRate column in each table is independently determined by the key of the table.  Payments by customers would be in a further table which references this table.
