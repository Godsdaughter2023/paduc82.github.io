thedbguy say:

Hi. I'm sure there is a way, but it probably won't be a good design. To properly design the tables for this, you'll need a record per temp per item, which means you'll actually have 20 records times 24 hours, which is a total of 480 records for one day of temp readings across 20 items.

Keep in mind it is more important to create a properly designed table structure than it is to worry about how the records should be displayed. Once you have the table structure designed properly, you can display the records in many ways based on your needs.

    Multiple projects one resource view - Microsoft Community

    john - project say:

    bevbanjos01,

    I assume you are not using a resource pool and that the same resources are shown on the individual Resource Sheets of each subproject. That effectively means that even though Joe is shown as a resource on more than one Project, each "Joe" is independent and not related.

    There are various ways to get the data you want. One is to use a Resource Pool file which is a separate independent file that only has resources and no tasks. Each individual project then shares resources from that pool using the Resource > Assignments group > Share Resources. With that approach you don't need a master file because the assignments of all resources from all projects are readily available via the Resource Usage or Task Usage view in the pool file.

    Another approach is to go ahead and leave each subproject file as is (i.e. replicate resource names in each file as necessary). Then when creating the master, un-check the "link to project" option in the lower right corner of the Insert Project window. That will create a independent snapshot in time of all the individual project files AND combine all resources into single individuals. This only works however if each resource is identically identified on each Project's Resource Sheet. For example, Joe must be listed as Joe on each project's Resource Sheet, Joe, joe and Joe J will not be the same resource.

    A word about linked structures in Project. If you have created a dynamic master (which is the default) or created a resource pool, you have a linked structure and unfortunately such structures in Project are prone to corruption. The chance of corruption increases if you have both a dynamic master and a resource pool. The second approach above does not create a linked structure and is therefore a safer approach. However, since a static master is a snapshot in time, it must be recreated each time you need to see all files together.

    Hope this helps.

    John

    john - project say:

    Bev,

    You're welcome and thanks for the feedback.

    First of all, is there a particular reason why the "business" wants a linked structure? As I mentioned in my first response, linked structures in Project are prone to corruption. None of the files in a linked structure should ever be renamed, moved, overwritten or "saved off" to another location. All files should reside in a single directory on a local drive and not be operated over a network. Failure to follow these rules sows the seed for corruption, which will occur at some point.

    Now to answer your questions. You still see multiple resource names when you look at a resource view (e.g. Resource Sheet or one of the Usage views) because you are seeing the individual subproject files which are linked but not actually part of the master. However, if you open the resource pool file and select the third option (see screenshot) then the pool file will be part of the master and all resources will be combined because the Resource Sheet view is now looking at the pool file, (remember it's not actually part of the master), and all resources are combined in the pool file.

    On the other hand if you select the first or second option and open your master separately, you will not be able to assign resources from the pool at master level IF the resources are not already assigned at each subproject level. Is this getting a little confusing?

    As far as not wanting to re-create a static master each time, there are a couple of ways to deal with that. First, if you primary goal is to view resource information, you do not need a master file. All that information is available in the resource pool file. If you do need to view task related information on all subprojects with a master, you can record a macro of you creating a static master. Once recorded, all you need to do the next time you need an update is to run the macro. This works fine as long as you never move any of the subproject files to a different location.

    With regard to having undesired indenture when inserting subprojects in a dynamic master. The easiest approach is to select all subproject at once when you do the insert. If you do find you need to add another subproject to an already existing dynamic master, I suggest you first collapse the last inserted subproject so only the insertion point summary line remains and then insert the new subproject after that. Then expand as desired.

    Hope this helps.

    John
