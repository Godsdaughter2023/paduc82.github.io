Hi Jared,

 

About the error message “There are multiple recipients matching the identity...Please specify a unique value”, it may occurs if multiple mailboxes or mail contacts have the same display name. Exchange examines the Display name field or the displayName attribute when it creates a rule. You can change the display name in the Exchange admin so that the name doesn't conflict with an existing mailbox or mail contact.

 

Meanwhile, to avoid this problem, please use the user's Simple Mail Transfer Protocol (SMTP) address instead of the display name when you create rules.

The basic code would be as follows. I have made the search not case sensitive. The text to find is strText - here set from an input box.

Option Explicit
Sub GetOLAppt()
Dim olApp As Outlook.Application
Dim olFolder As Outlook.Folder
Dim olItems As Outlook.Items
Dim olItem As Outlook.AppointmentItem
Dim strText As String
Dim bStarted As Boolean
Dim i As Long
On Error Resume Next
Set olApp = GetObject(, "Outlook.Application")
If Err <> 0 Then
    Set olApp = CreateObject("Outlook.Application")
    bStarted = True
End If
strText = LCase(InputBox("Enter text to find"))
Set olFolder = olApp.Session.GetDefaultFolder(olFolderCalendar)
Set olItems = olFolder.Items
For i = olItems.Count To 1 Step -1
    Set olItem = olItems(i)
    If InStr(1, LCase(olItem.Body), strText) Then
        '*********************************
        'do what you want with olItem e.g
        MsgBox olItem.Subject
        '*********************************
    End If
Next i
If bStarted Then olApp.Quit
Set olApp = Nothing
Set olFolder = Nothing
Set olItems = Nothing
Set olItem = Nothing
End Sub

Why should I have a separate row for each exam score even though the different exam scores are for the same student?

In the early 1970s Dr E.F. Codd came up with the concept of a relational database structure. That design used a concept called normalization to determine how tables should be setup within a database. These rules are still in effect largely unchanged in almost 40 years. The vast majority of current databases uses this same relational model. The design has stood the test of time.

One of the rules of normalization is that you don't define data using field names. This is what you would be doing by using field names like Exam1, Exam2, etc. I'm not going to go into all the nuances of normalization. There are plenty of web sites that discuss it if you want to do the research. What I can tell you is that structure of the data that you receive appears to be a properly normalized structure.

Now it is true, that sometimes a normalized structure may not make it easy to create a report. This would appear to be the case with you. But there are ways to get around that. Karl mentioned a Crosstab, I gave you a function that will concatenate the entries. or you can simply display the detail band of the report in columns.

qa guy commelec say:

These queries ---

   qry_ExamScore --
SELECT tblExamScore.[Student Name], tblExamScore.[Student ID Number], tblExamScore.[Date of Birth], "1" AS [Exam], tblExamScore.[Exam Score 1] AS [Score]
FROM tblExamScore
WHERE  tblExamScore.[Exam Score 1] Is Not Null
UNION ALL SELECT tblExamScore.[Student Name], tblExamScore.[Student ID Number], tblExamScore.[Date of Birth], "2" AS [Exam], tblExamScore.[Exam Score 2] AS [Score]
FROM tblExamScore
WHERE  tblExamScore.[Exam Score 2] Is Not Null
UNION ALL SELECT tblExamScore.[Student Name], tblExamScore.[Student ID Number], tblExamScore.[Date of Birth], "3" AS [Exam], tblExamScore.[Exam Score 3] AS [Score]
FROM tblExamScore
WHERE  tblExamScore.[Exam Score 3] Is Not Null;

TRANSFORM Sum(qry_ExamScore.[Score]) AS SumOfScore
SELECT qry_ExamScore.[Student Name], qry_ExamScore.[Student ID Number], qry_ExamScore.[Date of Birth], Sum(qry_ExamScore.[Score]) AS [Total Of Score]
FROM qry_ExamScore
GROUP BY qry_ExamScore.[Student Name], qry_ExamScore.[Student ID Number], qry_ExamScore.[Date of Birth]
PIVOT qry_ExamScore.[Exam];

Build a little, test a little.
