In a multi-user environment, each user must have their own local copy of the front-end on their pc.  Not share a common copy off of the server.

 

Typically, the db admin is the only person who should make any design changes to database objects.  If you start allowing users to do so, things will quickly become unmanageable.  Each user doing their own thing.  Then you make an update and they loose all their changes...  This is why there is an administrator.  Someone to manage things and keep control.

Nevertheless it does sound like all users are opening the same front end file, which is what is causing the problem.  Each user should have their own copy of the front end installed either on their local machine or in a 'personal' folder on the system.  Normally Access itself would be installed on each user's local machine, but in one organisation in which I worked Access was installed by the IT staff in a shared location on the system and separate copies of each front end on users' local machines.  While this was not what I recommended in the application's documentation, it seemed to work without any problems.

mukherjee ashutosh say:

Try the following steps to fix issues related to PowerPoint while printing posters using large-scale printers:

 

Open PowerPoint 2007 click on the ‘Design’ section and select ‘Page Setup’

 

Under the ‘Slides Sized for’ option select the size as ‘Custom’

 

Set the page size to the same size that you want the poster to be. For example, if you want your poster to be 42" x 48", then set the page size in PowerPoint to 42" x 48". Failure to set the above settings before creating the poster may result in a poster that is not printable, or may not get printed at the size you need.

 

PowerPoint has a maximum page size of 56". If you want a larger poster, set the page size (both width and length) to exactly half of the finished poster size and then enlarge it proportionally when you want to print it. For example, if you want a 42" x 60" poster, set the page size to 21" x 30". You can then enlarge it proportionally to Scale To Fit 42" x 60" at the time of printing. Do not set the page size to 42" x 56" if you want it to end up at 42" x 60". Any enlargement done enlarges both the dimensions. Remember, the maximum finished size of one dimension is 42”. When you submit your poster for printing ensure to mention the amount of scaling, e.g., 150%, 200% and the final size. Keep the scaling factor even.

 

You can only enlarge proportionally so ensure the aspect ratio of your page size in PowerPoint matches that of the final poster size, failure to so may result in a poster which may not print correctly. For example, if you plan to print at more than 100% (e.g. 200%) and you set the paper size to 42" x 46", then when it's enlarged both dimensions will be greater than 42" which won't work.

 

Once the poster is created, save the file, in PowerPoint 2007, then select ‘Save As’ from the ‘Office’ button

Click on the ‘Tools’ menu besides the ‘Save’ button and select ‘Save Options’,

Check the box ‘Embed Fonts in the file’

 

If PowerPoint encounters a font that it cannot embed it will throw an error message stating exactly which font it cannot embed. If you get an error message that means you are using a non-Microsoft font somewhere in your presentation. PowerPoint cannot embed non-Microsoft fonts therefore change the font. Failure to embed fonts may result in a poster that will print incorrectly.

    Linking master child fields - Microsoft Community

    kristendepuyt say:

    Linking master child fields

    I am unable to get my foreign key to populate in a subform but the property sheet does not give me the option to link the fields.  What am I doing wrong?

     

    Thanks!

    dirk goldgar say:

    I've looked at the database you sent me.

    The subsubform's name is SkillAssessment, its SourceObject is the form Skill Assessment, its Link Master Fields and Link Child Fields properties are both "EmpRatingID".  The RecordSource of the SkillAssessment form is the SQL statement:

        SELECT [tblRating].[EmpRatingID], [tblRating].[Rating], [tblRating].[Code], [tblSkills].[SkillDescription], tblRating.RatingID FROM tblSkills INNER JOIN tblRating ON tblSkills.SkillCode=tblRating.Code;

    All that is perfectly reasonable.

    The problem was that, on the subsubform "SkillAssessment", the text box named "EmpRatingID" had its ControlSource set to the expression:

        =[tblRating].[EmpRatingID]

    which rendered it non-updatable.  Note the leading "=" sign, which means this is a calculated field.  Calculated controls are never updatable.  What the ControlSource *should* have been set to is just:

        EmpRatingID

    I've sent you an updated version of the database with that change.  In the revised version, the form works fine.
