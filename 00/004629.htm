ken sheridan say: Are the queries restricted by parameters?  If so the parameters should be declared as DATETIME to prevent their being inadvertently misinterpreted as arithmetical expressions, which will not raise an error, but return the wrong results.  Also declaring the parameters caters for regional differences in date formats.

One caveat: if a parameter other than one of DATETIME data type is being made optional by means of the following syntax:

WHERE (ColumnName = <parameter> OR <parameter> IS NULL)

then the parameters which are not of DATETIME data type must not be declared as this would prevent them evaluating to NULL.  DATETIME parameters are unusual in this respect, however, and can still evaluate to NULL if declared.

Different versions can sometimes be more sensitive to this sort of thing than others, which might explain why the queries worked correctly in 2003, but not in 2013.

I'm not sure I understand what you mean.  When you say, "when the cursor moves to the Form header section field," so you mean when a control in the header section gets the focus?  Or when the mouse pointer just moves anywhere over the header section?

You can deselect a selected detail record with the VBA statement,

    Me.SelHeight = 0

You could execute that statement in the GotFocus event of one or more controls in the form header section.  Or, if you want to do this just when the mouse pointer moves over the section, you could potentially use the MouseMove event of the header section and probably the controls in that section, to call a function to deselect the detail record.  Because the MouseMove event fires many times as you move the mouse pointer, you would probably want to make the function test whether any record is selected, like this:

Function DeselectRecords()

    If Me.SelHeight > 0 Then

End Function

tryx3 say:

I think I have found the instructions you were trying to follow - the instructions that told you that there was a temporary folder in use by Outlook for its attachments.

The instructions are in Error opening attachments / Cleaning out the Temporary Outlook Files folder.  The instructions are by Roady, a user who is also active in the Office, Outlook section of this forum.

When you use his instructions & copy the folder path in RegEdit, go to your Desktop and create a new shortcut, paste in the folder path then call the shortcut something like Outlook Secure Temp folder - ThisUserOnly.  You can then use this shortcut to go straight to the folder to clear it out. 

This folder is not a normal "folder" -

  • This "folder" can apparently be full, unlike proper folders
  • You cannot browse to it, you have to use the shortcut.
  • You might not be able to copy the shortcut to another user account on the same computer [even by adjusting for the difference in user names] i.e. you might have to repeat the RegEdit steps in each user account to get the correct path for that user.

Here are two examples of the folder path to the Outlook Secure Temp folder.  The paths will not be the same on yours. 

  • Windows 10 example - C:\Users\UserName\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook\TT5TH1U1
  • Windows 7 example - C:\Users\UserName\AppData\Local\Microsoft\Windows\Temporary Internet Files\Content.Outlook\349T7SYI"

[the "folder" is not even visible from C:\Users\UserName\AppData\Local\Microsoft\Windows\Temporary Internet Files, it is "super-hidden".]

When you open an attachment directly in Outlook, a temporary copy of the file is created in the Outlook Secure Temp folder.  When you close the file and its email, the temporary file stays there.  When you close Outlook, the temporary file should disappear.  One of mine contains 10 files dated since Dec 2011 so perhaps Outlook does not clean them out properly every time or perhaps each file represents an occasion on which Outlook or my OS crashed while a file was open.  Perhaps this is how this "folder" can get full.

I have been running some tests.  As I understand it, Outlook invokes the installed antimalware application to scan the attachment before it opens.  However, my antimalware application creates logs of its scans [even scans of individual files] and no entries have been made for the attachments I have been testing.  So I am going to stick with my existing procedure for attachments:-

1  In MS Office Outlook, double-click on the email to open it,
2  Right-click on the attachment & select SaveAs,
3  Save it in a convenient folder,
4  Right-click on the file in its new folder and scan it with an anti-malware utility,
5  Edit the document on the computer,

6  Re-attach it to any email reply, if necessary.
