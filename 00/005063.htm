There's an old saying among database designers: "Fields are expensive, records are cheap". What this means in this context is that you should not add fields for the dependents; instead you would have two tables,, in a one to many relationship: a table of Families and a table of People. You would simply add another record to the People table (using a form based on the Families table and a subform based on People) to add the new person.

scottgem say:

My question here is how do set up a table to capture a person's dependents when I don't know how many dependents a person might have.  

This is very much the situation that relational databases were designed to handle. The answers here is you set up TWO tables (at the least). What you have here is referred to as a One to many relationship. One person can have multiple dependents. So it requires two tables, A parent and a child table. The parent table has a Primary key value that identifies the parent. The child has a foreign key value that identifies the parent.

ken sheridan say:
My other concern is how I'm going to present this to the end-user in a form that's not too cumbersome. 

You'll find examples of different ways of representing a simple one-to-many relationship type in FormsDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file each client can be related to many events and visits.  Think of clients as being analogous to your applicants, and either events or visits as being analogous to dependants in your case.  The first example, which uses subforms on the pages of a tab control might suit your requirements best as it provides a 'one stop shop' for data entry.  In this the basic data about a client is shown above the tab control, so is always visible, while the subsidiary data about the client is shown on the first page of the tab control.  The second and third pages each have a continuous forms view subform for events and visits respectively, each linked to the parent form on ClientID.

In this case the parent form is based on the Clients table, and you could do the same, or you could base it on an a query which joins two tables, Applications and Applicants say, in which the Applications table includes an ApplicantID foreign key column.  That way you could enter an application, selecting the applicant from a combo box bound to ApplicantID foreign key column in Applications, with text boxes bound to other columns from the Applicants table.  For an example of this see the InvoicePDF  demo in my same OneDrive folder (think of an invoice as analogous in data modelling terms to an application in your case).  In this when you select a customer in the combo box, the customer's address data is automatically shown in the text boxes below.  You can also enter a new customer via this form by typing their name into the combo box; code in its NotInList event procedure then opens a form in dialogue mode in which the additional customer data, is entered.  This does not work well for personal names, however, as they can legitimately be duplicated.  A better way is illustrated in the NotInList demo in my OneDrive folder; in this the first form has a button to enter a new contact record via a form in dialogue mode.  Note also how in this form the combo box's RowSource concatenates the contact's name and address data into a single string expression to differentiate between contacts with the same name when making a selection.  The importance of this was brought home to me once when two patients arrived at a clinic when I was present, both female, both with the same first and last names, and both with the same date of birth.
