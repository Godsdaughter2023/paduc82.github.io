BUG REPORT: "Automation error - Catastrophic failure" with - Microsoft Community

kjell.eikland say:

BUG REPORT: "Automation error - Catastrophic failure" with table and UDF

After several months I finally found the reason why I could not reliably open a VBA UDF-heavy spreadsheet from my main computer on other computers, even with the same OS and Office versions:


The problem is the use of UDF in tables with computed column values.  In my case, I store the distance between two coordinates by evaluating the UDF NMiles(LLpt1, LLpt2). This generally works fine in an open spreadsheet and on my main computer, but turns out to be the source of the automation catastrophic failure on other computers.  Removing this computed column removes the problem.


So, either Microsoft should prevent the use of UDFs in tables, or - preferably - fix the bug so that the functions are available to be computed in the table when opening the spreadsheet.


Have other people had this problem or even found a workaround?


Regards

fintec123 say:

I have started having the same problem with a complex Excel model I built that uses UDFs and tables. Your post was really helpful in calling out UDFs and tables as the source of the problem.

I seem to have found a "solution" that has eliminated the catastrophic failure, at least for the model in question. Based on the observations of BW86, I switched the calculation mode to manual before opening the problem model. I then added the following code to the WorkBook.Open event in the code module for ThisWorkbook:

Private Sub Workbook_Open()

    Application.Calculation = xlManual
    Application.Calculation = xlCalculationAutomatic
   
End Sub

By switching the automatic calculation off and then back on during the Open event, the model no longer locks up with the "Automation error - Catastrophic failure" message.

I'm not sure if this will work in other cases, but you might want to give it a shot.
