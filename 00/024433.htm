If you don't have a lot of data (100 rows is not a lot) you could just try sorting it twice, first to get the zeros to the top, second to send them to the bottom after substituting some other value that will force them to the bottom:

 

Sub SortDASList()
'
' Macro25 Macro
' This macro sorts the Active DAS list by sign-on and sign-off time.
    Dim i As Long
'
    'Range("Total_DAS_List").Select
    ActiveWorkbook.Worksheets("Active Roster").Sort.SortFields.Clear
    ActiveWorkbook.Worksheets("Active Roster").Sort.SortFields.Add Key:=Range( _
        "R_31_DASSignOn"), SortOn:=xlSortOnValues, Order:=xlAscending, DataOption:= _
        xlSortNormal
    ActiveWorkbook.Worksheets("Active Roster").Sort.SortFields.Add Key:=Range( _
        "R_32_DASSignOff"), SortOn:=xlSortOnValues, Order:=xlDescending, DataOption:= _
        xlSortNormal
    With ActiveWorkbook.Worksheets("Active Roster").Sort
        .SetRange Range("Total_DAS_List")
        .Header = xlNo
        .MatchCase = False
        .Orientation = xlTopToBottom
        .SortMethod = xlPinYin
        .Apply
    End With
    ActiveWindow.SmallScroll Down:=0
    For i = 1 To Range("Total_DAS_List").Rows.Count
        If (Range("R_31_DASSignOn").Cells(i, 1).Value > 0.000001) Then Exit For
    Next i
    i = i - 1   ' Number of rows with a zero sign-on time

' Make the zeros some other value so they sort at the bottom
    Range("R_31_DASSignOn").Resize(i, 1).Value = 9.999E99

    With ActiveWorkbook.Worksheets("Active Roster").Sort
        .SetRange Range("Total_DAS_List")
        .Header = xlNo
        .MatchCase = False
        .Orientation = xlTopToBottom
        .SortMethod = xlPinYin
        .Apply
    End With

' Now put the zeros back to zero
    Range("R_31_DASSignOn").Resize(i, 1).Offset(Range("R_31_DASSignOn").Rows.Count - i).Value = 0
    End Sub

ken sheridan say:

Computed or derived values should only be stored in columns in a table if they are functionally determined solely by the primary key of the table.  Let's illustrate this with two examples, both in the context of an OrderDetails subform within an Orders form:

 

1.  When a product is selected in the subform by means of a combo box with the following as its RowSource property:

 

SELECT ProductID, UnitPrice,TaxRate,Product

FROM Products

ORDER BY Product;

 

the UnitPrice and TaxRate values should be stored in columns in the OrderDetails table.  This is because they can change over time, so the values for the same product in different rows in the table might differ.  Consequently the two columns are functionally determined by the table's key and the repetition of the columns in the two tables does not introduce any redundancy into OrderDetails, which is therefore normalized to Third Normal Form (3NF) and probably beyond.

 

If the values were shown in computed text box controls in the subform, with their ControlSource properties referencing the relevant columns in the combo box's RowSource, then as the values in the Products table change over time, the values in orders prior to the changes would be incorrect.  The two control boxes should therefore be bound to TaxRate and UnitPrice columns in the OrderDetails table.  Values are then assigned to these columns in the AfterUpdate event procedure of the product combo box with the following code:

 

    Me.UnitPrice = Me.cboProduct.Column(1)

    Me.TaxRate = Me.cboProduct.Column(2)

 

The Column property is zero-based, so these reference the second and third columns.

 

1.  In the same subform we need to show the total price for this row.  For this we do use a computed text box, as the value in this is always the same result of the computation.  So the ControlSource property of a text box control to show the total price is:

 

    = Quantity*UnitPrice*(1+TaxRate)

 

where Quantity is also a column in the OrderDetails table and the values in the TaxRate column are fractional values, e.g. 0.2 for the current standard rate of VAT in the UK of 20%.

 

The computed total price value must not be stored in a column in the table as it can always be computed from the values of other columns in the table.  To store it would introduce redundancy, and the table would therefore not be normalized to 3NF, and consequently open to the risk of update anomalies.

I want to create a sum for column based on a cell fill color, is this - Microsoft Community

jschw5521 say:

I want to create a sum for column based on a cell fill color, is this possible?

I want to create a sum for column based on a cell fill color, is this possible?
jackpot. say:

There is no built-in excel function to do this. But you can use a UDF that
looks at the range and returns the sum of color.  But that function will not
recalculate if you change color.  Every time you change the color you will
need to recalculate or wait excel to recalculate...

To install the UDF (User Defined function). From workbook launch VBE using
Alt+F11. From menu Insert a Module and paste the below function.Close and get
back to workbook and try the below formula.

To use the as a formula in range A1:A10 try the below; which will count the
number of blue cells....The second argument denotes the colored cell.

=COLORSUM(A1:A10,B1)

Function COLORSUM(varRange As Range, varColor As Range)
Dim cell As Range
For Each cell In Application.Intersect(varRange, varRange.Parent.UsedRange)
If cell.Interior.ColorIndex = varColor.Interior.ColorIndex Then
COLORSUM = COLORSUM + cell.Value
End If
Next
End Function
