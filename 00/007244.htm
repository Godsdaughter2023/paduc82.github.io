Yes.

1)  Use Tools>Unprotect document.  If you are prompted for the password, you better have it :-)

2)  When the document is unprotected, Select All

3)  Issue the key combination Command + Shift + F9

4)  Save the document as a new file name

Command + Shift + F9 is "Unlink Fields".  It converts all the fields in the selection into ordinary text.

Be sure not to save the original form like this, because it can't work as a form after you have done this.

marshall barton say:

What are you doing to make a field appear as "compulsory".  If you are setting the field's column to a different color, then use the Conditional Formatting feature instead of using VBA code or a macro.

 

If you are using VBA code or a macro in the column's BeforeUpdate, LostFocus or Exit event, then you should probably change it and put it in the form's BeforeUpdate event.

psyberfox say: I'm putting it in the field's BeforeUpdate event, but when fired it changes it for all records. Remember this subform is currently set to datasheet layout. Will putting this in the form's BeforeUpdate event change the way it works?

ken sheridan say: You can apply a CHECK CONSTRAINT to the table.  To add the constraint enter something like the following (as one line)  in the debug window (aka immediate window - press Ctrl-G to open it)

CurrentProject.Connection.Execute "ALTER TABLE [YourTableName] ADD CONSTRAINT YourConstraintName CHECK((SELECT COUNT(*) FROM [YourTableName] WHERE [Invoiced / Cancelled] = 'Invoiced' AND ([FirstOtherField] IS NULL OR [SecondOtherField] IS NULL)) = 0)"

To remove the constraint:

CurrentProject.Connection.Execute "ALTER TABLE [YourTableName] DROP CONSTRAINT YourConstraintName"

NB:  Name of constraint must be distinct within the database.

You can reinforce this in the subform's BeforeUpdate event procedure with code along these lines:

Const MESSAGETEXT = "FirstOtherField and SecondOtherField must both be completed."

If  [Invoiced / Cancelled] = "Invoiced" _
    And (IsNull([FirstOtherField]) Or IsNull([SecondOtherField]) Then
    Cancel = True
    MsgBox MESSAGETEXT, vbExclamation, "Invalid operation"
End If

I would recommend that you do both.  While you can rely on the subform's BeforeUpdate event procedure to provide the validation at form level, the enforcement of the constraint at table level ensures that the former cannot be subverted by data being entered in some other way.  While this is unlikely in the normal operation of the database, it cannot be ruled out with absolute confidence.

ken sheridan say:
HOWEVER, will adding a constraint not force each record to conform to the same rules?

No, the whole point of the CONSTRAINT is that it will only be violated if a row has both a value of 'Invoiced' in the Invoiced / Cancelled column and a Null in either of the other two columns, so rows which have a value other than 'Invoiced' in the Invoiced / Cancelled column can legitimately have Nulls in the other two columns.

BTW you can only apply the constraint via code as I described.  You cannot enter the DDL directly in the query designer in SQL view.

marshall barton say:

What are you putting in the event?

 

If you are using some kind of valdation code, then it would only apply to the current record.  So, I'm guessing you are setting some kind of visual effect (e.g. color), and that needs to be done using CF instead of code.

 

For Validation across multiple fields, the best place is to set the table's ValidationRule, e.g.
