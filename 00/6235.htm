marshall barton say:

Not sure there is anything that is "secure", so it's a matter of degree.

The first thing you need to do is split your db.  Put the back end (tables and relationships) on a central machine's share area.  Copy the front end (queries, forms, reports, etc) to each user's hard disk.

A way to secure the VBA code in the front end is to make a copy of the front end and save it as an MDE.  Be sure to use a copy if you make an MDE because all source code is removed so future changes must be made to the original NDB file.  Each form/report also has the AllowDesignChanges property that can be used to prevent users from making changes to the front end.

langware say:

Andrewmiller14,

I am not sure if this pertains to your problem, but there is a known issue between the new Outlook Mail interface and email clients that fetch messages from the outlook.com servers using the POP protocol. 

When an email client that uses the POP protocol (such as an email app on a cell phone) fetches new (unread) email from the new outlook.com interface (which runs Exchange servers), the Exchange servers mark that mail (on the server) as being read (even if those email messages were never opened by the user).

The old outlook interface's servers did not do this ... the old servers left the fetched mail (on the server) marked as being unread.

The new Outlook Mail interface is not backwards compatible with the old interface. Many customers have complained about this .... both here in Microsoft Community and on the uservoice site .... but so far Microsoft has not done anything to fix it.

langware say:

If you are seeing your emails in your web based outlook.com account being marked as READ (even though you have never opened them), and if your cell phone is running a mail app that uses the POP protocol to fetch mail from the outlook.com server, then the problem is that the new outlook.com interface (on Exchange servers) automatically marks all fetched email as having been READ.

Only the Microsoft developers can fix this problem, and they rarely follow these forums.

To post a complaint where the developers might see it, go here to the Microsoft support site and enter your complaint.

shane devenshire say:

Need to associate the file type with Excel:

In Windows 7:
1. Right-click the file,
2. Select "Open With" 
select [Your Program Here]
3. Check always use this program. 

In Windows XP:
1. Locate the file as you have described above
2. Right click the file
3. Select Open with from the pop-up menu
4. Click Choose default program…
5. Select Excel in the Recommended Programs box
6. Check Always use the selected program to open this kind of file
7. Click on OK.

If this answer solves your problem, please check Mark as Answered. If this answer helps, please click the Vote as Helpful button. Cheers, Shane Devenshire

jan karel pieterse say:

Unless the data has dates as the category axis, Bar charts are not spaced according to the relative size of the category data. So even if your x data is 1, 2,3,7,8,9 it will show all bars next to each other without a gap between 3 and 7.

If you draw a trendline through those X values, a nonsensical trendline is calculated, because Excel treats the data as if being equally spaced.

Hence the advice to use an XY scatter chart.
