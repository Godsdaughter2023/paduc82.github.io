jay freedman say:
I should have read your post more closely and seen that you're using the Master Document feature. That feature has been so badly broken for so many versions of Word that I'm amazed it's still part of the program. The usual strike against it is that it routinely causes document corruption, especially when users try to edit the text of subdocuments while working in the master document. I don't recall seeing anything about the feature preventing the exclusion of subdocuments from the TOC, but it wouldn't surprise me at all.

Private Sub Document_Open()
    Dim WhatDoc As String
    Dim fld As Field

jllatham say:

As Jean-Yves Tfelt has said - look at ways to combine similar routines so that you can possibly combine the major working part of them into a single Sub with some parameters and just call that routine and send appropriate parameters from each of the buttons.

I actually just completed such an exercise for one project.  Picture a spreadsheet with 10 sets of 2-row by 15 column areas for data.  Each one has a button that opens a UserForm so that the user can identify equipment whose information will be placed on the 2 rows associated with that button.  All the button code does is put the 'button number'  into a Public variable that the UserForm then uses to determine where to put the information on the sheet.  Result: 10 2-instruction Subs calling a single UserForm that does all the work for all 10 data areas on the sheet.  In this case the button number is used to calculate the 1st row of the group using other Public Const values that hold the first row number of the first group and the number of rows per group.  So future changes to the worksheet only require a change to the Public Const values to keep the code working without further maintenance.  And if the layout of the groups changes (say moving fields from one column to another) - well the column IDs for the fields are also set up as Public Const values, so a single change in the Public declarations section fixes things easily.  And finally, if the logic of the data processing changes then it only has to be changed in the code for the one UserForm, not in 10 separate locations within the project.

Now, none of what I just described improves performance any - but it does set you up so that you can work on improving the performance of just one Sub or Function and once you get it tuned up you're done - you don't have to worry about remembering to revise the code in 2 or 3 or 15 or 20 additional locations in the project.  If nothing else this is a maintenance reduction feature along with a file-size reduction setup.  In the end it adds to the quality of your product.
