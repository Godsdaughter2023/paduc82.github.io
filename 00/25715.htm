jordan dom say:

Hi Brissa,

 

In Outlook 2016 and Outlook for Office 365, IMAP accounts use Offline Outlook Data Files (.ost). The Offline Outlook Data File (.ost) files stores a synchronized copy of your mailbox information on your device. per jessen say:

WIth is used to set a reference to an object, for instance a worksheet object. The advantage of this construction is that you do not have to write the (worksheet) object, but just use a leading dot to indicate that the statement is referencing the 'With' object.

 

With Worksheets("Sheet1")

   Set MyRange .Range("A1",.Range("A" & Rows.Count).end(xlup))

end With

 

WIthout the With statement we would write:

 

Set MyRange=Worksheets("Sheet1").Range("A1",Sheets("Sheet1").Range("A" & rows.count).end(xlup)

 

Hopes this clarifies it a bit...

jllatham say:

As Per Jessen has explained, it is a 'shortcut' way to deal with a higher level object.  Not only does it reduce your typing, but it can make your code clearer if you will indent the inside of the With ... End With block.  I believe there is also a speed advantage to be gained if you are going to reference multiple properties of an object; each dot-level that do not have to be re-evaluated represents a slight performance improvement.  You can even nest With ... End With blocks.  So...

    With Range("J9")
      .Value = "Hello World"
      With .Interior
        .ColorIndex = 6
        .Pattern = xlSolid
      End With
      With .Font
        .Name = "Arabic Typesetting"
        .Size = 10
        .Strikethrough = False
        .Superscript = False
        .Subscript = False
        .OutlineFont = False
        .Shadow = False
        .Underline = xlUnderlineStyleNone
        .ColorIndex = 55
      End With
    End With

should run faster than the following, even though both give exactly the same results

      Range("J9").Value = "Hello World"
      Range("J9").Interior.ColorIndex = 6
      Range("J9").Interior.Pattern = xlSolid
      Range("J9").Font.Name = "Arabic Typesetting"
      Range("J9").Font.Size = 10
      Range("J9").Font.Strikethrough = False
      Range("J9").Font.Superscript = False
      Range("J9").Font.Subscript = False
      Range("J9").Font.OutlineFont = False
      Range("J9").Font.Shadow = False
      Range("J9").Font.Underline = xlUnderlineStyleNone
      Range("J9").Font.ColorIndex = 55

joeu2004 say: Phil wrote:
I am looking for an explanation of, and the ramifications of with ... end with.
It seems like some forms of reference do not work within the confines of a with...end with block.
For instance:
    LastCol = .Cells(1, .Columns.Count).End(xlToLeft).Column
is valid inside a with..end with block, but outside of it gives me an error

As you noted, the key difference is the period before some class members like Cells, Columns, etc.

Conceptually, the With statement coupled with the period is shorthand.

For example, within With Application ... End With, you can refer to .ScreenUpdating, .Calculation, etc instead of writing the more verbose forms Application.ScreenUpdating, Application.Calculation, etc.

Without the period, some class members have a default qualifier.  For example, Range("a1") is equivalent to ActiveSheet.Range("a1").   But withing With Worksheets("sheet2") ... End With, .Range("a1") is equivalent to Worksheets("sheet2").Range("a1"), which is different object unless ActiveSheet is Worksheets("sheet2").

Walkenbach also claims that using With statements can improve performance by reducing what he calls "dot processing".  But I have not tested the effect myself.

On the other hand, the use of the With statement is error-prone.

Not only can it become unclear what .Range("a1") actually refers to, but also it is all too easy to omit the period inadvertently and the unqualified class member becomes a valid reference to something else altogether.  It takes "forever" to debug such mistakes because our brains tend to read what we expect, not necessarily what is there before us.

So personally, I avoid With statements, especially nested With statements.

I might use them in very short blocks, e.g. With Application ... End With.  And I might consider using them in some code paths if I can demonstrate a real performance benefit.

But generally, I favor readability and reliability over performance, except in CPU-intensive algorithms than might run for long periods of time.

[EDIT]
PS:  Jerry Latham's example is a good example where With ... End With makes good sense.  Because the dot-member name is always on the left, it is much easier to be sure that is present.  Moreover, it is an example where the use of With ... End With actually enhances readability and maintainability.
