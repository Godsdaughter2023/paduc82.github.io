    davidf2 say:

    I waited to see if the MS Support Engineer would come back with a solution or an explanation...

    The problem you are experiencing is not new to this forum. The 'jump to the anchor' linking feature will only work if the anchor is 'above the fold' of the page. I don't know quite how to explain this but I discovered it building web pages with Publisher.

    Publisher web pages can be any length and as long as you insert the anchor in the top section of the page it will work OK.  However 'top section of the page' is variable. Scroll any web page up until the bottom of the page reaches the bottom on your browser viewport and you cannot scroll any higher. For a link to the anchor to work correctly it must be above the page content that is visible with the page scrolled to the bottom page.

    The same is true of an anchor link on a PDF page. Publisher assumes that your PDF page is like a web page and the anchor will only work if it is in the top of the page that does not require scrolling up past the bottom of the page. Yes, I know that most or all PDF viewers automatically starts scrolling the next page as you approach the bottom of a page but Publisher assumes there is no following page much like the last page of your PDF file; you cannot scroll up beyond the point where the bottom of the page aligns with the bottom of the viewport.

    Clear as mud, right? I told you I didn't have the right words to describe this issue but the bottom line is I have never found a workaround and was hoping the MS engineer would provide one. It would seem that since most PDF viewers treat a multiple page file as one continuous single page when scrolling, then you should not be limited as you are with a web page. But that is the problem and to my knowledge there is no solution. Sorry, and hope I explained this adequately.

    scottgem say:

    I think you misunderstood me. There are a number of good reasons why one would want to be able to link external files to an Access database. I have several apps that do that. I wasn't asking why you wanted to do that. I was asking why use an Attachment field to do it. 

    The method I suggested doesn't use either OLE or the attachment datatype. It can be just as easy for the user to select files to attach, it just requires some standard code using the FileDialog property to select the file. But the user just triggers an event to run that code. It can store as many attachments as you need per record. PLUS, it allows the ability to easily search through attachments, accept any type of file as an attachment, and easily launch the file when needed. 

    All you need is an Attachments table like this:

    tblAttachment

    AttachmentID (PK Autonumber)

    ParentID (FK to parent record)

    AttachmentPath

    That last file is a text datatype containing the full path and filename for the attachment. That's the bare minimum you need. But you can expand on that as you can't do with the Attachment Datatype. For example fields like:

    AttachmentDescription (a brief synopsis of what the file is)

    AttachedBy (the user who attached the file)

    AttachDate (the date when the file was attached)

    AttachType (the type of file attached (image, spreadsheet, PDF, DOC, etc. ))

    Another possibility is applying security levels so only some users can see certain attachments. My point is that using this method provides a much more flexible approach to attaching external files to a record with none of the issues of the attachment datatype.
