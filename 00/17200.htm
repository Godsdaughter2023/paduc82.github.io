ken sheridan say: As Marshal says, do not include an Asset Number column in the table!   That introduces redundancy and leaves the table wide open to update anomalies.  Sooner or later Murphy's Law will come into play and your data will be screwed up.   Simply use an unbound text box in a form or report with a ControlSource property of:

=[PlantName] & [Category] & [Type] & Format([SeqNo],"0000")

or similarly in a query use a computed column by putting the following in the 'field' row of a blank column in query design view:

Asset Number:[PlantName] & [Category] & [Type] & Format([SeqNo],"0000")

You are then guaranteed that the computed value will always accurately reflect the values of the four columns.

Another important issue is the indexing of the table.  Are the SeqNo values distinct throughout the table?  If so the column should be indexed uniquely, or designated as the primary key, which creates a unique index.  Or are the  SeqNo values distinct per one or more of the other three columns, e.g. does the sequence start at zero for each plant name?  If so there should be a unique index on the multiple columns.

Another matter to be considered is whether the table is correctly normalized.  For each value of PlantName will the value of Category and Type always be the same?  With your sample values for example does a PlantName value of WI mean that the Category value is inevitably F and the Type value 1?  If so then the Category and Type columns are redundant in the table.  In the jargon they are said to be 'transitively functionally dependent' on the key of the table, so the table is again open to update anomalies as any of the three values can be changed independently of the others.  In this situation the Category and Type columns should be removed from the table.  The form can be based on a query which joins the current table to a Categories table, and this to a Types table, or you can include computed columns in the form which look up the category and type from the respective tables.  You'll find a demo of the latter approach at:

http://community.netscape.com/n/pfx/forum.aspx?nav=libraryMessages&tsn=1&tid=23626&webtag=ws-msdevapps

It uses the local administrative areas of parish, district and county in my part of the world, but the principle is exactly the same.  The form using 'multi-column combo box and text boxes' might suit your scenario.  It's also very easy to implement as, unlike the other forms in the demo which use correlated combo boxes, it  uses no 'hybrid' controls or VBA code whatsoever.  In your case, using this approach, you'd select a plant name from a list sorted by Type then category then plant name (the parish combo box in the demo)  and the category and type text boxes would show the correct values for the plant selected.  You can of course sort the combo box's list however you like, so it could be sorted by plant name if this fits your modus operandi better.

Ken Sheridan, Stafford, England
