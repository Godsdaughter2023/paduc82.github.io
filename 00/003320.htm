erickchu_msft say:

Hi Bing,

Question 1: EOP is the 1st line of defense for spam with Outlook/OWA being the 2nd line.

There are several levels to protect mails:

About “EOP is the 1st line of defense”, the answer is yes.

After that, there is the admin level. Admin can use some rules to protect emails, such as transport rules, spam filters.

The third level is the user’s level. They can use block and safe sender list or inbox rules to set up some conditions.

Question 2: The hybrid end user does not have the ability to create a “block or safe sender” list in EOP, only OL.

All end users cannot create a “block or safe sender” list in EOP.

Question 3: On-premise users can’t sync their block/Safe lists with O365 because they do not have a mailbox in O365, only an EOP license.

Since only one side has users’ mailbox in the Hybrid environment, the question is not related to license. If a cloud user with mailbox, the block/Safe list is also the user’s level. It will not be synced to EOP.

Actually, if users think it is a spam, they can submit false positive to our related EOP team to check. If they accept it, they will change the configuration of EOP. Here is a reference:

Submit spam, non-spam, and phishing scam messages to Microsoft for analysis

Question 4: Items in a user’s “block” list is acted upon on-premise, not through EOP or O365

The answer is “yes”. However, if the user mailboxes are moved to cloud in the Hybrid environment, it is acted by Office 365. Therefore, it is depending on the location of the mailboxes.

Question 5: what happens to blocked mail that is stopped on-premise?

If users add email addresses to block sender list, related emails will be moved to users’ Junk folder.

If anything is unclear, please feel free to let me know.

Erick

paulcordts say: You can only list each command button once, but you can get it to do multiple things depending on conditions, but both are triggered by the same button click

Generically 

Private Sub CommandButton1_Click()

   If Range("C2").Value = "D" Then
          'Do Stuff

           Exit Sub 'Macro execution ends

     End If   

    If Range("C2").Value = "Z" Then
         'Do other stuff    

    End If

End Sub


Reacts to the value in C2 and allows you to do one thing if C2="D" and another if C2="Z"


You could also react to multiple conditions with a select case

Private Sub CommandButton1_Click()

Select Case Range("C2")

Case "D"

'Do Something

Case "Z"

'Do Something Else

Case "E"

'Do a third thing

End Select

End Sub

If none of the cases are met -  nothing is done.


You can have more than one Case for an action set Case "Z","H" would do the same thing if C2="Z" or C2="H"

You can also use a Case Else to do something if none of the other cases are met.

Your table name is "2018"? And you have another table "2017"? That is almost certainly a mistake. If you would like feedback on your database design, please post a screenshot of the Relationships window in a new thread.

To your question: you cannot compare this way. How would Access know which row in the table to check? You need to be more explicit, and that is done with the DLookup function:

[Forms]![Open HomeAway]![text9] = DLookup("Reservation ID", "2018", "???")

It's up to you to replace ??? with a where-clause, something like "CustomerID=5".

scottgem say:
First you don't have a closing bracket on Reservation ID. But that's probably not the problem. What is 2018? I suspect its the name of a table. If so, you can't address a field value that way. Access would not know what record to look for. You have to use a Dlookup to pull the value from a table.

On a separate note, having a table names 2018 indicates a problem with your table design. You are using data to define an object. You shouldn't have separate tables for each year. You should have a date field or a year field in your table that defines the period for the record.
