marco burato say:

USBSer - WaitCommEvent issues on Windows 10

This is a bug report, I'm posting it here as suggested in the following blog post:

http://blogs.msdn.com/b/usbcoreblog/archive/2015/07/29/what-is-new-with-serial-in-windows-10.aspx

On Windows 10 the WaitCommEvent does not work correctly.

According to MSDN:

The WaitCommEvent function detects events that have occurred since the last call to SetCommMask or WaitCommEvent. For example, if you specify the EV_RXCHAR event as a wait-satisfying event, a call to WaitCommEvent will be satisfied if there are characters in the driver's input buffer that have arrived since the last call to WaitCommEvent or SetCommMask.

https://msdn.microsoft.com/en-us/library/aa363193(v=vs.85).aspx

My tests show that on Windows 10 WaitCommEvent does not report an event on some circumstances.

In particular, the function doesn't seem to report events that happened since the SetCommMask call. It doesn't even consistently report events happened since the last call to WaitCommEvent itself.

I've prepared a simple test program to demonstrate the issue.

To reproduce:

- Get an Arduino Uno (uses the USBSer driver) and program it with the Blink example sketch or any other sketch that doesn't configure pin 1, 2 as a serial port to keep the pins floating

- Bridge pins 1, 2 (TX and RX). This will cause all the data coming from the USB-UART bridge to be sent back to the PC.

- Run the following test program:

https://dl.dropboxusercontent.com/u/22455544/WaitCommEventTest.cpp

The program will run a test in an infinite loop and terminate in case of errors.

There are 2 test routines defined (Test1 and Test2). Both work properly in Win7, I've run Test2 for 1 million cycles.

On Win10, Test1 will fail immediately. Test2 will fail after some time depending on the hardware. On my Surface 3 it only takes a few cycles, on another PC it took a few hundred cycles to fail.

I can provide more details, this should be enough to repeat the issues though.

mmzafrog say:

Thought I'd take a moment and detail the workaround that I implemented and appears to work. The answer is, indeed, in creating a new column to hold the value that I lookup against. Steps taken:

1. Insert a NEW column in the Parent file immediately to the left of the column the user enters the ID values into

2. Add a function to the new column: =IF(ISERROR(INT(A11 * 100)), "NOVAL", INT(A11 * 100))

3. Modify the VLOOKUP in the Child file: =VLOOKUP(INT(B11 * 100), 'path-to\[parent-file.xlsm]Sheet1'!A11:D1048576, 4, FALSE)

So now I've got integers compared to integers on both sides. This works.

A further trick I implemented is to hide the NEW column in the Parent file. This is potentially problematic as copy & paste might fail to add the function to the hidden column for any new rows. As such, I added an Worksheet_SelectionChange() routine that double-checks the presence of the function and inserts it if not found. This was only moderately tricky. Since I take advantage of ActiveSheet.UsedRange.Rows.Count quite a bit, I have to make sure the Selection is within the used portion of the spreadsheet:

If Selection.Row <= ActiveSheet.UsedRange.Rows.Count Then ...

ken sheridan say:
Note that when I initially open the form, the LstCategory list box is blank.  It populates when I pick an HR Manager in the ComboHRManager combo box.  (I'm also working on that problem as I want the table  toshow data for all HRManagers upon opening the form).

In the WHERE clause you'll need to test for the parameter being Null.  The expression would be:

([HR Manager Name]=[Forms]![frmMainMultiListBox]![ComboHRManager]
OR [Forms]![frmMainMultiListBox]![ComboHRManager] IS NULL)

Note that the expression is parenthesized to force it to evaluate independently of the AND operation on the HRUserRole column.

What is the RowSource property of the ComboHRManager control?  It could be that the value of the control is a hidden numeric ID whereas what you see listed are the names.   A combo box of personal names should normally be set up like this as personal names can legitimately be duplicated, and often are, even within a small population (I worked with two Maggie Taylors.  Also on one occasion two patients turned up at a clinic at which I was present, both female and not only with identical first and last names, but also identical dates of birth.  It's a smaller world than we sometimes think!).  For this reason one or more additional columns should also be shown in the combo box's list to differentiate between identically named people.

BTW when building an SQL statement in situations like this it's better to concatenate the value of the parameter into the string expression rather than including a reference to it.
