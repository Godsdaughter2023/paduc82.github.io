albert d. kallal say:

If you  compile the accDB to a accDE, then all errors are ignored and some stray error will not shut-down the runtime.

Not only does this simple step make the application VERY reliable, but even local and global variables are NEVER re-set, and that includes any VBA code without any error handling.

If you run a accDB in the full version, then a un-handled (or un-expected error) will throw up and out a error message.

If you run a accDB in the runtime, then a un-handled error will toss up a error message, AND THEN shutdown Access.

Needless to say, the simple act of using a accDE will solve all your issues in this regards. So not only does the accDE not shutdown, but it will NEVER re-set and lose your local or global variables.

This simple act of compiing to a accDE and using that will thus result in your application working like a un-stoppable freight train and wll NEVER shut down due to an error, nor will you EVER lose global or local variables  EVER!

So, while you could go on a wild goose hunt and start introducing error handling, you can well still easy miss some routine, and any error occurring will blow out your application and shut it down.

If you use a accDE with runtime, then this will NEVER happen and errors will NEVER shut down your application, and your variables will NEVER lose their values.

For the above reasons, I thus STRONG recommend you save world poverty, and simply use a compiled accDE, and all your issues, problems and preventing shutdown of the runtime will go away.

kosh vorlon say:

Hi Amina,

 

Here's a couple of things about scheduled scans. To run, the PC must be on and connected to the internet (it cannot be shutdown, in sleep mode, in hybrid sleep mode, or in hibernation). The power saving settings must be such that the PC does not shut down or go asleep during the scan (or it will be as if it never happened). Scans do not begin precisely when scheduled - some can start an hour or so earlier or later than the officially set time (depending on Microsoft's servers and the loads at that time). There is a way to use Task Scheduler rather than MSE to awaken sleeping computers, run the scans, and then shut them down when done, but it is somewhat complex. More on that shortly.

 

Also, go to settings and check scheduled scans and uncheck the box that prevents them from starting unless the system is on and idle (if you're using the system when it's scheduled or it otherwise isn't idle - perhaps downloading Windows Updates or doing something else - then the scan will not start and will be missed). If a scan is missed it does not start at the next available time, but waits for the next scheduled scan.

 

Go to Start / Control Panel / Administrative Tools / Services / scroll to Task Scheduler and make sure it is started and automatic. If not set it to automatic and restart it. If it won't start, click on the dependencies tab and do the same thing for each of the ones listed. If any of those won't start, continue with it's dependencies and do the same thing. When you get to the end, reverse course and Start the ones you couldn't because of the dependencies as you go until you get to Task Scheduler and hopefully it will now go to automatic and start. If you changed nothing read on. If you changed anything, reboot and see if that helps.

 

There is a way to use Task Scheduler rather than MSE to awaken sleeping computers, run the scans, and then shut them down when done (and other options), but it is somewhat complex. You can also use it to create different scheduled scans of different types since MSE only allows one choice.

 

Please see http://answers.microsoft.com/en-us/protect/forum/protect_start/running-mse-full-scan-as-a-scheduled-task/f627f5ce-0b74-4b1d-be64-d6ceeb4d9b02 - especially the later corrections from GreginMich.

 

The post from GreginMich here http://answers.microsoft.com/en-us/protect/forum/protect_start/scheduled-scans-not-working/fd769043-f67e-4990-8805-a02427f22dff will also be useful.

 

I'd also suggest the post from GreginMich here http://answers.microsoft.com/en-us/protect/forum/protect_start/i-would-like-to-schedule-mse-to-scan-pc-with/58474560-4cca-49c1-a51a-76ca9bc3b9f2

 

I hope this helps. If not, please explain what did or did not happen with what you tried as well as the current situation which may be just the same, but I suspect may be different but related.

 

Good luck!

ken sheridan say: You can reset an autonumber column by compacting and repairing the database.  However, an autonumber is designed only to guarantee distinct values, not sequential values.  If the value in each row is semantically significant, which it rarely is with a surrogate key, then an autonumber should not be used; the value should be computed when a row is inserted into the table.

You'll find an example in CustomNumber.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

john w. vinson say:

Is the table now empty? If it has data in it, then the [rec_id] field must have a unique Long Integer value for every record; it would appear that you're trying to create a new (hence necessarily empty) field!

If your only purpose is to reset the starting point of the Autonumber in an existing table, well... don't; an Autonumber is NOT intended to be a count, or a gapless, meaningful, human readable identifier. It's intended to be a meaningless unique ID and NOTHING ELSE. In practice autonumbers will always have gaps, and can even become random (if you Replicate the database for instance).

If the table is in fact empty and you want to start the autonumber over at 1, simply Compact and Repair the database. This will reset (all) the Autonumbers to either start at 1 or at one more than the largest value.
