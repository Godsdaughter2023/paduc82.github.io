ken sheridan say:

As both John and Scott have pointed out you should not have a TechName column in FSRData as that introduces redundancy, and means the table is not normalized to Third Normal Form, which requires all non-key columns to be determined solely by the table's primary key.  It is consequently open to the risk of update anomalies.

So, having deleted the TechName column from FSRData, why not give your users the opportunity to either select the ID or the name?  You just need to set up two combo boxes on the form, with TechID as the ControlSource property of each.

To allow selection by TechID the RowSource property of the first combo box would be:

SELECT TechID FROM Table_TechnicianDropDown ORDER BY TechID;

It's other properties can be left as the defaults.

To allow selection by name the second combo box's RowSource property would be:

SELECT TechID, TechName FROM Table_TechnicianDropDown ORDER BY TechName;

The properties of this combo box would be:

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

Whether the user selects a TechID or a TechName the other combo box will automatically show the correct value.

One caveat: personal names can legitimately be duplicated, so if two or more technicians have the same name (not as unlikely as you might think - I worked with two Maggie Taylors) there will be two or more identical rows in the second combo box's list.  Ideally some other value(s) should be shown in another column or columns in the list to allow them to be differentiated.  You'll find an example in NotOnList.zip in my public databases folder at:

scottgem say:

Ok an Outer Join means that you are asking for all records from one side of the join and only matching records from the other side. This means that you have a related table that doesn't have related records for every record in the parent table. This is certainly possible, but if you don't understand the error message I'm wondering if you really need outer joins. Can you post the SQL statement for that query?

Ambiguous outer joins can occur if you have too many in a query. The answer to that is to use (what I call) interim queries. You can create a query with one outer join, then use that as the source for a query with another outer join. This way you can build up to the final query that is the recordsource of the report. I remember one report that required 6 interim queries to get the final recordset I needed for the report,

ken sheridan say: Ambiguous outer joins are where there is a conflict between two or more join types in a query.  You might have one LEFT OUTER JOIN for instance where you wish to return a row from one table regardless of whether there is a matching row in another, in which case the columns from the other table will return Nulls if there is no match.  At the same time you might have an INNER JOIN where only rows where there is a match in both tables will be returned.  The two join requirements might not be logically compatible.  In many cases changing the INNER JOIN to either a LEFT or RIGHT OUTER JOIN as appropriate will remove the incompatibility and allow the query to open.  Sometimes this is not the case and you will have to build the final query by joining separate queries in the way Scott described.

Bear in mind, however, that for the purposes of a report the whole report does not necessarily need to be based on one query.  You can have a parent report based on one query, and within it one or more subreports based on other queries.  A simple example of this would be a report of orders by customers in which you wish to return the details of each order by each customer.  For this a parent report can be based on a query which INNER JOINS Customers and Orders tables on CustomerID.  Within the detail section of this report can be embedded a subreport based on a query which INNER JOINs OrderDetails and Products tables on PoductID.  The subreport would be linked to the parent report on OrderID, so for each order returned in the parent report the subreport would return the set of order lines for the order.

An advantage of a subreport like this is that it's layout can be designed completely independently of the parent report's, and it can be positioned anywhere within the section. The main order data could be positioned to the left of the detail section and the subreport to the right, whereas with a single report based on one query which joins all the tables the order lines would be below the main order data, which would be shown in a group header in the report.  You'll find an example of a subreport used in this way as Payments.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

You can similarly use one or more subforms in a parent form.  For entering orders, for instance, an orders form with an order details subform would be the usual interface.

Welcome to Microsoft Community. Your interest in Windows 10 Technical Preview is much appreciated.

Since Windows 10 Technical Preview is still in development stage (testing environment), certain features might not work in the usual manner. Hence you may face some issue with Windows 10.We are looking forward to improve it.

I suggest you to follow the below methods and check if it helps.

Method 1: I would suggest you to run system file checker scan:

  2. Type the following command in the command prompt and press enter: Sfc  /scannow

Method 2: For Windows Store I suggest you to reset it as below.

  1. Press “Windows+R”.
