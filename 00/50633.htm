Like WinNT, Win2K, WinXP, and Vista before it, Windows 7's NTFS
file security paradigm doesn't rely on, or allow, the crude and
cumbersome method of password protection for individual applications,
files, or folders. Instead, it uses the superior method of explicitly
assigning file/folder permissions to individual users and/or groups.

As a crude work-around, you can place the file in a compressed
folder, and set a password to uncompress the folder to view/access its
contents. Also, some applications allow the user to password-protect
their specific data files. Read the specific application's Help files
for further information on this capability.

still largely applies to Windows 7:

HOW TO Set, View, Change, or Remove Special Permissions for Files and
Folders

This one is Windows 7 specific, but a bit more technical:

FINALLY! IT'S FIXED!

I wasn't expecting much from #1 or #2 as this isn't a performance issue, and I had already tried #3 without any success (as stated in the original post).

I hadn't thought to create a new use account to test the issue, which is a great idea! I set up a new account, and explorer.exe was fine in the new account. This got me thinking about what the differences are between a new account and an existing one that would affect explorer.exe. The first thing that I thought of was the items on the desktop, so I moved them all to a different location and explorer.exe immediately dropped down to between 0-2%.
