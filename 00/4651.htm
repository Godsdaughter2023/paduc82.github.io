ranmo_msft say:

Hi Chaule,

Please kindly note that user’s UPN rather than user’s SMTP primary address is the key when performing authentication, no matter where the authentication occurs. So, like what your example described, if you only convert domainC.com, it means for users whose UPN suffix is domainC.com (not the users with domainC.com SMTP address), the authentication type is changed to standard. Users with domainA and domainB primary SMTP address will be not impacted. The mail flow for them will not be affected.

Thanks,
Ran

ranmo_msft say:


Hi Chaul,

What you understand about “So it appears that this process will create a temporary password for the online account right? (hence the password file)” is correct. After you convert the namespace to standard, it will create a temporary file containing the passwords used to login Office 365 service. Then you should trigger a full Password Sync to synchronize on-premises passwords to Office 365.

You can also use the following cmdlet Set-MsolDomainAuthentication -Authentication Managed -DomainName Contoso.com to set the domain to standard authentication method. Then trigger a Password Sync from on-premises. The difference is that this cmdlet only sets the domain's authentication type to managed (standard). It does not convert the domain type.

Please kindly note that technically speaking, a seamless end user experience is not available in this situation as you mentioned above since the change of user’s authentication method might not take effect in time. Consequently, we recommend you perform this change at a non-peak time.

Ran
