ken sheridan say: Access is a development environment.  If you want to use a non-standard interface like this rather than a simple bound form  you need to do the necessary development to implement the desired functionality.

Rather than creating and saving a one-size-fits-all UPDATE query to update the relevant row in the operational table with the data held temporarily in the holding table, you will need to build and execute the SQL statement in code.  The code will step through the controls in the form and build the SET clause so that it updates only those columns where the corresponding control in the form IS NOT NULL.

Having built the SQL statement the code will call the Execute method of the database object, with the dbFailOnError option so that you can trap and handle any error which might occur.

scottgem say:

I agree with Ken here that you are making more work for yourself than is necessary. Access was designed to work best with bound forms. 

Allowing users to simply update specific fields means some complex coding. I would take a different approach if you want to continue to use a Temp table.

Step 1: Users selects the record they want to work with by choosing the piece of equipment. 

Step 2: The temp table is emptied and an Append query is run to place the requested record in the Temp table. 

Step 3: A form is opened bound to the Temp table which would then display that record. (Note: this form should not allow Additions)

Step 4: Have a Save button which runs an Update query that updates ALL fields in the record. 

So the user only makes the changes they need to make, but because the Temp record has all the values even the unchanged values are updated.

The call-hangup issue can be easily solved by disabling the KeyLock Application. Menu->Applications->Keylock->Options->Status->Disable.

 

This is the app which requires you to slide the finger to unlock device after pushing the hardware switch.

 

For the device stopping responding other times, ensure that you do not leave too much processes running in the background. Commonly, Log, Contacts, Music etc tend to remain open even after we have stopped using them. They continue occupying precious space leading to slow response times.
