Method 1:
I suggest you to check whether the Windows Installer service is started.
1. Press Windows key R, type Services.msc and press {ENTER}
2. Double-click Windows Installer
3. Set the Startup type of Windows Installer to Manual
4. Click Start to start the service. Note down the error message if any.
5. Click OK.

If the issue persists, I suggest you to try the next step and check.

Method 2: Unregister and re-register the Windows Installer to do that follow the steps provided below.

1.         Press Windows key + R, type MSIEXEC /UNREGISTER, and then click OK. Even if you
            do this correctly, it may look like nothing occurs.
2.         Press Windows key + R, type MSIEXEC /REGSERVER, and then click OK. Even if you

Hi Maverick,

If you're inserting multiple endnotes at the one location, you could insert a hyphen after the first one, then format the indicators for the remainder as hidden text. Similarly, if you're cross-referencing a series, insert just the cross-references to the first & last members of the series with a hyphen between them.

What Paul says is correct: You insert all the note reference marks, then insert an en dash after the first or before the last and then format the intervening ones as Hidden. ken sheridan say:

How to I tell access to store this value in the table?

You don't need to; the value can be computed from the Unit Price and Total Pounds column at any time, either in a computed control in a form or report as you are doing, or as a computed control in a query.  You'd only want to store the computed value if it could differ from the result of the expression, e.g. if a discretionary discount could be applied to it ad hoc.

ken sheridan say:

So is there no way to tell an access form to show a field with a calculated control in the table......

That's correct.  In a well designed database, what you see in the table's datasheet is really irrelevant.  It should certainly never be exposed to the user, who should only interface with the data by means of forms and reports.  A form can of course be in
It would be possible to assign a computed value to a column (field) in a table, but usually that would be bad design.  In the language of the database model it is a matter of 'functional dependency'.  All non-key columns in a table should be directly dependant solely on the primary key of the table.  If you had an amount column in the table to which you assigned the result of the expression [Unit Price]*[Total Pounds] the field would be dependent on those two columns, not directly on the key, so it would be 'transitively dependant' on the key.  The table would then be open to the risk of update anomalies which could result in a discrepancy between the result of the expression and the value in the Amount column.  The table would not be normalized to Third Normal Form (3NF).  Normalization is a very important mechanism in the database relational model, and you should become familiar with it, at least in its lower normal forms.  You'll find a simple explanation of it in Normalization.zip in my public
https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

You might also like to take a look at DatabaseBasics.zip in the same OneDrive folder.  This does pretty much what its name implies.

As I said earlier, the only time you would assign a computed value to another column in the table would be if it were legitimate for the value in the target column to be changed by the user from the computed value, i.e. if a discrepancy were valid, in which case the target column would be determined solely by the key, so the table would be in 3NF.  In such a case the value would be assigned to the target table by code in the AfterUpdate event procedures of both the Unit Price and Total Pounds controls:

    Me.Amount =[Unit Price]*[Total Pounds]

In this situation the Amount control would be bound to an Amount column in the table.

The current version of Access does actually include the facility for what are called 'calculated fields' in the table itself.  These act much like a computed column in a query, but are part of the table design itself.  Personally, I don't see what useful purpose this serves.
