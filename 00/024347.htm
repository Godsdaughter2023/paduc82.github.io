ken sheridan say: You can do that very simply with a single query and a limited number of reports.  In the query test for each parameter either matching the value entered, or being Null.  So let's say you want parameters for the school and school year, star by creating an unbound dialogue form, frmFieldTripsDlg say with two combo boxes on it:

1.  cboSchool, with a RowSource property of:

SELECT DISTINCT [School Name]
FROM [Field Trip Delivery Table]
ORDER BY [School Name];

2.  cboSchoolYear, with a RowSource property of:

SELECT DISTINCT [School Year]
FROM [Field Trip Delivery Table]
ORDER BY [School Year];

Add a button to the form to open the report, or maybe two buttons, one to open the report in print preview, the other to print it.

The query for the report would be:

SELECT *
FROM [Field Trip Delivery Table]
WHERE ([School Name] = Forms!frmFieldTripsDlg!cboSchool
OR Forms!frmFieldTripsDlg!cboSchool IS NULL)
AND ([School Name] = Forms!frmFieldTripsDlg!cboSchoolYear
OR Forms!frmFieldTripsDlg!cboSchoolYear IS NULL);

If you select a school in the form but leave the school year combo box blank you'll get all records for that school in all years.  If you select a school and a year, you'll get all records for that school in the selected year.  If you select a year only, you'll get all records for all schools in the selected year.  If you leave both blank you'll get all records in the table.

You can group the report to govern how the data is presented, e.g. you might group by school year, so the schools for one year are grouped together, then for the next year and so on.  Or you might group by school, so for each school all the years are grouped together, then the same for the next school and so on.  You could of course have two separate reports, each grouped differently, using the same query and select which to open in the form, which is easily done.  You can include summary data in the group footer in either case, and you can opt to show only the summary data if you wish by having a check box on the dialogue form in which you'd select only to see the summary data.  A little bit of code (one line only in fact) in the report's module is then used to suppress the details if the check box is True (checked).  But I'd concentrate on getting the form and query set up first; you can then think about what report layouts you want.

I think you'll see from the query above that it is very easy to add further parameters by adding more controls to the report and then additional parameters in the query, e.g. to include a parameter for Grade:

SELECT *
FROM [Field Trip Delivery Table]
WHERE ([School Name] = Forms!frmFieldTripsDlg!cboSchool
OR Forms!frmFieldTripsDlg!cboSchool IS NULL)
AND ([School Name] = Forms!frmFieldTripsDlg!cboSchoolYear
OR Forms!frmFieldTripsDlg!cboSchoolYear IS NULL)
AND ([Grade] = Forms!frmFieldTripsDlg!cboGrade
OR Forms!frmFieldTripsDlg!cboGrade IS NULL);

Just be sure that each OR operation is enclosed in parentheses to force it to evaluate independently of the AND operations.  But, be sure to create and save the query in SQL view.  Do not switch to design view before saving it.  If you do Access will move things around and it will be very difficult to see the underlying logic and make any amendments.

Once you have the form, query and report set up all you have to do is open the form, select however many or few parameters you wish and click the button to open the report.

As written above the query allows you to select only one value per parameter.  It is possible to use multi-select list boxes rather than combo boxes to enable multiple values of each parameter to be selected.  This requires more work, though, as you need to write VBA code to loop through the ItemsSelected collections of the list boxes.  A multi-select list box cannot be referenced directly as a parameter, however, though there are ways of simulating this using some code available online from Microsoft, but I find it easier not to use a query with parameters but to build a string expression in the Click event procedure of the button on the form to filter the report by means of the WhereCondition argument of the OpenReport method.  I can guide you through this if you want to go down that road.

Finally, you really should think about decomposing your table into a number of related tables.  At present you have a lot of redundancy, which is not just inefficient, but more importantly can allow update anomalies, e.g. the same school name could be entered slightly differently in two or more rows.  As far as the database is concerned these are then separate schools and your report will not give correct results.  I have personal experience of this as in one database I found three versions of my own name as the author of technical papers in my own field of work.  Two were correct, but excluding my middle initial in one case, the other had my middle initial as V instead of the correct W, clearly a simple typo.  Anyone looking for papers by me would have had a problem!  Decomposition into related tables where I appeared only once by name in an Authors table would have solved the problem.
