    -Alan

    don cuthbert say:

    One thing to note, which you may already know, is that files in a WIM file can be added with the option not to duplicate files. This is how there can be a multi-Edition Windows 10 ISO, with many different Editions included in an ESD file, without it multiplying in size. Boot.wim contains 2 indexes, but if many of the files are replicated, it does not mean it doubles in size. That means that exporting to ESD "recovery" compression is the best way to reduce the size of a Recovery partition. Also, capturing an image is not as cleaned up as Exporting that captured WIM to another WIM or to a more compressed ESD file.

    Upgrading Windows creates copies of the winre.wim that are used, when required, if an upgrade fails and is rolled back. Also, "Go back to the previous version" is an option that is preserved for 30 days, which is why the extra Recovery partition is made. It provides the means of restoring the original Recovery partition.

    Relevant to the original post:  If you apply (using PowerShell or DISM) all of the files from the main Index of a WIM file to a single partition and create a BCD to boot it, the end result will have the Recovery winre.wim in a hidden folder at C:\Recovery, instead of in a separate partition. This does not prevent a subsequent Windows version upgrade from creating a backup partition for Windows RE at the end of the Boot partition. But the purpose of that added partition (rollback to the last version) is done in 30 days.

    In many cases these days, 500MB is not a huge amount of disk space to dedicate for this purpose. With large drivers this size may grow (and not be fully compressed) so there may be some maintenance that can be done to reduce the disk space used, but not by the average user.

    anthony.b say:

    I think I would recommend Proposal 1 over Proposal 2. However, I think there are pros and cons to using a "split" database arrangement, as opposed to a single Access database, that contains everything (tables, queries, forms, reports, etc). What I have seen, within my work-place, is that sometimes when there is a split database arrangement, is that some of the following things happen:

    [1] There are several different versions of the database front-end, on various users' hard-drives, and each front-end gets changed slightly, putting each front end in a different state.

    [2] Someone puts the database front end out on the shared network, and the users either link to it there on the network, or navigate to the network to open the database front-end. Meanwhile other users use the front-end on their computer's hard-drive.

    [3] Users sometimes get confused between what the difference is between the database front-end and back-end.  For example, the user says my database has a problem, and when I ask is the problem in the front end or the back end, the user does now know what that means.

    [4] There sometimes are different versions of the database backend (with the tables) being used. This results sometimes in a user typing in data all day, and it not going into the same tables that everyone else is using.

    [5] Sometimes someone turns the front-end into an .accde or .mde file, and the original .accdb or .mdb file for the front-end is lost.

    [6] If the back-end database is moved to a different location, then the links between the front-end to the back-end might need to be re-done.

    I think there are sometimes advantages when an Access database is split, if the back-end and front-end are managed properly.  However, sometimes I think an un-spit Access database will sometimes have the same (or almost the same) performance without the risks of #1-6 above.  In general, I think that keeping a database as simple as possible (in this case, un-split is simpler) sometimes makes maintenance easier, especially if the developer who designed the database is no longer available to maintain the database.

    I'm going to disagree with you AnthonyB about a single database file being beneficial when used with multiple users.  There are too many BAD things that can occur that outweigh anything else. 

     

    Corruption is a much greater risk and the likelihood of losing your entire database and having to restore from a backup and entering in data from the corruption point could be costly and time-consuiming.  It is much better to manage your frontends better (which is NOT a problem if you do it correctly) than to go down a path of potential destruction.

     

    The best practice is to ensure that EACH user has their OWN copy of the frontend.  Not share it from a network drive.  And to manage it, you have several choices and here are a couple of the options:

     

    1. Each user is initially given a frontend.  Then you use an Auto Updater (of which I have a free tool on my website to enable auto updating) to always have the latest database frontend for them.

     

    2. You use a batch file as a shortcut and copy down the latest frontend from a shared drive (but not advertised to them as to where it is).  The drawback here is they might try opening from the last files opened list which then they would have a static version and wouldn't necessarily get the latest.

     

    So, the first is better because no matter which copy they open, the code inside it always ensures that they have the latest.  For example, the way the Auto Updater from my website works is that, once it has been enabled, when the user starts the database frontend, it checks to see if their version matches the version that that backend database says should be being used.  If not, it gives them a message that their frontend is out of date and then it will close the database, delete the old frontend, copy the new frontend down from the master location, and then reopen for them.

    http://www.btabdevelopment.com/ts/freetools
