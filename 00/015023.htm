steve rindsberg say:
The interop is the .NET equivalent of a type library file.  It tells your application about the properties and methods PowerPoint makes available.  Other than that, it does nothing.  It's a description of what PowerPoint can do IF PowerPoint is there.  

If PowerPoint isn't installed, you can't ask it to do anything.

If you want to automate PowerPoint, you must install it.

john mcghie say:

Hi Jo:

OK, Full Screen works fine in Word 2008, I just tried it.  Of course, we have to understand what "fine" means in this context -- The toolbars are hidden, the rulers are hidden.  The top bar remains, the ribbon remains.

What I need to find out from you is what you mean by "doesn't work" (exactly).  Everyone has their own ideas of what they expect software to do: but before I can begin to work on your problem, I need to know what you are seeing. I also need to know what you mean by "I have tried everything", because again, that's different for every person, and again, I need to know what you have done before I can suggest stuff that you haven't.

What Phillip is saying is that there are "limits" to how well Word 2008 can interact with OS X.  There are two "frameworks" that developers can use to connect their application with OS X (or they could write native code). A "framework" can be thought of as a bunch of plugs and sockets that enable an application to "plug in" to OS X.

Developers such as Microsoft who had applications on sale way back when... use the Carbon framework, which is intended to connect "legacy"applications such as Office 2008, which still contain code developed for OS 9 and earlier.  More modern applications, such as Office 2011, use the Cocoa framework, which is designed to connect 64-bit applications to OS X.  The very latest applications can be written in "native" Mac OS X code, so they do not need any plugs or sockets because they are designed as simple "extensions" to the OS X code.

Now, any fool with a compiler can make "software".  Making "money" is vastly more difficult.  On the Mac, a lot of what you can do is constrained by "how many can you sell"?  On the PC, that's a no-brainer.  With Office, you are going to sell a thousand million copies, so you can go ahead and chocolate coat it if you like, and put cherries on top!  You will easily be able to pay for the cost of your work.  On the Mac, it ain't so...  Last year, Apple sold 4.1 million Macs.  Only about half will have Office installed.  Mac customers are legendary for keeping old versions of software (e.g. Office 2008...).  And the anti-theft mechanisms on Apple Mac are trivial, so it's tough to get "paid" for your software.  So the story goes, "in places like China, you will only ever "sell" one copy..."

So the last thing you can do on the Mac is to say "Let's write a new version, just for the Mac, using Native Code!"  Microsoft Office literally cost "billions" to develop.  A few thousand developers working away since 1980...  You would NEVER get your money back, let alone turn a profit.

You HAVE to use a framework that can plug existing code into the latest OS.  Now, in Apple's case, they have dropped support for development in Carbon.  It still works, sort of, but it has no connections for the latest functions such as "Full Screen".  And it can't be used for 64-bit connections.

So legacy applications such as Office 2008 will operate increasingly imperfectly as OS X updates.  And there's no cure for that.  Microsoft is not going to send itself broke issuing updates for a suite of software that went out of support 5 years ago.

What you CAN do is wait for the next version of Mac Office.  We don't yet know when that will be, but late this year or early next year is my guess.  We do know that it will be faster, more powerful, more stable, and contain a much higher percentage of native code.

It will go better.  And it will give you a proper Full Screen display.  Probably won't spell any better (sorry...).

Hope this helps

suzanne s. barnhill say:

The article at http://homepage.swissonline.ch/cindymeister/LangFmt.htm is obviously very old, but not much has changed wrt Office 2003: it still works best when the application default language is the same as the Windows default. And, needless to say, you need to disable "Detect language automatically." Also, be aware that, if you paste text from outside the document (especially from the Web), it is likely to be formatted with US English. Language is a font property that can be applied to selected text just like bold or italics, and if it once infects your document (via pasted text), it will propagate unless you nip it in the bud.

charles kenyon say:

Suzanne knows more about this than anyone I know of.

Mastering the Spelling Checker by Suzanne Barnhill, MVP

The proofing language is a character-level setting. It is not document- or application-level. It is often embedded in styles and it can be imported when you paste text.

It can seem intractable and feel like a virus. Automatically detect language can throw real curve balls into the mix because Word is not that good at it.

If it continues to be a problem, I do have some macros, but they may be overkill and you may not want to use them.

Here is one for styles:

'   Written 21 March 2018
'   Charles Kenyon
'   Intended to set all styles to English UK, proofing, not automatitically update
'
    Dim aStyle As Style
    On Error Resume Next
    For Each aStyle In ActiveDocument.Styles
'        aStyle.AutomaticallyUpdate = False ' probably do not want this for TOC styles - future work
        aStyle.LanguageID = wdEnglishUK
        aStyle.NoProofing = False
    Next 'aStyle
'    ActiveDocument.UpdateStylesOnOpen = False ' For information on using this line, see:
'       http://www.shaunakelly.com/word/sharing/willmyformatchange.html
    On Error GoTo 0
End Sub
