There are likely 2 things to check + insure.

#1
The table has a PK. As a general rule linked tables to a server based system require the pk.

Note that for most (if not all) server based systems, the PK id is not available until AFTER the record is saved – not at dirty time like when using native Access tables.


#2  Any true/false column has a default set (don’t allow nulls for true/false). This would include a “bit” field if being used. (check default for number columns also - might want to use 0 in place of null).

Access also uses (wants) a “row version” number which is called a timestamp column. Access uses this to determine if a record is been updated when working with SQL server. However, Oracle typical does not have such a rowversion column and I not familiar with oracle.

So check above #1, #2 issues.

mattm570 say:

Nevermind.... I figured it out!

 

I had tried doing this as well as the Pattern in my original post:  "JW?*^p"

 

But that wouldn't work with the "Use Wildcards" checkbox checked. So I came across a page containing all the special characters available and I found that the other paragraph break character was "^13" so I tried this pattern below, and now it selects the entire line...!

 

 

Here is MY pattern to select the entire line:

                       "JWP?*^13"

 

So that says, find "JW" followed by any character, then any character 0 or more times till you hit a "Paragraph Break" (*i.e. ^p and ^13 are same but ^13 can be used with Wildcards) aka a "Hard Return" or hitting the enter key to create a newline.

Ignore blank means allow the cell to be cleared ie allow blank values in the cell, it does not mean ignore blank entries in the list.

jim gordon mvp say:

Unfortunately, Rena doesn't know the difference between Mac OS Safe Mode and Excel Safe Mode, nor does she know how AutoRecovery works in Excel.

Holding the Shift Key down when you restart your Mac does put your Mac into Safe Mode, which is a way to troubleshoot problems external to Excel. This will do nothing to help you with the problem you are having.

However, right up front you stated that holding the Shift key down when opening Excel avoids the error. Holding the Shift key down when opening Excel puts Excel itself (not Mac OS) into Safe Mode. When Excel itself is in Safe Mode, it skips checking for autorecovr workbooks to open.

The way autorecover works, if Excel crashes, the next time you open Excel normally, it checks for files in the AutoRecover location and tries to open them. The idea is that you will pick up roughly where you left off just before the crash, and then you will use File > Save As to rescue what you can of the crashed workbook. The flaw in the system is that if the workbook attempted to be recovered can't be recovered, you wind up in an endless loop, which you can break by using Shift to bypass the AutoRecover step.

Your damaged workbook is in a hidden directory called Library. Here's how to display the Library folder so you can navigate to the damaged file so you can delete it.

How to Show to ~/Library Folder in MacOS Sierra - OS X Daily

Navigate to User > Library > Group Containers

I think you'll find the damaged, unrecoverable file in one of the folders inside these:

I don't have any corrupt files, so I don't know exactly which sub-folder will have your bad file.
