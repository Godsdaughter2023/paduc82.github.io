Excel 2016 bug - cannot switch QueryTable to take parameters from - Microsoft Community

slepmog say:

Excel 2016 bug - cannot switch QueryTable to take parameters from sheet after refreshing it with a constant value

There appears to be a bug in Excel 2016.

If you have a QueryTable on a sheet, that query has a parameter, and at some point you refresh that query with a non-sheet value for the parameter, then you cannot switch the parameter to take its value from the sheet - Excel will crash. In order to make it work, you need to switch the QueryTable to take parameters from sheet, then save, then quit Excel completely, then reopen the file, and only then your query will work.

I believe it started after the latest Office update, and it is very annoying because it happens every time you create another query table. By default query tables are created with parameters prompted from the user each time, and they are refreshed when created, and the first thing you do after creating a querytable is repointing it to take its parameters from sheet cells, and when you do that Excel crashes and you lose your work and the table you just created.

The culprit appears to be the fact that the query table is refreshed with a non-sheet value as a parameter. As soon as that happens, it cannot refresh with a sheet value, it requires restarting Excel for that. If in the current working session you have not refreshed a query table with a non-sheet parameter, you can keep refreshing it with a sheet parameter, but as soon as you have, you need to restart Excel if you want to revert to taking the value from a sheet.

Steps to reproduce:

  • Start with a fresh instance of Excel and a blank workbook.
  • Create a query table (Data - From other sources - From Microsoft Query). Connect to a database using MS Query (in our case it's SQL Server 2016 via native client if that matters) and create a query against the database. Make sure it has a parameter.
  • Return the query to Excel. Answer the dialog about where to place it. Excel will also ask what value to use as a parameter, and whether to store that value for future reference. Enter any value as an answer.
  • You now have a query table on the sheet. Right click and go to Table - Parameters. Switch the parameter from "Prompt for value using the following string" or "Use the following value" to "Get the value from the following cell" and select a cell.
  • Refresh the query. Observe that Excel crashes.

Alternative steps to reproduce:

  • Start with a blank workbook and create a parametrized database query from MS Query like described above.
  • Switch it to take the parameter value from a sheet cell, but do not refresh the query.
  • Save the file and exit Excel. Open the file again, refresh the query, observe it is working fine and takes the value from the cell.
  • Now go to Table - Parameters and switch the parameter mode to a non-sheet option ("Prompt for value using the following string" or "Use the following value"). Then refresh the query providing a fixed value.
  • Now go to Table - Parameters and switch the parameter back to taking the value from a sheet cell. Click OK and Refresh the query. Observe that Excel crashes.

Work around:

Each time you create a query table and switch it to take parameters from a sheet cell, save the file, close and open it again, and do not touch the parameter mode after that, and if you do, save and restart again.

Do consider the fact that the setting of 3% probably means something like 4 minutes usage of the computer which ought to be enough time to save your files & then hibernate as long as the battery is healthy and does not just crash from 3% to 0% in the blink of an eye.  If you do want to change the level to below 3% then be prepared for power crashes as your battery ages. 

The built-in Powercfg utility is used to change the settings independently of the Power options UI.  If you want to go down this route then get its help file from a command window by running this [using a path that exists on your computer]

powercfg /? >D:\Desktop\PowercfgHelp.txt

You can also get it to report the settings for your current Power plan using

powercfg -qh >D:\Desktop\ThisPowerPlanProperties.txt

[Note that the undocumented switch -qh means report all properties including hidden ones.]

I find it best to copy the text results into Excel to review them.

You might find the results overwhelmingly complicated but it is actually just a large number of individually simple sections all strung together.

If you are going to use powercfg to change any settings then you'll find that the commands are very long but you can keep them slightly simpler by being in the power plan that you are trying to alter.  Even then they are long.  Here is what I think you would need to put in for a Critical battery level of 3% in your current power plan [but I appreciate that you will check this for yourself using the powercfg help file]:-

POWERCFG -SETDCVALUEINDEX e73a048d-bf27-4f12-9731-8b2076e8891f 9a66d8d7-4ff7-4ef9-b5a2-5a326ca2a469 3

jackie l say:

My response is based on not knowing your table structure or if it is correct using the normalization rules.  Hopefully, your Employee table has one record per employee and with an ID field as key.  Also, the audit table should have that same employee ID field in it..

Create a query which contains the two tables.  Create an outer join from the Employee table to the audit table on the employee ID field.  Make sure that the join is outer (all the records from the Employee table and only the records from the audit table where the joined fields are equal). 

Bring in the employee and add an expression field similar to the following:

AuditVar: IIF(IsNull([AuditDate]),"Not Audited","Audited").  If you put criteria under that field as "Not Audited", add Totals and choose Group by for the employee and Where for the new field, you SQL should look similar to this:

SELECT tblEmployee.EmployeID
FROM tblEmployee LEFT JOIN tblAudit ON tblEmployee.EmployeID = tblAudit.EmployeeID
WHERE (((IIf(IsNull([AuditDate]),"Not Audited","Audited"))="Not Audited"))
GROUP BY tblEmployee.EmployeID;

Your query results will only be the employees who have never been audited.  If you need to be able to remove the "audit" then it might be better to create a status field in the employee table which would be populated when the audit is entered and can be changed if needed without affected the audit records.  That is an entirely different scenario and you will have to let us know if you want to pursue that.

Hope this helps.

corey-g say:

create a new query, add both the audit table and the table with the names, and then join them by the persons ID.  Then right-click the line created between them and choose the option for all records from the person table, and only matching records from the audit table (An Outer Join - Left or Right is dependent on how the tables are positioned in the query window).

Then pick the fields you need to see, and include the person ID from the audit table, and set the criteria for the field to 'Is Null' - this will give you all people who haven't been audited yet.  Then you can change the criteria to Is Not Null to get the list of everyone who has been audited.

If you wanted to just have an indicator in the combobox (as suggested by scottgem), I think the best way would be to use an iif statement on the persons ID (fromthe audit table).

IIF(IsNull([Person_ID],"","*")

This will check to see if the person ID is null (not audited) - if it is null (AKA Not audited), leave it blank, otherwise (AKA audited) add an asterisk.  remember to adjust the combobox for the number of fields returned and have enough width to show the new indicator.

HTH's

Corey

thg250 say:

One way to get macron and breve is to go to system preferences/keyboard/input sources and use the plus and add buttons to activate the ABC Extended keyboard layout in the English category.  Then select that in the "flag" menu in the top right of the screen, and use these shortcuts.

macron =  option A, then the base letter:  ō

breve = option B, then the base letter:  ŏ

A full list is here.

Another way is to use the Character Viewer (Edit > Emoji and Symbols) and search for the combining macron and combining breve characters and then double click on them to insert after the base letter.

Macrons are also often available from the popup menu that appears when you hold down a key.  But I think breves are not.

If by chance you need these for Pinyin, it's not a breve but a caron (option V) which is used for that.
