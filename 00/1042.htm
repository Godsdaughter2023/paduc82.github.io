Move windows defender definitions to another drive letter - Microsoft Community

john_497 say:

Move windows defender definitions to another drive letter

Hi, Is it possible to move the definitions of Windows Defender to another drive? I have a PC that resets itself every time it is switched off or restarted with Deep Freeze.

I have set up a partition that will no be frozen and would like the definitions to be stored here so that the PC is up to date against the latest virus.

Kind Regards 

rob koch say:

The Microsoft security applications have always been designed to install and operate on the Windows system partition, so no, I don't believe there's a supported way to move these to another volume.

The Windows operating system has changed over the years from a relatively static file based system into a dynamically evolving set of modules, so the ability to save the state of the operating system and return to that point has become more and more problematic over time.  Though 3rd-party tools still attempt to offer this type of management, Microsoft itself is dropping features like System Restore in Windows 10, since it really doesn't function well in this modern environment.

The problem is that portions of the operating system including logs and database files are now sensitive to both date/time, as well as the integrity of these databases, so grabbing the system state at a given moment often doesn't recover in a stable manner when the system is returned to an earlier time.

For this reason I had recommended that clients move away from these older static methods years ago, since as you've no doubt discovered, applications like the Microsoft antimalware based security don't typically recover well after such a snapshot is restored.

This problem has only gotten worse with each successive Windows version, so though Win 8.1 hadn't yet dropped System Restore, it already had most of the design criteria in place that interacted badly with such tools.

Good luck, but I recommend that you start thinking dynamically when choosing support methods and looking for tools that fit this direction.

Rob

scott mcdaniel say:

That's fairly common with multi-table recordsets. You must be sure to include ALL relevant fields in the recordset from both tables - i.e. all the join fields, primary keys, etc.

However, in most cases you're much better off basing your forms on a single table, and using combos, listboxes, or subforms to show child data. You won't run into the 3226 error like that.
