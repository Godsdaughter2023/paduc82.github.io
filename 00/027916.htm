john sr wilson say:
I don't think so. You could add the animation 9 times with code though. Once you had one text animated the animation painter will paint all 9 animations to any other text.


Sub addMultiAnims()
Dim oeff As Effect
Dim osld As Slide
Dim oshp As Shape
Dim i As Integer
Set osld = ActiveWindow.View.Slide
Set oshp = ActiveWindow.Selection.ShapeRange(1) ' make sure there is a selected shape
For i = 1 To 9
Set oeff = osld.TimeLine.MainSequence.AddEffect(oshp, msoAnimEffectFade, , msoAnimTriggerOnPageClick)
oeff.Timing.TriggerDelayTime = 2
oeff.Timing.Duration = 5
Next i
End Sub

Here is the VFP code.  loTable already exists by this point.  ASC() is a function to return the character's numeric ascii value.  SUBSTR() is the substring function to pull a character out of the string.

   *** Removing Blank Table Rows ***

   * would like to remove blank table rows if they fall on the first line of a page; this is done by iterating through the
   * table again checking a range property that tells us which page we are on and if the page has changed from the last row
   * *and* this is a blank row, we delete the row; technique suggested by Doug Robbins, MVP, on Microsoft Office Word Forums
   * site on Aug 20/10 titled "Suppress blank row in table that falls at the top of a page" {Sept 21/10}

   * set lnPrevRowPageNo to 1 as there is need to check rows on first page (check once goes to page 2)
   STORE 1 TO lnPrevRowPageNo

   * Note: first started to do this via FOR lnAtRow = 1 TO loTable.Rows.Count but .Count changes as we delete blank rows

   FOR EACH loRow IN loTable.Rows

      * Information() method is property of the range so dig out range (as needed more than once)

      STORE loRow.Range TO loRange
      STORE loRange.Information(wdActiveEndPageNumber) TO lnCurrRowPageNo

      * if the number has changed, pull out the text from the row and if it is only CHR(13) or CHR(7) (paragraph or tabs I
      * think), then the row is actually empty

      IF lnCurrRowPageNo # lnPrevRowPageNo

         STORE loRange.Text TO lcText

         * loop through the chars checking each until a "real" character found; set a flag to indicate if all empty or not;
         * in testing, if the row is empty, it only contains CHR(13) and CHR(7)'s

         STORE .T. TO llAllEmpty

         FOR lnAtChar = 1 TO LEN(lcText)

            STORE ASC(SUBSTR(lcText,lnAtChar,1)) TO lnAscNum

            * check it
            IF lnAscNum # 13 AND lnAscNum # 7
               STORE .F. TO llAllEmpty
               EXIT
            ENDIF

         ENDFOR

         * if still all empty, delete the current row; if not empty, reset lnPrevRowPageNo as we might have gotten here after just
         * deleting a row and if we don't reset it now, every row from here on in will get checked when it is really only if we
         * are on the first row of the page; note that I first set lnPrevRowPageNo after each row but then if we had a case where
         * there was two blank rows in a row, the first one would get deleted but the next one was not checked (because the var
         * lnPrevRowPageNo would have been reset); so now, we reset only after we "hit" a non-blank row; note that I tested this
         * and if I forced the printout to have to blank rows right at the top of the page, they do indeed both get deleted now
         * with the change to the code {Sept 22/10}

         IF llAllEmpty
            loRow.Delete()
         ELSE
            STORE lnCurrRowPageNo TO lnPrevRowPageNo
         ENDIF

      ELSE

         * reset lnPrevRowPageNo to the new value
         STORE lnCurrRowPageNo TO lnPrevRowPageNo

      ENDIF  && for IF lnCurrRowPageNo # lnPrevRowPageNo

   ENDFOR  && for FOR EACH loRows IN loTable.Rows

scott mcdaniel say:

Access locks records based on "pages" of data, which could potentially lock more than one record.

Do you have a BOUND form that is displaying data to the user? If so, then you should use bound form methods to update your data, not recordset methods.

gary_unknown say: Hi Scott: No, this routine is used in the background to modify / delete records that will later be modified and ADDED back in.

scott mcdaniel say:

There's got to be something that is causing the lock. Unless the two machines try to do this at EXACTLY the same instant - which would be quite hard to do - then something in your application is locking those record pages.

Do you have anything that is bound - a form, report, etc. Note that in Access "bound" means you have Recordsource set for the Form or Report, and a ControlSource set for one or more controls.
