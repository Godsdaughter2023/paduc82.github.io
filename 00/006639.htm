Control's BeforeUpdate cancellation plus Required field allows - Microsoft Community

phil lucido say:

Control's BeforeUpdate cancellation plus Required field allows invalid values

There seems to be a long-standing bug in Access where setting Required to True on a field's property in its table, then using a control-level BeforeUpdate which gets cancelled on invalid data, can result in that invalidated data getting stored in a record, breaking data integrity in the database.  I'm hitting it with Access 2010, and searches see references back a decade or so.  I'm curious if this is still in Access 2013.

I've put a trivial repro database out at http://sdrv.ms/18eQj4f to demonstrate.  Open frmNumberTest to see what's wrong.  The table and form try various forms of validation to only allow numbers from 10 to 19, but you can see that case 4 and 7 allowed an invalid value into the database.  Hit the next record button to move to a new record, then try entering an invalid number in each control.  Everywhere except controls 4, 6, and 7, you'll get a popup with either the Validation Text from the table, or a BeforeUpdate msgbox, then you can't move out of the control.

But for controls 4, 6, and 7, the results are different.  Click on control "4) Required & BeforeUpdate", then type 1, and Tab.  The control's BeforeUpdate shows a msgbox about the invalid data.  Press OK, and you'll see a new msgbox with "The value violates the validation rule for the field or record."  Press OK, then Esc to try to undo the invalid input, and you'll see the invalid control value of 1 selected.  Press Tab again, and you'll move to the next control with the invalid 1 in the control.  If you fill the rest of the controls with valid values, you can save the record successfully, allowing invalid data into the table.

I've put a Debug.Print in Form_Error which shows that second msgbox is from error 2116.  This seems to happen whenever a control-based BeforeUpdate gets cancelled when the field is either marked Required or, equivalently, the Validation Rule is "Is Not Null".

Case 6 sets Required to True and Validation Rule to "Between 10 And 19", then uses BeforeUpdate to also validate.  Here, you can again get the invalid data in the control with the same keystrokes as in case 4.  But when you try to save the record, the Validation Rule kicks in and the record isn't saved.

In Case 7, I tried to see if the problem was the combination of BeforeUpdate cancellation and Required = True, by setting Required = False and the Validation Rule to "Is Not Null".  But this has the same issues as case 4.  I also changed the BeforeUpdate a bit - in addition to Cancel = True to cancel the update, I also do "Me.txtField7.Undo" to see if that improved things.  Instead, it makes it worse - the Undo selects the invalid text, so just hitting Tab after the msgboxes are dismissed causes the invalid data to be stored in the control and later saved in the record.

Best I can figure, it's just not safe to mix Required=True in the table definition with a control-based BeforeUpdate validation.  That seems a pretty bad bug to have around for years.  Any chance this was fixed in Access 2013?  I want to use Required=True both because it seemed like a good idea for data integrity, and because I key off that in my form-level BeforeUpdates to check for missing fields.  My workaround is to use a Validation Rule instead of control-level BeforeUpdate where possible, and where it isn't, set Required=False and find another way to check for a missing field (I'm setting the control's Tag to "[Required]" instead).

    simon1111 say:

    I spent the afternoon talking to a microsoft representative who looked at this problem within the document I was having this problem with.  He diagnosed the following problem:

     

    "This happens only with this document and its a document specific issue"

    A problem was caused by the document having a change in "the word spacing options etc."

    This can be caused when retained formatting from an outside source is copied into the existing document whose spacing options are different from those in the original document.

    Therefore the solution was:

    Copy the entire document into a new, freshly opened, blank document, leaving behind ALL the formatting.  DO NOT transfer any of the old formatting to the new document. 

    After the "text only" is copied into the fresh document the cursor resumes its proper functioning and can always be seen as it should be when using the arrow keys.

    Then manually reformat your document the way you want it to be formatted. (Font type, font size, tabs, italics, line spacing, etc...)

    Any new text you copy and paste into your document, make sure you paste it as text only without its outside formatting or you run the risk of recorrupting your document and losing your cursor again.

     

    This is the only way to get your cursor back. 

    For me it was worth it to do, as I am working with the same 300 page document for several hours per day for several months of the year.
