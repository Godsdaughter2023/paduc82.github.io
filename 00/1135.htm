mike h.. say:

Hi,

Try this ARRAY formula and see below on how to enter it. Put the look up value (Product) in F2

=INDEX(C2:C20,MATCH(MAX(IF(A2:A20=F2,B2:B20)),IF(A2:A20=F2,B2:B20),0))

 

This is an array formula which must be entered by pressing CTRL+Shift+Enter
and not just Enter. If you do it correctly then Excel will put curly brackets
around the formula {}. You can't type these yourself. If you edit the formula
you must enter it again with CTRL+Shift+Enter.

ken sheridan say:
Now I keep hearing not to have 2 different tables with same field name, especially linking them


I don't know where you heard that, but it's nonsense.  It is usual, and generally desirable for semantic clarity to have the same name for the primary key column of one table and the corresponding foreign key column in a table referencing the first table.  For example you'd have a CustomerID column as the primary key of a Customers table and a CustomerID foreign key column in an Orders table.

 

However, as regards your problem, your parent form is based on a query which returns rows from three tables, Customers, Enquiry and Item/Service.  Consequently there can be multiple enquiry rows per customer returned and multiple Item/Service rows per enquiry per customer.  I assume that the primary key of Enquiry is JobID, or at least that is what the SQL statement implies (though why not EnquiryID?)

 

You are relating the subreport to the parent report on the JobID, which implies that the items relate to each enquiry.  As multiple instances of each enquiry per customer can be returned in the parent report by virtue of the inclusion of the Item/Service table in the parent report's query the subreport can return multiple instances of each item related to an enquiry.  This will account for the duplication of rows returned.

 

As to a solution, I for one am far from clear as to what is the underlying logical model.  It would help if you could explain just what real world entity type each of the four tables models and how these are related.  It appears from what you have said is that the model is:

 

Customers----<Enquiry----<Item/Service

                      Enquiry----<Items

 

but I suspect this might not accurately reflect the real world relationships.  It's important to understand that a relational database models a part of the real world in terms of its entity types and the relationships between them.  If the model, as expressed by the tables and the relationships between them does not accurately reflect the real world, then the database is not going to function correctly.

 

BTW, I would recommend that you use plural or collective nouns for table names, as you've done for Customers and Items.  This better reflects the fact that a table is a set and makes for greater semantic clarity.  Currently you are mixing plural and singular nouns, having used the latter for Enquiry and Item/Service.  Singular nouns should be used for column names as these represent attributes of the entity type.
