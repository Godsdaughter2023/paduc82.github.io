ken sheridan say: Unless you have a real need to use a 'multi-valued field', i.e. your database is interfacing with SharePoint in the very specific context in which this feature is required, I would strongly advise against the use of this feature.  It conflicts with the basic principles of the database relational model as the table is not normalized to First Normal Form (1NF), whose definition is:

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple contains exactly one value for each attribute.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).

Instead, model the inherent many-to-many relationship type in the conventional way by means of a table which resolves the relationship type into two one-to-many relationship types.  If you have a need to return the values as a comma delimited string you can do so in a form or report.  You'll find an example of the former in StudentCourses.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates several possible interfaces for a binary (2-way) many-to-many relationship type, including presenting the values as a comma separated string expression.  Normally a conventional form/subform would be used, however.

For returning the values as a single string expression in a report a suitable VBA concatenation function can be used.  The Concat demo in my same OneDrive folder illustrates this using data from Northwind.  My file uses a function which calls the highly efficient GetString method of the ADO recordset object.  Allen Brown also provides a similar function using DAO at:

ken sheridan say:
So, if Parent is a MVF then it would have to be Children.Parent.Value or something like that. 

The cboParent combo box is not a multi-valued control.  There was some confusion on this point early in the thread but the OP clarified this by saying:

'I don't want to create a cascading combo from an MVF, I want the reverse'

i.e. they want to select multiple children of a single previously selected parent in the second control, not children of multiple parents.

It's not actually very difficult to correlate two many-to-many relationship types if they are modelled in the conventional way by a table, but it's done by correlating subforms not combo boxes.  The following, for example, is the RecordSource of a course sessions attendances subform in my StudentLog demo.  In this case the correlation is done by referencing the CourseID control of the currently selected row in a registrations subform, and the StudentID control of the parent form

SELECT CourseAttendances.CourseDate, CourseAttendances.CourseID, CourseAttendances.StudentID
FROM CourseAttendances
WHERE CourseAttendances.CourseID=[Forms]![frmStudents]![sfcRegistrations].[Form]![CourseID] AND CourseAttendances.StudentID=[Forms]![frmStudents]![StudentID]
ORDER BY CourseAttendances.CourseDate;

This restricts the subform's recordset appropriately when a row is selected in the registrations subform, so the attendances subform shows only the sessions for the selected course for which the student has registered.  For data entry in the subform the RowSource of a combo box for selecting the session by date/time is similarly correlated with the registrations subform on StudentID and CourseID:

SELECT Format([CourseDate],"dd mmm yyyy hh:nn AM/PM")
FROM CourseSessions INNER JOIN CourseRegistrations
ON CourseSessions.CourseID = CourseRegistrations.CourseID
WHERE CourseRegistrations.StudentID=[Forms]![frmStudents]![sfcRegistrations].[Form]![StudentID] AND CourseRegistrations.CourseID=[Forms]![frmStudents]![sfcRegistrations].[Form]![CourseID]
ORDER BY CourseSessions.CourseDate;

This allows only sessions of the currently selected course for which the student has registered to be selected.

But, as you said earlier, I would not dream of trying to do this by correlating two multi-valued combo boxes.

Well, yes, someone... The property Item.SentOnBehalfOfName returns the display name,

as stated in

https://msdn.microsoft.com/en-us/library/bb206695(v=office.12).aspx

So, it is missing the e-mail address.

https://www.slipstick.com/developer/read-mapi-properties-exposed-outlooks-object-model/

lists the MAPI properties...

I also had the same question... So, with sample code (that works in Outlook 2007) in

https://www.slipstick.com/developer/code-samples/outlooks-internet-headers/

I tried some of the properties, and finally,

PR_SENT_REPRESENTING_EMAIL_ADDRESS

is the one (at least, for me, because I did not try them all).

So, the code final is (the comments were my trials/attempts):

Function GetOriginalSenderAddress(olkMsg As Outlook.MailItem) As String
    ' use code from several URLs:
    'https://www.slipstick.com/developer/code-samples/outlooks-internet-headers/
    'https://www.slipstick.com/developer/read-mapi-properties-exposed-outlooks-object-model/
    Dim strFinalValue As String
    'Const PR_SENT_REPRESENTING_NAME = "http://schemas.microsoft.com/mapi/proptag/0x0042001E"
    'Const PR_SENT_REPRESENTING_ENTRYID = "http://schemas.microsoft.com/mapi/proptag/0x00410102"
    Const PR_SENT_REPRESENTING_EMAIL_ADDRESS = "http://schemas.microsoft.com/mapi/proptag/0x0065001E"
    Dim olkPA As Outlook.PropertyAccessor
    Set olkPA = olkMsg.PropertyAccessor
    'strFinalValue = olkPA.GetProperty(PR_SENT_REPRESENTING_NAME)
    'strFinalValue = olkPA.BinaryToString(olkPA.GetProperty(PR_SENT_REPRESENTING_ENTRYID))
    strFinalValue = olkPA.GetProperty(PR_SENT_REPRESENTING_EMAIL_ADDRESS)
    GetOriginalSenderAddress = strFinalValue
    Set olkPA = Nothing
End Function
