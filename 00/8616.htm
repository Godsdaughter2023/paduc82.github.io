bill manville say:

Forms toolbar controls often behave better than ActiveX ones.

My guess is that the controls were still there but had width of 0 so that even when visible you couldn't see them (!).

You could probably get them back by cycling through the shapes on the sheet and resetting the size of any which are width 0 (or close):

Sub Resurrect()
  Dim S As Shape
  Dim R As Range
  For Each S In ActiveSheet.Shapes
    Set R = S.TopLeftCell
    If S.Width<=1 Then
      S.Width = R.Width
      S.Top = R.Top
      S.Left = R.Left
      S.Height = R.Height
    End If
  Next
End Sub

dylanhunter say: Hey Mthinckley,

    After a good bit more research, etc, looks like you're right - this appears to be a bug in Excel 2010, which I believe was tagged Low Priority, so appears you're mostly on your own for now.  What I did to get beyond this was to create a Reset macro, manually forcing all ActiveX controls to their original positions.  A segment of this macro is below to give you an idea:

    With Sheet4.CheckBox23  '  Period 1
        .Top = 111.75
        .Left = 1037.25
        .Width = 13.5
        .Height = 10.5
    End With
    With Sheet4.CheckBox24  '  Period 2
        .Top = 111.75
        .Left = 1099.5
        .Width = 13.5
        .Height = 10.5
    End With
    With Sheet4.CheckBox25  '  Period 3
        .Top = 111.75
        .Left = 1161
        .Width = 13.5
        .Height = 10.5
    End With

   Biggest issue with doing this is that Excel appears to use relative positioning of ActiveX controls, (i.e. "place the control X pixels from the left of the screen") rather than Absolute, (i.e. "place the control in the middle of column K, regardless of whatever's happening with columns A-J") so hiding, displaying, grouping or resizing any rows or columns to the left or above the intended position will require you to modify the macro to get the controls to keep showing up where you want them.

   The way that I dealt with this latter issue was to have the macro set the spreadsheet to a display that you can count on, (i.e. hiding / unhiding rows/columns) as a first step before the above lines, then having it do all the above With statements, and then finally resetting the display back to either whatever it was before the macro started, or, (if you want to make life easier for yourself  :) ) a predefined display mode.  For this, you'll end up with something like:

    '  Unhide all columns since all .lefts below are relative references.
    Sheet4.Outline.ShowLevels RowLevels:=0, ColumnLevels:=2
    Sheet4.Columns("aa:as").EntireColumn.Hidden = False
   
(1 With block per control, as above)

   '  Close groups back up
    Sheet4.Outline.ShowLevels RowLevels:=0, ColumnLevels:=1

All a bit of a pain, to be sure, but linked to something like Ctrl-R, (and added to any user docs for the model) appears to be the best we can do until the bug is addressed someday.

bill manville say:

It's clear that the best that can be done is a work-around.

It seems that the problem only occurs when the file is saved with some rows/columns hidden so one workaround would be to have a Workbook_BeforeSave procedure that unhides everything.

 

The other workaround is to re-position the controls when they have gone awry.
It they are still in the correct cell but shrunk to height 0 then the approach I suggested in 2011 would work.  Otherwise DylanHunter's approach, to which I would suggest an improvement: set the LinkedCell property of each checkbox to contain the address of the cell in which it should be located.  Then instead of using absolute locations for each box you can use


Sub test()
  Dim O As Object
  Dim R As Range
  For Each O In ActiveSheet.OLEObjects
    If TypeOf O.Object Is MSForms.CheckBox Then
      If O.LinkedCell <> "" Then
        Set R = ActiveSheet.Range(O.LinkedCell)
        O.Top = R.Top
        O.Left = R.Left
        O.Height = R.Height

        O.Width = R.Width
      End If
    End If
  Next
End Sub

 

This could be adapted - e.g. to use part of the name of the Checkbox to tell you where to put it - if you don't want to use linked cells.

I would remove the parameter.  Then open the report using code like this:

 

DoCmd.OpenReport "ReportNameHere", acViewPreview, , "[DateFieldNameHere] Between #" & Forms!frmparameter.txtDate & "# AND #" & Forms!frmparameter.txtEndDate & "#"

boblarson say:

You don't put ANY criteria in the query.  Leave the criteria off of the query. 

And yes, you would create a form to open the report from and the date text boxes would be on them.  You could then build your where clause by using something similar to this:

 

    Dim stDocName As String
    Dim stWhere As String
        

        If IsNull(Me.txtFrom) And Me.txtFrom = "" Then
            If Not IsNull(Me.txtTo) And Me.txtTo <> "" Then
                stWhere = stWhere & "[TravelDate]  <=" & Me.txtTo & "#"
                End If
        Else
            If IsNull(Me.txtTo) And Me.txtTo = "" Then
                If Not IsNull(Me.txtFrom) And Me.txtFrom <> "" Then
                    stWhere = stWhere & "[TravelDate]>=" & Me.txtFrom
                    End If
            Else
                If (Not IsNull(Me.txtFrom) And Me.txtFrom <> "") And (Not IsNull(Me.txtTo) Or Me.txtTo <> "") Then
                stWhere = stWhere & "[TravelDate] Between #" & Me.txtFrom & "# And #" & Me.txtTo & "#"
                    End If
            End If
        End If

    stDocName = "rptTipsLog"

    DoCmd.OpenReport stDocName, acPreview, , stWhere

 

 

 

I used the code from my sample but you would substitute your date field names, your control names, etc.
