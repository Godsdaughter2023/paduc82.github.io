dana.d say:

If you wish, you can factor out the negative values...

joeu2004 say: gderooij wrote:

I tried to use a substraction formula like this =SUM(I4-G7-G4-E3-E6-E15-E19), the result should be 0, but instead of that I get this: -8,52651E-14.


What does this mean and how can I resolve this issue?


First, the use of the SUM function in this context is superfluous.  You should simply write:


=I4-G7-G4-E3-E6-E15-E19


And by coincidence, that might remedy the problem.  But only by coincidence.


In general, you should explicitly round expressions involving non-integers to the precision that you require.  For example, for most currency calculations, you might write:


=ROUND(I4-G7-G4-E3-E6-E15-E19; 2)

For other purposes, change 2 to the required number of decimal places.

Numbers in the form -8,52651E-14 are infinitesimal values presented in Scientific format.  That number is about 8 times 10 to the power of -14; that is, 8*10^(-14).

Such infinitesimal differences are common because most non-integers cannot be represented exactly in binary floating-point, the standard internal representation that Excel uses.  The binary approximation of decimal non-integers causes arithmetic anomalies that are inconsistent with mathematical expectations.

For example, even if I4 is =G7+G4+E3+E6+E15+E19, the result of =I4-G7-G4-E3-E6-E15-E19 might not be exactly zero, depending on the values in each cell, whereas =I4-(G7+G4+E3+E6+E15+E19) probably is exactly zero.  Of course, they are equivalent mathematically.

    daniel pineault say:

    It's very hard to help without seeing the macro details, or having access to a copy of your database.

    "it is needed to actually go to every specific field and change it via the drop down list"

    Normally, for this you would use code such as the following in the combo box's after update event.

    On Error GoTo Error_Handler
        Dim rs                    As DAO.Recordset

        Set rs = Me.RecordsetClone
        rs.FindFirst "[YourfieldName]=" & Me.YourComboBoxName
        If rs.NoMatch Then
            MsgBox "No entry found.", vbInformation
        Else
            Me.Bookmark = rs.Bookmark
        End If

    Error_Handler_Exit:
        On Error Resume Next
        If Not rs Is Nothing Then Set rs = Nothing
        Exit Sub

    Error_Handler:
        MsgBox "The following error has occured" & vbCrLf & vbCrLf & _
               "Error Number: " & Err.Number & vbCrLf & _
               "Error Source: YourProcedureName" & vbCrLf & _
               "Error Description: " & Err.Description & _
               Switch(Erl = 0, "", Erl <> 0, vbCrLf & "Line No: " & Erl) _
               , vbOKOnly + vbCritical, "An Error has Occured!"

    I'd also recommend never using Macro and doing all your work in VBA.  Macros are very difficult to work with and do not expose all the capabilities that VBA allows for.

    ken sheridan say: You appear to be attempting to navigate to a specific record by selecting a value in the combo box.  With a form bound to the table or query containing the records, the combo box must be unbound, i.e. its ControlSource property must be empty.  Otherwise you will not navigate to a record, but change the value of the current record's primary key, which will really mess up the data.

    With an unbound combo box you can navigate to the selected record.  You'll find an example of this, amongst other things, in FindRecord.zip in my public databases folder at:

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

    If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

    In this little demo file the RowSource property of the combo box is:

    SELECT ContactID, FirstName & " " & LastName, 1 As SortColumn,
    LastName,Firstname  
    FROM Contacts  
    UNION
    SELECT 0, "<New Contact>", 0,"",""  
    FROM Contacts
    ORDER BY SortColumn, LastName, FirstName;

    By means of the UNION operation the combo box's drop down list shows not only the names of the contacts, but also <New Contact> as the first item, which enables the user to go to an empty new record and insert a new contact record.  The combo box lists the names, but its actual value is the numeric ContactID, which is the table's primary key.  This is hidden by setting the combo box's ColumnWidths property to zero.

    To go to the selected record the following code is in the combo box's AfterUpdate event procedure:

        Const MESSAGETEXT = "No matching record"
        Dim ctrl As Control
        
        Set ctrl = Me.ActiveControl
        
        If Not IsNull(ctrl) Then
            If ctrl = 0 Then
                ' go to new record and move focus to FirstName control
                DoCmd.GoToRecord acForm, Me.Name, acNewRec
                Me.FirstName.SetFocus
            Else
                With Me.RecordsetClone
                    .FindFirst "ContactID = " & ctrl
                    If Not .NoMatch Then
                        ' go to record by synchronizing bookmarks
                        Me.Bookmark = .Bookmark
                    Else
                        MsgBox MESSAGETEXT, vbInformation, "Warning"
                    End If
                End With
            End If
        End If

    To go to a selected contact the code first finds the row in a clone of the form's current recordset, and then sets the form's Bookmark to the clone's Bookmark.  This causes the form to move to the selected record.

    You'll note that the form does not include any control for entering the ContactID primary key when a new record is inserted.  This is because the column is an autonumber, whose value is automatically inserted.  As the number has no semantic significance, there is no point showing it in the form.  In situations where the primary key is not an autonumber, and you do wish to show it in the form, it's bound control should be a text box, which would be additional to the unbound navigational combo box.

    ken sheridan say: The unbound combo box is purely for navigating to a selected record by means of the primary key of the form's recordset.  In the example I gave you this is the ContactID column.  The combo box's RowSource property in theory need return only that column, but this would require the  user to know the ContactID value for the contact in question, which is unlikely.  The combo box is therefore set up to list the contact's names in its second column and the first column, containing the ContactID, is hidden.  The user can then select the contact by name, but the code uses the hidden ContactID to find the record.  The contact's name is not in the form's recordset, however, so of the columns which the combo box's RowSource returns, only the ContactID matches a column in the form's recordset.  It is not necessary to include any other columns from the form's recordset.

    As Daniel has said, the types of bound controls in your form are irrelevant to the unbound navigational combo box.  Broadly speaking, where a column to which a control is bound is a foreign key referencing the primary key of another table a combo box would be used, where the column is an attribute which does not reference the key of another table a text box would be used, unless the column is of Boolean (Yes/No) data type, in which case a check box would most commonly be used.

    jay freedman say:
    Basically I agree with Charles, "nothing is foolproof because fools are so ingenious." But there is something you can try, as long as your users don't need to use Tracked Changes while making their modifications.

    After inserting your section breaks, go to the Review tab and click the Restrict Editing button. In the pane on the right, check the box under "2. Editing restrictions" and leave the dropdown set to "No changes (Read Only)." The only major function that's disabled throughout the document is Track Changes (but I'm not sure why). If you want to delete a section break, you'll have to supply the password.

    Deleting Graphics - Microsoft Community

    faxylady50 say:

    Deleting Graphics

    In a 400 page document, where I have copied and pasted contact information, I would like to delete all the pictures and graphics at one time.  Can I use the Edit-->Find function?  Using Win 7
    jay freedman say:
    If all of the pictures have their Text Wrapping set to In Line With Text, then you can use the Edit > Replace dialog. Type the code  ^g  in the Find What box, leave the Replace With box empty, and click the Replace All button.

    If more than a few pictures have any other Text Wrapping (such as Square or Tight), the Replace function won't find them. In that case the fastest method is to use a macro (see http://www.gmayor.com/installing_macro.htm if needed):

    Sub DeleteAllGraphics()
        Dim shp As Shape
        Dim i As Long
       
        ' use Replace to remove inline graphics
        With ActiveDocument.Content.Find
            .Text = "^g"
            .Replacement.Text = ""
            .Execute Replace:=wdReplaceAll
        End With
       
        ' remove floating graphics
        For i = 1 To ActiveDocument.Shapes.Count
            ActiveDocument.Shapes(1).Delete
        Next
    End Sub

    This macro will delete only the graphics in the body of the document, but not any that might be in text boxes, headers or footers, footnotes, or endnotes. If you need that, it can be added to the macro.
