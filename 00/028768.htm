jllatham say:

First a basic question:  what does the name of the workbook end with?  Hopefully .xlsm or .xlsb  and NOT .xlsx.   .xlsx files strip macros from them and so if you put a macro into one and open it later, the code is gone.  The UserForm itself is quite likely not there either.

 

Now, if it is a .xlsm or .xlsb file, then you should receive an alert near the top of the worksheet window in a yellow bar that looks like this:

 

If you see that when you open the file, click the [Enable Content] button!

 

Now, if you don't see that and the workbook is a .xlsm or .xlsb file then you probably need to change your Macro Security settings:

[File] tab

"Options" entry

In the Excel Options dialog that opens, choose "Trust Center"

Click the [Trust Center Settings] button over on the right side of the dialog.

Next choose "Macro Security" and I recommend the settings shown below:

 

Click the [OK] button on the dialogs to back out of the dialogs.

 

Close and re-open Excel  These settings do not take effect until this is done.  From then on, when you open a workbook with macros in it, you should get the alert and you can simply click the [Enable Content] button to make them available.

 

Now I'll repeat what I said earlier: if your workbook is a .xlsx file, then most likely you're going to have to recreate the code and very likely have to recreate the UserForm.  Open the file and press [Alt]+[F11] to go into the VBA Editor and see if there's any code or UserForm in the VBA Project at all.

 

Finally - last possibility is that you have "assigned" a macro of the wrong name that doesn't exist.  Open the workbook, go to the sheet with the button on it and right-click near its edge to get the popup that allows you to set its properties and such and choose "Assign Macro" and it will show you which macro is currently assigned to it, and if it is not one in the list shown, click the proper one in the list and click [OK] to assign it to the button.  OR - use TasosK's suggestion to automatically open it during the Worksheet_Activate event.

 

The error message "Cannot run the macro. The macro may not be available in this workbook or all macros may be disabled." should have been dealt with at this point.

 

The need to unprotect cells that the code in the UserForm might change is still there - you can either unlock those cells or do this kind of thing in the frmTime's code where it writes information to cells on the sheet.  In this example the word "myPassword" is an example, use the actual password for the sheet in the code:

 

ActiveSheet.Unprotect password:="myPassword"

ActiveSheet.Range("A1")= someValue

ActiveSheet.Range("X99")= someOtherValue

ActiveSheet.Protect password:="myPassword"

jllatham say:

TasosK - the 1004 error code is usually associated with something outside of the immediate VBA code itself.  And I suspect it's something in the code of the UserForm.

 

What happens is that the frmTime.Show is executed and that form attempts to open up.  During the opening of the form it either actually encounters an error during part of its _Initialize() or _Activate() event processing, if any is in use, or the VB engine detects an error of some type in another area of the form's code.  The form itself passes back a message saying "sorry, but I just threw up all over myself (and your back seat)"  and the code there in the macro simply says "some backseat passenger is not feeling well".

 

What I usually do to find those types of errors is put a STOP statement in the code in some part of the userform's processing to get it to open up in debug so I can [F8] through it to find the problem.  Even then it can get twitchy - you have to remember where you are as you step through, because I've seen it just pretty much go bye-bye without actually telling you what error occurred when you [F8] on a bad line of code.

 

This could be something as simple as referring to a control on the UserForm by the wrong name, or referencing an invalid property or method for one of them during the _Initialize() process for instance.  Let's say you'd done this and missed the notification:

Me!TextBox1.Caption = "Good Morning"

TextBoxes don't have captions!  Labels have captions.

 

But also, if during that process the UserForm tries to write to a locked cell on the protected sheet, the same type of situation can arise - the solution there would be to unprotect the sheet before using the frmTime.Show statement, or make darned sure that the code in the UserForm unprotects the proper sheet before it tries to write something to a cell on it.

macropod say:
Katrina: Simply telling someone to use Find/Replace to try to find duplicate paragraphs/sentences in a 300-page document isn't terribly helpful. For a document of that length, automation is called for.

macropod say:

Finding duplicated paragraphs can be done easily-enough with a macro like:

Sub DemoA()
Application.ScreenUpdating = False
Options.DefaultHighlightColorIndex = wdYellow
Dim RngA As Range, RngB As Range, i As Long, j As Long, strFnd As String
With ActiveDocument
  For i = 2 To .Paragraphs.Count - 1
    Set RngA = .Paragraphs(i - 1).Range
    Set RngB = .Range(.Paragraphs(i).Range.Start, .Range.End)
    With RngA
      strFnd = Trim(Split(.Text, vbCr)(0))
      If Len(strFnd) > 0 Then
        If .HighlightColorIndex <> wdYellow Then
          With RngB.Find
            .ClearFormatting
            .Replacement.ClearFormatting
            .Text = strFnd
            .Replacement.Text = "^&"
            .Replacement.Highlight = True
            .Format = True
            .Forward = True
            .Wrap = wdFindStop
            .MatchWildcards = True
            .Execute Replace:=wdReplaceAll
            If .Found = True Then RngA.HighlightColorIndex = wdBrightGreen
          End With
        End If
      End If
    End With
    If i Mod 100 = 0 Then DoEvents
  Next
End With
Application.ScreenUpdating = True
End Sub

For PC macro installation & usage instructions, see: http://www.gmayor.com/installing_macro.htm
For Mac macro installation & usage instructions, see: https://wordmvp.com/Mac/InstallMacro.html

300 pages is a lot to process, so don't expect instant results. Where duplicates are found, the first occurrences will be highlighted bright green; the duplicates will be highlighted yellow.

After running the macro, selecting any paragraph highlighted bright green and using 'Find' with the reading highlight setting will help you to see both the original and its duplicates for editing. After you've completed that, simply use Word's highlighting tool to remove all remaining highlights. 

Sentences are more problematic, because VBA has no idea what a grammatical sentence is. For example, consider the following:
Mr. Smith spent $1,234.56 at Dr. John's Grocery Store, to buy: 10.25kg of potatoes; 10kg of avocados; and 15.1kg of Mrs. Green's Mt. Pleasant macadamia nuts.
For you and me, that would count as one sentence; for VBA it counts as 5 sentences. If you're prepared to live with that, you could run the following macro. Initial and duplicate highlights are pink and teal, respectively. Be prepared for a much longer wait.

Sub DemoB()
Application.ScreenUpdating = False
Options.DefaultHighlightColorIndex = wdTeal
Dim RngA As Range, RngB As Range, i As Long, j As Long, strFnd As String
With ActiveDocument
  For i = 2 To .Sentences.Count - 1
    Set RngA = .Sentences(i - 1)
    Set RngB = .Range(.Sentences(i).Start, .Range.End)
    With RngA
      strFnd = Trim(Split(.Text, vbCr)(0))
      If Len(strFnd) > 0 Then
        If .HighlightColorIndex <> wdTeal Then
          With RngB.Find
            .ClearFormatting
            .Replacement.ClearFormatting
            .Text = strFnd
            .Replacement.Text = "^&"
            .Replacement.Highlight = True
            .Format = True
            .Forward = True
            .Wrap = wdFindStop
            .MatchWildcards = True
            .Execute Replace:=wdReplaceAll
            If .Found = True Then RngA.HighlightColorIndex = wdPink
          End With
        End If
      End If
    End With
    If i Mod 100 = 0 Then DoEvents
  Next
End With
Application.ScreenUpdating = True
End Sub

macropod say:

Well spotted. In that case, the first sub should become:

Sub DemoA()
Application.ScreenUpdating = False
Options.DefaultHighlightColorIndex = wdYellow
Dim RngA As Range, RngB As Range, i As Long, j As Long, strFnd As String, strTmp As String
With ActiveDocument
  For i = 2 To .Paragraphs.Count - 1
    Set RngA = .Paragraphs(i - 1).Range
    Set RngB = .Range(.Paragraphs(i).Range.Start, .Range.End)
    With RngA
      strFnd = Trim(Split(.Text, vbCr)(0))
      If Len(strFnd) > 0 Then
        If .HighlightColorIndex <> wdYellow Then
          If Len(strFnd) > 255 Then
            strTmp = Left(strFnd, 255)
            With RngB
              With .Find
                .ClearFormatting
                .Replacement.ClearFormatting
                .Text = strTmp
                .Format = True
                .Forward = True
                .Wrap = wdFindStop
                .MatchWildcards = True
                .Execute
                If .Found = True Then RngA.HighlightColorIndex = wdBrightGreen
              End With
              Do While .Find.Found = True
                If Trim(Split(.Paragraphs.First.Range.Text, vbCr)(0)) = strFnd Then
                  .Paragraphs.First.Range.HighlightColorIndex = wdYellow
                End If
                .Collapse wdCollapseEnd
                .Find.Execute
              Loop
            End With
          Else
            With RngB.Find
              .ClearFormatting
              .Replacement.ClearFormatting
              .Text = strFnd
              .Replacement.Text = "^&"
              .Replacement.Highlight = True
              .Format = True
              .Forward = True
              .Wrap = wdFindStop
              .MatchWildcards = True
              .Execute Replace:=wdReplaceAll
              If .Found = True Then RngA.HighlightColorIndex = wdBrightGreen
            End With
          End If
        End If
      End If
    End With
    If i Mod 100 = 0 Then DoEvents
  Next
End With
Application.ScreenUpdating = True
End Sub

dale howard [mvp] say:

DKapavik --

As you now know, the Team Planner view is useful for analyzing and resolving resource overallocations.  What you cannot do is to turn off the red formatting completely.

The reason why the red comes and goes depending on your level of zoom is because of how Microsoft Project calculates and displays overallocations based on the sensitivity setting you have specified in the Resource Leveling dialog.  Click the Resource tab to display the Resource ribbon, and then click the Leveling Options button in the Level section of the ribbon.  In the Resource Leveling dialog, you will see a pick list called Look For Overallocations On A _______ Basis.  The default value in this field is Day by Day, which means that when you have the level of zoom set to show Days, you will see the red overallocation formatting.  If you change this option setting to Month by Month, you will only see red overallocation formatting where the resource is overallocated on a monthly basis.  Maybe that would meet your requirements.

The alternative to make the red overallocation formatting to go away is to actually level the resource overallocations using a method of your choice.  Using the built-in leveling tool is one option, but you have many options as well, such as delaying tasks individually or assigning a task away from an overallocated resource and giving the task to a resource with availability.

Hope this helps.
