Hi Rob,

These references are related to OneNote 2016. So I assume that you are using OneNote 2016 instead of OneNote for Windows 10 (UWP). In OneNote 2016, you could try the following steps to import your personal notebook to your work account.

1.Open OneNote 2016 and sign in your personal account

2.Click File >click Open to open the notebook you want to export 

3.After that, click File >click Export >Select Notebook >Choose OneNote Package (*.onepkg) >Export to a local path like Desktop

4.Click File >click Account >Switch account >Sign in your work account

5.Click Open >Select This PC under other location >Find the previous path and select OneNote Single File Package or All files to display the notebook package >Open >Create 

6.Click File >Share >Select and click your work account's OneDrive location >Rename it and click Move

7.Then open your OneDrive for Business with your work account. You will see the notebook in that location

Hope this helps. Feel free to post back if you can import your notebook by following the steps above.

Prevent losing email in Outlook.com when you've set up POP3

Outlook.com has a feature that prevents losing email messages if you accidentally set up POP3 to delete them after reading them on the server. Here’s how it works:

    The first time that Outlook.com receives a POP3 instruction to delete email messages after reading from an email app, Outlook.com won't delete the messages. Instead, Outlook.com moves the email messages that were successfully downloaded by the email app into a POP folder and out of the inbox. The POP folder is available in the Folders list.

    Outlook.com keeps the email messages in the POP folder until you sign in to your Outlook.com account and verify whether Outlook.com should respect or ignore the POP3 delete command.

Don’t let another program delete messages from Outlook.com. (If your other program is set to delete messages from the server, we’ll simply move them to a special POP folder. They won’t be deleted.)

hansv mvp say:

The primary key in each of the tables (CategoryID, DesignerID etc.) is a Number (Long Integer) field.

The corresponding fields in the Charts table should be Number (Long Integer) too, but some are Number (Double), namely:

CategoryID

DesignerID
FormatID

You must change these to Number (Long Integer). When you save the table, Access will warn you that data may be lost. Don't worry about that, click Yes.

Similarly, change the MaterialTypeID in the Materials table to Number (Long Integer).

There is one other problem: the records with ChartID=1110 and 1160 in the Charts table have DesignerID=1149. This value does not occur in the Designer table, so currently it is not possible to enforce referential integrity for the relationship between Charts and Designers. You'll either have to add a record with DesignerID=1149 to the Designers table, or change the DesignerID of those two records in Charts to a value that occurs in Designers (or clear their DesignerID).
