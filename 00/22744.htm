I understand it can interact with the sheet, but haven't found anything about calling macros.





You said you can interact with the sheet, means for me you have access (at min.) to the sheet object.




The parent of the sheet is the file and the parent of the file is the application (Excel).




At first you can call only public macros, if they are in a sheet you can call them directly through the sheet object, example for VBA when Sheet1 contain a Sub named "Test":




Sheet1.Test




But when the "macro" is placed in a regular module, you can not call it that way. For this you have to use Application.Run




The first argument is the name of the macro and you can specify the exact location... if needed:




Let us assume that Sheet1 contain this sub:


Sub Test()

  MsgBox "Sheet1"

End Sub




And Module1 contain this sub:


Sub Test()

  MsgBox "Module1"

End Sub




And Module2 contain this sub:


Sub Test()

  MsgBox "Module2"

End Sub




And ThisWorkbook contain this sub:


Sub Test()

  MsgBox "ThisWorkbook"

End Sub




And we have another file "File2.xlsm" opened that also contain Module1 with this sub:


Sub Test()

  MsgBox "File2.Module1"

End Sub




In that case you can not just execute this and hope that the correct macro is called:




Sub MyClickEvent()

  Application.Run "Test"

End Sub




IMHO this calls always Module1.Test




When you want to be sure which macro is called you have to specify the exact "path" where the macro is located in:




Sub MyClickEvent()

  Application.Run "Module1.Test"

  Application.Run "Module2.Test"


  Application.Run "Sheet1.Test"

  Application.Run ThisWorkbook.CodeName & ".Test"

  Application.Run "File2.xlsm!Module1.Test"

End Sub




The reason for  the use of  ThisWorkbook.CodeName is that the name of this code module is language depended, e.g. in German it is named "DieseArbeitsmappe".

Another option is to add 12 months:

 sherrymanuel  say:


            Access 2010 Locking errors on a split database

             albert d. kallal  say:

    
    

            
            
            
            
In general, since users are on “different” records, then you will rare see the “locking” symbol.


However in a sub form with “small” records, then two users working on a different “main” record and editing (or adding) the sub form record “can” in some cases cause what we call a “page lock”.




The above is a fancy way of saying that Access will “lock” that page in question to update data, and records from “different” users can be in that same page.




A possible solution is to turn on what is called “row” locking. I shall warn that this feature will cause signification BLOATING in the back end database, but it certainly would be worth a try in your case.




The option is found via file->Options->Client Settings


The option you want to try is 




“Open database by using record-level locking).








Note that you can leave settings above "default" as to NO locks for default record locking. So EVEN WITHOUT ANY locking set, the record-level setting WILL force records to be placed in their OWN page, and thus no locking conflicts will occur.




ALSO NOTE that the above settings are a default. You will NOT see a lock symbol in a form UNLESS you set the form to lock edited record (or perhaps you changed a form to table locking). So it somewhat confusing, since setting the above options will NOT change
 how a form works unless you open up the form in design mode, and in the data properties tab check (or set) the locking choice.





In effect the above 3 settings are ONLY for new forms default, the existing forms are not change. However the above last option of Open database by using record-level locking IS GLOBAL.






I suspect this is only occurring when editing very small tables, since then a “lot” of records can be packed into one database page. The above option forces records to be placed in their own page. So this solves the “conflict” issue, but as noted will cause
 a substantial increase is back end file growth (bloat). As a result you have to adopt an “increased” frequency of compacting the back end database.




Also, the fact that users received a “lock” symbol in the record selector should NOT cause a freeze up, so there certainly something else here going on that does not seem quite right???. And users should NOT see the lock symbol UNLESS the form is set to
 lock edited records. If the form is set = no locks, then they will see the “edit pencil” and receive an error that the records been changed by someone else AFTER they edited the record. If you have locking = on, then they see a lock symbol RIGHT away and cannot
 edit the form. And if the form is record locked, then even VBA code that attempts to set or edit data likely can and will freeze up. So I would for first test REMOVE any locking settings you have on a form - you should NOT be seeing that lock symbol unless
 the form in question has locking = on.




So from your current description, you have that form (the sub form) set as row locking = table or edited record. Perhaps change that setting to none.




So I would check the form(s) in question, and see what the locking settings are. It not possible to see locking symbols unless the form has locking set.
