jay freedman say:

There is no way to edit an AutoText entry in storage. The way to change an entry is

  1. Open a document based on a template that has the styles defined and named as you want them to be.
  2. Insert an existing AutoText entry into the document.
  3. Make any changes needed to make the insertion look and behave as you want, including renaming and reapplying the styles.
  4. Select the properly styled material, press Alt+F3, and complete the Create New Building Block dialog, using the entry's original name (and Gallery and Category, if necessary). Be sure to select the correct template in the Save In box!
  5. Repeat for each of the other AutoText entries.

jatfdy say: Hello Jay, I did create all of the autotext in a precedent file where all the autotext was located with the customised styles already in the document. 

jay freedman say:

To clarify what you're seeing, please answer a few questions:

  • Does the change appear for all users, or just for one?
  • Do you want the Styles pane to show "Heading 1" and not "Main Heading", or the other way around?
  • It sounds like the affected styles have aliases. To see this, hover the mouse pointer over the style's icon in the Home ribbon. In the tooltip, is the built-in style name followed by a comma and the alternative (the alias)?
    • In the Styles pane, click the Options link at the bottom. In the Options dialog, the last check box "Hide built-in name when alternate name exists" enables two kinds of display of style names and aliases:
    • If the box is cleared, the built-in name of the style determines the position of the style in the list (only when the display order is alphabetical). The comma and alias are also shown there.
    • If the box is checked, the alias determines the position of the style, and the built-in name doesn't appear at all.
    • Neither setting of the option affects the way the style name is displayed in the Styles gallery in the Home ribbon: only the alias appears on the tile, but the built-in name is listed first in the mouse-over tooltip.

If the change affects only one user, then that user needs to change the option setting. If it affects everyone who uses the template containing the AutoText, I'm at a loss to explain that unless they all have the same setting.

sven johannsen say:

There are a lot of things that could account for this, but let me first explain what exactly conflict resolution means. When you first sync a WM device it may ask if you wish to replace all items on the device, or merge the two lists. Typically if there is nothing on the device, it may not even ask.  In that first sync, each entry, whether that is a contact, calendar entry, task, whatever, gets assigned a unique ID you don't ever see. That is how a particular item is paired with its mirror on the other device. When you make a change to an item, which could be a phone number in a contact, length of an appointment, etc., a flag is set that says something changed in that record. When you sync, those items with flags overwrite their twins on the other device, in either direction.  If you change something on both the device and the desktop in the same record, between syncs, Bob Jones birthday on the desktop, and his cell number on the device, for instance, a flag is set on both copies, and that constitutes a conflict. When you sync, the system sees a flag on both copies of an entry, and will respond in the way you have selected...by overwriting in the direction you specified. In my scenario, with your selection of 'replace on device', you would have Bob's correct birthday, but you would loose his new cell number. You replaced Bob's whole contact entry on the phone with what was on the desktop.

A long time ago, a third option existed, which was ASK. For each conflict, and remember that is only when something changed in the same record on both partners, AS would prompt you for which to keep. You would still loose something, but you had some control over what, and were alerted that something likely was lost. Been begging for that back, but it has fallen on deaf ears.  To me it is the only option that made sense. Either other option allowed you to lose data without being informed of that. My solution was to make darn sure I sync'd when I got home if I had made any changes on my device while out, before I did anything else. This ensured that nothing changed on my desktop before the mobile change had been sync'd. Then if I made changes on my desktop, make darn sure I sync'd before I left, so that I couldn't accidentally create a conflict while out.  I typically set 'replace on device' as I am much more likely to make changes on the desktop, so it is usually more up to date. I try to ensure a conflict never happens.

When things seem to get screwy, it appears that somehow the sync tables have got out of whack. Duplicates indicate that two records you think are identical have a different 'key'. Deleting one on one side, should delete it's mirror. They seem to be created when somehow the partners have lost track of each other, and both sides don't see a mirror entry and create a new one. With all the stuff you went through, it is hard to say what exactly happened there. but attaching, removing, changing, conflict rsolution, make lots of changes, can get things confused.

I would pick a conflict direction and leave it. Doesn't come into play often. If you are game, make darn sure everything on the desktop is right. Delete the partnership on both the PC and the device. When you delete the partnership on the device in AS, it will clear all the entries associated with the PC. Again make sure the PC is right, then re-partner. All the PC entries will go to the device cleanly (knock on wood).
