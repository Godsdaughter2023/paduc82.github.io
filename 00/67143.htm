A few things could be happening.
  
This MultiMediaCard (MMC) may be partitioned and you are only seeing ONE partition.  Unfortunately that may just be the way it is.
  
The card reader itself may not be able to properly read this particular MultiMediaCard (MMC) - that would depend on the card reader and the MultiMediaCard (MMC) specifications.  It 'fitting' and you even getting some result on your computer doesn't mean it can be read properly by that reader.  ;-)
  
The MultiMediaCard (MMC) may not only be partitioned but it may be of some proprietary format.  So your system may not know what to do with the format of the card as a whole when in that particular reader.
  
http://www.diffen.com/difference/MMC_vs_SD_Card
  
A few questions... (although likely your best bet is to get the cable and read it like you normally would - through the device it normally fits in):
  
1: What is the make & model of the digital camera?
2: How do you normally get pictures off this camera - physically and through what software?
3: Have you ever read this MultiMediaCard (MMC) in this manner before (even on a different machine with a different operating system and card reader?)

The Windows 10 Upgrade mechanism fails for many reasons, and - Microsoft Community

sba923 say:

The Windows 10 Upgrade mechanism fails for many reasons, and shouldn't

I want to share my experience with the Windows 10 Upgrade process.

I first had to fight to work around a C1900101-3000D error in FIRST_BOOT/MIGRATE_DATA, with a blocking "Visual C++ runtime" message box, caused by a memory leak related to the presence of many obsolete entries in the component registration of Windows Installer (see http://answers.microsoft.com/en-us/windows/forum/windows_10-update/windows-10-update-fails-with-c1900101-3000d-error/043db489-a09e-412d-9ebe-6c5b1d734fa5 and http://windows-update-checker.com/FAQ/SOLVED%20Windows%2010%20upgrade%20error%200xC1900101%20-%200x3000D.htm). To get past this crash, I had to cleanup that data using a handcrafted PowerShell script and CCleaner. I also had to move the user shell folders that I had relocated from C: (SSD) to D: (HDD) back to C:. Note: relocation had been done using a built-in, supported feature (the "Location" tab on a special folder's properties, which is still present in Windows 10). Before that, I had experimented quite a lot of things: uninstalling apps, repairing apps (e.g. Visual Studio 2015), removing the English language pack, using "SFC /SCANNOW" or "Dism /Online /Cleanup-Image /RestoreHealth" (which would never be able to repair everything), etc.

I then faced another error (80070004-4000D, in SECOND_BOOT/MIGRATE_DATA) because some data in my primary user account was not handled properly. I had to backup the data, delete the account from another administrator account, and start the upgrade process again. I now will have to recreate that account and reconfigure everything before everything gets back to normal.

Because the conversion of the Windows 7/8.1 activation data to Windows 10 activation data is done only upon a successful upgrade, I wouldn't have been able to just wipe out everything and do a clean install using a Media Creation Tool generated USB key.

Overall, upgrading my Dell Studio 1735 from Windows 8.1 to Windows 10 took me over a week.

Being a professional software developer and PC specialist for over 25 years, I was able to make some sense of the logs in C:\$WINDOWS.~BT\Sources\Panther, as well as capture and analyze memory dumps of crashing setupplatform.exe's.

But normal users such as one's relatives and non-computer-savvy friends should not have to ask people like me for help (which happens more often than not, for instance a friend of mine who's a professional photographer wouldn't have been able to upgrade his main production machine without me) because the experience is just not ironed out enough!

I understand the testing matrix for this kind of stuff is huge and one can thus easily miss some test cases, but still...

I hope Microsoft is listening... and looking at the countless failure reports they got via telemetry over the past week!

P.S. all logs and dumps are available upon request.

richardeiler say:

Hi Thor,

Error code 0x80073715 = ERROR_SXS_MANIFEST_IDENTITY_SAME_BUT_CONTENTS_DIFFERENT

The identities of the manifests are identical, but the contents are different.

It’s possible that the 0x80073715 error was caused by an incomplete download.

There are also other possible causes as well, one recent is a known problem that was caused by some changes made to the system by KB4056892 if it had been installed on the device previously.

Check your update history, is KB4056892 listed as installed?

If so try this:

Uninstall KB4056892.

Then in an elevated command prompt type or copy and paste the command below then hit enter (this will take some time to complete).

Dism.exe /online /Cleanup-Image /StartComponentCleanup

After DISM finishes close the command window, then;

Start the regular windows update to let the OS find its updates (it may or may not reinstall ..892) KB4074588 should then install.

If KB4056892 is not listed in your update history let us know and we can go from there.

Either way, let us know how it goes.

-Richard

How to Restore files backup using File History from an external drive - Microsoft Community

josemanstretta say:

How to Restore files backup using File History from an external drive to a NEW PC

I haven't found the answer yet on how after use File History to backup my data from an old pc I can restore those file and save them in a new one. Both PC have Win10 and the backup (using File History) was made to an external drive. Anyone?

richardeiler say:

Hi Jose,

This would be the procedure to follow:

Once you have Windows 10 installed on your new hard disk, reconnect your external hard disk and launch File History.

To do so, access Settings from the Start menu or from the Action Center, select the Upgrade & Security tile in the Settings window, and select the Backup tab. On the Back Up Using File History page, click the Add A Drive button. Windows 10 will then search for and prompt you to select a drive. When you select the drive containing your backup, File History will recognize that it already contains a set of files.

Foe a full tutorial follow this link:

http://www.techrepublic.com/article/restoring-files-from-windows-10s-file-history/

I hope this helps,

-Richard
