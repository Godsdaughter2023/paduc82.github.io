Proposed Treatment of Live Tiles in Windows 10 - Microsoft Community

antonio java say:

Proposed Treatment of Live Tiles in Windows 10

TL;DR Version:

Allow Live Tiles to be placed on desktop.

Long Version/Rationale:

Windows 8 adapted the “Metro” design language from Windows Phone in an attempt by Microsoft to unify the look and user experience of all Windows products. In doing so, however, the Windows 8 experience became a little convoluted. I'm not even going to touch on the “full-screen app” and “desktop app” issues (since this has thankfully been addressed in the Windows 10 preview anyway). This post will concentrate entirely on the Metro design's “Live Tiles” aspect.

When they were first introduced in Windows Phone 7, Live Tiles presented a new way for applications to interact with the user by combining what is essentially a program icon or shortcut with the functionality of a widget. With Live Tiles, an otherwise static icon/shortcut becomes a useful source of at-a-glance information related to the application, and the same system was carried into Windows Phone 8/8.1.

For Live Tiles to work on Windows Phone, a user had to select the appropriate program from the main application list and pin it to the Start Screen, where all the Live Tiles can be rearranged and resized. Because the Start Screen on the Windows Phone OS was the “default screen” – meaning it's what's displayed when programs aren't running – the functionality of Live Live Tiles is maximized by allowing the user to see tile information immediately and constantly when no other programs were open.

When Microsoft adapted Live Tiles on its PC OS in the form of Windows 8, it used the same principle as in Windows Phone – take an app from the main app list, then pin it to the Start Screen where the tile can constantly show its at-a-glance information. To maximize Live Tiles, Microsoft then made the Windows 8 Start Screen the “default screen” of that OS, much like it is in Windows Phone.

That, however, did not sit well with most Windows PC OS users, who, in previous iterations of the OS, have always considered their desktop screens as their default screen. Improvements introduced in Windows 8.1 allowed users to regain the feel that the desktop was the default environment, but the use of Live Tiles was still limited to the full-window “Start Screen”, which most users did not really care for, effectively decreasing the usability of Live Tiles.

Personally, I believe Microsoft was too fixated on terms and technicalities rather than functionality when it tried to incorporate Live Tiles into Windows 8. Because Live Tiles were on the “Start Screen” of Windows Phone, they thought it was only appropriate to also put Live Tiles on what was also called the “Start Screen” of Windows 8.

And now, the same is happening in Windows 10.

The Windows 10 preview has introduced many improvements that make it superior to all previous iterations of the OS, but I feel that Microsoft still missed the mark when it comes to how to implement Live Tiles. In the Windows 10 preview, by default, Start is no longer a screen like it was in Windows 8, but a menu much like how it was in Windows 7 (with a full-window “Start Screen” option should one go into tablet/touch mode). However, still of the mindset that Live Live Tiles should be plastered onto whatever was called “Start” on any given OS, Microsoft has made the Windows 10 Start Menu a hybrid of the folders/icons system found in Windows 7/Vista/XP and the Live Tiles-laden Start Screen of Windows 8/Windows Phone. This not only makes the Live Tiles redundant in the Start Menu, but also unoptimized, as the live information that makes a Live Tile awesome otherwise gets hidden when the Start Menu isn't in use, which is pretty much 99.9% of the time.

I believe what Microsoft needs to do to maximize the use of Live Tiles on PC is not to force it into whatever it calls “Start” in the different versions of Windows 10, but to allow it to be used in whatever “default screen” the OS uses.

For mobile and tablet users (including the PC version's “tablet mode”), that default screen is the Start Screen.

For PC users, that default screen is the DESKTOP.

On the Windows 10 PC version, this proposed setup essentially makes the desktop into THE Start Screen, but onto which windowed programs can be overlaid, while keeping the Start Menu purely an applications list (similar to how, in Windows Phone, a side swipe summons an applications list). For PCs with a “tablet mode”, the Live Tiles on the desktop then becomes the Start Screen of that mode. This better emulates the Windows Phone experience, but still completely respecting the Windows 7/Vista/XP experience as well. The bonus for PC users would be that they can open their windowed programs like they always do, then slightly resize them so that the user can see certain tiles on the desktop, allowing them to work while keeping an eye on the “live” information on the tiles.

Naturally, this setup replaces the Windows 7-style desktop shortcuts with the Live Tiles, since Live Tiles serve as shortcuts/icons anyway. For those who aren't big fans of Live Tiles, they can simply stick to the "Small", un-interactive tile sizes, creating an environment identical to Windows 7.

This system not only serves to further unify Windows 10 on all platforms, but also maximizes the use of Live Tiles on the PC version by always having them visible and showing their respective at-a-glance information when no applications are open or when the applications are windowed.

If a technical name is necessary for this kind of setup, let's tentatively call it the “Start Environment”. But the point is, this Start Environment should be whatever default screen the user prefers (or is limited to) – The Start Screen of mobile/touch devices, or the desktop of a PC.

In essence, instead of the mindset that the Start Screen and desktop are two different things, their functionalities should instead be merged, with the only difference being that mobile/touch devices have their applications open full-screen over the Start Environment, while PCs can open those same applications in resizeable windows over the Start Environment (desktop).

Similarly, the full list of applications can be accessed the same way: on mobile and tablets, a swipe from the side/bottom brings up the applications list in full screen, while on PC, a click on the Start Button brings up the same applications list in a Windows 7-style pop-up menu.

I hope this idea gets the attention of Microsoft as well as the developer community for consideration.

I will try to incorporate all these ideas into a mock “screenshot” in a future edit to this post. But as I am not particularly good with graphics editing, that might take a while.

Constructive criticisms welcome.

Thanks for reading!

brian tillman [outlook mvp 2007-2019] say:

If messages are deleted from the server when you delete them from the app, then I suspect you're not using a POP account like you believe because, in general, what you do in a POP client doesn't affect what's on the server. Messages travel in one direction only: from the server to the client, where they become completely local. The server no longer cares and cannot find out what you do with the local copies, except in one particular case.

There are two possibilities: one is you're using a synching protocol like IMAP or Exchange ActiveSync, in which case, whatever you do to a message in Windows 10 Mail will also happen on the server or two, you're using POP, as you say, but the client is deleting the messages from the server immediately when they get downloaded, not when you delete them on the client. You can easily test this by downloading messages, then visiting the mailbox on the server to see if the messages are there or not. If they're gone at that point, before you delete them from the server, then you know it's not when you delete them in W10M that they disappear from the server.

The "one particular case" I mentioned above is that some POP clients also have an option to send a delete command to the server when you empty the client's Deleted folder. So, while messages aren't deleted from the server when you click or tap Delete or swipe to delete on the client, when you empty the trash, the client will send a command to the server to delete the corresponding server-side copies of those messages.

I don't believe the option to choose whether or not to leave messages on the server when you configure a POP account exists for Windows 10 Mail. I think W10M always leaves messages on the server. Moreover, The Outlook.com server's default behavior is to ignore the DELE command it receives from the client when the Deleted folder is emptied and to move the messages to a folder named POP that is a subfolder of the Deleted folder on the server. Log into your mailbox on the server to see if that folder exists.
