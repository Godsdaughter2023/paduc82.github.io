accurate mgmt say:

Problems creating field-update macro

We're using Word and Excel (on a Mac) together to create a commission statement template. In our template we have calculation fields that need to be updated (since apparently theres no way to automatically update a calculation field?). So we were going to create a macro to update all the fields, and then assign the macro to the quick-toolbar as a solution. Heres how I tried to do this:

After it was recording, I pressed Cmd-A to select everything, then pressed Cmd-Opt-Shift-U to update the fields.

I then went under tools again and stopped recording the macro. 

However, then when I try to execute the macro I get an error:

Microsoft Visual Basic

Run-time error '9':

The item with the specified name wasn't found.

So what did I do wrong?

a. user say:

The others have made suggestions about what to do next, but as far as your question is concerned, i.e.

"So what did I do wrong?"

the answer is "nothing". 

But sometimes the macro recorder tries to record things that it shouldn't, and gets them wrong.

For example, if I put a text box in my document (or even a rectangle shape) then for some reason Word records the following code:

'

' Macro11 Macro

'

'

    Selection.WholeStory

    ActiveDocument.Shapes.Range(Array("Text Box 2")).Select

    Selection.Fields.Update

End Sub


When you try to run that code, it fails (with error 9) because of the line that starts "ActiveDocument". That line would actually work in Windows Word because in that case Shapes.Range specifies a "ShapeRange" and specifically the one where the shape is called "Text Box 2". 

That ShapeRange stuff is in Word's object model in Mac Word, but I do not think it has ever worked in the same way as on Windows, if it can be said to work at all.

In this case, the Macro Recorder seems to be trying to record a more "complete" operation than you have actually specified, because when you do cmd-A, then update all the fields, Word does not actually normally update the fields in text boxes, despite the fact that they are selected.

Even on Windows, playing/running the macro does not do what you might hope, because it selects the WholeStory, then fails to update its fields. It only updates the text box's fields. To do it all, you would need

    Selection.WholeStory

    Selection.Fields.Update

    ActiveDocument.Shapes.Range(Array("Text Box 2")).Select

    Selection.Fields.Update

The problem for the macro recorder is of course that it can only record what you do. It cannot tell what your "intent" was. In this case, somewhat unusually IME, it has substituted its own intent. But if your intent was to create a macro that updated the document body and all fields in it, then the recorded macro would be inadequate as soon as you added another text box. So as a general rule, macro recording can be very useful as a starting point, but in many cases you then have to recode in order to achieve what you really wanted.

suzanne s. barnhill say:
Right, that setting suppresses spacing between paragraphs of the same style, not before or after a block of those paragraphs. The purpose of this is, I think, primarily for bulleted or numbered lists, where you often want space before and after the list but not between the list items.

suzanne s. barnhill say:
If you set Spacing Before at 16 and Spacing After at 8, then that's what you'll get for that paragraph (unless the previous paragraph has Spacing After greater than 16 and/or the following paragraph has Spacing Before larger than 8). If you have two consecutive paragraphs with that formatting, then yes, you'll get 16 pts between them. But checking "Don't add space..." should suppress all the Spacing Before/After between paragraphs of the same style.
