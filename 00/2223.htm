Correction:  Using the Autonumber field created unpredictable results.  Instead, I used the fields that make up the Primary Key for the DCount -

 

(DCount("PONumber","customerdata","([sspart#] < '" & [sspart#] & "') or ([sspart#] = '" & [sspart#] & "' and customerpart < '" & [customerpart] & "')")*3) +1

mkeener say:

Although the above worked, it was very slow.  I changed the query to use a function to produce the "KeepOrder" field. 

 

Public Function fSeqNumBy3(vDummy) as Long

Static x as Long

Select Case vDummy

  Case "Initialize1"

     x=1

  Case "Initialize2"

    x=2

  Case "Initialize3"

    x=3

  Case Else

    fSeqNumBy3=x

    x=x+3

End Select

End Function

 

Then the query has column:  KeepOrder: fSeqNumBy3([PONumber]) that is appended to the KeepOrder field.

 

Works perfectly, and so much faster!

patrickcamblin say:

Evidently in its efforts to make help better in each subsequent version of MS-Office, Microsoft is making it progressively less so.  In earlier versions you could navigate to just about any menu command or select a dialog box item, or other objects, then press F1 and get context-sensitive help (CSH) on that particular item.  They even had a nifty Help menu icon that toggled your pointer, such that you could simply point & click on an object to get CSH.  With Office 2010 the CSH is even sketchier (see below), and very, very 'noisy' (you get all kinds of links to dead-end or incomplete posts in user-self-help forums (like this one, last updated in last year), matters having little to do with feature use, esoteric feature applications, etc).

 

To reduce the noise, do the following:  When you press F1 and Help opens, you'll see two search boxes.  Ignore the Bing box, and go to the Search bar in the upper left, just under the window's tool icons.  After entering your search argument into the text box (using quotes as necessary), click on the down-arrow of the Search drop-list and select "Excel Help".  This will restrict the search to the online help built into Excel, and will ignore the other help sources in the drop-list.

 

As to the "sketchy" nature of CSH, do the following:  Turn on Tool Tips in MS-Office Options.  With Tool Tips on you can hover your mouse over tool icons, ribbon-group-dialog buttons, commands and other objects to dispaly a pop-up showing the following (if available for the object in question): 1) The object name; 2) A brief description; 3) The keyboard shortcut for the object, if available; 4) A prompt to press F1 for CSH, if available.  The pop-up may not be available for all objects or my not contain all the indicated information (that's what makes it 'sketchy').  Try it on the Bold tool icon to see shortcut keys, and on the 'Fill Color' icon to see the F1-CSH prompt.  You can also take the object's proper name from the pop-up and enter it into the Help window search box.

 

Help within dialog boxes seems to still be context sensitive to a specific tab in the dialog, but no longer sensitive to the selected object within a tab.  Press F1 and get Tab help, then press [Ctrl]+F to bring up the text search dialog to search within the Search page for the object name.

time in a pivot table - Microsoft Community

simplyspiffing say:

time in a pivot table

I have a workbook that works fine in Excel 2008. 

On one sheet is a timesheet with columns to input start time, end time and task and another to calculate the time spent (the difference between start time and end time).

On the next sheet is a pivot table which shows, per week, the total time spent on each task. This is displayed as [h]:mm and has worked well for several years.

When I open this workbook in Excel 2011 (and Excel 2010, as it happens) the total time in the pivot table is a much, much larger number than expected and actually resolves as a closely related time with a (seemingly) random date from early in the 20th century added on. So, for instance, where Excel 2008 correctly shows 25:55, Excel 2011 shows 7 January 1928 02:55:00.

I've tried formatting the time differently. I've tried making sure that the date is part of the original calculation so that it really is just a difference in hours and minutes, but all to no avail.

Is it me? Or...?

j.e. mcgimpsey say: It's not you. Unfortunately, I don't have my Mac handy to check, but it's obvious what is happening.

Times and dates are the same thing to XL.

A date is just an integer offset from a base date - for instance, in the 1904 date system, today's date is 39317, or 39,317 days since 1/1/1904. Similarly, in the 1900 date system, it's been 40,779 days since 12/31/1899.

Times are stored as the fractional part of the date - so 12:00 noon = 0.5, 6:00 pm = 0.75, etc. 

Since you entered a time without a date, XL will translate 10:00 am to 0.416666666666667, which if you display it as a date, would be "1/1/1904 10:00"

Notice that the difference between the 1900 system and the 1904 system is 1462, or  4 years and 1 day (the extra day because XL erroneously thinks there was a 29 Feb 1900).

For some reason, the PT Sum() is treating your 1904 dates as if they occurred in the 1900 date system, and so adds 7x4 = 28 years (+ 7 days)  to the calculation.

As a workaround, try changing your workbook to the 1900 date system (Preferences/Calculation), and reenter your times.
