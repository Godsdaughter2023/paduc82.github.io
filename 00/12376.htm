ken sheridan say:
I meant the append query is constructed using three tables/query (I used the term recordset because a recordset can be either a table or a query...sorry for confusion)

A table can also be the result table of a query in fact, as distinct from a base table.  But that's by-the-bye.

You will need to identify what combination of columns in the query's result table identify a row as distinct.  You can then eliminate those rows by means of the NOT EXISTS predicate against a subquery, e.g. if a row in the query's result set can be identified on the basis of the primary key of one table in combination with a date in a column in another table:

INSERT INTO NewTable(NewTableID, SomeDate, <more columns>)
SELECT FirstTable.FirstTableID, SecondTable.SomeDate, <more columns>
FROM SecondTable INNER JOIN (FirstTable INNER JOIN JunctionTable
ON FirstTable.FirstTableID = JunctionTable.FirstTableID)
ON SecondTable.SecondTableID = JunctionTable.SecondTableID
WHERE NOT EXISTS
    (SELECT *
     FROM NewTable
     WHERE NewTable.NewTableID = FirstTable.FirstTableID
     AND NewTable.SomeDate = SecondTable.SomeDate);

This could also be done with a LEFT OUTER JOIN, but you may find it easier to use a subquery if the existing query is fairly complex.

However, you should consider very carefully whether there is a cogent reason for a new table rather than returning a result table from the original tables in a query.  Redundancy through the duplication of data gives rise to the risk of update anomalies, so should normally be avoided.  While the insertion of rows into a table for enhanced performance, e.g. as the RecordSource of a report, is sometimes appropriate, the data is usually appended only temporarily and the table is emptied prior to being filled again when next required, the duplication of data on a permanent basis can often be a sign of a less than perfect database design.
