braintrove.com say:

This can be resolved. A little background first.

You define what values the data source pulls from the actual source of the data. As you only configured a single data source, when you changed it for the second combo box, the definition changed, and of course all the other combo boxes that use the data source will change accordingly.

So let's say that all of your combo boxes use the same data source to populate their values and your SP List has columns A, B, C, D, and E. Now if your data source is configured to pull column A, then all your combo boxes will display the values in column A. Once you change the data source to instead pull column B, all of your combo boxes will change because they are all using the same data source.

To put it in a real world situation: Let's say there are two gum ball machines, one with all red gum balls one will all blue gum balls. You tell twenty children to put their pennies in the first. As expected, they all get red gum balls. Now you tell all of them to put their pennies in the second and, of course, they will all get blue gum balls.

Okay, how to fix the issue. There are two main things you can do:

1. Have your single Data Source, but have it pull back more data. You could have it return all the columns that each of your twenty combo boxes will need (there might be some overlap). This will be the most efficient because you will have a single call for data, even though the single payload will be larger. Then each combo box will get only the data it needs from the Data Source. You will not need to ever change the definition because it is configured already to pull back everything needed in one go.

2. Have individual Data Sources for each combo box. This is highly inefficient and not preferred unless there is no way around it (i.e. The source of the data is actually separate lists or on separate servers, etc.). Here then, comb box 1 would use data source 1, etc..., combo box N uses

Hopefully I understood your situation correctly and this is sufficient to explain what is going on and how to resolve the issue you are experiencing.

rick rothstein, mvp - excel say:
            If Err = 0 Then NamedRangeExists = True _
            Else NamedRangeExists = False
The above part of your code can be simplified. There is no need to check if something is True or False only to then directly assign True or False to a the function name (or, in general, to a variable)... just assign the test to the function name (or variable) directly. So the above can be written this way...

NamedRangeExists = (Err = 0) 

where the parentheses are optional, but many people think including them makes the statement easier to follow. So, if we implement this idea in your code, then it would become this...

'   Returns True if the named range exists in the active workbook.
Public Function NamedRangeExists(ByVal RngName As String) As Boolean
    Dim Adrs As String
    On Error Resume Next
    Adrs = Range(RngName).Address
    NamedRangeExists = (Err = 0) 
End Function

Of course, in thinking about it, as long as we are using an error condition to decide if a statment executes or not, we can eliminate the Adrs variable and assign a result for determining something about the range (lengtho of its address for example) directly to the function name. You want the cells in column B from B2 to the last one used to become shaded with the color indicated in cell A1 on the sheet.  Is that correct?  If it is, then the code below should help you - instructions for putting it into the proper place in your workbook are included, and you can find same instructions with pictures here:

http://www.contextures.com/xlvba01.html#Worksheet

this code goes into the Worksheet Module.

 

Private Sub Worksheet_Change(ByVal Target As Range)
'to put this code into your workbook:
'open the workbook, select the sheet to be used
'RIGHT-click on the sheet's name tab and choose
'[View Code] from the pop-up list.
'Copy this code and paste it into the code
'module presented to you.  Close the VB Editor.
'Save the workbook as type .XLSM (macro enabled)
'
'to test: put various 6-digit hex values into cell
'A1 on the sheet.
'
  Dim lastRow As Long
 
  If Target.Address <> "$A$1" Then
    Exit Sub
  End If
  lastRow = Range("B" & Rows.Count).End(xlUp).Row
  If lastRow < 2 Then
    Exit Sub ' nothing to color
  End If
  'Hex values in A1 must have 6 characters only
  'do not include # or &H, just hex value such as A2A2A2
  'make sure that A1 is formatted as TEXT
  On Error Resume Next
  Range("B2:B" & lastRow).Interior.Color = "&H" & Trim(Range("A1"))
  If Err <> 0 Then
    Err.Clear ' just ignore errors
  End If
  On Error GoTo 0
End Sub

jllatham say:

Here is a different approach: This will color a cell in B2:B# when you enter or delete a value from it (shades it when value entered/changed, clears shade when [Del] is used).  If you prefer, use this code instead of the other (you cannot use both at the same time).

 

Private Sub Worksheet_Change(ByVal Target As Range)
  'this will change the color of a cell in column B
  'when the value of a cell in column B is changed
  'if you [Del] the value in a cell in column B
  'it will become unshaded
  Dim anyCell As Range
  If Not Application.Intersect(Target, Columns("B:B")) Is Nothing Then

    Application.ScreenUpdating = False
    For Each anyCell In Target
      If anyCell.Column = 2 And anyCell.Row > 1 Then
        If Not IsEmpty(anyCell) Then
          On Error Resume Next
          anyCell.Interior.Color = "&H" & Trim(Range("A1"))
          If Err <> 0 Then
            Err.Clear
          End If
          On Error GoTo 0
        Else
          anyCell.Interior.Color = xlNone
        End If
      End If
    Next
  End If
End Sub
