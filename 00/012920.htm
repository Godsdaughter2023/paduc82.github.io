No; or not without great difficulty.  The problem is that the foreign key field in INSTRUMENTS is the autonumber PK of CAUSES.  You can't create the new records in INSTRUMENTS until you have created the records in CAUSES, and you don't know what the new CAUSES.ID is going to be for each record until you actually create that record.  Doing this in a separate set of work tables doesn't help, because when you copy the records back into the original tables, the autonumber PK that is assigned to each record in CAUSES will potentially be different.

However, if you can add an OldID field to your CAUSES table, then I think you can use that field to accomplish your goal without having to create separate work tables.  You would do this by setting that OldID to the original ID field when you copy the records:

    INSERT INTO CAUSES ( CAUSEID, SIFID, OldID )
        SELECT CAUSES.CAUSEID, [New SIF Number], ID
        FROM CAUSES
        WHERE CAUSES.SIFID=[SIF to Copy];

And then you could copy the appropriate records into INSTRUMENTS:

    INSERT INTO INSTRUMENTS ( INSTTAG, CID )
        SELECT INSTRUMENTS.INSTTAG, CAUSES.ID
        FROM CAUSES INNER JOIN INSTRUMENTS ON CAUSES.OldID = INSTRUMENTS.CID
        WHERE CAUSES.SIFID=[New SIF Number to Copy From]

Does that approach work for you?

ken sheridan say: I have doubts about the validity of what you are attempting.  If you think about this in terms of the database relational model, if you want to insert a new row into A whose related rows in B have identical data to those which reference the current row in A, and whose related rows in C have identical data to those in C which reference the current row in A via B, then it follows that the relationship types A----<B---<C are very unlikely to be a simple tree structured hierarchy, as B and C would almost certainly have invalid duplication of candidate keys.  In the abstract there is no implicit violation of the relational model, but in reality it's very unlikely to be a valid model.

I’ve succeeded in copying relational records within the same tables using SQL statements and maintaining the referential integrity of the new records.  There is certainly room for some more polish, but I’ll explain with generic table/field references how I’ve set it up.

Table A has a one-to-many relationship with Table B which has a one-to-many relationship with Table C.  The most important part of being able to copy an entire set of records from all 3 tables is setting up non-key fields in the tables to track the key fields so that values can be looked up in a “reverse referential integrity” order with the queries.  In my case I am cheating a little because the primary key field in Table A (I’ll refer to it as A.PK) is entered by the user so I can easily ask them to enter that value as a parameter in the queries which makes doing this a little easier in my particular case.  However, if A.PK is not available to the user as a data entry value the approach I’ve used in explaining the Table C and B interaction should have viability.

Table.Field references used in this description:

A.PK - User entered primary key, and left to the data entry user to ensure it is unique
A.DESC – Description field (user updateable after copied from another record)

B.FK – Foreign key tied to A.PK
B.PK – Primary key (AutoNumber)
B.PKOLD – Field to contain the value of B.PK from where the current record was copied from
B.DESC – Description field (user updateable after copied from another record)

C.FK – Foreign key tied to B.PK
C.PK – Primary key (AutoNumber but not referenced in SQL queries)
C.DESC – Description field (user updateable after copied from another record)
C.BFK – Non-key duplicated value from B.FK
C.BDESC –Duplicated data from B.DESC
C.PKOLD – Field to contain value of B.PK from the record that was originally copied.

Once the fields are setup 4 SQL queries performed in sequential order can copy all the records that are associated with one Table A entry and maintain referential integrity.

Query 1 (Copy Table A entry)

INSERT INTO A ( DESC, PK ) SELECT A.DESC, [Enter New A.PK Number]
FROM A
WHERE A.PK=[A.PK to Copy From];

Query 2 (Copy Table B entries associated with the newly created Table A entry)

INSERT INTO B ( DESC, FK, PKOLD ) SELECT B.DESC, [Enter Same New A.PK Number from Query 1], PK
FROM B
WHERE B.FK=[B.FK to Copy From];

Query 3 (Copy Table C entries without copying the foreign key)

INSERT INTO C ( DESC, BFK, BDESC, PKOLD ) SELECT C.DESC, [Enter Same New A.PK Number from Query 1], B.DESC, B.PK
FROM C INNER JOIN B ON B.PK = C.FK
WHERE B.FK=[ Enter Same New A.PK Number from Query 1];

Query 4 (Update foreign key in the copied records of Table C)

UPDATE C INNER JOIN B ON C.PKOLD=B.PKOLD SET C.FK = B.PK, C.BDESC=B.DESC;
