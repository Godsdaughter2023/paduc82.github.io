scottgem say:
An ambiguous join can occur when you mix inner and outer joins in the same query. 

Clearly you are doing that, by using an outer join between dte and an inner join between ID. You are not getting a false error, I'm not surprised at all at the error. 

So you need to better define the relationship between AllDates and Availability. Or you may need to use an interim query. First, have a query that joins on ID, then use that query to feed one that joins on dte.

Creating an attendance log - Microsoft Community

ordnance1 say:

Creating an attendance log

We have 200 employees and 8 supervisors and I need to create an attendance log to track absences.

I have created tbl_EmployeeNames, tbl_Supervisors, tbl_SupervisorsAssinments, tbl_AbsenceCodes and tbl_AttendanceHistory.

I am not sure how to handle the Suoervisor assingment since occasionally we will reassign employees to different supervisors but in my tbl_AttendanceHistory I want to see who the supervisor was at the time of the absence.

ken sheridan say: The foreign keys are in the tbl_SupervisorHistory table, which should have two columns EmployeeID and SupervisorID say, both of which should be of long integer number data type, not autonumbers in this case.  Columns are not defined as foreign keys in the table design, but by means of the creation of the relationships.  There are two relationships between tbl_EmployeeNames and SupervisorHistory , one from EmployeeID in the former to EmployeeID in the latter, the other from EmployeeID in the former to SupervisorID in the latter.  These are created separately by adding the tbl_EmployeeNames table twice to the relationship window in the way Karl described.  In each case enforce referential integrity.

The primary key of tbl_SupervisorHistory is a little more unusual than the normal single-column primary key you've probably been used to.  In this case the primary key is made up of three columns, EmployeeID, SupervisorID and AssignStart.  These three columns must, in combination, have distinct values because an employee can only be assigned once to a supervisor on any one date.  They may be assigned to the same supervisor on different dates of course, after a period assigned to a different supervisor.  To create multi-column primary key in table design view select each column while holding down the Ctrl key; all three should then be highlighted.  Right click and select Primary Key from the shortcut menu.

For data entry purposes an appropriate interface would be a form, in single form view, based on tbl_EmployeeNames, and within that a subform, in continuous forms or datasheet view, based on tbl_SupervisorHistory.  Link the subform to the parent form by setting the LinkMasterFields and LinkChildFields properties to EmployeeID.

In the subform you need controls for SupervisorID, AssignStart, AssignEnd and any other columns you've put in this table such as Comments.  You don't need a control for EmployeeID as that column is automatically given the correct value behind the scenes by the linking mechanism.  For the SupervisorID control use a combo box set up as follows:

ControlSource:    SupervisorID

RowSource:     SELECT EmployeeID, FirstName & " " & LastName FROM tbl_EmployeeNames ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first dimension is zero to hide the first column and that the second is at least as wide as the combo box.

For the other columns use text boxes.

To assign a supervisor for an employee you simply enter a new row in the subform, selecting the supervisor's name from the combo box's drop down list, and insert the AssignStart date (this will default to the current date if you've set its DefaultValue property to date() as Karl suggested.  When an employee changes supervisor you enter a value into the AssignEnd control in the latest row in the subform, and insert a new row in the subform for the new supervisor.

That handles the assignment supervisors, but not the attendance log for employees.  If you are recording absences only, rather than attendances you'd firstly have an tbl_AbsenceTypes table with columns AbsenceTypeID (autonumber primary key) and AbsenceType (text, e.g 'Sick leave', 'Annual leave', 'Unauthorised absence' etc).  This is then related to tbl_EmployeeNames by another table, tbl_EmployeeAbsences with foreign key columns EmployeeID and AbsenceTypeID, along with columns such StartDate, EndDate, and any others you want to include to record attributes of an employee's incidence of absence.  For data entry into this table you could use a form which includes a combo box to select the employee, set up similarly to that described above, another similarly set up combo box for AbsenceTypeID, along with text boxes for the other columns.  To show the employee's current supervisor in this form you could include an unbound text box control which looks up the supervisor from a query which joins tbl_EmployeeNames to tbl_SupervisorHistory on SupervisorID, with a criterion of IS NULL on the AssignEnd column (earlier supervisors of the employee will have a date in this column, so this will exclude them and show only current supervisors.  The ControlSource for this text box would be along these lines:

=DLookup("FirstName & " " & LastName", "qryCurrentSupervisors", "EmloyeeID = " & [EmployeeID])

Note that the qryCurrentSupervisors query should include the EmployeeID column from tbl_SupervisorHistory, not that from tbl_EmployeeNames.
