The six macros all call the autotext entries from the template attached to the document. Autotext entries cannot be stored in documents.

You refer to building blocks. These can include autotext entries and the default building blocks are stored in a specific building blocks template, but it seems improbable that the document was based on this template, so whatever you have done with building blocks does not address the original problem.

If you want to be able to distribute the new template with the macros then you are going to have to create or copy the autotext entries in question in that template and create new documents from that template.

Your default building blocks template will not be available to other users of the template so it is no use storing them there.

In my case I repeated the tests (sending the same attachment from Outlook 2010 and from OWA) to Outlook.com and having both delivered to the Outlook.com mailbox but only the email sent from OWA being pushed to device by EAS.
I've checked both emails (full headers) looking for diferences and found that the attachment sent with Outlook 2010 is "“Content-Type: application/ms-tnef; name="winmail.dat"" while the email sent from OWA the same attachment is sent has "“Content-Type: text/plain; name="quadro branco.txt"”.

I've tested the solution:
Delete the recipient's email address from the Auto-Complete List
 
1. Create a new mail.
2. Type the first few characters of recipients email-address.
3. While typing the first few characters, recipients address comes up on the drop-down list.
4. Point your cursor on that email-address and hit delete.
5. Then type the email address again.
6. Attach the file/s and hit send.

And can confirm that it resolved my problem :)
Maybe you can test and see if it will work for you too.
I admit it is a "strange" solution, but it worked for me. I've tested sending several types of attachements and all got pushed to my devices through EAS.

Hope it works for you too.
