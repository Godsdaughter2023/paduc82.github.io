scott mcdaniel say:

From what I can gather, you're referring to a sort of EAV system (Entity - Value - Attribute), and that always makes me cringe :)

EAV systems allow you to store/relate specific attribute values for each distinct entity, and you can pick and choose the attributes that comprise (or are associated with) a specific entity. You'd have a table that would store the Attributes, along with information about those Attributes (DataType, Size, DefaultValues, ValueRange, etc), and you'd have other tables that store details about the object that requires those Attributes, and you'd then have a 3rd table that would join all that together.

In your case, if Entity1 needs Attributes A, B and C, you can store those in your Join table(s). If Entity2 needs Attributes A, C and D you would store those. A common example of an EAV system would be a system storing data about Patients and their Afflictions/Diseases/Conditions. A Patient with Chron's Disease would not share many symptoms with a Diabetic, for example.

The troubles with EAV systems comes with user interface constructs and queries, since the data is stored in more of a "vertical" fashion. Often you must flatten that EAV data into temp tables in order to build a UI, and that can be very, very time consuming (and does NOT fit well into the Access environment, at least in most cases).

It's also very difficult to report on this sort of data, since you don't have any specific columns on which you can build your reports. You typically end up with reports that reference "ColA", "ColB", etc, and you create queries that output data in the same fashion.

I am currently involved with a client who is using an EAV system for a very popular online e-commerce solution, and it is maddening trying to figure out where and how to get data.

Which leads me to this:

In my mind, if you only have a handful of attributes (a dozen or less, perhaps) that aren't shared, then a standard relational database setup would be preferred. Your Application table, for example, would contain all columns needed by all Applications, but you would of course only fill in those that are needed by the specific applicant. While I'm no fan of have unneeded columns in a table, you could argue that all columns are needed - just not by every Applicant.

You also have to think about potential "attribute creep" - for example, are these attributes set in stone, or are they subject to be changed, added or removed? If so, then adding them as columns is a non-starter.

You can use a UDF.
 
To enter this User Defined Function (UDF), <alt-F11> opens the Visual

Ensure your project is highlighted in the Project Explorer window.

paste the code below into the window that opens.
 
To use this User Defined Function (UDF), enter a formula like
 
=SumNums(A1:A5,-B1)
 
 in some cell.
 
========================
Option Explicit
Function SumNums(ParamArray nums() As Variant) As Double
    Dim i As Long, j As Long
    For i = 0 To UBound(nums)
        If IsArray(nums(i)) Then
            For j = 1 To nums(i).Count
                SumNums = SumNums + Val(nums(i)(j))
            Next j
        Else
            SumNums = SumNums + Val(nums(i))
        End If
    Next i
End Function

charles kenyon say:

Any time there is a TOC field in the selection, you should get the dialog when you press F9. If the insertion point is in the TOC, the TOC field is in the selection. If you press Ctrl+A the TOC field is in the selection.

This is true even if there have been no changes to the document or if there have been heading added/modified. If the TOC field is locked, you will not see the dialog and the field will not be updated.

How to create a table of contents in Microsoft Word by Shauna Kelly

Generating a Table of Contents - Complex Documents

Update since this is marked as an answer:

If a heading paragraph is deleted completely (not just changed) the TOC will update (entire table, not just page numbers) without showing the dialog.

This does not occur when adding a heading. It may be because page numbers alone cannot properly update when a heading is deleted.

Also, if the heading is changed to a non-heading style, the dialog will pop up. If you choose page numbers only, the entry will remain in the TOC even though no longer a heading. If you choose to update the entire table, the entry will disappear from the TOC.
