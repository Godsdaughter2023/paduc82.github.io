Open A Windows Folder From A Form In Access 2010 - Microsoft Community

terryjenkins_790 say:

Open A Windows Folder From A Form In Access 2010

Does anyone have code that will open a Windows folder?  I want to program a button on a form in Access that when clicked will open a folder to a specific location.

Any help would be appreciated.

ken sheridan say:

Ken:  I apologize in advance, but I am very new to VBA.    Would I add the code above to the "On click" property of the button on my form to create an Event Procedure (Private Sub)?

If so, where do I add the "call the ShellToFile" portion above?

No need for any apology.  We were all new to this once.

The Windows API ShellExecute function and the FileDialog property of the Access Application object actually do two different things.  The former does the same as double clicking on a file or folder in Windows explorer, i.e. it executes the file in its associated application or opens the folder; the latter opens a file dialogue in which one or more files can be selected and returns the path(s).  I'm assuming from your use of the word 'open' that what you want is the former, to open the folder from a known path (which might have been obtained with the FileDialog property of course) in Windows Explorer.

If my assumption is correct, what you need to do is:

1.  Open the VBA window by pressing Ctrl-G..

2.  On the VBA menu bar select Insert | Module.

3.  A new standard module will open with the following two lines already in place:

Option Compare Database
Option Explicit

4.  Delete those two lines and paste in the whole of the code I gave you, from ' basShellExecute Module down to End Sub, into the module window.  You'll need to select whole module view from the buttons down at the bottom left of the module window to see all the code at once, but that's not really necessary; don't worry if you only see the ShellToFile procedure after pasting.

5.  Save the module as basShellExecute.  The actual name is immaterial so long as it doesn't duplicate any other object name.  I follow the long standing convention of prefixing module names with 'bas' to differentiate them from other types of object.

6.  You can now call the ShellToFile procedure from anywhere in the database, so if you have a button on a form you might put the following in its Click event procedure:

    ShellToFile "F:\MyFolder\MySubfolder"

ken sheridan say:

Ideally, I'd like to display "No active projects" in the Detail Section when there isn't a project assigned.

That shouldn't be difficult.  First add a text box control to the detail section, suitably positioned superimposed on top of the other controls and name it txtNoProjects.  Set its Controlsource property to ="No active projects"

Then put this code in the detail section's Format event procedure:

     Dim ctrl As Control
     Dim blnNoProjects As Boolean
     
     blnNoProjects = IsNull(Me.Project)
     
     Me.txtNoProjects.Visible = blnNoProjects
     
     For Each ctrl In Me.Section(acDetail).Controls
        If ctrl.Name <> "txtNoProjects" Then
            ctrl.Visible = Not blnNoProjects
        End If
     Next ctrl

This assumes that a bound control in the section is named Project.

ken sheridan say:
I tried your code and the same problem occurred as mentioned in my reply to Scottgem.  The "IF" conditional screws something up where I can't view or print

From your description of the set-up there is no apparent reason why that should happen.   Are you receiving any error message?  Post back with the full code you have used for the event procedure, including first and last lines which Access automatically inserts.  In the case of my solution, you should of course have amended this line:

    blnNoProjects = IsNull(Me.Project)

to:

    blnNoProjects = IsNull(Me.ProjNum)

However there is a more fundamental problem with your database design.  Firstly, you are using a personal name as a distinct identifier.  Names are unsuitable as keys as they can legitimately be duplicated.  Secondly,  you are recording the absence of a project for a technician by a row with Nulls in the ProjNum etc columns.  This is not how a relational database works, Projects should be a separate table whose primary key is referenced by a foreign key in the table which models the many-to-many relationship type between Techs, Tasks and Projects.  The table requires normalization by decomposition into a set of related tables:

Techs
....TechID (PK)
....FirstName
....LastName
....etc

TechProjects
....TechID (FK)
....ProjNum (FK)
....TaskNum (FK)
....CompleteDate

Projects
....ProjNum (PK)
....ProjDescr

Tasks
....TaskNum (FK)
....Task

PK = Primary key; FK = Foreign key. The TechProjects table is 'all key', i.e. its primary key is a composite one of all four columns.

A possible alternative model would be for TechProjects to reference Tasks only, not Projects, and for Tasks to reference Projects.  This would be the correct model if a type of task relates to one specific project, rather than being applicable to multiple projects.

The absence of any projects for a technician would be represented by the absence of any rows in TechProjects with that technician's TechID value.  To report on all technician's and their projects, including those technicians with no projects, a report should be based on a query which joins the tables, using outer joins to ensure that all rows from Techs are returned regardless of whether there is a match in TechProjects  or not.  The result table of this query will be the same as your current base table, so the solution for showing the absence of projects for a technician in the report would be identical to that described earlier.
