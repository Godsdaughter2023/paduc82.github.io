midders3 say:

I think you've misunderstood the workaround slightly:

- start the Map Loader normally and select PC Suite mode on your phone

- select the maps you want and click download

- use windows search to find a folder called download_temp (in c:\windows\temp on my system, but may be elsewhere)

- wait for the download to finish and start to load to your phone

(I've recently found out that if you wait for it to fail, it will have already deleted the download folder)

- either make the download_temp folder read only, or copy the .zip file (17210.zip for UK maps) to another location

- close the Map Loader or wait for it to fail

- reconnect your phone in mass storage mode

- unzip the .zip file and copy the 0,1,2...f folders to \CITIES\diskcache on your memory card

 

The memory card location used to store the maps seems to have changed with different versions of the map software, so have a browse through your memory card to find where the current 0,1,2...f folders are located before copying.

pete_uk say:

Rearrange your table so that it is in ascending order, like this:

1.00E-18 a 1.00E-15 f 1.00E-12 p 1.00E-09 n 1.00E-06 u 1.00E-03 m 1.00E-02 c 1.00E-01 d 1.00E+01 e 1.00E+02 h 1.00E+03 k 1.00E+06 M 1.00E+09 G 1.00E+12 T 1.00E+15 P 1.00E+18 E

I put this in A1:B16, but you could put it anywhere that's convenient for you. Then, using A20 as the number you wish to convert, put this formula in B20:

=IF(A20="","",TEXT(A20/VLOOKUP(A20,A$1:A$16,1),"0.0\"&VLOOKUP(A20,A$1:B$16,2)))

This will give you 1 decimal place, but if you don't want any (as in your example) you can change the "0.0\" in the middle to "0\". Note that this returns a text value, so you can't do any arithmetic directly on it.

Hope this helps.

Pete

The probability of SHA-1 collision is as below:

Assuming that SHA-1 behaves like a "random oracle" (a conceptual object which basically returns random values, with the sole restriction that once it has returned output v on input m, it must always thereafter return v on input m), then the probability of collision, for any two distinct strings S1 and S2, should be 2^(-160). Still under the assumption of SHA-1 behaving like a random oracle, if you collect many input strings, then you shall begin to observe collisions after having collected about 2^80 such strings.

(That's 2^80 and not 2^160 because, with 2^80 strings you can make about 2^159 pairs of strings. This is often called the "birthday paradox" because it comes as a surprise to most people when applied to collisions on birthdays. See the Wikipedia page  on the subject.)

Now we strongly suspect that SHA-1 does not really behave like a random oracle, because the birthday-paradox approach is the optimal collision searching algorithm for a random oracle. Yet there is a published attack which should find a collision in about 2^63 steps, hence 2^17 = 131072 times faster than the birthday-paradox algorithm. Such an attack should not be doable on a true random oracle. Mind you, this attack has not been actually completed, it remains theoretical (some people tried but apparently could not find enough CPU power ). Yet, the theory looks sound and it really seems that SHA-1 is not a random oracle. Correspondingly, as for the probability of collision, well, all bets are off.

so even assuming the most libveral interpretation here, there is a 1 in 9,223,372,036,854,775,808 or over 9 quintillion chance that there will be a collision, or matching SHA-1 values.  In other words, it is VERY unlikely that the file is corrupt.
