Use a Tab Control to Enter Multple Records to Support Transfers - Microsoft Community

sloopjohns say:

Use a Tab Control to Enter Multple Records to Support Transfers

I have a subform that I use to record material transactions.  Many transactions are transfers of material from one person to another.  This requires two records (a debit and a credit) to complete.  Id like to incorporate Access' Tab Controls to make this an easier process.  My goal is to have the TO tab be the default and have the user populate one record using that tab.  The second record is created using the second tab (FROM) and only in the event of a transfer.  I also hope to have the quantity related data auto populate as a negative amount equal to the amounts entered in the TO tab.

I've tried this but since I am not proficient at making it two separate records, I am seeing data mirrored.

scottgem say:
What does your table structure look like?

ken sheridan say: Place a separate subform on each page of the tab control.  Base the first on a query which restricts the rows to credits, and the second which restricts the rows to debits.  As you appear to be entering the values in to a single column as positive or negative values the criteria would therefore be >0 and <0 respectively.

In the AfterInsert event of the first subform set the DefaultValue property of the Quantity control in the second subform to the value of the Quantity control in the first subform multiplied by -1.  A control in a subform is referenced via the Form property of the subform control in the parent form's Controls collection, so the code would be along thses lines:

Me.Parent,SubformControlName.Form.Quantity.DefaultValue = """" & Me.Quantity * -1 & """"

Note that the DefaultValue property is always a string expression, regardless of the data type of the column to which a control is bound, so the value assigned to it should always be wrapped in literal quotes characters as above.

If you wish to use the subforms only for inserting new rows, then set the DataEntry property of each source form object to True (Yes).  Otherwise each will show all credits or debits respectively, and you'll need to navigate to an empty new record in each to insert the data.

The above will work if the user moves to the second subform on the second page of the tab control after inserting the record in the first subform, but will not enforce this.  A better approach with double-entry systems like this is to use an unbound form in which both transactions are entered into unbound controls, and then to execute two INSERT INTO statements in code in the form's module, executing both within a single Transaction.  This allows firstly for the incorporation of validation code before the transaction is executed, and for the Transaction to be rolled back in the event of one of the INSERT operations failing, thus avoiding the table being left in an inconsistent state.
