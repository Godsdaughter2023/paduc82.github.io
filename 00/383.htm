ken sheridan say: You have to identify the entity types, the relationship types between them and the attributes of the entity types, ensuring that the latter are determined solely by the key of the table modelling the entity type.

Customers are no problem.  They constitute a straightforward entity type with attributes such as FirstName, LastName, PassportNumber etc.  The last being distinct, could be the primary key, but most people would use an autonumber CustomerID as a surrogate key.

Designations are an entity type similarly easy to define, e.g. DestinationID, DestinationName.

Tours are not so straightforward.  Is a tour a fixed product, i.e. could there be a tour to Rome, Florence, and Venice for which there might be 3 or 4 separate instances in a season.  If so you'd have a Tours table with columns TourID, and TourName, and a table to model the many-to-many relationship type between Tours and Destinations by resolving it into two one-to-many relationship types:

TourDestinations
....TourID (FK)
....DestinationID (FK)

The primary key of this table is a composite one of both columns.  Each instance of a tour would then be modelled by a table like this:

Hotels used would be a table like this:

Hotels
....HotelID (PK)
....HotelName
....DestinationID (FK)

Each instance of a tour would then be modelled by a table like this:

TourInstances
....TourID (FK)
....StartDate

Again the primary key is a composite one of both columns.

and the TourDetails by:

TourInstances
....TourID (FK)
....StartDate (FK)
....DestinationID (FK)
....HotelID (PK)
....ArrivalDate
....DepartureDate

Note that this table has two overlapping foreign keys:  TourID/StartDate referencing the key of TourInstances and TourID/DestinationID (FK) referencing the primary key TourDestinations.

If a customer books a tour as a fixed package, i.e. they book a tour starting 1 July 2014  to Rome, Florence, and Venice then there is a many-to-many relationship type between Customers and TourInstances.

TourReservations
....TourID (FK)
....StartDate (FK)
....CustomerID (FK)

So far the model is quite simple.  When it comes to costs things might begin to get more complex.  At its simplest you'd have a TotalCost column in the TourInstances table, and as the Customers table is related to this by the above table the total cost is available through the relationship.  On the other hand, does the application need to compute the costs, which could involve differential tariffs at the hotels dependant on the time of year, type of room booked etc.  These things would need to be built into the model.

Modelling payments is not usually too difficult as each payment is related to TourReservations, so the Payments table would be along these lines:

Payments
....TourID (FK)
....StartDate (FK)
....CustomerID (FK)
....PaymentType
....PaymentDate
....PaymentAmount

Note that PaymentType could be a refund, which would then be used to sign the value as negative in any queries used to compute the total amount paid or outstanding.

I must stress that the above is only a simple schematic example of the sort of ways in which your model might be built up.  It makes assumptions which might not be warranted and omits many details, e.g. room types/costs per hotel are not modelled, so there would still be a lot of work involved in developing a robust operational database.  I'm rather surprised that you are using Access at all for this; I'd have thought that there would be purpose built software available to the travel trade for this sort of thing, which would have functionality very difficult, if not impossible, to achieve in Access.

My current animal shelter management database has 172 tables. It's only confusing if the underlying reality is confusing (which of course it is!)

 

A properly normalized database is less confusing, because it's made up of logical, modular pieces; each piece - each entity, each table - has its own existance, and is logically related to other pieces. Your current spreadsheet (it's not really a table if it has 255 fields!!!) is confusing because it's just all sorts of different things poured into a pot and mixed up; it's only clear to you because you're familiar with it, and have a whole bunch of useful knowledge about it encoded in neurons in your brain.
