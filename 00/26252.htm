ken sheridan say: I think it would be helpful if you could give us a few more details of what, in real world terms, you are attempting to model.  A table whose only attribute is a column of date/time data type would, unless the table is an auxiliary calendar table, be unusual.  Other non-key columns would normally be expected.  You might for instance have a table of events like this:

Events
….EventID  (PK)
….Event
….EventDate

Note that a surrogate autonumber primary key is used in the above table.  If the values in the EventDate column must also be distinct, i.e. the column is a candidate key, then the EventDate column should be indexed uniquely.  A further table of participants at events would be like this:

EventParticipants
….ParticipantID  (FK)
….EventID  (FK)

This table, in addition to referencing the primary key of Events by means of the foreign key EventID, also references the primary key of a Participants table, so it is modelling a many-to-many relationship type between Events and Participants by resolving the relationship type into two one-to-many relationship types.  The primary key of the table is a composite one of the two foreign keys.  It might well have other non-key columns each representing an attribute of the participant's participation in the event.

For data entry in this case a subform, in continuous forms or datasheet view, would be based on the EventParticipants table, and embedded in a parent form, in single form view, based on the Events table.  The subform would be linked to the parent form on EventID, and would contain a combo box bound to ParticipantID, along with other controls bound to columns representing attributes the participant's  participation in the event.

ken sheridan say: As I suspected you have a many-to-many relationship type.  In your case it is between what you term the 'main' table and a Mixes table, so the relationship type would be modelled by a table which references the primary keys of each.  I would not recommend using the date as the primary key of the 'main' table, however, but give it a surrogate autonumber key, and reference this by one of the two foreign keys in the table which models the relationship type.  Also, you do not need to give the table an autonumber key, as the two foreign keys in combination are a candidate key, and can this be made its composite foreign key.

In a simple binary many-to-many relationship type the table modelling the relationship type would have the two foreign keys only, along with other non-key attributes of the relationship type.  A quantity column in an OrderDetails table is an example.  Columns from the referenced table, Mixes in your case, would not normally be repeated in the table modelling the relationship type, as this would introduce redundancy as a result of the transitive functional dependency, and the table would not be normalized to Third Normal Form.  In your case, however, you say that values from the Mixes table can legitimately be varied.  You have two options to cater for this:

1.  You can treat each departure from a standard mix as a customized instance of that standard mix.  This would mean that columns from the referenced table can legitimately be repeated in the table modelling the relationship type, as in each table those columns would be functionally determined by the table's key, so there would be no redundancy and the table would be normalized to 3NF.  This is analogous to a row in an OrderDetails table (Northwind is an example) in which the UnitPrice of a product is a column in both the Products table and the OrderDetails table, reflecting the fact that the former will change over time, but the latter should remain fixed as the price when the order was created.  The value of the UnitPrice column from the Products table is assigned to the UnitPrice column when a product is selected in an OrderDetails subform

In your case you would assign the current values from the relevant columns of the selected row in the Mixes table to columns of the same name in the table modelling the relationship type.  This would be done in code in the AfterUpdate event procedure of a Mix combo box.  The values can then be edited as necessary.

2.  The other option is to have only the two foreign key columns in the table modelling the relationship type, and in the NotInList event procedure of the Mix combo box open a form to create a completely new mix record.

For an example of assigning values from a referenced table as in 1 above take a look at InvoicePDF.zip in my public databases folder at:
