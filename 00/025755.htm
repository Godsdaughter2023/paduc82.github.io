jllatham say:

Do you reference that code in any worksheet cells anywhere?  Or is it possibly referenced in another function that's declared as Volatile?  That might explain things.

 

But the process itself is darned fast!  I compared it to a couple of other routines that I have available to do conversion from a column number to get the column letter ID, and it was quite frankly, the fastest of the 3 (but the other 2 can handle 3-letter column IDs also).

 

I ran the tests in Excel 2003 and 2010, and they ran very slightly faster in 2010 (which kind of surprised me).  I have a tablet with 2013 on it, and I may run the tests on it to see if 2013 makes that much of a difference.  I'll post back if I do that. [EDIT: i ran my tests on a Lenovo Yoga with an i5 and Office 2013 and I still didn't see anything of significance: while the total time to complete increased as expected with the slower CPU, it was still fastest of the 3 routines tested and definitely did not hang the system.  My Office 2013 install on it is a .msi install, not click to run.]

 

Actually, in your process, I can see possibly 1 way to very slightly improve it's performance:  instead of:

Char2 = Chr(iCol - 26 * Int(iCol / 26) + 65)

ConvertLetter = Char1 & Char2
 
why not just
ConvertLetter = Char1 & Chr(iCol - 26 * Int(iCol / 26) + 65)
 
That would skip creating the Char2 value and make it part of the final answer.  Or go direct with a single statement:
ConvertLetter = Chr(Int((iCol - 1) / 26) + 64) & Chr(iCol - 26 * Int(iCol / 26) + 65)
 
With that last change, the improvement is small but measurable:  1 to 702 iterations ran with an average time of 0.000364 seconds (10 test samples) as written, with the one-line method it was .00034563 seconds. an improvement of about .00002 seconds (on a 3.6GHz system).

Function ConvertLetter(iCol As Integer) As String
Dim Char1 As String
Dim Char2 As String
    Select Case iCol
        Case 1 To 26 ' Single letter columns (1 = "A" - 26 = "Z").
            ConvertLetter = Chr$(iCol + 64)
        Case 27 To 702 ' Double letter columns (27 = "AA" - 702 = "ZZ").
            Char1 = Chr$(Int((iCol - 1) / 26) + 64)
            Char2 = Chr$(iCol - 26 * Int(iCol / 26) + 65)
            ConvertLetter = Char1 & Char2
     End Select
End Function

jllatham say:

gasx,

There is an error in your code - although it shouldn't affect the problem you're experiencing regarding cut'n'paste.

 

It looks like you obtained the original code over at FreeVBcode - and they realized their error and published corrected code.  Basically your code skips the AA, BA, CA etc columns - but then puts that id out in the xZ column, so the real column AA is noted as AB, while AZ is noted as AA.  The code below corrects all of that.  It was a matter of doing a MOD vs a / action:

 

Function ColumnLetter(ColumnNumber As Integer) As String
  'from: http://www.freevbcode.com/ShowCode.asp?ID=4303
  If ColumnNumber > 26 Then

' 1st character:  Subtract 1 to map the characters to 0-25,
'                 but you don't have to remap back to 1-26

'                 after the 'Int' operation since columns
'                 1-26 have no prefix letter

' 2nd character:  Subtract 1 to map the characters to 0-25,
'                 but then must remap back to 1-26 after
'                 the 'Mod' operation by adding 1 back in
'                 (included in the '65')

      ColumnLetter = Chr(Int((ColumnNumber - 1) / 26) + 64) & _

       Chr(((ColumnNumber - 1) Mod 26) + 65)
  Else
' Columns A-Z
      ColumnLetter = Chr(ColumnNumber + 64)
  End If

End Function

 

It runs on par with your version of the code as far as performance goes, so no tragic hit there.
