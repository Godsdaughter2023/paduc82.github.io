Protecting Access Database - Microsoft Community

[poni] say:

Protecting Access Database

Is there any way to protect an access database to restrict other people from being able to edit it? I have the database on a shared drive, and I want to make sure that I am the only one that is able to edit it.

ken sheridan say: Take a look at the access security blog by Tom Stiphout et al at:

http://www.accesssecurityblog.com

This has details of how to implement security either by means of VBA or by means of Active Directory.

You'll note that in the section regarding making a form read-only for relevant users with VBA it does this by setting the AllowEdits, AllowAddtions and AllowDeletions properties to False.  In some forms, however, setting the AllowEdits  property to False will deny access to unbound controls such as a navigational combo box. To allow for this, a solution is to firstly set the Tag property of each bound control you wish to make read-only to DisableMe or similar.  The Loop through the form's Controls collection and lock And disable the controls for relevant users:

    Dim ctrl As Control

    For Each ctrl In Me.Controls
        If ctrl.Tag = "DisableMe" Then
            ctrl.Enabled = False
            ctrl.Locked = True
        End If
    Next ctrl

A simpler method of making a form read-only for all but yourself would be to use the same code, but rather than making it conditional on the selection in a log-in combo box make it conditional on the user's Windows log-in name not being yours.  You can obtain the current user's log-in name by installing the following module in the database:

' module basGetUser
Option Compare Database
Option Explicit

Declare Function GetUserName Lib "advapi32.dll" Alias "GetUserNameA" (ByVal _
   lpBuffer As String, nSize As Long) As Long

Public Function GetUser() As String

   Dim strBuffer As String
   Dim lngSize As Long, lngRetVal As Long
   
   lngSize = 199
   strBuffer = String$(200, 0)
   
   lngRetVal = GetUserName(strBuffer, lngSize)
   
   GetUser = Left$(strBuffer, lngSize - 1)

End Function
' module ends

Let's assume that your log-in name is Poni.  The code in the form's Open event procedure would then be:

    Dim ctrl As Control

    If GetUser() <> "Poni" Then
        For Each ctrl In Me.Controls
            If ctrl.Tag = "DisableMe" Then
                ctrl.Enabled = False
                ctrl.Locked = True
            End If
        Next ctrl

        Me.Allowadditions = False
        Me.AllowDeletions = False
    End If

If you don't need users to have access to unbound controls like a navigational combo box, then rather than locking and disabling the tagged controls you can of course do as in Tom's blog and set the form's AllowEdits property to False.  Users will still be able to access other unbound controls such as command buttons of course.

You should of course ensure that users can access data only through forms, not directly in tables' and/or queries' datasheets.  This is true of any developed application, however, regardless of security issues.

amit tandon say:

You may try below formula: enter as an array formula - (CTRL+SHIFT+ENTER):

=SUM(ISNUMBER(SEARCH("*Ryan*",IF(OR(MOD(ROW($B$1:$H$100),15)<=1,MOD(ROW($B$1:$H$100),15)>=14),$B$1:$H$100)))*1)

This will count the occurrences of "*Ryan*" in the ranges B14:H16, B29:H31, B44:H46, B59:H61, B74:H76, and B89:H91 ie. every 3 rows starting from 14 and then a gap of 15 rows each. Hence MOD is used with the divisor of 15. You may update the formula easily for each employee accordingly.

will not retrieve anything unless you have BOTH defined the FirstName field's Allow Zero Length property to be Yes (it's No by default) and actually stored a zero length string in the field.

ken sheridan say: The WHERE clause should be:

WHERE (CustomerT.[First Name]=[Enter Customers First Name:]
OR Enter [Customers First Name:] IS NULL)
AND (CustomerT.Surname=[Enter Customers Last Name:]
OR Enter [Customers Last Name:] IS NULL);

Note that the parentheses are crucial to force the OR operations to evaluate independently of the AND operation.  You will find that if you switch to design view and save the query, Access will have moved things around next time you open it in design view.  The underlying logic is the same, but will be less clear.  Nevertheless, it's best with queries of this type to save them in SQL view.

As John said, referencing controls in a dialogue form as the parameters is a better solution, but the above will work.

ken sheridan say: One potential problem with that is if you have two or more customers with the same names.  I've frequently told the story here of the two patients who turned up at a clinic where I was present, both female, both with exactly the same first and last names, and both with the same date of birth!

My NotInList demo has a form which uses the following for a contacts list to differentiate people by address:

SELECT Contacts.ContactID, [FirstName] & " " & [LastName],
ConcatValues(", ",AddressLine1,AddressLine2,City,Region,PostCode)
FROM Regions INNER JOIN (Cities INNER JOIN Contacts
ON Cities.CityID = Contacts.CityID)
ON Regions.RegionID = Cities.RegionID
ORDER BY Contacts.LastName, Contacts.FirstName;

The address is concatenated using the following function, which is a variation of Microsoft's CanShrinkLines function:

Public Function ConcatValues(strSeparator As String, ParamArray arrVals())
 
    ' Pass this function a character or characters
    ' to be used as the separator followed by the values to be combined
    ' For example: strFullName =
    ' ConcatValues(" ",FirstName, MiddleName, LastName)
    Dim X As Integer, strLine As String
    
    For X = 0 To UBound(arrVals)
        If Not IsNull(arrVals(X)) And Trim(arrVals(X)) <> "" Then
          strLine = strLine & strSeparator & arrVals(X)
        End If
    Next
      
    ' remove leading separator character(s)
    ConcatValues = Mid(strLine, Len(strSeparator) + 1)
 
End Function
