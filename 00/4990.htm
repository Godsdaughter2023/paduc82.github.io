    With Me.SubformControlName

scottgem say:

First, The members name should NOT be in the donations table! One of the principles of a relational database is that data exists in one place and one place only. You should have a Members table with a ID field (preferably an autonumber). That value should then be used in your Donations table as a foreign key.

Second, I would have a mainform/subform combination with the mainform bound to the Members table and the subform bound to the donations table and linked on MemberID. The subform will show all donations for the member currently displayed on the main form. You can add a search combo (3rd option) using the combobox wizard to allow you to easily select a member.

So there is really no need for a query such as you describe. 

I almost never use parameter prompt queries because they are so limiting. Instead I will use a form to enter criteria using list controls etc. then enter the criteria in the query using the syntax:

Forms!formname!controlname

A little more info about the Sandisk SSD:

This thing works a lot like the old Promise Disk Controllers, With those, there were some memory slots on the disk controller board. You could populate the controller with one or more of the same memory sticks used on the motherboard. The result was a much faster system overall because it kept the most used sectors in the ram. Since it works on a sector layer, instead of files, it even cached parts of the ram that were paged out to disk. The disadvantage was that the memory was volatile, so all your cached data was lost when you powered off. The Sandisk ReadyCache is non-volatile, and the firmware even caches starting at boot, before the OS loads, So boot time is improved. Mine went from 82 seconds to 21 seconds booting Windows 8.1

Another advantage is that the  trim is handled in the SSD (ReadyCache) firmware. And , since you don't want to run Defrag on a SSD, the ReadyCache firmware defrags the HD it is caching in the background whenever the SSD is inactive.
