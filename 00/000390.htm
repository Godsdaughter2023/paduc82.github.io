Insufficient information.  I get exactly zero in D6 when I follow your instructions (correcting my typo in C3):

E6 is the formula =D6 formatted as Scientific.  0.00E+0 is exactly zero.

My suspicion is:  the values in B2:C6 are not exactly integers, as you say they are.  For example:

By coincidence, that demonstrates in E6 the infinitesimal difference (well, the same magnitude) that you see in D6, perhaps because you format D6 as General.

The point is:  the root cause of the problem is the usual rounding anomaly that arises because Excel uses industry-standard native-computer 64-bit binary floating-point to represent numbers.

Consequently, most non-integers cannot be represented exactly, and the infinitesimal differences in their representation often leads to infinitesimal arithmetic differences.

In general, the remedy is:  when you expect arithmetic to be accurate to a specified number of decimal places, explicitly round to that number of decimal places.

In my example, the formulas in column D should be of the form:

=ROUND(A1+B1-C1, 2)

because I display 2 decimal places, and that is the accuracy that I expect.  You might choose a different number, for example zero decimal places.

[EDIT] Formatting alone alters only appearance, not the actual value (usually).

Since the Formula Bar also formats only 209528, if you "re-enter" B2, it will correct the infinitesimal error.

By "re-enter", I mean:  cause Excel to re-interpret the constant.  That happens if we select B2 and press F2, then Enter.

In any case, you might ask:  how could a constant have infinitesimal digits that Excel does not format?

That happens when a cell value is the result of a calculation, then we copy and paste-value.

Paste-value preserves the exact binary representation, at least until we cause Excel to re-interpret the constant.

That can also happen when the calculation occurs in VBA, and we store the result into an Excel cell with a statement of the form Range("B2")=.... .  For example:

Range("B2") = 209528 + 2.91E-11

although I'm sure you never did anything quite so obvious.

djsteele, mvp say:

You cannot put a variable into SQL like that. You have two possibilities:

  1. put the value into a text box on a form, and refer to that text box (as I suggested earlier)
  2. dynamically generate the SQL statement and replace the SQL associated with the query

If you're going to use option 1, you need to resolve the parameter before opening the recordset. That means that you need to add the following to the beginning of the query's SQL:

PARAMETERS Forms![NameOfForm]!txtViewingID Integer;

Your code would then need to be something like:

Dim rs As DAO.Recordset
Dim qdf As DAO.QueryDef
Dim prm As DAO.Parameter

  Set qdf = db.QueryDefs("qryExportPhoneNbr")
  For Each prm In qdf.Parameters
    prm.Value = Eval(prm.Name)
  Next prm
  Set rs = qdf.OpenRecordset, dbOpenSnapshot)

doug robbins - mvp office apps & services (word) say:
While the Normal template is a macro enabled (dotm) template, the default user templates folder is a trusted location so even it the template contains macros, starting Word will not cause the macro notification message to appear UNLESS, under File>Options>Trust Center>Trusted Locations, the user templates folder has been removed from the list of Trusted Locations OR the box for "Disable all Trusted Locations" has a check mark in it.
