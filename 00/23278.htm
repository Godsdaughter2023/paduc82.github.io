The macro in that article will need a small change to satisfy the requirements.

First, to make this work at all, the header that contains the logo must be a First Page header, not just an ordinary header in Section 1. That's because a user can insert a page break in the middle of page 1, and an ordinary header would be repeated on page 2.

If the logo is not currently in a First Page header, select everything from that header and cut it to the clipboard. In its place, insert whatever you would want to appear starting on page 2 if a user does insert a page break on page 1. Then, on the Header & Footer Tools tab, check the Different First Page box. Finally, paste the header contents into the First Page header.

Next, save your document as a Macro-Enabled Document (*.docm) instead of .docx. (A template would generally be preferable in some ways, but getting everyone across a large set of locations to use a template instead of re-using existing documents is usually a lost cause.)

Open the macro editor (Alt+F11). In the Project pane on the left, expand the document's project, expand the Microsoft Word Objects icon, and double-click the ThisDocument icon.

Copy the code from the article linked above, and paste it into the large empty space on the right side of the macro editor.

Now select and delete the bottom procedure, from the line that starts Private Sub wdApp_WindowSelectionChange and ends with the End Sub line, and delete it. In its place, copy and paste the following code:

Private Sub wdApp_WindowSelectionChange(ByVal Sel As Selection)
    'quit if active doc doesn't contain this code
    If ActiveDocument <> ThisDocument Then Exit Sub
    'get out of the Section 1 First Page header if we're in it
    If Sel.StoryType = wdFirstPageHeaderStory _
        And Sel.Information(wdActiveEndSectionNumber) = 1 Then
            ActiveWindow.ActivePane.View.SeekView = wdSeekMainDocument
    End If
End Sub

Unlike the original macro, this will allow editing of all headers (and footers) except the one specific header on Page 1 of Section 1.

jay freedman say:

To use the macro in a template (and remember, all the code from the web article needs to be in the same file), replace the line

    If ActiveDocument <> ThisDocument Then Exit Sub

with the one that was originally in the article's code,

    If ActiveDocument.AttachedTemplate <> ThisDocument Then Exit Sub

In both cases, "ThisDocument" contains the name of the file (template or document) that holds the running code. "ActiveDocument" is the name of the file that's currently visible and active on the screen. If the code is in a .docm file, then those two names should be the same -- if they're different, that statement prevents the code from locking some other document that just happens to be open at the same time.

When the code is in a template, "ThisDocument" holds the name of the template. You want to lock a document that's based on the template, so for that case "ActiveDocument.AttachedTemplate" holds the name of the base template; those must be the same to allow the code to continue.

I don't know why you're getting an error on printing. In fact, printing the document should not even start running that macro. The WindowSelectionChange event is supposed to occur only when the cursor (the "Selection") is moved within the document -- for example, if you click in a different word or paragraph. Clicking the Office button and Print, or pressing Ctrl+P, doesn't generate that event.

ken sheridan say:

The ClientID field already has data in it and cannot be switched by back to autonumbering. (Unless you know a fix for that?)

It's not difficult.  Firstly add an autonumber column, ClientID_New say to the table, and a corresponding long integer ClientID_New column to any referencing tables which include a ClientID foreign key column.   Then it's merely a case of updating the foreign key columns in the referencing tables with 'update' queries such as below for an Orders table for instance:

UPDATE Orders INNER JOIN Clients
ON Orders.ClientID = Clients.ClientID
SET Orders.ClientID_New = Clients.ClientID_New;

If, in any of the referencing tables, the ClientID foreign key column is part of a composite primary key which is referenced by a corresponding composite foreign key in one or more other tables, you would need to do the same for those tables, joining the table with the composite foreign key to that with the composite primary key on both columns and updating a ClientID_New column in the table with the composite foreign key in the same way as above.  In a complex model this process might have to be repeated down the line until the lowest level referencing table has been updated in this way.

Once the ClientID_New columns in all of the relevant tables have been updated the original ClientID columns can be deleted from the table definitions, and the ClientID_New columns renamed ClientID.  The relationships on ClientID (whether as the complete key or as a component of a composite key)  can then be recreated and enforced.

So I am making an inventory and transaction database for a user and am running into a problem with one of my subreports.

I have a form called PrimaryAccessForm where the user will be able to do all her daily tasks without having to worry about the underlying tables, queries, and reports. On that form I have a tab control with three tabs each showing a different subreport she needs to be able to look at. One of the tabs pulls in the recent transactions report with two text boxes to control the date range that the report will show. My problem is that when I change the boxes to specify a new date range it doesn't update the report unless I switch views or manually save and refresh, neither of which is intuitive for the user. Is there some way I can force the report to refresh after each time the text boxes change?

Thank you in advance for you help.

marshall barton say:

Since you are displaying reports on a form, it would probably be best to use Report view instead of Print Preview.  If the reports are using Report view, then you can simply requery the reports from the main form (using each text box's AfterUpdate event?):

   Me.subformcontrol.Report.Requery

Note that Report view acts more like a form than a report in the events you can use and there are no page related objects or methods.

There are unintuitive ways to get the same effect for a PrintPreview report, but I don't think they're needed now that we have Report view.
