It's a common problem. Many people are careless when entering information into forms at websites. They intend to type A but type B instead, and don't notice that they've done so.

I have an email address that consists of my surname @live.com - I was very quick off the mark when these addresses became available several years ago. Sometimes it's possible to work out the name of the person involved; they have included a Christian lawyer in Uganda, the secretary of a parish council in Kent, a prospective Amazon customer called Tessa, a bingo aficionado in Newcastle and the editor of a newsletter in Raleigh, N. Microsoft SHOULD know if someone else has the same email address as me.

No-one has the same email address as you do. It's quite impossible for two people to have the same [Microsoft] email address at the same time. However, the address belonging to an account that has lapsed on account of neglect may be made available to someone else, but not until a year has passed from when it became inactive. This scenario is theoretically possible:

  1. Joe Alvarez opens an account called joehotmail.com
  2. Joe uses this account as his username for an account at eBay
  3. Joe gets another email account at Gmail and finds that easier to use, so he stops using his Hotmail account
  4. Nine months later, his Hotmail account is suspended for inactivity.
  5. Three months after that, his account is purged - all mail messages, contacts and calendars belonging to joeho are deleted.
  6. A year later, Joe Badajoz decides to open an email account at signup.live.com and is pleasantly surprised to find that joehotmail.com is available..
  7. Joe B. receives regular mail messages from eBay with special offers, even though he's never had any dealings with eBay.

The upshot of all this is that the Email Service Provider is completely powerless in the face of user carelessness or stupidity. Microsoft can't prevent anyone using your email address e.g. to register a user account at a website, to sign up for a newsletter or to join a political party. The onus is unfortunately on you to inform the senders involved that you are not the one they think you are.

saataja say:
alpha_sen wrote:

Hi i am using Nokia E63. I want to use it to connect to my office wi-fi. I have downloaded the CA certificate to my phone from my laptop but it still asks for WEP key whereas while connecting thru laptop it doesnt ask for it.The details of the office WLAN are as below pls help

 

 Wireless Network Key Setting

By looking at the Windows PC targeted instruction shown above, your office WLAN network seems to be configured so that it requires usage rather rare and dated WLAN security method known as "dynamic WEP encryption" with 802.1X EAP authentication. Dynamic WEP is getting really rare nowadays since it has been obsoleted for years by more recent WPA/WPA2 EAP security mechanisms.

 

"Smart Card or other certificate" part of the instructions indicate that EAP-TLS (certificate based two-way authention between client and network) authentication type should be used when accessing this network.

 

Note that EAP-TLS authentication requires that you have installed both your "Personal Certificate" (User certificate) which must also include the private key of the personal certificate) and matching "Certificate  Authority"certificate on the phone before you can continue configuring the EAP-TLS.

 

"Problem" with the phone asking you to enter the WEP Key (when attempting to connect via WLAN wizard or Search for WLAN etc.) is because WLAN network configured like this (dynamicly created WEP encryption keys with 802.1X/EAP user authentication) can't really be separated from the regular "static" WEP key secured networks, when WLAN client device is scanning for available WLAN networks. Thus your phone assumes that this is a "regular" static WEP encrypted network and phone prompts user to enter the static WEP encryption key. Naturally user can not know what would be the correct WEP key since with dynamic WEP, keys will be automatically generated during the connection attempt, after EAP-TLS authentication phase has been successfully passed.

 

To make this work you should first make sure that you have installed both your "Personal" certificate and "CA" certificate on the phone.

 

Then you need to manually create an approriate Internet Access Point on to phone. You can create new or edit the existing "WEP" security mode IAP in case such exists on the phone after you have entered the static WEP key when prompted while attempting to connect via WLAN wizard.

 

To create or edit Internet Access Point settings manually go to Menu -> Tools -> Settings -> Connection -> Access Points. Here you need to define your WLAN access point settings so that:

 

Connection name: Name of the IAP (usually same as WLAN network name)

Data bearer: WLAN

WLAN network name: SSID (i.e. name) of the given office WLAN network

Network status: Public (unless it's instructed somewhere that your office WLAN is in "Hidden SSID" mode)

WLAN network mod: Infrastructure

WLAN Security mode: 802.1X  (note that WEP or WPA/WPA2 does not work here)

 

Then go to WLAN security settings and define following:

 

WPA/WPA2: EAP

 

Go to EAP plug-in settings and:

 

Enable EAP-TLS  (via Options -> Enable)

Disable all other EAP methods than EAP-TLS (note that EAP-SIM and EAP-AKA are enabled by default, you must disable these)

 

Go in to the EAP-TLS settings (center click or Options -> Edit)

 

Personal Certificate: Choose your Personal Certificate which you had installed earlier on the phone. Note that if you can't see any Personal Certificates available on the selection list then you haven't installed approriate Personal Certificate on the phone and EAP-TLS authentication simply will not work.

 

Authority Certificate: Choose the correspronding CA certificate you had installed earlier. Note that CA certificate must be the one that your office WLAN network has been configured to utilize (on the server side) and you Personal Certificate must be "signed" by this CA certificate for two way EAP-TLS authentication to be possible.

 

User name in use: From Certificate

User name: <leave this empty>  (username should be included and read automatically from your Personal Certificate)

 

Realm in use: From Certificate

Realm:  <leave this empty>

 

Note that there is a "Cipher" tab (if you go right) on the EAP-TLS settings but typically you should leave it with default settings and since the default set of "TLS ciphers" should work. Notet that these "ciphers" are specific to EAP TLS / PEAP /TTLS tunneling and not to be confused with WLAN WPA/WPA2 level encryption "ciphers" like TKIP or AES

 

Your newly created Access Point settings will be saved when exiting the Access Point configuration menus by selecting Back (multiple times).

 

If everything wen't well, you should now have a functional  EAP-TLS authenticated and dynamic WEP encrypted "Access Point" defined on your phone and when you select this particular Access Point (instead of using "Search for WLAN" ) the application requesting connectivity should hopefully be able  to connect via your office WLAN network.

 

Note that Personal Certificate usage during the EAP-TLS authentication (private key storage access) will cause your phone prompt for the private key store password during the first EAP-TLS authentication so be prepared to enter the private key store password, which you should have been asked to define while you were earlier installing your Personal Certificate (with included private key).

One potential reason why your personal certificate get's installed to the Trusted Site Certificates instead of Personal Certificates is that the certificate you installed did not contain the "private key", which is essential part of any personal certificate you are suppose to use for EAP-TLS authentication purposes.

 

I assume that you "exported" this certificate from your Windows laptop, meaning that certificate and accompanied private key were already installed on the Windows? 

 

Try to re-export your certificate from Windows so that select the option "Yes, export the private key" during the Windows Certificate Export Wizard. In case you wan't to continue using the certificate also on your laptop ensure that option called "Delete the private key if export is successful" is unselected. You should select the "Include all certificates in the certification path if possible"option and leave the"Enable strong protection" option unselected. Then define a password that protects the exported ".pfx" certificate container file and define a folder and filename ".pfx" file will be created.

 

If everything wen't well you should now have a ".pfx" file which contains your Personal Certificate with a private key (private key is essential here) and also a Certificate Authority root certificate relevant for your personal certificate. Note that you must have a proper CA certificate installed on the phone and selected on EAP-TLS settings for EAP-TLS authentication to work properly.

 

If you successfully exported everything then simply copy the "pfx" file to your phone. Then you should probably just in case remove the previously installed version of the personal certificate by going in to the phone's certificate manager (control panel->settings->general->security->certificate management->Trusted site certificates-> select the certificate-> options -> delete).

 

Then start installing the "pfx" file by opening the file via File Manager. You will be prompted for the password you defined while exporting from Windows. Now that your personal certificate should also include the private key, your phone will ask you to define a new "private key store" password so make up a password that you will remember later on (has to be at least 6 characters long). Then simply move forward with the certificate import wizard and your personal certificate should get installed first followed by any CA certificates that were included in the "pfx" file.

 

In case you had already previously installed the relevant CA certificate it shouldn't be a problem since phone should just skip importing the CA certificate within the pfx file in case the included CA certificate is already installed.

 

Note that depending on how security policy of your personal certificate has been defined at time of creation and/or how it was initially installed on your Windows PC, you might not necessarily be allowed to export the private key at all since it has been locked explicitly to your Windows PC at the time of installation. If this is the case then you can't make your phone work with EAP-TLS by exporting the existing and already installed personal certificate from your Windows laptop (since private key is not exportable). Instead of exporting you basically need to install the personal certificate and private key to your phone from the fresh (non-exported) "pfx" file, which you might be able to get from your network administrators.
