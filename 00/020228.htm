1.  Create a new form bound to the Assessments table.

Dim strSQL As String

ken sheridan say: There are actually two errors in the code.  One is that you have unfortunately used spaces in object names, Assessment ID in this case.  While Access unwisely allows this, the widely accepted view amongst experienced developers is that it should not be done.  Where an object name is a composite noun, either use CamelCase or represent a space by an underscore character, i.e. AssessmentID or Assessment_ID.  As well as spaces, special characters should also be avoided in object names, the # character being a common culprit.  If spaces or special characters are used then the object name must be wrapped in square brackets when referencing it, i.e. [Assessment ID], but best practice is not to use them at all.

The second error was mine (Mea culpa!).  I omitted a comma.  The line should be:

CurrentDB.Execute strSQL, dbFailOnError

The reason only the second error was reported is that the first would not have become apparent until the SQL statement was executed, which the second error was preventing from happening.

The need to preserve the 'historic' value of the requirements is a matter of functional dependency.  It's a not unusual scenario, the classic case being the UnitPrice of a product, where you want to retain the price at the time of an order or invoice, regardless of subsequent price changes.  I address this in a number of my on-line demos, but the basic mechanism is that the current unit price is obtained from the UnitPrice column in the referenced Products table and inserted into a UnitPrice column in the referencing OrderDetails table.  Your situation is analogous to this.

What you need to do therefore, is insert the either the current [Requirement Name or ID] or [Requirement Text] value into the Requirement column in Assessments, whichever to wish to preserve.  So if we assume the latter for this example, the SQL statement needs amending slightly as follows:

strSQL = "INSERT INTO [Assessment Items]([Assessment ID],Requirement) " & _
    "SELECT " & Me.ID & ",[Requirement Text] FROM Assessments " & _
    "WHERE RegulationID = " & Me.cboRegulation

As regards the relationships, you should create an enforced relationship between Regulations and Assessments.  This will ensure that the integrity of the database is protected.

One other recommendation I'd make is that you avoid the generic ID as a primary key name.  It creates ambiguity.  Instead us a name which more specifically describes the attribute, i.e. RegulationID, RequirementID etc.

The size of the Builds will vary, depending on the computer where they are installed.

The download will range between 2GB and 3.2GB depending on CPU architecture, language and other variables. There is no way that I'm aware of to determine the size prior to downloading.

There are also a couple of of more things to consider about disk space. Each time a new Build is downloaded and installed a Windows.old folder is created that is used to Revert back to the previous build if there are problems with a new build. The longer you use the Peview Builds, the larger this Windows.old folder. You should not consider using the Windows 10 Technical Preview is disk space may be a problem.

When a new build is released it will be installed similar to any other windows update. You will always get the latest build that is available.

Let us know if you have more questions.
