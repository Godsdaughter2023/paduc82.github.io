SOLUTION:

Private Sub Form_Current()
Dim rst As Object
Set rst = Me.RecordsetClone
On Error Resume Next
rst.MoveLast
On Error GoTo 0
Me.txtRecordCount1.Value = rst.RecordCount

'Test Record Count and open Data Entry form if zero
With Me.fmProcessSub1.Form
    If .RecordsetClone.RecordCount = 0 Then
       .AllowAdditions = True
     .TxtProcess1.SetFocus
     .TxtProcess1.BackColor = vbYellow
    'DoCmd.GoToRecord , , acNewRec
       Else
        .AllowAdditions = False 
        .DataEntry = False
        .TxtProcess1.BackColor = vbWhite
   End If
End With

a. user say:

If you insert an XML "data" file into Word, all you see is the data in the elements, and possibly some element tag markers. To make it appear the way you want it you either need to 
 a. write VBA code (say) that reads the XML file "as data" and inserts whatever you need into the document or
 b. write an XSLT transform that converts the XML file to WordProcessingML (Word 2003 format), then insert that file into your document.

The trouble with (b) is that you have to understand a lot about the WordProcessingML vocabulary even to get started. Once you have written the XSLT, you can use { INCLUDETEXT } fields to include the transformed XML, e.g.

{ INCLUDETEXT "the path name of the .xml" \t "the pathname of the transform .xsl" \c XML }

which might be an advantage in your scenario, but I think you will probably be better off with (a).

a. user say:

> Is there any new convenient way to insert XML code in Word and Excel ?

There is nothing new in the out-of-the-box version of Word for that. I really don't know on the Excel side - you may be better off asking a separate question. The main change in recent versions is the availability of  Microsoft's JavaScript Office APIs, which may have a bearing on what you want to do.

I am assuming that you are not looking for the kind of XML-related facility that Word used to have in the 2007 version, and for a short time in 2010, but which was removed following a court case, but rather an ability to edit some thing like an XML code block.

If so, it sounds as if what you are looking for would actually be satisfied by an XML Editor that is capable of acting as an OLE server, or perhaps as an ActiveX control that can be embedded in Word.

There are certainly things out there that do something like that - e.g. the commercial TX Text Control, but AFAIK not all ActiveX controls will work well when the container is a Word document surface. Still, it suggests that you could do most of what you want if you are are capable of writing your own complete COM server.

Even in that case, if you need your XML objects to be able to span pages in Word, your server would have to be able to render the object's contents as plain text, RTF, HTML or possibly OOXML (not sure about the latter), which means it would be difficult to prevent users from "accidentally" editing the text without using the server.

Other approaches you might be able to take (and I am really just speculating here) include...
 a. you find or implement a .NET Windows form control that can do the editing you require, and use VSTO to enable you to host it on the document surface. I believe VSTO actually uses Word's ActiveX Control feature to do that.

 b. you isolate the XML code in a rich text content control and try to use its events to control how the content is edited (my guess is that realistically speaking you would have to do something like detect a click in the control, copy the content to an editing pane somewhere, then copy it back when done).

 c. Like (b), but provide a separate editing panel and rely on the user to indicate that they want to edit the content. At one time, you could use things such as "Smart Documents" or Task Panes constructed in VSTO. I do not know what the status of those technologies is now. However, this is I suppose where the Javascript APIs might come in, because (1) they are the thing Microsoft is actually trying to push at the moment, (2) you have a chance of re-using the vast collection of JS code "out there", and (3) the technology is intended to be cross-platform (desktop Windows Word, desktop Mac Word, Online Word), although there is still a long way to go on that front. None of the other suggestions made so far would have a chance of working on anything except desktop Windows Word.

Finally, what about Word ability to "write back" text included with an INCLUDETEXT field to the original file? Well, unfortunately that only works when the file is in one of Word's "native" formats, and I think you always have to use a named converter to include a .xml file that you want to be treated as text.

a. user say:

> Is there any attempt to have structured text into Word ?  

> (I believe there is something).

I would say that there *used to be* something, as I said before, and there may be 3rd party things (in which case a search engine will likely give you more help than I can).

There's nothing in the product itself that will take a .xsd and present you with a form that would constrain you to enter insert text. etc. that conformed to that .xsd. There used to be a kind of facility to do that (the Document Information Panel), but that has now gone, was really tied to SharePoint use, and panel itself was presented by InfoPath, which is now deprecated.

What does still exist is the ability to insert content controls, individually or grouped inside rich text content controls, protect the document so that you can constrain editing to those controls, and associate the controls with an item (such as an element) in an .xsd that you have imported into the "Office Schema Library", in which case Word will do some validation on the content. It's not an area I have visited for a long time.

In Windows Word 2013 and later you also get "repeating content controls" which may help you create things such as task lists. But you are then talking about having a fairly fixed layout, e.g. not one which would present different elements depending on the task type, etc.
