macropod say:

One of the problems you can have with your approach (aside from the selection issues), is that you'll get only whatever highlighting is currently active in the document - including none.

The following code explicitly applies a bight green highlight to the double spaces:

Sub HighlightDoubleSpaces()
Application.ScreenUpdating = False
Dim i As Long
i = Options.DefaultHighlightColorIndex
Options.DefaultHighlightColorIndex = wdBrightGreen
With ActiveDocument.Content.Find
  .ClearFormatting
  .Replacement.ClearFormatting
  .Format = True
  .Style = wdStyleNormal
  .Replacement.Highlight = True
  .Text = "  "
  .Replacement.Text = "^&"
  .Forward = True
  .Wrap = wdFindContinue
  .MatchCase = False
  .MatchWholeWord = False
  .MatchAllWordForms = False
  .MatchSoundsLike = False
  .MatchWildcards = False
  .Execute Replace:=wdReplaceAll
End With
Options.DefaultHighlightColorIndex = i
Application.ScreenUpdating = True
End Sub

I note, too, that you called your macro 'ConvertDoubleSpaces'. If what you really want to do is to eliminate them, try:

Sub ConvertDoubleSpaces()
Application.ScreenUpdating = False
With ActiveDocument.Content.Find
  .ClearFormatting
  .Replacement.ClearFormatting
  .Format = True
  .Style = wdStyleNormal
  .Text = "[ ]{2,}"
  .Replacement.Text = " "
  .Forward = True
  .Wrap = wdFindContinue
  .MatchWildcards = True
  .Execute Replace:=wdReplaceAll
End With
Options.DefaultHighlightColorIndex = i
Application.ScreenUpdating = True
End Sub

This version of the macro will clean up not only double spaces, but triples, etc. as well

john w. vinson say:

One to one relationships are quite rare: unless you are intentionally Subclassing, or using "Join based field security" or a few other specialized methods, it's probably not needed. Is it an accurate statement to say:

Every SCHEDULE record may have either zero or one DESIGN records, and every DESIGN record must have one and only one SCHEDULE record.

If so, a one to one relationship may be appropriate; but an unenforced relationship is really no relationship at all, and will indeed block updatability.  A one to one relationship is still directional - there is a Parent and a Child table, it's just a one-child family; you could use a Form based on SCHEDULE and a subform based on DESIGN. Another option would be just to put the "optional" DESIGN fields into the Schedule table and allow them to be NULL. 

But it's pretty important to pull the repeating groups out into a table or tables of their own and explicitly recognize the one to many relationships, as Scott advises.

I am not sure how Anne's suggestions would help in this case but I imagine she will elaborate.

In the meantime, here's a tested multi-step solution.

Suppose the only *numbers* in col A are the numbers in your list.

Then, in some cell, say C1 (but you can pick any cell), enter 10%.

Next, in another cell, say D1, enter the formula =ROUND(C1*COUNT(A:A),0)  This gives you the number of entries, rounded to an integer, in column A that constitute 10% of the total.

Finally, in another cell enter the formula =SUMPRODUCT(LARGE(A:A,ROW(INDIRECT("1:"&D1))))  This gives you the sum of the 10% of the largest entries in col. A irrespective of where they are located.

gary's student say:

Sort is an exception.

 

Say we build a formula that references D13.  If we add/remove columns/rows around D13 the formula will adjust automatically.  We can even click/drag/drop a block of cells around D13 and the formula will adjust.

 

Excel recognizes that the cell has moved.

 

If, however, we sort cells including D13, the formula will not adjust.  Excel "feels" that the cell has not moved, only the values.

Hi,

 

Select a range of cells where you want these unique random numbers and run this code.

 

 

Sub Unique_Randoms()
Dim FillRange As Range, c As Range
Set FillRange = Selection
For Each c In FillRange
Do
c.Value = Int((47125 * Rnd) + 1)
Loop Until WorksheetFunction.CountIf(FillRange, c.Value) < 2
Next
End Sub
