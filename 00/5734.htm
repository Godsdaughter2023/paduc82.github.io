ken sheridan say: It seems that you are in essence modelling a bill of materials.  This is a classic relational database problem, and there is a vast amount of literature on the subject, but the classic model boils down to a very simple one of two tables:

Parts
….PartID  (PK)
….Part

PartStructure
….MajoePartID  (FK)
….MinorPartID  (FK)
….Quantity

PartStructure is what is known as an 'adjacency list'.  Both of ikts foreign key columns reference the primary key of Parts.  The rows in Parts represent not only indivisible parts but assemblies made up of other assemblies and/or indivisible parts. I theory you could model a Boeing 747 right down to every last nut and bolt with these two tables.

The Parts table can have other attributes, many of which will apply only to the indivisible base parts, e.g. the weight of a part, as from this the weight of any sub-assembly, assembly right up to the complete item (the Boeing 747) can be computed.

Generating a bill of materials from this simple model is where the  complexity sets in.  The problem here is that it needs recursive querying to gerenearte a bill of materials to an arbitrary number of levels.  While some more advanced database management systems support recursive querying, Access does not.  The algorithms for doing so are well established, however, and it is possible to simulate it.  You'll find an example as BoM.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169


The BoM.accdb file in this little demo file generates the bill of materials by creating a temporary separate database file at runtime and progressively writing data to this file.  The code is in the basBom module in my demo.

Generating a bill of materials to a fixed number of levels from the same two tables is much simpler, using a query with multiple instances of the PartStructure table, and this is illustrated in the PartsTree.accdb file in the zip archive.

Importing non-normalized data from Excel into this model would use the same techniques illustrated in my DecomposerDemo file.  You'd need to relate other tables to the Parts table of course to build the full model.

You could try something along the lines of

Private Sub YourButtonName_Click()
    On Error GoTo Error_Handler
    Dim rs As DAO.Recordset

    With rs
        If .RecordCount <> 0 Then
            .MoveFirst
            Do While Not .EOF
                Me.YourFormControlName1 = Null
                Me.YourFormControlName2 = Null

        End If
    End With

    On Error Resume Next
    If Not rs Is Nothing Then Set rs = Nothing
    Exit Sub

Error_Handler:
    MsgBox "The following error has occured" & vbCrLf & vbCrLf & _
           "Error Number: " & Err.Number & vbCrLf & _
           "Error Source: YourButtonName_Click" & vbCrLf & _
           "Error Description: " & Err.Description & _
           Switch(Erl = 0, "", Erl <> 0, vbCrLf & "Line No: " & Erl) _

End Sub
