If the problem still exists

  • Copy all of the event code to Notebook
  • Delete all of the code from the Userform module.
  • Restore the code for one event at a time and test.

ossiemac say:

Your code is full of "On Error Resume Next". Extremely bad programming and it makes it impossible to identify the errors in the programming.

You need to remove the On Error Resume Next from all of your programming. I suspect that this code could be causing your problems because it is skipping past an error that results in further problems but we will not know until after you remove this code from your project.

Generally you use such code if you are using an error to determine an issue and then resume error trapping as soon as possible. Following is an Example of using "On Error Resume Next" to test if a specific worksheet exists.

Sub TestIfSheetExists()
    Dim ws As Worksheet
    On Error Resume Next
    Set ws = Worksheets("Sheet5")
    On Error GoTo 0     'Resume error trapping ASAP.
   
    If ws Is Nothing Then
        Set ws = Sheets.Add(After:=Sheets(Sheets.Count))
        ws.Name = "Sheet5"
    End If
End Sub

In the above example it is expected that the code will error if the worksheet "Sheet5" does not already exist and therefore the worksheet can be added. Note that error trapping is resumed as soon as possible.

If you use On Error GoTo then you should include code at the label to advise that an error has occurred otherwise you will never be aware that an error has occurred. Simple Example of GoTo below.

Sub AssignSheetToVariable()
    Dim ws As Worksheet
    On Error GoTo shtNonExistant
    Set ws = Worksheets("Sheet5")
    On Error GoTo 0     'Resume error trapping ASAP.
   
    'Place other code as required here
   
    Exit Sub    'If gets to here then no error so exit the sub


shtNonExistant:      'Code past this label only processed if On Error is invoked
    MsgBox "Error number " & Err.Number & " " & Err.Description & " in Sub AssignSheetToVariable()"

End Sub

If an error occurs in the above example then the developer can comment out the line "On Error GoTo shtNonExistant" and run the code and see on which line the code stops.

There is nothing wrong with the use of "On Error" but it is essential that you handle the error either as per the first example when it is used for a specific programming purpose or as per the second example so that a user will be aware that an error has occurred.

I actually use "On Error GoTo" as per the example at the link below so that if a User has a problem then I can locate what occurred. The error information is appended to a text file and I can view the text file periodically. Users are rarely able to describe what occurred and saving the information in the text file assists the developer to locate the problem.

Sub Test()

graham mandeno say:

Hi Irshad

That is the way subforms work.  When the focus moves to the subform, the record on the main form is automatically saved.  If the main form record fails to save, then the subform cannot get the focus.

This is usually necessary because in most cases the main form and subform have a one-to-many relationship, and the main form record needs to exist before subform records are created, to preserve referential integrity.

Perhaps you could describe a bit more what it is you want to achieve.

Best regards,
Graham
