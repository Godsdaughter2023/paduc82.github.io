quietman7 say:
A "Cannot find...", "Could not run...", "Error loading... or "specific module could not be found" message usually occurs when a file that was set to run at startup or as scheduled task in the registry has been deleted. It's not uncommon to find remnants of startup and scheduled task registry keys when booting into Windows after uninstalling programs or using anti-virus/security scanning tools to remove adware and malware. Windows is trying to load this file but cannot locate it since the file was most likely removed. However, an associated orphaned registry entry still exists and is telling Windows to load the file when you boot up. Since the file no longer exists, Windows will display an error message. You need to remove this registry or scheduled task entry so Windows stops searching for the file when it loads.

A safe and easy way to remove them is to download AutoRuns and save it to your Desktop.
  • Create a new folder on your hard drive called AutoRuns (C:\AutoRuns) and extract (unzip) the file there.
  • Open the folder and double-click on autoruns.exe to launch it.Vista/Windows 7/8 users right-click and select Everything tab.
  • In the top menu, click File > Find... and type suppor_1.DLL (the file name related to the error), then click Find Next.
  • Alternatively, you can scroll through the list and look for any entry related to suppor_1.DLL.
  • If found, right-click on the entry and choose delete.
  • Exit Autoruns and reboot your computer when done.

If you have only a single section following the front matter, then you can use the SECTIONPAGES field instead of NUMPAGES. Insert the Page X of Y construction, then press Alt+F9 to display the field code. Change NUMPAGES to SECTIONPAGES, Alt+F9 to toggle back, and F9 to update.

BTW, if you have used the LIKE operator and * wildcard characters to allow for Null parameters, as is sometimes recommended, this is not a universally reliable method as it does not allow for Nulls at the column positions in rows in the table.  The above method is both more reliable and more efficient.  One thing to note with this method, however, is that, with the exception of parameters of DateTime data type, the parameters must not be declared in a PARAMETERS clause in the query.  A parameter declared as TEXT for instance will not evaluate as NULL if left empty.

ken sheridan say: As far as I can see, at present your query appears to be designed to return rows which match the pattern of either of the text values entered into two unbound controls in the form, and which match the numeric values entered in the other two controls.  In all four cases the control can be Null. I would recommend that the WHERE clause be simplified as follows, to make the logic more apparent:

WHERE ([FragranceName] LIKE "*" & [Forms]![Find By Filter Form]![txtFragranceName] & "*"
    OR [Forms]![Find By Filter Form]![txtFragranceName] IS NULL
    OR [SubmissionName] LIKE "*" & [Forms]![Find By Filter Form]![txtFragranceName] & "*"
    OR [Forms]![Find By Filter Form]![txtFragranceName] IS NULL)
AND ([BriefNumber] LIKE "*" & [Forms]![Find By Filter Form]![txtBriefNumber] & "*"
    OR [Forms]![Find By Filter Form]![txtBriefNumber] IS NULL)
AND ([Number] LIKE "*" & [Forms]![Find By Filter Form]![txtNumber] & "*"
    OR [Forms]![Find By Filter Form]![txtNumber] IS NULL)

The query should be saved in SQL view, not design view.  Unfortunately Access will still screw up the layout, but at least in won't throw in any unnecessary garbage as it does in design view.  I save the SQL statements to a text file as part of the application's documentation.  It's then a simple task to copy and paste the logically formatted SQL statement into the SQL window if it's necessary to amend a query definition.

Pattern matching can result in specious substring matches.  To avoid this I would recommend that the first two unbound controls in the form be combo boxes.  The user can then select values from the list, or, by virtue of the AutoExpand mechanism,  progressively move to the first match as characters are typed into the control.  The query can then use the equality operator rather than the LIKE operator.

Two controls relate to a columns named Number and BriefNumber.   Are these columns of number data type?  If so pattern matching is inappropriate, and the equality operator should be used.  If the 'number 'is in fact a text data type then I would again recommend that a combo box be used as the parameter.

Because you are using a combination of restricting a recordset by parameters, and filtering the subform, the two should act independently of each other, provided that, when the former is used, the subform's FilterOn property is False, and conversely, when the latter is used, all parameters are Null.  You have catered for this in all but the cmdSearch button's Click event procedure, which should be:

Private Sub cmdSearch_Click()

    Me.subfrmFindByFilter.Form.FilterOn = False
    Me.subfrmFindByFilter.Requery
    
End Sub

When you 'reset' the form, so you also wish to remove the selections in the check box control bound to the AddToExport column?  If so, add the following to the cmdReset button's Click event procedure:

    Const conSQL = "UPDATE TableNameGoesHere SET AddToExport = FALSE"
    CurrentDb.Execute conSQL, dbFailOnError

Do this before requerying the subform.

The error message pops up because the LinkToFolder links can’t be protected when using the AutoSave feature in the OneDrive sync client.
