markuswegener say:

simple query doesn't return any records

Using Access 2007, XP

My actual goal is to create a cross reference query;  I have information about projects in one table, information about items related to these projects in another table, and one table that only has two fields; one matches a field in the projects table and one in the other table.  I found UNION queries and that sounds like it is what I need, however, I am having trouble with getting results in just a simple query ...

I created two queries, sql below:

1.  SELECT tblClients.ClientCode, tblClients.ClientName
FROM tblProjectInformation INNER JOIN tblClients ON tblClients.ClientName = tblProjectInformation.ProjectClient;

2.  SELECT tblProjectInformation.ProjectClient, tblProjectInformation.ProjectUnits
FROM tblUnits INNER JOIN tblProjectInformation ON tblUnits.UnitGroup = tblProjectInformation.ProjectUnits;

Not only are the queries essentially the same, but the fields they are using are as well:  All fields are text format, and the fields to the right of the = are filled in via a lookup from the field to the left of the =.  But query 1 will not return any results while query 2 returns all the records where the ON condition is true.

Why could that be?

I won't be able to work on my fancy UNION query until I figure this one out.

Markus

ken sheridan say:

I doubt you need a UNION query here.  It's more likely you just need to join all three tables in a query.  The tblProjectInformation table is modelling the many-to-many relationship between clients and units, so while this table is present in the query to map rows in tblClients to rows in tbUnits, it's the columns from the latter two that you'd normally return, e.g.

SELECT ClientName, UnitGroup
FROM tblClients INNER JOIN
(tblProjectInformation INNER JOIN tblInits
ON tblUnits.UnitGroup = tblProjectInformation.ProjectUnits)
ON tblClients.ClientCode = tblProjectInformation.ProjectClient;

Note that the join between tblClients and tblProjectInformation is on the ClientCode column from the former, not the client name.  I've assumed that this is the primary key of tblClients.  This is what John was referring to in his reply, the values of the ProjectClient column in tblProjectInformation being numbers which reference the primary key ClientCode of tblClients.  Using the 'lookup fields' wizard when creating a table in design view masks these real values, and leads to the sort of confusion you've encountered.  For this and other problems associated with 'lookup fields' this feature should be avoided.  For more reasons why see:

http://www.mvps.org/access/lookupfields.htm

In fact, as ProjectUnits seems to be successfully referencing UnitGroup in your second query (i.e. you are using 'natural' keys), you don't need to include tblUnits in the above query unless you want to return other non-key columns from tblUnits, so the following would give the same results:

SELECT ClientName, ProjectUnits
FROM tblClients INNER JOIN tblProjectInformation
ON tblClients.ClientCode = tblProjectInformation.ProjectClient;

If this isn't what you are looking to do post back with a detailed description ofwhat you are aiming for, rather than how you are trying to do it.

Ken Sheridan, Stafford, England

Date setting choices are different from PC Settings and Control - Microsoft Community

barthumphries say:

Date setting choices are different from PC Settings and Control Panel; how to change with more options

If you prefer to use the international format for your dates, such as "19 Oct 2014" instead of "Oct 19, 2014" but you live in the US and use a US time zone and want commas in your currency instead of periods (you want $5,000.00 to be the default money format), etc., then you're going to want to change the short and long date formats.

There's only one problem.  The available options in PC Settings to set the date are not as complete as they are in the Control Panel in the Region settings.

If you want to change the short date format from M/d/yyyy to dd MMM yyyy and the long date format from dddd, MMMM d, yyyy to dddd, dd MMMM yyyy then you can't do it in PC Settings.  PC Settings doesn't allow a leading zero in your long date format, and it doesn't allow a 4-digit year like that short date format has that I typed out.  You can only set this in the Region dialog.

Open the Region dialog by going into the Control Panel and searching for either "region" or "date".  Once you select it and it pops up in its own window, click "Additional settings" at the bottom, then click the Date tab.  This is where you can type in how you want dates to be formatted.

PC Settings won't refresh automatically until you change to look at something else in PC Settings then come back (or close it and reopen it), it'll show the correct settings.

Will those new settings be backup up if PC Settings is being backed up by OneDrive?  I don't know.  OneDrive doesn't have an option so say, "Don't backup my photos, just everything else", so I'm ignoring it.  It probably won't be backed up, though, so if you ever have to use your backed up PC Settings to restore a messed-up computer, you'll probably have to go make that change again.  Luckily, Microsoft hasn't yet removed all access to the Control Panel and it's a lot easier to access it in Windows 10 than it was in Windows 8 (without Classic Shell installed -- with Classic Shell it was always easy).
