joel ivory johnson say: The size you are seeing is the size of the cab (which is compressed). This number is relevant to a number of Marketplace users.  If a user doesn't have an unlimited data plan (which are not available in all markets) then bandwidth used has an impact on ones bill.  Also if a user is on a slower connection seeing the download size may give the user insight as to how long it will take to download.

I understanding what you are saying though.  If I am downloading a file onto my PC from a web site that web site may tell me how big that the file is in what ever form it is in when it is being transfered and not how big the file is when it is decompressed or installed. But on a mobile device with space being more constrained there's higher concern for the decompressed size of the application.

Though I don't know that the installed size can be easily assessed and described by a single number. For example some applications may require additional components (ex: SQL Server). So those needed packages will be packaged inside of the application install.  But the resulting space consumed on the device may not be the sum of the size of the decompressed files.  For a device that already has the dependent component preinstalled (either because another application installed it or it being built into the ROM on a  certain firmware version) then that additional component won't be [re]installed and thus doesn't result in more space being consumed on the device. But on a device that doesn't have that needed component both the application and the component will be installed resulting in a higher amount of space being consumed on the device.  There are also applications that may contain a number of other resources to accomodate different devices (such as multiple versions of graphics for different resolution devices) that may or may not be automatically deleted by the applications installer. So there will be variance in how much space that some applications consume.

So while I understand your concern here I don't think it's a problem for which there's a good solution.

insert static datestamp in column D when column E is >0 - Microsoft Community

mwaters say:

insert static datestamp in column D when column E is >0

I have an inventory spread sheet column D = units ordered and Column E = date ordered.  I want Column E to automatically put the date in when Column D >0.  The date cannot change.  I also have a Recieved Column F and Date Received Column G that requires the same code.

Is this possible?  I am currently using the NOW function and it updates everything.

dave peterson say:

I wouldn't rely on
Format(Now(), "dd/mm/yyyy")
to give the correct date.

This is like typing the value into the cell -- and excel will parse the input
the way it does manual entry (using the Windows short date format for that
user).

04/02/2011 would be treated as April 2, 2011 for me with my USA (mdy order)
settings.

I'd use:

Private Sub Worksheet_Change(ByVal Target As Range)

    If Target.Cells.Count > 1 Then
        Exit Sub
    end if

    If Not (Intersect(Target, me.Range("D:D")) Is Nothing) Then
        If IsNumeric(Target.Value) Then
            Application.EnableEvents = False
            with Target.Offset(0, 1)
                .numberformat = "dd/mm/yyyy"
                .value = Date
            end with
            Application.EnableEvents = True
        End If
    End If
End Sub

I also qualified some ranges and added some properties.  I like that explicit
syntax better.

Private Sub Worksheet_Change(ByVal Target As Range)
