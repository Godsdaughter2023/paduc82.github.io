550 5.7.133 RESOLVER.RST.SenderNotAuthenticatedForGroup for O365 - Microsoft Community

iustin irimia say:

550 5.7.133 RESOLVER.RST.SenderNotAuthenticatedForGroup for O365 groups that are synced with AD Connect Group Writeback

Summary

Since group writeback feature is still in preview, one of the current limitation, if you have centralized mailflow enabled or the MX pointed to the on-premises server, is that incoming mailflow to the group will fail for external senders, with:

'550 5.7.133 RESOLVER.RST.SenderNotAuthenticatedForGroup; authentication required; Delivery restriction check failed because the sender was not authenticated when sending to this group'

Even if you set the group in O365 to receive emails from external recipients with Set-UnifiedGroup -RequireSenderAuthenticationEnabled:$false


This is happening because the attribute 'msExchRequireAuthToSendTo' will not be updated in on-premises AD with the corresponding value from Exchange Online AD.

The sync rule in AD Connect will always update the attribute 'msExchRequireAuthToSendTo' to $true at the next sync cycle, even though we are manually altering it. So, as long as this attribute remains true in local AD, the on-premises servers will reject any emails from external senders.

Resolution

It is possible to adjust the default AD Connect sync rule in Synchronization Rules Editor to make sure we at least allow the manual update of the attribute in the local AD attribute editor.

For this:

-Open Synchronization Rules Editor

-Select 'Direction: Outbound' and 'MV Object Type: group' then select the 'Out to AD - Group SOAInAAD' rule -> Edit

-On the prompt select 'Yes' this will disable the default rule and create an active copy so you can edit this one. This way, you can disable the rule that we will create and go back to the default settings at any point. It is not supported to make changes on the default created synchronization rules, but creating a clone and working with that is. 

-Adjust the Precedence, as the default is set to '-1', to something that will not match any other rule precedence. If there is the same precedence on another rule, an error will be thrown when trying to save that rule.

-Go on 'Transformations' and click on the 'msExchRequireAuthToSendTo' in a space that is not clickable. Then 'Remove' this transformation.

-It will then look like this. We can now hit 'Save':

-After this we will see the original rule disabled and the clone enabled:

-At this point we now need to run a Full Synchronization of the directory from Powershell:

-Now the attribute is not forced anymore, and we can go ahead and adjust 'msExchRequireAuthToSendTo' for the synchronized groups in local AD:

-We can now keep this property to False for the desired groups. After the next sync cycle will run, this attribute will no longer be set to true anymore as we just adjusted the sync rule. 

-We should now be able to receive emails from external recipients.
