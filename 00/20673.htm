  6. Locate and delete the configor config.xml file.
  7. Press the Windows  and R keys on your keyboard at the same time again.
  8. Type %appdata% into the Run window and click OK.
  9. Locate and rename the Skype folder to Skype_Old.
  10. Open Skype.

scott mcdaniel say:

I'm not a big fan of the MultiValued Fields in Access. They tend to hide data, and can make reporting and querying much more difficult, and essentially lead to a de-normalized structure (even though they're storing data as normalized behind the scenes).

Your database really should have a Projects table, an Employees table, and a Project_Employees table. When adding Employees to a Project, you'd show the user a list of Employees, and they would select them, and you'd add each Employee ID to the Project_Employees table, along with the ID of the Project.

scottgem say:

First, a couple of points:

"I have created two databases. One with the tables and the second with the queries and forms so that I can copy the "front end" database to a couple of computers in the office"

This is not "two databases", this is called a "split" database where the the application is split into a back end (just the tables) and a front end, everything else. This is SOP for a multi-user application. 

"I was also taught that in this structure, a form should pull from a local query so the form is pulling from a "Projects" query "

I'm not sure what you were taught, but In a split database, the tables are linked to the front end from the back end. Once linked they can be used as if they were local tables. While there are some schools of thought that would recommend using a query (i.e. an unfiltered, complete field set, query), its not really necessary. 

"I like the idea of using a checkbox list because it would show all of the current employees and I can just check the ones that are working on the project". 

Yes, this makes for an easier interface, however, you should not confuse data storage with presentation. They are two different things. Yes a multi-value field makes for easier presentation but makes for a lot more work in integrating the data. 

That being said, while you have to create the MVF on the table level, you have some flexibility on the form level. So what you do is set the RowSource on the table level to the full employee list, not just actives. In the Default Rowsource on the form you leave it the same full list. But in the Got Focus even of the control on the form you modify the Rowsource to include only active employees. Something like:

Me.cboEmployee.RowSource = "SELECT EmployeeID, lastname & ", " & firstname as Employee FROM tblEmployees WHERE Active = True;

So when you pull down the list to get the checkboxes it only shows Active employees.

However, I tend to agree with Scott Mcd, that this is not a place I would use an MVF. MVFS are nice, but I would not use them where I may have to link to the contents of the MVF. Instead, since you have a many to many relationship here, I would use the Junction table to model that relationship. 

That doesn't mean you are restricted to the continuous form subform with combos. But it does mean you will need to do more work to get the check box interface. You would need to use an unbound subform where you populate unbound and locked comboboxes with each employee with a Checkbox next to the combo. I would hide the pull down arrow and Set Enabled to False so the user can't select the combo. They can only use the Checkbox. I would then use code to loop through the controls populating them from the employee and junction tables. Then use code to update the junction table after selection.

This little demo file illustrates a number of interfaces for representing a many-to-many relationship type like this, in my case between students and courses in your case between projects and employees.  One method is by means of a multi-select list box in which one or more courses can be selected for the current student.  It also illustrates the use of a conventional form/subform, a major advantage of which is that it is very easy to include other attributes of the relationship type other than the principle one of, in my case, the course name, in your case the employee name.  In my demo the status of the students relationship with the course can be entered.

I would advise against using a 'multi-valued field' as this provides no way that I'm aware of to represent other non-key attributes of the relationship type which it is modelling.

ken sheridan say: The Me keyword returns a reference to the current instance of the class in which the code is executing, but you can think of it loosely as a short way of referencing the current form or report.

I assume the method from my demo that you are using is the second one, using a multi-select list box, in which case the reason you are seeing the same selections for each record is presumably that you are not re-selecting the related items in the list box by means of code in the form's Current event procedure.  As the list box is an unbound control, without this it will retain the values you first selected whatever records you then move to.  You can only use in in single form view of course.

If, when you make a new selection in the list box, it does not write a row to the table, then there is something amiss with the code in the list box's AfterUpdate event procedure.

A word about ReFS and Windows 10 Pro for Workstations

If you are celebrating the announcement of the availability of ReFS in the upcoming W10 Pro for Workstations, take care.

Everything mentioned about ReFS in the announcement of W10 PfW is already available in your current copies of W10 Pro.  Note carefully that the announcement refers to Storage Spaces and ReFS.  ReFS is already offered as an option when you set up an SS on your present rig.  In fact you can make a simple registry change and have ReFS available for any data disk as well.  However there has been a bug since the Anniversary Update that makes the registry change problematical because the OS then thinks it cannot use VSS for backup functions.  That's a pretty big bug (spelled BUG).  If you want to explore ReFS for formatting single data disks I recommend you read the great tutorials on this on TenForums.  While it is OK to make the registry edit and then format a data disk, you must then immediately restore the default registry entry before continuing to use the OS.  Unless MSFT has just fixed the bug, of course.  Which I believe they have not since there has not been anything mentioned in the fixes in the preview build announcements in the FH.

The Windows 10 Pro for Workstations announcement by no means indicates that ReFS will be available as a format for system disks.  That appears to be a long way off still because, while ReFS is years past the beta stage and has been in production use in IT environments since Windows 8.1 debuted, it still is in feature development (as even NTFS is) and all the features needed to support installing the OS on it are not yet in place.  MSFT is being (rightly) super conservative as they approach this point with ReFS since it is the very foundation for an OS.

All of us should look forward to NTFS's successor, it is one heckava OS and pretty radically different.  But this announcement involves ReFS because W10Pro for Workstations is the improvement IT pros have been wanting, not so much for consumers, and ReFS is much more in use in enterprise networks than at home or office.  This is not an announcement of ReFS as the all-purpose Windows OS.  Not yet.

Your thoughts?  Make them known to MSFT in the Feedback Hub.

And the first question here, of course, is will we see insider builds for W10Pro for Workstations here or will they be part of the Server Insider track?
