I'm working on this symptom for XPsp3 and noticed a purported solution (Kelly's Tweak #371) could be partially implemented using an  sc.exe  command.   So, I'm curious if W7 already implements this or if it would help if it doesn't:

First, find out what it already does (e.g. enter in a cmd window):

     sc.exe  qc  audiosrv

In XP this showed me:

DEPENDENCIES       : PlugPlay
                           : RpcSs

The most significant difference that I could detect for Kelly's tweak could be effected by the following modification:

     sc.exe  config  audiosrv depend= DcomLaunch

I.e. that says don't try to start the   Windows Audio  service until the  DCOM Server Process Launcher  has finished.  

While researching this I found this Windows Server 2008 document which at least gives a partial implied rationale for this change/suggested change:

http://technet.microsoft.com/en-us/library/dd349799(WS.10).aspx

<quote>

DCOM Server Process Launcher

In versions of Windows earlier than Windows Server 2003 with Service Pack 1, the Remote Procedure Call (RPC) service (RPCSS) ran as Local System. To reduce the attack surface of Windows and provide defense in depth, the RPC service functionality was split into two services in Windows XP Service Pack 2 and Windows Server 2003 Service Pack 1.

</quote>

So, the idea must be that the part of rpcss that audiosrv depends on is the part that dcomlaunch is implementing.  If so, having it depend on (the now lesser) rpcss would not be sufficient and result in a possible indeterminancy (e.g. depending on the timing of the other services starting it might or might not be enough to let audiosrv do what it needs to do.)

As for the plugplay part I haven't worked that one out yet.  I have saved a ProcMon Boot Logging trace and am still looking at that part.  So far it looks to me that  plugplay  should be kept in that dependency list.  E.g. looking at the  ServiceGroupOrder\List   PlugPlay  occurs after  COM Infrastructure (dcomlaunch's LOAD_ORDER_GROUP)  and before  AudioGroup  but otherwise I can't see anything which would prevent AudioGroup from being started before  PlugPlay  had finished (which is what the original dependency required besides the completion of (presumably the assumed full function) rpcss).

Actually now that I have explained it here I am going to go away and give this alternative a try:

     sc.exe  config  audiosrv depend= dcomlaunch,plugplay

Wish me luck.  The first change wasn't enough.  (In fact, since I have my ProcMon Boot Logging trace still to look at I will also try to confirm that this really may be necessary; however, I'm not sure whether the trace entries will be that clear about when a service is finally ready; I assume it at least shows me when it starts getting ready.)

I hope this has some relevance for you with W7.   ; )
