Before you track the actual work (in hours) you should track the actual start and the actual duration up to the status date. Any actual work that occurred must have happened on the days of actual duration.

albert d. kallal say:

It not clear if you worked with split databases in the past.

 

That is you have the “front end” part that you install on each workstation, and then you have the back end. That back end can be an accDB file, or in your case SQL server.

 

Once you “link” the tables and point them to the back end (a file, or SQL server), then from that point onwards you should not see nor get some prompt on start-up.

 

It not clear if you “ever” had your links working to SQL server but the general approach is to launch the linked table manager IF ALREADY at one point in time you had the linked tables working to sql server.

 

So just make sure when you create the links (or use the linked table manager) you choose what is called a “file” DSN. This is easy, since that is the access default.

 

Just make sure that you check the box to save the password. Once you do this, and re-link, then at that point you can distribute the front end to any workstation. (no DSN, no config will be required on each workstation if you do this).

 

So as a general rule, access will and should and can remember the logon and connection information. And this information is “self-contained” in Access – no additional setup is required when you deploy (copy) the FE to other workstations.

 

So a few things:

 

DO NOT choose the Access data project option (this is depreciated, so you likely did not by accident choose this type of migration).

 

After you migrate, then your linked tables should work. If they don’t, then launch your access application (hold down shift key – you don’t want forms, code etc. attempting to run since your table links seem bad.

 

It also not clear what kind of sql logons you are using. Are you using SQL logons, or are you using windows authentication logons?

 

Ok, assuming you launched the application (shift key to prevent start-up junk from occurring). At this point, you can click on the external data tab. Now click on linked table manager.

 

At this point, you can selected all (if in fact all tables are linked tables). MAKE SURE you click in lower left

 

[x] Always prompt for new location.

 

So select all, above box checked, now hit ok.

 

At this point the odbc manager will launch, and you can choose new (again, MAKE SURE you leave the default setting of “File Data Source” as opposed to machine data source.

 

Choose new.

 

You now choose the ODBC driver.

 

You can choose the “legacy” driver called “SQL Server”

 

You can also choose SQL server native client 11.0 (or later). However this choice has some downsides. The BIG issue is that these newer native 11, 12, 13 drivers are PREFERRED choice, but this choice ALSO forces you to ensure that this driver is installed on each workstation. The long-time “legacy” SQL driver is installed on every copy of windows going all the way back to windows XP (and even before).

 

So the “SQL Server” is a preferred choice. However, if you did NOT tweak and change settings when using SSMA, then SSMA likely moved your date time columns to “newer” SQL data column types (datetime2).

 

So you have a choice:

Either change all datetime2 columns in SQL (using SQL manager studio to datetime), or re-migrate using SSMA again, and ensure that no datetime2 columns are used. Or simply ensure that every workstation has the native 11 or later ODBC drivers installed. These drivers have zero to do with Access and are not installed by access.

 

For now, let’s go with the “legacy” old time long time drivers. So after new, choose SQL server. When you hit next, the brose button can be confusing – but allows you to browse to a location to save the DSN file. (Use my documents – windows 10 often prevents the “default” location as read only, and thus you can use that. So browse to my documents, and then type in the name of the “connection” you going to save here.

 

Now just work your way through the prompts.

 

During this process, you also get a chance to “test” connection. If that test fails, then you not providing the correct information.

 

Last but not least:

 

Once (and only after) you get this working, and you still get a logon prompt when trying to open a table (or a form attached to one of those linked tables), then you may well have to delete all the table links, and then launch the external database tab, and choose odbc – and start over again (the reason you might want to do this “again” AFTER you get this working is that the prompt to “save” the password in the connection ONLY appears the VERY first time. If you miss that prompt, then you have to delete the tables to get that prompt.

 

Once you get the re-linked tables working, then your application is NOW BY DEFAULT what we call DSN-less linked.

 

So access ONLY uses the “file” dsn during the ONE TIME linking process – after that, the DSN is ignored. This is a good thing, since now you can copy this application to any workstation on your network, and it will work without any kind of re-linking, or password at logon, or any kind of config of each workstation.

 

Once you link, as noted, you can decide if is better to re-migrate the data again (but change SSMA to NOT use datetime2 columns). So you either have to change SSMA settings, or choose the sql “native” drivers. If just one or two tables have “datetime2” columns, then I just use SQL manger and change them back to datetime columns. (this might be far less work then re-migration of data – it really depends on how many tables you have.

 

So either get rid of datetime2 columns on the sql server side, or you have to ensure that each workstation have the newer “native 11” or later drivers (which are not installed by default).

I have to 100% agree. The only real downside is deployment of the application to each workstation. The drivers will have to be installed - as you note, it is only a one time install.

For someone just starting out - one might just go with the legacy drivers. This would especially be the case if they don't have rights to install new software on each workstation. (but of course have access on each machine).  I find more and more machines locked down  these days, and each machine don't have installing rights anymore (hence you now have to engage IT support for this task). And this tends to be around the time that SQL server is setup and available to users.

However, at the end of the day, no matter how I spin this, it certainly is a recommend choice to install these later drivers. If one has IT support for this task, or has rights to install newer software and drivers on each workstation, then "little" reasons exist to not jump to the newer drivers.

I have some rather large installs - and we not adopted nor added the ODBC drivers as part of our install package because we wanted to avoid the hassle of required elevated rights for our installs. So there are some legitimate reasons to avoid the newer native drivers, but given a choice, the newer native drivers are a better choice. I simply wanted to point out the pro vs cons here.

There are reasons why emails or messages end up in spam boxes. These are the most common reasons that cause emails to be marked as spam:

  1. Inadequate permission - In order to have email permission, the recipient must have specifically agreed to receive periodic messages or offers. Without this permission, you’re sending unsolicited mail, which is a surefire way to get caught in a spam filter.
  2. Unauthenticated Sender ID - This can happen when an agency is trying to send out emails using its own server. ISPs usually authenticate sender identities to make sure messages are coming from a legitimate source.
  3. Sending Emails with spammy Content - Examples are bad links, too much image, not enough text, sloppy html code.
