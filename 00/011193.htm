djsteele, mvp say:

I'm afraid you haven't given us enough information.

For one thing, you say "vDates are formatted for SQL correctly in a module from Microsoft". Please show how you've declared vDateOne and the code of that module.

Secondly, you need to include a value to reset BackColor for those occurrences when DCount is equal to 0.

Thirdly, you haven't really explained the problem! Is this a continuous form (or a datasheet view) and you're hoping to have only certain rows change? That, unfortunately, isn't possible using code such as you've shown. While it may look as though there are, say, 30 different occurrences of txtDateOne on the form, in actual fact there's only 1 occurrence, repeated 30 times. Any change you make to txtDateOne is going to apply equally to all occurrences. You need to look into Conditional Formatting to handle that case.

ken sheridan say: It may be just a typo in your post here, but in the following expression you are using the same variable twice, and there seems to be an additional character 'S' before the last one, and an extra closing parenthesis after it:

[HolidayStart] => " & vDateOne & " AND [HolidayEnd] <= " & S vDateOne)

I'm not sure what you mean by 'The vDates are formatted for SQL correctly in a module from Microsoft' but if these are variables of date/time data type formatting in a function elsewhere won't make any difference unless you use the return value of the function when building the expression.  I would normally format date/time values in this sort of context using the ISO standard of YYYY-MM-DD, and concatenate them into the expression as date literals, e.g.

If DCount("[HolidayID]", "[StudentHolidays]", "[StudentID] =" & _
    vStudentID & " AND [HolidayStart] => #" & Format(vDateOne,"yyyy-mm-dd")  & _
    "# AND [HolidayEnd] < #" & Format(vDateTwo,"yyyy-mm-dd") & "#+1") > 0 Then

Note how the date range is defined as on or after the first date and before the date following the last date.  This caters for the HolidayStart column inadvertently including values with a non-zero time of day element, which you cannot reliably assume not to be the case unless you have taken specific steps in the table definition to disallow such values.

ken sheridan say: Looking at your original post again, I realise the you have the criterion the wrong way round. You need to test for the text box's value being between the values in the column, not vice versa:

If DCount("[HolidayID]", "[StudentHolidays]", "[StudentID] =" & _
    vStudentID & " AND #" & Format(Me.txtDateOne,"yyyy-mm-dd")  & _
    "# >=  [HolidayStart] AND #" & Format(Me.txtDateOne,"yyyy-mm-dd")  & _
    "# < [HolidayEnd]+1) > 0 Then

I don't see any need for the vDateOne variable.  The same may apply to vStudentID  if this value is also in a control in the form.

I'm assuming this is a dialogue form, not a bound form so the question of setting the control back to its normal colour which Doug raises will be relevant if you change the value of the control, so should be included in the code, which would need to be in the AfterUpdate event procedures of both the txtDateOne control, and of a StudentID control if there is one on the form.

ken sheridan say: It would have been helpful if you'd posted that in the first place.  As I said before, you've got the date criterion back to front.  At present the code makes no sense as it is looking for a date between two identical values, which of course means equal to that value.  It would have to be like this to count rows where the date entered in txtDateOne falls between the start and end holiday dates.

    Dim vStudent As Integer
    Dim vDateOne As String
    Dim vCriteria As String
    
    vStudent = Me.StudentID
    vDateOne = FormatedDate(Me.txtDateOne)
    vCriteria = "[StudentID] = " & vStudentID & " AND " & _
        vDateOne & " >= [HolidayStart] And vDateOne < [HolidayEnd] + 1"
    
    If DCount("[HolidayID]", "[StudentHolidays]", vCriteria) > 0 Then
        Me.txtDateOne.BackColor = vbRed
    Else
        Me.txtDateOne.BackColor = vbWhite
    End If

You still haven't told us whether this is in a bound form or an unbound dialogue form, or in what event procedure(s) the code is.  If it's an unbound dialogue form and you just want to find out if a student is on holiday on a particular date then it would go in the AfterUpdate event procedures of both the StudentID and txtDateOne controls.  If it's a bound form in single form view then it would go in these event procedures along with the form's Current event procedure.  If it's a bound form in continuous forms view then it would be the same, but the txtDateOne  control would need to be in the form's header of footer, so it would be visually divorced from the row to which it applies, which would be rather confusing to the user.  In this context a better solution would be to base the form on a query which in addition to the column sin the table, also returns a computed column based on an expression which references the txtDateOne control as a parameter.  The form would be requeried in the AfterUpdate event procedure of the txtDateOne control.  The control bound to the computed column could be a check box, which would show checked where the date falls within the holiday dates, unchecked otherwise, or it could be a text box, either displaying values such as 'Yes' or 'No', or with the values hidden by setting both the controls ForeColor and BackColor properties by default to green for instance, and by means of conditional formatting to red if the date in the txtDateOne control falls within the holiday dates.
