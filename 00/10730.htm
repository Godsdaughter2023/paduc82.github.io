ken sheridan say:

Maybe the problem is I really don't know how to use relationships.   I don't know where to look for

referential integrity and cascade updates. 

 


On the Database Tools ribbon select Relationships from the Show/Hide area (I'm using 2007, so it may differ slightly in 2010).  This will open the relationships window.  To add tables to the window if they are not already shown, right click in the window and select Show table.  then select the relevant tables from the list.  Drag them about so that they form a logical layout within the window.

If a relationship exists you can edit it by right clicking on the line which joins the tables and selecting Edit Relationship from the shortcut menu.

To create a relationship you drag the primary key column from the referenced table (that on the one-side of a relationship type) to the corresponding foreign key of the referencing table (that on the many-side of the relationship type).  This will open the Edit Relationships dialogue and the columns in question should show in the list. You then select the Referential Integrity and Cascade Updates check boxes.

The relationship type at the bottom of the dialogue should now show One-to-Many.

If you find that it won't allow you to create the relationship this almost certainly means your current data infringes referential integrity, probably by having a row in a referencing table with a value in the relevant foreign key column which does not have a match in the primary key column of the referenced table, e.g. you have an appointment record without a matching member record.  You'll need to correct this first.  You can pin down such mismatches with a query like this:

SELECT *
FROM Appointments
WHERE NOT EXISTS
    (SELECT *
     FROM Members
     WHERE Members.MEM_ID = Appointments.MEM_ID);
