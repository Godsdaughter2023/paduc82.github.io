Publishing with microsoft frontpage__ - Microsoft Community

conéh say:

Publishing with microsoft frontpage__

Help will be appreciated.

 

_ronx say:

The error means exactly what it says - the server timed out.
Publishing with FrontPage Extensions is not a simple process of uploading files. It is (not necessarily in this order):
1) Logon to server - this may have to be repeated several times during the Publishing process
2) Get directory information from both local and remote sites to compare the files - looking for changed files and conflicts.
3) Upload files to temporary names or location on the server
4) Delete the original file (if it exists)
5) Rename the new file to its final name and location
6) Update meta data in the local website, and in the remote website
7) Change the code in pages as appropriate to work with the server extensions
8) Repeat 4) to 7) for each uploaded file
9) When all files are uploaded, rebuild the pages on the server with included files, shared borders, hit counters etc. (where used)
Each of these steps has to be completed within a fixed time frame set by the server configuration. If the time is exceeded, the server times out and the Publish fails.
The timeout can occur because the server is too busy handling other websites - this often happens on low cost shared servers.

If your website is large  then consider breaking it into subwebs - each subweb can be published as a separate website, thus reducing the load on the server. Or try publishing at a different time of day when the server is not so busy (very early morning, perhaps). Large is subjective. I once had the same problem with a site containing a mere 50 files: moving images into a subweb and publishing early eased the problem, moving to a different server with the same host cured it.
Have you asked your host if there is a problem at the server?

ken sheridan say:

Is sqlTKLForm a query, and if so does it include any parameters, e.g. references to controls on an open form?  If so you will need to evaluate the parameters before establishing a recordset object.  The following code snippet is from the start of a mail merging function which does this to establish whether there are any rows to merge:

 

    Dim dbs As DAO.Database, rst As DAO.Recordset

    Dim qdf As DAO.QueryDef

    Dim prm As DAO.Parameter

    Dim wApp As Object

    Dim wDoc As Object

    Dim wActiveDoc As Word.Document

   

    Set dbs = CurrentDb

   

    Set qdf = dbs.QueryDefs(strQuery)

   

    For Each prm In qdf.Parameters

        prm = Eval(prm.Name)

    Next prm

   

    Set rst = qdf.OpenRecordset

   

    ' exit function if recordset is empty

    If rst.EOF Then

        MsgBox "No data to merge.", vbInformation, "Mail Merge"

        GoTo Exit_here

    End If

   

    Set wApp = GetWordApp()

    ' etc

 

The alternative, when building an SQL statement in code is to concatenate the values of the parameters into the string expression.  In your case this would mean removing the assumed parameters from sqlTKLForm and adding a WHERE clause to the SQL statement when you build it, e.g.

 

Dim rs As DAO.Recordset

Dim strSQL As String

 

strSQL = "SELECT tblTKLMaster.TKLNo FROM tblTKLMaster INNER JOIN sqlTKLForm " & _

    "ON tblTKLMaster.[Part Number] = sqlTKLForm.[Part Number] " & _

     " WHERE tblTKLMaster.TKLNo =0" & _

     " AND sqlTKLForm.SomeNumberColumn = " & Forms!MyForm!txtMyNumber & _

     " AND sqlTKLForm.SomeTextColumn = """ & Forms!MyForm!txtMyText & """" & _

daniel g. grau say:
You may be headed for a lot of confusion. See  When I Highlight Text in Light Gray, Then Save, Close, and Reopen, the Highlighting Has Disappeared.

In addition, it will have to be Text Shading (that's what you are using, actually), as Highlight is not well integrated to Word (for instance, it cannot be defined as part of a style).

You could reassign the Del key to a macro, then restore it to the customary EditClear command (the Backspace key cannot be reassigned). But like Bob says, it would be rather complex and I can tell you are not a programmer, since your sample macro can be replaced by a one-liner: 

    Sub HighlightAsType()
    Selection.Font.Shading.BackgroundPatternColor = wdColorYellow
    End Sub

It would be a lot simpler to assign keys to various character styles. If you have reviewers A, B and C, you would have a DeletionA, DeletionB and DeletionC character style with overstrike and shading, and three matching AdditionA, AdditionB and AdditionC styles for insertions. Each reviewer would assign, say, Control D to their deletion style, and Control A to their addition style, in their own Normal template. To delete or add text, they'd select the text and apply Control D or Control A (the latter could also be summoned before doing the typing).

Using such character styles would not represent extra keystrokes, as pressing Backspace requires the same amount of work as pressing Control D.

So, define a common template with all styles, then customize the style keystrokes for each reviewer and distribute versions of the template.

Accepting or rejecting would require a Find and Replace operation, with a discretionary replace by nothing. Examples:
  • For deletions: Search for any overstrike text
  • For deletions and additions of a single revisor: Search for that particular shading color
  • For additions: Search for shading with no overstrike

The Find and Replace setup for those operations could be done through macros.

At the end of the revision cycle, delete those ABC character styles to restore the text, also through a macro.
