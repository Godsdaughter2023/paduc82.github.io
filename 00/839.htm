jay freedman say:
The macro doesn't have to be in the protected document; in fact, it's best if it isn't.

You have three other choices:

  • The Normal.dotm template. Macros in this template are always available when any document is open. But in some cases Normal.dotm gets damaged and needs to be replaced, so this might not be the best choice.
  • Any template that you store in the folder %appdata%\Microsoft\Word\Startup (see the note below) will also be available when any document is open.This is called a "global" template. You can create a new blank document, put the macro in it, and save it as a macro-enabled template (*.dotm) in that folder; you can give it any name you want. This is the choice I'd recommend unless the third choice is valid.
  • If your protected form is based on a specific template in your Templates folder, and if all the forms based on that template use the same password, then put the macro into that template. Then the macro will be available only to documents based on that template.


The expression %appdata% is an "environment variable" that automatically expands to the path C:\Users\<your name>\AppData\Roaming (or the equivalent in other Windows versions). Just paste the expression

%appdata%\Microsoft\Word\Startup

into the address bar of the Save As dialog in Word and press Enter, and it will open to the right place.


With any of these choices, it's important to have the active cursor in the document you want to unprotect or reprotect at the time you start the macro (because it refers to ActiveDocument, which is the document that currently contains the cursor).

Add the code to the Normal template by using the macro editor in Word. When you then close Word, during the closing operation you may be prompted whether to save changes to the Normal template (whether the prompt appears is controlled by a setting in the Options dialog). albert d. kallal say:

Well, you can link and just not update the data - so not really the end of the world here.

Are any of the columns memo?

Access will often pull in additional “blank” spaces during such an import – it supposed to trim them, but the ODBC driver does not in all cases.

 

What this means is that when you export the data as text from FoxPro, then the “blanks” in those columns are removed.

 

When you import from FoxPro, then the blanks are not, and Access has a limited record size compared to FoxPro.

 

So the processing of exporting the FoxPro table as text results in A LOT of extra junk and trailing blanks from the table being removed.

 

As noted, you could try linking the table, and try a append query – or even a bit of VBA code to “traverse” the linked table as a record set until it fails on a given record.

 

It’s possible that only “one” record is bad in the whole table.

 

So the real issue here?

 

Well consider that a flying saucer is different than say a dog.

 

And using an ODBC driver as compared to exporting data as text and then importing the text file are about the comparison between a dog and a flying saucer.

 

So the difference between that of a text file and a ODBC driver is spectacular.

atlr say: Hi Steve,

Thanks a lot for the reply. This approach works for finding the Y coordinate of the box. I guess, this is because the height of the slide in presentation mode is the same as the screen height (1080). But in case of the X coordinate, the slide width in pixels is 1440 while the screen size is 1920. I tried hardcoding the value for now, but that didn't work. Any idea how can I resolve this?

Thanks!

steve rindsberg say:
As I mentioned, if your screen aspect ratio and slide presentation aspect ratio are different, you've got some more homework to do.  

PPT will fill the screen with the presentation if the aspect ratios match or it will make the presentation as large as possible, w/o cropping, if the ratios do not match.

Your slide presentation is set to a 4:3 ratio on a wider ratio screen, so PPT fills the screen top to bottom but leaves black at left and right.

You know PPT will fill the 1080 screen top to bottom, you know that the width will be 1080 * 4/3 or 1440.  

Offset your results by (1900 - 1440) / 2 to get the number of pixels from the left on your screen and you should be good.
