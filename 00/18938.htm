Excel link change by month recognizing file name. - Microsoft Community

seon younglee say:

Excel link change by month recognizing file name.

I am now preparing monthly basis summary(Destination workbook).

In order to prepare this, I make to use link to another workbooks provided by monthly basis as well.

Can you please give me some tips to update  link  according to month base file name automatically

without going through manual link change whenever I open workbook.

Thanks in advance for your help... please.   

myveryownself say:

One way is to use INDIRECT().

 

I made a simple example with excel files names Jan source, Jan dest, Feb source, Feb dest, etc., to illustrate the approach. The month abbreviations are each 3 letters.

 

These formulas go in the dest files.

 

In F1 this returns the containing file's name (as long as the file has been saved).

  =MID(CELL("filename"),SEARCH("[",CELL("filename"))+1,

          SEARCH("]",CELL("filename"))-SEARCH("[",CELL("filename"))-1)

For example, in Jan dest you get

    Jan dest.xlsx

 

In F2 this returns the name of the file to draw from.

    =LEFT(F1,3)&" source.xlsx"

For example,

    Jan source.xlsx

 

In F3 this returns cell A1 of Sheet1 of the source,

    =INDIRECT("'["&F2&"]Sheet1'!$A$1",TRUE)

The idea is to use string concatenation to piece together the reference desired for INDIRECT. In the example, Sheet1 and $A$1 are fixed strings, but these fragments can be calculated if needed.

 

For the daring, the formulas in F1, F2, and F3 can be folded into one.

 

Hope this helps.

Blank pages in label mail merge - Microsoft Community

jturner215 say:

Blank pages in label mail merge

I am using mail merge to create labels in Word. I have set the print area on the sheet in the Excel file, but I am ending up with 4+ blank pages of labels preceding those containing data. Why? I have never had this happen before and have done the merge 6 times with the same result.


a. user say:

<<
The problem might lie in the fact that I am not able to enter a cell range in the "Complete Merge."
>>

Good point.

When you specify what you want to merge, there are two steps, and the way that you specify the records that you want is different in each of those steps.

In the first step, when you define the data source, you specify the excel file, (optionally) the worksheet within the workbook, and (optionally) an Excel Range. 

Let's suppose you specify that the Excel range is C5:T15. Then row "5" of your sheet would be the header row. Row "6" in the sheet would be "record 1" in the merge. ROw "15" would be "record 10."

Let's suppose that rows "6" to "8" of your sheet are blank. That would mean that you do not want to merge "row 6", "row 7," and "row 8". These are "record 1," "record 2," and "record 3"

In other words, what you want are records 4 to 10.

So, in step 6 in the Mail Merge Manager, when you specify which records to merge, you don't specify the Excel range. You specify the first and last record numbers you want to merge, i.e. 4 to 10.

I have to say that I can't help feeling that there is a much simpler way of conveying all that information, but that's the best I was able to do this time.


a. user say:

Do you have blank rows between the header row (that contains the column names) and the first row of label data? The rows could have been hidden.

If so, as far as I know you will either need to delete those rows in Excel before doing the merge, or  specify the range of records to merge in Word (you can do that in the "Complete Merge" part of the Mail Merge Manager dialog box).

As far as I know, Word 2008 expects your Excel data to have a header row (with column names) followed immediately by data rows. If you have empty rows above the header row, Word ignores them. If you have empty rows below the header row, Word treats them as data rows.

My understanding of Mailmerge on both Windows and Mac is that by default, Word does not in fact take any notice of any Print_Area you might have defined. It is possible that older versions of Word on Mac behaved differently in this respect, but that's how it seems to be here with the copy of Word 2008 that I have.

 Print_Area is an "Excel range name" that Excel creates when you define the print area. In order to restrict Word so that it saw only the rows in the print area, you would have to be able to specify "Print_Area" when you specify the data source. In fact, you can do that on Windows Word, but in Mac Word 2008, you only seem to be able to specify
 a. a worksheet name (you can try typing Print_Area as the name, but then the OK button in the dialog is greyed out)
 b. a range of cells, e.g. A1:C3. You do not appear to be able to specify "Sheet1" as the sheet and "Print_Area" as the range, either.
 
[Even if you could specify "Print_Area" as the range of cells to merge, you would need the header row to be at the top of the Print_Area. In the situation where you have a header row followed by blank rows, followed by non-blank data rows, that isn't actually what you have (i.e. you would have to do something like duplicate the header row prior to merging.)]
