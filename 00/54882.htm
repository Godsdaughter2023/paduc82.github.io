For example, if you have a folder with 100 GB of folder then the size will change but it still won’t be the correct size. This incorrect folder size problem isn’t specific to a certain drive or certain types of folders. Any folder can be affected by this incorrect folder size issue.

The problem is with the Windows 10. This is a known bug in the Windows 10 that was introduced in one of the latest Windows Updates. This bug prevents the File Explorer to read the metadata of a file correctly. This is what leads to this unexpected file sizes. That’s why you might also notice that your drive size isn’t correct either. The good thing about it being introduced by a Windows Update is that Microsoft will most likely release a fix in the next couple of updates. But if you don’t want to wait till the updates you can simply follow the steps given in the methods below. There is a way to actually look at the correct file sizes on Windows 10.

Method 1: Use TreeSize

TreeSize is a third party application that can help you get the correct sizes of your files and drives. It is a graphical manager for hard drive space. This tool comes with a free trial as well so you don’t have to spend anything on it. The free trial should be more than enough for you.

  2. Click Next
  3. Run the setup file once it is downloaded
  4. Everything should be self-explanatory. Simply install the software
  5. Once installed, open TreeSize
  6. You should see a File Explorer like window. It will automatically scan your drive and show you the folder sizes.

  1. If you want to look at other drives then click Select Directory and select your desired drive

  1. TreeSize will automatically scan the selected drive as well

That’s it. You can use TreeSize to have a look at the correct sizes. You can use this tool until you get a new Windows Update. So, keep checking for the latest Windows Updates.

terje rosenlund say: Figured it out my self

The problem was indicated by VM-ware which claimed that I did not have privileges to write to my catalog on the d: drive. VM-ware worked as expected but not the Virtual Machine I had downloaded from an UNC-path to the same location

When I checked catalog properties I observed that Attribute 'Write protected' was marked with a blue square inside the check-box. Clicking once in the check-box removed the mark. Pressed Apply and Ok. After closing and reopening the property-window the blue square was back and VM-ware was still complaining about missing privileges. My solution was to download the Virtual Machine once more but this time I placed it on the c:-drive and everything worked as it should. Finaly I removed the Virtual Machine from the d: drive by using explorer as administrator.

When I later looked at the 'Write-protected' problem I accidentally clicked twice in the checbox and was very surpised to se that the blue squere was removed by the first click and became checked by the seccond click! I figured that this probebly ment that the catalog was Not 'Write only' and the reason for the missing privilege had to be something else.

I assume that the reason for my problems may have been caused by downloading the Virual Machine using explorer as Administrator and therefore I had no user-rights to the catalog

My only remaining question was what the .... blue square in the checkbox meant and found this:

that checkbox only serves to set all the files under that folder. Thus will always display the same value regardless.

Very true but also very confusing and I'm shure it would have been bether to leave the box empty to avoid confusion
