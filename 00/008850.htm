You can do it like this. My lookup value in A1 and lookup table in Cols V & W.

=HYPERLINK("mailto:" & VLOOKUP(A1,V1:W27,2,FALSE),VLOOKUP(A1,V1:W27,2,FALSE))

scottgem say:
This starts with your table structure. You need an employees table, a certifications table and a junction table to manage the many to many relationship:

tblEmployees: EmployeeID (Primary Key Autonumber) Names, other employee info
tblCertifications: CertificationID (PK Autonumber), Certification, ExpirationPeriod (in months)
tjxEmpCert: EmpCertID (PK Autonumber), EmployeeID (Foreign Key), CertificationID (FK), CertDate

From there you create a query joining the three tables with a calculated column to calculate the Expiration date:
Expire: DateAdd("m",ExpirationPeriod,CertDate)

Next you set the criteria for the Expire column to:
>DateAdd("d",-45,Date())

That should give the records you want. Just e-mail a report based on that query,

Changing the Product Key for Office 2007 or previous requires you to go into the Registry.

1.    Browse to the following key:
HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Office\<version>\Registration

2.    Now is a good point to make a backup of that key;
Right click on the key, choose Export… and save the file to for instance your Desktop.

3.    Expand the Registration key in the Registry and you’ll see one or more sub keys which contain a lot of characters between curly brackets: { and }
Each sub key stands for an installed Office application.
The proper name for this identifier string is called a “GUID”.

4.    Click on the sub key to reveal its contents in the right pane.
The SuitName key will reveal the name of the package that you have installed. If you have multiple GUID sub keys listed, you can use this key name to determine for which installation you’ll be changing the Product Key.

5.    In the pane on the right, delete the following keys:

o  DigitalProductID

o  ProductID

6.    Close the Registry and start Outlook (or any other Office application); you’ll now be prompted to fill in your Product Key and activate Office again.

a. user say:

I have never done this for real, but no-one else has replied so I can only offer some starting points. Perhaps there are easier ways.

As far as I can tell, Dir and Dir$ no longer work the way you want and you will need to create an Applescript that gets the names for you, and modify your GetFileList function to call it using AppleScriptTask, which is a new function in Mac Office 2016. That means that if you need to distribute your Word document, you will also either need to distribute the necessary Applescript. You may also be able to create it programmatically from VBA, but I have not checked that possibility yet.

Traditionally, you could work around problems with old VBA commands related to file systems by using the Macscript function to execute some Applescript that got the names for you. Because you could specify a complete Applescript program in a text string in Word VBA, you did not need to have a separate Applescript program.

In Mac Office 2016 VBA, Microsoft has been implementing Apple's "sandboxing" rules. Apparently, many things can no longer be done, or can no longer be done the same way.

As a consequence, the chances are that you will need to change other parts of your code that interact with the file system.

To create the script, you can either create it in the Applescript Editor or in a text file. If you are unfamiliar with Applescript or its Editor, it may help to start using it now as youmay find script debugging easier.

As a starting point the text of the script can be like this - this script tries to retrieve the same file names that Dir$ retrieves by excluding files with certain names. Dir actually mangles long file names whereas this script should retrieve them intact.

on mydir(paramString)
  -- paramString must be a valid Posix (Unix) style folder name
  -- "-maxdepth 1" says "only look at the specified folder(s)
  -- "-type f" says "only look at 'ordinary files'", e.g. not
  -- folders. Not sure exactly what is excluded though.
  -- "-not name '~*'" says "exclude files whose names begin with '~'"
  -- these files are normally hidden on Unix and Mac.
  -- Actually, there is just one backslash, but we have to double it up 
  -- in Applescript

  try
    return do shell script "find '" & paramString & "' -maxdepth 1 ¬
    -type f -not -name '~*' -not -name 'desktop.ini' -not -name 'Thumbs.db'"
  on error
    return ""
  end try
end mydir

This script needs to be in the following folder: 

/Users/<yourusername>/Library/Application Scripts/com.microsoft.Word

e.g. if your user name is "barrya"you should put the script in 

/Users/barrya/Library/Application Scripts/com.microsoft.Word

You can either save the script as a text file or as a compiled Applescript. To save as a compiled Applescript from the Applescript Editor, choose the "Script" file formate in the Save dialog. The Editor will use the extension ".scpt" by default. If you choose the "Text" file format, the Editor will use the extension ".applescript". If instead you edit in Textedit, you should use the ".applescript" extension when you save.

Let's say you save your script as a text file called "dir.applescript". Then you can modify your GetFileList function so that it looks like this:

Function GetFileList(folderPath As String) As Collection

    Dim file As String
    Dim returnCollection As New Collection

#If MAC_OFFICE_VERSION >= 15 Then
    Dim s() As String
    s = Split(AppleScriptTask("dir.applescript", "dir", folderPath), vbCr)
    For i = LBound(s) To UBound(s)
      returnCollection.Add s(i)
    Next
#Else
    If Right$(folderPath, 1) <> Application.PathSeparator Then
        folderPath = folderPath & Application.PathSeparator
    End If

    file = Dir$(folderPath)

    Do While Len(file) > 0
        returnCollection.Add folderPath & file
        file = Dir$
    Loop
#End If

    Set GetFileList = returnCollection
End Function

Once you have got the basic thing going, you may need to modify the script so that it excludes more (or fewer) files, but I leave it up to you to find out how to make "find" work in your situation. If you want to use a similar method in Word 2011, you will need to use the Macscript function to call the AppleScript, but you will also need to make some other changes.

EDIT: I forgot to mention that in Word 2016, paths are basically Unix/Posix paths, i.e. the PathSeparator always seems to be "/" rather than ":", and you would need to pass a Unix-style path to your routine for your existing code to work. SO I have assumed that that is what you are doing. There is at least one advantage: you don't have to know the name of the DISK that the folder is on, but you do need to know the name of the top level folder (e.g. /Users, in the Library path I have given).
