luke moraga say:
The basic formula for linear equation is:
y=mx+b

If you want to move the y-intercept to a specific year, you could solve the equation for y by plugging in the serial values for the specific date you want. So, if the original equation is:
y=1.2x-40000

you could solve for y at the serial date 38353 (aka Jan 1, 2005) like this:
y=1.2*(38353)-40000

And your new equation, with the y-intercept based at Jan 1, 2005; would be:
y=1.2x+6023.6

pbezucha say:

Luke's answer is the most straightforward. The same effect would be achieved, however, by plotting in xy-scatter instead of line chart (years as numbers!), only you could have used plain 2005 integer instead of Excel code for 2005/01/01.

 

As a matter of interest, a direct but complicated way to achieve the desired effect is possible, i.e. true trendline for starting year as zero time, while still keeping years in x-axis legend. It requires the mentioned scatter chart and parallel helper range that was constituted as a difference between actual and start year + 2 (2007-2005+2). In the chart where you have drawn the basic series with original primary coordinates you would have to draw another series with such a difference as X-values, and the same number values, but (!) on the secondary axis (Series>Format>Axis>Secondary). Secondary axis y is created to accommodate this new series, but not secondary x-axis. You must do this manually by ticking off item Secondary axis on Chart Options. It is necessary to tune secondary x-axis boundaries; the lower to zero. Now the new series appears graphically identical with the original. You can add a trendlind to this series, and this is just what you wanted. Now, a dull work would follow with erasing redundant objects (i.e. numbers and, eventually, ticks on secondary axes) by declaring them as none - this is of any worth only in routine work.

The primary account that you use on the phone is the first windows live account you enter when prompted to set up your Microsoft account on the phone. By default, all contacts and appointments are set to sync automatically and you cannot choose to sync or not sync those items, so that's why you only see the option to sync emails...that's the only thing from the primary windows live account that you can choose to sync.

 

If you set up a second windows live account on the phone you can choose to sync contacts, appointments and/or email so you'll see options for those three things when you create another windows live account on the phone.

 

So the question remains: Why didn't your contacts sync to the phone?

 

Try this....Go log into your windows live account from a computer. boblarson say:

As for the different file sizes, it is normal because it all depends on what each user does in the database as different actions would result in behind the scenes file changes - like if you make a selection which populates a temp table, the amount of data from one would most likely be different from another person's.  There is much more beyond that but I don't have time, nor the desire and neither the knowledge of the behind the scenes stuff, to try to list more.

 

If they all get the Record Has Been Deleted error, it is likely that you have something that is referencing a deleted record somewhere (could be a persistent filter on a form, for example) which should be tracked down and removed which should fix it for everyone.

 

Instead of having a frontend for everyone in a network location, Gina has already made mention of this but I will give you an example of how to let users always have the latest copy.  Instead of having them open the frontend, give them a batch file for their computer and have a shortcut they use which calls that batch file.  It would then copy down the frontend from the network server and open it.  That way you can make changes to a master copy and then put that master copy in the network location referenced by the batch file when you are ready to deploy.  I also have, on my website, a tool which can make frontends auto updating if you prefer.  If you enable auto updating on the front end, all you need to do is distribute to the users the first time.  Then when you make changes to the master, you change the version number in two tables and copy that file to a network location that is accessible to users.  When they open their copy on their desktop machine, it will check to see if the version is the latest.  If not, it will inform the user that theirs is not the latest and then it will close the database, delete the old file, copy the new file from the server, and reopen for the user.
