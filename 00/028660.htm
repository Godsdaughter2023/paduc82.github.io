ken sheridan say: It's a subquery rather than a query.  The way it works is that the parent form is filtered to those records with a primary key (ProjectID) value which exists in the subform's table in those rows which also contain the value on which you wish to apply the filter in another column, ContactID in my case. It would be possible to base the parent form on a query which includes the subquery, in which the filter criterion is referenced as a parameter (see below), but my demo builds the subquery in code and sets the form's Filter property.

My demo has a projects form in which there is a contacts subform.  In the parent form's header is an unbound combo box in which the contact on which the form is to be filtered is selected.  It's called cboLastname, rather inappropriately because it was copied from a combo box of that name in another form and I forgot to change the name.  It actually allows selection by the contact's concatenated full name, but the value of the combo box is the ContactID in a hidden column, or zero if 'All Contacts' is selected.  The code to filter the form is in the combo box's AfterUpdate event procedure as follows:

Private Sub cboLastname_AfterUpdate()

    Const MESSAGETEXT = "No matching records found."
    Dim ctrl As Control
    Dim strFilter As String
    
    Set ctrl = Me.ActiveControl
    
    strFilter = "ProjectID IN(SELECT ProjectID " & _
        "FROM ContactProjects WHERE ContactID = " & ctrl & ")"
    
    If Nz(ctrl, 0) = 0 Then
        ' turn off filter
        Me.FilterOn = False
    Else
        If Not IsNull(DLookup("ContactID", "ContactProjects", "ContactID = " & ctrl)) Then
            ' filter form to name selected in combo box
            Me.Filter = strFilter
            Me.FilterOn = True
        Else
            ' inform user if no matching records found and show all records
            MsgBox MESSAGETEXT, vbInformation, "Warning"
            Me.FilterOn = False
            Me.Requery
        End If
    End If
       
End Sub

The alternative approach would be to base the parent form on the following query:

PARAMETERS Forms!frmProjects!cboLastname LONG;
SELECT *
FROM Projects
WHERE ProjectID IN
    (SELECT ProjectID
     FROM ContactProjects
     WHERE ContactID = Forms!frmProjects!cboLastname
     OR NZ(Forms!frmProjects!cboLastname,0) = 0)
ORDER BY Project;

The you'd only need one line of code in the combo box's AfterUpdate event procedure:

Private Sub cboLastname_AfterUpdate()

    Me.Requery

End Sub

The code I posted earlier for the parent form's Current event procedure also filters the subform to the selected contact, but in the demo as it stands the code leaves all contacts per project visible, but moves the subform's record pointer to the selected contact.  The code is:

Private Sub Form_Current()

    ' select contact in subform if combo box is not Null
    If Not IsNull(Me.cboLastname) Then
        With Me.fsubContacts.Form
            .RecordsetClone.FindFirst "ContactID = " & Me.cboLastname
            If Not .RecordsetClone.NoMatch Then
                .Bookmark = .RecordsetClone.Bookmark
            End If
        End With
    End If

End Sub

There are other ways it could be done, for instance you could loop through a recordset of the ContactProjects table's rows in code and build a value list of the ProjectID values for those rows with the selected Contact ID.  Rather than building a subquery in code the IN operator would be applied to the value list

ken sheridan say:

I'm a little confused here. Is there a 1:many relationship between the table on the main form and the table in the subform? If there is, then how can you filter by a field in the subform's table? If it is not 1:many, then it would have to be many to many in which case we need to know your table structure.

It doesn't actually matter whether the relationship type is one-to-many or many-to-many as, if the latter, the relationship type is modelled by a table which resolves the relationship type into two (or more) one-to-many relationship types, and it is that table on which the subform would be based, so the relationship type between the parent form's table and the subform's table is one-to-many.  The parent form is filtered to those rows where its primary key exists in the subset of rows in the subform's table in which the value of the other foreign key, or one of them in the case of a ternary or greater relationship type, is that on the basis of which the filter is to be applied.

See my last reply to the OP for the details of the methodology used by my demo, and an alternative, arguably simpler, approach which bases the parent form on a query with a subform which references a parameter.

My demo uses a simple model with a binary relationship type between projects and contacts, but exactly the same methodology would apply in the case of a ternary or greater relationship type, in which case the subquery could reference multiple parameters with the syntax:

(SomeColumn = <some parameter>
OR NZ(<some parameter>,0)  = 0)
AND (SomeOtherColumn = <some other parameter>
OR NZ(<some other parameter>,0)  = 0)
AND......etc
