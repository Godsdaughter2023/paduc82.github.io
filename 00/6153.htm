suzanne s. barnhill say:

If you have enabled "Different First Page" for the headers and footers in your document, then you will have a First Page Header and a primary Header for each section. In the primary Header, you can insert a StyleRef field referencing the style used for your STATEMENTS heading, followed by "CONTINUED."

 

For more information on the StyleRef field, see this Help article. For further details, see http://WordFAQs.mvps.org/StyleRef.htm.

What I'm suggesting is that the way to repeat text on every page automatically is to put it in the header. joeu2004 say: Jim wrote:

Here is what microsoft has to say on ther performance of long vs integer
http://msdn.microsoft.com/en-us/library/aa164754(v=office.10).aspx
The gist of it is to just use long. There is no advantage to integer.


While I agree with the conclusion "just use long", I am concerned about factual misstatements.

"No advantage"?  I agree there is no performance advantage.  But as I stated, type Integer consumes half the storage of type Long, at least in arrays.

This is readily apparent by executing the following macro and looking at Peak Mem Usage in Task Manager.

Sub doit
Dim x(1 to 100000000) as Integer    ' or Long
x(100000000) = 1
End Sub

When x is type Integer, the peak usage on my computer is 221,460K.  When x is type Long, the peak usage is 416,740K.  This is consistent with the fact that peak usage starts at about 26,000K on my computer after Excel with a new (empty) worksheet and entering the procedure above.

As for http://msdn.microsoft.com/en-us/library/aa164754(v=office.10).aspx, it does repeat the assertion that:  "In recent versions, however, VBA converts all integer values to type Long, even if they are declared as type Integer. Therefore, there is no longer a performance advantage to using Integer variables; in fact, Long variables might be slightly faster because VBA does not have to convert them".

But I have learned not to assume Microsoft technical explanations are literally correct.  For example, http://support.microsoft.com/kb/78113 states:  "although Excel can store numbers from 1.79769313486232E308 to 2.2250738585072E-308, it can only do so within 15 digits of precision".  That statement is factually incorrect [1].

I suspect that a Microsoft engineer explained that the CPU loads 16-bit type Integers into 32-bit registers ("type Long"), and a tech writer (probably an English major [2]) dumbed things down by replacing "CPU" with "VBA" and "load register" into "convert".

I cannot say that with impunity because I am not familiar with the VBA internal implementation.   But I have implemented products (not just student projects) like the VB compiler and interpreter, so I am familiar with their design in general.

And it is counter-intuitive that VBA literally converts type Integer to type Long (except in mixed expressions, of course).  Consider the following example:

Sub doit
Dim x as Integer, y as Long
x = 32767
y = x + 1
End Sub

That results in an overflow error because 32767+1 exceeds the 16-bit limit of 32767, even though the result would fit into the 32-bit type Long variable.

My explanation readily describes how that happens.  VBA does 16-bit loads of x and 1, and the CPU raises the carry-out flag based on a 16-bit add operation.

If VB(A) literally converted x to type Long, and especially if it performed type-Long addition, it would have to do a lot of work to detect the 16-bit overflow.

I think Occam's Razor should apply here.

Anyway, we are beating a dead horse.  I think we are all in "violent agreement" that type Long is the integer type of choice, except in the unusual case where we might be concerned about memory consumption.


-----
[1] Regarding Excel "can only [store numbers] within 15 digits of precision", consider this....    Put the formula =2^53 into A1, the formula =A1-1 into A2, and the formula =A1-A2-0 into A3.  Note that A3 correctly reflects the difference of 1, even though 2^53 requires 16 digits to be represented exactly (9,007,199,254,740,992) and for Excel to subtract 1 with correct 1 results.

[2] I mean no disrepect by referring to the tech writer as an "English major".  I am merely speaking from personal experience.  When I first started in the business, computer tech writers were people with an engineering background with some flare for writing.  Sometimes, they were the implementer him/herself.  A decade or two later, tech writers were often literally English majors with some flare for math or science, but not necessarily experts in the field they were writing about.

joeu2004 say: kala sha jahan wrote:
in my spreadsheet, i have a cell that has a value stored in cell A1.  as the macro is executed, the numerical value increments by 1.

should the variable as declared as Long or Integer, or something else?

i have the following:

Dim Mycount as Long

    MyCount = Range("C9") + 1
    Range("C9") = MyCount

I had the variable as Integer, but when the macro was executed, the value in C9 was set back to 0.


The only way I can imagine that might happen is if C9 contained 32767 to begin with.  But in that case, you should have gotten a VBA runtime error.

And of course when C9 is -1.  But in that case, changing to type Long would not change anything except by coincidence (C9 was not -1 when you tried type Long).

Nevertheless, between type Integer and Long, the "right" answer is:  it depends on how large you expect C9 to become.

I tend to always use type Long [*].  Then I usually don't have to worry about integer limits.  That allows C9 to be as large as 2,147,483,647.

If I need C9 to be even larger, I use type Double.  Caveat:  with type Double, the largest integer than can be represented exactly is 9,007,199,254,740,992.  Excel will display it as 9,007,199,254,740,990.  And incrementing by 1 will result in no change.

Better still, why not simply:

Range("C9") = Range("C9")+1

In effect, that does treat C9 as type Double, since Excel effectively uses type Double for all numbers.

-----
[*] The only time I might consider using type Integer is if I am using huge amounts of storage, like an array with 100 million elements.  Even then, I still tend to use type Long as a matter of habit.  Memory is "cheap" these days.
