This will return TRUE if any of the first 8 characters are not a number

Impact of extra comma in formulas - Microsoft Community

anankira say:

Impact of extra comma in formulas

I was calculating averages of noncontinuous values in a spreadsheet. Sometimes, when I selected the cells to be averaged, there would be an extra comma. Ex: instead of =AVERAGE(C5:C15,C19:C22) I would have =AVERAGE(C5:C15,C19:C22,). These two formulas result in very different numbers despite the fact that the same data are selected. Why is that?
mike h.. say:

Hi,

When you have an extra comma at the end like that with no value or cell after it then Excel interprets that as a zero and averages all the cells in that range and in addition a zero so in effect the formula is this

=AVERAGE(C5:C15,C19:C22,0)

You need to be careful not to include additional commas, the correct formula here is this

=AVERAGE(C5:C15,C19:C22)

hansv mvp say:

The formula =AVERAGE(C5:C15,C19:C22,) is equivalent to =AVERAGE(C5:C15,C19:C22,0). In other words, it is as if you added a 0 value to the data to be averaged.

ken sheridan say: You can save yourself a lot of headaches and unnecessary code by correct data modelling.  What you have is a many-to-many relationship type between children and parents.  If you are modelling the data over a single generation, then you can have separate Children and Parents tables; if you are modelling the data over multiple generations then you would have a single People table and the relationship type would be between two instances of the same table, as a child can also be a parent.

These sort of relationship types are very common in a variety of contexts other than personal ones, and the methodology is well established.  The relationship type is modelled by a table which has two foreign keys referencing the primary key of each referenced table (or of the one referenced table where the data is being modelled over more than one generation).  In your case the relationship type is ternary as it also references a ParentCategories table which would have rows for each category, Biological mother, Biological father, Social mother and Social father.

I'll assume for this example that two referenced tables are being used for children and parents, in which case the tables would, in broad outline, be as follows:

Parents
….ParentID  (PK)
….FirstName
….LastName

Children
….ChildID  (PK)
….FirstName
….LastName

ParentCategories
….ParentCategoryID  (PK)
….ParentCategory

And to model the relationship type:

Parentage
….ParentID  (FK)
….ChildID  (FK)
….ParentCategoryID  (FK)

The relationships must be enforced in each case of course.

As in your case you appear to be modelling only one set of social parents per child, rather than successive foster parents for instance, the Parentage table has two candidate keys, both composite, assuming that a person can be of only one parent category in relation to each child.  These are ParentID/ChildID, and ChildID/ParentCategoryID.  ParentID/ParentCategoryID is not a candidate key of course as a parent can be of the same category in relation to more than one child. ParentID/ChildID would be a logical choice for the primary key, with the other being included in a single unique index, though the choice of which candidate key to make the primary key is really an arbitrary one.

In each of the above tables further non-key columns can of course be added to represent other attributes of the entity type (a relationship type is just a special king of entity type), provided that each attribute is functionally determined solely by the table's primary key.

The interface for this classic model is a very simple one of a children parent form, in single form view, within which is embedded a parentage subform, in continuous forms view, linked to the parent form by setting the LinkMasterFields and LinkChildFields properties of the subform control in the parent form's Controls collection to Child ID.  The subform would contain two combo boxes bound to the ParentID and ParentCategoryID columns respectively.  It is then simply a matter of entering up to four rows in the subform, selecting a parent and category in each case.  The constraints of the primary key and unique index will prevent invalid rows being entered.

In the parent form an unbound navigational combo box can be included to go to a child.  The code for its AfterUpdate event procedure would be along these lines, taken from one of my on-line demo files:

Private Sub cboGotoContact_AfterUpdate()

    Const MESSAGETEXT = "No matching record"
    Dim ctrl As Control
    
    Set ctrl = Me.ActiveControl
    
    If Not IsNull(ctrl) Then
        If ctrl = 0 Then
            ' go to new record and move focus to FirstName control
            DoCmd.GoToRecord acForm, Me.Name, acNewRec
            Me.FirstName.SetFocus
        Else
            With Me.RecordsetClone
                .FindFirst "ContactID = " & ctrl
                If Not .NoMatch Then
                    ' go to record by synchronizing bookmarks
                    Me.Bookmark = .Bookmark
                Else
                    MsgBox MESSAGETEXT, vbInformation, "Warning"
                End If
            End With
        End If
    End If
    
End Sub

The RowSource property for the combo box in the above case is:

SELECT ContactID, FirstName & " " & LastName, 1 As SortColumn,
LastName,Firstname  
FROM Contacts  
UNION
SELECT 0, "<New Contact>", 0,"",""  
FROM Contacts
ORDER BY SortColumn, LastName, FirstName;

Its BoundColumn property is 1, its ColumnCount  property is 2, and its ColumnWidths property is 0cm to hide the first column.

rich007a say:

This will do what you want:

 

Sub UpGreenDownRed()
    Dim cht As Chart
    Dim ser As Series
    Dim i As Long
    Dim varY As Variant
   
    Set cht = ActiveChart
    Set ser = cht.SeriesCollection(1) 'SET SERIES NUMBER TO BE CHANGED
   
    varY = ser.Values
    For i = LBound(varY) + 1 To UBound(varY)
        With ser.Points(i)
            If varY(i) < varY(i - 1) Then
                .Border.Color = RGB(255, 0, 0) 'Red
            ElseIf varY(i) > varY(i - 1) Then
                .Border.Color = RGB(0, 255, 0) 'Green
'            Else 'Same value
'                .Border.Color = RGB(0, 0, 255) 'Blue
            End If
        End With
    Next i
End Sub

 

but note that if you only have one series, the legend will be changed to have one legend entry per point!  If that is not desired, plot a second series, with no lines or markers, and delete that "ghost" series' legend entry from the legend.

To use the above, in Excel press Alt+F11 to open the VBE, then Insert > Module, then paste the above code.

Go back to Excel, then do View tab > Macros... select the UpGreenDownRed macro in the list and click Run.

Hope that helps.
