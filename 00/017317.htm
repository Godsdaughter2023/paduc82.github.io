Re:  "...interrupt a macro that is being executed?"

The Escape (Esc) key works sometimes.
The way a macro is written affects whether it can be interrupted.
Liberal use of DoEvents is required, particularly within a Loop.

Interrupting "nicely" requires the use of "Application.EnableCancelKey = xlErrorHandler" at the beginning of code.  This allows one to trap for Error 18 and exit.

joeu2004 say:
How could I interrupt a macro that is being executed?  In good old days we had the CTRL-BRK combination, but now I am not sure anymore.

 

I am using Excel 2010 on Windows 7 platform.  On my laptop keyboard I do not see an explicit BREAK button.  I do see a SysRq and Pause button; and I have used them in combination with CTRL but nothing happens.  So I am guessing that the key combinations have changed.


As you discovered, the actual keystroke to use depends on the computer manufacturer, not the version of Excel or Windows.

On my (ancient) Dell laptop, I press ctrl+alt+Pause to have the option to End or Debug (break) a running macro.  I can press ctrl+Pause to simply break a running macro.

However, the success of breaking a macro depends on what it is doing at the time, and perhaps on how many CPUs (virtual and real) you have.

On my single-core single-CPU computer, if the macro is in a tight CPU-bound loop, often I cannot break it or it takes a v-e-r-y long time.  Also, I believe we cannot break a macro if it is waiting for input or the network, for example.

The prudent use of calling DoEvents in lengthy calculation loops will improve computer usage in general, as well as improving yours chances of breaking a running macro.

j.e. mcgimpsey say: By design, AutoRecover files are deleted when XL closes normally. AutoRecover only saves incremental changes, so once the file is saved, there's nothing to keep around. (Office no longer has an autosave capability.)

It sounds like XL closed rather than crashed. If you didn't save it since Friday, and AutoRecover didn't present a restored copy, it's unlikely that you'll be able to recover the data. There are data recovery services available (Google them), but they'll cost you much more than $50.

If you use Time Machine, you may be able to recover your AutoRecoverfiles, but I can't vouch for whether simply restoring them will cause XL to generate a recovered file.

One obviously should never rely on AutoRecover. And saving once a week is FAR too infrequent. It only takes a half-second to type CMD-s.

It will be interesting to see what happens with Mac OSX Lion - that's supposed to eliminate the need to manually save. I doubt I'll trust it...

j.e. mcgimpsey say: Sounds to me like XL is trying to automatically open another file that contains macros -

Check Preferences/General and make sure that the At Startup, open all files in:  input box is cleared.

Check that the HD/Applications/Microsoft Office 2011/Office/Startup/Excel folder doesn't have any files.

If you're using MacOSX Lion, quit XL holding the OPT key down to prevent Restore from trying to open all files open when the app closes.
