You could create a Before Update event procedure for the form.

Private Sub Form_BeforeUpdate(Cancel As Integer)
    If Me.Outcome Like "DMC*" Or Me.Outcome Like "MQL*" Then

SeqNo is the sequential number field, of type Number (Long Integer).

ken sheridan say: Adding the number is simple enough.  In the AfterUpdate event procedure of the Outcome control look up the last number used and add 1, assigning the value to the relevant column, which I'll call CallNumber for this example if one of the relevant values has been selected:

If Not IsNull(Me.Outcome) Then
    If Left(Me.Outcome,3) = "DMC" Or Left(Me.Outcome,3) = "MQL" Then
        Me.CallNumber= Nz(DMax("CallNumber", "Call Log"),0)+1
    End If
End If

The real problem is in preventing duplicates in a multi-use environment where two or more users are inserting a row simultaneously.  Normally, this would be done by indexing the column uniquely and handling the index violation error.  This cannot be done in your case, however, as some columns will either have a value of 0 or be Null.  I'd recommend the former, by setting the DefaultValue property of the column to 0 in the table design.

The best solution would be to apply a CHECK CONSTRAINT to the table, but for this you would need to know how to write and execute the ALTER TABLE statement.  An alternative solution would be to validate and adjust the number in the form's BeforeInsert event procedure.  This would be done by looking for an instance of the same number in another row, i.e. a row with a different primary key value to the current row:

Private Sub Form_BeforeUpdate(Cancel As Integer)

    Dim strCriteria As String
    
    strCriteria = "CallNumber <> 0 And " & _
        "CallID <> " & Me.CallID & " And " & _
        "CallNumber = " & Me.CallNumber
        
    Do While Not IsNull(DLookup("CallID", "Call Log", strCriteria))
        Me.CallNumber = Me.CallNumber + 1
        strCriteria = "CallNumber <> 0 And " & _
            "CallID <> " & Me.CallID & " And " & _
            "CallNumber = " & Me.CallNumber
    Loop
        
End Sub

This is not entirely bulletproof, but you would have to be extremely unlucky for two users to save the new record at exactly the same instant so that the code did not pick up the duplication.

ryanh_7 say:

{ IF { = OR ( { COMPARE { MERGEFIELD Field_1 } <= "Value" }, { COMPARE { MERGEFIELD Field_2 } >= "Value" } ) } = 1 "True Instructions" "False Instructions" }

But I haven't been able to find confirmation if there is an actual limit or if I'm just missing something.

macropod say:

Word's OR and AND fields are limited to two fields each. However, you can test multiple conditions via field maths. For example:

{IF{={IF{MERGEFIELD Field_1}= "Value" 1 0}*{IF{MERGEFIELD Field_2}= "Value" 1 0}*{IF{MERGEFIELD Field_3}= "Value" 1 0}*{IF{MERGEFIELD Field_4}= "Value" 1 0}}= 1 "True" "False"}

Functions as an AND test for four values and will only output "True" if all four are true. You could get the same results with:

{IF{={IF{MERGEFIELD Field_1}= "Value" 1 0}+{IF{MERGEFIELD Field_2}= "Value" 1 0}+{IF{MERGEFIELD Field_3}= "Value" 1 0}+{IF{MERGEFIELD Field_4}= "Value" 1 0}}= 4 "True" "False"}

which is actually the equivalent of an OR test for the same four values. Used conventionally, though, the OR tests would be more like:

{IF{={IF{MERGEFIELD Field_1}= "Value" 1 0}+{IF{MERGEFIELD Field_2}= "Value" 1 0}+{IF{MERGEFIELD Field_3}= "Value" 1 0}+{IF{MERGEFIELD Field_4}= "Value" 1 0}}> 0 "True" "False"}

and would output "True" if any one or more is true.

You can nest up to 20 AND/OR conditions this way and you can include various AND/OR combinations within the same field construct.

Note: The field brace pairs (ie '{ }') for the above examples are created in the document itself, via Ctrl-F9 (Cmd-F9 on a Mac); you can't simply type them or copy & paste them from this message. Nor, apart from the actual mergefields (which would appear as «Field_1» instead of {MERGEFIELD Field_1}), is it practicable to add them via the standard Word dialogues.

joel ivory johnson say:

Note that an application doesn't have to be actively running in the background for a push notification to appear. 

This information is a little more developer focused, but an application can register for push notifications in a way that allows an external server to send them. To save battery and for other reasons these notifications are routed through Microsoft machines that queue up the notifications and ensure that there's not a flood of them coming through to your device. A messaging server (WhatsApp in this case) could send the push notification message to a Microsoft server and your phone will be notified of it. This happens without the associated application running (and battery is also saved since the application doesn't have to be running).
