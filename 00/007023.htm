ken sheridan say:

he only difference from my 2nd form to my first form is the 2nd form has a query based on two tables.

A form can be based on a query which joins two or more tables provided that data is being entered or edited in the referencing table only.  Take the following example of a form based on a query which joins the following tables in a simple linear hierarchy:

Customers>----Cities>----States

In this form data can be entered into the Customers table, including into a foreign key CityID column, but not into the Cities or States tables.  A control bound to the State columns from States can be included in the form, but this control would be read-only.  Once a CityID value is entered into Customers, via a combo box in which the first column (CityID) is hidden and the second column(City) is visible, the state in which the city is located will automatically show in the read-only control.

In situations where it is necessary to enter data into two related tables the form should not be based on a query which joins the two tables.  A subform should be embedded in the form.  Taking the example of an Orders table and an OrderDetails table, a form, in single form view, would be based on the former and a subform within it, in continuous forms view, would be based on the latter.   The subform would be linked to the parent form on OrderID.  For each order entered into the parent form as many order details rows as necessary can be entered into the subform, one row for each item ordered.
