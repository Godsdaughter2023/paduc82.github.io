ken sheridan say:

Why do you think more than one table is necessary?  You simply insert a new row into the table via a bound form each time a call is logged.

To return the latest call you'd use a query restricted by a subquery:

SELECT *
FROM CallLog AS CL1
WHERE StartTime =
    (SELECT MAX(StartTime)
      FROM CallLog AS CL2);

To return the latest call per LeadID you'd correlate the subquery with the outer query:

SELECT *
FROM CallLog AS CL1
WHERE StartTime =
    (SELECT MAX(StartTime)
      FROM CallLog AS CL2
      WHERE CL2.LeadID = CL1.LeadID)

ORDER BY LeadID;

I agree with Ken, you should have a single Calls table that records each call. To add a call you add a new record in the form. I'm not sure why you have it set to overwrite an existing record.

ken sheridan say: We are not suggesting that you have only one table in the database, but only one call log table, and rather than overwriting data in that table, you'd insert a new row for each call.  You have two options for this via a form bound to the table:

1.  Show all records in the form and navigate to a new one, either via the built in navigation bar at the bottom of the form, or via a button in the form, which the button wizard can set up for you.

2.  Set the form's DataEntry property to True (Yes).  The form will then open empty and will only allow you to enter new records.  You'd need another wat to view existing records of course, which could be another form or a report.

A simple call logging database would, in broad outline, comprise tables along the following lines:

Employees
....EmployeeID  (PK)
....FirstName
....LastName
....etc

Callers
....CallerID  (PK)
....FirstName
....LastName
....etc

CallCategories
....CallCategoryID  (PK)
....CallCategory

The call log is a ternary many-to-many relationship type between the above tables and is modelled by a table like this which resolves the relationship type into three one-to-many relationship types:

CallLog
....EmployeeID  (FK)
....CallerID  (FK)
....CallCategoryID  (FK)
....StartTime
....EndTime
....Subject

The Callers table would, in a corporate environment, also include a CompanyID or similar column, as a foreign key referencing the primary key of a Companies table or similar.
