How to prevent email spoofing in O365 - Microsoft Community

melvin gante say:

How to prevent email spoofing in O365

Hello,

Just want your suggestion on how to prevent email spoofing in our domain in O365 from example emkei.

Thank,

Melvn

vincentchoy say:

The framework that many security companies use is SPF, DKIM and DMARC.

Essentially, security software on the recipient side need some way to tell if the emails they are receiving "from you" are legitimate.

So as a sender you put some settings into your DNS to allow the recipients security software to make a better assessment if the email they are getting really came from your email servers (SPF), and are not modified along the way (DKIM). And if they find something suspicious, what should the recipient security software do and where do they notify you (DMARC).

SPF is already set in your DNS when you provisioned Office365.

Office365 turns on DKIM by default, but in a round about way (as they do not have access to your DNS to set it properly). To properly set DKIM you need to insert the correct DKIM entries into your DNS and manually turn on DKIM signatures in Office365.

Then you set DMARC in your DNS to monitor if any of your email addresses may be used for spoofing. You will also need a report tool to help you manage the DMARC reports you will be getting (Eg Dmarcian).

Do note that certain services, such as third party email marketing, third party disclaimer services etc, might cause false positives to the framework. These services either generate mail that does not appear to directly come from you, or they might insert additional information (as in a auto disclaimer service) which the anti-spoof may consider as modified while in transit. These can be overcome by setting DMARC to "Take no action upon" and monitor the DMARC reports first.

This might sound complex, but after spending sometime understanding the framework and how it works you may get more clarity.

Come back for further questions if you need to.

Private or Public Functions? - Microsoft Community

jamesjoey say:

Private or Public Functions?

I have several Functions in a Module. (Not a form module).

I notice that some are simply Functions and others are Public Functions.

Any rule of thumb as far as whether they should be Public or not?

James

ken sheridan say: Declaring a function in a standard module Public exposes it throughout the database, so it can be called from queries, forms, code in other modules, etc.  Declaring a function in a standard module Private exposes it only within the module, so should be done where a function is called by another function or procedure in the same module, but not by anything outside the module.   The same name can be used for Private functions in different modules, but a Public function's name must be distinct within the whole database.
