tushar mehta say:
Obviously (from web searching) there is no actual constant string array.  The global string array is "effectively" constant if programmers remember not to change it.  From web searching, such an array must be initialized in Private Sub Workbook_Open().  However, if I click Reset while a macro is stopped, the constant array ceases to exist.  I have to browse to Private Sub Workbook_Open() again, run that, then run my macro.  Is there a more permanent way to assign the strings to a global array?  Constant string arrays would sure come in handy.


The simple solution is to create a function (or even a property get) that returns what you want.  Then, all your references are to this 'constant' function.  Here's an example.

 

Suppose you want the names of the months in an array.  Then the function MonthList or the property MonthGet give the required information.

Function MonthList() As String()
    MonthList = Split("Jan,Feb,Mar,Apr,May,Jun,Jul,Aug,Sep,Oct,Nov,Dec", ",")
    End Function
Property Get MonthGet() As String()
    MonthGet = Split("Jan,Feb,Mar,Apr,May,Jun,Jul,Aug,Sep,Oct,Nov,Dec", ",")
    End Property
Sub testML()
    Debug.Print MonthList()(0)
    Debug.Print MonthGet()(0)
    End Sub

For more on using properties in a standard module see

Case Study – Property procedures in a standard module

http://www.tushar-mehta.com/publish_train/xl_vba_cases/vba-property_procs_in_std_module.htm

 

 

jim thomlinson say: When you click reset both the stack and the heap are purged. In that instance all varaibles are cleared. The same is true of using the stand alone line End in your code or hitting an unhandled error. The only way to get your array back is to run code that will re-initialize the array. Generally speaking for this type of thing I do not initialize the array on open. I write a function that returns the array. The array is stored in the function as a static variable. If the array is blank then it fills the array. Generally this will only happen once but if anything goes wrong it will automatically right itself. The overhead on this is very low...

marshall barton say:

If you are using the Sum fubction in a Totals type query or in a form/report header/footer section, then the problem is probably because you are manually adding up the individual displayed values, which are rounded, and comparing that to the totalm which is not rounded until after the total is calculated.  In this case the total is the most accurate value and your manual calculation can be off by quite a bit.

OTOH, if you are writing checks then the individual rounded values is what you write on the check and the total need to be a sum of the rounded values.  That means instead of using an expression like  =Sum(Amount)  you need to use  =Sum(Rnd(Amount, 2))

If you are using Single or Double type number, but only need 4 decimal places, change to Curreny data type.

If that diesn't help, post a Copy/Paste of how you are summing the field and provide more details about how the numbers are off.
