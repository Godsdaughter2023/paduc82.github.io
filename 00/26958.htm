How to troubleshoot Office issues that happen randomly - Microsoft Community

erik davidsson (msft) say:

How to troubleshoot Office issues that happen randomly

Target audience: Expert

This article discusses some ideas that can be useful for getting more information about issues that happen randomly, and therefore are hard to get a better understanding of. The article is structured into several questions, and recommendations for each.


Did the issue you are investigating happen in the past, but you are not 100% sure what the errors/messages said, or what happened then?
If so, you may be able to "go back in time" by using the Microsoft Office Alerts event log:


(To reach the Event Log, press the Start button, then search for and start the Event Viewer)

Here, you can see most messages that Office has given you, such as "Want to save your changes to FILENAME?" in the screenshot above. You can also see the exact time when the message came up, and the version of Office that you had. You may be able to use this to e.g. figure out when the first time was that a specific error came up, or what happened just before and after the error came up.


Is the issue a crash or a hang that is occuring randomly, that you don't manage to get more information about?

If so, you can always make use of the "Application" event log. See the screenshot below:

As seen, by searching for "Faulting application name: EXCEL.EXE", I could find the last time the application crashed. I can also see the "Faulting module name" in the screenshot above that was the module it crashed in. If you see a 3rd party DLL or EXE there, it is probably caused by that one.

The "Fault offset" is interesting too. If I find that it almost always has crashed with the same "Faulting module" or even with the same "Fault offset", I may start to suspect that it is a generic issue. Sometimes, the "Fault offset" can be found when searching online, and in this way, potential matches to existing discussions could be identified.


Another interesting thing is to look at is the event that normally is logged directly after the crash. This can contain information about what files were open when it crashed. If you find that it seems to always mention the same file name there, you may want to clean that file up. If it is an Excel file that needs cleanup, refer to this.

To summarize, using the event logs can help you identify a pattern to the problems.



If you are a System Administrator and want to use this type of logic on a large scale, you can implement Office Telemetry in your organization.


Is the issue related to files/folders/registry keys randomly being deleted or changing permissions?
If so, you may want to set up some form of Audit on the suitable objects, so that you the next time it will happen afterwards can get information about what changed it exactly. See more information about that here.
