scottgem say:

Yes, you are approaching this from the wrong angle. First, you should have an Accounts table with a list of your accounts. That table should have an autonumber AccountID. Next you should have a Part Number table with a listing of all your parts with a PartNumberID. Then you pricing table would look like this:

tblPricing

PricingID (PK autonumber)

AccountID (FK)

PartNumberID (FK)

Price

EffDate

Now to search by Account, you would create a Continuous form bound to tblPricing. You can use Combos so it shows the Account Names and part Number instead of the autonumber PKs. 

In the header of that form you have an UNBOUND combobox that loos up from the Accounts table so you can select an Account. In the After Update event of that combo you have code like:

Me.Filter = "[AccountID] = " & Me.cboAccount

DoCmd.Runcommand acCmdApplyFiltersort

The form will now list pricing just for that account.

donleverton say:

Yikes! A different price for EVERY part for EVERY customer?

When I worked in an auto parts store, we had pricing matrices.

Waaaay back when these price lists were printed on paper, they were color-coded.

Retail was white (W)

Trade was Yellow (Y)

Dealer was Grey (G)

Stocking Dealer was Green (N)

..etc

When things got computerized, these designations "stuck"

Matrix:.

When our accounts finagled a "Better Deal" it became fairly easy to discount any of the pricing levels (eg: "Y-5" indicated a 5% discount off of the manufacturers Suggested Trade price.

But then the market and competition brought even more complication.

Here's an example:

There is a Product Line code ... i.e "NGK" for NGK spark Plugs

Within that group, you had "Families" which were numerical. 01 was probably standard plugs, 05 might have been Platinum, 23 might have been Iridium for example.

So now you might have a pricing profile that priced plugs at different discounts according to "Family".

As you can imagine, even building / maintaining a price profile on a "per customer" basis became quite a chore.

The next step was to develop generic key pricing groups. These were profiles that could refer to a pre-configured profile. eg "Preferred Trade" was assigned to multiple customers, which kind of assured that businesses belonging to this group all paid the same price.

The bottom line here is that I believe that it's much easier to design a few price matrices and assign customer pricing based on them rather than what you are suggesting. Yes it takes some time and effort to set these up, but the results pay for it in the long run as you have a bit better control of your Gross Profits ... which is what pays to keep the lights on at your business.

a. user say:

Here's something you can try, but be careful, as I have only tried this approach with very simple autocorrects, and only tested on Word 2010 on the Windows side. 

If you have any autocorrects that use more complex rich text, it may not work - for one thing, some Word constructs cannot be inserted in a table and since this approach relies on putting all the entries in a table, that would make the approach fail. It could of course be improved, but I expect there's better code out there already - its probably a question of knowing what to look for.

If you aren't familiar with Word macros, I suggest you visit 

http://word.mvps.org/FAQs/MacrosVBA/CreateAMacro.htm
for some hints on how to set them up in WinWord, and

http://word.mvps.org/Mac/InstallMacro.html

for the MacWord equivalent. There are some steps missing in both those descriptions as you will probably first have to enable the Developer tab on the ribbon in Word 2007/2010/2011.

Use the first macro (see below) to create a list of autocorrect entries in a Word document on your Windows system.

Inspect the document (just to be sure it all makes sense), then save it e.g. as WinWordAutoCorrects.docx and close it.

Copy that to your Documents folder on your Mac.

Make sure your Mac is properly backed up.

Then make a separate backup of your Mac AutoCorrect entries.  Use the same macro in Mac Word 2011 to save the existing Autocorrects from Mac Word 2011. Inspect the resulting file, and save it as MacWordAutoCorrects.docx and close it.

Then fix the second macro so that the line 

Set objDocument = Documents.Open("Macintosh HD:users:yourname:Documents:winautocorrects.docx")

has the correct pathname in it.

> Will this autocorrect file then work in applications such as word/excel? (version 2011).

If that works, it should work in Word 2011. However, AFAIK Excel has a separate set of autocorrects - in principle, you would need to copy those across separately. I don't have code to do that - I expect Excel VBA code to import autocorrect entries would be similar, but the Word and Excel object models are not identical in this area.
'The first macro:

Sub exportautocorrects()
Dim objACE As Word.AutoCorrectEntry
Dim objDocument As Word.Document
Dim r As Word.Range
Dim objTable As Word.Table
Dim i As Integer
Set objDocument = Documents.Add(Visible:=True)
Set objTable = objDocument.Tables.Add(objDocument.Content, Word.AutoCorrect.Entries.Count + 1, 4)
With objTable.Rows(1)
  .Cells(1).Range.Text = "ID"
  .Cells(2).Range.Text = "from"
  .Cells(3).Range.Text = "richtext"
  .Cells(4).Range.Text = "to"
End With
For i = 1 To Word.AutoCorrect.Entries.Count
  Set objACE = Word.AutoCorrect.Entries(i)
  With objTable.Rows(i + 1)
    .Cells(1).Range.Text = CStr(i)
    .Cells(2).Range.Text = objACE.Name
    .Cells(3).Range.Text = CStr(objACE.RichText)
    Set r = .Cells(4).Range
    r.Collapse Direction:=wdCollapseStart
    objACE.Apply r
    Set r = Nothing
  End With
Next
Set objTable = Nothing
objDocument.Activate
Set objDocument = Nothing
End Sub
 
'The second macro:

Sub importautocorrects()
Dim objACE As Word.AutoCorrectEntry
Dim objDocument As Word.Document
Dim r As Word.Range
Dim objTable As Word.Table
Dim i As Integer
' put the correct "yourname" in the following line
Set objDocument = Documents.Open("Macintosh HD:users:yourname:Documents:winautocorrects.docx")
Set objTable = objDocument.Tables(1)
For i = 2 To objTable.Rows.Count
  With objTable.Rows(i)
 
    If Left(.Cells(3).Range.Text, 4) = "True" Then
      Set r = .Cells(4).Range
      r.SetRange r.Start, r.End - 1

      Word.AutoCorrect.Entries.AddRichText Left(.Cells(2).Range.Text, Len(.Cells(2).Range.Text) - 2), r
      Set r = Nothing

    Else
      Word.AutoCorrect.Entries.Add Left(.Cells(2).Range.Text, Len(.Cells(2).Range.Text) - 2), _
        Left(.Cells(4).Range.Text, Len(.Cells(4).Range.Text) - 2)
      End If
  End With
Next
Set objTable = Nothing
objDocument.Activate
Set objDocument = Nothing
End Sub

bill mosca say:

Tiffany

What you are asking for is called cascading combo boxes. You use the first one
to select the category. You put code in the AfterUpdate event of that combo box
to change the Rowsource of the subcategory.

Let's say the Categories table has two fields:
    CategoryID autonumber primary key
    Category text
The SubCategoryies table has:
    SubCategory ID AutoNumber Primary Key
    CategoryID Long (foreign key to Categories table)
    SubCategory Text

The first combo has the categories and has the CategoryID as its bound column.

An example would look like this:
[[vbnet]
Private Sub cboCategoryID_AfterUpdate()
    Dim strSQL As String
    strSQL = "SELECT SubCategoryID, SubCategory " _
        & "FROM tblSubCategories " _
        & "WHERE CategoryID = " & me.cboCategoryID
    Me.cboSubCategory.RowSource = strSQL
    Me.cboSubCategory.Requery
End Sub
[/vbnet]

e.j. gun say:

The updated code below will create a new workbook and populate it with the worksheets from the "master" that have a match to the search text.  Try it out and let me know how it does.

Eric

Option Explicit

Public Sub ExtractWorksheets()
    Dim i As Long
    Dim ws As Worksheet, Found As Range
    Dim myText As String, FirstAddress As String
    Dim AddressStr As String, foundNum As Integer
    Dim searchWS As String
    Dim foundWS() As String
    Dim numWS As Long
    Dim newWB As Workbook, newShts As Long

    searchWS = ActiveSheet.Name

    numWS = 0
    ThisWorkbook.Worksheets(searchWS).Cells.ClearContents
    ThisWorkbook.Worksheets(searchWS).Cells(1, 1) = "Search String:"
    ThisWorkbook.Worksheets(searchWS).Cells(1, 2) = myText
    ThisWorkbook.Worksheets(searchWS).Cells(2, 1) = "Sheet Name"
    ThisWorkbook.Worksheets(searchWS).Cells(2, 2) = "Address of Found Item"
    For Each ws In ThisWorkbook.Worksheets
        With ws
'Do not search searchWS!
            If (Not ws.Name = searchWS) Then
                Set Found = .UsedRange.Find(what:=myText, LookIn:=xlValues, LookAt:=xlPart, MatchCase:=False)
                If Not Found Is Nothing Then
                    numWS = numWS + 1
                    ReDim Preserve foundWS(1 To numWS)
                    foundWS(numWS) = ws.Name
                    FirstAddress = Found.Address
                    Do
                        foundNum = foundNum + 1
                        ThisWorkbook.Worksheets(searchWS).Cells(foundNum + 2, 1) = .Name
                        ThisWorkbook.Worksheets(searchWS).Cells(foundNum + 2, 2) = Found.Address
                        Set Found = .UsedRange.FindNext(Found)
                    Loop While Not Found Is Nothing And Found.Address <> FirstAddress
                End If
            End If
        End With
    Next ws
'
' Finished searching.  If any worksheets contained the search text,
' create a new workbook, copy the worksheet(s) to the new workbook,
' and stop.
'
    If (numWS > 0) Then
Application.DisplayAlerts = False
        Set newWB = Application.Workbooks.Add
        newWB.Sheets(1).Name = "ZZZZZZZZZZZZZ"
        newShts = newWB.Sheets.Count
        For i = newShts To 2 Step -1
            newWB.Sheets(i).Delete
        Next i
        For i = 1 To numWS
            ThisWorkbook.Worksheets(foundWS(i)).Copy after:=newWB.Sheets(newWB.Sheets.Count) ' Add at end
        Next i
        newWB.Sheets(1).Delete
        newWB.Sheets(1).Activate
        Set newWB = Nothing
Application.DisplayAlerts = True
    End If

        MsgBox "Unable to find " & myText & " in this workbook.", vbExclamation
    End If
End Sub

jay freedman say:
No, "they" did not change the shortcut, although an add-in might have. The Ctrl+m shortcut is still assigned to the Indent command by default.

You can reassign it easily:
  • Click File > Options > Customize Ribbon. At the bottom left of the dialog, click the Keyboard shortcuts: Customize button.
  • In the Customize Keyboard dialog, scroll to the bottom of the Categories list and click All Commands.
  • In the Commands list, scroll down to the Indent command and click it.
  • Click in the "Press new shortcut key" box and press Ctrl+m.
  • Click the Assign button at bottom left.
  • Click the Close button, and then OK in the Options dialog.
