Hi Anita,

 

It’s an expected behavior that Microsoft Outlook client archive email message based on the received date or the last modified data and time, whichever is later. After you set a flag on an email message, the modified data is changed. You can press Alt + Enter to see the details of the email message.

 

If you want to make Outlook 2016 client archive emails by received data, please perform the following steps:

1. Press Win + R keys simultaneously to open the Run dialog box.

2. In the Run dialog box, type Regedit to open the Registry Editor.

3. Navigate to HKEY_CURRENT_USER\Software\Microsoft\Office\16.0\Outlook\Preferences

4. Please right click the Preference in the tree structure, select New > DWORD (32-bit) Value from the right-clicking menu, and then name the new items as ArchiveIgnoreLastModifiedTime.

5. Double click the new item and change the value data to 1, and then click the OK button.

Note:

1. Back up your regedit key in case this doesn’t help.

          2. Unfinished flagged emails won’t be archived even if their received dates or modified dates are earlier than the specified archiving date.

That bit is almost certainly being caused by the cells where the formula are going being formatted as text. To save you changing the format I've altered the code to do it

Sub Fill_Down()
  Dim FirstRow As Range, x As Long, r As Long
  Dim Cols As Long
  Set FirstRow = Application.InputBox("Select a single row of cells to sum", Type:=8)
  FirstCol = FirstRow.Cells(1).Column
  Cols = FirstRow.Columns.Count
  FirstRow.Cells(1).Offset(, Cols).Formula = "=sum(" & FirstRow.Address(0, 0) & ")"
  For x = 1 To Rows.Count
      r = WorksheetFunction.Count(Cells(x + FirstRow.Row, FirstCol).Resize(, Cols))
      If r = 0 Then Exit For
  Next
Range(FirstRow.Cells(1).Offset(, Cols).Address(0, 0)).Resize(x).NumberFormat = "General"
 Range(FirstRow.Cells(1).Offset(, Cols).Address(0, 0)).Resize(x).FillDown
 
 End Sub

graham mayor say:
If you check Control Panel > Regional Settings > Number Format and see what the short date setting is. For English US it should be MM/dd/yyyy

That being the case run the following macro, which assumes that the dates in question are in column A starting at row 2. This should fix ambiguous dates.

Dim xlSheet As Worksheet
Dim i As Long
Dim strDate As String
Set xlSheet = ActiveWorkbook.Sheets(1)
For i = 2 To xlSheet.Range("A" & xlSheet.Rows.Count).End(-4162).Row
    strDate = xlSheet.Range("A" & i)
    xlSheet.Range("A" & i) = Format(strDate, "MM/dd/yyyy")
Next i

suzanne s. barnhill say:

It might. There are field constructions in some non-English languages that require a semicolon instead of a comma, and some things vary depending on what characters you use as a decimal point and thousands separator. I can't see how either of these would affect the colon in an XE field, but perhaps Stefan Blom or someone else who uses a non-English version of Word may be able to suggest what the issue might be.

Oddly, there is a use for semicolons in index entry fields: according to http://office.microsoft.com/en-us/word-help/create-an-index-HP005189283.aspx:

"If you want to use a symbol, such as @, in the entry, type ;# (semicolon followed by the number sign) immediately following the symbol."

Actually, further reading suggests that this may be relevant: entries marked with ;# are sorted to the top of the index; this includes numbers as well as symbols. Word may be trying to be helpful here.

FWIW, there's what appears to be a rather useful page on indexes generally at http://taxonomist.tripod.com/indexing/wordproblems.html; I doubt that it has the solution to your problem, but it looks as if it might be generally helpful. One of the author's articles at http://taxonomist.tripod.com/indexing/wordflags.html demonstrates a use of the semicolon (sort override) that might be relevant.
