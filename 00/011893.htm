Regarding 65536 or 2^16, yes that was the old row limit.  I'm speculating the old limit had something to do with integers per two's compliment binary math: 10,000,000,000,000,000 (base 2) can be thought of either as 65,536 or as -32,767.  Whether it's 1 to 65,536 or -32,768 to 32,767, 16 bits will only get you 2^16 distinct numbers for rows.  The row and column limits are and were always powers of 2.  

Hi Peter,

If you will only get the MAX of the returned values, you can combine multiple formulas:

Sub Array_Copy_Test()
    Dim source_array As Range
    Dim onecol As Variant
    
    Set source_array = Sheets("rawdata").Range("$a$1:$g$65537")
    
    onecol = Evaluate("Max(Index(" & source_array.Address & ", 0, 4))")
    MsgBox onecol
    'index function will not work if number of rows is greater than the integer limit of 65536
End Sub

If you need all values then this will work:

Sub Array_Copy_Test2()
    Dim source_array As Range
    Dim onecol As Variant
    
    Set source_array = Sheets("rawdata").Range("$a$1:$g$65537")
    
    onecol = Evaluate("Index(" & source_array.Address & ", 0, 4)")
    Debug.Print onecol(1, 1), onecol(2, 1)
    
    'index function will not work if number of rows is greater than the integer limit of 65536
End Sub

The Integer definition has nothing to see with the number of rows. Earlier versions of Excel had 8192 & 16256 rows while an Integer has always been between -32727 & 32728.

ken sheridan say:

An MS Works file can't be directly imported into MS Access, so you will firstly need to export the data from Works, assuming you still have it installed, to a text file.  You'll then be able to import this into Access or into Excel.  If you import it into a table in Access, you'll then have the opportunity of decomposing it into a set of related tables.  You would still be able to work with the single table if you wished, but a relational database management system (RDMS) like Access is designed to use a set of normalized related tables in which redundancy is eliminated.  For example you would have a table of the hams with whom you communicate, with one row for each and columns for their names, callsign and any other attributes specific to each.  This table would have a primary key, usually an autonumber, and other tables which reference it would include only the primary key value in a foreign key column, so there is no need to repeat the names, callsign etc of the ham in question in a table of your contacts with them.

Once you have the data into a single table in Access this 'decomposition' of the data is something you can do subsequently, but before doing so you need to be sure you have a good understanding of how relational databases work.  There are many resources  available to help you with this.  You'll find a couple of simple introductory files in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to amend the form design accordingly. 

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

One is DatabaseBasics.zip, which as its name suggests is a simple introduction to the basic principles of relational database design and how to interface with the data using forms, reports and queries.

The other is Normalization.zip.  Normalization is the formal process by which redundancy is eliminated from a database by decomposition into a set of related tables.  This little file is a simple introduction to this.  You will need MS Access as part of your Office installation to open them of course.

If you import the data into Excel it will remain in the format used by Works.  You will be able to access the data and add new data, but you won't have the extensive database functionality of Access, which would be a far better choice for data of this nature.

dale howard [mvp] say:

Neeket --

 

John and Rod have given you sage advice.  I think there is a way to track your payment information for the Material resources you are using, but I am not sure it would be worth the bother.  Here is how you might handle it:

  • For each Material resource, enter a Standard Rate of 0.
  • Assign your Material resources to the tasks where you will use them and enter a Units value to show the consumption of the Material resources.
  • Add a Milestone task for every payment needed for your Material resources.
  • On each Milestone task, set a Must Start On constraint to show when you make the payment for the Material resource.
  • Apply the task Cost table and enter a Fix Cost amount for each of those payment Milestone tasks representing the amount you are paying for the Material resource.

By using the preceding process, you will see your costs for Material resources based on the Fixed Cost amount on the milestones, rather than on the tasks to which the Material resources are assigned.  I realize this is a workaround, but as John and Rod have clearly told you, there is no direct way to do what you want.  Hope this helps.
