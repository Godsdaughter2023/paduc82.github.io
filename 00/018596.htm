emerald77 say:

For debugging purposes, I inserted a MsgBox line (which included a function that tests for the existence of a given query) after my CreateQueryDef line to verify that the query was, in fact, created.  I also inserted similar MsgBox lines before my DoCmd.OutputTo and DoCmd.TransferSpreadsheet lines to again verify that the newly created query still existed.  In all cases, the output of the MsgBox lines confirmed that the newly created query existed.

 

What is interesting, however, is that with the insertion of the MsgBox lines, I received no error messages and everything ran as expected.  I then removed all the MsgBox lines, and the problem re-appeared.

 

I know this may sound silly, but could there be a timing issue?  In other words, given the speed at which the ACCDE code is being executed, and the fact that the DoCmd.OutputTo and DoCmd.TransferSpreadsheet lines are executed almost immediately after the CreateQueryDef line, maybe there is not enough time for the newly created query to "register" with Access.  But when the MsgBox lines are inserted, they cause enough of a lag to allow the query to be recognized by DoCmd.OutputTo and DoCmd.TransferSpreadsheet.

 

So I ended up "wrapping" the DoCmd.OutputTo and DoCmd.TransferSpreadsheet lines within an IF-THEN-END IF block, and used the aforementioned function to test for the existence of the newly created query.  By doing this, the DoCmd.OutputTo and DoCmd.TransferSpreadsheet methods "saw" the newly created query and executed with no problems.

 

I just thought I would share my discovery.
