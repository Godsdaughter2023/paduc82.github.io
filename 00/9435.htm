Hi Mark,

Please see below:

Publish a project to a SharePoint list

This procedure assumes you have created a list of tasks in Project prior to syncing the task list to a SharePoint list.

In Project, click the File tab to open the Backstage view. Click Save & Send

Click Sync with Tasks Lists.

In the Site URL list, select the URL name of the SharePoint site that contains the list that you want to sync to.

NOTE:  At this point, don’t use a URL that contains the name of the task list you want to use.

Click Validate URL.

In the Select an existing tasks list list, select the name of the SharePoint task list to which you want the project tasks list to sync.

If you want to sync the Project task list with a new list in SharePoint, type a new name. A new task list will be created in SharePoint, and will become synced up with the one in Project.

Click Sync.

After the two task lists are synced, changes to one list will be reflected in the other. If changes are made to the two lists at the same time, a dialog box appears that prompts you to decide which version of the change you want to keep.

TIP:  After the two task lists are synced, within Project you can go the SharePoint version of the task list. Click the File tab, click Info. In the Sync to Tasks Lists section, click the link to the SharePoint site.

    joeu2004 say: TimourKoupeev wrote:
    If the simulation (random number generator) is initialized with some seed value (e.g. = 1) and the distribution functions are always recalculated in a certain order, the result of the simulation will always be the same. However, if the recalculation order changes, the simulation result will be different even if the same seed value is used. I hope you understand the problem.

    I don't believe the order depends on whether or not you seed the RNG.  Perhaps it worked only by coincidence.

    AFAIK, the only way to control the order of calculation of cells is to set manual calculation mode [1], then selectively calculate cells and ranges in a macro, e.g. Range("A1").Calculate and Range("A1:A100").Calculate.

    Caveat:  If you use Range("A1:A100").Calculate, you cannot be sure of the order of calculation within the range.

    Common sense would suggest that if A1 depends on B1 and B1 depends on C1, Excel "must" calculate them in the order C1, B1 and A1.  But at least in XL2003, although that might be the final order of calculation, in fact sometimes cells are calculated out order with incomplete values of dependent cells.

    Normally, that does not matter.  But is causes errors if UDFs assume they are called in a particular order, and there is a side-effect if they are called multiple times in one recalculation cycle -- for example, assuming the repeatable random sequences will be generated in a particular order.

    -----
    [1] [EDIT] IIRC, if you subsequently set automatic calculation, it will trigger a recalculation cycle that will include all of the cells that had been calculated manually.  At least, that is the case with XL2003, IIRC.  You might double-check my recollection.  Bottom line:  for your purposes, you can never set automatic calculation :-(.

    [EDIT2] Confirmed with the following test, executing doit() after putting =myUDF() into A1.

    Function myUDF()
    MsgBox Application.Caller.Address
    End Function

    Sub doit()
    Application.Calculation = xlCalculationManual
    Stop
    Range("a1").Calculate
    Stop
    Application.Calculation = xlCalculationAutomatic
    End Sub

    myUDF is called both for Range("a1").Calculate as expected and for Application.Calculation=xlCalculationAutomatic, even when myUDF is not volatile.

    Using unbound control to dispay different values - Microsoft Community

    glint say:

    Using unbound control to dispay different values

    I have an unbound text box in continuous form. The challenge I have is to make it visible only on a selected record, instead of being visible to all records. If this is not possible, how can I make it display different values while it is unbound? I am trying to capture the value a user enters into the box but I don't want to confuse him when he sees the same value in all records in front of him.
    djsteele, mvp say:
    Unfortunately, you can't.

    While it may look to you as though you've got, say, 10 instances of the text box on your form, in actual fact you've got a single text box, repeated 10 times. Whatever you do to one text box is done equally to all ten of them.

    You might be able to use Conditional Formatting to change the text colour to the same as the background colour so that at least nothing shows other than the one you want.

    ken sheridan say: A control in a continuous form is a single control, so in each row you are merely seeing different instances of the one control.  If you set the Visible property of the control to True or False this will affect every row in the form.  Similarly the value of an unbound control will show as the same in each row.

    You can put an unbound control in the form's header or footer to show it only once, or you can include a column in the form's underlying table and bind a control to that, in which case the value can be changed independently in each row.  You will still not be able to hide the control independently in each row, however.
