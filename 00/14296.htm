jay freedman say:

You must define a character style (not a paragraph style) and put the shading color in that style.

Here's the simplest way to do it:

  • Select some text, not including any paragraph mark.
  • On the Home tab of the ribbon, in the Paragraph group, click the down arrow on the Shading button and select the color you want for the background.
  • While that text is still selected, click the lower down arrow at the end of the Styles gallery, and click Save Selection As a New Quick Style.
  • In the Create New Style dialog box, enter a name for the style and click the Modify button (not the OK button!).
  • In the next dialog (which confusingly has the same title as the previous one), click the Style Type dropdown and choose Character. If you want to add the style to your template, click the "New documents based on this template" option. Then click OK.

Unfortunately, once you have defined a style as a paragraph style, there is no way to change it into a character style; you must delete the paragraph style and start over.

ken sheridan say: I suspect that the columns (fields) are all of text data type, rather than a number data type.  Consequently the values, as string expressions, are being concatenated.

However, I'm concerned that the structure of the underlying table might be incorrect.  Each column appears to represent a subject of study or similar.  If so data is being 'encoded as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

If these are students' scores in subjects for instance, what you have is a many-to-many relationship type between students and subjects.  A many-to-many relationship type is modelled by resolving the relationship type into two or more one-to-many relationship types by means of an additional table.  So you'd have, in broad outline, tables along the following lines:

Students
....StudentID  (PK)
....FirstName
....LastName
....etc

Subjects
....SubjectID  (PK)
....Subject

And to model the relationship type between them:

StudentSubjects
....StudentID  (FK)
....SubjectID  (FK)
....Score

Score is an attribute of the relationship type.

To sum a student's scores for all subjects taken would now be a case of summation over each subset of rows, rather than addition over a set of columns, which is how a relational database is designed to work.

So Ken's StudentSubjects table would be a proper design. You can then total the grades for each student with a group By query:

SELECT StudentID, Sum(Score) As Grade

FROM StudentScores

Group BY StudentID;

Would give you a total score for each student.

lordalukard say:
Dear Mike,

I am an independent advisor, and I am trying to help other users in Community with my experience in Microsoft products.

When you are going abroad and try to access your account, system checks the location and device. If those two are unfamiliar system may block access to the account due to security reasons (as it thinks that it may be compromised https://support.microsoft.com/en-gb/help/10494/... )

In order to avoid this situation you can do the following:
1. Add your devices to the list of trusted devices https://support.microsoft.com/en-gb/help/12369/...
2. Enable 2step verification https://support.microsoft.com/en-gb/help/12408/... or (which is much better in my opinion) use Microsoft Authenticator app https://support.microsoft.com/en-gb/help/402672...
3. Update security information and add your phone number https://support.microsoft.com/en-gb/help/12428/...

This should allow you to get access to your account in any location (tested myself, as I had the same problem, and as for now it works)
