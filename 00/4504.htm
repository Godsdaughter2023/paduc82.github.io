lifescholar1 say: There is no workaround. You cannot have 2 files open in memory with the same name, even if they are saved in different locations.

I used to think this was odd until I thought through the situation where you might have a third file, call it "Workbook2.xls" that contains formulas that link to both "C:\Workbook1.xls" and "D:\Workbook1.xls". If both of these are closed, then the link formulas in Workbook2.xls will contain the full path name to the relevant Workbook1.xls. However, if you open one of the source workbooks, then the formulas in Workbook2.xls change to [Workbook1.xls] without the full path, i.e. when a source file is open in memory, the path information is not part of the link formula. If you were able to open both Workbook1.xls files, then there would be no way of distinguishing which workbook was being referred to in the link formula.

Thank you.

Lisa

jay freedman say:

The "official" way is to start each new drawing by inserting a drawing canvas (the bottom item on the Insert > Shapes menu) and creating the diagram in the canvas. Then you can select all the pieces by dragging a "marquee selection" rectangle that includes all of them. 

If you already have a drawing you created without a drawing canvas, the workaround is to save the document in Word 97-2003 (*.doc) format. In that format, marquee selection works without needing a canvas. You can select and group the diagram, save the document, and then go to File > Info and click the Convert button, then save again (which will make it a .docx file again).

Word 2010 is the only version that requires this workaround. In Word 2007 and 2013, click Home > Select > Select Objects to enable the marquee selection without a canvas.

thanks for the link... it makes all the difference....

your hosting provider is using E-tags to control the caching of your web pages on browser clients.

 

ref: http://en.wikipedia.org/wiki/HTTP_ETag

 

Since ETags are typically constructed using attributes that make them unique to a specific server hosting a site, the tags will not match when a browser gets the original component from one server and later tries to validate that component on a different server.

The problem with ETags is that they typically are constructed using attributes that make them unique to a specific server hosting a site.

ETags won't match when a browser gets the original component from one server and later tries to validate that component on a different server, a situation that is all too common on Web sites that use a cluster of servers to handle requests. By default, both Apache and IIS embed data in the ETag that dramatically reduces the odds of the validity test succeeding on web sites with multiple servers.

so .... it looks like your hosting provide has changed the IP address of your site since you first started developing it.

I think you can correct this on your machine by doing a dnsflush.

Start>Run>cmd

type ipconfig -dnsflush in the Command window.

 

further....

see How to prevent caching in Internet Explorer.... http://support.microsoft.com/kb/234067
