    Validate data entered in a table field based on a list of values - Microsoft Community

    orionis57 say:

    Validate data entered in a table field based on a list of values specified in another table

    Hi!

     

    I want to know how do I validate data entered in a table field based on a list of values specified in another table (Access 2007).

     

    For e.g., I have a table containing the names of all Sales people in TBL 1. In TBL 2, I have already used the LOOKUP Wizard to create a drop - down / list of values box. However, at times the data in TBL 2 is imported using an Excel spreadsheet and this results in the creation of sales persons that do not exist in TBL 1 or incorrectly spelled names etc.

     

    I know that I can create an expression in the "Validation Rule" section such as "James" OR "Byron" Or "Marrion" etc. However, this is possible only if there is a small number of names. Also, I don't want to "hard code" this expression but want it to remain dynamic, i.e., if a new name is entered in TBL 1, then that too should be considered as part of the Validation Rule.

     

    Any suggestions on this? I am using Access 2007.

     

    Thanks,

     

    Orionis.

     

     

    ken sheridan say: Firstly you should relate the two tables on the names and enforce referential integrity between the two tables.  This will 'validate' the rows in TBL 2 by preventing any which don't match being inserted.  Using personal names as keys is unusual, however, and normally something we would strongly advise against as names can legitimately be duplicated.  It's really necessary to include one or more other columns in the key to ensure distinct values, and even then this is not reliable.  I was once present at a clinic where two patients turned up, both with exactly the same first and last names, both female and both with the same date of birth!

    When you import the Excel data do so first into an empty 'master' table.  You can then append from this into your operational TBL 2 using an 'append' query.  Rows which do not have a match in TBL 1 will not be appended.  You can then have  a query which tells you which rows in the 'master' don't have matches, so won't have been appended, e.g.

    SELECT *
    FROM [Master]
    WHERE NOT EXISTS
        (SELECT *
         FROM [TBL 1]
         WHERE [TBL1].[FirstName] = [Master].[FirstName]
         AND [TBL1].[LastName] = [Master].[LastName]
    ORDER BY [LastName], [FirstName];

    You can base a form on this query and for each record include the means to:

    1.  Add the person to TBL 1 or:
    2.  Amend the names so they match with names in TBL 1 or:
    3.  Reject the new row completely and delete it from the master table.

    Once you've gone through this cleaning-up operation you can execute the append query again to append the now corrected names to TBL 2.  All the rows in the 'master' table can then be deleted.

    However, as the others have pointed out, the need for this very unsatisfactory procedure ultimately stems from the uncontrolled way it is being collected in Excel.  The real solution is to tackle the problem at the root.
