tony chen chn say:

Hi,


CORAND(CorrelArray, RandSource)
For any integer n greater than 1, the CorrelArray parameter may be a square n-by-n array of correlations for n random variables (as returned by MCORRELS), and then CORAND returns an array of n correspondingly correlated values. Such an n-by-n correlation array should be symmetric and must have ones on the diagonal. The CorrelArray parameter can also be a single number, in which case CORAND functions as if the CorrelArray parameter were a 2-by-2 array, returning two random values with the given correlation.
The optional RandSource parameter can be a reference either to a single cell or to n cells in a row. RandSource cells should contain independent uniform random values, as generated by RAND or CORAND.

john w. vinson say:

Do you mean linked table as in a split database, where the table itself is in a separate backend database? Or do you mean a linked table as in a (single) database, with Relationships defined between tables using this field?

If it's a split databse you must open the backend database separately, make the change to the datatype, open the frontend, DELETE the table (link), and use External Data... Link to relink to the table. Access caches some information about the backend table in the frontend, and I've found by hard experience that just using the Linked Table Manager isn't always flawless!

If you have the Serial Number as a primary key-foreign key link to other tables, it's more  difficult. You'll need to change the datatype in all of the tables containing the field. This will require opening the relationships window; deleting the relationship(s); opening each table in turn and changing the datatype; and finally reestablishing the relationships.

Of course the table could be linked in both senses of the term, in which case you'll need to delete all of the affected tables (the links to them, that is) from the frontend; open the backend and change all the tables as above; then open the frontend and use External Data to reconnect to the modified tables.

If you have any code which (say) increments the serial number it will obviously need to be modified - you can't do math on text fields.

If the subsidiary tables are related by some other field (such as an Autonumber Primary key to a Long Integer foreign key), then the child tables should not HAVE this serial number field so it shouldn't make any difference - you'ld just need to change the one table.
