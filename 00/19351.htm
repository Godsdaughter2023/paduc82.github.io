Hi Mike,

First, please make sure you have followed these steps to map a SharePoint Online library to your computer:

1. Sign in to the SharePoint Online site by using your Office 365 work or school account credentials. Make sure that you click to select the Keep me signed in check box.
2. Browse to the library that you want to map the drive for.
3. Click the LIBRARY tab on the ribbon, and then click "Open with Explorer".
4. Click in an empty area of the Explorer window address bar, and then copy the URL of the library.
5. Right-click Computer or This PC from the Desktop, or within Windows Explorer, and then click Map Network Drive.
6. Click the Connect to a Web site that you can store your documents and pictures link, and then click Next two times.
7. Paste the URL that you copied in step 4, click Next, and then follow the instructions in the wizard.

Note: "Open with Explorer" feature only works in Internet Explorer.

Reference: How to configure and to troubleshoot mapped network drives that connect to SharePoint Online sites in Office 365.

About "if there is any way that we could automate/simplify the reconnection process", I'd like to explain there is no automate way to achieve this.

If you get the above error, you may try the following workarounds:

A. Choose "Open with Explorer" from the document library

1. Log into the site with Internet Explorer.
2. Click "Return to classic SharePoint" at the lower left. 
3. Click on the "Library" tab > "Open with Explorer".
4. The library should now open up in File Explore, now you should be able to click on the mapped drive.

B. Ensure "Keep me signed in" is checked before logging in

1. Using Internet Explorer, log out of sharepoint (if you are logged in).
2. Go to your sharepoint site.
3. When it asks you to log in, do not click on your saved email address, but instead click "use other account".
4. Before you enter your email address, check the box "Keep me signed in".                                                  
5. Now enter your email and password. 
6. Once logged in, test the mapped drive connection. You should now be able to get into the site.

C. Adding sharepoint.com as a trusted site  

1. Open Internet Explorer.
2. Click the "Tools" menu, and then click "Internet options".
3. Click the "Security" tab, click Trusted sites, and then click Sites. 
4. In the "Add this website to the zone" box, type the URL for the SharePoint Online site, https://*.sharepoint.com, and then click "Add".
5. After you have added each site to the Websites list, click Close, and then click OK.

Meanwhile, we have already released the new OneDrive sync client which brings us a better user experience. It also allow us to sync SharePoint Online libraries. If you want to access your SharePoint Online documents from your local drive, you can also give it a shot.

Please follow the steps in "Test the feature before you enable it for your organization" part in this article and set it up and go to the library to sync files.

Simple Question on the Future of the Windows Phone - Microsoft Community

genecyprych say:

Simple Question on the Future of the Windows Phone

I have had a Windows phone for 15 months.  I understand that with 3% of the cell phone market developers are  not pulling all nighters to develop apps.  I have read that there is a plan to modify the phone's firmware and/or OS to allow Droid apps as well as iOS apps run on it/  Is this true?  When is it expected?  Do the techies really think it will properly host and support all of those apps?
christopher miller say:

When Windows 10 for the Phone is released, developers will be able to submit java based Android apps to the Windows store with little to no changes to the code.  Most functionality will just work out of the box and developers can add support for Windows Live Tiles.  Android developers will be able to do this using their existing development tools.  Microsoft will be releasing special libraries that work with the common Android development tools to allow developers to treat Windows 10 for Phones as just another target.

It's bit different for iOS.  Developers can take their Objective-C code for iPhone apps and recompile it using Microsoft's Visual Studio 2015 IDE.  Some minor changes may need to be made.  The Candy Crush Saga game currently in the Windows Phone store is the iPhone code recompiled in Visual Studio with some minor changes (I think they had to take out references to Apple's Game Center).  That has been available for Windows Phone for a while now and runs just fine with no performance issues.

At a technical level, yes this does work.  You can't take any app for Android or iOS and load it on your Windows Phone, the developer does have to submit it to the store.  

The real question will be if this have any affect.  Writing the code for an app is just one part of what it takes to get an to market.  You have test the app on each platform and you have to provide support for each platform.  Too many developers are just saying 3% isn't enough to spend the time supporting the platform.

What we do get with Windows 10 are Universal Windows Platform (UWP) apps.  UWP apps can run on desktop, tablets, phones, and Xbox One.  One binary, multiple platforms.  That mean you can write app for a full screen device, and it will run on the phone with a user experience optimized for both.  If that takes off, then we should get more apps for the phone.

greginmich say:

We don’t see any scheduled scan settings in Windows 8 Defender, but if we look under the hood, we do see a Windows Defender Scheduled Scan task defined in the new and improved (but not so transparent) Task Scheduler. Oddly, there appears to be no trigger for this scheduled scan task; so those wily Web gurus naturally surmised that the Windows Defender Scheduled Scan task must exist for the sole purpose of being modified by the user in order to create a scheduled scan. Notice that even the local experts felt compelled to direct users to the Windows Defender Scheduled Scan task as an afterthought in their new Wiki FAQ for Windows 8:

http://experts.windows.com/w/experts_wiki/163.aspx

 

I haven’t had enough time to look at this closely; but from what I’ve seen, it looks like the HANDS OFF rule that applies to the MSE scheduled scan task (the Microsoft Antimalware Scheduled Scan) should be extended in full force to the Windows Defender Scheduled Scan task. This task rightly belongs to the Windows 8 Operating System just like the MSE scheduled scan task belongs to MSE. There is already an Automatic Maintenance trigger for this task, and the task is configurable, to the extent that it was meant to be configurable, in the Action Center:

http://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_start/windows-defender-program-file-and-argument/5cc6d3dc-a3f5-4067-abd2-a5821ec649da

 

A Full Scan with Window 8 Defender adds very little value, and if you’re genuinely concerned about undetected malware, (which you probably should be, given the downtrend in Microsoft’s detection ratings), I would suggest scanning with third-party tools on a regular basis in lieu of scheduling a Full Scan with Windows 8 Defender. Nevertheless, a Full Scan can easily be scheduled by running the Create Basic Task Wizard in Task Scheduler, and entering one of the command lines described here:

http://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_scanning/in-the-new-windows-defender-for-windows-8-how-do-i/1548fe83-236e-4e72-9bd1-7c7033247d0d

greginmich say:

What I’m seeing is that Automatic Maintenance only allows the scheduled scan to run when the system is idle; and instead of pausing in response to user activity, as we might expect it to do, the scan terminates and then restarts when the system returns to idle. Those conditions create an environment that is pretty hostile to Full Scans.

 

The -RestrictPrivileges parameter is used to limit the functionality of the MSE scheduled scan. I’m not sure why you’re seeing it, but even with this parameter omitted, we can’t be sure that the Windows Defender Scheduled Scan could ever have its conditions modified to allow for the successful completion of a Full Scan. This task is really owned by the System, and it’s designed to run Quick Scans. So at least until we get a better understanding of Automatic Maintenance, the best course of action here would be to create your own independent task for a Full Scan.

greginmich say:

Dennis, we already know that when you add a new trigger to the Windows Defender Scheduled Scan task, the scan will run uninterrupted at the scheduled time, and everything will appear to be fine. But when you change the type of scan to a Full Scan, then Automatic Maintenance will also attempt to run a Full Scan on it’s own schedule - and that’s were the problem comes in; because the Automatic Maintenance scan will be interrupted by any kind of user activity and then have to start over from scratch - so a Full Scan will have a very difficult time completing.

 

If the new trigger somehow cancelled the Automatic Maintenance scan, then everything would be hunky-dory, but unfortunately that’s not how it seems to work. So if you choose to schedule your Full Scan this way, it will work, but you can probably expect some trouble from Automatic Maintenance down the road.
