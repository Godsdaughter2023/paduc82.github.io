Table recordset not updatable if sorted by field - Microsoft Community

robertolson-vb6 say:

Table recordset not updatable if sorted by field

This is so weird.

If I sort a table by one of the fields (ie: A-z) then all records and all fields are locked.  If you try to change any value in any field you get the note "Recordset is not updatable."  This is a table, not a query. 

If I remove the sort, the table is once again editable.  WTH?

UPDATE:  Just found out that this only occurs if you sort the table by a field which is a lookup field.

john w. vinson say:

One of many, many reasons to avoid table Lookup Fields: see http://www.mvps.org/access/lookupfields.htm for a critique.

The value that you see is not actually in the table; it's in the linked lookup table. Since it's not in the table, it's not in the table's indexes and therefore cannot be sorted.

Use a Query joining your table to the lookup table; include the text value from the lookup table in the query, and sort by it.

rick beesley say:

Hi Robert, the problem is the lack of a unique index in the referenced table on the field your table is looking up.

Example:

  • The tables
    • Teams
      • Teams.TeamID is an auto number field that I neglected to give a unique index
      • Teams.Name is the team name
    • Players
      • Players.TeamID is a reference to Teams.TeamID, and its lookup properties are set accordingly
  • Scenarios
    • Attempt direct edit in the Players table
      • Works fine until I sort by Team, after which case it's not updateable
    • Attempt to work around it with a query
      • I tried joining the two tables, selecting Players.* and sorting by Team.Name ascending
      • But no change in behavior. Recordset is not updateable when sorted by team name
    • Attempt to fix it by overriding the lookup in my query
      • In the query's properties of the TeamID column, I set the Display property to Text Box so I see numbers in that field, not team names, but still if I am sorting by Team.Name, the recordset is not updateable
  • The fix
    • Make Team.TeamID the primary key (or otherwise give it a unique index)

Run a macro containing the following code:

Dim i As Long
For i = 1 To 9
Selection.Find.ClearFormatting
With Selection.Find
    Do While .Execute(FindText:=i, MatchWholeWord:=True, MatchWildcards:=False, Forward:=True, Wrap:=wdFindStop) = True
        With Selection
            ActiveDocument.Fields.Add .Range, wdFieldEmpty, "=" & i & " \* cardtext"
        End With
    Loop
End With
Next i

It doesn't show the full "AND" logic, but in recent versions of Windows Word, you could also do this particular thing like this:

{ MERGEFIELD Fund_description_1 }{ MERGEFIELD In_Honor_of_1 \b " " }{ MERGEFIELD In_Memory_of_1 \b " { IF "{ MERGEFIELD In_Honor_of_1 }" <> "" "and " }" }.

macropod say:

You could use:

{MERGEFIELD Fund_description_1}{IF{MERGEFIELD In_Honor_of_1}= "" {MERGEFIELD In_Memory_of_1 \b " "} " {MERGEFIELD In_Honor_of_1}{IF{MERGEFIELD In_Memory_of_1}<> "" " and {MERGEFIELD In_Memory_of_1}"}}
or:

«Fund_description_1»{IF«In_Honor_of_1»= "" «In_Memory_of_1» " «In_Honor_of_1»{IF«In_Memory_of_1»<> "" " and «In_Memory_of_1»"}}

Note: The field brace pairs (ie '{ }') for the above example are created in the document itself, via Ctrl-F9 (Cmd-F9 on a Mac); you can't simply type them or copy & paste them from this message. Nor is it practicable to add them via any of the standard Word dialogues. Likewise, you can't type or copy & paste the chevrons (ie '« »') - they're part of the actual mergefields, which you can insert from the mailmerge toolbar.

Note also that, with the second example, you would still need to add the \b " " argument to the first In_Memory_of_1 field. Alternatively, if you're happy to always have a space after the Fund_description_1 field, you can omit the \b " " argument and the space before the second  In_Honor_of_1 field for both examples.
