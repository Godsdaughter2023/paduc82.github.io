gina whipp say:

Just so we are clear, you just want it to display there, correct? (Say yes...)  [Side note, those are Controls on your Form not Cells.  Cells are in Excel.]

If the information is in a row source of the Combo Box you can just use...

In Control source of your Text Box... =Me.YourComboBoxName.Column(2) and the other Text Box use Me.YourComboBoxName.Column(3).  You may have to change the numbers depending on the column count of your Combo Box.

OR

Include the table that has the information in the row source of your Recordsource (the one related to the Combo Box value) and display it in controls on your form.

scottgem say:

Access doesn't have cells That's a spreadsheet concept and spreadsheet concepts don't translate well to a relational database. You also don't have cells on a form, you have Controls that may or may not be bound to fields in a table. 

If you want to DISPLAY related values on your form when you select a value, please see my blog on Displaying Related Values. This describes various techniques to do that.

A combobox is certainly an option. There are ways to make selection easy or reduce the number of returned values.

ken sheridan say: I would not rule out a combo box.  While scrolling the drop down list would be tedious, provided that the control's AutoExpand property is True (Yes), the user can type in the value and the control will progressively go to the first match as each character is entered.  By setting the control's NotInList event procedure to True it will not be possible for the user to enter an invalid value.

You can then either reference columns in the combo box's list as the ControlSource  of unbound controls, as Gina describes, or you can base the form on a query which joins the current table to the referenced Products table on ProductID or similar, and return the Size and Weight columns from the Products table in the query.  You can then bind controls in the form to these columns, setting the Locked property of each to True, and their Enabled property to False to make them read-only, and thus avoid any possibility of the used inadvertently trying to edit the referenced values.

In the database the referencing table into which you are inserting rows via the form must be related to the Products table on ProductID (or whatever the name of the keys is) and referential integrity enforced.  This is important, not only to ensure the correct functioning of your form, but even more importantly to protect the integrity of the database.
