SIMPLE/ “SOFT” UNINSTALL SAVES CUSTOMIZATION SETTINGS, IN THEORY!
Since day one, what became Office applications have had quality problems. So MS designed them to be “user friendly”. You can do a simple uninstall, from the Control Panel and the process leaves behind your configuration files, like Normal.DOTM for Word, and Registry Entries. The developers came up with a 4 step “fast and dirty” trouble shooting process for themselves and for users:
.  Step 1, start the application in Safe Mode to disable addins.
.   Step 2, do an Office Repair
.   Step 3, un-install through the control panel then re-install The re-install picks up the exiting configuration information to save the user the effort of having to re-setup their installation
.   Step 4, do a full uninstall-reinstall using the appropriate “fix-it” script.  This wipes all configuration information, on the chance the problem in is in the configuration files/registry. 
This “soft” uninstall also plays a part in “upgrades”. During an upgrade do a simple unstall either manually or through the upgrade installer and the upgraded installation will pick up the existing configuration information making for a “painless” User Experience. 1.  Each city can have multiple zip codes.  That doesn't surprise me as the same is true of UK post codes.

2.  Each zip code can relate to more than one city.  This differs from the UK post code system, where each code identifies a very specific location.  The zip codes in the table are in 5 digit format.  In the data the term 'city' clearly can refer to quite small settlements.

If your data is similar then it is not possible to identify a city from the zip code.  The best you could achieve would be to present the user with a selection of cities in a combo box whose list is restricted to a selected zip code, i.e. you'd correlate the city combo box in the usual way.

If your data has the zip codes in their extended format, however, and these, like UK postal codes, identify a very specific location, then the zip code almost certainly determines the city.  In this case you would not need to store the city or state in columns in a table of locations, only the zip code.  To do otherwise would introduce redundancy and mean the table was not normalized to Third Normal Form (3NF) with the consequent risk of update anomalies.  This is how we do it here, where a postal code plus a house number or name is all that's needed in a table to identify an address.

Another important point to note is that a city name cannot be used as a key as city names are duplicated.  Nor can you have a composite key of the city name and state as some US states have more than one city of the same name.  A city should be identified by a surrogate numeric key, usually an autonumber, in a Cities table like this:

Cities
….CityID  (PK)
….City
….State

which would reference a States table with a single column State in an enforced relationship.  Some people would give the States table a surrogate numeric key, and the Cities table a corresponding long integer number foreign key, but it serves little purpose to do so, and only requires the inclusion of the States table in queries in addition to the Cities table.  Consequently a zip codes table would have a CityID foreign key column, but no State column, as state is determined by city, so to include a State column would introduce redundancy and again would mean the table was not normalized to Third Normal Form (3NF) with the consequent risk of update anomalies.

ken sheridan say: I perfectly understand your question, and of course it is possible to assign a value to one column in a table on the basis of a value in another column, but the point is that it is wrong to do so.  The database relational model is a formal model governed by formal principles.  These have developed over nearly fifty years, and are fundamental to the integrity of the data.  I'm sure you would be very unhappy if your bank ignored these principles and, as a result, large amounts of money were to disappear from your accounts.

One of the most important mechanisms of the relational model is that of 'normalization'.  This is the formal process by which redundancies are eliminated by the decomposition of a table into two or more related tables.  Fundamental to this is the concept of 'functional dependency', which means that where, for any one value of a column, the value of another column is always known the second column is functionally determined by the first.  Normalization (to Third Normal Form in this instance) requires that every non-key column in a table must be functionally dependant on the whole of the primary key of the table and on nothing else.  If you have a table in which there are columns LocationID (the primary key), CityID and State, then CityID is functionally dependant solely on the key, but State is not as it is functionally dependant on the non-key column CityID.  The table is consequently wide open to the risk of update anomalies as there is nothing to prevent a value being inserted into the CityID column, and, in the same row, a value being inserted into the State column where the latter is not the state in which the city identified by the CityID value is located.

When it comes to zip codes and cities, the relationship type is many-to-many, i.e. each zip code can be in one or more cities and each city can have one or more zip codes.  A binary many-to-many relationship type like this is modelled by a table which resolves it into two one-to-many relationship types, so this part of the model would, in broad outline, comprise the following tables:

ZipCodes
…..ZipCode  (PK)

Cities
….CityID  (PK)
….City
….State  (FK)

States
….State  (PK)

The many-to-many relationship type is thus modelled by a table like this:

CityZipCodes
….CityID  (FK)
…..ZipCode  (FK)

This table is 'all key', i.e. its primary key is a composite one of the two foreign key columns.  Even if you, unnecessarily, give this table a single column surrogate key, the two foreign key columns are still a candidate key and would be defined as such by their inclusion in a single unique index.

When it comes to modelling locations or similar, a table for doing this would have a composite foreign key of CityID and ZipCode, which references the composite primary key of CityZipCodes in an enforced relationship.  Data entry into this table can either be city oriented or zip code oriented.  This is a matter of choice for the developer in consultation with the end user.  The data entry form (and data must only be entered via forms, never directly into a table's datasheet) could have a city combo box with which a zip code combo box is correlated, or vice versa, i.e. the user can first select a city and then a zip code from a list restricted to zip codes in the select city, or, as you appear to wish to do, they can select a zip code first and then a city from a list restricted to those cities to which the selected zip code relates.  With a little more work, it is possible to restrict the number of steps to one where a zip code relates to one city only, making it necessary for the user to select a city only in those cases where the selected zip code relates to more than one city.

In no case will it be necessary for the user to select the state.  That would show automatically in an unbound control in the form once a city is selected.

For all this to work seamlessly, and with full protection of the integrity of the data, the tables must be correctly normalized as described above.  If this is not done then the database cannot confidently be regarded as fit for purpose.

ken sheridan say: I'm glad that Scott has used the word 'understanding' in his reply.  It echoes what Richard Feynman is saying to his students in the quote below my signature.  Just as in fundamental physics, which Feynman was referring to of course, the advice is equally relevant to relational database design.  Embarking on the development of a relational database application without a sound understanding of the principles of the relational model is asking for trouble, and part of that understanding is a knowledge of the vocabulary and what terms like 'normalization' and 'functional dependency' mean.

While it would be a tall order to acquire a level of knowledge and understanding as would be required by the professionals who design the databases behind our bank accounts and the like, for we mere mortals who use Access as a tool in our day to day lives, it is not beyond any person of regional intelligence to gain a level of knowledge and understanding sufficient for us to produce efficient and reliable database applications suited to the tasks with which we are presented.  This is why I post simple demo files in my OneDrive folder to give people simple introductions to the topics which they will need to understand if they are to design good database applications.

As regards forms and tables, for entering data in a grid layout you are not limited to a table's datasheet or a form in datasheet view; you can design a form in continuous forms view which gives you all the advantages of being able to include things like command buttons, unbound navigational controls etc which you can use in a form in single form view.  However, it would be rare to be able to enter data via one form, be it in single form view or continuous forms view.  Entering data almost always involves inserting rows into more than one related table, for which the interface would usually be a parent form (in single form view)  within which would be one or more subforms (usually in continuous forms view, but not necessarily so).

When it comes to the sort of time-saving tasks which you describe, these sort of ergonomic considerations are an integral part of good form design, so again it's a matter of having the knowledge and understanding to be able to include the necessary functionality in the forms.  Again it's Feynman's word 'understanding'.  If you don't have it at the outset then it's inevitable that you will make errors which will result not only in an inefficient database, but more importantly an unreliable one.  While its inefficiency will be self-evident and the cause of the sort of frustrations you have voiced, you will probably be unconscious of its unreliability, which is a dangerous position for you and the users of the database to be in.
