When changing certificates you need to make sure:

1. Assign the SMTP service to the certificate.

2. Set your send connector to use said certificate for TLS by running

Set-SendConnector <connector name> -TlsCertificateName <Certificate Name>

3. Having done that, restart the transport service.

teylyn say:

You have a circular reference not in a single cell but in a chain of calculations.

B8 refers to B32

B32 refers to B28

B28 refers to B24

B24 refers to B20

B20 refers to B16

B16 refers to B12

B12 refers to B8 and that closes the circle.

This is the cause of the error message and the answer to your question as asked.

You need to decide which of these cells is the first and which is the last in the cycle. The first cell of the new cycle needs to refer to the last cell of the old cycle, not the last cell of the same cycle. This is a time line progression, so you can't have next Sunday refer back to the previous Monday. Life does not work that way.

You may need to go back to the drawing board and re-design your data architecture to factor in these time frames.

scottgem say:

Richard, 

First let me ask why you need two fields to cover the same information. If a user enters the DateOfCompletion, then why do you also need a Yes/No field to say its completed? This is just redundant and a waste. If you want to know what records are not completed, then you filter for DateOfCompletion = Null. 

So my recommendation is to not bother with the Yes/No field. This will eliminate the need to lock the field. 

But as mimmo noted, this is something you do on forms. Unless you will be the only one using this database, then interaction with the tables should ONLY be done through forms. Users should NEVER access tables directly. So your assumption that "conditional stuff like that had to be done while still editing the tables". Is incorrect. Data macros are new to Access 2010, so you CAN do some things using Data macros. but most of the time you would use VBA code on events associated with controls or forms. 

marshall barton say:

Your problem in writing a Validation Rule is that multi field rules must be in the table's rule, not in a field's rule.  A Table level validation rule that should do what you want could be:

The long way to write it could be like:

   ([DateOfCompletion] Is Null And [CalibrationComplete] = False) OR ([DateOfCompletion] Is Not Null And [CalibrationComplete] = True)

but the short way is:

   [DateOfCompletion] Is Not Null Eqv [CalibrationComplete]

OTOH, I agree with Scott, the date field makes the check box redundant and removing the CalibrationComplete field from the table makes your question moot.  If you do that and you really feel the need, you can still display a check box that will be checked when the date is entered simply by setting the check box's control source to the expression:

   =CalibrationComplete Is Not Null

and because expressions are read only, users can not make the mistake of checking the box when there is no date or by entering a date and forgetting to check the box.


ken sheridan say:

..............I had just assumed that conditional stuff like that had to be done while still editing the tables.

Well, it is possible to apply the validation as part of the table definition, but not by means of the  ValidationRule property.  It would have to be done by applying a CHECK CONSTRAINT which requires some knowledge of DDL, the data definition language components of SQL.   However, I completely agree with Scott and Marshall that it would be pointless to do so; simply use an unbound check box in a form and/or report whose ControlSource is an expression which evaluates to TRUE or FALSE, as Marshall describes.
