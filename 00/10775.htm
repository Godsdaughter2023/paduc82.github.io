jay freedman say:

The macro recorder has at least two limitations that mean it can't record the kind of operation you want. First, it only knows how to use the Selection object, which represents the currently selected text (or the insertion point if nothing is selected). Second, it has no concept of repeating an operation in a loop. For some background on this, you can read http://www.word.mvps.org/FAQs/MacrosVBA/ModifyRecordedMacro.htm.

Here's the code you need, with comments to explain what's going on.

Sub NRLlist()
    Dim src As Document
    Dim dest As Document
    Dim srcrg As Range
    Dim destrg As Range
    
    ' initialize Document variables
    Set src = ActiveDocument
    Set dest = Documents.Add  ' new blank document is the default
    
    ' initialize the source range in the source document
    Set srcrg = src.Range
    
    ' set up the search
    With srcrg.Find
        .Text = "<NRL-*>"
        .Wrap = wdFindStop
        .MatchWildcards = True
        ' all the other properties keep their default values
        
        ' create a loop to find all of the instances;
        ' each time one is found, .Execute returns True
        ' and srcrg moves to the found text
        While .Execute
            ' (re)initialize destrg to the end of dest document
            Set destrg = dest.Range
            destrg.Collapse wdCollapseEnd
            
            ' assigning .FormattedText from one range to another
            ' does a formatted copy without using the clipboard
            destrg.FormattedText = srcrg.FormattedText
            
            ' add a paragraph mark at the end
            dest.Range.InsertParagraphAfter
            
            ' move srcrg to the position after the current found text
            ' to avoid finding the same one again
            srcrg.Collapse wdCollapseEnd
        Wend
    End With
End Sub

jay freedman say:
The following macro, which goes in the ThisDocument module of the form, will check the date picker's text and, if it isn't a valid date, will display a message box and then put the cursor back into the date picker:

Private Sub Document_ContentControlOnExit(ByVal CC As ContentControl, Cancel As Boolean)
    If (CC.Type = wdContentControlDate) And (Not IsDate(CC.Range.Text)) Then
        MsgBox "Please choose a date in the calendar.", , "Invalid Date"
        Cancel = True
    End If
End Sub

There are two important caveats:
  • The form must be saved as a .docm (macro-enabled document) or .dotm (macro-enabled template). Otherwise the macro will be discarded (you'll be prompted for this).
  • In Word 2007 there is a bug regarding the ContentControlOnExit event that makes this macro work. See http://gregmaxey.mvps.org/Validate_CC_Entries.htm for a workaround.

ken sheridan say: The key issue is whether each side of a node has its own attributes other than its name.  If not, treating it as an attribute rather than an entity type is reasonable, and if you find it easier to work with that model, I see no overriding objection.  If each side has other attributes, however, then a separate NodeSides table would be essential.

Personal names would be regarded normally as attributes rather than entity types, though we could decompose a Customers table by creating a table CustomerNames with columns CustomerID, NameType, and Name, which would reference an 'all key' table, Names, with columns NameType, and Name.  Somebody I used to help out in the OECD actually did this to prevent errors in entering a wide variety of names in different languages from countries around the world.  It might sound over the top, but we'd have no qualms with a table, CustomerPhones, with columns CustomerID, PhoneType, and PhoneNumber, in which CustomerID and PhoneType are foreign keys referencing Customers and PhoneTypes tables.  I've done this very thing myself.  In modelling terms there is no difference in principle between phone numbers and names, but we judge to treat them in different ways.

The bottom line here is that, while the database relational model is a formal one based on scientifically demonstrable formal principles, there is always an element of judgement.
