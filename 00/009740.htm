a. user say:

Hi Sushil,

I suggest you to manually reset the Windows Updates Components and then install the updates.

a) Press Windows Key + X on the keyboard and then select “Command Prompt (Admin)” from the menu. 

b) Stop the BITS, Cryptographic, MSI Installer and the Windows Update Services. To do this, type the following commands at a command prompt. Press the “ENTER” key after you type each command.

net stop wuauserv

net stop cryptSvc

net stop bits

net stop msiserver

c) Now rename the SoftwareDistribution and Catroot2 folder. You can do this by typing the following commands in the Command Prompt. Press the “ENTER” key after you type each command.

ren C:\Windows\SoftwareDistribution SoftwareDistribution.old

ren C:\Windows\System32\catroot2 Catroot2.old

d) Now, let’s restart the BITS, Cryptographic, MSI Installer and the Windows Update Services. Type the following commands in the Command Prompt for this. Press the ENTER key after you type each command.

net start wuauserv

net start cryptSvc

net start bits

net start msiserver

e) Type Exit in the Command Prompt to close it.

Now you may try running the Windows Updates and install the new build and check if the above steps resolve the issue.

For further reference, you can refer to the link:

Okay, that's a different question. To open the notes pane to view already created notes, you can do one of two things:

How to: Call sotred procedures from Word - Microsoft Community

grmihel2 say:

How to: Call sotred procedures from Word

I've been told from a dear MVP, that it is possible to create a stored procedure, and call it from Word some how?

The issue is that I need a mail merge, where I merge info from MORE than one database table (MSSQL express), and by creating a SP, I can select the info that I need (standard mail merge can only select one table per doc after what I could conclude).

 

So my question here is, how do I call a Stored Procedure from Word, so I can mail merge the info into my document?

a. user say:

I wish I'd seen your original question earlier. I suspect Cindy has momentarily confused Views and SPs

AFAICS your query could be a View, so unless that's just an example for discussion and the real situation requires parameters, if you need to use Mailmerge you need to use CREATE VIEW rather than CREATE PROCEDURE).

However, if it's a 1-many merge you need, MailMerge can't handle that out of the box. Two approaches are often referenced here:

Macropod's approach, which uses Merge fields and no VBA (except I think where he deals with merge to email, where that approach cannot work and he uses VBA):

http://www.gmayor.com/Zips/Catalogue%20Mailmerge.zip

Doug Robbins' approach, described on Graham Mayor's page at:

http://www.gmayor.com/ManyToOne.htm

You may be able to get one of these to work with your SP/VIEW. You might find that you can adapt Doug's code for your needs.

However, it's difficult to answer the question about "the best way." What to do depends a lot on the requirement, which version(s) of Word you have to work with, and your level of familiarity with the technologies involved.

As far as the requirement is concerned, it's usually down to "who exactly has to be able to do what?" - if end users have to be able to create or modify layouts, it's a lot harder to get this right no matter what technology you use. If you are defining the layouts yourself and only developers will ever change them, then you have a lot more leeway - you can either "roll your own Merge" in either VBA, using .NET+Automation, .NET+VSTO, or maybe even .NET+OpenXML SDK, although I suspect that using the OpenXML SDK for this is quite hard. Or you can use MailMerge+Word's MailMerge Events, set the Mail Merge Data Source to be the Customers. In the MailMergeBeforeRecordMerge event you then retrieve the records for that customer using ADO (in VBA) or ADO.NET (I guess, in .NET/VSTO), insert them, and format them however you need.
