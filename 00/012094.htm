steve rindsberg say:
It's a sort of bug/not-bug, I suspect.

In the Windows version of PPT, masters that have no slides depending on them will be deleted on save;  this is by design and generally it's a good thing, since unwanted/unneeded masters tend to accumulate like crazy in some situations.

BUT in the Windows version, you can right-click a master and choose Preserve Master.  If you do this, the master won't be deleted, even though it has no attached slides.

So much for the not-bug part of our program.  

The Mac version seems to lack this Preserve Master feature.  One workaround would be to add at least one slide based on every master you want to preserve.  Hide the slide so it doesn't appear in slideshows.

In the Windows version, this bit of VBA has the same effect as selecting Preserve Master for each of the masters in a presentation.  Give it a try on your Mac presentation:

Sub PreserveMasters()

' Design is VBASpeak for SlideMaster
Dim oDes As Design

With ActivePresentation
    For Each oDes In .Designs
        oDes.Preserved = True
    Next
End With

End Sub

daniel g. grau say:
The .tmp file you see is the result of an interrupted renaming process. Word creates a temporary file whenever you save the currently-open document. Usually, the original file is then deleted and the .tmp file is renamed to become the new saved file. It seems that Time Machine kicks in and establishes a lock on the .tmp file to back it up, but it either does not release it (unlikely) or Word does not retry accessing it (likely). In either case, Word is prevented from renaming it and panics. At that point, all you are left with is that .tmp file that could not be renamed.

A similar "create .tmp-delete original-rename .tmp" process happens during AutoRecover (wrongly referred to as "AutoSave")† and Backup, compounding the problem. And it becomes worse if there are pending Undos and unsaved changes, since these may also be stored in .tmp files until discarded by a Quit/Close or added to the file. That's why some .tmp files appear to be "useless" while others contain the last effectively-saved version of the document.

After some days of testing as per this thread, user JES910 concluded that the only way to achieve stability was to exclude the current working folder from Time Machine backups, plus either stop the AutoRecover feature or exclude the Office 2011 AutoRecovery folder from Time Machine's backup.

Thus, until the situation is fixed, create a common working folder for all your files and exclude it from Time Machine's backup, as well as the other folder mentioned above. You can then enable "Save AutoRecovery..." and "Create a backup copy" in Word's Save preferences.

† For a further explanation of why AutoRecover is not an AutoSave function, see these posts:

  • Bob Jones's 5/14/12 post in AutoRecover files missing
  • My post marked as Answer in How do I retrieve an unsaved Word file on a Mac...
