joeu2004 say:

Julie wrote:

Anyone knows whether the RANDBETWEEN formula second result is independent with the first result or not?

I assume that you are using RANDBETWEEN in Excel or WorksheetFunction.RandBetween in VBA.

Neither function uses the same "seed" as the VBA Rnd function.  So neither function is affected by the VBA Randomize statement.  See footnote [3] to demonstrate that fact.

Instead, RANDBETWEEN and WorksheetFunction.RandBetween use the same "seed" and pseudo-random number generator (PRNG) as Excel RAND.

The initial "seed" for RAND is randomized each time we start a new instance of Excel.  Excel does not allow us to control the initial "seed" [1].  Each new instance of Excel generates a new sequence of RAND values.

In contrast, VBA Rnd uses the same initial "seed" with each new instance of Excel, unless we use VBA Randomize to create a new "seed".

Technically, no software PRNG creates a truly independent sequence of random values.  But ideally, it emulates an independent sequence.

You will find a variety of opinions on the quality of the Excel and VBA PRNGs.  IMHO, they are good enough for most of us.

The biggest problems with VBA Rnd is its extremely short period (how many calls before repeating the same sequence) and the fact that the "seed" can degenerate to zero, causing Rnd to return zero repeatedly.

Before Excel 2010, Excel RAND had a significantly longer period.  And I do not believe the "seed" could degenerate to zero.

(Note:  That is not the same as whether or not RAND could return zero.  I believe it could.)

Starting in Excel 2010, Excel RAND was significantly improved, replacing the previous Wichman-Hill algorithm with a Mersenne Twister (MT) algorithm.

In general, the MT paradigm is thought to be the best PRNG.  But there are design choices that can "make or break" it.  So that says nothing about the quality of the Excel MT algorithm per se.

[ERRATA] Previously, I wrote that Excel RAND used a linear congruence algorithm before Excel 2010.  Actually, it is VBA Rnd that uses a linear congruence algorithm. [2]

-----

[1] Some very early versions of Excel did allow for specifying the initial "seed" on the Excel command line.  That was some time before Excel 2003.  I forget when.

[2] The VBA Rnd algorithm is described in support.microsoft.com/kb/231847.

The Excel RAND algorithm for Excel 2003 and 2007 is described in support.microsoft.com/en-us/kb/828795.

[3] To demonstrate the independence of Excel RAND (ergo, RANDBETWEEN and WorksheetFunction.RandBetween) and VBA Rnd, create a new Excel workbook with two worksheets and perform the following steps.

In the Sheet1 object, create the following VBA procedure:

Private Sub doit()
Dim r As Long, i As Long
r = Cells(Rows.Count, "a").End(xlUp).Offset(1).Row
For i = 1 To 10
Cells(r, i) = WorksheetFunction.RandBetween(1, 100000)
Next
MsgBox "done"
End Sub

In the Sheet2 object, create the following VBA procedure:

Private Sub doit()
Dim r As Long, i As Long
r = Cells(Rows.Count, "a").End(xlUp).Offset(1).Row
For i = 1 To 10
Cells(r, i) = Int(100000 * Rnd) + 1
Next
MsgBox "done"
End Sub

Save and close the workbook.

Open the workbook and execute each VBA procedure.

Save and close the workbook a second time.

Open the workbook and execute each VBA procedure.

In Sheet1 (created by RandBetween), you will see two different rows of values.

In Sheet2 (created by Rnd), you will see the same sequence of values in each row.

Well I have found a solution.  It is not the greatest solution from a user's perspective and I am really sad Microsoft has built something which needs a registry hack to correct.

 

Also this is NOT a SHarePoint issue but is an Office issue.

 

To fix this you do need a registry hack using regedit.  This is not normally advised and before you start export a copy of the registry in case the cat walks across your keyboard or something.

 

In the registry go to

<HKEY_CURRENT_USER>\Sofyware\Microsoft\Office\14.0\Common

 

In this section you will find something like Shared Sites inside of which you will find several references to MySite.  Inside each of these will be the url which references the site.  There will also be DWORD Values of IsMember and IsPublished.  Delete the whole tree containing the urls you no longer need.

 

If you want none of the Save and Publish entries in your Office applications you can delete the tree Shared Sites tree from Common.

 

Good luck and do not forget to take the backup before you start to hack

MS Word Quick Parts and SharePoint Collaborative Editing - Microsoft Community

karl f. kamysek say:

MS Word Quick Parts and SharePoint Collaborative Editing

I have a small team working on a fairly complex document that we are collaboratively editing on SharePoint.  In an effort to standardize the formatting I built several Quick Parts for inserting intricately formatted sections of the document that will be repeated throughout.

I would like the Quick Parts available to the rest of the team, much like the Quick Styles I built for the document are when they open the document for editing from SharePoint.  However, the Quick Parts I prepared are only showing on my local machine/copy of Word, and not on my team members.  How do I save/embed the custom Quick Parts I built for this document so they are available to all?

Thank you!

Karl

 

jay freedman say:

When you save a Quick Part, one of the dropdowns in the dialog box is "Save in". It defaults to Normal.dotm, but you can choose any template that's currently open/loaded.

For a document that's being edited collaboratively, the document should be based on a custom template specific to that project, and each editor should have access to that template (usually by placing the template in a shared folder and setting the Workgroup Templates location in each user's copy of Word to that folder). If that's the case, then the Quick Parts should all be in that template.

You can move the existing Quick Parts to the template as follows:

While that template (or a document based on it) is open, click Insert > Quick Parts > Building Block Organizer. Select one of the Quick Parts you made, click the Edit Properties button, and change the Save In setting to the name of the template. Repeat for the others. When you close Word, you'll be prompted (separately) to save changes to the template and to Normal.dotm; click Save for each one. You have now moved those Quick Parts out of Normal.dotm and into the document template.

If you aren't using a shared template, use the same process to move the Quick Parts into a new template. Move that template to the folder %appdata%\Microsoft\Word\Startup and restart Word. The Quick Parts in the template should be available in Word. You can then distribute a copy of the template to each of the other users, with instructions to store it in their ...Word\Startup folder.
