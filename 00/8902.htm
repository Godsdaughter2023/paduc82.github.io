ken sheridan say:

In the language of the database relational model, when the word key is used without any further qualification it normally refers to the primary key, but keys come in a number of flavours:

1.  Primary key:  This is a column or several columns in combination whose value is distinct for each row in a table and provides a unique identifier for the row.   The column or columns are defined in the table design as the primary key.  An example of a single column primary key would be an EmployeeID column in an Employees table.  This would be a numeric column, usually an autonumber, whose values are assigned automatically when a row is inserted into the table.  An example of a composite primary key would be in an OrderDetails table where the key comprises OrderID and ProductID columns in combination.

2.  Candidate key:  Tables will sometimes have more than one column or set of columns whose values are distinct in the table.  While only one can be the primary key the others are candidate keys and are defined as such by inclusion in a unique index.  A table of US states for instance could have an autonumber StateID column as the primary key, and a text State column, which would be a candidate key as US state names are distinct, so the State column would be indexed uniquely.

3.  Foreign key:  This is a column or set of columns in a table which maps one or more rows in that table to a single row in another table.  The value of the foreign key is that of the primary key of the row in the other table which it references.  An example would be a StateID column in a table of US cities.  This would identify the state in which the city is located.  A foreign key is defined in Access by creating a relationship between the tables, e.g. between States and Cities on StateID, and enforcing referential integrity.  A foreign key column should be indexed non-uniquely (duplicates allowed) except in those rare cases where the relationship type is one-to-one.

In the example of an OrderDetails table given above the OrderID and ProductID columns are each a foreign key referencing the primary keys of Orders and Products tables.  What this table represents is a many-to-many relationship type between Orders and Products, which it resolves into two one-to-many relationship types:

Orders----<OrderDetails>----Products

Can you confirm that you have notifications enabled for both the settings in the Mail app, as well as in Windows.

To enable notifications in the Mail app:

  1. From Mail go to Settings -> Notifications.
  2. Select the account that you would like notifications enabled on under the heading “Select an account”. 
  3. Make sure that “show notifications in the action center” is turned on.
  4. Check the options you would like to enable:  Show a notification banner, Play a sound, and Show notifications for folders pinned to Start.
  5. Note that for the last option you will need to have the folder pinned to receive a notification for a specific  folder. 

You will need to check this for each account you want to receive notifications.

To enable/configure notifications in Windows Settings:

  1. Open notification settings from the Start menu select Settings -> System -> Notification and actions.
  2. Click on Mail under the heading “Get notifications from these senders.”
  3. Make sure that Notifications is on, as well as any other options that you would like to enable.

Note that you can also change settings for pinned folders, however the folder needs to be pinned, the option needs to be enabled in the Mail app, and you must have received at least one email for the pinned folder before it will be listed under “Get notifications from these senders.” It will be listed with the title(s) of the folder(s) that have been pinned and meet these requirements.
