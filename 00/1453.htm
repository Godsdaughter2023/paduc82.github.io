Why aren't custom fields in Task view not available in Resource - Microsoft Community

philippebruno say:

Why aren't custom fields in Task view not available in Resource view?

The answer always boils down to one has to write a macro, but why? I understand this is some kind of limitation, but it does not make sense to me. By any chance, is there someone with a clear explanation as to why it is not possible?
dale howard [mvp] say:

Philippe --

What you are not understanding is the Microsoft Project Data Model.  In a Microsoft Project schedule, there are two types of data:  Task data and Resource data.  This data model controls what you can see in Microsoft Project.  When you apply a Task view, such as the Gantt Chart view, you can ONLY APPLY Task tables, Task filters, Task groups, and Task columns (fields).  When you apply a Resource view, such as the Resource Sheet view, you can ONLY APPLY Resource tables, Resource filters, Resource groups, and Resource columns (fields).  This is just how the software works, which is why you cannot display a custom Task field in a Resource view, and why a macro is the only way to get that data to populate.  Hope this helps.

john - project say:

Philippe,

Pardon me for jumping in. Dale gave you a good explanation but there are actually three types of data in Project's object model, task, resource and assignment. You might want to take a look at the following Wiki articles:

And yes, there are some Project fields that do cross data types (e.g. Project field).

richard xu msft say:

Hi EF,

If you deploy Outlook via RDSH, please do not use the cached Exchange mode, since the .OST file is designed to be local, and since RDSH environments create a new session for each user, that user’s session won’t include their .OST file. So the options are:

  • Copy the .OST file to the RDSH server as part of the login script. 
  • Remotely mount the .OST file across the network.

The .OST files can be pretty huge since they’re the full size of the user’s mailbox, often times being tens of gigabytes), and copying them to a user’s session on logon isn’t practical. And since .OST files were designed to be local, they assume local-level disk I/O, meaning if you put them on a file share accessed via SMB then you spike the network and slow down Outlook, so that doesn’t work either.

Also, Outlook 2016 in Exchange server is not a best practice. Normally, we use the Outlook in a normally personal computer (Windows operating system).
