OK, now to tackle the issue of finding the next empty cell in column B to begin putting your values from the UserForm into.  See my previous post for explanation of the use of Sheet9 in this example:

 

Replace the one line of code you have now that reads:

RowCount = Worksheets("Sheet1").Range("B17").CurrentRegion.Rows.Count

 

With this long winded piece of code:

 

Dim searchRange As Range
Set searchRange = Sheet1.Range("B:B")
On Error Resume Next
RowCount = searchRange.Find(What:="*", _
                        After:=searchRange.Cells(1), _
                        Lookat:=xlPart, _
                        LookIn:=xlFormulas, _
                        SearchOrder:=xlByRows, _
                        SearchDirection:=xlPrevious, _
                        MatchCase:=False).Row
If Err <> 0 Then
  'was an empty column!
  RowCount = 0 ' will get set to 1 later
  Err.Clear
End If
On Error GoTo 0 ' reset error trapping to 'normal'
'limit start row to specific row
If RowCount < 17 Then
  RowCount = 16
End If
Set searchRange = Nothing ' no longer needed
RowCount = RowCount + 1 ' points to next empty row now

 

At this point, RowCount will point directly at either row 17 or at the next empty cell below row 17 in column B.  So we can rewrite the With ... End With block like this now:

 

With Sheet9.Range("B" & RowCount)
         Offset(0, 0).Value = Me.txtPTitle.Value
        .Offset(0, 1).Value = Me.txtCNumber.Value
        .Offset(0, 3).Value = Me.cboDay.Value
        .Offset(0, 5).Value = Me.txtSTime.Value
        .Offset(0, 6).Value = Me.txtETime.Value
        .Offset(0, 9).Value = Me.cboPStage.Value
End With

 

And I think that should take care of things nicely.

jllatham say:

First piece: validating the value of the input

Entries on forms are really all text (as you probably already know), so let's say it was entered into TextBox1, then you can use the VBA VAL() function to turn text into a real number and then test.  After verifying that it IsNumeric() put this inside of that IF..,Then test as a second one:

If Val(TextBox1.Value) < 99 Or Val(TextBox1.Value) > 1000 Then
   MsgBox "Not between 99 and 1000"
Else
   MsgBox "Is 99-1000, inclusive."
   MyNumber = Val(TextBox1.Value)
End If

Of course, after having tested for IsNumeric() and finding that it was you could assign it to an Integer or other numeric variable and then test that for <99 or >1000 using the VAL() function again.

You can still use IsDate() to test if an entry is a valid time:

Private Sub CommandButton1_Click()
  Dim t1Entry As String
 
  t1Entry = Me!TextBox1.Value
  If IsDate(t1Entry) Then
    MsgBox "Is date/time"
  Else
    MsgBox "Is not date/time"
    Exit Sub
  End If
  MsgBox "Hour is: " & Hour(t1Entry)
  MsgBox "Minute is: " & Minute(t1Entry)
  MsgBox "Seconds are: " & Second(t1Entry)
End Sub

Of course that hasn't validated that it is a TIME rather than a DATE - you need to add more validation to see if it is one or the other - such as varifying that there are colons in it to separate hours:minutes:seconds (seconds being optional).  You haven't said what you are expecting the inputs to look like; 24-hour entries or with A.M./P.M. as part of the entry.

The code above will let you try various inputs and see how the hour/minute/seconds portions of your entry are evaluated.  If you just entered a date, then hour = 0, minutes = 0 and seconds = 0.  So non-zero values for those would indicate you have an entry that can be used for a time.

There are some other ways to check it - take a look at this page:

http://excel.tips.net/T003292_Checking_for_Time_Input.html

Projected % complete at a future date in MS Project - Microsoft Community

colij say:

Projected % complete at a future date in MS Project

I want to produce a report that shows the expected % complete at a defined date in the future, e.g. next month end, or year end.  Is there a standard report available or a reasonably quick way to calculate it?

john - project say:

Colij,

When you say "expected % complete", what is that based on? In other words, how would you determine the amount of completeness at any given time? If it is just based on a linear passage of time, that's pretty easy to calculate but it doesn't tell you anything. Why? Because Percent Complete is not a good measure of progress as it is based on duration and duration, (i.e. passage of time), accomplishes nothing. A much better measurement is percent work complete since it measures how much work effort has actually been accomplished compared to the planned effort.

The standard method for projected progress is earned value and there is a whole set of metric for that (i.e. BCWS, BCWP, ACWP, etc.). Earned value metrics measure performance compared to the original plan (i.e. baseline).Have you considered using that?

John

rod gill say:

As John said, earned value can tell you a lot, but only if the original schedule was reasonably accurate and a Baseline taken. Then accurate tracking done and the schedule updated to reflect lessons learned. For example if a particular team or contractor just doesn't have the resources expected/promised then future tasks need to reflect this.

A good schedule can communicate what needs to be done to achieve success, but cannot really say expected progress at a future date with any realistic accuracy. If management are insistent on getting a value, then I would keep some dice in your pocket to use for guessing estimating expected progress.

Good luck!

trevor rabey say:

Start with a complete and correct project plan which has no progress recorded for any tasks at all.

Set a status date to the end of the first week. Display the status date as a vertical red line on the chart. Select all tasks which are either entirely or partly on the left of the red line and click "mark on track". Note the result and save the file.

Then move/reset the status date to the following week, and repeat.

Then move/reset the status date to the following week, and repeat, etc week after week.

Then you will have the forecast progress as at the end of each week of the project. The result will include the progressive % complete, ie the progress of duration, as well as the progressive/incremental work and cost.

% complete is actual duration/total duration, where total duration is actual + remaining. It has some meaning at the task level, but no real meaning for collective groups of tasks when rolled up to the summary level, and less to zero meaning when rolled up all the way to the project summary.

I would not say that it is the "expected" % complete, because my expectation would be that things will not go as planned, but it is useful to forecast the incremental as planned progress in advance.
