ken sheridan say:
I had always used lookup tables to create reference lists to select from - I always thought it was that or free text (which I can't bear the thought of for an indexing field).  Is there another way then?

Don't confuse look-up tables with the look-up field wizard.  It's the latter which is deprecated, not the former.  A look-up table is really no more than the referenced table in a one-to-many relationship type, i.e. the table on the one side of the relationship, so is fine.  You might for instance have a table Cities related one-to-many to a table Addresses, with the later including a CityID foreign key column referencing the primary key CityID column of the Cities table.  In a form for entering data into the Addresses table you'd use a combo box set up like this:

ControlSource:    CityID

RowSource:     SELECT CityID,City FROM Cities ORDER BY City;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.

This would operate in the same way as if you'd used the look-up field wizard when designing the Addresses table, in that you'll see the city names listed, but the value of the combo box will be the numeric CityID from the hidden first column.  Unlike the look-up field wizard, however, this does not have the detrimental side effects which are detailed at the link Marshall gave you.

To cut to the chase, the main reason for my chipping in here is to come back to your original question about searching..  If you use the conventional way of modelling a many-to-many relationship type by means of a 'junction' table rather than a multi-valued field this is achievable relatively easily.  You'll find an example in the file MultiSelect.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

In this little demo file the table ProjectEmployees models a many-to-many relationship type between Projects and Employees - you'll see this if you open the relationships window.  On the opening form is a button captioned 'Filter Report on One or More Projects in Related Table', i.e. its a means of searching, via a dialogue form, on the multiple projects per employee and will return those employees who are assigned to any of the projects selected in the dialogue form.  It works by means of the following code in the Click event of the button on the dialogue form to open the report:

    Dim varItem As Variant
    Dim strProjectIDList As String
    Dim strCriteria As String
    Dim lngEmployeeID As Long
    Dim ctrl As Control
   
    Set ctrl = Me.lstProjects
   
    If ctrl.ItemsSelected.Count > 0 Then
        For Each varItem In ctrl.ItemsSelected
            strProjectIDList = strProjectIDList & "," & ctrl.ItemData(varItem)
        Next varItem
       
        ' remove leading comma
        strProjectIDList = Mid(strProjectIDList, 2)
       
        strCriteria = "EmployeeID IN" & _
            "(SELECT EmployeeID FROM ProjectEmployees " & _
            "WHERE ProjectID IN (" & strProjectIDList & "))"
       
        DoCmd.OpenReport "rptEmployees", _
            view:=acViewPreview, _
            WhereCondition:=strCriteria
    Else
        MsgBox "No Projects selected", vbInformation, "Warning"
    End If

The way it works is that it firstly loops through the selected items in the multi-select list box of projects and builds a list of ProjectID values, separated by commas.  This list, is then used to build a criterion for filtering the report when it's opened.  This is a little more complex than simply filtering on some field = some value as what is needed here is to filter the report to those employees who have any one of the selected projects assigned to them, i.e. those rows in the Employees table which is related to at least one row in the ProjectEmployees table with any one of the projects selected in the dialogue form.  This is done by building a string expression for the criterion as follows:

        strCriteria = "EmployeeID IN" & _
            "(SELECT EmployeeID FROM ProjectEmployees " & _
            "WHERE ProjectID IN (" & strProjectIDList & "))"

In plain English this is saying return those employees whose EmployeeID is in any row in the ProjectEmpoyees table which also has a ProjectID which is one of those in the list of ProjectIDs built from the selections in the list box.  It does this by using the IN operator against a subquery, which returns multiple EmpoyeeID values, whose WHERE clause uses the IN operator against the list of ProjectIDs, so the subquery only returns those rows with a ProjectID in the list. QED.
