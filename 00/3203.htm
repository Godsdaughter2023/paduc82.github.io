toddntucker say:

convert footnote to inline citation

I am looking for an efficient way to convert numbered footnotes from Word files to numbered inline citation.

I have tried using a macro (like the one here: http://bit.ly/16ZMr4t) to convert to inline citations.

One problem however:

1) In the macro suggested in the link, the footnote number is not preserved - a problem when one footnote refers by number to previous footnotes.

Any tips would be appreciated.
jay freedman say:
Try this version:

Sub foot2inline()
Dim oFeets As Footnotes
Dim oFoot As Footnote
Dim oRange As Range
Dim szFootNoteText As String
Dim index As Long

' Grabs the collection of FootNotes
Set oFeets = ActiveDocument.Footnotes

' Iterates through each footnote
For Each oFoot In oFeets
    index = index + 1
    szFootNoteText = oFoot.Range.Text
   
    'Start search from beginning of document
    Set oRange = ActiveDocument.Range
     
    With oRange.Find
        .Text = "^f" ' Looks for all footnotes
        .Forward = True
        .Wrap = wdFindStop
        .Execute
    End With
   
    ' Delete the footnote
    oFoot.Delete
   
    'Insert the footnote text
    oRange.Text = " [Note " & index & ": " & szFootNoteText & "] "

    'CHANGE COLOR HERE. Color code is below.
    'oRange.Font.Color = 6299648

    'Disables undo to save memory on very large documents.
    'ActiveDocument.UndoClear
Next
End Sub

The color change and the UndoClear are optional. If you want either or both, delete the apostrophe at the beginning of the corresponding code line.

a. user say:

I had to have quite an extensive look around to find out what was going on here, but am not quite in a position to summarise well. So here are some remarks..

Before anything else, doing this exploration has made me realise how using anything except character sets that are defined in Unicode increasingly spells "trouble". It shouldn't, because there will always be reasons why people need to define their own character encodings. 

Wingdings provides a good illustration. 
 a. The character set wasn't adopted into Unicode (unlike Dingbats, which is either older or much older and was not a Microsoft-only character set). You will find some of the same symbols scattered through the Unicode set, and as long as you can use a font that contains them, it's probably better to use those characters instead. Some of them are in the newer, higher Unicode ranges. For example, the Wingdings "Open Book" character is in the Emoji range. But as far as I know, you can't use anything above hex FFFF in either Windows or Mac Word. (Not even using the Unicode Surrogate mechanism).
 b. So you can't get Wingdings from the Mac Character Viewer. 
 c. There used to be support for special character sets such as Wingdings in Mac's onscreen keyboard viewer (this is perhaps relevant to your other question). But that was reduced in Mac OS X Leopard and eliminated in Snow Leopard. (If you have Leopard or Tiger, you may find a dropdown in the onscreen keyboard that lets you select Wingdings and type with that).
 d. Because Wingdings isn't in Unicode, Microsoft has decided that when they store a Wingdings character code such as Open Book (38, or hex 26) in a Word document, they will give it a code in the "Private Character" area of Unicode. That's despite the fact that they always actually store the font name too. So they actually store the character as hex F026. The rationale is given in the ECMA/ISO standards for the Office Open XML standards. In essence, Microsoft was trying to allow for other software that would ignore the font name and push out a Unicode 38 character (i.e. an ampersand). If the code's in the Private Use area, such software is more likely to substitute something potentially less confusing, such as a block or a question mark.

<<
It seems so weird that the chosen font set is so conspicuously absent from the code after Inserting a symbol from the menu bar, but I guess there is no defined (ie. readable) font selection during this process; just cursor dragging.
>>

First, what is recorded depends on which symbol feature you are using. 

If you are using Insert->Symbol->Advanced Symbol...
 a. Word ought to be able to insert the Font name that it has displayed in the dialog box.
 b. Mac Word 2011 does not insert the Font name.
 c. I looked at Windows versions of Word and Mac Word 2004 (and another language version of Mac Word 2011 FWIW) and they all insert the Font name when recording in the situation where the font is not the "current font". Only Mac Word 2011 does not. If you insert a character from the (normal text) font, newer versions of Windows Word insert something in the font name parameter (not necessarily the font name, it can be something like "+font")

If you are using Insert->Symbol->Symbol Browser...
 a. Word does insert the font name, but
 b. the selection available is very small. Perhaps I am missing something there, but I don't know how to change that selection. It would be nice if you could drag stuff over from the Mac Character Viewer or some such.

If you are inserting from the Mac Character Viewer,
 a. you won't find the Wingdings characters in there anyway because they are not part of the Unicode standard
 b. but if you do insert a character, VBA does record the insert, but in a different way (it records a Selection.TypeText command).

Second, VBA the VBA recorder and the Word Object Model have all evolved a long time. Even the youngest part (VBA) is over 15 years old. As with most commercial software, that tends to mean the "quirk count" is quite high. I'm actually pleasantly surprised that the Symbol Browser inserts are recorded at all, because several of the newer features on Windows Word are not recorded at all by the VBA recorder. And there's a salient quirk in this case, which is that on both Mac and Windows Word, the Object model says that the Character number is a "Long integer" (which ought to be able to hold the value of any Unicode character, even beyond the highest possible 16-bit character code (hex FFFF)). But as I mentioned before, VBA only actually allows us to assign a signed 16-bit value.

The other problem with the recording feature of VBA is that it's quite simplistic. At best, it could nly capture what the user does, and cannot know the user's intent. For example, if you select a piece of text and record the action of changing the font size from 12pt to 14pt, then re-use the macro, what was your intent? Set the font size to 14pt? Or increase the font size by 2pt? Or by 1/7th? And so on... And in fact I think in that case Word might record all the font characteristics - i.e. if you applied it to a different font, you might see an unexpected font change.

So recorded macros fall into the "useful if they do what you want, when you want" category. I don't suppose many end users actually want to start messing around with VBA, but that's one of the only other options available if they want to make their macros more useful. Generally speaking, recorded code is a bonus because it provides a starting point, unless it records nothing. IN some cases, it might set you off in an unhelpful direction, but often it's a question of finding out what you need to add (such as the font name) or remove (such as stuff you don't want to change).

OK, as I mentioned in the my other post just now, if you are still on Mac OS Leopard or Tiger you may find a drop-down in the on-screen keyboard that will let you select Wingdings (say), and show you which character you can get from which key. Since 224 is a lot larger than the key count on a keyboard, even in that case you will have to use modifier keys (alt/option etc.) to find all the codes.

However, I am not sure that will help, because all such a keyboard can do is send a character sequence to Word, and if Word actually has a different font selected, you will see the wrong characters appearing.

For example, if your keyboard is set up to insert Wingdings, and you type the "Open Book" key, it might generate a character hex 26.  c. as long as you have a single set of fixed sequences that you want to set up, it might be easier to work with, and fix the VBA.

ken sheridan say: The CustomerID column in the JobCard table must not be an autonumber, but a straightforward long integer number data type, indexed non-uniquely (duplicates allowed).  This should reference the autonumber primary key of a Customers table.  In database terms what you appear to have is a many-to-many relationship type between Contacts and Customers, modelled by the JobCards table, which resolves the relationship type into two one-to-many relationship types as below:

Contacts----<JobCards>----Customers

You can build an interface for data entry in a number of ways.  Firstly you should create simple forms, in single form view, based on the Contacts and Customers tables.  In these you'll enter data for each contacts and customer.

You can if you wish build a free-standing form to enter data into JobCards.  This will have controls bound to all columns in the table, of which the CiontactID and CustomerID controls will be combo boxes.  Set these up as in the following example for Contacts:

ControlSource:    ContactID

RowSource:     SELECT ContactID, FirstName & " " & LastName FROM Contacts ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

Alternatively, you can concatenate the names so that the last name is first, which would be better with a large list of names as it allows the user to enter the initial characters of the last name and progressively go to the first match as each character is entered:

RowSource:     SELECT ContactID, LastName & ", " & FirstName FROM Contacts ORDER BY LastName, FirstName;

However, selecting a person purely by name is not really satisfactory, as personal names can be duplicated (I worked with two Maggie Taylors).  To differentiate between them, further details can be shown in another column or columns of the combo box's list, e.g. job title, department etc.

While in this from you can enter a new contact or customer not yet represented by typing their name into the combo box.  For this you must first set the control's ListItemsEditForm property to the name of the customers or contacts form as appropriate.

The alternative would be to embed the job cards form as a subform in the contacts and/or customers from, linked on the relevant primary/foreign key columns.  That way you could navigate to a customer or contact in the parent form and enter job cards data in the subform.  You would then not need a combo box in the subform for the relevant 'parent' as the value would be automatically inserted into the JobCards table by the linking mechanism.

After upgrading to Office 2010 from 2007, I had this issue where Excel was not appearing in the "open with" menu and *.xlsx files were not automatically associated with Excel.

 

To fix this issue I did the following:

 

1. Open registry and navigate to : HKEY_CLASSES_ROOT\Applications\EXCEL.EXE\shell\open\command\

 

2. Change the "Office12" in the string to "Office14". Mine now looks like this "C:\Program Files (x86)\Microsoft Office\Office14\EXCEL.EXE" "%1" (with the quote marks).

 

Yours may looks slightly different depending on your installation folder.

 

Hope this helps someone.
