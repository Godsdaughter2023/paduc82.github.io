scott mcdaniel say:

That's not the way you work with references on end user machines.

Assuming the reference is deployed correctly to the end user machine, installed and registered, then VBA should make that reference for you. If it does not, then something is wrong with the reference on that machine, and you can't fix that with code.

If the reference is there, then you simply deploy your database with the reference ticked and Access/VBA should do the rest. 

With that said, you must be sure to comply with the "rules of references":

You must first determine the "lowest common denominator" in regard to that reference. For example, if 3 users are running Version 1.1.2 or that Reference and 2 users are running version 1.1.1, then you MUST use version 1.1.1 when creating your database. If you use 1.1.2, then the 1.1.1. users will NOT be able to remake the reference, and your app will fail.

Trying to use the References method you suggest is a haphazard (at best) solution, since an application with a missing reference has no guarantee that ANY VBA code will execute correctly (or at all).

To put it a different way - you should adapt YOUR development environment to fit your user's. You should NOT adapt your end user's environment to fit your own.

The only exception to that caveat would be where you've declared a "Prerequisite", which is a file/library/etc that is REQUIRED for your operation to function correctly. In those cases, you would generally have an installer that would check for that requirement. If the installer did not find the requirement, it would alert the user and exit the install.
