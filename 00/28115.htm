It also takes 1minute+ to then connect to the network.  During this process MsMpEng.exe has high cpu usage which is obviously causing this slowdown.  When I disable real-time protection in MSE (or uninstall) I don't have any hangs or slowdowns.

Is there a process or file I can add to the mse exclusions list to avoid these issues?


Thanks.

roland d. schneider say:

I found what seems to be the answer elsewhere on the Web. It appears that MSE is scanning files in the C:\Windows\tracing folder. Some of those files look like they are logs for the VPN connection process. Either MSE has changed, the files have grown over time, or something else has happened to make MSE spend a lot of time scanning those files when you do anything with the VPN connection.

The solution is to either:

1) Delete the files. I didn't do this because there are processes holding them open, and deleting files you don't understand is dangerous anyway.

2) Add C:\Windows\tracing to the list of excluded folders in MSE.

That fixed it for me.

*** UPDATE ***

I wasn't satisfied with the solution above because it shouldn't be necessary to exclude a Windows folder from scanning. I found that disabling tracing for VPN using the command (on Windows 7)

    netsh ras set tracing * disabled

fixed the problem without excluding the C:\Windows\tracing folder in MSE.

Looking at the contents of the log files in the tracing folder shows that tracing on my computer has probably been enabled since early September (about 4.5 months). I don't remember whether I enabled it or not. Once I removed the log files from the C:\Windows\tracing folder, I tried re-enabling VPN tracing. The log files where recreated and VPN continued to work normally (quickly), which, to me, implies that the problem is a combination of MSE behavior, having VPN logging turned on, and the size of the log files.

*** UPDATE 2 ***

I wrote a small test application in C# to see if I could reproduce the slow log file writing behavior. What I found was that repeatedly opening a large file (80MB) for append, writing a small amount of text to it (150 Unicode characters) and then closing it causes MSE to become CPU bound on one core of my workstation (in my case, that's 13% CPU usage) and makes the file writing process very slow. It appears to get slower as the file gets larger, but I didn't do any quantitative testing.

My guess is that MSE is processing the entire file every time it is opened for append, written to and closed. You do have to write to it - just opening and closing the file doesn't take any time. Most applications wouldn't write to files this way, which is probably why this doesn't come up all that often. However, it is a simple way to generate debug logs, which is what appears to be happening with the VPN software.

Once the file gets large enough (multiple GB), it becomes apparent that MSE is actually doing its work on file close, and that it appears to process the entire file, not just the new data that was written.

The template you are using has a header composed of Fields that update to display properties stored in the document or generated by Word.

ken sheridan say: Firstly you don't need to call both functions, the Now() function returns the current date and time.  You can either set the ControlSource property to =Now() and set the format property of  the control in its properties sheet to the desired format, e.g. the UK format of dd/mm/yyyy hh:nn:ss, or do it all in the ControlSource property with:

=Format(Now(), "dd/mm/yyyy hh:nn:ss")

For international use, a longer format such dd mmmm yyyy hh:nn:ss or the ISO standard for date notation of yyyy-mm-dd hh:nn:ss is better, to avoid ambiguity.

USB Tethering for Windows Phone 10 Internet Sharing - Microsoft Community

patchinswayze say:

USB Tethering for Windows Phone 10 Internet Sharing

I use my phone every day for a mobile hotspot, however, there are some disadvantages with using a wireless hotspot mode for tethering.

- Increased Battery Consumption

(With an active WiFi hotspot running, you are doing twice the work than what you would use with a USB connection, and of course you are charging off the computer as well)

- Less throughput than using a USB cable.

(Ref: http://www.imore.com/usb-vs-bluetooth-vs-wi-fi-whats-best-way-tether-your-mac-your-iphone-or-ipad: )

- Connectivity issues when used in area with many other WiFi networks.

( WiFi can only function on a handful of channels without overlapping transmissions. When you have four to five WiFi networks in an area, their signals will get tied up with other signals, causing packet loss, which slows internet connection.)

Apple and Android phones still provide this function which is just another thing that can turn people away from getting a windows phone. This feature used to available in Windows Phone 6.5, I would love to see it again. Other people think so as well...
