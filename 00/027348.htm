How to maintain correct reference to a cell in another spreadsheet - Microsoft Community

okau say:

How to maintain correct reference to a cell in another spreadsheet when sorting

Hello

I'm working in Excel 2010. I've got 2 spreadsheets (2 separate Excel files - let's call them 1 and 2). Spreadsheet 1 is the main work sheet where the data is entered, and Spreadsheet 2 shows results from the main spreadsheet.

Cells in spreadsheet 2 refer to the main spreadsheet 1.

The problem:

when I do sorting in any column in the main spreadsheet 1, formulas in the spreadsheet 2 still refer to the old cell number, although the correct value it is supposed to be reading is no longer there due to sorting.

For example:

 Cell A12 in the spreadsheet 2 refers to cell B1 with value CARS on the main spreadsheet 1.

When I do sorting in the main spreadsheet 1, value CARS is no longer in the cell B1. BUT spreadsheet 2 still refers to the same B1 sell which now contains some other value.

How can I make sure that my A12 cell in the spreadsheet 2 always finds value CARS in the spreadsheet 1 regardless of its location in the column?

Thank you in advance

vijay a. verma say:

You can do it with the help of a helper column.


1. In Sheet 1, in the last column enter sequential number 1, 2....till your last row. Type 1 in first cell, 2 in second cell, select first and second cell and drag down till last row. It will sequentially fill that column. Let's assume that this is cell C in sheet 1. (Edit - Column C not Cell C)


2. Now, to build the reference, you need to put following formula in cell A12 of sheet 2 (assuming only 100 rows. Adjust the rows as per your requirement) and drag down the formula.

=INDEX(Sheet1!$B$1:$B$100,MATCH(ROWS($1:1),Sheet1!$C$1:$C$100,0))

3. When you sort, you will have to sort along-with column C also where the sequential values are there.

Note: A12 is mentioned as per your example. Starting position might be A1 for you. The key is that it looks up your sequential number in sheet 1 and pulls out values from there.

Ok, I have a tip for you, but you have to use it at your own responsibility as it involves messing around with the windows registry. The following method will deactivate the new shortcut (captured by the Explorer) and reveal the still existing ("buried underneath" :)  ) old shortcut to OneNote (administrator rights to your Windows are required):

1. Copy the following command line to your clipboard:

reg.exe add HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Explorer\Advanced /v DisabledHotkeys /t REG_SZ /d S /f

2. Press [WIN] + [R].

3. Insert the copied command from the clipboard by pressing [Ctrl] + [V] and click OK

After reloading or relogging to Windows the old shortcut [WIN] + [Shift] + [S] should work again as before. In addition also [WIN] + [S] should work as it was in Windows 8. The latter was used to open Cortana search which is now disabled but still possible with [Win] + [V]

atif-hussain say:

Excel performance - prevent filtering triggering recalculations

Use case:

I have a large Excel file, with millions of formulae cells, all of it essential for its built-in logic. To make it work, all Excel performance aspects have been well researched and applied. And, the macro enabled Excel file (74MB xlsb) does work in doing its job. ... till... i try to filter something on the sheet.

General Excel performance issue:

Excel filter apply, row un/hide, cell resizing, etc. trigger volatile actions. Column un/hide no longer triggers recalculation.

For excel files of industrial applications, changing the view has no logical effect on values, except: subtotal.

Request:

We'd like a way to tell excel to stop causing-volatile-action/dirtying-all-cells/recalculation on view changes. This will make filtering really 'interactive'.

Suggestion:

option 1. Make subtotal as a volatile function. This will be a negligible performance hit compared to making view changes volatile action. For trade-off purposes, filtering is a very common usage (esp on large sheets) and needs to be interactive, compared to row hide, or subtotal. Subtotals even when used are limited and donot trigger downchain dependency.

option 2. Create 3 formulae types: non-volatile, view-dependent, fully volatile data, and 3 action types: input data change (recalc all 3), view change (recalc last 2), any event (recalc only fully volatile data). Could be more complex to implement, and may not be far better than option 1.
