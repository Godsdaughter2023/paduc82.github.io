First, your reply failed to answer the two questions that I asked. And that will probably be true for most -- if any -- other replies that others might post, because most people stop reading the OP after the first sentence.

Second, your remarks also describe Microsoft's interpretation of what comprises an Operating System (OS) and parrot the company line which mentions only one advantage of a 64-bit system. What you say about "64-bit Windows 10" with regard to "address space" is true for 64-bit Windows 7 and 64-bit Windows 8, too, so what is your point (rhetorical question)? Regardless, the "address space" is not the only reason for using a 64-bit system.

For a given clock speed, a 64-bit CPU can process almost twice as much No one has ever bought a computer because they wanted to use its OS!! Rather, we buy computers because they are useful tools for many things, all of which require processing as much data as possible, as fast as it can be feasibly processed.

Any 32-bit OS "components" cannot process data as fast as 64-bit components would process it. Thus, only the 64-bit components of "64-bit Windows" gain the advantages of 64-bit hardware, compared to 32-bits. The same is true with respect to comparing the 64-bit and 32-bit software applications that "64-bit Windows" monitors.

Each 32-bit process cannot address more memory on a 64-bit system than it can address on a 32-bit system, i.e., regardless of the size of the address space that is made accessible to the 64-bit Windows kernel. Also, 32-bit software will not process any more data in the same amount of time than it would process if it were running on 32-bit hardware at the same clock speed (whether it might even process less data while executed with the 64-bit Intel/AMD CPU "compatibility" mode). We sacrifice performance and potential productivity while running 32-bit software instead of 64-bit software in the bargain for "compatibility".

Further:  twice as much memory is required to store the same number of 32-bit data strings than is required to store the same data processed as 64-bit strings. If memory serves, 32-bit data is prefixed with 32 zero bits to make it the same size as the 64-bit

Third, the historical fact is that 64-bit hardware has been used only for "desktop" computers until recently. The long interim before the development of 64-bit portable devices is because electrical power consumption for 64-bit hardware made it infeasible for "laptop" computers and the subsequent "tablet" devices to have 64-bit CPUs and chipsets. The higher power consumption also produced significantly more heat, which made a "laptop" too hot to use while it was positioned on either a lap or a desktop.

So, all laptops of which I've been apprised are still 32-bit systems, regardless of whether current batteries can support 64-bit systems (albeit, for less time before recharging is required). Nonetheless, there are some 64-bit CPUs that have a relatively lower power consumption and produce less heat, but also run at a lower clock speed (for example, 2.5 GHz instead of 3 GHz).

Originally (in that historical context), Microsoft began developing Windows 7 as a fully 64-bit system, as Intel and AMD also began developing their respective 64-bit CPUs and chipsets. But when it became evident that 32-bit laptops were becoming the most commonly-used (most bought) devices, Microsoft persuaded Intel and AMD to design their 64-bit hardware with a native "32-bit compatibility mode". With it, "64-bit Windows" can effectively run both 32-bit and 64-bit software on the current Intel/AMD 64-bit architecture, but only 64-bit software has the data-processing performance advantage.

Of course, Microsoft continued to develop "32-bit Windows" for 32-bit hardware, which runs only 32-bit software. Unfortunately, the past preference for 32-bit portable devices relative to 64-bit computers has also discouraged 64-bit applications software development. Nonetheless, 64-bit software has been developed for applications that require the substantial data-processing throughput which 64-bit systems can provide (regardless of the OS).

The bottom line is that running 32-bit software on a 64-bit system is essentially foregoing the benefits and advantages of the hardware architecture. Period. It's like using a Maserati that has a governor installed to limit its speed to a maximum of 55 MPH so that you can use it for a school bus.  You will get there eventually, but the ride is longer.

That said, Microsoft has never made any commitment to a fully 64-bit operating system. They do not want to invest the resources (money) that would be required to revise the  Windows OS to execute on a 64-bit system that doesn't have a "32-bit compatibility" mode. Albeit, there are 64-bit systems that do not have the 32-bit "compatibility mode" of the Intel/AMD architecture on which 64-bit Windows runs. For those systems, Windows is not, of course, the OS.

What you want can't be done because it recognizes the names as alpha-numeric characters and sorts alphabetically (just like it does for the years - but there it works because they are numbers).  Try using a number designation instead of the alphabetic name of the month - 8-2006 for August 2006 and so forth.

If it is possible (and I don't think so), the answer may be in one of these articles (especially the one on sort order) - but most likely you will learn from them that it isn't possible as you have things set up:

Here's how to change the default template for a folder: http://www.vistax64.com/tutorials/70819-windows-explorer-folder-view-settings.html.  Default behaviour is to select the template based on the first type of file placed in the folder (it has to choose something).  Vista sets the Explorer window view settings and folder type template according to what folder types are in the window. This will show you how to reset the Windows Explorer folder view settings, turn off Automatic Folder Type Discovery (which is I think what you want), and increase the folder view cache to have Vista remember the view settings, folder type template, size, or position of a specific window for when it was last closed at the specific path location.
