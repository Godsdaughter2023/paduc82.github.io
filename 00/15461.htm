Try this VB code but note there's nothing to stop anyone unhiding rows/columns.

 

Sub Hide_Me()
Dim LastCol As Long, x As Long
LastCol = ActiveSheet.Cells(1, Columns.Count).End(xlToLeft).Column
Rows(1).Hidden = True
For x = 1 To LastCol
    If UCase(Cells(1, x).Value) = "HIDE" Then
        Columns(x).Hidden = True
    End If
Next
End Sub

The file is really intended to demonstrated correlated subforms, but it happens to be based on a student registrations and attendances database.  Take a look at the relationship window which gives the basic model for what you are doing.  The Students table equates to yours of course, and my Courses table would equate to a Classes table, with one row for Mrs Johnson's class, one for Mr Timms' etc.  The CourseRegistrations table equates to your ClassRoster table, and would have columns StudentID and ClassID.  The CourseAttendaces table is the attendance log you are aiming at.  You'll need an equivalent of the CourseSessions table, which in your case will have one row per class per day, so would have columns ClassID and ClassDate.

Your ParentGuardians table would be related to the Students table of course, but this should be via another table with columns StudentID and ParentGuardianID as the relationship is many-to-many (a student can, and usually will, have more than one parent/guardian, and each parent/guardian may have more than one child at the school).

The Instructors table will be related to the Classes table, either directly, or via another table with columns ClassID and InstuctorID if a class can have more than one instructor.

ken sheridan say: I think the model in my demo would fit your scenario with a few small changes.  What you are calling a class is modelled by the CourseRegistrations table in my file, i.e. it models the subsets of students registered for each course, which in modelling terms is a relationship between courses and students; in your case the subset of students registered for a teacher for a period of time.  So the current class of Mrs Johnson is the equivalent of a row in the Courses table in my file,  which would be the Instructors table in your case.

For each academic session for which a student is a member of a class, be it an academic term or a year, you would need a table, ClassPeriods say, between the CourseTable and CourseSessions table in my model (Instructors and ClassSessions in yours).  This would model the period being employed for registering each set of students to a class, so if the academic year is the period being employed, this table would for instance have a row for Mrs Johnson's class of 2011/12, and similarly for the other teachers.  Once the next academic year starts rows would be inserted for each teacher for 2012/13.

In your case a table equivalent to my CourseSessions table, ClassSessions say, would be needed to model each daily class session.  This would reference the additional ClassPeriods table to which I referred in the last paragraph.

Each student in Mrs Johnson's class a would be represented by a row in the equivalent of my CourseRegistrations table, ClassRegistrations say, and would have an InstuctorID column with the same value in each row referencing the primary key value for Mrs Johnson's row in the Instructors table, and a StudentID column referencing the primary key of the Students table.  A difference in your case would be that it would also have a column referencing the ClassPeriod.  So each student in Mrs Johnson's class for 2011/12 would be represented by a row with their StudentID in one column, Mrs Johnson's InstuctorID in another column, and 2011/12 in a third column.

When it comes to the daily class register this is modelled by the equivalent of my CourseAttendances table and would reference the ClassRegistrations  table on one side, and the ClassSessions on the other.  In my file a student's attendance at a course session is recorded by a row in the table, non-attendance by the absence of a row.  You could, however, have each relevant student recorded for each day, and a Boolean (Yes/No) Attended column, or a column with a set of possible values such as Attended, Authorised absence, Unauthorised absence etc.  If doing this the rows can be automatically inserted each day  with a little bit of code, ready for the Attendance column's values to be assigned.  There's a file Treatment.zip in my SkyDrive folder which illustrates this sort of thing using a simple medical procedures database, by means of a Create Treatment Programme button on the Patient Treatments form.  This inserts rows over a period of time, but the methodology would be near identical, and I could easily walk you through it when the time comes.  But concentrate on the model first.  Getting that right is the key to success.

jeeped say:

This behaviour can occur in a couple of ways.

First, if this only occurs within a single document, the Office button, Prepare menu item has a Mark as Final option which marks the current document as Read Only. If this is the case, a small icon to the right of the word count in the status bar will produce "Marked as Final" when hovered over. Solution: return to the same menu command and unmark as final.

Second, you could have a trial version that has expired its trial period. Solution: purchase a license, get a new trial of Office 2010 or switch to OpenOffice.org.

Third, your paid-for version of Office 2007 wasn't activated. Solution: activate it! (see Installation and activation information for 2007 Microsoft Office programs)

And last, the document has some sort of document protection applied to it. Access the document security commands from the Developer or Review tab's Protect group. (see You cannot open a Word document that uses Editing Restrictions in an earlier version of Word)
