ken sheridan say: Over the years I've regularly seen posts reporting problems opening a PDF file via a hyperlink.  A more reliable approach appears to be to store the path in a column of text data type and open it in its associated application by calling the Windows API ShellExecute function by pasting the following module into the database:

' basShellExecute
Option Compare Database
Option Explicit

Public Const SW_HIDE = 0
Public Const SW_SHOWNORMAL = 1
Public Const SW_SHOWMINIMIZED = 2
Public Const SW_SHOWMAXIMIZED = 3
Public Const OP_OPEN = "Open"
Public Const OP_PRINT = "Print"


Declare Function ShellExecute& Lib "shell32.dll" Alias "ShellExecuteA" (ByVal _
hwnd As Long, ByVal lpOperation As String, ByVal lpFile As String, ByVal _
lpParameters As String, ByVal lpDirectory As String, ByVal nshowcm As Long)

Sub ShellToFile(strPath As String, _
            Optional strOperation As String = OP_OPEN, _
            Optional lngShow As Long = SW_SHOWNORMAL)

    Dim lngRetVal As Long
    Dim lngHwnd As Long
    
    lngHwnd = Application.hWndAccessApp
    
    lngRetVal = ShellExecute(lngHwnd, strOperation, strPath, _
        vbNullString, CurDir, lngShow)
        
    If lngRetVal < 32 Then
        MsgBox "Unable to open file " & strPath, vbInformation, "Warning"
    End If
    
End Sub
' module ends

To open the file simply call it with:

ShellToFile [YourControlName]

This must be done in a form of course, but that's no problem as the user should never interact with data in a table's raw datasheet view.  If you wish you can partially simulate a hyperlink by formatting the control appropriately and using its Click event procedure to call the function.

ken sheridan say: You can do this if the four radio buttons are within an option group, e.g.

SELECT *
FROM YourTable
WHERE CoachID = Forms!YourForm!YourComboBox
AND ((1Cyle = TRUE AND Forms!YourForm!YourOptionGroup = 1)
    OR (2Cyle = TRUE AND Forms!YourForm!YourOptionGroup = 2)
    OR (3Cyle = TRUE AND Forms!YourForm!YourOptionGroup = 3)
    OR (4Cyle = TRUE AND Forms!YourForm!YourOptionGroup = 4));

However, as Duane has pointed out, having a Boolean (Yes/No) column for each cycle is poor relational database design as it encodes data as column headings.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

The correct design would be to model the many-to-many relationship type between clients and cycles by a separate table like this:

ClientCycles
….ClientID  (FK)
….CycleID  (FK)

where CycleID references the primary key of a Cycles table.  The primary key of the above table is a composite of the two foreign keys.  The table can have other non-key columns representing any attributes of the relationship type.

It is possible to automate the recasting of the data into a correct model, and you'll find an example in UnencodeColumns.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file includes examples where the columns are of Boolean data type, and where there are quantitative or scalar values in the columns.  The former applies in your case.
