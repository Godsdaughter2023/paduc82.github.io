For all those experiencing the same or similar issue:

The windows credential manager has been updated in Windows 10 such that it will only work with Samba 4.2 & above.  If you are still working with an older Samba version, as it was in my case, any program, such as Outlook, that utilizes windows to store credentials will not work properly.  Even if you are not using Samba to control the windows domain, I would still try the workaround below:

 

The workaround is to add the following registry key to revert the credential manager to the old method:

 

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Cryptography\Protect\Providers\df9d8cd0-1501-11d1-8c7a-00c04fc297eb]

"ProtectionPolicy"=dword:00000001

rich michaels - mvp (greatcirclelearning.com) say:

AutoRecovery is not the same as AutoSave.

There is not an equivalent AutoSave feature for documents stored on local drives. The AutoSave feature is only for files stored on OneDrive or SharePoint.

A timed based save function that produces the equivalent of a Command + S save (Ctrl + S on Windows) active document save does not exist. The only feature that does exist is the "Always Create a Backup Copy" option, which keeps a single backup of the file previous to the currently edited file. However, it's operation is not timed based either, it functions solely on a user action of clicking Save from the Ribbon or a Command + S shortcut key sequence.

And then there's my free Got-ur-Back add-in, but it also is not a timed based backup. It requires a manual user initiated Save action being performed.

One final option on a Mac is Time Machine ... it is time based but you probably don't want to set it for less than 1-hour because it will devour too much system resources.
