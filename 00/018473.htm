brian tillman [outlook mvp 2007-2019] say:

554 is an error from the SMTP server telling you that it will not accept the message. This can be due to multiple reasons. One is because your mail provider has the sending domain in a blacklist. One is because you're not properly authenticating to the outgoing server. One us because the server thinks the domain contains a reserved word. One is because the SMTP server's spam filter sees something it doesn't like. One is because the IP address of the sending system and the domain of the sending system don't match (reverse DNS doesn't report the correct domain name).

Of all of these, I think the failure to authenticate to the outgoing server is the most likely and also the one you most readily can fix. Click File>Account Settings>Account Settings, double-click the account, click More Settings, then select the Outgoing Server tab. Make sure the "My outgoing server requires authentication" is selected. Usually, "Use the same credentials as the incoming server" is the correct value as well, but not always. You can use different mail providers for incoming and outgoing, in which case the SMTP server credentials will be different from the IMAP or POP server credentials.

Microsoft Access 2010 Importing Tables with Date as Column Heading - Microsoft Community

sflath say:

Microsoft Access 2010 Importing Tables with Date as Column Heading

I am trying to import a table from Excel into Access that has dates as column headings.  The table contains sales figures and has locations in the rows, with the month and year as column headings.  When I import the table into Access, it converts the dates into numbers for the column headings (e.g. February 2011 --> 40575).  I want to be able to have Access calculate rolling 12-month sales totals and averages, and I would like to preserve the date format in the Access field names.  Is this possible?

Thanks in advance.

ken sheridan say: Date/time values are implemented as a 64 bit floating point number in which the integer part represents the days and the fractional part the time of day.  Zero represents 30 December 1899 00:00:00.  The reason you are getting 40575 for February 2011, is that this number represents 1 February 2011, the first of the month being assumed when only the month and year are cited.

You can probably do it by linking to the Excel worksheet and appending the data into a predefined local table in Access, but you'll end up with something which is not really a valid relational database table as it 'encodes data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

However, you can decompose this table into two correctly normalized tables such as:

Locations
....LocationID (PK)
....Location

Sales
....LocationID (FK)
....SaleMonth
....SaleAmount

The primary key of this table is a composite one of LocationID and SaleMonth.  The latter should be of text data type.

You'll find an example of how to to insert values into a correctly normalixed table as UnecodeColumns.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

It should then be a relatively simple task to compute the sales totals and averages.  You can call the CDate function, passing the SaleMonth value into it as its argument, to return a true date/time value from the string expression for sorting and grouping purposes.
