ajeesh_nair say:

hi,

 

As we look deeper into the differences between XLS and XLSX, we can see that the way the information is stored is vastly different for both XLS and XLSX formats. XLS is based on BIFF (Binary Interchange File Format) and as such, the information is directly stored to a binary format. On the other hand, XLSX is based on the Office Open xml format, a file format that was derived from XML. The information in an XLSX file is stored in a text file that uses XML to define all its parameters.

 

As XLSX is stored in a text file format, Microsoft decided to remove macro support for this file format. Instead they assigned a totally different file extension that allows the use of macros; it is named XLSM. The older XLS file extension does not have this issue and it is able to hold spreadsheets that contain macros or not.

 

Going forward, if someone is using lower verson of excel, even, they can open xls file without any issues. If you don't want the document properties to show in VBE, just right click on the properties window, then click on 'Dockable'.

Before we go there: what's up with these UDF1 - UDF6 fields? That sounds very much like a Repeating Group which is a big no-no in relational database design. And one of the reasons of such complicated expressions.

If you're up for it, let's work on your database design first.

ken sheridan say: Rather than use an expression of Byzantine complexity like this I'd be inclined to break down the logic into IF....End If constructs in a function into which the values of the controls are passed.  But, as Tom points out, the real problem here is more likely to be one of bad design.  Having multiple UDF columns is almost certainly what is known in the language of the database relational model as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

A correct design is likely to mean decomposing the table into two related tables as follows:

Foos
....FooID  (PK)
....Foo
....etc

FooUDFs
....UDF
....FooID  (FK)

Each UDF value per Foo is stored as an explicit value in a separate row in the UDF column in the FooUDFs table.  The primary key is a composite one of both columns unless there can be more than one row per Foo with the same UDF value, in which case another column would be required as part of the key.   This additional column could simply be a serial number from 1 upwards per Foo, which would in fact be storing the data incorrectly encoded in the column headings in your current table.  Other non-key columns can be added to the table if necessary of course if there are other attributes of the relationship between Foos and UDFs.

If the UDF values are a fixed set then a further table is required to ensure the of the data:

UDFs
....UDF  (PK)

The UDF column in FooUDFs references the primary key of the UDFs.  FooUDFs is modelling the many-to-many relationship type between Foos and UDFs.  By enforcing referential integrity in the two relationships between the three tables only valid values can be entered into FooUDFs.

The interface for data entry would be a form in single form view within which would be a continuous subform for entering rows into FooUDFs.  If there is a constraint which requires some Foos to have precisely 5 related UDFs and others 6 then this can be enforced via the interface.  You'll find an example as Families.zip in my public databases folder at:

joeu2004 say:

Claverie wrote:

This formula in the formula bar should result in zero. However, when using IFERROR, it give a result different than zero. As soon as I remove the IFERROR, the correct result is obtained (zero). What can the problem be?

The problem is a flaw in Excel, IMHO.  It arises because in some contexts, Excel arbitrarily replaces the actual binary arithmetic result with exactly zero.

When J9 is 15514:

=J9 - J9/100*100      results in exactly zero (0.00E+00 when formatted as Scientific)

=(J9 - J9/100*100)    results in 1.82E-12, which is the actual binary arithmetic result

Of course, the superfluous outer parentheses should not make any difference. They do only because the context no longer fits the very limited conditions under which Excel decides that the result of the last subtraction of a formula is "close enough" to zero that it replaces 1.82E-12 with exactly zero.

Likewise, IF(J9 - J9/100*100, 0) results in 1.82E-12:  it does not fit the conditions under which Excel chooses to make the arbitrary substitution.

BTW, =J9 - J9/100*100 - 0 also returns 1.82E-12.  Again, there is no logical reason why the superfluous -0 should make a difference.  It just demonstrates the capriciousness of the Excel decision.

-----

You might wonder why (J9 - J9/100*100) is not exact zero, in the first place.

The reason is:  the internal binary representation of J9/100*100 is not the same as the internal binary representation of J9 (exactly 15514).

Internally, J9/100 is exactly 155.139999999999,9863575794734060764312744140625, which is the exact internal representation of 155.14.  Excel uses 64-bit binary floating-point to represent numbers.  That binary form cannot represent most decimal fractions exactly.

(I use period for the decimal point and comma to demarcate the first 15 significant decimal digits, which is all that Excel formats (rounded).)

By coincidence, 155.14*100 is 15513.9999999999,98181010596454143524169921875, not 15514.

-----

In contrast, the following VBA function does return exactly 15514 when used as =myarith(J9, 100):

Function myarith(x As Double, y As Double) As Double
myarith = x / y * y
End Function

The reason is:  Intel-compatible CPUs use an 80-bit binary floating-point form to perform arithmetic with 64-bit binary floating-point operands.

In some contexts, VBA tries to use the 80-bit binary floating-point results.  And in this example, the extra precision of x/y allows x/y*y to be exactly x.  But that is only a coincidence.

(The extra precision does not always have better results.)

In contrast, Excel rounds the result of each 80-bit binary floating-point operation to 64-bit binary floating-point.  In this example, that loses sufficient precision to make a difference.

-----

In a nutshell, whenever you want a calculation with decimal fractions to be accurate to some number of decimal places, explicitly round to that number of decimal places.  Do not round to an arbitrary number of decimal places (e.g. 10), as some people suggest.

Only you can know the "right" number of decimal places to round to and when to round.  Again, it depends on the degree of accuracy that

For your example, I would round the final calculation to the accuracy of J9.  In this case:

=ROUND(J9 - J9/100*100, 0)

and

=IFERROR(ROUND(J9 - J9/100*100, 0), 0)
