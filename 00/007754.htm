ken sheridan say: If the Time column is a text data type then try this:

(([Forms]![Main Menu]![Day of Week]="Monday" And [Monday]=True) And [Time] >= "7:00") Or  ([Forms]![Main Menu]![Day of Week]="Monday" And [Tuesday]=True And [Time] <= "6:59")) And [Initials] Is Null

If it's a date/time data type then:

(([Forms]![Main Menu]![Day of Week]="Monday" And [Monday]=True) And [Time] >= #7:00#)
Or ([Forms]![Main Menu]![Day of Week]="Monday" And [Tuesday]=True And [Time] <= #6:59#)) And [Initials] Is Null

More usually with this sort of query parameters would be used for entering the times at runtime of course, rather than hard coding the values in the query. In fact you'd only need the one parameter, the start time, as the second time can be computed from the first with, if a text data type:

Format(DateAdd("n",-1, CDate([Enter start time])),"h:nn")

or if date time:

DateAdd("n",-1, [Enter start time])

Note that if the column is a date time data type the parameter should be declared as DateTime in the query.

A problem you'll encounter if the times are text data type is that a time like '7:00' ( 7.00 AM) will sort after a time like '14:00' (2:00 PM), so the hour should be entered as two digits, e.g. '07:00'.  I'd recommend using a date/time data type.  As this is implemented under the skin as a 64 bit floating point number, #7:00#  will sort correctly before #14:00#.

More fundamentally however, your table is not well designed.  By having multiple Boolean (Yes/No) columns for the days of the week you are doing what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  A correct design would involve decomposing the table into three related tables, one being your current table but without the days of the week columns, one being a DaysOfWeek table with 7 rows, and the third, colloquially known as a 'junction' gable, being a table which models the many-to-many relationship type between them by having a foreign key columns referencing the primary key of your current table and a DayOfWeek column referencing the primary key of the DaysOfWeek table.  Diagrammatically the model would therefore look like this:

CurrentTable----<JunctionTable>----DaysOfWeek

I'd also strongly recommend that you don't use Time as a column name.  It's the name of a built in function in Access, so as a 'reserved' word should be avoided as an object name.

ken sheridan say: Small or large, it's as well to get the model right.

The time is an attribute of the relationship type between procedures and days of the week, so the model would be pretty much as I described, with a column for the time of day in the junction table.  So the tables would be like this:

Procedures
....ProcedureID (PK)
....Procedure

ProcedureTimes
....ProcedureID
....DayID
....TimeOfDay

DaysOfWeek
....DayID (PK)
....DayName

The ProcedureID primary key column of Procedures can be an autonumber for convenience, as could DayID, but it would make more sense to make the latter a straightforward integer number data type with values 1 to 7.  The ProcedureTimes table is 'all key', i.e. its primary key is a composite one made up of all three columns.

If the times of day are a logical set rather than arbitrarily variable, then the relationship type is ternary (3-way), so a TimesOfDay table should also be introduced.  By enforcing the relationships between ProcedureTimes and each of the three referenced tables

kerwinyang_msft7476 say:

HI Jon,

In regard to your questions:
1. What exactly is a hit?
A:
Open document in Office Web App 3 times will count as hit (per one or multiple users) for that day, counter increment as following:

[VIEW AND EDIT both considered as separate hit]

not opened: count is 0
open 1st time: count is 0
open 2nd time: count is 0
open 3rd times: count is 3
open 4th times: count is 4
and so on

Open document in office client 3 times will count as 1 view (per one or multiple users) in a day, counter increment as following:
[VIEW AND EDIT both considered as separate hit]

View 1st time (not edited): count is 1
View 2nd time (not edited): count is 2
and so on

2. Does it mean someone has open just previewed the file or actually opened it online in their browser?
A: As I have mentioned above, for Office Web App and Office client, both view and edit can be considered as a hit. Kindly notice, for Office client, to make sure view can be considered as a hit, the Office client version should be 2013 or higher.

3. If they download a file does this constitute a separate hit?
A: No, the downloading file action won’t be considered as a separate hit.

4. In delve I can see how many time my document has been viewed. Is this the same as a hit in the usage report?
A: About the fourth question, I need some more time to consult the senior engineer and do some tests to confirm if the timer jobs for Delve view and popularity trend are the same. Once there’s any update, I will post back. Thanks for your time and patience.

After an account becomes inactive, all messages, folders, and contacts are deleted.
Incoming messages will be sent back to the sender as undeliverable.
Your account name is still reserved.

If you don't use your Windows Live ID for 365 days, your Windows Live ID will be permanently deleted. 
Once the account is marked as in-active all data is purged so even if you could get the email back the data would be gone.
