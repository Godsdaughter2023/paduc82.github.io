ken sheridan say: Unless your are interfacing your Access data with SharePoint I'd strongly recommend against the use of multi-valued fields.  They are completely foreign to one of the most basic principles of the database relational model, which is that only one value of each attribute should be stored in each row in a table.

A far better way is the conventional one of modelling a many-to-many relationship by means of a table, sometimes called a 'junction' table.  In the example you cite of employees and issues you would have an Employees table and an Issues table, with primary keys EmployeeID and IssueID say.  The relationship between the two tables would be modelled with an EmployeeIssues table with two foreign keys columns EmployeeID and IssueID referncing the primary keys of the other two tables respectively. Diagamatically the model would look like this:

Employees---<EmployeeIssues>----Issues

As you see, the model is symmetrical, so it’s a simple task to return all employees per issue or all issues per employee by joining the three tables in a query, e.g.

SELECT FirstName, LastName, Issue
FROM (Employees INNER JOIN EmployeeIssues
ON Employees.EmployeeID = EmployeeIssues.EmployeeID)
INNER JOIN Issues ON EmployeeIssues.IssueID = Issues.IssueID
ORDER BY LastName, FirstName, Issue;

This would list all employees in name order, with each issue to which they are assigned.  Conversely to list all issues with the employees assigned to them:

SELECT Issue, FirstName, LastName
FROM (Employees INNER JOIN EmployeeIssues
ON Employees.EmployeeID = EmployeeIssues.EmployeeID)
INNER JOIN Issues ON EmployeeIssues.IssueID = Issues.IssueID
ORDER BY Issue, LastName,FirstName;

In either case you can include a parameter to return the issues for a specific employee, e.g.

SELECT FirstName, LastName, Issue
FROM (Employees INNER JOIN EmployeeIssues
ON Employees.EmployeeID = EmployeeIssues.EmployeeID)
INNER JOIN Issues ON EmployeeIssues.IssueID = Issues.IssueID
WHERE LastName  = [Enter employee's last name:]
ORDER BY Issue;

Or to return the employees for a specific issue, e.g.

SELECT Issue, FirstName, LastName
FROM (Employees INNER JOIN EmployeeIssues
ON Employees.EmployeeID = EmployeeIssues.EmployeeID)
INNER JOIN Issues ON EmployeeIssues.IssueID = Issues.IssueID
WHERE Issue = [Enter issue:]
ORDER BY LastName, FirstName;

In a developed application you'd be more likely to use a dialogue form with a combo box to select an issue or employee, and open a report or form of the results.  If using a query as the RecordSource of a report, however, don't use an ORDER BY clause, but instead use the report's internal sorting and grouping mechanism.
Ken Sheridan, Stafford, England

Can you do a find and replace to insert a page break in Excel 2007 - Microsoft Community

uvr say:

Can you do a find and replace to insert a page break in Excel 2007

I have a spreadsheet that is over 30,000 lines long.  There is a blank between each job and each one needs to print on a separate page.  So I'm hoping for an automated way to add the page breaks.
jeeped say:
This macro will quickly place a manual page break between the blank cells found in column A and the data in the next row,

Press Alt+F11 to open the Visual Basic Editor and immediately use the pull-down menus to Insert, Module. Paste the following into the new pane titled something like Book1 - Module1 (Code).

Sub mcrPageBreakAtBlank()
    ActiveSheet.ResetAllPageBreaks
    Range("A1").Select
    Application.ScreenUpdating = False
    Selection.End(xlDown).Offset(2, 0).Select
    Do While ActiveCell.Row < ActiveCell.SpecialCells(xlLastCell).Row
        ActiveWindow.SelectedSheets.HPageBreaks.Add Before:=ActiveCell
        Selection.End(xlDown).Offset(2, 0).Select
    Loop
    Range("A1").Select
    Application.ScreenUpdating = True
End Sub

Press Alt+Q to close the VBE and return to your worksheet. From the ribbon's Developer tab, click Macros at the left end and Run mcrPageBreaksAtBlanks().

This macro assumes that any blank cell in column A is a marker to apply a page break.
