john korchok say:

Doug Robbins has the idea:

  1. Create a new style that will be applied to the hidden year. The font color can be set to white, to be invisible against the page background, or the font can be hidden, which means it doesn't take up a line space in your final document.
  2. Enter the date for the chapter, then apply the hidden year style. To keep track of it, it would make sense to place this just below the chapter title.
  3. Open the header and click on the position where the year will be entered, then choose Insert>Field... from the OS X Word menu at the top of your screen.
  4. In the Categories list, choose Links and References.
  5. In the Field names list, choose StyleRef.
  6. Click on the Options button, then on the Styles pane.
  7. Select the style you created earlier, then click on OK.

Now when you type a year, then apply the hidden year style, the year will automatically appear in the header where you placed the StyleRef field.

john w. vinson say:

Try using the string parsing functions:

 

Mid([Group_Prices], InStrRev([Group_Prices], "=") + 1, InStrRev([Group_Prices], "|") - 1)

 

InStr finds a substring in a string; InStrRev does the same but starts at the end working back, rather than at the start. So this will find the LAST occurance of = in the field, and the last occurance of |, and return the characters in between.

 

You may have some real trouble if there are NULL values or ill-formatted ones.

 

This is a really good example of why fields should be "atomic"; in your final database you really should have three fields for these three values, and you should feel free to whack whoever put this together originally with a wet noodle!

ken sheridan say: The following function should do it for both quantity and price where both are present:

Public Function GetListItems(strItemList As String, _
                    strDelimiter As String, _
                    intStart As Integer, _
                    intNumber As Integer) As String

    Dim intDelimLength As Integer
    Dim intSubStart As Integer
    Dim intSubEnd As Integer
    Dim n As Integer
    
    strItemList = strItemList & strDelimiter
    intDelimLength = Len(strDelimiter)
    intSubStart = 1
    
    ' get start of substring
    If intStart > 1 Then
        For n = 1 To intStart - 1
            intSubStart = InStr(intSubStart, strItemList, strDelimiter) + Len(strDelimiter)
            If intSubStart > Len(strItemList) Then Exit For
        Next n
    End If
    
    intSubEnd = intSubStart
    
    ' get position immediately after end of substring
    For n = 1 To intNumber
        If InStr(intSubEnd + 1, strItemList, strDelimiter) > 0 Then
            intSubEnd = InStr(intSubEnd + 1, strItemList, strDelimiter)
        End If
    Next n
    
    ' return value is substring starting at intSubStart
    ' and of length intSubEnd - intSubStart
    GetListItems = Mid(strItemList, intSubStart, intSubEnd - intSubStart)

End Function

Delete on premise user, but keep O365 mailbox active

Then we recovered it and run a full sync with dirsync and then the user mailbox was not getting deleted anymore.

Now we work with AAD sync and this tricks does not seem to work. Is their anyway that we can delete the on premise user, and keep the O365 user active? We would like to create shared mailboxes from the users so that we can archive their mail.

vasil l. michev say:

You can delete the O365 user and recover it from the Recycle bin, this will provision it as disconnector and any subsequent dirsync changes should not affect it. As there is some risk involved, make sure to backup or put the mailbox on hold first, and under no circumstances remove the user from the Recycle bin - it will hard delete the mailbox as well.

A more safer, but slower approach is to disable dirsync, clear out the ImmutableID attribute of the O365 and make sure that no soft-match will happen once you turn Dirsync back on (i.e. that is make sure that the PrimarySMTPAddress and the WindowsEmailAddress of the on-prem object are different than the O365 one). Or if you prefer, just set up a random ImmutableId that will never match an on-prem object.

None of those methods are supported, so use at your own risk :)
