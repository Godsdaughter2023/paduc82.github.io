ken sheridan say: A multi-valued field is a simple way to model a many-to-many relationship type, but, as you've found, can cause problems.  I'd recommend that you use the conventional way of modelling a many-to-many relationship type by a table which resolves it into two one-to-many relationship types, sometimes colloquially called a 'junction table'.  Instead of a control bound to the multi-valued field, you would then use a continuous subform based on the table which models the relationship type, with each value being entered as a separate row in the subform, for which a bar code scanner could be used as the input device.

If you have data in an existing multi-valued field you can automatically transfer than data into a predefined conventional model.  You'll find an example in MVFCorrector.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates how to transfer data to a conventional model, both where the MVF references another table, and where the available multiple values are stored as a value list as part of the object definition.

To use the barcode scanner for filtering a form on the basis of a value in a subform the FindRecord.zip file in the same OneDrive folder includes two options which illustrate how to filter a form on a subform value by means of an unbound combo box in the parent form.  One option filters the parent form only, the other filters both the parent form and subform.  A bar code scanner could be used as the input device for entering a value in the unbound combo box.
