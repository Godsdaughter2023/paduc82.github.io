The reason I was going with this ADO method, is because I am using this Access database as a front end to a SQL 2014 server database. My Forms are very slow/sluggish, so from what I've read, it seemed ADO might be the best approach for this situation. your thoughts? -ME

albert d. kallal say:

Using ADO, or writing some custom code in c++ is NOT going to increase performance, nor make a form load faster.

 

So SQL server does on all of a sudden “decide” to perform faster when YOU write some custom code in c++, or simply using Access and assigning the SQL as above.

 

SQL server does not “wake up” and make some decision as to how fast it going to retrieve some data. So using c++, vb.net, ADO, or DAO or a Access form to pull some data WILL ALL perform the same, and they ALL ARE using the same server, and same network – thus the speed of data transfer going to be the same.

 

If that system in question is slow, then likely your network speed is limited, or your designs are pulling too much data.

 

So always try to restrict the data pulled into a form.

 

If a form is bound to a table with 1 million rows, then RESTRICT the records by using the “where” clause of the forms open command.

 

When you do the above, then ONLY the ONE record is pulled from that table and only the one record travels down the network pipe.

 

Eg this:

 

Docmd.OpenForm "frmInvoice",,,"[InvoiceNumber] = 12345"

 

 

So using custom ADO code, or the one line above with a bound form will perform the same. You gain nothing by writing a whole bunch of code and using ADO and then assigning that ADO to the form.

 

So simply ask user what they need before you load a form! 

 

The above is so simple, but so often I see the above concept ignored. For example, when you walk up to a instant teller machine, does it download every account number and THEN ASK YOU what you want to do? Or when you seach the web, is everything download into your browser and you THEN search that huge result?

In access, you can well open up a form attached (bound) to a table, but FIRST ask what they want! Thus for a customer invoice, get the invoice number, and then load up the form with the ONE record (how can one record be slow?).

It is a "often" quoted myth that Access always pull down all records from the server. Access ONLY does what you tell it to do. So a form bound to a large table is only bad when you launch that form without telling Access what to load into that form.

When done editing the record...the form is closed, and you are back to the prompt ready to do battle with the next customer.

 

This type of UI and search is outlined here:

 

http://www.kallal.ca/Search/index.html

scottgem say:
Sum(IIF(IsNull([CustomerPrice]),0,,CustomerPrice)+IIF(IsNull(ShippingPerUnit),0,ShippingPerUnit)
AS SalePrice.

Note, I assume CustomerPrice and ShippingPerUnit are numeric datatypes, so your NZ should be 0 not "0".

qa guy commelec say:
This is the query in question 
Sum(Nz([tblOrderDetails].[CustomerPrice], "0")+Nz([tblOrderDetails].[ShippingPerUnit], "0")) AS SalePrice
It should not have quotes around the 0 (zero) like this --
Sum(Nz([tblOrderDetails].[CustomerPrice], 0)+Nz([tblOrderDetails].[ShippingPerUnit], 0)) AS SalePrice

Quotes makes it a text string and using the plus sign ( + ) will concatenate instead of add the two parts.

This is the IIF --
Sum(IIf([tblOrderDetails].[CustomerPrice] Is Null, 0, [tblOrderDetails].[CustomerPrice])+IIf([tblOrderDetails].[ShippingPerUnit] Is Null 0, ([tblOrderDetails].[ShippingPerUnit])) AS SalePrice

zenmasta say: I didn't know that adding quotes would convert to string but it seems access was smart enough to allow it to work previously.

Thanks, I changed it below per suggestions and I am able to use this query in my excel sheet now :)
Sum(Iif([tblOrderDetails].[CustomerPrice] Is Null, 0, [tblOrderDetails].[CustomerPrice]) + Iif([tblOrderDetails].[ShippingPerUnit] Is Null, 0, [tblOrderDetails].[ShippingPerUnit])) AS SalePrice

mike middleton say:

One way: Since your formula in column C refers to adjacent column B (and A), if the formula is in C1, double-click the fill handle (Excel's name for "the + symbol in the lower right hand corner of the cell").

Another way: Copy cell C1 (select C1, right-click and choose Copy). Select cell C2. While you hold down the Shift key, press End and press the down arrow (thus selecting C2:C1048576). Paste (right-click and choose Paste).

Excel 2007 and 2010 have 1,048,576 rows, not "1.6 million rows."
