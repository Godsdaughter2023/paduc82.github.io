Hi Wayne,

Thank you for posting your question in the Microsoft Community.

Windows Mail App in Windows 10 does not have an archive & backup function. Luckily all messages are stored locally in a Mail folder located deep in the hidden AppData folder.

Messages are stored as EML files. To read them outside the Windows Mail App you need a mail client capable of handling EML files (Microsoft Outlook, Windows Live Mail).

Note: Please be extremely careful when browsing and handling these folders as your Mail Application might stop working if the folders are removed, renamed or moved. Also do not delete individual email messages in File Explorer, use Mail App's Delete function for that.

If you go to “C:\Users\<username>\AppData\Local\Packages\”, open the folder that starts with "microsoft.windowscommunicationsapps", open the folders “\LocalState\Indexed\LiveComm” inside that, then select the folder with the email you want to look at, then one folder down you'll see two folders, "Mail" and "People". Your is the only report I've noticed of a crash related to customizing menus and toolbars, so I think this is a problem unique to your particular install.

Since I don't have any previous reports to rely on, the suggestions I have will be very general.

Not installing updates is the #1 reason for crashes, but this does not apply to you. Your OS and Office install are right up-to-date, which is a good thing.

The #2 reason for crashes is font issues, so let's go there next. Restart your Mac in Safe Mode by holding the Shift key down when you hear the startup chime. Try customizing in Safe mode. Is the crash gone?

Then do a regular restart without holding the shift key. Once you get to the desktop, empty the trash to get rid of the unneeded fonts and free up disc space. Now try customizing again. Is the crash gone?

Excellent idea Don but unfortunately when I tested it appears that there is an error. Should only subtract rows from the counta that will actually have data in them.

Because the JG-Chicago is starting at row 6 with the headers suggests that might have other cells above row 6 containing data. However, the unknown is how many have data and is it likely to change in the future?

I suggest that Don's named range formula be modified to the following and it will definitely start at row 7 and end at the last row with data by counting the rows with data from a range of B7 to the max rows in the worksheet. This will alleviate the need to know how many cells above B7 have data in them.

=OFFSET(Sheet1!$B$7,0,0,COUNTA(Sheet1!$B$7:Sheet1!$B$65536),1)

 

Note: if the worksheet grows and is migrated to a later version of xl and there is any likelyhood of max rows of data being greater than 65536 then need to edit the formula to the new max rows of the worksheet.

Tip when editing defined name formulas: Click in the field with the formula and then press F2 to go into edit mode otherwise you will have problems using arrows to move backward and forwards within the formula.

ossiemac say:

You have not said if you want to use this in VBA code or on the worksheet in the interactive mode.

One way is to use VBA Event code so that each time there is s change in the required column, the VBA code will rename the range.

To install the following code:

Right click the worksheet name tab and select View code.

Copy the code below between the asterisk lines and paste into the white area of the VBA editor.

Edit the name you want for the named range. (See comment in green after pasting into VBA)

Close the VBA editor. (Alt/Q or click Cross with red background top right)

Save the workbook.

 

'*******************************************************
Private Sub Worksheet_Change(ByVal Target As Range)

If Not Intersect(Target, Columns(2)) Is Nothing Then      'Column 2 is column B
    With ActiveSheet
        'Edit "MyName" in next line to the required name for the range.
        .Range(.Cells(7, "B"), .Cells(.Rows.Count, "B").End(xlUp)).Name = "MyName"
    End With
End If

End Sub
'*******************************************************

The code can be modified to include additional names for additional columns but I will need to know which columns to include and what names you want for each column.

steve rindsberg say:

PowerPoint pre-2007 used a mini-program called MSGraph to create charts; the ability to link data from Excel was one of MSGraph's features.

MSGraph is still an *optional* part of Office.  If you re-run setup, you can add it to your setup, then use it to make the old-style graphs (Insert | Object | Microsoft Graph Chart) .  The paste-link trick from an Excel 2007 range to an MSGraph chart data sheet in PPT 2007 still works.  Possibly with a few glitches ... it's been a while since I last used it in earlier versions, so do test for yourself.

At the other user's end, when they try to edit the chart, they'll get a message that starts "To edit this cuart using the new features available in the 2007 Microsoft Office System, you must first convert it to the Office 2007 system format. Do you want to convert ...?"

Options are Convert, Convert All, Edit Existing.

If they choose convert, PPT does a little dance then tells them that it can't convert one or more charts in the presentation.

A truly dumb error message, that, if they've only chosen to convert the one chart, but dumb error messages are our meat around here.  Enjoy the humor, move along folks, nothing to be done here once the mocking's over with.

The main point is this:  tell them to choose Edit Existing and if they've also opted to install MSGraph, they'll be in business.

I'm not sure what'll happen if they edit the data (w/o access to the linked file) and then return the presentation to you (where PPT can presumably still find the linked data if you activate the graph).  Will their edits update the linked XL sheet, or will the old contents of your XL sheet overwrite the more recent edits?  There's your homework assignment.  Quiz in the morning. ;-)
