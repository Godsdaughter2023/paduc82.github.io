graham mayor say:
If you want a macro to split the merge, then merge to a new document, set the printer to PDFCreator change the path where indicated in bold to where you wish to save the files and run the Splitmerge macro. The macro assumes the newsletters are one page. You would have to adjust the counter for multi-page newsletters.

Option Explicit

Sub Splitmerge()
Dim Letters As Long
Dim Counter As Long
Dim oPub As Publisher.Document
    Set oPub = ActiveDocument
    Letters = oPub.Pages.Count
    Counter = 1
    While Counter < Letters
        PrintToPDFCreator "Merged" & Counter, oPub, iFrom:=Counter, iTo:=Counter
        Counter = Counter + 1
    Wend
End Sub
Sub PrintToPDFCreator(sPDFName As String, oDoc As Document, iFrom As Long, iTo As Long)
Dim pdfjob As Object
Dim sPDFPath As String
    sPDFPath = "C:\Path"
   Set pdfjob = CreateObject("PDFCreator.clsPDFCreator")

    With pdfjob
        If .cStart("/NoProcessingAtStartup") = False Then
            MsgBox "Can't initialize PDFCreator.", vbCritical + _
                                                   vbOKOnly, "PDFCreator"
            Exit Sub
        End If

        .cOption("UseAutosave") = 1
        .cOption("UseAutosaveDirectory") = 1
        .cOption("AutosaveDirectory") = sPDFPath
        .cOption("AutosaveFilename") = sPDFName
        .cOption("AutosaveFormat") = 0        ' 0 = PDF
        .cClearCache
    End With

    'Print the document to PDF
    oDoc.PrintOut From:=iFrom, To:=iTo

    'Wait until the print job has entered the print queue
    Do Until pdfjob.cCountOfPrintjobs = 1
        DoEvents
    Loop
    pdfjob.cPrinterStop = False

    'Wait until PDF creator is finished then release the objects
    Do Until pdfjob.cCountOfPrintjobs = 0
        DoEvents
    Loop
    pdfjob.cClose
    Set pdfjob = Nothing
End Sub

karl timmermans [outlook mvp 2012-2018] say:

Short answer is "you can't" on any count.

- cannot change the name of any existing standard Outlook field - ever, under any circumstances
- cannot change the "field name" of a user-defined field via the Outlook UI
- cannot change the "field type" of an existing user-defined field (whether added to the item using standard messageclass (IPM.Contact) or a custom form name (IPM.Contact.SomeFormName)

If you are in custom form mode - you <can> change the field caption which appears on the form but <cannot> change the underlying field name (field caption and field name are two different items which "appear" to be the same since Outlook assigns the name you enter to both by default.

What you also <can> do is create a user-defined field at the folder level (for standard contacts), assign it any property type, delete it at any point, re-add the same field name and assign it a different property type. This is fine if you have not added to the individual contact items that already have the field and thus may have a value which does not correspond to the new field type definition found at the <folder level>

Some articles about user-defined fields that may be of interest

Understanding Outlook Contact User-defined fields - (in folder) - versus - (in item)
http://www.contactgenie.info/understanding-outlook-user-defined-fields-in-folder-vs-in-item

Understanding Outlook Contact User-Defined Fields in Custom Forms
http://www.contactgenie.info/understanding-outlook-user-defined-fields-in-custom-forms

Should there ever be a need to change either the <field name> or <field type> for any existing user-defined (custom) field that is not part of a custom form - this may be of interest
ContactGenie Toolkit - http://www.contactgenie.com/toolkit_4_outlook.htm
(CTK has special functions specifically for user-defined fields that enable the above capability to cover both the folder definition as well as any contact items that also have the same UDF field - however, changing a field type from <text> to <date> (as a for instance) and the field value does not contain a valid date value is not going to be productive resulting in loss of info).

In your case, a user is trying to edit the .xls/.doc files in the browser which are shared by another user and the user is unable to edit the file, as the edit in browser option is disabled.

This happens because the files are stored in the original users’ library and since the file is shared, the user is unable to convert the file in the supported format as the user do not have the access to the original library to save the backup file. In this case, the user has to open these files in the Office client application to edit them.

If a folder containing .xls/.doc file is shared, then it is easy for the user to edit the file in the browser because the user has the access to save the backup file in the shared folder.

I suggest you to try updating the display and graphics driver from Device Manager and restart the computer to check if this helps.

You may update the Display driver from the Device Manager and check if this helps.

1. Press (Windows key + X) and click on “Device Manager”.

2. Expand “Display Adaptor”.

3. Right click on the driver and select “Update Driver Software”. Once this is done, restart the computer and check if the issue persists.

Further, you may download the latest drivers from the manufacturer’s website and install it to check if this helps.
