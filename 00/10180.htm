ken sheridan say: Marshall's approach and that in my demo file are not really complementary.  The raison d'être of my demo is not so much to demonstrate how to build a set of correlated combo boxes per se, but how to do so in the context of a fully normalized table, i.e. one where the only column to which one of the combo boxes is bound is the final one in the hierarchy.  Marshall's suggestions, by allowing Nulls in the lower levels of the hierarchy where only those above have values which are relevant to the form's current record, like most of the correlated combo box solutions you'll find on-line, means that there must be columns in the table representing each level in the hierarchy, i.e. the table is not fully normalized as each level above the lowest is determined by the lowest level, which is a non-key column, so the columns representing all but the lowest are transitively dependent on the key via the level(s) below.  The existence of transitive functional dependencies means that the table is not fully normalized of course, as it contains redundancies which would allow update anomalies.

Nevertheless, there is no reason why the approach illustrated in my file would not work in the case of a non-normalized table, as all this would really mean is that the combo boxes representing the higher levels in the hierarchy would be bound to columns in the table (Locations in my demo) rather than being unbound controls.  If these columns are numeric foreign keys referencing the numeric primary keys of the referenced tables, however, as they would be in my demo, i.e. DistrictID and CountyID, then in a continuous form it would still be necessary to use hybrid controls, superimposing a text box on each combo box.  So, regardless of whether your table is normalized or not, if you mirror the approach of my demo, everything should work as expected in your form.

If you look at the code it my demo you'll see that once a selection is made in one of the higher level combo boxes it selects the first row in the next combo box and then drops down its list, e.g.

    ' requery Parish combo box to
    ' show parishes of selected District
    Me.cboParish.Requery
    Me.cboParish = Me.cboParish.Column(0, 0)
    Me.cboParish.SetFocus
    Me.cboParish.Dropdown

Where the next combo box only has one item, N/A, this is consequently selected automatically and the user can simply tab out of the control.  Where there are more available items the user is presented with the list from which to make a selection.  You could if you wish use code along the lines of Marshall's to only make a selection if the N/A item is the only one available, but you would need to test for the ListCount value being 1 rather than 0, and in the case of the last combo box, as the value of the control is not 'N/A', but the ValveID which corresponds to the relevant N/A row in the table which provides the control's RowSource, it would be necessary to include some additional code to determine this numeric value, so that it can then be assigned to the control.

Quite why you are not getting the restricted lists as expected is hard to say at this distance, but it is more likely to be a data issue than an interface design issue.  In essence you need to ensure that the referenced table for each level in the hierarchy contains the correct foreign key values which relate it to the relevant row in the referenced table for the next higher level in the hierarchy, plus a set of N/A rows, one referencing each row in the referenced table for the next higher level in the hierarchy.  If you look at the parishes table in my demo, you'll see that it has two N/A rows, one referencing district 8, the other district 9, these being Tamworth and Stoke-on-Trent, which are the only districts for which the N/A option is applicable.
