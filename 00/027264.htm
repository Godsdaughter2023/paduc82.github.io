ken sheridan say: I do this myself with my personal accounts database in a form based on a CurrentAccounts table, where I use a combo box for the TransactionDescription  column, the control's RowSource property being:

SELECT DISTINCT TransactionDescription
FROM CurrentAccounts
WHERE TransactionDescription IS NOT NULL
ORDER BY TransactionDescription;

i.e. the table in which the words are stored is the same table into which the value is being entered.  The combo box's LimitToList property must be set to False (No) to allow you to enter a new word.  In the form's AfterUpdate event requery the combo box with:

Me.[YourComboBoxName].Requery

This will add any new term you enter to the control's list.

This a cheap and cheerful solution, and for my own use is perfectly satisfactory.  I would not do this if I were producing a database for distribution, however.  For that I'd  create separate TransactionDescriptions table to which my CurrentAccounts table would be related, and to insert a new row into the TransactionDescriptions table when a new word is entered into the combo box I'd put code in the combo box's NotInList event procedure, having set its LimitToList property to True.  You'll find examples of the use of this procedure in a little demo file NotInList.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169


scottgem say:
I have am creating a database and would like to add a field that would autofill after you type a letter or two if you have entered that word or words before. How do I do that.?

First, you don't add a field that can do this. Fields are added to tables and you can't put code behind tables in Access 2003. Also, Users should NEVER interface directly with tables. Any interaction between a user and a table should be controlled through forms.

Second, you don't add fields to a form. You place CONTROLS on a form that may or may not be bound to fields in a table. This is a subtle but important distinction. 

The combobox control has an AutoExpand property, that when set to Yes, will do exactly what you describe. As the user types characters into the control, it will find the first match and display it. So using a combo is the easiest (and IMHO the best) way to handle such a situation. 

However, this capability can be simulated using the On Change event of the control. Whenever you type a character into the control. The On Change event triggers. You can then use code to search the database for the first match and display it in the control. Frankly, though the amount of time this will take, I suspect you will kile this process as soon as you test it.

Does it make sense to execute the code every time when any cell is changed? IMHO no...

Andreas.

Option Explicit

Private Sub Worksheet_Change(ByVal Target As Range)
  Dim H As Range
  'Get last cell in column H
  Set H = Range("H" & Rows.Count).End(xlUp)
  'Do we have data?
  If H.Row <= 3 Then Exit Sub
  'Events off, otherwise we call ourself
  Application.EnableEvents = False
  'Fill B3:H.. down
  Range("B3", H).FillDown
  'Copy formats
  Range("A3").Copy
  Range("A4", "A" & H.Row).PasteSpecial xlPasteFormats
  'Copy off
  Application.CutCopyMode = False
  'Events on
  Application.EnableEvents = True
End Sub
