a. user say:

There are a number of reasons why the @onmicrosoft.com domain exists (it is the parent domain for your tenant.  To remove your O365 tenant, the final user account for a tenant needs to be a @<tenant name>.onmicrosoft.com global admin account (after you remove all vanity domains, users, groups, etc.).  Basically, think of @onmicrosoft.com as the parent domain in a forest, and your vanity domains (i.e., your tenant in O365) is a trusted child domain with that parent.  If you want to configure specific settings for your tenant, you're going to need an account with access to your portion of the Microsoft's @onmicrosoft.com parent domain in that forest.  Each object in Azure must have a @<tenant name>.onmicrosoft.com address and since the definitive source (your local AD) didn’t provide one, Microsoft assigned an address to it automatically.

Many AD attributes sync from AD to Azure (O365 AD) and some attributes will sync back from the Azure MSOLDB (Microsoft Online DB aka Azure) to your on-premises AD objects.  What happens is that objects sync between your on-premises AD and the MSOLDB and then from the MSOLDB to the various O365 services such as Exchange Online, SharePoint, etc.  Check the FIM console on your AADsync system to see the various AD attributes that are syncing with the MSOLDB.  Only one AD attribute (that I am aware of) has an issue with syncing and updating properly (and that is the @<tenant name>.onmicrosoft.com SMTP address of course).  Once an object starts syncing with the MSOLDB, you have to temporarily stop AADsync to change the @<tenant name>.onmicrosoft.com address (i.e., like when people get married and change their name).  Microsoft doesn't allow changes to occur for that SMTP address when AADsync is in the “active” status.  Anyway, everything else can change and will sync up properly.  Yes, you can change the UPN in AD and it will sync (and update) the Azure object.  You can provide as many vanity SMTP addresses for an object as you want.  They will all sync, just so long as you’ve set up the vanity domain in your tenant first.  The SMTP addresses (i.e., proxy address attributes in AD) won’t sync between your on-premises AD and the Azure cloud objects unless you first define the vanity domain within your tenant.

You can’t edit the “proxyAddresses” attribute for objects in Azure directly because your on-premises AD objects is considered the definitive source for synced Azure objects (i.e., any change in your local AD objects will be reflected in the Azure objects).  What you actually want is a method to update your on-premises AD object attributes and not a method to update the Azure objects (which isn’t really part of Office 365 but instead is part of Active Directory and your on-premises environment).  Since the AD objects don’t have “proxyAddresses” values already, I assume you don’t have an email system integrated with AD.

Anyway, PowerShell will do what you want, just remember that the “proxyAddresses” attribute is a multi-dimensional array.  What that means is you have to access the attribute array in PowerShell then “process” the items of that array as individual array items.  To help, I’ve attached a TXT file to my post that will show you a portion of how I changed SMTP addresses for my local AD objects (which then synced up to the Azure cloud successfully, once I defined the domain within my tenant).  The attachment is provided as-is and only as an example to help guide you on your way (it is a function I used in a PS script).  The comments in the attachment were my own notes at the time and I didn't reread them prior to update (so I hope that they are up-to-date).  And

Best of luck,
