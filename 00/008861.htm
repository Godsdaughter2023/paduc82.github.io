joeu2004 say: Jim Thomlinson wrote:
Note to the op... if you are doing any kind of reference formula such as vlookup or checking for equality joeu2004's formula is going to work better than mine.
I was afraid I might have left the wrong impression.

When I implied that with my formula, $A$1 + (ROW()-1)/24, "a comparison with the date/time constant" is more reliable, I should have noted that that is probably due to the "close to zero" heuristic described in KB 78113 at http://support.microsoft.com/kb/78113.

Adding n/24 still results in an infinitesimal difference compared to the equivalent constant.  So it will not pass muster with lookup functions like VLOOKUP and MATCH.

Consider the following example:  A1 contains the constant 7/1/2008 1:00; A2 contains the constant 7/1/2008 2:00; B1 contains =A1; and B2 contains =B1+1/24.

Then IF(A2=B2,TRUE) returns TRUE.  But MATCH(B2,A1:A2) returns 1, which corresponds to 7/1/2008 1:00(!), and MATCH(B2,A1:A2,0) returns a no-match error(!).

The reason:  lookup functions do straight-forward binary comparisons without the "close to zero" heuristic described in KB 78113.  For the same reason, IF(A2-B2=0,TRUE) returns FALSE(!).

(Note:  The fact that B1+1/24 is less than 7/1/2008 2:00 is a coincidence.  It could have been greater if B1 were another date/time.  And that would mask the MATCH anomaly.)
Another way to remove the errors would be to round the formula such as this
=ROUND(A1, 12) + ROUND(TIME(1,0,0), 12)

Off-hand, I cannot think of any number theory to support that.  Rounding A1 (date/time) to more than 10 decimal places is useless for dates around 7/1/2008 (5 digits).  Rounding TIME(1,0,0) to 12 decimal places rounds the repeating decimal for 1/24 to fewer digits arbitrarily.

Arguably, your formula does consistently result in a smaller discrepancy (but never zero) starting with the 192nd derived date/time (7/9/2008 1:00).  But is that coincidence?  It results in a larger discrepancy for the first 167 derived date/times (through 7/8/2008 0:00).  (The results are mixed for the derived date/times in between.)

I think the best way to ensure a match with a constant date/time is:

=--TEXT(A1+1/24,"m/d/yyyy hh:mm")

Caveat:  In Excel 2003, that does not always work (i.e. match the constant time) for time-only data; that is, hh:mm with no date component v. --TEXT(A1+1/24,"hh:mm").  The discrepancy is a difference of only 1 in the least-significant bit, presumably due to a minor difference in the internal implementations for TEXT v. converting constants.  The problem seems to be fixed starting in Excel 2007, based on tests performed by Ron Rosenfeld.

(Note:  All details below assume that A1 contains 7/1/2008 1:00 and A2 contains 7/1/2008 2:00 as constants.  YMMV with other date/times.)

Executive summary....  Do one of the following to work around the problem.  See below for details.

Details....

First, instead of Autofill, you might consider using Date fill (Edit > Fill > Series > Date in Excel 2003).  Select A1 first.  For Step Value, enter 01:00.  For Stop Value, enter the date/time corresponding to your last row.  For 22000 rows, enter 1/3/2011 16:00, which I determined by the formula =A1+21999/24.

That Date fill seems to behave similar to the $A$1+(ROW()-ROW($A$1))/24 formula that I suggested.  In Excel 2003, the results exactly match the constant date/time 2 out of 3 times; and the results are off by only an infinitesimal amount (2^-37; one off in the least-significant bit) for the remaining 1 out of 3 times.

The problem is:  the infinitesimal error results in a smaller binary representation 1 out of 3 times.  Consequently, lookup functions like MATCH might not behave as expected.  However, straigtht-forward equality comparisons will behave as expected, thanks to the "closer to zero" heuristic described in KB 78113.

(On the other hand, the infinitesimal error that I see might be specific to Excel 2003.  That error might be remedied starting with Excel 2007, based on information that others have told me.  So there might be no problem with Date fill in Excel 2010.)

Second, returning to Autofill per se, the problem you enounter seems to be due to some dubious heuristics in Excel.

As Jim explained, date/time is stored as a number with a decimal fraction.  You can see this by temporarily formatting 7/1/2008 1:00 as Number with 10 decimal places.

So we might expect Autofill to determine subsequent cell constants effectively as if either =A2+(A2-A1) or =A2+($A$2-$A$1) were copied down the column.

As Jim explained, that does propagate a numerical error because of the way that Excel does arithmetic (namely, binary floating point).  Jim assumed that explained the error in appearance that you saw.  So did I.

But in fact, if we use either formula to fill the column, the cumulative numerical error is less than 0.028 seconds even after filling through row 65536.  (Less than 0.010 seconds in row 22000.)  It is not visible if you format time as h:mm:ss, much less h:mm.

(But the discrepancy still creates problems for equality comparisons with date/time constants as well as for equality lookups.)

However, that is not the results that we see with Autofill.  And in fact, the Autofill results depend on how the cells are formatted.

If A1 and A2 are formatted as Custom "m/d/yyyy h:mm" (without quotes), the normal format after entering the constant date/times when A1 and A2 are initially formatted as General, Autofill increments by about about 1 hour 0.005 seconds instead of A2-A1 (about 1 hour 0.00000042 seconds).

I cannot think of any good reason for that.  And because of that heuristic, we do see an error in the minutes starting in row 11903 (11/8/2009 23:00) when the time format is h:mm.

But if A1 and A2 are formatted as General or Number temporarily just for the Autofill, the cumulative numerical error is again less than 0.028 seconds after autofilling through row 65536.  And again there is no visible error if we format time as h:mm after the Autofill.

However, even in that case, Autofill does not increment by $A$2-$A$1 or A2-A1 (the difference being that A2-A1 changes to A3-A2, A4-A3 etc as we progress down the column).  Instead, the Autofill increments vary somewhat -- mostly about 1 hour 0.0000029 seconds and 1 hour -0.0000059 seconds.

I can only guess that this is due to some "smoothing" algorithm; for example, an average difference of some number of previous cells.   But I do not think that explains why the error in cell-to-cell difference is nearly 10 times larger.

In any case, the results with formatting as General or Number before doing Autofill are not as good as the results from doing a Date fill.  But the General/Number Autofill might be easier to do.

With both methods -- Date fill and General/Number Autofill -- it you want to ensure that the filled date/times exactly match constant date/times, select A1:A22000, format as Custom "m/d/yyyy h:mm" (without quotes), and use Text To Columns twice to convert the column first to text, then to General again.  In Excel 2003:

1. Click Data > Text To Columns; select Delimited and click Next; unselect all delimiters (i.e. not check marks) and click Next; select the Text column data format and click Finish.

2. Click Data > Text To Columns; select Delimited and click Next; unselect all delimiters and click Next; select General column data format and click Finish.

That should result in A1:A22000 formatted as Custom "m/d/yyyy h:mm" (without quotes).  If you prefer a different format, select A1:A22000 and chose the preferred format.

I believe this two-step post-fill procedure should be necessary after doing a Date fill in Excel 2007 or later.  But it would be prudent to test that yourself, since I do not have Excel 2007 or Excel 2010 to confirm that.
