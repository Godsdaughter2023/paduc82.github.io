a. user say:

I found out the problem.

We are sharing the file using Box.com

Box.com doesn't preserve the following extended attribute. Without this extended attribute, the file won't be loaded during startup.

com.apple.FinderInfo:
00000000  57 58 54 4D 4D 53 57 44 00 00 00 00 00 00 00 00  |WXTMMSWD........|
00000010  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00  |................|
00000020

Adding the attribute, fixes the issue.

I used the following command in the Terminal:

xattr -wx com.apple.FinderInfo '57 58 54 4D 4D 53 57 44 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00' startup.dotm

So I'm now thinking about the best way to share the file, including the extended attribute, or fixing the attribute sort of automatically

imikem say: I have run into this issue on our office network with mixed Windows and Mac Office clients. I was able to trace the problem to users disconnecting from the network without first closing Excel or Word documents on file server shares. This results in lock/temp files remaining on the shared directory, after which any Mac Office client attempting to open these files getting a read-only version since Office sees the lock file and assumes someone else has the file open already. 

If you can confirm these lock files are stale (if it's more than a day or two old, you're probably okay, but do your own homework!), deleting them should resolve the issue.

jim gordon mvp say:

Only one user can have an Excel file open at a time with read/write capabilities UNLESS you go to Excel's Tools menu and choose Share Workbook. 


If Share Workbook is NOT turned on, then any additional attempts at opening an already open file will result in an offer to open the already workbook as Read Only. This is standard procedure.


If Share Workbook IS enabled, then up to 256 users can open and work on the same workbook at the same time, and these users can be on both Macs and PCs. However, everyone doing this must have Excel 2011 on the Mac and Excel 2007 or later on the PC. Turning on Share Workbook disables certain features.


As was already mentioned, if someone disconnects from the network or for some reason does not do a normal close of an Excel file (or any other Office file for that matter) the server will wait forever for the file to be properly closed and won't allow anyone else to open it. The Share Workbook tool has a provision to kill the connection for an improperly closed workbook, but in a single-user situation, the server must be told to release the connection by the server administrator, or simply reboot the server.
