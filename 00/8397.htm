The export part of my macro works exactly like I want.  All that remains is to let me pick the file where I want the ported data to be saved.  Unfortunately, this is what is giving me trouble.

I want to put a dialog box on the screen, allow the user to name a file (existing or not), in any directory, where the exported data should be saved (in Excel spreadsheet format).  Such a dialog box is typically done using the FileDialog object in VBA.  Because the user might identify a file that does not yet exist, the FileDialog needs to be created using msoFileDialogSaveAs: msoFileDialogFilePicker and msoFileDialogOpen require the file to exist on the system, and msoFileDialogFolderPicker doesn't let the user pick a file (and the folder must be preexisting anyway).

Outlook does not support the FileDialog itself, so to access the FileDialog I have to use another Office program.  In the past I have used the FileDialog from a parallel executing Word application, and that works exactly as expected.  The problem is that Word focuses on its own file types, so it tries to suggest the file be saved with a .docx extension rather than a .xlsx extension, and doesn't show me any other spreadsheet files.  And the msoFileDialogSaveAs does not support adding Filters, so the lack of a filter for Excel spreadsheet files made the Word FileDialog pretty useless.

I thought it would be just a simple matter of replacing the reference to the Word application with the Excel application (which is also running in parallel) and using the FileDialog from the Excel application.  Unfortunately this doesn't work.  When I attempt to create the FileDialog object using

  Set FileSelectDialog = ExcelApp.FileDialog(msoFileDialogSaveAs)

I get one of two run-time error messages: either Run-time error '462' (The remote server machine does not exist or is unavailable) or Run-time error -2147023170 (800706be) (Automation error: The remote procedure call failed).  Which error message I get seems to depend on whether the Acrobat PDFMaker Office COM Addin is enabled or disabled in Excel: disabled gets the 462 error, enabled gets the -2147023170 error.

I know that Excel is running.  I have code before the problematic statement that checks both for ExcelApp not being Nothing and that I can add a new Workbook (if Excel was running when the code began but was terminated before I attempt to use Excel, the object would still exist but not point to a valid application).  I even can see the application in my taskbar when the function starts, and while the problematic statement executes the application disappears from the taskbar.  So something is shutting down Excel while the statement in question is executing.

For those who want the see the full code of the function I am trying to use to open the file, here it is:

Private Function SelectSaveFile(ByRef SourceFolder As String, ByRef DestFile As String) As String
  Dim FileSelectDialog As Excel.FileDialog
  Dim TempDoc As Excel.Workbook
 
  Set FileSelectDialog = Nothing
  Set TempDoc = Nothing
  SelectSaveFile = ""
 
  If (ExcelApp Is Nothing) Then
    Set ExcelApp = CreateObject("Excel.Application")
  End If
 
  On Error Resume Next
  Set TempDoc = ExcelApp.Workbooks.Add
  On Error GoTo 0
  If (TempDoc Is Nothing) Then
    Set ExcelApp = CreateObject("Excel.Application")
  Else
    TempDoc.Close xlDoNotSaveChanges
  End If
 
  Set FileSelectDialog = ExcelApp.FileDialog(msoFileDialogSaveAs)
  FileSelectDialog.InitialFileName = DestFile
  FileSelectDialog.Title = "Save calendar entries from " & SourceFolder & " to:"
 
  If (FileSelectDialog.Show = -1) Then
    SelectSaveFile = FileSelectDialog.SelectedItems(1)
  Else
    SelectSaveFile = FILEDIALOGERROR2
  End If
 
  Set FileSelectDialog = Nothing
  Set TempDoc = Nothing
End Function

ExcelApp is a global variable, defined as Excel.Application in another module.  FILEDIALOGERROR2 is a constant string (and should not be relevant to the problem at hand).

I am running Office 2013 (Outlook, Excel, and Word) on an HP Elitebook running the latest updates for Windows 10 and Office 2013.

Any help would be greatly appreciated.  Thanks in advance!

Ariel

therogueb97d say:

Ok, some experimentation has revealed at least a work-around, if not a solution.  It turns out that having an open workbook lets the code operate.  So adding

Set TempDoc = ExcelApp.Workbooks.Add

and then

TempDoc.Close (xlDoNotSaveChanges)

after using the file dialog seems to avoid the run-time errors.

I won’t (and can’t) call this an answer, but it at least seems to help.
