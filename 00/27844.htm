a. user say:

Almost anyone who has to do MailMerges from Excel in the past 8-10 years knows of this issue, and that certainly includes Graham and Paul. They don't really have to do the experiment you suggest: they have already seen what you have seen.

I do think it is a serious and irritating problem. You've put 59.3 in your sheet, and you want 59.3 in Word. Why should you have to add a switch in Word to get it to show the number as you first entered it? 

However, 
 a. what you see does not actually mean that /Word/ is the culprit, and below I will explain what I think /is/ the culprit.
  b. the difficulty for everyone in these groups is that it is not, in fact a new problem. It has been there since Office XP. Microsoft knows about it. They haven't done anything about it in all that time. So it doesn't look as if they are going to do anything about it, and perhaps they never will. 

This message is really about the Excel side of things. but I think much of the discussion is the same for Access, except that I don't know anything about the precise representation it uses for numeric values.

I believe that the main cause of this problem is located not in Word, nor in Excel, but in the OLE DB provider that Word uses to get data from Excel.

You will see people talking about Word getting the "underlying data" from Excel, and that is what I used to think was happening. It is certainly true that when Excel saves in .xls format, it will save a number such as 59.3 in a binary floating point format, so you might expect a conversion error.  AIUI Excel currently saves the highest 30 bits of the number converted to 64-bit IEEE 754 format. Or something like that. So for example, for 59.3, Excel stores 012AB740. AUIUI the "1" bit in "01" actually indicates that the number has been multiplied by 100. 

If you reverse that, right-pad with zeros to 40B72A0000000000, and enter it into an IEEE 754 converter, you get 5930, i.e. it is an exact representation of the decimal number * 100. So in that particular case, there is no conversion error.

It is of course possible that while Excel is running, it is using some other floating point representation that does introduce a conversion error. But I don't really see why it would. Further, when you connect Word, using OLE DB, to a .xls data source that is not already open in Excel, AFAICS no copy of Excel is started. Only the provider is running. Now the provider relies on some underlying Jet/ACE software, and that may rely on, or have the same code as Excel - in fact, I'd say it's highly likely that that is the case. But it isn't actually using excel.exe

You can see this even more clearly if you save your 5.93 in a .xlsx file, then look at the XML code for the worksheet. The value "5.93" is stored, i.e. as text. AFAICS no floating point representation is stored. Again, if you connect to that data source via OLE DB, excel.exe is not involved, but you will see the same conversion error.

So IMO the older "underlying representation" theory is incorrect. AFAICS conversion errors are almost certainly introduced by the OLE DB provider.

FWIW, what Word gets from the OLE DB provider depends partly on the mix of data in an Excel column. Excel can have mixed data types in a column. The OLE DB provider is "database-oriented" and tries to determine a single data type for each column, based on a number of parameters including the data that it finds, typically in the first 8 cells of each column. If you have a column containing 7 "59.3" numbers and 1 text such as "abc", the provider, as used by Word, will decide that the column is text. In that case, I think you will see "59.3" in Word rather than 59.299999999999997. However, if you have 8 numbers, the provider decides that the column is numeric. I suspect that that is the point where it all goes wrong - the provider probably tries to convert everything in the column to a numeric representation that is not IEEE 754, and introduces a conversion error.

AFAICS the behaviour of the OLE DB provider is essentially the same for Word XP and later. I see this problem in Word 2003, for example, and if you aren't that suggests to me that Word may not be using the OLE DB provider.

[Something that I don't actually understand at this point is that when I create a test sheet in Excel 2003 .xls format, and use it as a Mail Merge data source, the number I see in Word 2003 on Windows XP is 59.300000000000004. If I use the very same sheet as a data source in Word 2007 on Vista 32-bit, the number I see is 59.299999999999997. I thought there might be a difference between the ACE provider that Word 2007 uses by default and the Jet provider that Word 2003 uses, but switching to the Jet provider made no difference. Perhaps it's a Jet version issue. I wonder if someone with Office 2007 on a Win XP platform could look at that?]

The problem is that even the hypothesis that getting the OLE DB provider to see everything as text would solve the problem, there does not seem to be a parameter that would allow Word to tell the provider to do that. So there does not seem to be a lot that Word can do about it.

So, even if I am correct, is that Word's fault? I don't think so. Does Word rely on the OLE DB provider? Yes. Is the OLE DB provider therefore part of Word? Not really. The same is true of Excel. But unless Microsoft is able and willing to fix that problem, I'm not sure there is anything to be gained by pointing the finger more accurately. I doubt if Microsoft has an army of people who can remember how to work with OLE DB providers because they starting moving to ADO.NET 10 years ago or so.

Personally, I think of this type of problem as being a problem in the "Microsoft Office System." It is one of several types of problem to do with data transfer in Office that I do not think are likely to be solved unless/until Microsoft sees a need to provide new XML or json data interfaces between the various Office Apps. 

Finally, there are other potential problems when accessing Excel data. For example, it's easy to say that data should appear in Word as it appears in Excel, but if, for example, you are in the UK and you enter 59.3 into your sheet, then send your sheet to France where they mailmerge it, the chances are they'll want to see 59,3. And that's probably what they'll get, but there are circumstances in which they might not.
