brian tillman [outlook mvp 2007-2019] say:

First, Windows Easy Transfer is not a good way to move your Outlook data.  It nearly always creates a corrupted mail profile.

POP accounts have no way to mark something as read on the server.  That information is kept on the computer.  Since you have a new computer, which means you have a new mail profile, the record of what the old Outlook downloaded doesn't exist anywhere but the old computer.  Since you decided to leave all the mail in the Inbox on the Earthlink server, the new Outlook sees all the messages as new and downloads them.  If you don't wish to download them, log into the Earthlink server, create a new server-side folder named anythin you want, then move the messages from Inbox to that folder.  Since they won't be in Inbox on the server any more, the server won't tell Outlook about them and Outlook won't download them.

With only “Scan all downloaded files and attachments” enabled in the real-time protection options, I don’t get any detection when I save an “infected” attachment in Outlook 2003, whereas in Outlook Express, my eicar test file (eicar.txt) is detected as soon as the attachment is saved. So the IAttachmentExecute API is apparently not implemented in Outlook 2003.

 

The IOfficeAntiVirus interface is implemented in Security Essentials so that it will automatically perform an on-demand scan when it receives a “Save” or “Execute” notification from a browser or email client that implements the IAttachmentExecute interface; like Internet Explorer, Firefox 3.0, or Outlook Express. That’s all part of the same “Scan all downloaded files and attachments” feature, which as Rob points out, was inherited from Windows Defender. Except for monitoring automated downloads, this interface is redundant and simply automates an on-demand scan for those applications that implement it. It doesn’t provide any new type of protection, but does eliminate the need to apply the old “save and scan” rule that you should apply with applications that don’t implement the IAttachmentExecute API.

 

With Outlook 2003 you still have the full protection of the on-access (Monitor file and program activity on your computer) and on-demand scanners, but you will need to manually perform the on-demand scan when you save a compressed attachment. On-access scanning is done in kernel-mode with a file system filter driver (mpfilter.sys). The on-access scanner thus operates independently and provides protection for all applications, but it doesn’t scan compressed files. The on-access scanner detects and blocks my eicar.txt test file, which isn’t compressed, as soon as it’s saved in Outlook 2003: http://social.answers.microsoft.com/Forums/en-US/msescan/thread/91ec0a00-5002-470b-9d46-7db4a3fa2555

jay freedman say:

The "faulting module" MSGR3EN.DLL is the grammar checker, and the exception code 0xc0000005 indicates an "access violation error" -- the program tried to use a memory location that it doesn't have permission for. If there really was a bug in the program that caused this, it would happen to everybody with the same versions of the program files. Since that isn't the case, it's most likely that something in your copy of Office is damaged or misconfigured.

To fix that, follow the instructions to repair the Office installation.
