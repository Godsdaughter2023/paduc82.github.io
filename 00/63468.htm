cescal say: Thank you for your reply. I have another question, In my system 32 folder is a wininet.dll (2) file and also wininet.dll (3) file and 4 . My question is why are there so many different wininet.dll file with these different numbers and which one should I copy ?

a. user say:

I am not sure why you see stuff like that, but on a "normal" system in good health, there should only be one wininet.dll file in the system32 folder.

I think that if you see something like that wininet.dll(2), wininet.dll(3), etc. that means that you may have  been having problems updating your Internet Explorer and if you keep trying and trying or uninstalling and reinstalling IE, it may make a copy of the file then.   If these updates go smoothly, the installation or update should clean up after itself when it is done.  Any of that sound familiar?  Maybe somebody else can tell us why that is.  But something happened prior to those files being created.

Since the wininet.dll file is one of the 3498 files that Windows File Protection looks after, there should also be a backup copy here (assuming XP is installed on your C drive in the Windows folder):

c:\windows\system32\dllcache    (that is where XP keeps backup copies of important files).

To see Windows File Protection work, navigate to c:\windows\system32 and rename the wininet.dll file to something you can remember like wininet.xxx:

rename   wininet.dll   wininet.xxx

Then the important wininet.dll file would be "missing" and Windows File protection should automajically quickly and silently replace the "missing" file using the backup copy in the dllcache folder and put an event like this in the Event Viewer System log:

Event Type:    Information
Event Source:    Windows File Protection
Event Category:    None
Event ID:    64002
Description:
File replacement was attempted on the protected system file c:\windows\system32\wininet.dll. This file was restored to the original version to maintain system stability. The file version of the system file is 8.0.6001.23532.

That is Windows File Protection in action.  Try it!

If you installed Service Pack 3 from a download, there may also be a copy here (along with lots of other files):

c:\windows\ServicePackFiles\i386

Then if you ever install a MS Update where later and the update (especially IE updates) include a new version of wininet.dll, you will find old copies of the file (which may be the same) in the MS Update folders that get created when you installed the update.  This is because when you install an update, the installation makes a copy of the original file(s) when the update is installed.  That is so you can uninstall the update later if you need to - uninstalling the MS update will put the original file back.

So you may find lots of wininet.dll files on your system if you have been applying updates to your Internet Explorer (think in the inet in wininet to mean Internet and that makes sense).

If you search your system for wininet.dll, you may find dozens of wininet.dll files in various places on your system.  The same is true for lots of XP files - you could have lots of them scattered around on your system.  The only ones that really come into play when XP is running are the one in system32 and the backup copy in the dllcache folder.

Photo Gallery will do bulk conversion with this approach:

2) Click the Edit tab

richardeiler say:

Ok,

The point was that when the PC is shutdown the keep alive memory function that the CMOS battery provides will be lost and the time sync from the last session is then lost as well causing an immediate resync upon startup or a time that is slightly off of the correct time until a resync is accomplished. Also, the various time servers available are somewhat notorious for being unreliable from time to time (no pun intended). The other symptom, although less noticeable if applicable at all, would be that any BIOS settings that had been made to other than default would be reverted to the default BIOS settings.

If your device is newish there is a chance that it does not incorporate a CMOS battery type keep alive memory function, perhaps.

Again, it was a thought as to a possibility based on past experiences.

Good luck,

-Richard
