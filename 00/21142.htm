rohn007 say:

Keys are used in an index to speed up access to the table associated with the index.  In theory, for small tables you don't need an index, just a primary key

A "primary" key is a field (or fields) that allow you to uniquely retrieve a row from a table.

A "secondary" key is a field (or fields) that allow alternate access to data in a table.

A "foreign" key, is a primary or secondary key from ANOTHER table that gives you access to information in that table.

For example you have an order table. It has primary index of "order number".  Each order is associated with a specific customer. Rather than copying the customer information to the order row, you just insert the customer specific customer number key into the order row.  That customer number is a foreign key to the customer table.

ken sheridan say: Firstly you need to identify the candidate keys in the tables.  A candidate key is any column or combination of columns whose values are distinct in the table.  A table might have a number of candidate keys, of which one would be designated as the primary key, in which case the others would each be included in a unique index

Taking your SalesByCustomer table as an example, if all company names are distinct, and a customer makes only one purchase on any given date, then CompanyName and SaleDate would be a candidate key.  If a customer can purchase multiple products on the same day, but not purchase the same product twice  in one day, then CompanyName, SaleDate and Product would be a candidate key.

A candidate key of Contacts would be CompanyName, so you could make this the primary key, and create a relationship with SalesByCustomer on CompanyName.

As regards the BackLog table, if the CompanyName and RequestedDate columns reference the CompanyName and SaleDate columns in SalesByCustomer, and CompanyName and SaleDate columns are a candidate key of SalesByCustomer, then a relationship between the two tables can be created on these 2-column keys.  If, on the other hand, CompanyName, SaleDate and Product constitute a 3-column key of SalesByCustomer, then SalesByCustomer would need to be decomposed into two tables, in one of which CompanyName and SaleDate are the key, and in the other of which CompanyName, SaleDate and Product are the key, i.e. in the latter CompanyName and SaleDate are a 2-column foreign key referencing the 2-column primary key of the former.  The BackLog table can then reference the former in a relationship.

As described above all of the keys are 'natural' keys.  There is nothing in the database relational model which prohibits this, but often a table will be given an autonumber 'surrogate' primary key.  You can replace natural primary keys with surrogate keys by first creating an autonumber column in the referenced table in the relationship type, which will be populated automatically, and then adding a corresponding long integer number foreign key column to the referencing table.  Then, by joining the tables on the 'natural' candidate keys in an UPDATE query, the numeric foreign key column can be populated with the value of the autonumber primary key in the referenced table.

Whether you use 'natural' or 'surrogate' keys, the interface for data entry into two tables in a one-to-many relationship type would normally be a form based on the referenced table, and within it  a subform based on the referencing table.  By linking the subform to the parent form on the keys, the value of the primary key of the parent form's current record is automatically inserted into each row inserted into the referencing table via the subform.

If you do not want the Greek checked, you would want to mark it to not be checked. This is labor-intensive. Installing the Greek language pack would be less trouble.

To mark something to not be checked, you would want to create a special character style, i.e. "Greek No Check." That might have a component of your Greek fonts, but would also have the Language formatting of "do not check spelling or grammar."

You could assign a keyboard shortcut to that style. This is not automatic.
