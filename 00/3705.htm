JMH,
I'm not a big fan of the built-in reports but they may be useful in some cases. The Task report is basically the Task Sheet view (Gantt Chart view without the Gantt display).

In Project a view is made up of a screen, a view table, a group and a filter. The view table defines which Project fields will be shown as columns and how they will be shown (order, format, etc.). You can create your own custom view table or edit the current one under under View/Data group. Hit the little down arrow next to the Table icon. In a dynamic master file, if you add the Project field to the view, it will display which subproject each task belongs to.

The problem:

We have one supervisor who delegates the entire thing so naturally has to accept/deny all requests including requests from employees who may not directly be under him. 

The question: Is there a way to make 3 delegates (supervisors) who each receive their own employees time off requests but do not see the others and all from the same mailbox so everyone can still use the one calendar to see when everyone is off? 

Basically:

Supervisor 1 only sees/receives and approves/denies employees a, b and c requests.

Supervisor 2 only sees/receives and approves/denies employees d, e, and f requests. 

Supervisor 1 & 2 and Employees a, b, c, d, e, f are able to view 1 calendar for all of the time off requests. 

Thanks for any help!

vasil l. michev say:
You can't. A delegate can see/approve any and all requests, you cannot limit it to only specific ones. What you can probably do is try and configure some mailbox rules that will forward the request based on the recipient, but that means disabling the automatic "forward to delegates" option. Delegates will still be able to see/approve/deny all requests if they look directly in the Calendar though.

angelmesa say:
Hi S&SB,

I'm understanding you are using a personal account hosted on http://outlook.live.com, then, the procedure sent before still can help you to fix the issue.

However, you will need to set up your outlook.com account on Outlook client application, once you have connected your account on Outlook, you can follow the process

Download and Install the MFCMAPI tool from http://mfcmapi.codeplex.com/ You need to run it from a machine where Outlook is installed and properly configured to the faulty mailbox.

1. Open the MFCMAPI tool; Accept the initial popup window
2. Click Tools, then click Options.
3. Click to enable Use MDB_ONLINE when calling OpenMsgStore

4. In the Session menu, click Logon.
5. If you are prompted for a profile, select your default profile name (usually Outlook), and then click OK
6. At this point, you should see your stores

7. Double click your default store. A new window is going to open, expand the Root Container node; then expand Top of Information Store.
8. Right-click Inbox and select Open associated contents table. A new window should open. The top pane contains a list of entries (or rows) and the bottom pane contains a list of properties for a particular entry you select.
9. You will most likely find the ‘good’ Junk E-mail Rule as shown below (it has Message Class = IPM.ExtendedRule.Message). You can leave that rule alone. It’s not the rule that’s causing the problem.

10. The best way to search for the ‘corrupt’ junk mail rule is to walk each entry (or row) in the top pane where ‘Message Class = IPM.Rule.Version2.Message’. This is the list of all your Inbox rules. One of these rules is the corrupt Junk mail rule and you will need to find it. You MUST click on each row in the top pane (where Message Class = IPM.Rule.Version2.Message) and walkthrough each property in the bottom pane until you find the property that indicates ‘JunkEmailRule’ as shown on the documentation

11. When you find it, make sure that it is selected (in the top pane), then right-click it and select “Delete Message”.


12. At this point, you can close the MFCMAPI editor tool
13. Reopen Outlook
14. Try again the scenario to confirm if that solved your problem

Please see the documentation were you can see the pictures
https://support.microsoft.com/en-us/help/286040...

The only thing doesn't mention the documentation is the steps 2 & 3 "MDB_ONLINE when calling OpenMsgStore" must be enabled because the mailbox is hosted online.
