When you sign out of the computer, it shows the default picture because it doesn't know what to show.  No user account profile is loaded, so no preferences are loaded either.  It literally has no way to know what picture you want there.

When you shutdown or restart, it automatically selects the previous user's account (you can verify this because it starts up asking for that person's password).

So in summary, what you are seeing is the normal system behavior (although I agree it would be nice to be changed), but nothing is broken or wrong with your computer.

Our UEFI program to test TPM functionality does not function correctly on the surface pro 4, but the same code works fine on the Surface Pro 3 (and on other UEFI machines with a TPM 2.0 device).

The TPM 2.0 device functions correctly in Windows.

We are utilising the EFI_TREE_PROTOCOL (UDK 2014) and the equivalent EFI_TCG2_PROTOCOL on UDK 2015 to communicate with the TPM 2.0 device. In both cases the behaviour is the same.

The offending call is the protocol's SubmitCommand() function, which always returns EFI_STATUS code 2 which is  RETURN_INVALID_PARAMETER, regardless of which actual TPM command is encoded in the input buffer.

Another function exposed by the EFI_TREE_PROTOCOL/EFI_TCG2_PROTOCOL is GetCapability(), which does function correctly on the Surface Pro 4.

Secure boot was not enabled and the tests were performed both with a freshly cleared TPM and a provisioned TPM (ownership had been taken).

Obviously the UDK code is not Microsoft specific, and I will post the same question on the UDK forums, but this error only occurs on the Surface Pro 4, which leads me to believe it a UEFI firmware or TPM issue specific to the Surface Pro 4.
