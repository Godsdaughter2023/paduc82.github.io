stephen boots say:
I don't disagree that there's an issue with the item count and the progress bar -- confusion surrounding it. I do believe that the progress bar should present progress reasonably well, though all progress bars are approximations for any program using them, hence the oddball behavior observed at time. The count issue could be an indication of a deeper problem.

However, the thing to consider is that a scan -- any scan -- is after the fact. What is most critical is real time protection and monitoring of activity that might be suspicious. Things found in a scan are rarely active infections. They are typically dormant in saved files that were not detected when originally encountered and an updated definition set now recognizes the infection. Real time protection would then also recognize the infection on access.

A full scan does scan all files. A quick scan only scans the likely locations for infection and a custom scan is user defined. It is my understanding that a full scan will not scan inside compressed files/archives that have been previously scanned if the footprint (hash) of the file is known by MSE from a prior scan. So, let's say an archive contains 1,000 files. On the first encounter of this archive during a full scan, everything inside the archive is scanned and the container file is catalogued by MSE with a hash. On a subsequent scan, there is no need to scan deeply into the archive as long as the hash is intact.

-steve
