ken sheridan say: You should be able to do that relatively easily.  Join the TBL_lMovie table to TBL_Jct and this to TBL_Genre.  Also join the TBL_lMovie table to TBL_JctActor and this to TBL_Actor.  You might need to use outer joins if a movie record might not have a related genre or actor record.  Add the relevant columns to the query.  To avoid duplicate rows being returned set the query's Unique Values property to Yes in its properties sheet, or amend it in SQL view to SELECT DISTINCT etc.

As regards the parameters the best way to make a parameter optional is to test it for Null, e.g. in query design view the criterion for the Title column would be:

=Forms!frmMovieSearchQuery!Title OR Forms!frmMovieSearchQuery!Title IS NULL

and that for the Yearmade column:

=Forms!frmMovieSearchQuery!YearMade OR Forms!frmMovieSearchQuery!YearMade IS NULL

This method allows for Nulls in the columns, which your current one does not.

You will find, however, that once you've saved the query, if you open it again in design view Access will have moved things around considerably.  The underlying logic will be essentially the same, however.  The query should still work as expected, though there is a possibility with complex combinations of criteria that it might end up too complex to open.  It would also be difficult to amend the query should you want to add additional parameters.  This can be overcome by inserting the WHERE clause in SQL view (before any ORDER BY clause if you have one) and saving the query in SQL view once you've designed the rest of the query in design view  For the above two parameters it would be:

WHERE (Title=Forms!frmMovieSearchQuery!Title
OR Forms!frmMovieSearchQuery!Title IS NULL)
AND (YearMade=Forms!frmMovieSearchQuery!YearMade OR Forms!frmMovieSearchQuery!YearMade IS NULL)

Note how the OR operations are parenthesized to force them to evaluate independently of the AND operations; this is important.

To add further parameters is very easy; it's just a case of tacking on more parenthesized expressions using the OR operator with additional AND operators between each.

Tip:  when building complex SQL statements I find it a good idea to copy and paste the SQL statements for queries into a Word document or similar.  If things do go AOT when amending the query you can ten simply copy the statement from the Word document and paste it back into the query in SQL view.

You can open a form or report based on the query from your seach form, but the form might not be updatable.  For that you'd be better using a movies form with separate genre and actors subforms within it as John suggested.  It would be possible to then filter the parent form on its movie data, and within that form filter it further on the basis of the genre and/or actors data.  This would require a little work, however.  You'll find an example of this sort of thing in the StudentLog.zip in my public databases folder at:

ken sheridan say: The first question is are you sure that the combo boxes' bound column in each is in fact the text column of the movie title, the genre or the actor name?  Or is it a hidden MovieID, GenreID or ActorID column?   If the ColumnCount property of the combo box is 2, the BoundColumn property 1 and the ColumnWidths property something like 0cm;8cm, or simply 0cm (or equivalent in inches) then it is almost certainly the latter and the parameters referencing the combo boxes should be on the MovieID, GenreID or ActorID columns not the text columns.

Whichever is the case, the problem may also be due to the outer joins.  Restricting a query on a column on the right side of a LEFT JOIN or vice versa in effect turns it into an inner join.  Does the query return all the rows you'd expect if you remove the WHERE clause?  If so then you can probably restrict it by means of subqueries:

SELECT DISTINCT TBL_Movie.Title, TBL_Genre.Genre, TBL_Actors.ActorName
FROM (TBL_Movie INNER JOIN (TBL_Actors RIGHT JOIN TBL_JctActorMovie
ON TBL_Actors.ActorID = TBL_JctActorMovie.ActorID)
ON TBL_Movie.MovieID = TBL_JctActorMovie.MovieID)
INNER JOIN (TBL_Genre RIGHT JOIN TBL_JctGenreMovie
ON TBL_Genre.GenreID = TBL_JctGenreMovie.GenreID)
ON TBL_Movie.MovieID = TBL_JctGenreMovie.MovieID
WHERE (TBL_Movie.Title=[Forms]![FRM_Search]![TitleCmb]
OR [Forms]![FRM_Search]![TitleCmb] IS NULL)
AND EXISTS
    (SELECT *
     FROM TBL_Actors, TBL_JctActorMovie
     WHERE (TBL_Actors.ActorID = TBL_JctActorMovie.ActorID
     AND TBL_JctActorMovie.MovieID = TBL_Movie.MovieID
     AND TBL_Actors.ActorName=[Forms]![FRM_Search]![ActorCmb])
     OR [Forms]![FRM_Search]![ActorCmb] IS NULL)
AND EXISTS
    (SELECT *
     FROM TBL_Genre, TBL_JctGenreMovie
     WHERE (TBL_Genre.GenreID = TBL_JctGenreMovie.GenreID
     AND TBL_JctGenreMovie.MovieID = TBL_Movie.MovieID
     AND TBL_Genre.Genre=[Forms]![FRM_Search]![GenreCmb])
     OR [Forms]![FRM_Search]![GenreCmb] IS NULL);

Each of the subqueries is correlated with the outer query on MovieID.  The two tables used by each are joined by a criterion in the WHERE clause rather than by a JOIN clause.  This allows for restriction on either table, and for the join criteria to be combined with other criteria, so it is possible to reference the combo boxes to restrict the results.  Note how in this case the AND operations are parenthesized to force them to evaluate independently of the OR operations, so that if the parameter is NULL the subquery will return all rows and the EXISTS predicate will be TRUE for any row returned by the outer query.

craigsparkes5775 say:

Hi Martin,

You could either:

1. Create a distribution group for info@ and then add the mailboxes to the group. This will deliver the email to members of the group. They can then reply from their own email account.

2. Create a shared mailbox and give the additional users access to the shared mailbox (full permissions / send as permissions). They can then add the info@ mailbox within Outlook. Emails will then be received into the info@ mailbox and can be sent / replied from info@
