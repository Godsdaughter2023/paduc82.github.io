ken sheridan say: The check box should be unbound and you should  not have a field in the form's underlying table to indicate that the client is Level 2.  That field should only be in the Clients table from which the combo box draws its list.  To have a field in the form's underlying table as well would be to introduce redundancy, which, as you've found out, can easily lead to inconsistent data.

Lets assume you have a field named Level in the Clients table, with a value 2 for Level 2 clients.  What you can do is include that field as a hidden column in the combo box's list.  So if we assume the combo box's RowSource is currently:

SELECT ClientID, Client FROM Clients ORDER BY Client;

And its ColumnCount property is 2, and its ColumnWidths property is 0cm;8cm or something equivalent in inches, the it would become:

SELECT ClientID, Level, Client FROM Clients ORDER BY Client;

And its ColumnCount property would become 3, and its ColumnWidths property would become  0cm;Ocm; 8cm, i.e. the first two columns are hidden.

The ControlSource for the unbound checkbox would then be:

=([cboClient].Column(1) = 2)

where cboClient is the name of the combo box.  The Column property is zero-based, so Column(1) is the second column, the hidden Level value.  If this is 2 the expression used as the ControlSource for the check box evaluates to True, so the check box will be checked.

If you have any problem translating the above to your own scenario, post back with details of the combo box's current properties and of the clients table along with the names and data types of its relevant fields.

"Ken Sheridan" wrote in message news:4d3a58c7-68d2-4b88-babd-e2b24970c431...
The check box should be unbound and you should  not have a field in the form's underlying table to indicate that the client is Level 2.  That field should only be in the Clients table from which the combo box draws its list.  To have a field in the form's underlying table as well would be to introduce redundancy, which, as you've found out, can easily lead to inconsistent data.

Lets assume you have a field named Level in the Clients table, with a value 2 for Level 2 clients.  What you can do is include that field as a hidden column in the combo box's list.  So if we assume the combo box's RowSource is currently:

SELECT ClientID, Client FROM Clients ORDER BY Client;

And its ColumnCount property is 2, and its ColumnWidths property is 0cm;8cm or something equivalent in inches, the it would become:

SELECT ClientID, Level, Client FROM Clients ORDER BY Client;

And its ColumnCount property would become 3, and its ColumnWidths property would become  0cm;Ocm; 8cm, i.e. the first two columns are hidden.

The ControlSource for the unbound checkbox would then be:

=([cboClient].Column(1) = 2)

where cboClient is the name of the combo box.  The Column property is zero-based, so Column(1) is the second column, the hidden Level value.  If this is 2 the expression used as the ControlSource for the check box evaluates to True, so the check box will be checked.

If you have any problem translating the above to your own scenario, post back with details of the combo box's current properties and of the clients table along with the names and data types of its relevant fields.
