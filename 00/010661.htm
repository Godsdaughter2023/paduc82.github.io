ken sheridan say: I'd strongly endorse what John has said about using a single table, and you can achieve this quite easily with a few steps.  Before starting it is imperative that you back-up the tables, then:

1.  Change the name of the Enrolled Students table to Students.
2.  Add an Admission Status column (field) to the Students table.
3.  Create and execute an Update query which updates the Admission Status column in all the rows in the Students table to "Enrolled".
4.  Create and execute an Append query which appends the relevant columns in all the rows from the Prospective Students table to the Students table, assuming there is currently no overlap between the two tables.  If the tables have an autonumber primary key such AS StudentID, then don't include this in the columns to be appended; the new rows in Students will automatically be given new StudentID values.
5.  Create a ProspectiveStudents Query based on the Students table which returns all columns, with a criterion on the Admission Status column of <> "Enrolled".
6.  Create an EnrolledStudents Query based on the Students table which returns all columns, with a criterion on the Admission Status column of = "Enrolled".

The only complication which might arise is if there are any other tables which reference the current Prospective Students table on the StudentID column.  As new StudentID values will be assigned to the additional rows in Students, then the foreign key StudentID values in any referencing tables will not have the correct values to reference these.  These can be corrected by means of an update query, but this must be done before deleting the original Prospective Students table.  I won't confuse you further with details of this at present; we can come back to this if necessary.

If all has gone as expected, and there is no reason why it should not have, you can use the ProspectiveStudents and EnrolledStudents queries in exactly the same way as you currently use the separate tables.  If you are absolutely sure that the new Students table contains the correct data you can delete the original Prospective Students table.

There is probably no strong reason against having Null columns in Students where the values are no longer relevant to enrolled students, but if you want to be relationally purist you could include these columns in a separate table with a StudentID primary key, but not an autonumber in this case, and relate this one-to-one to Students.  In modelling terms what you then have is a Students 'type' with a prospective students 'sub-type'.  On the whole, though, it would be simpler just to accept the Nulls in Students.

ken sheridan say:
Out of curiosity, would there be any disadvantages down the line by not fully normalizing here and just allowing Null values on a master Students table?

It's not actually a question of normalization in this case.  The table would not be fully normalized only if it contains redundancies, i.e. the same 'fact' is repeated in more than one row.  As Scott said an unduly large number of columns in a table often points to this, e.g. a table of locations containing both CityID and State columns would not be normalized to Third Normal Form because the 'fact' that San Francisco is in California could be stated more than once in multiple rows.  This 'fact' should be stated once only, in a Cities table referenced by the locations table.  It gets rather more complex when we get into the higher normal forms, but let's not complicate matters further right now.

The objection put forward by some, particularly Chris Date, to Nulls stems from a basic principle of the database relational model, which is that each column position in a row in a table must contain a legal value of the attribute type which the column represents.  Null is not a value, it is the absence of a value, so a Null at a column position does not satisfy this requirement of the model as seen by those who take Date's position.  On the other hand, it is also widely accepted that a Null at a column position is a valid representation of the value of the attribute type in question being 'unknown' or 'not applicable'.  Codd himself implicitly supported the former use of Null by his proposal of a 3-way logic, True, False and Maybe.  In your case Null would be a representation of 'not applicable' and I think most people would be happy with this.

This is not to say that Null is always acceptable, however.  For many attribute types it would not be appropriate.  What would a Null mean at the CreditLimit column position in a row in a Customers table for instance?  Zero credit? Unlimited credit?  There is just no way of knowing.  Also, Null propagates in arithmetical expressions, so NULL + 1000 = NULL for instance, not 1000 as 0 + 1000 does.  Generally with attributes such as currency Null should be prohibited, and the column be given a DefaultValue property of 0.

There are other pitfalls arising from Nulls which we need to be aware of.  Null cannot be compared with anything else, even with Null as the result is always Null, 'unknown'.  This is not only true of individual comparisons, but of set operations in some cases; for instance the following expression using a subquery:

WHERE SomeField NOT IN(SELECT SomeOtherField FROM SomeOtherTable)

will fail if there is any row in SomeOtherTable with a Null at the SomeOtherField column position.  This is because the above can be expanded to something like this:

WHERE x <> a AND x <> b AND x <> NULL AND x <> c

While three of these non-equality comparisons are TRUE, x <> NULL evaluates to NULL, neither TRUE nor FALSE, so the expression as a whole does not evaluate to TRUE even though x is not represented in the set of SomeOtherField values.  The correct way to do the above  by means of a subquery would be:

WHERE NOT EXISTS(SELECT * FROM SomeOtherTable WHERE SomeOtherTable.SomeOtherField  = SomeTable.SomeOtherField)

So, where Null is concerned we need to proceed with caution, being aware of the implications, but not rule then out completely.  In your case you should be on safe ground.

Enter below formula as an array formula (CTRL+SHIFT+ENTER) in cell E2:

=SUM(--(FREQUENCY(IF(ROW(INDIRECT (MIN ($E$4:$AF$4)&":"&MAX ($E$4:$AF$4)))<=IF(MOD( COLUMN($F$4:$AF$4),2)=0, $F$4:$AF$4,0), IF(ROW(INDIRECT (MIN( $E$4:$AF$4)&":"&MAX( $E$4:$AF$4)))>= IF(MOD( COLUMN( $E$4:$AE$4), 2)=1,$E$4:$AE$4,0), ROW( INDIRECT(MIN( $E$4:$AF$4)&":" &MAX( $E$4:$AF$4))))), ROW( INDIRECT(MIN( $E$4:$AF$4)&":"&MAX( $E$4:$AF$4))))>1))
