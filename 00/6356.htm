dale howard [mvp] say:

Kelly --

Regarding task Duration values, enter a value that you think is the most likely time span it will take to complete the task.  You said you entered 7 days for the Duration of the task to find a cinematographer, but that it could take weeks.  Based on your experience with finding a cinematographer, how long does it TYPICALLY take?  Enter that number as the Duration.  When you save your Baseline for the project, Microsoft Project 2013 will capture that Duration value in the baseline.  Later, when you begin entering progress and you enter items like Actual Start and Actual Finish on that task, the software can tell you whether it took a longer time or shorter time to actually find the cinematographer.

Regarding the task on which you need to work during a given week, click the View tab to display the View ribbon.  In the Data section of the View ribbon, click the Filter pick list and choose the Date Range... item.  In the Date Range dialogs, enter the time period representing the current week and click the OK button.  Microsoft Project 2013 will show you the tasks scheduled during the current week.  Beyond this, only YOU know which tasks are higher priority than others.

Beyond this, when you have tasks in the past that are not completed, you need to reschedule them.  To do this, complete the following steps:

  1. Click the Project tab to display the Project ribbon.
  2. In the Status section of the Project ribbon, click the Status Date pick list button, select the date for FRIDAY OF LAST WEEK, and then click the OK button.  This is the date that tasks from last week SHOULD have been completed.
  3. In the Status section of the Project ribbon, click the Update Project button.
  4. In the Update Project dialog, select the Reschedule Uncompleted Work to Start After option.
  5. Leave the Entire Project option selected and then click the OK button.
  6. If you see a Planning Wizard about constraints, click the OK button.

Microsoft Project 2013 will then reschedule all uncompleted tasks from LAST WEEK into the CURRENT WEEK.  This let's you know what you did not finish last week so that you can work on those tasks this week.

Just a few ideas for you.  Hope this helps.

If I am right, there is still nothing keeping you from dynamically sizing the restored (that is, non-maximized) form to just fit in the application window, as I proposed before, and only then calling the ResizeForm procedure.

To the best of my knowledge, it is not possible to have a maximized form (in the normal Windows sense of the word) without a restore button.  Therefore, you either have to live with the presence of the restore button (but there's a way to reduce its impact -- see below), or else use the code I suggested to make the form fill the application window without actually maximizing it.

For what it's worth, there is a way to effectively keep the restore button from actually restoring the form.  You can use a combination of the form's Resize event and the form's Timer event to keep the form from restoring for more than a flashing instant.  The code would be something like this:

'------ start of code ------
Private Sub Form_Resize()

    If WM_apiIsZoomed(Me.hWnd) = 0 Then
        Me.TimerInterval = 1
    End If

End Sub


Private Sub Form_Timer()
    DoCmd.Maximize
    Me.TimerInterval = 0
End Sub
'------ end of code ------

To use the WM_apiIsZoomed API function that is declared in the code you posted, you would have to change its declaration from:

    Private Declare Function WM_apiIsZoomed Lib "user32.dll" Alias "IsZoomed" _
       (ByVal hWnd As Long) As Long

to:

    Public Declare Function WM_apiIsZoomed Lib "user32.dll" Alias "IsZoomed" _
        (ByVal hWnd As Long) As Long

The effect of the above code would be to watch for the form being restored and instantly -- a millisecond later -- maximize it again.
