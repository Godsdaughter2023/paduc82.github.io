ken sheridan say: It is a basic principal of the database relational model that values which can be time-independently computed are not stored.  There can be very occasional exceptions, but by and large this is the rule.

Where a value is computed from base data that value will always be the result of the expression which undertakes the computation.  The accuracy of the result is dependent on the expression being formulated correctly.  Computation can be within a single row, as for instance with the expression to compute a gross price: UnitPrice * Quantity * (1+TaxRate).  This can only ever produce a value which is correct in terms of the expression, so does not require any validation.

Computation might also be undertaken over a subset of rows, e.g. to return the total gross price per order the summation of the values returned by the above expression might be undertaken:  Sum(UnitPrice * Quantity * (1+TaxRate)).  Again this is guaranteed to be correct in the terms of the expression, so does not require any validation.

However, the result of the above expression could be incorrect in the case in the case of the European Value Added Tax (VAT), because the tax is being applied at order line level, whereas it should be applied at invoice level.  As there are differential rates of VAT this cannot be done a single computation per invoice.  It has to be done for each tax rate, for which the expression is Sum(UnitPrice * Quantity) * (1+TaxRate).  So this expression would be applied to each subset of items per tax rate and the results of each summation would be summed to return the total invoice amount including VAT.

So the result of the first method of computing the total invoice amount could be incorrect in reality, but not in terms of the expression used.  Consequently the fact that the amount is incorrect cannot be validated in the application.  The error can only be corrected by amending the application so that the correct expression is used.

Unfortunately you have not given us enough information about the context of your application to identify what you are doing in real life terms.  All we know is that you appear to be trying to ensure that the sum of the Values column per one  value of the product column is equal to the sum of the values per a subset of other values in the Product column.  On the information available this makes little sense as each column in a table should represent a legitimate attribute of the entity type modelled by the table.  I find it hard to see how your Values column can be doing this.  We need a more detailed explanation of what the Values column represents, and the rationale behind what you are attempting.

In this little demo file the BoM.accdb file uses the classic adjacency table model, in which two columns reference the primary key of a parts table which includes all assemblies, sub-assemblies and base parts.  To generate a bill of materials from such a model requires recursive querying, which JET/ACE SQL as used by Access does not support.  My demo illustrates a means of simulating this.  The number of each sub-assembly or part per assembly is computed, along with the total weight of the assembly.

A bill of materials is not usually a tree structured hierarchy.  Mathematically, a tree is characterized by there being one route only between each node.  A bill of materials, by virtue of the fact that a sub assembly or part will almost always be used in more than one assembly or sub-assembly, has multiple paths between nodes, so. mathematically, is a reconvergent graph.  Where a true tree exists Joe Celko's nested set model is a far more efficient than an adjacency list model.  You'll find a brief description at:

http://www.ibase.ru/files/articles/programming/dbmstrees/sqltrees.html

A detailed description is given in Celko's  book, 'Trees and Hierarchies in SQL', and there is a shorter account in a chapter on the subject in his 'SQL for Smarties'.

Note that a BoM model is not confined to what we normally think of as assemblies, sub-assemblies and base parts in an engineering or manufacturing context.  Many structures, both physical and abstract can be represented by such a model.

/ CheckHealth only tells you whether corruption has already been detected and flagged. It doesn't do a new scan.

/ScanHealth does a scan and reports the results.

/RepairHealth does a scan and then, if necessary, performs repair operations.

If you only want to run one, /RepairHealth is the one. However, the way /RepairHealth reports its results, you can't tell whether it found errors and fixed them, or just didn't find any errors. If you care about the difference, run /ScanHealth first, then run /RepairHealth if necessary, at the cost of a duplicate scan.

The MS documentation is at https://docs.microsoft.com/en-us/windows-hardware/manufacture/desktop/dism-operating-system-package-servicing-command-line-options

/CheckHealth Checks whether the image has been flagged as corrupted by a failed process and whether the corruption can be repaired.
/ScanHealth Scans the image for component store corruption. This operation will take several minutes.
/RestoreHealth Scans the image for component store corruption, and then performs repair operations automatically. This operation will take several minutes.
