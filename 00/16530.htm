Sub newer_method()
    Dim sLbl As String, c As Long, v As Long
    Dim vT As Variant, vCol As Variant
    vT = Array("Inventory", "Misc Item", "Comment", "Inactive", "Open", "Closed")
    vCol = Array(4, 5, 71)
    On Error GoTo FallThrough
    Application.ScreenUpdating = False
    Application.EnableEvents = False
    Application.Calculation = xlCalculationManual
    With ActiveSheet
        With .Cells(1, 1).CurrentRegion
            .AutoFilter
            ' column D, E then BS
            For c = LBound(vCol) To UBound(vCol)
                sLbl = .Cells(1, vCol(c)).Value
                For v = 0 To 2
                    .Cells(1, vCol(c)) = vT(v - ((c > LBound(vCol)) * 3))
                    .AutoFilter Field:=vCol(c), Criteria1:=Format(v, "0")
                    .Cells(1, vCol(c)).Copy
                    .Columns(vCol(c)).SpecialCells(xlCellTypeVisible).PasteSpecial Paste:=xlPasteValues

                Next v

    End With
FallThrough:
    Application.Calculation = xlCalculationAutomatic
    Application.EnableEvents = True
    Application.ScreenUpdating = True
End Sub

macropod say:

There is no such thing as an absolute line number in a Word document and never has been.

As for storing such information in the revision/comment object, anything added to such an object could be invalidated by any subsequent revision situated earlier in the document. Furthermore, due to all manner of formatting possibilities, line #s aren't even necessarily consistent from one page to the next. As for:

"Word stores the TOTAL line count, word count, paragraph count, etc"

some content doesn't even have meaningful line #s (e.g. content in textboxes, headers, footers, endnotes & footnotes). That said, you can retrieve the line # of designated content in the body of a specified page, via .Information(wdFirstCharacterLineNumber).

For a macro to export Track Changes data to an Excel workbook, see:

https://answers.microsoft.com/en-us/office/forum/office_2007-word/possible-to-export-word-track-changes-information/e0dee9dc-aedb-41d3-92bf-8dc609cc75af

The code there already exports page #s for revisions to content in the document's main text story (body). To include the corresponding page line #s, you might change:

StrRev = StrRev & vbCr & "Page: " & .Range.Information(wdActiveEndAdjustedPageNumber) & vbTab & .Author & vbTab & .Date & vbTab

to:

StrRev = StrRev & vbCr & "Page: " & .Range.Information(wdActiveEndAdjustedPageNumber) & "(" & .Range.Information(wdFirstCharacterLineNumber) & ")" & vbTab & .Author & vbTab & .Date & vbTab

macropod say:

Unfortunately,  embedded table rows are counted as line numbers (a table with a row of 9 will add 9 to the line count) but are not annotated as lines.  I can't figure the logic in that one out:  the line numbers show 68-98 (i.e., 30 lines) for the page but the line count returned by the Selection.Range.Information(wdFirstCharacterLineNumber) of the last line is 39 lines.

With Word's line count, when giving you the line count for a document, each table cell counts as 1 line, regardless of how many lines that cell actually spans.

Unlike Word's overall line count, .Information(wdFirstCharacterLineNumber) ignores how many cells a table has - every cell on a given row will return the same line # and every line in a table cell shares that line #. When you have merged cells in a table, .Information(wdFirstCharacterLineNumber) returns the line # for the top-most spanned row for all lines in that cell, but Word's overall line count treats ignores the fact of the merging.

Suppose you have one-page document with a 3-column by 5-row table in it. Word's overall line count will treat the table as representing 15 lines, regardless of whether any cells are merged, but .Information(wdFirstCharacterLineNumber) will only report the table as spanning 5 lines.

Obviously, if you go down this path, you'll also have to work out your own logic for the line numbering order for table cells (i.e. across-then-down vs down-then-across). The code in the link I posted simply reports table content by cell address.

honest abe say:

Hello John,

While we appreciate the recommendations of the MVP's, the suggestions here seem to ignore the fact that some printers do not register the paper properly (no matter what you do).  This is why many accounting programs include a printing alignment feature for checks, so you can make slight adjustments to the printout without changing the check formatting.  Furthermore, an occasional printer allows you to adjust the top/left starting print position for all print jobs, so check your advanced printer preferences or properties, just in case.

In our opinion, Microsoft should provide a similar feature to the accounting programs.  The benefit of a printing adjustment feature is that we can make adjustments for different printers without the need of fiddling with the basic document formatting.  This is good because fiddling with document formatting means it will print well on the current printer, but probably not on other printers.  This is a real problem when documents are being widely distributed within the company.

Over the years we have dealt with this frustrating issue hundreds of times.  As such, the feature for quickly adjusting the top/left print start position, appears to be the most efficient method for correcting small paper registration adjustments on different printers without changing the document formatting.  Furthermore, this concern applies to all the Office Suite applications, not just Word.

Hello Doug,

You are right, in that the problem may be an un-repairable printer problem.  However, we are not suggesting that the software can repair the printer, only that the software can re-align the printout to compensate for the improper paper registration by the printer.

marshall barton say:

Yes, the report's record source query runs again when the report is printed.  So, if you have a parameter that prompts for a value, it will prompt again for printing.

 

Using parameter prompts is a very weak user interface mechanism.  Instead you should use a text box on your form for users to enter the parameter value.  This way, as long as the form is open, the value can be retrieved by the query without any prompts.

 

Since you said you are using a Navigation Form, the control for the query parameter must be on that as a main form.  (Reason: the ONLY subform/subreport in a Navigation that can be open is the one you are looking at.)   

 

In your arrangement, the query parameter would look like:

   Forms!yournavform.thetextbox

If you want to reference the name of a form control in a parameter, you must include the full name of the form; if the control is on a subform, you need the full path from the mainform, to the subform, to the control. A paramter such as [cboYear] will not be recognized as being on a form, and you'll get a prompt.

 

Try - as suggested - [Forms]![NameOfYourForm]![cboYear]

 

using the actual name of your form; if cboYear is on a subform you must use

 

[Forms]![NameOfTheMainForm]![NameOfTheSubform].Form![cboYear]

 

using the name of your main form and the Name property of the Subform Control on the mainform (which may or may not be the same as the name of the form being displayed in that subform control).
