jalong say:

Why won't this left join query return all the rows in the left table?

Using MS-Access 2016. Been dabbling in databases for years but am strictly an amateur. I am probably missing some obvious answer but have not been able to find it with all my "left join only showing one row" searches. Never had this problem in Paradox (DOS)! Here's some sample data that shows where I am stuck.

Table 1 - Customers

CustID* | Name
1       | Jane Doe
2       | Jill Doe
3       | John Doe

Table 2 - Orders

OrderID* | CustID | Order
1        | 1      | Orange
2        | 1      | Banana
3        | 2      | Orange
4        | 2      | Apple
5        | 2      | Banana
6        | 3      | Apple
7        | 3      | Strawberry
8        | 3      | Banana

I want a list of all the customers and I want to see who has had a strawberry and who hasn't. That is, the results should look like:

Name      | Order
Jane Doe  | (null)
Jill Doe  | (null)
John Doe  | Strawberry

The query I am using is:

SELECT Customers.Name, Orders.Order
FROM Customers LEFT JOIN Orders ON Customers.CustID = Orders.CustID
WHERE (((Orders.Order)="Strawberry"));

What it returns is: John Doe | Strawberry. Doesn't a left join, by it's very nature, list all the rows in the left hand table? I am confused.

Could some kind soul please tell me what I am missing? Thanks in advance!

ken sheridan say: If you think about it, a LEFT OUTER JOIN is logically incompatible with the restriction WHERE  Order = "Strawberry".  On the one hand the JOIN is saying 'return all rows from Customers regardless of the existence of a matching row in Orders'.  On the other hand the restriction is saying 'return all rows from Customers only where a row in Orders has a value "Strawberry" at the Order column position'.  You can't have it both ways.

Adding 'OR Order IS NULL' to the WHERE clause as Duane suggests changes the restriction so that the JOIN and the restriction are no longer logically inconsistent, but, because the relationship type is a many-to-many relationship type, you will get unwanted rows returned, e.g. two rows for customer 3 will be returned with NULL at the Order column position, even though they ordered Strawberry, along with multiple rows for the other customers. Even if you introduce the DISTINCT predicate to the SELECT clause you'll still get one unwanted row returned for customer 3.

A solution is a UNION ALL operation:

SELECT Name,Order
FROM Customers INNER JOIN Orders
ON Customers.CustID = Orders.CustID
WHERE Order = "Strawberry"
UNION ALL
SELECT Name, NULL
FROM Customers
WHERE NOT EXISTS
    (SELECT *
     FROM Orders
     WHERE Orders.CustID = Customers.CustID
     AND Order = "Strawberry");

The first part of the UNION ALL operation will restrict the result set the those customers who ordered Strawberry, while the second part is restricted by the subquery to those customers who did not order Strawberry.

BTW I would strongly recommend that you change the column name Name to CustomerName.  Name is the name of a built in property, and as a 'reserved' word, should not be used as an object name.  For the same reason I'd recommend changing Order to ItemOrdered or similar; ORDER is an SQL keyword.

Is there a reason why a style won't apply and, instead, create a - Microsoft Community

markd731 say:

Is there a reason why a style won't apply and, instead, create a new one?

I have been working in Word creating a document. When I try to apply a style (e.g., Heading 3), it instead creates a new Emphasis style that appears to look like the Heading 3, but is still Emphasis (e.g., Emphasis, 14pt, bold...). This wouldn't be an issue except that this new Emphasis style doesn't appear in the document's table of contents.

 

Unfortunately, any formatting I do to remove these new styles seems to not hold after I save, close the document, and come back to it later. I've found that what was once a Heading 3 has returned to the Emphasis, 14pt, bold style later on.

 

Is there a reason why the program is doing this? Is there a better way to work around it?

 

Thanks!

suzanne s. barnhill say:

Heading 3 is a paragraph style. Emphasis is a character style. When a character style is applied to a paragraph, it is the style that is displayed, not the underlying paragraph style. If you want the heading to be bold and 14 pts., then you should modify the Heading 3 style. If you want that formatting to be applied to just part of a Heading 3 and to show up in the TOC, then you can add it on top of the paragraph style.

 

But note that the default paragraph font formatting of a heading is never reflected in the TOC; only manually added ("direct") font formatting is reflected. If you want your TOC entries to be formatted the same as the headings, you need to modify the TOC 3 style.

suzanne s. barnhill say:

I think what you're seeing is the result of the "Define styles based on your formatting" setting in AutoFormat As You Type. What this option really means is "Apply styles that resemble your formatting." If you modify a paragraph in a given style, then Word applies a style that has the formatting you applied. While that may not exactly apply to your case, I suspect that setting is still somehow the cause. Go to File | Options | Proofing | AutoCorrect Options | AutoFormat As You Type and clear that check box (you may see others you will want to clear as well--see http://word.mvps.org/FAQs/Formatting/TameAutoFormat.htm).

 

OTOH, if you have opted to display some kinds of formatting as a style (Options... in the Styles pane), then what is displayed is the formatting applied rather than the style. Again, if all you're really doing is just applying a style, without direct formatting, it's hard to see how this would be relevant, but it might be.
