This little demo file includes two options for using multiple multi-select list boxes, one to 'Filter Report on Multiple values from Fields, combined with AND' the other to 'Filter Report on Multiple values from Fields, combined with OR'.  From your description I suspect the first is the appropriate example for what you want to do.  The form in question uses two list boxes, but it would be a simple task to add more and extend the code accordingly.  The code behind the button to open the report after making the selections is as follows:

Private Sub cmdOpenReport_Click()

    Dim varItem As Variant
    Dim strEmployeeIDList As String
    Dim strProjectIDList As String
    Dim strCriteria As String
    Dim ctrl As Control
    
    Set ctrl = Me.lstEmployees
    
    If ctrl.ItemsSelected.Count > 0 Then
        For Each varItem In ctrl.ItemsSelected
            strEmployeeIDList = strEmployeeIDList & "," & ctrl.ItemData(varItem)
        Next varItem
        
        ' remove leading comma
        strEmployeeIDList = Mid(strEmployeeIDList, 2)
        
        strCriteria = " And EmployeeID IN(" & strEmployeeIDList & ")"
        
    End If
    
    Set ctrl = Me.lstProjects
    
    If ctrl.ItemsSelected.Count > 0 Then
        For Each varItem In ctrl.ItemsSelected
            strProjectIDList = strProjectIDList & "," & ctrl.ItemData(varItem)
        Next varItem
        
        ' remove leading comma
        strProjectIDList = Mid(strProjectIDList, 2)
        
        strCriteria = strCriteria & " And EmployeeID IN" & _
            "(SELECT EmployeeID FROM ProjectEmployees " & _
            "WHERE ProjectID IN (" & strProjectIDList & "))"
        
    End If
    
    If Len(strCriteria) > 0 Then
        ' remove leading ' and '
        strCriteria = Mid(strCriteria, 6)
        
        DoCmd.OpenReport "rptEmployees", _
            View:=acViewPreview, _
            WhereCondition:=strCriteria
    Else
        MsgBox "No employee(s) or project(s) selected.", vbExclamation, "Warning"
    End If
    
End Sub

The first thing to understand is that the report is based on a query which joins the Employees table to the ProjectEmployees and Projects tables.  To filter the results the report must be filtered by the key EmployeeID column.  This is simple to do when filtering to return one or more selected employees, and is done by the following part of the code:

    If ctrl.ItemsSelected.Count > 0 Then
        For Each varItem In ctrl.ItemsSelected
            strEmployeeIDList = strEmployeeIDList & "," & ctrl.ItemData(varItem)
        Next varItem
        
        ' remove leading comma
        strEmployeeIDList = Mid(strEmployeeIDList, 2)
        
        strCriteria = " And EmployeeID IN(" & strEmployeeIDList & ")"
        
    End If

What this does is loop through the items selected and build a value list of the selected EmployeeID values.  The IN operator is then applied to this value list.

When restricting the report's results by project, however, it is not simply a case of filtering by the selected ProjectID values, but by those employees who have been assigned to one or more of the selected projects.  This is done by the following part of the code:

    If ctrl.ItemsSelected.Count > 0 Then
        For Each varItem In ctrl.ItemsSelected
            strProjectIDList = strProjectIDList & "," & ctrl.ItemData(varItem)
        Next varItem
        
        ' remove leading comma
        strProjectIDList = Mid(strProjectIDList, 2)
        
        strCriteria = strCriteria & " And EmployeeID IN" & _
            "(SELECT EmployeeID FROM ProjectEmployees " & _
            "WHERE ProjectID IN (" & strProjectIDList & "))"
        
    End If

Again the code loops through the selected projects and builds a value list, but in this case the IN operator is applied to the value list within a subquery, and the report is filtered to those employees whose EmployeeID is at  least one value returned by the subquery.

To tack the restrictions on both employee and project together the AND operator is used when building the combined string expression for the filter.  A user might have selected from only one of the list boxes, however, in which case the AND operator is not needed and would cause an error.  The way this is handled is to begin each of the two string expressions with 'AND' before concatenating them together.  This leaves a surplus 'AND' at the start of the combined expression of course, so this is removed with:

        ' remove leading ' and '
        strCriteria = Mid(strCriteria, 6)

The option for using OR is the same , but uses the OR operator in place of the AND operator.

There is another option in the demo, to 'Filter Report on Multiple values from Fields, combined with AND
Where Employee Is Assigned to All Selected Projects'.  Whereas the above returns rows for any selected employee and/or any of the projects, this other option requires all projects selected to be assigned to the selected empoyee. This is done in a subquery query to which the IN operator is applied, but instead of a value list the subquery references a query, qryEmployeeExtendedDlg_BOTH.  It is in this query that the hard work is done.  The code to open the report is:

Private Sub cmdOpenReport_Click()

    Dim strCriteria As String
    
    strCriteria = "EmployeeID IN" & _
            "(SELECT EmployeeID FROM qryEmployeeExtendedDlg_BOTH)"
        
    DoCmd.OpenReport "rptEmployees", _
            View:=acViewPreview, _
            WhereCondition:=strCriteria
            
End Sub

The qryEmployeeExtendedDlg_BOTH query which is referenced is:

SELECT Employees.EmployeeID
FROM Employees INNER JOIN ProjectEmployees
ON Employees.EmployeeID=ProjectEmployees.EmployeeID
WHERE

InParam(Employees.EmployeeID,Forms!frmEmployeeExtendedDlg_BOTH!txtEmployeeIDList)
AND
InParam(ProjectID,Forms!frmEmployeeExtendedDlg_BOTH!txtProjectIDList)
GROUP BY Employees.EmployeeID
HAVING COUNT(*)=Forms!frmEmployeeExtendedDlg_BOTH!lstProjects.ItemsSelected.Count;

This query is probably rather more complex than those with which you are familiar.  Firstly it calls an InParam function, which is not built into Access, but published by Microsoft on-line.  The function, and a GetToken function which is called by the InParam function, are in my demo database in the module basInParam, so all you'd need to do would be to copy this module into your database.  What the InParam function does is to simulate the use of the IN operator, but with a parameter as its value list.  The IN operator itself can't be used because it only accepts a literal value list as its argument.

The main thing to note in this query is how it restricts the results to rows which match all the selections by means of its HAVING clause, which only evaluates to TRUE if the number of rows returned per employee is the same as the number of rows selected in the projects list box in the form, which means that only those employees selected who have been assigned to all of the projects selected will be returned in the report.

I appreciate that you have limited experience of Access, and the above might therefore be difficult to get your teeth around, but it would be wrong to pretend what you are attempting is simple.  The first two options are not too difficult to understand, and by replicating my code carefully in the context of your own application, hopefully you'll be able to implement a solution.  The last option is more difficult to grasp without a reasonable level of experience, however, and would be even more complex in the context of more than two list boxes (separate queries would be needed for each, and the number of rows returned per employee by each would have to match the number of rows selected in the relevant combo box).  At the end of the day it is really a question of what Richard Feynman said to his students about not writing things down before understanding them in the quote below my signature; the ability to transfer a more than trivial solution to another context requires an understanding of the solution.
