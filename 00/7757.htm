ken sheridan say: You firstly need to ask yourself whether the value at the column position in the table should zero or a Null (not really a value at all, but the absence of a value).

Null is semantically ambiguous, so can be misleading.  This is particularly the case with a column of currency data type for instance.  What would a Null means in a CreditLimit column for instance?  Is the customer afforded zero credit, unlimited credit, or what?  It's impossible to say.  Normally columns of currency data type should have a default value of zero.

On the other hand zero, as a value, might be inappropriate in other cases.  If whatever attribute of the entity type modelled by a table, which a column is representing is for some reason inappropriate in some rows, or is simply currently unknown, then a default value of zero would be misleading as it would be a categorical statement of a value of zero for that attribute.  In a case like this it is appropriate for the column position to be Null.

Should you decide that a Null rather than a zero is appropriate at the column position in the table, then you can update the table with an update query as follows:

UPDATE SomeTable
SET SomeColumn = NULL
WHERE SomeColumn = 0;

If on the other hand you decide that zeros are appropriate in the table then you can suppress the zeros in the report with an expression such as this as the control's ControlSource property:

=IIF([SomeColumn] = 0, Null, [SomeColumn])

or you can do the same in a computed column in the report's query and bind a control in the report to the column.  Formatting a control in the way Mimmo suggests would hide the value, but could lead to incorrect results if the data is being aggregated.  The COUNT operator will exclude Nulls for instance, but would count a hidden value of zero.

One other thing you should be aware of is that Null propagates in arithmetical operations, so the result of any operation which includes a Null will be Null, e.g.

2+Null = Null

You can avoid this by calling the Nz function to return a zero in place of a Null:

2+Nz(Null,0) = 2

marshall barton say: Following on from Ken's excellent explanation OR you can ignore all our advice about what should or should not be stored in the table's field and just set the table field's, query field's, form text box's or the report's text box's Format property to a custom format something like:   0;-0;"";"" if the field's data length is Integer or Long,  0.00;-0.00;"";"" for Single or Double, etc. or some minor variations of those.

scottgem say:
Hi couture,

You do not need to pull data from multi-tables in your subform. You can use comboboxes to display the related value. In a combobox you can set the width for each column in the Rowsource. The combobox will display the first non zero width column, but use the data from the bound column.

So, using your PersonID as an example. you would add a combobox bound to the PersonID with a Rowsource of

SELECT PersonID, PersonName FROM Persons ORDER BY Personname;

with a Bound column of 1 and Column Widths of 0";2". The combobox wizard will walk you through setting up such a combo.

You would then do the same thing with the RoleTypeID. Value lists should only be used for combo where there is a short, static list. I would not use a Value list for Roletype, especially since you have a RoleType table.

ken sheridan say: The subform's RecordSource should be a query which joins TitlePersonnel and Persons.  We'll come back to the need for Persons in the RecordSource later.  The LinkMasterFields and LinkChildFields properties of the subform control should both be TitleID.

You need two correlated combo boxes in the subform, bound to PersonID and RoleTypeID respectively.  How you correlate them depends on whether you wish to select a role type first or a person first.  For the purposes of the following example I'll assume the former.

The RoleTypeID combo box's RowSource should be:

SELECT RoleTypeID, RoleType
FROM RoleType
ORDER BY RoleType;

It's BoundColumn property should be 1, its ColumnCount 2 and its ColumnWidths 0cm (or inches)

The PersonID combo box's RowSource property should be:

SELECT Persions.PersonID, PersonName
FROM Persons INNER JOIN Person_RoleTypes
ON Persons.PersonID = Person_RoleTypes.PersonID
WHERE RoleTypeID = Form!cboRoleType
ORDER BY PersonName

where cboRoleType is the name of the combo box bound to the RoleTypeID column.  Note the use of the Form property to reference the control, rather than a fully qualified reference.  In the AfterUpdate event procedure of that combo box, empty and requery the other combo box, e.g.

Me.cboPerson = Null
Me.cboPerson.Requery

In the subform's Current event procedure requery the combo box, e.g.

Me.cboPerson.Requery

However!  Using correlated combo boxes in continuous forms view where the bound column of the second control is a hidden 'surrogate' key will not work on its own.  The control will appear empty in rows other than the current one.  No data will be lost, merely hidden.  The solution is to use a hybrid control in which a text box bound to the PersonName column from Persons is carefully superimposed on the combo box, leaving only its drop down arrow visible, to give the appearance of a single combo box control.

You'll find examples of the use of hybrid controls in this way in ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

The example to 'Open Form with Simple 2-level Hierarchy in Continuous Forms View Using Hybrid Controls' is the appropriate model to follow.  The form in the demo is not a subform, but that is immaterial in this context.

rob koch say:

Since the creation of the Windows firewall settings for Public and Private networks, the need to individually close ports in order to protect Windows from attack has largely disappeared.

What’s the Difference Between Private and Public Networks in Windows

Back in the old days of Windows XP and even to a small extent Vista, since its network configuration was less seamless than today's, the more simplistic firewall configuration often required some amount of tweaking.

However, with the more effective and easy to use firewall that's developed since Vista and Windows 7, there's little exposed by default even within a Private Home/Work network and absolutely nothing when using the Public setting typically used for travel.

So unless you have a unique need to configure specialized settings in Private network setting where there's little trust, I wouldn't even consider bothering with such outdated concepts.  Instead, take the time to learn how to properly use and configure the modern firewall that already exists in Windows 10 before messing up such an effective design.
