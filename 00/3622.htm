hansv mvp say:

sh.Cells is the collection of all cells on the worksheet specified by sh.

Please note that all the lines from

 

    LastRow = sh.Cells.Find(What:="*", _

 

up to and including

 

    MatchCase:=False).Row

 

are connected by continuation characters at the end of the lines - they form one single instruction.

 

To find the last used row in the sheet, we are going to look backwards (SearchDirection:=xlPrevious) through the rows of the sheet (SearchOrder:=xlByRows) until we find any content (What:="*").

If we would look only at the cell's values (LookIn:=xlValues), we would miss cells that have a formula resulting in an empty string. To incluse such cells, we look at the cell formulas (LookIn:=xlFormulas).

 

sh.Cells.Find(What:=..., ..., MatchCase:=False) returns a Range object representing the result of the Find action: it's a cell in the last used row (since we searched backwards through the rows).

 

sh.Cells.Find(What:=..., ..., MatchCase:=False).Row is the row number of that cell, i.e. the number of the last used row in the sheet.

 

Macros execute much more efficiently if Excel doesn't update the screen contents while the macro is busy. So it's usually a good idea to have a line

 

    Application.ScreenUpdating = False

 

somewhere near the beginning of a macro, and

 

    Application.ScreenUpdating = True

 

near the end.

 

If your workbook has so-called event handling code that reacts to changes in the workbook, changing a cell using VBA may cause a cascade of calls to the event handling code. In most situations, that is not desirable. To prevent such a cascade, we place a line

 

    Application.EnableEvents = False

 

near the beginning, and (very important!)

 

    Application.EnableEvents = True

 

near the end of the code.

 

The line

 

    DestSh.Cells(Last + 1, "H").Resize(CopyRng.Rows.Count).Value = sh.Name

 

copies the name of the worksheet sh into the cells in column H in the destination sheet corresponding to the range that has just been copied into it from sheet sh.

The range that was copied is CopyRng.

This range was pasted to DestSh.Cells(Last + 1, "A"). The cell DestSh.Cells(Last + 1, "H") is the corresponding cell in column H.

The copied/pasted range contains CopyRng.Rows.Count rows. We want the name of sheet sh in all those rows. DestSh.Cells(Last + 1, "H").Resize(CopyRng.Rows.Count) expands the cell DestSh.Cells(Last + 1, "H") to that number of rows. We then set the value of the expanded range to sh.Name, the name of the source sheet.

Word 2013 Opening multiple documents in one window? - Microsoft Community

empathydragon say:

Word 2013 Opening multiple documents in one window?

I have scrounged through the options settings of Word 2013 and cannot find the the option 'Show all windows in the task bar'! In all the other versions of word this option has always been available with the Display section for me to uncheck as I prefer to open all documents I am using within the same window.

Am I looking in the wrong place!
jay freedman say:
Empathydragon is correct, that feature has been removed in Office 2013.

It isn't really a question of what appears on the taskbar, but rather the behavior of the windows that hold the documents. (Thanks to MS for mislabeling the option and creating the confusion!)

In Word XP, 2003, 2007, and 2010, when the option "Windows in Taskbar" or "Show all windows in taskbar"  is checked and you open multiple documents, each document gets its own set of menus or ribbons and behaves as a completely separate window. (The technical term is Single Document Interface or SDI.)

In those versions, when that option is unchecked and you open multiple documents, there is only one set of menus or ribbons. You can tile or cascade the documents within the main Word window. (The term for this is Multiple Document Interface or MDI.) The fact that there's only one icon on the taskbar is really a side effect, but it was easier to dumb down the option name than to explain the difference between SDI and MDI.

In Word 2013, MDI no longer exists, and the option doesn't appear in the dialog.

The interesting part is that Microsoft dropped MDI from Word 2000, and the outcry was so loud that they had to put it back in Word XP (a.k.a. Word 2002).
