Here is a modified version. I added error handling.

Sub LoopAllExcelFilesInFolderDeleteSheet()
    'PURPOSE: To loop through all Excel files in a user specified folder and perform a set task on them
    'SOURCE: www.TheSpreadsheetGuru.com
 
    Dim myPath As String
    Dim myExtension As String
    Dim myFile As String
    Dim wb As Workbook
    Dim ws As Worksheet
 
    'Retrieve Target Folder Path From User
    With Application.FileDialog(msoFileDialogFolderPicker)
        .Title = "Select A Target Folder"
        If .Show Then
            myPath = .SelectedItems(1)
        Else
            Exit Sub
        End If
    End With
 
    If Right(myPath, 1) <> "\" Then
        myPath = myPath & "\"
    End If
 
    On Error GoTo ErrHandler
 
    'Optimize Macro Speed
    Application.ScreenUpdating = False
    Application.EnableEvents = False
    Application.Calculation = xlCalculationManual
 
    'Target File Extension (must include wildcard "*")
    myExtension = "*.xls"
 
    'Target Path with Ending Extension
    myFile = Dir(myPath & myExtension)
 
    'Loop through each Excel file in folder
    Do While myFile <> ""
        'Set variable equal to opened workbook
        Set wb = Workbooks.Open(Filename:=myPath & myFile)
 
        'Test whether sheet exists
        Set ws = Nothing
        On Error Resume Next
        Set ws = wb.Worksheets("Week 2 Life & Invest Commission")
        On Error GoTo ErrHandler
 
        If ws Is Nothing Then
            'Sheet doesn't exist - close workbook without saving it
            wb.Close SaveChanges:=False
        Else
            'Delete sheet without displaying a warning
            Application.DisplayAlerts = False
            ws.Delete
            Application.DisplayAlerts = True
            'Save and Close Workbook
            wb.Close SaveChanges:=True
        End If
 
        'Get next file name
        myFile = Dir
    Loop
 
    'Message Box when tasks are completed
    MsgBox "Task Complete!", vbInformation
 
ResetSettings:
    'Reset Macro Optimization Settings
    Application.EnableEvents = True
    Application.Calculation = xlCalculationAutomatic
    Application.ScreenUpdating = True
 
ErrHandler:
    MsgBox Err.Description, vbExclamation
    Resume ResetSettings
End Sub

rob koch say:

The Indian Oil e-Tenders website that has the invalid certificate error mentioned in the other thread was most likely the reason for the Avast! threat detection.

Figuring out why this was happening is well beyond most users and is why the original poster in the other thread was having so much difficulty understanding why it wasn't working with his Internet Explorer 11.  However, I'm not certain how he had it working with IE 10 either.

The core problem is that the certificate for this website is installed incorrectly and I'm not even certain it's the proper type of SSL certificate for a website in the first place.  Though the CA (certificate Authority) being used is supposedly a valid authority, they not only don't provide a clear listing of how their certificates can be used, but they definitely aren't a member of the Microsoft SSL Root Certificate Program.

http://social.technet.microsoft.com/wiki/contents/articles/14215.windows-and-windows-phone-8-ssl-root-certificate-program-member-cas.aspx

All of this and the fact that the domain for that e-Tenders website ends in ".gov.in" seem very suspicious, since this would be a typical method used by someone phishing for user logins with a fake website.

However, some additional research shows that the valid Indian Oil Tenders website with a valid SSL certificate contains the following PDF notice which not only references the e-Tenders website with the incorrectly configured certificate, but in fact discusses its purpose as well.

https://www.indianoiltenders.com/pipeline_noida.pdf

This notice also mentions that vendors wishing to submit bids via the e-Tendering website must acquire their own digital certificates in order to sign (and possibly encrypt?) the documents.  However, since these certificates can only typically be used for document signing, they should have nothing to do with the website access, so this doesn't explain that separate issue.

I would guess that the Avast! protection was simply an enhanced version (browser read-ahead?) of the SSL certificate validation that browsers do themselves and was detecting the improperly set up certificate chain, resulting in an error that generated the warning.  Since there's nothing apparently malicious at that site they must have removed the block after verifying this information themselves.
