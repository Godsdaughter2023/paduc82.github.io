karl timmermans [outlook mvp 2012-2018] say:

First step to success requires an understanding of the terminology (and items) used in Outlook

#1 - Outlook profiles are not contained in Account Settings. The Acct Settings window (multiple tabs) represents the information for the currently active "Outlook Profile"


#2 - To review your profile(s) - use the <Mail Applet> - in Win7 that is accessed by

Control Panel --> User Accounts & Family Safety --> Mail applet

#3 - An Email Account is not a profile. A single profile can include <multiple email accounts> which in turn may or may not encompass <multiple> data files (or none) depending on your email accounts and what has been set as the default delivery location for each. Some email accounts will <not> have any specific data file associated with it - such as a Hotmail account which will show <Not Available> under the File Name when looking at the Data Files tab in Account Settings.

#4 - When you start Outlook for the very first time, you are asked all kinds of questions about setting up your email account(s). When done, you have in reality created your first Outlook profile which in a new system would have the name of <Outlook>. That gets back to the point Russ was making - you can't run Outlook without at least one profile having been created.

The following article should provide with the required information in terms of goes into creating an <Outlook Profile>

How to create and configure an email profile in Outlook 2010, Outlook 2007, and Outlook 2003
http://support.microsoft.com/kb/829918

#5 - AddressBook is part of the Profile - if you want to start from scratch, create a new Profile and make sure that the AddressBook has been added to it

#6 - Finally, if a contact folder was added to the AB (ergo - the contact folder was set to "Show as an email Address Book" - right click on a contact folder, properties, <Outlook Address Book> tab to check the setting) AND the contact folder was subsequently deleted, it does not automatically get removed from the AB - you will have to manually remove it. Deleting a contact folder is not the same thing as "unchecking" the "Show as an email Address Book" option which does remove it from the AB list.

ken sheridan say: Actually, for this sort of concatenation you need to use a combination of the + sign and the ampersand.  In arithmetical operations Null propagates, so Null + "foo" evaluates to Null.  In concatenation operations it doesn't so, so Null & "bar" evaluates to "bar".  This behaviour can be usefully exploited to suppress redundant spaces, e.g.  

Firstname & (" " + MiddleName) &  " " & Lastname

If the middle name is missing you won't get the unnecessary extra space between the first and last names which would result if the ampersand had been used throughout.

However, if one of the values being concatenated is a number data type, then this will produce an error if you try and add a string expression, e.g. a space, to it, so you have to do a bit more work.  This may be what's happening if one of your fields is a 'lookup field' as John suggested might be the case, as its actual value will be numeric.  You can see what happens by entering a few things in the debug window;

var = 1
? " " + var
...results in a type mismatch

? " " + Cstr(var)
 1
...this works

var = Null
? " " + var
Null
...this works

but
? " " + Cstr(var)
....results in invalid use of Null error

so to work around it:
? " " + IIf(IsNull(var),Null,Var & "")
Null
... this works

var = 1
? " " + IIf(IsNull(var),Null,Var & "")
 1
....and so does this

Hopefully this will enable you to come up with appropriate expressions in your form.  

If you want to explore concatenation further, take a look at the demo file Concat.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

 BTW for reasons why not to use the lookup field wizard when designing a table see:

http://www.mvps.org/access/lookupfields.htm

Don't worry too much if you have used this feature however.  Base your form on a query which joins the current table to the relevant referenced table and return the 'looked up 'text field from that table.  the use this when concatenating the values for the address rather than the foreign key column in the current table.

Custom theme, color, and fonts on the Mac?

bob jones aka: cybertaz say:

No, implementation in Mac Office has not advanced to the same level.

What you can do:

  • In Excel/Word/PPT You can save custom Themes but you are limited to using the available built-in Theme Fonts schemes & built-in/custom Theme Colors schemes
  • Theme Colors can be customize only in PPT but then are available in the other apps

What you can't do:

  • Theme Fonts cannot be customized
  • There is no access to Theme Effects

However, if you create custom Themes on a PC they will be recognized in documents brought to a Mac. The Theme can then be saved as a Custom Theme there.

Also -- assuming proficiency with XML -- you can create your own Theme Fonts & Theme Colors files.
