ken sheridan say: Firstly, check the data type of the Broker Shop column (field) in the table into which you are entering data via the form.  If it is a number data type, which I'd expect, and which would be the case if you used the lookup field wizard when creating the column in table design view, then the column will hold the value of the corresponding ID primary key column in the Broker table.

Assuming the column is a number data type, the combo box's RowSource property should be:

SELECT ID, Phone, Email, BrokerShop FROM Broker ORDER BY BrokerShop;

It's other relevant properties should be:

Name:                  cboBroker
BoundColumn:    1
ColumnCount:     4
ColumnWidths:   0cm;0cm;0cm;8cm;

If you are using imperial rather than metric units Access will convert the dimensions to inches.

To show the Phone and Email; values in two unbound text boxes in the form use the following two expressions as their ControlSource properties:

=cboBroker.Column(1)
=cboBroker.Column(2)

The Column property is zero-based, so these reference the second and third columns respectively.

ken sheridan say:

However, on my table where I'm entering the data via the form, is returning only the ID number from the Broker table. 

That's exactly what it should be doing.  A referencing table should store only the foreign key, which, where the primary key of the referenced table is a number, will also be a number.  To duplicate the values from the referenced table's non-key columns in columns in the referencing table would introduce redundancy due to the transitive dependency of those columns on the key, and the table would not be normalized to Third Normal Form (3NF), with the consequent risk of update anomalies.

You have seen how the data can be shown in a form.  In other contexts, e.g. a report's RecordSource the data should be returned in a query which joins the tables, and includes the non-key columns from the referenced table.

samuelsantana1 say: So, Are you saying that I need to create a query where I have the information of the both tables in one place and use the form to update the query?

ken sheridan say:

So, Are you saying that I need to create a query where I have the information of the both tables in one place and use the form to update the query?

No.  The form updates the referencing table to which it is bound.  By virtue of the foreign key value inserted into the referencing table, any query which joins the tables on the keys will automatically return the correct values from the non-key columns of the referenced table.

Take a look at DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file, if you go to the section on 'retrieving data from the database' you'll find a form in which rows can be returned on the basis of unbound correlated combo boxes in the form header.  The unbound combo boxes are not relevant to this thread, but if you look at the form's RecordSource property you'll see that it is a query which joins four tables, and returns columns from each of them.  The tables include a foreign key columns which reference the primary key of one of the other tables.  By joining the tables on the keys data from non-key columns in each is returned in a single result table, which removes the need for invalid duplication of data, and thus eliminates any redundancy.

ken sheridan say:

One-to-one relationship types are used to model a type hierarchy, where an entity can be of two or more entity types.  This is not the case here. Consequently, no purpose is served by splitting the data over three tables.  A single table, with Student_ID as its primary key would suffice.  The Student_Roll_No and Fee_Challan_SNo columns can each be indexed uniquely in this single table.

 

While this would make the insertion of a row by means of a single form a simple task, it does of course mean, as does your current model, that there can be only one row of school record If you wanted to have one or more rows of school record and/or fee data per student, e.g. for each academic year, then it would be correct to have three separate tables but the relationship types would be one-to-many, not one-to-one.

 

For an example of a true type hierarchy and how to model it you might like to take a look at TypeHierarchy.zip in my public databases folder at:

 

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

 

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly. 

 

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

ken sheridan say: Unfortunately 'IT guys' don't always know a lot about relational database design.

If you are unable to alter the design of the present tables a solution would be to create a form based on Student_Data, and within it embed a subform based on Student_School_Record.  In this subform embed another subform based on the Student_Fee table.  The parent form and the subforms should all be in single form view.

The two subforms should be linked by setting the LinkMasterFields and LinkChildFields properties of each respectively as below:

Student_ID                      Student_ID
Student_Record_ID         Student_Record_ID

You'll then be able to insert data into each table via its subform.  If you go to an empty new record in the parent form with your button, each subform will automatically go to a new record.

It is possible that the problem is in the document, so try to print from a different document to test that idea.

1.- Create a new file in the Office program (Word, Excel, PowerPoint, etc.) in which you have problems to print. To do this, click on file> new

2.- Write a few random text words

3.- Try printing the document

NOTE: If it prints correctly, this indicates that the problem is in the existing document. See troubleshoot damaged documents in Word for Mac.

4.- Try to print from another program.

5.- Remove and reinstall the print driver files.

6.- Replace the printer preference files.
