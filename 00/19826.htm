fl-jim say:

I have a Treo 700W running Vista Home Premium & Outlook 2007 on a laptop. I am running Windows Mobile Device Center and sync using Bluetooth. I have learned a few things that help with the differences between ActiveSync and WMDC. I think these differences would apply no matter how you synced your devices (USB cable, etc). You must initiate action on BOTH the PC AND the Treo in order for things to work properly. Here is what works smoothly for me:

1. If you are going to manage files and options on the Treo - Start WMDC on the PC. If you don't need to manage files on the Treo, skip this step.

2. Start SYNC on the Treo. Click "Menu" and "Connect via ...." (I click "Connect Via Bluetooth"). You will see it start up and go thru the steps in connecting. When the connection is complete you will get a "Connected" message on the Treo. My Treo vibrates when the connection is complete. If you are managing files/options on the Treo, this is the place to do it on the PC.

3. After you get the "Connected" message (and do any file/option management), click "Sync" on the Treo. It takes a little while to get ready to sync, but it goes pretty quick when all the linkage is complete. It vibrates again when sync'ed.

4. When sync is complete the Treo will display a message "Synchronized: Today, hh:mm".

5. After you get the "Synchronized" message, click "Menu" and "Disconnect".

 

Hope this helps.

Back end on a MPLS connection - Microsoft Community

self untaught say:

Back end on a MPLS connection

Is it recommended to run MSAccess 2010, 2013 or 2016 with each front end running local and the shared back end on a 5MB MPLS  connection?

john w. vinson say:

No. NO. and NO. 

Access is VERY demanding of bandwidth and of a continuously open, two-way, rock stable connection. It does NOT "play nice" over the Internet, or anything other than a fast, stable local LAN. Most local LANs are now 100Mbps, 20 times faster than what you describe; and any Internet connection will have too many drops and retransmissions for Access; you'll get bad performance, lockups, and frequent database corruption.

You'll need either to have your backend in SQL/Server (and the frontends adapted for that use) or use remote terminal connections with the frontends in private directories on the same server (or at least the same fast stable LAN) as the backend.

First, If you are supporting this app, then it needs to be split into front end and backend. The backend contains just the tables and the front end everything else. This will make it easier to do maintenance without interfering with the user's data.

Second, if the user can install the app in any folder, then you need a system for checking that. I would create a function that e-mails you the location of the files and other system info. You can use CurrentProject.Filepath to return the path of the file. And place that, and other info in an e-mail to you. Add a button to your main menu to e-mail system info.
