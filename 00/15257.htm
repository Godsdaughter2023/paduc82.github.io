alanwu_msft say:

Hi Vikas,

A rule works while the email is on the way of mail flow. In other words, when an email is receives, the rule stops working.

Given the situation, it is not feasible to use a rule to deliver the email to another recipient (in this case, your supervisor) after the emails is received in 3 days.

However, we can set up a “transport rule” to Bcc the email to your supervisor while the email is sent out. This method is to ensure your supervisor can check the email even if the recipient hasn’t replied to the email. The detailed steps are:
1. Sign in to Exchange Admin Center (EAC) with an admin credential.
2. Click “mail flow”> “rules”> “+” (create a new transport rule)> “Apply this rule if (the sender is this person)”> choose the sender> “Do the following (Bcc the message to)”> type your supervisor’s email address> “OK”.
Note: The above method applies to the situation that the sender is the internal user. If the sender is not an internal user, you can change “the sender is the person” to be “the sender is located (outside your organization)”. Also, the above steps should be performed by an admin of your organization. If you are not an admin, you can ask for help from your admin or involve your admin in the post.

Lookups are great, but they should be done on the FORM level, not the table level.

So you apparently have lookup tables for street names and cities. So what you need to do is include those tables in your query and use the field containing the names in your expression.

Another thing I would recommend is that you name your foreign keys the same as the primary key they connect to. This way its clear they are a foreign key and you know to include the lookup table. I.E StreetNameID or CityID.

ken sheridan say: The other point about your tables, and arguably the most important one, is that neither the tblEstablishmentDetail nor tblOwnerDetail table is normalized to third Normal Form (3NF).  The formal definition is:

Third Normal Form: A relvar is in 3NF if and only if it is in 2NF and every non-key attribute is
nontransitively dependent on the primary key.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, and an attribute to a column (field).

What this means in plain English is that if a value in a column is implied by a value in another column, which is not the primary key, then that column is redundant and the table is open to the risk of update anomalies, resulting in bad data.  In your case the street name column is correctly determined solely by the key, but the city column is not, as this is determined by the street.  Note that we are talking not about the name of the street here as these can legitimately be duplicated in different locations, but the street as an entity.  The street names table can therefore have multiple rows with the same street name, but a different primary key, StreetID say, and a different foreign key, CityID say.

Similarly a cities table can have multiple rows with the same city name, as these are legitimately duplicable, but a different primary key CityID and a different foreign key StateID referencing the primary key of a states table.

In the tblEstablishmentDetail and tblOwnerDetail you then don't need the city or state columns as, once you know the street, these are automatically known via the relationships and can be shown in controls in a form, either as read-only bound controls by basing the form on a query which includes the referenced city and state table, or as unbound controls which reference columns in the RowSource of a combo box bound to the StreetID column.

Zip codes may be a different matter.  In the UK one street can have more than one post code (my street has two, one for each side of the street).  I'm afraid I don't know enough about zip codes to know whether this is also true in the USA.  If not, and each street has one zip code only, then the zip code column is also redundant as it is determined by the street.  Each row in the zip codes table would have a column for the zip code, which can then be shown in a form in the same way as the city and state.  In the UK the ideal way is to have a further table which models the many-to-many relationship type between streets and postal codes, whose two-column primary key is referenced by a two-column foreign key in the referencing table.  It is possible to purchase this

steve rindsberg say:

The incantations do generally work and have since I started using PPT in the early 90s, but with a few provisos:

1) That you can FIND the magic switches.  It used to be easy, but the UI Wunderkind who've dictated how the print dialog shall work have done a magnificent job of hiding important features in improbable locations.  Even if you know which switches you need to throw, it can be hard to find them. If you don't even know they exist, the likelihood of finding them is approximately nil.

2) That the printer doesn't impose large margins; most printers can print quite close to the edge of the paper these days but I imagine there are still some out there that impose wider margins than we might like.

Scale to fit doesn't distort the content.  Or at least I've never seen it happen, and I've been working the print dialog pretty hard since the 35mm slide and overhead transparency days.  But if the slide aspect ratio isn't matched to the aspect ratio of the chosen printer sheet size (or actually it's imageable area) there will be borders. 

In fact, even if you *want* distorted content, I don't think you can get it from the print dialog box; you'd have to change the slide size to force the distortion, then print.  But like you, I've no idea why anyone would really want to do this.
