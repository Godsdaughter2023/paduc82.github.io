Public Function ComboAfterUpdate(Optional strUnboundControlSource As String)
Dim strDelimiter As String
Dim frmForm As Form
Dim RstClone As DAO.Recordset

Dim strFindFieldName As String

    Set cboCombo = Screen.ActiveControl
    'Nulls in combo boxes don't trigger
    ' the NotInList event, so check for that,
    ' or the FindFirst will fail.
    If IsNull(cboCombo) Then Exit Function
    'Use Parent to get a reference to
    '  the right form, even if it's a subform
    '  (instead of using Screen.ActiveForm).
    Set frmForm = cboCombo.Parent
    Set RstClone = frmForm.RecordsetClone
    If strUnboundControlSource = "" Then
        strFindFieldName = cboCombo.ControlSource
    Else
        strFindFieldName = strUnboundControlSource
    End If

    strDelimiter = GetDelimiter(RstClone(strFindFieldName).Type)

    With RstClone
      .FindFirst strFindFieldName & "=" _
       & strDelimiter & cboCombo.Value & strDelimiter
      If Not .NoMatch Then 'should always be a match
          'Undo any update and navigate

          frmForm.Undo
          frmForm.Bookmark = .Bookmark
      End If

    Set RstClone = Nothing
    Set cboCombo = Nothing
    Set frmForm = Nothing

End Function

Public Function GetDelimiter(lngDataType As Long) As String
    Select Case lngDataType
        Case dBDate:         GetDelimiter = "#"
        Case dbMemo, dbText: GetDelimiter = """"
        Case Else
            'Do nothing for numeric types.
            'Return an empty string.
    End Select

suzanne s. barnhill say:

If you save the logo as an AutoText entry and insert it in the document as an AutoText field, then when you redefine the AutoText entry and update fields in the documents, the image will be updated.

Alternatively, you could insert the image as a link; when you save a new image with the same name (and in the same path) as the linked image and update fields in the documents, the image will be updated.

ken sheridan say: You firstly need an auxiliary calendar table of all available working dates, WorkdaysCalendar say, which is simply a table with a column of date/time data type, WorkDate say, stretching 10 years or so into the future.  You then need a table, TimeSlots say, which is again a single column table with the start time of all 15 minute time slots in the day in a TimeSlot column, also of date/time data type.

To record appointments you'd have an Appointments table with columns ContactID, AppointmentDate and TimeSlot, the latter two both of date/time data type. The primary key of the table is a composite one made up of the AppointmentDate and TimeSlot, columns.  Other non-key columns in the table would represent other attributes of the appointment.

To return all available time slots on a specific day would require a query lie this, which returns the Cartesian product of the WorkdaysCalendar  and TimeSlots tables, restricted to available slots by a subquery:

PARAMETERS [Enter date:] DATETIME;
SELECT WorkDate,TimeSlot
FROM WorkdaysCalendar, TimeSlots
WHERE WorkDate = [Enter date:]
AND NOT EXISTS
    (SELECT *
     FROM Appointments
    WHERE Appointments.AppointmentDate = WorkdaysCalendar.WorkDate
    AND Appointments.TimeSlot = TimeSlots.TimeSlot)
ORDER BY WorkDate,TimeSlot;

In reality, the parameter would be a reference to an unbound control in the header of a form bound to the Appointments table, not a simple prompt as above.  The query could form the RowSource property of a combo box in the form, but note that the combo box's BoundColumn property should be 2.  Consequently the combo box can be bound to the TimeSlot column in the Appointments table, and the value of its first column, i.e. Column(0) as the Column property is zero-based, would be assigned to a hidden control bound to the AppointmentDate column by code in the combo box's AfterUpdate event procedure.  The selected time slot could be shown in a read-only text box, also bound to the TimeSlot column.

ken sheridan say: It was actually Tom who gave you the link to the article. I haven't read it.  But let's look a where the parameters are used.  I've tidied the SQL statements of the queries below up a bit to get rid of the unnecessary garbage which Access throws in.  Note BTW that 'query' does not necessarily mean a saved query, which you'd see listed in the navigation pane.  An SQL statement, like those below, which is used as a property of an object is just as much a query.

First the form's RecordSource property:

PARAMETERS [Forms]![frmAppointments]![cboDate] DATETIME;
SELECT Appointments.*
FROM Appointments
WHERE Appointments.AppointmentDate=[Forms]![frmAppointments]![cboDate]
ORDER BY Appointments.AppointmentDate, Appointments.TimeSlot;

This references the unbound combo box in which the date is entered to restrict the rows returned in the form to that date.  The reference is made up of three parts:  [Forms] is the Forms collection, which is a collection of all open forms; [frmAppointments] is the name of the form: and [cboDate] is the name of the combo box.  The square brackets around the object names are only necessary if the name contains spaces or other special characters, which is not recommended and is not the case here, but even if not necessary they do no harm, so if in doubt, use them.

The RowSource property of the combo box bound to the TimeSlot control is the following query:

PARAMETERS Forms!frmAppointments!cboDate DATETIME;
SELECT FORMAT(TimeSlots.TimeSlot,"hh:nn AM/PM")
FROM WorkdaysCalendar, TimeSlots
WHERE WorkdaysCalendar.WorkDate=[Forms]![frmAppointments]![cboDate]
AND NOT EXISTS
    (SELECT *
      FROM Appointments
     WHERE Appointments.AppointmentDate = WorkdaysCalendar.WorkDate
     AND Appointments.TimeSlot = TimeSlots.TimeSlot)
ORDER BY TimeSlots.TimeSlot;

Again the combo box in the form header is referenced in the same way.  In the code from the form's module which I posted earlier, you'll see that the form and the time slots combo box are requeried where necessary when the value in the date combo box in the header might have changed.  Requerying reloads an objects recordset so that it reflects the new value of the parameter.  As I mentioned in the postscript to my last reply the time slots combo box should also be requeried when a record is deleted with:

Private Sub Form_AfterDelConfirm(Status As Integer)

    ' requery time slots combo box to reinstate available tome slot:
    Me.cboTimeSlots.Requery
    
End Sub  

So if you've created a form similar to mine, with a combo box to select a date in the header then you should reference the combo box in the form's RecordSource property and in the time slots combo box's RowSource property.  You should also requery these where necessary with code in the relevant event procedures in the form's module, taking your cue from my form.

What I've given you is a very simple way of recording appointments.  If I were doing it in a fully developed application I would do it in a rather more complex way by entering the start and end time slots for each appointment and inserting rows into the table with code, which would execute an INSERT INTO statement (an 'append' query in Access-speak).  The Appointments table would probably be decomposed into two tables, one in which each row had columns for the attributes of the appointment as a whole, and one which had rows for each time slot relating to each appointment.  However, it would take me rather longer to put that together than the simple solution I've posted, and it you might find it confusing at this stage.  I may well have a shot at it when time permits, with a view to adding it to my collection of little demo files, but I can't say when that would be.
