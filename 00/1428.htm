Why should I care about UTF-8 encoding?

Typically, you can import contacts without worrying about the underlying details of how the text is stored in the CSV file. However, if the contact information includes characters not found in the English alphabet, such as Greek, Cyrillic, Arabic, or Japanese characters, this can cause problems when importing contacts. Therefore, you should save the contacts file with UTF-8 encoding if you have that option during export.

If you can't export your contacts using UTF-8 directly, you can convert the exported CSV file using Excel or third-party apps. How to do this will be different between apps and app versions.

How to convert your CSV file to UTF-8 using Microsoft Excel 2016:

  1. In Excel, create a new blank document (workbook).

  2. From the Data menu, select From Text/CSV. Browse to your exported CSV file (you may have to select to display Text Files (....csv) to see it). Select Import.

  3. In the dialog box that appears, under File Origin, choose the encoding that makes the characters in the text appear correctly, for example, Cyrillic Windows 1251, and then select Load.

  4. Check that characters are displayed correctly in Excel.

  5. Select File >Save As. Enter a name for your file, and select CSV UTF-8 (Comma delimited) (*.csv) as your file type.

  6. Select Save.

You have to remember that because computers work with binary values decimal values are only an approximation.  The currency data type stores values to a precision of four decimal places to suppress cumulative rounding errors, so the underlying value might not be exactly 144.98.  What you see is the formatted value.   If, for instance the underlying value were144.9799, multiplying this by 100 would give you were14497.99 whose integer is14497.  You can see this in the debug window:

MyVal = 144.9799
? Format(MyVal,"Currency")
£144.98
? Int(MyVal*100)
 14497

If you want to multiply the returned value by 100 rather than the underlying value, you will have to ensure the value is truncated to to two decimal places first rather than rounded using the default algorithm.

>> You have to remember that because computers work with binary values decimal values are only an approximation.  <<

That is only true if floating point types are used (Double or Single in VBA; Number/Double or Number/Single for a Field).  Fixed point datatypes (Currency, Number|Byte/Integer/Long Integer/Decimal in a Field; Currency, Int, Long, LongLong in VBA) are exact representation of a number.  Floating point numbers work on the premise of significant digits (15 for a Double and 7 for a single) accompanied with a power of 10.

>> The currency data type stores values to a precision of four decimal places to suppress cumulative rounding errors, so the underlying value might not be exactly 144.98. <<

In what is shown, neither operand is of the Currency datatype.  So it is my guess that the poster of the question is using Number/Double as the Field datatype and Double in VBA.  Which are both floating point numeric values.

ken sheridan say: The point I was really trying to make is that even if the value is of Currency data type, if it is the result of an arithmetical operation it might not be exactly what you see when presented as a currency value to two decimal places.  I've come across people having problems with this on a number of occasions when adding VAT (Value Added Tax) to an invoice.  Sometimes they will add the VAT to each line item with an expression like:

UnitPrice * Quantity * (1+VATRate)

Where VATRate is a fractional value, usually formatted as a percentage.  They show the price inclusive of VAT per line item, and then wonder why the sum of these differs slightly from the total invoice price which aggregates the values.  The reason of course is that what is being aggregated are the values to a precision of more than two decimal places, i.e. the rounding error which results from manually summing the currency value which you see per line item is being suppressed.  This is why invoices usually show the VAT only as single values per invoice for each rate of tax.  I've just looked at the invoice for the recent purchase of my car for instance, and it shows S or E (standard or exempt) as the rate against each line item, and two amounts at the bottom, the exempt items being zero, the standard rate VAT being 70 times what I paid in total for my first car 53 years ago!  

In most day to day situations a series of calculations would be done on the underlying values, with only the final result being presented as a currency value, but there are some accounting situations which require the computed amounts to be rounded to the precision of two decimal places which we normally see, before moving on to the next calculation.  In such cases the rounded value has to be used as the basis for the next calculation.

bob jones aka: cybertaz say:

What you're asking is not possible... Any content added to a Slide (directly or in a Placeholder) will obstruct the view of any Master content it overlays. Masters are the bottom-most foundation on which everything else is placed.

The stacking order of Master layers can be rearranged & the stacking order of Slide layers can be rearranged, but the stacking order of Slide layers cannot be intermixed with the stacking order of Master layers. IOW, nothing added to a Slide can be sent behind anything on its Master. There's no getting around that... It's like wanting to install wall-to-wall carpeting but still have the underlying hardwood floors show through :-)
