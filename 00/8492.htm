bob jones aka: cybertaz say:
The Clipboard is a service provided by OS X, not the application program. It stores only the last content copied or pasted... IOW, when you Cut or Copy the new content replaces the previous. The content is not deleted regardless of how many times you paste it.

Unless there is an extremely large volume of copied/cut content, however, no disk space is involved. The Clipboard content is held in RAM & is cleared when you restart your Mac or log out.

If your HD is getting full what you need to remove are any files you've created but no longer need. Move them to another storage device for future use if necessary. Also, clear your browser caches & remove any software you don't use. I suggest that you not use any utility program that claims to 'clean up' your HD... They often cause more problems than they cure by removing support files needed by your programs.

In order to help improve performance, make sure that Office & OS X are fully updated, repair disk permissions & restart your Mac. Keep in mind that an occasional restart [not just closing the lid if it's a laptop] helps keep things running more efficiently.

  • My query has all three date fields
  • In the Criteria, I want it to only filter on date range IF that particular option was selected.
  • For example, if Form List box has "Processed Date" selected, I want the Criteria for "Trade Date" to return all records.
  • The IF statement was easy enough, but my problem is that because I'm doing a date range, SQL statements & Operators don't seem to work together.
  • If I use only one of the dates for the True & False expression, it works. But when I try to use BETWEEN or  ">=", I don't get any results.

(I.e Microsoft JET uses ".."  as in  12/2/2013..12/4/2013, but Access doesn't like this either).

 

Thanks.

 

john w. vinson say:

You can actually do it all in one query, with no IF and no VBA. It's a bit tricky and you need to realize that a query's criterion is an expression in Boolean algebra - if the expression evaluates to TRUE the record is retrieved, and if it evaluates to FALSE it isn't.

 

So a Where condition that should work would be:

 

WHERE

([Forms]![YourForm]![LIstbox] = "Processed Date"

AND [Processed Date] BETWEEN [Forms]![YourForm]![txtStart] AND [Forms]![YourForm]![txtEnd])

OR

([Forms]![YourForm]![LIstbox] = "Trade Date"

AND [Trade Date] BETWEEN [Forms]![YourForm]![txtStart] AND [Forms]![YourForm]![txtEnd])

OR

([Forms]![YourForm]![LIstbox] = "Billing Date"

AND [Billing Date] BETWEEN [Forms]![YourForm]![txtStart] AND [Forms]![YourForm]![txtEnd])

 

If the listbox contains "Processed Date" then only the first criterion will even be evaluated; the second and third parentheses will be FALSE whatever the date value might be.

sandykj2 say:

christopher miller say:
This add-in doesn't show up on any tabs, it just adds a new export type.  If you are using Office 2013, the installer for the add-in does not write the registry keys for that version of Office.  This is easy to fix.

  1. Run the Registry Editor (regedit.exe)
  2. Navigate the following key: HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Office\14.0\Word\Text Converters\OOXML Converters\Export\MediaWiki
  3. export that key to a file
  4. Open the file with notepad and change the "14.0" to "15.0" and save the file
  5. Import the registry file
If you are running the 64 bit edition of Windows, do the same thing for HKEY_LOCAL_MACHINE\SOFTWARE\Wow6432Node\Microsoft\Office\15.0\Word\Text Converters\OOXML Converters\Export\MediaWiki

Start Word and open or create a document.  Select File-.Save As and when you get to the "Save As" dialog, you'll see "MediaWiki (*.txt)" in the list of supported formats in the "Save as type" list.

I see that rmac3 is working with you to try to repair your system, but my typical approach is, as in the case of investigating an airplane crash, to learn as much as possible of the conditions that existed prior to the crash. 

So, prior to upgrading to build 16199, did you have any 3rd party security program installed, and if so, did you leave it installed, but only deactivated when you began the upgrade?

I pose this question because many of them are designed to automatically reactivate during a restart of the computer, such as during installation of the new build.

In that case, the 3rd party security program might grab and quarantine files that it doesn't recognize, but perhaps still not actually abort the install, so that the build appears to install successfully, but then it might crash or misbehave after login. 

Manufacturers of 3rd party security programs often claim that their program is compatible with Windows 10, but they aren't saying that it is compatible with preview builds, which contain files that were not present in the publicly released versions with which their program was tested.

If the above seems to fit your situation, I would suggest downloading the correct iteration of the ISO files from the Microsoft site at https://www.microsoft.com/en-us/software-download/windowsinsiderpreviewadvanced and doing a clean install (which means deleting both the current OS partition and the system reserved partition, and making a new partition to install the build into, allowing the OS installer to make a new system reserved partition).

After cleanly installing build 16193, you can upgrade to the latest build.

If you then choose to install a 3rd party security program, it is advisable to uninstall it, using the uninstall tool provided on its website, before beginning each new build upgrade.

Additionally, you might want to consider using one of the free for personal use disk image type backup programs to make an image of your OS partition after each build is successfully installed. It only takes a few minutes to make such an image (to be stored on a different hard drive or SSD), and when desired or needed, such as in your present circumstance, or that of a drive failure, you can restore your previous OS and installed programs in 10 minutes or less, using the related rescue bootable medium, not any malfunctioning OS.

I hope some of this helps.

jay freedman say:

Sorry, that was a slip of the fingers and lack of proofreading -- the shortcut for field codes is Alt+F9.

The Developer tab and Design Mode button have nothing to do with field codes. They display a different kind of tag, around different elements in the document (XML, content controls, and ActiveX controls). I mentioned them only as an alternative in case the field code setting didn't solve the problem.

The Compatibility Mode issue is a little difficult to explain.

  • If the older files you referred to are in Word 97-2003 (*.doc) format, then Compatibility Mode is expected. That's the way Word is designed -- features that aren't supported by the older versions are disabled for these documents.
  • If they're in Word 2007 (*.docx) format and were created in Word 2007, that suggests that until recently your Normal template was one that was created during an upgrade from a previous version, and was itself in Compatibility Mode. If that template was replaced because the old one was deleted or renamed, documents created since then would be not in Compatibility Mode. This scenario is extremely unlikely, because all the documents you created since installing 2007 would have had "Compatibility Mode" in the title bar, and I think you would have noticed that.
  • You can convert any document from Compatibility Mode to regular mode by clicking the orb and selecting Convert. For details, read this article.
