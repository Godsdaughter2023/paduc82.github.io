ken sheridan say:

I would also like to add further nesting, .............

Nesting multiple IIF function calls can result in a cumbersome expression of Byzantine complexity.  The SWITCH function is generally a lot simpler, as it accepts a simple set of paired arguments, the first of which in each case evaluates to TRUE or FALSE, the second of which returns the value.  The expressions are evaluated form left to right, so the first of the last pair should simply be TRUE to cause the final value to be returned in the event of all the preceding expressions evaluating to FALSE, e.g.

SWITCH([Noise] = "Woof", "Dog", [Noise] = "Miaow", "Cat", TRUE, "Unknown animal")

I'm a little concerned that you have columns named UnitCost1 and OrderQuantity1.  Does this mean you have multiple unit cost and order quantity columns in a table?  If so the table is not normalized to First Normal Form (1NF) whose formal definition is:

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple
contains exactly one value for each attribute.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).

A table not normalized to 1NF requires decomposition into related tables.

ken sheridan say: You should decompose the Products table into two tables, in broad outline as follows:

Products
….ProductID  (PK)
….Product

ProductPrices
….ProductPriceID  (PK)
….ProductID  (FK)
….LowerQuantity
….UpperQuantity
….UnitPrice

The LowerQuantiy and UpperQuantity columns would define the ranges for which each UnitPrice would apply, so if a product had six unit prices, each for a quantity within the range for the price in question, there would be six rows in ProductPrices for that product each referencing a single row in Products by virtue of the value in the foreign key ProductID column referencing the relevant row in Products.  The LowerQuantiy column should have a value of 1 for the lowest row in the range, and the UpperQuantity column should have an artificially high value which will never be reached in the highest row of the range.

Because prices can change over time the relevant price would be assigned to a UnitPrice column in an OrderDetails table or similar when an order is raised. Each order would then retain the price current at the time of the order regardless of subsequent price changes.  In relational-speak the UnitPrice column in OrderDetails is functionally determined solely by the primary key of the table, so an OrderDetails table would be along these lines:

OrderDetails
….OrderID  (FK)
….ProductID  (FK)
….UnitPrice
….Quantity

The primary key of the table is a composite one of the two foreign key columns, OrderID and ProductID.  In the AfterUpdate event procedures of the both the ProductID and Quantity control in a data entry form the relevant unit price would be looked up in the ProductPrices table and assigned to the UnitPrice column:

Dim strCriteria As String

If Not IsNull(Me.ProductID) And Not IsNull(Me.Quantity) Then
    strCriteria = "ProductID = " & Me.ProductID & " And " & _
        Me.Quantity & "Between LowerQuantity And UpperQuantity"

    Me.UnitPrice = DLookup("UnitPrice", "ProductPrices", strCriteria)
Else
    Me.UnitPrice = 0
End If

An unbound control in the form would compute the total price by means of the following expression as its ControlSource property:

=[UnitPrice]*[Quantity]

This value should not be stored in a column in the table, but always computed in a form, report or query.  Otherwise redundancy would be introduced into the table, which would then be open to the risk of update anomalies.

Yes; you probably need to re-think the entire exercise from the ground up (I appreciate this might be a major exercise).

Take a look at:-

http://answers.microsoft.com/en-us/office/forum/office_2007-excel/information-in-one-spreadsheet-to-automatically/17e2f494-fe37-4a76-9da6-c489e8151a16

 - which is possibly one way what you want can be achieved.

What you want can be done but it requires a fair amount of thought and planning to put it into practice.

Other things that you can consider here (which will almost certainly work) is setting up Data Connections between EXCEL Workbooks. Pivot Tables with the appropriate Data Connections will almost certainly work as well (for what you want).

Another thing that you should look at is:-

https://support.office.microsoft.com/en-gb/article/Consolidate-data-in-multiple-worksheets-007ce8f4-2fae-4fea-9ee5-a0b2c9e36d9b?CTT=1&CorrelationId=b5f148fa-0890-4356-bab1-ae8f945c39fc&ui=en-US&rs=en-GB&ad=GB

I very much doubt that you want a Shared Workbook in the situation that you describe.

The philosophy behind Shared Workbooks is that they have one central user (controller) who is responsible for exactly what's happening (like managing the conflicts that you mentioned).

It's a common misconception that they are best used by groups of people who work together: that, very often, is not what Shared Workbooks are best for.

You should also consider using ACCESS that is designed for multi user access with full record locking facilities. EXCEL is not designed to do this.

There are a whole variety of solutions available but this is not something that can be done, "on the fly".
