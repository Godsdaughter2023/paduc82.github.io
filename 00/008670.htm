hansv mvp say:

You don't need to include all the contact's details in the Leads table.

 

I assume that the Contacts table has a primary key field, say ContactID.

 

If the table doesn't have a primary key yet, add a new field as first field, name it (for example) ContactID, set its data type to AutoNumber, then click the Primary Key button, and finally save the table design.

 

Next, add a field of the same name to the Leads table; set its data type to Number (Long Integer) if the corresponding field in the Contacts table is an AutoNumber field. Save the table design.

 

In the Relationships window, add both the Contacts table and the Leads table (if necessary), then create a relationship between the two tables on the ContactID field. Double-click the join line and tick the check box "Enforce Referential Integrity", then click OK. This ensures that you won't be able to enter a ContactID in the Leads table that doesn't exist in the Contacts table.

 

On a form bound to the Leads table, you can use a combo box that has the ContactID field as Control Source, and the Contacts table or a query based on the Contacts table as Row Source.

If you set the Column Count property to 2 and Column Widths to 0, the combo box will store ContactID but display the second field of the contacts table/query.

john w. vinson say:

You don't.

 

Access ISN'T a flat file database, it's a relational database!

 

As such, you should store each piece of information once, and once only, and thereafter use Queries linking tables to bring information from different tables together. The Contact information (lastname, firstname, etc.) should exist in the contacts table, and noplace else. Your Leads table should have a ContactID as a link to the Contacts table; if you need to see Contact data in conjunction with information about Leads involving that contact you would use a Query linking the two tables, or a Form based on Contacts with a Subform based on Leads.

 

The whole point of relational data is that you avoid redundancy, thereby avoiding redundant redundancy. :-{)

hansv mvp say:

Create a new query in design view.

Add the Contacts table, then close the Show Table dialog.

Add the ContactID field to the first column of the query grid, and the field that you'd like to display on the Leads form to the second column. For example: a "last name" field, or an expression such as

 

[LastName] & ", " & [FirstName]

 

If you wish, you can select Ascending from the Sort dropdown in the second column.

Save this query as (for example) qryContacts, and close it.

 

Now open (or create) a form based on the Leads table.

Place a combo box on this form, and set the caption of its label to Contact.

Set the following properties for the combo box:

 

In the Format tab of the Properties window:

Column Count: 2

Column Widths: 0

 

In the Data tab of the Properties window:

Control Source: ContactID

Row Source: qryContacts

Row Source Type: Table/Query (this is the default setting)

Bound Column: 1 (this is also the default setting)

 

When you switch to form view and click the dropdown arrow of the combo box, you will see the data from the second column of the query qryContacts. The first column, containing the ContactID values, is hidden because its width has been set to 0. But when you select an item from the dropdown list, the corresponding ContactID value will be stored in the field, since the first column is the 'bound column'.

I was wondering if I could have the contact info to automatically appear in the Leads table so that the form for that table could be sent to the salesman?  For example, when I click on one of 3 items in the Contacts table (Lead, CB, or Email), could the contact data be automatically inserted into the Leads table?

john w. vinson say:

You're missing the point, Faxylady.

 

Tables are for storing data. They're not for sending, they're not for reporting, they're not for editing!

 

If you want to display data, use a Form. If you want to enter data, use a Form. If you want to email information to someone who isn't in the database, use a Report.

 

It is not necessary - in fact it's incorrect design - to store the contact data redundantly in the leads table.
