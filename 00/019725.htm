jay freedman say:

There are two ways to handle this.

One way is to precede the second half of the macro with the statement

If ActiveDocument.Range.Endnotes.Count > 0 Then

and put an End If just before the End Sub statement. That will prevent the second half from executing at all if there are no endnotes.

The other way is to "expect" the error. Put the statement

On Error GoTo Bye

before the second half of the macro, and put this before the End Sub:

  Exit Sub
Bye:
  If Err.Number = 5941 Then  ' member does not exist
    MsgBox "No endnotes", , "Skip Endnote Story"
  End If

If the endnote story doesn't exist, trying to address it (even before you try to select it) triggers error 5941, "The requested member of the collection does not exist". The On Error statement causes execution to jump to the label Bye, where a more appropriate message can be displayed (or you can comment out the MsgBox and just let the macro exit).

graham mayor say:

It would be possible e.g.

Sub CutLastLetter()
Dim arrFind As Variant
Dim i As Long
Dim oStory As Range
    arrFind = Array("towards", "backwards")
    For i = 0 To UBound(arrFind)
        For Each oStory In ActiveDocument.StoryRanges
            With oStory.Find
                Do While .Execute(FindText:=arrFind(i))
                    oStory.Text = Left(oStory.Text, Len(oStory.Text) - 1)
                    oStory.Collapse 0
                Loop
            End With
            If oStory.StoryType <> wdMainTextStory Then
                While Not (oStory.NextStoryRange Is Nothing)
                    Set oStory = oStory.NextStoryRange
                    With oStory.Find
                        Do While .Execute(FindText:=arrFind(i))
                            oStory.Text = Left(oStory.Text, Len(oStory.Text) - 1)
                            oStory.Collapse 0
                        Loop
                    End With
                Wend
            End If
        Next oStory
    Next i
lbl_Exit:
    Set oStory = Nothing
    Exit Sub
End Sub

rob koch say:

There is a lot of confusion about this on the Internet, since few understand how Java interacts with web browsers, the Java Cache and temporary file locations on the disk.

 

Even with my own past programming history it has taken me some time to get a feeling for what's really happening in these cases, but I believe I finally have enough understanding to state some of it here without making a complete guess.

 

Basically, I feel that most of the Java exploit detections that people come here to discuss are nothing to be worried about.  This is because in many of these cases the detections are simply related to known Java exploits that have been detected while performing an MSE Full scan.  In many cases these are found within the Java Cache folders, but in others they may also be located within the Internet Explorer Temporary Files folders as well.

 

Though the existence of these files is scary because it appears to indicate an attempted attack from the past that MSE didn't detect, I don't personally believe that most of these are a concern.  The reason is that though the files are actually present, if the particular exploits they contain have already been patched via an earlier Java update, they wouldn't be successful anyway.  So regardless of the fact that they are present and also that they may have been missed by MSE at the time they were originally downloaded, they really posed no threat to the system anyway.

 

The other thing I've noticed is that though I have disabled the saving of Java temporary files within the Java Control Panel applet, I still find these same exploit files on my PC on occassion.  However, rather than being found within the Java Cache folder, they are instead detected within the temporary files folder related to my logged in user.  Since the normal act of downloading a file or viewing a web page creates temporary file(s) in that folder this makes complete sense.  In fact, the files detected here are usually detected within a complete package or "jar" file, which is the way Java distributes applications or libraries.  So again, if these files are later detected by MSE as containing exploits, unless those particular exploits were unpatched in my installed Java platform at the time they were received, they wouldn't have been able to successfully compromise Java and thus the operating system anyway.

 

Now the question this brings to mind for me is whether MSE would ever present a real-time notification for such exploits, since in truth these aren't really malware themselves, simply the exploit code used to attempt to circumvent the normal operation of Java to then insert some additional truly malicious code that actually takes control of the system.  I don't know that this is true, it just seems to me that since the exploit itself only performs the initial steps, Microsoft may instead be concentrating on detecting and stopping the actual infection code.

 

Regardless of the answer to Microsoft's methods, in reality it's easy to see from the above that attempting to detect the exploit code within the jar distribution files, which are typically obfuscated and constantly changing is nearly impossible, since defintions can't change that fast.  So instead, I'd expect that Microsoft probably uses their behavior monitoring system to watch for the activity these exploits create and then ties this into addtional monitoring and detection of whatever payload the exploit attempts to deliver.

 

So as a result, I don't think that the detections of Java exploit files in temporary folders during an MSE Full scan really have much meaning by themselves.  They are really only the indication of an attempt rather than either a successful infection or true malware.  So the deeper question is are there other signs of additional malware either by detection during real-time operation or a scan, including files found within non-temporary folders or other indications such as issues with the operation of the PC in general?

 

Unless there have been additional signs of malware, I feel that these detected exploits are merely leftover remnants of a past attempt that failed and so are simply the same as any other temporary file trash found in these various temporary folders.  As such they can be cleaned out and ignored, since by themselves they have no more meaning than any such leftover garbage.

 

Rob
