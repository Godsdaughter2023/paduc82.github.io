ken sheridan say: A 'multi-valued field' models a many-to-many relationship type between two entity types.  While, at first sight, it might seem a convenient way to do this, in reality it can be problematic.  The conventional way to model a many-to-many relationship type is by a third table which resolves the relationship type into two one-to-many relationship types.  Unless there is a cogent reason for using a 'multi-valued field', I would strongly recommend the conventional model.

One solution would be to change the query to a 'make table query' and insert the query's results into a temporary table which you can then export to a text file.

A better solution, however, would be to decompose the table into a conventional 3-table model.  You'll find examples of how to do this in MVFCorrector.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the option for an 'MVF with Data as Embedded Value List' would be appropriate to your current set up.  You would firstly have to create an empty Ages table and an empty table to model the many-to-many relationship type.  It would then be a case of amending the code from the Click event procedure of the cmdCorrect button as necessary to reflect your object names.

It appears that the paragraph's formatting has been set to a "hanging indent" almost as large as the text column width. To fix it, right-click in the affected text and click Paragraph in the menu. In the dialog, change the "Special" dropdown from Hanging to (none).

If this problem appears in other paragraphs, look at the definition of the style applied to those paragraphs, and if you see a hanging indent there, modify the style to make the same change.

By the way, this problem probably isn't the result of anything you did. If you tried to set a hanging indent close to the right margin by using the Paragraph dialog, you'd get an error message, "The indent size is too large". A macro could do it, but that seems unlikely. @osamakhanjmi....  The issue is:  when Excel reads numbers, it interprets only the first 15 significant digits, replacing any digits to the right with zeros.

Beware that if you follow Gord's (correct) suggestion, the value is not interpreted as a number in some contexts.  And when it is interpreted as a number, Excel will still use the value 1805071317082310.

Ironically, Excel can represent your example (any 16-digit integer up to 9007199254740992) exactly as the result of a despite appearances.
