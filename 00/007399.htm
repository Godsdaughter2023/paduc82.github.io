mohdrafi say: Windows Update error 80070643

Windows Update error 80070643 can happen for several reasons. The most common cause is a problem with the .NET Framework installed on the computer.

The .NET framework is a part of Windows that's used to build, deploy, and run programs and applications. Unfortunately, it can be damaged by some programs, viruses, orhard disk problems.

To repair the .NET Framework automatically

  1. Click this link:

    Repair .NET Framework

    ‌

  2. In the File Download dialog box, click Run , and then follow the steps in the wizard.

  3. OpenWindows Update through control panel

  4. In the left pane, click Check for updates , and then wait while Windows looks for the latest updates for your computer.

  5. If any updates are found, click Install updates .  If you are prompted for an administrator password or confirmation, type the password or provide confirmation.

If the previous steps didn't fix the problem, you can reinstall the .NET Framework manually, but the steps involved are intended for advanced users. If you're not comfortable performing these steps, ask a friend to help you. You'll find the steps to follow in the reinstalling the .NET Framework article on the Microsoft website.

robert aldwinckle on forums say:
'Internet Explorer has stopped working' message


Means that something is causing iexplore.exe to crash.   Use the Reliability Monitor to get more details about your crashes.   You may be lucky and find that they point to a specific third-party module.   However, unfortunately it is more likely that it will point to a low-level system module and then the problem will only be able to be determined by extracting a Stack Back Trace for the crashing thread.   Previously this was available via drwtsn32.log but that tool is no longer used in your OS.   Now you need to use  windbg.exe  or resort to various third-party tools for some kind of substitute.   You might be able to get some related clues from Resource Monitor's list of loaded modules for the task which crashed.   E.g. for the instance of  iexplore.exe  before and after the crash.   If you got a list before a crash which showed a module which was no longer listed after a crash that could be very significant, e.g. it could represent a pointer to one of its routine's still being used invalidly after it had been unloaded.   Typically those kinds of crashes are represented by a StackHash representation since they don't occur in any loaded module.
