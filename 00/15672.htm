_ronx say:

You can't.
WordArt is not a problem in Internet Explorer - but can be terrible in other browsers.
Shapeart (autoshapes) uses a form of absolute positioning - referenced to the top left of the browser. This means that the only place they will go without floating out of place as text is resized or the browser portal resized is the top left corner of the page.
Convert them to images - either .jpg or .gif depending on colours.
A further complication is that firefox (and other non-IE browsers) cannot render VML (Wordart etc) - they see a low quality .gif representaion of the images, which is sometimes incomplete.

To covert, do a screen dump and paste into a graphics editor. Crop and save for the web.

See http://www.rxs-enterprises.org/fp/articles/2007/10/02/01/VML-graphics.aspx  for reasons why wordart and shape art should be avoided.

shyamal debnath msft support say:

Hi Daisy,

I would like to know which kind for migration they want to use. There are three types of email migrations that can be made from an Exchange Server.

In a cutover or staged Exchange migration, the mailbox contents, folder permissions, and delegate permissions of the on-premises mailbox are migrated to a new cloud-based mailbox. However, Send As or full mailbox access permissions aren't migrated.

In a hybrid deployment, on-premises mailbox permissions such as Send As, Receive As and Full Access that are explicitly applied on the mailbox are migrated to Exchange Online. However, the migrated permissions only work if the target and delegate mailboxes are both in the cloud, because Outlook Web App (OWA) and Microsoft Outlook can't access folders in cross-premises mailboxes.

For your reference: If I migrate mailboxes from Exchange to Exchange Online, will mailbox permissions be migrated over?

ken sheridan say: You've got the cart before the horse.  The end result you want is what you should be starting with.  What you have is what is known as 'encoding data as column headings', the data in this case being the test numbers.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  The correct model would involve three tables like this:

Patients
....PatientID
....Firstname
....LastName
etc

Tests
....TestNumber
....TestName
etc

PatientTests
....PatientID
....TestNumber
....TestDate
....OutCome

The last table models the many-to-many relationship type between Patients and Tests by having two foreign keys PateientID and TestNumber referencing the primary keys of the other two tables, i.e it resolves the  many-to-many relationship type into two  one-to-many relationship types, which is how a many-to-many relationship type is always represented in a relational database, though there can be more than two tables related in this way of course.

Having created the PatientTests table, you can easily, albeit tediously, recast your data into this correct model by means of a series of 100 'append' queries, though you don't need to save them, just amend and execute each without saving, staring with:

INSERT INTO PatientTests (PatientID,TestNumber,Outcome)
SELECT [ID], 1, [1]
FROM [YourCurrentTable]
WHERE [1] IS NOT NULL;

then:

INSERT INTO PatientTests (PatientID,TestNumber,Outcome)
SELECT [ID], 2, [2]
FROM [YourCurrentTable]
WHERE [2] IS NOT NULL;

and so on to:

INSERT INTO PatientTests (PatientID,TestNumber,Outcome)
SELECT [ID], 100, [100]
FROM [YourCurrentTable]
WHERE [100] IS NOT NULL;

However, if you are familiar with writing VBA code it would be quite a simple task to eliminate the tedium by writing a little procedure which builds and executes the 100 SQL statements one after the other automatically.  You'd just need to loop from 1 to 100 and change the value to be inserted into the TestNumber column and the name of the [1], [2] etc column at each iteration of the loop.

BTW avoid 'special' characters like the # character or spaces in column or table names.

Ken Sheridan, Stafford, England

ken sheridan say: So you don't in fact have the table you described in your original post?

Let's assume that the table you are now describing is named PatientTests and, as you've shown leading zeros, the PatientNr column is a text data type, the TestNr column is a number data type and, on the assumption that the TestValue column may include non-numeric values in the case of other tests, it is a text data type.

Let's also assume that the form includes a combo box control cboPatientNr in which you select the patient, and whose value is the PatientNr.  A caveat here.  Do not rely on a patient's name, even in conjunction with other personal details, to identify a patient.  I was once present at a clinic where two patients turned up for appointments, both female, both with exactly the same first and last names and both with the same data of birth.  The nursing staff had some difficulty in determining which procedures were to be given to each of these patients.

And let's assume that the four unbound text box controls are named txtAge, txtWeightGain, txtWaistFrom and txtWaistTo.

Firstly you do not need to have partially inserted the rows first; the code below will insert the row and assign the values.  In the AfterUpdate event procedure of each control put code as follows:

1.  In txtAge's AfterUpdate event procedure:

Const MESSAGETEXT = "Patient number must be selected first."
Dim strSQL As String

If Not IsNull(Me.cboPatientNr) Then
    strSQL = "INSERT INTO PatientTests(PatientNr,TestNr,TestValue) " & _
        "Values(""" & Me.cboPatientNr & """,1,""" & Me.txtAge & """"
   CurrentDb.Execute strSQL, dbFailOnError
Else
    MsgBox MESSAGETEXT, vbExclamation, "Invalid Operation"
End If

2.  In txtWeightGain's AfterUpdate event procedure:

Const MESSAGETEXT = "Patient number must be selected first."
Dim strSQL As String

If Not IsNull(Me.cboPatientNr) Then
    strSQL = "INSERT INTO PatientTests(PatientNr,TestNr,TestValue) " & _
        "Values(""" & Me.cboPatientNr & """,2,""" & Me.txtWeightGain & """"
   CurrentDb.Execute strSQL, dbFailOnError
Else
    MsgBox MESSAGETEXT, vbExclamation, "Invalid Operation"
End If

One thing I find surprising, however, is that this table, as you describe it, allows only one set of tests per patient.  I'd have assumed that it would either have a column such as TestDate to identify when the test was undertaken; or it would have a another column such as ExaminationNr so that PatientNr and ExaminationNr, as a composite foreign key could reference the composite primary key of a further table which would include non-key columns representing attributes of each coherent set of tests undertaken on a patient.

With the addition of a column such as TestDate or ExaminationNr, this column plus the PatientNr and TestNr columns would be the composite primary key of PatientTests.  Without the additional column PatientNr and TestNr would be the composite primary key of the table, but this would allow only one set of tests per patient.  Whichever is the case it is essential that these columns do not permit duplicate values (in combination), so must be uniquely indexed, which is most easily achieved by making them the composite primary key of the table.
