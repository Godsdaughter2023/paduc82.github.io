srimadhwa b say:

Hi,

I would like to let you know browsing history is never collected by Microsoft.

In Event Viewer, under Applications and Services, is an Internet Explorer log. By default, nothing is written to it.

To enable logging, download and install the Application Compatibility Toolkit from Microsoft and perform the following:

  1. Start the registry editor (regedit.exe).
  2. Move to HKEY_CURRENT_USER\Software\Microsoft\Internet Explorer\Main\Feature Control\FEATURE_ENABLE_COMPAT_LOGGING (if the key does not exist, then create it; you might need to create the Feature Control and FEATURE_ENABLE_COMPAT_LOGGING keys).
  3. From the Edit menu select New, DWORD value.
  4. Set a name of iexplore.exe and press Enter.
  5. Double-click the new value, set the data to 1, and click OK.

When Internet Explorer is restarted, logs would be written.

Outlook 2013 does not store the autocomplete cache in a separate file, as previous versions of Outlook did.

ken sheridan say:

And if I have the expression "where callsign like 'XYZ????'" in the where clause the length of callsign is always exactly 7 characters.

It seems that you want to remove the 'Z' where the value is seven characters in length, the first three characters are 'XYZ' and the last four characters are numerical digits.  The use of the IsNumeric function to determine the last of these criteria is not reliable as IsNumeric("12E4") for instance will return a True, the number in this case being 12 by 10 to the power of 4.

You can do it reliably and far more simply with callsign LIKE "XYZ####"

To remove the 'Z' call the Replace function:

UPDATE flightdata
SET flight = REPLACE(callsign, "Z","")
WHERE callsign LIKE "XYZ####";

There is no real need to check the current value of the flight column as it will merely be replaced with its current value if they match.

Your query is essentially:
update flightdata set flight=[expr]
where clause1 and clause2 and clause3

You state that this is a bug because clause1 should have eliminated all records that could make clause2 and clause3 fail.
This would be true if they were evaluated left to right, but there is no such guarantee. I could not find a reference in the 1 minute I took to look for it, other than this one for SQL Server: http://stackoverflow.com/questions/484135/select-where-clause-evaluation-order, but take my word for it: the SQL language does not include such guarantee.

So you build it yourself. I first created a query that applies clause1 and called it qryFlightData_CallSign:
SELECT FlightData.FlightID
FROM FlightData
WHERE (((FlightData.CallSign) Like 'XYZ####'));
(Note I am also applying some earlier-discussed enhancements)

Then I use that query to inner-join with in the Update query:
UPDATE FlightData INNER JOIN qryFlightData_CallSign ON FlightData.FlightID = qryFlightData_CallSign.FlightID
SET FlightData.Flight = 'XY' & Right([CallSign],Len([CallSign])-3)
WHERE (((FlightData.Flight)<>'XY' & Right([CallSign],Len([CallSign])-3)));
(I could have left the where clause out per earlier discussions)

This succeeds. QED.
