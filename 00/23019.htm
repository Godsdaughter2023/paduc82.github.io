qa guy commelec say:
Create a table CountNumber with field CountNUM containing numbers 0 (zero) through your maximum.

Join this query in your other and count DaysDiff --

SELECT YourTable.DateAssign, YourTable.[Date Due], Weekday(DateAdd("d",[CountNUM],[DateAssign])) AS DaysDiff
FROM YourTable, CountNumber
WHERE (((DateAdd("d",[CountNUM],[DateAssign]))<=[Date Due]) AND ((Weekday(DateAdd("d",[CountNUM],[DateAssign]))) Between 2 And 6))
ORDER BY YourTable.DateAssign, YourTable.[Date Due];

albert d. kallal say:

One way I do this is to create a shortcut in the "send to" dir.

Just browse to your shortcut dir, it usually something like this in win 7

C:\Users\Default\AppData\Roaming\Microsoft\Windows\SendTo

You could also use you own such as:

C:\Users\AlbertKallal\AppData\Roaming\Microsoft\Windows\SendTo

Ok, now once in that dir (paste above into windows explore), then go

New->new shortcut.

When the wizard comes up, browse to msaccess program dir, or even paste in this:


"C:\Program Files (x86)\Microsoft Office\Office14\MSACCESS.EXE" /repair

At this point you can then right click on any accDB, and choose sendto, and then choose the shortcut (you can call it my compact).

As a general rule you should hold down the shift key if there is startup forms.

Also, while /compact is a command line option, it does not work for send to, but they are functional the same after 2000 anyway.

You could also write some code in Access to popup a filedilog, and then allow the user to browse to a given file, and it would also to the above for you by execute (shell()) the above command line + the file name.

So, there not a standalone C+R utility, but the above is two ideas here. If this is just for two particular files in question, then you can simply provide a shortcut as per above that includes the file name along with a /compact option.

I think which approach is going to depend on how many different files you wish to do this to, and if you know the file(s) ahead of time or not. \

I don't believe there is a stand alone utility like their used to by, but then again the above quite much gives you a command line or utility type of approach here.

With a split database, then this issue is near always fixed with a persistent connection (9 out of 10 times).

However, since you state the issue exists with a local un-split database, then something else is likely the issue.

In the case of an un-split, then this usually suggests there is a default printer (often a network printer) that does not exist for some of the reports. Or simply that you have a default network printer that does not exist.

Another issue is linked tables pointing to some server that does not exist, or is on some kind of slow connection. (a persistent connection will fix this case).

Try changing your default printer to a “local” printer (I install a free pdf printer and set that as my default).

So look for a network printer as default (that can be slow, or even non existing). And check for linked tables that point to non-existing location(s).

And as others suggested such delays can be due to virus software. (so check this issue too).

And does this issue exist for all database(s), or just one in particular?

Cannot delete manual page break - Microsoft Community

jbubly6 say:

Cannot delete manual page break

I have a manual page break set in a document that I transferred from Microsoft Office 2008 on my PC.  I now am using Office for Mac 2011; I am unable to delete this manual page break by following the instructions given (hitting 'delete' when viewing as a draft).  Any suggestions?
john mcghie say:
I have a manual page break set in a document that I transferred from Microsoft Office 2008 on my PC.  I now am using Office for Mac 2011; I am unable to delete this manual page break by following the instructions given (hitting 'delete' when viewing as a draft).  Any suggestions?

It's probably not a "Page Break".  Does it say it is?  Not a "Section Break"?  There are some limits on deleting section breaks, particularly either side of a TOC or Index.

The page break is actually "part" of the paragraph following it.  If that paragraph is immediately following a table, Word won't allow you to delete it, in order to avoid corrupting the table.

On the other hand, it may be that the paragraph itself is corrupt.  In which case, the answer can become complex.

First:  Add a paragraph mark either side of it.

Then select all three things (the para above, the page break, and the para below).  Now will it delete?

If not, it probably is a corrupt paragraph.

Your only solution then would be to create a new document, copy everything above into the new document, followed by everything below.  Be careful not to copy either the page break or the paragraph mark either side of it, or you will copy the problem.

If that doesn't fix it, post back, we'll have to get serious with it.
