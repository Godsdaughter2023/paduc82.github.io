denis charland say:

I investigated a little bit more on this problem. I noticed that all apps except Calendar were showing up for a brief moment when they were started. So I had a look at the permissions of the directory and files of the Calendar app, located under C:\Program Files\WindowsApps. Here's a snapshot:

As you can see, most of the permissions are inherited from the WindowsApps directory, except the first three ones. The first and the second entries are special. Searching in the registries, I found that these IDs correspond to the application package IDs. They are different for each app. For many of the apps, these two entries were missing. I did not find a way to restore these missing entries, even "icacls /reset /Q /T /C" command was unsuccessful. It seems that the only way to substitute the role of these two entries is to add the "ALL APPLICATION PACKAGES" role to the WindowsApps directory and all it's subdirectories and files. Note that the "ALL APPLICATION PACKAGES" role is not normally present in the properties of these subdirectories and files. This is only a workaround.

Even adding the "ALL APPLICATION PACKAGES" role was not enough to completely solve the problem. Some apps like Camera, Photos, People and Store were still showing up for a brief moment when they were started. If you look at the third entry in the table above, it corresponds to the Users group. This entry is normally present in the properties of all subdirectories and files under the WindowsApps directory, but was missing except for the calendar app. After adding this role, all the apps were starting normally.

For security reasons, you should not modify the ownership and the permissions of any system files and directories using your user account. This is a security breach. Remember that you first need to take the ownership of these files and directories to be able to modify their permissions. This means that every time you are logged into your account, any malware or virus can take avantage of that to mess-up your computer. I highly recommend to use the built-in Administrator account to perform these kind of tasks. This account is not activated by default. You need to activate it first. Here's how to activate this account:

Start a Windows PowerShell as an administrator. In "All apps" --> "Windows PowerShell" menu, right-click on "Windows PowerShell"  and select "Run as Administrator". This will open a PowerShell window. Now type the following:

net user administrator /active:yes

The built-in Administrator account is not password protected by default, so you should add a password to this account by typing the following:

net user administrator *

Type "exit" to close the PowerShell window then logout from your account. Now, log into the built-in Administrator account. Open the an explorer window and switch to C:\Program Files. To be able to see the WindowsApps directory, you need to modify the view options and check "Show hidden files, folders and drives", click on "Apply to Folders" and then on "OK".

Now, select the WindowsApps directory and right-click on it and choose "Properties". Go to the "Security tab" and select "Advanced". At the end of the "Owner" line, click on "Change", type "administrator" and click on "Check Names" then on "OK". Check the option "Replace owner on subcontainers and objetcs" then on "Apply". The ownership will now be changed to "Administrator" on all subdirectories and files in addition to the WindowsApps directory.

Your are now ready to add the "ALL APPLICATION PACKAGES" and "Users ("your computer name"\Users)" roles. Click on "Add", click on "Select a principal", type "ALL APPLICATION PACKAGES", click on "Check Names" then on "OK" "OK".

Click on "Add" again, click on "Select a principal", type "Users", click on "Check Names" then on "OK" "OK". Now click on "Apply" and the two roles will be added to all subdirectories and files in addition to the WindowsApps directory.
