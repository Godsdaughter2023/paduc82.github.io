alwphantom say: I finally got it to work on both Mac and PC across multiple machines and permission levels. :-)

Thank you Peter, Paul and Jim for your time in helping to solve this.  Here is what the ultimate (incredibly simple) issue was:

I cannot explain why it worked in your SharePoint environment Peter, but I did the same thing when I tested the URL without the "?web=1" and changed the "https" to "http" (which worked on a PC but not on a Mac).  When I tried it again today I at the same time added a line with the {FILENAME \p} on it's own for some testing for Paul.  I noticed that the file name that appeared was "https" without the "?web=1".  Bingo, updated on a PC and Mac as "https://domain.sharepoint.com/[folders]/filename.docx".

So to recap for anyone else down the road who may be reading this and trying the same thing, here is what worked for me:

Include Text Field Code: { INCLUDETEXT "https://[domain].sharepoint.com/[folders]/[filename].docx" \*MERGEFORMAT }  Where domain=your domain, folders=the path to the folder on SharePoint, and filename=your file name

Realtive(ish) Path: To make it an easily adjusted file path where the file name remains the same but the rest of the URL changes.
- Create a column in the SharePoint library for the URL path (for instance IncludeTextPath)
- Create your document in that Library either by dropping it there or saving a new document to the library
- Define the data for that document in your "IncludeTextPath" as the path to the source documents excluding the file name
- Change the code to: { INCLUDETEXT "[IncludeTextPath][filename].docx" \*MERGEFORMAT } where "IncludeTextPath" is inserted by going to Insert Tab>Quick Parts>Document Property>InsertTextPath
- Note that for this example i included the "/" before the file name in the IncludeTextPath

FILENAME Field: This field does not appear with my testing to work with the relative fields, although I didn't spend a lot of time working on it. This is using the "\ ..\ method brought up earlier in the discussion that works for local files but appears not to play nicely with the SharePoint URL.

Permissions: When I tested this with a person who did not have edit permission on SharePoint, the updating of fields was a little off as he was not at the access level to edit. He could update the target document fine, but he needed to know to do that and because he downloaded a copy of it I'm not sure how it would have handled it if he went to print (I assume it would have auto updated). So good practice may be to update the fields before saving or set them to auto update on open (which could be time consuming).  We have a relatively small SharePoint environment here, maybe only 20 or 30 of us using it at the moment, so this isn't a huge issue for us, but I mention it as it could be if the files are reference regularly.

Path Length: Keeping it short seems to make Word happier. Using an underscore instead of a space is always better on SharePoint to save characters. :-)

Thanks again for the help guys, it is much appreciated!
Brian

Creating Paragraph Breaks within Conditional Field Codes - Microsoft Community

thomas-4543 say:

Creating Paragraph Breaks within Conditional Field Codes

I'm trying to create a sequential list using field codes and mail merge. My problem is that certain items do not need to appear on the list, depending on how the mail merge goes. This is confusing, so I will give an example.

Let's say I'm putting together a list of quantities for various fruits. I've got a mail merge document that tells me how many apples, oranges, and pears I have. So the list might look like:

1. There are 32 apples.

2. There are 27 oranges.

3. There are 3 pears.

Using field codes it looks more like:

(SEQ list1) There are (MERGEFIELD "Apples") apples.

(SEQ list1) There are (MERGEFIELD "Oranges") oranges.

(SEQ list 1) There are (MERGEFIELD "Pears") pears.

Now, if there are no oranges, I don't want it to be on the list at all. I just want the list to look this this:

1. There are 32 apples.

2. There are 3 pears.

To do this, I use if/then conditioning. (IF (MERGEFIELD Oranges) = 0 "" "(SEQ list1) There are (MERGEFIELD "Apples") apples. This is successful in removing the oranges from the list, but there remains a space:

1. There are 32 apples.

2. There are 3 pears.

I've tried every which way I can thing to get rid of that space, but I can't figure it out. Perhaps someone here could help me.

suzanne s. barnhill say:

What you need is something more like this:

{ IF { MERGEFIELD "Apples" } > 0 "{ SEQ list1}. There are { MERGEFIELD "Apples" } apples.

" }{ IF { MERGEFIELD "Oranges" } > 0 "{ SEQ list1}. There are { MERGEFIELD "Oranges" } oranges.

" }{ IF { MERGEFIELD "Pears" } > 0 "{ SEQ list1}. There are { MERGEFIELD "Pears" } pears.

" }

That is, the paragraph break must be inside the TrueText so that it (along with the rest of the numbered item) does not appear if the condition is not met.
