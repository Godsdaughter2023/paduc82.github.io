jllatham say:

This is relatively easy to set up and use.  Here is a screen shot of what I set up:

The numbers in A2, B2, C2 are your entries, and can be changed as needed.  The formula in D2 is

=B2 * C2

to give you the number of blankets per pallet although it is not used anywhere else.

The formula in C5 is simply =$A$1  to bring the total number of blankets needed into that cell.

The formula in D5 is =C5/$B$2  to determine the number of boxes needed for the blankets made on that day.

The formula in E5 is =ROUNDUP(D5/$C$2,0)  to determine the number of pallets you need for the blankets made on that day.

The formula in G5 is =SUM(B$5:B5)/$B$2 to tell you how many boxes you have used so far, including the ones needed for that day's blankets made.

The formula in H5 is =ROUND(G5/$C$2,0) to tell you how many pallets you have used so far, including the ones needed for that day's blankets made.

Finally, we need this formula in cell C6 (not C5)   =C5-B6

That tells us how many more blankets have to be made to meet the goal.

Fill all of those formulas down the sheet for as far as you need them.  Then as you add the number of blankets made each day into column B, the other numbers will update automatically.

    My design doesn't use a singular table repository that combines all of the data entered into 5 separate tables. Access is made to use Forms for data ENTRY and to use Reports to provide a "hard-copy" of the accumulated data.

    The idea is that you should be able to:

    1.) Select an Employee on the main form

    2.) Enter or view errors made by that employee in the first subform.

    3.) Enter or view corrective actions taken for each error in the second subform.

    Because we are entering and/or viewing this data on a main form / subform basis ... that IS the visual cumulative collection of data.

    The advantage of this approach is that because the data is stored separately ... it's nearly infinitely expandable!

    (i.e Add another error for this Employee -- 6 months, a year, 5 years from now ... and have the ability to view a history of corrective actions taken for EACH error!) "Easy-peasy" to have a cumulative history at your fingertips ... rather than having to pull an employee file out of the file cabinet and wade through it searching for the pertinent details.

    The "Access advantage" is that you don't have the limitations of a spreadsheet, where cells grow, need to be re-sized, etc.

    Access reports fields have a Can Grow / Can Shrink property, have Sorting, Grouping, Header/Footer capability ... and much more.

    We'll talk more about creating reports and subreports a little later ... I would concentrate on getting the tables and relationships properly structured and checking the functionality of the form and subforms.

    PS.

    If you have the tables and relationships designed at this point ... design a query that pulls data from the various tables (as long as you have defined your relationships, it will be quite easy to construct your PK & FK joins.)

    When you view the datasheet view of this query ... it should be evident on how you can enter the Employee ONCE, and have his/her name associated with multiple Errors and CorrectiveActions. This will, in effect, be that combination of cumulative data that you're looking for.

    The downside of viewing a query as opposed to a report is that the stories that you've entered in your Memo fields will almost certainly appear to be truncated. The data *is* there, you just won't be able to read it all. As I said earlier ... that is the purpose of the report.

    To recap:

    Data is stored in table fields. 

    Access is a relational database, which allows for multiple tables and the ability to display relational data in various ways (Queries, Forms, Reports)

    Queries *can* be used combine and display data (as explained above), and can also be used as a basis for Forms and Reports ... *BUT* I prefer to use linked Forms/Subforms and Reports/Subreports based on the tables directly ... as (IMHO) the tendency of errors and omissions is greatly reduced.
