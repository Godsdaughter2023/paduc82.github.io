The following formula will sum the four rightmost cells in range P5:W5 that are not empty. Is that what you are looking for?

=SUM(MMULT(--(COLUMN(P:W)=LARGE(COLUMN(P:W)*(P5:W5<>""),{1;2;3;4})),TRANSPOSE(--(P5:W5))))

=SUM(MMULT(--(COLUMN(P:W)=LARGE(COLUMN(P:W)*ISNUMBER(P5:W5),{1;2;3;4})),TRANSPOSE(IF(ISNUMBER(P5:W5),P5:W5,0))))

jay freedman say:

In case it isn't clear from Doug's response, it isn't necessary to do the unprotect and reprotect manually; it can (and should) be part of your macro.

At the beginning of the macro, insert

   ActiveDocument.Unprotect

if the protection doesn't have a password, or

   ActiveDocument.Unprotect Password:="password here"

if it does (of course, put the actual password in the quotes).

At the end of the macro, insert

    ActiveDocument.Protect _
        Type:=wdAllowOnlyFormFields, NoReset:=True

or

    ActiveDocument.Protect _
        Type:=wdAllowOnlyFormFields, NoReset:=True, _
        Password:="password here"

Because it will (should) take less than a second to run the macro, there's no chance for the user to delete anything while the document is unprotected.

If you worry about users reading the code to discover the protection password, you can apply a password to the VBA module in the Tools > Project Properties dialog's Protection tab. If you do that, keep an unprotected copy for yourself in case you lose the password.

Assuming it is also working for you, themanindbox, now we can see from this sample that IE11 can run HTML4 markup "natively", with Silverlight.

The F12 debugger confirms IE is running in IE11 mode.  Not the infamous IE5 Quirks mode.  (Or nearly as awful IE7 Compatibility mode).

However, if you type  document.compatMode  in the F12 console, IE does not report the expected "CSS1Compat".  Instead, it reports "BackCompat".  This means IE11 is running in the so-called "Interoperable HTML5 Quirks Mode".

https://blogs.msdn.microsoft.com/ie/2011/12/14/interoperable-html5-quirks-mode-in-ie10/

So, it's still running in some sort of Quirks mode.

The reason it's running in that Quirks mode is because the webpage's !Doctype has omitted the dtd url.  You can see this in the F12 DOM Explorer tab, in the very top line of markup.  Here's that line:

<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN">

Compare with the TechNet docs I linked earlier.  They explain that this is interpreted by IE to be an instruction to run in a non-Standards mode.  (a "Quirks" mode).  In order to run in full Standards mode, those TechNet docs say it would need to include the dtd url, like this:

<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN" "http://www.w3org/TR/html4/loose.dtd">

hmm well, not quite right.  The MSDN !Doctype tables contradict that, saying it produces "Almost-Standards (IE7)" mode.  I checked.  It still gets the "BackCompat" this way.

This following HTML4 doctype is listed in those MSDN tables as producing "full-Standards" mode.  I checked and confirmed it does in fact give "CSS1Compat" this way:

<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.0//EN" "http://www.w3org/TR/html4/strict.dtd">

rajesh c. say:

Immediately I can think of the following 2 options to solve this:

(there can be other possibilities also)

1.  Try to see if you can put the totals above the Header row.

     This way the position of the Totals will not change and your formula will continue to give you desired result

2.  In case if (say for example) you have a some information in Column-D and say, you have a Total in 

     cell D157.

     Now, in cell C157 if you put a header as "TOTAL : " 

     then you can use a VLOOKUP formula to get the answer as follows:

     write the following formula in your total-worksheet

        =VLOOKUP("TOTAL : ",BDM!$C$1:$D$50000,2,0)    

                 here I have assumed that your data can go upto row no. 50000

    Give it a try - it should work.

Hope this is Helpful.
