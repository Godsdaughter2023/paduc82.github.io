hansv mvp say:

Let's say your table is named tblIn with a field named TextField. I'll assume that it also has a primary key field named ID that defines the sort order of the records.

Create a new table tblOut with a memo field MemoField (if you are absolutely sure that the combined values will never exceed 255 characters in length, it could be a text field).

Activate the Visual Basic Editor (Alt+F11), insert a new module (Insert | Module), and copy the following code into the module:

 

Sub CombineRecords()
    Dim dbs As DAO.Database
    Dim rstIn As DAO.Recordset
    Dim fldIn As DAO.Field
    Dim rstOut As DAO.Recordset
    Dim fldOut As DAO.Field
    Dim strOut As String
    ' Current database
    Set dbs = CurrentDb
    ' Clear target table
    dbs.Execute "DELETE * FROM tblOut", dbFailOnError
    ' Open recordset on source table
    Set rstIn = dbs.OpenRecordset("SELECT TextField FROM tblIn ORDER BY ID", _
        dbOpenForwardOnly)
    ' Source field
    Set fldIn = rstIn.Fields("TextField")
    ' Open recordset on target table
    Set rstOut = dbs.OpenRecordset("tblOut", dbOpenDynaset)
    ' Target field
    Set fldOut = rstOut.Fields("MemoField")
    ' Loop through records of source table
    Do While Not rstIn.EOF
        If fldIn Like "#.#*" Or fldIn Like "##.#*" Then
            ' Do we have an output value?
            If strOut <> "" Then
                rstOut.AddNew
                fldOut = Trim(strOut)
                rstOut.Update
            End If
            ' Start new output value
            strOut = ""
        End If
        strOut = strOut & " " & fldIn
        ' On to the next record
        rstIn.MoveNext
    Loop
    ' Process the remaining string
    If strOut <> "" Then
        rstOut.AddNew
        fldOut = Trim(strOut)
        rstOut.Update
    End If
    ' Clean up
    rstOut.Close
    Set rstOut = Nothing
    rstIn.Close
    Set rstIn = Nothing
    Set dbs = Nothing
End Sub

 

Substitute the correct names for the tables and fields.

With the insertion point anywhere in the code, press F5 to run it.

ossiemac say:

Firstly the Conditional format. You say you are using If statements which are generally not required in Conditional format. Instead of the following formula

 

=IF(A2=10,TRUE,FALSE)

 

Simply insert the following

 

=A2=10

 

Try both formulas in a cell on a worksheet and if A2=10 then both return True (or False if A2 is not equal to 10.)

 

Another trap in Conditional Format is creating multiple conditions. If you set up a Conditional Format for a cell (or range of cells) and then copy that cell and PasteSpecial -> Formats to another selected range then it creates another set of rules for the range to which it is pasted. However, if you include the cell with the Conditional Format in the Paste range then it simply expands the "Applies to" Range in the Conditional format. You can also expand the "Applies to" range by selecting the cell or range with the Conditional format and then select Conditional Format -> Manage rules and select the required condition and then click the icon at the end of the "Applies to" Range and then select the new range, including the cell or range to which it already applies.

 

To check if you have multiple rules set up, select then entire worksheet and then Conditional Format -> Manage rules.

Creating new controls on existing forms or creating new forms is not really recommended as part of an application. Not only does it involve a lot of extra work that can be avoided by, as Bill recommended, simply changing the Visible property of existing controls, but it's actually not possible to do if you've converted the front-end application to an .MDE/.ACCDE

marshall barton say:

Absolutely NOT.  A running app should never manipulate the design of heavy duty objects such as tables, forms, reports, macros or modules.  In addition to the problems Bill and Doug pointed out, saving a design change will cause a major amount of bloat and all the issues that leads to.  It can also be a cause of catestrophic corruption if anything goes wrong at that time or if two users just happen to do it at the same time.

 

Istead, precreate as many text boxes as you will ever need and name them Text0,Text1, etc and make them all invisible.  Then when you figure out how many you want to use (in the form's Load event), Use a little loop to make them visible and set any other properties as needed:

   For k = 0 To Num
       With Me("Text" & k)
          .Visible = True
            . . .
       End With
   Next k

This is less code, much faster and way, way safer than creating and deleting objects.

 

Side note:  The Create... methods are only intended for use in your own design helper wizard like procedures where you are the only one in the database.  If something goes wrong while you are changing the design of your database you are right there and can either fix it or restore from a recent backup copy without affecting anything users may be doing in the production copy you gave them to use..
