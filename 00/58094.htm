robert aldwinckle on forums say:

The remote name could not be resolved: 'choice.microsoft.com'

That's telling you that your DNS does not know that host name.   You can use  nslookup  to prove this.   And you can use nslookup to find out what the address is from another DNS server.  E.g. let's try the Google public one

nslookup  choice.microsoft.com  8.8.8.8

So, then you could use the IP address that you get from that to see if you have connectivity to that server.  For example, mine would have me try:

Invoke-WebRequest -URI "http://65.55.130.50/"

That result looks the same as what the original test showed.  And if I enter that URL in IE's Address bar I get to see the same thing as if I had just entered it using the host name.   Note that these days that is no longer that common but at least it could prove that you were reaching a functioning HTTP server.

So, if you can get a lookup with a different DNS that shows you an IP address and you can use the IP address it would prove that the problem you have is with your DNS only.  The troubleshooter is actually implying that.  A problem is that without any details to substantiate the claim we tend to discount it, especially since frequently it turns out to be a bad guess on its part for a different cause of an identical symptom.

 

FYI

Robert
