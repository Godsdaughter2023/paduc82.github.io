marshall barton say:

If the Vendors table's primary key field is an AutoNumber, then the foreign key field in the products table must be a Number datatype with the FieldSize property set to long integer.

Since an AutoNumber field should never be changed (and it would be extremely difficult to change it), there is no use for setting Cascade Updates.  Your vendor name should probably have a Unique index so it would be impossible to enter the same vendor name twice.

john w. vinson say:

The most common strategy for primary and foreign keys in Access is to use an Autonumber as the primary key, with a Long Integer as the foreign key; users would typically never need to even see these values, much less edit or enter them. In your case the Vendor name should simply not exist in your Products table! Instead the table would have a (meaningless to the user) number; but this doesn't matter, since the user will never be looking at table datasheets anyway.

Instead the user will work with a Form based on the Products table (or more complex forms, most likely). They would use a Combo Box bound to the VendorID foreign key, but displaying the vendor name. This way if vendor "Joe's Widgets" changes its name to "Joe and Jane's Widgets, Gimcracks and Gadgets" you only need to change the vendor name once, in the vendor table; your products table will remain as it is, just pointing to the correct current vendor name.

You can - if you insist - make the vendor name the primary key; but a Primary Key should have three characteristics: It must be unique (only one Vendor can have that name); it should be stable; and it should be short, for efficiency (Access can search a four-byte integer field much faster than a 50-byte text field). You can get around the "stable" issue by using cascade updates, but you can't be certain that you won't someday have to deal with two vendors who just happen to have the same name, and you can't make the field shorter than the longest name you'll ever need. And that's all just additional complexity to manage... and it's not necessary, since the numeric keys work just fine both for the computer and for the user in millions of databases all over the world.

ken sheridan say:

1.  So, what I plan to do is make the vendor name my primary key in the vendors table (so I never change it and there is only one unique name) and reference the vendor name in the products table.  This way both will be text datatype.

2.  As far as cascading updates go, I was under the impression that the matching field in the master table has to be a primary key.  Is that not true?  Based on what you said, it seems that I would never have to have a cascading update when using a primary key.  

1.  The use of a 'natural key' like this can sometimes have advantages in some situations, e.g. when building correlated combo boxes where the values listed in one are restricted by the value selected in another, and there is nothing in the database relational model which bars the use of natural keys.  Nevertheless, in the absence of any cogent factors which say otherwise, the use of an autonumber surrogate key is generally to be preferred on the basis of efficiency.

The fact that you use the vendor name as a primary key does not mean you can't change it; that only applies to an autonumber key.  So when using a natural key it's imperative that cascade updates are enforced.  The real question, however, is whether you can say with absolute confidence that there will never, ever, arise the need for duplication of vendor names?  I think most experienced developers would answer no to that question and would consider it unwise to use anything but a surrogate autonumber as the key in this context.

2.  I'm afraid that you are still misunderstanding the nature of keys.  A table can have more than one 'candidate key', but only one can be defined as the primary key.  It's not unusual to have an autonumber primary key (for efficiency), and another column which is a candidate key, and defined as such by its inclusion in a unique index.  The autonumber column would be the one referenced in any referencing tables.  Cascade updates would be pointless in this situation because the autonumber value will never change.  There would be nothing to stop the other candidate key being referenced by a foreign key column in a referencing table, however, and on very rare occasions it is necessary to do so, in which case cascading updates would be enforced.

ken sheridan say:

.....................remember reading that I shouldn't include a field in two tables unnecessarily.

That's actually a slightly different, albeit allied, issue, one of normalization.  Normalization is the process whereby redundancies are removed from a table, or at least reduced as far as is possible.  Say we have two tables like this;

States
....StateID  (PK)
....State

Cities
....CityID  (PK)
....City
....StateID  (FK)

And then a table like this:

Contacts
....ContactID  (PK)
....FirstName
....Lastname
....AddressLine1
....AddressLine2
....CityID  (FK)
....StateID  (FK)

This table would be wrongly designed as the StateID is 'functionally determined' by the CityID, i.e. once we know the city we know what state the contact is in.  Telling us in this table that San Francisco is in California is redundant.  This is not merely inefficient, it leaves the table open to the risk of update anomalies as there would nothing to stop us entering a row where we put San Francisco in Colorado say.

In the language of the database relational model the table would not be normalized to Third Normal Form (3NF) because it contains a 'transitive functional dependency'.  For a table to be in 3NF all non-key columns must be determined solely by the whole of the primary key.  In this case StateID is determined by CityID, however, and is transitively dependant on the key; ContactID determines CityID, which in turn (transitively) determines StateID.  To put this table into Third normal Form we simply remove the StateID column, which we can do with no loss of information.  This process of normalizing a table in this way is known as 'non-loss decomposition'.

scottgem say:

Just to add, there are some data types in Access that have a slightly different correlation when used as a key. Autonumber is the prime example. When adding a foreign key to a table that will be joined to an Autonumber, the correct datatype (as noted) is a Number with a Long Integer sub type (under Fieldsize). 

Some other examples are the Boolean data type, which is an integer underneath and a DateTime which is a Number/Double underneath. Though it would be rare to use these as keys. Just wanted to help you with understanding datatypes more.
