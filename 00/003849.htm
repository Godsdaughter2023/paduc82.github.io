scottgem say:

In a split database, if you have to change the underlying tables, it can affect a great deal. It can change your queries, reports, forms and code. In some cases the forms, reports and queries may update automatically, but in others they may not, so you really have to check everything. 

Also, changes to the structure mean shutting access to the database while the changes are being made. 

ken sheridan say:

I assume that by 'modifying the structure' you mean changing existing table designs, adding new tables or deleting existing tables.  With a database split into 'front' 'back' ends you only have to do this to the single back end to which the front ends link.  What changes will be necessary to objects in the front end will depend on what changes have been made to the back end, so the amendments to forms, reports queries etc could be few and simple, or they could be many and more complex.  However, you won't have to amend every single user's copy of the front end; you should keep a 'master copy' in which you make the changes.

It's then just a case of replacing each user's existing copy of the front end with a copy of the new one.  If you have already linked the master copy to the back end then you should not need to do any more, but even should you move the location of the back end it's a simple task to refresh the links to the new location by using the built in Linked Table Manager in each front end.

It goeswithout saying that before doing any of this it's imperative that both the back end and the master copy of the front end are backed up to at least one secure location.

john w. vinson say:

Can someone point me to information on 1) how to successfully modify the data structure and 2) what effect changing the data structure has on existing queries, forms, etc.?  Cannot find it in the "Access 2010...the missing manual".

I see you've gotten good answers on what's involved in modifying table structures after the split... but I wonder if you're assuming that you'll have to make a lot of changes when you split.

You won't! A split database usually requires no changes at all to existing forms, queries or reports. They'll just use linked tables rather than local tables.

And it's actually much easier to modify forms, reports, and queries in a split database; you will just have a "master" .mdb or .accdb frontend file (that you keep in a safe place); each user will get their own copy of the frontend, usually "compiled" into a .mde or .accde file so they can't (accidentally or intentionally) mess with the structure. When you need to improve a form or a query, you can just make another copy of the master and make the changes in that copy. It's often helpful to temporarily link it to a "sandbox" copy of the backend so you can test out forms which update the tables, without damaging production data. When you have a new improved version of the frontend, you would relink it to the production back end, create the .accde, and delete everyone's old .accde and replace it with the new improved version.

Backup, backup, backup of course, at every stage!

They are both temporary directories used by windows to update your system.

It is safe to delete both to reclaim valuable space.

Microsoft has not updated the disk cleanup tool to delete them yet (but you can delete it manually).

To delete them you need to change permissions first.

Right-click start, Command Prompt (admin)

Copy and paste each set of commands and press enter

You might need to press Y - to delete and change permissions

If it does not work and you get "Access Denied" errors - repeat the instructions again

takeown /F C:\$Windows.~BT\* /R /A
icacls C:\$Windows.~BT\*.* /T /grant administrators:F
rmdir /S /Q C:\$Windows.~BT\

takeown /F C:\$Windows.~WS\* /R /A
icacls C:\$Windows.~WS\*.* /T /grant administrators:F
rmdir /S /Q C:\$Windows.~WS\

I am not sure about  $Sysreset - but you can probably use the same instructions above just change it to c:\$Sysreset
