Then the columns displayed in the Combo Boxes change. The first column is always the Primary Key, which is hidden. 

The second column is always what the user is explicitly looking for. If it is, say,  the "Name",  [LAST NAME, FIRST NAME] column is the second column and the first visible one.

graham mayor say:
Add the following macro to the ThisWorkbook module of the personal.xlsb workbook which stores global macros

Private Sub Workbook_Open()
    Application.OnTime TimeValue("03:55:00"), "QuitMacro" 'set the time to a time before the reboot
End Sub

and in another module in Personal.xlsb you need the macro to close your named workbook and quit Excel

Sub QuitMacro()
Dim xlBook As Workbook
For Each xlBook In Workbooks
    If xlBook.Name = "workbookname.xlsx" Then
        xlBook.Save
        xlBook.Close
    End If
Next xlBook
Application.Quit
End Sub


This assumes that you only have one workbook left running overnight. If you have more than one, then you will need to close them all in similar fashion.

ossiemac say:

I cannot think of a reason why you would design a project with multiple workbooks with Userforms in each workbook.

 

Personally if I have multiple workbooks then I think that it is preferable to have all of the code and Userforms in one workbook that does not contain any data. You can then simply address all of the ranges/cells in the other workbooks containing the data.

 

However, having said that, if you wanted to show a userform in another workbook then you would need to call a sub in that workbook that that will run and show the workbook.

 

The following code goes in the workbook with the Userform. (The workbook name that this is in is called   Example Book2.xlsm )

 

Sub ShowUserform()
    UserFormBook2.Show vbModeless
End Sub

 

Then in the workbook from which you want to show the userform, you call the above code like the following.

 

Sub ShowUserFormAnotherWorkbook()
    Application.Run ("'Example Book2.xlsm'!ShowUserform")
End Sub

 

Note that the single quotes are required around the workbook name if it has spaces in the name.

 

All code associated with the Userform needs to be in the workbook containing the Userform. I don't know of any way of addressing the individual controls within a Userform that is in another workbook.

ossiemac say:
I am only using PERSONAL to develop and test the utility. Later, it will be in a workbook that will go to the client.           


So you are developing exactly the same way that I do but instead of creating another workbook, you are using Personal.xlsb to hold all of the code and Userforms. Therefore why not try shifting the code out of Personal to another workbook because the code I gave you is tested and works. I should think that you will have to do this eventually to distribute because you can't expect your clients to have their Personal workbooks (if they have them) to be over written with your project one.

 

By the way the sub that is being called needs to be in a standard module otherwise the module name also needs to be included in the call. For instance if I placed the ShowUserform code in Sheet1 module then the code would be like the following.

 

Application.Run ("'Example Book2.xlsm'!Sheet1.ShowUserform")

 

Only speculating but this might be the problem in Personal and you need to nominate the module.
