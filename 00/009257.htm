a. user say:

Correct, an embedded object does not exist outside of Word.

The "best" way? Difficult - it depends.

There are two main ways to retrieve data from an Excel sheet using a field code with modern, Windows desktop versions of Word.

 a. Use a LINK field.

 b. use a DATABASE field. 

Of the two, only (a) lets you get what is basically an image of all or part of an Excel sheet. (b) is only any good if you have tabular data within the Excel sheet. It just brings across the data and lets you use a number of predefined table formatting features in Word. 

With (a), I believe you run into the problems as I described in the other thread at https://answers.microsoft.com/en-us/msoffice/forum/msoffice_word-mso_winother-mso_2010/word-mergemail-send-if-category-wildcards-not/af96cd63-1335-4e37-a4af-9a4dcee99651 . Is that what you found? I cannot say I have tried very hard to make it work.

IMO if that cannot be made to work it is probably worth making a request to have it fixed via word.uservoice.com (although it is difficult to imagine that Microsoft is unaware of the problem, they may have decided that not enough people need the feature to work.

With (b), you do not get automatic link updates as you should be able to get with (a) if you can get (a) to work at all). But a quick test here suggests that the field code works whether you provide the local pathname of the Excel file or the remote OneDrive pathname. So that might be an option in some cases.

(Just to explain, with an Excel worksheet, the DATABASE field will issue a SQL query via the Jet/ACE OLE DB provider to retrieve data from a sheet, like this, if you want all the data from a worksheet named "Sheet1"

{ DATABASE \d "the full path name of the .xslx with backslashes doubled up" \s "SELECT * FROM `Sheet1$`" \h }

DATABASE lets you apply various predefined table styles.

DATABASE is likely to cause problems if you have mixed data types in any of your Excel columns (e.g. text and numbers in the same column).

)

Another issue you may have to consider is the more basic one of coping with the possibility of multiple simultaneous updating of the same Excel workbook (even without having anything linked to it).

There are other non-field ways you can get data from Excel...

 a. copy/paste (no link). Redo when necessary

 b. use code such as VBA to automate Excel's object model

 c. use code such as VBA and the Jet/ACE OLE DB provider to retrieve tabular data from a .xlsx

 d. use code to open the .xlsx via the Office Open XML SDK and retrieve the data that way.

I suspect (d) is not really on the agenda (I have never done it nor see anyone else do it, but doubtless it is possible)

(a) is the only one of those that would bring over existing formatting (although you could probably do it with some effort with (b) and possibly (d)).

(c) basically does the same thing as the DATABASE field, but you would have  bit more control over certain things.

Here is a modified macro that
- fetches the data from columns B, C, D, I, J, K
- accumulated the values for North and East, starting with Northing and Easting
- highlights the cells for the tow points in the closest pair
(for the given example it is row 5 for line 1 and row 7 for line 2, with a distance of 9.0165958

Sub find_closest_distance()
  ' this macro finds the closest distance between curves built up from point in R3
  ' each curve consists of a collection of points with given coordinates
  ' the Z-coordinates are absolute while the X- and Y-coordinates are relative
  ' the result shows the pair of points, one from each curve, that are closer than any other pair

  ' the Z, X, Y coordinates for the points belonging to "curve" 1 are in columns B, C, D starting on row 5
  ' no other data in these columns
  ' the Z, X, Y coordinates for the points belonging to "curve" 2 are in columns I, J, K starting on row 5
  ' no other data in these columns
 
  With Worksheets("Sheet1")        'change if needed
    x1 = "C": y1 = "D": z1 = "B"   'change if needed
    firstrow1 = 5                  'change if needed
    lastrow1 = .Cells(.Rows.Count, x1).End(xlUp).Row
 
    x2 = "J": y2 = "K": z2 = "I"   'change if needed
    firstrow2 = 5                  'change if needed
    lastrow2 = .Cells(.Rows.Count, x2).End(xlUp).Row
 
    northing = .Cells(3, "F")      'change if needed
    easting = .Cells(4, "F")       'change if needed
 
    min_distance2 = 1E+300
    x1_acc = northing: y1_acc = easting
    For i = firstrow1 To lastrow1
      x1_acc = x1_acc + .Cells(i, x1): y1_acc = y1_acc + .Cells(i, y1)
      x2_acc = northing: y2_acc = easting
      For j = firstrow2 To lastrow2
        x2_acc = x2_acc + .Cells(j, x2): y2_acc = y2_acc + .Cells(j, y2)
        dist2 = (x1_acc - x2_acc) ^ 2 + (y1_acc - y2_acc) ^ 2 + (.Cells(i, z1) - .Cells(j, z2)) ^ 2
        If dist2 < min_distance2 Then
          min_distance2 = dist2
          min_i = i: min_j = j
          min_x1_acc = x1_acc: min_y1_acc = y1_acc: min_z1 = .Cells(i, z1)
          min_x2_acc = x2_acc: min_y2_acc = y2_acc: min_z2 = .Cells(i, z2)
        End If
      Next j
    Next i
    MsgBox "Then miniumum distance is " & Sqr(min_distance2) & " from point (TVD = " & min_z1 & ", North = " & min_x1_acc & ", East = " & min_y1_acc & ") to point (TVD = " & min_z2 & ", North = " & min_x2_acc & ", East =  " & min_y2_acc & ")"
    .Cells(min_i, x1).Interior.ColorIndex = 3: .Cells(min_i, y1).Interior.ColorIndex = 3: .Cells(min_i, z1).Interior.ColorIndex = 3
    .Cells(min_j, x2).Interior.ColorIndex = 3: .Cells(min_j, y2).Interior.ColorIndex = 3: .Cells(min_j, z2).Interior.ColorIndex = 3
  End With
End Sub
