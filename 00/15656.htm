Either use:

 

To show all (then use Sort and Group on your report)

 

SELECT [Orders]![DK] AS DKFromOrders, DKDATA.TxtName, DKDATA.Supplier

SELECT [Orders]![DK] AS DKFromOrders, DKDATA.TxtName, DKDATA.Supplier

ken sheridan say: What Wayne has given you are queries in fact.  Queries are written in SQL (Structured Query Language).  When you design a query visually in the query designs  it builds an SQL statement for you, which you can see by switching the view to SQL.   Each of the SQL statements Wayne posted is a query, and if you open the query designer and switch to SQL view, you can paste one or the other of them in, and this will create a query for you, assuming the table and column (field) names used are exactly as in your two tables.  Switch to SQL and the query should open.  Switch to design view and you'll see it in the format you are probably more accustomed to.

Assuming the query opens successfully you can save it under a suitable name and then use it as the basis of a report.  However, you don't need to save the query at all in fact.  You can simply copy the SQL statement to the clipboard, and open the report designer and then paste the SQL statement in as the report's RecordSource property in its properties sheet.  If you want to use the report wizard to build the report rather than hand-craft it, however, you will have to save the query first.

The second query Wayne posted includes a simple parameter which will prompt you for the DK value when the report on which the query is based is opened.  This is a cheap and cheerful method of restricting a report to a single DK value.  A far more user friendly method, however, is to use a reference to a control on an unbound parameter form as the parameter.  This will often be a combo box from which you can select an item.  A button on the form will then open the report restricted to the selected item.  A more flexible approach is to use a multi-select list box, in which you can select one or more items to restrict the report to however few or many you want,  You'll find an example as Multiselect.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file includes two methods, one which filters the report in code in the button's Click event procedure by using the WhereCondition argument of the OpenReport method, the other which restricts the report's query by means of a parameter.  In the former case the IN operator is used against a list of values, but in a query the IN operator does not accept a value list as a parameter, so this is simulated by using a couple of functions published by Microsoft.

BTW, note how the list box's RowSource property, itself an SQL statement, concatenates the FirstName and LastName columns into a single column to return the full name.  This gives a more meaningful list to select from.  The values use d to filter the report are in the first column of the RowSource, EmployeeID, however, which is hidden by setting the first dimension of the list box's ColumnWidths property to zero.
