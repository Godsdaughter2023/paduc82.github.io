macropod say:

The basic issue with inserting your tags before tables is that you can't have the final </b> hanging there without a paragraph break following. So, either Word would have to insert one, or it does the next best thing - it inserts the tag in the last possible place before the table, which just happens to be before the paragraph break. The reason it 'works' when a table isn't involved is that you have other paragraphs into which the tags can be (and are) inserted.

There is one anomaly, though, and I don't have an explanation for it: when a partially-bolded string occurs in one of your paragraphs before a table, you get two sets of tags: one that spans all except the last character in the string, then another for just the last character. I'd call that a bug, but: (a) I don't know that it is of any material consequence for the html output; and (b) it can easily be worked around by cleaning up pairs of end:start tags afterwards.

ken sheridan say:
What about the "Me!" term. Is it correct to apply it here?

Probably not.  The Me keyword returns a reference to the current instance of a class in which VBA code is running.  As you appear to be applying a data macro to a table, it would therefore not be appropriate in this context.  Try omitting the Me! completely.

I can't comment further on the use of an expression like this in a data macro, however, as I'm using an earlier version of Access.  I note that you are using this in the context of the AdterInsert event however.  I suspect the expression would always evaluate to True as it will detect the row which has just been inserted.  In a data entry form this would normally be done in a form's BeforeUpdate event procedure, conditional on the form's NewRecord property being True, and the Cancel argument of the procedure would be set to True if an existing row were detected.  To use the AfterInsert event you'd need to test for the count of matching rows being greater than 1.
