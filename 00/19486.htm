dianeoforegon say: Exchange Basics

The Inbox syncs more frequently than any other and should be kept as clean as possible. You mentioned (5000+) messages. That's the problem!

Any folder with more than a few thousand messages is going to take some time to fully come down when you first sync your account. Due to the design of Outlook's sync engine (and how exchange works), you will see the 512 newest messages in a folder at the initial start of sync then the rest of the folder's contents will be back filled before you see any mail that arrives after this point. This is why it will look like you are not getting new mail when a folder is still going through initial sync. This is also further exacerbated by an Exchange issue where it gets unnecessary change events before the newer mail arrives. If you carefully watch the progress during the sync of a folder with greater than a couple thousand messages, you will see "Updating local..." a lot before the newer mail starts arriving (newer than the initial 512 that come down).

Outlook checks the server for updates every minute, any folder that has updates will subsequently be synced. Since Outlook has a limit on how many folders can be synced at a time, there can be a queue of folders waiting. The Inbox does get high priority so it will generally sync before other folders that also need to sync.

Smallest numbers that can be represented in Excel

joeu2004 say: benjamj wrote:
What is the lowest positive value that Excel will recognize as a number?

Although Microsoft documentation says "[s]mallest allowed positive number[:]  2.229E-308", once again Microsoft documentation is incorrect.

Usually, the smallest computable number that Excel represents is 2^-1022, which Excel displays correctly as about 2.22507385850720E-308.

And indeed, Excel permits us to enter that number.  However, Excel misrepresents the value internally, and it incorrectly displays the value as 0.00E-01 [1].

So 2.22507385850721E-308 is the smallest positive number that we can enter reliably; that is, that Excel represents correctly internally and displays correctly.

You can use the N() function to protect yourself from smaller input that Excel does not convert to a numeric value.  For example, N(A1) returns zero if A1 contains the text "1E-330"; but it returns the correct numeric value if A1 is numeric.


-----
[1] Excel converts 2.22507385850720E-308 to a so-called "denormalized" representation internally, a value less than 2^-1022.  The difference is small enough that initially I thought it is just a rounding error in the conversion algorithm.  And I thought that 0.00E-01 is just Excel's way of denoting "denormalized" values.

In fact, Excel permits us to enter numbers as small as 1.11253692925361E-308 -- or so it seems.

But below 2.22507385850721E-308, the internal "denormalized" representation becomes increasingly incorrect.  For example, 1.11253692925361E-308 is actually represented internally as about 1.87744945419674E-322.

Moreover, at some point below 2.22507385850721E-308, Excel displays 0.00E+00 for "denormalized" representations instead of 0.00E-01.
