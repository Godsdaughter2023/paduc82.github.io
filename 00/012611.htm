AXRussell wrote:

Does anyone know the mathematical formula behind the Excel NPER function?

Why do you ask?  School assignment?  Idle curiosity?  Something else?

The PV help page (click here) [1] documents the formula:

It is important to note that "pv", "fv" and "pmt" are signed cash flows; for example, negative for outflows, and positive for inflows.

Also note that "rate" is the periodic rate per unit of "n".  For example, if "n" represents months and the annual rate is 12%, the

Finally, in that formula, "type" is presumed to be zero (payments at the end of periods) or one (payments at the beginning of periods).  But one.

So solving that formula for "nper", the usage is demonstrated as follows:

The formula in B7 is =NPER(B1,B2,B3,B4,B5).  It is provided as a check of the formula below.

The formula in B6 is:

=IF( B1=0, -(B3+B4)/B2,

        LOG( (B2 * (1+B1*(B5<>0)) - B4*B1) / (B3*B1 + B2*(1+B1*(B5<>0))) ) / LOG(1+B1) )

or stylistically:

=IF( rate=0, -(pv+fv)/pmt,

        LOG( (pmt * (1+rate*(type<>0)) - fv*rate) / (pv*rate + pmt*(1+rate*(type<>0))) ) / LOG(1+rate) )

That's a mouthful!  We can simplify if we know that "rate" is non-zero and "type" is zero or non-zero:

type=0:

=LOG( (B2 - B4*B1) / (B3*B1 + B2) ) / LOG(1+B1)

or

=LOG( (pmt - fv*rate) / (pv*rate + pmt) ) / LOG(1+rate)

type<>0:

=LOG( (B2*(1+B1) - B4*B1) / (B3*B1 + B2*(1+B1)) ) / LOG(1+B1)

or

=LOG( (pmt*(1+rate) - fv*rate) / (pv*rate + pmt*(1+rate)) ) / LOG(1+rate)

Note that B4 (fv) is typically zero, which leads to further simplication:  the expression -B4*B1 can be omitted.

You might note that the calculated "nper" is not exactly 360 in either case (B6 and B7).  The primary reason is:  we rounded "fv" to simplify the example.

But even if we hadn't, "nper" might not be an exact integer due to anomalies of binary arithmetic.

1. why do I get this connectivity error when I'm running both on the same machine?

Because your network connection is still required. Try unplug your network cable, or if wifi, turn it off. You then find that your network stack is now disabled. That why we often have to run what is called a loopback connector. It simulates that you have an active network running when in fact you have none.
 
In some cases, replacing local host with the name of your computer can eliminate this issue.
 
So, you are still using a network connection to the server and despite it running local, you still communication through a network protocol with tc/ip etc. That system will thus still often require that your computer is running with an legitimate network connection.
 
2. is there a way I can manage this connectivity issue? Should I use stored procedures instead of linked tables?
 
Not ideal, but
you can trap a logon error with sql server using DAO and you can avoid having that odbc message come up.
 
How this trick is done is outlined here:
 
ACC2000: How to Trap ODBC Logon Error Messages
http://support.microsoft.com/kb/210319
 
I use the following code based on the above KB article:
 
Function TestLogin(strCon As String) As Boolean
 
   On Error GoTo TestError
 
   Dim dbs          As DAO.Database
   Dim qdf          As DAO.QueryDef
 
   Set dbs = CurrentDb()
   Set qdf = dbs.CreateQueryDef("")
 
    qdf.Connect = strCon
    qdf.ReturnsRecords = False
 
    'Any SQL statement will work below.
    qdf.SQL = "SELECT * FROM calls"
    qdf.Execute
 
    TestLogin = True
   
    Exit Function
 
TestError:
    TestLogin = False
    Exit Function
 
End Function
              
 
To call the above, I would go:
 
 
            strCon = "ODBC;DRIVER={sql server};" & _
               "SERVER=" & ServerName & ";" & _
               "DATABASE=" & DatabaseName & ";" & _
               "UID=" & UserID & ";" & _
               "PWD=" & USERpw & ";" & _
               "APP=Microsoft Office 2003;" & _
               "WSID=" & WSID
 
            If TestLogin(strCon) = True Then
               ' logon is ok!

rrobinsoncpa say:

I am trying to add on to an existing program powered by VBA. This program can build an array 4 columns wide and anywhere from 1-50 rows long. I want to build a set of VBA commands that will detect the length of the array, and copy the populated rows from the active workbook to another workbook (which may be open or closed), detect the next open row, and post the results into the next open row of the other workbook. The places where I am having issues are:

  • Detecting the length of the 4 column array on the fly
  • Opening a workbook if it is not open

I believe that I have the commands for detecting the next open row within the open workbook.

Any help would be greatly appreciated.  

dave peterson say:

If you know your data, you may be able to use a column (it doesn't have to be
part of the range to copy, but usually it will be) to determine the last used
cell in that column -- and the last row of that table/range/array.

Dim LastRow As Long
Dim Wks1 As Worksheet
Dim Wb2 As Workbook
Dim wks2 As Worksheet
Dim RngToCopy As Range
Dim DestCell As Range

Set Wks1 = ActiveWorkbook.Worksheets("Somesheet")  'where do you get the data?

'check on 2nd workbook
Set Wb2 = Nothing
On Error Resume Next
Set Wb2 = Workbooks("otherworkbook.xls") 'no drive/path here!
On Error GoTo 0

If Wb2 Is Nothing Then
  'it's not open!
  Set Wb2 = Workbooks.Open("C:\my path\otherworkbook.xls") 'include path here!
End If
'there was no check to see if that file actually exists!!

Set wks2 = Wb2.Worksheets("anothersheet")
'again, no check to see if that sheet actually exists!!!

With wks2
  'I used column A to find the next row
  Set DestCell = .Cells(.Rows.Count, "A").End(xlUp).Offset(1, 0)
End With

'get ready to copy
With Wks1
  'I used column A to find the lastrow
  LastRow = .Cells(.Rows.Count, "A").End(xlUp).Row
  'A1 through D (lastrow)
  Set RngToCopy = .Range("A1:D" & LastRow)
End With

RngToCopy.Copy _
   Destination:=DestCell

========
All untested, uncompiled.  Watch out for typos!
