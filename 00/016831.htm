Function GetMail(ByVal S As String) As String
  'Return the mail address from S
  Dim i As Long, j As Long
  'Search the "@"
  i = InStr(S, "@")
  If i = 0 Then Exit Function
  'Search the left blank
  j = InStrRev(S, " ", i)
  If j = 0 Then j = 1 Else j = j + 1
  'Search the right blank
  i = InStr(i, S, " ")
  If i = 0 Then i = Len(S) Else i = i - 1
  GetMail = Mid(S, j, i - j + 1)
End Function

ken sheridan say: Assuming the form in question is bound, firstly you have to ask yourself this question:  Are all the columns whose values you are assigning to columns in the form's underlying table time-independently determined by the key of the referenced table?  You only assign a value to a column in the referencing table if the answer in respect of the column in question is No.  Otherwise you should simply reference the value in an unbound control, or include the referenced column in the form's query and bind a control to this.  Scott's blog will undoubtedly illustrate this.

In your case the requester's name is almost certainly time-independently determined by the key of the referenced table as people do not usually change their names over time, so you should not store the name in the referencing table, only the foreign key which references the primary key of the table of people.  The other columns' values, i.e. phone number, email and address can change over time, so in the case of these you then have to ask yourself whether the referencing table to which the form is bound needs to retain the values for these which were current when the row was inserted into the referencing table regardless of subsequent changes.  If the answer is Yes, then you do need to assign values to columns in the referencing table; if No, i.e. rows in the referencing table should reflect any subsequent changes to these value in the referenced table, then you should not, but simply reference them in the same was as with the name, storing only the foreign key value in the referencing table.

Should you decide that it is legitimate to assign the values (bar the name of course) then I'm at a loss why you are experiencing the behaviour you describe, viz '.......if I select a record that has data in each field then a record without data in a field, that field's data remains the same as the first record'.  If you navigate to an empty new record in the form and select a requester from the combo box bound to the foreign key column, then there is no reason that this should occur.

I've assumed above that your form is bound and you are entering data into a new row, but even if the form is unbound the same would apply.  With an unbound form, there would be no reason to assign the values to controls, however, and they should simply be referenced, but even in this scenario the behaviour which you are experiencing should not occur.

First, let me answer what a ControlSource is as it bears on the other answers to your questions. A ControlSource tells the control what to display.  Not all controls have Contolrsources (labels and buttons for example). but where a control has a Controlsource property, that property can be set either to a column in the Form's Recordsource or an expression to calculate a value. 

So when you set a control to a value in code, that suggests that the ControlSource is a column in the Recordsource. And if you are entering values from another table into the RecordSource, you have redundancy.

Controls should be bound if you want to store the data in the underlying table. Otherwise, you have to code the I/O between the form and tale yourself. With a bound form and controls, Access handles the I/O for you. 

Time dependent data is different from what you describe. Generally data like contact info is overwritten with new info, but its overwritten in the table where that info is stored. So, in your scenario if Joe Smith's phone number changes, you would want to change in the Contacts table, not in the table where you are indicating Joe Smith as the contact. Why would you need to know what Joe's phone number was 2 years ago? If you really do need that, then create a history table to save changes. Certainly there is to save contact info in another table. It can be displayed on the form without creating redundancies.

As for Subforms, this is one of the more powerful features of Access. As a general rule, it is not a good idea to base a form on a multi-table query, especially if that form will be used for data entry. This is where subforms come into play as you can embed a subform onto your main form and enter data into multiple tables that way. Also subforms are very useful in displaying 1: Many relationships. For example, you can have a main form bound to a Customers table and a subform showing all their orders in rows.

ken sheridan say:
I think I may only partially understand your first "time-independent" question. 

Let me illustrate it with one of the most common scearios in databases, that of an ordering database.  The basic model for this is:

Customers----<Orders----<OrderDetails>----Products

In this the OrderDetails table models the many-to-many relationship type between Orders and Products by resolving it into two one-to-many relationship types.  The Customers table will have a column such as CustomerName, assuming these are corporate customers rather than people.  This might change over time, e.g. The 'Widget Supply Company' might become 'Widgets R Us'.  It's still the same customer so the name is stored once and once only as a value at the CustomerName column position in a row in the Customers table and in no other table.  Once it's changed, wherever the name of that customer appears in the database it will now be 'Widgets R Us'.  The CustomerName column is functionally determined solely by the key of Customers.

The Products table will include a column UnitPrice.  This can, and almost certainly will, change over time.  If it was stored once and once only as value at the UnitPrice column position in a row in Products and in no other table, then every time the product appears in an order in the database it would have whatever is the current UnitPrice value for the product.  This would clearly be wrong as orders prior to the price change should reflect the price at the time the order was made, not the current price.  So in this case UnitPrice is also a column in the OrderDetails table, and is entered when an order is created by looking up the current unit price from Products.  Any subsequent changes to the unit price will not affect the value in the row in OrderDetails, so the price remains what it was when the order was made.  In this case the UnitPrice column in Products is functionally determined solely by the key of Customers, and the UnitPrice column in OrderDetails is functionally determined solely by the key of OrderDetails.  Nothwind has an example of how the price is assigned in its order details subform's module, but it's unnecessarily long-winded.  A simpler example can be found in my demo file InvoicePDF.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

An invoicing model and ordering model are pretty much the same.  In this little demo file the AfterUpdate event procedure of the ProductID combo box in the fsubInvoicedetails subform has the following code to assign the unit price and tax rate to the current row in the InvoiceDetails table:

Private Sub ProductID_AfterUpdate()

    Dim ctrl As Control
    
    Set ctrl = Me.ActiveControl
    
    Me.UnitPrice = ctrl.Column(1)
    Me.TaxRate = ctrl.Column(2)
    
End Sub

This little file's primary purpose is to demonstrate how to output a report as a PDF file, but it also illustrates how the data is entered via a main form which includes a set of subforms.

The process which governs the correct structure of the tables in a relational database is known as Normalization.  There are a set of Normal Forms starting at First Normal Form (1NF) and progressing upwards, each of which applies a further level of 'rules' which determine if a table is normalized to the normal form in question.  These rules are based on the functional dependencies of the columns in the table.  You should certainly familiarize yourself with the normal forms up to and including Third Normal Form (3NF) at least.  This is not to say that it's unnecessary to normalize a table beyond 3NF, but a table in 3NF will in most cases also be normalized to the higher normal forms.  The Wikipedia article on the subject provides a relatively straightforward introduction to the topic:

http://en.wikipedia.org/wiki/Database_normalization
