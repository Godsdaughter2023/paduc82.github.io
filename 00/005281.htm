sockatume---01 say: Unfortunately the standard SMS alphabet is limited to a particular set of characters. To send Greek characters (or most non-Latin languages and special characters) the phone has to switch to a different encoding, which uses twice as much data per character. All SMS services have this limitation, e.g. on Android and iPhone. Some phones work around it by sending the messing in "Greeklish" and translating "Greeklish" messages in return or other tricks, but this varies.

esentis13 say:

Before this firmware installation, everything was ok with Greek. Is there any way to install a fix about that?

sockatume---01 say:

Try Messaging -> Options -> Settings -> Text Message -> Character Encoding. It should read "full support", which seems to be the UTF-16 mode. Switch it to "Reduced Support". It should let you use 160 characters per message.

I guess they changed the default when they did the update. With "full support" it'll use UTF-16 and you have half the characters, but all receivers will be able to represent the text correctly. "Reduced support" doesn't work with all recipients but gives you more characters.

Let me know if this doesn't work ,a bit of searching suggests it doesn't work reliably on certain handsets.

How Do You Capture Unplanned "Soft" Hours in MS Project? - Microsoft Community

stephaniew777 say:

How Do You Capture Unplanned "Soft" Hours in MS Project?

They way we create our schedules today are based on physical work that is easily tracked.  We have hours required to produce drawings and documents, lead times for procured items, weekly meetings, milestones, etc.  What we seem to be missing are the unplanned "soft" hours.  We want to be able to track them and show how they impact the overall project progress.  By soft hours I mean activities like: unscheduled meetings with suppliers or customers; issues arise during production and time needs to be taken follow up; excessive time spent responding to comments, etc. 

 

What is a good way of handling this within the schedule?

julie sheets say:
I would add the unplanned meetings etc to the schedule as they occur and book the actual time against them.  As they are unplanned I don't think it makes sense to try to predict them.

rod gill say: Whilst you can't predict a particular meeting, you can predict an average number of hours spent on "soft" tasks per week. So if a resource averages 10h/week on soft tasks, only allocate them at 75% for all other tasks and 25% to a generic soft task.
This is the only time you might assign a resource to a summary task, as the time expands/contracts to match sub-tasks (not wise for beginner schedulers!).
So some weeks you will get ahead of schedule and other weeks behind. Track the average hours per week and update remaining soft tasks accordingly.

joe blake - pmp, mcp say: I've stuck with the old adage 'when in doubt, leave it out'.  Meetings and extra tasks don't belong in a project schedule.  They can interfere with critical path, allocation, etc.  We always keep an 'action item' list and a communication plan separate from the schedule.  You can keep this on a simple list in SharePoint or some other collaboration tool.  Remember, a project schedule is not a task list.  When you focus your schedule on deliverables and WBS specific items you will get much more value out of it.  Good luck.

dale howard [mvp] say:

Stephanie --

 

Yes, team members could use the Add New Task functionality in Project Server to track the unplanned tasks on which they work.  When the team member creates the task, he/she will need to give the task a name, pick a summary section in the project, enter a planned Start date (which is probably the date they actually started the work on the task), and enter either a Finish date or the number of hours they need to work on the task.  The PM will need to approve the task in the Approval Center in PWA, then open the project in Microsoft Project and decide if the task should be linked with any other tasks.  From there, the PM can see the overall impact of the team member working on the unplanned task, and if the task causes a resource overallocation, the PM can level the overallocation appropriately.  Hope this helps.
