BUG: Windows 10 network device IP caching on downed interfaces - Microsoft Community

alan brown ucl say:

BUG: Windows 10 network device IP caching on downed interfaces causing operational problems

In our network, we assign the same static IP to authenticated users on a machine whether they're using wireless or ethernet.

Unfortunately...... Windows 10 caches the IP address it last had on any given interface and HOLDS it when the interface is down. 

Previous versions of Windows did not.

If that interface goes down and that IP address is offered via DHCP on another interface, it goes through the entire DHCP request and ACK process for that IP, before deciding it's unsuitable and failing back to a link-local address - IE, it looks like a sucessful address assignment on the DHCP server side, but the WIn10 host is using a 169.254/16 address for no apparent reason

The result is that seamless transitions from ethernet to wifi and back again (or to a docking station) are no longer possible.

This is arguably a bug. A downed interface has no IP information and should not be preventing another interface from using that IP address. This is especially true for interfaces which are using dynamically assigned/server assigned IP addresses (DHCP).

Eventually (15 minutes or so) the cached IP data on the old interface evaporates and the new interface is able to get its IP and have it "stick".

The only way to shortcut this timeout is to disable and reenable the downed interface. Disabling causes cached IP data to be flushed instantly.

Please don't tell me it is supposed to work this way. Previous versions of windows did not have this behaviour, nor do other OSes. It was bad enough when windows wouldn't allow manually assignd IP addresses on a downed interface to be used on active interfaces (which is also broken behaviour), this new behaviour is not standards compliant,  breaking established systems and resulting in support calls from aggravated users.

It has taken quite a while to work out what the issue was, as users and first line staff were diagnosing it as an intermittent fault with no apparent cause.

What ETA can Micrfosoft offer to remedy this bug?
