How can I do this?

scottgem say:

Should give you what you want.

ken sheridan say: If you really want to store the structured value in a single column, called TransactionCode in the example below, you can insert the value automatically with the following code in the form's Current event procedure.  If the sequential numbering is overall:

    Dim strYear As String
    Dim strDay As String
    Dim lngSeqNo As Long
    
    If Me.NewRecord Then
        strYear = Format(VBA.Date, "yy")
        strDay = Format(VBA.Date, "y")
        lngSeqNo = Nz(DMax("Right(TransactionCode, 4)", "YourTableName"), 0) + 1
        
        Me.TransactionCode.DefaultValue = """" & strYear & "-" & _
            strDay & Format(lngSeqNo, "0000") & """"
    End If

Or if the sequential numbering is per year:

    Dim strYear As String
    Dim strDay As String
    Dim lngSeqNo As Long
    Dim strCriteria As String
    
    If Me.NewRecord Then
        strYear = Format(VBA.Date, "yy")
        strDay = Format(VBA.Date, "y")
        strCriteria = "Left(TransactionCode,2) = """ & strYear & """"
        lngSeqNo = Nz(DMax("Right(TransactionCode, 4)", "YourTableName", strCriteria), 0) + 1
        
        Me.TransactionCode.DefaultValue = """" & strYear & "-" & _
            strDay & Format(lngSeqNo, "0000") & """"
    End If

Note, however, that this could lead to conflicts in a multi-user environment if two or more users are inserting a new row simultaneously.  In this context you should trap the index violation error in the form's Error event procedure, incrementing the sequential number by 1 until no error occurs.  This is more easily done if the structured key is made up of three separate columns as I originally advised.  You'll find examples of this in CustomNumber.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the example for 'Sequential Numbering or 'Sequential Numbering by Group' would be the appropriate models to follow, depending on whether the sequential numbering is overall or per year.

However, there is another important issue to be considered.  If the table already contains a column with the date of the transaction (or whatever) as its value, then to store the year and day of the year in the structured key would be an unnecessary and incorrect approach.  The two values can, in this scenario, easily be computed from the date with the Format function and concatenated to the sequential number whenever required.  Storing the numbers in the table would introduce redundancy and thus prejudice the integrity of the database by leaving it open to update anomalies.

ken sheridan say: From your description it sounds like you are relating each meeting to one 'principal' contact, and recording multiple contacts as participants.  If so:

You need to model the many-to-many relationship type between meetings and participants in the conventional way by means of a table which resolves the relationship type into two one-to-many relationship types like this:

Meetings---<MeetingParticipants>---Contacts

The MeetingParticipants will have foreign key columns such as MeetingID and ContactID which reference the primary keys of the Meetings and Contacts tables respectively.  Its primary key is a composite one made up of both columns, or if you give the table a separate surrogate key, the two columns, as a candidate key, should be included in a unique index.

The Meetings table will also need a separate ContactID or similar foreign key column to reference the primary key of the Contacts table, to which I assume the parent form is bound.  This column will be the LinkChildFields property of the meetings subform control.

The meetings subform will need to be in single form view if you are to embed the MeetingParticipants subform in it, linked on MeetingID, but you could have the meetings subform  in continuous forms view if you wish and have the  MeetingParticipants subform as a separate correlated subform.  For a simple demo of correlated subforms, using data from Northwind as an example, see CorreledSubs.zip in my public databases folder at:

Password protection using the File > Save As dialog does not encrypt. It is light-duty protection that can keep most ordinary users from opening a workbook without having a password.

Encryption is the heavy-duty stuff. To use encryption you must have a rights management server available to you. Here is the Information Rights Management guide for Office 2011. There isn't a specific guide for Office 2016 for Mac, but the same concepts apply.

http://www.microsoft.com/en-us/download/details.aspx?id=20825

jay freedman say:

It appears that all the text in your document is contained within a field of some kind, possibly the field that creates the table of contents. 

Press the Alt key and the F9 key together and release them. That changes the display of fields from their "results" to the code they contain. The field code(s) will appear within bold curly braces, { }.

If the last closing curly brace appears after the text of the document, select the part that should be plain text, cut it to the clipboard, and then paste it at the end, after the closing brace.
