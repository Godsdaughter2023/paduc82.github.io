scottgem say:
I'm missing something here because you seem to be missing two tables. How do you record a shipment? The products table is fine, the employees table is fine (except you should be using an autonumber as the PK in that table). But the customers table should just be info about the institutions you send books to. You need a Shipments table to record each shipment and a shipments details table for each item shipped (unless you only send one book per shipment). 

Products and customers should NOT be directly related. They should be related ONLY through the shipments tables. The salesperson can be related directly to a customer (if a salesperson always deals with the same customer or with the shipment. That's a business rule that you need to answer. 


ken sheridan say: I too am having some difficulty understanding just what's being asked of you here.  The Products table more or less makes sense, though why Competency Number should be chosen as the name for its primary key I can't imagine.  Why not just BookID (note the lack of a space in this BTW; spaces in object names, while allowed in MS Access, are frowned upon by experienced developers).  Anyway, that's not the real issue.

The inclusion of a foreign key Competency Number column in Customers makes no real sense at all.  It assumes that a customer can be related to only one product (book).  This is known as a unary relationship type.  I'd have expected the relationship type to be many-to-many, i.e. each customer can relate to one or more books, and each book can relate to one or more customers.  This is known as binary relationship type.  For this there would be no Competency Number column in Customers, but the relationship type would be modelled by a table with two foreign keys, each referencing the primary key of the two main tables, e.g.

CustomerProducts
....CustomerID
....Competency Number

The Customers table would include a CustomerID column as its primaty key.  For convenience this can be an autonumber.

When it comes to Employees, like you, I'm assuming this will relate to the Customers table, i.e. will record each Customer's employees.  This is a genuine unary relationship type (one-to-many), assuming no body can be an employee of more than one customer simultaneously, so simply needs a CustomerID foreign key column in the Employees table, referencing the primary key of Customers.

There is another possible interpretation, however, which is that Employees does not relate to Customers, but is a distinct entity type representing the employees selling products to a customer.  The relationship type would then be ternary (3-way) between Customers, Products and Employees, so Employees would not have a CustomerID column and instead of the CustomerProducts table you'd have as follows:

ProductSales
....CustomerID
....Competency Number
....EmployeeID

There could well be other columns in this table such as SaleDate, Quantity (the number of copies of the book sold) etc., i.e. these columns represent attributes of each sale

The important thing is to understand just what in real worlds terms you are being asked to model,  even though the real world is a hypothetical one here.  The whole basis of the database relational model is that it models a part of the real world in terms of its entity types and the relationships between them, with each table modelling an entity type ( a relationship type is just a special kind of entity type, so tables modelling relationship types are in fact modelling entity types).  Columns in each table model the attributes of the entity type.  One you have a good mental picture of how the part of the real world with which you are concerned is made up in this way, the process of translating this into tables and relationships in the database is a relatively straightforward one.

If any of yours have been modified, edit the Target by removing the
