Prevent Duplicate Records in Many to Many Join Table

What I'd like to do is check for duplicates upon assigning a member to an event.

 

I use my main form with the table Members as the record source and a subform that uses the Join form as its record source.  I then use a combo box in the subform to look up the actual event to assign to the member and assign its Event PK in the Join table.  It all works perfectly, however...

 

As I can have Tom (ID of 1) assigned to many events and the event Army Ball (ID of 1) assigned to many members, I don't want to have Tom assigned to the Army Ball multiple times.  What is the best way to prevent multiple identical associations between the same member and event?

Thanks

Dan 

boblarson say:

In the form's BEFORE UPDATE event you can use:

 

 

If DCount("*", "EventTableNameHere", "[MemberIDFieldName]=" & Me.ControlWithMemberIDHere & " AND [EventIDNameHere] = " & Me.ControlWithEventIDNumberHere) > 0 Then

   MsgBox "This member has already been assigned to this event." , vbInformation

   Cancel = True

   Me.Undo

End If

john w. vinson say:

In addition to the BeforeUpdate event Bob suggests, you can absolutely prevent duplicates getting in from anywhere by creating a unique two-field index on the MemberID and EventID fields of the junction table.

Open the table in design view, and use the Indexes tool. Put some distinctive name for the index in the left column (you usually won't ever need to use this name but it must have one); select the two fields that row and the next row of the indexes toolbox; and check the Unique checkbox.

 

john p spencer say:

You can absolutely forbid it by using a unique index against the combination of MemberID and EventID. 


To create a multiple field unique index  (Compound index)
--Open up the table in design mode
--Select View: Index from the menu
--Enter a name for the index in first row under Index Name
--Select one field in the index under Field Name
--Set Unique to Yes
--Move down one line and select the next FieldName
   (Do NOT skip rows, do NOT enter the index name again)
--Continue moving down and selecting fieldnames until all needed fields are included.
--Close the index window and close and save the table

I would still use some code in the entry form to detect the atttempt at duplication, so I could give the user a better "error" message than the one generated by the database.

naming a print area - Microsoft Community

yveke say:

naming a print area

I have inherited a spreadsheet which has several print_areas named. Can I delete them? They have the print_areas pointing to various large areas on the worksheets. I can't understand how I would incorporate this in a formula or use such a large area or why they would name it. Can anyone tell me how you would use such a large named print area or how you would use it? Thanks....
vijay a. verma say:

1. To Delete - Go to Formulas Tab > Name Manager. Here, you can delete / edit them.

2. Use - Uses are very much context specific and use is limited to our imagination / creativity.

In your context, it seems it is related to print areas i.e. what areas to print. If you click on red area in Page Layout tab, you will reach Page Setup tab. Here, you can specify print areas under sheet tab by separating with commas. If you reopen this page setup again, you will notice named ranges are converted into ranges. Hence, excel will print only these areas.

Now the question is why use named area rather than cell range themselves. There are two possible reasons (in my view) :

1. Names are easy to remember rather than cells.

2. Names are also used for dynamic ranges. Hence, sometimes range is A1:G50 and sometimes it is A1:G58. In this case, you simply specify names and range will automatically be inputted. (Creation of dynamic range is a separate topic in itself).
