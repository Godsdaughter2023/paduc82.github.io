swischner say:
OCR'ing handwriting from a bitmap image is something that up to now NO program has been able to do (Well, Evernote sort of does, but with bad results in normal handwriting with connected letters). Handwriting recognition always required that the handwriting has been done in the corresponding app (like S-Note from Samsung, MyScript on all mobile devices and so on). The reason is, that the lines have to be stored as vector graphics (to "know" that you had drawn a circle and take it for an "O" or "0" instead of tracerouting a bunch of pixels to guess that it could be a circle). It may sound strange, but to some point handwriting recognition (from a bitmap image, e.g. a scan) is more complicated than voice recognition.

Now, so much for the bad news. There is good news too: Microsoft is doing a lot in the field of Artifical Intelligence recently. Handwriting recognition from bitmaps is a part of it and they included it in their online services called "Cognitive Services". And they have been putting this technique into OneNote as well. For now it's still "a little experimental" and limited to English language but it should work (I experimented a bit with it in OneNote 2016 and even wrote a blog article about it, but it's in German: http://onenote-blog.de/handschrifterkennung-jet...
Maybe if you copy the whole article to the clipboard and let deepl.com translate it, it will make a little sense.
When they introduced it two years ago, it was limited to ... now it comes ... notes that were taken with Office Lens, the Livescribe Smartpen, the Web Clipper OR ... and that could be your solution ... sent by mail to "me(at)onenote.com".
Apparently this restriction is still true, so I don't know about the planned future of this function.

Baseline: It's a little miracle that it works at all; not even very expensive OCR programs have been able to do bitmap-handwriting to text so far.

gustavbrock say:

It sounds like you are on two different locations, and data are replicated via DFS.

If so, the UNC path is virtual, meaning that at either location it refers to a local server, not the same physical server.

That won't work for simultaneous use as DFS for each modification at either location will try quite quickly to update the other location not respecting a change at the remote location.

You will have to revise this setup. Either run a remote desktop connection to one location, or update the backend to a server engine like SQL Server which the fronted access via ODBC.
