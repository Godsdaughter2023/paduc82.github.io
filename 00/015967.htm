ken sheridan say: As Scott Diamond says in the article at the link I posted, "….it should give you enough to expand on for more complex needs".  So it's down to you to build on his basic methodology.  The code he cites in the article relies on the 5th column of the login combo box to determine the user's access level, so is very limited.  To vary a user's Access levels to different objects, and conversely an object's to different users you could model the many to many relationship type between users and objects in a table with a composite key of UserID and ObjectName columns or similar, and a non-key column AccessLevel.  When a form is opened the code would look up the Accesslevel value in the row in this table for the current user, obtained from the login form, and the current form and by means of a Select Case statement, assign the values to the relevant properties.  Bear in mind, though that the AllowEdits property is a very blunt instrument , and locks all editable controls in the form, including things like unbound navigational combo boxes.  I would not use it, but would tag the bound controls and set their Locked and Enabled properties in a loop through the form's Controls collection.

daniel pineault say:

Creating a custom security setup to replace ULS will be as involved as your security needs to be, ie: simple login control is straightforward, but controlling access to individual forms can be very complex.  In all cases, coding is required.

My setup include the following tables:

Permissions -> Listing of all the database objects (forms, reports) and what permissions (open, edit, delete, filter, ...).  I end up creating multiple entries per object (read, read/write, read/write/delete)

Groups -> Listing a permission groups, can be simple or complex depending on your needs (read, read/write, read/write/delete Or could be task driven, IT, Management, Data Input Users, Database Administrator, ...)

Groups-Permissions -> A simple cross-reference between which permissions are assigned to each group

Users -> A listing of all the database users, password, isActive, ...

Users-Groups -> A cross-reference of which groups each user is part of

Then I created a couple simple procedures that are called by any secured objects to (i) ensure the user has the permission to open the object in question (ii) set the relevant objects properties (allowedits, allowdeletions, ...) based on the user's permissions.

I also use the User table as part of the login process and automatically locks out people who have no business even opening the database.

There is a considerable amount of work and testing required to get things running smoothly, but once you have the template setup, it only take a few minutes to implement in any project.

                   strNewfile As String, _

                   Optional blnShowMessage As Boolean = False)

       strMessage = "File successfully copied."
   Else
       strMessage = "File copy failed."
   End If
   
   If blnShowMessage Then
       MsgBox strMessage, vbInformation, "Copy File"
   End If
   
End Sub

   ' change value of following constant to 3704
   ' when using .accdb files
   Const FILEINUSE = 3356
   Dim dbs As DAO.Database
   Dim strMessage As String
   Dim strBackUpTemp As String
   
   ' if back up file exists get user confirmation
   ' to delete it
   If Dir(strBackUp) <> "" Then
       strMessage = "Delete existing file " & strBackUp & "?"
       If MsgBox(strMessage, vbQuestion + vbYesNo, "Confirm") = vbNo Then
           strMessage = "Back up aborted."
           MsgBox strMessage, vbInformation, "Back up"
           Exit Sub
       Else
           ' make temporary copy of backend file and then delete it
           strBackUpTemp = Left(strBackUp, InStr(strBackUp, ".")) & "bak"
           MakeFileCopy strBackUp, strBackUpTemp, False
           Kill strBackUp
       End If
   End If
   
   On Error Resume Next
   ' attempt to open backend exclusively
   Set dbs = OpenDatabase(Name:=strBackEnd, Options:=True)

   Select Case Err.Number
       Case 0
       ' no error so proceed
       dbs.Close
       Application.CompactRepair strBackEnd, strBackUp
       Debug.Print Err.Number
       If Err.Number = FILEINUSE Then
           ' file in use by current user
           strMessage = "The file " & strBackEnd & _
               " is currently unavailable. " & _
               " You may have a table in it open."
           MsgBox strMessage
           ' rename temporary copy of back up file
           ' if exists, back to original
           If Dir(strBackUpTemp) <> "" Then
               MakeFileCopy strBackUpTemp, strBackUp, False
               Kill strBackUpTemp
           End If
           Exit Sub
       Else
           On Error GoTo 0
           ' ensure back up file created
           If Dir(strBackUp) = Mid(strBackUp, InStrRev(strBackUp, "\") + 1) Then
               strMessage = "Back up successfully carried out."
               ' delete temporary copy of back up file if exists
               On Error Resume Next
               Kill strBackUpTemp
               On Error GoTo 0
           Else
               strMessage = "Back up failed."
               ' rename temporary copy of back up file
               ' if exists, back to original
               If Dir(strBackUpTemp) <> "" Then
                   MakeFileCopy strBackUpTemp, strBackUp, False
                   Kill strBackUpTemp
               End If
           End If
           MsgBox strMessage, vbInformation, "Back up"
       End If
       Case FILEINUSE
       ' file in use - inform user
       strMessage = "The file " & strBackEnd & _
           " is currently unavailable. " & _
           " It may be in use by another user."
       MsgBox strMessage
       ' rename temporary copy of back up file,
       ' if exists, back to original
       If Dir(strBackUpTemp) <> "" Then
           MakeFileCopy strBackUpTemp, strBackUp, False
           Kill strBackUpTemp
       End If
       Case Else
       ' unknown error - inform user
       MsgBox Err.Description, vbExclamation, "Error"
       ' rename temporary copy of back up file
       ' if exists, back to original
       If Dir(strBackUpTemp) <> "" Then
           MakeFileCopy strBackUpTemp, strBackUp, False
           Kill strBackUpTemp
       End If
   End Select
   
End Sub


Public Function GetBackEnd()

     Dim tdf As DAO.TableDef
     Dim strBackEnd As String
     
     For Each tdf In CurrentDb.TableDefs
        If tdf.Connect <> "" Then
            strBackEnd = tdf.Connect
            strBackEnd = Mid(strBackEnd, InStr(strBackEnd, "DATABASE=") + 9)
            Exit For
        End If
    Next tdf

If you want to create a series of cumulative backups then leave the code as is.

You'll then need to add a function to perform the backup by calling the BackUp procedure from the module, e.g. if overwriting a backup each time

PublicFunction CallBackUp()

   Const SOURCE_DB = "F:\Databases\MyDatabases\MyFile.accdb"
   Const BACKUP_DB = "G:\Databases\MyDatabases\MyFile_bak.accdb"

   BackUp SOURCE_DB, BACKUP_DB

   Application.Quit

End Function

or if creating a series of backups by building a new name each time, for which I'd suggest incorporating the current date/time in the format yyyymmddhhnnss as this sorts chronologically:

PublicFunction CallBackUp()

   Const SOURCE_DB = "F:\Databases\MyDatabases\MyFile.accdb"
   Dim strBackup As String
 
   strBackup = "G:\Databases\MyDatabases\MyFile_bak" & _
       Format(Now(), "yyyymmddhhnnss") & ".accdb"

   BackUp SOURCE_DB, strBackup

   Application.Quit

End Function

Then create an autoexec macro in the Access file which calls the CallBackUp function.  Test it by closing it down and opening it again.  All things being well it should create the backup and then close itself down.

In the Task Scheduler just work your way through the prompts for a basic task, entering a name for the task, frequency etc. and get it to run a program, entering your BackUp.accdb file as the program to run.

While this should work, I would personally recommend using a commercial back-up application such as Acronis.  Applications which are specifically designed for backing up include features for automatic scheduling along with back-up validation etc. so are a more robust solution than a simple file copy.

And one final point:  it's strongly advisable to undertake several back-ups each time to different locations, on different physical drives if possible, or at least to different partitions.  Relying on a single back-up is not really a safe option, as the single back-up can become corrupted or the drive on which it is located can fail completely.  This very thing did once happen to me.

    rick rothstein, mvp - excel say:
    The tilde is the "escape" character for special and wildcard characters (meaning placing it before a such a character makes that character a normal character). So, since the tilde is itself a special character, put one in front of it makes it a normal character. So, use two tilde characters (~~) instead of one when you want to find a single tilde.
