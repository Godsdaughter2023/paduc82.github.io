Your personal address  "*** Email address is removed for privacy ***" is an IMAP account. On an IMAP account, if you have  set up Outlook to retrieve new emails automatically upon startup and open the Inbox automatically, you will see each messages is duplicated. You need to click: Tools-> Accounts-> Advanced and change this setting:
Sync all IMAP folders every x minutes.

Moreover, you  need to click: Tools-> Accounts-> Advanced-> Folders and then click "Don't store a copy of sent messages".

If the issue  cannot be fixed,  I suggest checking your rules because if you forward your emails from one PC to another, then the forwarding rule can create loops where messages are continually bounced back and forth between them, creating an infinite number of duplicates.

1. Start Outlook.

2. On the Tools menu, click Rules.

3. In the Name colunm, select a rule that you created.

4. Click to clear Enable.

5. Repeat step 3 and 4 until all rules have no Enable option to clear.

6. Test Outlook.

7. If the issue is resolved, enable one rule at a time, and test Outlook after each step.

Sorting a report Group not working - Microsoft Community

ctdak say:

Sorting a report Group not working

Using Access 2013 and VBA - I have a report in my app that was created a long time ago (by me) and it has one group.  The report groups activity by "=[AccountNo]" with no problem, but now I want it to also sort the grouped results by "=[AcctLastName]", but no matter what I do the sort won't work.  Any thoughts?

ken sheridan say: You said in your original post 'now I want it to also sort the grouped results by "=[AcctLastName]"'.  That would mean there must be multiple account holders per account.  If adding a lower sort order on AcctLastName does not affect the report's output, this suggests that this is not the case.

Are you really saying that you wish to order the report by AcctLastName and, if a customer can hold multiple accounts, then byAcctNo for each customer?  If so then AcctLastName must be the first group level.  If AcctLastName contains personal names as values, however, then you need to go beyond this and sort first by AcctLastName , and then by CustomerID, or whatever the key is, and finally by AcctNo.  The CustomerID group level should have a group header and/or footer.  This is because personal names can legitimately be duplicated, and are therefore unsuitable as keys.  Sorting by name and grouping by the key thus returns any customers of the same name separately, with each individuals account data also reported separately.  If a report has only a personal name as a group level above the account group level then account data for two or more persons of the same name will be incorrectly intermingled.

The other possible scenario is that each person can hold only one account.  If this is the case then an AcctNo group level is pointless as each group will inevitably be a group of one.   In this situation the only group levels required are AcctLastName  and CustomerID.

Note BTW that a 'group level' does not necessarily imply that data is 'grouped' to return a single instance of duplicated values.  Data is 'grouped' in this way only where a group level has a group header and/or footer,  Where a group level has no header or footer the data is simply sorted.  In the current report design interface 'add a sort' simply adds a group level without header or footer.
