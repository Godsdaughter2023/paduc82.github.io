scottgem say:
First, you need to stop thinking cells. Access does not have cells. Access has fields which are part of records. You cannot attach anything to fields in a record. You CAN attach files to a record.


Access has a data type called the Attachment datatype. toy can add a field as an Attachment datatype to a record and use that to attach files to records.


The alternative and the way I recommend, is to use a child file to store the location of the file you want to attach in a text field and reference it that way.

It depends how the image files are named and stored.  Also you'd need to use VBA, not a macro.  If the files are in a single folder for instance, and if you wish to assign one file only to each row in a table, and if the rows in table can be sorted in the same order as the files in the folder, then you can iterate over the files in the folder with the Dir function, and through a recordset of the table's rows sorted accordingly, assigning the path to each file to a column of text data type in each row, as recommended by Daniel, not an attachment column.

For more information on the Dir function see:

https://docs.microsoft.com/en-us/office/vba/language/reference/user-interface-help/dir-function

If you only want to store one image per row in a table, the basic Images_Simple.accdb file in the zip archive should suffice.

ken sheridan say: I tend to agree with John that rather than reinventing the wheel, making use of Outlook's existing functionality makes a lot of sense, but if you do want to use Access, read on.

The format for use on other devices really depends what functionality you want in the address books on those devices.  If you want a simple list in alphabetical order which people can scroll through then I'd suggest you create a report in the Access database and export this as a PDF file.  This can be opened on pretty much anything.

For the Access database structure you'd need a model comprising tables along the following lines.  I've used UK geographical units below, but you'll be able easily to translate these to whatever is appropriate in your case.

Contacts
....ContactID  (PK)
....FirstName
....LastName
....AddressLine1
....AddressLine2
....CityID
....PostCode

PhoneTypes
....PhoneTypeID (PK)
....PhoneType

To model the many-to-many relationship type between Contacts and PhoneTypes:

ContactPhones
....ContactID  (FK)
....PhoneTypeID (FK)
....PhoneNumber

The above table is 'all key', i.e. its primary key is a composite one of all three columns.

Cities
....CityID  (PK)
....City
....CountyID  (FK)

Counties
....CountyID  (PK)
....County

For data input you'd use a Contacts form, in single form view, in which would be a phones subform in continuous forms view, the subform being based on the ContactPhones table and linked to the parent form on ContactID.

A report would be based on a query which joins all the tables and is grouped by contact.  All contact name and address data would be in a group header, with the phone numbers in the detail section.

I've assumed a simple scenario where each contact has one address only.  If it is necessary to record multiple addresses per contact then a separate related Addresses table would be needed, with a foreign key ContactID column to reference the key of Contacts.  Addresses would then be entered via a subform in the Contacts form.

To expand on what John's suggested:

When a shape has NO fill, PowerPoint doesn't see it as clickable.

If it has a fill, even if the fill is totally transparent, PPT sees it as clickable, even though the human doesn't see any fill.

So:  Give the shape a solid fill rather than no fill, but set it to 100% transparent.
