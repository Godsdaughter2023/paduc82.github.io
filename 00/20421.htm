Why is Access 2010 Not Compatible with 2007?!?! - Microsoft Community

jjmadero say:

Why is Access 2010 Not Compatible with 2007?!?!

Question says it all. After two weeks of working in 2010, put my access file on a computer with Access 2007 and low and behold, Microsoft does it again. Database refuses to open in 2007 even after installing SP2 on it....not sure where to go from here. My access file doesn't even have any really complicated components that should make a 2 year old product obsolete (very mad over this one).
nathan o. - microsoft support say:

Hi JJmadero,

 

The two versions are compatible with each other.  The problem people run into is they use controls in the new database that are not compatible with Access 2007.

 

Backward Compatibility between Access 2010 and Access 2007 - http://msdn.microsoft.com/en-us/office/cc907897.aspx

 

In addition to this list another common control specific to Access 2010 that will cause the database not to open in Access 2007 is an empty cell control on a form or report along with VBA code behind these specific forms or reports.  These empty cell controls are new to Access 2010 and usually get added to forms or reports when they are flipped to layout view (I think they are added to maintain formatting in layout view and to preserve formatting if it is published to the web).  So if you used any of the wizards to create a form or report, including the quick form or quick report button, these controls most likely were added because it creates the form and automatically opens it in layout view.  Then if you add any VBA code to the form or report, you will make the object and / or database unusable in Access 2007 or you will no longer be able to save it as an .mdb file.

 

To see an example of this, you can do the following:

+>Create a new .accdb file in Access 2010.

+>Create a table with two or three fields (number and text fields are fine) and add some sample data.

+>Save and close the table.

+>Select the table you saved within the Navigation pane.

+>Click on the Create tab and click on the “Form” button.  This should create a form in layout view.  At this point, you most likely won’t have an empty cell control.

+>While in layout view, delete one of the labels on the form.  After doing this, you will see a control with dashed lines around it.

+>Look at the properties for this control, you will see it is an “Empty Cell” control.

+>Add a command button to the form and put a simple VBA message box behind it (msgbox “test”).

+>Save the form and close it.

+>Try saving the database as an .mdb file by clicking on the File menu -> Save & Publish -> Select Access 2002 -2003 format -> Click the Save As button.  You most likely will receive an error due to the form that contains the empty cell control and code.

 

To correct the issue:

1. You must open your form in layout view in Access 2010

2. Delete the empty cells and save the form

3. Create a new database in Access 2010

4. Within Access 2010, import all of your objects from the previous database into the new database

 

Note:  You will need to make sure you do this with any forms and reports that you modified within Access 2010 and / or opened in layout view within Access 2010.

 

Unfortunately, to ensure you don’t run into issues like this, it is good practice to develop your database in the lowest version of Access your users will be using.

 

Best Regards,

Nathan O.

How to create a timer (VBA) - Microsoft Community

thomas. say:

How to create a timer (VBA)

Does someone know a trick to create a timer that is called say once a second or so? 

With the Windows versions of Office this was easily accomplished using the Windows API, but I couldn't find a way with Word for Mac (2011).
daniel g. grau say:
Word has an OnTime facility to program macro execution. The shortest interval is 1 second. Here's an example:

'----------------------------------------------

Sub BackgroundStart()

Application.OnTime When:=Now + TimeValue("00:00:05"), Name:="BackgroundRun"

Message$ = "Background process has been scheduled"

End Sub

 

Sub BackgroundRun()

Message$ = "Background process is running"

Application.OnTime When:=Now + TimeValue("00:00:05"), Name:="BackgroundRun"

End Sub

 

Sub BackgroundStop()

Application.OnTime When:=Now, Name:="BackgroundDummy"

Message$ = "Background process has been stopped"

End Sub

 

Sub BackgroundDummy()

'Nothing here

End Sub


'----------------------------------------------

"Start" will just schedule macro execution (the "Run" sub) and would be, for instance, in a toolbar button.


"Run" contains the bulk of your code and schedules itself. You can go as short as 1 second.


"Stop" interrupts the scheduling by programming another macro that does nothing, and it does it right away (Now). This could also be in a toolbar button.


"Dummy" is blank, but required, as the only way to stop a scheduled macro is by scheduling another. Only one macro at a time can be scheduled (a macro pending execution).
