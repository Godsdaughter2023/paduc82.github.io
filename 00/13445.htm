ken sheridan say: 'Foreign key' is not a data type; it means that this is the column which references the primary key column of the referenced table.  So if you have a table Sites with a primary key column SiteID (more than likely an autonumber) then in the Notes table you'd have a SiteID column of long integer number data type (NB not an autonumber in this case).

A column is not defined as a foreign key in the table design.  If every note relates to a site then you should create a relationship between the Sites and Notes tables and enforce referential integrity.  For data entry you have a couple of options:

1.  You could use a form based on the Sites table and include within it a subform based on the Notes table, linking the subform to the parent form on SiteID as its LinkMasterFields and LinkChildFields properties.

2.  Alternatively use a separate form based on the Notes table with text box controls for the NoteDate and Note fields and a combo box for the SiteID field set up as follows:

ControlSource:    SiteID

RowSource:     SELECT SiteID, SiteName FROM Sites ORDER BY SiteName;

BoundColumn:   1
ColumnCount:  2
ColumnWidths:  0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first dimension is zero to hide the first column and that the second is at least as wide as the combo box.

You can select a site by name from the combo box's list, but its underlying value will be the numeric SiteID.

You might want to set the DefaultValue of the SiteDate column to Date().  That will automatically insert the current date when a new note record is entered.  The date can be manually changed by the user, though.
