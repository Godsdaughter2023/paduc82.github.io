nonames---01 say:

Send Calls Right to Voicemail

I have a lumia 900.  Is there a way to send certain calls right to voicemail? I use my cell phone for my business and I get telemarketing calls on occasion (Business phones are exempt from DNC).   It's nice to send future calls right to voicemail.  My android phone used to have this feature and I have tried with Nokia to no avail.  Thanks!!! 

kosh---01 say:

 

Hi nonames,

 

Thanks for your post and thanks for posting.

 

Please use the forum search function. Often, as in this case, your question will have been asked (and answered) before. See the forum if you have any questions on posting.

 

Call forward features are a function of the network and as such need to be supported by the network for your contract. In the most recent firmware there is an additional settings option where these can be setup through a menu. (Settings>network).

 

In most cases call divert can be setup as follows:

 

 

Feature Mode Sequence
All calls Activate & register number ** 21 * Number # SND
Deactivate & deregister number ## 21 # SND
Activate to registered number * 21 # SND
Deactivate # 21 # SND
Verify *# 21 # SND
No answer Activate & register number ** 61 * Number # SND
Deactivate & deregister number ## 61 # SND
Activate * 61 * Number # SND
Deactivate # 61 #
Verify *# 61 # SND
Unreachable Activate ** 62 * Number # SND
Deactivate ## 62 # SND
Verify *# 62 # SND
Busy Activate ** 67 * Number # SND
Deactivate ## 67 # SND
Verify *# 67 # SND
All types Deactivate ## 002 # SND
All conditional types Deactivate ## 004 # SND

 

If you want to extend the time before a divert activates you can add **[time in seconds] after the number. So to activate divert for all calls to number 12345 after 30 seconds you would enter:

 

** 21 * 12345 ** 30 # SND

 

 

The codes mentioned are what most operators use, but they can be different so check with your operator. Also if these codes are not supported by your network you will get a 'Unknown request' or ' request denied' from the network as a response.

 

Hope this helps,

Kosh

ken sheridan say: The root of the problem is that you appear to have a fundamentally flawed design.   Having multiple Boolean (yes/no) columns in a table, where each represents a different value of an attribute, is what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

What you have is a many-to-many relationship type between two entity types, one being whatever your current table is modelling, the other being the entity type of which each Boolean column represents an attribute.  A many-to-many relationship type is modelled by a table which resolves it into two one-to-many relationship types.

Recasting the data into a correctly normalized set of related tables is not too difficult, and you'll find a simple example as UnencodeColumns.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

Once you have the correct tables it is then a simple task to output the data in the layout you describe by means of a report which includes a multi-column subreport with across-then-down column flow.  My file includes a simple example.  An alternative approach is to concatenate the values into a string expression using a VBA function.  Allen Brown has one at http://allenbrowne.com/func-concat.html and you'll find another, using the very efficient GetString method of the ADO recordset object in Concat.zip in my same OneDrive folder.

If you do want to input the data via check boxes it's possible to do so in the context of a correctly normalized structure by means of an unbound subform; VolunteerRota.zip in my OneDrive folder illustrates this as well as the conventional bound form/subform approach, which is the usual method of data entry with this sort of model.

dhookom say:

I agree with Ken (as usual). You haven't provided much information about your checklist table however...

Let's assume your table was a checklist of family members' readiness for a vacation. Create a normalizing union query with SQL like:

SELECT PrimaryKeyField, "Packed" as Check
FROM tblFamilyVacation
WHERE [Packed] = -1
UNION ALL
SELECT PrimaryKeyField, "Used Bathroom"
FROM tblFamilyVacation
WHERE [UseBathroom] = -1
SELECT PrimaryKeyField, "Passport"
FROM tblFamilyVacation
WHERE [Passport] = -1
SELECT PrimaryKeyField, "Cell Phone"
FROM tblFamilyVacation
WHERE [CellPhone] = -1
SELECT PrimaryKeyField, "Bathing Suit"
FROM tblFamilyVacation
WHERE [BathingSuit] = -1
SELECT PrimaryKeyField, "Suntan Oil"
FROM tblFamilyVacation
WHERE [TanningOil] = -1

You can then join this union query to tblFamilyVacation in the report's record source. The detail section would list only the [Check] details. You could also use the union query as the record source for subreport that uses the PrimaryKeyField as the LINK MASTER/CHILD.

john w. vinson say:

Just to agree with Duane and Ken: your design might work ok now, although you'll need some snarky and hard to maintain code to get the report you want. But sometime soon the insurance company will come out with a new Endorsement, delete two others, and change the wording on a fourth... and then what do you have to do? Change the design of your Table, build a new Form, rewrite the code for your Report? OUCH!

Much better is to recognize that you have a classic many to many relationship: each Quote may have zero, one, or many Endorsements; each Endorsement may apply to zero, one, or many Quotes. The proper structure has three tables: Quotes (or Policies, or these might be two tables); Endorsements (e.g. a record like EndorsementID = 123, Endorsement = "Home Business Customer Liability"); and EndorsementsUsed, with a link to the Quote and to the Endorsement. Rather than one yes/no field for each endorsement you would have one record in the EndorsementsUsed table. It would then be very easy to extract the endorsement text into a report, or even into a paragraph of comma-separated text if you prefer; and very easy to add, delete or rephrase endorsements.
