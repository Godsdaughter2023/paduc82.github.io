jp ronse say:

Hi,

VBA does not require variable declaration, it will automatically assign variables as type variant when needed. But this approach has certain risks. Say, I have following statement:

JP_Ronse = 1

JPRonse=JP_Ronse+1

Now, VBA will know 2 variables namely JP_Ronse (1) & JPRonse (2) where you may have expected JP_Ronse = 2.

It is good practice to use Option Explict in each module, you can set this in the VBA options.

This option requires explicit declaration of each variable.

Option Explicit

Dim JP_Ronse

JP_Ronse = 1

JPRonse=JP_Ronse+1

The debugger will now pop-up a warning that JPRonse is not declared.

It is also a good approach to use early binding. Early binding means that you also define the type of the variable at the declaration level. 

Dim JP_Ronse as Integer

JP_Ronse="s"

Will pop-up a type mismatch warning because "s" is not an integer.

If I had wrote:

Dim JP_Ronse

JP_Ronse="s"

It would have been accepted because it was declared as a Variant. But it can cause issues in your further code when you expect to have an integer value and face a string.

It is also good practice to use the Hungarian notation. (Invented by Charles Simonyi).

The basics of this are:

<level><type><Name>

E.g.

gintJpRonse: a global integer known in all modules

mintJpRonse: a module level integer

intJpRonse: a procedure level integer

While debugging, you know immediately at which level it was declared.

There are no rules about the <type>, you are free to use what you want. I am using a 3 letter type:

- bln Boolean

- sng Single

- byt Byte

Early binding can also significantly speed-up the execution of the code because the memory to read depends on the type. An integer is f.i. only 2 bytes.

About late binding.

You will often see Dim x as Object, that means that x can be everything, a worksheet, workbook, another application...

Nothing wrong with it but it implies that VBA at the declartion level doesn't know what type of object you have in mind. You have to define it in the statement CreateObject.

But even then after defining it is the Outlook object, VBA will not assist you in the methods and properties. 

One work-around is to include the Outlook reference to your workbook.

Dim objOL as Outlook.Application

But there is a viper under the grass. Each application verion has his own internal number.

And obvious, you code will not find the Outlook.Application anymore after an upgrade.

Therefore I always start with including the application and when finished I declare it back as an object;

HTH

shirley wang msft say:
Hi Sammurphy11,

OneNote 2016 for Mac lets you recover single pages or entire sections of notes that you have recently deleted. While viewing recently deleted notes, you can restore their pages or sections by moving them back to their original notebook or to another location you want.

OneNote keeps deleted notes for a period of 60 days, after which they are permanently erased. 

To view recently deleted notes, in any notebook, do the following:

1. On the View tab, click the small arrow next to the Deleted Notes button, and then click View Deleted Notes.
2. In the Deleted Notes view, click the tab of the deleted page or section that you want to look at.
3. When done, exit this view by clicking the curved arrow that appears just to the right of the Deleted Notes section label.

Restore recently deleted notes 

While viewing any page or section in the Deleted Notes view, do the following:
1. Control-click the tab of the page or section that you want to restore.
2. On the menu that appears, click Restore To.

andreas killer say:
I guess the issue is somewhere in the outside world, e.g. the given range objects contains non numerical values (errors, strings, etc.) or merged cells or something like that.

BTW, your function is very slow, did not cover any error situations and using Application.Sum is superfluous.

To find the closest value to a given set of values use a formula like this:
  {=INDEX(A1:A1000,MATCH(SMALL(ABS(A1:A1000-$B$1),1),ABS(A1:A1000-$B$1),0))}
Where the list of values is in A1:A1000 and the value to search for is in B1.

I assume the scenario that column A contains the numbers from 1 to 1000 and B1 the value 512.5 The formula needs 0.00132 seconds to calculate.

Your UDF =udf_Min_Abs_Diff(A1:A1000,B1) needs 0.04846, that is a significant difference.

A more safe but similar UDF looks like this:

Function ClosestR(ByVal Where As Range, ByVal What) As Variant
  'Returns the value from Where that is closest to What
  Const MaxDouble = 1.79769313486231E+308
  Dim Diff As Double, Min As Double
  Dim R As Range
  'Be sure What is a number
  If Not IsNumeric(What) Then
    ClosestR = CVErr(xlErrNum)
    Exit Function
  End If
  'Convert a range object to double
  What = CDbl(What)
  'Assume failure
  ClosestR = CVErr(xlErrNA)
  'Compare each
  Min = MaxDouble
  For Each R In Where
    If IsNumeric(R.Value) Then
      'Calculate the difference
      Diff = Abs(R.Value - What)
      If Diff <= Min Then
        ClosestR = R.Value
        Min = Diff
      End If
    End If
  Next
End Function

That UDF needs 0.01477 seconds, 4 times faster then yours, but 10 times slower as the formula.

We can speed up the code if we use an array instead of the range object:

Function ClosestA(ByVal Where As Range, ByVal What) As Variant
  'Returns the value from Where that is closest to What
  Const MaxDouble = 1.79769313486231E+308
  Dim Diff As Double, Min As Double
  Dim Data
  Dim i As Long, j As Long
  'Be sure What is a number
  If Not IsNumeric(What) Then
    ClosestA = CVErr(xlErrNum)
    Exit Function
  End If
  'Convert a range object to double
  What = CDbl(What)
  'Read in all values
  Data = Where.Value
  'Only one value?
  If Not IsArray(Data) Then
    ClosestA = Data
    Exit Function
  End If
  'Assume failure
  ClosestA = CVErr(xlErrNA)
  'Compare each
  Min = MaxDouble
  For j = LBound(Data, 2) To UBound(Data, 2)
    For i = LBound(Data) To UBound(Data)
      'Be sure we compare numbers only
      If IsNumeric(Data(i, j)) Then
        'Calculate the difference
        Diff = Abs(Data(i, j) - What)
        If Diff <= Min Then
          ClosestA = Data(i, j)
          Min = Diff
        End If
      End If
    Next
  Next
End Function

That UDF needs 0.00148 seconds, nearly the same speed as the formula. Keep this results in mind before you develop other UDF's.

BTW, this a good example that a larger code can execute faster then a shorter one, not what the most people would expect.

If you want to know how to get this timings and how Excel calculate things, read this article:
http://msdn.microsoft.com/en-us/library/aa730921.aspx
