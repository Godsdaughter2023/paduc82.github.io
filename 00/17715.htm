Tom's very valid point is that NULL values will trip you up here. Try just using a comparison

 

WHERE NZ([cage_code], "") <> NZ([cagecode], "")

 

and a similar expression in your IIF.

oliver wu msft say:

Hi Steve,

 

Based on your description, I know that the unread status not reflect correct after you search again. For this scenario, I’d like to share with you that the outlook fast search is based on the online search now, this means the status you changed will be synced to the email service side and then you search again, the outlook will reflect the status from the server side to the local cache side. And this process need some time to be completed, so if you mark the email unread and then mark it as read again immediately, the status of this email is still show unread on the email server side, so when you research it again, the status still show as unread.

 

Moreover, as you said when you turn off the cache mode, the research result is correct, that because your outlook is based on non-cached mode ( In general, we called it as online mode) so the status change process was reduced ( from server side to the local cache side), and it can show the status correct immediately.  And after you turn on the cache mode again, the outlook is based on cache mode, the status changes in the research result will need more time to reflect from the server to local cache ( Based on how many items you had in your mailbox, and outlook client need more time to update all folders in this scenario).
