a. user say:

Hi John,

Thank you for posting your query in Microsoft Community.

The 400 Bad Request error is an HTTP [Hyper Text Transfer Protocol] status code that means that the request you sent to the website server (i.e. a request to load a web page) was somehow malformed therefore the server was unable to understand or process the request.

When troubleshooting an HTTP 400 condition, it is important to remember that the underlying problem is that the client has sent a request to IIS that breaks one or more rules that HTTP.sys is enforcing. You will want to see exactly what the client is sending to IIS; to do this, capture a network trace of the client sending the bad request. You can analyze the trace to see the raw data that the client sends to IIS, and to see the raw response data that IIS sends back to the client.  You can also use an HTTP sniffer tool called Fiddler; this is a great tool as it allows you to see the HTTP headers even if the client and server are communicating over SSL.

The next data item you will want to use is the httperr.log file.  Beginning in IIS 6.0, the HTTP.sys component handles incoming HTTP requests before they are passed along to IIS, and is the component responsible for blocking requests that don’t meet the IIS requirements.  When HTTP.sys blocks the request, it will log information to its httperr.log file concerning the bad request.

NOTE:  For more information on the HTTP API error logging that HTTP.sys provides, see the following article:

Get help with website error messages (HTTP errors)

Hope this information is helpful. Do let us know if you need any further assistance, we will be glad to assist you.

Excel QUOTIENT function - Inaccurate results - Microsoft Community

raga_y say:

Excel QUOTIENT function - Inaccurate results

Hi,

The following formula is giving the result as 2 instead of expected result of 3.

=QUOTIENT(0.3,0.1)

Even when I use ROUND function in the formula =QUOTIENT(ROUND(0.3,1),ROUND(0.1,1))
 it is still giving the inaccurate result.  I'm using Excel 2010 (MS Office 2010 Standard) on Widows XP pc.   What could be the reason for this error?  How could this be fixed?  Can someone please advise me?

joeu2004 say:

Raga_y wrote:

The following formula is giving the result as 2 instead of expected result of 3.

=QUOTIENT(0.3,0.1)

[....]

What could be the reason for this error?  How could this be fixed?

The root cause is that most non-integers cannot be represented exactly in the standard 64-bit binary floating-point form that Excel uses to represent numbers and perform arithmetic.  In your example:

0.3 = 0.299999999999999,988897769753748434595763683319091796875
0.1 = 0.100000000000000,0055511151231257827021181583404541015625

[EDIT] I use period as the decimal point and comma to demarcate the first 15 significant digts.

As you can see, by coincidence, the binary representation of 0.3 is less than 0.3, and the binary representation of 0.1 is more than 0.1.

That explains why their (truncated) integer quotient is 2 instead of 3.

You might also notice that MOD(0.3,0.1) appears to be 0.1, not zero.

Actually, =MOD(0.3,0.1)-0.1-0 is about -2.78E-17, slightly less than 0.1, as we might expect given their true binary representations.

(We need to add or subtract zero "redundantly" in order to avoid an Excel feature, whereby if the result of the last subtraction is "close enough" to zero, Excel arbitrarily replaces the true arithmetic result with exactly zero.)

By the way, explicitly rounding constants to their original degree of precision -- like ROUND(0.3,1) -- does not change their binary representation.

If you require exact results with non-integers, it is best to convert them integers.  Multiply each integer by 10 to the power of the required precision (number of decimal places).  For example:

=INT(ROUND(A1*10,0)/ROUND(A2*10,0))

or

=QUOTIENT(ROUND(A1*10,0), ROUND(A2*10,0))

where A1 is 0.3 and A2 is 0.1.

Report Query with True/False - Microsoft Community

cpochard say:

Report Query with True/False

  I am still learning the ins and outs of Access. I have a table that lists different personnel, cities with dates with a True/False field that indicates when the last time the city was visited. I tried DLast() function but it only returns the last date entered. I my query, I get all the times the city was visited. I am looking for accessing the personnel list and when they last went to each city. But only the last time the city was visited. Is this possible?

hansv mvp say:

Let's say the table is named tblData, and the relevant fields are PersonnelID, CityName and VisitDate.

The SQL for your query would be


SELECT * FROM tblData
WHERE VisitDate =
    (SELECT Max(T.VisitDate)
     FROM tblData AS T
     WHERE T.PersonnelID = tblData.PersonnelID AND T.CityName = tblData.CityName)

ken sheridan say: A subquery as Hans describes is the most efficient method, but for future reference note that the DFirst and DLast functions do not do what you might expect from their names.  In essence they return an arbitrary value, but are occasionally useful if you want to ensure that values are returned from several columns, all in the same row.

To get the first and last values in the sort order of a set of values use the DMin and DMax functions.  As your query is used for a report it does not need to be updatable, so a subquery is the best solution, but if you wanted to do the same for a form in which you need to be able to edit data then an updatable query can be used as the form's RecordSource by using the DMax function:

SELECT * FROM tblData
WHERE Visitdate =
    DMax("VisitDate", "tblData", "PersonnelID = " & [PersonnelID]
    & " And CityName = """ & [CityName] & """");
