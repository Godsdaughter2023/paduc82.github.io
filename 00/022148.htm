tom van stiphout say:

In most cases you can simply replace the FE with a new version. No need for Pacakge Solution. I say in most cases, because if you start using an ActiveX control, you will want to deploy it to your users (provided licensing agreements allow it).

Change the structure of the BE on the fly: I am referring to CreateField, CreateIndex etc. methods (see help file). You could write code like (in pseudo-code)

if VersionOfBE = 1 then

  CreateIndex(...)

scottgem say:
This is one of the reasons behind splitting a database. This allows the developer to continue working on the app and deploying updates without affecting the user's ability to enter data.

If your environment exactly mirrors the user environment, then deploying is as easy as copying the local front end onto their PC. The links would be the same. However, if it is not a mirror, then you will need to include code that relinks to their back end. 

As for changing the back end, there are many ways to do so, Tom some, you could also write SQL code (like Alter Table) to do it. 

I use Custom Database properties to track the versions. I also put all my development work in a development folder. I check the path of the front end when I load and if its in a dev or test folder, I relink to a copy of the back end in the same folder. Otherwise I link to the production back end.

marshall barton say:

Deploying a new version of your front end db can be as simple as copying the file to each user's system or as complex as installing a new version of Windows or Office.  It all depends on what libraries you use in your front end's References list and what is preinstalled on your user's system.  If you only use the basic libraries (Access, VBA, DAO) and you have some adminustrative procedures to make sure your users akways have the exact same version of the libraries as you have (very unlikely), then you can just copy the FE file.  If users might have different versions of the libraries, then you should distribute a decompiled copy of a new FE and compile it on each user's system.

As other's mentioned, if you use other libraries (ActiveX, Office, etc) then you can run into conflicts when users install any other software that include it's own copy of a library you use.  The problem here is because you no longer control which library version is on each user's system.  At this level of installation complication, you need to minimize the number of libraries you have in References and use VBA code for late binding as much as possible.

Distributing a new version of your back end db is a question for another day.

I haven't seen a similar complaint nor can I reproduce it, so the first suggestion i would offer is to confirm that all software is fully updated, including the printer driver. Check the mfr's web site to be sure - This is particularly suspect since the PDF renders OK.( Have you tried locally to see if you can reproduce the problem on your systems?)
