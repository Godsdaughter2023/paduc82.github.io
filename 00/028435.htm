It's not apparent to me how a wildcard Find (or any other Find, for that matter) could determine the number of words in what you call a chain. Moreover, one of your examples (luon lone) even lacks the /. It's certainly possible to identify a single word followed by /, however.

eric aloha say:

Hi Paul,

Thanks for your reply. The omission of the slash between luon and lone was just a mistake. But I know there's a way to do this, because I actually got someone's help 2 years ago to do it. It involves identifying the characters that precede the words before the slashes. In most cases, this is a comma followed by a space. But it can also sometimes be an ending parentheses followed by a space.

There is some way, using wildcards, to tell the program to underline everything that occurs after those initial characters and before the slash. Does that help?

macropod say:

A wildcard Find/Replace can't do that without also underlining the comma or ending parenthesis, plus the /. You'd need to either:

• perform an initial wildcard Find/Replace underlines everything starting at the comma or ending parenthesis, and ending at the /, followed by another wildcard Find/Replace to remove the underlines from commas, ending parentheses, plus the spaces following them, and yet another to remove the underlines from the /; or

• use a macro that contracts the Found ranges to avoid underlining the commas or ending parentheses, and the /.

For the first approach, you'd use a wildcard Find/Replace with:

Find = [,\)][!^13^l^t.]@/

Replace = ^&

and set the replacement font to underlined, followed by:

Find = [,\)]^32

Replace = ^&

and set the replacement font to not underlined, followed by:

Find = /

Replace = ^&

You could, of course, record these actions as a macro.

I just discovered that there's one more situation in which I need to apply underlining to the words preceding a slash. It's in situations where the head word (which is in all caps and is 12 point instead of 10 point) has no parentheses or commas after it.

ACHIEVE  riich/reach, toch/touch 

macropod say:

It's at this point the complexity is such that it really does warrant a macro:

Sub Demo()
Application.ScreenUpdating = False
With ActiveDocument.Range
  With .Find
    .ClearFormatting
    .Replacement.ClearFormatting
    .Text = "[A-Z,\)] [!\),^13^l^t.]@/"
    .Replacement.Text = ""
    .Forward = True
    .Format = False
    .Wrap = wdFindStop
    .MatchWildcards = True
    .Execute
  End With
  Do While .Find.Found
    With .Duplicate
      If .Words.First.Text = UCase(.Words.First.Text) Then
        .Start = .Words.First.End
        .End = .End - 1
        .Font.Underline = wdUnderlineSingle
      End If
    End With
    .Collapse wdCollapseEnd
    .Find.Execute
  Loop
End With
Application.ScreenUpdating = True
End Sub

For Mac macro installation & usage instructions, see: https://wordmvp.com/Mac/InstallMacro.html

scottgem say:

 The secondary table needs to mirror certain fields of the main table but also have the capability to have information of it's own that isn't on the main table. Can this be done?

That is not how relational databases work. One of the principles of a relational database is that data exists in one place only. So you do not want any fields that are in the primary table in the secondary table except for a Foreign Key that links the secondary record back to the primary record. If you would give us details about the table structure of both tables we can be more specific. 

The general way you work this is by using a main form bound to the primary table and a subform bound to the secondary table and linked on the key fields. This way you see all the data from the primary record and the additional data that goes in the secondary table.

ronrosenfeld say: In that case, one has to test every character in every cell in the range.  That will take time, but is doable.

==================================
Option Explicit
Sub FindRedChars()
    Dim rSrc As Range
    Dim rResults As Range
    Dim c As Range
    Dim i As Long
Set rSrc = Range("B2", "BDK3354")
Set rResults = Range("BDM2", "BDM335")
rResults.ClearContents

For Each c In rSrc
    For i = 1 To Len(c.Text)
        If c.Characters(i, 1).Font.Color = vbRed Then _
            Cells(c.Row, rResults.Column) = _
            Cells(c.Row, "A") & "." & Cells(1, c.Column)
    Next i
Next c
End Sub

Can't speak to the C# side of things, but using C++ and the xll library

 

1. See the HELLO.XLL2 example below.

2. In the xll world, a sub cannot take any arguments. It is called purely for side effects.

3. and 4. Have a look at the documenation on http://xll.codeplex.com. Let me know how you think it can be improved.

 

Below is how you would write your functions in C++ using the xll library. You can find the add-in, help file, and spreadsheet in HelloXll.zip at http://sdrv.ms/JtaMIV. If you prefer to build it yourself, create a new XLL AddIn Project, replace the text of function.cpp, and hit F5. It should pop you into Excel with the add-in loaded.

 

Once you run the macro XLL.DOC, you can access the help file from the function wizard via 'Help on this function.' This comes in handy if you are creating add-ins for someone else.

The real win with using a C++ solution is that you do not have to marshal data back and forth between the managed and unmanaged world. This can make a big difference if you are using large arrays of doubles. There is no counterpart to the FP datatype in VBA or ExcelDNA.

 

// function.cpp - Rename this file and replace this description.
#include "xll/xll.h"

using namespace xll;

 

// short form
static AddIn xai_hello_xll(
 "?xll_hello_xll", XLL_LPOPER XLL_CSTRING,
 "HELLO.XLL", "Name"
);
LPOPER WINAPI
xll_hello_xll(const char* name)
{
#pragma XLLEXPORT
 static OPER o;
 
 o = Excel<XLOPER>(xlfConcatenate, OPER("Hello "), OPER(name));

 

 return &o;
}

 

// long form
static AddIn xai_hello_xll2(
 Function(XLL_LPOPER, "?xll_hello_xll", "HELLO.XLL2")
 .Arg(XLL_CSTRING, "Name", "is a string. ")
 .FunctionHelp("Append Name to the string \"Hello \".")
 .Category("My Category")
 .Documentation("Additional remarks go here.")
);

// reuse xll_hello_xll

 

#include "Mmsystem.h"
#pragma comment(lib, "winmm")

 

static AddIn xai_speed_timer(
 "?xll_speed_timer", XLL_LONG XLL_LONG,
 "SpeedTimer", "Exponent",
 "My Category", "Increment an integer by 10^Exponent and return time in miliseconds"
);
LONG WINAPI
xll_speed_timer(LONG e)
{
#pragma XLLEXPORT
 LONG e10 = (LONG)pow(10.,e);
 LONG n = 0;

 

 LONG t = timeGetTime();

 for (LONG i = 0; i < e10; ++i)
  n = n + 1;

 t = timeGetTime() - t;

 

 return t;
}

Prevent losing email in Outlook.com when you've set up POP3

Outlook.com has a feature that prevents losing email messages if you accidentally set up POP3 to delete them after reading them on the server. Here’s how it works:

    The first time that Outlook.com receives a POP3 instruction to delete email messages after reading from an email app, Outlook.com won't delete the messages. Instead, Outlook.com moves the email messages that were successfully downloaded by the email app into a POP folder and out of the inbox. The POP folder is available in the Folders list.

    Outlook.com keeps the email messages in the POP folder until you sign in to your Outlook.com account and verify whether Outlook.com should respect or ignore the POP3 delete command.

To verify your POP3 settings:

    In the upper-right of the Outlook.com window, click Options, and then click More options.
    Under Managing Your Account, click Pop and deleting downloaded messages, and then select one of the following:
   
Don’t let another program delete messages from Outlook.com. (If your other program is set to delete messages from the server, we’ll simply move them to a special POP folder. They won’t be deleted.)

Do what my other program says—if it says to delete messages, then delete them.

    Click Save.
