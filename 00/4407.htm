Why do LookUp fields default to 'Number' - Microsoft Community

seansykes2019 say:

Why do LookUp fields default to 'Number'

ken sheridan say: As you'll have seen from the link which Duane gave you the lookup field wizard should be avoided like the plague for a number of reasons.

When the wizard creates the lookup table it gives it an autonumber 'surrogate' primary key.  There are good reasons for this as we'll see below.  Consequently the foreign key column in the referencing table must be a long integer number data type.  In the table definition the wizard sets column's DisplayControl property to Combo Box and sets the combo box's RowSource property to something like this:

SELECT CityID, City
FROM Cities
ORDER BY City;

The first column is hidden by setting its column width to zero, so the user sees the name of the city in the column.  This is potentially confusing as an inexperienced user is likely to think that is the value in the column.  If they then create a query like this:

SELECT *
FROM Customers
WHERE City = "Newcastle";

the query will not work because the value in the City column for those customers in Newcastle will be a number.

To model customer's addresses (I'm assuming the database contains only UK customers)  would require the following tables:

Customers
….CustomerID  (PK)
….Customer
….AddressLine1
….AddressLine2
….CityID  (FK)
….PostCode

Cities
….CityID  (PK)
….City
….CountyID  (FK)

Counties
….CountyID  (PK)
….County

IN the USA the model would be extended with a States table.  Most developers would create these tables themselves rather than using the wizard, and enforce referential integrity in the relationships.

While that is the model which would usually be used, it is possible to build another model using 'natural' keys for the address data:

Customers
….CustomerID  (PK)
….Customer
….AddressLine1
….AddressLine2
….City  (FK)
….County  (FK)
….PostCode

Cities
….City  (PK)
….County  (PK)

Counties
….County  (PK)

You'll notice that Cities appears to have two primary keys!  By definition a table can only have one primary key.  In this case the two columns constitute a single composite primary key.  Correspondingly, what appear to be two foreign keys in Customers constitute a single composite foreign key referencing the composite primary key of Cities.  Again all relationships should be enforced.

With the use of 'natural' keys the query which would not work with the first model:

SELECT *
FROM Customers
WHERE City = "Newcastle";

will now work, but there is still one problem; there is more than one Newcastle in the UK, so in addition to the customers we really want, the query would return the customers in other Newcastles as well.  We can solve this easily by bringing the County column into play:

SELECT *
FROM Customers
WHERE City = "Newcastle"
AND County = "Staffordshire";

So while natural keys would not normally be used, there is nothing in the database relational model to say they should not be, and, as we’ve seen with the above query, they can have advantages.  To do the same with a model using numeric 'surrogate' keys the query would have to join all three tables.

Where 'natural' keys have a big advantage over 'surrogate' keys is where corelated (aka 'cascading') combo boxes are used, e.g. to select a county in one combo box, and then a city from a second combo box whose list is restricted to cities in the selected county.  You'll find an example of this in ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

This little demo illustrates the use of correlated combo boxes using both 'surrogate' and 'natural' keys.  It shows how much easier it is to build them using the latter, especially in forms in continuous forms view, where we have to use 'hybrid' controls made up of a text box superimposed on a combo box to give the appearance of a single combo box control.  This works, but gives rise to some behaviour which can be slightly confusing to users at first.

When it comes to data entry or editing, this should never be done in a table's raw datasheet view, but always in forms.  Consequently there is no point in using a combo box as the DisplayControl property of a column in a table.  A combo box cam be used in the form in exactly the same way.

So, to summarise:

1.  Never interface with data in a table's datasheet, but always in forms.  The only possible exception to this rule is when debugging.

2.  Never use the 'lookup field' wizard.  Always create the referenced ('lookup') tables first, and then the referencing tables.  You can then create foreign keys of the appropriate data type in the referencing table.

3.  Think carefully about the use of 'surrogate' or 'natural' keys.  While the former will be more commonly used, do not hesitate to use the latter where this will be advantageous to the design and efficient operation of the database.

For an example of the use of 'natural' keys in a more complex model see Relationships.zip in my same OneDrive folder.  This demo includes a high degree of correlation between combo boxes and uses 'natural' keys in all cases apart from the Patients table.  Personal names cannot be used as keys because they can legitimately be duplicated.  Nor is a combination of names and other columns as a composite key reliable.  I was once present at a clinic when two patients arrived within minutes of each other, both female, both with the same first and last names and both with the same date of birth.  People should always be identified by a 'surrogate' numeric primary key, e.g. PatientID, usually an autonumber for convenience.

While YOU may think that the envelope is being fed into the printer correctly, it may NOT be being fed into the printer in the position that Word thinks it is.
