Make word 2007 Default Program - Microsoft Community

sue goodwin say:

Make word 2007 Default Program

rohn007 say:
Try this:

If there is no error message, ie normal operations try adding the NoReReg key, ie manual method:

Before you close up shop and start working on this reg fix, open the version of Word that you will want to be your default. In other words, say you have two versions of Word on your system. You want to use 2007, but you want 2003 to be the version that opens by default when you click on a DOC file. Open Word 2003 before you insert the fix below. This will ensure that Word 2003 is the last version opened and it will, if necessary, reset the registry to that version. (More on getting around this default in the next section.)

  • Now make sure Word (office apps) is closed
  • Start regedit  <Win  R> and type REGEDIT
    Note that if this doesn't do anything, you're using a system where the administrator has taken away your ability to edit the registry so you don't mess it up!
  • Once within the registry, click the + (plus sign) to expand the /HKEY_CURRENT_USER/ folder.
  • Scroll down and locate the /Software/ folder and expand that one. We have a long journey ahead, so hang in there!
  • Once within the Software folder, scroll down that branch of the folder tree until you find the folder for /Microsoft/. Click the + to expand that one.
  • Now scroll down to locate the /Office/ folder.
  • There you'll see the numbered versions of the Office versions you have:
    2010 is 14.0
    2007 is 12.0
    2003 is 11.0
    2002 is 10.0e.
  • This fix can be used for all recent versions. Or at least, I know it works as far back as 2000. I can't confirm anything earlier. But if you're still using a version older than 2000, speed is obviously not an issue for you!<grin> Expand the folder for the version you want to speed up. (Note, this speed up fix will only help you if you are running multiple versions. So if you only have one version of Office on your system. Don't bother with this setting, as it won't do anything for you.)
  • Move to the /Word/ folder and expand that. Almost there!
  • Okay, locate the /Options/ folder and select that one. This is the folder where you'll be adding a new registry key.
  • With the Options folder selected in the route we took above, click Edit > New > DWORD Value
  • This will insert a new registry key into the Options folder. It will be highlighted in edit mode so you can go right ahead and start typing the name of this key: NoReReg

    Hit the Enter key after typing the name to have the name accepted.
  • Now double click this new key to open it. The Value data will read 0 (zero) by default. Change that setting to the number 1
  • click OK to accept that value.

Then close the registry. You did it!

So what is it that you've done? Well, you went into the Word registry settings for the version of Word you selected and you set a new registry key. The key is an option that stops Word from reregistering itself if another version is the current default. The value of 1 means Yes. So the NoRereg (no reregistering) is set to Yes...i.e., don't do it! This will cause Word to ignore the default version and just open, without taking several minutes to go through the reregistering hassle.

Shortcut method:

  • <win R> to start run dialog
  • Enter (copy) this command :

reg add HKCU\Software\Microsoft\Office\14.0\Word\Options /v NoReReg /t REG_DWORD /d 1

repeat for each office app and each version installed

Regedit commands that can be run from command line  for Office 2010 (Change 14.0 to 11.0 for 2003 or 12.0 for 2007):

reg add HKCU\Software\Microsoft\Office\14.0\Word\Options /v NoReReg /t REG_DWORD /d 1

reg add HKCU\Software\Microsoft\Office\14.0\Excel\Options /v NoReReg /t REG_DWORD /d 1

reg add HKCU\Software\Microsoft\Office\14.0\Access\Options /v NoReReg /t REG_DWORD /d 1

You must accept the Office End User License Agreement every time that you start an Office program - http://support.microsoft.com/kb/884202/en-us

Reset permissions on the registry container / folder:  HKLM/Software/Classes

Set owner to %computername%\Administrators and set to "Replace owner on subcontainers and objects"

Also make sure SYSTEM and Administrators are added to Security Tab / Advanced with Full Control and enable check-box "Replace all child object permissions with inheritable permissions from this object".

Then, uninstall Office 2010, (make sure all older office installations are removed), reboot TWICE (sorry, I don’t know why, but there appears to be some “Voodoo” at play here. We do it because it works!), reinstall Office 2010.

In A436 put in this formula,

=DATE(2013, ROW(1:1),1)

Format the cell as mmmm and fill down to A447. This will supply real dates that can be referenced within COUNTIFS() while displaying only the month name.

In B436 you can now use variations on the following,

=COUNTIFS($E$2:$E$432,">="&A436,$E$2:$E$432,"<"&EDATE(A436,1))

The above counts the dates in column E that are within Jan 1, 2013 - Jan 31, 2013.

Edit: removed some confusion with 2013 and 2014.

DATE function Returns the serial number of a particular date
ROW function Returns the row number of a reference
COUNTIFS function Counts the number of cells within a range that meet multiple criteria
EDATE function Returns the serial number of the date that is the indicated number of months before or after the start date

derek nishino say:

Misery loves company :) .  But the good news is that I think I found a fix.  Try using the -SessionCommand switch with a value of ReturnLargeSet:

$results = Search-UnifiedAuditLog -StartDate $startDate -EndDate $endDate -SessionCommand ReturnLargeSet -ResultSize 5000

If you then run $results.Count, you should get a number that you're expecting.  For me, it returns 5,000 as the Count.  When I run the search on file access events I'm now getting the result set that I expect.  The documentation says that using that switch causes results to be returned non-sorted.  I haven't confirmed that yet, but it is something I can script around if it is indeed the case.

I don't know if the documentation is just poor, or there is a bug if you don't use the -SessionCommand ReturnLargeSet switch.

=DLookup("MarketName","Markets","MarketCode = " & [Code])

Seems I should be able to add a column to the DetailedProposals table and use a dlookup to put the MarketName associated the Code field for each proposal in the DetailedProposals table. Right? 


Problem is that you are using the wrong approach. You are close, you have the door open but aren't stepping over the threshold. You did right by creating your Markets table. Such a table is generally called a lookup table. You store the MarketCode in your main table to link to the Market name in lookup table. This is called a Foreign Key.

Where you go wrong is wanting to ALSO store the Market Name in your table. This violates normalization rules. One of the main principles of a relational database is that data should exist in ONE place only. So leave it in your Markets table. You can DISPLAY the name anytime you want using a variety of techniques. I discuss many of those techniques in my blog article on Displaying data from related tables on a form.

you could also us a combo box with "code" as the control source and the row-source is the MarketCode and MarketName

I think the report wizard does that if the dreaded 'lookup field' wizard has been used in the table definition, but I don't favour using a combo box in a report myself.  It's really more a control for use in a form, not a report.  Joining the tables in the query is the best solution.

The only circumstance I can think of where a combo box in a report would help is if the Filter property of a form which uses a combo box and has been filtered via the 'filter by form' facility is passed to a report.  The string expression which this builds uses a lookup mechanism to get the value of the hidden key from the value in the combo box's visible column, so if a similar combo box is used in a report the form's Filter property can be applied to the report.  I've known cases where this has been done, but I'd always incorporate a custom interface for filtering forms and reports into any application I develop rather than allowing the built in 'filter by form' facility, which I believe is not available in Access's runtime environment in any case; that used to be the case at any rate, though off the top of my head I can't swear to it being so in more recent versions.
