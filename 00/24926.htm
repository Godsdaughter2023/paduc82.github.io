scottgem say:

First, your design is off. You have a many to many relationships here. One employee can have access to multiple applications and one application can be used by multiple employees. So you should have three tables here:

tblEmployees: EmployeeID, names, other info about employees

tblApplications: ApplicationID, name other info about apps

tjxEmpApp: EmpAddID (PK Autonumber), EmployeeID (FK), ApplicationID (FK)

The last is referred to as a junction table which models the many to many relationship.

Having a table for each application is not the right design. Neither is using Lookup fields on the table level.

I would have a mainform bound to tbleApplications with a subform bound to tjxEmpApp. On the subform (in Continuous Form mode) you would have a single visible control, a combobox to select an employee.

To create your query you would add all three tables, joining tjxEmpApp to tbleEmployees on EmployeeID and tblApplications on ApplicationID. Right click on the join line between tjxEmpApp and tbleEmployees and select Join properties. Select the option for ALL from tbleEmployees and matching from tjxEmpApp.

Then add the Employee name and Application name columns. This query will then list all employees with whatever application they have been assigned. If an employee has no app then the Application name column will be blank.

ken sheridan say:

I guess there are two parts I'm getting hung up on:

1) How can you have one table for "what we might call permissions" when APPLICATION 1 may have 50 different permissions type and APPLICATION 2 have 50 other application types that go by different names as APPLICATION 1's. If you aggreagate every single application's permissions into one table, it becomes confusing for hte everyday usage of the database.

2) There may be an easy solution for this, but a goal would be to have the ability add/remove/edit specific entries WITHOUT HAVING TO DODGE MY WAY AROUND A BUNCH OF OTHER ENTRIES FOR OTHER APPLICATIONS. And 'Im sure maybe queries and tables and forms can resolve this.

As regards 1 this is achieved by modelling the entity types correctly as tables.  Each table models one real-world entity type, so you have tables as follows:

1.  Employees: this has one row for each employee.
2.  Applications:  this has one row for each type of application.
3.  Permissions:  this has one row for each type of permission.  Each permission can apply to as few or as many applications as necessary, and as few or as many employees as necessary.  Which they apply to is irrelevant as far as this table is concerned; it simply models the total set of permissions.

4.  You then have a special kind of entity type, the relationship type between the above three 'referenced' entity types.  This table has three foreign keys, each of which references the primary key of one of the above tables.  Consequently any combination of employee/application/permission can be represented by a row in this table.  Each row must have a distinct combination of the three values of course, so the primary key is a composite one of all three columns.  Even if you add a separate autonumber primary key column to the table, the three columns are still a key and must be defined as such by inclusion in a unique index.

5.  If only some permissions can apply to some applications then there is another relationship type between applications and permissions, which is modelled by another table, this time with only two foreign keys, ApplicationID and PermissionID.  This is related to the table in 4 on these two columns, and the relationship is enforced.

6.  Similarly, If only some applications can apply to some employees then there is another relationship type between employees and applicationss, which is modelled by another table, this time with only two foreign keys, EmployeeID and ApplicationID.  Again, this is related to the table in 4 on these two columns, and the relationship is enforced.

7.  Similarly, If only some permissions can apply to some employees then there is another relationship type between employees and permissions, which is modelled by another table, this time with only two foreign keys, EmployeeID and PermissionID.  Again, this is related to the table in 4 on these two columns, and the relationship is enforced

The above is a pretty standard relational database model, and you'll find many situations which are modelled in a similar way.

As regards 2, with the above model it will not be difficult to design an interface by which data can be entered/edited in a very straightforward manner.  How you design the interface depends on the orientation by which want to see the data.  Let's say you want to orient it by employee, at its simplest you'd have an employees parent form in which would be embedded a subform based on the table described in 4 above.  In this subform would be combo boxes to select an application and a permission.  If the restrictions described in 5, 6 and 7 above apply, the combo boxes could be correlated so that the first lists only those applications applicable to the current employee, and the permissions combo box would then be correlated with the first combo box so that it lists only those permissions applicable to the selected application and to the current employee.

You could of course also orient the data in a different way, e.g. by application, in which case the parent form would be based on Applications and the subform would contain combo boxes to select the employees for the current application and the permissions for the selected employee/application combination.

For those that get "Can't connect to see who's online" when going to the Online pivot in Messaging on Windows Phone, the most likely causes are:

  2. Spotty connectivity. Having poor connectivity would cause the push notification system to be temporarily disconnected. This type of problem should recover automatically when the device regains connectivity. In rare cases where the device doesn't recover automatically and you can see that other services that use data work properly, you can try going "offline" and back "online" again by changing your status.

If you are unsure whether your Windows Live account works with Facebook chat (it needs to in order to work on the phone), you can try the following:

  1. Open a web browser, and navigate to http://live.com.
  2. Login with the same Windows Live ID you use on your phone. If you use multiple Windows Live ID's, sign in with your primary one.
  3. At the top of Windows Live, hover your mouse over "Messenger" and check what the status says. If it works here, it should also work on your phone. If it doesn't work here, and you see something like "Reconnecting..." it may indicate an issue with your Windows Live account or the service.

If you have issues with using the web messenger directly on Windows Live, then unfortunately this will have a downstream effect of it also not working on your phone. You can post your web messenger issue to the Windows Live forums here or go to http://windowslivehelp.com/. Once your web messenger works with Facebook, so should your Windows Phone. The phone uses Windows Live as a proxy for Facebook chat.

 

Note, also, that there are a small set of account types may not work with Facebook chat perfectly on the phone. You can read more about those accounts here:http://support.microsoft.com/kb/2599552. These account types should not show error "Can't connect", however.
