scott .. b say:

Typing delays are usually caused by insufficient RAM or slow keyboard response.  If your PC is dividing its available memory between several open apps, the keyboard response will be proportionally slower.

 

You can do a number of things to fix this:

 

1) Before typing, close ALL other apps.  Many people have several windows open at the same time and try to transpose between them.  This gobbles up RAM.  Windows considers your Desktop as an app, so close it and any windows you may have open on the Desktop.

 

2) Defragment your hard drive.  More than likely you have fragmented files scattered through the drive.  Windows must open and close files and windows from pieces of files scattered all over the drive. This takes up RAM.

 

At the METRO (Start) screen type "Defrag"

Settings

Select "Defrag and Optimize your Drives"

Analyze

Optimize

OK

 

3) Do a memory Troubleshot to see and fix any irregularities.

 

At the Start screen type" Memory"

Settings

Select "Diagnose your Computers Memory Problems"

 

(also select "Show your computers RAM")

 

4) Check your keyboard response speed.  If it is too slow, (because your RAM is minimal), you may have to increase it.

 

At the METRO (Start) screen, type "Keyboard"

Select "Settings"

Select "Check Keyboard Status"

Speed

Character Repeat (Test and Set Delay)

OK

djsteele, mvp say:

I hate to quibble with my fellow MVPs, but it's recommended that you do a Resume (or Err.Clear) once you've reported the error:

Private Sub Something_Click()
    On Error Goto Err_handler

    DoCmd.OpenReport "r_QuarterlyChecklist", acViewPreview
 
End_handler:
    Exit Sub
Err_handler:
    Msgbox Err.Description
    Resume End_handler
 
End Sub
 
The reason for directing the program back to End_handler is for those cases where you might have cleanup that has to run whether or not the code was successful.
 
You might read Allen Browne's Error Handling in VBA

a. user say:

Hi Dileep... Currently the option to 'force' 3G connections only is not available, and I'm not sure it would be in the future.  The way the cellular networks work, the phones negotiate with the cellular towers to determine the maximum workable speed.  The phone makes an initial request (based on the selection you make in your settings -- say 3G) and then the nearest tower talks back and forth with the phone to select a 'workable' speed, based on traffic congestion, proximity and signal strength, working down from the initial request (i.e. from 3G to 2G to GSM).  The goal is to keep the connection going as long as possible, using the best negotiated speed.  They don't differentiate between voice and data connections at this juncture.

At this point, there's no capability in the system to say, "just for data, if I can't get 3G, then forget it".

Printing track changes - only pages with changes - Microsoft Community

t. lynn simon say:

Printing track changes - only pages with changes

Is there a way to print only pages with track changes?

 

I understand how to use the "print markup" and other print commands, but there does not seem to be an option to print only pages with changes.

 

Thanks in advance!

 

Lynn

jay freedman say:
There isn't any built-in command to do that, but a macro like the following can. Read http://www.gmayor.com/installing_macro.htm if needed.

Sub PrintOnlyMarkupPages()
    Dim rev As Revision
    Dim rg As Range
    Dim lastPg As Long
    Dim pgList As String
    Dim oDoc As Document
   
    Set oDoc = ActiveDocument
   
    For Each rev In oDoc.Revisions
        If Len(pgList) > 0 Then pgList = pgList & ","
        Set rg = rev.Range
        rg.Collapse wdCollapseStart
        If rev.Range.Information(wdActiveEndPageNumber) = _
                rg.Information(wdActiveEndPageNumber) Then
            ' rev is contained on one page, so add that number to the list
            pgList = pgList & rev.Range.Information(wdActiveEndPageNumber)
        Else
            ' rev crosses at least one page boundary, so add page range to list
            pgList = pgList & rg.Information(wdActiveEndPageNumber) & "-" & _
                rev.Range.Information(wdActiveEndPageNumber)
        End If
    Next
   
    oDoc.PrintOut Range:=wdPrintRangeOfPages, Pages:=pgList
End Sub

There are probably a number of situations this macro doesn't handle properly -- if you run into a problem, post back and describe what goes wrong.

Use

Dim rev As Revision
Dim rg As Range
Dim lastPg As Long
Dim pgList As String
Dim arrpages As Variant
Dim PagetoPrint As Long
Dim oDoc As Document
Dim i As Long
Set oDoc = ActiveDocument
For Each rev In oDoc.Revisions
    If Len(pgList) > 0 Then pgList = pgList & ","
        Set rg = rev.Range
        rg.Collapse wdCollapseStart
        If rev.Range.Information(wdActiveEndPageNumber) = _
                 rg.Information(wdActiveEndPageNumber) Then
        ' rev is contained on one page, so add that number to the list
        pgList = pgList & rev.Range.Information(wdActiveEndPageNumber)
    Else
        ' rev crosses at least one page boundary, so add page range to list
        pgList = pgList & rg.Information(wdActiveEndPageNumber) & "-" & _
                 rev.Range.Information(wdActiveEndPageNumber)
    End If
Next
arrpages = Split(pgList, ",")
pgList = ""
PagetoPrint = arrpages(LBound(arrpages))
pgList = PagetoPrint
For i = LBound(arrpages) + 1 To UBound(arrpages)
    If arrpages(i) <> PagetoPrint Then
        PagetoPrint = arrpages(i)
        pgList = pgList & "," & arrpages(i)
    End If
Next i

oDoc.PrintOut Range:=wdPrintRangeOfPages, Pages:=pgList

The additional code will reduce your list that contains multiple instances of the one page to a list that contains just one instance of each page.  You may still get a problem if there are more than 255 separate pages that contain revisions.
