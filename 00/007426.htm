Suppress colon in cross-reference - Microsoft Community

emily scarfe say:

Suppress colon in cross-reference

Is there a way to suppress a colon in an automatic cross-reference? I know the switch to suppress a period but I haven't been able to find anything that will suppress a colon (or other punctutation, such as a hyphen, for that matter).

I have created automatically numbered headings in my document as follows:

"Part A: text text"

"Part B: text text" etc

The colon is included as part of the automatic numbering options (as when I typed it manually, Word was adding an additional space between the colon in my TOC).

When I cross-reference to Part A, the cross-ref shows up as "Part A:". How can I get rid of the colon?

Thanks.

a. user say:

The only way I know how to do this kind of thing is to replicate the heading texts in TC fields, like this:

1. Keep your Heading numbering as "Part A" etc.

2. Set up the actual text of each heading like this, where all the { } are the spcial field braces you can insert using ctrl-F9

{ SET T "the text you want for this heading" }: { T }{ TC "Part { SEQ H \*Alphabetic }: { T }" }

3. Use a TOC field like { TOC \f \h \z }

4. You will probably find that you need to update all the fields in the document a couple of times before the TOC shows the correct texts.

5. Because everything for the ":" onwards is the same for every heading, it's fairly easy to set up an autotext with, say

{ SET T "t" }: { T }{ TC "Part { SEQ H \*Alphabetic }: { T }" }

that you can insert for each heading, or use copy/paste. If there is any possibility that you might change the word "Part", you can do e.g.

{ SET HT "Part" }
at the beginning of your document and

{ SET T "the text you want for this heading" }: { T }{ TC "{ HT } { SEQ H \*Alphabetic }: { T }" }

FWIW, I think that's sufficiently nasty that compromising by allowing a space in the TOC (or [perhaps removing those spaces pre-publication) starts to look quite attractive, particularly if you are creating this for other people to use.

 

Peter Jamieson

There are some potential problems with the above. 

If the fields in ConflictOfInterestCases are not always null but a zero-length string, then the update needs to be

Caseload.[Primary SSN] = IIF([PrimarySSN] is Null Or [PrimarySSN] ="",[Primary SSN],[PrimarySSN]

If NCP Addr Line 1 is changed then NCPR Addr Line 2 should probably be changed to match NCPAddressLIne2 whether or not the field is null.  So you may need a more complex statement.

Caseload.[NCP Addr Line 2] =
    IIF([NCPAddressLine1] is Not Null,[NCP Addr Line 2],NZ([NCPAddressLine2],[NCP Addr Line 2))

You might even need to do something similar for NCP Addr Line 1 and for CP Addr Line 1 and 2.

I'm not sure what you mean.
 
SET is a key word in SQL that can only occur once.
 
If you use the query design view to build the query then you have
to write an expression in the "Update To" line of each column.
e.g. in the "Update To" of the [Primary SSN] column:
 
Caseload.[Primary SSN] = IIf(Caseload.[Primary SSN] Is Not Null,
ConflictOfInterestCases.[PrimarySSN])
 
in the "Update To" of the [NCPBirthDate] column
 
Caseload.[NCP BirthDate] = IIf(Caseload.[NCP BirthDate] Is Not Null,
ConflictOfInterestCases.[NCPBirthDate]
 
and so on for the 20 fields.
 
In SQL view of the query the SET clause would look like this:
 
SET Caseload.[Primary SSN] = IIf(Caseload.[Primary SSN] Is Not Null,
ConflictOfInterestCases.[PrimarySSN]),
Caseload.[NCP BirthDate] = IIf(Caseload.[NCP BirthDate] Is Not Null,
ConflictOfInterestCases.[NCPBirthDate],
etc.
