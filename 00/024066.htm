    daniel g. grau say:
    The Table > Formula thing creates invalid formula fields. You need to use fields that look like this (where the brackets are not brackets but field delimiters):

    • { = Sum (Above) }
    • { = Sum (Above \# "-0.00" } (The expression after # sets the format for the numbers.)

    I could not succeed in makingTable > Formula give me the proper syntax (as above), not even by carefully placing the cursor at the proper places. Thus, you need to correct the expressions by hand.

    Also, bear in mind that any error message will persist even after you've amended the formula, so you need to update the field manually (select it and press F9).

    Caveat: If the format of a number above the field is invalid, the calculation will reach up to that point and Word will not tell you it stopped there. You will then get a partial sum. You should double-check by selecting the cells with numbers and calculating the result. For this, Word has a command called "ToolsCalculate," which is not available in the default menus, but you can add it.

    To add ToolsCalculate as a menu choice:

    • Select View > Toolbars > Customize Toolbars and Menus
    • Make sure to select the Command bar in the top tab
    • A new toolbar (a duplicate of the menus in the main menu bar) will appear
    • In the list to the left, select All Commands
    • In the list to the right, select ToolsCalculate
    • Drag this command to a position in the menu toolbar that just showed

    To use the new command, select a numeric expression (like "2 * 4" or a string of numbers, like "2 3 4") or a series of table cells, then choose ToolsCalculate. The result will appear in the Status Bar at the bottom of your document's window and will also be copied to the Clipboard. Compare this value with what you are obtaining with the formula field.

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to amend the form design accordingly.

    Despite the number of vendors, you should still be able to use a combo box for the Ven_ID column.  You don't have to scroll the list as a combo box has an AutoExpand property by virtue of which when you type in a character the control progressively goes to the nearest match.  To add a new vendor the NotInList event procedure of the control can be used.  You'll find examples of the use of this procedure in various contexts in NotInList.zip in my same OneDrive folder.

    As you have rows in your vendors table which do not reference a supplier, the supplier control in your form will be Null if such a vendor is selected.  To add a supplier after selecting a vendor you could put some code in the supplier controls DblClick event procedure for instance, or in the Click event of an 'Add Supplier' button on the form.  One approach would be for the code to open an unbound form in dialogue mode, in which the new supplier name could be entered into a text box.  Code in the Click event procedure of a 'Confirm' button on the form would update the relevant row in the vendors table.  This would execute an UPDATE statement to update the value of the Supplier column to that entered in the dialogue form.  The control for doing this would probably be a combo box in which you could select an existing supplier from a Suppliers table, or insert a new row into the Suppliers table by means of the combo box's NotInList event procedure.  The UPDATE statement would be restricted by means of a WHERE clause, which would specify the current Ven_ID.

    scottgem say:

    OK, So you need something like this:

    tblSuppliers: SupplierID (PK Autonumber), SupplierName, other info about suppliers

    tblEmployees: EmpID (PK Autonumber), FirstName, Lastname, etc.

    tblOrders: PONumber (PK), PODate, SupplierID, (FK), BookedByID (FK)

    tblOrderTrack: OrderTrackID (PK autonumber), PONumber (FK), CheckedByID, StartTime, FinishTime, SKU, Lines, Errors

    Each PO is a separate entity and occurs once. The order has an original order date and a clerk who booked and a Supplier who it was booked with. These are all attributes of the order.

    However, because an order may be fulfilled over several shipments, each shipment becomes a entity. And the attributes of a shipment are the PO it is associated with, the person who checked it in, when it was checked in and what was checked in.

    The foundation of an efficient database is the proper table structure. The structure I outlined above is much more efficient then what you had.

    As to the Suppliers, as I said earlier, the data entry person selects the Supplier name from a combobox. If they are not listed, then the NotInList event is used to add it to tblSupplier. Similarly, you use comboboxes to select the Clerk who booked the order and the Checked who checked it in from the Employees table.
