rajesh c. say:

If there are multiple Pivot Tables placed Horizontally in the same sheet, then definitely there may be a possibility that the row which you are trying to insert may clash with one of the Pivot Table somewhere on the right hand side.

In such a case, you must first try to get the last row of each Pivot Table and compare the largest last row with the row number which you are trying to insert. 

In short, insert a row only after the last row of the longest Pivot Table.

You may have to insert code in your macro for checking this.

Alternatively, if your Pivot Tables are one below the other and if you are trying to insert a row below (between) and of the two Pivot Tables, then this problem may not arise at all.

albert d. kallal say:

What you do is link the tables WITHOUT the uid/password.

You then prompt the user for UID/logon, execute the logon, and THEN all tables and pass-though quires etc will by “magic” use the current logon. What Access does is cache the last used logon for the duration of the session.

The code to “test” and logon looks like this:

 

   On Error GoTo TestError

   Dim dbs          As DAO.Database

As DAO.QueryDef

   Set dbs = CurrentDb()

    qdf.Connect = strcon

    qdf.ReturnsRecords = False

    'Any VALID SQL statement that runs on server will work below.

    qdf.SQL = "SELECT 1 as test"

    TestLogin = True

    Exit Function

TestError:

End Function

And the code to create the conneciton string (dbCon) can look like this:

Public Function dbCon(ServerName As String, _

    ' returns a SQL server conneciton string

If IntegratedSecurity = False Then

Else

End Function

So once you logged in, then all of your linked tables etc. will use the uid/password you used for the logon. You can thus for example launch a custom form that does the above logon for you (ask for user id and password). You then execute TestLogon based on this connection string. Once done, you should be off to the races. All linked talbes, forms etc will then use this logon/password.

This “trick” and ability to logon to a database and use linked tables that DO NOT have a uid/password in the connection string is outlined here:

Power Tip: Improve the security of database connections

http://blogs.office.com/b/microsoft-access/archive/2011/04/08/power-tip-improve-the-security-of-database-connections.aspx

stefan blom say:

The symptoms you described in your original post suggest that numbering wasn't set up properly. By creating a numbered style you apparently got it right.

For future reference, here's how to change the indentation for a numbered list: Click the actual number in the document and then right-click and choose Adjust List Indents from the context menu.

For a single-level list, you will see the Adjust List Indents dialog box as illustrated above. "Number position" determines the position of the list number. "Text indent" sets the location for the second and subsequent text lines. "Add tab stop at" determines where text start on the first line. If you want a hanging indent, make sure that "Text indent" and "Add tab stop at" are the same (and larger than "Number position" of course). All measurements are from the left margin of the text container (page, table cell, text box, etc.).

Click the Office button, then Word Options in the lower right. Select Customize from the list on the left and click Keyboard Shortcuts: Customize in the bottom of the right pane.

                   

Select Home Tab from the Categories: list in the upper left and EditPaste from the Commands: list in the upper right. Note that whatever command is currently assigned will appear in the Currently assigned to: notation in the middle left (as circled in red above). Click the Assign button in the lower left to reassign Ctrl+V to Paste.
