Autorecovery not working - Microsoft Community

mrsfink say:

Autorecovery not working

My auto recovery is not working even though it appears to be set up to save every 5 minutes.  How can I get it to work?  I lost 45 minutes of work, nothing has been saved in the auto recovery file for months.
bob jones aka: cybertaz say:
Additional to Phillip's reply:

  • AutoRecover is a feature provided strictly as a tool to help avoid the loss of data in the event of a critical failure of the software or the system [e.g., power failure, crash].
  • AutoRecover is NOT auto-save. Under no circumstances does it automatically update the actual Excel file to which it pertains.
  • You must save manually in order for the feature to begin working. If you neglect to save again within the specified time frame an AutoRecovery file is created & it is updated on the same basis. It's still recommended that you periodically save your work [a simple Command+S when needed is all it takes].
  • AutoRecovery files are session-specific... When a file is Closed normally any AutoRecovery file established during that session is deleted. The file is not retained unless something happens to prevent that from happening -- They do not accumulate data over multiple sessions, let alone "months".

Contrary to Phillip's suggestion, though, I would not recommend turning the feature off. It can be a life saver as long as it's understood & not expected to do something for which it isn't intended. I would suggest that you restore it to the 10 minute default setting though.

Also, the backup copy feature is certainly a good one, but it isn't a replacement for AutoRecover in the way Phillip implies. My understanding is that once the copy is created on opening the file none of the work you go on to do is written to it. If you neglect to save as suggested above & do not have AutoRecover on a failure will most definitely cause the loss of everything you've done since opening the file.


bob jones aka: cybertaz say:
Assuming that there was no other software or network intervention...

I believe what you're referring to is an alteration to AutoRecover that was introduced in Office 2010 [Windows]. It is active by default & enables retention of a file's AutoRecovery file [assuming there is one] if the user closes the file without saving.

Conventional wisdom dictates to most experienced users that it immediately be disabled for exactly the reasons you've substantiated by way of your unfortunate experience:

  1. It lulls users into a false sense of security even more subversively than AutoRecover, &
  2. It isn't available in any other version of Office or in compatible programs people often use to work with their Office documents.

You might want to look into the features provided by OS X. They vary depending on which version of the OS you're running & some of the latest are not yet fully supported. Time Machine is one that should be available to you in any version. OS X Auto Save was introduced in 10.8 but I've had no experience with it... It may or may not be functional.

To address the Blank cells, use

To deal with the notification of duplicate entry in one column requires VBA...  create a list of all the teachers names and name that range "Teachers", then copy this code, right-click the sheet tab of the Entry sheet, select "View Code" and paste the code into the window that appears. Then save the workbook as a macro-enabled .xlsm.

Private Sub Worksheet_Change(ByVal Target As Range)
    Dim rngT As Range
    
    For Each rngT In Range("Teachers")
        If UCase(Target.Value) Like "*" & UCase(rngT.Value) & "*" Then
            If Application.CountIf(Intersect(Me.UsedRange, Target.EntireColumn), "*" & rngT.Value & "*") = 1 Then
                Exit Sub
            Else
                Application.EnableEvents = False
                Application.Undo
                Application.EnableEvents = True
                MsgBox "That teacher has already been assigned."
                Exit Sub
            End If
        End If
    Next rngT
    
End Sub

technichor say:

Macro naming rules from Microsoft:

  • You must use a letter as the first character.
  • You can't use a space, period (.), exclamation mark (!), or the characters @, &, $, # in the name.
  • Name can't exceed 255 characters in length.
  • Generally, you shouldn't use any names that are the same as the functions, statements, and methods in Visual Basic. You end up shadowing the same keywords in the language. To use an intrinsic language function, statement, or method that conflicts with an assigned name, you must explicitly identify it. Precede the intrinsic function, statement, or method name with the name of the associatedtype library. For example, if you have a variable called Left , you can only invoke the Left function using VBA.Left .
  • You can't repeat names within the same level of scope. For example, you can't declare two variables named age within the same procedure. However, you can declare a private variable named ageand a procedure-level variable named age within the same module.
