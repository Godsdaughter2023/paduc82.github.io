john - project say:
Jim,
Just for reference, using paste links is NOT the way to link tasks in a project. Doing so invites corruption. The correct method for linking tasks is to establish a predecessor/successor logic network for the whole schedule. Every task, except the completion milestone will have a successor and normally all tasks will also have a predecessor, unless they must start independently for some reason.

Now let me address your main issue. First of all I agree that for the manually scheduled tasks you describe Project should NOT move the milestone to 8/29/11 8:00 AM. For whatever reason the algorithm behind the Move Project utility was changed from previous versions. The algorithm has a glitch and I do not know if it was fixed in the SP1 update for Project 2010.

However, you would have fewer headaches in starting out with Project if you stick with auto-scheduled tasks. In your example task A would be entered with a duration of 5 days. So with a project start date of 8/1/11 task A would run from 8/1/11 (8:00 am)  through 8/5/11 (5:00 pm). Then if the milestone is linked in a finish-to-start relationship with task A, (i.e. assuming task A is ID 1, the Predecessor field for the milestone would show 1FS, or simply 1 since the FS is default), it will start and finish on 8/5/11 (5:00 pm). Just what you want. Unfortunately using the Move Project utility will still show the milestone incorrectly at 8/29/11 8:00 am. However, in your example you can achieve the desired schedule movement by simply changing the Project Start date from 8/1/11 to 8/22/11. This of course assumes you did NOT manually enter start or finish dates into the schedule. Doing so creates a constraint.

Lots of stuff to learn with Project. Lots of stuff. Ideally, to learn how to use Project properly you should enroll in at least a 2 day class with LOTS of hands on work. You might want to check out the offerings at, MSProjectExperts.com. For some initial self paced material, you could also check out Mike Glen's tutorials at, http://project.mvps.org/links.htm.

Hope this helps.

barb reinhardt say:

I have since discovered (though some fairly convoluted steps) that there were AutoShapes in both of these workbooks, that when deleted, allowed me to save the workbook.  

 

One of them showed the following as the formula:

=EMBED("Excel.Sheet.12","")

 

The other was

=EMBED("Excel.Sheet.8","")

 

There obviously is some commonality in this problem.  

 

I used this code to select the autoshape in the active workbook and manually deleted it. 

 

Option Explicit

Sub EvaluateShapesinActiveWB()

Dim WS As Excel.Worksheet
Dim myShape As Excel.Shape
Dim aWB As Excel.Workbook
Set aWB = ActiveWorkbook

For Each WS In aWB.Worksheets
    For Each myShape In WS.Shapes
        If myShape.Name Like "AutoShape*" Then
            WS.Activate
            myShape.Select
            Exit Sub
        End If
    Next myShape
Next WS

MsgBox ("Complete")

End Sub
