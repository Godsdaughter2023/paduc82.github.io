Glad to see the issue is resolved on your PC.

 

For explanation, it is a client issue which appears after upgrading. The issue occurs because Office 2016 doesn’t correctly load Office 2016 authentication component for authentication, but uses remaining Office 2013 component. Note that, by default, Office 2016 (modern authentication, ADAL) and Office 2013 (legacy authentication) use different authentication method. Therefore, it fails to redirect you to your Single Sign-On login page when it sends your input account to the server. And without your on-premises identity provider’s verification, you won’t gain access to Office 365 using your user account.

 

I understand that it will create some inconvenience to accomplish this task in a large organization.  From our perspective, to completely fix this client issue, the best practice is to uninstall all existing Office using the Fix it tool first and then re-install the latest one. This could make sure the client uses the Office 2016 authentication method (ADAL) to connect to Office 365.

Additionally, to automate the installation of Office 2016, you can use Office Deployment Tool 2016 for those problematic PCs.

a. user say:

Thanks to both of you for your help.

After having done some research on this topic, I understand why it's not possible to open a password-protected file in a public cloud service like OneDrive or Google Drive: To do so would mean that the password must be stored in the cloud service, which most people would find unacceptable.

The protection offered to files stored in the cloud comes not from the individual file's password, but from the password to the cloud service itself, which makes two-factor authentication that much more important. Without 2FA, a single password gives access to every file stored in the cloud service.

You can think of it this way: Were it possible to open a password-protected file in the cloud, you would need to enter two passwords: one for the cloud service and another for the file. With 2FA you would need to enter the password for the cloud service and the authentication code. Neither approach is more or less inconvenient, however, 2FA may be more secure because it doesn't only rely on a password.

Thinking further ahead: If you want to work on a sensitive file while travelling or with a team, it doesn't make sense to password-protect that file on the local computer, because that would make it impossible to sync changes to the file made while travelling or with the team. You really need to go 'all in' with the cloud service.

It makes sense to store password-protected files in the cloud as backup copies, but if you intended to actually work with those files then you want to dispense with a file-level password and rely instead on the cloud service's own password and 2FA.
