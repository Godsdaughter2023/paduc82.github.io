Extranet Smart lockout feature (ESL) - Microsoft Community

nuno-tavares say:

Extranet Smart lockout feature (ESL)

Extranet Smart lockout feature (ESL)

On March 22/2018 a new update was released for Windows server 2016 (KB4088889). This update brought us the new ADFS extranet smart lockout feature, or ESL.

The extranet lockout feature will stop the brute force attacks by locking the account on the ADFS while preventing the accounts to be locked in the Active Directory.
As the name suggests, this feature will only apply if the authentication request is coming from the extranet, and for Username/Password authentications.

The new ESL feature consists on the implementation of a new parameter called <ExtranetLockoutMode>

This parameter holds 3 different values:

ADPasswordCounter – This is the standard value. The “soft extranet lock”. It does not differ based on location.

ADFSSmartLockoutLogOnly – This is the extranet smart lock logs only mode. It won’t reject the requests. Instead, it will audit the events for analyses.

ADFSSmartLockoutEnforce – This is the effective extranet smart lock. It will block the bad password attempts when the threshold is reached.

Before you begin, we need to grant read/write permissions to our Administrator over the ADFS artifact database by executing:
Update-AdfsArtifactDatabasePermission
(Authenticate with an ADFS Administrator)

Once the permissions are set, we can start the extranet smart lockout configuration.

To enable the extranet lockout, we need to execute the cmdlet:

Set-AdfsProperties -EnableExtranetLockout $true -ExtranetLockoutThreshold <value> -ExtranetObservationWindow (new-timespan -Minutes <value>)

Let’s examine the parameters individually:

EnableExtranetLockout – This parameter represents the on/off switch. Is always a hard-codded value, either $true or $false

ExtranetLockoutThreshold– This parameter represents the number of failed password attempts before the account is blocked.
Note: The lockout threshold should always be smaller then the one on the Active Directory, so the accounts gets blocked in the ADFS only while the users are still able to access their Active Directory through the intranet or using a VPN.

ExtranetObservationWindow – This parameter represents the observation time window between blocks. When the lockout threshold is reach, the observation window timeframe begins. Once the observation window timeframe expires and a successfully authentication attempt is made, the account is unblocked automatically, and the count is reset to 0. If an incorrect authentication attempt is made, instead the observation window re-starts.

Managing the ESL:


Once you set the ExtranetLockoutMode to <ADFSSmartLockoutEnforce> the fun begins ��

PS: Make sure you reboot the ADFS service on all the ADFS nods on the farm:

Restart-Service adfssrv

By default, all the accounts logs will look like this:

Once the user successfully authenticates (password-based), the IP address is automatically added to the FamiliarIps list.

If the lockout threshold is reach, the account is automatically blocked, and the observation window starts:

Even though the account is blocked, it`s only blocked from unknown locations. The user will still be able access his account from a familiar location.

This is a massive improve on the extranet lockout feature.

On a brute force scenario, where the account is constantly being blocked from unknown locations, what happens when the account owner tries to access his account also from an unknown location?
He will not be able to access his account. However, he can simple contact the Administrator to have his current IP address added in the familiar locations:

Set-AdfsAccountActivity -Identity <UPN> -AdditionalFamiliarIps "IP Address(s)"

NOTE: At any time, the extranet smart lockout account activity can be set to default:
Set-AdfsAccountActivity -Identity <UPN> -Clear

The account access attempt log will be displayed on the respective ADFS nod under “Windows Logs/Security”

Update:

On June 2018, Microsoft introduced a Banned IPs list.
If any of the listed IP addresses are in the "x-forwarded-for" or "x-ms-forwarded-client-ip" headers, ADFS will block te authentication request immediatly.

Adding banned IPs:
Set-AdfsProperties -AddBannedIps "1.2.3.4", "::3", "1.2.3.4/16"

Allowed formats:
1. IPv4
2. IPv6
3. CIDR format with IPv4 or IPv6
4. IP range with IPv4 or IPv6 ( i.e. 1.2.3.4-1.2.3.6 )

There is a limit of 300 entries for banned IP addresses. You can use CIDR or range format to deny a large block of entries with a single entry.

View the banned IPs list:
Get-AdfsProperties | select BannedIpList -expandproperty BannedIpList

Removing banned IPs:
Set-AdfsProperties -RemoveBannedIps "1.2.3.4", "::3", "1.2.3.4/16"

Connect2XL_Exit:
On Error Resume Next
