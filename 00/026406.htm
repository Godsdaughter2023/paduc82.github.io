Slow LAN & WAN Save 512 byte SMB requests Powerpoint Office 2011 - Microsoft Community

ibxxx say:

Slow LAN & WAN Save 512 byte SMB requests Powerpoint Office 2011 Mac OS X

Hello Support,

Saving Powerpoint MAC 2011 files to SMB based shares is very slow (or to anything other than a local hard drive).

A packet capture of the process reveals that Powerpoint sends the file to the SMB share in 512 byte packets.

Network latency overhead of so many small packets makes the save process slow to a LAN SMB share, and painfully slow via a WAN with its extra latency. 

512bytes is the sector size of the local HD, where microsecond latency does not show any problem, and maybe why Powerpoint is choosing to save in this size segments.

Typical Powerpoint file size for me is between 15 & 20MB.

Of note, when the file is saved for the first time to the SMB share, save speed is OK.

The TCP_PSH bit is set in all of the SMB requests.

Trace file shows when a file has been opened from the SMB share, the save process sequentially re-reads the original file in 512 byte requests to the SMB share, & writes  back each 512byte segments to the SMB share to a new file (same filename with a 1 appended to the end of the filename).

Once the whole newly named file is written (hours later for a 20MB file size), the original is deleted, and the "filename with append 1" is renamed to the original name.

No problem with the process as such, but the 512byte read/writes make the file save process unworkable/too slow to anything other than a local disk.

Workaround is to save all Mac Office files opened from an SMB share, to a new filename, thus avoiding the 512byte at a time save routine.

I'd like this confirmed as a bug/to understand other workarounds.

Packet traces from two Mac OS X 10.6.8 , Office 2011 PowerPoint 14.0 (100825) through latest patches show same problem.

Thank you Support Team.

Slight logical problem with that, Ron. It's always going to tell you that you have at least one file, even if none are present, and if you point it at a non-existent folder, it throws an error.

A couple of minor mods fixes that and turned into a function, it may be more useful generally:

 

Option Explicit


Function FileCount(ByVal sFolder As String) As Long
   
Dim lFileCount As Long
Dim sFilename As String
   
sFilename = Dir(sFolder)

Do Until Len(sFilename) = 0
    lFileCount = lFileCount + 1
    Debug.Print sFilename
    sFilename = Dir
Loop
   
FileCount = lFileCount
   
End Function

Sub Test()
    Debug.Print FileCount("C:\temp\")
End Sub

Here is a function that should work in all VBA modules for the various Office applications...

Function FileCount(ByVal Path As String) As Long
  Dim FSO As Object
  Set FSO = CreateObject("Scripting.FileSystemObject").GetFolder(Path)
  FileCount = FSO.Files.Count
  Set FSO = Nothing
End Function

Simply pass in the path to the directory and it will return the count of the number of files in the directory. Here is an example...

Sub Test()
    Dim Path
