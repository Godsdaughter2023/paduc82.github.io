joeu2004 say: srcole1 wrote:
I am experiencing what seems to be a random problem that when My Combobox pulls specific values from my range of cells, it is reducing the value by .000000000001.

Example:
My ListFillRange is : T8:T348

These cells each have a formula that adds 0.05 to the cell above it. so I have a column of values that increase in increments of .05  (1.05, 1.10, 1.15, etc)

This is a common problem with computer binary arithmetic.  It is not a defect; just something we all must deal with.  For more details than necessary, see http://support.microsoft.com/kb/78113.

In a nutshell, most non-integer numbers cannot be represented exactly in binary.  For example, 0.05 is actually represented internally as 0.0500000000000000,0277555756156289135105907917022705078125, and 1.05 is actually 1.05000000000000,00444089209850062616169452667236328125.

The fix is:  explicitly round to the precision that you require.  For example, =ROUND(T8+0.05,2).

Formatting alone affects only the appearance of a value.  It does not round the actual value.

Alternatively, you might set the calculation option Precision As Displayed (PAD), which is under Tools > Options in XL2003.  But I do not recommend it.  PAD can have irreversible adverse effects if you enter constants before setting the cell format precision as desired.

ken sheridan say:

This sums all hours worked by the employee, but I want to see the total hours worked each day.  Also, does this round to the nearest quarter hour.  For example, clockIn time is 8:07am rounds down to 8:00am and clockin time of 8:08am rounds up to 8:15am.


For that you need to use the traditional 'round-half-even' algorithm colloquially known as 'banker's rounding'.  See http://eetimes.com/design/programmable-logic/4014804/An-introduction-to-different-rounding-algorithms for everything you ever wanted to know about rounding algorithms, and much you probably didn't.

My RoundTime function uses this method of rounding, so you can call this in a query grouped per day.  As you want to round the start and end times on which the summations are based you need to apply the function to each time, not to the result of the summation.

As the times per day will very likely sum to 24 hours or more you cannot simply format them as 'hh:nn:ss", but instead use my TimeElapsed function.  Putting all this together and using the TimeLog table from my TimeSheet demo at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

a query to sum the differences between the rounded times per day would be:

SELECT WorkDate,
TimeElapsed(
   SUM(RoundTime(TimeEnd,15)
   -RoundTime(TimeStart,15))
,"nn:ss") AS TotalTime
FROM TimeLog
GROUP BY WorkDate;

or:

SELECT WorkDate,
TimeElapsed(
   SUM(RoundTime(TimeEnd,15)
   -RoundTime(TimeStart,15))
,"nn") AS TotalTime
FROM TimeLog
GROUP BY WorkDate;

if you don't want to show the seconds, which will always be zero of course due to the rounding.

BTW:  as a supplementary note, the RoundToInterval function which I posted is really designed for use with the sort of scientific data used in my line of work, so if you were ever tempted to use it you need to take the advice of Richard Feynman in the quote below my signature.  It's basis is that when rounding up it rounds to the floor of the next interval above that in which the value is located, and when rounding down, to the floor of the interval in which the value is located.  

The basis of this is that, using time as an example, a value such as #08:30# does not represent a point in time, i.e. an infinitely small duration, but a duration of a particular precision, i.e. 1 minute in this case, 1 second on the case of  #08:30:00#.  Consequently the result of rounding up this value is:

? Format(RoundToInterval(#08:30#,#00:15#),"hh:nn:ss")
08:45:00

while that of rounding down is:

? Format(RoundToInterval(#08:30#,#00:15#,False),"hh:nn:ss")
08:30:00

The former might not be what's wanted with a more day-to-day interpretation of what's meant by rounding up.  In my line of work one area (quite literally) in which this comes into play is in location data using coordinate systems like the UK Ordnance Survey's National Grid Reference system, in which a location can be expressed at any desired precision.  Gunnery sergeants like my late father would understand this!
