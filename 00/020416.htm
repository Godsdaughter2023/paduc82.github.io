rod gill say:

Slack is calculated only. You can change lag, but not slack.

You only need to add lag if a resource can't start the next task immediately. Where a resource is working only 50% on your project, set the units to 50% and then no need for lag. However order equipment does have a lag before receive equipment happens.

If a resource can only work 4 hours a day on a task and the task requires 24 hours of work, then setting the percent allocation will simply cause the planned duration to be 6 working days rather than 3 working days. Do you schedule tasks so that a certain amount of slack is "created", or do you add lag in order to have more buffer?

normgoldsmith say:

Pug,

 

I'm curious as to why you think that  "scheduling task successor tasks right on the tail end of their predecessor tasks" is a mistake. The implication is that your task durations have been too optimistic. If your average error is 20%, why don't you schedule your tasks to take longer with lower effort per day - e.g. follow Rod's advice on changing resource usage?  If you do that, you can use slack to decide which tasks are likely to cause you to be even later.

 

What I have often seen is that people working on the project look at the Gantt chart and assume that they don't have to finish until that date. What you might want to do is to revert to the "old" PERT method where you create three schedules based on optimistic, pessimistic and likely durations. Publish the likely one to your staff, try to get your staff to deliver on the optimistic one,  and publish the pessimistic one to your client. Glen Searle of PCubed made available the macro code for creating PERT plans. Sorry I don't have the link immediately available.

 

NormG

win10nuc2 say:

Fail when setting up new account using external email address: must use an @outlook.com address

I've started this as a separate thread to highlight the issue.

If an account is set up using an external email address (i.e. not @outlook.com) then the Outlook.com Mail setup fails when it tries to set the region and timezone. It is then necessary to create an @outlook.com alias, make that the primary, logoff, logon to the alias in an InPrivate session, and then the region/timezone can be set. The original address can then be reset as primary. This was explained by John_E_M  (of MSFT) as the workaround in

http://answers.microsoft.com/en-us/outlook_com/forum/oemail-oaddacc/unable-to-set-time-zone-on-new-outlook-account

There have been other suggestions about using clearing browser caches, using a different browser, but they do not work (for me), only the use of @outlook.com / InPrivate.

This is an issue with the new web version of Outlook.com, it did not arise previously.

It needs to be fixed by Microsoft, as changing the primary will affect logins/display names on linked devices/phones, and may cause accounts to be reset (that used to be an issue but may have been fixed in latest Win10). Until then, there should be a warning to use an @outlook.com account at first when new accounts are being set up, and only when this has been successfully completed and the web Mail initialised can you create an alias for the external address and set that to primary.
