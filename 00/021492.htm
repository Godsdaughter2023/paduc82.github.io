greg maxey say:

Jonesey,

Why do you have command buttons showing on your document?  If you can move them to the Ribbon or QAT then you don't have the problem.

Otherwise you are going to have to add code to your document that intercepts the print command and "whites" out the command buttons:

Sub FilePrintDefault()
  HideCmdButtons
  ActiveDocument.PrintOut
  ShowCmdButtons
End Sub

Private Sub HideCmdButtons()
Dim oButton As InlineShape
  For Each oButton In ActiveDocument.InlineShapes
    If oButton.Type = wdInlineShapeOLEControlObject Then
      With oButton.PictureFormat
        .Brightness = 1
        .Contrast = 0
      End With
    End If
  Next
End Sub
Private Sub ShowCmdButtons()
Dim oButton As InlineShape
  For Each oButton In ActiveDocument.InlineShapes
    If oButton.Type = wdInlineShapeOLEControlObject Then
      With oButton.PictureFormat
        .Brightness = 0.5
        .Contrast = 0.5
      End With
    End If
  Next
End Sub

tom_bpc say:

Châu Trần,

  1. Answering your initial question… No, you can’t force the progress line to zag left at ID11.  That is a summary task whose current start date is in the future (i.e. to the right of the progress line), and Project omits those tasks from the progress line.  There’s no obvious reason for this behavior other than to de-clutter the Gantt chart.  If you advance the Status Date to 22Jul’19 or insert an additional progress line at that date, then the expected progress point will be inserted in the view.
  2. The progress line is rooted in old-time practices for daily task management using manual methods (i.e. weighted string and pushpins) – between schedule updates.  While intuitively useful for individual task bars, such methods were never applied to nor useful for summary bars.
  3. From the beginning, Project’s implementation of progress lines has been an exercise in creative hand-waving, with close examination exposing inconsistencies from version to version and from case to case.  You’d do well not to trust it in general.  For summary tasks in particular, progress lines (like progress bars) seem to use duration-summarizing methods that are ultimately invalid for these purposes, so in general the progress point locations are meaningless.  Unfortunately, Microsoft’s decision to include summary tasks on progress lines continues to confuse rather than clarify, and there is no way to turn them off.
  4. If the “qualified Consultant Engineer” on your project is in fact competent in project controls, then the “various requirements on the project report and display” would NOT include anything like a progress line.  Rather the focus would be on Earned Value or Earned Schedule curves (and associated variances), Critical and Near-Critical Paths, and forecast finish variance of key delivery milestones.  All of these are easily generated by a competent Contractor team that follows good practices including the recommendations that John and Trevor have provided.

Good luck, tom

trevor rabey say:

How can ID11 have any progress if none of its tasks have any progress?

ID11 is a summary. Summaries are not tasks. They are just a way of organising tasks within a WBS.

You have linked summaries, which should be forbidden because it is a very poor practice and a common mistake. Remove those links.

The progress of summaries is irrelevant, since only tasks can have progress.

It is v difficult to reproduce what you have because you have not displayed predecessor and successors.

Give us a screenshot which shows those columns.

Also, you have predecessors with negative lag, which should be avoided or forbidden.

Every task should have at least one FS0 predecessor and at least one FS0 successor.

When these mistakes are fixed, the progress status will become clear, and don't bother with progress lines (uninformative, un-interpretable)

After that, the real problem is that your tasks durations are too long.

Usually, a much finer level of detail is needed for a construction project.

Hello Guys,

I am from Brazil and have the same issue with solver.

After changing the MAC OS to English, like Mike Middleton said the Solver works normally.
We found that each Excel Language folder have a solver.xlam file.

We made the following steps and now the Solver is working with Excel in Portuguese.

Open Go > Applications > Then locate the Microsoft Excel App
Right click into the Microsoft Excel App and click in Show Packages Contents
Now, open the folder Contents > Resources

Now, find the en.lproj and open the folder Add-Ins
Copy the file Solver.xlam

After copying the file, back to the Resources folder and access the folder about your language (in my case I need to enter the in the folder pt.lproj - about Brazilian Portuguese). Then paste and replace the file Solver.xlam present in this folder closing all Finder Window after that.

Open Microsoft Excel and enable the Solver Add-in (If you don't enabled yet)

Now, the solver will be working normally.
