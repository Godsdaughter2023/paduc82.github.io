ken sheridan say:

There is no reason why the NOT EXISTS predicate cannot be used in JET/ACE SQL as used by Access, e.g.

SELECT *
from salestable AS ST1
WHERE saledate >= #2016-06-02#
AND NOT EXISTS
    (SELECT *
      FROM salestable AS ST2
       WHERE ST2.customer=ST1.customer
       AND saledate < #2016-06-02#);

The less efficient NOT IN predicate could be used:

SELECT *
from salestable
WHERE saledate >= #2016-06-02#
AND customer NOT IN
    (SELECT customer
      FROM salestable
       WHERE saledate < #2016-06-02#);

However it is necessary when using this predicate to ensure that the column returned by the subquery is not NULL in any row or the query will fail (you can easily see why by expanding it algebraically).  This is unlikely here as the customer column will almost certainly be defined as NOT NULL.

There is no reason why two instances of a table cannot be joined with an OUTER JOIN, but it would not work in this case as you would need to restrict the instance of the table on the right side of a LEFT OUTER JOIN, which in effect would make it an INNER JOIN.  The way to do it would be to join one instance of the table to a subquery:

SELECT DISTINCT ST1.*
FROM salestable AS ST1 LEFT JOIN
    (SELECT *
     FROM salestable AS ST2
     WHERE saledate <  #2016-06-02#) AS ST2
ON ST1.customer = ST2.customer
WHERE ST2.customer IS NULL;

Note that in this query you do not need restrict the ST1 table on the date column as any rows earlier than 2016-06-02 would not be returned by virtue of the NULL restriction on the ST2.customer column.

BTW, the dd/mm/yyyy short date format cannot be used for a date literal in JET/ACE SQL.  It must either be in US short date format, or an otherwise internationally unambiguous format such as the ISO standard format for date notation of YYYY-MM-DD, as used in the above examples.
