scottgem say:
Gina gave you the correct answer. The use of Lookup fields in Access tables is NOT recommended and this is one of the reasons why. A lookup field masks what is actually stored in the table. Lookup fields are ONLY recognized within Access. So, as soon as you try to reference the data from outside Access, you will only see the actual stored value. 

If you need to do a mail merge in Publisher, then you need to create a query that joins the foreign key field in your table with the lookup table it looks up to and then draw the description field into your query.

Junction tables give multiple records per design, one for each color. But for a catalogue merge we need all the colours for a design in the same data record.

 

Here is a simple workaraound:

Problem:

We have data records in a Table T-Group each of which has four fields for colour: Colour1, Colour2 etc. which we need for a catalogue merge in Publisher.

We don’t want to have to type in hundreds of descriptive colour names to our T-Group table, so we want a lookup for colours.

If you put a lookup to a table T-Colour with an ID and descriptive name field called colourName, the colour fields in T-Group store the ID, a number, of the colour record rather than the descriptive name and Publisher displays the number.

Solution:

Create a Query on T-Colour with one field viz. colourName, sorted on this field.

Make the type of Colour1, Colour2 etc. in T-Group short-text.

Add a lookup using the Query on T-Colour. Now when you add record to T-Group you have lookups for the colours, the descriptive names are stored in the record and displayed in Publisher.

Your workaround is a bandaid, and there is, at least one, simple answer to your question. 

What you do is add the colors table 4 times in your query as aliases. Then the SQL will look like this:

SELECT [T-Designs].DesignID, [T-Designs].Group, [T-Colors].Colorname AS [Color-1], [T-Colors_1].Colorname AS [Color-2], [T-Colors_2].Colorname AS [Color-3], [T-Colors_3].Colorname AS [Color-4]
FROM ((([T-Designs] INNER JOIN [T-Colors] ON [T-Designs].Color1 = [T-Colors].ColorID) INNER JOIN [T-Colors] AS [T-Colors_1] ON [T-Designs].Color2 = [T-Colors_1].ColorID) INNER JOIN [T-Colors] AS [T-Colors_2] ON [T-Designs].Color3 = [T-Colors_2].ColorID) INNER JOIN [T-Colors] AS [T-Colors_3] ON [T-Designs].Color4 = [T-Colors_3].ColorID;

The resulting query might look like this:
1  A Red Blue Green Yellow
2  B Purple Orange  Pink Black

There might be other, better ways of doing this and maintaining normalization. But assuming that every design has 4 colors, no more, no less, this is a viable. design.

jay freedman say:
I don't think Suzanne has all of that, but here are some references that should help:
  • Updating all fields: Make sure the "Update fields before printing" option (on the Display tab of the Options dialog) is checked. Go to Print Preview and back to the view you came from. [Using Ctrl+A to select all and F9 to update selected fields will get all the ones in the main body of the document, but may miss ones in headers/footers, footnotes, text boxes, etc.) There is also a macro you can write, but that probably isn't on the exam.
  • Converting all fields to text: Ctrl+A to select all, then Ctrl+Shift+F9 to "unlink" all selected fields (convert them to their results as plain text). Unlike updating fields, there's no keystroke way to get fields that aren't in the main body at the same time -- you'd have to work your way through all the other parts (which Word calls "stories"). Again, this could be done with a macro.
  • Creating custom fields: Either press Ctrl+F9 to insert field braces and type in the field code, or use the Insert Field dialog and choose options or type your own additions to the field code. For an extensive treatment of what you can do with field codes, see http://www.gmayor.com/formatting_word_fields.htm.
  • Change field display format: Also covered in Graham Mayor's article.
  • Document Information Panel: The DIP in Word 2007/2010 is really an InfoPath object, meant mostly for integration with SharePoint. I don't have a very good reference for it. Try http://www.google.com/search?q=document+information+panel+word+2007.
  • Custom document property field: First you have to create a custom document property. In the DIP, click the Document Properties "button" in the upper left corner and click Advanced Properties. (Far from being "advanced", this opens the Properties dialog from Word 2003.) Click the Custom tab, enter a name and value, and click the Add button. Now you can insert a DocProperty field, either through the Insert Field dialog or with Ctrl+F9 and using the DocProperty keyword and the custom property's name. Once more, custom properties and DocProperty fields can be created by a macro.
