Office click to run products install to the win drive, you cannot do a custom install, its an all or nothing install

There is a method, if you are prepared to delve in. Modify the following steps according to your specific situation:

I too was wanting to move my Office 365 install to a different drive. My main goal was to put it on a RamDisk to make it very quick to load.

  • I had created a RamDisk of 5GB, using http://www.radeonramdisk.com/software_downloads.php . The new drive was mounted as drive O:
  • Install Office 365 as normal, on the C drive.
  • Make a note of the two main Office directories:

C:\Program Files\Microsoft Office 15

C:\Program Files (x86)\Microsoft Office

  • Empty the Recycle bin before continuing.
  • Run Task Manager
  • Stop anything Microsoft Office related.
  • Delete both of the above directories
  • Open a CMD prompt, and run these commands:

MKLINK /J “C:\Program Files\Microsoft Office 15” “O:\Program Files\Microsoft Office 15”?

MKLINK /J “C:\Program Files (x86)\Microsoft Office” “O:\Program Files (x86)\Microsoft Office”

  • This created junction points for each directory on the C drive that Office installs to, and point them to the O drive. So the OS thinks its going to C drive for those folders, but in reality its going to the O drive.
  • Go to the recycle bin and restore all the deleted files. The files then appeared on the O drive.
  • Save the RamDisk to an image file (so it gets loaded when windows starts)
  • Reboot

After reboot, when I open an office app, its so fast you can’t time it!

You don’t have to do the ramdisk portion, its the junction’s that make it work.

I have 32GB of ram, and am only using about 5gb on average, so having a ramdisk makes sense for me.

RamDisks are about 8 to 10 times faster than SSD but are volatile memory, so it needs to be saved to disk if you don’t want to lose it.

In Ms Access 2010, how do I find the source table of a given cell on - Microsoft Community

reneropac say:

In Ms Access 2010, how do I find the source table of a given cell on an existing form?

I am working with a database that was created by someone else, and I need to find out which tables are feeding certain cells in an existing form. The Control Sources under Properties tells me the name of the specific cell where the data is pulled from, but where do I see the name/location of the table that cell is a part of?

We have several databases that are all interconnected, so looking for the table with the right cell has been quite painstaking and unsuccessful so far. Any help would be greatly appreciated!

john w. vinson say:

Well... Forms don't have "Cells". Access is not a version of Excel, and a database is NOT a Spreadsheet!

A Form has a "Recordsource" property; this can be a Table, or (more often) a Query selecting fields from one or more Tables. You can click the ... icon by the Recordsource property on the Data tab of the form's Properties, and Access will open a query design window showing the Query (or creating a Query based on the table - you needn't save the query but it will let you see the fields).

Each Control (not Cell, please!) on a Form has a Control Source property; this will usually be the name of a field in the Recordsource. 

Good luck - digging through and understanding someone else's database can be just as hard as building one from scratch, sometimes!

scottgem say:

First, Access doesn't have cells. That is a spreadsheet object, not a database object. Access has fields, that make up records. You need to get rid of the spreadsheet mentality when working with Access.

Second, A well designed Access form should only have one table in its Recordsource. the Recordsource property will tell you what table is being pulled from. If you are using a multi-table query as the Recordsource, and you have fields in multiple tables that have the same name, the controlsource should specify the table as well, otherwise you will get an error. 

ken sheridan say: Firstly, put all thoughts of spreadsheets out of your head.  A relational database is not a spreadsheet and does not have cells; it has tables with columns and rows, with each column representing an attribute of the entity type modelled by the table.  Each column position holds a value of the attribute in question.

To answer your question, the table can be identified from the form's RecordSource property.  This might be the table name, the name of a query or an SQL statement of a query.  In the latter two cases the table name will be in the query's FROM clause and the column name in its SELECT clause.

ken sheridan say:

A well designed Access form should only have one table in its Recordsource.

I'd qualify that, Scott, '….should normally only have one updateable table in its Recordsource'.  The inclusion of one or more referenced tables in a form's query is not unusual, if only for the purpose or ordering the recordset, e.g. a form for entering data into an EventParticipants table might well include the referenced Participants table so that the recordset can be ordered by LastName, FirstName.

Also there are occasional exceptions where updating of a column in the referenced table as well as those in the referencing table is legitimate.
