calculating ( 2 processor(s))--% - Microsoft Community

alokdubey say:

calculating ( 2 processor(s))--%

hi,

when I enter a calculation or formula my excel shows Calculating:( 2processor(s)) --% and it slow down the working.

joeu2004 say: Alok wrote:
when I enter a calculation or formula my excel shows Calculating:( 2processor(s)) --% and it slow down the working.  please advise because it is very irritating.

There are two separate issues....

Excel displays "Calculating" on the status line under the worksheet when the recalculation cycle takes a long time.  More about this below.

In your case, I suspect the "Calculating" status takes the form "Calculating:( 2processor(s)) --%" when your worksheet is complex and Excel takes advantage of multithreading.

Usually, multithreading is a good thing.  To learn more, see the http://msdn.microsoft.com/en-us/library/bb687899.aspx.  To learn how to control multithreading in Excel, see the paragraph "Change the number of processors used to calculate formulas" in http://office.microsoft.com/en-us/excel-help/change-formula-recalculation-iteration-or-precision-HP010054149.aspx.

Returning to recalculation....  The slow down is probably due to excessive recalculation.  There are many things that cause this.  For a wealth of information about this, see http://www.decisionmodels.com/calcsecrets.htm.  Be sure to look at many of the links at the top of the page as well.

Generally, a quick-and-dirty work-around is to set Manual calculation mode.  I don't know how to do that in XL2007 per se.  In XL2003, I click on Tools, Options, then the Calculation tab.

However, be forewarned that working in manual calculation mode can be very confusing.  For example, if you copy and paste a range of dependent formulas, most of the pasted formulas might not be recalculated, making it seem like they are not working.  You have to press F9 or ctrl+alt+F9 yourself when you are ready to recalculate.  That's the whole point:  you control when the recalculation is done.

A half-way step might be to simply disable automatic recalculation of tables, if you use them.  In XL2003, that option is under the Automatic calculation option on the Calculation tab.

macropod say:

Another option would be to automate the merge from Excel itself. That way, opening Excel gives you the opportunity to update the data to be used for the merge and the user needn't open both it and Word. The following Excel macro automates a Word mailmerge. As coded, the macro assumes the mailmerge maindocument is stored in the same folder as the Excel workbook and is named 'MailMergeMainDocument.docx' and that you want to merge every record on 'Sheet1' in the workbook.

Sub MailMerge()
'Note: A VBA Reference to the Word Object Model is required,
'set via Tools|References in the VBA Editor
Dim wdApp As New Word.Application, wdDoc As Word.Document
Dim strWorkbookName As String: strWorkbookName = ThisWorkbook.FullName
With wdApp
  'Disable alerts to prevent an SQL prompt
  .DisplayAlerts = wdAlertsNone
  'Open the mailmerge main document
  Set wdDoc = .Documents.Open(Filename:=ThisWorkbook.Path & "\MailMergeMainDocument.docx", _
    ConfirmConversions:=False, ReadOnly:=True, AddToRecentfiles:=False)
  With wdDoc
    With .MailMerge
      'Define the mailmerge type
      .MainDocumentType = wdFormLetters
      'Define the output
      .Destination = wdSendToNewDocument
      .SuppressBlankLines = True
      'Connect to the data source
      .OpenDataSource Name:=strWorkbookName, ReadOnly:=True, _
        LinkToSource:=False, AddToRecentfiles:=False, _
        Format:=wdOpenFormatAuto, _
        Connection:="Provider=Microsoft.ACE.OLEDB.12.0;" & _
        "User ID=Admin;Data Source=strWorkbookName;" & _
        "Mode=Read;Extended Properties=""HDR=YES;IMEX=1"";", _
        SQLStatement:="SELECT * FROM `Sheet1$`", _
        SubType:=wdMergeSubTypeAccess
      With .DataSource
        .FirstRecord = wdDefaultFirstRecord
        .LastRecord = wdDefaultLastRecord
      End With
      'Excecute the merge
      .Execute
      'Disconnect from the data source
      .MainDocumentType = wdNotAMergeDocument
    End With
    'Close the mailmerge main document
    .Close False
  End With
  'Restore the Word alerts
  .DisplayAlerts = wdAlertsAll
  'Display Word and the document
  .Visible = True
End With
End Sub

For PC macro installation & usage instructions, see: http://www.gmayor.com/installing_macro.htm

    kathleen orland say:

    Your nickname file will start to overwrite itself with new entries when it gets to a certain size, or it sometimes will corrupt.

    This happens if Outlook fails to close properly and doesn't write to the autocomplete cache. This is often due to add ins or synch software that integrate with Outlook (most common culprits: antivirus email scanning, synch software, FAX software, etc.).

    Open your Task Manager > Processes and look for Outlook running after you close it. If it is, you have an integration that interferes. http://www.howto-outlook.com/faq/outlookdoesntclose.htm 

    You may end up having to rename your .NK2 file (the nickname file). The location in Windows 7 is Drive:\Users\Username\AppData\Roaming\Microsoft\Outlook. If you do need to rename it, rename it to yourprofilenamegoeshere.nk2_old.

     

    You may even have to create a new profile: http://www.howto-outlook.com/faq/newprofile.htm 

    Just a note that the nickname cache is NOT a reliable feature and entries in the cache are NOT saved in your contacts. They are simply a list of your most commonly used recipients. Unless you deliberately create a contact for someone, you need to realize that the nickname cache is not doing it for you.

    suzanne s. barnhill say:

    Yes. In the "Define New Number Format" dialog,  you can choose Right as the alignment for the number. If you use a multilevel list (even if you're using only one level), then you can get considerably more control. In the Define New Multilevel List dialog, you can set the number alignment as Right and then also specify the position of the number, the text following it, and the hanging indent.

    In my experience, for 12-point type, the equivalent of flush-left numbers and a 0.25" indent is right-aligned numbers at 0.13" and the indent and tab position at 0.25". If you want the entire list to be indented roughly 0.25", then set the numbers at 0.38" and the indent at 0.5".

    For more on aligning numbers, see http://sbarnhill.mvps.org/WordFAQs/NumberAlignment2007.htm (also applies to Word 2010).

    andreas killer say:

    Yes, that is a stupid thing, I'm also stumbled over that.

    The reason is quiet simple: It is like as the names of the callbacks in the ribbon are pushed into a LILO stack, means a later loaded callback routine overwrites the earlier callback if they have the same name.

    So you have to ensure that the names are unique in your AddIn, I use GUID-names for my callbacks, e.g.:

    Public Sub OnActionButton_7A497E0F_7DAA_48F0_B4CC_E01B9FCAE4CC(control As IRibbonControl)
