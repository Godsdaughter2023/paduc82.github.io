ken sheridan say: While Tom does not explicitly say so in his reply, the point he is making is that this is an incorrect table design.  It's what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

The correct design is to store each genre per film in a related table which models the many-to-many relationship type between films and genres like this:

Films----<FilmGenres>---Genres

The genres for each film can then be entered in a subform based on the FilmGenres table.  You'll find an example of a means of automatically recasting a badly designed table like this into a set of correctly normalized tables as UnencodeColumns.zip in my public

fumei2 say:

One of the great advantages of using document objects (any objects really) is that you can action it directly without activating it.  By creating and setting an object VBA can do things  with it directly and\or in the background.

 

Using Doug example...

 

Dim myDoc as Document

Set myDoc = Documents.Open("C:\Temp\doc.docx")

With myDoc

'Do whatever you want to do with the document

End With

 

That is a literal statement.  myDoc IS now what you set it as - C:\Temp\doc.docx

 

You do not have to do things like ActiveDocument.whatever. You can use myDoc.whatever, and more importantly the document does not have to be active.  That is because you are telling VBA to do whatever to the object.

 

Declared objects can be just about anything, including multiple child objects.  For example:

 

Dim docFF as Formfields    'note the plural

Set docFF as ActiveDocument.Formfields

 

docFF now means ALL the formfields in the ActiveDocument.  So instead of have to point to a particular formfields with:

 

ActiveDocument.Formfields("Text1").Result

 

You can use:

docFF("Text1").Result

 

Objects are possibly the most powerful feature of VBA.
