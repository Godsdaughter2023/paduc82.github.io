tonyrr say:

vlookup won't return value when search value is numeric

I am a veteran Excel user, but this is throwing me. I have a workbook with two sheets; one is blank, and the other is a table with a column of numbers and a column of associated percentages (sheet name = SO RISK TABLE). On the empty sheet, I am trying to use vlookup to search for a numeric value in the first column and return the percentage in the third column. The numeric values in the first column start at 0.00 and go to 3.49 at 0.01 increments, and each value has a percentage associated with it.

The formula is =VLOOKUP(A1,'SO Risk Table'!$A:$C,3,FALSE)

The weird thing is, as long as the lookup value is from 0.00 to 0.09, vlookup returns the desired value. As soon as the lookup value is .010 or greater, it returns the no-value-found error (#N/A).

I have cleared the formats of both sheets entirely, and I've also ensured that the values in the data table are stored as numeric and not as text.

If it had a problem with the column A search values in the data table, I would have thought it wouldn't return a value with any lookup value. Why would it only do a few values correctly and not find the other 340 values?

-Tony
ronrosenfeld say:

The weird thing is, as long as the lookup value is from 0.00 to 0.09, vlookup returns the desired value. As soon as the lookup value is .010 or greater, it returns the no-value-found error (#N/A).

That's not exactly the case, at least when I download the workbook into my Excel 2007.

  0.06 --> #N/A

and 0.15-0.17 return the correct values.

 

The problem is that your values in Column A of the SO Risk Table are not exactly as they appear.  However, since they are "off" by less than Excel's precision limits, they appear OK both in the cell and in the formula bar.

 

I suspect that the values in Column A were generated by some kind of formula, and then a copy/paste special values was done to move the results to column A.  Although Excel only displays values to 15 decimals, it retains internal calculations to somewhat greater precision.  All this is in accord with the IEEE754 specification upon which Excel (and other SS programs) is based.  See:  http://support.microsoft.com/kb/78113

 

Also, there are issues where a particular decimal number cannot be represented exactly in binary, so it is approximated.

For a specific example, consider 0.12

 

The representation of 0.12 is

0.1199999999999999955591079015

 

But the 0.12 in your column A is

0.11999999999999998168132009368

 

The difference of 1.387778780782E-17 is enough to throw off the VLOOKUP.

 

I would suggest reviewing how the values in column A are generated, and round them to your desired level of precision.

 

(The above values were generated by xNumbers, a multiprecision floating point (free) add-in for Excel).
