It is not easy.

MS assumes users keep track of their product keys.  Re-installing Windows is effectively the same as dropping a hundred dollar bill in a shredder. You are out of luck.

  • Factory install, you have to go back to the factory
  • Product key card, you are SOL
  • online downloads, return to the online store you bought it from and log in with your email address and password. You can get your key from there and download the installation file
  • the product key is inside the big-ugly-ecologically-incorrect-yellow-plastic CD holder box

If you have an image copy backup of your system with Office activated on it, you can extract the key from there.


There is a very tiny chance that you can use some file recovery tools that claim to be able to recover files even after re-installing windows. The problem is, the file you are looking for is almost certainly in a location that was over-written, removing that option.

andreas killer say:
The VBA.MsgBox support only ANSI chars, but the OS has a MessageBoxW which support Unicode chars. But which Unicode chars are shown depends on the OS.

The official declaration of MessageBoxW has lpText and lpCaption as String, but when we pass a string with Unicode chars to that API from VBA, the Unicode chars are removed from VBA automatically. But we can use a trick and pass the string as pointer.

Copy the code below into a regular module.

Andreas.

'Version 1.0
'Andreas Killer
'24.01.2014
Option Explicit

#If Win64 Then
Private Declare PtrSafe Function MessageBoxL Lib "user32" Alias "MessageBoxW" ( _
  ByVal hwnd As LongPtr, ByVal lpText As LongPtr, ByVal lpCaption As LongPtr, _
  ByVal wType As Long) As Long
Private Declare PtrSafe Function GetForegroundWindow Lib "user32" () As LongPtr
#Else
Private Declare Function MessageBoxL Lib "user32" Alias "MessageBoxW" ( _
  ByVal hwnd As Long, ByVal lpText As Long, ByVal lpCaption As Long, _
  ByVal wType As Long) As Long
Private Declare Function GetForegroundWindow Lib "user32" () As Long
#End If

Private Sub Example_MsgBoxW()
  Dim Msg As String
  Dim Answer As VbMsgBoxResult

  'This is the usual way to build unicode chars in VBA
  Msg = "Is O" & ChrW$(&H2082) & " " & ChrW(&H2248) & " O²?"
 
  'But a little function makes it more easier
  Msg = ReplaceUnicode("Is OU+2082 U+2248 O²?")
 
  'Note: It depends on the OS if an unicode char is not shown as expected!
  Select Case MsgBoxW(Msg, vbYesNoCancel Or vbQuestion Or vbDefaultButton2)
    Case vbYes:    Debug.Print "vbYes"
    Case vbNo:     Debug.Print "vbNo"
    Case vbCancel: Debug.Print "vbCancel"
  End Select
End Sub

Function MsgBoxW(ByVal Prompt As String, _
    Optional ByVal Buttons As VbMsgBoxStyle, Optional ByVal Title, _
    Optional ByVal HelpFile, Optional ByVal Context) As VbMsgBoxResult
  'Same as VBA.MsgBox, but support unicode
  If IsMissing(Title) Then Title = Application.Name
  MsgBoxW = MessageBoxL(GetForegroundWindow, StrPtr(Prompt), StrPtr(Title), Buttons)
End Function

Function ReplaceUnicode(ByVal S As String) As String
  'Replace unicode terms like "U+1234" in S to unicode chars
  Dim i As Long, j As Long, k As Long
  Dim a As Integer
  Dim Digit As String
  'Find the start of an unicode term
  i = InStr(1, S, "U+", vbTextCompare)
  On Error GoTo Errorhandler
  Do While i > 0
    'Get the next up to 4 hex digits
    For j = i + 2 To i + 5
      a = AscW(Mid$(S, j, 1))
      If a < 48 Then
        Exit For '<0
      ElseIf (a > 57) And (a < 65) Then
        Exit For '>9 <A
      ElseIf (a > 70) And (a < 97) Then
        Exit For '>F <a
      ElseIf a > 102 Then
        Exit For '>f
      End If
    Next
    'Store the unicode char
    Mid$(S, i, 1) = ChrW$(CLng("&H" & Mid$(S, i + 2, j - i - 2)))
    'Overwrite the rest of the term with zeros
    For j = i + 1 To j - 1
      Mid$(S, j, 1) = vbNullChar
    Next
Skip:
    'Find the next term
    i = i + 1
    i = InStr(i, S, "U+", vbTextCompare)
  Loop
  'Strip zeros
  ReplaceUnicode = Replace(S, vbNullChar, "")
  Exit Function
Errorhandler:
  'Skip this part
  Resume Skip
End Function

andreas killer say:
I'm not that familiar with API calls and I don't understand the significance of the "L" in MessageBoxL, why you use an "Alias" "MessageBoxW," or the significance of the "W" in the alias?
Don't worry, I'm sure other ones are also interested in that.

Have a look at the MSDN side for MessageBox:
http://msdn.microsoft.com/en-us/library/windows/desktop/ms645505%28v=vs.85%29.aspx

At the top you can see the C++ declaration, when you convert that to VB you get:

Declare Function MessageBox Lib "user32" ( _
    ByVal hwnd As Long, ByVal lpText As String, ByVal lpCaption As String, _
    ByVal wType As Long) As Long

But when you try to call it:

Sub test()
  MessageBox 0, "prompt " & ChrW(&H2248), "caption", 0
End Sub

You get a RTE 453, means the function is not in the DLL file.

Go down at the MSDN side to the section "Requirements", you see:
  MessageBoxW (Unicode) and MessageBoxA (ANSI)

That means we can use MessageBoxA instead of MessageBox:

Declare Function MessageBoxA Lib "user32" ( _
    ByVal hwnd As Long, ByVal lpText As String, ByVal lpCaption As String, _
    ByVal wType As Long) As Long

But I already have written some code that use "MessageBox" as function name (in my sub test) and I don't like to rewrite it. So I can use an alias:

Declare Function MessageBox Lib "user32" Alias "MessageBoxA" ( _
    ByVal hwnd As Long, ByVal lpText As String, ByVal lpCaption As String, _
    ByVal wType As Long) As Long

That means i use "MessageBox" in VB but the name of the function in the DLL is "MessageBoxA".

Now we can run the sub, it works, but the prompt did not show the unicode char. Of course, we used the wrong function, let us change it:

Declare Function MessageBox Lib "user32" Alias "MessageBoxW" ( _
    ByVal hwnd As Long, ByVal lpText As String, ByVal lpCaption As String, _
    ByVal wType As Long) As Long

Run the sub again, and what it this? The prompt is messed up!

Now you have to know that a string is a pointer, means if you write S="123" in VB, you allocate some memory anywhere, copy the chars 123 into there and the variable S gets the address of the memory place.

That is the reason why I can use the trick and replace the STRING arguments with LONG arguments. But to let me (and other ones) know that I've done something special, I added the "L" to the VB name:

Declare Function MessageBoxL Lib "user32" Alias "MessageBoxW" ( _
    ByVal hwnd As Long, ByVal lpText As Long, ByVal lpCaption As Long, _
    ByVal wType As Long) As Long

Sub test()
  MessageBoxL 0, StrPtr("prompt " & ChrW(&H2248)), StrPtr("caption"), 0
End Sub
