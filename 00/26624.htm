Well the Word object model knows the concept sentences. So in theory you could let it check all sentences in a 2 loop fashion. Something along the following lines:

========================

Sub Test()
    Dim i, j, nrOfSentences As Integer
    Dim s As String
   
    nrOfSentences = ActiveDocument.Sentences.count
   
    For i = 1 To nrOfSentences - 1
        s = ActiveDocument.Sentences(i)
       
        For j = i + 1 To nrOfSentences
            If s = ActiveDocument.Sentences(j) Then
                MsgBox s, vbOKOnly, "Duplicate sentence"
            End If
        Next
    Next i

End Sub

===================================

You probably want to change the MsgBox line with whatever own logic you have.

But honestly, I wouldn't try the above code myself. Considering the inaccurate definition of what a sentence is, results might be unpredicatiable. For example, according to Word "John F. Kennedy was shot." are actually 2 sentences. But probably more important, the above function will be extremely slow. I would be surprised if it takes several hours, maybe even days, to check a large document.

ken sheridan say: When developing a database in Access the file is not saved in the same way that a Word or Excel document is saved.  Each object you create in the database is saved independently, so when you design a table, query, form, report etc you save each object as you go along.  When designing a complex object, e.g. a form it is a good idea to save the design frequently as you do so.  I use the save icon on the quick access toolbar, which I keep open below the ribbon.  That way, if something does go wrong, you have only lost whatever changes you've made to the object since the last save.

As regards the file as a whole you might want to consider saving this under a new name at intervals as you develop the database, e.g. MyDatabase.accdb might be saved as MyDatabase_01.accdb, MyDatabase_02.accdb and so on.  I normally do this in File Explorer by copying and pasting the file.  You can then easily revert to an earlier version if a particular line of development proves fruitless.

When it comes to saving data, this is mostly done automatically, e.g. once you move off a record in a form that record is automatically saved.  You might want to save all data periodically as dated backups, however, so that if you find that data has been lost or incorrectly amended, or has become corrupted (which can happen), you have earlier versions of your data to revert to.  To facilitate this you firstly need to split the database into separate front end and back end files, the latter containing the tables, the former all other objects along with links to the back end tables.  The built in database splitter wizard will do this for you.

Once the database has been split dated backups can be made of the back end from within the front end.  You might like to take a look at Backup.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates means of making both single backups, where the last backup is overwritten each time, or multiple dated backups where each backup file's name includes the date at which the backup was made.   You should of course also keep at least one backup of the front end in a safe location, and preferably two or more as a backup might itself become corrupted.

scottgem say:
If you save changes to an object while an Access session is open, then those changes are saved to the Access file. For example, you have a form, in Design mode you change the label for a control on that form. When you close the form it prompts you to save it. You do so, then close the file and re open it, that change should be on the form.

Similarly, when working with data, as soon as the current record loses focus, the record is committed to the data, So, if you change some data, close and reopen the app, the data should show changed.

However, if you close the file without saving the object or committing the data it would be lost.

So I'm not sure what you might have lost.

suzanne s. barnhill say:

In that case, you will also need to apply a style that includes a first-line indent. One way to do this is with three Replace operations as follows:

  1. Find what: ^p^t
    Replace with: ^p@@
  2. Find what: @@
    Replace with: Format: Style: Body Text First Indent (or indented style of your choice) 
  3. Find what: @@
    Replace with: <nothing>

To apply a style in this way, click More in the Replace dialog. With the insertion point in the empty "Replace with" box, click Format, then Style, then your chosen style; click OK. For the last Replace operation, leave the "Replace with" box empty and click No Formatting to remove the style information.
