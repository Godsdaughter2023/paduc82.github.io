ken sheridan say: You are approaching this from the wrong direction I'm afraid; your three 'managers' tables should not have any names in them at all, but numeric foreign keys, as personal names can legitimately be duplicated, and not infrequently are, even in small workgroups.

What you have here is an entity type Employees and three entity types Countries, Regions and Departments.  Between Employees and each of the other three entity types is a many-to-many relationship type, which in each case should be modelled by a table which resolves the many-to-many relationship type into two one-to-many relationship types.  So the four referenced entity types are modelled by tables like this:

Employees
....EmployeeID (PK)
....FirstName
....LastName
....etc

Countries
....CountryID (PK)
....Country
....etc

Regions
....RegionID (PK)
....Region
....etc

Departments
....DepartmentID (PK)
....Department
....etc

In each case the primary key column can be an autonumber for convenience.

To model the many-to-many relationship types:

CountryManagers
....CountryID (FK)
....EmployeeID (FK)
....possible other columns representing attributes of the manager's relationship with the country.

RegionManagers
....RegionID (FK)
....EmployeeID (FK)
....possible other columns representing attributes of the manager's relationship with the region.

DepartmentManagers
....DepartmentID (FK)
....EmployeeID (FK)
....possible other columns representing attributes of the manager's relationship with the department.

In these three tables the primary key is in each case a composite one of the two foreign key columns, which in this case are not autonumbers, but straightforward long integer number data types.

Data entry is achieved by a simple form/subform set-up. In this situation probably the most  suitable set-up would be to have firstly a Employees, Countries, Regions, and Departments form, the first as a simple standalone form.  The last three would each have a subform based on the relevant table from the last thee above which models the relationship type between the parent form's table and the Employees table.  In the subform the employees who are managers of the relevant entity type can then be selected from a combo box, inserting a row into the subform for each manager of the entity type in question.  You'll find an example of this type of many-to-many relationship type and how to represent it as a form/subform as ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

ken sheridan say: I've had a look at your relationships diagram now.  I'll limit my comments to the subject of this thread, i.e. the managerial relationships between people and countries etc.

If we take the many-to-many relationship type between people and Countries as an example, the people table contains only a few columns, People ID, First Name, Last Name and Position.  The Country Managers table, which is modelling the relationship type, contains many columns on the other hand.  In the case of First Name and Last Name, these are redundant as they repeat values already in People, which means that they are not only unnecessary in Country Managers, but the table is also at risk of update anomalies as there is nothing to stop different names being entered for the same person in the two tables, or even in different rows in Country managers as this will contain multiple rows per person.

The other columns below Last Name in this table, with the possible exception of Total People, which I'll return to below, also should not be in this table, but in the people table, because these represents attributes of the person, not of their managerial relationship with Countries.  These columns should, like First Name and Last name be in the Peoples table.

You also have columns COO and Region in the Country Managers table.  I don't know what COO represents, but I see you have a COOs table which is modelling a many-to-many relationship type between People and Regions, so I'm pretty certain that these columns should not be in Country managers.

You have a Country column in Country Managers.  Is this a text column with the country name?  If so it should not be, but a numeric CountryID column as a foreign key referencing the primary key of Countries.  I hope you have not used the dreaded 'lookup field wizard'  when adding this column to the table.  This feature is deprecated by most experienced Access developers.  For reasons why see:

http://www.mvps.org/access/lookupfields.htm

Most fundamentally, however, is the fact that there is no foreign key in Country Managers which is referencing the primary key of People.  You appear to have a relationship between Country Managers and people, but from your diagram I can't see on what columns.  There should be a People ID column in Country Managers, but not an autonumber and not uniquely indexed.

Returning to the Total People column in Country Managers, I cannot say whether this is a legitimate column in this table as I don't know what it represents.  With a table modelling a many-to-many relationship type, as this is, the basic principle is that apart from the keys, CountryID and People ID, all non-key columns must represent attributes of the relationship type, i.e. of the managerial relationship between the person and the country.  Whether the Total People column does represent such an attribute I can't say for certain, though its name suggests probably not.

Turning to the Countries table I notice that this has a Region column.  Again, as a foreign key this should be a numeric Region ID column referencing the primary key of Regions.  This assumes that a region is a group of countries, not a sub-section of a country; unfortunately the term is used in both senses.  Administratively the part of England where I live is the West Midlands Region,  but we also talk about the Middle East Region to refer to that loosely defined set of countries which feature in the news a lot these days.  Assuming that a region is used in the latter sense, it does raise another basic modelling issue.  You appear to have a many-to-many managerial relationship type between People and Regions.  Does this imply that a person who is the manager for a region has the managerial responsibility for the countries in that region?  If there were the case then there would in fact be no direct relationship type between People and Countries, but via regions, i.e.

People----<RegionManagers>----Regions----<Countries

I have not addressed the designs of other tables, as I'm concentrating on the managerial relationships which are the subject of the thread, but I suspect there will be issues with other tables; one thing I notice for instance is that First Name and Last Name columns appear in several other tables, which implies redundancy as I described above.

One final point.  I see you've included spaces in your table and column names.  While Access permits this, mistakenly in my view, avoid doing so,  Ales avoid special characters like the # (hash) symbol.  In place of spaces either use CamelCase, e.g  FirstName, or represent a space as an underscore, e.g. First_Name.  Don't use something like Transaction#, but something like TransactionNo or Transaction_No.

ronrosenfeld say:

It is pretty straightforward to do using VBA.

Using a Class Module, I created a User Defined Object consisting of a City name, and a collection of Zip codes for each city.  I then iterate through the source data in columns A:B, to populate each object, collecting them into a collection.

Then it is just a matter of going through the collection, and output in the format you want.

I did assume that your original data is in columns A:B, and your results will go into D:E.  The macro will also populate column D with a list of city names (the unique names from column A) so you don't need to do that manually.

To enter this Macro (Sub), <alt-F11> opens the Visual Basic Editor.
Ensure your project is highlighted in the Project Explorer window.
Then, from the top menu, select Insert/Class Module . 

With the class module (probable named Class1) highlighted, F4 opens the Properties window. Change the name to cCityZip

Then paste the Class Module  code below into that window.

Next, from the top menu, select Insert/Module and
paste the Regular Module code below into that new window that opens.

To use this Macro (Sub), <alt-F8> opens the macro dialog box. Select the macro by name, and <RUN>.

Class Module

=====================================

Private pCity As String
Private pZip As String
Private pcolZip As Collection

Public Property Get City() As String
    City = pCity
End Property
Public Property Let City(Value As String)
    pCity = Value
End Property

Public Property Get Zip() As String
    Zip = pZip
End Property
Public Property Let Zip(Value As String)
    pZip = Value
End Property

Public Property Get colZip() As Collection
    Set colZip = pcolZip
End Property

Public Function AddToZip(ZipCode As String)
    On Error Resume Next 'duplicate zip code for same city so ignore
    pcolZip.Add ZipCode, ZipCode
    On Error GoTo 0
End Function

Private Sub Class_Initialize()
    Set pcolZip = New Collection
End Sub

Regular Module

Sub Consolidate()
    Dim cCZ As cCityZip, colCZ As Collection
    Dim vSrc As Variant, vRes() As Variant
    Dim wsSrc As Worksheet, wsRes As Worksheet, rRes As Range
    Dim I As Long, J As Long, V As Variant
   
'Set sheets for source and results; and range for results
Set wsSrc = Worksheets("Sheet1")
Set wsRes = Worksheets("Sheet1")
    Set rRes = wsRes.Range("D1")
   
'Read source data into array
With wsSrc
    vSrc = Range(.Cells(1, 1), .Cells(.Rows.Count, 2).End(xlUp))
End With

'  unique list of zip codes

On Error Resume Next
For I = 2 To UBound(vSrc, 1)
    Set cCZ = New cCityZip
    With cCZ
        .City = vSrc(I, 1)
        .Zip = CStr(vSrc(I, 2))
        .AddToZip .Zip
        colCZ.Add cCZ, .City
        Select Case Err.Number
            Case 457  'Duplicate City, so combine the zipcode
                Err.Clear
                colCZ(.City).AddToZip .Zip
            Case Is <> 0  'unexpected error so stop to debug
                Debug.Print Err.Number, Err.Description
                Stop
        End Select
    End With
Next I
On Error GoTo 0
       
'Populate Results Array
ReDim vRes(0 To colCZ.Count, 1 To 2)
vRes(0, 1) = "City"
vRes(0, 2) = "Zip Codes"

    With colCZ(I)
        vRes(I, 1) = .City
            ReDim V(1 To .colZip.Count)
            For J = 1 To UBound(V)
                V(J) = .colZip(J)
            Next J
        vRes(I, 2) = Join(V, ", ")
    End With
Next I

Set rRes = rRes.Resize(UBound(vRes, 1) + 1, UBound(vRes, 2))
With rRes
    .EntireColumn.Clear
    .Value = vRes
    With .Rows(1)
        .Font.Bold = True
        .HorizontalAlignment = xlCenter
    End With
    .NumberFormat = "@"
    .EntireColumn.AutoFit
End With

End Sub
