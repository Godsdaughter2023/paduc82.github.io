tom van stiphout say:

You now understand that the Command function has the value given by the /cmd switch. How do we act upon it?

This is how I do it. I don't use a Startup form. Why? Because I may not even want to open the first form, for example if the BE could not be found, or user has no rights to open that form.
So we use the alternate (better) way of starting an Access application: via the AutoExec macro. This has one line: RunCode, and the name of a function. I call mine InitApplication.
This function resides in a standard module. It checks things out, initializes global variables, and then maybe, if all looks good, we may open the first form. This function is also the place where you check for the command line, and act upon it. I do it in a private sub called from InitApplication. In its simplest incarnation the code might be something like:
dim cmd as string
cmd = Command()
if cmd = "form1" then
  docmd.openform cmd
elseif cmd = "report2" then
  docmd.openreport cmd, dbViewPreview
elseif cmd<>"" then
  msgbox "Aarrcchh, there is no support for command line argument " & cmd
end if

tom-s say:

bobsmith12,

One way to do this is to use helper columns.

In E1 have this formula  =A1&B1&C1  and fill down as far as necessary.

Then from G1:H1 down list out all the pairs you want to search for: 0,0; 0,1; 0,2; ... 0,9; 1,1; 1,2; ... 1,9; 2,2; 2,3; ... etc etc

Then use column I to count the pair combinations, by placing this formula in I1 and filling down:

=COUNTIF(E$1:E$2000,"=*"&G1&H1&"*")+COUNTIF(E$1:E$2000,"=*"&H1&G1&"*")

 

Note that this is counting combinations, as requested, so there is for instance no distinction between 0,1 and 1,0

john mcghie say:

Philip is correct: your problem is lack of hyphenation.

Word can't actually "justify" text (I am one of the ones who frowns on "full justification, because it does make the text difficult to read). What Word does is vary the widths of the spaces between the words to place the end of the last word that will fit at the end of the line.

If you yet a large gap in a line, it means Word is unable to bring the word at the beginning of the next line up, and it is unable to split the word.

  1. Go to the Layout tab.
  2. In the Page Setup group, find the Hyphenation button.
  3. Click it, and choose "Automatic".

That will get you quite close.

Due to a bug in Word 2016, you can't set the hyphenation zone, so let's hope you like the result.

This is how it is supposed to work: https://support.office.com/en-us/article/Control-hyphenation-7d4d2a38-b0e2-4f04-873a-0a8d48ac3923

One more thing to check: In your styles, go to Modify>Format>Paragraph>Line and Page Breaks, and ensure that "Don't hyphenate" is turned off, otherwise that overrides your hyphenation setting for that style.

Of course, re-typing your document will not help: you would get the same result.

However, it is worth asking yourself whether anything "else" could be preventing Word from controlling the line breaks.  There is quite a variety of things that can interfere with Word's ability to lay out the document:

  • Empty paragraphs ("Blank lines").  There is no such thing as a blank line in Word: an empty paragraph still contains as many as 1,200 items of formatting, even if there is no text.  If Word attempts to honour that formatting, you will live in interesting times.
  • Hard page breaks.  Word will paginate the document for you automatically.  Intervene as little as possible.  If you are going to intervene, use the properties on the Paragraph>Line and Page Break tab to tell Word where you DON'T want a page break.  It will then do a great job of laying up the document for you.
  • Non-breaking spaces.  As the name implies, this character prevents Word from splitting a line at that point: if you use too many of them, the justification begins to look seriously weird.
  • Non-breaking hyphens: same effect, it prevents a line-break at that point.
  • Hard line-breaks (Shift + Return).  Nasty: if you put manual line breaks in, Word will attempt to justify but can only work with the lines that do NOT end with a hard line-break.

Word is NOT a Page Layout Program, it is a word-processor.  By design, a word-processor should handle the document layout automatically for you.  If you allow it to do that, you will get quite a nice result.  I often publish direct to press from Word: it won't produce a coffee-table showpiece -- but it does a commercially-acceptable job far quicker than a page-layout program.  However, if you intervene to prevent Word doing what it is designed to do, you can get quite a peculiar result.

Hope this helps

The reason for the delay in loading the page is the time required for the vURL Online service to analyze the website.  This would happen more quickly if it was simply analyzing the basic content, but things like requesting the PTR Record must complete a timeout failure in order to provide the "no PTR Record" results.

As for the original question as to whether a PTR Record is required or "it is a bona fide website", this would be aided if we understood your context.  If you chose this website as the result of a search the question might be less of a concern than if the original source of the URL where via a SPAM email message for example.

Regarding the missing PTR Record, this is unfortunately a common situation, since few websites typically take the time to properly configure these to resolve to the specific host name.  This is less a direct failure of website management than a simple matter of logistics in many cases.  The problem is that many websites today are hosted on servers where a single IP address resolves for several hostnames and in some cases these may actually be hosted across a number of redundant servers, so there is no simple one-to-one relationship by which the server may be identified.

The other issue is that though an individual ISP hosting service or server owner may be responsible for a block of IP addresses, they commonly aren't the true owner of the IP address block and so don't control the DNS server which should contain the PTR Record(s) for their specific sub-assigned IP block. For example, the analysis you provided indicates the following block and ASN as the responsible party.

208.73.200.0/21 NJIIX-AS-1 - NEW JERSEY INTERNATIONAL INTERNET EXCHANGE LLC,US

So in order to have their specific IP address identified by their hostname, the website owner would need to request this from their hosting service, who in many cases may not even own and manage the entire block and must request this of the ASN separately.

So due to these logistical issues, the most common configuration for PTR Records is something like a generic hostname containing portions of the ISP name and the specific IP address or in some cases such as this they don't even bother to create a record.  This clearly isn't "best practice", but it's unfortunately a common situation, especially for small websites or those hosted by others.

The larger organizations such as Google, Microsoft, etc. may sometimes include PTR Records for their more important public servers such as those providing email or secure web services (https), but even this isn't always done.  It's simply a function of the security concerns of the organization and the specific services involved as to whether their importance is seen as a great enough concern to perform this additional management.

In my own past life as an administrator for medium sized firms, I typically only requested the configuration of a PTR Record for the email and a couple of the most important web servers out of roughly a dozen IP addresses in use.  In many cases we actually didn't wish to identify some other private services such as a VPN, so leaving them generic or un-configured was preferred.

From all of this I think it's clear that this system isn't a good way to verify the validity of either a website or the company which owns it.  At best it's an indicator of good practices by the firm, but even that's not damning for the reasons I've mentioned above, so I typically only use these results for whatever additional information they may provide.  They are typically of no use to the everyday user since they are so difficult to view and understand fully.
