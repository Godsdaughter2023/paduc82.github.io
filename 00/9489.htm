Hi Mike,

You can do what you need to do with content controls. There are a few unintuitive steps you need to take to set it up.

A first consideration is that if you've mixed content controls and legacy form fields in the same document, you need to replace the form fields with content controls of similar function. When the two types are mixed, the required "filling in forms" protection interferes with content controls.

In the Restrict Editing pane, select the No Changes (Read Only) type of protection. Then select the all of the places (either content controls or plain text) that you want to allow users to edit, and check the Everyone box in the Exceptions section of the pane. You can add more exception locations later if needed.

To make an underlined area to your specifications, insert a Rich Text content control.

While the cursor is in the control, click the Properties button in the Controls group of the Developer tab. Check the box for "Use a style to format text typed into the empty control". Then click the New Style button. In the next dialog, enter a name for the style (e.g., Uline) and click the U button under formatting -- that will create a character style whose only property is underlining. Click "New documents based on this template" and click OK. For other underlined content controls, you can just select this style rather than defining a new one. Close the Properties dialog.

Next, click the Design Mode button. Delete the "Click here to enter text" placeholder, and in its place insert some underline characters to make it as wide as you want it to appear when there is nothing typed there. It will look like this:

Click the Design Mode button again to turn it off. The result looks like this while the content control is selected (the box disappears when not selected, and doesn't print):

Start enforcing protection, and save the form. You probably want to uncheck the box in the pane that says "Highlight the regions I can edit" to remove the ugly brackets and mustard-colored highlights, especially since the underlines show where to type.

jay freedman say:

To make sure this point is clear: Select the things that you want users to be able to edit, and check the Everyone box. That makes the selected things "exceptions to the restriction", that is, they'll be editable. Everything else will be restricted. (This kind of double-negative thinking is common among programmers, unfortunately.)

In general, the things you need to make editable are the content controls. You could -- but don't have to -- select pieces of regular text as well; for example, if there's a paragraph that you want the user to be able to delete, or a place where you want them to insert a photo.

You don't have to select all of the editable places at once. You could select one and check "Everyone", then another, and another, but that's tedious. It's easier to select one, then hold down the Ctrl key and use the mouse to select the rest of the ones that are currently visible, then check "Everyone". Scroll down to the next chunk of the document, select one, Ctrl+click to select more, then check "Everyone". Repeat until you get to the end of the document. As Stefan said, you only have to do this once per form.

brian tillman [outlook mvp 2007-2019] say:

Contact Groups (Distribution Lists for Outlook 2003) don't transfer well.  Instead, I'd recommend saving the Group as a TSV (tab-separated values) file.  Open the Group, then click FIle>Save As and choose "Text Only (*.txt)". Send this file to her.

Once she receives it, she should open her Contacts folder and choose New Contact Group.  When the form opens, she should click Add Members>From Outlook Contacts.  Now, she should open the text file you sent her and she should select all ot he name/address pairs in the file, right-click the selection, and choose Copy.  Now she should either click in the Members field and press Ctrl+V (to paste the list) or she should right-click in the field and choose Paste.  She shoudl then click OK, give the Group a name, then click Save and Close.

Of course, this doesn't add the individual members as contacts in her Contacts folder.  There are several ways to transfer those.  One is to include them as attachments to a Rich Text message.  Another is to put them into a separate PST and transfer that.  Another is to put them in a separate contacts folder, export the folder to a comma-separated values file (csv) and transfer that.

karl timmermans [outlook mvp 2012-2018] say:

Your question covers a lot of territory. What format are you importing from?

If it's from a PST file - no need to import. Better approach is to open the PST file that contains your contacts and copy them to the desired destination folder. Not withstanding that, if importing PST2PST - if the source PST file has an underlying error (or errors) - this could easily occur. Any import/export process cycles through the file on an item by item basis which could easily encounter things that you would not see in normal day to day activity if the item is question is never referenced.

If it's from a txt or xls file - you have a data related issue either in the field (column) names or the data itself. If it's a field name related issue (header row having the problem) - no contacts at all would be imported. If it's something related to the data itself - contacts would be imported up to the point that the invalid record was encountered (unless of course it coincidently also happened to be the first data record). If one or more items have been imported - check your file to find the last item that was imported - the one that follows is the one which has the problem.

In short - the error indicates a data related issue of some kind whatever the source but unfortunately the Outlook import/export wizard is not the most informative when it comes to explanations.
