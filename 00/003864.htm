scottgem say:

I would totally agree with Bill. The first flag was this:

I have 1 to 1 relationships with 12 tables split by subject but controlled by the model number.  

1 to 1 relationships are rare. Splitting tables by subject shouldn't be needed. This is called encoding data using object names. Data should be defined by the values of fields in a record, not by the fields themselves. 

Access is a relational database and should be used relationally. Relational database tables should be tall and thin, not wide. Your problems with fomrs, sounds like you are not using subforms, but trying to show all the fields from each table on the same form. Again, poor design. 

If you explain what data you are trying to capture we can help with a redesign. I would think you need at least 5 tables:

Product: Should include a ProductID as a Primary Key, and fields for price, cost, description and other fields specific to a product.

Specification: This table should have a SpecificationID as a PK and ProductID as a Foreign Key. It should then include a field for a specification type and a value for that type.

Packaging: Similar to Specification, but fields describing how the product is packaged

Prototypes and Development might be combinable into one table, but w would need to know more to tell.

Sorry, but your database probably needs a complete redesign to get it to work properly. We are willing to help you do that. But I would recommend that you read up on Normalization first.

logan05 say:

Hide a range of text based on a radio button?

Does anyone know how I can hide a range of text in a document based on a radio button?

I have a form with three sections to it: A, B, C.  When the user selects one of three radio buttons in the form, I have to hide the sections they did not choose. 

Would appreciate any way to do this or sugestions on an alternate way that might acomplish the same results.

Thank you,
Steve

 

jay freedman say:
You could modify Doug's code to delete sections instead of just hiding them.

The wrinkle here is that if you delete section 1, for example, then all the following sections are immediately renumbered (2 becomes 1, 3 becomes 2,...). So you can't just say "delete section 1 and section 3" because after deleting section 1, what was section 3 is now section 2.

With ActiveDocument

If Opt1.Value = True then

.Sections(2).Range.Delete

.Sections(2).Range.Delete  ' the original section 3

ElseIf Opt2.Value = True then

.Sections(1).Range.Delete

.Sections(2).Range.Delete  ' the original section 3

Else

.Sections(1).Range.Delete

.Sections(1).Range.Delete  ' the original section 2

End If

End WIth


The drawback to this is that if the user clicked the wrong button, there's no obvious way to correct the mistake -- the section is gone. (The Undo button might work -- as long as the document hasn't been closed and reopened.)

A different approach is to put each of the three sections into an AutoText entry or other building block, stored in the document's template (or use a template instead of a document, if you need to give the file to someone else). Then, when they make a choice, the code inserts the one needed section/entry. If they need to change their choice, the code can delete the previously inserted section and insert a different one. This can get a little complicated. :)

Combo Box not completely clearing - Microsoft Community

danumc say:

Combo Box not completely clearing

I have a cascading combo box on a form which displays the color, location, and manufacturer of an item. The three pieces of information are then displayed below the combo box as individual fields.

 

The combo box is unbound. The three fields are bound to the corresponding fields in the table.

 

The column count for the combo box is 4.

There are 4 column widths: 0; 1; 1; 1.

No column heads.

Row Source for the combo box is Table/Query.

SELECT [tbl_2016_main].ID, [tbl_2016_main].color, [tbl_2016_main].location, [tbl_2016_main].manufacturer FROM tbl_2016_main ORDER BY [Manufacturer], [color];

Bound column is 1.

Limit to List is set to Yes.

After Update an Event Procedure runs to clear the combo box of the displayed values.

 

Private Sub Combo197_AfterUpdate()

 

Color = Combo197.Column(1)

Location = Combo197.Column(2)

Manufacturer = Combo197.Column(3)

 

Me.Refresh

 

DoCmd.DoMenuItem acFormBar, acRecordsMenu, acSaveRecord, , acMenuVer70

 

End Sub

 

However, the combo box does not clear of all the values. When the user hits the dropdown arrow, all the records from the query, based on a table, display and populate in the appropriate fields on the form. However, when the user moves to the next record, the COLOR field still displays in the unbound combobox. As you progress to new forms, the COLOR entry of the previous form is displayed. You can still use the dropdown and see the choices but when you move to a new form, the color choice of the previous form is displayed.

 

I put “” in the Default Value property and that cleared the COLOR field of all old forms to be cleared but didn’t help any new records.

 

What am I missing?

 

Thanks.

scottgem say:

Use the On Current event  to set the combobox to Null.

Me.combo197=Null

A couple of points however. You have this working backwards. You should NOT have fields for Color, Location and Manufacturer in the table behind the form. The ID field from tbl_2016_main (the name of that table worries me as well) is all that should be in the table. you can then retrieve the other data using a join to tbl_2016_main. If you want to display the values on your form, set the Controlsource to =Combo197.Column(1) or 2 or 3.

Another point is don't accept the default names for controls. You should rename combo197 to cboProduct or similar.

You also refer to this as a cascading combo, but I don't see where there is any cascading. 

And one last point, you refer to fields on your form. Forms don't have fields, they have controls whihc may or may not be bound to fields in a table. This is a subtle but important distinction.
