gord dibben say:

See what you can do with this code which uses an Input Box to choose number of rows to unhide.

Sub Show_Hide_Rows()
    Rows("2:1000").EntireRow.Hidden = True
    mynum = Application.InputBox("How many rows to display (min 1 / max 1000)?")
    If mynum < 1 Or mynum > 1000 Then
        MsgBox "Please select a number between 1 and 1000", vbOKOnly, "Incorrect Entry"
    Else
        Range("A1").Select
        numRows = Selection.Rows.Count
        Selection.Resize(numRows + mynum).Select
        Selection.EntireRow.Hidden = False
    End If
    Range("A1").Select
End Sub

tom ogilvy say:

hard to write code without knowing where your data is.  However, I find the entry for FECHA and work from there.  Also, the use of merged cells make it even more challenging.   I believe I have accounted for that.

Sub showrows()
Dim rChoice As Range, rng As Range
Dim rng1 As Range, dItem As Double
Dim cell As Range
Set rng = Cells.Find(What:="FECHA", _
 After:=ActiveCell, _
 LookIn:=xlFormulas, _
 LookAt:=xlPart, _
 SearchOrder:=xlByRows, _
 SearchDirection:=xlNext, _
 MatchCase:=False)
 Rows.Hidden = False
 Set rChoice = Cells(rng.Row - 2, rng.Column + 1)
 dItem = CDbl(rChoice.Value)
 If Not rng Is Nothing Then
   Set rng = Cells(rng.Row + 1, rng.Column - 1)
   Set rng1 = Range(rng, rng.End(xlDown))
   For Each cell In rng1
     If cell.Value <= dItem Then
       cell.EntireRow.Hidden = False
     Else
       cell.EntireRow.Hidden = True
     End If
   Next
 End If
 
End Sub

ken sheridan say: You have two three options:

1.  Pass the values of the columns to the form as the OpenArgs property of the OpenForm method,  and assign the values to the DefaultValue properties of the relevant bound controls in the form in its Open event procedure.  You will find a method for passing multiple named arguments in Args.zip in my public OneDrive folder at the link given below.

2.  After opening the form, with further code in the button's Click event procedure assign the values to the relevant bound controls in the form.  BTW in Access event procedures and macros are completely different things, unlike in other Office applications where the term 'macro' is used for an event procedure.  

3.  In the Self_Peer_Eval_Form set the DefaultValue properties of the relevant bound controls to reference the matching controls in the calling form if it has such controls.

Note, however, that by storing the employee's name in the table to which the Self_Peer_Eval_Form form is bound you are introducing redundancy as the name is functionally determined by EmployeeID.  Consequently the table is not normalized to Third Normal Form (3NF) due to the transitive dependency of the name on the key.  The table is thus open to the risk of update anomalies.  In the case of the Position, Description, Location and Department columns, however, while at first sight it might be thought that these also introduce redundancy, this is probably not the case as, unlike the employee's name, these can change over time, so are functionally dependent on the key of the referencing table.
