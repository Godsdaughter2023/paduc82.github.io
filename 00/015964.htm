joeu2004 say: jhQC wrote:
if you calculate  =(AVERAGE(21,1;21,1;21,1;21,1;21,1;21,1)-21,1)*1e12  the result would be -0,00355 and not 0. Dropping one 21,1 from the formula corrects the result to 0.
Would there be any resolution for this kind of behavior.

This is due to a rounding issue, the potential for which is always present in arithmetic calculations due to the way that Excel represents numbers internally.

In general, if you want the result to be accurate to a specific degree of precision (e.g. 1 decimal place), you should explicitly ROUND the arithmetic expression.

In this case, the problem is that the internal representation of AVERAGE(21.1,21.1,21.1,21.1,21.1,21.1) is different from the constant 21.1 by about -3.55271E-15 (exactly 2^-48), the least-significant bit of the internal representation.

When you multiply by 1E12, of course that becomes about -3.5527E3.

Dropping one or more 21.1 from the AVERAGE remedies the problem only by coincidence.

PS:  This problem is not specific to AVERAGE.  For example, IF(10.1-10=0.1,TRUE) returns FALSE(!).  We must write IF(ROUND(10.1-10,1)=0.1,TRUE).

PPS:  You might notice that the following remedies the problem:
A1:  =AVERAGE(21.1,21.1,21.1,21.1,21.1,21.1)-21.1
A2:  =A1*1E12
or simply display A1 formatted as Scientific with 1 or more decimal places.  This, too, is only by coincidence.  In this case, Excel is arbitrarily "correcting" the arithmetic in A1 because the result of the last subtraction is "close to zero".  To demonstrate how dubious this "close to zero" algorithm is, note that the following does not correct the rounding problem:
=AVERAGE(21.1,21.1,21.1,21.1,21.1,21.1)-21.1-0
