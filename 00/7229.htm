When we select a single cell that contains text only, then drag the Fill Handle, the default behavior is to copy.

scottgem say:

First, don't confuse data storage with data presentation. The two are different concepts and should be viewed separately. Second, the table design I suggested follows rules for relational database design that have existed for forty years. There are good reasons for those rules. 

That being said, you could have the same form you currently have, but it would be unbound and you would have to use code to handle the I/O between the table and the form. 

Or you could do your form differently. First I would add a lookup table of AuditNames. I'm guessing here that it would be better to have descriptive audit names than generic ones.

tluAuditNames

AuditNameID (PK)

AuditName

This would mean a slight change to tblAuditDates:

tblAuditDates

AuditDateID (PK Autonumber)

ProjectID (FK)

AuditNameID (FK)

DueDate

So what you could do is automatically populate the Audit Dates table I suggested earlier from this table. Whenever you add a new Project have a button on your Project form with code like so:

Dim strSQL As String

strSQL = "INSERT INTO tblAuditDates AuditNameID, ProjectID " & _

               "SELECT AuditNameID, " & Me.ProjectID & " AS ProjectID " & _

               "FROM tluAuditNames ORDER BY AuditNameID;"

CurrentDB.EXecute strSQL, dbFailOnError

This code will add a record for each Audit. Finally, you create a Subform bound to this table and embed it on a mainform bound to your Project Table, the Subform would be in continuous form mode. So instead of having a row of Audits, you have a column of Audits with the DueDate column next to it and you just fill in the due dates.
