rob koch say:

Though it's quite possible that some of the warnings you received via the Java pop-ups were invalid, your description sounds more to me like otherwise valid notifications from various websites that the version of Java you then had installed was truly out of date.


It's very common for valid websites to verify this fact, since that test is built into the Java software these sites are attempting to invoke for their normal operation and in fact is intended to warn the user of an outdated version and to offer the valid information for where to get the Java update that is required.


The problem is that malicious websites are able to use this exact same software to confirm that an outdated version of the Java JRE software is present as well, then using this same information to redirect a client PC user to a malicious download in an attempt to infect the PC via the various known exploits in that out of date version.  Telling the difference is very difficult, since the malicious websites purposefully make themselves appear as "helpful" as possible in hopes that the user being attacked can be fooled (socially engineered) into installing whatever malicious software they have offered in it's place.


The problem you were actually having was likely due to a damaged older installation of Java that wasn't performing the normal update procedures properly, possibly due to an earlier attack on Java that either succeeded or simply caused the damage as a side effect.


What you also need to know is that even when the properly operating current version of Java JRE is installed, malicious portions of Java applets may be encountered even at otherwise "safe" sites that have become infected themselves.  But since the current version of Java JRE is installed, these otherwise outdated snippets of Java code will be retained in the Java cache of the PC as a normal part of the operation of Java.


However, since the installed version of Java is not susceptible to these outdated Java exploits, they will silently fail, but remain in the Java cache where they can have no effect unless that same website attempts to run them again at which point they would simply fail again.


It is these snippets or "packages" of code that your antimalware full scans are finding in the Java cache and detecting as potentially malicious.  However, since the version of Java you have installed isn't susceptible to these and no website is really currently trying to invoke them, MSE or any other antimalware detecting them is simply telling you that they are present.


In the most specific terms these detections might be considered a false positive, since whether these code snippets are present or not only really matters if the installed version of Java is outdated enough to be susceptible.  But since determining that would require that the antimalware also verify the currently installed version of Java and whether it is susceptible to that particular exploit, the more common response is simply to inform the user that these potentially malicious files were detected without any reference to whether this is really a problem.


So you are becoming confused by one of the many sets of technical anomalies that malicious and even otherwise normal code can create when detected "out of context" by an antimalware full scan.  Since the antimalware has no way of knowing all of the particular details of the configuration of a specific system, it must simply make the assumption that a user would want to know that these potentially malicious files are present when a full scan is run and that they will be able to understand such technical ramifications as the operation of Java and its cache as I've described above.


The problem here isn't the technology, it's the fact that these assumptions of technical knowledge by the PC user are often invalid and so confusing.

In Exchange Online, the first public folder mailbox you create will be the primary hierarchy mailbox, which contains the only writable copy of the hierarchy. Any additional public folder mailboxes you create will be secondary mailboxes, which contain a read-only copy of the hierarchy.

When the size of the primary hierarchy mailbox is close to 50GB, the Exchange Online server will auto-create a new primary hierarchy mailbox.

So, we suggest customers create one public folder mailbox and Office 365 will auto-create the new public folder mailbox while the size of the first one exceeds about 47.5 GB. phillip m jones say:

This situation is due to Apple's security system. They use a system called Sandboxing. Sandboxing means components are stored in specified safe locations. If you must use a folder that you create, it is my understanding you can do so but place that folder, in the (MacOS created) folder, called Documents.

I never have worked that way and had long set up my computer like a Filing Cabinet. I get the queasy feeling of putting all my "eggs in one basket." And plenty of MVP's think I am being, silly and, paranoid for thinking a single directory is going to blow up the works. But I've been doing things this way since I had any computers that used a system similar to Mac that will be close to 40 years. And doing so, I have yet to have any serious file problems.
