rob koch say:

These logs are related to the Windows Deployment Image Servicing and Management which is used to create installation images which are then later used to deploy Windows on other machines.

"Deployment Image Servicing and Management (DISM.exe) installs, uninstalls, configures, and updates the features and packages in offline Windows® images and offline Windows Preinstallation Environment (Windows PE) images."

http://technet.microsoft.com/en-us/library/dd744382(v=WS.10).aspx

Depending on when your particular PC went into operation, these logs may be the remnants of the initial process used to create the image that initially deployed your copy of Windows or they might be the logs of later executions of the DISM deployment tool, even if this was simply being started manually with no actual intention to deploy the image created.

In general though, these logs would have no relationship to an attack since this tool would be relatively useless for that purpose and far more dangerous things can be done that wouldn't require the amount of overhead or in fact the logging that occur with DISM.

In other words, quit digging around in protected directories, since unprotecting them simply increases the risk of you causing damage to your own operating system by tromping around in places that you shouldn't be looking.  Even people like myself who understand what these tools are for wouldn't mess with such things in such a clumsy way, since doing so simply makes us the "hacker" acting against the stability of our own PC.

When you press Tab at the beginning of a typed paragraph the AutoCorrect feature imposes a Left Indent, which is a Paragraph Formatting attribute. tim liu msft say:

Hi Daniel,

 

Firstly, let’s discuss on the Group site. The group site is related Office 365 Group. Each Office 365 Group will have a related site collection. We call it group site. The group site uses the team site template. We can create Office 365 Group in Outlook. Now, we can directly create group site in the SharePoint Home page. However, the group site won’t appear in the SharePoint admin center. The group site has different UI than other site collections created in SharePoint admin center. So, you don’t see the same options when clicking setting gear. To learn more, see the following articles:

Learn about Office 365 groups

New capabilities in SharePoint Online team sites including integration with Office 365 Groups

 

Now, let’s return to the initial issue. We should be able to edit subsite page in a group site. Please navigate to that problematic group site>setting gear>Site Content>New>Subsite>create a new subsite. Make sure it has the same permission of the parent site. Check if you can edit the page of the new subsite. For the problematic subsite, click Setting gear>Site Settings>Site Permissions>Check Permissions>enter your work/school account>Check Now>you should at least have Edit Permission for Editing the page. If you are group owner or have permission higher than Edit. We need to investigate the issue further.


(A subsite of group site, we can click Edit to edit the page)

 

Meanwhile, create a new group site>create a new subsite>check if you encounter the same issue in this new group site. We want to know if the issue is related to your account or a specific group site.

 

Moreover, are you the global admin or SharePoint admin in your tenant?
