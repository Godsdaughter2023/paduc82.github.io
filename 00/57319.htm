op-0 say:

M-Audio M-Track disconnects after 30-60 seconds of idle.

I recently got a Lenovo Thinkpad Yoga Business laptop with Windows 8.1. I previously owned the M-Track, and was using it with the HP Pavilion p6310y with Win7 loaded. However, when I tried to use it on the laptop, I was met with some unexpected results.

If the laptop is running on battery, the M-Track will just be disconnected for no apparent reason. This is a big problem, as the disconnection and reconnection of the device are heralded by a big pop, volume dependent on the setting on the Windows mixer. If the volume is set high, this pop can be quite painful. If the laptop is plugged in, this problem seems to disappear completely. The device will stay connected unconditionally.

I went over to the M-Audio support forum for answers, and a representative and I were able to compile some data:

1) The problem could be power related. The lack of the problem's appearance when the laptop is plugged in suggests that it is so. We weren't entirely sure how, though. Even when steps were taken to prevent Windows from unpowering the device, the device still disconnects. Turning off USB selective suspend did nothing. Preventing Windows from unpowering the USB slot it is connected to in the Device Manager only resulted in the occasional buzzing in the output following the pop.

2) The problem could be class-compliance related. On every other Windows version (XP, Vista, 7, 8), the M-Track has a dedicated driver. However, in Windows 8.1, the M-Track is class compliant. It is only on this version of Windows that this problem has surfaced (and from what the representative told me, it is very rare).

3) The problem is isolated. Other externally connected devices do not suffer this same problem. USB devices never disconnect, and Bluetooth devices only ever disconnect due to heavy wireless interference. In addition, these devices are not the source of the problem. The idle disconnect still occurs even when these devices have been removed.

4) ASIO works perfectly with the M-Track. There is no disconnection as a result of idle even if the device is left for hours without doing anything. This suggests that the issue may lie in the drivers.

5) I was fortunate enough to find a workaround in my experimentation. If I select "Listen to this device" on any available recording device, the problem goes away even if that recording device is muted. The representative told me that "Listen to this device" creates a constant stream of communication between the device and the Windows drivers. If this is the case, then it could be that the problem is the result of the driver not communicating with the M-Track when it has nothing to send to output, prompting Windows to disconnect it in an attempt to save power (which, from item 1, it should not be making).

I hope you could help me with this issue.
