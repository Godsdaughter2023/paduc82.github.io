Why does this happen and is there a way to fix it or avoid it?
brian tillman [outlook mvp 2007-2019] say:

Outlook always uses the format of the original message for the format of a reply or forward.  Thus, if someone sends you a Plain Text message, Outlook will reply using Plain Text and that format doesn't support fonts and styles.

It's a good idea, in my opinion, to honor the sender by replying in the format they used when they sent the message.  There are multiple reasons why the original sender may not want or cannot handle other formats.  That said, if you wish to change the format, click "More Move Actions" in the Move chunk of the message reply form and choose Edit Message.  You will then see the Format Text tab and can add what formatting, fonts, and styles you wish.

ken sheridan say: Generally the form's BeforeInsert event procedure is the best place for this sort of thing as it does not initiate a new record until the user begins to enter data, so they can abort the new record simply by closing the form or navigating back to an existing record.  If, however, you have a cogent reason to see  the SeqNo value as soon as you move to a new record then I'd do it like this:

Change the ADD button's code like this so it simply moves to a new record in the current form:

    DoCmd.GoToRecord acForm, Me.Name, acNewRec

In the form's Current event procedure put:

     If Me.NewRecord Then
        ' generate autonumber button by forcing form to a Dirty state
        Me.ProjID = 0
        ' assign new value to ProjID control
        Me.ProjID = Me.Seqno + 100
        Me.PALNUMBER.SetFocus
        ' This is for the popup forms, you must first load the is loaded
        ' function from the NWIND database.  The function is located in
        ' the Utility Functions Module
       If IsLoaded("PRJSTATE") Then
           Forms![PRJSTATE].Requery
       End If
       If IsLoaded("PRJSTAFF") Then
           Forms![PRJSTAFF].Requery
       End If
       If IsLoaded("PROJTASK") Then
           Forms![PROJTASK].Requery
       End If
        If IsLoaded("Project Type") Then
            Forms![Project Type].Requery
        End If
    End If

More fundamentally, however, this is a flawed methodology as the original developer was presumably under the impression that an autonumber would reliably generate a sequential set of numbers.  This is not the case; an autonumber is intended to guarantee distinct value and no more than that.  Consequently the ProjID values cannot be guaranteed to be sequential.  So far you might have been lucky, but your luck may well run out some day.  To generate sequential numbers reliably you should not use an autonumber, and should generate the values in code.  Scott gave you a link to his blog earlier.  You'll also find examples in CustomNumber.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file covers a number of different scenarios; the Sequential Numbering option would be appropriate in your case.  This includes error handling in the event of two or more users inserting a new row simultaneously and generating a conflict.  It also includes a means of 'seeding' the next number to be generated.
