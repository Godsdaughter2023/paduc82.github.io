dirk goldgar say:

I don't think you can do this with the .Seek method, at least not without significant modification, because Null never equals Null.  In your search, as I understand it, you want Null to equal Null.  To use the .Seek method, you'd have to use a ">=" comparision operator in the call to .Seek, and then even if that results in .NoMatch = False, check the located record to see if it's really a match according to your definition of "equals".

If I were in your position, I think I would use a dynaset-type recordset on the Inventory table, in order to use the .FindFirst method instead of .Seek, and dynamically build the search criteria like this:

'------ start of example code ------

Dim strCriteria As String

'...

strCriteria = "ClientID = " & strClientID & " AND ProductCode = " & strProductCode & " AND Item1 "

If IsNull(rstInvoiceDetail!Item1) Then

     strCriteria = strCriteria & " Is Null"

Else

    strCriteria = strCriteria & " = " & rstInvoiceDetail!Item1

End If

strCriteria = strCriteria & " AND Item2 "

If IsNull(rstInvoiceDetail!Item2) Then

     strCriteria = strCriteria & " Is Null"

Else

    strCriteria = strCriteria & " = " & rstInvoiceDetail!Item2

End If

rstInventory.FindFirst strCriteria

If rstInventory.NoMatch Then

    ' ... and so on ...

'------ end of example code ------

Note that in building the criteria string, I have assumed that all values are numeric, so I didn't insert quotes around any of the values being inserted into the string.  For any field that is not numeric, you would have to insert quotes around the literal values being built into the string; e.g.,

    strCriteria = strCriteria & " = """ & rstInvoiceDetail!Item2 & """"

I'm not sure whether it would be more efficient to open rstInventory each time on a query that includes the constructed criteria, and check for .EOF instead of .NoMatch, rather than using an unfiltered recordset and using .FindFirst as described above.  You can benchmark that, if you want.
