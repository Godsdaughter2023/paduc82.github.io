ken sheridan say: As Scott has said, in most situations only the foreign key IMEI value should be stored in the referencing table.  To store the other values in columns in Table 2 would introduce redundancy and the table would not be normalized to Third Normal Form, leaving it at risk of update anomalies.

For data entry use a combo box whose RowSource is an SQL statement which returns not only the IMEI value from Table1, but also the other three columns.  You can then reference the Column property of the combo box in unbound text boxes in the form to show the other values.

For an example take a look at DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In the section on Entering data via a form/subforms' in this little demo file there is an Orders form in which a customer is selected in a combo box, whose RowSource property is:

SELECT Contacts.ContactID, Contacts.Address, Cities.City, Regions.Region,
Countries.Country, [LastName] & ", " & [FirstName] AS Customer
FROM Countries INNER JOIN (Regions INNER JOIN (Cities INNER JOIN Contacts
ON Cities.CityID = Contacts.CityID) ON Regions.RegionID = Cities.RegionID)
ON Countries.CountryID = Regions.CountryID
ORDER BY Contacts.LastName, Contacts.FirstName;

The address data for the selected customer is then shown in unbound text boxes whose ControlSource properties are:

=[cboCustomer].[Column](1)
=[cboCustomer].[Column](2)
=[cboCustomer].[Column](3)
=[cboCustomer].[Column](4)

The Column property is zero-based, so Column(1) references the second column returned by the above SQL statement, Column(2) returns the third column and so on.

The only time where it is valid to duplicate columns from a referenced table in a referencing table is where the values in the referenced table might be changed, but those in the referenced table should remain fixed as the values when the row was inserted into the referencing table, or vice versa.  The order details subform in the above form has an example of this, where both the Items table and OrderDetails table have UnitPrice columns.  Consequently, the prices in existing orders will not change if the price in the Items table changes subsequently, as it inevitably will.  In this scenario the value is assigned to the control in the AfterUpdate event procedure of the item combo box:

Private Sub ItemID_AfterUpdate()

    ' assign current unit price from Items table
    ' to UnitPrice column in OrderDetails table
    Me.UnitPrice = Me.ItemID.Column(1)
    
End Sub

scottgem say:
Yes a  subform is the answer. You can use a multi-table query as the Recordsource of your for but often that results in an uneditable recordset.

brians_704 say: A sub-form seems like overkill in many situations. I do not remember having this issue in FoxPro or FileMaker - though it was a few years ago.

brians_704 say: This doesn't seem possible to me. With normalized data you often need to have a single form that enters data into several tables. Why does a list box show all the data when you have keys set up to set relationships?

scottgem say:

I don't know where you get your ideas, but subforms ARE the way to go. While you may be able to use a multi-table query, as I said, oftentimes such queries are uneditable due to the nature of SQL. 

I'm not that familiar with Filemaker, but in FoxPro, the forms interface was very different. Essentially Foxpro forms were unbound and the developer had to write code to mode data from the form to the tables and vice versa. So if you want to use unbound forms be my guest. 

One of the advantages of Access is that you can do a great deal without resorting to writing code. A continuous form subform gives you an easy way to display and enter data in a 1:many relationship, which is the meat and potatoes of relational databases.
