[BUG] Windows Update hogging all Internet bandwidth for itself - Microsoft Community

techie007l say:

[BUG] Windows Update hogging all Internet bandwidth for itself by running 40-100 downloads simultaneously!

    This thread was originally created during the Insider Preview phase, back when this unacceptable behavior was just a random bug.  The random bug has since been fixed, but this unacceptable behavior is now the new norm for Windows Update, bringing many home and office networks to their knees all over the world during updates.  This design flaw affects all three production releases of Windows 10:

  • Windows 10 TH1 (August 2015, RTM)
  • Windows 10 TH2 (November 2015)
  • Windows 10 RS1 (August 2016, Anniversary Update)

SYMPTOMS:

    When some Windows 10 PC is downloading updates via Windows Update (rollup security updates, driver updates, new builds of Windows), the Internet becomes very slow for all devices on the network.  We're talking about extremely slow, where a simple page like "www.google.com" takes several seconds to load, videos can't play without continuously buffering, and some networks even become practically unusable for anything else during the duration of the Windows Update download.

    If you are noticing these symptoms, please post a note about it in this thread.  Hopefully if we raise enough awareness of this issue, Microsoft will eventually fix it.  Remember that it is the squeaky hinge that gets the oil.  If you want to investigate Windows Update's behavior for yourself, the tools of the trade are Sysinternals TCPView, Nirsoft SmartSniff, and Process Hacker.

CAUSE:

    In Windows 10, Microsoft changed the way updates are downloaded.  In previous versions of Windows, updates were downloaded by BITS.  In Windows 10, updates are now downloaded by the Delivery Optimization service.  This service is supposed to reduce network and ISP loads by serving updates via P2P between computers.  Whether or not this feature works and is as effective as intended, I haven't yet fully assessed.

    However, let's forget about P2P for now.  The Delivery Optimization service has a couple serious design flaws in just its downloading feature alone!  When downloading updates, it splits them up into multiple slices.  If you've been in computing for many years, you may be familiar with this technique because it was commonly used back in the day by many download managers to "speed up" downloads.  How it works is that by dividing the download into multiple simultaneous streams (slices), the download's overall priority is boosted on all the networks between your PC and the remote webserver.  The most commonly used number of slices was four, which should boost the download's priority by approximately four times.  Here is the issue with the Delivery Optimization service: It is splitting these downloads into a whopping 40 slices for a 40x priority boost!  That means when another PC is trying to load another webpage, it gets one timeslot for every forty timeslots the Windows Update download gets.  Furthermore, on networks that implement QoS, many try to prioritize websurfing over file transfers.  How this is accomplished is by watching how much data is transferred over a connection.  The Delivery Optimization service not only bypasses this by using such an outrageously high number of slices (thus keeping the transfer amount per connection down), it regularly cycles these connections, closing them and opening new ones (thus resetting the data transfer counters to zero), which means that this already ridiculously high priority traffic also gets a free pass at high priority through QoS too!

    The other flaw I have seen in the Delivery Optimization service is that oftentimes, many of the slices will be redundant.  In other words, it is downloading the same parts of the same file multiple times.  I was alerted to this while waiting 25 minutes for a 200 MB driver to download on a 8 Mb/s connection.  It should have only taken 4 minutes to download.  The download speed was maxed out during most of the download, so there was no speed issue.  That's when I opened SmartSniff and started inspecting the HTTP request headers.  Low and behold, up to 10 of the parallel slices were for the exact same ranges of the same file, from the same server!  Downright outrageous!  So not only is Windows Update hogging all the Internet for itself, it may be prolonging the download time by up to 10x by accidentally downloading parts of the files that it is already downloading, over and over again.

SUSCEPTIBILITY:

    All networks are susceptible, but it may be under the threshold of annoyance on networks with Internet 40 Mb/s or faster if only hit by a single PC updating at a time.  This is for two reasons: At 40 Mb/s, the download bandwidth for one alternate stream should be 1 Mb/s, which while slow, is steady progress (some videos will still buffer however).  The other reason is that with a 40 Mb/s connection, most updates will be done downloading in a couple seconds, before the network impact is noticed.  Most people would chalk the delay up "to the Internet" (I am often appalled at what terrible performance people regularly put up with!).  And if the Delivery Optimization service is doing its job with P2P updates on your network, only a single PC should need to download the updates from Microsoft anyway.

----------------------------------------------------------------------------------------------------

    The last couple times I've upgraded to the latest Windows 10 Preview build, I noticed that it really seemed to bog my web browsing speeds down.  Which is surprising since I've got a router with powerful firmware configured with some pretty good QoS rules specifically to keep things running smoothly on my modest Internet connection with many users.  I never looked into why it was bogging down since I thought it was just my ISP lagging out on me (another issue).

    But when I upgraded from build 10130 to build 10158, I happened to check my bandwidth usage during the download, and was shocked to see my router downloading between 90-120% of my rated Internet speed—very unusual for my ISP (I usually get 40-60% rated speed on a single download).  So I thought my ISP had fixed something.  Next day, downloads were slow as usual all day.  Then build 10159 came out, and it proceeded to download beyond maximum speed again.

    During this download, my web browsing speeds were again very slow, even though the download was going fast.  As I was investigating this interesting phenomenon, I fired up Sysinternals TCPView.  That's when I noticed that Windows Update had approximately 100 TCP connections open, and was simultaneously downloading the new build with all of them at the same time.  Remember the download managers in the past that would let you split a download into several slices that downloaded at the same time?  Well, Windows Update was running a 100-slice download from Akamai Technologies (an Internet caching company).  Each second, the all numbers in the column on the right (received bytes) would increment higher and higher:

    After a couple hours, connections started closing.  But the Windows Update progress bar remained at 0% the whole time.  I eventually lost patience waiting, and restarted the Windows Update and BITS services.  The progress jumped right to 86% and the download resumed.  It reopened its 100-slices, but most of them quickly closed, and only a dozen ran while the download finished.

    So what's up with the 100-slice download?  That is  r a p e  of my Internet connection, not to mention the HTTP server.  This would not be tolerated on most other HTTP servers, but I guess Akamai is permitting it on their cache server.  In the past, I always had the opposite problem: New builds wouldn't download for long stretches of time, or would download very slow (less than a couple Mb/s) even though the connection was solid and fast.  It seemed the download was running at a very low priority and was pausing/throttling it whenever I used the laptop.  Now we have the opposite end of the spectrum: A 100-slice download!

  • What's up with the 100-slice download?!  It sure ruins any fair Internet download prioritization by creating a hundred bandwidth slots for itself.
  • Why are we using an HTTP connection for this critical download instead of HTTPS?  Someone could rather easily slip something malicious into the download.
  • Why are none of my machines doing peer-to-peer updates yet?  Each one downloads the latest build directly from the Internet each time.  I have never seen a single peer-to-peer update, and these machines are on the same network.  Even with file sharing enabled and occasionally used (which is still really buggy, by the way).
