john mcghie say:
In reply to deleted message

Fixy: Your answer is absolute rubbish.  Seriously: If you don't know the answer, shut the ***k up!

Clara:  bullets (and numbers) in Word are driven by the concept of "lists".  Each paragraph with a bullet on it is a member of a list.  There can be several lists in a document: the one applied to a particular paragraph is determined by the name of the style that formats that paragraph.

Your problem is caused because all of your paragraphs have the same style (probably "List paragraph".  So when you change the formatting of the bullet on one paragraph, it will automatically change the whole document.

To produce multiple different bullet formats in the document, use the built-in List Bullet styles designed for this purpose.  There are actually nine of them, List Bullet through to List Bullet 9, although in most documents only the first four or five are visible unless you use the others.

  1. Begin by applying the List Bullet style to a bunch of paragraphs (use Format>Style or click the Styles Pane button).  At the bottom of the list, change "List" to "All Styles" so Word will show you styles that are not in the document.
  2. In the Styles pane, drop down the arrow to the right of the List Bullet style name and choose Modify. Turn on "Automatically Update" and OK.
  3. Now use the controls you are familiar with to format the first level of bullets and their indents the way you want them. As you do this, all the paragraphs in your list will adopt your formatting.
  4. Now go back to Modify Style and turn "Automatically Update" off, so any further formatting you do will not wipe out your formatting in other paragraphs.

Now:  Select a bunch or paragraphs and apply the List Bullet 2 style.  Ignore the formatting: you are just about to set it...

Use steps 2 to 4 above to set the formatting for your second level of bullets.  Don't forget to turn "Automatically Update" off when you have them to your liking.

It is best not to use more than four levels of bullets in a document: the reader will quickly lose track of your structure and become very confused.  However, if you need to, you have nine levels available in the "List Bullet" series of styles, nine more available in the "List" series, and nine more available in the "List Continue" series.  If you need more than that, you can create your own styles :-)

So:  The answer to your question is "The bullet formatting is controlled by the style: to have different bullet formatting you must apply a different style."  About which Fixy should have known, because he is employed by Microsoft, but obviously hasn't a clue...

Hope this helps

wayne-i-m say:

You need to create a few items in your database.

 

A table to hold the booking In/Out information such as the person who has booked tyhe items In/Out. The time and date that the item was booked In/Out.  And maybe other information such as who has booked it  In/Out, etc
You will need to relate this table to you GaugeInfo table.  This is the simplest method but I would tend to go for a many/many relationship so you may want to also relate it to your personnel table - this way "many" people can book In/Out "many" patterns on "many" occasions.

A query - although this is just my personal choice.  As the calculation will be very simple you can just as simply perform them in a Form - but I tend to try where possible to get querys to calculate

 

A form for a user to input the details of the "In/Out".

The basics are that you should treat the Booking In/Out of a pattern as an “event” and it is this “Event” details that you will store in your new table.

 

I will assume that you already have 2 tables.
One called GaugeInfo and one called Personnel

 

1st Create a new table with these fields
EventID  = Autonumber format
EventInOut =  Yes/No format
EventTimeStamp = DateTime format (set this to General Date format)
EventNotes = Memo format

GaugeInfoID = Number format - linked to other table in the relationship window
PersonnelID = Number format - linked to other table in the relationship window

 

Of course you can create more fields subject to your needs – which I can’t see but you can ? But (very import) only create fields that will store information regarding that “specific” event

 

Next create a query based on your new table.

Double click each field to bring it into the query design.
You can add calculations later -


Next create a form based on this new query set it to Single Form format

Add all the fields into the form

Now the next bit comes down to personal choice and everyone on this forum will have different ideas about the UI (user interface) – that is “what should the form look like and what should it do”.


This bit is up to you. If it was me I would add a Toggle Button with just IN or OUT on each button. On change of this toggle button I may would set the value of EventInOut to either yes or no (maybe an idea to set the EventInOut control to hidden). Something like this


Private Sub FrameNameOfToggleHere_AfterUpdate()

Select Case Me! NameOfToggleHere

  Case 1

     Me. EventInOut = -1

  Case 2

     Me. EventInOut = 0

End Select

End Sub

 

Oh I would use -1 to denote a pattern going out and 0 to denote a pattern coming In. So now you know something is going out.
You can add to your toggle button to add the time date the pattern went either in or out like this

Private Sub FrameNameOfToggleHere_AfterUpdate()

     Me.EventTimeStamp = Now ()

Select Case Me! NameOfToggleHere

  Case 1

     Me. EventInOut = -1
        Case 2

     Me. EventInOut = 0

End Select

End Sub

 

So now you know if a pattern has gone in or out and when this happened.

Of course before you can do this you need to know which pattern you’re talking about and (again each person on this forum will have different ideas about this) I would just add a combo box based on your GaugeInfo.
On this combo set the bound field to the 1st column (ID) and the widths to allow the user to see the name of the pattern.
AfterUpdate of the combo add the ID to the GaugeInfoID control on you form.

 

Do the same type of thing with the personnel control – althougn if you only have a few employees you may want to just use another toggle button – there are “many” ways to do this ie, set the form user on open of the form (that’s a little more complex),etc.

 

So now you know

What has gone out/in
When it happened
Who did it.

 

Of course this is not a complete run-through but I hope it sets you in the right direction.

 

Remember that the you as the developer need to decide on really only three things.
What data you need to store
How is the data to be input into the table (by form, import, etc)
What user controls you need to add and code to allow the data to be input.

 

There are lots of other things to think about – eg. Do you allow a user to alter a record once it has been created – do you use the show the status of a pattern (taken from the EventInOut data) OnCurrent of the form – do you display elapsed time since Out – do you give warning messages when the user is entering information, etc, etc,etc

 

 

Good luck

I would like to start by discussing the recent clearing of your phone.  It sounds as if one of two things has happened.  Either the phone was factory reset, this would clear the device.  In this case any contacts and calendar entries associated with the Microsoft account, Facebook account, etc.., would come back to the phone when you add the accounts back to the phone.  The other possibility is that this was done remotely as a security procaution.  If you have an Exchange account on your phone this is something that can be initiated by an Exchange administrator or by entering the wrong PIN too many times.

 

Regarding the phones security, please review these articles, http://www.microsoft.com/en-us/download/details.aspx?id=8842.  Some of the highlights include:

  • Sanboxing of applications.  Each application is running in it's own chamber and only has access to the basic permissions needed to run the application.  This prevents information stroed to the phone by other application from being accessed.
  • Use of SSL encrypts data in transit.  As much of the Windows phone 7 is centered around the cloud much of the information accessed is not stored on the phone.  This allows the security of the server that stores that information to be the primary line of defense.  The biggest concern at this point is the transit of the information. 
  • Exchange ActiveSync management.  Multiple levels of control over EAS are incorporated on the phone.  This allows IT departments to manage policies that allow for secure use of Exchange on the Windows phone 7.

Please let us know if you have any additional questions or concerns we can assist you with.
