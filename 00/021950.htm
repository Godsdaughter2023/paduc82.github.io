greginmich say:

There’s no need to remove anything. When a threat is quarantined you could safely leave it in the Quarantine indefinitely, but it will be automatically removed after 90 days; or after the number of days specified with the -QuarantinePurgeItemsAfterDelay parameter.

 -QuarantinePurgeItemsAfterDelay

Specifies the number of days to keep items in the Quarantine folder. If you specify a value of zero or do not specify a value for this parameter, items stay in the Quarantine folder indefinitely.

https://docs.microsoft.com/en-us/powershell/module/defender/set-mppreference?view=win10-ps

The default value is actually set to 90 days, but you can set your own preference with PowerShell.

Right-click on the Start button and select Windows PowerShell (Admin); and then copy, paste, and enter this command (for example):

Set-MpPreference -ScanPurgeItemsAfterDelay 30

What generally confuses people here is the Recommended action: Remove threat now line in the Full History details report. That recommendation doesn’t apply to the threat once it’s been quarantined – it’s actually just the recommended action for dealing with the threat at the time of detection. And of course when we’re looking at the details, the threat has already been quarantined, or removed, or whatever – so this recommendation is always a “day late and a dollar short”. It really shouldn’t appear in the Full History report at all, as I pointed out many years ago. All it does is confuse people. For example, in my screenshot above, the recommended action is to Remove threat now, but the second line in the report just told us that the threat has already been removed: Status: Removed; so it’s a little too late for that guys.

Another point of confusion is that this threat is normally reported as “Quarantined”, and that report is erroneous – as we can see from the “not found” error reported for the Threat Status Error Code in the Get-MpThreatDetection output. -2142207965 translates to 0x80508023, and the description for this error is: The program could not find the malware and other potentially unwanted software on this device. But that’s probably because this threat was blocked and never downloaded to your PC, and not because it’s lurking somewhere – as you’ll see if you just run a Full Scan. 

GreginMich

For this I'd suggest that you simulate a split form with a standard form/subform.  You'll find an example in FindRecord.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

Rather than filtering a single source form object.  Create two separate forms for use as the subform, basing each on a query which restricts the rows to those for the relevant programme, using only the relevant columns in each case.  The buttons in the parent form would then set the SourceObject property of the subform control to the relevant form in each case.

To synchronise the parent form and subform so that a split form is simulated you'd repeat the code from my single subform's module in the modules of each of the source forms in your case.  Tip: if you start by creating a subform with all columns, and include the synchronization code in its module, you can copy and paste this as a second form.  Then you'd just need to change the query's criteria and remove the unnecessary columns as appropriate in each.  This would avoid your having to design the form from scratch and write the synchronization code twice.

While, in my demo, the subform is in continuous forms view you could easily set its DefaultView property to Datasheet if preferred.  You'd either put code in the DblClick event procedures of the relevant controls, or set their On Dbl Click event property to the name of macro.

PS: An alternative approach would be to put each subform on a separate page of a tab control, removing the need for the buttons in the parent form.

Try this code. . . .not tested beyond Excel 2007 version.

Paste into Thisworkbook module.

On any sheet select cells to see results in Statusbar

Private Sub Workbook_SheetSelectionChange(ByVal Sh As Object, _
                        ByVal Target As Range)
Dim s As String
Dim wfn As WorksheetFunction
    Set wfn = Application.WorksheetFunction
    On Error GoTo errH:
    If wfn.Count(Target) < 2 Then
        s = ""
    Else
        s = "Sum=" & wfn.Sum(Target) & "  " & _
            "Avg=" & wfn.Average(Target) & "  " & _
            "Count=" & wfn.Count(Target)
    End If
errH:
    Application.StatusBar = s
End Sub
