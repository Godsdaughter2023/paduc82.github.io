    Hi Matt,

    Please try to do the following checks and see if this is helping:

    1. Please check if the Windows 10 computer is accepting RDP requests from other network computers. You need to ensure that you are accepting requests from all devices, not just from the ones that know about Network Level Authentication.

    For this, please right-click on This PC, click on Properties. From the System window that is opening, please press on the Remote Settings link, from the left part of the page.

    On the System Properties window, Remote tab, select the Allow remote connections to this computer (you mentioned that this is already selected), but also uncheck the Allow connections only from computers running Remote Desktop with Network Level Authentication (recommended) check-box.

    2. Please ensure that you are connecting to the correct PC name (or IP address) the Windows 10 computer has at this point.

    a. For this please connect to the Windows 10 computer and:

    • right-click on This PC and click on Properties. From the System window that is opening, please check the value listed on the Computer name field.

    or

    • right-click on the Start button and press Run. On the Run window please type cmd and press OK. In the new Command prompt window that is opening, please type ipconfig and press Enter. Please check what IP address is assigned to the network adapter that you know it is connected (Wired or Wireless).

    b. On the computer from where you want to connect to the Windows 10 machine, please:

    • press the Start button and type mstsc and open the Remote Desktop Connection application. In the Computer name field, please type the Computer name (or IP address) of the Windows 10 machine (that you get at point 2.a.) and press Connect.

    or

    • right-click on the .rdp file (that you mentioned you have on the desktop of the computer from which you wanted to connect to Windows 10 machine) and press Edit. Please ensure that in the Computer name field is typed the Computer name (or IP Address) of the Windows 10 machine (that you get at point 2.a.). Ensure that the User name field is empty or you have a correct user typed in and press Connect.

    When you'll be asked to type the credentials, please type credentials for an user that is member of Administrators (or at least Remote Desktop Users) group on the Windows 10 machine.

    Cristian.

    It will no doubt be hard for users to believe, but the lag issue remains unresolved in Windows 10 build 1607 (anniversary edition) - one full year after it was reported in several forums.  MS level II support was made aware of it.


    This causes macros to hang and keyboard-actuated shortcut keys to execute only after an approximate four second delay.  It is a serious shortcoming in what is overall a fine O/S.

    Good try, but the issue has been pursued far beyond those generic fixes - all the way to extensive discussion with MS Level II administrators.  It's a known bug that some speculate may have been present in earlier O/S as well, though I didn't experience it until the advent of Windows 10.  The fault has been noted repeatedly in various forums.

    Apparently it ranks as low priority;   relatively few folks run macros or use the keyboard to launch shortcuts, but it should be fixed.  I courteously suggest that you (again) bring it to the attention of higher level support.

    It occurs, after a time and randomly, when certain apps (usually STORE-related) that are not running are propagated as 'suspended' background processes.  One common offender is CALCULATOR.  Another is SETTINGS.  Once they appear in the task manager background processes list the lag becomes obvious.  Kill them, and the lag is gone.

    Disabling SUPERFETCH here fixes the problem.  Of course, that negates whatever benefits SF may provide.

    Personally, I have sensed no performance hit by disabling that service.
