SELECT DISTINCT [tbl_jardiniers].[Prénom] & " " & [Nom] AS Prénom_nom_jardiniers, tbl_jardiniers.Jardin, [Téléphone] & (","+" "+[Téléphone2]) AS Tél1_Tél2, tbl_jardiniers.Adresse, tbl_jardiniers.Courriel, tbl_jardiniers.[Code Postal], tbl_jardiniers.Ville, tbl_jardiniers.[Actif], tbl_paiements_jardins.Saison_annee, ConcatRelated("[lot #]","tbl_paiements_jardins","ID_jardiniers = " & [tbl_jardiniers].[N°]) AS Lot

tom van stiphout say:

That is because the expression:

ConcatRelated("[lot #]","tbl_paiements_jardins","ID_jardiniers = " & [tbl_jardiniers].[N°])

does not limit the records to just those for 2017.

Try this:

ConcatRelated("[lot #]","tbl_paiements_jardins","ID_jardiniers = " & [tbl_jardiniers].[N°] & "AND Saison_annee=" & Year(Now())

ken sheridan say:

.................[Field] Like "*" & [Forms]![FormName]![TxtBoxName] & "*"

Don't believe everything you see on YouTube.  If you are going to use a query which references a form's controls as parameters that is not a good approach.  It is designed to make each parameter optional by allowing you to leave one or more of the controls in the form empty.  However, it is unreliable.

To understand why you need to know about NULL.  An empty field in a row in a table will be NULL, which means it has no value.  About the nearest you can get to a meaning for the field is 'unknown'.  In a comparative operation NULL is not equal to, greater than, lesser than, like etc anything else, not even another NULL.  The result of the comparison will always be NULL, nether TRUE not FALSE.

The effect of this is that if you don't put anything in the TxtBoxName control the expression [Field] Like "*" & [Forms]![FormName]![TxtBoxName] wiil evaluate to [Field] Like "**".  This should return any row whatever the value of [Field] is, i.e. the expression will evaluate to TRUE.  But as NULL is not a value, if [Field] is NULL in a row, that row won't be returned because the result of the Like operation will be NULL.

There is an easy way around this, however.  In relational databases we have at our disposal two special operators for handling NULLs, IS NULL and IS NOT NULL.  The way to do it in a query, therefore is this:

SELECT *
FROM [TableName]
WHERE ([Field] LIKE "*" & [Forms]![FormName]![TxtBoxName] & "*"
OR [Forms]![FormName]![TxtBoxName] IS NULL)
AND ([AnotherField]  LIKE "*" & [Forms]![FormName]![AnotherTxtBoxName] & "*"
OR [Forms]![FormName]![AnotherTxtBoxName] IS NULL)
AND ([YetAnotherField]  LIKE "*" & [Forms]![FormName]![YetAnotherTxtBoxName] & "*"
OR [Forms]![FormName]![YetAnotherTxtBoxName] IS NULL);

You can carry on like this for however many controls you have in the dialogue form.  The important thing is that each OR operation is enclosed in parentheses to force it to evaluate independently of the AND operations.  This will allow you to enter search parameters into as few or as many text boxes in the form as you wish, and the result will be those rows which match the combination of values you've entered.

When entering criteria in a query like this it is far simpler to do so in SQL view than in design view, and then to save the query in design view.  If the query is saved in design view Access will move things around a lot, and there is even the risk that the query might be too complex to open.  You can create the query without criteria in design view, but then switch to SQL view and add the WHERE clause.  If you've sorted the query the WHERE clause goes before the ORDER BY clause.  Then save it in SQL view.

The LIKE operator is a string operator, of course, so should only be used on text fields, not number or date fields.  With those you might use an exact search using the = operator, or search for values in a range using the BETWEEN.....AND opeartor.

You'll find examples of various search forms, including some which operate in the above way, in my little demo file, MultiSelect.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

jay sun msft say:
Hi Mykanne,

Thanks for your post. According to your description, the emails sent from the trusted sender (sent from your organization website) are verified as spam email and are delivered to your Junk Email folder. 

In this case, the cause of this problem may be related to the Sender Policy Framework (SPF) record. So, we suggest you add the SPF record in the domain register setting by following steps:

1. Login to Portal of Office 365 via the admin account.
2. Go to Admin center > Setup > Domains.
3. Click the domain name > click + New custom record.

Meanwhile, the SPF record is created as a TXT record and it will look similar to:

v=spf1 include:spf.protection.outlook.com –all

For related SPF TXT record information, you can refer to Set up SPF in office 365 to help prevent spoofing. In addition, you may need to add IP address to the SPF record. And you can get the IP address by looking for the “Received-SPF” header in the result of analyzing the header information of the problematic email in Message Analyzer of the Microsoft Remote Connectivity Analyzer.

Besides, we suggest you check the Block or allow list of the your account (login to Outlook Web App > click the gear icon at the top of the window > select Mail > select Block or Allow under Accounts). You can add the email address or domain of your organization website to the list of Safe Senders and Recipients or remove the related email address or domain from the list of Blocked Senders (For reference: Block or allow). 

Otherwise, if it doesn’t work, it might be related to the EOP spam policies of the organization. In this case, as you are the admin of the organization, you can try to prevent email from being marked as spam in EOP and Office 365 by referring to this article.
