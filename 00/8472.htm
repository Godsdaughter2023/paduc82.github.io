rick rothstein, mvp - excel say:
Okay, I have some worksheet event code (not a macro) that will react automatically when the value in E2 changes. To install this code, right click the name tab at the bottom of the worksheet you want to have this functionality and select View Code from the popup menu that appears (doing this will take you directly to the Module for that particular worksheet), then copy/paste the following code into the code window that opened up...

Private Sub Worksheet_Change(ByVal Target As Range)
  If Not Intersect(Target, Range("E2")) Is Nothing Then
    Rows(3).Hidden = Target <> "A"
    Rows(4).Hidden = Target <> "B"
  End If
End Sub

That's it. Go back to the worksheet and change the value in E2 to something other than "A" and Row 3 will hide, change it to something other than "B" and Row 4 will hide. I think this is the functionality you wanted.

john mcghie say:
Hi Karen:

You need to check all three parts of that error message.

"Document might be in use"?  Yes, it might be, if the documents are stored on another computer, and you have not logged in to that computer.

On a Mac, every user has a default ID of "501".  If you log in to the server each morning, the server will update that ID so that everyone on the network has a unique ID.  If that has not happened (no network login...) you get these misleading "in use" notices, because "everyone" on the network is known to the server as "501", and "every document" on the server is thus in use by "501".  If you are on a network, talk to your network administrator about this.

"Might not be a valid Word doc"? Yes, it might be, if the file extension is incorrect, or the file has been "unzipped".

The system (Mac OS X) relies on the file extension to tell it what kind of content is in the file, and thus, which component to use to open it.  If a .doc has a .docx extension, a .docx has a .doc extension, or the file has no extension, it is quite likely that the system will send the file to an opening component that cannot decode the content, so it thinks "This file is not a valid (expected) Word document."

Make sure your file extensions really are indicating what is in the file.  One way to find out is when you get such a message, start Word and go to "Open" on the Word File menu.  If you try to open from within Word, and the file opens straight up, that's a sure sign that the extension is wrong.  Word doesn't care about the extension, it reads the file itself to work out what is in there.  But Mac OS X will send the file to the wrong application if the extension is wrong.

To fix this, if the file opens, go to the File menu and choose Save As.  In the prompt, specify "Word Document (.docx)". That will make sure the extension is set correctly.  Do not change the extension on a file in the Finder unless you are sure you know what you are doing: chances are you will break the file :-)

Some email systems "process" attachments on the way through.  A Word .docx file is internally a Zip (Base-64-encoded) file format.  Some email services, particularly the free ones, may "helpfully" un-zip the file.  If they do, the file is permanently damaged and irreparable on a Mac.  (OK, you can fix it, but you need to be really good with the command line).  Normally much quicker to ask for a new copy.

"file name might contain invalid characters": Phillip has partially explained this.  It's not just the "file name" that must not contain funny characters, it is the entire PATH + file name + extension.

For example:
John's MacBook/Users/john/Documents/My Archives/MegaThai/Introduction.docx
is an illegal file name, because there is an apostrophe ' in the name of the computer HDD.

Make sure you have the latest updates, because there have been some fixes for this condition.

Hope this helps

rohn007 say:

As suzanne says, you can simply copy the example macro, but I understand that you may want to work through the steps to gain practical experience in recording macros.

Sure recording the macro will color some text.  But after recording the macro you have a couple of options for removing the unwanted green color.

1. insert some random text and select that, then delete it after you are done.

2. after the macro is recorded, use <CTL><Z> to undo that change

3. after the macro is recroded, select the text and use <CTL><Spacebar> to clear the formatting (oops, sorry not in this case, "highlight" is not "text" formatting that is cleared by this command).

4. Simply select some text that you want colored!

NOTE:

Recording macros can be a good way to get a starting point on a macro, but just like HTML that Word generates, the macros are often "cluttered" with extra "carp" / commands that are not absolutely necessary to perform the required function.  You may want to / have to "clean out" some of the "extra stuff" to have a macro that is more focused and easier to read and maintain/change in the future.

In addition to the default comments that the recorder inserts, I find it helpful, aka "good practice", to add some descriptive comments.  Things like:

  • In the header, describe the high level function of the macro. At a high level, what is it supposed to do? And why.
  • Use the default inserted date as a starting point for a change log (record when and who makes changes to the macro)
  • add comment line(s) to the body of the macro to describe specific functions of sections
