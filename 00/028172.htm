Assuming that by 71.03, you mean  71 Acres and 03 Gunta,  and

=(INT(((INT(A2)*40)+(A2-INT(A2))-((INT(B2)*40)+(B2-INT(B2))))/40))+((ROUND(MOD(((INT(A2)*40)+(A2-INT(A2))-((INT(B2)*40)+(B2-INT(B2)))),40),0))/100)

False Evalation in Excel - Microsoft Community

vahed say:

False Evalation in Excel

Hi all

Why Excel has a error during evaluating the formula ?

joeu2004 say:

Vahed wrote:

I Upload the main excel file here for more info.

Thanks.  Very helpful.

First, now that I see all of your data, I would suggest that you round to 5 decimal places, not 3, since some of your data has 5 decimal places.  In particular:

1. Formulas in column C should be of the form (C4 for example):  =ROUND((A4-1)*B4, 5)

2. The formula in D133 should be:  =ROUND(AVERAGE(D132,D134), 5)

3. Most importantly, the formula in G133 should be:  =ROUND((C134-C132)/0.05, 5)

It is really #3 that remedies the problem.

But #1 and #2 are "good practices" -- that is, if you agree that it is appropriate to round those formulas to some number of decimal places, in the first place.  Alternatively, perhaps you should only do #3.  Only you can make that decision.

Again, the root cause of the problem(s) is that Excel uses 64-bit binary floating-point to represent numeric values internally.  Numbers are represented by the sum of 53 consecutive powers of 2 ("bits") times an exponential factor.  Consequently, most decimal fractions cannot be represented exactly, notwithstanding appearances to the contrary.

Moreover, the summation series of powers of 2 to estimate a particular decimal fraction might differ because some bits are needed to represent the integer part .  For example, the estimate of 1/10 in 1.1 is different from the estimate of 1/10 in 0.1.  Consequently, IF(10.1 - 10 = 0.1, TRUE) returns FALSE(!) because the difference between the approximations is significant enough.

That is the problem with the formula in G133.  Even though C134 is exactly 3.25 and C133 is "exactly" 3.2 (coincidentally, even if we do not round, per #1 above), the fact is:  2/10 cannot be represented exactly, and C134-C132 is not "exactly" 0.05.  Consequently, (C134-C132)/0.05 is not "exactly" 1.

(By "exactly", I mean:  the same binary representation as the equivalent constant.)

Since you format F134 as General, you see the infinitesimal difference, namely about -1.15E-14.

You can see the inexact results in G133.  In the Formula Bar, if we highlight C134-C132 and press f9, Excel displays 0.0499999999999998.  (Be sure to press Esc to restore the formula as it was.)  Likewise, if we put the cursor anywhere in the Formula Bar and press f9, Excel displays 0.999999999999996 for the result of the entire formula.

In this case, the infinitesimal differences are large enough that even Excel displays them.

But recall that Excel formats only up to the 15 significant digits, rounding the 16th.  So what Excel displays is usually not the exact decimal representation.

For example, before applying #1 above, C7 appears to be 0.075, even we if press f9 in the Formula Bar.  And =C7-ROUND(C7,5) returns exactly zero, suggesting there is no difference.  But Excel is tricking us, trying to do us a favor.  In fact, =C7-ROUND(C7,5)-0 returns about 1.39E-17, which demonstrates that C7 is not "exactly" 0.075.

(The redundant -0 thwarts Excel's trickery.)

bernd p. say:

OneNote 2016 needs a MAPI compliant email program to send emails. Outlook 2016 is one. The Windows 10 Mail app is not.

That MAPI compliant email program must be set as default email program.

Once upon a time (in the time of Windows XP) Seamonkey was MAPI compliant by overwriting the MAPI32.DLL.

If this works in Windows 10, I don't know. Ask the Seamonkey community.

Or you could try Thunderbird.

Or you can use (it works) the good old and free Windows Live Mail.

Bernd
