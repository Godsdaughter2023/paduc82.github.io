ken sheridan say:

The reason for the error is that you are trying to reference a subform as a member of the Forms collection.  Only open parent forms are members of the collection.  To reference a control in a subform you have to do so via the parent form; the syntax is a s follows:

 

Forms.[ParentFormName].[SubFormControlName].Form.[ControlName]

 

However, unless you correlate the subforms the second subform could be referencing any row in the first subform, so I don't see how you can do this without correlation.  If you do not correlate the subforms then you would need to include the necessary referenced table(s) in the second subform's recordset so that the computation can be done per row in the second subform's query.

ken sheridan say:

If you'll exclude the pun, this seems to be a chicken and egg situation.  You have introduced a foreign key column PairID into Clutch in order that each row in the Clutch table can map to a row in the Pair table.  To update the existing values in the PairID column in Clutch, however, each row in the Clutch table needs to already be able to map to a row in the Pair table.  So, unless there are other columns in the tables which already establish a conceptual relationship between them you cannot automatically update the Clutch table and create a relationship object.  At first sight it looks to me, therefore, as if you might be faced with manually updating the 1725 rows in the table.

The only possibility of avoiding this would be if the Clutch table includes FemaleID and MailID columns which, in conjunction with the PairYear and the year of the FirstEggDate columns could be used to join the tables in an UPDATE query.  However, as the MailID column contains Nulls, you would be reliant on the values of FemaleID and PairYear being, in combination, distinct, i.e. each female being part of only one pair per year.  Without your expert knowledge of the subject, I don't know if that is the case or not.

Once you have the column populated and the relationship created and enforced.  It would be a simple task to include a combo box bound to the PairID column in a form for inserting data in the Clutch table.  the combo box's list could include whatever columns you need from the pair and other related tables to allow you to select a pair.  We can come back to that later once you've overcome the current problem.

ken sheridan say: As far as I can tell from the relationships there is no indirect path between Clutch and Pair which would allow you to map rows in one to the correct rows in the other, so it does look like you'll have to get your nose down to entering the values manually.

Time and space here do not allow a detailed analysis of your logical model, but I do notice that you have a couple of instances of what's known as 'encoding data as column headings' by having Observer1ID and Observer2ID columns.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.  The correct way to model a many-to-many relationship type like this is by means of a separate table which resolves the relationship type into two one-to-many relationship types, representing each observer as a separate row in the table.  This also means you can include other non-key columns to represent attributes of the relationship type.

nathan o. - microsoft support say:

Hi MJ,

 

It appears your old database file still has a referenced in it that is looking for dao235.tlb on  your machine.  Unfortunately, this is an old dao file that doesn’t ship with the newer versions of Office and /or newer operating systems.  In most cases, you should be able to uncheck the reference by doing the following:

 

+>Make a copy / backup of your database file.

+>Open your database in Access 2010.

+>Click Ok on the error message (you may have to do this several times).

+>Press Alt + F11 on your keyboard to bring up the VBA window.

+>Click on Tools and then select references.  You may receive the error message again when doing this.  If you do, once again just click Ok.

+>You will see a few references checked at the top of the reference dialog.  I would assume one of these references is for the dao235.tlb reference and it most likely is preceded with the word Missing.  Uncheck this reference.

+>I would then recommend trying to check the dao 3.6 reference by scrolling down in the list and checking “Microsoft DAO 3.6 Object Library”.

+>Now try closing and reopening your database.

johngeorgewebster say:

how do I enter a formula to find the age in the furture from a DOB entered

how do I enter a formula to find the age in the furture from a DOB entered
macropod say:

Hi John,

 

To see how to do this and just about everything else you might want to do with dates in Word, check out my Microsoft Word Date Calculation Tutorial, at:
http://lounge.windowssecrets.com/index.php?showtopic=249902
or
http://www.gmayor.com/downloads.htm#Third_party
In particular, look at the item titled 'Interactively Calculate A Person’s Age', which you could modify to the following format.

{QUOTE
{ASK BirthDate "What is the Birthdate?"}
{ASK EndDate "What is the end date?"}
{SET by {BirthDate \@ yyyy}}
{SET bm {BirthDate \@ M}}
{SET bd {BirthDate \@ d}}
{SET yy {EndDate \@ yyyy}}
{SET mm {EndDate \@ M}}
{SET dd {EndDate \@ d}}
{SET md{=IF((mm=2),28+(mm=2)*((MOD(yy,4)=0)+(MOD(yy,400)=0)-(MOD(yy,100)=0)),31-((mm=4)+(mm=6)+(mm=9)+(mm=11)))}}
{Set Years{=yy-by-(mm<bm)-(mm=bm)*(dd<bd) \# 0}}
{Set Months{=MOD(12+mm-bm-(dd<bd),12) \# 0}}
{Set Days{=MOD(md+dd-bd,md) \# 0}}
"If the Date of Birth was {Birthdate \@ "d MMMM yyyy"}, the age on {Enddate \@ "d MMMM yyyy"} will be {Years} Year{IF{Years}= 1 "" s}, {Months} Month{IF{Months}= 1 "" s} and {Days} Day{IF{Days}= 1 "" s}."}

How can I Find/Replace line breaks in Mac 2008 Excel

I'm using Excel within Office 2008 for Mac. In a large spreadsheet, I need to replace all line breaks within a cell with a space. I understand that to CREATE a line break, I would use Ctrl+Alt+Enter, but the Find box does not accept that. It also does not accept a pasted line break. I've tried everything I can think of. According to other forums, Windows versions of Excel can enter an ASCII value for a line break by holding down the ALT key while typing the ASCII code for a line break (0010). But this apparently does not work in Mac. Please help! Thanks.
j.e. mcgimpsey say: The simplest way (though with a large spreadsheet, it may still not be simple) is to insert a blank column next to cells containing line breaks. In that column (say, column B), enter

B1: =SUBSTITUTE(A1,CHAR(13),"")

or

B1: =SUBSTITUTE(A1,CHAR(13)," ")

depending on whether you want to keep a space between the end of one line and the beginning of another.

Copy down as far as required. Select the column. Copy it. Select the original column. Choose Edit/Paste Special, selecting the Values radio button.

Delete the column you inserted.

adam_w_ say:

Hello 8XyZxYz8,

 

In addition to the great suggestion above, this workaround may help:

 

1. Select the entire area that you want to edit within your Excel spreadsheet and select to Copy

2. Paste this data into an empty Word document.

3. Select Edit then Find….

4. Click the triangle in the bottom left corner, then select the bar in the bottom right corner which says Special and select “Manual line break”

5. The following icon should appear in the search bar: ^l

6. Next, select the Replace bar in the top of the Window

7. In the “Replace with” box, enter a single space

8. Select Replace All

9. Now paste your content back into your Excel spreadsheet

 

Hope this helps!
