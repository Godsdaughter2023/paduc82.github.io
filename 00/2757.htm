Item inventory and sales record in Access 2016

I want to have a form that saves data to tblSoldItems, but in it I want to select the product from a drop down list (from tbl Products) and customer from another drop down list (from tbl Customers), so then it fills in most of the form for me. dhookom say:

Several recommendations (probably more than you want):

ID is typically not a good name for a field since your database contains at least 3 fields with the same name which might cause confusion. I would recommend using ProductID in tblProducts. Notice I don't ever include spaces and punctuation/symbols in object names.

Name is reserved word since every object has a name property. Most of us prefer fields like FullName, FirstName, LastName, CustName, etc.

I wouldn't delete the records from tblProducts. If the ProductID is in tblSoldItems then we know it is not available for sale. Leaving the record in tblProducts means you don't have to store Description, Colour, Price in tblSoldItems since the values will be available in tblProducts.

My suggestion would be a simple form that allows you to select the ProductID and CustomerID from combo boxes and then enter the Payment Method, Card Type, and Card number.

scottgem say:

So your customers buy only one product at a time? And each product is a unique item?

That's an unusual business. 

tblSoldItem should NOT have the Description, Color or Price fields in it. One of the principles of a relational database is that you don't repeat data. 

As for displaying the information after selecting the ProductID please refer to my blog article on Displaying Data from Related Tables (see sig for blog address)

ken sheridan say: I fully agree with Duane about the inadvisability of deleting rows from tblProducts.  In your sales form set the RowSource property of a combo box bound to ProductID to:

SELECT ProductID, Description, Color, Price
FROM tblProducts
WHERE NOT EXISTS
    (SELECT *
     FROM tblSoldItems
     WHERE tblSoldItems.ProductID = tblProducts.ProductID)
OR SaleID = Form!SaleID)
ORDER BY Description;

Set its other properties as follows:

Name:                 cboProduct
ControlSource:    ProductID
BoundColumn:    1
ColumnCount:     4
ColumnWidths:    0cm;3cm;3cm;3cm
ListWidth:             9cm
 
Experiment with the last three ColumnWidths dimensions to get the best fit.  The first must be zero.  The ListWidth should be the sum of the  ColumnWidths dimensions.  Access will automatically convert the values to, inches if you are not using metric units.

In the AfterUpdate event procedure of the combo box assign the current price to a Price control in the form with:

Me.Price = Me.cboProduct.Column(3)

The Column property is zero-based, so Column(3) is the fourth column.

Requery the combo box in the form's Current event procedure with:

 Me.cboProduct.Requery

Note that a RowSource property like the above, which references the current record's primary key to include the product in the list in the case of already saved records, will work fine in a form in single form view.  In continuous forms view, however, this will not be the case without the use of a hybrid control in which a text box is carefully superimposed on a combo box to give the appearance of a single combo box control.  You'll find an example of this in ActiveEmployees.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

I also agree with Duane about the column names, and have renamed the relevant keys ProductID and SaleID in the above.

You cannot avoid the very limited use of VBA code in the above.  If you are unfamiliar with entering code into a form's, report's, report section's  or control's event procedures, this is how it's done in form or report design view:

1.  Select the form, report, section or control as appropriate and open its properties sheet if it's not already open.

2.  Select the relevant event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new line(s) between these.
