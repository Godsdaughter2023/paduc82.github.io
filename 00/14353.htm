Try this to pick the usedrange.

 

Sub select_used_range()
    Dim Lastrow As Long
    Dim LastCol As Long
    Dim myRng  As Range
    With ActiveSheet
        Lastrow = .Cells(.Rows.Count, "A").End(xlUp).Row
        LastCol = .Cells(1, .Columns.Count).End(xlToLeft).Column
        Set myRng = .Range("A1", .Cells(Lastrow, LastCol))
       ' MsgBox myRng.Address
    End With
End Sub

 

In "most" cases this will work.

albert d. kallal say:

As a general rule when you setup custom ribbons and un-check startup options to hide most of the Access interface from users, then as a developer you simply hold down the shift key during startup. This means your startup code, startup forms and the custom startup ribbon you specified will not run.

 

So, holding down the shift key during startup should get you back in business.

 

Of course, during development, you will hold down the shift key so yourstartup settings don't run. You then develop for awhile, and then to test in"user" mode, you exit, and then re-enter the application without the shiftkey bypassed. You will likely do this dance all day long as you run/test as

user mode, and then flip back in to developer mode (shift key used..so youdont get the custom ribbon, and all kinds of your startup settings that likely hide the access interface don't run). So, you cant develop, or really modify things when you run your application with the startup settings...so you must shift-by-pass them when you want to work.

 

You can also use alt-f4 to exit the application. The accdb file should still be highlighted in the windows explore, so, then you hit enter key (and, hold down shift key if you need be). This key stroke sequence and exiting and re-entering the application will occur CONSTANTLY all day long when you are developing. I often place a compact + repair in the QAT and use that in place of exit + re-enter into the application. So, just hit the Compact + repair and hold down shift key (or not) to flip between testing the application how it will run for your users as opposed to developing mode (hold down shift, prevent startup code that hides the access UI)..

 

When you finally have things just right...you create the accDE that you plan to distribute. (the accDE can't be modified by users).

saberman say:

Open Access with a new database.  Open a module create a public function:

Public Function RemoveCustomRibbon(ByVal vsDatabaseName As String, Optional ByVal vsPassword As String = "")

    Dim dbTarget As Database

    Set dbTarget = OpenDatabase(vsDatabaseName, False, False, "ms access;pwd=" & vsPassword)
    On Error Resume Next
    Err.Clear
    Do
        dbTarget.Properties.Delete "CustomRibbonID"
    Loop While (0 = Err.Number)
    dbTarget.Close
    Set dbTarget = Nothing
   
End Function

Run the function with the path-database and optionally a password.

Dates are really just numbers - the number of days since the start of the year 1900.

If you change the format and the display does not change, that means that your values are entered as text, not as true dates.

To correct that, select all the cells and change the format to the date format you want. Then use Ctrl-H (Find and replace) and replace / with /    That will re-enter the dates as dates, and will convert the 03/21/2017 to a date that will display at 2017-03-21
