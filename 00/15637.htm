Why does viewing a file in Excel modify it? - Microsoft Community

richpasco say:

Why does viewing a file in Excel modify it?

It's important to me to have confidence that my tools work as expected.  Two fundamental expectations are:

(1) If an application modifies the contents of a file at all, the operating system (OS) sets its modification time to the current time.

(2) If you open a file in an application and then quit that application without saving, the contents of the file should not be modified.

I'm running MS Excel 2007 under Windows 7 Pro, and a recent experience has shaken both assumptions.  Let original.xls be an existing spread sheet and run the following batch file.

--------------------------------------------------
rem First we make a working copy of our file
copy original.xls working.xls

rem Next we verify that the copy is good
comp original.xls working.xls

rem Now open it.  Please close it with Alt-F4 without making any modifications.
start /w "title" working.xls

rem now compare again - notice that they differ
comp original.xls working.xls

rem compare the dates/times
dir original.xls working.xls
--------------------------------------------------

OK, I can hear the folks at Microsoft muttering that the change was not significant, perhaps Excel just changed some meta-data in the file, not the spreadsheet itself, and then reset the file modification time to the original time so as to cover its tracks in the process.  That is beside the point.  The content of the file was changed, and said change was not reflected in the OS modification date.  To me this is simply wrong.

By the way, I can prevent this behavior by marking the file as read-only before viewing it:

attrib +r working.xls

but this defense against unwanted file modifications shouldn't be necessary.

I have been chasing this issue for some years and only today realized a work-around that I can live with. As far as I am concerned, this is just sloppy code from Microsoft but they don't have the will to fix it.

All file access should be done via the O/S interfaces to the file system and any changes to a file

Flagging a file as read-only will avoid the issue. However, it's not practical for regularly updated files - like your recent bank transactions.

However, with the introduction of the newer XLSX file type, Microsoft seem to have fixed the issue. Possibly inadvertently. Nonetheless, if you re-save your spreadsheets in the XLSX format (SaveAs...) the problem seems to disappear. Of course, older versions of Excel will then be unable to open the files. Hey, if you want a professional product, only open source will give you full control!

Hope that helps,

John

Note: There are API calls that can be used for managing files which can be used to change time stamps. Search for a utility called Attribute Changer for an end user level of access to these. Excel alters a handful of bytes every time you open an XLS to read something and then close without saving. And if you use checksum techniques (like MD5 hashes or similar) they will flag such changes as errors leading you on a wild goose chase because most sync software only compares files by last edit date and time to avoid the lengthy process of reading the full content of files.

warren b. say:

Some decimals are not exact in XML Spreadsheet format

We are saving spreadsheets in Microsoft Excel 2003 XML Spreadsheet format (so that a program can process the data and insert it into a database).  We have found a difference (bug?) in how certain decimal numbers are stored when using Excel 2007 compared to Excel 2003 (both on Windows XP).

Excel 2003
Create a new spreadsheet (for this example, just put data into 2 columns of the first row). 
In column 1 enter 20.1, in column 2 enter 20.2
Save as a type of "XML Spreadsheet".
The data in the XML is:
   <Row>
    <Cell><Data ss:Type="Number">20.1</Data></Cell>
    <Cell><Data ss:Type="Number">20.2</Data></Cell>
   </Row>

Excel 2007
Create a new spreadsheet (for this example, just put data into 2 columns of the first row).  
In column 1 enter 20.1, in column 2 enter 20.2
Save as a type of "XML Spreadsheet 2003".
The data in the XML is:
   <Row>
    <Cell><Data ss:Type="Number">20.100000000000001</Data></Cell>
    <Cell><Data ss:Type="Number">20.2</Data></Cell>
   </Row>
Why is 20.1 stored in the file (in this version of Excel) with 15 significant digits?

It didn't change using Excel 2007 if the cells are formatted General or Number (with 1 decimal).  And it still happens when the option "Precision as displayed" is turned on. 

Is there any way we can prevent this?  We don't care how it looks in the cell - we care that the XML contains exactly what was entered.  And it's strange that the two versions of Excel behave differently.

Thanks a lot.

Warren B.

joeu2004:

That said, I am curious:  how does all this adversely affect you?  Do you have any examples where it does?


I understand your points about 20.100000000000001 being numerically the same as 20.1 when using the value in a program (and the difficulty of representing floating point numbers using binary).  

We are collecting data that is supposed to have no more than 1 digit after the decimal point because we want all values to have the same measurement accuracy.   When we find a value with more than 1 decimal digit, we report an error to the user and reject their submission.  We had to put in special case code so that if the XML contains (exactly) 15 decimal digits, we suppress the "too many decimal digits" error message and let them file because it is not their fault!  Then we round the value to the desired 1 decimal place when inserting into a database.
