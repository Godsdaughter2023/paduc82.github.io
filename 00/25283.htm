bill mosca say: It sounds like you are trying to make one field dependent on another. That is not a normalized structure. What you should do is have a table with both of those fields and a primary key (autonumber). Then you just use that key in your first table. That's called a foreign key. It creates a relationship between the two tables.

Look at the Sample database Northwind to see how the Order Details table is set up with a Product foreign key.

A good place to learn about table structure is Crystal's free eBook "Access Basics".

ken sheridan say: Firstly you should only interface with the data via bound forms, never in a table or query's raw datasheet.

Assuming the combo box is bound to a foreign key column which references the primary key of a referenced table from which the value you wish to show is derived, in the vast majority of cases you should only store the foreign key value in the referencing table.  You can show the values from one or  more non-key columns from the referenced table in a number of ways, e.g.

1.  Return the relevant columns form the referenced table in the combo box's RowSource query and as the ControlSource property of one or more unbound controls in the form reference the Column property of the combo box.  This is zero-based, so to reference the third column for instance you'd reference Column(2) of the combo box.  You'll find an example of this in the file Images.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This demo file illustrates various ways of attaching images to records, but it so happens that the addresses form which it uses as an example includes a County text box which references a City combo box with:

=[City].[Column](2)

to show the county in which the selected city is located.

2.  Another way is to simply base the form on a query which joins the referencing and referenced tables on the keys, and return the non-key columns from the referenced table in the query.  You can then bind controls to these in the form.  To make the controls read-only set their Locked property to True (Yes) and their Enabled property to False (No).  NB the keys must be appropriately indexed.

The exception to the above is where the value of the column in the referenced table can change over time, but you would want the value in the referencing table to remain fixed as that at the time when the row was inserted.  An example is the UnitPrice of a product in a Products table which is referenced from an OrderDetails table.  As you would not want existing orders to reflect subsequent price changes, you would in this case also include a UnitPrice column in OrderDetails and assign the current unit price from Products to it in the AfterUpdate event procedure of a ProductID control, e.g.

Me.UnitPrice = Me.cboProduct.Column(2)

ken sheridan say: Don't print the form, but create a report and print that.  Forms are for interfacing with data on-screen, reports for printing or viewing data.  The report can mirror the set-up of the form by embedding a subreport in the parent report, though rarely will a form's layout be entirely suitable for printed output, and often you can do so as a single report by basing this on a query which joins the relevant tables.  The report's sorting and grouping mechanism can then be used to print the data from the referenced table once only in a group header, and that from the referencing table in the detail section.

To print a report filtered to the form's current record you can include a command button in the parent form with code along the following lines in its Click event procedure;

Dim strCriteria As String

strCriteria ="MyID = " & Me.MyID

'ensure current record has been saved
Me.Dirty = False

' print report filtered to current record, if any
If Not IsNull(Me.MyID) Then
    DoCmd.OpenReport "MyReport", WhereCondition:=strCriteria
End if

Hi Patrick,

In the Outlook client, autodiscover automatically loads all mailboxes for which the user has full access permissions without the password. This feature is called auto-mapping.

We can check and manage emails via the auto-mapping mailbox and the mailbox added as a separate OST file. The main differences between the two are:

1. When new emails come, auto-mapping mailbox has no notification.

2. Auto-mapping mailbox general has slower performance than the mailbox added as a separate account.
