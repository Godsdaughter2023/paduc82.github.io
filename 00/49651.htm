The  Certificate's CN name does not match the passed value.

This means that you're sending using SSL, which requires an exchange of security certificates to open a secure tunnel through which transmissions to and from the server can be passed. Each certificate is issued for one or more server names*, but the one on the server you're trying to connect to is not valid for the server name you're using.

Right-click on the account name in the folder pane and select Properties. What is the name in the Outgoing mail (SMTP) box?

Outgoing mail:  smpt.tds.net

If that server name was accurately transcribed, then your problem is easily solved: change the server name to smtp.tds.net 
Now compose a test message and send it to yourself. Do you still get the certificate error?

* This is the first case of its kind I've come across in many years of helping people with their email services. This is what I'm guessing is happening:

The commands needed to get a message from one email account to another are bundled together into a protocol called the Simple Message Transfer Protocol, abbreviated SMTP. Mail servers that use this protocol to transport messages around the Internet - like the one you're connecting to - often have names that start with smtp. However, it's very easy to type this wrongly, and the commonest typo is the one you appear to have made - smpt instead of smtp. They sound pretty much the same over the telephone, too, so tech support won't usually detect the difference. 

Because it's so common, some Email Service Providers (TDS in your case) take pity on their customers and make allowance for the typo by registering both the right and the wrong spelling as the name of the server**. TDS have done this, so a message sent to smpt.tds.net will go to the same server as if it had been sent to smtp.tds.net and you'd never know that you'd made a typo.

Your problem arose, though, when TDS started using SSL and had to provide a certificate to secure the connection between your computer and the SMTP server. The certificate was obtained and installed, but it was issued only for smtp.tds.net. So when you try to connect to smpt.tds.net, your computer security pricks its ears up and says Hey, that certificate isn't valid for that server name! You choose to ignore the warning and the message goes through anyway.

If you talk to TDS tech support about this, you might suggest to them that if they're going to allow secure connections to smpt.tds.net, they should add that as a Subject Alternative Name (SAN) to the server certificate.

If my guess is correct, then thank you for injecting a moment's light relief into an otherwise dreary day
