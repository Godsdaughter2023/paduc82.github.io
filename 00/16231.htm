vanderghast say:

There may be a possibility to use the Simplexe method, but that is not based on database. The only solution I can see with database would be to generate ALL the possibilities and examine them, explicitly, to find the "minimum". That is way too expensive, though, in ressources, to use such a brute force method.

 

For the Simplexe method, if you have access to the algorithm/program, the "problem" will be:

 

given variables  x[i][j] =  1 if piece [i] is cut in roll [j]; 0 otherwise  
         variables  y[j] = 1 if roll [j] is used, 0 otherwise
         constants   L[i]  = length of required piece [i]
         consants    R[j] = available length for roll [j]

 

Clearly, we look at a solution for the values for all x[i][j]. Note that if you need two pieces of 7 m and 5 pieces of 6 m,  you need 7 pieces ( 2 + 5)  so  i = 1 to 7. Indeed, EACH wanted piece has its entry, its 'i". Same for the rolls: you can handle the "used" ones, which are not making 10 m, as new ones, but each real roll has it own "j".

 

Constraints: 

                     x[i][j] in {0, 1}


                     y[j]  in {0, 1}

 

                     Since piece i is just cut once,   Sum over j ( x[i][j] )   = 1
                     That generates n constraints  ( n = maximum value of i )

 

                     Since what we cut cannot exceed the roll length:  Sum over i (  x[i]ji] * L[i] )   <=  R[j]
                     That generates m constraints  ( m = maximum value for j )

                     A roll j  is used if at least one x[i][ given j ]   = 1, so
                     y[j] >= x[i][j]                     m*n  constraints

 

Objective function:  the Simplexe method requires something to minimize (or to maximize), so, here, it is to "touch" the minimum number of rolls:

                    z =  MIN SUM over j (  y[j]     )

 

The standard Simplexe may find not-integral solution  (like  x[1][5] = 0.87469  )  which is not acceptable. You have to use the modified algorithm which returns integral solutions only. 

 

------------------------

You may find that a better objective function will be:

                 z= MIN SUM over j (  R[j] * y[j]  )

 

Indeed, using a roll of 1 m  should be less costly than using a roll of 10 m, so adding the length of the roll as "weight" to the "cost of touching a roll" is probably closer to the intuitive idea of what we consider a "minimum", in that case. Without such weight, using ONE roll of 10 meters will be prefered to use TWO rolls of 1 meter to get 2 pieces of 1 meter. With that new objective function, the optimal solution would be to use the two roll of 1 m (cost = 2) rather that one roll of 10m  (cost = 10).

-------------------------

http://www.zweigmedia.com/RealWorld/simplex.html supplies an on line Simplexe calculator, but I am not aware of how it works.

 NOTE 1:  generally,  x in {0, 1}   should normally be entered as  x <=1   since x>= 0  should be implicit (Simplexe method is always with positive variables; side note:  to use + or - values, the technique is to use    z  = a - b    with  both a, and b >= 0 and in the optimal solution, we would find that  only a or only b will be eventually not equal to 0, while both can be 0 too. )

 NOTE 2:   MIN z1 + z2 + z3   ===    - MAX -z1  -z2 - z3. or, if you prefer, max{x}  = -min{-x}

 NOTE 3:  it would be tedious to enter, manually,  more than  2*m*n  constraints, so nice tool to test the proposed "problem", but a programmed solution is definitively better, in the long run.

john w. vinson say:

This is a classic example of a "NP Complete" problem - for large numbers of rolls and large numbers of pieces, there is (almost certainly, mathematicians have been trying to remove the "almost" for over a century) no solution that is both optimum and efficient. To be sure you have the best solution, you have to compare all the possible solutions - and the number of possible solutions expands exponetially with the size of the problem. It doesn't take much to get into tens of millions of cases to test!

The best bet is to use the "greedy algorithm" that Vanderghast suggests. It may not get you the solution with the absolute least wastage but it will get you a valid solution (if there is one possible) that will usually be either the  best, or if not, at least pretty good.

 

marshall barton say:

To expand on what others have said, this is the classic Napsack problem.  Your situation is certainly simpler with one dimension (length) than with three, but it's still a very, very difficult problem where it is pretty much necessary to settle for a good solution rather than try to find a best solution (for your small to mefium size orders, the good enough solution may very well be the best solution).

You can find a lot of discussions on the Napsack problem by searching the web.  There are doctorial thesis on the subject so you may need to wade through some incomprehensible stuff to find something that addresses only your one dimensional problem along with any other simplifying constraints you may be able to come up with.

vanderghast say:

Technically, the problem is one order of magnitude LESS than the napsack problem since in the napsack problem, you have to consider to take n objects out of m to be in the  sack, for all possible n, that is, for n=1 to m. So, you have C(1, m) + C(2, m) + ... + C(m,m) cases to consider, with C(i, j) = how many possible choices to select i objects from a set of j objects.That makes a large number of cases to consider. But the actual problem forces you to take all the required "object"s", not just any subset., so we have just ONE case, instead of that ugly sum of cases, to consider. That helps a little.

Sure, part of the problem is about WHAT to optimize. SInce nothing is created, and nothing is lost, we have: Sum of Previous Inventory in meter = Sum of What Is Required + Sum of What Will be Left Over. So, for ALL possible solutions, What will be left , in meter, will always be the same, so, it cannot be minimized (it is a constant of the problem). So, indeed, we are left to DEFINE what is to be optimized, exactly. In fact, that *is* the problem, imho.

Note that the same problem is applicable to cut from reinforcing steel bar, from pipes, from steel beams, ... which are all delivered in standard length(s) from the mill.

It is much worse in 2D (cutting fabrics, as example), that is true.
