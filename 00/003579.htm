LogeshwaranA S wrote:

I believe I asked recently, and the answer is "no".

You might try array-entering (press

=XIRR(INDEX(B:B,N(IF(1,MODE.MULT(IF(B1:B8<>0,{1,1}*ROW(B1:B8)))))),

 INDEX(A:A,N(IF(1,MODE.MULT(IF(B1:B8<>0,{1,1}*ROW(B1:B8)))))))

MODE.MULT was introduced in Excel 2010, and I have Excel 2007.  So I cannot test that formula.  See footnote [1] for an explanation of the seemingly unnecessary syntax, all of which

Alternatively, you might try the following VBA code, which I did test:

Function myxirr(v0 As Range, d0 As Range)
    Dim n0 As Long, n As Long, i As Long
    n0 = v0.Count
    If n0 <> d0.Count Then myxirr = CVErr(xlErrValue): Exit Function
    ReDim v(1 To n0), d(1 To n0)
    For i = 1 To n0
        If v0(i) <> 0 Then n = n + 1: v(n) = v0(i): d(n) = d0(i)
    Next
    If n = 0 Then myxirr = CVErr(xlErrValue): Exit Function
    ReDim Preserve v(1 To n), d(1 To n)
    myxirr = Application.Xirr(v, d)
End Function

Usage:

=myxirr(B1:B8,A1:A8)

Remember to save the Excel file as macro-enabled (xlsm) or as Excel 97-2003 compatible (xls).

-----

[1] MODE.MULT returns an array of row numbers that meet the criterion (B1:B8<>0).

The syntax {1,1}* is needed to make MODE.MULT work as intended.  Otherwise, MODE.MULT returns a #N/A error because there are no duplicates.

That should cause INDEX to return an array of the relevant values.  But inexplicably, INDEX does not work that way in that context(!).

There is no rational reason for the extra syntax:  N(IF(1,...)).  But "XOR XL" discovered empirically that the extra syntax is necessary and sufficient to make INDEX work as intended.
