ken sheridan say: Firstly the appointments should be printed as a report.  This can be set up so that by default it would return all appointments in the database sorted by date/time.  You can easily set up an unbound dialogue form to allow you to select the period for which the report is to be printed.  You could for instance add an option group to select the time period, e.g. with options Today, Tomorrow, This Week, Next Week.  Then print the report from a button on the form using code as follows, where optTimePeriod is the name of the option group and whose values are 1,2 3 and 4 (the control wizard can easily set this up for you).  The code in the button's Click event procedure would be like this:

Dim strCriteria As String

Select Case Me.optTimePeriod
    Case 1
    strCriteria = "[AppointmentDate] = #" & Format(VBA.Date,"yyyy-mm-dd") & "#"
    Case 2
    strCriteria = "[AppointmentDate] = #" & Format(VBA.Date+1,"yyyy-mm-dd") & "#"
    Case 3
    strCriteria = "[AppointmentDate] Between #" & _
        Format(VBA.Date - Weekday(VBA.Date) + 1,"yyyy-mm-dd") & "#  And #" & _
        Format(VBA.Date - Weekday(VBA.Date) + 7,"yyyy-mm-dd") & "#  
    Case 4
    strCriteria = "[AppointmentDate] Between #" & _
        Format(VBA.Date - Weekday(VBA.Date) + 8,"yyyy-mm-dd") & "#  And #" & _
        Format(VBA.Date - Weekday(VBA.Date) + 14,"yyyy-mm-dd") & "#  
End Select

DoCmd.OpenReport "YourReportNameGoesHere", WhereCondition:=strCriteria

Alternatively, for maximum flexibility you could include two unbound text boxes in the form, txtStartDate and txEndDate, formatted as Short Date to force the date picker to pop up when the control receives focus.  You'd then enter the dates from which and to which you wish appointments to be printed.  For a single day you'd enter the same date in each.  The code in the button's Click event procedure would then be:

Const MESSAGE_TEXT = "Both dates must be entered."
Dim strCriteria As String

If IsNull(Me.txtStartDate) Or IsNull(Me.txtEndDate) Then
    MsgBox MESSAGE_TEXT, vbExclamation,"Invalid Operation"
Else
    strCriteria = "[AppointmentDate] Between #" & _
        Format(Me.txtStartDate,"yyyy-mm-dd") & "#  And #" & _
        Format(Me.txtEndDate,"yyyy-mm-dd") & "#  

    DoCmd.OpenReport "YourReportNameGoesHere", WhereCondition:=strCriteria
End if

ken sheridan say:

Word turns the quotes into 'smart quotes'.  Turn off smart quotes in Word and it should be fine.  Because I use Word to draft my replies here I have the following macro, run from a button in the quick access toolbar in Word, for toggling smart quotes on and off:

 

Sub ToggleQuotes()

 

    Dim strMessage As String

   

    With Options

        If .AutoFormatAsYouTypeReplaceQuotes = True Then

            .AutoFormatAsYouTypeReplaceQuotes = False

            strMessage = "Smart quotes are off."

        Else

            .AutoFormatAsYouTypeReplaceQuotes = True

            strMessage = "Smart quotes are on."

        End If

    End With

   

    MsgBox strMessage, vbOKOnly, "Smart Quotes"

   

End Sub

jay freedman say:

By default, Ctrl+F activates the search box in the Navigation pane, and words that it finds are highlighted in a medium yellow shading:

If you use the Advanced Find dialog and click the Reading Highlight button, the found text will be shaded with whatever color you last used on the Highlight button on the Home tab:

If you want to change the font color, you can click the More button in the Replace dialog, click in the Replace With box and enter the code ^&, click the Format button in the bottom left, click Font, and select a color. Then click the Replace All button. The same process can be made into a one-click operation by installing this macro:

Sub FindAndColorFont()
    Dim rg As Range
    Set rg = ActiveDocument.Range
    With rg.Find
        .Text = InputBox("Enter word to find:")
        If .Text = "" Then Exit Sub
        .Format = True
        .Replacement.Text = "^&"
        .Replacement.Font.ColorIndex = wdRed
        .Execute Replace:=wdReplaceAll
    End With
End Sub

Different MS Access ODBC Query times between instances - Microsoft Community

wcunkefe say:

Different MS Access ODBC Query times between instances

Hello,

     I'm attempting to run a query using three linked ODBC tables (MS Access 2010.  If I run the query in one instance of access, it runs instantly.  When I copy the sql over to another instance of access, the query takes >30 seconds to run (when it works) and frequently times out.  Both access files are located in the same folder and are being run on the same computer.  I need the query to function properly in the second access application.  Does anyone have any ideas why the same query has vastly different results in the two instances of access?

wcunkefe say: I deleted the links to the ODBC tables in the instance of Access that was having poor performance, then re-linked the ODBC tables.  After this, the query runs instantly.  If anyone has any insight as to what causes the problem and why deleting and re-adding the linked tables fixes it, please let me know.

albert d. kallal say:

It hard to “pin” to this down. However if one of the linked tables (even tables not included in your query) are pointing to a non-existing location or instance of the table, then “long” delays can occur as access “checks” and tries to open the table.

So having a linked table that points to a non-existing table or location or instance of SQL server can really slow things down.

If your re-link process ONLY re-linked the 3 tables, then perhaps one of the table had “different” connection.

As a result, access believed that one of the tables was on a “different” server, and thus how the data engine “decides” to join the data could and would be drastically different approach.

Remember, one table could be a local table, another could be dBase, and the other could be on SQL server half way around the world.

Thus when Access is faced with 3 completely different data sources, then Cleary the join process and how Access deals with 3 compete different data sources that are to be “magically” joined together is not only going to be slow, but Access will certainly choose a “different” approach to join such data.

So if any of the 3 tables is “linked” differently, then Access is likely “guessing” and “thinking” that the data in question is NOT from one single database source, but different data sources. And you can't really join different data sources very fast at all.

Thus joins and dealing with the data will require a far different and slower approach to join such data together given that Access is “thinking” and “guessing” that the data not on a single server.

If you want REALLY super performance here, try creating a view based on that 3 table join, and link to the view - you find spectacular performance since all "joining" of the data between table occurs 100% server side.

So differently linked tables can confuse Access into not realizing the data all coming from the one same server. This also would be in the case of using SQL logons for some tables, and windows authentication for some of the other linked tables.  So make sure all of the tables are linked the same way. Access will cache the logon for ONE linked tables and re-use that logon for ALL OTHER tables - but ONLY if the other tables are linked with identical connection strings. If you don't do this, then you have 3 different logons cached and again that will cause performance issues.

VERY important to ensure that all tables are linked the SAME way and with the SAME connection string - else Access cannot cache the one connection for re-use.
