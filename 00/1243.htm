Two basic principles you overlooked:

Note that some of the information on the Microsoft page is outdated and contains errors, particularly when applied to Outlook 2003, 2007 or 2010. Specifically:
1. Ignore any advice to copy your PST file to Outlook's default location to connect it to your profile. Doing so will often corrupt your Outlook profile. Copy it anywhere BUT the default location and then open it within your Outlook profile.

There are a variety of reasons that Auto-Save may not be showing up. Please first notice that the co-author feature for Excel has the requirement below:

1.       Everyone should be in Monthly Channel or Monthly Targeted Channel that support the co-authoring feature. (If just one person uses a version of Excel that doesn't support co-authoring, then everyone else will get the "locked" error — even if everyone else is using a version of Excel that does support co-authoring.)To check that, you can go to File > Account > Product Information to check it. If you are not in the related channel, please refer to

2.       Put the file on OneDrive, OneDrive for Business, or SharePoint.

3.       Using the co-authoring supported file format. They should be .xlsx, .xlsm, or .xlsb format.

4.       Refer to

Unsupported features, for example:

Turn off the Shared Workbook feature. This feature is an older method of sharing. It has many limitations, and has been replaced by co-authoring.

If possible, remove password encryption from the file: Go to File > Info > Protect Workbook. Then delete the password and click OK.

If possible, remove restricted access: Go to File > Info > Protect Workbook > Restrict Access and click Unrestricted Access.

If you are opening a workbook snapshot from SharePoint, choose to open the file instead of the snapshot.

Turn off the Refresh data when opening file setting on all tables and PivotTables in the file. Select the table or PivotTable, then go to Table Tools Design > Refresh > Connection Properties. Then uncheck Refresh data when opening file.

Inactivate an add-in that's causing AutoSave to be disabled. Select File > Options > Add-Ins > Excel Add-ins > Go and then clear the check box for the add-in.

To debug your websites

SCRIPT5009: 'mzR' is undefined
Abstract (21,15)

you should revise your Masterpage for the site/gallery and remove the mzR call in the body onload event handler as it accurs in all of your aspx pages in your gallery sub folder. This error does not affect the time taken to download and display the loader.gif file however...

3. To debug your scripts, select the Debug tab of dev tools and select "Always break on exceptions" from the dropdown (it looks like a blue stop sign on the Debug tab of dev tools), then WITHOUT CLOSING THE DEV TOOLS, return to your web page and either refresh the page (f5) or do your testing.... any script errors that are encountered will now halt execution and break at the error line number in your scripts.

3. To debug latency issues (the time it takes to request and then download a resource) select the Networking tab of dev tools and click the 'Start' (>) button (unselect the "Always refresh from server button if it was selected), return to your web page (WITHOUT CLOSING THE DEV TOOL) and refresh the page (f5).... the resource url's, request headers and responses are now listed in the Networking console.

My dev tools shows 175 requests totalling 17.5 mb of responses, taking 240 seconds to load......(optimum is 3 seconds.... ) on an ADsL 2 connection.. (500 kb/s)... what it also shows is that the request for the page-loader.gif does not occur until all of the other image resources have been downloaded ....

To correct the issue modify your html/aspx here:

<div class="se-pre-con-loaded" id="pnlLoader">

</div>

to

<div id="pnlLoader">

<img alt="" src="assets/images/page-loader.gif" style="width:100%;height:100%"/>

</div>

this tells the web browser to download the image as the rest of the web page is parsed and rendered... that is before the other resources are requested and downloaded. It will then appear at the top of the Networking tab list.

Retest step #3.... you should now see that the request for page-loader.gif is call at the very top of the network request stack. The browser will now 1. pase the img tag and make a request for the image src value. 2. while the image is being downloaded it will continue to parse and render the rest of the markup and scripts, but once page-loader.gif has been loaded it should display it, so you should see the loader gif spinning away while your other 17mb of images are downloading for the  next 3 minutes!
