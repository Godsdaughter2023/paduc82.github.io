albert d. kallal say:

Ok, how this works:

 

If the workgroup file you are using does not have a password for the Admin account, then access will NOT prompt for a password.

 

However, if the workgroup file has a password for Admin, then it will launch the logon dialog.

 

If you type in say “Albert” over the default “admin” that appears, then next time you launch access that default “Albert” will appear in place of Admin.

 

I seen customer sites where for years they just hit “ok” (because “Albert” did not have a password.

 

If they bump a key, or type in a different logon name, then that name will appear as the default (until you type I a different name). At that point, they are often “lost” because they don’t remember what the logon was they were using (that did not have a password). Or they typed in a password for years, but don’t remember what the logon was they were using!

 

HOWEVER: Typing in a different logon name will NOT “overwrite” the password. So you cannot OVERWRITE the password with the above process. And you cannot overwrite the password by accident. You have to change it. So no matter how crazy the logon process has become, that process NEVER changes the password by intention, or by accident – it cannot happen.

 

Next up: And this is WHOPPER of importance:

 

HOW ARE you specifcing the workgroup file?

 

There are two ways to do this:

 

#1

Use the work group manager, and browse to and select a work group file. ONCE you do this, then any and all launching of ANY access file mdb (or even a  accDB) will now launch the logon dialog, and prompt you (but this is assuming that the workgroup file has a password on the Admin account – if no password on the admin account, then in effect you disabled the workgroup logon prompt. (you will NOT get a logon prompt – even if other user accounts exist in the workgroup file)

 

#2

You use a shortcut to specify the workgroup file. This is “in general” the preferred setup, since every other access file you click on to launch will thus use the “built in” or “default” workgroup file, and no prompting for a logon will occur.

 

So this setup is nice, since only YOUR applcation will display a logon, but everything else will not.

 

So this limits the logon prompt to your “secured” application. (because you included a workgroup file specified in the shortcut. (and the Admin in that workgroup has a password – and BOTH of these things must be in place for the logon to appear.

 

So to be clear:

Logon on with “different” user does NOT overwrite the password, but it will certainly THEN result in Access displaying the LAST user logon.

 

So simply logging on does NOT overwrite anything in terms of the password. Simply logging on will NEVER change the password, and it cannot.

 

So you want to check + determine if your shortcut is specifying a workgroup file – and make sure you are using the correct one

 

I find that access tends to create several workgroup files, and I tend to seen one created in

 

c:\users\YourUserName\AppData\Roaming\Microsoft\Access\system.mdw

 

And if you installed access more than once you will see

 

System1.mdw

Ssytem2.mdw

System3.mdw

 

In other words, new installs of Access (or later versions) work VERY HARD to NOT overwrite a workgroup file – even the built in default one.

 

So what I would do (and it will take some time), is do a search from root (c:\) for *.mdw. Find all occurrences of the mdw files on your computer. (turn on view of hidden files, and system files before  you search).

 

It not 100% clear if you are just (were) specifying the user and password (or just user) in your shortcuts, or you were also specifying a full path name to the mdw workgroup file.

 

If you do specify a workgroup file that does NOT have a password for the admin account, then you do NOT get any logon prompts. And EVEN if you specify the /user in the shortcut, you will STILL NOT get a logon prompt if the admin account does not have a password.

 

As noted, if you use the workgroup manager to specify a workgroup file, then for any and all applications you launch, you will ALWAYS get a logon prompt. (Assuming that admin has a password).

 

So, so far with what you stated:

 

You either changed the workgroup file using the workgroup manager, OR YOU changed the workgroup file to be used in your shortcut.

 

Either way:

No, just typing in users and passwords will NEVER overwrite the password. (it does “remember” the last logon however as I pointed out, but you still always get a prompt for password since the dialog asks for both user + password).

 

So you have to find the workgroup file you were using before.

 

And I would just try clicking on the mdb file (without the shortcut), since it “amazing” how many times I seen files not secured correctly, and simply using a different workgroup file, or not using the one supplied allowed me full rights to the application anyway.

 

So you need to find the original workgroup file, and you need to find what the shortcut is (find a working computer – see what the shortcut is that specified the workgroup file).

 

Regards,

Albert D. Kallal (Access MVP, 2003-2017)

Delete a document in Word 2016 - Microsoft Community

geraldhouthoofd say:

Delete a document in Word 2016

How do I delete a document in Word 2016? There must be a simple way. Any help is welcome.

jay freedman say:

You don't delete a document "in" Word. You go to the folder where the document's file was saved, and delete that. You can do that in the File Explorer, or in any of the file management dialogs (Open, Save As, etc.) within any Office program.

If your question is about removing a file's name from the Recent Documents list on the File > Open page, right-click the file name and choose "Remove from list". That does not delete the document's file from the drive; you can still go to that folder and open the document.
