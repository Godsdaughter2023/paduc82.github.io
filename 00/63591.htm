The controller frequently disconnects from the adapter

  2. If so, isn't File History unsafe? If not, what's the point of keeping File History turned on while my external drive is disconnected (except for saving a few mouseclicks every time)?

spiritx say:
Hi,

Don't turn File History off when you disconnect the external drive.

"Changing the size of the local cache: File History uses a small amount of space on the local drive to store versions of files while the File History target drive is not available. If you create a lot of versions of files while disconnected or stay disconnected for longer periods of time, you may need to reserve more space on the local drive to keep all versions. Note that the versions stored in the local cache are flushed to the external drive when it becomes available again."

If you turn File History off then the local cache will not be updated. However when you reconnect the
drive and turn File History back on the File History will still be updated with the current file changes,
However there will not be updates for files that have been changed more than once which would have
been in the cache if active.

Why just not leave File History on when the external drive is disconnected? Then the cache helps
to ensure there should be copies of files changed during the periods when the external drive is
not connected. If there have been so many files changed that the cache is not large enough then
files are kept in the FIFO method (first in first out) so the File History will be updated with the most
current versions of the files.

Protecting user files with File History
http://blogs.msdn.com/b/b8/archive/2012/07/10/protecting-user-files-with-file-history.aspx

Hope this helps.

spiritx say:

Hi,

You can set the cache up to 20% of the size of the drive.

If you make file changes which exceed the capacity of the cache then the first/oldest versions in
the cache would be lost in favor of the newest/most current. So if you made 4 revisions to your
files and there was only space in the cache for 2 revisions then those last 2 would be the ones
saved in the cache. And those 2 would be written to the external drive when it is reconnected.
The 2 removed from the cache would be lost unless there were other backup methods in play.

Again there is really no reason to turn File History when the external drive is disconnected and
every reason to leave it enabled. File History is a background task and should not cause any
noticeable performance hits.

If you turn File History off then later reconnect the external drive and enable and run File History
the only versions of the files written to external drive is the current ones. Those revisions of files
changed but not saved in some manner would be lost.

One point : Some programs make a backup, such as a .bak file, when a change has been made. If
such a backup where written to a folder covered by File History then that file version would also
be saved. Which provides even further redundance in the methods.
