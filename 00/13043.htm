jllatham say:

As I said, you can't really do that with a formula.  The best way would be with some VBA code associated with the worksheet involved (where you want that formula).

BUT if you are willing to intentionally inject an error into the workbook/worksheet you can intentionally setup a 'circular reference' situation.

For this rather bad advice solution, assume that your formula above is going into cell G4 on the sheet, and you would enter this formula into G4:

=if(day(today)=10,SUMIF(Sheet2!$A$4:$A400,Sheet1!$A34,Sheet2!$D$4:$D400), G4)

Notice that the last parameter is pointing to the cell the formula is in and this creates what is known as a circular reference which is an error condition in Excel.  But it is one that you can set up to permit by going to the [File] tab and then choosing "Options" and go into the "Formulas" group and then check the box next to "Enable iterative calculation".  I would set the Maximum Iterations value to some very low number like 2 or 3 or even just 1.

The risk here is that in setting up formulas in the workbook later on that you accidentally set up a circular reference.  You won't get any alert that you have done that, which could make for a really bad day.

ken sheridan say: Your problems stem from the fact that you have not designed a valid relational database, but what amounts to a set of spreadsheets masquerading as relational database tables.  As well as introducing redundancy by repeating the same data you are also doing what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

In broad outline the model for what you are attempting would comprise the following tables:

Courses
….CourseID  (PK)
….CourseName

Holes
….CourseID  (FK)
….HoleNumber
….Par

The primary key of this table is a composite one of CourseID and HoleNumber.  It might have other columns representing attributes of the hole, e.g. Yardage.

RoundsPlayed
….RoundID  (PK)
….CourseID  (FK)
….DatePlayed
….RoundNumber

The final three columns of his table are a composite candidate key, so should be members of a single unique index.  The  RoundNumber column will not be necessary, however, if  the only one round is ever played on any one day.

HolesPlayed
….RoundID  (FK)
….HoleNumber
….Score

The primary key of this column is a composite one of RoundID and HoleNumber.

With this correctly normalized set of tables the interface for entering data would be a RoundsPlayed form in single form view, within which would be a HolesPlayed subform in continuous forms view, linked to the parent form on RoundID.  Within the parent form would be a combo box bound to the CoursesID foreign key column, set up in the usual way to show the referenced course names.  In the AfterUpdate event procedure code would first save the parent form's current record, and then execute an INSERT INTO statement to insert rows into the HolesPlayed table.  The SQL statement would return the RoundID value from the current record and the HoleNumber values from the rows in Holes where the CourseID is that of the CourseID selected in the combo box.  After executing the SQL statement the subform would be requeried to show the inserted rows, ready for you to enter the scores.  Other data  such as the par for the hole and any other attributes of the hole which are represented by other non-key columns in Holes can be shown in the subform in unbound controls.  I think you'll find that Scott's blog illustrates how to do this.

If you are entering data for a round played at a course not currently represented in the Courses table a new row can be inserted into the table by tying its name in the combo box and using code in the control's NotInList event procedure to insert a new row.  You'll find examples in NotInList.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file includes examples for transparently inserting a row into the referenced table (Courses in this case) or for opening a form in dialogue mode and passing the new name to the form, in which data can then be entered into controls bound to any other non-key columns, e.g. the course address, before returning to the main RoundsPlayed form.  This form would include a subform for entering the holes for the new course, the subform being linked to the parent form on CourseID.
