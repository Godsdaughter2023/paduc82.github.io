tonywilliams1944 say:

Using Access database online

scott mcdaniel say:

In short - no, there are no free resources which would allow you to share your Access database online. The only "online" Access experience I'm aware of is www.eqldata.com, which is most definitely not free.

If you have only a handful of users, and if those users will NOT use the database at the same time, you can setup a machine to allow remote access. Users could then use the Microsoft Remote Desktop Services (RDS) client to connect to that machine and use the database. Note that each connecting user would need a fully licensed copy of Office matching the same version as is being used on the "host" machine, in order to meet licensing requirements.

That sort of scenario rarely works well, however, and often you'll find that you need to setup an RDS server. This can be costly, of course, and if you're not able to rent web space then you certainly don't have the funds to do this.

Access 2013 introduced the concept of Web Apps, which can be run from any browser. You do have to have this hosted somewhere, and that costs money, of course. Microsoft provides hosting at a reasonable rate, or you could apply to them to have your organization deemed as "charitable", at which point they will provide you hosting at reduced rates. 

Note that if you've already created your Access database, then you will have to rewrite it to be able to use it in the Web Apps environment.

scottgem say:

Another option would be to use a remote access tool like TeamViewer. Teamviewer offers a free account for non commercial use, so this would qualify. Similar to Scott McD's suggestion, you would have to setup a PC in the organization's office that would remain on and connected to the Internet. It would be setup for unattended remote access. The users would be given the Computer name (set in TeamViewer) and password.

This would only allow one user at a time to remote in. But once connected, they could then run the Access app as if they were sitting at that machine.

ken sheridan say: The following queries are from my demo.  The first is the most efficient method, and joins two instances of the same table, Transactions.  This query does not return an updatable result set, however, so is not suitable as the RecordSource of a form in which you need to add or edit rows.  Note how the primary key TransactionID column is brought into play as the tie-breaker in the event of two or more transactions being on the same day.

SELECT T1.CustomerID, T1.TransactionDate, T1.TransactionAmount,
SUM(T2.TransactionAmount) AS Balance
FROM Transactions AS T1 INNER JOIN Transactions AS T2
ON (T2.TransactionDate<=T1.TransactionDate)
AND (T2.TransactionID<=T1.TransactionID OR T2.TransactionDate<>T1.TransactionDate)
GROUP BY T1.TransactionDate, T1.TransactionAmount, T1.TransactionID, T1.CustomerID
ORDER BY T1.TransactionDate DESC, T1.TransactionID DESC;

The second query does return an updatable result set, by calling the VBA DSum function.  This is far less efficient than the above query, but, unless the table is very large, performance should be acceptable.

SELECT TransactionID, CustomerID, TransactionDate, TransactionAmount,
CCur(DSum("TransactionAmount","Transactions","(TransactionID <= " & [TransactionID] & "  
OR TransactionDate <> #" & Format([TransactionDate],"yyyy-mm-dd") & "#)   
AND TransactionDate <= #" & Format([TransactionDate],"yyyy-mm-dd") & "#")) AS Balance
FROM Transactions
ORDER BY TransactionDate DESC, TransactionID DESC;

Note that in my case the rows are returned in descending date order, but you simply need to amend the ORDER BY clause to return them in ascending date order.

ken sheridan say: I'd go a little further than John, and say that, if the ID values are not in any meaningful order, then the table contains no information as to the order of transactions within the day.  Consequently, in the context of the database, the order in which the records per day are returned is immaterial.  If, in the real world, the order in which the records per day are returned is significant, then the database is deficient in this respect, and would need to record the date and time of each transaction rather than merely the date.

The queries from my demo, which I posted earlier, require the TransactionID column to have distinct values only,  the actual values are immaterial to the functioning of the query, which returns the rows in descending TransactionDate order, with multiple transactions with the same date returned in an arbitrary order determined by the autonumber TransactionID column.  The balance returned is per transaction.
