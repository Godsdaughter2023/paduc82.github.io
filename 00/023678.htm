General - Why a form or report should be based on a query rather than - Microsoft Community

justinkase42 say:

General - Why a form or report should be based on a query rather than a table

I am working with a client who has a database that he first created on DBASE IV.  He has several forms and reports that are based on a table rather than a query.  I know there is a technical reason why it is better to have forms and reports based on queries.  I think it is related to some kind of indexing or some kind query definition that will enable a query based form/report to run faster than one based on a table but I can't recall the precise technical reason. 

john w. vinson say:

It's not that big a deal. Access will create a hidden local query anyway if you specify a sort or filter.

The main advantage of a query is that you can get better efficiency by basing the form on a subset (often a small subset, perhaps even a single record) of the data rather than having the form potentially display all the records in the table, including thousands that nobody has any interest in at all.

John W. Vinson/MVP

peeshgooyy say:

FROM A BOOK NAMED ACCESS 2007 THE MISSING MANUAL::

 

UP TO SPEED
Doing the Heavy Lifting with a Query

"The most obvious way to build a report is to base it on an existing table. However, you can also create a report on top of aquery. This approach lets you use some heavy-duty filtering or sorting on your records before they reach the report. It also makes sense if you want to create a report that uses information from more than one table.

For example, imagine you decide you want to create a product list that includes additional details from another table (like the category description from the ProductCategories table). Although you can create this report from scratch, it often makes more sense to structure your data with a query first. That way, you can reuse the query for different purposes (like editing), and you can change it any time.

In this example, the first step is to create a query that joins the Categories and Products table. Then, you save this query, select it in the navigation pane, and choose Create >> Reports >> Report to create a report that's based on the query. You can then follow the normal steps to perfect your report."

gina whipp say:

I think you missedJohn's point...

Scenario 1

Table has 10 fields but only two show on the report.  Why base the report on the table which will *pull* all ten fields when only two are needed.

Scenario 2

Table has ten field but 100,000 records only 100 of which are needed for the report.  Better to runa query with criteria so need to *pull* the entore subset of data.

Both scenarios will allow for the form/report to load faster.
