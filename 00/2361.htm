There is NO reason I can think of that using the x64 or x32 bit version of Access will make ANY difference.

 

In other words, the fact that some users’ runs x64 and MOST run x32 means your software should work just fine on either platform. It is recommended you run x32 office, even on an x64 box.

 

> but now I think I might need to downgrade this database to make it universally compatible.

 

No, the above is 100% wrong and incorrect. I not sure where you found such information, but it is 100% wrong.

 

> Is there any way to convert it?

 

You don’t have to, so once again wrong question, wrong consulting, and wrong information.

 

 

So in summary:

Your information and consoling are incorrect. Those with Access 2013 x32 or 2013 x64 should have no trouble running your applications.

 

If other users are receiving an incompatible file type, they are likely running a PREVIOUS edition of Access, and that error message HAS NOTHING to do with them running x32, or you running x64.

 

There are only TWO issues that could cause problems for you with your x64 bit edition:

 

#1 – you are calling and using some windows API. However, MOST posted examples are x32, so once again this is HIGHLY unlikely.

 

#2 – you are compiling and attempting to distribute an accDE (a compiled version of your accDB. In this case you MUST compile with the same version you deploy to. However EVEN in this case, you do NOT need to convert your accDB to x32 or x64.

 

#3 – well, this is a LONG shot, but you using a x64 bit activeX control of FEW exist, and these would NOT work with x32 (but this is grasping straws possibility).

 

So no converting here at all is required. The error message suggests that the user is running a previous version of Access, and such error messages will occur REGARDLESS of using access x32 or x64.

 

So do feel free to follow up, but your application should run just fine on either platform and do so without changes. In fact access will automatic detect that your VBA needs to re-compile and do so on the fly. Of course if you distribute a compiled application (accDE), then this on the “fly” automatic re-compile cannot occur since a compiled accDE does not include source code. However, you can fix this issue by compiling the application with an x32 version before distribution to x32 users (and again, this is NOT likely your case).

 

I also DO suggest that if you KNOW ahead of time that the users will be running Access x32 (2013), then it is a good advice to do a de-compile as that cleans out the x64 compiled code. However, really at the end of the day, you should not have to do anything when jumping from the x64 bit edition of Access to the x32. I had a few cases where I did have to do a de-compile, but this is NOT the norm or a requirement.

 

So in effect, the above is likely rather good news for you.

 

The MOST likely issue is those users are receiving this error because they are running a previous version of Access and thus this has NOTHING to do with them running x32 or x64 versions of Access.
