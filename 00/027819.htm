ken sheridan say: Do not be confused by the superficial similarity between a table in datasheet view in Access and an Excel worksheet.  Relational databases and spreadsheets are completely different conceptually and operate in entirely different ways.  Tables are sets and by definition a set has no intrinsic order.  The concept of row numbers is meaningless in a database.  Every datum (atomic value) in a relational database is guaranteed to be logically accessible by resorting to a combination of table name, primary key value and column name.  This is the Guaranteed Access Rule (Codd's Rule #2).  Referencing a value by the intersection of a column heading and row number, as in a spreadsheet, is not how it's done in a relational database table.

As far as I can see you seem to be trying to compute balances per group.  There are a variety of ways of doing this by means of a query in Access, some which result in an updatable result table, some of which result in a read-only result table.  What is common to both, however, is that the computed value is not in a column in the base table, but returned in a computed column in the query's virtual result table.

You'll find example of queries which do this in Balances.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

bill manville say:

You could create a scheduled task to open your workbook once a day.

The workbook could contain an Auto_Open macro that would update the links and then save the workbook and quit Excel.

If the source of the link is a named range then the link can refer to that name and if the range moves within the source workbook the link will still work; otherwise, the Auto_Open macro would probably have to open the source workbook, locate the relevant cell and modify the link formula.
