Digital sign and publishing ACCDE best practises

We are publishing applications for Access Runtime 2010. We like to switch from MDE to ACCDE.

Since the new format of databases ACCDE, it's not possible to sign the VBA code and Access will show message that the file is not trusted.

As work around we setup the trusted location per user. This does not look very professional. 

Working with ACCDC files (Pack & Sign) does not do the trick since the ACCDC is just a signed package. When opening the ACCDC file the user must save the ACCDE to a particular location. The trusted location is not set. And again Access will show the message the file is not trusted.

scott mcdaniel say:

You MUST setup Trusted Locations with newer versions of Access. As you've found, signing a deployed Access application simply signs the entire package, and not the contents of that package. The only way to insure your code will run correctly is to setup a Trusted Location on the end user machine.

You can do so with code, if you'd like, or if you're using the Package Wizard you can include the code there. This article tells you how:

https://www.experts-exchange.com/articles/10805/Overcome-the-Trust-Center-nuisance.html

You can also create a .reg file that can be merged with the user's Registry, but of course that would force you to install the database to a specific location on each machine (which is not necessarily a bad thing).

The best way to deploy, IMO, is to use a true Windows installer and create your registry settings using that installer. Most Windows installers worth their salt allow you to easily work with the registry, so you could create the necessary registry keys and populate them with the values used during installation.

The Registry keys you must set are as follows, all of which are found under HKEY_CURRENT_USER\Software\Microsoft\Office\14.0\Access\Security\Trusted Locations. Under that registry entry you'd add a new Key, which must be the unique name for your new Trusted Location. Under that entry, you'd add the following keys:

Date

This will add the location found at the "Path" entry as a Trusted Location, and your end users will be able to run your Access application correctly.

julie fka profjulie say:

The way phone updates work:

  • Microsoft prepares & thoroughly tests an OS update for windows phone
  • Then Microsoft lets the OEMs (the phone manufacturers) know that an update is available
  • The OEM, if they feel so included, takes the update from AT&T and adds their own firmware and bits to it.  An OEM may decide NOT to update all the phones it sells that might be eligible for an update. So some customers may eventually get an update, while others who have different phones may not.
  • Then if a phone is privately branded to a carrier or mobile operator, the carrier/mobile operator is provided a version of the update (that includes the OEM parts). The carrier/mobile operator may choose to update or not update the various models of phones it sells.  For the phones it decides to update, it spends some time adding their bits to the update and testing the updated OS on their wireless system.
  • After the OEM & Carrier/Mobile Operator has finished up all their testing of the update, they release the update to their customers in stages. When an update is ready to be installed on a customer's phone, an update message will appear on the phone's home screen or, the customer can check that "check for updates" screen.

If you have clicked the "check for updates" screen and get the message that your phone is up-to-date, it could be because:

  1. Any available update is in the OEM's hands and the OEM is working on it
  2. Any available update is in the carrier/mobile operator's hands and the carrier/mobile operator is working on it
  3. Your phone model is not eligible for an update - either the OEM or the carrier/mobile operator have decided not to update your phone model
  4. Based on the OS that is currently running on the phone, there are no updates for the phone at all.

You have not provided much information about what version the operating system is running on your current phone, nor have you shared with us if your phone is privately branded to any carrier/mobile operator, so we have no idea as to the reason your phone hasn't received any updates.

You can review the Windows Phone 8 update history page to see the updates that Microsoft has prepared and made available to the OEMs and Carriers/Mobile Operators.

You can check the version of operating system that is running on your phone by going to Settings > About > more information > What OS version do you see?

If your phone is not running the latest version of OS, you might need to check with your OEM and carrier/mobile operator to find out what their plans are.

rdcinhou---01 say:

Oh...the Accuweather saga...

 

First...the "script" errors are from the GPS component. The default Accuweather setup tries to find your location using the GPS, but the GPS hardware and software are so flakey that it won't find a location 99.5% of the time.  I've had the official Nokia GPS antenna fix, loaded the v20 firmware, performed a total hard reset, reformatted E: drive memory and reinstalled everything from scratch and the GPS is still not up to expectations.

 

So...what do you do?  Go to Options->Add new location and enter just a city name.  It will connect to the network and give you a combo box to pick your location from.  I have had problems with entering zip codes or City-and-state, evidently the software isn't smart enough to parse these correctly.  By the way, be sure to use the stylus to select your city as the size of the box for selection is not "finger-friendly".

 

Once you have a city added, select Options->Manage Locations, select your city and select "Set deflt." to ensure that your city location is the Default one, select "Done." and then we're almost done.

 

What this has done is take the poor-quality GPS out of the equation.

 

Now select Options->Refresh data and you'll see the now-too-familiar "Loading..." but it should now actual find some data (at least 90% of the time).

 

The reason that the data doesn't always come is that Accuweather pulls this data from their Internet site, which I've found out by trying to access it on my PC, isn't always available online.

 

Did Accuweather underestimate just how much of a load all of the N97 owners trying to pull data from their website would inflict on their servers?

 

Oh-and-by-the-way, the Accuweather application was also designed to show you ads.  These appear at the bottom of the screen in portrait mode only, so there is probably an ad-server out there that it is trying to connect to as well.

 

Something else....when Nokia did their v20 firmware "enhanced memory management" what this means is that the constantly running homescreen widgets may be removed from running if another application needs more memory, so what will happen is that after you exit that application, Accuweather will be restarted and you'll see the "Loading..." indication on the homescreen until it can reconnect and refresh.

 

Sometimes it appears to help if you open it and select Options->Refresh data. 

 

Now that you see all of the pieces of this application and what it has to go through to work, you can understand why it appears to be so flakey.
