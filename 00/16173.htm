qa guy commelec say:

Example of running sum using subquery --

SELECT YourTable.Article, YourTable.Date, YourTable.Price, (SELECT Sum([XX].Price) FROM YourTable AS [XX] WHERE [XX].Article = YourTable.Article AND [XX].Date <= YourTable.Date) AS RunningSum
FROM YourTable
ORDER BY YourTable.Article, YourTable.Date;

Running without subquery (crosstab not allow subquery) --

SELECT YourTable.Article, YourTable.Date, YourTable.Price, Sum(IIf([YourTable_1].[date]<=[YourTable].[date] And [YourTable_1].[Article]=[YourTable].[Article],[YourTable_1].[Price],0)) AS RunningSum
FROM YourTable, YourTable AS YourTable_1
GROUP BY YourTable.Article, YourTable.Date, YourTable.Price
ORDER BY YourTable.Article, YourTable.Date, YourTable.Price;

ken sheridan say: The most efficient means of computing balances is with a join:

SELECT S1.Material, S1.Rank, S1.[Part Class], S1.[Future Spend],
SUM(S2.[Future Spend]) AS Balance
FROM [Sorted Spend] AS S1 INNER JOIN [Sorted Spend] AS S2
ON S2.Rank<=S1.Rank
WHERE S1.[Part Class] = "Raw"
AND S2.[Part Class] = "Raw"
GROUP BY S1.Rank, S1.Material, S1.[Part Class], S1.[Future Spend];

What are you trying to return as a percentage, however?  Is it the running balance as a percentage of the total spend regardless of Part Class?  If so:

SELECT S1.Material, S1.Rank, S1.[Part Class], S1.[Future Spend],
SUM(S2.[Future Spend]) AS Balance,
FORMAT(SUM(S2.[Future Spend])/
    (SELECT SUM([Future Spend])
     FROM [Sorted Spend]),"PERCENT") AS [Percent Spend]
FROM [Sorted Spend] AS S1 INNER JOIN [Sorted Spend] AS S2
ON S2.Rank<=S1.Rank
WHERE S1.[Part Class] = "Raw"
AND S2.[Part Class] = "Raw"
GROUP BY S1.Rank, S1.Material, S1.[Part Class], S1.[Future Spend];

Or is it the running balance as a percentage of the total spend for Part Class 'raw'?  If so:

SELECT S1.Material, S1.Rank, S1.[Part Class], S1.[Future Spend],
SUM(S2.[Future Spend]) AS Balance,
FORMAT(SUM(S2.[Future Spend])/
    (SELECT SUM([Future Spend])
     FROM [Sorted Spend]
       WHERE [Part Class] = "Raw"),"PERCENT") AS [Percent Spend]
FROM [Sorted Spend] AS S1 INNER JOIN [Sorted Spend] AS S2
ON S2.Rank<=S1.Rank
WHERE S1.[Part Class] = "Raw"
AND S2.[Part Class] = "Raw"
GROUP BY S1.Rank, S1.Material, S1.[Part Class], S1.[Future Spend];

In the above I've assumed that all returned columns are from the Sorted Spend table.  If so there is no need to join this to the Raw Total Spend table as the restriction on the Part Class value being 'raw'  is on the join column.

However, relying on an autonumber column to give a set of sequential values in order of the insertion of the rows into a table is living dangerously.  An autonumber is intended to guarantee one thing and one thing only, as set of distinct long integer number values which are semantically arbitrary.  You could well find that at some time a value which is not in sequence will be inserted.  If you wish to number rows sequentially in order of the insertion of the rows into a table, then you should not use an autonumber column, but should compute the value when the row is inserted.  If you merely need a set of values guaranteed to be in order of the insertion of the rows into a table then the simplest way is to date/time stamp each row by means of a column of date/time data type with a DefaultValue property of Now().

You'll find examples of queries for computing balances in various contexts as Balances.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In the same OneDrive folder you'll find an illustration of ways to compute serial numbers, both distinct and per group, as CustomNumber.zip, and an illustration of how to number or rank rows dynamically in a query's result table as RowNumbering.zip.

ken sheridan say: I think I referred you to my CustomNumber demo in another thread; this illustrates, amongst other things, how to compute sequential numbering per group.   But you first need to consider whether you should have a structured key like this.  General speaking structured keys as a single column are inappropriate as the column is storing multiple attributes, violating Codd's Guaranteed Access Rule.

If the prefix 'LH' is a constant then it should be stored in a single row in a single column in another table.  If not, in a column in the current table.

If the year value can be derived from another column of date/time data type in this or another table, it should not be stored but computed at runtime.  If not, the year should be stored in a separate column, e.g. CaseYear, and the serial number in a separate column of integer number data type.  The two columns should be included in a single unique index.

To return the composite value is a simple task of concatenating the three values, formatting the serial number to three digits.

ken sheridan say:

..............however what I am trying to do is just figure out a way that I can make sure that a number is never duplicated.


In this case, if you store the sequential number as a separate column, and derive the year from the Date Of Case column's value you cannot do as would normally be done and include the two columns in a unique index.  So you have two alternatives:

1.  One way to do it would be to apply a CHECK CONSTRAINT to the table which prevents there being more than one row where the serial number and the year of the DateOfCase column's value are the same.  This cannot be done via the table design interface, however; it has to be done by executing a DDL statement.  DDL (data definition language) is a part of the SQL language in which queries are written.  It's not difficult, but I would not expect an inexperienced developer to be familiar with it.  I can, if you wish, walk you through it.

2.  The other way is to break the rules and include a CaseYear column in the table, assigning its value when a row is inserted via a bound form with:

Me.CaseYear = Year(Me.[Date Of Case])

You can then include the CaseYear and the column containing the serial numbers per year in a unique index.  Automating the insertion of the next number per year is relatively simple.  As this does break the rules, however, it is not an entirely bulletproof solution, which is why the rules exist in the first place.  The 'LH' prefix is not relevant to this issue.  Having a single column as a 'structured key' is also not bulletproof of course as in both cases it would be possible for the year of the date and the year part of the key to be inconsistent with each other.

steve rindsberg say:

Audio recorded into a presentation is saved as a series of audio objects, one per slide. 

This does a pretty good job of making sure that the recordings match up with the slide changes, but doesn't allow for continuous audio; there'll be a break between slides.  No biggie if you're just recording narration, but a problem if you want a smooth musical background.

Another possible problem is that PPT records at rather low quality and no longer offers the option of selecting a higher quality recording as it used to.

There are ways of having an existing audio track play through an entire presentation, but PowerPoint isn't able to keep the slide changes synchronized to the audio accurately when you do this.

If you have Camtasia, I'd use it instead. Record video as the slide show plays, then you can add audio later, even mix music b/g and narration tracks, and it'll all stay in synch when exported to a format that makes YouTube et al happy.
