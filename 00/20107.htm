What you're describing is a document containing mergefields, not formfields.

Mergefields are used in mailmerge main documents connected to a data source. Upon opening a mailmerge main document, the user should get an SQL prompt, to which the user should answer 'Yes'. joeu2004 say: fnoots wrote:

First, I believe you are computing the retail price ("sell price") incorrectly.

Apparently, you want RP - RP*t - RP*r = BP, where RP is the retail price, BP is the base price, t is the sales tax rate (GST), and r is the rebate rate [1].  So mathematically:

RP = BP / (1 - t - r)

Thus, P = RP - RP*t - RP*r - C, and P = BP - C, where P is profit and C is costs.

Note that each component -- RP*t, RP*r, etc -- should be rounded separately, since they are actual payments or itemized amounts.


fnoots wrote:

The problem is I can not get the profit without rebate & GST to equal the same profi with Rebate & GSTt [...]. The profit columns don't match.


Second, it is unclear what difference in profit calculations you are referring to because you demonstrate only one calculation, namely the per-unit profit.

Nevertheless, we should expect some differences due to rounding, since GST and, apparently, the rebate amounts are based on the retail price for the total order (e.g. 2 boxes).  That is not the same as multiplying the per-unit GST and rebate amounts times the total units in the order.

The size of the discrepancy will depend on the magnitude of per-unit base price and on the tax and rebate rates.  It is unavoidable if you want the per-box price to be the per-unit price times the number of units per box.  For example:





The second image shows the formulas.  Note the careful use of rounding.  In F2, G10 and H10, it is needed to correct normal arithmetic.  In the other cells, it is used to correct anomalies that arise in Excel arithmetic:  infinitesimal differences that propagate into dependent calculations with possible unexpected consequences.

The key point is:  note the differences between rows 7 and 9.  Again, this is the result of how the GST and rebate rates are applied, namely:  per-box and per-order, not per-unit times the number of units.

However, also note that F10, G10 and H10 are the same per-unit profit when rounded.  This will not always be the case; but we can expect them to be "close" relative to the magnitude of the per-unit price.

If these are the differences that you see, they are a necessary evil if you want G2 and H2 to be multiples of F2, which I presume you do.

rick rothstein, mvp - excel say:

Because the array created by the assignment you indicated is a two-dimensional one and since for a column, that second dimension is fixed at 1, I don't believe you will be able to do what you want directly because in a dynamic array, only the last dimension (the one fixed at 1) can be changed while preserving the existing contents of the array. I can think of a kludge solution to do what you want though. Here is a function that you would pass in the ranges (qualified by their worksheets) as a delimited list and the function will return their content (in the order specified) as a single two-dimensional array...

' This function works on single columns of data only
Function ComboArray(ParamArray Data() As Variant) As Variant
  Dim LastRow As Long, Element As Variant, TempSheet As Worksheet
  If IsEmpty(Data) Then
    ComboArray = Split("")  'Return an empty **array** from the function
  Else
    With ThisWorkbook.Worksheets.Add
      For Each Element In Data
        Element.Copy .Cells(LastRow + 1, "A")
        LastRow = .Cells(.Rows.Count, "A").End(xlUp).Row
      Next
      ComboArray = .Range("A1:A" & LastRow)
      Application.DisplayAlerts = False
      .Delete
      Application.DisplayAlerts = True
    End With
  End If
End Function

This function is not limited to just two columns of data, so you could call it from your own macro for, say, three columns of data like this....

Sub TestTheComboArrayFunction()
  Dim X As Long, Contents As String, MyArray As Variant
  ' Get the array from individual columns
  MyArray = ComboArray(Worksheets("Sheet1").Range("A1:A10"), _
                       Range("B6:B10"), _
                       Worksheets("Sheet3").Range("C11:C17"))
  ' Let's look at the array that was created
  For X = LBound(MyArray) To UBound(MyArray)
    ' Note that the array that was created is two-dimensional and must be called as shown
    Contents = Contents & MyArray(X, 1) & vbLf
  Next
  MsgBox Contents
End Sub

Note 1: The second range was passed without a worksheet reference, so the ActiveSheet is used.

Note 2: The specified ranges may start at any location within their respective columns.

andreas killer say:
Under normal conditions: no.

The problem is you can only change the last dimension of an array with the keyword PRESERVE, i.E.:

'Basic initialize
ReDim Data(1 to 100, 1 to 10)
'or
Data = Range("A1:J200")

'Possible (add columns)
Redim Preserve Data(1 to 100, 1 to 20)

'Not possible (add rows)
Redim Preserve Data(1 to 200, 1 to 10)

But you can use a trick, use the Transpose method from the Worksheetfunction object to exchange the dimensions:

Data = WorksheetFunction.Transpose(Data)

After this your array is Data(1 to 10, 1 to 200) and now you can use

Redim Preserve Data(1 to 10, 1 to 500)
Data = WorksheetFunction.Transpose(Data)

to get an array Data(1 to 500, 1 to 10)
