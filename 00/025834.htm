Creating a time and user stamp anytime information is entered into a - Microsoft Community

ehowarth say:

Creating a time and user stamp anytime information is entered into a cell

Hello, I have a spreadsheet in which there are cells in range A7:A47. When someone enters information into one of these cells I would like their username to automatically populate in another worksheet (named "6") in the same cell and a timestamp in the corresponding cell in the B Column on "6".

For example. If add data into cell A10 on the activesheet, my username will populate in cell A10 on "6" and a date/time stamp will populate in cell B10 of "6"

Any ideas?

myall_blues say:

I'm pleased it worked for you!

The simplest way is just to add another code block for data being entered into the area of interest (B7:B47 in this case). See modified code below.

There's a more efficient way to do this since A7:B47 is a contiguous range, but since you have indicated you will extend this to the whole sheet I think this is the easiest way.

If this does what you want perhaps you could mark the thread as answered to assist others.

Regards

Murray

Option Explicit
 Private Sub Worksheet_Change(ByVal Target As Range)
     Dim cRow As Long
     Dim ws As Worksheet
     Set ws = ThisWorkbook.Sheets("6")
     '
     ' Date entered in A7:A47
     '
     If Not Intersect(Target, Range("A7:A47")) Is Nothing Then
         cRow = Target.Row
         With ws
             .Range("A" & cRow).Value = Application.UserName
             .Range("B" & cRow).Value = Now
         End With
     End If
     '
     ' Date entered in B7:B47
     '
     If Not Intersect(Target, Range("B7:B47")) Is Nothing Then
         cRow = Target.Row
         With ws
             .Range("C" & cRow).Value = Application.UserName
             .Range("D" & cRow).Value = Now
         End With
     End If
 End Sub

    jay freedman say:

    The quotation marks aren't the problem. What is a problem is that the paragraph's range includes the invisible paragraph mark, what appears as ¶ when you display nonprinting characters. That character is not valid in file names.

    Change the start of the code like this:

        Dim filename1 As String
        filename1 = ActiveDocument.Paragraphs(5).Range
        filename1 = Left(filename1, Len(filename1) - 1)

    The last of these statements removes the paragraph mark from the end of the string.

    By the way, any time you debug code and hover over a variable to see its value, a string variable will show double quote marks box is limited to one line.

    john w. vinson say: Well, it's generally better practice to keep the data all in the same table with a checkbox, but if you really want to do it the hard way you can.

    You would need two queries in succession: an Append query to copy the data for terminated employees to the Terminated table, followed by a Delete query using the same criteria to delete them from the main table. It's safest to run these using a Transaction; see the VBA Help topic for "How to use transactions in a DAO Recordset". A Transaction will ensure that either both operations get run completely, or if one fails it will roll them back so nothing gets changed.

    ken sheridan say: The problem with doing that is that you cannot enforce relationships between an Employees table and referencing tables such as EmployeeProjects or similar unless you enforce cascade deletes.  This could lose valuable 'historic' data.  If you don't enforce relationships of this nature then you lose the protection of the integrity of future data entered.

    If you really feel it necessary to have two tables, though I can see no advantage whatsoever in this, model the active and terminated employees as sub-types of a type employees, i.e. have an Employees table with a small number of columns representing only those attributes common to both sub-types and ActiveEmployees and  TerminatedEmployees tables, each of whose primary keys is also a foreign key referencing the primary key of Employees ( i.e. one-to-one relationship types).  The relationships with referencing tables can then be with Employees and you can append and delete rows to TerminatedEmployees ActiveEmployees in a Transaction in the way John described.

    However, you are just creating unnecessary complications by doing this.  The result table of a query 'WHERE Active' or 'WHERE NOT Active' is just as much a table as a base table, so what advantage is to be gained by having multiple tables?  I'd think again.

    ken sheridan say: It's apposite that you should mention Excel.  What you have by the sounds of it is not a relational database in the true sense, but an Access file which is mimicking a Workbook.  I'd be surprised if a table which has no relationship with any other table is correctly normalized, and by having multiple tables for employee types data is being encoded as table names.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.
