jaschwa say:

I found that "FIELDPICTURE(16)" changed the text field's format to inches with fractions and units displayed. 15 hides the units.

I put together this macros to update all the dimensions on a page:

                                     

Public Sub SetDimensionInchesFraction()
    Dim shpObj     As Visio.Shape, celObj As Visio.Cell
    Dim ShpNo      As Integer

    Dim DiagramServices As Integer

    DiagramServices = ActiveDocument.DiagramServicesEnabled
    ActiveDocument.DiagramServicesEnabled = visServiceVersion140

    For ShpNo = 1 To Visio.ActivePage.Shapes.Count
        Set shpObj = Visio.ActivePage.Shapes(ShpNo)

        If shpObj.LineStyle = "Dim-Architecture" Then
            shpObj.Section(visSectionTextField).Row(0).CellU(2).Formula = "FIELDPICTURE(16)"
        End If


    Next ShpNo


    'Restore diagram services
    ActiveDocument.DiagramServicesEnabled = DiagramServices

End Sub

karl timmermans [outlook mvp 2012-2018] say:

On a different but slightly related topic purely for the purposes of some added trivia when talking about "naming conventions" - Outlook allows a lot of things that are otherwise unacceptable elsewhere

Some examples:

Outlook custom field names allow the following characters which are illegal characters for MS Access (or as column header names when using the MS Excel file driver for that matter):

. - period
! - exclamation point
` - accent
[ - left square bracket
] - right sqaure bracket
" - chr$(34) - double quote

(this allows for some of the most creative field names mankind has ever seen when mixed in with a penchant for field name lengths that could rival the size of a minature encyclopedia based on things sent into us)

Outlook custom field names will NOT allow some characters that are allowed as valid MS Access field names

_ - underscore
# - number sign
, - comma

Outlook will allow space characters at the end of a custom field name so you can have three field names that "look identical" but represent three separate and distinct fields - i.e. <NewFieldName>, <NewFieldName > (one space char at end), and <NewFieldName  > (2 space chars at end) etc are 3 separate and distinct fields

nathan o. - microsoft support say:

Hi Trisbuffer,

 

Is your form based off of the table directly or is it based off of a query? If it is based off of a query, did you just select the attachment field name or did you expand the attachment field and select <Your Attachment Field Name>.FileData? If you used any of the attributes of the attachment field like .FIleData within your query or if you put a control on the form that is based off of one of these attributes, it will then list each attachment within its own record. So if id 1 has an attachment field with 3 attachments in it, you will see something similar to:

 

ID 1 – Attachment 1

ID 1 – Attachment 2

ID 1 – Attachment 3

ID 2 – No Attachment

ID 3 – No Attachment

 

I would look at your query or form to see how you added the attachment field to the form. If you don’t want to see a record for each attachment, then you need to ensure you only add the attachment field itself and not one of the attributes of that field.

 

The following article talks about attachment fields:

 

Attach files and graphics to the records in your database -http://office.microsoft.com/en-us/access-help/attach-files-and-graphics-to-the-records-in-your-database-HA001213852.aspx

 

And this article talks about querying multi value fields. Multi value fields work very similar to Attachment fields, so when you query them you can see similar results. This article talks about the different results (# of records) you can see depending on how you query these multi value fields.

 

Using multivalues fields in queries -http://office.microsoft.com/en-us/access-help/using-multivalued-fields-in-queries-HA010149297.aspx

 

Hope that helps.
