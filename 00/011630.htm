' timelog Macro
'

'
Dim t As Double
Range("A3").Select
t = Minute(Now())
While t < 61
If Minute(Now()) = t + 1 Then
    ActiveCell.Value = TimeSerial(Hour(Now()), Minute(Now()), 0)
    ActiveCell.Offset(0, 1).Value = Range("A1").Value

I had the same issue. The process below saved the day.

  1. Turn off all internet connections or One note will start syncing before you can complete the recovery

  3. Find: com.microsoft.onenote.mac (In Library containers folder)
    1. Close everything on desktop
    2. Click on Go in the menu bar
    3. Hold down option key, now you should see library click on Library
    4. Open Containers folder
    5. com.microsoft.onenote.mac should be in there (on to step 4)
  4. Open Time Machine
  5. Go to date you know files were there.
  6. Highlight com.microsoft.onenote.com and select recover
  7. Choose replace the existing file
  8. Then go to applications folder for the same date in time machine and recover the Onenote app

  10. Your files will be there but you’re not safe yet!
  11. Make another Notebook call it backup or whatever you want
  12. Copy all other note books contents into the new notebook you just created
  13. Turn on your internet connection allowing one note to sync
  14. Now some of the original files may be gone from original locations but if you copied them into the new notebook = backup they will all be there and you can put them back to their rightful place.

It shouldn't be very hard to write a function called, maybe, CheckForLocking(), in the General section of the form's module, in which you check whether all those controls have been filled in, and if they are (and the user is not an admin), either set the form's AllowEdits property to False or else set the Locked property of all controls to True.  If any of the controls is Null or "", then set AllowEdits to True or set the controls' Locked properties to False.

The question is, what form events would you use to call this function?  You would certainly use the Current event, since you would want to unlock or lock each record as you navigate to it.  But what other events?  You *could* use the AfterUpdate event of each of the 7 controls.  I see a potential usability problem with that, because a user might accidentally set the 7th control to some incorrect value, and immediately become unable to fix it.  I'd be happier, though, using the form's AfterUpdate event, because that would let the user modify any number of control values multiple times without getting locked out -- UNTIL the record is saved.  Once it's saved, it would be locked if th requisite controls are filled in.

Granted, that may not be the behavior you want.  But I do worry about locking a record with invalid values, such that the user is unable to correct them.

ken sheridan say: I would normally handle this sort of thing more selectively than by the blunt instrument of assigning a value to the AllowEdits property.  You can selectively lock/unlock specific controls, whether bound or unbound, by firstly setting the Tag property of only those you want to lock/unlock to something like 'LockMe', then looping through the form's Controls collection, e.g.

Function fncLockUnlockControls(frm As Form, blnIsNew as Boolean)

    Dim ctrl as Contol

    ' loop through tagged controls and set Locked
    ' property to False if at an empty new record or
    ' to True if RecordCompleted = True
    For Each ctrl In frm.Controls
        If blnIsNew Then
            If ctrl.Tag = "LockMe"
                ctrl.Locked = True
            End If
        ElseIf ctrl.tag = "LockMe" Then
           ctrl.Locked = (frm.RecordCompleted = True)
        End If
    Next ctrl

End If

The RecordCompleted  control in the form must be bound to a column of Boolean (yes/no) data type in the table, and it is important that the DefaultValue property of the column must be set to False in the table design, so that all controls will be unlocked when the user navigates to an empty new record in the form.  Otherwise the control will be Null when moving to a new record as, even though a Boolean column cannot be Null, a control bound to it will be Null until a new record is initiated in the form by the user starting to enter data.

In the form's Current even procedure call the function like this:

    Dim IsNew as Boolean

    IsNew = Me.NewRecord
    fncLockUnlockControls Me, IsNew

In the RecordCompleted control's AfterUpdate event procedure call it like this:

    fncLockUnlockControls Me, False

Rather than using a combo box as the bound control for the RecordCompleted column you can use a check box, which is the more usual type of control for a Boolean column.

Is it REALLY possible to be safe on-line? - Microsoft Community

a. user say:

Is it REALLY possible to be safe on-line?

Hello

I  read this today......

"By injecting an iframe or keylogger in a seemingly normal site, a thief can gain access to the visitor’s system, taking it over and copying account information, files, user actions, and anything else it might be interested in."

And this ........

"Cybercriminals succeed by crafting custom, specialized code that broad-based signature-driven tools don’t recognize and employing cocktails of techniques that consciously, laboriously, maneuver around layered defenses."  


It seems as if conventional anti-virus programmes are unlikely to protect us nowadays!


How does an iframe, placed on a 'normal' web page, make one so vulnerable?


Your thoughts, please.

kosh vorlon say:

Hi BDoTJ,

 

An iframe or keylogger or any infection on a website makes one "vulnerable" because first they are either disguised or actived simply by visiting.  You cannot reasonably check in advance to be absolutely certain if any of these things are there or what they are or how they are activated or how to visit the site and avoid them.  If it were possible, then all the precautions we take (such as those listed above would either not be needed or could be cut back or this new method would have been included).  The above response is the best (and virtually only) way to defend against such things.  There is no special tactic beyond what was mentioned there to protect against this (or similar threats) other than using internet options to block so much that you won't be able to effectively view and work on many if not most normal sites. 

 

The standard settings recommended are typically sufficient.  Even if you were to block far more to the point where you would find browsing cumbersome if not annoying or virtually impossible, I'm not sure you'd be absolutely sure to block all such code (but that's just simply too much and would make browsing a nightmare of access problems, page rendering problems, constant approval of prompts when you really have no idea whether or not it's safe to do so without looking at the code, unreadable pages and non-functioning features, buttons, links, and other things).

 

If you don't understand how an iframe itself can ever be dangerous, meaning you don't understand what an iframe is, try starting here: http://www.w3schools.com/tags/tag_iframe.asp and from there to the tutorial at the bottom http://www.w3schools.com/html/html_iframe.asp.  In essence it is a type and/or method of coding, and if present can, like all code on the web page, be used for good or evil - it can help direct you properly or can include malware or send you to another location to be hacked.

 

You can't find out in advance if a page contains such coding.  But once you are there, you can right click on the page and click "View Source" and, after reading the above articles on iframes, look through the source to see if any of the methods or coding techniques or terms to incorporate iframes exist anywhere in the code.  Then you need to see what it does and how it operates (and by then if it was the type to activate on opening the page, you've already been attacked and if it is the type that operates when you do something, then you need to analyze what it does to see if it's dangerous - and if it sends you to another page/site, then you need to check out that site since merely sending you to a site isn't necessarily harmful unless the site is harmful).  I suspect by now you realize that such effort makes browsing hideous and even still there's no 100% guarantee because these people live to make code that gets past all efforts to stop it.

 

I hope that explains the part I overlooked - not because I meant to avoid it, but because what I suggested is the only reasonable way to work regardless of the type or mechanism of infection (I suppose one could always do more, but then IMHO the burden begins to exceed the risks).  There's no special tactic for this one - it's just one of the many ways used to try to get by defenses.

 

The defenses suggested above aren't perfect, but they've kept me safe for years and should work for you.  Not 100% of the time (there's no such thing), but as much as is reasonable.

 

Good luck!
