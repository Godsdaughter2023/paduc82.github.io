Because of the way gmail works, this really isn't an issue. Messages in gmail have labels associated with them. These labels make messages appear to be in the various folders. So, a message can have the Inbox label or the Trash label or the Spam label causing the item to appear in the Inbox or Trash or Spam folder. When you click the Archive button in gmail, what that does is remove all labels from the item, except for the All Mail label.  See https://support.google.com/mail/answer/6576?hl=en for this information.

What deleting a message in gmail does is to remove the label for the folder where the message is, like Inbox, and add the Trash label. This makes the message disappear from the folder it was in and appear in Trash. Then, when you purge the Trash folder, this removes the Trash label. The item still has the All Mail label, however, and will still be in that folder. Thus, using the Archive feature and using Delete then Purge are equivalent operations. You do not lose the message. It will still be in All Mail and you can always recover it.

I've found a sort-of workaround for IMAP, for those who are interested. 

Go into your Gmail account on the website, manage tags, and delete any custom tag that is obviously a trash receptacle (deleted items, etc.). This will just archive any items that currently have that tag. Uncheck "Show in IMAP" for the official Trash system label.

In Outlook, right click on [Gmail] in the folder listing for your desired email account, choose "IMAP folders," and check "When displaying hierarchy in Outlook, show only subscribed folders." 

Right click on the main heading for the email account and make a new folder for trash (I named mine Trash, just to be simple) that isn't within the [Gmail] hierarchy (i.e., the subfolders under [Gmail]).

Go to Account Settings > {your email account here} > Change > More Settings... > Advanced, and make sure that all three checkboxes at the bottom are unchecked ("Do not save copies of sent items," "Mark items for deletion..." etc., and "Purge items when switching folders while online")

Close Outlook and re-open it. 

This should create a label in Gmail called [Imap]/{whatever your custom folder was called}. When you delete things within Outlook, they'll now have the [Imap]/{whatever} label appended to them and will be effectively deleted from Outlook.

The drawback to this set-up is that you have to periodically go into the [Imap]/{whatever} tag on the Gmail website, check all the items, and "Remove Label" to get them into the actual archive so that they're no longer synced (and taking up time in the sync process) in Outlook. Personally, I'd rather have a little extra syncing time than risk something being permanently deleted when I don't want it to be.

************************************************************************************

  Public Function RunSQLstr(strSQL As String, Optional db As Database, _
       Optional lngRecordsAffected As Long) As Long
  On Error GoTo errHandler
    Dim bolCleanup As Boolean

    If db Is Nothing Then
       Set db = CurrentDb
       bolCleanup = True
    End If
    DBEngine.Workspaces(0).BeginTrans
    db.Execute strSQL, dbFailOnError + dbSeeChanges
    lngRecordsAffected = db.RecordsAffected
    DBEngine.Workspaces(0).CommitTrans

exitRoutine:
    If bolCleanup Then
       Set db = Nothing
    End If
    RunSQLstr = lngRecordsAffected
    Debug.Print strSQL
    Exit Function

errHandler:
    If Not Err.Number = 3022 Then  'Bypass if Duplicate record error
        MsgBox "There was an error executing your SQL string: " _
                & vbCrLf & vbCrLf & Err.Number & ": " & Err.Description, _
                vbExclamation, "Error in RunSQLstr()"
        Debug.Print "SQL Error: " & strSQL
        DBEngine.Workspaces(0).Rollback
    End If
    Resume exitRoutine
  End Function

************************************************************************************

Hum, that looks ok.

You could "force" the connection for currentDB to be the SAME as the workspace transaction.

eg:

   Dim db As dao.Database
   Dim ws As dao.Workspace

   Set ws = DBEngine.Workspaces(0)
   Set db = ws.Databases(0)

   ws.BeginTrans

   db.Execute strSQL, dbFailOnError + dbSeeChanges

   ws.CommitTrans

Note how above uses the same database from the workspace - not currentDB which is a new instance of the database object. (I though above was necessary - but both seem to work for me).

albert d. kallal say:

PT queries as you found  out will not fix this issue!

Excellent – and yes my experience hints this is not the update query, but something else.

Often lack of indexing on the table will cause a full table lock + scan as opposed to just grabbing what is required.

As for a difference of 2000 vs 2010. As noted it could well be in the area of the pulling records into the form (a2000 might be pulling more or records faster – so the “query” filling the form completes – no left over table locks. If Access says “stop”, then trouble city. This issue “tends” to occur when a combo or form STOPS pulling data, but SQL server been given the command to pull the data. Forms can cause this, but most often its combo boxes.

Low hanging fruit:
If the form (an open form) has combo boxes based on tables – especially any table being updated, then that likely ground zero. If the form open during this “update” only has say 2 or 3 combo’s, then as noted change the SQL source to a “view”. You have to add these views as linked tables. Base the combo’s on that view. This should not result in you have to change any VBA code.

In addition to no-locks “hint”, there was a recent discussion as to a “better” setting – but I just for the life of me recall what the solution was in place of using NOLOCKS hint in the view. (Hopefully someone can jog my memory as per this recent discussion).

The locking issue is “hard” to re-produce with test data – but I OFTEN find this issue arise during production use with several users – 9 out of 10 times it is combo boxes. While this is “hard” to re-produce, it is RATHER COMMON issue when running SQL server back ends – seen this occur on client sites all the time – solution was the combo boxes. So this issue is NOT at all rare. I can't say I see this issue "more" in 2010 then 2000, but I don't have a 17 year time machine to compare - there likely been changes over that time period in performance optimizing - and that "stop" flow of data issue likely at fault here.
