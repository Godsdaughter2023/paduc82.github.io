suzanne s. barnhill say:
When you open any Word document, an "owner" or "lock" file is created that prevents another user from opening the document for editing while you are using it. When you close the document, this temporary file is deleted. When you use Save As to create a new document from an old one, the original lock file stays open until you close the new document, preventing you from deleting the old one.

Northwind.accdb design question - Microsoft Community

carengrandgenett say:

Northwind.accdb design question

There are two tables in this database that seem to contain the same info:  Customers and Shippers.  Is this good design or would it be better to have one table with many-to-many relationships between other tables?

Does the current design make sense if these are distinct entities even though the data in the tables is the same?

Thanks much!

scottgem say:
Sounds like you mean the table structure is the same, not the data. Generally it is better to use a single table with a flag to identify the type. However, if the scope of the data will be vastly different, as in Shippers and Customers you can use different tables. 

ken sheridan say: It's a very valid question, but I think in this case having separate tables is the correct design.  Tables model entity types, and in the context of the hypothetical reality which Northwind is modelling customers and shippers are quite distinct entity types.  It just so happens that their attribute types, and hence columns, are the same.

You can have a situation where one entity belongs to two or more different entity types.  This is where the entity types are sub-types of a single (super) type.  Chris Date in An Introduction to Database Systems gives the example of a type Employees with sub-type Programmers and further sub-types System Programmers and Application Programmers.  The way this is modelled is by one-to-one relationship types.  Each of these four tables would have a primary key EmployeeID say, but in the tables which model sub-types Programmers, System Programmers and Application Programmers the EmployeeID column is also a foreign key.

A sub-type is characterized by sharing all the attributes of its (super) type but not those of other sub-types.  So the Employees table will have columns for those attributes common to all employees, e.g. names, date of birth etc, whereas the other tables will only have columns which represent attributes specific to the entity type which the table is modelling.

In Date's example a single employee could be represented in all of the tables of course, being both a system programmer and an application programmer.  In the case of the customers and shippers table in Northwind these are not modelling entity types which are sub-types of some (super) type as a shipper will not also be a customer, or vice versa.  Each is a distinct entity type with mutually exclusive members.  Consequently there will never be any redundancy arising from duplication of the same data in each table (or that would be the case but for the other faults in these tables which I outline below).

However, these tables do illustrate other fundamental design flaws.  Both are badly non-normalized in that they have City, State/Province, Country/Region columns.  State/Province and  Country/Region are in neither case functionally determined by the whole of the primary key; State/Province is determined by City and Country/Region by State/Province.  These transitive functional dependencies mean that the table is not normalized to Third Normal Form (3NF).  Consequently the tables are open to the risk of update anomalies.

The tables should include only a CityID column which references the numeric key of a Cities table (city names are not suitable as keys as they can legitimately be duplicated).  Similarly the Cities table should include a foreign key referencing the primary key of a States/Provinces table, and this foreign key referencing the primary key of a Countries table.

I've omitted regions as it's not clear what is meant by this in Northwind.  Is it a region within a country or is it a region of the world comprising multiple countries?  By country I think most people would think of a nation state such as the USA, UK, France etc. I think region must mean a group of countries in Northwind, but if so it should be modelled as a separate table referenced by a Countries table.

Of course not all countries have a state or provincial structure, so the question arises how do we know which country a city is in if there is no equivalent of a States/Province in that country to keep the link in the chain.  The answer is that for those countries (I imagine Luxembourg would be an example) would each be given a single region of N/A, so for a city in Luxembourg the foreign key value in the Cities table would be whatever the numeric ID for the dummy N/A region of Luxembourg is.

From a brief examination of these tables they do now at least seem to have dispensed with one terrible design issue.  In older versions some columns used a combo box as their display control, with the list of items being a value list, i.e. data was being hard coded in the combo box's RowSource.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  This was Codd's Rule #1 when he introduced the model back in 1970.
