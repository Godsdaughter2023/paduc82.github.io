Closing forms with the Escape key

Is there a better way to close forms with the escape key, that will work for all forms (including those with DAO code), and  in both the full version and the runtime version?

albert d. kallal say:

I had some great opportunities to convert some expensive and longtime "green" screen or "text" based systems in which the Esc key was used to close a given screen. (and the record saved when you did this).

I have to 100% agree that changing such behavior is a bad idea. So this being good advice still stands here.

However having said the above, sometimes one just inherits a bad request or in fact you have 5-10 users on a previous system that used Esc to close+save for 15+ years and they really need/want the same behavior on the replacement system you are being asked to build.

So I have been around a GOOD many older text based systems, especially a number that ran on mainframe systems and yes, they did use Esc key to exit forms.

It either walk away from a rather lucrative contract, or simply let them continue to use the Esc key to close the forms. And to be fair adding such feature to the Esc key does ease the training and transition from the older system to the newer system as their minds are all hard wired into whacking that Esc key all day long.

I dare say that even today windows keyboards do have the Esc key up in the left – a high value high priority key location that is so often not used much these days. It is a great and easy key to hit/use while the right hand sits on the mouse.

Ok, now let's see if we can fix this issue.

Just looking at your code, keep in mind that while your attempting to re-purpose the Esc key, your code closes the form, but does not "kill" the Esc keystroke.

Using that recordset code should not case a problem. , but Esc on that form will be attempting to un-do all kinds changes to the forms. If your code (and recordset stuff) is attempting to reference controls, then I can well see this being a real mess.

In all your code where you plan to re-purpose the Esc key, MAKE SURE you KILL the keystroke so that Esc does not still attempt to do its previous job (undo).

So in your above code, you want this:


Case vbKeyEscape
keyCode = 0
doCmd.Close


So in ALL THOSE routines, try adding the KeyCode = 0. This will remove and kill the actions that Esc key normally does here.

In fact, having had to write and build such a number of systems, I would be hard pressed to think that your Esc key code worked without issue in Access 2003. In other words, without killing the Esc key, I would suspect one would have nothing but problems here.

I would also strongly suggest that you change your close command code, since a timer form or any kind of change of focus or even a quick bump of the mouse can result in that code running on a different form (you not close current form).

So you should use:
docmd.Close acForm, Me.Name

In most cases, you can just execute a close, but when you start re-purpose keys, or you executing some command in one of the forms events (as opposed to just clicking on a button), then issues of focus and timing rear their ugly head, and thus more caution and more "clarity" in code such as explicit naming the form to close tends to help and reduce erratic behavior based on timing issues.

So, kill the keycode, and watch out for focus issues - try explicit naming of forms to close when problems crop up.
