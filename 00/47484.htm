cmdrkeene say:

Take a look closely at what you're using.  The Picture library is entirely different from the Picture folder.  The library is just a shortcut way of seeing several places at once (nothing can ever be "stored inside" a library). 

Exercise:  Right-click your picture library and choose "properties".  Here you'll see all the places that will show up inside the Picture library view.  Likely, you have some places being shown in that library that aren't meant to be.  No problem, you can remove them from the list.

Just remember that the library is just showing you an easy combined view of all those locations.  The files are still stored in the original places and if you delete them from the library view, you're deleting the original.

dlan1000 say: I was having similar problems to you.  My issue occurred when a PC that was the creator of the homegroup was upgraded to windows 8.1 (and switched to a "connected" Microsoft user account).  My guess is that somehow during upgrading the OS, the computer retained its old id (pre-connected user account id?) leaving an orphaned member of the homegroup that was being broadcast by the homegroupprovider service (so deleting the local data, such as the idstore, would not help, because the homegroup provider running on that same machine was broadcasting that data to the homgroup listener running on that same machine)

I can't promise it will work for you, but I narrowed down the problem to the HomegroupProvider service running on my own computer claiming that it was a member of its own homegroup.  In order to change this, I needed to delete a key from the registry with system permissions.  Others experiencing this or related issue can try the following:

FIRST: A caveat: Before doing any of this, I had left the homegroup from all other computers on the network.  This is because other computers in the homegroup are broadcasting homegroup membership to any listeners on the network, so it is unclear if deleting a member on only one computer will quickly be reverted.

1) Check in the registry for some member keys in the location: HKLM/SYSTEM/ControlSet001/Services/HomeGroupProvider/LocalUserMembership

If there are no member keys there, this might not be your issue.  If there are, you can proceed to step 2.  

2) Export the LocalUserMembership key to a file on your desktop (just in case). You'd like to delete whatever keys are under "LocalUserMembership" -- but of course when you try, you will see that you do not have access to do so.  In order to get access, you need to run regedit as a system account.  See step 3.


3) Download pstools (google it) and put them in a folder (I put them in C:\program files (x86)\pstools\ ) and add that folder to your path (edit the system environment variable path... google that too if you don't know how.

4) Open a command prompt as admin and type: "psexec -i -d -s c:\windows\regedit.exe".  This will run regedit with system account access, giving you the permission to delete that offensive member key

5) Go back to 
HKLM/SYSTEM/ControlSet001/Services/HomeGroupProvider/LocalUserMembership
and delete all keys (folders) under it (you should have exported this to a .reg file on your desktop in step 2. In the event of an issue or if this fix doesn't work and you want to reverse it, you would have to reimport the .reg file with a system account).

6) run homegroup and see if it still reports that you are a member of the homegroup.   For me, it finally showed that I was not a member (and asked if I would like to join some other homegroup on the network.
