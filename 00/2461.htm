ken sheridan say: You do not need to create the labels report every time, only once.  Simply create the report using the wizard basing it on the same table or query as your form.  Save it with a suitable name, e.g 'rptMylabels'.  If you open the report normally from the navigation panel it will return all records.  To restrict it to only those you select by filtering the form you need to also filter the report.

When you filter a form using Filter by Form it creates a string expression as the form's Filter property.  This will be along the lines of the following:

[Code] = "ABC" Or [Code] = "XYZ"

When the filter is applied the form's FilterOn property is set to True.  So armed with these two pieces of information you can open the report filtered in the same way.  You can't do this via the user interface, however; you need to add a command button to your form.  In the Click even procedure you then need to write a little code as follows (I'll explain how to do this below)

    Dim strFilter As String
    
  ' is the form currently filtered
    If Me.FilterOn Then
        ' if so assign the filter string to the strFilter variable
        strFilter = Me.Filter
    End If
    
    ' open the labels report in print preview.  If the strFilter variable is not a zero-length
    ' string the report will open filtered to the form's current filter, if not it will open unfiltered
    DoCmd.OpenReport "rptMylabels", View:=acViewPreview, WhereCondition:=strFilter

In the above the lines beginning with a ' character are explanatory comment lines, not executable code.  You can leave them in or take them out as you wish.  If you want to be able to print the report directly from the form you can add another button for this.  The code will be the same apart from the last line, which will be:

    DoCmd.OpenReport "rptMylabels", WhereCondition:=strFilter

This should work for most filters, but in some situations the Filter property will be a little strange.  Often a control in a form will be a combo box which looks up a value from another table, e.g. the combo box may show a city name, but the actual value in the form's table will be a numerical CityID.  You filter by name, however, so the Filter by Form facility creates a special string which uses the name but looks up the corresponding number.  The labels report won't normally understand this string.  You can get round this by also using a combo box in the labels report, which is unusual, but can be done.  Hopefully this is an issue you won't have to address, however.

If you are unfamiliar with entering code in event procedures, this is how it's done in form design view:

1.  Select the button and open its properties sheet if it's not already open.

2.  Select the On Click event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new lines between these.

ken sheridan say: As Scott says, the records are being restricted in some way.  If the report's RecordSource property is a table, then the restriction must be in the report itself.  It may well have something entered as its Filter property, but don't worry about that; this property is persistent, so once you filter a report the Filter property stays there until you change it.  The important property is the FilterOnLoad property.  This should be No; if it's Yes change it to No and save the report.

If the report's RecordSource is a query on the other hand, does the query return all rows if opened independently of the report?  If it does examine the report's properties as described above.  If it doesn't then open the query in design view and see if there is anything in the 'criteria' row of any of the columns.  If there is then delete it and save the query.

Another possibility is that the report's RecordSource is a query in which two or more tables are joined.  If there are rows in one table without any matches in another table then, if the default INNER JOIN has been used, only rows where there are matches will be returned.  In this case the join would need to be changed to a LEFT OUTER JOIN.

jeeped say:
If the format can safely be assumed to be constant, e.g. empty row separating entries; a comma separating the city and state; zip codes used not zip+4, then 5 formulas should get you the first row and you can fill down as far as necessary.

                  

The formulas in B1:F1 are as follows:
=OFFSET($A$1,(ROWS($1:1)-1)*4,0)          in B1
=OFFSET($A$1,(ROWS($1:1)-1)*4+1,0)      in C1
=LEFT(OFFSET($A$1,(ROWS($1:1)-1)*4+2,0),FIND(",",OFFSET($A$1,(ROWS($1:1)-1)*4+2,0))-1)      in D1

=RIGHT(OFFSET($A$1,(ROWS($1:1)-1)*4+2,0),5)      in F1.

Those formulas can be filled or copied down as necessary to cover all entries in column A. john korchok say:

You can eliminate the footer warning by creating a custom page format with 0 margins and using that page size when printing to PDF:

  1. In Word, choose File>Page Setup.
  2. Click on the Paper Size dropdown and select Manage Custom Sizes...
  3. Click on the Plus sign to create a new page format.
  4. Click on Untitled1 and give it a name like Borderless Letter.
  5. Set all 4 margins to 0.
  6. Click on OK.

Copy the code below into a regular module.

Andreas.

[code]

Function IFS(Expression1, Value1, ParamArray Args()) As Variant
  'https://support.office.com/en-us/article/ifs-function-36329a26-37b2-467c-972b-4a39bd951d45
  Dim i As Long
  IFS = CVErr(xlErrNA)
  If CBool(Expression1) Then
    IFS = Value1
  Else
    For i = LBound(Args) To UBound(Args) Step 2
      If CBool(Args(i)) Then
        IFS = Args(i + 1)
        Exit Function
      End If
    Next
  End If
End Function
