bob jones aka: cybertaz say:

Your expectation is correct... a Continuous Section Break will provide the effect you want. The difference is that it does not force the following section to start on a new page. Accordingly, the Header & Footer for a section only can appear starting on the first full page of the Section. IOW, if a Break is inserted in the middle of page 5 the H/F for that Section will first display on page 6.

The default behavior of the program is to have a running Header/Footer throughout the entire document. It sounds like you haven't cleared the Link to Previous property for the Sections beyond Section One. As long as the Link to Previous check is in place for each of the other Sections, content entered in the H/F of any Section will appear in all.

rich michaels - mvp (greatcirclelearning.com) say:

There are potential 6 headers and footers per section and you diligently have to inspect all of them.

Each section can have:

  1. One primary page header (odd page)
  2. One even page header
  3. One Different First Page header
  4. One primary page footer (odd page)
  5. One even page footer
  6. One Different First Page footer

You may have to go section by section and temporarily enter forced page breaks in order to view each of them and see how the settings for Link to Previous are marked. For example, an even page footer in Section 3 that also has a Link to Previous marked, will connect to Section 2's even page settings.

1. Language is a character property in Word. Here is how to tell which language has been applied to a selection in Word 2007: Click the Review tab, and then click Set Language. 'Inserts current date
Selection.InsertDateTime _
DateTimeFormat:="dddd, MMMM d, yyyy", _
InsertAsField:=False
End Sub

Save the macro to the Normal template. For installation instructions, see http://www.gmayor.com/installing_macro.htm.

scottgem say:

Hi Jen,

Welcome to the Answers Community and to using Access. Since you said you were brand new to Access I hope you don't mind some suggestions here.

Gina and Ken have given you good answers but I generally use a different method. But before I describe that, I'm concerned by your statement "There are 584 records all up and each one has a status of either Active (179 records) or Closed (405 records)." This suggests that you only have one table. And if that's the case then you probably have Design issues. A database to track projects would rarely have a single table. If these projects are for customers, then you should have a customer table. I would think you would also need tables for employees assigned to projects, Departments and other lookup tables to make sure input is standardized. If you do have such tables, then just ignore this. If not, I would suggest you show us your table structure so we can suggest a better one.

Since you only have have 2 statuses I would suggest that having a field for status is redundant. Because I'm guessing you also have a field for Closed date. If you do then you can use the value of ClosedDate to determine status. If ClosedDate is Null then the project is Active, otherwise its closed. 

So the way I would do this is with an Option Group. I would put an Option group in the Header of your form and label it View. The Option group would have 3 radio buttons; All, Active, Closed. In the After Update event of the Option group I would use the Following code:

Select Case Me.optView

     Case 0 'All

          Me.Filter = ""

     Case 1 'Active

          Me.Filter = "[ClosedDate] = Null"

     Case 2 'Closed

          Me.Filter = "[ClosedDate] Is Not Null"

     Case Else

           Me.Filter = ""

End Select

DoCmd.Runcommand acCmdApplyFilterSort

The user selects one of the options in the group and the Filter for the form is set and applied.

scottgem say:

You don't "make a field a foreign key". A foreign key is a field in a child table that is the same datatype as the corresponding Primary Key. If you are using an Autonumber PK, then the FK needs to be a Long Integer.

Yes a table can have many FKs. But to impose Referential Integrity, There has to be a corresponding record in the related table. So if you have any project records, you can't impose RI until you have matching records. You would have to remove all the records from both related tables to impose RI if there are not all matching records.

Is it possible for a customer to have multiple PAs? If it can happen then the PAs should be listed in a child table of the Customers table.

 

You also shouldn't use special characters like % signs in object names.

You have CustomerContact linked to CustomerID, why? That should be linked to CustomerID. What is CustomerContact?

If you have a StartDate, you shouldn't also have ProjectYear as that can be derived from the StartDate.

ken sheridan say: You appear to be entering the code directly into the event property in the properties sheet, not into the event procedure.  If you are unfamiliar with entering code into a form's or control's event procedures, this is how it's done in form design view:

1.  Select the form or control as appropriate and open its properties sheet if it's not already open.

2.  Select the relevant event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new lines between these.
