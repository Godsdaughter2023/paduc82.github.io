You'll find an example of this sort of 'row level security' as SecDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169


If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

If we take the SecDemoSimple_PseudoLogIn.accdb file in the zip archive as an example it's model is as below:



The main data entry form for inserting or editing rows in the MyTable table is based on the following query:

SELECT *
FROM MyTable
WHERE EXISTS
    (SELECT *
     FROM RowUsers
     WHERE (RowUsers.MyID = MyTable.MyID
     AND RowUsers.UserNameID = Forms!frmLogin!cboUserName))
OR Forms!frmLogin!chkShowAll = TRUE
OR Forms!frmLogin!chkAdministrator = TRUE
ORDER BY MyTable.MyID;

The frmLogin form, which simulates a user login has two unbound check boxes chkAdministrator and chkShowAll.  The former is read-only and indicates whether the user has been designated as an Administrator in the Users table, the latter can be checked so that when the main data entry form is opened it will show all rows rather than those to which the user has permissions by virtue of a row in the RowUsers table.  This is done by the above query referencing the check boxes as parameters.

In your case you could either designate only yourself as an Administrator, or you could include a form such as my frmLogin form in a front end to which only you have access, and include a similar check box to my chkShowAll control in the form.

While the PseudoLogIn files in my demo, as the name suggests, only simulate a login procedure, you'll find examples of automatic logging in on the basis of the user's Windows user name in the AutoLogin files in the same zip archive.  In these the login form is not shown at start-up if the current user is already registered as a user in the Users table, but is opened hidden.  However, you can for demonstration purposes show the form from the main data entry form, and in it elect to see the rows for another user or all users.  In an operational database, however, I would restrict the hidden form to one which is read-only and does not allow another user, or all users, to be shown.  This would be done via a separate form available only to you as the administrator.


The fact that my demo is not split into front and back ends is immaterial to the methodology which it illustrates.  An operational database would of course be split.

ken sheridan say:
...............we do only have one frontend and one backend, no other files exist.

If so, that would be a bad set-up and very likely to lead to corruption.  Multiple users should not access a single front end.  The correct set-up would be to have a single back end, and multiple copies of the front end, each of which should be installed either on the user's local machine, or in a 'personal folder'  on the system to which only the user  and the administrator(s) have full permissions.   My office used the latter approach.

I am extremely doubtful that your set-up is in fact what you describe, however, as it does not explain the behaviour which you are experiencing.  You may think you have a single back end, but everything points you your having separate back ends per user, which might be the case without your realizing it.  Access has no built in mechanism for implementing the type of 'row level security' which you describe; you have to program the database yourself to achieve this, which is what my demos illustrate.

The basic mechanisms for 'row level security' are essentially quite simple:

1.  Each user must be distinctly identified when they open the database, which can be done by building in a log-in form into the database, or simply by using the Windows user name, which can be obtained in the way illustrated in the 'auto-login' versions of my demos.

2.  Each row in a table is related to a user by means of a column which stores the user's distinct identifier, or if rows can be available to more than one user, by means of a related table in which the primary key of each row in the main table is associated with the user's distinct identifier, e.g. if a row is available to three users, there will be three rows, each with the same value of the primary key of the row from the main table, and with a different user's identifier in the other column.  My demos work in this latter way.

3.  Users' access to the table is solely via a query which returns only those rows with their distinct identifier or where the current user is an administrator.  The query I posted in my earlier reply is an example.

4.  The table restricted in this way would be the main referenced table in the database, and other queries involving other tables should join those tables to this query not to the base table.

Take note of the 'health warning' in my demo's opening form.  While this is referred to as 'row level security' it is not secure to any real degree and any reasonably competent meddler would be able to access all the data, maliciously or otherwise.  It is essentially no more than a means of controlling access to the data in the normal day-to-day operation of the database.  While you can prevent a user gaining access to things like the navigation pane and the built in ribbon interface etc, MS Access is not a secure environment in any real sense.  I don't know what the 'spends' represent in your case, but in the authority in which I worked MS Access would never have been contemplated as a suitable basis for applications which involved the storage of employees' personal financial developed for my own personal accounting purposes is not of a sensitive nature which could be used maliciously.  I leave that to my bank in the hope that their systems are as secure as possible!
