It seems that with SharePoint Online, neither method is viable. In the case of the userName() function, the Windows login is what is pulled. In most SP Online systems, that will not match the username portion of the O365 login. For example, the Windows/network login might be jdoe where the O365 login might be *** Email address is removed for privacy ***. In any case, there's no guarantee that a user will be logged into the organization's network when they visit the SP site.

The User Profile Service solution won't work on SP Online, either, as there is a loopback protection. In on prem systems this protection can be turned off to allow connection to the web service, but currently in SP Online, it cannot -- not even by the Microsoft back end admins.  Therefore there is no way that I can see for an InfoPath form to hit any of the SP web services if published to SP Online.  http://support.microsoft.com/kb/2674193

If anyone has found a trick to have an IP form recognize the O365 current user in this circumstance, I'd love to hear about it.
