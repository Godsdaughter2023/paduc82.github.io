ken sheridan say: You can also do it by referencing the control as a parameter in a query, e.g.

SELECT *
FROM YourTable As T1
WHERE
    (SELECT COUNT(*)
     FROM YourTable AS T2
     WHERE T2.YourField < T1.YourField) < [Forms!YourForm!YourTextBox];

This will return the TOP n rows by the value of YourField, the smallest first.  Note that unlike using the SELECT TOP option this does not require that the result set be odered by YourField.

That's a pretty easy modification. Try,

Private Sub Worksheet_Change(ByVal Target As Range)
    If Not Intersect(Target, Range("C:C,E:I")) Is Nothing Then
        Application.EnableEvents = False
        For Each c In Target
            If Not Intersect(c, Range("C:C,E:I")) Is Nothing And Not c.HasFormula Then _
                c.Value = UCase(c.Value)
        Next c
        Application.EnableEvents = True
    End If
End Sub

Tap Alt+Q to close the VBE and return to the workbook when done.

albert d. kallal say:

Access does not have a macro recorder, but it has a AMAZING set of tools to build data entry forms. (so much so, that a macro recorder never been needed).

 

In other words is there a possibility that you the first person on planet earth that's come up with this concept and request of a macro recorder?

 

So how this works in Access somewhat different.

 

In the vast majority of cases if the need arises for repeating entry of the SAME data over and over, then this often suggests that data is being repeated in the tables.  And when data is being repeated in the tables, it often suggests that the database is not normnalized.

So for example if we doing a survey for today and we need to enter todays date. We might have this:

SurvayDate       QuestionNum       Answer:

03/15/2012         5                          A

03/15/2012         6                          B

03/15/2012         7                           A


In the above we have a design that suggests we have to type survey date over and over again.  In actuality it means the form and table designs are incorrect. The correct design would move the SurvayDate up to a parent table and then we use relational database concepts here. In fact the correct design would be to have a parent form with who did the data entry, the date of data entry, and the survey date. The child form only needs the QuestionNum and Answer columns. (thus only ONE row exists for the survey date). Therefore, we don't have to have some type of keystroke recording system because the SurvayDate is only entered wants by the person using the application. (and this explains why for the near 20 years of computing history on desktop PCs, most products designed for efficient data form entry don't have a macro recorder).

 

In other words by normalizing our database, in fact we only have to enter the survey date once into the system, and then start entering the question numbers and answer.  The beauty of the of the system is not only are we eliminating the repeating data but ALSO the need for the data etnry person to enter that data over and over.

Ok, having said the above, obvilsity great forms and great data entry systems do however automate common tasks.  And keep in mind that this is what Access as a tool was DESINGED for. So in the world of data entery forms Access is one amazing kick you know what kind of tool here.  Access is thus bristling with buckets of technology to build these amazing forms and streamlined those processes that you're attempting here.

 

The first thing is there are few keystrokes such as control ' (ctrl key and single quote) that will copy the value that was previously entered into the text box you are currently editing.

 

Next up, is that if you place a combo box to present the user a list of predefined choices in a form, not only does the speed up data entry, but those combo boxes also have what is called auto complete. This feature behaves much like when in excel that typing in the first few characters of the match will automatically complete the selection for you (and again, no code is required for this). And then toss in the "default" setting ability for a data entery form, then you'll find at the end of the day an amazing and in fact vast majority of streamlined data entry forms can be built without having to utilize any writing of code at all. In fact in the vast majority of cases, I seen people resort to writing code since they'd simply did not take the time to learn all the features of the product.

 

However, for code? Each form has a "key down" event that can run custom VBA code. And not only does each form built has this feature, but each each individual control placed on a form ALSO has such ability for custom code to run on a given keystroke.

 

And last but not least, while there's not a macro recording system, in Access there are two programming languages. One is called VBA (visual basic for applications), and the other programming language is an actual macro language. This macro language is less feature rich than that of visual basic . While most developers will not use the macro language, one advantage of that system is the ability to assign keys to execute particular bits of code. Thus the macro system is often used here. And even for experienced visual basic developers, we will still often use some of the macro language features because those macro language features have some abilities that we don't have in visual basic in terms of working with user keystrokes. (in fact specifically, global keystrokes settings to the whole application). However the macros can be set up to call your visual basic code based on those global key settings (and this is pretty much how we build and setup function key actions such as for f1-f12 function keys you have across the top on most keyboards).

 

Now before you go running off to start writing a bunch of code (which you may not need), we still have to determine if your "issues" are ones of database normalization (or lack of) in which you're attempting to copy repeated data over and over or are you looking to automate some particular tasks that a person typically has to do during the operation of this application.

 

So you'll have to cook up a typical example or scenario of what your tempting to do. Once you do that then I can suggest if that issue will be solved by coding, database normalization, or in fact using the Macro system in Access.

 

So Acess does have a number of very capable choices that will allow you to accomplish some of your goals here, but which were road I send you down will very much depend on the type of road and task you are attempting here.  

For example, to update many rows that already exist in the database, it has been suggested in this thread already that in place of the user interface and using search and replace, you can execute an update query.

 

So give a small hint of the type of task you attempting here. I can then suggest your best bet in terms of time.  In other words it's probably not likely that you're going to be able to learn all the systems and infrastructures you have available in Access, since they are rather extensive. 

 

This is kind of like an automobile. Maybe just need an air compressor for something in your tires. On the other hand maybe we have to take apart the whole engine and start changing camshafts out.  While I think most people are willing to put air in their tires, changing out the camshaft is going to be too large of an investment of time here.

 

There's a rather extensive help system built into access if you jump into the code editor, so you can check that out while you formulate and give us some additional information on the type of goals and tasks you're looking to automate in a typical data entry form.

Hello Mate,

Does the presentation only appear on two monitors and the third one will only be a black screen?

Anyhow, here are some few things that you can try:

How to Configure a Computer to Display on Multiple Monitors

  1. Connect the primary PC and the second and third PCs to the same network or use Ethernet cross-over cables to manually connect.

  2. Open the browser on your primary computer and go to the MaxiVista website to download a free demo version of MaxiVista.

  3. Click “Download Now.”

  4. Click “Open” in the dialog box which opens.

  5. Click “Run” to start the set-up and follow the set-up wizard's instructions.

  6. Repeat the download and set-up process on the other two computers by repeating the primary computer set-up procedures.

  7. Right-click the MaxiVista icon on the desktop of the second computer and click “Enable Secondary Display.”

  8. Right-click the MaxiVista icon on the desktop of the third computer and click “Enable Secondary Display.” Your computer is now set-up and ready to run multiple monitors.

How to Configure PowerPoint to Display on Three Screens

  1. Open PowerPoint presentation.

  2. Click “Slide Show.”

  3. Click “Set Up Show.”

  4. Check the box “Show Presenter View” in the dialog box which opens. This opens a navigation panel on the presenter’s monitor which allows the presenter to easily manage the multiple screens.

  5. Click the monitor you want the slide show presentation to appear on under the “Display Slide Show” list.

  6. Click “OK” to start the presentation.

Let me know how it goes. Cheers.
