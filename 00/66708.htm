lemp say:

You've been given some misleading and incorrect information.

First of all, if you use CHKDSK without specifying the volume, CHKDSK operates on the current volume.  Thus,

   G:\>chkdsk

will operate on G: (as your screenshot shows it did).  It is, however, probably "safer" to specify the volume you want checked, as in

   G:\> chkdsk G:

Second, exFAT is not "retired technology" (nor is it the same as FAT32), although it is about 11 years old.  Like FAT32, the exFAT file system is recognized by many different computer operating systems, including Apple's OS and Linux in addition to Windows.  While FAT32 will only permit files of approximately 4 GB or less, exFAT can handle much larger files.  exFAT is mainly used for USB flash drives and SD cards, especially large capacity ones.

There is a downside to using exFAT, as explained at http://www.digitalcitizen.life/what-is-exfat-why-useful, and this downside seems to have caught you.

In our opinion, there’s only one real “not so positive” thing about exFAT, and that’s the lack of support for journaling. Journaling is a feature that allows the file system to keep records of changes made to files stored on it. That’s useful when data corruption occurs because journals can be used to recover broken data. exFAT doesn’t have this feature, and that means that data can be corrupted more easily when unexpected shutdowns occur or when the removable drive formatted this way is not safely ejected.

When you run CHKDSK with the /F switch and file system corruption is detected, the file fragments are saved as .CHK files and those files are placed in a FOUND.nnn folder. 

What your screenshots appear to show is that you ran CHKDSK /F at least once prior to the time represented by your screenshot and file errors were found.  The file fragments were saved in (at least) FOUND.007.

Subsequently, your flash drive experienced yet more data corruption because now CHKDSK is reporting corruption in the file fragments that it had previously found.

In addition, the "Allocation Bitmap" is reported to be corrupted.  The Allocation Bitmap is part of how the file system keeps track of what physical clusters on the storage device are in use.

Given the level of corruption reported, I'm not surprised that UnChk couldn't recover anything.  You can try one of the other utilities here: https://www.raymond.cc/blog/how-to-recover-chk-files-created-by-chkdsk-and-scandisk/ or if you know that all of the files on the flash drive were JPG files, you can try renaming the *.CHK files to *.JPG (if they can't be opened by the usual Windows photo viewer, try free IrfanView (http://www.irfanview.com/).

I suspect, however, that your files won't be recovered easily, if at all.

Once you resolve this situation, either by recovering some files or writing them all off as irretrievably lost, I suggest that you reformat your USB flash drive using NTFS.

lemp say:

Windows determines how to deal with files by looking at the "file name extension" -- the 2, 3, or 4 characters after the "dot" in the file name.  

Application programs are "associated" with a "file type" (i.e., the extension). 

When CHKDSK "recovers" fragments of corrupted files (and the fragment may be the entire file), it assigns the "chk" extension to those recovered fragments.  There is no program associated with the "chk" file type.

What programs like UnChk do is examine the contents of a "chk" file (usually the first several bytes of the file) and make a guess as to what type of file it is. 

"When contents are recognized (for example, as a Word document), the CHK files are copied to a new file with the correct file extension (Filexxxx.CHK -> Filexxxx.DOC). The new file extension allows you to easily open the file using the associated program. That how UnCHK works to manage CHK file recovery." 

See https://www.easeus.com/file-recovery/chk-file-recovery.html

Or you can just rename the Filexxxx.CHK files yourself.

Giving the files the correct extension, whether manually or otherwise, is how "those relationships [of application programs] to actual files can be re-created" (icons have nothing to do with this; icons are just visual cues for the human user).  Of course, this only works if the Filexxxx.CHK files are complete and undamaged.

For more, please read:  http://www.ericphelps.com/uncheck/  The explanatory text is still valid although this page was obviously written a number of years ago:  SCANDISK was the predecessor to CHKDSK for Windows 9x and earlier and does not work with NTFS formatted disks, which is the default for Windows 2000 and later.  AFAIK, the last version of UnChk is version 3, "built for Windows 2000, but might work on computers as old as Windows 95."

And to repeat my earlier advice, once you're done with this exercise, re-format your USB flash drive to NTFS and you'll be less likely to experience unrecoverable file system errors.
