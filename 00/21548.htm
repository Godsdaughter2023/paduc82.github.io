It would appear that you have gone from a dreadful situation (course work in one huge undigestible memo field) to a merely very bad situation (encoding data in fieldnames, with a different FIELD for each course).

You're using a relational database - use it relationally! The proper structure for a course enrollment table involves THREE tables:

Students
  StudentID <primary Key>
  LastName
  FirstName
  <other biographical data>

Courses
  CourseID <e.g. an autonumber, or a unique stable course ID such as GEOIntro>
  CourseType <e.g. SS, HES> 
  CourseLevel <e.g. Intro, Advanced>
  <other information about the course as an entitiy in its own right>

Enrollment
  EnrollmentID <autonumber primary key>
  StudentID <who took the course>
  CourseID <what did they take>
  CourseDate <when they took it>
  Grade <how well they did>

This will make your calculations much, much easier; you can join all three tables, use criteria to select just intro or just advanced courses, just SS or just HES courses, whatever is needed.

ken sheridan say: You need to appreciate the fundamental basis of the database relational model, which is that it is a structured of each entity type are represented by columns in the table.

The interrelationships between the real world entity types are modelled in the database by relationship types between the tables.  A simple one-to-many relationship type, or less commonly a one-to-one relationship type, is modelled by the foreign key of one table referencing the primary key of the referenced table.  A many-to-many relationship type, which can be between two or more entity types, and thus tables, is modelled by a table which resolves the many-to-many relationship type into a set of one-to-many relationship types.  A relationship type is really a special kind of entity type, so itself can have attributes represented by non-key columns in the table.

The taking of a course by a student is a relationship type between the Students and Courses entity types, so is modelled by a table with foreign keys referencing the primary keys of the Students and Courses table.  However, the relationship type is not simply a binary (2-way) one but as it also includes the requirements entity type, also modelled by a table, it is a ternary (3-way) relationship type.  So if student S1 takes course C1 for requirement R1, then it would contain a row:

S1    C1    R1

whereas if student S2 takes the same course for requirement R2 it would contain a row:

S2    C1    R2

These three columns constitute a candidate key of the table, so should either be defined as a composite primary key, or included in a unique index if a separate surrogate primary key, usually an autonumber, is used.  The points achieved by a student in the course are a non-key attribute of the relationship type so are represented by a fourth column.  Computing the GPA is a simple task of grouping by student, and dividing the SUM(points) by COUNT(*), the latter returning the count of rows per student, i.e. the number of courses taken.

It is important in designing a relational database that this theoretical basis of the relational model is understood and reflected in the tables and relationship types between them (the logical model).  In essence, if the model is an accurate reflection of the part of the real world with which the database is concerned the it will act in the same way as the underlying reality and return accurate results; if not then it will be unreliable and not fit for purpose.  At a practical level it will also make it far easier to return meaningful information from the base data.  Otherwise you will find yourself having to resort to the sort of Byzantine solutions which you appear to be attempting, with very little assurance of the accuracy of the results.

The only way for Project to know a task should have finished (or started) on a particular date is if you saved a baseline. If you have saved a baseline, Project will calculate variance (start, finish, work, duration, and cost) by comparing the baseline data against the actual data.  No need to write a formula, the calculation is done. 

Did you save a baseline before you started tracking?

How to make "all markup" selected permanently - Microsoft Community

melisdba say:

How to make "all markup" selected permanently

I want Microsoft Word document to be in "all markup" mode everytime I open a document. However, I couldn't find any way to make "all markup" mode default. Is there any way to make "all markup" mode selected permanently?

stefan blom say:

Simply selecting "All Markup" in the Display for Review drop down on the Review tab should set that view as the new default for Print Layout view. To ensure that Word always starts in Print Layout view you should turn off the "Open e-mail attachments and other uneditable files in reading view" option at File tab | Options | General.

If it still doesn't work, you have to install a macro that forces "All Markup." This one is from Greg Maxey's post in the thread at http://answers.microsoft.com/en-us/office/forum/office_2013_release-word/how-to-change-simple-markup-in-word-2013-to-all/309b004e-d6bd-4716-9eee-3180ea496838:

Sub AutoOpen()
With ActiveWindow.View.RevisionsFilter
  .Markup = wdRevisionsMarkupAll
  .View = wdRevisionsViewFinal
End With
End Sub

Put the macro in your Normal template. For assistance, see http://www.gmayor.com/installing_macro.htm.

Why can't I move the Word 2007 insertion point cursor or highlight - Microsoft Community

k.c.s. say:

Why can't I move the Word 2007 insertion point cursor or highlight text or pictures with the mouse after I type text when using Word 2007 on Windows 7 64-bit?

Symptoms:

 

1)     The common fixes from the past do not resolve this problem on Windows 7 64-bit.

 

2)     When you open an existing Word document you are able to use the mouse as usual to move the insertion point cursor to different sections of the document, to select objects (like pictures) within the document, and to highlight existing text as expected. As soon as you type anything new anywhere in the document and push the Enter key (or just push the Enter key) the mouse can no longer be used to move the insertion point to another location in the document nor can it select text or objects. The arrow and shift keys still allow you to move around the document and select objects.

 

3)     When you create a new document you are able to type normally to add text to the Word document. As soon as you type anything in the document and push the Enter key (or just push the Enter key) the mouse can no longer be used to move the insertion point to another location in the document nor can it select text or objects. The arrow and shift keys still allow you to move around the document and select objects.

 

4)     You CAN use the mouse to select Word menus and buttons on the ribbon bar but you cannot use the mouse on or inside the Word document itself once the condition occurs after typing begins.  The mouse works fine and as expected in Excel and every other application.

 

5)     As soon as ANY open Word document is impacted by the "mouse stopped working" problem, all other open Word documents can't use the mouse either until Word is closed and you start the tests over again.

 

 

Solutions that did not work for me:

 

Similar symptoms have been reported by others using Word 2007 and 2003 on Windows 7, Vista, and XP but I have not found an acceptable solution to this problem with this system configuration.  It seems related to some kind of UAC or permission problem or is 64-bit specific but I am not certain.

 

I have already tried the following:

 

- Rerun Office Suite 2007 SP2 to confirm it was installed.

 

- Tried (twice) to rename the original, "HKEY_CURRENT_USER\Software\Microsoft\Office\12.0\Word\Data" registry key as suggested in these posts.  The new Data key gets created by Word and new values are added to the registry key but problem remains.

Bad Workaround:

 

- The only way I have found to USUALLY get the mouse to behave as expected and eliminate the problems described here is to force Word to "Run this program as administrator”.  I opened the properties of the "Microsoft Office Word 2007" shortcut in the start menu, chose Compatibility tab, and chose that option under privilege level. Without running Word as administrator I have been unable to eliminate these mouse problems.

 

- Once you force Word to Run as admin, you now get the annoying warning prompt from Windows that asks you if it is okay to run as admin every time you start Word.  This would be an almost tolerable workaround if all you ever did was create brand new Word documents. If you want to work on an existing Word document this solution isn't a good one, here is why:
