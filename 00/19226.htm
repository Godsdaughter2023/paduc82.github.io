IE8 gets KB927917 on pages being refreshed from server, even though - Microsoft Community

fixpertise1 say:

IE8 gets KB927917 on pages being refreshed from server, even though the pages can be loaded w/o error by setting the cache mode to "Always Refresh from Server".

We  have web site  with several web pages that  involve Google maps plotting vehicle locations and routes.   These pages load and work flawlessly in Firefox and IE 7.   In IE 8, If you use the Developer Tools to set the cache discipline to "Always Refresh from Server", the pages load flawlessly as well and you can naviagte the whole site without a problem.    However, if you use the default IE settings ("Always Refresh from Server" disabled) , you will often get the error "KB927917: HTML Parsing Error: Unable to modify the parent container element before the child is closed  at Line 0, Code 0, Char 0"  when you try to visit a page containing a map.   It is hard to understand how this can possibly be true if the pages load without error when you have set the cache discipline to "Always Refresh from Server."

The overall structure of the pages is as follows and conforms, as far as I can tell, to Google's model.  Notice that the body attributes indicate that the function load_gmap  is to be excecuted AFTER the body is loaded which would make it impossible to be modifying an element before it was closed.
<html xmlns="http://www.w3.org/1999/xhtml">
  <head>
  <body onload="load_gmap()" onunload="GUnload()">
    <div id="headerCell">
    <div style="clear: both;"/>
    <div id="topmenu">
    <div id="submenu">
    <div style="clear: both;"/>
    <div id="pagecell1">
      <h2>
      <form name="locHistory" id="locHistory" action="/LVMUI/vehicleLocationHistory.jsf" encType="application/x-www-form-urlencoded" method="post"target="">
      <div id="gmap" style="position: relative; background-color: #e5e3df; margin: 1em 0em; width: 700px; height: 500px;" __e_="[object Object]">
      <form name="buttons" id="buttons" action="/LVMUI/vehicleLocationHistory.jsf" encType="application/x-www-form-urlencoded" method="post">
      <h3>
      <h3>
      <table class="standardTable" id="data">
      <div style="height: 2em;"/>
      <h3>
      <table class="standardTable" id="data1">
      <p/>
      <div class="footer" id="footer"/>
      <!--  MYFACES JAVASCRIPT  -->
      <script type="text/javascript"><!--   function getScrolling()  {   var x = 0; var y = 0;if (self.pageXOffset || self.pageYOffset)   {    x = self.pageXOffset;    y = self.pageYOffset;   }    else if ((document.documentElement && document.documentElement.scrollLeft)||(document.documentElement && document.documentElement.scrollTop))   {    x = document.documentElement.scrollLeft;    y = document.documentElement.scrollTop;   }    else if (document.body)    {    x = document.body.scrollLeft;    y = document.body.scrollTop;   }   return x + "," + y;  }   //--></script>

When these pages successfully load, you can use the Developer Tools to validate them and they all pass W3C validation with only a warning about the lack of character encoding. 

I have tried different Google Maps API versions and several several "solutions" posted on the Internet and even though most don't make sense, they certanly do not help.   Some in fact make the problem worse.   

All map drawing in these pages is done in the onload function so there should not be any issue with modifying unclosed elements.  You certainly don't get that error when refreshing from the server  (via setting "Always Refresh from Server)  which would indicate that the pages are in fact OK. 
 
One other instersting observation is that if you manage to successfully navigate to a number of pages including maps by setting "Always Refresh from Server" and then disabling "Always Refresh from Server",  you can use the the forward and back buttons to revisit the pages without a problem.  The browser always retrieves the pages from cache and does not contact any server.  I have verified this by watching for the TCP/IP connections.  It is only when you try to navigate to another page containing a map by clicking on a link that you may get the error.  In this case, the browser successfully retrieves the page from the server and then fails with KB927917   I have verified this from both the server logs and watching the TCP/IP connections come and go.
 
This could well be a bug in the Google Maps APIs, in IE 8 or in the pages themselves  given the fact that it only occurs when visiting pages including maps.  But given the fact that FireFox and IE 7 do this flawlessly, I suspect IE 8 has a big hand in this failure.  (BTW, it doesn't matter if you have compatibility mode enabled or not, nor does it depend on the setting of the Browser Mode or Document Mode)

What is particularly frustrating about this is that (1) the KB927917 error does not include the names of the element involved and (2) the IE 8 Debugger is of no help whatsover because you have not completely loaded the page when the srror occurs.  (In fact, it apepars that you get only part way through loading the head section!)

This happens with IE 8 Version 8.0.6001.18702 on both XP Professional 5.1.2600 Service Pack 3 and XP Home Edition 5.1.2600 Service Pack 3.  I have been told, although I have not experienced it myself, that it happens on Vista and Windows 7 as well. 

Does anybody have any ideas for a workaround other than using the IE8 developer tools to set  "Always refresh from Server".   Keep in mind that this is a .jsp page so it is hard to re-aramge the order of stuff in the page.
