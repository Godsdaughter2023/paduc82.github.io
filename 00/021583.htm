ken sheridan say: It depends what you want to do.

1.  If you want to return everyone in the report, and show their ages you can do one of two things: (a) include a computed column in the query.  In design view this would be Age:CalcAge([BirthDate]) and bind a control in the report to the Age column. Or (b) include an unbound text box control in the report, with a ControlSource property of =CalcAge([BirthDate])

2.  If you only want to return those who are aged 5 then include a computed column in the query as above, and in its Criteria row in design view put 5

3.  If you want to return those who are aged 5 or above then include a computed column in the query as above, and in its Criteria row in design view put >=5

In either of the last two cases you don't need to include a control bound to the Age column if you only want to restrict the report's output, but not show the ages.

If the taxable price is in B2, the tax can be calculated in a parallel column (e.g. C2) as follows:

albert d. kallal say:

Keep in mind the setting for “no locks” is really ONLY the default setting for when you create a form. In other words, the setting really does not matter – since it is the setting used in the form that determines how editing will lock or not. So if you “change” that setting it will NOT affect existing forms – only set the default for new forms you create. So that setting is really more of a "default" setting for when you create a new form. A change in that setting will NOT effect existing forms.

>And would you expect such significant bloating (300 to 1,600MB) from the record locking?

Yes, often it is the case (it might not be in your case). Take this simple code:

   Set rst = CurrentDb.OpenRecordset("contacts")
  
   t = Timer
   Do While rst.EOF = False
      rst.Edit
      i = i + 1
      rst!City = "ci " & i
      rst.Update
      rst.MoveNext
   Loop
 
   t = Timer - t
   Me.Text1 = "done in " & t & " seconds"

If I run this on a 6 meg database, after running the above edit of 30,000 records – the resulting database is 6 megs (no real change). If I turn on row locking – the result is a whopping 126 megs (that is 120 megs of growth). So the difference in growth can be rather spectacular. I will admit that this setting may not necessary fix your issue – but it is a common result of using row locking (so you want to check or eliminate this issue).

Keep in mind that you need to ensure that all clients have this setting – since the FIRST user in who opens + runs the application will determine the setting for all additional users regardless of their client setting.

albert d. kallal say:

I would check if row locking is turned on.

File->Options->client settings

[x]Open databases by using record-level locking.

You want to un-check the above setting (and it requires a re-start.

The above option can substantially reduce bloat and thus the frequency of requiring a compact + repair. With this option turned on, then Access internally “pads” records to the size of a database page so only one record fits in that page. Access internal only has “page” locking, not row locking (thus this setting achieves true row locking by increasing the size of records to always take up one page).

So above setting can really effect "growth" or what we often here call bloat.
