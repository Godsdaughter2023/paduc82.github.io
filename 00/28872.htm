scottgem say:

Kristine,

As Daniel and John have noted, as a general rule we don't store calculated values. But you can easily do what you want using the Row source of the combobox (sometimes referred to as dropdown list). 

My standard RowSource for a combo that needs to select a person is:

SELECT PersonID, Lastname & ", " & firstname AS Fullname

FROM tablename

ODER BY Lastname, Firstname;

A couple of other tips if you don't mind.

Name your Primary Key fields tablenameID (ResumesID), don't just accept the default name of ID. This will make it clear what the link is when you use the field as a foreign key.

Don't use spaces in object names, this will come back to haunt you. Use camel notation (FirstName) or underscores (First_Name) instead. Use the Caption property when you need spaces in labels. 

Read up on Normalization, I can see at least three instances where you should be using child tables instead of LongText fields. For example: Technical Skills. You should have 2 tables here:

TL_Skills  The TL would indicate its a lookup table)

SkillID (PK autonumber)

Skill

You now have a many to many relationship. One applicant can have multiple skills and one skill can apply to multiple applicants. So you need a Junction table to model that relationship. Like:

TJ_ApplicantSkills   The TJ indicates a junction table

T_ResumeID (FK)

SkillID

ResumeID

You should also have child tables for References and Professional Designations, though I wouldn't use lookup tables for those so you don't need a junction table

For References I would use:

T_References

ReferenceID (PK Autonumber)

ResumeID (FK)

RefFirstname

RefLastname

RefStreetAddress

RefCity

RefState

RefZip

RefPhone

RefEMail

I also see you have a Posting table. That table should have an AutoNumber PK (PostingID) and that should be used as a Foreign key in T_Resume rather than the Short Text field you have for Job Posting Title.

scottgem say:

Hi Kristine,

The foundation of an efficient database is good table design. Before you start doing anything else, you need to get your design down right. I can't emphasize this more strongly. Cutting corners will just mean more work for you later on. And yes, your application calls for a database. Excel is not a database.

If you didn't get anything I said, please feel free to ask for clarifications. A Foreign Key is a field in a related table that identifies the record it is related to. Generally the relationship of Primary key to Foreign Key.

The reason why I suggested a separate Skills table is for searching. Lets say someone asks for a candidate with CAD skills. You can then easily pinpoint resumes that list that skill.

But no, Long Text is what you want for freeform text data.

As for the Postings table. Again, get your data structure done FIRST before you go further. Yes, get your postings table setup correctly and then change to a foreign key to relate the position being applied for to that table. This way you can pull all the other info about that posting easily. While I understand the idea of doing this in small chunks, you will be creating more problems for yourself if you don't get the tables right first. Even if the posting table is only Title and ID now, by setting this up properly with a relation, when you add more details about a position, you will already be set up to pull that data.

Let me add that Access is not an enhanced  spreadsheet program. It is a relational database with a great deal more power than Excel has to deal with data. You need to try and push back the spreadsheet mentality when working with Access. As you work with it more and more you will see many features and projects that it will do much better than Excel.

If you set the *limit* to a max of 50% of the CPU, MSE can not use more than 50%. If it doesn't need 50% it won't reserve it. If another program is consuming all available CPU, MSE will wait (like other processes) for cycles as managed by the CPU.

lz. say:

Hi,

Your 3rd IF (between blue parentheses) had no value_if_false. When an IF condition isn't true and no value_if_false is specified FALSE is returned. And when an IF condition is true and no value_if_true is specified TRUE is returned.

I removed a few parentheses (easier for fixing) in your formula. Please test the following before putting them back:

=IF( L10=AJ8, IF( K10="M", HLOOKUP(AJ7,Q:AB,10), IF( K10="Q", IF( AJ5="Q", HLOOKUP(AJ7,Q:AB,10) ), 0) ) )

Hope this helps
