john w. vinson say:

Fields that can be calculated should SIMPLY NOT EXIST in your table. If you store a TotalMiles or MilesSinceBase value in the table, you can be assured that the table will be WRONG, immediately after the car starts moving!

The table should store only static real-world data: VehicleCode, AsOfDate, and the milage reading as of that date. The MilesSinceBase, TotalMiles, and so on should be calculated on the fly with a query.

Could you explain the meaning of "AvgMilesPerYear"? Is that across vehicles, or for this vehicle? I presume that you will want to base a query on another query: first a Totals query grouping by Year([AsOfDate]) and summing the milage measurements for that year (with some provision for partial years, e.g. the car was put in service on December 17), and then a second query averaging those values.

If you're assuming that data must be in a Table in order to be reported, rethink that assumption: most reports are based on Queries, not on tables.

scottgem say:

OK, you need a few queries. One query To get the Base Date and beginning mileage:

qryBase

SELECT VehicleCode, AsOsDate, Mileage

FROM table

WHERE Base = Yes;

qryCurr

SELECT VehicleCode, Max(AsOfDate) As CurrentDate, Max(Mileage) As CurrentMileage

FROM table

GROUP BY VehicleCode;

Then

qryMileage1

SELECT VehicleCode, AsOfDate, Mileage, CurrentDate, CurrentMileage DateDiff("m",[AsOfDate],[CurrentDate] As Mths, DateDiff("yyyy",[AsOfDate],[CurrentDate] As Yrs, CurrentMileage-Mileage As TotalMiles

INNER JOIN ON qryBase.VehicleCode = qryCurr.VehicleCode

FROM qryBase, qryCurr;

Finallly

qryAnnualMileage

SELECT VehicleCode, TotalMiles, Mths,Yrs, TotalMiles/Yrs As AverageAnnual

FROM qryMileage1;

Email account info is maintained in an Outlook profile which survives any uninstall/re-install which is why when an inplace upgrade is performed, the new version of Outlook picks up all the information from the previous version even though the older version may already have been uninstalled.

Create a new profile using the Control Panel <Mail> app and make sure it is set to be the default profile the next time Outlook is opened.

autonumber the lines of a subform

My order form consists of a header with a subform in continuous view.  Many of my orders have 20 to 30 line items.   I would like to be able to have an autonumber generated for each line on the subform.  Of course I want the series to start at the number 1 for each new order.  This would make it easier for the user to stay on track as they enter orders.  Is there a way to do this?

ken sheridan say: You'll find an example of this in the file ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo illustrates a simple many-to-many relationship type and how to represent it in a form/subform.  The activities form in the parents form sets the DefaultValue property of the ActivityCounter control in the subform's Current event procedure as follows:

    Dim strCriteria As String
    
    strCriteria = "ParentID = " & Nz(Me.Parent.ParentID, 0)
    
    Me.AcitivityCounter.DefaultValue = """" & _
        Nz(DMax("ActivityCounter", "ParentActivities", strCriteria), 0) + 1 & """"

Note that the DefaultValue property of a control is always a string expression regardless of the data type of the column to which the control is bound, so should be wrapped in literal quotes characters as above.

This could of course give rise to conflicts in a multi-user environment where two or more users could get the same number simultaneously.  This would raise an error by virtue of the violation of the unique ParentActivity index which contains the parented and ActivityCounter columns.  In this demo the error is not handled, but you'll find an example of such error handling in the file CustumNumber.zip in the same SkyDrive folder.
