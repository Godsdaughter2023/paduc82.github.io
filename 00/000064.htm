Mike,

If you are still having this issue, I think I found the solution for our environment.  Look up the Java Plugin in the Manage Add-Ons section of IE and under More Info, get the Class ID.  In my case it was {8AD9C840-044E-11D1-B3E9-00805F499D93}.

If you lookup the following HKEY_CURRENT_USER Registry key entries (update the CLASS ID as needed):

32 Bit: HKEY_CURRENT_USER\Software\Classes\Wow6432Node\CLSID\{8AD9C840-044E-11D1-B3E9-00805F499D93}

64 Bit: HKEY_CURRENT_USER\Software\Classes\CLSID\{8AD9C840-044E-11D1-B3E9-00805F499D93}

You might see references to an older JRE version.  The fix is to just delete that whole key with a command similar to this:

reg delete "HKEY_CURRENT_USER\Software\Classes\Wow6432Node\CLSID\{8AD9C840-044E-11D1-B3E9-00805F499D93}" /f

Restart IE and the latest Java starts working.  Spent many hours trying to figure this one out and your post helped, so I am hoping maybe my solution will help you.

edwinarneson say:

We have had access to an application break in IE11 after a bunch of Windows patches were applied to Windows 7 PCs including KB3172605. But I haven't isolated the issue to that patch.

I have, though, watched the network traffic with Wireshark. The PC sends the TLS Client Hello packet and includes the ciphers it supports. The server replies back (Server Hello, Certificate, Server Key Exchange) and it selected one of the ciphers to use (TLS_DHE_RSA_WITH_AES_128_CBC_SHA; 0x0033). Then server sends a Handshake Failure (40) TLS packet, the session ends and the browser displays "page can’t be displayed" message. 

It's not clear what happened, but the server version is really old. The PC's Client Hello packet specifies TLS 1.2 (0x0303) while the server's reply specifies TLS 1.0 (0x0301).

If the patch is forcing IE11 to not use TLS 1.0 and the server doesn't support TLS > 1.0, then this may be causing the failure.
