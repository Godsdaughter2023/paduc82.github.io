This is called a Bill of Materials, and can be very complex. I doubt if the template will be able to handle it without a lot of modification. 

So you need at least 2 tables here, your Parts table which the template should already have and a components table like this:

tblComponents

CompDesc

ken sheridan say: If this is a bill of materials scenario as Scott describes, and I suspect he's right, there are a number of ways to model this, but the classical method is by means of an 'adjacency list'.  In this each item is represented by a row in a single table, whether they are indivisible base parts, complete assemblies or sub-assemblies which are components of assemblies (which can themselves be sub-assemblies  of course).

An item is identified as a component of another item by a row in the adjacency list table, which has two foreign key columns referencing the primary key of the main table of items, one as the major part, the other as the minor part, along with the quantity of the latter which are included in the former.  So the table is really modelling a many-to-many relationship type between two instances of the main table.

To produce a bill of materials (aka a 'parts explosion') it is necessary to recursively query the adjacency list.  The JET/ACE flavour of SQL used by Access does not support recursive querying, unlike some more sophisticated products, but it is possible to simulate it by using one of the three standard algorithms for this.  You'll find an example as BoM.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the BoM.accdb file builds a bill of materials by creating a temporary database at runtime and repeatedly inserting rows into its tables until no more need be added, i.e. when it reaches the level at which there are only indivisible base parts.

To incorporate this into an inventory database of the nature you describe, at its simplest would require you to maintain an inventory of the base parts which you would update as base parts are added to stock.  You'll see from my demo that the when the bill of materials is computed it calculates the total number of each base part (Parts 1 and 2 in my demo) in each assembly or sub-assembly, so if you were to remove an assembly from stock for dispatch the number by which the quantity of each base part in stock should be removed is available as the Quantity column for the MajorPart (the assembly) and the MinorPart (the base part) in question, e.g. to find the quantity of Part 1 items used in Assembly 2, both as direct components and via sub-assemblies which are components of Assembly 2

SELECT Quantity
FROM qryPartsExplosion
WHERE MajorPart = "Assembly 2"
AND MinorPart = "Part 1";

In reality you'd do this by referencing combo box control in an unbound form as parameters for the MajorPart and MinorPart columns of course rather than using literal values in the query.

This is a very simple scenario of course and my demo is only intended to illustrate the basic methodology for modelling a bill of materials.  Fully developed enterprise level inventory databases are very complex indeed.

Access Database Licensing - Microsoft Community

jc_2014 say:

Access Database Licensing

I created a generic database that I want a number of business' to use. Do they or I, need to pay any licensing fees to use this database? Furthermore, if it also uses other Microsoft applications like Excel to create spreadsheets, do they or I, also need to pay licensing fees for there use?
albert d. kallal say:

No, there no special licensing fees.

 

However the machine that you install your software on will require one of two things:

 

1)

The end user has purchased + installed MS Access and THEN you can install + run your software on that computer.

 

or

 

2)

The end user has installed the FREE RUNTIME edition of MS Access and THEN you can install + run your software on that computer.

 

So you have users install the free runtime. However to use the free runtime while costs nothing, you will have to spend extra time adding a user interface since things like the navigation pane and built in ribbons as a rule don't show.

 

So if your application already "hides" most of the Access interface and you provide startup forms etc. then adopting the free runtime system will not be much work. If your application requires the user to use the navigation pane, and launch forms on their own, then the runtime will not be appropriate.

 

As for using Excel or Word or Outlook for emailing? Again those programs will have to be installed on the target computer and thus the user will require paid version of those programs.

 

So in terms of special licensing for your database you are free to quite much do what you want and nothing special is required.

 

However the end user WILL require a version of Access to be installed that will ENABLE them to run your database. So that version of Access they use can be a paid full edition (like the one you are using to create the application) or as noted the free runtime edition that only lets you "run" the application but no do any design changes.

 

I explain many details about the runtime edition of Access here:

http://www.kallal.ca/RunTime/Index.html
