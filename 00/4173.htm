mr.anderson_503 say:

Now don't get your hopes up, I have a feeling you may still need Office365 (O365) Support when I'm finished, here.  ;^)

 

When you do, you will need to know specifically what kind of O365 Account you are using (Home, Small Business, Enterprise, etc.) and if you are using a Company Hub or an Enterprise Network.

 

1) As I understand it, it's not possible to use the "same login" on two services (MSN and Office 365).

 

Why not? I find it helpful to think of any Microsoft-related "identity" (login) as what they now call a "Microsoft Account".  Your MSN login is a "Microsoft Account" for MSN. You'll need a separate  "Microsoft Account" to use with O365.

 

2) The Microsoft Account you used to set up your new Phone is now treated as your "Primary Account". You cannot delete you Primary Account.

 

You CAN use another Microsoft Account as the Primary on your phone, but only after you "hard reset" your device, and start fresh.  (A hard reset removes all data, apps, documents and personalization you've applied to your phone, returning it to the state it was in when new. Proceed with caution.)

 

3) We used O365 Small Business at my last job, and here's what I remember:

 

We were told that the default setup for the WP Office Hub is essentially the same as it would be using an O365 account, EXCEPT that the Hub is "connected" with a specific O365 Account; as opposed to setting up new Outlook.com and Skydrive accounts "from the phone".

 

In other words, when you initially set up your Phone, your Office Hub pointed to a Skydrive and Outlook.com email for your MSN, and created the Live Tile to match those settings.

 

If you subscribed to O365 using the SAME account as the one used to setup your phone, then your account login will sync to that O365 account for you.

 

If you subscribed to O365 using a DIFFERENT account from the one used to setup your phone, Tap the Office Hub, Tap "Office 365" and follow the instructions to "Set up your Office 365 account on your phone...".  When you've finished you will not see anything different.  Your hub should now be associated with your office 365 account, which means the correct Skydrive, your Office documents, and will setup a Outlook.com email and Live Tile for the Office 365 account.

 

I know that's less-than-a-simple answer, but I thought it might give you more to go on that the other (zero) answers you've received since posting a month ago.  ;^)

I was just wondering whether it might be the code of a field code such as a LINK field. These might look something like 

{ LINK Word.Document.12 "/Volumes/someone/somedoc.doc" "OLE_LINK1" \a \p }

or  

{ INCLUDEPICTURE  "the path name of some picture" }

Field codes have a code, and a result. The code usually looks something like the above, but the result will typically be text/tabular material/a picture or some combination of those things.

If that's the problem, two things might help:

1. Select the field code and press shift-command-F9 (or fn-shift-command-F9 depending on your keyboard setup). In many cases the code should then be replaced by the result - i.e. the link/code will then be lost. (command=cmd). 

2. Press option-F9 (alt-F9) or fn-option-F9/fn-alt-F9, which changes Word's display so that it shows the result, not the code. In this case, the link/code remains.

ken sheridan say: In the hypothetical scenario I described you'll notice that the States table is not actually used.  So why have one?  The answer is that it protects the integrity of the data because you can create a relationship between Cities and States on the State columns and enforce referential integrity.  By having only legitimate states in the States table only valid states can be entered in the cities table, and the state value entered must be consistent.  I can testify to how easily slight inconsistencies can arise (through simple typos) as in one databases I once found three variations on my own name as author of technical papers in my own field of work.   A separate Authors table where I was represented by only one row would have prevented this.

So, while you don't need a States table for the set up I described, it's advisable to have one.  What is important, however, is that you have a CityID 'surrogate' key as city/town names are not distinct, even within a state.  There are five Springfields in Wisconsin for instance.  See:

http://en.wikipedia.org/wiki/List_of_the_most_common_U.S._place_names

Another reason for having a States table BTW is that, even if a state is not represented in any referencing tables in a database you can still query against that state, e.g in a sales database you might want to know which states a company has sold
