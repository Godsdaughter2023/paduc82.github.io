karl timmermans [outlook mvp 2012-2018] say:

I have a gmail account (IMAP protocol) profile in Outlook  that seems to be taking up far too much disk space, worsened by Outlook duplicating emails (your tech support have repeatedly failed to fix this) and everyone adding huge attachments for no good reason.  My ISP warned me that IMAP would take up a lot of space and I'd need to move the data at some point.

First step is making sure we're all on the same page in terms of the comment "taking too much space".

#1 - The data storage related to an Imap account is purely a reflection of the email data that resides on the server so if you have what you consider to be excessively huge amounts of data - then you primary focus needs to be what you "want" and "need" to keep on the server

#2 - Also need to understand that if you delete something in Outlook, it gets deleted at the server level (and vice versa).

Having said the above, the simplest solution to manage your email folders on an on-going basis is to "archive" them - either manually or automatically. There is no shortage of articles on how to go about that by doing a simple search using a phrase similar to "How to Archive email in Outlook 2016" (or any variation thereof) - some articles on the topic to get you started are

 
Using Outlook's AutoArchive Feature
http://www.slipstick.com/outlook/archive-outlook/using-outlooks-autoarchive-feature/
 
Archive older items manually in Outlook for Windows
https://support.office.com/en-us/article/Archive-older-items-manually-in-Outlook-for-Windows-fa03020e-55af-4eeb-94fb-925f640d942d?ui=en-US&rs=en-US&ad=US&fromAR=1
 
How to AutoArchive Your Email in Outlook 2016
http://www.dummies.com/how-to/content/how-to-autoarchive-your-email-in-outlook-2016.html

An alternative approach to archiving, is moving your email items that reside on the server (and by definition on your local computer) to different PST files according to whatever filing approach makes sense to you. There is no single "one size fits all" answer in terms of "what't the best way to retain historical data" - everyone has their own approach and retrieves info in their own way. Personally, have no use for storing historical data by "timeframe" - I have some data that goes back some 15 yrs organized in "subject category" oriented PST files with folders within those PST files further refining the data but that doesn't make that approach any better or worst then those who maintain data by calendar year and then do searches.

Just remember, unless you remove items at the server level, won't make any difference at the desktop level so deleting and then re-adding the Imap account won't solve anything.

john - project say:

Joe,

Just for reference, are you the original poster (i.e. Pete213)?

There are much better and more reliable ways to accomplish what I think you are trying to do. First of all, I wouldn't use paste links. As I indicated in my original response, paste links are probably the most fragile type of link. They do work, but unless all users are expert users of Project and have superb file management discipline skills, paste links will lead to file corruption. That may be why your links are not updating now.

Question, do you have SP2 installed? If not, install it. It fixes a host of issues with Project 2010.

I mentioned there are better ways to do what I think you want. The first, although not the best is to use external predecessors/successors from each of the individual files to the milestone file. A second similar approach is to create a dynamic master by inserting each of the individual projects into a master. Apply a filter to isolate the desired milestones at master level. However, both of the above approaches are still subject to file corruption, although somewhat less than with paste links. None of the files can be renamed, moved, or saved off separately and all files should reside on a local drive in a single folder.

The best approach would be to create a static master each time you want an update. A static master is created by unchecking the "link to project" option in the lower right corner of the Insert Project window. A static master is a wholly separate file that is a snapshot in time at the time of creation. It will not have dynamic updates, but, it is also will not be subject to file corruption. And yes, a new static master must be created each time an update is desired but this can be automated with a macro by simply recording a macro the first time it is created.

John
