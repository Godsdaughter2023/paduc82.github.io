graham mayor say:
There is no practical way to do this on the fly so that a worksheet is kept constantly up to date with the edits in your documents. The most practical way I can think of would be to keep the documents in question in a separate folder then run the following Word macro to update the counts of all the documents in that folder. In the example code you will need to change the two bold statements to reflect the name and path of the (initially empty) worksheet and the path of the folder containing the documents as shown. Running the macro will update the worksheet with the word counts of all the documents in the folder. The macro works with DOCX format documents (change to DOC if necessary)

http://www.gmayor.com/installing_macro.htm

Dim xlApp As Object
Dim xlbook As Object
Dim oDoc As Document
Dim NextRow As Long
Dim strFilename As String
Const strPath As String = "D:\My Documents\Test\Merge\"
Const strWorkBook As String = "D:\My Documents\Test\Merge\WorkbookName.xlsx"
On Error Resume Next
Set xlApp = GetObject(, "Excel.Application")
If Err Then
    Set xlApp = CreateObject("Excel.Application")
End If
On Error GoTo 0
Set xlbook = xlApp.Workbooks.Open(FileName:=strWorkBook)
xlApp.Visible = True
xlbook.Sheets(1).Cells.Clear
NextRow = xlbook.Sheets(1).Range("A1").Row
strFilename = Dir$(strPath & "*.docx")
While Len(strFilename) <> 0
    WordBasic.DisableAutoMacros 1
    Set oDoc = Documents.Open(FileName:=strPath & strFilename, AddToRecentFiles:=False)
    xlbook.Sheets(1).Cells(NextRow, 1).Value = oDoc.Name
    xlbook.Sheets(1).Cells(NextRow, 2).Value = oDoc.BuiltInDocumentProperties(wdPropertyWords)
    NextRow = NextRow + 1
    xlbook.Save
    oDoc.Close wdDoNotSaveChanges
    WordBasic.DisableAutoMacros 0
    strFilename = Dir$()
Wend
xlbook.Sheets(1).Cells(NextRow, 1).Value = "TOTAL"
xlbook.Sheets(1).Cells(NextRow, 2).Value = xlApp.WorksheetFunction.Sum(Columns("B:B"))
xlbook.Save
Set xlApp = Nothing
Set xlbook = Nothing
Set oDoc = Nothing

suzanne s. barnhill say:

I'm guessing that you have applied numbering to the built-in heading styles. The "Style for following paragraph" for all those styles is Normal, so when you press Enter at the end of a heading in Print Layout view, you will get a Normal paragraph.

If you are creating an outline, you can work in Outline view, in which the numbering/styles will behave as you expect, or you can use nonheading styles. It depends on what your plans for the outline are. If you want to enter all the headings first, intending to fill in the body text later, then working in Outline view makes sense. But if the outline will be the entire document, then you would be better advised to link your outline numbering to nonheading styles such as the List Number series.

If you want to continue with headings, you can change the "Style for following paragraph." If the document will consist of nothing but headings, you should also disable "Keep with next" for all but the top couple of levels.

scottgem say:

Well first you need to understand that quantity on hand is a calculation, not a stored number. QOH is calculated by adding the incoming transactions and subtracting the outgoing ones. 

So the way I would handle this is to store the case size as part of the Product record. Then, in your query that calculates QOH, divide the individual quantities sold by the Case size and truncate to the lowest integer value. I would assume incoming transactions would only be by case, so those wouldn't have to be adjusted.

scottgem say:

Sure. Your Product table would contain the info about the how many to a case, it would include a Unit Price per bottle and a discount price per case. If you need to you can have further discounts per lot size in a child table. For example:

john - project say:
Wendi,
One of the don't dos with regard to master/subproject structures is to save off files to a different location. As you found out it duplicates the link structure and will, (not may, but will), eventually lead to corruption of the link structure.

So how do you keep this from happening? The most direct way is training. Set up a series of ground rules for file editing and maintenance. Have one individual with write access to the files and does all editing, maintenance, and "what ifs". Everyone else has read access but is not allowed to save or edit the files. Obviously this requires training, discipline and auditing.

However the easiest approach is to not have two files and the tenuous linked structure it requires. Put everything in a single file with the milestone under their own summary at the top of the file. This is my recommendation. Is there a compelling reason you want/need two separate files?

If for some reason you have to have two separate files and want a more robust approach to sharing the files among multiple individuals, the two files can be converted from a dynamically linked master/subproject structure to a static file that has all the external links intact. However this cannot be done by simply converting the dynamic master to a static master by unlinking. The cross-project links will not be maintained. To convert this type of structure, you need a VBA macro to preserve the links. I have a macro that does that, but it is not freeware. However, if you are interested, you can contact me at the address below.

john - project say:
Wendi,
If you need the schedules to be linked, then your best option is with a dynamic master/subproject structure. However, even with that structure only one individual can have write access to a given file at a given time. The first one to log in gets write access, the rest are read only, which is the way it should be.

If you need multiple users to have write access, you could set up a user schedule. For example, Joe gets access from 8 to 9, Mary from 9 to 10, etc. Of course if different users are involved with each of the two files, then each file can be separately edited simultaneously (i.e. Joe works on the scheduler file at the same time Mary is working on the integrator file).

Or, maybe its a matter of breaking your two files into multiple files such that each file is maintained by a single individual. Cross-project links can be set up as necessary and a master can be created to pull the whole structure together. This is really the only way to allow multiple users simultaneous write access. The disadvantage is that it is a more complex link structure and training and discipline are of utmost importance to avoid file corruption.

Although I don't use it, have you considered moving to Project Server?
