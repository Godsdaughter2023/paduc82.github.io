hansv mvp say:

Assuming that the Type data are in A2:A6 and the Value data in B2:B6, use the following array formula, confirmed with Ctrl+Shift+Enter:

=SUM(IF($A$2:$A$6="Single",1/COUNTIFS($A$2:$A$6,"Single",$B$2:$B$6,$B$2:$B$6)))

If you enter the word Single in a cell, for example in F1, you can use

=SUM(IF($A$2:$A$6=F1,1/COUNTIFS($A$2:$A$6,F1,$B$2:$B$6,$B$2:$B$6)))

If Couple is in F2, you can fill this version of the formula down to get the unique values for "Couple".

amit tandon say:

Please refer below image:

All formulas to be entered as array formula (CTRL+SHIFT+ENTER). All formulas work even if column B has blanks when column A contains "Single".

Array Formula in cell D2 - Does not count blanks as unique ie. If column A contains "Single" & column B is blank, then the blank is NOT counted as unique:

=SUM(($A$2:$A$10="Single")*IFERROR((1/COUNTIFS($A$2:$A$10,"Single",$B$2:$B$10,"<>",$B$2:$B$10,$B$2:$B$10&"")),0))

Array Formula in cell E2 - Counts blanks ie. If column A contains "Single" & column B is blank, then the blank is counted as unique:

=SUM(IF($A$2:$A$10="Single",1/COUNTIFS($A$2:$A$10,"Single",$B$2:$B$10,$B$2:$B$10&"")))

Array Formula in cell F2 - Counts blanks ie. If column A contains "Single" & column B is blank, then the blank is counted as unique:

=SUM(($A$2:$A$10="Single")*IFERROR((1/COUNTIFS($A$2:$A$10,"Single",$B$2:$B$10,$B$2:$B$10&"")),0))

With a 32GB SSD drive you may not have enough free space available to update via the download and install method. Which is why a clean install of the latest build may be the best route.

trishp say:

Agreed that those who are experiencing this need to let Microsoft know - through the official channel of the Feedback Hub, which is the most efficient way.  Posting here may or may not be seen by the right person or persons at Microsoft in a timely fashion, and while it is useful to Insider peers to know and worth posting here, for sure, it should never be expected to be a reliable communications channel to Microsoft engineers - it's not the purpose of this group.

While your point about workarounds and fixes is valid, they can be very useful to post in many cases - you might not realize how many people here are actually pretty good amateur or professional software testers and developers who can use that information to help uncover what might be going on, plus there are people who would really like to be able to use their systems before a fix happens.  Additionally, if it's NOT a Microsoft bug (and it quite often isn't really a bug, just a piece of 3rd party software or a driver that hasn't "caught up" to the new code in the current build), it might very well not be something Microsoft would fix anyway.  Having a workaround allows people to continue to test a build.  That doesn't mean I think you should automatically use a workaround, but when it's a situation where the bug is already reported and known, and you can get around it with a workaround, you might be able to continue to test and provide feedback on other components of the build so that time isn't wasted.

As far as testing, yes, of course it's tested on systems at Microsoft.  Certainly it's run through an extensive automated test suite in labs every single day, and is also "dogfood" that's "eaten" (i.e., installed on devices that people actually have in their offices and use) by people in the Windows organization at the very least.  That doesn't mean they necessarily have the combination of hardware and software that is impacted by this. That's the whole reason for releasing a build to a beta group like the Insiders - to get exposure on many more combinations than could possibly be tested otherwise, and also find out if there IS a particular problematic combination that needs to be tested that isn't in their matrix of equipment.  And yes, it's also possible that it was missed for some other reason - maybe the system or systems that might have caught this particular issue were being held to track down a different bug they exhibited and didn't get run for this build, for example.

It's definitely a pain when something comes up like this,  but it's part of the life of a beta tester.  That's why we are supposed to use systems that we can afford to be without for a while when (not if) something goes sideways with a build.
