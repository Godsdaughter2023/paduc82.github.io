How to build an inspection checklist database? - Microsoft Community

ubatu say:

How to build an inspection checklist database?

I am an HSE officer trying to build a worksite inspection database. Please how do I start? 

dhookom say: The huge trap to avoid is to create fields with names that describe an inspection point. For instance if I was inspecting a home, I would not create a table with a ton of fields like [Driveway Condition], [Windows], [Livingroom Flooring], [Water Heater], ... These are data values that belong in tables, not in names of fields.

You should be able to add different inspections and categories without editing any tables, forms, queries, or reports.

When I have created solutions like this in the past, I basically have one table with only a few fields that contains all inspection detail results. This table would have fields like:

tblInspectionDetailResults
idrIDRDID autonumber primary key
idrInEID field that links to a table of inspection events like
a particular worksite on a particular day
idrInsID field that links to what specifically was inspected like an entrance
idrScore a numeric field that stores 0 - 10 maybe to rate the item
idrComments field to store comments

You could add more fields to store if you need to do a followup or whatever.

As you see, you will need to have an Inspection Events table that has a record for each worksite visit. Another table would store the "what was inspected" list.

I have a demo survey application at http://www.rogersaccesslibrary.com/forum/at-you... which might give you an idea of how to create your tables and forms. Think of an inspection like a survey.

scottgem say:
The way one starts designing a database is to first determine all the data points you need to capture. Once you have that, you then organize those data points into entities which become your tables. Only after you have a proper table design do you begin to design your workflow objects like forms and reports.

bob jones aka: cybertaz say:
Word for Mac never has used the dreaded 'drawing canvas' implemented in the PC version. Essentially, a PC Word canvas & anything within within it is treated as a single object when the document is opened on a Mac. There should be handles but they serve to scale the entire object - the individual components cannot be modified separately.

You most likely are unable to "slide the box over" because the object is formatted as In Line with Text. If you select the object then go to Format> Format Object - Layout you can apply Text Wrapping which will enable dragging the object around.

The List & Combo Box controls are also treated as graphic objects. You didn't specify which version of PC Word was used but those terms suggest that it was either 2007 or 2010. Also, although there are no details as to how the form was constructed, I would guess that the actual tools used were either ActiveX Controls or Content Controls, neither of which are supported on a Mac. They're displayed as picture objects & do not operate as they do in the Windows version. The only controls that operate properly cross-platform are the Legacy Forms Controls.

IOW, I'm not sure you can simply "bring it back to a PC and fix it" :-) It sounds as though the form will have to be recreated taking into consideration the requirements & limitations of both environments.

Regards,
Bob J.

tom van stiphout say:

You already got the answer you were looking for, but I want to explain why it is "Admin" for everyone.

It's because Access traditionally (up to A2003) had a security system called Workgroup Security where by default the current user was Admin. You could create other users in that system, and give them certain rights.

That system was done away with, but since it is so ingrained into the code, and not worth fixing, and not worth breaking existing applications, they just left that in, with the default user. Nobody uses it anymore.
