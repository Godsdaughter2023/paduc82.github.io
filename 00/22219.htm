ken sheridan say: The code does not make a lot of sense.  Firstly it checks for If rst!RecordDate = Begin_Date, and then for If rst!RecordDate >= Begin_Date Or rst!RecordDate <= End_Date, but the latter must evaluate to  TRUE as it will only be executed if the RecordDate is the value of Begin_Date.  I suspect what's really intended is to identity a date which falls within the range determined by the Begin_Date and End_Date values.

It then seems to check for the lane being any of the 24 lanes.  Assuming that there are no phantom lane numbers entered less than 1 or greater then 24, this would only make sense if the Lane column could be NULL.

So taking these points into account it looks to me like the whole thing can be done a lot more simply and efficiently by executing  a simple 'append' query which references the controls on the form as parameters:

PARAMATERS Forms!YourForm!Begin_Date DATETIME,
Forms!YourForm!End_Date DATETIME;
INSERT INTO ReviewTbl1(CallIDNbr, RecordDate, TechNbr, CallCatNbr, Lane,ShiftFlag,CallTime)
SELECT CallID, RecordDate, TechNbr, CallCatNbr, Lane,ShiftFlag,CallTime
FROM CallsTbl
WHERE RecordDate BETWEEN Forms!YourForm!Begin_Date AND Forms!YourForm!End_Date
AND Lane IS NOT NULL;

NB: it's important that the parameters are declared as DateTime to avoid any possibility of their values being inadvertently misinterpreted as arithmetical expressions rather than dates.

A more fundamental question, however, is whether this needs doing at all.  It seems to me that all the rows currently in the ReviewTbl1 table can be returned at any time by a simple SELECT query, so the ReviewTbl1 is redundant, and its removal would eliminate the possibility of update anomalies, to which the database is currently at risk.
