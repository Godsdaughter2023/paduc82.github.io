OnTime Saves the Day - Microsoft Community

zorvek say:

OnTime Saves the Day

Technical Level : Expert

Summary
The Application.OnTime method is often used to initiate a delayed activity such as executing code at a specific time or repeatedly running code every so many seconds, minutes, hours, days, etc. Beyond the obvious, the OnTime method can also be used to solve some interesting problems. The OnTime method is available in all Office applications but this article focuses on Excel.
Details

OnTime Basics

The Application.OnTime method is most often used to run code at a specific time or at some interval of time. The method has four parameters:

Earliest Time: A date/time value that determines when the code is to be executed. This parameter is required. If a date/time value earlier than now is passed the code will be executed as soon as possible.

Procedure: The name of the procedure to be executed. The procedure can be in a general module, the workbook module, or a worksheet module. Simple parameters can be included. This parameter is required.

Latest Time: The latest time at which the procedure can be run. If Excel is unable to execute the procedure by this time then the code is not executed. Pass a date/time value. This parameter is optional. If omitted Excel will try to execute the specified procedure when it can without any cut-off time.

Schedule: Pass True to enable the timer, False to delete the timer if it already exists. A timer is specified by the Earliest Time and the Procedure values. This parameter is optional. If omitted then True is assumed.

OnTime Caveats

If you create an OnTime event and the workbook containing the procedure to be executed is not open when the event triggers, Excel will open the workbook to run the code. This can be problematic, especially when you have created a repeating event. To prevent this from happening, delete any existing pending OnTime events when the workbook is closed. This is easily done by maintaining a global date/time variable that contains the last set timer date/time value and, when the workbook is closed, calling OnTime while passing False for the Schedule parameter.

Whenever an OnTime event is used, the code it executes should be kept to a bare minimum to avoid any interference with the user's experience. This includes displaying any dialogs and user forms. If the code has to do anything that requires time, either liberal use of DoEvents (to give Excel time to handle user actions), or ample notification to the user, or both, will help mitigate the negative impact on the user's experience.

If typing into the Name Box in the Formula Bar, Excel will close the Name Box when the OnTime event is triggered. If a recurring OnTime event is running every few seconds, this can be very annoying when trying to use the Name Box to navigate to named ranges or apply names to ranges.

If a recurring OnTime event is being triggered every few seconds or more frequently, the user will experience a flickering mouse icon which can be confusing and irritating.

A frequently recurring OnTime event will force the VBE to compile code being edited. This can be annoying when developing code—a closing quote will appear when entering a text string, spaces at the end of a line will disappear, and compile errors will be displayed. One way to circumvent this is to use conditional compilation that disables the code that generates the OnTime event when developing.

Scheduling a Task

The most common use of OnTime is to schedule code to be executed at a specific time or some number of seconds, minutes, hours, etc. from now. These two examples execute the code at 12:30 PM:

Application.OnTime TimeSerial(12, 30, 0), "Procedure"

Pass Now + TimeSerial(hours, minutes, seconds) for the parameter Earliest Start to execute the code a specific number of seconds, minutes, and/or hours, etc. from the current time. The next example executes the code 15 seconds from now:

Application.OnTime Now + TimeSerial(0, 0, 15), "Procedure"

Note that the date part of the date/time value is not being passed. In this case OnTime assumes the current date. The next examples execute the code at 12:30 PM on January 31, 2015:

Application.OnTime DateSerial(2015, 1, 31) + TimeSerial(12, 30, 0), "Procedure"

Application.OnTime #2015/1/31 12:30:00 PM#, "Procedure"

This example executes the code at 12:30 PM tomorrow:

Application.OnTime Now + 1 + TimeSerial(12, 30, 0), "Procedure"

Scheduling a Recurring Task

The OnTime method does not support recurring tasks so you have to write code to do so. The code below illustrates how to create a recurring OnTime event that triggers every five seconds:

Private mTimerTime As Date

Public Sub StartTimer()

   StopTimer

   DoTimer

End Sub

Public Sub DoTimer()

   ' Do timer work here

   ' Reset the timer

   mTimerTime = Now + TimeSerial(0, 0, 5)

   Application.OnTime mTimerTime, "DoTimer"

End Sub

Public Sub StopTimer()

   On Error Resume Next

   Application.OnTime mTimerTime , "DoTimer", Schedule:=False

   On Error GoTo 0

End Sub

Remember to delete the OnTime event when the workbook closes so that Excel won't try to reopen the workbook when the event triggers. Do this in the ThisWorkbook_BeforeClose event handler by calling the StopTime routine defined in the above example:

Private Sub Workbook_BeforeClose(Cancel As Boolean)

    StopTimer

End Sub

Referencing Procedures in Non-General Modules

Any public or private procedure in a general, worksheet, or workbook module can be executed. To reference a procedure in a worksheet or workbook module, include the code name of the module followed by a period before the procedure name as shown in these examples:

Application.OnTime Now, "ThisWorkbook.Procedure"

Application.OnTime Now, "Sheet1.Procedure"

Passing Parameters

Parameters can be passed using the OnTime method by including them as normally done in a regular statement. The examples below illustrate passing a single parameter to the OnTime routine:

Application.OnTime Now, "'Procedure 1'"

Application.OnTime Now, "'Procedure ""String Value""'"

Note that the entire string is enclosed in single quotes. Without the single quotes the OnTime event will fail.

To pass more than one parameter, separate them with commas:

Application.OnTime Now, "'Procedure 1, ""String Value""'"

In the above example the first parameter is the numeric value 1 and the second is the string "String Value".

Recurring Task Magic

When setting the next OnTime event, specific intervals at specific times can be specified. Below are some common intervals and times. These examples assume you are using the example code above.

Once a minute on the whole minute:

mTimerTime = Now + TimeSerial(0, 0, 60 - Second(Now) Mod 60 + 1)

Once every five minutes on the whole five minutes:

mTimerTime = Now + TimeSerial(0, 5 - Minute(Now) Mod 5 - 1, 60 - Second(Now) Mod 60 + 1)

Once every 10 minutes on the whole 10 minutes:

mTimerTime = Now + TimeSerial(0, 10 - Minute(Now) Mod 10 - 1, 60 - Second(Now) Mod 60 + 1)

Once every 15 minutes on the quarter hour:

mTimerTime = Now + TimeSerial(0, 15 - Minute(Now) Mod 15 - 1, 60 - Second(Now) Mod 60 + 1)

Once an hour on the whole hour:

mTimerTime = Now + TimeSerial(0, 60 - Minute(Now) Mod 60 - 1, 60 - Second(Now) Mod 60 + 1)

Once every three hours on the whole three hours:

mTimerTime = Now + TimeSerial(3 - Hour(Now) Mod 3 - 1, 60 - Minute(Now) Mod 60 - 1, 60 - Second(Now) Mod 60 + 1)

Once every six hours at midnight, 6 AM, noon, and 6 PM:

mTimerTime = Now + TimeSerial(6 - Hour(Now) Mod 6 - 1, 60 - Minute(Now) Mod 60 - 1, 60 - Second(Now) Mod 60 + 1)

Once every 12 hours at noon and midnight:

mTimerTime = Now + TimeSerial(12 - Hour(Now) Mod 12 - 1, 60 - Minute(Now) Mod 60 - 1, 60 - Second(Now) Mod 60 + 1)

Once every 24 hours at midnight:

mTimerTime = Fix(Now) + 1

The next day at a specific time (9:45 AM for example):

mTimerTime = Fix(Now) + 1 + TimeSerial(9, 45, 0)

Post Printing Restoration

When printing, sometimes we need to change the layout and formatting of our worksheets to print them so that they look good on paper. We use the ThisWorkbook_BeforePrint event handler to do this. But there is no obvious way to restore the layout and formatting so the worksheets look good on the screen. We can use an OnTime event to do this.

The code below illustrates how to format all of the worksheets about to be printed. The code that manages the formatting is placed in the ThisWorkbook code module. The code that formats each worksheet is placed in the worksheets.

Place this code in the ThisWorkbook code module:

Private mFormattedForPrinting As Boolean

Private Sub Workbook_BeforePrint(Cancel As Boolean)

   Dim FocusSheet As Worksheet

   ' Attempt to call the FormatForPrinting routine in each selected worksheet

   On Error Resume Next

   For Each FocusSheet In ActiveWindow.SelectedSheets

      FocusSheet.FormatForPrinting

   Next FocusSheet

   On Error GoTo 0

   mFormattedForPrinting = True

   Application.OnTime Now, "ThisWorkbook.Workbook_AfterPrint"

End Sub

Private Sub Workbook_AfterPrint()

   If Not mFormattedForPrinting Then Exit Sub

   ' Attempt to call the FormatForPrinting routine in each selected worksheet

   On Error Resume Next

   For Each FocusSheet In ActiveWindow.SelectedSheets

      FocusSheet.FormatForViewing

   Next FocusSheet

   On Error GoTo 0

  mFormattedForPrinting  = False

End Sub

Place the following code in any worksheet code module requiring special formatting.

Public Sub FormatForPrinting()

   ' Code to format the workbook for printing.

End Sub

Public Sub FormatForViewing()

   ' Code to format the workbook for viewing.

End Sub

How does this work? The BeforePrint event is generated when showing the print preview and when printing. This means that two calls might be made to a specific format routine in sequence. Therefore each worksheet's FormatForPrinting and FormatForViewing routine must be able to either detect which mode the worksheet is in and not format if already in the desired format, or be able to format the worksheet to its intended state without regard to how it is currently formatted. Additionally, the OnTime event is not triggered until all requested printing activity has been completed. This behavior allows the OnTime request to execute the custom "AfterPrint" routine without that execution overlapping the printing process.

Display a User Form with the Current Time Displayed and Updated Every Second

A user form can be displayed with an automatically updated time or clock by using the OnTime method. Add a label to a user form and name it lblTime. Add the code below to that user form's module. Note that this solution uses frequent OnTime events which, if the user form is non-modal, cause some user experience issues as discussed above.

Public NextTime As Date

Private Sub UserForm_Activate()

   If NextTime <> 0 Then Application.OnTime NextTime, "UpdateTime", Schedule:=False

   UpdateTime

End Sub

Public Sub UpdateTime()

   NextTime = Now + TimeSerial(0, 0, 1)

   lblTime = Format(Now, "H:MM:SS AM/PM")

   Application.OnTime NextTime, "UpdateTime", Schedule:=True

End Sub

Private Sub UserForm_QueryClose(Cancel As Integer, CloseMode As Integer)

   If NextTime <> 0 Then Application.OnTime NextTime, "UpdateTime", Schedule:=False

End Sub

Because a routine in a user form cannot be used with the OnTime method, the routine below need to be added to a general module. It assumes the form name is frmTime.

Public Sub UpdateTime()

   frmTime.UpdateTime

End Sub

The form can be shown as either a modal or modeless form.

Waiting for Excel to Load (and Preventing 1004 Errors)

In later versions of Excel (especially 2013), when Excel is executed while opening a workbook that executes code upon opening, that code can cause a 1004 error if Excel has not completely loaded all of its resources. It's not clear why Excel allows workbook VBA code to run before it is ready but it does. If you experience a 1004 error when opening a workbook in a new instance of Excel, put the automatic open code into a second routine and use DoEvents from within the Workbook_Open event handler to execute it. This will give Excel a chance to finish loading. The example code below illustrates this:

Private Sub Workbook_Open()

    Application.OnTime Now, "ThisWorkbook.Workbook_AfterOpen"

End Sub

Private Sub Workbook_AfterOpen()

    ' Do any workbook open tasks here

End Sub

Note that the OnTime event is not delayed. Just creating the event is usually enough for Excel to take some time to finish loading.

Two instances where you could see this type of error is using the Workbook_Open event handler, or having a customUI ribbon onLoad routine called. These both have the potential to cause the 1004 error in later versions of Excel if Excel is not fully loaded. Sometimes these errors are not traceable or debuggable as Excel will effectively freeze and even terminate.

Switching from One Workbook to Another

When when one workbook opens another workbook and the second workbook has to do something with the first after it has closed, you will end up with a deadlock if the second workbook, in its Workbook_Open event handler, tries to wait for the first workbook to close. This is because the Open method is asynchronous in that it waits for the other workbook to open and run any open event handlers. If the code in the second workbook's open event handler waits for the first workbook to close, a deadlock will occur. This is resolved by creating, in the second workbook's Workbook_Open event handler, an OnTime event to run a second routine that waits for the first workbook to close. This allows the Workbook_Open event handler code to continue and exit which frees the first workbook to continue and close itself. Below is the code used in the second workbook:

Private Sub Workbook_Open()

    Application.OnTime Now, "ThisWorkbook.Workbook_AfterOpen"

End Sub

Private Sub Workbook_AfterOpen()

    ' Wait for the first workbook to close here

End Sub

Determining if Running Within Internet Explorer

The only known method for determining whether or not a workbook is running in Internet Explorer is to examine the Container object of the Application object. The problem with this method is that the Container object is rarely initialized when the workbook Open event is called although it is usually initialized shortly thereafter. The code below illustrates how to handle this deficiency by using OnTime.

With the code below in place, query ThisWorkbook.IsRunningInIE any time after initialization to determine whether or not the workbook is open in Internet Explorer:

If ThisWorkbook.IsRunningInIE = True Then MsgBox "Running in IE"

If ThisWorkbook.IsRunningInIE = False Then MsgBox "Not running in IE"

If IsEmpty(ThisWorkbook.IsRunningInIE) Then MsgBox "Not yet known whether running in IE or Excel"

Add the code below to the ThisWorkbook code module.

Public IsRunnnigInIE As Variant

Private Sub Workbook_AfterOpen()

' Handle the post open timer event generated in the Workbook_Open routine. This

' routine is called every second until the Container object contains a valid

' object or five seconds have elapsed.

    Dim Container As Object

    Static Count As Long

    On Error Resume Next

    Set Container = ThisWorkbook.Container

    If Not Container Is Nothing Then

        IsRunnnigInIE = Container.Name = "Microsoft Internet Explorer"

        ' Call additional initialization routines from here

        Exit Sub

    End If

    Count = Count + 1

    If Count > 5 Then Exit Sub

    Application.OnTime Now() + TimeSerial(0, 0, 1), "ThisWorkbook.Workbook_AfterOpen"

End Sub

Private Sub Workbook_Open()

    Application.OnTime Now(), "ThisWorkbook.Workbook_AfterOpen"

End Sub
