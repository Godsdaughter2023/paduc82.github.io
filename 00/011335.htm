I suspect that it's your reference method that is causing the problem.  I
have not done a lot of work with datasheets and subdatasheets, but in
dealing with forms, you need to understand that a reference using the syntax
"Form_<FormName>" is a direct reference to an instance of the class module
for the form.  It can't even be used with a form that has no VBA code behind
it -- it will raise an error.  Furthermore, if you do use this reference
syntax, with a form that does have a module, it exhibits some interesting
behavior:
 
1. If there is not currently an instance of the form loaded, the form will
be opened.  I believe it will be opened hidden, so it you have to unhide it
if you want to see the form.
 
2. If there is currently one and only instance of the form loaded, whether
as a main form or as a subform, then the reference will return that
instance.
 
3. If there are multiple instances of the form loaded, the reference will
apply to only one of them, and *you have no control over which one that is*.
There is probably some internal logic followed, but I have no idea what it
is.
 
Now, I don't know exactly how this applies to datasheet subforms in Access
2010, but I strongly suspect that this is the root of the inconsistent
behavior you are seeing.  I strongly suggest you do not use the
"Form_<FormName>" reference syntax, because it doesn't reliably give you
what you want, and has unwanted side effects.
