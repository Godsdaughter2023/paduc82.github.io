Partially masked data

I am using Excel 2007. I want to hide the first few digits of a social security number. example: xxx-xx-1234. rick rothstein, mvp - excel say:

Okay, I was thinking... as long as we are using VBA code to handle the formatting of the cell, maybe you would like this simplify the method you use to enter your Social Security Numbers. I was thinking that instead of having to type 3 digits, a dash, 2 digits, dash and then 4 more digits, perhaps you would find it easier to just type in the 9 digits and let the code insert the dashes for you. The following code, in addition to providing your xxx-xx-#### format, will accept the Social Security Number properly formatted with the dashes at the 4th and 7th character positions, as before, and, in addition, will also accept up to 9 digits (leading zeroes will momentarily disappear if typed, so you do not need to type them... hence the "up to")... if you type anything else, it will appear in the cell as typed. The part highlighted in bold is the additional code that handles the non-dashed digit input.

Private Sub Worksheet_Change(ByVal Target As Range)
  Dim Cell As Range
  Const ColumnToMonitor As String = "B"
  If Not Intersect(Target, Columns(ColumnToMonitor)) Is Nothing Then
    For Each Cell In Intersect(Target, Columns(ColumnToMonitor))
      If Len(Cell.Value) > 0 And Not Cell.Value Like "*[!0-9]*" Then
        Application.EnableEvents = False
        Cell.Value = Format(Cell.Value, "000-00-0000")
        Application.EnableEvents = True
      End If
      If Cell.Value Like "###-##-####" Then
        Cell.NumberFormat = ";;;xxx-xx-\" & Format(Right(Cell.Value, 4), "0\\0\\0\\0")
      Else
        Cell.NumberFormat = "General"
      End If
    Next
  End If
End Sub

adam vero say:

Sorry, re-reading now I realise my answer was not clear.

What I was trying to say is that Patrik was already using a standard approach, and this could be extended to be used here.

I would still err on the side of using local dummy lists in many cases as it is clearer, easier to troubleshoot and so on.

So, the trick to break out of the restriction to get to other worksheets is that the data validation must use a defined name, say "MyList".This name can point directly to an explicit range in the other worksheet of the same workbook. We all accept that this should work, and all we are doing is fooling the error-checking code to allow us to do it. But it does not work if you use a name for your list and that name points to an external cell range, the error-checking is too smart for that.

But we can be smarter...

Taking this a step further, first you define a name for the correct range in the workbook which has the list of allowed values (I'll call this the source workbook). Now in the workbook where you want to use the validation you create another name, which refers to the first named range in the source workbook by name .

eg in source workbook Source.xls, "MyValues" refers to some cell range as usual. In the workbook which needs some validation you create a name such as "MyList" which points to [Source.xls]!MyValues (note if MyValues is local to one worksheet you need to include the worksheet name as well).

Now you set up validation using MyList and everything works. Hurray!

However, you will notice that if you close source.xls the defined name shows the full path to that file (this is good and means it knows which of your files called source.xls should be used). Normally, formulas would not be fazed by this, they just go read the file when they need to (albeit slowly). But having source.xls closed will break this name-to-name trick for validation, you MUST have the file open or you get nothing in your list, which might be a reason to revert tousing dummy lists.

One reason I like the approach of pointing a name to another name defined in the source workbook, is that whoever owns that file is responsible to make sure this range name points to the right region (eg updating it if items are added to the list). You are responsible for making sure that your local name points at the correct range name in the other file, and everything joins up nicely.

I've posted a pair of files on my website if you want to save time on testing this method. Download validate.zip from here:tinyurl.com/6zwjy4o and unzip to c:\temp\ this wil give you validate.xls and source.xls which I just created in Excel 2003 (sorry for the obviously thrown-together appearance!).
