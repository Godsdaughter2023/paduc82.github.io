Actually, it's a more simple explanation than you might expect :-) It stems from the fact that Word 2004 doesn't know
Word 2011, OTOH, is a bit more XML-savvy... but not much. If the document was created by because those formats include what you could call a "built-in XSLT" that typical XML files do not have. IOW, the internal coding of the file includes the supporting schema.

a. user say:

Unfortunately, Word 2011 just gets confused. There's absolutely no reason why it should do so, because it should be quite obvious to Word that an XML document that does not specify any of the namespaces that its documents use cannot be a Word document. Further, its .docx/.dotx/.docm/.dotm format files are all ZIP files with a very specific internal structure that no .xml file could possibly have. If your XML document actually specified those namespaces and used them, either the document would conform to WOrd's schema and Word could open it, or it wouldn't and Word would correctly not attempt to open it.

I generally ensure that Word->Preferences->General->"Confirm conversion at open is checked, then specify Other Encoding->Unicode 6.0 UTF-8. If the XML file has a BOM at the beginning you'll see three rubbish characters that you can delete, but XML files that are correctly UTF-8 encoded should otherwise be OK.

Bob's comments about "Internal XSLT" and schemas are wide of the mark. AFAIK Word XML documents contain no XSLT unless someone else put it there, and the only bits of schema you're likely to find in a Document created by Microsoft products are nothing to do with Word's own schema, but are descriptions of the properties that SharePoint may insert when it serves a document to a user.

tom_bpc say:

This works essentially the same (as Aljilogy suggests) in MSP 2010.

I've read this whole thread since I missed it on the first go-round, and I think it's still marked as unanswered.

The OP's issue is addressed (as Julie suggested) by making a custom Number field:

1. If the ONLY intention is to count ALL tasks.  It's easy to do the following:

    - (Optional) Rename the field, "Task Count" for example;

    - Set "Calculation for task and group summary rows" to "Rollup: Count Nonsummaries"

    - There is no formula needed here.  If there is a formula, then it will be ignored by the summary tasks anyway.

    - Obviously, make sure you have a visible project summary task to roll up everything.

2. If the intention is to count all task meeting a certain set of criteria, then two things need to change:

    - Enter a formula with a binary (1 or 0) result for each task.  The formula for the OP's specific issue (count all non-summary tasks with status of "Late") is:   IIf([Summary]=False And [Status]=2,1,0).  This formula computes a value of 1 when the task is non-summary and late, and 0 for all other cases.  (Like a lot of MSP fields, "Late" status actually indicates a number code = 2, which I had to lookup online.) 

    - Of course the user could also manually tag the tasks to count by entering the number 1 instead of a formula.

    - To count tasks meeting the criteria, the "Calculation for task and group summary rows" needs to be Rollup: Sum (not Count Nonsummaries).
