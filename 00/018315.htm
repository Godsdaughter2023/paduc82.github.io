jacobsladder---01 say:

4G is a new technology again, so you would in theory have a phone that supports GSM, 3G and 4G at the same time, plus having to support those technologies in different bands.

 

But yes, if you have a 4 band GSM phone, purchased in Asia/EU or America, then that will work in just about every country in the world. If you want to travel to Japan then you need 4 Band GSM + 2100 3G.

 

I think most N series phones today have 4 band GSM so will work in USA, but won't have the best 3G band unless you have bought the US version. For best US coverage you need 3G in 1900 & 850 bands, for Europe 900 & 2100.

 

You then get other mixes, eg Japan uses 2100 and 800 (not quite the same as US 850), Brazil  (I think) uses 850 and 2100, and I think so does Israel and Australia.

 

The holy grail is something called SW defined radio, where you can reconfigure the radio access in SW no matter what band you are accessing, however this is not easy to achieve and so most work has focused on optimising exisiting architectures for lower power and smaller size so that adding new bands doesn't make a bigger phone.

john w. vinson say:

Yes. Excel is a spreadsheet, best of breed; Access is a relational database. This is a good spreadsheet design but it is NOT a proper database structure!

Better would be to step back and design this relationally. You say each column is a Drawing; but what does each Row represent? Does one Project (??) have many Drawings? If so can one drawing appear in more than one Project?

If so you have a bog-standard many to many relationship which would use three tables: a table of Projects (or whatever your spreadsheet row entity represents); a table of Drawings, with one row per tag; and a third table to model the many to many relationship - it could have just two columns, a ProjectID and a Tag field. If a given project has twenty drawings it would have twenty rows - not columns! - for that project. You could put a unique Index on the two fields to prevent entry of duplicates, or (with a little easy VBA) highlight duplicates.

Hey John, thanks for your reply!

It is a set of facility drawings, so the drawings won't change often (occasional additions). Since I will always be using all drawings, do I just need two tables? One with the tags, one for the many to many relationship? The tag list will be somewhere in the vicinity of 10,000 tags once complete. The drawings are piping P&ID's, the tags are for valves. So each drawing has many tags on it. So do something like this?

john w. vinson say:

You've got it! The way to think about database design is to consider the "Entities" - real-life persons, things, or events - relevant to your needs, and the "Attributes" of those entities: the discrete "chunks" of information that you need to know about the entity. Each kind of Entity gets its own table - in your case, at least a table of Valves and a table of Drawings. 

Each table must have its own Primary Key - a field which should meet three requirements. It must be unique; it must be stable, not changing (or at least changing very rarely); and ideally it should be short so Access can use it efficiently. Many or most databases use an Autonumber primary key, which meets all three of these automatically, but in your case it appears that you already have reasonably good candidate keys in your DWG and TAG fields.

I would, however, recommend that you not include the constant text DWG# and TAG# characters in the actual field. If all drawing numbers start with DWG#, it's just waste space to include it in the table; just use Long Integer values for the numeric part and put the boilerplate text in a Label on your forms or reports, or display it using a Format. For example, if you set the Format property of a table or query field, or of a Textbox, to

"DWG\#000"

it will actually store a number in the table but display 213 as

DWG#213

Your example table is just what you want; it will be easy to create Reports listing all the Valves for a drawing, and to either prevent or easily find duplicates.

Hi Michelle and Paul,

Nice question! Actually, Windows phones have no "Reset" button to reset the data meter. The thing you can do to reset it, is to change the Time and Date settings of the phone. After changing it, go to Data Sense or Data usage app and open it and wait to load. Then close the app and go back to Time and Date settings then set it to the right date.
