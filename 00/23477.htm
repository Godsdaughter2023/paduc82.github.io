john - project say:

MS Project Geek,

You're right, you do have a rather complex plan. For reference, the leveling algorithm does pretty good but it is not designed to address all possible overallocation issues. It just isn't that sophisticated.

One thing that is slowing down the process is the "day by day" setting. This trips up a lot of users and the best "fix" is to change the setting to "week by week" or even "month by month". If you really have a hard rule for no overallocations on a daily basis, I'm afraid you're going to either have to do the process manually or develop your own algorithm with some VBA (not impossible, just challenging).

Here's something you might try. Instead of trying to level the whole plan, (which it sounds like you are trying to do), isolate tasks that are in the near term (e.g. the current and next week). Use Project's leveling for those and see how it does. It should make it easier to manually adjust overallocations the algorithm can't resolve.

Hope this helps.


John

a. user say:

Hi MS Project Geek,

Resource Leveling has some challenges in general.

* Requires a lot of detailed and accurate data, which is often not available

* When there are high levels of uncertainties (always in projects)

* Doesn't work genuine when task assignments are made for less than 100% (how to level 2 tasks assigned to the same person for 51% each?)

* Leveling multiple resources creates overcapacity, because resources are waiting to prevent over-allocation for each other. While all will have the feeling they could have some more work.

* Try to level 50+ projects with these challenges. Challenge!

There are work arrounds, but it still it makes it too complicated. PM is too difficult for the average pm without resource leveling. And in the end it doesn't really work.

What intrigues me: 30k+ lines.

I know nothing about your project and your plan. So don't take my speculation personal.

And I know in some industries, this kind of project are created (assembling airplanes)

But it could be that some planning guidelines are violated.
Project Plans should not try to solve complexity that that can be solved at task level. 
And Project Plans should not try to solve complexity that must resolved by clients or the project organization.

If you can delegate it, than don't put the details in your gannt chart.

If the course of the project isn't clear up front, then you better split up the project in smaller projects and deal with it as a multi-project. (When it deals with multiple points of delivery, GO/NOGO decisions you don't control, etc.)

This may simplify your plan, but don't necessarily fix your leveling issue.

I have some recommendations, but like to get some better understanding of your schedule.

How many more projects like this are those 50+ resources involved?

The ability to create Remote Mailboxes via the Exchange EAC in a hybrid exchange environment is to make sure the user trying to make the change has at least Exchange Organization Management membership on the on-premise environment.

Sub NewRowAK()
  Dim Dest As Range, R As Range
  
  'Insert a row below the current row
  ActiveCell.EntireRow.Insert xlDown, xlFormatFromLeftOrAbove
  'Refer to the new row columns A:I
  Set Dest = Intersect(Range("A:I"), ActiveCell.EntireRow)
  'Add a thin border around each cell
  For Each R In Dest.Cells
    R.BorderAround xlContinuous
  Next
  'Copy the values from A and D from the row above
  For Each R In Intersect(Dest, Range("A1,D1").EntireColumn)
    R = R.Offset(-1).Value
  Next
End Sub
