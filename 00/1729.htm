ken sheridan say:
What are some other ways other than inserting a lookup field for matching the auto number IDs from a parent table in a child table with mega amount of imported records?

You'd need to firstly insert the rows from the referenced table into the Access table which includes an autonumber column as its primary key.  Then join this table to the imported referencing table on the 'natural' candidate key, and by means of an append query, from this query insert the columns from the referencing table excluding the 'natural' key column, but including the autonumber column from the Access table, into an Access table which includes a long integer foreign key column into which the autonumber column's values would be inserted.

But as Scott has said, why bother?  Just use whatever columns constitute the candidate keys in the BI system as the keys in the Access tables by designating that in the referenced Access table as the primary key and creating an enforced relationship between it and the referencing Access table on the columns.  If it won't allow the relationship to be created, then that will indicate bad data in the original source.

ken sheridan say: One should not be dogmatic about the use of 'surrogate' or 'natural' keys, but decide on which to use in the context of the application.  While the former is in most cases the better choice, in some cases a 'natural' key such as your project team column has advantages.   An example of where this is the case is where it is necessary to restrict the items listed in a bound combo box in a form to a subset of the total rows in the referenced table, either because the combo box is correlated with another combo box from which a selection is made first to restrict the items available in the second combo box, or if the list is restricted on the basis of the status of each item which can change over time, e.g. a product might be discontinued.  In both cases, if the value of the combo box is a hidden surrogate key, the value will disappear in rows in a continuous form where it is not included in the rows available in the combo box in the currently selected row.  If the value of the combo box is the bound column, however, it will be visible regardless of whether it is included in the list or not.

With some entity types a surrogate key is necessary, however, as the 'natural' values can legitimately be duplicated, e.g. personal names or some geographical entity types such as cities.  In such cases a distinct numeric key should be used.

The 'lookup field wizard' in Access is really a means of achieving in a table's datasheet what one can easily achieve in a bound form by means of a combo box, but without the deleterious effects which the wizard produces.  As all data entry should be via forms of course, the use of this feature in the table design serves no useful purpose in any case.   In a form bound to a table which includes an EmployeeID foreign key column for instance, referencing the surrogate primary key of an Employees table the combo box would be set up as follows:

ControlSource:   EmployeeID

RowSource:     SELECT EmployeeID, FirstName & " " & LastName FROM Employees ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.

windows defender dosnot start and error 0*8007139f come up - Microsoft Community

zarabehpouri say:

windows defender dosnot start and error 0*8007139f come up

how can I fix this!
greginmich say:

If you have a third-party AV app installed on your PC and you get this error persistently when you try to start the Windows Defender Service, then that would likely mean that Windows Defender didn’t get properly disabled with the installation of the third-party antivirus provider. Defender needs to be completely disabled when you’re using a third-party AV app in order to prevent conflicts at both the kernel-level and user-level – and when this gets done properly, you’ll just see an informational message stating that “This app has been turned off and isn’t monitoring your computer” when you try to start Windows Defender:

But if Windows Defender didn’t get disabled properly in its own registry key, then its User Interface will appear with a message that “Your PC isn’t being monitored because the app’s service stopped. You should restart it now”. And then when you click on the Start now button, you’ll be informed that the service couldn’t be started because “The group or resource is not in the correct state to perform the requested operation” (Error 0x8007139f).

The appearance of the Windows Defender UI here gives people the impression that they should be able to continue using Defender right alongside their third-party AV app, and that there must be something that they can do to get its service started. That impression is totally false, and what people really need to do in this scenario is cease and desist in their efforts to start the Windows Defender Service.

The problems associated with the underlying error state frequently include failed Windows Defender definition updates, and Error 577 on trying to start the Windows Defender Service in services.msc. But none of these symptoms appear to have any serious consequences – so you should be OK if you just ignore the whole issue. On the other hand, if you’d prefer to take some remedial action, then you can refer to my post here:

http://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_updating/windows-10-desktop-pc-kb2267602-definition/d5b551bc-bf40-46d8-a05e-924d33882d6d

[Edit for update, 6/9/2016]

I think we’ve just found a quick and easy fix for this issue, although the prospective solution might well have to be modified slightly depending on the specific third-party AV app that you’re using. So people who get the error screen shown above when they try to start Windows Defender, and who maybe also have the related issue of failing Windows Defender definition updates (with a third-party AV app installed) might want to have a quick look here:

http://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_updating/definition-update-for-windows-defender-kb2267602/55510bcc-61b9-49cf-ba1a-e4e2bd083975?page=5#LastReply

The groundwork for this issue can be found here:

http://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_start/windows-defender-error-code-0x8007139f/93096739-16ce-471b-8716-1776f0c69f89?page=2

 [End Edit]
