bob jones aka: cybertaz say:
Please don't waste your time trying to figure out Phillip's suggestion... If he'd tried it himself he would have found that it cannot be done. His suggestion lacks detail for the same reason.

The ellipsis is a defined single character just like any other. The spacing between the 3 dots that comprise it is determined by the font designer based on standard guidelines. That spacing cannot be adjusted by the user any more than the height of a descender or ascender, or the length of the crossing of a "T"... or how much space is inserted when you press the Space Bar.

When you type 3 consecutive periods, AutoCorrect converts them to an ellipsis character compliant with the font & font formatting in use at the time. It's the same character you'd enter by typing the keyboard shortcut Option+; rather than the 3 dots. You can eliminate the item from AutoCorrect, but if you do you'll simply wind up with the 3 dots as separate characters, which could become a serious problem for several reasons, such as separating line breaks.

If you consider adding the ellipsis character from a differently designed font (such as Symbol) as an AutoCorrect item it won't solve the problem either. It would have to be saved as Formatted, which means that the same size, font would be inserted each time & could differ significantly from what was in use in the document.

Depending on your actual definition of "scrunched together" you may have a font issue that needs to be corrected. I'd suggest that you follow the procedure offered here for clearing up conflicts:  Font Weeding. There may be some differences in terminology depending on your version of OS X, but the Help for your version of Font Book should make it easy to interpret.

Run Disk Utility to repair disk permissions then Restart the Mac afterward.

If users can change the size of the form, then you'll have to calculate the number of records that can fit at the moment.  If the form has its Header and Footer sections, even if they might be invisible or zero Height, use:

   With Me.RecordsetClone  
      If .RecordCount > 0 Then
         NumRecs = (Me.InsideHeight - Me.Section(1).Height - Me.Section(2).Height) \ Me.Section(0).Height
         .MoveLast
         If .RecordCount < NumRecs Then
            .MoveFirst
         Else
            .Move 1 - NumRecs
         End If
         Me.Bookmark = .Bookmark
      End If
   End With
   DoCmd.GoToRecord, , acNewRecord

If the form does not have it's Header and Footer, then use:

   NumRecs = Me.InsideHeight \ Me.Section(0).Height

Opening Two Instances of the Same Form

Is it possible to open two instances of the same form with different records?  I have a form to enter contact records that checks to make sure the same person isn't entered twice for the same company.  Right now, it just opens up a message box that the record they are attempting to save is a duplicate.  I'd like to open the duplicate record for the user to see and/or modify without losing the data they have just entered in case it's not really a duplicate.  Is that possible, or will I need to create a copy of the form to use in this instance?

al borges say:

You can't have 2 free instances of the same form open at the same time, although you can embed 2 subforms based on the same form into a primary form. Just name each instance differently.

I've done this in one of my database projects where patients have a primary and a secondary insurance coverage. I named the first instance "frmINS1" and the second one "frmINS2." In this layout, both instances have the same recordsource, but you can try assigning a recordsource using the on the subform's OnCurrent event.

tom van stiphout say:

> You can't have 2 free instances of the same form open at the same time

Oh yes you can. A form is an object, and you can create a collection of form objects, and they can be the same form. As with any object, you use the New keyword to create them (not DoCmd.OpenForm).

See also: http://allenbrowne.com/ser-35.html
