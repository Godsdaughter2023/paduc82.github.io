rich007a say:

You can't use a UDF to change another cell's value, so you could never do this with a UDF.  See here: http://support.microsoft.com/kb/170787

Try this sub instead:

Sub MultipleGoalSeeks()
    Dim rngChg As Range, rngGoal As Range, rngSet As Range
    Dim i As Long
   
    Set rngChg = Range("A1:A10")    'Range to be changed
    Set rngGoal = Range("B1:B10")   'Range containing Goal values
    Set rngSet = Range("C1:C10")    'Range to be set to Goal value
   
    For i = 1 To rngSet.Rows.Count
        rngSet(i).GoalSeek Goal:=rngGoal(i), ChangingCell:=rngChg(i)
    Next i
End Sub


Note I have not included any error handling, and you'll need to do something to check that the goal seek has been a success.

To test, put the following in cells A1:A10:  1, 2, 3, 4....10
Put the following in cells B1:B10:     5
Put the following in cell C1:   =A1*A1    and drag it down to C10

Run the sub, each cell in A1:A10 is changed to close to the suare root of 5.

Now set cell C1 to   =-A1*A1  (note the minus sign) and run the sub again.  A solution is not possible in this case (the suare of a number can never equal a positive number), but the code doesn't highlight this.  Therefore, I would consider using some conditional formatting to highlight cells that are not within a certain margin of the target.

Probably a simple thing I'm missing, but I've two groups of resources, one group (the majority) works 5 days/week, another group works 7 days per week.  How do I manipulate/assign "change working time" so that a duration of, say, one week  represents 5 days versus 7 days work?

john - project say:

Ron,

Project only allows one definition for a "day", "week", and "month". If the definition of a "day" is different for the 5 day/week group versus the 7 day/week group, (e.g. 5 day/week people work a standard 8 hour day but 7 day/week people work a 7 hour day), then the only time you will get valid information in the Duration field is to set it to display hours because you won't be able to satisfy both work groups with a "day" or "week" value in the Duration field. That's just the way Project works.

With regard to your issue of setting this up, I'd do the following. Since at least one group works a 7 day week, create a custom calendar that has 7 working days with the appropriate work shifts. Set the Project calendar to the 7 day/week calendar (Project/Properties group/Project Information).

Next, create your two (or more) resource individuals or groups using the Resource Sheet view. You will note that the Base Calendar for each resource shows the 7 day/week. For those resource that work a normal 5 day week, change their Base Calendar to the Standard.

Now when you assign resources to tasks you will see that the 5 day/week resources only work 5 days but the 7 day/week resources work all seven days.

Hope this helps.

John

ronmilberger say:

Thanks, John.

Is there any reason I the base calendar can't be 5d/w and the 7d/w calendar be the exception?  

john - project say:

Ron,

No reason, and that might be a better choice since the majority of your resources are on the 5 day/week schedule.

But keep in mind that the Project calendar sets the calendar for all task scheduling, so, for those tasks that will only be assigned to 7 day/week resources, be sure to set the Task Calendar to the 7 day calendar. And, if any tasks have a mix of 5 day/week and 7 day/week resources, the calendar for those tasks should also be set to the 7 day calendar.

ken sheridan say: Firstly you need to insert rows into the two referenced tables, Table 1 and Table 2.  Values will be assigned automatically to the autonumber primary key columns of these tables.  You then need to assign rows into the table, Table 3, which models the relationship between the first two.  The two columns which comprise this table's primary key, Doc id and Trans id, must not be autonumbers, but of straightforward long integer number data type.  The values for these columns are obtained from the values of the primary keys of the two referenced tables.

So the first 'append' query is:

INSERT INTO [Table 1] ([Doc no],[Title],[Originator])
SELECT DISTINCT [Doc no],[Title],[Originator]
FROM [Base table];

The second is:

INSERT INTO [Table 2] ([Trans no],[Date])
SELECT DISTINCT [Trans no],[Date]
FROM [Base table];

The third query inserts rows into Table 3 by joining the two referenced tables to the Base table on the candidate keys Doc no and Trans No:

INSERT INTO [Table 3] ([Doc id], [Trans id])
SELECT [Table 1].[Doc id], [Table 2].[Trans id]
FROM ([Table 1] INNER JOIN [Base table]
ON [Table 1].[Doc no]=[Base table].[Doc no])
INNER JOIN [Table 2] ON [Base table].[Trans no]=[Table 2].[Trans no];

Note that, having used surrogate keys, you do not need Doc no or Trans No columns in Table 3 as this would introduce redundancy, the values of these attributes being obtainable via the relationships with the two referenced tables.  

It is imperative that the two columns Doc id and Trans id be defined as the composite primary key of Table 3 to prevent invalid duplication of rows.
