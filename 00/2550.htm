karl timmermans [outlook mvp 2012-2018] say:

I had someone attempt to change my desktop from Office  to 365 (so all 3 of my computers were the same and we could use One drive to all work on Outlook). 

#1 - and most important ignoring everything else, any Outlook files usd in any active Outlook session should not be stored on OneDrive. Not supported and can cause file corruption regardless of Outlook version in use. Outlook data files should always be stored on your local hard drive or more specifically, not in a folder that is getting sync'd with any kind of cloud service. 

Also just to be clear, the above is not the same thing as keeping a "copy" of any Outlook data file on cloud file sharing service. Difference being that the "backup" version is not being accessed by Outlook.

#2 - As for the rest, comes down to the types of email accounts that have been configured (POP, Imap, Exchange etc) on all three machines and what exactly it is that you are ultimately trying to accomplish in terms of sharing data across multiple devices

#3 - Don't have any experience with Carbonite but what needs to get restored (if anything) would depend on the type of email accounts configured on the target machine. As a for instance, backup data files for an IMAP account created in OL'2010 or earlier would not be the same for an IMAP account created in OL'2013/'2016 and in fact would be useless for OL'2013/'2016 IMAP accounts.

#4 - Just for the sake of accuracy - your reference to "Office" and "365" are one and the same thing. "Office 365" is just the name for the subscription service for the current released version of "Office" - currently being Office 2016 so one big piece of missing info is "What version of "office" are you referring to in " change my desktop from Office  to 365"? 

 

In a nutshell - OneDrive is not part of the equation in getting three devices to share information and to be perfectly honest - can be easily accomplished with any version of Outlook '2010 onward (or '2007 if multiple Exchange type accounts are not involved).

This problem is caused by an HP printer option called Mopier Mode being enabled. This mode causes only 1 copy to be sent to the printer which should then use its internal memory to print the multiple copies. You should then get multiple copies when you request them.

doyle nelsen say:

rich007a say:

Before you insert a new line, first try copying the line you want to duplicate.  e.g.:

select row 5. 

Ctrl+C to copy.

select row 6

right-click > Insert Copied Cells

Your formula and formatting should be duplicated.

 

If you want a "one button-click" solution you'll need a macro which could either insert a blank line and insert the formula(s) where required, OR it could copy a line and clear any cells that need to have fresh data inputted.  Below is one that does the latter.

You could use something like this:

Sub NewRow()
' Copies row 2, inserts new row under last filled cell in col A
' pastes and deletes contents of columns B and D in new row.
'
    Dim curRow As Long

    ActiveSheet.Protect

    Rows("2:2").Copy
    Range("A2").End(xlDown).Offset(1, 0).Select
    ActiveCell.EntireRow.Insert shift:=xlDown
    ActiveSheet.Paste
    Application.CutCopyMode = False
    curRow = ActiveCell.Row
    Range("B" & curRow).ClearContents   'Cells to delete
    Range("D" & curRow).ClearContents

    ActiveSheet.Protect
End Sub

To use the macro, hit Alt+F11, Insert > Module   then paste the above text in the pane that appears on the right.

You could insert a button control into your spreadsheet for your staff to use.  First, show the developer tab (show it via round Office button > Excel Options > Popular > Show developer tab in the ribbon).

Click Insert and click the top left control "Button (Form Control)" and then click on the sheet where you want the button to appear.  The Assign Macro window will appear, select the NewRow macro and OK.

You can resize the button and change the text on it.

You must save your workbook as a Macro Enabled Workbook (*.xlsm).  Staff will have to enable macros when opening the file, depending on your security settings.

You must unlock all columns that will need to be editable.  e.g. the above code clears columns B and D from the new row, so select columns B and D (hold down Ctrl while clicking on the column headings) and right-click>Format cells>Protection>uncheck Locked.

Hope that helps.

Columns in an Excel worksheet are ALWAYS named A, B, C, …

It doesn't matter if you insert or delete columns. Afterwards, the columns will still be A, B, C, …

The column letters are just a way to refer to columns; they don't have an intrinsic meaning.

Think of what happens when you insert a new column between columns B and C. If Excel didn't rename the columns A, B, C, …, how could we refer to the new column?
