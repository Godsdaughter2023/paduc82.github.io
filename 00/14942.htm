Access 2007 - Users (not DB creator) get the message "There isn't enough memory to update the display. Close unneeded programs and try again."

My users keep getting the message "There isn't enough memory to update the display. Close unneeded programs and try again." on an Access 2007 database (accdb).

I never get the error message (though I may have different permissions but not admin rights).

The database is split with a SQL Server BE and we're running Windows Server 2008. Each user has their own copy of the FE.

The error message doesn't keep anyone from using the database. However, the message popping up all the time is frustrating/time consuming.

I've already tried compact and repair, decompiling/compiling, exporting to text and importing to a new database. The message seems to pop up with any form/report. I don't know what else to try besides recreating the entire database from scratch (or copying all of my permissions to everyone else). Any suggestions would be very much appreciated!

sbkabes say:

I found the answer to my question; thought I'd post just in case anyone else runs into the same problem.

Even though we're all on the same terminal server, we have different fonts available to us. As an admin user, I had 297 and all of the other users have 246 (some are also similar to each other, but with different names). Even after my admin rights were removed (for testing purposes), I still had the 297 fonts available which is why I never go the error message.

After verifying all of the fonts match between our two logins, the message seems to have gone away. This could probably also be fixed on the Windows 2008/terminal server side, but it's not something I'm familar with or have access to.

joeu2004 say: Mohd.Ashran wrote:
I want to get the duration..
A- B =C
10.24.19
10: minutes
24: seconds

PLs Give me the formula..how to make..A deduct B and the answer in C

First, mm.ss.ff must be converted to Excel time.  It would be best to do that using helper cells.

And in order to do that, we need to know the frame rate.  From online information, I will assume it is 30 frames per second.  (But see "Caveat" below.)

We also need to know if seconds and frames are always 2 digits; for example, 10.02.01 for 10 min 2 sec 1 frame?  That would simplify formula immensely.

For example (note:  your table shows "B" in column B, "A" in column C, and "C" in column G; can you be more confusing?)

D3:   =left(B3,2)/1440+mid(B3,4,2)/86400+right(B3,2)/2592000
E3:   =left(C3,2)/1440+mid(C3,4,2)/86400+right(C3,2)/2592000
F3:   =E3-D3
G3:  =int(F3*1440) & "." & mod(int(F3*86400),60) & "." & mod(int(F3*2592000),30)

Note:  2592000 is 30*86400.

Caveat:  My assumption of 30 frames per second is inconsistent with your examples.  For example, C3 is 10.26.55.  I wonder if your data is really in the form m.ss.00 -- that is seconds and 1/100-th seconds.  Not only would that change things significantly; but also it would simplify things.

Please elaborate on the form of the input.

[EDIT] It is risky to use MOD with large values like F3*2592000 due to defects in MOD.  Arguably, it is more reliable to write INT(F3*2592000)-30*INT(F3*86400).

[ERRATA] I am not sure that even the corrected formula in G3 is reliable considering binary floating-point anomalies.  I will spend more time on it after your clarify the form of the input.
