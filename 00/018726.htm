albert d. kallal say:

You don’t mention what version of Access you are converting from, but VBA is un-changed when running Office x64.

The only exception is code that calls the windows API.

And your posted code makes no such calls – it should thus run fine without changes.

You also don’t mention if you running office x64 bit, or simply running the “recommended” x32 bit version of office on an x64 bit OS. So a typical and default install of office will be x32.

Regardless, EVEN if you are running Office x64 and thus an x64 bit version of VBA, there are in general ZERO code changes required to VBA. The only exception is if your VBA code is calling the windows API from Access.

None of your posted code makes calls to the windows API and thus no changes should be required.

I would assume you have attempted to compile the code BEFORE you run it (so in the VBA editor go

debug->compile).

If you VBA code does not compile, then likely you have a broken reference or some other issue – not one of running on an x64 bit OS.

So first, check what version of office you are running (99% of the time, it will STILL be an x32 bit version). You likely want to avoid the x64 bit version of office, especially if you have any ActiveX controls – they don’t exist for the x64 bit version. So say the tree view, or often the calendar (date picker) are two ActiveX controls and they do NOT exist for office x64.

So the only declare statements that MAY require changes are calls to the windows API and ONLY if you are running office x64.

If you are running office x32, then no changes to VBA should be required. The posted code should run just fine on any version of access released in the past 16 years.

albert d. kallal say:

As noted, the inspector not designed for use with Access code. And the access object model is so very different, that using the inspector is unlikely to give much of any value. (except large numbers of comments and suggestions that really don’t much apply to Access).

I suppose the inspector might help for code that uses the windows API – but how many of those exist in the application? (likely 1 or 2 tops – that is a few minutes of your time, and then after that you gain little if anything from the inspector).

I would in fact avoid the inspector – it going to suggest massive amounts of things that have little if any relevance to Access.

>The inspector references:  [mso]FileDialogFilter.Description

and provides this url:  http://go.microsoft.com/fwlink/?LinkID=215358.


See, the above is much useless information.

Better off to show the SIMPLE line of code that does not compile correctly. The file dialog object was supported in Access 2003 (first version, so “few” developers used that feature). However I recommend the filedialog since it works in both x64 and that of x32. (and I betting that the VBA code that does not compile is a result of the office object 15 library reference missing).

And I would be careful, since if you are going to modify the application to work with x64, then you may well break it in regards to working in x32. Remember, your existing version of Access 2003 is “happy” running without issues on the x64 machine! (so Access 2003 is x32, but runs fine on x64 machines).


The filedialog object can and will work in Access x64 2013 (and I in fact recommend using that feature in place of the windows API). However not seeing the offending 2-3 lines of code, it much like throwing darts in the dark!

I think you will obtain MUCH better results by simple using debug->compile in the VBA code editor and dealing with errors that occur as opposed to the code inspector.

You have to balance here how much you need/want to use the x64 bit version of Access (and it not the recommended version right now – especially for existing legacy applications).


So keep in mind the “reference” link above simply states that the message means some of the office object model has changed, but does not recommend or suggest the given code needs to be changed (and in fact it likely does not apply to Access in this case – and in fact I know that the filedialog object is supported in both 2003 and 2013 and for both x32 and x64).

I think a MUCH better approach to this migration project is the following:

You need to first determine these two things:

#1
Does the application use or reference ActiveX controls? (open up the 2003 application on a working machine with Access 2003 and ctrl-g to VBA editor – now go tools->references and look at the referenced software – are ActiveX objects in that list? (and working on a copy, if you un-tick one, can you still compile?)

The above is #1 step. The reason being is if the application uses ActiveX controls such as treeview or Calendar picker, then you are duck soup and could be faced with SIGNIFICANT requirements to re-write your software (more cost and time you have for such a migration).

And worse since you don’t have replacements for these controls in Access x64, then you might not be able to cost justify using Access x64. So keep in mind that using Access x32 (2013) will eliminate these required changes. (So Access 2013 x32 it is a far better choice then  x64, but for migration projects it often the ONLY PRACTICAL CHOICE from a cost point of view).

#2,
determine if the windows API is being used. Is there “one” routine that can be easily changed, or is there 200 windows API routines? If there is 200, then you don’t have the skill and budgets and time for such a conversion (but then using Access x32 would eliminate any need for change of code in this regards).

If only 1-2 API’s are used, then for sure one would justify fixing the API code since that only 1 hour of your time.

So I would take a step back, and determine the above two steps.

The file dialog object existed in Access 2003, and it exists in Access 2013 (for both versions).

Same goes for ActiveX controls. If there is 50+ forms and they all use the ActiveX calendar picker, then it going to be quite a bit of work to fix all those forms. And keep in mind for those ActiveX controls the code inspector will not help nor suggest the fixes for these cases anyway!

I hope the above gives the “approach” and "thinking" that experienced developers take for migration of such projects.

jeeped say:

The asterisk (e.g. *) is meant for the system time/date format, not to adjust the value according to the user's time zone. If one user has his/her computer set to dd/mm/yy and another to mm/dd/yy each would see the underlying values displayed correctly for their individual system time/date format.

If you want to shunt time values according to local time zone offsets, you would have to start with either UTC (aka Zulu) time or some standardized time zone offset; perhaps the main office's time zone differential. A drop down would apply a time (sometimes date) offset for the current user. Alternately, VBA could be applied to read the current computer's system time zone and apply an offset factor on opening the worksheet. A formula in another column would apply the offset to a column of UTC values to show the local date/time.

andreas killer say:

Correction:

After that line you have to use GetObject to get the correct instance of word with that object and some more code for the word object to replace the text.

After the application is started you can retrieve the application object from the OLEobject.

Andreas.

Sub Test()
  Dim S As Shape
  Dim objWord As Object 'Word.Application
  'Access the shape with the embedded word object
  Set S = ActiveSheet.Shapes(1)
  'Is it a word object?
  If S.Type <> msoEmbeddedOLEObject Then Exit Sub
  If InStr(1, S.OLEFormat.progID, "Word.Document", vbTextCompare) <> 1 Then Exit Sub
  'Open it with the associated application
  S.OLEFormat.Object.Verb xlVerbOpen
  'Get the pointer to the application
  Set objWord = S.OLEFormat.Object.Object.Parent
  'Do the work
  objWord.Selection.Find.Execute FindText:="qwe", ReplaceWith:="asd"
  'Close it (Note: There is no need to save it here, because it's saved within this file)
  objWord.Quit
End Sub
