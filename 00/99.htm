Agency, Title, Phone Number) so that we can export those lists with the additional fields.  Is there a way to do that?

 

We have an enormous number of contacts.    All of them are in at least one distribution list.  Some are in multiple distribution lists.    Although we can export all of our contacts, we have no way of knowing who belongs in which distribution lists.  For our COOP, it is important to us to know ALL of the contact information for each person assigned to a particular distribution list.

karl timmermans [outlook mvp 2012-2018] say:
We'd like it to show additional fields (ie. . Agency, Title, Phone Number) so that we can export those lists with the additional fields.  Is there a way to do that?

 

Adding fields to a PDL is not possible. There's also no PDL export function in Outlook and not aware of anything 3rd party that would do what you want (either free or commercial). A DL can contain different kinds of members including "one-off" with no details (beyond name and email address), active contacts from any contact folder in the Outlook Address Book or member of the Exch GAL. Those are 3 different item types in terms of available fields to be exported.

You seriously want to consider the use of "categories" as an alternative using mutliple categories for those contacts that belong to multiple lists.

As a useless bit of trivia - we gave up on doing anything "extended" with DL's (as opposed to EX Dist Groups) a very, very long time ago because they had the potential of becoming an "unholy mess" especially once "embedded" lists got included in the mix going by a number of customer supplied DLs that were reviewed.

jeeped say:
How can I set up an equation that will give me the number of cells in a single column that have data in it just by telling it what column to look under?

If you want to know how many cells contain numbers in column A then,
=COUNT(A:A)
If you want to know how many cells contain numbers, text or formulas (which might be returning an empty string) in column A then,
=COUNTA(A:A)
If you want to know how many cells contain numbers that are less than 10 in column A then,
=COUNTIF(A:A, "<10")
If you want to know how many cells contain numbers that are less than 10 and greater than 0 in column A then,
=COUNTIFS(A:A, ">0", A:A, "<10")

ken sheridan say: Let's start from square one. Using the table names you gave in your original post the tables should be like this:

Empttable
....Emp# (PK)
....FirstName
....LastName
....etc

Depttable
....Dept# (PK)
....Department

DeptDatatable
....Dept# (FK)
....Emp# (FK)

Each table can have other columns as necessary to represent other attributes of the entity type which the table is modelling.

The parent form should be in single form view and be based on the Empttable table, or a query on that table which orders the rows appropriately, e.g. by LastName, FirstName.

The subform should be based on the DeptDatatable table, be in continuous forms view and should be linked to the parent form by setting the LinkMasterFields and LinkChildFields properties of the subform control to [Emp#].  The subform should include a combo box set up as follows:

ControlSource:    [Dept#]

RowSource:     SELECT [Dept#], [Department] FROM [Depttable] ORDER BY [Department];

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column. In the SQL statement for the RowSource property above Department is the name of a text column in the Depttable table which has the department names as its values.  There is no need to show the Emp# or Dept# values in the subform; they do their work automatically behind the scenes.

To allow users to enter a new department for an employee, but not edit or delete existing rows in the subform, first set the AllowDeletions property of the subform's source form, and its AllowEdits property each to False (No).  Bear in mind, however, that you will almost certainly need to delete rows from the Depttabletable at some time, so will need a means to do so.  Assuming that the database is split into front and back ends you could have a front end for certain authorised users in which the subform's AllowDeletions property is True while the hoi polloi have a front end in which it is False.
