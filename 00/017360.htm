jllatham say:

Here's a possible solution that Andy Pope came up with over at OzGrid ( http://www.ozgrid.com/forum/showthread.php?t=50581 ).

No new column C needed!

This will work IF there is definitely a match up to be made, and with the caveat that if you specify row numbers to search, you must start at row 1:

=INDEX(Sheet2!C:C,SUMPRODUCT(--(Sheet2!A:A=A45),--(Sheet2!B:B=B45),ROW(Sheet2!B:B)),1)

that will work in Excel 2007/2010/later - not in 2003 or earlier, for 2003 and earlier you'd need to use something like:

=INDEX(Sheet2!C1:C1000,SUMPRODUCT(--(Sheet2!A1:A1000=A45),--(Sheet2!B1:B1000=B45),ROW(Sheet2!B1:B1000)),1)

If there is the possibility of entering a misspelled student name or month in A45/B45, then you can add some testing to it this way:

=IF(ISNA(AND(MATCH(A45,Sheet2!A:A,0)>0,MATCH(B45,Sheet2!B:B,0)>0)),"No Match Possible",INDEX(Sheet2!C:C,SUMPRODUCT(--(Sheet2!A:A=A45),--(Sheet2!B:B=B45),ROW(Sheet2!B:B)),1))

but that doesn't absolutely guarantee not to return a false return, but pretty close.

NOTE: Sheet2 is the sheet with the comments on it.  The formula(s) would be going into the report card sheet, cell C45.

Alternate part number schema - Microsoft Community

bkreft say:

Alternate part number schema

I am attempting to structure my inventory database to accommodate alternate part numbers. I have a master parts header record with an alternate part number table (one to many relationship). I want to be able to query a part number and it will list all part numbers that could be ordered (different vendors have different part numbers for the same part). The problem I am trying to overcome is the alternate number queried on it will only show the parent (or header record) and not show other numbers associated with the parent record.

 

I thought a many to many table would solve the problem but it only magnifies the complexity once you get beyond three part numbers as you must ensure that you have entered all possible combinations of all the affected numbers. (Alternate part numbers have header records in the master table)

 

Is it possible to use a one to many table and then query on that table to show all relationships?

 

Data entry must be pretty straight forward as most clerks will not be able to think of all the combinations for a series of part numbers.

ken sheridan say: A table to model the many-to-many relationship type between parts and Vendors would be the way to model this, so the tables would be something like this:

Parts
....PartID (PK)
....PrimaryPartNumber
....Part

Vendors
....VendorID  (PK)
....Vendor
....etc

and to model the many-to-many relationship type:

PartVendors
....PartID
....VendorID
....VendorPartNumber

The primary key of this table is a composite one of the PartID and VendorID columns.

PrimaryPartNumber would be a number used by you organisation internally to identify each part, VendorPartNumber would be that used by the vendor.

To query by PrimaryPartNumber is simply a matter of joining the tables, and referencing a parameter on the PrimaryPartNumber column.  The parameter would normally be a combo box in an unbound dialogue form, listing all parts.

To query by a vendor's part number to return all vendors who supply the same part, and the part numbers used by the vendors for that part would require two instances of the PartVendors table to be joined, e.g.

SELECT PV2.VendorPartNumber AS SelectedPart,
PV1.VendorID, PV1.VendorPartNumber
FROM PartVendors AS PV1 INNER JOIN PartVendors AS PV2
On PV1.PartID = PV2.PartID
WHERE PV2.VendorPartNumber = Forms!YourForm!cboVendorPartNumber;

where cboVendorPartNumber is a combo box of vendor part numbers.

The Parts and/or vendors tables can be included in the query, joined to the PV1 table to include the part and/or vendor names etc as columns in the query's result table.

ken sheridan say:

I took my cue from the statement in your original post that 'different vendors have different part numbers for the same part',  which in modelling terms suggests to me that PartNumber is an attribute of a many-to-many relationship type between Parts and Vendors, i.e. it is functionally determined solely by the whole of the key of PartVendors.  The same part number might or might not be used by different vendors, but this would not necessarily mean that the part is the same, e.g. Vendor 1 could use 123 as the part number for Nut and Vendor2 could use 123 as the part number for washer.  Vendor 3 might use 567 as the part number for Nut. Consequently Nut and washer must each be identified by a distinct key value, 42 for Nut and 66 for Washer say, and each would be represented by one row in parts, unlike your part header table where Nut is represented by multiple rows.

 

The PartVendors table would therefore reference this key of Parts and the key of Vendors:

 

PartID       VendorID          VendorPartNumber

42            1                      123

66            2                      123

42            2                      567

 

Your model does not seem to relate the part numbers to vendors.  That doesn't really have any bearing on the original question per se, however, which, if I understand it correctly, is in essence how does one relate the various part numbers used for the same part to each other.  So Vendors could be taken out of the picture and the table simply be:

 

PartID    PartNumber

42         123

66         123

42         567

 

Note, however, that whereas in the first table the same part supplied by multiple vendors using the same part number would be represented by multiple rows, in the second table it would be represented by one row only.

 

There is no need for an 'adjacency list' type table as you have as the alternate part numbers table.  Each row in the above table maps to another row via PartID, so a query of the type I described which joins two instances of the table can be used.  However, this query will return no information regarding vendors for the parts.
