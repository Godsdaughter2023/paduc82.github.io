dirk goldgar say:

The presence of the table indicates that Access encountered an error when compacting the database.  I don't think there's any way to tell from that particular error code and description exactly what the error was.  Maybe the Readme table is corrupt in some way, or maybe it was just happenstance that the error occurred when that table was being processed by the compact operation.

 

I would do the following:

 

1. Make a backup copy of the database.

 

2. Delete the MSysCompactError table.

 

3. Compact the database.

 

4. If the table doesn't reappear, assume the database is fine.

 

5. If the table reappears, try to eliminate any corruption by creating a new database and importing all the tables into it (except MSysCompactError), as well as any other objects that may need to be in it.  Also set any properties that may need to match the original back-end.

 

6. If all tables and objects import successfully, compact the new database to make sure the MSysCompactError doesn't reappear.  If it does, and indicates the same table, try recreating the table from scratch and copy/pasting or importing its contents.

 

7. When you get to the point that compacting no longer causes the error table to appear, swap this new database in place of the original back-end.

That is giving a different impression than "loading behind".   Lots of people have complained about that one, with no diagnostics to suggest.  This is the first time I have seen a report of "chasing a window".  The closest that I have seen that could be related to that was watching screens scroll all the way to the bottom and then resist any attempt to move them back.  My explanation for that phenomenon was that it could be an example of duelling pointer managers.  E.g. one getting a signal that would make it move the pointer and then another one getting the same signal and doing the same thing, so then the first would get a new signal and the loop would repeat until it would not make sense to try doing anything more, e.g. the window had scrolled to the bottom.  My workaround for that was to use a third pointer device Mousekeys to interrupt the (assumed) loop by trying to reverse the direction with an Up arrow.

In fact, I can imagine a scenario where a user would perceive that he was "chasing" a window.  It would involve the Move command.   You can initiate a window Move by Alt-Space M and a Cursor key.  The Cursor key engages the pointer device in a drag operation on the window's title bar.  Before the Cursor key is pressed the pointer device would be moving with a 4-arrow pointer.  So, if you were moving your mouse but its pointer was lagging I think you might get the impression that a window move being done by that was pushing the window instead of dragging it.  In this scenario you would want to click your left mouse button to "drop" the window being dragged and free the pointer from the drag operation.
