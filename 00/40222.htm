¡firedog say:

I think there are a few misunderstandings here, so some background may be of interest: 
 

  • You have at least two different calendars: the one you see when you're not signed in, the Default calendar, and the one belonging to the Microsoft Account you sign in with. You switch from one to the other by using the right-most button on the Home tab of the ribbon to Sign in or Sign out of Windows Live Mail.
     
  • The buttons on the Connection tab are commands, not settings. If you're signed in, the Stop signing in button will be active. If you're not signed in, it won't.
     
  • Windows Live Mail will remember the signed-in state from one session to the next. If you select Remember me and sign me in automatically in the Sign in dialogue, you won't be invited to sign in at the next program start - it will happen automatically. If when you're signed in, you click Stop signing in, you won't be invited to sign in at the next program start.
     
  • If you have been using the default calendar without ever having signed in, the first time you do sign in, this calendar's events will be transferred to the account calendar. This is apparently what happened to you two years ago ("About two years ago I was experimenting and did log into my WLM-account. After restarting WLM my calendar had disappeared"). 
     
  • You say that you have routinely been cancelling the request to sign in on program start, which presumably means that you were using the default calendar. You also say that the biggest instance of WLCalendarStore.edb is the one in the backup folder for the account calendar. This would indicate that the same transfer of calendar data from the default calendar to the account one happened again, probably when you applied the update patch as you surmise.
     
  • The ESENT databases containing calendars (and different ones for contacts and messages) consist of a principal repository (in this case WLCalendarStore.edb) and a number of auxiliary files. The database won't work properly without these files[1]. Any attempt to move a calendar in the file system must include the helper files.
     
  • I'm not sure of the distinction between  WLM Store Folder and C:\\Users\Pim\AppData\Local\Microsoft\Windows Live Mail. These will normally be identical unless you have relocated the store folder.
     
  • The patch that you applied is supposed to enable continued synchronization of mail, calendar and contacts between Windows Live Mail and Outlook.com after accounts are upgraded to Outlook Mail. Has the account you signed in with been upgraded? Visit mail.live.com to see. Do you have this same account configured in Windows Live Mail as a mail account?

While it may be possible to move calendars around in the file system, I think you're more likely to be able to recover your data from the backup that exists. Let's hear whether your account has been upgraded before doing anything else.

[1] Each transaction - adding a new event, or editing an existing one, for example - is written to a log file. Log files are capped at 4MB; transactions are recorded in edb.log, and when it reaches 4MB, it is renamed with a serial number (e.g. edb003E0.log) and a new edb.log file is created for the next transactions. These log files will normally be replayed in quiet moments or after program shut-down and their contents written to the main .edb file. The .chk file keeps track of how far the replay process has got so it can resume at the right place next time. Log files - two or more - should be in the LogFiles folder, along with two reserve files.

hairyfool say:

Setting up multiboot should not have required any changes to the BIOS settings. In a mutiboot environment all the BIOS does is hand boot control to the hard disk exactly the same as with a single O/S, it is the boot code on the hard disk (the Grub Loader if the second O/S in Linux) that handles choosing which actual O/S is loaded by default and giving you the choice within a timeout.

The alternative would have been if you had two hard drives and installed each O/S on their own drive with the other disk disconnected. That is strictly speaking not a multiboot system and in that case you would have to tell the BIOS which disk to boot from to get a specific O/S.

As your computer successfully boots Windows at the moment the next step is to  boot from your chosen distribution media for Linux and tell it to install it to the hard disk leaving the current O/S intact and to install the Grub Loader.
