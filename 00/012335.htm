For loan you should have at least 2 tables - one for widgets and second for usage.

I can reproduce something very similar, but I haven't been able to pin down the reason or a simple way to work around the problem.

Perhaps others here could try the following:
 a. create, save and close an Excel workbook called kt1234.xlsx with the following rows:
k t
1 text1
2 text2
3 text3
4 text4

 b. create a new MailMerge Main document (I saved mine and called it kt1234.docx). Make it a Form Letters type merge (probably irrelevant), and attach kt1234.xlsx

 c. drag the k mergefield into the document

 d. apply the following filter:
t Equals text2
 
 e. in the Preview (step 5) step through the records. There should be 1
 f. Apply the following filter:
t Greater Than text2

 f. in the Preview, step through the records. There should be 2
 g. APply the following filter again:
t Equals text2

 h. in the Preview, step through the records. Here, I still see 2, rather than the expected 1.
 i. when I look at the filter, it is still set to the "Greater Than" filter.

The problem does not seem to occur if I only have a header and 3 data rows rather than 4 data rows, and apply the same filters. As far as I can tell, if I don't actually move from record to record in step 5, the problem does not occur.

In a previous conversation about a similar issue, I made the suggestion that it could result from Word getting confused about the data type of the column in Word, and issuing SQL that was appropriate for a number rather than for a string (or vice versa). Now, I suspect it is something more basic than that. 

In this case, I can solve the problem quite easily by starting again and never applying the Greater Than filter. So it is probably worth clearing all the filters, save/close the document, re-open and apply the correct filters straight away.

The other thing I did here that made it work was to use VBA and issue the correct VBA for the Equals filter in the Immediate Window (it should all be on one line):

activedocument.MailMerge.DataSource.QueryString = "SELECT * FROM Macintosh HD:Users:myuser:Documents:kt1234.xlsx WHERE ((t = 'text2'))"

So that might be worth trying as well, but obviously requires that you know how to use the VB Editor and how to write the correct SQL for your filter/sort criteria (this is a very simple "Word" dialect of SQL which allows you to specify a field list, the FROM, a WHERE and an ORDER BY if you specify a sort).

  • Client seems to struggle with idea of slack.  They are instead used to Excel-Based plans where something starts on a given date, takes x time and ends on y date.  And our team is getting used to MS Project and the way it works (I.e., not like Excel).

  • If a predecessor is a placeholder in time but that predecessor's date changes so much that it no longer can act as placeholder, what is the best option to guide the start date of a given task(s)?  NOTE:  I have used constraints and now get constant error messages in the plan!

john - project say:

Georgie,

Project and Excel are two very different applications, each with it's own focus. Excel is very "open" with regard to what a user can do with the row and column cells. The only interaction between worksheet cells is what the user specifically programs with a formula or direct relationship (e.g. A2=2*A1, C3=sum(A1:A6)). Project on the other hand is specifically designed as a scheduling application. In order for Project to work effectively a minimum set of inputs are needed from the user (e.g. task name, estimated duration, predecessors, and assigned resources). That allows Project's scheduling engine to create the plan using those inputs.

In order to reflect scheduling dynamics (i.e. things that impact task completion like delays, resource availability, etc.) all tasks in a plan should be linked in a logic sequence of execution. Each task should have at least one predecessor and must have at least one successor. If a task has no successors, then it doesn't belong in the plan as it obviously is not required to complete the overall plan. Tasks that are independent of all other tasks MAY have a start-no-earlier-than constraint applied to "nail down" their start date, if it is later than the Project start date, but these tasks are normally rare. An example might be a task that is dependent on availability of a particular resource at some point after the project starts. However, if that resource is not available in time to complete the task such that the successor(s) of that task are not able to be completed in time to meet the end date of the overall plan, then action by the project manager is required to resolve the problem.

I'm not quite sure what you mean by "placeholder predecessor". Either the predecessor task is logically linked to a given task or it is not (i.e. required to start or complete before the next task can start). If by "placeholder" you mean a task that is independent of all others in the plan as noted above, then a start-no-earlier-than constraint may be appropriate. If that date changes, then so be it. The plan should reflect reality as best understood at a given point in time. If things occur that alter the plan (e,g. task is delayed), then the plan needs to be updated to reflect the new reality. That's just basic dynamic scheduling management.

It's nice to think that a task starts on a given date, takes "x" amount of time, and ends on "y" date, but in reality, that is rarely the case. Project is very good at helping to show that such that adjustments to the plan can be made, whereas Excel is not.

Hope this helps.

John
