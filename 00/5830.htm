scott mcdaniel say:

It is a huge request, and really beyond the scope of an online forum. It's often best to pick one thing to ask in these forums, get that right, and then move on to the next thing.

It sounds like you want to use your Contacts list to relate those Contacts with the Artwork they own. Assuming you have a table to track Art, you'd need a Join table between Contacts and Art:

tContact_Art

---------------

ContactID

ArtID

The Join table could be as simple as this, or you may need to add more information about this (like Notes). Just remember that the Join table's only purpose is to track the relationship between Contact and Art. It should not, for example, include information about an Exhibition, or Sales.

You typically represent a Join table by using a Form/Subform setup. Your main (or Parent) form would be based on one of the "joined" tables (Contacts or Art), and the Subform would be based on tContact_Art. For example, if you want to show a Contact and then show all the Art they currently have, you'd base the Subform on tContact_Art and use the ContactID as the child field in the Master/Child link fields on the parent form. On that subform, you'd create a combo based on the Art table and show the user the "name" of the Art from that table, but the combo would store the ArtID value, and the combo would be bound to tContact_Art.ArtID. When the user selects a record from the Art table, Access would store the values in tContact_Art.

As far as Marketing, that's a different topic, and should probably be explored separately. But I'd encourage you to get the structure right before you begin worrying too much about interfaces.

ken sheridan say: When converting a spreadsheet into a set of normalized related tables the basic principle is that data is first inserted into the referenced tables in each one-to-many or, rarely, one-to-one relationship types.  Only then should data being inserted into the referencing tables.  This applies not only to simple relationship types, but also to where a many-to-many relationship type is modelled by a table which resolves the relationship type into two or more one-to-many relationship types, i.e. what Scott refers to as 'join tables'.

The first step therefore is to decide upon the physical model (the tables and relationship types between them) for the database and create and relate the tables, enforcing referential integrity in all relationships.  The Excel data can then be imported and decomposed into the tables.  You'll find an example of the basic principles for undertaking the decomposition in DecomposerDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

This little demo file inserts the data from the imported 'master' table into the operational tables by means of a set of 'append' queries, with an explanation of each stage in the operation.  Note that the imported data is in a simple 'flat file' format, so if the workbook being imported is not in such a format, it should first be imported into a set of tables, and a query created to return the data in the required format from which it can be decomposed.  The query then takes the place of the 'master' table in my demo.

Once you have the data in the operational tables you can design the interface to suit your purposes.  We can help with this, but please post any questions as new threads, not on the back of this thread.
