john w. vinson say:

As a rule, there is no benefit to storing calculated values, and many reasons not to do so. For one thing, you can export data to another program from a Query just as easily as from a Table; for another, almost any calculation is much faster than a disk fetch.

You can certainly base a Form on a query and display the calculated expressions on the Form. They won't be editable of course, but the rest of the fields will be.

The only reason to consider storing the calculated values is hinted at in your last post - if you're rounding or manually tweaking the results of the calculations to "standardize" them. In this case, what you're storing is not in fact (just) the result of the calculation; it has additional intelligence applied to its value, and you have a good case for storing that intelligent input.

 

John W. Vinson/MVP

ken sheridan say: From your description it seems to me that a what you have here is a many-to-many relationship type between two instances of the equipment entity type and the drawings entity type, and would be modelled by an 'adjacency list' table, i.e. you would have a single Equipment table, with one row per item of equipment, with the Drawings table with one row per drawing, and a third table to model the relationship between them, DrawingDetails say, having two foreign key columns each referencing the primary key of Equipment, and a third referencing the primary key of Drawings.

When it comes to equipment dimensions, there are a number of ways these could be represented.  Let's assume a simplistic scenario where the items of equipment are all rectangular boxes with dimensions height, width and depth:

1.  The dimensions could be represented by Height, Width and Depth columns in the Equipment table, which would be appropriate in this simple scenario.

2.  Or the dimensions could be represented in a separate EquipmentDimensions table with columns EquipmentID (referencing the key of Equipment); DimensionType (referencing the key of a Dimensions table, which in this simple example would have three rows with values Height, Width and Depth; and a DimensionValue column.  This model would be more suited to a more complex scenario where  different items of equipment can have different dimension types, and where it might be necessary to introduce new dimension types.

When it comes to the 'standardisation' of dimensions by rounding , there is again a choice of approaches:

1.  The rounded values can be computed whenever necessary by calling the rounding functions in a query.  This is appropriate if the rounded values should always reflect the dimensions in the Equipment of DrawingDetails table (depending which of the above two models is adopted), i.e. if these values should change the 'standardised' rounded values in all existing rows should also change, or similarly, if the basis of rounding should change (e.g. the degree of precision is altered) then the values in  all existing rows should also change.

2.  Alternatively the rounded values can be computed and stored in columns in the DrawingDetails table, or in a related DrawingDetailDimensions table akin the the EquipmentDimensions  table.  This is appropriate only if the rounded values must remain fixed as those computed when the rows are inserted regardless of any changes in the equipment dimensions and/or the precision of rounding.  It is analogous to an invoice where you would not want the prices in old invoices updated as the prices of products is changed over time.

Whichever approach is adopted, the data from the various table can be brought together in a query for ex[ort to the CAD software, either applying the rounded functions to 'standardize' the dimensions from the values in the tables, or, if the rounded values are stored, including thse column sion the query's result table.

To find drawings which already cover particular items of equipment would require a simple query in which the EquipmentID values as parameters could be selected via an appropriate dialogue form, or in code in a form's module to determine if a user is 'reinventing the wheel'.

scottgem say:

Open Linked Table Manager from that machine and see where the tables link to. If the folder is on your C Drive, even if its shared, you can't use a drive letter. you have to use UNC. To use, go back to your machine and open Linked Table Manager. Check the Always prompt box and then start the relink. The standard Windows dialog box will open. On the left side scroll down until you see Network. Click the arrow to the left and you should see a list of computers or shares. You need to drill down through that until you find the back end file to select that.

UNC coding uses the following syntax:

\\servername\sharename\foldername\filename

So servername woukld be the name of your computer, sharename, would be the name you gave the shared folder. 

Note also that Read/Write permissions are not enough, they need Read/Write/Create/Modify/Delete. Basicially everything but full control.
