ken sheridan say: You said in your original post that you have separate Rents and Receipts tables.  That being the case you should be able to use my queries with only the minor alteration's of changing the table and column names and subtracting the Credit value per row in the final query from the Debit value, rather than vice versa as in my demos, which return credit balances.

You need to understand that relational databases use set operations, so it's not a case of carrying forward a balance from one row to the next in sequence in order to compute the new balance.  That would be how it's done in a spreadsheet, not a relational database.  The balances are computed in the latter by the summation of all credit minus debit values (or vice versa in your case) over the subset of all rows up to and including the current row in date order.  In my demo the primary key acts as the tie breaker in the event of two or more transactions on the same day.

Where separate Credits and Debits tables are used it is first necessary to bring both tables together into a single result table, which is done by a UNION ALL query.  The summation of the values up to and including the current row is then done by joining two instances of the query's result table, returning the detailed data from one instance, and the sum of the currency values from the other instance, which is the balance.

If you have trouble transposing my queries into your database, post full details of the two table definitions.  I should then be able to give you the relevant SQL statements for the queries needed.

ken sheridan say: These are the queries from my demo:

SELECT Q1.CustomerID, Q1.TransactionDate, Q1.Credit, Q1.Debit,
SUM(Q2.Credit-Q2.Debit) AS Balance
FROM qryCustomerTransactions AS Q1 INNER JOIN qryCustomerTransactions AS Q2
ON (Q2.CustomerID=Q1.CustomerID) AND (Q2.TransactionDate<=Q1.TransactionDate)
AND (Q2.TransactionID<=Q1.TransactionID OR Q2.TransactionDate<>Q1.TransactionDate)
GROUP BY Q1.CustomerID, Q1.TransactionDate, Q1.TransactionID, Q1.Credit, Q1.Debit
ORDER BY Q1.CustomerID, Q1.TransactionDate DESC , Q1.TransactionID DESC;

SELECT "C" & Format(CreditID,"000000000") As TransactionID,
CustomerID, TransactionDate, 0 As Debit, Credit
FROM Credits
UNION ALL
SELECT "D" & Format(DebitID,"000000000"),
CustomerID, TransactionDate,Debit, 0
FROM Debits;

In the latter, qryCustomerTransactions, the primary keys of the two tables are CreditID and DebitID, which are analogous to your ReceiptNo and RentInvoiceID.  Your TenantNo columns are analogous to my CustomerID columns.  The fact that they are of text data type is immaterial.  My Debit column is analogous to your Total column, and my Credit column to your RecieptAmount column.

So, if you take my queries and substitute your column names for mine the final query should return the balances in the same way that mine does.

gabriel bratton say:

I followed up on this and it turns out the Product Team put the feature to turn off searching nickname cache behind a registry key.  Since I am not able to repro the issue I have not been able to test it so please try setting this key and let me know if it works.

The first thing to do is confirm you are running the March 2015 Public Update version of Outlook 15.0.4701.1000. 

Next, add this registry key.  You can copy this text below into notepad and save it as a *.reg file and then double click to add into the registry.

Windows Registry Editor Version 5.00

[HKEY_CURRENT_USER\Software\Policies\Microsoft\Office\15.0\Outlook]
"DoNotIncludeNickNameCacheInSearch"=dword:00000001

Or, you can manually add it.

1.  Start, Run, Regedit, click OK

2.  Navigated to HKEY_CURRENT_USER\Software\Policies\Microsoft\Office\15.0\Outlook

3.  Right click Outlook and add a new DWORD (32-bit) value DoNotIncludeNickNameCacheInSearch

4.  Double click on DoNotIncludeNickNameCacheInSearch and for Value data put 1

gabriel bratton say:

The registry key is documented in this KB article, https://support.microsoft.com/en-us/kb/3054208/.

After you set this key Outlook will no long display entries from the nickname cache in people search.  If you see any other duplicates they must be coming from another source.  Carefully review them to make sure.
