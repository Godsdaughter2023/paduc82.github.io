Control Panel - Troubleshooting 

Check to see if you have the latest available driver. Download and install the latest

How to manually install drivers.

Check with System Maker (Motherboard Maker for custom systems) for updates
to the BIOS, low level chipset drivers, and the major on-board and separate device
drivers. Run DriverView - set VIEW to Hide Microsoft drivers - update those without
Dump in their name.

DriverView - Free - utility displays the list of all device drivers currently loaded
on your system. For each driver in the list, additional useful information is
displayed: load address of the driver, description, version, product name,
company that created the driver, and more.
http://www.nirsoft.net/utils/driverview.html

For Drivers check System Maker, as fallbacks, and the Device Maker's for which
are the most current. Control Panel - Device Manager - Sound - write down
the make and complete model of your sound adapter - double click - Driver's
tab - write down the version info. Now click UPdate Driver (this may not do
anything as MS is far behind certifying drivers) - then Right Click - Uninstall -
REBOOT this will refresh the driver stack.

Repeat that for Network - Network Card (NIC), Wifi, Display Adapters, Mouse
and Keyboard if 3rd party with their own software and drivers and any other
major device drivers you have.

Now go to System Maker's site (Dell, HP, Toshiba as examples) (as rollback) and
then Device Maker's site (Realtek, Intel, Nvidia, ATI as examples) and get their
latest versions. (Look for BIOS, Chipset, and software updates at System Maker's
site while there.)

Download - SAVE - go to where you put them - Right Click - RUN AD ADMIN -
REBOOT after each installation.

Always check in Device Manager - Drivers tab to be sure the version you are
installing actually shows up. This is because some drivers rollback before the
latest is installed (sound drivers particularly do this) so install a driver - reboot
- check in Device Manager to be sure it is installed and repeat as needed.

Repeat at Device Makers - BTW at Device Makers DO NOT RUN THEIR SCANNER -
check manually by model.

Manually look at manufacturer's sites for drivers - and Device Maker's sites.

They are all on the same subnet (saves on routers) 192.168.0.0/16, but each domain's machines are grouped into different pseudo subnets (192.168.0.0, 192.168.1.0, 192.168.8.0), and there is only one DHCP server with one scope handing out IP4 addresses in the range 192.168.0.100-199. As a machine is installed, it picks up an IP address from the scope, but after installation of the OS, it is given a static IP in the range designated for the domain to which the machine will be joined.

There was no IP6 scope configured on the DHCP server (but the IPv6 group was up) and IP6 on each machine was left enabled and set to DHCP. Therein lies the problem.

I was noticing some strange behavior. After installing a "new" server and promoting it to DC for a new domain, I then attempted to add another machine to the new domain, but the domain controller could not be found. Investigating the DNS on the new DC, I found that the DNS was missing all of the _msdcs entries (for which I still don't have a solid explanation, but I'm pretty sure it's related to the overarching DHCP issue). Manually adding all of the _msdcs entries in DNS (which was laborious) did not allow the new machine to join the new domain, but I noticed that the new machine was trying to get SRV records from the DC in the domain that managed the DHCP service via the link local IP6 address, which was not the new DC.

This forced me to investigate IP6 which, until now, I had happily ignored. There is some useful reading on the 'net around IP6, but the most concise is an old discussion by Rand Morimoto on NetworkWorld, which also points to a source of unique IP6 addresses that you can use behind your router.

I configured IP6 with static addresses on every machine from a range of addresses in the fdxx space (obtained from www.simpledns.com), including configuring IP6 addresses for DNS servers on all NICs and configured an IP6 scope on the DHCP server and all new machines were immediately able to join any nominated domain. 

The Problem (Search List):

However, frequently, particularly during database operations, machines could not find each other without a FQDN. 'ipconfig /all' showed that the search list contained one domain name (which was not the domain of which the machine was a member), so unqualified machine references would fail. The catch was that the properties for the NICs in both IP4 and IP6 did not specify a search list, so where was the spurious reference originating? This was a significant issue with a database server that couldn't find itself using the server\instance (viz CASSIOPEIA\POWER) notation. It was only accessible as localhost or by using its IP address. Another symptom was Report Server and SQL Server Agent failing due to not being able to find their databases.

Conclusion:

After some experimentation, I nailed down the source of the search list to the lack of (or an unconfigured) Microsoft DHCP server. By default, a Microsoft DHCP server, even without a configured IPv6 scope, will still have a default '00024 Domain Search List' property defined in the IPv6 group. This will contain the name of the domain to which the DHCP server belongs and it's this (if the IPv6 group is up) that provides the search list on all machines in the subnet even if NICs are configured with static IP6 addresses and the search list fields in the NIC properties is blank. 

If there is no Microsoft DHCP server and the subnet is using a DHCP server provided by some other provider, commonly a router, you will likely see 'mshome.com' in the search list as this appears to be hard coded into the registry.

The Solution:

The solution in my case was to complete configuration of the DHCP IPv6 group and configure '00024 Domain Search List' on the DHCP server in the IPv6 group with the list of domains on the subnet. Another option is to disable IP6 on the NICs, but at the cost of a lot of functionality. IPv6 is worth the time and effort to come to grips with.

During back-to-back meetings or when you’ve scheduled two meetings at the same time, sometimes you might see an “!UNINVITED!” meeting attendee on the meeting participants list.
