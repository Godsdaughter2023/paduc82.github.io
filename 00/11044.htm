faysal farooqui say:

while you find the difference in between 2 date with a single - (mins) sign, it returns the number of days. Just you have did A5 = A2 + A4 is correct way. While it is a question about years and months, simply it converts to Days and add these numbers of days with A2 date.

If you would like to return years, months or days from 2 different dates, then use the below formula:

=DATEDIF(Starting_date,Ending_Date,"Y") returning year

=DATEDIF(Starting_date,Ending_Date,"YM") returning months after removing year

=DATEDIF(Starting_date,Ending_Date,"MD") returning days after removing months

Use the below formula to find the difference in between 2 days with Year, Month and Days

=CONCATENATE(DATEDIF(A1,A3,"Y")," Y ",DATEDIF(A1,A3,"YM")," M ",DATEDIF(A1,A3,"MD")," D ")

This is known problem. Computers count in powers of 2, people count in powers of 10. Some powers of 10 numbers do now translate completely into powers of 2, specifically fractions ie 1/3.

joeu2004 say:

Bernie wrote:

using parens can force Excel's calculation engine to ignore the implied precision that is normal for formula results

More correctly, using parentheses defeats a kludge that Excel implemented in order to hide the normal precision of the arithmetic.

The kludge is demonstrated, but poorly described under the

But the kludge is implemented and applied inconsistently.

For example, even though =A2/B2 - INT(A2/B2) returns exactly zero (0.00E+00) when A2=13.95 and B2=0.45, =IF(A2/B2 - INT(A2/B2) = 0, TRUE) returns FALSE(!).  Moreover, =SUM(A1/B1, -INT(A1/B1)) returns the exact arithmetic result, about -3.55E-15.

In any case, A2/B2 - INT(A2/B2) is not a correct implementation of MOD(A2,B2).  The correct implementation is A2 - B2*INT(A2/B2).  Try it with A2=14.05 (13.95+0.10) and B2=0.45.

And MOD(100*A2,100*B2) is not the correct work-around.  Ostensibly, it should be MOD(100*A2,100*B2)/100.

But even that does not work consistently.  Try it with A2=8.55 (19*0.45) and B2=0.45.  As usual, when we expect arithmetic to be accurate to some number of decimal places, we should that number of decimal places.

Importing BMP Files into ACCESS DB shown as a package (file name and icon) instead of image/photo

Greetings,

This is probably caused by re-registering .bmp files with another photo editing/viewing software (e.g Photoshop) in association with the .bmp files.

Usually BMP Files in access DB used as an OLE Object, the default software that should open BMP files would be "Microsoft Paint"

Right Click any .bmp file -> Open with -> Choose Default -> Microsoft Paint

That alone will not fix it, you need to edit registry to accept the association with "PAINT"

From "RUN" or search or "Win key on keyboard + R"  type ->  regedit

First off (Backup your registry)

File -> Export -> Choose Export Range (ALL) -> Save it as backup just in case

Browse to

HKEY_CLASSES_ROOT\.bmp

Right Click on (Default) then Modify the value to Paint.Picture

Close Regedit

Start your DB and try to import again, should work by then.
