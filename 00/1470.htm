boblarson say:

Curtis:

 

SharePoint is a team collaboration tool as well as a content management system and document management system for sharing through an Intranet or over the Internet.  It has quite a lot of good functionality.  Microsoft Access worked (could use linked SharePoint lists) with SharePoint Services 3.0 (it might have worked with prior versions as well but I only have direct knowledge of the 3.0 as I had a contract I was working on where they used Access as the frontend to a SharePoint Services 3.0 set of lists) in Access 2003 and maybe earlier but Access 2007 added some functionality with that (but it was still slow and very limited in what it could do) but it also didn't allow for setting referential integrity or relationships for your tables.

 

Access 2010 also works with SharePoint but it is much more functional with SharePoint 2010 with Access Web Services.  That allows you to create forms and reports which render in a browser and allow for data entry (if you want) and reporting.  But you can also use the full Access client as well.  And, you can keep the Access frontend file there on SharePoint which lets anyone authorized to open the Access client file (ACCDB) directly from SharePoint which then downloads to their machine and opens.  It can then be worked on and "synched" which will ensure that the latest design changes are there. 

 

You can also mix web objects with client only objects which gives you a lot of flexibility so that the web objects will work from the browser and the client objects will work from the full Access desktop client.  It isn't perfect as the SharePoint lists (and Access Web Services Engine) can handle more rows of data than previously but it isn't optimal for tables with very large number of rows.  They are working on the next version of Access and I am optimistic that it will get better and better.

 

So, hopefully that helps a bit.

Here are procedures to correct the WMI repository that causes the Windows Security Center to report that no antivirus, antispyware, or firewall is active, yet MSE is actively protecting your PC and in green status.

 

When running this in Vista7, you must run the procedures as an administrator, elevated.

 

Method A

Start>Run>services.msc [enter]

Scroll down to Windows Management Instrumentation and double-click it.

Now click on the "Pause" button. Leave that window open and double-click

My Computer. Navigate to %systemroot%\Windows\System32\wbem (where

%systemroot% is the drive where XP is installed). Delete the Repository

folder and *only* the Repository folder. Now go back to the WMI service

window you left open and restart the service.

This will rebuild the Repository and hopefully straighten out the

incorrect entries for all your duplicates.

In order to see the Windows files, you may need to unhide them:

Make sure you are able to see all hidden files and extensions (View tab

in Folder Options).

Check "Display the contents of system folders".

Check "Show hidden files and folders".

Uncheck "Hide protected operating system files" and click "OK" to the

dialog box.

 

Method B from MVP Torgeir Bakken (more elegant)

Open a command window (Start/Run --> cmd.exe) and run the following commands:

 

net stop winmgmt

cd /d %windir%\system32\wbem

ren repository repository.old 

(or delete it using the command "rd /s repository" instead of the ren command)

net start winmgmt

It may take a minute or so to complete while WMI rebuilds the database.
