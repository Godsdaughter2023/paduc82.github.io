robert aldwinckle on forums say:

 Windows automatically started projecting my screen.

Some users with certain video adapters see what is known as "black screen with cursor".   So, since they see the cursor, it is really blank, not totally black.  An explanation for this phenomenon is that the video adapter has inexplicably created a phantom primary screen and what the users are looking at is a real secondary.  This can only happen if the Display configuration is Extend.  A solution then is to use the Project dialog to put it into Duplicate at least or Second Screen Only.  Note that you can switch from Extend to Duplicate by pressing Win-P, CursorUp, Enter; so you don't even need to see the Project dialog to do that.  In fact, under the hypothesis you would not see the Project dialog because it would be on the phantom primary.  There may be a problem doing this Duplicate method if the screen resolution of the phantom primary and the real secondary are different.  Perhaps that is what you were noticing that you described as "projecting my screen"?  Then in that case it would probably be better to switch to Second Screen Only or PC Screen Only.  Tip: you can start Narrator and use its assistance for navigating "in the dark".  Press Win+Ctrl+Enter to start Narrator.

For doing that the structure would be a tree-structured hierarchy, e.g. in a very simple model:

Categories
….CategoryID  (PK)
….Category

Subcategories
….SubcategoryID  (PK)
….Subcategory
….CategoryID  (FK)

Components
….ComponentID  (PK)
….Component
….SubcategoryID  (FK)

Note that the referencing table Components at the bottom of the tree references only that level in the hierarchy immediately above it.  In the above simple example it does not need to reference Categories as the category is functionally determined by the subcategory, so to reference both would introduce redundancy and the risk of update anomalies.

Press  Alt+F11 and paste the macro on the white surface.
Private Sub Worksheet_Change(ByVal Target As Range)
Dim N1 As String
On Error GoTo exitsub
Columns("C").Find(What:=Range("N1").Value, _
 LookIn:=xlValues, _
 LookAt:=xlWhole, SearchOrder:=xlByRows, _
 SearchDirection:=xlNext, _
 MatchCase:=False, _
 SearchFormat:=False).Activate
exitsub:
End Sub

Office 2007 creates "xxxxxxxx.tmp" files in same directory - Microsoft Community

twilobyte say:

Office 2007 creates "xxxxxxxx.tmp" files in same directory upon save/exit

This problem applies to both Word and Excel documents (and possibly other office programs as well, but I haven't tested them), saved in either 2003 or 2007 formats, and happens to users without full permissions on a network drive.

Duplicating the error:  A user with read/write permissions on a shared folder opens an Excel file owned by "Administrator" account.   A temporary file starting with "~<filename>" is created.  However, when the user saves and exits the file, the temporary file renames itself to a random 8-character followed by ".tmp" extension.  (e.g. A03948DD.tmp)

It seems the .TMP files are left behind because the User does not have permission to change ownership of the file (without the Modify flag).  Naturally, the workaround to this problem is by giving the user "Modify" permissions on the shared folder, but doing so introduces a security hole because the "Modify" flag allows the user to take ownership of all files in that folder and to delete them at will. 

Note:  The .TMP problem does NOT happen if the user already owns the file.  It only happens when the file is owned by somebody else (e.g. "administrator")

I've ruled out the following:

1) NOT due to antivirus software (we run Symantec Endpoint Protection 11).  I booted the computer in "Safe Mode" which only loads core Windows components, and the same problem with the "xxxxxxxx.tmp" files occurred. 

2) NOT due to third-party applications or TSR's because the problem still happens in Safe Mode. 

3) NOT due to Excel Add-ins.  I ran Excel /safe (launches Excel in safe mode) and the problem still persists.


It may be important to note that this problem is specific to Office 2007, and not 2003.   We are slowly upgrading our clients to 2007 from 2003, and it has only happened on 2007 systems thus far.

I have never had to give "Modify" permissions to a user to save Office documents correctly (e.g. without the leftover "xxxxxxxx.tmp" files), and would rather not leave our files in such a vulnerable state. 

Has anyone seen a solution for this?  Please help. 

Thanks,
Jon

Server is 2003 Server R2 Standard x64
Clients are Windows 7 32bit, Office 2007 Pro Plus).

 

 

 

joe's answer say:

Creating .tmp files is part of Word's genetic coding, and their generation
cannot be stopped... except of course by not using Word at all. ;-)

.tmp files are how Word manages the pieces of files as you work with them.
Think of them as pieces of scrap paper, without which Word could not do what
it does. When Word closes a file, the associated tmp files are deleted (in
theory). When Word crashes, tmp files often get left over. The leftover bits
can be deleted

As explained in http://support.microsoft.com/?kbid=211632, there are certain
kinds of temp files that must be stored in the same folder as the open
document, and if that's the desktop then the temp files will be in your
face. Store the documents in any subfolder on the disk, and the temp files
will be out of sight unless you happen to go looking for them.
