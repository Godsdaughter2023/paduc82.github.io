I "thought" I had the same problem... Here's what I found out.  You don't actually make a new note in the notebook, you have to make a new page on the note that is "Notes from [mobile device name]"

Adding and removing pages on the computer are reflected on the mobile device and vice versa.

 

Hope this helps (don't know why Microsoft couldn't give you the answers).

qa guy commelec say:

The easiest solution is to use a UNION query that will combine the sources into one.

Each department should not have a separate table but use a field in a common table to indicate the department.

My company had three layers - Annex, Department, Section - Using 3 digit numbering we could pull reports at each level - whole company, Annex, Department, Section.

scott mcdaniel say:

If your Production Data is in the same table for all Departments (and it should be), then you can create a single report and Group that by the Department.

If your Production Data is NOT in the same table, then as Karl suggested you'd somehow have to UNION that data together. Union Queries can be fussy, and they must contain the same number of columns, so this can sometimes be a non-starter.

Also, if your Production Data is not in the same table I'd have to ask "why". Unless you have distinctly different departments (like Automotive and Clothing, which would entail distinctly different production processes) then it would seem the production

You can also create individual reports for each Department, and then combine those in a single "master" report by using Subreports. 


scottgem say:

Without knowing the structure we have to guess. If you have queries where the department is hard coded as criteria, then it may be possible to just make a copy of one of these queries and remove the department criteria. 

On a separate note, criteria like that shouldn't be hard coded. You don't need a separate query/report for each department. Instead, you can have one query that shows all the data and one report. You can then filter the report for a selected department.

If you have a form OPEN that is bound to a table/query then you try to run an update query (for example what you're trying to do in the Before Update event), it is going to conflict.  It is as if you have two users updating the same data - it can't be done.  So, instead of running it in the BEFORE UPDATE event you will need to save the data in the form first before running the Update query in the AFTER UPDATE event of the form.
