iPad Pro handwriting not searchable in OneNote 2016 - Microsoft Community

scottmartindale say:

iPad Pro handwriting not searchable in OneNote 2016

I am having a problem where the handwritten text that write on my iPad Pro are not searchable after being opened in OneNote 2016. 

I have notebooks open on OneDrive in both the iOS and Windows versions and so they are synched over OneDrive.  The handwriting is searchable on the iPad, but not in OneNote 2016. 

I have looked for solutions online, but only found something about inserting an icon from the Gem add in and that will somehow force OneNote to process the page for handwritten text. However, that does not seem to be working for me.

Does anyone have any ideas?  I have the latest version of OneNote for the iPad installed as well as "Enable Experimental Features" is turned on. 

Thanks.

swischner say:
I have a short answer and a somewhat longer explanation for you.

The short answer: Just wait. Indexing of the handwritten notes takes a while as it is done on the Microsoft servers, not on your iPad. Usually it takes several minutes but I've also experienced hours.

Now for the explanation:
The OneNote developers are currently creating a big bess in terms of compatibility between the (how they see it) old and outdated OneNote versions for WIndows desktop (e.g. OneNote 2013 and 2016) and their "Modern" OneNotes (iOS, Mac, Windows 10 UWP....). Those incompatibilities involve several details like audio formats and, as you are experiencing, also ink formats.

Since the first release of OneNote for Windows (OneNote 2003) it always had the ability to recognize and index handwritten notes. It used a mechanism which is part of Windows itself, not part of OneNote.
OneNote 2016 is still using the handwriting recognition of Windows. Therefor the indexing is happening instantly. Interesting fact: The recognized text is even placed inside the note, it's just hidden.
Now, when OneNote for Mac, iOS, Android came, there was no handwriting recognition inside the operating system that OneNote could use. Instead it's using a server based service from Microsoft which is part of the "Cognitive Services". So if you create some handwriting on your iPad, it is sent to the Microsoft servers, placed in a waiting queue and eventually indexed. You can not search for your notes, until this has happened. And it can take quite a while. To check, if the indexing is finished, you can try and search the note on the ipad itself. If that works, the indexing is finished and it should work in OneNote 2016 also.

There is only one way to speed things up, but this is not how it is supposed nor very convenient:
If you see the handwritten note on OneNote 2016, you can select it, roght click and select "treat selected ink as handwriting". Then the Windows based indexing kicks in.

You can even distinguish which index for handwritten notes is used by the type of highlighting a searched and found text. If only the lines itself show a faint yellow highlighting, the online index is used. If there is a big yellow rectangle under the text, it's the WIndows index.

For Item 1 you do need some VBA. "I won't be adding rows.  I will as you termed it, always be adding a new field in each query every time there is a new spreadsheet added to the local folder."

For Item 3 you are think incorrectly and you really need to adjust your thinking. Data in a relational database is stored as values in fields within a record. The above quote indicates you are thinking spreadsheet not database. You don't add fields to a table for each new value. You add a record in a table. The values in the fields in that table will define the data stored in the record. Maybe an example will help. Lets say for example, you are reporting car sales. In your mind, if you get sales data on a new car model, you add a field to your table. But that is not correct. You would have a table like this:

tblCarSales

CarSalesID (PK Autonumber)

SalesPeriod (Date/time)

CarModel

SalesAmount

With a table like that, you don't change your table with each new model, you simply identify the model in the record. To put it more technically, your entity is car sales. The attributes of that entity are the period covered, the model of car and the sales amount. Your entities are your tables and your attributes are the fields within them.

The issue here is that you are mixing up spreadsheet layout with the concept of a database.  Excel and Access are distinct applications and data needs to be setup very differently.

If we look at simply the limits for the maximum number of columns of each application we get

Access ->      255 Fields (what you refer to as Columns)

Excel    -> 16 384 Columns

So while in Excel it may be conceivable to add new columns for every new import, this is not the way to go in Access as you will quickly hit the limit for the number of columns.  Normally, we don't really see tables with more than 20 fields.

If you truly wish to move forward with Access, and not Excel, you need to get back to basics here and review the raw data and normalize your data properly for use in a database -> determine the proper table structure.  Once this is done, then we can look at the import process.
