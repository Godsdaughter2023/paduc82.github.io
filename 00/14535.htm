Sounds like maybe you have "autoupdate" turned on for your normal style. Lets check:

  • Home tab / Styles group: click on the isty-bitsy-teeny-tiny-incredibly-stupid-little-dialog-box-launcher button (on the right end of the styles group name bar) to display the Styles pane

If you find that attribute is turned on use one of these macros to disable it

1.1 MACRO to disable “Auto Update” ALL Styles

NOTE:         The Automatically update document styles under Tools/Templates & Add-ins means that no matter what changes you make to style definitions (NOT direct formatting changes) will NOT stick. Every time you close & reopen the document (& possibly when you save, I can’t recall), the styles from the attached template are copied to the document & all your style changes are lost.

The automatically update under the Format/Style/Modify means that when you add any direct formatting to a charcter or paragraph, the style definition is changed to reflect your addition.  Therefore, all styles with that name in the document change.

By default in NORMAL.DOT MS creates some styles with “AUTO UPDATE STYLE” option turned on. So whenever a change is made to format of text using that style, all uses of that style are changed also. This macro will scan thru all styles and turn of the option. Most useful when you have to recreate NORMAL.DOT.

Public Sub RemoveAutomaticallyUpdate()

' RemoveAutomaticallyUpdate Macro

' Created by Gary Frieder

'Dim aSty As Style

For Each aSty In ActiveDocument.Styles

   If aSty.Type = wdStyleTypeParagraph Then

   aSty.AutomaticallyUpdate = False

   End If

Next aSty

End Sub

scottgem say:

Access is not like other programs. Other programs are file centric, in that when you are working with the program you are working with a overall file. So the program knows if you have made changes to the file when you close the program.

An Access file, on the other hand, is a collection of objects. Each object is a separate entity on its own. So each record in a database table is separate. Because of record locking requirements, when a record loses focus it is committed. it is up to the developer of an Access app to code in fail safes to the user to prompt them if they want to commit changes when an object within the Access file is closed.

albert d. kallal say:

Actually, if you use an Accounting package, or some inventory system, or some order entry system – NONE of them ask you to save your work when you exit the application. So you might be entering invoices and data all day long.

Also because this is a database system, when you enter a new customer or say a new invoice, then EVERYONE else in the building likely wants to see and be able to view and inquire about that invoice. Access is also multi-user.

So while perhaps Excel or word functions with a simple document save, but database and information systems don’t work that way. If you ever used any computers and other kinds of software, you often see that how access works is rather common in this industry.

Access is an “information” system. So think of using some order entry system, or an accounting program. So there exists significant amount of software that allow you to launch the application, work on information and enter data. You can well note that closing such applications (such as an accounting package) does not in general “ask” you to save your data.

Because Access allows you to build information systems, and is not just a simple document system, then it works like most other information systems and applications you typically see in our industry.

So such data and information systems “save” the data as you work so that others using that system ALSO can use and view the information in real time. So database systems tend to work this way due to the fact that they allow multiple users to “share” that

So you don’t want to compare Access to say a simple document application like word – but think of any information system you use at work – such systems rare ask you to save all your work and data entry at the end of day – but such systems tend to save data and the information you enter “as you work” so that everyone else can then view + use such data.

Access certainly is a rather different product compared to most of the office suite, and thus “simple” comparisons to the rest of the office suite does not apply. When you use Access you have to put on a “developers” hat and way of thinking.

You can certainly introduce a “save” prompt for data you enter into a form. So this can prevent changes to a given form, but that concept is far different then closing down an Access application. The opening or closing of the application has little to do with the concept of saving data into tables. And you could also provide a “edit” button on a form to prevent accidental changes (so in place of save - you make it harder to modify data on a form). With smartphone and touch screens - we see a big trend moving away from having to hit a save button.

.Out of the box form data is saved automatic for you – this is to speed up ease of use (people get real tired real fast when having to hit save over and over all day long). However, certainly the case exists where you want to make saving of one record more difficult or you wish to prevent accidental changing of data in a form.
