but try this sample...

(copy paste values only)

 

1) both workboks are open

2) define name on WB1 on sheet1 is range...  A1:D20 = DataWB1

3) paste on WB2 on sheet1, top left cell A1

 

Sub CopyPaste_TwoWBksOpen()

Const myData As String = "DataWB1" '<<< define name
Dim ws1 As Worksheet, ws2 As Worksheet
Application.ScreenUpdating = False
Set ws1 = Workbooks("WB1.xlsm").Sheets("Sheet1")  '<<< change names
Set ws2 = Workbooks("WB2.xlsm").Sheets("Sheet1")  '<<< change names

Application.CutCopyMode = False

Application.ScreenUpdating = True
End Sub

bart billiet say:

hello,

just wanted to let you know that i found the issue.

in the synchronization process with aad connect, i had specified employeeid as the immutableid (default = objectguid).
when the convert-msoldomaintofederated command runs, it creates a relying party trust in adfs, and not the one specified in aadconnect!

i wasn't aware that it requires a manual change to the adfs claim rule to actually change "objectguid" to "employeeid".
see also: http://blogs.perficient.com/microsoft/2015/04/office-365-why-you-need-to-understand-immutableid/ (impact to adfs)


once the claims rule was changed as described in the referenced article, login to the o365 portal was working as expected.
