1) There are multiple records in the table referenced by Subform1 for a record referenced by the Main form
2) I select (click) a record in Subform1 & the corresponding records appear in Subform2
3) If I then click a record in Subform2, the record in Subform1 CHANGES - & since the 2 subforms synchronize correctly, the selected record in Subform2 now also CHANGES to a DIFFERENT record than was displayed in Subform2 when I selected it

Note: If I start off by selecting a record in Subform2 w/o first selecting a record in Subform1, the record I select stays selected, which is the desired behavior.

I can get around the bizarre behavior programmatically by adapting the code in "ACC: Using Code to Dynamically Synchronize Two Forms (95/97)" (http://support.microsoft.com/kb/149940/en-us), but why should I need to? 

ken sheridan say: I'm not completely clear as to how your form and subform is set up.  It sounds from your description as though both subforms are independently housed within the same parent form.  For this they would have to be correlated.  The easiest means of doing this is via a hidden unbound control in the parent form whose ControlSource is a reference to the key of the first subform, and whose name is the LinkMasterFields property of the second subform.  This is an inefficient method of correlation, however, and a far better solution in terms of performance is to base the second subform on a query which references the key of the first subform as a parameter.  The second subform is requeried in the Current event procedure of the first subform.

The other option, and the simplest to implement, is for the second subform to be nested within the first subform, but this requires the latter to be in single form view.

You'll find examples of both solutions as CorrelatedSubs.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file uses data from Northwind to show the order dates per customer in one subform, and the order details per order in the second subform.  In one example both subforms are in continuous forms view and correlated as described above so that navigating to an order date in the first requeries the second; in the other the second subform is nested within the first, which is in single form view and includes an unbound combo box to navigate to an order from those made by the customer currently selected in the parent form.

scottgem say:
I can get around the bizarre behavior programmatically by adapting the code in "ACC: Using Code to Dynamically Synchronize Two Forms (95/97)" (http://support.microsoft.com/kb/149940/en-us), but why should I need to?  
Because, as Ken suggested, you have two separate subforms linked to the main form. For what you want, you would need nested subforms and you can't nest a subform on a continuous form. 

So, to keep subform 2 in sync with subform 1 you can't link subform 2 to the main form, or even to subform 1. You have to synchronize them.

The properties of a combobox are that it can display multiple columns for the user to select an item. But once the user chooses an item, then the list disappears and the control displays the first non-zero width column in the Rowsource of the combo.

Your table should have at least three columns. A TopicID (usually an autonumber), the next column should be individual topics. And the third column should be the topic headings. The first column should be the bound column and should be set to zero width. Since the user is selecting an individual topic, that should be the first displayed column, The third column would show the topic headings. This should only be necessary if some individual topics can appear under different topic Headings. This is the way combos are designed to work.

An alternative to this setup is to use Cascading combos. This is where the user will select a topic heading from one combo which would then filter the individual topics only for those under the Topic Heading.

Another alternative is to use a Listbox instead. A listbox can display multiple columns.

If you need further clarification please feel free to ask.

ken sheridan say: I would recommend you use two correlated combo boxes, the first being unbound and used to select a heading, the second to select a topic from a list restricted on the basis of the selection in the first.  You'll find an example in DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file, the section on 'entering data via a form/subforms' includes a Contacts form in which three correlated combo boxes are used tom select a country, then, region, then  city, the first two being unbound, the last bound to the CityID column in the form's Contacts table.  The correlation is achieved by referencing the prior combo box as a parameter in the RowSource properties of the second two controls, and by requerying the following combo box in the AfterUpdate event procedure's of the first two.  The demo also illustrates how the NotInList event procedure of each can be used to insert a new row into the relevant referenced table.

In a context like this, where the controls reflect a tree-structured hierarchy, it's important that the table into which data is being entered has a column bound only to the lowest level of the hierarchy, CityID in the demo.  For the table to also include RegionID and CountryID  would introduce redundancy, and the table would not be in Third Normal Form (3NF) due to the transitive dependencies on the key, with the consequent risk of update anomalies.  This is why the first two combo boxes are unbound in the demo.

Correlated controls like this are easily built in a form in single form view.  In continuous forms view, however, this would result in some controls in rows other than the current row appearing to be empty.  This can be solved in two ways.  Firstly 'natural' keys can be used rather than 'surrogate' numeric keys.  This not always possible, cities being an example, as city names can legitimately be duplicated, so cannot be used as a key.  The alternative solution is to use a 'hybrid' control made up of a read-only text box superimposed on a combo box to give the appearance of a single combo box control.  You'll find examples of this technique in ComboDemo.zip in the same OneDrive folder.

myall_blues say:

Do you get any other information other than not able to open the file? Can you navigate to the file and just open it normally (i.e. do you have appropriate permissions)?

It is possible to link to a server. I have found using the =HYPERLINK formula easiest because it is more transparent and a bit easier to troubleshoot.

So for example you can provide a full UNC path to the server:

=HYPERLINK("\\ServerName\Accounts\December\161207.pdf","Dec account")

or if you have part of the path mapped to a drive letter, you can use that:

=HYPERLINK("P:\December\161207.pdf","Dec account"), where P: is mapped to \\ServerName\Accounts
