dirk goldgar say:

You can trap errors of this sort in the form's Error event and subsitute your own message.  This error happens to be error number 3022, so you could have VBA code like this for the form's Error event:

'------ start of example code ------

Private Sub Form_Error(DataErr As Integer, Response As Integer)

    Select Case DataErr
    
        Case 3022
            MsgBox _
                "The Patient ID you entered already exists. " & _
                    "Please enter a different value.", _
                vbExclamation, _
                "Already Exists"

            
            Me.PatientID.SetFocus

            Response = acDataErrContinue
            
        Case Else
            Response = acDataErrDisplay
        
    End Select
    
End Sub
'------ end of example code ------

By assigning acDataErrContinue to the response argument, the above code is telling Access to continue without displaying its own error message.  In the "Else" case, for other errors the code isn't handling, assigning acDataErrDisplay to Response tells Access to go ahead and display its built-in error message.

dirk goldgar say:

I'm not completely sure I grasp which images have to do with which issues, but one thing is clear:  because the PatientID field is a required field, you can't clear it by deleting the value you entered in it.  Access won't let you edit a required field to set it to Null.  That's the cause of the error message, "You must enter a value in the 'PatientTb.PatientID field." 

Now, in the case of a new record where the value began with a Null value, you can clear the field manually by pressing the Escape key.  Or you can clear it programmatically by calling either the control's Undo method (which "undoes" the changes to the control) or the form's Undo method (which "undoes" all changes to the form).  Unfortunately, in the form's Error event, you can no longer undo the control, so you would have to undo the whole form.  If that is acceptable under the circumstances, you may could do it in your Form_Error code, by adding one line like this:

'------ start of code ------

Private Sub Form_Error(DataErr As Integer, Response As Integer)

    Select Case DataErr
    
         Case 3022

             MsgBox _
                 "The Patient ID you entered already exists. " & _
                     "Please enter a different value.", _
                 vbExclamation, _
                 "Already Exists"
            

             Me.Undo

             Me.PatientID.SetFocus

             Response = acDataErrContinue
            
         Case Else
             Response = acDataErrDisplay
        
     End Select

End Sub

'------ end of code ------

If you want to be able to undo just the control, though, I think you'd do better to check for a duplicate value in the control's BeforeUpdate event, rather than waiting for the form's Error event to be raised.  That might solve all your problems.  Code for that would be like this:

'------ start of code ------

Private Sub PatientID_BeforeUpdate(Cancel As Integer)

    If Not IsNull(Me.PatientID) Then

        If IsNull(DLookup("PatientID", "PatientTb", "PatientID=" & Me.PatientID)) Then
            ' This ID doesn't exist.
        Else
            MsgBox _
                "The Patient ID you entered already exists. " & _
                    "Please enter a different value.", _
                vbExclamation, _
                "Already Exists"
       
            Cancel = True
            Me.PatientID.Undo

        End If

    End If

End Sub

'------ end of code ------

You have too many double quotes in your field construction. Basically, the required field construction is:


{IF{MERGEFIELD APP_FEE}= "" "To pay the fee, you will need to do one of the following:¶

{IF«APP_FEE»= "" "To pay the fee, you will need to do one of the following:¶

Send a check or money order..."}


Note: The field brace pairs (i.e. '{ }') for the above examples are all created in the document itself, via Ctrl-F9 (Cmd-F9 on a Mac); you can't simply type them or copy & paste them from this message. Nor is it practical to add them via any of the standard Word dialogues. Likewise, you can't type or copy & paste the chevrons (i.e. '« »') - they're part of the actual mergefields, which you can insert from the 'Insert Merge Field' dropdown. The spaces represented in the field construction are all required. Instead of the ¶ symbols, you should use real line/paragraph breaks.

Darla you are correct. Since you didn't provide the date field names, Mimmo57 provided some "placeholders".

It's important to understand what you mean by "number of months between two date fields".

The DateDiff() function returns the number of intervals crossed from one date to another.

For instance:

DateDiff("M",#10/31/2015#,#11/1/2015#)  = 1

DateDiff("M",#10/1/2015#,#11/30/2015#) = 1

These have the same number of months but significant difference in days:

DateDiff("D",#10/31/2015#,#11/1/2015#) = 1

DateDiff("D",#10/1/2015#,#11/30/2015#) = 60

ken sheridan say: Following on from what Duane said about what is actually meant by "number of months between two date fields", the following function was originally written to determine the age in months of infants, e.g. a child born today, 2015-11-16, would be one month old on 2015-12-16, two months old on 2106-01-16 and so on:

Public Function GetAgeMonths(varDoB As Variant, Optional varAgeAt As Variant) As Variant

    Dim intmonths As Integer

    If Not IsNull(varDoB) Then
        If IsMissing(varAgeAt) Then varAgeAt = VBA.Date
        
        intmonths = DateDiff("m", varDoB, varAgeAt)
        
        If Day(varDoB) > Day(varAgeAt) Then
            intmonths = intmonths - 1
        End If
        
        GetAgeMonths = intmonths
    End If
    
End Function

You can see how it works in the debug window:

? GetAgeMonths(#2015-07-12#,#2015-11-16#)
 4
? GetAgeMonths(#2015-07-26#,#2015-11-16#)
 3

To get the age in months along with the remaining days in the format m:dd the following function can be used:

Public Function GetAgeMonthsDays(varDoB As Variant, Optional varAgeAt As Variant) As Variant

    Dim intmonths As Integer
    Dim intDays As Integer

    If Not IsNull(varDoB) Then
        If IsMissing(varAgeAt) Then varAgeAt = VBA.Date
        
        intmonths = DateDiff("m", varDoB, varAgeAt)
        
        If Day(varDoB) > Day(varAgeAt) Then
            intmonths = intmonths - 1
            intDays = DateDiff("d", DateAdd("m", intmonths, varDoB), varAgeAt)
        Else
            intDays = Day(varAgeAt) - Day(varDoB)
        End If
        
        GetAgeMonthsDays = intmonths & ":" & Format(intDays, "00")
    End If
    
End Function

Again you can see how it works in the debug window:

? GetAgeMonthsDays(#2015-07-12#,#2015-11-16#)
4:04
? GetAgeMonthsDays(#2015-07-26#,#2015-11-16#)
3:21

Leap years create a little ambiguity, e.g.

? GetAgeMonthsDays(#2015-02-28#,#2016-02-29#)
12:01
? GetAgeMonthsDays(#2016-02-29#,#2017-02-28#)
11:30

But should these really be 12:00 in each case?  It's a judgement call of course; there is no hard and fast answer.

This formula must be **array-entered**:

=MAX(COUNTIF(MyRange,MyRange))=1
----------------------------------------
To *array-enter* a formula, after entering
the formula into the cell or formula bar, hold down
<ctrl><shift> while hitting <enter>.  If you did this
correctly, Excel will place braces {...} around the formula.

Where MyRange would be, in your example, A1:D1

Caveats:

In versions of Excel prior to 2007, MyRange cannot refer to an entire
column.

If you are comparing text entries, the comparison will NOT be case
sensitive.
