debbie b@trhmc say:

ken sheridan say: It sounds like you might be using a memo field as a data structure to hold multiple values of a particular attribute of whatever each record represents.  If so it's not a good design.  Each value should be stored as a separate row in a related table which has two columns, one a foreign key referencing the primary key of the current table, the other a foreign key referencing the primary key of another table which contains one row for each attribute value.  Say for instance that the memo field is recording employee's qualifications you'd have tables Employees, Qualifications and EmployeeQualifications, the last model the many-to-many relationship type between Employees and Qualifications (what's sometimes called a 'junction' table).  This table might also have other non-key columns e.g. in my example, for where and when the qualification was obtained

You can parse out the individual items from the memo field using the following function:

Public Function GetListItems(strItemList As String, _
                    strDelimiter As String, _
                    intStart As Integer, _
                    intNumber As Integer) As String

    Dim intDelimLength As Integer
    Dim intSubStart As Integer
    Dim intSubEnd As Integer
    Dim n As Integer
    
    strItemList = strItemList & strDelimiter
    intDelimLength = Len(strDelimiter)
    intSubStart = 1
    
    ' get start of substring
    If intStart > 1 Then
        For n = 1 To intStart - 1
            intSubStart = InStr(intSubStart, strItemList, strDelimiter) + Len(strDelimiter)
        Next n
    End If
    
    intSubEnd = intSubStart
    
    ' get position immediately after end of substring
    For n = 1 To intNumber
        If InStr(intSubEnd + 1, strItemList, strDelimiter) > 0 Then
            intSubEnd = InStr(intSubEnd + 1, strItemList, strDelimiter)
        End If
    Next n
    
    ' return value is substring starting at intSubStart
    ' and of length intSubEnd - intSubStart
    GetListItems = Mid(strItemList, intSubStart, intSubEnd - intSubStart)

End Function

You can see how it works in the debug window like this:

? GetListItems("John" & Chr(10) & "Mary" & Chr(10) & "Bob" & Chr(10) & "Jean",Chr(10),1,1)
John
? GetListItems("John" & Chr(10) & "Mary" & Chr(10) & "Bob" & Chr(10) & "Jean",Chr(10),2,1)
Mary
? GetListItems("John" & Chr(10) & "Mary" & Chr(10) & "Bob" & Chr(10) & "Jean",Chr(10),3,1)
Bob
? GetListItems("John" & Chr(10) & "Mary" & Chr(10) & "Bob" & Chr(10) & "Jean",Chr(10),4,1)
Jean
? GetListItems("John" & Chr(10) & "Mary" & Chr(10) & "Bob" & Chr(10) & "Jean",Chr(10),5,1)

In the last one 5 is passed into the function as the intStart argument, but there are only 4 items in the list, so it returns a zero-length string.  You can write some code, therefore which loops through each row in a recordset of employees and for each row loops until the a zero-length string is returned by the above function, at each iteration of this second loop (bar the last of course) inserting a row into (in my example) EmployeeQualifications with the current EmployeeID and the value returned by the function.  Once the EmployeeQualifications table is filled in this way it's a simple task to fill the Qualifications table with an 'append' query in which you SELECT DISTINCT Qualification FROM EmployeeQualifications.  Enforced relationships between the tables can then be created.

Your report would then be based on a query which joins Employees and EmployeeQualifications on EmployeeID.  The report would be grouped by employee, with the employee data in a group header and the qualifications in the detail section.

Structuring your data in a correctly normalized set of tables as above also makes it much easier to query the database as you have specific values at each column position, so don't need to parse a memo field to find someone with a particular qualification for instance.  And when it comes to aggregating the data it becomes a very simple task, whereas with multiple values in a memo field this would be tricky.

BTW in case you are wondering what the intNumber argument in the function is for, it allows you to return multiple adjacent values, e.g.

? GetListItems("John" & Chr(10) & "Mary" & Chr(10) & "Bob" & Chr(10) & "Jean",Chr(10),2,2)
Mary
Bob

i.e. the 2 values starting at position 2 in the list.

     An array is just a data structure that allows you to store information in a matrix like construct.  It echos the organization of a worksheet in that a two dimensional array would have "rows" and "columns"

     

    A variant array is set up so you can store anything in the individual elements (intersect of row and column).  In the code you show they are creating a two dimensional array, but there is only a single column and the same number of rows as found in the listobject DataBodyRange.

    You can see this in the ReDim statement where vUserName is dimensioned to be of size 1 to the number of rows in the DataBodyRange and 1 column (1 to 1)  the first argument in the x to y, the x is the number of rows.  The second (the y) is the number of columns.

     

    An array can be a single dimension such as 

     

    redim v1(1 to 20)  and that would have 20 elements.  Or it can have 3 or more dimensions. 

    while v is a variant variable in the above code, it actually contains an array.  The array is all the values in the ListObject (List in Excel 2003, Table in Excel 2007 and 2010). 

     

    When created by referencing a multidimensional range, the lower bound in each dimension is 1 by default. 

     

    so they loop through all the rows in v and build the value of vUserName.  Then they put the values in UserName back in the List/Table in the column named "UserName".
