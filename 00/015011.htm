Indeed, I am familiar with this problem which I have experienced and often struggled with many years ago. That is until I realized that dragging photos or graphics on the Publisher work area to put them into position results in disproportional micro-resizing of the order of a single pixel, which the Publisher coding engine compensates by adding odd border lines. This problem exists even when using the
Publisher will always locate the insertion point of an image at the center of the window, so I have tried to slide the work document so as to insert the image as close as possible to its required location on the page, but while it minimizes the dragging of the picture, it is guess work. Sometimes it works, sometimes it doesn't.

I have long since adopted the technique of not leaving photos or graphics on the Publisher work document, not even background images, and it eliminated the problem completely.

To be more explicit, and for the benefit of less experienced users, the technique consists of:

in your website folder to store the photos and graphics and name them appropriately i.e. page1pics, page2pics, etc . . . then saving in them all the photos and graphics relevant to that website.

2- inserting one picture at a time on the Publisher work document and dragging it into position as normally done.

Tools menu on the top bar and ticking Snap to Object

HTML Code Fragment box (using the second-last icon on the left-hand tool bar) and placing the cursor to snap to the top left-hand corner of the photo or graphic, then dragging the cursor to snap to the bottom right-hand corner of the picture and releasing the left-click.

Send to Back icon (on the top bar). Now the HTML fragment box is under the picture. Deselect it by clicking in an empty spot outside.


HTML Fragment box and dragging the bottom right-hand corner down and to the right to enlarge the box by a couple of pixels in both directions.

HTML Fragment box to open the dialog box and inserting the following code:

<img src="name-of-folder-in-which-the-image-is-stored/name-of-image.jpg or whatever">

If the image carries a link, then the code is:

<a href="full-URL-of-the-link"><img src="name-of-the-folder-in-which-the-image-is-stored/name-of-image.jpg or whatever" border=0></a>

Done. Now proceed with the next graphic or photo.

Note that the pictures will not show up in preview. However, there will be no displacement of other content as the red cross picture substitute will appear within the HTML fragment boxes. You'll only see the pictures when your website has been saved in your website folder and you try it out on your local browser.

Epilogue:
This technique may seem like a time consuming task, but I can assure you that with practice it becomes very fast and second-nature. Besides, there is a couple of shortcuts which reduce the time spent doing this.

For example:

Insert and drag into the correct position all the images required for each page on the work document.
On the first page, apply the procedure described above for the first picture.
When the HTML Fragment box for the first picture is left on its own, copy it (Ctrl+C then Ctrl+V)
Then drag it to snap on the top left-hand corner of the next picture, deselect, and select the bottom corner button and extend ithe box, or reduce it as the case may be, to snap to the bottom-left corner of the image.
Send to Back, deselect, then select the picture and Cut.
Resize the box (1or 2 pixels down and right). Double click and change the details of the relevant folder and image in the code of the HTML Fragment.
Repeat the above steps until all pictures on the page are done, then proceed to the next page.

Also . . .
Photos or graphics that are common to all pages, like a banner or link buttons for example, can be saved into a folder called . . . say common, or shared. These will presumably occupy the exact same position on each page.
Once you've created an HTML Fragment box for each of the common images, you can group them and copy the group on each page by simply switching page button at the bottom of the window and clicking the paste icon at the top tool bar on each page in sequence. In fact this is the way to start the entire process.


One advantage is that your Publisher work document file is much smaller from the fact it contains no pictures whatsoever.

Anyway, each to his own . . . it works for me.

While we're at it,
the appearance of unsolicited odd border lines on text boxes is the result of element encroachment. It is prevalent on the vertical axis. I found long ago that it is best to observe a vertical separation between  elements of one button + 2 or 3 pixels.

In many cases you can get away with two elements touching vertically, but almost never three. For reasons I have not yet understood the problem rarely occurs on the horizontal axis.

Particular attention must be paid to the size of text boxes as the display size of  fonts varies greatly from one browser to the next. Odd border lines may appear in one browser and not in another.

If design constraints requires elements to have no separation at all, then it is best to insert all relevant elements, including text, in a single HTML Fragment box of suitable dimensions.

hansv mvp say:
If the name does NOT start with "Exception", you set Cancel to True. The code then starts to save the workbook under another name, but before that happens, the Workbook_BeforeSave event occurs, so you get stuck in an endless loop.

rik_uk say:

Hi Otto place this code in your workbook module. The public declared variable is used to test if the save action should be cancelled on the second iteration after the SaveAs statement is made...

Public SaveMyFile As Boolean
Private Sub Workbook_BeforeSave(ByVal SaveAsUI As Boolean, Cancel As Boolean)
Dim ThePath As String
Dim FileName As String
Dim BeginDate As String
    If Left(ThisWorkbook.Name, 9) = "Exception" Then
        Exit Sub
    ElseIf Not SaveMyFile Then
        SaveMyFile = True
        ThePath = ThisWorkbook.Path
        BeginDate = Format(Range("C4"), "mmmm d, yyyy")
        FileName = "Exception to my work book" & Range("A7") & " " & BeginDate
        ActiveWorkbook.SaveAs ThePath & "\" & FileName & ".xlsm", FileFormat:=52
    Else
        SaveMyFile = False
    End If
    If SaveMyFile Then Cancel = True
    SaveMyFile = Not SaveMyFile
End Sub

charles kenyon say:

To get rid of all built-in Building Blocks:

  1. Create a blank document.
  2. Type the following as a note:
    "Blank Building Blocks replacer. Save this in the appropriate building blocks folder."
  3. Save it as a template named "Built-In Building Blocks.dotx."
  4. Put it in the appropriate folder, replacing the file that is already there (with Word closed).
  5. You will need to do this on each computer for each user.

I believe this location will be:

  • AppData\Roaming\Microsoft\Document Building Blocks\1033\14\ for Office 2010,
  • AppData\Roaming\Microsoft\Document Building Blocks\1033\15\ for Office 2013, and
  • AppData\Roaming\Microsoft\Document Building Blocks\1033\16\ for Office 2016

The "1033" is the English Language designation. Each of those is in the user profile. Office 2007 uses a different file that includes custom building blocks. Mr. Tao's method would be more appropriate for that file.

Simply deleting the Built-In Building Blocks file will result in Word recreating it. You have to replace it with a template that contains no building blocks.

Note, this will not prevent your users from later deleting your file which will get them the building blocks back.

Where can Building Blocks be stored?

Building Blocks & AutoText

Note that Felix Tao's method will also remove all building blocks. My method is a chain saw, his is a scalpel. Both can be useful tools. The Built-In building blocks contains the built-in options for page numbers, headers and footers, cover pages, text boxes, and more. Emptying it is a drastic measure.

Word 2007 VBA: Deleting to Bottom of Current Page/Document

jay freedman say:

Sub DeleteToEndOfPage()
    Dim myRange As Range
    Set myRange = Selection.Range
    myRange.End = ActiveDocument.Bookmarks("\page").Range.End
    myRange.Delete
End Sub

Sub DeleteToEndOfDocument()
    Dim myRange As Range
    Set myRange = Selection.Range
    myRange.End = ActiveDocument.Range.End
    myRange.Delete
End Sub
