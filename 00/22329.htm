I built such a database for my company. It was completed in August 2001. It turned out to be invaluable as all our artwork was lost just a few weeks later on 9/11. Our insurers gave us 100% reimbursement because the catalog of artwork was so well documented.

Anyway, the trick here is to save each photograph of the art piece with the barcodeID as the file name. So, lets say the barcode was 123456, then that file would be saved as 123456.jpg.

All the photos should be saved in one folder.

Your table is simple:

tblArtwork

BarcodeID

Barcode2

Title

ArtDescription

On the form you include an unbound image control. In the On Current even of the form you use the following:

Me.imgArt.Picture = "d:\folder\" & Me.barcodeID & ".jpg"

As you cycle through the records, the image control will display the picture.

As for keyword searches. You can do something like this:

Dim strSql As String

strSql = "SELECT Title, ArtDescription, BarcodeID FROM tblArtWork WHERE ArtDescription Like '*" & Me.txtKeyword & "*';"
DoCmd.OpenForm "frmKeywordSearch", acNormal, , , acFormReadOnly, acDialog, strSql

This opens a form listing all titles where the entered keyword matches the description.

ken sheridan say: First thing, remove the CustFName and CustLName columns from the TransactionTBL table.  You only need the foreign key fkCustomerID column.  Repeating the names introduces redundancy and the table is consequently not  normalized to Third Normal Form (3NF) due to the transitive dependencies of these columns on the key, so is open to the risk of update anomalies.

You'll find examples of queries for returning balances in Balances.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates a number of queries, both updatable and read-only.  For a report the latter should be used as these use a join, which is far more efficient than calling the DSum function.  The latter is only necessary for an updatable query which you'd use as the RecordSource for a form in which you need to enter or edit the data.  The appropriate model for what you want is that for a 'Single table with credit and debit columns for amounts - read only'

A couple of points to note:

1.  The query brings the primary key TransactionID into play as a tie-breaker in the event of two or more transactions taking place on the same date.

2.  The values in the Credit and Debit columns are positive numbers in each case, so the query sums (Credit - Debit).  Only if the debit amounts wrer negatve numbers would you sum (Credit + Debit).

There are some quirks with the use of the default Heading 1, Heading 2 and Heading 3 styles within a table and, congratulations, you have found another one.

To overcome the problem, you can define and assign another style to the text to which the Heading 2 style is assigned and then modify the STYLEREF fields so that they refer to that style rather than to Heading 2.
