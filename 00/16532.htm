SELECT EnvLocationsForestPark.EnvLocID, EnvLocationsForestPark.LocationDescription, EnvLocationsForestPark.[Contact Area], Val(Mid(CStr(Rnd([EnvLocID])),3,4)) AS RANDOM_NUM INTO tblCollection
FROM EnvLocationsForestPark;


SELECT tblCollection.EnvLocID, tblCollection.LocationDescription, tblCollection.[Contact Area], ((SELECT Count(*) FROM tblCollection AS [XX] WHERE [XX].RANDOM_NUM <= tblCollection.RANDOM_NUM) Mod 4)+1 AS Collection_QTR
FROM tblCollection
ORDER BY tblCollection.RANDOM_NUM;

brian tillman [outlook mvp 2007-2019] say:

You can set up send/receive or send-only accounts in Outlook.com that will allow you to choose from multiple sending addresses so that you recipient sees the address you want.  It works well enough using the web interface.  If you don't validate the addresses you'll see "on behalf of" in the sending value, but if you validate the address, the "on behalf of" goes away.

 

A problem arises, however, when you want to do the same thing using a mail client like Outlook 2013, which I assume you're now using because of the change to Office 365.  If you create a send-only account with the alternate sending address you wish to use in Outlook 2013 and use the Outlook.com SMTP server for the outgoing traffic, specifying your Outlook.com credentials for authentication, even though Outlook 2013 includes the correct address as the sender, the Outlook.com server will rewite the headers of the message and send it using your primary address.  Your outgoing message will lose all trace of the sending address you specified.

 

Another problem is that you cannot specify alternate display names for the alternate sending addresses, even if you decide to send via the web interface.  While your sending address will be correct, whatever display name you specified for the primary account will be used as the display name for the outgoing message.

 

The only way I know how to prevent that is to authenticate using the credentials that match the sending address and those credentials must be for a separate mailbox on Outlook.com, not an alias of the primary address.  If your alternate addresses are for mailboxes on services other than Outlook.com, you will need to send your messages through those services' SMTP servers, not the Outlook.com server.
