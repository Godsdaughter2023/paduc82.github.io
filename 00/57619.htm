     A "DAX error" refers to Microsoft ERP technology used for also refer to the DAX language ("Data Analysis Expressions"), which is a Power Pivot "formula language" for manipulating operations within relational databases, and as such is not germane to the type of "DAX error" reported in this thread.  "TUCserver object" typically refers to Templated User Controls, which are web server controls (the buttons, text boxes, and so forth displayed on a web page) used in delivering ASP.NET pages and the like to a user's computer, while "secondary instance" refers to the AOS (Application Object Server) service, which coordinates communication between the local client (a user's computer) and the remote server, providing the objects (such as web server controls) and information used by a given application (program) when it connects to a given service using DAX (as do Skype and Office).  There are many reasons why a local client computer might receive this error, but the most common one is simple: lack of network connectivity, even fleetingly, during the "handshake" communication that establishes a connection between client and server.  One would need to parse the logs and/or network traffic - requiring fairly advanced troubleshooting skills and tools - to determine the exact cause in any particular occurrence of this error, but the standard user is likely to find that simply stopping and re-starting the offending program - again, usually Skype (especially if the scheduled task created when a user selects the option "Start Skype when I start Windows" is not re-configured with some delay in the "Advanced settings" section of the "New Trigger" dialog box invoked by clicking the "Trigger" tab of the may be required, and even more rarely, it may be necessary to actually repair Office, or uninstall, reboot, and re-install Skype.

     The connectivity scenario described above may be the most common reason for this error, but as

Support Engineer is required; the key is in the persistence nature of the error - if performing the above do not suffice to keep the error from happening consistently, and the user does not have overall network connectivity problems (i.e., is able to connect to the internet successfully despite the presence of the error), more drastic measures such as System Restore may well be in order.  A subset of such a problem is that described by the original poster to this thread, where the error crops up after a Windows Update.  Despite any recent snafus with Patch Tuesday updates, Microsoft does an excellent job of helping keep users' computers running both securely and smoothly; nevertheless, nothing is perfect, and Windows Update patches may occasionally fail to work as expected with a simple reboot.  Fortunately, the vast majority of such occurrences may typically be corrected if the user exercises a little patience and follows the "RSR" routine: reboot, shutdown, and reboot again.  This procedure, while more time-consuming, has yielded such consistent success for so many years that I always employ it after any updates or program installs on any computer (including those running operating systems other than Windows).
