Rename multiple sheets in a workbook with ascending dates

jeeped say:

Open a blank workbook. Right click on the first worksheet's tab and select View Code to enter the VB Editor. Once there, immediately use the pull down menus to Insert, Module. Paste this code into the new window titled Book1 - Module1 (Code),

Sub AutoDailySheetName()
' AutoDailySheetName Macro
Dim n As Integer
Dim i As Integer
Dim dt As Date
While Worksheets.Count > 1
    Application.DisplayAlerts = False
    Worksheets(2).Delete
    Application.DisplayAlerts = True
Wend
dt = Cells(1, 1).Value
n = DateSerial(Year(dt), Month(dt) + 1, 0) - dt + 1
Sheets(1).Name = Format(dt, "mmm dd")
For i = 2 To n
   Sheets.Add After:=Sheets(Sheets.Count)
   Sheets(i).Name = Format(DateAdd("d", 1, CDate(Sheets(i - 1).Name)), "mmm dd")
Next i
End Sub

... and press ALT+Q to close the VBE and return to Excel.

Now put the start date of the month in Sheet1!A1, e.g. 1/1/10. With that in place, from the Developer tab's Code group, click Macros. Highlight AutoDailySheetName and click Run.

The sheets are named in sequential days beginning with the date in Sheet1!A1. They continue until End-of-Month.

Warning: this macro deletes all worksheets but the first one in order to gain a finite reference point for creating new worksheets.

This worksheet will have to saved as a Macro-Enabled Worksheet.

jay freedman say:
There are several steps involved. It's all described in http://www.word.mvps.org/FAQs/Formatting/NumberingFrontMatter.htm.

The key thing is that each section has its own set of headers and footers, and there are three of each: main, first-page, and even-page. (If you turn on different even and odd pages, then the main header becomes odd-page.) So putting a page number in the first-page header doesn't affect the main (all other pages) header -- you have to put page numbers in both places.

Also, when you first insert a section break, the headers/footers in the later section are linked to the ones in the earlier section. If you need the sections to show different headers/footers, you need to unlink them by turning off the Link To Previous button in the Header & Footer toolbar.

dianeoforegon say: Yes, you never rebuild an Identity with Exchange unless you are forced to rebuild to recover messages under "On My Computer". If you do the rebuild, immediately put in work offline mode and delete Exchange account. That way no corruption is uploaded to the server.

If you rebuild an Identity with Exchange and you have subfolders under the Inbox this can lead to all sorts of problems like you describe including loss of data.

Exchange  Header

Drafts
Deleted Items folder
Junk
Folder A  <-- this is correct

It's best to move folders using your web interface (OWA) but you can drag in Outlook.

In Outlook:
First turn off Group folders. You'll see the header with the name of your Exchange account at the top. Drag folders to the header. Do this slowly. It will take some time for results to sync to server then back to Outlook.

In Browser via OWA
Because of the massive changes The process is best done in your browser. I would also put Outlook into work offline mode during this process. After all is moved on the server via the browser, I would delete the Exchange account and let it all download correctly from the server. If you only have Exchange, just start over in a new Identity.

Exchange Basics

The Inbox syncs more frequently than any other and should be kept as clean as possible. (this is the number one cause of problems with sync). Do NOT make subfolders under the Inbox.

Outlook checks the server for updates every minute, any folder that has updates will subsequently be synced. Since Outlook has a limit on how many folders can be synced at a time, there can be a queue of folders waiting. The Inbox does get high priority so it will generally sync before other folders that also need to sync.

ken sheridan say: Simply adding or subtracting a value to or from the local time is not sufficient.  It does not take account of British Summer Time, which does not start and end on the same date as daylight saving time in other parts of the world, including the USA, and many countries nearer to the equator do not have daylight saving time at all of course.

Coordinated Universal Time (UTC) can be returned by calling the Windows API GetSystemTime function by adding the basUTC module below to your database:

' basUTC
' returns current date/time as UTC (Coordinated Universal Time)
' by calling GetUTC() function

Option Compare Database
Option Explicit

Type SYSTEMTIME
  wYear As Integer
  wMonth As Integer
  wDayOfWeek As Integer
  wDay As Integer
  wHour As Integer
  wMinute As Integer
  wSecond As Integer
  wMilliseconds As Integer
End Type

Declare Sub GetSystemTime Lib "kernel32.dll" (lpSystemTime As SYSTEMTIME)

Public Function GetUTC() As Date

    Dim utctime As SYSTEMTIME
    
    GetSystemTime utctime  ' put the time and date in UTC time into utctime
    With utctime
        GetUTC = DateSerial(.wYear, .wMonth, .wDay) + TimeSerial(.wHour, .wMinute, .wSecond)
    End With
    
End Function

UK time is the same as UTC in winter as it equates with GMT.  British Summer Time (BST) begins at 01:00 GMT on the last Sunday of March and ends at 01:00 GMT (02:00 BST) on the last Sunday of October.  Consequently one hour would need to be added to UTC during that part of the year.  This is easily done by calling the DateAdd function, e.g. in the debug window:

? GetUTC(), DateAdd("h",1,GetUTC())
01/03/2018 12:52:16         01/03/2018 13:52:16

To determine when BST statrs and ends each year I'd suggest adding an auxiliary calendar table of the last Sundays of March and October.  This can be done manually or by using one of the examples in Calendar.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file select the option for 'Last Workday Each Month' and then switch the dialogue form which open to design view and amend the Click event procedure of the cmdCreateTable button as follows:

Private Sub cmdCreateTable_Click()

    On Error GoTo Err_Handler

    Const FORMNAME = "frmMonthLastWorkdayCalendar"
    Const MESSAGETEXT1 = "At least one day of the week must be selected."
    Const MESSAGETEXT2 = "A table name and both dates must be entered."
    Dim ctrl As Control
    Dim blnNoDays As Boolean
    Dim strSQL As String
    
    blnNoDays = True
    
    ' loop through day of week checkboxes and
    ' ensure at least one is True
    For Each ctrl In Me.Controls
        If ctrl.Tag = "CheckMe" Then
            If ctrl = True Then
                blnNoDays = False
                Exit For
            End If
        End If
    Next ctrl
    
    ' if all necessary controls are filled create calendar table
    If Not blnNoDays Then
        If Not IsNull(Me.txtTableName) _
            And Not IsNull(Me.txtDateStart) _
            And Not IsNull(Me.txtDateEnd) Then
            
            DoCmd.Hourglass True
            
            MakeCalendar_DAO _
                Me.txtTableName, _
                CDate(Me.txtDateStart), _
                CDate(Me.txtDateEnd), _
                IIf(Me.chkSun, 1, Null), _
                IIf(Me.chkMon, 2, Null), _
                IIf(Me.chkTue, 3, Null), _
                IIf(Me.chkWed, 4, Null), _
                IIf(Me.chkThu, 5, Null), _
                IIf(Me.chkFri, 6, Null), _
                IIf(Me.chkSat, 7, Null)
            
            ' delete all rows bar last weekday of month
            strSQL = _
                "DELETE * " & _
                "FROM " & txtTableName & " AS T1 " & _
                "WHERE calDate <>" & _
                "   (SELECT MAX(calDate)" & _
                "    FROM " & txtTableName & " AS T2" & _
                "    WHERE YEAR(T2.calDate) = YEAR(T1.calDate)" & _
                "    AND MONTH(T2.calDate) = MONTH(T1.calDate))"
                
            CurrentDb.Execute strSQL, dbFailOnError
            
            ' delete all rows bar March dates
            strSQL = _
                "DELETE * " & _
                "FROM " & txtTableName & _
                " WHERE Month(calDate) <> 3"
                
            CurrentDb.Execute strSQL, dbFailOnError

            ' refresh database window
            Application.RefreshDatabaseWindow
    
            DoCmd.Hourglass False
            ' close calendar form if open and reopen it
            ' using newly created table
            DoCmd.Close acForm, FORMNAME
            DoCmd.OpenForm FORMNAME, OpenArgs:=(Me.txtTableName)
                
        Else
            MsgBox MESSAGETEXT2, vbExclamation, "Invalid Operation"
        End If
    Else
        MsgBox MESSAGETEXT1, vbExclamation, "Invalid Operation"
    End If
    
Exit_Here:
    Exit Sub
    
Err_Handler:
    DoCmd.Hourglass False
    MsgBox Err.Description, vbExclamation, "Error"
    Resume Exit_Here
    
End Sub

Without saving the form switch back to form view and enter BSTCalendar as the table name.  Select Sunday only as the working day.  Then click the Create Table button.

Next, amend the code again as below:

            ' delete all rows bar October dates
            strSQL = _
                "DELETE * " & _
                "FROM " & txtTableName & _
                " WHERE Month(calDate) <> 10"

Then, back in the form enter BSTEndsCalendar as the table name, and again select Sunday only as the working day.  Create this table.

Finally execute the following UPDATE query:

UPDATE BSTCalendar
SET BSTEnds =
DMIN("calDate","BSTEndsCalendar",
"caldate > #" & FORMAT([BstStarts],"yyyy-mm-dd") & "#");

You'll now have a BSTCalendar table with start and end dates for BST over the next 10 years or so.  You can now close the dialogue form, without savings it, and delete the redundant  BSTEndsCalendar table.

This might sound complex, but it only took me a few minutes to do it (far less than it took me to write it down!), and, provided the UK government doesn't change the basis of BST, it will hold good universally for the next ten years (or more if you set the end date for the calendar further into the future in the dialogue form).

If you want to it manually, the table produced by the above is:

BSTStarts        BSTEnds
25/03/2018    28/10/2018
31/03/2019    27/10/2019
29/03/2020    25/10/2020
28/03/2021    31/10/2021
27/03/2022    30/10/2022
26/03/2023    29/10/2023
31/03/2024    27/10/2024
30/03/2025    26/10/2025
29/03/2026    25/10/2026
28/03/2027    31/10/2027

To determine the current UK time from anywhere in the world is then merely a case of calling the following expression:

DateAdd("h",IIF(IsNull(DLookUp("BSTStarts","BSTCalendar","#" & Format(GetUTC(),"yyyy-mm-dd") & "# Between BSTStarts And BSTEnds")),0,1),GetUTC())
