Here's another one.

Create this 2 column table...

Weekly 52
Bi-Weekly 26
Semi-Monthly 24
Monthly 12

Assume that table is in the range A2:B5.

You have a drop down list ro choose the time period in cell D2. To return the numeric value associated with the time period:

=SUMIF(A2:A5,D2,B2:B5)

Note that this method will only work if the value to be returned is a number.

This method will work if the value to be returned is text or numeric:

=VLOOKUP(D2,A2:B5,2,0)

macropod say:

I'd suggest creating a series of templates with bookmarks, document variables or custom document properties, instead of mergefields, wherever data are required. Your Access document-generation function can then open whatever template is required and update whatever bookmarks, document variables or custom document properties it finds - it is a simple matter to test whether a given bookmark, document variable or custom document property exists before trying to update it. With  document variables & custom document properties, if you need to replicate the same item in more than one location in the document, you only need the one document variable or custom document property (referenced via fields in the body of the document), whereas bookmarks would require either a separate bookmark for each reference or cross-references to the ones being updated by the code.

 

Also, if you use document variables or custom document properties, you could, theoretically have all 600 in every template and only reference those that are needed in the body of the document. That way, you don't need to test for their existence before updating them.

macropod say:

Document variables or custom document properties store content in the document's metadata, rather than in the body of the document itself. Their content can be refernced via DOCVARIABLE and DOCPROPERTY fields, respectively, wherever and however often you want the data to appear in the document. For some code to implement custom document properties, see:

http://word.mvps.org/faqs/MacrosVBA/MixedDocProps.htm

 

The reason for not using mergefields in a document automation context is that you need to replace them with whatever values you're outputting, plus you have the overhead of finding the field in the document. Furthermore, you'd need multiple copies of the same mergefield wherever you want to replicate the data. With bookmarks, you can simply input the data at the bookmarked location by referencing the bookmark's range. If you want to replicate the data, all you need to have is cross-references to the bookmarks and to ensure the bookmarked range is extended to span the new content.

 

As for bookmark visibility, you can toggle that on/off. In any event, once the template is properly configured, you really don't need to be able to see them.

alex wang msft say:
Hi Paul,

The official document which recording the limitation of Planner is still not available, but here are a thread in Planner forum has answers to the questions: Planner Limitations
To answer your questions:
1. 250 Active tasks (Not Started, In Progress) per plan
2. 250 assigned tasks per user irrespective of plans
3. The Planner web UI has the limitation to display 100 tasks per bucket, there is no official limit for the bucket limitation in Planner.
4. Planner associates with Office 365 Groups and the comments for tasks within a plan are stored in the Exchange Online mailbox of the Office 365 Group. The limit of the Office 365 group should be 2500.
5. Another limit worth notice is that 1500 tasks can be created by a user irrespective of plans.
6. About the permissions of Planner, currently, the admins could control who can create a plan but no classified permissions within a plan.

Sorry for all the inconvenience. We hope you could share your need in Planner UserVoice. This way other users who have same needs could vote for the idea and get the attention from Planner team. Here are some ideas about this has been posted, you could vote for them and share your comments:

Remove common limitations

Remove 1500 Tasks Created Per User Limit
