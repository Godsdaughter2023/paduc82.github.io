ken sheridan say:

You just need the one Comments table, in which there is a foreign key column such as ProjectID which references the primary key ProjectID column in the Projects table.  The primary key of Projects would normally be an autonumber, and the corresponding foreign key in Comments would be a straightforward long integer number data type.  In addition to this there would be an autonumber CommentID column as the primary key, and CommentDate and Comment columns of date/time and text data type respectively.

For data entry you can base a form on the Comments table and for the foreign key ProjectID column's control use a combo box set up as follows:

ControlSource:    ProjectID

RowSource:         SELECT ProjectID, Project FROM Projects ORDER BY Project;

BoundColumn:    1

ColumnCount:     2

ColumnWidths:   0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit

of the last one to inches. The important thing is that the dimension is zero to hide the first column.

Alternatively you could use a comments subform within a projects parent form, linking the subform to the parent form on ProjectID.

ken sheridan say: PS:  to sort the comments form by project/date base it on the following query:

SELECT Comments.*
FROM Comments INNER JOIN Projects
ON Comments.ProjectID = Projects.ProjectID
ORDER BY Project,CommentDate;

For examples of how to open such a form from a projects form, filtering it to the current project see FormsDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

For an example of how you can include an unbound combo box in a stand-alone comments form to filter it to a specific project, see the FindRecord demo in the same OneDrive folder.

Delete a row or rows - Either above or below certain text - Microsoft Community

mucko72 say:

Delete a row or rows - Either above or below certain text

Hi

I am trying to delete a row/rows above or below a certain row.

Is project capable of doing this using VBA code?

If so I am looking at finding information in the name field and deleting the 2 rows before or after depending on the result it finds.

Thanks... 

john - project say:

Mucko72,

Yes, it is definitely possible to do what you want with VBA.

I didn't discuss this before because it wasn't really applicable for what you were doing (i.e. custom formatting of text), but there are two ways to approach Project VBA code. The best approach is to use background processing since it is faster and generally easier to write, at least in my opinion. The other approach is foreground processing. That is what you get when you record a macro and we used it previously because unfortunately with Project, custom formatting can only be done using foreground processing. Excel on the other hand allows you to format using background processing.

So what's the difference? Background processing operates directly on Project's objects, independent of any views or selections. Foreground processing operates on elements of the current view by selecting those elements. Sometimes, a combination of the two is appropriate or necessary to accomplish what you want. For example, if you want to operate on a filtered set of tasks, the easiest approach is to apply a filter and then operate on the selected tasks (i.e. For Each t in ActiveSelection.Tasks). So, the code uses foreground processing to do the selection and then reverts to background processing to operate on that data. If this sounds/seems a little confusing, it will be more evident when you get deeper into writing macros in Project VBA.

Okay, on to your current question - deleting certain tasks based on Name field criteria. Perhaps the obvious approach is to loop through all tasks in the project and look for the Name field criteria. Once that is found, step back (or forward) and execute a Delete the rows before or after. Although it is not obvious the challenge is how to effectively select the rows to delete since each time a deletion is executed, all task rows are renumbered, so an absolute index using task IDs will not work. Fortunately Project has a fixed task ID and that is the Unique ID field. We can use that to our advantage when deleting tasks.

There are various ways to accomplish your need. A macro written to use either foreground or background processing can be used. I prefer background processing whenever possible.

The following code is a simplified version of code for deleting both task rows directly above a task row that contains the word "Train". A few things must be noted. The code will issue a runtime error if the task row with "Train" does not have at least two task rows above it. The code does not check to see if one of the task rows to be deleted is a summary line, and some other little items that need to be addressed. So, some tweaking will be necessary to apply it to your scenario.

Sub DelRws()
Dim t As Task
Dim ur1 As Single, ur2 As Single
For Each t In ActiveProject.Tasks
    If InStr(1, t.Name, "Train") Then
        ur1 = ActiveProject.Tasks(t.ID - 2).UniqueID
        ur2 = ActiveProject.Tasks(t.ID - 1).UniqueID
        ActiveProject.Tasks.UniqueID(ur1).Delete
        ActiveProject.Tasks.UniqueID(ur2).Delete
    End If
Next t
End Sub

Hope this gets you started.

John

jay freedman say:

I think we're discussing different things. In Microsoft Word lingo, "keyword tags" or just "tags" usually refers to one or more words that are placed in a document property on the File > Info page. These words can be used in the File Explorer's search box to find all documents that contain specific tags. The words themselves may or may not appear in the body of the document. That's what I described before.

The other meaning of "tag" that I'm aware of refers to making documents (usually PDF files) accessible to assistive technology. If this is what you have in mind, take a look at Making Documents Accessible. 

If you want to change the formatting of a word (e.g., make it bold or change its color) wherever it appears, you can use the Replace function for that.

One other feature that marks words throughout a document is the Mark Entry button on the References ribbon in Word. That inserts a hidden field next to one or all of the instances of a word, for the purpose of creating an index. I doubt that this is what you meant, but I include it for completeness.
