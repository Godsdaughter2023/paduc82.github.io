miekiemoes say:

Hi,

This is interesting indeed. By default, cmd.exe isn't supposed to act like this unless additional tweaks have been added to have cmd run in order to take ownership. In this case, it looks like a buggy tweak anyway, so can you remember you've ever tweaked this? Or ever tried a tweak or registry change to take ownership of certain files and folders, but the registry tweak was buggy?

I don't know exactly what has been tweaked here - so I hope you can still remember this, but my guess it's either a cmd association tweak or something that is running as a debugger under cmd.exe.

In case it's a cmd association tweak, you can get the default associations from here: http://www.eightforums.com/tutorials/8486-default-file-associations-restore-windows-8-a.html

Look in the list below there and search for and search for .cmd and download its registry fix in order to restore the default association for .cmd files.

Another possible cause can be that a debugger has been set for cmd.exe, pointing to an additional script or whatever that runs the "take ownership" commands.

So this means, when a debugger is set for a certain program, it will run its debugger instead of the program itself. These debugger references are present under the Image File Execution Options registry key.

You can verify this by going to the following registry key:

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options.

Look if the following subkey exists there: cmd.exe - so this means, if the following key is present:

HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options\cmd.exe - then it's safe to remove this key. Do NOT remove the Image File Execution Options key, only the cmd.exe key under it if present!!!).

By default, there's no cmd.exe key under the Image File Execution Options key anyway.

Hope this helps.
