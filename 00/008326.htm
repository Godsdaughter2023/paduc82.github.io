This is what I did.

If you do an  ipconfig  /displaydns  either after your ping or after your troubleshooter reports it couldn't identify the problem you will probably see your target host name represented by a CNAME record chaining it to one of those other aliases or (if you're lucky) directly to the canonical host name and hence to its address records.   But since there was no lookup done from the dnscache point of view there are no records in there which can help IE find an address to use.   It seems that IE gives up looking for addresses really quickly, so if you put a lookup in there for that name and its addresses it might help IE find one.   E.g.  do

before trying the IE request with the alias and see if your symptom changes.

Looks like I may have confused you by musing about the  ping  -i  TTL  specification?   You had the right command before.    Use  ping -?   to see the help.   The only reason we are using  ping  and hence the  -n 1  and  -w 1   specifications is because that is a convenient way of getting a DNS lookup done and loaded into the dnscache.   We don't care about the  ping's  ICMP  request because it has nothing to do with the ultimate protocol we want to use:  HTTP.   A better simulation of doing that would be to use  telnet  80.  That would also load some lookup into the dnscache  but then we would lose  ping's  display of the reverse lookup which shows us what happened to the lookup that we requested.

As soon as you get the ping's reverse lookup line that is when you should be pressing Enter on the  ipconfig  /displaydns  command that you have ready to go in the other Command Prompt window.

 
I also tried going to e144.cd.akamaiedge.net, but that returned an HTTP 400 Bad Request error.

After all this, the DNS cache was still empty, and I killed the infinite PING as the test was completed.


That's fine.   That proves that  IE  did a lookup for that host name and made an  HTTP  GET  request for its root page.   The thing to do then, again, ASAP that you see any sign that IE has achieved a lookup and is making that request, go back to your Command Prompt and get that  DISPLAYDNS  done.   The point is that the  Time To Live  of these entries is just a few seconds.   That should be loads of time for the things that  IE  would need to do but very little for us to try to use the dnscache as a diagnostic.   All that we are trying to do is find out if we can get a complete lookup cached so that (presumably) IE can use it without having to go to a real DNS and get it done that way.  That way we can claim that whatever the cause of  IE's  problem with your desired host name is it should not be the DNS lookup for it.

A problem with having IE make the request for the canonical host name is that as soon as it gets a lookup done it holds on to it internally for a long time.   So, you would never see signs of another lookup being done for this host by that instance of the browser.   E.g. you would have to close it, verify that all instances of  iexplore.exe  were gone (using Task Manger's Processes tab) and then start another IE.

However, this has just given me another idea for you.  This one would involve having two IE windows with URL ready to be entered in their Address bars.   E.g. one which would cause the canonical host name to be looked up and cached and another for the alias host name which would ASAP try to use it once the first one's lookup (and its addresses) had been cached.   Again, the reason for using two windows would be simply for the speed of making the requests.   I suppose another option would be creating two Favorites to be used instead.   Then you might be able to use only one window to do both requests, stopping the first if necessary.   My experience is that the Stop command is rather unreliable but that might only be because most often scripts are involved and in this case you would be making a single request which you would be assuming would fail anyway.

 

Have you also tried putting an entry for  www.cisco.com  into your  HOSTS  file?   That would avoid all this uncertainty about its lookup and the dnscache.   Another way of removing the dnscache from the picture would be to stop its service.   That would remove it as a source of diagnostics.   I don't know how it would affect the quality of the lookups you would be able to do without it but perhaps the lookup routine would have to change its timeout parameters accordingly and thus give you a possible workaround.   Who knows?   Isn't black box troubleshooting fun?   This is why Open Source is so successful.   Then (potentially) we could know what is really going on.

Using named array constants in a formula

I have several sets of numbers that I have entered into single cells as an array (e.g. Cell A1 contains {1,2,3,4,5} - with the braces). I've named that cell MyList.

I then want to use that array, referencing it by name, in further formulas, such as: =SUM(COUNTIF('OtherPage'!B:B,MyList)), where in column B I'm looking to count the total number of occurrences of any of 1, 2, 3, 4, or 5.

When I write out this formula with {1,2,3,4,5} instead of MyList, I get the correct answer. When I use the name, I get 0.

When I step through the evaluate formula option, the first step is replacing the name with the list, HOWEVER, it does so with double quotes around the array, making the COUNTIF look for that entire string rather than the individual strings within the array.

So --- is it possible to do what I'm seeking? Does my syntax just need a tweak?  I'd prefer to keep these number arrays in a name, as they're used in over a dozen formulas, and the contents of those arrays may change over time - if I just had to change the one named cell, there's far less room to introduce errors into this workbook.

Ann

mike h.. say:

Hi,

Delete that named range you have in A1 and ensure A1 contains this {1,2,3,4,5}

On the formulas tab click 'Define Name' and enter the name MyList

Put this formula into the 'Refers to' box

=Evaluate($A$1)

this will now work when entered as an ARRAY formula, see below for how

=SUM(COUNTIF(OtherPage!B:B,MyList))

or for a non-array formula that does the same

=SUMPRODUCT(--(ISNUMBER(MATCH(B1:B1000,MyList,0))))

This is an array formula which must be entered by pressing CTRL+Shift+Enter
and not just Enter. If you do it correctly then Excel will put curly brackets
around the formula {}. You can't type these yourself. If you edit the formula
you must enter it again with CTRL+Shift+Enter.

EDIT...You have used full columns in your formula and I left them there but really you should avoid this as it slows things down a lot. Use a large range instead.
