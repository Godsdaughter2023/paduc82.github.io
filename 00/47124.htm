wunders say: Somewhere in your wireless configuration, there is a page of "Profiles".  When you connect to a network, your computer will remember the particulars of that network in a profile which can be recalled the next time you connect to that network.  It is possible that there is a profile already set up for the network you are trying to connect to from a time in the past and since then something has changed on that wireless network.

Find the profile page on your wireless configuration (it will be one of the icons by the clock in the tray) and delete any existing profile that matches the SSID name of the network you are having problems with.  After deleting the profile, the next time you attempt to connect it will be a fresh try and it will connect without any pre-conceived notions of what the network is like and , if unsecured, should have no problem connecting.

Update on RAID 1, MIRROR, with Windows 7 - Microsoft Community

septimusfry say:

Update on RAID 1, MIRROR, with Windows 7

I have been struggling to come to terms with mirror behaviour under W7Pro, including following information posted here. So I thought I would post the conclusions of my struggle.

 

I set up a software mirror pair on Seagate Barracuda 1.5TB in April 2010 using the W7 disk management. I was very pleased with the performance and had no problems until recently. My system seemed sluggish, and in the course of my investigation I discovered that the mirror set was resynching and it was doing so for some time...

 

Using perfmon I was able to see that one was acting as a source and the other as a destination, the source was active at around 80mb/s and the destination at about 7Mb/s. I assumed that one of the disks was faulty and broke the mirror set so I could test the suspected disk. I found that it was perfectly OK (using Seagate Tools and HDTune). Incidentally, I am now not certain that the sluggishness was anything to do with the resynching problem.

 

After wasting a lot of time, including reading some useful and some unhelpful negative opinions on this forum, I managed to sort it out, and it is quite simple.

 

I believe that the 7MB/s being used to update the destination of the resynching is probably a design decision by MS which would reduce the impact of the resynching on the performance of the source. In the days when disks were hundreds of MB, resynching would be reasonably quick, but now, resynching may take some time - especially if, like me, you force a complete resynch by breaking the mirror set.

 

I finally got patient and waited. My prediction of the 60 hrs it was going to take turned out to be very accurate. I had learned here that it has to resynch the whole disk, including unused parts, and that turned out to be the case.

 

All is now OK and the performance of my mirrored Barracuda's running at 7200rpm is almost identical (with HDTune) with that of my Velociraptor 300 running at 10000 rpm.

 

The final question is 'why did the resynch start?'. Reading around the forums, I conclude that Windows Update may be the culprit. Some posters have suggested that Windows Update may operate on just one of the mirror set at some time in the  update process. This would indeed trigger a resynch of the affected blocks, but whether it would trigger a complete resynch or just partial is not clear. Unfortunately, there seems to be little source information on this made available by MS.

 

I posted this report simply in the hope that it would help any others encountering the problem. It will pass if you are patient.

 

A final word to a certain poster, to many forums, who ALWAYS simply rubbishes software raid in any form: have you ever been as convinced that you are right only to find out later that you are wrong?

Simple Solution for dll was loaded but the call to DllRegisterServer failed with error code   0x80004005 

         Try these steps to get the dll file register succeeded  with DllRegisterServer.

     1. Boot the system in safe mode.

(START – Run – MSCONFIG –System Configuration –Boot Tab check Safe boot and  select Network under Boot options.  Apply  OK. Then restart the system)

    2. Log in as Administrator.

       (Select “Switch User” option and enter your admin user name & password or           Right click on "computer" select "manage" look for the Local Users and Groups section, on the users screen right click on the administrator account and select properties, uncheck the "account disabled" option.)

    3. Register the dll files.

       (Right click at the dll file- select Open with – choose default program-Browse for file “C:\Windows\System32\regsvr32.exe” click OK.  Also Check - Always use the selected program to open this kind of file.)


  Start –Run – MSCONFIG –Tools Tab – select Change UAC settings- click Launch- drag the selection bar to bottom –set to never notify – click OK – OK . Turn off UAC in Vista      Start –Run – MSCONFIG –Tools Tab – select Disable UAC - click Launch- click OK.) and run Command Prompt with administrator rights ( Start – Programs – Accessories-  right click on Command Prompt – select Run as Administrator)  also failed to register the custom dll files.
