'550 5.7.501 Service unavailable, Client host blocked using Spamhaus

Split from this thread.

We are also experiencing this issue, with varied users using different domain names

the issue is actually caused by Exchange placing the user's IP address in X-Originating-IP field of the message header.

Recipient's SMTP server uses spamhaus.org to look this IP up, and if found, causes the message to be rejected.

This is NOT acceptable, as several of our users reported this issue when using mobile/cellular broadband /MiFi and one user reported this on home broadband connection.

This was NOT a problem when using Microsoft Exchange on our own server, the X-Originating-IP field was not induced. Microsoft's own Exchange online includes this for some reason.

Obviously  users cannot know if the IP assigned to them by ISPs DHCP pool (public WiFi, 4G/LTE) was being used by spammers before, this field needs to be removed ASAP

The suggestion that the users need to contact Spamhouse to remove IP addresses from the black list - IP addresses that belong to the ISP's DHCP pool, and by virtue are completely random - is unbelievable

ken sheridan say: You don't say how your tables are related.  If we assume that each album has only one manager, producer, recording studio and genre, then the Albums table will need a foreign key column for each, ManagerID, ProducerID etc, referencing the primary key of the relevant table in each case.  In a form based on the Albums table these would be entered via combo boxes in which you would select the manager, producer etc from a list.  the control wizard can set this up for you.

Artists are rather different in that an album could be by more than one artist, so you have a many-to-many relationship type between artists and  albums.  A many-to-many relationship type is represented by an additional table which resolves the relationship type into two one-to-many relationship types, so the table would be like this:

AlbumArtists
....AlbumID
....ArtistID

Each of these is a foreign key referencing the primary keys of the Albums and Artists tables respectively.  Taken together the two foreign key columns make up the composite primary key of the table.  Data entry would be via a subform in the albums form, liked to it on AlbumID.

There is a one-to-many relationship type between Albums and Tracks, so the Tracks table would include an AlbumID foreign key column.  To enter tracks per album a subform within the albums form would again be used, based on the Tracks table and linked to the parent form on AlbumID.

An albums form with artists and tracks subforms like this can be used for entering, editing and deleting data.  You will need other forms for entering data into the Managers, Producers, Studios and Genres tables.  These can be opened directly from within the Albums form  by code in the NotInList event procedures of the relevant combo boxes when you need to add a new manager, producer etc where they are not already represented in the relevant tables.  You'll find examples in NotInList.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates the use of this procedure in various contexts.
