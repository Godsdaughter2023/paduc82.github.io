jay freedman say:

Yes, you can assign a command (actually, a Math AutoCorrect entry) to any symbol. Here are the steps:

  • Insert the symbol in an equation using the menu as you showed.
  • Select the symbol (or even a group of symbols, or a whole equation).
  • Click File > Options > Proofing > AutoCorrect Options to open the AutoCorrect dialog, and click the Math AutoCorrect tab of the dialog.
  • The selected symbol or symbols will be in the "With" box. Type the command you want to assign to it into the "Replace" box.
  • Click the Add button, and then OK the dialogs.

You can now use the assigned command like any other.

albert d. kallal say:

The reference for odbc drivers to JET of course ONLY refers to code and systems that are going to read data from a mdb or accDB (access database).

 

In other words, 64 bit code TO READ THE ACCESS database code needs a 64 bit data engine.

 

Your question is about connecting to sql server (very differrent issue). So, the issue of a 32, or 64 bit driver for getting at access data is likey not the question nor the issue.

 

SQL server (64) allows both 32, and 64 bit clients to connect. And, you can run 32 bit clients on a 64 bit OS client.

 

So, you now have "two" ODBC datasource programs, one for setting up ODBC 32 clients, and one for 64 bit clients. Note that both of these can connect to a 64 bit server.

 

Now, it possible I not grasped your question, and perhaps you are using some "importing" or automated code that runs on the server side, and runs from sql server. In this case, YES you will need 64 bit drivers since it is a 64 bit "in-process" (sql server) that is running + reading + launching this code.

 

However, from a client side, 32, or 64 bit odbc works just fine.

 

However if you're writing .net code, or some other process code that is 64 bits, and you need to read and open An access database, then yes that Redistribute package you mention will be required.

 

So when you say data feeders, is this VBA code running on the clients, or System Automation code that runs on the server side ?

 

If it's automation and process code that runs on the client side, then all of that can remain in the 32 bit world (even on a 64 bit os machine), And all of this code should be able to freely without trouble connects to a 64 bit version of SQL server running on a 64 bit OS server.

 

However if those data feeders code and programs are writing is in 64 bits code, then yes you'll need that 64 bit edition of the JET (now called Ace) driver.

 

It's important to know that you can't mix and match automation space either. So, a 32 bit edition of word it can't go out and automate and use a 64 bit version of excel, they have to be the same. However in the case of ODBC drivers, they are running on different machines, and they both can connect to the server.

 

Note also that 32 bit automation code of office suite programs on the server should continue to work.

kevinwaddle say:

Janiszeh,

I had the same problem and solved it using the Microsoft XML, v6.0 reference.

I created a function that takes a URL, a file name and a file path.

The URL is the site you are trying to query.

The file name and path is where you want to temporarily store the results of your query.

The function will save your results to an HTML file.

You can then query this file in Excel.

Here is the function...

Function qryForceBrowser(strURL As String, strOutputFile As String, strOutputFilePath As String) As String

    Dim url                             As String
    Dim strHTML                         As String
    Dim HttpObj                         As MSXML2.XMLHTTP60
    Dim strPath                         As String
    Dim fso                             As Object
    Dim oFile                           As Object
    Dim Fileout                         As Object
    'Requires reference to Microsoft XML, v6.0

    Set HttpObj = New MSXML2.XMLHTTP60
    HttpObj.Open "GET", strURL, False
    HttpObj.setRequestHeader "User-Agent", "Mozilla/5.0 (Windows NT 6.3; Trident/7.0; rv:11.0) like Gecko"
    HttpObj.Send

    While HttpObj.ReadyState <> 4
        DoEvents
    Wend
    strHTML = HttpObj.responseText
    qryForceBrowser = HttpObj.responseText
    Set fso = CreateObject("Scripting.FileSystemObject")
    Set Fileout = fso.CreateTextFile(strOutputFilePath & "\" & strOutputFile, True, True)
    Fileout.Write strHTML
    Fileout.Close
    Set fso = Nothing
    Set oFile = Nothing
   
End Function


The line

HttpObj.setRequestHeader "User-Agent", "Mozilla/5.0 (Windows NT 6.3; Trident/7.0; rv:11.0) like Gecko"

...is what tells your web site "Hey, Look at Me I am IE11"

Search Google for MSXML2.XMLHTTP60 and User Agent to find the string for other browsers...

To call this function...

Sub TestURL()
Dim myURL As String
Dim z As Variant

'Set MyURL to the site you want to query

myURL = https://www.somewebsite.com

'Call the function

z = qryForceBrowser(myURL, "TempFile.html", ActiveWorkbook.Path)

'Now set MyURL to the path and file name of your temp HTML file

myURL = ActiveWorkbook.Path & "\TempFile.html"

'Then query that file
t.QueryTables.Add(Connection:="URL;" & myURL, Destination:=DataSheet.Range("a1"))
            .BackgroundQuery = True
            .TablesOnlyFromHTML = False
            .Refresh BackgroundQuery:=False
            .SaveData = True
        End With

'Then get rid of the temp file

        Kill ActiveWorkbook.Path & "\TempFile.html"

End Sub

suzanne s. barnhill say:

Recent versions of Word complicate this issue. Historically, the location for all Word templates (including those created by users) has been the folder where the Normal template is located. By default, this is C:\Users\[UserName]\AppData\Roaming\Microsoft\Templates, as Doug said. Recent versions of Word, however, create a Custom Office Templates folder that is in the Documents folder (C:\Users\[UserName]\Documents\Custom Office Templates.

To further complicate matters, at File | Options | Save, the Custom Office Templates folder is the default for "Default personal templates location." But if you go to File | Options | Advanced: File Locations..., you'll see that the default location for "User templates" is the Templates folder Doug cited (that's where Normal.dotm is stored).

If you leave these settings as is, then any template you create and save to Custom Office Templates will show up on the Personal tab at File | New. In order to see templates saved in the C:\Users\[UserName]\AppData\Roaming\Microsoft\Templates folder, you will have to add and use the New Document or Template button as Doug described. If you set the "Default personal templates location" to the same folder as "User templates," then you will see all your templates both at File | New | Personal and in the New Document or Template dialog.
