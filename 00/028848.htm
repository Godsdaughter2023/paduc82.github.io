normagreen say:

I am using Access 2010

I have a form (enter customer information). I have a customer table, state table, city table, and zip code table.

On the form I have a state combo box and a city combo box. I want to select the state and from that selection I want the city combo box to be populated with all the cities from the state that was selected from the state combo box.

The first Combo Box, State, has a RowSource is the states table which lists all distinct states.
When a user selects a state, the City Combo Box it is filtered to only the distinct cities which reside within the selected state.

The second Combo Box, City, I tried RowSource is the query and I tired AfterUpdate is set to the query.

I have a query that the following code has been entered into it:

SELECT CityTbl.CityName, CityTbl.CityStateName, StatesTbl.StateName
FROM [StatesTbl].StateName
WHERE StateName = cboState;

When I select the state the city is not being populated

I will appreciate any help on this matter. It has been a very long time(Years) since I have written a database.


ken sheridan say: Requerying the city control in the state control's AfterUpdate event procedure and the form's Current event procedure is required.  In the former you should also assign Null to the city control's value to cater for a user selecting a state then a city, but changing their mind and selecting a different state.

Some people might argue that by having columns for both the state and city in the customers table you are introducing redundancy as, once you know the city you know the state (state is functionally determined by city in the jargon) , and this means the table is not correctly normalized and therefore open to inconsistent data being entered.  This is not in fact the case in fact as you are simply using 'natural' keys rather than 'surrogate' keys, but it does mean that the relationship between CityTbl and StatesTbl must be on two columns, CityName and StateName.  From this it follows that the primary key of CityTbl must be a composite one made up of the CityName and StateName columns, or these two columns must otherwise be uniquely indexed (in combination, not individually).  This then caters for more than one city of the same name in different states, and you can prevent inconsistent data being entered by enforcing referential integrity in the relationship between CityTbl and StatesTbl.

So, if you don't already have this enforced 2-column relationship, creating it will mean that your design is sound in terms of the relational model, albeit not what most developers would do.  The usual approach would be to have a numeric CityID column as the primary key of CityTbl, and only a CityID foreign key column in the customers table, a column for the state being redundant as CityID does in this case, being unique to each city, determine the state.

You'll find a demo of the latter approach, using the local administrative areas of parish, district and county in my neck of the woods, as ComboDemo2K.zip, in my public databases folder at:

http://cid-44cc60d7fea42912.skydrive.live.com/redir.aspx?resid=44CC60D7FEA42912!169

This still allows a 'top down' selection from correlated combo boxes, equivalent to your selecting a state than a city, but does so by the former being an unbound control.
