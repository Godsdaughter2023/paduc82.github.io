The strange syntax used within the playlist files [wpl] defy any consistent method of converting XP playlists into something usable in in Windows 10.

<media src="..\..\..\Audio\Music\Ludwig van Beethoven\Les adieux\Das Lebewohl.mp3"/>

The folders that are not shown can include

  • the folder path you have designated as your My music folder,
  • any of the other folder paths you have told WMP to monitor in addition to the My music path

but the wpl file gives no information about which one applies.

I gave up and rebuild my playlists.  Actually I rebuilt them in Windows 7 and they continue to work in Windows 10 because I have exactly the same file paths for everything, the same designated music folder, the same additional monitored folders, the same usernames - not everybody was able to get this far and had to rebuild them in Windows 10 as well.

I did try to play some old playlists in the hope that I could figure out from their error messages what I could do to edit the wpl files directly in Notepad.  But there was not enough consistency to allow me to do this - what worked in fixing one playlist did not work in another even though I could not identify any reason for them to be different.

So I really do recommend that you just accept the need to rebuild them.  If your XP is still working you could have each playlist open in Notepad just to help you along the way by reminding you what decisions you took about which files to include & in what order.

    nocturnecsh say:

    Why would Defrag be available on a Windows RT device if it would damage the drive? Seems counterintuitive to me. And don't say, "It's legacy in the Windows 8 build", because RT is written/compiled differently to run on ARM.

     

    Would love to know why. If it IS damaging, maybe 8.1 RT will remove it??

    hairyfool say:

    HDDs benefit from a defrag because if blocks of data are distributed around a drive it takes time for the head to step to a new cylinder and normally at least one rotation to read the required sector.

     

    With an SSD it makes no difference if the next block required is the next sector or at the other end of the memory, it is just load a sector address >read that sector. So there is no speed improvement by making the sectors contiguous.

     

    It is potentially damaging because flash memory has a nominal write count limit. I am not sure if there is one for a magnetic disk but if there is it is very high. A defrag involves rewriting data into a new location so using one of those writes for no purpose and potentially reducing the life overall of an SSD.

     

    I don't know why Defrag is in Windows RT, I don't think you can completely dismiss the "legacy" angle as there are some odd hangers on from x86 code throughout RT and I wonder if Microsoft saw other options for RT perhaps in embedded systems. It could be for USB devices of course as these could be HDDs although this article seems to recommend disabling Defrag on all drives in a system with an SSD.
