christopher miller say:

Hi Gyan,

Windows Phone and iOS really don't have or need task managers like what is provided with desktop Windows.  The operating system manages apps and flushes out apps that not being used.  Windows Phone 8 doesn't allow background apps, except for isolated cases like music players.  Apps can be periodically woken up by the OS, but it's not the same thing as a background process.

The Data Sense app will show cellular and Wi-Fi usage by each app, but it's not a real time monitor.

The Battery Saver app show relative CPU usage by app, but like Data Sense, it's not a real time monitor.

Having a close all apps button can actually use more battery power.  When an app isn't in the foreground, the OS suspends it and keeps it in executable memory.  If the foreground app needs more memory, then a suspended app will be flushed out of memory.  If you relaunch an app that you forced closed, it now has to spend the cycles reloading from storage.  You don't save anything forcing an app out of memory and you can draw more battery reloading the apps.

Windows Phone and iOS are designed for the model that you just use the device, you don't need to clean up after it.  If you want to be able to monitor everything like a desktop OS, then you want Android.

ken sheridan say:

I do not want to send all at once, because then a new excell will be created, and I want excell to be such an archive (base) so that once sent records can not be updated anymore.

You could do this through automation, but it would require some familiarity with the Excel object model.  Another way is to overwrite the Excel file each time with its current data plus the new row.  This might seem like overkill, but it is actually quite efficient.  Basically it works like this:

1.  A temporary link to the current Excel file is created in the database.

2.  All rows from the linked table, plus the new row, are exported to a new temporary Excel file.

3. The current Excel file is overwritten with the new temporary Excel file.

4. The temporary Excel file and the temporary link to the original Excel file are deleted.

The function below is an example of this where a new row entered into a Contacts table in a form in Access is added to an Excel file Contacts.xlxs.  For the function to work it requires two additions to the Access data base:

1.  A query named qryTemp.  This can be anything as its SQL statement is going to be replaced by the code.  In my case it was simply SELECT * FROM Contacts;  BTW this query will not subsequently open from the navigation pane in Access as the temporary linked table on which it draws only exists during the execution of the function.

2.  The following module which calls the Windows API CopyFile function to copy a file and replace the original, i.e. in effect overwrite the original file:

' module basMakeFileCopy
Option Compare Database
Option Explicit

Declare Function CopyFile& Lib "kernel32" Alias "CopyFileA" (ByVal _
lpExistingFilename As String, ByVal lbNewFileName As String, ByVal _
bFailIfExists As Long)

Public Sub MakeFileCopy(strExistingFile As String, _
                   strNewfile As String, _
                   blnDoNotOverWrite As Boolean, _
                   Optional blnShowMessage As Boolean = False)
                  
                  
   Dim strMessage As String
                  
   strExistingFile = strExistingFile
   strNewfile = strNewfile
  
   If CopyFile(strExistingFile, strNewfile, blnDoNotOverWrite) = 1 Then
       strMessage = "File successfully copied."
   Else
       strMessage = "File copy failed."
   End If
  
   If blnShowMessage Then
       MsgBox strMessage, vbInformation, "Copy File"
   End If
  
End Sub
' module ends

With these two things in place the function to add the new row to the Excel file is as follows.  You would need to adapt it to suit your own table and file names of course:

Public Function ExportRowToExcel(strPath As String)

    Dim strSQL As String
    Dim strPath_1 As String
   
    ' create link to Excel file
    DoCmd.TransferSpreadsheet acLink, , "ContactsXL", strPath, True
   
    ' build SQL statement to return all rows from Excel worksheet
    ' plus current record in frmContacts form
    strSQL = "SELECT * FROM ContactsXL WHERE ContactID IS NOT NULL " & _
        "UNION ALL SELECT * FROM Contacts " & _
        "WHERE ContactID = " & Forms("frmContacts").ContactID
    ' Assign SQL statement to qryTemp query
    CurrentDb.QueryDefs("qryTemp").SQL = strSQL
   
    'export result table of qryTemp to temporary ContactsXL.xlsx file
    strPath_1 = Left(strPath, InStrRev(strPath, "\")) & "ContactsXL.xlsx"
    DoCmd.TransferSpreadsheet acExport, , "qryTemp", strPath_1

    'delete link to Excel file
    strSQL = "DROP TABLE ContactsXL"
    CurrentDb.Execute strSQL, dbFailOnError
   
    ' overwrite original Contacts.xlsx file
    MakeFileCopy strPath_1, strPath, False
   
    ' delete temporary ContactsXL.xlsx file
    Kill strPath_1
   
End Function

In the frmContacts form's AfterInsert event procedure the function is called like this:

    ExportRowToExcel "F:\SomeFolder\SomeSubFolder\Contacts.xlsx"

PS:  It goes without saying that before trying this both the Access database and the Excel file must be backed up.

cztbg0 say: For what it's worth, I found out what the issue was. Lenovo Smart Update (located by default at C:\Program Files (x86)\Lenovo Smart Update) is the program that is causing issues with the mouse cursor in Microsoft Word. If you go to Start Menu > Run > msconfig > Startup, and uncheck Lenovo Smart Update so it does not start when your computer boots up, it will fix the issue. 

Alternatively, you can simply uninstall that program of course.

Thank you for the Guest account tip, I would not have discovered that interaction otherwise. Smart Update requires administrator privileges to run so it does not run on Guest accounts.
