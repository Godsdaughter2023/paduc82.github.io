graham mandeno say:

Hi Keith

I think the error message is pretty clear.  It says:

Data in the table 'Reservation' violates referential integrity rules.
For example, there may be records relating to an employee in the related table, but no record for the employee in the primary table.

In your case, you have three records in Reservation with GuestID values of 202, 205 and 206, but these GuestID records do not exist in the Guest table.

You can find these 'orphan' records with a query like this:

SELECT Reservation.*
FROM Guest RIGHT JOIN Reservation ON Guest.GuestID = Reservation.GuestID
WHERE Guest.GuestID Is Null;

Incidentally, it appears that all your primary keys (GuestID, ReservationID, PropertyID) are text fields and yet the values are all numeric.  I suggest you change the data type to numeric (long integer).  Of course you will also need to change the data types of the "foreign key" fields in the related tables.

If the values of these primary keys do not have any special significance then you could make them AutoNumbers, and thus remove the hassle of generating unique numbers for new records.

Good luck :-)

Formula in Word 2007 - Microsoft Community

sheryl millsap say:

Formula in Word 2007

Can a "fillable" formula be added to a Word 2007 document?  Or can an excel formula be added to a Word document?
doug robbins - mvp office apps & services (word) say:

If you can live with the document being protected for filling in forms, you could use the legacy Text FormFields to gather the date and also use a Calculation Type Text FormField to display the result of the calculation.

The advantage of this method is that for each of the input FormFields, you can set its Calculate on Exit attribute so that the result is automatically displayed in the result field.  The disadvantage however is that there are some restrictions placed on the use of other functionality of Word.

There is a quick and dirty way of doing it with Content Controls, but it has the disadvantage of not automatically updating the display of the result which would need to be done by updating the field in which the result is calculated.  The method makes use of the fact that if you insert any of the 15 Document Properties that can be inserted by clicking on Quick Parts in the Insert menu, it is inserted into the document as a Content Control.  You can change the title of the Content Control to something that is appropriate for the data that is to be entered and probably you will want to give it an initial value - maybe 0.  Then where you want the result of the calculation to be displayed, you can use an equation field { = } whose arguements are the Docproperty fields that you inserted from the Quick Parts utility

For example, the field construction { = { DOCPROPERTY Author } * { DOCPROPERTY Keywords } } will give the produce of the two numbers that are entered into (what were) the Author and Keywords Content Controls.

As mentioned, the disadvantage of this method is that the result is not automatically updated as the data is entered.

-- Hope this helps.

cindy meister say:

Hi Sheryl

Besides that which Doug proposes:

Click on the "selector" for each content control, in turn. Insert/Bookmark and assign a bookmark name.

Now Ctrl+F9 to create a field code. In the field code, type: = bookmark1 + bookmark2

Alt+F9 to hide the field code; F9 to update.

This has the same drawback as Doug's proposal in that the field code that executes the actual calculation will not update automatically. If the option to "update fields" is activated for printing, you can be sure this information will have updated latest at the time of printing.

Other possibilities present themselves if you could use macros...
