MS Access 2010 Runtime Ribbon - Disable File Tab Controls - Microsoft Community

danielhugos say:

MS Access 2010 Runtime Ribbon - Disable File Tab Controls

When I was trying to disable Access 2010 File tab controls (in both the runtime and full versions of Access) I encountered a lack of clear answers online. Thanks to Avelino Sampaio at Monta Ribbons for getting me out of a very frustrating situation. I must have looked at a hundred posts.

Maybe I was missing the obvious, but for those who also miss the obvious, or majored in something other than Computer Science or Engineering in college, or do not wish to simply be directed to yet another online article or post, here's what I did:

The object was to put together a custom ribbon with two tabs: File, and Mytab (ie, give it a name). I mainly wanted to disable the Privacy Options button on the File tab.

So in order to do that, I held the SHIFT key down when I was opening the database and chose Options > Customize Ribbon, and unchecked everything. I saw that there was the ability to make a custom ribbon the default ribbon - using XML seemed to be the easiest way to build it and using Access was the way to make it the default.

Yes, I understand that there are other approaches possible that do not require you to disable the controls, but I did not want to hide the entire ribbon with the Showtoolbar command, I did not want to disable the SHIFT key, nor did I want to write VBA code or macros. I just wanted to do it with XML.

I had built an XML table called USysRibbons with column names RibbonName and RibbonXML (id field is optional) which only needs to hold the one record in it, the one that would provide me with that application's custom ribbon. I input the XML associated with the custom ribbon. I closed the database, so that the XML and any subsequent changes would take effect (this is kinda lame it seems to me).

I reopened the database with the SHIFT key down and went into File > Customize Ribbon and checked off Add New Tab and then Add New Group. I named them the same thing. Then I made my new ribbon (give it a name) the default.

Microsoft apparently does not let you disable the File tab, so then the next best option is to disable the controls. But I could not find out clearly how to do that. 

I was getting the custom tab just fine, but the File tab controls were not disabled. There were always the Print, Privacy Options, and Quit buttons there and I could not find the problem - very annoying and time-consuming!

Avelino found a couple of errors that were causing the whole XML statement to break:

1. The structure <backstage> was in the wrong position.  It has to be below the </ ribbon>. No one told me that.

2. I also forgot to add the <tabs> tag. Oops, missed it.

3. Note: Enable the error detection in your Access: File button > Options > Client Settings > Check “Show add-in user interface errors”

Clear and concise. Tags in my XML code were in the right order now. Everything worked. When I opened my database, I got the File Tab with everything disabled, and my custom tab.

So I resolved to put something out there that would clarify this problem for others like me who are desperately searching for a solution to these rather straightforward questions, because I personally couldn't find anything. I hope this helps somebody else who doesn't write XML for a living.

Here's some simple example XML code for you - with these fixes, my XML looked like:

<customUI xmlns="http://schemas.microsoft.com/office/2009/07/customui">

<ribbon startFromScratch="true">

<tabs>

<tab id="tab1" label="My Custom Tab">

<group id="group1" label="Undo">

 <gallery idMso="Undo" size="large"/>

</group>

<group id="group8" label="Find">

<control idMso="FindDialog" size="large"/>

</group>

<group id="group2" label="Sort ASC">

<toggleButton idMso="SortUp" size="large"/>

</group>

<group id="group3" label="Sort DESC">

<toggleButton idMso="SortDown" size="large"/>

</group>

<group id="group4" label="First Record">

<control idMso="MailMergeGoToFirstRecord" size="large"/>

</group>

<group id="group5" label="Previous Record">

<control idMso="MailMergeGoToPreviousRecord" size="large"/>

</group>

<group id="group6" label="Next Record">

<control idMso="MailMergeGoToNextRecord" size="large"/>

</group>

<group id="group7" label="Last Record">

<control idMso="MailMergeGotToLastRecord" size="large"/>

</group>

<group id="group10" label="Print Preview">

<control idMso="FilePrintPreview" size="large" enabled="true"/>

</group>

<group id="group9" label="Print Preview Close">

<control idMso="PrintPreviewClose" size="large" enabled="true"/>

</group>

<group id="group12" label="Print">

<control idMso="PrintDialogAccess" size="large" enabled="true"/>

</group>

</tab>

</tabs>

</ribbon>

<backstage>

          <tab idMso="TabPrint" visible="false"/>

          <button idMso="FileExit" visible="false"/>

          <button idMso="ApplicationOptionsDialog" visible="false"/>

</backstage>

</customUI>

Another tidbit was how to get the names of the icons for the various controls that you'll want to use. Why is there not a list that's actually easy to use? I found this also to be very frustrating.

But I found out that you can get them (you will need them for your XML code) this way:

Just go File tab > Options > Customize Ribbon > All Commands, and then hover over the name of the control and you'll get the name. I forget where I read that but whoever posted that information saved me a lot of time.

Be aware that there are a couple of these icons that are "officially" misspelled or whose names are otherwise typos. Even Microsoft makes mistakes.

vijay a. verma say:
Use the below formula

=INDEX(A1:A5,MATCH(SMALL(B1:B5,1),B1:B5,0))

Change 1 to 2, 3, 4...for 2nd, 3rd...

A more robust construction would be

=IFERROR(INDEX(A1:A5,MATCH(SMALL(B1:B5,1),B1:B5,0)),"")

You can also use full column references

=IFERROR(INDEX(A:A,MATCH(SMALL(B:B,1),B:B,0)),"")

jamesyan_0708 say:

Hi Jason,

Below are the main differences between MDM for Office 365 and Microsoft Intune:

1. Cost

For MDM for Office 365, the cost is included in Office 365 commercial subscriptions (Business, Enterprise, EDU and Government),
while Microsoft Intune is a paid subscription (single $6 per user per month or with the Enterprise Mobility Suite $7 to $12 per month).

2. Device Management

MDM for Office 365 manages devices through the Office 365 admin center while Microsoft Intune manages devices through the Microsoft Intune Cloud console or the System Center Configuration Manager console.

3. Supported devices

MDM for Office 365 supports IOS, Android, Windows Phone while Microsoft Intune supports IOS, Android, Windows Phone and also Windows.

4. Capabilities

MDM for Office 365 is limited to the following: Conditional access, Device management, Selective wipe.
Microsoft Intune includes all of the MDM for Office 365 capabilities, plus the following: Advanced mobile device management, Mobile application management, PC management.
