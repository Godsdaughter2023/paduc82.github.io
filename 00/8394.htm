craigdkelly say:

How can I add a password protected button to a DB to allow access to design, tables, macros etc

ken sheridan say: Using the by-pass key doesn't enable you to open a file 'in design mode' per se.  What it does is by-pass whatever start-up options have been set.  There is not really any way you can intercept this and prompt for a password, as the mechanism set up to prompt for the password would itself be by-passed.  Catch 22!

The best option is to disable the by-pass key completely in all operational copies of the front end, so that holding down the shift key when opening the file does nothing, but provide yourself, and only yourself, with a mechanism for re-enabling the by-pass key when necessary.  Normally you should have no occasion for doing so, as all interface development work would be done in a copy of the front end to which only you have access (what Gina has called the 'Design Master').  This will have the by-pass key enabled, whereas all other users' copies of the operational front end will have the by-pass key disabled.

On occasions you might have to break into a user's front end.  I sometimes had to do so when a file had been distributed to a remote site, in order to debug some problem the user had encountered.  For this I used a separate file which I carried around with me on a portable medium.  You'll find this file in ByPass.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little file enables you to enable or disable the by-pass key by browsing to the file in question, so you can temporarily enable the file while you debug it, or whatever, and then disable it again once you are finished.

If you need to modify the design of the back end, you can password protect the back end separately, and include the password in the Connect property of the tables in the front end.  This will be done for you if you link the tables in the usual way via the interface.  Users will consequently never need to know this password, so it can be kept private to yourself and used by you only if you need to modify the design of the back end.

Bear in mind that Access provides very little in the way of true security, and whatever measures you take can be subverted relatively easily by anyone with the necessary skills.  The measures you take will give protection against inadvertent meddling in the day to day operation of the database, but little more than that.  As Tom said, distributing the front end as an ACCDE file is your best safeguard against inference with the design of the database.

hansv mvp say:
In a crosstab query, you MUST apply an aggregate function (Min, Max, First, Last, Sum, Average, Count, ...) to the Value field. You can't list the individual values.

qa guy commelec say:

There is a work-around but it entails using a temp table.  You number them using a ranking query.

TRANSFORM First(Data.[Date of Consent]) AS [FirstOfDate of Consent]
SELECT Data.Ckit
FROM Data
GROUP BY Data.Ckit, Data.IndividualID
PIVOT Data.Region;

Data.Ckit is the row data and Data.IndividualID is the numbered items.

Build a little, test a little.
