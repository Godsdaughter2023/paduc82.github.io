phillip m jones say:

It's a font conflict. It first showed up in the 2011 version of Excel.

To verify that it is a Font conflict.

  1. Save a copy of the document.
  2. Now open the copy.
  3. Hold down "Command" key (⌘)
  4. Type "A" Key. 
    (This is "Select All".)
  5. Now change to any other font listed.

If document shows up, then you have a Font Conflict. 

Open "Font Book" and validate fonts using the method I have on this Web site, titled, "FontBook Lion and above". 

Now you can either save the copy or discard. Then open the original and it show in all its glory.

The problem lies with the sender of the file. The 0kb size is the tip-off.

In Microsoft Windows, when Windows is set to show hidden files, they see the name of the file they are working on twice in Windows Explorer (the Windows version of Mac OS Finder). The first file name is a zero kb temporary file, and then there's the actual file name displayed further down the file list in the window. Your Windows friend attached the zero kb temp file instead of the actual word document file.

jay freedman say:

After examining a sample of the document, it seems that part or all of it came originally from a Word installation that has Chinese (PRC) language support. The additional grid setting available for East Asian languages increased the line spacing.

Initially I corrected the spacing by editing the document's XML to change the <w:docGrid> node. While that worked, it's an ugly method. Eventually I found the VBA command to do the same thing:

ActiveDocument.Range.Paragraphs.DisableLineHeightGrid = True
