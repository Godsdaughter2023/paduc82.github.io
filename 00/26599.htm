Hi,

What you're seeing is well documented and is caused by rounding errors when doing floating point arithmetic in Excel. Excel works in binary and often there is no precise binary equivalent of a floating point number. It's not specific to Excel and happens because Excel works to a standard called IEEE 754 which describes how binary computers should deal with floating point numbers, see the link below

In your case you should add up using ROUNDED numbers like this

=ROUND(82.21,2)

I followed your advise and also the instruction from the weblink  (Set precision as displayed) in Office 2010, however it still shows the the number with the 9th decimal as a 1. Is there a way to show the exact number without somewhere along the decimals the 1 is showing up ?

First, setting "Precision as displayed" is ill-advised.  And before you do that in a pre-existing Excel file, you should make (should have made) a backup copy of the file.  PAD can have unintended and sometimes irreversible consequences.

Second, PAD is effectively only if you also format the cell(s) with a format that specifies the number of decimal places.  It is less effective for cells formatted as General.

But PAD applies only to the final cell value.  It does not apply to expressions within a formula.  So, if you have a cell with the formula =SUM(A1:A1000) formatted as Number with 2 decimal places, PAD should remedy the problem.  However, PAD will not help if you have a formula of the form IF(SUM(A1:A1000) > 1000,....).

The advice to follow is to explicitly round the sum of the numbers, not the individual numbers.  If your original formula is =SUM(A1:A1000), change the formula to =ROUND(SUM(A1:A1000),2).

In general, if you expect a calculation to be accurate to n decimal places, explicitly round the calculation to that number of decimal places.

I am not Mike.  But since you responded to my posting, I presume that you are asking me.

The issue is: Excel uses 64-bit binary floating-point to represent numeric values internally.  Most decimal fractions cannot be represented exactly.  And the approximation of the same decimal fraction might vary depending on the magnitude of the integer part.

That is why, for example, IF(10.01 - 10 = 0.01, TRUE) returns FALSE(!).

It is difficult, if not impossible, to predict when the result of binary arithmetic will differ infinitesimally from the result that we might expect based on decimal arithmetic.  Sometimes, even the order of the binary arithmetic makes a difference.

In the following example, the same numbers summed in different orders have visibly different results in A1 and B1.

Sometimes we are not aware of the infinitesimal difference between binary and decimal arithmetic results because Excel formats only the first 15 significant digits.

In the following example, the MATCH formula in C1 demonstrates that the same numbers summed in different orders have different results in A1 and B1, even though A1 and B1 appear to be the same when formatted to display at least 15 significant digits.

Function Rank(vNumber As Variant, sTableName As String, sFieldName As String, Optional bAscending As Boolean = True, Optional sWhereCondition As String) As Variant
    'Version    1.4
    'Created    12/07/2010 22:10    Apollo67
    'Purpose    Emulates MS Excel's Rank() function, for use in queries
    '
    'Description:
    'Returns the rank of a number in a list of numbers. The rank of a number is its size relative to other values in a list.
    '(If you were to sort the list, the rank of the number would be its position.)
   
    'Inputs:
    'dNumber = The number whose rank you want find
    'sTableName = The current table name
    'sFieldName = The field of numbers to be evaluated
    'bAsceding = Optional ranking order - True for Asceding Else False
    'sWhereCondition = an optional criteria constraint to query records on
   
    On Error GoTo Err_Rank
   
    Dim rs As DAO.Recordset
    Dim db As Database
    Dim sSQL As String
    Dim sWhere As String
    Dim sOrderBy As String
    Dim vReturn As Variant
    Dim lRow As Long
    Dim dLastValue As Double
    Dim dCurrValue As Double
    Dim dNumber As Double
   
    vReturn = Null
   
    If (Not IsNull(vNumber)) Then
        dNumber = CDbl(vNumber)
   
        sSQL = "SELECT " & sFieldName & " FROM " & sTableName
        sOrderBy = " ORDER BY " & sFieldName & IIf(bAscending, "", " DESC")
        If (Len(sWhereCondition) > 0) Then sSQL = sSQL & " WHERE " & sWhereCondition
        sSQL = sSQL & sOrderBy
       
        Set db = CurrentDb
        Set rs = db.OpenRecordset(sSQL, dbOpenSnapshot)
       
        If (rs.RecordCount > 0) Then
            lRow = 0

                If (dNumber = dCurrValue) Then

                    Exit Do
                End If
                dLastValue = dCurrValue
                rs.MoveNext
            Loop
        End If
        rs.Close
    End If
   
Exit_Rank:
    Set rs = Nothing
    Set db = Nothing
    Rank = vReturn
    Exit Function

ken sheridan say: I don't know why your query is not returning any rows, but another thing to bear in mind is that a 'word' is a substring of the total string, but a substring of the total string is not necessarily a 'word'.  You can get round this to a degree by wrapping both the 'sought' string and the 'sought in' string in spaces in the expression, but even then it's not completely reliable as a 'word' may be terminated by a punctuation character, not a space (or in some languages, e.g. Spanish, preceded by a punctuation character).  The following function provides a more robust solution:

Public Function FindWord(varFindIn As Variant, varWord As Variant) As Boolean

   Const PUNCLIST = """' .,?!:;(){}[]/"
   Dim intPos As Integer
  
   FindWord = False
  
   If Not IsNull(varFindIn) And Not IsNull(varWord) Then
       intPos = InStr(varFindIn, varWord)
      
       ' loop until no instances of sought substring found
       Do While intPos > 0
           ' is it at start of string
           If intPos = 1 Then
               ' is it whole string?
               If Len(varFindIn) = Len(varWord) Then
                   FindWord = True
                   Exit Function
               ' is it followed by a space or punctuation mark?
               ElseIf InStr(PUNCLIST, Mid(varFindIn, intPos + Len(varWord), 1)) > 0 Then
                   FindWord = True
                   Exit Function
               End If
           Else
               ' is it precedeed by a space or punctuation mark?
               If InStr(PUNCLIST, Mid(varFindIn, intPos - 1, 1)) > 0 Then
                   ' is it at end of string or followed by a space or punctuation mark?
                   If InStr(PUNCLIST, Mid(varFindIn, intPos + Len(varWord), 1)) > 0 Then
                       FindWord = True
                       Exit Function
                   End If
               End If
           End If
          
           ' remove characters up to end of first instance
           ' of sought substring before looping
           varFindIn = Mid(varFindIn, intPos + 1)
           intPos = InStr(varFindIn, varWord)
       Loop
   End If

End Function

Although I've named the function FindWord it will also find a string of words, e.g. "Baker's Lane", in "21 Baker's Lane", and will find "Baker" in this as the apostrophe is in the PUNCLIST constant.  You can call it in your query like so:

WHERE FindWord([Expr1], [Forms]![frm_Import_Points_Manager]![frm_Points_Finder_Existing_Points].[Form]![txt_Point_Name])
