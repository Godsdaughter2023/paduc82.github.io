How to add multiple totals in a report - Microsoft Community

ellensilverberg-brennan say:

How to add multiple totals in a report

My report is based on a query.
Three of the fields are called
Number of contacts
Number of contacts 2
Number of contacts 3

How can I add the totals in order to see one number in the report?

Thanks for your help.

ken sheridan say: To get the total per row you can add a text box to the report's detail section, with a ControlSource property of:

=Nz([Number of contacts],0)+Nz([Number of contacts 2],0)+Nz([Number of contacts 3],0)

To get a grand total for the report add a text box to the report footer, with a ControlSource property of:

=Sum(Nz([Number of contacts],0)+Nz([Number of contacts 2],0)+Nz([Number of contacts 3],0))

However, when I see multiple columns with similar names like this it does set my normalization antennae twitching.

marshall barton say:

You can Get the total of the three fields by using a text box expression Like:

   =[Number of contacts] + [Number of contacts 2] + [Number of contacts 3]


But, if any records have a Null in any or those fields, use This for the first one above:

   =Nz([Number of contacts], 0) + Nz([Number of contacts 2], 0) + Nz([Number of contacts 3], 0)


Regardless of how you do the above, you can get the grand total of those three fields across all records using this expression in a Report Header or Footer text box:

  =Sum([Number of contacts]) + Sum([Number of contacts 2]) + Sum([Number of contacts 3])


Now the big BUT, you should not have field like num1, num2, num3, ....  Normally, that kind of "repeating fields" structure is a strong indicator that you need to put those values in a separate table with one record for each value.  Your posted question is based on just one of the many problems you will encounter by using that kind of badly designed table.

I agree with Marshall.

 

If they might do three different activities in one shift, it's at least logically possible that someday they might do four.

 

Your table structure only allows three. What will you do? Redesign your table, all your queries, all your forms, all your code? Put in a duplicate record? Put the fourth activity in an almost unsearchable memo field?

 

You have a perfectly classic one (person-shift) to many (activities) relationship. Model it as a one to many relationship! That's how databases are designed to work; it'll make your totals much easier (just use a simple Totals query); it'll be more flexible; it'll be easier to maintain. What's not to like?

ken sheridan say: You should certainly decompose the table into two related tables; it's simply one of the fundamental mechanisms of how a well-designed relational database works.  For a simple example of one-to-many relationship types you might care to take a look at Forms.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file has as its main table a Clients table, which is related to Events and Visits tables, each in a one-to-many relationship type.  The main purpose of this little demo is to illustrate three different ways of representing this in forms; (a) a single clients form with e events and visits shown in subforms on separate pages of a tab control, (b) separate 'linked' forms for the events and visits where the form is opened from the clients form in dialogue mode, forcing the user to close it before they can return to the main form, and (c) separate 'linked' forms for the events and visits where the events or visits form is opened in normal mode, but is synchronized with the main form, so if a user moves to a different client record in the main form while one or both of the linked forms is open, the latter will show the events or visits for the client in question.

The most common method of representing a one-to-many relationship type in forms is to use a form with a continuous subform within it.  The subform need not be on a separate page of a tab control as in my demo, however; it can be within the main form so that  all the data for the current client, or employee in your case, is visible.

Most of the time, the redirection occurs when the Exchange server you connect through instructs Outlook to move to the new address. The issue typically has to be fixed on the server itself and I'm not 100% sure you can do anything to prevent this on your end.

I've had the issue myself in the past and figured out a way around it by connecting directly to the sub-server that had my account.
