albert d. kallal say:

The issue of GOTO a control was fixed for the retail edition. I remember this as it was a bug that I actually filed. So, yes, goto does for for me.

 

So just keep mind that running the pre-release edition then you often find things are fixed in the retail version. And additional features such as returning vales from data macros was added. However, sometimes we see features removed.

 

For example the forms on-dirty event was removed just before RC (or if it in the RC version it is now gone in retail version I can't remember anymore). I was using the on-dirty event to set a global refresh variable that the "calling" form would thus know to do a re-query or refresh of data when that form on top was closed. A web refresh can be quite slow so one as a good idea should pay more attention to this issue for WEB applications as opposed to desktop applications . However there is still a forms dirty property that I can check. So I simply moved the code over to my code that closes the form and set the global refresh variable in that code. Again being flexible with my designs means I did not really miss the on dirty event at all.

 

I should mention that in most cases for long time products the RC edition is not that big of a deal and tends to be very close to the final retail. And features that been in for years (and code that also thus been tested for years) tend to work well much the same as previous versions. However in the case of Access Web we are talking about 100% of every feature you are using is brand new. So it is subject to MORE change from the beta and test and RC versions compared to the retail then average. This is not a huge deal here but I thought it a good idea to point out that not only will have some things will have changed in the retail edition but things will have changed more then average as compared to other products that already existed.

 

One last issue on the RC, you find that what you develop now with an pre-release version will not convert without issues and work in the final edition without problems. So this is just a heads up that what you build now will not seamless convert to retail.

 

Now, lets get back to the query problem. One really important tip here is to always check the usys Application log. I not really been thinking about your question out since placing a forms expression or even expressions with a tempVar is NOT legal in the WEB query builder. You should be seeing this error in the USYSApplication log. I am not sure why you are not. It is possible you not checking this log when testing/debugging code after you published? (just go file view applicaton Log). You can also set the nav pane to display system objects, but I find that clutters up the tables too much).

 

However if you not looking at this log after publishing then I am not at all sure how you been testing this code? (trial and error?). It is difficult (if not impossible) to figure out if something is wrong if you not checking the USYSApplicationLog after each publish attempt to guide and tell you about errors (this would be like driving a car with blindfold on, you not see any of your new errors). So as a another good tip do make sure you become good friends with the USYSApplicaionLog. Perhaps you are checking the log and that error is not showing up because you are using RC? If that error not showing up then you in a real tough situation since I can't imagine any other reasonable way to know if something not working and you just flying blind. I not sure how the "there are new application" errors is supposed to work, but when publishing since that table is synced, then it will state no new errors, but when testing something like above you best just check that table after every new publish.

 

Here is a screen shot and you can see it notes the error of a forms expression in the query builder is not allowed.

 

 

In many cases we can find a good workaround but in case of a combo box there just not a good work around. Note that for a query attached to a form you can just removed the forms!someform expression from the query builder and replace it with the new parameter feature. This is a new feature to replace the forms! reference. You also have the WHERE CONDTION. Once you add parameters to the query then you will see the parameter option in the code editor appear for OpenForm (it just like passing a parameter to a data macro). This works well for a form (or report). However the query parameter feature is NOT supported for a combo or list box.  So your hands are quite tied on this.  

 

About the best I can come up with is to build a query that joins to the combo box data source to another table with two columns. One column has user name (so this example can be multi-user). The 2<sup>nd</sup> column would have the filter value that you joined the query to. You can then set that value of the join by calling data macro and that will get you your filter. This works quite well. However this will only work before the form is loaded. I don't believe even re-query would force the combo box to re-load while the form is open. So at the end of the day you can't do cascading combo boxes.

 

Another work around that would be my choice is just simply launch a form that presents the records listed for selection. Place a small button on each row that when clicked on sets the id in a global variable and then closes the form and you have your id value back in the calling code.

 

While I am at this I noted in other questions of yours you attempting to use where clause and filters as strings. They are expressions ONLY. Also the were clause and those query run on the server side.

 

Here is a diagram that explain this

 

 

The above diagram also explains why you don't have record processing on your desktop code either. (UI macro). As the above shows a web browser on your desktop does not have a recordset or any type of data engine to process data inside the browser. Of course what is very cool is that since the code is running on your desktop then often your code can run without having to request or do anything from the web server. This makes for a very rich and windows like responding web form.

 

A really cool thing to try is bring up the expression builder in a form when you building the where clause. You see a near VBA like function set. Now, try editing a expression just one or two lines up, and your see that the expression builder only has about 12 functions. The reason for this is that code that runs in the web browser has VERY limited functions. So, there no Hour() and very limited or where clause is a expression and is not a string.

 

Also for most where clauses etc. it is better to NOT use the forms!someFormName reference. Better to set a local var since then if you re-name the form or even use the new navigation control then the form becomes a sub-form and your code will have to be changed. In fact by adopting this coding standard then you don't miss the me! Property that we had in client forms. There also the case with un-bound text boxes used for parameters that the data type of the expression is often messed up (it not clear).

 

Anyway the above diagram should help you with some of your code issues.

bob jones aka: cybertaz say:
Placement of Footnotes is dependent on pagination of the document which is almost a constant process while a document is being revised. That's why you see the correct adjustment when you print or reopen the file. Either process forces re-pagination based on the current status. Switching to Draft view then back to Print Layout view also will cause pagination to occur. Although what you're describing isn't 'typical' it isn't unusual either. If you're using Track Changes [or have been using it] with unresolved changes existing in the document it will exacerbate the behavior.
