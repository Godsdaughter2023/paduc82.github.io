fin_modler say:

The difference between "Beg Dec 14" and "9-Dec-14" could be any number between -1 and 8. It all depends on how you want the calculation done.

Are you expecting to see this range ("minus 1 to plus 8") in your answer or are you expecting "-1" or "8" or are you looking for the difference against the mid-point (ie 4 or 5 Dec) of the "Beg" range. Or are you seeking something else?

Excel can do any of the above (and others) but you need to stipulate.

As an illustration, say if you want to use the first day of each range 1 for Beg, 11 for Mid and 21 for End, and your column A has Expected Date, column B has Actual Date and column C has the difference,

First put a lookup table for the months in L2:M13

Jan 1
Feb 2
Mar 3
Apr 4
May 5
Jun 6
Jul 7
Aug 8
Sep 9
Oct 10
Nov 11
Dec 12

Row 2 is your first row of dates (A2 Expected, B2 Actual)

C2 =IFERROR(B2-IF(ISNUMBER(A2),A2,DATE(RIGHT(A2,4),INDEX($M$2:$M$13,MATCH(MID(A2,5,3),$L$2:$L$13,0)),IF(LEFT(A2,3)="Beg",1,IF(LEFT(A2,3)="Mid",11,IF(LEFT(A2,3)="End",21,""))))),"Bad Expected Date")

or if it a case of seeing if the actual date was within or after the expected period:

C2 =IFERROR(B2-IF(ISNUMBER(A2),A2,DATE(RIGHT(A2,4),INDEX($M$2:$M$13,MATCH(MID(A2,5,3),$L$2:$L$13,0)),IF(LEFT(A2,3)="Beg",10,IF(LEFT(A2,3)="Mid",20,IF(LEFT(A2,3)="End",DAY(EOMONTH(DATE(RIGHT(A2,4),INDEX($M$2:$M$13,MATCH(MID(A2,5,3),$L$2:$L$13,0)),1),0)),""))))),"Bad Expected Date")

The message can't be sent right now. Please try again later. - Microsoft Community

andries bos say:

The message can't be sent right now. Please try again later. Problem resolved by code when using OLE automation and OWA

Searching for a solution for the annoying and actually mostly undefined/ unexplained error 

The message can't be sent right now. Please try again later The message can't be sent right now. Please try again later

show in OWA ((Outlook Web Access)) , I searched nearly three weeks to solve this mistery. All OLE automated created emails (created by thir party program) save to the draft within an outlook client, could not be sent until you manually re-enter the email address already present. That's not an option ;(

As all of these question threads within the community has been closed, it is not possible reply to these unsolved threads.  I hope this post will help other users or programmers within the future.

This problem **** when an email has been created and save to an office 365 account that has been added to a local outlook 201x client installed on a pc :

If an email has been entered and saved manually in outlook, it works fine and the email could be sent using OWA . If not try using local cache or clear your local cache first (see other threads).

The problem **** for all emails that has been added using the client outlook via OLE Automation ( incl third party software ).

The cause for the problem is that email adresses that has been added to the email created (by OLE automation) , has never been 'resolved' by outlook. Within outlook normally all manually added emails will be resolved (only) if these options has been checked.

Unchecking these options generates the same problem.

For some strage reason OWA will not sent this saved email and will not give any explaination on why this happens ! the message "The message can't be sent right now. Please try again later" does not help at all.

The solution for me is to alter the program code that created the email via OLE automation : you have to explicit push a resolve event after adding the emailadresss to the mailitem recipients list ! 

e.g. 

aRecipient = Mailitem.recipients.Add("*** Email address is removed for privacy ***")
aRecipient.Resolve

This will trigger the outlook event that results in the recipients table in a iMessage object property tag PR_SMTP_ADRESS_W . Creating this tag in this way will solve your problem and the email could be created in a local client and be sent without any problem using your online OWA (Outlook Web Access)

Tips to solve/ debug  : Use the freeware program Outlookspy to compare emails that could and cann't be sent to detect differences
