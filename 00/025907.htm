karl timmermans [outlook mvp 2012-2018] say:

To get some of the simple things out of the way


#1 - Your information regarding the email accounts and associated data files is maintained in the Outlook profile which persists any uninstall/re-install which is why it would be looking for the same data files. To put this another way, whatever folders were created in those PST/OST files would remain. The way to get to the equivalent of a fresh install of Outlook would be to create a new profile.


#2 - Not clear what state your machine is in (more specifically, which version of Outlook you have running at the moment or what all you have done to the registry) so with that said, assuming you have only one version of office installed, use the Control Panel Mail app to see what data files Outlook is looking for in the current profile. When 2 versions of Office are installed ('2013 and anything earlier - there would be two Mail apps showing in the control panel - one for '2013 and one for the older version. When '2013 is first installed, it migrates the profiles from the earlier version.


#3 - Re: the strange behavior, don't have a definitive answer but does sound like something that could be caused by a 3rd party addin so that would certainly be something to check (i.e Anti-virus, Anti-Spam etc) and remove any to see if that resolves the issue


#4 - To get to your ultimate question - how to get things going


- With item #3 in mind, create a new Outlook profile


How to create and configure an email profile in Outlook 2010, Outlook 2007, and Outlook 2003
http://support.microsoft.com/kb/829918


How to create a new profile in Microsoft Outlook
http://www.slipstick.com/tutorials/profile/profile.htm


- When creating the new profile, you have the option to use an existing PST file (if this involves a POP account) or creating a new one. To get a clean PST file, let Outlook create a new one but before writing off all your contacts and old email, try and find the PST file that contained the old data and open that file in Outlook (File --> Open Data File .....) and copy the items you want from the respective folders to your new PST file. Once done, right-click on the root of the old PST file --> Close.


If the strange behavior continues after creating a new profile, then the issue is most certainly something related to a 3rd party program that is still in play. You can check for this by running Outlook in Safe mode and see if the issue still occurs


How to start Outlook in safe mode
http://www.outlook-tips.net/outlook-2010/start-outlook-in-safe-mode/

locutus deborg say: 9879 is abusing the PUIS setting,
PUIS = Power Up In Standby,
actually it's a bad name because to me I read that as disk remains powered up in Standby, Sleep or Suspend mode, but what it acutally means is the drive will not spin until a specific ATA / SCSI command is sent to complete the Power Up of the drive. 

The original implementation was a SCSI only feature called Staged or Staggered Spin up
and the intent of Staged / Staggered Spin up, and PUIS is for multi-disk systems where simultaneously powering up several HDDs at once might cause a PSU to drop the load or PSU failure due to too much current draw.

This has been covered in at least 2 other threads:
- build 9879 bricking drives!!!!
- Build 9879 hard drive disaster (read this)

PUIS should only be used on systems with an Enterprise Class SAS / SATA / SCSI HBA controller and not on consumer grade systems

young yang msft support say:

Hi LuRe,

Thanks for the reply.

Suppose an AD user already has an on-premise mailbox, and this user is synced to Office 365. DirSync service can detect that this user has on-premise Exchange attributes, so, DirSync service knows that this user already has an on-premise mailbox. Therefore, Exchange Online mailbox cannot be provisioned for this user.

If an AD user has no on-premise mailbox, it is just a normal AD user, after DirSync syncs this user to Office 365, we can create online mailbox for this user.

DirSync only syncs user objects to Office 365, and it does not sync mailbox or mailbox data. On-premise mailboxes (and mailbox data: email message, calendar, etc) need to be migrated to Office 365, via Office 365 migration tool.  

One user cannot see emails or access mailbox on both locations (on-premise and cloud).

ken sheridan say:
Yes, the users would be on the same LAN. Would this work even if the users are updating at the same time or will it lock some users out?

You have to decide on which is the appropriate locking strategy for your operational requirements.   It really comes down to a choice between 'optimistic locking' and 'pessimistic locking'.  These are set by means of the RecordLocks property of your data entry form.  It goes without saying that data must only be entered/edited via a form, never in a table or query's raw datasheet view.

1.  Optimistic locking: This involves setting the form's RecordLocks property to 'No locks'.  If a user is editing a record, another user will be able to edit the same record simultaneously.  If User A starts to edit a record, and User B edits and saves the same record before User A saves their edits, User A will be informed when they attempt to save their edits that another user has changed the record since they started editing it.  User A will be given the opportunity to abandon their edits or overwrite User B's edits.

2.  Pessimistic locking: This involves setting the form's RecordLocks property to 'Edited Record'  If User A begins to edit a record all other users will not be able to edit that record until User A has saved their edits.  Until then all other users opening the record in the form will see a 'no entry' symbol in the record selector on the form.  All editable controls will be locked.  In the past pessimistic locking was unpopular because Access did not support true record locking, only page locking, so it was possible for records other than that being currently edited to be locked unnecessarily.  For a long time now Access has supported support true record locking, however, so pessimistic locking is feasible as users will still be able to edit any other record.

Pessimistic locking does impose a modest additional burden on the locking engine, so if you don't anticipate two or more users editing one record simultaneously optimistic locking would be appropriate.  In my own line of work, however, I've nearly always adopted a pessimistic locking strategy as, wearing my managerial hat, I would not want staff wasting time and money editing a record only to find that another user has simultaneously edited the same record.  At the end of the day it's your judgement call which to use.
