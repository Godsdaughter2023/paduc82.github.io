When is Optimization Not Optimization? - Microsoft Community

cmalcheski say:

When is Optimization Not Optimization?

I have been reading a lot of discussion all over the web for some time that indicates an overwhelming consensus that many parts of Windows are "optimized."  Trust me - it is not the case.  Because a) people just get lazy, and b) what today's universities are churning out is a dumbed-down "I know how to use macros" level of "developer" who knows little about what they are working with but carries that almighty degree so they are presumed to know everything.

Citing DirectX as the most glaring example, because optimization is so critical in that realm, the inefficiencies and mistakes I see are staggering.  It was written somewhere that the calling convention for 64-bit functions was designed around "wanting to clean up" the old 32-bit method.  What resulted was a nightmare of waste and bloat that helps nobody and drags down the system to an amazing degree.  The first four parameters are passed in r8, r9, rcx, and rdx.  But you still have to create space for them on the stack, even though most (all?) times they don't actually need to be placed there; just reserve the space.  WHY???  Then subsequent parameters go to [esp+20h], etc.  Then the entire stack has to be 16-byte aligned for each call - necessary for later calls to functions that might use SSE, but won't actually care about that stack alignment since each function realigns its local variable usage anyway .  "Inline" DirectXMath library functions are even worse.  Undocumented parameters are rampant, XMM registers are used for parameter passing and obey few to none of the rules for register preservation, and the entire resulting mess is like a circus of incompetence that reflects a disturbing lack of understanding about what is going on at the hardware level.  Intel requires that XMM registers must access memory on even 16-byte boundaries; movdqu can override this but it's much slower to execute - so in these special cases, the stack alignment MIGHT make some sense except that parameters are always passed in the XMM registers.  The DirectXMath library is a nightmare of inefficiency, hitting memory with reckless abandon when typically 20x faster register usage could be employed instead at no cost, all indicating the authors just did not have any clue about what optimization really means.  The ten matrix functions I hand-coded in ASM out of the DirectXMath library ran between eight and one hundred six times faster than their XMMatrix___ counterparts because they got rid of the excessive, sloppy memory hits all over the function and did all their work in registers.  Half the XMMatrixMultiply function was removed and SIMD capabilities were easily employed to cut out most of the execution time of the DirectXMath version.  What I saw in the DirectXMath library is an amateur hack at best that the average devoted high school freshman would have no problem increasing the execution speed of many times over.

The situation is not just bad ... it is catastrophic.  Left and right are insane implementations like moving a value from memory to a register then, on the very next instruction, moving that same value from the same register to the same location in memory it just came from.  Filling memory that should not be used in the first place with 0xCC - micro-managing the developer and ensuring no uninitialized pointers are used - in cases where every last byte of the "fill zone" is guaranteed to be overwritten by the code ahead of it.  Stupid, stupid, inept, incompetent things like this are out of control in the DirectXMath library, and one can only assume that this would be the best case scenario. 

IMHO this is why Aero was removed: dumbing down the OS to the lowest common phone (iPaci) denominator created a slew of room for covering up this lack of programming knowledge Windows-wide.

Don't assume anything is "optimized."  The deeper you go into the code of any API, the more horrific it becomes.
