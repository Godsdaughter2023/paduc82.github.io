sba923 say:

Closing (Reader) app doesn't release file locks, need to kill app in Task Manager

This issue was already present in Windows 8.1 but I'm reporting it here so that it gets fixed in Windows 10.

Take the following scenario: open a (local) PDF file in Reader, then close reader e.g. using the X button or Alt+F4.

If you attempt to delete the file, File Explorer reports a "file in use" error. If you open Task Manager, you'll notice the Reader process is still running (though the app isn't visible among running apps, e.g. in the taskbar). Once you kill Reader, you can  delete the file.

It's OK to keep the process running so that, maybe, reopening the app is faster, but the app in such an "invisible state" must close all open files!

chrispeng_msft6626 say:

Hi Ramiz,

Quick Parts are stored in the Normalemail.dotm file in the following location:
C:\Users\%username%\AppData\Roaming\Microsoft\Templates

Based on my experience, if the Normalemail.dotm file is damaged or corrupted, the Quick Parts saved before will disappear and we cannot restore them. Such issues usually occur when the contents are not properly saved in the NoralEmail.dotm if we turn off the operating system before Outlook shuts down properly. Moreover, some add-ins which hold on to Outlook.exe may cause the issue.

So, for the best practice, we suggest you close the application completely and make sure the Outlook.exe is not in the Processes tab in Task Manager before shutting down the OS. You could also back up the Normalemail.dotm file regularly to prevent Quick Parts from disappearing.

In addition, our forum focuses on Office 365 for Business Online services. As the Quick Parts is a feature in the Outlook client, if you need any further assistance regarding it, you can post to the Outlook client forum which a dedicated support channel for client related issues.
