brian tillman [outlook mvp 2007-2019] say:

Here is the answer: you CANNOT turn off all junk mail filtering with Outlook.com.  There are only two levels: exclusive and standard.  The first will put any messages you receive into your Junk folder if the sender address is not either in your Contacts or in your Safe Sender list or the sending mailing list is not in your Safe Mailing List list.  The second will allow more messages to be delivered to your Inbox, placing in Junk only those messages that trigger the Junk filter (and no one can tell you what those triggers are).

 

What evidence do you have that your messages are being intercepted and that Outlook.com's servers are doing the interception.  If they're being delivered to your Junk folder, then add the sending address to your Contacts or your Safe Sender list.  If they're not being delivered at all, then there's no telling what is preventing the delivery.  Any mail router between the sender's server and the Outlook.com receiving server could have a hand in it.  The only way you can know for certain that Outlook.com is actually preventing the delivery is if your sender receives a message from the Outlook.com server telling the sender that Outlook.com has blocked the message.  If the sender received such a report, then it will contain the steps the sender must do to have their mail server or their server's domain removed from the block.

ken sheridan say: Why are you specifically inserting the AccountID value into the subform's underlying recordset?  By setting the LinkMasterFields and LinkChildFields properties of the subform control to AccountID the value of the AccountID column in the parent form's underlying recordset will be automatically inserted into the AccountID column in the subform when a new record is added via the subform.  However, the process of creating a new record will not be initiated until you insert other data.  By specifically inserting the value as you are doing, this does initiate the creation of a new record, so when, having changed your mind, you attempt to navigate away from the record an attempt is made to save it, but this fails because all the required data has not been inserted.  If you try to delete the new record before its saved, then there is of course an inherent contradiction in this, as you are trying to delete something which doesn't yet exist. You'd need to Undo the record instead, so the code in your Delete button should include error handling which undoes the record if it is not yet saved and can't be deleted.

Simply allowing the linking mechanism to insert the AccountID value should solve the problem.  In circumstances where you do need to insert a value, e.g. when adding a related record in a separate linked form rather than an embedded subform, then what you should do is assign the value as the DefaultValue property of the relevant control.  Unlike assigning the value as the Value property (which is the default) this does not initiate the creation of a new record, so you can back out gracefully.  If in your case you were adding related records into a separate form rather than a subform you'd open the separate form at a new record and then assign the current AccountID as the DefaultValue property of the AccountID control in the second form.  You can do all of this with a button on the first form, e.g.

DoCmd.OpenForm "YourSecondForm", DataMode:=acFormAdd
Forms("YourSecondForm").AccountID.DefaultValue = """" & Me.AcountID & """"

Note that the DefaultValue property is always a string expression regardless of the data type of the underlying column, which is why it's value is wrapped in quotes character in the above.

Often, however, a better way is to open the second form in dialogue mode and pass the value to it via the OpenArgs mechanism as this forces the user to close (or at least hide) it before returning to the first form, for which the button's code would be:

DoCmd.OpenForm "YourSecondForm", DataMode:=acFormAdd, WindowMode:=acDialog, OpenArgs:=(Me.AccountID)

You then set the DefaultValue property to the value passed to it in the second form's Open event procedure with:

If Not IsNull(Me.OpenArgs) Then
    Me.AccountID.DefaultValue = """" & Me.OpenArgs & """"
End If

kimc2 say: Thank you so much!  I have been struggling with this for a while.  Now, how do I handle the case where the user clicks the button for a new record and the begins to enter data and then deletes the data from the form manually instead of using the undo or delete buttons?  I am just trying to allow for all possibilities. 

ken sheridan say: Try the following for the delete record button's Click event procedure;

    Const DELETE_CANCELLED = 2501
   
    On Error GoTo Err_Handler

    If Me.NewRecord Then
        If Me.Dirty Then
            Me.Undo
        End If
    Else
        RunCommand acCmdDeleteRecord
    End If
   
Exit_Here:
    Exit Sub
   
Err_Handler:
    Select Case Err.Number
        Case DELETE_CANCELLED
        ' anticipated error so do nothing
        Case Else
        ' unknown error so inform user
        MsgBox Err.Description, vbExclamation, "Error"
    End Select
   
    Resume Exit_Here
