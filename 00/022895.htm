scottgem say:

Relations are created between key fields in tables. The most common relationship is 1:many where a primary key in a parent table (the 1 side) is related to a foreign key in the many side. 

While you can have composite keys, they are more cumbersome to work with. Generally using a surrogate key like an autonumber. So what happens is the Foreign key becomes a meaningless numeric value that the user never sees. 

For example: You have an Order database. Customers make orders so there is a foreign key in the Orders table for the CustomerID. Most users won't know customers by their IDs so when you populate the FK in the Orders table, you use a combobox. Such a combo will store the CustomerID, but the user sees a list of customer names and chooses from that. 

Microsoft has often been guilty of not following best practices in tutorials and templates. The whole lookup fields issue is an example. 

ken sheridan say:

Thank you for your advice.  I am not familiar with combo box but I truly believe what you said is what I want to do with my database.  If you can elaborate the last part of your comment, that will be great.  If not, I will first look up combo box and see what I can find/understand.  Again, thank you for your time.

Let's say you have a table Employees, which includes columns EmployeeID, FirstName and LastName, of which EmployeeID is an autonumber primary key.  You also have a table ProjectEmployees which includes columns EmployeeID and ProjectID.  In this table EmployeeID and ProjectID are foreign keys, the former referencing the primary key of Employees, the latter referencing the primary key of a Projects table.  Both of these foreign key columns are straightforward long integer number data types, not autonumbers.

If we assume you have a form bound to the ProjectEmployees table in which you wish to select employees for projects, the form would includes combo box bound to the EmployeeID column, set up as follows:

ControlSource:   EmployeeID

RowSource:     SELECT EmployeeID, FirstName & " " & LastName FROM Employees ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.  The combo box's drop down will list all employees by name, but when you select one the value of the control, and hence of the column to which it is bound, will be the value of the EmployeeID in the hidden first column.

A form set up as above would usually be embedded in a projects form as a subform, the subform being linked to the parent form on ProjectID.  There would be no need to include a combo box bound to the ProjectID foreign key column as its value is automatically set to the parent form's current ProjectID value by the linking mechanism.
 

With reports, to list all projects and the employees assigned to them for instance, you would base the report on a query which joins all three tables:

SELECT Projects.Project,
Employees.FirstName & " " & Employees.Lastname AS FullName
FROM Projects INNER JOIN (Employees INNER JOIN ProjectEmployees
ON Employees.EmployeeID=ProjectEmployees.EmployeeID)
ON Projects.ProjectID=ProjectEmployees.ProjectID;

In the report you would bind text box controls to the Project and FullName columns.  The latter is a computed column in the query.  This is similar to a 'calculated' column in a table, but in this case the first and last name values are concatenated on the fly by the query, rather than in the base table.

A few supplementary points:

1.  Notice how in the above no table and column names include spaces or other special characters.  This is another area where the Microsoft templates and tutorials fall below best practice.  While spaces or special characters can be used in object names, it is strongly advised that you do not do so.  Instead use CamelCase or represent a space with an underscore character, e.g. Last_Name.

2.  In the ProjectEmployees table above, the EmployeeID and ProjectID columns represent a composite key of the table, assuming an employee can only be assigned once to a project.  Even if you were to add a separate autonumber primary key, the two foreign key columns are still a candidate key, and as such must be included in a unique index to protect the integrity of the data.  In a case like the above there is no advantage in adding a separate 'surrogate' key; it merely adds an additional overhead as another index has to be maintained by the database engine.  It is simpler to make the two foreign key columns a composite primary key.  There can be advantages in some similar situations in adding a surrogate key, but only if the table is itself a referenced table in a one-to-many relationship type, as it then avoids the need for a composite foreign key in any referencing table(s).

jllatham say:

This would be possible in VBA - but some rules need to be known about what to shade green and what to shade red.  Since you want to base things on whether a cell in column E has "Solved" or not, then it would be a Worksheet_Change() event most likely that 'monitors' column E entries -- but big question: how does Solved get into that cell?  User types it in, chooses it from a list, or does some IF() formula display it?

First, I wouldn't worry about whether the sheet is protected or not - if this is a cell that user's can type entries into, then make it UNLOCKED so that they can use it without having to unprotect the sheet.  Even if it is a cell with a data validation dropdown list in it, it has to be unlocked on a protected sheet to allow the user to make new selections.

But again, the heart of the matter is how to decide what to make green and what to make red in the cell.  You have to give us some clue as to what might show up in the cell and how to deal with it.

It could get really complicated if you want to make entries by Tom red and entries by Mary green.

If you want to see a little of what the code has to look like:

Type some text into a cell. 

Then start recording a macro - select just one or two words in the cell and change the color of the font for those words, repeat with other word(s) and different color. 

Stop recording.  Look at the code to see the type of thing that has to happen.  When it's stripped down to the essentials, it's going to look something like this:

Sub MultiColoredFontInCell()
'
' MultiColoredFontInCell Macro
'

'
    Range("C9").Select
    ActiveCell.FormulaR1C1 = "This is test text to work with."
    'text is all black at the moment, now
    'changing test text to red font
    With ActiveCell.Characters(Start:=8, Length:=10).Font
        .Color = -16776961
        .TintAndShade = 0
        .ThemeFont = xlThemeFontMinor
    End With
    'now changing work with to green
    With ActiveCell.Characters(Start:=22, Length:=9).Font
        .Color = -11489280
        .TintAndShade = 0
        .ThemeFont = xlThemeFontMinor
    End With
    Range("C10").Select

So as you can see, you need to be able to determine the starting character in the cell and how many characters behind it to set to a color within the cell.  That could be based on a word, or the location of spaces in the cell or other things, but there has to be some guidelines/rules to build the code around.
