In a Hybrid environment, the Autodiscover record points to the Hybrid Exchange server in general. Hello Jason,

By default, Office applications automatically save temporary backup copies of your documents as you work and there’s a good chance you can recover them. The recovery feature has been around for a long time, so if you’re using an older version of Office (going back to at least Office 2007), you’ll still be able to attempt recovery. You might just have to hunt around a bit for the actual commands. If you do run into a problem with an unsaved Office file, here are some methods on how to recover it:

Method 1:

  1. Launch Office application you were working in where your file didn’t get saved.
  2. From there, you may see the Document Recovery Task Pane. Simply click the file that was unsaved to open. Then you will have the option to recover and save it.

Method 2:

  1. Launch PowerPoint, then click the File menu.
  2. Click Options and then the PowerPoint Options dialog box will appear.
  3. Click Save at the side menu, then locate the AutoRecover file location box.
  4. Copy the AutoRecover file location folder path.
  5. Close the PowerPoint Options dialog box.
  6. Now, access the backstage view. Click File again, select the Open option, click This PC, and then click on Browse button.
  7. This brings up the Open dialog box. At the address bar, paste the previously copied AutoRecover file location folder path.
  8. Within the Open dialog box, you can see the names of all presentations that were open in the last session of PowerPoint, but as you can notice, there are folders instead of PowerPoint files. Just double-click on the folder that shows the name of the presentation you want to recover.
  9. This will show all the recoverable auto-saved (and also unsaved) versions of the selected presentation. Select the file and click the Open button to recover it.
  10. This will open the selected presentation in PowerPoint. You will see a strip above the Slide Area to indicate the fact that this is a recovered presentation. Just click the Restore button to save it as original file. A window will show confirming that you are overwriting the last saved version. Click the OK button in the window to accept.

Method 3:

  1. Open Office application you were working in where your file didn’t get saved. Click the File menu.
  2. On the File menu, click Info.
  3. On the Info page, click “Manage Document”. From the drop-down menu, choose “Recover Unsaved Documents”.
  4. The Unsaved Files folder contains all unsaved files Office has created temporary backups for. Select the file you need and then click Open.

Let us know if these methods resolve the issue.

Private Sub Worksheet_Change(ByVal Target As Range)

    If Not Intersect(Target, Range("B11")) Is Nothing Then     'If not nothing then is something

ken sheridan say: A relational database is radically different from a spreadsheet in concept, structure and most importantly in the formal principles which determine how the latter should be defined.  A relational database is a formal model of that part of the real world with which the database is concerned.  In a relational database:

1.  Each entity type is modelled by a table.
2.  Each attribute of an entity type which is modelled by a column which is functionally determined solely by the whole of the key of the table.
3.  All data, without exception, are represented by explicit values at column positions in rows in table and in no other way.
4.  Relationships between entity types are modelled in the case of a one-to-many or more rarely one-to-one relationship type (unary relationship type) by the primary key of the referenced  table being referenced by a foreign key of a referencing table; in the case of a many-to-many relationship type (binary, ternary etc relationship type) by a table which resoles the relationship type into two or more one-to-many relationship types.

From your original post it follows that you have two entity types, one of which is suppliers, the other of which is undefined (jobs??), but will be represented by a table on which the parent form in which the subform is embedded is based.  The many-to-many relationship type between these entity types is modelled by the table on which the subform is based.  The attributes of this entity type are the foreign keys which reference the primary keys of the referenced tables, suppliers and the undefined other table which is party to the many-to-many relationship type with suppliers, along with other attributes which are determined by the key of this table.  Because 'each material in the row will have a different cost associated with it every time an estimate  is made' the cost is an attribute of the table which is determined by a key, so must be modelled by a column in this table in which the values are stored.  This is directly analogous to the UnitPrice column in the OrderDetails table in Northwind.  An important point to note is that UnitPrice is also an attribute of Products so there are UnitPrice columns in both tables.  This is because each is determined by the key of the table in question; Unitprice in products represents the current price per product; UnitPrice in OrderDetails represents the price at the time of the order.  The former provides the default for the latter when a row is inserted into OrderDetails.  This has parallels in very many situations.

The other factors to which you refer, type of contractor, labour costs, auxiliary costs such as freight, distance to site etc are all data, so must be stored as values at column positions in rows in tables, which will be referenced by other tables which model relationship types.  Note, however, that data mist not be 'encoded as column headings', e.g. there must not be columns whose names represent a type of cost. A type of cost is a datum so must be a value in a column, with the cost of that type as a value in another column (this is one of the principle ways in which databases and spreadsheets often differ)   Beyond that I cannot say more without a thorough understanding of you business model, an analysis of which will determine the 'logical model' of your database in term of the tables and relationship types required to model the reality.

Whatever final result you wish to produce by way of an estimate must be entirely derived from base data stored in columns in tables.  Where it is necessary to compute values from the base data the computed value must not be stored, but only the base data from which the computation is undertaken, a simple example is the computation of the gross price on the basis of stored unit price, quantity and tax rate vales for each row in an order or invoice lines table.  To store the gross price and/or any intermediate value between the base data and the gross price would introduce redundancy and the risk of update anomalies as the table would not be correctly normalized, the intermediate and gross prices being transitively dependent on the key.

I don't think I can do any more than outline these basic principles.  Neither space nor time here permit the level of detailed analysis of the business model which would be required to be categorical with regard to the appropriate logical model or its representation by forms.  Although it has no direct functional relevance to what you are attempting you might care to take a look at InvoicePDF.zip n my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This demo file is really intended to demonstrate how to output a report as a PDF file, but it happens to be based on a simple invoicing database which illustrates in a simple way the basic principles I've outlined above.  In particular note how the Invoice Details table models the many-to-many relationship type between Products and Invoices, and, as with unit price in Northwind, includes columns for both UnitPrice and TaxRate which are also represented in the Products table.  Note also how the final invoice report is produced entirely from the base data stored in the set of correctly normalized related tables with no redundancy, and with no data hard-coded in code or as properties of objects such as controls.
