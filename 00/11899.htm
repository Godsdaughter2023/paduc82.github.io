jason (office engineer) say:

Hi Martin,

The same URL is used for people to edit and view your Sway.  When a non-author opens the URL, it will open in view mode automatically (you can test this by opening the URL after signing out or in an In Private window).  By default, only the person that created the Sway is an author.  If you send out an Edit Link to anyone, those people can also become authors, but if you haven't done that, nobody else will be able to edit it.

If you want to open a Sway directly into play mode when you are signed in as the author, add ?play to the end of the URL.  There is also a button on the ... menu for an individual Sway that opens it directly into view mode.

Hope this helps!

Jason

jason (office engineer) say:

When you create a Sway, people logged in as your account will be the only people that are able to edit it or get to the edit page with or without a password.  The same URL is used by everyone to open it and if the opener is signed in as the author they will automatically get the edit page while everyone else gets the play page.  Putting ?play on the URL is only necessary for authors to open their Sway directly into play mode without having to go through edit first.

If you add coauthors to your Sway (by having them follow the special edit link), they too will get the same behavior as the Sway's primary author, but that is the only exception.

The link that you mention to a template is special.  You aren't actually in edit mode when you view it, but when you click edit, a copy is created in your account and opened into edit mode for you.

Access Desktop database are very different from Access 'web' Apps.  You cannot simply 'upgrade' and existing app.  I which there was a way to do so but there isn't.  You have to completely restart the project as a Web App.  The only thing you can salvage is the tables (and even then).

Also, Web Apps do not allow any form of VBA programming!  So if you cannot perform all your automation using Macros, Web Apps will not work for you.

One possible approach is creating a Hybrid Database (Tables (BE) in the cloud, FE local desktop db), but this has issues with performance.

Web Apps run in the user's browser, so it works for anyone with a web browser.  They don't need Access and you don't need to deploy an FE to them.

Developer Addition hasn't been around for a while now.

Not to burst anyone's bubble, but for me, for a web application of the scope you are talking about, Access would not be my technology of choice.  I'd be looking at MySQL and PHP, .Net, ...  something that you can code and add proper adaptive programming (AJAX, js, ...)

Good luck to you, this sound like a big project.

daniel pineault say:

Another approach I've used with numerous clients now, to avoid redo everything, is to deploy the database using

  • Terminal Services
  • CITRIX

you keep developing as you always have, nothing changes for you.  Your users simply connect through a remote computer.  This work very well, can be accessed from anywhere you have a network/internet connection and typically performance is excellent.

Just a thought.

jay freedman say:
That can certainly be done in Word, and probably just as well in Publisher.
 
In Word, I'd start with a three-column table. The left-most column is the width from the
left edge of the gray bar to where the words "Career Overview" and the horizontal lines
start, about 0.25 inch. The second column is where the words "Career Overview",
"Professional Experience", and so on are located, about 1". The third column contains
everything to the right of the gray bar.
 
After inserting the table, press Ctrl+Alt+U to turn off all its borders. If you can't see
where the table is, click Table > Show Gridlines; these appear only on screen and don't
print.
 
Select the first and second columns. In the Borders & Shading dialog, set the background
to the shade of gray you want to use. If that shade is dark enough, and if the text in the
second column has Automatic font color, the text will automatically become white.
Otherwise you can explicitly set the font color to white.
 
The horizontal bars are made by selecting the second and third columns, going to Borders &
Shading, and turning on only the top, bottom, and horizontal middle borders of the cells,
setting them to a darker gray than the first column.
