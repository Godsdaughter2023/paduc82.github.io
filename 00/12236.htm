ken sheridan say: I'd suggest that you don't filter the subform but restrict its recordset by making its RecordSource property a query which references an option group as a parameter.  The option group can go wherever you like, e.g. in the subform's header or in the parent form immediately above the subform.  If we assume the values of the option group are 1 for Industry, 2 for Aircraft, 3 for Agriculture and so on, the syntax for the query's WHERE clause would be along these lines:

WHERE ((Industry = TRUE
  AND Forms!YourParentForm!YourOptionGroup = 1)
  OR Nz(Forms!YourParentForm!YourOptionGroup,0) <> 1)
AND ((Aircraft = TRUE
  AND Forms!YourParentForm!YourOptionGroup = 2)
  OR Nz(Forms!YourParentForm!YourOptionGroup,0) <> 2)
AND ((Agriculture = TRUE
  AND Forms!YourParentForm!YourOptionGroup = 3)
  OR Nz(Forms!YourParentForm!YourOptionGroup,0) <> 3)
AND etc.............

This assumes the option group is in the parent form.  If it is in the subform it would be referenced with the syntax:

    Forms!YourParentForm!YourSubformControl.Form!YourOptionGroup

Wherever it is located, in its AfterUpdate event procedure Requery the subform control to restrict the subform's recordset.

I hope that you realize that the table design breaks one of the fundamental rules of the database relational model by doing what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  In essence what you have is a spreadsheet masquerading as a relational database table.  For this reason this design error is sometimes known colloquially as 'committing spreadsheet'.   It makes querying of the data much more difficult.  With a correctly designed table where the data are all stored as explicit values in multiple rows at a single column position restricting the results on the basis of a value in the one column would  be very simple indeed.

karl timmermans [outlook mvp 2012-2018] say:
Some background on how things work with IMAP accounts (following is assuming that you only had Imap accounts configured in the profile)

#1 - In Outlook 2010, you would have one PST file for each IMAP configured along with one PST file for the default contact/calendar folders. As a for instance, if you had 4 IMAP accounts configured, there would be 5 PST files (one for the email folders associated with the Imap accounts and one set as the "default data file" applicable to all accounts. The default data file is the one that you need to be sure to save if ever moving to another machine).

#2 - Under normal circumstances, upgrading to OL'2016 on the same machine, Outlook would create OST files for each of 4 PST files related to email folders and retain the default data file. Each account would then be re-sync'd with the server.

#3 - The fact that you state that your profile only shows PST files after upgrading to '2016 indicates that something went wrong during the upgrade process which may well have been related to whatever was causing '2010 to regularly crash. If you have an ongoing problem, usually not a good idea to upgrade purely in hopes of resolving whatever the problem is since the cause is likely just going to get carried forward. In this case, sounds like there is an issue with one of the IMAP accounts on the server or there was a problem with one (or more) of the PST files causing the upgrade not to complete as expected.

Suggestions given that there is no way of knowing what the original problem is/was

#4 - With Outlook closed, use the Control Panel <Mail> app --> click on the Data FIles tab and take note of the PST file that is set as the default data file.

#5 - Return the window with <Show Profiles> to create a new Outlookk profile and configure the IMAP email accounts. Once the email accounts are configured, add the PST file from step #4 to the newly created profile and make sure that it's set as the default data file for the profile. When done, make sure that the new profile is configured as the default profile for when Outlook is opened.

How to create profile and set up an e-mail account in Outlook
http://support.microsoft.com/kb/829918

*** if you still encounter any kind of problem when Outlook is opened, you may need to repeat step #5 - one email account at a time to determine which account is causing the problem and go from there.
