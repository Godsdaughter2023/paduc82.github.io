=if(month>=Apr, Income!$B$4,original formula)

You probably need a different test to check for April. Something like: month(now())>4

ken sheridan say: A relational database is a very different creature from a spreadsheet.  Don't be mislead by the superficial similarity of a table or query's datasheet with an Excel worksheet.  A relational database table is a set.  By definition a set has no intrinsic order, so concepts like 'previous' have no meaning to set.  A row might be identifiable as immediately previous to another row when they are sorted in a specific sort order, and it is on this basis that balances are computed, not on any fixed ordinal relationship between the rows per se.  When you say 'Access, it seems, has to separately add every row from row one to the current row for every single row' that is not really the case conceptually.  As we are dealing with a set the process is one of summation, not addition, which are mathematically not the same thing.  How this is actually done under the skin is of no concern to the relational database model.

You'll find simple examples of how to compute balances in a variety of contexts in Balances.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

Note that the DSum function is not an efficient method, and should only be used where an updatable recordset is required.  For a report a JOIN is the most efficient method, as the recordset need not be updatable.  If you experience performance problems pay particular regard to the correct indexing of the columns in the base tables; it makes a big difference.

Your bank will be using a relational database management system to compute your balances.  It won't be Access, but the underlying data modelling and processing will be essentially the same.

ken sheridan say: The T1 and T2 aliases allow the two instances of the same table to be differentiated so that they can be joined.

When using separate credit and debit tables you firstly need to bring them together into a single result table by means of a UNION ALL operation.  In my demo only two tables are used, Credits and Debits, but with multiple tables it would simply be a case of extending the UNION ALL query to include the other tables.  My query for this is:

SELECT "C" & Format(CreditID,"000000000") As TransactionID,
CustomerID, TransactionDate, 0 As Debit, Credit
FROM Credits
UNION ALL
SELECT "D" & Format(DebitID,"000000000"),
CustomerID, TransactionDate, Debit, 0
FROM Debits;

In the result table credits and debits are distinguished by concatenating 'C' or 'D' as appropriate to a formatted value of the CreditID or DebitID primary key of each table as the computed TransactionID column.  The formatting with leading zeros allows for the concatenated values to be correctly sorted in the subsequent query based on this one's result table. In your case as the primary keys from the multiple tables might conflict you'd have to cancatenate 'D1', from one table 'D2' from another and so one rather than just the letter.  Similarly for credits.

The subsequent query which returns balances per customer is:

SELECT Q1.CustomerID, Q1.TransactionDate, Q1.Credit, Q1.Debit,
SUM(Q2.Credit-Q2.Debit) AS Balance
FROM qryCustomerTransactions AS Q1 INNER JOIN qryCustomerTransactions AS Q2
ON (Q2.CustomerID=Q1.CustomerID) AND (Q2.TransactionDate<=Q1.TransactionDate)
AND (Q2.TransactionID<=Q1.TransactionID OR Q2.TransactionDate<>Q1.TransactionDate)
GROUP BY Q1.CustomerID, Q1.TransactionDate, Q1.TransactionID, Q1.Credit, Q1.Debit
ORDER BY Q1.CustomerID, Q1.TransactionDate DESC, Q1.TransactionID DESC;

In this each instance of the UNION ALL query's result table is differentiated by the aliases Q1 and Q2.

One thing to note about the queries in my demo is how the primary key TransactionID is brought into play in the JOIN criteria.  This acts as a tie-breaker where two or more transactions occur on a single date.  The balances are therefore per transaction, as with a conventional current account (checking account in USA speak) statement for instance.  Without this the balances at close of business per day would be computed.

ken sheridan say: The aliases for the separate instances of then tables are simply assigned in the query's FROM clause with:  TableName AS Alias.  The same goes for query's.  The aliases then enable the query to treat each instance of the same table or query separately, in exactly the same way as would be the case if they were separate base tables.  This ability to relate two sets of the same data is an important part of data processing in a relational database.  A common scenario, for instance, is with a bill of materials in which a part might be an indivisible base part, or it might be a component made up of multiple parts, some of which can themselves be components made up of multiple parts, and so on down the line.  This is modelled by having a single table of Parts which encompasses all base parts and components.  This table, known as an 'adjacency list'. is then related to itself by another table with columns such as MajorPart and MinorPart, each of which is a foreign key referencing the primary key of the Parts table.  Consequently any component, however complex (a Boeing 747 say) can be modelled right down to its individual nuts and bolts etc by this very simple two-table model.  Producing the bill of materials as a result table of a query is the tricky part as it requires recursive querying, which the JET/ACE flavour of SQL used by Access doesn't support, though more developed products like Oracle do.  It can be simulated in Access, however, and one of my online demos does this.

Your report solution is fine, and a very appropriate one if the data is to be presented as a report.  Queries like mine are only necessary if the data is to be presented as a form.  A report is not just a query on steroids, however; it operates very differently from a query.  A query is a single set operation which, conceptually at least; what the system actually does under the skin is irrelevant to the database relational model which only concerns itself with the conceptual level.  A query basically uses two types of mathematics, relational algebra, on which the original relational model introduced by Codd in 1971 was based, and relational calculus, of which elements have been introduced as the SQL language has developed since then.  A report on the other hand is more procedural, and processes the data procedurally before the final output is formatted.  Mechanisms like the RunningSum property do this behind the scenes, but you can do similar things yourself with code in the Format and Print event procedures of a report's sections.  One could say that a 'running sum' mechanism is available in a query, but you have to build it yourself through the SQL, which while a relatively compact language, is a very powerful one.

melchizedek qui say:

Hello Jim,

If you want to set Internet Explorer as the default browser and want all your links to open always in Internet Explorer with the Metro UI, select the Always in Internet Explorer option. It's best to use this option if you normally browse the internet and open link using Internet Explorer.

The Let Internet Explorer option, decides how it opens links. So in this case, sometimes Internet Explorer may open with the Metro UI and sometimes on the desktop, depending on the app or program that is accessing it.

While the Always in Internet Explorer on the Desktop option allows you to launch links in the desktop environment.

The Open Internet Explorer tiles on the Desktop option automatically launch Internet Explorer tiles on the desktop.
