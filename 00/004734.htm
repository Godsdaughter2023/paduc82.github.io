john w. vinson say:

The HAVING  clause is best for criteria which apply to the result of an aggreagate function - if it's a static table field you should use the WHERE clause instead. In the grid you can select the pseudo-aggregate function WHERE. When you have multiple criteria for the same field it's also more compact and more efficient to use an IN() clause instead of multiple OR.

 

Try

 

SELECT DISTINCTROW [LOD A].TYPE, [LOD A].RRC_district, [LOD A].COUNTY, Sum([LOD A].VALUE) AS SumOfVALUE, [LOD A].OWNER, [LOD A].ADDRESS1, [LOD A].ADDRESS2, [LOD A].ADDRESS3, [LOD A].ADDRESS4, [LOD A].CITY, [LOD A].STATE, [LOD A].ZIP, Count(*) AS [Count Of LOD A], [LOD A].[TYPE]
FROM [LOD A]

WHERE [LOD A].[RRC_District] IN ("7", "8", "9", "8A", "7B", "7C")

AND [LOD A].[TYPE] = "RI"

GROUP BY [LOD A].TYPE, [LOD A].RRC_district, [LOD A].COUNTY, [LOD A].OWNER, [LOD A].ADDRESS1, [LOD A].ADDRESS2, [LOD A].ADDRESS3, [LOD A].ADDRESS4, [LOD A].CITY, [LOD A].STATE, [LOD A].ZIP, [LOD A].TYPE
HAVING Sum([LOD A].[VALUE]) > 5000

 

One question: do you have multiple VALUEs for a given address?

 

I suspect the criteria weren't being applied properly because of the OR logic - it was looking for District "7" AND Type = RI, OR District 8 AND any type, OR District 9 AND any Type, etc.

ken sheridan say: On the basis of your original description the logic is:

.....
WHERE [Field 1] = "R1"
AND ([Field 2] = "9" OR [Field 2] = "8" OR [Field 2] = "7")

HAVING SUM([$]) > 5000

So you need to restrict the query on the Field 1 and Field 2 columns by means of a WHERE clause, and on the aggregated Field 3 column by means of a HAVING clause:

SELECT [TYPE], RRC_district, COUNTY, SUM([VALUE]) AS SumOfVALUE,
OWNER, ADDRESS1, ADDRESS2, ADDRESS3, ADDRESS4,
CITY, STATE, ZIP, COUNT(*) AS [Count Of LOD A]

OR RRC_district="7B" OR RRC_district="7C")
GROUP BY [TYPE], RRC_district, COUNTY, OWNER,
ADDRESS1, ADDRESS2, ADDRESS3, ADDRESS4,
CITY, STATE, ZIP
HAVING SUM([VALUE]) > 5000;

I would avoid Type and Value as column names however, as they are 'reserved words' in Access, so are best not used as column names.  Use more specific descriptors like TransactionType, TransactionValue or whatever appropriately describes the attribute represented by the column.

john w. vinson say:

I think Ken and I are in agreement. Access' query grid will (unfortunately!) automatically put all criteria into a HAVING clause in a Totals query, even if it's not really appropriate.

 

It does work but maybe not in the way you want: the WHERE criteria are applied before the totalling and grouping take place; the HAVING clause is applied afterward. The drawback here is that you can spend a lot of time summing and calculating results which just get discarded unused after they're calculated.

 

You can override Access' use of HAVING by choosing "WHERE" from the dropdown on the Totals row - you can have Sum, Max, Min etc. on this row but you can also choose WHERE. This will remove the field from display unless it also is used in the GROUP BY. Try:

 

SELECT DISTINCTROW [LOD A].TYPE, [LOD A].RRC_district, [LOD A].COUNTY, Sum([LOD A].VALUE) AS SumOfVALUE, [LOD A].OWNER, [LOD A].ADDRESS1, [LOD A].ADDRESS2, [LOD A].ADDRESS3, [LOD A].ADDRESS4, [LOD A].CITY, [LOD A].STATE, [LOD A].ZIP
FROM [LOD A]

WHERE ((([LOD A].TYPE)='RI') AND (([LOD A].RRC_district) In ("9","8","8A","7B","7C")))

GROUP BY [LOD A].TYPE, [LOD A].RRC_district, [LOD A].COUNTY, [LOD A].OWNER, [LOD A].ADDRESS1, [LOD A].ADDRESS2, [LOD A].ADDRESS3, [LOD A].ADDRESS4, [LOD A].CITY, [LOD A].STATE, [LOD A].ZIP
HAVING Sum([LOD A].VALUE)>5000;

Then switch back to grid view to see how Access displays it.


 

ken sheridan say:
I am curious how to do the Having statement in the "grid".  Is that possible?  I am already using the SUM function on the field.  Is this something I have to add manually in SQL?

With a 'totals' query to create a HAVING clause in design view you just enter the criteria in the column which sums the values.  It's creating a WHERE clause in a 'totals' query which is a little bit trickier, and where you went wrong originally.  What you have to do is enter the Type and RRC_district columns a second time to the design grid, uncheck the 'show' check box in each, select Where as the 'total' and put the criteria for each in these columns.

BTW, using the IN operator should be fine with a relatively modest recordset, but I think I'm right in saying that using the OR operator as in my example is more efficient as it makes use of the indexes.  Unless you have a very large table you probably won't notice any difference, though, the end result will be the same (there's often more than one way to skin a cat!).
