albert d. kallal say:

Ok, so we making progress.

I think the best work flow would be:

Scan, and THEN cursor is placed in the QTY box. If the 1 value is ok, then you hit the enter key. When you do that, the barcode and qty is then written to the table (or QTY update processing code can run).

And of course after hitting enter, in the qty box, and after your code runs then the barcode box should be emptied, and the cursor placed back into that box ready for the next scan.

So the workflow would be:

Scan – and if qty does not need changing then you hit enter, and you ready for the next scan.

So

Scan – enter (qty 1)
Scan – enter  (qty 1)
Scan – type in 5, hit enter (qty 5)

Futhermore, you likely could (should) shoot for the goal that when you scan, and if the qty is 1, then you don’t have to touch or do anything with the keyboard.

The “way” you can achieve the above is that most scanning software allows you to configure the scanner to HIT (type) in a starting keycode. I like to use F12 or some such key. Now what happens when you scan the barcode you get this:

<F12>33506102<cr key>

You thus THEN setup an auto keys macro. When you hit F12, then your code runs to flip “focus” to the above form, and then set focus to the barcode text box. The reason for this design is then you can have a regular working Access form for the user, but when they hit F12 (or scan), the F12 code will launch the above form, set focus to barcode, and then the box will be filled with the barcode. This means you can build an application that is “ready” for scanning and use of the scanner at all times. (even if a user is in a different part of the application).

Also, looking at your screen shot it does look like the cursor after the scan is on the NEXT line – this suggests the scanner types in the number and then hits the linefeed key (either 10 or 13 as a key code). If the scanner after the number types in the Enter key, then the cursor would jump to the next text box – since it does not, then the linefeed (ctrl-j) is likely what is being pressed here (by pressed I mean the key hit by the scanner after the barcode is typed in).

So before you crack open the scanner manual and play with the F12 key code suggesting, try placing this code in the textbox key up event:

Private Sub Text0_KeyUp(KeyCode As Integer, Shift As Integer)


   Debug.Print KeyCode
   If KeyCode = 10 Or 74 Then
     
      KeyCode = 0
      ' move focus to qty text box
     
      Me.Text2.SetFocus
     
   End If

  
End Sub

The above will thus allow you to scan, and have the cursor jump to the second text box. (you have to replace above text2 with the actual name of YOUR text box).

And the above will ALSO display all keycode values in the debug window – that will allow you to determine the leading + trailing keys entered by the scan.

As for “code” and updating the actual tables – I would not worry about that until you get the above basic scan + jump to qty text box working. Once you have that, then you can start to write code that would enter/update the actual data tables.

I do recommend you continue to use an un-bound form – especially for a scanning form since miss-scans etc. would thus not be updating the table(s) in question directly.

- respect above hint
