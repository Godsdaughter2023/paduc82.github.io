The error message "currently locked by another user" has nothing whatever to do with field size: it means what it says, that the record is currently locked by another user or another process. Most commonly the "other user" is yourself - i.e. you may have a Form open based on a table or query and you're trying to run code to update the same record that you have open.

A TEXT datatype field in a table is limited to 255 bytes (including carriage return-new line pairs as two characters); a Memo field to a billion bytes if you're updating it programatically, 65536 manually.

albert d. kallal say:

Well, first there are several types of locks that occur in Access tables.

You don’t mention what object model you are using to edit the data, but a few things:

Access WILL use page, or what we call FRAME locks when updating data. In fact if several “small” records reside in one of these frames (or often called buckets), then during an update that frame will be locked.

In a multi-user environment, this often showed up in sub forms with small records.

However the above ONLY in general occurs if you opening the table with locking. (it not clear which object model, and how you are opening such tables).

You can however set the database to use row locking to eliminate the above issue – but the cost of row locking (as compared to page locking) is RATHER LARGE file bloat (growth). Access actually fakes row locking by simply expanding all records to take up one whole database frame (bucket). Since multiple reocrds are never placed together in a frame bucket, then the issue goes away! So you can choose page locking in access, or row locking – but real row locking is in effect still page locking!

I suppose the other possible issue could be some type of threading or other issue – the record is not yet released, and you have some other code/process attemting to use the same row of data (but issues only show up because of bucket/frame shuffling that ONLY occurs when adding larger reocords – if the record cannot fit BACK into the currect bucket, then additional buckets are added to the file – this can take time.

Because ACE/JET is not thread safe, then using a Access back end for a web site can often show up issues like above. (because the web software is threaded).

I would check existing update code – MAKE SURE you use the same connection for the update code and not multiple ones.

As a general rule, the simple addition of a memo column should not cause this issue. As you say, you been using Access for a long time and you not seen this in 15+ years. Is this occurring on the desktop client, or on the web site side of things?

It not clear if you using the .net providers to update the ACE table, or ADO.

And while being able to simple upload the Access file to the web server – what are the default locking options and settings for that server when using ACE/JET?
