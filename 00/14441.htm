the sands say:

I figured it out last night and posted the answer - I have no idea why it's not here???  I worked hard on that post and don't appreciate having to redo it...

 

Here is what you do to start an auto number list, on a number other than 1 - when it has to be 1 not 1. or 1)

Insert

Quick Parts

Field

Seq

in the field codes box type this:

SEQ Step \r 70 \* MERGEFORMAT

(my list starts with 70, put in your number after the \r)

For the next number on the list use

Insert

Quick Parts

Fields

Seq

(in the field codes box type this:)

SEQ Step \n \* MERGEFORMAT

 

The first \r will start the list on the number you want... the second will "count" from thereafter... you can copy the 2nd numer or 3rd or whatever to keep the numbers going...

 

As mentioned above, the list will not update automatically, but you can highlight the numbers/list and right click - hit "Update fields" and it will update.

 

By doing the above you get the number 1 or any number you choose without a period or a parenthesis after it.  If you don't mind the period if 1. or 1) is okay with you, you can use Insert/Quick Parts/ListNum and enter the "start at value" as any number you want (which will start the list) then for the second number do Insert/Quick Parts/ListNum and leave the start at value empty and it will starting "counting."  If you select "LegalDefault" you will get a number with a period after. 

 

Hope this helps others... and I hope the moderators don't take it down.

ken sheridan say:

You might like to take alook at the following:

http://community.netscape.com/n/pfx/forum.aspx?tsn=1&nav=messages&webtag=ws-msdevapps&tid=23133

Its the file attached to my post in the thread in what was the old CompuServe MsDevApps forum.  Its really intended to demonstrate a method of creating bills of materials in Access, but could be adapted to handle staff records.  It uses the conventional method of an adjacency list table with two columns referencing the key of the Parts table (staff in your case).  The demo includes two files, one which simulates recursive querying, the other which uses a query with multiple instances of each table.  It is this file, PartsTree, which includes a diagrammatic report.  This however limits the maximum number of hierarchical levels to 9, which might not sound a lot but in fact should cope with most situations.

By using an adjacency list table what is modelled is a convergent graph rather than tree in that there can be more than one path between two nodes; a tree has only one path between any two nodes.  For a staff model this might or might not be necessary.  I once had a member of staff who was responsible to another section head as well as to myself, in effect holding two part-time jobs, so model would be a convergent graph.  Where a member of staff is responsible to only one line manager, however, a tree would be sufficient.  Joe Celko devised a very efficient method of handling trees as a a nested set model.  For details see his article in Intelligent enterprise magazine at:

http://www.intelligententerprise.com/001020/celko.jhtml

where he uses a personnel model as the example.  Using this model you'd need to work out how to show the data diagrammatically, however, as Joe's article is purely concerned with the model and its implementation in SQL.
