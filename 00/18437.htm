graham mayor say:
You can do it with a script run from a rule.

Create a rule to identify messages with your name in the recipient field from the association and run the following script from that rule:

Sub CopyToLiza(olItem As Outlook.MailItem)
Dim olFolder As Outlook.MAPIFolder
Dim olCopiedItem As Outlook.MailItem
Dim olNS As Outlook.NameSpace
    Set olNS = Outlook.GetNamespace("MAPI")
    Set olFolder = olNS.folders("Liza's Account Name").folders("Inbox")
    With olItem
        If InStr(1, olItem.To, "Liza@ourinternetaddress") > 0 Then
            Set olCopiedItem = olItem.Copy
            olCopiedItem.Move olFolder
        End If
    End With
End Sub

Change the two items in bold to reflect your wife's account details. Then when the message arrives that has your address and your wife's address in the TO field, it will create a copy of the message in her inbox.

You are still going to get two messages, but now you will both get two (unless there is some way to distinguish between them, which seems unlikely). Checking the folder to ensure that the message doesn't already exist is going to slow down the process markedly.

mbs it say:

Well, 3 months later we seem to have finally resolved this issue.

We started by following advice here and updating to Office 365.

This temporarily solved the problem, public folders reappeared, but after a week or so they started vanishing from profiles.

Restarting Outlook would bring them back for a couple of minutes, but they would disappear again before you could do anything useful.

Deleting and starting a new outlook 2016 profile would restore full functionality, but it could reoccur days or weeks later.

Eventually running SARA tool brought up an issue with invalid autodiscover responses coming from our web host.

Our domain web hosting is on a cpanel site totally separate to office365.

It appears outlook was looking for autodiscover info on the web host (since it has the same domain as our email address). 

In our hosting Cpanel the Email routing item was set to “Automatically Detect Configuration:Local
I was able to change that to “Remote Mail Exchanger” (The server will not accept mail for this domain. The system sends all mail for this domain to the lowest numbered mail exchanger.)

This appears to have entirely resolved the problem. 

Thought I would post here in case it helps someone else.

The problem seemed to have been triggered by a change in web hosting to new hardware, that must have changed some cpanel behaviour too.

The problem appears to be that you are using content controls for the text, together with form field check boxes. The latter require that the document is protected, while the former do not. Use text form fields (found at Developer tab | Legacy Tools) instead of plain text content controls. You'll find that they will work better with the form field check boxes.
