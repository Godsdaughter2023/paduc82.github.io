john w. vinson say:

It's not easy, at least in part because the "size" of an object isn't all that well defined. A Table will have space for the data stored in it - 4 bytes for a Long Integer, 8 for a Double or a Date, 2 bytes per character for Text, etc. - but there is also some overhead per record, indexes take up space, there is additional (undocumented mostly) space taken up in system tables, and so on. On top of this, there is always uncompacted "waste space" from deleted records or deleted objects which doesn't get cleared except by compaction, and not always then. Your proposed size check would almost surely givee different results if you compact after each addition!

Finally, what benefit is there to knowing this bit of information? You can tell in general terms that a 1,000,000 row table of 400 byte records will be big, as will a 100 row table with a memo field containing ten printed pages each record, while a 100 row table with two number fields will be much smaller; what is the benefit to knowing that a given table is 23.8MByte as opposed to 15.31MByte?

alden rey say:

Hi,

Whenever you save a Microsoft Office document (Word, Excel or PowerPoint) to your local hard drive or to a fileserver, an associated temporary file is created to the same location as your MS Office document. However, you will not be able to see the associated temporary file from within the Finder because it is a hidden file.

The associated temporary file remains present in the same location as your MS document until you close your document.

The purpose of this associated temporary file is to lockout editing of your MS document to other users while you have it open. The associated temporary file also stores the name of the person who currently has the MS document open, and if another user attempts to open the document, it displays a warning dialogue (see below) to inform them that, “The file is locked for editing”.

In this case, we suggest to update to fix this concern by starting one of your Office applications like Word, and then clicking File > Account > Update Options > Update Now. After install the update, the Office version should be the latest.

milt0424 say:

How to construct a mail merge rule using "IF . . . AND"

I need to construct a mail merge rule that will insert the word "and" into the document when two specific columns both contain data. a. user say:

There are various possible ways to do it, including

{ IF "{ MERGEFIELD E }" <> "" "{ IF "{ MERGEFIELD F }" <> "" " and " }" }

Each pair of {} needs to be the "special field braces" that you can insert using ctrl-F9. Everything else is ordinary text.

Peter Jamieson

doug robbins - mvp office apps & services (word) say:

You need to use the following field construction (where you use Ctrl+F9 to insert each pair of field delimiters { } )

in memory of { MERGEFIELD "Column E " }{ IF { MERGEFIELD "Column F" } <> "" " and in honor of { MERGEFIELD "Column F" }." "" }

You can simplify entering this by inserting the merge fields into the text using the insert merge fields facility

in memory of «Column E»«Column F» <> "" " and in honor of «Column F»." ""

then select the

«Column F» <> "" " and in honor of «Column F»." ""

part of the above and press Ctrl+F9 which will enclose it in field delimiters

{ «Column F» <> "" " and in honor of «Column F»." "" }

Then press Alt+F9, which will convert the whole thing to the construction shown above.  Pressing Alt+F9 again, will toggle off all of the field codes in the document and you can then execute the merge.

If there was a possibility that the donation was only made "in honor of" then you would need to use

{ IF { MERGEFIELD "Column E" } <> "" "in memory of { MERGEFIELD "Column E" }{ IF { MERGEFIELD "Column F" } <> "" " and in honor of { MERGEFIELD "Column F" }." "" }" "in honor of { MERGEFIELD "Column F" }." }

Hope this helps.

suzanne s. barnhill say:

It would appear that the document contains a number of section breaks. Although you can work your way through the document, accessing the header/footer for each section and using Format Page Number to select "Continue from previous section," it may be more efficient to remove all the section breaks temporarily, make sure that numbering in the remaining single section is set to "Continue...," then reinsert section breaks as needed.

 

The reason copying and pasting didn't help is that the formatting (including numbering restarts) is stored in the section breaks and the last paragraph mark of the last section, which you copied and pasted.

doug robbins - mvp office apps & services (word) say:

Run a macro containing the following code to have the page numbering in each section continue from that in the previous section.

 

Dim i As Long
With ActiveDocument
    For i = 2 To .Sections.Count
        .Sections(i).Footers(wdHeaderFooterPrimary).PageNumbers.RestartNumberingAtSection = False
    Next i
End With
