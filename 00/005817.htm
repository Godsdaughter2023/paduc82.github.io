And, if you using linked tables or quires to sql server to fill out combo boxes, then again you get stray locks.

 

The simplest solution to all of the above issues, and also that of forcing the linked table to be read only?

 

Create views for the combo boxes on sql server side. Include the WITH (NOLOCK) hint in those queries.

 

The result is;

You not messing around with a whole bunch of quires and code and stuff to pull data from sql ever to update local tables that drive combo boxes. That’s just a messy approach, and you can dump all of those hand stands.

 

And, by using a view, the linked table (view) from Access will be seen as read only.

 

So, no question, the issue of stray locks is a very nasty issue and problem. So, just in place of linking to the table, create a view on the sql side, and include with (NOLOCK) hint, and that will eliminate this locking issue.

 

Eg this:

 

select * from tblHotels with (nolock)

 

I would if possible simply eliminate the access back end if possible. The FE should be free to query the data from SQL server, and you thus eliminate the issues of having to maintain + update + mess with an accDB back end. There is always problems with larger user counts, and accDB back ends.

 

Get rid of the access back end – you will 100% eliminate all of the maintains and that messy linking to it. With your user counts, then you be glad you di.

 

You wind up with a nice clean FE, and your back end is now sql server.

 

Regardless?  Those append queries that you run also as you are finding is causing some sql server locking issues. Use linked views in place of the tables – this will fix the locking issue.

 

And, as noted, if you update your FE to directly use links to sql server to drive those combo boxes, then once again, use linked views to sql server to drive the combo boxes (with the nolock hint). If you don’t do this, you will once again get all kinds of stray locks on sql server.
