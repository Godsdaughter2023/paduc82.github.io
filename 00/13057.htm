jllatham say:

NOTE: The above code opens the tilde files as workbooks, and after transferring their data into the workbook with the code, it closes the source file.  You could strip that code down to just the

Workbook.OpenText ....

statement and then it would simply open each file as a separate workbook.  That would look something like this:

 

Sub OpenTildeFiles()
  Dim fileName As Variant ' will be an array if file(s) selected.
  Dim FPL As Integer ' File Pointer Loop counter
 
  fileName = Application.GetOpenFilename("Tilde Delimited Files *.txt, *.txt", , _
   "SelectTilde Files", , True)
  On Error Resume Next
  'this is just a test using FPL as a temporary value
  'if fileName is not an array, user cancelled file selection.
  FPL = LBound(fileName)
  If Err <> 0 Then
    'had an error, it is not an array, probably user cancelled
    Err.Clear
    On Error GoTo 0
    If fileName = "False" Then
      'user cancelled the file selection
      Exit Sub
    Else
      MsgBox "Something quite unexpected has happened!"
      Exit Sub
    End If
  End If
  'speed things up and help prevent screen flicker during operation
  Application.ScreenUpdating = False ' will turn back on automatically at End Sub
  'filename will now be an array of from 1 to n depending on number of files selected
  For FPL = LBound(fileName) To UBound(fileName)
    'open each file in its own window
    Workbooks.OpenText fileName:=fileName(FPL), StartRow:=1, _
     DataType:=xlDelimited, TextQualifier:=xlDoubleQuote, _
     ConsecutiveDelimiter:=False, _
     Tab:=False, _
     Semicolon:=False, _
     Comma:=False, _
     Space:=False, _
     Other:=True, OtherChar:="~"
  Next ' end FPL loop
  MsgBox "All selected files processed."
  ThisWorkbook.Activate ' back to this workbook
  'other workbooks will be open "behind" this one.
End Sub

albert d. kallal say:

As Daniel points out, are you talking about migration of your EXISTING application to SQL server?

Or are you talking about building an application?

If you are asking about adopting SQL server in place of the EXISTING back end (accDB), then such a project should not involve that many changes to your application. In other words, about 99% of your existing code can and should remain un-touched.

How long of a ball of string is required to reach across town? (well how big is the town??)

How much of a budget you require is quite much a guess here. There no fast rule, but **if** you are experienced with Access and SQL server, then you quite know “what” kind of hill you about to climb.

So for any existing form, report and 99% of your existing code? If you move the data from an Access back end to SQL server, then VERY little code changes are required. Things should continue to work as before.

However, from your question(s) it is not clear if you are developing a NEW application , or you already have a working Access application?

Some of your comments are rather confusing, such as:

> I have been modifying the existing programs that update in batch to update in real time

This is rather confusing (and sets off some serious alarm bells here). Whatever changes you are making and are considering now, those SAME changes and approaches will work with, or without SQL server.

So it not clear what you mean by “batch” programs, since in general any Access application I am aware of means that when you edit data in a form, then the data is directly saved to the table(s) in question. And if you move the tables to SQL server, then the application will continue to work EXACTLY the same. Any form you have now will continue to operation and update the tables just like they do now – only difference is the tables now reside on SQL server.

So ANY existing process you have now will and should work continue to work a near identical

fashion with SQL server.

In other words, using Access and Access tables
Or
Using Access + SQL server has ZERO bearing on some batch process.

So this issue of some batch process has ZERO to do with SQL server, but is a fundamental design approach you are free to use – and SQL server will have little or no bearing on this issue.

So anything you doing now in Access should work the same when using SQL server. You would not re-write some code or change some process that is of sound design now. If you have sound design approaches that work with Access, then those same approach(s) will work 100% the same with SQL server.

Of course after you migrate the data from Access to SQL server, then near everything should work as before. However, some reports, and some forms CERTAINLY will benefit from tweaking and using some of the SQL server features to improve performance.

However, beyond this “improving” performance issue (which can occur over time – not all at once after the migration to SQL server), it is not clear why you feel “many” application changes are required here due to moving from Access to SQL server?

How well and how easy the application will work with SQL server is going to depend on how well (or poor) the existing application is designed. The better the designs of the application, the more ease the transition of the back end data from access to SQL server will be.

So no extensive changes are required to the application.

However, you certainly need someone who done such migrations, and this is the list of criteria that the Access developer needs to follow:

  1. Done several Access migrations of data to SQL server, and knows the kind of changes required on the Access front end.
  2. The developer KNOWS that it is a bad idea to re-write tons of VBA code based on say DAO and attempt to convert to ADO (this is a bad idea).
  3. Realizes use of views for slow queries will allow existing filters and VBA “where” clauses to work un-changed after the data is migrated to SQL server. (So existing forms that prompt for user data to launch a report can continue to be used without changes – and you get near pass-through query performance without changes to that VBA filter prompt form).
  4. The developer realizes that bound forms to those SQL tables are just fine. Even if a form is bound to a large 1 million row table, loading of such forms should be “instant” since the developer always opens such forms with a “where” clause that Access always had to restrict data pulled from that table (but again, the existing applcatation should also have been written this way). So the developer knows to avoid large code re-writes and knows to keep bound forms and thus avoid racking up costly re-development when none is required.
  5. Realizes that DAO works for pass-through quires and executing of SQL store process. And that that connection strings are not required in code (the pt quires are to be re-linked at table link time).

As long as the Access developer realizes the above general concepts, then I see little reason why one Access developer can’t migrate the access data to SQL server and have the system up and running in 1 month of time easy. If the above 5 suggestions don’t make sense to the developer involved in the migration then you have the wrong developer. Those 5 suggestion are not the “only” rules, and even if the developer approaches things a bit different then the above 5 suggestions – the suggestions should be perfectly sensible approaches to that developer.

f.     To turn off the Information bar for file downloads, scroll to the Downloadssection of the list, and then, under “Automatic prompting for file downloads”, click Enable.
