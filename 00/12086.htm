ken sheridan say:

Take a look at the file Balances.zip in my public databases folder at:

 

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

 

This little demo file illustrates how to compute balances in a query in various contexts.  If you open the updatable form for balances grouped by customer (fourth button down on the left) and in the form scroll down to a new record, then select a customer in the combo box, you'll see that their current balance is immediately shown.  Once a transaction is entered the record is updated the new balance is shown.

 

If you just want to see the current balance for any customer you could create a query, qryCurrentBalances say, like this:

 

SELECT Customers.CustomerID, FirstName, LastName,

NZ(SUM(Transactions.TransactionAmount),0) AS CurrentBalance

FROM Customers LEFT JOIN Transactions

ON Customers.CustomerID = Transactions.CustomerID

GROUP BY Customers.CustomerID, FirstName, LastName;

 

and in a customer form, look up the current balance in an unbound text box control with a ControlSource property of:

 

=DLookup("CurrentBalance", "qryCurrentBalances", "CustomerID = " & [CustomerID])

 

Format the control as Currency.


I've assumed in the above that the transaction amounts are stored in a single column, with negative values for debits.  If using separate and credit columns then you'd need to sum Credit-Debit in each case of course, as in some of the other examples in my demo file.

 

The forms in my demo are in descending data order, as is common for account statements like this, and are in continuous forms view.  They would of course work equally well in singe form view and ascending date order.

 

One thing to note about the queries in this demo is how the primary key TransactionID is used as the tie-breaker in the event of there being two or more transactions on the same date.

Sub Copy_and_Clear()
