The appearance of the Table of Contents is set by the formatting of the TOC # styles.

By default TOC 1 will be used for the items to which Heading 1 is applied in the document, TOC 2 for the Heading 2 items and so on.

You can access the TOC # styles to modify them via the Table of Contents dialog>Modify button.

stefan blom say:

If the problem is that formatting from the headings turn up in the TOC, then you have direct formatting applied to your headings. Direct formatting is the result when you apply bold, a larger font size, etc. by clicking the Bold, Size, etc. buttons on the ribbon as opposed to modifying your styles properly.

Remove the direct formatting by selecting heading text (in the body of the document) and pressing Ctrl+SpaceBar. Update the TOC.

To properly reformat your headings, you will have to modify each of the heading styles. For example, click inside a Heading 1 and press Ctrl+Shift+S. In the Apply Styles pane, click Modify and make the desired font changes.

Don't think that 100% stacked column charts can have percentage as the datalabel values.

But you can calculate the percentages and display them as text with a macro like this:

Sub set_data_labels_to_percent_for_100_stacked_column_chart()
'Note: the macro has to be rerun when series data change
  With ActiveChart
    For i = 1 To .SeriesCollection.Count
      .SeriesCollection(i).HasDataLabels = True
    Next i
    For i = 1 To .SeriesCollection(1).Points.Count
      s = 0
      For j = 1 To .SeriesCollection.Count
        v = .SeriesCollection(j).Values
        s = s + v(i)
      Next j
      For j = 1 To .SeriesCollection.Count
         v = .SeriesCollection(j).Values
        .SeriesCollection(j).Points(i).DataLabel.Text = Format(v(i) / s, ".0%")
      Next j
    Next i
  End With
End Sub

Change the number of zeroes in the Format parameter to control the number of displayed decimals.

Generally what happens here is that Word opens a file and cannot get the file server to release it.  Word then opens another file, etcetera.  Word needs up to 20 open files for each "document" you see.  Eventually, the file server runs out of "file handles" and you get this condition.

Here is a simple work-around that alleviates the problem for some users:

a)  When you go home at night, Log Out (don't just allow the computer to go to sleep, log right out so you have to put in your login name and password next morning.  This enables your computer to release all its files.

b)  One night a week (Friday night?) Shut Down the computer.  Not just a sleep, not just a log out, this time shut it down all the way until the lights go out.  When it next restarts, the Operating System can then do its clean-up housekeeping.

If this works for you, it will completely prevent the problem.  Of course it shouldn't be necessary, but for some users, it is.

macropod say:
That suggests the table has been split on page 9. If there's a paragraph break after the cell on that page, delete it. It's also possible for this behaviour to occur after two tables with different alignments, widths, etc. have been merged to behave this way. Although they may look like a single table, they don't behave that way in all respects. In that case, they need to be split again and the parts re-aligned & resized so they're exactly the same - sometimes its easier just to add new rows to the part that's working properly and transfer the data.
