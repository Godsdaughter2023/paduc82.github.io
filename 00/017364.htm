nicpaulroberts say:

TLDR: when having this sort of problem, use an error handling function around references to your variable cell to allow solver to step to the next iteration without getting stuck throwing an error.

Not sure if someone else figured this out before me, but I had this problem and it stumped me for quite a while before I found a way around it.

In my case, when solver selected a value outside the constraint range it would cause a #NUM error to propagate through the model, which would cause it to fail to find a solution. This is why I had included the constraint in the first place.

However, it seems that solver selects values for variable cells and calculates the cells before checking if the value meets the constraint(s).

In my case, I used MIN(variableCell, 0.995) to eliminate the #NUM error, where 0.995 was the upper bound constraint. This prevents solver bugging out, and allows it to step to the next iteration and keep looking for a valid result.

In this case, you'd want to use MAX(variable cell, lower bound) to stop solver getting stuck.

Date expired Yes/No field dependant on another field's date - Microsoft Community

colin cerimagic say:

Date expired Yes/No field dependant on another field's date

I have a field that stores the date a quoted price was updated. I would like to have another field (Quote Expired) that shows whether the quote is expired  (expired if date in price field is 3 months old). I have been unable to find a way to do this and need help. 

Thanks.

john w. vinson say:

This Yes/No field should simply not exist in your table, since its value depends on another field. If the value in the field is False today, it might be True next week!

Instead, use a Query with a calculated field: in a vacant Field cell type

[Expired]: [datefield] < DateAdd("m", -3, Date())

When this query is opened or used as the Recordsource of a Form or Report, it will compare the date field to today's date and return True or False as required.

Thanks John. I am new to Access and need to remind myself that tables are not where the information is ultimately viewed.

john w. vinson say:

Sorry for not getting back to you.

True is stored as -1, False as 0. You can base a Form or Report on the Query (you shouldn't be using table or query datasheets for data presentation in any case), or you can set the Format property of the calculated field to display True or False. Right click the little grey bar at the top of the column in query design view, view the field's Properties, and choose True/False from the dropdown for the Format property.

charles kenyon say:
Statements (vba) or fields (merge)?

To troubleshoot your field I would want to actually see it.

Often nested fields are not needed, successive fields will produce the same result and be far less complex to debug. This is the case when the field is testing one variable or condition. Nested IF fields, are, by their structure, IF/THEN/ELSE.

charles kenyon say:
OK, these are not nested but rather successive or sequential mergefields. Nested fields are fields inside of fields.

 If none are true in this pattern nothing shows up in the document except a blank line. You can include a line break or paragraph return at the beginning of your truetext to start a new line if there is something to insert. It looks like that is what you are doing.

If you put the line break at the beginning of your truetext rather than at the end, if none of your conditions are met, you will not have a blank line.

More on Mail Merge: http://www.addbalance.com/usersguide/mailmerge.htm

Hope this helps.
