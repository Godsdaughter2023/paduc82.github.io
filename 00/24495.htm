john w. vinson say:

It is possible to store the calculated value in a table.

But it is a bad idea to do so, and it's not necessary!

Tables aren't spreadsheets, and don't work like spreadsheets. Typically in a spreadsheet you will have a mix of raw data and expressions. That's perfectly appropriate - for a spreadsheet!

In contrast, a Table should be used to store only "real data". It's a basic principle of normalized design that a field in a table should be functionally dependent only on that table's primary key. Storing derived data (such as this product) in a table accomplishes only three things, none good: it wastes disk space (trivial these days, I know); wastes time (a multiplication is VASTLY faster than a disk fetch); and most importantly risks data corruption; if one of the underlying values is edited, your product is now wrong with no easy way to tell that it's wrong, or which of the three fields might have been edited.

Keep your tables "under the hood". Finance (or even you, once the application is working) never need to even SEE the tables. When you need to present data to Finance, they'll be seeing real data direct from tables, or calculated data calculated in a query, a form, or a report, or a mix of all of these, using whatever database tool is appropriate. But just don't assume that "if I'm going to show it to Finance then it has to be a field in a Table" - because that's simply not how databases work!

ken sheridan say:
...............but I was hoping there is a way that this revenue figure would show up directly on a Finance table with fields Revenue, Costs, Depreciation etc,-- so that middle step of having to run the revenue report/query  (calculation) is not necessary....

An important concept you need to grasp is that in a relational database a table is not necessarily a base table, but can equally be the result table of a query.  As such it can be used in the same ways as a base table, e.g. as the RecordSource of a form or report, and can be referenced in another query in the same way that you would a base table.  Care needs to be exercised in doing the latter, however, as things can easily get out of hand and inefficient if you have queries referencing other queries, which themselves reference other queries.  I have often received badly designed databases to work on, where the same computations are done more than once in queries within queries.

When you save a query what you are saving is simply the query definition.  A query can do a variety of things; it can apply restrictions to determine which rows are returned; it can apply projections over columns from one or more tables to determine which columns are returned; it can compute values on the basis of values in multiple columns; it can aggregate values over a set of rows using aggregation operators like SUM, AVG, MIN, MAX etc.  Any database application beyond the trivial is likely to contain a significant number of queries, and these will be the main way in which the users interface with the data, by means of forms or reports based on the queries.

A relational database is a model.  It models that part of the real world with which the database is concerned in terms of its entity types and the relationship types between them.  Each column in a table represents an attribute of the entity type, and each attribute must be determined solely by the whole of the table's primary key.  This eliminates, or at least reduces as far as is practically possible, redundancies which can otherwise lead to update anomalies resulting in unreliable data.  The elimination of redundancies is formalized by the process of normalization, which is defined by a set of Normal Forms the basis of which is an analysis of the functional dependencies in the table, and between tables when one comes to the higher normal forms.  The sort of model you are contemplating would involve major transgressions of the principles of normalization, resulting in an unsound and unreliable application.  That's by the bye, though, as it's unnecessary; you can achieve the same ends by means of queries, whose result tables would be the basis of the interface objects used by users of different types, and for different purposes, without prejudicing the integrity of the database.  But they would all be using the same data from the same base tables.

Before embarking on the design of any database application of more than trivial import, it is important that the developer has a good understanding of the principles of the database relational model.  By applying those principles the database will be an accurate model of the underlying reality, and as such, will reliably behave in the same way as the reality being modelled.  As Richard Feynman says in the quotation below my signature, it's necessary to understand things before writing them down.  It cannot be pretended that the learning curve is shallow, but it's manageable.  There are no short cuts, however, the homework must be done.

Hello Jeroen,

Welcome to Microsoft Community.

To resolve the issue follow these steps:

  1. In Word 2010 click on the Review tab and click on the Word Count button on the Proofing group-of-options.

  2. Make sure to uncheck the option: Include textboxes, footnotes and endnotes.

  3. Close and reopen Word 2010.

Check if the issue resolves.

If the above does not work, then the issue might not just be related to the Word Count not working or malfunctioning. It could also be an issue with some Add-in in Word 2010. To check this restart Word in the Safe Mode. Follow these steps to open Word in the Safe Mode.

Follow these steps to launch Word 2010 in the Safe Mode:

  1. Close Word 2010 if open.

  2. Click on Start > Run OR Start > All Programs > Accessories > Run to open the Run box. In the Run box that opens type winword /safe (note there is a space between winword and /) and click on Ok.

This action will open Word 2010 in the Safe Mode. Check if the Word Count stops counting Word and allows you to work normally.

If it resolves the issue and Word 2010 works fine in safe mode, disable the Add-Ins for the application and check the result.

  • Click the File tab.
  • Go to Options.
  • Click Add-ins.
  • Select COM Add-Ins under the Manage drop-down box.

  • Try disabling the Add-ins – clear all the options and click on OK to save the changes made. Note: In case you see some Add-ins getting checked again after applying OK it is still fine. Ignore and proceed further.

After completing Com Add-Ins follow the same steps as above and check under Word Add-Ins to see if there are any Add-in there which needs to be disabled. If yes, proceed with the same steps as mentioned above.

Check if the above step helps you to resolve the issue and you are able to work normally in Word 2010 without the Word Count issue.

Note In case if it works briefly and the issue returns after a while then make sure to navigate back to Com Add-Ins and Word Add-Ins (following the steps as detailed above) and remove / un-install the Add-ins from the list.
