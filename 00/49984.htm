manuelcapiro say:
Hello SimonTytler,

When the USB flash drive does not show up in Windows 10system, it means that the computer fails to read the USB drive. You must make it readable again.

Update USB flash drive driver

1. Connect the not working USB flash drive to PC, right-click on This PC/My Computer and choose Properties.

2. Click "device manager" on the left taskbar > Expand USB Serial Bus controller, then you'll see a yellow exclamation mark, and right click on it and select "Update driver" or "update driver software".

3. Click "automatically search for driver software online"...
Wait patiently and let the driver to be installed and reboot your computer.
After doing so, you can try to reconnect your USB drive to PC and check whether the USB drive can be detected or not.

Use Disk Management and make computer recognize USB flash drive

1. Right-click on Start icon and select Search > Type control panel and hit Enter > Tab on Control Panel.
2. Click Administrative Tools > Select Computer Management.
3. When the window opens, click on Disk Management and then find the device in the drive viewer.
4. Right click on the USB flash drive > Select Format to directly format the USB drive to reusable again.


Sometimes, you may also try to reinstall the disk driver for USB drive when it fails to show up on your PC. This method works on any Windows systems - Windows 10etc. Now, let's see how this works now:

1. Right-click This PC/Computer > Properties > Devices Manager.
2. Click on Disk driver > Right-click on General Flash Disk USB Drive (your USB flash drive) and choose Uninstall.

3. Make sure the driver you'll uninstall is named "Flash Disk".
4. Click OK.
Wait patiently to let the process to be finished. If it takes too long, you can remove the USB flash drive from the PC.
5. Reconnect the USB flash drive to the PC and it will now reinstall new device software.

Then PC will read and recognize your USB drive. Then you'll see the USB drive shows up in Windows 10 and you can even view data inside the drive again.

Hope it helps, have a good day

  • net start bits

robertjwin say:

A clean install involves formatting or deleting the system partition, so the fact that your existing files were moved into Windows.old means you didn't perform an actual clean install.

Regardless, as you found the upgrade overwrote the original Windows.old with a new one. Technically when the Windows 10 Upgrade Assistant said that all your files weren't gone it wasn't wrong, because it was only talking about your existing (newly installed in your case) OS' files and not about anything already inside an old Windows.old. Windows.old is considered disposable by the OS and whether you copy your files out of it or not, the folder is auto-deleted after only 10 days (used to be 30 days earlier). So never rely on files in Windows.old remaining safe forever.

I assume you installed Win10 RTM or 1511? If you'd used a Win10 1607 ISO directly to install then you wouldn't have had to upgrade, but there's no use crying over spilt milk now. Since you must have confirmed after searching that your files are truly gone, the only thing you can do now is try various data recovery software. However don't hold your breath and expect 100% retrieval of your deleted files. Next time I'm sure you'll backup to an external drive before attempting something similar.

The security principle ALL APPLICATION PACKAGES has to have permissions to various registry hives.  In our case, we had modified the key HKEY_CLASSES_ROOT\CLSID (via GPO) by adding a permission (long before we even heard of Windows 8).  This caused the loss of the ALL APPLICATION PACKAGES permissions to that key.  Its permissions should be:

Type: Allow

Applies to: This key and subkeys

Advanced permissions: Query Value, Enumerate Subkeys, Notify and Read Control

 

By adding ALL APPLICATION PACKAGES back to the permissions in the GPO, everything started to work normally.

 

ALL APPLICATION PACKAGE should have these permissions to all hives. I haven't checked, but it may be blocked by default on certain keys within each hive. Plus, it should have Traverse folder/execute file, List folder/read data, Read attributes, Read extended attributes and Read permissions on the c:\program files, c:\program files (x86), users, and c:\windows folders.  The permissions are blocked someplace down the users folder, so make sure the folder C:\Users\<user name>\AppData\Local\Microsoft\Windows\WER has List folder/read data and Create folder/append data. 

 

I found our issue by creating a test OU in AD and blocking GPO inheritance to it.  I then made a copy of the GPO we knew was causing the issue and applied that copy to the test OU. Then I started deleting registry entries in the GPO until I found the one causing the whole issue. By the way, if you run your own tests this way, you should have a VM of a Win 8 machine and a snapshot of a working installation so you can easily roll back to a working condition after each test.
