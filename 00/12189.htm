Although it may be possible to extract the source code from the database
without opening it as the current database of an Access application, I don't
know how to do it.  However, you can use automation to open a second
instance of Access and then process the modules in it from your own
application  For example:
 '------ start of example code ------
   Dim app As Access.Application
   Dim mdl As Access.Module
    Set app = New Access.Application
    With app
       .OpenCurrentDatabase "C:\Path To\YourDB.mdb"
       .DoCmd.OpenModule "mdlWhatever"
       Set mdl = .Modules("mdlWhatever")
       Debug.Print mdl.CountOfLines, mdl.CountOfDeclarationLines
       .DoCmd.Close acModule, "mdlWhatever"
       .Quit
   End With
    Set app = Nothing
'------ end of example code ------
 Of course, that's not exactly what you're doing in your code above, but it
should point the way.

suzanne s. barnhill say:

Spacing Before is suppressed after a "soft" (automatic) page break but not after a manual page break, whether inserted via Ctrl+Enter or by applying "Page break before" formatting. Spacing Before is also honored after a section break and for the first paragraph in a document.

If the top paragraph on the page is too close to the header, then you need to add Spacing After to the header paragraph or increase the top margin (or reduce the header margin).

dirk goldgar say:

Are you user 'Admin', and is your machine 'xxxxx'?  Is this database on your own computer, or on some other computer on the local network?  The most likely scenario is that something unusual happened when you were in the database earlier, and Access didn't really close, though you may not see its window onscreen.

Close every instance of Access you can see.  Then open up Task Manager and see if msaccess.exe is listed among the running processes.  If it is, select it and choose "End Process".  Then see if you can delete the lock file, and after doing so, see if you can open the database.

If you don't see msaccess.exe among the running processes, and the database is on your own PC, try rebooting your computer.  After rebooting, you ought to be able to delete the lock file and open the database.

If the database is on some other computer, then probably *someone* has it open exclusively, and you need to find them and make them close it.  If you can't do that, but you have control of the machine that is hosting the database, rebooting that computer ought to terminate all connections.  However, if it's a server that has mutliple functions, that may not be an option.

What probably happened is one user shut their PC without closing the database. That left the locking file open causing your problems.
