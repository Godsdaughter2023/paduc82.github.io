john w. vinson say:

Short answer: the query as written isn't working because NULL (a blank control) is not equal to anything, even to another NULL. You can use a criterion on each field of

 [Forms]![FrmQryPrinc-Details-DT-A]![Type]  OR  [Forms]![FrmQryPrinc-Details-DT-A]![Type]  IS NULL

However, if you put this in all the criteria, Access will scramble the query grid into a real MESS. You can do it in SQL view by using a criterion such as

WHERE ([Type] =  [Forms]![FrmQryPrinc-Details-DT-A]![Type]  OR  [Forms]![FrmQryPrinc-Details-DT-A]![Type] IS NULL)
AND ([ThisField] =  [Forms]![FrmQryPrinc-Details-DT-A]![ThisField] OR  [Forms]![FrmQryPrinc-Details-DT-A]![ThisField] IS NULL)

AND ... 

,etc. through all the controls on the form. But opening this query in the grid will scramble it again!

Better is to use some VBA code on the form to poll through each control, building up a SQL string as you go. There are examples on the web for this technique - Bingoogle for "Access Query By Form".

One (very emphatic!!) suggestion - do NOT use punctuation such as hyphens, periods, slashes or blanks in the names of tables, fields, or controls. Access can and will get confused, and it can make your queries all but unreadable. Much better is to use "camel case", e.g.

FrmQryPrincDetailsDTA

or use underscore characters

FrmQryPrinc_Details_DT_A

ken sheridan say:
The most reasonable format seems to be:

IIf(IsNull([Forms]![FrmQryPrinc-Details-DT-A]![Contract]), “*”, [Forms]![FrmQryPrinc-Details-DT-A]![Contract])


To elaborate a little on what John has said, don't do as above; as you've found out it will fail in the event of a Null in the column.  Use the following in the criteria row of the column in query design view::

[Forms]![FrmQryPrinc-Details-DT-A]![Contract] OR [Forms]![FrmQryPrinc-Details-DT-A]![Contract] IS NULL

This translates to the following SQL:

WHERE Contract = ([Forms]![FrmQryPrinc-Details-DT-A]![Contract] OR [Forms]![FrmQryPrinc-Details-DT-A]![Contract])

This approach will work with any data type. When using this approach do not declare the parameters, however.  Declaring a parameter means it cannot evaluate to Null.  The exception is a parameter of DateTime data type, which is always best declared.

If you do this one thing you'll find is that if you save the query in design view, when you re-open it in design view Access will have moved things around.  With a simple query of few parameters this won't matter, but with a complex query it could result in it being too complex to open.  For this reason it's better to write and save such queries in SQL view as this preserves the simple logic.  Here's an example from one of my on-line demos:

PARAMETERS Forms!frmMultiFieldDlg!txtDateFrom DATETIME,

  Forms!frmMultiFieldDlg!txtDateTo DATETIME;

SELECT Employees.*, FirstName & " " & Lastname AS FullName, Projects.Project

FROM Projects RIGHT JOIN (Employees LEFT JOIN ProjectEmployees

ON Employees.EmployeeID=ProjectEmployees.EmployeeID)

ON Projects.ProjectID=ProjectEmployees.ProjectID

WHERE (City=Forms!frmMultiFieldDlg!cboCity

  OR Forms!frmMultiFieldDlg!cboCity IS NULL)

AND (Department=Forms!frmMultiFieldDlg!cboDepartment

  OR Forms!frmMultiFieldDlg!cboDepartment IS NULL)

AND (DateAppointed>=Forms!frmMultiFieldDlg!txtDateFrom

  OR Forms!frmMultiFieldDlg!txtDateFrom IS NULL)

AND (DateAppointed<Forms!frmMultiFieldDlg!txtDateTo+1

  OR Forms!frmMultiFieldDlg!txtDateTo IS NULL);

Note how each OR operation is parenthesized to force it to evaluate independently of the AND operations, and how the DateTime parameters are declared, but not the others.

john w. vinson say:

In this case I'd follow Ken's advice. (Heck, I've been following Ken's advice and learning from him since back in the last century!)

Queries ARE SQL. The query grid is a very handy tool to make it easier to build SQL. But it's a limited tool; you could build Ken's query in the grid, by putting the criteria for each field on a different row - but Access will be forced to create as many new fields in the grid as you have criteria, just to accommodate the limitations of the grid. The query itself - the SQL - doesn't need all those extra fields; they're not needed to run the query, just for the grid display.

Don't be scared of SQL. People have learned to translate Sumerian cuneiform and Mycenean Linear B; SQL is much easier... and even comes with documentation! :-{)

Sub FillValues()
    Dim Ws As Worksheet
    Dim i As Long, k As Long, l As Long
    Dim Cell As Range, Rng As Range
    Set Ws = Worksheets("Sheet1")
    Set Rng = Ws.Range("A1:A" & Ws.Range("A" & Rows.Count).End(xlUp).Row)
    For Each Cell In Rng
        i = i + 1
        For k = 1 To 3
            l = l + 1
            Ws.Cells(l, 4) = Ws.Cells(i, k)
        Next k
    Next Cell
    Ws.Columns("A:C").Clear
    Ws.Columns("D").Copy Ws.Range("A1")
    Ws.Columns("D").Clear
End Sub
