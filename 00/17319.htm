jay freedman say:
None of the characters shown in that chart as letters surrounded by a dashed box actually have printable glyphs. In normal text they're all invisible, if they can be entered at all (most of them are 'control characters', such as the ESC that corresponds to the Esc key).

One thing you could do is to take a screen shot of the box from that PDF, using either the Windows 7 Snipping Tool or the Word 2010 Screenshot tool (on the Insert tab). Save that graphic as a building block, and then use the Replace function to replace actual space characters with the graphic.

Of course, if you do that with Replace All, it will make your entire document act like one very long word. The line breaks will be ... broken. :-(

dhookom say: Think of a "slot" as a new record in a subform. If you need to add another ingredient to a recipe, you add a record to a RecipeIngredient table.

tblRecipes
==================
recRecID autonumber primary key
recTitle values like "mom's meatloaf" or "tuna hotdish"
recRCaID link to tblRecipeCategories.rcaRCaID


tblIngredients
=====================
ingIngID autonumber primary key
ingTitle values like "tuna", "ground beef"

tblRecipeCategories
=======================
rcaRCaID autonumber primary key
rcaTitle values like "desserts", "Drinks", "Vegetables"

tblRecipeIngredients
=============================
rinRInID autonumber primary key
rinRecID link to tblRecipes.recRecID
rinIngID link to tblingredients.ingIngID
rinQty quantity like 1, 5 , 4.3
rinUOM unit of measure like "cup" "teaspoon"
rinComment

You would create a main form for tblRecipes with a continuous subform to add/edit tblRecipeIngredient records.

You could create a table for units of measure which would make data entry easier and more consistent.

If you want to track inventory, you would need to have a transaction table for purchases and usage but that's a lot more work.

scottgem say:
Hi Adam,

The foundation of a well working database is a proper normalized table structure. Normalization is a set of rules for designing relational databases.

First, thing, you need to have an ingredient in your table BEFORE you can select it for use in a recipe. This can be done on the fly using the NotinList even of a combobox. But you have to have the table setup first.

So you need a Recipe table and an Ingredients table and a recipe details table. at the least. The Recipe table hols the specific info about the recipe itself, Name, prep instructions, etc. Then the Recipe-Ingredients table lists each ingredient that goes into the recipe with amounts. The Ingredients table lists all the ingredients you may use in a recipe.

Inventory is not stored, its calculated. You will need a table that lists all your stock movements. When you make a recipe, you will add a record to this table listing the ingredients and the amount used. When you purchase ingredients, you add records to the same table indicating the amounts purchased.

Hope this helps,

The Microsoft Print to PDF is such a print driver. Many low-end HP printers also use such version 4 print drivers. PostScript printer drivers are typically version 3 Windows printer drivers and don't suffer from this problem.

We are expecting a fix from Microsoft sometime in the next month.

rich michaels - mvp (greatcirclelearning.com) say:

If you're drawing it on a paragraph there isn't a left or top setting and that due to the range being inline.

A method you might try is to insert a one cell table then select the the table cell, collapse the range, and draw the shape there. Your left and top settings would be "0" in this case. The command would look like this:

  Set rng = ActiveDocument.Tables(1).Cell(1, 2).Range
  rng.Collapse wdCollapseStart
  Set shp = ActiveDocument.Shapes.AddShape(92, 0, 0, 34, 34, rng)
