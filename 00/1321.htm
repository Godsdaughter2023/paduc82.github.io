Hi, sometime ago I found this solution to check who open an excel file see if it helps you

As romperstomper says, this is reliant on macros being enabled on opening
Code:
Option Explicit
Private Sub Workbook_Open()
    Dim NextRow As Long
    Dim wsLog As Worksheet
    Set wsLog = Sheets("LogRecord")
    With wsLog
        NextRow = .Range("A" & Rows.Count).End(xlUp).Row + 1
        .Range("A" & NextRow) = Date
        .Range("B" & NextRow) = Time
        .Range("C" & NextRow) = Environ("USERNAME")
        .Range("D" & NextRow) = Environ("COMPUTERNAME")
        .Visible = xlSheetVeryHidden
    End With
End Sub
To view the sheet enter this in the Immediate pane of the VBa editor

Sheets("LogRecord").Visible = xlSheetVisible
Press Return
When you have finished viewing the log either enter this in the Immediate pane
Code:
Sheets("LogRecord").Visible = xlSheetVeryHidden
Press Return
or just close the file.

rick rothstein, mvp - excel say:

The key is to use error trapping. Andreas showed you one way to structure its use, here two other ways....

Method 1
==========================
Sub YourMacro()
  Dim R As Range
  ......
  ......
  On Error Resume Next
  Set R = Application.InputBox("???", Type:=8)
  On Error GoTo 0
  If R Is Nothing Then Exit Sub
  '
  '  Your code goes here
  '
End Sub

Method 2
===========================
Sub YourMacro()
  Dim R As Range
  On Error GoTo NoRangeInutted
  Set R = Application.InputBox("???", Type:=8)
  MsgBox "OK!"
  '
  '  Your code goes here
  '
  Exit Sub
NoRangeInutted:
  '
  '  Put any clean up code needed here
  '
  MsgBox "Oops!"
End Sub

Note that I showed the error trapping in a suggested macro (no arguments in the Sub), but it can be in use in a full subroutine or function as well.

I am creating an address book in Word, using addresses from Excel as a mail merge. I need several entries per page. How do I "insert new record"


If you use a Directory type mail merge main document, and just set up one entry the way you want it (I would suggest that you set up the fields in the cell of a one row one column table, with the row set so that it does not break across pages, then execute that merge to a new document, you should get what you are after.  The new document will contain a table with a row of data for each record in the data source.

hansv mvp say:

Create code like this:

Sub DropDownChange()
    Select Case ActiveDocument.FormFields("DropDown1").Result
        Case "This"
            Call DoThis
        Case "That"
            Call DoThat
        Case "Other"
            Call DoOther
        Case Else
    End Select
End Sub
 
Sub DoThis()
   ' Action for 'This'
    ActiveDocument.FormFields("Text1").Result = "This"
End Sub
 
Sub DoThat()
   ' Action for 'That'
    MsgBox "You selected 'That'", vbInformation
End Sub
 
Sub DoOther()
   ' Action for 'Other'
    Beep
End Sub

You'll have to modify the names, options and macros of course.

While the document is unprotected, double-click the dropdown, and assign the DropDownChange macro to be executed when you exit the dropdown.

ken sheridan say: It looks like you do have a multi-valued field, though why you experience the error is unclear.  These type of fields are best avoided unless there is real need for them, which would only be the case if the database is interfacing with SharePoint in the very specific context in which they are required.  Otherwise model the many-to-many relationship type in the conventional way by a third table which resolves it into two one-to-many relationship types.

You said that you are logging applications, so let's assume each application can relate to one or more counties.  That's a binary (2-way) many-to-many relationship type between applications and counties, so would be modelled with, in broad outline, tables like this:

Applications
....ApplicationID  (PK)
....ApplicationDate
....etc

Counties
....CountyID  (PK)
....County

and to model the relationship type:

ApplicationCounties
....ApplicationID  (FK)
....CountyID  (FK)

The primary key of the last table is a composite one comprising both foreign key columns.  So diagrammatically the model is:

Applications----<ApplicationCounties>----Counties

in which the two relationships between Applications and ApplicationCounties, and between Counties and ApplicationCounties should both have referential integrity enforced.

You'll find an example of this sort of basic many-to-many relationship type as StudentCourses.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file various means of interfacing with the data are illustrated, but I'd strongly recommend the conventional one of a form/subform, which, unlike the others, is very simple to implement, being code free.
