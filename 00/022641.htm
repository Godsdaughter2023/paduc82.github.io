ken sheridan say:


I'm just sort of working my way through Access via trial and error.


Here are a few errors to avoid:

1.  Do not use the generic ID as a column name.

2.  Do not use spaces or other special characters in table or column names.  Use CamelCase or represent a space by an underscore character like_this.

3.  Do not use words like 'Name' as a column name.  This is the name of a built in property in Access, so as a 'reserved' word should not be used as an object name.  The same goes for things like Date and the like.  Use explicitly descriptive terms like ProductID, CategoryID, TransactionDate etc

I know you'll find at least the first two of these done in Northwind, but Northwind is full of bad relational database design features and is a poor example of good practice.

So your tables should be set up along these lines:

Products
....ProductID  (PK)
....ProductName  (indexed uniquely)
....CategoryID  (FK - indexed non-uniquely)

Categories
....CategoryID  (PK)
....Category  (indexed uniquely)

PK = Primary Key; FK = Foreign Key.

Create a relationship between the tables on CategoryID and enforce referential integrity.

I'm sure Scott's blog will give you excellent examples of how to show the category for the product when you select a product in form based on a referencing table.

When naming objects I follow the convention, as far as possible, of naming tables with plural or collective nouns to reflect the fact that a table is a set, and naming columns with singular nouns to reflect the fact that a column represents an attribute of the entity type which the table is modelling.  This makes a real difference to the ease of writing and reading complex queries, which one day you'll be doing; in fact there has been psychometric research on computer science students which demonstrates this.

Re: How do I fix corrupted installation database

I ran into the same problem on one of our computers trying to install the Office 2007 Service Pack 2 issued April 28, 2009.

The catch is that you've got to go into the registry at HKEY_CLASSES_ROOT\Installer\Products\ and rename the patch key to patch_old on every entry for an Office 2007 application - Excel, OneNote, PowerPoint, Expression Web, etc. The number of entries you'll have to change depends on which Office 2007 programs are installed. The entries to change include those for:

  • Microsoft Office Shared Setup Metadata MUI (English) 2007
  • Microsoft Software Update for Web Folders (English) 2007
  • Microsoft Office Excel MUI (English) 2007
  • etc. -- everything involved in Office 2007.

I would strongly suggest that before making these edits, that you export the registry to your desktop as a safety valve and run a system restore from the Control Panel.

After the install works, be sure to reboot. I suspect that you can then delete the patch_old keys, but I am not certain (and after losing so much time to finding this solution, I'm not ready to be a guinea pig right now).

Force at least one record into subform - Microsoft Community

tim shep say:

Force at least one record into subform

I have a main form with a continuous subform.

I would like to ensure that at least one record is entered in the subform and I am unsure how to accomplish this.

Thanks, Tim

marshall barton say:

I don't think that's feasible.  There is no way I know of to prevent users from just navigating to another main form record without entering a subform record.  Without going through all kinds of contortions, you can not prevent a mainform record from being saved before a subform record can be created because the focus can not move to the subform until the mainform record is saved.

If your main form is guaranteed to be filtered to a single record, then you could use the main form's Unload event to check for a subform record and cancel the unload so the form remains open.  But that is seriously clumsy and may motivate users to resort to drastic measures like killing Access, which will leave the mainform record with no subform record.

Besides, why prevent users from going on to another task and coming back later to add a subform record.  It's easy enough to check an existing mainform record to see if it does or does not have a subform record and pop up a warning message with code like this in the mainform's Current event:

   If Me.subformcontrol.Form.Recordset.RecordCount = 0 Then
      MsgBox "Please enter a subform record"
   End If

You may also want to consider creating an audit form that displays all main records that do not have a subform record.  This audit form would simply be based on a FindUnmatched query.
