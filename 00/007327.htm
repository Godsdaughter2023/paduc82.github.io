albert d. kallal say:

When you say "navigation form", are you using the term in the general context that you have some forms, or are you using the Access 2010 navigation form?

 

All forms in the new Access navigation system are in fact just sub forms. (you could have coded this in previous versions by hand – but now Access does most the work for you).

 

Because the navigation system is in fact based on sub forms, if you are NOT displaying another form then that form is NOT loaded, and thus you cannot require that form. However, if you have a form with some tabs, or some other setup and the other form IS loaded, then the standard syntax to re-query that sub form should work.

 

Eg this:

   Me.custChild.Requery

 

In above, this would be code in the current form, and custChild would be the reference to the sub form – and this could even a sub form on a different tab.

 

In general, since with the new navigation system forms behind other tabs are NOT loaded, then when you switch to that tab, the form IS re-loaded, and in general no requery is required.

 

If you are in fact using tabs (as opposed to a navigation form), then as noted, you will have to re-query  such forms, and a standard reference as per above with a requery command should work.

So, in general you should not need a requery command if you are using the new navigation system (because forms you cannot see are NOT loaded at that point in time). With the navigation system EACH change of form closes the other form and the one you see is loaded (so forms are not all loaded – JUST the one you are looking at and thus you cannot re-query other forms and you should not need to).

 

You might have to "expand" on which case you are using here (navigation form, or a tab with a sub form).

Like I said, are the tabs or forms that don't requery behind an Access navigation form, or are they behind a tab?

And it rather hard just looking at a screen shot if the tab is in fact a tab control or as noted a navigation form (they look very similar – hard to tell apart in a screen shot).

 

As noted, the way to re-query a sub form is to simply use INSIDE of that given form this VBA syntax:

 

Me.MySubFormControlName.Requery

 

Note that the above WILL work if the parent form is dropped into a navigation form set. (but again as noted, we on a wild goose chase if you cannot clear up if we are talking bout tabs on a navigation form as opposed to tabs on a tab control).

 

And as noted, in general you should not need to requery a sub form, but the above VBA command WILL requery the form in question if that is your goal + question.

 

You can consider using a fully qualified forms path name in place of "me" as per above, but you want to avoid that type of code/syntax if possible.

 

As noted, you can requery the navigation  sub form, but WHERE and WHEN the code resides that you attempting to run will affect the syntax you use.

 

From the main form to requery a form inside of the navigation form set, you would use this:

 

   Me.NavigationSubform.Requery

 

But, in fact even more correct syntax would be this:

 

   Me.NavigationSubform.Form.Requery

 

And if it was a sub sub form, then you would have this:

 

   Me.NavigationSubform.Form.custChild.Form.Requery

 

However, it would be RARE to run such code from the main navigation form. And as noted if the code resides in the form that is part of the navigation set, then this code will work:

Me.custChild.Form.Requery


And last but  not least:

If the form in question is on A DIFFERENT navigation tab, then you cannot re-query such forms since when using the navigation system.

 

Why?

Because with the navigation form system those other forms behind tabs are NOT YET loaded. What occurs is when you click on a different navigation tab, the current form is UN-LOADED and the new form is THEN loaded. So keep in mind that previous forms are also automatic  un-loaded. This means you cannot reference controls on those PREVIOUS FORMS when using a navigation form as opposed to a tab control.

 

So this is why I stated it is VERY significant to distinguish between a tab control and navigation forms.

Without such distinctions between tabs and navigation forms, then any information provided here is going is unlikely to be of any use to you.

jasles say:

OK - that's a terrible template that Microsoft provided. There's so much that needs to be done to make it work!

To start - only the January tab has all of the defined names added. To use every other month's tab you would have to add defined names referring to the cells in row2 on each worksheet that Microsoft provided. On top of that, you would have to add defined names to each worksheet (January to December) for the cells that you want to add to row 2.

Then, you would need to change the conditional formatting on each cell in each worksheet to change to the correct color based on what you typed.

For the employee names worksheet, the data validation source is only cells B4 to B8. You would need to change the source to continue further down the column.

Also, the data validation for each row on each month's sheet for the names would need to be changed to refer to the same cells as the employee names sheet.

If you want to use this template, I'd delete all month tabs, except for January, make the changes to January tab and then duplicate the January tab for each month. I'd get rid of all the defined names and just refer to the cells also.

In all, it's a fair chunk of work.
