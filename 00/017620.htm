jim gordon mvp say:

I am not sure exactly which file formats you mean by "native Office file formats."

Since 2007, the default file format for word processing, spreadsheet and presentation programs was changed from proprietary binary (.doc .xls .ppt) to open text (.docx .xlsx .pptx). The old binary formats could contain VBA code. The current file formats are absolutely safe from visual basic macros because it is impossible to include a macro in any of these formats. You do not have to worry about VBA macros when you stick to the default file format.

Even if you make a VBA macro and try to save your file in the default format, the macro will be stripped away during the save process.

Unfortunately, the new open file format is exactly that - open. Anyone with a text editor can open these files and muck around with their contents and structure undetected unless the files are encrypted using a digital rights management (DRM) server. Most people don't have access to a DRM server. However, even a hacked document still will never have VBA in it, so if you are worried about the default file formats ever having VBA code in them you are wasting your time worrying about something that is impossible. The default file format does support Javascript, which in my opinion is far more dangerous than VBA.

Files having extensions ending in the letter m signify that those files probably contain macros. These are macro-enabled files or macro enabled templates having VBA and sometimes AppleScript. Excel is the only Microsoft application that still has a binary file format option (.xlsb) and this format does support macros. 

Files saved as add-ins in Microsoft Office are visual basic programs (aka applications). These contain VB code and can also have AppleScript and/or Javascript.

It sounds like you want to have your cake and eat it, too. Microsoft Office is extensible using VBA, Applescript and Javascript. These technologies can be employed to create extremely valuable customized solutions and automation. Yes, there is inherent risk in using a product that is as powerful as Microsoft Office. There is risk in opening web pages and even pictures. If you can't resist clicking every ad or opening every file without considering its source, then maybe you should not be using computers or smart phones. A certain level of education and responsibility is required if you want to take advantage of the amazing capabilities of these devices.

You can limit your exposure to risk by choosing applications that do not support programmability and are not scriptable. Even TextEdit and iWorks are scriptable, so they are not risk-free. To view the AppleScript dictionary for any application, drag its icon onto the Script Editor app in Applications > Utilities. The Microsoft Office and Google web applications are scriptable, so they are not safe either. Web pages can be made malicious. Adobe has its own scripting language. I can't think of a single productivity application that has any usefulness at all that is not able to be scripted.

wayne-i-m say:

You don't use the NZ function in a table, you use it in a query, form or report.

If it's a number field set the default to 0.
Or you could use an update query to change the null data


But then you're storing data with no need.  You can always run the Nz function so there really is no need to store a 0 in most cases - but you can if you want

Nz ([FieldName], 0)
Or 
IIf (IsNull ([FieldName]) , 0)


john w. vinson say:

That's correct. You can't use ANY function as a calculated field in a table.

As Wayne says, it's not necessary to do so; you can always use NZ in a query, and then use that query instead of the table. If you do need to replace nulls with zeros in the table, you can run an Update query; use a criterion of 

IS NULL

on the field and update it to 0. If you then change the field properties so that the Required property is Yes and the DefaultValue is 0, you won't get any more nulls to worry about.

hansv mvp say:

When the problem appears:

  • Switch to design view.
  • Click in an empty part of the upper pane of the query window.
  • Look at the Property Sheet.
  • Do you see the field name in one of the properties, for example in the Filter or Order By property? If so, clear that property.

Another option:

  • Switch to SQL view.
  • Select and copy the SQL statement.
  • On the Create tab of the ribbon, click Query Design.
  • Close the Show Table dialog without adding any tables.
  • Switch to SQL view.
  • Paste the SQL statement.
  • Check whether the new query works.
  • If so, discard the old one and continue working with the new one.
