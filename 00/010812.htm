What we've discovered so far:
- Using updated mscomctl.ocx (6.1.98.34) on development machine, decompile and compact source database, then distribute new version, essentially recompiling to the new mscomctl.ocx.  This works on some, but not all machines (possibly Win7 and/or those with Office 2010 installed) that have the new version of this file already.
- On one machine (XP, no Office installed), it had the newest version of mscomctl.ocx but the recompiled version didn't fix it.  We copied an older version of the file and registered it, and then copied the newer version of the file and registered it -- probably doing what Nathan O. suggested above (unregistering/registering).

We're still trying to get a grip on the problem and develop an automated fix so we don't have to remote in to every single installation, but this is sure a big headache to deal with, and we haven't found fixes for some clients yet, so still looking for more things to try with them.
