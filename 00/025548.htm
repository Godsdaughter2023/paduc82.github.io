beth melton mvp say:

When Style Sets were first introduced, was told (by MS) that, as you observed, the styles in the Style Gallery are those saved in the Style Set. One piece that I didn't see mentioned in this thread is the Document Defaults are also saved and when a Style Set is applied, the Document Defaults are also updated. This is the primary difference between using Style Sets and the Organizer. Actually, that may be the only difference. 

Here's what happens: When you apply a new Style Set, all styles in the Style Set file with the attribute of tw:qFormat="1" in styles.xml along with the Document Defaults are "copied" to the destination document. (They aren't really copied, the style properties in the destination document are updated in the xml.) If a style isn't part of the Style Set, it remains untouched when a new Style Set is applied. With that in mind, any custom styles that aren't defined in the Style Set will not be updated and remain in the Styles Gallery, provided the style includes that attribute and if it's in-use in the document. It will, however, inherit formatting from the Document Defaults.

If there are discrepancies when applying a Style Set to other documents, it's likely due to the Theme that's in use in the destination document and whether the styles were formatted using theme elements, such as Heading and Body fonts, theme colors, etc. rather than absolute formatting.

Another discrepancy possibility is there is a conflict between the Normal style and the Document Defaults. This shouldn't happen, but I've encountered it a couple times. As with most Word features that pertain to formatting, it's recommended the Normal style remain an empty style (especially in Style Sets) in order to avoid the table style bug and other style hierarchy issues.

For those not familiar with how the Normal style works in Word 2007 and up, when the file format changed to xml in Word 2007, the Document Defaults were no longer hard-coded and could be modified. (Prior to Word 2007, they were hard-coded in every document as TNR, 10pts, single space, etc.)  In all versions, the Normal style is empty by default. IOW, by default, there is no formatting defined for paragraphs or characters for the Normal style and all formatting is inherited from the Document Defaults.

Now that we can modify the Document Defaults, there's no reason to modify the Normal style and it should remain empty in order to avoid the issues previously described. The reason for those issues is due to order in which Word applies formatting: first it applies formats defined in the Document Defaults, then Table Styles, then Paragraph Styles, and then Character Styles. Since the Normal style is a Paragraph Style, when you're working with tables its formats are applied 3rd, which is why hierarchy issues could be encountered if it's not an empty style.

Why Excel VBA is a fools' language

John Franco *** Email address is removed for privacy ***

Today you will get a complete explanation of the fundamentals of the VBA command structure. The majority of professionals out there don't get this right so don't be one of them.

Pay attention to this short VBA story. You don't have to remember anything, once you understand.

Commanding a fool

Imagine you are out of office and need to change a report slightly and resend it to your manager. You know Joe is there, so you call it and ask him a favor. If you want to just change the content of a cell of your report to "Aug report", you would say something like, "Joe, please change the cell A1 to "Aug report" and then email the book to *** Email address is removed for privacy ***". But Joe would immediately ask, "the cell A1 from which book and from which sheet in that book?"

You must be specific and convey the full command "Joe, please open the book "Aug sales.xlsx", change the cell A1 of the sheet "Aug" to "Aug report", and then email the book to *** Email address is removed for privacy ***".

You must be specific. And you are talking to a person. A person is intelligent, but a machine not. So, to command a machine, you've got to even more specific.

Excel is a controllable fool

The good news is that Excel is controllable. In fact, you control Excel every day. Well, you control the Excel objects.

You are already familiar with Excel objects (cells, charts, pivot tables, sheets, etc). You either change an object's property, for example, you change the cell height to 20, you set the book name to "Aug report", etc. Or you do something with an object, for example, you open a book, you update a pivot table, etc. (well, it's the object that does it).

But you are not familiar with instructing Excel (a machine) to do the work for you using VBA. You've got become empathetic with the machine and be very specific.

The command "change the value of that object property to VALUE" is written in VBA this way OBJECTEXPRESSION.Property = VALUE

The command "object X, do Y" is written in VBA this way OBJECTEXPRESSION.Method (Arguments)

The root "OBJECTEXPRESSION" above represents an Excel object. Because there are many objects, you've got to specify which one you are referring to. You need to specify which book from the books, which sheet from the sheets in the book, which cell from the cells in that sheet.

Here's the step by step guidance to write the full VBA syntax of a command. You can use such commands to automate your formatting, data collection, or as part of bigger applications that automate the report creation every week, etc.

1) Unambiguously specify the object you want to control using the Excel object hierarchy

Specify which object you refer to. But you cannot point to objects if you don't know where they belong. E.g. A cell is contained in a sheet. A sheet contains many cells. A sheet is contained in a book. A book can contain many sheets. A chart is contained in a sheet. A sheet can contain many charts.

Here are some examples that refer to different A1 cells (the root is in bold font):

  • Range("A1").Value = “Hello”
  • Workbooks(1).Worksheets(1).Range("A1").Value = “Hello”
  • Workbooks(3).Worksheets(1).Range("A1").Value = “Hello”

Here are some examples that refer to different charts in different books and sheets:

  • ActiveChart.Legend.Position = xlTop 'sets the legend position
  • Workbooks(3).ActiveChart.Legend.Position = xlTop 'sets the legend position
  • ActiveSheet.ChartObjects("Chart 1").Chart.Legend.Position = xlTop

Here are some examples that select different ranges:

  • Selection.CurrentRegion.Select
  • Range("A1:C20").Select

Here's how to delete columns on different sheets and books:

  • Columns("H:H").Delete Shift:=xlToLeft
  • Workbooks(3).Worksheets(1).Columns("H:H").Delete Shift:=xlToLeft
  • ActiveSheet.Columns("H:H").Delete Shift:=xlToLeft

After you have provided the Excel VBA object you are referring to, it’s time to:

2) Unambiguously specify what you want to do with that Excel VBA object you specified (set the property/method name)

Just type the name of property/method you want to invoke (in bold below).

  • Range("A1").Value = “Hello”
  • ActiveSheet.Columns("H:H").Delete Shift:=xlToLeft
  • Workbooks(1).SelectedSheets.PrintOut Copies:=1,
    Collate:=True,  IgnorePrintAreas:=False

Where do you find the property/method name? Use the Excel VBA help. Each object has a list of members: properties, methods, and events (not coded directly in macros). You will see a section called: Workbook Object Members, Windows Object Members, etc.

At this stage, you have pointed to the right object, typed the desired property or method, now it’s time to:

3) Unambiguously specify how that Excel object performs the action you just set (set the Excel VBA object arguments) or what's the new value for the object's property you specified

You can't plainly order someone to paint. You must specify what to paint and what color. The verb Paint has its own arguments. "Paint" is incomplete. "Paint the wall yellow" is complete.

Each object action has its own arguments.

How do you provide the rest of the syntax (arguments)? Use the Excel VBA help and provide the arguments as needed.
 
The following code prints 1 copy of the selected sheets:
 
expression.PrintOut(From, To, Copies, Preview, ActivePrinter,
PrintToFile, Collate, PrToFileName, IgnorePrintAreas)
 
Where expression is a variable that represents a Workbook object.

Workbooks(1).SelectedSheets.PrintOut Copies:=1,
Collate:=True,  IgnorePrintAreas:=False
 
The following code moves the given shape to the left.
 
expression.IncrementLeft(Increment)
 
Where expression is a variable that represents a Shape object.

ActiveSheet.Shapes("Chart 1").IncrementLeft -49.5

If you are changing an object's property, you need to specify its new value. An object expects a specific kind of value for each of its properties. For example, the Height property of the Range object cannot receive a text.

The following code sets the cell A1 value to "Hello".

Range("A1").Value = “Hello”
 
Now you can easily write and understand the full syntax of Excel VBA objects.
