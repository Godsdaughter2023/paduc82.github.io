ken sheridan say: Although you've reached a solution it might be helpful to review the issue in a little more detail as there are one or two points arising from your post which have not been addressed:

The only real reason for basing the form on a query, as you seem to have done originally, would be if you wanted the rows (records)  in the form ordered by some value.  As columns (fields) in the Link table contain arbitrary numeric values, with no semantic significance whatsoever, then there would be little point in using a query on that table alone.  You might as well make the RecordSource property of the form the name of the table itself.

If you do want the rows ordered then you'd have to include at least one of the referenced tables in the query.  Say you want the rows ordered by item name, the query for use as the form's RecordSource property would be:

SELECT Link.*
FROM Link INNER JOIN Item
ON Link.linkItemID  = Item.itemID
ORDER BY itemName;

The combo boxes in the form would, as you appear to understand, have the hidden key column as the bound column, so if we take the combo box for items as an example it would be set up as follows:

ControlSource:    linkItemID  

RowSource:     SELECT itemID, itemName FROM Item ORDER BY itemName ;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.  You do not need to include a value for the second column's dimension.  It's only necessary to include further column width values if more than one column in addition to the key column is returned, in which case you also need to set the ListWidth property to the sum of the ColumnWidths dimensions, e.g. with a ColumnWidths property of 0cm;2cm;3cm the LisatWidth property would be 5cm.

The other two combo boxes would be similarly set up.

If you do use wizards for tasks like this you should try and understand just what the wizard is doing.  Wizards should be used to cut out the drudgery, not as a substitute for understanding.  Hand crafting combo boxes like this is such a  simple procedure, however, that I can't see any need to use the wizard at all.

One further point about the Link table is that there is almost certainly no good reason to have an autonumber primary key column. The other three columns constitute a candidate key, so can be made the composite primary key of the table.  If you do have a separate 'surrogate' key, however, then, it is imperative that the other three columns, as a candidate key, be included in a single unique index.  Otherwise there is nothing to prevent invalid duplication of data.

Finally, when using combo boxes like this you may need to enter a new value not yet represented by a row in the referenced table, e.g. a new item.  For this you can put code in the combo box's NotInList event procedure which enables you to type the new item name directly into the combo box for a new row to be inserted into the Item table.  You'll find a examples of the use of the NotInList event procedure in various contexts in the demo file NotInList.zip in my public databases folder at:

how to repeat action in word

in Word for mac, if I want to repeat any action, how to do this, is there is an icon I can use to repeat what ever am doing, for example if I insert a row, and i want to repeat inserting how to do this. thanks
john mcghie say:
You're looking for "F4".  This is special to Word: it repeats all keystrokes since the most recent mouse-click.

So you can type a whole paragraph, cursor back, capitalise a word, bold a word, go to the end of the paragraph, hit Enter -- then hit F4 and you will get another paragraph just like it.

As Bob says, it's keystrokes only: as soon as you touch the mouse you end the run.  You cannot use it to repeat the result of a mouse action.

Hope this helps

bob jones aka: cybertaz say:
Why Phillip confused the issue by by interjecting the Undo command into the conversation I don't have any idea. What you're looking for isn't "different" at all except that you don't want [or need] to use Command+Z [Undo] before you use Command+Y :-) That would be pointless.

What does differ is the circumstance at the time the Command+Y keystroke is used:

  • If the last action taken was to Undo [Command+Z] ReDo that action -- IOW, it reverses the Undo, however
  • If the last action taken was something other than Undo, Command+Y repeats that action [but I believe there are some actions which cannot be repeated using that keystroke].

For many actions, though, you don't need a special Repeat keystroke. You can just use the same keystroke again. Eg., If you paste content then want to paste it again just use Command+V again. It will continue to paste the last content copied/cut.
