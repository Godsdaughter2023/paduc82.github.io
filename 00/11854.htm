talking moose say:

Shouldn't be a sandbox issue. Scripts the user invokes himself (i.e. by double-clicking or running in the Script Editor) should have access to any app that's sandboxed. What I've found is scripts invoked by an application (i.e. an AppleScript set to run when triggered by a rule) won't be able to control other applications.

For example, this works:

End-user double-clicks an AppleScript applet or runs a script in Script Editor. The script can open Outlook, copy the content of a message to the clipboard, open TextEdit and paste the content to a new window in TextEdit.

This won't work, though, due to sandboxing:

End-user creates a rule triggered by receiving a message. Outlook can copy the content of the message to the clipboard and instruct the Finder to open TextEdit, however, it cannot tell TextEdit to paste the content into a new document.

Note that pasting into a new Word 2011 document still works because Word is not sandboxed.

Usually I've used "current messages" only in scripts automatically triggered by a rule and I've used "selection" when I'm manually running the script. For consistent behavior, I'd stick with "selection" when manually triggering a script and/or manually selecting messages.

steve rindsberg say:

That's one way of looking at it.  Another is like the difference between leasing and buying a car. 

You buy the car (or in this case Office) and you know that you'll have the same features for as long as you want to keep the car.  If something newer and shinier comes along, you have to decide whether it's worth buying a new car to upgrade. 

With a lease, you get the opportunity to upgrade more frequently.

If you want a new car/Office every couple years, you may be better off leasing than buying. If you prefer to drive the same car/Office for years and years, then buying is probably the most economical option.

For example, $99/year buys (well, rents) you an Office 365 subscription.

The approximate one-time purchase would cost $230 or so.

It's easy to make a case for either version, depending on your upgrade habits.

If you have multiple computers, the subscription version starts looking more attractive, since it allows you to install to up to 5 computers (Windows or Mac) AND up to five other devices like smartphones/tablets.  On the other hand, it wants to call the MotherShip at least once a month or so and if it can't chat with MS, may go demo-restricted on you. 

The purchased, not leased, version doesn't have that potential problem. And for some users, stability is a real plus.  A company with hundreds or thousands of users isn't going to want to keep up with supporting every new feature or change that comes down the upgrade pipe to subscription versions. They'll want to stick to the one-time purchase and live without all the shiny new interim features that come out.

Having the choice is welcome, I think.  What's not so good is that the usual consumer doesn't get the whole story up front; the one-time purchaser has no good way of knowing that he/she won't have access to the cool new toys that the others are getting with their rental-ware.

sven johannsen say:

Hard to tell if you are trying to have your phone use your Mac's Internet connection or the Mac use the phone's. Neither way will work over BT, as the neccessary profile isn't supported on the phone.

 

There are options if you are trying to set up the phone using Internet Sharing, which has the phone act as a WiFi Access Point using the cell data as internet access. The phone needs to have the Internet Sharing option, which is device and provider dependent.

 

You can probably set the Mac up as an Access Point via some software, (it is available for Windows PCs) which would allow the Mac to be a  WiFi router to whatever internet connection it has, wired or WiFi, for the phone.
