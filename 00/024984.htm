jeeped say:
... four cells are displaying 92%, with true numbers of 92.3076923076923 and 91.6666666666667 the ranks are coming out as 4 (92.307..)  and 6 (91.6666..) ...

If the numbers are true percentages, wouldn't the true numbers be 0.923076923076923 and 0.916666666666667?

... Any idea on how to get excel to read them both as just 92% so that they will rank the same?

RANK() does not like to be used in an array formula format but you can imitate the results with a SUMPRODUCT() counting the number of values that are less than the examined value.

=SUMPRODUCT(--(ROUND(J2,2)<=ROUND($J$2:$J$10,2)))

Note that I've rounded to two decimals to achieve a true 92% (i.e. 0.92).

The answer to this question is “yes”: The Tech Support Scam webpage that you encountered is directly related to the detection for Rogue:JS/TechBrolo. But the exact relationship isn’t clear because this is purportedly a detection for a “webpage”, whatever that amounts to:

https://www.microsoft.com/security/portal/threat/encyclopedia/entry.aspx?Name=Rogue:JS/TechBrolo.A

I did extensive testing of these Tech Support Scam pages over a period of several months in order to confirm that no actual malware (file-based malicious code) was involved in this pervasive scam. In the course of that research, none of the browser-locking variants of the Tech Support Scam was ever detected by any third-party AV/Antimalware app – and out of the hundreds of URLs that I tested, I only had a single Microsoft detection for Rogue:JS/FakeCall.D. But this “threat” wasn’t detected by real-time protection while the scam page was loaded – there was only a belated detection with a file path that pointed to the Edge cache. As I recall, the “threat” wasn’t properly removed in this case and I had to manually remove it by clearing the cache. So it would be helpful to know the detection path for Rogue:JS/TechBrolo.A, especially since we’ve already had some reports of remediation failures with that specific detection.

It’s been an uphill battle to get people to stop routinely recommending AV/Antimalware apps for these script-based modal dialog exploits, but the fact is that these Tech Support Scam pages have been an ever-growing menace just because they can lock up any browser without using malware – they just run an onboard script that reopens a fake alert dialog every time you try to close it by clicking on the OK button. That prevents you from closing the fake alert dialog, and that means that there’s no way  to reactivate the browser. Since these pages don’t need to depend on downloading file-based code to do their dirty work, they can generally avoid detection by AV/Antimalware apps, and these Microsoft “webpage” detections really don’t make a whole lot of sense. But while it would certainly be interesting to know the details with these oddball Microsoft detections, it wouldn’t change the basic ground rules for dealing with page-based scripts that exploit the browser’s modal dialog feature – these lockups can be fully resolved in most browsers by simply ending the browser process in Task Manager, or in the case of Microsoft Edge, by utilizing the Dialog Loop Protection checkbox:
