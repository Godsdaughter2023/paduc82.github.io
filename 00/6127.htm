john w. vinson say:

One of the basic principles of relational databases is that fields should be "Atomic" - having only one value. You're having trouble because your Competencies field contains multple values, not one. Access works best with properly structured data... and this isn't!

 

What do you mean by "can't... use a comma delimited series"? If you have a properly vertical table you won't HAVE a comma delimited series. The proper structure has THREE tables, not one:

 

Employees
  EmployeeID <primary key>
  LastName
  FirstName
  <other biographical data>

Competencies
  CompetencyID <primary key>
  Competency (text, e.g. Negotiation, Heavy Equipment, Database Design)

 

EmployeeCompetencies
  EmployeeID <link to Employees>
  CompetencyID <link to Competencies>

If Joe Schmoe, is good at six things, there will be six RECORDS (not six comma delimited text strings!) in EmployeeCompetencies with his EmployeeID. Your report then becomes much simpler, because you can just apply a criterion to the CompetencyID.

You'll need to run some Append queries to create this EmployeeCompetencies table from your existing comma-delimited string. You can use the Split() function in a little simple VBA to extract the numbers.

 

 

 

ken sheridan say: What you have here is firstly a many-to-many relationship type between Employees and Profiles, which is modelled by three tables:

Employees
....EmployeeID (PK)
....FirstName
....LastName

Profiles
....ProfileID (PK)
....Profile

and to model the relationship type:

EmployeeProfiles
....EmployeeID (FK)
....ProfileID (FK)

The primary key of the last table is a composite one made up of both columns.

You then have a one-to-many relationship type between EmployeeProfiles and Competencies which is modelled by a further two tables:

Competencies
....CompetencyID (PK)
....Competency

and to model the relationship type:

EmployeeProfileCompetencies
....CompetencyID (FK)
....EmployeeID (FK)
....ProfileID (FK)

The primary key of the last table is a composite one made up of all three columns.  Of these  CompetencyID  references the primary key of Competencies and the composite foreign key made up of EmployeeID  and ProfileID references the composite primary key of EmployeeProfiles.  The relationships between the tables should be created on this basis, and referential integrity enforced in each case.

Do not be tempted to think you can do without the EmployeeProfiles table and reference the primary keys of Employees, profiles and Competencies individually by the three columns in EmployeeProfileCompetencies.  That way there is no control over which profiles an employee can have competencies in.  There will also be information loss if an employee is not recorded as having competency in a particular profile.

For data entry you would have an Employees form and within that a subform based on EmployeeProfiles in which the profiles for the employee can be selected.  Both should be in single form view.  Within the EmployeeProfiles subform you'd have a further subform, in continuous forms or datasheet view (preferably the former), based on EmployeeProfileCompetencies, in which the competencies in the currently selected profile can be selected.

Another option would be to have continuous EmployeeProfiles and EmployeeProfileCompetencies subform both directly within the single form view Employees parent form, with the combo boxes being correlated so that as you navigate from row to row in the EmployeeProfiles subform the EmployeeProfileCompetencies is requeried to show the competencies for the currently selected profile.  This is achieved either by having two hidden unbound text box controls in the parent form which reference respectively the current ProfileID and EmployeeID of the EmployeeProfiles  subform and act as the LinkMasterFields property of the EmployeeProfileCompetencies  subform control; or by basing the EmployeeProfileCompetencies subform on a query which references the ProfileID and EmployeeID controls of the EmployeeProfiles  subform as parameters and is requeried in the Current event procedure of the EmployeeProfiles  subform.  The latter approach tends to perform better, but requires more work.

ken sheridan say: I think you are making a big mistake with this.  It violates First Normal Form, one of the most basic principles of the database relational model, which is defined formally as:

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple contains exactly one value for each attribute.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).

Your table structure is fine, provided that competencies in all profiles are recorded for an employee, no more no less.  Otherwise you need to introduce a further table to model the many-to-many relationship type between employees and profiles as I described.  Whether this is needed or not, however, there is only one correct way to store the values in the CompetencyID column in tblEmployeeCompetencies, which is a as one value per row.  This is how a relational database works, it's a simple as that, and doing otherwise is just making a rod for your own back.

To see the competencies per profile per employee set out horizontally in one line you can use a crosstab query or a multi-column subreport with across-then-down column flow.
