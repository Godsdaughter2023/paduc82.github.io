rob koch say:

Unfortunately I believe that explaining how the Behavior Monitoring and now blocking already built into MSE works is going to be difficult , since your thoughts are actually mired in the archaic ways these features work in older antimalware products.  In MSE they are part of the active protection system and as such require that you completely forget how you 'think' that they operate in order to understand how they really work.

 

The key to the protection system within MSE is that it is dynamically coupled into the SpyNet (now MAPS) system via the signature based and behavior systems.  This allows MSE to query the vast MAPS database using a signature generated by combining file hashes, network and other behavioral activity collected from its various monitoring modules.  If this signature matches a local definition then it can act immediately, but if it's unrecognized or even simply not a perfect match it can supplement this by uploading the signature to MAPS, which if known to MAPS can quickly return the associated definition to take action.  If it's completely unknown the item(s) can be optionally uploaded to MAPS for further investigation.

Though the above may not be as perfect as you'd like, it's really the most effective thing that can be done for a completely unknown item, since it at least makes MAPS and so the MPCC aware of a [potential] new threat.  Known malware families with minor variations are already covered via family type definitions which again include both file hash as well as various behavioral signature information.

 

Another ability of the MAPS system is the reversing of false positive detections, since the query made to MAPS allows an existing local definition to be replaced dynamically, updating it on the fly to remove the invalid portion of the detection once this has been identified and corrected in the MAPS database.

Stating that this system isn't effective is forgetting that most 'new' malware is experienced while online, not disconnected from the Internet.  Though this is possible, especially in the case that the PC is disconnected for multiple days, this situation is really no different than it has ever been  Again, the family and also heuristic detections will combine to provide some protection, but as with all such systems these will decay in value as they age offline.

 

The primary reason I've explained the operation of the MAPS system before the updates themselves is to show why constant updating of MSE is unnecessary, since it can update just what it needs on demand whenever a new item is detected.  This precludes the requirement for constant almost pathological updates and so allows a more reasonable approach of daily differential updates that contain all of the definitions which have been added since the last successful update cycle.

The reality that most don't understand is that very few new definitions updates are actually created and released in less than a few days, with only rare large outbreaks being produced in a day or less.  This is because they have been fooled by the apparent quick release cycles that appear to exist when updates are requested and sometimes even provided every couple hours.  The simple truth is that these systems typically just 'thrash' a similar number of actual definitions updates over a larger number of smaller updates, while the actual age of the definitons delivered may still be days.

Though the higher speed of the release cycle (e.g. 1 to a few hours) appears to provide quick response, the on demand ability of MSE and MAPS allows an even faster response with a more targeted type of request.  This ability precludes the need for nearly constant updates which along with adding no real improvement in true security, actually only serve to cause overhead for both the local PC and the update servers.

 

On a different theme, you have missed an important point relating to your understanding of how MSE operates relative to the network and firewalls with Windows Vista and 7  Though it does need the Windows firewall to be enabled to provide inbound protection, MSE also provides it's own network monitoring abilities in the mpnwmon filter driver.  This module allows MSE to monitor and detect network activity via the Windows Filtering Platform, so it performs similarly to the various third-party firewalls that provide dynamic protection, but simply uses this activity as another thing it can monitor and/or block if malware is involved.  So for this reason there's no need to configure the firewall beyond the normal inbound protection, since MSE itself will do the rest as required.

The difference with MSE in the case of network monitoring and many other situations is that it doesn't pop-up useless prompts about network activity, since these just serve to confuse most users and again provide no true additional security.  It simply monitors and if malware is detected it then provides network blocking, file quarantine/removal and any other actions as required as well as notification of the user.  This allows MSE to be normally quiet, but take action including notification when it's actually required.

 

In reference to your last item about the file hashes, as I recall this is how MSE avoids constantly re-scanning all existing files, by storing the file hashes in a cache.  However, these must be flushed after every reboot or other shutdown of the Real-time protection, since there's no way to be certain they haven't been modified by something malicious during the period that MSE was offline.

 

I won't try to address your other requests since some are covered better already in other threads here or you are asking for things that I really don't care about.

As for the age of detections found on the local PC, since these USB delivered malware tend to be days or even weeks old before they get to a new PC it's highly unlikely that you'd ever experience one before you'd updated again anyway, unless you are offline for several days at a time, but then that's the real issue isn't it?

 

In terms of the firewall, the issue there is less about additional control than it is about protecting any open ports created by installed software, which MSE already does with it's network monitoring module.  I don't see how adding more controls to another program would result in an easier interface, since Windows 7 already blocks by default and notifies the user when new software attempts to open connections.

Since attempting to autoconfigure the firewall for every possible application would be nearly impossible, it's simpler for MSE to monitor for known malware activity and simply block once any is found.  If the network involved is isolated from Internet access then using MAPS won't be available, but that is again an issue of connectivity and not for Microsoft to resolve.

 

As for the file/folder hashes, my point was that since the storage location for these would be known they would need to be regenerated each time the PC was restarted, since nothing about the state of the system while MSE was shut down could be guaranteed.  Thus the added security you were hoping for couldn't be acheived without re-scanning and regenerating the hashes anyway, since they could simply have been modified by malware while MSE wasn't enabled.

 

None of this or the above ever stated that MSE is perfect, since that's not true for any antimalware system, it's just the state of the art today.  It also isn't telling you what you can and can't have, just why most of it is unnecessary or too costly for Microsoft to implement.  Understanding your antimalware and how it works helps you to better understand your security as a whole, so take it or leave it, I just thought you might want to understand better what you are using.
