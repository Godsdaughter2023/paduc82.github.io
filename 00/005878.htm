greginmich say:

I was able to follow these steps and set up a daily scan to run in the early morning hours.  However after a couple of days, I checked Windows Defender and it still shows last scan date as a date from early in the week when I did a manual scan.  Why did my setup not work?

The Last scan info should be updating with the daily Automatic Maintenance Quick Scan. The steps provided above will only start the Windows Defender app – they won’t run a scan. Also, a scheduled scan won’t wake up a sleeping PC unless that option is checked in the Conditions tab. There's generally no need to run your scans in the wee hours, and they're much easier to monitor if they run while you're using the PC. Try this setup:

Windows Defender Scheduled Scan Setup:

Open Task Scheduler (Start button > All apps > Windows Administrative Tools > Task scheduler)

1.) In the right-hand (Actions) panel, click on Create Basic Task… and name the task “WD Scan” and click Next

2.) Click on the Daily option and click Next, and then pick a time and click Next

3.) We’re using the default action (Start a program) so just click Next

4.) Browse to (or just copy and paste) "C:\Program Files\Windows Defender\MSASCui.exe" in the Program/script box (with quotation marks)

5.) Enter -QuickScan in the Add arguments box (no quotation marks) and click Next

6.) Check the Open the Properties dialog for this task… box and click Finish

7.) Click on the Conditions tab, and check the Wake the computer to run this task box, and click OK

To schedule a weekly Full Scan, just change scheduling option to Weekly, and the Add arguments entry to -FullScan.

[Edit for update, 5/7/2016, 9/26/2017]

You might have to turn off Require a password on wakeup in Power Options and/or enable wake timers in Power Options > Change when the computer sleeps > Change advanced power settings.

If you want a quicker way to schedule a scan, then you can use SchTasks.exe: Right-click on the Start button and select Command Prompt (Admin). Then copy, paste, and run this command line: 

Windows Defender Quick Scan:

SchTasks /create /sc DAILY /tn “Defender Quick Scan” /st 19:00 /IT /rl HIGHEST /tr "'C:\Program Files\Windows Defender\MSASCui.exe' -QuickScan"

Windows Defender Full Scan:

SchTasks /create /sc Weekly /tn “Defender Quick Scan” /st 19:00 /IT /rl HIGHEST /tr "'C:\Program Files\Windows Defender\MSASCui.exe' -FullScan"

You can modify the scheduled start time (/st 19:00) for this scan either before running the command, or in the Task Scheduler Library after the task is created. The scan should be set to start at a time when the PC will normally be in use, since it won’t run when the PC is sleeping, or on the next startup when a scheduled time is missed. If you want a more reliable scan, then you’ll have to add these features by checking the Wake the computer to run this task box (Conditions tab), and the Run task as soon as possible after a scheduled start is missed box (Settings tab).

You can also run a daily Quick Scan at startup with a startup short cut:

http://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_start/is-windows-defender-icon-supposed-to-be-in-system/6cfa9d1d-3db8-49a9-add7-280a284ccfdc

If you want your scheduled scans to run at full speed, then you’ll also have to run this command at the elevated PowerShell prompt:

Set-MpPreference -ScanAvgCPULoadFactor 100

ken sheridan say:
It should be a relatively easy fix but I can't find it

Maybe not quite so easy.

My guess, and it's no more than a guess, would be that the date and time are inserted into the subform by means of the DefaultValue property of the date/time column  in the subform's underlying table and/or the control bound to it in the subform being set to Now().  When you set the DefaultValue property it does not in itself initiate a new record.  That only happens when further data is inserted into the row or the default data updated, so without doing this, while the data will show in the subform, no row will be inserted into the table.

Probably the best solution here would be, in the AfterInsert event of the parent form insert row into the referencing table, and then requery the subform to show it, with code along these lines:

    Dim strSQL As String
   
    strSQL = "INSERT INTO VolunteerLog(Volunteer_Id,DateTimeStamp " & _
        "VALUES(" & Me.Volunteer_Id & _
        ",#" & Format(Now(), "yyyy-mm-dd hh:nn:ss") & "#)"
       
    CurrentDb.Execute strSQL, dbFailOnError
   
    Me.sfcVolunteerLog.Requery

where:

VolunteerLog is the name of the subform's underlying table.
Volunteer_Id and DateTimeStamp are the names of its columns.
sfcVolunteerLog is the name of the subform control in the parent form.

You should set the subform's AllowAdditions property to false as the row is now being inserted by the code.  otherwise you'll get two rows showing in the subform, one the inserted row, the other an unsaved new row as at present.

ken sheridan say:
and/or using something like the RunMenuCommand > SaveRecord macro

If I'm right about the DefaultValue property being used here to insert the date/time value, the problem would again be that there is nothing as yet to save.  This often confuses people because when you see a form with default values in it, the natural assumption is that there is something there to save.  That's not the case, however, until the user begins to enter more data or change a defaulted value in the form.  A trick in cases like this is to set the value of one of one of the defaulted controls to itself, which does begin to initiate a new record, so it can then be saved.  It might be possible to do this in a macro before saving the record, but I think the best solution here is to insert the record with code as I described.
