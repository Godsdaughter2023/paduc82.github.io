jeeped say:

In this macro example, I changed column A in the first image to column A in the second,

                

 The macro only acts on one state pair at a time, but moves to the next position in order to be quickly run again. Assign a keystroke shortcut to facilitate this repetition.

Sub mcrSplitStates()
' todo: assign keystroke shortcut
    ActiveCell.Offset(0, 1).Range("A1").Select
    Selection.FillRight
    Selection.TextToColumns Destination:=ActiveCell, DataType:=xlDelimited, _
        TextQualifier:=xlDoubleQuote, ConsecutiveDelimiter:=False, Tab:=False, _
        Semicolon:=False, Comma:=True, Space:=False, Other:=False, FieldInfo _
        :=Array(Array(1, 1), Array(2, 1)), TrailingMinusNumbers:=True
    ActiveCell.Offset(1, -1).Range("A1:A2").Select
    Selection.Insert Shift:=xlDown, CopyOrigin:=xlFormatFromLeftOrAbove
    ActiveCell.Offset(-1, 1).Range("A1").Select
    Selection.Cut Destination:=ActiveCell.Offset(1, -1).Range("A1")
    ActiveCell.Offset(0, 1).Range("A1").Select
    Selection.Cut Destination:=ActiveCell.Offset(2, -2).Range("A1")
    ActiveCell.Offset(3, -2).Range("A1").Select
End Sub

a. user say:

<<
I would expect that this information is reflected inWord Help. 
>>

I would certainly hope that this kind of stuff was explained in some easy-to-find place. But hope and expectations are two different things.

<<
The influence of local regional settings is at all a big surprise... Does it mean that if a person composes a document in UK and sends it to colleague in USA, he will see different results for table formula? 
>>

That is exactly what it can mean. FWIW I do not think these problems are easy to solve, because in Word it would probably require users to signal their intent in some way (like they do with Excel by formatting a cell as numeric or text).

In a way, your document is a simple example of what can go wrong: presumably in your locale, "," is used as the decimal point character, whereas here it's ".". 

Question 1 is really "how should your document be displayed here, and who should control that?"
 - option 1 is "display it exactly as you see it", with "," rather than ".". But then all the field updates go wrong
 - option 2 is "change it to match my locale settings". Word would have to know to change "," to "." and not to change $ to £ - unless, of course your document is some kind of example and you want people to see their local currency values

Either way, Word would have to detect which things in your document should change and which should not. It could probably do such stuff automatically reasonably well (e.g. distinguishing between a number like 2.3 used in a caption and 2.3 used in a calculation), but IMO this is where the "intent" thing comes in and a user would have to have a way to specify a number in a universal way and specify whether the formatting should be fixed or locale-dependent.

At the moment, the only reasonably locale-independent way I know to enter a number is to enter 3,49 (or 3.49) as an + field like this { =3+49/100 }. I can't see many users doing that for long! Nor is it simple to specify any kind of locale-independent formatting in the \# switch.

When Word gets numeric data from outside, for example from an Excel Sheet during a mailmerge, things are a little different because numbers in Excel are stored as numbers (basically in the universal XML numeric formats). The provider that retrieves the data will apply the correct localised "." or ",". So Word should see localised numbers and do the calculations correctly. But as for currencies, well that is trickier. If I make a sheet with UK locale and format a cell as a currency, by default Excel inserts a "£" (actually, AFAICR I can choose). If I then change my regional preferences to France (actually I am doing this on a Mac which may behave differently in this area from Windows) and re-open the sheet, the numbers are still displayed as "£" (which is probably what most users would want) but the cell format has been modified to "Custom". When I use that as a merge data source, the data comes through with "£" and "," as the decimal separator. Which is probably OK.

<<
What about sharing the document online via Word Online – what regional settings will be applied on the background?
>>

I don't know the answers. I cannot test all the current "cut-down" versions of Word, but suppose my OS has the French locale settings and I create a document with a number using { =3+49/100 }. Then when I save it, I see 3,49.

When I copy that to OneDrive and open it in either the Web-based Word Online app or the Android one, it displays as "3,49". AFAIK these apps might update fields such as { PAGE } fields but do not update all field types (even when you try to print etc.).

So in a sense you are stuck with what you started with until you load the document into either WIndows or Mac desktop Word, and then the current locale settings will be used (but probably only when you next update the field.

But these are all details, really. If you need to know how it all works at present, you just have to slog through the necessary research - knowing exactly what you want to work and how would make that easier. Designing a set of facilities that would deal with this kind of problem more systematically is another matter.
