daniel g. grau say:
The language setting of text is defined separately from whatever version of Word you are running, or the language of the OS X. It's just another attribute, like Bold, Size or Color.

  • Select the text in question
  • Go to Tools > Language
  • Choose your version of English. Make sure the bottom checkmark is absent.

You can also change the language globally in the styles. As most paragraph styles are based on the Normal paragraph style, usually all you need to do is define the language right there:

  • Select Format > Styles…
  • Click on Normal
  • Then click on the Modify button
  • In the bottom dropdown menu, choose Language and proceed as above

If all your documents are based on the Normal.dotm template (as when you do Command-N or chose File > New Blank Document) and their language is wrong, then you need to change the language in the template:

  • Go to the Finder*
  • Locate Normal.dotm in this folder: YourUserName:Library:Application Support:Microsoft:Office:User Templates
  • Double-click* on Normal.dotm and redefine the language of the Normal paragraph style as explained above
  • For good measure, select the document's content (usually just a blank ¶) and apply the proper language
  • Close and save Normal.dotm

Note: It's just a coincidence that there is a "Normal" paragraph style and a "Normal" template. No relation, though.



* See Bob's correction to this method in his post below. You must open Normal.dotm through Word's File > Open menu.

dianeoforegon say: Any mail stored under "On My Computer" is subject to loss. However, since a search shows the messages we can try some options to recover, but I highly suggest you move your mail to IMAP. Comcast is supposed to offer IMAP soon, but there are other free ways to get IMAP like iCloud and Gmail for example.

See this chart "Compare Sync options for POP, IMAP & Exchange Accounts"

1) Drag the sent folder to the Desktop. Check the size of the folder in the Finder. Does it contain data? I dragged a folder with 1 message and it was 10MB.

If the folder is larger like KBs or GB, you can try importing this folder into a new Identity.

2) Duplicate your Identity then do a rebuild.

Rebuild Basics:

  • Do Not Rebuild Identities with Exchange and IMAP accounts. Delete the accounts (this deletes the cache files in Outlook). Do the rebuild then add back accounts. Better yet start over in a new Identity. If the rebuild does not work then you could sync unwanted changes or deletions to your server and lose data.
  • Make sure you have plenty of free space on your drive. Make sure that you have at least three times as much hard disk space available as the current size of your Outlook identity PLUS 10% free for virtual memory.
  • Rebuilding does not fix all issues. It can make some problems worse. Even though the Database Utility will make a backup, I advise making a copy in the Finder first.
  • You can delete the rebuild version after you are satisfied with the results. This copy will be named like this: Main Identity [Backed up 1-21-2012 13.35]  

Tip: If your Identity will open, before you rebuild, check "On My Computer"Inbox and Sent folders to be sure they are under the 2GB limit. Generally this is less than 10,000 messages but with large attachments could be less. Ellie wrote:
can you give me an example of calling a "DoEvents"?

Very simply:

SolverSolve
DoEvents

If you get a syntax error, perhaps DoEvents is not available on a Mac.

On a Windows system, DoEvents yields the CPU to any other processes that are ready to run.  But that is no guarantee.  If those processes themselves "block" (stop running), the VBA process might start running again prematurely.

The logic you really need is conceptually:

SolverSolve
Do: yield CPU: Loop Until Solver is done

As you discovered, Wait and Sleep affect the entire process, not just the calling thread.  So you need some other way to accomplish "yield CPU".

But the important question is:  how to determine "Solver is done"?

I presumed that SolverFinish would encapsulate that logic.  You seem to say it does not, at least not on a Mac.

Since you believe the Solver operation will take "about 60 seconds", the following hack might work around the problem.

Dim y as Variant, x as Double
y = SolverSolve
x = Timer
Do: DoEvents: Loop Timer - x > 60

Caveat:  Do not execute that across midnight.

If DoEvents does not work, you might try a statement of the form:

Range("a1") = Range("a1").Formula

where A1 is some cell that is "safe" to access.  At the very least, that will schedule the Excel thread to run, which might yield the CPU to the thread running Solver.

Of course, it is a "terrible idea" to always wait 60 seconds.  It is even worse and probably useless to code a loop like that without some way to "yield CPU".

But it might get you off the hot seat while you look for a better solution.

Note that I wrote y=SolverSolve.  You might want to take a look at what SolverSolve returns on the Mac when the operation is not yet complete.  That might give you a clue as to what else you might try.

There is a bug in Solver under Mac Excel 2011. I submitted a report to the feedback site on this. My summary of the problem for others searching for help (hard to find) is this:


I too have (re)discovered this. It still occurs in version 14.3.1  I posted a feedback report to Microsoft and because of the difficulty I had just trying to find info on the problem I will post my conclusions here:

There is a serious problem using  Solver for Excel in Office 2011 when it is called from vba. Because Solver is a separate application, control must be transferred from Excel to Solver, and this control does not occur until after vba has terminated. This means that any processing that occurs in the vba program after the call to SolverSolve will be completed before Solver has done its task.

Thus a vba program to call solver and then capture or process the results _cannot_ work. It also precludes calling Solver from vba in a loop even though the documentation suggests this is possible

Code that adheres to the published rules for using Solver from vba will work perfectly on a PC but does not work on a Mac.

I have tried many work-arounds using delays of various sorts but none work - one must actually exit vba before Solver will run.

This means that any post processing of solver results must be completed using events which are very complicated to use for multiple calls to Solver.

Full scans by any security application are known to display false positive detections, since this type of scan looks at every file anywhere on the disk drive without any regard for context or whether the potential malware detected is active or not.

For this reason, virtually anything detected by a full scan that isn't also detected by a quick scan should be taken with a gran of salt.  That's why Microsoft has only recommended the use of a full scan either when other malware has been recently detected or may be suspected for other reasons such as slow operation or other possible indications that malware might be involved.
