billmeye1 say:

Over the last week I gave Sharepoint a try using Microsoft Live Small Business.  It is a great idea but really impracticle.  In my mind I thought I would get a WAN and be able to work anywhere at anytime, replication (which is initself multiple backups) and not need to wory about copying backends from one computer to another if you want to take it on the road, ability to keep track of changes since Sharepoint does this automatically, have a secure location for my data that requires user level access and passwords and be working in the Clouds.  Here are my experiences:

1. It is very slow.  Much too slow to be practicle.  My users were ready to kill me.  Most of my tables have less than 500 records and even the ones with less than 100 took much longer to load then when just running over a network (wireless at that).  It kept losing connection to the lists and then would ask to reconnect.  I had to disconnect from my network to force it to run locally and that helped some but it still seemed quirky.  It seems like even though all the tables are supposed to be cached locally, it still wants to verify all the records and update the tables even if nothing has changed.  Also, many times when I would click synchronize it couldn't find my tables at the Sharepoint Server and I would have to wait if I truely wanted to synchronize data with another user or synchronize before shutting down so all my data updates the Sharepoint Lists.

2. No referencial integrity so you can't cascade update or cascade delete.  AND you can only have 1 field per table indexed for connection to other tables.  Not allowed to have composite indexes or secondary indexes.  It becomes much more difficult to keep your niclely written relational database organized.  you really have to work much harder to keep your data clean and up to date.

This may seem like something you can code around, but one thing I noticed was when your working offline Sharepoint lists decrement the autonumber ID's so they count from -1 down.  Once you recoonect to the Sharepoint list it will then assign the correct incremental value.  If you are like me and are using this autonumber ID in daughter records created using VBA in AfterInsert Events for the Parent record, then you are first filling in the field with this negative number.  You have to make sure that each value you want to keep updated once reconnected to Sharepoint Server has been created using a lookup field when creating the original table.  If it is not it will not get changed once you reconnect to the Sharepoint server and you will lose the parent/daughter relationship since you are stuck with this negative number.  Access always increments all the time so the first value assinged is always the value. 

Also, if you are doing anything with the newly created negative autonumber behind the scenes, such as using dmax() to determine the value of the autonumber ID you won't get the correct value, you need to use dlast() and if you want to do any math for something else your out of luck since Sharepoint just assigned a negative and so adding to it will inevitably duplicate an existing record.

3. So many times it would lose connection and not be able to populate a form or combobox or list because it can't find the associated tables.  I know that the great thing about Access 2010 is it caches all the tables locally so you are supposed to be able to always work offline, but it just wasn't consistant.  Access would hang up for a very long time while it tries to connect to this table on the Sharepoint Server which could not be any more annoying when your trying to get work done.

4. Sharepoint does not seem to allow TIME only in a date field.  So, if you have any fields that store TIME only you will lose this data when exporting to Sharepoint and you can only store date and time in a date field.  I tried to work around this my stoing a number in the Sharepoint field using MyTimeNumber = Cdbl(NewlyEnteredTime) and converting it back to time by using TimeValue(CVDate(myTimeNumber)) but I just gave up due to my database used TIME in too many locations.

There are many other restrictions on field types, such as no binary allowed, that would require some rewritting and restructuring.

5. When you synchronize to the Sharepoint Server it needs to close everything to do so.  You have to decide if syncing is more important then keeping your workflow going.  I'm not expierienced with Sharepoint or Access 2010 but I could not find any way to intentionally disconnect from the Sharepoint Server to work offline and prevent the continual interuptions asking to synchronize.

In the end I had to recover all my data and switch back to our local LAN and use remote desktop to use the database remotely.  It might be that Microsoft is still using Sharepoint 3.0 and not Sharepoint 2010.  I did read that Sharepoint 2010 allows referencial integrity.  If anyone has experience with Sharepoint 2010 please share your experience.  Still, I guess I'm like so many others who wish I could just use Microsoft Access over a WAN without having to change 1 line of code!

jgt say:

word 2010 macro to automatically update TOC

How can I create a word 2010 macro to automatically update the TOC if it is present? I have several documents with a TOC and frequently I add content and would like to have the TOC automatically updated.
deanh say:

There is a built-in field (which a TOC is) update mechanism in Word (has been for many years), first check File tab, Options and that the "Update Fields" is checked in the Print options.

Apologies, I am away from my 2010 machine and cannot tell you exactly where this option is.

Once checked, anytime you want to update all the fields in the document, go to different views but one must be the Print Layout view. Does 2010 still have the view icons at the bottom of the window on the left of the horizontal scroll bar? if so you can do this quickly.

This will refresh all the fields in the document.

Now if you are after a macro that will register that you have done a change in the document that requires the TOC to be updated, I think you may be out of luck.

I do have a macro that refreshes the whole document and which I have assigned to my own icon but this does require a manual activation of "pressing the icon" sorry, not automatic ;-)

Sub UpdateDocument()

' Macro to update whole document, Fields and TOCs

 

Dim oStory As Range

Dim oTOC As TableOfContents

Dim oTOF As TableOfFigures

  For Each oStory In ActiveDocument.StoryRanges

    oStory.Fields.Update

    If oStory.StoryType <> wdMainTextStory Then

      While Not (oStory.NextStoryRange Is Nothing)

        Set oStory = oStory.NextStoryRange

        oStory.Fields.Update

      Wend

    End If

  Next oStory

For Each oTOC In ActiveDocument.TablesOfContents

        oTOC.Update

    Next oTOC

For Each oTOF In ActiveDocument.TablesOfFigures

        oTOF.Update

    Next oTOF

    ActiveDocument.Save

End Sub

In addition, the Teams MSI will place an installer in Program Files. Whenever a user signs into a new Windows User Profile, the installer will be launched and a copy of the Teams application will be installed in that user's appdata folder.

%userprofile%\AppData\Roaming\Microsoft\Teams

%userprofile%\AppData\local\Microsoft\Teams

 

If a user already has the Teams app installed in the appdata folder, the MSI installer will skip the process for that user.
