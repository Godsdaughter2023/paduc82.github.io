karl timmermans (outlook mvp 2012-2018) say:
>>

First question is: "What type of email account is configured in OL'2010? (where type = POP3, Imap etc)

Next , if it's an IMAP account, then the email folders would be in a different PST file then where the contact/calendar data is being stored. IMAP accounts in '2010 and earlier use 2 PST files for an IMAP account. In other words, you need to make sure that you are referencing the right PST file.

Lastly, if this account is also being configured as an IMAP account in '365 then would not recommend "importing" the contact/calendar data into the OST file used by the IMAP account. Contact/calendar folders would have the added words "This Computer Only" in the folder description which means exactly that. If the IMAP account is ever removed OR Outlook can't find the OST file for the account, it would create a new OST file and then sync the email folders but anything that was stored in "This Computer Only" folders would be lost since contact/calendar data is not sync'd with the server for an IMAP account.

Instead, find the correct PST file that has the contact/calendar data, open it in '365
(Go to File --> Open & Export --> Open Outlook Data File). When it's added, then go to File --> Account Settings --> Data Files tab --> select the PST file and set it as the default data file for the profile. The default data file is where the default contact/calendar folders are stored. PST files never get deleted (or "recreated") by Outlook.

karl timmermans (outlook mvp 2012-2018) say:

Instructions were based on the source version being '2010 (or earlier) where the IMAP account would use 2 different PST files. First one would contain the email folders which would mirror what's on the server. The second one would contain the default contact/calendar folders or what you're referring to as the "address book" with lots of contacts. For sake of accuracy, contacts are stored in a contact folder which is a member of the Outlook Address Book (the default contact folder automatically becomes a member of the Address Book.

Given that this is an IMAP account, the contacts stored on the server have no relevance since IMAP accounts don't sync contact/calendar data.

What is confusing is the statement

"this is a laptop that has been through far too many operating system and application updates since it was "born" in 2013."

Specifically the reference "born in 2013". If that was meant to include Office 2013 then the account could not have been configured as an IMAP account without there being an .ost file - no ifs, ands or buts about that. If Outlook '2010 was upgraded to Outlook '2013 (or later) then an .ost file would have been added when the upgrade took place, email folders would have sync'd with the server AND the original default data file, which would have been a PST file - the exception to that rule would be if an Exchange based account was also configured in the profile and the data file for that account was configured to be the default in which case the contact/calendar data would be stored there and immediately available when the account gets configured again.

So given all of the above, the answer to your question "What do you suggest? " - the answer is to

1) find the PST file that contains the contacts,
2) copy it to the new system and set it as the default data file on the new system.

You really don't want to have contact/calendar data stored in folders with the added words "This Computer Only" (if using Outlook '2013 or later). Reasons are that (1) if the IMAP account is ever removed, then the associated OST file will be deleted causing all data in the "This Computer Only" folders to be lost and (2) if Outlook cannot find the OST file for the account it will recreate the file, sync the email folders but the contact/calendar data will be lost. The "This Computer Only" folders will be empty in the newly created OST file.

marvin ma msft say:

Hi Jeffrey215,

It is my pleasure to write here for you. As far as I know, if the other Office 365 tenants try to spoof sending emails as your domain, the Exchange server will detect if the domain is owned by them. Once the Exchange server find the domain is not owned by them, the spoofing emails will not be sent out. 

As for the SPF record, it’s mainly used to prevent spoofing and phishing in the receiving server. For example, if someone using other mail server tries to use your domain to send spoofing and phishing, the receiving server will check if the IP address included in the SPF record is spf.protection.outlook.com. And if the receiving server finds it’s not the correct IP address, they will handle the spoofing emails according to their policy. Also, if you are interested in how the SPF record works, you can refer to How Office 365 uses Sender Policy Framework (SPF) to prevent spoofing. 

Besides, if you are an admin, I suggest you turn on the hard fail option in Exchange Admin Center. This operation will help to mark the messages failed in the SPF check as spam.

cody c. say: Hi SeamusSu,

Unlike Outlook for Windows, Outlook for Mac can only perform a one-time PST import. This import will create a folder with the name of the PST file and add all of the information in the PST file to the Outlook for Mac database.

Outlook for Mac does not support linking to a PST file. In addition to that, Outlook for Mac does not use a PST file to store emails. Instead, it keeps it's information in a database that is made up of individual files and folders. This is partly to ensure support for Time Machine.

If full PST support is a feature you’d like to see in future versions of Office for Mac, be sure to send your feedback by clicking Help > Send Feedback… in any Office application or by clicking on the link below:

Office for Mac | Product Feedback

young yang msft support say:

Hi Loreman,

1. First, it is strongly recommended that we do not set the primary SMTP domain as Internal Relay in Exchange hybrid environment.

2. For your situation, hybrid configuration in this scenario is possible:

When on-premise users send emails to online mailboxes (have corresponding on-premise mail users), hybrid connector will route the emails to cloud.

When on-premise users send emails to external users (no on-premsie mail users), emails will be routed to the smart host due to Internal Relay domain.

3. Moreover, HCW does not check or change on-premise domain type.
