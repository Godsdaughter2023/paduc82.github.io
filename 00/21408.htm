There are a number of possible gotchas in this process, and your post doesn't say enough about what you did or what you verified to say where the problem lies.

In which template is your macro stored? To be available in a new document, it has to be stored either in Normal.dotm or in a template in the folder %appdata%\Microsoft\Word\STARTUP. (The macro recorder, if that's what you used, defaults to saving in Normal.dotm, but there's a box in the dialog to let you change that.)

In which template is your shortcut stored? Again, there's a box in the Customize Keyboard dialog to let you set the location, and it defaults to Normal.dotm.

What is in the macro's code? If you recorded the macro, it's possible the macro has no executable statements, so it actually runs but doesn't do anything. Copy the macro from the VBA editor and post it here so we can take a look.

For the specific example of inserting today's date, there is already a shortcut Alt+Shift+D for the InsertDateField command. Is there a reason you aren't using it?

jay freedman say:

In fact, there is a way to insert the date in the format of your choice without a macro. On the Insert ribbon, click the Date & Time button in the Text group. Select your preferred format in the dialog and click the Default button at the bottom. Then use the Customize Keyboard Shortcuts dialog to assign your preferred shortcut to the built-in InsertDateField command, instead of or in addition to the default shortcut of Alt+Shift+D. Your format will be applied to the Date field this command inserts.

If your shortcut assignment doesn't "take", or doesn't survive closing and reopening Word, then the problem actually be in saving the Normal.dotm template rather than in assigning the shortcut. Usually the cure for that is renaming the template and letting Word build a new one.

dale howard [mvp] say:
RC PM --
 
Although I do not work for Microsoft, I can assure you that this behavior is most definitelyNOT a bug.  Microsoft designed this functionality to resolve another problem that has been in the tool for a long time.  This problem occurs when the PM applies the Task Usage view (or Resource Usage view) and enters Actual Work in the timephased grid on the final day of the task, and the Actual Work entered is greater than the Remaining Work on that final day, and then the PM increases the Remaining Work after that.  Microsoft introduced the Peak field to resolve that problem, which it did very nicely.  However, in solving one problem, Microsoft created a lot ofconfusion in the minds of experienced users (such as yourself) who are used to the tool behaving as it has in all other previous versions.  We address this issue directly in ourUltimate Study Guide to Microsoft Project 2010 book and class, where we have the students learn how Task Types now interact with the new Peak field.
 
I feel your pain, my friend.  Hope this helps.
