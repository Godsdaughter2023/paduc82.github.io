ken sheridan say: Count the rows per Org returned by a subquery which returns a distinct set of Org/SSN values, e.g.

SELECT Org, COUNT(*) AS StudentCount
FROM (SELECT DISTINCT Org, SSN
           FROM Enrolments) AS DistinctStudents
GROUP BY Org;

Yes, Windows Defender advanced configurations where removed (back in Windows 8).  However you still have access to PowerShell to tweak some Windows Defender settings.

By default - Windows Defender is set to use 50% (on average max) of you CPU during a scheduled scan.  If you want to give it more - the following command will change the value : Set-MpPreference -ScanAvgCPULoadFactor 100

Is this CPU cap for Windows Defender scheduled scans for real? CPU throttling was the solution for the impact of MSE’s scheduled scan on system responsiveness and usability in Windows 7 – where the scheduled scan and the user had to share the system’s resources. In Windows 8/8.1/10, the scheduled scan’s performance hit is mitigated by only-on-idle scanning. And Automatic Maintenance doesn’t need to share the system’s resources with the user any more than the user needs to share those resources with Automatic Maintenance.

Moreover, a 50% CPU cap effectively doubles the scheduled scan’s run time.  Now with Automatic Maintenance we have a scheduled scan that’s being confined to idle-time, and that also has to start over from scratch (it’s not being paused) every time the user interrupts it. In this context, where time is of the essence, and where the scan isn't competing with the user for the system's resources – why shouldn’t the scan run at full tilt? Why in the world would MSE’s default CPU cap still be imposed on a Windows Defender scheduled scan?
