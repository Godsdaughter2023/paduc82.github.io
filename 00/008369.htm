zorvek say:

Relative to the Objects, where can bits of this code be relocated?

It can be located anywhere and do whatever needs to be done. The only difference between one module and another is what the default parent objects are. For example, within a worksheet code module, any use of the object Range without qualifying it refers to a range in that worksheet. In another module is refers to a range on the active worksheet.

Is there a preferred location to declare all my variables?  If so where and how?

Always declare your variables with as narrow a scope as possible. This means putting them into the routine if possible. If a specific variable is used in different routines then it should be defined in one routine and then passed to the others. If a variable is used across many routines in a single module and you need that variable to persist over time then define it a Private in that module - this is called a private module level variable, it's scope is only within that module, and all routines in that module can reference it. If a variable is accessed across multiple modules and it has to persist over time, then define it a Public in a general code module - this is called a global variable and it has global scope. Use this last type of variable sparingly and only when needed. Overuse of globals is considered bad structured programming technique.

Is there a preferred location to set my variables? If so, where and how?

VBA always initializes variables to 0 or a null string or Empty. Initialize them when you need them. When, exactly you initialize module-level and global variables depends on how they are used.

In your specific example, I would put the code in the worksheet code module that you want to manipulate. I'm suggesting this because I'm assuming the code only manipulates one worksheet. The next question is, what triggers the code to run?

You probably do not want to define the routine ShowColors as Private because you won't be able to call it from outside the worksheet code module or as a macro. If you call it from the worksheet's Worksheet_Change event handler then it can be private.

Kevin

jan karel pieterse say:

OK, some basics.

The modules in the branch "Microsoft Excel Objects" are in fact class modules belonging to their rsesective objects. Being:

One for each worksheet in your workbook (named after the sheets)

One for the workbook itself (called ThisWorkbook)

This group of modules is primarily meant to hold event code: code responding to events on those objects.

For example, in ThisWorkbook you would write code that is called when your workbook is opened, closed, saved, printed. In a worksheet module you write code that actions when the user selects a cell, changes a cell things like that.

The group called "Modules" contains "normal" code modules. In these you place macros. This is also where the macro recorder puts them.

So where your code should go depends on what should start the code to run.

What object your code addresses depends on your code entirely. In the example you gave,

Range("D4:D4")

addresses cell D4 on whatever worksheet is the active worksheet. If you need to work with a cell on a specific worksheet, prepend range with the worksheet:

Worksheets("FooBar").Range("C4")

If you also need to address a worksheet in a specific workbook, then you also need to add the workbook object reference to that same object:

Workbooks("SomeBook.xlsm").Worksheets("FooBar").Range("C4")

If you want to address the workbook in which the VBA code is running:

ThisWorkbook.Worksheets("FooBar").Range("C4")
