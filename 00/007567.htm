mimmo57 (d. laurenza) say:

Hi,

to archive both in one archive table, you can do a query creation table so as

SELECT tblMember.Id, tblMember.C1, tblSubscription.[Date of subscription] INTO tblArchive
FROM tblMember INNER JOIN tblSubscription ON tblMember.Id = tblSubscription.Id_Member
WHERE (((tblMember.Id) Not In (Select Id_Member From tblSubscription As A Where A.Id_Member = tblMember.Id And  Year([Date of subscription]) > year(Date())-5)))
ORDER BY tblMember.Id, tblSubscription.[Date of subscription];

ken sheridan say: Moving data to a separate archive table serves no real purpose.  A better solution is to simply include a Boolean (Yes/No) column named Inactive or similar in tblMember and set its value to TRUE for those rows you wish to 'archive'.  This can be done easily with:

UPDATE tblMember
SET Inactive = TRUE
WHERE NOT EXISTS
    (SELECT *
      FROM tblSubscription
      WHERE tblSubscription.Id = tblMember.Id
      AND DateOfSubscription > DATEADD("yyyy",-5,DATE()));

And conversely to FALSE with:

UPDATE tblMember
SET Inactive = FALSE
WHERE EXISTS
    (SELECT *
      FROM tblSubscription
      WHERE tblSubscription.Id = tblMember.Id
      AND DateOfSubscription > DATEADD("yyyy",-5,DATE()));

The execution of the two queries could easily be automated at start-up to keep things up to date.

For day to day operation of the database base any forms or reports on a query which restricts the result table to:

WHERE Inactive = FALSE

By virtue of an INNER JOIN with the tblSubscription table only the personal and subscription data for non-archived members will be returned.  Should a member be reactivated by the insertion of a row with current date into tblSubscription their data, both past and present will immediately be returned once the UPDATE queries have been executed.
