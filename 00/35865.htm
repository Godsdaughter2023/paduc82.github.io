baffin say:

I had the same problem (Indexing would not index a partition assigned the letter B).  Windows Explorer sees that drive B is indexed so it uses the index -- but since Indexing apparently has a bug and didn't scan B, nothing is found.  If you modify the Indexing configuration to NOT index B, you'll find that Windows Explorer will then use 'brute force' searching on B -- and it'll find your files but it'll be slow because it is doing the search right then.

I tried to solve this by changing the drive letter to something other than A or B (I picked E).  Disk Management appeared to change the drive letter but after a reboot, E disappeared!  I had to use diskpart to get it back to B.

This solved (worked around) it for me:  Create a new partition, call it E or something (not A or B).  Copy all but System Volume Information and Recycling Bin to the new partition (from your B).  Set things up (eg., Indexing) for E, not B.  Everything should work.  Later, when you are sure all is well and nothing is using B, delete B partition.  (If you are comfortable doing it, check the registry for references to B.)

I expect Microsoft will fix all these Win 7 bugs sometime.  Meanwhile, the above procedure worked for me.

Other s/w manufacturers might have similar bugs, so even after Win 7 is fixed, it's still probably risky to use A or B as partition letters.  (A and B used to be reserved as removable floppy drives -- coping with that history is probably what's screwing up.)

lemp say:

It is highly unlikely that rundll32.exe is corrupted, especially given that sfc /scannow did not show any problems.

Windows programs are either "executable" files (*.exe) or "dynamic link libraries" (*.dll).  Without going into why a software designer would use one of these rather than the other, the basic difference is that a DLL file can't run on its own but must be "hosted" by another process. 

Rundll32.exe is a host process for running DLL files.  If rundll32.exe is not present in Task Manager, that simply means that no application that is coded primarily as a DLL file is running.

Two excellent tools for investigating problems involving DLL files and excessive CPU use are Process Explorer and Process Monitor.

Go here:  https://docs.microsoft.com/en-us/sysinternals/downloads/process-explorer and download Process Explorer (you may later want to download Process Monitor as well:  https://docs.microsoft.com/en-us/sysinternals/downloads/procmon)

Although you can just run Process Explorer, you probably would benefit by first spending a hour or two watching the videos linked under "Learn More" at the bottom of the Process Explorer web page.

There also are a number of "how to use Process Explorer" guides available through a Google search:  https://www.google.com/search?q=process+explorer+guide
