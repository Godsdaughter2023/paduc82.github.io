scottgem say:

I am not a fan of Composite keys and the problem you are encountering is a prime reason for it. 

What I would recommend is to add an Autonumber field as your PK. Then set a unique multi-field index on the combination of fields that you have now as your composite key. This will allow you to make your relationships work more easily by being able to use a single foreign key while preserving the uniqueness of the combination of values. 

ken sheridan say: For the combo box's RowSource property use an SQL statement which returns both components of the composite key of the table which models the binary relationship type, along with the relevant non-key columns you wish to see listed.  Hide the key columns by setting their dimensions to zero in the ColumnWidths property.  Bind the combo box to whichever component of the composite foreign key references the combo box's BoundColumn, and in the control's AfterUpdate event procedure assign the value of the second hidden column to the second component of the composite foreign key.

andre for directly say:
Its a legitimate component in Windows.

he dllhost.exe process goes by the name COM Surrogate and the only time you're likely even to notice its existence is when it crashes and you get the message COM Surrogate has stopped working. What is this COM Surrogate and why does it keep crashing?

The COM Surrogate is a fancy name for Sacrificial process for a COM object that is run outside of the process that requested it. Explorer uses the COM Surrogate when extracting thumbnails, for example. If you go to a folder with thumbnails enabled, Explorer will fire off a COM Surrogate and use it to compute the thumbnails for the documents in the folder. It does this because Explorer has learned not to trust thumbnail extractors; they have a poor track record for stability. Explorer has decided to absorb the performance penalty in exchange for the improved reliability resulting in moving these dodgy bits of code out of the main Explorer process. When the thumbnail extractor crashes, the crash destroys the COM Surrogate process instead of Explorer.

In other words, the COM Surrogate is the I don't feel good about this code, so I'm going to ask COM to host it in another process. That way, if it crashes, it's the COM Surrogate sacrificial process that crashes instead of me process. And when it crashes, it just means that Explorer's worst fears were realized.

In practice, if you get these types of crashes when browsing folders containing video or media files, the problem is most likely a flaky codec.

Source: https://blogs.msdn.microsoft.com/oldnewthing/20...
