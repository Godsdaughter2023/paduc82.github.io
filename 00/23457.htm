Aha... the very, very common "checkbox trap".

 

Don't use checkboxes! They aren't the correct design. You appear to have a perfectly normal Many to Many relationship here: each Ingredient may have zero, one, or several Allergens, and each Allergen may appear in zero, one or many Ingredients. Right?

 

If so you need THREE tables:

 

Ingredients

  IngredientID <autonumber primary key>

  IngredientName

  <other information about the ingredient>

 

Allergens

  AllergenID <autonumber primary key>

  Allergen <text, e.g. "gluten">

  <other information about the allergen as a thing in its own right>

 

IngredientAllergens

  IngredientAllergenID <autonumber primary key>

  IngredientID <Long Int, link to Ingredients>

  AllergenID <Long Int, link to Allergens>

 

If a given ingredient has four allergens you would have four rows in this last table; on your data entry form you would use a Subform with the IngredientID as its master/child link field, and a combo box to select which allergen.

graham mandeno say:

Hello Sarah

 

Designing a database is a very skilled task and there are many pitfalls, especially for the beginner.  A well-designed database should be like Grandma's pantry - a place for everything and everything in its place.  You will find some very good resources here to get you started:

http://www.accessmvp.com/JConrad/accessjunkie/resources.html#DatabaseDesign101

 

Pay particular attention to the articles on normalization.  Here is a particularly good one to get you started, but there are many more:

http://www.fmsinc.com/tpapers/genaccess/databasenorm.html

 

I suggest you start with a large sheet of paper and a sharp pencil.  Identify the different entities or “things” that you wish to model and write them down.  I can see that your list would include:

  • Suppliers

  • Contacts

  • Ingredients

  • Allergens

 

Each of these entities will become a table in your database.

 

For each entity, draw a box on your paper and write down in it a list of attributes or properties that apply to that entity.  For example, under Suppliers, you could write:

  • SupplierName

  • SupplierAddress

  • SupplierPhone

  • … etc

 

Each of these attributes will become a field in your table.

 

Be careful to ensure that the attributes relate ONLY to that entity – for example, the phone number to order an ingredient relates to the supplier, not to the ingredient.

 

Also, be careful to avoid repeating several attributes of the same or similar type – for example, Allergen1, Allergen2, Allergen3, etc in the Ingredients table.  This is an indication that a separate table – Allergens – is required.

 

As you write down the attributes, write (K) for “key” next to any attribute that should be unique in that table – for example, SupplierName and IngredientName.  Also, write (I) for “index” next to any attribute that you are likely to use for sorting or searching.

 

Next, look at your (K) fields to identify a primary key (PK) for each table.  A PK field MUST be unique, and should be:

  • short – preferably numeric, but maximum 10 characters if text

  • unchanging over time

 

SupplierName does not fit these criteria, because it is not short and it could change over time if a company changes its name.  If there is no suitable PK candidate, then add an ID field – SupplierID – which will contain an automatically generated number (AutoNumber), and make that your PK.  If in doubt, use an AutoNumber PK.

 

Next, decide how the different entities are related to one another.  For example, each supplier has one or more contacts, so there is a one-to-many relationship between Suppliers and Contacts.  Draw a line between the two boxes with “1” at the Suppliers end and “M” at the Contacts end.  You will also need a field in the Contacts table to act as the foreign key (FK) in the relationship.  This field will contain the PK value (SupplierID) of the supplier record to which the contact record is related – you might name it ContactSupplierFK.

 

Some of your relationships will be many-to-many (with “M” at both ends) – for example, an ingredient can contain more than one allergen, and an allergen can occur in more than one ingredient.  For this, you need a junction table containing TWO foreign keys – one for the IngredientID and one for the AllergenID.  You could name this table “IngredientAllergens” and name the fields “IngredientFK” and “AllergenFK”.  Cross out the M-M line, and replace it with two 1-M lines between the PK tables and the junction table.

 

Only now should you go to your computer and open Access.  Create a new table in Design View for each entity on your sheet of paper, and one for each junction table.  Add the fields to each table as required, starting with the PK field.  Click on the key icon on the ribbon to nominate the PK field.  Set the data type of each field appropriately – for an ID field choose “AutoNumber”.  For each (K) field, set the Indexed property to “Yes (No Duplicates)”, and for each (I) field, set it to “Yes (Duplicates OK)”.  For a FK field where the related PK is an AutoNumber, set the data type to “Number” and the Field Size to “Long Integer”, and delete the Default Value.

 

For the junction tables, create a composite primary key comprising both the FK fields.  Do this by selecting both fields and then clicking the key button.  The key symbol should appear beside both fields.

 

Note that you should avoid all spaces and special (non-alphanumeric) characters in your field names, and every field name should start with a letter.  The underscore character (_) is OK.  Also, avoid names such as “Name” and “Date”, because these have special meanings in Access.  (You can google “microsoft access reserved words” for a list.)  I use a convention of starting each field name with a short suffix which is unique to that table – for example, supID, supName, supPhone, etc.  That way I can instantly tell in my code which table a field belongs to, there is no ambiguity, and I’m very unlikely to stumble on a reserved word.

 

Some of your attributes might be split into several fields – for example, ContactName should be split into cntFirstName and cntLastName, and SupplierAddress should be split into supAddress1, supAddress2, supCity, supState and supZIP.

 

Once you have created all your tables, open the Relationships window and add all your tables into it.  Then, create the relationships by dragging each PK field to its corresponding FK.  In the Edit Relationships window that pops up, be sure to click “Enforce Referential Integrity” as this ensures that you cannot have “orphan” FKs without a corresponding PK.  For a relationship with a junction table, you can also choose “Cascade Delete Related Records”.

 

Then you can start creating queries to bring your tables together and forms for data entry and reports for data output.  Have fun!

 

Best wishes,

Graham

 

PS: For your list of allergens in an ingredient, setting up a list of all allergens with checkboxes is possible but difficult.  The easiest way is to use a subform linked to your junction table where you can select an allergen from a combo box.  You will see the list of associated allergens and you can add or remove items from the list by adding/deleting records in the junction table.

steve rindsberg say:

Strictly speaking, you can't do this.

However, there's a free viewer that Windows users can download and install; that would let them view your presentation. 

If you have Windows PowerPoint, you can choose File | Export | Package Presentation for CD.  This gives you the option of saving to a folder that you can burn to CD.  This may or may not automatically launch a web page with a link to download the viewer.  Depends on the user's settings.

You can also upload your presentation to OneDrive and share a link with the user.  The presentation will play in the online version of PPT, which doesn't have all the features of the desktop version but may be sufficient for your needs.  If you choose this approach, test your font choices first; the online app has limited font choices and you can't install any new ones.

Method 1

I would suggest you to check if you are able to access Windows Update services and also make sure that the following services are started. If it is not started follow the below mentioned steps: 

a. Press Windows key + R, type "services.msc" (without quotes) in start run box and press Enter. 

b. Locate Windows Update service.

c. Right click on the Windows Update service and select Properties.

d. In Start-up type, select Automatic from the list.

e. Click Start under Service status.

f. Click OK.

g. Repeat the steps c to f for the following services also:

Background Intelligent Transfer Service & Cryptographic Service.

If these steps do not help to resolve the issue, then you may proceed with the step mentioned below and check.

Method 2

Step by step instructions to get the issue fixed.
 
a) Press Windows key + X and select Command Prompt (Admin).
   
b) On the open Command Prompt window copy and paste the commands (all at once)
 
net stop wuauserv

net stop cryptSvc
 
net stop bits

net stop msiserver

ren C:\Windows\SoftwareDistribution SoftwareDistribution.old

ren C:\Windows\System32\catroot2 catroot2.old

net start wuauserv

net start cryptSvc

net start bits

net start msiserver

c) Wait for the process to complete.
 
d) Close the Command Prompt.
