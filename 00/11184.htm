Only scans that are run by the Windows Defender Scheduled Scan task will be interrupted by user activity (mouse or keyboard activity). That includes the daily Automatic Maintenance scan and (presumably) any scan that’s created by adding a trigger to the Windows Defender Scheduled Scan task – like the one suggested in the Microsoft article.

The only-on-idle priority that’s used by Automatic Maintenance is designed to prevent the scheduled scan from using system resources while you’re using the PC, and it serves this purpose very well for many users; but it doesn’t really run on a schedule in the same way that a classical scheduled task does.         Next shpNum
    End With

ken sheridan say: Arising from your last post there are a number of points:

1.  As you say there can be multiple synonyms per host species you cannot merge the synonyms and hosts tables.  There is a one-to-many relationship type from the latter to the former as your model correctly reflects.

2.  Data should never be stored as a value  list.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  Where the data represents a set of values which are immutably fixed in the external works, e.g. days of the week or months of the year, then the use of a value list is permissible, but that is not the case here, so you should have a HostCategories table.  This would be referenced by the HostSpecies table.

3.  So, one possible model would be like this:

CollectionRecords>----HostSynonyms>----HostSpecies>----HostCategories

In this model the CollectionRecords table consequently needs only a foreign key referencing a row of the HostSynonyms.  It then maps via the relationships to a single row in each of HostSynonyms, HostSpecies and HostCategories.  For data entry into CollectionRecords you would then need only a combo box for the foreign key HostSynonymID column as you show in your diagrammatic model.  The host name and category would then be shown in read-only text box controls in the form.  The frmContacts form in my NotInList demo is a direct analogy as this contains a city combo box, but the referenced region in which the city is located is shown automatically in a text box bound to the City column in the Cities table, to which the Contacts table is joined in the form's RecordSource query.  This could be extended to show a country for instance, if the database also contained a Countries table.

With this model inserting a new host species requires that a new row must be inserted into HostSynonyms first.  This can be done via the HostSynonymID  combo box's NotInList event procedure, which would open a form based on HostSynonyms, which would contain a combo box for selecting a host species value, the NotInList event procedure of this would similarly open a form based on HostSpecies, which would contain a combo box for selecting a category, whose NotInList event procedure would directly insert a row in to the HostCategories table in the way I described in my first reply.

4.  There is however another possible model, which is one where the CollectionRecords table would reference the HostSpecies table directly, i.e. the species would be identified by its formal taxonomic Latin name, so a row in CollectionRecords would map to all possible synonyms for the selected host species.

CollectionRecords>----HostSpecies>----HostCategory
                                          |-----------<HostSynonyms

In this model the combo box in the CollectionRecords  data entry form would be bound to a HostSpeciesID foreign key column, and the HostSpecies form opened from its NotInList event procedure to insert a previously unrecorded host species would have, in addition to a combo box bound to HostCategoryID, a subform, in continuous forms view, listing all the synonyms for the species in question.  The subform would be linked to the parent form on HostSpeciesID.

ken sheridan say: Firstly do not be surprised at the number of tables you have.  The more common mistake is to have too few and introduce redundancy.

The traditional Linnaean classification, being a tree in the mathematical sense, in that there is only one path between each node, is very straightforward to model as it is simply represented by a set of tables in one-to-many relationship types from Kingdom down to Species.  The one thing which does not fit into the tree is the Authority, as the relationship type for this is in the other direction, being many-to-one from Species to Authorities.  So having established the set of tables from Kingdom down to Genus for the types of species with which you are concerned, along with a table of Authorities in which each authority is represented by one row, a Species table would then have foreign keys GenusID and AuthorityID referencing the keys of the Genus and Authorities tables.  As genus determines Family, and Family determines Order and so on up the tree, each row in Species maps to a single row at every level of the hierarchy right up to Kingdom.

Modelling locations can similarly be done as a simple tree-structured hierarchy, provided that each level of the location is discretely contained within the next level up.  For this reason we use administrative geographical units, with civil parish as the smallest, ranging upwards through districts and counties to countries, the UK being composed of 3 countries, England, Scotland and Wales (which make up Great Britain) plus the province of Northern Ireland.  So only the civil parish need be recorded in any table of locations.  There is a fly in the ointment, however, in that urban districts have no parish structure, and districts which are unitary authorities are not administratively within any county, so it is necessary to have rows with N/A values at these levels to maintain the 'chain'.  You'll find an example of how to handle these sort of models by means of correlated combo boxes as ComboDemo.zip in my SkyDrive folder.

As regards the latitude/longitude references for locations, if you have a single intersection for a group of locations then I'd simply include two additional columns for the actual values, so you might have four columns, NominalLongitude, NominaLatitude, ActualLongitude, ActualLatitude.  Here we don't normally use longitude and latitude for location, but the Ordnance Survey National Grid Reference (NGR).  This, as its name suggests is based on a grid covering the whole of the UK, with its origin somewhere to the south-west of the Scilly Isles, these being the most south-westerly land in the UK.  Normally we recorded locations to a precision of 10 metres, though we had to pad this with two zeros as our GIS required values to a precision of 0.1 metres.  but being a decimal system it is very easy to compute to a lower precision then this by simple integer division of the eastings and northings.  It's one of the advantages of being a small country that a grid like this can be used as for most purposes we can, without undue distortion, project land surface over a flat plain, rather than having to use spherical trigonometry to allow for the curvature of the surface.

john - project say:

Kristopher,

That's the way Project works.

Project is a dynamic scheduling program. When you create a plan, and just before you are ready to start execution, you need to save a baseline (Project > schedule group > Set baseline). There are several baselines fields (e.g. Baseline Start, Baseline Finish, Baseline Duration, etc.) that store your original plan so that the data can be used for variance analysis after the program starts.

When you enter an actual start date in the Actual Start field, Project adjusts the plan to agree with reality. There is no point in having a active plan that still carries outdated information as part of the on-going activities. Right?
