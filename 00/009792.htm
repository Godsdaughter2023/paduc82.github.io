A little food for thought...  Think before you post. 

In your first paragraph you state that you have two tables, but you never tell what the two tables are.  Instead, you tell what field names seem to be involved in your scenario.

Understand that it makes it difficult to provide accurate and complete help if we don't have the accurate and complete details to your problem.

So, now that we have all that straight, on to your problem.

In your Registration form, you can place the following code to prevent a student from registering for more than three courses.

Private Sub Form_BeforeInsert(Cancel As Integer)
   
    If (DCount("StudentID", "Registration", "StudentID=" & Me.StudentID)=3) Then
        MsgBox "You have reached the limit of courses allowed.", vbExclamation, "Can't Add Another Course"
        Cancel = True
    End If
   
End Sub 

ken sheridan say:

Problem is: How do I ensure each student will only sign up for maximum of 3 courses?

You can do so via the interface by calling the DCount function, as described by the other respondents, and it is a good idea to do so.  This will not prevent an invalid row being inserted by other means, however.  More robust protection of the data integrity can be achieved by applying a CHECK CONSTRAINT to the table, which would done by executing the following code:

CurrentProject.Connection.Execute "ALTER TABLE Registrations ADD CONSTRAINT RegistrationLimit CHECK((SELECT StudentID FROM Registrations GROUP BY StudentID HAVING COUNT(*) > 3) IS NULL)"

Where Registrations  is the name of the table which models the many-to-many relationship type between students and courses.  To remove the constraint, e.g. if you need to change the limit, you'd execute the following:

CurrentProject.Connection.Execute "ALTER TABLE Registrations DROP CONSTRAINT RegistrationLimit"

You can execute the code from the debug window (aka 'immediate' window), but be sure that the Registrations table is closed when you do so.

dillopk say: OK, I found the answer, I will post it here for others.  If Internet Explorer is not able to save your downloaded file in its normal temporary file folder, where it normally goes when you click on RUN, there is no choice but to ask you where to save it, and the RUN button is not even presented.  In Tools > Internet Options > Advanced tab > scroll down to Security > you will find an option "Do not save encrypted pages to disk".  By default, this option is probably turned on, preventing the download to be saved to temporary disk and thus making it ineligible for the RUN option.  UN-check this box, APPLY the new setting, and try your download again.  This not only fixed my problem of not having the RUN button appear, but also fixed a problem downloading from a bank and a problem defining one-step update within Quicken.
