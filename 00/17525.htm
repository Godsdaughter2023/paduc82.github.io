sloopjohns say:

Hide or display vertical scroll bar via VBA code

I'd like to change the scroll bar visibility property of a subform based on the property of a field in the main form.

I have a form (f_Holding) that contains a subform (f_subholding).  Most frequently, the holding contains only one metal type (item), but at times it contains more than one.  To prevent errors, I am trying to limit entry on the subform to one line item UNLESS the user has indicated (via a check box) this holding is an alloy.  How would I use the "Alloy" check box on f_Holding to control the visibility of the vertical scroll bar of subform f_Subholding?

dirk goldgar say:

I'd like to change the scroll bar visibility property of a subform based on the property of a field in the main form.

I have a form (f_Holding) that contains a subform (f_subholding).  Most frequently, the holding contains only one metal type (item), but at times it contains more than one.  To prevent errors, I am trying to limit entry on the subform to one line item UNLESS the user has indicated (via a check box) this holding is an alloy.  How would I use the "Alloy" check box on f_Holding to control the visibility of the vertical scroll bar of subform f_Subholding?

I don't think controlling the scroll bar will necessarily give you the control you want, because there are several ways of moving to a new record on the subform without using the scroll bar.  I'll post the code for it if you like, but I'd recommend taking a different approach.

To my mind, it would be better to control the subform's AllowAdditions property instead.  You can set it up so that the subform only allows additions if either the subform is has no records yet, or the check box is checked.  Here's how to achieve this:

First, place this function in a standard module -- not the form's module:

'------ start of code for standard module ------

Public Sub LimitRecords(frm As Access.Form, Optional NoLimit As Boolean, Optional RecLimit As Integer = 1)

    If NoLimit Then
        frm.AllowAdditions = True
    Else
        With frm.RecordsetClone
            If .RecordCount > 0 Then .MoveLast
            frm.AllowAdditions = (.RecordCount < RecLimit)
        End With
    End If
   
End Sub

 '------ end of code for standard module ------

For the rest, I will assume that the check box on your form is named "chkAlloy".  Change that name where it appears in the following code to the actual name of your check box.

Create the following two event procedures in the main form's module.  If you already have code in these events, merge the code into those procedures:

'------ start of code for main form's module ------

Private Sub chkAlloy_AfterUpdate()

    LimitRecords Me.f_subholding.Form, Me.chkAlloy

End Sub

Private Sub Form_Current()

    LimitRecords Me.f_subholding.Form, Me.chkAlloy

End Sub

'------ end of code for main form's module ------

Finally, create the following two event procedures in the subform's module:

'------ start of code for subform's module ------

Private Sub Form_AfterDelConfirm(Status As Integer)

    LimitRecords Me, Me.Parent!chkAlloy

End Sub

Private Sub Form_Current()

    LimitRecords Me, Me.Parent!chkAlloy
   
End Sub

'------ end of code for subform's module ------

That ought to prevent anyone from adding more than one record to the subform unless the "chkAlloy" check box is checked.

john - project say:

PhilippeBruno,

Even though I have Project Pro, I do not use Project Server so I've never worked with Enterprise fields but I will tell yo what I can.

The macro code in the Wiki article sets task and resource assignment fields through a direct reference to the object (i.e. Task_As and Reso_As) property, namely the Text1 field property. However, Enterprise fields cannot be directly referenced in this same manner. In order to set values for Enterprise fields, the SetField Method is needed.

The FieldID argument of the SetField Method is a pjField constant. However, I find no pjField constants for the Assignment object, they only exist for Task and Resource objects. Based on that it appears that all you need to transfer Enterprise custom field values between task and resource is the simple form of the macro shown for Project versions prior to Project 2007. Of course you will need to modify the macro code to use the GetField and SetFields methods.

And to address your second question, the field name of the custom Enterprise field is not "Project Cost Type". That is simply the friendly name you gave to whatever Enterprise text field you used (e.g. Enterprise Task Text1). Although you can work with friendly names using the FieldNameToFieldConstant Method, it just makes the code more complicated. I suggest you reference the Enterprise custom field using the actual field name (e.g. ResourceText1 or TaskText1).

Hope this helps.

John

barbara henhapl say:

Hi,

John is correct for Task and Resource Enterprise Custom Fields: to set an ECF for Resources and Tasks needs "SetField".

Unfortunately, for Assignment Fields there is no "SetField". You need to set field on assignment level with Task_As.YourFieldName and Resource_As.YourFieldName. Unfortunately, for doing so, YourFieldName must not contain any spaces. I suggest to change your field name from "Project Cost Type" to "Project_Cost_Type" to be able to set this field.

After starting Excel, create a blank new workbook (or open an existing one).

Select File > Options.

Scroll down the main pane on the right.

Clear the check box 'Show the Start screen when this application starts'.

Click OK.

Quit and reopen Excel.
