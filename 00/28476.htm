These two groups are totally different.

Distribution groups (sometimes called distribution lists) have been the go-to solution for Outlook Web App users to communicate and collaborate with colleagues and partners. They're still a great solution for Outlook Web App users. Autodiscover SRV records - Microsoft Community

hughtc say:

Autodiscover SRV records

Hi,

we recently migrated from a CPanel-hosted IMAP system to Office365. I've done a few migrations before, so I was comfortable with the process. But things have been buggy since with Outlook refusing to start,  warning that 'Microsoft exchange info service in your profile is missing required information.'

Turns out that although I'd implemented the required Autodiscover CNAME record, I'd left in an SRV record, ie

_autodiscover._tcp.ourdomainname.com.au. 14400 IN SRV 0 0 443 cpanelemaildiscovery.cpanel.net.

That URL was being accessed by Outlook 2016 (Mac and Windows), which, on PC, would then cache the result of the query in the %localappdata%\Microsoft\Outlook\16 directory, leading to a mismatch between the saved Exchange account and the result of the lookup, which points back to the old server.

The symptom and a fix (but not he possible cause) is discussed in this thread

If Outlook 2016 is using this URL, there are 2 possible solutions that I can see:

1. Delete the DNS record in question (which seems a bit daft, as Outlook 2016 is obviously using it) 

2. Reconfigure it to point to the correct Office 365 server 

Option 2 seems the most logical, but I haven't seen any documentation on what the URL should be; it's not in the 'required DNS records' section of the setup wizard. 

thanks,

Hugh
