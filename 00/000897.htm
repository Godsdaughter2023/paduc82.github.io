With Me.RecordsetClone

     !fieldb = Me.fieldb    'repeat for each record source field (not control) you want

   .Update

  Me.Bookmark = .LastModified

ken sheridan say: You are quite right to want to build the model from scratch, but you can, with a little care, do so without any data loss.  The first thing to do is to work out the appropriate logical model in terms of the entity types which make up the underlying reality, the relationships between them, and the attributes of the entity types.  Each entity type will be represented by a table, and each attribute by a column in a table.  It is important that there is no redundancy in the attributes, i.e. each 'fact' is stated once and once only in the database.  This is important not merely for efficiency, but because it ensures the integrity of the database by preventing update anomalies.

The process of eliminating redundancies is known as normalization, and is achieved by decomposition of a non-normalized table into a set of correctly normalized tables.  You'll find an example as DecomposerDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file imports data from Excel, but that is not material to what you intend as you are starting with data in Access.  In my demo the data is first imported into a single 'master' table and this is then decomposed into a set of normalized related tables.  The model is as below:



I'd draw your attention to two points of this simple model:

1.  The relationship type between Contacts and Employers is a many-to-many relationship type, which is resolved into two one-to-many relationship types by the ContactEmployers table.

2.  The address data is a tree-structured hierarchy of Cities, Regions and Countries, so only the lowest element in the hierarchy, Cities, is referenced by the referencing table Contacts.  If Contacts were also to reference Regions and/or Countries directly this would introduce redundancy into the table so it would not be normalized to Third Normal Form (3NF) and would be at risk of update anomalies as the same city could be incorrectly recorded in separate rows the table as being in different regions and/or countries.

Once you have decided on the correct logical model and are satisfied that it represents the underlying reality without any redundancies, you can create the tables and the relationships between them.  In each relationship referential integrity must be enforced; an unenforced relationship serves no purpose whatsoever.

The process of decomposition is achieved by executing a set of 'action' queries.  My demo takes you through this step by step with some sample data.  The rule of thumb is that rows are inserted firstly into the referenced tables (those on the 'one' side of a relationship) before rows can be inserted into the referencing tables (those on the 'many' side).

In addition to decomposing the data into a set of related tables, you will also need to build the user interface.  The final stage in my demo is the opening of a contacts form with an employers subform.  Note how in this form (a) only the city combo box is bound to a column in the underlying table, the region and country combo boxes being unbound; (b) the multiple employers per contact are shown in the subform as separate rows.  If you look back to my paragraphs 1 and 2 above you'll see that this reflects the two points to which I draw your attention with regard to the model, i.e. the form/subform interface is a reflection of the underlying model.

I have deliberately not addressed your data specifically in this reply as it is the underlying principles and the methodology of decomposition which I wish to describe as far as it is possible to do so in the limited time and space available here.  You will understand the nature of the reality which you are modelling far better than we can, and are consequently in the best position to come up with the correct logical model - provided that you follow the advice given to his students by Richard Feynman in the quote below my signature.

Fix for Lumia 521 rebooting unexpectedly - Microsoft Community

lisa66---01 say:

Fix for Lumia 521 rebooting unexpectedly

My new lumia 521 starting rebooting itself unexpectedly.   Screen would go black, then I would get the Nokia splash screen followed by the T-Mobile splash screen then Windows Phone start screen.   I noticed it happened mostly after the phone was physically moved such as when I set it down on a table or moved it quickly in the air.  I suspected the problem was the battery was becoming unseated.

 

Nokia Support live chat suggested that I buy a new battery.  What I did instead was I took a kleenex tissue and folded it until it was about 8 thicknesses deep.  Then I cut a piece about 1 inch by 2 inches and I used scotch tape to cover the black cushion that is on the inside of the back cover.  Essentially, I made the cushion thicker.  Now the battery is held in place much more securely.  I  had to experiment with how many tissue thicknesses I could put over the existing cushion so as not to interfere with the side buttons.

 

Problem solved - no more surprise reboots.  Much faster and cheaper than buying a new battery or sending the phone back for service.

 

It is annoying though that the phone came with a back cover that doesn't hold the battery in place securely.  Nokia take note:  this appears to be a design flaw that could be remedied.

 

Hope this helps anyone else experiencing the same problem . . .
