Ms Access Front end to mysql Database Back end. - Microsoft Community

sherifarafa say:

Ms Access Front end to mysql Database Back end.

I have built a ms access front to a mysql backend and connected using the ODBC driver, everything is working perfectly but the problem is that the connection parameters is saved in mysql ODBC data source used to connect ms access Front End to mysql Back End. Accordingly anyone can open a new access file and use the same ODBC data source with the (saved connection parameters) to get to my mysql database tables and mess with my raw tables, delete data, modify data. So my Question is: Is there a way to connect to mysql DB from a ms access Front End via a connection string or VBA code that stores all connection parameters ? So only authorized and protected ms access files would be able to access mysql database? instead a having an ODBC data source that anyone can use to get to the raw data and change it.

albert d. kallal say:

Yes, you can.

However, if users can use + view the data, then they have the ability to view + use such data!

So there is not a lot you can do. You can remove the user id + password from the connection string, and at application start-up have the user enter their id + password. You then execute a logon, and you do NOT have to re-link the tables when you do this.

I mean, if you allow ODBC to your database, then Excel users with the correct userID + password can use that data. So essentially your question is how to prevent Excel or even word users from using that database?

You could create a MySQL user that you DO NOT include in the connection strings. Then on application start-up you execute that logon and NOW all your connection strings (without UID + password) will work.

Thus if a user attempts to import the connection strings to another application, those connection strings will NOT work since they don’t have nor include the UID/password that is required.

So about the best suggestion would be to NOT include the UID/password in ANY of your connection strings, and then have the application on start-up execute a logon to the database with ONLY your application supplying that unknown logon + password.

ONCE you logged in then ALL OF your connection strings will now work like magic, and work without the password/User included into those connection strings. And this also means you don’t have to re-link the front end if you change that password.

I use the above approach for all and any Access applications that connect to SQL server, and the same approach works for MySQL.

How this works is outlined here:

Power Tip: Improve the security of database connections

http://blogs.office.com/b/microsoft-access/archive/2011/04/08/power-tip-improve-the-security-of-database-connections.aspx
