dirk goldgar say:

This is made harder -- or at least, harder to maintain -- because your data are structured vertically rather than horizontally.  A normalized relational structure for this would be:

PersonName TestName RestResult
You Test1 Fail
You Test2 Pass
You Test3 Fail
You Test4 Pass
Me Test1 Pass
Me Test2 Pass
Me Test3 Fail
Me Test4 Pass
He Test1 Fail
He Test2 Pass
He Test3 Fail
He Test4 Fail
She Test1 Pass
She Test2 Pass
She Test3 Fail
She Test4 Pass

This is glossing over the possible use of numeric keys for people and tests, rather than text fields, and the possible encoding of "Pass" and "Fail" as boolean values -1 (True) and 0 (False).

But regardless of that, with a vertical structure like that, the "Fail count" of each person could be easily retrieved by a query like this:

    SELECT PersonName, Sum(IIf(TestResult = "Fail", 1, 0)) As NumberFailed
    FROM TestResults

What becomes harder with this structure is getting the results each test to show in a column rather than as a row.  To get that, you would use a crosstab query.

ken sheridan say: I think it would help if you were to post the SQL statement for your current query.  The query Dirk gave you computes the number of failures per person by a process of summation, which is how it should be done with a correctly designed table, whereas I think you will need to do so by a process of addition.  Mathematically these are not the same thing.  Essentially what you will need to do is add the results of a fixed number of multiple expressions, each of which returns a value of 0 for a pass, or 1 for a fail, but without knowing the expressions you are using currently it's not possible to say more than that.
