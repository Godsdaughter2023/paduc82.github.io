001-11

002-11

.

078-11

where 11 - 12 are from the year (2011, 2012, 2013).

hellosaw say: Hello RubaAbo Arrob,


You can try the Workflow workaround to your highlighted scenario:
  1. Create a custom list called Autonumbering List, rename the Title field to NextNum
  2. Create two calculated columns: FollowingNum; NextID
  3. FollowingNum: Enter the formula 
    =NextNum+1
  4. NextID: Enter the formula 
    =LEFT("000",3-LEN(NextNum))&NextNum&"-"&RIGHT(YEAR(Modified),2)
  5. Create a workflow and configure to start on item creation in the list you want autonumbering. 
  6. In the workflow, get the NextID and saves it as a workflow variable: Numbered, then update list item in the Autonumbering List, setting NextNum equal to FollowingNum. 
  7. Then set the Title (or other field) of the list you want autonumbering to the workflow variable: Numbered (the value of the NextID from Autonumbering List from before we changed it).
PS: The Autonumbering List is a separate list from your desired list where you want autonumbering.

daniel g. grau say:
It is synchronous if you use an interpreted script, as it needs to return a result. This is the case for MacScript("do shell script blah, blah, blah"). However, if you compile a script to create an application, you can launch and forget. For instance:

  • MacScript ("tell application ""Finder""" & vbCr & "open application file ""Macintosh HD:Applications:My Scripts:My Application.app""" & vbCr & "end tell")

where "My Application.app" could take its input from a file and store it in a variable:

  • set InputData to do shell script "cat  /Applications/My\\ Scripts/Input.txt"

and could create a file to store any output:

  • do shell script "blah, blah, blah … | cat >/Applications/My\\ Scripts/Resuls.txt"

ken sheridan say: When you say 'the form allows data to be entered into several table through one form', is the form bound to a query in which the tables are joined, or does the form contain subforms in which data is entered into the referenced  tables?  The latter is strongly recommended, but in some cases the former is possible, albeit not advised.  

Where data is being inserted into more than one table via a query which joins the tables, it is essential that the relationships are enforced, and the relevant primary and foreign key columns are appropriately indexed.  This must be the case in the back end.  While relationships can be shown in a front end they do nothing apart from setting the default join type when a query is built in design view.  It is the relationships in the back end which are important to the integrity of the database.

If, when splitting the database you created base tables in the back end, and links to them from the front end, but did not create relationships between the base tables, this might explain the behaviour which you are experiencing.  Even if this is the source of the problem, and is rectified, I would still emphasise that this is not recommended as an interface for inserting data into multiple related tables.

ken sheridan say: As Duane has said, experienced Access developers would not normally use the crude interface of a subdatasheet, but would design forms and subforms, where the latter is based on a source form object.  Whereas a subdatasheet will show by default in the case of a local table's datasheet view, in the case of a linked table this is not the case.  So, if you want to use this view in your form you will need to set this up manually as Duane described.

If you try to open a linked table in design view, you will get a warning message, but opt to go ahead.  While you cannot amend the table definition per se, you can amend its SubdatasheetName property, which by default will be [Auto], to the name of the referencing table.
