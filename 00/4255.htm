jetta48 say:

Assuming that McAfee has been successfully removed then to turn Windows Defender on:

 

1. On Start screen, begin typing Windows Defender until it appears on the left then left-click on it.

2. Select Settings tab.

3. Tick Turn on real-time protection (recommended).

4. Click Save changes.

5. Click Update tab then Update button then wait for the latest updates to be applied.

6. It would now be sensible to do a full-scan to start with, so click on Home tab then select Full then click Scan now.

7. Wait for scan to finish and action results as necessary.

(full scans may take some time)

8. Select Home tab and you should see a green computer monitor screen with a white tick in it with the words "Windows Defender is monitoring your PC and helping to protect it." Underneath that it should be reporting that "Real-time protection:" is "On" and that "Virus and spyware definitions:" are "Up to date". At the top left of Windows Defender it should also say "PC status: Protected". If you see these things, everything should be OK with Windows Defender.

 

 

Don't forget to make sure you have a working firewall. The built-in Windows Firewall is sufficient for most users - just check its turned on in Control Panel.

10. Check the new library and see if the files and version histories are moved to the new library.

Tables relationships in Query - Microsoft Community

bkny4321 say:

Tables relationships in Query

Using Access 2010.  I have two tables. 

Table 1 - Members contains Member ID (autonumber/yes no duplicates), Member Number (text/yesterday no duplicates) either of which can be set as primary key as well as name, demographic information as well as status and member type. 

Table 2 - Dues contains a Dues ID (autonumber/yes no duplicates), first name, last name, year dues were paid, the date dues were paid and the amount of dues paid.  The dues would be inputted year after year for each member.  So there will be many records (one for each year of dues paid) for each member. 

I made a subform with a combobox to pull up a list of the member's first and last name to then enter the dues information but for some reason I can't seem to pull information on the query from Table 1 and Table 2. 

I want to set up a relationship between these two tables but I am not sure I did this correctly to build a query that will list the members name, status, member type and the date and amount of dues paid by pulling this information from the two tables. 

Something is not right.  I keep getting a "type mismatch in expression error."   I am a newbie to Access so some help would be appreciated. 

Thanks


tom van stiphout say:

tblDues should not have Firstname, Lastname, but rather MemberID, long integer, required.

 

Btw, avoid spaces in table and field names; they cause notational difficulties.

 

Then go to the Relationships window, put both tables on the design surface and draw the line from tblMembers.MemberID to tblDues.MemberID, and in the dialog that comes up check the box to enforce referential integrity.

 

Once that is done, select the Members table in the Navigation pane, and on the Create tab of the ribbon click on Form. The wizard will create the form for members and subform for dues.

scottgem say:
As Tom noted, you don't have memberID in your Dues table. Instead of the names, you should just have the MemberID as a foreign key. The name info belongs in the member table. 




bkny4321 say: Unfortunately, this doesn't create a one-to-many relationship, it creates a one-to-one relationship (the problem I was running into).  I will have many dues records (one for each year) for each member in the member table.  I don't want to be having to put each member's information in manually in the dues form.  I want to be able to make the subform so that it pulls in the Identifying demographic information from the members table but it does not seem to be doing that. 

For example I want the form to be something like this:

   Top of form has:  Member ID, member #, name, status, member type and start date  (comes from table 1).

   Subform ties to top of form and contains the information which will be entered each year which will include:

          Some type of identifier that ties to top of form (like member # and name)
         
          Fiscal Year, Date Dues Paid and Amout of Dues Paid. (table 2)

Does this make any sense?

scottgem say:
If it creates a 1:1 then you have set the MemberID column in the Dues table as either a PK or Unique. That's not what it should be.

What you should be doing is using a subform. Your main form is bound to the Members table. The subform to the Dues table and linked on MemberID. The main form will display your Member demographic info, the subform the dues record for the currently displayed member. 

This is all pretty much SOP.


bkny4321 say: I found the error.  It was the way the dues table member ID was indexed.  That was fixed and then I was able to make a 1 to many relationship with the member table member ID.  That was my whole problem.  From there on smooth sailing.   Thanks a bunch.
