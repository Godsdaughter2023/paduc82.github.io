scott landry say:

When you add a domain to Office 365, you are telling Office 365 that it should accept email for that domain (that is, specifically any email destined TO the domain).  Simply accepting a domain doesn't have anything to do with open relay -- it simply means that the domain is accepted for delivery.  Once added, the domain can be fully hosted (authoritative) or can allow relay to an on-premises server (relay domain).  If the domain is set to relay, then you also should have an "outbound connector" which tells Office 365 where to send the email which is intended for any domain which you have told Office 365 to accept email for.  This particular configuration is often used by customers who have mailboxes which are hosted on premises but wish to have Exchange Online Protection provide anti-spam capabilities for their domain.  If the domain is set to authoritative, then a match must exist in the Office 365 directory, whether the mailbox is hosted on-premises or in Exchange Online.

As for the MX record, that is controlled by you and tells you whether you want EOP to do the scanning or your on-premises environment (a common misconfiguration is to have multiple MX records, only some of which are pointed to Office 365, something which should generally be avoided).  These two pieces control the inbound behavior for messages going TO your domain.

Some customers also choose to route messages from on-premises out through Office 365 -- or, in the case of hybrid, to internal mailboxes which happen to be hosted in Office 365.  To configure this, an "inbound connector" is created.  This allows authenticated connections (IP address or certificate based authentication) to relay mail TO any domain, whether hosted or not.

As far as FROM goes -- that is far more complex.  Basically any SMTP server which needs to accept unauthenticated Internet email will accept email FROM any domain.

Some customers also choose to route messages from on-premises out through Office 365 -- or, in the case of hybrid, to internal mailboxes which happen to be hosted in Office 365.  To configure this, an "inbound connector" is created.  This allows authenticated connections (IP address or certificate based authentication) to relay mail TO any domain, whether hosted or not.

As far as FROM goes -- that is far more complex.  Basically any SMTP server which needs to accept unauthenticated Internet email will accept email FROM any domain.  Even a server which you would only expect to originate email for a particular domain may need to accept email FROM that domain as well -- situations like listserv groups, forwarding, etc. all have to be supported.  However, most respectable servers should do basic authentication checks.  In Office 365, there are 4 basic types of sender authentication methods: 

  • Hosted mailboxes can authenticate directly with smtp.outlook.office365.com and send authenticated email using SMTP AUTH w/ TLS.  Only mailboxes which are authorized to send as a user are allowed to send as that email address.  No spoofing at all should be supposed with this method.
  • The other two cases -- EOP treats on-premises servers and non-customer external parties largely the same: the sender is not explicitly trusted, but can prove authorization using SPF and/or DKIM and/or DMARC (none of which is explicitly required, but we highly recommend using these as appropriate).  Certainly there are many other variables used by our spam engine -- SPF and DKIM aren't a free pass; but for proving domain authorization, these are important variables.

Hopefully this helps. Don't hesitate to ask follow up questions, but it would be good to know what you're trying to accomplish / prevent and why.

Regards,
-Scott
