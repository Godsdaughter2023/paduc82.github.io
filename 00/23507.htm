ken sheridan say:
I only want the option to type a new supplier or select one from the list, not open a new form.

With respect, in a situation like this it's not a question of what you might want to do, but want you need to do to achieve the objective.

It sounds like you might have used the 'lookup field' wizard when creating the table to which your form is bound.  This feature is deprecated by most experienced developers.  For reasons why see:

http://www.mvps.org/access/lookupfields.htm

Even if you did not use this feature your foreign key SupplierName column's data type is almost certainly a long integer number which references an autonumber primary key of a Suppliers table in which a text data type non-key SupplierName column or similar stores the name of each supplier.  The combo box bound to the SupplierName foreign key column in the form's table should be set up like this:

RowSource:     SELECT SupplierID, SupplierName FROM Suppliers ORDER BY SupplierName;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

If your Suppliers table includes no other non-key columns than SupplierName then you don't have to open a form via code in the combo box's NotInList event procedure to insert a new row into the Suppliers table; you can do so entirely in the code.  You'll find an example of how to do this in the RegionID combo box in the frmCities form in my NotInList demo.

It's hard to conceive that no other attributes than the supplier name would be stored for a supplier. What about their address, phone number, email address etc?  If there are such other attributes then you really have no option but to open a form in dialogue mode to do so when a new name is typed into the combo box as at least some of those columns are likely to be 'Required', and disallow Nulls.  The example of the city combo box in my demo to which I referred you earlier illustrates how to do this with code in the combo box's NotInList event procedure, and the dialogue form's Open event procedure (frmCities in my demo).

hansv mvp say:

ActiveDocument.SpellingErrors is the collection of the ranges that contain spelling errors. You can loop through this collection and store the text of those ranges in an array or in a string variable, or do something else with it. For example:

Sub ListErrors()
    Dim docCur As Document
    Dim docList As Document
    Dim i As Long
    Set docCur = ActiveDocument
    Set docList = Documents.Add
    For i = 1 To docCur.SpellingErrors.Count
        docList.Content.InsertAfter docCur.SpellingErrors(i).Text
        docList.Content.InsertParagraphAfter
    Next i
End Sub

This is just to give you an idea, of course

My issue was that my account was not select to be included in the Send/Receive.

On the Send/Receive tab, click Send/Receive Groups and select Define Send/Receive Groups.
