bill mosca say:

It's not really a one-to-one. It's just an outer join. If you set up a relationship in the relationships window you will see the one-to-many join, but it doesn't mean anything. It's jsut fluff. The query will act the same way without it.

If you want to see All the programs in the program table, double-click on the join line between it and the Courses table and choose "All records from Programs and only matching records from Courses.

But I'm worried about your Groups-to-Courses relationship. Wouldn't the Program need groups of courses instead of individual courses? If not, what is the purpose of a group?

graham mandeno say:

A relationship with referential integrity will show as one-to-many if the foreign key field either has no explicit index, or if its index allows duplicates.  If the foreign key field has an index disallowing duplicates then the relationship will show as one-to-one.

Check your Courses.ProgramID field to see if it has a unique index.  You presumably want to allow more than one course per program!

Furthermore, could a course be included in more than one program?  If so, then you actually have a many-to-many relationship, which must be implemented with a junction table and two one-to-many relationships:

Table ProgramCourses:  ProgramID (1-M to Programs table), CourseID (1-M to Courses table)

Best wishes,
Graham

graham mandeno say:

OK, so a Group is defined as "a group of people attending one presentation of a particular course", and no group can attend more than one course.  Fair enough :-)

The problem with your totals query is almost certainly that you are using the Count() function on a select query involving all three tables.  This gives you a count of records in the select query, not a count of records in the source table, so your count of "courses" will always be the same as your count of "groups".

For the count of courses, use a subquery:

NumCourses: (Select Count(*) from Courses as x where x.ProgramID=Programs.ProgramID)

Sub Test3()

  Set Oo = ActiveSheet.OLEObjects(1)

In addition to the knowledge base article issue, this may well be just from the way Access handles design changes to the structure of already-populated tables. It makes (or tries to make!) two complete copies of the entire table in memory - and you don't get the full physical memory of your box either.

 

I would suggest a two step process: 1. Copy the table to a new table, design mode only (no data), and change the field types and sizes in the empty copy; 2) run an Append query to migrate the data from the old table to the new.

 

You can then rename the new table, reestablish relationships, and delete the old table.
