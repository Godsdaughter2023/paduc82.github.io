joeu2004 say: Viljoet wrote:

I'm trying to run the following, rather simple, VBA code:

Sub Simple_Loop()
    Dim dToday As Date
    Dim i As Integer
    dToday = DateSerial(Year(Now), Month(Now), Day(Now))   
    For i = 1 To 24
        ThisWorkbook.Sheets(1).Cells(i, 2) = dToday
        dToday = dToday + 1 / 24
    Next
End Sub

For some reason, on the Excel sheet where I print the results, I get funny results. When the timestamp should increase from hour 23 to hour 0 on the next day, the result is hour zero on the same day.
[....]
Can someone explain what is going on, because this looks like a bug to me?


Perhaps.  Apparently, when we assign a Date variable to a Range variable, it is as if we wrote:

Cells(i,2) = Int(dToday) + TimeValue(Format(dToday, "hh:mm:ss"))

As Bernie conjectured, when we add 1/24 incrementally, especially to a relatively large number (today is 40695), there can be an infinitesimal anomaly (not really an error) due to binary floating-point limitations.  See http://support.microsoft.com/kb/78113.

After adding 1/24 to 40695 for 25 times, the result is about 40695.9999999999, not 40696.  Format(0.9999999999,"hh:mm:ss") rounds seconds, then truncates minutes and hours.  Thus, the result is 00:00:00.

Arguably, the defect is rounding the time part without carrying over any excess into the date part, as Format(dToday,"m/d/yyyy hh:mm:ss") does.

Note that it does not matter how you increment a date/time serial number by adding one hour -- although the DateAdd could have been smarter about it.

Doing Cells(i,2) = CDbl(dToday) instead is not an ideal alternative.  It simply retains the infinitesimal computational anomaly.  Consequently, if you use the Custom format "m/dd/yyyy hh:mm:ss.0000" without quotes, you will see the time 09:59:59.999 starting in row 2387 (Sep 8 2011) if you add 1/24 that many times.

Bernie's suggestion of computing dToday + (i-1)/24 works because that is how Excel converts hours to a date/time serial number.

But in general, it is best to compute n/m instead of summing 1/m for n times.  This is not a "work-around".  It is good numerical practice.  Nevertheless, the potential for infinitesimal computational anomalies still exist.

BTW, instead of doing dToday = DateSerial(Year(Now),Month(Now),Day(Now)), it is better to do dToday = Date.

Besides being more efficient, errors can arise when calling Now multiple times near midnight.
