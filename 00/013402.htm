While this file illustrates a simple invoicing application the model would be the same if you substitute orders for invoices. Forget the Payments table for the moment and concentrate on the many-to-many relationship type between Invoices and Products which is modelled by the InvoiceDetails table.  This resolves the relationship type into two one-to-many relationship types.

A few points to note:

1.  The Invoices table duplicates the address data from the Customers table.  At first sight this might seem like redundancy, but it is not.  The model assumes that in the event of a customer changing address the address at the time an invoice was raised should remain static with the invoice, i.e. in database terms, the customers current address is functionally determined by the key of Customers, the address per invoice is functionally determined by the key of Invoices.  The current address from Customers is automatically inserted into Invoices when a new invoice is raised by means of code in the InvoiceDetails subform's module.  This is not mandatory, however.  You might prefer prior invoices to show the customer's current address rather than their historical address, in which case the address data would not be stored in Invoices.

2.  The same applies to both UnitPrice and TaxRate, which are columns in both Products and InvoiceDetails for the same reason.  In this case, however, this is mandatory as each invoice should store the unit price and rate of tax (in the demo VAT, the European tax on the supply of products or services) which was current when the invoice was raised.

Your model should be broadly similar to this, but if you need to import data from Excel as well as/in addition to entering it directly in Access, the question arises as to how the data from Excel can be imported into such a model.  If you have multiple rows in the worksheet, one for each item ordered, then the data can be imported into a temporary 'holding' table in access and then be decomposed into a set of correctly normalized related tables along the lines of the above.  The Decomposer.zip demo illustrates the generic methodology for doing this by means of a series of update queries, taking you through each step by step, with a brief explanation of each stage.  The rule of thumb is that rows are first inserted into the 'referenced' tables (those on the 'one' side of each one-to-many relationship type), and then into the 'referencing' tables (those on the 'many' side).  It's more than a simple cut and paste I'm afraid.

To check that the mailbox is being blocked, open the Exchange Admin Center in your Office 365 portal, click on Protection and on Action Center.

You will see the user listed there with an unblock. Do not unblock the user for now.
In the Protection screen, click on dkim and highlight your external domain. Click on Enable. You will get an error message that CNAME records required are not found.

Open your domain DNS management portal on your hosting company and add the following CNAME entries

Host name: selector1._domainkey.<domain>
Points to : selector1-<domainGUID>._domainkey.<initialDomain>
TTL: 3600

Host name: selector2._domainkey.<domain>
Points to : selector2-<domainGUID>._domainkey.<initialDomain>
TTL: 3600

Once your DNS records have propagated, click on the Enable button on the dkim section.

Once enabled you can go under the Action Center and unblock the user. The process may take up to 2 hours to be cleared.

This will protect you from email message header spoofing. On another note to know immediately if a user has been blocked, you need to setup a notification as below.

Under the Exchange Admin Center open the Protection/ Outbound Spam section. Double click on default. Click on Outbound Spam preferences and tick send a notification when a sender is blocked as below and enter the admin email address. Click Save.

john w. vinson say:

If you're using an Access database in multiple cities, you've got much bigger problems than ULS. Access is NOT suitable for use over the Internet; it's much too demanding of speed and rock solid stability of the network connection. You could have a database (split, into a frontend with the forms, reports, queries and code and a backend with the tables) on the server, and use remote terminal connections such as Citrix  or Microsoft's Remote Terminal Service. Better would be to store your data in Sharepoint or SQL/Server and link to it from an Access frontend.

For the city/city security you could have each user link, not to the Table, but to a View showing only Boise data to the Boise users, only Poughkeepsie to the folks from there.

You may want to seriously consider ditching Access and going with a fullfledged web application.

A) Try repairing your update components

1. Disconnect your internet and turn off your wifi

2. Reboot your pc and login

3. Right-click start, Command prompt (admin)

4. Copy and paste all these commands into the DOS window and press enter

net stop wuauserv
net stop cryptSvc
net stop bits
ren C:\Windows\SoftwareDistribution SoftwareDistribution.old
ren C:\Windows\System32\catroot2 catroot2.old
net start wuauserv
net start cryptSvc
net start bits

5. Reconnect your internet and turn on your wifi

6. Reboot your pc and login

B) Try uninstalling dotnet 3.5 (.net 3.5) if you installed it

Right-click start, Control Panel, Programs and Features

Turn windows features on or off

Untick [ ] .net Framework 3.5

Reboot your pc and login

C) Change the registry key using windows + R keys (run) and type regedit

Change the entries from fbl_Impressive to TH1

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\WindowsSelfHost\Applicability

Ring = WIF
BranchName = TH1
UserPreferredBranchName = TH1

2. Run windows update and it might download and install TH1 Professional Windows build 10240.

D) or click start, settings, Update & security

Windows update

Advanced Options

Change from SLOW to FAST or FAST to SLOW

click back arrow

Click [Check for updates]
