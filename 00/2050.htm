As far as the filtering is concerned, this problem may be a result of the converter (or Word) determining that a column is "text" when it is numeric, or vice versa. 

However, I haven't been able to reproduce the exact problem here yet - each time I produce a similar result, Word pops up a message that says  "This Word document contains outdated sorting information. Word is removing it." then reverts to the criteria you had previously. In the case where you had no criteria previously, Word merges all the records.

For example if you have a column that contains mainly numbers, but with a single text value, it looks as if Word regards the column as "text". So if the column is called M_2012 and has values 1, 2, 3 , 10 20, 30, a filter criterion such as M_2012 > 1 will select the values 2, 3, 20 and 30 but not 10 as you might hope. (Something similar happens in Windows Word, but the situaiton is arguably better known and understood on that platform.)

But it may be useful if you can open the VB Editor to display the query that Word has saved. To do that, you may have to 
 a. save everything and restart Word (I have found that Word sometimes crashes during this process)
 b. open your Mail Merge Main Document
 c. use Word->Preferences->Ribbon to enable the Developer tab
 d. click the Editor button in the developer tab
 e. use View->Immediate Window to reveal the Immediate Window
 f. in that window, type

?Activedocument.mailmerge.datasource.querystring

and press Enter.

If you defined a filter such as M_2012 > 1, the SQL Word displays will either be something like

SELECT * FROM something:thedatasource.xlsx WHERE ((M_2012 > 2))
or
SELECT * FROM something:thedatasource.xlsx WHERE ((M_2012 > '2'))

In the first case, you have a numeric comparison, which is what you need here. In the second case, it's a text comparison that won't work correctly for numbers.

The reason that doesn't explain things completely is because I don't see a situation where the SQL that Word creates will always match the PAYE value (unless the criterion has not been applied). But the other thing to bear in mind is that the SQL is created when you apply the criteria. If the data has changed when you next open the Mail Merge Main Document, the data types perceived by WOrd may have changed and the SQL's comparisons may be the wrong type for the data. hansv mvp say:

The following custom function was created by sdckapr (Steve Aprahamian):

 

Function VLookupAll(vValue, rngAll As Range, iCol As Integer, Optional sSep As String = ", ")
    Dim rCell As Range
    Dim rng As Range
    On Error GoTo ErrHandler
    Set rng = Intersect(rngAll, rngAll.Columns(1))
    For Each rCell In rng
        If rCell.Value = vValue Then
            VLookupAll = VLookupAll & sSep & rCell.Offset(0, iCol - 1).Value
        End If
    Next rCell
    If VLookupAll = "" Then
        VLookupAll = CVErr(xlErrNA)
    Else
        VLookupAll = Right(VLookupAll, Len(VLookupAll) - Len(sSep))
    End If
ErrHandler:
    If Err.Number <> 0 Then VLookupAll = CVErr(xlErrValue)
End Function

 

Use like this:

 

=VLookupAll(K1, A1:C25, 3)

 

to look up all occurrences of the value of K1 in the range A1:A25 and to return the corresponding values from column C, separated by commas.

 

If you want to sum values, you can use SUMIF, for example

 

=SUMIF(A1:A25, K1, C1:C25)

 

to sum the values in C1:C25 where the corresponding values in column A equal the value of K1.
