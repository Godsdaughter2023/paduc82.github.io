john w. vinson say: This is probably an example of a "Bill of Materials" database - a recursive structure where a Recipe can consist of one of many Ingredients, but any of those Ingredients may itself be another Recipe, which in turn may have ingredients which are themselves Recipes. This can be done, but it's far from trivial;  google for the term (or BoM databases) for examples. But bear in mind that the "recipe" for building such a database is more like a cordon bleu gourmet dish than it is like steak and fries!

ken sheridan say: As John says, this is analogous to a bill of materials.  The conventional model for this is to have a table of all parts(ingredients in your case) and an 'adjacency list' table which has two columns, each referencing the primary key of the first table.  The important thing to understand is that a part can be not only an indivisible base part, but a sub-assembly, which would be analogous to your base recipes, so the rows in an ingredients table would include both base ingredients and base recipes.  The relationships via the second table would enable any base recipe to be mapped to its constituent ingredients, which could be base ingredients (e.g. flour, milk etc) and/or a base recipe (e.g. a basic white sauce).

The problem is that to generate a 'bill of materials', or a 'parts explosion' as it's sometimes called, to an arbitrary number of levels requires recursive querying, for which there are several standard algorithms, but which Access doe not support.  It can be simulated, however, and you'll find an example in BoM.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In the zip archive, in addition to the main BoM.accdb file, which simulates recursive query by progressively writing data to a temporary database created at runtime, there is also a simpler PartsTree.accdb file which generates the bill of materials to a fixed maximum number of nine levels by means of a query which joins multiple instances of the two tables.  It would not be difficult to extend this query, but I think nine levels should cope easily with even quite complex recipes; though be aware that I speak as someone whose cooking expertise largely comprises moving something from the freezer to the microwave!
