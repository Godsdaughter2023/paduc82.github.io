john w. vinson say:

I would suggest doing the totalling on the PellPayments subform's form footer.

 

View the form in design view; right click it and use the menu on the dropdown to enable the Form Header and Footer. Put a textbox in the Footer named, say, TotAmtUSD; set its Control Source to

 

=Sum([AmtUSD])

 

Do the same for a textbox txtChecksRtd. Set the Name property of the Subform Control on the mainform (the box containing the subform, not the form object itself) to subPellPayments.

 

You can then put a textbox on the mainform with a control source

 

=NZ([AwardAmt]) - NZ([subPellPayments].[TotAmtUSD]) + NZ([subPellPayments].[TotChecksRTD])

 

 

john w. vinson say:

Don't confuse data STORAGE with data DISPLAY. They are separate tasks!

 

A basic principle of relational databases is that you should store data once, and once only. If the Pell_Award is stored in the parent table then it should simply NOT EXIST in the Pell Payments table. It's redundant; having the same information stored in two places raises the possibility that the values in the two tables could be different, which would throw all your calculations into confusion. You can DISPLAY the Pell_Award value on the subform if you wish (though if it's right there on the mainform on the same screen, I don't see why you would want to); similarly with the Req#. To display one of these values (without storing it) you can put a textbox on the subform with a control source of either

 

=Parent![Pell_Award])

 

or

 

=DLookUp("[Pell_Award]", "[FISA Table]", "[FISA_ID] = " & [FISA_ID])

 

In the same vein, the Balance should be calculated dynamically, and should not be stored in any field in any table; to calculate a running balance on the subform, you could use an expression such as

 

= [Parent]![Pell_Award] - DSum("[Amt_USD]", "[Pell_Payments]", "[FISA_ID] = " & [FISA_ID] & " AND [PP_ID] <= " & [PP_ID]) + DSum("[Checks_Rtd]", "[Pell_Payments]", "[FISA_ID] = " & [FISA_ID] & " AND [PP_ID] <= " & [PP_ID])

-Andy

julie fka profjulie say: Unfortunately, there's an assumption inherent in ActiveSync that the data on the computer takes priority over the data on the pocket pc when a new partnership is activated. (I don't think this was always the case, but with the most recent versions of ActiveSync that must be used with WM5.0 or WM6.0 devices it is).

With a new partnership, you only have three options:  1) Combine the items on the mobile device with the items on the computer; 2) Replace the items on the device with the items on the computer; 3) Do not synchronize the information at this time. So when you create a new partnership, there is no option to overwrite the data on the computer with the data from the pocket pc. 

There is an option in ActiveSync that handles conflicts with specific data items.  If you change an appointment or a contact on both the computer and the pocket pc, this option tells ActiveSync which item takes precedence, but that option only works on a case by case basis and does not apply when you're setting up a new partnership.

I recently did something like what you are trying to do.  Here's what I did:
  • Manually added data (appointments, contacts, etc.) to the Pocket PC so the Pocket PC incorporated all unique records from my computer.
  • Deleted all the data from each Outlook application (appointments, contacts, tasks, email, and notes) on my computer so it had an empty .pst file.
  • Created an ActiveSync partnership and selected "Combine the items on the mobile device...."
Of course, you could also do the same thing by manually entering all the Pocket PC data to the computer and then deleting the Pocket PC data and syncing.  In my case, most of my data was on my Pocket PC and it was just easier for me to manually add the computer
