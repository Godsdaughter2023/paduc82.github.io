ken sheridan say: Firstly the relationship between volunteers and programmes is many-to-many, not one to many, as each programme might involve many volunteers, so you need a separate Programs table.  The tables will then be like this:

ProgramEnrollment
....VolunteerID (FK)
....ProgramID (FK)
....<more columns?>

Programs
...ProgramID
...ProgramName
...<more columns?>

The ProgramEnrollment table resolves the many-to-many relationship type between VolunteerInfo and Programs into two one-to-many relationship types by VolunteerID referencing the primary key of VolunteerInfo  and ProgramID referencing the primary key of Programs.  This is how a many-to-many relationship type is always modelled in a relational database.  The primary key of ProgramEnrollment is a composite one made up of the VolunteerID and ProgaramID columns.  A separate autonumber key is unnecessary as the table is not referenced by any other table.

Any more columns you add to these tables should, in the case of Programs represent attributes of the programme as a whole, and in the case of ProgramEnrollment represent attributes of the volunteer's enrolment in the programme.

As regards a form this can be set up quite easily in single form view, based on the VonunteerInfo table.  For the control bound to the HouseholdID column  (don't call columns merely ID, it conveys nothing and only caused confusion)  use a combo box set up as follows:

Name:    cboHousehold

ControlSource:    HouseholdID

RowSource:     SELECT HouseHoldID, Address, [Name], Address FROM Employees ORDER BY [Name];

BoundColumn:   1
ColumnCount:   3
ColumnWidths:  0cm;0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first two dimensions are zero to hide the first two columns.

Add an unbound text box to the form with a ControlSource property of:

= cboHouseHold.Column(1)

This will show the household address once you select a household.  Set the text box's Locked property to True (Yes) and its Enabled property to False (No) to make it read-only.

Next create a form based on the ProgramEnrollment  table in continuous forms view.  This should include a combo box bound to the ProgarmID column set up as follows:

Name:    cboProgram

ControlSource:    ProgramID

RowSource:     SELECT ProgramID, Program FROM Programs ORDER BY Program;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If you have included other columns in ProgramEnrollment representing other attributes of a volunteer's enrolment in a programme controls bound to these columns should also be included in the form.

This form can now be embedded in the volunteers form as a subform, linked to the parent form on VolunteerID.  You'll find a simple example of a subform like this in the demo file ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you need to add a new households record for the volunteer, or a new programs record when assigning a volunteer to programmes in the subform, you can do so by tying the household or programme name directly into the relevant combo box in the parent form or subform, and using code in the combo box's NotInList event procedure.  There's a file NotInList.zip in my SkyDrive folder to which I gave you the link above which demonstrates the use of this event procedure. As you'll need to open a households or programmes form to insert values into the other columns than the name of the household or programme, the relevant form in my demo file would be the frmContacts form in which a new city name can be typed into the city combo box.  This opens the frmCities form, passing the city name to it, ready for the region in which the city is located to be selected or entered.

Sub RandomMixAcross()
 'Jim Cone - Portland, Oregon USA - July 2016
  Dim rngToSort As Excel.Range
  Dim vTemp     As Variant
  Dim vNewRng   As Variant
  Dim ColCount  As Long
  Dim Rd        As Long
  Dim C         As Long
  Dim R         As Long
 
  Set rngToSort = Selection.Cells
  vNewRng = rngToSort.Value
  ColCount = rngToSort.Columns.Count
  Randomize
  For R = 1 To rngToSort.Rows.Count
     For C = 1 To rngToSort.Columns.Count
     Rd = VBA.Int(VBA.Rnd * ColCount + 1)
     vTemp = vNewRng(R, C)
     vNewRng(R, C) = vNewRng(R, Rd)
     vNewRng(R, Rd) = vTemp
    Next
  Next
  rngToSort.Value = vNewRng
End Sub

tom van stiphout say:

There is a lot wrong with this code imho but I'm not sure it explains the error. Here are some opportunities for improvement:

 

1. You say "I'm pretty sure that the error occurs...". How about "I'm 100% sure, and it happens exactly on this line"? That is certainly possible and a best practice. You accomplish it by (a) installing the free "MZ Tools" and using it to add line numbers, and (b) by adding a good error handler in every procedure. "On Error Resume Next" is the WORST error handler. An improvement would be:

public sub mySub

on error goto Err_Handler

'code goes here

Exit_Handler:

'possible cleanup goes here

Exit sub

Err_Handler:

msgbox "ErrNo=" & Err.Number & " on line " & Erl & ": " & Err.Description

resume Exit_Handler

end sub

 

2: Use early binding or late binding, not both. If you know all users will have Excel installed, use early binding:
dim objXL as Excel.Application   'Requires a reference to Microsoft Excel [version] Object Library in Tools > References

set objXL = new Excel.Application

'code goes here

set objXL = Nothing

Replace all code that uses late binding (CreateObject).

 

3: Procedure arguments need to be typed and ByVal, unless a special case applies.

So your line:

Sub WorkBook_RO(FileToRead, bskip As Boolean)

becomes:

Public Sub WorkBook_RO(Byval FileToRead as String, ByVal bskip As Boolean)

 

4: Remove redundant code

    strFileName = FileToRead

    Set xlBook1 = XlApp.Workbooks.Open(strFileName)

This should be written as:

    Set xlBook1 = XlApp.Workbooks.Open(FileToRead)

 

Another example:

Public Sub WorkBook_RO(Byval FileToRead as String, ByVal bskip As Boolean)

    If bskip = True Then Exit Sub

It is curious to pass in an argument to decide not to run a procedure. There may be rare cases where that is correct, but mostly one would write in the calling routine:

if not bskip then WorkBook_RO myFileName

 

By the way, extra points if you realized that this line:

  If bskip = True Then Exit Sub

needs to be fixed up like this:

  If bskip = True Then Goto Exit_Handler

 

Finally, make sure you have Option Explicit in every module. It wasn't clear from your code fragment if you had set it.
