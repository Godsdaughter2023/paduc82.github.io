tom van stiphout say:
It only works on bound textboxes, i.e. ones with a ControlSource, not for unbound ones.

ken sheridan say: Rather than examining the ControlType property, set the Tag property of those controls you want to check to a value such as 'CheckMe'.  You can then prescribe exactly which bound controls you wish to check for changes, and examine the Tag property of each control in the form's Controls collection in your code.

BTW do not use the Environ function to get the current user name.  It is unreliable as an environmental variables value can be changed very easily by a user to masquerade as another user.  Instead call the Windows API GetUserName function with a module such as this:

' module basGetUser
Option Compare Database
Option Explicit

Declare Function GetUserName Lib "advapi32.dll" Alias "GetUserNameA" (ByVal _
   lpBuffer As String, nSize As Long) As Long

Public Function GetUser() As String

   Dim strBuffer As String
   Dim lngSize As Long, lngRetVal As Long
   
   lngSize = 199
   strBuffer = String$(200, 0)
   
   lngRetVal = GetUserName(strBuffer, lngSize)
   
   GetUser = Left$(strBuffer, lngSize - 1)

End Function

You can then call the GetUser() function.

daniel g. grau say:

I saved a document in the AutoRecovery 2011 Folder (it was the default folder when I saved) and I cannot find the file. 

That's a reserved folder. Word deletes everything in there immediately upon quitting. If it does contain files when Word is not running, it's because Word did not quit normally or froze. The next time you start it, Word checks there for files that will allow it to recover whatever documents you had open at the time of the crash.

Sometimes you can find in the Trash a folder called "Recovered files" containing bits and pieces. If you see it, move it to the Desktop and check its contents.

I do not have a "library" folder that many posts refer to- and I have used the "go" tab to search it. 

We all have it, but it's a hidden folder (your file would not be missing if the "Recovered files" folder were properly hidden by Word). Go to the Finder, press the Option key and open the "Go" menu.

In this thread you'll find several ways to look for missing files.  I hope one of them works for you!

Figured out the problem. It had nothing to do with Outlook. Also, there was a bit of a discovery process as to what port will be allowed for outgoing mail.

For those of you poor people who also been sucked into susing Verizon FIOS internet with Outlook, here's what is different -

Settings: Outgoing email sever is: outgoing.verizon.net

Check "My outgoing server (SMTP) requires authentication.

Log on using: Username and password that corresponds to the Verizon email account.

Check the 'Remember password'.

Uncheck Require Secure Password Authentication (SPA).

The outgoing port for me turned out to be 587 although the tech told me 465 (and that number is bandied about elsewhere on the net).

ken sheridan say: I've had a look at your file and I see that the subform's LinkChildFields and LinkMasterFields properties are:

DATE;DESCRIPTION;ENTRY_NO;DESCRIPTION_CASH;DR_CR;AMOUNT

However, you have code in the subform's module which changes the value of the DR_CR column when an account is selected in the subform.  Consequently, while a row is inserted into subform's table it no longer matches the row in the parent form on the above columns.

If you remove the DR_CR column from the subform's LinkChildFields and LinkMasterFields properties you should find that the link between the parent form and subform is retained, and the corresponding credit and debit entries will show as intended.

However, when I examined the JOIN between the two tables with the following query:

SELECT tblCHR_DataBase2.*, tblTRS_TYPE_DataBase4.*
FROM tblCHR_DataBase2 INNER JOIN tblTRS_TYPE_DataBase4
ON (tblCHR_DataBase2.DATE = tblTRS_TYPE_DataBase4.DATE)
AND (tblCHR_DataBase2.DESCRIPTION = tblTRS_TYPE_DataBase4.DESCRIPTION)
AND (tblCHR_DataBase2.ENTRY_NO = tblTRS_TYPE_DataBase4.ENTRY_NO)
AND (tblCHR_DataBase2.DESCRIPTION_CASH = tblTRS_TYPE_DataBase4.DESCRIPTION_CASH)
AND (tblCHR_DataBase2.AMOUNT = tblTRS_TYPE_DataBase4.AMOUNT);

I found that the result table returned considerably more rows than the maximum number of rows in the two base tables; roughly by a factor of 5.7:1.  This indicates that the JOIN is not one-to-one as would be expected if each row in one table represents a corresponding credit or debit entry of a single row in the other.

With the methodology you are employing I would suggest that you make the ID column of tblCHR_DataBase2 a straightforward long integer number rather than an autonumber, and make it the primary key of the table.  Then add this column to the subform's LinkChildFields and LinkMasterFields so that its value is automatically inserted when a row is inserted in the subform.  This will ensure that there is a one-to-one relationship type between the two tables.

TheLinkChildFields and LinkMasterFields properties will therefore become:

ID;DATE;DESCRIPTION;ENTRY_NO;DESCRIPTION_CASH;AMOUNT

To update the existing data in the tblCHR_DataBase2 tables in this way you will firstly have to delete the ID column and add a new one of straightforward long integer number data type, but do not make it the primary key.  Then you will need to identify a completely reliable composite candidate key in both tables on which the tables can be joined one-to-one, if this is possible, which might not be the case.  If it can be done, this will allow you to join the tables on the candidate key and update the ID column in tblCHR_DataBase2 with the values in the ID column in tblTRS_TYPE_DataBase4.  The ID column in tblCHR_DataBase2 can then be made its primary key.  If Access does not allow you to make it the key, however, that would indicate that the one-to-one relationship type cannot be established, and the composite candidate key which you used to join the tables is not valid.  In fact, with the data in the tables at present a one-to-one relationship type will not be possible as the tables do not contain equal numbers of rows, so, at the very least, the deletion of some rows from one and possibly both tables would be necessary before a one-to-one relationship type can be established.

ken sheridan say: The LinkChildFields property must be a foreign key in the subform's table, not its primary key.  You therefore need to change the ID column in the referencing table from an autonumber to a long integer number data type and index it non-uniquely (duplicates allowed).  This will then allow multiple rows to be inserted in the subform.  I would recommend that you use a more meaningful name for the keys than the generic name 'ID'.

To delete a parent record and all matching records in a referencing table you should create a relationship between the tables and enforce referential integrity and cascade deletes.  When a record is deleted in the parent form all matching records in the subform's table will then be automatically deleted.

To duplicate a parent record and all matching records in a referencing table is more difficult.  You firstly have to execute an SQL statement in code which inserts the values from all non-key columns in the parent form's current record into a new row in the same table.  You then have execute another SQL statement in which the values from the non-key columns in the referencing table are inserted into new rows in the same table, at the same time inserting the value of the ID primary key of the row inserted into the parent table, into the ID foreign key in the referencing table.  For an experienced Access developer this is not an unduly difficult task, but I do not think it would be one within your current abilities.
