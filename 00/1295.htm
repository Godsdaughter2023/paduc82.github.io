bill mosca say:

The usual practice is to use combo boxes and to use their values as criteria in the target queries. For instance, if you have a form named frmFilter with a combo box named cboInitials (RowSource uses a table of initials) the report query would look something like this:

SELECT stuff

FROM myTable

WHERE Initials = [Forms]![frmFilter]![cboInitials]

As long as the form is open, Access can see the value in the combo box and will use it in the WHERE clause.

scottgem say:

First are you using the Switchboard Manager that is included with Access or are you creating your own Switchboard?

If using the Switchboard Manager, I suggest you have the switchboard open a separate form. On that form will be a combobox control (sometimes referred to as a dorp down).  Otherwise you can just add the combo to your switchboard form.

The combobox with have a RowSource that will list the options to choose from. If you are using the combo as a filter, then you can just reference it in the WHERE clause of your OpenReport method:

DoCmd.OpenReport "reportname",,,"[fieldname]= " & Me.comboname

In the case of a text value use: "[fieldname] = ;" & Me.comboboname & "'"

However, when we try to create a profile with Outlook 2010 or Outlook 2013 from a domain joined computer, the user's Outlook profile still goes to the local Exchange Server and isn't doing an Autodiscover to the Office 365 servers.

There seem to be several different suggestions on how to fix this but so far nothing has worked. We are following the Microsoft document "Perform a cutover migration of email to Office 365". In the article under Complete post migration tasks it talks about setting the Set-ClientAccessServer AutodiscoverInternalConnectionURI to Null. We have done that but are unable to get the domain joined computer to connect to Office 365 and setup Outlook. What are we missing?

Art

art dekneef say:

Got it working. Needed to expand my search terms to get the answer. Needed to remove the local Virtual directory.

First, from elevated Exchange Management Shell.

Type Get-AutodiscoverVirtualDirectory | fl Name, Server, InternalUrl, Identity

It will display 4 lines. You want the information for Identity listed on the right hand side. Copy or write down this information.

Then type Remove-AutodiscoverVirtualDirectory -Identity "Server\autodiscover (Web site)"

The information you want here is what was returned in the first command enclosed in quotes.

We also did this first but it didn't solve the problem. Set-ClientAccessServer name -AutoDiscoverServiceInternalUri $null

Removing the Autodiscover Virtual Directory did the trick.
