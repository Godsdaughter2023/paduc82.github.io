rich007a say:

Sounds like you've recorded a macro, but you haven't looked at the resulting code (otherwise you would have posted it here!).

 

In Word press Atl+F11 to open the Visual Basic Editor.  Try to find your macro there, probably in a module called NewMacros, either in your docuement or in the Normal template, look in the top left corner in the Project Explorer pane of the VBE (post back if you can't find it).

 

In the mean time, try recording your macro by using Replace.  Ctrl+H opens the Replace dialog, and you can replace "@@" with "" (nothing), then click Replace All.  The resulting macro code should look like this:

 

Sub Macro1()
'
' Macro1 Macro
'
'
    Selection.Find.ClearFormatting
    Selection.Find.Replacement.ClearFormatting
    With Selection.Find
        .Text = "@@"
        .Replacement.Text = ""
        .Forward = True
        .Wrap = wdFindContinue
        .Format = False
        .MatchCase = False
        .MatchWholeWord = False
        .MatchWildcards = False
        .MatchSoundsLike = False
        .MatchAllWordForms = False
    End With
    Selection.Find.Execute Replace:=wdReplaceAll
End Sub

 

but the macro recorder doesn't always record efficient code.  The above could be simplified to:

 

Sub Macro1()
    With Selection.Find
        .ClearFormatting
        .Replacement.ClearFormatting
        .Text = "@@"
        .Replacement.Text = ""
        .Forward = True
        .Wrap = wdFindContinue
        .Format = False
        .MatchCase = False
        .MatchWholeWord = False
        .MatchWildcards = False
        .MatchSoundsLike = False
        .MatchAllWordForms = False
        .Execute Replace:=wdReplaceAll
    End With
End Sub
 

Hope that helps.

Incorrect line terminator when pasting from Excel as plain text - Microsoft Community

hamiditized say:

Incorrect line terminator when pasting from Excel as plain text

Pasting a column from Excel as text into other programs uses Carriage Return instead of Line Feed as line terminators.
Apple OS 9 used CR (0x0D) as line terminator.
OS X uses LF (0x0A), same as other Unix-like operating systems. Excel should be using LF, and not CR.

To reproduce this bug:

Select any column of data from Excel 2011.
Right click and select "Copy"
Open a new document in TextEdit, make sure it is plain text (Format->Make Plain Text).
Select Edit->Paste.
Save as normal.
Examine the resulting file in a hex editor and note that lines are terminated with hex 0x0D and not 0x0A.

While OS X is tolerant of this behavior, most other Unix like systems are not. For example, if you copy this resulting file to a CentOS system and try to print out its contents, you will see only a single line printed, as the CR character does not imply a newline in those systems, as such all the lines print on top of each other on the same line.
