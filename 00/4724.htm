ken sheridan say: A simple system generated parameter prompt may be a cheap and cheerful solution, but far better is to reference a control on a dialogue form as the parameter and open the query from the form, though that is itself a poor solution as users should not interface with data in a raw datasheet, and it is better to open a form or report based on the query.  This has many advantages in addition to presenting the user with a more visually coherent interface; the parameter can be referenced multiple times, multiple parameters can be included in one form, validation code can be included so that invalid values are not entered or the parameter is left Null (unless that is a valid option), etc.

Another option is not to reference a control on a form as the parameter in the query at all, but to filter a form or report based on an unrestricted query by means of code in the form's module which uses the WhereCondition argument of the OpenForm or OpenReport method to filter the form or report.

You'll find examples of both approaches, and more, in MultiSelect.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file opens a report for most of the options available, but opening a form is exactly the same.

You can if you wish get the report to open the dialogue form so that you can open the report directly in the normal way.  The following is an example of code in the report's Open event procedure which does this:

    Const FORMNOTOPEN = 2450
    Dim frm As Form
    
    On Error Resume Next
    Set frm = Forms!YourFormName
    If Err = FORMNOTOPEN Then
        DoCmd.OpenForm "YourFormName", OpenArgs:=Me.Name
        Cancel = True
    Else
        If Err <> 0 Then
            ' unknown error
            MsgBox Err.Description, vbExclamation, "Error"
        End If
    End If

If you wish you can close the form in the report's Close event procedure with:

    DoCmd.Close acForm, "YourFormName"

As the Click event procedure for the button on the form use:

    Const REPORTCANCELLED = 2501
    
    On Error Resume Next
    DoCmd.OpenReport Me.OpenArgs, acViewPreview
    Select Case Err.Number
        Case 0
        ' no error so do nothing
        Case REPORTCANCELLED
        ' anticipated error so do nothing
        Case Else
        'unknown error so inform user
        MsgBox Err.Description
    End Select

Because the report passes its name to the form as the OpenArgs property this form can be used with more than one report which takes the same parameter.  Note that the form's code handles the error which would occur if the opening of the report is cancelled.  This is to cater for reports whose opening is cancelled in their NoData event procedure if there is nothing to report.  If you open the report directly in code from elsewhere in your application you'd also need to handle this error as the code in the report's Open event procedure cancels the opening of the report if the dialogue form is not open, so say you want to open the report from a button on a form which provides a choice of available reports you'd use code the same as the above but instead of Me.OpenArgs you'd put the name of the report, in quotes, or more likely a reference to a control on the form in which you select the report.

An alternative method is:

Private Sub Report_Open(Cancel As Integer)

    Const FORMCLOSED = 2450
    
    DoCmd.OpenForm "YourFormName", WindowMode:=acDialog
    
    On Error Resume Next
    Cancel = IsNull(Forms("YourFormName").SomeRequiredControl)
    Select Case Err.Number
        Case 0
        ' no error
        Case FORMCLOSED
        ' cancel report
        Cancel = True
        Case Else
        ' unknown error; close dialogue form
        DoCmd.Close acForm, "YourFormName"
        MsgBox Err.Description, vbExclamation, "Error"
    End Select
    
End Sub

In this case the button on the form simply needs to hide the form with:

    Me.Visible = False

Again the form is closed in the report's Close event procedure.

ken sheridan say: Whether you view the query's result table as a datasheet, form or report is a side issue.  The real point is that, as well as solving your problem, parameters are far better entered via a simple little dialogue form rather than a crude system generated prompt regardless of the context.  It takes only a few minutes to set such a form up.  The type of control used in the form depends on the nature of the value on which you wish to restrict the query.  If it's one of a set of values, e.g. the name of a customer, product or service for instance then a combo box is the obvious choice.  It not only saves time as you can either select from the control's list or progressively go to the first match as you type in each character, but it also avoids typos.

christophe---01 say:

Please note that a VW Golf 5 and 6 does not have an "ignition sense" wire. You could buy a CAN BUS adapter or connect simply the blue wire from the car kit to the red-and-black cigar ligher plug lead. The cigar lighter loses power 5 to 10 minutes after removing the key from the ignition. Power comes back on as soon as you open the car door.
For signal wiring you need a KRAM type cable when you use the ISO variant (e-bay, 30 to 50 euro for a basic version or 80 to 150 euro for a CAN BUS adapter version). For the direct input variant you need a DIY connector (e-bay, 5 euro).
The ISO variant has the disadvantage that all audio from the radio (traffic warnings, route guidance, parking radar signals) is interrupted during call handling. The direct input variant "overlays" these when they occur.
Before ordering any type of cable or connector, you should remove the unit from the dashbouard and have a look at the top of the unit. It has a sticker attached to it which gives the pin-out for your specific model. There can be differences, depending on the exact production date. The "phone mute" wire can be on the main fakra plug (pin 11) or on the "phone" sub-plug (pin 10).
