ronrosenfeld say: To do this automatically, you would need to use workbook event code.  In general, you would add code that has this kind of structure to "This Workbook":

To enter this event-triggered Macro, right click on the sheet tab.
Select "View Code" from the right-click drop-down menu.  When the VB Editor opens, select "This Workbook" from the relevant project and double-click on it.

Then paste the code below into the window that opens.
=====================
Option Explicit
Private Sub Workbook_NewSheet(ByVal Sh As Object)
    Dim s As String
    Dim sOldSheet As String
    Dim sNewSheet As String
 With Worksheets("Sheet2").Range("A1")
    s = .Formula
    sOldSheet = Mid(s, 2, InStr(s, "!") - 1)
    sNewSheet = Sh.Name
    .Formula = Replace(s, sOldSheet, sNewSheet & "!")
End With
End Sub
=========================

The above will work only on formulas which have the format noted in your example.  If you will be replacing the worksheet name in more complex formulas, post some examples here so I can provide a more appropriate replacement routine.

jllatham say:

There is no formula that can do that.  The best you can hope for would be a rather complex macro that would run when a new sheet was added to the workbook.

There are formulas, called 3D formulas, that can deal with new sheets being added in between two existing sheets.  But those add all of the same referenced cells on all of the sheets between the 2 identified and any in between them.  For example:

=SUM(Sheet1:Sheet3!A1:A6)

Would add all of the values in cells A1:A6 on Sheet1 and Sheet3 and any sheets in between those two, even when added later as long as they're placed inbetween Sheet1 and Sheet3.

robert aldwinckle on forums say:
I need to click RESTORE to maximized the windows  to full screen.


The window will already be in the Restore state.   Again, check the window's window menu.   Now that you can see it you can also do that by pressing Alt-Space.   See that Restore  is grey and unusable?   That means that the window is open in its normal mode.   What you are doing is clicking on the button at the top which maximizes the window's size by putting it into its Maximize state.   Again, check the window's window menu.   Restore is now available but Maximize is grey.   There are tooltips associated with that button but IMO it is better to get familiar with what the icons are telling you.   E.g. the one that has Maximize as its tooltip is a single box, indicating that when you click it the window will fill the screen.   Similarly the one that has Restore or Restore Down as its tooltip has two boxes in a cascade, indicating that it will restore to whatever its normal mode size is.

 

Having a window size that is less than maximal is actually useful, e.g. being able to see parts of other windows behind the one you are using.   And you can have both by using Fullscreen mode with its toolbar not set to Auto-hide.   E.g. press F11 and then press Alt-Space,H  to uncheck that option.   Then if you need to see something other than your IE window you would just need to press F11 to toggle it out of Fullscreen mode.


Fullscreen mode is something which will persist, Maximize state will not.   That's why you would have to keep clicking Maximize each time you open your IE window for the first time but only have to do the Fullscreen mode operations once or simply toggle into Fullscreen mode when you have left the window in its normal mode.

 

However, if you really want to have all your IE windows maximally sized, even in normal mode--e.g. so that there is no advantage to using F11 the way that I described, you can do that quite easily by taking advantage of a Taskbar function,  Show windows...  (formerly called Tile...)  when one of those operations is applied to a single window.   So, you would minimize all your windows, e.g. press Win-D.   Then restore the one that you wanted to maximally size.   Then apply a Show windows... command to it.   Presto!   One window, maximally sized in its normal mode.   Or you could also resize it by dragging on its corners until it filled the screen if you preferred to just use your mouse.

 

BTW if you ever do find a maximally sized window awkward to use you can use the Taskbar's Cascade windows command in a related fashion.   E.g. minimize all windows but the one(s) you want to resize and then apply that command.   That resizes the chosen windows to a default cascade size but then you can easily resize from there by dragging on corners or sides.

Payroll db Table Normalization - Microsoft Community

_db say:

Payroll db Table Normalization

I've just taken a Microsoft Level 1 and Level 2 Access training class and am starting to look at an existing Access payroll db in my work group.  I can't find a single thing about it that seems to conform to the rules and practices to which I was introduced.  I've made a test copy of the db and thought I would take a stab at applying some of the things I've learned.  The first thing I'm seeing is that the data is not normalized.  Some tables have no primary key and one table has 4 primary keys, and everywhere I look there are redundancies.  In my imperfect understanding of normalization, one should eliminate repeating data in the same field, so with respect to payroll related data, you have such things as hours worked, payroll codes, shift names, shift hours, date worked, employee, etc., etc.  So how, with these kinds of necessary fields, does one avoid repeating various common values down the column?  All the shifts are 8.00 hours long.  The "num_HoursWorked" field will, with the exception of overtime, have a value of 8.00 repeating over and over.  Since on any given day there are between 30-40 employees working, the "dtm_DateWorked" field values would show that date repeated an equal number of times.  My question is then, do the rules of 1st, 2nd, and 3rd Normal Form not apply in some cases? 

Thanks, and hoping not to look like a fool...

db

 

john w. vinson say:

You're misinterpreting the "repeating data" rule. There's nothing at ALL wrong with the HoursWorked field having many rows containing 8 hours. However I'd recommend NOT using a Date/Time field; store 320 minutes as a Long Integer, not 8:00 as a Date/Time field - the reason is that you can sum integer minutes, but summing Dates will give wierd results that you don't want!

The multiple rows with 8 hours aren't really "repeating data". If 31 employees each worked 8 hours, one worked 6 hours and took the rest of the day as sick time, that is 32 valid, separate, real pieces of information that should be stored in 32 rows in your table. Repeated data would be something like storing the employee's name repeatedly in the Payroll table - it should be stored once, in the Employee table.

The rules apply always - but how you interpret the rules may need some tweaking!
