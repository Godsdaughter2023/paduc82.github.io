Daily CSV file into Access - Help! - Microsoft Community

katiebou say:

Daily CSV file into Access - Help!

I'm fairly new to Access but have experience working with Sybase and Oracle DBMS. This is my first time finding myself being asked to build a new DB in a professional setting and I'm struggling a bit.

We have a daily CSV file that my manager would like to be imported into Access. My current set-up is that I'm loading the file into a working table and then I have a series of queries that copy the data from the working table into several different tables. Where I'm getting stuck is that the rows of data in the daily file may change and I need to be able to update that row with the new data. For instance, one of my tables is a member table with address information. If the member's address changes, I need the record to update with the new address but I'd also like the old address to copy into a historical table.

When I upload a new CSV file into the working table, I run a series of append queries and any records that are brand new will append into the appropriate table but I'm not sure how to handle updates. I'd like to write a query that matches records by the primary key but I'm not quite sure how to go about it. Most of the SQL I've written was for read only and not making changes to data in tables. I'm also not sure how to trigger a historical record. In Oracle, I'd use a trigger statement but from what I've been reading online, there's no triggers in Access. Can someone help me or point me to a good resource to figure this out?

ken sheridan say: With your example of addresses, rather than archiving the old addresses in a separate table, I would recommend that all addresses are stored in a single table which includes a Boolean column name Active or similar, along with a MemberID or similar foreign key column referencing the primary key of a Members table.  In the day-to-day use of the database you would use queries restricted to WHERE Active to return each member's current address.

When inserting new data, unless your text file includes a distinct MemberID column or similar you would need to determine some other combination of columns which are a candidate key.  This is not always easy.  Names are completely unsuitable as they can legitimately be duplicated.  Sometimes a person's date of birth is used in conjunction with their names, but I have regularly told the story here of when I was present at a clinic where two patients arrived within minutes of each other, both female, both with the same names, and both with the same data of birth.  So, it's for you to identify a suitable candidate key.

Having identified a candidate key you can then include its columns in a unique index.  If you then execute an INSERT INTO  statement (what Access calls an 'append' query) to insert a row into Members, any rows in the text file with the same candidate key values will be rejected, so only new members will be inserted.

The same will apply when executing a query to insert the address data into a referencing Addresses table, in which you have again identified a suitable candidate key.  Additionally, however, you will need insert a value of TRUE into the Active column in any new row inserted, and update the value in the Active column to FALSE in the row in which its value was TRUE prior to inserting the new row.  The sequence would be

1.  Identify whether the address being inserted is a new address, which you can do by first joining the table with the new data to the existing table on whatever you have identified as a candidate key.  If the join returns no match, then the address is a new one. If so you'd proceed as follows, if not you'd abort the routine.  You can do the same in code by calling the DLookup function.

2.  Update the Active column in the client's current rows in addresses to FALSE.

3.  Insert the new address as a new row in Addresses, with a value of TRUE being inserted into the Active table.

For any other attributes which might change, you'd do similarly, but in cases where you simply want to update values, and not retain the 'historic' values, it would simply be a case of joining the table with the new data to the operational table on a candidate key and executing an UPDATE statement.  You don't need to identify whether a value has changed, as if it has not, it would be updated to its current value.

The key thing is the identification of candidate keys which are present both in the operational tables, and in the text file being imported.  How easy this is will depend on how well structured the text file is.

As regards triggers, the nearest thing in Access is what are known as 'data macros'.  These are executed automatically on the basis of changes in the data, regardless of how those changes are made, thus avoiding the need to do so programmatically.  I'm not sure that they'd help in this context, however, but you might want to explore their capabilities.
