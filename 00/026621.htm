ken sheridan say: The fact that you have different packaging values per product reinforces the need to store these in a table along the lines Karl describes.  The table can then be joined in the query to get the value for the product in question.  This does raise another question as to whether there are more than one size of box, however, with different capacities per product?  If so the table would need to also include a Box column so that different capacities per product per box could be stored.  Extending Karls' example:

BoxID  Product BarsPerBox
 1         X          12
 2         X          6
 1         Y          15
 2         Y          8
 1         Z          7
 and so on

The table is really modelling a many-to-many relationship between Boxes and Products, with BarsPerBox as an attribute of the relationship type.  In this scenario the table would be joined in the query on both BoxID and Product.

Leaving the question of whether, and is so, how, the capacities per product are stored, you said at the outset that you currently have a query, which presumably returns the current inventory.  It may well be that that query can be modified without too much trouble to create another query which will do what you now need. If you can post the SQL of the query we might be able to suggest the SQL needed for other queries, firstly on the basis of not storing the capacities per product (which can be entered at runtime as a parameter), and secondly on the basis of an amended design in which the capacities are stored n a table.  For the latter we'd need to know whether there is more than one box size which can be used for a product, however, i.e. whether Karl's 2-column table or its 3-column extended version described above would be needed.
Ken Sheridan, Stafford, England
