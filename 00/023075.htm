andrea jones - all about resources say: Well, you need to create your conditional formatting formula so it returns TRUE on the cells that match and FALSE on the ones that don't.  I'm not clear what you're trying to do with the IF, but the error relating to too many arguments seems to have occurred because of all the AND functions entered as the arguments for the IF function.  If you were trying to test whether A1 was 5 and B1 was 10 and C1 was 15 and D1 was 20 you would use =AND(A1=5,B1=10,C1=15,D1=20).

jllatham say:

To go along with what Andrea Jones has been saying, one way to test your formulas that (for me at least) is easier than constantly revising the conditional formatting formula is to pick an empty cell on a row that would be tested and enter your formula there.  Then you can more easily see when it is returning TRUE or FALSE as its result.

You can even split a complex formula like you have into pieces and put the pieces into separate cells on the row to determine which are giving returns you expect and which may be the piece that is fouling up the whole thing.

My initial guess for a formula would have been the same as Andrea Jones' first one.  You can put more than just 2 things to AND test within an AND statement.  When you do, all of them must be true to return TRUE and it only takes one of them to be FALSE to create the FALSE return.

If combined AND statements aren't returning expected results, I might even split them all into separate cells on a row (and fill down as needed to test more) to see the results of each, like this, assuming you have empty cells beginning at column X and you want to start testing at row 10.

In X10:  =INDIRECT"$C" & ROW())="commissioning"

In Y10:  =$E$1=INDEX(ganttTypes,1)

in Z10:  =L$4>=$E6

in AA10: =L$4<($E6+$F6)

in AB10: =L$4>=$H6

in AC10: =L$4<($H6+$I6)

each should return TRUE for the rows that would be conditionally formatted, you can validate by eye.  When you get them set up properly, you can combine them into a great big AND() formula.

albert d. kallal say:

This can often be confusing.

To be “really” fair it is ACE in Access that replaces DAO. Note that your DAO code works fine in Access.

Note that you do NOT USE or Set a reference to DAO anymore in Access.

The updated successor to DAO is now included with Access. We generally call this ACE.

So for Access versions from 2007 onwards you don’t set a reference to DAO but in fact use ACE.

The reference looks like this in Access 2010:

[x] Microsoft Office 14.0 Access database engine Object

So in fact references stating that DAO is being depreciated are “kind of” correct, but they fail to mention that DAO been superseded by ACE.

So ACE is a new version of DAO. It is fair to say that the “older” DAO reference is not used nor required in Access (unless you using older mdb features).

So in Access code you can and most certainly should continue to use

Dim rst    as DAO.Recordset

But if you look in tools references from the VBA editor, you will NOT see nor require a reference to be set to DAO.

You now set your references to “ACE” as per above and this is how you obtain use of the “new” DAO system included with Access.

The reference to ACE gives you both use of the database engine (ACE) and a “DAO” library. So only one reference is required now in Access to use the database engine + DAO code.
