This should be close. You may need to chage the Sheet names (Sheet1, Sheet2) and the Starting cells(A2).

Public Sub CopyStuff()
    Dim wksFrom As Worksheet
    Dim wksTo As Worksheet
    Dim rng As Range
    Dim rngSource As Range
    Dim rngPaste As Range

    Set wksFrom = Sheets("Sheet1")
    Set wksTo = Sheets("Sheet2")
   
    With wksFrom
    Set rngSource = .Range("A2", .Cells(Rows.Count, "A").End(xlUp))
    End With
    Set rngPaste = wksTo.Range("A2")
   
    For Each rng In rngSource
        rngPaste.Value = rng.Value
        Set rngPaste = rngPaste.Offset(1, 0)
        rngPaste.Resize(9, 1).Value = 0
        Set rngPaste = rngPaste.Offset(9, 0)
    Next rng
   
End Sub

ken sheridan say: Correlating the subforms is easy.  You change the Stocks for Selected Ingredient subform's RecordSource property to:

SELECT ContainerID, MaterialName, Description, Stocks.MaterialID
FROM [Stocks Query]
WHERE MaterialID=[Forms]![Edit Operation]![ProtocolIngredients Edit Operation subform].[Form]![MaterialID];

And enter the following in the Current event procedure of the ProtocolIngredients Edit Operation subform:

    Me.Parent.[Stocks for Selected Ingredient].Requery

What you then need to do is execute an INSERT INTO statement in code on the basis of whichever row you select in the Stocks for Selected Ingredient subform which inserts a row into a table which models the relevant relationship type.  My guess  is that the relationship type is modelled by the OperationIngredients table, and you will need to build and execute the SQL statement in code to insert a row into this table.  With your present set-up you will need to do this row by row on the basis of which row is currently selected in the Stocks for Selected Ingredient subform.  This would require a separate button.  A better solution would be to use an unbound list box, with the above query as its RowSource property.  You would then be able to build and execute an INSERT INTO statement in the list box's AfterUpdate event procedure, thus eliminating the need for the separate button.

The OperationIngredients Subform would need to be based on a query whose result set is restricted by a parameter to the ingredients for the current operation.  The subform would be requeried in the AfterUpdate event procedure of the list box, after executing the INSERT INTO statement, or in the Click event procedure of a separate button if you stick with your current subform.

This not the sort of thing you'll be able to do with the aid of a few online videos I'm afraid.  It requires you to have the necessary understanding of the processes, and the knowledge of VBA and SQL to be able to implement the solution.  I've tried to point you in the right direction, but time and space here allow no more than that.

scottgem say:

First, what you are describing is a fairly standard manufacturing process. Just so you know there are lots of companies offering software designed to do this.

Second, Instead of the right hand subform, I would use a List box to select the container rather than a subform.

The thing is, how this works depends on your table structure. So you start with a table of finished products with a child table that lists all the components of a finished product and their quantities and another child table of the operations needed to complete a finished product. You then have a table of manufacturing orders for finished products. This also has a child table that lists the operations which in turn has a child table that lists the components and where they are being taken from. And that means a table of containers with the components that are stored within.

So the main form is bound to the first child table of the mfg orders. You select the MO which will filter a combo to select an operation. When you select an operation the subform linked to the operation lists the components for that operation. When you select a record in the subform. it will modify the Rpwsource of the list box to show stocks of that component. When you select a stock the subform control is populated with the containerID. I don't see a need for the bottom subform as this is handled with the correct table structure.

Not sure what is your issue with the UI. It is actually pretty straightforward. But it does require a knowledge and understanding of data structures to make it work. Finally a continuous form is one that displays a set of controls as multiple rows. You can accomplish a similar effect using datasheets as you show in your screen shot, but Continuous forms allow for a much greater flexibility in formatting.
