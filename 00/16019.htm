scottgem say:
The first thing you need to do is gain a better understanding of relational database design. Proper database design is an art, but its an art that can be learned. Do some research on Normalization.

Relational database tables should be tall and thin, not wide. I would suspect that you have violated the first law of database design which states that data should be defined by a column and row and no other way. That may sound confusing, but I suspect you are doing what's called encoding data using Field names. what this means is you are defining what data the field holds by its field name, rather than by the data in a record. Let me give you an example.

Lets say your app tracks resort hotels. So you need to track information about the hotel, like name, location etc. You also need to track things like Amenities the hotel offers. One may be tempted to include a field for each amenity like Pool, Fitness cneter, Business Center, etc. But that would be defining data by fields names. The proper way is to have a table of Resort Amenities like so:

ResortAmentitiesID (PK)
ResortID (FK)
AmenityID (FK)

And also a table of possible Amenities

AmenityID (PK)
Amenity

As you can see tall and thin

You said your app tracks projects. So you probably added fields to help track the progress of a project. But these attributes of a project should probably go in a separate table like this:

ProjectStatusID (PK)
ProjectID (FK)
StatusID (FK)
Status

One record for each status where the status is identified by a value in the StatusID field and the value of the status in the Status field.

If you give us an idea of what your table look like we can get more specific.

dhookom say: Hi Craig,

15 employees involved in an incident should equate to 15 records being added to a related table. The "employee incident" table with have a field for the Incident primary key and the employee primary key. Other fields would store information directly related to the employee's involvement in the incident.

You would not store the employee's first name in any table other than the employee table. If Emp No. is the primary key in the employee table, this is the only field value that should be stored in any other table in your application.

This is how relational databases are created. Fields are expensive and records are cheap.

Use subforms to enter the related information. Your main form would be bound to the incident table with a subform bound to the "employee incident" table.

Please come back with more questions. A professional application is based on a solid, normalized table structure. If you don't get the tables correct, your application will not be correct.

The problem was with the fact that my PST file was configured to receive mail from multiple POP3 servers and the safe senders list show only those safe senders defined for the default mail account.  As a result when I added a safe sender based to an e-mail from one of my non-default accounts it did not appear in the safe sender list.  I did not notice or understand this behavior until I began rebuilding my PST file.  Further during testing I had deleted all the safe senders from my default account and apparently the subsequent e-mails that I added to the safe sender list where not for the default POP3 account.  This gave the appearance that the display of safe sender list was broken, when it was not.

In order to see the safe sender list defined for any particular POP3 mail account I have to set that account to be the default in File->Account Seetings->E-mail.  I can then remove any safe sender defined for that POP3 acccount.  A little inconvenient, but perfectly workable now that I understand the behavior.

dbo_111 say:

The clean boot didn't help either. 

It was some VBA code left over from the 2010 suite to expand all of the account folders on initial startup of Outlook. We used to have this on all our machines because under Outlook 2010 the folders always started out collapsed. Here's the code snipped:

Private Sub Application_Startup()
 ExpandMyFolders
End Sub
Private Sub ExpandMyFolders()
Dim Ns As Outlook.NameSpace
Dim F, myF As Outlook.MAPIFolder
Dim str As String

Set Ns = Application.GetNamespace("Mapi")
Set myF = Application.ActiveExplorer.CurrentFolder

For Each F In Ns.Folders
    str = F.Name
    Set Application.ActiveExplorer.CurrentFolder = F
    DoEvents
    If F.Folders.Count Then
        Set Application.ActiveExplorer.CurrentFolder = F.Folders(1)
         DoEvents
    End If
Next
Set Application.ActiveExplorer.CurrentFolder = myF
End Sub

It's the DoEvents which is causing it. Can't believe I didn't think of that. The 2013 suite doesn't require the code anyway since folders now start expanded. Never mind at least I found it.

The removal of the code not only resolved the white panel fix, but it also prevents the crash event on a 1st window close. It's definitely very odd; the DoEvents must prevent something critical from completing internally to the app. 

Anyway, I can stop pulling hair out over this one now.
