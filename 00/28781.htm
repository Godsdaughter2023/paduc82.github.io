unravel a formula in Excel - Microsoft Community

stormy girl say:

unravel a formula in Excel

This formula breaks up a suburb and postcode, can someone talk me through it so that I understand it pls? thx

=MID(B2,MIN(IF(ISNUMBER(FIND({1,2,3,4,5,6,7,8,9,0},B2)),FIND({1,2,3,4,5,6,7,8,9,0},B2))),4)

vijay a. verma say:
I can explain but I need to know the value in B2. Could you please let us know what does B2 contain?

vijay a. verma say:
The formula, in short, is locating the position of first numeric character and extract the 4 digits following that.

Let's break it up.

Let's assume that B2 has MR5807

FIND({1,2,3,4,5,6,7,8,9,0},B2) - This looks for the position of 1, 2,.....9, 0 in B2 and populates the position number found

Hence, you get an array of
{#VALUE!,#VALUE!,#VALUE!,#VALUE!,3,#VALUE!,6,4,#VALUE!,5}

Now, ISNUMBER will convert them into TRUE or FALSE depending upon whether a value is number of not.

Hence you get for ISUMBER(FIND({1,2,3,4,5,6,7,8,9,0},B2)) = {FALSE,FALSE,FALSE,FALSE,TRUE,FALSE,TRUE,TRUE,FALSE,TRUE}

Hence, what we have is =MID(B2,MIN(IF({{FALSE,FALSE,FALSE,FALSE,TRUE,FALSE,TRUE,TRUE,FALSE,TRUE}
},FIND({1,2,3,4,5,6,7,8,9,0},B2))),4)

Now, you have an IF condition...When you have IF condition like IF(A1=A2,"Y"), then you get "Y" if A1 and A2 are equal. if A1 and A2 are not equal, you get FALSE as second part in IF condition is ommitted. Now, you have
=IF(ISNUMBER(FIND({1,2,3,4,5,6,7,8,9,0},B2)),FIND({1,2,3,4,5,6,7,8,9,0},B2))
You know what FIND({1,2,3,4,5,6,7,8,9,0},B2) will generate as I already covered this above.

Hence, you have now this -
IF({FALSE,FALSE,FALSE,FALSE,TRUE,FALSE,TRUE,TRUE,FALSE,TRUE},{#VALUE!,#VALUE!,#VALUE!,#VALUE!,3,#VALUE!,6,4,#VALUE!,5})

Hence, your expression becomes
=MID(B2,MIN({FALSE,FALSE,FALSE,FALSE,3,FALSE,6,4,FALSE,5}),4)

Now minimum is 3 here, hence, you have now
=MID(B2,3,4)

As Bob said, you are taking an incorrect approach to this. Access is a database platform that you use for the collection, storage and retrieval of information. Information is stored in tables. XPanel comes in 2 flavors - the original, and "Version 2". I'm not sure what the original version uses as the "carrier" program, as we're insulated from it, however Version 2 runs on Adobe AIR 1.0.

Unfortunately, when we try to run either version of XPanel on a PC running Win 7, 8.1, or 10, the behavior is not quite what one would expect.

For example, I draw a Volume Up button. The desired operation of the button would be that you press the button and release it, the volume would go up 1 "tick". If you press and hold the button, the volume should ramp up - the rate of which would be set in a separate program running in the Crestron processor.

What we find however, is this: When we push the button, (on the 'initial' press,) nothing happens. When you take your finger OFF the button, the volume THEN goes up by one 'tick'.

If you hold your finger on the button (for 1, 2, 5, 10, or even 30 seconds), nothing happens at all - until you release it, whereupon the volume goes up one tick. You can't get any ramping action.

I assume that this is due to the desire to have right click functionality on a touchscreen. If the screen were to respond on the initial press, there would be no way to implement right click, as the program would have gone off and done whatever it was a left click was supposed to do.

After some experimentation, we discovered that if you enable right click in the Pen / Touch settings dialog, set both time sliders to zero, then disable right click, we can get almost acceptable response.

The system still waits for about 2 seconds before it realizes that the button is being held down, but at least it works.

I can't find any way to remove the delay. No delay would be better for our purposes, as we could then set the desired press/hold response time in the processor.

This [somewhat annoying] button press behavior is not always present, however. If it was, you wouldn't be able to swipe anything, or slide switches back and forth in some settings dialogs found in Win 10.

Which tells me that this is programmed behavior built into Windows that is somehow only functional under certain conditions - such as when a program is running.

Apparently, AIR 3.0 allows for instantaneous button responses, and we've verified that it does work. Unfortunately, Crestron isn't planning on upgrading to 3.0 anytime soon, so we still have the problem.

It's a huge amount of work to try to re-create an entire program outside of VT Pro-e, so I guess we will have to live with slow button responses until they get around to going to 3.0.

Unless someone can come up with a way to get the touchscreen to respond the way one would expect it to when right click functionality is turned off.
