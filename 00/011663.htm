albert d. kallal say:

Ok, first your shortcut does not look right.

It should be:

C:\Program Files (x86)\Microsoft Office\Office16\MSACCESS.EXE


The above assumes Access x32, and a non click to run version.

It is possible you are running the x64 bit version of office, and thus you would use:


C:\Program Files\Microsoft Office\Office16\MSACCCESS.EXE


However, if you using an office 365 version, then you are using what is called CLICK TO Run (CTR)

In that case, then:

For x32: (CTR)

C:\Program Files (x86)\Microsoft Office\root\Office16\MSACCESS.EXE

For x64: (CTR)

C:\Program Files\Microsoft Office\root\Office16\MSACCCESS.EXE

So you have 4 possible shortcuts!

So you first need to determine if you running office x32 or office x64. You THEN need to determine if you are running a CTR version of office (most if not all 365 versions of office will be CTR).

So I suspect when you say “not working”, are you saying the program launches and does not work, or the shortcut does not even run Access? (a REALLY important detail here)

Ok, now that we cleared up the above, there is ALSO a SECOND way to test + run + view your application as runtime.

What you do is make a copy of the application, and re-name the extension from accDB to accDR.

However, you REALLY (and I mean, really really) want to first try a debug->compile in the VBA editor and THEN create an accDE. You THEN re-name that accDE to an accDR for testing as runtime. I simply don’t have the pages of space here as to why you want to do this (but do ask and I will follow up as to why this is so important).

So if you not sure if you running office x32 or x64, and ALSO if you not sure if you running a CTR version of office (or a MSI version) or not, then a quick and simple way to test as runtime is to re-name the extension to end in r.

Eg:

MyApp.AccDB  -> MyApp.accDR

MyApp.AccDE  -> MyApp.accDR   (I recommend this choice).

Also, last but not least:

It is ZERO harm and trouble to install the runtime, but in fact as pointed out it DOES NOTHING if you have the full edition of Access. So you really can’t mess this up since if you have the full edition of Access, then the runtime install is IGNORED and DOES NOT really occur. In fact it not possible to install runtime when you already have full edition. (and not to split hairs - the install occurs but it is a "fake" install).

So I can well say that using the “R” rename approach is less effort then setting up a full shortcut, but you eventually DO WANT to get familiar with the correct shortcut, as you will VERY much want to use a shortcut for the final “end users” distribution when using the runtime.

So I ONLY recommend the “rename R” trick on your development machine. Of course the slight downside of this is you have to turn on display of file extensions in windows, but that quite much the FIRST change I make when starting out with a new computer/new windows.

Good luck!
