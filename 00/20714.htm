john w. vinson say:

This is a bit confusing in that it appears that you're mixing Forms with Tables.

 

A Form is just a window - a tool to display and edit data. Forms do not have primary keys or foreign keys or relationships; only Tables do.

 

Forms with Subforms do have Master and Child Link Fields which can be used to automate the updating of foreign keys; i.e. if your mainform is based on a table with a primary key, and you use that primary key as the Master Link Field to a subform with a foreign key as the Child Link Field, Access will (at least try to) assign the value of the current record's autonumber to the foreign key on the child table. If you want the relationship to be optional, allowing orphans, then you cannot use the master/child feature.

 

What are the actual table (not form!) relationships in the Relationships window? Are you using forms and subforms, or are Form A to D independent freestanding forms? Could you post a sample of the code which triggers the runtime error?

 

More basically, what real-life entities are modeled by these very nonstandard relationships?

Because only one of the foreign keys is allowed to be filled in


Well... that's the problem. A basic principle of normalization is that a value in the table should depend on that table's primary key and nothing else. Here, you have two fields in your table which are dependent both on the primary key of the table, and also on what's in the other foreign key field! This is a violation of normal form.

john w. vinson say:

All I can suggest is that you appear to have a good example of "Subclassing" - every HardwareID has zero, one, or many Models, but each Model may be one of a whole lot of classes: Converters, Amplifiers, Microphones, and so on. Each class of models has its own distinct attributes, mostly not shared with other models; it's not clear to me whether all Models share any attributes in common other than their existance.

 

One suggestion would be to have a table of Models with a ModelID primary key; a ModelType field (indicating that this model is a Converter, this next one is a Microphone, and so on); and any other fields that are in common to all models, if any. This table could have a one-to-one relationship to each of your existing Model tables (joining primary key ModelID to primary key ModelID), and related one to many to the Hardware table.

 

It's tough to build a really clean and simple user interface, in my view, because it's not a clean or easy problem! Good luck!
