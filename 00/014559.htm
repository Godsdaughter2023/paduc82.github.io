ken sheridan say: As with any relational database the key to an efficient and robust application ids to get the logical model correct first, i.e. having correctly normalized tables, each with a set of columns which represent the attributes of the entity type which the table is modelling, and which are correctly related.  A relational database, being a model, should reflect the entity types and relationships in the underlying reality.  If the model accurately reflects the reality, it will behave like the reality, if not problems will inevitably arise.

When embarking on the design of a relational database application it's important to take note of what Richard Feynman is saying in the quote below my signature.  What's needed above all is an understanding of how the database relational model works, so that you can apply that understanding to the task in hand.

An important element of relational database design is the elimination of redundancy through the process of normalization.  I shan't go into details of normalization as time or space do not allow, and you'll easily find plenty of resources online.  The key thing to understand is the concept of functional dependency.  In the language of the database relational model this is expressed formally, but in a gross oversimplification you might think of it as meaning that every fact in the database is stated once and once only, e.g. a city determines a state, so in a table of addresses once we are told which city the address is in we know which state it's in.  If the addresses table also told us which state the address is in by having a column for the state that would be wrong as we know the state already from the city by virtue of having a state column in a separate table of cities.

In your case there will be some attributes of a printer which are determined by the key of a row in a table of printer models, e.g. the manufacturer.  It would be wring therefore to include the manufacturer as a column in a table of individual physical printers as this will include a column for the model.  Appropriate attributes represented by columns in this table would be its serial number and its current location, which are functionally determined by the key of this table.

When it comes to your 'master' table of repair jobs you need a foreign column which references the primary key of the  table of individual physical printers, but you do not need columns for the model, serial number, manufacture etc. as these are not determined by the key of the table.  However, you do need a column for its current location for instance, despite the fact that this is already in the table of individual printers.  This is because, while the current location of the printer is recorded in the latter table, the printer might be moved over time, so a column for its location at the time of the repair job is needed in the table of repair jobs.

For data entry therefore, a form should be based on a query which joins the 'master' table to the table of individual printers, and which in turn joins this table to the table of printer models, and this to the table of manufacturers.  Other tables in the query would bring in other relevant data.  The data from the referenced tables can be shown in controls bound to the relevant columns in those tables.  For data such as the current location, however, the control would be bound to the column in the 'master' table and the value from the table of individual printers would be assigned to this control in code in the form's module.

To insert the correct value into the foreign key column which references the relevant row in the table of individual printers this column's control would be a combo box whose bound column is hidden so that it shows a list of serial numbers form which one can be selected.  Once this is done the values form the columns in the referenced tables would be shown in their relevant controls, and it would then be a case of inserting values into the controls bound to the columns in the 'master' table.

john w. vinson say:

We can only tell you how to do this if we know how your data (not your form) is currently structured. We can't see that from here!!

 

If... as I suspect... you're trying to copy the Asset # and Building Location from the Printer table into Asset # and Building Location fields in your Call Log table - your tables are incorrectly designed. Relational database should store each fact ONCE and once only, not copy it redundantly into multiple tables!

 

If you just want to display these fields, you can use a Combo Box control on your form to select the serial number. The combo would be based on a query of the Printer table, including these fields; you can use the ColumnWidths property of the combo to set the width of the fields to 0 so you don't see them in the dropdown. Then you can put textboxes on the form with control sources like

 

=comboboxname.Column(n)

 

where n is the zero based column position - e.g. if the Building Location is the third column in the combo, you would use Column(2).

 

If you are stuck with storing these fields redundantly, please explain - it's a problem that should be fixed but we can give you a "band-aid" to patch it if need be.

The first point is that for this to happen the Asset # and Building location are ALREADY in the database with the Serial #. Therefore, you should NOT be populated the Asset # and Building Location FIELDS in your table. However, as my blog says, you can DISPLAY those values on your form once the the Serial # is selected.

=cboSerial.Column(x)

where cboSerial is the name of the serial number combobox control and x is the column # in the RowSource of the combo (starting with 0) of the value. This is all detailed completely in my blog.
