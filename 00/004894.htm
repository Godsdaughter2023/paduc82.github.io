john - project say:

Adam,

I guess your experience and my experience in working with VBA is quite different. I would guess that a good 40-50% of complex macros I've written involve transfer of data from Project to Excel (90%) and Excel to Project (10%).

A good plan in Project reflects the logical order of tasks to be completed to reach an end result. So in your example whether travel is a single occurrence in a group of tasks or occurs multiple times over the course of plan execution, it should be included where and as many times as appropriate. As far as plans changing, it is rare that a plan in Project will not go through several changes as it unfolds. That's part of project management.

Many users want/need to transfer Excel based information into Project to produce a Gantt style plan. This can be done in a few ways. One is via an import map although that requires very careful setup for fields to transfer successfully. Another is to manually copy Excel data fields to Project but this is better suited for a one-time transfer as it is tedious and can be prone to human error. The third (and my preferred) method of transferring data from Excel to Project is with VBA.

On the subject of tying Project's custom fields into its scheduling engine I though I made it quite clear previously but apparently not so let me re-iterate. To "bind those custom finish fields", there are only two viable options. Either manually copy the custom fields into Project's scheduling fields (e.g. Duration1 into Duration) one by one, or use VBA. So, which would you rather have, a requirement that users do a lot of manual manipulation or provide a custom button on the ribbon to run a macro that does the transfer automatically?

If I were doing this I would transfer data from Excel to Project with a macro that automatically populates the required fields to produce the desired plan in Project. No custom fields and no manual effort other than required for normal management of a project plan.

John

christoph müller say: Hi Jack,

I was a bit busy last week, but now I wanted to let you know that I finally managed to get everything sync'ed. It seems that WMDC has a problem with very large amounts of data during the first sync. The problem were the hundreds or thousands of mails I had flagged (or marked as finished tasks). It seems that WMDC tries to syncronise these in the initial step, too, and then timeouts over the sheer amount of data.

In case s.o. else is interested, this is what I did: I sorted the mails by flags, selected all mails having a red flag or having the check for finished tasks. Then, I remove these marks via the context menu. After that, WMDC sync'ed without problem. If you backup your PST before, you can replace the version without flags with the backup version afterwards. I am not sure whether this works in any case, but in my case it did. I have all flags back and I still can synchronise.
