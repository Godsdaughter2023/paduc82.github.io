     A      B    C      D

countif($B:$B,"no")

countif(B,"no")

countif(b,no)

countif(B:B,no)

$B="no"

B="no"

bill manville say:

First clear your existing rules, if any (Home > Conditional Formatting > Clear Rules)

Then select the area you want to have the conditional formatting and not which row the active cell is in.  Suppose it is in row 2...
Home > Conditional Formatting > New Rule > Use formula to determine which cells to format > Format values where this formula is true: =$B2="No" > Format > set the colour you desire.

The 2 in the formula should be the row number containing the active cell.
Note there is no $ before the 2, which means the row number adjusts for the other rows

Changing "On Error..." at start of error handler - Microsoft Community

paul domaskis say:

Changing "On Error..." at start of error handler

I just debugged some code in which "On Error GoTo errHandler" was not reset to "On Error GoTo 0" after the "Resume Next" in errHanlder.  That is, in the main body of code (not errHandler), there should have been a "on Error GoTo 0" after the area where you wanted error handling by errHandler.  So errHandler got run at all sorts of times when it would have been helpful for VBA to simply quit and give the user/programmer an error message.  The most obvious newbie fix (i.e., good enough for me) is to  ensure that "On Error GoTo errHandler" and "On Error GoTo 0" bookend the piece of code for which you want errors handled by errHandler.

Then it occurred to me that even when an error is encountered within errHandler itself, you don't want program flow to be transferred to the start of errHandler.  So it seemed to me to be a good idea to have "On Error GoTo 0" as the first line of error handlers in general, unless you have a specific alternative handler for errors that occur within error handlers.  However, I never see this being mentioned in my web browsings.  Is this in fact a rational good practice?

I can imagine that if "On Error GoTo errHandler" is automatically disabled upon entry into errHandler, then the above is unnecessary.  But i'm not sure if that's really how things work.  If I set a break point within errHandler, is there a way to ask the VBA environment what error handling is in effect when it breaks within errHandler?

steve rindsberg say:

If On Error GoTo 0 is left as is or commented out, Err.Raise within the error handler throws an error so I think it's reasonable to say that the local errorhandler doesn't handle errors that occur *within* the errorhandler.

On Error GoTo 0 also seems to do an implicit Err.Clear.  If you comment it out and run the code below, you'll get a message box that lists the error description and number.  If you leave it in place, the message box gives you a blank.

On Error GoTo 0 as the last line of the errorhandler before Resume Next disables the error handler in the main code.  From that point on, any errors will throw an error dialog box at the user.

I would think you'd want OnError Resume Next at the beginning of the error handler if you're doing anything there that might throw errors.  You can then test for errors after any suspect lines of code:

If Err.Number <> 0 Then

   ' Ooops.  Fix it.

End if

Or if the error handler's going to do anything more complex (logging, for example), have it call another sub/function that handles its own errors rather than passing them back.

Sub BadErrHandler()

    On Error GoTo BadErrHandler
    
    Err.Raise 404, "Here", "Random error number"
    
    MsgBox "After the error"
    
NormalExit:
    Exit Sub
BadErrHandler:
    On Error GoTo 0
    MsgBox "Err: " & Err.Description & Err.Number
    Err.Raise 666, "In errorhandler", "Devilish error"
    Resume Next
    
End Sub

doug robbins - mvp office apps & services (word) say:

Click on the Windows Start button and then on Control Panel and then on Change keyboards or other input methods under Clock, Language and Region. Then in the Region and Language dialog, go to the Formats tab and then select English (Australia) from the Format pulldown.

The Date and time formats will then change to d/MM/yyyy, dddd, d MMMM yyyy, etc and if you enter 10/9/2010 into Excel it will be taken as being 10 September 2010
