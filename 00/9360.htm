Inserting a line break in a row source - Microsoft Community

jennifer a davis say:

Inserting a line break in a row source

In a report I have a combo box with row source:

SELECT [BEST Active Sites primary chosen by me].[Facility Name] & [BEST Active Sites primary chosen by me].[CCCE's Name] AS Expr1 FROM [BEST Active Sites primary chosen by me];

Row source type is Table/Query.

Is there a way to have the second field, CCCE's name, on a line below the "Facility Name field?  I guess I need to insert a line break.  Thank you!

scottgem say:
Try:
SELECT [BEST Active Sites primary chosen by me].[Facility Name] & Chr(10) & Chr(13) & [BEST Active Sites primary chosen by me].[CCCE's Name] AS Expr1 FROM [BEST Active Sites primary chosen by me];

That should put it on a second line.

However, I see some design issues with your database. First, Using spaces in object names is not recommended, it can come back to haunt you. Second, it would be better to use shorter object names. But the biggest issue is that it looks like you may be storing redundant data. When selecting data from another table, you should be using a primary key value when selecting from another table, not repeating data.

dhookom say: A combo box in a report should be bound to a column in your report's Record Source. Since your Row Source in the combo box returns only a single column, it must be the bound column. Adding a carriage return and line feed would remove the link between the Control Source and the Row Source.

I would not use a combo box in the report since you can add [BEST Active Sites primary chosen by me] to the report's record source.

Scott's solution with the Chr(10) & Chr(13) does split the result into two rows.

ken sheridan say: In a report you should not use a combo box.  Base the report on a query which joins the report's current table to the [BEST Active Sites primary chosen by me] table or query on the [Facility Name].  In the report bind two text boxes in the to the [Facility Name] and [CCCE's Name] columns from [BEST Active Sites primary chosen by me].

This assumes that [Facility Name] is the primary key of [BEST Active Sites primary chosen by me] and is referenced by a foreign key, probably of the same name, in the report's current table.  While there is nothing in the database relational model against using a text column of distinct values as a primary key, it would be more usual in a context like this to use an autonumber column in [BEST Active Sites primary chosen by me] as the primary key, and a corresponding long integer number column in the referencing table as a foreign key.  For data entry into the referencing table in a form you would then use a combo box set up to hide the numeric bound column in the same way as I described in your other current thread.  In a report's query the join would then be on the numeric keys, not the text columns.

Note that if a text column is used as a primary key, cascade updates should be enforced in the relationship between the table and any referencing table.

ken sheridan say: I think you are misunderstanding the means of achieving correlated subforms.  A continuous forms view subform cannot be embedded in the detail section of another continuous forms view form, be it a parent form or a subform.  In this case, however, both the services subform and the goals subform are in continuous forms view, and (this is the important point) both are subforms of the main parent form, not one of the other.  The second is correlated with the first via the hidden text box control in the main parent form.  Consequently the goals continuous forms view subform is requeried as you navigate from row to row in the services continuous forms view subform.
Ken Sheridan, Stafford, England

ken sheridan say: You'll have to include a Boolean (Yes/No) field in the underlying table and bind the check box to this.  That way you can select rows individually.
