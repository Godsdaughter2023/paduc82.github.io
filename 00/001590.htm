ken sheridan say: Karl's reply prompted me to think a little more about your question, and it seems to me that your problem inherently stems from an incorrect design.  As I understand it you have six columns in a samples table, one for each tissue type which might be sampled.  This is what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  A correct design for this would be to have each tissue type sample as a separate row in a related table, so the model would be like along these lines:

Samples
....SampleID (PK)
....PatientID (FK)
....SampleDate

TissueTypes
....TissueTypeID (PK)
....TissueType

SampleDetails
....SampleID (FK)
....TissueTypeID (FK)
....SampleResult

The SampleDetails table is here modelling a many-to-many relationship type between Samples and TissueTypes by resolving it into two one-to-many relationship types, which is how this kind of relationship type is always modelled in a relational database.

To average the results per sample is now a simple task of joining the Samples and SampleDetails tables in a query and using the AVG operator to return the average of the results per sample, e.g.

SELECT PatientID, Samples.SampleID, SampleDate,
AVG(SampleResult) AS AverageResult
FROM Samples INNER JOIN SampleDetails
ON Samples.SampleID = SampleDetails.SampleID
GROUP BY PatientID, Samples.SampleID, SampleDate;

As there will be no rows in SampleDetails where the SampleResult column is Null the average will be over the number of tissue types examined per sample.

To see a simple example of this kind of relationship and how it can be represented via a form and subform you might like to take a look at ParentActivities.zip in my public databases folder at:

ken sheridan say: Firstly, having multiple tables for each tissue section per tissue type is not the way data is structured in a relational database as it constitutes what I described earlier, but to an even greater degree than encoding data as column headings, it being encoded as table names.  It's as well to consider the basic principles of the database relational model before turning our minds to the specifics in your case.

Tables model entity types. The word 'types' is important here as it is the physical or abstract class of entity which a table represents not an instance of that class (an entity).   It is each row in the table which represents a specific entity. In a table Patients for instance, which models the patients entity type each patient is an instance of that entity type and is represented by a single row in the table.

Each column in a table represents an attribute type of the entity modelled by the table, so in the case of the Patients table there will firstly be a column, designated as the primary key, with distinct values which uniquely identifies the patient, PatientID say.  Often this will be an autonumber which automatically assigns an arbitrary distinct number of long integer data type when a row is inserted.  This number will not usually be exposed to users of the database as it has no semantic significance.  Other attribute types.  Sometimes another column whose values are distinct to patient will be included in the table, in the UK the patient's NHS number for instance.  This could be used as the 'natural' primary key, though by and large an arbitrary 'surrogate' key is preferable, in which case the NHS number column, as a 'candidate key' would be uniquely indexed to prevent invalid duplication of the value.

Other non-key attributes such as FirstName, LastName, DoB etc would be represented by other columns in the table.  We now come up against another important principle, that of 'functional dependency', which is the basis of 'normalization'.  Normalization is the process of eliminating redundancy from a table by the decomposition of the table into a set of related tables.  A column in a table is functionally determined by another column if, by knowing the value of the one column the value of the other column is known.  In a correctly normalized table each non-key column must be functionally determined solely by the whole of the primary key of the table (a key might be made up of two or more columns, hence the 'whole').  Say we have a CityID and StateID column in the patients table as part of the patient's address, once we know the city we know the state, e.g. if the city is San Francisco the state is California, so CityID determines StateID.  But CityID is not the primary key, nor even a part of the key of the Patients table, so StateID is not functionally determined solely by the whole of the primary key of the table.  It is said to be 'transitively functionally dependant' on the key.  This allows inconsistent data in that it would be perfectly possible for a row to erroneously put San Francisco in Colorado!  The StateID is not needed at all in the table, it is redundant, as a Cities table will tell us once and once only that San Francisco is in California, so there can never be any inconsistencies in the database as to which state a city is located in.

I know that this basic patient data is not really what interests you in the context of this thread.  It simply provides a simple example of the basic principles which underlie the relational model, and which it is important to understand when considering how to model the data with which you are primarily concerned.

So, the starting point is to determine what are the entity types, what are the relationship types between them, and what are the attribute types of the entity types.

The first thing I should say is that you are in a far better position than we are to determine this as you have an intimate knowledge of the real world situation which the database will model.  We are reliant on your description, so be sceptical of everything I say below as it stems from a very superficial understanding of the situation.  As I see it the entity types are:

1.  Samples.  This represents the sample as a whole.  The attribute types of this entity type will be this which apply to the sample as a whole (functionally dependant on the key), so in addition to the sample number (a candidate key, so can be the primary key) might include the date the sample was taken, by whom etc.  Only you can say for certain what is appropriate and therefore what non-key columns the table will have.  One column would presumably be the species from which the sample is taken, which should be a foreign key referencing the primary key of the second entity type below.

2.  Species.  This is the set of different species from which samples can be taken, human canine, pig and whatever other species might now or in the future be used.  Each would be represented by one row in the table.  So if at some time samples were to be used from a species other than the three you mention it is simple a question of inserting a row in to the Species table.  You may never envisage this happening, but it does illustrate why it is so important that data be stored only as values at column positions in tables, as, if it is encoded as column headings or table names it then becomes necessary to amend the design of the database by adding further tables and/or columns.  You might ask why have a separate table at all rather than simply entering the species value into a column in the Samples table?  The answer is that it allows you to create a relationship between samples and species, enforcing referential integrity.  This then means that only valid species can be entered in Samples, avoiding errors resulting in inconsistent data.

3.  TissueLocations.  By this I mean the stomach body, or colon.  You can probably think of a better name for this entity type, however.  So this would have just two rows, but as with species more rows can be added should it ever happen that samples are taken from other parts of the body.

4. TissueTypes.  A table with one row for each of mucosa, submucosa, muscularis mucosa, and muscle.

5.  TissueTypeLocations.  This table models the many-to-many relationship type between TissueLocations and TissueTypes, so would have foreign key columns TissueLocationID and TissueTypeID.  In your case samples of each of the four tissue types can be taken from each of the two locations, so there would be eight rows, one for stomach mucosa, one for stomach submucosa etc.  However, it allows for a situation where some tissue types would be available from only specific locations and thus eliminates the possibility of data being entered erroneously for a tissue type from a location from which the tissue type in question cannot be taken.  This situation does not arise at present of course, but the model should allow for it.

6.  Measurements.  We now get to the table where the real work is done.  All of the above tables will be set up once as the general framework of the sampling regime.  The Measurements table is where data will be entered on a regular basis as the slides are received.  In relational database terms this table is really modelling a relationship type between Samples and TissueTypeLocations.  So it will have a foreign key SampleID column, a foreign key TissueTypeLocationID column, a MeasurementNumber column ( with values 1 to 3 to distinguish between the three measurements per slide),  and the Measurement column itself.  The primary key of this table would be a composite one of the SampleID, TissueTypeLocationID and MeasurementNumber columns.

When it comes to aggregating the measurement data as averages, with this model you can now do this on any basis you wish.  In a query it is simple a question of how you group the query before returning the AVG(Measurement).  It can be by tissue type as you wish, which can be per species The query would include the relevant tables, joined on the keys, group by species and then by tissue type location and then by tissue type and average the measurement column.  It can also be done in a report based on a query which joins the table, but does not group the data, in this case grouping the report in the same way and returning the averages in the tissue type group footer. Because the database is a model of the real world entity types and relationships you van extract information from it in pretty much any way you want to.  That's the great advantage of a well designed database; it does not impose preconceived ideas on what you want from the data; it stores the data in a logical way so that it gives you complete flexibility over how you turn data into information.
