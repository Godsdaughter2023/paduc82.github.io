Q: Your means is even user not change to a new password, there is no effect to Office applications (word excel..), is it right?
A: Yes, there are not related each other in terms of activation. However, if the user don't change to a new password, he will not be able to access SharePoint Online resource which is associated with the user account from within Office application.

Q: When password expiration date reached, users are not change to new one, is it OK?
A: It's OK if your users don't need to access Office 365 and SharePoint Online resource. However, as a best practice, I would suggest that you set users' passwords to never expire in Office 365 admin center > service settings > passwords or via PowerShell. You may refer to Charles's reply in the thread below:
https://community.office365.com/en-us/f/156/t/199722

Q: About the information before reach, it will appear on each PC of user or where?
A: Do you refer to notification before password expiration? Yes, it will appear on each user's PC. Users can see a message whenever they sign in(Word, Excel,etc..), starting at the number of days before password expiration your specified.

Vincent

Suppliers - Microsoft Community

shelly moothart say:

Suppliers

Our inventory lists the most used supplier of the product.  I need to change the supplier to a different one but keep the product the same and have it show up on the list of inventory under that supplier.

Example:  We normally buy from ABC Company but this time we bought from DEF Company.  When I go to add a purchase order and update the inventory it only shows the product under ABC Company not DEF Company.

Help!  I am a first time user.

thedbguy say:

Hi Shelly. Can you post your table structure? Do you have a table of suppliers to include both companies?

ken sheridan say: As each product can be purchased from more than one supplier, and presumably each supplier can supply more than one product, three is a many-to-many relationship type between Suppliers and Products which should be modelled by a table which resolves the relationship type into two one-to-many relationship types.  In broad outline:

ProductSuppliers
….ProductID  (FK)
….SupplierID  (FK)

The primary key is a composite one of the two foreign keys.  The table is likely to have other non-key columns which represent other attributes of the relationship type between the supplier and the product, e.g. the current unit price charged by the supplier for the product.  You could also have a Boolean (Yes/No) column in this table to indicate which supplier per product is the most used supplier.

In a purchase orders form, I would normally expect the SupplierID combo box to be correlated with a ProductID combo box so that, having selected a product, the list of suppliers would be requeried to show only those who supply the product.

scottgem say:

There are several issues all related to the design of your tables.

Issue 1: Do you need to track which suppliers supply which products? If you do, then you need the structure that Ken describes. A suppliers table, a products table and the junction table that models the many to many relationship.

Issue 2: Do you need to just track the last supplier you bought a product from? In that's case, you would just have SupplierID as a Foreign Key in your Products table. But then you would need to manually update the Products table when you make a change,

Issue 3: Do you only need to show the supplier that was most used for a specific product. In that case, you need to create a query that calculates the supplier with the most purchases. You would then match the product against that query to pull the correct supplier.

All three scenarios can be valid depending on what you need.
