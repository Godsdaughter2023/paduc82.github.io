ken sheridan say: As a general rule, a form can be based on a query on multiple tables, provided that data is only entered /edited in the referencing table, not in referencing tables in the relationships.  Controls can be bound to the latter, but these should be made read-only by setting their Enabled property to False (No) and their Locked property to True (Yes).  Otherwise the query is likely to return a non-updatable result table, which I suspect is what's happening in your case.

For entering/editing data over a set of related tables the usual interface would be a form with one or more subforms embedded in it.  The parent form would be based on the referenced table, and the subforms on the referencing tables.  You might like to take a look at DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the section on 'entering data via a form/subforms'  has a contacts form which provides a simple illustration of this.  For examples of entering data into a more complex model via forms/subforms see StudentLog.zip in the same OneDrive folder.

ken sheridan say:

this is the SQL View that came up

SELECT EastTickets.*, Trucks.Description, CropsAndFarms.Variety, CropsAndFarms.FarmNames
FROM CropsAndFarms INNER JOIN (Trucks INNER JOIN EastTickets ON Trucks.TruckID = EastTickets.LicensePlate) ON CropsAndFarms.GrowerBlockID = EastTickets.GrowerBlockID;

I can see no reason why that query should not be updatable, but for a form's RecordSource you don't actually need to include the Trucks or CropsAndFarms tables in the query.  You can show the Trucks.Description, CropsAndFarms.Variety,  and CropsAndFarms.FarmNames values in a form based on the EastTickets table alone.

To show the Trucks.Description value in the form use a combo box set up as follows:

Name:    cboTruckDescription

ControlSource:  LicensePlate

RowSource:     SELECT TruckID, Description FROM Trucks ORDER BY Description;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

To show the Trucks.Description value in the form use a combo box set up as follows:

Name:    cboCropVariety

ControlSource:  GrowerBlockID

RowSource:     SELECT GrowerBlockID, FarmNames, Variety FROM CropsAndFarms ORDER BY Variety;

BoundColumn:   1
ColumnCount:    3
ColumnWidths:  0cm;0cm;8cm

To show the FarmNames values in the form use a text box with a ControlSource property of:

=cboCropVariety.Column(1)

The Column property is zero-based, so Column(1) references the second column, FarmNames.

If you don't want the user to be able to change the description or variety values set the Locked property of each combo box to True (Yes) and the Enabled property to False (No).  The farm names text box, as a computed control will not be updatable.

It does successfully connect to the Internet. It also created a homegroup for itself and fails to recognize any of my other local computers or the existing homegroup. mike donehoo say:

I have learned that the way the VM was set up was the problem. Oracle VM VirtualBox shows the virtual LAN adapter as "Attached to: NAT" by default. Once I changed this selection to "Bridged Adapter," everything started working properly.

To access this setting, select the Settings for the Win10 virtual machine, then select Network in the left column, and set "Attached to:" to Bridged Adapter. This solved the problem for me.
