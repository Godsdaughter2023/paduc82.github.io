1. Don't create 3 tables for 1 type of data. Rather you have 1 table with a QuestionLevel (inventory/ invoice/ factory).

2. Don't use the word "query" if you mean "question": it has very specific meaning in Access.

3. Build for flexibility: what is 10 questions now, may be 12 tomorrow. The 3 levels today may be 4 tomorrow (and when that happens, you do NOT add another table - see point 1).

You need at least a Questions table and an Answers table. Audits can be done multiple times, each time adding rows to the Answers table.

Do all questions have numeric answers? I would not count on it. See point 3. So answers may have to be stored in a text(255) field just to cover all bases.

I would really need more info to design the tables for you, but this may be a start:

tblQuestions

QuestionID autonumber pk

QuestionNumber integer required uniqueindex   (is this field really needed?)

QuestionText text(255) required uniqueindex

QuestionLevel text(10) required duplicateindex   (could spin this off in its own QuestionLevels table)

tblAnswers

AnswerID autonumber pk

Answer text(255)   (should this be required or are blank answers ok?)

AnswerDate datetime required default=Now duplicateindex

AnsweredByID long int required FK   (links to Users table)

tblAudits

AuditID autonumber pk

AuditGeneralDate datetime required duplicateindex   (this is the "July audit" which took 3 days to do)

As you can see some of this is speculation and requires more analysis.

Once the tables are created put them on the Relations design surface and draw the lines between the corresponding fields. In the dialog that pops up check the box to enforce referential integrity. This takes care of the R in RDBMS.

Good luck with your project.

apostrophe (') or single quote (`) in record - Microsoft Community

nigel parsons say:

apostrophe (') or single quote (`) in record data ruins my SQL statement.

The town of AIRE-SUR-L'ADOUR is screwing up my SQL statement.  What is the rightway to deal with fields or strings containing these characters?
hansv mvp say:

Change the line

        strCity = StrConv(rs!CityTownSuburb, vbUpperCase)

to

        strCity = Replace(StrConv(rs!CityTownSuburb, vbUpperCase), Chr(39), Chr(39) & Chr(39))

hansv mvp say:

Chr(39) is the single quote / apostrophe character.

If we look at

"WHERE tblAirports.CityTownSuburb = '" & strCity & "' "

and set

strCity = "Riscle"

the WHERE-part becomes

"WHERE tblAirports.CityTownSuburb = 'Riscle' "

The first ' tells SQL that a literal string begins there, and the second ' that the literal string ends. But if

strCity = "Aire sur l'Adour"

the WHERE-part becomes

"WHERE tblAirports.CityTownSuburb = 'Aire sur l'Adour' "

the second ' comes before after l, so SQL thinks the literal string is "Aire sur l" and it then becomes confused because it doesn't know what to do with Adour.

The workaround is to double the apostrophe within the literal string. By replacing Chr(39) with Chr(39) & Chr(39), we get

strCity = "Aire sur l''Adour"

so that the WHERE-part becomes

"WHERE tblAirports.CityTownSuburb = 'Aire sur l''Adour' "

SQL has been programmed to interpret a double apostrophe as a literal apostrophe within the string.
