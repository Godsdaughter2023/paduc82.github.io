john - project say:

LadyWhitmofGP,

Well, from your description of the end result, it sounds like you do have the external link established. Externally linked tasks will appear as "ghost" tasks in the other file and that is what you describe in your last bullet. I'm not sure what you were expecting to see.

However, I STRONGLY suggest you do NOT set up a linked structure particularly over a network, secure or not. In Project linked structures are very fragile and subject to corruption. You must never rename, move, overwrite, or "save off" to another location any of the files in the structure. Further, all files should reside in a single directory on a local drive, not on a network. Oh it may work for a while but I can practically guarantee that in time, and that may be a few hours or several weeks, the structure will become corrupt.

I have never worked with files on SharePoint but I have heard of others who use it successfully even with linked structures. Nonetheless, the same "rules" above apply.

For a large corporation I would suggest you look into Project Server. It is a much better choice for a large enterprise environment. If for some reason, Project Server is not possible, then your next best option is to create a static master whenever you need to coordinate tasks between various Project files (stored anywhere). A static master is created by using Project/Insert group/Subproject and unchecking the Link to Project option in the lower right corner. That will effectively copy all the selected files into a new independent project file. You would then need to link the tasks in that file and do whatever reporting you need. Since a static master is a snapshot in time of all the files that are inserted, you would need to build a new static master each time but this could be automated by recording a macro of you going through the process the first time.

John

a. user say:

Since you haven't received any response, I'll try to respond to the bits that I can. Please let us know if you have already moved on!

For the Part Number,
 a. Word is somewhat schizophrenic when it comes to "chapter-page" numbering. To avoid unhelpful presuppositions, let's refer to this as "segment-page" numbering.
 b. Some parts of Word assume that each new "segment" begins with a numbered heading style. e.g. { PAGE } number display works like this, on a per-section basis. You say (for example) that each segment begins with a numbered Heading 1 style, and you specify in each section of your document that page numbers should be composed of the heading number, a character and a page number, e.g. 3-1
 c. Other parts of Word assume that each new "segment" begins with a { SEQ } field, e.g. { SEQ part }, and that the complete page number consists of the SEQ number, a separator, and the page number without any Heading numbering.
 d. The INDEX and TOA fields rely on (c), not (b). (Whereas AFAICR, TOC relies on (b), not (c),)

So you may need to coordinate your (b) and (c) style numbering in some way. For example, if you are using Heading 1 to do "chapter-page" numbering for your TOC, in order to replicate that numbering scheme in either INDEX or TOA, you will probably need to put a { SEQ } field in, or near each Heading 1 paragraph. You could for example insert { SEQ segment \h } (the \h hides the result). Then in your INDEX or TOA field, you specify \s segment \d "-" or some such.

(FWIW I have never understood why these differences of approach exist and why we can't choose between them, but accept that perhaps I just don't "get it")

It's harder for me to comment on the TA/XE choice because I have rarely needed to use either. Further, it is not completely clear to me to what extent TA/TOA has been superceded by the newer CITATION/BIBLIOGRAPHY features. As far as XE/INDEX is concerned, the main advantage seems to be that INDEX creates a two-level structure (main items, and subitems), and it isn't clear to me how that works with TA/TOA. But TA/TOA also assumes that each TA and each TOA is for a single category, which is easy if you decide to have just one category, but involves planning/design choices if you want more than one. Also, I notice that the example you give shows an XE field that contains more information than is contained in the visible reference (a subitem, 1-2 and/or 01-02). So in other words, another XE can have the same main reference, but with a difference subitem reference, and the INDEX field should group the references by the main reference. TA/TOA does not appear to me to have that option.

So for converting XE fields to TA, I think you would first have to consider which parts of the XE field should be in the TA - the whole reference, including (referrring to my example above), the 1-2) or or just the "main reference". But once the design decisions have been made, I think either converting XEs to TAs, or adding TAs to existing XEs, or even creating a newer structure that avoids unnecessary duplication, is a matter of producing some fairly straightforward VBA.

I also hope that someone who understands this stuff much better than I do will jump in!
