joeu2004 say:

I am trying to format cells in excel so that they display elapsed time and NOT time of day.  [....] for example, the 400 m dash when some times are below one minute and are stored as a number, such as 58.97, and other times are above a minute and are stored as a time of day, for example 1:01.15.  This is actually stored as 12:01:01.15 AM when viewed in the formula bar.  I know it's possible to convert these times their fraction of a minute, for example 1:01.15 becomes 1.019 minutes and 58.97 becomes 0.983 minutes.

There is no difference between how elapsed time and time of day are stored.  And there is no difference between those and how any other number is stored.

The differences that you see are merely due to formatting and human interpretation.  The Formula Bar always formats any cell with a Time or Date format as time of day.  We have no control over that.

But you might interpret the time value as elapsed time.  And you can select a Time or Custom format that dispays the time value in a form that is consistent with your interpretation of the number.

As for data entry....

In general, elapsed times can always be entered in the form h:m:s.fff, where "fff" is the fractional seconds.   To that end, 58.97 seconds can entered as 0:0:58.97.  Alternatively, we can enter it as 0:58.97.  But "fff" is zero, we must enter 0:58.0 (with a fraction second of zero) or 0:0:58, because Excel interprets 0:58 as 0 hours 58 minutes.

Moreover, all times can be entered as m:s.fff, even if minutes exceeds 59.  But remember to enter m:s.0 if "fff" is zero.  For example, Excel interprets 64:32.98 as 64 minutes 32.98 seconds; it is the same 1:04:32.98. That might apply to 10k runs and longer.  But if you are talking about dashes (short runs), you probably don't care.

Note that if we enter m:s.fff into a cell formatted as General, Excel changes the format automagically.  If we do not like Excel choice, we can always format the cell before or after data entry.  The point is:  the form of the data entry does not have be the same as form of the cell format.

[EDIT] And when you interpret a cell value as elapsed time, I suggest that you use the Custom format [m]:ss.000 or [h]:m:ss.000 (omit any part of the end that you don't want).  The [m] and [h] specifiers display minutes greater than 59 and hours greater than 23.  The latter is particularly important because otherwise, Excel interprets 24 hours or more (stored as 1.xxx or more) as a date.  For example, 25:43:21 might otherwise display as 1/1/1900 1:43:21 AM.  (And that is how it will appear in the Formula Bar, regardless the cell format.  Remember:  we have no control over the Formula Bar format.)

Some clarifications....

Excel stores time (be it time of day or elapsed time) as a fraction of a day.

So 1:01.15 is stored as about 0.000707754629629629, not 12:01:01.15 AM [sic].  Again, the latter is simply how the Formula Bar chooses to format the cell if we format the cell as Time.  If we change the cell format to General, the Formula Bar

And if we enter 58.97 (seconds), it is stored as 58.97, not as about 0.982833333333333 (minutes) and not as the time value 0.000682523148148148 (approximately), which we might format as 0:0:58.970.

Finally, if we enter 1:01.15, the Formula Bar displays 12:01:01 AM, not 12:01:01.15 AM.  At least, that is true for Windows Excel.  Admittedly, I am not familiar with Mac Excel.  This is one of the problems for (Windows?) users:  we cannot edit fractional seconds because they do not appear in the Formula Bar.

andreas killer say:
They are not the same right, their are differences between them??
Just a few, if you are familiar with VBA, you have 95% of the VBscript ready.

The first step is to record a macro in Excel to get the basics of what you want to do. This macro could be look like this:

Sub Makro1()
  Workbooks.Open Filename:="Z:\Test.xls"
  ActiveWorkbook.SaveAs Filename:="Z:\test shared.xls", AccessMode:=xlShared
End Sub

The next step during development is to do the same from another VBA application, e.g. Word:

Sub Makro2()
  Const xlShared = 2
  Dim xlApp As Object, xlBook As Object
 
  'Access to Excel
  On Error Resume Next
  Set xlApp = GetObject(, "Excel.Application")
  If xlApp Is Nothing Then Set xlApp = CreateObject("Excel.Application")
  If xlApp Is Nothing Then
    MsgBox "Excel is not accessible."
    Exit Sub
  End If
  On Error GoTo 0
  'Open the file
  Set xlBook = xlApp.Workbooks.Open("Z:\test.xls")
  'Save it as shared file
  xlBook.SaveAs "Z:\test shared.xls", AccessMode:=xlShared
  'Close it
  xlBook.Close
End Sub

As you see it is very similar, but Word did not know anything from Excel, so we must declare any constant and use a pointer to Excel (xlApp) to access the objects.

The next step is the script, we must/can not use DIM and named arguments are not possible.

  Const xlShared = 2
  'Access to Excel
  On Error Resume Next
  Set xlApp = GetObject(, "Excel.Application")
  If xlApp Is Nothing Then Set xlApp = CreateObject("Excel.Application")
  If xlApp Is Nothing Then
    Wscript.Echo "Excel is not accessible."
    Wscript.Quit
  End If
  On Error GoTo 0
  'Open the file
  Set xlBook = xlApp.Workbooks.Open("Z:\test.xls")
  'Save it as shared file
  xlBook.SaveAs "Z:\test shared.xls", , , , , , xlShared
  'Close it
  xlBook.Close

The rest is self-explanatory, I hope.

You can use Word's Catalogue/Directory Mailmerge facility for this (the terminology depends on the Word version). To see how to do so with any mailmerge data source supported by Word, check out my Microsoft Word Catalogue/Directory Mailmerge Tutorial at:

The tutorial covers everything from list creation to the insertion & calculation of values in multi-record tables in letters. Do read the tutorial before trying to use the mailmerge document included with it.
 
Depending on what you're trying to achieve, the field coding for this can be complex. However, since the tutorial document includes working field codes for all of its examples, most of the hard work has already been done for you - you should be able to do little more than copy/paste the relevant field codes into your own mailmerge main document, substitute/insert  your own field names and adjust the formatting to get the results you desire. For some worked examples, see the attachments to the posts at:
