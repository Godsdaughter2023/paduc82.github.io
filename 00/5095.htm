Set a reference to outlook in your VBA project, and run code like this to make sure that Outlook is running prior to generating the email:

 

Sub OutlookStuff()
    Dim olApp As Outlook.Application
    Dim myNameSpace As Outlook.Namespace
    Dim myFolder As Outlook.MAPIFolder
    On Error Resume Next
    Set olApp = GetObject(, "Outlook.Application")
    If olApp Is Nothing Then
        If MsgBox("Outlook is not running. Start it?", vbYesNo) = vbNo Then Exit Sub
        Set olApp = CreateObject("Outlook.Application")       
        Set myNameSpace = olApp.GetNameSpace("MAPI")
        Set myFolder = _
        myNameSpace.GetDefaultFolder(olFolderInbox)
        myFolder.Display
    Else
        MsgBox "Outlook is runnning so all is good...."
    End If

    'Rest of code, using olApp object.....
End Sub

Assigning an AutoNumber data type to a field that already has data in - Microsoft Community

kylefaas say:

Assigning an AutoNumber data type to a field that already has data in it

I have normalized a single table database (in Access 2007) into 3NF with multiple tables, each with a primary/foreign key relationship to a field which, in the original database, was an AutoNumber data type.  The people using this database would like to keep the AutoNumber data type, as it facilitates adding/tracking new records.  What I'm looking for is a macro to open a form and insert the next available primary key into this field in the new record.  Looking around, it seems I might be able to do this in Visual Basic, but I've never used this language before.  
ken sheridan say: You have two options:

1.  Keep the primary key column of the referenced (one-side) table as a straightforward long integer number and compute the next number in the BeforeInsert event procedure of a form base on the table with:

Me.MyID = Nz(DMax("MyID", "MyTable"),0)+1

In a multi-user environment conflicts could arise if two or more users are simultaneously inserting a new record via the form.   Roger Carlson has a simple solution to this at:

http://www.rogersaccesslibrary.com/forum/topic395.html

2.  Change the primary key to an autonumber, and retain the current values.  Succeeding numbers will then be inserted automatically.  You cannot change the column directly to an autonumber, however.  What you have to do is append the rows from the current table into a new one, where the MyID (or whatever) primary key column is an autonumber, so:

2.1. Rename the existing table to, e.g. MyTable_Old

2.2. Create a new empty table of exactly the same design as the original MyTable, but with the MyID primary key column now an autonumber, and save it as MyTable

2.3. Execute the following 'append' query:

INSERT INTO MyTable
SELECT * FROM MyTable_Old;

2.4.  Make sure MyTable has exactly the same values in its rows as MyTable_Old, and if so delete MyTable_Old (you'll need to delete the relationships of which it's a part first).

2.5.  Recreate the relationships between the new MyTable and the referencing tables.

My recommendation would be to go with the second option.  It's easy to implement and succeeding numbers will then be inserted automatically with no need for any code.  The first method is only appropriate if the numbers must be sequential, but if they are merely unique identifiers of arbitrary value an autonumber makes more sense.

The foreign keys in the referencing (many-side) tables must not be autonumbers of course, but should be of straightforward long integer number data type.   These will be given values behind the scenes via the interface, e.g. if using a form and subform linked on the keys the foreign key value corresponding to the primary key of the form's current record will be automatically inserted.  Or if you are selecting values form the referenced table in a combo box in which the first numeric column (e.g. CityID when selecting a city by name, the numeric value will be inserted transparently into the field to which the combo box is bound.

robert aldwinckle on forums say:

After following your instructions, the words, " Successfully flushed the DNS Resolver Cache" came up.  I did a restart and I am afraid it is still the same.


The main value of   ipconfig  /flushdns  IMO is to make   ipconfig  /displaydns  clearer and more relevant.

Also, the only "restart" which might be necessary when using ipconfig /flushdns is restarting IE.   If IE has connected  successfully it holds on to the IP address it gets for a long time and does not try to do another lookup.   (Rebooting may be advisable for those other netsh commands but then IMO the others would be superfluous, in particular the ipconfig /flushdns.)  

To see this use  ipconfig  /flushdns,  retry using the same instance of IE and then do an  ipconfig  /displaydns.

Sometimes a problem happens when the host name is an alias of another host name (CNAME) and then it is helpful to make sure that a lookup is cached for that other host (and any others which it is chained to), in order to have a cached lookup which includes an IP address.   However, in the case of the host name that you are using, (I am seeing that) it is completely cached by one lookup (though with a relatively short Time To Live.)   So, use  ipconfig  /displaydns  to see what you are getting from your connection attempts.

Another problem that sometimes happens is related to the way that you make your request.   E.g. you said "I type in www.facebook.com and it just will not load."   The problem there is that if you don't give an explicit protocol prefix for your URL and if you are having a problem with slow lookup, then IE will try to treat the request as a search from the Address bar (if that feature is enabled.)   So then your symptom could be related to the performance of your Search from the Address bar Search Provider.

In fact, if that provider is Google I recently noticed that somehow using its result link can enable lookups to be cached which otherwise weren't and thus help IE connect to the desired host when it previously couldn't.
