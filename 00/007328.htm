graham mayor say:

You can make the process dynamic, so the height (width too if you wish) depends on the page you are currently viewing. Initially set all the pages to the same (larger) size having first noted the height (and optionally width) of the original size then use the multipage value to determine the height and width for the individual pages. e.g.

Private Sub MultiPage1_Change()
    With MultiPage1
        Select Case .Value
            Case 0, 1, 2, 3        'Pages 1 to 4
                Me.Height = 200
            Case 4        'Page 5
                Me.Height = 300
        End Select
        .Width = Me.Width - 25
        .Height = Me.Height - 75
    End With
    CommandButton1.Top = Me.Height - 52
End Sub

Don't forget to set the positions of any items outside the multipage, such as the command button(s)

ken sheridan say: You appear to have HWID and LID columns in both tables, which implies that the two columns constitute the composite primary key of Sys.  Is this the case?  If so then the LinkMasterFields LinkChildFields proprieties should include both columns, i.e. should be HWID;LID in each case.  If not, then the LID column is redundant in AA if it is transitively dependant on the key via HWID.  If it is functionally determined solely by the key (analogous for instance to a UnitPrice column in an OrderDetails table whose value is assigned from a UnitPrice column in a Products table when a row is inserted into OrderDetails) then you would expect it to show in the subdatasheet.

If you are expecting the foreign key of the referencing table to show in the subdatasheet, while I never have occasion to use a subdatasheet, my understanding is that this is not the case.  It would be pointless to do so as it would merely repeat the value of the primary key of the referenced table.
