Sub HTMLCleanup()
  Load HTMLForm
  HTMLForm.Show
  Select Case SubToRun$
    Case "DeleteHTML"
      Call DeleteHTML
    Case "FindAmpersand"
      Call FindAmpersand
    Case "FindExtended"
      Call FindExtended
    Case Else
  End Select
End Sub

'Part 1 -- Remove HTML comments
Sub DeleteHTML()
  Application.ScreenUpdating = False
  WordBasic.EditSelectAll
  SelText$ = WordBasic.GetText$(WordBasic.GetSelStartPos(), WordBasic.GetSelEndPos())
  InitLen = Len(SelText$)

Loop1:
  SelText$ = WordBasic.GetText$(WordBasic.GetSelStartPos(), WordBasic.GetSelEndPos())
  NewLen = Len(SelText$)
  'WordBasic.Print Int(100 - (NewLen * 100 / InitLen)); "%"
  StartPos = InStr(1, SelText$, "<!-")
  EndPos = InStr(1, SelText$, "->")
  If StartPos = 0 Or EndPos = 0 Then GoTo EndSub1
  WordBasic.EditGoTo "\StartOfSel"
  If StartPos <> 1 Then WordBasic.CharRight StartPos - 1
  WordBasic.CharRight EndPos - StartPos + 2, 1
  WordBasic.EditClear
  Count = Count + 1
  WordBasic.ExtendSelection
  WordBasic.EditGoTo "\EndOfDoc"
  WordBasic.Cancel
  GoTo Loop1

EndSub1:
  Application.ScreenUpdating = True
  Beep
  MsgBox Str$(Count) + " '<!-->' codes removed" + Chr$(13) + "End Part 1"

'Part 2 -- Remove HTML proper
  WordBasic.EditSelectAll
  Application.ScreenUpdating = False

Loop2:
  SelText$ = WordBasic.GetText$(WordBasic.GetSelStartPos(), WordBasic.GetSelEndPos())
  NewLen = Len(SelText$)
  'WordBasic.Print Int(100 - (NewLen * 100 / InitLen)); "%"
  StartPos = InStr(1, SelText$, "<")
  EndPos = InStr(1, SelText$, ">")
  If StartPos = 0 Or EndPos = 0 Then GoTo EndSub2
  WordBasic.EditGoTo "\StartOfSel"
  If StartPos <> 1 Then WordBasic.CharRight StartPos - 1
  WordBasic.CharRight EndPos - StartPos + 1, 1
  WordBasic.EditClear
  Count = Count + 1
  WordBasic.ExtendSelection
  WordBasic.EditGoTo "\EndOfDoc"
  WordBasic.Cancel
  GoTo Loop2

EndSub2:
  Application.ScreenUpdating = True
  WordBasic.Beep:  Beep
  MsgBox Str$(Count) + " '<>' codes removed" + Chr$(13) + "End Macro"
End Sub

Sub FindAmpersand()
  WordBasic.EditSelectAll
Loop3:
  SelText$ = WordBasic.GetText$(WordBasic.GetSelStartPos(), WordBasic.GetSelEndPos())
  StartPos = InStr(1, SelText$, "&")
  EndPos = InStr(1, SelText$, ";")
  If StartPos = 0 Or EndPos = 0 Then GoTo EndSub
  WordBasic.EditGoTo "\StartOfSel"
  If StartPos <> 1 Then WordBasic.CharRight StartPos - 1
  If StartPos > EndPos Then     'Check for ";" not part of HTML code
    WordBasic.ExtendSelection
    WordBasic.EditGoTo "\EndOfDoc"
    WordBasic.Cancel
    GoTo Loop3
  End If
  WordBasic.CharRight EndPos - StartPos + 1, 1
  WordBasic.CharColor 9
  Count = Count + 1
  WordBasic.CharRight 1
  WordBasic.ExtendSelection
  WordBasic.EditGoTo "\EndOfDoc"
  WordBasic.Cancel
  GoTo Loop3
EndSub:
  MsgBox Str$(Count) + " '&;' codes highlighted blue."
End Sub

Sub FindExtended()
  With Selection.Find
    .ClearFormatting
    .Replacement.ClearFormatting
    .Replacement.Font.Color = wdColorRed
    .Text = "[^0127-^0255]"
    .Replacement.Text = ""
    .Forward = True
    .Wrap = wdFindContinue
    .Format = True
    .MatchWildcards = True
    .Execute Replace:=wdReplaceAll
  End With
  MsgBox "Extended ASCII characters highlighted red."
End Sub

vincentchoy say:

Yammer is a social platform.  Most of the concerns you raised are better suited in a more regulated intranet platform like SharePoint.

Let me try to give you some answers to your questions  - I can learn form your experience as well.

1. Reporting a concern - Yammer has a key word monitoring function, which will send a email to a designated individual should words from the list be included in a conversation. There is no report this post function, but a ad-mention to the @networkadmin and writing a comment to the post should do the job.

2. Audit trail - As of now, it is not possible to edit a Yammer post once posted - so in a sense the conversation and who posted it is already there. I dont recall Yammer having a audit trail function.

3. There is no workflow to approve a new group. Think of a group as simply a online meeting where items are shared and discussed. So in that sense if you dont need an approval to have a meeting in the office, you dont need an approval to have a meeting in Yammer. That being said, you might be worried about 'official' groups. It's taken a bit of digging, but I have the solution!  Nothing to do with Creator's Windows update, looks like an error in an O365 update package for 32-bit installs some time around April / May 2017. It's also been rolled into the installer package too as I've just done a clean install confirming this is broken too.

The problem occurs because the 32-bit version of the OneNote search filter ONFILTER.DLL has been incorrectly installed in the directory

where the 64-bit version of the DLL should be. For the current O365 (16.0.8229.2045) the 32-bit DLL has a file size of 395,464 bytes whereas the 64-bit DLL is 464,072 bytes.

To fix the problem, you need to obtain the 64-bit version of the DLL and copy it to the above directory. I installed 64-bit O365 on a spare machine (having de-installed the 32-bit version first) as I couldn't find any other reliable way of getting the latest version of just the DLL. Who knows, maybe Microsoft will read this post, pay me $10,000 for debugging their programs and roll the patch into the next update.

Once you have the correct 64-bit DLL in place, close and re-open OneNote indexing and searching speed is soon back to super-fast - it looks like OneNote is now updating the index as you search so the first search was a bit slow, but after that it was back to instant.

If you check in Indexing Options the previous "OneIndex16://xxxxx" and "unavailable" tooltip error message have been replaced with "Microsoft OneNote". The icon is generic rather than OneNote; if you want to fix this too, the answer is to also copy the 64-bit ONENOTE.EXE into the same place you just put ONFILTER.DLL (there's no existing file to replace in this case).

After applying this fix to three machines last evening, I woke to found that two of them were "broken" again this morning, i.e. the 32-bit DLL was back in place. I guess this makes sense i.e. 64bit DLL needed by 64-bit Windows indexing service to create / update the index - 32-bit OneNote doesn't / can't be using it.

From event logs, both machines ran an unattended install of Click 2 Run Integration packages c2rint.16.msi and c2rint64.16.msi overnight. Looking further, there's a scheduled task running C2RClient each night, looks like this decided to run the reinstall. Time will tell if this happens every time I swap the DLLs or if it was one off triggered by something else I did yesterday. OK that's the fix, for those interested, here's the how I found it. Just not quite this quickly :/

  • As documented elsewhere, every time you started OneNote you got an event 3083 with description "The protocol handler OneIndex16 cannot be loaded. Error description: (HRESULT : 0x800700c1).".
  • HRESULT 0x800700c1 generally means you're trying to execute something that's not a valid executable, found various posts about 32/64 bit conflicts causing this, and earlier issues with searching in 32bit OneNote on 64-bit o/s.
  • Searched registry for OneIndex, found HKCR\OneIndex.ShellFolder\OneNoteApplicationPath with key IconPath pointing to ONENOTE.EXE in the ... root\VFS\ProgramFiles64 ... directory. Noted that the ONENOTE.EXE file didn't exist in this location.
  • Searched registry for other keys referring to this directory and found several including e.g. HKCR\CLSID\{0875DCB6-C686-4243-9432-ADCCF0B9F2D7} with description "Microsoft OneNote Namespace Extension for Windows Desktop Search" and with an InProcServer32 key pointing to ONFILTER.DLL in that same VFS directory.
  • Compared ONFILTER.DLL in this directory with the copy in the general program directory ... Microsoft Office\root and found they were the same file size.
  • Downloaded systinternals Sigcheck64 and confirmed that the DLL in the VFS directory was in fact the same 32bit DLL
