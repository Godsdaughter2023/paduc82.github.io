hansv mvp say:

Be careful - many Ctrl+letter/digit combinations already have a meaning in Excel:

Ctrl+A = select all

Ctrl+B = bold

Ctrl+C= copy

Ctrl+D = fill down

etc. Although it's possible to override these with your own shortcuts, that would be confusing to many users.

You could use Ctrl+Shift combinations instead - some aren't in use, and others are rather obscure.

You can activate the shortcuts in the Workbook_Open event procedure of the add-in and deactivate them in the Workbook_BeforeClose event procedure (both in the ThisWorkbook module):

 

Private Sub Workbook_Open()
    Application.OnKey "+^M", "ThisMacro"
    Application.OnKey "+^T", "ThatMacro"
End Sub

Private Sub Workbook_BeforeClose(Cancel As Boolean)
    Application.OnKey "+^M"
    Application.OnKey "+^T"
End Sub

 

+ is the "code" for the Shift key, ^ for the Ctrl key (and % for the Alt key).

The relationship should be on TestID and referential integrity must be enforced.  The reason for the change of table name is that the second table does not model a Ranges entity type, but a Tests entity type, of which Range is a virtual attribute defined by the two attributes High and Low.

The TestID column in Tests would normally be an autonumber for convenience.  The TestID column in CMP_Results, as a foreign key, should be along integer number data type.

A CMP_Results form should include a combo box set up as follows:

ControlSource:    TestID

RowSource:    SELECT TestID, Test FROM Tests ORDER BY Test;

BoundColumn:    1
ColumnCount:     2
ColumnWidths:    0

By setting the ColumnWidths property to zero the first column, TestID, is hidden and the user can select a test by name from the drop down list.

In a query the tables should be joined on TestID, using an INNER JOIN, which is the default in the query designer unless the join type is set to a LEFT or RIGHT OUTER JOIN in the relationship dialogue.

With the present tables, you can determine whether the lack of an enforced relationship has resulted in any invalid rows in CMP_Results by means of the following query:

SELECT *
FROM CMP_Results
WHERE NOT EXISTS
    (SELECT *

jan karel pieterse say:

Jim already pointed you to a great resource. I wanted to point you at a small programming error.

Despite what you might think, in this declaration:

Dim LinhaIniOri, LinhaFinOri, LinhaIniDes, LinhaFinDes As Integer

ONLY the last variable is declared as integer. The other ones are of type Variant.

In VBA you have to add the type definition to each one:

Dim LinhaIniOri As Integer, LinhaFinOri As Integer, LinhaIniDes As Integer, LinhaFinDes As Integer

For readabilities' sake I personally prefer:

Dim LinhaIniOri As Integer

Dim LinhaFinOri As Integer

Dim LinhaIniDes As Integer

Dim LinhaFinDes As Integer
