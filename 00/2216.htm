The solution is to log in using the new account.

ken sheridan say:

What you are referring to here is a structured key rather than a composite key; the latter term is usually used for a key made up of two or more columns.  The first 6 digits are easy:

 

Format(VBA.Date,"yymmdd")

 

However, is there a column in the table in which the date is stored?  If so you should not include these 6 digits in a structured key as this introduces redundancy, but include the date column in a composite primary key.  If there is no such column I would recommend that you include one, as a date/time data type with a DefaultValue property of Date(), rather than building a structured key.  The latter has no advantage, violates good design principles by using a column as a data structure, and complicates matters unnecessarily as you can return a single string at any time with a simple expression by concatenating the values in a computed column in a query or a computed control in a form or report.

 

The state code should not be included in a structured key as this would also introduce redundancy and the consequent risk of update anomalies; the state column should be a part of a composite key.

 

Is the 3-digit serial number per state/date, i.e. starting from 001 for each state/date combination?  Or are the values distinct within the table?  If the former then the column should be part of a composite key, if the latter then it can of course be the primary key.  Whichever is the case store the values in a column of long integer number data type and compute them when a row is inserted into the table via a form.  You'll find examples of how to do so in both scenarios in the file CustomNumber.zip in my public databases folder at:

 

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

 

The option for Sequential Numbering would be the appropriate one if the serial numbers are distinct in the table, that for Sequential Numbering by Group would be the appropriate one if the serial numbers are per state/date.  In my demo the group is by gender (you'll see that the two columns are the key of the NamesByGender table), in your case it would be by state and date.

 

With the 3-column composite key you can return the structured value at any time with the following expression:

 

Format([YourdateField], "yymmdd") & Format([YourSequentialNumberField],"000") & [State]

j.e. mcgimpsey say: In addition to Guruprasad's observation about the print driver (which, as he says, is the biggest factor), other issues that can change layout in XL (since you're posting in the XL forum, I'll ignore Word - but they're much the same) -

  • The default font selected in Preferences/General is used to determine column widths, which are set in units of "numeric characters" of that particular font, regardless of which font is used/displayed in the cells. Using a different default font from the one on the WinXL platform is almost guaranteed to give you a different layout.
  • Fonts - even fonts that have the same name sometimes have slightly different metrics, depending on their source and version - if you stick to those provided by MS with Office, you'll get best compatibility with WinOffice
  • Printer - if you're using a printer with built-in fonts, or with a generic print driver, different printers will produce different output.
  • Quality setting: In File/Page Setup, changing the quality setting can have dramatic effects on the layout.
  • "Fit to": I've seen the Fit to x pages wide by y pages tall setting cause massive differences in layout.
