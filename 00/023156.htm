ken sheridan say: Why use code at all for this?  Conditional formatting is far simpler, and is operative in all views.

I think you may be misunderstanding how the OnCurrent event property works.  In the report's Current event procedure enter some code like this:

Private Sub Report_Current()

    Debug.Print Me.[SomeBoundControlName]
    
End Sub

Open the report in report view.  Press Ctrl-G to open the debug window. You'll see nothing in the debug window.  Click on two or three rows in the detail section with different values in the SomeBoundControlName control.  You should see those values on separate lines in the debug window.  As Karl explained earlier, the event is triggered when you interact with the report.  If you want to selectively format all or part of the detail section (or any section for that matter) at runtime in a way that conditional formatting can't handle, governing the format by values in controls in the current instance of the section, then you should not open the report in Report View, but use the Format event procedure of the section, and open the report in Print Preview or send it to a printer.

They, I and You call it Floating Points. It's the way Excel does it's calculations. Excel works in Binary and often there's no precise binary equivalent of a decimal number and this is what;s happening here.

 

Excel (nearly) follows IEEE 754 (See link).

Far from introducing innacuracy rounding here will maintain it so. Use this and it round up to 6 decimal places...

 

=ROUND(SUM($A$2:$A2),6)

joeu2004 say: Lynn wrote:

I am adding up a set of numbers that have no more than 2 decimal places but in the result, Excel is displaying values in up to 14 decimal places.  See the last running total in my data below. The total should just be 5.18. Where are these numbers coming from & how do I stop Excel from doing that?


If you want the result to be accurate to 2 decimal places, change your formula to:

=ROUND(SUM($A$2:$A12),2).

Do something similar to any other formula where you expect the result to be "exactly" dollars-and-cents.

Alternatively, you can set the "Precision as displayed" calculation option.  It might be okay to use in your situation.  But generally, I do not recommend it.

The problem arises because of the way that Excel (and many applications) represents numbers and performs arithmetic.  Infinitesimal differences creep into calculations with non-integers (and division of integers).  The infinitesimal differences can accumulate and become significant.
