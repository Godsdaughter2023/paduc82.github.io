macropod say:

The problem you described is far from being normal for Word. I have used footnotes extensively and have never encountered such behaviour except when the citation in the footnote uses a CITATION field. By design, those will revert to Microsoft's interpretation of the referencing format you're using (Note: Microsoft's implementation is often non-compliant). However, that behaviour is due to the use of the CITATION field - any edits/formatting changes you make to the content within any field will be lost when Word next updates the field display (e.g. @ Print Preview).

If the behaviour you're seeing isn't caused by using a CITATION field, you may have a faulty Office installation or a corrupt document. You might start by repairing the Office installation (via Windows Control Panel > Programs > Programs & Features > Microsoft Office (version) > Change > Repair). If that doesn't remedy things, corrupt documents can often be 'repaired' by inserting a new, empty, paragraph at the very end, copying everything except that new paragraph to a new document based on the same template (headers & footers may need to be copied separately), closing the old document and saving the new one over it.

ken sheridan say: These days you don't need any code at all.  Since Access 2007 (or maybe even earlier) an Image control has had a ControlSource property which you can simply set to the name of a column in the table in which the path to the image file is stored.  

If the location of image files is a single folder, and you want to split the path to cater for the file being opened on different systems or the location of the folder being changed, rather than computing the path every time I'd suggest you either do this once only at start-up and assign the path to the folder to a public variable in a standard module, or, and this is the approach I've adopted in similar circumstances, store the path in a column in a local single row table to which you can bind a form which is opened automatically at first start-up.  You'd then need a little function which returns the concatenated full path as the Image control's ControlSource property.

You'll find examples of the use of image controls in reports in Images.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the Images_Simple.accdb file for instance associates a single image with each row in the table by storing its full path in a text column.  For an example of a situation where the path to a single folder is stored in a separate table see the InvoicePDF demo in the same OneDrive folder.  This does it for PDF files and opens with a simple little dialogue form in which the path can be entered by browsing to the folder.  The form includes a check box which can be unchecked to prevent the form opening subsequently, which is done by changing the database's start up form property in code.

When loading image files at runtime be sure that the image resolution is no larger than necessary for your purposes.  Modern digital cameras produce very large high resolution files which will be quite slow to load with a report of multiple images.  A simple image management application like Picasa easily enables a reduced resolution copy of a large image file to be made.
