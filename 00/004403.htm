tom van stiphout say:

That's not quite what I meant. I'm guessing QryDonationsDDP is your report query, and you're thinking of simply moving the generation of the invoice numbers from the report (running sum) to the report query.

I wrote: You would create the invoice number when the payment is entered in your Payments form, and saved with the Invoice record. Then simply display it on the report.

What I mean by that is a change in your database design. You would have an Invoice table, and it would have an InvoiceNumber field. Records would be created from the Payments form each time a payment comes in. The InvoiceNumber would be created using the DMax+1 technique. For example in the Form_BeforeInsert I would write:
Me.InvoiceNumber = DMax("InvoiceNumber", "tblInvoices") + 1
Then I would change QryDonationsDDP to join with tblInvoices and pick up the InvoiceNumber field. Then in the report I set the InvoiceNumber field's ControlSource to InvoiceNumber.

tom van stiphout say:

OK, fair enough. Then at the end of the year you generate all Invoice records, one per donor. Still the same database design with an InvoiceNumber using DMax+1 technique, and a report query that picks up that value.

ken sheridan say:

I will need it to work on a select query that groups and sums.

Invoices, as a separate entity type related to multiple payments, should be modelled by a related Invoices table of which the InvNo column will be the primary key, or if a separate autonumber column is used as a surrogate key, defined as a candidate key by being uniquely indexed.  It is when a row into this table is inserted that the value will be computed and assigned to the InvNo column.

You can group and aggregate values in a query which joins the Invoices table to whatever other tables are relevant, but for invoices the grouping and aggregation is more commonly done in a report, which may also include one or more subreports.  InvoicePDF.zip in my same SkyDrive folder includes an example of such a report, though the primary purpose of the demo is to illustrate how to output a report as a PDF file; it just happens to be based on a simple hypothetical invoicing application.

scottgem say:

As Ken has said, you should have an Invoices table. This could be very simple:

tblInvoice

InvoiceID, (PK Autonumber)

DonorID (FK)

If you need the Invoice numbers to be Sequential within a Year, see my blog article on Sequential Numbering and add a Sequence field to tblInvoice. 

The Actual Invoice number would be generated as indicated in the blog using an expression like:

=Sequence & " - " & Year(InvDate)-1 & "DDP"
