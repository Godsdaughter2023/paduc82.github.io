My emails send and receive in one .pst email folder, but the sent emails show up in the Sent Items folder in the other.pst folder. How can I have just one folder showing in the viewing panel on the left side?

Email accounts configured using IMAP will have two (.pst) data files associated with the account

(1) - Will contain all your email folders. If Outlook doesn't find or cant use this file for any reason, it gets automatically created and then re-sync all the data from the server. This data store is a "mirror" of what's on the server

(2) - The other will contain your default contact/calendar folders but will also show an extraneous set of other folders (<inbox>, <Sent> etc) . This is the critical one to keep since this info doesn't sync with the server

The above applies to OL'2010 and earlier. Data stores for Imap accounts in OL'2013/2016 work differently.

Not in the way you want. Rather then repeat what was included in my previous response - quickly review the description provided. For the sake of this thread, going to assume you are only dealing with IMAP accounts. The FIRST Imap account will always create 2 PST files as described. Each additional IMAP accounts will only add one PST file. These PST files will appear in your Outlook Nav Pane. There is no way to set the destination of a given IMAP account to another set of folders. The only way you can accomplish that would be by creating <rules>.

Re: Q#3 - Backing up your IMAP data (Outlook '2010 and earlier - not applicable to OL'2013/'2016)

Again two parts to that answer. The PST file for an IMAP account containing your email folder data is a mirror of what resides on the server. If it ever gets corrupted/deleted etc - Outlook will re-create it. The "default data file' on the other hand contains the local contact/calendar data.

edhanswork say:

I use PowerQuery (Get & Transform) to pull data from other workbooks. The basic steps are:

  1. Your source data is best in a named range or Excel table, not just a grid of data.
  2. Using PowerQuery, get files from SharePoint
  3. Filter the result to the file you need
  4. CLick the Binary link next to the file name. This will just get the data from THAT file as opposed to the more powerful "combine binaries" which is designed to combine multiple files.
  5. Select the range or table you want to pull in
  6. further filter the data and/or delete columns to only have what you ultimately want in your destination Excel file.
  7. Load that query to an Excel range.

Now when you REFRESH ALL, the data will update.

Depending on the size of your sharepoint site, it can be a bit slow, but it has two primary benefits:

  1. The files can be closed when updated.
  2. You can pull in an entire range (table) of data, not just a single cell. Your destination cells will grow/shrink as needed depending on what happened in the source workbook.

1.  You should create a Campuses table to ensure data integrity.

2.  All relationships must be enforced.  See the image below in which the relationship dialogue for the relationship between Schools and Classes is shown as an example.  An unenforced relationship is as useful as a chocolate poker.

ken sheridan say: Time does not permit a detailed analysis of the model, but one obvious fault stands out:

The Campus table contains transitive dependencies and is consequently not in 3NF.  Assuming that  the geographical units represent a tree structured hierarchy then each determines the next in the hierarchy:

Id--->Ward--->District--->City--->Province

Consequently the District, City and Province columns are redundant.  Ward (or more usually WardID) should reference the primary key of a Wards table which would in turn contain a foreign key referencing the key of a Districts table and so on.  You'll find an example of such a structure in DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file has Contacts, Cities, Regions and Countries tables related in this way.  In the section of the demo is a contacts form which illustrates, amongst other things, how to enter data in a top-down sequence, by selecting a country, then a region and finally a city.  The combo boxes, the first two of which are unbound, also illustrate how to insert new rows into the three referenced tables by typing the new country and/or region and/or city names directly into the combo box.  Code in the relevant control's NotInList event procedure then inserts a new row into the relevant table.  Again this must be done in a top-down sequence of course.

A few points regarding object naming:

1.  You have used plural nouns for table names in some cases, e.g. Schools and Contacts, but singular nouns in other cases, e.g. Campus and Class.  I strongly favour the use of plural, or occasionally collective, nouns as table names to reflect the fact that a table is a set.  For column names I favour the use of singular nouns to reflect the fact that a column represents an attribute.

2.  Why have you given tblMonths a 'tbl' tag.  I dislike the use of tags for table or column names intensely.  They merely interfere with the semantics.  I do use tags for other object types, and for variable names in code, however, by and large following the Reddick convention.

3.  You have used 'Id' as a generic name for primary keys.  I know Microsoft does this in templates and wizards etc, but you'll find no shortage of bad practice in those.  I'd strongly recommend that each primary key name reflects the entity type modelled by the table, e.g. ContactID, ClassID etc.  Using, wherever possible, the same name for a foreign key as that of the primary key which it references gives greater semantic clarity, especially when writing complex SQL statements.
