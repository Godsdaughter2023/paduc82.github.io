bobdubery say:

Trouble opening microsoft outlook files downloaded with IE11 on Win 10

Hi all,

We have an app with a web browser front end. We stipulate that the browser to be used is IE. An important function of this system is the ability to upload documentation in various forms and to later download it for viewing.

With win 7 client machines we run IE in compatibility mode. All types of attached files download and IE can determine what to do with them - even if the mime type is incorrectly declared in the response headers from the server. I am assuming this is down to the mime sniffing that IE uses.

When we started testing with a Win 10 client machine attachments would open if we gave a fully qualified URL EG

https://172.32.2.27/attachedFiles/claimsImages/00009662_008_001_005_01.MSG

But most of the time we present files via a DLL (this is to help with a security requirement that some of our customers have - they don't want simple URLs that could be copied into other windows or email out, and they don't want the path name to the file to be visible in the URL). So the URL is more likely to be something like

https://172.32.2.27/scripts/xxxdevdata.dll?ContentType=application/vnd.ms-outlook&OpenURL=00009662_008_001_005_01.MSG

The first problem we ran into was that the URLs (generated by a script) ALWAYS specified ContentType=application/vnd.ms-word and that content type is returned in the response header the server sends to the browser.

So we changed this for WIN 10 (making sure that it still works in WIN 7) so that the appropriate content type is specified. EG for BMPs we specific image/bmp, for PDF application/pdf and so on. For outlook messages saved as a type .msg we specify application/vnd.ms-outlook

So just about every type of file we have tested with on WIN10, with the appropriate content type declared, opens. The exception is .MSG and that's an important one for us. Our users will attach saved Outlook messages to build up a trail of communications.

With the type declared as application/vnd.ms-outlook and working on WIN 7 with a URL like https://172.32.2.27/scripts/xxxdevdata.dll?ContentType=application/vnd.ms-outlook&OpenURL=00009662_008_001_005_01.MSG

https://172.32.2.27/scripts/xxxdevdata.dll?ContentType=application/vnd.ms-outlook&OpenURL=00009662_008_001_005_01.MSG

The browser tries to download a file with the name xxxdevdata.MSG and then asks if the message should be saved or opened.

On Win 10 the browser (which runs in compatibility mode for the site and has the site trusted) tries to download a file name

xxxdevdata.DLL. Then warns that "this file does not have a program associated with it for performing this action." Not surprising because it downloaded a .DLL instead of a .MSG file.

I checked the browser settings for trusted sites. Mime sniffing is enabled. I have tried using different file types application/vnd.msoutlook and message/rfc822. The different file types do not modify the browser's apparent behaviour in any way.

Has anybody run into something similar or is able to point me in the right direction?

Thanks


Bob

bobdubery say:

UPDATE! I have solved the problem.

I used the Content-disposition header to force the issue.

Content-disposition: attachment; filename="myEmail.msg"

Now download manager still starts up, but the file is now named correctly (myEmail.msg in this example).

I'm still interested to know why this was necessary when running IE 11 on Win 10 but not on Win 7 (it does no harm on win 7).

3. boot Windows in safe mode (with networking)

If any or a combination of the above solves your problem there is third-

       a so-called desktop search.
