Hmmm..........let me try a longer answer with an appropriate embedded "disclaimer" as it applies to my own experience along with suggestions that probably don't apply to the vast majority of people who may end up reading this.

#1 - I have a variety of test PST files that I move between various Outlook versions on a regular basis. None of the Outlook versions use any kind of 3rd party sync/anti-spam addins nor are any 3rd party apps in use that integrate themselves with Outlook - or to put this another way, chance of any kind of corruption resulting from 3rd party apps (addin or otherwise) is non-existent. This movement of files is not something I do on a daily (or otherwise frequent regular) basis.

#2 - Next item is that all PST files that get moved around are <Unicode> format files. I would never suggest anyone to move ANSI format PST files be moved back and forth or to put it even more directly, highly recommend that (1) scanpst file is run on any ANSI based PST file and then (2) get converted to UNICODE in whatever older OL version is being used BEFORE it's opened in OL'2013/'2016.

My experience over a lot of years is that I have never run across an issue moving a PST file between any Outlook versions nor have I run across any scenarios/docs that would contradict that experience. The critical caveat to all of this are the scenarios outlined in item #1. It is possible to have a 3rd party program cause a potential issue in a data file that may not be immediately (or ever) be noticed but get impacted when moved between old and new versions of Outlook.

To summarize, if it were me and I wanted/needed to do what you want to do with an already working procedure to move PST data files that has been working, introducing OL'2013/'2016 into the mix wouldn't deter me but to repeat, I don't move production PST files between systems on a day-to-day basis as part of any kind of standard operating procedure.

For those who may be reading this and not familiar with how to determine if a PST file is ANSI or UNICODE - the process is simply right-clicking on the root of the PST file in any version of Outlook --> Data File Properties --> Advanced --> if the <Format> description includes "97-2002" then  it's an ANSI file (something that may well be the case for those who have migrated a PST file created in OL'2002(XP) to succeeding new versions.

Lastly, would advise anyone following this practice to maintain multi-generational backup copies of the PST file(s) being moved (multi-generational = a fancy term for multiple copies that get rotated.

steve rindsberg say:

Thanks for sharing the file.  I get the same results here, so either neither of us is going mad or we're both in good company. ;-)

I extracted the original EMF from the PPTX (made a copy of the PPTX, changed the extension to .ZIP, opened that, went into the \ppt\media folder and dragged the single EMF out onto the desktop).

I opened that in CorelDraw, which is a bit more sophisticated than PPT about these things and picked it apart, found that the four pictures in the graph are 7x7 pixel images.  Each of the little squares within them is literally one individual pixel.  Blown WAY up.

PowerPoint has always tried to help with images by antialiasing them; or in plain English, blurring and upsampling them to remove the jaggies.  And there's no way to make it stop "helping".

So what you have here is extremely small images, which provoked PPT into an utter frenzy of antialiasing. 

If you had the original images that made up the EMF, you could upsample them in an image editing program w/o antialiasing, but since you don't, about the best suggestion I have is to make a copy of the slide, ungroup the graphic until you can grab just the image with four little pictures and the gradient scale. 

Copy that then use Paste Special to paste back a PNG.

Crop the pasted PNG back down to just one of the four pictures.

Copy and crop three more times until you've got four pictures. 

Move them atop the original pictures in the EMF and NOW they should go to PDF as you expect them to.  Tested and working as described here.

WAY more work than you should reasonably have to do, so if anyone's got a better idea, please, let's hear it.
