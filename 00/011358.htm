steve8d say:

Excel 2010 "Event Handler"

Hello From Steved

VLOOKUP(B2,'Race Meeting'!$B$2:H519,6,FALSE)
End Sub

hansv mvp say:

I wouldn't use the Worksheet_SelectionChange event for this - it occurs each time you select a different cell or range, even if you don't modify it. I'd use the Worksheet_Change event of Sheet1 and monitor changes in column B (the column with the lookup value):


Private Sub Worksheet_Change(ByVal Target As Range)
    Dim cel As Range
    Dim varReturn As Variant
    If Not Intersect(Range("B:B"), Target) Is Nothing Then
        Application.EnableEvents = False
        For Each cel In Intersect(Range("B:B"), Target)
            varReturn = Application.VLookup(cel.Value, _
                Worksheets("Race Meeting").Range("B:G"), 6, False)
            If IsError(varReturn) Then
                varReturn = ""
            End If
            cel.Offset(ColumnOffset:=2).Value = varReturn
        Next cel
        Application.EnableEvents = True
    End If
End Sub

john - project say:

Tim,

You say you deleted all the rogue calendars from your file and the pool file. Did you also check each of the other sharer files. Custom calendars are saved with the file so it's possible that if one of the other sharer files has those calendars, it might be showing up through the pool/sharer files link structure.

Also, do you regularly share the files with anyone else? Or does anyone else have access to the linked structure such that their Global may be getting "cross-pollinated" into the structure.

If none of the above is applicable, then I'd say the pool/sharer structure is corrupt. In order to clean that up, open each sharer file and switch the resource structure to "use own resources". That will transfer the resources used by that file from the pool to the using file. Once that is done for all files, save the pool and all sharer files. You might also want to double check the pool and each file to insure all the rogue calendars are gone. If not, delete them and save again. Then save the pool and each sharer file as XML. Then re-open each of the XML files in Project and re-link the pool to the sharer files. Normally that will clean up a corrupt structure.

John

john - project say:

BeccaHolmes,

Resource pool/sharer files are linked structures and are therefore prone to corruption. None of the files in the structure should ever be renamed, moved, overwritten, or saved off to another location. All files should be in the same directory on a local drive and not be operated over a network.

It sounds like that last part (network drive) may have had a glitch and the files got corrupted. I suggest you unlink all the files from the pool, and save each file as XML. Then re-open each XML file in Project and re-link to the pool. Hopefully this will clear any corruption but be advised that you are still at risk if you continue to operate over a network.

Hope this helps.

John
