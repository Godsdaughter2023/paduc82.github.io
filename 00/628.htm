joeu2004 say: Adam wrote:
Although we have not noticed any real activity on the network utilization section of Task manager, we are wondering if a 1GB link would help solve this issue instead of the current 100MB.

No one can really say without looking at the network topology and the load on the route end-to-end.

If Task Manager can be believed and it shows no significant network traffic, obviously replacing a 100MB link with a 1GB link into the local computer will have no impact.

But replacing a lossy network interface might.  You might try monitoring local network traffic using WireShark or something similar to see if you are plagued by packet loss.  This can be recognized by looking at the TCP ACK packets.

My guess:  networking is not the problem at all.  It would be very poor design if Excel were to loop without delay while waiting for network input.

Adam wrote:
Maybe the calculations are extremely complex and Excel just needs that amount of time.

Ding!  I'd put money on it.

Excessive use of volatile functions, poorly-designed event macros, use of "lookups" over large ranges (including COUNTIF, SUMIF et al) can all contribute to this.  See http://www.decisionmodels.com/optspeed.htm and http://www.decisionmodels.com/calcsecrets.htm, among many other articles that easily found using a Google search.

Adam wrote:
We have an executive in our company that is constantly using Excel 2007 and is always complaining of how slowly it runs when retrieving data from our SQL server.

One strategy for isolating the root cause of performance problems is "divide and conquer".

If you can minimally modify the offending workbook to eliminate the data retrieval, that would go a long way toward determining whether the SQL data acquisition per se or the Excel data processing is the problem.

I am not familiar with how Excel might acquire data from an SQL server.  But if you can modify the method so that it returns a constant without accessing the SQL server, that would eliminate the SQL data acquisition factor without eliminating the processing of the "SQL" data that Excel might do.

charles kenyon say:
This is not an infrequent problem. There are two likely culprits. The least-likely is corporate IT if you are on a network. The most likely is a poorly written Add-In that blocks changes to normal.dotm.

Rich told you how to find normal.dotm. You can right-click on it and open it, make changes and try saving it.

Another way to open it is to go into the vba editor by pressing Alt-F11 with a document open and pasting the statement below into the immediate window:
application.NormalTemplate.OpenAsDocument

Then press Enter and your normal template will open as a document. When you have normal.dot open be careful to not type any text (including headers or footers).

Getting to the likely root of the problem: Add-Ins go to your Developer tab and try both of the buttons for Add-Ins. Try unchecking all of them and then see if you can save changes to your defaults. If you can, uncheck them one at a time to see what is blocking you. Decide if that is an Add-In you really need, and in any case, complain to the manufacturer.

This usually happens when an Add-In makes changes to the Word user interface that is stored in normal.dotm and doesn't want that change saved. It tells Word that the normal.dotm file has already been saved.
