ken sheridan say:

Are these bound controls in a form bound to a table or query?  If so the first thing you need to understand is that the table should only contain a SubActivityID column, not an ActivityID column.  To have both would mean the table contains redundancy because SubActivity determines Activity, and is not normalized to Third Normal Form (3NF), which requires all non-key columns to be determined solely by the whole of the table's primary key.  Otherwise it is wide open to the risk of update anomalies leading to inconsistent data.

 

It is still possible to do what you want by using an unbound combo box to select the activity.  However, because you are using surrogate keys, probably autonumbers, you cannot simply use two combo boxes, one unbound, the other bound.  Strictly speaking that's only true of continuous forms view, not single form view, but it would require code to assign a value to the unbound (Activity) combo box in the form's Current event procedure, so using the same solution regardless of whether the form is in single or continuous forms view makes sense.

 

The way it's done is to use 'hybrid' controls in which a text box is superimposed on each combo box to give the appearance of a single combo box control, basing the form on a query which includes the referenced tables (Activities and SubActivities) so that the text values can be shown in the text boxes superimposed on the combo.

 

Don't worry, it's not as complex as it might sound and you'll find an example in ComboDemo.zip in my public databases folder at:

 

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

 

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

 

The model to follow in your case is that for a simple two-level hierarchy.  In this Disticts are analogous to your Activities, and Parishes to your Sub-activities.
