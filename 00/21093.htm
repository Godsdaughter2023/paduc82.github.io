This problem occurs when you select Ignore on an email message and then a later message from that thread comes into your mailbox. Outlook deletes the email and it also keeps track of all future email related to the ignored message. If a future message related to the originally ignored message arrives in your Inbox, Outlook automatically moves these future messages to your Deleted Items folder.

To resolve this problem, remove the "ignore" status of the email thread using the following steps:

1. Select your Deleted Items folder.
2.Select the message that is currently set to be ignored by Outlook.
3.Click Ignore on the Delete group of the ribbon.

    Original Title: Delayed Email - August 2013

    colinbowern say:
    Hi Andreo - in the private message area you said "Upon checking your your domain and it appears that its MX record is not yet configured, which is why your currently experiencing this issue."

    At first I thought this can't be true because my zone record on my DNS provider clearly shows the correct MX record in place. I found a tool to give me a view into MX resolution -- MXToolbox. When it ran the MX resolution test it returned an error that it was getting caught in a redirect loop.

    Logging into the DNS hosting service to discover that the primary name server for the zone was down. I went to update my zone record with the alternatives and noticed that the secondary name servers had also changed at some point -- likely to deal with something like a DDOS attack.  The remaining valid name server was down so no one could access the domain.  The delivery failure messages started to make a bit more sense in terms of the big picture - they were coming from my outbound SMTP server, not a stuck inbound queue.  The status code of 4.4.7 on the failure notification message header probably should be 4.4.1 in these cases, but that is besides the point as that is how Exchange or whatever SMTP server you are using interprets it.  Furthermore with both of us using Outlook.com I didn't clue into the fact that the path of the message had not left the outbound queue because it could not contact the MX server for the domain.

    To resolve I updated the NS entries in my zone record with servers that are operational.  I then synced those with the domain record held by my domain registrar.  Once the internet managed to sync up those records everything started flowing through.

    Thanks for the hint as that led to the resolution of the issue.  Not only that it also allowed the domains.live.com record for the custom domain hosting to finalize the verification of domain ownership.  That had been an issue since the start but because that offering does not provide any technical feedback as to why it could not verify the domain ownership I never clued into the fact that it might have been trying to connect to one of the bad name servers.  The final string holding this together must have been severed last week when the remaining name server went down.

    The end result is that this was not an Outlook.com issue but a DNS issue. Troubleshooting this problem required, however, a few pieces of information to identify where and when these things get stuck.  Some thoughts for the future that would help the tools provide more details to help resolve this type of issue:
    • Outlook.com could expose a view into messages stuck in the outbound queue
    • Windows Live Domains could expose a technical description as to why it cannot verify domain ownership
    • Exchange (or whatever the SMTP server in use is) could expose a failure message that indicates that DNS resolution is the problem
    • Support personnel should ask if this is a custom domain earlier in the support cycle (albeit I'm sure the majority of cases they are not custom domains)
    • Support personnel should be more forthcoming with technical details as to what they are seeing rather than generalized statements like "the possible reason that you are getting a delayed delivery notification is that the domain xxx has configuration issue" --  I understand this may not always be ideal if you are dealing with non-technical users though
    • Support personnel can recommend verifying MX and SMTP connectivity through services like MXToolbox (or ship their own on the custom domains management area)
    Thanks Andreo, Ralph, Jessa, Leilani for your assistance in resolving the issue.  I wish turnaround was a bit quicker but am thankful to have found a solution to the issue with your help.
