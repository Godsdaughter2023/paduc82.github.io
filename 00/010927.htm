ken sheridan say: For inserting data into a single table you need not have any subforms at all.  You can simply bind a form to the table and place logical groups of controls bound to the columns in the table on separate pages of a tab control.  Data which you need to be permanently visible can be shown in bound controls above and/or below the tab control.

However, as Scott has pointed out, and as has been made clear to you in your other current thread, a single table is a bad design.  It will inevitably mean that there is major redundancy in the database.  This not only makes data entry much more difficult and time-consuming for users, but more importantly, it introduces a high risk of update anomalies which prejudice the integrity of the data.  The database will consequently be unreliable as an effective business tool, and cannot be regarded as fit for purpose.

I would very strongly recommend that you draw this to the attention of the organisation in question.  The user-friendly interface which they desire can be achieved in the context of a correctly normalised database comprising multiple related tables, each of which models a valid real world entity type, and whose columns represent valid attributes of the entity type.  This underlying model should be of no concern to the users, and not exposed to them in the operation of the database.

You might like to take a look at DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

As its name suggests, this file provides an introduction to some of the basic principles and methodologies of relational database design.  You might also care to look at Normalization.zip in the same OneDrive folder.  This is as simple as possible (but no more so) introduction to the process of formal normalization, an undertanding of which to at least Third Normal Form is essential to anyone designing a relational database.

There are multiple ways this could have happened.  One is that they could have been the owner of that address at one time but allowed it to lapse or deleted it without changing their Facebook information.  Later on, you requested the account and since it wasn't taken, you got it.  If that person ever forgets the Facebook password and tries to use the address to recover, you'll get the messages from Facebook and you'll be able to take over their Facebook account if you're so inclined.  In my opinion, this is exactly why mail providers should never allow addresses to be reused.

Another way is that they made a mistake when creating the Facebook account and accidentally used an address they don't own.  Perhaps they thought their address is abcdef@outlook.com but it's really abc.def@outlook.com. They entered the first value when they should have used the second.  Then along you come and get an account that looks like the first.  Since that account wasn't in use, there was no problem on the Outlook.com side.

There undoubtably are other scenarios as well.

erick chu msft say:

Hi Livio Jeangoudoux,

 

Question 1 - Do we need to have two ADFS farms and different federated domains for each?

 

Some admins will use one ADFS farms. However, based on my knowledge, you can also use two ADFS farms. It is depending on your requirements.

 

 

Question 2 - Do we need to synchronize user from their respective AD forest A.lan and B.lan with Azure AD Connect to Office 365 as multi forest?

 

If you use one ADFS farms, the answer is yes. You need to sync them to Office 365 as multiple forests.

 

 

Question 3 - Does ADFS Farm from AD forest A.lan could authenticate users from AD forest B.lan for Office 365 with the federated domains departement1.com or departement2.com or we really need two ADFS farms.

 

No, you can only have one ADFS farm. However, forest trusts exist between the forests this will work. Meantime, each forest much have unique UPN login suffixes for this scenario. If no trusts exist between the forests than multiple ADFS farms are required.

jeeped say:

The cause of this error is Adobe's PDFMaker.xla, which loads at startup from the XLSTART folder. This is an add-on that add the Adobe toolbar in Excel upon startup.

The easiest solution is to close all instances of Excel and delete PDFMaker.xla from your computer. Typically, PDFMaker.xla is stored in your profile directory:

C:\Documents and Settings\<user>\Application Data\Microsoft\Excel\XLSTART\PDFMaker.xla

In other scenarios, PDFMaker.xla could be stored in your Program Files directory. Be suree to search your C: drive for any and all instances of the file.

After you deleted the file PDFMaker.xla, the error "Compile Error in Hidden Module: Distmon" when starting Excel shouldn’t appear.
