Use the IIF function with the following specs:
If the Final Grade (field) from the Students table is less than 60, Letter Grade (field) should be "F".
If the Final Grade (field) from the Students table is less than 70, Letter Grade (field) should be "D".
If the Final Grade (field) from the Students table is less than 80, Letter Grade (field) should be "C".

marshall barton say:

The Switch function is better than using a sequence of nested IIf functions:


   Switch([Final Grade]<60,"F", [Final Grade]<70,"D", [Final Grade]<80,"C", [Final Grade]<90,"B", [Final Grade]<100,"A", True,"I")


But it is a poor practice to hard code the data values in an expression.  Data really needs to exist only in a column in a table.  In this case, the table LetterGrades would look like:

Score  Grade

   60      F

   70      D

   80      C

   90      B

 100      A


Then you can get the letter grade in a calculated query field by using a subquery:


LetterGrade:  (SELECT TOP 1  Grade FROM LetterGrades WHERE [Final Grade]<Score ORDER BY Score)


Based on your original question, I presume that you are not fully up to speed in the fundamental rules of relational databases so you may not grasp the importance of this.  But, for now, you at least need to be aware that this long winded explanation of a valid approach to do what you asked really is far better than using functions such as IIf and Switch.

ken sheridan say: Bear in mind that by doing this you are encoding data in an expression, which is not good practice.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

The correct way to model this would be by means of a table like this:

LetterGrades
....LowerLimit
....UpperLimit
....LetterGrade (PK)

So it would  have rows:

0      59    F
60    69    D
70    79    C
80    89    B
90    200  A

The upper limit for grade A is an arbitrarily high one which will never be reached.  BTW what happened to grade E?

It is then a matter of a simple join to return the grade for any score, e.g.

SELECT StudentID, Score, LetterGrade
FROM Scores INNER JOIN LetterGrades
ON (Scores.Score BETWEEN LetterGrades.LowerLimit AND LetterGrades.UpperLimit);

As well as respecting the Information Principle this has the important practical implication that maintenance is now simply a matter of amending values in the table rather than modifying the code.

marshall barton say:

Ken,


There are certainly convenience considerations in favor of using a three column table, but if a range boundary is ever changed, two records will need to be edited.  While I can't remember which Normalization Rule (4?, 4.5?, ??) a three column table violates for bounding contiguous ranges, I still use a two column table in this kind of simple situation.


Marsh

ken sheridan say: We've discussed this before, Marsh.  I fully acknowledge the redundancy as the price of convenience.

Thinking a little further about the current case, however, I do think a two column table would be best as the number of possible score values is very limited.  So I'd simply use a table with serial score values from 0 to 200 (or whatever arbitrary top score is appropriate - when I was at school marks were always out of 100) as the key and the grade for each score in a non-key column.  Data entry could be via an unbound form in which the grade and its upper and lower limits were entered, executing some simple SQL to either insert or update each row as appropriate.

The join is then a simple one of equality.

It would be very easy to constrain the table to the score value being an integer >=0 and the cardinality of the table being the top score+1.  It would also be advisable to include some mechanism to ensure that each subset of grades corresponded with a continuous sequence of scores, and the grade's values were (descending alphabetically) in the same order (ascending numerically) as either the lower or higher limit values per grade.

marshall barton say:

Yeah, well, Ken, er, um, my memory of these discussions gets more and more vague as a geometric function of age  :-)


I hadn't thought of a table with all possible scores with their corresponding letter grade.  In this case it would only be 100 records and the query would then be a trivial join (outer to deal with invalid and missing scores).  This definitely sounds like the most straightforward and flexible approach.  Good idea.
