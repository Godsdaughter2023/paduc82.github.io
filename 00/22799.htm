Office 365 & SharePoint Profile inconsistencies

When I delete a user account/email in Office 365, they still show up on our SharePoint team sites as users, ie. they continue to have a SharePoint user profile.  Any insights?  Do we need to delete their profile in both places when they leave the company? Is there any better way to manage this?   

iris he msft say:

Hi MLSchuk,

 

That's the normal behavior.

 

After we delete a user from Office 365 portal admin center, his user profiles user profile is marked for deletion. The profile will be preserved in the database in a deleted state for 30 days. This coincides with the time that's allocated to restore a deleted user account from the Azure Active Directory recycle bin.

 

To let his user profile disappear before 30 days, we have two ways:

1. As a global admin or SharePoint service administrator, we can manually delete his user profiles in SharePoint Admin Center->User Profiles->Manage User Profiles.

2. Manually remove it from deleted users recycle him before it 30 days in Office 365 admin center using PowerShell.

The second suggestion is recommended. Because SharePoint user profiles are synced from Office 365 Azure AD in portal. That means, if we deleted the user's profile in SharePoint Admin Center, the user still exists in deleted users recycle bin. However, if we delete the user from deleted users recycle bin, the user's profile will be removed. So, in case the user will be restored from deleted users recycle bin by someone someday, I'd suggest you remove the user from deleted user's recycle bin.

 

To do that, follow the steps below:

1. Install Windows Azure Active Directory PowerShell by following Install the Azure AD Module part in this article: https://msdn.microsoft.com/en-us/library/jj151815.aspx (64-bit is recommended)

2. Then open and connect to your Office 365 Admin center by typing: Connect-MsolService.

3. Then input your ID and password.

4. Run this command: Remove-MsolUser -UserPrincipalName *** Email address is removed for privacy *** -RemoveFromRecycleBin

Feel free to let me know if we can discuss further.

 

Thanks,
Iris

scottgem say:

You create a child table the same way you create any table. The difference is in the usage, not the process. A child table is a table that has parent records. So in the example we have the table of instructors is the parent, mainly because each instructor has ONE record. The child table is the child because it can have MANY records related to an instructor. In your case 1-3 records representing the course each instructor must take.


So InstructorID is the primary key in the Instructor's table and the foreign key in the courses taken table. Foreign keys are generally entered using a combobox. The Rowsource of the combo is the table where the key is the primary key, but the Controlsource is the foreign key field. So lets say you have a form where you enter the courses taken. You have a combobox to select the InstructorID, another to select the CourseID, maybe another to select the LocationID. So the courses taken table can have three parents all sharing the same children.
