ken sheridan say:
The only value which you need to store in tblSafeguarding to reference the row in tblAppointments is that in the AppointmentID foreign key column.  To do otherwise is not only unnecessary, but more importantly introduces redundancy and the consequent risk of update anomalies as tblSafeguarding would not be normalized to Third Normal Form (3NF).

You can do this with no code or macro whatsoever by including a safeguarding subform linked to the parent appointments form on AppointmentID.  Alternatively you can use a separate liked safeguarding form and open it filtered to the current appointment in the appointments form.

You'll find example of both approaches in FormsDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly. 

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.
This little demo file includes examples using subforms on a tab control of the parent form, and linked forms in both synchronized and dialogue mode.  It also includes an example of linked forms where the referencing form, rather than being filtered to the matching records, shows all records and moves the record pointer to the first match.  This won't concern you, however.

If you want to open a linked form at an empty new record, use the AcFormAdd constant as the DataMode argument of the OpenForm method, e.g to open the events form in my demo at  a new record the code would be amended as follows:

        DoCmd.OpenForm FORMNAME, _
            WhereCondition:=strCriteria, _
            DataMode:=acFormAdd, _
            WindowMode:=acDialog, _
            OpenArgs:=(Me.ClientID)

ken sheridan say:
The safeguarding table should contain an AppointmentID foreign key column and must not duplicate any other column from the appointments table, so having a MemberID column in the safeguarding table is incorrect.

The subform should be linked to the parent form by setting the LinkMasterFields and LinkChildFields properties of the subform control, i.e. the control in the parent form which houses the subform, to AppointmentID in each case.

The subform need not include any controls bound to the primary key SafeguardingID or to the foreign key AppointmentID.  The values of both of these columns are inserted automatically, the first by virtue of it being an autonumber, the second by the linking mechanism.  Neither have any semantic significance, so making them visible in the user interface is pointless.  However, the subform must contain a control or controls bound to at least one non-key column representing an attribute or attributes of the entity type which the safeguarding table is modelling.  The problem here is that, on the basis of waht you said in your original post, having discarded the invalid MemberID column, you don't have any such columns(s) in the table.  This begs the question as to what this table is meant to be modelling in real world terms?  You will have to explain to us exactly what this is, as at present the table contributes nothing whatsoever to the information content of the database.

Having said that, I'd like to go back to something you said in your original post, viz.' One appointment would have one safeguarding record'.  This would mean that the relationship type between the two tables would be one-to-one.  One-to-one relationships are quite rare.  They are used to model a type/sub-type.  The DatabaseBasics demo in the OneDrive folder to which I gave you a link earlier has an example of a store selling books and CDs, where each is a sub-type of type Products.  A sub-type is characterised by sharing all attributes of its (super) type but not those of other subtypes.

There is nothing to suggest that you have a type/sub-type model here, so a one-to-one relationship type would not be appropriate. As far as I can see, all you have to do is include one or more columns for whatever are the attributes of safeguarding in the appointments table, and dispense with the safeguarding table completely.  At its simplest this might mean including a Boolean (Yes/No) column named Safeguarding in the appointments table.  Without knowing what is meant by safeguarding in your real world scenario, however, I can't say more than that.

ken sheridan say:
The first comment I'd make is that I don't see a single enforced relationship in your model.  An unenforced relationship serves no purpose whatsoever, and it's questionable whether such a relationship can legitimately be called a relationship at all in the context of the database relational model.

To comment on the totality of your overall model is beyond the time and space available here.  It would require a substantial systematic analysis of the conceptual model, which is basically the set of real world entity types which constitute the reality which the database is intended to model, followed by the formulation of the best logical model in which the relationship types between the entity types are established, and finally the translation of this into a physical model which faithfully represents the logical model by a set of correctly normalized tables.  I'll confine my comments to the appointment/safeguarding issue therefore.

While there is no need for separate appointments and safeguarding tables, as the relationship type between them is one-to-one, having separate tables is not a problem per se (I'll come back to this at the end), provided that the AppointmentID column in tblSafeguarding is uniquely indexed (no duplicates).  You should then find that when you enforce the relationship by selecting 'enforce referential integrity' in the relationship dialogue, a one-to-one relationship type will be created.  It's important to note, however, that a one-to-one relationship type is directional, so one table must be the referenced table and the other the referencing table.  In this case tblAppointments must be the referenced table as you will need to insert a row into this before a row can be inserted into tblSafeguarding.

One thing which doesn't look right.  I note that the appointments table contains a SafeguardingID foreign key column referencing an unseen  table below the window.  What is the intended purpose of this?

As regards the interface this can simply be a parent form based on tblAppointments, within which is a subform based on tblSafeguarding, linking the subform to the parent form on AppointmentID.  The process of inserting data would thus be to firstly insert appointment into the subform after the user has inserted data into the parent form.  It can be done with a little programming trickery, however, and you'll find an example in Families.zip in the same OneDrive folder to which I gave you a link earlier:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

In this simple little demo file the first option, using a form/subform, is the relevant one.  This requires at least one family member to be inserted in the subform after inserting a family name in the parent form.  If you don't and try to move to another family record or close the form you'll be prompted to insert a family member.  If you don't your only option is to delete the family record in the parent form (NB, not just the name in the control, but the record itself).

The way it works is that code in the parent form's Current and Unload event procedures checks to see if there is at least one related row in the FamilyMembers table for the family record just entered.  If the code in the Current event procedures finds none, it moves the record pointer back to the newly entered record, as this code is executed as soon as the record pointer moves to another record.  The code in the Unload event procedure simply cancels the unloading, and hence closure, of the form if no family members for the current family are found, which it does by setting the return value of the event procedure's Cancel argument to True.

Whether you want to go to this trouble to enforce the insertion of a safeguarding record when an appointment record is entered is for you to decide.  You might feel you can trust users to do so.  It's your judgment call.

The other issue is one of space in the form. The safeguarding table contains a large number of columns, so you might find it difficult to squeeze enough controls into a subform without requiring scrolling by the user, which is never very satisfactory.  One possibility you might like to consider is to have a number of different safeguarding subforms, all linked to the parent form on AppointmentID, putting these on separate pages of a tab control.  The controls in each subform would be bound to a separate subset of the table's columns, with the first page containing those which it is mandatory to enter when a safeguarding record is created, and the other pages containing controls which can be entered subsequently.  The columns to which control in these subsidiary subforms on page two and following are bound would have to accept Nulls, i.e. their Required property would be False (No) in the table design, or some at least might be given a default value in the table design, so that a value is inserted automatically.  These values can then be subsequently edited where necessary.

I said I'd revisit the question of having separate appointments and safeguarding tables.  While this can be accommodated by a form/subform as described above, if the two tables were coalesced into one, then it immediately overcomes the question of how you enforce the insertion of safeguarding data when an appointment record is entered.  All you'd have to do would be to make the relevant safeguarding columnsmandatory by setting their Required property to True (Yes) in the table design.  By doing this for those columns which must have data inserted when an appointment record is created, but not for those whose values can be inserted later, the user must enter values in the mandatory columns before they'll be able to save the record.  Again, to save space, the controls can be separated into logical groups on separate pages of a tab control within the form, which would now be based on the single table, with no need for a subform.

Win 8/8.1 Total Backup System: File History + RecImg.EXE + Refresh + SlimImage

 

Win 8 has refined your backup and restore options. Before Win 8 you could create an Image Copy backup of the full drive (or selected directories), but it was hard to separate backups of the OS, Applications and personal Data unless you created a separate

 

Out of the box, MS has been advertising the “Refresh” and “Reset” options. Personally, I think they missed a valuable selling point by not mentioning RecImg.EXE in conjunction with Refresh. It removes my main concern/objection to Refresh.

 

Reset is equivalent to using the “Recovery Partition” on previous versions of Windows. The factory would create a “Recovery Partition” that you could use to “reset the computer to factory fresh”. It would typically include all of the factory “crapware” and it would wipe out all of your data files and installed applications.  A less than desirable outcome but an appropriate reaction to a bad malware infection.  The “recovery partition” type of restore typically had to be accessed at boot time while Reset can be accessed from within Windows 8.

 

Refresh is roughly equivalent to an image copy backup.  A Reset will restore a backup of the OS and Metro applications. It will wipe out “legacy” / x86 desktop type applications BUT it will not affect your personal data files, which is much better than a Reset.  But there is still that issue of wiping out your installed desktop applications. I do not have documented confirmation, but I’m guessing that Metro apps are treated like personal data and those directories/folders are simply not reset back to the old condition.

 

RecImg.EXE is a backup tool that ships with Windows 8. It creates the backup “image” that is used by the Refresh tool. This image that was created at the factory is why using Refresh will wipe out installed desktop programs in the “C:\Program files” and “C:\Program Files (x86)” folders because they haven’t been backed up. However, using RecImg.EXE you can create new images that will include recently installed desktop applications.

 

RecImg Manager is a third party tool the provides a GUI to access the RecImg.EXE command line tool and backup files.

 

File History is a new Win 8 feature. It is an extension to the “Previous Versions” and “Volume Shadow Copy” features. File History continuously monitors files stored in Libraries, Desktop, Contacts, and Favorites folders. It runs on a predetermined schedule, and when it detects changes in any file, it makes a backup of that file to an External drive. It isn’t enabled by default.

Dear Pamela,

I am an independent advisor, and I am trying to help other users in Community with my experience in Microsoft products.

I understand that this is frustrating to receive lots of spam and unwanted emails. It is even more frustrating to receive scams, spoofing and phishing emails. Scammers nowadays are using multiple methods to intimidate users.

Please note that Postmaster ( https://sendersupport.olc.protection.outlook.co... ) is actually blocking lots of spammers ( and I quote ):
For example, we've implemented a number of mechanisms to reduce the burden of junk email which currently prevents nearly 4.5 billion email messages from reaching Outlook.com users every day!

There are couple of ways to deal with scammers and unwanted emails and improve how your anty filter will work (I am sure that you are aware of some and you are using them, but maybe additional information in support articles will help you):

1. First of all check your spam filters and see suggestions on how to keep spam out of your inbox https://support.office.com/en-us/article/help-k...
2. Report the phishing, spam and unwanted emails https://support.office.com/en-us/article/deal-w...
3. Add emails or domains to block sender (those that were actually used to send emails - see #5) list https://support.office.com/en-us/article/block-...
4. Create specific rules for subjects or common phrases that are present in those emails https://support.office.com/en-us/article/use-in... and add conditions, that are common for those emails, for example additional keywords, recipient address includes etc.
5. Nowadays some scammers have learned how to mask their emails addresses that you can see in the emails. However, in the message source you will find the real one (and the real domain) In order to view message source please check this article https://support.office.com/en-us/article/receiv...
6. You can also use the option at the bottom https://outlook.live.com/mail/options/mail/junk... to receive only trust email from addresses in my Safe senders and domains list and Safe mailing lists

You can also check tips on how to stay safe online from Microsoft https://www.microsoft.com/en-us/digital-skills/...

Hope this will help your email free from Junk!
