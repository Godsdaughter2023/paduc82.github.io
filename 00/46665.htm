lemp say:
If chkdsk doesn't run on startup, when does it "cry about every external drive"?

I strongly suspect that your drive issues are caused by something else, and that "getting rid of chkdsk" will only mask the symptom and not address the real problem with your failing drives.  However, if you insist, there are a number of things you can do.

First, you should realize that chkdsk will run if the "dirty bit" for a drive is detected on boot up.  This bit is set when Windows determines that the file system may be in an inconsistent state.  Typically, this happens when Windows is not shut down properly, e.g., the power is turned off abruptly.  Generally this does NOT mean that there's a hardware problem with the disk.  Unless you specify the /r or /b parameters, chkdsk will only attempt to fix the file system, which should not physically affect the drive.

If a hard drive is already failing, using the /r or /b parameters for chkdsk may hasten the failure.  See, e.g., post #3 here: http://www.overclock.net/t/1603282/what-does-the-chkdsk-b-argument-do

You can query the dirty bit on any drive by using an Elevated Command Prompt to run the following command:

   fsutil dirty query <drive>

For example:

If the dirty bit is set, then (normally) chkdsk will run on the next boot.  If this bit is set, there is something wrong with your file system ... and the problem is not chkdsk.

Although it's not recommended, you can disable automatic checking on boot (but you will be ignoring the fact that there is something wrong with your file system) by using the chkntfs command

     CHKNTFS /X <volume>

You can list multiple drives, e.g., --

     chkntfs /x c: d: e:

There are more drastic methods to disable automatic disk checking.  None of these are particularly good ideas.  Rather than give instructions, however (because all of the following methods are far more likely than not to cause harm to your system), I'll just list them.  If you really want to risk damaging your system, you can find each of these methods using a search engine.

  1. You can edit the registry instead of using the chkntfs command.  There's no good reason to make registry changes instead of using the UI, but if you do, be sure to make a backup of the registry first.
  2. You could also either remove or disable the entire registry key that controls automatic checking, for example by using AutoRuns
  3. You can manually clear the dirty bit (a particularly bad and dangerous idea)
  4. You can defeat Windows File Protection and delete chkdsk.exe (if you do this without also disabling automatic checking, you'll just create more error messages).
