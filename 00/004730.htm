With the current design of Outlook.com, the feature of dragging custom folder to change its order is already not available.

For a workaround, you try to following:

jay freedman say:

Tabs set in a document that was created in an earlier version of Word should continue to work the same in Word 2016 (with the exception of tab stops set to the right of the right margin, which stopped working in Word 2013). You can change the positions of tab stops for the selected paragraphs on the horizontal ruler or in the Tabs dialog box (http://wordfaqs.ssbarnhill.com/SettingTabs.htm), and you can modify them in paragraph styles through the Modify > Format > Tabs sequence.

If tabs are replaced with underlines in printed copy but don't appear on screen, that may be a display issue. Changing the zoom factor slightly (to 90% or 110%) may make the lines appear on screen. If they're there, it indicates that the tab stop is defined with an underline "leader". You can use the Tabs dialog box to change the leader to "None".

... it correctly lists 'libTest.dylib' as located in the directory. It looks like the Error53 is misleading, the file is found but an error occurs that is declared as Error53: File not Found. Also, the error only occurs when I call a .dylib compiled with links to PostgreSQL's libpq C library, that is, compiled using:

clang++ libTest.c -dynamiclib -I$(pg_config --includedir) -L$(pg_config --libdir) -lpq -o libTest.dylib

If I compile without linking to libpq then the .dylib library is called without problems.

alejandro dopazo say:

Solution:

Excel for Mac misleads by reporting error53: File not found when it doesn't find a library the .dylib links to. In my case libTest.dylib linked to a library located in a directory different that the one set by pg_config.

I uninstalled brew versions of both postgresql and psql and set the path to the library installed by Mac's posgreapp to:

 

PATH="/Applications/Postgres.app/Contents/Versions/latest/bin:$PATH"

in Terminal. Now Excel for Mac runs without problems.
