ken sheridan say: It can be done in a form in continuous forms view, but you need to use a hybrid control made up of a combo box bound on which is superimposed a text box so that only the arrow of the combo box is visible to the right of the combo box.  To the user it appears as a single combo box control.

The requirements are as follows:

1.  The form is based on a query in which the referencing table (Evaluations) is joined to the referenced table (Employees) on the key of the latter, e.g.

SELECT Evaluations.*,
FirstName & " " & LastName AS FullName
FROM Evaluations INNER JOIN Employees
ON Evaluations.EmployeeID = Employees.EmployeeID
ORDER BY LastName, FirstName, EvaluationDate;

2.  The combo box is bound to the EmployeeID column from Evaluations and set up to show FirstName & " " & LastName as its RowSource's visible column.

3.  The Text box is bound to the computed FullName column.

You can now vary the RowSource property of the combo box in the form's Current event procedure to restrict it to those rows WHERE Archived = FALSE in a new record, i.e. where the form's NewRecord property is True, and unrestricted otherwise.

When the user selects an employee in the combo box the value of the superimposed text box will be the name of the selected employee, so this will remain visible in the text box, whereas it will be hidden in the text box element of the underlying combo box.

You'll find examples of the use of hybrid controls, albeit in a different context, in ComboDemo.zip in my public databases folder at:

Now it's time for a little into into how IPv6 works:  Routing with IPv6 does NOT involve NAT in any way.  In short, *EVERY* IPv6 address is a *PUBLIC* address.  It's sort of like what happens when you use a dial-up service that does not NAT you via an RFC1918 (or similar 'private') address:  your machine is exposed on EVERY open port for potential attack.  if anyone remembers "Win Nuke" from a while back, it's *THAT* kind of POTENTIAL vulnerability.  yeah.

This is why nearly everyone who knows anything about public IP addresses and the benefits of a NAT firewall uses a NAT firewall.  But you don't have this with IPv6.

To combat this on my OWN network, I use FreeBSD's built-in firewall [which is similar to what Linux has in many ways] to BLOCK all incoming ports that Windows listens to.  I'll probably have to REVISE this list now because of listening ports in windows 10.  That's ok, it's like "block incoming Ipv6 traffic on this tunnel for this port".  not too difficult, just a matter of doing it.

But it would HELP if windows 10 wasn't LISTENING! ON! THESE! PORTS! IN! THE! FIRST! PLACE!

And I doubt that the Windows firewall would protect you.  It's likely to have security craters built right in.

Now, on the more POSITIVE side, I noticed that Edge defaults to an IPv6 address when both IPv6 and IPv4 are available.  I used my own domain as a test, verified it connected to the IPv6 address.  However, I don't know if this is configurable.

I also noticed that the IPv6 address and routing came up working without any intervention from me, using the default configuration.  This is ALSO good, and as it should be.  Previously, I only had IPv4 NAT available [I had to do some hacking on the VM config side and plug a network cable in for a bridged adaptor so IPv6 would work].

But as I suspected, there are those OPEN PORTS to deal with, and IPv6 addresses are *PUBLICALLY* *ACCESSIBLE*.

So is anyone ELSE out there testing IPv6?  I probably won't be doing much else other than verifying basic functionality, and finding those open ports.
