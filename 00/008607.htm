Is there a way in MS Access to limit the ability to edit data to the - Microsoft Community

tim4692 say:

Is there a way in MS Access to limit the ability to edit data to the user who entered it?

I have a database where supervisors enter notes on their subordinates. I would like to make it so that only the supervisor who made the entry (or someone with admin privileges) can edit these entries once they are made. Is there a way to do this?
ken sheridan say: Yes, but rather than setting the properties of each control individually you can designate which controls in a form you wish to disable/lock by setting the Tag property of each control to a string expression such as:

 LockMe

You then need a means of determining the rank of the current user, which might not be available in the RecordSource of every form to which you want to apply the restriction.  As you are using user and group security you can get the current user with the CurrentUser function.  If you have a table Users say, with UserName and Rank columns, you can look up the current user's rank in this, so the code for the form's Open event procedure would be like this:

Dim ctrl As Control
Dim intRank As Integer

intRank = DLookup("Rank", "Users", "UserName =""" & CurrentUser() & """")

For Each ctrl In Me.Controls
    If ctrl.Tag = "LockMe" Then
        ctrl.Enabled = (intRank < 4)
        ctrl.Locked = (intRank >=4)
    End if
Next ctrl

By setting both the Enabled and Locked properties of a control the appearance of the control is unchanged, but the user won't even be able to move focus to it.  Setting only the Locked property allows a user to move focus to the control, but not update it.  Setting only the Enabled property on the other hand denies the user access to the control, but also greys it out.

This of course only prevents lower ranked personnel from adding or editing the values in the controls; it does not hide the data from those personnel.  If you wanted to do the latter you could add the following line within the loop after setting the Locked and Enabled properties of each control:

    If intRank >= 4 Then ctl.ForeColor = ctrl.BackColor

This hides the contents of the control by setting the colour of the text to the colour of the background.

If you ever contemplate converting the database to the .accdb format then you won't be able to use user and group security, so it would be necessary to use the user's Windows log-in name rather than their Access log-in name.  The former can be obtained by calling the GetUser() function after pasting the following module into your database:

' module basGetUser
Option Compare Database
Option Explicit

Declare Function GetUserName Lib "advapi32.dll" Alias "GetUserNameA" (ByVal _
   lpBuffer As String, nSize As Long) As Long

Public Function GetUser() As String

   Dim strBuffer As String
   Dim lngSize As Long, lngRetVal As Long
   
   lngSize = 199
   strBuffer = String$(200, 0)
   
   lngRetVal = GetUserName(strBuffer, lngSize)
   
   GetUser = Left$(strBuffer, lngSize - 1)

End Function

This is the module used by the demo files to which I referred you in my earlier post.

It does sound as if you have entered ACTUAL values (actual start? actual duration? Actual Work?). Now for Project the word "actual" means "done". So when you tell project that a task has an acutual start in the future, it will accept that because after all it's your project. Project isn't "stupid" or "smart", it's just a set of formulas.

Try to get rid of all Actual values (you yourself say it's unlogical they are there) then Project will react more "logically"
