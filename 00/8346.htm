' Input Variables:
' ~~~~~~~~~~~~~~~~

'
' Usage:
' ~~~~~~
' ReadURLFile("http://www.google.ca/index.html")
' ReadURLFile("http://www.SomeDomain.com/SomeFolder/SomeFIle.txt")
' ReadURLFile("http://www.SomeDomain.com/SomeFolder/SomeFIle.dat")
'
' Revision History:
' Rev       Date(yyyy/mm/dd)        Description
' **************************************************************************************
' 1         2012-Jul-07                 Initial Release
'---------------------------------------------------------------------------------------
Function ReadURLFile(sFullURLWFile As String) As String
'Returns an "" (VbNullString) if an error is reported back from the server
'        otherwise returns the Response Text from the server
    On Error GoTo Error_Handler
    Dim oHttp           As Object

    Set oHttp = CreateObject("MSXML2.ServerXMLHTTP")

    Call oHttp.Open("GET", sFullURLWFile, False)
    'Call oHttp.setRequestHeader "Content-Type", "text/xml"
    Call oHttp.Send
    'Check for any errors reported by the server
    If oHttp.status >= 400 And oHttp.status <= 599 Then
        ReadURLFile = ""
        GoTo Error_Handler
    Else
        ReadURLFile = oHttp.ResponseText
    End If

Error_Handler_Exit:
    On Error Resume Next
    Call oHttp.Close
    Set oHttp = Nothing
    Exit Function

Error_Handler:
    If oHttp.status >= 400 And oHttp.status <= 599 Then
        MsgBox "The following error has occured." & vbCrLf & vbCrLf & _
               "Error Number: " & oHttp.status & vbCrLf & _
               "Error Source: ReadURLFile" & vbCrLf & _
               "Error Description: " & oHttp.StatusText, _
               vbCritical, "An Error has Occured!"
    Else
        MsgBox "The following error has occured." & vbCrLf & vbCrLf & _
               "Error Number: " & Err.Number & vbCrLf & _
               "Error Source: ReadURLFile" & vbCrLf & _
               "Error Description: " & Err.Description, _
               vbCritical, "An Error has Occured!"
    End If
    Resume Error_Handler_Exit
End Function

albert d. kallal say:

You have to distinguish between a file sitting in a folder on the web server (that you can use via FTP), and that of a HTTP file exposed in a web page.

 

In your example, if the web page has a URL (link) to a file to download, then the code example you have should work (better yet is Daniel's example). However, if the file is simply sitting on the web server and NOT exposed as a link in the web page, then you need to use FTP to determine if the file exists.

You can try/test this by hand. Take the URL and file name you are trying in your code, and type it into your browser - when you do so, do you see the picture?

 

So make sure you include the WHOLE correctly formatted URL as Daniel posts shows:

 

Eg:

 

http://www.SomeDomain.com/SomeFolder/SomeFIle.JPG

 

So, try this manually, and fire up your web browser and type in the url with the file name – does it display?

  

As noted, if the link is NOT in some web page, and is only a file in a directly on the web server, then you CAN NOT use your browser or the above sample code.

That being the case, then:

 

The folder MUST be exposed and setup to allow FTP

You must use FTP and NOT http. (in some cases the web server can be setup to dish out files from a dir via http, but in “most” cases you require to logon to the server (FTP) and then use FTP commands to determine if the file exists).

 

So you need to clear up the URL issue and if you are/can/will using HTTP or do you have to use FTP?

  

So the problem here (as I see this) is if the file(s) in question have “hyperlinks” in a standard web page and you can click on those links (then examples posted here will work).

However, if the file(s) in question reside in a folder on the web server, then unless that dir is exposed via http (you type in the http web URL path with folder), and THEN YOU see file dir listing in your web browser? If not, then you MUST use FTP to determine if the file exists on the  web folder.

By default, Word includes both the number and the text of a heading in a table of contents.

What you have to do is make use of two styles: one numbered (add no text to paragraphs using that style) and one unnumbered (for the heading text). ricky_brundritt say:

No, streetside is not dead. In fact, the Bing Maps team just made it available to developers a month ago in their new map API.

The issue you see is one of coverage. Google has been collecting streetside imagery for nearly a decade. A large percentage of the imagery they have is very old, yet it still gets highlighted in the coverage map. So it can be a bit misleading. Also worth noting that the radius of a data point in the Google maps coverage map is larger, so it looks like it has a lot more coverage than it actually has. You can see this when looking at how far the coverage area goes out over the water.

It costs a lot money to capture new imagery so when it comes to expanding coverage, a lot of things are taken into consideration, such as where users are or go. The current coverage in Bing Maps likely covers 80-90% of there current user base.

rohn007 say:
The name of the general concept is "OCR", Optical Character Recognition.

The general process is:
  • you scan the document into an image file on your computer\
  • you pass the image to an OCR program to extract the text.
  • you now can edit the text.

Since you have OneNote 2010, you already have a basic "OCR" program. 


After you scan the paper, in OneNote:

  • click on the Insert tab
  • click on the picture button
  • in the dialog box, navigate to the place you have image stored
  • single click on the file name
  • click OK to insert it
  • right click on the image of the file in OneNote
  • select "Copy Text from Picture"
  • you can now paste and edit the text anywhere

Note, the formatting will be gone and depending on the quality of the image you may have to correct spelling errors.


Or, as Palcouk pointed out, so printers / scanners include better OCR software. By better, I mean that they do a better job of retaining character and paragraph formatting.


You can also get free OCR tools from the internet that do a better job than OneNote.

FYI: Word 2013 has PDF OCR built in. You can open a PDF file and Word will automatically create a DOCX version of the file.
