joeu2004 say: Ryno33 wrote:
Current I am using the following two formulas to calculate monthly income from YTD values

1.  To calculate complete months: =DATEDIF(G6,G7+1,"m")

2. To calculate partial month: =DAY(G7)/DAY(EOMONTH(G7,0))

The problem that arises is when the person starts on a day that is not the first.  Example: Start date of 4/19/2010 to an end date of today (7/29/2010).  The formula still completes the partial month from the first of the month even though its really not.

It is always good practice to tell us what you expect for results versus what you are getting.

I presume that your complaint is:  for the partial month, you are computing 29/31, whereas as you would like either 10/31 or 11/31 (which?) -- i.e. the partial month from 7/19 to 7/29.

Your formula for complete months suggests that the start and end dates are inclusive.  So I believe your intent is:  the 3 complete months run from 4/19 through 7/18 inclusive, and the partial month is the 11 days from 7/19 through 7/29 inclusive.

Thus, one way to compute the partial month fraction is:

=(G7-EDATE(G6,B1)+1)/DAY(EOMONTH(G7,0))

where B1 represents the result of the complete-month formula, DATEDIF(G6,G7+1,"m").

You should test both formulas (complete and partial) with start 2/29/2008 and end 2/28/2009 as well as start 2/28/2008 and end 2/28/2009 to be sure the results meet your expectations.

Using your complete-month formula and my partial-month formula, I get 12 months plus 1 day (i.e. 1/28) for both combinations.  I believe that is consistent with US law.  But it might not be consistent with law in your jurisdiction or with your intentions.

Note that start 1/31/2008 and end 1/31/2009 is also 12 months plus 1 day (i.e. 1/31).  That is based on my interpretation that the start and end dates is inclusive.

The numbers actually appear to be getting rounded somewhat randomly.  Here are some results of a test:

There is nothing random about it.  The devil is in the details.

And at least some of the "changes" are due to formatting after you re-open the CSV file.  So you have an opportunity to remedy the problem sometimes.

I will explain what is happening in detail below.  But cutting to the chase, these are the important take-away points.


1. Do not save as a CSV file if you intend to access the data using Excel.

Instead, save as an Excel file.  If you might access the data using XL2003, save as an Excel 97-2003 Workbook (xls).  Otherwise, save as an Excel (2007-10) Workbook (xlsx) or Macro-enabled Workbook (xlsm) file.

Alternatively, if you want to access the data from a non-Excel application, perhaps you can save as an XML file.  Check the non-Excel application to see if it supports XML input.


2. When you save as a CSV file, numbers are written to the file as they appear in cells.

So if you want to save numbers with greater precision, you need to change the cell format accordingly before saving as a CSV file.

You should also change the cell format accordingly after re-opening the CSV, especially before saving the worksheet as a CSV file again later.


3. When you use Paste-Special Value, only the value is pasted, not its format.  The format of the pasted cell will remain as it was before; typically, the format is General.

You might consider using Paste-Special Value-and-Number-Formatting instead in order to preserve the original format.


4. If you must save as a CSV file and re-open it in Excel, you might consider doing that with VBA code for both writing and reading the CSV file.

That will allow you to ensure that the data are written to the CSV file with the required precision, and that cells are formatted appropriately after re-opening the CSV file.

A CSV file is simply a text file.  It does not retain any formatting information.

Excel 2010 VBA ADODB: execute access 2010 stored query

I have been trying to execute a previously created stored query from Excel VBA. The query is stored in Access. I could re-create the SQL string in Excel VBA, but it involves long and complex joins in Access. It would be better to create the query in Access and connect to it from Excel VBA.

When I started, I though I could simply use the query name in "FROM [My Access Query Name]" in a SQL statement. But I found this did not work.

A Web search indicates that an ADODB command is needed. I found the code below, but it does not work. I get the following error message:

"Run-Time-Error '2147467259 (80004005); Method 'Execute of object'_Connect failed".

Can someone offer corrections to my code or a new method that works?

Thank you.

Sub excelquerycmd()
Dim cnn As ADODB.Connection
Dim rs As ADODB.Recordset
Path = Application.ThisWorkbook.Path
file = Path & "\MyFile.accdb"
Set cnn = New ADODB.Connection
With cnn
    .Provider = "Microsoft.ACE.OLEDB.12.0;"
    .Open file
End With
Dim cmd As New ADODB.Command
cmd.CommandType = adCmdStoredProc
cmd.CommandText = "[My Access Query Name]"
cmd.ActiveConnection = cnn
Set rs = cmd.Execute()
Sheets(1).Cells.Clear
Sheets(1).Range("a2").CopyFromRecordset rs
rs.Close
End Sub
