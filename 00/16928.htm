Sub macro_01()

Dim r As Long, c As Long, r1 As Long, c1 As Long
Dim t As Long, i As Long, j As Long
Dim ws1 As Worksheet
Set ws1 = ActiveSheet

c = Selection.Columns.Count
Sheets.Add

For Each rr In ws1.Range(ws1.[B1], ws1.Cells(r, c)).SpecialCells(xlCellTypeConstants)
Cells(t, 1) = rr.Value
t = t + 1
Next
Range("A:A").Sort , Key1:=[A1], Order1:=xlAscending, Header:=xlNo
r1 = ActiveSheet.UsedRange.Rows.Count
For t = r1 To 2 Step -1
If Cells(t, 1) = Cells(t - 1, 1) Then Rows(t).Delete
Next
r1 = ActiveSheet.UsedRange.Rows.Count
Range("A1:A" & r1).Copy
[B1].PasteSpecial xlValues, Transpose:=True
Columns(1).ClearContents
ws1.Range("A1:A" & r).Copy
[A2].PasteSpecial xlValues
Application.CutCopyMode = False
c1 = r1 + 1
For i = 1 To r
For j = 2 To c1
For Each rr In ws1.Range(ws1.Cells(i, 2), ws1.Cells(i, c))
If rr.Value = Cells(1, j).Value Then
Cells(i + 1, j).Value = "X"

Next

john - project say:
Michael,
The question is, is "Data Designer" a single unique task shared by both business services or is it in fact a similar task performed separately by both business services? If the former, then it should be its own separate task and the two business services can link to it using normal predecessor/successor links. If the latter, then the data task should be listed as two separate tasks under each business service (e.g. Data Designer A and Data Designer B).

Just for reference, "Data Designer" is not a task, it is a statement. It almost sounds like a resource and if it is, it should be listed a such on the Resource Sheet.

I find many users tend to get lax with task descriptions when laying out a project. A true task that has work content, as opposed to a summary line heading, should be described in action verb format. For example, perhaps the "Data Designer" should really be something like "Design Data Structure for A".

Another pitfall I see in many project schedules is a task description that is overly abbreviated such that it is not unique to the work being performed. For example, a schedule describing the build of a house would probably have a painting effort. If the schedule is broken down by rooms the painting task under each room category should not be "paint room" because painting the living room is not the same task as painting the kitchen, etc. Each task should be described in sufficient unique detail without getting too verbose.

My two cents.

john p spencer say:

First thing to check:  Open the query directly and see if you can edit a record there.

If not, you may have to use a sub-form to enter referer details.

In Access Help type the following in the Answer Wizard tab
         When can I update data from a query
In the results window select About Updating Data and select the appropriate option.

If you cannot edit the data in a query, this list may help you identify why it is not updatable:
--Query based on three or more tables in which there is a many-to-one-to-many relationship
--Query that includes a linked ODBC  table with no unique index or a Paradox table without a primary key
--Query that includes more than one table or query and the tables or queries aren't joined.

--It has a TRANSFORM clause. A Crosstab query is always read-only.
--It uses First(), Sum(), Max(), Count(), etc. in the SELECT clause. Queries that aggregate records are read-only.
--It has a GROUP BY clause. A Totals query is always read-only.
--It contains a DISTINCT predicate (i.e.; Unique Values is YES in the query's properties.)
--It involves a UNION. Union queries are always read-only.

--It has a subquery in the SELECT clause.
--It uses JOINs of different directions on multiple tables in the FROM clause.
--The fields in a JOIN are not indexed correctly: there is no primary key or unique index on the JOINed fields.
--The query's Recordset Type property is Snapshot. Set Recordset Type to "Dynaset" in the query's Properties.
--The query is based on another query that is read-only (stacked query.)

--Your permissions are read-only (Access security.)
--The database is opened read-only, or the file attributes are read-only, or the database is on read-only media (e.g. CD-ROM, network drive without write privileges.)

-- Access 2007: The query calls a VBA function, but the database is not in a trusted location so the code cannot run.
