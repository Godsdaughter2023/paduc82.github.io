hansv mvp say:

You can use an array of edge types:

 

Sub SetBorders()
    Dim xlEdgeType As Variant
    For Each xlEdgeType In Array(xlEdgeLeft, xlEdgeTop, xlEdgeRight, _
            xlEdgeBottom, xlInsideVertical, xlInsideHorizontal)
        With ActiveCell.Range("A1:C50").Borders(xlEdgeType)
            .LineStyle = xlContinuous
            .ThemeColor = 5
            .TintAndShade = 0
            .Weight = xlThin
        End With
    Next xlEdgeType
End Sub

a. user say:

There is a BorderAround method to apply borders  to the cell periphery.
There were problems using that in the XL2010 beta and I have avoided it since.

Note:  ActiveCell.Range("A1:C50") is not the same as ActiveSheet.Range("A1:C50")
'---
Sub LoopThruBorders()
Dim N As Long
Dim vBorderNdx As Variant

vBorderNdx = Array(xlEdgeLeft, xlEdgeTop, xlEdgeBottom, xlEdgeRight, _
                              xlInsideVertical, xlInsideHorizontal)
For N = LBound(vBorderNdx) To UBound(vBorderNdx)
  With ActiveSheet.Range("A1:C50").Borders(vBorderNdx(N))
      .LineStyle = xlContinuous
      .ThemeColor = 5
      .Weight = xlThin
  End With
Next
End Sub

You must write -100000 without the comma, as I did.  -100,000 is two parameters, with PMT based on a loan of -100(!) instead -100000.  (But in that case, starting with $100,000, I get a residual of $126,846.49, not merely $32,000.  In other words, the loan has a negative amortization!)

If that does not explain your problem, post a complete example with cell names, formulas and numbers.  Even better:  upload an example Excel file (redacted) that demonstrates the problem to a file-sharing website, and post the public/share URL in a response here.

It works fine for me, as demonstrated below.

Formulas:

B4: =PMT(B2/12, B3*12, -B1)

H2: =B1

E3: =$B$4

F3: =E3-G3

G3: =H2*$B$2/12

H3: =H2+G3-E3

E27: =SUM(E3:E26)

H27: =H26

Copy E3:H3 into E4:H26

Copy E27 into F27:G27

Format H27 as Scientific

The point of H27 is to demonstrate that H26 might not be exactly zero due to binary arithmetic anomalies.  Remedy:  change the formula in H26 to =ROUND(H25+G26-E26,2).

Likewise, in the real world, the formula in B4 should be =ROUND(PMT(B2/12, B3*12, -B1),2).  In that case, H26 will usually not be exactly zero, even when rounded.  In this example, it is only -$0.07.  But much larger differences are possible.

Also in the real world, lenders might round interest calculations in any of various ways, but usually with more decimal places than just 2 (cents).  So do not be surprised if your calculations do not match a lender's amortization schedule exactly.

david a matson say:

Greetings everyone:

I'm sorry that you all are experiencing this problem.  It is actually due to WMDC, not Outlook.    You can get the official word on this problem by reading about  KB980513. 

Outlook 64bit actually works as it should, but certain chunks of 64bit code in WMDC were not programmed properly.   Sadly, Microsoft admits that while it does acknowledge this problem, it has not intention of fixing this particular problem.   Based on info that I have been able to research, it seems that any fix they provide would only serve to break other aspects of WMDC, so it's a no-win situation for Microsoft.   However, I should state for the record that the previous sentence is just rumor, I have no evidence supported by fact.

One of the key sentences in that article says "Windows Mobile Device Center syncs correctly with a 32-bit version of Outlook 2010 that  uses a standard MSI installation."  

This installation file-type should located be on your CD/DVD.   However, if you lack this type of installer on the CD/DVD, you can download the  Office Trial -which in my experience, is usually provided in MSI Installer format.   You can then use the Serial Key from your CD/DVD to register Office.

I hope this helps!
