To the "Higher-Ups" who can influence Windows Phone 8

There is no denying that Windows Phone 8 has huge potential and the industry is crying out for a viable third party. All the other companies trying to come up with a viable mobile OS should just stop trying now. Even desktop Linux OS's such as Ubuntu and Red Hat should worry that Android will take over and become the only Linux client available. My personal take on the strategy of Windows Phone is that it has positioned itself in between Android and Apple. Where Apple has great optimisation due to having control over both Hardware and Software, its proprietary take on the phone ecosystem provides lack of choice in features and forces users to be locked in a walled garden environment, which I, for one, am glad to not be part of. Android on the other hand gives users an abundance of choice in handsets, from processor type to screen sizes and resolutions. But the lack of control over the hardware means fragmentation is a SERIOUS issue, with programming for the OS a horrible experience.

 

And that's where Windows Phone comes in, concentrating on the optimality of performance by providing restrictions on screen resolutions and processors but also giving OEMs a license of  creativity to design the size of the handset to whatever specifications they want. However, if you were to imagine a scale in between Apple and Android, Microsoft would definitely lie closer to Apple's ecosystem than Android's and this, unfortunately, is Windows Phone's biggest issue. This post will therefore address what those issues are:

 

1) File Explorer

For a company that has built 2 decade's worth of software on the fundamental notion of the File Explorer, it is ABSOLUTELY UNBELIEVABLE this it is not included on Windows Phone (as of a matter of fact, nor in Windows 8's modern UI [formerly known as Metro]). Microsoft, if you do read this, please note that THERE IS NOTHING more UBIQUITOUS than the File Explorer and hence is one of the main reasons people switch from using iOS products to Android products. The File Explorer gives the smart phone, the heart of a PC, and its absence thereof is the reason WHY many have not bought an iPad. Having a digital device without a File Explorer, especially a Microsoft one, is unforgivable and should be corrected immediately.

 

2) Network File Sharing

Homegroups and Workgroups are an integral part of any person's personal or professional network system. At the heart of the Windows experience since Vista, Microsoft have missed a trick by not incorporating this feature natively in Windows Phone 8. Android's Es File Explorer is one of the most downloaded apps and rightly so. For a device that is always on, I cannot see a better experience than lounging in your bed with your phone, and being able to play a video that is on the shared folder of you home PC without having to transfer it on via USB or installing any ridiculous software on either my PC or my phone. Such a tool would trump Android as I'm sure Microsoft can build a perfectly great file explorer that incorporates network file sharing. Get this done and it will also open up Windows Phone to be built on a phablet. But before that can even happen, WP8 must support higher resolution screens (1080p and above).

 

3) Higher Resolution Screens (1080p and above).

WP8 has a tendency to JUST catch up with the hardware available on Android. Announcing that WP8 can only support 720p is ridiculous when HTC releases their Droid DNA that has a 1080p screen, and with numerous other OEMs releasing 1080p screens on their handsets at CES, WP8 is once again lagging behind. With a little intuition, if WP8 had announced support for 1080p resolution and ABOVE (1440p), it would mean when OEMs are able to step it up with their Hardware, software is already supported. Come on Microsoft, its not difficult, these trends are so plain for everyone to see. Step up your game.

 

4) Processor Chip

Speaking of catching up with hardware, having the dual core S4 Chip instead of the quad core S4 at launch was ridiculous! To be the best, you have to have the best hardware and if OEMs aren't able to launch WP8 phones on the latest hardware but they are doing it with Android, again WP8 will always be catching up. Microsoft should SYNC the release of WP8 phones with the launch of Qualcomm's new chip. As soon as Qualcomm releases a chip, the first phone that has that chip SHOULD BE a Windows Phone, in fact WIndows RT should follow that same principal. Consumers follow that geeks buy, and despite whatever Apple wants people to believe, specs STILL matter. If at launch I could get a WP8 phone, which is the first phone EVER to have a 1440p screen with Quakcinn's S5 chip, regardless of my preference of Android or WP8, I'm getting that phone. There is no competition. Specs matter and for WP to survive, it has GOT TO BE the best.

 

5) Advantages of being in the Windows Ecosystem

This is the main selling point Microsoft is trying to get to the consumer, that if you buy WP8, it'll work perfectly in your ecosystem of Windows and XBox. Well, I bought into the dream but I didn't get the results. If I'm browsing the web on my HTC 8x, I can, via smartglass, push that page onto my Xbox. Great. If I want to continue reading that page, I can push it onto my Surface RT or my Windows PC, again great. Where is this synchronicity I heard about if I joined the family suite of Microsoft products? Well, its not there. Why can't i start angry birds on my phone, continue it on my Xbox, then my PC and then my Surface RT just before I sleep? Its an Xbox live game, is it not?

 

I'm a Microsoft fan and will be til the day I die (assuming the company doesn't go bankrupt), but I cannot be the only person to see that these glaring issues are holding back the love from the geeks, and hence holding back from the acceptance with consumers. As Microsoft thrives to be a viable alternative to OEMs, my advice is, be more like Android and less like Apple.

Linking an Excel document to PowerPoint - Making sure the link works, - Microsoft Community

aurelio74 say:

Linking an Excel document to PowerPoint - Making sure the link works, also if the 2 files are sent to another user, that saves the 2 files in a different location than the original files.

MS Office PowerPoint 2007:
Objective: I want to link a table from an Excel document to a PowerPoint document, in a way that the table in PowerPoint is automatically updated if changes are made in the Excel file.
Identified solution: I "copy" the table in Excel, and make a "Paste special" / "Paste link" / "As: Office Excel file". The 2 documents (PowerPoint and Excel files) are in the same folder.
Issue: The link works. However if I want to send the 2 documents to another person, and the 2 files are again saved in a same folder, PowerPoint tries to updated the table from the "old location" of the Excel file. The location is not automatically updated, as I believe it was the case in the previous version of Office.
How can this be solved? How can the link be defined, not related to the exact location of the original file, but only related to the name of the Excel document - to be saved in a same folder as the PowerPoint file?
Thanks in advance for your help!
steve rindsberg say:

This used to work exactly as you want it to;  PowerPoint would look for the linked XLS file in the folder you originally linked from.  If it couldn't find it there, it'd look in the folder the PPT file's in.   It doesn't do that any longer.  MS broke it. 

The recipient will need to

Manually relink the file or

Run a bit of VBA code that lets them search for the original path and replace it with the new path (available on PPT FAQ; let me know if you'd like the link, or

Use a utility program or add-in that will fix the links automatically (I sell one called PPTools FixLinks; there's a free demo athttp://www.pptools.com/fixlinks/ that lets you repair up to five links in a presentation; the full version has no such limitation)

The BIG problem here:  this puts the burden of fixing the damage on the recipient; I don't think there's anything you can do at your end to fix it for them.  You can "pre-repair" some other kinds of links by removing the path from the file link; then *usually* PPT will look for the file in the current folder.  Unfortunately, with OLE links (e.g. your links to Excel content, other stuff pasted from one app to another), Windows requires a full path; it won't accept relative paths, so we're stuck.

Ah. Wait.  Another thought:  if you and the recipient agree on a drive and folder name, say C:\LinkedExcelFiles\, you might have a way around the problem.  You create that folder on your system and put the XLS files there.  They create it at their end and put the XLS files you send them there (shouldn't matter where the PPTs go, but may as well put them in the same place).  The links should work then.

I'd avoid using network drives for this.  Some releases of PPT 2007 have bugs following links to network drives, and if the drive letter is mapped to a UNC path ( ie,\\servername\sharename\) you're liable to get that in the link rather than the plain drive letter.
