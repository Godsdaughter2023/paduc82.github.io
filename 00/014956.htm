ken sheridan say: If you combine the data from all three Excel worksheets, and add a SchoolYear column to distinguish between the rows for each year, then you will be able to base a form on that table and enter data.  However, a spreadsheet stores data in a very different way to that in which a relational database operates.  In Excel you might well have multiple worksheets in a workbook, one worksheet per year.  In a relational database storing the data from all three worksheets in a single table, while usable, would involve a huge amount of duplication of data, e.g. student names and other personal data.  In designing a relational database one of the fundamental principles is that such redundancy is eliminated.  It's not only wasteful, but more importantly leaves the database open to the insertion of bad data.  I once found three versions of my name as the author of papers in my own field of work in a badly designed database.  As far as the database was concerned there were three separate authors, each responsible for a district subset of the papers.

The process by which such redundancy is eliminated is known as 'normalization'.  This is a formal process with a set of 'normal forms' each of which has a formal definition which determines if a table is normalized to that normal form.  Correct normalization is achieved by the decomposition of a table into a set of related tables.  Each normalized table models an entity type, which can be a real world entity type such as students, school years or grades or can be a relationship type between two, or more entity types, e.g. the many-to-many relationship type between students and grades which would be modelled by a table like this:

StudentGrades
….StudentID  (FK)
….Grade  (FK)

The primary key of this table is a composite of the two foreign keys, each of which references the primary key of the Students and Grades tables.  The many-to-many relationship type has thus been resolved into two one-to-many relationship types.  Other tables might reference the StudentGrades table with composite foreign keys of the same two columns.  By enforcing referential integrity in the relationship, no rows can be entered into those table with an invalid grade for a student.

Once data has been imported into a single non-normalized 'master' table its decomposition into a set of normalized related tables can be automated by the execution of a series of 'append' queries in a specific order.  With the example above for instance, rows would have to be inserted into the Students and Grades tables before rows can be inserted into the StudentGrades table.  You'll find an example of the process in DecomposerDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file takes you through the import and decomposition of the data step by step, with a brief explanation at each stage.  Before looking at this file, however, you might like to familiarise yourself with some of the basic principles and methodologies of relation database design by taking a look at a couple of other files in the same OneDrive folder.  Firstly, DatabaseBasics.zip is a general introduction to the topic, and covers, amongst other things, how the different relationship types are used in the modelling of the data.  The second is Relationships.zip, which builds on the first and illustrates how relationships are built up across a database to achieve the final model.  It uses a simple medical prescriptions database as its example.  If you want to learn about normalization, which I'd advise, then Normalization.zip in the same folder provides as simple as possible, but no more so, introduction to the topic.  Concentrate on normalization up to Third Normal Form (3NF) initially, and don't worry to much about the formal definitions, whose language uses the technical terms of the database relational model, and can be quite formal.

Please try to new a separate send/receive group for the IMAP account.

4. Click "New..."

13 Select "All Accounts."
14. Click "Edit..."
15. Choose every IMAP account in the left and uncheck "Include the selected account in this group" box.
16. Click "OK."
17. Finally, click "Close".
