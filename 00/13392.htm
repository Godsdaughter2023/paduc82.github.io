I have this same problem, on a Toshiba DX735-D3201 with Atheros AR9002WB-1NG. My solution: Install 10041 clean. Keep a copy of the latest Atheros drivers handy (the ones from atheros.cz, not necessarily what pushes on Windows Update). Watch, however, that Windows doesn't decide it doesn't like that you flipped auto-updates to manual, when you move to a your Microsoft account from a purely local account.

If Win Update pushes the new Atheros drivers, you WILL get an immediate BSOD. So watch those settings carefully!

jllatham say:

Here is a way to deal with this and know that an error took place.  You can do this in any module where you set Application.EnableEvents = False, and maybe also if you use Application.DisplayAlerts = False.  You can even use the same name for the Goto part in each one.

On Error GoTo RecoverFromError

Application.EnableEvents = False

Application.DisplayAlerts = False ' just shown as example doesn't have to be there

... the rest of your code here until it's all done, then

RecoverFromError:

If Err<>0 Then

  MsgBox"Error " & Err.Number & ": " & Err.Description, vbOKOnly, "Error Encountered"

  Err.Clear

End If

On Error Goto 0 ' clears error trapping

Application.EnableEvents = True ' reset this

Application.DisplayAlerts = True ' if needed

End Sub

If you start getting messages saying that an error was encountered, to track it down you can make a comment out of the On Error GoTo RecoverFromError statement and run the code to try and fix it -- of course .EnableEvents/.DisplayEvents will remain off if you set them to False.  You can set them to True again using the Immediate Window.

jllatham say:

More about what you show above:

=======

hi,  thanks.  not sure if there is a way to force it back to Events = True after an error then,  but this is what i have so far:

i think the msgbox below, will come up even if other areas have a on error resume next, etc. for:

  MsgBox "Error " & err.Number

===============

If you use an On Error Resume Next, then it erases any On Error GoTo RecoverFromError - and will continue to execute code even if errors are created until you somehow change what to do if an error does take place:

On Error GoTo 0

will clear all error trapping and let Excel halt and show you an error message and the line of code that generated the error.

But if you use another

On Error GoTo RecoverFromError

statement, then once again it will go to that exit routine if an error does take place.

Remember what I said above: the checkV() routine can only set .EnableEvents = True the way it is written now, it will never toggle it to .EnableEvents=False.

Test this short routine to see how this works:

Sub Test1()

   Dim myX as Integer

   Application.EnableEvents = False

   On Error Resume Next ' Tell Excel to ignore reporting errors to you

   'create a divide by zero error

   myX = 1/0 ' generates an error

   'create some more errors, none of which will halt the program

   myX = 5/0

   myX = .5/0

   If Err <> 0 Then

      Err.Clear

      MsgBox "Error Encountered and IGNORED!"

   End If

'after you click [OK] this code will continue to run

   On Error GoTo RecoverFromError

   'create the same error again

   myX = 1/0

   MsgBox "You Should NEVER See This Message"

RecoverFromError:

   If Err<>0 Then

     MsgBox "Error " & Err.Number & " took place, and code jumped to RecoverFromError"

     Err.Clear

   End If

   On Error GoTo 0 ' clear all error trapping

   Application.EnableEvents = True ' make sure .EnableEvents is set to True

End Sub
