qa guy commelec say:

Participation table to have foreign keys for Programs and Locations.

Participation table --

ParticipationID - primary key - autonumber

VolunteerID - foreign key - number integerProgramID - foreign key - number integerLocationID - foreign key - number integer

StartDate - DateTime

EndDate - DateTimeRemarks - memo

Active - yes/no

scottgem say:

Mimmo is right you need a junction table to model the many to many relationship between volunteers and Programs. You have to have a primary Key value for each table, then add a field in the junction table of the same datatype as the corresponding PK. Name it the same as the PK so you can easily see the relationships.

Once you have the table setup, you can create a form with 2 comboboxes to select the Volunteer and the Program. 

ken sheridan say: The following is the model for my StudentCourses demo and illustrates a basic many-to-many relationship type.  think of students as analogous to your volunteers and courses as analogous t your programmes:



You'll find the file as StudentCourses.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

The main purpose of this little demo file is to illustrate various interfaces for data entry, but I'd recommend the conventional form/subform approach.  In my case the StudentCourses table has only the two foreign key columns, which together make up the composite primary key of the table, but a table like this can have other non-key columns representing attributes of the relationship.  For instance it might have StartDate and EndDate columns for when a student starts and ends a course.  A table modelling the relationship type between volunteers and programmes in your case would be likely to have similar non-key columns in addition to the keys.  The subform would then have additional controls bound to those columns in addition to a combo box for selecting a programme.

While my demo shows courses per student, which would equate to programmes per volunteer in your case, you could of course also have a programmes form with a volunteers subform showing volunteers per programme.  The ParentActivities demo in my same OneDrive folder includes both ways for entering data in this way.
