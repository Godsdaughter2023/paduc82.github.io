doug robbins - mvp office apps & services (word) say:

Of course, if you modify the formatting of the font of the Footnote Reference style to remove the superscript, it will be removed from both places where the footnote reference appears (in the body of the document and in the footnote itself.

You can use a macro containing the following code to add a tab before the footnote text and that will move the beginning of the text to the tab stop,

Dim afn As Footnote
    For Each afn In ActiveDocument.Footnotes
        With afn.Range
            .Text = vbTab & .Text
        End With
    Next afn

However, if you use

Dim afn As Footnote
    For Each afn In ActiveDocument.Footnotes
        With afn.Range
            .Text = "." & vbTab & .Text
        End With
    Next afn

You will get a space between the footnote reference and the footnote reference and the period.

If you are dealing with Endnotes, you could use a macro such as the following to convert the endnotes

' Macro created 29/09/99 by Doug Robbins to replace endnotes with textnotes at end of document
' to replace the endnote reference in the body of the document with a superscript number.
'
    Dim aendnote As Endnote
    For Each aendnote In ActiveDocument.Endnotes
        ActiveDocument.Range.InsertAfter vbCr & aendnote.Index & "." & vbTab & aendnote.Range
        aendnote.Reference.InsertBefore "a" & aendnote.Index & "a"
    Next aendnote
    For Each aendnote In ActiveDocument.Endnotes
        aendnote.Reference.Delete
    Next aendnote
    Selection.Find.ClearFormatting
    Selection.Find.Replacement.ClearFormatting
    With Selection.Find.Replacement.Font
        .Superscript = True
    End With
    With Selection.Find
        .Text = "(a)([0-9]{1,})(a)"
        .Replacement.Text = "\2"
        .Forward = True
        .Wrap = wdFindContinue
        .Format = True
        .MatchWildcards = True
    End With
    Selection.Find.Execute Replace:=wdReplaceAll

Exporting Fields from MS Word Template - Microsoft Community

andrewparris say:

Exporting Fields from MS Word Template

I am creating a template in MS Word 2010. I am using the "Developer" functionality to create fields (text, pull-down, date) for people to fill in the template. Is it possible to easily/automatically export all of the values (that users of the template have entered) in the fields, say, to Excel? (Apart from copy and paste.)

macropod say:

The following Excel macro extracts data from all formfields & content controls in all Word documents in the selected folder and populates the first available row in the active Excel worksheet with the data for each document.

Sub GetFormData()
'Note: this code requires a reference to the Word object model.
'See under the VBE's Tools|References.
Application.ScreenUpdating = False
Dim wdApp As New Word.Application, wdDoc As Word.Document
Dim FmFld As Word.FormField, CCtrl As Word.ContentControl
Dim strFolder As String, strFile As String
Dim WkSht As Worksheet, i As Long, j As Long
strFolder = GetFolder
If strFolder = "" Then Exit Sub
Set WkSht = ActiveSheet
i = WkSht.Cells(WkSht.Rows.Count, 1).End(xlUp).Row
strFile = Dir(strFolder & "\*.doc", vbNormal)
While strFile <> ""
  i = i + 1
  Set wdDoc = wdApp.Documents.Open(FileName:=strFolder & "\" & strFile, AddToRecentFiles:=False, Visible:=False)
  With wdDoc
    j = 0
    For Each FmFld In .FormFields
      j = j + 1
      WkSht.Cells(i, j) = FmFld.Result
    Next
    For Each CCtrl In .ContentControls
      j = j + 1
      WkSht.Cells(i, j) = CCtrl.Range.Text
    Next
  End With
  wdDoc.Close SaveChanges:=False
  strFile = Dir()
Wend
wdApp.Quit
Set wdDoc = Nothing: Set wdApp = Nothing: Set WkSht = Nothing
Application.ScreenUpdating = True
End Sub

Function GetFolder() As String
    Dim oFolder As Object
    GetFolder = ""
    Set oFolder = CreateObject("Shell.Application").BrowseForFolder(0, "Choose a folder", 0)
    If (Not oFolder Is Nothing) Then GetFolder = oFolder.Items.Item.Path
    Set oFolder = Nothing
End Function

You can comment-out/delete the loop relating to FormFields or ContentControls, as applicable.

Note: As coded, the macro looks for files with the .doc extension (which can't contain content controls) as well as files with the docx & docm extensions. To work with docx files only, for example, change:

strFile = Dir(strFolder & "\*.doc", vbNormal)

to:

strFile = Dir(strFolder & "\*.docx", vbNormal)

ken sheridan say: Scott is absolutely right to point to the fact that you appear to have a one-to-one relationship type between plans and sub_plans.  A one-to-one relationship type is used to model a type/sub-type scenario, i.e. where each sub-type shares the attributes of the (super) type, but not those of other sub-types.  That does not appear to be the situation here on the basis of the information you've provided, so having two tables is unnecessary. If you wish to add supplementary data to the current record in the same table via separate form, then the code to open a second form bound to the same table is simply:

    'ensure current record is saved
    Me.Dirty = False
    ' open second form filtered to current primary key value
    DoCmd.OpenForm "YourSecondFormName", WhereCondition:="Plan_ID = " & Me.Plan_ID

If Plan_ID is a text data type, not a number data type, it would be:

    DoCmd.OpenForm "YourSecondFormName", WhereCondition:="Plan_ID = """ & Me.Plan_ID & """"

The above could easily done with a macro which carries out the same actions.

However, with current two-table set-up the code for a button's Click event procedure would be:

    Dim strCriteria As String
    
    strCriteria = "Plan_ID = " & Me.Plan_ID
    ' alternatively:
    ' strCriteria = "Plan_ID = """ & Me.Plan_ID & """"
    ' if Plan_ID is a text data type

     If Not IsNull(DLookup("Plan_ID", "Sub_Plan", strCriteria)) Then
         DoCmd.OpenForm "YourSubPlanFormName", _
              WhereCondition:=strCriteria, _
              WindowMode:=acDialog
    Else
         DoCmd.OpenForm "YourSubPlanFormName", _
              DataMode:=acFormAdd, _
              WindowMode:=acDialog, _
              OpenArgs:=Me.PlanID
    End If

In the Open event procedure of your sub-plan form put:

    If Not IsNull(Me.OpenArgs) Then
        Me.Plan_ID.DefaultValue = """" & Me.OpenArgs & """"
    End If

Note that (a) Plan_ID must be a bound control in your sub-plan form, but can be hidden if desired by setting its Visible property to False (No); (b) setting the DefaultValue property of a control does not initiate a new record.  That only happens when the user begins to enter data into other non-key controls in the form.  This is what you'd want of course as saving a row with only a value in the Plan_ID column would be meaningless, and would prejudice the integrity of the database by forcing you to allow Nulls in all other non-key columns; (c) the DefaultValue property is always a string expression regardless of the data type of the column in question, so is wrapped in literal quotes characters in the above.

I don't think it would be possible to do the above with a macro in Access 2007.  The extended macro language in the current version of Access, primarily for use in 'web' databases, might be able to do it, but never having occasion to use macros I can't say for certain.

[Client] table

[Plan] table attached to [Client] table via [Client ID] field. Each [Client] record can have multiple [Plan] records.

[AFS] table attached to [Plan] table via [Plan ID] field. This table stores the answers to a series of questions related to the plan's contract. Its purpose is to be used to generate a Word Merge of the contract at any time. We could have multiple entries if they have more than one contract (we keep track of the historical contract changes by entering a new record if the terms change and mark the old record as "inactive"). Many fields on this table pull their list of possible answers from one of the "Answers" tables (see below)

[Addendum] table attached to the [AFS] table via [AFS ID]. This contains data necessary to merge a custom language addendum to a contract that is abnormal in some way. Can have multiple [Addendum] records for each [AFS]. Also relies upon the "Answers" tables for many of the fields.

[Fee Disclosure] table attached to the [AFS] table via [AFS ID]. This stores the information that needs to go onto our fee disclosure documents. All Y/N questions at the moment.

[Compliance Report] table attached to [Plan] table via [Plan ID] field. Every [Plan] record only needs one of these records. This stores information that is used to merge an annual report. Many fields rely on the "Answers" tables for the answers, though there are also many text fields and number fields that can change each year. This table has the most fields (so far) which could potentially be further categorized into sub tables if needed.

[AFTAP] table attached to [Plan] table via [Plan ID] field. Every plan should only have one of these records. Basically, it functions the same way as the [Compliance Report] table but is used for a different annual document.

[PPA Statement] table attached to [Plan] table via [Plan ID] field. Every plan should only have one of these as well. It also functions the same way as the [Compliance Report] and [AFTAP], though it does not change very frequently (only when major revisions are made to the client's plans).

"Answers" tables:

I have a few additional tables that hold the potential choices for various fields throughout the database. I tried to use Row Source text for answer types that do not have overlap across fields in order to avoid having a lot of unneeded tables.
