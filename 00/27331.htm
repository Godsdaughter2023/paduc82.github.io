scottgem say:

We have been converting a lot of apps to 64 bit usage. And I ran into the same problem with fOSUsername. That is a very common piece of code that has been around for ages. It is used to determine the name of the user logged into that PC. 

I decided to forget about using fOSusername() anymore. So what I did was comment out all the code within that function and replace it with:

fOSUsername=Environ("Username")

So the function no longer uses an API call, but the built in Environ function. While the Environ function is more easily spoofed, its only a problem with a determined hacker.

scottgem say:

Yes, For an accdb to run in a 64 bit version you have to update All Your DECLARE functions. The error message is telling you how. 

So wherever you have DECLARE FUNCTION you need to change that to DECLARE PTRSAFE FUNCTION. The same with SUBs. Also any parameters declared as Long need to be changed to LongPtr. 

That code you have is a common API call. I use it too. Here is my 64 bit version:

Private Declare PtrSafe Function apiGetUserName Lib "advapi32.dll" Alias _
    "GetUserNameA" (ByVal lpBuffer As String, nSize As LongPtr) As Long

Notice the nSize declaration, but you don't change the Function declaration. 

Making those changes will allow the accdb to be run in either 32 or 64 bit. It won't, however, allow an app compiled in 32 bit to run under 64 bit.

For more details on this just do a search on PtrSafe.

The suggestions made above to correct your problem did not work for me.

Open outlook in /safe mode using the run command:

Select Send/Receive

Click once on Work Offline icon and watch the bottom of your screen.  It should revert from offline to online.

Note, the red X on Work Offline is part of the icon and does NOT disappear when you flip between online and offline modes.

Attempting to switch between online and offline mode when NOT in safe mode may work, but often it does not.  Safe mode will prevent any add-ons from interfering with the normal operation of Outlook.

Putting outlook in safe mode is also recommended when attempting to disable or enable add-ons, as sometimes they are not activated or de-activated correctly when in regular mode.
