ken sheridan say: This is not as simple as you might think.  Firstly you must get out of a spreadsheet mindset.  While a relational database table's datasheet has a superficial resemblance to a spreadsheet, that's where the resemblance ends.  In a relational database a table is a set, and by definition a set has no intrinsic order, so concepts like 'above' or 'below' are meaningless.  Tables do not have cells; they have values at column positions in rows.  Data in a table is addressed solely by value, not by position.

One value in relation to another value is dependent on a sort order, e.g. in a table of distinct daily transactions a row for a transaction with today's date would sort after one with yesterday's date if the data is sorted by date, but could sort before or after the same row if the data is sorted by the transaction amount.  There is no intrinsic relative position of one row vis-Ã -vis the other.

To do the sort of thing you are attempting requires that you identify the two values on which the percentage difference is to be computed on the basis of one being the value in the row with the latest date prior to the date in the other row.  To do this in a query is not trivial. Perhaps I can illustrate it using a simple table of transactions in which each transaction date is distinct, but not necessary sequential:

SELECT T1.TransactionDate,T1.TransactionAmount,
(T1.TransactionAmount -
   (SELECT MAX(T2.TransactionAmount)
     FROM Transactions AS T2
     WHERE T2.TransactionDate =
        (SELECT MAX(TransactionDate)
         FROM Transactions AS T3
         WHERE T3.TransactionDate < T1.TransactionDate)))/
    (SELECT MAX(T2.TransactionAmount)
     FROM Transactions AS T2
     WHERE T2.TransactionDate =
        (SELECT MAX(TransactionDate)
         FROM Transactions AS T3
         WHERE T3.TransactionDate < T1.TransactionDate))*100
AS PercentIncrease
FROM Transactions AS T1
ORDER BY T1.TransactionDate;

In this query the outer query returns the transactions ordered by date.  The percentage difference between the value of one transaction and the next in date order is computed by using a subquery to return the amount where the transaction date is the date returned by a further nested subquery which returns the latest (MAX) date prior to the outer query's current date.  The use of the MAX operator against the TransactionAmount in the first subquery is here used not to return the highest amount, but simply to force this subquery to return only one value, so any aggregation operator could in fact be used in this case.

The percentage difference is computed by subtracting the earlier amount returned by the subquery from the current amount, dividing the result by the earlier amount returned by the subquery (which is why he subqueries are repeated) and multiplying by 100 to return the value as a percentage.

While this can be done in a query as above, there is a far simpler solution, which is to compute the percentage difference in a report.  The report would be a simple one of transactions ordered by date, and in the Format event procedure of the detail section the percentage would be computed on the basis of the current transaction amount and the value of a module level variable which stores the previous transaction amount.  The result of the computation would be assigned to an unbound text box control in the detailed section, and the value of the variable would then be set to the current transaction amount ready for it to be used in the next row of the report.

ken sheridan say: Once you've created and tested a suitable query you simply have to create a report with the query as its RecordSource property.  The report wizard can be used to do this.

If you want to do the computation in the report rather than a query then you'd firstly declare a variable in the report's module's Declarations area:

    Dim PreviousMaxOfGrade As Currency

Then in the Format event procedure of the detail section you first assign a value to an unbound text box,  txtPercentIncrease say, with code like this, leaving the control in the report's first row as Null by examining the value of the PreviousMaxOfGrade variable for a non-zero value, and then assign the current value of the Max of Grade column to the variable ready for the computation in the next row:

    If PreviousMaxOfGrade <> 0 Then
        Me.txtPercentIncrease = (Me.[Max of Grade] - PreviousMaxOfGrade)/PreviousMaxOfGrade
    End If

    PreviousMaxOfGrade = Me.[Max of Grade]

Format the txtPercentIncrease control as Percent to show the fractional value as a percentage.

Doing the computation in the report is not only simpler, but has the advantage that the dates of the rows need not be distinct.  It's unlikely with salary increments of course that you'd have two rows with the same date, but with other types of transactions this might be the case.

The report can then be a simple one based on a query which merely returns the rows from the table with no need for the complexities of the subqueries needed if the computation is done in the query.  If the report is grouped by employee say, then re-initialise the variable to zero in the Format event procedure of the employee group header with:

    PreviousMaxOfGrade = 0

ken sheridan say: You are absolutely right of course, Marshall.  I'd lost sight of the fact that the variable would need to be incremented only in the last incidence of the event.  The RunningSum property doesn't help in this context, though, as this is not a case of aggregation.  It's necessary to know the previous row's value as the expression to return a percentage increase is (y-x)/y*100 where y is the previous value and x the later value.

The query I posted will work in the current context on the assumption that there can be no more than one transaction in one day.  In the context of this post the subqueries would of course need to be correlated on the employee key if data for multiple employees is being returned.  In cases where multiple transactions can occur in a single day, unless distinct date/time values per transaction are recorded (desirable but rarely done in my experience), a candidate key would need to be brought into play as a tie breaker.  This is not difficult when computing balances by means of a join of two instances of a table, or even if calling the DSum function for an updatable query, but would make the query I posted even more complex.  I think the simplest approach would be to firstly compute the row numbers per group by a join, by customer in the following example from one of my on-line demos:

SELECT COUNT(*) AS RowNumber, T1.CustomerID, T1.TransactionDate, T1.TransactionAmount
FROM Transactions AS T1 INNER JOIN Transactions AS T2
ON (T2.TransactionID<=T1.TransactionID OR T2.TransactionDate<>T1.TransactionDate)
AND (T2.TransactionDate<=T1.TransactionDate) AND (T2.CustomerID=T1.CustomerID)
GROUP BY T1.CustomerID, T1.TransactionDate, T1.TransactionAmount, T1.TransactionID;

Save this as Q1 say, then create another query on it:

SELECT Q1A.*,
 (Q1A.TransactionAmount-
   (SELECT TransactionAmount
    FROM Q1 AS Q1B
    WHERE Q1B.CustomerID = Q1A.CustomerID
    AND Q1B.RowNumber = Q1A.RowNumber-1))/
   (SELECT TransactionAmount
    FROM Q1 AS Q1B
    WHERE Q1B.CustomerID = Q1A.CustomerID
    AND Q1B.RowNumber = Q1A.RowNumber-1)*100 As PercentIncrease
FROM Q1 AS Q1A;

ken sheridan say:

The following code in a form's Open event procedure will fill an unbound list box with a list of report descriptions, i.e the Description property of a report object as added via the database window.  Reports with no description property are omitted. 

Private Sub Form_Open(Cancel As Integer)

    On Error GoTo Err_Handler

    Const NODESCRIPTION = 3270
    Dim ctrl As Control
    Dim dbs As DAO.Database
    Dim ctr As DAO.Container
    Dim doc As DAO.Document
    Dim strReportname As String
    Dim strDescription As String
    
    Set dbs = CurrentDb
    Set ctr = dbs.Containers("Reports")
    Set ctrl = Me.lstReports
    
    For Each doc In ctr.Documents
        strReportname = doc.Name
        On Error Resume Next
        strDescription = doc.Properties("Description")
        Select Case Err.Number
            Case 0
            ' no error
            Case NODESCRIPTION
            ' report has no description
            ' so ignore and go to next report
            GoTo EndLoop
            Case Else
            ' unknown error
            GoTo Err_Handler
        End Select
        Err.Clear
        On Error GoTo Err_Handler
        ctrl.AddItem strReportname & ";" & strDescription
EndLoop:
    Next doc
   
Exit_Here:
    Exit Sub
   
Err_Handler:
    MsgBox Err.Number, vbExclamation, "Error"
    Resume Exit_Here
   
End Sub

As it stands the code lists all reports with descriptions, but you could, by using suitable naming conventions for reports, limit the list to particular reports, e.g. to list only reports whose names end with "_Sales":

        If Right(strReportname,6) = "_Sales" Then

several possible reasons and solutions:

  • The spreadsheet shows formulas instead of values -- use Ctrl -` (that is the key to the left of the 1 on a US keyboard) to toggle between formulas and values
