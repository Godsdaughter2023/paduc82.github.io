ken sheridan say: Restricting a form's recordset means exactly what is says on the tin.  In your expression [forms]![frmSearchProducts]![txtProductName] is a parameter.

You need no code whatsoever in the form's Load event.  The only code you need is the line:

    Me.Requery

which in your case would be in the AfterUpdate event procedure of the unbound text box.  As you want to restrict the recordset on any of three columns the query's WHERE clause would be like this:

WHERE [Column1] LIKE "*" &  [forms]![frmSearchProducts]![txtProductName] & "*"
OR [Column2] LIKE "*" &  [forms]![frmSearchProducts]![txtProductName] & "*"
OR [Column3] LIKE "*" &  [forms]![frmSearchProducts]![txtProductName] & "*"

As all three controls will be NULL when the form opens the WHERE clause will evaluate to NULL, so no rows will be returned.  Only when the user enters a value will rows be returned where there is at least a partial match in any of the three columns.

Trying to archive attachments in access 2010 - Microsoft Community

jeremyrich say:

Trying to archive attachments in access 2010

How can I archive my attachments/documents when I archive the records in access 2010? Any help would be greatly appreciated.

albert d. kallal say:

Access will allow you to export a single table to another database.

So you can create a blank database for archiving.

Then re-open your original database, and from the ribbon choose External data tab, and then Access in the Export area.

The above will thus allow you to export the table and this will INCLUDE attachments.

Normally one often will use a append query, but with an attachment column (multi-value), then append query cannot be used. However the above export idea will work, but will always export the whole table.

It is also possible to write VBA code to export those files out to standard windows files outside of the database.

So you have some options, it not clear if you looking to pull out the attachments to a standard windows file folder(s), or you want to export the table out to another database for backup.

And I suppose after you exported the whole table, then you could delete records that are say older then 1 or two years by creating a delete query.

So you have to export the whole table at a time. Thus in place of choosing what rows to export, you export the whole table and  THEN delete the rows you don't  want.

It is also possible to write VBA code to export the data based on some date criteria etc, but simply exporting the whole table as per above can be done without any code. As noted you could then delete rows you don’t want in your production database anymore.
