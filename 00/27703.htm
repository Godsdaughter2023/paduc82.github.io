ken sheridan say:

Firstly, a table can only have one primary key.  I suspect what you mean is that the table has a composite primary key made up of the PatientID and CaseID columns, where the latter is a serial number starting at 1 per patient.

 

There is no reason to save the record before entering further data.  In fact, to do this potentially undermines the integrity of the database as it forces you to either give every other column a default value, or allow every other column to accept a Null.  The former is very unlikely to be appropriate and the latter is certainly undesirable.  I assume the justification for doing this is to prevent conflicts in the event of two or more users inserting a new case for the same patient simultaneously; conflicts can be resolved in other ways, however.

 

You'll find an example of how this sort of model can be handled in CounterDemo.zip in my public databases folder at:

 

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

 

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

 

The appropriate option in the demo is that for Sequential Numbering by Group.  In this form the key of the table is a composite one of NameID and Gender, analogous to your CaseID and PatientID.  In the form the NameID is computed in the AfterUpdate event procedure of the gender control by calling the GetNameID function in the form's module.  This calls the DMax function to get the last number for the gender in question, and increments it by 1.  In the event of a key violation error the form's Error event procedure calls the IncrementField function in the form's module, which increments the value by 1 again.  Should multiple conflicts occur this process repeats until no error is raised.

 

Not that it is the DMax function used to get the next number, not the DCount function.  The latter would fail, as would your query, if for any reason an earlier  patient's case record had been deleted.  You may say that this should never happen, but it cannot be absolutely ruled out unless specifically prevented  in the database design.  However, if that were the case then there would be no need to store the case number as a value in a column at all as it could be computed at any time.

suzanne s. barnhill say:

It sounds as if the template in which you are saving the toolbars is not being saved when you quit Word, and most likely this is the Normal template (both because that's the default save location for toolbars and because it's the one most likely to be not being saved). The cause of this is usually a poorly written add-in, and ultimately it will be best for you to figure out which add-in is causing the problem and then, if you decide you can do with it, to remove it. As a temporary workaround, however, you can open the Normal template itself, make the changes directly in it, and then manually save it.

 

If you know how to locate the Normal.dot template, then you can navigate to it and open it just as you would a document (you will see "Normal.dot - Microsoft Word" on the title bar, not "Document1 - Microsoft Word"). If you have difficulty finding the Normal template, then here's another approach:

 

In Word, press Alt+F11; the Visual Basic Editor opens. Press Ctrl+G to activate the Immediate window. In that window, type

     NormalTemplate.OpenAsDocument

and press Enter. Normal opens, but note that it may not be on top of all windows. You can close the Visual Basic Editor. Make the desired changes to the Normal file, and then save and close it. If the changes you have made have not been recognized as changes (the Undo button is not activated), press Spacebar, then Backspace to "dirty" the file, thus ensuring that the template will actually be saved.

halfadime354 say:

MS Word 2010 headers Different first page, but need to carry some information to all other pages.

I have a multiple page document.  The first page header contains a logo, and a couple of other items that do not need to appear on later pages.  However, I do have information, such as review dates, reviewer, etc. that needs to be on all pages.  Headers on all pages are set up in a table format.  How can I get the information I need to carry over from the first page on other pages without retyping on page 2?  If I let all the information carry over (same header all pages), document length increases significantly.

Thanks

charles kenyon say:

See the StyleRef Field. It is what you want. Create a character style, perhaps "RevDate" for your revision date and use the StyleRef Field  to put that information in your headers and footers.

See also: Useful StyleRef Field Tricks

suzanne s. barnhill say:
To add to what Charles said, note that you can create both primary Header and First Page Header in your template (with StyleRef fields in the primary Header) even though the template is only a single page. Create the Header first (before enabling Different First Page) or add a second page temporarily to create the Header. Even though you don't see the Header in the one-page template, Word will remember it when the document spills over to a second page.
