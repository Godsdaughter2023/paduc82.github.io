PeekMessage does not recognize keyboard when passing Application.Hwnd

BACKGROUND

Windows 8.1 - 64 bit / Excel 2013 - 32 bit. I work for a civil engineering lab. I am an amateur programmer with fairly extensive experience with Excel VBA. I have only limited experience with Windows programing.

PROJECT DESCRIPTION

I developed my own magnifier similar to Microsoft's. It uses Excel as a platform and is intended for use in Excel only. It uses StretchBlt, BitBlt and Peekmessage as the main ingredients. It is intended to let me select an exact point in a photograph and place a green crosshair on this point by left clicking or by pressing the <Enter> key.  An example of the magnifier is provided at my Public SkyDrive folder at this link:   http://1drv.ms/1BPku1X

PROBLEM

The magnifier window has white crosshairs at the centre used to select the point where the green crosshair gets placed.  I want to be able to move the white crosshairs using the arrow keys when the magnifier window is active instead of having to gently move the mouse. Moving the mouse to select the exact point is difficult because the area under the mouse pointer is magnified, and so the effective movements are coarse.

If I pass the Desktop window handle of 0 to the PeekMessage routine then I get the desired functionality with respect to the arrow keys and <Enter> key. In other words, the keyboard is recognized. However, doing this somehow corrupts the workbook. In particular, the green crosshair doesn't get pasted to the selected point until I deactivate the worksheet and reactivate. The worksheet is unresponsive in other respects also. This corrupted state does not correct until I close and reopen the workbook.

If I instead pass the application window handle (XLMain or Application.Hwnd) or one of its children (XLDesk or Excel7) then the corruption doesn't occur but the keyboard isn't recognized. Therefore, I don't have the desired arrow key and <Enter> key functionality.

I believe I have properly prevented memory leaks:  deleted created DC's, released the Desktop DC, and destroyed the magnifier window etc. See the code in the workbook provided.

Very appreciative of help.

Greg Wilson

greg wilson 2 say:

I seem to have answered my own question:

When I create the magnifier window it must be automatically assigned the keyboard focus. So the keyboard messages were sent to it as well as its parent window(s). That is why I could receive keyboard and mouse messages for the Desktop window. For whatever reason, mouse messages are also sent to the application's main window (XLMAIN) as well as child windows such as EXCEL7. So I could detect mouse messages for these windows.

The solution was simply to call the SetFocus API function after creating the magnifier window referencing the main window, i.e. "SetFocus(Application.Hwnd)" so that the keyboard messages are directed to it instead.

Greg Wilson

Edit:  I have applied the fix to the Public SkyDrive version.

PowerPoint 2016 has a new method that uses a simpler format for caption files, called WebVTT. Closed captions are stored in a text-based file with a. vtt filename extension. CC button in PowerPoint 2016 does appear when captions are added successfully.
