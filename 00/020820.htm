Build 16288.1 - Recover a "Won't Boot" Surface Pro 3 - Microsoft Community

jason[msft] say:

Build 16288.1 - Recover a "Won't Boot" Surface Pro 3 Device

Hello Insiders,

Last week we released build 16288.1 to both the Fast and Slow rings.  As data and feedback has come in, we've tracked reports from Surface Pro 3 users that their devices may end up in an "unbootable" state where the Windows OS will not load and it appears to be stuck on a "spinning dots" screen.  Teams have been hard at work to triage these reports and we have uncovered the cause of devices that have arrived in this state.

Who:  Surface Pro 3 devices on builds 16288.1 or 16291

What:  The device may end up in an undesired state where the Windows OS may not load

Why:  One of the required boot files on Surface Pro 3 devices may end up in a corrupted state after an abnormal shutdown

Resolution:  We now have a workaround that Insiders can use to recover their devices.  Please see the steps below to recover your device.

----------------

Workaround Steps:

  • Prepare a bootable USB image from a working PC
    • https://www.microsoft.com/en-us/software-download/windows10
  • With the device powered off, boot your Surface Pro 3 device from the USB drive
    • Boot Surface from a USB device
  • When the Windows setup box appears, press Shift + F10
  • In the Command Prompt window that appears, type:
    • wpeutil reboot
  • Hit enter to execute this command and immediately remove the USB drive from the computer
  • The device will reboot and load the Windows OS as expected

The above steps will allow your computer to boot.  To permanently fix the issue, complete the following additional steps:

  • Open a web browser and navigate to aka.ms/fixsp3ec
  • Download the file:  fixsp3.zip
  • Open File Explorer on your PC and navigate to the Downloads folder
    • Downloads is the default folder, it may be different if you have changed your default location
  • Right-click the file fixsp3.zip and select extract all
  • A window will open after extraction. Right-click the file fixsp3.cmd and choose Run as Administrator
  • The process will execute and implement the necessary patch to prevent this bug from recurring
  • Reboot your Surface Pro 3 device.  It is now fully patched.

We have a permanent fix for this issue in the build pipeline already.  If your device has not hit this bug, it is still possible to in the short term if you have not run the above script.  We will have a permanent fix with no required workaround steps in our next flighted build that is 16294 or greater.

If you have any questions while working through these steps, please let us know.

Thank you as always,
Jason

john w. vinson say:

You'ra misunderstanding the purpose of referential integrity. A relationship between two tables does not automagically create new data in the child table; instead itprevents  the addition of invalid records to the child table.

Relational databases use the "Grandmother's Pantry Principle" - "a place - one place! - for everything, everything in its place". The LastName and FirstName fields should exist only once, in only the Names table. If you need to see the name in the first table in conjunction with data in the second table, then you should either create a Query joining the two tables, or use a Form based on the parent table, with a Subform based on the child table.

What kind of data (other than names) do you have in the two tables? What real-life Entities (people, things or events) does each table represent?

boblarson say:

It would seem that you don't understand the basics of how a database works, so let's go into it a bit.

1. You don't join the primary keys of two tables together.  You would have a the primary key (let's say autonumber) of the first table and then you would have a field (Long Integer) in the second table which is known as the FOREIGN KEY.  This is where the link happends between the two tables.  The first table has its primary key jointed to the foreign key in the second table.

2. Your second table would also have a primary key (autonumber for instance) and that would not be joined to anything.

3. You don't store the same data in both tables.  So, you would not store last name and first name in both tables but only in table 1 and then you store the primary key of the table 1 record in the foreign key spot in the second table.  This allows you to join the data together and get the first name and last name from the first table by means of a QUERY which would have both tables in it and you would select the names from table 1 along with any other data from table 2.

I hope that helps.

suzanne s. barnhill say:

Actually, Ribbon versions of Word make it easier than ever to see just the styles you want to see and where you want to see them. Word 2002 and 2003 permit "custom" settings for the Styles pane that then also apply to the Styles dropdown, but in Word 2007 and above, you can right-click any style in the Styles pane and specify whether it does or does not appear in the Quick Styles gallery (and of course you can right-click any style in the Quick Styles gallery and remove it).

In the Manage Styles dialog, however, you can set the priority and visibility of every single style, so that when you set the Styles pane to Recommended (in the Styles Pane Options), you'll get your custom list, sorted according to priority ("As Recommended"), alphabetically, or some other way.

It requires a little (well, to be honest, a lot of) work to set this up initially, but if you choose "New documents based on this template," you need do it only once for the Normal template and any others you want to customize. The styles to which you assign the highest priority (lowest number) will appear first in the Quick Styles gallery (which is always sorted "As Recommended"); styles with the same priority will be in alphabetical order. By manipulating the priority numbers (and, in the case of custom styles, the style names), you can assure that the styles you use most often are on the first row of the Quick Styles gallery. Choose a few styles to be high-priority, set the rest to "Hide until used," and you may get what you want. I say "may" because it has been observed that some styles don't seem to hide until they're used, and some may not show up even after they're used. The reason for the first situation might well be that, in the Style Pane Options, the box for "Show next heading when previous level is used" is checked; so when you use Heading 3, Heading 4 will appear. This is actually a convenience, and I keep that box checked.

You can also add the classic Styles dropdown to the Quick Access Toolbar. It mirrors the display of the Styles pane (both the selection and the sort order), with several differences (aside from being always visible). It displays the styles with their formatting (there's no option, as in the Styles pane, with the Show Preview check box, to display all the styles in plain text). When All Styles is selected, it displays a long list of table styles (which don't appear in the Styles pane). And, even if the Styles pane is displaying a subset of styles, you can force the Styles dropdown to show all styles by pressing Shift while clicking the arrow.

Here is my two cents from a VBA perspective.  Personally I rarely use Word for more than a casual letter or make a To Do list so styles are not something that important to me.  However, when I do need to use them, I too feel the frustration surrounding their classification.

If you open a new document and run the code provided at the end of this post, you may be enlightened a bit about the meaning of "In current document" and "In Use" related to the options in the Style Pane. 

If you are unfamiliar with macros see See: http://gregmaxey.mvps.org/word_tip_pages/installing_employing_macros.html

for instructions to employ the VBA code provided above

In the VBA world "In Use" is really more like "In Current Document" and from the Style Pane perspective "In Current Document" means basically the same thing as "In Use" to VBA.

From the Style Pane perspective "In Current Document" is ALMOST the same thing as "In Use" to VBA.  I say almost because there can be built-in styles InUse and In the current document that are never listed in the Style Pane e.g., the No List and Table Normal styles.

I would summarize these observations as:


In current document means:  Most built-in styles that have been modified or applied in the document template or a new style that has been created in the template\document."

In Use means:  A styles that is physically applied to an existing paragraph or text in the document.

Note:  The macro may take several seconds to run so be patient.

Sub StyleQuery()
Dim oStyle As Style
Dim oDoc As Document
  Set oDoc = Documents.Add
  oDoc.Range.Select
  Selection.ParagraphFormat.TabStops.ClearAll
  Selection.ParagraphFormat.TabStops.Add Position:=InchesToPoints(3), _
    Alignment:=wdAlignTabLeft, Leader:=wdTabLeaderSpaces
  Selection.Collapse wdCollapseStart
  Application.ScreenUpdating = False
  oDoc.Range.Text = "These are all of the styles built-in and user defined associated with the document templAte:" & vbCr + vbCr
  For Each oStyle In ThisDocument.Styles
    If oStyle.BuiltIn = True Then
      oDoc.Range.InsertAfter oStyle.NameLocal & vbTab & " Built-in Style" & vbCr
    Else
      oDoc.Range.InsertAfter oStyle.NameLocal & vbTab & " User Defined Style" & vbCr
    End If
  Next oStyle
  oDoc.Range.InsertAfter vbCr + "These are the styles ""InUse"" in VBA speak.  This speak says: " & vbCr + vbCr _
  & "A style is InUse if it is a built-in style that has been modified or applied in the document or a new style that has been created in the document." & vbCr _
  & "The InUse property doesn't necessarily indicate whether the style is currently applied to any text in the document. For instance, if text that's been formatted with a style is deleted, the InUse property of the style remains True. For built-in styles that have never been used in the document, this property returns False." & vbCr & vbCr _
  & "This sounds more like what you actually see when you select ""In current document"" as the option for the Style Pane." & vbCr + vbCr
  For Each oStyle In ThisDocument.Styles
    If oStyle.InUse Then
      If oStyle.BuiltIn = True Then
        oDoc.Range.InsertAfter oStyle.NameLocal & vbTab & " Built-in Style" & vbCr
      Else
        oDoc.Range.InsertAfter oStyle.NameLocal & vbTab & " User Defined Style" & vbCr
      End If
    End If
  Next oStyle
  oDoc.Range.InsertAfter vbCr + "These are the styles that are physically applied to one or more paragraphs of text in the active document." & vbCr + vbCr _
  & "In my mind that is what ""In Use"" means and what you see when you select ""In Use"" as the option for the Style Pane:" & vbCr + vbCr
  For Each oStyle In ThisDocument.Styles
    If oStyle.InUse Then
      If FindAnywhere(oStyle.NameLocal) Then
        oDoc.Range.InsertAfter oStyle.NameLocal & vbCr
      End If
    End If
  Next oStyle
  Application.ScreenUpdating = True
lbl_Exit:
  Exit Sub
End Sub

Public Function FindAnywhere(strStyle As String) As Boolean
Dim rngStory As Word.Range
Dim lngJunk As Long
Dim oShp As Shape
  FindAnywhere = False
  'Fix the skipped blank Header/Footer problem
  lngJunk = ActiveDocument.Sections(1).Headers(1).Range.StoryType
  ResetFRParameters
  'Iterate through all story types in the current document
  For Each rngStory In ActiveDocument.StoryRanges
    'Iterate through all linked stories
    Do
      If SrcInStory(rngStory, strStyle) = True Then
        FindAnywhere = True
        Exit For
      End If
      On Error Resume Next
      Select Case rngStory.StoryType
        Case 6, 7, 8, 9, 10, 11
          If rngStory.ShapeRange.Count > 0 Then
            For Each oShp In rngStory.ShapeRange
              If oShp.TextFrame.HasText Then
                If SrcInStory(oShp.TextFrame.TextRange, strStyle) Then
                  FindAnywhere = True
                  Exit For
                End If
              End If
            Next
          End If
        Case Else
          'Do Nothing
      End Select
      On Error GoTo 0
      'Get next linked story (if any)
      Set rngStory = rngStory.NextStoryRange
    Loop Until rngStory Is Nothing
  Next
lbl_Exit:
  Exit Function
End Function
Public Function SrcInStory(ByVal rngStory As Word.Range, strStyle As String) As Boolean
  SrcInStory = False
  With rngStory.Find
    .ClearFormatting
    .Replacement.ClearFormatting
    .Style = strStyle
    If .Execute Then SrcInStory = True
  End With
lbl_Exit:
  Exit Function
End Function

  .Replacement.Text = ""
  .Forward = True

  .Format = False
  .MatchCase = False
  .MatchWholeWord = False
  .MatchWildcards = False
  .MatchSoundsLike = False
  .MatchAllWordForms = False
  .Execute
End With
End Sub
