This little demo file illustrates a method for duplicating a tree structured hierarchy of three levels (rather than two as in your case) of country, regions and cities.  This is of course an absurd example, and deliberately so, but it serves to illustrate the methodology.  The function which copies the data is:

Public Function CopyCountryTree(lngCountryID As Long)

    Dim strSQL As String
    Dim lngNewCountryID As Long
    Dim lngNewRegionID As Long
    Dim lngNewCityID As Long
    Dim rstRegions As DAO.Recordset
    
    ' get next CountryID value
    lngNewCountryID = DMax("CountryID", "Countries") + 1
    
    'insert new row into Countries
    strSQL = "INSERT INTO Countries(CountryID,Country) " & _
        "SELECT " & lngNewCountryID & ",Country " & _
        "FROM Countries WHERE CountryID = " & lngCountryID
    CurrentDb.Execute strSQL, dbFailOnError
    
    ' get Max RegionID value
    lngNewRegionID = DMax("RegionID", "Regions")
    
    strSQL = "SELECT Region FROM Regions WHERE CountryID = " & lngCountryID
    Set rstRegions = CurrentDb.OpenRecordset(strSQL)
    
    ' loop through set of Region values in country being copied
    ' and insert a new row into Regions, incementing value of RegionID by 1
    ' at each iteration of the loop
    With rstRegions
        .MoveLast
        .MoveFirst
        Do While Not .EOF
            lngNewRegionID = lngNewRegionID + 1
            strSQL = "INSERT INTO Regions(RegionID,Region,CountryID) " & _
                "VALUES(" & lngNewRegionID & ",""" & .Fields("Region") & """," & lngNewCountryID & ")"
            CurrentDb.Execute strSQL, dbFailOnError
           .MoveNext
        Loop
    End With
    
     ' get Max CityID value
    lngNewCityID = DMax("CityID", "Cities")
   
    strSQL = "SELECT R1.RegionID, City FROM Regions AS R1,Cities AS C1 " & _
        "WHERE CountryID = " & lngNewCountryID & _
        " AND EXISTS" & _
            "(SELECT * " & _
            "FROM (Cities AS C2 INNER JOIN Regions  AS R2 " & _
            "ON C2.RegionID = R2.RegionID) " & _
            "INNER JOIN Countries " & _
            "ON Countries.CountryID = R2.CountryID " & _
            "WHERE Countries.CountryID = " & lngCountryID & _
            "AND R2.Region = R1.Region " & _
            "AND C2.CityID = C1.CityID)"
          
    Set rstRegions = CurrentDb.OpenRecordset(strSQL)
    
    ' loop through set of Region/City values in country being copied
    ' and insert a new row into Cities, incementing value of CityID by 1
    ' at each iteration of the loop
    With rstRegions
        .MoveLast
        .MoveFirst
        Do While Not .EOF
            lngNewCityID = lngNewCityID + 1
            strSQL = "INSERT INTO Cities(CityID,City,RegionID) " & _
                "VALUES(" & lngNewCityID & ",""" & .Fields("City") & """," & .Fields("RegionID") & ")"
            CurrentDb.Execute strSQL, dbFailOnError
           .MoveNext
        Loop
    End With
    
End Function

In previous versions of Office, Microsoft included a kind of "junior" version of MathType from Design Science.  

In Office 2013, they replaced MathType with their own equation editor.

If you're inserting new equations in 2013 using Insert |  Symbols | Equation, you should be able to edit them after having inserted them.  Let us know if that's not the case.

If you've inserted equations in previously created presentations using earlier versions of PowerPoint, they would be Objects created by MathTypeJunior, or as MS calls it, Equation 3.0.

So far, facts.  Now we speculate a bit:

If memory serves, the old equation editor is available as an optionally installed component in Office.  

And while I'm not certain of it, I think that the greyed out/slashed handles you're seeing (and thanks for adding the screen shot of that!) may be PPT's way of telling you "OK, there's this embedded Object thing here, but Windows is telling me that the application needed to edit it isn't installed.  So you can't.  Edit, that is."

You can test this:

Choose Insert | Text group | Object 

Click Create new if it's not already selected, then under Object type: look for Microsoft Equation 3.0

If it's not there, then I think we may've fingered the problem.  And in that case, try a re-install of office, do a Custom install and make sure to opt into MS Equation 3.0
