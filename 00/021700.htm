john mcghie say:

Yes.  It drives me crazy, regularly.  OK, so it's a short drive from here to "crazy" but still...

In a nutshell, the problem will turn out to be Capitals, Colons, and Spaces.

The index generator is extremely sensitive to the presence or absence of capital letters, semi-colons where there should be colons, and spaces.

It really helps to turn on your non-printing characters so you can see what you are doing.

And of course, if you have tracked changes in the document, you will get some very weird results, because the supposedly "deleted" text is in fact only "marked for deletion", and all its tags, fields, and numbering are still active until the changes are accepted and the text is actually deleted.

The index generator in Word 2011 is rock solid, you can trust it.  I am not aware of any bugs in it at all.  So whatever is happening in the index is in a tag somewhere!

The other thing that may catch you out is that you should HIDE the non-printing characters before generating the index, otherwise the extra characters can throw the pagination and produce strange page numbering.

Hope this helps

This is the sticky part. When you use foreground processing (macros that are recorded or macros that do everything by selecting view elements), code generally runs slower. It is much more efficient to use background processing (working directly on Project's objects independent of the current view). Prevent direct access to tables on accde - Microsoft Community

self untaught say:

Prevent direct access to tables on accde

If someone has the full version of msaccess.exe and they open the accde, how do I prevent them from opening the navigation pane and viewing the tables directly?  Before creating and distributing the accde, I have added the option in "current database" to not display the navigation pane on startup, but anyone with knowledge of the correct function key can then display the navigation pane and have direct access to the tables. albert d. kallal say:

Well you can’t modify the accDE anyway.

However the general approach is:

Of course, during development and design, you will hold down the shift key so your start up form, and the start-up settings don't run. So the display of all those special keys and even the start-up form, and the hiding of the nav pane setting will be IGNORED!

You then develop for a while, and then to test in "user" mode, you exit, and then re-enter the application without the shift key bypassed. You will likely do this dance all day long as you run/test as user mode, and then flip back in to developer mode.

You cannot really develop or modify things when you run your application with the start-up settings and custom ribbon set (or 100% hidden). And there are SO MANY settings you need DISABLED for users and ENABLED for developing.

So just use shift-by-pass key when you want to work during application start up.

So in windows explore - click once on your application to select it. Then with left hand, I hold down the shift key - keep holding ti down, with right hand you give the enter key a quick tap - but keep holding down that left shift key. This will by-pass ALL of your access startup settings and allow you to work on your accDB.

And if you don’t want to exit + re-enter the application 100's of times all day long to filip into “test” as user mode?

A really cool trick is to place a compact + repair button on the QAT. So when working, it is ONE mouse click and you simply do NOT hold down the shift key – presto without an exit + re-enter you are now in "user" mode and all your start up settings are now "active" and engaged for you to test your application in "user mode". And this includes your custom ribbons.

Another c+r and HOLDING down the shift key will flip you back into developer mode. This assumes you have the c+r in your QAT - but OFTEN by your desire, you your start up settings may well hide the QAT.

So in “user” mode, you may well not see or have the QAT.

So in this case to "flip" back to developer mode I simple hit alt-f4 to exit the whole application. At this point the accDB file in windows explore will STILL be highlighted so I now tap the enter key (and hold down shift key during start up).

 So exiting and shifting between "user" and "developer" mode "dance" will occur all day long. 

When you finally have things just right? You THEN compile the accDB down to an accDE to lock your code and forms from design changes.

You also can and THEN should disable the shift key by-pass and distribute this compiled accDE application to your users.

Compiled applications not only lock up your forms and code from any changes, but such code run MUCH more reliable since un-handled errors will NOT reset global or local vars – you can REST assured that all variables remain intact during the WHOLE time the user is running your application.  The above is really quite much the only practical approach to developing in Access. 

Better yet this approach ALSO eliminates the need to have all kinds of VBA code on start-up that attempts to hide or show the developer surfaces in Access. So you don’t need code to hide navigation pane. Just use the “check box” to show or hide the nav pane in the settings. Holding down shift key during start up not only ignores all those settings, it ALSO means your start up code and start up form DOES NOT run.

So at the end of the day, you really never need to set or change the settings of the compiled accDE – it is only the resulting “executable” application that you distribute to all your users. As noted about the only “last” thing you do is disable the shift key.

And as others stated here – you don’t want to accident lock down your developer accDB copy – only the resulting accDE that you distribute to each user’s workstation.

The above approach will quite much lock down your application from all but rather experienced access developers.

By Pass Shift Key Code
