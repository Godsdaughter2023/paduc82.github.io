Known Issue: Windows Insider Builds 17060 through 17077 - Word unable to save with add-ins running

Hello Office and Windows Insiders.

Recently, Windows Insider builds 17060 through 17077 were released to a subset of Windows Insiders. These builds contain a bug which prevents Word files from being saved if any add-ins are running. Anybody running a Windows Insider build in this range that run Word add-ins will be impacted by this bug. This impacts all versions of Word, including production builds, as well as both traditional save and auto-save.

The issue has been identified and a fix is being worked on, however we do not have an ETA at this time.

In the meantime, users impacted by this issue have two alternatives as a work around:

  1. Use the web version of Word 
  2. Uninstall any Word add-ins you have and continue to use Word as normal

-Jesse

ken sheridan say:

The chances of having a credit or debit of the same amount, on the same day, for the same compny is extremely unlikely

'Extremely unlikely' is not really good enough to be confident of not losing data.  You need to be sure this is the case.  You can find this out by creating a query which is groups the rows by the three columns and returns COUNT(*) in a fourth column.  If this returns a count of more than 1 in any row, then you have duplicates.  If you add the following clause to the query:

HAVING COUNT(*) > 1;

this will retun only those rows with duplicates, so you'll be wanting it to return an empty result table.

You don't mention a column such as Company or Customer?  Is the memo column the Customer?  Its name would suggest it’s a memo field (now known as 'long text') but it seems to be the only candidate for the company identifier.  If so, and your assumption that there will be no transactions with the same date, amount and company values on the same day is correct, then you can return all transactions not currently represented in tblTransactions with:

SELECT tblData.Date, tblData.Amount, tblData.Subcategory, tblData.Memo
FROM tblData
WHERE NOT EXISTS
    (SELECT *
     FROM tblTransactions
     WHERE tblTransactions.Date = tblData.Date
     AND tblTransactions.Amount = tblData.Amount
     AND tblTransactions.Memo = tblData.Memo);

If this returns the correct rows you can easily convert it into an 'append' query to insert rows into tblTransactions.

In the absence of any other columns with customer data, I'm guessing that, if my assumption that the memo column is the company identifier is correct, this contains things like the company name, address etc as a single string expression.  If so you will then be faced with the problem of parsing this into separate columns in order that you can insert distinct rows into a tblCompanies table, and then reference the primary key of this table in a CompanyID or similar foreign key in tblTransactions.

BTW do not use Date as column name.  As the name of a built in function it's a 'reserved word'.  Use TransactionDate instead.

ken sheridan say: There are still some unanswered questions:

1.  You referred to credits or debits in respect of companies.  How is the company identified in the data?  The memo column seems to be the only candidate.  Is that the case?  If the creditor or debtor is identified as a substring within the column, e.g. taking an example from my own current account 'DIRECT DEBIT PAYMENT TO BRITISH GAS SERV REF ############ MANDATE NO ####' the company is British Gas.  It's simple enough to insert the complete string into a column in a Transactions table, but in a relational represented by the column.  So while inserting the complete string into a column is a first step, the substring which identifies the creditor/debtor will first need parsing out in order that it can firstly insert distinct rows into a separate table of creditors/debtors, and from that insert values into a foreign key column in the Transactions table to reference the primary key of the table of creditors/debtors.  This does not need to be done simultaneously with the initial import, but it will need to be done subsequently if the database is to be correctly structured.  You'll find an example of this sort of thing, but without the parsing, as DecomposerDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file imports non-normalized data from an Excel worksheet and decomposes it into a set of correctly normalized related tables in Access.

2.  How are credits and debits differentiated in the source data?  Are they positive and negative values?  Statements from my own bank have then as separate columns.

Turning to your specific questions:

1.  I would rename the Date column as Transaction date only in the tblTransactions table.  You can live with the bad name in the text file and the tblData table provided the column name is qualified with the table name in the queries.

2.  The HAVING clause goes in the first query to check for duplication:

 
SELECT tblData.Date, tblData.Amount, tblData.Subcategory, tblData.Memo,
COUNT(*) AS NumberOfDuplicateRows
FROM tblData
GROUP BY blData.Date, tblData.Amount, tblData.Subcategory, tblData.Memo
HAVING COUNT(*) > 1;

This is entered in the query designer in SQL view.  Open the query designer, switch to SQL view, and paste in the above in place of what's there already.  When you open the query, if you see an empty result table, all is well.

3.  The main query is entered in the same way.  It is converted to an 'append' query as follows:

INSERT INTO tblTransactions(TransactionDate,Amount,Subcategory,Memo)
SELECT tblData.Date, tblData.Amount, tblData.Subcategory, tblData.Memo
FROM tblData
WHERE NOT EXISTS
    (SELECT *
     FROM tblTransactions
     WHERE tblTransactions.TransactionDate = tblData.Date
     AND tblTransactions.Amount = tblData.Amount
     AND tblTransactions.Memo = tblData.Memo);

Note the change of name of the Date column in tblTransactions to TransactionDate.  This change to the table design must be made before executing the query.  You can preview what the query will insert by clicking on the datasheet view icon in the Results area of the Design ribbon.  To execute the query click on the adjacent Run icon.

4.  To empty the tblData table ready for the next inport, execute the following query:

DELETE *
FROM tblData;

which is entered and executed in SQL view in the same way.

ron coderre say:

Here's the point I want to make sure you understand: If your information and processes are truly so sensitive that nobody outside the company should ever see them, do not look to Excel to protect them.  Excel security is so meager that a relatively unskilled person can, with little effort, discover how to unlock protected workbooks, worksheets, and VBA projects. Consequently, anyone with access to the file can simply transfer it to another location and break into it.

If that situation is acceptable, there are various approaches you could use. Here's one:

In a general module, this function tests if a file exists:

Function FileExists(strPath As String, strFileName As String) As Variant
   Dim fsoFSO As Object
   Dim strFullPath As String
   If Right(strPath, 1) <> "\" Then
      strPath = strPath & "\"
      [FileSearchPath].Value = strPath
   End If
  
   Set fsoFSO = CreateObject("Scripting.FileSystemObject")
  
   If fsoFSO.FolderExists(strPath) = False Then
      FileExists = False
   Else
      strFullPath = strPath & strFileName
      If fsoFSO.FileExists(strFullPath) = True Then
         FileExists = fsoFSO.getfile(strFullPath).DateLastModified
      Else
         FileExists = False
      End If
   End If
   Set fsoFSO = Nothing
End Function

In the workbook module, this code runs when the workbook is opened AND macros are enabled:

Private Sub Workbook_Open()
Dim strPath As String
Dim strFileName As String
    strPath = "C:\"
    strFileName = "YourFileName"
   
    If FileExists(strPath:=strPath, strFileName:=strFileName) Then
        'Do something
    Else
        MsgBox "You are not authorized to use this application."
        ThisWorkbook.Close SaveChanges:=False
    End If
End Sub
