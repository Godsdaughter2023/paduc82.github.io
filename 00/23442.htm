bernie deitrick say:

Don't use links to the cells. Instead, either:

1) Remove the Subtotals and create a pivot table based on your data set and the subtotals will be properly shown no matter what you do to the source data - well, within reason.

2) Use the Subtotals but collapse the data to show just the subtotals, and don't use a second sheet.

3) Use a formula like

=VLOOKUP("East Region Total",Sheet1!A:B,2,FALSE)

replacing the "East Region Total" with a cell reference or a hard-coded string that will find the subtotal of interest.

bernie deitrick say: Tables of one way of addressing the additional data, but an easier and more foolproof way (tables are good but are not entirely errorless) is to use entire columns as your data source rather than specific rows within the columns.

    Ahh that's easy. You don't! As a general rule we don't store calculated values. So its fine to use the expression as the controlsource of a control on your form or use it in a query. But the value shouldn't be stored in a table.

    ken sheridan say: If you are inserting the PipeSize and NominalDensity into columns in a referencing table then you are introducing redundancy, which means the table is not normalized to Third Normal Form (3NF)  and consequently at risk of update anomalies.  The referencing table should contain only the foreign key column ProductCode.

    The PipeSize and NominalDensity values can be shown in unbound text box controls in the form, with ControlSource properties of:

    =cboProductCode.Column(1)
    and
    =cboProductCode.Column(2)

    The Column property is zero-based, so Column(1) is the second column, and so on.

    The removal of these columns from the referencing table would of course rule out the use of a calculated column based on one of those columns.  The calculation can be done in an unbound computed control in the form, however, or in a computed  column in the form's RecordSource query.  Calculated columns in tables serve no useful purpose; the place for such computation is in a query or in a control in a form or report.

    You say that 'Some of the product sizes are listed with no product codes'.  How can that be the case?  From the information that you have provided ProductCode, as the combo box's BoundColumn, must be a candidate key of the Pipe Product Codes table.  A candidate key cannot be NULL.
