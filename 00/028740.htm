Hi,

For any minute in the plan, Peak=Work (done during that minute)

Then for any larger slice of time, Peak is the highest peak number for all the minutes of that period.

This is calculated task by task (or to be precise, assignment by assignment) o whether you have 2 tasks or 10000 is not relevant.

albert d. kallal say:

Well, if you going to adopt a web interface, then you really much have to adopt a web server also.

 

The reason for this is many, but a stand-alone browser will STILL have to use + open that back end database. And if you write all your code inside of the browser, then you essentially still writing a desktop program. So to answer your question, being JUST limited to the desktop browser is NOT really able to open + pull data from that file share. And you would STILL be required to have use of a shared folder on your local network. (So JUST adopting the web browser will NOT allow users say outside of the building or people on the road to use the resulting application.

 

And worse, the browser would require references to the ACE/JET database engine. And that would mean say an iPad or smartphone browser would not work. (You would be limited to 100% compatible windows computers, and EACH windows workstation wold STILL requite the Access database engine to be installed on each workstation.

 

What the above means, is you still be writing software 100% limited to windows PC computers that run on the user’s desktop.

 

To eliminate the above issue, then in addition adopting a web browser, you ALSO have to adopt a web server. If you adopt a web server, then you get to write SERVER side code. The result is that the access database engine ONLY needs to be installed on the web server, and thus any standard web browser on an iPad or anything else can then run that application.

 

So keep in mind that when you write a web application, then the web server you choose has to support all of the tools and systems you plan to develop that web based system with.

 

So if you choose say a Linux hosted system, then you can’t install windows software on that web server (which means you can’t use Access). So most web applications that run on Linux are built around the so called “lamp” stack (Linux (the OS), Apache (the web server), M is MySQL (the database server), and “p” stands for PHP – the web side programming language you use.

 

So when choosing a web server, you have to:

  1. Adopt the program system and tools that he web server offers.

  2. Adopt a set of programming tools you like and THEN choose and ensure that the web server allows you to run that software.

     

    So while the browser can run on near any computer, the web server not really any different then you’re desktop. You have to install the required programming tools and systems onto that computer). So while the web server can “spit out” web pages to be consumed by any computer, the web server will be LIMITED to ONLY the programming systems and languages that the web server supports.

     

    So if for example you like VBA, and thus choose vb.net and the net platform to develop a web application, then the web server has to be a windows based web server and support the .net programing system.

     

    In summary:

    You have to choose a set of programming tools and development tools that you like and THEN ensure that your web server supports those programming systems. If the web server is missing “any” part of the systems you choose to develop your software with, then you can’t install + run your software on that web server.

     

    And keep in mind that if you use Access as the database, then you not use ANY feature of Access except that of the tables. (Since web servers don’t support access forms or reports). Since you are ONLY uses Access to store the table data and zero other features of Access, then you likely best to adopt a server based database like MySQL or Microsoft SQL server.

     

    So if you choose Linux as your web OS, then you likely be using the Apache web server, and your database choice likely will be MYSQL.

     

    If you want to use say vb.net, and SQL server (or Access), then you have to choose a web server that runs on the windows OS. That means you likely choose IIS (that is Microsoft web server system they offer). And since the system runs on windows, then the database system can be SQL server or Access. However since you only using the tables from Access, then you might as well just choose SQL server, since all it really does is hold table for you, but SQL server is FAR better and works far better with the IIS web server then does Access.

     

    If your programing language for the web server is PHP, the likely the LAMP stack is a better choice. That quite much suggests a Linux server (but that does not support Access). I never used PHP on IIS – but it likely is supported.

     

    However, coming from a VBA environment, then I much prefer using Visual Studio, and VB.net. The reason for this is that vb.net is VERY similar to VBA, and creating of web forms in Visual Studio is a drop + drop process VERY much like Access.

     

    So in addition to your given tools, you also have to choose a web server.
