scottgem say:

Based on what you have said, you have not optimally deployed your application. Every experienced developer I know recommends that EVERY multi-user application NEEDS to be split between a back end  (the tables) and a front end (everything else). The back end is then stored on a shared network drive and the front end is stored on each user's PC. In addition, any important application should be backed up regularly. 

I know this doesn't help you recover the data, but before you put this application back into production you need to split it and deploy it as I outlined.

ken sheridan say: As Tom says, this appears to be due to the database having been amended in some way in a later version of Access than 2007.  However, the root of the problem is the fact that a single file is apparently being opened by multiple users.  This is asking for trouble.  If you can open the file with a later version of Access you hopefully might be able to recover the data and correct the installation as described below.  But you might not be lucky.

A correct installation would involve splitting the file into separate front and back ends, the latter containing only the tables, the former containing all other objects, along with links to the back end tables.  The built in database splitter wizard can do this for you.

Once you have the database split the back end should be installed in a shared location on the system, to which users have full read/write privileges.  Separate copies of the back end should be installed on each users local machine, or, as in my organisation, in 'personal' locations on the system to which each user only has access.  The built in linked table manager can be used to refresh the links to the back end if necessary, but do this before copying the front end to each user to avoid having to do so multiple times.

The use of separate front ends and a shared back end not only makes corruption of the data far less likely, it also allows any development of the front end to be done independently, following which copies of the updated front end can be distributed to each user to replace their existing copies.  Splitting also allows the data to be backed up independently of the front end on a regular basis (daily at the very least) as it is merely a question of copying the back end file to a secure location, though back-ups can themselves become corrupted, so making at least two back-ups is strongly advised.
