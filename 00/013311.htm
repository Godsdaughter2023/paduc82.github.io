If you upgrade from a OEM version of Windows 7 or Windows 8/8.1 to the free Windows 10 upgrade this summer, the license is consumed into it. Because the free upgrade is derived from an OEM base qualifying license, Windows 10 will carry this type of licensing too.

If you upgrade from a retail version, it carries the rights of a retail version.

Full version (Retail):

- Includes transfer rights to another computer.

- Doesn't require a previous qualifying version of Windows.

- Expensive

Upgrade version (Retail):

- Includes transfer rights to another computer.

- require a previous qualifying version of Windows.

- Expensive, but cheaper than full version

OEM :

OEM versions of Windows are identical to Full License Retail versions except for the following:

- OEM versions do not offer any free Microsoft direct support from Microsoft support personnel

- OEM licenses are tied to the very first computer you install and activate it on

- OEM versions allow all hardware upgrades except for an upgrade to a different model motherboard

- OEM versions cannot be used to directly upgrade from an older Windows operating system

ken sheridan say: One reason why experienced developers object so strongly to the use of the lookup field wizard when assigning a data type to a column in table design view is that it gives rise to this kind of confusion.  For further details see:

http://theaccessweb.com/lookupfields.htm

When a lookup field is created the column in the referencing table will normally be a long integer number data type, referencing an autonumber primary key of the referenced table.  An autonumber is merely a special king of long integer column whose values are inserted automatically.  Your referenced FieldPositions table, therefore, is likely to be structured as follows:

Positions
‚Ä¶.FieldPositionID (autonumber PK)
‚Ä¶.FieldPosition (short text)

It follows from this that you cannot call the DLookup function in the way you have attempted, because the FieldPosition column in that column will not contain text values such as 'CEO', but a number which maps to the row in FieldPositions with the same value in its primary key column.  You could call the DLookup function against a query which joins the two tables, or you could do so in two stages in code as follows:

Dim varFieldPositionID As Variant
Dim strEmployee As String

varFieldPositionID = DLookup("FieldPositionID", "Positions", " FieldPosition = ""CEO""")

If Not IsNull(varFieldPositionID) Then
    strEmployee = DLookup("FieldName","Table1","FieldPosition  =" & varFieldPositionID)
    MsgBox strEmployee & " is CEO", vbInformation", "CEO Search"
Else
    MsgBox "No CEO record located", vbInformation", "CEO Search"
End If

Without using the lookup field wizard you could, and normally would, set up the tables manually with exactly the same columns, but without the unfortunate consequences detailed at the above link.  The referencing table would then show the true numeric values in the foreign key column when the table is opened in datasheet view.  You should never interface with data in raw datasheet view, however, but only in forms.  In this case a combo box in a form would be bound to the foreign key column, but set up to show the text values from the referenced Positions table.

Another option would be not to use 'surrogate' numeric keys at all, but text 'natural keys, i.e. the Positions table would be set up as follows:

Positions
‚Ä¶.FieldPosition (short text PK)

The foreign key in the referencing table would similarly be of short text data type.  While numeric keys are generally used and are, in most cases, a more efficient approach, there is nothing whatsoever in the database relational model to say that natural keys should not be used where the values are distinct.  In some circumstances they can have significant advantages over surrogate numeric keys.  Which to use is a matter of informed judgement by the developer.  The code would in then be as follows:

Dim varEmployee As Variant

varEmployee = DLookup("FieldName","Table1","FieldPosition  = ""CEO""")

If Not IsNull(varFieldPositionID) Then
    MsgBox varEmployee & " is CEO", vbInformation", "CEO Located"
Else
    MsgBox "No CEO record located", vbInformation", "CEO Located"
End If

Note that, as a text value CEO must be delimited by literal quotes characters, "FieldPosition  = ""CEO""".  In this expression each delimiting literal quotes character is represented by a contiguous pair of quotes characters.

I've used your table and column names in the above.  I'm assuming that in reality they have different more meaningful names.
