Word 2010 Error! Bookmark not defined! - Microsoft Community

bewareofdog say:

Word 2010 Error! Bookmark not defined!

 I have a document with multiple {XE} index fields. When I go to print, they all turn into Error! Bookmark not defined. I have tried updating the index just before printing to no effect. I have done a REPLACE, and replaced all with a DELETE. The Fields disappear from the document, but when I go to print, the error message returns in all the same places. Help! This is a 300-page document with 10,000 index fields!

jay freedman say:
Whenever Word encounters any field code that doesn't start with a recognized keyword (such as XE or FILENAME), it assumes that it's a REF field, and that the first word in the field code is a bookmark name. This is what happened to all of your XE fields when you changed XE to ZZ -- each field was looking for a bookmark named ZZ, which doesn't exist.

Now you've fixed most of the fields, maybe all of them except one.  To look for that, make sure you see field results instead of field codes (and turn off the display of hidden text so you don't see the XE fields), and then do a Find for the word  Error!  (including the exclamation mark). For each one you find, click it and press Shift+F9 to display the field code, fix it, and press F9 to update it.

suzanne s. barnhill say:

It would appear that you already have the fields set to update on print, since that's when you're getting the error, but the setting is at File | Options | Display: "Update fields before printing."

 

If the only thing wrong with the XE fields is that you've replaced XE with ZZ, then you can reverse the process, replacing ZZ with XE (the field codes must be visible when you do this). If you do want to delete them all and start over, then search for ^19 XE (which will select the entire field) and replace with nothing.

jay freedman say:
In theory, there should be no reason that you can't use Office or other programs while Windows Maintenance Mode is running. They may be slower, because of the extra load on the CPU and the hard drive, but they should be usable.

However, there are two things you can do to get Maintenance Mode out of your way.

The best one is to schedule maintenance to happen in the middle of the night -- say, 2:00 AM. In the Windows Action Center, expand the Maintenance heading, click "Change maintenance settings" (if maintenance is running, you have to click "Stop maintenance" to see this link), and set the time. If you're in the habit of turning off your computer every night, check the box for "Allow... to wake up", and set the computer to sleep instead of turning it off.

The other alternative is to just go to the Action Center and click "Stop maintenance" when a program isn't usable because of maintenance. It can take a couple of minutes for the stop to take effect, because it has to finish copying whatever file it's working on.

ken sheridan say: You've been given two workable solutions, but when I see something like I have several fields, "Type A", "Type B" my relational antennae begin to twitch.  If these are columns in the result set of a crosstab query, that's fine, but if they represent separate columns in a base table, then it suggests that you might be doing what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

I'd be interested to know what these columns represent.  If they represent different categories of the same attribute then the table should be decomposed into related tables.  You'll find an example of how to automate this as UnencodeCoulmns.zip in my public
