ken sheridan say: You can open the report in print preview and automatically open the print dialogue with:

    Const PRINT_CANCELLED = 2501
    Const REPORT_NAME = "Voucher Hotel A"

    DoCmd.OpenReport REPORT_NAME, acViewPreview
    On Error Resume Next
    RunCommand acCmdPrint
    Select Case Err.Number
        Case 0
        ' no error
        Case PRINT_CANCELLED
        ' anticipated error, do nothing
        Case Else
        ' unknown error, inform user
        MsgBox Err.Description, vbExclamation, "Error"
    End Select

Another approach is to allow the user to select a printer from a list or combo box of installed printers first to tempoaraily change the printer used by Access from the Windows default printer.  You'll find an example of this in LabelPrinter.zip in my public

    scottgem say:

    What is the code behind the View Report button? Is the Select report control an option group?

    You need to use the WHERE clause of the Open Report method to filter the report. 

    ken sheridan say: I take it from the label in your form that each combo box allows the selection of an asterisk to return rows regardless of the value in the column in question.  You say you want to restrict the report by a date range, but you also include a number of combo boxes in the form, so I assume that these also act as parameters for the report.

    To restrict the reports to the selected criteria in the text and combo boxes base each on a query with a WHERE clause along these lines:

    WHERE (StartDate] >= Forms![YourForm]![txtStartDate]
    OR NZ(Forms![YourForm]![txtStartDate],"*") = "*")
    AND (EndDate] < Forms![YourForm]![txtEndDate]+1
    OR NZ(Forms![YourForm]![txtEndDate],"*") = "*")
    AND (Department] = Forms![YourForm]![cboDepartment]
    OR NZ(Forms![YourForm]![cboDepartment],"*") = "*")
    AND (Shift] = Forms![YourForm]![cboShift]
    OR NZ(Forms![YourForm]![cboShift],"*") = "*")
    < and so on to>
    AND (ClassType] = Forms![YourForm]![cboClassType]
    OR NZ(Forms![YourForm]![cboClassType],"*") = "*")

    I imagine that not all reports, if any, will require the full set of parameters, however.  It's simple to remove/add specific parameters from the WHERE clause, but note that the parenthesizing is important to ensure that the OR operations are evaluated independently within each AND operation.

    To select the report to be opened you need to identify the report name on the basis of the value of the option group, e.g.

    Dim strReport As String

    Select Case Me.YourOptionGroup
        Case 1
        strReport  = "rptTrainingHistoryAllEmployees"
        Case 2
        strReport  = "rptTrainingHistorySpeficEmployee"
        < and so on to>
        Case 7
        strReport  = "rptTrainingNearDueByClass"
    End select

    DoCmd.OpenReport, strReport, View:=acViewPreview

    However, it appears that you might be duplicating reports unnecessarily.  For instance you appear to have separate training reports for all employees and a specific employee.  You only need one report for this.  By virtue of the parameters in the query, if the EmployeeID combo box is left Null or an asterisk selected the report would return all employees, otherwise a single employee.  Without knowing more of your other reports I can't comment on the degree of unnecessary duplication which might be involved in them.

    You'll find examples of various ways to restrict a report's results by means of controls in an unbound dialogue form as MultiSelect.zip in my public databases folder at:

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

    dhookom say:

    The WHERE clause should be between the FROMs and the GROUP BY.

    Do you really need this to be a GROUP BY since there are no aggregations in the SELECT clause?

    Ideally the HAVING clause filter should be moved to the WHERE clause.

    scottgem say:

    First, you are using a Group By query where there are no aggregate functions. A Group By is uses when you want to sum, avg, etc some values across multiple records. If the reason is because you were getting duplicate records then just use SELECT DISTINCT.

    Second, your WHERE clauses are coming after the ; hence the error messages. WHERE clauses go after the FROM clause and (when using one properly) before the Group By clause.
