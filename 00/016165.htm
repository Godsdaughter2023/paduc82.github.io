Access and Outlook - Manually Adding Attachments produces Array Index - Microsoft Community

bill j. say:

Access and Outlook - Manually Adding Attachments produces Array Index Out of Bounds

I posted this back on Feb 27, I think I didn't give a good description in the Subject line.  Reposting...

I have a routine in Microsoft Access that produces an email.  I had to modify the code for Office 2016 where .Display needed to be brought to the top of the With (.) stack:

The following works with Windows 7 and MS Access 2013

Public Function ProcessEList(stTo As String, _
                            Optional stCC As String, _
                            Optional stSub As String, _
                            Optional stRpt1 As String, _
                            Optional stRpt2 As String, _
                            Optional stMsg As String, _
                            Optional stRpt3 As String) As Boolean

    Dim objOutlook
    Dim objMail

On Error Resume Next
    Set objOutlook = GetObject(, "Outlook.Application")
    ProcessEList = False

    If Err.Number <> 0 Then
        Set objOutlook = CreateObject("Outlook.Application")
    End If

    If IsNull(stCC) Then
        stCC = ""
    End If

    Set objMail = objOutlook.CreateItem(cnMailItem)

    With objMail
        .To = stTo
        .CC = stCC
        .Subject = stSub
        .HTMLBody = stMsg
        .BodyFormat = cnHTMLBodyFormat
        .Attachments.Add stRpt1
        .Attachments.Add stRpt2
        .Attachments.Add stRpt3
        .Display
        .Display
            AppActivate stSub, True
    End With

    ProcessEList = True
End Function

Moving .Display up to the top of the stack of Dot objects is the charm to allow it to work with Windows 10 and MS Access 2016.  HOWEVER! 

If the user tries to attach additions files manually  that weren't attached with the VBA Code,

1.  The attachment(s) added by the VBA code is removed

2.  I get the dreaded "Array Index Out of Bounds" error.  I believe that it has something to do with the order of events.

3.  The Error Message Appears with no Error Number, so there isn't a method I can use to trap it with error handling (I tried)  

3.  A work-around we've come up with is:  

     a.)  Save the Email as a draft.

     b.)  Re-Open the draft (any attachments added by VBA will be there) and continue to add any additional attachments.

Another note, 2013 users got errors when they used the code I modified for Access 2016.  I don't know if it was only because .Display was moved or, more likely, the modifiations changed some of the reference library items that aren't compatible with the older version of Access (I'm unable to check for that).

Note:  In my production version of this Module I have added a couple of lines to check if Outlook is running or not.

Hi Hugh,

I'd like to firstly explain some processes of Multi-factor authentication:
1.When admin enable MFA for one user in Office 365 admin center, it will display Enabled next to that user.
2.When that user sign into Office portal the first time after MFA enabled, he will be redirected to the page as the first screenshot in your post. He needs to finish Additional security verification. Note: User will get an app password during this process. Let them copy and save the password. It will be used to sign into Office 2013 applications. More infomation, see this article.

You may let them try the two options below to see the outcome:
Option1.Resign into Office applications.
Open one of the Office applications>File>Account>Sign out the existing account>sign in>enter the same account again.
Option2.Remove Credentials.
a.Open one of the Office applications and sign out the existing account.
b.Open Control Panel>Credential Manager>remove or Microsoft related credentials in both Web Credentials and Windows Credentials.  Restart
c.Restart computer and resign into Office applications.

Let us know some updates when you have time.
