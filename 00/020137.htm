Hi Jeremy,

The "Save list as template" option is available only when custom script is enabled on this site collection. Please follow these steps to check this settings and make sure custom script is enabled on this site collection:

a. Download and install the SharePoint Online Management Shell from this article.
b. Launch SharePoint Online Management Shell, connect to SharePoint Online via the the following commands using your Office 365 global admin account,

Connect-SPOService -Url https://contoso-admin.sharepoint.com

c. Check the custom script settings using this command:

get-SPOsite https://contoso.sharepoint.com/sites/sitename | select DenyAddAndCustomizePages

d. If the above result shows "enabled", it means custom script is disabled on this site collection, please run this command to enable custom script on this site collection:

Set-SPOsite https://contoso.sharepoint.com/sites/sitename -DenyAddAndCustomizePages 0

For your reference:
Turn scripting capabilities on or off.

After running the command, you can also check the result by running:

get-SPOsite https://contoso.sharepoint.com/sites/sitename | select DenyAddAndCustomizePages

make sure DenyAddAndCustomizePages shows Disabled.

Remember to change https://contoso.sharepoint.com/sites/sitename to the URL of the affected site collection.

bernie deitrick, excel mvp 2000-2010 say:

The date and time of the rollback is - for the year entered into a cell named Yr, and for countries that fall back on the first Sunday of November:

=DATE(Yr,11,9)-IF(WEEKDAY(DATE(Yr,11,1))>1,WEEKDAY(DATE(Yr,11,1)),7+WEEKDAY(DATE(Yr,11,1)))+1/12

You can use that to compare your date and time.

Of course, a time between 1:00AM and 2:00AM on 11/6 is ambiguous, since that hour technically happens twice this year.

greg maxey say:

Yes.

You can put the controls in a Frame control and show or hide the frame, or you can loop through controls by type, or by evaluating the string name:

Private Sub UserForm_Initialize()
Dim oCtr As Control
'Or
Me.Frame1.Visible = True
'Or
For Each oCtr In Me.Controls
 If TypeName(oCtr) = "Label" Then
   oCtr.Visible = False
 End If
Next
'Or
For Each oCtr In Me.Controls
 If InStr(1, oCtr.Name, "Q") > 1 Then
     oCtr.Visible = False
 End If
Next
End Sub

busylabtech say:

Finally found that the culprit was the 'enter' return mark made by the program after I would change from whatever font I was using (say Arial) to the barcode (some version of 39).  Word will leave a mark in the barcode font at the very end of the barcode, but after the necessary *.  I just have to identify this spot, change it to some other font, and everything works the way it used to. 

Don't know if this works for everybody, but I find it a much simplier method for barcoded labels than some other methods I have seen out there.

To reiterate:

I use Excel to create a data sheet for all my samples, assigning a consecutive control number to each one.  Then I go to Word, and do a mail merge, for labels, specifying whatever sheet of the excel file I'm working from. I use the Avery 5160 sheet of labels, so I get 30 labels per sheet.  For the sample label at the beginning, I write in the items that will stay the same for all of the labels; the year and what study, for instance, and then use the 'insert mergefield' to specify which column to pull the information for the rest of the label.  The final line of the label will have the barcode, written this way- *year-study-mergefield*.  The year and the study will be the same for each code, and the mergefield will change with the sample.  Then I highlight the whole phrase, and convert it to the barcode font, (I like IDautomation's free Code 39).  After the conversion, I highlight the vey tail end of the phrase, making sure I leave the final * alone.  I am just trying to get the character or keystroke left from entering the font conversion.  I'm not sure what the terminology is for that bit of excess; mine is probably not printable, but it is the key to the whole business.  Once I have that little tail end of excess barcode identified, I convert that to a regular font, and the whole thing is ready to move to update labels, merge and print. 

All I am looking for is a quick way to ID the sample.  We have a scale that is tied into the computer, and will dump a weight into an Excel cell.  Our process is to read the bag label with the scanner, move to the cell to the right, print the weight of the bag, and move on.  The crew doesn't have to sort out the bags, enter the name ( with the potential of mistakes), find the sample out of what could be hundreds to enter the weight in the right spot, etc.  When all the bags have been read and weighed, we do a data sort, put everything in order and match the list up with the original data sheet.  Fairly simple, and we already own the technology.

Not sure who to thank; somewhere along the way I saw a mention of the 'excess keystroke', and finally followed it up.  I can send a set of demo files if someone wants to see how this all works.
