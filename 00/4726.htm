But (not surprisingly) all I can see in the "Electronic" PDF is that it is created by Microsoft Word, not the name of the agent that creates it.

The dates format OK here during Print Preview, which suggests to me that Microsoft has fixed something in Word since the version Adobe is using. 

Doing a few more experiments, it became clear that

 a. in the footer, the date switch is completely ignored during the "Electronic" PDF process, at least for DOCPROPERTY fields. I think it probably affects all field types

 b. attempting to apply the date switch using a different field code, e.g. wrapping the DOCPROPERTY inside a QUOTE, does not make any difference:

{ QUOTE { DOCPROPERTY VersionDate } \@"MMMM d, YYYY" }

 c. what does work is to put a SET field in the document body (e.g. at the beginning) to set a bookmark to the formatted DOCPROPERTY, then reference it in the footer, e.g.

In the main body:
{ SET VDate { DOCPROPERTY VersionDate  \@"MMMM d, YYYY" } }

In the header/footer:
{ VDate }

If it is of any use to you, another way to do this that appears to work in both Mac and Windows versions of Word 2016 and works with Microsoft's service is to insert (say) the Publish Date property (Word inserts a mapped date picker content control) and format each instance of the control using MMMM d, YYYY format. Then the user would have to know where to insert/pick the date.

ken sheridan say: If only one item is to be selected, there is little point in using a list box, a combo box will suffice and occupies less space.  More usefully it has the advantage that if a user types in a value it will progressively select the first match as each character is entered.  Whether you use a list box or combo box, however, the method is exactly the same.

For a search form you don't even need to use a subform; you can simply use a bound form in continuous form's view and put a combo or list box in the form header.  The combo box would be set up as follows:

Name:    cboFindResourceType

ControlSource:    Leave this property empty

RowSource:     SELECT ResourceTypeID, ResourceType FROM ResourceTypes ORDER BY ResourceType;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

The form's RecordSource would be the following query:

SELECT ResourceTypeDetails.*
FROM ResourceTypeDetails INNER JOIN Resources
ON ResourceTypeDetails.ResourceID = Resources.ResourceID
WHERE ResourceTypeID = Form!cboFindResourceType
ORDER BY Resource;

The combo box would have one control, a bound combo box set up as follows:

Name:    cboResourceID

ControlSource:   ResourceID

RowSource:     SELECT ResourceID, Resource FROM Resources ORDER BY Resource;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

To implement the search when you select a resource type in the combo box in the header you need to put the following code in the AfterUpdate event procedure of the combo box:

    Me.Requery

To enable you to enter a new row in the subform to add a new resource for the selected resource type put the following line of code in the form's BeforeInsert event procedure:

   Me.ResourceTypeID = cboFindResourceType

This  assigns the  value of the combo box to the ResourceTypeID column in the new row being inserted as soon as you select a new resource in the empty new row at the bottom of the subform.

Finally if you don't want users to be able to edit existing resources for the selected resource type in this form, put the following code in the form's Current event procedure:

    Me.cboResourceID.Enabled = Me.NewRecord
    Me.cboResourceID.Locked = Not Me.NewRecord
