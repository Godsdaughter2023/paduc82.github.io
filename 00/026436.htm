john w. vinson say: Sounds like you need to change your table design. If one Item can have multiple Prices, then you have a one to many relationship: a table of Items with information about the product (but no price info), related to an ItemPrice table with whatever fields you need to distinguish the different prices. You would then need to use a Query joining the two tables, with whatever criteria are needed to determine which price applies.

scottgem say:

Yes, you do need to review your table design. It sounds like your invoicing links to the product ID but does not have a field for unit price. Price is a time sensitive value so should be stored in invoice detail record. It sounds like your invoices are pulling the price from the Product table instead.

If the price for a job is dependent on multiple factors you need to account for that in your app. So if your pricing is tiered (i.e. lower due to quantity) you either need to apply a discount factor or you need to have a table of pricing per quantity. If price is also dependent on the paper used, then that should be a separate line item on the invoice.

ken sheridan say: In relational database terms you have a many-to-many relationship type between products, paper types, and finishings, with minimum quantity, maximum quantity and unit price as attributes of the relationship type.

A many-to-many relationship type is modelled by a table which resolves the relationship type into multiple relationship types, so you'd have a table like this:

Prices
….ProductID  (FK)
….PaperTypeID  (FK)
….FinishingID  (FK)
….MinQuantity
….MaxQuantity
….UnitPrice

The table has two overlapping candidate keys, one being all of the foreign keys plus MinQuantity, the other  being all of the foreign keys plus MaxQuantity.  Either of these can be made the composite primary key. The foreign keys reference Products, PaperTypes, and Finishings tables respectively.

For sales orders and invoices an OrderDetails or InvoiceDetails table would be like this:

OrderDetails
….OrderID  (FK)
….ProductID  (FK)
….PaperTypeID  (FK)
….FinishingID  (FK)
….UnitPrice
….Quantity

The primary key of this table is a composite of the four foreign keys, and UnitPrice.  The price should be inserted into the UnitPrice control in the form with the following code in the BeforeUpdate event procedure of the Quantity text box control:

    Const MESSAGE_TEXT = "A product, paper type, finishing, " & _
        "and quantity must be entered."
    Dim strCriteria As String
    
    strCriteria = "ProductID = " & Me.ProductID & _
        " And PaperTypeID = " & Me.PaperTypeID & _
        " And FinishingID = " & Me.FinishingID & _
        " And " & Me.Quantity & " Between MinQuantity And MaxQuantity"

    If Not IsNull(Me.ProductID) And _
       Not IsNull(Me.PaperTypeID) And _
       Not IsNull(Me.FinishingID) And _
       Not IsNull(Me.Quantity) Then

       Me.UnitPrice = DLookup("UnitPrice", "Prices", strCriteria)

    Else
       MsgBox, MESSAGE_TEXT, vbExclamation, "Invalid Operation"
       Cancel = True
    End If

If there are products to which a paper type or finishing is irrelevant then the PaperTypes and Finishings tables should each include a row with a text value in the non-key column of N/A or similar.  This can then be selected in the relevant combo box in the form where  appropriate.  The Prices table must include rows which reference the N/A rows in each table.

This is not necessarily the end of the story, however.  The above model would allow the section of any combination of paper type and finishing.  In reality there might be constraints on this, i.e. a paper type might be available in not all finishings.  If so, this constraint should be modelled by a further table:

PaperFinishes
….PaperTypeID  (FK)
….FinishingID  (FK)

This table is 'all key', i.e. its primary key is a composite of the two foreign keys.    In addition to valid combinations of paper type and finishing, this table must also include rows which reference the N/A rows in PaperTypes and Finishings,. By enforcing referential integrity in a relationship on these two columns between PaperFinishes and Prices the constraint is enforced.

noeneel --

Open the project in question, apply the Gantt Chart view, and then temporarily insert the Status Manager column.  The Status Manager column displays the name of the PM who will receive task updates for each task in the project.  When a PM creates a new project, he/she is automatically designated as the Status Manager for every task in the project.  If a different PM is set as the Owner of the project, the system does not automatically make the new Owner the Status Manager for tasks in the project.  That must be done manually by the new Owner.

HKEY_CURRENT_USER\Software\Microsoft\Office\11.0\Common\Xlstart

Set the value to match the sub new folder name e.g. Office11XLSTART. That should sort out Excel 2003. Create a similar sub folder and registry entries for Office 2010 i.e.Office14XLSTART and at HKEY_CURRENT_USER\Software\Microsoft\Office\14.0\Common\Xlstart

The same idea applies to Word’s startup and template folders.
