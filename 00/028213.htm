Ok...well I solved it.

Whilst an error was presented, it did not actually have any supporting evidence to the problem.
I was able to alter the presented error by using other functions of the Excel macro I was having issues with.

The summary of my resolve is:

I continued down the Security/Permissions route as I suspected. I found reference on a website about the errors presented that an addin maybe missing.
The means to check the state of these addins is via References from the Tools Menu within the Visual Basic Editor. As the option was greyed out, I had to stop the script running to allow access to the References. Upon attempting this, I received a new error "Unable to access registry". Thus supporting my theory.

I looked this error up via google and found reference to the user being unable to access various registry keys.
I opened REGEDT32.EXE and browsed to HKEY_CLASS_ROOT\Typelib\  I arrowed down through the list of folders and received access denied errors. I tested this on a working computer and found that 100% of these keys were accessible to non-administrative users. I then logged out of the user and in as a local administrator in attempt to change the security permissions of these registry keys, only to recieve further errors. I then logged out of th elocal administrator and in as a domain administrator and had complete success in altering the permissions of all registry keys and subsequent keys. I altered the Owner of each key to Administrators and "replace owner on subcontainers and objects". I then closed the permissions window and then removed all inherant permissions and added Everyone|full control and "replace auditing entries on all child objects with entries that apply to child objects".

Once all this had been completed, I logged out and back in as the previously affected user and tested the macro with complete success.

So in conclusion, Excel does depend on the windows registry to process macro code; regardless of what the end result is. Even something simple like copying a sheet.

greginmich say:

You’re absolutely right, Sandra; these Tech Support Scam pages generally aren’t associated with any kind of external malware that you have to hunt down with malware-removal tools. The “malware” here is usually just an active script that’s running on the scam page. The script loops a modal alert dialog. Now, a modal alert dialog normally deactivates the browser only until the user responds by clicking on the OK button, and that closes the dialog and reactivates the browser. But when the alert dialog gets looped, you can’t close it, and the browser never gets reactivated, so you can’t close the browser either.

Whenever you encounter a Tech Support Scam page just right-click on the taskbar and choose Task Manager, and then select the browser (or page) and click on the End task button. When you restart the browser, avoid the last-session recovery option, and go directly to your home page. Microsoft Edge is the exception, because you can’t avoid the last-session recovery in this browser; so you have to restart Edge indirectly after force-closing it. You can just type a short search term into the Search box to bring up a search-result page and then right-click on the search page tab and choose “Close other tabs” to remove the scam page. In the most recent version of Edge, there’s a “Don’t let this page create more messages” checkbox that appears in the alert dialog. Checking this box breaks the alert loop and allows you to close the dialog and then close the scam page:

Dialog Loop Protection for Microsoft Edge

There’s a little more information, and some screenshots, in this thread:

scottgem say:

Some data types may not translate between Excel and Access. So the solution you found was assuming that the hyperlink column in Excel was coming across as text in Access. So it was suggesting that you use the Application.Followhyperlink command. 

The instructions you posted left out some things.

First, the solution only works on a form. Second, you have to open the form in Design Mode and select the control that contains the hyperlink text. Then you open the Properties dialog (either right click on the control or select properties from the ribbon). Then select the Events tab and click the ellipses [...] to the right of the On click or On double click event. Select Code Builder and then enter:

Application.FollowHyperlink Me.controlname

Once that is done, if you click or double click (depending on the event chosen) it should open what is being linked to.
