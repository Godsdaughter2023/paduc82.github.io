charlie123cm say:

Access 2010 autocomplete field with multiple values?

Can you create a field in Access 2010 that will allow for multiple values and also autocomplete without having the check boxes. I have a field which references over 100 people and therefore, autocomplete would be much more efficient... but I need multiple values also. ken sheridan say: You almost certainly don't need a multi-valued field, and are only likely to create problems for yourself by using one.  At face value the use of a multi-valued field violates one of the basic rules of the database relational model because it would mean the table is not normalized to First Normal Form (1NF), which is defined as follows:

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple contains exactly one value for each attribute.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).

In fact, under the skin, Access maintains normalization because the data is invisibly stored not as multiple values in one row, but as individual values in multiple rows in a related table.  It makes far more sense to store the data in this way in the first place.  You don't say what entity types are involved her, other than people, so let's take a simple example of a database of documents each of which is distributed to employees within an organization.  The two principle entity types are documents and people, so these are modelled by tables along these lines:

Documents
....DocumentID (PK)
....DocumentTitle
....DocumentDate
....etc

Employees
….EmployeeID (PK)
….FirstName
….LastName
….etc

You could simply model the relationship between documents and employees, i.e. the distributions by a table with foreign key columns DocumentID  and EmployeeID which reference the keys of Documents and Employees, which would be the equivalent of a multi-valued field in Documents, but you'd then have to assign each document to each relevant employee individually.  It's likely that there will be defined groups of employees to whom it would be appropriate to distribute a document.  So to model the groups you'd have a  table

Groups
....GroupID (PK)
....GroupName

Almost certainly you'd want it to be possible for an employee to be a member of more than one group, i.e. the relationship type is many-to-many, so the relationship type is modelled bty a table in the usual way:

GroupMembers
....GroupID (FK)
....EmployeeID (FK)

A document would probably be distributed, at least in some cases, to more than one group of employees, so again the relationship type is many-to-many and modelled by a table:

DocumentGroupDistribution
....DocumentID (FK)
....GroupID (FK)

With this model it is now a simple task of distributing a document to multiple employees simply by inserting rows in to the DocumentGroupDistribution table.  So if there are 20 members of group 1 and 30 members of group 3 say, assigning a document to those groups would distribute it to up to 50 employees; not necessarily 50 in total of course as some employees might be members of both groups.

The trouble with this model, however, is that, while it is very simple, it is also very inflexible as it is only possible to assign documents to all members of one or more groups.  It is not possible to exclude one or more members of the selected groups from the distribution, not to add individual employees who are not members of the group to the distribution on an ad hoc basis.  A better solution therefore might be to revert to the solution I originally dismissed of a table with foreign key columns DocumentID  and EmployeeID which reference the keys of Documents and Employees:

DocumentEmployeeDistribution
....DocumentID
....EmployeeID

Rather than assigning a document to individual employees by inserting rows into this table one by one, though, once a document is assigned to a group, rows could be inserted into the DocumentEmployeeDistribution table automatically.  This is easily achieved with a little VBA code which inserts the rows by executing an SQL statement in the AfterInsert event procedure of a subform bound to the DocumentGroupDistribution table.  The new rows can then be displayed in another subform, so can be edited to remove specific employees if necessary, or add others who are not members of one of the selected groups.

The above hypothetical model probably won't be anything like the real world situation which your database is modelling, of course, but that's beside the point.  The purpose of describing it here is to illustrate how these sort of relationships are correctly modelled in a normalized relational database without recourse to multi-valued fields.  Not only does it not violate any of the principles of the relational model, at a more practical level it makes it much easier to query the database.

ken sheridan say: I think you'd find the standard method of modelling a many-to-many relationship type more efficient for your purposes than a multi-valued field, and you don't need to redesign things too much.  the way you'd do this is to create a table to model the relationship type between the Course Schedule and patron tables.  You don't say what the primary key of the Course Schedule table is.  It could be a composite one of Course Title, Course Date, and Course Time, but that would be cumbersome, so if you don't already have one add a CourseScheduleID autonumber column to the table, and make it the primary key.  Similarly the Patron table can have a PatronID autonumber column as its primary key.  The table to model the relationship type would then be:

CourseAttendees
....CourseScheduleID (FK)
....PatronID (FK)

The primary key of this table is a composite one made up of the two columns.  To do this, in table design view hold down the Ctrl key and left click on each of the columns, being sure to click on the 'field selector' (the small square to the left of the column name).  Then right click and select 'Primary Key' from the shortcut menu.  You should then see the key symbol against each of the columns.

For data input you can have a Course Schedule form, which must be in single form view.  With this you'd have a CourseAttendees subform in continuous forms view.  This would be linked to the parent form by setting its LinkMasterFields and LinkChildFields properties to CourseScheduleID.  It would contain just one control, a combo box set up as follows:

ControlSource:  PatronID

RowSource:     SELECT PatronID, PatronName FROM Patron ORDER BY PatronName;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first dimension is zero to hide the first column.  Change the names of the table and columns in the RowSource property to your real ones.  If the patron table has separate FirstName and LastName columns (which is better than a single name column) the RowSource property would be:

SELECT PatronID, FirstName & " " & LastName FROM Patron ORDER BY LastName, FirstName;

As the combo box is not now multi-valued you'll be able to make use of the AutoExpand property to select names in the subform by tying the first few characters, or you can select from the list with the mouse.  For each attendee per course schedule session you'd enter a new row in the subform, selecting a different student in each case.

You'd do exactly the same for instructors with another table:

CourseAttendees
....CourseScheduleID (FK)
....InstructorID (FK)

and another subform in the main Course Schedule form, set up in the same way as above.

For an example of this basic form/subform setup take a look at the file ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file shows how to use a basic form and subform for entering data into a table modelling a many-to-many relationship type.

In the same SkyDrive folder you'll fund a file NotInList.zip.  This shows how to use the NotInList event procedure of a combo box for adding new values to the control's list by inserting a new row into the referenced table.  It covers a number of different scenarios.  You could do this if you wish to enter a new patron or instructor into the relevant table simply by typing the name directly into the combo box in the subform

A more advanced use of subforms is demonstrated in the StudentLog.zip file in the same folder.  This file is intended to demonstrate the use of correlated subforms, where there are two subforms in continuous forms view  and as you navigate form row to row in one, the other is requeried to show the related records for the current row in the first subform. As it happens the file is based on a simple course registration and attendance database.  The file uses a slightly more complex method of correlating the subforms than you might find demonstrated elsewhere, involving a certain amount of code.  A simpler, but less efficient method in my experience, is to have a hidden text box control in the main form, whose ControlSource property is a reference to the key of the first subform, and whose name is used as the LinkMasterFields property the second subform.  One feature which might be of interest to you is the All Students button in the attendances subform in the Courses form.  This automatically shows all students registered for a course as attendees of the currently selected session.  The rationale behind this is that it may well be quicker to show all registered students attending and then simply delete the non-attendees for the session, rather than inserting the attendees one by one, though the latter can also be done of course.

If you do decide to normalize your tables it should not be too difficult to fill the new tables with existing data using append queries.  However, never using multi-valued fields myself, I have to confess to not knowing how the multiple values would be extracted for insertion into separate rows in the new tables.  I'm sure one of my colleagues here will know how it's done, though, so don't remove the multi-valued fields from the Course Schedule table just yet.  You can do that once the data has been successfully migrated to the new tables.
