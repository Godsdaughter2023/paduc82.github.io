ken sheridan say: The current cumulative balance per transaction is the sum of all credits less debits (or vice versa for debit balances) prior to and including the current transaction.  This can be done in the report's query.  To compute the balance in a report you can add a control to the detail section which subtracts the debit value from the credit value (or vice versa) and set its RunningSum property to 'over all'.  However, unlike a query, this will not automatically take account of the B/L brought forward at a start date for the report later than the first transaction in the account, so you would have to get this value independently, e.g. by a DSum function call, and add this to the values returned in the computed control.

You might like to take a look at Balances.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates queries for computing balances in a variety of contexts.  The most efficient method is a JOIN of two instances of the table.  The following is a query from the demo, with a start date parameter added:

PARAMETERS [Enter start date:] DATETIME;
SELECT T1.CustomerID, T1.TransactionDate, T1.Credit, T1.Debit,
SUM(T2.Credit-T2.Debit) AS Balance
FROM TransactionsCD AS T1 INNER JOIN TransactionsCD AS T2
ON (T2.TransactionDate<=T1.TransactionDate)
AND (T2.TransactionID<=T1.TransactionID
OR T2.TransactionDate<>T1.TransactionDate)
WHERE T1.TransactionDate >=[Enter start date:]
GROUP BY T1.CustomerID, T1.TransactionDate, T1.TransactionID,
T1.Credit, T1.Debit
ORDER BY T1.TransactionDate DESC , T1.TransactionID DESC;

Note how the TransactionID primary key of the table is brought into play as the tie-breaker in the event of two or more transactions occurring on the same date.

An important point to note is that, because Null propagates in arithmetical expressions, the Debit and Credit columns must not allow Nulls, so their Required property should be set to True (Yes) and  their DefaultValue property to 0.  In my own accounts' forms and reports I use conditional formatting to show zero credit and debit values in light grey, though I haven't done so in the demo.
