ken sheridan say: If all else fails you can write the rows from a query's result table directly to a file.  The following function does this, allowing for flexibility with the regard to the choice of delimiter and whether text values are enclosed in quotes or not:

Function ExportToText(strQuery As String, _
                    strExportTo As String, _
                    strDelim As String, _
                    blnQuoteText As Boolean, _
                    Optional blnHasFieldNames As Boolean = True)
                    
'Accepts:
' strQuery - Text - name of query to be exported
' strExportTo - Text- path to file to export to
' strDelim - Text - delimiter character(s) to separate fields
' blnQuoteText - Boolean - True to enclose text or memo data in quotes
' blnHasFieldNames - Boolean - (optional) - True (default) to export field names as first line

   Dim dbs As DAO.Database
   Dim rst As DAO.Recordset
   Dim qdf As DAO.QueryDef
   Dim prm As DAO.Parameter
   Dim Fld As DAO.Field
   Dim n As Integer
   Dim strPrintList As String
   Dim strQuote As String
   
   Set dbs = CurrentDb
   Set qdf = dbs.QueryDefs(strQuery)
   
   ' evaluate query's parameters, if any
   For Each prm In qdf.Parameters
       prm = Eval(prm.Name)
   Next prm
    
   Set rst = qdf.OpenRecordset
   
   If Dir(strExportTo) <> "" Then
       If MsgBox("Overwrite " & strExportTo & "?", vbQuestion + vbYesNo, "Export Query") = vbYes Then
           Kill strExportTo
       ElseIf MsgBox("Append rows to " & strExportTo & "?", vbQuestion + vbYesNo, "Export Query") = vbNo Then
           Exit Function
       End If
   End If
           
   With rst
       If Not (.BOF And .EOF) Then
           Open strExportTo For Append As #1
           
           If blnHasFieldNames Then
                ' include column headings in text file
                For n = 0 To qdf.Fields.count - 1
                     strPrintList = strPrintList & strDelim & qdf.Fields(n).Name
                Next n
                ' remove leading delimiter
                strPrintList = Mid$(strPrintList, Len(strDelim) + 1)
                Print #1, strPrintList
                strPrintList = ""
           End If
           
           Do While Not .EOF
               For n = 0 To qdf.Fields.count - 1
                   Set Fld = .Fields(n)
                   strQuote = IIf(blnQuoteText And (Fld.Type = dbText Or Fld.Type = dbMemo), """", "")
                   strPrintList = strPrintList & strDelim & strQuote & _
                   .Fields(n) & strQuote
               Next n
               ' remove leading delimiter
               strPrintList = Mid$(strPrintList, Len(strDelim) + 1)
               Print #1, strPrintList
               strPrintList = ""
               .MoveNext
           Loop
           Close #1
       End If
   End With

Exit_Here:
   rst.Close
           
End Function

For example the following query:

SELECT FORMAT(Contacts.ContactID,"0000") AS ContactID,
FirstName, LastName
FROM Contacts
WHERE Lastname = "Sheridan";

saved as qrySheridans, can be exported as follows, writing the column names as the first row, using a comma as the delimiter and wrapping the text values in quotes for instance, with:

ExportToText "qrySheridans","C:\Users\Sheridan\Documents\Sheridans.csv",",",TRUE,TRUE

giving the following file:

ContactID,FirstName,LastName
"0004","Gavin","Sheridan"
"0005","Martha","Sheridan"
"0022","Sarah","Sheridan"
"0247","Ken","Sheridan"
"0253","Ian","Sheridan"
"0367","Fiona","Sheridan"

in which the first column is formatted to 4 digits with leading zeros.  Whether the ContactID is a number, as in my case, or a text data type is immaterial.

To omit the quotes characters for instance, which is why the function was written in the first place, would be:

ExportToText "qrySheridans","C:\Users\Sheridan\Documents\Sheridans.csv",",",FALSE,TRUE

giving:

ContactID,FirstName,LastName
0004,Gavin,Sheridan
0005,Martha,Sheridan
0022,Sarah,Sheridan
0247,Ken,Sheridan
0253,Ian,Sheridan
0367,Fiona,Sheridan

To omit he column headings and quotes for instance:

ExportToText "qrySheridans","C:\Users\Sheridan\Documents\Sheridans.csv",",",FALSE,FALSE

giving:

0004,Gavin,Sheridan
0005,Martha,Sheridan
0022,Sarah,Sheridan
0247,Ken,Sheridan
0253,Ian,Sheridan
0367,Fiona,Sheridan

strQuote = IIf(blnQuoteText And (Fld.Type = dbText Or Fld.Type = dbMemo), """", "")

or add an additional line to enclose date/time with whatever character is desired.

As it stands date/time values would be exported unenclosed in whatever regional format is set for the system.  Alternatively, you could export the date/time value as text by calling the Format function in the query, as with the ContactID column in my example.

hansv mvp say:

If someone creates a formula in Excel 2007 or later that uses IFERROR, and if you open the workbook with that formula in Excel 2003 or earlier, Excel will display _xlfn.IFERROR to indicate that (1) it doesn't support the function, and (2) it can see that it is a built-in function from a later version.

If you have a formula

=_xlfn.IFERROR(expression, replacement_value)

you can replace it with

=IF(ISERROR(expression), replacement_value, expression)

This will work in all versions of Excel.

daniel g. grau say:

Things that can increase CPU usage:

  • Tracked changes (turn off or accept all changes to test)
  • AutoRecover (turn off in Word > Save > Preferences)
  • Corrupt font cache (restart Mac with Shift pressed—boot time will be longer—, then restart as usual)
  • Duplicated fonts (locate and disable with Application > Utilities > Font Book; while you are there also validate your fonts and disable any faulty ones)
  • Scrolling in very long and complex documents (test with simpler documents)
  • Wrong permissions in folders or files used by Word (open Application > Utilities > Disk Utility and repair disk permissions)

… plus several more things that I'm sure I can't recall at the moment.

Unfortunately, the brainiacs at Microsoft have labelled the Text Form Field with the words Text Box, which is also the name of Text Box on the Insert tab, but they're completely different objects.

A form can have protected and unprotected sections. If a form field (Developer>Text Box) is in an unprotected section of a protected form, it can be deleted. If a text box (Insert>Text Box) is anchored to an unprotected section, it can be deleted.

I've never seen an instance of a form field in a protected section being deletable when the form is protected.
