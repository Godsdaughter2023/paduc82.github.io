HTTP 206 Partial Content responses and multiple Content-Length headers - Microsoft Community

bnaberhuis say:

HTTP 206 Partial Content responses and multiple Content-Length headers

Apologies if this is not the relevant place to make a bug report, but I have been unable to find a more proper location to do so.  

When IE 11 makes a request for continued content from some resource that didn't download all the way (or was truncated for some other reason), it makes a request for the rest of the object, which is reasonable.  

What is unreasonable is the method that bing.com's servers respond to such a request: it has 2 Content-Length headers attached to it.  The first one holds the length of the entire object, and the second one holds the length of the data that is in the body that follows.  This tends to play havoc with various proxy servers / ICAP server combinations.  

RFC 7230 explicitly states that multiple Content-Length headers should cause a browser / proxy / etc. to outright reject any message that has 2 Content-Length headers (for security reasons; there are a number of exploits that use this specific ambiguity in RFC 2616 to deliver malware).

Ideally, IIS should never attach 2 such headers to a request; it causes wastes of bandwidth and (admittedly) some small amount of time.

marshall barton say:

No,  The field size is determined by the number type in the properties.  E.g  Integer is a field size of 2 bytes, Double is 8 bytes, etc.  Only the Text (and Decimal) data type allows you to specify the number of bytes the field can use.  Other datatypes, such as Memo, have no field size and are only limited by the max size of the database file (2 GB).

OTOH, you gain nothing by limiting a Text field to less than 255 bytes because the field only takes up as much space as needed (up to 255) to hold it text string.  So you should leave Text fields at size 255, unless you have a specific data requirement from a business rule for a smaller limit on the number of characters (e.g. soc sec # must be 9 characters)

Solver doesn't look like the rest of the dialogs in Office 2016 for Mac because it is actually a custom VBA UserForm, and not a standard OS X dialog. This means that most of the inconsistencies that you're seeing in the Solver UI are a side-effect of the underlying Visual Basic code (which for the most part needs to stay consistent across versions of Office). That isn't meant to be an excuse for the design, but hopefully the context makes more sense. I'm also not seeing the white outline from your screenshots in our latest version (15.22). As for any other non-Retina assets you find in the product, please report them through the in-app feedback - these fixes are a priority to us and the changes are usually as simple as upconverting the existing icon and swapping them in.

We also shouldn't be adding trailing spaces to formulas, since that would be changing your data and we're committed to the integrity of your documents.  It's possible that there visually seems to be a trailing space, but we've included the TRIM function for almost a decade because we know how painful debugging whitespace can be :)

=============

Symptom:

When you try to access OneDrive for Business in Office 365 portal, stuck at “Sorry for the wait. Your OneDrive is almost ready”.

Root cause:

When we assign SharePoint Online license to an user, the user will be synced from the Azure Active Directory Service to the SharePoint Online Active Directory Service. After that, user can use OneDrive for Business service.

In your case, a syncing issue occurred in this process, so you cannot access OneDrive for Business.

Solution:

We have triggered a new syncing process from backend to resolve this issue.

genemiao_msft6664 say:

Hi Arjun,

Here are the steps to verify a domain:
1. Sign into Office 365 admin portal with your work or school account.
2. Click DOMAINS > Add a domain, then type in your domain name and click Next.
3. Then, you will see a TXT record. Add the TXT record to your domain hosting provider and save it. If the domain host provider doesn't support TXT records, click Use an MX record instead, and add the MX record to the domain hosting provider and save it.
4. Click Okay, I've added the record, then Office 365 will verify the domain.
5. When Office 365 finds the correct TXT or MX record, the domain is verified.

Verify your domain and change nameservers
