jay freedman say:

The lists certainly were created using a style. The way you'll know it is this: If styles aren't used, then right-clicking the AutoTextList field will display used to create the list.

The other way to see what was done -- and the only way to know the name of the style that was used for a particular list -- is to examine the field code of its AutoTextList field. In the template or in a document based on the template, press Alt+F9. (This is a shortcut for going to the Advanced page of the Options dialog and checking/unchecking the "Show field codes instead of their values" option.) You'll see something like this:

{ AUTOTEXTLIST "Name" \s "Names" \t "Right-click to select name" }

The text in quotes that comes after the \s -- in this case, Names -- is the style that was applied to each item when it was made into an AutoText entry. (Press Alt+F9 again to change the display back to the field value.)

If the AutoText entries are stored in the Normal.dotm template, they would be available in all documents on your computer, regardless of which template a particular document was based on. However, they wouldn't work on anyone else's computer unless you copied the entries to their Normal.dotm. While that's possible, it's a bit fiddly and almost impossible to maintain if the list changes.

If the AutoText entries are stored in the individual templates used to create documents, instead of the Normal.dotm template, then the templates are all independent of each other. You cannot "break" a working template by putting other entries into a different template, and updating a template for all users is as simple as copying it to each computer.

To control which template stores an AutoText entry at the time you create the entry by pressing Alt+F3, open the "Save in" dropdown in the dialog box and select the proper template before clicking OK.  If you need to copy it later to a different template, go to Insert > Quick Parts > Building Block Organizer, select the entry, click the Edit Properties button, and change the Save In dropdown.

macropod say:

For the basic formatting, you could use:

{={MERGEFIELD Percentage}*100 \# "0%;-0%"}

or:

{={MERGEFIELD Percentage}*100 \# "00%;-00%"}

The second form will preserve leading 0s for amounts less than +/- 100%. The use of such fields is standard practice with mailmerges. Unless you reformat the source data to express the percentages as whole numbers, not decimals, the only way of avoiding the need for the switches is to use the old DDE connection for the mailmerge connection. DDE, though, is severely limited in what it can do compared to the default OLE DB method.

It's also not clear what you want to do regarding amounts equal to or grater than +/- 100%.

Note: The field brace pairs (i.e. '{ }') for the above example are created in the document itself, via Ctrl-F9 (Cmd-F9 on a Mac); you can't simply type them or copy & paste them from this message.

    ken sheridan say: The fact that you have created multiple tables solely on the basis of the excessive number of columns does point to a badly structured database.  The structure of a database should reflect the entity types in the real world being modelled by each table and the relationship types between them.  It would be very unusual for a table which correctly models a real world entity type to need more columns than can easily be encompassed by Access.  However. let's leave that particular can of worms unopened for the moment.  

    From your reference to a LEFT OUTER JOIN, my guess is that the subforms which have queries as their RecordSource are including the referenced Clients table in the query.  This is unnecessary.  Each subform should be based either on the referencing table only, whether that be as the table itself or as a query, e.g. where you want to return computed columns in the subform.

    The fact that some rows in the Clients table will not have matching rows in some of the subforms is immaterial.  All rows from the Clients table will still be returned in the parent form.

    The error which you are experiencing could point to the ClientID column being the primary key of the subform's tables.  If more than one row in a  referencing table on which a subform are is based the can reference a single row in the Clients table the ClientID column is a foreign key and should accept multiple values.  Consequently it should not be the primary key, but should be indexed non-uniquely (duplicates allowed).

    The only circumstance in which a referencing table's primary key is also a foreign key referencing the primary key of a referenced table is where the second table models a subtype of the first table, i.e. the model is a type hierarchy.  You'll find an example in TypeHierarchy.zip in my public databases folder at:

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

    If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

    This little demo file illustrates how a type hierarchy of personnel in an academic institution can be modelled.  In the demo Employees and Students are sub-types of Personnel, and AcademicPersonnel, AdministrativePersonnel and WagedPersonnel are subtypes of Employees.  A subtype is characterised by sharing all attributes of its (super) type but not those of other subtypes.

    As regards the subject of data modelling in general in the context of a relational database you might like to take a look at a few other files in the same OneDrive folder.  One is DatabaseBasics.zip, which is a general introduction to the design of relational database in Access.  Another is Relationships.zip, which builds on some of the basic concepts outlined in DatabaseBasics, and illustrates how relationships are built up across a database.  Finally, Normalization.zip is an outline of the process of normalization in as simple terms as possible, but no more so.  Normalization is the process by which redundancy and the consequent risk of update anomalies is eliminated by the decomposition of a table into a set of normalized related tables.  The process is a formal one, subject to formal rules, by which a table is progressively put into increasingly rigorous 'normal forms'.  A good understanding of normalization to at least Third Normal Form (3NF) in the first instance is essential for anyone attempting to design a relational database.  Most tables normalized to 3NF will also be normalized to the higher normal forms without further decomposition.
