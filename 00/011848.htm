ken sheridan say:

You'll find examples of queries to return balances in various contexts in balances.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

The following query using the Transactions table from this little demo file will return a result table in the format you wish:

SELECT FORMAT(TransactionDate, "mmmm-yy") AS [Transaction Month],
CustomerID, SUM(TransactionAmount) AS [Total Monthly Credit],
DSUM("TransactionAmount", "Transactions", "CustomerID = " & [CustomerID] &
    " AND FORMAT(TransactionDate, ""yyyymm"") <= " & FORMAT(TransactionDate, "yyyymm"))
AS Balance
FROM Transactions
GROUP BY CustomerID, FORMAT(TransactionDate, "yyyymm"),
FORMAT(TransactionDate, "mmmm-yy");

But the following, using a subquery will do the same, and is likely to be more efficient:

SELECT FORMAT(T1.TransactionDate, "mmmm-yy") AS [Transaction Month],
T1.CustomerID, SUM(T1.TransactionAmount) AS [Total Monthly Credit],
    (SELECT SUM(T2.TransactionAmount)
     FROM Transactions AS T2
     WHERE T2.CustomerID = T1.CustomerID
     AND FORMAT(T2.TransactionDate, "yyyymm") <= FORMAT(T1.TransactionDate, "yyyymm"))
AS Balance
FROM Transactions As T1
GROUP BY T1.CustomerID, FORMAT(T1.TransactionDate, "yyyymm"),

FORMAT(T1.TransactionDate, "mmmm-yy");

john w. vinson say:

I agree that the size of the table is very likely the problem, especially since your criterion is using the Format function - defeating any indexes on the table. You might get better results applying the criterion directly to the date field, assuming that IT is indexed. Try

SELECT FORMAT(TransactionDate, "mmmm-yy") AS [Transaction Month],

That will at least use the index in the DSum, but it will still require a function call for every row. You might need temp tables!

bill manville say:

The use of multiple sheets containing similar information usually worries me, because of the difficulty of keeping them in step, as you describe.

But, to answer your question, if you want B1 to B100 in Sheet B to always refer to A1 to A100 in Sheet A  even if rows have been inserted or deleted,

Sheet B!B1: =OFFSET('Sheet A'!$A$1,ROW()-ROW('Sheet A'!$A$1),0)

and copy down to B100.

That will work even if someone inserts a row before row 1 in Sheet A, but not if someone delete's row 1

doug robbins - mvp office apps & services (word) say:

Do you have an = separating the fields?

 

You will get an error if there is a mis-match between the field separator used and the Separator:= in the code.

 

Here if I use = as the field separator, sorting on Field 2 works with your code.  However, if I use

Separator:=vbTab and the fields are separated by =, a 9125 error occurs.  If is separate the fields with a tab space then the code using Separator:=vbTab works.

 

The reason why you can sort on Field 1 with a mismatch between the field separator that is use and that in the code is because the string does not then have to be parse to obtain the field on which to sort.
