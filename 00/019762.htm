Error in Rank function when ranking values calculated from sums of - Microsoft Community

*.pjp say:

Error in Rank function when ranking values calculated from sums of reciprocals

The rank function does not work when ranking floating point values calculated using a sum of reciprocals formula.

For example,

3,  4,  5               
4,  3,  5
5,  4,  3

if you add the sums of the reciprocals in each row (1/3 + 1/4 + 1/5, etc.) you get a decimal value (.7833333 in this case).
Since the components of each row are the same integers, you get exactly the same value for each sum.  This can be verified using any combination of
IF, =, EXACT, <, >, etc.   In any formula I can dream up to compare the resulting sums, all three sums are identical, with the exception of using the rank formula.

if you attempt to rank the three sums, the answer should always be 1 for each rank since the values are equal.

e.g.,

values            sum            rank
3,  4,  5        .7833333       1
4,  3,  5        .7833333       1
5,  4,  3        .7833333       1

But instead, the row that has the "3" in the first position will be ranked as 3 instead of 1.

No amount of rounding in any order will change this behavior.

I've found many other examples where this occurs in slightly different ways.

I have a sample spreadsheet with many examples of the error and a few examples where it works as expected, but I don't see a way to upload that spreadsheet here. Please let me know how I can submit the spreadsheet. 

thanks,
Philip
Windows XP SP3, Office 2010 Professional Plus, version 14.0.4760.1000 (32-bit)

RANK is returning the correct result for your example.  The sum of the 3rd row is indeed infinitesimally greater than the sum of the 1st and 2nd rows.  The sum of the 1st and 2nd rows are indeed exactly the same.  But that is a coincidence that we cannot rely on.

Philip wrote:
Since the components of each row are the same integers, you get exactly the same value for each sum.  This can be verified using any combination of IF, =, EXACT, <, >, etc.   In any formula I can dream up to compare the resulting sums, all three sums are identical, with the exception of using the rank formula.

Try dreaming up the following comparison.  Assuming your values are in column A, B and C starting in row 2, and the sum is in column D, enter the following:

E3:  =IF(D3-D2=0,"exact","different")
E4:  =IF(D4-D3=0,"exact","different")

Even better:

F3:  =D3-D2-0
F4:  =D4-D3-0

formatted as Scientific with 1 or more decimal places.

You will find that E4 is "different".  And F4 is about 1.11E-16, exactly 2^-53, which demonstrates that D4 is indeed larger than D2 and D3.

Once you realize that D4 is larger the D2 and D3, perhaps everything will be clear.

But to demonstrate, try putting 1, 1 and 2 into A6, A7 and A8; and put the formula =RANK(A6,$A$6:$A$8,1) into B6 and copy it into B7 and B8.  You will see that B6 and B7 are 1, and B8 is 3.  Read the RANK help page for the explanation.

Now, there are a lot misconceptions to explain.  I will work backwards....

Philip wrote:
Since the components of each row are the same integers, you get exactly the same value for each sum.  This can be verified using any combination of IF, =, EXACT, <, >, etc.

You were fooled by the dubious heuristic poorly described under the misleading title "Example When a Value Reaches Zero" at http://support.microsoft.com/kb/78113 (click here).

In a nutshell, in some (emphasize some) circumstances, Excel arbitrarily decides to replace the true arithmetic result with zero or the true comparison with "equal".  It is well-intentioned, but poorly designed and executed.  I cannot begin to tell you how "dubious" (read:  bad) this heuristic is.

Consequently, =D4-D3 returns exactly zero, misleading you into thinking they are equal.  But =D4-D3-0 returns 2^-53, demonstrating that they are indeed not equal.

"Huh?!  How could subtracting zero change the result?", you say, demonstrating more wisdom than a Microsoft engineer.  There is no intelligent answer for that [2].  The implementation is just plain "stoopid", IMHO.

Philip wrote:
Since the components of each row are the same integers, you get exactly the same value for each sum.

You are correct mathematically.  But computer arithmetic does not follow the same laws.

In particular, all computer arithmetic is eventually limited by some finite precision.  In the case of Excel, it is limited to 53 bits (64-bit floating-point).  Complicating matters, arithmetic is actually performed with the precision of 64 bits (80-bit floating-point); then it may or may not be rounded to 53 bits between operations.

Consequently, the exact result of a sequence of arithmetic operations is often dependent on the order of operations. 
Whether or not you understand that purposely incomplete and vague explanation, the fact remains:  the sums of the same components in different orders are not necessarily exactly the same.  And they are not exactly the same in all 3 instances in your example.

You neglect to say how you computed the sum.  Several ways come to mind.  (See the assumptions of your data layout above.)  In D2:

1. =1/A2 + 1/B2 + 1/C2
2. =SUM(1/A2, 1/B2, 1/C2)
3. =SUMPRODUCT(1/A2:C2)

Whichever you used -- or something else entirely -- it is copied into D3 and D4.

Each of those formulas presents a different opportunity for Excel to implement the sum slightly differently, potentially allowing the entire sum to be calculated with 64-bit precision (i.e. 80-bit floating-point).

(However, even the precision of 80-bit floating-point is no guarantee of equality.)

Actually, I tested all 3 formulas, and they all have the same result:  an infinitesimal difference in the 3rd row.

But to demonstrate the point, try the following:

4. =vbsum(A2, B2, C2)

doug robbins - mvp office apps & services (word) say:

The following code will not insert a space if the cross reference is followed by either a period or a space, so if it is run a second time, it will not add another space.

Dim i As Long
Dim rng As Range
With ActiveDocument
    For i = 1 To .Fields.Count
        With .Fields(i)
            If .Type = wdFieldRef Then
                Set rng = .Result
                rng.End = rng.End + 2
                rng.Start = rng.End - 1
                If rng.Text <> Chr(46) And rng.Text <> Chr(32) Then
                    rng.InsertBefore " "
                End If
                .Result.Font.Italic = True
                .Result.Font.ColorIndex = wdDarkBlue
            End If
        End With
    Next i
End With
