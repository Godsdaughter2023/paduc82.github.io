Stop playing after 2 slides in 2007 means that if the audio starts on slide 1, if you hyperlink from slide 1 to slide 3, the music will stop. You can jump from slide 1 to slide 2 forever and it won't stop.

In 2010 and 2013 it means that after two transitions it will stop. You could link from slide 1 to slide 100 hundred and it would play. However, if you go from slide 1 to slide 2 and back to slide 1 it will stop like the problem you have now.

I unfortunately haven't thought of a way to get that functionality back in newer versions.

I'm using several CBs on a form called Orders to show info according to the cb selected option which in turn calls a Query called Orders_qr.

I fill the cb up with data from a column, let's say product.

The following SQL works fine with a TextBox Value, but not with the ComboBox Value since it returns a number.

WHERE (((Product.CarMake) Like "*" & [Forms]![Order Search]![CarMake] & "*"));

In the Design View of the Select Query, I tried to solving the index problem by adding the Text Property to get the Option text:

But it gets transformed into this:

hansv mvp say:

If the Row Source of the combo box has two (or more) columns, and if the first column has been hidden by setting its column width to 0, the value of the combo box will (usually) be that of the hidden first column, while the display value is that of the second column. You could try

WHERE Product.CarMake Like "*" & [Forms]![Order Search]![CarMake].[Column](1) & "*";

Columns start counting at 0, so Column(0) is the first column, Column(1) is the second column etc.

henry habermacher say:

To reference the value of a ComboBoy that is bound to the ID you can use the columns() property. Give this a try (assuming the ID is in Column 0 and the displayed column is Column 1.

WHERE (((Product.CarMake) Like '*" & [Forms]![Order Search]![CarMake].Columns(1) & "*'));

HTH

Henry

scottgem say:

Bert,

While using the Column should get you what you want, I suggest its not the right way to go. If your database is designed properly, then your table should NOT have text in the foreign keys. Lets look at your first example the Car Make. I will assume you have a table of Car makes that has an autonumber ID field and a text description. So in your orders table you have a foreign key field for car make that is populated with ID field from the car makes table. Now in your Orders query, you are joining your orders table to the Car makes table on the ID field. So you can add the text field from the car makes table to your query. This is all good and SOP.

But, what you should also be doing is adding the ID field to the query as well. And you should be setting the criteria of THAT column to your combo:

WHERE (((Product.CarMake) = [Forms]![Order Search]![CarMake]));

You don't need to use Like, since you are using a lookup table to make sure input is standaridized. If you are using the Query Design grid, you can uncheck the Show box, if you don't want the ID field to show in the query results or just include the field in the WHERE clause if you are typing in the SQL. 

To summarize; you should use lookup tables to standardize input. You should include a numerical ID field as the PK of your lookup tables. Your comboboxes should return the numerical ID, but not display it. So when filtering a query, you filter on the numerical ID rather than the text.

For an IMAP migration, a maximum of 500,000 items can be migrated from a user‚Äôs mailbox, and the maximum message size that can be migrated is 35 MB.
