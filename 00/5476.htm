Using a macro there is an Open Query Action You can add as many of these actions as you need. There is also a Set Warning action. You would have that as your First action and set it to False, then have another as the last action and set it back to True.

ken sheridan say: A Description column in the sales orders table is almost certainly redundant as it is not functionally determined solely by the whole of the table's key.  The table is consequently not normalized to Third Normal Form (3NF), so is open to the risk of update anomalies.  In the data entry form for sales orders bind a combo box control to the foreign key PartNumber column and set up the control as follows:

Name:                cboPartNumber

ControlSource:    PartNumber

RowSource:        SELECT Description, PartNumber FROM Products ORDER BY PartNumber;

BoundColumn:    2
ColumnCount:     2
ColumnWidths:   0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

Add an unbound text box to the form with a ControlSource property which references the hidden column of the combo box:

=cboPartNumber.Column(0)

The Column property is zero-based, so Column(0) is the first column.

The exception to this would be if the Description property were not time-independently determined by the part number, i.e. if the description in the Products table might change over time, but the description associated with each sales order should remain static as that current when the order was created.  In this scenario the sales order table should have a Description column, with a bound text box in the form, and the current value should be assigned to it in the cboPartNumber control's AfterUpdate event procedure.  You'll find an example in InvoicePDF.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the unit price and current VAT rate of the selected product are assigned to the columns in the InvoiceDetails table in the AfterUpdate event procedure of the ProductID combo box in the invoice details subform.

I'm surprised that you say you have two tables, rather than three.  Normally in this sort of situation, there is a many-to-many relationship type between products and orders, which is modelled by a third table such as the InvoiceDetails table in my demo.  Otherwise each sales order can only be for one product.

scottgem say:

Please review my blog (see address in sig) on Displaying Data from related tables. 

What Ken and Duane are explaining is that it is a principle of relational databases that data should exist in one place and one place only. Its not the Description that is redundant, its storing the Description in two different tables that is redundant..

The exceptions to this are foreign keys and time sensitive data (like prices). A Foreign Key is a field that stores the value of the primary key of the related table. So all you need in your Sales table is the Part reference as the foreign key. You can then link back to the part record and pull any other information you need from that table.

Another point here is you say; "each item on a sales order is a separate "job" in the factory so each needs a separate record and print out"  If a sales order has multiple items, then you need two tables; one for the info about the sale and one for the info about each item in the sale. You can still create separate reports for each line item if that is what is needed by the factory.

One last point. Correct table design is the foundation of a well run database. And there are a set of rules that govern that design. Those of us who have been designing databases for a long time strongly advocate adhering to those rules.

I think you may be making the very common beginner's mistake of assuming that all of the information in a printout must be in one table. That is NOT the case!

Don't confuse data storage with data presentation. They are two different tasks. As the guys have said, your tables should be designed with the "Grandmother's Pantry Principle": "A place - one place! - for everything, everything in its place." Your summary printout should be based - not on the order table - but on a Query joining the Order table, to the OrderDetails table, to the Parts table; this query will draw the needed information from all three tables onto one sheet of paper.

Taking my Invoice PDF demo, to which I gave you a link earlier, as an example, if we firstly look at the database's logical model:



You'll see a number of things, including:

1.  The description of each product is stored in the Product column of the products table only.  It maps to each invoice line by the relations ship between Products and InvoiceDetails on the ProductID columns.  This is because the product description is always the same, so there is no need to store it in InvoiceDetails.  That would be redundancy, and would put the data at risk of update anomalies in that a different description for the same product could exist in each table.  Even if the Product value form Products were to be inserted into InvoiceDetails when a new row is inserted into the latter, there is always the possibility that one or the other could be changed subsequently.  

2.  The customer address columns, on the other hand, are stored in both the Customers and the Invoices tables.  This is because a customer might change their address, so the address data in Customers would be changed to their current address.  However, the address data for each invoice remains that which was current when the invoice was raised.  This is a policy decision.  It might have been decided that all invoices, past and present, should show the current customer address, as was done with the product description, in which case the Invoices table would not include any address columns.

Consequently there is no redundancy in the database, and there can never be any update anomalies which would result in inconsistent data

When it comes to showing the data from the tables this is done by means of a query which joins them:



In this, columns from all of the tables are returned, so the product description from Products is returned  as well as the invoice date from Invoices for instance.  This query is the basis for the invoices report.  If you download the demo you'll see this for yourself.
