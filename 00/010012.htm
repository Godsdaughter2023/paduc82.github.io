I would investigate the File Type and Creator codes on those files.  See http://support.apple.com/kb/TA25699

I don't remember the codes now: one is WBBN and MSWD, I can't remember the others.  These codes have not been used in OS X since OS X came out, and are supposed to be ignored by Word.  But in some cases, Word does not ignore them.

There are three kinds of ".doc" file:  I know them by their PC descriptions of Word 2, Word 6, and Word 97.  Respectively eight-bit, 16-bit, and 32-bit binaries.  Files in the Word 2 format can no longer be opened by MS Office on the Mac, files in Word 6 format should produce the error you are seeing.  The file type is blocked because it was judged to be too expensive to validate the 16-bit import converter to make it safe from the modern internet nasties.

If you open or save such a file in PC Word, it will not upgrade the format unless you force it.

It would seem that Zamzar is simply re-writing the File Type and Creator codes to the modern equivalent, which is supposed to be ignored anyway.  You should be able to accomplish the same thing by forcing a save to .docx in Word 2010 on the PC.  That would upgrade the code to 32-bit and at the same time re-express it in XML, which would come in through the XML import filter that does ignore the codes.

It is not unknown for external applications to "impose" File Type and Creator codes on files that did not originally have them (WebMail interfaces provided by the various freemail purveyors are legend for doing this).  Often, they choice the old 16-bit codes that are inhibited.

I understand that it's relatively trivial to write an AppleScript routine to go through and look for outdated OS 9 File Type and Creator codes, and substitute them for the modern equivalents, which theoretically will solve the problem.

Sorry I can't provide specific technical detail.  But that's my understanding of what is going on.  The cure would be to either apply the latest codes, or remove them entirely.

john - project say:
AsathMuhammathu,
In project management an S-curve is simply a graph of cumulative data over time. For example, let's say you have an overall project budget and you want to graphically see how your ongoing project is performing towards that budget. If you plot cumulative cost over time, you can see if costs are on track to gradually reach the budget or if costs are accelerating such that they likely will exceed your budget.

Unfortunately no version of Project will create and display an S-curve directly. However there are multiple ways to get the necessary Project data into Excel where that data can be plotted as an S-curve. One method is to copy and paste the Resource Usage or Task Usage view data from Project to Excel. This is a manually intensive approach that is prone to error and there are better ways to get there. A better method is to use the visual reports feature. This is more automated but still requires the user to manipulate the data in Excel. Perhaps the best approach is through VBA. Project data can be exported to Excel and then automatically plotted in the desired S-curve format. In Rod Gill's book on Project VBA programming, Rod has a sample macro for producing an S-curve. For more information on Rod's book go to, http://www.project-systems.co.nz/project-vba-book/index.html.

john mcghie say:
If you have an Exchange Server, "Yes", otherwise, "Not Really".

If the task is created within an Exchange Server account, you can forward or re-assign it to another Exchange Server account.  Both users need to be connected to the same Exchange Server.

The theory is described here:
http://office.microsoft.com/en-au/outlook-help/create-and-track-assigned-tasks-HA001229368.aspx

If you do not have an Exchange Server, you can still forward tasks to an email address as an attachment.  But Outlook on its own cannot keep track of the task for you.

To email a task in Outlook for Mac, drag it to your desktop.  Outlook will convert it to an .ics file, which you can attach to an email.
