FWIW, the following macro will search the active document for chemical formulae, and power expressions, superscripting/subscripting the numbers as appropriate. Thus, C5H8(N2S)4 becomes C5H8(N2S)4, whilst 3(CaO)•2(SiO2)•4(H2O)(gel) + 3Ca(OH)2 becomes 3(CaO)•2(SiO2)•4(H2O)(gel) + 3Ca(OH)2, and 100m2 becomes 100m2. Unless you're working with isotopes, the results should be correct - you'll need to apply the isotope superscripting yourself (if the numbers are already superscripted, they’ll be left alone).

Hmm - the forum software here messes with the subscripts, especially, making them appear as superscripts.

If your document has other alphanumeric strings in which a non-superscripted number follows a letter (eg Table cell references), you’ll need to select only the range(s) containing the text to be converted and answer ‘No’ to the prompt.

Sub ChemPwrFmt()
Application.ScreenUpdating = False
Dim RngSel As Range, RngNum As Range, RngTmp As Range, bState As Boolean
Select Case MsgBox("Do you want to process the whole document?", _
    vbYesNoCancel + vbQuestion, "Chemical/Power Formatter")
  Case vbYes
    bState = True
  Case vbNo
    bState = False
  Case vbCancel
    End
End Select
Set RngSel = Selection.Range
With ActiveDocument.Range
  If bState = False Then .Start = RngSel.Start
  With .Find
    .ClearFormatting
    .Text = "[A-Za-z)][0-9]{1,}"
    .MatchWildcards = True
    .Wrap = wdFindStop
    .Forward = True
    .Execute
  End With
  Do While .Find.Found = True
    Set RngNum = .Characters.Last
    If bState = False Then
      If RngNum.InRange(RngSel) = False Then Exit Do
    End If
    Set RngTmp = .Words.Last
    RngTmp.MoveStartUntil " ", wdBackward
    If LCase(RngTmp.Text) = RngTmp.Text Then
      RngNum.Font.Superscript = True
    ElseIf RngNum.Font.Superscript = False Then
      RngNum.Font.Subscript = True
    End If
    .Collapse Direction:=wdCollapseEnd
    .Find.Execute
  Loop
End With
RngSel.Select
Set RngTmp = Nothing: Set RngNum = Nothing: Set RngSel = Nothing
Application.ScreenUpdating = True
End Sub

hi, there i have resolved the problem.

Things that didn't work. 

  • Update to latest build from Windows update (same error)
  • Download any windows technical preview ISO (same error)
  • Windows refresh (New Error - : Windows cannot be refreshed because you are on windows to go)

This gave me a new angle to work on, and suddenly the Original errors we have seen about trying to install windows on a USB drive makes sense, windows has mislabeled itself as a to go installation.

Now for the fix:

To change the PortableOperatingSystem registry, follow these steps:

  1. Open Registry editor (regedit.exe)
  2. Locate and then select the following registry subkey: 

    HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control
  3. From the right-side pane double-click on PortableOperatingSystem Dword value
  4. Change the value data to 0 (The default value is 1)
  5. Close the registry editor
  6. Restart the computer.


You can also you the command line option to make the change. Run the below command from an elevated command prompt:

reg add HKLM\SYSTEM\CurrentControlSet\Control /v PortableOperatingSystem /t REG_DWORD /d 0

this was found here: https://support.microsoft.com/en-us/kb/2778881

i used the cmd way to do it and restarted the machine, downloaded from windows update again and could finally upgrade to build 10041.

Hopefully this will be useful to someone else in the future.

Mailboxes - Unable to Create O365 Mailbox after Restoring Original AD - Microsoft Community

alex rizeanu msft say:

Mailboxes - Unable to Create O365 Mailbox after Restoring Original AD User Account

Technical Level : Intermediate

Summary

Consider the following scenario:

An administrator migrates a mailbox to Office365 using the hybrid remote move migration process.  Some time passes and something happens with the end user - they leave the company for instance.  The administrator removes their account from Office 365 using directory synchronization but retains it in local on-premises Active Directory - for example, they move the account into a non-syncing OU.  The mailbox was not placed on hold first, so after 30 days it is purged from the Exchange Online service.  The end user then returns to the company afterwards, and the administrator pulls their user account from the non-syncing OU to place them back into synchronization, which copies it back to Office365.  After applying an Exchange license to the account, the administrator finds that there is no mailbox provisioned for this account.  They also find that the following warning is displayed:

"This user's on-premises mailbox hasn't been migrated to Exchange Online.  The Exchange Online mailbox will be available after migration is completed."


Details

CAUSE

This is by-design.  When you synchronize a user account with an on-premises mailbox to Office365 you synchronize its msExchMailboxGUID value to the corresponding cloud mail-enabled-user object.  This tells Exchange Online that there is an on-premises mailbox.  The reason for this is simple; if the administrator assigns an Exchange license in O365 to this user, Exchange won't provision a cloud mailbox for the account as well.  This also prevents potential conflicts surrounding moving this mailbox to the cloud in the future.  This guarantees that Exchange Online can't assign that GUID value to any other cloud-only (non-synchronized) mailbox that a tenant administrator may create in the future.  If that happened, you wouldn't be able to move the mailbox to the cloud.

If you remove the user from sync and place them back well after 30 days as described in the above scenario, for the reason stated above, Exchange Online will not provision a mailbox - no matter if it once hosted a migrated mailbox for this account and regardless of the true absence of an on-premises mailbox.  In other words, if it detects a populated msExchMailboxGUID , it makes no assumptions.  However, if the user is removed and re-added to directory synchronization within the 30 days grace period, then the soft-deleted mailbox in Exchange Online should be restored to the user after sync is complete.

OLUTION

To resolve this issue properly, you need to zero out the msExchMailboxGUID value on the on-premises object and then run directory synchronization. Afterwards, the account will appear to Exchange Online as a recipient for which it needs to provision a mailbox: 

  1. Run the following command from on-premises Exchange powershell (replace Identity with the actual Identity of the account):

    Set-RemoteMailbox Identity -exchangeGUID 00000000-0000-0000-0000-000000000000
  2. Run directory synchronization
  3. Wait until you confirm that the change was replicated to AzureAD before checking the state of the user in Exchange Online.

  • Collect the immutableID and objectID from the msoluser object using Windows AzureAD Powershell:
    Get-MsolUser -UserPrincipalName upn | select immutableID, objectID

  • Convert the immutableID from base64 to a GUID:
    New-Object -TypeName System.Guid -ArgumentList(,(([System.Convert]::FromBase64String(" base64stringValue "))))

  • Search for the matching recipient in on-premises Exchange shell and confirm that the user has a populated ExchangeGuid value (synonymous with msExchMailboxGUID) and has a recipient type of MailUser:
    Get-Recipient guid | select ExchangeGuid,RecipientType

  • Use the objectID from step 1 and search for a matching recipient in Exchange Online powershell. If you find a match, confirm its recipient type. If it matches 'mailuser' then you are experiencing this scenario.
    Get-Recipient objectID | select *type*
