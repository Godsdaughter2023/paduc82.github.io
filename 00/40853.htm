omoikane1122 say:
Hello my name is Bradford and I'm an Independent Advisor for the Microsoft Community and I will try to resolve your problem.

It sounds like your Icon database in Windows is corrupted and has to be rebuilt. In Windows 10 the procedure is a bit long but I've outlined it below for you:

To rebuild the icon cache database on Windows 10, close any application that you may be running, and then do the following:

1. Press Windows Key+R to open the “Run” box.
2. Type “cmd” into the box and then press Ctrl+Shift+Enter to run the command as an administrator.
3. Type the following command to navigate to the icon cache database location and press Enter:

cd %homepath%\AppData\Local\Microsoft\Windows\Explorer

4. Type the following command to verify the icon cache database files are at the location and press Enter:

dir iconcache*

The output of the dir command should list these database files:

iconcache_1280.db
iconcache_16.db
iconcache_1920.db
iconcache_256.db
iconcache_2560.db
iconcache_32.db
iconcache_48.db
iconcache_768.db
iconcache_96.db
iconcache_custom_stream.db
iconcache_exif.db
iconcache_idx.db
iconcache_sr.db
iconcache_wide.db
iconcache_wide_alternate.db

--VERY IMPORTANT about step 5: Once you terminate File Explorer, your desktop background and taskbar will disappear, leaving a black screen — but it's OK, it's only temporarily. You need to close File Explorer to be able to delete the icon cache files. Otherwise, you won't be able to do it.
------

5. Type the following command to stop File Explorer and press Enter:

taskkill /f /im explorer.exe

6. Type the following command to delete the iconcache files and press Enter:

del iconcache*

Note 1: If you get access denied trying to remove the files, you're probably not running Command Prompt with as an administrator. Repeat these steps, but making sure Command Prompt starts with administrative privileges.

7. Type the following command to verify that you successfully deleted the icon cache files and press Enter:

dir iconcache*

Note 2: If you still see some iconcache files, then it's likely that some apps are still running, which will prevent you from deleting the database files. Make sure to close them all, and repeat the steps.

8. Type the following command to start File Explorer and press Enter:

explorer.exe

9. Close Command Prompt to complete the task.

Once you completed the steps, Windows 10 will automatically recreate the database files with fresh icons, this should fix the problem

If following the above steps doesn't resolve the issue please let me know!

syriel say:

The posted answers to your question aren't exactly correct.

  

With the Xbox 360, it breaks down like this:

 

A NTSC unit will not output a PAL signal. In standard definition, it will always output 480, which is NTSC. Most newer TVs in Europe are multi-standard and will accept a NTSC signal. 

 

If you output in HD (720p or higher) then there is no difference between NTSC and PAL. Those designations really only apply to SD signals. 720p is 720p no matter where you are located.

 

As far as games are concerned, your console will play any games that are region coded for North America as well as any games without a region code. Most Microsoft first party games are region free and do not have a region lock. With third party titles, the region lock is decided on by the publisher. If a game is region free, it will play (and output properly) on your console. Unfortunately, the only way to confirm if a game is region free or not is to look it up online or try it (I suggest renting) as this is not something that is publicized on the game boxes.

 

Finally, there is Xbox Live Arcade. Your XBL account is probably set to the US region. Anything you've downloaded will still work. However, going forward, you may run into some games that cannot be downloaded from NA XBL while you are physically located in Europe. This is due to licensing. Any MSP cards you buy in Europe will not work on your US account (you would need to get US cards). XBL Gold time bought in Europe *may* work on your US account, but again, you would be better off buying subscription time from a US seller.

 

If you are going to be in Europe for the long haul, you may want to use the online tool to change your XBL account region from North America to your current location in Europe. After using the migration tool, your account will connect to the local version of Xbox Live and locally purchased MSP and XBLG time cards will work. However, American cards will not.

 

Hopefully this helps answer some of your questions.

cary l. brown say:
I recently updated my XP computer with all the latest rollups, security updates, etc. so I could update my Windows Media Player from version 10 to version 11.  All fine and great -- I now have WMP 11.  But the reason I went through all that was because WMP wasn't playing CERTAIN audiobook files I downloaded from Audible.com in their "enhanced" CD quality format.  In fact, trying to play the file(s) just makes WMP shut down / crash every time.  However, I can play OTHER files I have IN THE SAME FORMAT, also from Audible.com, with no problem.  Any idea why?

You are definitely not the only person ever to encounter this.   I have also experienced it, on some (but not ALL) of the computers I use.

To those who would related this to CODECS... I'm sorry, that's nonsense, and if you don't understand a problem, you should probably avoid making "suggestions" on that issue.   The "Audible Manager" program, provided by Amazon Audible, is a self-contained program which can play the two AUDIBLE formats, and allows these to be played by either iTunes or WIndows Media Player/Windows Media Center as well.

This is NOT a "CODEC" issue.  If it were, the media files would not play on ANY application...   but that, clearly, is not the issue.

The telling point in the original poster's (henceforth "OP's") question is "why do some of these refuse to play," essentially.  And REFUSE is the correct point.   You'll note that the OP had just downloaded a series of "Windows Updates."   This is also something that everyone else who has experienced this has also done just prior to losing their ability to play back these files.

The thing that you need to note in what I just said is "REFUSE to play back."   As in... the file is there, it contains the proper data, and CAN be played (by some applications).

It's very clear, this is a "Digital Rights Management" (DRM) issue.   Somehow, the Windows Media "DRM" system is no longer recognizing that you have a legitimate right to play these files, so the files are not allowed to be played back.

If you want to confirm that this is what is happening, find a friend who has some DRM-protected files, and transfer those files to your system, and try to play THEM.  You will see the exact same behavior.   You can click on the file, but it will simply not play back.

Note that Audible has two principle file formats... the *.aa file format (their original format, mono, and with several bit-sampling rates, rated from "1" (the lowest rate) to "4" (the highest rate), and the *.aax" format (better quality overall, stereo...  this sounds much better but is a MUCH larger file format).  For those of you who understand what "bitrates" are...  Format 2 is 3.7MB, format 3 is 7.2MB, format 4 is 14.4MB and format E (enhanced) is 28.8MB.

You will find that the DRM system used in the AA files is fully supported by the latest/greatest "DRM system" updates provided by Microsoft, but that the "AAX" format files, which use a more "complicated" DRM system, will often become recognized by the Microsoft DRM system as "unauthorized" (even though they are, in fact, fully authorized).

On one of my machines, this happened to the entire batch of AAX files... but on another, only on a subset.   And I've noticed that NEW files seem to be "accepted" but files I've previously registered on a given machine will not.

And on my little netbook, I just wiped the OS and did a clean reinstall...   I do that regularly on that machine (as I take it with me when traveling and worry that, with all the unsecured WIFI I have to use, something will get inside of it).   Of course, a "clean reinstallation" means, among other things, that the DRM database is also "cleanly reinstalled."

Remarkably, on that netbook, I can play back all of my Audible files, with all current "Windows Updates" installed.  Prior to the most recent "clean reinstallation," however, I couldn't play any of them, even though the system was in the same state (as far as Microsoft products are concerned, anyway).

So...  I have a suspicion.  I suspect that there is an encrypted database, maintained by the OS, of all files which are "authorized" to be played by DRM, and I suspect that this same database contains a list of "known bad actor" files as well...  and somehow, Audible "enhanced" files can end up on the wrong list.

If there were some way to view, and edit, the DRM database (I'm sure that there is, but that we, the end-users, are not permitted access to it, right?)...  You would see entries for what files are "deemed unsafe" or "unauthorized files" or "authorized files" and so forth, and you'd find that, either as a class, or as individual files, these "AAX" files are not permitted to be played.

You have three options, it seems.

1)  Reinstallation of your OS.   Depending on the complexity and level of customization of your installation, this may be an easy solution or it may be an amazingly painful one.   In my case, I have one machine where it's easy (the netbook), one where it's an annoyance (my home theater PC), and one where it's a nightmare which would take months to recover from (my main working machine).

2)  Downgrade from the (much nicer) AAX files to the smaller, somewhat less nice AA (level 4) files.   This is certainly acceptable for most users...  but not for me on my HTPC setup (played back through a studio-level 7.1 sound system).   These files are always permitted, it seems, as long as they're installed into WMP by the Audible Download Manager.

3)  Play back the AAX files from another application.  I have noticed that iTunes doesn't seem to have the same problem (let me know if you do have an issue).  I'm not a fan of iTunes, though... but this is a solution.  Or, just play the AAX files back through Audible's program.

In my case, I sort of did a hybrid of #2 and #3 on my HTPC.  I have both the "enhanced" and the "level 4" file versions downloaded.  I can play back the "4" files in WIndows Media Player/Windows Media Center, and I can play the high-quality files in Audible Manager, or in iTunes.

Whether the issue is with Amazon/Audible, for not keeping up with "improvements" in the Windows DRM system, or if it's with Microsoft...  the issue is WITH ABSOLUTE CERTAINTY one of "Digital Rights Management."

(Of course, legally speaking, you have a right to whatever files you've purchased (as long as you don't redistribute them to anyone else).   So, I suppose the fourth option would be to convert the files (using any of a number of "playback and record the playback" type utilities).  The problem with this is that it's a LOT of work...  you have to recreate your metadata, break the file down into reasonable "chapters" and so forth.   There are definite advantages to the "Audible" formats...)

cupawntae say:

In case anyone else comes across this thread as I did, in my case the problem was caused by LAV splitter from LAV filters.

Switching off mp4 in LAV Splitter Configuration -> Input Formats allowed WMP to play my aax files again.

I can't say for sure if any other third-party media splitters might have the same effect (certainly if they do try to split the aax stream it will fail).

In case it helps, in my case I figured it out by starting with a clean install of Windows 7 in a VM, and installing components until it stopped working. Once I knew it was LAV, I disabled all input formats in LAV splitter to check that it started working again, which it did.

Then to figure out which particular format LAV thought it was handling (because it wasn't AAC as I first guessed), I went back to the real machine, enabled the system tray icon for LAV splitter and I could see from that that it was mp4.
