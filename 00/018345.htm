ChelseaP_43 –

In the Design tab of the Form Layout tools, click the dropdown in the Controls panel, and turn on the Use Control Wizards.

ken sheridan say: Am I to understand that you wish to select a  row from the MI table in the unbound combo box in the form's header and then select an item in the bound combo box in the detail section, where the list is restricted to those rows from the referenced table CTS_MI where the value at the CTS_ID_fk column position references the row in MI selected in the unbound combo box in the header?

Firstly, with a form in single form view it is immaterial whether the unbound combo boxes is in the header or not.  That only applies to forms in continuous forms view, to retain the visibility of the unbound combo box in the header.

The RowSource for the unbound combo box in the header would be:

SELECT mi_id, mi_number, mi_desc
FROM mi
ORDER BY mi_number;

Assuming you wish to hide the mi_id column, the control's other properties should be like this:

BoundColumn:    1
ColumnCount:     3
ColumnWidths:    0cm;2cm;3cm
ListWidth:             5cm

Experiment with the ColumnWidths dimensions to get the best fit.  Leave the first dimension as zero to hide the mi_id column.  The ListWidth property should be the sum of the ColumnWidths dimensions

You can order the result set by whichever column you wish; I've arbitrarily used mi_number.  In the unbound combo box's AfterUpdate event procedure requery the bound combo box (Combo2?) in the detail section with:

    Me.[Combo2] = Null
    Me.[Combo2].Requery

This clears the control of any current value and reloads the RowSource recordset so that it is restricted to those which reference the value in the unbound combo box in the header.

The RowSource property of the bound combo box in the detail section should be as follows to correlate it with the unbound combo box in the header:

SELECT cts_mi_id, airframe, release_date
FROM cts_mi INNER JOIN cts ON cts_mi.ctd_id_fk = cts.id
WHERE mi_id_fk = Form![Combo24]
ORDER BY airframe;

Again I've used airframe arbitrarily for the sort order.  The other properties should be as for the unbound combo box above.

The final task is to set the value of the unbound combo box to that referenced by the bound combo box if you navigate to an existing record.  This is done in the form's Current event procedure:

    Me.[Combo24] = DLookup("mi_id_fk", "cts_mi", "cts_mi_id = " & Nz(Me.[Combo2],0))
    Me.[Combo2].Requery

BTW, I would strongly advise that, immediately after adding a new control to a form in design view, you change the Name property of the control from the meaningless name such as Combo2 which Access assigns to it, to something meaningful such as cboMI.  Also, I note that you have AIRFRAME as a foreign key in CTS.  Consequently AIRFRAME in the AIRFRAME table must be a candidate key.  You can therefore delete the current AIRFRAME_ID column, and make AIRFRAME the primary key of the AIRFRAME table.  You can then enforce the relationship between the tables.  An unenforced relationship serves no purpose whatsoever.

I note that you appear to have 'repeating columns' SOI_POS1, SOI_POS2, and SOI_POS3 in the SOI table.  What do these columns represent?  Having multiple columns holding values of the same attribute is bad design as it means the table is not normalized to First Normal Form (1NF), and should be decomposed into two related tables.

Finally, we never judge, but we always give the best advice of which we are capable on the basis of the information provided, even if it might not always be appreciated as such.

ken sheridan say:

So, in the unbound box, I need the row source to be the CTS_ID, CTS_AIRFRAME, and CTS_RELEASE_DATE.  In the bound box, I need the results of all of the MIs (MI_ID, MI_NUMBER, MI_DESC) that are related to the selected CTS.

That doesn't make any sense in the context of the model illustrated in your original post.  The combo box in the detail section is shown in your screenshot as bound to the CTS_MI_ID_fk column.  This is a foreign key referencing the CTS_MI_ID column of CTS_MI.  Consequently the values in the column must be members of the set of values in the CTS_MI_ID column in CTS_MI, not the MI_ID column in MI.  I suspect your model is in need of amendment, but without any knowledge of the real world entity types being modelled and the relationship types between them, I can't say more than that.

Another issue is that, if you want to use a continuous form, simple correlated combo boxes will not work where the BoundColumn property references a hidden key column, as is usually the case where a surrogate key is used rather than a natural key.  This is because, when the form's record pointer is moved to a record the combo box is requeried.  In other rows in the form, where, as a result of requerying the combo box, the row in the RowSource referenced by the value in the hidden column is not present, this will result in the combo box in those rows being blank due to there being no row for the value of the control to reference.  No data is lost, merely invisible.

The solution is to use a 'hybrid' control made up of a read only text box carefully superimposed on the combo box to give the appearance of a single combo box control.  The form's RecordSource must be a query which includes the relevant referenced table.  The text box is bound to the appropriate non-key column from the referenced table, and the combo box to the foreign key column in the form's recordset.

to create a new address book policy for users with specific attribute, please follow the steps below:

set-mailbox <mailbox> - customattribute1 domainname
set-distributiongroup "newdg" -customattribute1 domainname

new-addresslist -name "address list1" -recipientfilter { (((recipienttype -eq 'usermailbox') -and (customattribute1 –eq 'domainname'))) -or (((recipienttype -eq 'mailuniversaldistributiongroup ') -and (customattribute1 -eq 'domainname')))}

set-mailbox <mailbox> -customattribute1 domainname
set-distributiongroup "newdg" -customattribute1 domainname

new-globaladdresslist -name "gal_1" -recipientfilter {(customattribute1 -eq "domainname")}
7. create new offline address book:
new-offlineaddressbook -name "oab_1" -addresslists "gal_1"
8. create new address book policy:
new-addressbookpolicy -name "abp_1" -addresslists " address list1" -offlineaddressbook "\oab_1" -globaladdresslist "\gal_1" -roomlist "\al_rooms"
9. assign an address book policy to mail users
