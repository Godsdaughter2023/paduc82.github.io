john w. vinson say:

Well, first off... Access is emphatically NOT "Excel on Steroids". It's different and requires a different mindset, and probably some "unlearning" of the way things are done in Excel.

In this instance, the Customer list ("TABLE") should simply not contain any date of service. Instead you need (at least!) two tables - a table of Customers, with a unique customer ID, last name, first name, contact information, etc., related one-to-many to a table of Services. This table would have a primary key, a CustomerID field as a link to the Customers table, probably a ServiceTypeID with a link to a table of Service Types, and a ServiceDate field.

The Customer table would probably include a ServiceFrequency field to indicate how frequently this customer receives services. The Next Service Date field would not exist anywhere in ANY table - instead, it would be calculated on the fly from the maximum of the service dates in the Services table and the frequency. Users (even you, when you're not actively developing) should never even see a raw Table; instead, you would use Queries to arrange the data (and calculate the next service date, perhaps) and Forms to display and edit it.

John, we went further by having a tblWorkOrders to record each time a service was performed for historical purposes.  It had ReqDate and CompDate.   We would run an append query for the next period of workorders - week in advance OR month in advance.

But wait...What if you are using outlook 2016 with office365????

Hummm.....Let me explain a bit.  Outlook connects to exchange online following FOUR basic operations

1. It looks for SCP(internal network) for a URL to locate A record

2. It tries to find A record associated with domain like https://techsupporto365.com

3. It tries to locate A record with URL prepended within auto discover.xml

4.It tries to locate SRV record for autodiscover service in DNS zone that matches the user's SMTP domain.

If you are office 365 user and getting  the above error with outlook 2016, than believe me you must look for SCP in your AD which is still pointing towards your old exchange 2007.

so the outlook connecting with a unsupported exchange version.

FIX:

 If there is SCP, you need to remove it so the all users could configure outlook2016 easily.

If you want to do this fo a single user or may be you don't have access to local AD, than force outlook not see local SCP.

ADD the registry values:

This screen shot is from outlook 2015, you will do the same step for outlook 2016.

If issue still there, than you need to go to your DNS and make sure MX record for exchange is created their.

In my case it was DNS. Create MX record and configure the profile. I hope this will help.
