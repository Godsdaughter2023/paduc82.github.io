rohn007 say:
Office is written with the assumption that there is only one version ever on the computer. The logic being along the lines of <sarcasm> "... what reasonable user would want to keep the carpy old version when they have the wonderful new & improved version ...." </sarcasm>.

But it can be done.  I currently have Office 2003, 2007, 2010 and 2013 all running on my computer in the same windows session.

On that basis when you simply have 2 versions installed on the same computer you run into 3 main issues:
  1. "configuration": each time you open an application from the "other" version then Office performs the "configuration" process to setup the required registry entries on the assumption (wrong) that they are not there
  2. your normal templates and other configuration files will be in the same locations
  3. Windows can only make one application "default" for a file type.

The issues can be worked around.


#1 MS developers needed to be able to work with more than one version so they created a simple registry hack, adding a registry key to suppress "configuration" process

Here are instructions on how to do this:

Dual Install - Working with Multiple Versions of Office

https://office.microsoft.com/en-us/word-help/keep-earlier-versions-of-office-programs-when-installing-office-2010-HA102597134.aspx

This tip describes the specific steps and choices during the 2010 custom install. After you have the multiple versions installed you still have some additional steps to do. They are described in this tip:

 

http://pubs.logicalexpressions.com/pub0009/LPMArticle.asp?ID=762

With multiple versions installed, there is one issue you can’t avoid, Default Windows File Associations. Only one app can be used to open the files by double clicking in Windows Explorer. You have to pick which one.

FYI: NoReReg key works with Office 2010 / 2013 installed on Windows 7 and 8 (I tested it myself Rohn007)

Note: in user accounts, when you log in to do the NoReReg hacks, DO NOT start REGEDIT using “Run As Admin”. You will be editing the wrong “Current User” key <grin>

 

Office 2010 configures each time you start

http://support.microsoft.com/kb/2528748

In Method 1, this tip has a Fix it, 50780, that applies the required Registry fix.

 

Alternate Fixes

#2 After you install, you should go into File tab > Options command > Advanced option > General Section > File Locations button and changing the locations for the config files

john w. vinson say:

I have to agree with Tom here. You are using Access as if it were Excel; it is NOT Excel! Databases work differently than spreadsheets; your design is possibly ok for a spreadsheet but NOT for a relational database. It will give you trouble forever!

A spreadsheet has many rows and many columns. A database has many, many rows and very few columns in each table, but uses multiple related Tables.

For your case, it appears that you have a "Many to Many" relationship: each Word can be shown in many Scripts, and each Script can depict many Words. In a spreadsheet you can do it as in your example; but in a Database you will need THREE tables: a table of Words; a table of Scripts (Latin, Greek, Cyrillic, Pashtun, etc. etc.); and a third table related one to many to each of these, containing the transliteration for that language pair. This table would have only three fields - a link to the Words table, a link to the Scripts table, and a field for the word in that script.

I wish you luck; this is a very challenging project, just because of the differences between phonemes in different languages. The English letter "o" can be pronounced - and transliterated into other languages - many different ways depending context.

If fName <> "" Then
Application.ScreenUpdating = False
Do Until fName = ""
Application.DisplayAlerts = False
myWB.Sheets(fName).Delete
Application.DisplayAlerts = True
Workbooks.OpenText Filename:=fPath & fName, DataType:=xlDelimited, Tab:=True
Set wb = ActiveWorkbook
wb.Sheets(1).Copy after:=myWB.Sheets(myWB.Sheets.Count)
myWB.Sheets(myWB.Sheets.Count).Name = fName
wb.Close False
fName = Dir()
Loop
Application.ScreenUpdating = True
Else
MsgBox "no files"
End If
Else
MsgBox "Cancel"
End If
End With
End Sub
