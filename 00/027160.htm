wildcard in my query criteria is returning too many values for search - Microsoft Community

e-zshelving say:

wildcard in my query criteria is returning too many values for search purposes.

Greetings,

I've made an Access 2013 search form modeled after this example, in which a list box displays a very long list of manufactured parts that the users sort by entering keywords into text boxes, or by selecting part attributes from option groups.  The source of the list box data is a query.  The criteria for the query are all patterned similar to 'Like "*" & [forms]![frmPartSearch]![Depth] & "*"'. 

When the form loads the list box displays all records, then sorts the records dynamically as user types and selects.  Where I'm having trouble is with the sorting by option group selection values.  If the user selects a Depth of 6" (DepthID and option group value 1) the list box also returns parts where the Depth is 24" (DepthID and option group value 11) due to *1* indicating both 1 and 11 as acceptable values.  Instead I would like the criteria to only return parts where the depth value is 6" (DepthID and option group value 1).  If I remove the wildcards the list box starts out with no values when the form loads.

To resolve this I've tried adding a new field to the query as follows:

Depth: IIf([forms]![frmPartSearch]![Depth] Is Null Or [forms]![frmPartSearch]![Depth] > 9, Like "*" & [forms]![frmPartSearch]![Depth] & "*", Like [forms]![frmPartSearch]![Depth]). 

I'm trying to tell the query to use the wildcards if the Depth option group is set to Null or a double-digit number, otherwise drop the wildcards and sort by the single-digit number.  The form has a "Reset Depth" command button that sets the Depth option group back to Null in order to clear the depth sorting.

1) Is there a better way to over come the query criteria issue of *1* returning 1, 11, 21, etc.?  I'm aware that a work-around would be to eliminate the single-digit DepthIDs from by Depth table.  I would like to learn a solution using criteria syntax if it exists. 

2) If I'm on the correct track with my iif statement, could you help me correct the syntax?  I'm receiving the error, "The expression you entered includes invalid syntax. You may have entered a comma without a preceding value or identifier."

Thank you!

scottgem say:

What is the Recordsource of the form? Why not use a filter instead of query criteria?

Why would you use the Like Operator when you need an exact match? The LIKE operator is generally used with Text datatypes where you want to search on a partial string. 

ken sheridan say: To allow for a Null parameter and return all rows, the LIKE operator with wildcard characters is not a particularly reliable method as it is prone to (a) specious substring matches (as in your case), and (b) it does not allow for Nulls in the column in question if all rows are to be returned (Null is not a value, but the absence of a value, so can never equal anything).  The more liable method is to use simple Boolean logic in the query's WHERE clause

WHERE ([SomeParameter] IS NULL
  OR SomeColumn = [SomeParameter])
AND ([SomeOtherParameter] IS NULL
  OR SomeOtherColumn = [SomeOtherParameter])
AND etc....

Assuming good indexing in the table, this is a very efficient method because of the simplicity of the logic.

However, in a case like yours, involving the interactions of two parameters, the option group and the text box, I would not use a WHERE clause to restrict the query's result table, but build the SQL statement in code on the basis of the value of the option group, and the value of the text box, assigning the statement to the list box's RowSource property.  This allows you infinite variety in the criteria being applied, so the equality operator can be used where appropriate (particularly with numeric values), the LIKE operator can be used where you wish to do pattern matching, or the BETWEEN....AND operator can be used where a range is defined by two parameters.
