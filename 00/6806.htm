It is very common for calculated values to appear identical even when formatted to 15 significant decimal places, but in fact they are not, due to limitations of how Excel represents numbers and performs arithmetic (64-bit binary floating-point).  For example (using comma to demarcate 15 significant digits):

A1:  =1.01-1  is exactly 0.0100000000000000,088817841970012523233890533447265625

A2:  =0.01     is exactly 0.0100000000000000,0020816681711721685132943093776702880859375

=A1-A2         is exactly zero (0.00E+00 when formatted as Scientific)

=A1=A2        returns TRUE

=A1-A2-0      returns FALSE

The latter is the correct comparison.  The former comparison (A1=A2) returns TRUE because Excel arbitrarily and inconsistently treats two values as equal when their difference is "close to zero", but not really zero.

That probably explains why you "used IF function to check if the two numbers from the two lines

If you want numbers to be accurate to a predetermined number of decimal places, you should

Formatting alone is not sufficient.  It changes the appearance, but not the actual value.

[EDIT] PS....  Re: "I have removed the formula by copying and pasting value only."

Sometimes, paste-special-value does indeed change the internal representation so it is equal to the equivalent constant.  But sometimes, it does not.

The following probably would have worked:  select the cell with the calculated value, press F2, then F9, then Enter.

That works when you cannot see any difference even when the values are formatted to 15 significant digits.  But I do not recommend it as a long-term solution.

Besides, it does not work when you can see a difference.  For example, =10.01-10 appears as 0.00999999999999979 when formatted to 15 significant digits, not 0.01.  So pressing F2, F9, Enter would not remedy the difference.

ken sheridan say: Let's examine where you are going wrong:

1.  Do not use the lookup field wizard in table design.  It disguises the true data in the column (field) and should be avoided like the plague.  For reasons why see: http://theaccessweb.com/lookupfields.htm  You can achieve the same in a form by binding a combo box to the column.

2.  Do not interact with data in a table's raw datasheet view.  All interactions should be via forms.  The only exceptions might be when debugging.

3.  Your design of your Masterfile table is wrong.  By having multiple TSR_# columns data is being encoded as column headings.  This violates one of the basic principles of the database relational mode, the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.  What you have is a binary many-to-many relationship type between MasterFile and TSR which should be modelled by a third table, so in broad outline the model would be like this:

MasterFile
….masterID  (PK)
….master

TSR
….tsrID  (PK)
….tsr

and to model the relationship type between them:

MasterTSR
….masterID  (FK)
….tsrID  (FK)

The primary key of the last is a composite of the two foreign keys.  The table might also have further columns representing attributes of the relationship type.

4.  Masterfile is a bad name for a table.  Tables are not files, but sets modelling an entity type. attributes of which are represented by its columns.  Table names should identify the entity type being modelled, e.g. a table Employees would have columns employeeID, firstname, lastname etc.  A common convention, which I'd recommend, is to use plural or collective nouns for table names and single nouns for column names, reflecting the fact that a table represents a set and a column represents an attribute.
