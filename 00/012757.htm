reference checkboxes in VBA

I have created VBA that is pulling data from a form and appending it to a master Excel file.  The form includes Form Control checkboxes and I’m not sure how to reference these checkboxes within VBA.  Below is a picture of some of the checkboxes.  What I would like is if any of the checkboxes are checked the value returned to my master file would be the names of the checkboxes concatenated with commas in between.  So the below example would return FLM, SystemOther.

Thanks

hansv mvp say:

Perhaps you can use this as starting point...

Sub ListCheckBoxes()
    Dim wsh As Worksheet
    Dim chk As CheckBox
    Dim strVal As String
    ' change sheet name as needed
    Set wsh = Worksheets("Sheet1")
    ' loop through check boxes on sheet
    For Each chk In wsh.CheckBoxes
        ' test whether check box has been ticked
        If chk.Value = 1 Then
            ' if so, append name to string
            strVal = strVal & ", " & chk.Name
        End If
    Next chk
    ' if any check box was ticked ...
    If strVal <> "" Then
        ' ... then remove first ", "
        strVal = Mid(strVal, 3)
    End If
    ' do something with the value
    Worksheets("Sheet2").Range("A1").Value = strVal
End Sub

Tracking external deliverables - Microsoft Community

jpstewar say:

Tracking external deliverables

 -->How do you track external deliverables (potential constraint - may be provided by another company or external resource)?  Do you use a milestone as a predecessor to start the dependent task?
jim aksel say:

We have a section at the top of the schedule for Givers and Receivers.  You are talking about a Receiver. 

Receiver- A milestone containing no predecessors (internal to the file) representing the delivery of something from an external source such as a supplier, or another schedule..  Receivers must have a sucessor task that points down into the subject file.  In effect, Receivers must drive something inside this schedule, but are not driven by anything within this schedule.  Receivers have a Date Constraint of "Start No Earlier Than" along with a date.  Sometimes they also have a Deadline date that will notify the user if the anticipated receipt date is late.  The dates for receivers come under configuration control and require permission for them to move.  A Receiver must have a corresponding Giver in some other schedule, otherwise you are requesting something that no one plans to create.  In a stand alone schedule, Receivers have NO predecessors and DO have successors.

Giver - A milestone representing an end product produced somewhere within this schedule.  Givers contain no successors while having one or more predecessors from "deep within the project file".  Givers have no date constraints.  Specifically AVOID using Finish No Later Than as this is not the logic you think it is.  Instead, Givers have Deadline Dates which let the uder know if any action taken in their schedule violates an agreed to date.  Givers are also under control of Configuration Management. A Giver matches up to someone else's Receiver, otherwise you have no reason to produce the product represented by the Giver. In a stand alone schedule, Givers have NO successors and DO have predecessors.

Configuration management can be as simple as a spread sheet as long as it meets the company rules.

The requirements for No Predecessor/Successor is different if you have Master Project/Subproject files are Linked files.  Consider that a Giver may have NO "internal" successors but will have at least one "external" successor matching to a receiver in another file.  In a similar manner, in a linked file setup, Receivers will have only "external" predecessors and "internal" successors.

Hopefully my ramblings are useful for you.

If this post was helpful, please click on something :) I am also at http://www.msprojectblog.com Jim

Yes, one way to model an external deliverable is to use a Milestone task with an FS dependency on the next task in your project.  You can place a constraint on the Milestone task to show the anticipated date of completion for the external deliverable.  When the other company or external resource completes the task, enter an Actual Finish date on the Milestone task to show when the deliverable was finally completed.  Hope this helps.

Much of the formatting in Access is determined by the default Printer, so be sure you've setup the correct Printer on your home machine. If you're pointing to a PDF printer, or a Label printer, for example, you can get some oddly formatted pages.

Also be sure you're not using any Themes in Access. You can review these in the File - Options menu.
