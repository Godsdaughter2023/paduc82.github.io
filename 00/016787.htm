Predecessors - Microsoft Community

a. user say:

Predecessors

I'm trying to create a project in which there is a final date which is set in stone, and I would like all of its predecessors to fall in place behind that.  I tried setting a constraint of "Must finish on" and linking previous tasks finish-to-start with lag time, but that does not seem to be working.

Thanks in advance!

john - project say:

AustinU,

My advice, forget about setting a "must finish on" constraint or even reverse scheduling, those approaches will give you more problems than simple forward scheduling.

I'd start by laying out all the tasks required to achieve the finish date or output. Link them in a logical sequence using basic finish-to-start dependencies. All tasks, except the beginning milestone, should have at least one predecessor and all tasks, except the final milestone should have at least one successor. Use the Deadline field to set the "stone" finish date. And remember, if you have a realistic workable plan but it still doesn't meet the required end date, then it's time to question the "stone". Forcing the issue is a bad idea all around.

Once the above is all laid out, does it meet the finish deadline? If so, great, press on. If not, review your plan to see where you can make realistic adjustments to bring the finish date back to the deadline. When it's all ready, set a baseline so you can track the current plan against the original plan.

Project plans are iterative. It is very likely that the schedule will become delinquent at one or more points during execution and it is your job as the project manager to make changes as necessary to achieve the final goal.

Hope this helps.

John

trevor rabey say:

This approach to scheduling that you describe, where you begin with the finish date in the future, is very common but it is a mistake, or at least it is not the only way to phrase the problem.

Look at it another way. If the start date is selected first, and then all of the tasks are listed with their durations and their predecessors, what you get is the earliest overall finish date, as the output from the durations and their predecessors, which is the input.

If that is earlier than your target, then you can accept that or change something to make your finish date later. What you can change is either the start date, or durations of tasks (mainly those on the critical path) or the predecessors which form the sequence or order of performing the tasks.

If that is later than you want it to be, then either accept it, or change something.

If your managers intend to only start at the latest possible date, then all of their projects are destined to be late.

Put the following macro in the ThisDocument class. (same place as your "Document_New" sub) This issue you are having is timing, which is a problem on the Mac.

Sub AutoNew()
    Dim PauseTime, Start
'    Add a pause to allow time for Word to open before continuing
    PauseTime = 1  ' Set duration in seconds
    Start = Timer   ' Set start time.
    Do While Timer < Start + PauseTime
        DoEvents   ' Yield to other processes.
    Loop
End Sub

As the note in the code says, this will allow Word to fully load and continue before executing the private Document_New routine. Optionally include the code you have in your Document_New routine following the "Loop" statement in the public AutoNew routine. You can then delete the Document_New routine. Let me say again, this is an option, you don't have to change anything in your code, other than adding the AutoNew routine. The private Document_New will execute when the AutoNew (with pause) completes.

    %AppData%\Microsoft\Windows\Recent\AutomaticDestinations

  3. Identify files that are comparatively larger in size than the rest of the file, such as files that have file size of 1 MB or more.
