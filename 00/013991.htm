ken sheridan say: If each row in LessonPlans relates to only one row in each of the other tables, then you have a simple one-to-many relationship type so you simply need a foreign key column for each in LessonPlans.  You'll need an additional short description text column in each of the other tables so that you can show that in a combo box in a form for inserting lesson plan records.  Taking Prep as an example the combo box would be set up like this;

ControlSource:    PrepID

RowSource:     SELECT PrepID, PrepDescrip FROM Prep ORDER BY PrepDescrip ;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.  The PrepDescrip column is the text column in Prep containing the short description you'll see in the combo box.

If on the other hand there can be more than one row from one of the other tables relevant to each lesson plan then what you have is many-to-many relationship types between the LessonPlans table and the other table. A many-to-many relationship type is modelled by a table which resolves the many-to-many relationship type into two one-to-many relationship types.  This is sometime called colloquially a 'junction' table.  If we again take the Prep table as an example.  You'd model the relationship type with a table like this:

LessonPlansPrep
....LessonPlanID (FK)
....Prep (FK)

The two columns are foreign keys referencing the primary keys of LessonPlans and Prep respectively.  The primary key of the table is a composite one made up of both columns.  Diagrammatically the model looks like this:

LessonPlans ----<LessonPlansPrep>----Prep

The relationship types with the other tables would be modelled similarly.  In the form for entering the lesson plans you'd now have a subform for selecting multiple Prep rows per lesson plan.  The combo box would now be in the continuous subform, which would be linked to the parent form on LessonPlanID.  You'll find a simple example of this sort of many-to-many relationship type and how to represent it in a form/subform as ParentActivitie.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

While you'll see only the short description in the combo box it would be a simple task to add a small button beside it which opens a separate form in dialogue mode to show the full memo field corresponding to the selected description.

As regards the report you can either join the other tables to the LessonPlans table in a query which returns the fields from LessonPlans and those from the other tables, including the memo fields, and base the report on the query, or you can use subreports within the parent report to show the fields from the referencing tables.  You may well use a combination of both approaches, joining some of the referencing tables in a query, and showing others in subreports.  How you do it will be determined on how you want the report laid out.

If a lesson plan does not necessarily reference all of the other tables then  you'd need to use LEFT OUTER JOINs in the query to ensure all the relevant rows are returned.

john w. vinson say:

I think you may be making two very common mistakes, one of them aided and abetted by Microsoft's poor design.

 

For one thing, you're assuming that you must have all of the lesson plan data in one table in order to print a report of it. That is emphatically NOT the case! You can, and should, base your Report on a query joining your main table to the related tables, pulling the memo field data from the Prep table or the End table or whatever, along with the needed fields from the main table.

 

The second is that you're another victim of Microsoft's "Lookup Wizard" misfeature. See http://www.mvps.org/access/lookupfields.htm for a critique. The lookup wizard lets you see what appears to be a table containing data from other tables - BUT IT DOESN'T. The table actually just contains links to the child tables; these are typically long integer numbers, but they're concealed from your view by the combo box.

 

So you don't need the lookup fields at all (which is just as well, given that they don't work for memo fields!); instead, just use a Form with Subforms to enter your data, and a Query joining all your tables appropriately to generate the Report.
