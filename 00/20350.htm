Delete rows if a cell value is not equal to worksheet name

I have a worksheet where I would like to use a macro to search the first two characters of the value in a certain column and delete the entire row if that value does not equal the first two characters of the worksheet name.  Is there a way to do that?
lars-Åke aspelin say:
I have a worksheet where I would like to use a macro to search the first two characters of the value in a certain column and delete the entire row if that value does not equal the first two characters of the worksheet name.  Is there a way to do that?
Try this macro:

Sub delete_rows_in_active_sheet()
  column_to_look_in = "A"
  firstrow = 1
  lastrow = ActiveSheet.Range(column_to_look_in & ActiveSheet.Rows.Count).End(xlUp).Row
  For r = lastrow To firstrow Step -1
    If Not Left(Cells(r, column_to_look_in), 2) = Left(ActiveSheet.Name, 2) Then
       Cells(r, 1).EntireRow.Delete
    End If
  Next r
End Sub

Note that the comparison is case sensitive. If you dont want to delete a row from Sheet1 where the content in the "column_to_look_in" is e.g. SH, use the function UCase to convert the strings to upper case before they are compared, like this:

    If Not UCase(Left(Cells(r, column_to_look_in), 2)) = UCase(Left(ActiveSheet.Name, 2)) Then

help getting MS Access 2013 forms on SharePoint 2010 - Microsoft Community

pkemme say:

help getting MS Access 2013 forms on SharePoint 2010

I created a database in Access 2013 with several tables.  I have a main form which contains several subforms.  It is essentially a log for activity.  I have a Sharepoint site in Sharepoint 2010 and when I moved data from Access to Sharepoint from my Access database, only the tables moved over and created lists in Sharepoint.  My forms did not show up.  I have not found a solution for this on any forum.  

albert d. kallal say:

The Access (client) forms cannot move to SharePoint.

A few choices are:

Move your data as you already done and then link your application to that data. You thus can now distribute this “front end” part to any desktop that has a running copy of Access (including the free runtime edition). They will all share the same data, but you still require Access and your application program to be installed on each desktop.

If you wish to migrate forms to the web, then you need to use Access 2010. I suppose you could attempt to download a 2010 web template, delete everything and that would give you a blank 2010 web database. You then import all your existing forms and tables, and then use the 2010 web publishing feature.

So at the end of the day, while moving your data to SharePoint is quite easy, migration and creating forms for the web means your existing forms cannot be used – they must be re-created as web.

And with SharePoint 2010, then I would recommended you use Access 2010.

Note that the above “web forms” for 2010 are much deprecated, and it is recommended you use both Access 2013, and SharePoint 2013 for web design.

However it is “possible” for you to take your existing application and create some web forms that will publish to SharePoint 2010, using Access 2013, but you have to get your hands on a blank web database created with Access 2010. (so if you have Access 2010 someplace, you can do this). You ALSO must be running what is called "Access Web Services" on your SharePoint site (so before doing anything - check if you have this feature).

Note in all cases the existing forms you have cannot be used for the web, and you have to re-create forms as WEB specific.

Regards,
