daniel g. grau say:

I think you are looking at "base64-encoded" files, which is what email programs receive via the Internet when a file is attached to an email message. If that's so, your email program is failing to decode them properly. Some reasons for this:

  • Your email program is malfunctioning
  • The file has been badly encoded by the sender's email program
  • The email message somehow was corrupted along the way

Those types of files start with a heading, similar to this one:

--36908767-896244226-1393621369=:34804--

... followed by the main body (similar to your screenshots), followed by a closing sequence that matches the heading:

--36908767-1059637450-1393621369=:34804--

As you see, this data includes the file name, so the fact that you do have a file name indicates that your email program is interpreting that information, albeit partially, which is why Word is asking you to confirm the file's format. Of course, none of those options will work. The content remains in the original form—text only—in spite of the file's name ending in .doc.

Try using another email service (like a free Hotmail account) or perhaps zipping the files will shield them against corruption. If that fails, post back and I'll tell you how to manually decode that information.

karldaly say:

Calculation based on previous record

Hi,

I have a table full of meter readings. Each record will contain a reading value for a each month.


On each record I also want to have a field called consumption which is automatically populated.  The consumption should be calculated by looking at the meter reading value entered for that month and then subtracts the previous months reading for that meter which leaves the consumption.

This image below should explain

[URL=http://s90.photobucket.com/user/KTJ_SBS/media/meter.png.html][IMG]http://i90.photobucket.com/albums/k258/KTJ_SBS/meter.png[/IMG][/URL]

hansv mvp say:

If the readings are always dated the first of the month, you can use

 

Consumption: reading-DLookup("reading","name of table","[meter no.]=" & [meter no.] & " AND [date of reading]=#" & Format(DateAdd("m", -1, [date of reading]), "yyyy-mm-dd") & "#")

ken sheridan say: You should not store the consumption figure in a column in the table.  This introduces redundancy and leaves the table open to the risk of update anomalies.  The consumption should be computed in a query, which can be done by means of a subquery:

SELECT R1.MeterID, R1.ReadingDate, R1.Reading,
R1.Reading -
     (SELECT NZ(MAX(Reading),0)
      FROM Readings AS R2
      WHERE R2.MeterID = R1.MeterID
      AND R2.ReadingDate < R1.ReadingDate) AS Consumption
FROM Readings AS R1
ORDER BY R1.MeterID, R1.ReadingDate;

Or for an updatable query by calling the DMax function to get the previous reading:

SELECT MeterID, ReadingDate, Reading,
Reading -
     NZ(DMAX("Reading","Readings", "MeterID = " & [MeterID] &
      " AND ReadingDate < #" & FORMAT([ReadingDate],"yyyy-mm-dd") & "#"),0) AS Consumption
FROM Readings
ORDER BY MeterID, ReadingDate;

This will return the consumption per meter for multiple meters, each identified by the MeterID column.  In each case the consumption for the earliest row per meter will be the reading for that date as the Nz function is used to return a zero in place of the Null (non-existent) previous reading value.

The above assumes that the readings are always incrementing, which will normally be the case, but if there is the possibility of a meter being recalibrated, subtracting the maximum earlier reading per meter  would not work of course.

dguidry7150 say:

This was a pain in the **** to trace through, but I did figure it out on my own. Microsoft was useless.

This ended up being related to a commercial security program we use.

Even though the security application wasn't triggering on OneNote, the real-time security scanner would still scan OneNote components during OneNote I/O. The scan would lock files for milliseconds, and that could happen concurrently when OneNote would try to write. This conflict is what was causing our crashes.

This conflict would often result in lingering corruption, and that's why I could often replicate the crash with that security application uninstalled.

I ended up having to exclude OneNote.exe and Onmain.dll from the security scan. Even after that was done, it was not uncommon for the safe mode options to pop up one more time. Usually you'd still have to blow out the cache once.
