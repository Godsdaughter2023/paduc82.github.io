ken sheridan say: Date literals in Access must be in US short date format, mm/dd/yy or an otherwise internationally unambiguous format regardless of the system short date format.  So in SQL to update to 7 February you'd put:

SET MyDate = #02/07/2012#

If it's written as #07/02/2012#  this will be interpreted as 2 July.  However Access will second guess the date if the value is not a legitimate date in US date format, so if you put:

SET MyDate = #14/02/2012#

this will be interpreted as 14 February because there is no month 14.

To confuse matters further, if you enter the date in query design view as 07/02/2012 it will show in design view as #07/02/2012#, but in SQL view it will show as #02/07/2012# if the system short date format is dd/mm/yyyy, and will be correctly interpreted as 7 February.

I always recommend the use of the ISO standard for date notation of YYYY-MM-DD when entering date literals, so, whether in query design view or SQL view, instead of #02/07/2012# you'd put #2012-02-07#.  Unfortunately Access will still change this back to the local regional format #07/02/2012# in query design view, so the confusion is only partly alleviated, but if you use the ISO format whenever entering a date literal you can be guaranteed that it will be correctly interpreted whatever the local system date format has been set to.  For data entry in forms you'd still use your local format of dd/mm/yyyy of course, as I do here, and the dates will show in queries, reports etc in that format by default unless you have specifically changed the Format property of the column or control.
