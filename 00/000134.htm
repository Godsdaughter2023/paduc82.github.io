suzanne s. barnhill say:

It probably depends on the type of document. In the types of documents I deal with, where most of the text is going to end up being some flavor of Body Text, I usually just apply Body Text (or Body Text First Indent) to the entire document, then go back and tag the headings and other non-Body Text paragraphs with the appropriate styles. I retain an unmodified copy of the document (hard copy if available) for reference, but the "nuclear option" or tabula rasa is what works best for me. At the same time, I make a couple of passes with Replace to remove extra spaces and empty paragraphs.

Note that you can create styles based on direct formatting, and you can also update existing styles based on the formatting of a given paragraph. So if you, for example, have a heading paragraph (in Normal style) with the formatting you want for, say, Heading 1, select that paragraph, then right-click on Heading 1 in the Quick Styles gallery or Styles pane and choose "Update to Match Selection."

While these details are general minor, when you talking about software breaking, these minor issues and omissions become VERY large in the scope of things.

 

As for the certificate? Not clear if you talking about a self signed or commercial one. However for the most part I've not bothered with the hassles of using certificates. The simple soltion is to add a trusted location to the the registry.

 

For 2007, that is (don't have 2010 ones handy as of yet).

 

 

HKEY_CURRENT_USER\Software\Microsoft\Office\12.0\Access\Security

\Trusted Locations\

Location(x)

AllowSubFolders (REG_DWORD) = 1

Path (REG_SZ) "C:\MyDatabaseFolder\"

 

Even more simple then above is to set macro security low. This means all nag prompts go away and you don't even need a trusted location or a certificate. I used this approach for years, but for some, perhaps security will now allow this. For 2007, that key is

 

Root: HKCU;Subkey:

"Software\Microsoft\Office\12.0\Access\Security";ValueType: dword;

ValueName: "VBAWarnings"; ValueData: "1"

 

So, one key setting, and then any mdb/mde/accde/accdb file simply copied to the target computer can be double clicked on to run. no prompts, no hassles this way.

I not yet tested the above keys and changed them for 2010, but for your testing, macro security = low is a good testing start untill you at least have things working, and then you can start to tighten things back up if that is important to you.

 

As for word failing, again, try testing with macro security set = low in word, and see at least if it can work.

 

So, check references, and then next I would test and ensure that the code compiles correctly in access 2010. I would then try creating the accDE file in using access 2010 as that will also ensure at least the code compiles with out any broken references.
