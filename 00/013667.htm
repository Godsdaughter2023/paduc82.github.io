    How do you READ the ruler? My page is 6 x 9 the ruler shows under - Microsoft Community

    marilynd. say:

    How do you READ the ruler? My page is 6 x 9 the ruler shows under 1" and under 9" when margins are set.

    I am writing a book in Microsoft Word 2010 version. My paper size is set at 6" x 9".  Margins are set at . 5" top/bottom  Inside .75" outside .5" (mirrored) My publisher tells me there are problems with the resulting PDF file I sent. The ruler does not show a 6"w x 9" long, it shows a minus 1" and minus 8" in the grey areas. How do I read this ruler????????? I am getting so frustrated with this program!  How do I read this ruler?
    steve house say: If that's the paper size those are very unusual dimensions for a manuscript but be that as it may, the ruler zero position is the first character position in the typing area.  If the paper size was set to 6w X 9h, the left margin to .75" and the top margin to .5, the zero point on the horizontal ruler would be .75" in from the left side of the page while the zero point on the vertical ruler would be .5" down from the top.  The light areas in the ruler indicate the typing area on the page, the grey areas indicate the margins.  So just looking at the horizontal ruler, it would start at -0.75", go up to zero at the transition between darker shading and lighter shading, then continue up ending at 5.25 for a total length of 6 inches (0.75 + 5.25), the right-hand end being darker again starting at 4.75 (5.25-0.5).

    Just FYI, if you're trying to actually typeset the finished book so that what you submit to the publisher is the actual page layout ready for the printing press, you might want to consider MS Publisher as that has much more capable tools for pre-press layout work.  AFAIK it can import your Word files directly so you don't have to retype everything.

    If statement for tbl Look Up on Row Source

    I would like to have a field display only certain information in a combo box based on what is selected in a previous data field.

    The first fields is "Highest Degree Level Obtained" which contains the options of Associate's, Bachelor's, Master's, Doctorate's, and Certification. The degrees are in a separate table called "Degrees" and each type of degree is in its own field labeled accordingly. 

    If I just have all degrees listed at once, this is the Row Source formula: SELECT [Degrees].[ID], [Degrees].[Degrees] FROM Degrees; 

    I am thinking it would be something like this:

    SELECT [Degrees].[ID], [Degrees].[Masters] FROM Degrees IF SELECT [Information].[Highest Degree Level Obtained] = "Master's"; 

    How far off am I or is this even possible?

    Thank you.

    scottgem say:
    Not only is this possible, but its a standard technique called Cascading or Synchronized comboboxes. If you search on those terms you will find many articles describing how to do it.

    First, however, you need at least 2 tables:

    tluDegreelevel
    DegreeLevelID (Primary Key Autonumber)
    DegreeLevel

    tluDegree
    DegreeID (PK Autonumber)
    Degree
    DegreeLevelID (Foreign Key)

    Second, you need to understand that you do not have fields on a form. You have controls that may or may not be bound to fields in a table. This is a subtle but important distinction. A combobox is an example of a control.

    Third, so what you need is to have 2 combobos on your form; cboDegreeLevel and cboDegree. 

    The Rowsource for the first would be:

    SELECT DegreeLevelID, DegreeLevel
    FROM tluDegreelevel
    ORDER BY DegreeLevel;

    The Rowsource for the second would be:

    SELECT DegreeID, Degree 
    FROM tluDegree
    WHERE DegreeLevelID = Forms!formname!cboDegreeLevel
    Order BY Degree;

    In the After Update event of the first combo you want to put code in the After Update event that requeries the second combo.

    Note, If you actual have a field named Masters in your Degrees table, then your design is the best. And you may want to spend some time readin up on database design.

    ken sheridan say: I think Scott has inadvertently missed an important word in his reply.  He probably meant to say something more along these lines:

    Note, If you actually have a field named Masters in your Degrees table, then your design is the best - NOT!  And you may want to spend some time reading up on database design.

    If the table is designed in this way then it violates one of the fundamental rules of the database relational model, the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  You appear to be doing what is known as 'encoding data as column headings'.  As the rest of Scott's reply implies, each type of degree should be a value in a row in a separate referenced table (tluDegreelevel in Scott's example).

    A further point here is that only the degree obtained should as a column be stored in the table of Alumni, not the type of degree.  The latter is functionally determined by the former so to include columns for both would introduce redundancy due to the transitive dependency, and the table would be open to the risk of update anomalies as it would not be normalized to Third Normal Form (3NF), whose formal definition is:

    Third Normal Form: A relvar is in 3NF if and only if it is in 2NF and every non-key attribute is nontransitively dependent on the primary key.

    Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).
