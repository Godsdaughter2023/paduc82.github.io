zachd123 say:

The problem with the methodology suggested is that it involves hacking tightly versioned critical system files.  These system files are not intended to be compatible with older versions of Windows.  They are not intended to be compatible with newer versions of Windows.  When you create a hacked system file of this sort, it should only ever be used with that specific version of Windows (if at all).  When a system update comes out, that hacked system file is invalid and should no longer be used.  A "patching" utility that corrupts system memory to make that work should not try to make it work on the new system without some sort of validation.

This theme "version race" has always been a concern, but it's much more noticeable now that Windows is getting major updates multiple times a year.  Without a methodology change, hacked theme file usage is probably unsustainable over time.

If you think the theme system should be (more) extensible, you should create and upvote feedback upon that in the Feedback application.

natalieriddick say:

I figured this out myself.  

This had nothing to do with the printer.  This had to do with my antivirus - we had turned on application and device control that blocked programs from running in certain locations.  

What threw me off is that I disabled the AV and it still didn't work.  This morning I came in and looked at the device control logs and saw that it was still blocking it.  To be specific it was blocking dllhost.exe from running.  

I fixed that and now it is popping up fine.
