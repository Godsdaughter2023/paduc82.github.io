paul vine say: Not being possible to disable a "Form Control" type of buttons placed on a spreadsheet in Excel 2010 from VBA is, in my experience, a new bug (or a new feature) only found in Excel 2010 and not in Excel 2007 and earlier.

Detailed description of the bug (or the feature):

1) I have a workbook with sheets with buttons on them.
2) The buttons are "Form Control" type of buttons (as opposed to ActiveX type of buttons) and are located on the spreadsheets (and not on a VBA Form).
3) The buttons are enabled and disabled from VBA using statements similar to this:

        Sheets("Sheet with buttons").Buttons("Button 2").Enabled=False

4) The workbook functions as expected in Excel 2007, as well as in Excel 2003 (when converted from 2010 format to 2003 format). The buttons can be properly disabled and enabled. When they are disabled and the user tries to click on them they do  do not react (i.e. nothing happens).
5) The same workbook does not work as expected in Excel 2010. The buttons cannot be disabled. When their property ".Enabled" is set to False, the buttons remain enabled and when clicked execute the assigned to them actions. When queried, their property ".Enabled" correctly returns the value to which it has been set (but this value has no effect on the behaviour of the button, because the button remains enabled always). When a button is disabled and the user tries to click on it, the button responds as if it is enabled -- it is seen to sink down and then it executes the assigned to it action. This only happens in Excel 2010 and  not in any previous versions. This can be a bug (because it results in a counter-intuitive behaviour of the control and because it makes its property ".Enabled" useless. However, it can also be a new feature, intentionally introduced by Microsoft.

6) One possible partial workaround for this bug (or feature) is to assign to the buttons only VBA subroutines (as opposed any action possible in Excel). Then to write the VBA subroutines, assigned to the buttons, so that these subroutines first check the buttons' property ".Enabled" and only execute the assigned actions if this property is True. For example something similar to this:

Subroutine Button3_Execute_on_Click()
        If Activesheet.Buttons("Button 3").Enabled then
                ' some command or code goes here -- it will be executed
                ' if the button is enabled when the user clicks on it.
        End If
End Subroutine

This workaround is not ideal, because the user will be able to click on a button and will see the button sinking down, but nothing else will happen. This may be a bit confusing for the user, because if a button is disabled, it should not sink down when clicked.

Excel - I wish to have Excel open a spreadsheet with Autofit Row - Microsoft Community

jolly9roger say:

Excel - I wish to have Excel open a spreadsheet with Autofit Row Height already ON, is this possible ?

I wish to share a spreadsheet with less experienced users.

Anticipated amount of text entry to cells requires Autofit Row Height to be On and working.

Currently whether it is active on opening a spreadsheet seems to be hit and miss.

Is there a way to configure Excel - or a specific spreadsheet - such that this option is ON when the spreadsheet is opened ?

Any suggestions ?

bill manville say:

As I understand it, Excel will automatically fit the row when the user performs various actions on a cell in the row including entering data, clearing data, setting the font size, setting or clearing Wrap Text PROVIDED THAT the row height has not previously been set explicitly by the user and not been subsequently autofitted.

Certain operations do not cause autofit even when the row is in autofit "mode" - e.g. changing column width where the column contains wrapped text, entering wrapped text in a merged cell, a formula result changing.

The following VBA code will autofit all rows in all sheets in the active workbook
  Dim WS As Worksheet
  For Each WS In ActiveWorkbook.Worksheets
    WS.Rows.AutoFit
  Next

You could include such code in various events e.g Workbook_Open and Workbook_SheetCalculate

Or for a particular worksheet's Worksheet_Calculate event:
  Me.Rows.AutoFit

cortig say:
With IMAP, all e-mails are synced with the server. If you delete the
IMAP account, even though the e-mails are still on the server, the
local cache is purged and the e-mails are not accessible anymore.
 
There are two *very* simple workarounds:
1) Copy all the e-mails in the On My Computer section of Outlook to
keep a local copy.
2) before you delete the account, add the new IMAP account and copy all
the e-mails there from within Outlook.
 
In either cases, the e-mails will persist after you stop using the old account.
As for PSTs, Outlook doesn't use them for anything except for importing
archives from Windows. Outlook can only store emails in its own
identity and does not use external storage like the Windows version of
Outlook does.
