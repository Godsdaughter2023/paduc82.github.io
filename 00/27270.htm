MINIF, MAXIF and MODEIF with Standard Formulas - Microsoft Community

jeeped say:

MINIF, MAXIF and MODEIF with Standard Formulas

Summary
Technical Level: Intermediate Applies to: Excel, all versions This article will deal with providing functionality for pseudo MINIF(), MAXIF() and MODEIF() statistical formulas without the necessity for entering the formulas as arrays (e.g. without Ctrl+Shift+Enter). Formulas for MIN(), MAX() and MODE() with conditions are very commonly requested and the theory behind their implementation can easily be applied to other functions for similar conditional processing.
Details
The basis of each formula is the use of INDEX() in its array format to provide a range of numbers to calculate against. In order to get MIN(), MAX() and MODE() to only consider the values that match our criteria, a certain degree of mathematical manipulation will be applied to exclude numbers that do not meet the criteria. Each formula will be slightly different as the calculation to provide exclusion will be based upon the definition of the function being interpreted. For example, the following two formulas will return the same results.

                =MIN($C$2:$C$99)

                =MIN(INDEX($C$2:$C$99,,))

When used without adding criteria, the first is sufficient. However, to add criteria that will exclude some values from consideration, the latter form is necessary.

 

The next point to be made is the use of a boolean TRUE or FALSE in a formula. When using TRUE or FALSE in an Excel calculation, TRUE is treated as 1 (one) and FALSE as 0 (zero). We can use these conditional values multiplied by another number to return either the number (1 × «number» = «number») or zero (0 × «number» = 0).

 

These sample formulas will use 1E+99. This number is scientific notation for a very, very large number (a 1 followed by 99 0's). It will be used to increase or reduce some numbers within the data set thereby removing them from consideration while not affecting others.

 

My examples will use the following data matrix. Due to space limitations, a limited number of values will be used but statistical functions work better on larger data sets. MODE() for example will return an #N/A error if looking at a set of unique (no duplicate) numbers.

 

 

For the sake of these examples, we will use Apples as the criteria and I've applied some conditional formatting to highlight the rows that hold the Apple data.

 

For the MINIF() equivalent, we want to exclude any non-Apple quantities from consideration. It is probably safe to say that if we add 1E+99 to any non-Apple quantity, that quantity will not be the minimum.

                =MIN(INDEX($C$2:$C$99+(($B$2:$B$99<>$E3)*1E+99),,))

This formula looks at the quantities in column C. Any quantity that is not associated with Apples has 1E+99 added to it. As mentioned above, any of these increased numbers will not be the minimum so by extension, the result of the formula will be the minimum quantity for Apples. With the data shown in the image above, this result will be 3.

 

For the MAXIF() equivalent, we will reverse the process to exclude any non-Apple quantities from consideration. If we subtract 1E99 from any non-Apple quantity, that quantity will not be the maximum.

                =MAX(INDEX($C$2:$C$99-(($B$2:$B$99<>$E3)*1E+99),,))

This formula looks at the quantities in column C. Any quantity that is not associated with Apples has 1E9+9 subtracted from it resulting in a very low negative number. For all intents and purposes, it is highly unlikely that one of these adjusted numbers will be the maximum so by extension, the result of the formula will be the maximum quantity for Apples. In this case, the result is 15.

 

The conditional MODE() function presents a different problem. MODE() is defined as the most frequently occurring number so we have to individualize each non-Apple quantity. Adding 1E+99 is a start but you cannot repeatedly add the same large number and expect uniqueness. To gain truly unique values you should add 1E+9 multiplied by the row number that each quantity value is on so that a different amount is added for each row.

                =MODE(INDEX($C$2:$C$99+(($B$2:$B$99<>$E4)*ROW($2:$99)*1E+99),,))

 

1E+99 has an additional role here in that multiplying it by the current row and adding to the quantity will also use Excel’s 15 digit precision limit so that the left-most digits are the ones considered. The result is a set of unique, sequential, large numbers associated with non-Apple quantities that will be excluded in the MODE() calculation. The remaining Apple quantities will be the ones that MODE() selects a result from. The result from our sample data is 6.

john w. vinson say:

Well, you almost certainly do NOT want to "delete certain records" from the table. That would remove all record that Dr. Smith ever got a BS or MS!

Instead, you can use either a Top Values query or a Subquery to display only the most recent degree for the person. What's the context? Where do you want this information displayed?

Note that you should certainly have (at least) two tables - one for static biographical data about the person, related one-to-many to a table of degrees awarded. You don't want to have the biographical data repeated in separate records; not only does it waste space but much more importantly it leaves you open to incorrect data if one record gets edited and the other doesn't.

ken sheridan say: When posting a question like this it's important to give full information.  As it stands your question leaves room for ambiguous interpretations and leads to the sort of questions which John has raised.  

By 'the same table' are you referring to a table of students which includes columns representing both their personal data (names etc)  as well as the degree awarded.  If so this is the wrong design, as John has pointed out, as it includes redundancies which mean the table is not normalized to Third Normal Form.

Or do you mean that it's a table which models a many-to-many relationship type between Students and Degrees tables.  This is the correct design, so the model would comprise the following tables:

Students
....StudentID  (PK)
....FirstName
....LastName
....etc

Degrees
....DegreeID  (PK)
....DegreeTitle

and to model the relationship type:

DegreesAwarded
....StudentID  (FK)
....DegreeID  (FK)
....DateAwarded
....Subject
....ClassAwarded

By ClassAwarded I'm referring to (in the UK at least) degree classifications like First, Upper Second, Lower Second etc.

For each student, each degree they receive will be represented by a row in DegreesAwarded.  To return each student's details, along with the latest degree awarded (if any)  would therefore require a query along the following lines which restricts the outer query to the latest degree per student by a subquery, or to a Null where a student has not yet received a degree.  If you are only interested in graduates use INNER JOINs throughout and omit 'OR DegreesAwarded.StudentID IS NULL' from the outer query's WHERE clause.

SELECT FirstName, LastName, Degree,DateAwarded,
Subject, ClassAwarded
FROM (Students LEFT JOIN DegreesAwarded
ON Students.StudentID = DegreesAwarded.StudentID)
LEFT JOIN Degrees
ON DegreesAwarded.DegreeID = Degrees.DegreeID
WHERE DateAwarded =
    (SELECT MAX(DateAwarded)
     FROM DegreesAwarded
     WHERE DegreesAwarded.StudentID = Students.StudentID)
OR DegreesAwarded.StudentID IS NULL;

You can see how it works in the sample Northwind database which comes with Access where the following query would return the latest order per customer, if any, along with products ordered per order.  This will, unlike the above, return multiple rows per order of course as each order can include multiple products:

SELECT Company, [Order Date],[Product ID]
FROM (Customers LEFT JOIN Orders
ON Customers.ID = Orders.[Customer ID])
LEFT JOIN [Order Details]
ON Orders.[Order ID] = [Order Details].[Order ID]
WHERE [Order Date] =
    (SELECT MAX([Order Date])
     FROM Orders
     WHERE [Customer ID] = Customers.ID)

john w. vinson say:

That's different from your original question (the most recent degree), but it's pretty easy. Create a Query joining STUDENTS to DEGREES (I'd use the plural for the table name, singular for the fieldname), joining by StudentID; include the student name field and the DEGREE field. If you wish, you can sort by student name, or studentID, or degree, whatever suits. If desired put a criterion on the student name or the studentID to limit the query to that student.

Open the query as a datasheet and you'll see all that student's degrees.

To get the most recent degree you will need some additional field in the DEGREE table specifying the date the degree was attained, or a sequence number indicating that the BSc was reached first and the MSc next. You can use this field to construct a query that returns only the most recent date - but you can't without it!
