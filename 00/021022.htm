Unfortunately, the answer is not good. Access just doesn't work that way. If you are using Access tables for your data store, the back end has to be on a local server. Access needs to be able to create a lock file when it opens the back end and can't do that in the cloud. With most cloud storage, the cloud is synced with your local drive. The only way to store your data in the cloud would be to use an Azure (or other online) database and link to that.

andreas killer say:

To search for a cell format you must use Application.FindFormat to setup the format and Range.Find to find the first and last cell in your column. Then build a range from this cells, use INTERSECT or Range.Offset to map the range to another column and fill down this range.

BTW, it seems that Range.Find is not able to find a ThemeColor (added in XL2007), I guess you are using this format.

Andreas.

 

Sub Init()
  'Setup a test scenario
  With Range("A1")
    .Value = "Top"
    .Interior.ColorIndex = 15
  End With
  Range("B1") = "X"
  With Range("A20")
    .Value = "Bottom"
    .Interior.ColorIndex = 15
  End With
End Sub

 

Sub Test()
  Dim F As Range, L As Range, R As Range
  'Apply the format to search for
  With Application.FindFormat
    .Clear
    .Interior.ColorIndex = 15
  End With
  With Columns("A")
    'Find the last cell
    Set L = .Find("", LookIn:=xlValues, LookAt:=xlPart, _
      SearchDirection:=xlPrevious, SearchFormat:=True)
    'Find the first cell
    Set F = .Find("", L, LookIn:=xlValues, LookAt:=xlPart, _
      SearchDirection:=xlNext, SearchFormat:=True)
  End With
  'Build a range from this cells and map them to column B
  Set R = Intersect(Range(F, L).EntireRow, Columns("B"))
  'Fill first cell down
  R.FillDown
End Sub

You didn't respond as to whether or not you reviewed the mappings on both systems given an earlier statement that you expected everything to work the same because it was an Outlook to Outlook import/export

One thing that Outlook does is retain the last set of import settings on a given system so while something may work on one - it may not work on another depending what the user of each system did the last time.

Regarding CSV not working with "multi-line" fields - that would be true if the CSV file was not created with a field "text qualifier" character (usually a double quote at the start and end of each field

i.e. "First Value","Second Value" etc

When field qualifiers are used - Outlook imports things just fine. The same problem would occur with Tab-delimited text files that do not use text qualifier characters (and Tab-delim files can cause problems even in Outlook without text qualifiers). It's the text qualifier character fields that determines the actual start/end of a field even line break characters are contained within the field.

Unfortunately, there is no single universal rule regarding what makes up a valid "CSV" file so anything is possible. Multiline fields give a great many programs a lot of difficulty even when text qualifiers are used but Outlook is not one of those programs

I'm sorry to inform you that it is not possible to set one filter to have all those conditions to be considered on the rule. You need to manually enter each condition to your filter settings to have your preferred message rule.

  2. Click the cog icon located on the upper right of the page and select More mail settings.
