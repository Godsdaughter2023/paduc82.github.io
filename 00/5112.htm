vincentchoy say:

Hi Ripper2020,

Office365 uses SPF, DKIM and DMARC as a form of Spoof prevention. Note that these settings are set by the SENDER organization to prevent outsiders from Spoofing their email.

For mails coming into your organization, your anti-spam filter checks the incoming mail to see if they have  any SPF / DKIM inconsistencies (set by the email sender) and takes action based on the sender DMARC settings.

SPF defines the authorized sending servers. These are set when you configure your office365.

DKIM allows checking if your outgoing mail has been modified in any way

DMARC sets what you want the recipient anti-spam to do if it receives a mail that is inconsistent with SPF or DKIM (report, quarantine or reject). 

Note the onus is still on the recipient anti-spam to act, and there are ways a spoofer will circumvent SPF and DKIM so your DMARC doesnt trigger.

More reading can be found here - 

https://dmarc.org/wiki/FAQ#How_does_DMARC_work.2C_briefly.2C_and_in_non-technical_terms.3F

suzanne s. barnhill say:

Finding elusive "no proofing" text can be frustrating. When you select the entire document, you are selecting only the text layer. This doesn't include headers and footers and may not even include footnotes and endnotes. In one instance, I found "no proofing" text in an unused header. Here's how to ferret it out.

  1. Press Ctrl+H to open the Replace dialog and then select the Find tab (to my mind this is the easiest way to get to the Advanced Find without bothering with the Navigation pane).
  2. Click More>> to expand the dialog.
  3. Leaving the "Find what" box empty, click Format, then Language.
  4. Check the box for "Do not check spelling or grammar" and click OK. You'll see "Format: Do not check spelling or grammar" beneath the "Find what" box.
  5. Since "Find in" allows you to search only one layer at a time, it will probably be quicker to use "Find Next" (the likelihood is that there will be only one hit, anyway).
  6. Note where the found text is located. In the instance I cite, it was in an Even Page Header. Although "Odd and Even" was not currently enabled for headers and footers, Word had remembered the text entered for the Even Page Header, and it was formatted as "Do not check spelling or grammar." I could never have found this merely by inspecting the document in its current state.

doug robbins - mvp office apps & services (word) say:

Assuming that you are going to use Level 5 for the questions, in the Define new Multilevel list dialog, select that level and then select the .1.1.1. between the first and the last 1s and replaced them with a hyphen

and then, set the numbering for that level to Restart after Level 1

Also make any adjustment that you required to the Text indent and the position of the tab stop after the number in the Define new Multilevel list dialog

If necessary, assign that heading numbering to a new style, but continuing to use Heading 5 for it, with appropriate changes to the font etc. should not cause any problems.

It appears that BT Sport uses the Silverlight add-on, so please check that you don't have ActiveX filtering turned on:

1. Open IE.

3. Select Safety and make sure that ActiveX filtering does not have a tick against it.

If that isn't the issue please try:

1. Open IE.

2. Press Alt + X   on your keyboard.

3. Select Manage add-ons.

4. select Toolbars and Extensions on the left.

5. Select All add-ons from the dropdown menu half-way down on the left.

6. Locate Microsoft Silverlight in the list and check that it is Enabled. If it is not enabled, select it then the Enable button then the Close button.

A Secure Hypertext Transfer Protocol (HTTPS) page has content from a non-secure source.

Make sure all content in the page, including scripts, stylesheets, and images, are from HTTPS sources.

Access is denied

A script tried to access data from a source other than the host of the current page. The Same Origin Policy followed by Internet Explorer and other browsers allows scripts to access data only from sources with the same scheme, host, and port of the URL of the current page.

For example, if the current page is https://employees.mycompany.com, you can't access data from the following URLs:

  • http://data.contoso.com, because it is using HTTP instead of HTTPS.

  • https://somedatasource.com, because it is a different domain.

  • https://employees.mycompany.com:8888, because it uses a different port.

To correct this error

  • Investigate whether the API you're trying to call supports JSONP or CORS, which are two approaches that safely allow cross-origin scripting.

  • If you're trying to call a REST API, refactor this call to your server-side code, then expose a new REST endpoint for your client-side scripts.

    For additional information, look for online documentation related to the Same Origin Policy, JSONP, and CORS
