1969l46 say:

Apparently, the people who programmed the automation events for PIDs had no experience in making PIDs. If automation events are turned on (options->advanced), then changing an instrument type in one instrument changes the instrument type to the same type for ALL instruments with the same value in the bottom half of the instrument symbol (normally the loop number) to the same instrument type.

Suppose you have a typical level-control loop for a tank that has a level sensor (LE) that sends a signal to a level controller (LC) that, in turn, sends a signal to a flow controller (FC). Standard practice would be to have all 3 of these instruments labeled with the same loop number. So a user might insert the LE instrument and put it in loop 100 and then copy and paste that instrument to have 2 more instruments labeled as being part of loop 100. As soon as one of the LE instruments is set to LC, all of the instruments in loop 100 are changed to type LC. Then when the other instrument is changed to FC, once again, all the instruments in loop 100 is changed to LC. Fortunately, the automation does not change the instrument types if you just change the loop number to an existing loop number.

There are a couple workarounds:

  1. Turn off automation events. This will also turn off reconnecting pipelines that are split by a shape when the shape is deleted, or
  2. Give every instrument a unique loop number (or leave the loop number blank). ONLY when you have all of your instruments in the PID and their types set, then renumber them, or
  3. For existing instruments that you want to change, change the loop number to a unique number, change the instrument type, and then change the loop number back to the correct value (e.g., set loop 100 to loop 100ZZZ, change LE to LC, then change the loop number from 100ZZZ to 100.

Maybe they were trying to have loop numbers be the same for instruments that are connected and totally screwed it up.

Hello Louie_Solcorp,

The SPF update for your domain is incorrect. You can only have one entry v=spf1 on your DNS for a domain, if you have multiple SPF record published, it can not be validated and the recipient will mark your domain as 'Permanent Error'.

If you are only sending e-mail from Office 365 keep this SPF v=spf1 include:spf.protection.outlook.com -all  and delete the v=spf1 include:outlook.com ~all . Send a test e-mail to any domain and validate the header.

You should not be receiving this error.
