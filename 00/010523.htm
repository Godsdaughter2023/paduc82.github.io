albert d. kallal say:

So in the video I am using SharePoint and SharePoint lists. There is NO sql server involved in ANY context at all here.

 

In your case, you ARE using SharePoint lists. That is what 2010 publised Access databases use.

 

The linking of Access tables to SharePoint are also based on the NEW RELATIONAL features that are now part of SharePoint lists (you need SharePoint 2010 or later for this ability). This is a SharePoint "list"

ability, not Access ability.

 

In fact, you can use ANY version of SharePoint 2010 to do this. This means you can use the free edition of SharePoint and this linking of tables INCLUDES relational support that you see in that video.

 

So to be 100% clear:

 

SharePoint 2010 lists now have relational database features.

 

This relational feature when using with Access does NOT require Access web services and you do NOT have to publish your database (you can export the tables or what is termed up-size to SharePoint).

You still have use of the off line table, and the "sync" mode – but you only syncing data (not forms, and code).

 

As I noted, I still often publish that back end (as opposed to exporting to SharePoint). And the reason for this is as I note, often I may want to start adding a "few" web forms over time.  This means that the web forms I create are placed in the back end

 

In fact, if you published your Access application, then you are NOW USING SharePoint lists (all Access tables are SharePoint lists for Access 2010.

 

> I have heard, that in Sharepoint Lists are a limitations like the count of records and other things. Do you know something about that?

 

Yes, there most certainly a number of limitations to be aware of.

And more important there are additional limitations when using o365 as compared to hosted SharePoint or having your own SharePoint server.

 

While there are "many" small issues, the basic two issues with SharePoint tables are:

 

Your primary key is a single auto number column. For me, this was never a problem or issue since I always used the same auto number default of ID for the last 15 years anyway. However, for applications that use complex or compound primary keys, this setup is not supported with SharePoint. So just stick to a primary key of Auto number, and have your foreign keys in child tables relate back to that PK.  And how this is done is outlined in that video.

 

About the only other real "got ya" issue is when 365, then if you INITIAL table size is larger than 5000 records, then indexing settings don't stick (get saved) This in many ways may be a bug, but regardless, once the indexing of such columns is set correctly then your table can easily grow beyond 5000 records. 

 

As for how large of a table can you work with? Well, this is much depending on the type of application. I mean, we often see posts here that Access runs slow with one user – such application likely will run even slower or worse.

 

I certainly now experience good performance up to 30,000 or 50,000 records. Much of this will depend on how well the application is written. Any application that has to frequent UPDATE MANY rows simply will NOT work well at all with 365/SharePoint. In these cases, then using SQL server for the back end will be a FAR better choice.

 

However, if you have a good application that does not do silly things like open up a form without a where clause, then again your setup should perform quite well.

 

I would say that for most basic Access applications a good performance tip is to NEVER open up a form bound to a large table UNLESS I have some type of where clause.

 

So you can get great performance here. And remember, the local data cache will still remain in place (and if you are off-line, then you have to sync – the sync is supported here without Access web publishing).

 

In fact due to the "local cache" of data, then in some cases Access + SharePoint lists setup runs absolute circles around Access + SQL server. A good example is a report - no data need come down the network pipe with a the local data cache of data.

 

So tables cannot be huge, but more important is when launching a form bound to such tables, then you want to ensure you include some type of "where" clause to restrict the records loaded into that form.

 

In fact, be is a simple Access application, one with linked tables to SQL server, or one with linked tables to SharePoint?

 

In ALL THREE cases if you ask the person "what" they want to look at, such as

 

What invoice number to view:

 

If you now open the form with a where clause, EVEN with SQL server linked tables, and the form being BOUND to the base table, then ONLY the one invoice record comes down the network pipe. (this is true in all 3 cases above, and including when a network is involved – the whole table is NOT pulled down, but only the one record).

So in ALL 3 cases
Access split on a with a network
or

Access with SharePoint
or

Access with SQL server,

 

Only the ONE record comes down the network pipe. And this is even the case when the form is bound to the base table without a server system. But all 3 work this way.

 

So this idea of limiting records pulled into a form is a good idea even with just a single desktop Access application.

 

Such design considerations becomes FAR more import if you use SQL server, or in this case SharePoint.

If your application has temp data tables, then you can setup code to work them local. (don't attempt designs that use temp tables for data processing with 365 – keep those tables local).

 

Another danger or weak spot is Access applications that have to update many rows of data – that setup will run quite poor with linked lists to SharePoint/o365.

 

So if you using 2010 publishing, then all the tables ARE converted to SharePoint lists. The tables used for a published 2010 Access database are the SAME if you export the tables using the Access export option to SharePoint.

 

In both cases they are standard SharePoint lists, but as noted SharePoint 2010 now have relational features built in.

john mcghie say:
The Product Key is not on the computer. It never was.

When you typed the product key in, Microsoft checked it with the server and then issued you with a PID (Product ID).

If you have discarded the Product Key, then don't wipe the disk without taking a full backup.  There are two matching PID files stored in the Preferences folder and the Application folder.  If you restore both, the software will work without needing the product key re-inserted. The system will prompt you to go online.  It will handshake with the Microsoft server.  If both the PIDs are intact, the server will re-validate them and Office will work as normal.

If you miss either one, the system will prompt for the product key and you will need to buy a new copy.

Are you certain the product key was in the packaging?  Most copies are purchased online these days, and the product key arrives as an email.

Hope this helps

To make Google's I’m Feeling Lucky your default search provider, do this:

 

 

1.       In Internet Explorer 7 or 8 go to:  http://www.ieaddons.com/en/createsearch.aspx

http://www.google.com/search?q=TEST&btnI=3564

4.       Click “Install search provider”, and check "Make this my default search provider"
