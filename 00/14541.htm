Hi Judith. You may have created a look up field at the table level, which is discouraged practice. What you really need is create the Combobox on the form and add unbound Textboxes to display the other columns of the Combobox.

ken sheridan say: There are three ways you could handle this:

1.  Use correlated combo boxes so that you can drill down through the attributes, e.g. you select a commodity in one combo box, then a variety in a second combo box whose list is restricted to those varieties pertinent to the selected commodity, then a size in a third combo box and so on until you select the item in the final combo box.  The basic methodology for this is that each combo box's RowSource is an SQL statement which references the previous combo box up the line as parameter.  In each combo box's AfterUpdate event procedure those down the line are assigned to Null and requeried.

2.  Use a bound multi-column combo box to select the item, and unbound text boxes which reference the Column property of the combo box to show the other columns when an item is selected.

3.  Concatenate the values of the various attributes into a single string expression in the visible column of the combo box.

You'll find examples of the first two approaches in ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

You'll find an example of the third method in the NotInList demo in the same OneDrive folder.  In this little demo the opening form concatenates the various address columns with the contact's personal name to allow for two or more contacts of the same name to be distinguished.  The concatenation is done by means of the ConcatValues function from the basConcat module in the database.

This is NOT how relational databases work. The ONLY value from the Products table that should be in the Order Details table is the ProductID (the primary key in the Products table) as a Foreign Key. One of the principles of relational databases is that data should exist in one place only.

If you want to DISPLAY the additional values from the Product table on your form when you select a product, there are a variety of ways that are detailed in my blog on Displaying Data from Related tables. For reporting purposes, you would base the report on a query that joins the two tables on the ProductID.

Also please heed DBGuy's advice on doing lookups on the form level not on the table level.

=IF(condition, do if true, do if false)

=IF(ISBLANK(F35,"",IF(E35<=E28,"X","")
