ken sheridan say: Firstly, you appear to be encoding data as table names.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

When referencing a tree-structured hierarchy the referencing table should contain a foreign key only for the lowest kevel of the hierarchy.  In your case you have a hierarchy of two levels  FoodGroups and Foods, so only Foods should be referenced.  To also Reference FoodGroups would introduce redundancy as FoodGroup is functionally determined by Food, and would this be transitively dependant on the key,.  The table would consequently not be normalized to Third Normal Form and open to the risk of update anomalies.

The relationship type between Customers and Foods is many-to-many, so should be modelled by a table which resolves the relationship type into two one-to-many relationship types.  The overall model will therefore, in broad outline, comprise the following tables:

FoodGroups
….FoodGroupID  (PK)
….FoodGroup

Foods
….FoodID (PK)
….Food
….FoodGroupID  (FK)

Customers
….CustomerID  (PK)
….Customer
….etc

To model the many-to-many relationship type:

CustomerFoods
….CustomerID  (FK)
….FoodID (FK)

This table is 'all key', i.e. its primary key is a composite of the two foreign keys.

Diagrammatically the model is:

FoodGroups---<Foods---<CustomerFoods>----Customers

When dealing with tree-structured hierarchies it is often necessary to cater for a discontinuous hierarchy.  This would be so if you need to relate a customer to a food group as a whole, and not to individual foods.  This would be done by, for each FoodGroup, including a row in Foods with a Food value of 'N/A' or similar.  You'll find an example of this in DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

This little demo file includes, in the section on 'entering data via a form/subforms', a contacts form in which a country, region and city can be selected in correlated combo boxes.  You'll se that the country Monaco includes a region 'N/A', the country of Monaco comprising the city only, with no regional structure.  Note also that only the city combo box is bound, reflecting what I said in the second paragraph above about referencing only the lowest level of the hierarchy.

ken sheridan say:

As the list of all the food is approximately 150, i thought that it would be easier to have it in 5 different tables.

That is not how data is stored in a relational database.  It's what is meant by 'encoding data as table names' and violates the Information Principle, which is fundamental to the database relational model.

You should have a single Foods table, which can reference a FoodTypes table in the way I described.  This in turn can reference a table of 'subsections', which in turn can reference a table of 'choices'.  The 'choices' table should be the only one referenced by the Customers table via a CustomerChoices table which models the many-to-many relationship type between customers and choices.

You should not have check boxes bound to Boolean (Yes/No) columns to enter customer choices.  That again violates the Information principle, in this case by encoding data as column headings.

You might understandably be reluctant to undo the work you've done to date, and go back to square one, but the first thing to do when in a hole is to stop digging.

ken sheridan say: No database should contain an indeterminate relationship.  All relationships between tables should  be one-to-many, or in rare circumstances, one-to-one.  In each case the relationship should be between the primary key of the referenced table, and the corresponding foreign key of the referencing tables.  By definition a primary key is indexed uniquely (no duplicates).  A foreign key in a one-to-many relationship type should be indexed non-uniquely (duplicates allowed).

All relationships should have referential integrity enforced.  An unenforced relationship serves no useful purpose whatsoever.  The following is an image of relationships from my DatabaseBasics demo, to which I gave you a link earlier, which illustrates a simple contacts model.



As you can see all of the physical relationships are one-to-many.  There is also a conceptual many-to-many relationship type between Contacts and Employers, which is modelled by a ContactEmployers table, which resolves the relationship type into two one-to-many relationship types.  The CustomerFoods table in the model I suggested earlier does the same, modelling the many-to-many relationship type between Customers and Foods.

steve rindsberg say:

Can you put an example PPT/PPTX file someplace where we can download it?  Better yet, an example as saved from 2013 and an example or screenshot of what you see in the Mac version.

But as a general bit of advice, I'd avoid using EPS graphics in PowerPoint.  I've been using PowerPoint since version 3, have used EPS graphics in other software all that time, and have never had anything but trouble with EPS+PowerPoint.  

I'd try PNG if possible.  But again, if that doesn't work, put an example where we can see it.

EPS isn't so much a graphics format as a programming language that can describe graphics (in both raster and vector format as Rich has mentioned).  Typically, EPS also includes a (usually but not necessarily) raster/bitmap preview image for the computer to display and/or to print to non-PostScript printers.  It uses the PostScript content of the EPS when printing to PS printers.

Since there aren't that many PS printers out there, EPS have been something of a problem for developers; if they handle the EPS as per spec, a lot of the customers will be unhappy because the preview image they see and print is usually low rez and kinda nasty.

Instead, recent versions of PPT actually behave something like a PS printer ... they interpret the programming code in the EPS and convert it to a Windows Metafile (PC) or Enhanced Metafile (Mac).  It does quite a good job, at least for simple graphics, but it's not a real PostScript printer, so sometimes the graphics get munched.
