tony jollans say:
> If I convert the document to a template (.dotx), ...
 
If you, instead, convert the document to a macro-enabled template (.dotm)
you can then set a reference to the original .dotm template, and it will be
opened whenever the first template is used.
 
Firstly, have both templates open. Then it helps if you rename the VBA
Project of the original .dotm - this is not essential but it makes for
easier identification. Next, in the VBE, select the new template (that was
dotx but is now dotm) and goto Tools > References and find - and check - the
original dotm template, and click on OK. You can confirm you have a
Reference by expanding the References branch of the new template in the
Project Explorer.
 
In order to force Word to save the Reference with the project, you must make
use of it. Create a dummy procedure in the original template, and in the new
template (in ThisDocument will do) create a procedure that calls it
(project_name.ThisDocument.dummy_name - whatever you called all the bits).
 
Now save both Templates, and when you create a new document based on the
first one, the second one will be opened as well. There are a couple of
gotchas - the referenced template is located by the same rules as an
attached template - that is, if a template of its name exists in the same
folder as the referencing template, that one will be used, otherwise Word
will look in the hard coded path where it existed when the reference was
first set up. Second, in Word 2007, you will not normally get prompted to
enable macros in both templates, so the referenced one needs to have them
automatically enabled - probably by having it in a trusted location.

dale howard [mvp] say:

hshayhorn --

 

There is an easy way to do this.  Display the Gantt Chart view and select the task in question.  Right-click anywhere in the Gantt Chart pane and select Show Split on the shortcut menu.  In the Task Form pane, click the Task Type pick list and select the Fixed Duration item, and then click the OK button.  In the Task Form pane, enter 4 days in the Duration field and click the OK button.  In the Task Form pane, change the Work to 16 hours and click the OK button.  Now you have 16 hours of Work spread over the 4 day Duration of the task.  Lastly, you should click the Task Type pick list again and select either Fixed Units or Fixed Work, and then click the OK button.  Hope this helps.

First off, you need to understand how conflict resolution works with
Jet Replication.
If you don't define a conflict resolver routine, Jet will
automatically resolve all your conflicts according to its default
rules (which are fairly complicated, but pretty sensible). When as
conflict is reported, it's only informational -- the conflict has
already been resolved and the data in both replicas that
participated in the synch are identical.
Well, then, what's the point of all this about "resolving
conflicts?"
The point is that Jet keeps track of how it resolved the conflicts
and maintains a copy of the losing data so that you, the user, can
decide if Jet guessed right or not, and either accept Jet's default
resolution, roll it back, or merge the records.
So, when you check for conflicts and resolve them, what you're doing
is just checking that Jet did it right. If you trust Jet's rules in
all cases, you could just delete all the conflict records and be
done with it and never bother. Your data would be in a consistent
state in all your replicas, though it might not have exactly the
resolution).
(well, there are exceptions to that -- certain kinds of replication
errors are reported as conflicts, and those can prevent the replicas
from being identical after the synch. Those are usually related to
RI)
So, in reality, you have two choices:

1. use the SQL Server replication conflict resolver (Access and SQL
Server have used the same one, in fact, since A2000/SQL Server 2000)
anyway, which is what you had before A2010. This entails little more
than finding the executable from an earlier version of Access and
copying it to the machine on which you want to resolve conflicts.
2. write your own routines to check the conflict tables and present
to the user only the types of conflicts you feel need a human being
to check that they were done properly. For instance, if you have a
Updated date/time field, you could check that the losing record has
a date/time that is earlier than the winning record, and accept the
resolution by deleting that conflict record. Then you'd only have to
present the ones where there's a tie or the winning record was for
some reason earlier than the losing record.
So the issue is perhaps not as complicated as people might think
(because they haven't ever really understood how conflicts really
work in the first place).

 Sub ViewSyncConflict()
      Dim Db As Database
      Dim Td As TableDef
      Dim i As Integer
      Set Db = CurrentDb
      ' Step backward through the TableDefs collection so you
      ' do not miss any tables when you delete conflict tables.
      For i = Db.TableDefs.Count - 1 To 0 Step -1
         Set Td = Db.TableDefs(i)
         If (Td.ConflictTable <> "") Then
            ' Open a recordset based on the conflict table.
            ' Insert code to do conflict resolution.
            ' Delete the conflicting record when you are done.
            ' Delete the conflict table when all its records are deleted.
            ' Set the ConflictTable property to "".
         End If
      Next i
   End Sub
  
  
Sub ViewSyncError()
      Dim Db As Database
      Dim Rs As Recordset
      Dim MsgString As String
      On Error GoTo ErrorHandler
      Set Db = CurrentDb
      Set Rs = Db.OpenRecordset("MSysConflicts", dbOpenSnapshot)
      Rs.MoveLast
      If Rs.RecordCount > 0 Then
         Rs.MoveFirst
         Do Until Rs.EOF
            ' Build the error message string.
            MsgString = "Table ID: " & Rs!BaseTableGUID & vbCr
            MsgString = MsgString & "Record ID: " & Rs!BaseRowGUID & vbCr
            MsgString = MsgString & "Failed Because: " & Rs!Reason
            MsgBox MsgString
            Rs.MoveNext
         Loop
      End If

ExitProc:
      Exit Sub

ErrorHandler:
      ' If the MSysErrors table is empty...
      If Err.Number = 3021 Then
         Resume ExitProc
      ' display any other error that occurs.
      Else
         MsgBox Err.Description
         Resume ExitProc
      End If
   End Sub


 Sub ViewDesignError()
      Dim Db As Database
      Dim Rs As Recordset
      Dim MsgString As String
      On Error GoTo ErrorHandler
      Set Db = CurrentDb
      Set Rs = Db.OpenRecordset("MSysSchemaProb", dbOpenSnapshot)
      Rs.MoveFirst
      Do Until Rs.EOF
         ' Build the error message string.
         MsgString = "Operation: " & Rs!Command & vbCr
         MsgString = MsgString & "Failed Because: " & Rs!ErrorText
         MsgBox MsgString
         Rs.MoveNext
      Loop

ExitProc:
      Exit Sub

ErrorHandler:
         ' If the MSysSchemaProb table does not exist.
         If Err.Number = 3078 Then
            Resume ExitProc
         ' If the MSysSchemaProb table is empty.
         ElseIf Err.Number = 3021 Then
            Resume ExitProc
         ' Display any other error that occurs.
         Else
            MsgBox Err.Description
            Resume ExitProc
         End If
   End Sub

Avoid them...

They are genuine Microsoft product keys, they are actually retail licenses, but it is intended for a particular product channel either the Microsoft Software Developer Network (MSDN) or TechNet for IT Professionals who pay a subscription fee. The main purpose is for evaluation purposes. The great thing about them, unlike trial Microsoft software, MSDN or TechNet keys don't expire.

Because the agreement under which the subscription is provided is a single license, none of the software should be distributed outside of it. Even though its $50,000 worth of licenses, it is for one person only to use and no one else. Unfortunately, regardless of the licensing terms, persons still abuse the program, either giveaway product keys or resell it on auction sites. Microsoft licenses it in good faith that customers won't do so, but I guess human nature wins out.

If you bought an MSDN licensed key, which carries up to 10 activations unlike full packaged retail licenses which only carry 1 activation, the person who sold it to you probably sold it to 10 other persons. Somewhere along the way, one of those persons might have installed it on a second system, activated it, because it went past 10 activation threshold, Microsoft detected it that it was being abused and blocked the key from further use.

If you want to be guaranteed a genuine license, purchase it through the Store or Amazon.com (not the market place section).

An immediate sign its not genuine is if they are selling the Pro Plus edition, which is only available to large organizations.
