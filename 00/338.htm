DSUM in case of Criteria of Dates - Microsoft Community

faraz a. qureshi say:

DSUM in case of Criteria of Dates

While I have a control on a form representing DATE type data labelled LASTDT I want to bring the sum of the numerical field LastBen from the table Latest. DSUM doesn't seem to be working with the following syntax being inserted in the expression builder for the control source:

=DSum("LastBen","Latest","[LastRep]=" & [LASTDT])

What could be the error?


john w. vinson say:

You need # as a date delimiter: Number fields use no delimiters at all, Text fields use ' or ", Date/Time fields use #. Try

=DSum("LastBen", "Latest", "[LastRep]= #" & [LASTDT] & "#")

or, better because it controls for vagaries of how LASTDT might be entered, explicitly format the LASTDT:

=DSum("LastBen", "Latest", "[LastRep] = #" & Format([LASTDT], "yyyy-mm-dd")  & "#")

This does assume that LastRep is just a pure date with no time portion.

How to ignore empty field on Access 2013 report - Microsoft Community

caesar3 say:

How to ignore empty field on Access 2013 report

I've built a report that lists the names of teachers by school year for a local historic (now closed) school. I started with an Excel spreadsheet that holds the data and linked that spreadsheet to a table in Access 2013.

The spreadsheet has these fields: SY (for School Year), Prefix (Mr., Mrs., Ms., Miss), LastName, and FirstName.

In Access, I built a very simple select query from the linked table. The query includes a calculated field (TeacherName) that combines Prefix, FirstName, and LastName.

Next, I built a report that lists the teachers, grouped by SY. Here's my problem: Not all records have a prefix. I gleaned the data from a Web site, where they did not provide a prefix for every teacher. The few men in the list are no problem, of course, but the women present a different situation. The printed report looks okay, except there's an empty space in front of the names that lack a prefix and the left edge looks ragged. I'm sure there's a way to force the report to ignore the missing prefix or to modify the query for the same purpose. I just don't know how to do it, and I hope someone here can show me how.

john w. vinson say:

You can use the quirky fact that both the & and + operators concatenate strings, but they do so a bit differently: [A] & [B] returns just the contents of B if A is null, but [A] + [B] returns NULL if either argument is NULL. So you can change your calculated expression to

([Prefix] + " ") & [FirstName] & " " & [LastName]

This will show

Mr. David Roberts

Miss Lydia Smith

Gene Aldrich

and so on; if there is a prefix it will be included followed by a trailing blank, but if there is none the parenthetical expression will just become a NULL.

It's not obvious to me how the gender of the teacher interacts with the presence of a prefix, but I hope this trick solves your problem.

jm6 say:

I'm a beginner at this, and have not been able to find this documented.  Here's what I've found through experimentation:

  • When in the Macro Designer you select the SetProperty method and type into the Control Name text box, type F. t\ The IntelliSense will provide a list of existing objects, including Forms.  Once you've entered that, type a !, and it'll show all the forms.  With every object you enter, type either a dot, a !, or a >.  Some won't do anything, but at least one will make IntelliSense list more objects.
  • Once you have the object you want, the Property pulldown box will show a list of its properties.
  • When you run the app locally, it'll work.
  • If you've published it and try to run it in a browser, it won't work (if it's a form on a different Navigation page).  It'll give the error you mentioned (about the control not existing).  It's not that it doesn't exist - I think it's just that security when running in a browser is more restrictive, and doesn't allow access to the control.
  • So it appears that you cannot set the property in a form on a different page.  You can only set properties on objects within the current form.  So you have to wait till you're in a macro on that form before you can set the property.

johnlv say:

Roberto-

Are you using an embedded macro or calling a saved macro object?  In either case, the context of the macro is that of the caller.  Because the macro is called from F1, then C2 tries to reference a control on F1, not F2.  You have to fully qualify the name of the second form and its control if you want to reference something on F2 in a macro called from F1.
