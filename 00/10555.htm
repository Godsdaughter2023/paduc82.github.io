Great.   Thanks for that confirmation.   So it looks like a DNS problem but just with that host name.   Then there are two ways to get past the possibility that that is what the error message actually means, e.g. no addressability for that host.

1.    You can try proving that IE should have a usable lookup in the dnscache

2.    You can insert a temporary override for it in your HOSTS file.

 

Have you tried using the Diagnose connection problem... button that comes up with that message?  Sometimes it is enough to get past the problem.   There is an automatic Refresh done if the Troubleshooter thinks there is nothing wrong.   And as part of its troubleshooting it may load a lookup into the dnscache.

You can see how this happens by opening an Administrator Command Prompt (aka elevated cmd.exe window).    Run... (e.g. press Win-R and enter:)   cmd.exe  to start an ordinary Command Prompt and then pointing at its Taskbar icon press Ctrl+Shift  and Click.   Then follow the prompts to allow it to be elevated.  In fact, it is looking as if we don't need to elevate the Command Prompt to do what I am going to suggest but it can't hurt to have the full command set available if needed.

In the Command Prompt's command line enter:

ipconfig  /flushdns

The main benefit of this command IMO is to simplify inspection and analysis of any other commands and actions you may take which change the dnscache.    However, if you had not cleared your HOSTS file you could see entries due to it, so now I would suggest also an initial

ipconfig  /displaydns

 

to either verify that it is empty or to see what it now contains.

 

Next switch to your IE window with the problem message and start the troubleshooter.   Let it proceed through up to the last step when you would close it.   Take note of any messages it gives but usually it will indicate "no trouble found".

So switch back to the cmd window and see what it did, if anything

OMG.   I think I see a potential problem with the  www.facebook.com  lookup.   There are initial  IPv6 entries for it.   I don't know what  IE  would try to do with them so they might be contributing to your problem symptom.   More significantly if it tried to choose an IPv6 address and the rest of your Networking support really couldn't support it there would have to be error recovery, e.g. at least a timeout and retry which would switch to an IPv4 address instead.   I have read recently someone complaining that IE does not do such error handling as well as other browsers.

 

<aside>

I actually did a  ping  to load the lookup and then noticed something unexpected in the displaydns output, so then I did

nslookup  www.facebook.com

 

and that's when I noticed the IPv6 addresses.   Fortunately my undifferentiated ping did not try to use one.

</aside>

 

So, in case that is the problem for IE and in case the troubleshooter is still loading an IPv6 address what I would try is first doing another flushdns (e.g. to get rid of any confusion that the troubleshooter might have caused by perpetuating use of IPv6) then do:

ping  -n 1  -w  1  -4   www.facebook.com

 

Check what you accomplished with another  displaydns.   BTW  Tip:  the cmd window has an F7 shortcut you may like if you find you are re-typing the same command more than once.  

Another reason for using the F7 shortcut is that there is a  Time To Live  value given for each record (in seconds) so you want to be as quick as possible to let IE try to take advantage of your cached lookup.   These facebook entries that I'm seeing seem rather short so you would have to be on your toes to get the diagnostics done and the Refresh started.   Of course if the troubleshooter already loads an appropriate lookup then it would be best just to let IE try to take advantage of them as soon as possible.  The problem is that you can't just assume that it is usable, so that's why it is necessary to do the diagnostics and do them efficiently.

 

After checking the dnscache and fixing it up, switch back to the IE window to close the troubleshooter, allow IE to do its automatic Refresh and see if IE will use your IPv4 address.

 

 

If your facebook addresses are more complicated than that, e.g. if they involve it being an alias of a caching host (a very common technique for large sites) we would have more to do to ensure that IE would have a usable lookup.

 

As for option 2,  inserting a lookup in your HOSTS file, that might be the most effective workaround, at least while testing.   So, FWIW here is the IPv4 address that my DNS shows and the syntax that you would use to append it to your HOSTS file:

69.171.228.74   www.facebook.com

Just in case, you should check that this is the same address that your DNS is telling you is one of the ones to use to connect to that host.   Also, if you did that you would have to remember that it was there, e.g. in case the lookup changed sometime later and this address was removed from use or reassigned to a different host.


 

HTH

bob jones aka: cybertaz say:

I can't be certain from a screen shot, but I believe what you're seeing is caused by the use of a Column Break, indicated by the blue line that appears below the word Together:.

The overriding issue, though, is that you are attempting to use columnar alignment in a manner contrary to its purpose :-) Columns in Word refer to snaking or newspaper-style columns which flow top-to-bottom, left-to-right.

What you're attempting to achieve is better handled by the use of a Table, where text on the left pertains to or is relative to content to its right. I'd suggest using a 2 column, 1 row Table such as what is shown in the screen shot below:

The Table paragraph is formatted with 6 points Before & 6 points After Paragraph Spacing, but that can be adjusted as you'd prefer. Also, the Borders can be removed from the Table if you don't want them to be visible & the width of the columns can be adjusted as necessary. A Table also provides more flexibility in virtually all respect for this type of project as each cell can be dealt with individually without impacting the others.

bob jones aka: cybertaz say:

The problem is that (combined with forcing the Table down on the page) you Split the right-hand cell which causes it's lower portion to be treated as a separate row. There is no [practical] way to prevent a Table from splitting across a page break. However, this is one of the few instances where a single empty ¶ comes in handy...

Click in the empty ¶ you've inserted above the word Together, then go to Format> Paragraph - Line & Page Breaks. Check the box for Keep with Next.

Another option is to merge those 2 cells so that the Table comprises only 1 row. If you want some form of demarcation separating them use Format> Borders & Shading to apply a Bottom border to the upper of the 2 paragraphs or a Top border to the lower one. [Note that a paragraph border will remain visible even when Table borders are not displayed.]

I'm not sure why you're using all those ¶s to force the Table toward the bottom of the page, though. As I suggested before, that method creates more problems than it resolves. However, if you insist on that method just press return a few more times & the Table rows will rejoin on the following page.
