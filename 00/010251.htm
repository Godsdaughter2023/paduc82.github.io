ken sheridan say: If you do use one of the methods from the MS site to which Hans referred you, the second, using the GetToken and InParam functions is more reliable as it avoids any possible substring mismatches which the first method can sometimes produce.  A better method in my view, however, is not to use a parameter in the report's query at all, but to filter the labels report by selecting one or more values from a multi-select list box on an unbound dialogue form, which also includes a button to print the labels report.

To do this set up the list box's RowSource property like this:

SELECT DISTINCT [MP] FROM [YourTable] ORDER BY [MP];

And set the MultiSelect property to Simple or Extended as preferred.

Add a button to the form to print the report, called rptLabels in this example, with the following in its Click event procedure:

    Dim varItem As Variant
    Dim strMPList As String
    Dim strCriteria As String
    Dim ctrl As Control
    
    Set ctrl = Me.lstMPCodes
    
    If ctrl.ItemsSelected.Count > 0 Then
        For Each varItem In ctrl.ItemsSelected
            strMPList = strMPList & ",""" & ctrl.ItemData(varItem) & """"
        Next varItem
        
        ' remove leading comma
        strMPList = Mid(strMPList , 2)
        
        strCriteria = "MP In(" & strMPList & ")"
        
        DoCmd.OpenReport "rptLabels", WhereCondition:=strCriteria
    Else
        MsgBox "No codes selected", vbInformation, "Warning"
    End If

If you wish you can also add a button to select all labels, with this code in its Click event procedure:

    Dim n As Integer
    
    For n = 0 To Me.lstMPCodes.ListCount - 1
        Me.lstMPCodes.Selected(n) = True
    Next n

and one to clear the selections, with:

    Dim n As Integer
    
    For n = 0 To Me.lstMPCodes.ListCount - 1
        Me.lstMPCodes.Selected(n) = False
    Next n

Instead of opening the report you'd now open the dialogue form, select one or more codes and click the button to print the selected labels.  You can if you wish set the report up to open the form automatically by including code in the report's Open event procedure:

    Const FORMNOTOPEN = 2450
    Dim frm As Form
    
    On Error Resume Next
    Set frm = Forms!YourFormName
    If Err = FORMNOTOPEN Then
        DoCmd.OpenForm "YourFormName", OpenArgs:=Me.Name
        Cancel = True
    Else
        If Err <> 0 Then
            ' unknown error
            MsgBox Err.Description, vbExclamation, "Error"
        End If
    End If

If you wish you can close the form in the report's Close event procedure with:

    DoCmd.Close acForm, "YourFormName"

As the Click event procedure for the button on the form use:

    Const REPORTCANCELLED = 2501
    
    On Error Resume Next
    DoCmd.OpenReport Me.OpenArgs
    Select Case Err.Number
        Case 0
        ' no error so do nothing
        Case REPORTCANCELLED
        ' anticipated error so do nothing
        Case Else
        'unknown error so inform user
        MsgBox Err.Description
    End Select

Because the report passes its name to the form as the OpenArgs property this form can be used with more than one report which takes the same parameters.  Note that the form's code handles the error which would occur if the opening of the report is cancelled.  This is to cater for reports whose opening is cancelled in their NoData event procedure if there is nothing to report.  If you open the report directly in code from elsewhere in your application you'd also need to handle this error as the code in the report's Open event procedure cancels the opening of the report if the dialogue form is not open, so say you want to open the report from a button on a form which provides a choice of available reports you'd use code the same as the above but instead of Me.OpenArgs you'd put the name of the report, in quotes, or more likely a reference to a control on the form in which you select the report.
