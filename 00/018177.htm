macropod say:

In that case, you'll need either:

• an empty rectangle the size of your bulleted content placed 'behind text' or a transparent one place 'in front of text' and IF field that determines whether to display it; or 

• two versions of your bulleted content (one outlined, the other not) and an IF field that determines which version to display.

For the first approach, you'd use a field coded like:

{IF{MERGEFIELD Contition}= "True" ""}

and anchor the rectangle 'relative to character', between the "" pair. This will likely bit a bit fiddly to implement. The field itself would be inserted at the start of the first bullet point.

For the second approach, you'd use a field coded like:

{IF{MERGEFIELD Contition}= "True" "• Bullet point¶

• Bullet point¶

• Bullet point¶

• Bullet point¶

• Bullet point¶

" "• Bullet point¶

• Bullet point¶

• Bullet point¶

• Bullet point¶

• Bullet point¶
"}

and anchor the field to the paragraph preceding where the bullet points now appear.

daniel pineault say:

There are some fundamental structural issues, and I think this is in part responsible for your current headaches.

Firstly, I avoid any Lookup fields, no exceptions, ever!  If you need lookup, then create the proper related table.

Fields should never store more than one value, so as soon as I see Part_Level housing multiple values, I know that the structure is wrong.  The Part Level needs to be broken off into its own related table, then each value get's its own record.

There is no need to have a field such as Part_Level_2, this should simply be part of the lookup table as an extra field that can be referenced whenever needed.

ken sheridan say: This is one of a number of reasons against using a multi-valued field (MVF), but instead modelling the many-to-many relationship type between Roster_Info and Part_Levels in the conventional way by a table which resolves the relationship type into two one-to-many relationship types.

As you are already restricting the query on the Roster_Info.Part_Level.Value, you might be able get around this by using  a true aggregation operator such as MAX rather than the FIRST operator, which, to all intents and purposes returns an arbitrary value in any case.  You can avoid the use of the FIRST operator completely as the FirstName and LastName columns are functionally determined by the key of Players, so can be included in the GROUP BY clause.  The following might work, though I would not be unduly optimistic:

SELECT Players.Player_ID, FirstName, LastName,
MIN(Season.Fall_Year) AS MinOfFall_Year,
MAX(Season.Fall_Year) AS MaxOfFall_Year,
[MaxOfFall_Year]+1 AS Maxx_Fall_1,
[MaxOfFall_Year]-[MinOfFall_Year]+1 AS Num_Yrs_Plyd,
[MinOfFall_Year] & "-" & Right([Maxx_Fall_1],2) AS Career,
MAX(Roster_Info.Part_Level.Value) AS PartLevel
FROM Season INNER JOIN (Players INNER JOIN Roster_Info
ON Players.Player_ID = Roster_Info.Player_ID)
ON Season.Season_ID = Roster_Info.Season_Play
WHERE Roster_Info.Part_Level.Value LIKE "*Var*"
GROUP BY Players.Player_ID, FirstName, LastName;

The best solution would be to change the model to a conventional one, however, by eliminating the multi-valued field and modelling the relationship type by a table.  You'll find an illustration of how to automate the insertion of rows into the table which models the relationship type in MVFCorrector.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates how to insert rows into a StoreStocks table which models the relationship type between stores and stocks, with examples where a MVF currently stores the data internally as a value list, or references another table.  In the former case, the referenced table, Stocks, analogous to a Part_Levels table in your case, has to be created and populated first, but this is a trivial task where the number of values is so small.
