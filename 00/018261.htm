hal hostetler - mvp-outlook say:

The standard behavior for deleting an item in an IMAP account is that the message is not moved anywhere, it is marked for deletion with a strikethrough line.  in Outlook 2013, go to "File > Account Settings", select your IMAP account, click the "Change" button, click the "More Settings" button on the right, click the "Advanced" tab, and in the "Deleted Items" section at the bottom, REMOVE the checkmark from the "Mark items for deletion, but do not move them automatically" checkbox, then click OK.  This changes Outlook's behavior so that messages are moved to Deleted Items.  The reason you can currently see Deleted Items that are marked with a strikethrough is because your View setting is NOT set to "Hide messages marked for deletion". if you change to this View setting, they'll all disappear. With that in mind, you might also want to add a checkmark to the "Purge items when switching folders while online" checkbox in the same section.  This will purge any items marked for deletion when you switch folders

VBA TextBox Decimal places - Microsoft Community

christiansierra1 say:

VBA TextBox Decimal places

Hi I want to use a text box where i can write 12.34 with the dot, i cannot add numbers with decimals. I am managing weight which requires decimal places and the TextBox does not allow me to input decimals. any code?

macropod say:

The following code, attached to a textbox (TextBox1) on a UserForm, ensures that only numbers can be entered. Any non-numeric keystroke generates a beep and the offending character is deleted.

Private Sub TextBox1_KeyPress(ByVal KeyAscii As MSForms.ReturnInteger)
If Not (KeyAscii > 47 And KeyAscii < 59) Then Beep: KeyAscii = 0
End Sub

Or, if you want to allow the user to input decimal values:

Private Sub TextBox1_KeyPress(ByVal KeyAscii As MSForms.ReturnInteger)
If Not (KeyAscii > 47 And KeyAscii < 59) Then
  If KeyAscii = 46 And Len(TextBox1.Text) = 0 Then
    TextBox1.Text = "0"
  ElseIf Not (KeyAscii = 46 And InStr(TextBox1.Text, ".") = 0) Then
    Beep: KeyAscii = 0
  End If
End If
End Sub

Alternatively:

Private Sub TextBox1_KeyPress(ByVal KeyAscii As MSForms.ReturnInteger)
Select Case KeyAscii
  Case 48 To 59
    ' Allow digits - do nothing
  Case 46
    ' Allow only one dot symbol
    If InStr(ActiveControl, ".") Then
      Beep: KeyAscii = 0
    Else
      If Len(TextBox1.Text) = 0 Then TextBox1.Text = "0"
    End If
  Case Else
    ' Block others
    KeyAscii = 0
End Select
End Sub

Note: With the second & third subs, only a single period can be input and a leading ‘0’ is automatically inserted if the period is the first character input by the user.

john w. vinson say:

I see a structural problem at the root of your difficulties. You say "value in the list box contain some kinds of coding, then they are very similar, just a little bit different at the end. " That means to me that the ShortDesp field is a composite, containing at the very least two disparate pieces of information, probably more: it looks like a date, some other numbers, and a three letter code. For this purpose you're trying to distinguish records based on the three letter code.

This is a very good example of why Codd and Date, back in the mid-20th century, insisted that fields should be "atomic" - having only one piece of information. If you have a date, a number, and a code, you need three fields, containing a Date, a number, and a code! You can easily use conditional combo boxes to select the date, and the number, and the code, at each step selecting only those values pertinent to the previously selected values.

But... if you're copying this 16-byte string from a report and want to use it as a filter, why mess around with the listbox at all? You're copying a string from a report, into a textbox; copying that textbox into a listbox; finding that record in the listbox; and then using that selection to filter the second form. Why not just cut out the middleman? You HAVE the filter right there in the textbox. Or am I still missing something?
