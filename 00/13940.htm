albert d. kallal say:

You can certainly blow out the variable holding the forms object. However, when that object is forced to go out of scope, then the objects “Terminate” event WILL fire. That applies to standard class objects in Access, and ALSO to form objects.

So you can blow out the variable – but the forms “events” will STILL fire. The same goes for any custom class object you create in access. Setting the variable to “nothing” or killing it will STILL fire the close event.

The following code demonstrates how you can kill any “instance” of your form. You can cut + paste the following into a new code module and play with this

Option Compare Database
Option Explicit

Dim colForms      As New Collection

Sub Open5Forms()

   ' open the form 5 times
   Dim i    As Integer
   Dim f    As Form
  
   For i = 1 To 5
      Set f = New Form_Form6
      f.Caption = "form" & i
      f.Visible = True
      colForms.Add f, "form" & i
   Next i
  
End Sub

Sub CloseAForm()

   Dim i       As Integer
  
   i = InputBox("what form instnace to close")
  
   colForms(i).Myclose
   colForms.Remove (i)
  
End Sub

Sub CloseAForm2()

   Dim i       As Integer
   Dim f       As Form
  
   i = InputBox("what form instnace to close")
  
   Set f = colForms(i)
   Set f = Nothing
   colForms.Remove (i)
  
End Sub

Sub showForms()

   Dim f    As Form
  
   For Each f In Forms
  
      Debug.Print f.Name, f.Caption
     
   Next
  
End Sub

Just run the first sub (place cursor in sub – hit f5 to run). You now have 5 copies of the same form open.

Now, you can simply run either of the two additional subs that “kills” a given form. In the first example I run a custom public “MyClose” function in the form – eg this:

Public Function Myclose()

   DoCmd.Close
  
End Function

So any public function can be run inside of each form instance.


However as the second “kill” form code snip shows, I simply kill the given form instance in memory (set f = nothing) – in BOTH cases the forms close event will fire. So if I type in “5” to the input box, you see I the debug.window the close event fires with this:

Immediate window output:
close form = form5

In the form6 close event I have this code:

Private Sub Form_Close()

   Debug.Print "close form = " & Me.Caption
  
End Sub

So you can blow out any instance of a form, but the forms events will still “run” since like any class object BEFORE the given variable/object goes out of scope the “terminate” event will fire. This means for custom class objects the VBA “terminate” event still fires, and for form class objects, all of the correct forms “shutdown” events also run before the form goes out of scope.

Note that a custom collection is a ideal way to “hold” the given instances of those duplicate copies of the given form(s). And note how I set the “caption” property of the form. This means I can loop the forms collection and “find” the given copy. If you run the “last” routine you get this output:

Form6         form1
Form6         form2
Form6         form3
Form6         form4
Form6         form5

Note how in the forms collection you can’t get at the “additional” form name with forms("form6") and the forms collection ONLY allows  you to grab by forms(1-5) or forms("form6"). But “form6” will always return the first one in the standard forms() collection. So the forms ARE added to the built in forms collection, but all having the same name means you can get at "one" particular form you want!

So if you add a “caption” to the form, then you can close any instance by “searching” for the correct caption, grab a form ref,  and closing that form anyway you want as per above (nothing custom close - whatever). Just keep in mind that in all cases the forms “terminate” events do fire (so on-close etc do fire in all cases).

Try the above sample code. To run any of the subs - click inside the sub and hit f5 to play with and see how this works. You can create a form called "form6" or use any form you have now.

Edit:

I should point out that in vb.net, c#, c++, Delphi or in ANY modern programming language the developer has to keep track of those open forms - and dispose of them in the above like manor. So how VBA works, or vb.net, or c++ works is really the same. You are most free to dispose of any object in code - but the developer has to "know" what object you want to dispose of.
