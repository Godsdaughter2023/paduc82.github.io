ALT+F11 to open VB editor, right click 'ThisWorkbook' and insert module and paste the code below in and run it.

 

 

Sub Sortem()
Dim x As Long, y as Long
For x = 1 To Worksheets.Count
    For y = x To Worksheets.Count
        If Left(Sheets(y).Name, 3) < Left(Sheets(x).Name, 3) Then
              Sheets(y).Move before:=Sheets(x)
         End If
    Next
Next
End Sub

nathan o. - microsoft support say:

Hi Gregory,

 

Access 2007 and Access 2010 both make use of Ace as its database engine. I also believe certain components of Project 2010 make use of the 2010 version of Ace. So when you launch Project 2010, it most likely is configuring to use the 2010 version of Ace, which then in turn breaks Access 2007, since it needs to use the 2007 version of Ace. In order to get Access 2007 to run properly again, I would first recommend running a repair of Access 2007 / Office 2007. Doing this, should reconfigure the product and then set the current version of Ace to the 2007 version. 

 

Normally we don’t recommend having multiple versions of Office on the same machine, because problems like this can occur. When you have multiple versions of Access on the same machine, if you flip between versions, you will see Access reconfigures itself automatically.  The reason it does this is because it has to ensure it loads the proper .dll files for that version of Access and one of those .dll files is for Ace. In your scenario, Access 2007 doesn’t reconfigure itself if Project 2010 is launched, thus as mentioned above, it doesn’t use the right version of Ace. So if you launch Project 2010, you will then most likely need to repair Access each time before using it, unless you remove Access 2007 and install Access 2010.

 

Hope that helps.

Ok, if you have a requirement to keep a historical status, that would be done with a child table. Something like this:

 

tblCaseHistory

scottgem say:

First, A report displays whatever records are in its recordsource. So its not the report that is dropping records, its the query. The problem could definitely be your structure. Having one history table with 2 main tables. could cause a problem.

 

If you have a history table like I suggested all the more reason that you don't need and shouldn't have a separate table for closed cases.

 

And yes, when you add a record for a new case, you can append a record to the history table. So you would use the After Insert event of your case form to run code like:

 

Dim strSQL As String

 

strSQL = "INSERT INTO tblCaseHistory (CaseID, StatusID, StatusDate, Notes) " & _

               "VALUES(" & Me.CaseID & ", 1, #" & Date() & "#, 'Open Case');"

CurrentDB.Execute strSQL

 

This assumes that the statusID for Open is 1. Also, I forgot to include the StatusDate in my earlier note.

I'm assuming you reinstalled Access 2003? If so, then you may have had troubles with that installation. Try removing it again, reboot the machine, and then try to reference the correct libraries.
