varun shankar say:
It is not possible to delete the default .pst file in Outlook.
How many data files do you have in Outlook?
When you configure an email account as an IMAP account, a different set of folders is created to synchronize the data with the IMAP server.
Since that data file does not have a few folders which have to be part of an Outlook data file (i.e. Calendar / Contacts / etc.), Outlook creates two .pst file to save for IMAP account. One for the local personal folder and one for the actual IMAP mail store.
The IMAP account data file cannot be set as default because of the same reason as above.
The .pst for the local personal folder is used to store the local items, such as your Tasks, your Calendars, Notes, etc.

ken sheridan say: If you have three hours to cut down a tree, spend two of them sharpening the axe, i.e. when new to database development invest time in learning the principles of the database relational model, and the nuts and bolts of how databases are designed in Access in accordance with these principles.  If you don't you will continue to make a rod for your own back by committing fundamental errors.

Your table at present is badly non-normalized and in need of decomposition into a set of correctly normalized related tables.  The basic model would be like this:

Assets>---Addresses>---Locations

In this I'm using the term addresses to mean the specific location of an asset, e.g. a building or other facility.  As you can see the model is a simple tree-structured hierarchy, in that there is a one-to-many relationship type between Locations and Addresses (each city can contain one or more addresses, but each address is in one location only), and similarly a one-to-many relationship type between Addresses and Assets.  The term locations is used in your sense, and could refer to a geographical unit of any size such as a city, town or village.   

To determine which columns are included in each table you need to be familiar with the 'rules' of normalization, but in very simple terms all non-key columns in each table must be determined solely by the key, e.g. the Assets table would not have columns AddressLine1 and AddressLine2 as these are determined by the key of Addresses.  Similarly Addresses would not have a column County (or Region) as this is determined by the key of Locations.  For this there should therefore also be a Counties ( or Regions) table referenced by Locations.  Note that this should contain a N/A county or region to cater for metropolitan areas which are not within any administrative county or region, e.g. London, Birmingham etc in England (this might not apply in Scotland; I'm unfamiliar with the administrative structure north of the border).

You might like to take a look at the files DatabaseBasic.zip and Normalization.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

These files provide simple introductions to the basics of relational database design in Access, and to the subject of normalization.  With the latter, concentrate on the first three normal forms to start with.  Most databases normalized to Third Normal Form (3NF) will also be normalized to the higher normal forms, and paying too much attention to the latter at this stage will probably confuse you.  That's not to say that they are not important of course.

When it comes to retrieving data from the database, my DatabaseBasics demo gives some simple examples, but for the situation you describe you would probably want to use correlated combo boxes in which you 'drill down' to assets by selecting a location first in one combo box, then an address in a second combo box whose list is restricted to those addresses in the selected location.  The ComboDemo.zip file in my OneDrive folder includes examples.  The simplified form for 'drilling down' through a hierarchy would provide an appropriate model to follow.

scottgem say:

As Ken said, you need to understand database design before you can start designing tables.

You said that each building has a UPRN. Therefore you need a table of buildings. It would also appear that each building belongs in a Classification and each Classification belongs in a Unit. So that'sTHREE tables;

tblUnits

UnitID (PK Autonumber)

UnitName

tblClassifications

ClassificationID (PK Autonumber)

ClassificationName

UnitID (FK)

tblBuildings

UPRN (PK)

BldgAddress

BldgName

ClassificationID (FK)

Now notice I haven't included UnitID in tblBuildings. That's because each Classification is associated with a single unit. So if you know the Classification you know the Unit.

Lets now look at your main table:

tblAssets

AssetID (PK Autonumber)

AssetNumber

BuildingID (FK)

ConditionID (FK)

TypeID (FK)

MfgID (FK)

PATTestDate

NextPATDate

Again notice that we only have BuildingID as a FK. Since BuildingID is associated with a single ClassificationID, we do not need to include it in tblAsset. As for displaying the related values see my Blog On Displaying Related Data. Notice also the FKs for Condition, Type and Mfg. You will need lookup tables for these items.

scottgem say:

I don't understand your rational here. Are you porting to an EXISTING database. If so, then why not use the same structure as in that database? If not, then why not build that database properly?

But whether you use a street address to identify the Building or some other method, the structure I outlined still stands. A building is an entity. Attributes of that entity as the Classification, physical location and name or other identifier. 

Unit is an attribute of the Classification entity. Asset is another entity whose attributes are Building, Type, Mfg, Condition and test dates.

VBA to do what? 

I outlined how the relationships would work by indicating the PKs and FKs. Don't accept the default ID name for autonumber PKs. rename them as tablenameID, which is the naming convention I used. Then name your FKs the same, this makes your relationships clear. Adding an FK to a table is simple. The only requirement for an FK is that it match the datatype of the PK it relates to. If you are using Autonumber PKs, then the FKs need to be Long Integer data types.
