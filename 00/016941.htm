stephen boots say:
Because I've been moderating this forum since the initial introduction of MSE and the forum for its predecessor, Windows Live OneCare.
MSE is designed for a consumer audience. It was originally licensed only for home users and specifically prohibited for use in any business environment. That licensing has changed to allow use on up to 10 PCs in an business. 
Although certainly possible, domain joined PCs in a home environment are rare.
There is an installer available to OEMs in order to deploy MSE to PCs in the factory image. To obtain it, you'd need to be a "partner" - http://windows.microsoft.com/en-US/windows/products/security-essentials/partners

For typical domain environments, the product Microsoft sells is Forefront, which includes management capabilities, product versions, and licensing for servers and clients.

So, assuming that you are a home user or plan to use MSE in a business environment within the terms of the licensing (and 8 computers certainly fits that bill), manually installing MSE to each PC initially would be the solution. It will work on domain joined PCs. In the event that you also use WSUS, definitions and client upgrades can be handled via WSUS, too.

SortField: IIf(InStr([LineNumber],"."),1000*Left([LineNumber],InStr([LineNumber],"."))+Mid([LineNumber],InStr([LineNumber],".")+1),1000*[LineNumber])

greginmich say:

The Defender NIS and ATP services each have a “manual” startup type by default. The Defender ATP Service is probably set to a “manual” startup because it’s only used on Windows 10 Pro machines that serve as clients in enterprise environments (it’s not included in the Windows 10 Home edition); and the Defender NIS Service is probably set to a “manual” startup because it's only required to run when unpatched vulnerabilities in network protocols have been identified.

But the only Defender services that are pertinent to the issue at hand would be the Defender AV and the WDSC services, which are both set to the “automatic” startup type by default. Because the connections of the basic Defender services with the WDSC hub are quite complex, changing any one of those services to a delayed startup could potentially have unpredictable consequences.

That being said, the Windows Security Center Service (which was integrated with the Windows Defender Security Center belatedly in Version 1709), does appear to have an Automatic (Delayed Start) startup type by default, and although that might seem counterintuitive, there’s probably some good technical reason for it, so it shouldn’t be modified for any purpose other than testing.

But our present understanding is that this issue simply won’t be fixable without either finagling things with repeated restarts, or maybe by using Group Policy to prevent Defender from being disabled – and we’re still not really sure that the registry edit will resolve the issue in this particular case:

https://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_scanning-windows_10/windows-defender-virus-threat-protection-restart/9215a1e1-d386-477b-b22f-d608365b255a

So people will need to try these methods to get the Windows Defender Antivirus Service up and running if it’s down, since this is the service that provides Defender’s critical real-time AV protection. If that fails, then replacing Defender with a third-party AV app should at least provide folks with a viable alternative for Defender’s real-time protection, and thus allow them to safely connect their PCs to the internet:

https://www.pcmag.com/article2/0,2817,2388652,00.asp

ihrigt say:

Importing Currency values from Excel adds unwanted decimals

I am importing data from an Excel spreadsheet into an Access 2010 table using the External Data > Import function.  The data in Excel includes several columns of data that are currency values (ie., number with 2 decimal places).  The corresponding fields within the Access table are designated as "Currency" data types.  After the import, some values in Access now have 4 decimal places and are off by 0.0001.  I cannot discern a pattern as to which rows will be affected and it does not occur in a regular interval (every 10th row, for example).

 

I discover that this has occurred when summing a range and the sum is off by a few pennies.  I have tried to isolate where this occurs.  When I find a row where the currency value has 4 decimal places, I look at the original value in Excel and can see in the cell status bar that only 2 decimal places are present.  This seems to indicate that the added precision is happening in Access, presumably during the import.

 

I assumed that by setting the receiving field to a "Currency" data type with 2 decimal places that the import values would retain only the 2 decimal places of the original data.  This does not appear to be correct - so I have two questions:

 

1. How can I prevent this from happening and force the database to 2 decimal places?

2. How can I "clean" my existing data in the table to remove any fields that now have more than 2 decimal places of precision?

 

Thanks in advance for your help.

hansv mvp say:

You could create an update query that rounds the currency fields to two decimal places.

For example, update a field Amount to Round([Amount],2)

hansv mvp say:

I don't know how to prevent this happening - Access and Excel appear to have different ways of interpreting numbers.

 

To create an append query:

Activate the Create tab of the ribbon.

Click Query Design in the Queries group.

The Show Table dialog will pop up. Select the relevant table, click Add, then Close.

In the Design tab of the ribbon, in the Query Type group, click Update. This will remove the Sort and Show rows from the lower part of the query design window, and add an Update to row.

Add one of the currency fields from the field list to the query grid. Let's say it is called Amount.

In the Update to row of this column, enter the expression Round([Amount],2) using the name of the field.

Repeat for the other currency fields.

 

To execute the update query, click the Run button in the Results group of the Design tab of the ribbon.
