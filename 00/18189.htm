pamcaswell say: Rhetoric101,

You can set heading row formatting in the table style.  The workaround is that you should NOT set the heading row to repeat in the table style.    When the  table style is applied, users will see the table with one heading row formatted.  If they need more than one row formatted as a heading row or need more than one row to repeat, they should select the rows and click the Repeat heading row button. This extra step only has to be done once per table.

I use this method to format multirow headers even when the tables don't cross pages.  I think of it as a way of telling Word how many rows are heading rows.   In the early months of W2007, this feature wasn't working well on tables with joined or split cells, and I had to manually format them.   So I think this is a treat.

Thankfully, I don't often have to work with restricted documents, but I did create one to test what the poster in the other thread suggested and find that it works.

I'm experiencing the same problem and would like to add some information and a (silly and annoying, but workable) workaround.

First of all confirming the OP isn't the only one with this problem. Secondly I'm using Windows 8.1 and have the latest updates installed. Using OneNote 2013 and Excel 2013. Colleagues of mine are experiencing the same problem. It seems to be related to the way OneNote stores the excel document in its OneNote document.

The error that is being logged in the EventLog:

Faulting application name: EXCEL.EXE, version: 15.0.4701.1001, time stamp: 0x54e312a6
Faulting module name: EXCEL.EXE, version: 15.0.4701.1001, time stamp: 0x54e312a6
Exception code: 0xc0000005
Fault offset: 0x00000000000ee6ab
Faulting process id: 0x2bec
Faulting application start time: 0x01d066fb7ebafb18
Faulting application path: C:\Program Files\Microsoft Office\Office15\EXCEL.EXE
Faulting module path: C:\Program Files\Microsoft Office\Office15\EXCEL.EXE
Report Id: 068391ed-d2ef-11e4-8289-ecf4bb2c7455
Faulting package full name: 
Faulting package-relative application ID: 

The workaround to fix the issue temporarily:

1. Open the excel workbook from OneNote

2. Make a copy of the worksheet and remove the old worksheet within the same workbook, save the workbook so OneNote is updated

3. The copy allows inserting comments again

Problem reproduction scenario:

The problem (for us) only seems to appear when multiple users have access to the same OneNote.

1. User A inserts an excel workbook, edits, and can without problem insert comments and save the file multiple times. This user then closes excel and lets OneNote sync properly.

2. User B opens the onenote, lets it sync properly and then opens the excel workbook. User B's Excel will crash upon inserting a comment with the EventLog as posted above.

3. Using the workaround depicted above User B fixes his excel and can properly insert comments again. 4. User B is finished, saves and lets OneNote sync.

5. User A opens the onenote, lets it sync properly again and then opens the excel workbook. User A's Excel will crash upon inserting a comment. It now repeats starting with step 2.

In conclusion it seems that when the excel workbook is synchronized over OneNote to another machine, certain data is lost or a situation is achieved that causes Excel to generate a null reference exception (0xc0000005 access violation is most commonly a nullref) because some data is missing. Copying the worksheet fixes this, until the workbook is updated by another user via OneNote.

Hoping this sheds some light on the issue and provides a workaround for those having the issue. Also hoping this will be fixed in an update.

suzanne s. barnhill say:

The behavior of the decimal tab stop is unchanged. It aligns text on the first non-numeric character in a string (aside from the thousands separator). What this means in practice is that numbers with decimal points will align on the decimal point. Numbers in parentheses or with following note references (such as an asterisk or raised letter) will be aligned on the closing parenthesis or reference mark. If you have some numbers with decimal points and some in parentheses (but without decimal points), then you may not get the alignment you seek. See http://wordfaqs.mvps.org/NumberAlignment2007.htm#DecimalTabs

 

In tables, text to be aligned at a decimal tab must be left-aligned and need not contain tab characters. If tab characters are used in the cell for other purposes, then a tab will be required for the decimal alignment.
