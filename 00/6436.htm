It has nothing to see with the search in my opinion, but it is due to the poor 512 apps limitation, and how windows handles the tiledatalayer database (or the tile data server service...) : it won't reset or change once it's broken...

So, the best workaround for now, is :

  1. Reduce the number of links/apps (by moving some folders, or deleting useless shortcuts)
  2. Delete the Tiledatalayer/database folder, to force the start menu to "reset"
    This folder is situated in  "your user name"/appdata/local/tiledatalayer, but its a bit tricky to delete, you need do it in a few steps : 
    • first you need to open task manager and go down to windows processes, leave it open on the side. 
    • then you delete the database folder, but it will state "the file...is in use"
    • so in task manager under windows processes, end  "appmodel task" (click  yes/ shutdown blabla... it won't shutdown in fact.)
    • finally you click retry and the folder will be deleted.
  3. Restart Windows and after a moment, it will "reconstruct" itself, and all apps (if less than 512) will reappear, less the store and window native apps (see step3 for that).
  4. Re-deploy store apps, so they appear again in start menu.  For that, open a PowerShell window as Administrator. Copy and paste the following line into the PowerShell window and press Enter:

Get-AppxPackage -AllUsers| Foreach {Add-AppxPackage -DisableDevelopmentMode -Register “$($_.InstallLocation)\AppXManifest.xml”}

  1. All your apps are back, however all your pinned apps will be reset (the right panel will be empty) so you need to re-pin everything that you want. Personnaly I did a screen capture before, to remember what I pinned and where (use the "snipping tool" in startmenu/windows accessories, with a delay to be able to open the start menu before the capture)
  2. Optional step, backup the tiledatalayer folder. In case your start menu breaks again you'll just have to restore it (to be able to copy the folder and/or restore it, see step 2 and taskmanager trick for that)

Until it get fixed, hope it helps someone...

PS : Creating a new user profile is not a very good option and can be a real disaster, as the user SID in registry will be a different one, it means you can break all your security/permissions (if you delete the first user profile and data). So - very important- in case you're a real kamikaze and want to do that,  before deleting the original/migrated profile, go in the registry, under "HKLM/SOFTWARE/Microsoft/Windows NT/CurrentVersion/ProfileList/S-1-5-xx-xxxxxxxx-xxxxxxxxxxxxxx/" , note the last 4 digits of your original profile (usually user SID finishes by -1001 if you did a clean install). After the migration you'll have to change the new profile SID by replacing with the original user SID. I guess there is plenty of topics explaining better than me how to migrate a profile in a good way...imho my solution is much less dangerous.

edit: copying the tiledatalayer database from a newly created user is another option, but due to the new user SID, the file permissions would be changed (to different SID or "account unknown" if you deleted that new profile)

ken sheridan say: The code I posted is merely to illustrate how you can examine the TableDefs and Fields collections using VBA.  To see it in operation just create a new procedure in a standard module in the usual way, paste in the code and press the F5 key to execute it.  The tables and their columns will be listed to the debug window.

How you'd  use this method of returning field names in an application depends on what you want to do with the field names.  You'd most probably reference a specific table or query e.g.

    Dim tdf As DAO.TableDef

    Set tdf = CurrentDb.TableDefs("YourTablename")

and then examine its columns by looping through the tdf.Fields collection.  What you'd then do with the names is for you to say.

If you just want a simple way to examine list the column names of each table you can set up a little dialogue form to do this very easily.  First create a new form, then:

1.  In the form's Open event procedure put:

    On Error GoTo Err_Handler

    Dim dbs As DAO.Database
    Dim tdf As DAO.TableDef
    Dim ctrl As Control
    Dim strTablename As String
   
    Set ctrl = Me.lstTables
    ctrl.RowSourceType = "Value List"
   
    Set dbs = CurrentDb
    For Each tdf In dbs.TableDefs
        strTablename = tdf.Name
        If Left(strTablename, 4) <> "MSys" _
          And Left(strTablename, 1) <> "~" Then
            ctrl.AddItem strTablename
        End If
    Next tdf
   
Exit_Here:
    Exit Sub
   
Err_Handler:
    MsgBox Err.Description, vbExclamation, "Error"
    Resume Exit_Here
   
End Sub


Private Sub lstTables_AfterUpdate()

    On Error GoTo Err_Handler

    Dim ctrl As Control
   
    Set ctrl = Me.lstColumns
   
    ctrl.RowSourceType = "Field List"
    ctrl.RowSource = Me.ActiveControl
    ctrl.Requery
   
Exit_Here:
    Exit Sub
   
Err_Handler:
    MsgBox Err.Description, vbExclamation, "Error"
    Resume Exit_Here

2.  Add a list box named lstTables to the form.

3.  In the lstTables's control's AfterUpdate event procedure put:

    On Error GoTo Err_Handler

    Dim ctrl As Control
   
    Set ctrl = Me.lstColumns
   
    ctrl.RowSourceType = "Field List"
    ctrl.RowSource = Me.ActiveControl
    ctrl.Requery
   
Exit_Here:
    Exit Sub
   
Err_Handler:
    MsgBox Err.Description, vbExclamation, "Error"
    Resume Exit_Here

4.  Add a list box named lstColumns to the form.

The first list box will display the table names.  When you select a table in it the second list box will display the column names from the selected table.
