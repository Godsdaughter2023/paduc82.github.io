I noticed the mouse response time seemed to be slightly slow in general (milliseconds) so I started looking for any software that might be sitting in-between the driver and the user interface. I discovered ipoint.exe in the Task Manager, which belongs to IntelliPoint/IntelliMouse. A quick search revealed that some people have had "problems" with IntelliPoint on Windows 7, and that updating the software fixed these problems. I followed the link to http://www.microsoft.com/hardware/en-us/d/intellimouse, installed the latest version (8.15 for USB mice), rebooted the system, and my problems were less frequent, but still noticable. I then tried terminating the ipoint.exe process, which was a slight improvement but not a fix.

The problem went away for me only when I replaced the mouse with a new Windows Certified, "Microsoft IntelliMouse Optical" mouse. I noticed this one uses a different driver than the previous mouse (by Sun Microsystems) which had been using a generic Microsoft driver.

The pattern of failure seems to indicate a bug somewhere between the mouse and the UI which results in the (apparent) occasional loss or double-acknowledgement of interrupts from these devices. The problem is also related to performance, which may explain why it only appears some time after a fresh installation of Windows 7 - it takes time to install enough services and programs to load the CPU/memory to the point where the problem becomes noticable. My best guess is that there is a bug in the generic USB mouse driver which results in missed or double-acknowledged interrupts. If you were dragging a window and a mouse interrupt was missed, it would look like you'd dropped the window and clicked again; if you clicked once and the interrupt was double-counted, it would look like you had double-clicked. Missing interrupts would explain 100% of my symptoms.

A mismanaged buffer, of sorts, would fit the profile of the problems I was having. Suppose there's a buffer (FIFO) for mouse interrupts that only gets used when the machine is loaded and the interrupts cannot be handled immediately. Suppose there's a bug in the algorithm that manages the buffer; for example (and I'm purely speculating), maybe there are two processes that access the FIFO - one that reads, and one that writes. If both try to read/write at the same time, and there's no mutual exclusion being enforced (or it's buggy), you might get race-conditions when the FIFO is nearly empty, and some interrupts could missed or counted twice.

If in doubt, the quickest solution is probably to get a Windows 7 certified mouse.
