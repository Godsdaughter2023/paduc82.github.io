deep-thought say:

Problem Solved!!! (sort of)

 

Hi,

 

I've discovered the scenario which leads to this problem, at least in my case.

 

It appears you can have as many POP3 and IMAP accounts, each with their own COMODO email certificate, as you like and Outlook automatically selects the appropriate certificate to sign an outgoing email correctly.

 

However, as soon as you add a second Office 365-like account Outlook can no longer find the certificate for the first-installed account automatically and you get the error message, "Microsoft Outlook cannot sign or encrypt this message because you have no certificate which can be used to send from your e-mail address.".

 

So, the answer is only use one ActiveSync Exchange (Office 365-like) account with a COMODO email certificate and all is well.

 

On my machines I have several POP3 accounts, Hotmail (IMAP), an internal domain Exchange Server and my Office 365 account all working ok now. All have their own certificates and they are selected correctly automatically by Outlook depending on the account the email is being sent from.

 

It is a shame Outlook gets confused when using multiple externally hosted Exchange accounts, but I doubt this will affect too many people until organisations start moving their email infrastructure into Microsoft's cloud and users who have their own personal Office 365 account have an employer's email account to connect to as well. Since this is Microsoft's aim, the day will come.

 

Perhaps someone should let Microsoft know they need to do some testing and fix Outlook so that it can select from multiple email certificates correctly when a user has more than one Office 365-like email account?

Glyn.

bob jones aka: cybertaz say:
You might first try just running Disk Utility to repair disk permissions followed by a restart of your Mac. It may require nothing more than that.

There are 3 "most likely candidates" which cause that message to start being triggered where it had not been showing up previously. In order of decreasing probability they are:
  1. Word is attempting to load external add-ins which are failing [e.g., Endnote, Zotero, or some other utility that is intended to add functionality to the program]. Usually it's a matter of the add-in being outdated, or
  2. The MS Visual Basic for Applications framework has been damaged, removed, or cannot be accessed due to permissions issues. It may be necessary to reinstall only that from your Office 2011 source media, or
  3. The Normal.dotm template [or document/template] being opened contains VBA that cannot be executed & may be viral. You can easily test this by either holding Shift while launching Word or creating a new User Account to see if the message appears when logged in as that user.
There are existing conversations that address in more detail all 3 possibilities, such as Word could not fire event?. Despite a couple of sarcastic replies advocating full remove/install, that conversation covers just about all of the bases. (Although remove/install may well fix the problem it's an overkill approach likely to precipitate other issues... Kinda like the old "throw out the baby with the bathwater" expression :-)

Running on OS X 10.6.x the Normal.dotm template is stored in:
~/Username/Library/Application Support/Microsoft/Office/User Templates

If you are interested in an anti-virus utility, one that's highly reliable is Sophos which is available for home use as a free download.

vijay a. verma say:
The problem of .xls file not getting opened after upgrading to Excel 2016 / Windows 10 is a common problem. This problem may occur after any windows / MS Office update as well.

1. Start the Excel and CTRL+O (or File > Open) > Locate the file and open the file. This should open the file but this  may not solve the problem of double clicking the file and opening it.

2.A quick repair usually fixes the file association

https://support.office.com/en-NZ/Article/Repair-an-Office-application-7821d4b6-7c1d-4205-aa0e-a6b40c5bb88b

3. If this doesn't fix the association, when you right click on .xls file > Open With > You should get the list of programs which can open this file and you should select Excel.

In case, the list of program doesn't have Excel, then choose BROWSE and select Excel.exe. But, in case the folder which you get after clicking BROWSE doesn't have Excel.exe, you will need to find the location of Excel.exe to select Excel.exe. To find the path, you can refer to below link -

http://eforexcel.com/wp/tips-tricks-147-find-the-location-of-excel-exe/

4. If there is a file block issue

4.1 Right Click on the file > Properties > In the General tab, you will get Unblock option down below > Check the unblock box

4.2 If you don't get unblock option, ALT+T+O (or File > Options) > Trust Center > Trust Center Settings > File Block Settings > Uncheck the box in front of .xls file

marvin ma msft say:

Hi MJR1968,

As far as I know, you can only set up the group to let external users to send emails to this group like the picture below.

It's different from allowing the group members to email with each other. This option means all the external users outside your organization are able to send emails to this group, that is, it's open for extranet. 

Besides, when the member replies to a group email, it will be split to many copies by Exchange Online Protection (EOP). One of the copy will be delivered to Office 365 group inbox in your organization, while the other will be routed by EOP to the mail servers of the external users. And for the later, EOP only works as a router, that is, your organization doesn't make much influence on the whole mail flow.

To confirm it, you can check if Paul's reply has been delivered to the inbox of this group. If so, it indicates EOP works well and the issue should be related with the sender side and the recipients who failed to receive the email. And it could be caused by many factors. For example, the untrusted relationship between sender and recipient, the lack of SPF record of the sender side, or even the recipient blocked the sender. And it would be better for the sender side to do the further investigation since the issue is out of your side. 

On the other front, if Paul's reply hasn't been delivered to the group inbox, then the issue could be related with the EOP or Paul's email server side. To further confirm it, you can run a message trace to track if this email has been inbound.

Regards,

Marvin
