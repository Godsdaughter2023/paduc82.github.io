ken sheridan say: If you are using a split form I would recommend that you do as Tom suggests and use the built in functionality.  Trying to make the single form part of a split from using one set of controls is not really a feasible proposition.  Over the years I've see a number of posts where people have tried to make a set of controls dual purpose, and they usually end up inadvertently changing the values of data in the tables.

If you want to build a search form in which a set of unbound controls is positioned at the top of a form in continuous forms view, that is a simpler and safer solution.  There are a number of ways you can implement the search:

1.  Build a string expression based on the values entered/selected in non-null unbound controls in the form header, and assign the string expression to the form's Filter property, also setting the form's FilterOn property to True.

2.  Build a string expression based on the values entered/selected in non-null unbound controls in the form header, and change the form's RecordSource property, with the string expression forming the basis of its WHERE clause.

3.  Base the form of a query which references the unbound controls as parameters, and requery the form to implement the search.  The following is an example of such a query from my ComboDemo file in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

SELECT Counties.County, Districts.District, Parishes.Parish,
Locations.Location, Parishes.ParishID  
FROM ((Counties INNER JOIN Districts ON Counties.CountyID = Districts.CountyID)  
INNER JOIN Parishes ON Districts.DistrictID = Parishes.DistrictID)  
INNER JOIN Locations ON Parishes.ParishID = Locations.ParishID  
WHERE (Counties.CountyID=[Forms]![frmDrillDown_Simple]![cboGotoCounty]
  OR [Forms]![frmDrillDown_Simple]![cboGotoCounty] IS NULL)
AND (Districts.DistrictID=[Forms]![frmDrillDown_Simple]![cboGotoDistrict]
  OR [Forms]![frmDrillDown_Simple]![cboGotoDistrict] IS NULL)
AND (Parishes.ParishID=[Forms]![frmDrillDown_Simple]![cboGotoParish]
  OR [Forms]![frmDrillDown_Simple]![cboGotoParish] IS NULL)
ORDER BY County,District,Parish;

In this little demo file the query is the basis for the form for 'drilling down through a hierarchy' (simplified version).

With any of the above approaches the form can be filtered to the matching records either as a single operation via a command button, or, as in my demo, progressively by means of code in the AfterUpdate event procedures of the unbound controls.  The unbound controls in my demo are correlated combo boxes, in which a selection in one restricts the items available for selection in the next, but this is not a prerequisite of this approach.  the controls could be independent of each other.

ken sheridan say: You can change a form's RecordSource property in code at runtime without affecting the form definition.  This will remain unaltered, so the unrestricted query will still return the full recordset next time you, or somebody else in a multi-user environment, open the form.  It's simply a question of building the SQL statement in code and assigning it to the form's RecordSource property whenever you want to restrict the rows returned.

Building and assigning a string expression to the form's Filter property will have the same effect, but you also have to set the Form's FilterOn property to True in the code to apply the filter.  You appear to be doing this.

I think the problem is with the form set-up, and I'm unclear as to what this is as you refer to a split form and a subform.  If you simply want a search form in which you can  filter the rows returned, the form should not be a split form, nor should it contain a subform.  The form should be a basic bound form, and can be in single form or continuous forms view as preferred.  Whichever is the case the unbound controls on which the form is filtered would be in the form header, though this is not essential in the case of a form in single form view, they can go in the detail section provide that you visually separate them from the bound controls.  With a simple set-up like this your code should work as expected.

If you want to be fancy, however and have something which looks and works much like a split form you can simulate this.  You'll find an example in FindRecord.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the simulated split form, which uses a continuous subform, includes an unbound combo box in the parent to go to a contact.  At present this navigates to a row in the recordset, rather than filtering it.  However, you could filter both the parent form and the subform by changing the code in its AfterUpdate event procedure to:

    Dim strFilter As String

    If Not IsNull(Me.cboGotoContact) Then
        strFilter = "AddressID = " & Me.cboGotoContact
        
        Me.Filter = strFilter
        Me.sfcAddresses.Form.Filter = strFilter
        
        Me.FilterOn = True
        Me.sfcAddresses.Form.FilterOn = True
    Else
        Me.FilterOn = False
        Me.sfcAddresses.Form.FilterOn = False
    End If

Note that sfcAddresses is the name of the subform control, not its source form object.

With a set-up like this, in your case you could include the multiple unbound combo boxes in the parent form, like my cboGotoContact control, and assign a string expression to a strFilter variable in the same way as you are doing by examining each of the unbound controls in turn.  You'd then set the Filter properties of both the parent form and subform to the variable and set the FilterOn properties of both the parent form and subform to True to apply the filter to both.

Thank you for reporting this issue.

 

In Office Desktop, the use of the font is determined by the Unicode character values of the run content.

In your example, the Office Desktop determines that the font specified in the eastAsia font slot is not appropriate for the characters used in the run.  Office Online Reading View is simply an image of the document that is rendered in Office Desktop (on the server), and therefore it also shows the same fonts as Office Desktop.

In Editing view, we use JavaScript to recreate the document in the browser. Unfortunately, it is very difficult (and in some instances impossible) to recreate the above character-by-character Unicode analysis that Office 2013 performs from inside a browser. In the example you provided, Editing View has no knowledge that the eastAsia font should not be applied to the characters in the text run.  This is a known issue and are looking into how we can refine our algorithms to get closer to the behavior of Office Desktop.

Below are general troubleshooting steps to solve various problems in Excel. Try them one by one and check the result after each step -

1. Update your office as well as Windows 10 to latest patches from Microsoft site. Microsoft keeping releasing Office and Windows 10 patches to correct known errors.

2. Perform Quick Repair of the office installation - https://support.office.com/en-NZ/Article/Repair-an-Office-application-7821d4b6-7c1d-4205-aa0e-a6b40c5bb88b

3. Please look into C:\Users\User_Name\AppData\Roaming\Microsoft\Excel (replace user name with your user name, you can directly reach here by copy and pasting following without quotes in Explorer address bar - "%appdata%\Microsoft\Excel\") and see if is contains few files / folders. Move those files to some other location (i.e create a backup of those files / folders and delete all files / folders from here). Hence, make Excel folder blank.

Now open the file and see if the problem disappears or not. 

Note - to find Excel path, you need to find XLSTART path. Excel folder contains XLSTART folder in itself (In case, your XLSTART path is not in Appdata)

1. Open Excel
2. ALT+F11
3. If Immediate Window is not visible, type CTRL+G. Same can be also be accessed through View > Immediate Window
4. Type "? application.StartupPath" without quotes and press enter
5. You will have your path below.

4. You may login to Windows with a new user profile and see if it helps.

5. Review following link - https://community.office365.com/en-us/w/manage/office-365-and-windows-10-problems-and-solutions

6. Turn off Hey Cortana and reboot your computer (useful particularly if Excel performance is slow).

7. Sometimes, this is caused due to excel not being able to access the default printer. Close all office applications and change your default printer through Start > Devices and Printers to another printer. It may be some physical one or One Note or XPS or PDF or anything else. If this is the problem, then you will need to update your printer driver for default printer. 
Also upgrade all your Printer Drivers to latest from the internet.

8. Many third party applications also cause the problem like Tuneup Utilities, Abby Finereader, Kaspersky Anti Virus  etc. See if you have any of them installed. Close them one by one and see whether it helps.

9. Sometimes, add-ins can cause problems. To determine, if add-in is a problem start Excel in safe mode and see if problem has gone away. If yes, then start Excel normally and disable add-ins one by one and start Excel again every time you disable an add-in to determine the culprit add-in.

To start Excel in Safe Mode -

Hold CTRL key and click on Excel icon
OR
In the Search Box above Windows icon, type "Excel /s" without quotes and enter.

10. Issues in Protected View

File > Options > Trust Center > Trust Center Settings > Protected View > Uncheck all options
File > Options > Trust Center > Trust Center Settings > File Block Settings > Uncheck all options

(For reference, You may review the following thread - http://answers.microsoft.com/en-us/office/forum/office_2013_release-excel/cannot-open-some-office-files-after-upgrading-from/c103ee49-2e87-4421-ad96-f433ec77ec54)

11. Go to C:\Program Files (x86)\Microsoft Office\Office15, look for EXCEL.EXE
Right Click on this, Properties, In Compatibility tab, check ""Run this Program as an Administrator""
Start the Excel. You might again get the error. Close Excel and uncheck "Run this Program as an Administrator" and start Excel again. You should not get the error anymore.
(If folder path is not correct, you will have to find EXCEL.EXE manually, like Program Files (x86) may be Program Files. Or you can even search for EXCEL.EXE in your C: drive

12. File > Options > Advanced - In General section (go very much down in Advanced), check "Ignore Other applications that use Dynamic Data Exchange (DDE)". Close and re-start the Excel. Uncheck "Ignore Other applications that use Dynamic Data Exchange (DDE)". Close and re-start Excel again. You should not get the error anymore. (Basically, DDE should remain unchecked)

13. You may try to disable the Hardware Graphic Acceleration and verify the result:

    Open Excel.
    Click on file, Options.
    Go to the Advanced tab.
    Under the Display section, check the box for 'Disable hardware graphics acceleration'.
    Click ok and restart Excel.

14. If above don't solve your problem, re-installation of office may (and may not also) solve your problem. Remove MS Office completely by downloading Fixit tool and from http://support.microsoft.com/kb/2739501 and running it. Make sure that you have license keys ready before you attempt re-installation. If you don't have license keys, please do not attemp re-installation. You can also install MS Office by logging into www.office.com/myaccount if you have registered your copy.
