jerryduan_msft8021 say:

Hi QS-IT,

Please find the answers below:
1. What video formats does Office 365 video support for upload?

    Office 365 Video is built on Azure Media Services. Therefore, Office 365 Video supports only the codecs and file formats that Azure Media Services supports. For the details, check this out: Video formats that work in Office 365 Video.

2. What format do the videos get converted into, and at what resolution?

    When you upload a piece of content to Office 365 Video, SharePoint uploads that content to Azure Media Services where the content is encoded into various formats for adaptive smooth streaming so that the consumer can get the best playback experience based on the combination of device and bandwidth capabilities. Playback is based on H.264 at 720p resolution with bandwidth ranging from 400kbps – 3000kbps. Throughout playback the service monitors the quality of service and adjusts the playback every 2 secs to provide the best possible experience for each client.

3. How much storage are Office 365 tenants allotted for their videos, and where would I find capaticity utilization?

   When you upload a video, you upload it to a specific channel. The original file is stored in SharePoint Online and it counts against your storage quota. For more information, see Manage site collection storage limits.

I don't think Word will let you do this. The only reason for having a blank page after page 3 would be an Odd Page section start, and Word will insist that the page number of such a section be odd; if you try to make it page 4, Word may change the section start type, throw in another blank page, omit the one it already inserted, etc.

doug robbins - mvp office apps & services (word) say:

If you are duplexing, I do not understand why you would want a blank page after page 3 and then have the next page (the fifth page in the document) be numbered as page 4.

However, if you put the selection at the end of the text on the page, which you want to be followed by this anomaly, and run a macro containing the following code, it will insert a Next Page Section Break and a Page Break and unlink the Footer in the new section from that in the previous section, set the page numbering so that it restarts at the section with the same number as the page on which the selection is located and suppress the page number on the first page of the section so that the second page of the section has the number that is one more than the number on the page on which the selection was located.

Dim i As Long, j As Long
i = Selection.Information(wdActiveEndSectionNumber)
j = Selection.Information(wdActiveEndAdjustedPageNumber)
MsgBox j
Selection.InsertBreak wdSectionBreakNextPage
Selection.InsertBreak wdPageBreak
With ActiveDocument.Sections(i + 1).Footers(wdHeaderFooterPrimary)
    .LinkToPrevious = False
    With .PageNumbers
        .RestartNumberingAtSection = True
        .StartingNumber = j
        .ShowFirstPageNumber = False
    End With
End With

marshall barton say:

Go back and reread Wayne's reply where he said to use the record source field name in the sum, not the name of a control (text box).

Remember that form/reports do not have fields, they have controls.  Many people use the two different kinds of object names interchangably, but they are very different.  Aggregate functions (Count, Sum, etc) only understand expressions with field names, they are totally unaware of controls in a form/report.

marshall barton say: As for your "weird" part, Layout view is an alternate design view where you can change a form/report's design, particularly how things look and their position.  It uses some real data as a sample just so you can see how it will look, but it is not intended for users to view full sets of real data.  FYI, Report View is a web thingie that acts more like a form than a report, so don't try it until you have a good understanding of when/where it is useful.

marshall barton say:

Report View is a COMPLETELY DIFFERENT thing than a report.  It does not do the same things and operates in a different way (no pagination, different set of events, etc).  If that's what you want, then forget about Print Preview and printing the report in Normal View, if not then don't use Report View for anything, especially comparing what it does to what Print Preview does.

I do not understand what you are saying about the sum only picking one detail.  I suspect that your report's record source is joining two tables and the field you want to sum is in the one side table of the join.  If that's the case and you really must display the sum in the report header, I think things would be a lot easier if you put the many side table in a subreport instead of joining in a single report's record source query.  Then the main report records can be summed across its details and the subreport in the main report's detail section will not get in the way.

If that's too much to do, the only other thing I can think to try in a single report arrangement is to use Pages in a text box (e.g. =Page & " of " & Pages) somewhere in your report.  It will be slower, but it might(?) be that forcing the report to be formatted twice will make the report footer text box's value available to the report header the second time around.
