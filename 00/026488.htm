jay freedman say:

That "template" (actually a .doc document instead of a .dot or .dotm template) isn't suitable for a mail merge. The boxes on the second page are text boxes, not tables, which prevents the use of {NEXT} fields that select the next record from the data source.

You can download a "template" (again, really a document) from the Avery website, https://www.avery.com/templates/presta-94101. This also handles six 3"x3" square notes, but they're spaced slightly differently on the page than the ones in the document you have. You'll have to make a copy of Avery's document in which you turn on the borders of the table cells, and print that as a carrier for the notes.

Unfortunately, Word in Office 365 doesn't include a built-in Labels option for the 94101 product. Unless you want to use the New Label button in the Label Options dialog and enter all the measurements to create the definition, the alternative is to use the Avery document directly and do a Directory-type merge.

In the Directory merge, you need to insert a {NEXT} field at the end of each note, after the other merge fields. (Press Ctrl+F9 to insert the field braces, and type the word NEXT between the braces. You can copy the first one and paste it into each of the other notes.) Word will display these as «Next Record».

Yes, the quick scan is very useful. It looks at all of the actively running processes on the machine, as well as files that are pointed to by the various registry entries and other locations used by spyware to start automatically or otherwise interfere with your system.

If you have a particular piece of spyware or other unwanted software that is detected by Windows Defender, the vast majority of the time it should be detected by the quick scan.

The full scan, in addition to performing all of the checks in a quick scan, also examines every single file on your system. This is a quite time-consuming operation if you have a lot of files.

I would always start with a quick scan - if it gives you a cləan bill of həalth, you can be pretty confident in the results. If the quick scan finds anything, you probably should follow it up with a full scan to find any additional junk that the malicious software dropped on your machine that may have been missed by the quick scan. It's also a good idəa to schedule the full scan to happen periodically (--weekly, monthly--or only when something is found or you have some reason for suspicion, for instance) at a time when you are not typically using your computer and the added time to scan all of the files won't be noticed.

How new are you to programming? With Bound forms Access handles ALL the I/O between the form and the tables. With unbound forms, you have to write code to do this yourself. If you are comfortable with coding, then go ahead and use unbound forms, but being new to Access I would, like Albert suggest you start with bound forms.

Based on your code, you have a very simple table. I would definitely not recommend an unbound form in this situation. Unbound forms have their place. Where you need a lot of verification before committing the record or where you want to present a user interface that doesn't fit the table structure.

The way you have to code setup, it checks each control in turn. If it is null, then it prompts you sets focus to that control and exists. The user then fills in that value and starts the process again. That's a pretty standard UI. 

albert d. kallal say:

Well, first I hope you did think long and hard as to why you going with un-bound forms. They don’t save anything in terms of performance, and all it results in you having to write TONS of code.

And bound forms do have some GREAT verification options built in that would reduce the amount of code. You could for example place your above “verification” code in the forms before update event along with the messages + set focus to deal with this issue (it would be far less code).

As for the item not found in collection? These lines look bad/wrong:


Do you really have a column in your database called RecordsForms_Combos3) ??? I am betting this should be

Rst[name of column in table goes here] or in your case

Rst![Status] = me.Status

You have rst![Recordsform_Combo3] = Me.Status
(so above suggests that you have a actual column named Recordsform_Combo3 in your table)

So as a “general” rule, you will use rst![name of column] in table.

I still think you should dispense with un-bound forms as they don’t work well in Access, and the fact that you are having to “track down” wrong syntax and wrong column names quite much shows the “difficulty” in having to write all that code. With bound forms then rather little code need be written, and better you don’t spend time having to track down incorrectly column names in your code.
