ronnie vernon say:

Hi Ken

Modern Windows versions include a 32bit sub-system that allows 32bit programs to be installed on a 64bit version.

This is the reason that you see a C:\Program Files and C:\Program Files (x86) folder. The former is where the 64bit programs are installed and the latter is where the 32bit programs are installed.

You will also see a C:\Windows\System32 folder for the 64bit System Files and the C:\Windows\SysWOW64 folder where the 32bit System Files are stored.

The system isolates 32-bit applications from 64-bit applications, which includes preventing file and registry collisions.

When Windows Update runs it will detect any 32bit applications and offer to install any updates that apply to those 32bit applications.

It's not a good idea to ignore the 32bit updates since this could potentially cause problems with the 32bit programs. This is especially true with security updates.

Running 32-bit Applications:

Temps are good, No worries there.

There are two non-hardware things I can think of that may cause games to crash after playing for a period of time...

-  Anti-virus software. Try disabling or uninstalling your AV temporarily (or disable Windows Defender if you are using that).

Fortunately you don't have much software installed, so this helps reduce potential for conflicts from other software.

- I've only heard of this affecting one game, Far Cry 3 (crash after 15min).  Tablet Mode. In Settings > Tablet Mode.

Verify - When I Sign - In is set to  Use Desktop Mode (I think this only crashed the game, not the machine).

- The error messages implicate several things, as you seem to be aware, but your drivers are all up to date, so we should be able

to rule them out.

RAM is not necessarily the source of the crashes, however, because the crash happens after a period of about 15 minutes,

if the two items above don't help, I would be looking toward the RAM, but also run a disk check on your SSD & HDD, as errors

there could also lead to the memory_management error.

Did you use the Windows Memory test tool, or MemTest86 ? I'm not sure if you can setup Windows memory test for more than

two passes, which is (unless things have changed) not always adequate, particularly with RAM intensive things like games.

Eight passes are traditionally suggested for a proper test. If you can run Windows memory test for eight passes, no need for MemTest86.

It is a bit tedious, but you should test one stick at a time, and test in the two slots your mobo uses for two stick Dual Channel

(some mobo's use two adjacent, and others use alternate slots) as well, as a test for the mobo slots.

https://www.memtest86.com/download.htm

https://www.wikihow.com/Test-PC-RAM-with-MemTest86

- You may want to try repairing the games, but I think the issue is more likely hardware related. Especially as you said you haven

not installed anything other than your AV and some games.

Both Steam and UPlay have tools for this in their client software.

- With Steam R/click on the games title and go to Properties > Local Files and run the Verify Integrity of Game Files tool.

- UPlay - R/click on game title and click on Verify Files.

"I don't know much about BIOS, so I don't dare to fiddle with the settings,"

- I thought you had tried some over-clocking because you mentioned it in your first post. I just wanted to verify you set full defaults.

because a lot of people (myself included about twelve years ago) try to set their RAM speed to match what is on the label, which is

the maximum overclock speed for the RAM, not what the cpu/memory controller is capable of without overclocking it as well.

Motherboards generally use the lowest 'failsafe' speed by default, but you can generally setup BIOS to use the

RAM's XMP speed ( 2600MHz with your RAM I believe) without the system crashing.

- Note that if your graphics card is a factory over-clocked model resetting the motherboard BIOS to default would not

affect the gpu factory OC. It's not common but some brands of the Nvidia GTX500 series had issues with factory OC's due

to poor gpu BIOS.  This was remedied by using tools such as MSI afterburner to set the clocks to the base speed for the model,

and in some cases with a new gpu BIOS .

People at the Nvidia forums would know if this is an issue with the GTX1000 series.

- Personally, if I bought a pre-built machine from a retailer (I'm assuming OEM pre-built, or custom retailer built ) I would have wanted

to run a game so the retailer  could see what is happening, unless of course he is happy to use your description of the issue and

take the machine in for warranty repair. Which is what you are paying a premium for when buying pre-built machine.

You've done your bit to this point (hopefully you will find what the cause of the problem is soon, and it's not hardware,

before having to take the machine back. It's a drag to not be able to use your machine potentially for weeks).

The sooner the better though. If hardware is faulting you want to get in during the initial return to base warranty through the retailer

as having to do a RMA warranty through the mfg of the suspect component later is more trouble, and takes more time.

cmdrkeene say:

By default, if you're already logged in with an administrator-level account, you won't be prompted for a username/password, because you've already unlocked and proved your identity.

If you want to get prompted for more than just "yes, I confirm" by forcing it to require a password, you have two options:

  1. Lower your account down to a "standard" account.  Then any time you need to elevate your security privileges, it will prompt for a full password instead of just your consent.

    or


  2. Run the program SECPOL.msc (Security Policy) and navigate down to the Security Settings\Local Policies\Security Options section.  Here you can set the option called "User Account Control: Behavior of the elevation prompt for administrators in Admin Approval Mode".  Here you can choose if you want to "prompt for credentials" (need password) or just "prompt for consent" (just click yes).
