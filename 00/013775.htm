john mcghie say:
The key to this is to open the template itself to make the changes.

It's in /Users/~/Library/Application Support/Microsoft/Office/User Templates, where ~ stands for "My home directory.  You may have to open Finder, hit Command + g, then paste that location in -- in Lion it's a hidden folder.

My first suggestion would be to simply quit Word then DELETE your Normal.dotm and any other versions of Normal.dotm and Normal.dot and just "Normal" that you find.  If Word can't find the real Normal.dotm on startup, it goes hunting for previous versions and copies them, bringing in a variety of settings that are incompatible.

If you delete all the other versions of Normal (some of which will be in your Microsoft User Data folder in your Documents folder) you will get a nice clean default template with no rust or dust that will be a good starting point.

To set defaults for Normal, close all other documents then use File>Open to open Normal.dotm.  You must do it that way, or Word will make a copy and you will go around in circles.

Now use Format>Style to change the properties of Normal style to what you want.  In a default Word Normal.dotm, every style inherits its settings from Normal.

While you are at it, switch from Arial to its new replacement, Calibri.  Calibri is a much nicer font, engineered to look much better on screen as well as when printed.  Most documents these days are consumed on screen more often than on paper.  Calibri also supports the full modern Unicode character set, which means you get a whole lot of characters that Arial does not contain.  Of course, if you send such a document to a system that does not have Calibri, it will automatically fall back to Arial.  I doubt if there are any systems like that still running, but you never know.

In Calibri, you may wish to go up 1 point on your font sizes: it is a little more compact than Arial to enable more characters per line.

When you have finished making changes to Normal.dotm, save and close it, then quit Word to ensure that Word does not attempt to write the old version of Normal.dotm back to disk from its memory.

Hope this helps

graham mayor say:
In theory at least

Private Sub MultiPage1_Change()
    With Me.MultiPage1
        Select Case .Value
            Case 0: Me.Calendar1.Visible = False
            Case 1: Me.Calendar1.Visible = True
        End Select
    End With
End Sub

should work, but I know from previous experience that calendar controls do not always behave as expected when you attempt to hide them, so an alternative approach is to move them off the userform's visible area e.g. assuming the userform width is (say) 240, put the left property of the control beyond this e.g. 250, then when you want it visible, change its left property back to its original position - here 60. This invariably works where the visibility option doesn't.

It helps to set the userform width in the vba editor wide enough to display the moved control, and set the width to its normal state in the userform initialization.

Private Sub MultiPage1_Change()
    With Me.MultiPage1
        Select Case .Value
            Case 0: Me.Calendar1.Left = 250
            Case 1: Me.Calendar1.Left = 60
        End Select
    End With
End Sub

Recently a couple readers wrote to me with very similar problems. One was having trouble getting Internet Explorer to run; the other, Windows Explorer.

In other words, a couple of Windows' own built-in tools were broken. (Note to Microsoft: That should not happen. Come on!)

Usually, for situations like these, I recommend an end-run around the problem. Internet Explorer won't work? Install Firefox or Chrome and use that instead. Can't load Windows Explorer? Switch to any number of even better Windows Explorer replacements.

Of course, that doesn't really solve the problem. If you're hoping to fix your broken tool, you'll need to do a little trouble shooting. To start, I recommend switching to (or creating) another user account, then seeing if Explorer runs properly. If it does, you've narrowed down the problem significantly (and should consider "moving" to a new user account permanently). If not, well, back to square one.

Either way, it's a simple method of figuring out how deeply rooted into Windows the problem is.
