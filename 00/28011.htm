ken sheridan say: A multi-select list box has an ItemsSelected collection which is a collection of variants, pointing to a selected row in the control.  You need to loop through the collection and, at each iteration of the loop execute an INSERT INTO statement to insert a row into the subform's table.  Then requery the subform control, i.e. the control in the parent form which houses the subform, to show the newly inserted rows.

If we assume the subform's table is name EqipmentPerJob (note the use of CamelCase here BTW; object names should not include spaces or other special characters), and this models a many-to-many relationship type between Jobs, on which the parent form is based, and Equipment tables by means of JobID and EquipmentID foreign keys, you should only have an EquipmentID column in the table.  To include other columns duplicating other attributes of Equipment would introduce redundancy, and the table would not be normalized to Third Normal Form (3NF), leaving it at risk of update anomalies.  Consequently the code would be like this:

    Dim ctrl As Control
    Dim strSQL As String
    Dim varItem As Variant

    Set ctrl = Me.List54

    If ctrl.ItemsSelected.Count > 0 Then
        ' if at least one item has ben selected in the list box
        ' loop through the control's ItemsSelected collection
        ' and insert a row into EqipmentPerJob   table
        For Each varItem In ctrl.ItemsSelected
            strSQL = "INSERT INTO EqipmentPerJob(JobID, EquipmentID) " & _
                "VALUES(" & Me.JobID & "," & ctrl.ItemData(Varitem) & ")"

            CurrentDb.Execute strSQL, dbFailOnError
        Next varItem
    End If

    Me.NameOfSubformControlGoesHere.Requery

I've assumed autonumber primary keys for both referenced tables here.  If a 'natural' key such as SerialNumber has been used, and this is of text data type, then the value must be wrapped in literal quotes characters when building the SQL statement, e.g.

            strSQL = "INSERT INTO EqipmentPerJob(JobID, SerialNumber) " & _
                "VALUES(" & Me.JobID & ",""" & ctrl.ItemData(Varitem) & """)"

Whichever is the case the BoundColumn property of the list box must point to the  EquipmentID or SerialNumber column.

The code should be executed not in the list box's AfterUpdate event procedure, but in the Click event procedure of a separate button, which is pressed after all items have been selected.

What makes you think that?

The first value inserted into the EquipmentPerJob table should be the value of the primary key of the current record in the jobs table.  This would usually be the primary key of the table to which the form in whose module the code is being executed is bound.  Is this the case?

Regarding the definition of the EquipmentPerJob table there are a number of issues:

1.  Giving the table a surrogate autonumber primary key does not eliminate the necessity to define the CoBook Number and EquipID columns as a candidate key by including them in a single unique index.  A simpler solution is to delete the autonumber column, which serves no useful purpose, and make CoBook Number and EquipID the composite primary key.

2.  Item_Name and SerialNumber are not functionally dependant on the primary key as they presumably already exist as columns in the Equipment table.  Consequently the table is not normalized to Third Normal Form, due to the transitive functional dependencies, and therefore open to the risk of update anomalies.  The only situation in which non-key values should be duplicated like this is where they are independently determined by the keys of their relevant tables.  The classic example is that of a UnitPrice column in both Orders and OrderDetails tables.  As prices change over time the unit price in each table is functionally determined solely by the key of its table.  I don't think this is the case with the Item_Name and SerialNumber columns in your table, however, in which case the columns should be deleted from EquipmentPerJob.

Private Function FindFileInSubFolder(ByRef strPath As String, _
           ByRef strName As String, ByRef strFolder As String) As String
 'Jim Cone - Portland, Oregon USA - April 2006
 'Requires project reference to "Microsoft Scripting Runtime" library      <<<
 'Determines file a path if only a partial folder/file name is known.
  On Error GoTo ScriptErr
  Dim objFSO       As Scripting.FileSystemObject
  Dim objSubFolder As Scripting.folder
  Dim objFolder    As Scripting.folder
  Dim objFile      As Scripting.file
 
 'Bring it to life...
  Set objFSO = New Scripting.FileSystemObject
 
 'Check for top folder
  On Error Resume Next
  Set objFolder = objFSO.GetFolder(strPath)
  If Err.Number <> 0 Then
    FindFileInSubFolder = "No Top Folder:"
    GoTo FinishUp
  End If
  On Error GoTo ScriptErr
 
  For Each objSubFolder In objFolder.SubFolders
   'Verify secondary folder contains the specified characters.
    If objSubFolder.Name Like strFolder Then
     'Does file exist in the secondary folder...
      For Each objFile In objSubFolder.Files
       If objFile.Name Like strName Then
         FindFileInSubFolder = _
         "Path is " & objSubFolder.Path & Application.PathSeparator & objFile.Name
         GoTo FinishUp
       End If
      Next 'objfile
    End If
  Next 'objsubfolder
 FindFileInSubFolder = "Secondary Folder/File Not Found  "
 
FinishUp:
  On Error Resume Next
 'whatever is needed
  Exit Function
ScriptErr:
  FindFileInSubFolder = "Error " & Err.Number & "  " & Err.Description
  GoTo FinishUp
End Function
