Not completely sure, but I think it is as follows with the old SQL Server client (before the version that is called the "Native Client")

1.. leave out the Trusted_Connection parameter

2.. include the UID and Password parameters.

In that situation, I believe that a trusted client actually connects using the Trusted connection (even though you have specified a username and password), and an untrusted client connects using the login details that you provide.

If you specify Trusted_Connection=No then  then both clients will try to connect using the logon/password you provided

If you specify Trusted_COnnection=Yes then the untrusted client request will be positively rejected

Right now I cannot tell you whether it is the same for the newer client.
