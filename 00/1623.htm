techfreak_ say:

There is only 2 years of call history stored in the Skype cloud. This is the reason it only goes back to ~ April 2017. Since Skype's servers have never stored more than 2 years, any history greater than 2 years would rely on the history stored locally on your device in your main.db file (used by classic versions of Skype). If you still have the device you used in the past 3+ years and your main.db file exists and is not damaged, you should be able to retrieve 3+ years of history. 

To retrieve call history from a main.db file, you have two options:

1. You can use third party software like SkypeLogView(er) [nirsoft.net] to view the data contained in the main.db file.

-- or --


2. Use the computer that contains your history with the main.db file or copy your Skype name folder onto your newer computer that has no such history. You can use the export feature in Skype for Windows Desktop 8.x to export that older history to a searchable .html file. You might have to recreate a Skype folder in the %appdata% folder on any other computer and copy the Skype name folder, so that the desktop software can detect that older history.

Hi LASSD,

The 'NotifySender' condition applies to client workloads like Outlook & OWA.  These workloads only support a small set of the conditions available in Exchange.  If a client-side action (like NotifySender) is combined with any conditions which are not supported on the client-side (like AttachmentIsUnsupported), then you get an error about it.  You cannot use them in the same rule.

If the attachment contains sensitive information as detected by the DLP feature then the attachment will be highlighted in Outlook and the Policy Tip will be displayed.

Currently, however, Policy Tips do not support the condition AttachmentIsUnsupported.  You will need a server-side-only rule for such a case.
