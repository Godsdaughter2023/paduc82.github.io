a. user say:

There are actually many more accounts than that. From the very beginning, computer accounts have been based on the 'principle of least privilege' which is that each type of account has the least amount of access to the computer ('privilege') that is necessary to fulfill its role. Many accounts have been created for a singular purpose, so they have only the privileges that are needed to carry out that role.

In addition, some accounts are 'system' accounts that are used only by the operating system while other accounts are 'user' accounts that are used by you and me. As Andre explained, system accounts are normally hidden from you and me because we're not supposed to be using them.

Administrator is a system account while Administrators is a user account. (Technically, Administrators is a category of user accounts.) Both the system account and the user account have 'elevated privileges' (lots of access) on the computer.

Network Service is a system account for granting access to network services. Sorry, I'm not familiar with the Network account.

Guest is a user account that is a member of the Guests category of accounts. Not long after Windows 7 was released, Microsoft asked people to stop using the Guest account because it's not secure.

Default is a system account that is the basis for all user accounts. Don't ever change the Default account.

Defaultuser is the username assigned to a user account when the user hasn't chosen their own name or has chosen an invalid name.

It's very normal to have dozens of processes loaded at any given time. Even more than a hundred. By itself, the number of processes means nothing. What matters is what those processes are doing. Many processes are just sitting around waiting for something to happen.

Modern web browsers like Chrome open each tab in its own process. That way, if one tab crashes it doesn't take down the entire browser.

All this information and lots more is available by searching the internet, where simpletons with a spirit of adventure don't remain simpletons for long  ;-)

Although the procedure you tried to use (the infamous procedure outlined in KB307545) sometimes is helpful, it should NOT have been the first thing you tried! So what you need to do now is undo what you did and THEN check your disk for errors, etc.

Step 1:

While in Recovery Console, run these commands (carefully, they must be EXACT!):

copy c:\windows\tmp\system.bak c:\windows\system32\config\system

chkdsk /r

That's pretty much it. It will likely take a long time. It might even seem to be stuck. You must be patient! Actually, you might even need to run chkdsk /r two or three (!) times until it finally fixes your disk.

EDIT:

It's possible that might not solve your problem. But it should still be the first thing you try. In the event you still have a problem, you essentially need to try the other methods outlined here:

http://support.microsoft.com/kb/314477

I will simplify it. First of all, you can totally disregard Method 3. :-)

You will basically need to correct your boot.ini file. The following is courtesy of regular contributor JoseIbarra (and it synthesizes the information into something more coherent and accurate):

cd \   [if you are not already at the C:\ prompt]
attrib -shr boot.ini      (removes the System, Hidden, Read Only attributes)
ren boot.ini  boot.ini.bak     (rename the potentially afflicted file)
bootcfg /rebuild    (run bootcfg /rebuild and create a new c:\boot.ini file)

The reason you need to rename the current (potentially afflicted) boot.ini is so that when you run the bootcfg /rebuild command, you do not build on top of the current suspicious boot.ini file.  

If you are going to use bootcfg /rebuild, always rename or delete the current boot.ini first, or you will end up with multiple XPs being offered the next time you boot.  This tidbit is missing from the so many instructions... including Microsoft instructions so trust me on this one.

When bootcfg /rebuild asks for Load identifier enter whatever text is appropriate (this is strictly for show), such as:

Microsoft Windows XP Professional  or  Microsoft Windows XP Home

When it asks for load options enter:

/fastdetect  (thats a forward slash and fastdetect)

Press Enter to create the new c:\boot.ini file.

When you get done, you can see the contents of your new c:\boot.ini file by entering this command:

type  boot.ini

If the c:\boot.ini file doesn't look right, just delete it and start over.

Here is what a generic c:\boot.ini file looks like for a single partition XP Pro System:

[boot loader]
timeout=5
default=multi(0)disk(0)rdisk(0)partition(1)\WINDOWS
[operating systems]
multi(0)disk(0)rdisk(0)partition(1)\WINDOWS="Microsoft Windows XP Professional" /fastdetect
