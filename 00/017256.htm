ken sheridan say: Rather than calling the IIF function in the query use simple Boolean logic, e.g.

PARAMETERS ([Forms]![frm_Main_Menu]![Eff_In_Symbol] TEXT(2),
[Forms]![frm_Main_Menu]![Eff_Out] DATETIME;
SELECT *
FROM [YourTable]
WHERE ([Forms]![frm_Main_Menu]![Eff_In_Symbol] = "="
    AND [YourDateColumn] = [Forms]![frm_Main_Menu]![Eff_Out])
OR ([Forms]![frm_Main_Menu]![Eff_In_Symbol] = ">"
    AND [YourDateColumn] > [Forms]![frm_Main_Menu]![Eff_Out])
OR ([Forms]![frm_Main_Menu]![Eff_In_Symbol] = "<"
    AND [YourDateColumn] < [Forms]![frm_Main_Menu]![Eff_Out])
OR ([Forms]![frm_Main_Menu]![Eff_In_Symbol] = ">="
    AND [YourDateColumn] >= [Forms]![frm_Main_Menu]![Eff_Out])
OR ([Forms]![frm_Main_Menu]![Eff_In_Symbol] = "<="
    AND [YourDateColumn] <= [Forms]![frm_Main_Menu]![Eff_Out]);

Be sure to build WHERE clause in SQL view, and save the query in SQL View.  If you switch to design view before saving the query Access will move things around. At best the logic will be obscured; at worst the query will become too complex to open.  Note that the parentheses are crucial to force the OR operations to evaluate independently of the AND operations.

Also be sure to declare the [Forms]![frm_Main_Menu]![Eff_Out] parameter in particular to ensure that the values are not misinterpreted as arithmetical expressions, and to cater for variations in regional date formats.

a. user say:

Yammer OAuth2 SSO fails on non-standard JSON Access Token response deserialization

Integrating with Yammer's OAuth2 provider is much harder than it needs to be, due to Yammer's non-standard /oauth2/access_code response.

What the standard tells OAuth2 client implementations to expect: http://tools.ietf.org/html/draft-ietf-oauth-v2-22#section-5.1

{
  "access_token":"2YotnFZFEjr1zCsicMWpAA",
  "token_type":"example",
  "expires_in":3600,
  "refresh_token":"tGzv3JOkF0XG5Qx2TlKWIA",
  "example_parameter":"example_value"
}

What Yammer provides: https://developer.yammer.com/docs/oauth-2

{
  "access_token": {
    "token": "ajsdfiasd7f6asdf8o",
    "view_subscriptions": true,
    "expires_at": null,
    "authorized_at": "2011/04/06 16:25:46 +0000",
    "modify_subscriptions": true,
    "modify_messages": true,
    "network_permalink": "yammer-inc.com",
    "view_members": true,
    "view_tags": true,
    "network_id": 155465488,
    "user_id": 1014216,
    "view_groups": true,
    "network_name": "Yammer",
    "view_messages": true,
    "created_at": "2011/04/06 16:25:46 +0000"
  },
  "...": "..."
}
There might be various ways to work around this problem (https://github.com/spring-projects/spring-security-oauth/issues/623) but it would be much better, if Yammer could provide a way to give a standard=true parameter to the /oauth2/access_token endpoint, which would then return an Access Token response which would follow the standard.

doug robbins - mvp office apps & services (word) say:
Check the formatting for the first paragraph on Page 4.  Possible it is formatted to have a page break before it, or to have the lines kept together AND to be kept with next.

suzanne s. barnhill say:

If you have created an outline composed solely (or primarily) of numbered headings, then that's the problem. All of Word's built-in heading styles are formatted as "Keep with next." When you have a long series of nothing but headings, they will stay together in a clump until a single non-heading paragraph appears, when, with a sigh of relief, they will break to the next page. If you're creating an outline document that consists primarily of headings, it it wise to remove the "Keep with next" formatting from most of the styles (leave it for just the top few--say Heading 1 and Heading 2).

 

The "Keep with next" setting is on the Line and Page Breaks tab of the Paragraph dialog, which can be accessed by right-clicking in a paragraph and choosing Paragraph... from the context menu. Make the change in any paragraph of a given style, then right-click, choose Styles..., and, on the flyout gallery/menu, choose Update Heading x to Match Selection.
