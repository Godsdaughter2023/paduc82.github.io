32bit QueryInterface for Excel _Application interface fails after - Microsoft Community

dmitry matrokhin say:

32bit QueryInterface for Excel _Application interface fails after upgrade from 32bit to 64bit office

Hello!

Steps to reproduce:

1. Install 2013 32bit office with ClickToRun installer;

2. Install 365 64bit office with ClickToRun installer (You should click "Install Anyway");

3. Run some 32bit application that uses excel COM-interfaces (see examples section);

It will fail on QueryInterface of GUID {000208D5-0000-0000-C000-000000000046}.

If we go to HKCR\WOW6432Node\Interface\{000208d5-0000-0000-c000-000000000046}, we will see that it has no value. In normal scenario it should have "_Application" (Default) value instead.

You can check a lot of interfaces in registry that are corrupted after this upgrade.

Some examples of corrupted interfaces:

{000208d5-0000-0000-c000-000000000046} - _Application

{000208d6-0000-0000-c000-000000000046} - _Chart

{000208d8-0000-0000-c000-000000000046} - _Worksheet

If you install 64bit version of office without upgrading, then these keys will not be in registry. And everything will work, because it exists not in WOW6432Node.

Root cause:

Using procmon I found that while installer is running it calls AppVCleaner to delete old registry values. And what I see is AppVCleaner calls RegDeleteValue but never calls RegDeleteKey. That's why we are leftover with all these corrupted keys.

Fixes:

There are 2 possible fixes I see right now:

1. Uninstall 64bit version, than install 32bit version using MSI installer, than uninstall 32bit version, than install 64bit version. But if you don't have MSI installer of some 32bit version it's not the case.

2. Delete all corrupted keys manually. But there are too many of those. I found ~600 Excel keys that are corrupted. In procmon I see ~1900 office keys that are corrupted by AppVCleaner.

Please provide some script/diagcab/some other easy way to fix all these keys.

I think it will be better if somebody from Microsoft write such a script, because he will know why it was done this way initially.

Examples of code that fails:

C++:

Excel::_ApplicationPtr app;
HRESULT hr = app.CreateInstance(L"Excel.Application", NULL, CLSCTX_LOCAL_SERVER);

C#:

Microsoft.Office.Interop.Excel.Application excel = new Microsoft.Office.Interop.Excel.Application();

excel.Visible = true;

Why MSI installer works?:

It works, because it successfully removes all interface keys when we do "Uninstall" from "Apps & features".

ClickAndRun uninstaller just forgets all these keys.

This would be considerably easier without errors in the range. IMHO you would be better off dealing with those if that's possible.


To highlight yellow the bottom 10%, select A1 then drag to capture all the rows in Col A that you want then:-


Home tab | Conditional formatting | New rule | use a formula to decide which cells to format and enter this formula. Click the format button | Fill tab | Choose yellow and OK out.


=AND(A1>0,A1<=SUMIF($A$1:$A$100,"<1E100")*0.1)

=A1=SMALL(IF(ISNUMBER($A$1:$A$100),$A$1:$A$100),COUNTIF($A$1:$A$100,0)+1)


Note that both these formula work for up to 100 rows. If you have more than that then increase the 100. Also note that both formula assume column A and that the data start in A1.

Your stated logic is a bit ambiguous so here are two potential solutions,

=IF(AND(AB3="X",AC3="X"),"","N/A")        'both of the cells have to contain X for blank cell, N/A otherwise
