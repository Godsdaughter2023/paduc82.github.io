  1. Login to https://people.live.com
  2. Click on Import from file, located on the center of the page.
  3. Under Step 2, choose Microsoft Outlook (using CSV).
  4. On Step 3, click on Browse and choose the file that you exported.
  5. Click on Import contacts.
To import your calendar to Outlook.com, follow the steps below:
  1. Log-in to Outlook.com calendar in this link:  https://calendar.live.com .
  2. Click Import located on the top of the page.
  3. Under Import, select Import into a new calendar and then click Choose File and find where you save the .CSV file from Office 365 and then click Open.
  4. You can create a Calendar name, choose a Calendar color, Charm and Reminders.

Resetting Windows Update Components will fix corrupt Windows Update Components and help you to install the Windows Updates. Please follow the below steps to reset the Windows Updates Components:

1. Press Windows Key + X on the keyboard and then select "Command Prompt (Admin)" from the menu.

2. Stop the BITS, Cryptographic, MSI Installer and the Windows Update Services. To do this, type the following commands at a command prompt. Press the "ENTER" key after you type each command.

net stop wuauserv

net stop cryptSvc

net stop bits

net stop msiserver

3. Now rename the SoftwareDistribution and Catroot2 folder. You can do this by typing the following commands in the Command Prompt. Press the "ENTER" key after you type each command.

ren C:\Windows\SoftwareDistribution SoftwareDistribution.old

ren C:\Windows\System32\catroot2 Catroot2.old

4. Now, let’s restart the BITS, Cryptographic, MSI Installer and the Windows Update Services. Type the following commands in the Command Prompt for this. Press the ENTER key after you type each command.

net start wuauserv

net start cryptSvc

net start bits

net start msiserver

5. Type Exit in the Command Prompt to close it.

Now you may try running the Windows Updates and check if the above steps resolve the issue.

jllatham say:

First, you don't need the Load frmFinder statement, the frmFinder.Show statement that follows it will automatically load it.

The Subscript out of Range could be caused by a statement within the form's code failing.  Does the form have code associated with the UserForm_Initialize() event?  I suspect that it does, and that the actual line of code that is failing is within it.  These are a little tough to debug.

To see if the problem is in the code in the form's _Initialize() event, go into that section of code and early in it, put aStop statement.  Then when your code above is run and the form is being loaded, it will hit that statement and the VB Editor will open showing that line of code highlighted.  Use the [F8] key to single-step through the code in the process, watching closely until it suddenly fails on you.  That will be the line with the actual error in it.  If it is a particularly long process, you can put comments like'Ok to this point into it so you know how far you've gotten without error.  Then if you hit an error and want to to get closer to it quicker next time, move yourStop statement down to the last 'OK entry you made and try again.  Repeat until you pin down the errant line of code and then determine what array or collection you're trying to reference that is not being dealt with quite right.

chip pearson say:

The problem is not with the actual line of code "Load frmFinder". The problem lies in the code module of the frmFinder form. The reason that VBA reports the line "Load frmFinder" as the problem is that you have the VBA Error Trapping option set to "Break on unhandled errors". If you change this setting (VBA, Tools menu, Options item, General tab) to "Break in class module", VBA will enter break mode within the frmFinder code module on the line that is really causing the problem. As a general rule, you should always use "Break In Class Module".

A "subscript out of range" error arises when you attempt to access a non-existant item in a collection. For example, Set WS = Worksheets("NoExists"), will throw an error if there is no worksheet named "NoExists".

Cordially, Chip Pearson Microsoft MVP, Excel Pearson Software Consulting, LLC www.cpearson.com

chip pearson say: No need for all that Stop code business. (And embarassing if you forget to remove it when you're done, too.). Just change the VBA Error Handler option to "break in class module" and the debugger will take you directly to the offending line of code within the object module. This applies to errors in all types of  object modules (classes, ThisWorkbook, the Sheet modules, and user form code modules). If the error handler is not set to "break in class module", the debugger takes you to the line of code that instanitates the object (e.g., a Load function) rather than the offending code within object module itself.
