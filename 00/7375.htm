ken sheridan say: Or you can do it the other way around.  Base a form on the Accounts table and for the ClientID control use a combo box set up as follows:

ControlSource:    ClientID

RowSource:     SELECT ClientID, FirstName & " " & LastName FROM Clients ORDER BY LastName, FirstName;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.

If you want to see other values from the Clients table in the form base it on a query which joins the Clients and Accounts tables on ClientID, returning all the columns from Accounts table plus those from Clients you want to see in the form.  You won't need to return the ClientID, FirstName or LastName columns from the Clients table in the query of course, as they are already handled by the combo box.  In the form add text box controls bound to the additional columns from Clients you want to show and set their Locked property to True (Yes) and their Enabled property to False (No) in each case.  Once you select a client in the combo box the text boxes will show the values from the other columns in Clients for the selected client.

ken sheridan say: PS:  an additional piece of functionality you can add to a form set up as I described above is the ability for you to enter a new client directly by typing their name in the format FirstName LastName into the combo box in the Accounts form.  By putting code in the combo box's NotInList event procedure this will open a form in which you can then insert additional client data before returning to the accounts form.

You'll find an example in the file NotInList.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you type a new name into the combo box in the opening form in this little demo file, after being asked to confirm, a form bound to the Contacts table will open with the new names already in the FirstName and LastName controls as their default values, ready for the other data to be added.  Note that setting the DefaultValue properties of the two controls does not initiate a new record, which only happens when further data is added, so if the form is closed without doing so, a new row won't be inserted into the table.

To see how it works take a look at the code in the NotInList event procedure of the combo box on the opening frmGotoContact form, and that in the Open event procedure of the frmContacts form.  The latter code parses the name passed to it as its OpenArgs property into separate first and last names and assigns these to the DefaultValue properties of the relevant controls.

jay freedman say:

If the results of the two merge fields are in two cells of a single table in the document, you should be able to extract those strings in the macro. You need to know which table in the document (the first, second, etc.) contains the data, and which row(s) and column(s) in that table contain the individual strings -- that is, which cells in the template contain the corresponding merge fields.

The small gotcha in dealing with the contents of tables cells is that each cell ends with an invisible cell marker. If you just assign the cell contents to a string variable, the cell marker will be included, which you don't want. There are a couple of ways to exclude the marker; I'll show one of them in the sample code.

As an example, if the fields are in the first table in the template, in the second row of that table, and in columns 3 and 4, then you can use code like this in the template:

    Dim subj1 As String
    Dim subj2 As String
    Dim oTbl As Table
    Dim oRg As Range
    
    Set oTbl = ActiveDocument.Tables(1) ' first table in document
    
    Set oRg = oTbl.Cell(Row:=2, Column:=3).Range ' second row, third column
    oRg.MoveEnd Unit:=wdCharacter, Count:=-1 ' exclude the marker
    subj1 = oRg.Text
    
    Set oRg = oTbl.Cell(Row:=2, Column:=4).Range ' second row, fourth column
    oRg.MoveEnd Unit:=wdCharacter, Count:=-1 ' exclude the marker
    subj2 = oRg.Text
    
    ' when setting up the mail message...
    With oItem
        .To = "me@me.com"
        .Subject = subj1 & " " & subj2
        ' ...
    End With

phillip m jones say:

There are three types of VBA in use at Microsoft. 

  1. Content Controls
  2. Active-X Controls
  3. Legacy Form Controls

On Microsoft Office 2011 and lower only the third type works both on Mac and a Windows machine.

On Office2016 Mac, it is reported type #1 although can not be written by the Mac version, it does recognize type #1 and will acknowledge, accept, and work when using the Mac version. Unless Apple rescinds its ban on Active-X, Active-X will never work on a Mac.

 

A little history: In 1995 a company called Netscape came out with an Active-X plugin. After install of that plugin there many reports to Apple about Strange issues with the system at the time. So Apple tested the plugin. (At the time Active-X was a Microsoft design, and since placed in the public domain)  They discovered that as written there were no safety controls built in to prevent anyone from writing malicious code to damage operating systems. So next system update they wrote code to reject  Active-X and put out a ban on Active-X.
