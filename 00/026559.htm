john w. vinson say:

Yes, it's called an "Append Query".

But don't do it!

Storing "the same" data - tasks - in two tables is generally bad design. Unless you have some demonstrated - not assumed - need to move the data out, it's almost certainly better to keep all the tasks in one table with a Status field. It's dead easy to create a Query selecting just "not started" or "completed" tasks from the table; the data in this query can be shown on a form, used as the rowsource of a combo box to select a task, printed on a report, whatever. With proper indexing and query design this would let Access comfortably handle tens of thousands of records.

If you have a million task records, and/or a demonstrated performance problem, you can run an Append query to copy the records, followed by a Delete query to delete the old records. It's doable but a bit risky if the append and delete queries get out of synch, and generally not necessary.
