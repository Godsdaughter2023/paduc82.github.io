The measurements you've given for the vertical pitch and horizontal pitch must be wrong -- each of them is as large as the entire page dimension. The vertical pitch (as shown in the Preview picture in the dialog) is the distance from the top of one label to the top of the label immediately below it. Measure the distances on an actual label sheet and enter those.

willy denoyette say:

This normal.

Following built-in SID's

S-1-5-18

S-1-5-19

S-1-5-20

are the built-in SYSTEM, LOCAL SERVICE and NETWORK SERVICE account SID's, created during Windows install time.

The S-1-5-21-xxxxx-xxxxx-xxxxx.yyyy

Is a normal user account SID

The yyyy is the RID, user accounts are starting with 1000 and this number is upped each time a new user is added.

The xxxx-xxxxx-xxxx is 96 bit number created at Windows install time, this number is a statistically unique number, no two devices in the world will have the same number. 

A SID that has no corresponding user account, is called an orphaned SID. Orphaned SID's are the result of a incomplete/failed account deletion.

Orphaned SID's are nothing to worry about, you can simply removed them, just like you can remove them from them from the file system objects (folders/files) security properties.

Some other built-in user account SID's (not found in the registry) are:

S-1-5-21-xxxxx-xxxxx-xxxxx.500   This is the ADMINISTRATOR SID

S-1-5-21-xxxxx-xxxxx-xxxxx.501   GUEST SID

S-1-5-21-xxxxx-xxxxx-xxxxx.503   HOMEGROUPUSER$

Note that xxxxx-xxxxx-xxxxx is the same for all user account SID's in the system.

And there are a lot more of them......

ken sheridan say: The firstdayofweek argument of the DatePart function defaults to Sunday, so you should be alright on that score.  However, if you have data for more than 12 months duration the function will return the same value in each year, so data from two years will be aggregated together rather than independently.  You can avoid this by first grouping on the year of the date.  My WeekStart function avoids this by returning a date/time value of course.

To return a string expression put the following in the field row of a column in query design view:

WeekName:"Week " & DatePart("ww",[Date])

I would strongly advise against the use of Date as a column name, however.  As the name of a built in function it is a 'reserved' word, and should not be used as an object name.  Always use a name more specifically descriptive of the  attribute which the column represents, e.g. TransactionDate.
