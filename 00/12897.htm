john w. vinson say:

If you want to join Mgr_1 to Mgr_1, Mgr_2 to Mgr_2 and so on, I don't see why that would be "a mess" - you can join by up to ten fields. NULL values can be an annoyance but a manageable one. Karl's expression should work too, but the query will probably not be updateable.

If, however, you want to match Bob as Mgr_1 in Table1 to Bob as Mgr_3 in Table2 - i.e. if you want an exact match irrespective of the order of names in the record - you have a real problem!

At the root of the problem is your table structure: the Mgr fields are a "repeating group". If you have a Many to Many relationship (each Employee may have one or more Managers, and each Manager may supervise one or more Employees), you should really have three tables: a table of Employees, a table of Managers (which very likely would actually be just another instance of the Employee table selecting those people who are in fact managers), and a "resolver" table Management:

Management

  ManagementID (autonumber primary key)

  MgrID (fk to Managers)

  EmpID (fk to Employees)

  <other fields, such as MgrNo with values 1 to 5, date they became manager, etc.)

With this structure a self join query would allow you to find employees with the same set of managers, regardless of the sequential order.

What is the real-life meaning of the Mgr fields? I may be misinterpreting; and how are you determining the desired value of Area? It's not obvious from your sample!

ken sheridan say:

It's actually hierarchical.

As John has pointed out the fundamental problem is in the faulty table designs, neither being normalized.  A simple tree structured hierarchy, in which there is only one possible path between any two nodes, is easily modelled by a single table, Employees say, in which a foreign key ManagerID references the primary key EmployeeID of the same table.  A more complex hierarchical structure, the classic example being a bill of materials, in which there can be multiple paths between nodes, is modelled by an 'adjacency list' table with foreign keys EmployeeID and ManagerID, each referencing the primary key of Employees.

In each case it is a simple task to generate the hierarchical 'chains' to a fixed number of levels by means of a fixed number of outer joins.  In the case of an arbitrary number of levels recursive querying is necessary, and while JET/ACE SQL as used by Access does not support this, it is possible to simulate it.  You'll find an example as BoM.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

As well as illustrating the generation of a bill of materials to an arbitrary number of levels, the demo also includes a file which illustrates the simpler means of doing so to a fixed number of levels.  Either approach could be applied to a management hierarchy, but the latter, simpler, method would probably suit your requirements.
