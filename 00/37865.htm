tryx3 say:

Windows 10 - Transfer downloaded Maps to another computer

Technical Level : Intermediate

Summary

This procedure transfers downloaded maps for the Windows 10 Maps app from one computer to another without their having to be re-downloaded from the internet.

This procedure uses the Command prompt and Windows backup.  For that reason, I have marked it as suitable for intermediate users rather than by all users.  Despite this, I have still endeavoured to describe each step in sufficient detail for beginners to follow.


Details

1                   Introduction

I want to use the Windows 10 App Maps on both my computers but their downloads are necessarily quite large.  I am always keen to find ways of avoiding duplicate downloads because I use my mobile phone’s limited data allowance as my internet connection.

I had found conflicting ideas in existing forum threads about transferring maps from one computer to another so I decided to work through each idea until I managed to get the job done.

2                   The reason this transfer is not a straightforward copying task

The reason for having to transfer the maps using a quite convoluted procedure rather than a straightforward copy is that the MapsData folder has unique security permissions that cannot be set up manually.  The folder is owned by MapsBroker but that username cannot simply be added back in after a copy; Windows rejects it as an invalid name even though it is the name it uses.  So the transfer mechanism needs to transfer not just the maps themselves but also MapData’s ownership & security permissions.

Windows 10 Backup & restore facility does preserve folder & file ownership properties and security permissions so it is the appropriate tool to use for this transfer.

3                   Initial conditions

On one computer [the “Source”] I had downloaded maps of Belgium, Ireland & the UK and I had started to download a map of Germany. 

On the other computer [the “Target”] I had started to download maps of Ireland & the UK.

The downloads were run in the normal way whilst I was logged in to an MSAccount-linked user account - Settings, System, Offline maps, Download maps.

Maps was working correctly on both computers.

I was logged in a local user account user account when transferring the maps across.  I only log in to an MSAccount user account when I want to check for Windows 10 App updates in the Store or to download new maps.

Maps are, by default, in a hidden folder C:\ProgramData\Microsoft\MapData 
 
Changing the storage location merely serves to get the maps off the OS drive thereby both saving space on what might be a limited size drive [such as an SSD] & reducing the size of system images.   
 
Changing the storage location creates the folder E:\MapsData and transfers existing maps into it - this step never has to be repeated.

7                   Potential complications

7.1            Strange indications in the Target computer

On the Target computer, I checked the entries in Settings, System, Offline maps

This list is not the same as that shown on the Source computer [It is not the same as that originally shown on the Target computer either].

I have yet to identify any problem emanating from this strange condition.

7.2            Storing maps on another drive affects backup requirements

I had wanted to get the maps off my OS drive so I decided to do so before running through this procedure.  I did not want these large files using up space on my relatively small OS drive.  However, I had not considered the subject properly.

The folder E:\MapsData is, as far as it affects my backup planning, part of the system itself.  It is not a simple collection of data files like everything else on my E:\ drive.  Because of its unique ownership & security permissions, I cannot simply copy back files into E:\MapsData in the way that I can copy / move / delete / edit my own audio files in E:\Audio. 

But the downloaded maps are large.  The entire collection is about 15GB.  I do not want to use space on my OS drive for them and I do not want to enlarge every single system image by up to 15GB.

As soon as I decide how many maps I am going to download, I will

  • Create & format a partition from my E drive that is big enough for the maps I want

  • Assign it a drive letter using Disk management or another partition manager [I’ll probably call it Z: as that will always remind me that it has a rather strange status]

  • Use Settings, Offline maps, Storage location to move the maps to Z:

    - Then I will include the Z: drive in my next system image so that it is fully recoverable in the event of system corruption or HDD failure

    - After which I will only need to bother including Z: in system images if I download additional maps [which I do not expect to do very often]

8                   The alternative procedures I tried but could not get to work

8.1            Simple folder copying

I copied the whole of the MapsData folder from the Source computer to the Target computer.  There were several warnings that files were not overwritten because they were in use by the system.

I opened Maps on the Target computer but just got the blank green window that sat there for a few seconds then disappeared.

8.2            Setting Mapdata security permissions using those of the Target computer’s OS drive

I found a command in one thread that would apparently correct the security on E:\MapData by copying the settings on the root of the C:\ drive. 

Get-Acl -path C:\ | SetAcl -Path E:\MapData

but could not use it as I do not know what type of command it is.  It is not a Command prompt command.  It looks to me like it might be a Windows PowerShell command, a subject about which I am ignorant.  I tried running the command in a PowerShell session but it did not recognise the command either.

8.3            Copying across security permissions from the source computer

In the hope of correcting the evident problem with simple copying, I backed up the security permissions on the Source computer MapData folder using icacls.  I opened an elevated Command prompt window and navigated to my Desktop folder then ran

icacls E:\MapData\*  /save aclfile /t /c /l /q

and copied the file across to the Desktop of the Target computer where I ran

icacls E:\MapData\ /restore aclfile

but this had no effect on the fault symptom I was seeing [blank green disappearing Maps window].

8.4            Transferring maps using Windows backup without having deleted the original Target MapsData folder first

After I saw the suggestion by the forum user, Special K, about using Windows backup as a transfer mechanism for the downloaded maps, I tried it.  I used the procedure I gave above without preparation steps 3 & 4 i.e. without first deleting the existing MapData folder on the Target.

During a very early part of the Restore stage I had several error reports to the effect that <…> file was not being overwritten [by the backup] because it was in use by the system.  My objective was to have the Target MapData folder exactly the same as the Source MapData folder. 

So I aborted the Restore and decided to add the preparations given in section 3  Preparations above and this modification to the procedure worked.

8.5            Deleting all maps in the Target first

What did not occur to me until afterwards was that I might have been able to use Special K’s procedure without amendment if I had first gone to Settings, System, Offline maps and deleted all maps.

Perhaps this step would have cleared out MapData sufficiently for restoring the Windows backup to have worked without throwing up file in use errors.

I cannot try that now because I no longer have my original system to experiment with.  The system has been changed by what I have done.  The system image that I made beforehand did not include my E:\ drive [and hence its MapData folder] because I did not appreciate the significance of MapData’s unique ownership & security permissions.

System restore files size - Microsoft Community

lucyy2 say:

System restore files size

Is it possible to reduce the size of your system restore fil

bill smithers say:

Yes it is possible.

Read this information:

"How to Do a System Restore in Windows 8 and 8.1"

http://www.eightforums.com/tutorials/4692-system-restore-how-do-windows-8-a.html

But it does not mean that Restore Points that you created 6 months ago will still be there.

It all depends on the amount of space allocated for System Restore.

New restore points are being created automatically/regularly and as that happens older ones are deleted to fit that space allocation.

If you find that you need earlier restore points as they are no longer there you will have to alter that space allocation to allow that in the future.

You can't alter the restore points that are available presently. What's there is there, what's gone are gone.

But remember that you use too much hard drive space if you wish to keep restore points back to the year "dot" for your machine and which you would have to keep increasing to accommodate them and it is not practical to do that as you would lose installed programs/app, Windows updates, computer settings, etc. when you do a SR onwards from that earlier restore point.

"How to Change Maximum Disk Space Used by System Protection and System Images in Windows 8 and 8.1"

http://www.eightforums.com/tutorials/4691-system-protection-change-maximum-disk-space-usage-windows-8-a.html
