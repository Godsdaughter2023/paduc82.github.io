SpyNet membership basically means that information about detection, usage, allowed, blocked, quarantined, cleaned, etc. gets uploaded to Microsoft for analysis - anonymously.
Advanced will occasionally submit bits of data for analysis as well.

rob koch say: On the MSE main GUI screen, click the down-arrow next to Help near the uppper right corner, then select View privacy statement.

Scroll down to the section titled Microsoft SpyNet, there's a more specific description there.

It's unlikely you will ever be allowed to completely 'opt-out' of SpyNet, since this is how the system is designed to 'learn' about new threats and collect data worldwide to help defend against them.  It's actually the only real 'cost' to the product, since by allowing MSE to collect this information you are contributing to the overall protection of everyone using it.

Every attempt is made to avoid collecting any personally identifiable information, since it really has no value to the security systems that perform the collection.  In fact, it generally just gets in the way, since it's the malware and/or information directly related to it that SpyNet is really trying to collect.  These statements are merely an admission that the software doing the collection isn't perfect and may inadvertently take some personal data along with it.

If you think about it, there will be literally millions of pieces of information sent to SpyNet on a daily basis, so it's likely that very few will ever even be seen by a human.  Most are simply counted or cataloged and treated in the aggregate, for example, how many PCs found a newly detected file and where in the world (region, e.g. US) were they being found?  So the chance that anything personal of yours would ever be collected by mistake and then that anyone would actually look at is almost non-existent.

The latter.  If you upload an example Excel file that demonstrates the problem to a file-sharing website (e.g. box.net/files), I can answer the question dispositively in this case. Post the public/share URL of the downloadable file. Test the URL first, being careful to  log out of the file-sharing website.

I don't know how the SQL server and Excel exchange data, and whether that depends on choices that you can make.

If the SQL server creates a CSV or other unstructured text file, Excel does indeed truncate 1377.826289855995 to 1377.82628985599. In general, for manually-entered and other unstructured text input, Excel interprets only the first 15 significant digits, substituting zero for any digits on the right.

However, if the SQL serve creates an Excel or XML file, Excel will see the exact internal binary representation, which has more precision. Contrary to most online explanations, that is not limited to 15 significant digits.

In either case, neither decimal presentation can be represented exactly in 64-bit binary floating-point, which is what Excel uses to represent numeric values internally.  The exact decimal conversions of the internal binary representations are:

1377.82628985599 = 1377.82628985598,9898569532670080661773681640625

1377.826289855995 = 1377.82628985599,4900790392421185970306396484375

I use period for the decimal point and comma to demarcate the first 15 significant digits, which is the most that Excel formats (rounded), substituting zero for any digits on the right. That is an arbitrary limitation of Excel, not due to the industry-standard binary representation, again contrary to most online explanations.
