Junction Table - Microsoft Community

preschool mike say:

Junction Table

Just a question:  How do I know if I need a junction table?  What are the determining factors?
ken sheridan say: You'll know when you need one when you have a sound understanding what a 'junction' table represents.  It's really just a colloquial term for a table which models a many-to-many relationship type between two or more entity types.

First, though, you need to understand that a relationship type is really just a special kind of entity type, so like any table modelling an entity type it has attributes, each of which is represented by a column.  In the formal language of the database relational model, very loosely speaking a relation is the equivalent of a table, and an attribute is the equivalent of a column (the loose equivalent of a row BTW is a tuple).  The columns might be keys or they might be non-key attributes.  A table which models a many-to-many relationship type does so by resolving the relationship type into two or more one-to-many relationship types.

Let's take a simple example.  You have a database in which includes four tables, Students, Teachers, AcademicYear and Subjects.  In each year a student can be taught by multiple teacher, and each teacher can teach one or more, but not necessarily all, subjects, and might teach a different set of subjects in different years.  Firstly you have a ternary (3-way) many-to-many relationship type between Teachers, AcademicYears and Subjects modelled by a table like this:

TeacherSubjects
….TeacherID  (FK)
….SubjectID  (FK)
….AcademicYear (FK)

This table is 'all key', i.e. its primary key is a composite one of all three foreign key columns.  It has no non-key attributes, though there is no reason why it should not have if there are any attributes of a teacher's teaching of a subject in a year which you'd want to record.  Note that these attributes must be functionally determined by the whole of the key.

You also have a binary (2-way) relationship type between the TeacherSubjects entity type and the Students entity type, which is modelled by a further table:

StudentTeacherSubjects
….TeacherID  (FK)
….SubjectID  (FK)
….AcademicYear (FK)
….StudentID  (FK)

This table has four foreign keys, but the first three of them constitute a composite foreign key referencing the primary key of  TeacherSubjects.  When setting up the relationships for the table therefore, you would create three separate relationships between TeacherSubjects and Teachers, Subjects and AcademicYears respectively, but no direct relationships between StudentTeacherSubjects and Students, Teachers, Subjects and AcademicYears.  Instead you would create a single relationship between TeacherSubjects and StudentTeacherSubjects on the 3-column composite keys.  All relationships must be enforced.

You'll find it easier to visualise this if you draw out the model as described above on paper, reflecting what you'd see in the database's relationships window.

So the first step is to analyse the reality being modelled by the database in terms of its entity types and the relationships between them.   If the model accurately reflects the real world entity types and the relationships between them, the database will behave like the real world and do the job it's designed for.

As an addendum to the above, one other point to note is that a unary (1-way) one-to-many relationship type can also be modelled by a table rather than simply by means of a foreign key of one table referencing the primary key of another.  This is done where it is important that the referencing table does not have any Nulls in a foreign key column, which might be semantically ambiguous.  It's done in exactly the same way that a binary many-to-many relationship type is modelled, but in this case the foreign key in the 'junction' table which references the primary key of the table on the 'one side' of the unary relationship type is indexed uniquely.

amit tandon say:

Option 1:

Enter below formula as an array formula in cell E2 and copy down:

=INDEX(C$2:C$100,MATCH(LARGE(B$2:B$100-ROW(B$2:B$100)/10^5,ROW(A1)),B$2:B$100-ROW(B$2:B$100)/10^5,0))

Option 2:

Enter below Non-array formula in cell F2 and copy down:

=INDEX(C$2:C$100,MATCH(LARGE(INDEX(B$2:B$100-ROW(B$2:B$100)/10^5,),ROW(A1)),INDEX(B$2:B$100-ROW(B$2:B$100)/10^5,),0))

This will consider the Duplicate values. The 1st duplicate value will be returned first. If you want to return the last duplicate value first then instead "-ROW(B$2:B$100)/10^5" use "+ROW(B$2:B$100)/10^5", ie. use plus instead of minus with row number which is mentioned twice within the formula.
