Drive letters can be changed by going to Disk Management and requesting the change manually. For this, please follow the instructions below:

  1. While on the Desktop, move the mouse pointer to the bottom-left of the screen, and right-click on top of the Start icon.

Windows 7 Windows Explorer Unexpectedly Changes Files' Date - Microsoft Community

carefulobserver say:

Windows 7 Windows Explorer Unexpectedly Changes Files' Date Modified Attribute

I am in the process of switching from XP Pro to Win 7 Ult.  During this process, I came across a disturbing behavior in Windows Explorer under Win 7. 

In Windows 7 (and I hear Vista as well) Windows Explorer will sometimes change the file's Modified Date when dragging and dropping to a new location.  I have spent hours researching this issue.  In some discussions, some participants claim this is a reasonable thing, since copying a folder with files creates a new folder and new files, thus a new Modified Date is warranted as well.  I can understand this as the basis for changing the Creation Date.  This was the behavior in XP and I think NT as well.  However, changing the Modified Date is a fundamental departure from the way the File System behaved in the past, and it is also a serious departure from common sense.  At the very least, it will make it impossible to search or organize files by Modified date, since the original Modified Date (the date the actual contents of the file were last changed) is lost.  It also seriously diminishes the ability to find duplicate files by that attribute when simply copying a file alters it. 


Changing folder view options can prevent this behavior (Folder Options:View: Check Always show icons, never thumbnails).  Also, changing a file to Read Only will also stop this Modified Date change.  However, neither of these options provide a viable solution to this problem.  First, changing the folder view option then prevents viewing thumbnails of picture files.  Changing it back after the file copy causes the copied files' Modified Date to change to the Creation Date.  The same holds true for the Read Only attribute (removing it causes the Modified Date to change), not to mention it is often impractical to efficiently apply that strategy. 

I also note that this behavior is not consistent across the board.  Folders copied from a shared drive on a network PC didn't seem to exhibit this behavior.  But, files and folders transferred by using Dropbox (www.dropbox.com) did exhibit the unwanted behavior.  Worse yet, the behavior was not consistent even within the folder.  I transferred a folder with eml files from 2007 via Dropbox.  In most tests, once the folder was dragged and dropped on the Win 7 machine (either copied or moved) the Modified Date of the files was changed to reflect the date and time they were dropped.  However, in one case, two of the files did not have the Modified Date changed.  Clicking on the file or even opening it to read it did not make a difference.  The Modified Date remained unchanged.  I don't really know why that was the case in that instance, since these same files did experience the Modified Date change in earlier experiments. 

Frankly, I'm stumped and dismayed that this has not become a more major issue.  It doesn't take a lot of imagination to see how this behavior could adversely affect users archiving old e-mail files as emls and anyone copying older files who later needs to search or organize them by the Modified Date.  There are solutions that I am aware of, such as zipping files before copying or using Robocopy with the appropriate options, but these are a lot less convenient than Drag and Drop functionality.  Moreover, Drag and drop is now so ingrained in users that this is their likely natural choice for file copy or move.  Once the Modified Date is lost, it can't be retrieved without a restore from backup files - again, an impractical and time consuming option. 

I think this is a serious issue that needs to be quickly addressed by Microsoft.  Good design maintains logic and mitigates factors that can result in human error.  The above behavior fails both of these needs.  Has anyone found a fix for this (Registry change maybe) or know if and when Microsoft plans to address this?
