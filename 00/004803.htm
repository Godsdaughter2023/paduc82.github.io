Curious about the logic for VLOOKUP with "true" - Microsoft Community

oz du soleil say:

Curious about the logic for VLOOKUP with "true"

I used VLOOKUP to retrieve the ages. It's perfect if I use "false" or sort the list of names alphabetically.

Just to see what happens, I used true instead of false and the results were strange. So strange that it's not obvious why Excel generated the results that it did. 

Examples:
  • Carla resulted in an error even though she's in the lookup range
  • Marc was assigned Bill's age
  • Freddie was assigned the right age
  • Kay isn't in the look-up range at all but she was assigned Bill's age

Can you help walk through the logic?

A screenshot of the issue is here

Thanks much!
joeu2004 say: Oz du Soleil wrote:
In the example in the screenshot it makes sense that Bill's age gets assigned to so many people because 
  1. alphabetically, Marc is after Freddie.
  2. since Freddie is in the center of the list, the next step is the center of the second half of the list: Bill.
  3. It makes no sense that Bill is where he is so, the search stops.

You have the basic idea, but the details are off.  The search does not stop because Bill "makes no sense ... where he is".  Instead, the search returns the Bill's age only because Bill is the largest value less than or equal to the search key.

Change Ron to Mamie, and you will see the difference.
  1. First we look at #7 = INT((1+14)/2), Freddie.
  2. Since Marc > Freddie, we look at #11 = INT((8+14)/2), Bill.
    The search does not stop here even though Bill < Freddie "does not make sense".  Instead....
  3. Since Marc > Bill, we look at #13 = INT((12+14)/2), Mamie (now).
  4. Since Marc > Mamie, we look at #14 = INT((14+14)/2), Carl.
  5. Since Marc > Carl, we set the low limit to 15.
    The search stops because that is larger than 14, the number of items in the list.

Since Marc > Carl, Carl is presumed to be "the largest value less than" Marc.  Hence, VLOOKUP returns 29, Carl's, age.


But if #14 were Robin instead of Carl, VLOOKUP would return 31, Mamie's (was Ron's age).  The reason:  in step #5, since Marc < Robin, VLOOKUP presumes that #13 (before #14) is "the largest value less than" Marc.  (If it had been equal to Marc, VLOOKUP would have stopped at step #4.)


The #N/A error is returned if the lookup column is misordered in such a way that VLOOKUP "falls off" the bottom; that is, VLOOKUP wants to search past #1.  To see this, Change Freddie to Tom, and change Les to Xavier.

  1. First we look at #7 = INT((1+14)/2), Tom.
  2. Since Marc < Tom, we look at #3 = INT((1+6)/2), Xavier (now).
  3. Since Marc < Xavier, we look #1 = INT((1+2)/2), Todd.
  4. Since Marc < Todd, we set the high limit to zero.
    The search stops because that is less than 1.

Since Marc < Todd, VLOOKUP might want to return the index below #1.  But since that is zero, VLOOKUP returns #N/A, "not found", despite the fact that Marc is right there in #2.


In fact, if we change Todd to Larry, VLOOKUP returns 20, Marc's age, because in step #4, VLOOKUP would look at #2 = INT((2+2)/2), which is Marc.


Forgive me if that is TMI.  I just wanted to make the point that the algorithm stops either when an exact match is found or when the high limit < low limit of the search.

dhookom say:

Do you understand normalization and the fact that your table 2 has data values as column headings? Perhaps this is what you are stuck with. Also it looks like your data value in table 1 is "C1 + PD" which contains spaces while the column name in Table 2 does not. Is this correct?

The first step is to create a normalizing union query with SQL of:

SELECT [Table 2].Exp, "C1" AS Differ, [Table 2].C1 AS TheValue
FROM [Table 2]
UNION ALL
SELECT [Table 2].Exp, "C1+PD", [Table 2].[C1+PD]
FROM [Table 2]
UNION ALL
SELECT [Table 2].Exp, "C2", [Table 2].[C2]
FROM [Table 2] ;

This creates a normalized view of your data:

quniTable2
1A C1 1234
1B C1 2367
2A C1 2345
1A C1+PD 2345
1B C1+PD 4321
2A C1+PD 3456
1A C2 2345
1B C2 7896
2A C2 7543

You can then join this union query to your Table 1 to get TheValue.

Query4
1A C1 1234
1B C1+PD 4321
2A C2 7543

SELECT [Table 1].Step, [Table 1].Differential, quniTable2.TheValue
FROM [Table 1] INNER JOIN quniTable2 ON ([Table 1].Differential = quniTable2.Differ) AND ([Table 1].Step = quniTable2.Exp);

moocho9923 say:

I was using the third option in the wizard. The information looks like " Seller # 1, Seller # 2..." The values that appeared were like this, "2, 2, 2, 3, 2, 2...". The following row source is from the combo box.

SELECT [Sellers].[SellerID], [Sellers].[SellerName] FROM Sellers ORDER BY [SellerID], [SellerName]; 

The form that the combo box in in has a record source called  'TicketEventWithMyData'. The row source I used for the combo box though comes from a table called 'Sellers'. 'TicketEventWithMyData' and 'Sellers' have the SellerName in common, which is value I want to use to search for data. moocho9923 say: Okay, using the first option in the wizard allowed me to go straight to the 'Sellers' table and my values look fine now but they aren't navigating the records. scottgem say:

OK, several issues here. 

First, using the First option in the wizard is only when you want to select a value that will be stored in your table. The third option is the correct option for a search. 

Second, However, based on what you explained you do NOT want a search combo and your table relationship is wrong. "'TicketEventWithMyData' and 'Sellers' have the SellerName in common," That is not good design. SellerID should be your foreign key in your Event table. Names may not be unique plus number types perform better in Joins. So SellerName should not even be included in your Event table.

Third, I'm not understanding what is turning to numbers. The RowSource you showed looks correct. The Bound column should be 1, the Column count 2 and the Column widths should have the First column as 0. But, since you don't have SellerID in the Events table it wouldn't work. 

Fourth, A search combo should be used to find a single record in your table. But I'm guessing there is a one to many relationship between Sellers and Events. So what you want to use is a Filter, not a search combo. So what you need to do is the following:

1) Remove SellerName from the Events table and Replace it with SellerID.

2) On your form, create a combobox using the wizard. it should be bound to the SellerID field. It should display the Seller names but store the SellerID

3) Make a copy of that combo but make it unbound (delete the Controlsource). 

4) In the After Update event of that combo use code like:

If Not IsNull(Me.cboSellerFilter) Then

     Me.Filter = "SellerID = " & Me.cboSellerFilter

     DoCmd.RunCommand acCmdApplyFilterSort

End If

So when you select a Seller from that combo, you should see Filtered appear in the Navigation bar at the bottom and the numbers should show 1 of x indicating how many records in the Events table are associated with that seller. 

Some other points. People names should be broken out into their components. Its easier to concatenate fields together then to parse them apart. I generally use five fields for People names (Salutation {Mr, Ms, etc.}, First, Middle,Last and Suffix {Jr, Sr, etc.}). Your combo should hide the key column so that they can display the text column. A combo displays the first non zero width column after selection. My RowSources for a person selection look like this:

SELECT PersonID, Lastname & ", " & Firstname As Person FROM table ORDER BY Lastname, Firstname;

This way I see the person's full name, but store the ID.
