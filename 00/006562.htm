ken sheridan say:

You would include a subform in the Titles form, basing it on the TitlePersonnel table.  The subform would be in continuous forms view and linked to the parent form by setting the LinkMasterFields and LinkChildFields properties both to TitleID.  It would contain a combo box bound to Person-JobTypeID, but set up to show the referenced JobType column from the Job Type table, and an unbound  text box bound with a ControlSource property of =[cboPerson-JobTypeID].Column(2).  The latter would automatically be populated when an item is selected in the combo box.

For an example of such a subform take a look at DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

In the section on 'inserting data via a form/subform' you'll find a contacts form which includes a subform which shows the contact's employer(s) in a combo box.  In your case the combo box would be set up as follows:

ControlSource:   [Person-JobTypeID]

RowSource:     SELECT [Person-JobTypeID], [JobType], [PersonName] FROM ([Person-Job Types] INNER JOIN Persons ON [Persons].[PersonID] = [Person-Job Types.PersonID])  INNER JOIN [Job Type] ON [Job Type].[JobTypeID] = [Person-Job Types].[JobTypeID] [ORDER BY [JobType],[PersonName];

BoundColumn:   1
ColumnCount:    3
ColumnWidths:  0cm,3cm,3cm

ListWidth:           6cm

If your units of measurement are imperial rather than metric Access will automatically convert the units to inches.  The important thing is that the dimension is zero to hide the first column. Experiment with the other two dimensions to get the best fit.

Edited

ken sheridan say: You are quite close.  Firstly you should drop all the 'job type' columns from Titles.  As there is a many-to-many relationship type between Titles and Person-Job Types, this needs to be modelled by a table like this:

TitlePersonnel
….TitleID  (FK)
….Person-JobTypeID  (FK)

The primary key of this table is a composite of the two foreign keys.

You can now enter as many rows as necessary into TitlePersonnel to represent the entire cast and crew of any film, including where one person undertakes more than one role, e.g. as both director and lead actor, Kenneth Branagh's Hamlet for instance.

Additionally I would strongly recommend that you change the table and column names wherever necessary to exclude spaces and other special characters like hyphens.  These can cause notational difficulties.  Instead, either use CamelCase, as I've done with TitlePersonnel or represent a space by an underscore character, e.g. Title_Personnel.

OL'2013/'2016 uses .ost files for IMAP accounts which can only be opened by the Outlook prodile/email account used to originally create the file. In other words, can't be used on two different computers even if the same email account is configured in each.

karl timmermans (outlook mvp 2012-2018) say:
Just for the sake of accuracy, there's no difference in the way IMAP fundamentally works when it comes to email folders. Copying the "email folder" PST file in OL'2010 served absolutely no purpose whatsoever - the whole concept with IMAP is to enable multiple devices to access the same data. IMAP is email account based - not data file based.

Now the difference comes to the contact/calendar data. OL'2010 used 2 different PST files whereas OL'2013/'2016 only uses one .OST file where it stores the contact/calendar folders (under typical scenarios). If the IMAP account is deleted from the profile - the ost file is also deleted thereby all data in the "This Computer Only" folders (i.e. contact/calendar) will be lost. Also, if Outlook is unable to find the OST file associated with the account, it will automatically create a new OST file, re-sync the email folders and all "This Computer Only" folders will be empty.
