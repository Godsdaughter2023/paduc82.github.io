suzanne s. barnhill say:

The message you quote is the one you get when the column width is the problem. The issue is that "Equal column width" has to be checked in order for Word to automatically adjust the width of a column (even when there's just one column that's full margin width) when you change the margins.

For example, on your landscape A4 page, if you had the margins set to, say, 2 cm, the space between margins would be 25.69 cm. When you change the margins to 2.54 cm, the line length is 24.61. But if the Columns dialog had the column width set to 25.69 as an absolute measurement (not "floating"), then you wouldn't be able to increase the margins. Clicking on the One preset picture checks the "Equal column width" box, which changes the measurement to a "floating" one. If your document contains more than one section, you may need to do this for each section separately.

For this example, I want "Related City" to display 'City 2, City 4' in Excel. How should I do?

 

How to make the multivalued fields display the lookup values based on another table in Excel after data are exported from Access?

scottgem say:
The best answer is to NOT use a multivalue field, but use a child table. Then use a query to join to the child table and then export that query.

ken sheridan say: You need to export the result set of a query which joins the two tables, e.g.

SELECT SomeField, SomeOtherField, City
FROM TableB INNER JOIN TableA ON TableB.CityID = TableA.RelatedCity.Value
ORDER BY SomeField, SomeOtherField, City;

Note that to effect the JOIN you must qualify a multi-valued field name with .Value in the query.

However, the lookup field wizard is generally deprecated by experienced developers and should be avoided.  For reasons why see http://www.mvps.org/access/lookupfields.htm

Since the article at the above link was written Microsoft have introduced the multi-valued field facility.  This feature should only be used in very limited circumstances when interfacing with SharePoint (see Help for details).  In normal circumstances the conventional method of modelling a many-to-many relationship type by a third table (sometime called a 'junction' table) should be adopted.  This table contains two foreign key columns, referencing the primary keys of the two main tables respectively.  No column I either of the main table references a column in the other.  For example a many-to-many relationship type between tables Contacts and Cities would be modelled by a table ContactCities with columns ContactID and CityID.  Diagrammatically the model looks like this:

Contacts---<ContactCities>---Cities

The primary key of ContactCities is a composite one of both columns.

You'll find an example of this basic many-to-many relationship type and how to enter data via a form and subform as ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that, whether you use a multi-valued field or model the many-to-many relationship type by means of a table in the conventional way, when you export the result set of a query multiple rows will be imported for each row in TableA which relates to more than one city in TableB, repeating the data from TableA, but with a different City value from TableB in each.  In fact a multi-valued field is implemented under the skin by creating a hidden 'junction' table.

Update Document Properties with Merge Data - Word 2010

Does anyone know of a way to update Document Properties with merge fields? I'm implmenting mail merge templates in MS Dynamics CRM, and my users want the document properties to be filled in with customer information from the merge - as far as I can tell this is not possible. Anyone know of any tricks?

a. user say:

> Anyone know of any tricks?

I don't think anything "out of the box" will update any of the built-in properties or custom properties /in the output/

Using the "field language", you can update the values of 5 of the built-in document properties by using nested fields. e.g. you can set the Title property using

{ TITLE { MERGEFIELD someothertitle } }

where someothertitle is field in your merge data source, and each pair of {} are the "special field code braces" you can insert using ctrl-F9

The properties that allow this are AUTHOR, COMMENTS, KEYWORDS, SUBJECT and TITLE.

However, I think this will only let you set the properties of the /Mail Merge Main Document/, not the output, from the values in the first record in the Mail Merge Data Source, and I also suspect that that will not always happen early enough for the values to be copied to the output document, unless you use code to force an update.

There's no equivalent for the other builtin properties or old-style custom document properties. You could consider using Word MailMerge Events+VBA (or whatever) to set the output document properties. Beyond that, I don't know if Dynamics CRM lets you use the same type of Properties as Sharepoint does, in which case they could probably be set by code in various ways.

 

Peter Jamieson

rich michaels - mvp (greatcirclelearning.com) say:

You said you wanted to learn some VBA to do this, but I also want to point out that you do not need VBA for accomplishing it.

What you have in your current document are tabs being used to move and align the paragraphs content. What also could be used are Paragraph Left Indents. The Left Indent setting would replace any tab settings, which might be a bit faster versus changing the tab setting on each paragraph.

The concept is to make the selected signature line(s) content left aligned, and then add a left indent amount. Again, you can do this manually, but if your goal is to also learn some VBA, here is what a macro for it might look like:

Sub SetIndentLevel()
    Selection.Range.Paragraphs.Alignment = word.WdParagraphAlignment.wdAlignParagraphLeft
    Selection.Range.Paragraphs.LeftIndent = Application.InchesToPoints(4.5)
End Sub

Notice that the indent position in this example is hard coded into the macro. You could enhance it, making it more dynamic, by adding a InputBox VBA command, which prompts for a value, and then taking that value and using it for the indent setting.

This involves learning about Dim statements that can hold the value. And InputBox which I just mentioned.

Let me know if that helps.
