This will return TRUE or FALSE

=AND(A1="",T1<>"")

EDIT.. If you want to just see TRUR or nothing then you can use this

=IF(AND(A1="",T1<>""),TRUE,"")

If this post answers your question, please mark it as the Answer.

rick rothstein, mvp - excel say:
That does make somewhat more sense now.  If you have the time can you explain exactly what the "" and the <> bits do, i am trying to get a handle on what various things do in formulas.


The "" is called the empty string (where string is Excel's way of saying "text"). When you have text, such as "Hello", you enclose it with quote marks to tell Excel it is text (a collection of characters) and not a Range Name. When you us "" you tell Excel that the text string has no characters in it... that is, it is empty or blank. The <> is the way you test two values for being not equal to each other. So, this...

A <> B

is the opposite of this...

A = B

There for, this part of the formula...

AND(A1="",T1<>"")

does this... the AND says both arguments inside the parentheses must be true in order for the AND function to return TRUE as its result... the first argument says that A1 must have no characters in it and the second argument says that T1 cannot have no characters in it (that is, when you unwind the double negative, it says there must be at least one character in T1).

scottgem say:
Yes duplicating data is bad in a relational database. what sets off a red flag here is this "even though all fields are in one table". If this database is tracking appointments then there needs to be at least two tables, a person table and an appt table.

You should have a main form bound to the person table and a subform bound to the appt table, linked on PersonID. Then there would be no need to duplicate. When you enter a new appt in the subform, the PersonID is automatically populated.

ken sheridan say: The trick when doing this is not to allow the primary key of the new row in the referenced table to be automatically generated if it is, as will often be the case, an autonumber, but to firstly obtain the next number in the sequence, and then assign this to the column.  As you know the value of the key in advance, it can then easily be assigned as the value of the corresponding foreign key in a referencing table.  You'll sometimes see it said that a value cannot be assigned to an autonumber column, but that is not the case.

You'll find examples in CopyTree.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file includes an example to copy a tree-structured hierarchy, in this case of three levels.  This is rarely valid, and the demo uses the deliberately absurd example of copying a country, all its regions and all the cities in those regions.

What is more usually required is that a row in a referenced table be copied along with its many-to-many relationships with another table.  This could be so in your the case, e.g. an employee's appointment with one or more clients might be copied and the date of the appointment changed, i.e. a row from an Appointments table, along with the referencing rows from an AppointmentClients table modelling the relationship type between Appointments and Clients would be copied.  The employee's details would not of course be duplicated in the Appointments table of course, only the value of the EmployeeID foreign key column.  In the demo this process is illustrated by the copying of rows from an Orders table and an OrderDetails table.

scott mcdaniel say:

You can (and should) have a separate record for each visit.

What the others are saying is that you don't store the PERSON information in the APPOINTMENT table. Instead, you store the Primary Key value of the Person record in the Appointment record. If you then store details about that appointment, you would store them in a Appointment_Details table, and you'd store the Primary Key of the Appointments table in the Appointment_Detail record.

I have said this MANY times. Users do NOT dictate table design. If they knew how to design relational databases they wouldn't have someone else do it.

Of course there is a separate record for each visit. That's what the appointments table I described is for. You just don't need to duplicate the person information of each record. Other than the PersonID, you don't need any info about the person in the appointment record.
