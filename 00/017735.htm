ken sheridan say: As with most things in relational database design the key is to first get the underlying model, i.e. the tables, their columns and the relationships between them, correct.  You have three principle entity types, parts, sizes and colours, so you start with three tables, in broad outline like this:

Parts
....PartID  (PK)
....Part

Sizes
....Size  (PK)

Colors
....Color  (PK)

The fact that the sizes and colours are each a fixed set is immaterial.  These are data.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way, so you should have tables for these entity types.

In relational database terms the 'items' constitute a ternary (3-way) many-to-many relationship type between the above three entity types, so this is modelled by a table like this:

Items
....ItemID  (PK)
....PartID  (FK)
....Size  (FK)
....Color  (FK)

You could omit the surrogate key, ItemID, if you wish in which case the primary key of this combo box would be a composite one of all three foreign key columns.  Any table referencing this table would then need a 3-column composite foreign key, however, so an autonumber surrogate key is advisable.  The three foreign key columns, as a candidate key, should then be included in a single unique index.

In a form with three correlated combo boxes the Part combo box would be set up like this:

Name:    cboPart
RowSource:     SELECT PartID, Part FROM Parts ORDER BY Part;
BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

The Size combo box like this:

Name:   cboSize
RowSource:     SELECT DISTINCT Size FROM Items WHERE PartID = Form!cboPart ORDER BY Size;

The other properties of this control are the defaults as it has only one column

The Color combo box like this:

Name:   cboColor
RowSource:     SELECT ItemID, Color FROM Items WHERE PartID = Form!cboPart AND Size = Form!cboSize ORDER BY Color;
BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

Note the use of the Form property in the above to return a reference to the current form.  This is possible as all three controls are in the same form.

In the AfterUpdate event procedure of cboPart put:

Me.cboSize = Null
Me.cboColor = Null
Me.cboSize.Requery
Me.cboColor.Requery

In the AfterUpdate event procedure of cboSize put:

Me.cboColor = Null
Me.cboColor.Requery

You'll have noticed that the RowSource for cboColor includes the ItemID column as the BoundColumn.  This control can therefore be used for data entry in to a table which references the Items table, e.g. an OrderDetails table, in which case its ControlSource would be ItemID.  Alternatively it could be used as an unbound control, e.g. for filtering a form to a particular item or navigating to an item record.  In either case, the cboPart and cboSize controls should be unbound (unless you do opt for a 3-column composite key of course, in which case they'd be bound).

Where the cboColor control is bound, in a form for entering data into a referencing table, you will also need code in the form's current event procedure to assign values to the cboPart and cboSize controls so that they show the part and size of the current record's item.  You'll find an example in ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file, which illustrates the use of correlated combo boxes in various contexts, the Current event procedure of the frmLocations_Single form includes the following code to set he values of the county and district combo boxes to the right values for the current parish.

    If Me.NewRecord Then
        Me!cboCounty = Null
        Me!cboDistrict = Null
    Else
        Me!cboDistrict = GetDistrict(Me!cboParish)
        Me!cboCounty = GetCounty(Me!cboDistrict)
    End If
    
    Me!cboDistrict.Requery
    Me!cboParish.Requery

Don't worry about the other two lines of code in this procedure in my demo.  They are for 'undoing' the form.  You can always come back to building in that functionality later if you wish.  The above code calls the following two functions to get the district and county:

Private Function GetDistrict(varParishID)

    ' get DistrictID for current value of ParishID field
    If Not IsNull(varParishID) Then
        GetDistrict = DLookup("DistrictID", "Parishes", "ParishID = " & varParishID)
    End If
    
End Function

Private Function GetCounty(varDistrictID)

    ' get CountyID for current value of District combo box
    If Not IsNull(varDistrictID) Then
        GetCounty = DLookup("CountyID", "Districts", "DistrictID = " & varDistrictID)
    End If

End Function

After the 30 day grace period, the data is deleted and can’t be recovered, except for documents that are saved on SharePoint Online. The user could access the OneDrive for Business library via the URL. You may refer to the explanation in the Assign or unassign a license for one user section of the following article:

https://support.office.com/en-us/article/Assign-or-unassign-licenses-for-Office-365-for-business-997596b5-4173-4627-b915-36abac6786dc

 

Yes, their manager would receive the email. If the Manager field is populated for the deleted account, the manager will receive an email message that states that the site will be removed in 30 days and that access to the site is granted to the manager.

For the details, please refer to the following KB :

https://support.microsoft.com/en-us/kb/3042522


The Manager won't receive the granting permissions email when the user's SharePoint Online license is removed. The Manager can only recieve the email when deleting a user.
