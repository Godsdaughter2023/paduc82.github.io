john w. vinson say:

All fields in an Access database are "searchable", so that's not really the issue! What you need is a database that's properly structured (so the computer can work with it), with a user-friendly interface (so a person can work with it).

Part of this process is what's called normalization. If that term isn't familiar, look at some of the resources posted below and study up on it - it's absolutely VITAL to any good database!

One feature of normalization is that every kind of "Entity" - real-life person, thing, or event - should have its own Table; another is that every Table should have a "Primary Key" - a field (or sometimes a combination of fields) that uniquely identify each record in the table. An Autonumber is a very common choice for a primary key (not absolutely essential but typical). So you need to identify what Entities are of importance to your application; and you've done so with your listing. I'd expect that you would have several simple "lookup tables" such as:

tblCompanies
  CompanyID <autonumber primary key>
  CompanyName <e.g. "Star Tribune">
  MediaTypeID <Long Integer number, link to tblMediaTypes>
  <other information about this company as an organization>

tblMediaTypes
  MediaTypeID <autonumber primary key>
  MediaType  <text, e.g. "Newspaper">

tblContacts
  ContactID <autonumber primary key>
  LastName <text, indexed but NOT uniquely - different people can have the same name!>
  FirstName
  CompanyID <long integer link to tblCompanies, who does this person work for>
  MediatypeID <ditto to tblMediaTypes, what genre is their specialty>

and so on.

rohn007 say:

The lists, like the screen cap Nathan provided, are all there is.  Word has very few non printing characters that are visible.  All of the rest of the format information is not visible other than as the format displayed.

It sounds like what you are looking for is a "html" view where you can see all of the formatting codes, like WordPerfect used to do. There is no such thing built-in to Word. Word is built to be "user friendly" by hiding all of the formatting codes.

There are some features in Word you can use to show paragraph and specific character formatting. But that presumes you have an idea of where the problem lies.

Here are the articles that I have on recreating WP "reveal codes" in MS Word:

Reveal Codes1- Word for Word Perfect Users
http://addbalance.com/word/wordperfect.htm

Reveal Codes2- Checking Formatting - Word's Reveal Codes 
http://www.addbalance.com/usersguide/basic_formatting.htm#RevealCodes

Reveal Codes3 in Word - Is there life after “Reveal Codes”? -
http://sbarnhill.mvps.org/WordFAQs/RevealCodes.htm . 
This is an excellent article describing all of the native Word tools for discovering formatting.

Reveal Codes5- Reveal formatting <SHF><F1>- Displays the Reveal Formatting Pane showing all of the formatting on the selected text

Reveal Codes6- Style Inspector - display Style Pane, then click on middle button at bottom of the style pane. Displays an abbreviated version of the Reveal Formatting pane

Reveal Codes7- Formatting Problems? Install CrossEyes -
http://www.levitjames.com/Products/CrossEyes.aspx
This tool adds a pane that converts Word formatting into HTML/WordPefect like “codes”, displaying a WordPerfect like “Reveal Codes” pane. It is a great complement to MS’s inadequate attempt in the <SHF><F1> “Reveal Formatting” pane.

The last link is for an addin. I haven't looked at it for a couple of years now. But in general I did like what I saw the last time I looked.

The (relatively) new "x-file" formats introduced in Office 2007, ie DOCX in Word, converted Office from MS "native" "binary" file formats to an underlying XML format.  The DOCX file is actually a renamed "ZIP" file format that contains the document content in a small folder structure, that contains mostly XML format text files.  One of the files contains the body text embedded inside of XML formatting codes.  But Word has a tradition / history of generating really carpy HTML files. This has carried on in the generated XML code. Unlike WP, which just inserted the new codes required, Word inserts "all" of the formatting associated with a style. It makes reading the XML codes .... "challenging".  Here is an example taken from the middle of a very short file

The body text is at the top of this screen capture, I just grabbed a random section of the XML code and placed it below. Then I weeded through the XML to find the body code in that segment. Finally I highlighted the same body text in both places.  Notice the seemingly random break between 23 and files.  The carp after "doing." is part of the setup for the next paragraph ...

In other words, good luck trying to wade through the underlying XML code find the formatting being applied to the body text.

But, if you wish to, simply make a copy of the file, then add the new file type suffix ".ZIP".  Say yes to the warning you are changing the file type, then Windows File Explorer treats the file like a zip file, which is actually a folder structure with embedded files.  You are looking for the "document.XML" file in the structure, ie

... - Copy.docx.zip\word\document.xml
