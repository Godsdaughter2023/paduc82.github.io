vijay a. verma say:

Switch off the Auto Calculation Mode by Formulas tab > Calculation Options > Manual


Now, the randbetween will not change when you press enter.

If you enter randbetween formula in any cell and enter, no other cell will change. If you want to change only the cell where randbetween is entered, just press F2 in that cell and press enter, that cell will recalculate itself and no other cell will recalculate.


When your task is over, revert to Auto Calculation Mode (If you want to save your answers, you can copy and paste special as values before reverting to Auto Calculation Mode)

If you are open for VBA solution, then I have found a way at following link -

http://datapigtechnologies.com/blog/index.php/stupid-random-numbers-keep-recalculating/

You need to follow following steps -

1. Save your Workbook as .xlsm

2. ALT+F11

3. Insert > Module

4. Copy following code ans Save.

Function RandomBetween(Low As Long, High As Long)
            Randomize
            RandomBetween = Int(Rnd * (High + 1 - Low)) + Low
End Function

5. Now, in your worksheet, you can put following code

= RandomBetween(1,13)

It will generate a random number every time without changing previous one.

If it's happening with a .doc, then do definitely check whether "Remove personal information from file properties on save" is checked at File | Options | Trust Center | Trust Center Settings | Privacy Options. sylvain lafontaine say: This could happen if you create a lot of named querydefs - even if you want
to use them only temporarily - or maybe even if you simply change the
definition of a named querydef frequently.

Each time you change the definition of a named querydef, a new query plan
must be stored and Access won't free the previous - and not useless - query
plans until you do a compact&repair.

If this is your case, you should create and use unnamed querydefs if
possible (but beware, it's not always possible).

Also, most of mdb files - even those without any data - will grow up rapidly
at the beginning of their usage because Access must compile all the modules
and forms and store the result of this compilation into the mdb file; so
what you are seeing for the first hours is possibly quite normal.
