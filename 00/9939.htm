Someone who had seen this before suggested disabling "robustified UNC." The description is here. I'll also copy the relevant instructions in case the link eventually stops working:

[The OP's Outlook] issue appears to be specific to running Outlook 2007 or Outlook 2010 on Windows Vista or Windows 7. It also appears to happen more often to network shares that are either part of a Distributed File System (DFS) structure or located on storage hardware which isn’t running on Windows like a dedicated NAS system.


The solution is to disable a feature called “Robustified UNC” via the Registry;

  • Outlook 2007 Key
    HKEY_CURRENT_USER\Software\Microsoft\Office\12.0\Word\Options
  • Outlook 2010 Key
    HKEY_CURRENT_USER\Software\Microsoft\Office\14.0\Word\Options

Value name: DisableRobustifiedUNC
Value type: REG_DWORD
Value: 1


While the registry key is located in the Word hive, it actually applies to Outlook (and other Office applications) as well. Due to its location, it could be required to close all open Office applications before changes to take affect (or simply restart your computer).

...

While the effect of the registry key has not been publicly documented by Microsoft, the “Robustified UNC” feature affects the security mechanisms used when opening a file from or saving a file to a network share in Office applications.


Disabling the “Robustified UNC” feature could also solve issues such as documents not opening when double clicked from a network share (while Word does open) or hangs or slow performance when opening/saving files that are located on a network share.


Apparently, it also can solve the issue of very slow opening of documents on a network share accessed via hyperlink from within Word.

a. user say:

Just to add a little to what Bob has said, in order to ensure that numbers typed in those cells are formatted as currency automatically, you really have to use Word's forms technology. Although that's probably a bit over the top for what you're doing (because it introduces complications that you may not want), it can be done, although it will split your table. The overview is:

a. Enable the Developer tab in Word->Preferences-Ribbon
b. Use the Developer tab to put a forms "Text Box" in each cell where you want to enter data
c. right-click on each text box and select Properties, then under Type, select Number, and choose a suitable number format (it is possible to modify the formats)
d. click in the first cell in the row
e. Insert->Break->Section break (Continuous)
f. click in the first cell in the row below, or select the paragraph immediately below the table if the fields are in the last table row, and Insert->Break->Section break (Continuous)
g. Click Tools->Protect Document..., then select "Forms". Click the Sections... button and ensure that only the section containing the form fields is checked (if there are no other section breaks in the document, there will be 3 sections and you should only protect section 2.
h. You should now find that in section 2, you can only type in cells that have form fields, and that an amount such as 3000 is formatted once you tab out of the field, or click outside the field. You should be able to edit anything outside the protected area.
i. You can unprotect the form using Tools->Unprotect.

Each section break inside the table will split the table, but should not introduce any gaps. What it will do is introduce cell borders for each table, so you may need to remove one of them using Format->Borders and Shading to make it look like it's a single table. Or it may make sense to put form fields in all the other fields where you need text or numbers and "protect" the whole table or a larger portion of the document.

To illustrate the points Gina and Gary have made, below is a clip from one of my online demo files of a simple form with a subform on a tab page.  I've temporarily added a hidden control to the subreport's footer with a ControlSource property of =Count(*) to count the number of rows currently in the subform.  In the parent form I've added a text box which references this control to show the number in the parent form even when the subform is not visible.

1.  The subform control is selected.  Note from its properties sheet that its name property is sfcEvents.  This differs from the name of the underlying source form object, which is fsubEvents.

2.  Note that the expression used as the ControlSource of the text box in the parent form is =[sfcEvents].[Form].[txtEventCount], i.e. it references the hidden control in the subform via the Form property of the subform control.
