Hi Sofie,

Thanks for your updates. As the issue can be solved with the “keep me signed in” check, I’d like to explain how “keep me signed in” works in Office 365.

When we login to Office 365 without “keep me signed in” checked, session cookies are created in browser for storing our login information. When the session cookies lifetime expired, the session cookies are automatically cleared by default. So, our login information is lost. We need to re-login to Office 365 to save the changes.

When we login to Office 365 with “keep me signed in” checked, persistent cookies and session cookies are both created. When the session cookies lifetime expired, the session cookies are automatically cleared. But the persistent cookies still remain in the browser. So, our login information still exists and we are able to save the changes.

In conclusion, you may need to check the “keep me signed in” box when login to Office 365 to avoid the issue.

ken sheridan say: You are encoding data in the expressions.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

The length of training in days should be stored in a column of Integer data type in a TrainingTypes table, e.g.

TrainingTypes
….TrainingType  (PK)
….TrainingDays

Because the default length of a training type could conceivably be varied over time it is not time-independently determined by the key of the above table, so a ClientRaining table or similar would have the foreign keys TrainingType and ClientID, and the non-key column StartDate, and this table does also need a FinishDate column.  No redundancy is created.  Otherwise, if the length of a training type were changed by amending the value in the TrainingDays column in a row in TrainingTypes, existing training records would be inaccurate if these simply reference that value.  The same is true of your current incorrect method of assigning the value in code, but that would be even worse of course, as any amendment would require amendment of the code (one reason why Codd included this rule in his seminal paper back in 1985).

With the data stored correctly in the TrainingTypes table, in your data input form for inserting/editing rows in the ClientTraining table you would select a Training type in a bound combo box with a RowSource property of

SELECT TrainingDays, TrainingType
FROM TrainingTypes
ORDER BYTrainingType;

Set its ColumnCount property to 2, its BoundColumn property to 2 and its ColumnWidths property of zero to hide the first column.

In the AfterUpdate event procedures both of the combo box and of the StartDate control, assign a value to a bound FinishDate control with:

Me.FinishDate = Me.StartDate + Me.cboTrainingType.Column(0)

where cboTrainingType is the Name property of the combo box.  The Column property is zero-based, so Column(0) is the first (hidden) column.

One other factor you might need to consider here is whether you wish to add only working days to the start date, i.e. excluding weekends and public and concessionary holidays.  If so you'll find functions for doing so in Workdays..zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file includes functions for adding/subtracting workdays, for use with a single country with a single set of holidays or for in multiple countries with different public holiday dates.
