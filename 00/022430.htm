jay freedman say:

The large orange blocks on the first page contain content controls that are inserted from Insert > Quick Parts > Document Properties. The box labeled Title is the Title document property, but the Subtitle box was originally the Status document property (its name was changed later).

The orange text in the header also consists of Document Property content controls inserted the same way, using the same document properties.

Each of the items in the Document Properties list is linked ("mapped" in Microsoft jargon) to a bit of XML in the document's file. When you update any one of their content controls, all the controls mapped to the same XML automatically update with the same value.

The appearances of the various content controls are set by applying different paragraph styles to them.

This is covered in more detail in section 7, "Content Controls", at http://gregmaxey.com/word_tip_pages/repeating_data.html.

ken sheridan say: Let's first deal with the problem of the values entered into the Task Number, TT Impacted and Time controls in the form not being inserted into the columns in the Issues table.  The form should have as its RecordSource property either the name of the Issues table or a query which returns al the columns from the table.  Using a query is generally better as it allows you to order the records on one or more columns so that they appear in a logical order in the form.

Within the form you should have controls, text boxes in this case, whose ControlSource property is in each case the name of the relevant column in the table, i.e. Task Number, TT Impacted or Time.  Note however that naming a column Time is not a good idea as Time is the name of a built in function, so as a 'reserved' word should be avoided as an object name.  Something more specifically descriptive like CR_Time or similar would be better.

You should then have no difficulty entering data into the controls and it being assigned to the relevant column in the current row in the table when the form is close, you move to another record or otherwise explicitly save the record.

As regards the CR Workgroup column in the Issues table, the first question to be addressed is whether that column is either necessary or desirable.  As I said in my first reply this depends on whether it is functionally determined by the contact or not.  What this means is that when you select a contact as the CR Raiser should the value of CR Workgroup in the Issues table always be the value of Workgroup in the row in the Contacts table for the selected contact?  Or is it legitimate for the value of CR Workgroup in the Issues table to differ from that value?  If the former then you don't need the column in Issues and should return the value from Contacts in a computed control as I described in my earlier post, changing the column names in the combo box's RowSource property to the actual names.  BTW I should have mentioned that the CR Raiser's combo box's ColumnCount property should be changed to 3 in this scenario.

If on the other hand the value can legitimately differ from the value in Contacts you will need to address a fundamental design issue in your database.  You will firstly need to introduce a Workgroups table if you do not currently have one.  This will be a table with one row for each possible workgroup, so would be set up like this:

WorkGroups
....WorkgroupID (PK)
....Workgroup

The primary key WorkgroupID column can for convenience be an autonumber.

Before doing anything further back up the database.

You then need to fill this table with the names of all workgroups currently listed in Contacts, using the following append query:

INSERT INTO Workgroups(Workgroup)
SELECT DISTINCT Workgroup
FROM Contacts
WHERE Workgroup  IS NOT NULL;

You next need to add a WorkGroupID column, of long integer data type, to Contacts and insert values into it with the following update query:

UPDATE Contacts INNER JOIN Workgroups
ON Contacts.Workgroup = Workgroups.Workgroup
SET Contacts.WorkgroupID = Workgroups.WorkgroupID;

Once you are happy that the new WorkgroupID column in Contacts has the correct values you can delete the redundant Workgroup column from Contacts.  Then create a relationship beteen Contacts and Workgroups and enforce referential integrity.

Whether or not you need a Workgroup column in Contacts as discussed above, you should do the above in any case; it's just a matter of good database design.

If on the basis of the discussion above you conclude that a CR Workgroup column is legitimately required in the Issues table, this should be of a long integer data type and its bound control in the issues form should be a combo box set up as follows:

RowSource:     SELECT WorkgroupID, Workgroup FROM Workgroups ORDER BY Workgroup;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.

For selecting a workgroup in a contacts form you'd use a combo box set up in the same way.

To enter a workgroup not currently represented in the Workgroups table you can out the following code in the NotInList event procedures of the workgroup combo boxes in both the issues and contacts forms:

    Dim ctrl As Control
    Dim strSQL As String, strMessage As String

        CurrentDB.Execute strSQL, dbFailOnError
        Response = acDataErrAdded
    Else

        ctrl.Undo
    End If

When the user types a new workgroup name into the combo box and confirms it at the prompt, a new row will be inserted into the Workgroups table and the combo box's list updated to show the new workgroup.

Your description of the real life situation helps.  Let's look at things in terms of the real world entity types and how they'd be modelled by tables and relationships between them, and what the attributes, represented by columns, of each entity type would be.  The entity types as I see it are Employees, Workgroups, and Issues.  So we start with tables like this:

Employees
....EmployeeID (PK)
....FirstName
....Lastname
....etc

WorkGroups
....WorkgroupID (PK)
....WorkGroup

Issues
....IssueID (PK)
....Issue

This table records the categories of issues, not specific instances of an issue.

An employee might of course move from one workgroup to another during their period of employment with the company, so the relationship type between employees and workgroups is a many-to-many one, in this case it's a 'binary' (2-way) relationship type.  A many-to-many relationship type is modelled by a table which resolves it in to a set of one-to-many relationship types, in this case, it being a binary relationship type, a set of two:

EmployeeWorkgroups
....EmployeeID (FK)
....WorkgroupID (FK)
....StartDate
....EndDate

The primary key of this table is a composite one made up of the EmployeeID and WorkgroupID  columns.  The StartDate and EndDate columns represent non-key attributes of the relationship type, when an employee joins and leaves a workgroup.  I'm assuming for the moment that an employee will not be a member of the same workgroup for more than one stretch, but if it's possible that they could move from group A to group B and then back to group A the primary key of the table should also include the StartDate column.  Data entry would be via a continuous subform in an Employees form.

Now we come to the crux of the database, which is where the specific instances of an issue are recorded.  This is another many-to-many relationship type, so is modelled by a table.  In fact a relationship type is really just a special kind of entity type, so the table is still modelling an entity type.  The table will be like this:

IssueIncidents
....EmployeeID (FK)
....WorkgroupID (FK)
....IssueID (FK)
....Comments
....IssueDate
....etc

The primary key for this table is a composite one of EmployeeID, WorkgroupID, IssueID and IssueDate, assuming that there can be no more than one incident per EmployeeID per WorkgroupID, per IssueID per day.

In creating a form for this table, which is where the incidents of an issue will be entered, we need to ascertain whether the workgroup logged for the incident will be the current workgroup at the time of the incident, or whether an issue can be logged for an employee but with a different workgroup from that of which they are or have been the member.  This is crucial to how the relationships are set up.  If the former than an enforced relationship should be set up between EmployeeWorkgroups and IssueIncidents and would be on the two columns EmployeeID and WorkgroupID, i.e. the composite primary key of EmployeeWorkgroups is referenced by a corresponding composite foreign key in IssueIncidents.  If the latter then there should  separate relationships between IssueIncidents and Empoyees and between IssueIncidents and WorkGroups.

Whichever is the case then the form would be set up in the same way with only one small difference, which I'll come to.  There should be combo boxes bound to the EmployeeID, WorkgroupID and IssueID columns with RowSource properties drawing the data from SQL statements referencing the Employees, Workgroups and Issues tables respectively, hiding the first ID column in the usual way in each case.

When an employee is selected in the EmployeeID combo box you can automatically select their current workgroup in the WorkgroupID combo box by putting code in the AfterUpdate event procedure of the EmployeeID combo box:

Dim strCriteria As String
Dim dtmDate As Date

strCriteria = "EmployeeID = " & Me.EmployeeID

' get latest workgroup start date for selected enployee
If Not IsNull(Me.EmployeeID) Then
    dtmDate = DMax("StartDate", "EmployeeWorkGroups", strCriteria)

    ' get current workgroup for employee and assign to WorkgroupID
    strCriteria = "EmloyeeID = " & Me.EmployeeID &_
         " And StartDate = #" & Format(dtmdate,"yyyy-mm-dd") & "#"

    Me.WorkgroupID = DLookuP("WorkgroupID ","EmployeeWorkGroups", strCriteria)
Else
    Me.WorkgroupID = Null
End if

If you need the ability to change the logged workgroup from the employee's current workgroup then leave the WorkgroupID combo box as set up so that it can be edited by selecting from its list.  Otherwise set its Locked property to True (Yes) and its Enabled property to False (No) to make it read only.

One final point: you might have noticed that I've not used any spaces in table or column names.  While Access inadvisably allows spaces in object names these should be avoided.  You can use CamelCase or represent a space by an underscore character Like_This.
