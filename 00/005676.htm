You can have Excel do this automatically, without needing VBA:

  • Select the cell or cells.
  • Press Ctrl+F1 to activate the Format Cells dialog.
  • Activate the Alignment tab.
  • Tick the check box "Shrink to fit".

Range("A1:C10").ShrinkToFit = True

tushar mehta say:


AFAIK, detecting mouse movements has always depended on how busy the system is and how fast the mouse moves.  One can create this "problem" on web pages, in PP, in Excel, and I imagine in every other program.

Steve's suggestion of moving the code into another routine should *not* work because it doesn't change the amount of work the system has to do.  What you may want to consider -- and I've used this techniquie quite effectively -- is to delay processing the code by zero seconds (in Excel I'd use the OnTime method, in PP you will have to use a SetTimer/KillTimer combo).  This priortizes event detection over event handling.  The downside, of course, is that this just "stacks" up all the event processing.  Then, you have to decide if you have to execute all those requests or somehow detect that the currently executing request has been superseded by a subsequent one and if so maybe ignore the current request.  Gets messy fairly quickly. ;-)

Where I typically run into trouble is with somewhat heavy macros, like when building and firing animations during a slideshow on a mouseover, or when hiding other shapes on the slide. I use the typical settings for my mouse, so I don't think that's the issue.

No, not to sleep.  It schedules the macro for execution at or after a particular time and there are other options like 'on or after time1 but no later than time2.  The big difference between the OnTime and the SetTimer/KillTimer approaches is that OnTime is a one-off event whereas SetTimer causes the macro to be called repeatedly until stopped by KillTimer.

Excel does have a Wait method that puts the executing thread to sleep but I stay away from it because it also puts Excel to 'sleep.'

Hi,

I might suggest to capture a user mode dump of the crash and see if any further useful information can be obtained on the crash - it might point to a third party module.

You can configure Windows to create user-mode dumps. Create a System Restore Point first.

If you copy and paste the following in Notepad and save as a .reg file (save with the .reg extension and give it any name such as iedump.reg), then right-click the .reg file and select "Merge" to add to the registry, a .dmp file at the time iexplore.exe crashes should be created in the C:\CrashDumps folder:

Windows Registry Editor Version 5.00

00,6d,00,70,00,73,00,00,00

http://windows.microsoft.com/en-ca/onedrive/sha...

http://msdn.microsoft.com/en-us/library/bb787181(VS.85).aspx

Windows Registry Editor Version 5.00

You can delete the C:\CrashDumps folder anytime.
