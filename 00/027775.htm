scottgem say:

It sounds like you have a lookup field set up on the table level. This is not recommended and this is one of the reasons for it. Please see http://access.mvps.org/access/lookupfields.htm for details. Lookups should be done on the form level not on the table level. 

Had you not setup the lookup field what you would have had to have done is add your stock table into your queries with a join on the stock reference code. Then pull the description field into your query instead of or in addition to the stock reference code. And that's what you have to do here. Without seeing your expressions, I can't say where you have to substitute the description for the code. 

I would recommend that you remove the lookup field from the table level and modify your queries and modify your queries.

john w. vinson say:

Well...

 

Don't.

 

Storing multiple pieces of information in a single field (an invoice number) is a bad design.

Storing fields which depend on other fields anywhere, in any field, is an even worse idea. If you have to delete an invoice, or add an overlooked one, all your invoice numbers would now be WRONG. You can change the ones in the table, but you can't change the ones that have been printed and mailed to customers, or that have been seen and stored in someone's mind!

 

The count of invoices is just data - display it separately from the invoice number, but do so by calculating it as needed.

ken sheridan say:
But not sure even how to set up the Count in my query 
Here's an example using the Orders table from Northwind:

SELECT [Customer ID], [Order Date],
YEAR([Order Date]) AS OrderYear,
DATEPART("ww",[Order Date],2) AS OrderWeek,
    (SELECT COUNT(*) +1
     FROM Orders As O2
     WHERE O2.[Customer ID] = O1.[Customer ID]
     AND YEAR(O2.[Order Date]) = YEAR(O1.[Order Date])
     AND DATEPART("ww",O2.[Order Date],2) = DATEPART("ww",O1.[Order Date],2)
     AND O2.[Order Date] < O1.[Order Date])
AS OrderNumber
FROM Orders AS O1
ORDER BY [Customer ID],[Order Date];

One thing to note, however, is that the dates per customer must be distinct.  If two invoices for the same customer have the same date the computed number would be the same.  The above works in my copy of Northwind because in the one case where there are orders with the same customer/order date values the second has a non-zero time of day if you look at the unformatted value for the date.  If there were not the case then it would be necessary to bring in the table's primary key as a tie-breaker.  You'll find examples where this is done in RowNumbering.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file includes means of numbering rows per group.  In your case the group would be customer/year/week.

ken sheridan say: Again using the Northwind Orders table as an example, for an updatable query which breaks any ties by date/time on Order ID:

SELECT *,
Year([Order Date]) AS OrderYear,
DatePart("ww",[Order Date],2) AS OrderWeek,
DCount("*","Orders", "[Customer ID] = " & [Customer ID] &
" And Year([Order Date]) = " & Year([Order Date]) &
" And DatePart(""ww"",[Order Date],2) = " & DatePart("ww",[Order Date],2) &
" And [Order Date] < #" & Format([Order Date],"yyyy-mm-dd hh:nn:ss") &
"# And ([Order ID] <= " & [Order ID] &
" Or [Order Date] <> #" & Format([Order Date],"yyyy-mm-dd hh:nn:ss") & "#)")+1
AS OrderNumber
FROM Orders
ORDER BY [Customer ID],[Order Date];
