jay freedman say:

The setting works correctly here, so it isn't a bug in the program itself; something else is going on.

There's a single setting to display or hide the Paste Options button in all the Office programs of the specific version. To make

that possible, the setting is stored in the registry as the value named PasteOptions in the key HKEY_CURRENT_USER\Software\Microsoft\Office\14.0\Common\General (for Office 2010) or HKEY_CURRENT_USER\Software\Microsoft\Office\15.0\Common\General (for Office 2013).

It appears that your copies of Office are not saving the settings from the dialog to the registry. If you weren't seeing the same behavior in separate versions on separate computers, I would guess that an add-in was somehow interfering, but in this case it seems unlikely.

You could try changing the value in the registry to zero (0), which should be reflected in the dialog after restarting Word. To do that for your home computer, start with an empty Notepad file and paste in the following text:

Windows Registry Editor Version 5.00

[HKEY_CURRENT_USER\Software\Microsoft\Office\14.0\Common\General]
"PasteOptions"=dword:00000000

(The blank line after the first line is required.) Save it to your desktop and name it PasteOptions2010.reg. Double-click the file's icon, and accept the two message boxes. Then test Word, Excel, etc.

Assuming that works, make a copy of the file and name it PasteOptions2013.reg. Right-click the file and choose Edit, which will open it in Notepad. Change the 14.0 to 15.0 and save. Move the file to your work computer, and double-click it there.

macropod say:

Here are two macros for you:

1. This macro writes all the source xml data to the current document.

Sub SourcesBackup()
Application.ScreenUpdating = False
Dim oSrc As Source, StrSrc As String
With ActiveDocument
  .Range.Text = vbNullString
  For Each oSrc In Application.Bibliography.Sources
    StrSrc = StrSrc & vbCr & oSrc.XML
  Next
  .Range.InsertAfter StrSrc
  .Paragraphs.First.Range.Delete
  .Save
End With
Application.ScreenUpdating = True
End Sub

2. This macro restores all the source xml data from the current document

Sub SourcesRestore()
Dim i As Long
'In case the source is already present
On Error Resume Next
With ActiveDocument
  For i = 0 To UBound(Split(.Range.Text, vbCr))
    Application.Bibliography.Sources.Add Split(.Range.Text, vbCr)(i)
  Next
End With
End Sub

Over here, converting from Word to PDF only preserves the color and underline of the hyperlink, but it stops working. The same thing happens with other types of links, like URLs (this is a known problem).

 

The only workaround I can suggest is to open the PDF in Preview, press Cmd Shift A to bring up the tools at the bottom of the window and then:

  1. Select the link's text
  2. Click on the chain link icon at the bottom
  3. Select "Link type: URL"
  4. Type "mailto:*** Email address is removed for privacy ***?subject=Subject line here"

bob jones aka: cybertaz say:

I'm not sure I fully understand the question, but if I do...

The Subject information is stored in a field. When you send an email message that field populates the Subject line in the email message which is a part of the designed structure of an email message. Generating a PDF is not the same thing... it's a print operation just like printing the document on paper. The document output has no such field or Subject line to receive the Subject content. The only document content that prints is the content that is visible in the document.

The one exception would be if the Subject content were a part of the content to Display: in the document. If so, the email address would be followed by ?subject=[subject content].

vincentchoy say:

Has the client appreciated the differences between EA and CSP? For example

EA SKUs 

Most Traditional EAs are configured with a CORE CAL and MS-Office License.

If the client came from a traditional EA moving to Office365 EA, his EA plan would likely be Office365 + Bridging CAL. The bridging CAL covers the Windows and SCCM CAL avaliable in the CoreCAL under the traditional EA contract.

Does their EA plan cover a bridging CAL?

If you move them to CSP without taking into account the bridging CAL then they might lose their SCCM and Windows CAL license (if on EAS) or the SA to those licenses (If on EA).

Also Office365 via EA, the client has the ability to contact Microsoft directly for support via support ticket at no additional charge, whereas for CSP you are responsible for level 1 support and there is the addition of the delegated admin (or its subsequent removal if the client is not comfortable). If you charge for level 1 csp support can you still compete with EA pricing at 1500 users?  Also the support turnaround time might end up slower depending on how your level 1 support is provided.


Also there is the possibility of geographical region issue.

https://docs.microsoft.com/en-us/partner-center/regional-authorization-overview

This is a question that should be directed at Microsoft, with a black and white answer to back you up.

That being said, my experience is that Microsoft does not take too kindly to a reseller dismantling a existing EA. So there is that as well.

On the migration, there is really nothing much to migrate. It is more of a license reassignment which little revenue can be charged.
