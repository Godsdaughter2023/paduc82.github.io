That's correct, provided that you have properly baselined the plan by setting a baseline before.

>> This seems good at face value, but if a task actually started late and subsequently finished

You can create custom filters for that purpose but pls first check the following views and filters (you can also review the built-in filters in Filter Definition dialog box to see how they're created):

Custom filters can be applied to the built-in reports. Pls also check the built-in reports. Pls see that for more:

Just consider splits with a fixed duration task. Also consider the case where you add an exception day to the project calendar; finish date of a task may slip while there is no duration and finish variance.

ismet kocaman say: The values here are the scheduled (or current) and planned (or baseline) values. You will get actuals only when you feed actual data or other progress data into the plan; e.g. Actual Start, Actual Finish etc... Pls see the following representation of scheduled vs planned: 

Finish Variance = (scheduled or current)Finish - (planned)Baseline Finish
Start Variance = (scheduled or current)Start - (planned)Baseline Start

So "actual days" is not a correct term to use in this context. If you mean the variance in duration, it is the Duration Variance field. No need to calculate it by using "Finish Variance - Start Variance", instead just insert the Duration Variance field into the Variance table to see the corresponding values. I believe that'll do it.
