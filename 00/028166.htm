If we expand the expression:

([WelBalDue]+[PenBalDue]+[AnnBalDue]+[OtherBalDue]>0)

we get:

(([WelAmountDue]-[WelCollect]-[WelAdj])+([PenAmountDue]-[PenCollect]-[PenAdj])+([AnnAmountDue]-[AnnCollect]-[AnnAdj])+([OtherAmountDue]-[OtherCollect]-[OtherAdj])>0)

([TotalWelFind]+[WelBillAuditFee]+[WelLiqDam])

and so on with the references to any other computed columns.  Try it again, and keep expanding the expressions until it works or you are left with references to base columns only.  I would not rule out the possibility of the final expression being too complex to evaluate, however, in which case the best thing might be to break it down into two queries by removing the WHERE clause completely, and then create another query:

SELECT *

WHERE......

The WHERE clause could then reference the column headings returned in the query rather than the expressions.

marshall barton say:

The traditional way to do that is to add an unbound text box (named txtSearch) to the form header section where you can enter the first or last name.  Assumin the first name and last name are in two different fields, use the text box's AfterUpdate event to run VBA code like:

With Me.RecordsetClone
   .FindFirst "[FirstName]=""" & Me.txtSearch & """ OR [Lastname]=""" & Me.txtSearch & """"
   If .NoMatch Then
      Beep
   Else
      Me.Bookmark = .Bookmark
   End If
End With
