     Among the things I do to prevent finding myself in a similar situation is to "carry my eggs in more than one basket"; to this end I deploy a local e-mail client (program) installed directly on my computer.  My choice is Windows Live Mail 2012 (part of Windows Live Essentials), because it offers both automatic synchronization with the "in-the-cloud" webmail client Outlook.com as well providing a locally-maintained copy of each message.  For many years I basically ignored the webmail portion of Microsoft's ever-evolving e-mail solutions (Passport/Hotmail.com/Outlook.com), preferring to manage mail through whatever locally-installed client I was using (Outlook Express/Outlook/Windows Live Mail for me, but there are many others available) and the backup mechanisms I was comfortable with (I prefer the free MailStore Home for this.)  

     The proliferation of multiple devices in today's Bring Your Own Device ("BYOD") environment prompted me to begin using Outlook.com more regularly, and I find that as long as I perform a couple of simple "maintenance tasks", which I originally learned through the use of the venerable Outlook Express local client, have seemingly sufficed to keep my instance of WLM virtually trouble-free, and that Outlook.com - through the automatic synchronization mechanism I mentioned previously - follows suit.

    These tasks involve 1) "compacting the databases" and 2) keeping the "Sent items" folder from

becoming too bloated with old messages, both of which could cause problems with Outlook Express

functionality, and both of which, I believe, can adversely affect both the Window Live Mail local client and the Outlook.com webmail client.  For more information about the first task, see Community star ¡Firedog's April 30, 2012 answer at the top of the thread "I cannot delete messages from the "Sent" File"; the method I use to perform the second task involves the use of WLM's "Storage Folder" - I haven't actually done this in the Outlook.com webmail client's "Sent" folder, since the afore-mentioned synchronization with WLM yields the same end result, but a user can do in Outlook.com exactly what I describe below for the local client by using the "Move" feature to transfer old sent messages to a different folder. 

     Hovering the cursor on the "Storage folders" icon (in my local client, located at the bottom of the folder tree, below the Outbox) invokes an explanatory dialog box with the phrase "Store items here that you want to save on your computer but not on your mail server".   I use this location to "archive" my old sent messages in a folder other than the "Sent items" folder itself.  As I prefer to keep a copy of every message I send, I simply create a new folder each month, which I title in a format that allow me to easily see that the folder contains sent messages for that particular month (I use 0113, 0213, 0313, etc.). I create these folders "nested" under a folder I created called "OldSent" in the "Storage Folders" location, and at the beginning of every month I move all the messages I've sent during the previous month out of the "Sent items" folder and into the folder I've created for that month.

     One last thing I should mention is that I find the ability to create various folders -whether in the webmail client or the local one - and then to apply message rules to various senders, thereby routing messages from them to whatever folder I choose, makes life much easier, as I no longer have to sort through the 75-100 messages I receive daily all jumbled together in my Inbox.   The article I've just linked applies to Outlook.com message rules; to do the same for Windows Live Mail I suggest having a look @ "Use Message Rules" (part of Community star ¡Firedog's excellent Windows Live Mail Stuff blog).

john w. vinson say:

For an "all of the above" answer, I prefer to avoid the LIKE operator and instead use a "fieldless criterion":

WHERE
([fieldname] = [Forms]![frm_Runs].[cbo_Points_Test_Point_Type_From]
OR [Forms]![frm_Runs].[cbo_Points_Test_Point_Type_From] = "All")
OR
([fieldname] = [Forms]![frm_Runs].[cbo_Points_Test_Point_Type_To]
OR [Forms]![frm_Runs].[cbo_Points_Test_Point_Type_To] = "All")

If you select "All" in either combo box, the expression will be TRUE (regardless of the value of the field in the table) and the record will be retrieved.

One problem with the LIKE criterion is that you might have overlapping values - MICRO, MICRON and MICROSOFT or SMITH, SMITHERS and SMITHSON for example - which would give you false hits.
