atlbo say:

Formula to Count Values for a Range

Need a formula that will count the times a value can be found in a range of a worksheet

Here are the specs:

-I need a count of the number of times a specific value appears in a row over a range in the worksheet "DataSheet".  This formula will be pulled down and applied to numerous unique values in the destination worksheet,  "2 Year Mojo Per", where the formula will be located... 

-The data area of "DataSheet" begins on row 18 and ends on row 962 (the data area could be filled to row 962 by year's end in "DataSheet", but there will almost always be fewer than the maximum number of rows used in the area)...

-The range to be surveyed is the bottommost 135 rows (bottommost rows that contain a column E value) in the data area of "DataSheet".  The range moves down each week as 15 rows of new data are added

-The value to be counted is in column B in the destination worksheet for the formula and in column E in "DataSheet"

-In summation...Count the number of times B in the destination worksheet for the formula is found in E of "DataSheet", in the range of the last 135 rows of data containing an E value in "DataSheet", in the data area range of "DataSheet" between 18:962

Just need a straight count...

Thanks for any assistance

jllatham say:

AtlBo - this looks like something I've seen over at Excelanon and worked with in just the last 24 hours to help with in other areas.  Think adding another VBA routine would help?  Actually I'm thinking of a User Defined Function that you could reference in a cell as a formula, something like

=CountInLast135(B2)

and the UDF itself handles dealing with determining the last 135 rows of the range and getting the count and returning the result.

Here's a UDF that should work - just put it into any regular code module in the workbook:

Function CountInLast135(seekValue As Range) As Integer
'to be used on the [2 Year Mojo Per] sheet in any column except column B
'seekValue is a reference to the cell in column B on that sheet to get a count
'for from the last 135 entries beginning at row 18 on the [DataSheet] sheet, column E
'if there are not at least 135 entries beyond row 18, then just the available entries
'from row 18 on are used.
'
' use on the sheet in a formula as:
' =CountInLast135(B2)
'
  Const srcWSName = "DataSheet"
  Const srcColumn = "E"
  Const minRow = 18
  Const MaxRows = 134 ' this will give us last 135 rows
 
  Dim seekRange As Range
  Dim srcWS As Worksheet
  Dim firstRow As Long
  Dim lastRow As Long
 
  Set srcWS = ThisWorkbook.Worksheets(srcWSName)
  lastRow = srcWS.Range(srcColumn & Rows.Count).End(xlUp).Row
  'handle no data in DataSheet!E18 on down
  If lastRow < minRow Then
    CountInLast135 = 0
    Exit Function
  End If
  firstRow = minRow
  'handle less than 135 rows in use below row 18
  If (lastRow - MaxRows) > (MaxRows) Then
    firstRow = lastRow - MaxRows
  End If

  Set seekRange = srcWS.Range(srcColumn & firstRow & ":" & srcColumn & lastRow)
 
  CountInLast135 = Application.WorksheetFunction.CountIf(seekRange, seekValue)
  Set seekRange = Nothing
  Set srcWS = Nothing
End Function

Mail Merge prints 0 for Canadian Postcode - Microsoft Community

honker2012 say:

Mail Merge prints 0 for Canadian Postcode

I use MailMerge in Word to create e-mail messages. My data source is an Excel worksheet and contains both US and Canadian customers. One column contains Post Codes, which are either 5-digit Zip codes or 6-character Canadian post codes (like LNL NLN) (L=letter, N=number). The column is formatted Special>Zip Code.

When I run MailMerge the <<Zip>> field reports US Zip Codes correctly, but it produces a '0' for the Canadian Post Codes. I want to correct this problem.

I have tried formatting the Zip column in Excel as Text, but I lose the leading zero in some US Zip Codes. I get the same result if I format as Number with zero decimal places.

What is the fix?

(Btw, I am posting this as a Word issue, but it certainly could be an Excel issue.)

Thanks.

Mike

a. user say:

1. The underlying problem is an issue to do with the OLE DB provider that Word uses to get data from Excel. When you have mixed data types in a column, such as "numeric" and "text", the provider decides a data type for the column (e.g. numeric) then tries to convert data of other types to that type. The provider uses the data in the first 8 rows of the column to do that. If even one of the first 8 cells in the column has a text value (e.g. a Canadian postcode) then the provider should treat the column as "text" and convert US ZIP codes entered as numbers to text. However, if all 8 are US Zips entered as numbers, subsequent Canadian codes will be lost.

So one workaround is to insert a "Canadian" row at the top of your sheet and exclude the first record from the merge in Word.

Another workaround is to make a backup of your workbook, then
 - select the postcode column
 - use the Data->Text to Columns option to change the format of the data to "Text" in step 3 of the process. Altering the cell format any other way usually isn't enough because it does not actually change the data in the cell, just the formatted appearance.

2. The problem with the DDE option not showing for .xlsx files is to do with Word. DDE is an old, deprecated method of getting the data, and (despite the fact that it is still often the best choice for many users) I assume that in Word 2007 and later Microsoft either chose not to list the DDE option or forgot to include it for .xlsx/.xlsm file types. So you do have to check "Show All" as you have realised. With some merge problems, it is possible (and sometimes even quite sensible) to work around the problem using Word VBA and its OpenDataSource method, but in the case where the extension is .xlsx, I do not believe that you can get OpenDataSource to open the file using DDE.
