suzanne s. barnhill say:

While Jay's workaround is helpful, I find it more useful to know the keyboard shortcuts for all the quote characters so that you can force the one you want (this is also helpful when you want a word to begin with an apostrophe rather than a single opening quote):

Ctrl+`, ` = opening single quote

Ctrl+', ' = closing single quote

Ctrl+`, " = opening double quote

Ctrl+', " = closing double quote

The way these shortcuts work, you press Ctrl and either the grave accent key (on U.S. keyboards this is the first key on the number row, paired with ~ tilde) or the apostrophe (paired with the " double quote at the right end of the home row on U.S. keyboards) and release. This is the "setup key." You then press ` or ' or " to get the desired type of quotation mark.

scottgem say:
Is the combobox based on a Value list or a table/query? Looks like a value list which is your problem. 

You should be using 2 tables:

tblPart
PartID (Primary Key Autonumber)
PartCode
ProjectID (Foreign Key)

tblProject
ProjectID (PK Autonumber)
Description

Then you RowSource for your Combo would look like this:

SELECT PartID, Description, PartCode
INNER JOIN tblPart.ProjectID = tblProject.ProjectID
FROM tblPart, tblProject;

Then you could filter the report on PartID, to get only the parts for the selected project. Or you could have a RowSource of:

SELECT ProjectID, Description, PartCode
INNER JOIN  tblProject.ProjectID= tblPart.ProjectID
FROM tblProject, tblPart;

And just select a Project, then filter just for the Project. 

There are a few ways to do this, all better than what you are doing. But to be more specific we need to understand the structure of your database.


ken sheridan say:
However, I need my "Program" list like in this way.

But that doesn't mean you need to store the data like this.  In fact, by doing so you are violating one of the fundamental rules of the database relational model, Codd's Rule #2, 'The Guaranteed Access Rule':

'Every datum (atomic value) in a relational database is guaranteed to be logically accessible by resorting to a combination of table name, primary key value and column name.'

You are storing two data, the program and the part in one column, so neither is accessible in the way the rule describes, which is essentially the problem with which you are faced.  By decomposing the table as described by Scott the rule is not violated and your problem is solved.

As regards the need to show the values together you simply join the tables in a query and concatenate the values of the two columns, with an intervening hyphen character, as a single string expression in a computed column.
