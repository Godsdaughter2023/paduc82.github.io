marshall barton say:

I think you are mixing your sources  ;-)


A form or report has a RECORD source that is either a table name, a query name or an SQL statement (which is a SELECT query).


Controls (text box, etc.) on a form or report have a CONTROL source that can be the name of a column (field) in the form/report's record source or an expression.  You can type the name or expression directly into the control source property.


Just for completeness, combo and list box controls also have a ROW source that can be the name of a table, the name of a query or an SQL statement.


Your form based on a query with multiple tables in its FROM clause is very likely to be read only and not useful for entering or editing data into the tables, which is fine in most reports.


For normal operations a form, you should have a separate form for each table.  These forms can be used as subforms within other forms to display and edit data in related tables.  I think this may be the crux of your problem and the first thing you need to do.  Set the competitive books form's record source to a query (with sorting) based only on the CompetitiveBooks table.  Create a purchasing form with its record source to a query based on the OnlinePurchasing table.  Then Drag and Drop the purchasing form from the objects list onto the competitive books form and set the subform control's LinkMaster/Child properties to the BookID fields.


If you have just started to use Access and only beginning to learn how to use a wizard to create things, be warned that the wizards only go a little way to get things started.  Your question kind of implies that you have already reached a limit in what they can do to make things easy.  When you start to get confused or can't figure out how to do something in a wizard, get out of the wizard and work in the form's design view where you can specify and manipulate everything.

scottgem say:
As Marshall has said, you need to understand your terms better. A Form has a Recordsource, not a Controlsource. Controls have Controlsources. 

A form's Recordsource can be either a table, query or a SQL SELECT statement. The help article actually was helpful to people with more knowledge than you. If you go to the property dialog, select Form in the combo at the top, then click the ellipses next to the Recordsource property, it will open Query Design mode. There you can use the query grid to construct a SQL statement that pulls in all the fields you want on your form. As Marshall also noted, such a Recordsource will most likely render the form uneditable. That's fine if you just want to use the form for viewing data. But a better way to deal with multiple tables is via the use of subforms. Create a main form bound to one table and then embed subforms linked on your key fields. This is especially important if you have one to many relationships between your main table and the others. The subforms can display multiple matching records.

jim gordon mvp say:

The code you posted assumes that you want the chart on the active sheet.

All you need to do is to tell Excel the name of the sheet you want near the beginning of your code. For example, right beofre you select the range you could put:

Sheets("Sheet2").Select

replace "Sheet2" with the name of your worksheet

jim gordon mvp say:

The error message you received is well documented on the web. From MrExcel.com

"The problem occurs whenever the default chart type is set to other chart types than the ones you are trying to create. For an example, if you try to create line chart and if the default chart in Excel is OHLC (Candlestick Stock Chart) then Excel will be fast to complain "The specified dimension is not valid for the current chart type". The same occurs even if you try to create charts from .NET. Hence, first of all change the default chart type to some basic chart types like Line charts. The problem will be solved."
