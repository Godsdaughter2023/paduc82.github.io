Word - Mail Merge Formatting - Microsoft Community

barb_mac6 say:

Word - Mail Merge Formatting

In Mail Merge, how do I change the format of the date from 1/1/12 to January 1, 2012?

a. user say:

On Mac, the suggestion Bob made is probably the easiest way to do this if the field is a { MERGEFIELD } and comes from an Excel sheet.

But if that doen't work, another thing you can do is to modify the MERGEFIELD field that inserts the date.

To do that, you have to work out how to "reveal field codes" - could be Option-F9 or fn-Option-F9.

Then look for your date field in your Mail Merge Main Document - e.g., it might be

{ MERGEFIELD mydate }

or you might even be talking about a { DATE } field

For the MERGEFIELD, modify it so it looks like this:

{ MERGEFIELD mydate \@"MMMM d, YYYY" }

Then Option-F9 again, ideally select the field and update it (e.g. via F9 or fn-F9), then merge.

If it's a DATE field you could either change it to 

{ DATE \@"MMMM d, YYYY" }

or delete the DATE field and insert the date manually.

An example of one of the array formulas that I use is:
{=SUM(IF('C-By Organisation by Test'!$B$1:$B$60000=O3,IF('C-By Organisation by Test'!$D$1:$D$60000=R3,IF('C-By Organisation by Test'!$AB$1:$AB$60000=S3,'C-By Organisation by Test'!$F$1:$F$60000,0),0)))}
[....]
Maybe there is a different way of doing these formulas....but i have always just used nested ifs.

You might have done that in XL2003.  But even in XL2003, you can avoid the array-entered formula by using SUMPRODUCT.  For example:

=SUMPRODUCT(('C-By Organisation by Test'!$B$1:$B$60000=O3)
*('C-By Organisation by Test'!$D$1:$D$60000=R3)
*('C-By Organisation by Test'!$AB$1:$AB$60000=S3),
'C-By Organisation by Test'!$F$1:$F$60000)

It is unclear whether that is more, less or equally efficient.  But since array-entered formulas are error-prone (you might press just Enter and get some GIGO result), I try to avoid them.

However, if XL2003 compatibility is not an issue, you can use SUMIFS in XL2007 and later.  For example:

=SUMIFS('C-By Organisation by Test'!$F$1:$F$60000,
'C-By Organisation by Test'!$B$1:$B$60000, O3,
'C-By Organisation by Test'!$D$1:$D$60000, R3,
'C-By Organisation by Test'!$AB$1:$AB$60000, S3)

That might be more efficient, especially if you have less than 60,000 rows of numeric data in column F.

Of course, reducing the rows 1:60000 as much as reasonably possible would also help.

greginmich say:

These are definition updates, and the command line parameters appear in the Windows > Temp > MpCmdRun.log file. So far, I’ve only tried scheduling updates with the PowerShell Set-MpPreference cmdlet, which takes the place of Group Policy in the Home edition of Windows 10, but which works interactively with Group Policy in the managed editions. So I’m not sure that the automatically generated update checks can actually be turned off with the Set-MpPreference cmdlet, although the documentation seems to hint at that possibility.

https://docs.microsoft.com/en-us/powershell/module/defender/set-mppreference?view=win10-ps

https://docs.microsoft.com/en-us/windows/security/threat-protection/windows-defender-antivirus/manage-protection-update-schedule-windows-defender-antivirus

To set up my scheduled update, I ran these commands at the Administrator PowerShell prompt:

Set-MpPreference -SignatureScheduleDay 0  

Set-MpPreference -SignatureScheduleTime 10:20  (The interpreter doesn’t seem to accept minutes after midnight)

RandomizeScheduleTaskTimes $False 

This adds a Windows Defender Update task to the Windows Defender folder in Task Scheduler – but this task appears to run with the usual Automatic Maintenance only-on-idle priority, and it isn’t starting the update at the scheduled time. Adding a trigger to the task might help, but I’m afraid that you might have to depend on your independent user-scheduled update task to keep the updates running on time. And then the next step would be to try managing the automatically generated update checks with the -SignatureFirstAuGracePeriod and -SignatureUpdateCatchupInterval parameters.

-SignatureFirstAuGracePeriod

Specifies a grace period, in minutes, for the definition. If a definition successfully updates within this period, Windows Defender abandons any service initiated updates. This parameter overrides the value of the CheckForSignaturesBeforeRunningScan parameter.

-SignatureUpdateCatchupInterval

Specifies the number of days after which Windows Defender requires a catch-up definition update. If you do not specify a value for this parameter, Windows Defender requires a catch-up definition update after the default value of one day.

For example, you might try setting both of these intervals to 2 days:

Set-MpPreference -SignatureUpdateCatchupInterval 2

Set-MpPreference -SignatureFirstAuGracePeriod 2880

This is probably a long shot – because the documentation is still pretty fuzzy here, and also because some of these parameters just don’t seem to work the way they’re supposed to.
