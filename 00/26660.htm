doug robbins - mvp office apps & services (word) say:

What you need to do is insert a bookmark at the end of each section - I usually use Section# for each bookmark where # is the Section number (Section1, Section2, Section3, etc

Then, set the page numbering to be continuous across each section and in the header of the second page, unlink it from that in the first Section and insert the following field construction:

{ = { page } - { PAGEREF Section1 }} of { Sectionpages }

Then move to the header of the third section and unlink it from that in the second section and modify the field construction to:

{ = { page } - { PAGEREF Section2 }} of { Sectionpages }

Then go to the header in the fourth section and unlink it from that in the third section and modify the field construction to:

{ = { page } - { PAGEREF Section3 }} of { Sectionpages }

and so on.

Note, you must use CTRL+F9 to insert each pair of field delimiters and you use ALT+F9 to toggle off their display

This a copy of answer Bob Jones gave to another question about this sub to another OP (original poster).  There may be  Sentences That apply to that post that may not this. But For Most part the information

"Word for Windows supports controls [fields] that are not supported in the Mac version. There is nothing you can do to change that. In order for forms to work in both Mac & PC the only type of controls used must be what are referred to as Legacy Form Fields, so the document would have to be redesigned with cross-platform compatibility in mind. Apparently the document uses either Content Controls or ActiveX Controls.

Also, the Windows versions of Word support document protection features that the Mac versions lack. If those features have been used Mac Word will simply see the document as "protected" & not allow any changes whatsoever.

Additionally, not all macros which run on a Windows PC will run on a Mac & vice-versa. It depends on the commands written into the macro & what it is intended to do."

Courtesy of Bob Jones (Cyber Taz)"

zack barresse say:

Hi there,

Let's clarify a few things. First, you do not run Office 365 in the cloud. It is installed locally and runs locally. Your files may originate there, but even those open locally, then re-save back to the cloud. There is no such thing [atm] of running Excel in the cloud on a local machine. The closest you could get to that would be to have a browser open on your local machine which has Excel Online open on it, but you can't run VBA there.

Office 2007 will no longer update on your machine if you uninstalled it correctly. Any installed updates would have been removed with the application, unless it had to do with other applications, i.e. a Windows update, not an Office update.

When you say "Office 365", that doesn't tell us what version you are running. That tells us that you are paying for a subscription to Microsoft. There are many flavors of Office 365, ranging in prices. Those subscriptions come with software. For Excel, this is generally the latest version, although you can downgrade one version to 2013. It also installs differently than previous Office versions (known as MSI's), but the program is the same. The reason for the different install paths/properties is for updating. O365 can get updates pushed from the mothership (i.e. Microsoft) at a faster cadence - sometimes monthly even. I'm assuming you're running Excel 2016 for desktop.

VBA for Excel 2016 for desktop is indeed slightly different from VBA for Excel 2007 for desktop. But those differences are minute. First, Excel 2016 has VBA7, whereas Excel 2007 has VBA6. There is a 64-bit and 32-bit Excel 2016, but only a 32-bit Excel 2007. And the object model (OM) has had updates with the new features.

All this to say, it is rare to find code which worked in 2007 but does not work in 2016. The best thing you can do is post your code and tell us what the expected outcome is.

bob jones aka: cybertaz say:

Word stores your macros in the Normal.dotm template by default*. However, what you now have in the User Templates folder is a Normal.dotm template but not the one you had prior to the "CSR" tampering with your installation.

(*Depending on what is specified when a macros is created, macros can be stored in other templates or in the document, itself.)

The good news is that you apparently still have the files that were moved to the Trash... Your former Normal.dotm should be in the same relative location of the folder hierarchy that was deleted. Locate it & move it to the Desktop.

Once you've retrieved the file, in Word use the Organizer to copy the macros from the old template to the new one.

Just a suggestion :-) - If you rely heavily on macros and/or other customizations it's a good idea to save a backup copy of your Normal.dotm in a safe place.
