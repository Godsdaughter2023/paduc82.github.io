john mcghie say:

VBA supports ANSI, not ASCII.  Is that what you mean by "extended ASCII"?

Word stores all characters as doubler-byte Unicode, so you should be using the "Wide" functions for that (ChrW and AscW).

VBA itself does not support the extended character set internally: if you are passing strings back and forth, stick to the first 256 characters.

See http://support.microsoft.com/kb/145745

ken sheridan say: I share Duane's reservations about your table design.  Having multiple Boolean (Yes/No) columns in a table usually means that data is being encoded as column headings.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

dhookom say:

I'm not sure what Ken's download looks like but I expect it's most appropriate. I also don't have much of a clue regarding the purpose of your tables.

Let's pretend your records store what types of animals are in various zoos. Your current table might have fields like:

ZooID

A normalized structure would have a table for zoos, animal types, and a junction table of zoo animals:

tblAnimals

tblZooAnimals (Junction table)

You can use a crosstab query to display the "spreadsheet" type view. You can also add more animal types without changing the table structures, forms, reports, queries, etc.

ken sheridan say:
So, I should change the structure to include 5 or 6 fields where people can choose which of these topics are relevant?  Then create a make table query on those fields?

No.  What you appear to have is a many-to-many relationship type between whatever entity type your current table is modelling and a Topics entity type.  There is only one way in which such a relationship type should be modelled, which is by means of a third table which resoles the many-to-many relationship type into two one-to-many relationship types.  My demo, to which I referred you, models a relationship type between Stores and Stocks, i.e. the relationship represents the items stocked per store.  It is modelled by three tables like this:

 Stores----<StoreStock>----Stocks

in which the StoreStock table has two foreign key columns, StoreID and StockID referencing the primary keys of the other two tables respectively.

To return records on the basis of multiple values in one of the related entity types, in your case multiple topics a multi-select list box can be used to restrict the result table of a query.  You'll find an example as MultiSelect.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

In this little demo file the bottom pair of buttons on the opening form each opens a form with a multi-select list box of employees.  In one form selecting one or more employees will generate a list of projects to which any of the selected employees is assigned; in the other form it will generate a list of projects to which all of the selected employees are assigned.  Think of employees as analogous to your topics, and projects as analogous to whatever entity type your current table is modelling.

As regards creating a new table, this is rarely necessary.  A table does not need to be a base table, but can be the result table of a query, which could be used to export the data to Excel or whatever format is required.
