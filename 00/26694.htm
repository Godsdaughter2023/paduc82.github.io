The answer to your question is DON'T DO IT. The only way to do this is to use an unbound form and then you have to change the form design every time you change the list. Or you use a denormalized table and then have to change the table and form every time the list is changed.

ken sheridan say: As rider to Scott's last post (and I endorse all he's said) you can always recompute the index numbers so that they increment by 10, allowing however many more intervening rows you need to be inserted.  The following is some code which does exactly this in a form bound to a table of regular payments from an account:

    Dim rst As DAO.Recordset
    Dim strSQL As String
    Dim strRS As String
    Dim n As Integer
    
    Me.Dirty = False
    
    ' get current RecordSource property
    strRS = Me.RecordSource
    
    ' set form's RecordSource to a zero-length string
    ' to allow change to table definition
    Me.RecordSource = ""
    
    ' drop index
    strSQL = "DROP INDEX PaymentIndex ON RegularPayments"
    CurrentDb.Execute strSQL, dbFailOnError
    
    strSQL = "SELECT PaymentIndex FROM RegularPayments ORDER BY PaymentIndex DESC"
    Set rst = CurrentDb.OpenRecordset(strSQL)
    With rst
        .MoveLast
        n = .RecordCount
        .MoveFirst
        Do While Not .EOF
            .Edit
            .Fields("PaymentIndex") = n * 10
            .Update
            n = n - 1
            .MoveNext
        Loop
    End With
    Set rst = Nothing
    
    ' reinstate index
    strSQL = "CREATE UNIQUE INDEX PaymentIndex ON RegularPayments(PaymentIndex)"
    CurrentDb.Execute strSQL, dbFailOnError

    ' reinstate form's RecordSource
    Me.RecordSource = strRS
                    
    Me.Requery

As an alternative to a subform you can use a multi-select list box in a single form, but you would need to write code to select the relevant rows in this when you navigate to each existing record, and to write the data to the referencing table when you select items relating to a new record or edit items relating to an existing record.  The code would also need to delete rows from the table when items are deselected in the list box of course.  This would give you the visual comparison you are looking for, as the list would show both selected and unselected items, analogous to checked or unchecked check boxes, and the list would be dynamic, but would require more work to implement than the entirely code free solution of a continuous forms view subform.

Based on my research, there’s no official documentation about cross-premises eDiscovery search in Exchange 2010-based hybrid deployment.

suzanne s. barnhill say:

Double-click in the header area to open the header pane. Type your short title in all caps, then tab twice to get to the right margin and press Alt+Shift+P to insert a page number field.

Note: If your title is long enough to extend past the center-aligned tab stop in the center of the paragraph (which it likely will if it's 50 characters), you'll need to tab only once to get to the right margin.

Include a page header (also known as the "running head") at the top of every page. To create a page header/running head, insert page numbers flush right. Then type "TITLE OF YOUR PAPER" in the header flush left using all capital letters. The running head is a shortened version of your paper's title and cannot exceed 50 characters including spacing and punctuation.
