john mcghie say:
There are two sets of dates involved: one stored in the document and maintained by Word, the other stored in the file meta data and maintained by the operating system.

You can preserve the operating system's dates only by using Terminal to copy the file from one OS X file system to another. 

In your email, the "file" is a copy made for the purposes of attaching it.  It's not the same instance as the original file, and it gets new dates.  Those will be the date current when the email file was created for attachment to the email.  When you save from your email to the file system, again you are not saving the "same" file, you are creating a new file on your computer, and it will have the current dates on your system.

If you open the document and go to the File>Properties command in Word you will see the Operating System dates on the General tab, and the internal file dates on the Statistics tab.  The dates there usually make little sense.  I am looking at a file that was apparently "printed" 12 hours before it was "Created" and was "Modified" when I opened it (even though I did not either modify or save it).

No:  Nothing has "changed" in Word.  However, there have been some changes in the way Apple OS X interprets file dates, and there may have been a change in which dates Word selects to read.  On the PC, the same mechanism operates, but will show different dates because the PC file system stores different dates.

The dates on the Statistics tab are written into the file by Word, and will be the same in PC or Mac.  On the Statistics tab, the "Created" date is the date that this VERSION of the file was created by Word.  The "Printed" date may refer to an earlier date, the latest date on which any version was printed.  The "Modified" date will always be the current date, which will be written to the file when you save it, or remain unchanged if you don't.

To directly answer your questions:  "No", nothing has intentionally changed, and "No" there is no way to alter how these dates appear.

What you can do is use the Help>Send Feedback menu to remind Microsoft that their current date mechanism is neither helpful nor reliable, but that it is an important and useful function that people are attempting to rely on, and should be fixed.

This issue occurs when EXO fails to provision the ExchangeGUID attribute for an user(msoluser will have a validation error), because this ExchangeGUID value is already used by another object from EXO, it could be an active mailbox/mailuser or a soft deleted mailbox/mailuser.

Search in EXO PowerShell for the object that is using the mentioned EXchangeGUID or ArchiveGUID: 
Get-Recipient -IncludeSoftDeletedRecipients 'ExchangeGUID value'|ft RecipientType,PrimarySmtpAddress,*WhenSoftDeleted*

Get-MsolUser -UserPrincipalName user@domain.com |fl *objectID*
Redo-MsolProvisionUser -ObjectId 'paste the *objectID* value from above command'

(Get-MsolUser -UserPrincipalName user@domain.com).errors.errordetail.objecterrors.errorrecord| fl

lance cai msft say:

Hi Shan,

It is not suggested that you delete the batch based on the status. After you changed the MX record,  it’s time to verify that all your email is being routed to Office 365, and then you can go ahead and delete the migration batch. Doing this stops the synchronization between your source email system and Office 365. Before you do, make sure of a few things:

  • Your users are using Office 365 exclusively for email. After you delete the migration batch, email that is sent to mailboxes on your source email system isn’t copied to Office 365. This means your users can’t get that email, so make sure that users are all on the new system.

  • Let the migration batch run for at least 72 hours before you delete it. This makes the following two things much more likely:

    • Your source email system and Office 365 mailboxes were synchronized at least once (they synchronize once a day).

    • The email systems of your customers and partners have recognized the changes to your MX records and are now properly sending email to your Office 365 mailboxes.

When you delete the migration batch, the migration service cleans up any records related to the migration batch and removes it from the migration dashboard.

ken sheridan say:
What is exact meaning of normalizing data?  Thanks!

Normalization is the process of the elimination, as far as possible, of redundancies by the decomposition of a table into one or more related tables.  Tables can be in various normal forms, from First Normal Form (1NF) upwards.  A table is in 1NF if in each row it contains one value of each attribute only.  A table in 1NF is said to be normalized; tables in the higher normal forms of 2NF, 3NF etc are said to be normalized to the normal form in question.  For an outline of the subject see:

 

http://en.wikipedia.org/wiki/Database_normalization

 

One misinterpretation of the definition of First Normal Form which is sometimes made is that it assumes it refers to a ban on multiple values in a row at a single row position, e.g.

 

Store        FruitStocked

 

Store1      Apples, Pears, Oranges

Store2      Oranges, Bananas

 

rather than:

 

Store         FruitStocked1              FruitStocked2               FruitStocked3

 

Store1      Apples                         Pears                            Oranges

Store2      Oranges                       Bananas

 

Neither table is in 1NF in fact as each contain multiple values of the attribute FruitStocked in each row.  The storing of multiple values at the same column position in a single row is sometimes said to be illegal on the basis that each column position's value must be 'atomic' (indivisible).  This is true, but it stems not from the definition of 1NF per se, but from another fundamental rule of relational database design, the Guaranteed Access Rule (Codd's Rule #2), which states:

 

Every datum (atomic value) in a relational database is guaranteed to be logically accessible by resorting to a combination of table name, primary key value and column name.

 

Clearly this is not the case with the first example above as a reference to the FruitStocked column for the row with a value Store1 in the Store column (the primary key) does not tell us whether it is Apples, Pears or  Oranges which are being referenced.


Note that a value in a column might not be 'atomic' but the two or more values at the column position could be of different attributes, so the definition of 1NF is not breached, but that of the Guaranteed Access Rule is.

 

Another point which is sometimes made incorrectly is that it is sometimes said that it is acceptable to normalize a table to 3NF only.  Third Normal Form requires that all non-key attributes must be determined solely by the whole of a table's primary key: 'the key, the whole key and nothing but the key, so help me Codd'.  This arises from a misinterpretation of the fact that a table normalized to 3NF will, in the majority of cases, automatically be normalized to the higher normal forms.  If so then there is no need to attempt to take normalization beyond 3NF.  However, if a table in 3NF is not normalized to the higher normal forms, then it should be normalized to those  levels.  If not then the database could still be open to the insertion of bad data.

 

While the normal forms are closely defined in formal language, by and large the process is one of applied common sense once a good grasp of the underlying principles is obtained.

 

A non-normalized table, which will usually be the case with data imported from Excel, can be decomposed into a set of fully normalized tables by firstly designing and relating the tables, and then executing a set of 'append' queries in a specific order.  In essence the procedure is to append rows to a referenced table (a table on the 'child' side of a one-to-many or, rarely, a one-to-one relationship type) before appending rows to a referencing table ( a table on the 'parent' side of a relationship type).

 

You'll find an illustration of the process as DecomposerDemo.zip in my public databases folder at:

 

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

 

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

 

This little demo file imports data from a simple Excel worksheet into a temporary 'master' table, and, step by step, appends it to the relevant tables in the set of correctly normalized operational tables.
