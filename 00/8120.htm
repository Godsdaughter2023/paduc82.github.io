"Not responding" is a symptom, not single specific defect. It is actually a symptom of many defects. The problem can be a large / complex (inefficient) spreadsheet calculation that simply takes forever to process.  Or it can be that the application is "looking for something" it can't find, so it waits a predefined time before going on it's merry way.  What that "something" is could be bad code, or it could be something located on the internet that the application is having a problem finding. Gib:

No, repair should not make a difference if a install did not help.

Unfortunately the tech that did the uninstall and install was an ID-10-T.  There are 2 levels of "uninstall" for Office. The first is from the control panel > programs and features.  This type of install leaves various configuration files, like normal.dotm for Word, and registry entries in place. The new installation picks up this left over information to make your new installation look and feel exactly like the old one. it is this type of uninstall that is used when doing an upgrade to a new version, ie from 2016 to 2019.  The original idea for this type of uninstall was for simple trouble shooting and upgrades to make them transparent to the user.

The other type of uninstall uses a "fix it" tool from MS to remove all traces of the Office installation. This type of uninstall is intended as the second step in trouble shooting, just in case the problem is being caused by the configuration files / registry entries. 

The MS tech should not have gone directly to the full uninstall without warning you. That is just rude and lazy.

Taking a step back, have you tried starting the Office applications in "safe mode"?  Safe Mode disables addins. Addins have a long history / "tradition" of causing freaky, unexpected interactions (ie CRASHES) with Office applications.  Actually, the same applies to a lesser degree with Windows addins, sometimes Windows Safe Mode helps.

Office Safe Mode

    vladw say: Here is actually the source of the confusion I had with VBA, in a distilled form:

    Sub ClearValue(ByRef v As Variant)
        v = Empty
    End Sub

    Sub TryIt()
    v = "full"
    ClearValue v             ''This works right
    Debug.Print IIf(IsEmpty(v), "empty", v)
    v = "full"
    Call ClearValue(v)    ''This works right
    Debug.Print IIf(IsEmpty(v), "empty", v)
    v = "full"
    ClearValue (v)           ''This does not pass out internal changes!
    Debug.Print IIf(IsEmpty(v), "empty", v)
    End Sub

    The debug output is:
    empty
    empty
    full

    -------------------------------------------------------
    In short:  
    VB allows to call subroutines having one argument without Call in front of its name and with parentheses around the argument.
    Such call works without an error, the called subroutine performs all its activity, the only problem is that it does not return out any changes made inside to its argument, even when one can naturally expect this for an argument sent ByRef.

    So, never call a Sub having just one argument ByRef, from which you expect a return, this way: SUBNAME (ARG), call it either like: Call SUBNAME (ARG), or just SUBNAME ARG.

    In typical Microsoft misstatement, what I think they really mean is:  if we omit the Call keyword, any parentheses are not interpreted as surrounding an argumentlist.

    That means the parentheses are interpreted as surrounding an expression, which of course results in a value, not a variable reference.

    Hi Md.ShamimReza,

    Let me calrify the POP setting.

    POP works by contacting your email service and downloading all of your new messages from it. Once they are downloaded onto your PC or Mac, they are deleted from the email service, which means that there is no emails from web. This means that after the email is downloaded, it can only be accessed using the same computer/device. If you try to access your email from a different device(like mobile), the messages that have been previously downloaded won't be available to you.

    However, we have a choice to download and don't remove messages from the POP3 mail server in Outlook, you can take a look as below:

    https://support.office.com/en-us/article/Read-POP3-email-messages-on-multiple-computers-cf375a53-a3df-47c8-b3b6-17f6139bd9d1

    You are right. For normal user mailboxes with Kiosk plan, messages will be remain in the mailbox. If you don't delete them from inbox, the maximum size is 2GB. For more detailed information about the limitation, you can refer to the link below:

    https://technet.microsoft.com/en-us/library/exchange-online-limits.aspx?f=255&MSPPError=-2147217396
