If you open a Word 97-2003 .doc file in Word 2010 and just resave it normally (using Save rather than Save As), it will be saved in its existing format. scottgem say:

Again, the Runtime has all the same functionality of a full Access license except for the ability to enter design mode. Once installed, the user runs the actual mdb/accdb file so its basically transparent to the user that they are using the runtime.  So the user is not really "using" the runtime, so there is no issue of ease of use. Any ease of use issues are inherent in the app itself. 

If there is only going to be one user who maintains the data, and the need is to just copy that file to local machines, then this will work. However, there is one issue here. An Access app needs to be in a "Trusted Location". This means that a folder needs to be set up on the users PCs as a Trusted Location. I would not put this file on a user's desktop but in a folder on their local drive and place a shortcut to the file on their desktops. So, some initial setup is going to be necessary to create the folder, make it a Trusted, copy the file and create the shortcut. From there it will only be necessary to copy the latest version into that folder.

However, the question is how often is the data updated? If the data is being updated frequently, then this will become a chore to frequently refresh the local file. It would be much easier to split the app as described and just load the front ends on the local machine. This way those users will always see the latest data without doing anything. The forms in the front end just need to locked to prevent editing which is easy to do.

Here are a few causes of #NAME? error

  • A formula refers to a name (name: A word or string of characters in Excel that represents a cell, range of cells, formula, or constant value.) that does not exist.
  • A formula refers to a name that is not spelled correctly.
  • The name of a function that is used in a formula is not spelled correctly.
  • You may have entered text in a formula without enclosing it in double quotation marks.
  • A colon (:) was omitted in a range reference.
  • A reference to another sheet is not enclosed in single quotation marks (').
  • A workbook that you open calls a user-defined function (UDF) that is not available on your computer.

Perhaps you could post the formula which returns the error.

gord dibben say:

As has been pointed out, Excel's dates are Serial Numbers.

Excel stores dates as sequential serial numbers so that they can be used in calculations. By default, January 1, 1900 is serial number 1, and January 1, 2008 is serial number 39448 because it is 39,447 days after January 1, 1900. Most functions automatically convert date values to serial numbers.

Just format that serial number to your choice of Date format.

43273  is 6/22/2018 or you can format as  June 22, 2018
