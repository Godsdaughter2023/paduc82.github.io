I experienced this issue as well and here is how it got in that condition and the simple solution I found to correct it; I am using Exchange 2007 and Outlook 2010 (on Win7).

How the contacts appear as unread?

In my case, our company has a public contacts folder and when one of the contacts is copied from there to a user's Outlook Contacts folder it populates with an un-read status; unfortunately the actual "un-read" contact does not appear bold.  Not to get into why, but my simple reason for copying the contact is that my Blackberry syncs to my contacts.

Solution (simple one that works for me):

1) Highlight Contacts on left pane.

2) From Home tab, select "By Category"

3) From the View tab, select "Add Columns"

4) From Show Columns box, choose "All Contact fields" from the "Select available columns from:" drop down list.

5) In the newly refreshed "Available columns:" list, select "Read" and click on the Add button.

6) Click "OK"

7) On your Contacts listing, there should now be a new column, on the right, with a closed envelope icon as the column heading.  Opening any contact that has a closed envelope icon (un-read state) should change the icon to an opened envelope (read state).  Evidently this column does not support sorting.

Disclaimer:  As with so many other applications, there may be other ways to accomplish this.  That this approach works for me, is no guarantee that it will work under other conditions, scenarios, etc.  I got curious about this situation and worked off someone else's comment about fields in order to figure this out.  Hope this is helpful for a majority of the cases.

tom-s say:

Cindy,

Because AVERAGEIF can't be used in a 3D reference, one alternative would be to pull the data from each sheet's F37 cell into a table on your data analysis sheet, and then do the average from the table.

Continuing with the August 2012 example, I'm assuming there's a sheet for each weekday (Mon-Fri). So you could form a table as follows, formatting column B as text, and typing a space before each 08:

 

      A                       B            C

1   DAY                  DATE   WORKDAY

2   Wednesday     08-01    1

3   Thursday          08-02    2

4   Friday                08-03    3

5   Monday             08-06    4

6   Tuesday           08-07    5

7   Wednesday     08-08    6

:           :                      :         :

23 Friday               08-31   23

 

In column D you can pull in the F37 values with this formula e.g. in D2

=INDIRECT("'" & A2 & B2 & "'!F37")

 

And then the average formula would be

=AVERAGEIF(C2:C23,"<=" & H2,D2:D23)

 

where H2 holds, as you've indicated, the current workday number (assuming you have a formula for deriving this).

 

Hope that's something close to what you need.

jeeped say:

... Is there a way to change my view options so a selected cell is always filled in colour, or bordered in a colour other than black?


The following will highlight a selection of a cell or range of cells while emphasizing their row and column.

Right-click the name tab of the worksheet and select View Code. When the VBE opens, paste the following into the pane titled something like Book1 - Sheet1 (Code), 

Public OldRng As Range
Private Sub Worksheet_SelectionChange(ByVal Target As Range)
    If Not OldRng Is Nothing Then
        OldRng.EntireRow.Interior.ColorIndex = xlNone
        OldRng.EntireColumn.Interior.ColorIndex = xlNone
        OldRng.Interior.ColorIndex = xlNone
    End If
    Target.EntireRow.Interior.ColorIndex = 19
    Target.EntireColumn.Interior.ColorIndex = 19
    Target.Interior.ColorIndex = 6
    Set OldRng = Target
End Sub

I've used ColorIndex as the change mechanism so you can refer to the image I've supplied below to change the colors being assigned. Press Alt+Q when you are done to return to your worksheet. You will have to save your workbook as a macro-enabled workbook.

ken sheridan say: Firstly you need to return multiple instances of each row in the report's underlying query, for which the standard method is to create a CounterTable with numeric column Counter, with values from 1 to an arbitrarily high number at least as great as the number instances of each row to be returned.  You can then determine the number to be returned either by hard coding the value in the query's WHERE clause if the number is fixed, or by means of a parameter if its variable.  The following is an example of such a query, in this case of Contacts:

SELECT Contacts.*, CounterTable.Counter
FROM Contacts, CounterTable
WHERE Counter <= [Number of copies:];

For this example the report will be grouped by LastName, with multiple instances of the LastName footer printed:

1.  Add a LastName group.  You can give this a header and/or footer if you wish, but these will print once only.  Note that if you give this group a footer it will print below the multiple footers.

2.  Add a Counter group, with a footer section.  Include the footer contents you want to print multiple times in this section.

3.  Include a control bound to the Counter field in the detail section, and set its Visible property to False (No).

4.  In the detail section's Format event procedure put:

Cancel = (Counter > 1)

With this example the user is prompted to enter 'Number of copies:' at runtime.  When the report opens one instance of each contact record is printed, grouped by LastName.  Below each LasName group the footer is printed multiple times determined by whatever number the user entered at the parameter prompt.

The above is clearly not a very realistic scenario, but that's not relevant.  The result will be the same however a report is grouped if it is designed in the above way.

Ken Sheridan, Stafford, England

marshall barton say:

It would be better to use a separate report for this purpose.

A good way to to get multiple copies of a report is to create a utility table (named Numbers) with one Interger field (named Num) populated with values 1, 2, 3, ... up to 99 or at least more than the greatest number of copies you will ever need to print.  Then the report's record source can be a query like:
   SELECT table.*, Num FROM table, Numbers WHERE Num <= {the numbier of copies} And {criteria for report}

If the report is just for the letter, the letter can be in the detail section (with ForceNewPage set to Before Section.

If you feel that you must use your existing report, then you will need a top level group on the Num field to keep all the other sections of the report propery prganized.  Ynu will also need some way to tell the report that you want the whole report or just the letter along with some code in the report's Open event to make all the sections, except the letter section, invisible.

I think the Wikipedia-article on Deprecation answers this one pretty well:

In the process of authoring computer software, its standards or documentation, deprecation is a status applied to software features to indicate that they should be avoided, typically because they have been superseded. Although deprecated features remain in the software, their use may raise warning messages recommending alternative practices, and deprecation may indicate that the feature will be removed in the future.
