atimatik say:

@Ar Rom, could you be more specific with your answer? Could you please paste here the method - if there's any - how to make Ctrl+C/Ctrl+V work in OWA without duplicating the hyperlink?

It seems that this bug has been there for years, this is a detailed description of the cause of the problem, found in the forum linked below the quote:

Comment 32 by ************, Oct 1 2015 
The issue appears to be due to a bug in the OWA javascript, although after a quick scan through the bisect I'm not quite sure why it worked before.  Perhaps someone more familiar with chromium can explain.
Not sure how advisable quoting blocks of OWA js code would be, so I'll give as much of an overview as I can and link to an exemplifying jsfiddle.
The OWA message typing region is an embedded iframe with document.designMode = "on".  The javascript files are loaded in the outer window (which is relevant for this issue)..
Within the typing region iframe, there is a listener attached to the keyDown event, which checks the previous word if the event key is spacebar.  If the previous word is identified as a link, it is selected by means of a Range and then document.execCommand("inserthtml",...,...) is used to replace the selected text with a link-ified version.  In Chrome, the text is not selected and so the link is instead written at the cursor position, replacing the space keypress, which has the effect of 'doubling' the link text as reported above.
The key lines of OWA js:
   link_word_function() {  //called by keyDown event handler
      ...
      var owr = createObjectWithRange //Calls a function that returns an OWA object with a Range value 'objectWithRange'
      ...
      owr.setRangeStart(textContainer,textStartIndex); //textContainer is the textNode in the typing region iframe containing the word to link
      owr.setRangeEnd(textContainer,textEndIndex);
      owr.select();  //Calls select method of objectWithRange, see below
      document.execCommand("inserthtml",false,linkifiedText);
      ...
   }
   ...
   objectWithRange.prototype.select() { //Calls select method of objectWithRange, see below
      ...
      var selection = window.getSelection();  //This returns a Selection, but in the context of the parent window, not the iframe contentWindow
      ...
      selection.removeAllRanges();
      selection.addRange(this.range);  //This should add the Range to the Selection, thus selecting the text to be overwritten, but fails
      ...
   }
While the code is running as an event fired from within the iframe, it's namespaced to the outer window so window.getSelection() returns a Selection from the outer window rather than a Selection from the iframe.  The Selection from the outer window will not accept an addRange giving an iframe window range as the argument, so the select fails and the link gets doubled.
The OWA bug actually would affect Firefox too, however the Range in Firefox is explicitly (i.e., if UserAgent is Firefox) initialised in a different way:
document.defaultView.getSelection().getRangeAt(0);
rather than 
document.createRange();
The effect of this (I think) is that the Range is already associated with a Selection when owr.SetRangeStart and owr.setRangeEnd are called, and so the text gets selected anyway owr.select() is superfluous (even though it's still broken).
Hopefully this illustrates the issue and why it works in Firefox: http://jsfiddle.net/w86zkgyj/
In terms of a fix, one solution for any admins out there might be to patch the js served by OWA, if that's possible.  You'd need to find the function SelectionRange.prototype.select in the served file uglobal.js and change the line 
var a = window.getSelection();
to
var a = this.document.defaultView.getSelection();

SOURCE: https://bugs.chromium.org/p/chromium/issues/detail?id=410989
