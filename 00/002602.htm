melodyemswiler say:

VLookup

Is it possible to create a vlookup when the lookup value is a cell populated by another formula?  If so, how would the vlookup formula be written?

jp ronse say:

Hi,

Two things:

1. Your lookup range has only one column (H), it should be at least H:I.

2. The FALSE argument in =VLOOKUP(I4,LISTS!H1:H60,2,FALSE) requires an exact match of (G4-H4)*12/26 in the lookup range LISTS!H1:H60 and I am afraid that is not always the case also resulting in the N/A error.

Sort the table H:I ascending on H and omit the FALSE argument.

Round the formula (G4-H4)*12/26 to get an exact match.

Different Accounting Formats in Excel - Microsoft Community

aleisha gottlund say:

Different Accounting Formats in Excel

I have inherited a spreadsheet in which the values for Row 15 are calculated by subtracting Row 14 from Row 13. The resulting value should always be zero, and almost all of these cells show the zero as (0.00), which is correct. I have one cell which does not show (0.00) but instead shows - , a dash. When I go up to format the cell, the drop-down shows "Accounting $-". For all of the cells which display (0.00), the formatting drop-down shows "Accounting $(0.00)". Why is it showing these two different methods of displaying the Accounting formatting, given that the formulas are the same in each cell?
joeu2004 say:

I wrote:

Such infinitesimally small values are a common side-effect of computer arithmetic using 64-bit binary floating-point, which is how Excel (and most applications) chooses to represent numeric values internally.

Sorry, I was pressed for time. I can explain a little more now.

You wondered why there seems to be only one exception -- one cell that displays a dash, which is exactly zero.

In a nutshell, it is because there are so many ways that a calculation can result in an infinitesimal non-zero value, but only one way that it can be exactly zero. So the odds are in favor of an infinitesimal non-zero value. 

It usually goes unnoticed because most formats show some form of zero for both exact zero and non-zero values

(The General format also distinguishes between exact zero and infinitesimal non-zero values.)

Also, it is difficult to predict when a binary floating-point calculation, which should be exactly zero mathematically, will actually result in exactly zero. Sometimes, the result depends on the order of operations.

For example, if A1:A4 are $(4.40), $79.89, $(202.11) and $126.62, =SUM(A1:A4) displays $-, exactly zero in Accounting format.

But if A1:A4 are $79.89, $(202.11), $126.62 and $(4.40), =SUM(A1:A4) displays $(0.00), a negative value that rounds to zero in Accounting format. The actual sum is about -8.88E-15.

The only difference is the order of the summands.

-----

The root cause of the problem is: most decimal fractions cannot be represented exactly using 64-bit binary floating-point. Instead, they are approximated by the sum of powers of 2.

Moreover, the approximation of a decimal fraction might vary with the magnitude of the integer part.  That is why 10.01-10 does not equal 0.01. Note the difference in the approximation of 1/100 in 10.01 and 0.01:

10.0099999999999,997868371792719699442386627197265625
 0.0100000000000000,0020816681711721685132943093776702880859375

(I use period for the decimal point and comma to demarcate the first 15 significant digits, which is all that Excel formats, rounded.)

The first 15 significant digits of both decimal fractions round to x.01. But 10.01-10 becomes:

0.00999999999999978,68371792719699442386627197265625

I hope that makes sense when you look at 10.01.

There are other binary forms that can represent decimal fractions exactly; VBA type Decimal, for example.

But the advantage of binary floating-point is the wide range of values that can be represented.

Also, there are some calculations that are more accurate using binary floating-point.  For example, in 64-bit binary floating-point, 3*(1/3) results in exactly 1, whereas CDec(3)*(CDec(1)/CDec(3)) results in 0.9999999999999999999999999999.

Of course, the flipside is also possible: type Decimal calculations are sometimes more accurate than 64-bit binary floating-point, especially simple addition, subtraction and most multiplication.

I hope that helps your understanding. In summary, it's a crap-shoot. And again, the way to get reliable results is to explicitly round calculations when you expect accuracy to some number of decimal places.
