john - project say:

Imogg,

You didn't miss a setting but you did miss a field, it's the Unique ID field. In Project the ID field will change to reflect a contiguous numbering sequence for all tasks, however the Unique ID field is permanent. Once a task is created it is given a unique ID and if that task is moved, it will retain it's unique ID. If that task is deleted, that particular unique ID is gone from the file. If a file is heavily edited over a period of time, the unique IDs may be "all over the place" even though the ID will show a nice continuous sequence.

Project has a built-in utility called "Compare Projects" and it is found under Project/Reports group. That utility uses the Unique ID field to index two genetically related projects to find differences. When I say "genetically related" I mean that both projects came from a single source (i.e. a project is created, saved and then edited and saved under a new name). The original version and a later version of that same file can be compared using the utility. However, the utility will NOT work for project that are simply "similar" but not related.

Hope this helps.

John

david j parker say:

If your shapes do not contain the database table unique ID, but they do contain pseudo keys, then you could create an updatable view in your database that has the pseudo key.
For example, your Employee table could have a guid as the primary key, but the combination of FirstName + LastName may be sufficient to uniquely identify a record.  Therefore the FisrtName and LastName fields are a pseudo key.  You should be able to create an updateable view defning these as the complex key in our view.  You can then use the Viso Database add-on to link to the view, and revords should get updated, added, or deleted - thus performing these actions on your base table.
