jeeped say:
You can split off the last word separated by a common space with this,

=PROPER(TRIM(RIGHT(SUBSTITUTE(A2, " ", REPT(" ",99)),99)))

I've changed every space to be 99 spaces then taken the right-most 99 characters. That is then trimmed to remove extra spaces and proper case is applied.

talking moose say:
I perfectly understand your dilemma but without assistance from Microsoft I'm not sure anyone will be able to help you identify what you're needing.

First, the Office for Mac 2011 installers are practically universal and contain all applications. If you download a trial version, for example, you'll be able to use all applications including Outlook for 30 days. Once you purchase and install the license key then the applications are restricted. Outlook for Mac could very well be present on a machine that has a Home & Student license. It will be inactive but it will be present.

Furthermore, any Home & Student version can be upgraded to Home & Business by simply purchasing a new license key. The bits of the applications do not need to change.

That means the only thing different between the editions is the com.microsoft.office.licensing.plist file found in /Library/Preferences. It's not unreadable but it's gibberish unless you know how to interpret it.

I'm not sure Microsoft is even part of the software ID tagging project that TagVault.org has initiated. While they may have defined a standard (and they're working on newer versions too) that doesn't mean this is an industry standard. It has been adopted by Adobe and Symantec but those are the only major players I know.

I'm escalating this to Microsoft to ask if anything in the licensing file I mention above can be used to identify the edition or if another file can do this. I suggest you send Microsoft feedback using the Help menu in any Office application to let them know you need this support. It probably won't make Office 2011 but there's plenty of time for the next version.

joeu2004 say:

The problem arises because of the way that Excel and most applications represent decimal numbers in binary, called 64-bit binary floating-point.

In general, when we expect a calculation to be accurate to some number of decimal places (probably 2 in your case), we should explicitly round the calculation to

For example, ROUND(7206.51-(705900.51-700000), 2) and ROUND(7206.51-705900.51-700000, 2) return the same expected result, namely the exact integer 1306.

-----

The reason for the difference is because most decimal fractions cannot be represented exactly in 64-bit binary floating-point (the sum of 53 consecutive powers of 2 times an exponential factor).  And the approximation of the same decimal fraction (x.51) might vary depending on the magnitude of the integer part (7206 and 705900).

Moreover, when large and small values are added or subtracted, precision might be lost.  The consequences are unpredictable.  Sometimes, the lost precision causes unexpected results.  In your case, the lost precision is corrective; but that is only by coincidence.

7206.51 is represented internally as exactly 7206.51000000000,0218278728425502777099609375.

705900.51 is represented internally as exactly 705900.510000000,00931322574615478515625.

(I use period for the decimal place and comma to demarcate the first 15 significant digits, which is all that Excel formats, rounded.)

When 700000 is subtracted first from 705900.51 (the effect of the parentheses), the result

(By the way, the difference should have appeared to be 5900.51000000001.)

And when that is subtracted from 7206.51, the result is exactly 1305.99999999999,0905052982270717620849609375.

In contrast, the constant 5900.51 is represented internally as exactly 5900.51000000000,0218278728425502777099609375.  If that were subtracted from 7206.51, the result would be exactly the integer 1306, because the representation of x.51 is the same and the magnitude of the integers is similar enough.

On the other hand, when 705900.51 is subtracted first from 7206.51 (without the parentheses), the result is exactly the integer -698694, despite the different representations of x.51.  This is a coincidence due to the large difference in the magnitude of the integers and the concomitant loss of precision in the computer arithmetic.

Of course, when 700000 is added to the exact integer -698694, the result is exactly the integer 1306.
