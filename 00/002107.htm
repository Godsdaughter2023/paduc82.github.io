    scottgem say:

    First, using lookup fields on the table level is not recommended. Lookups should be done on the form level using list controls (see http://access.mvps.org/access/lookupfields.htm).

    Your mail merge should work from a query that joins all your tables on your key fields and then pulls in the description fields into your query. Your Word Merge doc will then link to the columns in the query as necessary.

    ken sheridan say:
    To achieve this, I'd prefer creating a separate field in addition to the lookup field that would have store the contents of the display value..............

    Don't even think about doing this.  The table would not be normalized to Third Normal Form (3NF) and would consequently be subject to the risk of update anomalies.  There is no advantage to be gained in doing so as you can return non-key values from the referenced table in a query in which the tables are joined, and use this as the source for the merge, or in any other context in which you need to see the referenced values for that matter, e.g. a report.

    cortig say:

    To relocate your user folder, you need to log out of the account, copy the account over to a different location/drive, then from another user account open the Users & Groups preference pane and unlock it.

    Then you Ctrl-click (not right click) your user account and you get the Advanced user options.

    In that pane, you can specify the POSIX path to the user account (e.g.: /Volumes/My Other Drive/Users/Me).

    There are some risks associated with it, especially when the Mac crashes and the drives are not properly unmounted: a "My Other Drive" folder appears in /Volumes and the real one becomes "My Other Drive 1". Some apps don't follow the Apple Guideline and get confused (though I haven't seen that in aaaagggeeeeessss).

    No names displayed via Bluetooth in the car - Microsoft Community

    ralphvde say:

    No names displayed via Bluetooth in the car

    I had an old Nokia phone but recently changed to the Lumia 610. When somebody calls me while driving ,  the screen in my car does not display names anymore via Bluetooth but only the phone-number so difficult to see who's calling . Anybody knows why ?
    sven johannsen say:

    Let me take a wild guess here. BT is implemented by a number of profiles, that are all designed to do different things. You can see a pretty decent list here. http://en.wikipedia.org/wiki/Bluetooth_profile#Headset_Profile_.28HSP.29  Windows phone, supports a subset,  http://support.microsoft.com/kb/2449475

    • Advanced Audio Distribution Profile (A2DP 1.2)
    • Audio/Video Remote Control Profile (AVRCP 1.3)
    • Hands Free Profile (HFP 1.5)
    • Headset Profile (HSP 1.1)
    • Phone Book Access Profile (PBAP 1.1)

    Your phone does not send anything but a number to another device over HFP/HSP, which is what lets you do a phone call. To display a name that goes with that number, that last profile PBAP needs to be working. In most cases the car kit just downloads your phone book, and then displays a name, based on the number the Lumia sends it. May be the rev levels of PBAP aren't compatible, or there is a step in the pairing setup, or in the car kit settings, to go get the address book. Hopefully that points you somewhere you could look.

    Column Fill error? or Excel bug or processor bug? - Microsoft Community

    kieran osborne say:

    Column Fill error? or Excel bug or processor bug?

    Hi all, not sure if this is a known bug, a new one, or just an error with my system, I'd be grateful if someone could have a go and see if they get the same, and/or know of a fix.


    Put the value 1.001 in cell A1

    Put the value 1.002 in cell A2

    Highlight both cells and then do a drag fill down to a value of say 2

    on my system all is well up until and including 1.046

    after this point excel starts adding a few extra decimal places

    What should be 1.047 comes out as 1.04700000000001

    these extra 0's and a 1 continue until 1.14 which then becomes 1.14000000000002


    This continues probably ad infinitum.


    Any help would be appreciated.


    Thanks

    K

    PS I'm using windows 7 with an i5 2500K 

    j.e. mcgimpsey say: It's neither an error nor a bug. It's an artifact of XL (as most other spreadsheets) using double-precision floating point numbers to store values.

    Most fractional amounts (including those represented as decimals) can't be exactly represented in a fixed number of binary digits , any more than most fractional amounts can be represented in a fixed number of decimal digits (e.g., 1/3 = 0.333333333....).

    What you're seeing is the small rounding error in the last binary digit accumulating during the auto filling of the range.

    See http://www.cpearson.com/Excel/rounding.htm for a fuller explanation.

    Instead of autofilling, if you want to prevent that accumulation, use

    A2: =A1 + 0.001

    and copy that down.
