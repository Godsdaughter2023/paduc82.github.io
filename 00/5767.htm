robert aldwinckle on forums say:

My question is: is there any way to get more details on what happened? A crash dump, stack trace, log, something??

Simplest place to start would be the Reliability Monitor.  Next System Information for  Windows Error Reports (bottom item in msinfo32.exe).  Next Event Viewer.  Unfortunately all of those only report in terms of "error signature", so a very common case is not covered:  crashes in low level system modules which were caused by their callers.  To see those cases you need at least a list of loaded modules and Stack Back Trace, such as the ones that used to be provided in drwtsn32.log.  Now that could mean having to extract those things from an associated dump, using a tool such as windbg.  However, you can actually get a list of loaded modules from ResMon, while the crash report message window is still showing.  That might be a sufficient clue to allow you to do further fault isolation.  And a way to get both a trace and a pseudo Stack Back Trace for each trace event is to run ProcMon.  Another interesting and relatively simple way to do traces of iexplore.exe is to use DependencyWalker for its Profiler feature.  E.g. that would show you .dll being loaded and called.

 

FYI

Robert Aldwinckle

Is this a home computer? You can only use roaming profiles if you log into a client-server network.
