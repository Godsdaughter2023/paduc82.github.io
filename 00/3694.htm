Excel corrupts large, complicated spreadsheets - Microsoft Community

r. shackle say:

Excel corrupts large, complicated spreadsheets

Excel 2010 has corruption problems with certain, large spreadsheets... Keep adding records, formulae and hyperlinks, and Excel 2010 eventually corrupts the data, it seems. (I've had this problem multiple times.) If there is any warning, it's that some cells won't calculate unless you select them and press Enter. (No refresh button?)

The Corruption/Deletion Problems (C/D): 
1. Some columns are blanked out. Even the header is gone. The column is the same width as before. It may have following columns. 
2. Some columns have many records blanked out. Other columns will be unaffected. Earlier and later records in those columns are fine.
3. A few records will be entirely blanked out. No data in any column.
4. Some cells are blank, but the formula is still there. It shows up as blank, but the formula remains. It refuses to calculate under any circumstances. Editing that cell doesn't help. Neither does copying the cell to another column.
5. Some records have the data disappear, but the hyperlink remains. However, the hyperlink won't work if you click on the blank cell. 
6. Sheets with many records may become connected to other sheets. In one Corruption/Deletion (C/D) episode, a large number of records from sheet 1 suddenly appeared just below my few records in sheet 2. When I deleted the unwanted data from sheet 2, they also disappeared from sheet 1. Undo. A change to any of the linked records in sheet 1 will appear in sheet 2, and vice-versa.

All spreadsheets ruined by Excel's C/D:
* Were over 5 MB
* Had 2 sheets with many records, plus 2-4 smaller ones
* The largest sheet:
-- Had 40k+ records with columns up to about J-M. (Many of these columns disappeared.)
-- Had multiple columns with formulae, typically COUNTIF, IF, LOWER, LEN, etc. formulae, including CONCATENATE, MID, LEFT, etc.
-- Had many records with a single column with hyperlinks, but not all
-- Was constantly re-sorted in various ways, with up to 5 columns per sort
* The second biggest sheet:
-- Had 0 - 20k records
-- Was occasionally cut/pasted to the end of the bigger sheet.

The main solution, if you can call it that, is to have a good, uncorrupted backup to fall back to. With that in mind, here's what I've done ...
To counteract C/D:
* Constantly save new versions, as backups. I've got 38 versions, now!
* Simplifying the spreadsheet and how to use it:
-- Remove unneeded hyperlinks
-- Simplify formulae to use smaller ranges. E.g., don't use COUNTIF(A$2:A$42828,B1000). Sort carefully and use a smaller range.
-- AsMuchAP, replace calculated cells with plain text. (First, copy the formula to Notepad.)
-- Delete all but 1 or 2 sheets (after a backup)
-- Don't cut/paste 10k+ records at a time. That caused an immediate C/D one time!  Copy/pasting 2500 records at a time went OK.
* Wait for all sorts to complete.
* Double-check that you don't have any 3rd party Excel addins running. If you do, turn them off.

What C/D is not caused by... C/D is not caused by user error, obviously. "Undo" won't help you, typically, unless you notice right away. Even then, it won't be undo-ing a user error. It's nothing so simple as white-text-on-white-background, or accidental pasting, deleting or partial sorting. It's not a virus or malware, at least according to all the tools I've used. It's not too many "extra" programs running in the background; I keep things simple. It's not superfluous services. It's probably not me turning off a needed service, unless Excel needs some service at rare times. There were no 3rd party addins to Excel when C/D occurred, I'm pretty sure.  I've got plenty of RAM, I would think (4 GB, but only 3 GB usable). This is Excel 2010 on 32-bit Win7, running on a name-brand laptop with Intel T4300.  The PC is/was stable, and ran everything else fine, except Windows Update was failing to run a .NET Framework 4.0 update.

I haven't seen C/D recently, even though my trimmed XLSX file is 5.7 MB. The concern is still there, however. Probably, I will move the data to an SQL database, or to an older version of Access (XP). Excel lets me do a lot of data manipulation, though, that would be painful to do in code.
