js4653 say:

Scanning/Testing Compressed Files: RAR support

I see that MSE scans and tests compressed files in the format ZIP and CAB and looks inside to see if the contents are contaminated. However, ZIP is not the only popular compressed format out there. The .RAR format is becoming more and more popular, as it compresses tighter than ZIP and has many newer features. I've seen numerous programs and files distributed in this format that could come from places that potentially could distribute "nasties". My feeling is that MSE should include support for the .RAR format as part of it's compressed files test. I do not know if the details of RAR are 'open' for use by others or by license. randy treit say: MSE can scan inside RAR files the same way as ZIP, CAB, and many other compression formats. The behavior should be the same. On-access (e.g. realtime protection) does not scan inside containers for performance reasons but as you note if the files are extracted they'll be scanned when they hit the disk.

On-demand scans (quick, full custom, or right-click context menu) should work fine with RAR. Put EICAR in a RAR file and then scan the directory with a custom scan. It should be detected the same as with ZIP.

Hi kmgsg,

Your install of Excel is right up to date. Since you're the kind of person who keeps their software updated (a very good thing IMHO), I'll assume you also keep Mac OS up to date.

You also mentioned that you have a workbook that crashed and you're trying to get it back using AutoRecover.

Here's what I think is happening. When Excel crashed, it saved a copy of your workbook in the AutoRecovery location. The trouble is, that copy is bad. When you open Excel, it tries to recover the bad file and crashes all over again. You're caught in an endless loop.

To break the loop, hold the Shift key down and then click on the Excel icon in your dock. This starts Excel in "Safe Mode," which skips the step to check for AutoRecovery files. This should allow Excel to run.

Now we have to deal with the bad file. Apple thinks it is very important to keep certain of your files hidden in secret places so you can't find or use them. Microsoft is stuck with this bad idea. Your next move is to unhide super secret Library folder, where many of your important files reside. Here's one of hundreds of internet articles explaining how to show the hidden Library folder:

Unhide And Make Library Folder Visible In macOS Sierra

Now you can navigate to the clumsily named location where your AutoSave workbook copy was saved. You should be able to find the bad workbook here:

 ~/Library/Containers/com.microsoft.Excel/Data/Library/Preferences/AutoRecovery/

Drag the bad workbook to the desktop. Then open Excel and use File > Open and try to open the file. I suspect all you'll get is a crash.

You can try using Excel On Line to open the file. Perhaps that version will be able to recover your work.

I think the best solution would be to use Time Machine and restore the latest version of the file that doesn't cause the crash and go forward with that version.
