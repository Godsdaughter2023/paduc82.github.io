ken sheridan say: I assume that this is a 'make table' query.  You can convert the autonumber column to a straightforward number by means of the Val function.  Here's an example using a couple of my own tables:

SELECT Contacts.ContactID, Contacts.FirstName, Contacts.Lastname,
VALl(Cities.CityID) AS CityID, Cities.City, Cities.County
INTO NewTable
FROM Cities INNER JOIN Contacts
ON Cities.CityID = Contacts.CityID;

Note that a 'make table' query creates a column of double precision number data type for the CityID rather than a long integer number, which an autonumber is, so the data type of the column should be changed.  Better still, however, is not to use a 'make table' query at all but to insert the rows into a previously designed empty table.  There is then no need to call the Val function as the values from the autonumber column will successfully be inserted:

INSERT INTO NewTable ( ContactID, FirstName, Lastname, CityID, City, County )
SELECT Contacts.ContactID, Contacts.FirstName, Contacts.Lastname,
Cities.CityID, Cities.City, Cities.County
FROM Cities INNER JOIN Contacts
ON Cities.CityID = Contacts.CityID;

cscottkelly say: As an addendum to my prior post, I determined that the solution was even simpler.  The issue is not the name of the underlying datafile "outlook.pst".  The error is caused by the name assigned to the datafile by Outlook.  When you add a datafile in Outlook by going to File>Info>Account Settings then clicking on the Datafiles tab, Outlook has a "Name" property for each datafile.  This property is defaulted to the name of the pst file.  (e.g. if your pst file is called "outlook.pst", then the Name property is defaulted to "outlook".  This is what is causing the bug.  You do not need to change the name of your underlying pst file.  All you need to do is highlight the datafile, click settings, change the Name property to be something other than "outlook" (I used "My Outlook Datafile") and then when you go back into Access and select the contacts when linking the table you should get no error.  Any Microsoft sales rep who wants to contact me about m

I received this solution from Aaron Hafele, in Microsoft Tech Support.  This did the trick, and the Outlook Data wizard is now working in Access.

1.       Check that the following directory is located in your trusted locations:  “C:\Program Files (x86)\Microsoft Office\Office12\ACCWIZ\”.  The folder will be named “Program Files” if there is not a “Program Files (x86)” folder.  You can see this setting under Access Options | Trust Center | Trust Center Settings | Trusted Locations.

 

2.       Verify that wizards are not disabled according to the following KB article:

924738  Error message when you try to open a wizard in Access 2007: "This feature isn’t installed, or has been disabled"

http://support.microsoft.com/default.aspx?scid=kb;EN-US;924738

 

3.       Rename the wizard folder and repair Access.  The wizard folder is the one specified in item #1.  Simply rename it from ACCWIZ to ACCWIZold and then launch the Office 2007 repair process.  This can be done from within Add/Remove programs.

doug robbins - mvp office apps & services (word) say:

Assuming that you have the document setup for the first page to be taken from a particular tray and subsequent pages in the document to be taken from another, I suspect that the issue is caused by the printer driver not respecting that Word has the document setup in that way when it is printing multiple copies.

However, as a work-around you could run a macro containing the following commands, which will send the document to the printer as an individual print job the number of times that you specify:

Dim numcopies as Long

Dim i as Long

numcopies = InputBox("How many copies do you want to print?",,1)

For i = 1 to numcopies

   ActiveDocument.PrintOut
Next i
