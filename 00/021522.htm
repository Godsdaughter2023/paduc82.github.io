nathan gwozdecki say:

Outlook contact notes field changing font and style when edited on mobile device

This is a known bug when using outlook exchange between PC's and mobile devices.  Editing a contact on the mobile device causes the notes field of the contact being edited to change style and change fonts, line spacing ect.  Blackberry tells me the problem is hot having a microsoft font on my android phone.  Anybody have any work arounds for this?

***Post moved by the moderator to the appropriate forum category.***

nathan gwozdecki say:

I fixed the issue myself.   The solution is below.

To fix the Outlook Contact Notes field from being corrupted by mobile devices, make the following changes in Outlook.  Basically, you are turning off auto formatting to prevent Outlook from inserting styles into the plain text of the notes field.  Editing on a mobile device turns that field to plain text because the mobile device won’t support different fonts, formatting and styles.  The work around is to disable all the formatting styles and just make the notes field plain text.  A side effect is that you might notice some of the auto formatting missing when you compose emails because the same editor is used for the notes field.  But, at least you won’t have corruption.

 

In the Outlook main screen Click   File / Options / Mail / Editor Options / Proofing / AutoCorrect Options

Then Choose AutoFormat Tab then uncheck the following:

  1. In the Replace Section UNCHECK Internet and Network Paths with Hyperlink
  2. In the Always AutoFormat Section UNCHECK Plain text e-mail documents

It is best to keep hyperlinks out of the notes.  They become corrupted when edited on the mobile.

  1. Choose the AutoFormat as You Type Tab.  Then in the Replace as You Type Section UNCHECK Internet and network paths with hyperlinks.

 

Also, as a general rule keep any embedded files and documents at the top of the notes field.  They will end up migrating up every time mobile edits anyway.

albert d. kallal say:

Good follow up. You have 3 things you asking for. We only addressed the third issue or "how" are we going to display what filter was/is used to create the report.

 

The next issue to address is how do we allow more (multiple) conditions in our prompt form. And, I assume that dates or whatever prompts on our form will be optional.

 

So, lets now address this issue. Note that the above solution to display that filter will continue to work. I had assumed that you were creating that filter in code when you mentioned what the value of doitall is (it not clear if you creating that string in code, or you had just hard coded it for this example, and thus wanted me to expand on how to create that string on the fly?).

 

So, as a general rule, you will and should continue to use the where clause of the openReport command. (you don't need the filter nor filter on command, and in fact I recommend you don't attempt to use it).

 

So, the solution use now is simply to take the values from the text boxes on this form, and build your own where clause in code. This where clause is thus assumed to be created in the doitall variable you now how.

 

Above approach works well since then the report does not need a query with parameters etc. (and we don't mess up the reports query source with a bunch of messy conditions.

 

So, to "send" the conditions to the report (or perhaps form), you simply use the "where" clause.

 

The code you create the date clause would look like:

 

dim strWhere        as string 

' or in your case above would be doitall

dim strStartDate   as string

dim strEndDate     as string

 

 

strStartDtae = "#" & format(me.StartDateControl,"mm\/dd\/yyyy") & "#"

strEndDate = "#" & format(me.EndDateContorl,"mm\/dd\/yyyy") & "#"

 

strWhere = "[InvoiceDate] is between " & strStartDate & " and " & strEndDate

 

 

Lets assume that you also had a combo box, and the sales rep was optional

(blank = all).

 

if isnull(me.cboSalesRep) = false then

 

  strWhere = " and (SalesRep = " & cboSalesRep & ")"

 

end if

 

Above assume saleRep is a number collum

 

 select what City for the report

And, for optional code, lets assume a text combo box with the city

 

if isnull(cboCity) = false then

  if strWhere <> "" then

     strWhere = strWhere " and "

  endif

  strWhere = strWhere & "City = " & cobCity & ""

end if

 

Note how the 2nd combo test is setup. You can add as "many" more conditions you want (that what the above "and" code allows.

 

Lets say we have a check box to only include Special Customers. We can add to our very nice prompt screen a check box to

 

[x] Show Only Special customers

 

The code we add would be:

 

if chkSpeicalOnly = True then

  if strWhere <> "" then

     strWhere = strWhere " and "

  endif

  strWhere = strWhere & "SpecialCust  = true"

endif

 

Note how the above code is setup to add as many more conditions as we please.

 

For sure, each combo and control we add to the nice report screen takes a bit of code, but no more messy then the query builder. The building of the where clause in code is quite nice because we DO NOT HAVE to change the reports query source (and that part of your first question, and we address that in a second).

 

By not having any parameters in the query then we can re-use the same query for different reports, and have no worries about some form that is supposed to be ope. So, a tiny bit more code eliminates the messy query problem.. For me, this is very worth while trade.

 

In above code example, I using strWhere in palce of your doitall variable.

 

As for changing the data source of the report? That should be required at all. If you have multiple tables of the same data structure, then your data not normalized (just combine all the tables into ONE table, and then add one column that differentiates why you have different tables).

 

So, I answered quest 2 and 3, but for question #1, my answer is you don't want nor should need to change the reports data source. It not done, and tends to be problematic. Since you can change the criteria as above with a where clause, then changing the data source is a rare requirement.
