In Office, you shouldn't use a feature unless you have to. In almost all cases, the only reason why you ever need a second Slide Master is to apply a different color theme (each master can only have one color theme at a time).

Unfortunately, MS Tech Support is not always up on best practices. If you created a new blank presentation, it wouldn't have any of the Slide Master and Layout formatting, so while the pasted material would initially look the same as the old presentation, editing would quickly break it down.

ken sheridan say: It appears from your examples that you are not using a weighted GPA, so you would firstly have a Grades table like this:

Grade    GradePoint
A           4
B           3
C           2
D          1
F           0

The grades achieved by students  would be modelled by a StudentGrades table or similar, with columns such as StudentID, SubjectID, AcademicYear, Semester, Grade.

The GPA is the arithmetical mean of the numeric grade points achieved over a period of time, so to compute a single GPA per student over an academic year would means a query like this:

SELECT StudentID,
AVG(GradePoint) AS GPA
FROM StudentGrades INNER JOIN Grades
ON StudentGrades.Grade = Grades.Grade
WHERE AcademicYear = "2017/18"
GROUP BY StudentID;

To compute the GPA per subject per student:

SELECT StudentID, SubjectID,
AVG(GradePoint) AS GPA
FROM StudentGrades INNER JOIN Grades
ON StudentGrades.Grade = Grades.Grade
WHERE AcademicYear = "2017/18"
GROUP BY StudentID, SubjectID;

ken sheridan say: The table you have illustrated is incorrectly designed, as the LG, GP and GPA are not attributes of the Subjects entity type.  Moreover, GPA is a computed value, so should not be a column in any base table, but returned as a column in a query's result table.

The LG and GP columns represent attributes of the Grades entity type, so should be columns in a Grades table, as I described in my first reply.  If you are using a weighted GPA system, then the Grades table should also have a Subject column, and there would be multiple rows per subject, one for each grade.  The GradePoint values would then differ from subject to subject, i.e. the more highly weighted subjects would have a higher grade point per 'letter grade' than the lower weighted subjects.

A GPA is, as I said earlier, the arithmetical mean (mean average) of a set of grade points, each of which is equivalent to a 'letter grade'.  It is therefore computed from the grades achieved by a student over a period of time by joining a table of the grades achieved by each student to the Grades table on the Grade columns.  The GPA is then returned by the AVG of the GradePoint values which correspond to the 'letter grades' achieved by the student.   I gave examples of the queries for this in my first reply.  Where a weighted GPA system is used the JOIN in the query would be on the Subject columns as well as the Grade columns.
