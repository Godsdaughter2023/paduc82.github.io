ken sheridan say: The root of the problem is you have a flawed table design.  By having separate Boolean columns for Student, Tutor and Relative you are doing what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

What you have here is a many-to-many relationship type between people and roles, which should be modelled by tables as follows:

People
.....PersonID
.....FirstName
.....LastName
etc

Roles
....RoleID
....Role

and to model the relationship type between them

PeopleRoles
....PersonID
....RoleID

The primary key of the last table is a composite one comprising both columns.  You'll find an example of this basic many-to-many relationship type and how to represent it in a form/subform as PeopleRoles.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

So if a person has 1,2 or 3 roles the PeopleRoles table will have three rows for that person, each with the same PersonID but with three separate RoleID values.

To restrict rows being inserted into your other table to those where the person is a student you can firstly prevent the second form being opened from the first if this is not the case by using code like this:

Const FORMNAME = "YourSecondFormName"
Const MESSAGETEXT = "This person is not a student."
Dim lngRoleID As Long
Dim varPersonID as Variant
Dim strCriteria As String

If Not IsNull(Me.PersonID) Then
    lngRoleID = DLookup("RoleID","Roles","Role = ""Student""")
    varPersonID = DLookup("PersonID","PeopleRoles","RoleID = " & lngRoleID")

    If Not IsNull(varPersonID) Then
        strCriteria = "PersonID = " & varPersonID
        DoCmd.OpenForm FORMNAME, WhereCondtion:=strCriteria
    Else
        MsgBox MESSAGETEXT, vbExclamation, "Invalid operation"
    End If
End If

To also allow for  the second form to be opened independently you can do as Hans says and restrict the list of a combo box in bound to the PersonID column to:

SELECT People.PersonID, Firstname & " " & LastName
FROM People, PeopleRoles, Roles
WHERE PeopleRoles.PersonID = People.PersonID
AND People.RoleID = Roles.RoleID
AND Role = "Student"
ORDER BY Lastname, FirstName;

Set the BoundColumn property of the combo box to 1, its ColumnCount property to 2 and its ColumnWidths property to 0cm (or inches if you are not using metric units).  Make sure its LimitToList property is True (Yes).

But the fundamental way to ensure this would be to add a CHECK CONSTRAINT to the other table to prohibit a row being inserted where the person is not a student.  In Access this cannot be done by executing an SQL statement in the usual way, but it can be done from the debug (aka immediate) window like this (as a single line).  Be sure the table and any forms etc using it are closed first:

CurrentProject.Connection.Execute "ALTER TABLE YourOtherTable ADD CONSTRAINT StudentsOnly CHECK((SELECT COUNT(*) FROM YourOtherTable WHERE NOT EXISTS (SELECT * FROM PeopleRoles INNER JOIN Roles ON PeopleRoles.RoleID = Roles.RoleID WHERE  PeopleRoles.PersonID = YourOtherTable.PersonID AND Role = ""Student""))=0)"

If you need to remove the CONSTRAINT you can do so with:

CurrentProject.Connection.Execute "ALTER TABLE YourOtherTable DROP CONSTRAINT  StudentsOnly"

ken sheridan say: I've given a little further consideration to this problem as I was not very happy about using a CHECK CONSRAINT.  Firstly it relies on the value 'Student' being used in the Roles table, so if it should ever be decided to use a different term to represent the attribute value 'Student', and the row in the table edited, the constraint would first have to be dropped and then re-created using the new value.  This would also be the case should it ever be decided that people of another role can also be referenced in the table in question.  So I've looked at how it might be done through the logical model itself.  You'd start as before with the following tables:

People
.....PersonID
.....FirstName
.....LastName
etc

Roles
....RoleID
....Role

and to model the relationship type between them

PeopleRoles
....PersonID
....RoleID

You haven't told us what your other table is, so I'll assume for this example that  it's called CourseRegistrations, with columns PersonID and CourseID, the latter referencing a table Courses.  

To represent the restriction of a course to people of one or more roles you'd then have a table CourseRoles, with columns CourseID and RoleID.  The two columns make up the table's composite primary key.  In your case the RoleID value would in every case be that for the Student role of course.

An additional column RoleID would be introduced into CourseRegistrations, so this would be:

CourseRegistrations
....PersonID
....CourseID
....RoleID

An enforced relationship between CourseRegistrations and CourseRoles on CourseID and RoleID would then ensure that only rows can be inserted into CourseRegistrations  for the role(s) applicable to the course.   Similarly an enforced relationship between CourseRegistrations and PeopleRoles on PersonID and RoleID would then ensure that only rows can be inserted into the table for a person who has been assigned the role in question, which would of course be restricted to Student if all rows in CourseRoles have the RoleID value for Student.

This solution would avoid the need to create the constraint in the first place, and would allow modifications of the sort I outlined in the first paragraph above to be undertaken simply by editing the data as appropriate.  On reflection I think this is a better approach.  While I've used a CourseRegistrations table as an example, the solution can of course be applied to any table which includes PersonID and one or more other columns as a candidate key.
