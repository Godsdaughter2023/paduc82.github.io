zorvek say:

Bernard is correct as are you. Your professor does make a valid point about the third IF function in that you are not providing a FALSE parameter. However, while your formula is not best form, it will never produce a "FALSE" result (the result if the third parameter to an IF function is not provided and the test proves false) because that IF function will never be evaluated for a result less than or equal to 1,000.

As Bernard stated, Excel evaluates using the parentheses to determine the order of evaluation. The first part of the formula to be evaluated is the first IF (looking at the formula from left to right) and, depending on the result of the IF test parameter, only one of the second (TRUE) OR the third (FALSE) parameters is evaluated. If B11 is greater than or equal to 100 then and only then will the third parameter to the first IF be evaluated.

Overall, your formula will always produce correct results but it is not good form in that it contains redundant logic.

And, as usual with Excel, there are more than a few different ways to implement a solution. Here are two more:

=B11*CHOOSE(MATCH(B11,{0,100,1000.0001}),B3,B4,B5)

=B11*INDEX(B3:B5,MATCH(B11,{0,100,1000.0001}))

And the professor is simply incorrect.

-----

First, Excel evaluates the parts of the IF expression from left to right; and it stops when the first true condition is encountered.  So there is no ambiguity between the first and second conditions.

Good Excel programmers rely on this left-to-right evaluation all the time.  The "best" way to write your formula with minimum change is simply:

=IF(B11<100, B11*B3, IF(B11<=1000, B11*B4, B11*B5))

Even more terse (but no more or less efficient):

=B11 * IF(B11<100, B3, IF(B11<=1000, B4, B5))

-----

Second, your original syntax will indeed work correctly 100% of the time, for the very reason that "there is no use" for an "IF FALSE" part.  That is, any "IF FALSE" part would be useless because there is no way it would ever be evaluated.

My suggested "best way" makes that clear by making B11*B5 the "IF FALSE" part.

ken sheridan say: In your first post you refer to a report, but in the last post to a form.  The method of sorting differs in each.  With a form you can either order the form's underlying query with:

ORDER BY Abs([Service_Completed])

or you can set the form's OrderBy property to:

Abs([Service_Completed])

and in the form's Open event procedure put:

Me.OrderbyOn = True

In a report you should use the report's internal Sorting and Grouping mechanism and create a group level on the expression Abs([Service_Completed]).

This works because Access implements a Boolean TRUE as -1 and a FALSE as 0, so returning the absolute value with the Abs function allows the FALSE values to sort first as 0 sorts before 1.  If you regard this, as the head of one software company of my acquaintance calls it, as 'being unduly chummy with the implementation' you can use one of the following expressions:

IIf([Service_Completed],1,0)

or:

Not [Service_Completed]

or:

([Selected]=False)

instead of calling the Abs function.

jay freedman say:
The Avery Wizard installs a template in the Word Startup folder, and the macro code in that template is not compatible with any recent version of Word. The solution is to find the template and delete it.

The template is typically named AVWiz12s.dotm, although the number may be different depending on which version of the wizard you installed.

There are three possible locations to find the template:

%appdata%\Microsoft\Word\STARTUP
C:\Program Files\Microsoft Office\Office15\STARTUP
C:\Program Files (x86)\Microsoft Office\Office15\STARTUP

For the first one, paste the path into the address bar of the file explorer and press Enter, so the %appdata% part gets expanded to the proper place in your user profile.

The last of the locations will exist only if you're running 32-bit Office on 64-bit Windows.

Look in each place and delete the AVWiz file wherever you find it. Then restart Word.
