IMO, building in XML in this way is only really worth trying if there is a significant performance issue, because you are basically building a big XML string in memory and only interacting with Word once (when you insert it). The fact that you have to build a whole document is likely to be a technicality unless you need unusual formatting. If you build as a Word 2003 XML document, it may be as simple as :

 - the first part is boilerplate

 - the second part is similar to specifying a table in HTML

 - the third part is boilerplate.
A lot depends on whether you have to do stuff like format cells differently depending on their content. If you have to do that, you either have to be able to generate suitable XML, or format the cell post-insertion via the object model, which begins to undermine the simplicity of the approach.

The other consideration is how big such an XML string might need to be, and/or whether you would generate it using VBA string operations alone, or make use of MSXML to build an XML document. You may run into size limitations either way - that's one area where the .NET approach may help.

guessed say:

Since you must be a big enough Far Side fan to recognise the photo, here is a macro to help in your exploration of xml and content controls.

This function embeds an xml file (sFile is a file path) into a document (oDoc). When this xml is embedded you can use the CC toolkit to link content controls to elements within the xml file. Then you can use the function again to swap out the xml file anytime you need a new build of the document. Note that the first part of the function removes all custom xml but I have left in the remmed out lines which allow you to remove only a specific custom xml file based on its namespace.

  

Private Function AttachXML(oDoc As Word.Document, sFile As String) As Boolean

  Dim oPart As Office.CustomXMLPart
  'remove custom parts if any are already in the document
  For Each oPart In oDoc.CustomXMLParts
    'If oPart.NamespaceURI = "http://www.chrysalisdesign.com.au" Then
      If oPart.BuiltIn = False Then oPart.Delete
    'End If
  Next oPart
  'embed fresh custom part
  Set oPart = oDoc.CustomXMLParts.Add
  On Error GoTo ErrCatch
    oPart.Load sFile
  On Error GoTo 0
  AttachXML = True
  Exit Function

ErrCatch:
  If Err.Number = 432 Or Err.Number = -2147467259 Then
    MsgBox "The file was not found: " & sFile, vbCritical + vbOKOnly, "File Not Found"
  Else
    MsgBox "An error was encountered: " & Err.Number & vbCr & Err.description, vbCritical + vbOKOnly, "Error"
  End If
  AttachXML = False
End Function

net stop bits
net stop msiserver
ren C:\Windows\SoftwareDistribution SoftwareDistribution.old

net start wuauserv
net start cryptSvc
net start bits
net start msiserver
Now rename the file to update.bat, and run the batch file. Keep us posted on the results.

graham mayor say:
The common causes of label misalignment are as follows.

1. You have added a header/footer to the normal template (rename the normal template)
2. The page size set in the printer properties doesn't match the page size of the label sheet.
3. You have a scaling option set in the zoom section of the print dialog
4. Your printer has poor paper handling. Some printers will slip when faced with label stock and thus are incapable of feeding the label correctly. Cleaning the rollers in the print path may help.

You may find http://www.gmayor.com/Envelope_Label_Add_In.htm useful. It can be used with any (portrait format) label template, and you can store your commonly used label data in an Excel worksheet.
