ken sheridan say: The value of the Date Modified column will only be updated when you save the updated record, not when you change the value in the address control (or any other control for that matter).  The same goes for my code.  If you want the it to update immediately then you should do it in the AfterUpdate event procedure of each bound control with:

Me.[Date Modified] = Now()

This is far simpler than my code of course, but that's because you are time-stamping updates, not necessarily changes to the data.  You could for instance change the address, but then change it back to its original value, and the date/time value would update.  The value would not have changed from the original, however, so the date/time value is logging the act of changing it.  It's this behaviour which my code overcomes, as it detects changes to the data, not just updates.

ken sheridan say: As you want to time-stamp a changed record you'll need the basChangedRecord module form my demo file, and the model for your form will be the frmChangedRecord form.  When you say 'doesn't work' that doesn't give us much to go on.  Does it raise an error?  Does it fail to time-stamp the record? Or what?

Let's go through the form's module step by step.  I've changed it slightly from the one I posted on SkyDrive by introducing a variable to flag when a new record is entered.  Relying on the NewRecord property didn't fully work as once the AfterUpdate event procedure had executed the value of the property became False.

1.  Firstly, the module's declarations area is as follows:

Option Compare Database
Option Explicit

Dim blnNewRecord As Boolean

This is where the blnNewRecord varaiable is declared to make it available throughout the module.

2.  The form's Current event procedure is:

Private Sub Form_Current()

   On Error GoTo Err_Handler

   Dim dbs As DAO.Database, rst As DAO.Recordset
   Dim strSQL As String
   
    If Not Me.NewRecord Then
       strSQL = "SELECT * FROM Contacts " & _
           "WHERE ContactID = " & Me.ContactID
       
       Set dbs = CurrentDb
       Set rst = dbs.OpenRecordset(strSQL)
   
       StoreOldVals rst
    Else
        blnNewRecord = True
    End If
 
Exit_Here:
   Exit Sub
   
Err_Handler:
   MsgBox Err.Description, vbExclamation, "Error"
   Resume Exit_Here
   
End Sub

If the record is an existing one (Not Me.NewRecord) this firstly establishes a recordset based on an SQL statement which returns one row from the Contacts table, for the current Contact, identified by the value of the ContactID primary key column.  It then calls the StoreOldVals procedure in the basChanged record module, which stores the values of the columns in the current row in an array.  If a new record is being inserted the value of the blnNewRecord variable is set to True.  This is used later in the module to determine if a new record is being inserted or an existing record is being edited.

3.  The form's BeforeInsert event procedure is:

Private Sub Form_BeforeInsert(Cancel As Integer)

    Me.DateTimeStamp = Now()
    
End Sub


This simply sets the value of the DateTimeStamp control to the current date/time as soon as a user begins to enter a new record.  This will overwrite the default value, which is set when the user moves to the empty new record, not when they actually begin to enter data.  A user could move to a new record, but go to lunch before they begin to enter data, so this avoids the time being too early.  The default value is really there in case a record is inserted in some other way, e.g. via code in a function or procedure.

4.  Finally the form's AfterUpdate event procedure is:

Private Sub Form_AfterUpdate()

 On Error GoTo Err_Handler

    Dim dbs As DAO.Database, rst As DAO.Recordset
    Dim strSQL As String
       
    strSQL = "SELECT * FROM Contacts WHERE ContactID = " & Me.ContactID
       
    Set dbs = CurrentDb
    Set rst = dbs.OpenRecordset(strSQL)
    
    StoreNewVals rst
       
    If Not blnNewRecord Then
        If RecordHasChanged() Then
            ' MsgBox "Record has changed"
            strSQL = "UPDATE Contacts SET DateTimeStamp = #" & _
                Format(Now(), "yyyy-mm-dd hh:nn:ss") & _
               "# WHERE ContactID = " & Me.ContactID
           dbs.Execute strSQL, dbFailOnError
        End If
    End If
    
    blnNewRecord = False
    
Exit_Here:
   Exit Sub
   
Err_Handler:
   MsgBox Err.Description, vbExclamation, "Error"
   Resume Exit_Here
   
End Sub

This is where the real work is done.  Firstly it again establishes a recordset based on an SQL statement which returns the current record.  It then calls the StoreNewVals procedure, which stores the values of the columns in the updated record in an array.  So we now have two arrays, one containing the values before the record was updated, the other containing the values after the record was updated.

Next the code examines the blnNewRecord variable to see if a new record has just been inserted.  If not it calls the RecordHasChanged function in the basChangedRecord module.  This function loops through the contents of the two arrays and compares them.  If it finds that a new value differs from an old value the function returns True.  My demo next pops up a message box to say the record has changed, but I've commented this out in the code above as you don't need that; it's really only there for demonstration purposes.  It then builds and executes an SQL statement which updates the current row in the table, setting the value of the DateTimeStamp column to the current date/time, using the ISO format for date/time notation of YYYY-MM-DD hh mm ss ( the standard uses case to distinguish between months and minutes, whereas the Format function, not being case sensitive, uses m and n respectively).  This ensures that the literal date/time value is interpreted correctly regardless of the regional date format in use.  The reason for updating the value in this way rather than simply assigning the current date/time to the form's DateTimeStamp control is that if this were done it would trigger the AfterUpdate event again and the code would execute once more.

Finally the blnNewRecord variable is set to False as, even if a new record has been inserted, it is now an existing record, so if the user were to save it without navigating off the record, but then make further changes, these changes would be detected by the code and the time-stamp updated.

jim thomlinson say:

You have to be careful when aggregating and using averages. In the end you need weighted averages not simple averages. Here is an example

        Units       Price         Total
1  $   9.00  $     9.00
999  $   1.00  $  999.00
1,000  $   5.00  $1,008.00

I sold 1 @ $9 and 999 at $1

The average (9+1)/2 = $5

But the true average is $1,008 / 1,000 = $1.008

If I simply multuply through 1,000 * 5 I get $5,000

The long and the short is that your average unit price needs to be calculated within the pivot table based on the Sum of Sales / Sum of Units. This is done as a calculated field.
