ken sheridan say: What do you mean by 'main table'.  Your model as it stands is rather confusing:

1.  tblEasementWatersheds appears to be modelling a 3-way relationship between easements, drop down watersheds and drop down sub-watersheds, in that it has foreign keys columns EasementID, WtaershedID and SubwatersheadID.

2.  On the other hand the also seems to be a relationship between drop down watersheds and drop down sub-watersheds in that the WSHEDID column in the latter references the ID primary key of the former.

The implication of this is that the relationship between tblEasementWatersheds  and tblDropDownWatersheds is unnecessary as each row in the former maps to a row in the latter via tblDropDownSubwatersheds.  If we assume a tblEasments table in addition to those you've described, the model would, diagrammatically, look like this:

tblEasments----<tblEasementWatersheds>----tblDropDownSubwatersheds>----tblDropDownWatersheds

For data entry an appropriate interface would be an Easements form with an EasementWatersheds subform based on tblEasementWatersheds .  The controls in the subform would be a combo boxes bound to the WatershedID column but showing the value of the NAME column from tblDropDownSubwatersheds.  Code in the combo box's NotInList event procedure would enable a new row to be inserted in tblDropDownSubwatersheds if a new name is entered in the control, by opening a form based on tblDropDownSubwatersheds in add data-mode and passing the name to it as its OpenArgs property.  This form would in turn include a combo box bound to the WSHEDID column and showing the WSHED_NAME from tblDropDownWatersheds.  This control would also make use of its NotInList event procedure to insert a new row into tblDropDownWatersheds where necessary, though in this case it would not be necessary to open a form based on that table as the row can be inserted entirely in code.

BTW do not use NAME as a column name.  It's the name of a built in property in Access, so as a 'reserved' word should be avoided for object names.  Always use a specific descriptive term such as DropDownSubwatershedName.
