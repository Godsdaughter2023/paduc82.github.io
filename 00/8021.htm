petrib say:

The Symbian OS is from the ground up based on Unicode.

 

A particular phone might not have Unicode fonts with all languages as part of the firmware, but they might be installed afterwards.

 

A hard reset might (and probably does) remove fonts from phone memory that are not in ROM (read-only memory).

 

A firmware update might restore them, if the fonts are part of the firmware update.

 

Otherwise, a firmware update might also cause languages to be lost, if the firmware has been updated to a different language variant after the phone left the factory:

Split Database - Linked and unLinked Tables - Microsoft Community

jeffreybeck say:

Split Database - Linked and unLinked Tables

I am in the process of splitting our company database because it keeps crashing and I have read that splitting it may help.  A short history of this is I started for this company a year ago.  I came in as they were migrating from XP to Win7 and Office 97 to Office 2007, (a few have Office 2013) they use Access as their "ERP" system.  The company hired an outside contractor to do the Access upgrade and conversion.  I have taken the Microsoft Access 2010 course and understand the basics and learning more on VB.

Now, here is my situation.  The database I split has queries and macro that either import tables from other databases, renames tables, or copies a table to another "new" table.  These tables are then in the local front-end and not the back-end.  Is there a way, without having to search every query and macro for code, to set maybe a default source database or current database?  (I have thought about not using local front-end but a shared network front-end.)  Any recommendations would be great.  

albert d. kallal say:

As a general rule any code or process that re-names a table is not going to work in any kind of multi-user environment. Such designs tends to suggest the data structure is un-normalized. While not the end of the world, such designs are usually not suited for a multi user database system.

And how will code handle some form or report is based on a table, but now some other user wants to re-name the table that is in use! You cannot re-name a table that is opened and in use by someone else. In other words, the whole approach to application development assumes that table re-naming and copying should not occur for the “main” application tables.

Keep in mind the suggestion in most cases is to try and normalize your data. So in place of creating a new table every month say like this:

SALESJun2015
SALESJul2015
SALESAug2015

You simply add a SINGLE new field (column) to the database called SalesDate. Now you ONLY have one table, and pull out required data based on the date column.

The beauty of above is with one simple column, you eliminate all the need to create a new table each month. And even BETTER is you can now run reports on several months of data into a single report (and ones that cross year boundaries also – say from Dec-Feb for example). And a single report and single query can thus operate on that set of tables. If you start re-naming tables then you now be faced with having to change quires and reports or worse copy reports over and over to point to those other tables. Your application will eventually reach a point in time which the application becomes un-maintainable. (it will cost too much to run and you risk the application being abandoned as a result).

So normalizing table data allows you to eliminate all that code and time and resources you are chewing up to create new tables, and copy data over and over. As noted, such table data over time tends to have quiers, forms, reports AND also code that works on such data.  You cannot mange the issue of all those query, form, reports etc. now must attempt to point to different tables.

So the fact of having to create such tables all the time will not work in the vast majority of database systems.

The exception to above is when you need to do some pre-processing of data or work on some data that you toss out after having been process. So in this case, of course it is “common” to use “temp” working tables. Even in the case of importing data, we often will pull such data into a temp table(s), process the data, re-format data before importing or pushing into the main application tables. However such “working” tables are NOT multi-user and can (should be) local tables that you are free to re-name, delete etc.

So the possibility exists here that the tables you speak of are for pre-processing data and then tossed out when done. If this is the case, then as noted keep and place those tables in your front end – such tables (and data) is really only single user data anyway. Keep in mind that using such temp tables will cause significate bloating (file size growth) of the front end. Such space is not recovered until a compact + repair is done. A more elegant solution is to create an external temp database that resides in the same folder as the front end, and have your application use that "local" to one user external database for such temp data processing needs.

So in fact, splitting often solves these types of problems without any code changes for working with temp tables since such tables are only being used by the front end, and each user will thus not “trip” over each other. Thus you are free to re-name, copy, and delete such tables.

However the actual “final” application data will reside in the back end as noted MUST be a static and un-changing design in terms of table structure and table names. If this is not the case, then you need to address this serious design flaw. As noted, this design issue is not limited to Access, but the whole software and database industry in general.

You can certainly change your code to open + use the back end database, and you can write code that will copy tables in the back end. However for table re-naming etc. you will find many of these operations are not allowed if the back end is opened in multi-user mode. I note this, since you could attempted to adopt code that copies and re-names tables in the back end from the front end, but it simply not a recommend approach and thus tends to be rather problematic if such a system is to be multi-user.

So the “how” and for what reasons such table re-naming and why repeated copying of tables is required is likely the first area to address (this should not be required). Data normalizing can often eliminate the need to create a table each month or for whatever reason/time period.

And if such table(s) are for temp data import and processing before such data goes into the main application tables (which as noted should not change in name nor in structure), then such code and tables should work fine if such tables remain in the front end.

Regards,
Albert D. Kallal (Access MVP)
Edmonton, Alberta Canada
