john - project say:

This requires a macro or a bunch of manual effort. Years ago I wrote a macro that basically does the opposite of what I understand you want to do. 1. Arrange the original file such that each of the future subproject is under it's own summary

10. Use the information stored in the extra text fields to establish cross project links. When done, delete the data in the extra text fields

Now you have a master and 8 subprojects with cross project links that, if everything went okay, replicated your original file.

ken sheridan say: Firstly note that your academic years table must not have overlapping ranges, so its data should be:

AcademicYearID    YearName    DateFrom       DateTo
1                             2018/2019    01/09/2018    31/08/2019
2                             2019/2020    01/09/2019    31/08/2020

If we assume that the FeeAmount value is the fee for the academic year, then you can return the current balance due for the year per transaction per student with the following query:

SELECT SP1.StudentID, SP1.AmountPaid, SP1.PaymentDate, AY1.YearName,
SP1.FeeAmountDue -
    (SELECT SUM(AmountPaid)
    FROM tblStudentPayments AS SP2 INNER JOIN tblAcadimicYears AS AY2
    ON SP2.PaymentDate >= AY2.DateFrom AND SP2.PaymentDate < AY2.DateTo+1
    WHERE SP2.StudentID = SP1.StudentID
    AND AY1.AcademicYearID = AY2.AcademicYearID
    AND SP2.PaymentDate <= SP1.PaymentDate) AS BalanceDue
FROM tblStudentPayments AS SP1 INNER JOIN tblAcadimicYears As AY1
ON SP1.PaymentDate >= AY1.DateFrom AND SP1.PaymentDate < AY1.DateTo+1;

This query does not return an updatable recordset, so in a form you should use an unbound text box control with the following ControlSource property to return the balance:

=DLookUp("BalanceDue","qryBalances","StudentID = " & [StudentID] & " And PaymentDate = #" & Format([PaymentDate],"yyyy-mm-dd") & "#")

VARIOUS UNINSTALL METHODS FOR OFFICE VIA: CONTROL PANEL / FIX IT / TROUBLE SHOOTER / MANUALLY

SIMPLE/ “SOFT” UNINSTALL SAVES CUSTOMIZATION SETTINGS, IN THEORY!
Since day one, what became Office applications have had quality problems. So MS designed them to be “user friendly”. You can do a simple uninstall, from the Control Panel and the process leaves behind your configuration files, like Normal.DOTM for Word, and Registry Entries. The developers came up with a 4 step “fast and dirty” trouble shooting process for themselves and for users:
. Step 1, start the application in Safe Mode to disable addins.
. Step 2, do an Office Repair
. Step 3, un-install through the control panel then re-install The re-install picks up the exiting configuration information to save the user the effort of having to re-setup their installation
. Step 4, do a full uninstall-reinstall using the appropriate “fix-it” script. This wipes all configuration information, on the chance the problem in is in the configuration files/registry.

This “soft” uninstall also plays a part in “upgrades”. During an upgrade do a simple unstall either manually or through the upgrade installer and the upgraded installation will pick up the existing configuration information making for a “painless” User Experience. That has been my experience with Office over a couple of decades. Except now, I read in this article that this author had a different experience, Office 365 has been sometimes losing his existing configuration. <hmmmm> You can read the details here under the subsection “My Settings Matter”: http://www.theregister.co.uk/2016/07/29/office_...

.
The reason for this error is the installer will remove everything, except the "ClickToRunSvc", which will be left in ZOMBIE state.
.
The service will be listed in the services.msc console, but in a state that won't run or be enumerated.
.
Fix

• Open a CMD window (using Run As Admin)
• Type

• Hit Enter.
.
Install setup should now run without issues.
