tom_bpc say:

Dale's advice is always on the money, but maybe I'm reading Madpig's problem a little differently - it is Friday evening after all.

I'm seeing IDs 4/5 and IDs 10/11 as two different options for scheduling the work.  The first uses an early constraint on ID 4 followed by a conventional FS+10 relationship to ID5.  This is unsatisfactory because it doesn't reflect the desired logic (ID 4 to be as late as possible but no later than 10d before ID5).  The second option reverses the link to make the second task a predecessor of the first, with a 10 day lead.  This is exactly what Dale has suggested (even to the removal of the superfluous constraint.)

The problem with this, as Madpig stated, is that with "calculate multiple critical paths" checked, MSP will force total slack to zero on open-ended tasks, making ID10 in the second option (or ID4 per Dale's suggestion) critical. 

Ideally we would treat this with a "zero-free-slack" constraint that is unfortunately not available in MSP.  A possible workaround for this particular schedule logic is to replace the constraint on ID4 with a FF-10 link from ID3, which is the current driving predecessor for ID5.  This logic will break if a different driving predecessor for ID5 is introduced in some future iteration of the schedule.

I've never had much use for the "multiple critical paths" checkbox, as it offers a positive treatment for what is generally accepted as poor scheduling practice (open ends inside the schedule network.)  For those tasks that are intentionally left dangling (open ended) just to make them critical, I would suggest introducing a true successor relationship and assigning a deadline.  Then uncheck the box.

Have a great weekend.

 

trevor rabey say:

This comes up quite often, and the confusion is caused by trying to mix scheduling forwards with scheduling backwards.

Pick one and stick to it.

The critical path method, which is the only viable method available, is about scheduling forwards, with everything scheduled asap. Unless a task has a predecessor which is preventing it from starting now, then it should be scheduled to start now (or otherwise asap).

In your first scenario, the test cannot start until the notice has been sent, so the notice at task 4 is (at least) a FS0 predecessor of the test, as you have shown. Task 4 is currently being scheduled to start no earlier than 7/29 (or 29/7 as we say in Australia) by the date constraint. The date constraint should be removed and the notice should be scheduled asap, perhaps with the purchase at task 2 being a SS0 predecessor of the notice at task 4, because task 4 should have a predecessor (because every task should have a predecessor. If task 2 has a FS0 predecessor (not shown), then that predecessor should also be the FS0 predecessor of task 4.

In addition to the test not being able to start until the notice is sent, the test also cannot be started until some time after the notice is sent, so task 4 should be a FS+ predecessor of task 5. That is, task 4 determines when, ie how early, task 5 can start, not the other way around.

This is true, but is not what is probably being asked for, which is that the notice must be sent no later than a certain time before the test starts. I think that this requirement, which seems reasonable at first glance, is at fault, because it attempts to force you to schedule task 4 based on an event which will occur in the future relative to task 4.

Why don't you want 10 and 11 to be critical? They are. Also, 5 and 6 are critical, but 4 is not. That's just the way it is, and it is what you are trying to find out.

In the second scenario, the notice cannot finish being sent until, ie no earlier than, a certain time before the test starts, so task 11 is a SF- predecessor of task 4. This is not a condition that meets the requirement for the notice to be sent no later than a certain time before the test starts.

All that is needed here is for task 4 to be a FS+ predecessor of task 5, which is true, and then provide the notice as early as possible, or else wait, and check to see that the free slack of task 4 does not go below whatever the notice period is.

diane poremsky mvp (slipstick.com) say:
Hi Nicoletta,

>>However, whenever I add or remove someone from my distribution list I need to update that list in that occurrence meeting.
>>

If you are sending the event to a distribution list,  Outlook is not detecting a change - it's still going to the DL. It doesn't know the membership changed. If you want to resent it, make a change and send an update to all. People no longer on the DL will not get the update or have the event canceled on their calendar though. 

When you send to a DL, you should expand the DL before sending, then add/remove members as needed for updates.
