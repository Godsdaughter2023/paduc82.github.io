scottgem say:

 

The field name Xmit Number has the data type of text assigned to it.  Users manually type in the Xmit Number, for example: WBR-T-999.  Now that we are entering numbers exceeding 999, the numbers are positioned after WBR-T-100 and not in sequence after WBR-T-999.  I believe that a numerical data type should have been used for this field instead of text. 

 

No it shouldn't You can't use a numerical datatype where the data is alpha numeric. I suspect the length of the filed is set to 9 characters. Changing the field length in table Design mode may be what you need. 

However, if the Xmit Number is always prefaced by WBR-T-, then there is no reason to force the user to enter it or even store it. In that case, you can have the user enter a number (or have the system generate it--see my blog on sequential numbering) and then have another control display the full code with an expression like:

Format(sequence,"WBR-T-0000")


ken sheridan say:
 Now that we are entering numbers exceeding 999, the numbers are positioned after WBR-T-100 and not in sequence after WBR-T-999.

That's because, being text, the values are sorted alphabetically, '1000' as a string expression precedes '200'  in the sort order for instance.

The root of the problem is that you are using a 'structured key' which has 3 elements, each separated by a hyphen character.  Multi-element keys should be stored as separate columns in the table, three in your case, with the third as an integer number data type.  In the form you can position the three controls side by side with the hyphen character as a label between each, so the empty controls would look roughly like this:

[   ] - [ ] - [    ]

The three columns can either be the composite primary key of the table, or if a separate surrogate key is used, be included in a single unique index.

For output in a report you can concatenate the three columns in a computed column in the report's query or a computed control in the report, using an expression like this:

[XMitNo_1] & "-" & [XMitNo_2] & "-" & [XMitNo_3]

You can doubtless think of more appropriate names for the columns.

This assumes that the values of XMitNo_1 and XMitNo_2 will vary of course, but as Scott says, if they are constants you don't need to have columns for them in this table at all.  As data they should, to conform with Codd's Rule #1 (The information Rule), be stored in two columns in a separate single row table, however (or possibly even two related tables - it depends on what these data actually represent).  You can include this table in a query to return the full concatenated value.  Don't specifically join it to the current table, however; it will then return the Cartesian product of the two tables, i.e. the single row will be joined to every row from the current table.

smittypro1 say:

"SheetReference" should be the name of the sheet to which you're referring.

If the sheet name doesn't have any spaces in it, then =SheetName!A1 is fine. If you do have spaces, or the sheet name begins with a number, then you need to add apostrophes: ='Sheet Name'!A1.

The error message you get with =Sheet1!D4 indicates that a sheet named "Sheet1" doesn't exist.
