Correlations: trouble coding data - Microsoft Community

gorzo55 say:

Correlations: trouble coding data

I'm having trouble coding my data to find a correlation coefficient in Excel 2010.  I've never tried this before so I'll explain my approach below.  I'm using data that are "yes" or "no" answers so there is no value.  When I assign a yes=1 and no=0 and try to calculate the correlation, I get a divide by zero error. 

 

I'm trying to compare two columes of "yes" or "no" answers.  The first column segregates 27 yes answers that I'm comparing to the second column, which breaks the first column down into two yes or no categories.  So the value I'm getting at is of the 27 who answered yes to column A, there are 16/27 who answers "yes" to column B, and I want to calculate the correlation coefficient between the two columns.  So I made a colume with 27 ones and a column of 16 ones and 11 zeros.  When I use this for my cell range for the CORREL function I get a divide by zero error.

 

What am I doing wrong?

mike middleton say: The correlation coefficient is usually used to describe the degree of straight-line relationship between two continuous-valued variables, i.e., the relationship you see in a typical XY scatter plot.

Your data are the simplest kind of discrete-valued data, i.e., binary and not-at-all continuous-valued, and I don't think the correlation coefficient is appropriate for summarizing the comparison you describe. Instead, I suggest you consider using a conditional relative frequency, i.e., conditional on Yes in A, the relative frequency of Yes in B was 16/27.

The reason you get a divide by zero error is that the CORREL function is the covariance of X and Y divided by the product of the standard deviation of X times the standard deviation of Y, and the standard deviation of 27 ones is zero, thus the error.

-  Mike Middleton, www.MikeMiddleton.com

    ken sheridan say:

    I'm not overly familiar with junction tables but it seems like even with a third table I'd have the same problem.

    No, you wouldn't, because you would not need to call the IIF function to list the student's strengths, only to determine their gender.  A report would be based on a query along these lines:

    SELECT Students.StudentID, FirstName & " " & LastName AS FullName,
    IIF(Gender = "Male", "He", "She") AS PersonalPronoun, Strength
    FROM (Students INNER JOIN StudentStrengths
    ON Students.StudentID = StudentStrengths.StudentID)
    INNER JOIN Strengths
    ON StudentStrengths.StrengthID = Strengths.StrengthID;

    A report would be grouped on StudentID, with the names in a group header, and the Strength in the detail section.  For the strength use an unbound control named txtStrength or similar, with a ControlSource property of:

    =[PersonalPronoun] & " " & [Strength]

    If you offset the name control and the Strength control as follows:

    [FullName]
                                  [txtStrength]

    And out the following in the group header's Format event procedure:

    MoveLayout = False

    Then the first strength will print on the same line as the student's name.

    ken sheridan say:

    It makes absolute sense in terms of database design and simplicity.

    It's more than a question of simplicity or elegance of design.  The database relational model is governed by certain rules and principles which are intended to guarantee as far as possible that databases will be robust and efficient, and will not be open to update anomalies giving rise to conflicting information in the database due to redundancy.  The process of normalization aims to eliminate redundancy within the database by means of a set of normal 'forms' of which the first, is formally defined as follows:

    First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple contains exactly one value for each attribute.

    Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).

    In your case you have up to three values for the attribute Strength by virtue of having three separate columns, so the table is not normalized to 1NF.  The same would be true of multiple values were stored in a single column in the row, e.g. as a comma separated value list, which would also violate another rule, the Guaranteed Access Rule:

    Every datum (atomic value) in a relational database is guaranteed to be logically accessible by resorting to a combination of table name, primary key value and column name.

    This was number 2 in the set of rules which Codd, who first introduced the database relational model around 1970 while working at IBM, produced in 1985 in response to what he saw as a growing departure from the principles of the model.

    What you have is a binary (2-way) many-to-many relationship type between Students and Strengths.  Such a relationship type is modelled by a third table which resolves the many-to-many relationship type into two one-to-many relationship types (sometimes colloquially called a 'junction' table).  You might like to take a look at DatabaseBasics.zip in my public databases folder at:

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

    If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

    This little demo file, as its name suggests, is a simple introduction to the design of relational databases in Access, and, amongst other things, explains the different types of relationships, and how they are used to model that part of the real world with which the database is concerned.  A relational database is really just a model of the reality in terms of its entity types and the relationships between them.  The Relationships.zip file in the same OneDrive folder takes things a little further by illustrating how relationships across the whole of the model are built up to form the overall set of tables and relationship types which make up the database.

    The same OneDrive folder also includes a Normalization.zip file which explains each Normal Form (up to 5NF) in simple terms, or at least as simple as possible, but no more so.  Don't concern yourself too much with the higher normal forms for the moment; concentrate on getting a solid understanding of the first three normal forms.  This is not to say that the higher normal forms are unimportant, but they are not quite so easy to get your teeth around as the first three, and most tables normalized to 3NF will be normalized to the higher normal forms already, without the need to do any more to them.
