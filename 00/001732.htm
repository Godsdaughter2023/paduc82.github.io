dirk goldgar say:

If your report and subreport show data, then they are based on queries, even if those queries aren't stored queries that show up on the Queries tab of the navigation pane.  Looking at the Record Source property in your screenshot, it appears that the Record Source is specified as the name of a table.  That means that when you run the report, Access will generate a query that selects all fields and records from the table.

If you click the "build" button at the end of the Record Source property line (caption "..."), then Access will offer to build a query based on the table.  If you click Yes, then you will be placed in the visual query designer, with that table in the table pane.  You can then, for example, drag the Department field down to the field grid, and click the Totals button in the ribbon to group on it.  If you then click the Close button on the query design window, you'll get a prompt to "update the property", and if you say Yes there, the Record Source will be set to the SQL of the query you just created, which will look something like this:

    SELECT tblPhoneList.Department FROM tblPhoneList GOUP BY tblPhoneList.Department;

ken sheridan say: The fundamental problem here is that you have only one table.  You should have two, a tblDepartments table and the tblPhoneList table.  The former should have a column, Department say, with distinct values, which is the table's primary key.  It can have other non-key columns representing attributes of the department if you wish.  The Department column in tblPhoneList should be indexed non-uniquely (duplicates allowed).  The two tables should be related on Department with referential integrity and cascade updates enforced, but NOT cascade deletes.  The integrity of the data is then protected.

The tblPhoneList table should have one row per employee.

By basing the parent report on the Departments table and the subreport on the tblPhoneList  table, and setting the LinkMasterFields and LinkChildFields properties of the subreport control to Department you should have little difficulty returning the employees per department without any duplication, as in my demo to which I referred you, which your report design would then be closely mirroring.  Normally the RecordSource properties of the parent report and subreports would be queries on the respective tables, but in this case that is not an absolute necessity.  Basing each on a table and sorting them appropriately by means of the reports' internal sorting and grouping mechanism will work fine.
