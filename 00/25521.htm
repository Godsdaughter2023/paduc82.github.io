paul bowden [-msft-] say:

Hi Y,

There are several issues talked about in the thread, so I just want to make sure I understand which problem you're having here. Is it that AutoUpdate *never* detects any updates for you (and never has), or that it usually works but you're not seeing the updates you expected?

I'm one of the engineers for AutoUpdate (aka MAU) and can go into some depth about how it works if you'd like to me if you want to dig in and narrow down the problem. Fundamentally, to check for updates, AutoUpdate attempts to read a pair of files (xml/cat) from a web location. The base URL will be slightly different depending on which channel you're in (production/insider fast/insider slow), and there are a pair of files for each registered app. As an example, if you're in the Production channel (i.e. non-Insider), AutoUpdate will read the following file to check for a Word 2016 update - https://www.microsoft.com/mac/autoupdate/0409mswd15.xml   ....there's version information in this file that AutoUpdate will use to compare against what you currently have installed. If it determines that you don't have the latest version installed, it will use the contents of the XML to figure out the smallest delta download package you need to get up to date (usually a 100 MB package if you're less than a month old). If you're many versions old, AutoUpdate will select a 'full' package, which for Word is just shy of 1GB.

The XML file contains the exact path to all the packages. These are all located on the Office Content Delivery Network (CDN) which has a base URL of https://officecdn.microsoft.com. There are now a complex set of networking rules which are used to figure out the best download server for your machine. At the highest level, a CDN provider is selected, for which there are three - Azure, Akamai and ChinaCache. Depending on where you are in the world, load-balancing rules, policy, etc, you'll hit one of these. Within each CDN provider, you'll be routed to a server node geographically closest to you for the fastest possible download speed.

When everything is working great, you should have a really fast update experience. Unfortunately, unexpected things can go wrong along the way - they could be things under your control or ours, but the AutoUpdate error message might not give you enough information to understand where the failure occurred. For the tech-savvy, here's where you should start for troubleshooting.

1. Determine if AutoUpdate can actually read the pair of xml/cat files on the web that it needs to determine if you are out of date. AutoUpdate will say 'no updates available' for both the scenario where it read the xml and determined that there were no updates, and for the case where it couldn't read the xml it was looking for, For this, a network sniffing tool is indispensable. Personally, I use Charles Proxy and configure it with SSL proxying. If you see red symbols against HTTP requests to the xml/cat files, it means that AutoUpdate failed out the gate. This points to a local machine problem - either your networking config, or something blocking access to the URL. Sometimes, it's useful to open Safari and try to get to the same URL manually (i.e. https://www.microsoft.com/mac/autoupdate/0409mswd15.xml). If it doesn't work there, then AutoUpdate won't work either. If it does work in Safari, then it means that either the AutoUpdate process itself is being blocked (may be by a security program on your machine?), or perhaps you have a network proxy server that for some reason AutoUpdate isn't working with.

2. Assuming the xml/cat files were read by AutoUpdate (the requests completed with a 200 response), but AutoUpdate is still saying 'no updates available' this usually really does mean that you are up to date. There are some corner cases where this is not true, and are usually situations where you have moved the applications out of the default /Applications folder, or if you somehow have multiple versions of the same app. For example, if for some reason you have two copies of Word 2016 installed, one in /Applications and one on your Desktop, you could be launching/running the version on your desktop, but AutoUpdate is only checking the version in /Applications. This will mess with your head, because the version running on your desktop (and actively running) may be 15.19.1, the one in /Applications is 15.23.0 but MAU says there are no updates available. Bottom line, unless you know what you're doing, always use /Applications.

3. At this stage, if AutoUpdate read the xml/cat successfully, you only have a single copy of the apps installed, and they are in /Applications, MAU really should be presenting you with a list of updates if you really are out of date. The only case where MAU may still refuse to list updates is if there's a signature mismatch in the XML, which is indicative of a man in the middle attack where something/someone has tampered with the contents of the XML and its hash doesn't match the security catalog (which is the cat file). Signature mismatches may be an accidental problem caused by us as well, but we usually rectify these very quickly (it's where the cat and xml become out of sync).

4. So now, you should be able to select which updates you want to download, and hit the Install button. If within seconds MAU says that there were errors, then it was not able to download the file from the Office CDN. Again, use a network sniffer to see if the requests are actually failing, or use Safari to try and download the package manually (e.g. https://officecdn.microsoft.com/pr/C1297A47-86C4-4C1F-97FA-950631F94777/OfficeMac/Microsoft_Word_15.22.1_160512_Updater.pkg) Assuming they are failing, there are a host of different causes, and it's usually a server-side issue we need to fix. Sometimes, one of the CDN Providers doesn't get updated with a particular package, sometimes, one of the server nodes in the CDN loses contact with its origin. The typical manifestation here is that only you or people in your city are seeing the issue, but others around the globe are working just fine. Again, we actively monitor the CDNs so we try to resolve these quickly (usually by taking out the bad node/CDN). This can be the cause of where you were seeing problems yesterday, but its working today, or you tried to update 10 times, and on the 10th time it worked.

5. So now the update packages should be downloading. If the download seemed to work (i.e. progress bars were moving steadily and you didn't see an error within seconds), the next step is that you should be prompted for the admin password to apply the updates. If you don't get the password prompt, but instead an error saying that there was a problem with the update, it means that the signature check failed - the downloaded file got corrupt or hijacked aong the way and the hash doesn't match what we expected. Try again.

6. After entering your admin password, you should see the progress bars moving steadily and updates being applied. If after 2-3 mins, there is no progress/activity at all, AutoUpdate might be hung. This is an active bug we are investigating, and happens sporadically. Your only workaround for now is to force quit AutoUpdate and try again. If you're unlucky enough and hit this same issue over and over, your autoupdate helper is probably bad. For this, you need to delete the com.microsoft.autoupdate.helpertool file in /Library/PrivelegedHelperTools, and start AutoUpdate again.

7. You should now be in the state where updates are being applied, and progress bars are moving toward completion. We might need you to close apps along the way, but at this point, there's not much that usually goes wrong.

We are actively working on some major improvements to AutoUpdate that make us more resilient to the types of problems that can go wrong here, as we really want you to have a great experience and not spend your time troubleshooting these types of issue. In the mean-time, I hope this information helps everyone out there understand a little bit more about how to narrow down update issues.

Thanks, Paul.
