a. user say:

If you inserted all the links in the same way as your sample dialog box shows, then Word should have inserted a "relative link" in the resulting HYPERLINK field, i.e. if you use Option-F9 (or fn-Option-F9, depending on your keyboard & setup), then you should see

{ HYPERLINK "everything/book%20notes.docx" }

In that case, what Word should do is treat the "everything" path as relative to the place where you opened the Dictionary.docx file form. So if you actually move the entire Dictionary folder somewhere else, the links should still work without you haveing to do anything.

Although the tooltip shows a file:// URL to the complete path of the file (probably file://localhost/something)  Word should fix up the tooltip and open the correct link when you open it.

If that isn't happeneing, possible explanations include:
 a. you have actually created Hyperlinks that have the complete path to the URL. I don't think these will be fixed up automatically.

 b. your dictionary.docx document has a "Hyperlink base" specified in File->Properties->Summary. When you have a Hyperlink base URL, relative links will be evaluated relative to that URL.


But there may be other reasons why it doesn't work.

IMO if you haven't actually tried moving your folders yet, one thing you can do is copy your Dictionary.docx and some sample documents from Everything into a DIctionary folder elsewhere, rename the  sample documents, open the copied DOcuments.docx and try the relevant links.

Otherwise, you *may* be able to fix up the links using Find/Replace . Make a backup of your Dictionary.docx file, thenuse Option-F9 (or fn-Option-F9 as appropriate) to display the HYPERLINK field codes, then..

 a. if you have absolute links and you want them to be relative links, you should be able to remove the first part of the URL (e.g. "file://localhost/Users/yourname") by finding it and replacing it with an empty string.

 b. if you have relative links and you want them to become absolute links, you should be able to find the following:

HYPERLINK "

and replace it with, e.g.

HYPERLINK "http://somewebaddress/Dictionary/

If you have absolute links and you want to replace them with a different absolute link, you can either do (a) then (b), or directly replace the initial Base URL with the one you want.

You may then need to select the document (cmd-A) and update all the fields (F9 or fn-F9).

I haven't verified that that process works here, but it should be hard to check for yourself whether it works or not. 




Otherwise, you'll need some code to modify the links as RIch suggests. One or two people have written such code


However, you may also find that some files open differently when they are located at an http: URL than when they are stored in a file system, depending on your browser and settings. For example, here when I click on a link to a .pdf stored locally, it opens fine. If Word has to open Safari to load it from a website, I see encoded text. If I set Chrome to be the default browser instead, the PDF opens fine.

Are you really using such numbers in arithmetic?

If so, then you need to understand the limitations of Excel numbers.  In particular, on input, Excel will only interpret the first 15 significant digits, replacing the rest with zeros.  On output, Excel will format only the first 15 significant digits, rounding the 16th digit.  Moreover, integers larger than 9,007,199,254,740,992 cannot be represented exactly.

-----

Or are such "numbers" actually some form of id, e.g. an account id?

If the latter, you need to enter such ids as text.  The method for doing that depends on the origin of the data.

If you are entering the data manually (typing), either prefix the id with an apostrophe (aka single-quote) -- for example, '10443720001094716 -- or format the cells or column as Text
(It is not sufficient to change the format to Text after entering the data.  Digits following the first 15 are lost forever.)

If you are inputing the data from a file or pasting the data from a webpage, you might be able to use an import or paste "wizard" to force the data to be treated as text.  If you need help with that, specify exactly what you are doing in a response here.

suzanne s. barnhill say:
When you insert a section break, the section that follows initially has the same settings as the original one, which means that "Different First Page" is enabled, and the first header in the section will be the First Page Header; you won't get the primary Header until the second page of the section. If you want the primary Header to appear on every page of that section, you will need to disable Different First Page for that section. For this purpose, it is not necessary to unlink either header from the previous one (in fact, if you want the Header in the second section to reflect any changes in the Section 1 Header, you need to leave them linked).

suzanne s. barnhill say:

To omit a header on the first page, you enable "Different First Page." If you insert a section break, you get a new section with DFP enabled. There is no way around that unless you create the template with two sections to begin with. You can't unlink headers unless there is more than one section. Be aware, too, that linking/unlinking applies only to like headers/footers. You can unlink the Header in Section 2 from the Header in Section 1, and you can unlink the First Page Header in Section 2 from the First Page Header in Section 1; in the latter instance, you could copy the primary Header content into the First Page Header of Section 2, but you still wouldn't be able to do that until Section 2 has been created.
