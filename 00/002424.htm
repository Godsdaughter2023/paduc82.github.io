albert d. kallal say:

Datetime2 columns work without issue with Access.

However, you WILL have to link your tables using the SQL Native 11 (or later drivers). And you thus will have to install these drivers on each workstation.

If you link your tables using the SQL Driver (this is a older driver installed on all versions of windows by default). If you use the older ODBC driver, then datetime2 columns will be seen by Access as text columns.

So your choice is:

Always migrate using the older datetime format. You can change this setting in SSMA.

After you migrate simply use sql design tools (SSMS) and change the column type to datetime. If  you just have say a few tables, then this is not much work.

If you have a lot of tables, then I would consider adopting the native 11 (or later) drivers - that way you don't have to change anything server side, and access will see + work with those types of columns correctly.

The newer drivers are better, and supposedly faster - but I not experienced performance increases.

So Access has no problem with datetime2, but you have to re-link your tables using a Native 11 or later driver for Access to see these columns correctly. About the only downside of this option is the noted require to ensure that these drivers are installed on any workstation you plan to run your application on .

ricw-nw say:

Fortunately many of the WMP library management problems are sorted in W10 (but don't upgrade - wait until you need a new pc!)

However renumbering tracks on two discs sets will always cause complications. I understand why you thought it would be better - I have done it with some of my albums because I don't like them being displayed as two discs. Both Google Music and iTunes have tidied this up so that in their libraries for example 'Now 70' will show as one album, but when you open it you will see CD1 and CD2 listed correctly... However if that was in your iTunes library

What you need to do is rip the CD but then copy the files from both discs to a location that is not monitored by WMP - a temporary folder in 'Documents' would do. Then you need to change the album title in 'properties' to NTWICM 70 (or whatever)for all the tracks (you can highlight them all and do that in one hit) and afterwards go through the track numbers and alter those on disc 2 from 1-20 to 21 - 40 (or whatever).

Then cut and paste the contents to your explorer library - they should then appear correctly in WMP... but after you have done this you must avoid re-inserting the original CD because they may well get changed back again! You also need to make sure that in 'Organize' - 'Library' Tab you uncheck all the boxes about adding /updating information.

I've found this out over quite a few years by trial and error and my library is pretty stable - so you can get there!

Ric.

georgeogden say:

This sounds like you're possibly another victim of the failed 16288.1 update and is in need of a manual rest of the Windows Update process..... in my case the "NewOS" directory wasn't being created in the update directory C:\$WINDOWS.~BT which may have also corrupted the Windows Update process. 

If you've already tried running the Windows 10 Update Troubleshooter and that hasn't fixed the problem (and it quite often doesn't) then you'll need to take more drastic action and attempt to reset the Windows Update process manually.

A simple method is to copy the text between the ++++ lines and paste it into Notepad

++++++++++++++++++++++++++++++++++++++++++++++++

net stop wuauserv

net stop cryptSvc

net stop bits

net stop msiserver

ren C:\Windows\SoftwareDistribution SoftwareDistribution.old

ren C:\Windows\System32\catroot2 Catroot2.old

net start wuauserv

net start cryptSvc

net start bits

net start msiserver

+++++++++++++++++++++++++++++++++++++++++++++++++

Save as UpdateReset.BAT (note the .BAT and NOT .TXT) to your desktop and close it.

Go to the desktop and right click on the .BAT file and select "Run as Administrator"

That should flush out any corruption in the Windows update files and enable you to move on to download the latest build 16296.0. If you get a message saying that wuauserv can't be stopped then run the Update Troubleshooter and then try the batch file again.

Once done, go back and check for updates and also check to see if the C:\Windows\SoftwareDistribution.old and C:\Windows\System32\Catroot2.old directories were created and, if they exist, delete them in case you need to re-run the batch file again.

Finally, before trying to update again make sure you detach any non-essential peripherals, USB devices and completely remove any 3rd party anti-virus software as they may interfere with the update process,

Hope that helps and good luck.
