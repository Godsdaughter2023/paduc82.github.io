Making Backups as You Work in Word 2007

http://www.msofficetuneup.com/2008/06/01/making-backups-as-you-work-in-word-2007/

Note: this site has been taken off line

Even if you’re using the macros earlier in the chapter to save your work frequently, you can still lose data if your hard drive crashes. So we’ve all learned from hard experience not only to save our work regularly, but also to make periodic backup copies. The macro I use most often in Word is one that does both in a single procedure! That is, the macro not only saves your work, but it also makes a backup copy on another drive, such as a removable disk, a second hard drive, or a network folder. Listing 1 shows the code.

Listing 1 A Procedure That Creates a Backup Copy of the Active Document on Another Drive

Sub MakeBackup()

   Dim currFi1e As String

   Dim backupFiIe As String

   const BACKUP_FOLDER = ‘G:\Backups\’

 

   With ActiveDocument

   ‘ Don ‘t bother if the document is unchanged or new

        If .Saved Or .Path = ** Then Exit Sub

   ‘ Mark current position in document

        .Bookmarks.Add Name = “LastPosition”

   ‘ Turn off screen updating

        Application. ScreenUpdating = False

   ‘ Save the file

        .Save

   ‘ Store the current file path, construct the path for the

   ‘ backup file, and then save it to the backup drive

 

        currFi1e = .FullName

        backupFi1e = BACKUP FOLDER + .Name

        .SaveAs FileName:=backupfile

   End With

   ‘ Close the backup copy (which is now active)

   ActiveDocument.Close

 

   ‘ Reopen the current file

   Documents.Open FileName:=currFi1e

 

   ‘ Return to the pre-backup position

   Selection.GoTo What:=wdGoToBookmark, Name:=”LastPosition”

End sub

 

The backupFile and currFile variables are strings that store the full pathnames for the active document and the backup version of the document. Use the BACKUP_FOLDER constant to specify the folder in which you want the backup stored.

The procedures first check to see if the backup operation is necessary. In other words, if the document has no unsaved changes (the Saved property returns True) or if it’s a new, unsaved document (the Path property returns “”), bail out of the procedure (by running Exit Sub).

Otherwise, a new Bookmark object is created to save the current position in the document, screen updating is turned off, and the file is saved.

We’re now ready to perform the backup. First, the currFile variable is used to store the full pathname of the document, and the pathname of the backup file is built with the fol­lowing statement:

backupFile = BACKUP_FOLDER + .Name

This is used to save the file to the specified folder. The actual backup takes place via the SaveAs method, which saves the document to the path given by backupFile. From there, the procedure closes the backup file, reopens the original file, and uses the GoTo method to return to the original position within the document.

TIP

Using a Bookmarkobject to reset the insertion point is useful because it takes you back to the exact point in the document where you were before the backup started. However, you may be interested only in returning to the last position within the document where an edit occurred. If that’s the case, use the following statement in place of the Selection.GoTo statement:
Application.GoBack
Note, however, that there’s a bug in the GoBackmethod, whereby Word doesn’t save the last edit position (technically, it’s a hidden bookmark named \PrevSel1) in some cases. Specifically, when you exit Word, if you elect to save changes in the last document that gets closed, Word doesn’t save the last edit position in that document.

rich michaels - mvp (greatcirclelearning.com) say:
I may be wrong but I think this problem occurs only on systems that are using OS X Lion.  I speculate that the problem is caused by a threading issue because of too many independent applications are simultaneously trying to grab an image of the active file. This results in partial images and lost indexes that both the operating system and the application running under the OS needs.  Depending what is missing either the OS or the Application hangs and that's what makes the problem look different.  Meaning the OS saying there's a disk problem versus Word saying it can't open or save a file.

Supporting my theory are the experiences of many high volume Word users (MVP and MCC's) on this forum who have not had this problem. What do we have in common?  We are not running OS X Lion.

Lion contains an AutoBackup feature, Time Machine is a backup function and both are controlled by the Operating system.  The AutoRecovery and Backup Copy functions are controlled by Word.  And then add to that mix the random saves of the file as the end-user edits it.  The result is in a nano-second of time one of these functions jumps out of line (threading) and grabs an incomplete file image resulting in "shirt hitting the fan."

Until the two teams, Mac and MS, get it worked out us poor end-users must make choices.  Lion vs. Snow Leopard, Lion Auto Backup vs. Lion Time Machine, Word AutoRecover vs Word Create Backup or some other combination of the above that keeps our system running.

Finally, I want to be clear. I'm not saying that OS X Lion is the problem. What I am speculating is that the combination of imaging functions  including backup, recovery, auto save, timed based backups, and end-user file saves are not being threaded properly and one or more of the activities are "taking cuts in line."

rohn007 say:

Question: Is the outline you showed us made up of headings and what you are showing us just the outline view where body text is collapsed under the headings.

Track changes is the feature specifically built in to Word to allow users to monitor changes being made to the file.  But the feature does not include the visual component you are looking for.

There are only 2 features that allow you to apply colors to the background of text. One is the highlighter the other is paragraph background fill.  Unfortunately neither gives you what you are looking for.

In my quick test in outline view you can use <CTL><Click> to select multiple discontinuous headings.  Which is what you want to do.

If you try to apply highlight to the selections, only the last (current) selection is highlighted.  The heading and body text is highlight.

If you try to apply the paragraph background fill, the color is applied to all of the selections, BUT you cannot see the color in outline view.

Can you use either tip?

daniel g. grau say:

I managed to produce a similar error.

Word lets you save web pages using two options: "Save entire file into HTML" (to include non-browser, Word-specific information like headers and footers) and "Save only display information into HTML" (to include only browser-relevant code). I used both to create two files.

In both files, I inserted pictures as links (not that it should make any difference, as pictures are stored by Word in an ancillary folder anyway). Word produced these respective links:

{ INCLUDEPICTURE ":FULL PATH HERE:NameOfFolder:Name of image ... }

{ INCLUDEPICTURE \d "Macintosh HD:FULL PATH HERE:NameOfFolder/image001.png ... }

The differences:

  • the \d (display) option
  • missing disk name
  • slash at the end (Word will no longer locate the file; if this is "corrected," Safari won't locate it)
  • changed image name

Both versions opened fine in Safari, but the second one displayed a blank picture in Word (on account of the slash). And since there's a lot of hidden HTML code, the field cannot be repaired by hand.

If your problem fields resemble the above \d example (press Option F9 to toggle field codes), you should have created the HTML file with the "Save entire file" option in order to re-open it with Word.

How do you restore the invisible images? The only reason would be to be able to open the HTML file in Word and see the pictures. So either reconstruct the file (there's no going back with Save As) or fix manually:

  • Open the HTML file in Word as Text Only (you'll have to enable "Confirm conversions at Open" in Word > Preferences > General).
  • Locate the lines that say src="FOLDER_NAME/IMAGE_FILE_NAME"
  • Change them to src="file:///Users/YOUR_NAME/PATH_TO_FOLDER/FOLDER_NAME/IMAGE_FILE_NAME

This can be done with a global replacement. For instance, if the two strings are:

  • src="Test%20files/image001.png"
  • src="file:///Users/daniel/Desktop/Test%20files/image001.png"

Search for src=" and replace by src="file:///Users/daniel/Desktop/

Make sure the "Straight quotations marks" option is turned off in Word > Autocorrect > AutoFormat as You Type.

Conclusion: If you want to open the HTML file in Word, use "Save entire file into HTML." Better yet, keep a docx working file and do the final save with "Save only display information into HTML," as this will produce a simpler HTML file.
