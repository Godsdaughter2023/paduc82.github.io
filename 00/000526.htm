Age calculation.

Use this --

john w. vinson say:

It is neither necessary nor appropriate to store the age in days in any table: if you were to store it today, you could be absolutely certain that it would be wrong tomorrow!

Instead, just store the birth and death date and calculate the age on the fly, as needed; you can do this in a Query or in the control source of a textbox on a form or report. Tables should not be used for routine interaction with your data - use a Form instead.

You can calculate the age in a query by typing

Age: DateDiff("d", [DateOfBirth], NZ([DateOfDeath], Date()))

in a vacant Field cell in a query. To do it on a textbox just use

=DateDiff("d", [DateOfBirth], NZ([DateOfDeath], Date()))

as the Control Source of a textbox.

Fill-in-as-you-type Combo Box in Access

franklinlan say: Customer is a local variable on the Order table. This will be easier for my manager to pull data just at the Order table, instead of linking to another Customer table in a query. 

ken sheridan say:

Customer is a local variable on the Order table. This will be easier for my manager to pull data just at the Order table, instead of linking to another Customer table in a query. 

That is not a good design as it removes any possibility of enforcing referential integrity.  The fundamental principles of the database relational model are quite clear;  each entity type is modelled by a table whose columns represent attributes which are functionally determined solely by the table's primary key.  Customers are an entity type and the customer's name (or first and last names if they are personal rather than corporate entities) an attribute of the entity type.  Names are not suitable as keys as they can legitimately be duplicated, hence the need for a distinct numeric CustomerID value per customer, usually an autonumber, as the key.  The referencing table Orders will then include a CustomerID foreign key of long integer data type, and the relationship between Customers and Orders on CustomerID will be enforced to protect the integrity of the data.

Inserting a new customer by typing the name into the combo box could be done by means of the control's NotInList event procedure.  You'll find an example in NotInList.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the opening form allows a new contact to be inserted in this way, but as this assumes that contact names are distinct, which cannot be assumed in reality, an alternative method via a command button is illustrated.  This allows two or more contacts of the same name to be inserted.  In the combo box's list they would be distinguished by their address data, which is concatenated with the FirstName and LastName columns in the list.

There is no foundation for your assumption that the inclusion of the contact name in a column in the Orders table would make things easier for your manager to retrieve data.  With a correctly structured set of related tables and a properly designed search interface it can easily be done.  The MultiSelect demo in my same OneDrive folder includes various forms which illustrate methods for retrieving data in a number of different ways in the context of a set of correctly normalized tables.

tom van stiphout say:
It is *not* easier to pull data from an incorrectly designed Orders table than it is to pull data from a correctly designed database with an Orders query. In either case you need to open ONE object, either a table or a query.

scottgem say:

"Local variable" is not a term associated with tables. Your Orders tables should have a foreign key for the CustomerID. 

AutoExpand is a feature that allows the combobox to search through the RowSource and match the first item that matches what is being typed. So if the user types S, it will position on the first item that starts with S. if the user then types c, it will position at the first item that starts with Sc and so on until no match is found. So it has nothing to do with populating the rowsource.

You DO need a table of Customers. And that table would be the RowSource of the Customer's combo. You would then use the NotInList event to add the new Customer name to that table. But as Tom pointed out, a new Customer would generally involve more info then just a name. So, I would be inclined to add the name and then open a popup form to fill in additional info about the new customer.

Getting table design right is the most important factor in starting a new database. Storing the customer name as part of the order record, is not good design and has the potential for creating update anomalies. 

Finally, unless your business is just starting up, there will be a list of previous customers somewhere, even if only on paper. It behooves you, as the developer, to populate the customers table with some data before deploying.
