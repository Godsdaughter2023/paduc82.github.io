=((SUMIF($P$2:$P$120,"01:00:00",$T$2:$T$120)*SUMIF($P$2:$P$120,"01:00:00",$V$2:$V$120))+(SUMIF($P$2:$P$120,"01:30:00",$T$2:$T$120)*SUMIF($P$2:$P$120,"01:30:00",$V$2:$V$120)))/(SUMIF($P$2:$P$120,"01:00:00",$T$2:$T$120)+SUMIF($P$2:$P$120,"01:30:00",$T$2:$T$120))


You can reduce the length of that formula by using cells to hold the time values.

  • A1 = 1:00
  • B1 = 1:30

You can also reduce the denominator from:

(SUMIF($P$2:$P$120,"01:00:00",$T$2:$T$120)+SUMIF($P$2:$P$120,"01:30:00",$T$2:$T$120))

To:

SUM(SUMIF($P$2:$P$120,TIME(1,{0,30},0),$T$2:$T$120))

Putting it all together without the error trap:

=((SUMIF($P$2:$P$120,A1,$T$2:$T$120)*SUMIF($P$2:$P$120,A1,$V$2:$V$120))+(SUMIF($P$2:$P$120,B1,$T$2:$T$120)*SUMIF($P$2:$P$120,B1,$V$2:$V$120)))/SUM(SUMIF($P$2:$P$120,TIME(1,{0,30},0),$T$2:$T$120))

What result do you want to replace the #DIV/0! ?

This version will return a blank:

=IF(SUM(SUMIF($P$2:$P$120,TIME(1,{0,30},0),$T$2:$T$120)),((SUMIF($P$2:$P$120,A1,$T$2:$T$120)*SUMIF($P$2:$P$120,A1,$V$2:$V$120))+(SUMIF($P$2:$P$120,B1,$T$2:$T$120)*SUMIF($P$2:$P$120,B1,$V$2:$V$120)))/SUM(SUMIF($P$2:$P$120,TIME(1,{0,30},0),$T$2:$T$120)))

That's still pretty long but not as long as it could be!

This version will return 0:

=LOOKUP(1E100,CHOOSE({1,2},0,((SUMIF($P$2:$P$120,A1,$T$2:$T$120)*SUMIF($P$2:$P$120,A1,$V$2:$V$120))+(SUMIF($P$2:$P$120,B1,$T$2:$T$120)*SUMIF($P$2:$P$120,B1,$V$2:$V$120)))/SUM(SUMIF($P$2:$P$120,TIME(1,{0,30},0),$T$2:$T$120))))

Outlook/Word eMail merge using Birthday = Today as a criteria - Microsoft Community

karl timmermans [outlook mvp 2012-2018] say:

Outlook/Word eMail merge using Birthday = Today as a criteria

Was asked by someone if it's possible to create an email mail merge where the criteria is "all people whose Birthday is today".

Outlook filter doesn't allow for the creation of any filter where "Birthday" is dynamically evaluated to "today". The objective is to have something that can be run on a daily basis where Birth/Today dates <Month & Day> are equal.

Does anyone have a sample of how it would be done via Word (if possible) using an Outlook folder as the primary data source? The second desired objective is not to export contacts on a daily basis and work with a different data source to pre-filter the data prior to running a merge.

a. user say:

This probably isn't going to answer your question but I hope it will be a helpful start.

There are two main parts to this:
 1. how to get the data out of Outlook.
 2. how to select the records you want

I'm not sure whether you have already determined what to use on  (1) - the usual methods are 
 a. start from Outlook Mail Merge option to initiate a mail where you specify the same name for the exported contacts each time, and the same name for the Mail Merge Main Document each time. "push the data into a named data file from a list of contacts in Outlook". This isn't particularly straightforward, especially if you don't save the Mail Merge Main document with the data source disconnected.
 b. pull the contacts from within Word, which I cannot test right now because I have the problem where Word crashes when you do that. There's often an Outlook login dialog and always a contect folder selection dialog which apparently cannot be avoided by using VBA

It is also possible to consider using an Access database that has a table linked to the appropriate contacts table, and use that as the data source. I think that still works in Office 2010. But you may not be able to retrieve all the fields you need from Outlook.

As for the filtering, the problem is that if anything reconnects the data source, you will probably lose any filtering that you set up in Word. That will probably happen for both methods (a) and (b) in which case you are probably going to be better off using SKIPIF fields (see below). If you use Access to link to the data, you can probably use an access query to do it and use that as the data source.

I don't particularly like using SKIP fields because they are not recommended by Microsoft but if you put a SKIPIF field at the top of the Mail Merge document like this:

{ SKIPIF { MERGEFIELD Birthday \@MMDD } <> { DATE \@MMDD } }

then you should only get emails for people with birthdays on today's date. All the {} have to be the special field code braces, as usual.

However, I suppose you also have to deal with leap birthdays, but I don't know how people with birthdays on the 29th actually do that. But you could try for example

{ SKIPIF { ={ COMPARE { MERGEFIELD Birthday \@MMDD } = { DATE \@MMDD } }+{ COMPARE "{ MERGEFIELD Birthday \@MMDD }{ DATE \@MMDD }{ QUOTE { DATE \@"YYYY-02-29" } \@Y }" = "02290228{ DATE \@YYYY }-02-29" } = 0 }

i.e. 
 a. if the dates match, the first COMPARE evaluates to 1
 b. if the year is not a leap year and the birthday is the 29th February and the date is 28th february, the second COMPARE evaluates to 1
 c. so leap year birthdays are dealt with by (a) in leap years and by (b) in non-leap years
 c. this relies on the fact that in a leap year
 { DATE \@"YYYY-02-29" } will be a valid date and { QUOTE  { DATE \@"YYYY-02-29" } \@Y } will return the year, but 
in a non-leap year, e.g. 2013
{ QUOTE  { DATE \@"YYYY-02-29" } \@Y } will return the string  2013-02-29
