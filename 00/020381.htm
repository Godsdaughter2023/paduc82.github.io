    joeu2004 say:

    Roland....  First, I hope you did not accede to Zixuan's requests.  The problem has nothing to do with your version of Excel; so there is no need to share product information.  And it is better to share examples with everyone by uploading to a file-sharing website.  Share privately only if the examples cannot be redacted; obviously not a problem in this case.

    Second, you ask two separate questions:  (a) how to avoid the value change when formatting to/from hh:mm:ss; and (b) how to ensure correct results from the time arithmetic B-C>D, given the value change?

    -----

    The answer to #b is:  when you expect time arithmetic to be accurate to some unit of time (seconds, in this case?), explicitly round to that unit of time by using the TEXT function.  For example:

    =--TEXT(B2-C2,"[h]:m:s")>D2

    It should be noted that that conversion is limited to 9999 hours.  So the arithmetic B>C+D must be written:

    =B2>INT(C2+D2)+TEXT(C2+D2,"h:m:s")

    Caveat:  That can lead to surprising results if the time is greater than 23:59:59.5.  Not a problem for you, I believe.

    -----

    Unfortunately, the answer to #a is:  it cannot be avoided, especially if you intend to write the number-formatted value (you use the General format) to a text file (e.g. CSV).

    As you suspect, "there seems to be a rounding issue" that "results out of the internal floating format".

    The root cause of the problem is the fallacy that Excel stores numbers with 15 decimal digit precision.  That is incorrect.  But it is true that Excel

    So when you copy-and-paste-value (date and) time, format it as a number (General, Number, etc), and press Enter in the Formula Bar, you lose the full precision of the time value.  That has unpredictable consequences when doing comparisons.

    Consider your first example:  comparing "5/8/2018 11:30" minus "5/8/2018 11:15" with the constant 00:15.

    By coincidence 11:15 can be represented exactly in binary with few enough bits that there is no loss of precision when the date is included.  It is exactly 43229.46875.

    However, 11:30 cannot be represented exactly in binary.  And the precision of its binary representation depends on the magnitude of the integer part (date).  So 5/8/2018 11:30

    (I use period for the decimal point, and comma to demarcate the first 15 significant decimal digits.)

    When we copy-and-paste-value in the same Excel instance, the exact binary precision is preserved.  But when we press Enter in the Formula Bar, we are re-entering 43228.4791666667, not the time value.  And that is exactly 43228.4791666667,00621135532855987548828125.

    The infinitesimal difference is enough to affect the comparisons, as you noticed.  That is why explicit rounding is so important.

    Ironically, the comparison of the date/time-formatted arithmetic worked only by coincidence.  You can see that by formatting =B3-C3 and =B6-C6 as Number with 16 decimal places so that we see 15 significant digits.

    "5/8/2018 11:30" minus "5/8/2018 11:15" appears to be 0.0104166666642413, which is obviously different from and indeed not greater than the binary representation of 00:15, which

    But "5/8/2018 16:50" minus "5/8/2018 16:35" appears to be 0.0104166666715173, which is indeed greater than what

    That is why rounding to the minute (or second) is essential for getting the correct comparison.

    (Although rounding to 6 decimal places, as Bernard suggested, works in this instance, it does not work in all instances.  Generally, it is a arbitrary number of decimal places.)

    FYI, even the comparison B>C+D can yield unexpected results because C+D might be infinitesially different more or less.  We are more likely to see unexpected results if we write B-(C+D)>0 or MATCH(C+D,B,0).  But coincidentally, we are less likely to see unexpected results with B>C+D because the ">" operator compares the left and right values rounded to 15 significant digits, not their exact binary representations.
