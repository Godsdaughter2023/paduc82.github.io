george x3 say:

Hi Patrizia and Zune folk

The issue identified here is not present in Windows Media Player. Windows Media Player uses the appropriate API to prevent sleep from kicking in while playing audio. Zune.exe doesn't. Multimedia applications on Windows can (thankfully) prevent sleep while playing media.

The SetThreadExecutionState API is straightforward to use (one line to disable sleep, 3 lines to check the error code, and one line to re-enable sleep). In Win7/Server 2008 there is a new API PowerCreateRequest that appears to offer similar functionality to SetThreadExecutionState (though the documentation for the new API is spartan).

For consumers with machines in the home that use power settings, this makes Zune.exe second choice...except without Zune.exe consumers can't sync with their Windows Phone or take advantage of their Zune Pass. W have a PC in the kitchen and one in the living room and both are configured to sleep when inactive. I know I'd prefer Zune.exe to match Media Player's behavior, or offering the user a configurable option to choose whether to sleep when playing back audio.

I had many of the same concerns back in the Access 2.0 days, and began "hiding" queries by using a "Usys" appeneded to the start of the query name. i.e "MyQuery" became "UsysMyQuery". The only way to see them was to "View Hidden Objects".

From there, I evolved into building most of my queries in VBA behind the forms / reports, and hiding the database window. I figured that would discourage the casual users, at least.

As I said this was all in Access 2.0 and Access 97, where you had to buy a developer version of Access in order to make an MDE.

Since Access 2007, though ... anyone of us can make an ACCDE file and distribute our apps with a Runtime Version of Access which does not expose your code or allow design changes by the user.

daniel pineault say:

There are a number of things to do when developing a database.

  • You always split your database.
  • Password protect the back-end.
  • Convert your front-end to an accde.
  • Disable the shift bypass
  • Create an AutoKeys macro to disable certain key sequences.
  • Go through the database options to disable menus, ... to minimize user access and limit them to use your menu/ribbon instead.
  • Disable accessing the design option directly through your database objects.
  • Ensure you have proper error handling throughout all your VBA code to avoid user's being able to get into your code through the debug menu (converting to accde also resolves this issue) and also ensure proper error reporting by doing so.
  • ...

As for securing your data from accidental deletion, my application all have a user management system, either a formal login form or a hidden one that automatically recognizes the user's network username. and then associates the appropriate permissions for the forms based on the user's profile.

If you lock down your application and create a menu/ribbon and only allow your users to use those to access forms, reports, and stop them from having access to the navigation pane, you should be able to minimize any potential damage they can cause.

On the other hand, as with any system (not just Access), if someone wants to do harm, they will find a way.  This is where management comes in.  If someone can't be trusted, then they shouldn't have access to the data.

Also, by setting up a backup routine, you can minimize any potential problems which can occur naturally, or intentionally.

As Daniel said, there are many ways to help lock down your app. You can review my blog on Login Security for some hints there. Another tool is not to install a full Access license.
