Error: The name in the end tag of the element must match the - Microsoft Community

john korchok say:

Error: The name in the end tag of the element must match the element type in the start tag

This error is a common type of corruption in OpenXML Office documents. It almost always results from a particular error that Word makes when writing files. I'm still investigating the precise cause, but it seems to occur when different types of objects are anchored to the same paragraph mark. I've read confirmations that grouping text boxes with equations will cause this, since the grouped object shares a common anchor. But I'm also seeing it in documents that group photos with text boxes or that group autoshapes with photos. The XML error occurs at the end of the object group, at the point where it is anchored to the paragraph.

It always occurs within an <mc:AlternateContent> tag. The pages I've read about this tag are that it contains 2 subsections for presenting graphics. The first section within <mc:AlternateContent> is <mc:Choice>, which opens the preferred content. Apparently the <mc:Choice> contains content that may require newer features in Word for graphic desplay.

Following <mc:Choice> is the <mc:Fallback> tag, which is supposed to supply alternate content for older versions of Word. The XML bug occurs when there is no content in the <mc:Fallback> tag. When this happens, Word writes <w:pict/>. The backslash at the end of the tag means it is self-closing. It's a placeholder without any content.

This is when the bug happens. After writing <w:pict/>, Word takes a mental break and doesn't finish closing the rest of the tags that are open. When the user tries to reopen, they get the error message that is the title of this discussion. 

This XML error always takes the same form:

<mc:Fallback>
  <w:pict/>
</w:p>

And you can always solve it the same way, by adding in the bolded tags:

      <mc:Fallback>
        <w:pict/>
      </mc:Fallback>
    </mc:AlternateContent>
  </w:r>
</w:p>

Since I've discovered all this by looking at files that are already broken, I'm still trying to nail down exactly what actions lead to it. A reproducible bug is a nearly dead bug, or at least a potentially avoidable one.
