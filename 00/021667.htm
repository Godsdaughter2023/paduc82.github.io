Description of Word Doc Structure

“Why Master Documents corrupt “ Article contributed by John McGhie

http://www.mvps.org/word/FAQs/General/WhyMasterDocsCorrupt.htm

 

http://word.mvps.org/faqs/general/RecoverMasterDocs.htm

… a Word document is a great big “list” of objects. An object can be anything you can put in a Word document. Each of these objects has many, many “properties” that determine how it appears and how it behaves.

The properties are all contained in several giant “tables” inside the file. The connection between any given object (say, a paragraph) and its properties is made with an amazingly complex lattice-work of “pointers”. These pointers are large binary numbers that cause Word to look at an exact byte location in the file to see what shape, size, or colour this object should be. Most objects have more than one pointer. Some pointers go to “collections” of properties (for example, a “List Template” that describes all the formatting for a numbered list) and some go simply to a single entry (for example the “language” that is just a single name).

Whenever we experience a “Word document corruption”, what has actually happened is that the pointer, or the entry in the table it points to, has become corrupted. The information found there is either nonsense, or it does not apply to the object in question. For example, a paragraph is trying to inherit page margins: a paragraph cannot have page margins, so Word gets terribly confused.

All these property tables are stored in Section Breaks. A Section Break is not just a “page break”, it is a binary container that stores several hundred properties in multiple tables. The largest Section Break is the “Default” Section Break. You will never see one. The default Section Break hides in the very last paragraph mark of a document. Because it is absolutely essential to the document (without it, the file is just a stream of bytes, not a document) Word maintains the contents itself and hides it from you and me.

cove3 say: The problem is fixed.  It's very subtle because a hyperlink in a group header does not behave the same as a hyperlink in the detail.

1.  Assume hyperlink format is wik#restoflink## and is displayed in datasheets/reports as a field 3 characters wide to save space
2.  There's a property setting which I didn't notice called Is hyperlink.  This is set to no when you create a text box
3.  Despite being set to no, the hyperlink in the detail line shows wik or whatever abbrev no matter how wide it's sized.  That is, it correctly shows it as the 3 character name of the hyperlink.  The popup shows the full URL, and it correctly goes to the URL.
4.  However, in the Group Header, it does not behave this way.
    (a)  If the hyperlink is sized 3 characters wide, it does not show the popup and does not go to the URL
    (b)  If the field is sized wide enough, it does not shows  wik like in the detail, but shows wik#restoflink## but at least then it works
    (c)  However, if the Is hyperlink property setting for the Group Header hyperlink is set to yes, then only wik shows and the URL works
5.  In creating the report, I copied the hyperlink from the detail to the Group Header, changing the source, but not realizing I was carrying along the No setting
6.  The solution is to set Is Hyperlink to Yes for Group Header links, and despite not being required, do the same for the detail hyperlinks

Ron

hansv mvp say:

Try this:

 

Sub RemoveHeaders()
    Dim sec As Section
    Dim hdr As HeaderFooter
    For Each sec In ActiveDocument.Sections
        For Each hdr In sec.Headers
            If hdr.Exists Then
                hdr.Range.Delete
            End If
        Next hdr
    Next sec
End Sub
