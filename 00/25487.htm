is there any sis emulator to make files run on a pc?? - Microsoft Community

abhishekkai08 say:

is there any sis emulator to make files run on a pc??

is there any sis emulator to make files run on a PC?? if not, then if it is possible to create that?

petrib say: Well, yes and no and it depends.

The .sis file can contain UI themes or applications.

The themes you should be able to insall on the S60/Symbian emulator (which is in the S60 C++ SDK, Software Development Kit).

For applications/executables it won't work. The Symbian emulator expects programs to be compiled for the emulator (theyr're really Windows DLLs, Dynamic Link Libraries).

Programs that are compiled for target device are ARM process binaries, and run only on the phone. Currently there is no ARM emulator for Symbian programs that runs on a PC.

So, usually the easiest and most likely correct answer is: No, there is no PC emulator that can run .sis (or .sisx) files.

abhishekkai08 say: bt if we install the SDK on the pc then??
the programmers also code through pc nt on mobile itself, i hope :)

petrib say: Yes, you install the SDK on a PC and you run the emulator on the PC.

However, you can't run applications on the emulator, which are intended to be installed on the phone.

The developer makes a different version (build) of the application for the emulator, and do not generally distribute emulator builds at all (just the finished application for the device).

karl timmermans [outlook mvp 2012-2018] say:

Perhaps the easiest solution of all is to simply use one of the User fields (User1, User2, User3, User4) and enter the "exact" value that you want to use for your specific purpose in the merge. You are likely going to spend far more time trying to figure out a "merge" solution that works 100% of the time (questionable if 100% accuracy is even possible) then it would take to create the correct saluation for each contact item.

Outlook contacts by definition are intended as "single" entity items (one primary first and last name). If your issue is understood correctly, you have a <mixture> of formats which is going to be a contorted effort to try and parse in a merge (for instance, one example that you didn't include is where a person's first or last name consists of 2 words versus being an entry of <First Last> so would be impossible to determine if <Mary Jane> was strictly a first name or <First Last>. This is an over-simplified example but assure you that based on the names seen in contact files sent to us - there will be no shortage of "variations".
