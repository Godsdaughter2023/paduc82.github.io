ken sheridan say:

The reason I want to use a subform to use as criteria on the query is because I want to be able to select multiple data in a single column.

You don't necessarily need a subform for that.  Any form in continuous forms view allows you to do so; whether it’s a parent form or a subform is immaterial.  The only situation in which a subform is required is where it is in a bound form, in which case its recordset is normally restricted to those rows which reference the current row in the parent form, which is in most cases achieved by setting the LinkMasterFields and LinkChildFields controls to the primary and foreign keys of the parent form's and subform's recordsets respectively.  It can also be done by referencing the parent form's primary key as a parameter in the subform's query, but this is usually only done when correlating two or more continuous subforms.

Where a continuous form's recordset needs to be restricted on the basis of values entered or selected in one or more unbound controls, while a subform can be used, there is no actual need to do so, as a single form in continuous forms view can have its recordset restricted to those rows which match values in unbound controls in the form header.  It can do so by referencing those controls as parameters, or by filtering the form in the AfterUpdate event procedures of the unbound controls.

All of a sudden, all of the update queries of both kinds started timing out.  After a full day of troubleshooting why my update queries kept hanging, I discovered that if I limited the number of rows in the listbox to <65,535 (SELECT TOP 65533), the queries suddenly ran as expected.

What does the listbox have to do with your update queries? Can you provide some SQL.or code?

d:  The listbox displays a "SELECT * FROM [TableName]" pass-thru query.  Both update queries update a field on the displayed table "UPDATE [TableName] SET [DateModified]='1/1/1900'".  Otherwise, there is no issue.  BOTH update methods timeout if the listbox tries to hold more than the 65,535 record limit.  All I had to do was change the listbox rowsource to "SELECT TOP 65533 * FROM [TableName]" and both update methods completed normally and in the expected time.
