One way to do it would be to create a class as follows (C#):

}

PeterL,

Actually I'm going to tell you that there is nothing wrong with using Project 2003 if it meets your needs and you do not need to interface with other companies or users who have a more current version. The biggest disadvantage, other than more whistles and bells added in later versions, is support. Microsoft stopped updating Project 2003 long ago so you are on your own with any latent bugs that exist, and for sure they do exist.

You say you are using a resource pool but not a master file and infer that will ward off any linked structure problems. A resource pool/sharer file structure is no different from a master/subproject structure. They are both dynamically linked structures and therefore must be managed with strict discipline. The files in the structure must never be moved, renamed, overwritten, or saved off to a different location. Those are the hard points. Ideally, all files should reside on a local drive in a single directory and not be operated over a network. These are the "softer" points.

Assuming none of the users violated the hard points, you did mention a network and if I had to guess I'd say that may be the source of the problem. Any glitch in network operation could easily "drop a link" between the pool and sharer file.

If you cannot live without using a network for your files, then here is an approach that may work better for you. Assuming resource pool updates are infrequent (i.e. not daily), you could designate a particular time (e.g. weekend or off hours) for pool updates. After the update from Excel, update each sharer file with the new resource/calendar information and then unlink the sharer file from the pool. The link and unlink process is critical in that it must be done correctly to avoid creating link fragments. Each time a link or unlink process is performed, BOTH files must be saved to preserve the link structure. Note, this whole process can be automated with VBA.

Okay, if that's a little too drastic this is more of a band aid approach but it might be all you need. Create an audit process (VBA) that detects when a sharer file is "dumped" from the pool and then re-establishes the link if necessary.

Just some suggestions, hope it helps.

John

john korchok say:

Here's a technique that works with both legacy Form Fields and Content Controls:

  1. In Word, choose File, then Options.
  2. Select Display.
  3. Under Page display options, check Show document tooltips on hover.
  4. If you're using Content Controls, select the content control and use Insert>Bookmark to open the Insert Bookmark dialog. Insert a bookmark.
    If you're using legacy Form Fields, the field already has a bookmark.
  5. Check that the Content Control or Form Field is selected.
  6. Choose Insert>Hyperlink and click on Place in This Document.
  7. In the center panel, you'll see the bookmark for the field or control. Select it.
  8. Click on the ScreenTip... button.
  9. Enter your text prompt in the ScreenTip text field. OK out.
  10. If you're using a Content Control, click in the Text to display field and enter the text you want to display inside the content control.
  11. Protect the form.

When you float your mouse over each field, the screentip appears and the first line displays your prompt text. For all computers using these forms, the display option must be set as described in the first 3 steps.
