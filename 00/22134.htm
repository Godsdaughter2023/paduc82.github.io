albert d. kallal say:

I’ll add a few more things.

The web pages and front end web parts resides on SharePoint. The data store resides in SQL server, and that instance of SQL server is not part of SharePoint nor is the SQL part running on SharePoint (it runs on SQL Azure which is a good thing). So no matter how you slice and dice this, while you are using a low cost 365 account, you web forms are running on SharePoint.

End users have access to the front end in real time

Well, in theory, if the web forms achieve what you need then end users would not need Access front ends, would they? However since no reporting features exist for Access web, then for reports of any kind of complexity, then deploying a Access front end to each of the windows desktop PC would be required. Users could also have the full edition of Access desktop. In both cases the Access front end only runs on a windows desktop (so no mobile users etc.). So tablets, phones etc. can use the web forms, but they would not have reporting since these devices don’t support or allow you to install Access on them.

Each user has log-in privileges and only access to their data (I believe it does this, but each user requires their own SharePoint Level 1 access for $5 / month?)

Each user will require a logon. However, you can use one paid $5 account and user, and that user can then invite up to 500 users for free. So users with a Gmail or other non-Microsoft account can use that web site. (they have to associate that Gmail or non-Microsoft account with a live ID). So the logon process is the same for 365 accounts, or the 500 free accounts that can be any email type of user – including non 365 accounts. So you only need the one $5 user + account – the rest can be invited for free.

and only access to their data

The above issue is SO VERY often overlooked!

That not how Access or most systems work. Think of Access desktop. Because you have 5 or 10 users and they open a form attached to a table, then all users can simply edit data in that form. You the developer have to “restrict” the data to each user. So how you do this in Access desktop, or Access web, or any web development system is the SAME. So just because you jump to web does not out of the blue make that form attached to a table of data by some “magic” restrict data in that form to some particular user. All users can edit all data.

Restricting data is NOT part of the database system or the web system just because it is web based. If data is to be restricted to individual users, then you have to design that into the system. The security model for Access web is rather poor at this ability. After all the idea is to allow an easy way build Access web forms. Those forms edit all data in that table.

And even if you did manage to restrict data in those web forms to a specific user, the deployed Access front ends would be able to edit all data in a given table + form. So nothing in the system will “automatic” restrict the data to an individual user.

So if your goal is to have Access form and edit data, you are fine. However a VERY common scenario often required that data and customer information is restricted to EACH customer (the logged on user). The “goal” of restricting records and data to each user is HUGE different kind of application! You don't want to overlook this issue. If this "sepeate" customer data is a requirement in which each user of the application ONLY sees their own data, then Access desktop (or Access web) is likely the wrong tool.

- Lastly, I need the ability to have a coder integrate it with some existing external software

You really cannot integrate 3d party software into Access web forms. So customizing and using features beyond the limited Access web forms is really not much practical.

I suspect coding in vb.net + asp.net which at least allows one to build WEB forms with vb.net code (very much like VBA) in those web forms, and allows high degree of customizing, and also allows the all-important ability to restrict data to each individual user is a better choice here.
