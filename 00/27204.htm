don cuthbert say:

The Microsoft Software License Terms for stand-alone software allow you to transfer the software to any other device, even if that device does not belong to you.

Linking the Windows 10 license to your Microsoft Account is separate from linking it to a specific physical or virtual device. Your license can be linked to a specific device even if you never login with a Microsoft account on that device. Linking your Digital License for Windows 10 to your Microsoft Account can help to activate Windows after a significant hardware change, such as replacing a motherboard. There is no limit to the number of Windows licenses that you can have linked to the same Microsoft Account.

Most Windows 10 licenses that were upgrades from Windows 7 or 8 have the same generic Product Key (one for Home Edition and another for Pro). Having the same Product Key does not mean that the devices are sharing the same license. Each activated Windows 10 device, whether virtual or physical has its own Digital License for that Edition of Windows 10.

So, if you have four Windows 10 VMs and one Bootcamp install, all activated with the same Edition of Windows 10 (Pro or Home) it means that you have 5 Digital Licenses for Windows 10 (assuming that Bootcamp means a separate install of Windows 10).

Because your ID here is tagged as an Insider, you may have picked up some Digital Licenses through the original Windows Insider Program, during the time when Windows 10 devices could become activated with a Product Key, even if they were never licensed for Windows 7 or 8. A newly created Virtual Machine, for example, needs its own license for Windows 10. During the original Windows Insider period, it was possible to activate such a VM with a Windows 10 Insider Preview Product Key. Now that is no longer possible.

So that means that you should not delete any Virtual Machines (which are usually very tiny files, not including the virtual disk) that are licensed, as each License could be said to represent a value of about $US200.

In November, 2015, it became possible to activate Windows 10 clean installs using a Windows 7 or 8 Product Key. I believe that only worked if the Product Key had at one time been activated on that same VM or physical computer, but perhaps it also worked if the Windows 7 or 8 Product Key had never been activated on any other device. So it is possible that Windows 10 might be activated with a Windows 7 or 8 Product Key. However, most upgrades only show the same generic Product Key. I don't know if a Windows 10 computer that was activated with a Windows 7 or 8 Product Key retains that old Product Key or not, as I have never personally seen this done. The Product Key reader that you have should show if this is the case or not.

So it may be impossible to tell which of your VM's or devices actually now has the activation that came from your 2 Windows 7 Retail stand-alone licenses, unless they actually now show the Windows 7 Product Key or if you are able to trace the use of the Windows 7 Product Keys in some manner through your own records.

I wonder what would happen if you try to install Windows 7 to a new physical device or VM using those original Windows 7 Product Keys? Could you transfer the Windows 7 license without compromising any of the other VM's activation status? If the VM's do not show the Windows 7 Product Keys, that may be the only way to know which VM's are activated -- based on those original Windows 7 licenses (i.e. if activating Windows 7 on a new device caused a VM to lose activation).

GetSaveAsFilename Mac Excel 2011 & 2004, and Win Excel 2000 thru - Microsoft Community

rjbayha say:

GetSaveAsFilename Mac Excel 2011 & 2004, and Win Excel 2000 thru 2010

The programs I have developed are required to work in Windows Excel 2000 thru 2010. The programs must also work in Mac Excel 2004 and 2011. Finally, the programs must work on Windows computers, and Mac PowerPC and Intel based computers.

 

The problem is GetSaveAsFilename(initial file name, FileFilter:=, Title:=, ButtonText:=) does not work as expected-documented in Mac Excel as it does in Windows Excel. With the exception of the initial file name, Mac Excel ignores (does not use) FileFilter:=, Title:=, and ButtonText:=; even though, the parameters are specified.

 

In my case, I want to force the user to save the workbook in use as an XLS, pre-2007 workbook, and the workbook being saved should not be saved to the TEMPLATE folder. For Mac, the user’s Template folder is general contained within the user’s hidden Library folder.

 

The following procedure is an example of how I handled GetSaveAsFilename issue while meeting my requirements – it works for me. Please let me know if there is a better way of handling the issue.

 

 

Sub SaveAs_File()

 

   ‘variables defined-declared

   Dim mThisFile As String    ‘ for the current path and workbook name

   Dim mFileSave As String    ‘ for the desired path and workbook name

   Dim mFileType As Integer    ‘ for the file type enumeration

   Dim mFileLength As Integer   ‘for the number of characters in mFileSave

   Dim mFileExt As Integer   ‘for determining the location of the “.” preceeding the file extension

   

   'bring up saveas dialog box

    If Left(Application.OperatingSystem, 3) = "Mac" Then

    mThisFile = ActiveWorkbook.Path + ":" + ActiveWorkbook.Name

    mFileSave = Application.GetSaveAsFilename _

        (mThisFile, FileFilter:="XLS8", Title:="Save Movement Table", _

        ButtonText:="Save Mvmt-Tbl")

    Else

    mThisFile = ActiveWorkbook.Path + "\" + ActiveWorkbook.Name

    mFileSave = Application.GetSaveAsFilename _

        (mThisFile, FileFilter:=" Excel Workbook 2000-2004 File " + _

        "Format (*.xls), *.xls", Title:="Save Movement Table", _

        ButtonText:="Save Mvmt-Tbl")

    End If

   

    'make sure file is not saved to template folder

Debugging is as much an art as it is a science.  And we do not know how skillful you are.

Some problems are difficult to debug when we are sitting there at the computer.  Many problems are nearly impossible to debug at arm's length.  At the very least, you need to show us all of the relevant code.  "Relevant' is necessarily a subjective term; but certainly it includes all of the type declaration statements (Dim, ReDim, etc) as well as functional statements.

From your description, I would suspect something environmental that is influencing the behavior.  But I would not rule out unexpected code behavior due to unintended and improper values of variables.

Look at the Regional and Language Options control panel on at least two computers, one where the failure occurs and one where it does not.  You might also look at Task Manager to see if the "failing" computer is stressed more than the others.

But if the problems fails intermittently with the same data and code, I would lean toward "uninitialized" (or poorly-valued) variables or timing issues.

Unfortunately, I do not know how to work with an SQL DB using VBA.  But if that relies on another thread or process, perhaps you need to be sure that some interprocess transfer of data is completed before proceeding.

Alternatively (and more likely), perhaps the random value of some "uninitialized" variables causes unpredictable behavior of your code.

Again, sight-unseen, the possibilities are endless, and we can easily mislead you down a rat hole.

When I suggested using On Error, it was not just to determine what field caused the problem.  The important thing is:  look at all values at the time of the error, using Print in the Immediate Window for example.  Challenge your assumptions about what things are; test everything.

One off-the-wall thought:  change as type Date variables to type Double; and change any references to range.Value to range.Value2.

Sometimes VBA's special interpretation of dates and other "types" (formats) of Excel data cause problems for me.

Frankly, I would expect that any such problems would be consistent, not intermittent as you describe.  But again (and again and again ;->), without the benefit of seeing the code, really anything is possible.
