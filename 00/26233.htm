How to determine which Outlook.com server you are on? - Microsoft Community

a. user say:

How to determine which Outlook.com server you are on?

How can an Outlook.com user know which server they are on?

 

a. user say:

Outlook.com hosts numerous servers and user accounts are spread across different servers.

 

If you are able to sign in and are not encountering any login error, please follow these steps to know the server that hosts your account:

 

1. Log into account

2. Hover over the "Inbox" button

3. In the bottom of the browser in the status bar it'll show the server URL that we're familiar with:

Example: http://bl113w.blu113.mail.live.com/mail/Inbox...

In this example, blu113 is the server name.

 

Note: If you do not see a server name in the status bar when you hover over the inbox or any other folder, please point your mouse pointer to any of the emails in the Inbox, and you should be able to the server info in status bar on the bottom left hand corner. 

 

If you can't sign in to the account and encounter an error message because the server is down:

 

1. Click on this link: http://mail.live.com/mail/browsersupport.aspx

2. Input your Hotmail credentials if it prompts you to

3. You'll be taken to a default error page - hover over the "continue to Outlook" link

4. In the bottom of the browser in the left corner, the status bar will show the server URL that we're familiar with:

Dim DocSource As Document, DocTarget As Document

TryAgain:

Set DocSource = ActiveDocument

With DocSource

        i = i + 1

   Loop

What are objects starting with dt_ in an Access Project - Microsoft Community

alisida say:

What are objects starting with dt_ in an Access Project

We are using an old Access Projet that was created by an intern back in 2000. I am trying to make improvements to the setup but I am not ery familiar with Access projects. There are a tone of objects listed in the Queries menu and I would like to clean it up but soem of them I don't know what they are. There names all start with dt_ except one that has fn_. No one at work knows what they are or has ever used them. If I look at their properties they are Stored Procedures. If I open them in design view nothing shows. I need to know what they are and how to determine what they do.

Any help would be great.

 

 

albert d. kallal say:

The lists of objects starting with dt_ are usually the long list of server side "utility" store procedures that gets created by the client tool (in this case Access).  In fact I believe historically that prefix means "DaVinci Tools".

Remember, the SQL editor in an Access Data Project (ADP) is DIRECTLY editing SQL on the server side – you not only have full SQL server syntax and functions, but also the ability to write + edit T-SQL (store procedures).

So all those dt_ objects are simply "utility" functions that the client (Access) created to achieve a common set of functions that the client can use.

So you see common routines like
dt_adduserobject

Or

dt_checkinobject

And if I recall correctly, I think even a set of routines was uploaded to support diagramming in the Access client (the relational diagrams for an Access Data Project are stored SERVER side).

So the routines are not directly used by your application, but they are and have been created by the client tools for correct operation of features that Access requires to use with the server side.

Last but not least, since Visual Studio and other products ALSO can create those objects, then it possible that many of the objects were not created by the Access client program, but some other system that was also used during the development process with SQL server.

At the end of the day?

I would not worry about these objects - they are as a rule not directly used by your code and application, but I certainly would not go about deleting them either.

So it is unlikely that these objects where created by your developer – they are "client tool" generated objects in most cases.

Last but not least, the dt_ prefix is often used for date type of objects in SQL server. So they could be a set of store procedures that are for handling dates, but I think the best guess is those objects are "generated" store procedures from the client tools and the dt_ prefix was common for that occurrence. I think a quick "glance" at the types of names for those sprocs would quite much hint if they are system generated such as:

 

dt_adduserobject

or

dt_PayrollWeekStart

 

>No one at work knows what they are or has ever used them

 

You should be able to view them (you often need admin permissions), but as noted, they are not likly used direclty by your code.
