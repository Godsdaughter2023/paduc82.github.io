Date in MS Access Reverses in the Form - Microsoft Community

patmurray_1690 say:

Date in MS Access Reverses in the Form

The date 01/10/2016 shows up correctly in my table and query.  However, when I type it into the form, it reverses the day and month to 10/01/2016.  How can I correct it?  I have tried the following: selecting "short date;" typing "mm/dd/yyyy".  Both work fine in the table and in the query, but changes every time in the form.

scottgem say:
You have to look at a) how the control on the form is formatted and b) what your regional settings are. Access normally picks up the default date format from the Windows Regional Settings. 

ken sheridan say:

It sounds like your regional short date format could be set to UK format of dd/mm/yyyy rather than the US format of mm/dd/yyyy, so when you think you are entering 10th January, you are in fact entering 1st October.  If this is the case the dates in the tables will be incorrect.  Changing the system date format to mm/dd/yyyy will be necessary.

Any dates in tables which you've entered in US format while the system format was incorrectly set will need changing after you change the regional settings, which you can do with an update query like this:

UPDATE YourTable
SET YourDateField = CDate(Format(YourDateField,"dd/mm/yyyy"))

WHERE YourDateField IS NOT  NULL;

Note that if any dates were entered in a format valid in the US, but invalid in UK format, e.g. 07/13/2015, these will have been inserted into the table as the correct date because Access will have detected the invalidity of the date and corrected it.  These will also be preserved as the correct dates after executing the above query.

If you have dates in a table, some of which were entered when the system short date format was set to US format, and some of which were entered when the system short date format was set to UK format, then you have the problem of determining which need to be corrected and which don't.

PS: Updating to Windows 10 changes the language and regional settings of UK machines to US formats, but I've not heard of the reverse happening.

patmurray_1690 say: Thank you for your responses.  I was finally successful.  I changed the date field in the table to "medium," meaning 10-Jan-16, then deleted the field from the form, dragged it back in, and the date appeared properly in the form as well as table and query.  I think deleting the field from the form, then putting it back in was the fix.

scottgem say:

You don't have fields on a form. You have Controls that may or may not be bound to fields in a table. This is a subtle but important distinction. Controls have their own set of properties. When you drag a field from the list of Available fields onto a form in Design Mode it retains the properties from the form. However, once the Control is created its properties are not synchronized with the field's properties in the table. So the probability here is that somewhere along the line, the properties of the control were changed. By deleting the control and then recreating it by dragging it from the table, you essentially changed the properties of the control to match what was in the table.

Website made with Publisher 2007 - image filenames are changed to - Microsoft Community

krwinch say:

Website made with Publisher 2007 - image filenames are changed to default names

I have built a simple website using Publisher 2007. after I publish the site, the images are saved in the "index_files" folder and they are all renamed to image####.jpg instead of the original filename. How can I save these images and retain the original names?

davidf2 say:

You can't if you embed the images into the page before you Publish to the web and produce the index_files folder. In fact if you have embedded an image that is larger than the picture box, Publisher will make two copies of the picture; one for IE and one for all other browsers.

 

You can observe this if you look at the contents of the index_files folder in thumbnail view. You will have one image file named: image####.jpg and another named image####1.jpg. Note that the first image retains the exact dimensions and resolution of the orginal image and is generallly a much larger file, and takes a long time to load in IE. The second image is the one for non Internet Explorer browsers and note that it has been resample and sized to the exact dimensions of the picture box at 96 ppi. It is a much smaller, web optimized file and loads much more quickly.

 

You can avoid the two copies by compressing all the images before you produce your web files. Right click one of the embedded images > format picture > picture tab > compress and choose the 'Web' target output (96ppi). You also have the option to apply to all the images in your publication; you can resample them all at one time.

 

I suspect that you want to retain the image name for SEO reasons. You can get much the same effect if you include 'Alt text' with each image. Once again right click each image > Web tab > and add the alternative text to the image. You can read the primary reasons for the alt text, on that tab, but in addtion the search engines will crawl, read and index this alt text. In fact by using Alt Text you can add much more than just a file name worth of information.

 

And with all that said you can import the images under the original name if you want. Create a subfolder on your web server at the same level as the index_files folder called 'images'. Resample, size and optimize all the images you want to import into your web page in a third party image editor and upload them to that 'images' folder. Then using the insert > html code fragment tool and insert a verison of this code fragment:

 

<IMG SRC="http://yourdomainname.com/images/yournamedimage.jpg" ALT="the alt text you want to add to the image" border="0" width="120" height="60">

 

The height and width are to correspond with the resized image to fit exactly that size in your page. Make your code fragment box that size for layout purposes. Personally I don't think it is worth the effort, but it can be done.

Matching Title metadate to File Name - Microsoft Community

corinos say:

Matching Title metadate to File Name

My question stems from my recent adoption of Sharepoint as a document tracking tool.  The Search feature in Sharepoint wasn't bringing up the results I was expecting and some research leads me to believe that it functions, by default, using document Titles, not File Names.  Many of my documents had titles unrelated to the documents actual purpose due to some issues with Template use, but that has been resolved with one exception.

When I save a word document in Word 2013 after creating it from a template, the Title field in the metadata is blank.

I would like that field to include the file name, or, if it must, the first line of the document.

NOTE: I can manually add the title when I save the file, but I don't want to have to.  It would like it taken by default.

jay freedman say:
This can be achieved with an event handler in a global template (add-in).

Create a new document and save it as a macro-enabled template (*.dotm) file, temporarily in your Templates folder.

Open the macro editor, and insert a module and a class module. Using the Properties pane, rename the module as AppEvents and rename the class module as ThisApplication.

In the module, paste this code:

Dim oAppClass As New ThisApplication

Public Sub AutoExec()
    Set oAppClass.oApp = Word.Application
End Sub

In the class module, paste this code:

Public WithEvents oApp As Word.Application

Private Sub oApp_DocumentBeforeClose(ByVal Doc As Document, Cancel As Boolean)
    With ActiveDocument
        If .Type <> wdTypeDocument _
           Or .BuiltInDocumentProperties(wdPropertyTitle) = .Name _
              Then Exit Sub
       
        If .Path = "" Then
            .BuiltInDocumentProperties(wdPropertyTitle) = .Range.Sentences(1)
        Else
            .BuiltInDocumentProperties(wdPropertyTitle) = .Name
        End If
       
        .Saved = False ' force a save
    End With
End Sub


Then save the template, and copy it to the folder %appdata%\Microsoft\Word\Startup. Templates stored in that folder are global; they are loaded whenever Word starts up. Restart Word to make the code active.

When you create a new document (and presumably save it one or more times while working on it), the Title property will be empty. When you close the document or shut down Word, the code in the add-in will run. If the Title has already been changed to the file's name, the code will do nothing. If the document has never been saved, the .Path will be empty, so the Title will be set to the first sentence (Word doesn't really know what a "line" is!). If the file has been saved before, but the Title is different from the file name, then the Title will be changed to the file name and the document will be saved again.
