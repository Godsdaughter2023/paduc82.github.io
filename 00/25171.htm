eliuri,

In an earlier thread you started you'd complained about issues with removing the McAfee antivirus product originally installed on your PC as I recall.  This is one of the most common reasons for problems with both re-enabling Windows Defender as well as the stability of its operation.

If you have not gone to the McAfee support website and found their removal program, you should do that and run its installation as soon as possible., making sure to restart the PC both before the installation and after, whether the instructions tell you to do this or not.

Partially installed 3rd-party security applications leaving remnants behind are an infamous problem with any Windows system and are the most common reason for issues like those you've described in both this and that earlier thread.

The other thing you need to realize is that the Microsoft Controlled Folder Access is actually designed to block the outdated installation methods used not only by ransomware, but also quite a few older Win32 applications.

In most cases of reputable programs like Malwarebytes Anti-Malware (MBAM), this only interferes with the installation of things like a program icon in the Desktop folder, which is no longer considered best practice and so the Desktop is a CFA protected folder.  I've installed and upgraded MBAM myself on Windows 10 and the only thing it's ever blocked was this desktop icon.

Lastly, the simplest workaround to problems like the blocking of icon installation is to temporarily disable the CFA protection when installing a new trusted application, then immediately re-enable CFA once the install is completed.  Since CFA is designed to block this very action, it will ever be "fixed" by Microsoft, since it's the act of installing a desktop icon that's actually "broken".

So in most cases, the problems with CFA are actually misunderstanding of its purpose and function by the consumer, which is exactly the same type of consumer confusion that allows the ransomware to socially engineer the PC owner into installing these malicious programs in the first place.

scottgem say:

One of the principles of a relational database is that data should exist in ONE place only. So the site name should ONLY be in the Sites table. In your Artifacts table, you have a field named SiteID that is the foreign key linking the Artifacts table with the Site table. When you add an artifact to your database, you should have a Combobox on the form where you can select the Site where the artifact was found. This comobox can be set to allow you to select from the site names, but to store the SiteID. The combobox wizard will do this by default.

If you have been entering artifacts without setting the foreign key, then you would need to run an Update query that populates your SiteID foreign key with the correct ID based on the stored sitename. Then delete the Sitename field from the Artifiacts table as it will no longer be needed.

Hope this helps, Scott<> P.S. Please post a response to let us know whether our answer helped or not. Microsoft Access MVP 2009 Author: Microsoft Office Access 2007 VBA Technical Editor for: Special Edition Using Microsoft Access 2007 and Access 2007 Forms, Reports and Queries

john w. vinson say:

I'll have to agree with ScottGem here: your table design is incorrect!

Relational databases use the "Grandmother's Pantry Principle": "A place - one place! - for everything, everything in its place". The SiteName should simply notexist in the Artifacts table, only in the Sites table. You only need the SiteID in the Artifacts table.

If you're entering data (site names for example) directly into the Artifacts table datasheet... don't. Tables are designed for data storage, not for editing or data entry! Create a Form based on the Artifacts table; on the form you can put a Combo Box based on the sites table, bound to the SiteID field. This will let the user select a site name from a list (avoiding typos, spelling errors, and excessive creativity on the user's part regarding site names); the user sees the human-meaningful site name, the computer sees the computer-meaningful SiteID, and they are both happy.

That's means, when you add a domain of particular organization (please note that domain is just part after @ - in your example domain is just contoso.com), all your users will be able to communicate with that organization using Skype for Business.

So for example, if you have users: 

  • ex1@contoso.com
  • ex2@contoso.com

and you want to let them to communicate just with users with @microsoft.com email addresses via Skype for business, you need to add this domain to the list with On only for allowed domains settings. You can add more domains to that list.

If you set external access to On except for blocked domains, without adding domains to the list, your users will be able to communicate with all domains. If you add some domains to the list, the users will be able to communicate with all domains except the domains on the list. 

Regarding the second questions, if the communication is enabled it will work both ways.

I hope this helps. If you need additional information, please let me know what kind of settings you want to have for your organization and I'll try to help you configure it.
