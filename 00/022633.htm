To answer your questions, assuming you are using MFA in cloud,

Q1: Is there a way to force the users to only be able to choose Microsoft Authenticator and NOT SMS/Message or Phone call as verification?

Yes, here are the steps:

  1. Sign in to the Azure portal.
  2. On the left, select Azure Active Directory > Users > All users.
  3. Select Multi-Factor Authentication.
  4. Under Multi-Factor Authentication, select service settings.

  5. On the Service Settings page, under verification options, select/unselect the methods to provide to your users.


  6. Click Save.

Q2: Is there a way for me to see on the users who already has 2 factor authentication if they used Message or Authenticator when they set it up?

Yes for users who have set up MFA.

  1. Sign in to the Azure portal.
  2. On the left, select Azure Active Directory > Users > All users.
  3. Select Multi-Factor Authentication.
  4. Under Multi-Factor  Authentication, select users > Enforced for Multi-Factor Auth status.

Q3: Is there an option to make all new users to be forced to use 2 factor authentication? 
No, I'd like to invite you to vote for this similar idea via Office 365 admin User Voice. It is the direct way for the related team to know your requirements.

rich michaels - mvp (greatcirclelearning.com) say:
IMO a twenty page document is not a large document.  Documents we work with are usually 200 + pages in size and have many tables, graphics, etc and we don't have the performance issue you're experiencing. Maybe the difference is Tracked Changes because we try to use Document Compare instead and when there are documents with Tracked Changes sent to us, we accept or reject them and start fresh.

I think the performance issue comes when the Tracked Changes get stacked upon each other.  i.e. your colleague makes a change and then you make another change to the change, etc. and never is there a fresh starting point.

Take a look at the Document Compare feature, it's on the Review Tab, and see if you can fit that into your revision work process with your colleague.

A couple of final thoughts.
  1. Hopefully after you repaired the disk permissions you restarted your system that insures all adjustments are now stored in memory.
  2. Does this performance problem happen on all documents or just certain ones?  You might try experimenting between creating a "doc" format document versus a "docx" format and checking the speed of editing using your existing work process. The "doc" format should be faster but then you lose the ability to use some new features of Word.
There are always trade offs :-)

rich michaels - mvp (greatcirclelearning.com) say:

Performance issues can be caused by any number of the following:

  1. Duplicate fonts.  See Font Weeding
  2. Disk Permissions. See Repair Permissions
  3. Unresolved Tracked Changes
  4. AutoRecovery time set to less than 10 minutes
  5. Time Machine and VMWare 4.0 conflict when the VM partition is included in the backup
  6. Badly fragmented memory ... reboot the system
  7. Not working locally ... editting files from shared drives or drop boxes
  8. Not being current with updates on the software

There are probably more but those are the ones that first came to mind
