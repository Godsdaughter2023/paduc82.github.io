greg carmack - mvp 2010-2020 say:

You posted in the comments for Post a screenshot of Disk Mgmt

Here is my reply:

Any drive marked Active is going to cause the BIOS/UEFI to stop to query it for bootable System files since the Active flag only exists to point to a bootable partition.  It is always wrong to have any other than the partition actually booting Windows marked as Active in a Legacy MBR install and needs to be corrected because it will also almost always interfere at some point. 

In this case D is marked Active but doesn' t appear to be bootable itself so should be marked Inactive:  Partition - Mark as Inactive - Windows 7 Forums

Then Disk 4 has an apparently abandoned EFI System partition which is only intended to boot a UEFI system drive which is formatted GPT.  If you don't have a UEFI bootable OS on the drive then the drive should have it's files moved off to wipe it of all code with Diskpart Clean Command or at the minimum deleting all partitions to clear the partition table (which doesn't remove old boot code however).  Then reformat as a data drive.

Finally the apparent actual OS drive is Disk5 which we know is the booted OS because of the presence of the Boot flag, and boots itself from C drive itself as evidenced by the System and Active flags on C.  This is OK but not optimal.  First, the drive with the bootable OS will boot faster if it's the first drive in order because all those other drives are causing it to pause to query each, and spend extra time querying the errant Active drive D, and EFI System drive if this is a UEFI system that's set to also boot legacy drives.  Second, if this is a UEFI system using the new replacement for BIOS the OS should optimally be installed to UEFI and not Legacy to take advantage of enhanced drive security, boot level mouse support and other features of the modern UEFI firmware.  If this isn't a EFI system then the presence of a EFI System partition on a drive is just sloppy and wrong.

I'm not going to debate whether these clearly wrong configurations are causing your present problem, but in any case they are wrong and would be corrected if there is any concern for Best Practices.  Their present condition would also be expected to cause problems anyway.

lonnie_l say:

Microsoft is aware of reports that the Absolute Volume feature is incompatible with some Bluetooth devices, resulting in the inability to adjust the Bluetooth device’s volume.  If you are experiencing this issue, neither the volume slider in the taskbar, nor the device’s volume controls have any effect on volume.

To resolve this issue, first make sure you’ve installed the latest Windows Updates.  A solution was included in the To check for updates, select the Start button, and then go to Settings> Update & security> Windows Update, and select Check for updates.

Three devices that are known to experience the issue will automatically have the Absolute Volume feature disabled and should no longer experience the issue once you’ve installed KB4462933:  Creative T30 Wireless Speakers, Edifier R2000 DB, and the JBL Go.

For other Bluetooth devices, once you’ve installed KB4462933 or once you've installed Windows 10 version 1809, you can resolve the issue by disabling Absolute Volume manually by making a change in the registry.  (Warning:  Serious problems might occur if you modify the registry incorrectly. For added protection, back up the registry before attempting to modify it.)

To disable the Absolute Volume feature manually:

1.     In the Search box, type cmd.  Right-click Command Prompt in the Search results and select Run as administrator.

2.     In the Command Prompt window, type the following command:

reg add HKLM\SYSTEM\ControlSet001\Control\Bluetooth\Audio\AVRCP\CT /v DisableAbsoluteVolume /t REG_DWORD /d 1 /f

To re-enable the Absolute Volume feature:

1.     In the Search box, type cmd.  Right-click Command Prompt in the Search results and select Run as administrator.

2.     In the Command Prompt window, type the following command:

reg add HKLM\SYSTEM\ControlSet001\Control\Bluetooth\Audio\AVRCP\CT /v DisableAbsoluteVolume /t REG_DWORD /d 0 /f

You may need to restart Windows for the registry change to take effect.

techfreak_ say:

Skype does not provide any automated migration of your local chat history (main.db) from the Desktop version to the App version, you would have to do that manually.  If you do not, you will only receive the last ~30 days of cloud history.

Your Desktop history is located in your Skype name folders located in %appdata%\Skype

Your App history is located in your Skype name folders located in %localappdata%\Packages\Microsoft.SkypeApp_kzf8qxf38zg5c\LocalState\

You can transplant the files yourself or you can use the following program I made, if you find it more convenient. 

CopyDB (.NET 2.0 version) [pcdust.com] - link removed.

If you get a complaint about needing Framework 3.5 or older (and don't want to take the time to install it), run the below version instead:

CopyDB  (.NET 4.0 version) [pcdust.com] - link removed.

This program currently does not merge the contents of files.  It simply overwrites them.  Be aware that you need to sign into your account at least once (if you have multiple) in the Skype App.  If you do not, there will be no location to copy the files to.  This same rule applies when copying the files over manually.

Be aware that the newer Skype applications have changed heavily since this program was initially created (skype.db and leveldb data). It will not function in a usable way in regards to the latest versions of Skype. The links have been removed.
