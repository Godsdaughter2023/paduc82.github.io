Hi,

Assuming that the 2 leftmost letters in D are the ones that you want to match in column A.

  1. Select the range D1:E6
  2. In the Home tab, conditional formatting
  3. New rule
  4. Use a formula to determine which cells to format
  5. Enter this formula: =MATCH(LEFT($D1,2),$A$1:$A$100,0)

However, a better model would be to have a single Employees form, which includes all employees, including managers, and to include a foreign key ManagerID column which references the EmployeeID primary key of the same table.  An enforced relationship on these columns can then be created between two instances of the table.  This will allow you to model the complete management hierarchy from the CEO down to the most lowly employees in the organisation.

It may well be that you would want to record other attributes of some or all managers which do not apply to other employees.  This would be done by creating a further table, Managers, with a primary key ManagerID of long integer data type (NB not an autonumber).  This would also be a foreign key referencing the primary key EmployeeID of Employees.  The Managers table is consequently modelling a sub-type of Employees and would have non-key columns representing those attributes relevant only to managers.

Data entry for all employees would be via a form as described above.  Where an employee is being inserted as a manager, or there is the need to edit the manager attributes of an employee for whom a row already exists in the Managers table, a command button could be included in the form, whose code would (a) save the current record so as to allow a row to be inserted into the referencing Managers table, and (b) open a form bound to the Managers table in dialogue mode, filtered to the current EmployeeID value, passing the EmployeeID value to the managers form as the OpenArgs argument of the OpenForm method.  Code in the managers form's Open event procedure would then assign the value of its OpenArgs property to the DefaultValue property of a bound ManagerID control.

While it is possible in some scenarios, as a general rule data should not be inserted into more than one table via a single form bound to a query which joins the tables.  Where it is necessary to insert data into multiple rows in another table, e.g. into an OrderDetails table via an Orders form a subform based on the referencing table (OrderDetails) should be embedded into a parent form based on the referenced table (Orders).  Where it is necessary to insert a single row into a referenced table, e.g. a new customer in an Orders form this should be by means of the NotInList event procedure of a combo box bound to the foreign key column (CustomerID), or more crudely, by setting the combo box's ListItemsEditForm to the name of a form bound to the referenced (Customers) table.

1/2.  These are closely connected as they mean there is a binary (2-way) many-to-many relationship type between  each instance of the employees entity type.  A binary relationship type is modelled by a table which resolves the many-to-many relationship type into two one-to-many relationship types.  Where the many-to-many relationship type is between two instances of the same entity type, as here, the table which models the relationship type is an 'adjacency list'. So the tables are:

Employees
â€¦.EmployeeID  (PK)
â€¦.FirstName
â€¦.LastName

EmployeeManagers
â€¦.ManagerID  (FK)
â€¦.EmployeeID  (FK)

The primary key of the latter table is a composite one of the two foreign key columns, each of which reference the primary key of Employees.

The model is therefore as follows:

Employees(1)----<EmployeeManagers>----Employees(2)

NB: Employees(1) and Employees(2) are not separate tables, but are instances of the single table Employees.  The many-to-many relationship type between employees and roles would be modelled in the same way, but between Employees and Roles tables, not between two instances of a single table.

3.  Here things are not so straightforward.  It's not clear to me what you mean by 'record'.  A record is a row in a relational database table, and each row represents an entity of the entity type modelled by the table.  What entity type your EmployeeRecords table is modelling is unclear, partly because of its generic name.  A table should be given a name which describes the entity type being modelled, usually as a plural or collective noun to reflect the fact that a table is a set.  Employees is a case in point.

Your EmployeeRecords table also has a fundamental design error in that you have multiple ObjMet# columns, which mean that the table is not normalized to First Normal form (1NF) whose definition is:

First Normal Form: A relvar is in 1NF if and only if, in every legal value of that relvar, every tuple
contains exactly one value for each attribute.

Loosely speaking, in the language of the relational model, a relvar (relation variable) equates to a table, a tuple to a row (record) and an attribute to a column (field).
