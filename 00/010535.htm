Why can't ORDER BY expression involve DISTINCT fields - Microsoft Community

paul domaskis say:

Why can't ORDER BY expression involve DISTINCT fields

I have the following query:


   SELECT DISTINCT t1.field1, t2.field2
   FROM table1 t1 INNER JOIN table2 t2
   ON t1.field1=t2.field2
   ORDER BY t1.field1 & t2.field2


Access 2010 (SQL-92 enabled) says that the ORDER BY clause conflicts with DISTINCT.  Why is that?  Note that the above ORDER BY expression is a simplification for troubleshooting only.  In practice, I use field1 & field2 in an arbitrary expression, e.g., someFunction(t1.field1,t2.field2).  But I don't understand why the expression involving field1 and field2 can't simply be evaluate for each distinct combination of values of field1 and field2.


I found that I can achieve the same effect with:


   SELECT t1.field1, t2.field2 FROM
   table1 t1 INNER JOIN table2 t2
   ON t1.field1=t2.field2
   GROUP BY t1.field1, t2.field2
   ORDER BY t1.field1 & t2.field2


However, it seems like the SQL involving DISTINCT is logical is well.  And shorter, too, and requires no vast amounts of time lost to debugging.

ken sheridan say: You can't pic n' mix between the two clauses.  This will work:

SELECT DISTINCT Lastname, FirstName
FROM Contacts
ORDER BY Lastname, FirstName;

As will this

SELECT DISTINCT Lastname & " "  & FirstName AS FullName
FROM Contacts
ORDER BY Lastname & " "  & FirstName;

But not this:

SELECT DISTINCT Lastname & " "  & FirstName AS FullName
FROM Contacts
ORDER BY Lastname, FirstName;

In SQL an ORDER BY clause is actually part of an implied CURSOR declaration.  Consequently, where the two clauses differ as in the last example it won't work because the LastName and FirstName columns don't exist in the result set of the cursor.  The reverse, as in your first example, is also true of course.  Since the SQL-99 standard it has been legitimate in standard SQL to use the name of the computed column in the ORDER BY clause, but JET/ACE SQL does not support this, so it is necessary to repeat the expression as in the second example above.

However, the following will work in JET/ACE SQL:

SELECT Lastname & " "  & FirstName AS FullName
FROM Contacts
ORDER BY Lastname, FirstName;

which the explanation given above would suggest would be illegitimate.  Curiouser and curiouser!  This suggests that the use of the DISTICT predicate invokes a CURSOR, but no CURSOR is invoked without the DISTINCT predicate.  But this is really no more than guesswork on the basis of the behaviour, and the real reason for the behaviour might be something completely different under the skin.
