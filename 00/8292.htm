jay freedman say:

The "Do not check spelling" option does not disable spell checking throughout the document, in Word Online or in the desktop Word. It applies only to the selected text, in a manner similar to bold or italic formatting.

In desktop Word, you can go to the Options dialog's Proofing page and turn off "Check spelling as you type", which also turns on "Hide spelling errors in this document only". However, that just removes the red squiggly underlines; pressing F7 still runs the manual spelling check. Word Online doesn't even offer the ability to hide errors.

As far as I know, there is no way to disable spell checking in Word Online; and if there is a way, there's still no way to prevent students from turning it on again. Fundamentally, Word is not a good tool for giving spelling tests.

In addition to the two excellent responses you've already received, you can also specify what you want returned if the inputvalue is Null.

tom van stiphout say:

Nz is an abbreviation for "Null to Zero". It turns any Null value into a 0 value or a "" value depending on its best guess of whether you'll want a number or a string.

The advantage of using this function is that calculations can continue. For example:

z = x + y

z will be null if either x or y are null. Often not what you want.

z = Nz(x) + Nz(y)

Now the calculation will continue normally. If x=Null and y=5,  z will be 5.

In your code you have:

"[Property Details]=" & Nz([PropertyID],0)

So if PropertyID were null, this would result in "[Property Details]=0"

rather than:

Null

 

-Tom. Microsoft Access MVP

a. user say:

Broadly speaking, you can't use an IF "inside a table" to say @reveal this column@ or "do not reveal this column". WHat you can do (particularly if you are talking about MailMerge, is to use an IF field to insert one table or another,,,

{ IF <the condition> "insert table with n columns" "insert table with n+1 columns" }

However, if you are populating the table with rows from a merge data source, that wont work either as it will give you 1 table per row, rather than a table with 1 row per data source record.

Depending on the data source, another approach might be to use a DATABASE field to insert your table - you can control the row count in various ways (in the SQL SELECT statement or by specifying start and end rows), and if necessary you could use IF statements to generate a SQL statement that included the columns you wanted. The main drawback is that you have a lot less control over the formatting of the tablle and the data within it, especially of the table flows over multiple pages. You can't apply formatting switches to the data in individual columns, so you either have to do that using your SQL SELECT (but the statement length is severely limited byWord), or using a suitable query/view/TRANSACT-SQL table-valued function in your database (there always used to be problems using a regular TRANSACT-SQL procedure but I suppose they might have been fixed now. OLE DB rather than ODBC connection is generally still needed by Word.

If you are not talking about a mailmerge, and  trying to create a user-editable document, bear in mind that unless the IF field is unlinked (i.e. replaced by its result) then any edits done by the user will be lost the next time the IF field is update - in that case you are probably better off considering a macro approach.
