Calendar permissions (Known solutions) - Microsoft Community

alex rizeanu msft say:

Calendar permissions (Known solutions)

Technical Level : Intermediate

Summary

This page contains known solutions related to Calendar permissions issues:

  • Outlook user is unable to manage the Calendar of a Kiosk user
  • You do not have permission to view this calendar. Do you want to ask user to share his or her Calendar with you?
  • The sharing invitation couldn't be sent to the following recipients: xxx@domain
  • Your request couldn't be completed. Http Status code:500.
  •  Microsoft.Exchange.Data.Storage.ObjectNotFoundException: Default folder Notes not found.
  • Couldn't update the following event <EventName>  

Details

  • If an Outlook user is unable to manage the Calendar of a Kiosk user, then note this is by design. Accessing and managing Kiosk user's calendar is limited to OWA only.
  • If in a Hybrid set up, a user having permissions is unable to create or open the items in the Calendar of a shared/user/room mailbox, due to the error: You do not have permission to view this calendar. Do you want to ask user to share his or her Calendar with you? then make sure the targetaddress/remoteroutingaddress is pointing to SMTP:*** Email address is removed for privacy *** and there is a secondary proxyaddress matching smtp:*** Email address is removed for privacy ***. If the secondary smtp pointing to smtp:*** Email address is removed for privacy *** is missing the users will see the issue in Outlook 2013 while Outlook 2016 or OWA will not exhibit the issue.
  • If it's a Hybrid set up, make sure the targetaddress/remoteroutingaddress is properly configured. The value should be SMTP:*** Email address is removed for privacy ***. Additionally this address should be listed as a secondaryaddress in proxyaddresses. You can use Pwershell (Get-MsolUser -UserPrincipalName <UPN> | fl) to review the attributes or you can collect a local AD dump (using Active Directory Module for Windows PowerShell) of the user's properties. 

get-aduser alias -Properties * > c:\temp\mn_aduser.txt

  • If the issue started to occur after the migration to O365, and the impacted user had delegate/folder permission, make sure either both mailboxes are in O365 or, in case of cross-premises, full access permissions should be granted.

  • The send on behalf of permission is not needed for meeting-related messages. Microsoft Exchange Server specifically does not require that you have send on behalf of permissions to send a meeting request on behalf of another user. Therefore, even without the send on behalf of permission, delegates can successfully send meeting requests on behalf of the calendar's owner. However, when you use a delegate, that does not have the send on behalf of permission to send a non-meeting-related message on behalf of the owner, the operation fails. For example, if a delegate tried to send an informational message "from the boss," that operation would fail. This is because the send on behalf of permission was not successfully granted.
  • If a user is unable to share or assign calendar permissions, then review if the user has any orphaned delegates Using MFCMapi to inspect the LocalFreeBusy message. Remove any orphaned delegate that you identified while inspecting the LocalFreeBusy folder and test again the issue. If the issue persists, run also outlook with the switch /resetfolders.
    • OWA Error when sharing:

The sharing invitation couldn't be sent to the following recipients: *** Email address is removed for privacy ***

    • OWA error when trying to pull the Calendar permissions

Your request couldn't be completed. HTTP Status code: 500.

    • Fiddler error resembling to the below one:

{"Body":{"ErrorCode":500,"ExceptionName":"ObjectNotFoundException","FaultMessage":"Default folder Notes not found.","IsTransient":false,"StackTrace":"Microsoft.Exchange.Data.Storage.ObjectNotFoundException: Default folder Notes not found.td>

  • If the user is unable to update items as he is getting an error resembling the below one:

Couldn't update the following event <EventName>

    • then review the dumpster of the mailbox on which the user is trying to update the meetings and make sure is not full.
  • If a user is unable to modify items on another user's Calendar even permissions are shown in PowerShell nor Offcat, then run the following commands: 

  • Remove-MailboxFolderPermission -Identity kim@contoso.com:\Calendar -User john@contoso.com
  • Add-mailboxfolderpermission kim@contoso.com:\Calendar -user john@contoso.com -accessrights editor

  • If a user is able to modify items on another user's Calendar even no permissions are shown in PowerShell nor Offcat, then run the following commands: 

  • Add-mailboxfolderpermission kim@contoso.com:\Calendar -user john@contoso.com -accessrights editor
  • Remove-MailboxFolderPermission -Identity kim@contoso.com:\Calendar -User john@contoso.com

  • In case the issue is with the Calendar delegation and the Calendar was opened as a shared folder, remove the target's mailbox Calendar and re-add it from the Address Book.

  • In case the Calendar was added via the Full Access permissions with automapping on, remove the permissions and test without auto-mapping. Follow the steps here. If the steps help solving the issue remove again the permissions and re-grant them with automapping set to $true.
   

  • In case the calendar was added via Additional mailbox, remove the additional mailbox and test by opening the shared Calendar from the Address Book.

  2. Select Account Settings - Account Settings.
