>>fact that the query continues to work after the initial execution affirms that the connection data stored with the query and in the defaults of the SQL login dialog is correct. 

Actually this suggests quite much the opposite (that the connection string is wrong).

Since the FIRST time you get a prompt, when the user enters the correct information, that information is used IN PLACE of your connection string!

What this suggests is that the connection string is wrong, or not been setup correctly.

I would link a simple table to SQL server and DURING THE link process ensure that you check the save pass-word option.

Now test the above simple table link. Exit Access and try opening the linked table. If you get no prompts, then copy the connection string used for that linked table into the connection properties for the pass-though.

So something is wrong or missing with that connection string. ONCE the user enters correct information for the connection, then that is used for connection – not what you have in the string.

Of course this “caching” effect is a HUGE bonus and feature that you can use WELL to your advantage in Access. This means in effect you do NOT have to store the UID/Password in the connection string anymore!

How this works is outlined here:

Power Tip: Improve the security of database connections

http://blogs.office.com/b/microsoft-access/archive/2011/04/08/power-tip-improve-the-security-of-database-connections.aspx

So you can use some simple code to “logon” to the database, and once done then all the linked tables and PT queries should work without prompts.

Some simple SQL logon code I use is thus this

(just pass the connection string).

Function TestLogin(strCon As String) As Boolean

   On Error GoTo TestError

   Dim dbs          As DAO.Database

As DAO.QueryDef

   Set dbs = CurrentDb()

    qdf.Connect = strCon

    qdf.ReturnsRecords = False

    'Any VALID SQL statement that runs on server will work below.

    qdf.SQL = "SELECT TOP 1 * FROM sys.all_columns"

    TestLogin = True

    Exit Function

TestError:

End Function

windows 7, office.com : when I access a file I get a ghost file ~$-----.doc that effects using the file, why and how do I fix??

Whenever a file is accessed in office a ghost file is created. The file name is preceded with ~$  (file name less a letter or two).doc.    Why is this happening and how to I fix it??
jay freedman say:
Gordon, that's almost correct. A temporary file whose name starts with ~$ is an "owner" file, which isn't the file you edit. There are many kinds of temporary files that Word may create, and they're explained in http://support.microsoft.com/?kbid=211632.

If you don't display hidden files, and especially if you never store and edit Word documents on the desktop, then you'll never see any of these files -- but that doesn't stop them from being created, used, and deleted.

If Word crashes, or for some other reason fails to delete the temporary files, you can get rid of them after Word is shut down. See http://www.gmayor.com/what_to_do_when_word_crashes.htm.

jay freedman say:
The owner file is explained near the bottom of the KB article. It's what tells Word that the document is already in use by someone else, and prevents you from editing it simultaneously -- you can make a copy and work on that, but you can't open the same file. (If you're both on SharePoint, the coauthoring feature changes this behavior.)

Where this goes bad is if the owner file doesn't get deleted as it should when the document is closed. Then Word tells you the document is in use when it isn't. The solution is to delete the owner file.
