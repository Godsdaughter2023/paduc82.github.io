steve rindsberg say:

And as an aside to David's answer:

>> " i know that there's a auto save of office that you can arrange it to save every 10min and so"

That option is quite misleading.  It doesn't do what you'd expect it to do.

It saves recovery information every 10 (or whatever) minutes so that IF PowerPoint crashes and IF it's smart enough to realize "OH!  I'm about to crash", it can restart and recover whatever was last saved.  

The feature doesn't save your work in any other fashion; for example if you've worked on a presentation for three days nonstop, go to quit PowerPoint and click NO when it asks if you want to save, you've just said NO to three days of your life.  It's gone.


>> "but it hasn't work,"

It does work; it just doesn't do what we think it should do.

joel ivory johnson say:

Hello Caz62.

There are not any Windows Mobile 6.x devices that will be upgradable to Windows Phone 7.

Windows Mobile had been designed to be an OS that could be adapted to different hardware configurations and customizable according to design needs. But the consequences of this high flexibility in adapting the OS is that there were many varried hardware designs, each of which had it's own customized build of Windows Mobile. Windows Mobile build updates have never come directly from Microsoft. Instead the updates are made available to the device manufacturers. If both the device manufacturer and carrier through which a device was sold are in agreement then a new OS image could be made available for owners of the device. But when either the device manufacturer or carrier doesn't decide to make an update available then a device is nolonger able to get OS updates. Given that there are hundreds of different hardware designs running Windows Mobile (unlike the desktop PC in which all of the hardware designs are based on a set of standard designs) there's not a generic Windows Mobile image that would be able to run on each customized design. Thus when the hardware manufacturer or carrier decide to nolonger allow updates to get to a WM6.x device then there was no sanctioned way for a user to get more updates. The statement you've received from HTC about it being up to Microsoft to deliver the update to users is a little odd given that over the past 10 years devices running this family of OS have always gotten their OS updates from the manufacturer and never directly from Microsoft.

In recognition of the problems that the above causes Microsoft made a common hardware design for Windows Phone 7 and has provided requirements that will be enforced to ensure that an OEMs hardware conforms.  The OS images and updates for Windows Phone 7 are not coming from the OEM and carrier but instead are coming from Microsoft (which is doable under WP7 because of the common set of requirements).  The HD2 and other Windows Mobile devices don't conform to the Windows Phone 7 requirements. Looking at the surface level requirements all of the existing Windows Phone hardware doesn't conform to the requirements for screen resolution, required RAM, the storage memory requirements, button requirements, processor and GPU requirements, and so on.  I can only imagine that these same devices don't conform to architectural requirements.

The problem isn't unique to Windows Mobile. It has happened on Android devices to, though to a lesser extent. For some Android devices there have been instances where Google has made a new Android version available but an OEM has decided to either nolonger produce their customized version (in which case the end user gets no further updates^) or takes a long time to produce an update (in which case when the customer does receive the update it is still behind the latest version available). Google will be responding in a manner similar to Microsoft with enforcing stricter requirements on future Android devices.

In either case (for both the Windows and Android scenarios) inorder for a customer to move out of the no-upgrade scenario to having hardware that receives continuous updates from the OS vendor it is necessary to acquire new hardware. Historically Microsoft has been pretty good about supplying updates to hardware that conforms to common designs (Zune, XBOX and Xbox 360, PCs). I'd expect the common hardware platform for Windows Phone 7 to address this problem, but there's no way to take advantage of forthcoming changes without getting new WP7 hardware.

brian tillman [outlook mvp 2007-2019] say:

Windows Live Mail is a Windows-only application and it cannot be installed on an Android tablet.

Don't use a POP account.  On your tablet, in the Email app, create either an "Outlook" or "Corporate" or "Exchange" or "Hotmail" account (depending on your Android version) to access your Outlook.com mailbox. This will allow you to see the same data as you see if you use a web browser to access your Outlook.com mailbox.  In WLM on the PC, use an HTTP account.  Allow WLM to create the account automatically.  It, too, will then sync with the Outlook.com mailbox and you'll see the exact same stuff on the PC  and the tablet.

For the Gmail account, use an IMAP account.  You can set this up in WLM on the PC, in the Email app on the tablet, or you can continue to use the Google Mail app.

For these types of accounts, sync is two-way so that anything you do on any device gets echoed to all of the devices.

For your third account, check with the mail provider to see if they also allow IMAP access.  You can continue to use POP if you wish, just as you do now.
