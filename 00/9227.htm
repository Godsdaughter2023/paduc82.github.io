jay freedman say:
As often happens, the macro recorder isn't capable of creating a working macro for the type of manipulation you need. It uses the wrong objects and assumes certain things that are almost certainly not true. Its usefulness is mainly for very simple jobs and for learning how to write macros from scratch.

Here's the working macro you requested:

Sub InsertTrianglePageNumber()
    Dim tmp As Template
    Dim shp As Shape
    Dim rg As Range
   
    With ActiveDocument
        Set rg = .Sections(1).Footers(wdHeaderFooterPrimary).Range
        If rg.ShapeRange.Count > 0 Then rg.ShapeRange.Delete
        Set tmp = GetBBTemplate
        If Not tmp Is Nothing Then
            tmp.BuildingBlockEntries("Triangle 1").Insert _
                Where:=rg, RichText:=True
            Set shp = rg.ShapeRange(1)
            With shp
                .Fill.ForeColor.ObjectThemeColor = wdThemeColorBackground1
                .Fill.ForeColor.TintAndShade = -0.15
                .Fill.Solid
                .Fill.Visible = msoTrue
                With .TextFrame.TextRange.Font
                    .Color = 0 'seems to give the same black as -587137025
                    .Name = "Calibri"
                    .Size = 28
                    .Bold = True
                End With
            End With
        Else
            MsgBox "Could not find Building Blocks.dotx template", , "Error"
        End If
    End With
End Sub

Function GetBBTemplate() As Template
    Dim tmp As Template
    Templates.LoadBuildingBlocks
    For Each tmp In Templates
        If LCase(tmp.Name) = "building blocks.dotx" Then
            Set GetBBTemplate = tmp
            Exit For
        End If
    Next
End Function

If necessary, you can call this code from another macro by using its name, InsertTrianglePageNumber, as a statement.

Alternative to sumif then index/match - Microsoft Community

ando_888 say:

Alternative to sumif then index/match

I have a sheet with more than 200 columns representing regions - for the sake of this question call them Nth, Sth, East, West etc. In each column there are 50 different types of expenses eg salary, annual leave, superannuation. Each type of expense is one of 10 different main expense types eg salary & on-costs. I need a 2nd sheet in the workbook to take a selection of the columns and just the main expense types. I could do a sumif on the first sheet to get the main expense types summarised below the data above and then pull them through to the 2nd sheet using match/index, but ideally I would like to go straight from the data on the 1st sheet to the 2nd sheet.

 

Any ideas would be appreciated.

jllatham say:

I've uploaded an Excel 2003 file to MediaFire that you can download to see the formula I'll present below in action.  Link to for_Ando_888.xls (17KB):

http://www.mediafire.com/?0ivyyazalq5q1t2

Assumptions:

Sheet1 - The entry Category is at A1 and locations extend across row 1.  Major categories are then all in column A down to row 10.

Sheet2 - A1 is actually empty and A2 contains "Income" and "Salary & on-costs" is down at A8.  NorthAct is in B1, NorthBud is in C1 with Variance in D1.

In cell B8 on sheet 2 put this formula:

=SUMIF(Sheet1!$A$1:$A$10,$A8,OFFSET(Sheet1!$A$1:$A$1,0,MATCH(B$1,Sheet1!$A$1:$H$1,0)-1))

The formula can then be filled down and across the sheet and will automatically adjust to get the correct totals for you.

What we've done with this is set up the "SumRange" portion of the SUMIF() function to use an offset to find the proper column on Sheet1 to use.  We do that by matching the location entry in the formula's column, row 1 (B1) to all the location entries in row 1 on Sheet1; trying to get a match starting at the same column (A) that the categories are listed in even though we know that 1st column won't ever match.  Then subtract 1 from that result to turn the value returned by MATCH() into the proper offset.

The use of $ symbols (and non-use of them in places) is crucial to it continuing to work as you then fill that formula down and across Sheet2.

Triggered by something OssieMac said, I think I have this figured out.  Microsoft please take note: you should better document this.  Here's how it works:

When using the Get Data -> From Text/CSV function and using the Edit option, DON'T use the Data Type selection over the Transform group/tab. 

Instead, right-click the column header over the column containing the dates, and select Change Type -> Using Locale.  Then, in the dialog that appears, select data type Date and a locale that uses the date format that matches the CSV date format, and click OK. 

In this case I needed mm/dd/yy.  There are a LOT of options and you might have to search, but at least it does show examples for the one you pick.  In my case English(United Stated) worked perfectly.
