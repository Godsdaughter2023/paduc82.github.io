I don't want to spend too much time trying to explain the difference between the HEAD section of the source code and the BODY, but try this. Just right click this page > View Page Source. The source code will probably open in NotePad. Note line 5 you will see <head> and then line 56 you will see </head>. This is the HEAD section of the page of code. Below that you will see the BODY section. If you want to learn more then read this page and peruse this whole website to understand HTML coding. http://www.w3schools.com/html/html_head.asp

When you insert a code fragment into a Publisher page you can only insert into the BODY section. If you want to insert anything into the HEAD section you have to open the *.htm file that Publisher produces with NotePad or some other editor and then manually add the code. I found it helpful to right click the index.htm file that Publisher produces and Open With > NotePad and study the coding. It was clear as mud at first but then I started understanding a bit about how the Publisher HTML coding engine produces the code and learned a bit about editing it manually.

But one of the reasons we use Publisher is to avoid learning HTML coding and having to manually edit anything. So I generally have avoided anything that requires me to edit the html code after it is produced by Publisher. Another important reason is that I would have to do that each time I produce a new .htm page because the manual edit is not reflected in the Publisher publication page. If I use the insert HTML code fragment that will insert the code each time I produce updated pages.

Now, if you look at the arbitrarily chosen script on Dynamic Drive: http://www.dynamicdrive.com/dynamicindex4/featuredzoomer.htm 

You will see that there is script that you are supposed to copy and insert into the HEAD section and script for the BODY section. In general I skipped these and looked for scripts where it only required me to insert code into the BODY, but occasionally I would just copy both the HEAD and BODY script and insert it into the BODY of the Publisher page and it would work. Most of the time it does not. Bottom line is with so many different scripts available I would suggest you try the ones that have script for the body only for now.

What Folder? When you produce your web pages Publisher will produce an index.htm file (your home page) and an index_files folder that contains all the other .htm files (the other pages of your web site) and all the picture files for your site. When you upload those to your website you upload to your root directory or root folder. You would upload your slideshow folder with the webfiles within it to the root folder right beside the index.htm file and the index_files folder. You can create the folder locally and upload the whole thing or you can create the folder on your webhost under or within the root folder...that is why I call it a subfolder. The link to that folder would look something like this: http://yourwebsite.com/slideshow/index.htm Now many slide shows will have another folder just to contain all the different image files used by the slide show. In this case it might be a subfolder within the slideshow folder and the link might look like this: http://yourwebsite.com/slideshow/images/slideshowimage.jpg

As per PDF file, do the same. Create a subfolder on your website under the root directory and upload your PDF files to that folder. A sample link might look like this: http://yourwebsite.com/pdfdownloads/yourfile.pdf

And finally I gave you links to two sample Publisher websites that were using scripts. I would suggest that you open those pages and look at the source code and find where the code is inserted. It will be a good learning tool and will illustrate how they wrote the links. Although right clicking an empty or blank section, outside a text box or image of a webpage with usually give you the option to view the source code you can also find that option in the tools of the web browser you use. It will vary with the browser. With FireFox I find it Tools > Web Developer > Page Source.

And actually one more suggestion. Remember to test your pages in both IE and FF to make sure they enjoy good cross browser compatibility. If your pages load and work correctly in both those browsers they should work in the other major browsers out there.

scottgem say:

You should have a table of Contact_Types. You place an unbound combobox in the form header (not the one bound to the Contact_Types field) that looks up the Contact Type.

If your database is setup properly. Your Contact_Types table should have an Autonumber key and a Text description field. The Contact_Types field in your main table should be bound to the the Autonumber key. The unbound control will display the description.

In the After Update event of the unbound combo use code like this:

Me.Filter = "Control_Types = " & Me.controltypesearch

Me.FilterOn = True

That should do what you want.

ken sheridan say: An easy way to do this without any VBA code would be not to filter the form, but to restrict its recordset, as suggested above by the DBguy,  by using a query as the form's RecordSource, which references an unbound combo box on the form, e.g. in its header, as a parameter.  The query would be like this:

SELECT *
FROM Contacts
WHERE (Contact_Type = Forms!frmContacts!cboGoToContactType
OR Forms!frmContacts!cboGoToContactType IS NULL);

where frmContacts is the form's name and cboGoToContactType is the name of the unbound combo box.  You can add an ORDER BY clause to sort the records if you wish.

As the AfterUpdate event property of the unbound combo box call a macro which requeries the form, though in VBA it only needs one line in the event procedure:

    Me.Requery

You can easily add the unbound combo box to the form by copying the existing bound combo box and pasting it into the form as a new control.  Then delete the value in the new control's ControlSource property and change its Name property to cboGoToContactType.  Whether the Contact_Type column is a numeric value, as described by Scott, or a text value is immaterial when referencing the control as a parameter.

To restrict the form's recordset to a contact type select an item in the unbound combo box; to show all contact types simply delete the value from the combo box.
