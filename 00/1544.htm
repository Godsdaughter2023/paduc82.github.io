What you're remembering is Publishing Layout View which was available in Word 2008 & 2011, but has been excluded from Word 2016.

jeffconrad say:

Hi,

You can't dynamically toggle hiding/showing entire views within the Access web app interface shown in your browser using macros or other means. As you noticed, you can hide an entire table caption which hides all of the views attached to it within your browser. That is essentially an "all or nothing" type of thing.

You can use the SetProperty macro action to dynamically show/hide controls on a view at runtime, but not the whole view itself. Incidentally, you cannot use the [Forms]![Contact List] type of syntax anywhere in web apps like you can in Access desktop applications.

Here are some techniques you can use to "simulate" some extra control over the navigation of views. Keep in mind, these techniques aren't completely without some trade offs.

1.  You can make standalone views - views not attached to any table caption in the Table Selector - for any view you need to restrict. Standalone views can only be opened via Open Popup macro actions. You can then control who can open these since there is no way for a user to navigate to them manually through the Address Bar. a potential downside is that these are in a smaller popup window.

2. For a really hacky approach, you can clear out a view's caption in client designer (select to rename from the options menu of the view) and just leave an empty string. Then move that view to the far right of any other views attached to a table caption. At runtime, you won't see anything to click on. However, users could still open this view if they happen to click the small spot on the page where the empty view caption is. Yes, hacky I know, but it might work for your scenario.

3. You can use the Change View macro action to navigate to a specific Table/View combination, perhaps even with your On Start macro logic. If you hide an entire table caption and it's associated views, you can still use the Change View macro to open and navigate to them by providing the specific table and view names in the macro arguments.

4. Building on option 3, in the On Load event of any "restricted" view, you can check the current user and if they are not one of the people supposed to open this view, then use the Change View macro action to navigate them anyway to some other view they can open. So in this case even if the view caption is shown to users in the browser, if they click on it and are not approved, they just get navigated somewhere else.

Hopefully this gives you some ideas to use in your Access web apps.
