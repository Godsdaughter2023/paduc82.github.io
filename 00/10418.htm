AzureAD Account Matching - Source of Authority Transfer to On-Premises - Microsoft Community

alex rizeanu msft say:

AzureAD Account Matching - Source of Authority Transfer to On-Premises

Technical Level : Expert

Summary

These are the 3 typical ways accounts are created in Office 365 and Exchange Online:

  • Created directly in AzureAD and forwardsynced to EXO
  • Created directly in EXO and backsynced to AzureAD
  • Created on-prem and dirsynced to AzureAD, then forwardsynced to EXO

In each of these examples, an account from a source directory is copied to the destination directory (plus or minus certain attribute values). At that moment a permanent synchronization link is established between the two. This is an automatic process between AzureAD and EXO. We cannot create a user in AzureAD without automatically triggering an account creation in EXO, and vice-versa. When syncing an on-prem account to AzureAD using DirSync, a link is also established between the account automatically.

Aside from this automated provisioning process, there are two methods an admin can use to forcibly link a pre-existing AzureAD account with a specific on-premises account. The accounts must not be currently linked to any other account for this to work.

  • SMTP Soft-Matching: Using the same primary SMTP address between the AzureAD and on-prem account.
  • ImmutableID Hard-Matching: Overwriting the AzureAD account's ImmutableID value with the on-prem account's sourceAnchor value.
For an explanation on 'immutableID' see Sync Anchor Values

Normal account provisioning should be left up to the automated process. An admin would only use these manual methods under very specific circumstances because they are labor intensive and prone to human error.


Details

Common Scenarios

SMTP Soft-Matching

The following are common scenarios where soft-matching would be implemented:

  • Accounts are accidentally created directly in the cloud when business processes require the objects to be directory synced. The accounts have been in use for some time and there is a risk of data loss if the account is deleted and recreated.
  • Accounts are created in the cloud before introducing directory synchronization.

ImmutableID Hard-Matching

This is the typical scenario where you will apply hard-matching:

  • An account is dirsynced to the cloud to begin with. The on-prem account is then filtered from synchronization. This causes the cloud account to disconnect from the on-prem account and to be moved to the soft-deleted state. The Administrator restores the cloud account, but permanently deletes the on-prem account. Because the two are no longer linked, the purging of the on-prem account has no effect on the cloud account. The Administrator later creates a new on-prem account and is required to link it to the cloud account.

Which Method Should Be Used?

If the AzureAD account has a blank ImmutableID value, use SMTP soft-matching; otherwise use ImmutableID hard-matching. The SMTP soft-match is easier to use, but it will fail for accounts with populated immutableIDs.

Procedure:

SMTP Soft-Matching

The EXO account, and therefore the AzureAD account, is already going to have a primarySMTP address that you probably don't want to change. In order to successfully link the cloud account to an on-prem account you just need to stamp the same primarySMTP to the on-prem account and run a directory sync. However, you must take great care when dealing with any existing secondary smtp X500 addresses (or any other attribute value for that matter). If you fail to copy these onto the on-prem account before you link the two, you will end up overwriting these addresses in the cloud.

See Procedure: Account Soft-Matching


ImmutableID Hard-Matching

Both the On-premises and the Azure AD user object will potentially have the same UPN. However, the AzureAD user object will show a status of "In Cloud". The goal is to link a new on-prem user object to a pre-existing Azure AD user object/mailbox. In order to successfully link the Azure AD user object to a new on-premises account, you need to convert the on-premises ObjectGUID to Base64 and paste this to the ImmutableID attribute on the Azure AD Object. The ImmutableID attribute is how Azure knows it is linked to an on-premises object. However, you must take great care when dealing with any existing secondary smtp X500 addresses (or any other attribute value for that matter). If you fail to copy these onto the on-prem account before you link the two, you will end up overwriting these addresses in the cloud.

See Procedure: Account Hard-Matching

Verifying Success

Below is a before and after screenshot of an account that was soft-matched. Note the addition of 'LastDirSyncedTime' and 'ImmutableID' as well as the addition of a 'mail.onmicrosoft.com' address and changing of the UPN to match the on-prem account.

SMTP mismatch

The primarySMTP of the on-prem account you are exporting does not match the primarySMTP of any account in AzureAD; however it's secondary address does. This conflict causes the error and fails the export. We cannot link to an existing account, and we cannot create a new account in AzureAD. To remediate this situation, double-check the primarySMTP of the two accounts you are trying to link and adjust as necessary. If they do match, check for replication delays on-premises amongst the domain controllers involved with the account's provisioning, especially if the account's address changed recently.
