a. user say:

Hi GertBert,

I suspect that you havent received many replies in here because most people would not claim to have good information about the relative performance of different solutions, and would also want to pin down exactly what you need to do.

There was a discussion about DATABASE fields in the MSDN thread, but 
 a. do you also need something that overcomes the problem with single-cell fields containing text that I mentioned on the MSDN site
 b. there is still a question mark about what you mean by "linking". If you mean "linking using builtin features of Word that enable users to update links without code, then the only options are the field types that enable linking - LINK, DATABASE, and I may as well mention DDE and DDEAUTO and Mergefields. Of those, 
 - Mergefields are really geared to record-by-record data processing and I do not think they are suitable for your requirement
 - DDE/DDEAUTO were deprecated a long time ago, rely on intertask communication (e.g. you have to have Excel/Access running to extract data from anything)
 - I treat DATABASE as deprecated as it has been  removed from the standard Word UI, but as long as it exists, it does have the advantage that it can work directly with (more or less) anything that has an ODBC driver or an OLE DB provider.
 - LINK requires that your data is maintained via a program that can act as an OLE server. In effect that means Excel (I don't know about non-MS spreadsheet programs). If you move your data to Access or any "backend" such as SQL Server, LINK won't do anything for you.

If your data is Unicode (i.e. goes beyond ANSI/OEM character sets) DATABASE needs an OLE DB provider, which also starts ruling out most non-Microsoft solutions. I think the limitation is actually in Word, but am not sure. LINK should deal with Unicode better, and overcome the problem with "extra paragraph marks" that DATABASE suffers.

Whenever you retrieve data using Jet/ACE (which you will if you use DATABASE, ODBC, OLE DB with Excel files, Access .mdb/.accdb, and plain text files) you need to be aware that the driver/provider may lose or mangle information if you do not organise your

If you are talking about creating your own "LINKs", it might be something like...
 - you have a document and some way to specify what data needs to be inserted where in the document

then either
 a. you populate the document and the user has no way of refreshing the "LINKs" or
 b. you populate the document and you also provide code that lets the user update the "LINKs"

then there are obviously more options, and perhaps you could clarify what you are thinking of first. 

But I don't think using the OpenXML SDK is going to help all that much because it can only generate  .docx type documents, whereas you're talking about Word 2003. (Obviously if you can rely on the OpenXML format converter, all you need worry about is the performance and reliability of the converter). I would have thought an approach based on either VBA+ADO or .NET+VSTO+ADO.NET could do the job. Same potential problems with OLE DB providers, though, although there are ways you can exert a little bit more control over how the provider behaves.
