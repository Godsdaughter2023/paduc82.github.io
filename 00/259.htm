greginmich say:

Well, as I’ve previously pointed out many, many times, the Microsoft method for scheduling a scan in Windows 10 is totally degenerate. First of all, it will only schedule a Quick Scan, with no option at all for a Full Scan. In addition, when you “schedule” a scan by adding a trigger to the Windows Defender Scheduled Scan task, what you actually get is a scan that won’t run at the scheduled time if you’re using your computer at that time. You’d have to stop using your PC and wait for the scan to finish running before doing anything else – because this scan gets immediately cancelled anytime you start using your PC.

And let’s please not overlook the fact that Automatic Maintenance is already triggering the Windows Defender Scheduled Scan task on a daily basis, and that maintenance can be scheduled for any hour of the day, with the same proviso that the scan will only run at the scheduled time if you take your coffee break at that time:

Right-click on the Start button > Control Panel > Change maintenance settings.

http://answers.microsoft.com/en-us/protect/forum/protect_defender-protect_updating/windows-defender-atomatic-scan/462f2612-9824-4140-b7b3-8ddf333accc7  

So what possible reason could there be for adding another trigger to the Windows Defender Scheduled Scan task? Of course someone who keeps busy at their PC might want to schedule a Quick Scan that actually starts on time and runs without interruptions, or maybe even a Full Scan that runs at midnight, and that’s something that you would do with the Create Basic Task Wizard:

ken sheridan say: Measurement is an attribute of the relationship between recipes and ingredients, so should be a column in the RecipeIngredients table, not the Recipes table.  The latter should have the recipe name as a non-key column.  I'd suggest that you also have a column IngredientIndex in the RecipeIngredients table in which you can enter a number from 1 upwards to, indicate the order in which you want the ingredients to appear in the recipe.  So the model would comprise the following tables:

Recipes
....RecipeID (autonumber PK)
....Recipe

Ingredients
....Ingredient

and to model the many-to-many relationship type between them:

RecipeIngredients
....RecipeID (FK)
....Ingredient (FK)
....IngredientIndex

The primary key of the RecipeIngredients table is a composite one made up of the RecipeID and Ingredient columns.

Regarding your point about the layout, you can achieve want you want in a form or a report.  Not in a query, but the interface with the data should always be via a form or report, never directly with a query or table in datasheet view.

For data input create a form in single form view based on a query which sorts the recipes by name:

SELECT *
FROM Recipes
ORDER BY Recipe;

Add a text bio bound to the Recipe column to the form.  Then create form based on a query on the RecipeIngredients  table:

SELECT *
FROM RecipeIngredients
ORDER BY IngredientIndex;

This form should be in continuous forms view with a detail section just one row deep.  In the detail section put three controls, a combo box bound to the Ingredient column and text boxes bound to the Measurement and IndredientIndex column. Put the controls side by side in one line, and put a label for each in the report header above them.  The ingredient combo box's RowSource property would be:

SELECT Ingredient FROM Ingredients ORDER BY Ingredient;

Embed this form as a subform in the recipes form and set the LinkMasterFields and LinkChildFileds properties of the subform control to RecipeID  You'll now be able to enter a recipe name in the parent form and for each ingredient enter a row in the subform, selecting the ingredient from the combo box in each case and entering the measurement and ingredient index into the two text boxes.  In the AfterUpdate event procedure of the subform put:

Me.Requery

This will sort it by the index value when you enter each row.

You may of course wish to enter an ingredient which is not yet represented in the Ingredients table.  You can do this by putting code in the NotInList event procedure of the ingredient combo box:

    Dim ctrl As Control
    Dim strSQL As String, strMessage As String
    
    Set ctrl = Me.ActiveControl
    strMessage = "Add " & NewData & " to list?"
    
    strSQL = "INSERT INTO Ingredients(Ingredient) VALUES(""" & _
            NewData & """)"
   
    If MsgBox(strMessage, vbYesNo + vbQuestion) = vbYes Then
        CurrentDB.Execute strSQL, dbFailOnError
        Response = acDataErrAdded
    Else
        Response = acDataErrContinue
        ctrl.Undo
    End If

When you type the new ingredient into the combo box, whose LimitToList value must be True (Yes), you'll be prompted to add it to the list.  Once you answer Yes the code will automatically insert a new row into the Ingredients table and add it to the combo box's list.

To show the recipes in a report base the report on a query which joins the Recipes and RecipeIngredients tables:

SELECT Recipes.RecipeID, Recipe, Ingredient, Measurement, IngredientIndex
FROM Recipes INNER JOIN RecipeIngredients
ON Recipes.RecipeID = RecipeIngredients.RecipeID;

Group the report by Recipe and give the group a group header and group footer.  Then sort it by IngredientIndex.  In the group, header add a text box control bound to the Recipe column, and in the detail section add text box controls, side by side, bound to the Ingredient and  Measurement columns.  The detail section should be one line deep.  The group footer does not contain any controls, but make it deep enough to give a blank area between the end of one recipe and the start of the next in the report.  If you want each recipe on a new page set the ForceNewPage property of the group footer to 'After Section'.

Finally, you might want to be able to preview and/or print an individual, recipe from the recipes form.  For this add command buttons to the parent form for this.  The code to preview the recipe report would be like this:
