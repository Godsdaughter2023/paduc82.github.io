If this is an unbound text box, in a format setting of the properity sheet change the data type or formatting of the text box to "general number".  This will actually FORCE the actual value you stuff from the text box into that temp VAR to be a number data type as opposed to a string data type.

 

In fact you'll notice this issue, because if you run the same software in the access client, and use the ID or number value from that text box after being stuffed into a temporary variable, using a where clause of an open report or open form works just fine (client side).

 

However if you publish to the web, you'll find exact same code does not work, and the reason is is when you run these things from the web there's not that automatic conversion or what we call casting of the type of value from a string to a number that's required for this to work correctly (Access has always done this over the years)..  This pretty much means at the end of the day that you must pay greater attention to that of data types for applications that are going to be publish to the web.

 

So open up your current form and layout mode, and display the property sheet for the text box in question, and in fact if this is an unbound text box, then change the format to a number type, and try your code again.

 

Also the exact same advice applies to that of using date values. Again for unbound text box for prompting the user to enter a date value, again make sure in the property sheet of that control on the format tab you set the format to a short or long date.  When you do this, once again then the system will know if the value in question is in fact a text value, or a date value, or as your example a number value.

 

Last but not least, you keep in mind that this problem does not exist for bound text boxes and controls on a form, because they automatically inherit and accept and take on the data type of the underlying column of the table that the controls based on.

 

So in effect, this problem is more common when you use unbound controls.  Last but not least, while a text box and that of a combo box does have a format section for setting the data type, a list box does not.

 

If that list box is to return a number value, and you attempt to filter by the value, it will not work in less you explicitly forced the data type of the value you've received to be that of a number.  Thus you'll have to do the following:

 

SetTempVar:
        lngID
Expression:   cdbl([listboxControlname])

 

Notice we wrap the expresison with a cdbl() command - convert to double number. So uses cdbl() forces and means to convert to a number type of value. 

 

By the way the above function can be used for text box or combo box if you don't bother to go into the format tab and setting of the control in question and simple set the value to "general number". I think in most cases it's probably a really good habit to simply set the control to general number format and then you don't have to bother to remember to use the cdbl() function.

 

So keep in mind the exception to the above format setting is as mentioned is the case of using a list box, since you don't have a format section setting to cast or force the results to being a number, and then your ONLY choice is to use the above suggestion of the cdbl() function to force the expression to be a number.
