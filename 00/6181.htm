milesweinberger say:

endnotes in Word

In submitting a manuscript to a publisher, they ask that the references (endnotes in the manuscript) be submitted separately from the body of the manuscript and that the body of the manuscript should not have the references (which are in fact, endnotes).  But if the endnotes are stripped from the body of the manuscript, the reference numbers in the manuscript will be lost. 

Is there a way, to cut the reference list at the end of the manuscript without losing the reference numbering in the text?

Mike

macropod say:
For a macro to do the conversion, see: http://www.msofficeforums.com/word/12920-convert-endnote-footnote-regular-text.html#post34554

doug robbins - mvp office apps & services (word) say:

' Macro created 29/09/99 by Doug Robbins to replace endnotes with textnotes at end of document

' to replace the endnote reference in the body of the document with a superscript number.

'

    Dim aendnote As Endnote

    For Each aendnote In ActiveDocument.Endnotes

        ActiveDocument.Range.InsertAfter vbCr & aendnote.Index & vbTab & aendnote.Range

        aendnote.Reference.InsertBefore "a" & aendnote.Index & "a"

    Next aendnote

    For Each aendnote In ActiveDocument.Endnotes

        aendnote.Reference.Delete

    Next aendnote

    Selection.Find.ClearFormatting

    Selection.Find.Replacement.ClearFormatting

    With Selection.Find.Replacement.Font

        .Superscript = True

    End With

    With Selection.Find

        .Text = "(a)([0-9]{1,})(a)"

        .Replacement.Text = "\2"

        .Forward = True

        .Wrap = wdFindContinue

        .Format = True

        .MatchWildcards = True

    End With

    Selection.Find.Execute Replace:=wdReplaceAll

rich michaels - mvp (greatcirclelearning.com) say:

Hi Afreen

This morning I worked further on the problem of deploying a PowerPoint add-in along with an auto loading User Interface toolbar. Unfortunately it cannot be done… at least simultaneously; it requires a 2-step process

 

Step 1 – load the “.ppam” file to the Applications/Microsoft Office 2011/Office/Add-in folder.

Step 2 - Place a “.ppsm” file onto the user’s desktop and instruct them to open the PowerPoint “Read Me” slideshow.

 

The “.ppam” add-in will not load a toolbar… this is because PowerPoint 2011 does not support “Events”, which means the VBA commands of AutoOpen, AutoNew, AutoExec, et cetera will not function. You must use a different method to load a User Interface toolbar that allow access to the “.ppam” macros.

To do this programmatically you can choose one of the following methods.

  1. Create a “.pptm” file that contains the macro to create the custom toolbar and ask the user to execute it from the PowerPoint Menu Bar > Tools > Macro >Macros… > Run > “your build custom toolbar macro”
  2. Create a “.ppsm” slideshow file that contains your “build custom toolbar macros” and that executes when the user clicks on a specific  slideshow “Action” button that you have assigned to your macro.

It seems to me that #2 (“.ppsm”) above is the better alternative. You can create the slideshow as a “Read Me First” instructional file and it tells the user about your custom add-in and how to access it. It does have a catch however…

Probably because of a timing delay problem inherent to Slideshow automation your macro will not complete fully unless you alter it with a user interaction. I found during testing that it would create the new “blank” toolbar but it would not populate it with the buttons. It doesn’t make sense because if you run the same macro manually via “Menu Bar > Tools > Macro > Macros… > Run” it works perfectly.

So I first attempted to insert a time delay between the VBA code that adds the new toolbar (bars.Add (CustomBar) the VBA code that populates it. But of course this led to another problem because PowerPoint 2011 VBA does not appear to support the VBA OnTime, Sleep, or Wait commands that you could otherwise use to put in a time delay. I ended up with code like the following, which does work fine in the “.ppsm” solution for auto loading a toolbar in PowerPoint.

Sub AddCustomToolbar()

    Dim cmdBar As CommandBar, cmdButton As CommandBarButton

    Dim bars As CommandBars, barMenu As CommandBarControl

    Dim bN As String

    On Error Resume Next

    bN = "George Test"

    Set bars = Application.CommandBars

For Each cmdBar In bars

        If cmdBar.Name = bN Then cmdBar.Delete

    Next

bars.Add (bN)

MsgBox "The new toolbar is created", vbOKOnly, "George for PowerPoint"

   

    For Each cmdBar In bars

        If cmdBar.Name = bN Then

            With bars(bN)

                .Visible = True

                .Controls.Add (msoControlButton)

                Set cmdButton = .Controls(1)

                cmdButton.style = msoButtonIconAndCaption

                cmdButton.FaceId = 1192

                .Controls(1).Caption = "Insert Equation"

                .Controls(1).Tag = "Equation"

                .Controls(1).TooltipText = "Inserts the Legacy Equation Editor"

                .Controls(1).OnAction = "GCL_RibbonClk"

            End With

        End If

    Next

MsgBox "Process Complete", vbOKOnly, "George 5"

End Sub

 

In the above code example the button I added is not meant to make sense for my test User Interface toolbar. I simply used some code I had and that would quickly prove if my workaround actually performed correctly.

 

I hope this helps and please us know if this workaround is viable for you.

About this part:

If you open your .xls workbook and it is running in Compatibility mode, you can turn it into a native Excel 2007/2010 file using File --> Save As.

When you get into the Save As dialog, in the "Save as Type:" section, choose the

Excel Macro-Enabled Workbook (*.xlsm)

entry.

It should actually keep the same name, only changing the file extension from .xls to .xlsm.  The file will be saved into the same folder the original .xls came from by default but with the new name, so you would end up with:

myFile.xls  and myFile.xlsm in the same folder.  It will not overwrite your original .xls file.

Hope this helps you understand that aspect of it all a little better.

OH - almost forgot.  After using the File --> Save As to change the file's type, you need to then close the file and reopen it to actually use the new features.  Even though initially it will show the filename as .xlsm, until you close and reopen it, it still acts like a .xls file running in Compatibility Mode.
