ken sheridan say: If I understand you correctly, it sounds like the [field product name] column is a foreign key column in a referencing table whose value is a long integer number in the hidden bound column of a combo box when viewed in the table's datasheet.  This will be the case if you've used the 'lookup field' wizard when designing the table.

A report should be based on a query which joins the current table to the referenced 'products'  table, and the text product name column from the referenced table returned in the query.  You will then be able to concatenate a literal string expression with the value of this column instead of the foreign key column from the referencing table.

For future reference, the 'lookup field' wizard is deprecated by most experienced MS Access developers, and should be avoided like the plague.  For reasons why see:

http://www.mvps.org/access/lookupfields.htm

Design flaw: Office [PowerPoint / Word] (2007, - Microsoft Community

fstallmann say:

Design flaw: Office [PowerPoint / Word] (2007, 2010, 2013) derives proofing language from keyboard layout

I'm posting this here as I have found no other way to report this bug (or rather unhelpful feature).

I am working with Office 2013 on Windows 8.1, but the behavior has been consistent since at least Office 2007. My system language is English; in Office, I have English and German proofing tools installed; *English is set as the default editing language in the Office language preferences*, and pretty much all documents that I edit are US English.

However, my hardware, in particular my keyboard, is German, so I am using German (DE) keyboard layout.

Now, whenever I create a new element such as a new text box (or even add a new bullet point) in PowerPoint, the editing language is set to German regardless of my default editing language (or the language set for the previous bullet). [Edit: The same is true for creating a new document in Word.] While language autodetection works reasonably well in Word, this does not seem to be available in PowerPoint anymore; the English text I enter remains classified as German and is flagged by the spell checker. This behavior is very annoying and a design flaw (made worse by the fact it is impossible to simply select all slides and set the proofing language for all elements on all slides at once).

Whoever designed this meant well, but ultimately fell into a trap of US-centric thinking. "I am typing US English on my US keyboard. Now I want to type French. Oh, they have all these weird diacritical marks not present on my keyboard. I guess I have to switch to a French keyboard layout to write efficiently. Hmm, why would I ever switch to a French keyboard layout if I didn't want to write French? So I can safely infer that I am typing French when I use the French keyboard layout!" 

Unfortunately, this does not hold true for the rest of us from the countries using these weird diacritical marks. Our keyboard layouts are usually supersets of an English language keyboard (at least for normal text - German programmers pay for this by having to use ALT GR to type { [ ] } and \), so there is no reason whatsoever to change the keyboard layout in order to type English text. It is actually highly counterproductive as some letters (e.g. y and z for Germans, even more keys for French users) switch around, and people would have to adjust their touch typing when they could just as well keep using their accustomed layout. Non-touch typists are even less inclined to use a layout that deviates from what is printed on the keys. Personally, I don't know anybody who switches to US layout for typing English on German hardware, and I work for a German company where 80% of the internal communication is in English.

In short, determining language based on keyboard layout is a bad idea, at least for non-English latin alphabet languages. It would be ok to keep this as an optional feature - as mentioned above, I see this working for US users, and probably also for languages with wildly different character sets - but it absolutely needs to be possible to switch this off in the language preferences. A simple checkbox 'use keyboard layout to determine editing language' would do it.

This may only be at the level of an annoyance, but it affects millions of users. I just hope somebody reads this here who knows how to forward this to the right people...

How can I number different sized merged cells automatically? - Microsoft Community

abu hashem al hasani say:

How can I number different sized merged cells automatically?

How can I number different sized merged cells automatically?
rick rothstein, mvp - excel say:

Give this a try. Select all the cells you want to number in Column A (actually, any column will work) and then run this macro...

Sub NumberCellsAndMergedCells()
  Dim Cell As Range, Number As Long
  Set Cell = Selection(1)
  Number = 1
  Do While Not Intersect(Cell, Selection) Is Nothing
    Cell.Value = Number
    Number = Number + 1
    Set Cell = Cell.MergeArea.Offset(1)
  Loop
End Sub

ken sheridan say: The DefaultValue property does not initiate a new record per se, it merely sets a value for when a new record is initiated.  You'll see the value in the empty new row, but no new record will as yet have been initiated, i.e. the subform's Dirty property will not  be True.

If you are seeing the pencil icon without entering anything into the subform manually, i.e. its Dirty property is True, it suggests that a value is being assigned to a column in the subform's recordset in some other way than by setting the DefaultValue property, e.g. in code in an event procedure in the subform's module.
