Need to AutoFill based on TWO criteria - Microsoft Community

gp_racer say:

Need to AutoFill based on TWO criteria

So, I am (attempting) to develop a database for students going through our class.  I have the following tables (actually a lot more, but these are the relevant ones)

tblStudents

tblGrade

tblService

tblStudents has some drop down lists that are selected from tblGrade and tblService

tblServce has these options: Army, Navy, AirForce, Marines

tblGrade has these options: E-1, E2, E3, E4, E5, E6, E7, E8, E9, WO1, CW2, CW3, CW4, CW5, O1, O2, O3, O4, O5, 06, 07, 08, 09, O10

Within tblStudents, I would like to take the value of tblService and tblGrade, and then apply the correct rank without user input into the Rank field in tblStudents.

For example: Student PK 1, is in the US Army and his grade is E-5... the autofill would input SGT into the Rank Field.

The issue is, for a given grade, the different services call those ranks different things.  Another example:

Student PK1 has a grade of E-7.  If he is in the Air Force, he is a MSgt.  If he is in the Marines, he is a GySgt, If he in the Navy, he is a CPO.  If he is in the Army, he is an SFC.

So basically, I need this field to analyze those two fields and apply the correct rank, but have ZERO idea how to do it.  

Your help is very welcome.  

john w. vinson say:

It sounds like there's some redundancy here: if you can unambiguously determine the Rank given the Service and the Grade, then in a fully normalized table, you would not actually store the Rank AT ALL - just look it up as needed. In fact, it could be a real mistake to store it; say you had student 23 in the database as an E-7 CPO in the Navy. She sees her listing and storms into your office saying "I'm no Swabbie, you nitwit, I'm a MARINE!". Even if you correct her service, the CPO for Rank will still be in the table - it's wrong but as you have redundant fields, there's no easy way for Access to detect the inconsistency. 

However, I can see that you may want to store both the Grade and the Rank in the table nevertheless. "Judicious denormalization, only when necessary" is in some developers' vitas. 

Either way, you need a three field translation table with fields for Service, Grade and Rank with the correct translation. You could then create a Query joining the personnel table, joining Service to Service and Grade to Grade to pick up the corresponding Rank. You will need to make a considered judgement whether to use this Query whenever you need a Rank, or to make it into an Update query to actually store the Rank in the personnel table. If you do so, you should use some validation code to be sure that the ranks are all correct.

ken sheridan say: The other issue you might need to consider here is that of 'time-independence'.  What you have is a ternary (3-way) many-to-many relationship type between students grades and services, which would be modelled by a 3-column table with three foreign key which reference the primary keys of the three referenced tables respectively in the way John describes.  The primary key of this table is a composite one of the service and grade foreign key columns.  

If you can be absolutely sure that the determination of rank by service and grade will remain the same for all time, i.e. it's time-independent, then you do not need to store the rank in the students table, and should not do so as it would introduce redundancy and the consequent risk of update anomalies.  Any form or report simply needs to be based on a query which joins the tables to show the rank.

If, on the other hand, the determination of rank by service and grade is not time-independent, i.e. it might change at some time in the future, then you should store the rank in the students table.  The way to do it would be to put code in the AfterUpdate event procedures of both the service and grade controls in the data input from which looks up the rank from the table modelling the relationship type and assigns it to a rank column in the students table.  The code would need to be conditional  on neither the service nor grade column being Null of course, as you cannot say for certain which would have a value entered first.

I don't know whether this would apply in the USA, but in the UK another determinant of rank would be the regiment in which the student serves, as, in the army at least, ranks can differ between regiments.  The ranks of Lance Corporal and Corporal in most of the army are Lance Bombardier and Bombardier in the Royal Artillery for instance.  So the table modelling the relationship type would require a fourth column as the relationship type is now a quaternary (4-way) one.

ken sheridan say:

The answer is quite simple.  There are right ways to do things, and wrong ways.  The wrong ways are likely to result in a database which is inefficient, unreliable and possibly dangerous in that it will give the wrong answers to the questions asked of it.  Over the years I've done a lot of work on clinical databases.  Would you be happy with a database of clinical results which said you were perfectly healthy and didn't require treatment when you actually had a life-threatening condition?

If someone is clearly doing something wrong it's part of our role as respondents to tell them this and guide them as to how to do it correctly.  

As regards what John calls a 'translation' table, in your case it would have rows like this:

Service       Grade    Rank
Army          E-5        Sgt
Air Force    E-7        MSgt

The primary key of this table is a composite one made up of the Service and Grade columns, because the combination of values in these two columns must be distinct.  Consequently in the students table the columns Service and Grade act as a composite foreign key.  Rank in the above table is a foreign key column referencing the primary key of a table in which rank is the primary key.

You don't need to 'autofill' a Rank column in a Students table; you simply join the two tables in a query to return the student's rank (assuming the answer to the question which I posed in my last reply is no, and the determination of Rank by Service and Grade will remain fixed for all time).  If you did have a Rank column in the students table it would be redundant and there would be nothing to stop its value being changed to one which is not determined by the student's Grade and Service, resulting in incorrect can go wrong, it will go wrong'.

From my view, the concept of "a Student's Rank at the time they took the Class" is an attribute of that Student at the time they took that specific Class (as Ken indicated above). If that's the case, then it should be stored with the Student + Class, but not with the Student (or the Class). I assume you have a structure for storing data like this:

With a setup like this, you'd store the StudentRank at the time they took the Class. You may already have this, but of course we don't know your table schema.

So the question is: In your source data, do you have the Student's Grade and Service data for the point in time at which they took the Class? If you do, then you could use a Rank table that will "map" the Service and Grade to the Rank:

tRank

-------------

RankID

ServiceID

GradeID

Fill that table, and you can then pull the needed information from that table when adding a Student to a Class. For example, in a Form where I might add my Student to the Test I might do this after the student is selected or input:

Me.Rank = DLookup("RankName", "tRank", "ServiceID=" & Me.StudentServiceID & " AND GradeID=" & Me.StudentGradeID)
