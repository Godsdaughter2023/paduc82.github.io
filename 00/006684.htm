I don't believe there is a built-in method to do this, but it is easy enough to achieve with a pair of macros assigned to keyboard shortcuts of your choice. The macros increase or reduce the zoom level by 5% with each activation. You can change the 5 to a preferred figure if you wish to have greater or lesser increments.


Sub MyZoomIn()
Dim iZoom As Long
iZoom = ActiveWindow.View.Zoom
iZoom = iZoom + 5
ActiveWindow.View.Zoom = iZoom
End Sub

Sub MyZoomOut()
Dim iZoom As Long
iZoom = ActiveWindow.View.Zoom
iZoom = iZoom - 5
ActiveWindow.View.Zoom = iZoom
End Sub

a. user say:

> Why would it truncate in one field and not another?

As background...

Outside Microsoft (which is where we are in these groups), the "rules" for truncation have been assembled from what various pieces of Microsoft documentation say, plus information gathered from experiment. Without knowing what Microsoft's code actually does, it's impossible to be sure that every possible case has been covered. For example, it appears that values generated from Excel formulas do not always behave the same way as constant values inserted in a cell. (Until a couple of weeks ago, I had a web page about these rules, but it's down at the moment).

The component that Windows Word (2002 and later) uses to get merge data from an Excel workbook is the Jet (later, ACE) OLE DB provider and an Excel-specific component that it calls. An older ODBC driver works in a similar way, but is not identical. The basic problem with using this provider is that it's designed to return data in a way that is compatible with Access and other relational database packages. Acces insists on having a specific data type for each column, and every value in that column has to have thattype. In Excel, of course, each cell in a column can have a different type. So when the OLE DB provider gets data from an Excel column with mixed data types, it (a) has to decide a specific type for each column and (b) has to do something about all the data in each column that has another type.


To complicate matters, for historical reasons, there are two text types: "text" (up to 255 characters) and "memo" (can be longer than that), and there are rules that deal with that, too. I wish it were not so, but it is.

So, just to be sure I have understood, the column that "works" does not have a text longer than 255 characters in the first 8 cells in that column?

The "8 cells" rule is the default rule. The number of rows that the OLE DB provider checks can be changed in the Windows registry. I doubt if many people change the rule (I wouldn't advise it because of the possibility that other software that relies on the rules being a certain way will break), but it is possible that a different number has been specified on your PC. It used to be possible to specify 0 rows, which would make the provider check a large number of rows (if it was actually "all rows", it might be worth doing, even though you then have to wait for the provider to do all that checking. But it isn't).

And as I said earlier, there may be other rules that no-one has yet documented.

So the answer to your question "How do I know what field is going to be a problem?" is, I am sorry to say, "you don't". The best you can really do is try to ensure that you avoid the problem in the first place, or use the kind of workaround we have already discussed. Precisely how much effort to put into that depends on the situation - if you are creating your own data and using it for your own merges, you now know roughly what you have to do. If you're trying to design and create a reliable business process, I would try not to start from data created in Excel :-)

doug robbins - mvp office apps & services (word) say:

Word sets the type of data in each field by looking at the data in the first record in the datasource.  The reason that some of your datasource merge OK would be due to the first record in those sources containing more that 256 characters in the the field that is giving an issue with other data sources.

If due to the method of generation of the data sources, it is not feasible to have a record with more that 256 characters in the field(s) with which you have an issue as the first record in the data source, you may be able to work around the issue by clicking on the Office button and then select Word Options and then go to the Advanced tab and scroll down to the General Section and then check the box for the "Confirm file format conversion on open"   After having done that, when you attach the data source to the mail merge main document, you will be asked for the method by which the connection should be made.  Using the DDE method may overcome the issue.

-- Hope this helps.
