hansv mvp say:

Open the form in design view.

Select the Name text box.

On the Format tab of the ribbon (under Form Design Tools), click 'Conditional Formatting'.

Click 'New Rule'.

Select 'Expression is' from the 'Format only cells where the' dropdown.

Enter the following expression in the box:

DCount("*","TableName","[Name]='" & [Name] & "' AND [ID]<>" & [ID])>0

where:

TableName is the name of the table the form is bound to.

Name is the name of the field/text box containing the name.

ID is the name of the primary key of the table (preferably an AutoNumber field)

Select the desired formatting, then click OK.

ken sheridan say: In Access, like all relational database systems, a personal name should not appear in the same table more than once unless they are two or more different people with the same name.  Because personal names can legitimately be duplicated, they are unsuitable for use as keys; a person should always be identified by a distinct numeric value.  If we take a simple example of patients undergoing treatments this would be modelled, in broad outline, by tables like this


Patients
....PatientID  (PK)
....FirstName
....LastName
....etc

Treatments
....TreatmentID  (PK)
....Treatment

The relationship type between patients and treatments is many-to-many, so is modelled by a table like this:

PatientTreatments
....PatientID  (FK)
....TreatmentID  (FK)
....TreatmentDate

The primary key of the last table is a composite one made up of all three columns, i.e. the two foreign key columns and the TreatmentDate column.  This table does allow duplicate PatientID table of course as a patient might have more than one treatment.  Duplicates in this case can be highlighted in a form in the way Hans has described, but the expression would be slightly different:

    DCount("*", "PatientTreatments", "PatientID = " & [PatientID]) > 1

Note that this would not be implemented until the record has been saved, however, as until then it is not committed to the table and consequently not included in the count.

The problem here is that blanks are meaningful. You see Assigned To as one name; Access sees it as two unconnected words, "Assigned" and "To", and doesn't understand you. 

If you choose (unwisely IMO) to use special characters such as blanks in field or object names, you must wrap them in [square brackets] to tell the program that this is one thing: try

DCount("*","TableName","Active='" & [Assigned to] & "' AND ID<>" & ID)>0

Note that none of the object names in the above include spaces.  If you have unwisely included spaces in any object names, remember that they must be enclosed in square brackets in the code, as John has pointed out.

forgiven---01 say: Delivery reports are pretty reliable when you send sms to a correspondant who is on the same network as you, and who has a good working handset. In the case of sending a sms to a foreign number, operators can never garantee the good delivery simply because they have no control over the other networks.

In the case of overloading(in a vacation resort for ex), networks can delete incoming sms from abroad to prevent their own subscribers to not be able to send/receive sms.
