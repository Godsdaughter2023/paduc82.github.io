albert d. kallal say:

Actually VBA code ports VERY well over to vb.net code.

 

They are VERY similar in syntax.

 

The problem is not so much converting code modules (separate from forms) to vb.net. I find that such code ports extremely well over to vb.net

 

And VBA code does support class objects, and I use them all the time in Access. So as far as a programming language, you don’t want to confuse Access “macros” with that of VBA (Visual Basic for Applications). VBA is a VERY powerful and able language.

 

 

The vast bulk of your code in Access will be VBA, not macro code.

 

As noted, VBA code modules (and even class objects) convert VERY well to .net if you coding in VB.net.

 

However, the problem is NOT really the syntax of VBA vs that of Vb.net (they are remarkable compatible and similar).

 

The huge issue and problem is Access gets all of its “RAD” ability (rapid application development) from the fact of what we call bound forms. In other words, code to fill a form, code to edit a form is NOT in general requited. And Access bound forms have a gazillion data events (such as on-insert, on change, before update, after date). The whole power in Access that saves your software budget of about a factor of 5 times less cost is this bound forms model. You don’t have these events in .net forms, and thus the issue not the syntax of VBA code, but that of the vastly different event model Access has for bound forms (of which you don’t have in .net).

 

So a $10,000 project in Access will cost you $30,000 to $50,000 in .net.

 

So there is no conversion system say from FoxPro to .net. and there is no real conversion system say from Pascal to VBA, or say in your case (and that’s been the way of the computer industry for 30+ years (so zero surprise that you looking for a magical unicorn that never existed in our industry).

 

However, at the end of the day, I can say that VBA code modules (not form ones) port over to vb.net VERY well and VERY little syntax changes are required. However, because Access does not separate UI code, and forms code, then the code in forms is where the “bulk” of the value + effort in Access was no doubt where developers spent their time. And this area has the least possibility of porting over to .net – even if a really good code converted existed, the whole design philosophy is so different, that your effort and time is not code converting, but converting the logic used in Access to drive that form – that kind of code logic does not port at well to .net EVEN if code syntax was the same. The problem is not code syntax, but the vastly different event model, and the data bound forms in Access that don’t exist in .net.

josh black (josh.black) say:

Agreed with Albert.

Transferring your code isn't that bad. It's the other parts that make a transition to .Net difficult. I've transferred a few smaller Access applications to .Net. The code portions were the easiest part, and the RAD portions (data binding with automatic CRUD, reporting, active-record pattern, etc.) took the longest to replace.

We have a much bigger Access application we've been talking about upgrading for almost a decade. The reason we haven't been able to make the transition has very little to do with the VBA code.

WPF and UWP and even Xamarin have great data binding, but it's nothing like what you have built into Access forms and controls. You also have to rewrite the data access code to get your data into the UI, most likely by providing objects via an ORM like Entity Framework.

In our application, the biggest hurdle is actually figuring out what to do with reports. We have hundreds of reports in our Access database. Stand-alone reports that users pull with a filter form could just remain in Access until we can recreate them in something like SSRS. Most of our reports, though, are generated directly from the "active record" on a form, such as letters customized for the customer visible on the form, or an order report for the currently selected order. Our users would revolt if we told them they had to leave the new application, go back to Access, and find the same customer or order to get the report. I've tried generating an Access report directly from .Net, but haven't had much luck. So, we need to also replace the "current record" reports to run in something like SSRS with the native report viewer before at the same time we migrate our data forms.

Even for the VBA code migration, I don't think I'd recommend trying to transfer it to vb.net. While similar, there are important differences. Also, you'll have an easier time finding developers to manage c# code rather than vb.net. Finally, Access encourages some bad design patterns, such as mixing UI logic with business logic. It's pretty rare for an Access database to have business objects as classes or modules and have the form interact with those objects. Rather, you usually see a bunch of logic behind buttons on the forms. You'll probably want your new code to have a better separation of concerns to make it easier to test, reuse logic, and understand.

RAD tools are nice. But they don't encourage good design and. When what started as a RAD application turns into a complex system, it can be quite painful to transition.

If you create an AD user without an on-premises mailbox, you should run Set-RemoteMailbox cmdlet. If the user is created from on-premises, but its mailbox is provisioned in Exchange Online, then you need to run enable-remotemailbox.
