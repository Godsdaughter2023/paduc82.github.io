nathaniellindsey say:

Working with gridlines in a template in Word 2010

I'm working in Word 2010 with a template that will NOT show gridlines no matter how many times I select View > Gridlines.  I've fixed the template once before, so I know there's a way, I've just forgotten what it is.  Can someone help?
jay freedman say:

Word has two different things it calls "gridlines", so let's be sure which one is being discussed. When you insert a picture or drawing, you can get a graph-paper type of grid that covers the whole page. When you insert a table and turn off some or all of its borders, you can display thin dashed lines that show the edges of the table cells. Each of these has a "View Gridlines" setting, and they don't interact with each other.

When you select a table, the Table Tools pair of tabs appears on the ribbon, and the Layout sub-tab has a View Gridlines button. That turns the table gridlines on or off, but does nothing for drawing gridlines.

On the View tab, there's a Gridlines check box, which turns the drawing gridlines on or off, but does nothing for tables.

There's a Grid Settings dialog that modifies the drawing gridlines, and something might be set wrong in there.

In a document, insert a picture or a drawing shape. While that's selected, the Picture Tools or the Drawing Tools tab appears on the ribbon. On either tab, click the Align button near the right end and click Grid Settings.

Here's what the settings should look like for the default grid:

The check box for "Display gridlines on screen" should always have the same state as the Gridlines box on the View tab and the View Gridlines item on the Align button's menu -- change any one of them, and the other two should change the same way.

If none of this helps, but if gridlines work correctly in other templates, this template might be corrupted in some way. You can try copying everything in the template except the very last paragraph mark, and pasting that into a new blank document. If that one works, save it as a template and discard the old one.

rob koch say:

Again no one seems to understand why Microsoft security products truly exist.

The MSRT (Malicious Software Removal Tool) was created for just that purpose, to remove a relatively small number of the most common malicious software programs that had been widely distributed and due to their highly infectious nature, have affected many Windows systems often for many years up to and including today.

The MSRT is automatically downloaded with the other monthly Windows Security Updates in order to determine whether a system contains any of these common malware, primarily as a sanity check to determine whether the installed antivirus or other security applications on that particular PC are operating properly.

The advantage of the MSRT is it's a relatively small application that doesn't require installation and instead simply downloads and runs once, whether automatically with Windows Updates or manually, so it's not as easily disabled or crippled by malware as installed security apps like Defender or 3rd-party AV products.

MSRT has never been in competition with MSE, Defender or any other Microsoft security product, it's designed to compliment these similar products and perform automated or manual sanity checks when these or other 3rd-party installed security apps may be in a questionable state or known to be crippled.

Comparing the MSRT to any other security product has no meaning and of course it's still needed, for exactly the reasons it was originally created.

Rob

rob koch say:

Billy,

You can't use the built-in components of a system to fully insure the integrity of that system itself, the sanity check or verification must be done via some external monitor to insure that other core elements of the system haven't been compromised in order to cripple the built-in functions such as Windows Defender or another AV.

Microsoft actually attempted to add something similar to what you are thinking by integrating Windows Defender Offline into the operating system, though in truth that was done more to make the deployment and operation of the offline version easier for the user.  Even that version depends on other pre-boot operating system components contained on a separate partition, so it only works properly when all of those additional pre-boot components are functioning as well.

My entire point wasn't that anyone is wrong, since there are times when using the built-in Defender is better, as well as times when using 3rd-party products might make more sense.

The primary point is that assuming the MSRT no longer has a purpose is flawed, because it's still completely valid in terms of its original purpose to sanity check the operation of the installed AV product.  It doesn't matter whether that AV is Defender or a 3rd-party product, since the point is to verify whether that security app is functioning to protect the system as it should.

What you're probably missing is the fact that just like Windows Defender itself, the MSRT and in fact all current Microsoft Security products are reporting their findings back to the MAPS cloud in order to inform Microsoft which known or potentially new, unknown malware they have detected, allowing both the automated and manual processes performed by Microsoft's security analysts to extend the protection provided beyond what currently exists.

It's the automation of this latter portion of data collection and even analysis to some extent that makes the Microsoft Security products so powerful, while most of the other things that the general population believes are important often have less impact or are at least improved greatly due to this ability for Microsoft to view the entire Windows ecosystem from their artificial intelligence systems including MAPS.

Running a 3rd-parrty security product like Malwarebytes may serve to remove a specific set of less malicious PUP type software that Microsoft doesn't consider true malware from a specific system, but this does nothing directly to inform Microsoft via MAPS of any real malware it does find.

Rob
