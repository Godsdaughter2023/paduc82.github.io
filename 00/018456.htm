charles kenyon say:

I am a bit confused. Is your table something to use in a document, that your macro adds rows and contents to, or is it intended to be used to create building blocks that can later be used?

Assuming the second, your Autotext entries may well carry with them character formatting and or styles. You seem to be inserting table rows. Those will automatically have paragraph formatting.

If you are carrying paragraph formatting in your AT entry, the best you can do is have it use a style that is already in use, in which case it will adopt that style's definition from the document.

You do not need a macro to insert a building block in that you can attach a keyboard shortcut directly to such insertion. That shortcut can be stored in the same template that stores your building block.

As Graham Mayor pointed out, if you are using a macro, you can first gather information at the insertion point on formatting and apply that formatting to the inserted text. I've got this code that grabs table entries and makes building blocks for every row.


Public Sub updater_Click()

Dim aTextDoc As Document
Dim cTable As Table
Dim rName As Range, rText As Range
Dim i As Long
Dim x As Long
Dim sFname As String
sFname = "C:\Users\edwardp\Desktop\Format Notes.docx"
Set aTextDoc = Documents.Open(sFname)
y = aTextDoc.Tables.Count

For x = 1 To y
Set cTable = aTextDoc.Tables(x)
    For i = 0 To cTable.Rows.Count
    Set rName = cTable.Cell(i, 1).Range
    Set rText = cTable.Cell(i, 2).Range
    rName.Font.ColorIndex = wdBlack
    rText.Font.ColorIndex = wdBlack
    rText.End = rText.End - 1
    rName.End = rName.End - 1
    If rText = Null Then
    Set rText = cTable.Cell(i, 1).Range
    End If
    NormalTemplate.AutoTextEntries.Add Name:=rName, Range:=rText
    Next i
Next x

aTextDoc.Close wdDoNotSaveChanges
MsgBox "Successfully updated building blocks (Click OK to continue)", vbOKOnly, "updating building blocks"

End Sub

cortig say:

I don't know why that message got tagged as being the answer to the original message since it's about a completely different matter.

The boot-related issue is irrelevant to calendar and contact syncing.

Here's the situation with a bit more details than my previous post:

  • The only type of server Outlook can sync to is Exchange.
  • iCloud and Google use other types of servers (CalDAV and CardDAV). Outlook doesn't support them, but previous versions of the System offered a workaround: You could configure the accounts in the System apps (Contacts and Calendar/Address Book and iCal) and an internal mechanism called Sync Services allowed other apps to directly sync locally to the same resources. It had its bugs, but it sort of worked for a number of people.
  • In 10.10, Apple told everybody that the local syncing mechanism was going away and people had to stop using it. In 10.11 (El Capitan), they effectively took it away. So in El Capitan, that workaround no longer exists and Outlook essentially has no way to sync to anything related to iCloud and Google calendar and contacts. No System update is going to change that…

Obviously, the solution now would be for Outlook to (finally) support CalDAV and CardDAV. Lots of people have been asking for it but it's still not there.

As I was saying, if that's an important feature for you, I would strongly encourage you to use the Send a Smile button on the top right corner of the main window in Outlook to let MS know, and help them prioritize future development. If there is enough pressure, they might change their mind and support these protocols.

Hi Kon,

Question 1: Merge AD of company B in to company A

You can use the Dell AD Migration tool to merge domains. Since it is a third-party tool, it is not supported in our forum. Your understanding is highly appreciated.

You can also use the ADMT tool (Active Directory Migration Tool) to merge two domains into one. Here are some references:

ADMT Guide: Migrating and Restructuring Active Directory Domains

Active Directory Migration Tool (ADMT) Guide: Migrating and Restructuring Active Directory Domains


Question 2: Setup Office 365 under company A and configure same-password with AD Connect

For this question, you only need to enable the password sync when setting up AD Connect. The AD password will be synced to Office 365 cloud.

Question 3: Move all users from Company B (SBS2011) directly in to the Office 365 under Company A (as there is no capacity to host users on On-Prem)

About SBS servers’ migration, based on my experience, you can use IMAP or Cutover migration to move data to Office 365.Here are some references:

Migrate other types of IMAP mailboxes to Office 365

Perform a cutover migration of email to Office 365

About your plan, I think it is correct. However, I just want to remind you that you may need to use SMTP match to connect AD users and cloud users. Since you directly move user B (users in company B) to Office 365, it means mailboxes (accounts) will be existed in cloud. After that, you enable the AD Connect to sync users and it may have some issues. Here are some steps I think: (I assume you still use the CompanyA.com and CompanyB.com for two company users.)

1.    Create Office 365 tenant under Company A.
2.    You can merge two AD domains and create different OUs for company A and B users. Meantime, you can use different OUs to filter users who you want to sync. You can only sync company A users at this step.
3.    Install and enable AD Connect to sync Company A users.
4.    Migrate company B data to Office 365 (the tenant under Company A). You can create a test user to migrate first.
5.    Use SMTP Match to connect AD company B users and cloud Company B users.

It is my suggestion. You can do it based on your actual situation. If you have any questions, please feel free to let me know.
