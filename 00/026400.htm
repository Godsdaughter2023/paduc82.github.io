phillip m jones say:

You have it set a little too fast. Less than 5 Minutes doesn't give it time to save a snapshot.

  1. Go to Word Preferences
  2. Locate Setting for Auto Recover and if you are going to use it, set it for no less than 5 minutes. (And that is on the fine edge of reliability.)
  3. Make sure the checkbox for using it is checked.

As for looking in the Folder where auto recover files are kept I thought you could only see items in the folder only as long as Word was open and running.  These type files are supposed to do an auto dump into the trash as soon as the application has quit.

A much more reliable method than auto-recover. Is to set Word to create a Backup File.

It requires a little more effort on you part. And it will take up Room on your hard Drive

The difference between Auto-Recover and create backup Is that Auto-Recover depends on Word Crashing to Take that snapshot of that 5 minute period to recover information.

Create Backup on the other hand, Physically creates a backup file with all information up to the point you hit "save". You need to get in the habit of touching ⌘-S every time you pause to think about what you want to write next, or scratch your nose. If you "save" often enough you should be able to open the backup file and fill in missing parts. The big advantage is, suppose you add a Paragraph, and save, then re-read and discover what you added was said wrong or didn't convey the message you desired. You can open one of the previous versions and make you alterations having left out the part you want to re-work.

To turn on "Create Backup":

  1. Click on Word menu.
  2. Choose preferences
  3. Choose Save
  4. Tick checkbox beside of Create Backup.

Now get in the habit of doing  ⌘-S about every 10 minutes. And leave on auto-recover.

Once you finish a Project you can go through the "Backup of  ...." (whatever the file name) they will be dated and timed. And remove all of them except maybe for the last one or two.

Hi there,

I feel your pain. I was working on my thesis proposal last night for HOURS and suddenly Word decides to crash. To be honest, I'm not sure if this will help but take a look anyway, this was my fix.

This is a fix for people who; 1) Did not "Save as...", 2) Did not have AutoRecovery/Save features turned on, 3) Did not have a recent Time Machine Backup OR SnapShots. This is also a fix for people who remember what time their Word application crashed.

I exhausted all forms of recovery (looking in library folders under AutoRecovery, looking under "Open Recent...", ect.). I finally found something interesting.

Follow this path:
- "Users",
- "<Your Account>" (In this case, it was my name)
- "Library"
- "Containers"
- "com.microsoft.Word".... by this point you're probably saying to yourself, "I've done this already..." but keep up with me here......next, go to...
- "Data"
- "Library"

- "Application Support"
- "Microsoft" and finally.....

- "Temp"....

This is it. This is where my mind was blown and I figured out the meaning of life and existence. This is also where I found my .tmp (temporary) word document file that I thought I had lost. All you need to do is look for the most recent file before the crash. At this point there a few things you should do. First, if you find a .tmp file that resembles a recognizable time-stamp (for me, I saw that a .tmp file was modified at 2:56 am which was right before word crashed, so this signaled to me that this is probably the file I was looking for), make a copy of it on your desktop. Next, open that copied document in Text Edit. Now this is part is bittersweet. At this point, the good news is that the file you've opened SHOULD contain all of your changes up until the application crashed, the bad news is that this .tmp file records every single keystroke in ORDER, meaning your file will look like a Jackson Pollock painting (if you don't know what his art looked like, look at it and you will understand) except with letters. In essence, Sentences will be out of order. Words will be cut off. Portions of sentences will be out of place as well.

This is, however, as best as I have been able to recover my file and the only method that has worked for me. It has been difficult to decipher, but I have my work back. It will save crucial time and energy that you already put into the document when you typed it to begin with.

I'm hoping this will work for you. Please let me know and Good luck!

Hello,

simple question but could only find vague answers on the net.

I read that the way to do it is using the After update instead of ONchange  and the sum will be displayed once a textbox loses focus.

Isn't there a way to use ONchange without vba?

Thanks

ken sheridan say: Whichever event you use you'll need to write some VBA code (or at the very least a macro), but it's minimal.

First set the ControlSource of the third combo box to an expression which adds the values of the first two, calling the Nz function in the expression to return a zero if either of the first two control's is Null, e.g.

    =Val(Nz([text1],0))+Val(Nz([text2],0))

Then in the Change event procedure of each of the first two controls put:

    ' recalculate the form
    Me.Recalc
    
    ' move insertion point to end of value in active control
    With Me.ActiveControl
        .SelLength = 0
        .SelStart = Len(.Text) + 1
    End With

Note that if the first two controls are bound, the current record will be saved as soon as the value in either is changed.

ken sheridan say: The following article includes guidance on the use of the Hijri calendar in Access:

http://office.microsoft.com/en-us/access-help/about-using-different-calendars-HP005257922.aspx

Try using the CDate function rather than a date literal:

     strCriteria = _
         "InstructorID = " & Me.InstructorID & _
         " And Start <= " & CDate(Me.txtEnd) & _
         " And End >= " & CDate(Me.txtStart) & _
         " And ID <> " & Me.id

As you are concatenating the value into a string expression, you may need to convert the return value of the CDate function to a number, however:

     strCriteria = _
         "InstructorID = " & Me.InstructorID & _
         " And Start <= " & Clng(CDate(Me.txtEnd)) & _
         " And End >= " & Clng(CDate(Me.txtStart)) & _
         " And ID <>" & Me.id

The date/time data type is implemented as 64 bit floating point number staring at 30 December 1899 00:00:00, with the integer part representing the days and the fractional part the time of day.  As you are concerned solely with dates here using the CLng function to return a long integer number should give you a value which is directly comparable with the underlying numeric values in the Start and End columns.

Another possibility would be to use the following function of mine:

Public Function WithinDateRange(dtmParamRangeStart As Date, _
                       dtmParamRangeEnd As Date, _
                       dtmDataRangeStart As Date, _
                       dtmDataRangeEnd As Date) As Boolean
                       
' Accepts: date at which parameter range starts
'          date at which parameter range ends
'          date at which data range starts
'          date at which data range ends
' Returns: True if data range intersects with parameter range
'          False if whole of data range outside parameter range
 
   WithinDateRange = _
       dtmDataRangeStart <= dtmParamRangeEnd And _
       dtmDataRangeEnd >= dtmParamRangeStart
       
End Function

This just wraps up the same logic as in your expression in a function, so you'd call it like this:

     strCriteria = _
         "InstructorID = " & Me.InstructorID & _
         " And WithinDateRange(" & CDate(Me.txtStart) & _
         "," & CDate(Me.txtEnd) &",[Start],[End])" & _
         " And ID <>" & Me.id

As the function's arguments are all declared as date data type it should correctly interpret the values passed into it.
