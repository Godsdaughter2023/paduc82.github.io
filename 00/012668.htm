joshhhhhhhhhh say:

Is it possible to change the picture import settings in Word so photos are automatically sized and rotated?

As the title suggests I would like to know if there is any way I can automatically set the size and rotation of  pictures when i drag and drop them into a word document. I fill in a lot of reports and when I have to adjust hundreds of photos that need to be all the same size and rotation it takes up a fair bit of time.
macropod say:

The macro code for this isn't exactly obvious and isn't something you could generate with the macro recorder. The following macro rotates and resizes all inlineshapes in table cells in a document.

Sub Demo()
Dim iShp As InlineShape, Shp As Shape, sngHght As Single, sngWdth As Single
For Each iShp In ActiveDocument.InlineShapes
  With iShp.Range
    If .Information(wdWithInTable) = True Then
      Set Shp = .InlineShapes(1).ConvertToShape
      With Shp
        .IncrementRotation 90#
        .ConvertToInlineShape
      End With
    End If
    sngHght = .Cells(1).Height
    sngWdth = .Cells(1).Width
    With .InlineShapes(1)
      .LockAspectRatio = True
      .Height = sngWdth
      If .Width > sngHght Then .Width = sngHght
    End With
  End With
Next
End Sub

eriqvanbibber say:

Please be aware of the impact of public DNS and responsiveness of Outlook and other apps that connect to Office365.

There's an endpoint testing tool that can help you understand if you are being sent to some far off endpoint.

www.priasoft.com/.../

This tool will do a "TCP Ping" to all the known Microsoft Office365 endpoints world-wide from your host and will rank them by how quickly they respond.  We have found that about 4 time out of 10, clients are not connecting to the nearest Microsoft endpoint.

The importance of this has to do with network latency and not bandwidth.  Latency is the delay inherent in a WAN connection, often simply due to physical distance.  The farther away from a particular endpoint, the more delay.

Since public folders (and mailboxes too) are not stored in a file-system, latency become very important to understand with regards to responsiveness and usability of applications that connect to Office365.  Consider that for each item and each call to retrieve each request of data.

Furthermore, since items in Exchange are not files, when a request for an items data is made, SEVERAL transactions occur, not just one.  If a typical item has a body, and attachment, and one or more recipients, that can mean 3 or more network trips just for one item!

When you attempt to copy or move an item from a mailbox to a Public folder, especially if you are running in cached-mode, then you are actually copying data from the local host up to Office365.  MAPI isn't yet cool enough to see that you are trying to move

So, one of the very first things to review is your network latency to outlook.office365.com and to monitor the consistency over time to that hostname.  You will likely find that you need to "lock in" some IPs to guarantee local clients are always communicating with the endpoints with the lowest latency.
