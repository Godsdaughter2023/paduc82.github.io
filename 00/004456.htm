When we ask for the error message we mean all of the message. You have only provided the error code number. We also need the message that goes with it because an error number 1004 can mean a number of different problems.

An explanation of the code.

Option Explicit should be the very first line in every code module. It means that every variable must be dimensioned (Dim statements) and then before running the code you can select menu item Debug -> Compile and then many errors get picked up before attempting to run the code. Unfortunately it does not pick up all errors.

You can get the system to automatically insert the Option Explicit. In the VBA editor select Tools ->Options and check the box against "Require variable declaration". Once selected, it applies to all new projects and inserts the Option Explicit at the top of each module.

If using a password, declare the password as a Const at the top of ONE standard module and that way if you want to change the password, it only has to be done at one place in the project. In the example, Password is now a variable that contains the password.

You will see 2 test lines in the example starting with "Debug.Print". I have inserted these so you can see how much information is actually stored in range variables. To see the information, after running the code, select the VBA editor and use Ctrl and G to open the Immediate window and you will see that the variables contain the workbook name, worksheet name and the range. Close the Immediate window with cross at top right of the Immediate window. Delete the 2 Debug lines for your production code because they are only used in testing.

You will see where I have inserted code to remove the password protection before copy and pasting the data to the workbook and then re-applying the password protection after pasting. If password protection was not your problem then you don't need either of these lines. (However, if using protection, then  where the constant is declared, change the password "ossie" to your password.

I don't understand why you are using a worksheet name with a space at the end. I suggest editing the name on the worksheet tab and remove the space and then edit the VBA code accordingly.

joeu2004 say: CMS wrote:
in excel 2003, importing data from a csv file that contains credit card numbers.

16th character of number automatically turns to a zero. I have tried sending the csv file thru notepad, and using text to columns.

tried copying via copy, paste special. tried creating new, clean column and retyping the number, still get the same results. I have played around with the custom formatting, but my expertise in there is short.

The 1.11E+15 is not an error code.  It is the way that Excel displays large numbers in General format.

(The format is called Scientific.  You can see that in the Number format menu.)

And yes, you lose the digits after the 15th digit because Excel limits entry of numbers to 15 significant digits.  For that reason, it does not help to format the cells as Text after data entry because you cannot recover the lost digits after the fact.

You do not mention the origin of the CSV file.  No matter.  When you open a CSV file directly in Excel, Excel 2003 does not respect double-quotes around numbers (i.e. treat them as text).  When you save to a CSV file, Excel 2003 does not put double-quotes around numbers even if the cell is formatted as Text.  Sigh.

You need to use the Import Text wizard (Data > Import External Data > Import Data), set the delimiter to comma, and be sure to select Text for the column type.  Note that it is not sufficient to format target cells as Text before importing.

Caveat:  This is problemmatic if the column has a mix of true numbers and text numbers (like IDs).  In that case, to be on the safe side, you will need to select Text for the column type, then convert the cells with true numbers to Number format later.  Unfortunately, that is not as easy(?) as it sounds because simply changing the format to Number is not sufficient.  You also need to re-enter each cell (press F2, Enter), or copy 1 and paste-special-multiply, or write a macro to do the conversion.  Sigh.

If you are copy-and-pasting from Notepad, it is sufficient to make the cell format Text before pasting.  However, that does not work if you are pasting objects from other applications such as HTML (web pages).  I always paste HTML text into Notepad first, then copy and paste into Excel 2003.  Sigh.

If you type the text numbers manually, either set the cell format to Text before entering the data, or type an apostrophe (aka single-quote) before each text number.

albert d. kallal say:

The xml to open a form looks like:

 

          <button id="btnOpen" label="Open Form Test"
             imageMso="FileOpenDatabase"  size="large"
             onAction="=OpenForm('frmTest')" 

           />

And in a standard VBA module, palce this code:

 

Public Function OpenForm(strForm As String)

   DoCmd.OpenForm strForm
  
End Function

 

So you can open any form with the above xml (in this example we open form test). Note how we not using a call back, and thus the above is only ONE line of VBA, and the form is passed from the ribbon.
