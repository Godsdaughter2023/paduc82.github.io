ken sheridan say: Admirable as it is, I think your ambition may be running a little ahead of your experience if you are only on day 2 of using Access, but I would not discourage you from pressing ahead.

Firstly your third table, let's call it Inventory, does not need a Manufacturer column, and it would in fact be quite incorrect to include one as it introduces redundancy into the table.  This in turn leaves the table open to inconstant data, i.e. it would be quite possible to enter a row with a brand name not produced by the manufacturer entered in the same row.  I realize that preventing this is what you are trying to achieve by means of the correlated combo boxes, but relying on the interface for this is not really good enough; it should be built into the table designs themselves.

Once you have a brand name recorded in a row in the table you know the manufacturer via the relationships.  In the jargon of the relational model it is said that manufacturer is determined by brand name.  To include both columns in the Inventory table would mean that the table is not normalized to Third Normal form (3NF)  which requires that all non-key columns are determined by the table's primary key.

You can still do as you wish by including a combo box bound to the BrandName column.  This must as its RowSource property use a query, sorted by brand name, which references the manufacturer combo box (more about this below) as a parameter in the criteria row of the Mnaufacturer column in query design view, e.g.

    Form!cboManufacturer

Where cboManufacturer is the name of the manufacturer combo box and Form is the Form property, which is a shorthand way of referring to the current form.  In SQL view the query would be like this:

SELECT [BrandName]
FROM [BrandNames]
WHERE [Manufacturer] = Form![cboManufacturer]
ORDER BY [BrandName];

Add  the other combo box listing the manufacturers, but the latter must be unbound, i.e its ControlSource property is left empty.  The problem is then that, being unbound the combo box will show whatever manufacturer you select in every row in the form.  You can get round this by using a 'hybrid' control.  This is made up of two controls, an unbound combo box listing the manufacturers, and superimposed on it so that it appears to the user as a single control, a text box bound to the Manufacturer column, i.e. the combo box's arrow is all that is seen to the right of the text box.  But, we've already said that the Inventory table does not include a Manufacturer column, so the form has to be based on a query which joins the Inventory table to the BrandNames table on the BrandName columns.  The query should return all columns from Inventory, plus the Manufacturer column from BrandNames.  It is this last column to which the text box on top of the combo box is bound.

To get the brand names combo box to show only the brands for the selected manufacturer you need to put code in the AfterUpdate event procedure of the manufacturers combo box.  This code does two things; (i) it sets the brand name combo box to Null to remove any existing value; (ii) it requeries the brand name combo box so it shows only the brands for the selected manufacturer.  The code is like this:

    Me.[cboBrandName] = Null
    Me.[cboBrandName].Requery

You also asked about reversing the process, but if you think about it that's illogical.  If the brand names combo box is showing only the brands for the currently selected manufacturer, then whatever you select in it will be that same manufacturer.  It's a one way street.

One final point.  Set the Enabled property of the bound text box which overlies the unbound combo box to False (No), and its Locked property to True (Yes).  That will prevent the user inadvertently trying to change its value other than by the combo box lurking underneath it.

You'll find examples of the use of correlated combo boxes in various contexts as ComboDemo.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

The forms in this demo are a little more complex than you will require, so don't think you have to do everything I've done in it.  If you follow carefully what I've outlined above I see no reason why you should not be successful.

ken sheridan say: As you've found, correlated combo boxes in continuous forms or datasheet view are a problem where hidden surrogate keys are used.  The reason is that once a combo box is requeried, any rows where the value of the control is no longer represented in rows returned by the control's RowSource will appear empty because the value is not in the hidden bound column so there is nothing to which the value in the visible column can refer.  The value of the control is unaltered, however, so no data is lost.

There is a solution in continuous forms view, but it's a bit kludgy, which is to use a hybrid control made up of a text box overlain on a combo box so that to the user it has the appearance of a single combo box control.  You'll find an example as ComboDemo2K.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

You might have to copy the text of the link into your browser's address bar (not the link location).  For some reason it doesn't always seem to work as a hyperlink.

As regards the parameter prompt it appears that you are trying to reference a subform as a member of the Forms collection with Forms!sfrmClientTargetsVer2!TOutcome.  A form used as a subform is not in the Forms collection and must be referenced via the Form property of the parent form, e.g.  Forms!mfrmClientRecord.sfrmClientTargetsVer2.Form!TOutcome.  Note, however, that sfrmClientTargetsVer2 is in this case the name of the subform control, i.e. the name of the control in the parent form which houses the parent form.

If this reference to the TOutcome control is in the subform itself, however, e.g. in the RowSource of a combo box in the subform, you can simply reference it with Form!TOutcome.  As it is in the same form the Form property is enough on its own.  In fact you can even omit that and reference the control solely by its name TOutcome, though I prefer to include the Form property to avoid any possible ambiguity over just what is being referenced.
