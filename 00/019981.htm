ken sheridan say: The question of how 'people' are modelled in a relational database such as yours is, I think, in need of little more consideration.

Firstly, let's get the terminology correct.  Each table in a relational database, including those which model a relationship type, models an entity type.  The word 'type' is very important here.  Each row in a table represents a legitimate entity of the type modelled by the table.

We now need to examine the concept of type hierarchies. While every entity must be of at least one entity type, it can be of multiple entity types.  Every person represented in your database is clearly an entity of type people, so it is correct to do as you've been advised, and include them in a single table.  However, a person might be an employer or a doctor, or even both.  Consequently such a person is also an entity of type employers and/or type doctors.  Employers and doctors are subtypes of type people.

What does a subtype actually mean?  I can do no better than cite Date on this:

If T' is a subtype of T, there should always be a type constraint such that, if it is satisfied by some given value of T, then the value in question is really a value of type T'.

Thus T and T' are both sets, and T' is a subset of T, and T and T' both have membership predicates, such  that a value is a member of the set in question (and hence a value of the type in question) if and only if it satisfies the predicates in question.

So a person obviously satisfies the predicates for membership of the set people, and hence type people, but if they are a doctor they also satisfy the predicates (e.g. registration by the BMA or other registration body in their area of practice), for membership of the set doctors, and hence type doctors.  This is known as 'constraint by specialization'.

How then do we model such a type hierarchy in a relational database?  We simply have a table for each type and for each of its subtypes, and for each of their subtypes, and so on.  So as well as a People table the model does require Doctors and Employers tables, the latter two modelling subtypes of the first.  It is the case of course that for each row in a people table there can be no more than one row in a Doctors table, so the relationship type is one-to-one with the People table being the referenced table in the relationship type.  A one-to-one relationship is created by the primary key of the subtype table being also a foreign key referencing the primary key of its (super) type table.  Consequently the primary keys of subtype tables cannot be autonumbers.

A subtype is characterised by sharing all the attributes of its (super) type, but not those of other subtypes.  Consequently a people table will have columns representing attributes shared by all people, e.g. personal names.  A Doctors table would therefore not have any personal names columns, but would have columns representing attributes relevant only to doctors.  While this eliminates any redundant duplication of data, don't be confused by the fact that two tables might have columns of the same name.  In the demo database to which I refer below for instance, three tables have DateAppointed columns.  This is because these columns represent the dates when the person was appointed to the entity type modelled by the table in question.  A person can, and usually will, have three different appointment dates, so this is not a shared attribute of the three tables' (super) type.

For an example of a type hierarchy modelled in this way you might like to take a look at TypeHierarchy.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This little demo file uses a simple model of personnel in an academic institution as its example.  In addition to the model itself it illustrates a simple interface by which a person can be entered as members of any, some or all of the different subtypes of personnel.  I've used separate forms for each, selected from an opening menu form, but the interface could equally well be a unified one in which each of the four forms were embedded as subforms on separate pages of a tab control in a single unbound parent form.  An unbound combo box in the parent form would enable the user to select an person, and, by setting the combo box as each subform control's LinkMasterFields property, cause all of the subforms to go to the selected person, or to an empty new record in those subforms where the person is not currently a member of the subtype in question.

stephen boots say:

Open Task Scheduler

Create new basic task

Give it a name

Go to Actions/New

Browse to c:\program files\Microsoft security client\mpcmdrun.exe and select it to populate the action box.

In the Argument box type -SignatureUpdate without the quotes

Don't schedule it or add triggers.

Now right click and run it to see if it works. A command Prompt window should load and the text in that window should indicate that an update is happening and then the window will close. MSE should reflect that run time as the last check for updates.

Add a schedule if the above worked.

 

Note that it will not work if MSE cannot check for update manually. (Open MSE and click Update Now).

-steve
