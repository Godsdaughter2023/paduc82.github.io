john w. vinson say:

You really shouldn't rely on Autonumbers for human-meaningful purposes; in particular, they are NOT at all suitable as a record count.

A new Autonumber value will be assigned the instant a new record is started. Even if the user hits <Esc><Esc> one second late, cancelling that record before it's even entered, that autonumber value is gone forever! Likewise, if a record is deleted, its autonumber value will not be reassigned. So autonumbers will, in practice, always have gaps.

If you realize this, and can accept gaps in the sequence (and never use Replication, which will make all autonumber fields Random) you can use autonumbers - but you have to recognize the limitations!

 

ken sheridan say: As John has pointed out, if you need to fix them then you shouldn't be using an autonumber; if you are using an autonumber for the purpose it's designed for, i.e. to guarantee a set of distinct arbitrary identifiers, then you don't need to fix them as the values per se are completely irrelevant.

If you really do need sequential numbering then you should compute the values when a row is inserted.  You'll find an example as CustomNumber.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

The Sequential Numbering option in this little demo file would be the relevant one.  You can easily convert your current table with the autonumber column into one with a straightforward long integer primary key:

1.  Add a new column of number data type to the table.  If your current autonumber column is called MyID say call the new column MyID_New or similar.

2.  Create a simple update query to fill the MyID_New column:

UPDATE MyTable
SET MyID_New = MyID;

3.  Once you are happy that MyID_New has the correct values delete the MyID column from the table.

4.  Change the name of the MyID_New column to MyID and make it the primary key.

You can now recreate the relationships between the table and the relevant referencing tables. Using the method described in my demo file the referenced table's MyID column will be given the next number following the current highest number, and so on as further rows are inserted via the form.  But note that they must be inserted via the form.

guessed say:

The success you have in importing styles depends heavily on how the styles are defined. In the last few versions of Word there are additional theme layers for Fonts and Colors. This can have a large effect on the styles imported into the document if they are defined to rely on the theme fonts and colours.

As an example, check the style definition of one of the styles which is appearing in the wrong typeface after importing the style. Is the typeface listed as 'fontname' or 'fontname (Body)'? The second option shows that the style is not actually defined to use fontname as its font but instead uses whatever font is chosen for body in the Theme Font.

You can choose to apply a different font theme by going to Home > Change Styles > Fonts. If changing your font theme results in a considerably different looking document then you have the choice of either modifying your theme to use your preferred fonts or modifying each style that uses the 'theme' fonts to specify the font explicitly (ie independent of the theme font).
