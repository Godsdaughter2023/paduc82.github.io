curtis ness say: So I solved this issue with lots of trial, error, and searching for bits of info in all dark alleys of the internet.  In the end it is a MAC issue, not a Microsoft issue, however the only way the issue is evident is through the Microsoft RDC Client...  

Cause:  This is the short version...  When the RDS server was set up initially, I didn't have the licensing server correctly configured and licensed so the users were running in the Grace Period.  When connecting to the RDS server, the MS RDC client pulled down licensing information related to the grace period.  When I finally got the licensing server to work, that's when the Macs lost their ability to connect.  Due to permissions on the Mac client, the new license information was unable to be downloaded to the client.

So this seems simple enough right?  It is once you find the area where the permissions issues happened.  Here are short and to the point steps to fix the Mac client:

a. Navigate to Macintosh HD > Users > Shared > Microsoft > RDC Crucial Server Information
b.  Deleted the contents of this folder
c.  Change the permissions of this folder to Everyone -> Read and Write
d.  Change the permissions of the Microsoft folder as well to Everyone -> Read and Write
e.  Reconnect with the MS RDC Client v2.1.2 and verify creation of the files
f.  you should be able to reconnect easily at this point with no licensing error

If you still encounter the licensing error you will need to follow the steps outlined to correct the certificates.  Follow the steps below then return to the above client steps:

Locate and then click the following registry subkey:
HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Terminal Server\RCM

** Make a backup if you want to but it should not be needed

Right-click each of the following values, click Delete, and then click Yes to confirm the deletion:

Certificate
X509 Certificate
X509 Certificate ID
X509 Certificate2
Quit Registry Editor, and then restart the server.
Reactivate the Terminal Services Licensing server by using the Telephone connection method in the Licensing Wizard. (I just reactivated online though and it worked just fine.)

marshall barton say:

It fails because referring to a subform control is more complex than referring to a main form control.

 

The way to refer to a control in a subform is:

 

   Forms!mainform.subformcontrol.Form.controlinsubform

or

   Forms!mainform.subformcontrol!controlinsubform

 

There are subtle differences between what those two syntaxes work, but either should be ok in your case.

 

Thus, you should change the query to something more like:

 

SELECT LocationID, LocationName, CompanyID

FROM qryActiveLocations

WHERE CompanyID = Forms!frmContacts.NavigationSubform!comboContactCompany

ORDER BY LocationName

 

Note:  NavigationSubform is the default name of the subform CONTROL on the main form so if you changed the name, replace that with the name you used.

 

Note:  I removed the redundant table/query name, parenthesis and brackets just to make it easier to read.  The query designer is not very sophisticated and adds them in whether they are needed or not.

steve rindsberg say:

Try the Divide and Conquer method:

Make two copies of your presentation.  Delete the first half of the slides from one copy, the other half of the slides from the second copy.  Save both of these as PPT.  If a problem with one or more of the slides is preventing the save, it'll likely happen with one of the "halves" and not the other.

Similarly, divide the "problem" half presentation into halves, save each as PPT and so on.  Keep dividing until you've found the problem slide.   Delete that same slide from the original presentation and try saving as PPT again.  You'll have to re-create the problem slide.
