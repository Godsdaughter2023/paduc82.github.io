Our driver uses calls the ExtEscape GDI function in order to communicate between the UI DLL and the core printer driver. In the UI DLL, within the DrvDocumentEvent implementation, we have various calls to ExtEscape using the printer's device context. The core printer driver processes the call to ExtEscape in its implementation of DrvEscape.

All this has been working great since Windows NT4 till the latest official release of Windows 10 (July 2015.) With the insider preview builds starting from build 14295, the call to ExtEscape is not transmitted to the core DLL when the application that is printing is 32-bit. Most of you know by now that when printing from a 64-bit application, the application interacts directly with the printer driver, in this case the calls to ExtEscape work as expected. With 32-bit applications, all the calls to the printer driver go through the SPLWOW64.EXE module, in this case the calls to ExtEscape fail (or do not reach the core DLL, there is actually no error code.)

Note that these calls work when a 32-bit application calls ExtEscape with a printer DC. The calls fail ONLY when the driver UI tries to communicate with the driver core.

Any hints before the release of the new Windows 10 would be greatly appreciated.

The problem (of course) is that Ctrl+Shift+F9 removes fields. Since links in Word are maintained using Hyperlink fields...
