All you need do is to specify a fixed height and width for the rows & columns.

macropod say:

The "I try and move the table down the page slightly it goes on to the second page" suggests your table is too large to fit within the page's print area. Try reducing the row height and/or page margins.

If, by "I've changed margins down to zero too", you mean you've changed the page margins, do be aware that most printers can't handle that - they need about 0.25in on the sides and sometimes at the top or bottom as well, for paper handling.

You cannot lock a table so it can't be moved - it has to allow for whatever might be inserted before it. If you want to control that too, then put whatever goes before the table into more fixed-height table rows. What you can also do is change the table's text wrapping to 'Around' which gives access to options that allow you greater flexibility with positioning.

As for "how to make an image fill the cell and not resize too", I thought the whole idea was to make the image fit the cell. Obviously, if the image has a different aspect ratio to the cell, it's not going to fill the cell unless you distort the image. Note: you may need to format the cell's paragraph indents and before/after spacing to 0, so that images can align with the cell top & bottom (assuming their aspect ratio allows them to do so).

Note: The automatic resizing to fit the cell dimensions only applies to images inserted from outside of Word, not to images that are already in the same document or another document. You will also need to ensure the images are inserted into the cells using the 'in-line with text' format.

suzanne s. barnhill say:

I'm assuming the paragraphs are not autonumbered. If they are, then you can just remove the numbering as the first step. Otherwise, here's the two part Replace that will work.

  1. Using wildcards, search for [0-9](1,}= and replace with nothing. This will remove the numbers and equals signs.
  2. Before the paragraph mark of the first answer, type the number of periods you want to precede the number (or a tab character if you've set a tab stop with a period leader). Then press Ctrl+F9 and, between the braces, type SEQ number.  Select the text you just inserted and Cut it to the Clipboard. Then, without wildcards, and with just the desired text selected, search for ^p and replace with ^c^p.

That second replace will appear not to have (completely) worked, but, with the text selected, pressing F9 will cause the numbers to be displayed. You can then press Ctrl+Shift+F9 to unlink them, converting them to ordinary text.

rob koch say:

I wonder if there is better security  and protection or any other advantages for using MSE 64 bit verses 32 bit version?

I have 64 bit Win7 and I love it. I never had any virus problems even in vista. 

I am using now MSE2 beta 64 bit and curious to know if it has any advantages for protection/security over 32 version.


Dr. Strangelove has already answered your question completely correctly, since your operating system defines which version of MSE you must use, so my comment is only to address your larger question of which is better, 32-bit vs. 64-bit.

The simple fact is that much of the malware that has been created to this point is 32-bit, so though it may run on a 64-bit operating system via compatibility mode, it's not truly native.  This and the fact that both Vista and Windows 7 themselves have much improved security over the older Windows XP operating system means that both the newer and the wider (32 vs. 64 bit) the better.

This is NOT a guarantee of protection, simply a fact that the newer os is inherently more secure and some softwware specifically designed for 32-bit systems may either not run or be more easily controlled by a 64-bit version of the operating system and anti-malware.

So obviously you can see that there's not a simple 'yes or no' answer to this in every case, but the tendency at the moment is that 64-bit operating systems which must use 64-bit MSE are somewhat more secure.  This is likely to change some over time as malware writers spend more time developing true 64-bit malware, but the fact that the newer operating systems start with a higher level of security in place should tend to dampen this a bit.

At this point, many of the most successful malware attacks already require that the user accept the installation of something bad via social engineering (fooling the user), such as the fake anti-malware applications that are so common today.  Though they are newer and faster, no anti-malware has yet been very successful at protecting a user from themself.  Maybe we need a '64-bit upgrade' for the users?

Rob
