ken sheridan say: You'll find examples of this sort of thing in CombDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file uses the administrative areas of County, District and Civil Parish in my neck of the woods, and allows top-down selection of county, then district, then parish in correlated combo boxes.

The key thing to take on board when modelling hierarchies like this is that only the lowest level is stored a value in a column the referencing table.  In my file the locations table has a ParishID column, but not a DistrictID or CountyID column.  This is because, in the language of the database relational model, district is functionally determined by parish, and county is functionally determined by district, so once we know which parish a location is in, we automatically know which district and county it's in.  If the Locations table included DistrictID and CountyID columns they would be said to be transitively determined by the table's key.  For a table to be correctly designed, to Third Normal Form (3NF) at least, all non-key columns must be functionally dependant solely on the whole of the primary key.  Otherwise the table includes redundancies, and these can allow inconsistent data, e.g a parish could be recorded incorrectly in two different districts and/or counties.

With tree-structured hierarchies like this we then have to allow for gaps in the 'chain' e.g. in my demo the districts of Tamworth and Stoke-on-Trent do not have any parishes, so we can't record which parish a location is in.  The way to handle this is to give each such district a dummy N/A parish.  When entering a location in either of these districts the N/A parish is selected for the district in question.  It is of course the only one available for selection in such cases.  This same approach would be followed as we move up the hierarchy, so a county which had no districts would have a N/A district in the database.  This doesn't happen in real life, but the principle would be the same for other hierarchies which might have gaps at any level.

When it comes to data entry, the county and district are selected in unbound combo boxes, the district combo box being correlated with the county combo box by referencing it as a parameter in its RowSource.  In the county combo box's AfterUpdate event procedure the district combo box is requeried so that its list shows only the districts in the selected county.

There is a fly in the ointment, though.  When such correlated combo boxes are used in a continuous form, if the value of a combo boxes is a hidden surrogate key, e.g. DistrictID, and the visible value is the text value, i.e. the district name, rows other than the currently selected one where the selected county differs from that in the current row will show a blank in the combo box.  No data is lost, it's merely invisible by virtue of the fact that the DistrictID which maps to the district name is no longer in the set of DistrictID values in the combo box's list.  This is overcome by the use of a hybrid control made up of an unbound combo box superimposed on a bound text box, so that it appears to the user to be a single combo box control.  The text box is bound to the district name text column, but as this is not in the Locations table, the form must be based on a query which joins the Locations, Parishes, Districts and Counties tables.  This is illustrated in my demo.

Your locations, sub-locations and individual offices are directly analogous to my counties, districts and parishes entity types, so your form can be modelled on one of mine, depending on whether the form is in single form or continuous forms view.

scottgem say:

The correct search term would be Cascading or synchronized combos. This is a standard technique where one combobox is filtered by a value selected in another combo. Ken's demo illustrates how its done.

You said: "but I don't know how to populate a table with the selections." There are two ways to populate a table with data entered into a control (you don't have fields on a form, you have controls that may or may not be bound to a field in a table). Either you bind the field to the control (and the form to a Recordsource). Or you write code to take the data from the table and populate the fields in the table. Using bound forms is FAR easier.

But what worries me is you use of selections (plural). Ken alluded to this in his response. But when you have a cascading situation, the earlier combos would be unbound. In your example, you have an individual Office. That office would be part of a sublocation, which, in turn, is part of a main location. So once you know the individual office, you know both the sub and main locations. So there is no need to store anything more than the Individual Office code.
