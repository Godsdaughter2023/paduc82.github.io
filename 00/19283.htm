suzanne s. barnhill say:

Pace Paul, I don't think mail merge is suitable for your situation. WordPerfect has a feature called (I think) "keyboard merge" that is just what you would be looking for. You create a standard document with spaces to insert information, and then, ad hoc, you create another document that contains the data. You then merge those two documents. Simple and easy, and you can then throw away the document that provided the data.

Mail merge in Word is much more cumbersome, though not impossible to use for this purpose. You must create a data source (in Word, Access, or Excel) with at least one record and then create a document in Word that has merge fields corresponding to the fields in each record in the database. It would not be difficult to add a new record to the data source when you get a new tenant, and you could then, in the mail merge main document, use Select Recipients to select just one specific record to be merged. This would have the advantage that you would be keeping a database of all tenants in one file, so it might not be a terrible idea. If you were familiar with the mail merge process, though, I suspect you might already have taken this route, so if you had to learn how to create a mail merge, it might be almost as much trouble as learning to code a UserForm, and my sense is that it would be a little more trouble to create an individual lease document as well.

macropod say:

I would have to change the Data Source document by adding the information for each new tenant each time I create a new lease. Would that be a problem with the mail merge function? I thought the mail merge required a static data source from which to pull information.

Also, how would I "tell" the mail merge to only pull the information for the new tenant and not the other tenants that are listed in the database?

Suppose, for example, your data source is an Excel workbook, with columns like:

Property Address, Monthly rent, Lease Start, Lease End, Tenant1 name, Tenant1 SSN, Tenant2 name, Tenant2 SSN, Lease ID

with a separate row for each tenancy agreement (the Lease ID might be nothing more than the row #).

Your lease document could then have mergefields wherever you need them, referencing the relevant field names (i.e. Property Address, Monthly rent, etc.). The column order is unimportant for the mailmerge and field coding can be used to conditionally display the Tenant2 data according to whether there's one or two tenants.

Whenever you're preparing a new tenancy agreement, simply add a new row to the workbook with the relevant data, open the lease document, select the Lease ID from the mailmerge filters, then 'Finish' the merge. Done.

The process works just as well with a database or any other file Word can use as a mailmerge data source. The only sense in which the data source should be static is that it should retain the same name & path so the mailmerge main document can find it. You can add/delete columns to it, too, provided you don't delete any that are used for the merge. Once you have a data source set up, that also allows you to generate reports, such as the tenancy history of a given property.

A mailmerge is not at all difficult to set up and, once done, is quite easy to maintain.

Do drawing canvases prevent photo compression? - Microsoft Community

rj1peterson say:

Do drawing canvases prevent photo compression?

Hi all -

I'm using drawing canvases to group pictures in word and I keep ending up with extremely large file sizes that are leading to issues with saving. I have been individually compressing each photo, but it doesn't seem to make a difference as to the file size. Any help? 

rich michaels - mvp (greatcirclelearning.com) say:

I don't believe that they do and I can't think of a logical reason why they would. The automatic compressing of images is controlled by a setting in Word > Options or alternatively as it sounds like you are doing, a manually chosen action.

I think your file size issue would best be handled by resizing your images before you import them into Word. When you import an image into any Office application it stores the image in its original size and pixel density. When you compress an image with MS-Word, it uses a resampling process that eliminates pixels in the image and this can lead to a loss of image sharpness and clarity. Yes, compressing does reduce image size but it does so by reducing image quality, i.e. PPI.

IMO - It is far better to resize by scaling the image using an external graphics editor program and to keep the PPI between 150 - 220 on the imported "smaller sized" image.

The other factor to consider is the image format. JPG will be the smallest sized but it will lose detail if it is reedited. JPG format is best for photographs, it does not do well with text or detailed drawings. PNG will be much larger and will do well with both photographs, text and line drawings. TIF will be the best all around but again, very large file size.

Hope this helps
