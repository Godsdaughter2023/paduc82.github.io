tc4076 say:

The dialog to change Outlook note's font, size, and default color was removed from Office 2013 but from my experiment, Outlook still reads those settings from the registry.  The easiest way to set this is to find a machine with Outlook 2010.  Start Outlook, set the font/size for Note, then export the registry settings, and import it to your Outlook 2013 machine.  Once you have Outlook Note font/size set the way you want, run the following command to export the settings.

reg export "HKCU\Software\Microsoft\Office\14.0\Outlook\Options\Note" NoteSettings.reg

Now, use notepad (or any other text editor) to open the NoteSettings.reg above and change this line

[HKEY_CURRENT_USER\Software\Microsoft\Office\14.0\Outlook\Options\Note]

to

[HKEY_CURRENT_USER\Software\Microsoft\Office\15.0\Outlook\Options\Note]

Save the file.  Copy the file to your Office 2013 machine and import the settings.

reg import NoteSettings.reg

The change will take affect the next time you start Outlook.

Multi-Level List Numbering is replacing dots with commas in other environments

We have a set of Arabic language templates with multi-level numbered lists in them using Hindi numbering.

However when we place the same template in our other environment, the numbering appears as 1. then 1,1 then 1,1,1 i.e. when there is more than one number, the dot converts to a comma, even though the style explicitly states a dot.

We have checked each environment for different settings as we have read that auto-correct can often be the reason for swapping out dots for commas, but this does not seem to be the case as the auto-correct options seem to be the same in each environment.

The only difference I can see is one environment is running Office 365 Business and the Other Office 365 Professional Plus (both Word 2016) but I'm not sure that is the factor.

barkoosh say:
For Arabic, Word in some versions automatically changes the display of the decimal separator "." to comma "," when between two hindi digits. Why? It's to avoid confusion, since zero has the form of a dot in Arabic.٠ (This dot to the right is actually the Arabic zero)

This feature has nothing to do with Word's autocorrect.

Since the dot in multilevel lists is between two digits, Word is going to think that this is a decimal separator and converts it to comma when the digits are hindi.

I suggest going to Define new Multilevel list and replace the dot(s) with hyphen under Enter formatting for number.

Well, you're going to think I'm mad, but I actually have a solution for contacts not appearing when you type their names in the e-mail address.  At least, this worked for me. are the same as mine.

Open up one of the contacts you're having trouble with.

Note the contents in 'Display as' (don't confuse this with 'File as').

If  it contains a name, and not an e-mail address, delete the content of the field.

If you click on another field, you'll see that it has changed to Name(e-mail address). 

Save and you should be good to go.

You might find that you have the same problem if the field contains only an e-mail address, although I haven't had time to play around with this.  If it is problematic, then the same solution applies.
