Shuffling a two-column table that has 200 numbered rows... - Microsoft Community

chirogeek say:

Shuffling a two-column table that has 200 numbered rows...

Good morning all,

I'm a medical student and use word to make flash cards.  So I have created a two column table with many rows (usually over 200) with the first column of a row, which is very narrow, used exclusively for numbering (1, 2, 3, 4, etc) and the second column of that row used for asking myself questions with the answer in that same cell--only in very small font. 


QUESTION: I would like to be able to shuffle these questions (mix them up) without manually re-numbering them--which takes a very long time.

So is there any option for shuffling a numbered table?  I know word can put the tables numbered rows in order, but what about shuffling them up?

I hope that made sense and thanks in advance.

Doug
jay freedman say:

Use this macro. See  http://www.gmayor.com/installing_macro.htm if needed.

Sub Scramble()
    Dim i As Integer, j As Integer
    Dim rg1 As Range, rg2 As Range
    Dim tmp As String
    Dim tbl As Table
    Dim maxRow As Long

    If ActiveDocument.Tables.Count = 0 Then
        MsgBox "No table to scramble."
        Exit Sub
    End If

    Randomize Now

    Set tbl = ActiveDocument.Tables(1)
    maxRow = tbl.Rows.Count

    ' duplicate the original first col.
    tbl.Columns(1).Select
    With Selection
        .Copy
        .MoveLeft Unit:=wdCharacter, Count:=1
        .Paste
    End With

    ' scramble the new first column
    ' by swapping each one with a random other one
    For i = 1 To maxRow - 1
        Do
            j = maxRow * Rnd + 1
        Loop Until j <> i

        ' store the current number in tmp
        ' Note: the MoveEnd is needed to avoid the cell marker
        Set rg1 = tbl.Cell(i, 1).Range
        rg1.MoveEnd Unit:=wdCharacter, Count:=-1
        tmp = rg1.Text

        ' swap
        Set rg1 = tbl.Cell(i, 1).Range
        rg1.MoveEnd Unit:=wdCharacter, Count:=-1
        Set rg2 = tbl.Cell(j, 1).Range
        rg2.MoveEnd Unit:=wdCharacter, Count:=-1

        rg1.Text = rg2.Text
        rg2.Text = tmp
    Next

    tbl.Sort

    tbl.Columns(1).Delete
End Sub

rob koch say:

The following Toast notification overview document, includes the paragraph below, which hints at something that may relate to your issue.

https://docs.microsoft.com/en-us/previous-versions/windows/apps/hh779727(v=win.10)

"Long-duration toast: This notification looks the same as a standard toast but stays on the screen for 25 seconds and optionally can play longer, looping audio. This type of toast is used in situations where you want to grab the user's attention because there is a human waiting on the other end of the connection. This is appropriate for person-to-person communication like instant messages and VOIP calls. This type of toast can also be used for calendar reminders."

This reminded me that I'd become annoyed by notifications from Skype and had thus gone into App settings to disable the ability for that app to display them.

Though I'm unsure whether there are any similar messaging abilities remaining in the Outlook apps, it might make sense that another installed messaging app like Skype could be used to send such a notification of a waiting message.

That document also mentions using the toast notification to launch the app, so it's possible that it might be used to begin some form of interaction with the malicious individual via the app as well.

I've seen other mention of this particular type of hacked password message being displayed, but had assumed these were simply browser popups as those are so common.  It's possible though, that these may have also been displayed via toast and the individual reporting just wasn't familiar enough with these to describe them fully.

This might actually make sense, since with the Microsoft account being directly associated with an email address, it would be reasonable to expect that those 3rd-party websites which have been breached in the past may often use the same email address as a username.  This could result in the symptom mentioned by others where the password is an old one they recognize, but no longer use.

All of this fits well with attempts by malicious actors to use trusted apps and notifications in order to build trust and allow them to display their messages to the PC user without being blocked.  The assumption by the PC user that these apps and notifications are somehow protected makes them less likely to recognize that the messages themselves may originate from any valid sender via the app in question.
