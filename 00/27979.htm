VBA Regular Expression patterns with word boundaries - Microsoft Community

therogueb97d say:

VBA Regular Expression patterns with word boundaries

To whom it may concern:

I am using regular expressions to search through text for various patterns that I want to try and match.  Because it could sometimes happen that one pattern (which I would want to match) might be embedded inside other text (where I wouldn't want it to match), I need to use word boundaries to block my searches.  For example, I have a pattern that looks like [0-9]{4}-[0-9]{1,4}: in the input text "123456-7890-123" I would want it to match "7890-123", but not "3456-7890".   By putting word boundaries around the pattern ("\b[0-9]{4}-[0-9]{1,4}\b") I can prevent the undesired match from occurring.

But I have recently run into a problem where the word boundary in the pattern creates a problem, due to the fact that the word boundary has to separate a word character from a non-word character.  Because the beginning or the end of the line is considered a non-word character, if the first character in the search string is not a word character, that character (and potentially others as well) get skipped.

Again, an example will help clarification.  For technical reasons that are not relevant here, one pattern I need to check for will match anything: ".*".  Delimiting this with the word boundary anchor produces the pattern "\b.*\b".  If I then search the text "12345 ABCDE", I get back "12345 ABCDE" as expected.  But if I search the text "[12345] [ABCDE]", I get back only "12345] [ABCDE": since the square bracket characters are not considered word characters, the word boundary is found on the inside of the square brackets.

I have my code set up to perform my searches using a loop for generalization.  That way, I can repeat the process as many times as I need, based on the number of patterns I have to test.  In all instances but this special case, the use of the word boundary has been fine, and I don't want to have to implement a special case in my code for this pattern.

Does anyone know of some pattern expression I could use instead of the word boundary anchor that works like the word boundary anchor does, but always considers the start and end of the string to be word boundaries, even if the adjacent character is not a word character?  Note that it is important that I not "lose" a character to this alternative to the word boundary anchor: returning to the first example, I don't want to lose the extreme characters ("1" and "3") from the input text, nor do I want to lose the first character of the match (e.g., "7" from "7890-123").  Nor can I add something as a prefix and/or suffix to the input string before applying the pattern, which would taint the results of the search.

Thanks in advance.

Ariel S. Rogson

therogueb97d say:

To all: I have found that instead of using "\b" to define the "word boundary" that I need, using "(^|$|\b)" does the trick, as it will force a match at the start and end of the string, even if the start or end of the string does not qualify as a word boundary.

a. user say:

Basically this is what happens when a MSRT build, containing false positive signature/s, does during a scan:

1. MSRT makes a detection on the client machine during the scan, the UI displays 1 detection

2. Scan completes, and the signature is checked with Microsoft AV backend, which has the offending signature marked as a false positive.

3. Disable notification is sent from the backend to the client machine, and the offending signature is disabled.

4. the UI displays the scan result, and since the offending signature is disabled, no detections are reported.

quietman7 say:

All scanning tools are susceptible to glitches, bugs and heuristic analysis technology. Heuristics is the ability of a scanning program to detect possible new variants of malware before the vendor can get samples and update the program's definitions for detection. Heuristics uses non-specific detection methods to find new or unknown malware which allows the anti-virus to detect and stop if before doing any harm to your system. The disadvantage to using heuristics is that it is not as reliable as signature-based detection (blacklisting) and can potentially increase the chances that a non-malicious program is flagged as suspicious or infected.

john w. vinson say:

If you're doing this one record at a time using a Form, I'd suggest a different technique. You can make these 25 fields "sticky" by setting the form control's DefaultValue property to the value currently in the field in the control's AfterUpdate event. Put the following code in your form's Module:

Public Function SetDefault(ctl As Control)

If ctl.Value & "" <> "" then ' did the user enter data?

ctl.DefaultValue = Chr(34) & ctl.Value & Chr(34) ' the default value must be a string for any datatype

End Function

Then in the AfterUpdate event of each of the 25 controls put

=SetDefault([nameofthecontrol])

That is, in the AfterUpdate property of the textbox named txtLastName put

=SetDefault([txtLastName])

I still have to wonder if your table design is correctly normalized - having 25 fields with many lines of repeating data suggests that you might be better to have these 25 fields in a parent table and the remaining fields in one or more related child tables, and use a Form - Subform arrangement.
