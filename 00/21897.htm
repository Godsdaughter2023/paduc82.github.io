Count Check Boxes in an Access Form - Microsoft Community

kenmarwick say:

Count Check Boxes in an Access Form

I have a continuous form in Access 2013 with a list of items, each item has a check box to select the item. Is there a way to count the number of boxes that are checked on the form?

I appreciate any help on this.

john w. vinson say:

Yep, and it's pretty easy. A Yes/No field is stored as 0 for False/No/unchecked, -1 for True/Yes/Checked; so you can put a textbox in the form Footer with a control source

=Abs(Sum([fieldname]))

If there are five rows with the field checked, the sum will be -5, and Abs will make it just 5.

scottgem say:

Just a point of clarification here. Forms don't store data so you don't count values on a form, you count the values in the Recordset the form is bound to. This is an important concept to understand. If you have to deal with the data a form displays, you have to remember you are dealing with the recordset returned by the form's Recordsource. So the Sum function that John mentioned will sum the values of the checkbox control in reference to records currently displayed on the form. 

Change the records the form displays and the expression will adjust. 

ken sheridan say: This suggests to me that you are doing what is known as 'encoding data as column headings'.  This is a common error (sometimes colloquially known as committing spreadsheet), but violates a fundamental principle of the database relational model, the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

The correct way to model this sort of thing is by two related tables, with each value currently represented by a TRUE value (checkbox ticked) being represented by a row in a referencing table.  Counting the values then becomes a simple matter of aggregation over each subset of rows by means of the COUNT operator.

You'll find an example of how to automatically decompose an incorrectly designed table into correctly normalized related tables as UnencodeColumns.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the option for Boolean (Yes/No) Values is appropriate to your situation.  If you take a look at the relationships window in the file you'll see that the corrected model is made up of three tables as follows:

Stores---<StoreStock>----Stocks

In which StoreStock models the many-to-many relationship type between Stores and Stocks by resolving it into two one-to-many relationship types.  In my file to count the number of items stocked per store a query as follows would be used:

SELECT Store, COUNT(*) AS ItemsStocked
FROM Stores INNER JOIN StoreStock
ON Stores.StoreID = StoreStock.StoreID
GROUP BY Store;

You can if you wish enter data via check boxes in the way you are doing, while maintaining a correctly normalized model, but you need to use an unbound controls and code to read and write the values from the tables.  The VolunteerRota demo in my same OneDrive folder includes forms illustrating both the conventional code-free method of a bound form/subform, and the use of unbound checkbox controls over which transparent command buttons are superimposed.

ken sheridan say:

1. I am actually only using a check box per record to indicate whether the item is active or inactive.

2. I can't count them in the underlying table as this is a weekly operation and the table contains checked boxes from all previous weeks.


1.  That is exactly what is meant by 'encoding data as column headings'.  The 'item' descriptors currently used as column headings are data, i.e. values.  The way this would be correctly modelled  in a related table would be for the descriptor to be a value in a column, so if there are 5 items active in relation to a row in the table, the referencing table would have 5 rows with the same foreign key value in one column and the descriptors of those items as values in the other column.

2.  As Scott says this is easily achieved by grouping the data per week, assuming there is a week or date value in a column to allow this.  The query I posted which would count the stocked items per store in my demo does this by grouping by store.  In your case the grouping is likely to be by two columns, one being whatever the count of valid items relates to, the other being the week.

You might think we are being unduly relationally purist about this, but this is not a matter of theory for theory's sake; getting the design correct in accordance with the tried and tested principles of the theoretical relational model really is the key to an efficient, reliable and robust database application.

One possible reason why you are seeing two accounts under one email address is because you might have signed in to Skype using your account's associated email address instead of your Skype Name without merging them first, thus resulting to a new Skype account.

All Skype accounts were transitioned to basic Microsoft accounts last October 2016, although users with legacy accounts (Skype Name-created accounts) still need to fully update their accounts to enjoy the benefits of a full Microsoft account.
