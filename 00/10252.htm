jllatham say:

I hope I don't mess this up because I'm definitely not an expert on the fine differences between a standard module and a class module.  But I'll go so far as to say what you wrote in your response to Andreas Killer seems you've got a handle on things.

My background is mostly in Excel and I think speaking from that background may help you some.  In a workbook there several objects 'exposed' within the VBProject right away.  By default the Workbook object and all worksheet objects are exposed.  You can choose one of those objects and write code that actually responds to certain defined events for them - and it turns out that the code modules for the workbook and each worksheet is a Class Module.  But we don't have to define the events for them, they are pre-defined within Excel.  But we can add a Class Module and define a class, which generally requires us to define what events it can respond to and how it will react to that event.  Then we can declare a NEW object based on that class module and that object will respond to the set of events we defined, and each NEW object we create based on that one class module will inherit the ability to respond to the events, with each object's responses being "remembered" distinctly from the others while the object exists.

This all goes along with Andreas' statement "A class module is an object..."

I don't know that there's a rule that says you can't put more than a call to a routine in a standard module within an object's defined event process.  What it does when you click it or double-click it or whatever, becomes one of its properties (or probably more appropriately one of its methods).  But I think at least convention says you just set up to call a routine in a standard code module; if nothing else this would appear (at least in my mind) reduce the size of the application at runtime: having a dozen objects each with just one line of code calling a routine in a standard module should take less resources than those same dozen objects each with the same several (hundreds?) lines of code effectively duplicated for each of those objects.

But in a standard module you cannot say *when CommandButton1 on a worksheet (or slide) is clicked, do this*.  But you can set up in a Class Module to say *when an object of this class is clicked on, then run the code in a standard module named "CommandButton1Action"*.  That's because standard modules are ignorant of any given object's events [and again it goes along with Andreas' statement "..., a normal module is just code."].  If you take a PowerPoint slide and go to the [Developer] tab and create a command button on a slide, you can view the code for it and you'll see a stub inserted like:
Private Sub CommandButton1_Click()

End Sub
and you can put a whole lot of code within that stub or simply have it execute a routine in a standard module.  And the reason that controls work that way is that they already have been defined as a particular class with specific events identified for use.  You don't have to create a class module for the controls available on the [Developer] tab.  You'll also notice that the code for the control is placed within the slide's code module, which in itself is a Class Module.

I hope this helps a little and doesn't add to your confusion.

Option Explicit

'Our "light bulb"
Event ShowLight(ByVal Light As LightCondition, Position As Range)

Public Enum LightCondition
  Out = 0
  Red = 1
  Yellow = 2
  Green = 3
End Enum

Dim Light As LightCondition

'This is the light one the same street from the other direction
Public TLsame As TrafficLight
'This is one light of the crossing street
Public TLcross As TrafficLight
'Position of the light in the sheet
Public Position As Range

Property Get Condition() As LightCondition
  Condition = Light
End Property

Property Let Condition(Value As LightCondition)
  'Change the light
  Light = Value
  'Show it
  RaiseEvent ShowLight(Light, Position)
  With TLsame
    'Make sure our counterpart has the same color
    If .Condition <> Light Then .Condition = Light
  End With
End Property

Sub StopTraffic()
  'Already stopped?
  If Condition = Red Then Exit Sub
  'Do the change
  Condition = Yellow
  Wait
  Condition = Red
  Wait
End Sub

Sub StartTraffic()
  'Already goes?
  If Condition = Green Then Exit Sub
  'Stopp the crossing street first!
  TLcross.StopTraffic
  'Do the change
  Condition = Yellow
  Wait
  Condition = Green
  Wait
End Sub

Private Sub Wait()
  'Wait one second
  Application.Wait Now + TimeSerial(0, 0, 1)
End Sub

carmellob. say:

I had this same issue. To fix this, do the following:

1. Open PowerShell as Administrator

 

2. Find the oem<xx>.inf

pnputil /enum-drivers | Out-File oem_all.txt

3. Open C:\Windows\System32. Scroll down or search, then open oem_all.txt. In Notepad, click Find in the top menu. Find "netbridge".

4. Read the "oemxx" next to the block of text with "netbridge" in it.

 

in my case it's oem28.inf

 

5. Open PowerShell as Administrator again, and type

pnputil /delete-driver oemxx.inf

(replace xx with your number)

6. Open C:\Program Files (x86)\VMware\VMware Workstation, and right click "netbridge.inf" and click Install.

7. Open Control Panel and add the Bridge Protocol.

NOTE in build 18290 Windows may run into a GSOD while starting a VM.

ken sheridan say: The Execute method executes an 'action' query; it does not open a query in datasheet view, which I assume is what you are attempting.  For this sort of 'on the fly' task I use the following function:

Public Function OpenTempQuery(strSQL As String)

    Dim qdf As DAO.QueryDef
    Static n As Integer
    
    n = n + 1
               
    ' delete temporary querydef object if exists
    On Error Resume Next
    CurrentDb.QueryDefs.Delete "Temp" & n
    Select Case Err.Number
        Case 0
        ' no error
        Case 3265
        ' temporary querydef does not exist
        ' ignore error
        Case Else
        ' unknown error
        MsgBox Err.Number, vbExclamation, "Error"
    End Select
    ' create temporary querydef object
    Set qdf = CurrentDb.CreateQueryDef("Temp" & n)
    CurrentDb.QueryDefs("Temp" & n).SQL = strSQL
    ' open query and then delete temporary querydef object
    DoCmd.OpenQuery "Temp" & n
    ' delete temporary querydef object
    CurrentDb.QueryDefs.Delete "Temp" & n

End Function

Call it like this:

OpenTempQuery "SELECT [mess] FROM [etypes] WHERE [etypecode] = 'CURR'"

Because the integer variable n is declared as Static, giving each temporary querydef object's name a distinct suffix, the function can be called multiple times while leaving each datasheet open.  This can be useful for ad hoc 'what if?' comparisons, which is why I wrote it in the first place.

Doug, this is not about sets of styles but rather [Quick] Style Sets. Custom Quick Style Sets are special .dotx templates stored in the QuickStyles folder.  They could be regular Word templates. They are not stored in the normal template, although the styles can come from the normal template. They can work in conjunction with Themes. I agree that document templates are the most straightforward solution. However, quick style sets provide for a global solution, like themes, in that they will allow comprehensive changes in document or template format in a single step.

I certainly agree that the normal template should never be shared.
