ken sheridan say:
Yes the fields have null values :)


In that case change the query to:

 

PARAMETERS [Forms]![Form1]![date from] DATETIME,

[Forms]![Form1]![date to ] DATETIME;

SELECT kat.Date, kat.[Wallet no], kat.Status, kat.Location, kat.[Box no]
FROM kat

WHERE (kat.Date) >= [Forms]![Form1]![date from] OR  [Forms]![Form1]![date from] IS NULL)

AND (kat.Date < [Forms]![Form1]![date to]+1 OR [Forms]![Form1]![date to] IS NULL)

AND (kat.[Wallet no] = [Forms]![Form1]![wallet] OR [Forms]![Form1]![wallet] IS NULL)

AND (kat.[Status] = [Forms]![Form1]![Combo10] OR [Forms]![Form1]![Combo10] IS NULL)

AND (kat.[Location] = [Forms]![Form1]![Location] OR [Forms]![Form1]![Location] IS NULL)

AND (kat.[Box no] = [Forms]![Form1]![Box] OR [Forms]![Form1]![Box] IS NULL);

 

Testing for equality with the parameter OR the parameter being NULL makes a parameter optional without failing to return rows where the relevant column position is NULL, as the use of the LIKE operator does, as any comparative operation involving NULL evaluates to NULL, neither TRUE nor FALSE.  The correct parenthesizing of the Boolean OR operations is crucial here to force them to evaluate independently of the AND operations.

 

Note that the date/time parameters are declared, which they should be to avoid the values of the parameters being misinterpreted as arithmetical expressions, but the other parameters are not.  A date/time parameter can be declared and still be NULL, but declared parameters of other data types cannot be NULL.

    I suspect that you were - or Access erroneously thought you were! - adding a second textbox to the form where there already was one.

You can change the font size or any of the other display properties at the same time.

    Also, don't confuse "fields" with "controls". Tables have Fields containing data; Forms don't have fields at all. Instead they have Controls of various types: Textbox controls (usually bound to Fields in the form's recordsource), Label controls (containing human-readable information about the associated textbox), Combo Box controls, Checkbox controls, on and on. The properties of any of these controls can be easily changed by just changing its properties in design view.

    Why can't MS create a fix for 97-2003 excel for the colour - Microsoft Community

    lindaoraya say:

    Why can't MS create a fix for 97-2003 excel for the colour palette?

    The one thing that is driving me insane is that any colours I choose in Excel 2010, turn into garish neon colours in 97-2003 Excel, or sickly ill yellow.  Come on MS, it's not a lot to ask for.
    jllatham say:

    " The colours from 97-2003 carry over to 2007-2010 no problem.  It's just the other way around that's the problem."

    Think about that for a moment - what you have is a case where 2007-2010 have more color display capabilities than did 97-2003, so the 97-2003 colors become a kind of sub-set to the ones available in 2007/2010.  So colors from 97-2003 can be rendered in 2007/2010, but not the other way around.  Kind of a "all Scotties are dogs, but not all dogs are Scotties" situation.

    See this web page for some explanation and possibly even a free tool that could help you with the problem (can't vouch for the tool as I have not tried it).

    http://excelcampus.com/tools/excels-color-palette-compatibility-solution
