I probably code things more like

Private Sub Form_Current()
    On Error GoTo Error_Handler
    Dim db                    As DAO.Database
    Dim rs                    As DAO.Recordset

    If IsNull(Me![ColorID]) = False Then
        Set db = CurrentDb
        SQL = "SELECT Table.R, Table.G, Tables.B " & vbCrLf & _
              "FROM Table " & vbCrLf & _
              "WHERE (Table.ColorID=" & Me![ColorID] & ");"
        Set rs = db.OpenRecordset(SQL, dbOpenSnapshot)
        With rs
            If .RecordCount <> 0 Then
                Me.Control.BackColor = RGB(Nz(![R], 0), Nz(![G], 0), Nz(![b], 0))
            End If
        End With
    End If

Error_Handler_Exit:
    On Error Resume Next
    If Not rs Is Nothing Then
        rs.Close
        Set rs = Nothing
    End If
    If Not db Is Nothing Then db = Nothing
    Exit Sub

Error_Handler:
    MsgBox "The following error has occured" & vbCrLf & vbCrLf & _
           "Error Number: " & Err.Number & vbCrLf & _
           "Error Source: Form_Current" & vbCrLf & _
           "Error Description: " & Err.Description & _
           Switch(Erl = 0, "", Erl <> 0, vbCrLf & "Line No: " & Erl) _
           , vbOKOnly + vbCritical, "An Error has Occured!"
    Resume Error_Handler_Exit
End Sub

a. user say:

For Word 2016, you may be able to connect via DDE as follows:

 a. open the Worksheet in Excel (this is the only way I can currently get the rest of the procedure to work)

 b. go to Word->File->Options->Advanced->General (near the bottom of the Advanced options)

 c. check the Confirm file format conversion on open option

 d. click OK

 e. go through the connection process again. When you try to connect to the Excel file, you should see a dialog titled Confirm Data Source, and probably just one option (OLE DB Database Files)

 f. Don't choose that option. Check Show All then select MS Excel Worksheets via DDE (*.xls), then click OK.

 g. you may then see the usual DDE connection dialog.

This problem is usually a "mixed data types in the same column" problem. I would expect to get it if

 a. the first 8 cells in the text column had "short texts" (fewer than around 256 characters), and

 b. one or more cells further down had longer texts. 

If that is the situation, you may also be able to fix it by putting a long note in one of the first 8 cells in the column. Some people do that by inserting a dummy data row under the title row, with a "long note" in the appropriate column, then excluding it or discarding it from the merge, 

[If it is that. it's because the default method that Word uses to get data from Excel (OLE DB) treats Excel data as a database, and wants to assign a specific data type to each column. It applies various rules if the types are mixed. Unfortunately, it regards TEXT (short string, up to 255 or 256 long) and MEMO (longer string) as different types.]

teylyn say:

Hello,

AFAIK, the hidden columns/sheets never get included in the PDF, so there is no way to unhide them.

With a pdf printer tool, the output that would normally be sent to a printer and be rendered on paper, is now saved in an electronic portable document format.  The PDF contains only what you can see in the Print Preview dialog of Excel. Nothing more.  

In addition to that: If you need to unhide columns, rows or sheets that are hidden with Excel, you would need a piece of the Excel functionality in the PDF reader. A PDF reader tool does not have that functionality.
