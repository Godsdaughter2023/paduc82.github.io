Ligatures are decorative or joined characters that are available for certain characters in some fonts.

Ligatures 2- Enable Ligatures in Microsoft Word https://www.vletter.com/help/font-faq/enable-li...
Advanced features like ligatures provide for special rules within the fonts that can tell the text routines in the system to substitute one representation of a character for another.



This next article is old, but especially at the end it presents a couple of worthwhile tips
Ligatures in Microsoft Word
http://editorium.com/archive/ligatures-in-micro...

Ligatures, in case you didn’t know, are letters that have been mashed together as one character. Why would anyone want that? For aesthetic reasons. (Yes, there is an ae ligature for words like “aesthetic.”) The ligatures used most often are fi and fl, accompanied by their friends ff, ffi, and ffl. That’s because these letter combinations really do look kind of ugly in many fonts, with the dot on the i running into the top of the preceding f, and so on.

*****************************
If you are at the point of using ligatures, you should also look into using Kerning. Kerning is reducing space between specific letter pairs that normally appear to be spaced too far apart. ie AV or AW or LT. It is most commonly used with larger font sizes in places like heading and banners.

Word doesn't reinsert a deleted \* MERGEFORMAT switch unless there is a formatting change... You can confirm that it does so by inserting a cross-reference via Insert|Cross-reference, checking the field code, then changing it's bold/italic formatting, and re-checking the field code. joeu2004 say:

Jim Michaels wrote:

perhaps I have caught a fluke, a floating point oddity

Exactly right.  But it's not a "fluke" insofar as it is very common.  Often we fail to notice it because its appearance depends on a number of coincidences.  See my comments in the discussion at http://social.answers.microsoft.com/Forums/en-US/excslcreate/thread/ed394a0a-5339-4034-a95f-c2383a36d2aa.

Jim wrote:

this didn't used to happen with office 2003
Actually it does.  For example, IF(10.1-10=0.1,TRUE) returns FALSE(!) in Excel 2003 because of the very problem that you are running into.  And the aforementioned discussion is about a similar observation by an Excel 2003 user.

Jim wrote:

the only workaround I know of is to change the C5 function to
=IF(ABS(A5)<0.01,"Reconciled!----------","Not Reconciled--------")
and to change A5 to =ROUND(A7-A3,2)

IMHO, explicit rounding of arithmetic expressions is the better solution; and it should be sufficient.  In other words, the formulas should be:

in A5 put =ROUND(A7-A3,2)
in A7 put =ROUND(A4+E9-D9,2)
in D9 put =ROUND(SUM(D11:D47),2)
in E9 put =ROUND(SUM(E11:E52),2)

Then it should be sufficient to write IF(A5=0,...) as you wish.

However, sometimes we also need to use round in the IF() expression.  For example, IF(ROUND(10.1-10,2)=0.1,TRUE) fixes the aforementioned problem.

Jim wrote:

is floating point error taken care of in excel, or do I have to deal with that nastiness in all my stuff and make all my functions longer?

Well, Excel does have a dubious heuristic that is poorly described under the misleading title "Example When a Value Reaches Zero" in the web page athttp://support.microsoft.com/kb/78113 .  BTW, that KB attempts to explain the nature of the fundamental problem in gory detail.

But the design and implementation of that heuristic is half-baked.  It does not "correct" all such floating-point anomalies even when you might expect that it should (e.g. your formula in A5); and it does not apply to such floating-point anomalies that arise in Excel functions like SUM.

For that reason, I suggest explicitly rounding most formulas, for example any expression that should result in an "exact" dollar-and-cents value.

For the particular numbers that you present, the problem can be summarized as follows....

Most numbers with decimal fractions cannot be represented exactly due to the way that Excel stores numbers and performs arithmetic, namely 64-bit binary floating-point.

Consequently, the constant 179.36 in A4 is actually exactly 179.360000000000,0136424205265939235687255859375.  The SUM(D11:D47) in D9 is 164.5, which can be represented exactly.  But the expression A4+E9-D9 in A7, which appears as 14.86, is exactly 14.8600000000000,136424205265939235687255859375.

However, the constant 14.86 in A3 is actually exactly 14.8599999999999,994315658113919198513031005859375.  Note the infinitesimal difference from the representation in A7.  That infinitesimal difference is reflected in the result of the expression A7-A3 in A5.

Note that you cannot see these differences even if you format A3, A4 and A7 to 15 or more decimal places.  This is because Excel will format only up to 15 significant digits, rounding the 16th digit.

PS:  Alternatively to the prolific use of explicit rounding, some people might suggest setting the Precision As Displayed (PAD) calculation option.  However, I deprecate its use for a number of reasons.  For one thing, it would not fix the problem with IF(10.1-10=0.1,TRUE).  Also, PAD can result in changing constants undesirably and irreversibly if you are not careful.  For that reason, if you choose to experiment with PAD, first make a back-up copy of the Excel file.
