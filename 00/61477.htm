Doc,

You really should have started your own thread.  If you thought that there was some relevant information in this 20-month-old thread, you could have just included a link to it.  I only became aware of your post because I never turned off notifications on the old thread.

My initial observation is that you really shouldn't be using WEP encryption, especially in a medical practice subject to HIPAA.  These days, WEP (even 128-bit WEP) is very close to no security at all.

Passing that, however, the usual reason that people used to have connecting to WEP-secured wireless networks was that they entered the "password" rather than the hexadecimal encryption key. 

Whoever setup your wireless security entered some password in the router's configuration utility.  The router's internal software then converted that password into a hexadecimal encryption key of the correct length.  In your case, the 128-bit WEP key is generated from a 13-character ASCII password. See pages 10-11 (pdf pages 14-15) of the User Guide for your SMCWEB-N.

 
The problem arises if the router software uses a different algorithm for generating the hex key from the ASCII password than the algorithm used by whatever utility is managing your wireless adapter.  In that case, when you enter the ASCII password, your wifi adapter ends up using a different encryption key ... and naturally can't connect to the wireless access point.

The solution -- other than using WPA2 or WPA encryption (and, if your practice can afford it, WPA2-Enterprise rather than WPA2-Personal) -- is to find out from the person who has access to the router's configuration utility what the 26-character hexadecimal encryption key is (rather than the 13-character ASCII password) and enter that when asked for your password.

quietman7 say:
 Regsvr32 is a legitimate Windows process used to register and unregister OLE controls such as .dll (Dynamic Link Library) modules or ActiveX Controls (OCX) files which too can be legitimate or sometimes malware related.

It is not uncommon to receive such an error(s) when booting into Windows if the file was removed inadvertently, during the uninstallation of a program or after using anti-virus and other security scanning tools to remove malware.

A "Cannot find...", "Could not run...", "Error loading... or "specific module could not be found" message is usually (but not always) related to a malware file that has been deleted. Windows is trying to load this file but cannot locate it since the file was mostly likely removed. However, an associated orphaned registry entry still remains and is telling Windows to load the file when you boot up. Since the file no longer exists, you need to remove this registry entry so Windows stops searching for the file when it loads.

To resolve this, download AutoRuns and save it to your Desktop.
  • Create a new folder on your hard drive called AutoRuns (C:\AutoRuns) and extract (unzip) the file there.
  • Open the folder and double-click on autoruns.exe to launch it.
  • Vista/Windows 7/8 users right-click and select Run as Administrator.
  • Please be patient as it scans and populates the entries.
  • When finished scanning, it will say Ready at the bottom.
  • In the top menu, click File > Find... and type the file name (glu32.dll) related to the error message, then click Find Next.
  • Alternatively, you can scroll through the list and look for an entry related to the file in the error message.
  • If found, right-click on the entry and choose delete.
  • Exit Autoruns and reboot your computer when done.

Hi,

Disk Defragmenter is a tool that rearranges the data on your hard disk and reunites fragmented files so your computer can run more efficiently. We should keep in mind that Disk Defragmenter uses low-priority disk access and CPU usage so that it doesn't influence your current running programs. This automatically translates into a slower process. Disk defragmentation can take a long time if you are running it for the first time and also depend on the hard drive space and the data available. If can take up to 15 to 16 hours also depending on the data and bad clusters on the hard drive.

Disk Defragmentation takes a long time for one of the following reasons:-

• The number of file extents.

• Also the number of free space extents and the size of each will affect defrag time.

• Defrag runs at low priority, so if you are doing something else on the computer defrag will yield to whatever else is happening.
