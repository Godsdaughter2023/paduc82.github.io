scottgem say:

I would suggest you do some research on Error Trapping. There are different ways errors are triggered. Some errors are triggered when a form is loaded, such errors would be handled in the Form's On Error event. Some errors are triggered during the processing within a module. Those errors are trapped with an On Error statement. something like this

On Error GOTO Err_label

Your code

Exit_label:

Exit Sub

Err_Label:

SELECT CASE Err.Number

     Case xxxx

          MsgBox "your error message"

     Case else

           MsgBox Err.Number & "-" & Err.Description

End Select

Label can be anything you want, I usually reference the name of the procedure. The xxxx is the number of the error being trapped. You can have as many Case statements as you need if there might be multiple errors thrown. The On Error statement goes near the top of the code (usually) and the rest at the very bottom. 

If you are still getting the error message then its probably being triggered in the procedure where you open the form. Try putting the code I gave you there. Dave's code is only for Form errors.

ken sheridan say: Another way to handle runtime errors is by in-line error handling.  Say you have a button to move to a Next Record, but no next record currently exists, e.g. the form's recordset is empty or the form is at an empty new record:

Private Sub cmdGotoNext_Click()

    Const NO_NEXT_RECORD = 2105
    Const MESSAGE_TEXT = "You cannot currently move to a next record."
    
    On Error Resume Next
    DoCmd.GoToRecord acForm, Me.Name, acNext
    Select Case Err.Number
        Case 0
        ' no error; ignore
        Case NO_NEXT_RECORD
        ' anticipated error; inform user
        MsgBox MESSAGE_TEXT, vbInformation, "Invalid Operation"
        Case Else
        ' unknown error; inform user
        MsgBox Err.Description, vbExclamation, "Error"
    End Select
    
End Sub

This technique can be used for instance if, following an anticipated runtime error, you  need to conditionally execute one of two or more possible courses of action depending on the error, possibly before further code in the procedure is executed, which you can do in the Select Case construct.  The same can be done by branching to an error handler in which the code Resumes at a point in the code following the line which has raised the error, but I find in-line error handling to be a simpler technique in this context.

a_user29 say:

I’d like to think that the developers might have belatedly realized that consistent worst-in-class rankings for “self protection” are not really such a good thing after all. But I suspect that this was more likely prompted by the specific realization that stopping the MsMpSvc service is currently an attack vector for some very prevalent malware:

http://social.technet.microsoft.com/Forums/pl/FCSNext/thread/9937bf63-c6f8-4ab0-adef-8a3f77270968

Live Security Platinum specifically attempts to terminate the MpCmdRun.exe, MsMpEng.exe, msseces.exe, and NisSrv.exe processes, as well as to stop and disable the MsMpSvc service – as if it was custom designed to exploit MSE’s well known self-protection vulnerability.

http://www.microsoft.com/security/portal/Threat/Encyclopedia/Entry.aspx?Name=Live+Security+Platinum

I certainly don’t buy into the philosophy that you can’t take privileges away from malware without also taking those same privileges away from the user; but if taking unnecessary privileges away from the user also takes those privileges away from malware; then I think that’s a step in the right direction.
