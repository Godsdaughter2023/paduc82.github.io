The ntdll.dll, which appears to cause the crash, is a component of Windows.

However another module may actually be causing the crash.

To get more information on the crash I might suggest to try to obtain a user mode dump which has the potential to pinpoint the cause.

You can configure Windows to create user-mode dumps. Create a System Restore Point first.

If you copy and paste the following in Notepad and save as a .reg file (save with the .reg extension and give it any name such as dump.reg), then right-click the .reg file and select "Merge" to add to the registry, a .dmp file at the time  "Scripted Diagnostics Native Host" crashes should be created in the C:\CrashDumps folder:

Windows Registry Editor Version 5.00
[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\Windows Error Reporting\LocalDumps\sdiagnhost.exe]
"DumpFolder"=hex(2):43,00,3a,00,5c,00,43,00,72,00,61,00,73,00,68,00,44,00,75,\
00,6d,00,70,00,73,00,00,00


After  Scripted Diagnostics Native Host (sdiagnhost.exe) crashes, can you then make the .dmp file available (provide link) via a public folder on OneDrive or similar site?

Here's a link on using OneDrive:

http://windows.microsoft.com/en-ca/onedrive/share-file-folder

More info on collecting user-mode dumps:

http://msdn.microsoft.com/en-us/library/bb787181(VS.85).aspx

To stop the creation of the user mode dump files run the following registry file:

Windows Registry Editor Version 5.00

[-HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\Windows Error Reporting\LocalDumps\sdiagnhost.exe]

You can delete the C:\CrashDumps folder anytime.

Ok after much searching and reading and copious quantities of coffee I finally struck on a work around until M$ resolves this patch issue.  It appears that KB4480970 is the culprit.  Uninstalling that KB does not fix the problem.  Using System Restore and selecting a restore point before the update does not fix the problem.  I also tried modifying my registry but that didn't fix the problem.  These were suggested in some of the posts that I read.

As I am reading it occurs to me that the update somehow screws up Network Level Authentication.  

So here is the change I made to get Remote Desktop Connection to work.

  1. Go to "Control Panel\All Control Panel Items\System"  (open Control Panel and click on System)
  2. Click on "Remote Settings".  That will open the System Properties window at the "Remote" Tab.  Currently the selection is "Allow connections only from computers running Remote Desktop with Network Authentication (more secure)"
  3. Select "Allow connections from computers running any version of Remote Desktop (less secure)"
  4. Do this on all your computers.
  5. Reboot all the computers
  6. RDC now works.

This is a workaround until M$ fixes that bad patch.  Once the fix is in place I will go back and again use the more secure connection method.

I can ping, but I can't surf - Microsoft Community

dptb_dptb_dptb say:

I can ping, but I can't surf

Two weeks ago I bought an ASUS EeePC 1005PE Netbook with Windows 7
Starter Edition. At first, it was operating without any trouble, but as
of last week I am no longer able to surf the internet over my wireless
home network. This is despite the fact that I can ping the outside world
over that connection. Below I will list the symptoms that I have
observed and the remedies that I have tried. (All have failed so far.) I
would be very grateful for any advice that readers of this forum might
have.


HARDWARE and SOFTWARE

Machine: ASUS EeePC 1005PE

OS: Windows 7 Starter

Processor: Intel Atom N450

Wireless Adapter: Atheros AR9285

Wireless Driver version 8.0.0.238 (Appears to be the most recent)

Router: Netgear 3700, with updated firmware and SSID broadcast

Wireless security: WPA2-PSK [AES]

The netbook is set up for DHCP and automatic DNS, and the router is set
up accordingly.


SYMPTOMS

The "Network and Sharing Center" reports that my connection to the
router and my router's connection to the internet are active. This is
confirmed by my ability to ping outside addresses (e.g., www.yahoo.com)
from the "Command Prompt". However, when I try to surf the web using
Internet Explorer, the connections time out. Typically, the comment in
the bottom left hand corner of Internet Explorer says "Website found",
but the connection times out. The offer to diagnose the connection
problem that is offered by Internet Explorer offers no solutions.


To try to troubleshoot this issue, I have tried a number of
configurations. Here I will divide the results of those tests into
configurations that generate the same symptoms, and configurations that
work as expected.


Configurations that yield the same symptoms:

* Windows 7 Starter under "Safe mode with Networking" option.

* Trying to surf with Firefox also yields the same symptoms.

* Disabled windows firewall and removed all security from the wireless
connection. Still the same symptoms, even after rebooting the router.

* As one might expect, Skype also fails to connect, even though ping is
able to.


Configurations in which I can both ping and surf:

* Restarting the machine under the "Express Gate" operating system that
ships with EeePCs. My understanding is that this is a version of the
"Splashtop" Linux distribution. Under this operating system I am able to
both ping and surf the internet without any trouble.

* Disabled the wireless facilities and connected to the same router over
the wired ethernet. Everything works fine in this case, too, under both
Windows 7 Starter and Express Gate.

* Took the netbook to work, and I can ping and surf on the wireless
network there, too, under Windows 7 Starter and Express Gate.

* My other laptops (Old IBM Thinkpad running Win 2000 and a Mac Book
Air) operate as one would expect, as does my iMac when I remove the
ethernet connection and employ the on-board wireless.


Based on these tests, it appears that there is something about the way
in which Windows 7 Starter interacts with my wireless router that is
leading to this problem. However, it is interesting to see that this
problem does not arise under the Express Gate/Splashtop operating
system.



ATTEMPTED REMEDIES

Based on my reading of some of the rather informative discussions on
this forum I have attempted the following remedies.

* Applied Windows 7 Updates

* Reset TCP/IP

* Flushed the DNS cache

* Reset Winsock

* Made sure that the only enabled wireless connection is the one to my
router. (under "Manage Wireless Networks")

* Searched for updated diver for Wireless Adapter, both automatically
and manually. It appears that the driver that I have had all along is
the most recent.

* Searched for any evidence of the "Bonjour" service. I didn't find any.

* Checked the list of services that sometimes appears on this forum, and
they all seem to be running appropriately.

* Performed a "clean boot", with everything except for the Microsoft
services disabled.

* Performed a "hard reset" of the router.

* Disabled the "IP Helper", and disabled IPv6. Reset TCP/IP and restarted.


Unfortunately, none of these attempts resolved the situation.


I would like to thank those of you who have made it this far. If anyone
has an ideas for different test configurations, or different remedies, I
would be very grateful to hear of them.

Thanks very much.

dptb-dptb-dptb say:

Once again, thank you all for your interest in my problem.

 

I am happy to say that I have a resolution to report.

 

Following some feedback on a forum on the ASUS web site, it was suggested that under Windows 7 I disable the 802.11n functionality and force the internal Atheros adapter to use 802.11b/g.

 

This resolved the problem, and, fortunately, my requirements are not so sophisticated that I will need the additional features of 802.11n.

 

Just in case any one else wants to try this, I did this by:

Going to the Network and Sharing Center

Clicking on "Change adapter settings"

Right clicking on the Wireless Network Connection and selecting Properties

Clicking "Configure"

Selecting the "Advanced" tab

Clicking on "Wireless Mode Selection", and choosing "802.11b/g" rather than "Auto"

 

Thank you, once again, for all your suggestions.

a. user say:

Hi,

Thanks for the reply. None of those options you suggested were available - but I found the answer.

It appears that for some reason the Registry key "SensorPresent" at this location {HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\AutoRotation} was set to 0.

 I modified it to 1 and the auto rotation returned, along with the usual methods of turning it on and off!
