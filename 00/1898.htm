ken sheridan say: The approach I've always used is to have a separate table, Defaults with columns FormName, ControlName and DefaultText, all of text data type.  You can then open a continuous forms view form bound to this table from any bound form via a command button, e.g.

DoCmd.OpenForm "frmDefaults", _
     WhereCondition:="FormName = """ & Me.Name & """", _
     WindowMode:=acDialog, _
     OpenArgs:=(Me.Name)

In the same form's Open event procedure put:

    Dim ctrl As Control
    Dim strCriteria As String
    Dim varDefaultValue As Variant
   
    For Each ctrl In Me.Controls
        strCriteria = "FormName = """ & Me.Name & """ And ControlName = """ & ctrl.Name & """"
        varDefaultValue = DLookup("DefaultText", "Defaults", strCriteria)
        If Not IsNull(varDefaultValue) Then
            ctrl.DefaultValue = """" & varDefaultValue & """"
        End If
    Next ctrl

Then in the AfterUpdate event procedure of the DefaultText control in the frmDefaults form put:

    Forms(Me.OpenArgs).Controls(Me.ControlName).DefaultValue = """" & Me.DefaultText & """"

And in frmDefaults Open event procedure put:

    Me.Controls("FormName").DefaultValue = """" & Me.OpenArgs & """"

The above gives the ability to set or amend the default value for any control, but if you want to limit it to specific controls then fill the Defaults table with the form and control names, and set the frmDefaults' AllowAdditions and AllowDeletions properties to False (No) so that users can't add a new record.  Set The Locked property of the FormName and ControlName controls to True (Yes) and their Enabled property to False (No) to prevent users editing the form or control names.  They will then be limited to editing the DefaultText control, though a really perverse user could open and edit the table directly of course.

You may be wondering why the DefaultText column is a text data type when it has to cope with values for numeric and date/time data types as well.  The DefaultValue property is in fact always a string expression, regardless of the data type, so the text column can happily contain values for all these data types provided the value are entered in a format which can be interpreted correctly for the control in question.  One thing this does mean is that default dates can be entered in the local regional format, e.g. dd/mm/yyyy in my case.  The fact that the value is a string expression is why its wrapped in literal quotes characters when assigning the value to the property.  This is especially important with dates as otherwise a value such as 13/01/2011 would be interpreted as an arithmetical expression and give the wrong result (30 December 1899 00:09:19 as it happens, this being day-zero in Access's date/time implementation, and 13/01/2011 as an arithmetical expression evaluating to 0.00646444554947787).

Ken Sheridan, Stafford, England
