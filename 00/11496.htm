jay freedman say:
I'm afraid your memory of previous versions is wrong. Right now I'm looking at the same document in both Word 2003 and Word 2007, and their status bars display the same information:
  • The first item is the "formatted page number", which is the value that will be displayed by a {PAGE} field on that page. It's the same as the physical page number, unless you go to the Format Page Numbers dialog and change the starting page for the section. That is, if you set the section to start with 1, then the formatted page number shows the "page number within the section" that you're looking for; but otherwise it will show some other number.
  • The second item is the section number.
  • The third item is the physical page number followed by the total for the document. The display is different (Word 2003 shows "4/5" while Word 2007 shows "Page: 4 of 5"), but the information is the same.
If you haven't change the section starting page number, there is no display of the "page number within the section" in either version; you have to do the arithmetic yourself (or write a macro to show it).

For use in the Print dialog, though, you need the formatted page number, not the count of pages from the start of the section. For example, if you change the numbering of Section 3 to make it start on page 10, then to print the second page of that section you would enter p11s3, not p2s3.

How do you lookup wildcard characters (~, ?, *) as 'literal' in VLOOKUP?

 In Excel, I am trying to do a VLOOKUP; however, some of the lookup values contain the wildcard values of *, ~ and ?.  So when I do my lookup, Excel thinks it's actually a wildcard and not a literal value. For instance, I have the following table:

 

~A     B1

?A     B2

*A      B3

A~     B4

A?     B5

A*      B6

AA     ZZ

 

If I try to do a VLOOKUP on any of these codes, Excel will return 'ZZ' instead of the correct value I am looking for.  Is there a way to make Excel look at those values that contain those wildcard values as literal so that my VLOOKUP will work correctly without modifying the code?  And, if not, is there another way to do it or another formula that I could use?  Thanks, in advance, for any help you could provide. 

jeeped say:

... ~?~? up to ~?~?~?~?~? ... ~*~* up to ~*~*~*~*~* ... and I could also have an instance of this: ~~ up to ~~~~~ ...


You will need to use a series of SUBSTITUTE() functions in a specific order to transform multiple wildcards to a literal lookup value that will work within VLOOKUP().

               

The formula in E2 is,
=VLOOKUP(SUBSTITUTE(SUBSTITUTE(SUBSTITUTE(D2,"~","~~"),"?","~?"),"*","~*"),$A$1:$B$99,2,FALSE)
It may be of worth to note that you need to modify the tilde first, then the question mark and asterisk. The order of the latter two is inconsequential, but it is important to modify the tilde first.

a. user say:

That type is actually a wdMailmergeDataSource enum value and '5' means wdMergeInfoFromODSO. i.e. it isn't an ADO cursor type.

This reflects the fact that Word still uses different methods to open data sources depending on what they are (e.g. at one time you could connect to Excel via DDE, ODBC, OLE DB or a Word text converter). "ODSO" is "Ofiice Data Source Object", introduced around Word 2002 as a worker object that does OpenDataSOurce's job for certain connection types. AFAIK there is actually no way to get a VBA reference to the ODSO object that OpenDataSource uses so that route is also closed off.

I don't know of a way to specify an ADO cursor type when connecting via OpenDataSource, as I don't think there is any mechanism in the Connection String to do it. (In ADO it's something you specify on the Recordset or Command object AFAICR). My understanding is that when Word opens sources via ODSO it actually ignores anything you put in the ConnectString parameter, and if you actually want Word to take notice of specific connection parameters you have to create an external file such as a .odc and put what you need in that.

Many years ago I tried to suggest to Microsoft that they exposed a new OpenDataSource interface that would let people write their own Open, PrevRecord, NextRecord, Close type methods to give them more control and enable precisely the thing you are trying to do. But even if my message got through, it was clearly regarded as a minority requirement or possibly even a likely security risk. Pity!
