Bringing in an local image in mail merge - Microsoft Community

schlaefers say:

Bringing in an local image in mail merge

I'm working on a project where I use mail merge to create profiles on specific people.  I have a repository of head shots for these people and I want to use mail merge to programmatically update these pictures.  In my mail merge data I can put in the location of the file on my computer.  I want to have the mail merge throw whatever image file is in that location into my template.  Any help would be awesome.

PS. I'm using Microsoft Word for Mac 2011

a. user say:

The quick view (minor update made for Word 2016):

1. Let's suppose you are using Mail Merge to create a Catalog.

2. In your mail merge data source, create a column that contains the full path name of the image. Let's say the column is called imagepath, then it might contain

Macintosh HD:Users:userx:Pictures:pic1.jpg
Macintosh HD:Users:userx:Pictures:pic2.jpg
Macintosh HD:Users:userx:Pictures:pic3.jpg

etc.

If you are using Word 2016, you would need a path name that is separated by single forward slashes. Typically, the equivalent path names would look like this (i.e., the volume name "Macintosh HD" does not appear in the path):

/Users/userx/Pictures/pic1.jpg

/Users/userx/Pictures/pic2.jpg

/Users/userx/Pictures/pic3.jpg

Other path name examples in this message need to be treated in a similar way if you are using Word 2016 rather than 2011.

All these files must exist at the time when you perform the merge.

3. Create your mail merge main document, set it up to be a Catalog type merge, attach it to the data source, and save it as a .doc format (Word 97-Word 2004 format) file, not a .docx

4. If you want the catalog to contain links to the original pictures, insert the following nested field code in your mail merge main document:

{ INCLUDEPICTURE "{ IF TRUE "{ MERGEFIELD imagepath }" }" }

5. Alternatively, if you want the catalog just to contain the pictures and lose the links, or you are doing a Letter merge to email, insert the following nested field code:

{ IF { INCLUDEPICTURE "{ IF TRUE "{ MERGEFIELD imagepath }" }" } { INCLUDEPICTURE "{ IF TRUE "{ MERGEFIELD imagepath }" }" } }

6. Just to be sure, select the field codes and update them.

7. Save the mail merge main document and perform the merge.

Notes...

A. To insert the field codes, you may be able to use the following keys if they have not been "reserved" for other Mac features such as Expose:
  - cmd-F9 to insert each pair of field code braces {}, type everything else between the braces in the usual way
  - F9 to update selected field codes
  - Alt-F9 to toggle between field code view and field result view.

See, e.g. http://word.mvps.org/mac/FieldsInMacWord.html for further info. about key reassignment.

B. If you use the simpler construct
{ INCLUDEPICTURE "{ MERGEFIELD imagepath }" }
the problem is that Word does not insert the correct images as it goes along. That means that you have to merge to an output document, select it, and update all the fields. Using these IF TRUE constructs forces Word (somehow) to resolve the field codes properly. There are other things that affect field code behaviour, e.g. there are "Update fields on print" and "Update links on print," but the IF TRUE approach ultimately seems to be the simplest.

Tests here suggest that with the IF TRUE approach, you can merge to a catalog document, merge Letters to a new document, merge to printer, or merge to email (at least via Outlook, using HTML or attachment format), and the fields are correctly resolved in all cases. However, you should test that!

Notes on point (2) 
 a. Putting the entire path in the data source means you can easily use images from any location, at least on your local hard disks. However, you can use plain text + field codes to construct the full path "on the fly". For example, if all the files are in the same folder "Macintosh HD:Users:userx:Pictures", you can
    - use a column called imagefile in your data source. This would just contain 
        pic1.jpg
        pic2.jpg
        pic3.jpg
    - use the following field codes instead
{ INCLUDEPICTURE "{ IF TRUE "Macintosh HD:Users:userx:Pictures:{ MERGEFIELD imagefile }" }" }

If, e.g., all your files are .jpg files, you could leave the ".jpg" part out of your data source and use
{ INCLUDEPICTURE "{ IF TRUE "Macintosh HD:Users:userx:Pictures:{ MERGEFIELD imagefile }.jpg" }" }

and so on...
 b. Make sure that the full path name, however you construct it or store it, does not contain any characters such as spaces that should not be there. (It is very easy to introduce such spaces when copying/pasting with the smart copy options switched on).
 c. Tests here suggest that you can use either "colon-separated pathnames," e.g.
Macintosh HD:Users:userx:Pictures:pic1.jpg
or Posix-style pathnames, e.g.
Users/userx/Pictures/pic1.jpg

For the benefit of anyone who has used this approach on Windows Word, you do not need to "double up" the separator characters in the same way that you need to double up Windows backslash path separators "\\"

 d. If one of the image files does not exist, your output will probably show the same image as the last merged record instead. In some versions of Word, it is possible to avoid that by inserting a \d switch in the INCLUDEPICTURE field, e.g. as follows:

{ INCLUDEPICTURE "{ IF TRUE "Macintosh HD:Users:userx:Pictures:{ MERGEFIELD imagefile }" }" \d }

In that case, you may see a blank space instead of a picture. However, that fix does not appear to work in Word 2011, and it's probably quite hard to work around that particular problem any other way (i.e. you would probably have to resort to using VBA to do the merge).

Notes on point (3) 
The reason for using .doc is that .docx tends to try to remove { INCLUDEPICTURE } fields and does other nasty things with them. This is unfortunate because it means that you cannot use all .docx features and so on.

Notes on points (4) and (5)
The credit for working most of this stuff out on the Windows platform belongs to Word MVP "macropod," although AFAIK the simpler "IF TRUE" construct was known to others before that. I am not sure whether anyone knows why the undocumented "IF TRUE" construct functions at all.
