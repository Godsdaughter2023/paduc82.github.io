chrispeng_msft6626 say:

Hi Casaadmin255,

If a user has applied personal tags to folders or emails, messages that have a personal tag applied are always processed based on the personal tag's settings.

For the tags with an action “Delete and Allow Recovery”, emails will be moved to the Recoverable Items folder when reaching the specified retention age.

For the tags with an action “Move to archive”, if the archive mailbox is enabled, emails will be moved to the archive mailbox. If not, no action is taken.

Let us know if anything is unclear.

ken sheridan say: John has said it all really, but it's important to understand that, unlike a spreadsheet, a relational database, whether its created in Access or any other product, is a model of the part of the real world with which the database is concerned (usually shortened to the reality in the literature).  It models the reality in terms of its entity types (each modelled by a table), the attributes of the entity type (each modelled by a column), and the relationships between each entity type, which might be represented simply by a foreign key in one table referencing a primary key in another (a one-to-many, or more rarely a one-to-one relationship type) or by a table (in the case of a many-to-many relationship type) which resolves the relationship type into two or more one-to-many relationship types.

If the model accurately reflects the reality then it, and consequently the database, will behave in the same way as the reality, which is exactly what we want a database to do of course.  As it happens there is another file, DecomposerDemo.zip, in the same SkyDrive folder to which I referred you earlier, which shows how to 'decompose' (yes, that is the technical term) a single table imported from a simple Excel worksheet into a set of correctly related tables in Access.  Comparing the original single table with the set of related tables might give you an idea of how a relational database fits together, and how it differs radically from a spreadsheet.

ken sheridan say: Scott has previously posted details of the three tables for the model you describe.  In this the tblTimeSheet table models a many-to-many relationship type between tblEmployee and tblProject by resolving it into two one-to-many relationship types, on EmployeeID and ProjectCode respectively.  Diagrammatically the model is like this:

tblEmployee ----<tblTimeSheet>----tblProject

As regards a data entry form this will depend on how you wish to input the data.  You could for instance have form in single form view based on tblEmployee,  with a subform embedded in it, in continuous form view, based on tblTimeSheet.  Alternatively the parent form could be based on tblProject, with the subform again based on tblTimeSheet, to enter timesheet data per project rather than per employee.

A report based on a crosstab query is not straightforward as the column names returned by a crosstab query vary, each being a date.  So it is not simply a case of binding controls in the report to the columns.  An approach which avoids these sort of problems is to use a multi-column subreport to return the dates, and below this another multi-column subreport to return the hours per project per date.  You'll find an example of this approach as Payments.zip in the same SkyDrive folder to which I referred your earlier.  The report in this returns payments per customer per date.  In your case it would be the sum of hours per project per date, but otherwise it is analogous to mine.
