ken sheridan say:

I can't reproduce that behaviour, and I would not expect the code to do anything to an 'NA'  value other than strip any leading or trailing spaces and convert to upper case.  Let's look at how the piece of code in question works:

                ' The first line checks to see if the column is not a long integer data type
                ' and only executes the remaining code if that is the case.
                ' This is to prevent it trying to update an autonumber and raising an error.
                If fld.Type <> dbLong Then
                   ' The following line allows the values to be edited.
                    .Edit
                    ' The following line trims off any leading or trailing spaces
                    .Fields(fld.Name) = LTrim(RTrim(.Fields(fld.Name)))
                    ' The following line converts the value to upper case
                    .Fields(fld.Name) = UCase(.Fields(fld.Name))
                     ' The following line checks to see if the value is a valid
                     '  date and/or time value.
                    If IsDate(.Fields(fld.Name)) Then
                        '  If the value is a valid date/time value the following lines in
                        '  the current If….End If construct are executed.
                        '  The following line, which is split over two physical lines by
                        '  the underscore continuation character, converts the value
                        '  to a value of date/time data type, and then checks to see if
                        '  the date/time value is greater than or equal to 1, and if the value
                        '  formatted as a date is the same as the unformatted value.
                        '  If the value is greater than or equal to 1 this means that the
                        '  value contains a date.  This is because a date/time value is
                        '  implemented as a 64 bit floating point number in which the
                        '  integer part represents the days and the fractional part the
                        '  times of day.  Consequently a value of 1 or more must contain
                        '  a date.  It's actually a little more complex than that, but I'm
                        '  simplifying here for clarity.
                        '  If, as well as containing a date, the unformatted value is the
                        '  same as the value formatted as a date, then it does not contain
                        '  a time of day (again I'm simplifying here for clarity).  If both of
                        '  these criteria are met, i.e. the text value is a date without a time
                        '  then the value is formatted as yyyy/mm/dd.
                        If CDate(.Fields(fld.Name)) >= 1 _
                           And CDate(.Fields(fld.Name)) = Int(CDate(.Fields(fld.Name))) Then

                             .  the following line formats the value as yyyy/mm/dd

                             .Fields(fld.Name) = Format(.Fields(fld.Name), "yyyy/mm/dd")
                        Else
                            ' otherwise, i.e. if the text value is either a date with a time or
                            ' a time only, it is formatted as hh:nn:ss
                           .Fields(fld.Name) = Format(.Fields(fld.Name), "hh:nn:ss")
                        End If
                    End If
                    ' The following line applies the edits
                    .Update
                End If

So if the value IS 'NA', or any text value which is not a valid date and/or time, the following line:

    If IsDate(.Fields(fld.Name)) Then

will evaluate to False, so none of the code in the If….End If construct will be executed, and the only edits will be the stripping of any leading or trailing spaces, and the conversion of the text to upper case.

If you have faithfully copied and pasted the code I gave you in the right places, therefore, I can see no reason why you should be experiencing the behaviour you report.  The correct nesting of the code in the three If….End If constructs as above is crucial, so check your code carefully against the above.  If it tallies, copy all the values from the columns in one line, in which the value in the column is 'NA', and paste them into a reply here.  I'll then be able to test the code against your data.

                                                                                                                                                                   

ken sheridan say: The function should be exactly as I posted.  A line like:

     .Fields(fld.Name) = LTrim(RTrim(.Fields(fld.Name)))

for instance should be exactly that, not your:

     .Fields(1) = LTrim(RTrim(.Fields(1)))

The fld object variable returns a reference to the current field object in the recordset's Fields collection,  and its Name property returns the name of the field.

You should paste the function into a standard module in the database and save the module under a name which does not conflict with any other object, including the function itself, e.g. mdlUpdateTable.  In the form add an unbound text box named txtTableName, into which you should, at runtime, enter the name of the table you want to update.  The code for the button's Click event procedure would simply be:

    UpdateTable Me.txtTableName

With data imported from Excel, which I think is what you are validating, the code should cope with any of the fields.  It would probably fall over with weird field types like multi-valued fields, attachments etc.  If any such fields are in the table then you should exclude them from the update in the same way that I've excluded fields of long integer data type by examining the field's Type property, which I've done merely to cope with autonumber fields, which might well have been added as part of the import process.  Trying to update an autonumber would raise an error.

One final word.  It is of course imperative that you first back up any table you are updating.
