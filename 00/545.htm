ken sheridan say:

Hi, I am unsure about a shared front-end, multiple users can access this but it is only saved in one place.

It sounds like each user is opening a single copy of the front end file (your 'viewer') in a shared location on the system.  This is not a good modus operandi, and probably explains the behaviour which you are experiencing.

You should install a separate copy of the front end on each user's local machine, or in a 'personal' location on the system to which only the user has permissions.  Each copy of the front end should then be linked to the back end.

You might like to take a look at Refresh.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In the zip archive extract the RefreshLinksSingle_07.accdb file.  In the opening form of this little demo code in its Open event procedure checks the current links and if they are invalid, opens a dialogue form via which the user can browse to the back end file and refresh the links.  If neither the front end nor back end are moved this needs to be done only once.

You would need to copy the code into your front end's opening form's Open event procedure.  Note that this must not be a bound form or access any of the tables in the back end in any other way, though it can be a switchboard form or similar, using a table located in the front end, not the back end.  You would also need to copy the two modules from my file, one a standard module, the other a class module, into your front end, along with he frmUpdateLinks form.  Copies of the front end should then be distributed to the users.  The first time each user opens the front end they will be prompted to refresh the links via the frmUpdateLinks form.  Refreshing the links sets the Connect property of each linked table, so when the user subsequently opens their copy of the front end the code will not detect any invalid links and the database can be used as normal.

With what do you want to replace the text?

 

If you copy the text to be replaced to the clipboard and then run a macro containing the following code, it will ask you for the replacement text and then replace each instance of the text that you copied to the clipboard with the replacement text:

 

Dim MyData As DataObject
Dim strClip As String
Dim strReplacement As String
Set MyData = New DataObject
MyData.GetFromClipboard
strClip = MyData.GetText
strReplacement = InputBox("Enter the text to be used for the replacement", "Find and replace long text")
With ActiveDocument.range
    .Text = Replace(.Text, strClip, strReplacement)
End With

macropod say:

It's indeed a custom style, but it's defined with the correctly identified outline level (hence it shows in the navigation pane, or in the list of headings in cross-references).

The issue with the 'place in this document' hyperlinks is that they rely on the existence of bookmarks at the destinations. Word generates these automatically for its Heading Styles but not for your custom Styles. Your choices come down to using Word's Heading Styles an their automatic bookmarking, or using your own Styles and having to add your own bookmarks.
