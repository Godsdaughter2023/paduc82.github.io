The key thing to understand about a relational database is that, unlike a spreadsheet, it is a model of the reality in terms of its entity types, each represented by a table, the relationships between them, and the attributes of the entity types.  Relationship types can be one-to-one, one-to-many or many-to-many.  One-to-one and one-to-many relationship types are modelled by a foreign key column in one table referencing the primary key column of another table.  Many-to-many relationship types can be binary, ternary etc depending how many entity types are involved in the relationship type.  They are modelled by a table which resolves the relationship type into a set of one-to-many relationship types, e.g. an OrderDetails table models a binary relationship type and would have foreign keys OrderID and ProductID referencing the primary keys of Orders and Products tables.

Columns in a table represent the attributes of the entity type which the table is modelling.  The process of normalization requires that each non-key attribute must be determined solely by the key, e.g. in  a table of Addresses you might have a CityID column, but you should not have a StateID column as state is determined by city.  The Cities table would have a StateID column as a foreign key referencing the primary key of a States table.  Consequently we are told only once in which state a city is located, so there is no redundancy and inconsistent data cannot be inserted, as would be the case if the Addresses table had both CityID and StateID foreign keys.
