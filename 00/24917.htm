To the MS technician in charge of the Myanmar XAML Keyboard - Microsoft Community

ianrg say:

To the MS technician in charge of the Myanmar XAML Keyboard

Please note I have provided feedback on this in the Feedback hub - This is a primer for whomever is in charge of the Myanmar XAML touch keyboard.

Myanmaza is primarily a left to right script, however it has some complications. These complications have lead to a variety of solutions over the years, first for what order the characters were written, then typed on mechanical keyboards and, finally, in how different computer based fonts and font shaping engines handled the complications. Many of these solutions did not take into account the needs of different cultures and peoples within the Myanmarza family of scripts. This was all supposed to be fixed by adoption of Unicode. However, between those who had already learned older methods, those who adopted non-Unicode compliant fonts that used the Myanmar Unicode range (Esp. Zawgyi, a font based on stealing Arial from MS) and those who want to type in Unicode/Dictionary order vs those who want to type in Visual order, there has been ongoing problems.

The issue arises in that, despite being a left to right script, there are characters that are typed to the left, right, above, below, left and right, above and below of the preceding character plus one that encircles these. For these medial and rhyme characters, there are characters that change if there is another medial or rhyme character or depending on the shape of the primary head character.

In the feedback MS and everyone else receives, you will have seen that there are people who want a Unicode compliant font (Eg Type the primary head character first, then the medial or medials, then the Rhymes - vowel character or characters, finals (Nasals or Stops) and finally a tone marker if necessary. You will also see people that want to use Zawgyi type fonts which uses the Myanmar Unicode zone but with some characters on different code points and other cultural letters missing to allow the poor shaping in Zawgyi for specific characters to work properly. And still some who want old style ASCII based solutions.

This conflict has resulted in MS providing a Unicode order solution for Windows 10 PCs and different Visual order solution for Windows 10 Mobile / Windows Phone (before it died). Now with the move to XAML touch keyboards, MS appears to severely stuffed up a change from the internationally agreed Unicode order to a visual order. This is bad move, not just because it is a backwards step, but because it means that programs like Microsoft Word and Microsoft WordPad cannot be used using the new XAML keyboards. There is a simple solution:

Step 1 Provide a working XAML keyboard (ie unlike in 17035).

Step 2 Return the now lost ability to load third party keyboard layouts lost with the move to XAML.

Step 3 Provide a Unicode compliant Unicode order Keyboard and a Unicode compliant Visual Order Keyboard - This last point is what is supplied on the Bagan Keyboard on Android: it has a setting that switches between Unicode order or visual order.

Please contact me if you wish to discuss

ianrg say:

There are two ways to create a မေး /meí/

1.      The correct Unicode order is that the ေး - the /eí/ sound is typed and stored in its phonetic place (i.e. at the end of the syllable) rather than its visual place (i.e. to the left of the consonant (shown as a မ). This was decided in 1999 and Microsoft Windows Desktop OSK and touch keyboards have followed this for a long time.

2.      The deliberately rejected old visual method where the ေ is typed before the မ and ေ’s tone marker(shown as း) is typed after the မ. This is used by non-Unicode compliant fonts and programs.
