ronhatcher say:

Word erroneously reports the disk is full in the middle of a document. Then locks up?

john mcghie say:

Hi Ron:

It is a bug.  Actually, it seems to be several of them.  And the reason you got the run-around is that nobody (including us...) knows for sure what is causing it.

Late last year I was talking to the Senior Software Engineer who has been working on Mac Word for about the past 20 years.  He has spent a huge amount of time chasing this.  And he still hasn't found it.  The best we can say now is t hat there appears to be more than one cause, and it is due to undocumented "features" of the Apple file system.  But Apple can't find it either.

"Disk Full" is a very ancient error message, which dates back to the time when people worked on floppy disks.  What it means is "A write failed writing to disk". Back in the day, the most likely cause was that the disk actually was full.  Now, it is the least likely cause.

The cause appears to be a "deadlock".  One application is holding a file the other wants.  The other application is holding a file the first wants.  Each one is waiting for the other to let go.  They will both wait forever, until eventually OS X gets sick of them and returns "Write Failed".

Unfortunately, it does not tell Word WHICH file failed, or WHY.  While you are working, Word has around 20 different files open for each document in edit.  At least two of them are returning an error.

Since we don't know what's causing it, we can't tell you how to fix it.

I can tell you some things that will make it happen a lot less often (I have not had one in the past two years...):

1)  Apply the latest updates.  Every time they find one of the causes of this, they issue a patch.  Go to Help>Check for Updates and run it until it tells you you are fully up-to-date.

2)  Work on the local drive.  Working on external or network drives involves a more complex process that is more likely to throw up the errors. Use the Finder to copy the file back to the external drive after you have closed it.

3)  Every day when you finish work, log out. Leaving Word running with files open overnight is asking for trouble. (Leaving a computer connected to the network and unattended is inviting a drive-by shooting from the bad people out there...)

4)  At the end of the working week, shut down the computer.  Start fresh on Monday morning: this gives OS X a chance to run its house-keeping clean-up tasks.  If you leave the system running "continuously", eventually it clags up with various temporary files and throws this error a lot.

If you try to do those four things whenever you think of it, your mileage will greatly improve. As I say, I have not seen that error for at least two years.

And I wouldn't waste any more of your time with Product Support: they simply don't know the answer to this (or much else...).  They are Level 1 Phone Staff.  The people in here are Level 3 industry professionals who use Word in a professional capacity all day.

Hope this helps

bill manville say:

Wow!  I haven't seen anyone using DGET for about 20 years.

The function doesn't allow arrays for the criteria.

You could use =SUMIFS($C$11:$C$18,$A$11:$A$18,B27,$B$11:$B$18,C27)

which would give a wrong answer if there were duplicate rows.

Or, if you named the ranges for clarity using the table headings from A10:C10:
=SUMIFS(Rating,Group,B27,Subgroup,C27)


Or maybe

=SUMPRODUCT((Group=B27)*(Subgroup=C27)*Rating)

which has the same problem with duplicates

Or maybe

=INDEX(Rating,MATCH(B27&C27,Group&Subgroup,0))

entered as an array formula using Ctrl+Shift+Enter
This will not have the problem with duplicates - just returning the first rating

UDFs may be convenient, but INDEX+MATCH in array formulas can be arbitrarily complex.

=INDEX(Table,MATCH(TRUE,IF(condition1,IF(condition2,IF(condition3,condition4))),0),columnindex)

If you're going to use VBA, might as well use it to wrap a UDF around ADO in order to use SQL queries on worksheet tables.
