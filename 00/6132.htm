gregretro say:

Unfortunate that this wasn't answered sooner, and that the suffix list is not updated with this one (located at https://www.microsoft.com/security/portal/mmpc/shared/malwarenaming.aspx, bottom of the page).

The "plock" suffix is a generic cloud-based rule (and not a signature) that marks something as malicious based on data that has been accumulated from other machines participating in MAPS.

Essentially, the detection is of something almost definitely malicious based on data that the scanner pulls from the cloud, but the data is too new to have made it into a deployed anti-malware signature.

bob jones aka: cybertaz say:

Word for Mac hasn't supported Forced Justification for eons - possibly forever, but I honestly can't recall as I never have used it due to the readability considerations. The 'bottom line' is that the last line of text in a paragraph will not get 'stretched' when Justified is used. [The exception is that Distributed Alignment is available when using East Asian Language script.]

The reason you're seeing the effect is that when you insert a Column Break in the midst of a paragraph it automatically causes a paragraph break [¶] to be created at that spot. Accordingly, the line preceding it loses the Justification because it now is the last line of a paragraph.

To avoid that, create a Line Break [Shift+return] just before the newly added ¶. However, there are at least 2 considerations: One being the visual impact, the other being that it can be problematic for future revision, especially if another user editing the document is not aware of what has been done. Also, if attempting to balance the columns it can make it difficult to do so.

ken sheridan say:

You insert a new row into the plot table by assigning a value to the SiteID field.  Once you move off the parent form to the subform the new plot table record is saved and the linking mechanism (the LinkMasterFields and LinkChildFields properties of the subform control both being PlotID) automatically assigns the value to the PlotID foreign key field in the new row in the sample table without the need for any code.

I don't understand why the parent form only has a PlotID control.  It would be impossible to insert any meaningful record, as a record which contains only an arbitrary autonumber value contains no real data, merely a unique but otherwise meaningless identifier.  The only circumstance where I can see it might be necessary to do this would be if the SiteID field has a value set as its DefaultValue property in the table design.  Should this be the case you can force a new record to be added by assigning the default value of the SiteID to itself with:

Me.SiteID = Me.SiteID

which you could do in the DblClick event procedure of the SiteID, or any other control, on the form for instance.

The above would in fact work even if the SiteID field has no default value and accepts Nulls (i.e. is not Required), but as I said, the resulting record would be meaningless.

Ken Sheridan, Stafford, England

nadiasz say:

We have to clarify what you want to accomplish.

I thought that you want to be able to add a new record to the Plot table if you enter a record in Sample table with PlotID that was not found in Plot table.

Usually, the form and subfotm are linked on a certain field, and this field is not shown in the subform. When you add a new record to a subform - it picks up the value of the linked field from the parent form - that is how one to many relationship is implemented (you have one record in a Plot table and many records in a Sample table, connected by PlotID field). In this scenario, you don't need to add a value to the Plot table.

However, if you want to enter a new record in a subform that would have a value in the linked field that is not found in master table, then you have to enter a new record to master table first - so that the child can connect to the master on a linked field.

Nadia

ken sheridan say: Are there separate Cover, Density and Species tables for tree, shrub and ground vegetation sub-plots?  If so I can see no logical reason for having the Tree Plot Table in the model as it merely models relationships from Plot Table to the Cover, Density and Species tables.  The table can be dispensed with and PlotID columns in the Cover, Density and Species tables can reference the key of Plot Table directly.  Although you've used surrogate keys (CoverID etc) for these tables each would also have a composite candidate key (PlotID/CoverClass etc).

If in the other the Cover, Density and Species tables each relate to tree, shrub and ground vegetation plot tables by means of separate foreign key columns, this would not be a good design as (a) the attribute types would presumably differ between each, so there would be considerable redundancy, and more fundamentally, (b) a more correct way to model this would be to have a single column PlotType with possible values 'Tree', 'Shrub', 'Ground' say.  The keys would then be composite ones such as PlotID/PlotType/CoverClass etc.

My guess would be that of the two alternative models above the first is the more likely to have been used in view of the different attribute types involved.
Ken Sheridan, Stafford, England

If you really need the tab in the text to which the Heading 1 style is applied, all that you can do is select the STYLEREF field in the header and then use CTRL+SHIFT+F9 to unlink it from its source, which will convert it to ordinary text from which you can then delete the tab.  That does however destroy the functionality of the STYLEREF field and you will then see the same result for it on all pages, rather than it updating dynamically to display the text to which the Heading 1 style is assigned on the various pages of your document.

Do you really need the tab in the text to which the Heading 1 style is applied.
