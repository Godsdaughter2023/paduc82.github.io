FIXED - Ok the code below will need to be implemente as your HoverControl Subroutine, I have highlighted my additions... The reason for the borders not updateing would appear to be due to the repaint not occuring, so it needs to be forced. So that the form is not continuously repainted I added additional checking to ensure it only repaints once per label change.

I have also added an extra check to exclude other controls...

Sub HoverControl(PassCtrlB As Control)
    Dim CtrlB As Control
    For Each CtrlB In Me.Controls
        If PassCtrlB.Name = CtrlB.Name  And TypeName(CtrlB) = "Label" Then
            If CtrlB.BorderColor = &HFFFFFF Then Me.Tag = "Y"
            CtrlB.BorderColor = &H808080        'Grey border
        ElseIf TypeName(CtrlB) = "Label" Then
            CtrlB.BorderColor = &HFFFFFF         'White border
        End If
    Next CtrlB
    If Me.Tag = "Y" Then Me.Repaint
    Me.Tag = ""
End Sub

I would also suggest you add the additional control check to your ClickControl routine

Sub ClickControl(PassCtrlA As Control)
    Dim CtrlA As Control
    For Each CtrlA In Me.Controls
        If PassCtrlA.Name = CtrlA.Name  And TypeName(CtrlA) = "Label"  Then
            CtrlA.BackColor = &HFF&               'Red fill
        ElseIf TypeName(CtrlA) = "Label" Then
            CtrlA.BackColor = &HFFFFFF            'White fill
        End If
    Next CtrlA
End Sub

bob jones aka: cybertaz say:

What you're experiencing isn't a 'bug' :-) It's just that the Preview is not the merge. Previewing the data is done in your single page main label document. When you advance from one record to the next the records simply shift up one label cell bringing the next record into the lower right cell on that one page. The intent is merely to see how each record fits based on variable field lengths. IOW, you are not seeing separate individual pages of labels.

It isn't until you actually perform Step #6 that the necessary number of pages are generated to accommodate the total number of records. That step gives you the option of merging directly to the printer or to generate the merge as a separate new document. If you do the latter that document will contain the requisite number of pages. You can then print that document or return to the main label document & repeat the merge to the printer.

 

Regards,

Bob J.

christopher harwood say: I ran the large mail merge to a new document last night, and the resulting document was correct this morning.  It looks like this is just a bug, or unforeseen odd result, with Preview Results.  Viewing each record in turn gives the appearance of page changes, but the actual merge does not create a new page for each record.  For each record, preview shows how that particular record will appear in the upper leftmost label and does not show where that record will be printed on a page.  The preview then uses the records following that one to fill the page, giving the impression when moving between records that each new record is causing Word to create a new page.

So, the way preview handles multi-label pages is a bit annoying, but the merge itself is correct.
