ossiemac say:
  • Set up the tables with the last name and first name like the screen snippet below. (Columns E and F are only to display the formula that you are to use in columns C and D)
  • Column C has a formula to concatenate the Last Name, Space and First Name (See formula column E)
  • Column D has a formula to concatenate the First Name, Space and Last Name (See formula column F)
  • Use the list in Column C for the Last Name list in the Data Validation.
  • Use the list in Column D for the First Name list in column in the Data validation.
  • Both Data Validation DropDowns will have Last Name and First Name but the one list has Last name first and the second has First Name first in the list. This virtually allows you to select by the Last or First name even though both are displayed in the DropDown
  • When setting the data validation, select the "Error alert" tab and uncheck the box against "show error alert after invalid data is entered"

Copy the VBA code below into the worksheets module as follows.

  • Right click the worksheet tab name
  • Select "View code" (will open the VBA editor window at worksheet's module)
  • Copy the VBA code below and paste into the VBA editor.
  • Do not rename the sub
  • Close the VBA editor (Cross very top right of VBA editor window)
  • Save the workbook as Macro enabled.
  • Ensure macros are enabled. See help for how to do this. (Option to "Disable all macros with notification" should be OK.)
  • When a selection is made in either of the DropDowns, the VBA code splits the selection and inserts the Last Name and First Name in the appropriate columns.

Screen Snippet of the tables for the Data Validation.

Copy all of the following code into the VBA editor as per guidelines above.

Private Sub Worksheet_Change(ByVal Target As Range)
    Dim arrSplit As Variant
   
    On Error GoTo ReEnableEvents
    Application.EnableEvents = False
   
    If Target.Cells.Count = 1 Then  'If Only one cell changed
        If Target.Column = 1 Then   'If change is in Column A
            arrSplit = Split(Target.Value)
            Target.Value = arrSplit(0)
            Target.Offset(0, 1) = arrSplit(1)
        End If
       
        If Target.Column = 2 Then   'If change is in column B
            arrSplit = Split(Target.Value)
            Target.Value = arrSplit(0)
            Target.Offset(0, -1) = arrSplit(1)
        End If
    End If
   
ReEnableEvents:
    If Err.Number <> 0 Then
        MsgBox "An error occurred in module " & Me.Name & " Private Sub Worksheet_Change"
    End If
   
    Application.EnableEvents = True
   
End Sub

daniel pineault say:

Set Column Count 1 (don't want to see column 2)

This does not control the visibility of the columns.  As the title suggests, it is the number of data columns.  In your case it should be 2.

The visibility is controlled by defining the column widths.  If you want to hide the 2nd column, set it to 1";0" (replace 1" by whatever dimension you would like).

scottgem say:

That property controls the width of the list when you pull it down. For example, a combobox only shows the first non-zero width column after a selection is made. But you might want to show additional columns to help the user select the proper record (usually when you use a table as the Rowsource). So you may need the list width to be wider than the control.

And, by the way, a Value List should only be used if you have a short, static list of items. otherwise it is better design to use a table.

ken sheridan say:

...........what does List Width Auto do?

For examples of how the various properties of a combo box control are used you might like to take a look at Prescriptions.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file, if go to the patients form and open it in design view, you'll see that both the parent form and its prescriptions subform include a number of combo boxes.  Take a look at the 'Go to patient' combo box first.  Its RowSource property is:

SELECT PatientID, FirstName & " " & LastName, 1 As SortColumn, LastName,Firstname  
FROM Patients  
UNION
SELECT 0, "<New Patient>", 0,"",""  
FROM Patients ORDER BY SortColumn, LastName, FirstName;

This illustrates a number of things.  Firstly the BoundColumn is the first column PatientID.  This is hidden by setting the first dimension of the control's ColumnWidths property to zero, leaving the concatenated name as the visible column.  It is usual to hide the first column in this way, rather than allowing a second column to be invisible in the way that you have done.  Note also how the <New Patient> option is introduced into the list by means of a UNION operation, and shown first in the list by means of the SortColumn column whose values are the constants 1 and 0.  The ListWidth property in this case is the default of 'auto', which sizes the list automatically to the width of the control

If you take a look next at the 'Strength' combo box in the subform, this control's RowSource returns two columns, the strength and the units in which the strength is expressed.  Both columns are shown in the list by setting the control's ColumnWidths property to 2cm;1.501cm (Access is sometimes reluctant to set a value like 1.5cm!).  In this case the ListWidth property is 3.501cm, i.e. the sum of the ColumnWidths dimensions.  This makes the list the precise width to accommodate both columns.  By having both columns listed the drug strength and units are selected simultaneously, meaning that only a valid combination of strength and unit for the selected drug can be selected, which is rather important from the patient's point of view!  The unit value is assigned to the read-only init control by the following code in the strength combo box's AfterUpdate event procedure:

Private Sub cboStrength_AfterUpdate()

    On Error GoTo Err_Handler

    ' assign selected unit to txtUnit control
    Me.txtUnit = Me.cboStrength.Column(1)
    ' requery cboDrugForm control to show forms
    ' for selected drug/strength/unit
    With Me.cboDrugForm
        .Requery
        .SetFocus
        .Dropdown
    End With

Exit_Here:
    Exit Sub

Err_Handler:
    MsgBox Err.Description & " (" & Err.Number & ") ", vbExclamation, "Error"
    Resume Exit_Here
    
End Sub

You'll notice that the code also requeries the drug form combo box.  This is because this combo box is correlated with the strength combo box so that it only list the forms, e.g. tablet, capsule etc in which the drug is available in the selected strength.  This done by the control's RowSource referencing the drug name, strength and unit controls as parameters:

SELECT DrugFormStrengths.DrugForm
FROM DrugFormStrengths
WHERE DrugFormStrengths.DrugName=Form!txtDrugName
     AND DrugFormStrengths.Strength=Form!cboStrength
     AND DrugFormStrengths.Unit)=Form!txtUnit
ORDER BY DrugFormStrengths.DrugForm;

You'll have noticed that all of these controls use SQL statements (queries) as their RowSource property, never a value list.  A value list should only be used where the set of values is immutably fixed in the external world, e.g. the 7 days of the week or 12 months of the year.  Otherwise the values are data, and data should never be encoded in an object definition as a value list.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.
