ken sheridan say: Your data_table has, in addition to its primary key of data_id, a composite candidate key of item and location.  Consequently, to identify which locations each order is sourced from your order_table will need to be decomposed along the following lines:

Orders
….orderID  (PK)
….orderDate

and a related table either as:

OrderDetails
….orderID  (FK)
….data_id  (FK)
….quantity

or as:

OrderDetails
….orderID  (FK)
….item  (FK)
….location  (FK)
….quantity

In both cases the primary key is a composite one of the two or three foreign keys.  To automatically determine the quantity value in either case an unbound TotalQuantity control could be included in an OrderDetails subform.  The user would then enter a value into this, and then execute some code, e.g. by means of a command button which would determine if the quantity can be sourced from whichever location currently contains the minimum non-zero quantity of the item in question.  If so, the TotalQuantity value would be inserted into the Quantity control; if not, the current quantity in the location in question would be inserted into the Quantity control, and the balance computed.  In the latter case the code would then insert a new record into the subform and repeat the operation with the new balance.  This process would be repeated as necessary until the total quantity has been sourced.  Such a routine would not be difficult for an experienced developer, but if you are not comfortable with writing the VBA to do so, hiring professional assistance would be a sensible business decision.

It is then a simple task to return your result table with a query.  There is no need to append data to a base table, and more importantly, you should not do so as it would introduce redundancy and the risk of update anomalies.

If you do wish to record the quantity on hand per item/location in the data-table then you will need to update the qty value for the relevant row by executing an UPATE query in the AfterInsert event procedure of the OrderDetails subform (embedded in an Orders form).  However, due to the redundancy of storing the quantity on hand values, the table would still be open to the risk of update anomalies.  The safer method is to compute the quantity on hand per item/location, which in theory is easily done by subtracting the SUM of all movements out of stock per item/location from the SUM of all movements into stock per item/location.  In real life, it's not quite so simple, however, as this does not take account of unrecorded stock movements (pilfering)  or data entry errors, so account needs to be taken of stock takes.

You'll find an example in Inventory.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file, does not include any provision for recording stock locations by bin, rack etc, so would require amendment to provide a model for your purposes.
