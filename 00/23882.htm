So you tried upgrading to Windows 10 final with the ISO -- not activated.  (I have upgraded from 10162 to 10240 using the ISO and remained activated).  You then rolled back to 10162 -- it is activated again.

http://windows.microsoft.com/en-us/windows-10/media-creation-tool-install#

In order to best duplicate the Windows Update approach I always do this with the ISO:

  • Ensure that I have the correct ISO to match the version I am upgrading (Home, Pro, Single Language, etc.)
  • Run setup.exe from the ISO from within an activated Windows 10 Insider Preview, Windows 8.1, 8 or 7
  • Always choose to Keep Windows settings and files (otherwise I have seen the "upgrade" display "Installing Windows" instead of "Upgrading Windows" as it is processing).
  • Never enter any Product Key during or after the upgrade.  Choose Skip or "Do this later" on these screens.
  • Stay connected to the Internet during the upgrade, especially at the end.  Login properly to the wireless network if necessary to connect.  However on the one I did not do this, it was not activated after install, but activated instantly as soon as an Internet connection was made, without having to choose Activate Windows.

Choosing "Upgrade this PC now" from the Media Creation Tool eliminates the potential of doing one of the above steps wrong.  For example, many people will at least attempt entering some Product Key when asked to enter one, which might prevent activation of the free upgrade.

scottgem say:
Again, without knowing your data model and structure its hard to answer. But I can tell you that using a heavily horizontal structure is not good design. Let me give you an example. Lets take a database with info on resorts. So you have a table that lists information about the resort (i.e. Address, phone # etc.). This would be like your primary report. Next you want to list the amenities offered by the resort. One way to do this would be to have a table with fields that list each amenity. This would be a horizontal structure and would not be good design. The better way is with a table like:

tblResortAmentities
ResortAmentiyID (PK Autonumber)
ResortID (FK)
AmnityID (FK)
OtherInfo

This is a vertical structure that uses one record to describe each amenity in each resort. This is a normalized structure.

Now this could be applied to your Diseases table like so:

tblReportDisease
ReportDiseaseID (PK Autonumber)
ReportID (FK)
DiseaseID (FK)
DiseaseAttributeID (FK)
OtherInfo

So you would have ONE table for diseases (a similar table for workflow). Each record in this table would be tied to the Primary Report record, a specific Disease and an attribute of that disease that you want to track. The Disease and DiseaseAttributes would be chose from lookup tables. This eliminates the need for disease specific table it won't matter how many attributes you need to track per disease.

There are two factors to syncing times:

1. Yes, the sync is somewhat slow by design. I remember a OneNote engineer having mentioned something about 250 KB/s max wit the current protocol. I know they are working on the sync protocol so hopefully that will improve soon.
2. Given the above speed statement and the fact that many users do not use OneNote as it most likely was desiged to (as a note taking tool) but more as a document management system, printing hundreds of PDF pages into notebooks (where they get PNG images), there can be a large amount of data in a notebook.

OneNote also originally was designeds for PCs, not for mobile devices with maybe a limited storage capacity (local storage is used by the cache copies for open notebooks).
