trevor rabey say:

This is not a problem or an issue. It is how it works.

You have correctly described what happens, but have not said what you expect otherwise.

% complete is actual duration/total duration.

Total duration is actual duration + remaining duration.

If you type in a % complete for a task which does not yet have an actual start, MSP must have a date/time to assign as the actual start, so that the actual duration can start somewhere..

The only date/time that it has is the scheduled or planned start, so it uses that, and it makes no difference if the scheduled or planned start is in the past or the future. You can change it if the actual start date/time was something else (tracking table).

You seem to be wanting MSP to decide that the remaining duration should be changed, or that the actual start should be something other than the scheduled or planned start, based only on the % complete. How can MSP possibly know that? How can it know what the actual start date/time was, if you don't type it into the actual start field in the tracking table?

There is no need t type in a % complete, and also doing it doesn't work. It is best to type in the actual start date/time and the actual duration (both must be in the past because they are "actual", and re-estimate the remaining duration. Let MSP calculate the % complete.

dirk goldgar say:

If your field is an actual AutoNumber field, data type "AutoNumber" in the table's design view, then there is no way to store a text value such as "NJ10401" in the field.  Access will generate a new *long integer* value for each new record, and that is what will be stored in the field, no matter how you format it for display. 

Also, if you have specified the Format property of the field to "NJ1"0\400" -- or "NJ104"00 -- in the table, that doesn't necessarily carry over to a form that displays this field.  Access will usually copy the Format property from the table design to a new form created from the table, but for pre-existing forms, or if it doesn't copy the property, you have to set that format on the text box that is bound to the field.  So if you want to keep the autonumber but use this format to display it, you will have to copy the Format property to any existing form that uses this field.

An alternative is not to use an autonumber at all, but rather to make the field a text field and use a function to generate a new value for the field, in the desired format, for each new record you create.

scottgem say:
First, an Autonumber has one purpose and one purpose only, to provide a unique identifier to a record. There are many developer who hold that an autonumber PK should not even be visible to the user. Personally I don't see that, it can be used as a visible ID as long as you understand that it won't necessarily be sequential. 

But the autonumber will still be stored in the database as a long integer. And any foreign keys will have to match what is stored. 

You can force the Autonumber to start at a specific value if you want. But you can only add text for display.
