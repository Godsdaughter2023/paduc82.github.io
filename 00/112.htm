Glenn Glidden Replied

This has worked for me (usual backup warnings apply);

1) Rename .docx file to .zip

2) Extract the files into it's own folder.

3) Look in the word\_rels folder for the file document.xml.rels


4) Open this file in notepad or your basic text editor of choice and search for your rID string.

5) In my case, the target bit in the example below was set to NULL

<Relationship Id="rId39" Type="http://schemas.openxmlformats.org/officeDocument/2006/relationships/image" Target="media/image30.png"/>

6) To fix, put in a filename that exists in the media folder.


7) Save the file, ensuring the filename is the same as before (make sure notepad or your editor of choice doesn't mangle the filename by adding .txt to the end).


8) Zip up all the files and folders back into one file again.


9) Rename your .zip file back to .docx, open in Word and you should be sorted.


Note that one of your images in the document will be wrong, but unfortunately you can't tell which one, so a bit of proofreading is needed.

Hope that helps.  Had about four different incidences where I work, all due to a large number of images being placed in the docx file.

jamesdemers say:

In my case, the lost images are nowhere to be found.  I did some testing and found that the .emf files were never written to the media folder, even when I "saved" the document.  The images are there on the monitor, so you THINK you just saved them successfully, but in fact they don't exist except on the screen and, I suppose, in the video RAM.  There's no hint of a problem until you close the document, and you don't discover that the graphics are missing until you next open it.  I would bet that exhaustion of the video RAM with large documents causes people to notice the issue sooner.

The best solution I've come up with is this: Do not simply "paste" graphics into a docx or pptx document, but "paste special".  That gives you a chance to specify the format, and selecting png creates .png files in the same media folder where the .emf files (default for cutting and pasting via the clipboard) are refusing for some reason to make an appearance. 

The problem, in my case, only happens with graphics that would generate large (>100k) emf files.  Smaller ones seem to be created without a problem.

All of this is with Word for Mac 2016; ymmv.

scottgem say:
I just tested this with two forms. Even though the button was set to Display Always, they did not print. 

This isn't a bug but the way Access was designed. There are some properties that are common to Forms and Reports but are ignored in one or the other. Another example is Can Grow/Shrink. Even the property can be set on a form, its ignored. That's because forms allow scroll bars on text boxes, Reports don't. Forms may appear to have lots of print options but that don't really. 

Clearly, there is a difference in the design and usage of forms and reports.  I'm not denying that sometimes it can be useful to just print out a form, but there are very good reasons why its not a good idea.

ken sheridan say: In my ReportDialogue demo the first method cancels the opening of the report and opens the dialogue form if it is not already open, passing the name of the report to the  dialogue form via the OpenArgs mechanism with:

Private Sub Report_Open(Cancel As Integer)

    Const FORMNOTOPEN = 2450
    Dim frm As Form
    
    On Error Resume Next
    Set frm = Forms!frmMultiFieldDlg_1
    If Err = FORMNOTOPEN Then
        DoCmd.OpenForm "frmMultiFieldDlg_1", OpenArgs:=Me.Name
        Cancel = True
    Else
        If Err <> 0 Then
            ' unknown error
            MsgBox Err.Description, vbExclamation, "Error"
        End If
    End If
    
End Sub

The code in behind the button in the dialogue form is:

Private Sub cmdOpenReport_Click()

   Const REPORTCANCELLED = 2501
    
    On Error Resume Next
    DoCmd.OpenReport Me.OpenArgs, acViewPreview
    Select Case Err.Number
        Case 0
        ' no error so do nothing
        Case REPORTCANCELLED
        ' anticipated error so do nothing
        Case Else
        'unknown error so inform user
        MsgBox Err.Description
    End Select
    
End Sub

In this the value passed into the form as its OpenArgs property is used as the ReportName argument when the OpenReport method is called.  

From the error which you've incurred suggests to me that the dialogue form has been opened independently, rather than as a result of opening the report.  In my demo the report is opened from the button on the opening form with the following code:

Private Sub cmdMethod1_Click()

    On Error Resume Next
    Me.Visible = False
    DoCmd.OpenReport "rptEmployees_MultiFieldParam_1", View:=acViewPreview
    
End Sub

This hides the opening form (not important from your point of view), and then attempts to open the report.  As the dialogue form is not open at this stage the opening of the report is cancelled and the dialogue form opened.  When the opening of the report is cancelled this will raise an error on the DoCmd.OpenReport…. line above, but the error is ignored by virtue of the, admittedly rather crude, error handling.  

The premise behind this demo is that the report is opened directly, not the dialogue form.  The latter is opened only by the report, so if an attempt is made to do so independently the error which you've experienced will occur.  So if this method is used, your application's user interface must be designed so that the report is opened directly.  If you want to pursue the alternative of opening the dialogue form first, then there are examples of that in my MultiSelect demo in the same OneDrive folder.  These use a similar dialogue form, but the methodology behind the form differs.

ken sheridan say: The code in the Click event of the button on frmMain should simply open the report, but you can also ensure the dialogue form is closed if it has inadvertently been left open. This shouldn't happen, but it's as well to play safe:

    On Error Resume Next
    DoCmd.Close acForm, "frmReport"
    DoCmd.OpenReport "YourReportNameGoesHere", View:=acViewPreview

The opening of the report will actually be cancelled as the frmReport form is not yet open.

In the report's Open event procedure you should include code like that in that which I included in my last reply, changing the name of the form in the two relevant lines to frmReport.

In the report's Close event procedure close the dialogue form:

Private Sub Report_Close()

    DoCmd.Close acForm, "frmReport"

In the Click event of the button on frmReport open the report with:

    DoCmd.OpenReport "YourReportNameGoesHere", View:=acViewPreview

WHERE (tblAgency=Forms![frmReport]!cboAffiliationAgreementType

       Important When creating XML Maps and exporting data in Excel to XML data files, there's a limit on the number of rows that can be exported. Export to XML from Excel will save a maximum of 65,536 rows. If your file has more than 65,536 rows, Excel will export the first (#ROWS mod 65537) rows.For example, if your worksheet has 70,000 rows, Excel will export 70000 mod 65537 = 4464 rows. Our recommendation is to 1) use xlsx instead or 2) save the file as XML Spreadsheet 2003 (.xml), which will lose the mappings, or 3) delete all rows after 65,536 and then export again which will keep the mappings but lose the data at the end of the file.

    1) xls or xlsx

    everything is okay.

    2) xlsx

     if excel file has 65537 rows (headers in row 1 and 65536 entries)
