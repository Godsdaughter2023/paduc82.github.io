rick rothstein, mvp - excel say:

VB uses quote marks to delineate the beginning and end of strings of text, so if you just put a single quote mark inside, VB can't tell if you mean an internal quote mark or an end/beginning of a new text string... so it issues an error. VB provides a mechanism to allow you to include internal quote marks though... just use two quote marks for each internal one. Example...

If your quote text is this...   My name is "Rick"

then double the quote marks....

My name is ""Rick""

and then put the surrounding quote marks around it to tell VB is is a text string...

"My name is ""Rick"""

This applies to quoted text constants... once the text is in a variable, you do not have to do anything extra afterwards... VB knows what to do with the text contained in a variable once it is in that variable.

albert d. kallal say:

It now appears that the databases will be moving to a new environment (due to a merger of two companies) on which they run an even older version of Office: XP. Which leads me to 2 more questions:

1. Are Access PIA files available for Office XP?

2. Is there a difference in formats and accessability through PIA between Access XP/2000 and 2003?

I don't think you need to bother with the .net office interop assemblies at all. In fact I would avoid this whole issue.

 

Remember, ms-access and those PIO's are for automaton and using ms-access. However, you don't need to use or bother with ms-access at all here.

 

Let me explain where I going with this:

 

Ms access is comprised of two parts, there's the whole big application part with the forms and reports and VBA code. It is for using this part of the application that those PIO's are designed for.

 

Then there is the data engine or the part that reads those data files. This is what we call the JET data engine. JET is what reads the data from those mdb files and it turns out that every version of windows since 98 SE has shipped with that jet database engine.

 

In other words even computers without having installed any version of ms access can read and open and manipulate data in those Access files.  In fact on a pure virgin clean install of an OS, you can write some windows scripts that'll open and read data out of that file.

 

In a nutshell this means that you don't want to bother with ms-access. In fact you're not really trying to bother with things like forms + reports + VBA code and all of the aspects that make up an access application anyway. If you use the inter-op stuff, then you are launching those mdb as "applications" and that even means the startup code in the mdb file will run when you do this. 

 

All you care about is reading data out of the tables in the mdb files.  The other advantage of not worrying about ms-access means that the file formats from access 2000 to 2003 or are readable by the same data engine and again you don't have to worry about versioning of the office or even bothering that you have the correct PIO assemblies.

 

Here's an example windows script that'll open and read an access database file:

 

Set dbeng = CreateObject("DAO.DBEngine.36")

strMdbFile = "C:\Documents and Settings\Albert\My Documents\Access\ScriptExample\MultiSelect.mdb"

 

Set db = dbeng.OpenDatabase(strMdbFile)

strQuery = "select * from contacts"

Set rs = db.OpenRecordset(strQuery)

rs.movefirst

If rs.EOF = true Then

   quit

End If

 

strTextOut = "C:\t5.txt"

set fs = Wscript.CreateObject("Scripting.FileSystemObject")

Set ts = fs.OpenTextFile(strTextOut, 2, True)

 2 = write, 1 = read

 

do while rs.EOF = false

   strOutText = rs("LastName")

   ts.Writeline strOutText

   rs.movenext

loop

ts.close

rs.close

 

The above uses a simple sql statement, and then writes out the last name collum out to a text file. However, you can see the idea here. 

I can't think of any particular advantage of trying to open and manipulate ms-access as a  automation object using the Interop assemblies as compared to simply using the JET data engine directly. If you have some familiarity you can even use the ADO object model if that is your cup of tea.  

As mentioned you can thus open and read mdb files without installing any additional software. And, you will be able to read any of the access database formats from 2000 to 2003, and in fact I think the above would even automatically open and read access 97 files.

 

So you don't need ms-access here, you wanna play with and use the jet engine, as mentioned if you're more familiar or comfortable using ADO, then all means use that.

Note however there is not a 64 bit verison of JET, so you have to stick to 32 bit running here.

 

Keep in mind for access 2007, and now Access 20 10<sup>th</sup>, you will have to install a new version of the jet engine, or what is now referred to as the ACE data engine. 

However once again that data engine can be separately downloaded and installed on your computer, and again you don't have to bother with installing or setting up ms-access on the computer
