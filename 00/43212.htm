There are at least two possible causes:
 
Cause 1:    You entered your password incorrectly or your Internet service provider (ISP) (ISP: A business that provides access to the Internet for such things as electronic mail, chat rooms, or use of the World Wide Web. Some ISPs are multinational, offering access in many locations, while others are limited to a specific region.) changed your password.
Cause 2:   You saved your password but the registry contains incorrect information for the Protected Storage System Provider subkey for the user account.
Resolution
Cause 1:    Microsoft Outlook needs to be updated with your new password. Microsoft cannot retrieve your old password. Contact your ISP, and ask them to assign a new password to your account. After you get your new password, you must update your e-mail account settings.
 
Cause 2:   The Microsoft Knowledge Base article 290684 is available to help you correct this behavior by stepping you through the process of removing your user account information from the Protected Storage System Provider subkey in the Windows Registry and entering your password options again in Outlook.

tryx3 say:

"There's no need for me to insert a pause in my script" - Yes, I understand [and should have done before].

I have just tested a network operation for the first time.  It hangs forever just like yours does even after I've closed Outlook & waited for the W:30 I set.  {I also ran it with Outlook closed the first time just to ensure that the network itself had no problems}

I read that link you provided.  So it seems that there is no solution.

- - - - - - -

You seem to be doing a much larger & probably more structured operation than mine so this workaround might not be of any use to you.  This other open files issue concerns a simple hourly backup of my core files that includes pst files.

1  I xcopy all the files to an internal folder InterimBackup first.  Then I use RoboCopy or XCopy to send them from InterimBackup to the real destination.

2  If Outlook is open then it does not copy to InterimBackup and, if the previous Outlook.pst is still in InterimBackup it gets deleted by the xcopy attempt.  [conclusion - XCopy checks the existence of a file to be overwritten, deletes it and only then discovers that the source file is locked and cannot be copied.] 

So I precede this with

0  I run [if exist] copy D:\InterimBackup\Outlook.pst to D:\InterimBackup\Outlook2.pst

The result is that the whole process always completes but includes only the latest Outlook.pst that has been available.  So I always have to make a point of closing Outlook at some stage during the day to allow it to take part in this backup.

If you also tried this as a two part operation, your xcopy / robocopy to the InterimBackup folder would always succeed but would only include Outlook.pst if you had closed the application first.  The subsequent RoboCopy operation would also always succeed with the same possible omission.

The downside of this might not suit your overall aim though.  You would have to separately RoboCopy all your other folder trees but keep the Outlook.pst in a tree that was not included.  You could do this either by restructuring your data or by using RoboCopy's exclude switches for the Outlook.pst file's folder.

You would have to accept that you could not be sure about the last time the Outlook file had been included unless you had deliberately ran the RoboCopy after closing it.

I have considered improving this by using Task scheduler to react to the Outlook closing event and use that to initiate copying outlook.pst to InterimBackup and then running that portion of the main RoboCopy operation for that folder.  But my needs are so simple that I've never felt it worth the studying of event logs that I'd have to do in order to set this up.

As I say, this might not suit your needs.
