Christine wrote:

Why does my Formula Bar still show 12:09:40 AM?  I have it formatted as [m]:ss.  It's just confusing that it shows up as a time of day in the formula bar. 

It can be more than just confusing.  If we enter fractional seconds (e.g. 12:34.567), we cannot edit the fractional part because it is not displayed in the Formula Bar.  (Aarrgghh!!)

Anyway, to answer your question:  AFAIK, we cannot control the format of the Formula Bar; only the format of the cell after we exit edit mode.  So it is what it is.  Of course, 12:09:40 AM is the 24-hour time 0:09:40, which can be interpreted as the duration 9 minutes 40 seconds.

Christine wrote:

Also, my average formula at the bottom of the column is not calculating correctly.  It was when I used h:mm (which is why I used that.)  My average for the column is now showing as .10 instead of 10.22 (it's cutting off the .22)  My formula is =AVERAGE(I2:I31).

I assume that I2:I31 are entered in the form m:ss.0 or 0:mm:ss .  It does not matter how they formatted, as long as it is not Text.

One way to format the average is:  Custom [m]:ss.000 .  I explained the use of "[m]" in my previous response.

Alternatively, if you want decimal minutes, the average formula might be:

=1440*AVERAGE(I2:I31)

formatted as Number with 1 or more decimal places.

It might help to know that Excel time is stored as a fraction of a day.  So 12:34.567 (12 minutes 34.567 seconds) is stored as 0.00873341435185185 (about 0.0087 days).

I'm not sure how you are ever seeing 0.10 and why you would expect to see 10.22 instead.  I think you omitted some details of the details, such as what amount of time you are really talking about.

(BTW, 0.10 might still be 0.1022, if you change the cell format to display 4 decimal places.)

Oh, I wonder if your average formula is still =24*AVERAGE(I2:I31), using 24 because the time used to be hours and minutes.  Then an average of 10 "hours" 13 "minutes" (intended to be minutes and seconds) would display as 10.22 "hours" (intended to minutes).  But then, an average of 10 minutes 13 seconds would display as 0.17 (really 0.170333333333333), not 0.10.

ken sheridan say:

Updating a front end is merely a matter of replacing each user's current copy with the amended copy of the file.  

Updating a back-end without losing the current data requires more work, however.  Alternative approaches are:

A.

1.  Copy the new back end, empty of all data to the user's location under a new name from the current back end file by including a version number in the file name.

2.  Import all data from the current back end into the new file.  To avoid violating referential integrity data must first be imported into referenced tables before it can be imported into the relevant referencing tables.

3.  Re-link the front end to the new back end.

B.

1.  Amend the table definitions in the existing back end by executing the relevant DDL statement, i.e. the ALTER TABLE statement to amend the definition of an existing table, or the CREATE TABLE statement to add a new table to the back end.  If new relationships are required as a result of the amendments the referenced table in each relationship must be altered or created before the referencing table in the relationship is created.

2.  For details of the DDL statements see:  https://msdn.microsoft.com/en-us/library/office/dn125729.aspx

However, you can avoid any of this if you adopt the normal strategy of having a single back end in a shared location on the system.  Where individual users have 'ownership' of specific data each user's access can be limited to that data by including their user name in the data and controlling access via queries restricted on the user name.  You'll find examples of this sort of 'user level security'  in DecDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file controls access to data by including each user in a group, but it can of course be easily adapted to control access on the basis of the individual user.
