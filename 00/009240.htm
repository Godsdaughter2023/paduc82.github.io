Unfortunately, there is no foolproof way to get a TOC to break where you want it to. I create many documents with multipage TOCs, and this is always an annoying issue. Sharing word documents with embedded macros. What permissions are - Microsoft Community

keeleymj say:

Sharing word documents with embedded macros. What permissions are required (read, Read / Execute)

User has a shared folder with word docs with embedded macros. What NTFS permissions are required?
jay freedman say:
As far as NTFS permission is concerned, Read is sufficient -- you just have to be able to open the document in Word.

To be able to execute the macros, you have to satisfy Word's macro security, which is separate from NTFS. Go to Office button > Word Options > Trust Center > Trust Center Settings > Macro Settings and select the item "Disable all macros except digitally signed macros".

If the macros inside these documents are signed with a digital certificate, that should be sufficient. (Each user who opens the document and wants to use the macros will have to accept the certificate the first time.)

Otherwise, on each user's computer, go to the Trusted Locations tab of the Trust Center dialog. Check the box for "Allow trusted locations on my network" and add the shared folder as a trusted location.

jllatham say:

I'll be blunt here - you need to give strong consideration to switching to a relational database such as Microsoft Access or SQL Server if you're going to have files used like this.

It is generally thought of as a "old adage" that it is not a matter of IF a shared workbook will become corrupted, but rather just a matter of when it will happen.  I can think of blunders that Microsoft has made over the years in Office tools - allowing shared workbooks without getting them to truly be stable was one of the larger ones.

I'd recommend that you find an empty folder somewhere with lots of space on the drive and start making regular (daily) backups of these shared Excel files so as to preserve as much of your data as possible.  Don't just overwrite a single backup file over and over, keep several days worth of backups so that if the file(s) become completely unusable you don't find that your backup copy is equally useless.

With a relational database application, you'd have a single backend storage file and each user would have a copy of the front end.  Multiple users creating new records isn't a new issue because you aren't ever working on "the same row" - its more like you are always working on the first empty row below the last used row even as others are entering data into "new rows".

You can even set up your business rules to validate data entry as each person tries to add new records to the database, or edit older records (and you can even control who's permitted to edit or delete old records if you want to).

jllatham say:

Sorry ... but on the bright side, your current Excel workbook gives you the table structure you need, I mean for this you would probably not even need to get real fancy and 'normalize' the data into multiple related data tables: you could probably get away with just defining a table that has fields relating to the columns in the Excel workbook.  Yes, purists will burn me at the stake for that statement, but the object here is to get something that works in place without making a major development effort out of it.  After it's up and running in this 'primitive' state, then you can kick pack and design a really efficient database normalized out to about the 3rd level of normalization if you care to (might not be a bad idea).

Also you should be able to import all of your current data into the database, so there's not a huge effort there.  Then you tweak the fields in the table to identify what defines a unique record, and what fields are mandatory.  Build a data entry form based on the table - put a "New Record" button on it and a couple of others (Save record, Quit, etc).  Split the database into front-end/back-end and take the front-end around to your users and install it on their computers and get the link to the back-end set up for their machine and you're in business!  They open and use the front end just as they have been using the shared Excel workbook.

Actually if you set up the link to the backend using its network URL instead of a mapped drive letter, then you are really down to easy distribution of the front end.
