ken sheridan say: You need to model the entity and relationship types by tables in the database.  In your case there is a many-to-many relationship type between orders, products, colours and product types.  This relationship type would be modelled by an OrderDetails table which resolves the relationship type into four one-to-many relationship types, and, in addition to the four foreign key columns has columns for attributes of the relationship type , e.g. the quantity ordered.

So, broadly speaking the table would be like this:

OrderDetails
….OrderID  (FK)
….ProductID  (FK)
….ColourID  (FK
….ProductTypeID  (FK)
….Quantity

The primary key of the table would be a composite one of the four foreign key columns, each of which would reference the primary key of one of four tables modelling the orders, products, colours and product types entity types.

Such a table on its own, however, would assume that every product is available in any possible combination of colour and product type.  This is unlikely to be the case, so the relationship type between colours and product types would need to be modelled by a further table:

ProductTypeColours
….ColourID  (FK
….ProductTypeID  (FK)

The same is true of the relationship type between Products and Colours:

ProductColours
….ProductID  (FK)
….ColourID  (FK

and between products and product types:

ProductProductTypes
….ProductID  (FK)
….ProductTypeID  (FK)

This is known as 3-decompostion and results in normalization to Fifth Normal Form (5NF). Each of the tables is 'all key' i.e. its primary key is a composite of the two foreign keys.  The corresponding pairs of columns in OrderDetails are therefore each a composite foreign key referencing the composite primary key of one of the above three tables.  By the enforcement of referential integrity in all of the relationship types the ordering of a product in a non-available colour and product type combination for that product is avoided.

You'll find an example of how relationships like this are built up across a database to achieve the final model as Relationships.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file uses a simple medical prescriptions model as its example.  As well as illustrating how the model is built, it also illustrates a simple interface for data entry, using correlated combo boxes which reflect the constraints imposed by the relationship types.  Note that the construction of the interface in a case like this is made much easier by the use of 'natural' keys, e.g. DrugName, DrugForm etc rather than 'surrogate' numeric keys like DrugID, DrugFormID. Prescriptions are broadly analogous to orders in your case.

suzanne s. barnhill say:

From your description, you must have Word 2013 or 2016 (previous versions cannot open PDFs at all), and you should edit the "Applies to" for your question accordingly.

The point of opening a PDF in Word is to be able to edit it. To accomplish this, Word must "reflow" the text into a Word document. If you just want to be able to read or print a PDF, you need to open it in Adobe Reader or another PDF reader. When you open it in Word, you must be prepared for some changes in layout, but the text and graphics should theoretically be intact. The more complex the layout, the less likely it is that Word will get it right.
