=[Parent]![SubForm1]![Comments]

 

where SubForm1 is the name of the first subform as a control on the main form.

Private Sub Comments_AfterUpdate()

Private Sub Form_Current()
    Me.Parent!SubForm2!txtComments = Me.Comments
End Sub

 

where SubForm2 is the name of the second subform as a control on the main form and txtComments is the name of the text box that you want to populate.

Instead of putting the same code in both places, just put it in the After Update event of the comments and then call that same after update event from the On Current:

 

Private Sub Comments_AfterUpdate()
    Me.Parent!SubForm2!txtComments = Me.Comments
End Sub

Private Sub Form_Current()
   Comments_AfterUpdate   

End Sub

 

That avoids having duplicate code and makes it so you only have to maintain it in ONE place.

Exchange Online users are unable to access on premises Public Folders - Microsoft Community

alexandru cazacu say:

Exchange Online users are unable to access on premises Public Folders in a hybrid deployment

I've been meaning to write a few lines about this issue, as it is often encountered in Office 365 support issues and topics.

A lot of customers that have an Exchange hybrid deployment (Exchange Online in rich coexistence with Exchange on premises), and still have their Public Folders infrastructure hosted on premises, have at least dealt with this cross forest access issue at least once, during their migration process to cloud. 

For Exchange Online users to be able to access on premises, remote public folders, some conditions must be met first and below is a list of them:

1. Check that these articles were followed correctly:

https://docs.microsoft.com/en-us/exchange/collaboration-exo/public-folders/set-up-modern-hybrid-public-folders?redirectedfrom=MSDN  - for Exchange 2013/2016 hybrid deployments

https://docs.microsoft.com/en-us/exchange/collaboration-exo/public-folders/set-up-legacy-hybrid-public-folders?redirectedfrom=MSDN - for legacy Exchange hybrid deployments (Exchange 2010)

2. Check that the Exchange Online Organization Configuration, has the PublicFoldersEnabled set to Remote instead of Local (default setting). This can be seen by running in Powershell connected to EXO, the Get-OrganizationConfig | fl *Public*,*Folder* command. 

3. Check the output of the command Get-OrganizationConfig | fl *remote* executed in Powershell connected to Exchange Online. The output should give us the names of the remote on premises public folder mailboxes.

4. Check in Powershell connected to Exchange Online, that those on premise public folder mailboxes, are synchronized in Office 365 as Mail User objects.

5. Check that the cloud users have either the DefaultPublicFolderMailbox or the EffectivePublicFolderMailbox attributes, populated with the name of the on premise public folder mailbox, when you run Get-Mailbox | fl PrimarySmtpAddress,DefaultPublicFolderMailbox,EffectivePublicFolderMailbox in Powershell connected to Exchange Online.

6. Check that the cloud users are synchronized from on premise, and that the on premise Exchange server, is seeing them as Remote User Mailboxes. You must take a note about the LegacyExchangeDn and the X500 addresses of those users that will be created with a mailbox directly in cloud instead of being migrated, as not having the correct X500 address stamped on the user, will make him unable to access the on premise public folders. See these articles/threads for more details:

https://answers.microsoft.com/en-us/msoffice/forum/msoffice_outlook/o365-hybrid-public-folder-access-for-new-o365/87c06fcf-c0d1-48df-a7c0-7ebc4b0a6249

https://answers.microsoft.com/en-us/msoffice/forum/msoffice_o365admin-mso_exchon-mso_o365b/cross-forest-full-access-permissions-in-a-hybrid/f1b8fa00-4bf7-4e49-8945-299ad85a4575

https://support.microsoft.com/en-us/help/3106618/exchange-online-users-can-t-access-legacy-on-premises-public-folders

7. Make sure that the appropriate permissions are assigned for the affected users, on the on premise public folders. In the on premise Exchange Management Shell, a command like Get-PublicFolderClientPermission <name or full path of the public folder> | fl can be executed to see the users that have permissions on a given public folder. It is also important, to have on the root public folder, the default permission set to at least Author, this can be seen with Get-PublicFolderClientPermission \ | fl.

To see corrupt permissions on the public folder, use this command (Get-PublicFolderClientPermission <name or full path of the public folder>).User. We are looking at UserType column, for "Unknown" values. These type of permission entries can cause access issues as well, but they can be removed with these commands:

$a=Get-PublicFolder -Identity "<\full path of the PF>"  | Get-PublicFolderClientPermission | ?{$_.User.UserType -like "Unknown"} 

$a | ForEach-Object {Remove-PublicFolderClientPermission -Identity "<\full path of the PF>" -User $_.User.DisplayName}


8. Check that the Autodiscover process is working correctly. DNS records for Autodiscover should point to the on premise Exchange server. Run the EXRCA Outlook Autodiscover tests for Office 365 and confirm that Autodiscover is correctly configured (the test should pass).

NOTE: Just to be more clear on the X500/LegacyExchangeDn statement from point 6. When we have a Mail User synced with Office 365 (the user has a mailbox in on premise Exchange), there will be a proxy address stamped in Exchange Online, as an X500 address that contains the value of the on premise LegacyExchangeDn (this X500 address is visible only in cloud, it doesn't exist on the on premise user's properties). 

Similarly, when a user has been migrated to Exchange Online, or his mailbox has been directly provisioned in Office 365 (with New-RemoteMailbox command for example executed in on premise Exchange), the local Remote User Mailbox object, will need to have at least a proxy address of type X500, with the value of the Exchange Online LegacyExchangeDn.

Back to Exchange Online Support Corner

PS: Since we are speaking about hybrid deployments and cross forest permissions, this new support article might be very useful as well: https://support.microsoft.com/en-us/help/3064053/overview-of-delegation-in-an-office-365-hybrid-environment

Tags: Public Folders, Access, Hybrid, Exchange, Permissions

Windows Mobile 6.1 POP and IMAP Send Mail Patch

 

This behavior is associated with a feature that is introduced in Windows Mobile 6.1. The feature allows for mobile operators to specify an alternate SMTP server name that is used if e-mail messages cannot be sent by using the user-specified SMTP server name. If the mobile operator does not specify an alternate SMTP server name and if the Windows Mobile 6.1-based device does not connect, the e-mail account is corrupted and cannot send e-mail messages.

 

Important

Follow all steps below completely and in order. Be sure to complete all steps.

Follow only one set of instructions below, depending on whether you synchronize your device to a PC:

·         If you connect your device directly to a PC with a USB cable or cradle, follow the instructions for Windows Mobile users who connect to a PC to synchronize.

·         If do not synchronize your device or phone with your PC or you only synchronize your phone or device wirelessly with a Microsoft Exchange Server or other e-mail and calendar systems, follow the instructions for Windows Mobile users who do not connect to their PCs to synchronize

 

Windows Mobile users who connect to a PC to synchronize

 

Download and run the Windows Mobile 6.1 POP and IMAP Send Mail Patch.

Check to make sure you have the latest version of ActiveSync or Windows Mobile Device Center:

ActiveSync 4.5 or Windows Mobile Device Center 6.1.

 

Connect your Windows Mobile device to your PC, download and then run the Windows Mobile 6.1 POP and IMAP Send Mail Patch for Windows Mobile

 

Windows Mobile users who do not connect to their PCs to synchronize

 

For users that do not connect to PCs, there is an update file (.CAB) available to make this update. (Note: There are a number of ways to install a .CAB file and below outlines a popular method).

Using the Microsoft Internet Explorer Mobile browser on your device, go to this location http://www.microsoft.com/downloads/details.aspx?FamilyId=D9D71B2E-D2DD-44F2-86E5-1E53AAD7FB7A&displaylang=en and download this file, WindowsMobile61-KB958639.msi, to your Windows Mobile device. This file will install the update directly on the device.

 

Installation Instruction:

 

From your device, navigate to the file you downloaded, and then tap it (or select the file and press “OK” button on a non-touch screen device) to install the update. If you are unable to download the file proceed to step 2.

 

If you are unable to access the file in step 1 on your device, open Windows Internet Explorer on your PC and click here to start the download process. After downloading the file, attach it to an e-mail and send it to an account you can access on your device (the send mail bug will not prevent you from receiving messages). On your device, open the e-mail and save the attached file; on some devices, you may need to synchronize your e-mail a second time to download the attached file. Then navigate to the file on your device and click on it to install the update.

Note: After you install this update, your device will automatically restart to ensure that the update is complete

 

Installation tips

Check the messages in your Outbox

 

After you have applied the update to your Windows Mobile device, messages in the Outbox folder of your affected POP and IMAP accounts will be sent. You may no longer wish to send these messages if they are out of date.  You can keep these messages from being sent by navigating to the Outbox folder of these accounts and either deleting the messages or moving them to another folder. On a WM 6.1 Professional device you can access account folders by tapping on the folder drop down located on the upper left of the message list.  On WM 6.1 Standard devices you can access account folders by Pressing the Menu soft key, then selecting Folders.

One of the problems we have with this thread is your use of terminology which is causes confusion.

1) In MS Outlook you add a mail account, which may be pop, IMAP or Exchange type of mail account

2) Reference to email address means a contacts email address, it does not mean your mail account

3) A backup is not an export

In any export/import there should not have been any reference to a email address/account, it should only have contained references to the default folders, e.g Inbox, Sent etc

Your initial post said you needed more space - this is a confusing, as in an IMAP mail account, the msgs that you see in MS Outlook is simply a copy of the data held on your IMAP mail server, and as such unless you are receiving msgs for your mail provider regarding capacity/space there should be no issues.

Copying/exporting msgs from an IMAP mail account does not remove the msgs from the IMAP mail account, and as such you will not gain any 'space'

I think you need to state, explicitly what error or other msg you are seeing with regard 'space'
