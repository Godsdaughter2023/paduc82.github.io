feidaman say:

(I've just stumbled upon a reliable solution. this problem has been bugging me for a long time and I'm reposting here what I've just posted over at stackoverflow just to spread the word)

Here is my solution, it works consistently and you don't need to manually copy the sheets and code across to a blank workbook. I've tested this method on several corrupted workbooks that were giving me the "Automation error - Catastrophic failure" error upon launch.

NOTE: Original corrupted file was saved as .xlsm

1. Open a blank Excel workbook
2. Developer tab > Macro security > Disable all macros without notification
3. Close Excel
4. Double-click corrupted file, for example, MyFile.xlsm
5. File > Save as > MyFile.xlsb (not .xlsm), choosing the .xlsb format is what does the trick
6. Developer tab > Macro security > Enable all macros (or whatever level of security you prefer)
7. Close Excel
8. Double-click MyFile.xlsb

The file is now fixed! You can re-save the MyFile.xlsb file as .xlsm if required. In my experience .xlsm files become corrupted quite easily, so I'm going to get into the habit of always using the .xlsb format.

Hope somebody finds this helpful :)

graham mayor say:

It is relatively simple to create a Word document from Excel using a Word template e.g.

Dim wdApp As Object
Dim wdDoc As Object

    On Error Resume Next
    Set wdApp = GetObject(, "Word.Application")
    If Err Then
        Set wdApp = CreateObject("Word.Application")
    End If
    On Error GoTo 0

    wdApp.Visible = True

    Set wdDoc = wdApp.Documents.Add(Template:="C:\Path\TemplateName.dotx")

    'do what you want with the document here

If the template already contains bookmarked locations, or it contains DOCVARIABLE fields, or even content controls it is a simple matter to populate the fields/bookmarks with the required data from your worksheet.

If the Worksheet is a list of similar items, then it might be easier to start in Word, and create a userform to display the list and then compile the document from your list. Either way the process is pretty much the same. See http://www.gmayor.com/Userform.htm  and http://www.gmayor.com/Userform_ComboBox.html which between them cover the code techniques required to populate the document.

john mcghie say:

You can't do it.

Microsoft Office for Mac applications will use ONLY TrueType fonts, and OpenType fonts provided they contain TrueType outlines (glyphs).

It would appear that the character you are after is not one of those.

If you create a Document in TextEdit and save as a Microsoft Word 2007 .docx document, put your signature in, then save and close, what happens when you open that in Word?

Square Boxes?  You're hosed, it's not going to do it.

If the characters DO appear in Word, then you can create an AutoText or AutoCorrect to insert it:

See here: http://www.dummies.com/how-to/content/how-to-use-autotext-in-word-2011-for-mac.html

Make certain to select "Formatted Text" to store the formatting in the Autocorrect.

What you are calling "Text Substitution", Word calls "Autocorrect".

In fact, if you follow the process for creating an Autocorrect even though Word can't display the characters, it will correctly insert them into the document (yeah, I know, don't get me started...).  Sadly, Word won't be able to print them, although TextEdit will.

There are any number of utilities out there that will convert that font into a TrueType font that Microsoft Office for Mac can use. If you were to ask on one of the Mac groups where folks do a lot of this, some kindly soul may offer to do it for you: it takes only a few seconds provided you have the correct software.  Note: If the reason Word can't use it is that the glyphs are coded as 8-bit characters, you will have to fix that: Office requires all characters to be 16-bit Unicode with a byte order mark correctly assigned.

Hope this helps
