Start may become unresponsive in build 10130. Here is a possible workaround.

Start may become unresponsive in case someone chooses to turn off the three options under Settings-Personalization-Start-Customize list.

In that case: right click on an empty desktop space and choose Personalization. That should open the Desktop Settings. Navigate to the Start settings and re-enable the three options under Customize list. Start will work again.

This workaround is valid only for the described scenario. If the Start does not open for reason, other than the prior change of Customize list options, this workaround will not help.

marshall barton say:

I'm not entirely comfortable with requerying the combo box in the Enter event, but I can't summon an argument against it either.

OTOH, requerying the form should be independent of the combo box so it can be done in lots of places,except the form's Current event and while entering/editing a record.  Trying to keep multiple users up to date with changes from all users is an extremely complex issue that is impossible to do in all situations.  Doing a Me.Requery at appropriate times is the best you can do so there will always be some time lag before one user can see the edits and additions made by another user.

One thing that needs to be considered in this context is the Refresh method is significantly different from the Requery method.  Refresh will update a form's data for saved edits made by other users without repositioning the form's current record.  Refresh will not reflect records that other users added or deleted.  But Requery will save any changes to the form's current record and make the form's first record current while it picks up all added, edited and deleted records from other users.  Note that Access usually(?) does a Refresh on a periodic interval (Tools - Options - Advanced - Refresh Interval) so it is an unusual situation where you need to force it to happen.

Another thing is to keep the locking scenario in mind for situations where multiple users are editing the same record.  Optimistic locking will notify a user that the edits they are trying save have already been changed by another user and ask which set of changes should be kept.  Pessimistic locking will prevent a second users form editing a record that another user is editing.  It is very important to realize that the pessimistic locking mechanism actually operates on a 2K memory buffer, so if multiple records will fit in the buffer all the records in the buffer are locked so a user eiting a record may lock out a user that wants to edit a different record.

Back to your immediate problem, I don't see how requering the form can mess up the operation of the combo box.  For a search/navigate combo box, the ControlSource should be empty so searching does not edit a data field.  I also wonder if there may be something about the combo box's BoundColumn, ColumnCount or ColumnWidths properties that might not be right.
