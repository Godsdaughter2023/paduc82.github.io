A problem with all these procedures is that their diagnostics are obscure, intended really only for support staff who have access to the source and support documentation.  That is why I will always try using ProcMon to capture more detail about what is happening.  We can filter first by Operation Is WriteFile and thus get a good impression of how the writes of each log file occurred.  More significantly we can try to infer which write wrote which log record and thus find the place in the trace where messages that we are interested in can be examined without the filter in order to see what Registry and other File accesses may have gone before the generation of that one.

Alternatively you can at least search for all files which had  DateModified:Today   (That search term is actually allowed in File Explorer but it is only useful if all standard inhibitions of it are disabled, e.g. make sure that System directories are searched and Hidden files allowed to be found.)  Then you can sort the results by Date Modified and scroll to the minute(s) that the update ran and see what files were written in that time period.

In the case of CBS.log if it is corrupted I suspect renaming it would allow a new uncorrupt one to be started.  In the case of DISM.log, notice that DISM has some command line parameters which can be used to both redirect and filter the output.

BTW how far did you go with your FixIT?   E.g. if you got to step 5 you would be able to use the DISM command line modifications I alluded to above.  In fact, I thought it was usually the DISM.log which gets corrupted more frequently than the CBS.log.
