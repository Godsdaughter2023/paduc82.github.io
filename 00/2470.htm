Archiving mailboxes of departing employees in a dirsync environment - Microsoft Community

briandial- [o365] say:

Archiving mailboxes of departing employees in a dirsync environment

I've recently been developing a procedure on how to archive the mailboxes of employees who leave as shared mailboxes for quick access should a person say "Hey can you get me so-and-so's email they were working on a project...". Apart from that, I also have the requirement to free up the address for a future employee. 

I've read a few threads with some information and have been trying to cobble together a process but i'm running into an issue with how dirsync works. 

The first thing I do is delete the user from active directory. After a drsync, this puts them in the Deleted Users section of office365. I then restore the user, and it restores them but changes their status to In Cloud instead of Synced with Active Directory. At this point I thought I had broken the link between our AD and them ailbox. So now I can go into exchange admin and convert the user to a shared mailbox.

Next time dirsync runs though, it deletes the user again. I can see in the synchronization service manager the delete request but I have confirmed that this user is no longer in active directory.  And since the user says in cloud as their status, why is dirsync continuing to perform operations on this user?

young yang msft support say:

Hi Brian,

This is normal/expected behavior in DirSync environment.

1. When an AD user is deleted, DirSync will sync the change to Office 365. So the Office 365 user is deleted and moved to Deleted Users (recycle bin).

2. In Office 365, when a user is deleted, there are 30 days grace period before the user is permanently removed. During this grace period, we can temporarily restore the user in Office 365. After the restoration, the status is “In cloud”. During this short period, you can manage the user as a cloud user. This is not due to that the link between AD and Office 365 is broken. The reason is that: right now, DirSync has not run. Incremental sync is not performed from local AD to Office 365.

This kind of restoration has no practical effect.  

3. The next time DirSync runs, DirSync service detects that this AD user is already deleted, but the corresponding Office 365 user still exists. The incremental sync is performed. So, the Office 365 user is deleted again.

This is a normal behavior. So, if you continue to use the user as a shared mailbox in Office 365, we cannot delete the user.

Anything unclear, please let us know.

Thanks,

Young
