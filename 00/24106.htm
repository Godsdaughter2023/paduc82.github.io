[feature request] real time background execution for UWP, - Microsoft Community

mihaicosmin say:

[feature request] real time background execution for UWP, including on mobile

I recently came across a question in the MSDN UWP forum and I figured to leave some feedback here, and also open the thing for suggestions.

The question was about real time processing in UWP.

As we all know, UWP has a rather unfriendly life cycle for applications, that leave any apps at the mercy of the OS when they are in the background, thus cutting any sort of real time approach to multi tasking.

My idea is to extend the current "sponsored background execution" to any application that opts in to a capability called "real time background execution". This will allow applications to continue to run in the background, much the same way as media apps are in W10M (shed memory when pressure gets high be unloading UI).

Users should be given a notification when downloading the applications from the store, much the same way it happens with location-based applications, so they knew these application may continue running in the background forever.

The task manager should flag these apps and keep them available to the user to switch to or to close them at any time (unlike now, when apps can disappear from task manager even if they continue to run in background). Users can run as many of these apps in background as the hardware allows it: probably enforce limits depending on processing power.

Tiptoe,

Approximately how many messages do you have in your inbox?  10? 100? 500? 1000?

People (whose accounts have been moved to the new Outlook Mail interface) with many messages in their inbox (hundreds or thousands) are complaining about having scrolling/performance problems. 

The reason is that in the new interface, ALL messages in ones inbox are kept in one long scrollable page (the old interface kept messages in short pages with 35 messages per page).

In the old interface, in boxes (with many messages) were easily scrolled (because the longest page one had to scroll only has 35 entries). In the new interface, attempting to scroll one long list of messages (with hundreds or thousands of messages in one page) causes performance issues.

If you have many messages in your inbox, that might be the cause of your problem. You could try various browsers to see if any handle scrolling a long list better than the browser you are currently using. But ultimately, only Microsoft can fix this problem.

Microsoft knows about this issue .... many people have complained. Either Microsoft never bothered to test this case (inbox with many messages), or they tested and found the problem ... but did not care enough to fix it before rolling out the new Outlook Mail interface. 

Microsoft needs to provide an option in the new Outlook Mail interface that would allow users to define how many messages are displayed per page.
