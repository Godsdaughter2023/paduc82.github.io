That's because you set the width of the first column to 1".  A combo box will always show the first visible column when it is not dropped down.  The natural arrangement of a lookup field is to *store* the foreign key field, but not display it.  To do that, it would have its ColumnWidths property specify that the the first column has a width of 0; e.g.,

    Column Widths: 0";1"

A more fundamental question I have about the lookup query convention is why the key field of the lookup table needs to be in the select clause at all, if it is not being displayed.  The order of the records can still be determined by the presence of the key field in the ORDER BY clause.  Or perhaps the 1st field in the query return *is* used to match the bound value in tMain.fForeignKey.  I mean, I was only *assuming* that the matching with tMain.fForeignKey is based on the row number of the query return rather than first field (with the order of the records in the dropdown list being determined by the ORDER BY clause).

Your original assumption was wrong.  The whole point of a lookup field -- for good or ill -- is to *store* the foreign key but *display* the associated description or other related data.  Therefore, that foreign key must be in the SELECT clause of the combo box's RowSource property, and that foreign-key column must be set as the bound column of the combo box.

dirk goldgar say:

(4) At the point where the Lookup Wizard asks "When you select a row in the lookup field, you can store a value from that row in your database, or you can use the value later to perform an action.  Choose a field that uniquely identifies the row.  Which column in your lookup field contains the value that you want to store or use in your database?", you need to select the Description field (i.e., the field containing the value from the lookup)

Unless I'm misunderstanding you, this is not correct.  In a lookup field, the stored value would be the actual foreign-key field, not the description field.  Then Access will automatically look up the value of the description from the related table when you open a datasheet on that table.  That is the purpose of lookup fields:  to establish the relationship between the tables and tell Access to perform the appropriate lookup automatically, by using a combo box with the appropriate properties to display the field.

You should be aware that many professional Access developers disapprove of lookup fields defined in tables, because they tend to hide the actual data that is being stored.  See this article for an expanded explanation of that point of view:

    The Evils of Lookup Fields

That doesn't at all mean that you shouldn't establish relationships between tables, use foreign keys to implement relationships between entities, or use combo boxes on forms to implement lookups so that users don't have to manually look up foreign-key values.  The issues only surround the use of lookup fields in the table design.  The only real reason for using lookup fields in the table design is if your users are going to be working directly in the tables, rather than in forms and reports, and that is to be avoided.
