Assuming all your chemical details for each email are part of the same record (e.g. they're on the same row in an Excel workbook), you can conditionally output them, with punctuation, using a field coded as:

{MERGEFIELD chemical1}{IF{MERGEFIELD chemical3}= "" " and {MERGEFIELD chemical2}" ", {MERGEFIELD chemical2}{IF{MERGEFIELD chemical4}= "" " and {MERGEFIELD chemical3}" ", {MERGEFIELD chemical3}{IF{MERGEFIELD chemical5}= "" " and {MERGEFIELD chemical4}" ", {MERGEFIELD chemical4} and {MERGEFIELD chemical5}"}"}"}

or:

«chemical1»{IF«chemical3»= "" " and «chemical2»" ", «chemical2»{IF«chemical4»= "" " and «chemical3»" ", «chemical3»{IF«chemical5»= "" " and «chemical4»" ", «chemical4» and «chemical5»"}"}"}

Note: The field brace pairs (i.e. '{ }') for the above example are all created in the document itself, via Ctrl-F9 (Cmd-F9 on a Mac); you can't simply type them or copy & paste them from this message. Nor is it practicable to add them via any of the standard Word dialogues. Likewise, you can't type or copy & paste the chevrons (i.e. '« »') - they're part of the actual mergefields, which you can insert from the mailmerge toolbar. The spaces represented in the field construction are all required.

Julian Date

Dear Microsoft,

Your Julian date offset is completely arbitrary; JD 2415018.5 corresponds to December 30, 1899. I can tell you from experience that no one in astrophysics, trajectory analysis, or planetary science uses this offset. There are two standard offsets out there:

1.  2430000.0     Jan 5, 1941

 (Preferred for those working in trajectory analysis)

2. 2400000.5     Nov. 17 1858 (Preferred for those working in astrophysics and planetary science)

While we appreciate you having an automatic conversion from JD to MM/DD/YYYY built in, anyone working in the fields that actually would use it would be pretty confused about your offset.

-A Concerned Aerospace Engineer

ken sheridan say: Are the subreports merely for returning the totals per category?  If so then I would question the need for subreports at all.  You can conditionally sum the units per category in the ID group footer by summing the return value of an expression which returns the quantity sold/used or a zero.  To sum those sold unprocessed per ID the expression for the ControlSource property of a text box in the ID group footer would be along these lines:   

= Sum(IIf([Category] = "Sold unprocessed",[Quantity], 0))

You can do the same in the report footer to return the counts per category for all IDs.

To return the balance in hand:

= Sum(IIf([Category] <> "Sold unprocessed" And [Category] <> "Used in production" And [Category] <> "Sold processed",[Quantity], 0))

NB: if Category (or whatever) can be Null you'd need to cater for this in the expression with Nz([Category], "XXX"), the "XXX" being an arbitrary string expression which will not be the value of any category.

To sum the overall quantity of all units sold/used/in hand is simply:

=Sum([Quantity])

Even if you still need to use the subreports for showing other data not encompassed by the parent report you should still be able to return the aggregated values in this way.

dhookom say: Without knowing how your table structure is setup, I would create a query that sums the units by category and add it to the main report's record source. Then, you don't have to worry about pulling data from the subreports. 

ken sheridan say: I see no need for subreports here.  All you need to do is:

1.  Return all the details currently returned in the subreports in the report's recordset.
2.  Group the report by ID
3.  Then group the report by the category (sold uprocessed, etc)
4.  Return the total per category in the category group footer with =Count(*) for a simple count or =Sum([Quantity]) if it's the sum of quantity per row.
5.  Return the total per ID in the ID group footer.
6.  Conditionally compute the grand count or sum per category in three controls in the report footer in the way I described.

It's very easy to get into a mindset that where a subform would be used in a form, a subreport should be used in a report.  Form's have no grouping mechanism, however, and it's been my experience that in many cases subreports are not necessary.  Where they do come into their own is in presenting related data in a format which differs from that of the parent report.  Where data is simply being grouped, with aggregated values at one or more group levels, then that can generally be achieved more easily by a single report, which from your description, seems to be the case here.
