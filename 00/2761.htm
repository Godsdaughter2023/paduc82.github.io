ken sheridan say: A key violation error occurs when an attempt is made to insert a row into a table in which the value being inserted into the primary key columns (or columns in the case of a multiple key) already exists in a row in the table.

I'm unclear why you are using 'append' queries here.  At first sight it would appear that your form could be constructed by means of an Orders parent form within which is embedded an  OrderDetails subform, linked to the parent form on OrderID (or whatever the key is).  What you appear to have in relational database terms is a many-to-many relationship type between orders and products (each customer can receive one or more product (paper), and each paper can be supplied to one or more customers).  The model for this, in broad outline, would comprise the following tables;

Customers
....CustomerID  (PK)
....FirstName
....LastName
....AddressLine1
....etc

Rounds
....RoundID  (PK)
....Round

Orders
....OrderID  (PK)
....CustomerID  (FK)
....OrderDate
....RoundID  (FK)

Products
....ProductID  (PK)
....Product
....UnitPrice

ProductCategories
....ProductCategoryID  (PK)
....ProductCategory

In your case the categories would be things like Sunday or week day papers, magazines etc.

and to model the many-to-many relationship type:

OrderDetails
....CustomerID  (FK)
....ProductID  (FK)
....ProductCategoryID  (FK)

The primary key of this last table is a composite one of the two foreign key columns.

The tables listed in the navigation pane seem to tally to a large degree with the above model wt the notable exception of an OrderDetails table.

This type of model is very common.  Northwind is an example, and you'll find a similar model, and its representation in a form/subforms as InvoicePDF.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

One significant difference to the model which might or might not apply in your case however relates to the unit price of a product. in both Northwind and my demo the Products and OrderDetails (Invoices and InvoiceDetails in my file, but that's immaterial) tables both include UnitPrice columns.  This is because prices change over time so instead of simply referencing the UnitPrice column the OrderDetails table also includes a UnitPrice column to which a value is assigned when a row is inserted into OrderDetails via subform.  The price at the time of the order is therefore retained, regardless of any subsequent price changes of the product.  In the context of a paper round database, however, it's possible that this might not be appropriate as you would want the order to show the current price per paper for the purposes of invoicing customers on a regular basis.  That's for you to decide.

Hopefully the above will give you some general background for fleshing out the details of your model, but as a coursework assignment the onus is on you to do the legwork of course.  We can help point you in the right direction, but it would be wrong for us to provide you with too much detail as you would then not have the learning experience of working it out for yourself.

Coming back to your original question, it would help us to help you if you can provide a description of what you are attempting by means of the 'append' queries.  The fact that you have an 'Add Order' button suggests that this is what your queries might be for.  If so, a bound form and subform would be a simpler solution, and a more usual one it this context.  Unbound controls are appropriate for data entry in some situations, usually where a format for data entry which does not reflect the underlying model is required.  The VolunteerRota demo in my same OneDrive folder shows how the same data can be entered via a conventional bound form/subform, or by means of unbound controls.  From the image you've posted, however, I would have thought a conventional bound form/subform would be the appropriate interface in your case.

scottgem say:

A key violation is just what it says. Your query violates the constraints placed on a key field. A key field is general set to No Duplicates, so you are apparently trying to add a record that already exists in your database.

Append queries are not "applied" to a form. Append queries add data to a table. Forms are just a means to interface with tables. There are two ways to build a form, Either bound or unbound. A bound form means that the Recordsource of the form is a table (sometimes a query). With a bound form Access handles all the I/O between the form and the table. An Unbound form will have a blank Recordsource property. With an Unbound form, the developer has to write all the code to move the data entered in the form between the form and the table. While unbound forms are an important tool in many instances, bound forms are much easier to work with.

As Ken illustrated it appears you have the need for multiple tables. Using subforms bound to those tables will make your work a lot easier. Unless, your assignment was to create an unbound form.

ken sheridan say: In a relational database multiple instances of the same entity type (in this case a paper) are not modelled by multiple columns like Sunday1, Sunday2, Daily1, Daily 2, but by multiple rows in a table.  If we look back at the model I suggested, on reflection I'd change it as follows:

Products
....ProductID  (PK)
....Product
....ProductCategoryID  (FK)
....UnitPrice

OrderDetails
....CustomerID  (FK)
....ProductID  (FK)
....DateFrom
....DateTo

ProductCategoryID  has been moved from OrderDetails to Products.  DateFrom and DateTo columns have been added to record when a customer starts to order a product, and when they remove that product from their order.  The DateTo will be Null until this is done.  So let's assume that a customer with a CustomerID value of 42, orders the Guardian and the Times as their daily papers, and the Observer as their Sunday paper.  They also order the Radio Times.

Let's assume that the ProductCategoryID, for daily papers is 1, for Sunday papers 2, and for weekly magazines 3.  These would be the values in products in the ProductCategoryID   column for the Guardian, The Times and the Radio Times respectively.  Let's assume the ProductID values for these are 8, 12 and 23 respectively.

The products ordered by Customer 42 can therefore be represented in the OrderDetails table by three rows like this:

CustomerID    ProductID
42                 8
42                 12
42                 23

When it comes to determining how many of these a customer received there are a number of ways this could be modelled, but I would suggest a table like this:

IssueDates
....ProductID  (FK)
....IssueDate

You might think this is going to produce a very large table, and you'd be right.  There are approximately 300 issues of the Guardian each year, so this means approximately 300 rows in the table for the Guardian each year.  Ditto for the Times and other dailies.  The Observer and other Sunday papers will have around 50 rows each per year, as will the Radio Times and other weeklies.  Monthly magazines will have 12 rows per year.  But bear in mind that the number of dailies, weeklies and monthlies stocked by a newsagent is not really that large, so even over 10 years or so the number of rows in this table, while it might seem large at first sight, is chickenfeed in database terms.

Of course people do not take every issue of a paper or magazine.  They cancel them while on holiday for instance.  So another table would model this:

Cancellations
....CustomerID  (FK)
....ProductID  (FK)
....IssueDate

So taking the OrderDetails, IssueDates and Cancellations tables together we know which products a customer received and on which dates.  From this we can compute the number of each product received in any date range, and with the current UnitPrice of the product in the Products table we can thus compute the total price of products ordered in the range.  To this would normally be added a weekly service charge in order to compute the total invoice amount for the range.  These values would be inserted into an Invoices and an InvoiceDetails table by means of an append query when the invoice is issued.  Another column in the Invoice table would record the amount paid by the customer, so provided this equals the total invoice amount the customer's account is in balance; if it's less than the total invoice amount the account is currently in debit.

Once the referenced tables like Products, ProductCategories, IssueDates have been populated all regular data entry is easily done through bound forms/subforms. To populate the IssueDates table over a number of years with rows for all products can be achieved quite easily by executing  a set of append queries, one for each product category, but for the purposes of your project, which will persuadably include a limited number of products over a relatively limited data range it should not be an unduly onerous task to insert the rows manually.

I imagine that in your studies you will have come across the use of 'entity relationship diagrams'.  In essence this is similar to what you see in the relationships window of a database, but compiling a diagram on paper before creating the tables for a database is both a time-saver and, more importantly, a way of marshalling your thoughts about the design of the model.  I'd suggest, therefore that you do this with the tables I've described here to see how the model fits logically together.  Then you can mentally test it by considering how it would handle the sort of functionality you envisage for the database. Only then should the tables be built, and after that the interface of the forms, reports queries etc.  But don't assume the model I've suggested is a cut and dried solution.  This is you project and you know what's wanted of it, so you'll almost certainly want to make changes, perhaps radical ones.  Nevertheless I hope it will give you a foundation on which to build a model which respects the principles of the relational model.

Userforms are NOT useful if you want the radio buttons visible/printable in the actual document. For this functionality you will need to use a Legacy Tools > ActiveX Controls > Option Button from the Developer Tab.

The basic process is to bookmark the chunks of your document that need to appear/disappear and trigger macros when the option buttons are changed. Those macros should examine the state of the radio buttons and set the relevant bookmarked regions as visible/hidden. 

Your macros will also need to configure initial settings to not display or print 'hidden' text. You may want to reconfigure these settings to the user's personal preference as the document is closed.

doug robbins - mvp office apps & services (word) say:

Based on your second point, you cannot use a userform.  Instead, you could insert a pair of ActiveX Radio buttons and have the code that runs when you click on either of them set the value of a document variable and have a DOCVARIABLE field for that variable in the document.  The code would also update the fields in the document so that the DOCVARIABLE field showed the relevant text.

There's a sample of a template set up that way at:
