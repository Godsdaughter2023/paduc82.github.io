ken sheridan say: However, there is an unasked question, which is whether you should be using a multi-valued field at all?   Unless you are interfacing the data with SharePoint multi-valued fields should not be used.  Instead store the multiple values as rows in a separate table which models a many-to-many relationship between two other tables, e.g.

Patients-----<PatientProcedures>-----Procedures

Here PatientProcedures models the many-to-many relationship type between Patients and Procedures by having PatientID and ProcedureID foreign key columns referencing the primary keys of Patients and Procedures.  The appropriate interface for this would be a Patients form in single view, with a PatientProcedures subform in continuous forms view containing a conventional combo box bound to the ProcedureID column.  Assigning multiple Procedures to a patient is simply a case of inserting multiple rows in the subform.  PatientProcedures (and therefore the subform) would almost certainly contain other non-key columns such as ProcedureDate etc. modelling the attributes of the application of a procedure to a patient.

Of course, the chances are you are not referring to medical procedures at all, but the principles are the same whatever entity types are involved.
Ken Sheridan, Stafford, England
