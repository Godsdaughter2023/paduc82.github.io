tommarasmus say:

So I have a design change request in the with the Outlook for Mac product group on this. Apparently this is an issue with Outlook 2011 but Outlook 2011 handles the error differently. Outlook for Mac for Office 365 (Outlook 2016) just lays out the error.

It was explained that this happens on recurring meetings that have exceptions created on them. An exception is any modification to one instance of the meeting (deletion, change of subject, attendees, body text, etc.). This occurs when Exchange is reading the exception list for the meeting and is unable to find a specific exception. When this occurs in most cases, Exchange will create one, which is by design. However, in this scenario, the user who is accessing the meeting does not have permission to create the attachment, as they only have reviewer permission, and therefore Exchange returns an Access Denied error. A meeting having a list of exceptions and then not being able to find one of the exceptions is not expected and points toward there being a problem with that tempting.

At this point, the communication protocol used by the client is being blamed. Since Outlook for Mac uses EWS, it handles the errors different than Outlook for Windows, which uses MAPI. But OWA uses EWS as well and I don't know about anyone else, but me and others do not get the errors and can see the meetings in OWA but not in Outlook for Mac.

At this point, I had the DCR submitted for the product group to review and hopefully have the Outlook for Mac client handle the errors/meetings w/exceptions the same was as OWA. If they can make that happen, then all should be happy.

So fingers crossed they accept the request. Currently it is under review.

jay freedman say:

There certainly doesn't appear to be any direct access in VBA to the
action that occurs when you uncheck the "Keep Content Control" box in
the Paste Options. I was able to cobble together a similar function by
using the content control's Temporary property. When that property is
set to true, the control is removed when its content is edited,
leaving the text behind.

It isn't easy to get an object that points to the content control (if
any) in the newly pasted material, either. I'm not happy with the code
here, but I'm too tired to think of something more elegant.

Sub PasteRTFandRemoveContentControl()
    Dim bStartDone As Boolean, bEndDone As Boolean
    Dim oRg As Range
    Set oRg = Selection.Range
    oRg.PasteSpecial datatype:=wdPasteRTF

    ' Alternately expand the range toward the
    ' start of the document and toward the end
    ' of the document until it includes a content
    ' control or it reaches both limits.
    While (Not bStartDone) Or (Not bEndDone)
        If (oRg.ContentControls.Count > 0) Then
            bStartDone = True
            bEndDone = True
        End If

        If (Not bStartDone) And (oRg.Start > ActiveDocument.Range.Start) Then
            oRg.MoveStart unit:=wdParagraph, Count:=-1
        Else
            bStartDone = True
        End If

        If (Not bEndDone) And (oRg.End < ActiveDocument.Range.End) Then
            oRg.MoveEnd unit:=wdParagraph, Count:=1
        Else
            bEndDone = True
        End If
    Wend

    ' Turn on the control's Temporary property
    ' so the control will be removed when the text is
    ' edited. Then add and remove a space to do the deed.
    If (oRg.ContentControls.Count > 0) Then
        oRg.ContentControls(1).Temporary = True
        Set oRg = oRg.ContentControls(1).Range
        With oRg
            .Collapse wdCollapseStart
            .Text = " "
            .Delete
        End With
    End If
End Sub

jay freedman say:

Unfortunately, both of those things are not only possible but true.
There are quite a number of operations in Word that function only in
the user interface (keyboard/mouse) and are not available in the
object model exposed to VBA. This is more often true of features that
were added in the last two or three versions, but some of them date
back to the very beginnings of Word.

Specifically in the case of the clipboard, VBA has access only to
plain text without formatting, which can be placed into a String
variable
(http://www.word.mvps.org/FAQs/MacrosVBA/ManipulateClipboard.htm).
There is no way to tell whether the clipboard contains anything else,
such as content controls, graphics, or other objects. It may be
possible by invoking functions in the Windows Win32 API, but I'm not
familiar with any such functions.

The Quick Access Toolbar's ready-made Print command executes the "Quick Print" command--which doesn't give you any options. If you like, you can put the regular Print (dialog box) onto the QAT as well or instead. Click the Office button, Print, hover over the arrow to expant, then right-click the Print command and choose Add to Quick Access Toolbar. If you never use Quick Print (I don't), you can right-click it on the QAT and click Remove so you don't click it by accident.

Or... if you're okay with keyboard shortcuts, Stefan's Ctrl+P solution works great.

herb tyson say:

Ctrl+P now brings up the Backstage Print window. To get the print dialog using the QAT, I believe you now have to use a macro. You can still assign FilePrint to a keyboard keystroke, though.

Because I like seeing the Print dialog regardless of which method I use, I use the following macro for creating a QAT tool:

Sub PrintDialog()
On Error Resume Next
Application.Run MacroName:="FilePrint"
End Sub
