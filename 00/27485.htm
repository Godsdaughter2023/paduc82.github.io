Try this macro to get the trendline equation from the chart into a worksheet cell from where you can filter out its different parts.

Sub get_trendline_equation()
   ActiveSheet.Shapes("Chart 2").Chart.SeriesCollection(1).Trendlines(1).DataLabel.NumberFormat = "0.000000000000000"
   Cells(1, 1).Value = ActiveSheet.Shapes("Chart 2").Chart.SeriesCollection(1).Trendlines(1).DataLabel.Caption
End Sub

The first code line sets the format to display without rounding and the second line copies the equation to cell A1.

scottgem say:

Before you can design a database, you need to understand normalization rules. This stie can help:

http://www.accessmvp.com/strive4peace/

You seem to have the entities correct (Customers, Bids, OutsideSales), but not the relationships. You shouldn't have the Customer NAME in Bids, just the customerID, Similary, if the Outside Sales Reps are associated with a customer, then the RepID should be a foreign key in the Customer table. 

What you have is a Many to Many relationship between customers and bids. One customer can have many bids and one bid can apply to many customers. The traditional ways to model this is with a junction table:

tjxCustBid

CustomerID (FK)

BidID (FK)

What you described in creating a lookup field was to create a multi-value field (MVF). This is a construct that exists in Access (and hardly anywhere else) that simulates a junction table. But because the junction table is hidden, you can't address it directly. So if you want to display a list of customers in your query, you can just include the the field in the query and it should display a comma separated list. But if you want to pull the RepName (or really the ID) from the customer table, then you have to use the .Value property of the MVF to join it back to the Customer table.

Using an MVF makes for a very nice interface, you just check off the items you want to select. But it makes it more cumbersome to link to other tables. Using a Junction table makes linking easier.
