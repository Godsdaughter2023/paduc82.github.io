jay freedman say:
If the same Letter B will be displayed in all of the merged copies of Letter A, then all you need is an INCLUDETEXT field that contains the path to Letter B. If you later edit the Letter B file, and then you run a new merge of Letter A, the merged copies will contain the new text from Letter B. (Existing merged letters won't show the new text unless you open each one and update the fields.)


jay freedman say:
If you anticipate making formatting changes in the included file, the INCLUDETEXT field should not have "preserve formatting during updates" checked. (In the code of the resulting field, you should not see a \* MERGEFORMAT item, which is what that check box causes.)

I ran a test with a simple merge main document, where I put in two INCLUDETEXT fields pointing to the same file -- one with a \* MERGEFORMAT in the code and one without it. I ran a merge with plain unformatted text in the included file; then I changed the formatting in the included file and reran the merge. The field with the \* MERGEFORMAT did not show the changed formatting, while the field without the \* MERGEFORMAT item did show the new formatting.

None of this involved any manual updating. The formatting change in the included file won't show up in the main merge document (unless you manually update it there), but the updates will appear in the result of the merge automatically.
