"Excel" (VBA) can never know when you are finished executing OnTime operations.  It is a logical state that only your implementation can determine.


Ordinarily, one solution might be to have a global public variable (e.g. Dim listsDone As Boolean) which you can test in a loop elsewhere.

However, the problem is more complicated when calling VBA procedures in other workbooks.  I believe global public variables are accessible only by code in the workbook where they are declared.

(There might be syntax for accessing variables in other workbooks.  I don't know.)

I would create a function in the "other" workbook (ABC.xls) that returns your state of Update_Lists.

To demonstrate, try the following paradigm (changing workbook names as needed).


In Book2:

Const cntLimit As Long = 5
Dim cnt As Long

Sub startit()
cnt = 0
doit
End Sub

Sub doit()
cnt = cnt + 1
If cnt < cntLimit Then
    Application.OnTime Now + TimeSerial(0, 0, 1), "doit"
Else
    MsgBox "book2 doit done; cnt = " & cnt
End If
End Sub

Function isDone() As Boolean
isDone = (cnt >= cntLimit)
End Function


In Book3:

Sub doit()
Dim cnt As Long
Application.Run "book2!startit"
cnt = 0
Do While Not Application.Run("book2!isDone")
    cnt = cnt + 1: DoEvents
Loop
MsgBox "book3 doit; cnt = " & cnt
End Sub


Both workbooks must be open in the same Excel instance.

Start by executing in Book3!doit.

In Book2:

Const cntLimit As Long = 5
Dim cnt As Long
Dim nxtTime As Double

Sub startit2()
Dim t As Double
Do: DoEvents: t = Timer: Loop Until t <> Int(t)
cnt = 0
doit2
End Sub

Sub doit2()
cnt = cnt + 1
If cnt < cntLimit Then
    nxtTime = Now + TimeSerial(0, 0, 2)
    Application.OnTime nxtTime, "doit2"
End If
End Sub

Function isDone() As Boolean
isDone = (cnt >= cntLimit)
End Function

Function myNxtTime() As Double
myNxtTime = nxtTime
End Function

Function myCnt() As Long
myCnt = cnt
End Function


In Book3:

Dim cnt As Long
Dim st As Double

Sub startit1()
st = Timer
cnt = 0
Application.Run "book2!startit2"
doit1
End Sub

Sub doit1()
Dim nxtTime As Double
Dim et As Double
If Not Application.Run("book2!isDone") Then
    cnt = cnt + 1
    nxtTime = Application.Run("book2!myNxtTime")
    ' delay extra 1 second to avoid a "race condition"
    Application.OnTime nxtTime + TimeSerial(0, 0, 1), "doit1"
Else
    et = Timer
    MsgBox "book2 cnt = " & Application.Run("book2!myCnt") & _
        vbNewLine & "book3 cnt = " & cnt & _
        vbNewLine & Format(et - st, "0.000000") & " sec"
End If
End Sub

Ajeesh wrote:

Use Application.Wait (1000)

1000 means 1 second.


For posterity, that information is incorrect.

The parameter for Application.Wait is a time of day, not an amount of time, much less a number of milliseconds.

Ironically, if we execute Application.Wait 1000, it will return immediately(!) because presumably the system clock is a date later than 26 Sep 1902 (day 1000).

Ostensibly, use Application.Wait Now+TimeSerial(0,0,1) in order to "suspend" [1] the process for 1 second.

However, that really "suspends" the process for up to 1 second.  It, too, might return immediately, depending on how close the system time is to the next second on the system clock.  Note that VBA Now returns time accurate only to the second, not to the 1/100th second as Excel Now does.

If we really want to ensure a delay of at least 1 second, use Application.Wait Now+TimeSerial(0,0,2).  But beware that that might "suspend" the process for

-----
[1] Application.Wait does not really suspend the process, at least not in Excel 2003 and 2007 with WinXP and Win7 on a "PC" (non-Mac).  Instead, it executes a "busy loop", presumably the equivalent of (but obviously not implemented in VBA):  Do Until Now >= endTime: DoEvents: Loop.


[2] Application.Wait accepts a date/time specification with finer granularity than the VBA Now function provides.  The following code comes closest to a delay of as close to 1 second as possible.

(We can never guarantee a delay of exactly 1 second because we are always competing with other processes and interrupt routines for CPU time.)

Dim d As Double, t As Double
Do
    d = Date
    t = Timer
Loop While d <> Date    ' unlikely race condition at midnight
Application.OnTime d + (t / 86400) + TimeSerial(0, 0, 1), "doit"

But of course, Now+TimeSerial(...) is sufficient for long delays like 3 minutes.

john w. vinson say:

It would appear that you have a classic Many to Many relationship: each Parcel may have zero, one, or many Deficiencies; and each Deficiency may be manifested in zero, one or many Parcels. This requires THREE tables (at least): a table of Parcels (with the information specific and unique to a parcel, such as its address, geocoordinates, ownership, etc.); a table of (types of) Deficiencies, e.g.  one record would be "Heaving", another "Cracking", a third "Tree Roots" and so on (I'd use an autonumber DeficiencyID); and a third table modeling the many to many relationship, with a pointer to the ParcelID, another pointer to the DeficiencyID, and possibly others (e.g. text such as "Crack extends across SW quadrant" or "Cavity under pavement is not visibly obvious").

What you have is what's called "Committing Spreadsheet": having data ("Heaving", "Cracking") stored in fieldnames, not as data in your table. That's normal for a spreadsheet but it is JUST FLAT WRONG for a relational database!

john w. vinson say:

No. I'm certainly NOT saying one table for each deficiency. I'm saying THREE TABLES:

Parcels (primary key TaxID)

Deficiencies: primary key DeficiencyID, a two-field lookup table identifying generic types of deficiencies. This is not a deficiency at a particular parcel; it's a TYPE of deficiency, e.g. "Bulge", "Tree root", etc.

DeficienciesObserved (a tall thin table containing the same kind of data as your wide-flat table, but with one ROW per deficiency seen at this parcel, rather than one FIELD per deficiency).

scottgem say:

Let me try to explain this. First, while John is technically correct that you have a many to many relationship, This is handled with a lookup table. So you have your parcel table, and your deficiency table, then you have a lookup table that lists the deficiencies. This is to standardize input. So your Deficiency table is really your junction table. 

John mentions all the Yes/No fields you have like Heaving, Cracking, Tree Roots, etc. I think he is assuming these are the deficiencies. Are they? Using a lot of Yes/No field are a sign of a non normalized database. If those are, indeed, the deficiencies, then you should have one foreign key field to hold the DeficiencyID from your Deficiency lookup table.


John's other point about Committing Spreadsheet is also valid. You have a great many fields. In a relational database, tables are tall and thin, not wide. So the fact that you have so many fields leads us to believe that you are not normalizing properly, hence our advice that you need more than 2 tables. 

Access is not an extension of Excel. It is a different animal entirely and you need to get away from spreadsheet thinking.

The problem here is we don't understand your business model. So we don't know what all those fields represent. So we don't know how to reconstruct your table without a better understanding. But I see a lot of fields where you should probably have a child table or more with a structure like:

RecordID (PK)

ParentID (FK)

MeasurementID (FK)

Measurement

For example, you have several fields about driveways. Each of those fields should be a record in a child table like above where the MeasurementID points back to a lookup table of measurements and the Measurement field is the actual value.
