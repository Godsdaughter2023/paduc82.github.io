Headers and footers are controlled by Sections.  For each Section in the document your Header (or footer) choices are:
  1. All the same, which sounds like what you have
  2. Different First Page, which means that the first page's header (or footer) can be set differently from the remaining pages in the Section.
  3. Odd and Even Pages, which means the left hand page's header (or footer) can be different from the right hand page's header (or footer).
  4. The Different First Page option can be combined with the Odd and Even Pages option.
Word Help has pretty good coverage on the Header and Footer topics.

It would be easier to answer your questions if you would punctuate and capitalize properly.

You are certainly going to need a number of tables, including Customers, Jobs (what you do for a customer), Parts (all parts you use), JobsParts (the parts used in a particular job), and maybe Invoices (depending on how you do your accounting).  Quite likely you will need more tables than just those.  Before you do anything else, you need to figure out exactly what entities (types of things) your database needs to be concerned with, and the characteristics of each entity.  You will normally represent each type of entity with a table (wherein a record represents a particular instance of that entity, as a record in the Customers table represents a specific customer), and the characteristics of each type of entity will be fields in that table.  There will likely be some tables (like JobsParts) that represent complex relationships between entities -- JobsParts represents parts that are use on a particular job.

Only when you have got the table structures figured out will you begin building forms and reports to allow the user to interact with them.

As a novice at Access, I recommend you review some tutorial material.  Access MVP Crystal Long has a nice set of tutorials, both written and video, that may help you a lot.  You can find them here:

     http://www.accessmvp.com/Strive4Peace/

The grand question, "How do I build a database for my business," is really too broad for a forum like this.  But if you read Crystal's tutorials, think hard about exactly how your business works and what are the entities and relationships between them, you can probably build a working application that does exactly what you want.  If you run into snags or have specific questions, feel free to post them as new threads.

dirk goldgar say:
I am trying to create a job costing tool for myself using access. I have started off by making a table for each different each part for ex. nipples, heads, turn seals, pipe. I dont know how to set up a relationship so that I can pull all of these items into another table with all the parts listed. I also wanted to know how I could establish a cost feature and a invoice. Although I don't know the details of your job, you probably don't want to create a separate table for each type of part.  It will be very difficult to bring all those tables together in any useful way.

Instead, have a table of Parts.  In that table, you would have a field "PartType" (maybe), which would identify what type of part this record represents.  You would probably also have a table "PartTypes", keyed by "PartType", listing all the possible distinct types of parts.  Each record in the PartTypes table would containing any information that is true of all parts of that type.

The Parts table would have a field for the cost of the part, as well as the other details describing this particular part (including its type).

Along with Customers, Parts, and PartTytpes, there are many other things your database is likely to need to keep track of, which only you can really identify.  Are these parts used in a job or project for the customer?  If so, you're going to need a Jobs or Projects table, related to a particular customer record.  Also, you'll need a table called, maybe, JobParts, identifying which parts were used in a particular Job.  An Invoice is usually related to a Job and the Parts it used, but needs to be captured and saved separately at the moment of creation, so that later changes in part costs don't affect the invoice amount, and so that (if this is how your busines works) payments can be applied to the Invoice.

As you see, the details of a project like this can be quite complex.  Access can certainly be a good tool for building a database such as you describe, but it will take time for you to build it exactly what you need.  If you have the time and interest to put into it, that is fine, and the helpful people in the Access community will be happy to give you pointers.  Also, it's possible that there is an existing database template on Microsoft's Office website that you can adapt to your needs.  But you should be aware that you may be able to find commercial software that does what you want, or even hire a professional to do the work for you, and that may be more cost-effective if all you are concerned about is your bottom line.

As an Access enthusiast, my approach is always to build the application myself, but I know other people don't necessarily feel the same way about it.

mike middleton say: The Histogram tool determines a frequency distribution for your raw data and prepares a histogram chart. A frequency distribution shows the count of how many data values are in specified intervals of values. Excel refers to the maximum value of each interval as a bin. The histogram is an Excel column chart of the frequency distribution.

If you already have a frequency distribution, i.e., a list of interval maximums with associated counts, you can construct the histogram as a column chart without using the Histogram tool.

If you have the raw data in a column (Input Range) and if you have bins, preferably equally-spaced interval maximums in ascending order, in a column (Bin Range), the Histogram tool can create the frequency distribution and its associated histogram (column chart).

Alternatively, if you have raw data in a column, you could use my free Better Histogram add-in, available at http://www.treeplan.com/download-free-better-histogram-add-in.htm

Looking at your situation, it's not clear to me if you have raw data or a frequency distribution that summarizes the raw data.

Export Citations from Word to End Note - Microsoft Community

turbon5.0 say:

Export Citations from Word to End Note

Hi,

Is there a way that I can export a list of citations that I have in a word document into a new end note library?

Thanks
macropod say:

Hi turbon,

Word can export your bibliography sources in XML format, which you may then be able to import into EndNote. Here's a macro that exports the list as a series of paragraphs in the current document:

Sub SourcesExport()
Dim oSrc As Source, StrSrc As String
With ActiveDocument
  .Range.Text = vbNullString
  For Each oSrc In Application.Bibliography.Sources
    StrSrc = StrSrc & vbCr & oSrc.XML
  Next
  .Range.InsertAfter StrSrc
  .Paragraphs.First.Range.Delete
End With
End Sub

Each paragraph will look like:

<b:Source xmlns:b="http://schemas.openxmlformats.org/officeDocument/2006/bibliography"><b:Tag>The95</b:Tag><b:SourceType>Book</b:SourceType><b:Guid>{A9F88CD0-3296-4707-8F5D-12EAE6681D82}</b:Guid><b:LCID>0</b:LCID><b:Title>The Concise Oxford Dictionary</b:Title><b:Year>1995</b:Year><b:City>Oxford</b:City><b:Publisher>Clarendon Press</b:Publisher><b:RefOrder>1</b:RefOrder><b:Edition>Ninth</b:Edition></b:Source>
