    ken sheridan say:
    =[cmbcodepcg].[column](1)

    That's a much more efficient solution, but, for the record, in situations where a DLookup function call is used note that where both controls are in the same subform you don't need to reference it via the parent form's subform control.  Simply reference the combo box.  If the Code PCG column is a number data type you could use:

    =DLookup("[Intitulé]","[PCG]","[Code PCG]= " & [cmbcodepcg])

    If it's a text data type:

    =DLookup("[Intitulé]","[PCG]","[Code PCG]= """ & [cmbcodepcg] & """")

    Another efficient solution would be simply to include the referenced PCG table in the subform's RecordSource query and bind a control to the Intitulé colum, setting its Locked property to True and its Enabled property to False to make it read-only.  This latter approach also enables rows in a subform to be ordered by a column from the referenced table, which is often desirable where the subform's table is a one which models a many-to-many relationship type, e.g. an OrderDetails table, where the inclusion of a referenced Products table would enable the subform's rows to be ordered by product name for instance.

    You'll find an example of this latter approach in InvoicePDF.zip in my public databases folder at:

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

    In this little demo file's invoices form the Invoices table is joined to four referenced tables in a set of linear relationships to return the customer name and all their address data in bound text boxes in the form.  The invoice details subform's query includes the Products form in order to order the rows by product name as described above.
