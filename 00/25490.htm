ken sheridan say: Firstly, a relational database table is a set, so the concept of 'next record' in the table is meaningless.  A row can follow another row in a sort order of the set, so you need to define the basis for the sort order before you can determine the 'next' row in that sort order.

However, having defined the basis for the sort order, there is no need to assign duplicate values to another row, and to do so would introduce redundancy and the consequent risk of update anomalies.  The values can be returned in a query which joins two instances of the table, and returns the Depart_ values from a row in the second instance as the Arrive_ values, where the row in the second instance has the MAX value of whatever is the basis for the sort order where this is less than the value of the same column(s) in the current row from the first instance of the table.  NB: whatever is the basis for the sort order must be a set of distinct values.

ken sheridan say: The trouble is that you are duplicating the same data in separate rows of the table.  This is not good design as, due to the redundancy, if any of those values changes the data is inconsistent until the appropriate change is made in the other row. This is an update anomaly in the jargon of the database relational model.

What you are attempting of course is to cater for this by executing automatic updates.  This might seem a reasonable approach, but it's not how a well designed relational database should work.  A good design eliminates the update anomaly by normalizing the table.

The fundamental issue here is that storage of the data is being confused with presentation of the data.  Presentation of the same data twice, as you currently have it, is fine.  Storage of that same data twice is not.  However, there is no conflict between the two; the data can be entered and stored once, but presented twice in the way you currently have it.

When inserting a row into the table you would insert the arrival location, the arrival date and time, and the departure data and time.  You would not enter the departure destination or the arrival data and time at that destination, however, as these columns would not exist in the table.  Those values would be entered into another row in the table as the arrival location etc.  They can still be shown in the preceding row in the sort order (arrival date/time) by joining two instances of the table.  So as each arrival location is inserted the data will show as departure data in the previous row in the sort order.  Consequently the redundancy is eliminated and the database, in this regard at least, is safe from update anomalies, but the data is nevertheless presented in your current format.

That sounds simple enough.  Well, yes and no.  The flies in the ointment are firstly the obvious one that you need to possess the SQL skills to write the query which will do this.  Such a query is not trivial, and while I'd be confident of being able to write it, I could not do it off the top of my head with any degree of confidence on the information available from your screenshot.  I'd want to know the full and exact definition of the table in question.

The second point is that a query to do this would not be updatable, so would not be the basis for a bound data entry form.  The simple solution would be to have one form for data entry, showing only the arrival data in each row, and another form, in the format of your current form, based on a query.  A better solution, however, would be to have a form, in continuous forms view, bound to the query, and another form, in single form view, bound to the table only.  By embedding the  continuous form as a subform in the other form, data could be inserted into one form and the second form then requeried so that it shows the newly inserted data.  The appearance would be similar to a split form, and it would be possible to simulate the behaviour of a split form so that you could use the continuous form for navigating to a row in the parent form.  This would mean a rebuilding of the interface, but allowing the interface to determine the design of the table, as is currently the case, is putting the cart before the horse.

It's now your judgement call in the light of what I've said.  Do you stick with the flawed model and try to work round its faults in the way you've proposed, or do you recast the model and make the consequent changes to the interface?

ken sheridan say: I assume that there is some foreign key column in the table which identifies the set of rows which constitute a 'trip'.  Let's call this TripID for the following example.

What you need to identify is the row with the same TripID as the current row and the Arrrive_Date+ArriveTime value which equals the Dest_Arrrive_Date+ArriveTime value in the current row. If such a row exists then you need to execute an UPDATE statement, if not an INSERT INTO statement.  This can be done with code in the form's AfterUpdate event procedure, which covers both the insertion of a new row and the amendment of an existing row:

    ' declare a variable, which has to be a Variant
    ' as it will be Null if no later row exists
    Dim varDestination As Variant
    
    Dim strCriteria As String
    Dim strSQL As String
    
    ' build a string expression of the criteria to find the next row, if any
    ' note the use of the ISO standard notation for date/time to ensure
    ' international unambiguity
    strCriteria = "TripID = " & Me.TripID & " And Arrrive_Date + ArriveTime > #" & _
        Format(Me.Dest_Arrrive_Date + Me.Dest_ArriveTime, "yyyy-dd-mm hh:nn:ss") & "#"
    
    ' call DLookup function to determine if next row exists
    varDestination = DLookup("Arrive_Destination", "tbl_Main", strCriteria)
    
    ' if row exists then build an UPDATE statement
    If Not IsNull(varDestination) Then
        strSQL = "UPDATE tbl_Main " & _
            "SET Arrive_Destination = """ & Me.Depart_Destination & """, " & _
            "Arrive_State = """ & Me.Depart_State & """, " & _
            "Arrive_Time = #" & Format(Me.Dest_Arrive_Time, "hh:nn:ss") & "#, " & _
            "Arrive_Date = #" & Format(Me.DepartDate, "yyyy-mm-dd") & "# " & _
            "WHERE " & strCriteria
    Else
        ' build an INSERT INTO statement
        strSQL = "INSERT INTO tbl_Main(Arrive_Destination,Arrive_State,Arrive_Time,Arrive_Date) " & _
            "VALUES(""" & Me.Depart_Destination & """, " & _
            """" & Me.Depart_State & """, " & _
            "#" & Format(Me.Dest_Arrive_Time, "hh:nn:ss") & "#, " & _
            "#" & Format(Me.DepartDate, "yyyy-mm-dd") & "#)"
    End If
    
    ' execute the SQL statement
    CurrentDb.Execute strSQL, dbFailOnError
    
    ' requery form to include new row if inserted
    Me.Requery

Go to File -> Options and choose Formulas.  At the top in the Calculate options section there is an item "Enable iterative calculation".  Check that box.  Basically that allows circular references.  For a use like you've described you would want to set Maximum Iterations to 1.
