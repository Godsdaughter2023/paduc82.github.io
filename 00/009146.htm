ken sheridan say: The first thing to point out is that the referencing table into which data is being entered via the form should have a foreign key column referencing the sub-standards table only, and no column referencing the standards table.  The standard is functionally determined by the sub-standard, so to have a foreign key column referencing the former would introduce redundancy due to the standard being transitively dependant on the key of the table via the sub-standard column.  The table would not be normalized to Third Normal Form (3NF) which requires all non-key columns to be determined solely by the whole of the table's primary key.

You can still do what you wish in the context a normalized design, however.  You'll find an example as ComboDemo2K.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

The relevant option in this little demo file your case would be that for a simple 2-level hierarchy.

The way to do it is base the form on a query which joins the referencing table into which data is being entered to the sub-standards table and that to the standards table.  You then use hybrid controls for the standard and sub-standard columns, as in my demo, each being a text box superimposed on a combo box so as to appear to the user as a single comb box control.

For the standard the text box would be bound to the Standard column from Standards, the combo box unbound.   For the sub-standard the text box would be bound to the Sub-standard column from Sub-standards, the combo box bound to the foreign key substandard column in the referencing table.

The RowSource property of the combo box bound to the foreign key substandard column in the referencing table should reference the Standard unbound combo box as a parameter in the same way as that for the cboParish control in my demo references the cboDistrict combo box.  

The combo box bound to the foreign key substandard column should be requeried in the AfterUpdate event procedure of the Standard unbound combo box.

Both of the text boxes should have their Locked property set to True (Yes) and their Enabled property to False (No) to make them read only.

ken sheridan say: You can see how the tables are related in my demo in the relationship window.  The model is like this:

Counties---<Districts---<Parishes---<Locations

where the < character indicates the 'many' side of each one-to-many relationship type.

As regards the lack of queries, bear in mind that a query need not be a saved querydef object, it can be an SQL statement.  In my demo I've followed my usual practice of using SQL statements as the RecordSource properties of the forms.  In the case of the continuous form for the simple two level hierarchy it's:

SELECT Locations.*, Parishes.Parish,
Districts.District, Districts.DistrictID
FROM (Districts INNER JOIN Parishes
ON Districts.DistrictID=Parishes.DistrictID)
INNER JOIN Locations
ON Parishes.ParishID=Locations.ParishID;

whereas In the case of the continuous form with the hybrid controls for all three levels of the hierarchy it's:
 
SELECT Locations.*, Parishes.Parish,
Districts.DistrictID, Districts.District,
Counties.CountyID, Counties.County
FROM Counties INNER JOIN ((Districts
INNER JOIN Parishes
ON Districts.DistrictID=Parishes.DistrictID)
INNER JOIN Locations
ON Parishes.ParishID=Locations.ParishID)
ON Counties.CountyID=Districts.CountyID;

Note that in each case only the non-key Parish column from parishes is returned, not the ParishID key column, as this is a foreign key in Locations, but in the case of the higher levels of District and County both the key DistrictID and CountyID columns are returned as well as the non-key District and County columns.  It's the non-key text columns to which the text boxes over the combo boxes are bound.  Only the parish combo box is bound, however, to the ParishID column from locations, the district and county combo boxes are unbound.

rob koch say:

It isn't over your head rockydean, you're just not wanting to understand it as with most consumer users, so disabling CFA may be the only thing you can do.

I'll try once more by saying that if you know what a folder is, also known as a directory in the past, you can easily understand what Controlled Folder Access (CFA) does.

Since a folder is just a container for files, CFA is simply a group of settings that enable or disable the ability for programs to write into the "Protected folders" list which can be viewed using that hyperlink.

When CFA protection is turned On, the folders in the protected list can't be written into by programs by default, so you must add them to an exemption list using the "Allow an app through Controlled folder access" setting if they need to.  This is done by clicking "Add an allowed app" and then simply browsing to the trusted program's executable that displayed the warning message in the popup notification.

As I stated earlier, if this is too much for you then all you typically need to do is just turn Off CFA as you already have while installing any new program you trust and then return to turn CFA On again once the installation is completed.  Few programs are actually causing these warning messages to display while operating, though they do exist and would then either need to be added to the Allow an app through CFA setting I just described above or simply turn CFA completely Off at that point if you can't figure this out.

Yes, you are opening your system up to attacks by ransomware that might otherwise be stopped by CFA if you turn it Off, so simply turning it Off before installing new programs and back On after is worth trying, since most people install new programs quite rarely.  The other thing that will usually keep these messages from displaying is to uncheck the selection to "add a desktop icon" while installing, since this option is what most often causes these CFA messages to display.

1. From Internet Options> General > Browsing History> Delete (next screen) uncheck Preserve Favorites website data then check and delete Temporary Internet Files, History and Cookies.

2. Shut down/power down the computer, modem and router for about 5 minutes then restart/power up.

3. With the warning that a reset will delete AutoComplete data, such as passwords and you will have to reinstall or re-enable browser add-ons, use the Reset IE Settings button in Internet Options> Advanced.

Useful for recovering stored passwords before you clear all stored login data:
IE PassView- NirSoft:
http://www.nirsoft.net/utils/internet_explorer_password.html

What a reset of IE settings will do:
Windows Vista Help- Reset Internet Explorer settings:
http://windowshelp.microsoft.com/Windows/en-US/Help/81fe3b46-2460-459e-b826-8f134a7fe1771033.mspx

4. From Start> All programs. Accessories, right-click Command Prompt and choose "Run as administrator". At the flashing cursor type in ipconfig /flushdns (note the space between config and / ) hit [Enter] then Restart.

There are additional troubleshooting steps in this article:
Error "Internet Explorer cannot display the webpage" when viewing a website in Internet Explorer:
