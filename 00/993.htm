shyamal debnath msft support say:

Hi Eoin,

Thanks for the update. Here is the basic descriptions and configuration requirements about both the method follows.

SMTP client submission:

• If your device or application can authenticate and send email using an Office 365 mailbox account, this is the recommended method.

• In this scenario, the device or application uses an email account to send email to recipients just like an email client.
• If you want to send email from a third-party hosted application or service, you must use SMTP client submission.
• If your device or application supports or requires user name and password authentication, consider the SMTP client submission method instead of SMTP relay.

SMTP relay:

• This method allows Office 365 to relay emails on your behalf by authenticating using your public IP address (or a certificate). This requires a connector to be set up for your Office 365 account. (Might be the complexity of configuring a connector, SMTP client submission is the easiest over Office 365 SMTP relay).
• However, in this method your device or application can send email using any address (including ones that can't receive mail), as long as the address uses one of your Office 365 domains. The email address doesn’t need to be associated with an actual mailbox.

Moreover, you may check the summary of table inside the article it will help you decide which one of these method options will meet your business needs.

If anything is unclear, please feel free to let us know and we will be glad to provide further assistance.

Best regards,
Shyamal

Compatibility View may be telling websites you're running an older version of Internet Explorer.

 

Check these settings:

IIf Statement - Checking two values - Help

I am trying to write an IIF statement that checks two values. I have a value in two columns that I want to return a certain color code.

Here is my example:

CONDITION

Color

REPAIRABLE BAD BOTH ENDS WH/RD

If I just check the CONDITION, by using this function:

Everything works great, but I would like to check both CONDITION and TYPE to return the COLOR value.

I thought I could write something like this:

IIF([Condition]="Prime" & [Type]="NO DEFECTS", "White", IIF([Condition]="Prime" & [Type]="NO HEAT", "WH/GR",...

I realize that this will end up being a long IIF statement and maybe there is a better way to get my end result but I cannot combine Condition and Type because I use these as values for other reports.

I hope I provided enough information here to for you to help me without taking up to much of your time. If I didn't and you still would like to help please email me. Thanks inadvance for you time.

john w. vinson say:

To directly answer your IIF question, you're close: the problem is that the & operator is used in Access to concatenate two strings. It is not at all equivalent to the logical operator AND. Your IIF would work if you phrased it

IIF([Condition]="Prime" AND  [Type]="NO DEFECTS", "White", IIF([Condition]="Prime" AND [Type]="NO HEAT", "WH/GR",...

However, your concern about the long IIF is well taken - this is not at all the best way! I would suggest instead using a table-driven approach. Create a new table tblColors just like your example, with three fields. You can then create a Query by joining your existing table to tblColors, joining on both the Condition and Type fields - drag Condition to Condition, and Type to Type. This will let you display the color for that combination; and you can easily add new rows to tblColors to handle other combinations.

ken sheridan say: You should never encode data in an expression like this.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1).  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

In fact you have shown how to correctly represent this with your example.  You should have a table with the columns Condition, Type and Color.  This table should contain all possible combinations of the Condition and Type values, with the appropriate Color value for each.  The primary key of the table is a composite one made up of the Condition and Type columns.  The Color value for each record in the referencing table can then be shown in a number of ways:

1.  In a bound control in a form or report based on a query which joins the referencing table to the new table on Condition and Type.  This is probably the most efficient method.

2.  In an unbound control which shows the value by means of the DLookup function, passing the values of the Condition and Type control in the form or report.

However it's done there should be no Color column in the referencing table, only in the new referenced table.  An enforced relationship between the tables on Condition and Type should be created.
