What you need to do is, starting with a blank document that is in the same orientation as the certificate, insert Content Controls in the positions required so that the text that you enter into them will land in the correct position when you print that document onto a certificate.  To avoid wasting certificates, just print to plain paper and compare the output to a certificate until you get the positioning correct.

brian tillman [outlook mvp 2007-2019] say:

When you say "MS Outlook", I'm assuming you mean Outlook.com, Microsoft's mail service replacing Hotmail.  Usually, "MS Outlook" means Office Outlook, a mail program you can install on your PC that can be used to access mail services like Gmail.  So it's not a choice between Outlook and gmail.  Outlook can access gmail.

The only way you'll be able to see both your Outlook.com and Gmail messages in a single Inbox is if you either forward your messages from gmail to your Outlook.com address (or vice versa) or if you configure gmail to fetch your Outlook.com messages from that mailbox via POP (or you can go in the other direction instead).

It is not possible at this time for the one service to see the folders in the other service.  This would require IMAP access between the services and that's not possible at this time.  You can, of course, connect to both Outlook.com and gmail with the Office Outlook mail client, which will allow you to see folders from both services in Office Outlook, but you can't combine them into a single folder set.

julie fka profjulie say:

The difference between GPS software that needs a data connection and GPS software that doesn't need a data connection has to do with where the maps are stored. 

 

GPS services such as Google Maps and the Windows Live option require a data connection (and this data connection could be either through GPRS/EDGE/3G or Wifi) because the maps are stored on the web.  The GPS radio will still work without the

 

GPS services provided by TOMTOM and other such applications do not require any kind of data connection because the maps are stored locally (on an installed SD card).

 

The two different COM port settings you refer to have to do with whether you are connecting your device to an external GPS device versus connecting to an internlly installed GPS radio.  Since you are simply trying to activate the GPS radio that is already installed in your device, you only need to be concerned with setting the correct hardware COM port and baud rate (you should be able to find these settings in your user documentation).  There is nothing you should do with or set in regards to the Program/software COM port unless you are connecting to an external GPS device.

 

Once you know the hardware COM port that is used by the GPS radio in your device, you will need to reference this COM port the application connection options of the software you are using.

The first problem can be addressed as follows:

Dim xlApp As Object
Dim xlBook As Object
Dim dSource As String
Dim ExcelWasNotRunning As Boolean
Dim fd As FileDialog
Const strFolder As String = "C:\Users\JohnDoe\Dropbox\"

Set fd = Application.FileDialog(msoFileDialogFilePicker)
With fd
    .Title = "Select the file that you wish to use."
    .InitialFileName = strFolder
    .Filters.Clear
    .Filters.Add "Excel Files", "*.xls?"
    .AllowMultiSelect = False
    If .Show = -1 Then

    Else
        MsgBox "You cancelled the operation"

    End If
End With
On Error Resume Next
Set xlApp = GetObject(, "Excel.Application")
If Err Then
    ExcelWasNotRunning = True
    Set xlApp = CreateObject("Excel.Application")
End If
On Error GoTo 0
With xlApp
    Set xlBook = .Workbooks.Open(dSource)
    MsgBox "The workbook" & vbCr & dSource & vbCr & "is open!"

    'Insert code to do what you need to do with the workbook here

    xlBook.Close SaveChanges:=True
    Set xlBook = Nothing
End With
If ExcelWasNotRunning Then
    xlApp.Quit

Set xlBook = Nothing
Set xlApp = Nothing
Set fd = Nothing

jeeped say:
Excel attempts to interpret data to numerical values when it comes from a text file. It expects data to come in according to a limited number of formats (e.g. MDY, DMY, YMD, MYD, DYM or YDM). Yoru 'saved' data is not including the year (according to Excel's import correction routine) so Excel is attempting to construct an actual date using 2014 as the default year. Solution: include a day, month and year when saving to CSV so that the resulting text value can be interpreted by Excel when re-imported correctly. Best to use the system default of dd/mm/yyyy vs mm/dd/yyyy to avoid more complications.

joeu2004 say:

Jake wrote:

My date cells in a CSV are formatted to display mon-YY, so Oct-77 is October 1977, for example. When I enter 04/01/1925, it appears correct at first as Apr-25. However, when I save, close, and reopen the CSV the format and actual input value has changed to 25-Apr (DD-mon) with a value of 04/25/2014 (which if I reformat to mon-YY is Apr-14). How can I fix this? I have never had the program unilaterally change actual values before. 

If you look at the CSV file using Notepad or the equivalent, you will see that the problem is not that dates are change "upon save".  Instead, the problem arises when you open the CSV file.

When you save as CSV, values are written as they are displayed.  So the file contains Oct-77 and Apr-25.

When you open a CSV file, Excel interprets the text exactly as if you had typed it.

Apr-25 is interpreted as 25 April 2014 because 25 looks like the day of month in that context.  If you type Apr-25 into a cell manually, you will see that Excel interprets it as 25 April 2014.

In contrast, Oct-77 is interpreted as 1 October 1977 because 77 cannot be interpreted as the day of month.

Off-hand, I don't know of any way to coerce Excel to interpret Apr-25 and month-year.  I tried using Get External Data > From Text and selecting MYD in the last dialog box, to no avail.

If you reformat date cells in some format that displays month, day and year, then save as CSV, Excel should interpret the CSV data corrrectly, at least on a computer with the same regional settings.  But then you would have to reformat the cells as Custom mmm-dd after opening the CSV file.

Alternatively, you might consider writing VBA code to interpret the CSV with Apr-25 in the manner that you require.

Not a "great" solution, I know.

I expected the same thing, so I thought my explanation covered the situation ("written as they are displayed").

But it appears we are both wrong.

When we enter 12/31/2014 into a cell and purposely reset the column width to the default, we see "########" in the cell.

When we Save As ".csv", close the Excel file and open the file in Notepad (or the equivalent), we see 12/31/2014.

When we open the CSV file in Excel, we see "########" in the default-width cell.  But the Formula Bar shows the correct date (using its own method of formatting), namely 12/31/2014.

-----

In contrast, when the original Excel cell displays "########" because of an error (negative time, for example) and we save as ".csv", "##...##" (ad nauseum) is indeed written into the CSV file.

(Note:  255 "#" characters are written to the CSV file even if fewer "#" characters were displayed!)

So when we open the CSV file in Excel, we see a cell with a text value (ISTEXT is true) of 255 "#" characters.
