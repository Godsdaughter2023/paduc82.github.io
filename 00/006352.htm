ken sheridan say:

That's what I thought initially, but somethin I read lead me to believe that I should be doing it the other way.

What you have read might have been referring to the modelling of types and sub-types.  This would be a more valid model if the attributes of owners and tenants differ, in which cases owners and tenants would be regarded as sub-types of type contacts.

A sub-type is characterized by sharing all attributes of its (super) type, but not those of other sub-types.  So in your case the Type contacts (or whatever you wish to call it) would be modelled by a table like this:

Contacts
....ContactID  (PK)
....FirstName
....LastName
....etc

Each column in this table would represent a attribute which is shared by both tenants and owners, both of whom have first and last name attributes of course.

The two sub-types would be modelled by Tenants and Owners tables.  The relationship type between each of these and Contacts would be one-to-one, i.e. the primary key of each would also be a foreign key referencing the primary key of Contacts.  It follows from this that the primary keys of these two tables cannot be autonumbers of course.  The non-key columns in each case would be a distinct set of attributes which are applicable in one instance to tenants only, and in the other instance to owners only.

A type/sub-type model like this avoids the need for Nulls (which can be semantically ambiguous) in a single Contacts table, where some columns would be inapplicable to tenants and others inapplicable to owners.  A person might of course be both a tenant and owner, in which case they would be represented by a row in both tables, along with one in the referenced  table Contacts.

The many-to-many relationship type between contacts and properties would be modelled by a table which resolves it into two one-to-many relationship types, Daniel's Properties_Contacts table being an example.  This table is 'all key', i.e. its primary key is a composite one made up of all three columns.  It might have other non-key columns of course, representing other attributes of the relationship type e.g. the date of commencement of ownership or tenancy of the property.  BTW a ContactType column in such a table should be a foreign key referencing the primary key of a ContactTypes table.  This protects the integrity of the database.

On the subject of relationship types in general, you might like to take a look at DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In this little demo file, note in particular how address data is decomposed into separate referenced tables.  Again this protects the integrity of the data by ensuring that all tables are correctly normalized.  The contacts form in the section on 'Entering data via a form/subforms' illustrates how data is entered in the context of a set of normalized tables.

brian tillman [outlook mvp 2007-2019] say:
In order to have the ability to push your data up to the Outlook.com server, you will need to either create an IMAP account for accessing the Outlook.com server (see the "Apps that support IMAP and SMTP" section at http://windows.microsoft.com/en-us/windows/outlook/send-receive-from-app) or download and install the Outlook Hotmail Connector (see http://support.microsoft.com/kb/2735989) and configure the account there.

Either method will create a new set of folders matching the folders of your Outlook.com mailbox.  You can then copy the data in the PST to the Outlook.com folders in Outlook, removing the PST when you finish.  This is a manual process.  If the folders in the PST are NOT the standard Outlook folders (Inbox, Outbox, Drafts, etc.), you should be able to copy the folders themselves and all the data in them should transfer.  If a folder is one of the default, open the folder and copy its contents, not the folder itself.

bob jones aka: cybertaz say:
Nope -- What Derek is referring to is that prior to Excel 2011 there was a Color choice in Excel's Preferences.

Before 2008 it allowed for replacing any of the Standard Colors, Chart Fills & Chart Lines with any colors you preferred.

In Excel 2008 the Color Preference was reduced to customizing Standard Colors only... Chart Fills & Chart Lines were excluded.

For some reason the Color Preference was removed altogether in Excel 2011. [My guess is that it relates to the incremental implementation of the new drawing engine.]

The Theme Colors imposed by Themes have no bearing on the default Fill Color the button is set to by default... which for some reason is bright Yellow. The Theme Colors control only the default fills for Shapes, Word Art & Smart Art Objects.

BTW Jimmy -- I thought you had it straight by now :-) Custom Themes [a user's choice of existing Theme Colors & Theme Fonts] can be created in any of the Office apps. It's custom Theme Color schemes that can be created only in PowerPoint.
