ken sheridan say:

You don't really need a subform at all for this; you can put the unbound combo boxes in a form bound to the CombFinalSignoffQRY query, in its header if the form is in continuous forms view.

The RowSource properties of the Region, Business  and CreditSubStat combo boxes need to reference the next combo box up in the hierarchy e.g.

SELECT Region FROM Regions WHERE Country = Form!cboCountry;

SELECT Business FROM Businesses WHERE Region = Form!cboRegion;

SELECT CreditSubStat FROM CreditSubStats WHERE Business= Form!cboBusiness;

Note the use of the Form property in the above rather than fully referencing the form as a member of the Forms collection.  This is possible as the controls are in the form being referenced.  In fact the control name alone can be referenced, but I prefer to qualify it with the Form property to avoid any possible ambiguity as to what is being referenced.

Then in the AfterUpdate event procedure of cboCountry set the values of the controls lower in the hierarchy to Null and requery them, and the form:

Me.cboRegion = Null
Me.cboBusiness = Null
Me.cboCreditSubStat = Null
Me.cboRegion.Requery
Me.cboBusiness.Requery
Me.cboCreditSubStat.Requery
Me.Requery

Similarly in cboRegion's AfterUpdate event procedure:

Me.cboBusiness = Null
Me.cboCreditSubStat = Null
Me.cboBusiness.Requery
Me.cboCreditSubStat.Requery
Me.Requery

and in cboBusiness's AfterUpdate event procedure:

Me.cboCreditSubStat = Null
Me.cboCreditSubStat.Requery
Me.Requery

The CombFinalSignoffQRYquery will need to reference the controls on the form as parameters, testing each for a value or Null.  Or, if the query is being used for other purposes you may want to create another query for the form's RecordSource in which you include the references to the parameters, e.g.

SELECT *
FROM CombFinalSignoffQRYquery
WHERE (Country = Forms!YourForm!cboCountry
OR Forms!YourForm!cboCountry IS NULL)
AND (Region = Forms!YourForm!cboRegion
OR Forms!YourForm!cboRegion IS NULL)
AND (Business = Forms!YourForm!cboBusiness
OR Forms!YourForm!cboBusiness IS NULL)
AND (CreditSubStat = Forms!YourForm!cboCreditSubStat
OR Forms!YourForm!cboCreditSubStat IS NULL);

By parenthesizing each OR operation these evaluate independently of the AND operations so in effect 'optionalize' each parameter.  Consequently as a selection is made in each combo box the form will progressively be restricted to the matching rows.

 

Ken Sheridan, Stafford, England

ken sheridan say: You only really need to use a subform if its linked to the parent form.  Normally this would be where the parent form is based on a table related one-to-may to the subform's underlying table.  It is possible to link a subform to an unbound parent form by specifying the unbound control(s) in the parent form as the LinkMasterFields property, but that would require all the relevant controls in the parent form to have values, so you can't 'drill down' through the hierarchy in the way the approach I suggested allows.

The main points are:

1.  The form is bound to the query, i.e. the query is its RecordSource property, and will include controls bound to the relevant columns in the query.

2.  The query references the unbound controls in the form as parameters, and in each case tests for the parameter being NULL.  This is how it enables you to drill down through the hierarchy, as if a parameter IS NULL this part of the WHERE clause evaluates to TRUE for every row, so only where a parameter has a value will it restrict the rows returned by the query.

3.  The controls  cboRegion, cboBusiness and cboCreditSubStat are unbound.

4.  As a value is selected in each control, starting at the topmost level of the hierarchy with cboRegion then the controls representing the lower levels of the hierarchy, cboBusiness and cboCreditSubStat, are set to Null and requeried.  This process is repeated as you progress down through the hierarchy.  At each selection the form itself is progressively requeried, so as a selection is made in each combo box the form updates to show only those rows for the value selected, first the region, then the business and so on.

5.  The unbound combo box controls only need to go in the form's header if the form is in continuous forms view.  This is simply to keep them visible at the top as you scroll down through the records.  If the form is in single form view then the unbound controls can just as well go in its detail section and be positioned anywhere you like in the section.
Ken Sheridan, Stafford, England

Hi, my VBA code for a budget spreadsheet has a Worksheet_Change procedure with Intersect and target as part of it.

When users copy and paste a cell using their mouse to Edit, Copy then Edit, Paste, the spreadsheet freezes. In these scenarios the copy paste actions are not within the target, so the two Ifs solve to 'Nothing', so most of the code is skipped. Yet it still freezes when using the mouse.

When it freezes, using the mouse, the only way I can get out of it (other than closing Excel), is to hit F5 GoTo, then choose a reference that is on another sheet. It will change to that sheet. I can then come back to the first sheet and continue as normal.

townfield say:

Bill / Harold,

I think I have got to the bottom of this one - there are 3 issues here...

1. As noted above the maro leaves ScreenUpdating set to false

2. From the tests I have done, it looks like Excel will usually reset ScreenUpdating to TRUE all on its own - unless there are a lot of changes made to the sheet - e.g. delete a row or two

3. Harold's Change macro does not appear to have protection against being re-entered if the marco itself changes the content of the sheet...

Try this macro...

Private Sub Worksheet_Change(ByVal Target As Range)
'
Application.ScreenUpdating = False
Range("B1").Value = Range("B1").Value + 1
End Sub

If you change ONE cell on the sheet which has this Change macro, the count will (unexpectedly) be 96 or 97.  Sometimes the screen freezes, sometimes it does not.

I suggest that this is due to the macro being called again (and again) when the value in B1 is incremented by the Chnage macro.  Somewhere inside Excel this goes up a loop and then decides that it has done enough /  its job and it exits.  Usually ScreenUpdating is TRUE on exit, other times it is not.  If you try to test the value of this in the immediate window it is reported to be false and then (all on its own) flips to TRUE (I kid you not).

The behaviour is variable - on one test, deleteing a row generated an "Out of stack space" error.

Conclusions...

1. If you want to change cell vaules on the sheet to which the Change maro belongs, you MUST make the code non-renterant - see below

2. If you 'manage' system parameters such as ScreenUpdating on entry to a maro, you must remember to set them back to something sensible on exit.

3. When the screen was left frozen I suspect that a 'quiet' stack over flow occurred leaving ScreenUpdating set to FALSE.

 

Suggested full solution... code to be placed  on the worksheet object

 

'Control flag variable that exists outside of the macro to inhibit re-entry

PUBLIC blnChangeActive as boolean

 

Private Sub Worksheet_Change(ByVal Target As Range)
'

'Look to see if this code is already active, if it is go away!
if blnChangeActive then Exit Sub

'Set the macro is active flag so that changes made to this sheet in this macro do not
'cause this macro to run again
blnChangeActive = true

'Inhibit screen updating to make the code run faster and avoid screen flashing
Application.ScreenUpdating = False

'
' This is where the Change macro code goes... whatever you want to do!
'

'For example my cell value incrementer...
Range("B1").Value = Range("B1").Value + 1

'Enable screen updating again
 Application.ScreenUpdating  = True

'Remove the reentry lock
blnChangeActive = false

End Sub

 

The above macro results in the value of B1 being increment by only 1 - as might have been expected!

HTH and fully answers the problem.
