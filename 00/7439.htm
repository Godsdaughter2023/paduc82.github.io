ken sheridan say: That's because the computed Expr1 column is taken from the qryAttendanceData query, not from the tblStudentData table.  The column from the query will be Null for each 'perfect' student because of the lack of matches in tblStudentAttendanceData.  You can overcome this by changing the references in the cross tab query to 'qryAttendanceData.Expr1' to the expression 'tblStudentData.[Last_Name] & ", " & 'tblStudentData.[First_Name]'.  But the result table will not include any counts for these students as counting Nulls gives you Null.  This may be what you want, but if not I've tried to put forward a solution below which gives you at least some indication that these are students with perfect attendances.

As far as I can see the root of the problem is that those students with perfect attendance are not recorded as such per se, but 'perfection' is implied by the absence of rows in tblStudentAttendanceData.  I have to confess that this sounds somewhat illogical to me, as I'd have expected a perfect attendance record to be represented by rows in tblStudentAttendanceData for all possible attendance dates.  However, this apparently not being the case, probably the best you can do is change the original qryAttendanceData so that it returns all students regardless of matched, by means of LEFT OUTER JOINS, and using the NZ function to substitute values for the Nulls which would otherwise be returned at the attendance data and attendance code column positions where there are not matches.  Try something like the following:

SELECT
    tblStudentData.Student_Number,
    tblStudentData.Home_Room,
    [Last_Name] & ", " & [First_Name] AS FullName,
    tblStudentData.First_Name,
    NZ(tblAttendanceCode.AttCode,"Uncoded") AS AttCode,
    NZ(tblStudentAttendanceData.Att_Date,"All dates") AS Att_Date,
    NZ(tblStudentAttendanceData.Att_Comment,"Perfect attendance") AS Att_Comment
FROM (tblStudentData LEFT JOIN tblStudentAttendanceData
ON tblStudentData.Student_Number=tblStudentAttendanceData.Student_Number)
LEFT JOIN tblAttendanceCode ON tblStudentAttendanceData.Att_Code=tblAttendanceCode.AttCode
ORDER BY tblStudentData.Home_Room, [Last_Name], [First_Name];

You could then base your original crosstab query on this, but note that I've changed the default and rather meaningless Expr1 computed column name to FullName.

While this should ensure that all students are returned the values at the intersections in the crosstab query's result table will be rather inconsistent as for those students with an attendance record the value will be the number of attendances per code recorded, whereas for those with a perfect attendance record the values will, I think, be 1 for the 'Uncoded' column heading, so there are no counts of attendances as such for the 83 students.  But without any data in the base tables to record their attendances there cannot be any counts of course, i.e. the problem stems from the way in which the attendances are recorded.

ken sheridan say: Peggy:

The counts could be done quite easily in the report by grouping by attendance code, but the problem would be that the codes would not be arranged  in columns in the same way as those returned by a crosstab query.  There is an alternative way you can count the absences per code in a query, given that there are a fixed set of codes, which is to conditionally sum the return value of an expression using the IIF function, which returns 1 or 0 depending on the value of the code.  In the example below I'm assuming that the AttCode is a text data type with values Code 1, Code 2, Code 3 etc, but you should be able to amend it easily to use the real values:

SELECT
    tblStudentData.Student_Number,
    tblStudentData.Home_Room,
    Last_Name & ", " & First_Name AS FullName,
    SUM(IIF(NZ(AttCode,"") = "Code 1",1,0)) As [Code 1],
    SUM(IIF(NZ(AttCode,"") = "Code 2",1,0)) As [Code 2],
    SUM(IIF(NZ(AttCode,"") = "Code 3",1,0)) As [Code 3],
    SUM(IIF(NZ(AttCode,"") = "Code 4",1,0)) As [Code 4],
    SUM(IIF(NZ(AttCode,"") = "Code 5",1,0)) As [Code 5],
    SUM(IIF(NZ(AttCode,"") = "Code 6",1,0)) As [Code 6],
    SUM(IIF(NZ(AttCode,"") = "Code 7",1,0)) As [Code 7],
    SUM(IIF(NZ(AttCode,"") = "Code 8",1,0)) As [Code 8],
    SUM(IIF(NZ(AttCode,"") = "Code 9",1,0)) As [Code 9]
FROM tblStudentData LEFT JOIN tblStudentAttendanceData
ON tblStudentData.Student_Number=tblStudentAttendanceData.Student_Number
GROUP BY
    tblStudentData.Home_Room,
    Last_Name & ", " & First_Name,
    tblStudentData.Student_Number;

This should give you a result set of all students with the number of instances of each code per student.  Those with a perfect attendance record will have zero in every column.

Should the AttCode happen to be a number data type, e.g. 1 to 9 then amend the SUM lines of query as follows:

    SUM(IIF(NZ(AttCode,0) = 1,1,0)) As [Code 1],
    SUM(IIF(NZ(AttCode,0) = 2,1,0)) As [Code 2],
    and so on

You can name the query's computed columns whatever you like of course, you don't have to use Code 1, Code 2 etc.

Note that the tblAttendanceCode table is no longer included in the query as its unnecessary (It wasn't needed in your original query in fact).  Also there is no longer an ORDER BY clause.   When creating a query for use as a report's RecordSource an ORDER BY clause serves no purpose and merely  slows things down.  The rows should be sorted in the report by means of its internal sorting and grouping mechanism.
Ken Sheridan, Stafford, England

djsteele, mvp say:

DatePart("ww", Date()) will give you a week number since the beginning of the year.
