ken sheridan say: In the academic literature of the database relational model the term key tends to be used to denote both:

(a) the primary key of a table, which is a column or set of columns which are designated as a unique identifier of each row in a table.

(b) a candidate key of a table, which is any column or set of columns which non-trivially uniquely identify each row in a table.  A table can have more than one candidate key, and where the key is made up of more than one column, a column or columns can be a part of different keys.

Chris Date makes the important point that the 'logical notion of a candidate key should not be confused with the physical notion of a "unique index"'.  Whether the column or columns are indexed makes no difference to it/they being a candidate key, though all candidate keys should of course be uniquely indexed to ensure integrity of the data.

In a paper in Relational Database Writings, 1991-1994 entitled 'The Primacy of Primary keys: An Investigation' Date presented arguments in support of a position that it is sometimes not a good idea to make one key 'more equal than others'.  I would have to refresh my memory on the details of this, but while the points made are, as always with Date, of interest, it's not something of which I've ever found it necessary to consider the practical implications.

A foreign key is simply a column or set of columns in a table whose values are required to match the values of a candidate key in another table.  Referential integrity requires that each value of a foreign key in a referencing table must have a matching value in a candidate key (not necessarily the primary key) in a row in the referenced table.

It is the existence of a candidate key and corresponding foreign key in a referencing table which defines a logical unary (one-to-many or one-to-one relationship type between the tables).  Again note that this should not be confused with the physical notion of a relationship object.  A relationship object must be created to enforce referential integrity, however, and in Access it is necessary for the candidate key of the referenced table to be uniquely indexed before an enforced relationship type can be created.  In Access it is normally the creation of the relationship object which defines the foreign key, though it can be designated as such in DDL, in which case the converse applies; the DDL definition results in the creation of a relationship object.  In Access the object is actually called a Relation, which is a little confusing as in the database relational model the term Relation is loosely equivalent to a table, though more strictly speaking it is a mathematical term for a type of set.

Finally, note that a one-to-one relationship type is directional, i.e. the referenced table's candidate key  is solely that, but the referencing table's candidate key is also a foreign key. A one-to-one relationship type is used to model a type/sub-type scenario, e.g. an Employees entity type is a (super) type of SalesStaff and ProductionStaff sub-types.  Each employee can be a member of one or of both sub-types.

JOINs are operators by which tables are, not surprisingly, joined in queries, most commonly on the primary key of one table equalling the foreign key of another, though not invariably so.  A JOIN expression in a query simply needs to evaluate to TRUE for a row to be returned, so is not necessarily an equality operation.  An INNER JOIN only returns rows where there is a match in both tables; an OUTER JOIN, which comes in flavours LEFT or RIGHT, returns all rows from one table (subject to any restrictions in the query's WHERE clause) along with those from the other where there is a match.

Sub Tyre()

stefan blom say:

When you apply numbering to the built-in headings, note that pressing Enter at the end of a heading will apply the Normal style, by default. That is why you lose the numbering. 

You can change the "Style for following paragraph" setting for the affected headings in the Modify Style dialog box.

To display the dialog box: In the Apply Styles pane (Ctrl+Shift+S), type Heading 1 and click Modify button. For the Heading 1 style, pick "Heading 1" as the "Style for following paragraph" and repeat for the other styles. See screen shot below.

However, unless you are setting up a list of numbered headings, you should be using other styles for your numbering. The general principles are discussed in the article at https://shaunakelly.com/word/numbering/numbering20072010.html?p=1768 (with the built-in headings as examples).
