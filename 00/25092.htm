vanderghast say: The query has to be 'deleteable'. A query lose its possibility to delete records, among other conditions;
- It contains an aggregate (MIN, MAX, ... ) somewhere else than in a WHERE clause (trough a subquery). DMax, DMin, ... are ok.
- It contains GROUP BY, or DISTINCT
- It contains a join which is not an inner join
- It involves another not updateable query.

So, in your case, your top query involves another query using the keyword DISTINCT, so it is not updeatable... although it can DISPLAY (view) the data.


Your solution is one of the proper ones. Another one is to fill a table from your query using DISTINCT: the query using DISTINCT cannot modify its data, but its data can be appended into another table. Another solution would have been to identify the primary key (pk) of the records you want to delete:

DELETE * FROM table WHERE primaryKey IN ( SELECT DISTINCT primaryKey FROM ...    )

where the sub query, the one within the ( ), return the primary key values of the records to de deleted. THAT query can use DISTINCT, GROUP, aggregate...

scottgem say:
You can't run a Delete query with DISTINCT. Access will not know which records to delete. What you need to do is write out the dups to a separate query, then do a join to that query for the matching records.

Ex: DELETE * FROM maintable WHERE PK IN(SELECT PK FROM dupsquery);

You can test this first by doing a SELECT query with the same WHERE clause.
