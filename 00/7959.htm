Issues....

-----

1. The structure of the file is not exactly as you describe (and expect?).

Each "record" (sequence entry) that you describe is preceded and followed by a 32-bit integer whose value is the number of octets comprising the data in between.

In some cases, the "sequence entry" that you describe is composed of several "records",
If you are not writing those octet counts, it appears that FORTRAN is.

I would be surprised, since you opened the file with the option FORM="UNFORMATTED".

But I wonder if you are writing FORTRAN structured variables instead their individual fields, and FORTRAN is bracketing each structured data with its size in order to facilitate "random access"

(FORTRAN did not contain structured data types, like struct in C, when I used it 40+ years ago.  But it might have been enhanced since then, either the standard language or Intel Visual FORTRAN in particular.)

-----

2. Type Single (VBA) v. type Double (Excel)

You might notice that when your type Single data are written into Excel worksheets, they might not be exactly what you expect.  For example, YSURF should be 108.80000000000; but Excel displays it as 108.800003051757.

The reason has to do with the way that binary type Single is "converted" to binary type Double.  The result is infinitesimal differences in the decimal representation.

(I could explain this further, if you are interested.  It gets a little technical.)

If that is problematic for you, you can use the following paradigm to do the conversion more precisely:

=--TEXT(A1,"0.000000E+0")       ' Excel

Dim d As Double, s As Single     'VBA
d = Format(s, "0.000000E+0")

where A1 contains the "converted" binary representation.

-----

3. In some cases, you describe the sequence of NW data as representing multiple sets of 3 data.  For example,  "6 coordinates stored [... as ...] X, Y, Z of End1 and X, Y, Z of End2" for the "rigid body coordinates" sequence.  Similarly for the "rigid body displacements" and "rigid body forces" sequences.

It is unclear exactly how you want to disambiguate the NW data.  More to the point:  it is unclear if FORTRAN disambiguates the sets of triads in the way that you expect, if indeed you leave that up to FORTRAN to do.

You will need to pay close attention to the order in which those data are stored, and retrieve them accordingly.

-----

4. The point in #3 is especially true if ILGSEL and IFRSEL are truly 2-dimensional arrays in FORTRAN.

If you allow FORTRAN to determine the order in which the data are written to the file, I don't know (remember) whether that is in "row order" or "column order"; that is, whether FORTRAN increments the two indices in the order that you intended, if indeed you leave that up to FORTRAN to do.


You will need to pay close attention to this detail.

-----

5. I did not understand the structure of the data for ILGSEL and IFRSEL in the "rigid body and leg selection arrays" sequence (readSelectionArrays).

Or I do understand it, and it presents a significant difficulty to implement the input routine in VBA based on my design.

In particular, your description indicates that the number of rows (first index) varies for each column (second index) based on NUMNOD(j) and NUMHED(j) respectively.

(The index is "j", not "i".  They are difficult to read in this font, especially when surrounded by parentheses.)

The problem is:  the size of NUMNOD and NUMHED are variables, namely NLEG and NFR respectively.

Ordinarily, that requires a ReDim statement in VBA to declare NUMNOD and NUMHED.  But ReDim is limited to procedure-local variables, whereas NUMNOD and NUMHED would need to be module-global variables in order to refer to them in multiple procedures.

The ideal solution would be to declare NUMNOD and NUMHED globally using some reasonable maximum size.  [EDIT] I chose this approach.

An alternative solution would be to restructure the implementation so that both uses of NUMNOD and NUMHED are encapsulated in the same VBA procedure.

But since the corresponding file data are separated by other unrelated data, that would require that we also encapsulate the calls to other VBA procedures within the same single VBA procedure that references NUMNOD and NUMHED.

Not that difficult, really.  Just a little less "structured" than I would like.

Bottom line:  you will need to pay close attention to my procedures readNodes and readFrequencies, as well as readSelectionArrays.

6. I did not implement readLegForces because the description of the data seems to be different from what we see in the formatted file dump.

The file dump shows that the corresponding data starting at offset 0b5644 are a sequence of structured data probably in the form:  KEY, NW, and NW "real" values (type Single).

The KEY seems to be incremented by one for each set of structured data.  And each set of structured data is bracketed with the 32-bit integer octet count described in #1 above.

That pattern seems to continue until the end of the file.  There does not seem to be any indication in the file of how many such "records" of data there are.

(Your pseudocode description suggests that it, too, might depend on NUMNOD.)

Bottom line:  you will need to pay close attention to this final algorithm.

Usually, the endianness of numerical data types in binary files is determined by the endianness of the computer.   (Exception:  standardized files that are intended to be system-independent.)

Since Intel-compatible computers write numerical data types to main memory in little-endian order, I would expect that numerical data types are usually written to binary files in little-endian order as well.

And that is consistent with a wiki page [1] that states, under the title "Endianness in files":  "most Windows programs use Little Endian format".

-----

Based on my experiments, Excel VBA does indeed write numerical data types to binary files in little-endian order.  That is, the least-significant octet is written to the file first, at least on an Intel-compatible computer.

The VBA procedure below can be used to ascertain the endianness of numerical data types written to and read from binary files.

I confirmed the results of the VBA procedure using other tools that I have.



Option Explicit

Type Tdouble
val As Double
End Type

Type Tlong2
val(0 To 1) As Long
End Type


Sub testEndian()
Const fname As String = "c:\temp\data.dat"    ' *** customize ***

Dim ell1 As Long, ell2 As Long, ellT As Tlong2
Dim d1 As Double, d2 As Tdouble
Dim i As Long, fd As Long
Dim b As Byte, s As String

' create 32-bit Long &h01020304
ell1 = 16909060

' create 64-bit Double &h01020304,05060708
' must enter as shown, with 15 significant digits
d1 = 8.20788039913184E-304

' write Long and Double to new file
On Error Resume Next
Kill fname
On Error GoTo 0
fd = FreeFile
Open fname For Binary Access Write As #fd
Put #fd, , ell1
Put #fd, , d1
Close

' read octets from start of file
fd = FreeFile
Open fname For Binary Access Read As #fd
s = "octets:" & vbNewLine & " "
For i = 0 To 3      ' read ell
    Get #fd, , b
    s = s & " " & Format("0" & Hex(b), "!@@")
Next
s = s & vbNewLine & " "
For i = 0 To 7      ' read dT.val
    Get #fd, , b
    s = s & " " & Format("0" & Hex(b), "!@@")
Next

' read Long and Double from start of file
Get #fd, 1, ell2
Get #fd, , d2.val
Close

s = s & vbNewLine & "long:" & vbNewLine & _
    "  &h" & Format("0000000" & Hex(ell2), "!@@@@@@@@")

LSet ellT = d2
s = s & vbNewLine & "double:" & vbNewLine & _
    "  &h" & Format("0000000" & Hex(ellT.val(1)), "!@@@@@@@@") & _
    "," & Format("0000000" & Hex(ellT.val(0)), "!@@@@@@@@")

s = s & vbNewLine & "long in file = memory?  " & (ell1 = ell2)
s = s & vbNewLine & "double in file = memory?  " & (d1 = d2.val)
MsgBox s
End Sub
