ken sheridan say:

I'm unsure of why not have Payment 1, etc. or Add-On 1 as I'm trying to capture details of all that goes into a Tour Pricing.

It's what's known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

The correct way to store multiple payments is to store each in a separate row of a separate table which references the primary key of the TourBookings table.  TourBookings would include the first four columns (fields) of your current Tour Payments table along with TourID and ClientID foreign keys.

The new TourPayments table would have a TourBookingID foreign key column referencing the primary key of TourBookings, along with PaymentDate and PaymentAmount columns.  You can then enter as few or as many payments per booking as necessary simply by inserting a row into TourPayments  for each.

Exactly the same applies to the TourAddOn columns in TourPricing.  These should similarly be split off into a separate related table in which each add on is represented by a separate row.  Similarly the three Price columns in this table should be represented by separate rows in a related table.

A few other points:

1.  Do not use the generic ID as a column name.  Always use a name which identifies the entity type in question, as you have done with ClientID and TourID

2.  As far as I can see you don't need separate GroupTours and TourPricing tables as the relationship type between them is one-to-one.  One-to-one relationship types are legitimate when modelling a Type/Sub-type scenario, but that's not the case here.  The two tables can be coalesced into a single table.

3.  Using spaces or other special characters in table or column names is considered bad practice.  Use CamelCase or represent a space by an underscore character, e.g. Tour_Name..

albert d. kallal say:

The so called issue of tall + narrow as opposed to short + wide is the issue here. The fancy term is called data normalization.

 

However, an easy way to think of this is how you ADD UP a column in an Access report.

 

You have something like:

 

ProductName        Price

Mars Bar           $1.25

OH Henry           $1.10

Snickers           $1.30

                   =====

                   $3.65

 

Note how in above we can TOTAL up the rows. And we can have 2 or 20 rows of data. So think of this like an “invoice”.

 

In above we only have ONE price column, but repeat the row many times.

 

And access forms allow repeating rows, but you can’t add a new column for each new price, since then the form and table will become too wide. And you can’t query and manipulate the data with any kind of ease if every Price value is in fact a new column.

 

In the follow sample tour booking screen, you can see for pricing I only have two rows of data, but I could add 10 more options. And note how EACH row has an option type – so I can have 3, or 30 options and NOT have to add new columns to the database. And no code was written to TOTAL UP the rows of pricing - NOTE how the total price is displayed at the bottom of the screen. And this includes GST taxes - if we had 20 rows of price options, the table design would be a "mess" if we had to create a new Price1, GST1, Price2,GST2 column. In fact such a design becomes near impossible to code. As noted, ZERO code is used below to show the price and GST total - it is a built in feature of Access and most database systems (if you create Price1, Price2 etc., then you cannot use the built in features since you not telling the database system to simply total up some rows of data. Even in Excel, you typical add more rows to the sheet - not columns because you want to add in one more "item" you sold.

 

I actually divulge the table designs I used for the above tour system in the following article of mine:

 

Notes on a Relational database tour system:

 

http://www.kallal.ca/Articles/fog0000000003.html

 

(the table design is on the last page of the article)

 

So normalizing of data is MOST important, and it often seems a bit confusing at first, but you gradually grasp why you want to use repeating ROWS of data, and not add columns to ONE row of data when multiple things such as a payment or price is involved.

And take a good read of the above article - it much outlines many cool ideas for building a tour booking system.

scottgem say:

First, you made a good start by identifying the data points you want to capture. But you need to gain a better understanding of normalization to properly organize those data points.

Your Master Clients table is OK, but as Ken said use Camel Notation (FirstName) or underscores (First Name). Each field has a caption property that you can use for a headings and labels if you want to display First Name.

You may, however, want to separate out the contact info from the identifying info. If you book Tours, then the likelihood Is what you really want is a Passengers table. Where you list the names of everyone who will be a Passenger. Since some passengers will share addresses and Phone numbers you will probably want store addresses and phone numbers in a separate table. This means that each Passenger would have an AddressID. Phones, on the other hand maybe multiples. So you have a many too many relationship which requires a Junction table to model that relationship.

Your Group Tour table is Ok, but you should have a Vendor table and that means VendorID becomes a Foreign Key.

Your Tour Pricing table is badly flawed. It should be a child table of Group Tours like so:

TourPricing

TourPricingID (PK Autonumber)

TourID (FK)

PricingTypeID (FK)

Price

Which requires:

PricingTypes

PricingTypeID

PricingType

Pricing types would be things like Ground transport, Air Fare, Cruise, Hotel, Rental Car, etc.

Tour Name, by the way should not be in any other table but Tour Group.

Similarly, your Tour Payments table is flawed for the reasons Ken described. It should look like this:

tourPayments

TourPmtID (PK Autonumber)

TourID (FK)

PaymentTypeID (FK)

PaymentDate

PaymentAmount

Plus

PaymentType

PaymentTypeID (PK autonumber)

PaymentType (Deposit, Regular, etc.)

You will also need a table to identify the passengers:

TourPassengers

TourPassengerID (PK Autonumber)

TourID (FK)

PassengerID (FK)

BillingPassenger (Yes/No)

The fields I've identified as FKs are what you use to relate back to the related table.

The system selects the active power plan after reboot, that means that your active powerplan is Balanced. If you want another power plan as active you need to set it as such.

Run the following commands from a command shell (as administrator):

powercfg /l

in the output you will notice (Balanced) * which means it's the active PP.

run ....

powercfg /S guid to set an active powerplan.

Example:

powercfg /s 381b4222-f694-41f0-9685-ff5bb260df2e

or...

powercfg /s SCHEME_MAX

sets the active PP to Max Power.
