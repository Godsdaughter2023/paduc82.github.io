scottgem say:

Well first, your database should be split with all the tables in a back end on shared network drive. Everything else should be in LOCAL front ends on each user's PC.

But this means that there isn't a way to tell who is logged in based on your login routine. There are a bunch of ways you can read the backend's laccdb file to list the computername accessing the file.

If you want to know who is in based on the login, then what you can do is have a login log in your back end. Whenever a user sucessfully logs in, you add a record to that log that indicates who logged in and when. When they shut down their front end, the log record gets update with their logout time. So you can see who is in the database but querying the log for whoever has a log record without a logout time.

As an aside, you have me curious about your "extensive piece of code". I've written a blog on doing login security using VBA (see blog address in sig) and I would call the amount of code to be extensive. So it makes me wonder if what you have done is overkill.

 

Hope this helps, Scott<> P.S. Please post a response to let us know whether our answer helped or not. You can read the LACCDB file. You won't get user names (they will all be Admin) but you will get the computer names.

Another thing you can do is to create a LoggedIn table. When the startup form opens it can put the username into the table. When the application closes you can delete that record.
