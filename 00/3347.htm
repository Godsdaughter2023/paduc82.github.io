I have several IMAP email accounts, archived email, contacts and calendar.

Your indicating that all accounts are configured as IMAP in '2007 then the key thing is t copy the right PST files to the new computer.

Imap accounts in '2007 use two PST files for a given account. One is for the email folders and the 2nd is for the contact/calendar data. When multiple IMAP accounts are configured, all of those accounts will use the same PST file for the contact/calendar data. This PST file is set as the "default data file' for the profile. The PST files for the email folders do not need to be used on the new computer since that data will get sync'd when the account is configured. Any other PST files that are in use (i.e. Archive) should also be copied to the new computer.
(Any PST files copied from the old computer should have Outlook closed on that computer to avoid potential data corruption).

On the new computer, configure the IMAP accounts and add the PST file containing the contact/calendar data to the profile by going to
Outlook menu --> File --> Open & Export --> Open Outlook Data file
(this should be also be one for any non-IMAP Email folder PST file that are configured in '2007).

When the PST file containing contacts/calendar folders is added to the profile then go to File --> Account Settings --> Data FIle tab --> select the PST file and set it as the default for the profile. This will now be where the default contact/calendar folders will be stored.

Why you do not want an .OST file for an IMAP account set as the default data file for the profile ..................

#1 - Should the IMAP account be removed from the profile, the ost file for the account will also be deleted from the system. Any data in folders with the added words "This Computer Only" in the folder name will be lost. The same is true if Outlook cannot find/open the OST file for the account - Outlook will automatically create a new OST file and resync the email folders but the contact/calendar folders will be empty.

I have a background article on the topic

How to Find & Copy the Right Outlook PST Files

  Set oAccess= CreateObject("Access.Application")

hansv mvp say:
SysCmd(acSysCmdRuntime) will return False if the user has the full version of Access, and True if the user has the runtime version.

kaspencer say:

Thankyou for the reply Hans.

My problem with the SysCmd(acSysCmdRuntime) seemed to be that it wouldn't run in my VBScript. I think it needed to be run within the Access Application. What I was looking for was a means in the VBScript, and outwith any instance of Access actually running, to detect whether Access was installed, either as a Full Licence or as the Runtime Engine only.

Anyway, after a bit more thought (and I should have come to this sooner), I decided on the following solution, which is now in the VBScript which kicks of Access with all my desired parameters:

...
' Decide whether  Full Access or Runtime only is installed:

On Error Resume Next
Set oAccess = CreateObject("Access.Application")
If Err.number<>0 then
   MsgBox err.description & crlf & ". Starting under Runtime", vbOKonly
'  above line can be omitted and a smooth change for running under Runtime Access set up:
  sExecute = "CMD /C Start msaccess /runtime /wrkgrp C:\[workgroup file & location] [access MDE file & location]"
  Set oWShell = WScript.CreateObject("WScript.Shell")
owShell.run sExecute
end if
wscript.quit
On Error goto 0

' Rest of code for Full Access startup
...

Maybe that'll help others in search of a solution.

Thanks again,

Ken

It sounds like you installed Outlook 2010 using the older ANSI format for a .pst file which has an upper limit of 2 gigs for a mail store (the practical limit is somewhat lower before you start having problems.)  You need to create a new .pst file in the Unicode format (which can go to 32 terabytes!) and move your mail from OE to the new .pst file and close the old .pst file.
