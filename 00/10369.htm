jeeped say:

There are very likely more elegant ways of accomplishing these but here are two examples.

(1) All worksheets in the same excel file in same location , i.e. A1

Sub Copy_Sel_to_All_Sheets()
    Dim aSHTs() As Long, j As Long, k As Long, l As Long, a As String
    ReDim aSHTs(1 To (Sheets.Count - 1))
    l = ActiveSheet.Index
    For j = 1 To Sheets.Count
        If j <> l Then _
            k = k + 1: aSHTs(k) = j
    Next j
    a = Selection.Address
    Selection.Copy
    Sheets(aSHTs).Select
    Sheets(aSHTs(1)).Activate
    Sheets(aSHTs(1)).Range(a).Select
    ActiveSheet.Paste
    'Application.CutCopyMode = False
    Sheets(l).Activate
End Sub

The above copies whatever is selected to all remaining worksheets.

(2) Several, but not all worksheets in same excel file, again in same location , i.e. A1

Sub Copy_Sel_to_Some_Sheets()
    Dim vSHTs As Variant, j As Long, k As Long, l As Long, a As String
    ReDim aSHTs(1 To (Sheets.Count - 1))
    l = ActiveSheet.Index
    vSHTs = Array(1, 2, 5, 6)
    a = Selection.Address
    Selection.Copy
    Sheets(vSHTs).Select
    Sheets(vSHTs(0)).Activate
    Sheets(vSHTs(0)).Range(a).Select
    ActiveSheet.Paste
    'Application.CutCopyMode = False
    Sheets(l).Activate
End Sub

The above copies whatever is selected on the active sheet to the first, second, fifth and sixth worksheets.

doug robbins - mvp office apps & services (word) say:

With Legacy FormField DropDowns, assuming that the FormFields have the bookmark names "DropDown1", "DropDown2", . . . "DropDown#", if you run a macro containing the following code on entry to one of the formfields where you want the totals shown, it will populate those formfields (assumed to be bookmarked with the names "One", "Two", "Three", "Four" and "Five") with the totals of each respectively

Dim One As Long, Two As Long, Three As Long, Four As Long, Five As Long
Dim i As Long, j As Long
Dim strBookmark As String
One = 0
Two = 0
Three = 0
Four = 0
Five = 0
With ActiveDocument
    j = 1
    For i = 1 To .FormFields.Count
        If .FormFields(i).Type = wdFieldFormDropDown Then
            strBookmark = "DropDown" & j
            Select Case Left(.FormFields(strBookmark).Result, 1)
                Case 1
                    One = One + 1
                Case 2
                    Two = Two + 1
                Case 3
                    Three = Three + 1
                Case 4
                    Four = Four + 1
                Case 5
                    Five = Five + 1
            End Select
            j = j + 1
        End If
    Next i
    .FormFields("One").Result = One
    .FormFields("Two").Result = Two
    .FormFields("Three").Result = Three
    .FormFields("Four").Result = Four
    .FormFields("Five").Result = Five
End With

The above will work with any number of formfields.

lance cai msft say:

Hi Tony,

Verifying synchronization of single user

To verify the synchronization status of a single user, execute Get-MsolUser PowerShell cmdlet from an elevated PowerShell command prompt and retrieve date and time stored in LastDirSyncTime attribute of users. For example, if you wish to check synchronization status for a user named user@domain.com, you will execute the command below:

Get-MSOlUser -UserPrincipalName "user@domain.com" | Select-Object UserPrincipalName, LastDirSyncTime

When you run the above command, it will show last date and time the user was synchronized to Office 365. To check the last synchronization date and time for another user, just replace user@domain.com with another username.

Verifying synchronization of all users

You can execute the above command in case you are troubleshooting synchronization for a single user, but in case you need to check synchronization status for all users, you will be required to export the value of LastDirSyncTime attribute for all users by executing the command below:

Get-MSOlUser -ALL | Select-Object UserPrincipalName, LastDirSyncTime | Export-CSV C:\Temp\SyncStatus.CSV

The above PowerShell command retrieves all users from Office 365 with their user principal name and last synchronization date and time. The output is exported to C:\Temp\SyncStatus.CSV file. You can navigate through the file to ensure last synchronization date and time for all users matches with the current synchronization date and time.

Checking health of synchronized users

Not many Office 365 administrators know that Office 365 provides a quick way to check the health status of synchronized users who sync from on-premises Active Directory to Office 365. The user properties that you can use to report on the health status of synced users are “ValidationStatus” and “DirSyncProvisioningErrors.” ValidationStatus property stores the health status of synced users and DirSyncProvisioningErrors indicates if any errors were encountered during the synchronization. To retrieve synchronization status for users along with ValidationStatus, LastDirSyncTime, and DirSyncProvisioningErrors data, execute the command below:

Get-MSOlUser -ALL | Select-Object UserPrincipalName, LastDirSyncTime, ValiadtionStatus, DirSyncProvisioningErrors | Export-CSV C:\Temp\SyncStatusHealth.CSV

Once you have executed the command above, open the CSV file in Excel and filter ValidationStatus and DirSyncProvisioningErrors columns to ensure there are no errors reported. If you find any users with errors, check the last synchronization date and time for the user. You may need to troubleshoot further as to know what caused synchronization failure for a user.

Ok, first, your Subcontractor table is not well designed. You are encoding data using field names which violates normalization rules. Instead of 6 Yes/No fields for contractor type and Union you should have 2 fields and 2 lookup tables:

tblContractorType

ContractorTypeID (PK Autonumber)

Contractor

While the above should work, I should point out that your design is not a good one.  Having multiple Boolean (Yes/No) columns representing values of the 'services' and 'unionisation' attributes is what's known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.  The SubcontracterList table should really be decomposed into a set of related 'Services' and 'Unionisation' tables in which each attribute value is represented by a separate row rather than by separate columns as now.
