charles kenyon say:

See another of Suzanne Barnhill's pages on changing defaults:

http://wordfaqs.ssbarnhill.com/CustomizeNormalTemplate.htm#Word2003Font

stefan blom say:

The way I understand it, the Default Paragraph Font represents the default font for any given paragraph style. You can use it in Find & Replace operations to remove direct font formatting only while preserving similar formatting which is part of a paragraph style (as Suzanne demonstrates in the article for which you provided a link).

The Default Paragraph Font does not, in itself, control the font formatting of other text in a document.

The font formatting in a paragraph style (such as List Paragraph) is inherited from the document defaults (= the Set Defaults tab of the Manage Styles dialog box) or from the Normal style. Of course, you can modify List Paragraph, or any other paragraph style, so that it has whatever font formatting you want.

When you are modifying the font for any paragraph style, you are setting its default font.

ken sheridan say:

Looks like i need to learn about queries



Learning how to create queries would be a good idea as they are such a fundamental part of good database design, but as far as your form is concerned you seem to be doing everything right at present, and don't need to change anything.  In the form Total Nominal Length is not a field, it's a text box control; tables have fields (aka columns), forms have controls.  A table can be a base table or the result table of a query.  A text box control can be bound or unbound.

A bound control is one which has the name of a field as its ControlSource property.  An unbound control can have an empty ControlSource property, where it's used for entering a value which is then used to do something else, e.g. go to a particular record; or it can have an expression as it ControlSource, which is what your has, the expression being =([Calc Shk lm]+[Calc CTL lm]).  This type of unbound control is called a computed control (aka calculated control), and shows the result of the expression.  That's all that's necessary.  You don't need to store that result in the table, and more importantly it would be dangerous to do so because it introduces redundancy.  So long as you have the values for [Calc Shk lm] and [Calc CTL lm] in separate columns in the table you can compute the Total Nominal Length at any time, and always be sure the result is correct.  If you were to store the value in a third column in the table then there is nothing to stop any one of the three values being changed independently of the others so that Total Nominal Length is no longer equal to the sum of the other two values.  That would be an 'update anomaly', and the data would be inconsistent, with no way of knowing which of the three values is incorrect.

You can also have a computed column in a query, which is what Doug has described.  In this case you'd enter the following in the 'field' row of a blank column in query design view:

Total Nominal Length: [Calc Shk lm]+[Calc CTL lm]

There is no overriding  advantage in doing the computation in a query or in a form, though doing it in a query would mean you could base a form and report on the same query for instance, without having to do the computation twice.

There are some situations, however, where you might at first sight think that storing a computed value would be redundant, but where that would not actually be true.  The classic scenario for this is one of orders or invoices for products.  Say you have a Products table with columns, ProductID, ProductName and UnitPrice, and you have an InvoiceDetails table with columns InvoiceID, ProductID and Quantity.  You might think that to compute the gross price for an invoice line you merely need to join the two tables in a query on ProductID and multiply the unit price by the quantity with Products.UnitPrice*InvoiceDetails.Quantity.  That would give you a gross price for each invoice line, but the unit price of a product will change over time, usually upwards.  This would mean that the gross price in every invoice detail record for that product would also change because it's using the new UnitPrice value.  This would be incorrect; the gross price should remain static as the value it was when the invoice was created, not change over time.  The correct design therefore would be to have InvoiceID, ProductID UnitPrice and Quantity columns in the InvoiceDetails table, i.e. you have UnitPrice columns in both tables.  When you enter the product into an invoice line record (usually by selecting it in a combo box bound the ProductID column) you can assign the current unit price to the UnitPrice column in InvoiceDetails.  This would be done in code in the AfterUpdate event procedure of the combo box which looks up the UnitPrice from the Products table like this:

    Dim strCriteria As String
    Dim curCurrentPrice As Currency

    strCriteria = "ProductID = " & Me.ProductID
    curCurrentPrice = DLookup("UnitPrice", "Products", strCriteria)

    Me.UnitPrice = curCurrentPrice

Now that you have the current unit price stored in InvoiceDetails, it will remain static in the row no matter what changes are made to the UnitPrice value in the Products table, so the expression for the gross price is now in a query:

GrossPrice:InvoiceDetails.UnitPrice*InvoiceDetails.Quantity

or as the ControlSource property of a computed text box control in a form or report:

=[UnitPrice]*[Quantity]

and this will always be the value current when the invoice was raised.
