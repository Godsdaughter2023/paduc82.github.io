lahirufernando say:

How to Import Data from one Excel file to another.

Hi,

when i have this type of Excel sheet,How can i Import the all datas to other Excel sheet by filling only colomn A Numbers.

That means when i fill the colomn A values in Excel sheet,other data would Import automaticaly without typing.

Please reply me ASAP.

bernie deitrick say:

Open that file and use a VLOOKUP in the other file

To get the values from column B:

=VLOOKUP(A3,'[File with data table.xlsm]Sheet1'!$A:$C,2,FALSE)

and to get the values from column c:

=VLOOKUP(A3,'[File with data table.xlsm]Sheet1'!$A:$C,3,FALSE)

To get the correct file and sheet name, select columns A to C of that file while creating the formula - this part will change:

'[File with data table.xlsm]Sheet1'!$A:$C

ken sheridan say:

................and decided one table was ok for my data.

Did you mean to say 'one form was ok for my data'?  If so, that's where you have gone wrong.  In a relational database you can rarely get away with a one-size-fits-all interface.  But before considering the interface we need to examine the underlying model.  I assume that you have tables such as Projects, Activities and Employees.  If so, you would also need the following tables to model the relationship type's between them:

ProjectEmployees
….ProjectID  (FK)
….EmployeeID  (FK)

ProjectActivities
….ProjectID  (FK)
....ActivityID  (FK)

EmployeeProjectActivities
….ProjectID  (FK)
….EmployeeID  (FK)
....ActivityID  (FK)
….DateWorked
….HoursWorked

The first two tables are 'all key' i.e. in each case the primary key is a composite one of the two foreign key columns.  The primary key of the third table is a composite one of the three foreign keys and the DateWorked column, assuming that each row represents one employee's inolvement in one activity on one project on one day.  ProjectID/EmployeeID and ProjectID/ActivityID are each composite foreign keys referencing the composite primary keys of ProjectEmployees and ProjectActivities respectively.

As regards the interface I would envisage the following forms:

1.  A projects form, in single form view, within which are two subforms, in continuous forms view, bound to the ProjectEmployees and ProjectActivities tables respectively.  Each subform would be linked to the parent form on ProjectID, and would contain a combo box to select the employees and activities which relate to the project.

2.  There are various ways you could insert the hours worked per employee per project per activity each day.  A simple solution would be a form, frmTimeLog say, in continuous forms view with two unbound combo boxes in the form header, cboProject and cboEmployee.  These would be correlated so that the cboEmployee combo boxes lists only those employees who have been assigned to the selected project.

The form would be bound to the following query:

SELECT *
FROM EmployeeProjectActivities
WHERE ProjectID = Forms!frmTimeLog!cboProject
AND EmployeeID = Forms!frmTimeLog!cboEmployee
ORDER BY DateWorked;

In both unbound combo boxes' AfterUpdate event procedures requery the form with:

Me.Requery

When a project and employee are selected the form will show the selected employee's record of involvement in the selected project.  The form will include a combo box to select an activity,  and text boxes to enter the date and hours worked.

ken sheridan say: Firstly, as regards the model, it assumes that any task/activity can apply to any fund/project.  If this is the case the model is valid, but if not, then the relationship type between tasks/activities and funds/projects should be modelled by a further table and an enforced relationship type between this table and the hours table should be created.

As regards the form this should be bound to the Hours table or a query on that table, with combo boxes bound to the EmployeeID, FundID and TaskID columns, with text boxes bound to the DateWorked and HoursWorked columns.  If the form's DataEntry property is set to True (Yes) the form will open at an empty new record, and the user will not see other users' data or any earlier records of their own.  Nevertheless, you should be aware that any user able to open the database would have little trouble accessing other users' data, so there could be confidentiality issues here.

Note, however, that if the HoursWorked column were also to be a date/time data type, then you can enter the duration in the format hh:nn, setting the control's Format property to that format.  You'll sometimes see it said that a date/time data type should not be used for durations, but only for a point in time.  That is only true if durations of 24 hours are more are being entered, as the date/time data type is no more than a floating point number in which the integer part represents a day, and the fractional part the time of day, so by using it to enter a duration of less than 24 hours the duration is merely being stored as the fraction of one day.

When summing durations over a length of time the result will usually be more than 24 hours of course, which cannot be shown in the usual date/time format.  By the use of simple functions it can be shown as a duration in hh:nn or hh:nn:ss format.  You'll find the use of such functions illustrated in TimeArithmetic.zip in my public databases folder at:
