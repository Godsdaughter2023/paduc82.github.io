peter thornton say:

Do you mean that if I use CreateObject, I don't actually need to implicitly reference the Outlook library, it will take care of itself?

If in development you set a reference to an earlier version of (say) application, when your app is used in a later version the reference will update to the later version, all will work fine. But the reverse is not true. One cause of problems is if your app originally saved with the early reference gets resaved with the later version, when sent back to someone with an early version it may fail.

There are two good reasons to set a reference, say to Outlook.

1. Speed: Object model stuff will be found much more quickly.
2. Ease of developement
2a: You get all the intellisense as if you were working directly in the other app
2b: You can use named constants that are defined in the other app's type library.

The downside is the possibility of your app with the new reference failing with someone's older version.

What some do is set the reference and develop with Early Binding, to get 2a above, then remove and convert to Late Binding by removing the reference before distribution, particularly if speed is not an issue or known problems will occur. Most will not notice the difference between say 0.01sec and 0.0001 seconds

Apart from removing the reference you will need to change any full declarations to Outlook's object model to "As Object" eg

Dim objMI As Object ' not As Outlook.MailItem

Also change any named constants to their intinsic value, or define them yourself, eg

Private Const olMailItem As Long = 0

Also you can't use the "New" to create an instance but use CreateObject

One more thing, if developping in a new version for distribution to an old version, be sure not to use any methods or properties that were introduced in the newer version (although you can work If NewVersion do New-Stuff Else do old-stuff).

Regards,
Peter Thornton

jp ronse say:

Hi Geoff,

I was facing the same issue and you don't need to include the object reference of Outlook to get it working.

Put following in a module but not in a procedure:

Declare Function FindWindowByClass Lib "user32" _
    Alias "FindWindowA" (ByVal lpClassName As String, _
    ByVal lpWindowName As Long) As Long

 

Call it as:

lngRet = FindWindowByClass("rctrl_renwnd32", 0&)   

 

rctrl_renwind32 refers to Outlook.

If lngRet =0 then Outlook is not running else Outlook is running.

 

When Outlook is already running, uou can use the GetObject method:

Set gobjOutlook = GetObject(, "Outlook.Application")

Set gobjMailItem = gobjOutlook.CreateItem(olMailItem)

Set gobjMailAddress = gobjMailItem.Recipients.Add(strTo)
            gobjMailAddress.Type = olTo

And so on.

You can of course not use the Outlook constants without declaring them.

Some examples:

Public Const olMailItem As Long = 0
Public Const olTo As Long = 1
Public Const olCC As Long = 2
Public Const olBCC As Long = 3
Public Const olAppointmentItem As Long = 1

Public Const olFree As Long = 0
Public Const olTentative As Long = 1
Public Const olBusy As Long = 2
Public Const olOutOfOffice As Long = 3
Public Const olFormatPlain As Long = 1

Const olPRIVATE                   As Long = 2 ''' Outlook constant
Const olRequired                   As Long = 1
Const olOptional                    As Long = 2

Programming with late binding is (without a reference to the object) is more difficult then with early binding because you do'nt get the proposals per instruction.

To work aroun, include the reference, write your code, remove the reference and adapt the code.

Cascading styles question - Microsoft Community

crtate say:

Cascading styles question

I'm a bit confused as to what parts of a style cascade in Word (2007).

I've read Shauna Kelly's document. However, when I try to get my styles to cascade only certain elements seem to filter down through the styles.

For example, if I set Heading 1 to Arial then I can get Headings 2, 3 and 4 to change to Arial too. However, I cannot seem to get the colour of the text to filter through or the size. Does the cascading bit only apply to the actual font type or should all the other elements be changeable too?

Thanks.

jay freedman say:

A property will cascade only if the initial setting of that property in the dependent style is exactly the same as the setting in the based-on style.

One way to tell whether you've gotten them right is to look at the list of defined properties in the Modify dialog of the dependent style, just below the preview panel. If the property you want to cascade isn't listed there, then its value is inherited from the based-on style. If it is listed there, then its value is different, and the property won't cascade.

The colour setting is particularly pernicious. In the default setup, the property list for the Heading 1 style says "Font color: Accent 1", referring to one of the theme colours. What it doesn't tell you is that it's the "Accent 1, 25% Darker" variant. The Heading 2, 3, and 4 styles use the plain Accent 1, while Heading 5 and 6 use "Accent 1, 50% Darker." Thus changes in the Heading 1 colour definition won't cascade to any of the other Heading styles. If you want the colour to cascade, you must modify all the styles to use the same variant of the same theme colour or one of the non-theme (standard) colours.

The alternative, if you stick with the theme colours as in the default set of styles, is to select a different set of colours to use in the theme. On the Home tab, click the Change Styles button and choose Colors, then select a palette of colours that you find suitable or define a custom set. That will maintain the variants (darker/lighter) but switch the base colour for the Accent 1 and other selections. Similarly, you can choose different pairs of heading and body fonts by clicking Change Styles > Fonts, and the different sizes defined in the styles will be maintained while switching the fonts.

daniel pineault say:

The IIF() statement I posted doesn't care whether your field has 5 characters or 500.  So the issue, which I have been trying to figure out since yesterday, lies elsewhere.

I could be wrong, as I don't have access to your data, but could your field have "spaces" at the end?  Try this instead and see if it doesn't help:

IIf(IsNumeric(Right(Trim([ProjectNumber]),1))=True,Left(Right(Trim([ProjectNumber]),4),3),Right(Trim([ProjectNumber]),3))

The Trim function will remove any spaces found at the beginning or end of your string so then we are actually working with the text part.  I have tested it on a table with several hundred record and it does work properly.  If this still doesn't work properly for you, could you post a sample of 2 or 3 ProjectNumbers that fail so we can try to identify the source of the problem.
