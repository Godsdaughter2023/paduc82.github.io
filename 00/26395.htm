ken sheridan say:

Why would I remove those fields if they are needed in the report (those are used as filters. I need to know how to do it with what you see in the example above

To restrict the result set of a crosstab query on columns other than those by which the query is grouped you need to reference those columns in the query's WHERE clause, not its GROUP BY clause.  As Duane says, to be able to enter the criteria for the restriction at runtime you can reference controls in a form as parameters.  If you wish to be able to use the parameters electively, leaving some empty while entering data for others then you need to use the following syntax:

WHERE (SomeColumn = <some parameter>
OR <some parameter> IS NULL)
AND (SomeOtherColumn = <some other parameter>
OR <some other parameter> IS NULL)
AND etc

The parentheses are crucial to enforce the OR operations to evaluate independently of the AND operations.  You cannot realistically restrict a query this way in design view, but the logic is very straightforward as you can see, so that should not be a problem.  The query should also be saved in SQL view as, if you switch to design view before saving it, Access will move things around.  At best the logic will be obscured; at worst the query might become too complex to open.

There is a possible fly in the ointment, however.  I don't know whether it is still the case in later versions of Access, but in earlier versions it was certainly the case that parameters in a crosstab query must be declared.  If this is still the case then the above syntax would only work for parameters declared as DateTime data type.  This is because, with other data types, a declared parameter cannot evaluate to Null (dates are unusual in this respect).  The way to get round this would be to give each control referenced as a parameter a default value of the appropriate data type, 'N/A' for text, 0 for numbers or currency say.  That way it would be possible to declare the parameter and, in the WHERE clause, test for OR <some parameter> = "N/A" or OR <some parameter> = 0 as appropriate, rather than testing for Null.
