greginmich say:

My only experience with a paid AV app was with Kaspersky in the days when I wanted to maximize the protection for my code projects – and that experience was very positive. Kaspersky is one of several paid AV products that consistently scores detection rates close to 100%, and if you’re using one of the apps in this class, then you’re actually less likely to benefit from adding the protection of Windows Defender’s Limited Periodic Scanning, simply because of the inherent detection capabilities of these apps. The apps that would stand to benefit the most from Limited Periodic Scanning would be apps that score in the lower end of the empirical distribution – like McAfee, because those apps will tend to miss more malware. Limited Periodic Scanning should be moderately useful to people using the best-rated free AV apps.

 

I’m still using the same simple “additive model” for on-demand detections that I’ve been using for many years now. On-demand scanners provide “additive” or “supplemental” protection because they provide extra coverage through additional signatures, although that coverage isn’t the ideal, proactive, “on-access” type. Defender is now capable of providing fully automated, AV-inclusive, supplemental protection for any third-party AV solution. So according to the additive model, the overall level of protection would be optimized by pairing up the real-time protection of the best available AV app with Windows Defender’s Limited Periodic Scanning (plus lots of additional on-demand scanners). The potential benefit from Limited Periodic Scanning won’t be limited to people who already have a third-party AV app installed, it will also be available to anyone who decides to install a higher-rated third-party AV app – so the Limited Periodic Scanning feature should actually provide some incentive for installing a third-party app, because this will potentially result in an improved real-time detection rate without sacrificing any of the on-demand detections built into Windows Defender.

 

I don’t see any advantage at all to running Windows Defender as a standalone AV solution simply because it’s now starting to overtake the bottom-rated AV apps in lab testing. Replacing a bottom-rated app with a free app that has a higher detection rate than Defender – and then turning on Defender’s Limited Periodic Scanning is the way to optimize your protection if your choices are limited to the free AV apps. I started recommending periodic scans with third-party malware-removal tools many years ago based on this “additive protection” model for on-demand scans. This model now implies that using Windows Defender in conjunction with the real-time protection of a better-rated third-party AV app will step up your overall level of protection – but it also implies that you should keep right on scanning with Malwarebytes Free and whatever else you’ve found useful for finding “missed malware”. The real implication of the old quip that “no antivirus is 100% effective” has always been that you need to leverage the detection capabilities of multiple engines.

But here’s the reason why dragging out the old argument that Windows Defender is actually better than it appears to be in the mainstream AV test results won’t help your cause: Even if we give Windows Defender the benefit of the doubt in a huge way (which we really shouldn’t do), and assume that it provides exactly the same level of protection as the top-rated free AV app, then the most effective strategy (by just a little) would still be to run the top-rated free AV app as the real-time AV provider, with Limited Periodic Scanning turned on. john w. vinson say:

A Date/Time value in an Access table is not stored in any particular format: it's stored as a double float number, a count of days and fractions of a day since midnight, December 30, 1899. For example, where I am it's now #3/9/2012 5:56:40# or 40977.7475925926. That value can be displayed in any format you like - as 3/6/12, or 6/3/12, or innumerable other choices.

If you have some users familiar with the American and others with the British format, you may want to consider inconveniencing both of them for their own safety: you can use an unambiguous format such as the ISO yyyy-mm-dd format. 2012-03-09 only makes sense as March 9 so it's safe. If need be, you can use an Input Mask of 0000-00-00 to enforce entry in this format.

I notice that you add the shared mailbox via the Add Account button in the Outlook client. However, I’d like to let you know that it’s an incorrect way. Since the shared mailbox has no licenses and passwords, it’s not feasible to add it directly in Outlook client like common user mailboxes.

To add the shared mailbox, please follow the steps below and give yourself full access to it and use Auto-Mapping to let it appear automatically in your Outlook client.

a. Configure your Exchange account in Outlook client via Exchange.

Note: The steps below need admin permission. If you’re not an admin, please let admin do this for you.
b. Connect to Exchange Online using remote PowerShell
c. Run the command: Add-MailboxPermission –Identity shared mailbox -User your mailbox -AccessRights FullAccess -AutoMapping $true

If you configure your account via Exchange and have full access permission of the shared mailbox, it will appear automatically in your Outlook client.
