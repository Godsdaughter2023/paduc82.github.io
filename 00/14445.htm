The default template is named Normal.dotm in Word 2010, and it's stored in C:\Users\<username>\AppData\Roaming\Microsoft\Templates.

The AppData folder is hidden so you need to show hidden files and folders in Windows Explorer in order to see it.

Quit Word, move Normal.dotm out of its folder (don't throw it away yet), then start Word - you should get a blank document now.

If you had custom styles, macros etc. you can copy them from the old Normal.dotm to the new one.

Does O365 have any means of preventing these applications from still - Microsoft Community

logananicich say:

Does O365 have any means of preventing these applications from still connecting via Basic authentication that would still allow ActiveSync to work and without breaking any Outlook functionality?

First, our general configuration is that we have A single Office 365 account and we federate our sign in to Okta for SSO and MFA sign-ins.  With Okta we can specify that if someone is on our network (based on external IP) they are NOT prompted for MFA, but when they are off network they are.  We are currently using ActiveSync for our mobile phones with an app called Blackberry Work.  This doesn’t support the modern authentication flow, but we can’t have ActiveSync just open everywhere otherwise there’s no point in using Blackberry Work in the first place.  So we have O365 setup to quarantine all Active Sync devices unless it has the Blackberry Work UserAgent string.  We then have Okta setup so that mobile devices using ActiveSync are allowed to authenticate anywhere.  This allows us to prevent unauthorized devices for the people that do have ActiveSync enabled.

 

Hopefully that all makes at least some sense.

 

Now onto the issue and questions at hand.

 

Issue:

About a month ago an employee of ours was phished and gave up his credentials.  With MFA this shouldn’t be a big deal.  However, the attacker apparently used an email client called eM Client to connect to his account and it worked, and they were still able to send messages to all of his contacts and deleted the messages from his mailbox.  We never got the ActiveSync quarantine notification so it went under the radar entirely.

 

I downloaded this client and confirmed the same with my account. It’s not using ActiveSync it’s just using Exchange Web Services with basic authentication.  Ultimately this was found to be allowed through Okta under the policy that allows “Unknown” ActiveSync devices.  (They specifically have iOS, Android, and “Unknown” device types listed under ActiveSync).  Since it doesn’t use ActiveSync this doesn’t make sense, and is clearly some kind of Okta issue.  They’re working on that and have seemingly acknowledged the existence of an issue on their side, but Okta support has nonetheless asked me to ask some questions of MS.

 

We’ve handled this through a separate policy in Okta that blocks the “Unknown” devices, but some Android devices report back as “unknown” so we have to have exceptions for them.

 

Questions

  1. Does O365 have any means of preventing these applications from still connecting via Basic authentication that would still allow ActiveSync to work and without breaking any Outlook functionality?
    1. Note we still have some active Outlook 2010 installations so those can’t break either, yet?
  2. The UserAgent / ClientInfo String for eM Client is: Client=Microsoft.Exchange.WebServices; eM Client/7.1.33101.0 (ExchangeServicesClient/7.1.33101.0), does O365 have any way of using that to prevent that agent (and maybe others?) from connecting.

Thanks
