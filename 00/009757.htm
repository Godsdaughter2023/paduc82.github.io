In your example if you reference the variable lo.Name it will return “c031_tblName”. If you reference the variable lo.DisplayName it will return “_c031_tblName”.

jacques raubenheimer say:

Doug is right. Instead of two sections for the TOC, just set a different first page for your section, then you can suppress the TOC header on the first page.

I think the confusion about all of this stems from the slight disconnect between what we see and "experience" on screen and what happens when a document gets printed.

From what you have written, it seems as if this document will be printed in duplex (both sides of the actual physical page) and bound. In printing terms, the "front" side of a page is called the recto, and the recto is odd numbered in a book (when it appears in an opened book, it is on the right). The "back" side of the page is the verso, and it is even numbered. It appears on the left of an opened book.

So in essence, the numbering scheme you are proposing is a bit contrary to the convention for a duplex-printed document, and that is why Word is not playing along with you--it's just trying to do the numbering the way it knows it should be done (or let me say, the way it knows a printer--the person, not the device--will expect it to be done, which is also what every style guide will recommend).

So I first want to ask you to reconsider your numbering scheme. It's not just about sticking with convention, as if convention were some holy thing, but of doing things in a way that your eventual readership will grasp and be able to understand.

Having said that, if you decide that you want it to stay like that, you will simply have to do the following:

1. Avoid all Odd page/Even page section starts, and use only Next page section starts.

2. Ensure that page numbering is always continued from the previous section.

3. You will then have avoided Word's automatically added blank pages which aims to balance things out according to Printers' (the people, not the devices) specifications, but the tradeoff is that you may find yourself having to manually insert page breaks to regain balance. To me, that seems like more hard work than letting Word manage it for you.

So try this. Make sure that Section 4 has a Next page section start. Then set the page numbering to start at zero (this only works for Arabic numbers, but I suspect you are using those). Then add a page break right at the start of Section 4. Now Task 1 will start on the left (it has been moved to the verso by the page break) but will be numbered 1. Not logical, and still has a blank page, but "something" (even nothing but blank space) must be printed on that side of the sheet, as it's a physical sheet. So you can't get around that.

Just note that if you do take that document to a print house, they might be unhappy with you...  :-)

Prevent losing email in Hotmail when you've set up POP3

Hotmail has a feature that prevents losing email messages if you accidentally set up POP3 to delete them after reading them on the server. Here’s how it works:

    The first time that Hotmail receives a POP3 instruction to delete email messages after reading from an email app, Hotmail won't delete the messages. mike & ann mariedurso say:

I opened a problem ticket with Microsoft on this issue.  A technician contacted me the next day and helped me resolve the issue. 

The problem was related to MS Access 2010 not recognizing the file type extension ".xls" on the original Excel 98 version of the import spreadsheet.  To correct the issue we did the following:

  • converted the database to a 2010 version 
  • coverted the original Excel 98 import workbook to an Excel 2010 version
  • modified the import macro arguments in the database to file type "Excel Workbook" and file name extension ".xlsm" for the worksheets that contain data be imported into the associated db tables  

With these changes the import macro is now working exactly as originally desinged.
