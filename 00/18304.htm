Sub ScratchMacro()

Dim oPar As Paragraph
Dim oRng As Word.Range
Dim lngIndex As Long
  For Each oPar In ActiveDocument.Paragraphs
    Set oRng = oPar.Range
    oRng.Collapse Direction:=wdCollapseStart

    Do

    Next lngIndex
 Next oPar
End Sub

hansv mvp say:

One could easily argue that 0^0 is undefined:

If you look at the sequence 3^0 = 1, 2^0 = 1, 1^0 = 1, you'd predict 0^0 = 1.

But if you look at the sequence 0^3 = 0, 0^2 = 0, 0^1 = 0, you'd expect 0^0 =0.

In other words, the function x^y has a discontinuity at (x,y) = (0,0).

chip pearson say:

I don't believe this is a bug. Rather, I believe it was a deliberate design decision when Excel was developed. Many authors consider 0^0 to be undefined, since two conflicting mathematical principles are involved. The first is that 0^N = 0 for all N <> 0. The second is that N^0 = 1 for all N <> 0. In some circumstances, 0^0 should be treated as 1, and in other circumstances it should be treated as undefined. My memories from my college days 25 years ago as a Mathematics major are that 0^0 is undefined. The treatment of 0^0 as equal to 1 is largely a matter of convention and convenience rather than solid mathematics. From Wikipedia ( http://en.wikipedia.org/wiki/Exponent#Zero_to_the_power_of_zero )

>>>

According to Benson (1999), "The choice whether to define 00 is based on convenience, not on correctness."[14]

<<<

 

In its overview the, Wikipedia article on exponents of 0  (http://en.wikipedia.org/wiki/Exponent#Powers_of_zero)  states that

 

>>>

If the exponent is zero, some authors define 00 = 1, whereas others leave it undefined, as discussed below.

<<<

 

Wikipedia goes into great detail of the problem at

 

http://en.wikipedia.org/wiki/Exponent#Zero_to_the_power_of_zero

 

The IEEE standards provide several definitions of exponentiation, and some of those treat 0^0 as NaN (not a number). However, many if not most standard math class libraries (e.g., NET System.Math and Java) treat 0^0 as 1. Most, but not all, programming languages treat 0^0 as 1, but others, particularly Mathematica and Wolfram Alpha, treat 0^0 as an indeterminate form. VBA treats 0^0 as equal to 1.

 

While treating 0^0 as undefined and returning #NUM was probably a wrong decision on the part of Microsoft programmers, I would assume that it was a deliberate design decision, not a real bug. If memory serves, Lotus 123 considered 0^0 to be undefined, and at the time, Lotus had 95% of the spreadsheet market. When MS designed Excel to compete with Lotus, it deliberately replicated bugs in Lotus so that Excel would always return the same results as Lotus, even if Lotus's results were wrong. It is for this very reason that Excel calculates the day after 28-February-1900 as 29-February-1900, not the correct date of 1-March-1900. Lotus made the mistake in thinking that 1900 was a leap year, and MS followed suit so that Lotus users would get the same results in Excel and be more likely to adopt Excel.

 

You can read far more that you probably want to know about 0^0 on Wikipedia at http://en.wikipedia.org/wiki/Exponent#Zero_to_the_power_of_zero .

If you clean-installed instead of upgrading, it did not have a chance to detect the Windows 7 activation during the upgrade.  It needs to either detect that an activated Windows 7 was upgraded, or if you are using a Windows 10 install created after the November 2015 update, you can activate using the same Windows 7 Product Key that was properly activated on that same device before the clean install (and has not been activated elsewhere since then).
