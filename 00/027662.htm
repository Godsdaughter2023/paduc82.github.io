Hi Dayspring,

To verify the domain with Office 365, we need to add TXT records or MX records to the domain hosting provider. Here are the steps to add TXT records on 1&1:
1. Login to your 1&1 control panel.
2. Click Manage Domains from the Domains panel.
3. Click the arrow icon next to the domain name in question to display an options menu.
4. Select Edit DNS Settings from the Domain Settings section of the options menu
5. Click the Add Record button.
6. Enter the value of the TXT record and then click on Add.
7. Review your changes and then click save.

Meanwhile, here are two articles for your reference:
Create DNS records at 1&1 Internet for Office 365
Add your users and domain to Office 365

Macro below:

Private Sub Worksheet_Change(ByVal Target As Range)

    If Target.Cells.Count > 1 Then Exit Sub

    If Not Intersect(Target, Range("A:A")) Is Nothing Then

        Target.Offset(0, 1) = Date

    Else

        If Not Intersect(Target, Range("C:C")) Is Nothing Then

            Target.Offset(0, 1) = Format(Now(), "long time")

        Else

            If Not Intersect(Target, Range("F:F")) Is Nothing Then

                Target.Offset(0, -1) = Format(Now(), "long time")

                Target.Cells.Offset(0, 1).FormulaR1C1 = "=RC[-2]-RC[-3]"

            End If

        End If

    End If

End Sub

rakesh_agrawal say:

Hi,

The following line is usually used to find the last row of a range:

     LastRow = Cells(Cells.Rows.Count, "F").End(xlUp).Row

However, if some of the rows at the end of the range are hidden, it ignores then and returns the last visible row.

It may be a cause of your problem.

However, the following code has been tested by me and works.

Sub temp()
    Dim firstRow As Long
    Dim lastRow As Long
    Dim i As Long
    
    firstRow = 12
    lastRow = firstRow + Range("F12").CurrentRegion.Rows.Count - 1
    
    Debug.Print lastRow
    
    For i = firstRow To lastRow
        If Cells(i, "F").Value = 0 Then
            Cells(i, "F").EntireRow.Hidden = True
        Else
            Cells(i, "F").EntireRow.Hidden = False
        End If
    Next i

End Sub

Hope it helps.

Rakesh

jllatham say:

rakesh_agrawal is correct - hidden rows are not seen in the test I used for finding the last row.  Another thing that could make that test invalid would be if you had the data on the sheet filtered: again, the test would not see hidden rows caused by the filtering, and the .Hidden = False statement would not work as expected/hoped for.

But the changed code below should deal with both cases for you - at the expense of removing any data filtering you may have previously set up on the sheet, and I've cleaned out no-longer-required code from it.

Private Sub Worksheet_Calculate()
Const firstDataRow = 12
Dim LastRow As Long, c As Range
Application.EnableEvents = False
Application.ScreenUpdating = False ' does not have to be set =True later

'**********  New Line of Code

ActiveSheet.ShowAllData ' unhides all rows/columns and removes data filtering

'***********************************************************************************

LastRow = Range("F" & Rows.Count).End(xlUp).Row
'safety net to make sure we don't hide rows above row 12
If LastRow < firstDataRow Then
  LastRow = firstDataRow
End If
 
  For Each c In Range("F" & firstDataRow & ":F" & LastRow)
   If c.Value = 0 Then
        c.EntireRow.Hidden = True
   End If
Next
Application.EnableEvents = True
End Sub

daniel pineault say:

Create a normal form, setup the detail section as you see fit by adding the various fields of interest.

In the form header, add a combo box named cbo_Registration which has a Row Source to list all the Registration.  Then add a Click Event like:

    Dim RId                   As Variant
    Dim sSQL                  As String

    RId = Me.cbo_Registration

    If IsNumeric(RId) = False Then
        Me.Section(acDetail).visible = False
        sSQL = "SELECT * " & _
               "FROM YourTableName " & _
               "WHERE (FALSE);"
    Else
        If Me.Section(acDetail).visible = False Then Me.Section(acDetail).visible = True
        sSQL = "SELECT * " & _
               "FROM YourTableName " & _
               "WHERE Registration =" & RId & ";"
    End If

    Me.RecordSource = sSQL

The above assumes that Registration in a Numeric Data Type.

If you are binding to the entire table, another approach would be to simply filter based on the combo box's value.

scottgem say:

Access is a relational database. Data stored in an Access database needs to be stored according to a set of rules designed for relational databases. These rules are called Normalization. Before creating a database you need to do some research on this.

A single table is not appropriate in your model. Owners can own more than one car. I see the need for at least three tables probably more:

tblOwner: OwnerID (PK autonumber), Surname, firstname, other info about owner

tblVehicle: Registration (PK), OwnerID (FK), DateIssued, CarMakeID (FK), BadgeNo, Location, ExtensionNo

tblCarMake: CarMakeID, CarMake

Not knowing what BadgeNo, Location or ExtensionNo mean you may need more tables.

If you use a split form, you can use datasheet filtering. Each column in the datasheet part of the form has a pull down with filtering options. So you could pull down the registration column and filter for the registration number.

If you use a regular form, you can use the Combobox wizard to create a "search" combo (3rd Option) where you can select the Registration number and it will bring up the matching record.

ken sheridan say: Having decomposed your table into a set of correctly normalized tables as described by Scott, you can create a form based on a query which joins the tblVehicle and tblOwner tables on OwnerID, and returns all the columns from tblVehicle along with whatever columns from tblOwner you wish to show in the form.  Note, however, that the owner's names will be shown in a combo box bound to the foreign key OwnerID column from tblVehicle.  Other controls in the form bound to other columns in tblOwner should be made read-only by setting their Locked property to True (Yes) and their Enable property to False.  Similarly the car make will be shown in a combo box bound to the foreign key CarMakeID column from tblVehicle; there is no need to include tblCarmake in the query.

In the form header add an unbound combo box, named cboGoToRegistration say, with a RowSource property of:

SELECT Registration FROM tblVehicle ORDER BY Registration:

In the combo box's AfterUpdate event procedure put the following code:

    Const MESSAGETEXT = "No matching record"
    Dim ctrl As Control
    
    Set ctrl = Me.ActiveControl
    
    If Not IsNull(ctrl) Then
        With Me.RecordsetClone
            .FindFirst "Registration = """ & ctrl & """"
            If Not .NoMatch Then
                ' go to record by synchronizing bookmarks
                Me.Bookmark = .Bookmark
            Else
                MsgBox MESSAGETEXT, vbInformation, "Warning"
            End If
        End With
    End If

I've assumed that the Registration  column is of text data type.  If it's a number data type amend the code as follows:

            .FindFirst "Registration = " & ctrl

To find a car by its registration number the user can either select from the combo box's drop down list with the mouse, or type in the registration number.  If the latter is done the first matches will progressively be selected in the control as each character is entered.

If you are unfamiliar with entering code into a form's or control's event procedures, this is how it's done in form design view:

1.  Select the form or control as appropriate and open its properties sheet if it's not already open.

2.  Select the relevant event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new lines between these.

In the form's Current event procedure put the following code to keep the unbound combo box in sync with the form:

    Me.cboGoToRegistration = Me.Registration

You will also need to create simple form's for inserting or editing rows in the tblOwner and tblCarMake tables.  To insert a new car make not currently in the database you can do so by typing the new make directly into the car make bound combo box and putting code in the combo box's NotInList event procedure to insert a new row into the tblCarMake table.  You cannot do this with owner names, however, as personal names can legitimately be duplicated, but you could add a button to the form to open the owners form in dialogue mode for you to enter a new owner.  You'll find examples of both approaches in NotInList.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

You might also like to take a look at the DatabaseBasics demo from the same OneDrive folder.  This little demo file illustrates some of the basic principles of relational database design and how to build forms etc to interface with the data.
