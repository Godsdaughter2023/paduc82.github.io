Problem:

The other day we migrated a user from Win7 to Win10.  Whenever the user would logon, explorer would crash constantly and you couldn't do anything.  However, if other users logged on it wasn't a problem.  If that user logged onto a different Win10 system, the problem would follow him.

Solution:

We use Work Folders with Win10 anniversary addition at our company.  We have had great success with Work Folders with a bump or two in the road but still very satisfied.  It turns out, this person had a shortcut that pointed to an invalid network location that was trying to sync down to his Work Folders folder.  I should also mention that we have redirected the desktop to point to a Desktop folder in Work Folders.  For some reason I am unsure of, this shortcut was causing Explorer to crash.  Since Work Folders was constantly trying to sync it down from the network, explorer was constantly crashing since the file copy was never successful.  After identifying the problem shortcut and deleting it, the crashing has stopped.

Notes:

In event viewer, I not only saw explorer.exe crashing but also searchUI.exe.

I doubt anyone else will run into this very odd issue but if you do I hope this helps.

I wanted to give everybody an update.  We are still working on an official fix through a KB download or a windows update, although I cannot provide an ETA.  However I can provide a very unofficial fix now.  Warning, the steps are complex.

 

Step 1:  Upgrade to the fall creators update and have the latest updates from Windows update.

 

Step 2:  Download devcon.exe

There are a couple ways to get this tool.  One way is by installing the Windows Driver Kit (WDK).  You can install it here:  https://developer.microsoft.com/en-us/windows/hardware/windows-driver-kit

After installing it you will find devcon.exe in “%SystemDrive%\Program Files (x86\Windows Kits\10\Tools\x64” for 64 bit versions or in “%SystemDrive%\Program Files (x86\Windows Kits\10\Tools\x86” for 32 bit versions.

 

An alternate way to get it is to use the steps documented here:  https://superuser.com/questions/1002950/quick-method-to-install-devcon-exe

To summarize the steps:

For 64 bit:

Download

Extract the file in the cab “filbad6e2cce5ebc45a401e19c613d0a28f” and rename it to devcon.exe.

 

For 32 bit:

Download

Extract the file in the cab “fil5a9177f816435063f779ebbbd2c1a1d2” and renamed it to devcon.exe.

 

Step 3:  Insert the device.

Wait 30 seconds

 

Step 4:  Open a elevated Command Prompt

Right click on the Windows Start Button

Select Windows Powershell  (Admin)

Select Yes to run Powershell with Admin privileges

Type the command “CMD” to switch powershell to Command Prompt

 

Step 5:  Verify the device has a 169.x.x.x IP Address

Run the command ipconfig

You should see a the device listed as a 169.x.x.x address.

(The nature of this bug is the device is not getting a proper 192.168.x.x address)

 

Step 6:  Run Devcon.exe to reload the driver

Switch to the directory that contains devcon.exe.  Such as cd “%SystemDrive%\Program Files (x86\Windows Kits\10\Tools\x64”

 

Run Devcon.exe status "USB\VID_12D1&*"

Run Devcon.exe update %SYSTEMROOT%\INF\wceisvista.inf "USB\Class_e0&SubClass_01&Prot_03

 

Wait 10 seconds

 

Run Devcon.exe remove "USB\VID_12D1&*"

Run Devcon.exe rescan

 

After 20 seconds the device should now be functioning normally.  If you run ipconfig again you should see the correct IP address.

 

 

Please let me know if this works for you or not.  If these steps are too complex then please wait for the official update.  I do not have a date when that update will be available.
