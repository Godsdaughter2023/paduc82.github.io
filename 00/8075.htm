dbOpenDynaset: is it safe for split database with multiple user - Microsoft Community

chris parker say:

dbOpenDynaset: is it safe for split database with multiple user access?

Hello, all. I am just a the point where I am going to split my database into a front end and a back end. I have it working the way that I want it to on my computer, but of course, during testing, I have been the only person using it. Once I release it to my staff, there will be several people using it. Before I do that, I want to make sure that I haven't done anything "unsafe" with it, particularly in the case where there may be a race condition between two or more users trying to access a table/record.

All of my code that makes changes to the tables uses transactions (DBEngine.BegineTrans, CurrentDb.Execute strSQL, DBEngine.CommitTrans, etc) with a rollback when an error occurs during the final commitTrans step, so I feel safe there.

However, when I retrieve data to populate my forms (I use unbound forms), I am using DAO.Recordset objects that I am opening as follows:

Set myRecSet = DBEngine.Workspaces(0).OpenDatabase(CurrentDb.name).OpenRecordset("tblMyTable", dbOpenDynaset)

Once I have opened the Recordset, I'm just moving to the first record (myRecSet.MoveFirst), and then find the first item with the primary key value that I am looking for with FindFirst. Assuming a record is found, I copy the record's fields to my table's textboxes and then close the Recordset.

It all seems pretty innocuous, but the "dyna" in dbOpenDynaset makes my naive brain wander to dynamic access, and then I start worrying about multiple users causing corruption by monkeying with the same data at the same time. I imagine that the database engine takes care of protecting my data, but I want to make sure that I am not inadvertently telling the database not to worry about such things!

Could someone set my mind at ease about this?

Thanks,

-Chris

dirk goldgar say:

Dynaset-type recordets are fine in a split database -- in fact, they're the default when working with linked tables.  However, there are some peculiarities in the methods you describe that bear discussing.

Your use of unbound forms in the way you describe means that Access can't lock records and display a conflict dialog if, say, one user loads a record and begins editing, and then another user loads the same record, edits it, and saves it,  and then the first user saves the record.  I don't think that Access would recognize that the record was changed by another user and display any resolution dialog -- the last user to save would overwrite the other user's changes without notice.  That's fine if that's the behavior you want, but it may not be.

Aside from that, a couple of your statements puzzle me a bit.  First, this:

Set myRecSet = DBEngine.Workspaces(0).OpenDatabase(CurrentDb.name).OpenRecordset("tblMyTable", dbOpenDynaset)

This seems to me to involve way more overhead than you need.  Why open a new Database object on the current database?  Why not just:

Set myRecSet = CurrentDb.OpenRecordset("tblMyTable", dbOpenDynaset)

?  You may have some reason behind this, but at the moment I don't see the point.

Then there's this:

Once I have opened the Recordset, I'm just moving to the first record (myRecSet.MoveFirst), and then find the first item with the primary key value that I am looking for with FindFirst. Assuming a record is found, I copy the record's fields to my table's textboxes and then close the Recordset.

If you only want a specific record, it would be much more efficient to open the recordset on a SQL statement that returns just that one record; e.g.,

Set myRecSet = CurrentDb.OpenRecordset( _

    "SELECT * FROM tblMyTable WHERE MyKeyField = " & MyKeyValue, dbOpenSnapshot)

Then, if the recordset has any records, load the form controls from the recordset and close it.  Note that I changed the recordet from Dynaset to Snapshot, because you're not going to be updating the recordset and I believe a snapshot will have less overhead.

hansv mvp say:

You can't directly copy cells from two rows (301 and 302) into one...

Sub Loopy()
  Dim wsh As Worksheet
  Dim r As Long
  Dim m As Long
  Set wsh = Worksheets("Sheet3")
  m = wsh.Range("A" & wsh.Rows.Count).End(xlUp).Row
  For r = 11 To m
    Worksheets("Sheet1").Range("B8") = wsh.Range("A" & r)
    Worksheets("Sheet2").Range("E301:G301").Copy _
      Destination:=wsh.Range("FE" & r)
  Next r
End Sub

    Worksheets("Sheet2").Range("E301:G301").Copy _
      Destination:=wsh.Range("FE" & r)

    Worksheets("Sheet2").Range("E301:G301").Copy
    wsh.Range("FE" & r).PasteSpecial Paste:=xlValues
