Hello dkjedi,

When Media Center records a TV show, it stores the file in the Public Recorded TV folder on your primary hard disk (usually located at C:\Users\Public\Recorded TV). If you have another hard disk installed in your computer, or an external storage device, you can change where Media Center stores recorded TV shows.

1.      Click the Start button, click All Programs, and then click Windows Media Center.

2.      On the Windows Media Center start screen, scroll to Tasks, click Settings, click TV, click Recorder, and then click Recorder Storage.

3.      Under Record on drive, click the Plus‌ or Minus‌ button to choose a new drive for Media Center to save recorded TV shows to. (If there are no other drives installed in or connected to your computer, these buttons will be unavailable.)

4.      Click Save.

Tips

If you're not sure how much free space a particular drive has, select the drive, and then check the chart under Total recording time before saving.

To limit the amount of hard disk space that Media Center can use for recorded shows, change the Maximum TV limit setting to a lower number.

techfreak_ say:

The Skype for Windows 10 App switched from using a Skype.db file (not the same as a main.db) in 12.x to a s4l-skypename.db in Skype 14.x. You could attempt scrolling through all your almost 2 years worth of history (~mid April 2017) using that specific version. Upon doing so you could attempt to duplicate the file and devise a way to export the data from it. If any chat message data exists, it would be in the messages12 table and nsp_data column viewable, with any SQLite Viewer. I have no idea when they will provide a tool to perform a similar task.

Skype for Windows desktop 8.x does not utilize SQLite database files. Any cached data is stored in LevelDB format (.ldb) and .log format. Since I am unaware of a compatible viewer, you can either view the collected data in something like WordPad, searching for strings, or create software to parse the files for specific data. I have noticed that if you scroll through your cloud history, the current .log file will update with that history. You could archive that data and extract what you need.

There is no guarantee the data you are looking for is currently present in any locally cached files. These files can be affected by updates and removals. All you can to is scroll through the cloud history to download data from the cloud, attempt to make a copy (open/closed), then see if you can extract/archive the data you want.

This is how it works now.  Who knows how it will work in the future?
