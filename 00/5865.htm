Hi Gary,
Whenever I get a new database I first review the database design, because it is the foundation that everything else is built on top of. At this time I'm only focused on major structural issues and major design challenges.

Following are must-fix items from my perspective. This is not just opinion, but based in the rules of normalization and set theory.
1. Change primary key from double to long integer. Double is an inprecise data type ("approximate numeric") and thus not suitable as a PK. In your case you may want to keep RecipeNum as a double, but then you have to introduce a RecipeID Autonumber as the PK. This will also require fixing up all FK's that are currently linked to RecipeNum. This can be done with update queries.
2. PK values should not be shown to end users, in most cases. Rather they should be meaningless numbers that are there for structural integrity only.
3. tblRecipe contains a "repeating group" with regards to pricing. Pricing levels should be spun off in their own child table, 1:M related to the parent table. An exception would be if this db is specifically written for a POS system that we will keep for the lifetime of this application, and that always will support 4 prices and no more. The same is true for tables such as tblMicrosConddef: if the unusual field names reflect the POS system you have my blessing. It may be useful to use the Description field in table design to further describe the field (e.g. 1-1 = Egg Style).
4. tblRecipe contains fields such as Price@1025 which I don't fully understand at this time but are possible calculated fields and as such belong in queries, not tables. I know we have a Calculated Field data type now, but it should not be used here.
5. tblRecipe contains calculated fields such as ImagePath with literal values "C:\PMWeb\Images\" & [RecipeNum] & ".jpg". The path to images should not be hard-coded but be in a SystemSettings table and the full path to an image should be calculated in a query.
6. Speculating, I think tblRecipe.[Photo?] is a calculated field. It can be eliminated or calculated in a query by checking if ImagePath is null. Similarly [No Allergen?] can be calculated from Allergen being null.
7. I don't understand the child tables such as tblMicrosGeneral well enough yet, but it appears you are duplicating information. Several fields are shared between this child table and its parent, and that is most likely wrong. If those parent values are needed, they can be looked up using a query. Only the FK is needed.
8. Use the most precise data type. tblRecipe.Price1 should be Currency, not Single.

In the category of recommendations:
1. Don't use spaces or other funny characters in object names. They may cause notational difficulties.
2. Don't use reserved words such as tblRecipe.Name as object names.
3. Don't use negative logic like [No Allergen?]. It always causes me to pause and ask what exactly a checkmark means.
4. You could probably use more required fields, e.g. tblRecipe.Price1.

Sorry this does not directly answer your question about copying a recipe, but in my view it is a necessary precondition.

scottgem say:

The Copy command does not commit the record, basically what it does is fill in the controls on the form. If you tried to commit the record without changing the PK it wouldn't allow it. 

But when you try to append with a SQL query, that is committing the record.

You have a choice. You can either use an Autonumber, or you can generate a new ID BEFORE you append it. Either way will work. So you can use the latter method if you want to preserve your ID. However, using Double Precision number as a PK is NOT a good idea. Double precision numbers are accurate to (I think) the 15th decimal place. So its possible to have 1216.5 as a visible number but what is stored is 1216.50000000000003. This will not match to 1216.5!

If you will have 130 fields, then it is highly likely your database is not normalized properly. It is highly likely that you are encoding data using field names. You should be striving for a more vertical structure rather than horizontal.

jay freedman say:

You can use the macro below to change ActiveX check boxes to plain text. This one is more complicated than the one for content controls, for two reasons: Simply deleting an ActiveX object doesn't leave a plain-text copy of its contents the way a content control does; and VBA access to ActiveX objects is inherently more complex because they aren't really Word objects but are sort of shoe-horned in.

This version looks at all the inline shapes, because ActiveX check boxes belong to that collection (along with pictures that are inserted in-line). For each one, it looks at the OLEFormat.ClassType. Pictures don't have an OLEFormat property, so just trying to look at it causes an error; the On Error statement makes the macro skip to the end of the loop.

If there is an OLEFormat property, the macro checks its ClassType. Only check boxes are "Forms.CheckBox.1", while other kinds of ActiveX controls have different ClassType values.

Once we're sure we have an ActiveX check box, its Object property is assigned to oCtrl. We store its Range, Value, and Caption in variables.

If the Value is True, we set the rg.Text to "2611" (the Unicode value of the checked-box character in the Segoe UI Symbol font). Doing this replaces the entire ActiveX control with the number, just like selecting the control and typing the number would do. Then the ToggleCharacterCode command replaces the number with the checked-box character (like pressing Alt+X).

If the Value is False, we do the same thing but with "2610", the number for the cleared-box character.

Finally, we add the original caption saved from the ActiveX control as text after the box.

When the loop finishes, the content of the document is copied to the clipboard.

Here's the macro:

                     

Sub ConvertCheckActiveX()
    Dim oCtrl As Object
    Dim lgIdx As Long
    Dim rg As Range
    Dim bChecked As Boolean
    Dim strCaption As String

    With ActiveDocument
        For lgIdx = .InlineShapes.Count To 1 Step -1
            On Error GoTo skipILS
            If .InlineShapes(lgIdx).OLEFormat.ClassType = "Forms.CheckBox.1" Then
                Set oCtrl = .InlineShapes(lgIdx).OLEFormat.Object
                Set rg = .InlineShapes(lgIdx).Range
                bChecked = oCtrl.Value
                strCaption = oCtrl.Caption

                With rg
                    If bChecked Then
                        ' checked box
                            .Text = "2611"
                    Else
                        ' cleared box
                            .Text = "2610"
                    End If
                    .Select
                End With
                Selection.ToggleCharacterCode        

                Selection.InsertAfter " " & strCaption
skipILS:
            End If
        Next lgIdx

        .Content.Copy
    End With
End Sub

ken sheridan say: The number of rows inserted into the table will increase exponentially every time the query is executed.  As more rows are inserted then the query will return that number of rows, so if the table originally contains one row only, the query will insert one row, then 2, then 4, 8,16 and so on.

To insert a single row each time:

PARAMETERS [Forms]![AFS]![Plan_ID] LONG,[Forms]![AFS]![AFS_ID] LONG,
[Forms]![AFS]![AddendumEffDate] DATETIME, [Forms]![AFS]![Client_ID] LONG;
INSERT INTO AFS_Addendum (Plan_ID, AFS_ID, [Addendum Effective Date], Client_ID)
VALUES([Forms]![AFS]![Plan_ID], [Forms]![AFS]![AFS_ID],
[Forms]![AFS]![AddendumEffDate], [Forms]![AFS]![Client_ID]);

I've had to guess the data types for the parameter declarations on the basis of their names, so you might need to change them if I've guessed wrong.

    Next PPSlide
   
    'loop through shapes on slide master
    For Each PPShape In PPPres.SlideMaster.Shapes
        If PPShape.Type = msoLinkedOLEObject Then
                 With PPShape.LinkFormat
                    SrcName = .SourceFullName
                    IndexNum = InStr(1, SrcName, "!")
                    'reset source file
                    If IndexNum > 0 Then
                        NewPath = MyXLFilePath & Mid(SrcName, IndexNum, Len(SrcName) - IndexNum + 1)
                    Else
                        NewPath = MyXLFilePath
                    End If
                    .SourceFullName = NewPath
                    'refresh links
                    .Update
                End With
        End If
    Next PPShape
         
'SAVE UPDATES AND SAVE AS PDF
    SelfName = Range("SelfName").Value
   
    PPPres.Save
    PPPres.Activate
    PPApp.Run PPPresName & "!PrintToPDF.PrintToPDF", MyPPFilePath, ClosePPApp, SelfName

'CLEAN UP
    MyPPFile = Dir()
    PPPres = Nothing
    PPApp = Nothing
    ClosePPApp = False

MyWorkbook.Activate
MsgBox ("Report Created!")
 
End Sub

Function FileOrFolderExistsOnMac(FileOrFolder As Long, FileOrFolderstr As String) As Boolean
'By Ron de Bruin
'30-July-2012
'Function to test whether a file or folder exist on a Mac.
'Uses AppleScript to avoid the problem with long file names
    Dim ScriptToCheckFileFolder As String
    ScriptToCheckFileFolder = "tell application " & Chr(34) & "Finder" & Chr(34) & Chr(13)
    If FileOrFolder = 1 Then
        ScriptToCheckFileFolder = ScriptToCheckFileFolder & "exists file " & _
                                  Chr(34) & FileOrFolderstr & Chr(34) & Chr(13)
    Else
        ScriptToCheckFileFolder = ScriptToCheckFileFolder & "exists folder " & _
                                  Chr(34) & FileOrFolderstr & Chr(34) & Chr(13)
    End If
    ScriptToCheckFileFolder = ScriptToCheckFileFolder & "end tell" & Chr(13)
    FileOrFolderExistsOnMac = MacScript(ScriptToCheckFileFolder)
End Function
