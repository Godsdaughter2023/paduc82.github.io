I managed to find a solution for this.

I created DimDate using instructions from this website. Then, I converted DimDate and my input data into tables using Power Query and after that I just created new excel connection and used following query:

SELECT ...

FROM `data.xlsx`.`DimDate$` `DimDate$`, `data.xlsx`.`Transactions$` `Transactions$`
WHERE `Transactions$`.`Period Begin Date` <= `DimDate$`.Date AND `Transactions$`.`Period End Date` >= `DimDate$`.Date

macropod say:

Hyperlinks have three basic attributes, the:

• url/filepath
• display text
• tooltip text

I've already told you how to manage both the first and last of these. Neither the display text nor the tooltip text need display the url/filepath.

The display text can be modified by right-clicking the hyperlink and choosing Edit. In the dialogue that pops up is an option for setting the display text. Do note that editing the display text this way will turn any embedded fields, such as the {FILENAME \p } field back into their output, so you'll need to re-do that part if you edit the display text.

Workaround for runtimebroker.exe high CPU usage in Groove - Microsoft Community

the_d0lph1n say:

Workaround for runtimebroker.exe high CPU usage in Groove

For a long time, I had the issue where if I set Groove to look in the Onedrive Music folder for songs, the Runtime Broker (runtimebroker.exe) will constantly consume between 25% and 45% of the CPU (Core i5 Surface Pro 3) while playing music. This seems to be a result of OneDrive constantly syncing with Groove. The result is that my battery life is destroyed from the CPU running at full speed constantly. Removing the OneDrive Music folder from the Groove library solved the high CPU usage, but then I don't have much of my music collection anymore (especially songs in FLAC files). This issue was reported in this thread too. This issue happened across multiple Groove versions and Windows 10 Fast Ring builds, as well as on the regular 1511 version.

The Workaround:

Don't minimize Groove. The Groove window doesn't have to be visible; another window can completely cover it, but do not minimize Groove to the taskbar. As long as Groove is not minimized, the high CPU usage will not occur even when the OneDrive Music folder is in the library. All functions will work properly and CPU usage stays in the single digits. This workaround allows me to enjoy my full music collection on Groove without having only 3 hours of battery life and a Surface that is too hot to touch.

Speculation on the cause:

This is pure speculation on my part, but I suspect that minimizing Groove places it in the background state, which is a state that is supported in UWP apps, and one of the tasks that runs in the background is syncing with OneDrive. By not minimizing Groove, it never goes in the background state, and the CPU-hogging background sync task doesn't run. This is just speculation based on what I know about the UWP app model and the background APIs, but I recommend that the Groove developers check what happens when the app is minimized and fix the CPU usage issue.
