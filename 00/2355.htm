a. user say:

As John says, it's not straightforward. Microsoft Word doesn't really do  this type of "one-to-many" merge out of the box, and some of the approaches that work on the Windows version either do not work on the Mac version at all or would require a lot of manual intervention.

I took my time on this because it is some time since I checked the situation on Mac Word 2008 and I wanted to be sure that the approaches I'm about to describe actually work on that version.

The one approach that can be used without VBA/Applescript etc. on all the desktop versions of Word (going back quite a long way!) is the one outlined in the following Knowledgebase article:

https://support.microsoft.com/kb/294686

This approch is covered in depth by macropod in his "Microsoft Word Catalogue/Directory Mailmerge Tutorial at

http://windowssecrets.com/forums/showthread.php/154370-Microsoft-Word-Catalogue-Directory-Mailmerge-Tutorial

I think you will have to sign up to that site to get that, but there is also a version here...

http://www.gmayor.com/Zips/Catalogue%20Mailmerge.zip

I have also provided two samples based on that technique.

However, that approach basically assumes that your data is in a single "flat file" that contains all the student and grade data needed by the merge. Typically it would have one row per student+subject, with any student-related data repeated in all the rows for that student. Usually, it would also be sorted into the sequence you need for the merge, although Word Mailmerge's own sequencing facility may do the sorting you need.

What isn't completely clear to me is where you are getting your data from. Your comment about database categories suggests that you might be getting it from something like FileMaker, where it would typically be stored in two separate tables (perhaps Students and Grades). In that case, 
 (a) you may be better off using the reporting facilities in whatever database product you are using, rather than going via Word
 (b) to use the approach mentioned above, you first have to create the flat file data source that you need.
 (c) there is another approach that uses one table as the mail merge data source (In this case, it would be a table with one row for each Student) and uses a DATABASE field to insert the Subject/grade data for each student. That approach suffers from some problems even on Windows Office where the data retrieval features are generally rather better than on Mac, but on Mac whether the approach is usable at all depends very much on your data source(s). For example, if the data is in an Excel table, you would have to respond to Mac Word's Excel converter prompt for each student. I don't know what the situation is if the data is inside FileMaker, but if you can export each table's data to a separate Word file that could be used as a data source, then you have a fighting chance of using this technique. That said, you get very little control over the layout of the table that is inserted for each student, especially if the table splits over more than one page.


But here are some samples anyway. You will need to download the ZIP from

http://goo.gl/0piLyT


and unzip the files into a suitable folder on your Mac's desktop. For each sample, you will need to open the mail merge main document and use the Mail Merge Main Manager to set it up as a Mail Merge Main Document and connect the appropriate data source. In the case of the two examples that use a DATABASE field, you may need to modify the field to include the full pathname of the data source that contains the grades.

Student grades sample 1.docx shows the basic technique for a one-to-many merge using "ordinary field codes. Have a look at the source data, which is in Grades1.xlsx (if for some reason you do not have Excel, you can use Grades3.docx instead)

Then you will need to
 - open "student grades sample 1.docx"
 - open the Mail Merge Manager (MMM)
 - in step 1 of the MMM, set the document up as a Catalog merge
 - in step 2, select the data source (grades1.xlsx).
 - you should be able to skip to step 6 and merge to an output document.

One consequence of this technique is that, because you have to use a Catalog merge, you cannot merge directly to a printer or send an email to each student with their results. You would have to code that separately. Also, if you want to display the data in a table, the best that you can reasonably easily achieve using this technique is a "table" that actually consists of one Word table per data row (plus the header).

Student grades sample 2.docx shows how you can insert text after the table of subjects+grades. This example uses a "dummy record" at the end of the data, which is in grades2.xlsx. If you need to use grades3.docx, you will have to add one row to the end of the data with Student set to (say) "zzz". Other than the different data file, the steps to use this sample are the same as above.

Student grades sample 3.docx shows how you can use a DATABASE field to list the subjects/grades per student. In this case, the Mail Merge Data source has to be a list of the students, with one record per student. So the procedure is
 - open "student grades sample 3.docx"
 - open the Mail Merge Manager (MMM)
 - in step 1 of the MMM, set the document up as a Form Letters merge
 - in step 2, select the data source (grades3.docx)
 - if necessary, modify the name of the data file in the DATABASE field to be the complete path name of the grades3.docx file, e.g.

Macintosh HD:Users:yourname:Documents:strudents3.docx

or whatever.  Notice that if you preview the merge, the results of the DATABASE field probably will not change. You have to perform the merge.  When you merge, you will probably also notice that the database column widths may change for each student. You can try inserting a \*Mergeformat switch to prevent that, but it may not be effective and you may find that it breaks down if you have more data and it spills over to the next page (not likely with this particular type of data, I would have thought, but perhaps you are actually doing something more complex).

I believe that you should be able to print or email this output, but you should check that the data generatedby the DATABASE field appears correctly.

Student grades sample 4.docx shows what happens if you try to use an Excel file as the data source for a DATABASE field. Set it up the same way as sample 3, but use grades3.xlsx instead. Edit: one way you may be able to avoid the Excel converter prompts is to set up another Mail Merge Main document that uses the same data source that you want to use in the DATABASE field. Open that Mail Merge Main document and respond to the Excel prompts. Then leave it open and open the Mail Merge Main document that contains the DATABASE field.

Please let me know if you cannot get the download, or find errors in the samples - I haven't really had time to re-check them.

macropod say:

Typically it would have one row per student+subject, with any student-related data repeated in all the rows for that student. Usually, it would also be sorted into the sequence you need for the merge, although Word Mailmerge's own sequencing facility may do the sorting you need.

The only thing my tutorial's approach requires to be repeated on every row in whatever is used for the 'key' field(s). All the other rows require only whatever you need to output for that row(and even that could be worked around). Likewise, the data do not need sorting if you configure Word's mailmerge to do that for itself (not sure if Macs can do this).

An alternative to using a mailmerge would be to use a DATABASE field. Again, I'm not sure of Mac support. For tabular output you could also combine a DATABASE field with a mailmerge, but that's really getting beyond most users' abilities...

  1. Use "File, Save As" to create a copy of your notebook at any file system location (local within My Notebooks or on a file share or ....); using a new name.

Hi Paul,

The answer is no.

To manage the whole Office 365 tenant, you need the global admin permission. However, you don’t have to assign an Office 365 license to the user. If the global admin wants to use Exchange Online service, he must have a mailbox.

In this situation, since you want to use this alias to manage your Office 365 tenant, you need to create a new user named the alias and  assign the user related permission. However, the alias cannot be a contact.
