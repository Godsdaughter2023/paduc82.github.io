john w. vinson say:

Are these Microsoft Works databases? If so, no version of Access has ever been able to open them natively. You can open and save the "database" in Excel, as a spreadsheet, and import the data from Excel into Access; but the user interface will have to be redone - Works and Access have very different architecture and are not compatible.

 

ken sheridan say: As far as I'm aware  Access has never been able to import or link to Microsoft Works database files.  You need to first export the Works file to a file format which Access can import or link to, such as a delimited text file.

Works databases are very simple in structure, more akin to a spreadsheet than a true database, so if you have Access installed you may well want to consider importing the Works data into Access via an intermediary text file and then working entirely in Access, which is a far more powerful database management system and will enable you to do far more with the data than you can in Works.  The Works files can then be ditched.

Unlike Works, Access is a relational database management system, so the likelihood is that the single table into which you import the Works data would best be decomposed into several related tables to eliminate redundancy.  Redundancy is not merely inefficient, but more importantly it leaves the database at risk of update anomalies, whereas a correctly 'normalized' set of related tables eliminates redundancy and therefore ensures the integrity of the data.

You'll find a demonstration of how a single table, in this case imported from Excel, but that's by the
 bye, can be decomposed into correctly normalized related tables as DecomposerDemo.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

You might have to copy and paste the text of the above (not copy the link location) into your browser; for some reason it sometimes doesn't seem to work as a hyperlink.

The file uses a series of 'append' queries to insert data from the 'master' table into previously created empty related tables.  The sequence is important.  In essence it's a question of firstly inserting rows into the 'referenced' tables (those on the 'one' side of one-to-many relationships) before inserting rows into the 'referencing' tables (those on the many-side of the relationships).

john mcghie say:

Hi Lisa:

Your problem is caused by "Mirror Margins", and it's not part of the Paragraph setting, it's part of the Section Break setting.

Go to Format>Document>Margins and uncheck "Mirror margins".  (On the Ribbon it's Layout>Margins>Custom...)

Mirror Margins swaps a 1 cm margin from left to right depending on whether you are printing a right-hand page or a left-hand page, to allow for the binding, when printing double-sided.

If you are not printing double-sided, turn it off.

Hope this helps

a. user say:

Hi,

Thank you for posting your query on Microsoft Community.

It must be difficult when things do not work as expected and I appreciate you for providing details about the issue and your efforts towards resolving it.

I would like to inform you that this is by default and if you plug in a head set, it will automatically changes the channels from 5.1 to default.

I suggest you to uninstall and reinstall the display and audio drivers and check if it helps to resolve the issue.

1. Press “Windows key + X” and click on “Device Manager”.

2. In the list of devices, you should see the Audio Drivers.

3. Right click on the device and then select "Uninstall".

4. In the uninstall dialog, make sure you select the option to delete the driver package from the system. This causes the driver package installed earlier to be deleted.

5. In the Action menu of Device manager, select “Scan for Hardware changes” option. This will cause the device driver to be reinstalled. Once the driver is installed, close all windows and restart the computer.
