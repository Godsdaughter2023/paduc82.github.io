djsteele, mvp say:
First of all, regardless of your Regional Settings, dates in Access query statements must be in mm/dd/yyyy format, or a non-ambiguous format such as yyyy-mm-dd or dd mmm yyyy. See Allen Browne's International dates in Access for more details.

That being said, as long as you're not trying to use = in your comparison, it should still work.

ken sheridan say: Changing the format in which the date/time values are displayed should make no difference.  Whatever format you use the date/time is stored as a 64 bit floating point number in which the integer part represents the days and the fractional art the times of day.  This makes the values unambiguous, so if I enter a date as 04/07/2113 and send the table to one of my American friends they will see it as 07/04/2113, i.e. it's 4th July to both of us.  It would actually be stored under the skin as 77983, which is because the date is 77983 days from 30 December 1899 00:00:00 which is the origin Access uses for date time values.

However, your system seems to be using a short date format of dd/mm/yyyy, as does mine.  Date literals in Access cannot be in this format, however, but must be in US short date format or an otherwise internationally unambiguous format.  The date literal #01/01/2013# will obviously work for 1st January because it is the same in dd/mm/yyyy and mm/dd/yyyy formats.  As it happens #21/03/2013# would also work as in US format it is not a valid date, so Access detects this and automatically  sees it as #03/23/2013#.  On the other hand for 5th February, #05/02/2013# would not work because in US format it is a valid date, so would be interpreted as 2nd May.

The upshot of this is that when using a date literal it must be in US format or otherwise internationally unambiguous.  I would recommend that you always use the ISO standard for date formatting of YYYY_MM_DD.   So 5th February 2013 would be #2013-02-05#.   Users can of course continue to enter dates in the normal day-to-day operation of the
There is one other point you need to consider.  If you use an operation in a query such as:

WHERE MyDate BETWEEN #2013-01-01# AND #2013-03-21#

a row with a date time value of 21 March 2013 12:52:29 would not be returned by the query.  This because #2013-03-21# is not a date value per se, but a date/time value of midnight at the start of the day, i.e. 21 March 2013 00:00:00, which is before 21 March 2013 12:52:29, so the latter is not within the specified range.  To avoid this the range can be defined differently:

WHERE MyDate >= #2013-01-01# AND MyDate < #2013-03-22#

i.e. on or after the start date and before the day following the end date of 21 March 2013.  When using parameters this would be expressed like this:

WHERE MyDate >= [Enter start date:] AND MyDate < [Enter end date:]+1

The user can now enter 21/03/2013 as the second parameter, and be  confident  that rows with a value of that date and any time of day will be returned.

Trevor wrote: "I know the current day cash value (e.g. 100,000) [....] I know the late I left the fund (e.g. 15-February-1989)".

You want to calculate the "present value" of today's 100,000 on 2/15/1989.

Unfortuntely, Excel does not provide a PVSCHEDULE function, the complement of FVSCHEDULE.

Suppose the average monthly rates are in B3:B352.

(I estimate the rate for 3/17/2018 as 1.75 percent by using FORECAST based on the previous 2 months, since the rate for that month will not be determined until 3/31/2018.)

And suppose that the fund applies the average monthly rate to the average daily balance and compounds monthly.

(Since you do not mention any external cash flows, I assume the average daily balance is simply the previous month's outstanding balance.)

Then the present value of today's 100,000 on 2/15/1989 is approximately 18,610.09, calculated by the following formula:

{ =100000 / PRODUCT(1 + B3:B352%/12) }

Note: That formula is array-entered. Do not type the curly-braces. Press ctrl+shift+Enter instead of just Enter to commit the formula.

More precisely, since 2/15/1989 and 3/17/2018 are not full months, the present value is about 18,759.71, calculated by the following:

{ =100000 / (1 + (A352-A351)*B352%/365) / PRODUCT(1 + B4:B351%/12) / (1 + (A3-A2+1)*B3%/365) }

where A3:A351 are the end-of-month dates starting with 2/28/1989, A2 is 2/15/1989, and A352 is 3/17/2018.

joeu2004 say:

As an aside....

Trevor wrote:

the problem I see is that the PRODUCT [....] would determine a single average rate across the 29 years and so not reflect actual compund growth needed.

Actually, the average compounded rate is calculated by GEOMEAN(1+B3:B352%/12)-1, which is the same as

PRODUCT(1+B3:B352%/12)^(1/COUNT(B3:B352))-1.

So another way to approximate the present value is with the following array-entered formula (press ctrl+shift+Enter instead of just Enter):

=PV(GEOMEAN(1+B3:B352%/12)-1, COUNT(B3:B352), 0, -100000)

That has about the same result as array-entered 100000/PRODUCT(1+B3:B352%/12) which, as I said before, is an approximation because the first and last two dates are not at the end of the month.

(The infinitesimal difference is probably due to binary floating-point arithmetic anomalies, which is no surprise, considering the very different arithmetic methods.)

Access requires TWO characters - line feed (chr(10)) plus carriage return (Chr(13)) - to indicate a new line.  Excel if I recall correctly only uses Chr(10) a line feed to indicate a new line in a cell.

Access cannot display a tab character (chr(9)) in fields and simply shows in place of actually tabbing.
