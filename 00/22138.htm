gord dibben say:

Excel 2007 has no AutoSave.    No version of Excel since 2000 has had AutoSAVE

 

2007 has AutoRecovery which creates and saves a temporary file while you are working on your open workbook.

 

The temp file is deleted after you successfully close that workbook.  No backup remains.

 

AutoRecovery is not a backup feature, only a recovery feature in the event of a crash.

 

If you want a real backup up of any saved file you must set that through...............

 

Office Button>Save As.....your type>Tools>General Options.

 

Checkmark "Always create a backup"

 

Now save the workbook............a backup will be created in the default Excel folder..........not in the AutoRecovery Folder.

Bug in Excel 2016 (Version 1707): Excel starts with no - Microsoft Community

govert-roningms say:

Bug in Excel 2016 (Version 1707): Excel starts with no default workbook or start page if an XLL is installed

By default, when you starts Excel from the Start menu, Excel is started with the Startup screen showing the File->New backstage screen. 

In recent version of Excel 2016 (I'm testing with 32-bit Version 1707 - 16.0.8326.2062), when an XLL Excel add-in is installed, Excel start up with no startup screen. In the past, installing an XLL Excel Add-in did not change this start-up behaviour, es expected. 

Similarly, if you've turned the Startup screen off (say, as described at https://support.office.com/en-us/article/Turn-off-the-Start-screen-1f2834b4-09d4-490d-9c04-5d9f1b25ff70), then starting Excel should create a default workbook called Workbook1. But, if you have an XLL Excel add-in installed, Excel starts with no default workbook opened.

To sum up, if any XLL Excel add-in is installed, Excel incorrectly opens in the "all workbooks are closed" state (when you start it from the Start Menu).

Detailed steps to create an XLL Excel add-in, install it and test:

  • Install Visual Studio 2017 with C++ tools.
  • Download and install the Excel 2013 XLL Software Development Kit from here: https://www.microsoft.com/en-za/download/details.aspx?id=35567
  • Open a "Developer Command Prompt for VS 2017" and then change directory to the SAMPLES subdirectory of the XLL SDK.
  • Rebuild the samples by entering "MAKE.BAT RELEASE BUILD X86" at the developer command prompt.
  • Check that the file SAMPLES\EXAMPLE\RELEASE\EXAMPLE.xll has been created.
  • Open Excel and press Alt t, i to open the Add-ins dialog. Press Browse and navigate to select the EXAMPLE.xll add-in. It should display as "Example Standalone DLL", with the checkmark next to it. Click OK - a message will pop up indicating that the add-in was installed.
  • Close Excel and reopen.
  • Notice that Excel has opened with no Workbook, and with the Recent / New backstage display that is normally displayed.
  • Create a new workbook, press Alt t, i to display the Add-ins dialog and remove the add-in.
  • Close Excel and reopen to confirm that everything is back to normal.

I have tested this with various different .XLL Excel add-ins, including native C++ add-ins like the sample from the official Excel SDK, and using add-ins created with the Excel-DNA library.

The problem was initially reported and discussed on the MSDN forum here: https://social.msdn.microsoft.com/Forums/office/en-US/2bb21bfb-4940-4a68-9656-743ae0b8b4da/bug-in-excel-2016-excel-starts-with-no-default-workbook-or-start-page-if-an-xll-is-registered?forum=exceldev

talking moose say:

You've done a lot of the heavy lifting in troubleshooting this problem. Your description is very thorough!

First, verify you're user is running the most current version of Office for Mac 2011. That should be 14.5.3, I believe. You can use Help > Check For Updates in any Office application to update your software.

Sounds like the document itself is not the problem if you've used Finder's Connect To Server to access the file from a file share. That rules out SharePoint too. I think you're right that something in the user's home folder is the problem. Let's test that theory first by creating a new OS X user account and connecting from the same Mac.

  1. Choose Apple menu > System Preferences > Users & Groups.
  2. Click the plus button ( + ) to create a new temporary test user on the Mac.
  3. Log out of the current account and log in to the new temporary test account.
  4. Use Word to access the same file on SharePoint.

Do you still see the problem? I suspect you won't.

If you don't then either user caches or font caches are likely the problem.

  1. Restart the Mac.
  2. Just after you hear the startup chime, hold the Shift key down until you see the progress bar under the Apple logo appear. You can release the Shift key.
  3. Wait for the Mac to come up to the login window or the user's desktop. This may take longer than normal.
  4. You should see something like "Safe Mode" in the upper right corner of the screen.
  5. Restart the Mac normally.
  6. Test accessing the document now.

Let's start with that. Let us know what you see and we'll keep troubleshooting if that doesn't help.

talking moose say:
Not including after you noticed the problem but had you rebuilt your identity within the prior six days? Outlook may have opened an old identity instead of your more current one.

Launch Outlook with the Option key held down. This will launch the Database Utility application. If your current identity is a "backup" then choose the identity that's not a "backup". Make it the default using the menu button at the bottom of the window.

Quit the Database Utility and launch Outlook normally.

Hope this helps! 

redseablue say: Yes, I had rebuilt the database six days before the upgrade.
