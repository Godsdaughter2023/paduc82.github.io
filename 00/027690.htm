diane poremsky mvp say:
I cant fix the servers but I can try to explain the messages. The errors relate to EWS (Exchange Web Services) API, which eM uses to sync.

Does only the one account generate errors?

Only the first 2 really have any meaning - the last ones relate to the backend.

"The specified object not found in store" - What does this mean?
An item - could be email, appointment, or contact cant be found in the mailbox but is in the index. This often indicates an item is corrupt, or was deleted but the index was not updated yet. If eM generates log files, it might hold clues as to what is triggering this.

"Can't connect to the mailbox of user" - What does this mean?
It can't connect to your mailbox on the office 365 servers.

"Mailbox database guid: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - What does this mean?
That is the database (within the vast server farms) where you mailbox is stored.

"the ExchangePrincipal object contains outdated information" - What does this mean?
Something in your account on the server is does not match other data. Its not really anything to worry about.

"The mailbox may have been moved recently" - What does this mean?
The mailbox may have been moved to a new database. This can happen for a number of reasons - server failure or server update is the most common reason.

julie fka profjulie say: Notifications for appointments are handled through the mobile device's notification queue. The notification queue is a type of database that is separate from the Outlook database (called PIM.vol).  Because data is stored in these two separate places, it is possible for the data in the Outlook database to be deleted and still have notifications fire.  

You can get a piece of 3rd party software like SKTOOLS, which will allow you to view, edit or delete the contents of the notification queue (open up SKTOOLS, tap View > Maintenance > Notifications Queue)

SKTOOLS will also let you view the number of records that are in your PIM databases.  Then you'll know for sure whether or not the data is gone for good or just hidden. (open up SKTOOLS, tap View > Maintenance > scroll down and choose Appointments, Contacts, and Tasks and see how many records are in each of these files.

If you see a count of 0 in these files, then your data is gone for good and there is no hope for recovery...And unfortunately, no good explanation about what could have caused this to happen.

But if you see counts that are more than 0 in these files, then the data is still there and is probably hidden behind a filter.

First of all, you can remove the Requery line: assigning a RowSource requeries the object.

farey say:

ken sheridan say:

Still i am not clear why it should go blank when data form is open in edit mode!

It's almost certainly because at this stage the RowSource property is returning FlightID values which do not include the current value of the control because it is restricted to an AirlineID value which does not relate to the current FlightID value.  The value of the control is still the correct one, but you can't see the corresponding referenced value in the control because of the use of a surrogate key, as there is no row returned by the RowSource property containing the key value.  This is not usually a problem in single form view because, as Tom says, you simply need to set the RowSource property in the form's Current event procedure, exactly as you are doing in the control's AfterUpdate event procedure.  It can be a problem in continuous forms view, however, where you need to resort to the use of a 'hybrid' control where a text box is superimposed on a combo box to give the appearance of a single control.

When correlating combo boxes like this I tend not to change the RowSource property at all, but reference the first control as a parameter.  In your case it would be:

    SELECT [Flights].[FlightID], [Flights].[AirlineID], [Flights].[FlightNum]
    FROM Flights
    WHERE [AirlineID] = Form!AirlineID;

Normally I'd also include an ORDER BY clause to sort the list.  Note the use of the Form property to return a reference to the current form.  You can do this as both combo boxes are in the same form.

You then don't have to change the RowSource at all, but merely requery the FlightID combo box with:

    Me.FlightID.Requery

Do this not only in the AirlineID control's AfterUpdate event procedure, but also in the form's Current event procedure.  It's also a good idea to do so in the form's AfterDelConfirm event procedure in case a record is deleted.

For examples of the use of correlated combo boxes in various contexts you might like to take a look at ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file differs from many examples you'll find elsewhere as it is specifically intended to illustrate the use of such controls in the context of a correctly normalized set of tables.  Consequently the examples might be a little more complex than you'll need.

farey say: Thank you so much. I just changed the RowSource of the combo control flightID to the code you have given and it is working like magic. Though i am not fully aware what problems may cause. Do i need to add Me.FlightID.Requery to the after update event of combobox AirlineID (i mean replace with Me.FlightID.Requery)?

ken sheridan say: Assuming that FlightID is the name of the second combo box you put Me.FlightID.Requery in two places at least:

1.  The AfterUpdate event procedure of the AirlineID combo box.
2.  The Current event procedure of the form.

If changing, adding or deleting a record in the form can change the values listed then you should also put it in:

3.  The form's AfterUpdate event procedure.
4.  The form's AfterDelConfirm event procedure.

If in doubt put it in the last two as well as the first two.  Even if not necessary it won't do any harm.
