tom van stiphout say:

Re #1:

Let's say both users are on customer #5; one changes the phone number, the other one the email address. #1 saves the record. #2 saves the record. What happens? Answer: the phone number edit will be undone, because #2 record overwrites it. This is completely normal and it works like that with all multi-user systems. Yes, there is esoterica like excessive record locking, but believe me it is not worth it; the above scenario will almost never happen.

Re #2: Last person to edit is relatively easy if you know just a little bit of VBA; in the Form_BeforeUpdate write the user's ID to a LastChangedUserID field. A full audit trail of every edit of every field is also possible but requires more skills. Search the web for "Access audit trail" and you may find some pointers. In A2010 and above there also are data macros at the table level that can be used for audit trail.

#3 has already been answered by Mimmo.

#4: Not for this scenario. Store the back-end on a file server in your network.

scottgem say:

Access is designed to be a multi-user platform. However, experience has determined that the best setup is a split database where the tables are stored in a separate file located on a shared network drive and the front end (everything but the tables) are stored on each user's local PC. Its that simple.

To be able to determine who made the last change is easy, but you need to have a method for determining the current user (see my blog on Login Security). If you want a history of all changes, then you need an Audit Trail. There are plenty of examples of such (including one on my blog).
