I looked at the Settings of the Onedrive desktop application and I saw that the option “Use Office to sync files faster and work on files...” was checked. What this process does is build a very large cache of indexed Office files on your local drive.  For me, the cache was ~5GB, or about equal to the combined size of my Office files.  That would be fine provided I had extra space available on my SSD, but I don’t.     

I discovered that simply turning off this option does not automatically delete the cache and free the space.  To recover the disc space I closed the Onedrive application and then deleted everything in C:\Users\UserName\AppData\Local\Microsoft\Office\15.0\OfficeFileCache.   Keeping the “Use Office” option turned off has resulted in a much smaller cache. 

MS needs to document the cause/effect of this option so people understand the impact it has on their system.

Hello Jacques,

I just saw this, and I can explain it. In 2013 we went to a more secure algorithm across Office - not just in Excel. Excel 2013 uses SHA-512. It makes only milliseconds of difference with one call, but in those cases where code runs, protecting and unprotecting many sheets, it all adds up to a terrible performance issue.

ken sheridan say: The use of aliases in Access queries is not quite so straightforward as might be thought.  A column name can be used as an alias provided that the column name is qualified by the table name, e.g.,

SELECT Contacts.LastName AS LastName
FROM Contacts;

A ludicrous example of course, but it demonstrates the point.

 It can also be done where two tables in a query have columns of the same name, but only one of those is returned, e.g.

SELECT Contacts.Lastname AS LastName,
FROM Contacts INNER JOIN Employees
ON Contacts.ContactID = Employees.EmployeeID;

Again a ludicrous example, but an allowable one.

Another allowable use is where two tables in a query have columns of the same name, and both are returned, but one is given an alias which differs from the column name:

SELECT Contacts.Lastname AS LastName, Employees.LastName AS Employee
FROM Contacts INNER JOIN Employees
ON Contacts.ContactID = Employees.EmployeeID;

Where it cannot be done is where two tables in a query have columns of the same name, and both are returned, but only one is given an alias of its own name, e.g.

SELECT Contacts.Lastname AS LastName, Employees.LastName
FROM Contacts INNER JOIN Employees
ON Contacts.ContactID = Employees.EmployeeID;

Ludicrous again, but the query will open, and return columns with headings LastName and Employees.LastName.  However. the first column, LastName, will be empty.

The real question, though, is why it would be considered necessary to give a column an alias of its own name, rather than one which distinguishes it from another column of the same name.  In a developed application queries' result tables would be presented to the user as forms or reports in which columns can be captioned with any name whatsoever, regardless of the column headings returned by the query.  If it's felt to be appropriate to return the result table as a datasheet, then a form bound to the query can be opened in datasheet view.

    Dim rst As DAO.Recordset
    Dim strSQL As String
    Dim strRS As String
    Dim n As Integer
    
    Me.Dirty = False
    
    ' get current RecordSource property
    strRS = Me.RecordSource
    
    ' set form's RecordSource to a zero-length string
    ' to allow change to table definition
    Me.RecordSource = ""
    
    ' drop index
    strSQL = "DROP INDEX PaymentIndex ON RegularPayments"
    CurrentDb.Execute strSQL, dbFailOnError
    
    strSQL = "SELECT PaymentIndex FROM RegularPayments ORDER BY PaymentIndex DESC"
    Set rst = CurrentDb.OpenRecordset(strSQL)
    With rst
        .MoveLast
        n = .RecordCount
        .MoveFirst
        Do While Not .EOF
            .Edit
            .Fields("PaymentIndex") = n * 10
            .Update
            n = n - 1
            .MoveNext
        Loop
    End With
    Set rst = Nothing
    
    ' reinstate index
    strSQL = "CREATE UNIQUE INDEX PaymentIndex ON RegularPayments(PaymentIndex)"
    CurrentDb.Execute strSQL, dbFailOnError

    ' reinstate form's RecordSource
    Me.RecordSource = strRS

2leon say:

random switch to desktop - moving mouse recovers open windows

computer switches to desktop from other applications at random .  It recovers after moving the mouse but it is annoying. Peculiar that it continues to register typing while on application (browser ) even that it shows the desktop
steven. s say:

Hey 2leon,

 

This can happen if your mouse is moved over the “Show desktop” button on the taskbar. When the mouse hovers over this button, you can “peek” at your desktop (this is what happens when your windows disappear). The Internet Explorer window is still open during the “peek” which is why your typing is still recorded.

 

You can disable the peek when the mouse is hovered over the button by:

·        Right click your taskbar

·        Click Properties

·        Uncheck the Use Aero Peek to preview the desktop box

·        Click OK

 

Hope that helps, look forward to hearing back from you.
