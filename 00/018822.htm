tom_bpc say:

Kim (and Trevor),

Although you appear to have taken the discussion off-line, the question is still marked as unanswered, so I'm chiming in with that caveat:

  1. As pointed out by Dale, Julie, and Trevor, project planning based on the Critical Path Method involves compliance with a number of "best practices" for networking logic, including having a single starting node (the only task with no predecessors) and a single finishing node (the only task with no successors).  Then:
    • Every other task in the project is a logical successor/descendant of the project's starting task and a logical predecessor/ancestor of the project's finishing task.
    • That collection of tasks - the delay of any single one of which leads to a delay of the project's finishing task - is the project's "Critical Path".  This is the chronological longest path through the logic network.
    • Under relatively simple scheduling conditions (i.e. No constraints or deadlines, no resource leveling, no variable task calendars, no "Calculate Multiple Critical Paths"), the Critical Path / Longest Path can be identified by Total Slack=0.

  2. MSP does not enforce compliance to "best practices", and when the "relatively simple scheduling conditions" above become more complex (e.g. deadlines, late constraints, variable task calendars), then Total Slack becomes unreliable as a Critical Path indicator.  Unfortunately, MSP 2010 provides no other (native) method to identify the longest path.

  • [In light of those issues, my firm developed an MSP Add-In (called BPC Logic Filter) for internal use and for sharing with our clients.  Our tool ignores Total Slack and simply traces the Driving Path for the project finishing task (i.e. Longest Path) or for any other task in the project (e.g. Kim's three building completions).  It's our go-to tool for Critical Path reporting on real-world complex projects.  We also use it for schedule logic review, including relative float analysis, Drag calculation, etc.]

Finally, with respect to Kim's "one red bar at the end of the project...."  That's a tell-tale indicator of a constrained completion task.  That is, the completion task has one of the following conditions, which should be removed:

  • As Late as Possible (ALAP) constraint. (In MSP, this constraint and "Critical Path Method" are totally inconsistent.)
  • Task Mode = Manual (Same comment)
  • A constrained or manually scheduled summary task any levels above (Same comment.  These can be hard to spot if you aren't looking for them.)
  • Start No Earlier Than (SNET) or Finish No Earlier Than (FNET) constraint (or MSO or MSF constraint).  This overrides the logic flow through the task, increasing the total slack of the task's predecessors, if any.

Less likely causes might be a driving predecessor that has been completed out of sequence, a unique and strange task calendar for the one visible task, or some very aggressive resource leveling....
