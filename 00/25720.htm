dale howard [mvp] say:

Chris --

The "Move/Completed Parts to Status Date" option you have selected conflicts with your process of manually entering progress.  You cannot have it both ways.  Either you use the "Move/Completed Parts to Status Date" function and Microsoft Project calculates the Actual Start date, or you manually enter the Actual Start date yourself.  Personally, I would never use the "Move/Completed Parts to Status Date" function because Microsoft Project makes assumptions about when the task must have actually started, and usually those assumptions are wrong.  Hope this helps.

trevor rabey say:

I think you are going wrong at the point where you say "During subsequent updates if I progress the task to a higher percentage" etc.

The purpose of "Move Completed Parts to Status Date" is to correct a task which appears to have actual duration (and by necessity and actual start) in the future relative to the status date.

Since any actual duration should be in the past, that's why MSP helpfully moves the actual part far enough to the left to ensure that it is all in the past, without you having to calculate how far that is.

It is not clear to me what you expect will happen when you "Move Completed Parts to Status Date", but it is doing exactly what it is meant to do.

Where you say "When a task line starts early, I fill in actual date in the "Actual Start" column.", I think you need to say "When a task starts, I fill in actual date in the "Actual Start column in the tracking table"". A task acquires an actual start date when it actually starts, regardless of whether it starts early or late or exactly on the scheduled start date. That actual start date must always be in the past.

Suppose a task has a scheduled (earliest) start date which is in the future relative to the status date, as determined by a predecessor, but somehow the task actually starts earlier than that date (in defiance of the predecessor). Lets' say that the actual start is at least 2 days before the status date. If the task has say 10 days duration, and you type in the actual start, and then type in that % Complete = 20%, then this is identical to saying that the actual duration is 2 days. The entire 2 days actual duration will be on the left of the status date. However, the remainder part still "honours" the predecessor, the task bar appears to be split. It becomes more obvious if you make the actual start and actual duration even earlier, and/or if you make the finish of the predecessor even later.

This is true whether the scheduled start date as determined by the predecessor is in the past or in the future relative to the status date.

I think this is what is happening for you when you fill in an actual start, as you say "early".

Another way to end up with a split task is to have actual duration in the past as well as some remaining duration. Since the remaining duration must be re-scheduled to asap in the future, you can use "move Incomplete parts to the status date", and there is the split. Actuals stay where they happened, remaining goes to where it can happen.

Two pieces of advice:

  • Always show the status date as a red vertical line on the Gantt Chart
  • Do not type in % Complete
