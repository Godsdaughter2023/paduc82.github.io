dirk goldgar say:
The database is a .mdb file.  Calculated fields were not available in that version of access?  Is there a way to do something similar with that file extension?


Calculated fields (in tables) are not available in the .mdb file format.  Heck, they aren't even supported for the .accdb format by Access 2007 -- they were introduced in Access 2010, and even though A2010 and A2007 both use the .accdb file format by default, calculated table fields are among the A2010 features that aren't supported by A2007.

 

The most common approach prior to A2010, and still often the best approach, was to create a query that defines the calculated field, and then base forms and reports on that query.  Calculated fields in queries have always been supported.

 

Calcculated fields in queries are created by entering the expression in field grid of the query's design view, as for example:

 

    ExtendedPrice:  Quantity * UnitPrice

 

Or, in the SQL view of the query, it would look like this:

 

    SELECT ..., (Quantity * UnitPrice) As Extended Price, ...

    FROM YourTable

lars-Åke aspelin say:

I have complex time issue that requires me to exclude off time and weekends. Work hours are 8:30 am to 5:00 pm. In cell A2 I have the date and time delivered and in cell B2 the date and time started. I need to find the time it took from delivered to started (C2) but I need to exclude off work hours ( 15.5 hours) and if necessary exclude weekends. Is this possible with a formula?

 

 

Date Del. Date Start Deliver to Start Time
7/18/13 10:00 AM 7/18/13 2:30 PM 4:30
7/18/13 10:00 AM 7/19/13 2:30 PM 13:00
7/19/13 2:30 PM 7/22/13 11:00 AM 5:00
7/19/13 2:30 PM 7/23/13 11:00 AM 13:30

Try this formula in cell C2:

=(INT(B1)-INT(A1)-(INT((INT(B1)-INT(A1))/7)+(WEEKDAY(B1,2)<WEEKDAY(A1,2)))*2)*(TIME(17,0,0)-TIME(8,30,0))+MOD(B1,1)-MOD(A1,1)

Note: Both deliver times and start times have to be within the working hours in order to get the correct result from this formula.

Hope this helps / Lars-Åke

jeeped say:
It may be best to thing of this formula as the SUM() of three parts.

     

The time on the day of delivery from the delivery to the start will either be a portion of the same day or from delivery until the end-of-business.

=MIN(B2,INT(A2)+TIME(17,0,0))-A2

The time on the day of starting from the delivery to the start will either be the same day or from the start-of-business until the start.

=(MOD(B2,1)-TIME(8,30,0))*(INT(A2)<>INT(B2))

Finally, there may be days in between that need to be accounted for (excluding weekends).

=MAX(NETWORKDAYS(INT(A2),INT(B2))-2,0)*TIME(8,30,0)

Put these three formulas together in a SUM() for,

=SUM(MIN(B2,INT(A2)+TIME(17,0,0))-A2,(MOD(B2,1)-TIME(8,30,0))*(INT(A2)<>INT(B2)),MAX(NETWORKDAYS(INT(A2),INT(B2))-2,0)*TIME(8,30,0))

It may be worthwhile to point out that NETWORKDAYS() can also exclude holidays. Click through the following link for more information.

NETWORKDAYS function

Changing primary key to include more than 1 field - Microsoft Community

chelonidae say:

Changing primary key to include more than 1 field

I needed to change my primary keys to include more than 1 field. In my case Year needs to be included to a unique Location ID# (current primary key)  to account for samples that have now been sampled repeatedly. I tried establishing these relationships using the same ID field that was used previously and using both fields that are the primary key (ie also adding the Year).

scottgem say:
Wrong approach. If you have multiple samples you need 2 tables. Your Locations table and a Samples table, Even if there is only one sample for a location as longs as some locations have multiple samples, the samples should be broken out int a separate table. LocationsID would then be a Foreign Key in your Samples table. 

chelonidae say:

Thanks Scottgem.

john w. vinson say:

You'd break the table apart the same way as if it had 10 records or 10,000,000: create a new table (or in this case, tables) and run Append queries to migrate the data.

It would appear that you have a standard one to many relationship: each Location may have zero, one, or many samples (over time); and each Sample will be taken at one location at one time. I don't know of course what information constitutes a sample, but your Samples table should have at least a SampleID (autonumber primary key, I'd suggest), LocationID (foreign key to the master Locations table), SampleYear (don't use the reserved word Year for a fieldname) or better SampleDate (Date/Time). Other fields that pertain to a location regardless of any particular sample would remain in the Locations table; information pertaining to an individual sample would be migrated into Samples.

ken sheridan say: One further point.  If you do include an autonumber SampleID as the primary key of the Samples table, it is important, if the combined values of  LocationID and SampleYear (or SampleDate if you record samples by a specific date) are distinct within the table, that these two columns are included in a unique index.  This defines them as a candidate key and prevents invalid duplication.  The alternative would be to not use a 'surrogate' SampleID key, and simply make the LocationID and SampleYear (or SampleDate) columns the composite primary key of the table.  A 'surrogate' key in a situation like this only serves a useful purpose if the table is referenced by another table as it avoids the need for a corresponding composite foreign key.

Decomposition of the current table is very simple.  If you can describe the structure of the present table, i.e. its name and all of its columns, with a description of what attribute a column is modelling if this is not obvious from its name, then we'll be able to be more categorical.

ken sheridan say: PS:  I've just seen your other thread, from which it appears you do have a column of date/time data type.  I don't know why you feel it necessary to extract the year value as part of the candidate key, but it makes a lot more sense simply to use the existing date/time column.  Wearing my day job hat as an environmental planning officer I'd have thought this in combination with the LocationID column is more likely to provide distinct values in any case, rather than assuming there can be no more than one sample per location per year.

scottgem say:

Without seeing the current structure of your tables we can't advise how to break them up.  You have to look at the data and judge what fields go in each table. From there it should be fairly easy to run Append queries to move the data into the correct tables.

And no it would not be a quicker fix. With a bad structure, you will constantly be trying to work around it. The better choice is to fix the problems rather than apply a bandaid approach.

rock909 say:

Error when opening file share link from email.

    file://localhost/Volumes/filepath

talking moose say:
First, the "file://" protocol doesn't translate between Mac and Windows
because they will connect to the same location using different paths.
Windows to Windows will generally work. Mac to Mac may work provided
your machines are connected to the same location using the same network
protocols.
 
Second, the "localhost" in the path name means "this computer". If
someone sends you a link with "localhost" in the path then he has
included a link that works for him but it won't work for you. The file
is not on your own computer.
 
Third, there's a difference between "file sharing" and "web sharing".
Any time you receive a link starting with "file://" then the user is
attempting to provide you a link that's using file sharing. Unless you
are connected to the same location using the same operating system (Mac
or Windows) then this will almost always fail.
 
Web sharing, on the other hand, is practically universal. Links will
generally begin with "http://". If you have anything like a Microsoft
SharePoint service then you can upload files to it and then share the
link from there. The same would apply to DropBox or any other web-based
file sharing service.
 
Hope this helps!
