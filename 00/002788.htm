This refers to the old 1904 date system that use to be used in Mac. Today although the setting there it's left unchecked. Turning on add an extra Leap year making the date 366 days off. When Mac first came out they used the 1904 date system Microsoft used 1900 date system. Since back then Mac Users and PC users were independent groups and rarely if ever shared anything with each other. Suggestion leave off unless you exclusively share only with Mac Users.

Replaceing cell ranges - Microsoft Community

howieem say:

Replaceing cell ranges

As you can see from the following code, this macro check for certain cells (or cell ranges) and applies the select case function to allow inputting time without using the colon.  The problem is that first, I have to define every cell either by cell reference, or by range.  The second is that this code only applies to the sheet where the code is inserted. 

What I am wondering, is there a way to not use line 5 which defines each cell or range, and instead, look for any cell which is formatted as a time (hh:mm AM/PM) and apply this function to that cell?  Also, can this function be applied to the workbook so that it would work on any sheet with times entered into cells?

This way, anywhere that I format a cell to the time format at defined above, the function would automatically run to allow entering time without the colon.

 

Private Sub Worksheet_Change(ByVal Target As Excel.Range)

Dim TimeStr As String

If Target.Address = "$J$2" Then Call B_selectthecase

On Error GoTo EndMacro
If Application.Intersect(Target, Range("L6:M61,E48:F61,D73:D83,D87:D105,L87:L93,M73:M83,D6:D24,D28:D43,B109:B127")) Is Nothing Then
    Exit Sub
End If
If Target.Cells.Count > 1 Then
    Exit Sub
End If
If Target.Value = "" Then
    Exit Sub
End If

Application.EnableEvents = False
With Target
If .HasFormula = False Then
    Select Case Len(.Value)
        Case 1 ' e.g., 1 = 00:01 AM
            TimeStr = "00:0" & .Value
        Case 2 ' e.g., 12 = 00:12 AM
            TimeStr = "00:" & .Value
        Case 3 ' e.g., 735 = 7:35 AM
            TimeStr = Left(.Value, 1) & ":" & _
            Right(.Value, 2)
        Case 4 ' e.g., 1234 = 12:34
            TimeStr = Left(.Value, 2) & ":" & _
            Right(.Value, 2)
        Case 5 ' e.g., 12345 = 1:23:45 NOT 12:03:45
            TimeStr = Left(.Value, 1) & ":" & _
            Mid(.Value, 2, 2) & ":" & Right(.Value, 2)
        Case 6 ' e.g., 123456 = 12:34:56
            TimeStr = Left(.Value, 2) & ":" & _
            Mid(.Value, 3, 2) & ":" & Right(.Value, 2)
        Case Else
            Err.Raise 0
    End Select
    .Value = TimeValue(TimeStr)
End If

hansv mvp say:

Remove the Worksheet_Change event procedure from the individual sheets.

Copy the code that follows below into the ThisWorkbook module.

It will handle changing multiple cells at once.

It uses a very primitive test to determine whether a cell is formatted as time: it looks for a colon ":" in the number format. This means that a custom format such as "Profit: "$#,##0.00 will be seen as a time format too, hence the code will return an incorrect result.

 

Private Sub Workbook_SheetChange(ByVal Sh As Object, ByVal Target As Range)
    Dim rngCell As Range
    Dim TimeStr As String
    On Error GoTo EndMacro
    Application.EnableEvents = False
    For Each rngCell In Target
        With rngCell
            If .Value <> "" And .HasFormula = False And InStr(rngCell.NumberFormat, ":") > 0 Then
                Select Case Len(.Value)
                    Case 1 ' e.g., 1 = 00:01 AM
                        TimeStr = "00:0" & .Value
                    Case 2 ' e.g., 12 = 00:12 AM
                        TimeStr = "00:" & .Value
                    Case 3 ' e.g., 735 = 7:35 AM
                        TimeStr = Left(.Value, 1) & ":" & _
                        Right(.Value, 2)
                    Case 4 ' e.g., 1234 = 12:34
                        TimeStr = Left(.Value, 2) & ":" & _
                        Right(.Value, 2)
                    Case 5 ' e.g., 12345 = 1:23:45 NOT 12:03:45
                        TimeStr = Left(.Value, 1) & ":" & _
                        Mid(.Value, 2, 2) & ":" & Right(.Value, 2)
                    Case 6 ' e.g., 123456 = 12:34:56
                        TimeStr = Left(.Value, 2) & ":" & _
                        Mid(.Value, 3, 2) & ":" & Right(.Value, 2)
                    Case Else
                        Err.Raise 0
                End Select
                .Value = TimeValue(TimeStr)
            End If
        End With
NextCell:
    Next rngCell
    Application.EnableEvents = True
    Exit Sub
EndMacro:
    Resume NextCell
End Sub

sven johannsen say:

Any reputable charger using 5v, over 750ma, and has a micro USB connector will charge your phone just fine. The voltage is important, and specified by it being USB and having that connector. The current (amps) spec is what the charger is capable of safely supplying. It is not what it will force into your device.

Using a charger that can produce less than what your phone wants can result in an overheating charger, and at the least slower charges. A charger with a higher capability will not hurt your phone, and may result in faster charges, though that is not guaranteed. The device being charged determines at what rate it will accept current, and will not accept more than it is designed to handle.

I used the term reputable charger for a reason. A poorly engineered charger can drop it's voltage if the current requested by the device exceeds it's specification which can damage the charger and probably not charge at all. On the other hand, a poorly engineered charger can try to raise it's voltage if it is being asked to deliver more current, and that is not good for the phone. I would avoid $1 eBay chargers, but pretty much anyone sold at a decent shop should be fine.

ken sheridan say: A Boolean (yes/no) column can only be TRUE or FALSE and cannot be NULL.  One way is to use a column of integer number data type with a ValidationRule property of:

Is Null Or (>=-1 And <=0)

A check box bound to this column in a form should have its TripleState property set to True (Yes).  This allows it to be Null.  This tallies with Codd's 3-way logic, TRUE, FALSE or MAYBE.  While Codd does not explicitly express support of Nulls, as Date has pointed out, his support for a 3-way logic does imply support of Null.  Date does not.

However, what you are proposing is not really a 3-way logic in the sense that Codd meant.  Your  third option is N/A, which is a value.  Null is not a value of course, but the absence of a value.  So the use of an integer column in which Null is allowed to represent N/A is something you will need to consider carefully.  Many people would be happy with interpreting Null in this way, but I'd be cautious about it.  Null is semantically ambiguous, so could be interpreted as ' don't know'  rather than 'not applicable'.  It's your call, but I'd be happier with text values as Scott suggests.
