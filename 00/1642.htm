    My query by Year Combobox doesn’t work? - Microsoft Community

    efandango say:

    My query by Year Combobox doesn’t work?

    I have an unbound combo box (cbo_Payment_Year) on a main form that has a Row Source Type set to Value List of: 2011,2012,2013… The combo box has a Format value of ‘yyyy’.

     

    I have an underlying field query called [PaymentFilterYear: Payment_Date] with a format of yyyy. The field has a criteria of: [Forms]![frm_Expenses_Report]![cbo_Payment_Year]

     

    When I try to use the combo box to filter the query, I get an error saying:

     “The value you entered isn’t valid for this field”

    “For example you may have entered text in a numeric field or a number that is larger than the FieldSize properties permits”

     

    I cannot figure out where I am going wrong with this, as it seems to me that both ‘items’ are of the same type.

     

    The SQL Query:

    PARAMETERS [Forms]![frm_Expenses_Report]![cbo_Payment_Year] DateTime;

    SELECT Expenses_Log.Expense_Rec__ID, Expenses_Log.Cost, Expenses_Log.Quantity, Expenses_Log.Unit, Expenses_Log.Expense_Type_ID, tbl_Expense_Type.Expense_Type, Expenses_Log.Due_Date, Expenses_Log.Payment_Date, Expenses_Log.Note, Expenses_Log.Supplier, Expenses_Log.Payment_Date AS PaymentFilterYear

    FROM Expenses_Log INNER JOIN tbl_Expense_Type ON Expenses_Log.Expense_Type_ID = tbl_Expense_Type.Expense_Type_ID

    WHERE (((Expenses_Log.Payment_Date)=[Forms]![frm_Expenses_Report]![cbo_Payment_Year]));
    ken sheridan say: You'll find an example of exactly that sort of thing, amongst other things, in MultiSelect.zip in my public databases folder at:

    https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

    This little demo file includes means of restricting a report's results on the basis of one or more fields optionally in the way you describe.  In your case, if we strip out all the
    unnecessary garbage which the Access query designer throws in, and then rearrange the WHERE clause we get:

    SELECT Expenses_Log.Expense_Rec__ID, Expenses_Log.Cost, Expenses_Log.Quantity, Expenses_Log.Unit, Expenses_Log.Expense_Type_ID, tbl_Expense_Type.Expense_Type, Expenses_Log.Due_Date, Expenses_Log.Payment_Date, Expenses_Log.Note, Expenses_Log.Supplier, Expenses_Log.Payment_Date AS PaymentFilterYear
    FROM Expenses_Log INNER JOIN tbl_Expense_Type ON Expenses_Log.Expense_Type_ID = tbl_Expense_Type.Expense_Type_ID
    WHERE (Expenses_Log.Expense_Type_ID = Forms!frm_Expenses_Report!cbo_Expense_Type
    OR Forms!frm_Expenses_Report!cbo_Expense_Type IS NULL)
    AND (YEAR(Expenses_Log.Payment_Date) = Forms!frm_Expenses_Report!cbo_Payment_Year
    OR Forms!frm_Expenses_Report!cbo_Payment_Year IS NULL);

    The parenthesizing is crucial here.  Each Boolean OR operation is parenthesized so that it evaluates independently of the Boolean AND operation.  So each will evaluate to TRUE for those rows where the value in the column is that of the parameter, if one is entered, or for every row if the parameter is NULL.  Rows will be returned where both of the OR operations evaluate to TRUE, by virtue of the AND operation.

    PS: with queries of this nature be sure to save them in SQL view.  If you switch to design view Access will move things around and at best the logic will be obscured; at worst the query may become too complex to open.

    ken sheridan say: Read my previous post again and try and understand how the logic works.  It's really quite straightforward:

    WHERE (SomeColumn = <some parameter>
    OR <some parameter> IS NULL)
    AND (SomeOtherColumn = <some other parameter>
    OR <some other parameter> IS NULL)
    AND (YetAnotherColumn = <yet another parameter>
    OR <yet another parameter> IS NULL)

    .....and so on, tacking each parenthesized OR operation together with an AND operation.

    And remember to write it and save it in SQL view.  Don't atempt to do it in design view.
