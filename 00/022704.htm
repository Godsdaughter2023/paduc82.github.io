I don't think there's ever been any type of implied harmony between syntax of forms refs between macros and that of VBA. 

 

In other words for the last 15 or more years, I don't think there's been article(s) or writings or any kind of suggestions that assume there is some type of harmony or similarity between the two systems.  In fact I'm pretty sure the way the web macros reference things is the same as to how macros always worked even many moons prior to even web based development in access. (so macros work like macros always did).

 

>There is no explanation given as to why this is, but the limitation is clearly stated.

 

Well I'm not sure of I've ever seen any limitations stated as to the difference between a local and global variable inside of access either but they are certainly different. 

 

However the case of web based development, you really can't reference other forms because the other form actually might just been clicked on and is now doing your banking or you're about to purchase something on EBay.

 

I suspect people would not take kindly to you being able to reference other browsers open on the user's desktop.

 

In fact if you DO launch another browser to the same application then you will find that this setup is identical to having a split VBA database and launching two copies of the front end..

 

In a split database we OFTEN see the question in this forum as to if the VBA variables are SHARED between each front end running - we know they are not).

 

And EVEN if you launch two copies of Access and the SAME front end two times on your computer then AGAIN the LOCAL and GLOBAL variables between the two instances are separate. However I don't see this explained the manual, but that is the way Access has always worked.

And the SAME goes for web based. If you launch two separate browsers and send them to the same Access web application then you will find that BOTH local and global vars are SEPARATE BETWEEN EACH INSTANCE of the browser. This much explains why we have a new command browseto in which in place of opening a form, we "browse to " a form.

 

At the end of the day we could speculate as to why you can't ref the forms! in that command. On the other hand a "wild guess" would be that we don't have the ability to reference other instances of the user's browser running on their desktop. I suppose it would be cool if I could do a re-query on another browser running your Facebook.

 

Hey maybe I can refresh something on your machine and get you to purchase a bunch of my things from EBay?

 

Without question we are dealing with a new architecture and different paradigm and how we think has to be changed.

 

One more concept:

 

Remember the code you're writing in a web form has to run on the user's desktop inside of a web browser. 

 

I going to say this again since so few seem to grasp this issue:

 

Remember the code you're writing in a web form has to run on the user's desktop inside of a web browser.  (it does NOT run server side).


This explains why there are no record sets in web forms. In fact last time I looked when you run a safari browser on your apple iPad, the apple iPad does NOT have DAO recordset support. In fact that apple iPad has ZERO recordset support.  And the same goes for your cool windows smartphone – that mobile browser also does not support recordsets.

So the many different kinds of browsers do not support record sets and this explains why form code is so limited.  This means to do record set "type of" processing then you have to call "server side" code from the web form. In Access Web services such code turns out to be named table macros.

 

One more tip:

 

One of the things about I liked about VBA is you always had "me" to reference the current form. In Web based you do not. As a result in ALMOST all cases I find it much preferable stuff text box values into local (or global) vars FIRST and THEN use those for filters etc.  The reason being for this tip is it just so turns out that when you stuff values from web text boxes into variables you do NOT have to include a forms ref, but JUST the name of the control.

 

This is rather nice as then you can rename the form, copy the form, or insert the form into one of the new navigation controls (which in effect means you're running the form as a sub form).  So adopting a neutral approach and avoiding forms! references in code are a really wonderful thing to do. 

 

I think we've naturally avoided hard coded forms refs in most VBA applications, but I can't say in the help or manual it stated somewhere that we should use "me" to help us write neutral form ref code – however that just something we as developers learned over time.

 

So for the most part keep in mind that these things work the way they do and 9 out of 10 times there is pretty good explanation as to why things work the way they do.

Well, using

Forms!FormName.Controls!Combo72

is not the normal way to refer to a control anyway.  The normal methnod in a macro is

[Forms]![FormName]![Combo72]

 

But when you are working with macros, as opposed to VBA, you would simply use the [Combo72] as the name because it knows what form you are referring to since it is on the event on that form.  If you had another form then you might want to use the full method, again like:

 

[Forms]![SomeFormNameHere]![ControlNameHere]

 

In VBA it would be

 

Me.Combo72

 

if on the same form and then

 

Forms!FormNameHere.ControlNameHere

 

if on a different form than the code is on.

glenn_n---01 say:

I have E75 and have recently been getting the annoying prompt to accept certificate permanently - over and over.

Today I took the time to investigate and fixed it easily. The problem was that the email provider had changed their SSL certificate a few weeks ahead of schedule. The name was the same as the previous certificate. All I had to do was delete the existing certificate and then the next time I was prompted to accept certificate, it was able to store the new cert permanently.

As explained in an earlier reply:

- Open control panel/settings/general/security/certificate management/trusted site certificates.

- You will probably see that the certificate is already stored - but it is old.

- Delete the certificate.

- Next time you connect, you will again be asked to accept permanently. Do it and all will be well!
