scottgem say:

OK, while MVFs make for a nice interface, most pro developers don't use them because they cause more problems then they are worth. MVFs model a many to many relationship, but they do so with a hidden table that makes things harder to work with.

Access is a relational database, therefore, data needs to be stored relationally to work efficiently. Your member table should only contain identification info about members. I.E. Names, address contact info.  Since members may take multiple sessions, that is why they should not be in the Members table. In my opinion you need at least three tables here, besides your members table. The first table, I'll call Courses:

tblCourses

CourseID (PK Autonumber)

CourseName

other info about a Course

Then you have the sessions table which will show when each course is given. So there will be a 1:Many relation between Courses and Sessions

tblSessions

SessionID (PK autonumber)

SessionTime

CourseID (FK)

Finally you have a Registrations table to show which members are registered for which sessions. This models the many to many relationship between members and sessions.

tblRegistrations

RegistrationID (PK autonumber)

MemberID (FK)

SessionID (FK)

Now your query joins all the tables to pull all the info you want for your report and you can then group the report by Session.

Let me first describe my last two purchased systems, both of which perform very well.

#2 - Just 2 months old now:

OS: As before - Windows 7 Ultimate 64-bit.

And the Office 2003 & 2010 versions are the same!

 

Now - if you want to spend the bucks on a Xeon CPU based system, go ahead - but remember that you're actually paying a lot extra for a CPU that is designed for use in a server, not in a workstation.  but they do process stuff really fast!  If you have to balance your budget for the CPU, pick the fastest CPU you can afford, regardless of number of cores.  If you go with an i7 you're going to get at least 8.  But I'll let you in on a 'secret' - I've got a CPU usage monitor running on both systems, and most of the time the average usage across all the cores is below 25% although you might see one or two of the 8 up at even 100% from time to time, but the others will be near zero.

 

Right now I have IE open doing this response; I have Excel 2003 running with a 1MB file, and even Excel 2010 open with a 1.5MB file, there's a background task that always keeps the CPU occupied, and I've got Outlook 2010 open with a couple of reply emails in progress.  My CPU usage is at 13% :), and I'm using right at 3GB of my 12GB of RAM.  Task manager tells me I'm running 98 processes.

 

As for the number of cores that Excel 2007 and later can use - just leave the setting on "automatic" or "all" and it'll use what's available.  There were instances with Excel 2007 where you could actually get better performance by cutting down the number of cores being used to just 1, but I believe that's been fixed in 2010.

 

With 12 to 16GB of RAM you'll probably never really use it all.  I have a rule of thumb of at least 2GB per cpu core.  And if you plan on setting up virtual machines, at least 2GB allocatable to each of them.

 

BUT there is a point that simply tossing more RAM at Excel doesn't help.  There are certain internal features of Excel that only allocate themselves a certain amount of RAM and regardless of the amount of physical RAM you have in the computer, if they hit one of these internal limits, then you're going to see one of those "not enough memory..." messages.

 

Your last question was basically asking "what's the sweet spot combination of CPU type, clock speed and installed RAM?":

CPU is almost a no-brainer again: given a particular class of CPU by a manufacturer, the higher the clock speed, the more work you'll get done in a given amount of time.

RAM - I haven't seen performance graphs for RAM installed with Windows 7 64 bit.  But there is typically a curve that's steep up to some point and then almost levels out at some point.  If you can afford the 12 or 16 GB, then go for it, but I've got a kind of gut feeling that the sweet spot is at 12 GB or perhaps less.  I don't think the extra $$ for something like 34 or 32GB installed would get you much bang for the dollar.

 

Now about that OS - definitely go with the 64-bit version!  Without the 64-bit version of Windows, you're not going to "see" more than 4GB of RAM no matter how much you actually have installed.  So that's pretty much a no-brainer.  If you buy 12 or 16 or more GB or RAM, you're going to need a 64-bit operating system to use it.  Also, if given the option, set the drives up as NTFS formatted drives, not FAT-32.  Lots of advantages to the NTFS (NT File System) format.

 

Office 32-bit vs Office 64-bit??  Well, in theory the 64-bit version should do things faster.  I have the 64-bit version on my office computer.  Quite frankly I don't really see a big performance difference.

 

Also, with the 64-bit version of Excel, there are still a lot of 3rd party "Add-Ins" that have not been set up to work in 64-bit Excel.  So if you want a smooth path to using Add-Ins in Excel, stick with the 32-bit version of Office.

 

A couple of other things to think about:  one of the slowest things happening in and around your computer (besides user keyboard/mouse input and disk I/O) deals with the display.  If you can afford a high end video card with lots of RAM on it, go for it.  It'll pay off in improved performance in getting the data from the CPU/RAM onto the display.  Gamer systems usually make great workstations: they have high end CPUs, lots of RAM and really hard working video cards.

 

Another place you can gain some performance improvement for relatively little cost is by adding an external RAM cache in the form of a thumb drive or memory card to be used as ReadyBoost memory.  I'll let you read about that here: http://technet.microsoft.com/en-us/magazine/ff356869.aspx

You can add up to 32GB of cache RAM in this way (see the article for limitations on that).  I've got both of my machines set up with small, 4GB ReadyBoost USB drives.  I'm thinking of moving up to a larger sized device on at least one of them to see if I notice any improvement, just haven't done that yet.

 

I hope I've not confused you too much.

#1 - There's absolutely no question about whether or not <any> Outlook version supports a PST file - all do including '2019 which leads into #2
