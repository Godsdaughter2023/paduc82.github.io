Can IF Statements use Contains as a function and If so how can I do - Microsoft Community

scottstevenson1 say:

Can IF Statements use Contains as a function and If so how can I do it?

I have the need to create a Merge field that uses an IF statement with a Contains function that evaluates a string of values to find a specific value

e.g.

The Merge Field pulls in a string of 6 values (this could be more or less depending on the related data)

I need to know if the string contains a specific word
If so show "A" Else "B"

The value searching for could be anywhere with the string and therefore cannot use the * to denote start or end of merge field.

Is there a way of doing this within a Word Document?

Thanks

a. user say:

There is no particularly good solution in Word's "field language".

As far as I know, there are basically five ways, none of them particularly simple and they all come with various restrictions and/or "gotchas".

In summary:

 a. you use a large number of IF statements to try to match each location in the Merge field.  

 b. you do the "contains" in the data source.

 c. you use a DATABASE field (this approach requires that you use an external file)

 d. you use an INCLUDETEXT field (this approach also requires that you use an external file).

 e. you use VBA to do your merge, with or without the MailMerge Events.

Approach (a) is the only one that does not rely on either VBA or "something outside Word". Suppose your merge field is called "mytext" and you are searching for the word "difficult". Then you could use

{ SET match 0

}{ IF "{ MERGEFIELD mytext }" = "difficult*" "{ SET match 1 }"

}{ IF "{ MERGEFIELD mytext }" = "?difficult*" "{ SET match 1 }" 

}{ IF "{ MERGEFIELD mytext }" = "??difficult*" "{ SET match 1 }" 

}

If you do not want case-sensitive matching yu can use

{ IF "{ MERGEFIELD mytext \*Upper }" = "DIFFICULT*" "{ SET match 1 }" } and so on.

and so on until the number of ?s + the length of the word ("difficult") reaches the maximum length of your input string, or the maximum Word will recognise (probably about 128).

Then you should  be able to use { IF match = 1 "contains difficult" "does not contain difficult" }

That is probably the simplest approach, but you may find that with all the necessary field codes, execution is slow.

With approach (b), there are different possibilities depending on your data source and your level of access to it.

 1. Your data source lets you create a view, query or procedure that does the "contains" and returns a flag for each word you are trying to match. That would typically require that the words you are looking for are hard-coded in the view or query. Typical source type - SQL Server, Access Jet/ACE. In the case of Excel, you might be able to create additional columns in the source sheet that do the necessary contains for each word.

 2. Your data source lets you use a reasonably powerful dialect of SQL (e.g. SQL Server, Jet/ACE) and you can construct a suitable SQL statement to retrieve the data and create new columns for each "contains" you need. Then you use Word VBA and OpenDataSource to issue the query. A significant constraint is that the query length is restricted either to 255 or 511 characters (depends on the situation AFAIK). It may sound like a lot but those characters get used up very quickly. Another constraint is that you may not be able to control certain aspects of the match (e.g. it may be difficult to ensure a case-sensitive match).

With approach (c), you use a Word DATABASE field to do the comparison via a piece of Jet (Access) SQL. The broad outline is that you need a nested field like this:

{ IF { DATABASE \d "databasefilepathname" \s "SELECT iif('{ MERGEFIELD mytext }' Like 'difficult',1,0)" } = 1 "contains difficult" "does not contain difficult" }

There are several constraints with this approach including:
 - you have to have an external file (the "databasefilepathname")
 - you must know in advance that the texts you want to match must not contain a mixture of single- and double-quotation characters
 - there is a limit of around 255 characters on the total length of the SQL statement

 - It's for Windows desktop Word only (Mac Word does not currently allow data sources with a sufficiently rich dialect of SQL) 

- and so on.

I have posted a set of example files that use the DATABASE field approach at https://goo.gl/gJAd9y - download the .zip and follow the instructions in the readme.txt file.

With approach (d), you use a special INCLUDETEXT field and a small external file. The broad outline is that it is rather similar to approach (c). I have posted a set of example files that use this approach at https://goo.gl/UU44Za - download the .zip and follow the instructions in the readme.txt file.

For most approaches, there are also "edge" cases that you may or may not need to solve - e.g. what if you need to match something that contains a wildcard character such as "*" or "?".

Since approach (e) relies on writing VBA or other code, IMO it is really another question.

    Would anyone be able to walk me through how to do this? 

    ken sheridan say: I'm not familiar with the template, so can only give you a generic answer.

    Let's say you have an unbound dialogue form with unbound text box control's txtStartDate, txtEndDate, txtTaskTime and an unbound combo box cboTask whose BoundColumn is a numeric TaskID column.

    It's unlikely that  you would want to insert a task for every day over a period of months say.  A more likely scenario is that you would want to insert rows for each working day in the period.  So, rather than iterating through a loop as I suggested in the earlier thread, the first thing to do would be to create an auxiliary calendar table of all working days over the next ten years or so.  This table can periodically be updated by deleting rows as and when the dates of public and concessionary holidays become available.  You'll find a means of creating such a table in Calendar.zip in my public databases folder at:

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

    If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

    Armed with the WorkdaysCalendar table, if we assume rows are to be inserted into a table TaskScedule with columns TaskID and TaskDateTime then you can write code in the Click event procedure of a button in the dialogue form:

        Dim strSQL As String

        strSQL = "INSERT INTO TaskShedule(TaskID, TaskDateTime) " & _
            "SELECT " & Me.cboTask & ", #" & _
            Format(calDate + CDate(Me.txtTaskTime),"yyyy-mm-dd hh:nn:ss") & "# " & _
            "FROM WorkdaysCalendar " & _
            "WHERE calDate BETWEEN #" & Format(Me.txtStartDate,"yyyy-mm-dd") & _
           "# AND #" & Format(Me.txtEndDate,"yyyy-mm-dd") & "#"

        CurrentDb.Execute strSQL, dbFailOnError

    The above obviously won't be a precise fit for whatever table you need to insert rows into in the template, but it should give you enough idea of the methodology for doing so.
