ken sheridan say:

The value shown in a combo box is that of the first non-zero width column.  To show the last name column it would need to be the first column.  However, it is mandatory for the bound column of a combo box to be guaranteed to contain a set of distinct values.  As personal names can legitimately be duplicated this cannot be guaranteed.

Names should never be used as keys.  A person should always be identified by a distinct numeric value, which is usually an autonumber as the primary key of a Personnel table or similar.  Consequently, to satisfy your requirements the properties of the combo box would be something like this:

RowSource:            SELECT PersonID, LastName, FirstName, Community
                               FROM Personnel INNER JOIN Communities
                               ON Communities.CommunityID = Personnel.CommunityID
                               ORDER BY LastName,FirstName;
BoundColumn:      1
ColumnCount:       4
ColumnWiths:       0";1";1";1"

ListWidth:             3"

scottgem say:
Hi mom, I'm an independent advisor and will try to help you.

I'm afraid you have structural issues with the design of your database that should be corrected before you go further. I'm guessing here that the TeamMaster table Is a listing of people who are in charge of a team. You should also have a table of people, and the combobox (which should not be in the table, but should be on your form) should lookup against that people table. You should not be storing the Last name, since that is redundant data, but rather just store the primary key value of the person record.

So your Rowsource should look like this:
SELECT PersonID, Community, Lastname, Firstname
FROM table
ORDER BY Lastname, Firstname

Actually I would use:
SELECT PersonID, Community, Lastname & ", " & Firstname As Person

hansv mvp say:

You could run the following macro:

Sub MergeRows()
    Dim r As Long
    Dim m As Long
    Dim c As Long
    Application.ScreenUpdating = False
    m = Cells(1, 1).End(xlDown).Row
    For r = m - 1 To 2 Step -1
        If Cells(r + 1, 1).Value = Cells(r, 1).Value Then
            For c = 2 To 3
                If Cells(r + 1, c).Value <> "" Then
                    Cells(r, c).Value = Cells(r + 1, c).Value
                End If
            Next c
            Cells(r + 1, 1).EntireRow.Delete
        End If
    Next r
    Application.ScreenUpdating = True
End Sub

bryan_656 say:

For me this problem started when I converted my GMAIL account from POP to IMAP.  I followed these steps:

- In Outlook I suspended my GMAIL POP account from send/receive actions (File,Options,Advanced,Send/Receive )

- copied and renamed 'All Accounts' as 'IMAP accounts', removed all send/receive options in 'All Accounts'.  This stopped send/receive actions.

- In GMAIL settings I changed to IMAP

- In Outlook I set up a new account and pointed it to my GMAIL address with automatic configure and added it to the 'IMAP accounts' group (along with an existing IMAP account from another provider).  This had Send/Receive options enabled.

All worked perfectly with Outlook getting all new messages as expected, with all old messages stored in the POP account in Outlook.

I then began the process of creating new folders for the IMAP account that I had previously in the POP account - with the same folder name in both accounts.  I also set up all auto-archive settings for each folder, and progressively created 'Rules' to move messages from the inbox to each folder..

As messages reached the age that required archiving (probably in both POP and IMAP accounts), the problem reported here started to occur.

I have avoided the problem by renaming all the folders I created in the IMAP account so that they were different names to those used in the POP account.  Since then the problem has been avoided.

So I suggest that to avoid this problem ensure that all folders in all accounts have a unique name.
