anil kumar b say:

Hi,

 

-What is the issue are you facing with Internet Explorer 10?

 

The error 0x80042308 is related to Volume Shadow Copy.  When you backup files on a Windows 7 computer you may receive error 0x81000019, this error can happen because of having too many Windows language packs installed on your computer.

 

The cause is that the Backup process involves the system writer that reports files in the important applications installed on the machine. In the case where a large number of language packs is installed, the metadata file reported by this writer becomes huge in size and does not fit the 100 MB volume. This causes backup to fail since there is not enough space for the shadow copy creation. This is not something that you can check on to validate since this metadata file is created during backup and deleted at the end.

 

Method 1:

To resolve the issue, uninstall all of the unused language packs that are currently installed on your computer. You can do this by

a) Clicking Start, and then typing Lpksetup.exe in the Search box

b) And then clicking Lpksetup.exe in the list of programs.

c) Once the Language Pack installer starts, click Uninstall display languages.

d) Uninstall the languages that you do not use, and then restart your computer and restart the backup.

 

Method 2:

Check whether Volume shadow copy is running:

a) Click Start.

b) In the search box type Servcies.msc

c) Right click Volume shadow copy and Go to Properties.

d) Select Automatic (Delayed) from the Startup type drop down menu.

 

Method 3:

I would suggest you to run SFC scan.
How to use the System File Checker tool to troubleshoot missing or corrupted system files on Windows Vista or on Windows 7
http://support.microsoft.com/kb/929833

 

Let us know the status of the issue. If you need further assistance, please post back. We will be glad to help you.

ken sheridan say: The rules as regards delimiters for literal values of the different data types are as follows:

Text:  The values must be enclosed in single or double quotes characters 'like this' or "like this".  

Number or Currency:  No delimiters are required.  With currency values the currency symbol should be omitted.  The Currency data type is really a number data type whose value is stored to a precision of four decimal places to avoid rounding errors.

Boolean (Yes/No):   No delimiters are required.  The values are usually entered as the Boolean constants TRUE or FALSE.  Under the skin Access actually stores these values as -1 or 0.

DateTime:  The delimiter is the # sign, and the value must either be in US short date format, mm/dd/yyyy, e.g. #25/03/2018# or an otherwise internationally unambiguous format such as the ISO standard  of YYYY-MM-DD, e.g. #2018-03-25#.  Under the skin date/time values are stored as a 64 bit floating point number, with zero representing 30 December 1899 00:00:00.

ken sheridan say: As written, that query assumes that the Cost column (field) is a currency data type, and the other columns are all of text data type.  However, one possibility is that, when building the table you used the 'lookup field' wizard when assigning the data type of one or more of the columns.  If so the actual values in the columns won't be the text values which you see, but will be numeric foreign key values referencing the numeric (usually an autonumber) primary key of another table.

This sort of confusion is one reason why most experienced Access developers recommend that the 'lookup field' wizard be avoided like the plague.  For reasons why see http://theaccessweb.com/lookupfields.htm  If this is the case what you need to do is join the table to the referenced  table(s) in the query.  Let's take a simple example in which the wizard was used when creating the Location column, and assume we want to return all rows from the table where the location is Malaga.  The referenced  table might be called Locations or something similar, so the query would be like this:

SELECT Table1.*
FROM Table1 INNER JOIN Locations
ON Table1.Location = Locations.LocationID
WHERE Location.Location = "Malaga";

In this Location is the foreign key column in Table1, LocationID is the primary key of Locations and Location is a text column in Locations in which the actual location names are stored.  You can of course design this query easily in query design view, rather than writing the SQL.

andreas killer say:
It's tough to get only that information, but a simple and clean way (and all programmers around the world do that) is to install an error handler.

So if anything went wrong (invalid filename, access denied, disk full, disk write error, whatever), the code jumps to the error handler label and you can show a message, try again, whatever...

Below is a sample code how to use an error handler.
Don't run it! Debug it step by step and see what happens.

Andreas. 

Option Explicit
Sub Example_ErrorHandler()
  On Error GoTo ErrorHandler
  Err.Raise 1, "Example_ErrorHandler", "Error description"
  
  Example_SubErrorHandler
  Application.EnableCancelKey = xlErrorHandler
  Do
    DoEvents
  Loop Until False
  Exit Sub
ErrorHandler:
  If Err.Source = "" Then Err.Source = Application.Name
  Debug.Print "Source     : " & Err.Source
  Debug.Print "Error      : " & Err.Number
  Debug.Print "Description: " & Err.Description
  If MsgBox("Error " & Err.Number & ": " & vbNewLine & vbNewLine & _
      Err.Description & vbNewLine & vbNewLine & _
      "Enter debug mode?", vbOKCancel + vbDefaultButton2, Err.Source) = vbOK Then
    Stop 'Press F8 twice
    Resume
  End If
End Sub
Sub Example_SubErrorHandler()
  'Install a 2nd error handler
  On Error GoTo ErrorHandler
  'Something goes wrong:
  Debug.Print 1 / 0
ExitPoint:
  'Usual finish
  Exit Sub
ErrorHandler:
  Select Case Err.Number
    Case 123
      'We handle this error here
      Resume ExitPoint
    Case Else
      'Call the upper error handler with this error
      RaiseError ErrSource:="Example_SubErrorHandler"
  End Select
  Stop
End Sub
Sub RaiseError(Optional ByVal ErrNumber, Optional ByVal ErrSource, Optional ByVal ErrDescription, Optional ByVal ErrHelpFile, Optional ByVal ErrHelpContext)
  'Save the current error
  With Err
    If IsMissing(ErrNumber) Then ErrNumber = .Number
    If IsMissing(ErrSource) Then ErrSource = .Source
    If IsMissing(ErrDescription) Then ErrDescription = .Description
    If IsMissing(ErrHelpFile) Then ErrHelpFile = .HelpFile
    If IsMissing(ErrHelpContext) Then ErrHelpContext = .HelpContext
  End With
  'One level up (this clears the error object!)
  On Error GoTo 0
  'Raise it again
  Err.Raise ErrNumber, ErrSource, ErrDescription, ErrHelpFile, ErrHelpContext
End Sub
