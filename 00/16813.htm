When you configure accounts in Outlook 2010, if you allow Outlook to autoconfigure them, it will add a new PST for each account.  You should manually configure them if you want to control that.  If you manually define the account, you have the choice of having Outlook create a new PST for the account or use an existing one.

brendanball- [o365] say:

You need to post some more information about the on-premises Exchange server configuration. Single Exchange 2013 server with both roles? Full namespace design? Hopefully these thoughts can get you pointed in the right direction.

1. You cannot have any 3rd party between Office 365 and your on-premises Exchange server. It sounds like you've already fixed this.

2. The FQDN that Office 365 uses for your on-premises server must resolve to an external IP that will allow port 25 inbound. This should be NAT'd to your Exchange server (I'm assuming here that you have a single Exchange 2013 server running CAS/Mailbox roles).

3. Say your domain is 'contoso.com' and you want to keep your MX record pointed on-premises. Under accepted domains in your on-premises Exchange server you can set contoso.com to authoritative. In the Office 365 tenant you can would set 'contoso.com' to internal relay. This way, if a message is sent from a cloud mailbox to an address that Office 365 doesn't know about (say a distribution group or mail enabled public folder) it will be routed through the connector that goes on-premises. NOTE - if you do it this way, then you should not create any new groups/users in Office 365 as the on-premises server won't know about them and will drop the message.

Personally, I prefer to point the MX to Office 365. Configure the domains in Office 365 as internal relay, then configure the domains on-premises as authoritative.
