It is my belief that your Windows 7 license would be "invalid", at least at always has been in the past.

What is offered here for free is an upgrade.  Notice not a standalone copy of Windows.

The basic difference between an upgrade key and a "full" key is the fact that the upgrade replaces the existing license.

I have upgraded many times.  Windows XP to Windows 7 to Windows 8 to Windows 8.1.  I got a reduced price because I was upgrading not buying the full version, so I still only have one license not one for every version of Windows I have upgraded from.  I'm not allowed to run copies of Windows XP, 7, 8, 8.1 from the same license all at the same time on different machines, I did not pay for that.  On the other hand at any time I could have decided to downgrade to the older version if that is all I was going to run.  So "invalid" isn't accurate it is more accurate to say that you have a license to run Windows on one machine, but nothing has authorized you to run the same license on multiple machines.  Now please note that I talking about the single machine license.  Microsoft has had a variety of licenses and some are for multiple copies/machines with those the terms would be different.

Also Microsoft activation software does look for this.  If the software finds two different devices using the same license it will "un-activate" at least one of the devices or prevent the second one from activating in the first place.

32 bit shell32.dll attempts to load 64 bit devmgr.dll

Win XP 64-bit

The issue is that c:\windows\syswow64\SHELL32.DLL is attempting to load c:\windows\sysnative\DEVMGR.DLL.

I got this information from dependency walker.

I installed a fresh copy of XP 64-bit in a VM to compare the two side by side.

On the new VM copy of XP64, loading c:\windows\syswow64\SHELL32.DLL in dependency walker shows DEVMGR.DLL with no path, and listed as a missing delay-load dependency module. All programs of interest on this XP64 box work correctly.

The version of SHELL32.DLL on the fresh install, where everything works OK, is 6.0.3790.1830. This is the one that correctly finds no DEVMGR.DLL in SysWOW64.

The version of SHELL32.DLL on the box I depend on, where everything fails, is 6.0.3790.5018. This is the one that incorrectly loads c:\windows\sysnative\DEVMGR.DLL.

Does anyone know why version 6.0.3790.5018 of the c:\windows\syswow64\SHELL32.DLL is attempting to load c:\windows\sysnative\DEVMGR.DLL while version 6.0.3790.1830 of the c:\windows\syswow64\SHELL32.DLL correctly finds no DEVMGR.DLL in the SysWOW64 folder?

All referenced dependencies from SHELL32.DLL on the bad box are to SysWOW64 except for the one reference to DEVMGR.DLL.

Could there be something in the registry on the old box that is not on the fresh box that would cause sysnative to be invoked?

I searched every reference to DEVMGR.DLL in both registries and both are identical. All references are to c:\windows\system32\DEVMGR.DLL

Thank you in advance for any insight you may provide.

eewiz say:

ANSWER

For those that may not be aware.

On the XP x64 platform, a 32-bit application cannot access the system32 folder with a Wow64.dll version prior to 5.2.3790.4156.

At version 5.2.3790.4156 of Wow64.dll and beyond XP 64-bit can access the system32 folder from a 32-bit application.

See https://support.microsoft.com/en-us/kb/942589 for more information on Wow64.dll.

 

I have forever had "C:\Windows\Sysnative" in my path statement. I do not know how or when it appeared there.

But, after an unknown update installed version 5.2.3790.4156 of Wow64.dll, all kinds of things broke.

After installation of version 5.2.3790.4156 of Wow64.dll, any 32-bit program searching for a not found 32-bit .dll file in the SysWOW64 folder, would now be able to search Sysnative which is actually the C:\Windows\System32 folder where all of the 64-bit .dll files are stored and attempt to load the 64-bit version of the searched for .dll into the 32-bit process.

 

Of course trying to load a 64-bit .dll into a 32-bit process crashes the 32-bit program.

After removing "C:\Windows\Sysnative" from my path variable, many broken things started working again.

Sysnative is an internal feature of the "Windows 32-bit on Windows 64-bit (WoW)" functionality provided on 64-bit windows and "C:\Windows\Sysnative" should never be added to the path variable.

 

Now, why the 32-bit Shell32.dll has a dependency on the Devmgr.dll file is beyond me.

On 64-bit windows, the only Devmgr.dll is 64-bit. This is the .dll that makes Device Manager work and there is only a 64-bit version of it residing in the System32 folder. There is no 32-bit version of Devmgr.dll residing in the SysWOW64 folder.

Although Dependency Walker does clearly show that the 32-bit Shell32.dll has a dependency for the 64-bit Devmgr.dll. Admittedly, windows handles this impossible dependency with grace, until one puts "C:\Windows\Sysnative" into the path variable.

 

All for now.

cybertek say:

Hello gafa46

The System Volume Information folder is where System Restore keeps the Restore Points.

Opening it will not help you as all you will see is a list of Restore Points which you cannot open.

Also it is not possible to delete one Restore Point.

The only way to get rid of an infected Restore Point is to turn off System Restore. This will delete all of your System Restore Points.  Then you can turn System Restore back on and a new Restore Point will be created. So bear in mind it will not then be possible to do a System Restore to a time earlier than when you turned System Restore back on again.

The System Volume Information folder is hidden

To see it

In Windows Explorer click [Tools] [Folder Options]
Click the [View] tab, click [Show Hidden Files and Folders]
Clear [Hide protected operating system files (Recommended)] check box.
Click [Yes] on the change confirmation box and click [OK] to exit.
Open the System Volume Information folder in the root folder. (C:\System Volume Information)

You will see all the Restore Points.

There is a System Volume Information folder on every partition on your computer
