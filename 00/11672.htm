van dinh say:

Your Table is incorrectly structured...  One thing you need to watch out for is repeating groups of Fields and your Table clearly has 10 groups, each consisting of 1 Field. 

 

Repeating groups of Fields violates the First Normal Form and your database should satisfy to at least the Third Normal Form which include the First and Second Normal Form.

 

You should check up on Database Normalization and then re-structure your Table(s) so that your database satisfies the Third Normal Form, at least.

 

 

ken sheridan say: What you are doing here is what is known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.

In this case the data is the downtime number for each shift.  To correct the design you need to decompose the table into two related tables:

Shifts
....ShiftDate (PK)

Downtimes
....ShiftDate (FK)
....DowntimeNumber
....DowntimeHours

The tables should be related on ShiftDate and referential integrity enforced.  The primary key of Downtimes is a composite one made up of the ShiftDate and DowntimeNumber columns. So, with your sample data the rows in DownTimes would be:

2012-09-06    1     3
2012-09-06    2    1

I've shown the dates in the internationally unambiguous  ISO standard format of YYYY-MM-DD, but you can format them however you wish of course.  If you wish the DownTimeHours column could be of date/time data type rather than a number, assuming no individual downtime is 24 hours or more, so that the downtimes could be entered as hours:minutes.  If you do this you'll need to use a function to convert a sum of the values over a set of rows into hours:minutes, however, as if the sum is 24 hours or more the value returned will otherwise be a date around the start of the 20th century, arising from the way in which Access implements date/time values.  I can let you have the code for the function if you want to do this.

With this normalized model you can enter as few or as many downtimes per shift as necessary.  Aggregating the values is very simple, e.g. to get the total downtime per shift you'd group a query by ShiftDate and sum the DowntimeHours:

SELECT ShiftDate, SUM(DowntimeHours) AS Total DownTime
FROM Downtimes
GROUP BY ShiftDate;

For data entry you'd normally use a form based on the a query on the Shifts table ordered by ShiftDate, in single form view, and within this you'd place a subform, in continuous forms view, based on a query which orders the downtimes by ShiftDate and DowntimeNumber:

SELECT *
FROM Downtimes
ORDER BY ShiftDate, DowntimeNumber;

the parent form and subform would be linked on ShiftDate.

In a report you could use a query which joins the tables in an outer join, so all shifts are reported, even if there are no downtimes for a shift:

SELECT Shifts.ShiftDate,
Downtimes.DowntimeNumber, Downtimes.DownTimeHours
FROM Shifts LEFT JOIN DownTimes
ON Shifts.ShiftDate = Downtimes.ShiftDate;

and group the report by ShiftDate, with the DowntimeNumber and DownTimeHours controls in the detail section.  You can of course include parameters in the query to restrict it to a date range entered by the user at runtime.  Alternatively, to give a more economical layout, you could base a report on Shifts and within it include a multicolumn subreport with across-then-down column flow to show the downtimes horizontally across the page in columns.  You'll find an example of this sort of layout as ClubMembership.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

This demo file also includes a single report solution which does the same thing by manipulating the layout in code in the report's module at runtime, but don't bother with that; a subreport solution is far simpler.

The AfterUpdate event procedure of the plant combo box has two lines of code:

Me.cboDownTimeReason = Null
Me.cboDownTimeReason.Requery

Where cboDownTimeReason is the name of the second combo box.  Its best to give controls a name which differs from the underlying column to which they are bound.  Do this by changing the Name property in the control's properties sheet, but always do this before writing code in any event procedures for the control.  If you do so afterwards the link between the code and the control will be broken, and you'll have to recreate the event procedure.

A subform does have a Current event procedure.  You are confusing the subform control, which is the control in the parent form's controls collection which houses the subform, with the underlying source form object.  It's the latter whose Current event procedure has the one line of code:

Me.cboDownTimeReason.Requery

If you are unfamiliar with entering code in event procedures, this is how it's done in form design view:

1.  Select the form or control as appropriate and open its properties sheet if it's not already open.

2.  Select the relevant event property and select the 'build' button (the one on the right with 3 dots).

3.  Select Code Builder in the dialogue and click OK.  This step won't be necessary if you've set up Access to use event procedures by default.

4.  The VBA editor window will open at the event procedure with the first and last lines already in place.  Enter or paste in the code as new lines between these.
