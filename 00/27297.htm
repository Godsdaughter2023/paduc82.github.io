How do I calculate an invoice date each month based on either a table or formula in MS Access

Hi

I am trying to create an "Invoice Date" that is around about the 24th of every month. This could change on a monthly basis (maybe the 25th depending on bank holidays etc)

Would it be better to strictly say "Invoice date is the 24th of the month" and try to do a formula based on that?

Does anyone have any ideas? I'm pretty awful with sql but am OK with creating tables and formulas.

Christine

ken sheridan say: I'd suggest you create an auxiliary 'calendar' table.  You'll find means of doing this as Calendar.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

 1.  Use the Monthly Calendar option to create a table named InvoiceCalendar say, with dates of 24th of each month from the earliest date you'll need until some date far in the future.  Copy and paste this into your own database and in your database do the following:

2.  Create a PubHols table as in my demo, and fill this with public holiday dates as far ahead as these are known.

3.  Now you need to use a little simple SQL, but it's not very scary.  Execute the following 'update' query repeatedly until in returns no more rows, a maximum of twice will normally be enough, but executing it beyond that won't do any harm even if there are no rows to update:

UPDATE InvoiceCalendar INNER JOIN PubHols
ON InvoiceCalendar.calDate = PubHols.holDate
SET calDate = caldate+1;

4.  Execute the following query once:

UPDATE InvoiceCalendar
SET calDate = caldate +
SWITCH(WEEKDAY(calDate,7)=1,2,WEEKDAY(calDate,7)=2,1,TRUE,0);

This will move any weekend dates to the following Monday.  This could mean the date is moved to a public holiday of course, so repeat step 3 above again to move any such dates to a working day.  You'll also need to repeat step 3 whenever you add dates to the PubHols table as and when they become known, so be sure to keep the query handy.

You should now have a calendar table of all invoice dates on 24th of each month or the next working day thereafter.  How you use it depends on how and where you'll be returning an invoice date.  If you simply wanted the next invoice date on or after the current date, say, a simple expression calling the DMin function will do it:

DMin("calDate", "InvoiceCalendar", "calDate >= #" & Format(VBA.Date,"yyyy-mm-dd") & "#")

Instead of the VBA Date function you could of course use a date field in a table, to return the next invoice date on or after that date in a comouted column in a query , or a computed control in a form or report.

albert d. kallal say:

I going to give you the answer first, but then give some additional tips here.

First, in your main form where the save button is, you can grab the row value from your sub form with the following code:


[sub form control name]![Text box on sub form]

So in code that means you can go:

SetProperty (TextBoxOnMainForm, Value, =[AllEmployeesDS]![Name])

 

So, the above is all you need.  

 

Now that you have the above one line of code required, I should point out a few things:

 

First up, in a roundabout way, it's probably a really terrific thing that a combo box is limited to 500 entries.  Not only does such a design prevent a huge waste of processing and wasted bandwidth as all kinds of records are pulled from the database into the web form, after all that slow loading up of the form you are THEN going to select ONE employee!

 

I kind of much like to use the analogy of an instant teller machine.  The instant teller machine does not download every single account on the planet into the machine, and THEN ask you what account number to work on!

 

If you think about this, even if 10 people were to launch your application, each user would've downloaded 500 employees into a combo box. That means we now download a total combined usage of 5000 records, but you've done ABSOLUTELY NOTHING of any real USEFUL WORK at this point in time!

 

And ignoring the waste and abuse of Computer Resources, forcing a user to scroll through potentially 500 choices does much suggest you have a desire to inflict much cruelty and pain and suffering to your users as they have to trudge thought 100's of choices to select one employee!

 

And I dare say that adoption of a sub form likely is not much better unless you are not loading records into the sub form before they search? (Let's hope not).

 

Worse, the sub form likely will be even clumsier of a user interface.

 

Last but not least, access is what we call a relational database system.  I don't know if you're new to computers, or of ever used computers or access much before, but to display information from related tables, we generally DO NOT NEED to copy information.

 

In place of copying things like the user name, or phone number, we use the built in relational database ability to display the information for you!.

 

How this works and is done is the same if you're building web based applications, or desktop based.  In a nutshell it means you should not be copying the EmployeeName, but saving the employee primary key "ID" that represents the one employee.

 

The reason for the above means then if you are all of a sudden asked to display the extension number, or even the employee phone number in that form for quick reference, you now do not have to write ANY code!

 

So, if you do this right and use what is called relational database design, then once you've saved the employee ID into that form, then display of the phone number, their name, e-mail address etc. can be done without you having to write any additional code and you NEVER copy the data. That means if you down the road change the employee name (or phone number etc.), then again with ZERO code the correct changed employee name and phone will appear in all your event forms.

 

And regardless of you ignoring and not using the relational abilities of the database system, I also suggest you should probably pop up a form for searching and have that form return the value to the given form you are working on.

 

And before you consider the above idea of a popup form, or continuing to use your sub form idea, why not consider reversing the steps in the application?

 

In other words if I'm going to do a customer invoice, I can fire up a screen to start building an invoice, and then have to select and search for a particular customer.  However it tends to be a lot better of the user interface to REVERSE the setup.

 

What do I mean by reverse the above setup?  What I'm suggesting to do is you build a form that allows you to search and find a particular customer first, and then add the invoice.  The reason for this is many, but at the end of the day you're going to have to spend Time Energy & Software development dollars building a system in which you're going to have to be able to search and find and edit and work on that customer with great ease. 

 

And once you found the customer, and that point it's a simple matter to add a button that you click on to add an invoice, and the issue of determining which customer you're working with for that invoice is now eliminated.

 

Furthermore it means the same form that you build for great searching and finding of the customer, also now allows you to search for a customer, select the customer. And while you are looking at the customer it's going to be rather easy to have some type of sub form listing that can display all of the active invoices attached to this one given customer.  That means before you even add a new invoice, you see an existing invoice that you didn't realize had already been partially completed.  If you jump a right to the invoice adding screen, then you don't see other information about the particular customer and often it's too late. This tends to create lots of duplicates in the database and does not encourage users to check if something been enterted (they are lazy and just re-enter the invoice or event in your case again).

 

 And such information might even be about overdue accounts or who knows what.

 

And the above scenario also implies to employees also. Maybe that employee moved across the country, and you don't know that? 

 

So consider adopting a design in which you slice and dice and search for and select the employee FIRST before you actually move onto the next step or screen in the application in which you set up an event.

 

So while the one line of code above I've posted should solve your problem, I think a better approach would be to reverse the series of steps in your application. And it may well be more approipate to do things as you have and as such I would suggest with your existing design have some type of little magnifying or button you click on that pops up a search form that allows you select and choose an employee ID. When you close this form it sends back to the calling form the primary Key ID of the employee (so, try to avoid the copy back of the EmployeeName).

 

And ironically, such a popup form set is likely less code in which you are tempting to do now and as noted is bandwidth friendly as again were not loading up sub forms or forms with lots of records.

How To Copy the ROUND Function - Microsoft Community

dezdavis say:

How To Copy the ROUND Function

I want the formula I have in the cell to stay the same, but I want to add the ROUND function to it.  Is there a way to avoid typing the ROUND function into each cell?  Can I just copy it somehow without altering the formula I have in the cell?
bernie deitrick, excel mvp 2000-2010 say:

Select the cells and run this - change the 0 to the rounding level you want.

Bernie

Sub AddRounding()
Dim myCell As Range
Dim strForm As String

Application.EnableEvents = False
For Each myCell In Selection
   strForm = Right(myCell.Formula, Len(myCell.Formula) - 1)
   myCell.Formula = "=Round(" & strForm & ",0)"
Next myCell
Application.EnableEvents = True

End Sub

You could also select all the cells, and then apply the rounding to cells with formulas:

Sub AddRounding2)
Dim myCell As Range
Dim strForm As String

Application.EnableEvents = False
For Each myCell In Selection.SpecialCells(xlCellTypeFormulas)
   strForm = Right(myCell.Formula, Len(myCell.Formula) - 1)
   myCell.Formula = "=Round(" & strForm & ",0)"
Next myCell
Application.EnableEvents = True

End Sub
