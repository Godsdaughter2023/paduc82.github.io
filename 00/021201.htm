tom van stiphout say:

This can get complicated quickly. This is why shopping cart database design typically has at least 50 tables. You can simplify by taking advantage of your business rules.

For example if in YOUR business price is ONLY determined by Product and Size, that's manageable. If it's also determined by Colour, and or any other Product Attribute, it gets more complicated.

If the former, you might have a size table with surcharges: T-shirt $20, XL surcharge $1.50.

tom van stiphout say:
A few other suggestions for your db design:
1. Don't use spaces or other funny characters in object names. They may cause notational difficulties down the road.
2. OrderedItems currently has no link to Products, but I would guess you sell products without a color or with unspecified colors. Same for sizes. I think ProductID should be a FK in OrderedItems.
3. Typo: should be "Dispatched".
4. Customers.ChildsClass does not belong, for same reason you don't put Age in such table: it gets stale a year later.
5. You need more Required fields, and fewer Default Values (especially 0).

ken sheridan say: Prices are an attribute of the relationship type between products and sizes, so would be modelled by a table like this:

ProductSizes
....ProductID  (FK)
....SizeID  (FK)
....UnitPrice.

The primary key of the table is a composite of the two foreign keys. Remember that unit prices are not time independently determined by product/size; they will change over time.  Consequently its important that an OrderDetails and/or InvoiceDetails table also includes a UnitPrice column.  When an order or invoice is raised, the value of the UnitPrice column for the product/size is looked up in ProductSizes and assigned to the UnitPrice column in OrderDetails or InvoiceDetails.

As regards the relationship types in your database you will need to model constraints such as which colours a product is available in, and which sizes a product is available in, as tables, and then which products of those colours and sizes in combination are available, as a further table modelling the relationship type between the first two relationship types.  You'll find an example of how relationship types are built up across a database in this way in Relationships.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file uses a simple medical prescriptions database to illustrate how the relationship types are built up, and how they are then reflected in the user interface.

ken sheridan say: When entering rows into OrderDetails via a subform in an orders form you would bind a combo box to the ColourSizesID column and set it up so that its RowSource is a query which returns the ColourSizeID column from ColourSize in its first column, and a concatenated string of the Item Name, Colour and Size columns from Products Colours and Sizes respectively in its second column.  So the query would join 5 tables.  The query would be ordered by Product, Colour, Size for instance, though the choice of order is yours to decide.  By setting the combo box's ColumnWidths property to 0, and the ColumnCount property to 2, the first column would be hidden, so you'd select the product by its name, colour and size.  The numeric values are never exposed to the user.

The following is an example of a query which concatenates four values into a single column, using the tables in my Relationships demo.  This returns the first four columns as the composite key rather than a single first column, because my demo uses composite 'natural' keys rather than 'surrogate' numeric keys.

SELECT Drugs.DrugName, FormTypes.DrugForm, Strengths.Strength, Strengths.Unit,
Drugs.DrugName & ", " & FormTypes.DrugForm & ", " & Strengths.Strength & Strengths.Unit
FROM Strengths INNER JOIN (FormTypes
INNER JOIN (Drugs INNER JOIN (DrugForms
INNER JOIN DrugFormStrengths
ON (DrugForms.DrugForm = DrugFormStrengths.DrugForm)
    AND (DrugForms.DrugName = DrugFormStrengths.DrugName))
ON Drugs.DrugName = DrugForms.DrugName)
ON FormTypes.DrugForm = DrugForms.DrugForm)
ON (Strengths.Unit = DrugFormStrengths.Unit)
    AND (Strengths.Strength = DrugFormStrengths.Strength)
ORDER BY Drugs.DrugName, FormTypes.DrugForm, Strengths.Strength;

I could have used this query as a combo box's RowSource in my demo for entering prescriptions, rather than correlating separate combo boxes as I did.

Inserting new rows into the Products Colours and Sizes tables, and the tables which model the relationship types between this and OrderDetails, would be done via a separate interface, in much the same way as drug, form and size data are inserted in my Relationships demo.  The product base is thus built up in advance of processing orders, though you could include a means of opening the forms to do this in the orders form, so that new product data could, if  necessary, be added while in the process of entering an order.
