    • net start cryptSvc
    • net start bits
    • net start msiserver

Now you may try running the Windows Updates and check if the above steps resolve the issue.

ken sheridan say: As you have two one-to-many relationship types, the interface should be a parent form in single form view based on the Project Main table, and within that two subforms based on the two referencing tables respectively.  In each case the subforms are linked to the parent form by setting the LinkChildFields and LinkMasterFields properties to File Number.

The subforms can be in single form view, continuous forms view, or datasheet view, though either of the latter two are more common in the case of subforms.

A key point to note is that neither of the referencing tables should have any columns other than File Number which duplicate columns in the referenced Project Main table.  To do so would introduce redundancy and the tables would not be in Third Normal Form, with the consequent risk of update anomalies.  The primary key File Number value of the current record in the parent form will automatically be inserted into the File Number foreign key column in the subform's table when a new row is inserted in the subform.

ken sheridan say: You haven't responded to Tom's request to post an image of the database's relationships window.  Seeing the structure of your database in this way would help us assist you.

Spreadsheets and relational databases are very different in both concept and methodology, so if you import an Excel worksheet into Access as a single table it will almost always be riddled with redundancy, and consequently require the sort of wasteful duplication of effort inserting data which you are trying to avoid.  More importantly the database will be wide open to the risk of update anomalies.

The database relational model is a formal model subject to formal rules.  Of these the process of normalization by decomposition is central to the design of robust and efficient databases.  So when data is imported into Access from Excel it needs to be decomposed into a set of related tables, each of which is correctly normalized and consequently contains no redundancy.

Having created a set of empty normalized tables, and created enforced relationships between them, the process of decomposition of the imported table then involves the execution of a set of 'append' queries in a specific order.  The rule of thumb is that data must first be inserted into a referenced table before any data is inserted into a table which references it, e.g. you cannot insert rows into an Orders table before rows have been inserted into a Customers table, and you cannot insert rows into an OrderDetails table until row have been inserted into both an Orders table and a Products table.

You'll find an illustration of how data imported from Excel can be imported into Access and decomposed into a set of correctly normalized related tables in DecomposerDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly. 

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file firstly imports some simple data from Excel and then decomposes it into a set of related tables.  An explanation of each 'append' query is given as you step through the demo.  Finally it shows how the data is then represented in the interface by a form/subform, in which data could be added or edited, and includes a simulation of how further data can then be imported and inserted into the tables.

In answer to the first sentence. You are missing what Ken told you. The ONLY value from the Projects table that should be in the other tables is your FileNumber. One of the main principles of a relational database is that data exists on ONE place only. So you neither need nor want the Project name and Vendor info in the child tables.
