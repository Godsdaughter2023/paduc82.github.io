On Excel, how do I copy the row heights on part of a spreadsheet to another set of rows on the same spreadsheet? The data, formulas and formats copy OK, but the row heights do not change with the copy, so I have to readjust all the various different row heights in every one of the new copies. Thanks

ron_at_buy say:

Instead of just highlighting the cells you wish to copy, click on the complete row (i.e. the row number to the left of your spread sheet) Hold the left mouse button and move down the row numbers you wish to copy. (This will highlight the rows you wish to copy).

Then Copy.

Move to the first row where you want to copy to - click on the row number (highlighting the complete row) then Paste. The row heights should be copied. Move down to the next area to copy to and repeat the Paste process. This will of course copy all the cell contents, formulae etc. so you may have to delete the contents if you only want the row heights copied.

Hope this helps.

ken sheridan say: It's important when designing a relational database that the validity of the logical model based on an analysis of the functional dependencies takes precedence over the design of the interface.

The key issue here is whether there can be more than one bid per request, i.e. can there be more than one row in the Bids table which references a single row in the Requests table.  If not then the Bids table is redundant as its non-key attributes are attributes of the request.  The question then is whether there can be more than one bid-service per request-service.  If not then the logical model, and hence the physical model, then becomes a very simple one of:

Clients---<Requests---<RequestServices>---Services

As with Bids, the non-key attributes of BidServices are attributes of RequestServices, making the former unnecessary,

The simplest interface  would be a request form with a request-services subform embedded in it, linked on RequestID, but you can if you wish use separate interfaces for inserting the request data initially and the bid data subsequently.

ken sheridan say:

1.  But for the purposes of my database, I have to keep Requests and Bids separate according to my assignment guidelines, as they represent separate operational business events.

2.  I plan to multiply this field, which is input by user into the subform, by the [Daily_Rate] field of the Services table.

1.  They can be presented as such through the interface, regardless of the underlying physical model, of which the user should be completely unaware, just as you are when you interface your bank account via an ATM or online.  By modelling the data as currently, you are not only making life unnecessarily difficult for yourself, but more importantly are introducing redundancy which leaves the database wide open to the risk of update anomalies.

2.  This raises the question of the time-independence of Daily_Rate on the key of Services.  Unless your company intends to keep the same daily rate per service until the end of time, which I think we can assume is not the case, only the current Daily_Rate is functionally determined by the key of Services.  You either have to assign the current value to a separate Daily_Rate column in the referencing table with code (analogous to the assignment of the current unit price of a product to an OrderDetails table which is found in countless ordering and invoicing databases, including Northwind), in which case the line total would be computed at runtime, or store the line total in a column in the referencing table and assign a value to it by multiplying the daily rate in the referenced table by the estimated days in the referencing table.  The former approach is the more usual one, but both are valid.
