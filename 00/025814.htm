How do you create a Lookup Field in a Table? Well... don't! See http://www.mvps.org/access/lookupfields.htm for a critique of what many of us consider to be a misfeature.

To create a Combo Box on a FORM is a different matter and very much a good thing to do. However, you cannot link two tables Autonumber to Autonumber. The Autonumber should be the primary key of the parent table; each child table should have its own primary key, independent of any other table's Primary Key, and should have a Foreign Key field, a long integer linked to the ID of the parent table.

Perhaps you should also come up with more descriptive names than Field1 and Field2. ken sheridan say: You first need to identify the entity types, their attributes (remembering that each attribute must be functionally determined  solely by the key of the table), and the relationship types between the entity types.  You appear to have entity types which, in broad outline, would be modelled by tables along the following lines:

Subjects
....SubjectID  (autonumber PK)
....Subject

....Authors
....AuthorID  (autonumber PK)
....FirstName
....LastName

Publications
....PublicationID  (autonumber PK)
....Publication

As each author might write for one or more publications, and there will be one or more authors per publication, there is a many-to-many relationship type between Authors and Publications which is represented by a table which resolves the relationship type into two one-to-many relationship types:

PublicationAuthors
....PublicationID  (long integer number FK)
....AuthorID  (long integer number FK)

The primary key of this table is a composite one made up of both columns.

Citations represents a many-to-many relationship type between Subjects and PublicationAuthors, and is modelled by a table like this:

Citations
....CitationID  (autonumber PK)
....SubjectID  (long integer number FK)
....PublicationID  (long integer number FK)
....AuthorID  (long integer number FK)
....Notes

PublicationID and AuthorID constitute a composite foreign key referencing the composite primary key of PublicationAuthors, so there should be an enforced relationship between these two tables on the two columns.  All other relationships are on the single primary and foreign keys columns, and should also be enforced.

Once you have this logical model set up and have tested it with some hypothetical data building the interface is relatively straightforward.  A subjects form, in single form view, would have within it a citations subform, in continuous forms view, linked on SubjectID, and within the subform would be combo boxes in which the publication and author can be selected, and text boxes bound to the Notes column and any other non-key columns, e.g. the pages in the referenced publication and/or the date of the publication.  A command button could be included in the subform to open another form in dialogue mode, in which a new row can be inserted into PublicationAuthors if one is not already present for the author/publication combination you wish to select.  In this second form the NotInList event procedures of the PublicationID and AuthorID combo boxes can be used to insert new rows into the Publications and/or Authors tables if necessary.  On closing the PublicationAuthors form the PublicationID and AuthorID combo boxes in the original subform would be requeried to show any new author and/or publication in their drop down lists.

That is wrong.  It's not quite clear what you are attempting.  If you are trying to filter the form by subject the RowSource property should be:

SELECT SubjectID, Subject FROM tblsub_Subjects ORDER BY Subject;

The ColumnWidths property should be zero.

If, on the other hand, you are trying to navigate to a specific citation record the RowSource property should be:

SELECT [tblCitations].[CitationID], [tblsub_Subjects].[Subject], [tblCitations].[Citation] FROM tblCitations INNER JOIN [tblsub_Subjects] ON [tblsub_Subjects].[SubjectID] = [tblCitations].[Subject] ORDER BY [tblsub_Subjects].[Subject],[tblCitations].[Citation];

In this case as you would wish to show two columns in the list the ColumnWidths property should be something like 0cm;3cm;3cm, the ListWidth property would then be 6cm (the sum of the ColumnWidths dimensions) and the BoundColumnn property should be 1.

General notes

Take some time to make a rough sketch of the database you plan to create. “A good database design is the most important step in creating a database that does what you want it to do” (Napier and Judd 2001, 89). 

To make the rough sketch, “you first need to identify entity types, their attributes (remembering that each attribute must be functrionally determined solely by the key of the table), and the relation types between the entity types” (Sheridan 2014).  An example of a rough outline for database SUBJECT_CITATION_SOURCE_NOTES.accdb (henceforth “SCSN”) is shown in Figure 1.

 

Figure 1: Rough Draft Diagram of SCSN


 

Data Types and Data Type Properties

Design View gives you control over table content.  When fields are added a data type is assigned.  “A data type is a classification of the type of data that a variable or object can hold in computer programming. Data type is an important factor in virtually all computer programming languages, including visual basic, C#, C++, and JavaScript” (Computer Hope 2014).

The data types used in SCSN follow:

  • AutoNumber is a unique number Access automatically inserts each time you add a new record to a table.  These numbers follow a consecutive number sequence.  You cannot change or update an AutoNumber data type. Access uses this data type to generate primary key values.  A primary key is a designated field that uniquely identifies a record. (Napier and Judd 2001, 94–97).
  • The Memo Data Type is used to enter lengthy text.  Memo fields can contain up to 65,535 characters per record. A property is an attribute of a Data Type that you can set.  For example, the Text Format property of the Data-Type Memo controls the format applied to the Memo field—it can be set to Plain Text or Rich Text.
  • Number Data Type is used to include numbers, such as those that identify a record. One of the properties of the Number Data Type is Long Integer, which defines the property Field Size as a whole number from
     – 2,147,483,647 to 2,147,483,647 (Hart-Davis 2009, 109).
  • Text Data Type, the default in Access, is used to enter text and numbers that  do not require calculation.  A text field can contain up to 255 characters per record.

Define Relationship

Relationships are established “by creating a link between fields in different tables.  The most common relationship is one-to-many” (Napier and Judd 2001, 195).  For example, there is a one-to-many relationship between tblCitations.CitationID and tblNotes.CitationID.  Each Citation and associated Source could have 0-to-many Notes (e.g., notes from one biology source could have several notes taken from just the one source).  The one-to-many relationship between the two tables is set as follows:

  1. Click the Relationships button on the Access Database Tools toolbar.
  2. Right-click and select Show Table from the context drop-down list.
  3. Add the tables tblCitations and tblNotes, and then close the Show Table dialog.
  4. Click and hold the CitationID field in tblCitations
  5. Drag the mouse pointer to the CitationID field in tblNotes
  6. The Edit Relationships dialog appears indicating a One-To-Many Relationship Type.
  7. Check the Enforce Referential Integrity check-box.  Enforce Referential Integrity means related tables need to stay connected to one another, even if you change the data.  For example, suppose the Citation “Korol 2000” is deleted from tblCitations.  Since tblCitations is related to tblNotes, the notes associated with Korol 2000 would become orphans—they wouldn’t belong to any table.  Access helps avoid orphans when Enforce Referential Integrity is check-marked on the Edit Relationships dialog, which means you cannot delete a record that has an existing relationship unless you first delete the relationship (Napier and Judd 2001, 198, 199).
  8. Click the Join Type button; this opens the Join Properties dialog.  “A join defines the relationship between two tables.  Access lets you set up three types of joins.  An inner join (option 1 on the Join Properties dialog) is the most common type.  This option includes only the rows where the joined fields from both tables are equal.  Because the CitationID field in tblCitations matches the CitationID in tblNotes, an inner join is selected (option 1).  Notice tblCitations.CitationID and tblNotes.CitationID are not exactly the same.  The first field has an AutoNumber field type (Long Integer property) and is the Primary Key (PK) for tblCitations, while tblNotes.CitationID has a Number field type with a Long Integer property and is the Foreign Key (FK) for tblNotes.  This is as it should be—the PK represents the 1 record in the relationship, and the FK is the many.
  9. Click OK → Create.  A line from tblCitations to tblNotes appears with the symbols 1 and ¥ indicating each record in tblCitations can be associated with many records in tblNotes.

While you cannot use the inbuilt watermark facility for both of the watermarks, you can do it by setting the Different first page option in the Page Layout>Page Setup>Layout dialog, and insert the graphic for the first page into the header of the first page with the text wrapping set to be behind text and adjust the brightness and contrast as necessary to achieve the desired "washed-out" effect.  Then repeat that process with the second graphic, inserting it into the Primary header (Add a second page to the document so that the Primary header is visible).

suzanne s. barnhill say:

As Doug says, a "watermark" is just a graphic anchored to the header, with wrapping set to "Behind Text" and color set to "Washout" (a specific combination of brightness and contrast). Although you can't insert two separate watermarks using the Watermark feature in Word, you can use that feature as a shortcut. Here's how:

  1. Before enabling "Different First Page," from the Page Layout tab, insert the watermark you want on the first page of the document.
  2. Double-click in the header area to access the header.
  3. Select the watermark and press Ctrl+X to cut it.
  4. Now add the watermark you want on the second and subsequent pages.
  5. In the Options group of the contextual Header & Footer Tools | Design tab, check the box for "Different First Page"; you'll now be in the First Page Header, and there will be no watermark.
  6. Press Ctrl+V to insert the watermark you cut.
