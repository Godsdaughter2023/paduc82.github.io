scottgem say:

Yes the problem is the lookup fields on the table level. You don't say what you tried, but the answer is simple. Create a query that joins your tables on the ProductID field and pull the ProductName field from T_Products. 

However, you have another problem. Your database is not structured properly. use of fields like Ingredients 1, 3, etc. are a violation of normalization rules. This is know as a repeating group. What you have is a many to many relationship here. One recipe can include many products and 1 product can be in many recipes. So you should have tables like:

T_Recipe

RecipeID (Pk)

ReciepeName

and a junction table to model the many to many relationship:

tjxRecipeIngredients

RecipeIngredientID (PK Autonumber)

RecipeID (FK)

ProductID (FK)

You may want to add fields to the junction table like quantity, etc.

However, this will cause a problem when doing a mailmerge. Because a Word mailmerge produces one document per record (there are supposedly ways around this, but I haven't gotten them to work) so, you will have a problem with mail merging. What you describe would probably work better as an Access report.

scottgem say:

You are working with a very old version of Office (XP). This is probably why you were having a problem with Northwind. You want to find an MDB version. 

But the way to do this is with a mainform/subform. The mainform will be bound to your Recipes table and the subform to the junction table and linked on RecipeID. The subform should have a hidden control for the RecipeID. This will be filled in automatically. Then it should have a combobox that lookups against the Product table for the ProductID. You don't add multiple products to one field. You need a record in tjxRecipeIngredients  for each ingredient in each recipe. So the subform will be in continuous form mode so it can display all the ingredients for the recipe displayed on the main form. 

If you want to do costing, I would add a Unit of Measure (UOM) field to your Products table along with a UnitCost field. Even if you don't use them now you will need them in the future.

marshall barton say:

In design view, reduce the height of the detail section by dragging the detail section's bottom edge up near the controls.

If that's mot the problem and you are displaying the form in continuous view, then you can reduce the form's overall height when there are not enough records to fill the form using VBA code in the form's Load event:

Me.RecordsetClone.MoveLast
If Me.RecordsetClone.RecordCount < N Then     'max number of records that fit in the form
   Me.InsideHeight = Me.Section(1).Height + Me.Section(2) _
         + Me.Section(0).Height * (Me.RecordsetClone.RecordCount  + 1)
End If
