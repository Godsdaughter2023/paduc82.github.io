peter schleif say:

Perhaps the file is UTF-8 encoded but has no BOM. In this case, Excel can not detect the encodcing and opens the file as ASCII. So you have to add the BOM manually. You can do that with a good text editor or a hex editor or also with VBA.

In Excel start the VBA editor with <ALT> <F11>. Copy the code below and paste it into the VBA-Window. Execute the code. It asks you for the CSV-file, opens it and saves it under a new name - this time with BOM.

Peter

 Sub AddUtf8Bom()
   Dim file As Variant

   file = Application.GetOpenFilename("CSV Files (*.csv), *.csv")
   If file = False Then Exit Sub

   With CreateObject("ADODB.Stream")
      .Charset = "UTF-8"
      .Open
      .LoadFromFile file
      .SaveToFile file & "_with_UTF-8_BOM.csv", 2
      .Close
   End With
End Sub

vasil l. michev say:

The trouble is that when you are using dirsync, disabling the account in AD will disable the corresponding user object in O365 and thus they will not be able to log. You can create separate, non-synced account for them, but accessing the old mailbox will require for you to assign a license to those accounts, so it will cost you. Or just copy the old mails to the newly created account, then move over the aliases and other attributes as well (which will take a lot of administrative effort)

Probably the best solution in your case will be to exclude the this accounts from the scope of dirsync, which will in turn provision them as disconnectors. Note that this will delete the associated account in O365 and the mailbox, so you will have to restore those manually from the O365 portal. And you will face some downtime in the process.

You can also consider excluding the 'useraccountcontrol' AD attribute from the scope of dirsync, but this will affect all accounts and it's not a very good idea. Now, if you are using the newest tool, AADConnect, you have a bit more flexibility with the rules, so you might be able to exclude said attribute for some users only, but I haven't tested that scenario...

Any idea what is causing this?
jay freedman say:
On the computer where it applies to the whole paragraph, open the Styles pane (Ctrl+Alt+Shift+S) and uncheck the box at the bottom for "Disable Linked Styles".

A linked style is one that appears in the styles list with both a ¶ mark and an 'a' to the right, meaning that it can be applied as either a paragraph style (including all the settings that apply to paragraphs, such as indents and space before/after) or as a character style (only the font formatting), depending on whether the whole paragraph is selected or just part of it. Disabling linked styles means that they can only be applied as paragraph styles.

Most of us who have studied Word over many versions feel that linked styles are a bad idea, because they can lead to odd behavior and sometimes to tangled and broken markup in the file. It would be far preferable to keep paragraph styles as just paragraph styles, and to make separate character styles with the same formatting if that's what you need.

stefan blom say:

On the "other" machine, display the Styles pane, for example by pressing Ctrl+Alt+Shift+S. Make sure that the "Disable Linked Styles" check box is cleared.

However, note that Heading 3 is a paragraph style (just like all of the built-in headings). In other words, by design it should apply to a whole paragraph.
