ken sheridan say: You can seed an autonumber by first compacting the database with the empty table, and then inserting a row into the table by means of an 'append' query, e.g.

INSERT INTO YourTable(CodeID)
VALUES(1000);

All other columns in the table must temporarily allow Nulls or you must also insert dummy values into those which do not.  The 'dummy' row can then be deleted and the table definition amended to disallow Nulls where appropriate.

However, this will not be a reliable  means of generating sequential values.  An autonumber guarantees distinct values, not sequential values, and should not be used for the latter.  If you need to do the latter it is necessary to generate the numbers yourself.  You'll find an example as CustomNumber.zip in my public databases folder at:

Advantage/Disadvantage of back end mdb vs accdb - Microsoft Community

self untaught say:

Advantage/Disadvantage of back end mdb vs accdb

Is there any advantage/disadvantage to keeping the back end data as an mdb when the front end is accdb/accde? Or should I convert the back end to an accdb? Or does it make any difference?
5 people found this helpful

ken sheridan say: The back end should be in the format appropriate to the earliest version of Access being used by the users.  If one or more user is using Access 2003 say, then it should be a .mdb file.  If all users are using Access 2007 or later then it can be a .mdb file, but normally it would be a .accdb file.

scottgem say:

There are some advantages to the 2007 accdx format. new features that were introduced with 2007 and later. 

But if you aren't using those features doesn't make much of a difference.

daniel pineault say:

There are new "features" available in accdb that don't exist in mdb, such as

Data types:

   Attachments - avoid them

   Calculated

   Big int

   ...

Data macros

Better encryption, if security is a concern and even then, Access altogether probably isn't the appropriate tool in such a case.

At the end of the day mdbs date back to access 2003 and prior, so 14 years old.  Unless you are needing to support older versions of Access you are probably best using accdbs.

Do be aware though that a 2007 accdb is not the same as an 2016 accdb.  Their are versioning issues.  Be sure never to open an accdb in a newer version or else you may get compatibility issue.  So as always, do all your development in the oldest version that will be used to use the database.

genevieve cri say:

Hi Dan Mastin,

 

Having an issue with sending a meeting invite using a third-party domain is caused by the following:

 

  • The sender’s IP or domain has a lower reputation with Outlook.com because of Spam activities, the IP is new, or the IP address of the domain is sending a million emails to Outlook.com users (Note: IP based domain have limitations of emails to be sent to Outlook.com users).
  • Anti-virus software’s “Internet Email Auto Protocol” or “Internet Email Protection”. If this setting is enabled, disable it and try sending a test message to our servers again.
  • The DNS of the sender’s domain is not configured correctly.
  • Namespace mining - This method is commonly used by malicious senders to generate lists of valid e-mail addresses that they can send spam, phishing emails, or malware.

 

With the above mentioned causes, Outlook.com users have nothing to do with the issue as it is created by design to protect Outlook.com users for any possible compromise of your personal information online. If you’re an Outlook.com user, it is best that you contact the email administrator of the domain and they need to contact our Outlook.com Postmaster by visiting this link. If you’re an email administrator, please follow the troubleshooting on this link. You can also request for delisting your IP address thru our Anti-Spam IP Delist Portal and/or Outlook.com Deliverability Support.
