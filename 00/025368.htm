Duration to show calender days - Microsoft Community

jerryagriffin say:

Duration to show calender days

To make this simple I have a project duration that I want to show as 4 calender days but Project shows it as only 2 working days (Friday to Monday)  I have Saturday and Sunday set as non working time and would like to keep it that way.  I would also still like to show the tasks as working days.  How do I make this happen?
john - project say:
Jerry,
Sounds like I guessed right. Unfortunately you can't do what you want in the Duration field, or any single field, unless you are willing to use a little VBA "magic" and a custom field.

You can enter either working days,  ("d"), or calendar days, ("ed"), in the Duration field for any performance task but a summary line will always display working days. However by adding the customized field as I noted above, and setting the option to use the formula for calculation of group and summary rows, then the normal Duration field will show working days for all task rows, (summary and performance), and your custom field will display calendar days. The following tweak to the formula will make it a little more evident:
DateDiff("d",[Start],[Finish])+1 & " edays"

If you want to can relabel the Duration field as Duration (working) and rename the custom field as Duration (calendar).

Now, about that VBA magic. The following code will give you what you want in a single field (i.e. Text2). If you decide to rename the field I suggest you not simply call it "Duration". Doing so can lead to confusion with the normal Duration field. I'd call it, "Duration (modified)", or similar.

Sub Duation_modified()
Dim t As Task
For Each t In ActiveProject.Tasks
    If Not t Is Nothing Then
        If t.Summary = True Then
            t.Text2 = DateDiff("d", t.Start, t.Finish) + 1 & " edays"
        Else
            t.Text2 = t.Duration / 480 & " days"
        End If
    End If
Next t
End Sub

A final comment about using two fields versus one with VBA. If you use the two field approach you may see more duration data than you want in your view but it will update automatically via the formula each time you make a change to task duration. If you elect to use the single field approach with VBA, you will need to run the macro to see changes each time a task's duration changes.

Hope this helps.

John

macropod say:
The formatting change has most likely occurred because the bolding was applied to the fields via hard-formatting rather than via a character Style or the Charformat switch. To apply a Charformat switch to, say, a «First_Name» field:

• Select the «First_Name» field and press Shift-F9. You should now see { MERGEFIELD First_Name } or { MERGEFIELD First_Name \* Mergeformat};

• Format the 'M' of MERGEFIELD as bold text;

• Delete the '\* Mergeformat' if it's there and insert '\* Charformat', so you end up with { MERGEFIELD First_Name \* Charformat}

• Select the field and press F9 to update.

Done. If a label merge is being used, you can propagate the change to all labels via 'Update Labels'.

macropod say:

Formfields have existed for well over 20 years - more like 30 - so that shouldn't ever have been an issue. Regardless, the problem you'll have with using mergefields is that, if you overtype them, you lose whatever formatting you'd applied via the \* Charformat switch and, if you type within them, whatever you've typed will get deleted as soon as anything causes the field display to refresh (e.g. a Print Preview).

If you want to persist with overtyping mergefields, retaining any formatting applied to them entails formatting the entire field with the desired attributes rather than using the \* Charformat switch. This is best done by applying the appropriate Character Style (e.g. Strong for bold text) to the field.
