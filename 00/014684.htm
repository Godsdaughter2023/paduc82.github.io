rob koch say:

Prashant is basically correct, since the multiple methods provided by Microsoft to update definitions for their antimalware products like Windows Defender can often create such anomalies if manual updates are performed in the client GUI.

The reason is that though all of the automatic methods get their updates from the same Microsoft Update system, the precise update package version these are pointing at can vary at any moment in time.  So if you have already downloaded the latest update available via a manual method, then this will often cause the automated Windows Update method to "fail" with an error.

The point here though is that this isn't truly a failure, it's simply a notification of an anomaly, since the earlier update has already succeeded making the later attempt unnecessary.

So in reality it's most often just the overzealous use of manual definitions updates which causes this, though it can also occur on occasion when the Windows Update client and Windows Defender client both attempt to perform an update cycle at nearly the same moment.  This causes a "race" condition where the first client to complete the update process "wins" the race, with the second client resulting in an error as it attempts to install the same or earlier definition version.

The reason that Microsoft originally decided to create these multiple update paths was to bypass potential problems where the antimalware (e.g. Defender) client wasn't properly updating itself, so the alternate Windows Update client method might still work in its place.  Since these 2 client systems are independent and mostly unaware of what the other is doing, they can collide at various points in the process resulting in several different possible error messages.

Rob

I don't believe that either of these things is possible; PowerPoint doesn't expose anything that would allow you to intercept clicks on content, or similar placeholders as such.  However ...

>> For example, I have code for creating and formatting PowerPoint tables and I would like users to be able to select a tableplaceholder, click a button and the code will create the table and embed it in the table placeholder? (Same for charts and pictures and the relevant placeholders.)

Suppose the user doesn't select a placeholder first but simply clicks a button that you supply.  You could then create the table (or other object as needed).  You could also consider the current selection; if it's a shape, you might use the shape's dimensions for your newly created object.

Another thought, though:

    With osh  ' reference to the shape
        If .Type = msoPlaceholder Then
            If .PlaceholderFormat.Type = ppPlaceholderTable Then
                If .HasTable Then
                    ' it's a table placeholder but there's already a

                Else
                    ' Add your table
                End If
            End If
        End If
    End With

stefan blom say:
When multiple versions of Office programs are installed on the same computer, they will inevitably "compete" over file associations, but you can work around that by setting the NoRereg value (to 1) in the registry for the version that you do not want to be the default. See the "Hacking the Registry" section in MVP Graham Mayor's article at http://www.gmayor.com/toolbars_in_word_2007.htm (note that the title and URL of the web page are a bit misleading).
