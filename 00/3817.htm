Cannot open Office documents from network shares named "home" - Microsoft Community

starglobe say:

Cannot open Office documents from network shares named "home" after 14.3.8 update

We received a report yesterday that a user could no long open office documents on their home network drive.  The OS is 10.8.5 and the Office version is 14.3.8.  The home drive is connected via smb://server1/home/org/USERNAME.  These files can be opened from other network shares, but not the home drives for server1 or server2, which both have the same home/org/USERNAME structure.  Also, none of these mac's are bound to the domain.

For an Excel document, the error is "'home:org:USERNAME:file.xlsx' could not be found.  Check the spelling of the file name, and verify that the file location is correct.  If you are trying to open the file from your list of most recently used files on the File menu, make sure that the file has not been renamed, moved, or deleted."  For a Word document, the error is "Word cannot open this document. The document might be in use, the document might not be a valid Word document, or the file name might contain invalid characters (for exampel,\/). (filename.docx)".

The files open with Preview, but not with the Office applications.  While testing from a 10.8.5 machine with Office 2011 v14.3.7, the files in home directories open fine, so it seems like something was changed in 14.3.8.

In testing again on the 14.3.8 version, I am able to open files when I manually mount the folder via Terminal - "mkdir /Volumes/home" "mount_smbfs //username@server1/home/org/USERNAME /Volumes/home".  This is fine for me, but this is not a viable method of mounting home directories for our users.

As this update was just released last week, we have not had many reports, but I expect this to change as more users update their Office.  Thank you for any insight you can provide.
starglobe say: The issue is partially resolved in Mavericks, thanks to a change in how network shares are mounted.  In Mavericks, when you mount a network share, smb://server/home/org/username , the share is mounted as /Volumes/username .  So in this sense, there are no issues opening Office documents, because the share is mounted as "username" instead of "home".

However, if you mount just the share itself, and attempt to drill down to a particular folder, you receive the same error.  If you mount smb://server/home , then the share is mounted as /Volumes/home and the problem remains.  In Mountain Lion and Lion, the share is mounted as /Volumes/home whether you use the full path or not.

After my initial post, I began searching for others having similar problems, and this seems to have been causing issues for a year or so now, though it didn't impact us until the 14.3.8 update.  For reference, see http://www.officeformachelp.com/2012/11/office-2011-applications-fail-to-work-with-volumes-named-home/ or http://arstechnica.com/civis/viewtopic.php?f=19&t=1182205 .  From the officeformachelp link, the issue applies to local volume as well (usb stick, hard drive, etc).  Any volume with the name "home" or "file" will cause these issues.
