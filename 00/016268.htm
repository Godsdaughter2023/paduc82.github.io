kosh vorlon say:

Hi 9ytxiS,

 

I don't know if it's a "bug" in the system or merely a problem with the documentation of the feature.

 

Consider that many people would find the automatic starting of a scan at an unscheduled time sometime later, likely by surprise, not all that convenient as it could cause performance and other issues if it were to occur without warning at essentially a random time.  Not to mention what might occur if the computer were shutdown for several weeks and several scheduled scans previously missed all began to run (either in parallel or in sequence).

 

Though I tend to run scans in the background while working (and generally recommend that), there are other alternatives.   Perhaps the following will help.

 

Here's a couple of things about scheduled scans. To run, the PC must be on and connected to the internet (it cannot be shutdown, in sleep mode, in hybrid sleep mode, or in hibernation). The power saving settings must be such that the PC does not shut down or go asleep during the scan (or it will be as if it never happened). Scans do not begin precisely when scheduled - some can start an hour or so earlier or later than the officially set time (depending on Microsoft's servers and the loads at that time). There is a way to use Task Scheduler rather than MSE to awaken sleeping computers, run the scans, and then shut them down when done, but it is somewhat complex. More on that shortly.

 

Also, go to settings and check scheduled scans and uncheck the box that prevents them from starting unless the system is on and idle (if you're using the system when it's scheduled or it otherwise isn't idle - perhaps downloading Windows Updates or doing something else - then the scan will not start and will be missed). If a scan is missed it does not start at the next available time, but waits for the next scheduled scan.

 

Go to Start / Control Panel / Administrative Tools / Services / scroll to Task Scheduler and make sure it is started and automatic. If not set it to automatic and restart it. If it won't start, click on the dependencies tab and do the same thing for each of the ones listed. If any of those won't start, continue with it's dependencies and do the same thing. When you get to the end, reverse course and Start the ones you couldn't because of the dependencies as you go until you get to Task Scheduler and hopefully it will now go to automatic and start. If you changed nothing read on. If you changed anything, reboot and see if that helps.

 

There is a way to use Task Scheduler rather than MSE to awaken sleeping computers, run the scans, and then shut them down when done, but it is somewhat complex. You can also use it to create different scheduled scans of different types since MSE only allows one choice.

 

Please see http://answers.microsoft.com/en-us/protect/forum/protect_start/running-mse-full-scan-as-a-scheduled-task/f627f5ce-0b74-4b1d-be64-d6ceeb4d9b02 - especially the later corrections from GreginMich.

 

The post from GreginMich here http://answers.microsoft.com/en-us/protect/forum/protect_start/scheduled-scans-not-working/fd769043-f67e-4990-8805-a02427f22dff will also be useful.

 

I'd also suggest the post from GreginMich here http://answers.microsoft.com/en-us/protect/forum/protect_start/i-would-like-to-schedule-mse-to-scan-pc-with/58474560-4cca-49c1-a51a-76ca9bc3b9f2

 

I hope this helps.

 

Good luck!

 

Kosh

james yan msft say: Hi Jean,

I'd like to explain that the "OneNote notebook" from the "New" menu is included in the default "Document" content type. There is no option to hide/remove the "OneNote notebook" from the New menu in a tenant level.

If we don't want users to create OneNote notebook from the "New" menu, we have to change this in each library. We can add custom site content types for Word/Excel... to the library and set the default "Document" content type to "invisible" under "Change new button order and default content type" in "Library settings".

For how to add custom site content type, you can follow these steps:

Take Word document as an example:
1. Open a new document in Word > "File" > "Save As" > "Browse" > select "Word Template" in "Save as type" > "Save". You will get a .dotx file.

2. Go to the site > "Site settings" > "Site content types" > "Create" > select "Document Content Types" under "Select parent content type from:" > select "Document" under "Parent Content Type:" > "OK" > Click "Advanced settings" > select "Upload a new document template:" > Browse > choose the .dotx file we created in step 1 > "Open" > OK.

3. Go to the library > "Library settings" > "Advanced Settings" > Select "Yes" under "Allow management of content types?"  > OK.

4. Back to the Library settings page > click "Add from existing site content types" > under "Select site content types from:", choose "Custom Content Types" > select the Site Content Type we created in step 2 > "Add >" > "OK".

For Excel/PowerPoint, the steps are the similar to the above.

You cannot set the Display Control of a Yes/No field in a table to Text Box.

And a Yes/No field in a table is never empty - it is either Yes or No, not blank.

You can create a text box on a form or report bound to a Yes/No field and set its Format property to

;"text for yes";"text for no"

However, when you click in the text box you'll see -1 for Yes or 0 for No, and you'll have to enter -1 or 0 to change the value, because -1 and 0 are the underlying values.

You cannot set the Display Control of a Yes/No field in a table to Text Box.


Have things changed in the current version of Access?  In earlier versions there was no problem setting a Boolean column's DisplayControl property to 'Text Box', and its Format property to 'Yes/No'.  However, I can see little point in doing so as users should of course never interface with data in a table's datasheet, but only via forms.  The same could be done with a bound control in a form, but, as you pointed out this makes editing of the values extremely clumsy, and a check box or combo box is really the only sensible type of control for a Boolean values.  Using the latter the visible values corresponding to the Boolean TRUE or FALSE values can then be whatever are desired.

If 3-way logic (which Codd himself supported) is required a Boolean (Yes/No) column cannot be used of course.  Instead a column of integer number data type should be used, with a ValidationRule property which allows 0,-1 or NULL only.

There would of course be no objection to a bound text box formatted 'Yes/No' being used in a report.
