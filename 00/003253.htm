jakivi say:

Hello all,

 

Comparing battery endurance is very difficult as there are so many variables but please find below some general guidelines. Some of them have already been mentioned here but I try to make some kind of summary.

 

-       3G (both talk and data usage) consumes more power than GSM. This is due different modulation methods allowing 3G carry more data than GSM.

-       The distance to nearest base station basically determines which transmitter power your phone is using. Higher transmitter power obiviously means higher power consumption. One base station can handle only certain amount of simultaneous connections so on rural areas the base stations are scattered whereas in cities there may be several on relatively small area.

-       WLAN (in phones which have it) and Bluetooth are trancievers which consume power. They should be switched off when not in use.

-       Multitasking takes power so shut down not needed programs which are running in the background.

-       The brighter the display the more power is being used.

-       If packet data connection has been set to “When available” and there is no packet data coverage, your phone periodically tries to establish a packet data connection which consumes power. Use Packet data connection setting “When needed” instead.

-       When the Network mode is set to dual mode in the network settings, the device searches for the UMTS (3G) network. If the signal strength of the UMTS network varies much in your area, your device must scan for the available network repeatedly which takes power. To use only the GSM network, select Network mode > GSM.

-       If the battery loses the power fast even though there is no features consuming the power, the battery may be wearing out and you need to replace it with a new one. This naturally should not be the case with a phone you have just purchased.

 

If your battery lasts very short time in normal usage, your phone should be checked at local Nokia Care point. If you have possibility to switch batteries for example with a friend whose device uses the same battery than yours, you can make some basic trouble shooting of your own. As mentioned several things affect to power consumption so the settings and environment should be as stable as possible during the testing.

Ask yor friend to charge his/her battery to full and charge your battery full using your device. Then switch batteries.

 

-       If your battery in your friends device does not show full bars your device charging calibration may be wrong. In practice this means that your device thinks that the battery is fully charged before it really is and therefore your device stops charging too early

-       Your battery might be faulty if your battery in your friends device shows full bars but doesn’t last as long as his/her own battery

-       If your battery in your friends device shows full bars and lasts about the same time as his/her own battery but your friends battery in your phone wears out quickly, then your phone may consume more power than it should.

Access 2010 Sort editable datasheet combo box Sort broken - Microsoft Community

normd say:

Access 2010 Sort editable datasheet combo box Sort broken

If there is a sub-form datasheet (or continuous form) with columns that are text boxes, check boxes AND combo boxes that allows the user to add or edit records and everything works correctly when the form is first opened.

If the user sorts the records on a column that is not a combo box, they can continue to add or edit records OK.

If the user sorts on a column that contains a combo box, then the user can no longer add new reccords or edit existing records - THE BUG

          This worked OK in earlier versions of Access

 

The record source for the form is a select from a single table (no joined tables)

The table contains a primary key

The table is lined to a SQL Database backend

The problem described above exists both in the full retail IDE environment and in the runtime environment when released to users.

 

Has anyone run into this and found a resolution, or do I need to raise the problem with Microsoft as a priority issue and request a hot fix from them.

02/16/2012 Additional information:

If the following scenario is performed, then the records remain editable:

  1. Open the form
  2. Right Mouse click and select filter by selection for one of the combo box current entries
  3. Right Mouse click and select Remove FIlter/Sort
  4. Right Mouse click one of the combo box entries and select Sort
  5. NOW THE RECORDS REMAIN EDITABLE AND THE USER CAN ADD NEW RECORDS!

Thanks

NormD

normd say:

FYI for anyone, I have determined what caused the problem and a solution (description is generic).

 

The combo box in the sub-form datasheet used a SQL View as its row source.  When Access generated the new query to perform the sort on the combo box column, it generated a query using an outer join to the SQL View and re-issued the query to SQL.

 

To correct the problem with not being able to edit after sorting on the combo box column I changed the linked View.  When the View was re-linked I explicitly selected the column that represents the primary key of the rows in the view.  Once this was dont, Access generated cursort to handle the sorting rather than a new outer join query.

 

In this instance, the number of rows is small so using cursors is not a noticeable performance hit.
