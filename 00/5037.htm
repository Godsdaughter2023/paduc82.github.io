First, you need to understand that forms don't hold data, tables hold data. What is not clear is what the workflow is. A subform is linked to a main form on a key field. If you are trying to show records in your Bid Services table for the RequestID shown in the main form, then this is accomplished by linking the subform to the main form on RequestID.

ken sheridan say: Scott, this is a classic road to Dublin question which I've previously addressed in the OP's other thread.  EstimatedDays and Negotiated price are merely attributes of RequestServices, so there is no need for, and should not be, a separate BidServices table. Similarly the attributes of Bids are merely attributes of Requests, so there is no need for, and should not be, a Bids table.

The root of their problem is that the OP is confusing the interface and the underlying physical model, and consequently making a rod for their own back.  With the simple model:

Clients---<Requests---<RequestServices>---Services

all that's necessary to differentiate the 'business events' is to have a Request form based on Requests with a RequestServices subform, the controls in the parent form being limited to the attributes of the original request, and those in the subform to the attributes of the service; then a separate Bids form and subform set up in the same way with the controls in the parent form limited to the attributes of the bid, along with whatever is necessary to identify the request in question, and the controls in the subform being limited to the attributes of the bid per service, along with whatever is necessary to identify the service in each row of the subform.  The only code/macro which might be required is to navigate in the Bids form to the request in question via an unbound combo box, which the control wizard can set up.

ken sheridan say: The value of a course which requires you to create an incorrectly designed database has to be questioned.  As Scott says, this should be taken up with whoever has set this assignment.

The batch insertion of multiple rows into a referencing table is not uncommon in the correct context, but this is not such a context.  You'll find a valid example in StudentCourses.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

In this little demo file the sfcCourseAttendances subform in the parent frmCourses form includes an 'All Students'  button which inserts rows representing all students currently enrolled for the selected course into the CourseAttendances table using the following code:

    Dim strSQL As String
    
    strSQL = "INSERT INTO CourseAttendances" & _
        "(StudentID,CourseID,CourseDate) " & _
        "SELECT StudentID, " & Parent.[CourseID] & ", #" & _
        Format(Parent.[sfcCourseSessions].[Form].[CourseDate], "yyyy-mm-dd hh:nn:ss") & _
        "# FROM CourseRegistrations WHERE CourseID = " & Parent.[CourseID]
    
    CurrentDb.Execute strSQL
    
    Me.Requery

One thing to note about this code is that it does not include the dbFailOnError argument when calling the Execute method of the database object, as would normally be done.  This is because, if the table, and hence the subform, already contains some of the registered students, we want the key violation error to be ignored, and the remaining students registered for the course to be inserted.

strSQL is a string variable that holds a SQL statement. Doesn't matter what data type your fields are. 

ken sheridan say: A string expression is being built as an SQL statement, do the variable is declared as String.  The data types of the columns in question are immaterial.

As for using my code as model I only posted it as an example of the basic methodology.  The SQL statement in my case is specific to its context in my demo; in your case you'd need to build an SQL statement which does what's required in your case.  In my case the SELECT clause returns only one column, StudentID, from the CourseRegistrations table, so this differs for each row returned; the other two columns are taken from the CourseID control in the subform's parent form, so this is the same value for every row returned, and  is inserted into the CourseID column in the CourseRegistrations table, and from the CourseDate control in the parent form's sfcCourseSessions subform, so again this is the same value for every row returned, and is inserted into the CourseDate column in the CourseRegistrations table.

Where your SQL statement will get its values from is for you to work out, and build the string expression accordingly.
