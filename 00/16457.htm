armen stein say:

This is a big topic, but here are a few comments.

1.  You don't have what you need.  You'll need SharePoint 2010 or 2013 if you want to publish on premises - the Enterprise edition, which is pricey.

2.  Alternatively, you can use Office 365.  You'll need a plan that supports Access services.  This is more difficult to determine than it should be.

3.  Access 2010 and Access 2013 have very different web publishing models.  Access 2010 stores data in SharePoint lists and can be designed offline and "published".  Access 2013 stores data in SQL Azure and needs a live connection to the database to design.

4.  Access 2010 web applications do not upgrade smoothly to Access 2013.  The database structure will transfer, and macro programming is reusable, but forms themselves will need to be rebuilt.  For this reason, Access 2010 web applications are essentially a dead end.  We recommend that people start with Acces 2013.

5.  When you say your application needs to be "available on for use on the web", I'm assuming you mean outside your own company.  If that's true, then Access 2013 web apps have more limitations, as anonymous (non-logged-in) users can't add or edit data.  You must create a named account for each user, or you could set up a generic account for users to share (but then you wouldn't be able to tell them apart and they'd all have the same password).

Overall, Access web apps are still new.  They're evolving and improving, but they don't have the same rich capabilities as traditional Access client applications.

Hope this helps,

Armen Stein

Those are good clarifications about your requirements.

Access 2013 web apps store their data in SQL Azure (SQL Server in the cloud).  That SQL Server data is accessible to other traditional Access applications, and it will perform okay if good design principles are used and you have a fast, reliable connection to the internet.

albert d. kallal say:

Well, if you move some Excel data to SQL server, or some Access data to SQL server, then at the end of the day what do you plan to use for the UI (the user interface).

Any kind of web publishing in regards to Access will require a complete re-write of the given application.

However, the goal may well be that you don’t mind having to install the application on each workstation, but all such applications then use + work + share the same common data.

The comments and suggestions about migrating the access tables to SharePoint in this thread have not changed. So you could consider pushing the data from Access to what are called SharePoint lists, and then you link your Access application to this tables. In general, using SQL for the tables is a better choice, but SharePoint tables (lists) can work.

But the considering now or when this post started has not changed.

So if you switching from a car to a helicopter, you want experience on that new machine. Same goes for a SharePoint migration of tables – you want to have experience and have done this a few times. That way you can identify what parts of Access will work well, and what parts will not. If your tables are less than 5000 rows – maybe to top out around 10,000 rows, and your relationships are standard primary key of auto number, and the child key is a standard long number, then this setup can work quite well.

You could also consider migrating the tables to SQL server, and continue again using your front end.

However, given the lack of experience and me not recommend to jump onto some technology platform without previous experience, one of the best choices would be to setup and adopt Terminal Services (and remote desktop). Such a platform would allow your application to be run by anyone with an internet connection anywhere, and zero changes likely would be required to your application. So this choice is great since “once” someone sets up the server, then your application will run as it does now without modifications.

If you move your data tables to SQL server, or say SharePoint tables, then your application will require modifications, and you having "good" experience with those new server systems.

So don’t confuse the issue of migrating the Access data (tables) to SharePoint or SQL server and that of SharePoint web publishing. SharePoint web publishing from Access will require a 100% re-write of your application. Such a setup does not support VBA or reports. So you be faced with a re-write of the application and learning a new platform. And if you have Access reports, then you really in trouble.

So most easy is terminal services as that means no changes to your application (and you don’t have to learn new tools).

Next up would be keeping Access as being installed on each workstation, but you migrate the data tables to SharePoint, or SQL server.

Keep in mind however that internet connections are VERY slow, and if you going to make Access work over the internet, then a LOT of work will be required in terms of performance. I explain this issue and solutions here:

http://www.kallal.ca//Wan/Wans.html

There are some Names that should not really be deleted. Consider this...

Sub DeleteNames()
  ' First seen posted by Bob Phillips
  Dim N As Name
  For Each N In ActiveWorkbook.Names
    If N.Name Like "*_FilterDatabase" Or _
      N.Name Like "*Print_Area" Or _
      N.Name Like "*Print_Titles" Or _
      N.Name Like "*wvu.*" Or _
      N.Name Like "*wrn.*" Or _
      N.Name Like "*!Criteria" Then
    Else
      N.Delete
    End If
  Next N
End Sub

andrea jones - all about resources say:

You have several offending names like _A123Graph_ACHART 1

First run a macro to unhide all the hidden names:

Sub allnamesvisible()

   For Each N In ActiveWorkbook.Names

     N.Visible = True

   Next N
End Sub

Now click on Insert ->Name ->Define and manually delete all the offending names which have a space before the number at the end.  After this there will still be some names Excel doesn't like.  I modified the code above to display the bad names:

Sub deletenames()
Dim N As Name
On Error GoTo ErrorTrap
   For Each N In ActiveWorkbook.Names
     If N.Name Like "*_FilterDatabase" Or _
       N.Name Like "*Print_Area" Or _
       N.Name Like "*Print_Titles" Or _
       N.Name Like "*wvu.*" Or _
       N.Name Like "*wrn.*" Or _
       N.Name Like "*!Criteria" Or _
       N.Name Like "*A123Graph*" Then
       Else
       N.Delete
     End If
   Next N
Exit Sub
ErrorTrap:
MsgBox (N.Name & Err.Description)
End Sub

When you've found each bad name you can manually delete it and then run the code until it meets another bad name but it should still be a bit quicker.
