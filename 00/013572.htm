robert aldwinckle on forums say:

No win Logo key.  Is there a workaround. 

Yes.  Learn how to avoid using a Win- key.   E.g. instead of pressing and releasing it by itself to open the Start menu use Ctrl-Esc.   Settings (Win-i) would be there.   Cortana (Win-c) could be there.     The Touch Keyboard for its Microphone button for Dictation (Win-h) could be there.   Also, The Touch Keyboard (and the OSK itself) could be used to generate  Win- key combinations.   So, you really don't need a real Win- key; however, if you have a Touch Screen the Task View (Win-Tab) could be there with swipe from the left and the Activity list (Win-a) could be there with swipe from the right.  Then via your swipe from the right you would also have access to the equivalents of Win-K and Win-P but both of those Touch screen shortcuts could be entered indirectly via their Taskbar icons anyway (Task view on the left and Notifications button on the right).  Etc.  Some things might be more awkward than they would otherwise be (e.g. Windows Snap especially) but you could learn how to work around not having easy access to those functions too.

FYI

Robert Aldwinckle

In Exchange hybrid environment, we have the following ways to create an Office 365 mailbox:

 

1. Move an on-premise Exchange mailbox to Office 365.

 

2. From on-premise Exchange, use the New-RemoteMailbox cmdlet to create a new Office 365 cloud mailbox and also create an associated mail-enabled user in the on-premises AD.

 

3. From on-premise Exchange, use the Enable-RemoteMailbox cmdlet to create a mailbox in the cloud-based service for an existing user in the on-premises Active Directory.

 

Both New-RemoteMailbox and Enable-RemoteMailbox apply to Exchange 2010.

 

If one user already has an on-premise Exchange mailbox, we cannot provision cloud mailbox for this user. This is an expected behavior. In Exchange hybrid, one user cannot have an on-premise mailbox and an Office 365 cloud mailbox at the same time.
