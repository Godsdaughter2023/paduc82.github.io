hpholm say:

The Windows Phone operating system maintains everything in the hidden WPSystem folder and its subfolders, and it is not possible to transfer the information to another SD memory card and have it work. Thumbnails and other files have unique names linked into other system files in the phone internal memory, which cannot be accessed.

Based on practical experience I suggest format the new memory card in the phone, then transfer only media files and documents via USB into their proper locations, reinstall apps from the Store, then let the phone handle the rest (and cry out loud about all the hours spent and stuff missing).

I'm currently trying to convince a Lumia 720 there are ~1,250 more pictures on the SD memory card than it displays in the gallery, and I will probably have to format the card and start over to fix it.

    john w. vinson say:

    The Filter string needs to be a valid SQL Query WHERE clause (without the WHERE keyword). If you can build a query to do what you want, you can just go into SQL view for that query and look at the WHERE clause to get what you need. It's also preferable to filter the report first rather than opening it for the entire table and then throwing away 95% of the data! You can use either the Filter or the WhereCondition argument of the OpenReport method to do this.

    So you could just build up the string piecemeal: something like

    Public Sub cmdViewEmpNotes_Click
    Dim strDocument As String ' for future flexibility with other reports
    Dim strWhere As String
    strDocument = "[Employee Notes]"
    strWhere = "EmpID = " & Me!EmpID
    If Me.Dirty Then
       Me.Dirty = False ' Save the record to disk if need be so it's available for the report
    End If
    If Len(Me!cboCategory & "") > 0 Then ' Did the user select a category?
       strWhere = strWhere & " AND NoteCategory = " & Me!cboCategory
    End If
    If Len(Me!txtNotesFromDate) > 0 Then ' Did they select a start date?
      strWhere = strWhere & " AND [NoteDate] >= #" & Format(Me!txtNotesFromDate, "yyyy-mm-dd") & "#"
    End If 
    If Len(Me!txtNotesToDate) > 0 Then ' Did they select an end date?
      strWhere = strWhere & " AND [NoteDate] < #" & Format(DateAdd("d", 1, Me!txtNotesToDate), "yyyy-mm-dd") & "#"
    End If 
    DoCmd.OpenReport strDocument, WhereCondition:=strWhere
    End Sub

    The date criteria insert the # date delimiters, and convert the date to an unambiguous legal date format regardless of what the user typed; and the DateAdd ensures that you'll see the last day's data if there is a time component to the note date.

    You should add error trapping code, etc. but this should get you started.

    Note that it's best to avoid using blanks or other special characters in names - I'd call it rptEmpNotes for instance. If you do use blanks then the [brackets] are [[absolutely essential]].

    ken sheridan say: In your original post you said the combo box was named cboCategory, not cboNoteCategory as in your code.

    I note that you are using a value list as the controls' RowSource property.  This is not good design as it is storing data as part of the form definition.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  You should have a table NoteCategories say, with a single column NoteCategory.  This table should be related to whatever table the notes are entered into, with referential integrity and cascade updates enforced.

    The combo box's RowSource property would then be:

    SELECT NoteCategory FROM NoteCategories ORDER BY NoteCategory;

    Should you wish to add, amend or delete a category it's then a simple task of editing the NoteCategories table, rather than amending the form definition as would be necessary at present.  You can also set up the combo box s that a user can enter a new category if necessary directly into the combo box, putting code in the control's NotInList event procedure to insert a row into the NoteCategories table.  For examples of the use of the NotInList event in various contexts see the demo file  NotInList.zip in my public

    swischner say:
    Maybe this is a (very common) misunderstanding. It's impossible that you have been using Autosave in Excel for the last 20 years, as this feature is rather new to Office 365 (introduced sometime in Summer 2017).

    Autosave is a kind of very granular working versioning system for your Office files, merging (almost) every little change you make on a document or workbook to the stored version and creating copies of the older versions.
    Autosave absolutely needs the files to be stored on SharePoint Online / OneDrive for business, as only these use the protocol needed. It basically is a side effect of the also rather new coworking functions (multiple authors working on the same file at the same time), which also require the files to be stored in the cloud (amongst a few other things).

    Now .. what you are thinking of is not *Autosave* but *Autorecover*, which is a somewhat similar function and much older. That's a mechanism which also saves versions of your document or workbook to recover from in case of a crash or similar problem. It's triggered on a timed basis (e.g. every 10 mins) and not on changes or edits to the file. Autorecover is what you are referring to in the Options menu and that should still work, no matter where the files are stored.

    Here is a quite good article I found about it: https://www.petri.com/autosave-office-documents
