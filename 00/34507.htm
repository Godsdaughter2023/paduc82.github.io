a_user_here say:

The sizes reported in the Control Panel and now in the Settings for a given application is based not on looking that the actual disk space that an application uses (would probably be too slow).  The number is a value that is put in the registry when the program is installed as an estimate of its size.  So it is the install program that is at fault.  Also please note that it is just an estimate that software manufacture puts in and is generally updated manually.  So some people tend to put in a slightly high estimate, and don't tend to update it on every release of their builds.  In the case of calibre I would suspect that the developer putting in the size estimate didn't understand what the units are being used.  Maybe he wanted 90 MB, and got 90 GB.

I do find it interesting that you show different sizes for calibre in the Control Panel and the Settings.  I looked at my own program, since I know what I put in, and in both cases it is 11.7 MB.  In my case I just checked and it is actually 11.1 MB on disk.  Note other reasons that the number can be off by some is if there application installs different things based on options.  In that case they probably would just pick a number that is greater than max it would take.  The original intend of this number was to let Windows know the size before installing, so that it could calculate if there is enough disk space free to install the program.

On the Temporary file clean up.  I'm also seeing that it does nothing, at least as I can see.  Maybe it sets some kind of schedule task to clean it up at a later time, but it is basically useless as it sits.  And the removed the disk clean up from the tools in the Properties for a disk.

As for where it is looking, it is in various places.

One is the standard temp directory: C:\Users\USERNAME\AppData\Local\Temp

Another is the cache that IE for browsing history:

C:\Users\USERNAME\AppData\Local\Microsoft\Windows\INetCache

Note that I found that the folders under INetCache are very "special" you can't see them in Explorer even if you select the options for showing hidden files, and the system protected files.  I found/deleted them using a program called WinDirStat.  I use WindDirStat for when I want to find out what is taking up disk space.

But you should also be able to do the same clean up from the IE settings.

Clearing out the two above took me down from 1.5 GB to 365 MB, so there are clearly other temp/caches out there that are included.

But on the whole I don't worry about such directories/files.  They are there for a purpose and usually are managed just fine.  The one exception being some files in the standard temp folder, but in fact that is mostly caused by the fact that a lot of programs don't clean up after themselves, and Windows doesn't have a periodic clean up of them. But even still I have seldom found that constant cleaning of the temp folder not really worth my time.  1.5 GB on a 256 GB drive is really just noise.

lemp say:

What happened to your system shortly before the problem started?  Any power failures/surges/nearby lightning strikes?

This line in the configuration report for your Ethernet (wired) network connection

      Autoconfiguration IPv4 Address. . : 169.254.81.171(Preferred)

means (1) your adapter is configured to obtain its IP address from a DHCP server; this is both the default configuration and the correct configuration for virtually all home networks and (2) your adapter did NOT receive an IP address from the DHCP server.

Essentially, your computer is not communicating with your router.  This is confirmed by the fact that no "Default Gateway" is listed.

This could be because

  1. The DHCP server in the router is broken
  2. The DHCP server in the router has reached the maximum number of IP addresses it is set to allocate
  3. The port on the router is broken
  4. The Ethernet cable is bad
  5. The port on your network adapter is bad

You didn't say how your other PC is connected, but however it's connected if it's able to access the Internet, the DHCP server in the router is almost certainly working properly.

Number 2 is highly unlikely.  That would only happen if you had used the router's configuration utility to set the number of allowable IP addresses to a very small number, lower than the number of devices connected to it (not counting the problem computer).  You would know if you had done this.

That leaves reasons 3, 4, and 5. 

To test for #3, connect your Ethernet cable to another port on the router (most home routers have 4 ports).  If your other PC also is connected to your router by Ethernet cable, disconnect its Ethernet cable from the router and connect the cable from the problem PC to that port.

You didn't say why you believe that it's not a cable problem (#4).  IMO, that's the most likely cause.  If your other PC also is connected to your router by Ethernet cable, disconnect the cable from the other PC (leave it connected to the router) and connect it to the problem computer.  Otherwise, obtain a known-good Ethernet cable and test using that.

Number 5 is difficult to test.  Apart from physical damage, it's unlikely, but still possible, that the port would fail while the rest of the adapter is functioning enough to give your the ipconfig data that you see.  With a cable connected to both the router and the problem computer, take a close look at the Ethernet jack on the computer.  You should see 1 or 2 small LED lights.  What do you see?  What color?  Are any of them flashing?
