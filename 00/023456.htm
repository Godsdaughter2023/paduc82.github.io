doug robbins - mvp office apps & services (word) say:

Even though a style may be based on Normal style, if the style has been modified to change an attribute, such as the Font, that change will not be overwritten by a change to the font in the Normal style.

Reur Heading styles, it is a good idea to have Heading 2 based on Heading 1, Heading 3 based on Heading 2 and so on so that a change to Heading 1 will cascade down to the other styles.  It is also a good idea to have Heading 1 based on nothing, so that a change to the Normal style will not impact on your heading styles.

a. user say:

I would have responded to this earlier but have been away.

The trouble with trying to fix these bibliography style .xsl files is that the way the code has been constructed makes it very difficult to assess the scope of any given change in the code. It is very easy to make a change that affects far more than you intend (e.g., that changes almost every ":" that the .xsl inserts to something else) but also very easy to make a change that fixes the results for one type of source but not for another.

Someone else recently posted a request for a fix to a style they called "TABOR", which seems to use the comma-separated layout you are looking for. You can have a look at the conversation at https://answers.microsoft.com/en-us/msoffice/forum/msoffice_word-mso_winother-mso_2016/custom-bibliography-style-works-with-word-2010-but/a835900f-9a74-4754-93ed-f683596591b0 - somewhere I have posted a link to a .xsl for that style. But perhaps that is not what you need either.

Other than Microsoft's own .xsl Referencing Style support, almost the only other .xsl s I have are based on Yves Dhondt's work - you may be able to find out about that at https://github.com/codingo/BibWord

and perhaps adapt something from there. 

My personal opinion is that either your school insists on using certain tools that it knows provides the formats you need (e.g. maybe it's easy to do this in LaTex or online citationbibliograpy systems) or if they want to allow people to use tool such as Microsoft Word, they either accept the results from a Microsoft-supplied standard .xsl, or they point you to one that they will accept and which actually works. My guess is that the number of people who could actually take one of these .xsls and modify it to work exactly as they require, in a short space of time, is vanishingly small, so the likelihood of "producing their own .xsl" is also small.


For example, I was able to modify the code to fix the specific example you gave by doing the following, but it is unlikely to work in the case where the Bibliography needs to generate an entry where the appropriate format is not "templateA" (you'd have to refer to the .xsl code to see what I mean).

Further, the values of some of the variables referenced in the code (e.g. the value of the ListSeparator, the GroupSeparator etc., are supplied by Word, as far as I can tell using a hard-coded table of values that depend either on one of the language settings relevant to the Word Application, the document, or perhaps the language specified for the citation. So doing these fixes may only work for English language citations and/or citations in other languages that use the same values.

In this template

<xsl:template name="templ_prop_GroupSeparator" >

change this line

    <xsl:value-of select="/*/b:Locals/b:Local[@LCID=$_LCID]/b:General/b:GroupSeparator"/>

to

    <xsl:value-of select="/*/b:Locals/b:Local[@LCID=$_LCID]/b:General/b:ListSeparator"/>

That should modify the ":" to a "," in most cases.

In this template,

  <xsl:template name="appendField_Dot">

change the following line near the end of the template 

<xsl:call-template name="templ_prop_Dot"/>

to 

<xsl:call-template name="templ_prop_ListSeparator"/>

(that should fix the "." at the end of the author, but will probably also affect rather a lot more than you might hope).

In this template, 

<xsl:template name="Field_Dot">

change the following line near the end of the template 

<xsl:call-template name="templ_prop_Dot"/>

to 

<xsl:call-template name="templ_prop_ListSeparator"/>

That should change the "." at the end of the title to a ",", but again, it may affet more than you expect, and may not work for everything that looks like a title.

Does Visio pro 2013 use Cartesian coordinates?

I'm trying to draw concentric 90 degree arcs with Visio and I'm great difficulty understanding data in Visio's 'Size and Position Sheet'; normally the x-axis is horizontal and y- is vertical.

paul herber say:

Yes, Cartesian coordinates are used, the X/Y values are relative to the shape's X/Y position, not absoluate on the page. However, be aware that the Shape and Position window only allows values to be entered for the shape values, there might be a formula there but you won't see it and entering a new value will overwrite the formula. Basically, don't use the Shape and Position window except to have a look at values. Use the shapesheet instead, that will show you the formulae used and any other cells used for the calculation of coordinates etc.

(You will need to enable the developer tab to be able to see the shapesheet editor).

For example, I want to calculate sales between (7/7/2014 4:00:00) and (7/8/2014 3:59:59). 

Is that possible?

hansv mvp say:

Let's say the sum range is A2:A100, and the date/time range is B2:B100.

Enter 7/7/2014 4:00:00 in one cell, say K1, and 7/8/2014 3:59:59 in another cell, say K2.

You can now use

=SUMIFS(A2:A100, B2:B100, ">="&K1+TIME(5,0,0), B2:B100, "<="&K2+TIME(5,0,0))

graham mayor say:
Your header says Word, your message says Spreadsheet? Let's assume Word.

Create a template containing a three column one row (or two rows if you want a header). Then add the following macro to the template (not to the normal template). http://www.gmayor.com/installing_macro.htm

Sub NextCell()
Dim oCell As Range
If Selection.Information(wdWithInTable) = True Then
    Set oCell = Selection.Cells(1).Range
    oCell.End = oCell.End - 1
    If oCell.Information(wdEndOfRangeColumnNumber) = 1 Then
        If oCell.Information(wdEndOfRangeRowNumber) > 1 Then 'If you have a header row
            oCell.Text = Format(Time, "HH:MM:SS") 'Set the format to taste.
        End If 'If you have a header row
    End If
End If
WordBasic.NextCell
End Sub

Create a new document from the template.

Whenever you tab out of the first column of the table the time is entered in the cell of the first column that you have just left. Note that if you go back to that cell, the time it contains will be overwritten.

Start in the first empty cell and use Tab to move between cells and to create a new row when you tab out of the last cell of the table.

Office application icons do NOT appear on the task bar by default. They are displayed when the application is running.

Lets make sure we are talking the same terminology.  The "Task Bar" is the black thingie at the bottom of your screen (it can be moved) that displays icons of running applications.  Applications MAY also be "pinned" on it to make them easier to start.

The "Start menu" is the default location where you launch programs from. You open it using the <CTL><ESC> shortcut or by clicking on the first icon on the task bar, an "angled window".  If you start typing on the start menu you invoke the Windows search function.  There is an area on the right side of the menu that displays "pinned" program icons. By default it is filled with a bunch of useless carp (including a bunch of advertising). You will have to clean it up to show icons you actually need. If you click on the "All apps" icon at the bottom of the start menu it will display all of the installed applications sorted in alphabetical order.

If you right click on an icon in the start menu you will see options to "Pin to Start", to place it on the pane on the right side of the Start menu or "Pin to Taskbar" to place them on the Taskbar so you can launch the applications "faster" from there.
