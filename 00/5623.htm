My guess is that your problem is with port numbers. After some research
I see that by default Communicator uses a minimum port of 5350 and a
maximum port of 5353. These can be adjusted to a minimum of 1024 and a
maximum 65535.

Ask your VPN administrator if he's blocking any ports outside this
default range. Both TCP and UDP should be enabled for this range as well.

babul a. mukherjee say: I am currently running Mac Cisco Anyconnect VPN 2.5.3055 and Communicator 2011 13.1.2 with no issues, including Enterprise Voice, desktop sharing, video, and voice.  I've been doing this since I got my Mac 18 months ago without issues.  It worked on Snow Leopard and now on Lion (after the 13.1.2 update).  No special setup on the Mac was needed.

I hope your IT/Lync administrator isn't tunneling the Lync/Communicator traffic.  It should be talking directly to the Lync Edge server regardless of whether the VPN is connected or not.

Tunneling Lync traffic is going to result in some poor/uneven performance at a minimum.  

If the Lync auto discovery server address is being tunneled, then my guess is the VPN administrator has set split-tunnel incorrectly on the domain name in use.  The Lync edge server name should not be tunneled.  You can test this by connecting the VPN and pinging the Lync edge server name and make note of the IP.  Then disconnect the VPN and do it again.  If the two IPs are different, the Lync server is being tunneled.

If the dial pad in Communicator is not showing up, have your Lync administrator refer to Enterprise Voice on Communicator 2011 for Mac.
