ken sheridan say: A tree structured hierarchy can be represented to a fixed maximum number of levels by a query which joins multiple instances of the table, each of which is joined to that above it in an outer join.  The result table of such a query would be extremely repetitious however, and not suitable as a means of presenting the hierarchy to the user.  It can be presented more coherently by means of a report, however, and you'll find an example in BoM.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

The primary file in the zip archive is the BoM.accdb demo which illustrates how to simulate recursive querying in Access to an arbitrary number of levels, to generate a bill of materials.  The PartsTree.accdb file in the same zip archive generates a bill of materials (parts explosion) to a fixed number of levels by the use of a query as described above, and presents this in an indented report as below:




A bill of materials is not a tree of course, because there can be multiple paths between nodes.  Mathematically it is more akin to a convergent graph.  Consequently the model differs from that for a tree, in which a single table refences itself in the way Tom described.  The classic model for a bill of materials on the other hand is the adjacency list model, in which a separate table has two foreign keys referencing the primary key of a single 'parts' table.  Consequently, a query to represent a tree would join multiple instances of the single table , rather than multiple instances of two tables as in my demo.  The basic principle is much the same, however.  You could of course model a tree by means of an adjacency list table, the difference being that one of the foreign keys in the table would also be its primary key, in which case the query would be similar to mine.  Whichever way it is done the design for the report in my demo would be similar, with multiple group levels, and the visibility of controls at each level being controlled by code in the report's module.

jay freedman say:

If all you want is a plain number, position the cursor where you want it, click Insert > Quick Parts > Field, and select the Page field in the list. The keyboard shortcut for inserting a Page field without going through the dialog is Alt+Shift +P.

The fancier "page number" items on the Header & Footer Tools > Page Number button come from the building blocks file, C:\Users\<user name>\AppData\Roaming\Microsoft\Document Building Blocks\1033\16\Built-In Building Blocks.dotx. (If your default language isn't English (US), then the 1033 will be replaced with the language ID for that language.) In order to see the AppData folder and everything below it, you must enable display of hidden folders/files in File Explorer.

To see what's in that file, in Word click Insert > Quick Parts > Building Block Organizer. Click the word "Gallery" at the top of the second column to sort the entries by that column. Scroll down to find the entries (if any) in the Page Numbers gallery.

If there are none, and especially if there are no entries that say "Built-in Building Blocks.dotx" in the fourth column, the file may be missing, corrupted, or had entries deleted from it. To fix that, shut down Word and start the File Explorer.

  • Go to the folder C:\Program Files (x86)\Microsoft Office\root\Office16\Document Parts\1033\16 and copy the Built-in Building Blocks.dotx file there -- that's the one that the Office installer created. (Again, if your default language isn't English (US), replace 1033 with the correct language ID.)
  • Paste the file into the folder C:\Users\<user name>\AppData\Roaming\Microsoft\Document Building Blocks\1033\16 (or use the correct language ID).
  • Restart Word and check that the Page Number gallery works.

Private Sub Workbook_BeforePrint(Cancel As Boolean)
  Dim Prompts, Defaults, Answer
  Dim i As Integer
   
  'Assume failure
  Cancel = True
 
  Prompts = Array("Enter Your Name", "Enter the day", "Enter Weather")
  Defaults = Array(Application.UserName, FormatDateTime(Date, vbGeneralDate), "")
 
  With ActiveSheet.PageSetup
    For i = 0 To UBound(Prompts)
      'Ask the user
      Answer = Trim(InputBox(Prompts(i), , Defaults(i)))
      'Abort if no answer
      If Len(Answer) = 0 Then Exit Sub
      Select Case i
        Case 0
          .CenterHeader = "&20" & Answer
        Case 1
          .LeftHeader = "Day: " & Answer & Chr(10) & Chr(10) & "Day:_________________"
        Case 2
          .RightHeader = "Weather: " & Answer
      End Select
    Next
  End With
  'Success
  Cancel = False
End Sub

If there are continuous section break in the document, it can be difficult to actually access the requisite headers and footers.

Running a macro containing the following code will however do it for you.

Dim i As Long, j As Long
With ActiveDocument
    For i = 2 To .Sections.count
        With .Sections(i)
            For j = 1 To .Headers.count
                .Headers(j).PageNumbers.RestartNumberingAtSection = False
            Next j
        End With
    Next i
End With

ravdberg say:

Finally found the solution on the Kerio forum. Apparently this problem had nothing to do with Windows Phone, but with a setting on the Kerio mail server. It didn't recognize Windows Phone as a valid device, but with this easy change it's fixed. Now everything is working smooth!. Everybody thanks for their help!

As some other Users posted in a Forum you have modify the mailserver.cfg of kerio Connect.

When you do this you do it at y our own risk.
We do it and it works now for over 2 months without problems.

Stop the Kerio connect services. Search for "mobilesync" in the mailserver.cfg file.
There is a variable called "AllowUnsupportedDevices". Set this Variable to 1.
Save the cfg file and start teh Kerio Connect Service.

One importend think is that you need the Certification File of Kerio on your Windows Phone.
This instruction is only needed if you have an self signed certificate.
Since there is no offical way to copy files to the phone the simplest way is to use e-mail with a attachment.
So export the cert File from Kerio Connect and rename it to "Name of Certificat.cer" and mail it to an e-mail account where you can access via webmail. Open this webmail via Windows Phone and install the attachment (the Certificat). Restart the Windows Phone. Have fun
