Setting up Page numbers for booklet size docs

you need to use the columns approach to set up your own virtual pages on each physical page. To place the page numbers (and have them work properly), you need to use calculations within nested fields. The following two fields show, respectively, how you would create the page number for the left virtual page and the right virtual page on each printed page:

{= {PAGE} *2-1 \* MERGEFORMAT }

{= {PAGE} *2 \* MERGEFORMAT }

Each pair of braces in this code is created by inserting a field using Ctrl+F9. In other words, position the insertion point where you want the first page number, press Ctrl+F9, and type the contents. To create the {PAGE} field, again press Ctrl+F9 and type the word PAGE within the resulting field braces. When you are done inserting the fields, they should look just like what is shown above.

Word gives you complete control over how it prints your document. You can either print to a printer or to a file. To print to a file, follow these steps:

1.   Choose Print from the File menu. You will see the Print dialog box.

2.   Make sure you select to print using the same printer on which your file will eventually be printed.

3.   In the Range box, specify what you want to print.

4.   Click on the Print to File check box. A check in the box means the output will be sent to a file.

5.   Click on OK. You will be asked to provide a filename for the output.

6.   Enter a filename, including a path if desired.

7.   Click on OK.

Once the output has been sent to a file, you (or someone else) can later copy the file to the destination printer from the Windows command line by using the copy command. For instance, let’s assume that you created an output file called Report.prn. You could later send this file to the printer connected to your parallel port by using the following command at the command prompt:

copy report.prn lpt1:

This command line syntax will work fine if your printer driver creates ASCII output. (For instance, if you use a PostScript printer driver.) If you use a different printer driver, the output file may actually contain non-ASCII characters. In this instance, you must issue the command line prompt as follows:

copy report.prn lpt1: /b

The addition of the /b switch causes the file to be sent to the printer in binary format, which is required for some types of output files.As a side note, if you want to produce a print file that you can later load into a text editor (such as Notepad), then you should make sure you are printing using the generic, text-only printer driver.

Properties such as ThemeColor, TintAndShade, PatternTintAndShade were introduced in Excel 2007. They are unknown in Excel 2003 and earlier. You'll have to remove all lines that use one of the above.

For text, you'll have to use code like

 

Selection.Font.Color = vbRed

or

Selection.Font.ColorIndex = 3

 

And for the fill color:

 

Selection.Interior.Color = vbYellow

or

Selection.Interior.ColorIndex = 6

soularts7 say:

XML format compatibility

I have Office for Mac 2011 which used to work like a charm but now I can no longer save a document in its proper folder." I am told that XML based format that word documents are saved in by default cannot store VBA macro codes." What is an XML based format? What are BVA macro codes? My documents are always saved in docx and I don't know how to change the  compatibility issue. Help!
john mcghie say:

Oh, here we go...  This can be a very long answer :-)

XML stands for "Extensible Markup Language".  It is one of a collection of applications of IBM's SGML (Standardized Generalized Markup Language).  You can read about them all here: https://en.wikipedia.org/wiki/Markup_language

Effectively, it is a way of describing the content of a document in such a way that both computers and humans can understand the content.  Compare it with the obsolete "binary" format of the original Microsoft .doc format. The .doc format was understandable only to a computer, and only if the computer was running software at the correct version for the .doc file you were trying to open.

Now:  To answer your implied question: There are four sub-types of XML format for Word. They are all written in XML.  Remember the word "Extensible" in the XML format?  That means you can "extend" XML in any way you like, to describe anything you like.  An exact analogy to using "English": you can use English to describe anything you like, from a love poem to a scientific paper, provided you follow the rules of English.  Same with XML.

Now the four different Word XML formats vary according to what they are "allowed to contain". This is a set of safety mechanisms designed to protect unskilled users.  There's a very detailed description here: https://msdn.microsoft.com/en-us/library/bb266220%28v=office.12%29.aspx?f=255&MSPPError=-2147217396

The simplest is the .docx format.  In it you can store text, tables, graphics.  That's all.  Word will remove anything else.

Next is the .dotx format.  This is a "Template".  Still coded in XML, but this time it adds a couple of things: it contains a style sheet that can be copied, and a "rule" that causes Word to always make a copy of it.  You use this when you want users to create a new copy of the document each time they use it: they can't change the original, but each time they open it they get a new file that is a copy of the template.

The .dotm format is the replacement for the "old" template file type. This is again a template, and again hands the user a new copy each time they use it: they can't change the original.  But this version is permitted to contain everything Word can make, including macros and other active content.

Finally we get to the rarely-used .docm format.  This is the deprecated replacement for the obsolete .doc format, in that it is allowed to contain "anything", including macros and other active content. In this version, the users can change the original. I say this format is rarely used because security software treats it with an extreme level of paranoia.  This is the terrorist's version of a soft-drink can: it can, and probably does, contain things that can seriously hurt computers.

Having said all that, let's get to your real question: "Why can't I put anything I like anywhere I want?"  well, of course, you can.  But you will run up against mechanisms designed to keep people safe who do not understand this stuff.

  • When you choose to save in .docx "Word document" format, Word allows you to save it "anywhere", and strips from the file anything that could hurt someone, such as active content.
  • When you want a "template" other users can only copy, you choose the "Word template" .dotx format. The same content as a .docx is allowed, saving anywhere is OK, but users can only copy it.
  • If you want to distribute macros or other active content, save as a "Word macro-enabled template" .dotm.  Word will now permit the VBA container to exist in the document, and will force a save into the "Templates" folder where the system security software can run it through the airport body scanner.
  • If you want a document that contains active content (say, form fields that have drop-down lists to enable the user to select content, and macros to control what happens when they do...) then you have the option of saving as a "Word macro-enabled document" in the .docm format.  Good luck emailing that anywhere: most email systems will simply strip the entire file out: those that don't will subject it to the computer equivalent of a body-cavity search :-)

Just to recap then: what you have is not a "compatibility" issue.  All of the files are in XML format.  But there are four "classes" of file, with different rules to what they are allowed to contain, where they can be saved, and how Word behaves when it opens one.

Don't get me wrong: you can still get issues in the XML format, but now it has to do with the age or "model" of Word.  There are several versions of XML out there, of which the latest is the flavour saved by Word 2016 on PC and Mac. This version is cable of containing things that older versions of Word can't create or edit.

Any version of Word, from Word 97/Word 2001 on, can "read" XML and is thus "compatible" with the format.  But the older the version of Word, the more likely it is to encounter something in the file that it does not understand.  In XML this is covered by a simple rule: "Display/edit what you can, ignore and preserve the rest!"

If you open a Word 2016 .docx or .docm in Word 2008, there will be quite a few things Word 2008 can't understand.  Mainly graphics, and some macros.  It will display and allow editing of everything it knows about.  It will attempt to downgrade the graphics. The macros may produce errors.  If you edit one of the downgraded graphics, Word will replace the original with the downgraded version.  Everything you don't change will remain in its original format, and your Word 2016 user will get it back undamaged.

So it's not correct to describe the issues we have as "compatibility".  The XML format can be extended to describe and store "anything". It is compatible with anything that can read XML, including your browser.  The issue is whether the receiving application can understand the content.  Sometimes it can't.  Usually, that doesn't matter.

If you need one of the new abilities, then you can simply upgrade to the latest version of Word to be able to create and edit them.  Other users will still be able to read and edit your documents, except that their version of Word may downgrade the new stuff.

Hope this helps

An Enterprise Agreement is designed for global companies, that enjoys the largest volume discounts, as well as these discounts can be applied to subsidiaries worldwide. The client can just add additional licenses without having to resort to buying CSP licenses, which would probably cost more than the EA price the customer currently enjoys.

The process of adding CSP licenses to an existing tenant, will require the existing tenant to be converted into a CSP tenant.

In the process of conversion, the support section of the tenant will be changed. The CSP partner will also be added as a delegate admin.  The CSP reseller will then be responsible for first line support as part of the T&C of being a CSP license reseller, where Microsoft Support desk will refer all support to be handled through the CSP reseller/Distributor before contacting Microsoft. It is currently not possible to have 2 tenants share a single domain.

Please ensure that your client HQ is comfortable with converting their main tenant into a CSP tenant before proceeding.
