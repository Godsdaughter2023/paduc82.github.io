The function is passed a string with a date value and 'first day of the week' parameter is set to vbSunday. The spreadsheet is basically a one month calendar (31 days layed out horizontally) and the macro colorizes the weekends columns based on the month and year provided. For example, in May 2012 the first weekend is May 5-6, however on the problem Mac it returns a value of 1 (Sunday) for May 2. To add insult to injury, after it falters in the first half of the month, it paints correctly the last two weekends and this is happening consistently. Any suggestions?


Thanks,

Arik


ariks say: Hi Jim,

After trying just about everything, I figured out the problem. The way the function Weekday() sees [uncustomized] system date differs between Microsoft OS and Mac OS. Originally I wrote the macro for PCs with region set to USA, meaning I fed the date to the function in mmddyyyy format. When the macro ran on Canada region, the function faltered and it turned out I had to feed the date in ddmmyyyy format to work correctly. Then I had to customize the macro to run on Mac OS but Weekday was feed the date in the same way as on PCs, one way for US, and another for Canada, and the "problem" Mac kept faltering. On the other Mac I mentioned in the post, the date was actually slightly customized and even though the region read Canada, the long date formats (first two rows in settings) were set to US format mmddyyyy therefore it worked OK. On my Mac it ran OK because I changed the setting to Canada but I really had to restart my Mac in order to be truly in Canada region, and as soon as did that, it started faltering on mine as well. So, I changed the code to only feed the date in ddmmyyyy format if Application.International(xlMDY) = False and in Windows, otherwise feed mmddyyyy.

The expiration dates for Windows 10 Technical Preview vary by build number.  The table below summarizes the expiration dates for each of the builds we’ve released so far.  (Dates are subject to change.)

Build number Expiration warnings begin License expiration date Windows will stop booting
9841 4/2/15 4/15/15 4/30/15

10049 9/17/15 10/1/15 10/15/15

Approximately 2 weeks before a build’s license expires, you will start getting warnings that this build will expire and that you should update to the most recent build. 

When the build reaches the license expiration date, the system will automatically reboot every three hours.

Approximately 2 weeks after the license expires, the build will no longer boot at all.

If you are still on an older build and are getting expiration warnings, it is likely that Windows Update is failing to install the latest build automatically.  To resolve the problem, download the latest ISO file from http://windows.microsoft.com/preview-iso-update.  After the download is complete, double-click the ISO file to mount it and then launch setup.exe.

If Windows has stopped booting due to an expired license, to resolve the issue you’ll need to download the ISO file from another computer and then transfer the ISO file to installation media such as a USB flash drive or DVD.   Then boot the PC from the installation media to install the latest build of Tech Preview.

  • Created the first POP account when Outlook started (creating a new unwanted pst file).
  • Redirected the new account's incoming mail to the old/restored pst file and set that pst file as the default.
  • After closing and reopening Outlook, deleted the unwanted pst file.
  • Added two additional POP accounts each directed to the same existing pst file.
  • Added an IMAP account after which I first noticed the ghost account/OST file.
  • Deleted the new IMAP account in an effort to remove the ghost account/OST file (but it didn't help)

Any help would be greatly appreciated!

diane poremsky mvp say:
Hi Ross,
Try deleting it from the registry.
Close Outlook.
Open the registry and go to HKEY_CURRENT_USER\Software\Microsoft\Office\16.0\Outlook\Profiles (you may want to export this key just in case you delete the wrong key)
Expand your profile (probably called Outlook).
Find this key: 9375CFF0413111d3B88A00104B2A6676 (it will have a triangle on the left)
Expand the key.
Look in each key to find the ghost account - delete it.

rossj_972 say:

Hi Diane,

  • Account Name = Outlook Address Book, Service Name = CONTAB
  • Account Name = Outlook, Service Name = MSUPST MS

Both these keys are consistent with another PC I have where this ghost account doesn't exist. So I assume they are unrelated.
