Why a-a doesn't equal to 0 in excel? - Microsoft Community

gracezhao414 say:

Why a-a doesn't equal to 0 in excel?

I was doing a formula, all figures were keyed in to 2 decimals:

254,499.77-282,631.34= -28,131.57

459,648.63-431,517.06= 28,131.57

but when i run (254,499.77-282,631.34)+(459,648.63-431,517.06)=(0.00)

out of curious, why it's (0.00) instead of -, i've opened more decimals to check. And was surprised to find out the answer was actually: (0.00000000002910383045673370000000)

why a-a doesn't equal to 0?

joeu2004 say:

The problem that you encountered is an anomaly of the way that Excel represents numeric values internally: 64-bit binary floating-point.  Consequently, most decimal fractions cannot be represented exactly. And the approximation of a particular decimal fraction depends on the magnitude of the integer part of the number.

(I will explain in more detail in a follow-up response.)

So in general, the remedy is: whenever you expect a calculation with decimal fractions to be accurate to n decimal places (n=2 here), you should explicitly round to

In other words, write:

ROUND((254499.77-282631.34)+(459648.63-431517.06), 2)

or

ROUND(254499.77-282631.34+459648.63-431517.06, 2)

-----

In this particular example,  the parentheses that you use are unnecessary.  And by coincidence,

=254499.77-282631.34+459648.63-431517.06 is exactly zero (0.00E+00 in Scientific format).

Nevertheless, =254499.77-282631.34+459648.63-431517.06=0 returns FALSE(!).

That anomaly is the result of a flawed design in Excel that tries to minimize the binary anomalies that you encountered. If the last subtraction of a formula is "close enough" to zero, Excel replaces the actual arithmetic result with exactly zero.

The design is flawed because, as demonstrated, it is not implemented consistently.

gracezhao414 say: In this case, can I conclude that, it was the parentheses, which caused the binary anomalies? thank you, Joeu.

joeu2004 say:

Grace wrote:

In this case, can I conclude that, it was the parentheses, which caused the binary anomalies?       

Not exactly.  It caused a different binary anomaly.

However, by coincidence, omitting the redundant parentheses allows Excel to apply a dubious "feature" that tries to hide some binary anomalies.

Consider the following more-complete explanation.

I wrote:

The problem that you encountered is an anomaly of the way that Excel represents numeric values internally: 64-bit binary floating-point.  Consequently, most decimal fractions cannot be represented exactly. And the approximation of a particular decimal fraction depends on the magnitude of the integer part of the number.

The exact decimal representation of the internal binary value is shown on the right.

254,499.77     254499.769999999,98952262103557586669921875
282,631.34     282631.340000000,0256113708019256591796875
459,648.63     459648.630000000,004656612873077392578125
431,517.06     431517.059999999,9976716935634613037109375

I use period for the decimal point and comma to demarcate the first 15 significant digits, which is all that Excel formats (rounded), substituting zero for any digits to the right.

Consider the following presentation of your example.

Formulas:

C1: =A1-B1

C2: =A2-B2

C3: =C1+C2

D1: =C1-ROUND(C1,2)-0

D2: =C2-ROUND(C2,2)-0

A4: =(A1-B1)+(A2-B2)

A5: =A1-B1+A2-B2

A6: =A1-B1+A2-B2-0

A7: =A1-B1+A2-B2=0

As D1 and D2 demonstrate, the values in C1 and C2 are infinitesimally different from how they appear, even when we format them with 10 decimal places (15 significant digits).  The exact decimal representation of the internal binary values in C1 and C2 are:

-28,131.57     -28131.5700000000,3608874976634979248046875
 28,131.57       28131.5700000000,069849193096160888671875

In contrast, the exact decimal representation of internal binary value of the equivalent constant is:

 28,131.57       28131.5699999999,997089616954326629638671875

The redundant -0 in some formulas works around the dubious "feature" wherein Excel replaces the actual arithmetic result with exactly zero (0.00E+0 in Scientific format) if the last subtraction of a formula is "close enough" to zero.

This is demonstrated in A5 and A6.  Note that A5 is exactly zero (0.00E+00), but A6 displays the infinitesimal difference in actual arithmetic (-5.82E-11).

The effect of the inconsistent implementation of the "feature" is demonstrated in A7. Even without the redundant parentheses around sub-expressions, the comparison returns FALSE.  This is because -B2 is the last subtraction of an formula.  So the calculation in A7 behaves more like A6 than A5.

The different infinitesimal anomalies in A4 (with redundant parentheses) and A6 (without redundant parentheses) arise because Excel is working with different binary values in the final arithmetic operation.

The exact decimal representation of the internal binary value of A1-A2+B2 is:

431,517.06     431517.059999999,939464032649993896484375

Compare with the exact decimal representation of the internal binary value of the equivalent constant, shown above.

-----

All of that might be interesting or overwhelming.

The take-away is:  we cannot predict when binary arithmetic anomalies might appear. So it is prudent to explicitly round arithmetic results, as explained previously.

coxy.t82 say:

I have fixed this problem on my machine by following the steps in that link and substituting "14.0" with "15.0".  That is:

 

1. Run regedit

2. Go to HKEY_CURRENT_USER\Software\Microsoft\Office\15.0\Common

3. Create a KEY called "Security" here by right clicking on the Common key and selecting New->Key and typing "Security"

4. Create a DWORD in the Security key by right clicking on Security, selecting New->DWORD and type in "DisableHyperlinkWarning"

5. Change the value of this DWORD to 1

 

Worked for me, good luck to the rest of you.

As you point out, having pdfs available for printing on SharePoint seems to defeat the purpose of Visio Services providing automatic refreshes. However, it is probably the best solution at present, but here is a tweak that does work....

You could have the source Visio documents open on a workstation that has code running listening for the recordsets refreshed event. The code can then automatically save as pdf, xps or even as web format. This method has the advantage that it could also perform shape additions and deletions, and inter-shape connections ... tasks that are not possible with just the data linked refresh in Visio Services.

By the way, Visio does have an InvisibleApp, so it could be running invisibly on a workstation...
