xp rocks. say:

MULTIPLE_IRP_COMPLETE_REQUESTS (44)
A driver has requested that an IRP be completed (IoCompleteRequest()), but
the packet has already been completed.  This is a tough bug to find because
the easiest case, a driver actually attempted to complete its own packet
twice, is generally not what happened.  Rather, two separate drivers each
believe that they own the packet, and each attempts to complete it.  The
first actually works, and the second fails.  Tracking down which drivers
in the system actually did this is difficult, generally because the trails
of the first driver have been covered by the second.  However, the driver
stack for the current request can be found by examining the DeviceObject
fields in each of the stack locations.

¡firedog say:
Any news?

This is a problem that many users have faced, and I don't know the precise reason for it. The fault is in either the encoding or decoding of the Quoted-Printable text. Because email only understands the ASCII character set, text containing any characters lying outside of it will be encoded one way or another before transmission.

If I take a Danish text and send it to myself using UTF-8 character encoding, it arrives and is displayed properly. However, if I then try to print it, the result varies with the message encoding: if it's QP, then I see the same result as you do. If it's Base 64, then it prints properly:

å, for example, is Unicode codepoint 229. Its two-byte UTF-8 representation is 1100 0011 1010 01012 or C3 A516. This appears in the QP text as =C3=A5. Instead of being converted back to 229 and the single character å, this is sometimes displayed (and printed) as the two characters C316 = 19510 = Ã and A516 = 16510 = ¥.

To prevent this from happening to people you send mail to, then, change the mail sending option for HTML to Base 64 encoding: in Windows Live Mail, press Ctrl-Shift-O for Options. On the Send tab, click HTML Settings under Mail Sending and select Encode text > Base 64. Select Allow 8-bit characters in headers as well. You can't do much about it as regards messages you receive.

You can see how the message is encoded by selecting it in the message list and pressing Ctrl-F3. The message body starts below the first blank line, and each part will have a header telling you the encoding (if any) and character set (charset) used.

You can copy QP-encoded text and paste it into any number of online Quoted-Printable decoders. On some sites, you'll get the right display, while on others you'll get a wrong one. I can't work out why.
