dirk goldgar say:

Thanks for posting that.  I was able to reproduce the problem and pin it down to a factor I hadn't considered.  It seems to be a bug that occurs only for PopUp forms.  Your form's PopUp property is set to Yes, which you hadn't mentioned and I hadn't tested.  With it set that way, it shows the incorrect behavior you described.  If I change the PopUp property to No, then it behaves correctly.

As I said, I think this is a bug, and I'll report it to the Microsoft Access product team.  It's not just a problem with the view-switching, either -- after switching the view, even references to Me.Recordset raise an error.

As for how you can work around it -- if the form really has to be a popup form -- I've found that this works:

Private Sub BegTime_DblClick(Cancel As Integer)

    '*** BEGIN Crazy workaround for bug
    Dim formName As String
    Dim currentPosition As Long

    formName = Me.Name
    currentPosition = Me.Recordset.AbsolutePosition  ' save current position
    '*** END Crazy workaround for bug
   
    Select Case Me.CurrentView
        Case 1: RunCommand acCmdDatasheetView
        Case 2: RunCommand acCmdFormView
    End Select
   
    '*** BEGIN Crazy workaround for bug
    Forms(formName).Recordset.AbsolutePosition = currentPosition
    '*** END Crazy workaround for bug

End Sub

The change in the view-switching code itself -- "Select Case" instead of "If .. ElseIf" and elimination of the unnecessary "DoCmd" -- is just a simplification of your original code, and not relevant to the fix.  While I think it's better this way, those changes aren't important. 

I should mention that this fix only works if the form is a main form, not a subform on another form, and only if it's a default instance of the form.
