You will need a vba solution to apply the same formula automatically in multiple sheets. Refer below code:

Sub AutoFillFormulasInMultipleSheets()
'this macro autofills formula in 10 sheets (specified by name - P1 to P10), range (L1:L10) from other 9 workhseets is referenced in each worksheet in consecutive columns starting from column A ex.
'sheet P1 range A1:A10 is populated ='P2'!L1 to ='P2'!L10; sheet P1 range B1:B10 is populated ='P3'!L1 to ='P3'!L10 & so on
'sheet P2 range A1:A10 is populated ='P1'!L1 to ='P1'!L10; sheet P2 range B1:B10 is populated ='P3'!L1 to ='P3'!L10 & so on
'sheet P3 range A1:A10 is populated ='P1'!L1 to ='P1'!L10; sheet P3 range B1:B10 is populated ='P2'!L1 to ='P2'!L10 & so on

Dim i As Long, l As Long, c As Long
n = 10
sheetlist = Array("P1", "P2", "P3", "P4", "P5", "P6", "P7", "P8", "P9", "P10")
For i = LBound(sheetlist) To UBound(sheetlist)
    For l = LBound(sheetlist) To UBound(sheetlist)
        If l <> i Then
            strFormula = "='" & sheetlist(l) & "'!L1"
            If l >= i Then
                c = l - 1
            Else
                c = l
            End If
            Sheets(sheetlist(i)).Range("A1").Offset(0, c).Formula = strFormula
            Sheets(sheetlist(i)).Range("A1:A" & n).Offset(0, c).FillDown
        End If
    Next l
Next i

End Sub

You mat adjust / change range per your requirement.

CONSTANT EXPRESSION REQUIRED? - Microsoft Community

faraz a. qureshi say:

CONSTANT EXPRESSION REQUIRED?

While tryin to use the NOW as the default value in one of the arguments in a UDF in VBA I am getting a msg "Constant Expression Required". What may be the reason and the way out?

 

Thanx in advance.

chip pearson say:

While Rick's approach will certainly work fine, I prefer not to use Variants as optional parameters. Rather I give them a specific type and a default value. E.g.,

Public Function YourFunction(Optional Count As Long = -1) As Boolean
    If Count = -1 Then
        ' argument not passed, execute missing param logic here
    Else
        ' argument passed, your code here
    End If
End Function

You should choose the default value of the optional parameter to be a number that would not be valid in the normal context of your code. For example, if YourFunction added up salaries for Count number of employees, -1 is a good default value because you would never really want to add up salaries for -1 employees -- it doesn't make sense.

Note that the IsMissing function applies only to Variant types. If a variable is any type of than a Variant, IsMissing will return False, even if the parameter is missing.  E.g.,

Sub BBB(Optional L As Long)
    Debug.Print IsMissing(L)
End Sub

The Debug message will be False regardless of whether L is passed to BBB.


This type of explicit optional parameter can be useful when passing values of built in Enums. For example, a function that compares two text values can test them case-sensitive ("A" <> "a") or case-insensitive ("A" = "a"). You can specify the comparison parameter as Optional and give it a default value of what option you want. E.g.,

Public Function CompareText(S1 As String, S2 As String, _
        Optional CompareMethod As VbCompareMethod = vbTextCompare) As Long
    CompareText = StrComp(S1, S2, CompareMethod)
End Function

Here, the user has the option of specifying the ComapreMethod but can omit it and use the default value. In this particular example, you would get the benfits of Intellisense, something you would not get with an Optional Variant.

I'm not saying that all this is better than Rick's approach. It is just another way of doing things and a matter of personal programming style.

See http://www.cpearson.com/Excel/OptionalArgumentsToProcedures.aspx for additional information about Optional parameters to procedures.

red sun setting say:

Well, I finally did find the answer on my own. Go figure after months of searching I figure it out only hours after asking the question online. I'll share the answer in case it helps anyone else.

I *did* have a rule configured. The rule was "If any item is over 60 days old and is not flagged, move it to the folder 'Follow up'". Most rules only run on the inbox, and then only on new messages that arrive. This rule apparently ran on my entire mailbox on a regular basis. So all of my old messages had been moved into my Follow up folder (which I never look at, in spite of its name).

I discovered this by looking at my Follow up folder today and seeing an item in there that I did not expect - I looked down and saw that there were over 50 pages worth of items in the folder.

The solution is to add a new criteria to the rule, "if the message is in the folder 'Inbox'" then move it. Now I have to move all of those messages back to their original folders. Thank goodness for the "Sweep" feature!

So, Microsoft, word of advice - when a rule runs on a schedule rather than on an event, you should let us know. Even better, let us edit the schedule ourselves ;)

rob koch say:

I'm not certain why there'd be a difference between the count displayed by these two interfaces, but it could easily relate to the way they were designed to count items like compressed files or in memory items such as processes.

The larger point is that, though it's still included in the Windows Defender folder, the Windows Defender Antivirus interface isn't linked from anywhere within the existing menu structure in the Creators Update interface that I'm aware.  To me this means that the Windows Defender Security Center is now considered the official interface, so I wouldn't be using the old Windows Defender Antivirus interface, nor consider it in any way official going forward.

If you recall, Microsoft Security Essentials (MSE) included the display of the file currently being scanned near the bottom of the GUI window, which I don't recall ever being the case in even earlier versions of Windows Defender.  However, watching this while scanning with MSE, you'd have seen not only the display of compressed files and their contents, but also those for processes and other special items not normally considered files by the typical user.

So the decision of whether to include these items, including whether to count the individual file contents of larger compressed files or simply consider them one container file is highly subjective.

Personally I've never considered these numbers of any true consequence, since they provide nothing other than a relative idea of the amount of work the application has performed and having nothing to do with the functions of security.  In fact, they've more often simply added to user confusion, since the very issues I've mentioned can result in their causing confusion when the typical person doesn't understand these technical differences.
