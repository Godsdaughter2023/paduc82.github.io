a. user say:

If you possibly can, add a column in Excel that holds the length of the whole expression to test, and test that against 25.

Word's "field language" does not have any of the familiar string functions such as LEN, LEFT, MID, RIGHT, CONCAT etc., and does not have a visible concatenation operator.

In fact concatenation is the simple part. For example, 

{ SET lcfsm "{ MERGEFIELD Last },{ MERGEFIELD First } { MERGEFIELD Middle }" } would set bookmark LCFSM to the first concatenated expression you have. The trouble is checking its length. Without using "tricks", you then have to do something like this:

{ SET LTTF 0}{ IF lcfsm = "" { SET LTTF 1 } }{ IF lcfsm = "?" { SET LTTF 1 } }{ IF lcfsm = "??" { SET LTTF 1 } } and so on until you have 25 question marks in the second comparand, then { IF LTTF = 0 { lcfsm \*Charformat } { lcfsm \*Charformat } }, and apply the appropriate formatting to the two { lcfsm \*Charformat } fields. 

You may not need the \*Charformat bits.

(Actually, I notice that the string you are testing does not have a space after the comma whereas the string you want to output does, so you might not be able to use the { lcsfm } field for output> or, you can do the final IF field as

{ IF LTTF = 0 "{ MERGEFIELD Last }, { MERGEFIELD First } { MERGEFIELD Middle }" "{ MERGEFIELD Last }, { MERGEFIELD First } { MERGEFIELD Middle }" } and apply the appropriate formatting to each part of the output.

But if you can bring the length in s an extra Excel column, you can do

{ IF { MERGEFIELD theLength } < 25 

as the first part of your IF field and there would be no benefit in creating a new bookmark called lcfsm. If you intend to format the entire output string the same way you might also create an Excel column that does the string concatenation as well.

Some syntax notes:
  - all the { } have to be the special field code brace pairs you can insert using ctrl-F9. You can't just type them

  - put a space on either side of the comparison operand (in this case " < "). That isn't actually the complete rule but IMO it is the easiest thing to remember that always works

  - you need the double quotation marks that I have indicated in the "result" part of the IF field, otherwise WOrd cannot know where the Tru and False results begin and end

 - you can eliminate a lot of the other space if you really want.

There are other ways to do the length comparison if you cannot do it in an Excel column, but they are all harder, and some of them may not work with some of the data you have.

macropod say:

What you could do is use a single-cell table with a fixed height and width for the data and a macro for the auto-resizing.

The following macro intercepts Word's 'Finish & Merge'>'Edit Individual Documents' command, so your merge runs normally, but then uses the table 'fit text' option to shrink the text in any 'first' paragraphs in table cells to fit the available with. This is far more flexible than changing font sizes based on character counts.

Naturally, for this to work, you'd need to either add the macro to the mailmerge main document's template or add it to the mailmerge main document and save that in the .doc or .docm format. Other than that, you'd notice no real change to your mailmerge processing.

Sub MailMergeToDoc()
Application.ScreenUpdating = False
Dim Tbl As Table, Cll As Cell, Par As Paragraph, sCllWdth As Single, sParWdth As Single
With ActiveDocument
  With .Tables(1)
    For Each Cll In .Range.Cells
      Cll.WordWrap = False
    Next
    With .Cell(1, 1)
      sCllWdth = .Width
      With .Range.Paragraphs(1)
        sCllWdth = sCllWdth - .LeftIndent - .RightIndent
      End With
    End With
  End With
  .MailMerge.Execute
End With
With ActiveDocument
  For Each Tbl In .Tables
    For Each Cll In Tbl.Range.Cells
      If Len(Cll.Range) > 2 Then
        For Each Par In Cll.Range.Paragraphs
          With Par.Range
            sParWdth = .Characters.Last.Previous.Information(wdHorizontalPositionRelativeToPage)
            sParWdth = sParWdth - .Characters.First.Information(wdHorizontalPositionRelativeToPage)
            If sParWdth > sCllWdth Then .FitTextWidth = sCllWdth
            If .Characters.Last.Previous.Information(wdVerticalPositionRelativeToPage) <> _
              .Characters.First.Information(wdVerticalPositionRelativeToPage) Then
              .FitTextWidth = sCllWdth
            End If
          End With
        Next
      End If
    Next
  Next
End With
Application.ScreenUpdating = True
End Sub

For PC macro installation & usage instructions, see: http://www.gmayor.com/installing_macro.htm
For Mac macro installation & usage instructions, see: https://wordmvp.com/Mac/InstallMacro.html

To create a single-cell table with a fixed height and width, use Insert|Table and choose one cell. Then, for the:
• rows, set the exact row height under Table Tools>Layout>Properties>Row>Specify Height>Exactly.
• columns, set the preferred column width under Table Tools>Layout>Properties>Columns,
and uncheck the 'automatically resize to fit contents' option under Table Tools>Layout>Properties>Table>Options. You might also want to set the cell margins to 0 all round.

If necessary, you can also change the table's text wrapping to 'Around', via Table Tools>Layout>Properties>Table, so as to reposition the table wherever you need it to appear on the page.

Subscription is for unlimited calling for the duration of that package. If you are heavy caller, better to use subscription. But they also cover limited countries, in general 63 countries for World subscription. Also while you can call on all countries within the package on landline, only few countries are supported for mobile calling. It also excludes special, premium and non-geographic numbers.

Credit it pay as you go, hence this is per minute kind of calling plan. Covers even those countries which are not covered by subscription and you can call both mobile and landlines. Hence, if you want maximum cost advantage and flexibility - You need to have both. Wherever you can use Subscription, use that. Otherwise you will need to use Credit.
