For example I wish to have a field where I can put "Version 1.0" text and then that is copied to places on other pages. And if I change that to "Version 2.0" it updates it on all the pages where I have that field. Much in the same way that a header or footer works but for other parts of the document.

Please help!!

 
john korchok say:

There are several ways to do this, I'm sure others will chime in with their favorite. 

If the words are appearing on the cover page and then are to repeat in the headers or footers or the document, create a separate Style for each item of information. The styles can look the same, they just need to have different names. If each item of original information can be on its own line, you can create paragraph style. If each piece of data is embedded within a paragraph, then create character styles and apply them to the words or phrases to be repeated. Then create StyleRef fields in the locations where that info should appear. The fields will automatically echo the data, though you may have to print or Preview the file to force the fields to update.

You can also create Bookmarks around each piece of information, then use a Ref field to echo that. Bookmarks are a little more fragile, as they can be erased by editing the bookmarked text.

feelew say: I had that problem before. Now, I'm strictly using the HYPERLINK function. The path being included within the formula instead of somewhere in a location that I am not quite sure about seems to maintain them better.

HYPERLINK(link_location, [friendly_name])

The first argument should be a string, and the optional friendly name can be just about anything , except a value that returns an error, in which case the hyperlink will not be created. If the friendly name is omitted, the link will display the link_location.

In a cell, it could look like this:

=HYPERLINK("C:\Users\Public\Music\Sample Music","Here")

The link can point to either a folder (like in this case) or a specific file.

Hope that can help.

albert d. kallal say:

Is it possible that intStartHr unbound?

If that text box on the web form is un-bound and let's pretend you enter 5 into that text box.

The following code will fail:


If [intStartHr] = 5 then
   MessageBox  (this is 5)
End if

The reason why the above fails when intStartHr is un-bound is it is a text data type, but we doing a comparisons to a number.

Two ways to fix the above

One: cast/convert such as this;

 

If cdbl([intStartHr]) = 5 then

 

Or in the text box property sheet in the format tab, change the format to "general number". This will fix the original first code posted.

The other issue could be that the system tends to be more "picky" if the column you are referencing does not have a corresponding text box placed on the form – so try ensuring that a text box bound to the underlying column is on the form if in fact this is a bound column. If this works, then set the text box visible = false if you did not need the value displayed to the user.

So, be REALLY careful of data types when using un-bound text boxes on a web form.

If the text boxes such as intShartHr is in fact bound, then check the data type of the underlying type in the table it is bound to - if this is a text type, then that explains this issue. And if it is a text type, then your "range" checking would not make sense.

Note that the above issue will NOT be seen when you run the form local as client in Access, but only as web published (access in the client side tends to do more "auto casting" of the data types, but when running 100% web, then forms code is VERY sensitive to data type miss-matches).

albert d. kallal say:

As noted for ANY type of filter needs a full qualified forms reference, and since we don't have a "me" to represent the current form like we do in VBA, then I always set controls into local vars and then use those for filters.


Eg:

setLocalVar  (lngInvoiceNum, [PromptInvoiceNumBox])

setfilter ([InvoiceNumber = [LocalVars]![lngInvoiceNum])

 

The above thus allows one to re-name the form, and also use the above code if place the form inside of a sub form or inside of a navigation control (so no forms! name in the above code).

 

Also one more big got-ya:

 

When the form does not have a record, then those bound controls become un-defined. That means EVEN code to with a nz() for null values will fail.

Eg:

setLocalVar  (lngInvoiceNum, [PromptInvoiceNumBox])


Above will NOT set lngInvoiceNum and that var is thus un-defined at this point in time.

 

Worse, even using this fails

setLocalVar  (lngInvoiceNum, nz([PromptInvoiceNumBox],0))

 

So even adding nz() to above does not work.

 

You  have to set error handling to "next" and then this will work:

So you actually set on error for next, and do this:

 

setLocalVar  (lngInvoiceNum,0)
setLocalVar  (lngInvoiceNum, nz([PromptInvoiceNumBox],0))

 

You have to do the first line to put a value into lngInvoiceNum since the next line of code will fail to execute if there is no record.

 

So perhaps you code is failing since there is no record, and that means controls at that point are un-defined?
