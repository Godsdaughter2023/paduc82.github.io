based on your description, i understand that you want to create a shared calendar where only 10 people in your organization can view, edit and modify items on it.

it is feasible to do that in office 365, here are the steps:

1. create a shared mailbox and add the 10 people to the member of it. to do this, sign in to exchange admin center (https://outlook.office365.com/ecp) with the admin account, navigate to recipients—shared and click “+” to create a new shared mailbox and add the 10 people to the member of it.

please note that after adding the users to the members of the mailbox, they have full control for not just the calendar but also the emails in it.

2. if you just want to give the calendar permission of the shared mailbox to the users, follow the steps below:

a. create a shared mailbox but don’t add the 10 users to the members of it.

b. create a security group and add the 10 users to the members of the group. to do this, sign in to exchange admin center, navigate to recipients—groups and click “+” to create a new security group, add the 10 users to the members of the group, click save at last.

c. connect to exchange online using remote powershell.

then run the following command in powershell:

add-mailboxfolderpermission -identity *** Email address is removed for privacy ***:\calendar  -user *** Email address is removed for privacy *** -accessrights editor

please replace *** Email address is removed for privacy *** and *** Email address is removed for privacy *** with your actual shared mailbox and security group email address.

after the permission is set for the group, users inside the group can open the shared calendar via the below steps:

1. in outlook 2013, right click my calendars, then select add calendar –from address book, choose the shared mailbix and it will appear under the main calendar.

2. in outlook web app (https://outlook.office365.com/owa) – calendar, right click my calendars, select open calendar, type the shared mailbox address under from directory, click open.

marshall barton say:

I would use two recordsets, one for getting the fields from the old table and the other to write the new records to the new table.  For well delineated list like you have, the InStr is kind of tedious when the Split function can do it all in one shot.  The general outline of the code could look like:

 

Public Sub SplitField()
Dim db As Database
Dim rsOld As DAO.Recordset
Dim rsNew As DAO.Recordset

Dim varList As Variant
dim k As Integer

 

Set db = CurrentDb()

Set rsOld = db.OpenRecordset("old table")

Set rsNew = db.OpenRecordset"new table)

 

Do Until rsOld.EOF

   varList = Split(rsOld!field1)

   For k = LBound(varList) To UBound(varList)

      rsNew.AddNew

         rsNew!Field1 = varList(k)
         rsNew!Field2 = rsOld!Field2
      rsNew.Update
   Next k

   rsOld.MoveNext
Loop

 

rsOld.Close
rsNew.Close

End Sub

 

You can the call that procedure from a command button on a form, or, if you only need to that once, call it from the immediate window.

bob jones aka: cybertaz say:

No, the changes you make are only held in RAM [memory] until you actually save the file. At that time, what you've done is written to disk [storage]. Once you close without saving, though, memory is cleared -- any unsaved changes simply "evaporate".

Just a friendly word of advice :-) Don't ever go extended periods without saving your work, even if you have some sort of 'automatic' features in place. One grows reliant on such "conveniences" & when they fail it's all the more devastating. A simple Command+S once in a while keeps instances like this to a minimum & reduces the loss when they do occur. How frequently you save is up to you -- there is no rule. Some people save every 15 minutes, some save after every paragraph, some save any time they make a critical change. The important point is that you get in the habit of saving regularly so that even if there is a system failure you're most likely to lose only what has been done since the last Command+S.
