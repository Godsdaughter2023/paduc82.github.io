marshall barton say:

The typical arrangement for that kind of structure is to add a subform to the patient form.  The subform is bound to the relationshpXT table with an invisible text box for the patientID and a combo box bound to the parentID field.  The combo box's RowSource would be a query based on the parent table.  The subform control's LinkMaster/Child properties are set to the ParentID field.

You will need the standard NotInList code so you can enter new parent records and that can open a parent for if you need to enter additional parent information such as the relationshp.

vanderghast say:

You can make a query which will have the two tables joinned, and use that query. Doing so, using an existing ParentID in the childtable.ParentID field, will 'lookup' the values from ParentID for you, and using a not existing ParentID (a new one), a new record will also be appended in the parent table and in the child table... Sure, that assumes this is what you want, and it is NOT a typo. So, generally, to avoid adding new records in parent tables due to typo, we prefer to use a form (or subform) for the children, and if a new ParentID is dectected, a MODAL form is open to add the new parent (like a NOT IN LIST combo box logic).

 

So, in short, yes, there is an easier way, but due to typo, it is not often used, and two forms approach is generally used.
