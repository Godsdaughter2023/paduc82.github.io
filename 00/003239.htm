cellular plus sim card - Microsoft Community

tobbyxp say:

cellular plus sim card

what can I do if my Lumia Phone keep hiding sim card?

itsonlyacomputa say:

Ok this can be a card type issue. However you need to do a few things first

1: Check that it is the right card type for your phone. You can't stick a micro sim in a Nano sim port or vice-versa. They have different pin positions and settings.

2: Go to Settings/Extras/Network Settings. Check that sim is visible and networks is turned on. If it is dual sim go back to Extras and go into Smart Dual Sim and check that both sims are on.

3: Go to Settings/Network and Wireless/Mobile and Sim. Check mobile service is on and sim is on.

4: Go to Search window scroll to find Messaging and Phone. Both these appear in your Apps and Services on your phone as Icons. Both are required to run the sim module to access your plan service. If these don't have a relevant service listing beside (e.g. Messaging - Optus and Phone - Optus) you probably have a faulty Sim card.

5: Call (phone) your service provider for a replacement Sim Card.

ken sheridan say: You can do it in a query:

SELECT
SUM(IIF([Play baseball],1,0)) AS [Basball players],
SUM(IIF([Play football],1,0)) AS [Football players]
FROM [YourTable]

or in computed text box controls in a form or report bound to the table with a ControlSource property of:

=Sum(IIf([Play baseball],1,0))

and

=Sum(IIf([Play football],1,0))

However, having separate columns (fields) for each sport is not a good design.  It's what's known as 'encoding data as column headings'.  A fundamental principle of the database relational model is the Information Principle.  This requires that all data be stored as values at column positions in rows in tables, and in no other way.  A correct design would be to have two tables along these lines:

Players
....PlayerID
....FirstName
....Lastname
....etc

Sports
.....SportID
.....Sport

In each case the primary key ID column can be an autonumber

What you have is a many-to-many relationship type between Players and Sports, i.e. each player can play one or more sports and each sport can be played by many players.  This is modelled  by a third table:

SportPlayers
.....SportID
.....PlayerID

In this case SportID and PlayerID are foreign keys referencing the primary keys of Sports and Players, so are straightforward long integer number data types, not autonumbers.  The primary key of this table is in fact a composite one made up of the two columns, which must of course be unique in combination.  The table can have other columns which represent any other attributes of the relationship between a player and sport, e.g. the level at which the player plays the sport.

With this correctly normalized model you can of course add as many sports as you wish simply by adding new rows in the Sports table.  To count the number playing each sport is just a question of a simple query which groups by sport and counts the rows for each:

SELECT Sport, COUNT(*) As NumberOfPlayers
FROM Sports INNER JOIN SportPlayers
ON Sports.SportID = SportPlayers.SportID
GROUP BY Sport;

For data entry an appropriate set up would be a form, in single form view, based on the Players table, and within it, a subform, in continuous forms view, based on the SportPlayers table, the parent form and subform being linked on PlayerID.  The subform would have a combo box set up as follows:

ControlSource:   SportID

RowSource:     SELECT SportD, Sport FROM Sports ORDER BY Sport;

BoundColumn:   1
ColumnCount:   2
ColumnWidths:  0cm;8cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the first dimension is zero to hide the first column and that the second is at least as wide as the combo box.

You can then select as many sports for a player as you wish while at a player record in the parent form, simply by inserting new rows into the subform, selecting a different sport from the combo box's drop down list in each case.  If you've included other columns in the SportPlayers table, such as the level at which the spot is played by the player, you'd include other controls in the subform bound to those columns.
Ken Sheridan, Stafford, England

maashoff say:

Excel is not adding up sum correctly

I have a spreadsheet / database that has daily cash payment details. When payments are netted there is a common reference for the netted payments. When I filter and/or do a pivot table to sum the netted payments I am getting false results. Each payment is whole value to the cents. Meaning.01 or.02 not .0299999. Optically when you expand the number its all zeros but the sum is giving me a result of .999999999? How is this possible.  I have expanded the numbers out to the maximum that excel allows and all digits aver the tens are 0

joeu2004 say: Maashoff wrote:
Thanks Ron I understand. However what we have here are flat  rounded numbers from a PeopleSoft query. They are not formulated. So using your example what I am doing is effectively adding .34 + .34 which should be = .68.

I understand your incredularity.  It is not like 1/3 at all.  We expect 1/3 not to be exact to 2 decimal places.  But we expect 0.34 to be.

And that's the mistake.  In fact, most decimal non-integers cannot be expressed exactly in binary, which is the sum of a finite number of powers of two.  For example, 0.34 is actually represented internally as 0.340000000000000,024424906541753443889319896697998046875.

(I use period for a decimal point and comma to demarcate the first 15 significant digits, which is all that Excel formats arbitrarily.)

Moreover, you are not simply adding 0.34 + 0.34.  You are adding numbers like 100.34 + 1,000,000.34.  The point here is:  the approximation of x.34 differs because of the magnitude of the integer part, which "steals" away some of those powers of two that otherwise would represent 0.34.  Specifically:

100.34:                  100.340000000000,003410605131648480892181396484375

1,000,000.34:  1000000.33999999,9967403709888458251953125

You can see that that does not sum to 1,000,100.68 exactly.  So, for example, =SUM(100.34,1000000.34,-1000100) is about 0.679999999934807, not 0.68.

Maashoff wrote:

However when I do a pivot table and / or sum them and expand the decimals it becomes. .666666666.

Usually, I recommend explicitly rounding when you expect the result to be accurate to some number of decimal places.  For example, =ROUND(SUM(100.34,1000000.34,-1000100),2) does return the same binary representation of 0.68, which is 0.680000000000000,04884981308350688777863979339599609375.

However, I'm not sure if and how you have that much control when using a pivot table.  I believe you would use a "calculated field".  But I am not a pivot table person.

Alternatively, you might consider setting the option "Precision as displayed".  Click on File, Options, Advanced, and page down to the option.

However, it is important that you save a copy of the Excel file before you try setting PAD.

The reason is:  PAD can change constants irreversibly.

A typical example is:  interest and tax rates that you might display with only 2 percentage decimal places, but which you enter with more decimal places.  In that case, PAD would round the rates to 2 percentage decimal places; and that might change other calculations.  More importantly, if you unset the PAD option later, the rates (as constants) retain only 2 percentage decimal places.

Also PAD applies to the entire workbook, not just selected cells or even the active worksheet.  So it can change the value of some formulas unexpectedly.  You should review every calculated cell in every worksheet.  (Sigh.)

Those are two of several reasons that I usually deprecate the use of PAD.

But if you are only doing simple arithmetic wth only dollar-and-cents amounts throughout the entire workbook (not just the pivot table), that might be an easy option for you at this point.

(But it is oh-so very dangerous, IMHO.)
