don cuthbert say:

If you use a Command Prompt and diskpart to assign a drive letter to the hidden partition, you should find hidden folders and a winre.wim file, plus a few small files. DISM /Get-WimInfo /WimFile:winre.wim /Index:1 should show the build number of the winre.wim contents (if run from that subfolder).

Winre.wim also exists in the first Recovery partition on the disk and checking that file will show its build number. During the initial development of Windows 10, I used Insider builds as my only OS on the same PC I have now. Due to the series of build upgrades, one after the other, I developed several recovery partitions at the end of my C: drive. I think the most I had was 3, plus the original one from the beginning of the hard disk. Basically, with each build it shrinks C: and adds a new Recovery partition.

So the extra Recovery partition winre.wim usually matches either the installed build or the previous one. Windows. Winre.wim is mounted and runs on a RAM disk (shows up as drive X:).

The extra partition provides the means to roll back and have Windows RE also revert to the matching one for the previous build that you upgraded from.

I have noticed some variance in whether the extra recovery partition gets created or not, but I have never tried to track down any reasons for it. I thought that maybe it was due to not doing so many sequential build upgrades (on my part).

_________________________________________________

Your UEFI setup does not look standard UEFI/GPT. So although it may be UEFI-capable, you must be running in a Legacy/compatibility mode to be able to boot from an Active partition and not have an EFI (FAT32-formatted) System partition.

There is a new utility called MBR2GPT.EXE which could convert from MBR to GPT disk initialization, but you would need to switch the UEFI firmware to full UEFI mode to boot from the hard disk after doing that.

https://docs.microsoft.com/en-us/windows/deployment/mbr-to-gpt

The 16MB MSR partition does not show up in Disk Management, but is required on a GPT-initialized disk. The layout below shows standard GPT disk and the Recovery partition contents. I suspect that your 817MB partition has the same contents, but different file size for winre.wim.

don cuthbert say:

I could add that from other reports here in the forums, a much larger Recovery partition is sometimes added. I also find that after an upgrade, two copies of winre.wim are preserved under:

C:\Windows.old\$WINDOWS.~BT\Sources\WinRE\winre.wim

C:\Windows.old\$WINDOWS.~BT\Sources\SafeOS\winre.wim

On my upgraded build 16232, both of those files are identical and are for build 16232 (as is the copy in the Recovery partition - total of 3 copies of it on disk). I don't see a copy from the prior build, but maybe that got deleted by running Disk Cleanup, as I had trouble upgrading and ran Disk Cleanup with every option selected, in order to get the upgrade to succeed.

EDIT: I think I had one of the large winre.wim files once, but found that it had the same contents as a smaller one, but could be compressed using DISM. The scenario where a new Recovery partition is created is if it will not fit on the existing partition of 450-500MB.
