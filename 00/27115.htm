ken sheridan say: First, to get the age of the member, add the following function to a standard module in the database:

Public Function GetAge(varDoB As Variant, Optional varAgeAt As Variant) As Variant

    If IsMissing(varAgeAt) Then varAgeAt = VBA.Date
    
    GetAge = DateDiff("yyyy", varDoB, varAgeAt) - _
        IIf(Format(varAgeAt, "mmdd") < Format(varDoB, "mmdd"), 1, 0)

End Function

Then try this to get the counts per age group:

SELECT SUM(IIF(GETAGE(MemberAddInfo.BirthDate, DATESERIAL(YEAR(DATE()),9,1)<25,1,0)) AS Under25Count,
SUM(IIF(GETAGE(MemberAddInfo.BirthDate, DATESERIAL(YEAR(DATE()),9,1)>=25,1,0)) AS 25OrOverCount
FROM MemberInfo INNER JOIN MemberAddInfo
ON MemberInfo.Number = MemberAddInfo.Number
WHERE Active = TRUE AND JoinDate <= DATESERIAL(YEAR(DATE()),9,1)
AND (LeaveDate IS NULL OR LeaveDate > DATESERIAL(YEAR(DATE()),9,1));

Obviously I can't test this, so cannot rule out the need for some debugging of the query.  I would point out, though, that a LEFT OUTER JOIN in this context would be pointless as two tables only need be included in the query if one or more of the columns on which criteria are being applied in the WHERE clause are in the MemberAddInfo table.  Applying a criterion (other than testing for NULL) on a column in a table on the right side of a LEFT OUTER JOIN in effect makes it an INNER JOIN.

However, I am concerned by your statement 'I have a ton of information about each member so I've split into multiple tables to make it easier to control.'  That is not a valid reason for decomposing a table, and strongly suggests that the tables are not normalized.  Decomposition of a table is undertaken to eliminate redundancies arrising form nontrivial functional dependencies, multi-valued dependencies or join dependencies, so that each table is normalized to at least Fifth Normal Form.  Decomposition into a set of tables in a one-to-one relationship type is valid where a type and subtypes are being modelled, but not merely because the number of columns is excessive.

Without knowing the attributes represented by the columns in each of you tables I cannot identify the redundancies, and consequently would be required to achieve a set of normalized tables.  All I can say is that whenever I encounter a design where tables are being used like this in a manner more akin to a spreadsheet, normalization by decomposition is invariably required.

Not repeating the criteria is an easy thing to handle. Instead of the cumbersome way you are doing it using global variables, simply use a form. This is a standard technique. Create an unbound form with unbound textboxes for the 3 date values. Then use the following syntax for the criteria:

=Forms!formname!controlname

where formname is the name of the form and controlname the name of the textbox for that the criteria value. When the query is run, it will look to the form for the value and not prompt you for it. This has added advantages in that you can verify the values that are entered. And, in other circumstances, use different types of controls to enter the criteria.

As for copying the database, if you are afraid of accidentally overwriting data then its pretty clear that your database has design flaws. This would not be a problem in a properly designed database. 

Finally, as to Ken 's point. It IS a valid technique to use tables in a 1:1 relationship where the data in the additional tables is not applicable to ALL the records in the master table. However, the other issues you are encountering do indicate that you really may need to revisit your design. Posting your table structures might help us help you more.

ken sheridan say: You can concatenate the literal strings in to the expression:

="Bank " & [Bank] & (Chr(13) & Chr(10) & "A/c Name " & [A/cName] & Chr(13) & Chr(10) & "A/c Number " & [A/cNo] & Chr(13) & Chr(10) & "Sort Code " & [SortCode] & Chr(13) & Chr(10)

To start Excel in Safe Mode -

Hold CTRL key and click on Excel icon
OR

File > Options > Trust Center > Trust Center Settings > File Block Settings > Uncheck all options
