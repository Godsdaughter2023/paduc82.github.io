Sub InsertEndnotenow()

bob jones aka: cybertaz say:

As for the summing accuracy: The key factor is that values you type add up correctly. That suggests that at least some of what you have in the other cells actually are Text rather than Numbers (numerical values). Copying from one location to another will not resolve the problem -- it simply duplicates the same text -- nor will simply reformatting the cells.

  1. Enter the digit 1 into any unused cell, then copy that cell
  2. Select the cells in column A
  3. Go to Edit> Paste Special
  4. Click the Multiply option in the Operation group
  5. Click OK

You should find that the column will then sum correctly. If so, repeat the process for the other ranges.

The printing issue is completely unrelated. The first steps to take are;

  • Confirm that your installation of OS X is fully updated
  • Confirm that your installation of Office 2011 is fully updated [currently 14.4.5]
  • Confirm that your printer driver is current -- check the mfr's web site to be sure
  • Run Disk Utility to repair disk permissions & restart the Mac after applying any updates.

If those steps do not resolve the issue, submit a separate question. Provide a complete & accurate description of that issue, including information regarding the origin of the file(s) involved. Also, be sure to describe what does happen when you attempt to print the file.

Keep in mind that reinstalling is seldom a recommended solution on a Mac & never is the first step to take when a problem needs to be addressed. It isn't the software that "goes bad", it's the support files generated after installing which have a tendency to become damaged or corrupt. Reinstalling does absolutely nothing to replace or repair those files unless a proper & complete Removal of Office is done first. That can lead to other, unnecessary complications. Further, reinstalling does nothing, whatsoever, to correct problems with individual user document files.

Hi Simon,

You need to think in terms of a relational database. You are still thinking in terms of a spreadsheet.

In a relational database a maintenance event is a separate entity so should be recorded in a separate table. Essentially you wouldn't need any field in the log to identify the maintenance associated with a log entry since you can use the date of the maintenance.

If you really want to include a field in your log book, you can use the Before Insert event of the form to pull the Maintenance ID value of the of the next scheduled maintenance.

Somehow after an update, my server setting went from imap to POP. Now I am unable to change them back to imap. The drop down is not highlighted. How can I do this without having to recreate/add a new account. I tried that option but when it created my account, it now put it into a folder ending in "ost". Something is not making sense with the description. A POP account doesn't use an .ost file and have never heard of an update that changes the protocol of an email account since they would use different server settings but that aside,

You can't simply change an account from POP to IMAP (or vice versa) - each type requires different settings. If you want to configure your account as a POP account - don't use <auto-discover> (only entering your email address/password) but rather manually configure your email account by checking the "Manual setup or additional server types" option at the bottom of the window.

Getting back to your original description "..........I tried that option but when it created my account, it now put it into a folder ending in "ost"" - A POP account uses a PST file and an IMAP account uses an .ost file so it did exactly what it's supposed to do. HOWEVER, whenever you configure an IMAP account, it creates a new .ost file and then syncs the email data from the server. Any data that you had in folders such as contacts/calendar which include the description "This Computer Only" will be lost if you haven't backed up that info by exporting it to a PST (or .csv) file
