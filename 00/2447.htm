ss it say:

Outlook 2007: "There was a problem reading your reminders..."

Hello, thanks ahead of time for any helpful input.

I have an Exchange user who's been experiencing this error for several weeks now. When she opens Outlook 2007, she gets the bubble "There was a problem reading one or more of your reminders. Some reminders may not appear."

We are running Exchange 2010 on Server 2008.


Any help would be appreciated. Thank you!

ss it say:

I'm posting the solution here so if anyone else has this issue, they can retain their sanity. This is somewhat related to the solution of removing an extra PST in Outlook. Having two PSTs can cause this error for some users. When they remove the extra PST, their reminders work properly.
1) Go into Exchange and find the user who's having problems.
2) Right-click and select Mange Full Permissions.
3) My user had her own name listed in the permission settings. Generally, this is not necessary because the other permissions should be set to allow access. This is a roundabout way of assigning permissions for the user.

This is where the extra PST comes in. In Exchange 2010, when a user adds Full Permission on a mailbox, it is automatically assigned to whomever the permission was added. For example, if I add Full Permission for myself on Joe Smith's mailbox, his mailbox will appear in my folder list automatically. You don't need to add the box manually, it is there when you restart Outlook. The downside of this is when you have many user permissions assigned to you. Your mailbox will be populated with all of their accounts. You can't manually remove these mailboxes from your Outlook unless you remove your name from the user's Full Permission.

Similarly, when a user adds their own name to their Full Permission, their own mailbox will appear again (twice) in their folder list. If you remove the user from their own Full Permission, it will disappear from the folder list. I believe this is related to the 'extra PST' issue. Since their own account is duplicated, it is functioning similar in error.

Depending on how your mailboxes were set up, you could receive an error after removing a user from their own Full Permission. When you open Outlook after removing this permission, you may see "Cannot Open your default e-mail folders. You do not have permission to log on." This means that the user doesn't have their own permission to access their own mailbox. This setting is generally automatically set when the mailbox is set up. If it's not, you must add 'NT Authority\SELF' to the Full Permission.

To recap.:
Remove user from their own Full Permission in Exchange.
Add NT AUTHORITY\SELF if it is not listed.

Note: You may need to kill the profile and create a new one.

hansv mvp say:

Clear the Default Value property. Instead, populate the control in the On Load event of the form:

 

Private Sub Form_Load()
    Me.txtDate = Date
End Sub

 

where txtDate is the name of the unbound text box. The date will be entered automatically when the form is opened, but if the user clears it, the text box will remain blank.

ken sheridan say: The 'proper' solution is the one you have identified, i.e. assigning the value to the control in code.  While the DefaultValue property emphasis on its use in this context.

Setting the DefaultValue property of  a bound control in a form, whether as part of the form definition or in code, has the significant advantage that it does not Dirty the form, so a user can back out simply by closing the form or navigating to an existing record if they decide to abort the insertion of a new row before entering any data.  In the case of an unbound control setting its DefaultValue property has no advantage however, as, unlike when applied to a bound control, the value has no relation to the context, so a value can simply be applied to the control, i.e. to its Value property, this being the default.

You may find this reply 'unhelpful' but, like Hans, I am simply asserting that the solution you identified is the more appropriate one.
