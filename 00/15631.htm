You probably see about 2.08E+12 for 2083421907530.  That is not a formula.  It is simply how Excel represents certain large numbers in a cell whose format is General.

As long as such numbers are 15 digits or less, Excel will input them correctly (exactly).  However, they will be numeric, not text.

Since the third column (after two commas) contains a mix of text (206T025193091) and numeric (2083421907530), it would be prudent to format that column (or at least that range of cells) as Text.

However, when you open or download the CSV file directly in Excel, you have no control over those formats.  Although you can fix it post facto, it might be easier to control it when you input the data.

There are two ways.  If you can download the file to disk first, change the extension from ".csv" to ".txt".  Alternatively, use the Import External Data wizard to input the data from the file.  Either way, the wizard will provide a way to set the type of that column of data to Text instead of General.

However, that will require that you treat corresponding data in the first parameter of your VLOOKUP similarly.  In other words, you should enter identifiers of the form 2083421907530 as text, not numeric.

You can do that one of three ways, depending on circumstances.  If those identifiers are in cells, either prefix the data entry with an apostrophe (aka single-quote) -- for example, '2083421907530 -- or format the cell as Text

joeu2004 say: Crazed wrote:

Why won't my vlookup recognize these values as a match?


Another thought....  If you are using VLOOKUP with only 3 parameters or with TRUE (or non-zero) in the 4th parameter, and you lookup table is composed of mixed text and numeric data as you posted (not in any particular order) [1], you are violating the requirement that the lookup column is in ascending order in the special collating manner described in the VLOOKUP help page.

Essentially, all text data must appear in ascending order following all numeric data, which also must be in ascending order.  Excel Sort will order data according to those collating rules.

Change or add the 4th VLOOKUP parameter as FALSE (or zero).

Caveat:  Although that might allow the VLOOKUP to work as-is, it might slow things down considerably, depending on the amount of data in the lookup table and the number of VLOOKUP expressions that you have.  Ultimately, it would be better to sort the lookup table according to the first column, and either omit the 4th VLOOKUP parameter or set it to TRUE (or non-zero).

Explanation

http://answers.microsoft.com/en-us/windows/forum/windows_8-performance/blurry-text-in-windows-8/ed053d17-b525-4e09-bdbc-de91114d975c

The cause for Office, Modern UI and IE10 to look so bad is that they use a new graphics rendering API offered in Windows 8 (and with updates on Windows 7, too). 

The new font rendering engine offered by this new API simply doesn’t have Clear Type implemented. So unless Microsoft patches this new API to support Clear Type, no program using this API will ever be able to do so.

The reason why Firefox (and Chrome, to an extend) are able to use Clear Type is because they still rely on the old Windows font-rendering API.

This is a bad decision by Microsoft, but their reasoning behind that (from what I think) is also quite clear: It is based on the assumption that their future will be in the tablet market. Since ClearType only works in one direction (horizontally, that is), it is totally useless if you rotate the tablet. The other thing is animation: if you animate a Clear Type anti-aliased font, its boundaries will start to flicker. And since Modern UI is a lot about animations and transitions, this would offer a bad experience. 

Last thing is the much higher ppi (pixels per inch) offered by tablet screens (>200ppi) in respect to computer monitors (mostly <=120ppi), where grayscale-anti aliasing is totally sufficient for crisp fonts.

And the worst thing: the Surface has a display with only 148ppi! So fonts look crappy on their reference design tablet. 

The bad thing is: no one can help you with the font rendering problem. You’re on your own, as a consumer. For me that meant to downgrade back to Office 2010, ignoring Modern UI and all apps completely and ditching IE 10 (but hey, that’s a no-brainer, isn’t it?).

Performance becomes more dependent on CPU characteristics, like memory, number of cores, and on how many apps are open, and can strain the CPU resources.

On Windows 7, Office animations are disabled. On Windows 8, animations are enabled because of D2D WARP improvements found in Windows 8.
