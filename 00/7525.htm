The detection likely only happens occasionally since the method based on the detection name is due to "Behavior" rather than a simple signature detection that always denotes malware.

Since these types of detections occur as a result of generic behavior monitoring performed within various portions of the operating system or browser, they tend to be more susceptible to false positive triggers when some piece of code appears to be acting in a similar manner to the malware name specified.

In this specific case, the detection trigger was due to the internalbehavior: 6AC3F45D03A99DFCD6FE5899A66BCE4E item displayed in the threat details you provided, so I'd be looking closely at that component for things it might be doing that are similar to the actual malware actions.

I found this additional detail including a history of the evolution of this RAT (Remote Access Tool) on the following CheckPoint blog article.

https://blog.checkpoint.com/2016/02/24/adwind-malware-as-a-service-reincarnation/

Looking closely within that article, there are key items such as actions the malware takes, file names its various incarnations have used and how its distribution and function were accomplished that might provide clues as to why your own software can occasionally trigger a detection for similar behavior.
