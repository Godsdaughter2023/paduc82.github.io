ken sheridan say: Before looking at your problem let's look at a more general issue:

Using the LIKE operator and wildcards is not a good way of doing this as it will fail with a NULL parameter for rows where there is a NULL at the column position in question.  This because NULL, being the absence of a value, cannot be compared with anything, so NULL LIKE "**" evaluates to NULL, neither TRUE nor FALSE.  To get round this you test for OR <parameter>  IS NULL in the query.  If we take a simple example with three controls referenced as parameters, a query would be along these lines:

SELECT *
FROM [Contacts]
WHERE ([Company] = [Forms]![SEARCH]![company]
OR [Forms]![SEARCH]![company] IS NULL)
AND ([City] = [Forms]![SEARCH]![city]
OR [Forms]![SEARCH]![city] IS NULL)
AND ([State] = [Forms]![SEARCH]![state]
OR [Forms]![SEARCH]![state] IS NULL);

Note how each OR operation is parenthesized to force it to evaluate independently of the AND operations.

So if you entered a new row without a State value for instance, your method would not pick this row up for the reasons given above if the State parameter is NULL, i.e. NULL LIKE "**" is nether TRUE nor FALSE, but NULL.  With the above method, however, it would be returned because [Forms]![SEARCH]![state] IS NULL evaluates to TRUE where the parameter is NULL, so the row would be returned regardless of the value or lack of value in the column..

Turning to your current problem, you say you have entered a value 'Test' into each column in a new row in the table, so the potential problem of Nulls should not arise.  Are you absolutely sure that you have entered a value in every column for which you are referencing a control in the form as a parameter?  Are the 'search boxes' text boxes into which you are typing a value, or are they combo boxes which derive their lists from the table?  If the latter then you will need to requery each combo box when a new row is inserted in the subform.  Until this is done the new items won't show in the lists.  You can do this in the subform's AfterUpdate event procedure, which will cover not only new rows, but any edits to existing rows which change the values.

As regards the other actions you want to undertake once the relevant rows have been returned, this is a separate matter which should have no bearing on the current problem.

There is one other issue which should be mentioned, however.  You say you want to create a duplicate row for the same contact with different data in some columns if they move to another company.  This is not how it's done in a relational database as it introduces redundancy and the consequent possibility of update anomalies.  What you will need to do is decompose the table into two tables, one with columns for the contact's the name etc, i.e. those attributes of the contact which remain static however many companies they may move to.  The second table will include a ContactID foreign key column and columns for those attributes such as their email address which will change.  So if a contact has worked for three companies there would be only one row in the main Contacts table, but three rows in the related ContactCompanies table. You could base the subform on a query which joins the two tables, but better options would be (a) to have a subform in single form view based on Contacts, and within it a subform in continuous forms view based on Companies; or (b) to have two subforms in continuous forms view and to correlate the subforms.  You'll find a simple example of the latter set-up as CorreatedSubs.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Querying now becomes a little trickier, though, as to query by company for instance you are not querying the referenced table (Contacts) but the referencing table (ContactCompanies), but you need to restrict the contacts subform's recordset to those contacts at the company selected.  You'll find an example of how to do this in StudentLog.zip in my same SkyDrive Folder.  In this you'll see that the students form includes a button for filtering by course, which opens a dialogue form in which one or more courses can be selected.  This filters the students form to those students registered for any one of the selected courses.  In this case the form is filtered in code in the Confirm button's Click event procedure, not by restricting the form's recordset by means of a parameter.

My answer wasn't the best it could have been, as I merely recorded my experimentation and wandering around. suzanne s. barnhill say:

It is very mysterious that you would see this issue in one document and not another, especially given that they're based on the same template. The issue in this case might be the customization context: when you customize a menu or toolbar, you can save the changes in Normal.dot (the default), in the template attached to the document (if other than Normal.dot), or in the document itself. So it's just barely possible (though it seems very unlikely) that somehow the Picture command has been removed from the menu just in the given document.

To restore it, in the affected document, open Tools | Customize, select the Commands tab, and set "Save in" to the document. Then select the All Commands category, scroll down to the InsertPicture command, and drag it to the Insert menu (hover over the menu until it opens, then drop the command where you want it). This command is also listed in the Insert category as "From File..."

There is a slight complication here, however: the InsertPicture command just inserts a picture, whereas the Insert | Picture... menu item opens a submenu with Clip Art, From File..., From Scanner or Camera..., etc. To restore all of that, you'd have to choose New Menu, name it Picture, and then add all the other menu entries manually. This seems like a lot of work! If what you need is really just InsertPicture, then you can equally well add that to a toolbar instead of the Insert menu.

The best solution, however, if you haven't customized the Insert menu, is this: While the Customize dialog is open, right-click on the Insert menu item and select Reset.
