scottgem say:

Can you please help me to display the remaining time left in a form?

To do this involves the use of a Timer. There are two elements of a Timer; the timer interval and the code behind it. When a timer interval is set on a form, the form executes the code in the On Timer event at the interval indicated. 

The problem here is that there is overhead involved with running code in the On Timer event. so I would be careful and keep the code very simple. But in your case I would do it differently. 

I would have a separate form to display the clock. A Timer Interval of 1000 will run every second The code is simple:

Me.Labelname.Caption = Format(Date() & TimeSerial(16,30,0) -Now(),"hh:nn:ss")

Use a label to display the time. 

On your actual form, I would use the Before Update event to run code like:

If Timevalue(Now()) > TimeSerial(16,30,0) Then

     MsgBox "it is too late to edit"

     Me.Undo

     Me.AllowEdits = False

End If

So this way you main form is not running the timer event but if someone tries to edit past the deadline, it will be prevented.

To display the time in hh:nn:ss until the deadline first add the following function to a standard module the database:

Public Function TimeDuration(dtmFrom As Date, dtmTo As Date, _
            Optional blnShowdays As Boolean = False) As String
            
    ' Returns duration between two date/time values
    ' in format hh:nn:ss, or d:hh:nn:ss if optional
    ' blnShowDays argument is True.
    
    ' If 'time values' only passed into function and
    ' 'from' time is later than or equal to 'to' time, assumed that
    ' this relates to a 'shift' spanning midnight and one day
    ' is therefore subtracted from 'from' time

    Dim dtmTime As Date
    Dim lngDays As Long
    Dim strDays As String
    Dim strHours As String
    
    ' subtract one day from 'from' time if later than or same as 'to' time
    If dtmTo <= dtmFrom Then
        If Int(dtmFrom) + Int(dtmTo) = 0 Then
            dtmFrom = dtmFrom - 1
        End If
    End If
    
    ' get duration as date time data type
    dtmTime = dtmTo - dtmFrom
    
    ' get whole days
    lngDays = Int(dtmTime)
    strDays = CStr(lngDays)
    ' get hours
    strHours = Format(dtmTime, "hh")
    
    If blnShowdays Then
        TimeDuration = lngDays & ":" & strHours & Format(dtmTime, ":nn:ss")
    Else
        TimeDuration = Format((Val(strDays) * 24) + Val(strHours), "00") & _
            Format(dtmTime, ":nn:ss")
    End If
    
End Function

In the form set the TimerInterval property to 1000, add a text box, txtTimer, to the form, and put the following code in the form's Timer event procedure;

    If Now() < TransactionDate + #16:30:00# Then
        Me.txtTimer = TimeDuration(Now(), TransactionDate  + #16:30:00#) & _
            " until deadline."
    Else
        Me.txtTimer = "Deadline has expired"
    End If

ken sheridan say: The column you have in the table is in essence equivalent to my TransactionDate column, but its DefaultValue property should be Date() not Now().  You don't want the time of day when the row was inserted, only the date,  though the value is really midnight at the start of the day as there is no such thing as a date value or a time value in Access, only a date/time value, which is implemented as a 64 bit floating point number.

In the scenario you have described there is no real need to record the time when the user inserts the row, only to validate that it is within the parameters.  If you do wish to record the time in another column however, do not do so by means of the column's DefaultValue property.  Contrary to what is commonly thought, that does not insert the time when a row is inserted by a user, but the time when the user navigates to an empty new row in a form or, in a badly designed database, in a datasheet.  The difference could be substantial.

To insert the time when a user begins to enter data in a row via a form the current date/time value should be assigned to the column in the form's BeforeInsert event procedure.  To do so when the row is saved to the table the AfterInsert event procedure should be used.  In either case the code is the same, e.g.

    Me.TimeStamp = TimeValue(Now())

If inserting rows by other means, e.g. an 'append' query, then the query should insert the return value of the TimeValue(Now()) expression into the column.  Note that this inserts the time only (or strictly speaking the time on 30 December 1899, which is day-zero in Access's date/time implementation).  The date/time when the row is inserted is returned by adding the values of the two columns: TransactionDate + TimeStamp.  An alternative of course would be to have a single column DateTimeStamp and return the transaction date with:  DateValue(DateTimeStamp). You'll often see it said you should always have only one column in this way, and you can do that if you prefer, but in the database relational model there is no cogent reason in support of that view unless you have a situation where you have a single date and two times representing the start and end of a range, e.g. a shift worked, and the range can be 24 hours or more in duration.

Theoretically the same problem could arise when inserting the date only of course, but that would require the user to navigate to a new row before midnight, but insert the data after midnight.  In most working situations this is unlikely so the use of the DefaultValue property should be reliable.

The reason for setting the ValidationRule property of the TransactionDate column is that it guarantees only values with a zero time of day element will be entered.  You might wonder why, as the data is entered automatically via the DefaultValue property, this is necessary.  In the normal day to day operation of the database it won't be of course, but there can be no absolute guarantee that the value might be amended by some other means.  If you think this is unlikely, Murphy's Law states otherwise, viz. "If something can go wrong, it will go wrong".

3.       Type “C:\Program Files (x86)\Microsoft Office\Office14\MLCFG32.CPL” (without the quotes). It is worth noting that I had to try both Program Files and Program Files (x86) to find the MLCFG32.CPL file.

4.       In the resulting Mail control panel applet, look to see where your PST file is located.

5.       Reboot to leave Safe Mode

6.       Make a copy of the corrupted PST file, this is critical if you are using POP3 to get your mail. The PST locates your mail on your machine. If you delete this file you’ll lose all your mail.

7.       Delete the corrupted PST file.

8.       Start Outlook and when prompted, create a new profile. If you are using IMAP your account information should be there and your mail should start working again. If you are using POP you’ll have to import mail from the old profile that you copied in step 6.
