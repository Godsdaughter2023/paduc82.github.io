ken sheridan say:
I wish Microsoft would fix that control so that it only is activated when you first put focus on the date field at issue.

The date picker only enters a value into the control, it doesn't update it.  You can force it to do so by moving focus to the next control in the date control's Change event procedure:

   Me.[YourN/ACheckBox].SetFocus

However, this will also attempt to move focus off the control as soon as the user enters or changes a character in the control if the user edits the date manually.  This can cause an error if the value is at that stage an invalid date.  This can be trapped:

   On Error Resume Next
   Me.[YourN/ACheckBox].SetFocus

but as soon as the value becomes a valid date focus will move to the next control.  This does not require a full date to have been entered, as a date defaults to the current year once a day and month have been entered.  This could become very annoying to a user trying to edit a data.

Even with focus on the date column you can of course tab to the next control in the tab order, in which case the data picker will disappear.

You could add the DConcat function from http://www.eileenslounge.com/viewtopic.php?f=29&t=6361&p=48878#p48878 to a module.

Create a query based on the table, and turn on the Totals button (Σ).

ken sheridan say:

Our Records Management System outputs records in an annoying way, creating multiple rows for what should be one record. I am trying to combine similar records so that they are all in one row.


Provided that the Incident Number and Date columns are a composite foreign key referencing the composite primary key of an Incidents table your data is stored correctly as, for a table to be normalized to First Normal Form (1NF) there must be only one value of each attribute (officer in this case) in each row.  You would encounter big problems querying the database otherwise.  You can return a result table which concatenates the officers into a single row, however, with a query which calls a suitable VBA concatenation function.  Hans has given a link to one and you'll find another in Concat.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file includes a function which uses the highly efficient GetString method of the ADO recordset object to concatenate values in to a single column, using data from Northwind as an example.

Another approach is to output the data as a report, with one row per incident, and which includes a multi-column subreport to output the officers in a horizontal format. From a presentational point of view I favour this approach as it aligns the columns vertically.  The ClubMembership demo in my same OneDrive folder is an example.  The rptAddresses_Sub report is the one to look at, ignore the rptAddresses_Code report other than as a technical curiosity.  I only produced it in response to a question by a reader of a databases column written by a contact of mine in a magazine.  For some reason the reader specifically wanted a code solution.  A code-free report/subreport solution is far simpler.

BTW 'Date' is a bad name for a column.  As the name of a built in function it is a 'reserved word', so should be avoided as an object name.  IncidentDate or similar should be used.
