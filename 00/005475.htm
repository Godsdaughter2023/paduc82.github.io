don cuthbert say:

Normally, you can only assign a letter to the Recovery partition via diskpart.  Even if you do, Disk Management will not see it as having a drive letter.  In File Explorer the folders, etc. are protected operating system files, so usually hidden without even browse rights to Administrators via File Explorer.  If you do assign it a drive letter you can most easily see what is there via a Command Prompt if you look for hidden files.

The file at \Recovery\WindowsRE\ReAgent.xml on each Recovery partition contains the info on which build the Recovery partition contents came from.

don cuthbert say:

Is there a chance that the recovery partition at the end of your disk has been there since prior to the public Windows 10 release?  The contents of the ReAgent.xml file in that last partition should reveal which build it came from.  Also, since it is right beside your C: drive partition, you could reclaim that disk space by extending C: if you delete it.

I have done multiple upgrades on my main Insider machine and it still only has the one Recovery partition at the beginning of the disk, but it was clean installed (after first wiping all partitions from the disk) after the November public release.  Default Windows 10 partitioning is different now than it was prior to the first RTM last July.

UEFI computers, with a GPT disk layout do not have an Active partition like Legacy MBR-initialized disks require.  The 16MB partition is the MSR (Microsoft Reserved) partition.  It used to be larger, but not since the Windows 10 RTM.

debugini say:

I had this problem. Found two solutions: first one solves recording for new presentations, the second one is a way to fix a slo-mo recording you already have.

1. The problem's root cause is an unsigned 16 bit overflow. It is caused by a sample rate setting of 96000 Hz of the internal microphone, leading to a sample rate of 30464 (which is 96000-65536) to be written into the wav files, where the real sample rate is 96000. Use "Audio-MIDI-Setup" (from Utilities) to set the sample rate back to 44100 Hz at 16 bits. This solved the problem for me and is a good explanation, why the slo-mo effect does not occur with an external mic connected (as reported by ProfPete)

2. to fix a presentation you already have recorded:

a. rename from .pptx to .zip, then double-click the file to unzip it.

b. navigate to the ppt/media subfolder

c. change the speed of each audio file (.wav) by factor 3:

I did this in a batch using sox (http://sourceforge.net/projects/sox/) from terminal because I had 30 wav files, but you could also use Audacity to do it manually (note you have to enter a factor of 200% in Audacity to get a 3-fold speed increase). With sox, the commands in terminal are as follows. Replace <path to unzipped files> with the actual path on your machine:

cd <path to unzipped files>/ppt/media

for i in *.wav; do mv $i $i-slomo; sox -r 96000 $i-slomo $i; done

rm *.wav-slomo

cd ../..

zip -r ../rezipped1.pptx *

d. open rezipped.pptx in PowerPoint. It will prompt you to repair the file. Click "Repair". 

e. done

Axel

If any of these documents were created with Word 2007 or later, they're most likely in the docx or docm formats, not the older doc format. If you examine the files with a Hex editor, any whose first two bytes are PK will have the docx or docm formats (most likely docx, as docm is for macro-enabled documents).

This shouldn't be a factor. If a document is created in docx format, and the extension is later changed to doc, then Word 2007 and all later versions will still open it -- the program recognizes the format and ignores the extension. It even works the same in Word 2003 with the Compatibility Pack.

It may still be worth examining a few of the documents with a hex editor. The first two bytes having ASCII values PK indicates a docx or docm format; the first four bytes having hex values of D0 CF 11 E0 indicates a doc format. Anything else indicates that the file has been damaged and isn't likely to give back any useful data.

jay freedman say:

It sounds like that section has been set to start on an even page.

Click on the page that should be 39. Then click the dialog launcher arrow in the bottom right corner of the Page Setup group on the Page Layout tab. In the dialog, click the Layout tab. If the "Section start" dropdown says "Even page", open the dropdown and select "Odd page".

By default, the Apply To box at the bottom of the dialog will say "This section". Since you want every section of the document to start on an odd page, you can change that to "Whole document". Then click OK.

Page numbers in headers and footers should change immediately. Page number fields in the body text (which are rarely used) will have to be updated by selecting the text and pressing F9, or by going to Print Preview.

john mcghie say:

Richard:

This is a pain and a chore, isn't it :-)  As Bob says, nothing short of Indesign will handle this properly.

I tend to specialise in very long documents that will have only a few of these "sideways pages" so it is a real chore to make them up.

So what I often do is rotate the TEXT, not the PAGE.  Put the text in a single table cell, with no borders so the reader cannot see it: then rotate the text so that when it prints, the text prints sideways.

This technique enables the headers and footers and page numbering to carry on as normal, and you can print and bind the result without having to do anything extra.

The only problem occurs if some of your sideways sections occupy more than one page.  If they do, then you have to use Bob's method of creating separate sections for them and fiddling to get the headers, footers and page numbers into the correct position for binding (I sometimes just take the headers and footers off the sideways pages: life is too short...)
