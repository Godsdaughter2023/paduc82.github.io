scottgem say:

First, you need to research the difference between a natural and a surrogate key. An Autonumber is a surrogate key. A natural key is a field or combination of fields that define a unique record. 

So, if you have a website table. I would definitely use an Autonumber key rather than the URL. This will allow you to create multiple records for the same URL that have different connections.

Another point you should not have BusinessName as a field I would call that table tblIncidents rather than tblBusiness. You should have a Businesses table and the only thing in the Incidents table would be the BusinessID. 

weissschnee say:

Thank you. That is what I was thinking... I was hoping to use the AutoNumber as a unique identifier for my websites. However, because I have an autonumber as my primary key, I am not able to have duplicate of that number. Which I need.

Also, I do have a separate tblIncidents. tblBusiness actually just lists the business information. (I didnt give all the details in my original post because I did not think it was necessary)

ken sheridan say: Every table must have a primary key, but it need not be a single column, and it need not be the only key in the table.  A table might have more than one 'candidate key', in which case one should be made the primary key.  This will be the one referenced by foreign keys in other tables.  The other candidate keys are each defined as such by being uniquely indexed.

Your fundamental problem here is that your Businesses table is not correctly normalized.  As there can be more than one incident per business, there should be no IncidentNum column in the table.  Instead each incident should be a row in a separate related Incidents table, of which IncidentNum would be the primary key, and which, if each incident relates to one business only, would contain a BusinessID foreign key referencing the BusinessID primary key column of Businesses.  The Businesses table now has only one row per business, so the redundancy which currently exists in the table is eliminated.  Consequently the Website column can be in this table, and assuming that more than one business does not share the same website, this is a candidate key and should be indexed uniquely.  Other attributes of the website can be represented by column  in the same table.  There is no need for a separate websites table, as this would be in a one-to-one relationship type with Businesses.  A one-to-one relationship type is used to model a type/sub-type, which is not the case here.

If, on the other hand each incident can relate to more than one business, then the Incidents table would not contain a BusinessID foreign key column.  The relationship type between businesses and incidents is, in this scenario, many-to-many.  A many-to-many relationship type is modelled by a table which resolves it into two one-to-many relationship types, so the table would be like this:

BusinessIncidents
….BusinessID (FK)
….IncidentNum (FK)

The primary key of this table is a composite one made up of the two columns.  Even if you give the table an autonumber surrogate key, the two columns are still a candidate key, and would have to be defined as such by their inclusion in a unique index.  This table might well gave other columns to represent attributes of the relationship type.

Editting a word template with quick parts - Microsoft Community

paul-latif say:

Editting a word template with quick parts

We currently use a word template with quick parts at work.  I am trying to edit the document but when I save it as a new template, the quick parts do not save.  It will not let me save over the current template.  Is there a way to edit the template and save it without the quick parts disappearing?
stefan blom say:
From within Word, display the Open dialog box (Ctrl+O), locate the template file, select it, and click Open.

Note that you should never modify a template by overwriting it with a document saved as a template. When you do that, you lose data that can't be saved to documents, such as Quick Parts (or more generally Building Blocks).

stefan blom say:

To edit a template, open it as a document (via the Open dialog box), make the desired changes, save and close it.
