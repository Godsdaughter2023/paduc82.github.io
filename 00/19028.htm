ken sheridan say:

The list I am looking up is text


Are you sure that is the case?  Normally a 'lookup field' will be a long integer number data type which references an autonumber primary key of a table which also has a non-key text column.   For example you might have a table Addresses which includes a column (field) for the city.  This would reference a table set up like this:

Cities
....CityID (autonumber PK)
....City  (Text)
....State (Text FK)

You cannot use the text column as the primary key in this case as city names can legitimately be duplicated, so a 'surrogate' numeric key is necessary.

The Addresses table must also use a numeric data type column for the foreign key to reference the primary key of Cities, so if the lookup wizard is used the display control of this column would be a combo box set up as follows:

RowSource:     SELECT CityID, City FROM Cities ORDER BY City;

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the last one.  The important thing is that the dimension is zero to hide the first column.  The combo box will show the city names, but its value will be the hidden numeric CityID value.

Of course, there is a problem with this example.  As I said, city names can be duplicated, so if the same city name appears more than once in the combo box's list there is no way to know which is the correct one to select.  This is just one of the many reasons why the lookup field wizard is deprecated by experienced Access developers.  You could amend the column's properties to overcome this, but as a user's interface with the data should only be via forms, never directly via a table's datasheet, this would serve no purpose; in fact the use of the lookup field wizard serves no purpose other than to cause the problems detailed at the site to which John gave you a link.  While you obviously have to follow your tutor's instructions in this case the correct approach would be use a combo box in a form bound to the table, setting it up as follows:

Name:    cboCity

ControlSource:  CityID ( or whatever the foreign key column in the Addresses table is called)

RowSource:     SELECT CityID, City, State FROM Cities ORDER BY City;

BoundColumn:   1
ColumnCount:    3
ColumnWidths:  0cm;3cm;2cm
ListWidth:          5cm

You'd need to experiment with the last two ColumnWidths dimensions to get the best fit.  The ListWidth dimension should be the sum of the ColumnWidths dimensions.  You can now select from a list which shows the state in which each city is located, so you know which one to select and don't end up selecting Newark, New Jersey when you wanted Newark, California.  You can take it a step further by adding an unbound text box to the form with a ControlSource property of:

=cboCity.Column(2)

The Column property is zero-based, so this will show third column's value, i.e. the State for the selected city.

I realize I've gone beyond the immediate substance of your question, but it's important (see the quote from the great Richard Feynman below my signature) that (a) you understand how a 'lookup field' works under the skin, and (b) that there is a way of achieving the same end result in a form without the unnecessary and potentially harmful baggage which comes with the use of the wizard.  You might like to draw these points to your tutor's attention.
