louisyan_msft say:

Hi Ziid,

The answers are listed below for your reference:

Q1: If we want to encrypt messages that are stored within a user’s mailbox (or a shared mailbox) is that something thats part of RMS?

A1: No. RMS does not help to encrypt messages that have already been stored within a user’s mailbox.

Q2: If I read about the RMS the message encryption there is only when sharing a message or attachment with other users, so one can ensure that the recipient are using a RMS enabled client and reads the message in a secure manner right?

A2: No. Message encryption in RMS does not require the recipient to enable RMS. However, if the recipient sends emails back to you, his email may not be encrypted.

Q3: Is there a way to have all or some of the messages encrypted within the mailbox?

A3: Yes. We can specify what kind of email needs to be encrypted. You can refer to the following article:

technet.microsoft.com/.../dn569289.aspx

Q4: All messages to and from Exchange Online (in transit) is encrypted by default without using any types of RMS? TLS?

A4: For RMS: No.

   For TLS: Office 365 will try to set up the SMTP connection with the recipient via TLS by default. But, if the recipient's side doesn’t support TLS, the SMTP connection will be unencrypted. In this kind of scenario, you can use Message Encryption as a workaround.

Q5: If the target environment does not allow TLS, how does this work? will it fall back to like clear text?

A5: Yes.

Moreover, here are some points about the concerns:

1. Office 365 Message Encryption isn't equal to TLS encryption. TLS encrypts the SMTP connection. Message Encryption encrypts the mail itself.

2. Message Encryption is just one of the functions supported by RMS. Preventing the email from being forwarded is another function supported by RMS.

If you have any questions when referring to the information above, feel free to post back.

Best regards,

Louis

Second, it would be good to know what exactly your customer means "contents encrypted in the mailbox".

In Office 365, email data at rest is encrypted using BitLocker Drive Encryption. BitLocker encrypts the hard drives on a computer to provide enhanced protection against data theft or exposure on computers and removable drives that are lost or stolen, as well as more secure data deletion when BitLocker-protected computers are decommissioned or recycled.

When you subscribe to Azure RMS, or get it if you subscribe to E3, you get the features of message encryption and also Rights Management (RMS).

Message Encryption is only applied to a message upon sending provided it is triggered by a rule eg If email subject contains Confidential then encrypt.

RMS is applied on files, and its applied via templates. For example you specify a Salary Protection template where only HR director can edit, and dept heads can view. Eg Apply the Salary Protection RMS Template on this June 2015 spreadsheet. This can be applied to data at rest (the file itself) or only when sending.

Your customer must bear in mind that in order to be more secure, you trade off convenience.

So for example, if you encrypt every message, that would mean the recipient will have to take additional steps before he can read a email message. Message in, log in to portal, key in credentials to confirm identity, view email. So in theory it is secure, but productivity wise will suffer.

RMS is similar. If you RMS  the file itself, everytime you want to edit it or view it, you will  key in your credentials before you can open the file. A lot of work if you are trying to edit it many times.

What is causing the fonts to change using the save as?

bob jones aka: cybertaz say:

I'm not sure why others are running you around like this... The simple fact is that Mac Office does not support font embedding in documents. Having you look for someplace to turn on font embedding is a waste of your time because no such setting exists. All the more, it isn't an Option/Preference issue in the first place :-)

Generating a PDF is a print function... Microsoft only stuck their PDF command in the Save As path for the purpose pf pandering to users who can't seem to understand that :-) When using the Save As method it invokes the Microsoft PDF generator, which provides 2 options: If you use the Best for electronic distribution and accessibility option fonts are not embedded in order to minimize file size. If you use the Best for printing option you'll get a different result and what could be a significantly larger PDF file.

Also, the File> Share method invokes the OS X PDF generator as does using File> Print - PDF. The common problem is that both of these methods embed all fonts used in the document in their entirety, rather than subsets, which can result in an extremely large PDF file. Most other programs [such as Pages] use the OS X PDF generator.
