Stopping on the sub name is indicative of an error occurring in a called sub but I cannot say that is always the case. You could try inserting debug.print and a string to identify the location in the code at various points and when the code stops you will have an indication of how far the code progressed.

Also the file it is looking for could be a system file used to complete some processing. You could have some processing where the system needs to catch up. Try inserting some DoEvents at critical points. (Before and after each sub call and between each line of the following code as per the example):

Set User = New clsSystem
DoEvents
User.GetSystemInfo Mid(Application.OperatingSystem, 10, 2)........
DoEvents
userColl.Add User, "System"

If you have any complex code in the called subs, you could also insert DoEvents there to ensure that external processes complete before attempting to process following lines.

There are two types of forms from the standpoint of interacting with data. Those types are bound and unbound. A bound form is one where the recordsource of the form is a table (sometimes a query). The controls are then bound by setting the Controlsource to a field in the Recordsource. With a bound form, Access handles all the I/O between the form and the table.

The unbound type means the Recordsource is left blank. This means that the developer needs to handle all the I/O between the form and table using code. While there are some good reasons to use unbound forms, unless you have one of the good reasons, you should use bound forms. Especially as a novice developer.

If you use a bound form, you can start off with a search form that selects the record you want to work with. You can use code like:

DoCmd.OpenForm "formname",,,"keyfield = " & me.keyfield

to open a form filtered for the selected record. You can then edit whatever you need to. If you don't want to edit certain fields once they have been entered, you can lock those controls on the form.
