ken sheridan say:

It's not merely a question of changing the controls used from combo boxes to subforms.  If a multi-valued field has been used, i.e. one where you can select more than one value in the combo box by checking them then this is a method (a  very bad one in the view of many experienced database developers) of representing a many-to-many relationship type between two entity types.  The conventional way of doing this is by a third table which models the many-to-many relationship type by converting it into two one-to-many relationship types.  You'll find a simple example as ParentActivities.zip in my public databases folder at:

https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

which shows how to represent this sort of model as a form/subform.  If you look at the relationships window in my demo you'll see how the model comprises three tables as follows:

Parents----<ParentActivities>----Activities

If a multi-valued Activities field had been used in the parents table, then the ParentActivities table to model the relationship type would not exist.  That would also be so in your case if multi-valued fields have been used, and to be able to use a subform rather than a combo box you'd need to first rebuild the model to include a table to model the relationship type.  This is not difficult, but does require some knowledge of building the necessary 'append' queries to insert rows into the additional table prior to deletion of the multi-valued fields from the current tables.

You can still bring your data together into a single table to export to Excel without rebuilding the model, but the resulting table does not need to be a base table, but the virtual result table of a query which joins the existing base tables.  You might find this can't be done in a single query, however, and it will be necessary to build separate queries to join subsets of the tables, and then join these queries until you arrive at a final all encompassing query.

scottgem say:
OK, lets explain comboboxes a bit. A combobox (sometimes called a drop down) allows a user to select from a list of values. There are two ways populate a combo. One is via a Value List, the other by a query against a table. If one uses a value list, then generally, the select value is stored in the table. If one uses a separate table, then generally a code  (called a Foreign Key) is stored in the table. 

So you need to look at the properties of the table first and see if these fields are lookup fields. If they are, then you need to see if they are populated with value lists or queries. If they are not lookup fields, then you need to look at the comboboxes on the data entry form. Again check the properties to see what the RowSource is. 

If the database uses lookup tables where a code is stored as a foreign key, you can then create a query that joins the main table to those lookup tables and get the description value rather than the code. This query, can then be exported to Excel.

GMAIL'S RECOMMENDED IMAP CLIENT SETTINGS

Sending:

Do NOT save sent messages on the server. If your client is sending mail
through Gmail's SMTP2 server, your sent messages will be automatically
copied to the [Gmail]/Sent Mail folder.

DO save draft messages on the server. If you want your drafts in your mail
client to sync correctly with Gmail's web interface, set your client to save
drafts to the [Gmail]/Drafts folder.

Deleting:

Do NOT save deleted messages on the server. Messages that are deleted from
an IMAP folder (except for those in [Gmail]/Spam or [Gmail]/Trash) only have
that label removed and still exist in All Mail. Hence, your client doesn't
need to store an extra copy of a deleted message.

Do NOT save deleted messages to your [Gmail]/Trash folder because this will
delete a message in all folders.

Do NOT save deleted messages to your [Gmail]/All Mail folder as some clients
will try to empty this folder and ultimately fail. This can lead to delayed
mail access or excessive battery consumption on a mobile device.

Junk mail and spam:

Do NOT enable your client's junk mail filters. Gmail's spam filters also
work in your IMAP client, and we recommend turning off any additional
anti-spam or junk mail filters within your client. Your client's filter will
attempt to download and classify all of your existing messages, which may
slow down your client until the process is complete.
