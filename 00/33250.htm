}firedog say:

A few general points to aid your understanding of what's happening:

  1. The folder structure in Windows Live Mail bears only a superficial resemblance to the folder structure in the store folder in the file system. This is not normally a problem; the message database index keeps track of which messages are in which folder and should ensure that they're displayed properly regardless of their actual location in the file system.
  2. The database uses folder names with a maximum of 15 characters. This is partly an attempt to restrict the length of the path to message files so as not to run afoul of Windows' limitations on path lengths. To achieve this, folders with longer names have them truncated to eleven characters followed by a space and three random characters.
  3. Each exported folder should contain a file called wlmail.fol. This is a simple text file giving the original name of the folder so it can be reproduced on import.
  4. The existence in the file system of folders with appended serial numbers indicates that there has at some stage been a corruption in the store folder that Windows Live Mail has recovered from. This will usually be transparent to the user, for the same reason as given in (1) above. It may nevertheless become a problem after export/import, meaning that messages have to be relocated. If this is the case, any such relocation should take place within the program and not in the file system, so that the database can keep track of what's going on.
  5. The store folder is hidden by default for a reason: any file or folder manipulation conducted there will be invisible to the database, with the result that the index will become confused, even to the point of not being usable.
  6. It's important to give the program peace and time to index the store folder after a major operation like importing from an exported store folder. This might be a long process - taking hours, for example - if the store folder is very large. It may be as well to disconnect from the Internet and allow the program to run overnight in cases like this.

mouneshwar r say:

Hi wolfman01,

Here are the steps that you need to follow in order to setup a secure wireless network using linksys router on Windows Vista. Firstly, you need to get to your router's settings.

1. Open up a web page and type: 192.168.1.1 into the address bar at the top. This is linksys default IP address for the router.

2. It will then ask for a username and password.
username: admin
password: admin

3. Create a WEP encryption.

4. Click on the Wireless tab at the top. Then click the Wireless Security sub-tab right underneath of that.
Make sure the 64 bit WEP encryption is selected. Next to the Key 1, create a ten digit password which will be the password used to get you on the network and keep others out.

5. Now click Save Settings at the bottom and you are all set!

debio2 say:

The issue is actually a lot more subtle than drivers.

As a poster noted above, this device has no vendor drivers. Rather, it uses hidclass.sys, hidparse.sys and hidusb.sys, which are provided by Microsoft to support the Human Interface Device standards. These drivers or their equivalents are still present in Windows 8.

The problem is actually this:


No drivers are loaded for the device because it never initializes properly on the USB bus. Its hardware ID is set as USB\DEVICE_DESCRIPTOR_FAILURE rather than an actual vendor and device ID.

I observed this problem on a laptop - the dance pads worked fine under Win7, then consistently failed to enumerate with this error under Win8, then started working again after I went back to Win7. Normally this particular error would indicate a device failure, but since it only occurs under Win8, we're probably looking at a USB controller driver bug.

Unfortunately, I don't know any troubleshooting steps to take after this because all the normal steps revolve around changing drivers or reinstalling drivers. Since the device never reaches a state where drivers can be loaded, I'm left twiddling my thumbs.

Any thoughts?
