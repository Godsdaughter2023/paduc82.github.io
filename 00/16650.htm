jcmu99 say:

Are header/footer settings supposed to supersede margin settings?

Just trying to understand a little bit of how margins and headers/footers interact. I have a document where my margins are set at 1 inch all the way around. But I set my footer at 1/2 inch and I noticed that my footer text goes beyond the 1 inch margin. So just trying to understand if that is normal behavior for the header/footer settings to supersede the margin settings. Thanks in advance.
doug robbins - mvp office apps & services (word) say:

The settings "From edge:" for the Header and Footer in the Headers and footers section of the Layout tab of the Page Setup dialog control the distance of the Header and the Footer from the top and bottom of the page respectively.  

They have no bearing on the width of the header and footer text, which will normally be constrained by the left and right margins as set on the Margins tab of the Page Setup dialog.  However, those settings can be superseded by formatting the paragraphs in the header and or footer to have negative indents before and after the text.

suzanne s. barnhill say:
Doug's answer addresses the width of your header or footer; as to the depth, that is determined by the amount of content. The default settings of 0.5" header and footer margins and 1" top and bottom margins allow for 0.5" of header/footer content without encroaching on the document body. If the header or footer contains more content, it will push the document body down/up accordingly. This can be used to advantage in a letterhead-type document (see http://wordfaqs.ssbarnhill.com/DifferentMargins.htm). For more on how headers and footers work, see http://wordfaqs.ssbarnhill.com/headerfooter.htm.

doug robbins - mvp office apps & services (word) say:

Dim i As Long, j As Long
With Selection.Tables(1)
    MsgBox "The selected table contains " & .Rows.Count & " rows and " & .Columns.Count & " columns."
    For i = 1 To .Rows.Count
        For j = 1 To .Columns.Count
            MsgBox "The dimensions of cell " & j & " in row " & i & " are " & .Cell(i, j).Height & " x " & .Cell(i, j).Width & "."
        Next j
    Next i
End With

scott mcdaniel say:

You can't do that with an AutoNumber field, nor should you. An AutoNumber field is not guaranteed to be sequential, and you cannot store a value in the format you desire (nor should you).

Instead, store the JobSequence and JobDate values separately, and combine them as needed to view on a form, report, etc:

Year(Me!JobDate) & "-" & Me!JobSequence

I'd still use an AutoNumber field to relate the table to any others, however. This would known as a "surrogate" key, since it would really have no "value" to the record, and would instead simply be a pointer to that record. You would still insure uniqueness through other indexes/fields.

I want to append each year's data to a combined table.


You should already be storing all Jobs in the same table and then querying it as needed to view whatever data is needed.

hansv mvp say:

In the 2014, 2015 etc. tables, the prefix is only a matter of formatting; the prefix is not part of the stored values in any way whatsoever. When you append records from these tables to the combined table, only the stored values are transferred, not the display values.

If you want 2014-1 etc. in the combined table, you must change the job number field from Number to Text, because values such as 2014-1 are not numeric.

In the append query that adds records from the 2014 table to the combined table, use

"2014-" & [JobNumber]

to append to the JobNumber field. For the 2015 table, use

"2015-" & [JobNumber]

etc.

Alternatively, leave JobNumber as a number field, and append the values "as is", so you'll get 1 from each of the tables.

Create a new number field JobYear in the combined table, and populate it with 2014 in the append query for the 2014 table, etc.

The combination of JobYear and JobNumber will be unique (this combination could be the primary key).

In a query based on the combined table, you can concatenate year and number:

JobID: [JobYear] & "-" & [JobNumber]

This will display 2014-1, 2015-1, etc.

ken sheridan say: Firstly, you are absolutely right to coalesce the tables into a single table.  By having multiple tables, one for each year you are doing what is known as 'encoding data as table names'. A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

The autonumber columns in your current tables should be amended to a column of long integer data type, indexed non-uniquely.  As you are using the year to format the values this suggests that you might have a column such as JobDate, of date/time data type, in the tables (see my final paragraph below if you don't) from which you are obtaining the value for the current year.  Consequently, when the data is appended to a single table you can do the same in the single table and format the values as at present.  You cannot use the combination of the year and serial number as the table's composite key, however, so you will need to introduce an autonumber column into the table as a surrogate primary key.  You could get round this by introducing a JobYear column into the table of course, but you should not do so as this would be transitively dependent on the key, so the table would not be normalized to Third Normal Form, and consequently wide open to the risk of update anomalies.

To compute the next number per year when inserting a row into the new table you can do so by means of code in the AfterUpdate event procedure of the control bound to the JobDate (or whatever) column, e.g.

    Dim strCriteria As String

    strCriteria = "Year(JobDate) = " & Year(Me.JobDate)
    Me.JobNumber = Nz(DMax("JobNumber", "Jobs", strCriteria),0)+1

However, in a multi-user environment this gives rise to the risk of conflicts where two or more users are inserting a row simultaneously.  This cannot be prevented by trapping a key or unique index violation as would normally be done (see CustomNumber.zip in my public databases folder at: https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169 for an example).  You would have do so by other means, the most effective being the application of a CHECK CONSTRAINT to the table with:

CurrentProject.Connection.Execute "ALTER TABLE Jobs ADD CONSTRAINT YearJobNumber
CHECK((SELECT JobNumber FROM Jobs GROUP BY YEAR(JobDate), JobNumber HAVING COUNT(*)> 1) IS NULL)"

 If necessary at any time the constraint can be dropped with:

CurrentProject.Connection.Execute "ALTER TABLE Jobs DROP CONSTRAINT YearJobNumber"

The error can then be handled in the same way as in my demo by trapping it in the IncrementField function in the same way that the key violation error is currently trapped and handled in my demo.

If you do not have a JobDate column or similar in the tables, then you can of course legitimately add a JobYear column to the single table.  You can easily populate it by including a constant such as 2014 or 2015 in each append query's SELECT clause.  This can then be part of the composite key, as in the NamesByGender column in my demo, removing the need for a CHECK CONSTRAINT, and the key violation error can be handled in exactly the same way.

scottgem say:

Let me summarize something here.

First an Autonumber has one and one purpose only, to provide a unique identifier to a record. It cannot be used be where there is the need for sequential numbering. While it can be used as a visual identifier either alone or with other data, it is meant to stand alone as a unique identifier. If you do need sequential numbering (and you should justify that you do), then I would suggest checking out my blog on Sequential Numbering for one way to accomplish it.

Second, using separate tables to categorize data is not good design. Data should be categorized by values in a record, not field or table names. If you want to identify a record as belonging to a year, then include a date field in the record.

Third, SQL is really a fairly easy to language to grasp. Plus, since you can generate your SQL using the Query Designer its pretty easy to pick up. But it has nothing to do with your question.

Finally, If you started out by creating table for each year, I have to wonder whether you have any other design issues with your database, If you haven't read up on normalization I suggest you do so beforse proceeding with your application.
