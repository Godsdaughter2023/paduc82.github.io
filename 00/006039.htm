Go to the Excel menu and choose About Excel to check your update level. If you're not at version 15.20 or 15.21 then head to Excel's Help menu and choose Check for Updates to get the updates. http://spreadsheetpage.com/index.php/dataform/home

After reading about the add-in, go to the download page. 

http://spreadsheetpage.com/index.php/dataform/download/

Click the Download link for version 3b (under the Excel 2007 heading). The download file will be named dataform3.zip.

In Finder, double-click the zip file and it will open a folder called dataform3. Go inside the folder and find dataform3.xlam. You can drag this file to a permanent location in which it can be used. I recommend making a folder called Add-ins in your Documents folder and drag dataform3.xlam into that folder.

Your Add-ins dialog should have the add-in listed and checked, like this:

Now you're ready to use the new add-in.

Select a cell in your data table. Then go to the Data tab of the Ribbon and you will see a new button off to the right labeled J-Walk Enhanced DataForm. Click this button to activate the J-Walk Enhanced Data Form.

scott mcdaniel say:

The link you included discusses Properties and Variables, not Functions. In your case, since you're using the Environ function, you could not really use a Variable.

In my view, a Property is part of a Class module. It would only be available when you create an instance of the class. 

A Function performs some sort of work and then returns a value. Assuming your function is in a standard code module, it would be available at all times.

In your case, it would seem a Function would be the preferred method. 


albert d. kallal say:

Both of your code examples will work just fine, and even work fine in a standard code module.

However, the property GET (and LET) are usually used in a class module, and not a standard module. While they both work, unless you are creating a class module, you should use the function.

In the case of a class module, you are free to use a public function, or GET, but in most cases a public function is considered a method of the class, and a GET/LET is considered a property of the class module.

Class modules are different in that you can (have to) create an instance of that module BEFORE you use it. This allows you to have multiple instances of that code module, and MORE importantly multiple copies of that code with different values in the given set of variables you declared for that code module.

Since your needs are global and simply a standard function call, then using GET would only confuse a developer since it would suggest the code they are reading is part of a class module. As noted you donâ€™t have to follow this convention, but using GET would strongly suggest to anyone reading the code that such code is part of a class module, and not a regular code module.

As for using class modules, I explain why one would want to use class modules in this article of mine:

http://www.kallal.ca/Articles/WhyClass.html

So it is simply better coding practice to use a function. I kind of wish that Access forced you to use a class module when using LET/GET, but Access is flexible in this case - you can and are free to use LET in a standard code module and do so without having to create an instance of that class in code.

As for performance? Well, since Access can easy execute about 100,000,000 VBA instructions per second, then performance a non-issue, and in this case like zero performance difference exists anyway.
