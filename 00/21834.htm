john w. vinson say:

Only by using Passthrough Queries to be run directly by SQL. 

A 25 field Table is pretty big. A 255 field Table is either part of a (big!) SQL data warehouse, or is misdesigned; using a spreadsheet as if it were a table is pretty much a recipe for trouble!

scottgem say:

A relational database is NOT a spreadsheet. Keeping the database as a single table is a very bad idea. So is allowing users to modify the front end by creating their own queries.

It is up to you, as the developer, to provide users with an interface that they can use to extract the data that they need. It is certainly possible to provide the users with a list of fields to select from and values to filter on that will make it easy for them to generate reports or queries with the data they want.

ken sheridan say: You might like to take a look at DecomposerDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates how to decompose non-normalized data imported from Excel into a set of normalized related tables.  It does so by executing a set of 'append' queries in a specific order to insert rows into a set of predefined tables.  At each stage a brief explanation is given of the current SQL statement being executed.

The rule of thumb when doing this is that data must be inserted into the referenced tables first, and then into the referencing tables.  Otherwise referential integrity would be violated.  In my case the source table has a relatively small number of columns, but for a source table with a large number of columns like yours it is only a matter of scale; the same principles apply.

As regards allowing the users to create queries, a well designed database application should provide an intuitive interface for users to return data.  In most applications the basis for doing so will be predefined in unbound dialogue forms through which bound forms and/or reports based on appropriate queries can be opened.  In some workgroup environments, greater latitude might be appropriate.  In my own area of work, environmental planning, I did make provision for my team, all fellow specialists, to be able to search the database on pretty much any basis, but this was done by building an interface which allowed them to build complex search criteria without any knowledge of the structure of the database, and the results were returned in forms or reports in which the ability to edit the data was carefully controlled, e.g. controls in a form which were bound to a referenced table would be made read-only to prevent their being edited in such a way that the changes would be incorrectly applied in the case of other rows in a referencing table which referenced the same row in the referenced table.  Allowing users to create and open queries directly is very risky, as they might easily create an updatable query which would allow them to inadvertently change values which would result in unfortunate consequences.  The risk would be even greater in the case of a single non-normalized table, as by its nature, such a table would allow update anomalies resulting in inconsistent data, making the database unfit for purpose.  This would be equally true of basic data entry of course, as with such a table, there is no way that the integrity of the data can be enforced.  I once encountered a simple example of this when I found that in one database, not of my design, my name appeared in three separate ways as the author of articles in my own field of work, K Sheridan, K W Sheridan and K V Sheridan, the last being an obvious typo.  As far as the database was concerned this made me three separate people of course, and anyone searching for articles by K W Sheridan, as I'd normally be cited, would not get accurate results.  Murphy's Law tells us that simple errors like this are inevitable as 'if something can go wrong, it will go wrong!'

albert d. kallal say:

You can execute a query on the large column table, but ONLY return 255 columns at one time.

So you might have some “prompt” form in which the user enters date rate/month data, and then your VBA code would create the query that includes the correct “time” or “date” based columns on that data warehouse data.

So you can “hit” and “query” the larger tables, but any query will still be restricted to 255 columns.

If you just common making queries on this data, then Excel might be a better choice.

So if you have “some” means to restrict the columns you need for a query, then Access still might work for you, but any single query you execute will still have to be less than 256 columns.
