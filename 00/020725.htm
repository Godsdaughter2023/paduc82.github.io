Exchange can't disable this mailbox because it is placed on hold - Microsoft Community

alexandru cazacu say:

Exchange can't disable this mailbox because it is placed on hold

I've been meaning to write this article for quite some time now, as it is an issue we frequently see in support cases that we receive from our customer.

I will start by saying that this scenario is by design. As long as the Office 365 user account still exists, the mailbox will not go to inactive mailboxes, even if the license has been removed. The user must be deleted from Office 365, in order to break the link between Azure AD and Exchange Online.

From article https://docs.microsoft.com/en-us/microsoft-365/compliance/create-and-manage-inactive-mailboxes paragraph "Create an inactive mailbox" :

Making a mailbox inactive involves two steps: 1) placing the mailbox on Litigation Hold or applying an Office 365 retention policy to it, and 2) deleting the mailbox or corresponding Office 365 user account. 

After the mailbox is inactive, its contents are retained until the hold or retention policy is removed.

In our scenario, because the user might be synchronized with local Active Directory, and still exists in Office 365 active users, Exchange cannot disable the mailbox and convert it to an inactive one, even if the license was removed.

There is another attribute of interest on the mailbox, besides the litigation hold/in-place hold ones:

- ExternalDirectoryObjectId - this is the link between Azure AD and Exchange Online; the value of this attribute will be populated on the Exchange Online mailbox, as long as the Msol User still exists.

So, in order to fix this situation, the customer must hard delete the user associated with the mailbox from Office 365:

- put the user in an OU that isn't synced with Office 365, in the local Active Directory

- this will soft delete the account in Office 365

- use Powershell to hard delete the Office 365 account with the command Remove-MsolUser -UserPrincipalName *** Emai Get-OrganizationConfig | fl *Hold*. The identity of such a hold, would begin with "mbx" as the first 3 characters, but you might also see "Uni" from Unified (this is an older acronym which should no longer be in use actually, unless the hold comes from an e-Discovery case).

IMPORTANT:  If the user account must be preserved because of existing data in Sharepoint Online and OneDrive for Business, and only the mailbox must be disabled, then please consider removing the hold from it and allow Exchange Online to completely disconnect it (the mailbox will be recoverable for 30 days, if the Exchange license is reassigned).

More information:

Disable-Mailbox command in an Exchange on premise environment, would convert a user mailbox to a simple User object (not Mail User). In Office 365, removing the Exchange license, will trigger a Disable-Mailbox command.

Remove-Mailbox command in an Exchange on premise environment, would delete the mailbox AND the AD user. In Office 365, this command is usually given, when you delete a user without removing its license prior of this (removing the license will perform a Disable-Mailbox first). 

For example, you cannot use the Remove-Mailbox -PermanentlyDelete command in Exchange Online, while we still have an active/soft deleted Msol User. You must first purge the Msol User, to break the link between Azure AD and EXO AD (the ExternalDirectoryObjectId will have its value set to null). 

Pay attention to below 3 attributes as well, that have been recently introduced in Exchange Online, which can be seen on a given mailbox with Powershell command Get-Mailbox -Identity *** Email address is removed for privacy *** | fl *hold*,*compliance*:

a. DelayHoldApplied - this is an attribute that will be set to True, once a mailbox is removed from a hold policy; this prevents the user from instantly losing all the content that is placed on hold in his mailbox; the attribute can block and admin from disabling the mailbox, but it can be set to False by running the command Set-Mailbox -Identity *** Email address is removed for privacy *** -RemoveDelayHoldApplied.

b. DelayReleaseHoldApplied - specifies delay holds on cloud data generated by non-Exchange apps (such as Teams, Forms, and Yammer) from the mailbox. Data generated by a non-Exchange cloud-based app is typically stored in a hidden folder in the mailbox. It can be set to False by running the command Set-Mailbox -Identity *** Email address is removed for privacy *** -RemoveDelayReleaseHoldApplied.

c. ComplianceTagHoldApplied - this comes from Security and Compliance Center, and as its name suggests, it is originating from a Compliance Tag; if this parameter is set to True and the admin wishes to disable a mailbox, it will not be possible; the value of the attribute cannot be changed by the administrator in any way; contact support for this matter. This will require an approval email which includes:

- The ExchangeGuid, DistinguishedName, Guid and UserprincipalName of the user you want the ComplianceTagHold to be removed from.

- Acknowledge in the email that removing the ComplianceTagHold from the mailbox may result in the permanent deletion of the data protected by the hold in that mailbox. 

EXTRA Information:

Excluding a mailbox from the Retention Compliance Policy (hold policy), must be done from the Security and Compliance Center/Data Governance/Retention section. Then pay attention to the DelayHoldApplied flag, not to be set to True (if it is, set it to false and then try to remove the mailbox again).

If the mailbox is already inactive (its associated Azure AD user account has been deleted and the mailbox is on hold), and you want to purge it, you can use the following:

- connect Powershell to Exchange Online: https://docs.microsoft.com/en-us/powershell/exchange/exchange-online/connect-to-exchange-online-powershell/connect-to-exchange-online-powershell?view=exchange-ps

- run one of the following commands:

Set-Mailbox -InactiveMailbox *** Email address is removed for

What about that article :)? That's for when the mailbox has already been disabled. When there is a hold on the mailbox, you should not remove the license as it will result in the error message in the title of the article. You need to either delete the user while the license is still assigned to make the mailbox inactive or you exclude the user mailbox from the hold policy, so that you can disable the mailbox, but keep the account for OneDrive and SharePoint for example.

rob koch say:


< SNIP >

And will there be a security program that will never allow virus or malware ? Can we expect such program from Microsoft ?        

I won't address the other question, since you've already received that answer.

The reason there can't be a security program that stops all virus and malware is Windows itself, since it was originally created at a time when such concerns didn't exist and the PC wasn't Internet or even modem attached to any others.

The open programming interfaces that Windows provides were also created in those early days and so didn't inherently contain any protection, which meant they could be used to do anything that the PC user wished.  This was valuable at that time, since the primary need was to allow the easy creation of programs that the user needed to do various things.

However, as time went on the PC became attached to various networks, first by modem to things like bulletin boards and newsgroups and later to local and wide-area networks in companies and eventually the Internet.  Each of these progressions exposed the individual PC to larger numbers of typically less trusted people and systems, which due to Windows open nature meant greater risks to the security of the data and operating system.

Over time, Microsoft has tried to slowly close many of the obvious holes in security that earlier Windows versions allowed, but every time they do this a major upgrade is required, since this also inherently breaks the operation of many programs that were designed for the older security rules that were then in place.  This has made the process of securing Windows relatively slow and sometimes problematic, since application developers must learn to write for the changes as well.

So though in theory it might eventually be possible to create a version of something called Windows that was highly secure and didn't allow true malware to be easily installed, it would look little like what you now see and instead would look far more like Microsoft's own Windows Phone or the Windows RT that they originally designed to run on the same type of processors that many phones and tablets do.  This change is so radical though, using the newer "app" structure and placing limitations on what programs can do, that few customers were willing to accept it as Windows, since none of their old software would run and only those redesigned to operate on the new platform would work.

So long story short, no it's not possible to perfectly secure the Windows that exists today, though Windows 10 is far better than any version that's come before it.  The choice that exists is whether to continue with the next version of Windows as it is released and improve towards better security over time or simply jump to something entirely new that though possibly more secure, may not be able to run the software you want or prefer.

Microsoft is trying to make this process easier with Windows 10, by automatically updating you to slightly improved versions over time without the need to perform a major upgrade every few years.  This should allow them to fix or remove the more significant security issues over time, though it will also require that application developers occasionally adjust their own software to fix any problems this might create.

Thus, the answer to your last question is that as with any evolving product, Windows will never be "perfect" and will always be changing to improve not only security, but also adding new functionality that developers and so Microsoft's customers desire.

Rob

No major operating system software is bug free, that's a commonly spouted fallacy that Linux fanboys like to repeat.  The reality is simply that practically no one uses Linux, so virtually no one spends any time trying to find its bugs, since it doesn't provide any significant money to either sell this information to malware purveyors or to the Linux developers who maintain it.  Along with that,the few Linux users often don't have much money themselves and since they are often technically knowledgeable they're more difficult to fool with the simple social engineering tricks that have become most common recently.

Windows, on the other hand, has millions of less technical users worldwide, so there's lots of money to be made in selling the information about a newly detected bug before Microsoft is aware and can fix it.  For security firms there's also the notoriety of publishing the details before everyone else, whether they first cooperatively provide this information to Microsoft or not.

In any case, the most secure platforms are inherently limited, with the applications running not as executable modules as with Linux or Windows, but rather as sub-processes that are often unable to operate directly on the processor itself.  This leaves the operating system in complete control, with the apps processed primarily as data, which keeps even a misbehaving app from directly taking control.

This also makes it easier for the operating system to decide what the app can and can not do, monitoring every access it makes in order to decide if that particular security access is allowed.  Doing this with Windows would require a complete emulation of every single step a program takes, which though this is becoming theoretically possible with today's faster processors, would still slow a more complex program too badly to be usable.

So until we reach that ultimate point where everything can run as an emulation, we'll continue to see malicious code as relatively common for Windows.  Since as long as someone can make money from millions of users, there'll also be a motivation to make these programs as well.

Add-ins (both XLAM, which are created through Excel, and COM add-ins, created through a programming language like VB, C#, or others) are basically hidden files that open with Excel. The XLAM file versions created within Excel are actual workbooks, but the file property "IsAddin" is set to True, whereas for all 'normal' files it's set to False. Excel still opens these but always keep them hidden. These are nice when you want some kind of functionality or automation available in all files you open, as opposed to just within the file(s) you open.

COM add-ins are a different beast and aren't necessarily standard Excel files, but a compiled solution which runs in Excel, much like an XLAM, but not editable in the Excel app. This is nice for securing code and such.

The reason it's good to disable all add-ins when testing odd behavior like this is because they have code which runs behind the scenes. Sometimes this can have adverse effects and cause erroneous behavior like you're seeing. A known add-in to cause problems is the Adobe add-in (in previous versions, I think the more current versions are more stable). In order to "rule out" any add-ins causing the problem, the process is simply 1) disable all add-ins, 2) close Excel, 3) re-open Excel, 4) attempt to replicate the error. If you can't, one of the add-ins is the culprit (most likely), and likewise if you can replicate the error the add-ins probably weren't the cause of the problem.

If an add-in is found to be the problem, repeat this process, but instead add them back (check their box - those you had installed) one at a time, closing/re-opening/re-testing between each one you add back. Once you replicate the problem, the last add-in you installed is the offender.

To see your add-ins, click on File, Options, Add-ins (in the left pane), and look at the bottom of the screen. You should see a drop down which defaults to 'Excel Add-ins'. Click the 'Go' button next to it with that selected and you'll get a list of the available add-ins (XLAM). You can change this drop down to 'COM Add-ins' to check that list as well. Repeat the same process of taking them all off and adding them back individually to test for problem reproduction.

I would also try to open Excel in Safe Mode, then try to reproduce the problem (http://office.microsoft.com/en-us/excel-help/work-with-office-safe-modes-HP010354300.aspx). This might be easier to try before checking the add-ins. :)

Alright - assuming the example you just showed is showing how the General Ledger sheet is set up.

Then if your T Accounts sheet is set up like this:

        A                  B
1  T Acct#
2    1101             TR #
3   formula here

You can set this formula up in A3:

=SUMIFS('General Ledger'!D:D,'General Ledger'!B:B,$A$2,'General Ledger'!A:A,$B3)

You can fill that formula on down for that particular account number (as indicated in A2) for as far as you have entries for the TR#s in column B for it and it will work.

If you have more T Account entries on down the sheet, you'll have to start the formula series over again, for example:

      A                B
31 T Acct#
32 1001           TR #
33 revised formula here

At this point the formula needs to be revised to point to the new T Acct# in A32 and the row number for the last parameter needs to reflect the row you enter that formula into, which would be row 33 since this formula would go into A33:

=SUMIFS('General Ledger'!D:D,'General Ledger'!B:B,$A$32,'General Ledger'!A:A,$B33)

'General Ledger'!D:D refers to the column that has the DEBIT/CREDIT amounts in it.
'General Ledger'!B:B refers to the column that has the T ACCT numbers in it
$A$32 refers to the cell on the T Accounts sheet that has the T Account number in it.
'General Ledger'!A:A refers to the column that has the TR #s in it.
$B33 refers to the (first) cell that has a transaction number in it on the T Accounts sheet for the T Account number in question.

robertson-fam say:

I have created an Order Entry database for apparel. The database includes a Table for "Orders" and "Order Details", which are populated using an associated Form/Subform. Nothing fancy.

I also have a related table that lists the items we sell and the details specific to those items (RN Number, Manufacturer, Color, Price, Size Options, etc.)

For example, one shirt may come in XS-2XL, while another may come is Sizes 8, 10, 12.  I would like to have a combo box that lists only the right set of sizes.

ken sheridan say:

My Question: When I enter an order (on my form/subform), how do I limit the "Size" field to only the sizes available in each specific item?

You model it in the database by a table.  What you have is a many-to-many relationship type between Products and Sizes, i.e. each product is available in one or more sizes and each size can apply to one or more products.  A many-to-many relationship type is modelled by a table which resolves it into two or more one-to-many relationship types.  In your case the relationship type is a binary one (2-way) so it is resolved into two one-to-many relationship types.  In broad outline the tables for this would be:

Products
….ProductID  (PK)
….Product

Sizes
….Size  (PK)

and to model the relationship type:

ProductSizes
….ProductID  (FK)
….Size  (FK)

The primary key of the last table is a composite one made up of the two foreign key columns.

A relationship type is really just a special kind of entity type, so can have non-key attributes, just like any other entity type.  If the price of a product varied on the basis of its size then this would be an attribute of the relationship type, so the ProductSizes table would also have a UnitPrice column to represent the current price of the product in that size.  Where a product is not available in different sizes, and the price is consequently a single one then it would be represented in the ProductSizes table by a single row for the product in question in which the Size would be 'N/A' or similar.  This would require a single 'N/A' row in Sizes of course.

For data entry you would have a combo box to select the Product, whose RowSource property would be:

SELECT ProductID, Product FROM Products ORDER BY Product;

Top select a size from that product you'd have a combo box 'correlated' with the first combo box by having a RowSource property of:

SELECT Size, UnitPrice FROM ProductSizes WHERE ProductID = Form!ProductID;

To cause the Size combo box to list the correct sizes for the product in the AfterUpdate event procedure of the ProductID combo box you'd put the following code:

    Me.Size = Null
    Me.Size.Requery

And in the subform's Current event procedure you'd put:

    Me.Size.Requery

so that as you move from row to row in the subform the combo box would show only the sizes for the product in that row.

Another important point with this sort of database is that UnitPrice is not merely an attribute of ProductSizes , but also of OrderDetails.  This is because prices change over time, but each order line must retain the price current when the order was raised.  Consequently the current price must be assigned to the UnitPrice column in the current row of OrderDetails when the size is selected.  This is done by code in the Size combo box's AfterUpdate event procedure:

    Me.UnitPrice = Me.ActiveControl.Column(1)

The Column property is zero-based, so Column(1) is the second column of the combo box's RowSource query, UnitPrice.
