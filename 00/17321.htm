roady [mvp] say:

Well, the SRV method is a whole other method again which is being checked last. So if a method before that returns any results which are incorrect, it will just fail and never reach the SRV method anyway.

 

You first need to decide which method you want to use in your environment to reach the Autodiscover service;

  • LocalXML (client-side solution)
  • SCP (AD object)
  • https://domain.com/autodiscover/autodiscover.xml
  • https://autodiscover.domain.com/autodsicover/autodiscover.xml
  • http://autodiscover.domain.com/autodiscover/autodiscover.xml
    (redirect URL for another https URL which does host the Autodiscover service)
  • SRV record

Of course, this all works together with the published URLs on your CAS so you need to make sure these are set correctly for both internal and external users.

 

To find out which method your Outlook 2010 clients are using, hold the CTRL button while clicking on the Outlook icon in the Notification Area. Then select: Test E-mail AutoConfiguration...

Fill out your e-mail address, password and make sure that only "Use AutoDiscover" is selected. On the Log tab, you can see which method was successful. The results tab gives an overview of your retrieved settings (for both RCP/MAPI as for HTTP/Outlook Anywhere) while the XML tab shows you the raw XML which was returned.

willy denoyette say:

Since 14965, Microsoft has moved from a multi service per service host model to a (more robust) single service per host model.

The net result is an extra of +/- 50 more processes to be created before the desktop is completely initialized. The  result is a pre-desktop slowdown of +/- 2 sec.

I have measured this on a quad core, SSD boot disk system. Your mileage may vary depending on CPU and Disk speed.

I don't know whether this service process model will be maintained or that they will return to the multi service per process model in a later build.
