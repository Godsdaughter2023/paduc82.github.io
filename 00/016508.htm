bill manville say:

I'm not clear what you mean by "not getting any response":

- BeforePrint not being called? (see below)
- Printout not appearing? (check default printer is what you expect it to be)
- Printout appearing but being coloured (see below)
- something else?

 

You need to be careful using EnableEvents that you don't leave it False.

For example, if the user cancelled the ActiveSheet.Printout that would cause an error in the Worksheet_BeforePrint procedure that you have not handled so Application.EnableEvents=True would not be executed.

 

So future prints during the same Excel session would be colourful.

 

I would add On Error Goto TidyUp at the start of the procedure
and
TidyUp:
  MsgBox "Error " & Err & ": " & Error & vbNewline & "in Worksheet_BeforePrint"
  Application.EnableEvents = True
before Exit Sub

jay freedman say:

I tend to agree with John. To answer the original question, though, you'll find the list of crashes in the Windows Event Viewer.

In Windows 10, click the Start button and click All Apps. Expand the category Windows Administrative Tools, and click Event Viewer.

In the Event Viewer, expand the Windows Logs folder and click Application. It may take a minute or two for the program to read the log. Once the list appears in the top part of the window, go to the Actions panel on the right and click Filter Current Log. In the Filter dialog, check the box labeled Error in the Event Level section and click OK. The list will now show only the errors and not the ordinary "Information" events.

Click the heading of the Source column to group the errors by their sources. Select the ones whose source contains "Office" or "Microsoft Office" (you can hold the Ctrl key and click to select multiple items that aren't grouped together).

With all the Office errors selected, go to the Actions pane and click  Save Selected Events. In the Save As dialog, click the Save As Type dropdown and click Text (Tab delimited)(*.txt). Enter a file name and choose the desired folder, such as your desktop, and click Save.

You can open the file in Notepad to see what you saved; you may want to delete event entries that aren't concerned with Word crashes.

Besides showing how often Word has crashed, the information in the entries may help to diagnose why the crashes occur. Two pieces of information that are especially useful to a knowledgeable technician are the exception code and the name of the faulting module, if those items are listed (not all error events include them).
