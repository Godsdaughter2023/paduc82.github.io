graham mayor say:

I don't think you need to store the date. You could just compare the date from the file with today's date each time Word is started e.g.

Option Explicit

Sub AutoExec()
    If GetLastDate("C:\Path\Filename.csv") < Format(Date, "yyyymmdd") Then
        MsgBox "It's older than today so do something"
    Else
        MsgBox "It's not older than today so do nothing"
    End If
lbl_Exit:
    Exit Sub
End Sub

Public Function GetLastDate(strPath As String) As String
Dim oFSO As Object
Dim oFile As Object
    Set oFSO = CreateObject("Scripting.FileSystemObject")
    Set oFile = oFSO.GetFile(strPath)
    GetLastDate = Format(oFile.DateLastModified, "yyyymmdd")
lbl_Exit:
    Set oFSO = Nothing
    Set oFile = Nothing
    Exit Function
End Function

How do I resolve "Sharing Violation" when saving an Excel 2007 - Microsoft Community

sooner_dave say:

How do I resolve "Sharing Violation" when saving an Excel 2007 spreadsheet downloaded through Internet Explorer?

All,

I have spent the better part of the last two-plus days trying to investigate and resolve a "Sharing Violation" issue associated with an Excel 2007 spreadsheet downloaded from Internet Explorer 8 and opened within Excel.

When I attempt to save the file within Excel 2007 via the "Save" button on the top bar, or the "Save.." menu option, the Sharing Violation will occur 100% of the time. "Save As..." efforts always succeed.

My environment:

* Software: Microsoft Office Excel 2007/32-bit, version 12.0.6565.5003 SP2 MSO 12.0.6562.5003
* OS: Windows 7 Enterprise/64 bit, joined to a domain (Build 7600).


In researching this problem, I have seen numerous similar reports of this issue affecting other users, and have attempted to implement the solutions provided in those circumstances. To date, I have attempted and/or verified the following:

  • No version of Trend Micro antivirus product is running nor has ever been installed.
  • Expressly turned off Indexing Service, and verified it is not running
  • Disabled all scanning features of the existing McAfee antivirus product , to the point where Windows is consistently yelling at me that all my "protection" software is disabled or turned off
  • I have installed the following hotfixes, all of which were reported to have solved the "Sharing Violation" issue in other circumstances, but do not solve the problem in this case:
  • ->KB970950
  • ->KB982125
  • ->KB983314

What I have observed so far:

  • Downloading the *exact same file* via the same website under a Windows XP Professional box running IE8 causes the file to be loaded into Excel 2003, which then converts the file into the 2003 native format. It can be saved directly (no Save As required)
  • Under Windows 7, the sharing violation can be terminated IF the "hosting" Internet Explorer process is terminated. That is, the IE file download induces Excel to start with the downloaded file that cannot be directly saved. Killing the "parent" IE process then allows the direct file save to succeed. This strongly suggests to me that a file handle is being improperly shared between IE and Excel
  • Running Process Monitor during the "Save" effort reveals a sequence of approximately ten attempts to save the file, with the "SHARING VIOLATION" occurring each time. Immediately prior to each instance of the "SHARING VIOLATION," an instance of a "PRIVILEGE NOT HELD" error occurs following an IRP_MJ_CREATE operation. The attempted creation is performed with Desired Access of "Generic Read, Delete, Access System Security", and options "Synchronous IO Non-Alert, Non-Directory File, Open Reparse Point", with Share Mode of "Read, Write, Delete".
  • There is a delay of approximately ten seconds between the time the Save button is pressed and the "Sharing Violation" error dialog appears.
  • The literal filename does not appear as locked within Process Explorer, but that filename prepended with a tilde IS shown as open by the Excel process.
  • Direct download of a non-Office file, such as a simple text file, works without exhibiting this behavior. The file opens in Notepad, and a direct save (File-Save) works as would be expected.

What I believe is happening:

  • Internet Explorer downloads the file and saves it to Temporary Internet Files under a random file name. This is shown in Process Explorer, with the file being opened having "ShareMode" of "Read."
  • When Excel tries to direct-save the file, it tries to open the "transmitted" file name with (as noted above) Share Mode of "Read, Write, Delete."
  • I believe that Excel's attempt to save the file is actually being interpreted as, first, an attempt to reopen the temporary file with a *different* ShareMode. This results in an error as it conflicts with the documented behavior of CreateFile as noted at http://msdn.microsoft.com/en-us/library/windows/desktop/aa363858(v=vs.85).aspx, and generates precisely the error being experienced. Note the CreateFile documentation excerpt here:

    "You cannot request a sharing mode that conflicts with the access mode that is specified in an existing request that has an open handle. CreateFile would fail and the GetLastError function would return ERROR_SHARING_VIOLATION."

I have yet to find a complete solution for this issue, and at the moment am able to offer only a clumsy workaround that instructs users to perform a "Save As..." immediately after downloading the file. While this is a stop-gap, a more robust solution to support the proper application behavior would clearly be preferred.

If anyone knows of other hotfixes, service packs, or other fixes I can attempt to resolve this issue, I would be most appreciative. If there is any other information I can provide, please let me know.

Many thanks in advance for your suggestions and help.

-David
