Problem with this solution is that if Word continues to be unstable, you're forced to manually save the document every sentence or two, or risk losing work.

AutoRecovery is not an AutoSave function. You need to periodically press Command S to save. This is what happens:

  • If you've never saved your document (it is still unnamed) and Word crashes, you will lose all your work, whether AutoRecover was enabled or not.
  • If a document contains unsaved changes and you attempt to close it, Word will ask you to save changes. This will be your only chance to get the document on disk, whether AutoRecover was enabled or not.
  • If the document is on disk and AutoRecover is enabled, Word will periodically save temporary files that will be used to rebuild your document only if Word crashes. After you quit normally, these files disappear.

tblNextIDs

SystemID long int required PK

NextNumber long int required

Records would include:

Then you could write a function in a standard module:

public function IncrementAndGetNextID(byval SystemID as Long) as long

dim sql as string

sql = "update tblNextIDs set NextNumber=NextNumber+1 where SystemID=" & SystemID

currentdb.execute sql, dbFailOnError

end function

First, I would suggest that you check out my Blog on Sequential Numbering.  It gives techniques and describes what you are trying to accomplish here.

What you are asking is actually a violation of Normalization rules. the design of a Relational database should reduce redundancy. Each item of data should be stored in a separate field.

ken sheridan say: A table's primary key need not be a single column; it can be a combination of columns which, in combination, are a 'candidate key' i.e. when taken together the composite value is distinct within the table.  Even if you add an autonumber column as the primary key, the candidate key is still a key, and must be defined as such by the inclusion of all of the columns which make up the candidate key in a single unique index.  Sometimes, the addition of a 'surrogate' single column primary key can be useful, other times it serves no useful purpose and merely adds an unnecessary overhead.

Before addressing the  issue of keys in your case it is worth considering what your database is modelling in relational terms, both in the real world and in the database itself.  What you have are three real world entity types, CarModels, Systems and Hazards.  In a database each of these is modelled by a table.  Each table can be given an autonumber primary key of ModelID, SystemID and HazardID respectively.  Note that each row in the Hazards table represents not an instance of a hazard in a particular system in a particular model, but a type of hazard which might be encountered in more than one model and/or system.

A specific instance of a hazard in a particular system in a particular model is a relationship between CarModels, Systems and Hazards (don't confuse the use of the word relationship in this context with a Relationship object in Access).  Each such relationship is represented by a row in a further table which models the relationship type between the other three tables.  The word 'type' is important here.  Think of the relationship type, and consequently the table which models it as the set of many possible relationships between CarModels, Systems and Hazards, each of which is represented by a row in this table.  In the same way a CarModels table models an entity type, and each row in the table represents one entity of that type.  A relationship type is really just a special kind of entity type

The relationship type between CarModels, Systems and Hazards is a many-to-many relationship type, in this case a ternary relationship type as it relates three entity types.  To model a ternary many-to-many relationship type by a table it is resolved into three one-to-many relationship types, so the table would have three foreign keys like this:

HazardInstances
....CarModelID
....SystemID
....HazardID

Do these three columns constitute a primary key of this table?  That depends one what is being recorded in this table.  If you require each row the table merely to say that a particular hazard has been found in a particular system of a particular model, then they are a key, and can be made the composite primary key of the table.  You do not need a column for a sequential number per model/system.  If you do want to number the hazards per model/system then you can compute the numbers at any time in a query.

If, on the other hand, you want the table to say that a particular hazard has been found in a particular system of a particular model on one or more occasions and you wish to identify each of those occasions as a separate row in the table, then the three columns are not a candidate key as their values would be duplicated in more than one row, each of which would have data in other columns.  You could do as you intend and add a further column of numbers, which would be sequential per model per system, in which case the candidate key would be a composite one of the CarModelID. SystemID and the SeqNumber column, but this would not add to the information content of the database, as the numbers can be computed in a query at any time.  Making the  CarModelID. SystemID, HazardID and the SeqNumber column a candidate key would make more sense, or you might use some more meaningful attribute the fourth member of the key rather than a sequential number, e.g. a date or date/time value when the hazard was identified.

If you do wish to use a sequential number, however, then take a look at CustomNumber.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

This little demo file illustrates methods for computing sequential numbers.  In your case the example for numbering per group would be the appropriate model,  This sequentially numbers rows by gender, so the relevant code is called in the gender control's AfterUpdate event procedure.  In your case as the grouping is on multiple columns it would be better done in the form's BeforeUpdate event procedure.

By modelling the relationship type by a single HazardInstances table as above, this assumes that any hazard might be encountered in any system in any model.  In reality this is unlikely to be the case, e.g. a hazard in the electrical system might not be possible in the hydraulics system.  So to cater for this the constraint would need to be modelled by a table like this:

SystemHazards
....SystemID
....HazardID

So, if a hazard is possible only in the electrical system that hazard would be represented in this table by a single row, with the values of the primary keys for the electrical system, and the type of hazard in question.  If on the other hand a hazard can be encountered in electrical and hydraulic systems, it would be represented by two rows.  By enforcing referential integrity in the relationship (object) between SystemHazards and HazardInstances on SystemID and HazardID, the invalid insertion of a row into HazardInstances for a hazard not possible in the system in question is prevented.  You'll find an illustration of how relationships are built up across a database to model constraints like this in Relationships .zip in my same OneDrive folder.  This uses a simple medical prescriptions database as its example.

albert d. kallal say:

When Access opens up a file, it also creates a temporary working file of the same name but with a different extension. This temp working file allows access to manage multi-user mode (to allow more than one person to work on the database at the same time).

 

While most users don't turn on the display of file extensions, most developers as a course of action on all of our computers turn on the display of file extensions . If you turn on file extensions, then you'll see for documents like word you see ThankYouLetter.Doc or ThankYouLetter.Docx.

 

For Access the typical file extension is mdb, and for 2007 and later it is accDB. And, for the access locking file, it is .ldb.

 

It's also interesting to note that this temporary working file is used for access to allow multiple users to enter data at the same time into a database. So it manage the locking to prevent two users from editing the same record at the same time (what Access does internal is save information out to that little locking file as to who is currently using a particular record. When a another user tries to the edit that read, then Access can check that file to see if the record is in use. In fact what this means is, if the user does not have permissions to create a file in the folder where the access database resides, then Access will open up that database in single user mode (in Access terminology single user mode is often refer to as what we call exclusive mode). You thus will find that multiple users trying to open that database at the same time will not be permitted, and you'll even get a message to that effect.

 

When you exit the database, Access delete that file. In the case of multiple users, the very last person out closes out the lights so to speak and that file should be deleted.  However if while using Access, perhaps your program freezes or you have an access crash (or you kill with the task manager), then Access does not correctly shut down and locking file will remain in place. If you open the file again, when you exit, the locking file should then again be deleted when you exit. In other words if you're seeing locking files, it means you have some users where Access is not correctly being shut down. Or Access is freezing or perhaps they are doing something like turning off their computers without first closing or allowing all programs to gracefully shutdown .

 

So if you're absolutely sure that nobody is in the database, then you can manually delete the file, or simply open up the database and close it and then if you are the last person out, then Access will simply delete the file for you. If the file does not get deleted when you exit, this means that you have another copy of Access accidentally running on your desktop that you don't realize, or perhaps as another user who has the file open. So if you open the database file and then exit, then that locking file should then disappear. If the file does not disappear up to doing that, but it means it opened by someone else, or are there some issue of permissions to the folder with the file exists. It is the single user computer at all times, then I would reboot your computer, and try this again.

 

So as a general rule, if no other user has that file open, there's no damage or consequences of deleting that locking file manually in place of the going into access
