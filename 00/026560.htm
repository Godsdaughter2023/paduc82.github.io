bofra say:

Maher,

To re-phrase what Brook already answered, and to be specific to your asks:

1. One federation server is enough to federate multiple custom domains (this was a new feature released about 2 years back, prior to which one federation server used to support only one federated domain).

2. Certificate is required only for the service domain, both custom domains will be federated to the same ADFS proxy and FQDN of this proxy would be the service domain (Ex: fs.xyz.com, fs.123.com, or even svc.abc.com). Ensure this service domain is the 'Common Name' of the certificate as well.

3. If you intend to use the ADFS endpoint for Device registration Service as well, you would need additional SAN entries in the certificate: enterpriseregistration.xyz.com, enterpriseregistration.123.com

4. You don't need SAN entries for each of the custom domains (xyz.com, 123.com), unless you are using the ADFS endpoint for any additional purposes as well.

If this isn't clear, let me demonstrate:

   1.  Table 1 has a field, Field 1, with a data type of Long (AutoNumber field)

   2.  Query 1 begins with "SELECT [Table 1].[Field 1] AS ID..."

   3.  Table 2 has a field, Field 2, with a data type of Long (Numeric field to be used to link to Table 1's Field 1)

   4.  I want to create, Query 2, to be something like: "SELECT [Table 2].* FROM [Query 1] INNER JOIN [Table 2] ON [Query 1].ID = [Table 2].[Field 2]"

michael gallop say: Never mind...  I figured out my problem.

It was that I was using a UNION query where some SELECT statements were setting the value of ID to Null and that was why the result for the Non-Null results were treated as strings.
