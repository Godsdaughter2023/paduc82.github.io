  • Like many others, I had trouble with all the “Security Monthly Quality Rollups for Windows 7” - from October 2016 onwards: Oct 2016 - KB3185330, Nov 2016 - KB3197868, Dec 2016 - KB3207752 and Jan 2017 - KB3212646. Each one failed with error code 80004005 and then reverted - and the only solution I could find, as a workaround, was to hide each update and manually apply the equivalent security-only update.
  • In previous research I had noted that some people thought this problem was associated with dual-boot computers. My Dell precision M6600 laptop is NOT dual boot – so I had not pursued that line of enquiry – but when I saw Dave Brammer’s post I realised that I too, in the past, had swapped my HDD for an SDD and so the problem might, after all, be something to do with partitions on the disk.
  • So, this is what I did – and it solved the problem:-
  • Looking at my disk using the built-in Disk Management tool (“create and format hard disk partitions”) – which appears to be a GUI version of DISKPART - I could see I had 3 partitions:- Partition 1 was small and unused,  Partition 2 was small and called RECOVERY, Partition 3 was the large partition, called OS. I could also see that Partition 2 was marked “Boot, Active” whereas Partition 3 was not. So somehow the computer was booting using Partition 2, even though the operating system and everything else was on Partition 3. I don’t really understand how.
  • Maybe this was how Dell set it up – or, more likely, this somehow happened when the cloning software (Acronis) created the partitions on the SSD.
  • Preparation:- I made sure I had a recovery disk that I could boot from - with access to DISKPART.
  • Then I followed Dave Brammer’s instructions, using DISKPART to make Partition 3 active. (NB I believe I could equally have done this using the Windows Disk Management utility)
    BUT . . .
  • WARNING – this made the computer unbootable, presumably because there was no ‘boot loader’ on Partition 3 (I’m not 100% sure of my terminology here)
  • I could probably have fixed this new problem using a windows repair disk / utility – but I was nervous about that – so, instead, I backed out by booting onto a recovery disk and using DISKPART to make Partition 2 active again.
  • Then . . .  INSTEAD, I used the commercial tool, EasyBCD, to “change boot drive” to C: - and that worked! It not only changed the active partition but also did whatever is necessary to make the main partition bootable. I rebooted and everything was fine. And then I applied the January rollup KB3212646 – and that was all OK. Job done  . . . well almost . . .  I also discovered that Windows had assigned a drive letter (E:) to Partition 2 – so the last thing I had to do was use the Disk Management Tool to un-assign the drive letter.
  • So MANY thanks to Dave Brammer for putting me onto the solution. Hopefully this will work for others who have the same problem. But do please note my warning above.
  • PS – My girlfriend also has a DELL laptop (17R - 7720), also with an SSD that I installed in place of the original HDD. Same problem. Same solution.

Hi

 

Follow these steps to uninstall a Windows Update.

 

1. Press the WinKey+X to display the system menu and select Control Panel.

 

2. In Control Panel, select Windows Update.

 

3. On the left side menu, look at the bottom of the window and click Installed Updates.

 

4. Scroll down and locate the update you wish to uninstall.

 

5. Click that update and then click the Uninstall Button at the top of the window.

 


Let us know if this helps.
