richmilburn3813 say:

I suspect that the issue is related to the fact that DirSync used a local account and/or group when the hosting server was a member server. When a server is promoted to be a domain controller, the local security context (SAM database) is replaced with the domain's security context (SAM database aka Active Directory). The domain local groups and accounts are now that server's local groups and accounts, and for all DCs after the first DC in the domain, the former local security database is discarded. So when DirSync tries to connect to its database, the security context it's trying to use no longer exists. Demoting the DC temporarily to uninstall DirSync might work, but chances are, DirSync would need to be removed the hard way. Or, do it the easy way - demote the DC, wipe the server, reload Windows, re-promote it to a DC, and install AAD Connect fresh on another server. The most time-consuming part of that would be patching the new Windows install :)
