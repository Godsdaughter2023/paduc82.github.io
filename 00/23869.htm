ken sheridan say:

Somehow the date has picked up the date/time formatting..............

You can prevent that happening again by setting the ValidationRule property of the Date column to disallow dates with a non-zero time of day element.  If the column allows Nulls, i.e. is not Required, set it to:

    Is Null Or DateValue([Date])

If Nulls are disallowed, i.e. it is Required set it to:

    DateValue([Date])

You can easily remove any non-zero times of day from existing data with an UPDATE query:

UPDATE Donations2
SET Donations2.Date = DateValue(Donations2.Date)
WHERE Donations2.Date IS NOT NULL;

I would advise against using Date as a column name, however.  As the name of a built in function in Access it is a 'reserved word' and should not be used as an object name.  Use something like DonationDate instead.  Provided you qualify it with the table name, as I've done above, there should be no ambiguity as to what is being referenced, but nevertheless its better avoided.

The W10 Mail app and Outlook desktop (part of Office) are two entirely separate and distinct <Email Client> programs that do not interact with each other in any way shape or form which is why you haven't found any articles.

scottgem say:

An inventory application that will be used to report on quantity on hand should be based on a Transactions that records all movement of items in and out. 

To enter a new item, you can't just scan a barcode since the item ins't in the database yet. It has to be added. You could scan the barcode, but will need to fill in all the other info. 

The Transaction table can be used to scan the barcode to enter the Item as a foreign key. I'm assuming that items are checked out to individuals. So you would have a Transaction to check out each item, again scanning the barcode as a Foreign key. 

QOH is then CALCULATED (not stored) by adding the incoming transactions ans subtracting the outgoing ones. 

You should have lookup tables for things like mfg, model, colors, etc.
