    scottgem say:

    This is a standard technique called Cascading or Synchronized comboboxes. There is a lot of documentation for this technique if you search on those terms.

    The first thing you need is to have a field in your Accounts table for AcctCAtegID. This means that an Account is associated with only one Category. It also means that you should NOT have a field in the Transactions table for

    I'm not sure what these queries are that you refer to, but they are unnecessary for this task. The CboAcctCateg should be unbound since its value is not stored. The RowSource of CboAccount should look like this:

    SELECT AcctID, AccountName FROM Accounts WHERE AcctCAtegID = Forms!formname!CboAcctCateg ;

    In the After Update even of CboAcctCateg you want to requery CboAccount. CboAccount will then show only those accounts that match the selected category.

    ken sheridan say: As Scott has pointed out the relationships with a set of correctly normalized tables would be like this:

    Transactions>----Accounts>----AccountCategories

    You'll find an example of the use of correlated combo boxes in this sort of context as ComboDemo.zip in my public databases folder at:

    https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

    If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

    In this little demo file I would suggest that you take a look at the form for a simple 2-level hierarchy, in which my Locations table is analogous to your Transactions table, my Parishes table to your Accounts table, and my Districts table to your AccountTransactions table.

    The relevant form in my demo uses 'hybrid' controls in which a text boxes is superimposed on a combo box to give the appearance of a single combo box control.  This is necessary because of the use of hidden numeric key columns as the BoundColumn of each combo box.  If combo boxes alone were used, data in some rows other than the current one would disappear (not lost, just invisible) because when the combo box is requeried there would be no value in the visible column in the control's list to which the numeric column can map.  This value is shown in the text box superimposed on the combo box.

    Note how the form is based on a query which joins the three tables, and In the form's module take a look particularly at the AfterUpdate event procedure of the cboDistict combo box, and at the form's Current event procedure, in both of which the cboParish control is requeried to show only those parishes within the selected district, analogous to showing the accounts within the selected account category in your case.

    ken sheridan say: You seem to have a form based on a Customers table, in which you have a subform based on an OrderDetails table or similar.  A more usual approach is to use a form based on an Orders table, within which is an order details subform.  A combo box bound to a CustomerID foreign key column in the Orders table is then used to select the customer for the order, and by linking the subform to the parent form on OrderID the value of the parent form's primary key is automatically inserted into the OrderDetails table's foreign key OrderID column.

    You'll find an example of this sort of set-up as InvoicePDF.zip in my public databases folder at:

    https://skydrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

    This demo file is primarily intended to illustrate how to output a report (an invoice in this case) as a PDF file, but the invoices form is set up in the same way as would be the case for orders.  Ignore the subforms at the bottom of the parent form for the moment, just concentrate on the parent form and the invoice details subform, which are the equivalent of an orders form and order details subform in your case,

    To create an invoice you just go to a new record in the parent form and select the customer from the combo box's list.  The other customer details will show in the controls below the combo box.  These are bound to columns from the Customers table in the form's underlying query, and are read-only by virtue of their Locked property being True and their Enabled property being False.  To enter a new customer not yet in the list you simply type the customer name into the combo box, answer Yes at the prompt and then enter their other details in the form which opens.  This is achieved by the code in the combo box's NotInList event procedure.

    Once you've entered the invoice date you can then enter rows in the invoice details subform, one for each item charged in the current invoice.

    With your current customers form you would need to have an orders subform within it to enter orders records before you can enter order details records, which could be done in a subform within the orders subform, which would have to be in single form view.  Alternatively you could have two subforms in continuous forms view within the parent form, and correlate the subforms so that as you navigate from order to order in the first, the second is requeried to show the order details rows related to the currently selected order.  That would require a little more work, however.  The StudenLog.zip file in my same SkyDrive folder illustrates correlated subforms, though these are rather more complex than you'd need.

    scottgem say:
    The After Update you cited, sets a filter to the selected customer. So the form would show an existing customer. If you want a newly entered customer to appear, you have to requery the combobox. 

    You also have a relationship with referential integrity set between customers and orders. So you cannot create an order record unless the customer record exists in the customer table. You may have entered the data, but not committed the record.
