The May 2019 Feature Update (Windows 10 Version 1903) appears to be build 18362.30. You can create an unofficial ISO for that, using a website called UUP dump.

Here is a link to the x64 build 18362.30. It will take quite a long time to assemble the .30 version, but the source files used come directly from Microsoft's Windows Update servers.

https://uupdump.ml/known.php?q=Feature+18362.30+amd64

Step through the link provided on the website and select your Language and Edition, such as Pro or Home. Select Download using Aria2 and convert. A Zip file will download.  Unzip the file to a folder to get a couple of CMD files. SmartScreen may give some warnings, since the files came from the Internet, which you will have to allow to access the Zip contents.

Run aria2_download_windows.cmd, which will begin downloading files from Microsoft and processing them into an ISO. It has to mount the Windows image and add the .30 Cumulative Update to complete, so this takes a long time.

When it is done, you will have a 18362.30 ISO with a long name. Mount that in Windows and run setup.exe from the drive letter that it produces, to either upgrade to that build, or to repair your existing install, if you already have that build number. It will preserve your files and data, just as a Windows Update upgrade would do.

This is not an official way to create an ISO, nor is it recommended by Microsoft, but many of the Community Forum users have used this method to produce newer ISOs than what is available otherwise.

 

If you leave your PC off of the Insider Program you should not need to worry about getting further Insider Builds.

Without seeing your code, it is hard to give you an answer I can be sure of; however, maybe using a global variable would work for you. To declare a global variable, declare it as Public in a general Module outside (and before) of any procedures that you might have in the module. So, simply put something like this on a line all by itself at the top of the module...

 

Public CopyCell As Range

 

Then, in whatever procedure you now issue the Copy command from, do this instead...

 

Set CopyCell = Range("C3")

 

In the above, C3 is the assumed cell you are copying... use the reference method you now use to identify the cell. Next, in whatever procedure you now issue the Paste operation, do this instead....

 

CopyCell.Copy Range("D4")

 

In the above, D4 is the assume cell you are copying to... use  the reference method you now use to identify the cell.

You cannot undo it, but you can try the following:

  • Right-click the Publisher document in File Explorer.
  • Select Properties from the context menu.

The Normal template (Normal template: A global template that you can use for any type of document. You can modify this template to change the default document formatting or content.) opens whenever you start Word, and it includes default styles, AutoText, macros, toolbars, and other customizations that determine the basic look of your document.

  1. Double-click the Normal.dot file to open it. To be certain that you're working in the default template, check to see thatNormal.dot appears in the Word title bar.

Hi Www_w_,

 

There are several causes why users receive NDR when sending emails. NDRs include a code that indicates why your email isn’t delivered, you can refer to the list in this article.

 

In your case, the emails you send have an invalid sender address (null sender), so it would be rejected by Office 365. Sometimes a rejected message will appear as “550 5.7.512 Access denied, message must be RFC 5322 section 3.6.2 compliant and include a valid From address”. This is an expected behavior, as the NDR cannot be sent back to the sender whose address is invalid. Therefore, NDR will not be sent and it will be “discarded”

lance cai msft say:

Hi www_w,

 

If you send an email from a null/invalid address, this email won’t even reach the Office 365 exchange server, so no NDR will bounce back. An error message will be sent to the sender like what I showed above. In the log, you didn’t type in the address in the From part, so it also means a null address. You can run a message trace and view result to check whether the null sender has sent out the emails to Office 365 exchange server. No results means that the email didn’t touch the server.

 

A summary of the problem: if an email is sent from a null/invalid address, an error message will bounce to the sender. There is no business with Exchange server and no NDRs for you checking.

If an email is sent from a valid address and it failed to delivery to the receipt, the Exchange server will bounce a NDR to the sender.

 

Thanks,

 

Lance
