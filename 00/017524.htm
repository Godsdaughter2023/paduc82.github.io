bulk convert odt to docx? - Microsoft Community

a. user say:

bulk convert odt to docx?

Does anyone know of a method (preferably free!) to BULK convert odt documents to docx? (I've done a fair bit of searching and can't find anything...)
graham mayor say:
You could use a batch macro e.g.the following run from Word 2007 will save all ODT documents in a folder as DOCX (retaining the ODT version) - assuming you have the ODT filter installed.

Sub SaveAllAsDOCX()
Dim strFilename As String
Dim strDocName As String
Dim strPath As String
Dim oDoc As Document
Dim fDialog As FileDialog
Dim intPos As Integer
Set fDialog = Application.FileDialog(msoFileDialogFolderPicker)
With fDialog
    .Title = "Select folder and click OK"
    .AllowMultiSelect = False
    .InitialView = msoFileDialogViewList
    If .Show <> -1 Then
        MsgBox "Cancelled By User", , "List Folder Contents"
        Exit Sub
    End If
    strPath = fDialog.SelectedItems.Item(1)
    If Right(strPath, 1) <> "\" Then strPath = strPath + "\"
End With
If Documents.Count > 0 Then
    Documents.Close savechanges:=wdPromptToSaveChanges
End If
If Left(strPath, 1) = Chr(34) Then
    strPath = Mid(strPath, 2, Len(strPath) - 2)
End If
strFilename = Dir$(strPath & "*.odt")
While Len(strFilename) <> 0
    Set oDoc = Documents.Open(strPath & strFilename)
    strDocName = ActiveDocument.FullName
    intPos = InStrRev(strDocName, ".")
    strDocName = Left(strDocName, intPos - 1)
    strDocName = strDocName & ".docx"
    oDoc.SaveAs FileName:=strDocName, _
                FileFormat:=wdFormatDocumentDefault
    oDoc.Close savechanges:=wdDoNotSaveChanges
    strFilename = Dir$()
Wend
End Sub

robert aldwinckle on forums say:

Chrome?


"chrome"  (not capitalized) is a way of referring to the parts of the IE window that contain the UI elements but not the content of the tabs.

Apparently according to this document I could have referred to it as the "UI Frame".   Who knew?

 

http://blogs.msdn.com/b/ie/archive/2008/03/11/ie8-and-loosely-coupled-ie-lcie.aspx

 

You can identify the the chrome/UI frame task in Task Manager's Processes tab by two things:

  1. the "chrome" iexplore.exe task has a simpler command line syntax
  2. the other iexplore.exe tasks refer to that other ones PID via a SCODEF: parameter

Sort the Processes tab by Image Name to see all the iexplore.exe tasks grouped together and then scroll over to the Command Line "column" to see these details.   You may have to enable the Command Line column via the View menu.

 

So I think provided you don't kill the chrome task you have a chance of keeping the rest of the tasks running, e.g. assuming they are doing something.   You can determine that by using the Resource Monitor checking all iexplore.exe tasks in the Overview tab's CPU section and then sorting them by Average CPU Utilization (%).   E.g. sometimes I have found a new tab task in a deadlock state (using right-click Analyze Wait Chain)  and killing it lets everything resume (including the thing I "killed" which then pops up in a separate window which is kind of cool <w>).   The deadlock has had the effect of "freezing" the chrome, which conceivably might include some of its rendering activity, such as removing an old tab label.   OTOH I suppose it might also be possible that your symptom indicates that the chrome crashed but hasn't either fully recovered or fully terminated.   In that case I would try looking for clues in the Event log which would support that hypothesis.   E.g. use Event Viewer to search the Windows Applications log for events which mention iexplore.exe.   (A related diagnostic is the Reliability Monitor.  That actually might be easier to use.  I keep forgetting about it since it is a new feature with Vista, whereas Event Viewer has been in NT forever.)  Otherwise I'm not sure how I would try to diagnose this, probably run ProcMon and hope that something interesting popped up that way--kind of impractical, especially when it is not something that is easily reproducible.

 

Good luck

Robert
