tom van stiphout say:

Most of the regular in this forum will promote best practices, because they have worked well over the years. Part of that is that tables are for STORING data, not for displaying data (other than the most elementary) and not for editing. We have queries to add more richness such as

joining with other tables to pick up more fields, and for end users forms and reports to interact with the application.

One reason we don't often use lookups is because they hide the underlying information. For example if I download the Northwind sample application and open the Orders table, I see an Employee column with in one of the rows a value of Jan Kotas. Two things are confusing here: the underlying field is [Employee ID], not Employee, and the value is 3, not Jan Kotas. When I'm witing code or creating queries, I usually want the raw data, not the flowery version for human consumption.

That said, one advantage of using lookup controls in tables is that when you create an auto-form based on this table, you will automatically get a combobox. To me that's not enough of a benefit. I spend the vast majority of project time on business logic, not on cosmetics.

scottgem say:

What problem(s) will I encounter as a result of creating a lookup combo box at the table level using the lookup wizard?

Assuming I want to create the lookup combo box at the table level is there a way to include a value from another field in the table in the Row Source SQL statement on the Lookup tab? Thanks.

Well first, you CAN'T do what you want in a table. You can only modify the Rowsource of a combobox on a form. 

SELECT [Event].[Event Number], [Event].[Event Name] FROM Event WHERE [Event].[Customer Number] = Forms!formname!controlname ORDER BY [Event Name]; 

If you set the RowSource using the syntax provided above (substitute your form and controlname)the list in the combox will only show those events for that customer.

ken sheridan say: You were nearly there with your SQL statement for the combo box's RowSource property.  The parameter is a reference to the [Customer Number] control in the form:

SELECT [Event].[Event Number], [Event].[Event Name] FROM Event WHERE [Event].[Customer Number] = Form![Customer Number] ORDER BY [Event Name];

In the above Form is the Form property and returns a reference to the current form, Customer Number is the name of the control in the form bound to the Customer Number column.  The name of the control and the column might or might not be the same.

To get the combo box to show the restricted list of events you need to requery the combo box in two places.  I'll assume for this that the name of the combo box is cboEvent:

1.  In the Customer Number control's AfterUpdate event procedure with:

    Me.cboEvent = Null
    Me.cboEvent.Requery

2. In the form's Current event procedure:

    Me.cboEvent.Requery

Note that in the first case the combo box is also set to Null.  This is in case you are changing the existing value of the Customer Number control to a different value, and removes any inappropriate value from the cboEvent combo box before Requerying it.

I assume that you wish to show the event names in the cboEvent combo box, rather than the event number to which it is bound.  This is what the lookup field wizard does, but as the others have said, doing it at table level is pointless and has some unfortunate consequences, though nothing fatal.  To do it in a control in a form set the combo box's BoundColumn property to 1, its ColumnCount property to 2, and its ColumnWidths property to 0cm (or inches).  This will hide the first column and show the event name.

One caveat:  this will not work correctly in a form in continuous forms view, only in single form view.  This is because, in continuous forms view, if the current record restricts the combo box to events which do not include events selected in other rows in the form, the combo box in those rows will appear blank because the hidden value of the control does not map to a value in the second column of the list.  No data is lost, merely hidden.  This can be overcome with a little trickery, by the use of a hybrid control, in which a text box is superimposed on a combo box to give the appearance of a single combo box control.  You'll find examples in ComboDemo.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to amend the form design accordingly.  

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.
