andrea jones - all about resources say:

First step, sort your lookup values so the first column is in descending order.

Use a combination of the INDEX and MATCH functions to select the value.  If our lookup table is in cells F2:G5 we can use the following to find the closest match for a value in cell A2:

 

=INDEX($F$2:$G$5,MATCH(A2,$F$2:$F$5,-1),2)

Regular expression not working in Outlook Add-in, but only on the - Microsoft Community

sam miles say:

Regular expression not working in Outlook Add-in, but only on the Desktop version

I am currently developing an Outlook add-in and may have found a bug in how the desktop client renders third party add-ins. 

I currently am using a regular expression to extract some information from an selected email in outlook. 

Here is some javascript that will be run in the add-in code:

Assume that body is a string including the entire body of the email read in from the email currently open in Outlook Desktop.

var re = /#(.*)(?= and)/;

var matches = new Array();

var match = re.exec(body);

while( match != null) {

   matches.push(match[1])

   match = re.exec(body);

}

This regex parses emails and provides a list of used hashtags followed by the string ' and'. This works in Outlook 365 on all modern browsers including back to IE 10. However, when this regex is run from the Outlook Desktop application, it doesn't work as expected.

Any insight would be greatly appreciated. 

sam miles say:

After some digging and debugging, I found that the Desktop Outlook appears to be making manipulations on line endings in the email. Outlook 365 Web does not do this. When the javascript regex was reading the email on the desktop version, the body had been altered and didn't match the same email from the web experience. Just another case of weird line endings causing headaches. 

I ended up using another regex to replace the vertical tabs implanted with new line characters. Not sure if its a bug or a feature but hopefully this helps anyone else who has a similar issue
