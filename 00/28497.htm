amit tandon say:

Reason:

Excel follows the IEEE 754 specification on how to store and calculate floating-point numbers. Excel therefore stores only 15 significant digits in a number, and changes digits after the fifteenth place to zeroes. 

Workaround:

To work around this behavior, format the cell as text, then type the numbers. The cell can then display up to 1,024 characters.

For further details refer below link:

https://support.microsoft.com/en-in/help/269370/last-digits-are-changed-to-zeroes-when-you-type-long-numbers-in-cells

joeu2004 say:

Amit wrote;

Excel follows the IEEE 754 specification on how to store and calculate floating-point numbers. Excel therefore stores only 15 significant digits in a number, and changes digits after the fifteenth place to zeroes. 

Although Amit is correct that the root cause is limits due to the internal representation of numerical values, he is wrong about the details, which are often misstated, even by "experts".

First, the IEEE 754 standard specifies using 17 significant digits, not 15, to convert between binary and decimal representation with no loss of accuracy.  And indeed, Excel writes and reads (interprets) 17 significant digits in XML files, which is used to store "xlsx" files (compressed).

Second, Excel does not store and calculate with only 17 significant digits, much less 15.  That is merely the decimal "presentation" (appearance) of numeric values.  Generally, the exact binary values are used.  (There are some exceptions, where Excel does not follow the IEEE 754 standard.)

For example, if we enter =1/3 into A1, that appears to be 0.333333333333333.  But =A1-0.333333333333333=0 returns FALSE because the exact decimal representation of the internal binary representation of 1/3 is 0.333333333333333,314829616256247390992939472198486328125 .  In contrast, the exact decimal representation of the binary representation of 0.333333333333333 is 0.333333333333332,981762708868700428865849971771240234375 .

(I use period for the decimal point and comma to demarcate the first 15 significant digits.)

However, it is correct that when reading decimal input (typed or read from a text file), Excel truncates after the 15th signficant digit, subsituting zeros for digits on the right.  And when presenting decimal values (formatting in worksheets or writing into text files except XML), Excel usually rounds the 15th significant digit, substituting zeros for digits on the right.  (There are expections due to a formatting defect.)

imacken wrote:

I suspect Dana was asking:  what problem are you trying to solve that requires more than 15 significant digits of precision?

Are you trying to calculate trajectories to land within 3 feet of a target on Mars, for example? <wink>

But it is good that you showed us a concrete example of the calculation.  There are other issues to consider.

Your original question suggested an interest in integers with more than 16 digits.  Instead, your example uses decimal fractions (although we don't know what "constant" is).

Even though the decimal fractions are within the 15 significant digits that Excel interprets correctly for data entry, most decimal fractions cannot be represented exactly in 64-bit binary floating-point, which is the internal representation.

The reason is:  decimal fractions are represented by the sum of consecutive powers of 2.  For example, 1/10 (0.1) cannot be represented exactly by a sum of consecutive powers of 2, no matter how many binary digits we use.  Instead, 0.1 is represented exactly by 0.100000000000000,0055511151231257827021181583404541015625 .

For your example, the exact representations are 0.00944282812657065,0126847539240770856849849224090576171875 and 9.71216791700000,03722407200257293879985809326171875 .

For this particular example, the following VBA function could be used for an exact result of the first product pair:

Function mymult(x As String, y As String) As String
mymult = CDec(x) * CDec(y)
End Function

But that might not work for the second product (times "constant") and for other examples.

To apply the example from joeu2004:

Right-click on the sheet tab
Choose "View Code"
Within the menu click Insert \ Module
Paste the code 

Function mymult(x As String, y As String) As String
  mymult = CDec(x) * CDec(y)
End Function


Close the VBA editor
Write your values in A1 and B1
Apply the formula in C1: =mymult(A1,B1)
Save the file as .XLSM (otherwise the code is lost if you close the file)

Note:
This code is what we called an UDF.
UDF means User Definied Function
