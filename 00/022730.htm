I too was wanting to move my Office 365 install to a different drive.

My main goal was to put it on a RamDisk to make it very quick to load.

I had created a RamDisk of 5GB, using http://www.radeonramdisk.com/software_downloads.php

The new drive was mounted as drive O:

I installed Office 365 as normal, on the C drive.

I then made a note of the two main directories:

C:\Program Files\Microsoft Office 15

C:\Program Files (x86)\Microsoft Office

I emptied my Recycle bin before continuing.

I then ran Task Manager and stopped anything Microsoft Office related.

Then I deleted both of the above directories, trust me on this.

After that I went to a CMD prompt, and ran these commands:

MKLINK /J "C:\Program Files\Microsoft Office 15" "O:\Program Files\Microsoft Office 15"?

MKLINK /J "C:\Program Files (x86)\Microsoft Office" "O:\Program Files (x86)\Microsoft Office"

This created junction points for each directory on the C drive that Office installs to, and point them to the O drive. So the OS thinks its going to C drive for those folders, but in reality its going to the O drive.

I then went to my recycle bin and restored all the deleted files.

The files then appeared on the O drive.

I then saved the RamDisk to an image file (so it gets loaded when windows starts)

Finally, I rebooted.

After reboot, when I open an office app, its so fast you can't time it!

You don't have to do the ramdisk portion, its the junction's that make it work.

I have 32GB of ram, and am only using about 5gb on average, so having a ramdisk makes sense for me.

RamDisks are about 8 to 10 times faster than SSD but are volatile memory, so it needs to be saved to disk if you don't want to lose it.

Here is an example for removing duplicates. plug in the fields that constitute a dup. Be sure to make a copy of the table first in case something goes horribly wrong. [grin]

DELETE *  
FROM T_Data  
WHERE (SELECT  Count(*) 
       FROM T_Data AS T  
       WHERE Nz(T.F1, "") =  Nz(T_Data.F1, "") 
       AND Nz(T.F2, "") =  Nz(T_Data.F2, "") 
       AND Nz(T.F3, "") =  Nz(T_Data.F3, "") 
       AND Nz(T.F4, "") =  Nz(T_Data.F4, "")  
       AND T.ID <= T_Data.ID) > 1;

ken sheridan say: Here's a simple example which would delete all rows bar one with the same LastName value, retaining the row with the highest (MAX) value of the key (ContactID) per last name:

DELETE *
FROM Contacts AS C1
WHERE ContactID <>
     (SELECT MAX(ContactID)
      FROM Contacts AS C2
      WHERE C2.LastName = C1.LastName);

If the table does not currently have a candidate key, simply add an autonumber column to act as the key.

albert d. kallal say:

>>Will there be compatibility problems if we share an
>>Access 2010 32 bit front end and back end?

The back end will ALWAYS be neutral and will work fine for all versions x32/x64.

The front end? This becomes more complicated. If the front end is just plane Jane VBA, and does not use ANY windows API calls, and does not use ANY ActiveX controls, then that application should run without ANY issues with Access x32 or x64. And it should run without any changes. Both x32 and x64 front ends can consume the back end, and do so at the same time if this is a shared/multi user application.

However, if you distribute a compiled version of your application (that means an accDE as opposed to an accDB), then you MUST compile that front end with the appropriate version of Access.

So you must compile the accDB to accDE with version x32 for computers with x32 Access.

So you mustcompile accDB to accDE with version x64 for computers with x64 access.

The above "assumes" you need or want to use a compiled accDE as opposed to a accDB.

As noted, if using an accDB, then you can distribute that front end to BOTH x32 and x64 without issues and WITHOUT having to compile separate versions (assuming the above points about windows API and ActiveX).

>>I understand that

- we should use Office 2010 32 bit unless we really need 64 bit

Correct, it is “in general” and most common to install office x32 on BOTH x32 and x64 bit machines.

>>that we can't change windows 7 32 bit to 64 bit

Correct - You would have to re-install windows. However, since windows 7 came out VERY FEW machines are x32, so it is unlikely that you have many if any windows 7 x32 machines.

>>I might be able to change Office 2010 from 64 bit to 32 bit without having the product key.

No, you have to un-install x64 complete. It is a different bit size software system and different processor architecture.  You then have to re-install to office x32 – and yes, this will require your product key, however you can use your EXISTING key – you don’t need a different one.

So there are no steps that exist to uninstall and re-install office without a product key. However the SAME keys you used to install the existing office should work – it not possible that you installed such office without a product key – so you MUST have had such a key at one time. 

As noted, you “can” continue with a mix of x32 and x64 office.

How “challenging” this setup is will depend on the given Access application (does it use windows API), have you been distribution a compiled accDE as opposed to accDB etc.

So depending on your setup, flipping everyone over to the same version may not be required, it will much depend on the given Access application.

albert d. kallal say:

You can also consider using the free Access runtime. The runtime allows most applications to run, but simply does not allow design changes.

So presumably, you could install the Access runtime on that computer.

However if all other people are running Access x64 (and I would double check, and triple check), then this might not be worth the trouble for JUST one machine. (and as noted you could NEVER have installed access x64 on that one computer anyway.


As noted the runtime (even for x64 machines) can be considered. The only machines that really need the full version of Access are those “developing” the application. The users or “consumers” of the application only need the free runtime.

Keep in mind that not applications will work with the runtime. The application in question will need a startup form and some kind of custom UI that lets the user launch forms and reports. If the application requires the user to pick and launch and “know” what form to launch from the navigation pane, then such application cannot be used with runtime.

However, if the application is quite “polished” up and provides startup forms to launch other forms and reports, then the runtime system can be used. (so applications that provide a user interface as opposed to the “users” opening and working on a given report or form work well with the runtime).

And if you ONLY have one computer running Access x32, then while the runtime install of Access is free and does not require a license, the additional trouble and work to support that one computer likely is not going to be worth the time an effort.

So you are free to install the runtime (x32) on that computer. As noted, it really depends on how “polished” the application is. And as noted, if all the existing machines use x64 bit version of Access, then likely changes need to be made to run on x32.

However, supporting that one “different” computer could result is as much work as supporting the other 20 so to speak.

First up, I would really double check as to what versions of Office the computers are running. If you have a “mix” of x32 and x64 office/Access installed on the x64 OS machines, then efforts to ensure the application runs on x32 and x64 are worth your time and effort. However, for just ONE computer, that is another story and likely not worth the efforts and time to support the one computer.

davidh888 say:

Calculating the coordinates of points along a line

I have a line of points, with a start coordinate, and azimuth to the next...as part of a channel. So the orientation of the channel will change, as may the elevation (entered angle from horizontal). Is there a way to calculate the rest of the coordinates in the line?
Thanks,

gary's student say:

A straight line is of the form:

Y=AX+B

 

say C1 contains the x-coordinate, D1 contains the y-coordinate and E1 contains the angle in degrees.

 

We calculate the slope (A) in cell C2:

=TAN(E1*PI()/180)

 

We calculate the intercept (B) in cell C3:

=D1-C2*C1

 

We list the desired x-values in column A and in B1 enter:

=A1*$C$2+$C$3   (Excel's version of Y=AX+B )  and copy down

 

For example, if C1,D1,E1 are 1, 3, and 35 then in column A & B:

 

0 2.299792
1 3
2 3.700208
3 4.400415
4 5.100623
5 5.80083

9 8.60166

12 10.70228
