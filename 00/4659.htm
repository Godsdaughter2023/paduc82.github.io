Sub AttachLabelsToPoints()

   'Dimension variables.
   Dim Counter As Integer, ChartName As String, xVals As String

   ' Disable screen updating while the subroutine is run.
   Application.ScreenUpdating = False

   'Store the formula for the first series in "xVals".
   xVals = ActiveChart.SeriesCollection(1).Formula

   'Extract the range for the data from xVals.
   xVals = Mid(xVals, InStr(InStr(xVals, ","), xVals, _
      Mid(Left(xVals, InStr(xVals, "!") - 1), 9)))
   xVals = Left(xVals, InStr(InStr(xVals, "!"), xVals, ",") - 1)
   Do While Left(xVals, 1) = ","
      xVals = Mid(xVals, 2)
   Loop

   'Attach a label to each data point in the chart.
   For Counter = 1 To Range(xVals).Cells.Count
     ActiveChart.SeriesCollection(1).Points(Counter).HasDataLabel = _
         True
      ActiveChart.SeriesCollection(1).Points(Counter).DataLabel.Text = _
         Range(xVals).Cells(Counter, 1).Offset(0, -1).Value
   Next Counter

End Sub

cspgtp2 say:

Database planning in Access 2010: audit/quality checklist

Hello all!
I am an experienced Excel user but absolute Access newbie.  I am trying to expand my quality audit tracking.  Essentially, I have a multi-question checklist which we use for reviewing quality.  The spreadsheet does all of the calculations to weight each response then provides a weighted score at the end.  What I want is to build this directly into a database so auditors can fill out their checklists but then I can analyze results either by users or time or both.   The goal is to be able to address individual and group quality.  Unfortunately, I am not sure how to structure my tables so I can do calculations in reports to give me what I want.  The outputs should be a report for an individual audit, an output that shows me average score by employee over a given time and outputs which will show me what errors are most prevalent (so I can target group training).  I hope this makes sense.  Here is a sample from my current audit checklist.  There are more checklist questions but hopefully you get the idea.  The error number is the type of error (they are categorized) and the error level is the weight with higher numbers being more serious.  The "check" column is a drop-down which drives whether it is counted or not.  

 


henry habermacher say: Hi cspgtp2

What you want to do here is a more complex thing than just having one table. You have to break the data structure down to relations. For example:

Table Checklists: contains the description of the check list and other checklist details as well as CheclistID, the primary key for the checklist

Table Questions:
Contains any Questions you want to be able to present to the user. It also contains the QuestionID, the primary key for the Table Questions.

Table ChecklistQuestions:
This is a relation entity. It sets the Table Questions and Checklists into a many to many relationship. In here you have a compound primary key that consists of ChecklistID and QuestionID. Both fields are Foreign Keys to the mentioned questions.
What this table does is just keep the information: For Checklist XYZ the user will have the Questions 1, 2, 3, 5 and 7 presented.

Now you can create your checklists.
To store the answer you need a similar structure on the Answer side.
So you need a Table ChecklistAnswer, containing the header information about the answered questions, like: when answered and by who. Maybe there is also an approval required that goes into this table. The ChecklistAnswers table has a ChecklistAnswerID (Primary Key) and a foreign key referencing a Checklist.

For each question you now need the answer to the question. This can be stored in a table ChecklistQuestionAnswers. This one references now the ChecklistAnswers table (foreignkey) and has a compound primary key that consists of the CheclistAnswersID and the QuestionID.

No you also need a vew Domain tables holding your coding. If the user has to select a value from a list, then the values that are presented in the list are stored in such tables. For each list you will need a seperate table. Let's say your user can choose between different ErrorLevels. You would in a Listbox like to present the Error Level Number with a description when to choose this level. You then would place this decoding into a table ErrorLevels that contains the ErrorLevelNr (that could be a primary key) and the ErrorLevelDescription. Similar then to all other values the user can select from a list.

When you once have the base datamodel you can start to build up your forms to allow the (power-)user to enter Checklists and select Questions that are presented in the Checklists. Then you'll need a From to present the Checklists created before to the users that have to fill the Checklist. The result of the answers you store in the corresponding table.

The tricky thing is that you have to initialize the ChecklistAnswers somehow. As soon as a user is answering a Checklist it would make sense to initialize all answers for this checklist by adding for every question in checklistquestions that the user has to answer a row into the answers table. You could do this on the fly while the user is answering questions but I'm pretty sure it will be easier also for reportings later to initialize the ChecllistQuestionAnswers immediately when the user want to fill  the form. This can be done by a INSERT query on first opening a new checklist for answering by the user.

For your reporting I'd wait until you have the first testdata in your database, else it will become difficult to setup the required datasources for your reports correctly.

This was now just an entry about how this could be done. There are many different approaches that will lead you to a final solution. As your title suggest you need to plan your database. A questionnaire application requires a specific format to store the data. Generally it looks like this:

tblQuestion
QuestionID (Primary Key Autonumber)
Question

tblRespondent
RespondentID (PK Autonumber)
RespondentName
other info about respondent (or, in your case, the subject of the audit)

tblResponse
ResponseID (PK Autonumber)
RespondentID (Foreign Key)
ResponseDate

tblAnswer
AnswerID (PK Autonumber)
ResponseID (FK)
QuestionID (FK)
Answer

With this structure it will be easier to analyze the data.
