a. user say:

This turns out to be quite a curious problem. I don't know if the other suggestion eventually worked, but if not, it may take one or two attempts to solve it, but here goes.

1. The background

When you initiate a Merge via Outlook->Tools->Mail merge, Outlook exports the contact data as a Unicode format text file, but with a .doc extension. I think this is to force Word to open the data source as if it is a Word document, probably using Word's internal text converter). Word has several options for opening text files, and all the others introduce more dialog or other problems.

The trouble is that as soon as Word opens the data source as if it is a Word document, it appears to get the sort wrong. I do not think this has anything to do with the Unicode text encoding - it has something to do with Word's internal sort. And that is a little bit strange, because as Rohn007 says, if you put these values in a Word table and sort them, the sort works OK. However, Word cannot be using either that sort (well, not exactly) because the number of columns in the exported contacts file is well over 100, far more than are allowed in a Word table.

So the solution seems to be to save the contacts file in another format that sorts correctly. Unfortunately, that is more easily said than done in the general case. As long as all the characters in your data are in the "Western European Windows" set, the following may work:

2. A possible solution

(NB, another approach may be to export Outlook contact data in the usual way and use that as your data source)

 a. In Outlook's Tools->Mail Merge dialog box, ensure that you specify a "Permanent file" in the Contact Data File section. For example, I used c:\a\cancon. DOn't add an extension - Outlook will add ".doc"

 b. Close and discard the empty Mail Merge Main document that Outlook creates 

 c. In Word, open c:\a\cancon.doc . You may see a "Convert File" dialog that presents "Encoded Text" as the format. If so, click OK. Then you will probably see a File Conversion encoding dialog that will present "Unicode" as the encoding. Click OK.

 d. Now click Word->Office Button->Save As->Other Formats, select Plain Text in the "Save As Type" selection, and save the document as c:\a\cancon.txt (say). You should then see another File Conversion encoding dialog box. Don't click OK yet. Select the Windows radio button at the top left of the box. Click OK.

 e. Ensure that Word Office Button->Word Options->Advanced->General (near the bottom)->"Confirm file format conversion on open" is checked

 f. Connect to the data source, c:\a\cancon.txt. As a consequence of (e), you should see a "Confirm Data Source" dialog box. Select the OLE DB Database Files option. You should then see a "Text File Connection Parameters" dialog box. Select the "Comma" button, and leave "First rwo of data contains column headers" checked. Click OK. You will probably see the same dialog box again (don't ask!). Make the same choices. 

 g. Open the Edit Recipients dialog, locate the column and sort on it, and see if that sorts correctly.

This is obviously far more tedious than anyone would hope. I would hope that you would be able to get away with no more than doing steps (a), (b), and (e), rename c:\a\cancon.doc to a .txt, then selecting it as the data source. But if you do that, Word does not get the encoding correctly and misreads the file. I also wondered whether it would be easy to open the .doc in Excel, then save it as a .xslx, but unfortunately the multiline addresses make that hard as well. So right now, this is about the simplest procedure I can think of.
