ken sheridan say: I think you need to do some background work on how a relational database works.  Each table in a database models an entity type, and each column in the table represents an attribute of the entity type, and of that entity type only.  Data from columns in a referenced table must not be duplicated in the referencing table; that introduces redundancy and leaves the table wide open to the risk of update anomalies.  The only column from a referenced table which should be duplicated in a referencing table is the foreign key column which references the primary key of the referenced table.  Occasionally the primary key and the corresponding foreign key may be made up of more than one column each, but in your case it appears to be one column only.

So in your case the referencing table into which you are inserting data via your form should have a foreign key column, e.g. Head_SN_ID, which references the primary key ID of the Head SN List table.  BTW you should avoid spaces or other special characters in table or column names, and should use descriptive terms for keys, not the generic ID.  I know the sample Northwind database does both, but Northwind contains many examples of very bad practice.

In your form the control for the Head_SN_ID column should be a combo box set up as follows:

ControlSource:    Head_SN_ID

RowSource:     SELECT [Head_SN_ID], [Head SN] FROM [Head SN List] ORDER BY [Head SN];

BoundColumn:   1
ColumnCount:    2
ColumnWidths:  0cm

If your units of measurement are imperial rather than metric Access will automatically convert the unit of the last one to inches.  The important thing is that the dimension is zero to hide the first column.

To show the values of the non-key columns from  Head SN List base the form on a query which joins its current table to Head SN List on ID/Head SN List.  Both of these columns must be indexed, the first uniquely (as it will be if it's the primary key, which it must be), the second non-uniquely (duplicates allowed).  The tables must be related and referential integrity mist be enforced.

In the query return all columns from the referencing table, and only those columns from Head SN List whose values you wish to show in your form, but not the Head SN] column whose value will show in the combo box  In the form bind a text box to each of these columns from Head SN List, and set the Enabled property of the text boxes to False (No) and the Locked property to True  (Yes) to make them read only.  Once you select a value in the Head_SN_ID combo box the values from the referenced row in Head SN List will show in the form.  You can do the same in a report if necessary.

To see an example of a form which works in this way take alook at InvoicePDF.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

If you have difficulty opening the link copy its text (NB, not the link location) and paste it into your browser's address bar.

You'll see that when a customer is selected in the combo box bound to the CustomerID column in the main invoice form the customer's address details automatically show in the controls below the customer name.

donleverton say:

Ahhh, the first thing that I've noticed is that you have decided to change the spelling to "Warranty" versus "Warrantee" :D

OK ... the first thing to understand is that BOTH the RowSource Property and the AfterUpdate Event are intended to be for the listbox control. The RowSource is used  to build a list of items to be selected. Selecting something from the listbox inserts the selected value into the text-box (which is bound to the table field for the current (NEW) record. The reason why you couldn't find a RowSource property for a textbox ...  is that a textbox is not intended or capable of doing that (unless it has been defined as a "lookup" type of field in the table design). I am not in the habit of doing that, but I suppose that may be a possibility for you, considering that you seem to want that functionality for multiple fields.

I would also recommend that you consider using a naming convention for your objects (especially for form controls like list-boxes, combo-boxes, subforms, etc) to help you to differentiate them from bound controls that will have the same names, by default, as the fieldnames from your tables. you can also save yourself some grief by not using spaces in your field names. i.e. "HeadSN" instead of "Head SN"

If you look at the picture that I attached to my post ... the unbound listbox is named "lstWarrantee", and the textbox is named "Warrantee", which is bound to a table field of the same name.

In B2 on sheet 1:

=IFERROR(VLOOKUP(A2,'price list'!$A$2:$C$1000,2,FALSE),"")

In C2 on sheet 1:

=IFERROR(VLOOKUP(A2,'price list'!$A$2:$C$1000,3,FALSE),"")

Replace price list with the actual name of that sheet, and adjust the range if necessary (it doesn't matter if it's too large).

Fill or copy down from B2 and C2 as far as you want.
