suzanne s. barnhill say:

To add to what Doug has said, the initial "bug" reported to the product team was that Word no longer honors tab stops beyond the right margin. This does break some older documents (if converted to the new format), but it is not a showstopper provided that a right indent can be used. The fact that tabs to the right of a right indent are also ineffective is in fact a major bug and must be fixed with a hotfix. If no hotfix is forthcoming, then Microsoft is going to have a major problem, assuming that adoption will be much slower as long as such a bug is not fixed.

OssieMac,

I tried this and it worked. If it won't work in all cases, let me know.

Regards,

John

Function FormatPhone(strText As String) As String
    Dim strPhone As String
    Dim i As Integer
    Dim strChr
    
    Application.Volatile
    strPhone = ""
    For i = 1 To Len(strText)
        strChr = Mid(strText, i, 1)
        Select Case strChr
            Case "0" To "9"
                strPhone = strPhone & strChr
        End Select
    Next
    
    If Len(strPhone) = 11 And Left(strPhone, 1) = 1 Then
        GoTo FinalFormat
    End If
    
    If Len(strPhone) = 10 Then
        strPhone = 1 & strPhone
        GoTo FinalFormat
    End If
    
    'If phone number is blank (0), convert to 10000000000 then
    'goto finalformat
    If Len(strPhone) = 0 Then
        strPhone = 1 & "0000000000"
        GoTo FinalFormat
    End If
    
    'If does not meet either of above 3 conditions then
    'the original text does not have correct number of numerals
    FormatPhone = CVErr(FormatPhone)    'Return error because wrong number of numerals
    Exit Function
    
FinalFormat:
    FormatPhone = Left(strPhone, 1) & "-"
    FormatPhone = FormatPhone & Mid(strPhone, 2, 3) & "-"
    FormatPhone = FormatPhone & Mid(strPhone, 5, 3) & "-"
    FormatPhone = FormatPhone & Mid(strPhone, 8, 4)
    
End Function

jim gordon mvp say:

With an Office 365 subscription you could install the Windows version of Office on your Mac if you also install Parallels and a retail copy of Microsoft Windows.

The thing is, you're still going to have to learn how to generate a pivot table from a 3 table query and it will wind up being the same procedure in Windows as on the Mac unless you go all the way to using PowerPivot, which is a feature of certain Excel 2013 editions on Windows.

This assumes you downloaded and installed the Actual Access driver from Actual Technologies. You can download this driver and try it out for free. In free mode, you'll only get the first 5 rows in a result set, but it's good enough to test to see if the concept works for you.

Whichever platform you decide on, there are some steps you will take:

1. Put one table onto each worksheet in the data source. (Sounds like you've already done that). Be sure each table has one (and only one) row of headers. Rename the sheet tabs, as those names become the names of the tables you will use as your data source. Save and close this workbook.

2. On the Mac, you open the ODBC manager (this is built in Mac OS X and is in Applications > Utilities). Set the source workbook as a data source and give it a name. Once you set a name you can reuse that data source as many times as you want. Just don't rename or move that data source workbook. On a PC there is an analogous process to add data source name.

3. Open Excel to a new, blank workbook. On the Data tab of the Ribbon, click the triangle for Pivot Table and choose Create Manual PivotTable. Choose the Use External Data Source option, then click the Get Data button. Choose the Data Source Name you created in step 2 to connect the PivotTable to your data source tables.

4. Microsoft Query then opens. This is where you build the query of your data tables that feeds your pivottable with data. If you don't know SQL, get this book from your local library.

5. When you quit Microsoft Query, your source data is returned to a PivotTable in Excel.

That's the list of steps I think you should take.
