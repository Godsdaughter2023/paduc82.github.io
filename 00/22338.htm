How to add custom namespace metadata in Office files - Microsoft Community

alex_va say:

How to add custom namespace metadata in Office files

What is the approach for adding custom metadata for a unique namespace to Word/Excel/PowerPoint files without using a enterprise solution like SharePoint? I'm hoping someone might be able to point me to a starting point for setting up custom file properties in a namespace that matches what we are using with other applications.

Thanks

a. user say:

It depends on what you mean, really. I haven't come across any info. on "how best to do this stuff," so apologies if I have missed the mark but... 

For Word/Excel/Powerpoint, there are some advantages to following the SharePoint way of doing things even if you are not using SharePoint. For all three, you may be able to use a slightly more helpful DIP. FOr Word, you may get some advantages when using Content Controls. But more about that later.

Without following the SharePoint way of doing things, the facilities offered by Office are more limited. 

What you get to do is store XML documents in Custom XML Parts, and manipulate them via the relevant object models, e.g. via VBA. The only "freebies" AFAICS are that you can always connect suitable Word Content Controls to an element or attribute in an XML document in a Custom XML Part. In Word you may also get some advantages if you can insert the associated XML Schema in the Office Schema Library. 

[Incidentally, if you need to use non-XML stuff such as data in JSON data in Custom XML Parts, as far as I know you have to put it in an XML wrapper. I did read once that you could put non-XML data in Custom Parts, but AFAIK Word will not then open the document, and the object model doesn't seem to be able to do it either. Perhaps it was possible in the Office 2007 rereleases.]

But assuming that you are going to want to manipulate your metadata via the object models, I think the questions are then really about:
 a. is your existing approach to namespaces adequate, or does it need to be improved? e.g., is there a standard or a set of best practices for defining XML documents that contain metadata rather than other types of data? (don't know). Do you need to review how you assign and document namespace URIs? Are there widely used systems out there such as Document Management systems that impose requirements on the XML, and is it worth your while trying to conform to them? I believe these kinds of questions are not really to do with office, but are more to do with "being well plugged into the XML standards world"
 a. once the namespace thing is sorted, you can generate XML documents that conform to the schemas that define your metadata. You can put them in Custom XML Parts. You can reach any of the data via the object model. You can link Word Content Controls to them. Do you need anything more. e.g.
  - does your processing require access to the metadata schemas, a subset of them, or even additional information? If so, perhaps that can also be put into Custom XML Parts. But then proliferation may be a problem if the schemas are large.

Back to SharePoint now...

AFAICS there is only one pattern for defining your metadata that allows you to take advantage of as many of the features of Office as possible without actually using SharePoint. And that's to follow what SharePoint does. But 
 a. the only document I have found that describes the Custom XML Parts that Sharepoint creates is this one (Other than the older one that relates to the ECMA standards)

http://msdn.microsoft.com/en-us/library/ee908652(v=office.12).aspx

"Standards speak" generally leaves me cold so the only way I have found to understand this is to look at the Custom XML Parts that SharePoint actually creates, and 
 - I haven't gone far along that path
 - it isn't always clear what is essential, what is optional, and in terms of naming conventions, what has a chancing of playing well with other systems long term.

As far as I can tell, by following that pattern in your own documents and by using Microsoft's namespace names, in Word (and probably Excel and Powerpoint) a "freebie" is you get a "Server DIP" that can present some basic properties not included in the basic DIP, even if you don't have SharePoint. But 
 - you don't get the same validation stuff as you get in a real DIP
 - you don't get to use all the types of value that you can use when connected to SharePoint

(I would have to check even that, as it's a while since I looked).

However, if you are using metadata definitions that have in essence been reserved for use by SharePoint, if anyone checks the document into a SharePoint system and out again, the metadata will be gone. (That can probably happen even by "copying" a document from one SharePoint library to another).

So there's a case for avoiding "hijacking" the SharePoint namespaces. You will lose the opportunity to have a slightly more helpful DIP. In Word, you will also lose the fact that Word generates entries in the "Document Properties" list in the Insert->Quickparts menu, and that inserting content controls linked to those properties will require some other approach. AFAIK there is no equivalent of those features in Excel/PowerPoint.

So if you decide you don't need those features, what does the Schema part inserted by Sharepoint actually contain? Well, it contains enough stuff to allow both Word and InfoPath present the right kind of controls (text boxes/dropdowns), do the right thing in terms of looking up stuff in SharePoint, value lists where the SharePoint column is an enumeration, and so on. Also, it contains some name mapping info. and some attributes that are primarily to do with presentation in the DIP (whether the property is visible, some formatting stuff and so on). i.e., not much useful for metadata that's not going to be in the DIP.

How do I fill a series across multiple worksheets - Microsoft Community

odd ball say:

How do I fill a series across multiple worksheets

I have 52 worksheets, one for each week of the year, can I automatically fill cell A3 with the date of that week's monday?
rich007a say:

Try this formula:

=DATE(YEAR(TODAY()),1,8)-WEEKDAY(DATE(YEAR(TODAY()),1,6))+((MID(CELL("filename",INDIRECT("A1")),FIND("]",CELL("filename",INDIRECT("A1")))+1,256)-1)*7)

I'm sure the usual suspects will come up with significantly shorter versions though!

It assumes week 1 starts on the first Monday of the year.

To put in cell A3 or each sheet, select all the sheets (right-click on a sheet tab > select all sheets), then select cell A3 and paste in the above formula.

Cheers
Rich

PS:

This year is YEAR(TODAY()).

DATE(YEAR(TODAY()),1,8)-WEEKDAY(DATE(YEAR(TODAY()),1,6))  is the first Monday of the year.

(MID(CELL("filename",INDIRECT("A1")),FIND("]",CELL("filename",INDIRECT("A1")))+1,256)  is the sheet name (must be a number).

[date of first monday of year]+( ([sheetname]-1) * 7) = date of monday in weeknumber [sheetname]
