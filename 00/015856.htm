Were you an insider when we were running build 14393, which became the public release, and doesn't expire? If so, did you make a disk image backup at that time of your OS partition? Since that build doesn't expire like the rest of the preview builds do, you could restore your OS partition in about 10 minutes, and then upgrade from there to the current build ... if not, and you can't trick the system with reset dating long enough to let you run an upgrade with an ISO, you may well be in need of a clean install with said ISO.

Many people won't bother to make such a backup for many vague and questionable reasons, but it's well worth the few minutes that it takes to do it. If you have it, and don't need to use it, that's OK ... but, if you need it, and don't have it, that's not good.

At my place, we test with quite a bunch of machines, and it's standard practice to always have on hand, for each machine, disk images for the previous licensed OS, the most recent public release build, and the two previous preview builds, as well the ISOs for each version of at least the last two builds. If a hard drive fails, we lose a few minutes, not hours and hours.

I say all that to point out that it isn't much of a bother, if one gets in the habit, to maintain quick, easy, restore and/or reinstall capabilities.

I waste quite a bit of ink telling people this ... before the failure, they don't believe it will be needed, and afterward, they say it doesn't do any good to tell me that, now ... why don't you try to be helpful, instead of criticizing?

I can't win, but I keep hoping that one's current situation might cause them to see the benefits of being prepared for future failures. It never works, though. They go right back to thinking it won't be needed.

Lets take these "suggestions" one at a time:

#1 Permissions

Exactly what folder are you trying to open these documents in?

There is a small chance Windows has messed up permissions. So here are a couple tips on how to "take ownership" of the folder you are saving these files in

Take Ownership of Files / Folders

How to Take Ownership (Permission) of a File or Folder Manually in Windows?

http://www.askvg.com/guide-how-to-take-ownership-permission-of-a-file-or-folder-manually-in-windows/

 

#2 Free Mem or Disk Space

This one is very unlikely to be the issue. If you were opening an outlook attachment or an IE download, there is a small chance this would be the problem. There is a way of setting limits on TEMP folder sizes, but I really don't think it applies to this problem.

#3 Text Recovery Converter

This one implies there is some sort of file damage, file corruption.  This is very unlikely with new files. There is one possible cause for this sort of problem. That is if you are saving the files to a USB connected drive. If you don't use the proper "shut down" proceedure before pulling the USB plug there is a chance you could corrupt one of the files, but not all of them.

Try starting Word, then use the File > Open dialog. Find one of the problem files. Click on the drop down arrow beside the "Open" button.   Select "Open and Repair".  I doubt this will work, but it is worth a try.

“Word experienced an error trying to open the file”

Click file>options

Click Trust Center>Trust Center Settings

Click Protected View

Uncheck the following boxes

Enable Protected View for Files originating from the Internet

Enable Protected View for Files located in potentially unsafe locations

Enable Protected View for Outlook attachments

filename._docx - Microsoft Community

gvidok-s say:

filename._docx

Why sending an e-mail in MS Word document "filename.docx", the recipient receives a "filename._docx"? What then is the reason? It is also seen with MS Excel files.

***Post moved by the moderator to the appropriate forum category.*** 

I have found the reason why the file with the extension *.docx file occurs *_.docx.

In this case, the file name ended with a dot ".", for example "filename5." So the file name extension is "filename5..docx" with two dots "..". E-mail program for Windows live mail upon receipt of the file, apparently do not understand those two points and displays the file as "filename5._docx" file extension *_.docx
