tom van stiphout say:

This is where you go wrong: "The fields in the LEDGER table include clients name, phone, advertising source and date, date of the payment, payment method etc..."

Client name is already in the TST table. Relational database design rules require that LEDGER only has a CustomerID field, which refers to the TST table.

Then if you ever want to know for whom a ledger entry is, you create a query which joins the two tables (draw a line between LEDGER.CustomerID and TST.ID) and select fields from both tables.

You also need to go to the Relationships diagram, draw the same relation, and in the dialog that comes up check the box to enforce the relationship. This ensures LEDGER.CustomerID can only have values that actually occur in TST.ID.

Repeat as necessary.

It is critical you get this relational model correct. Hire the necessary talent if you have to.

scottgem say:

I strongly suggest that you due a search on database normalization. Your database is cumbersome because you don't understand proper normalization which is key for a relational database.

 

One of the principles of a relational database is that data is entered in ONE place only. So wherever you need to indicates that a record in another table is related to a specific customer ALL you need is the CustomerID as a FOREIGN KEY. So you should NOT have redundant fields in your Ledger table, just the Customer ID.

 

Another thing you need to learn how to do is use subforms. If you want to see all ledger records for a customer, then you create a main form/subform. The main form would be bound to the customers table and the subform to the Ledgers table, linked on CustomerID. this will show all the ledger records for the currently displayed customer. You can do the same with your Payments table.

 

Can your merchant accounts give you an electronic file with the information you need? If so, there is no reason you can't import that data. As long as they can produce a file in a format Access can read (many choices), then you can import that data.

 

I totally that "The amount of repetitive information that is entered into each table is crazy." But its crazy because of how poorly designed your database is (I'm sorry to have to say). Because you are not using the relational capabilities of Access properly, nor are you using all the tools Access gives you to make your database more efficient.

 

You say, you think eliminating this repetition will reduce your payroll considerably. Therefore, you might want to seriously consider hiring a consultant to properly design your database. The cost of doing so will be paid back by the productivity savings.

You cannot paste into a locked area of a protected sheet.

Activesheet.unprotect password:="pword"

      do the pasting

Activesheet.protect password:="pword"

 

As far as hidden sheets go...............you can add data to a hidden sheet just by sending it there

 

Assume in both cases below that Sheet2 is hidden.

 

Sub copy_to_hidden_sheet()
    Dim rng1, rng2 As Range
    Set rng1 = Sheets("Sheet1").Range("A1:F23")
    Set rng2 = Sheets("Sheet2").Cells(Rows.Count, 1).End(xlUp) _
               .Offset(1, 0)
    rng1.Copy Destination:=rng2
End Sub

 

Or by this method

 

Sub value_only()
Dim rng1, rng2 As Range
    Set rng1 = Sheets("Sheet1").Range("A1").EntireRow
    Set rng2 = Sheets("Sheet2").Cells(Rows.Count, 1).End(xlUp) _
               .Offset(1, 0).EntireRow
    rng2.Value = rng1.Value
End Sub

rich lloyd say:

I don't know if you got a solution for this but I had the same problem.

The solution for me was that the Mac (an iMac) was running in mirror mode and all looked good on the output to the projector. When entering Presenter mode on PowerPoint the projector was displaying a smaller image.

This was being caused because Presenter Mode forces the Mac temporarily in to extended desktop and in this mode the Mac was defaulting the second screen to 1024 x 768 resolution.

The fix for this is to go to display settings, set screen to extended desktop (mirroring off) and set the second screen resolution correctly (probably 1920x1080). If that resolution is not available in the list hold the Command Key and click on the Scaled radio button to see a full list.

This should solve your problem.

I would also recommend installing the Display Menu app from the App store which gives great control over screen resolutions. A very useful little free app.
