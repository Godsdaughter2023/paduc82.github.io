ken sheridan say: It sounds like you might be encoding data as table names.  A fundamental principle of the database relational model is the Information Principle (Codd's Rule #1). This requires that all data be stored as values at column positions in rows in tables, and in no other way.

If each of the tables has the same structure then the tables should be coalesced into a single table, with an additional column which represents whatever attribute is currently being represented by the separate tables.  It then becomes a simple matter to restrict the query on that column by referencing the check box controls in the dialogue form as parameters.  However, unless the check boxes represent a set of values which is immutably fixed, then a better interface will usually be a multi-select list box, which, being data driven, will automatically reflect any changes in the set of values.

ken sheridan say: Categories are data, so by having a separate table for each the data are each being represented by a table rather than a value in a column in a table as the Information principle requires.  Let's take a simple example of products each of which is of one of two categories 'Tool' or 'Hardware'.  The tables to model this would be:

Products
….ProductID  (PK)
….Product
….CategoryID  (FK

Categories
….CategoryID  (PK)
….Category

In a form, frmSelectCategories, you have two checkboxes, chkTool and chkHardware.  The query to allow you to select one or both categories would be:

SELECT Product, Category
FROM Products INNER JOIN Categories
ON Products.CategoryID = Categories.CategoryID
WHERE (Category = "Tool" AND Forms!frmSelectCategories!chkchkTool = TRUE
    OR Forms!frmSelectCategories!chkchkTool = FALSE)
AND (Category = "Hardware" AND Forms!frmSelectCategories!chkchkHardware = TRUE
    OR Forms!frmSelectCategories!chkchkHardware = FALSE);

Note that the parentheses are crucial here to force the OR operations to evaluate independently of the AND operations.

As you can see, it would be a simple matter to extend the query with further parenthesised expressions to cater for other categories should the product range be extended.  However, you'd also have to amend the form design by adding further check boxes.  If, on the other hand, a multi-select list box is used rather than multiple check boxes, the list in the list box will be automatically extended when further rows are added to the Categories table.  You'll find an example in DatabaseBasics.zip in my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.

In the section on 'retrieving data from the database' in this little demo is a form in which one or more contacts can be selected from a list box.  The report which then opens incudes the SQL statement for its query in the report header.  Note that it uses functions from the basInParam module, so you'd need to copy and paste the module into your own database to use this method.

ken sheridan say: I should have mentioned in my last reply that the multi-select list box in my demo is not referenced directly by the query.  A multi-select list box has no value as such, but an ItemsSelected collection.  In the form is a hidden text box, just above the 'Select All' button.  Code in the form's module inserts a value list into the hidden text box, and it is the text box which the query references as a parameter.
