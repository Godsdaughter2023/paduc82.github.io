macropod say:

Field coding cannot perform the kind of wildcard test you're after using asterisks on both sides of the expression. As you've see, you can use:

DO NOT SEND - to test for an exact match with DO NOT SEND

DO NOT SEND* - to test for a string starting with DO NOT SEND

*DO NOT SEND - to test for a string ending with DO NOT SEND

To test for a string that contains DO NOT SEND, you would have to use the ? wildcard to test every conceivable position before/after any other content. For example:

???DO NOT SEND*  - to test for a string starting with three characters before DO NOT SEND

?????DO NOT SEND*  - to test for a string starting with five characters before DO NOT SEND

*DO NOT SEND??? - to test for a string ending with three characters after

*DO NOT SEND????? - to test for a string ending with five characters after DO NOT SEND

I think everyone probably agrees with you that a "contains" function would be very useful, but in the absence of one the main options are

 a. preprocess your data in some way

 b. use one of several rather nasty tricks, all of which have limitations of some kind

In this case, with Windows Word, you might be able to do it using one of the methods I described in the following conversation: https://answers.microsoft.com/en-us/msoffice/forum/msoffice_word-mso_other-mso_o365b/can-if-statements-use-contains-as-a-function-and/8d1f04fd-d454-41ce-ac7e-e9f8fffdb590 , but only if

 - you don't have any single or double quotation marks in your Categories field

Specifically, I think you could use approach (d), where you put a small XML file in the same folder as your Mail Merge Main document .docx, and use the following field codes at the beginning of your document, using ctrl-F9 as usual to insert all the special field brace pairs { }:

I've edited this a couple of times to try to clarify it, but am now going to try to answer this part of your post first. Yes, what you understand is close to the truth.

It really comes down to the fact that we are using (or trying to use) INCLUDETEXT as a workaround. The INCLUDETEXT is trying to extract text from a file. All the xPath function specified by the \x tells the field is which part (s) of the file to extract it from. 

So even though in some sense the INCLUDETEXT can evaluate an xPath contains() function, there is nothing that lets us ask it to return the actual result of that function. The best we can do is use the xPath to identify a node (e.g. an Element or Attribute), and then the INCLUDETEXT will get the plain text value of that node. 

So we cannot return anything that is not already in the .xml file. Actually, we can also specify an XSL transform using the \t option. If we do, INCLUDETXT invokes that transform, passing the node identified in the xPath \x option. But it does not pass the text of the xPath expression, so again, we cannot AFAIK use the XSL to extract our text and use XSL functions to manipulate it. So at best, we can return stuff that is already in our .xml file or already in our .xsl file. Also AFAIK the XSL is limited by what we can do in XSLT version 1 and a small number of predefined Microsoft addon XPath and XSL functions. 

Later I'll touch on the DATABASE field option.

To go back to your other questions:

-----------------------------------------

<<
as I cannot find the \x switch on the INCLUDETEXT
>>

To summarise, for an XML file, the INCLUDETEXT field should really look like this:

{ INCLUDETEXT "path/name of an XML file" \x "a piece of xPath that identifies a sequence of  nodes" \t "path/name of an XSL file" \n "list of any necessary namespaces" \c XML }

\x: In practice, returning more than one node is rarely useful. It doesn't usually do what you would hope without an XSL transform, and the XSL transform basically cannot cope if you return more than one node. It can of course process multiple nodes by processing the node and all its descendants.

\t: AFAIK the XSL specified in the \t option can return text in any format that Word can cope with internally. I always used to believe it had to be WordProcessingML (either the Word 2003 version or perhaps the newer single-file OOXML version, but in fact you can return plain text or HTML.

\c: Identifies the name of the Word converter, in this case "XML"

\n: If you need to identify specific namespaces in the XML file, you can use the \n switch to specify prefix names that you can use in your xPath statement instead. I will have to check the exact syntax. In my x01.xml file, no namespace names are used so we do not have to worry, but suppose instead we had

<r xmlns='myns'>

then all our elements would be in the 'mins' namespace and we would need to use

\n "xmlns:something='myns'"

and qualify element names in our xPath with something:, e.g.

\\something:i[1]

If there is more than one namespace you can use e.g.

\n "xmlns:ns0='first' xmlns:ns1='second'"

-----------------------------------------

<<
I guess the {FILENAME \p} is there to find the path to the xml file and follwoing "\\"  sequence is to replace the name of the file.
>>

Yes, if your .docx is in, say, c:\mypath\mydoc.docx 

then

{ FILENAME \p }\\..\\x01.xml

expands to

c:\mypath\mydoc.docx\\..\\x01.xml

The \\ is a bit of a red herring because you (sometimes) need to double up backslashes in fields, so it's really

c:\mypath\mydoc.docx\..\x.01.xml

which is a relative path address, i.e. 

c:\mypath\mydoc.docx - locate that file

\.. locate its folder

\x01.xml locate x01.xml in the folder

(It has always been surprising that that works starting from a file name rather than a folder name, but it does, at least with local file systems and AFAIK with documents in Sharepoint folders with http: names (though I'd have to re-check that).

So the basic includetext action is to include the .xml file.

In this case we do not need \t or \n, just the xPath, and that is what the \x specifies. And it pretty much does what you suggest in this case. The xPath query is

//i[1+contains('{ MERGEFIELD Categories }','FullCategoryName')]

//i is, in effect, "select the values of all XML elements named 'i'

The bit inside the [ ] is a qualifier, a bit like a SQL WHERE clause

//i[1] selects the first Element i in whatever list //i returns, //[2] selects the second, and so on.

"contains" is one of the few text functions in xPath 1.0 (which is the only version we get to play with here - no xPath 2 allowed). 

contains(string1,string2) will return 1 if string1 contains string2, and 0 otherwise.

So if { MERGEFIELD categories } contains FullCategoryName, we get the *value* of //i[2], i.e. our 2nd i element, i.e.  <i>1</i> element, i.e. 1, otherwise we get the value of <i>0</i>, i.e. 0

-----------------------------------------

The conversation I pointed to also shows how you can use a DATABASE field to do much the same thing (with similar limitations). In that case you use an SQL select statement to do the contains, and because most Windows desktop copies of Office have either the Jet or ACE OLE DB provider installed, it uses either 

 - the Jet/ACE "Like" operator with wilcards (it's similar to VBA's "Like") or 

 - the instr() function to do the "contains". 

In that case, we can use SQL + Jet/ACE functions to return stuff such as substrings. But there are similar constraints with characters such as ' and ", and the total length of the string we can pass to SQL. And although in the case of the DATABASE field we do not have to return any data from its data source, to use the Jet/ACE dialect of SQL, we still have to specify something that tells it that it is dealing with a Jet/ACE data source, which requires that we provide an external file. It turns out that it doesn't have to be an actual database, but can just be a file containing a rather short connection string.

If you're going to resort to one of these techniques, and my suggestion is that it is usually going to be better to find another approach, the main advantage that the INCLUDETEXT/XML approach has is that it does not actually return field execution errors when the xPath syntax is invalid. It just doesn't return the expected result. With the Database field, errors are harder to avoid.

As described, the answer is NO.  Now, I have to assume that you have W8.1 in 1 partition and W10 in the other partition and not 3 partitions.

Come July 29th, you have a decision to make.  You can quit Insiders and move from W10 Preview to W10 RTM.  This means that 1 partition leaves the scene and you now just running W10 as your sole O/S.  Case closed.

However, come July 29th, you can choose to do nothing and just continue with Insiders.  This is just business as usual, the same as now.  You have not burnt your bridge to get to W10 RTM final.  This is just doing later what you could have done Jul 29th.  Case open.

The same applies to non-Insiders when their turn comes.  They can go to W10 when announced or take 1 year.  This group will not be bothered with dual boots or VM.

Please don't dispute my facts without reading Gabe Aul - June 22 - when he finally took both feet out of his mouth.

Mel

don cuthbert say:

I do not see why that would not work. If your Windows Insider Preview partition was an upgrade from a qualifying Windows 7 or 8 OS, it should get the same final Windows version as you get on the Windows 8.1 partition.  Following that, there will be further Insider Preview Builds for that partition, but you will have upgraded your Windows 8.1 license to the final public Windows 10 release license.

Each computer has the option to join or leave the Windows Insider program, so you could probably leave it from your former Windows 8.1 partition and continue with Insiders on the other partition.  The end result would be the equivalent of someone running dual boot as you are today, except both would be versions of Windows 10.  I currently have multi-boot versions of Windows 10 on some computers.

Since your Windows 8.1 license will be upgraded, you will not own a license to dual boot Windows 8.1 any more.

I would ensure that you make an ISO of the Windows 10 final release, so that you can install your licensed copy at any time that you choose and have it available if you at some point leave the Insider Program.
