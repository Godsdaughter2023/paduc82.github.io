ken sheridan say:

I am surprised how complex this is, as it seems to me it would be a basic function in any invoicing database.

Access provides the tools.  It's the responsibility of the developer, in this case you, to build the functionality.

In an invoicing database the price of a product should not be returned in a computed column or control.  Over time prices change, so prior invoices will show incorrect values.  The price should be looked up from the referencing table and assigned to a column in the invoice details table.

You'll find an example in DatabaseBasics.zip my public databases folder at:

https://onedrive.live.com/?cid=44CC60D7FEA42912&id=44CC60D7FEA42912!169

Note that if you are using an earlier version of Access you might find that the colour of some form objects such as buttons shows incorrectly and you will need to  amend the form design accordingly.  

If you have difficulty opening the link, copy the link (NB, not the link location) and paste it into your browser's address bar.

In the section on 'Entering data via a form/subforms in this little demo file you'll find an Orders form in which there is an order details subform.  When an item is selected in the subform the current unit price of the product is assigned to the UnitPrice column in the subform's OrderDetails table in the item combo box's AfterUpdate event procedure with:

Private Sub ItemID_AfterUpdate()

    ' assign current unit price from Items table
    ' to UnitPrice column in OrderDetails table
    Me.UnitPrice = Me.ItemID.Column(1)
    
End Sub

The unit price of each item is returned in a hidden second column in the combo box's RowSource.  Note that the Column property is zero-based, so Column(2) references the second column.  By storing the unit price with the order (it's the same with an invoice) the price remains static as that current at the time of the order regardless of subsequent changes in the price of the product.

For an illustration of invoicing in a more complex scenario see InvoicePDF.zip in the same OneDrive folder.

rob koch say:

Though it's possible there might actually be malware contained in that pagefile, you have to be careful when performing a full scan, especially on the files from an entirely separate filing system.

That's due to the fact that you are looking at the files "out of context" from the outside, not from the standpoint of the filing system within which they are being examined and not from within that operating system itself.

This may not seem to matter, but the problem is that the full scan has no idea what the correct status of any file from another operating system might be, so it's merely looking at these files in a static manner, displaying anything if finds that might indicate possible infection.  On the surface this seems ok, but since the scan is entirely unaware of the other operating system, it's also unaware of any differences this might imply or whether what it's able to see is appropriate, since as an example that particular file might normally be locked and so unable to be read when the system is operating.

This situation is why even a full scan of the current operating system disk can often result in false positive detections, since looking at files that aren't contained in their normal installation locations may confuse the detection logic into thinking something's wrong when in reality it's just the abnormal location and/or status that causes the scanner's confusion.

Is this possible, or do I need to use cross-reference?

suzanne s. barnhill say:

The "display text" of a hyperlink can be anything you want. So, for example, you can type the word "here," select it, and add a hyperlink to the desired location. The display text can be edited in the Hyperlink dialog; you could conceivably reduce the display text to a single space, though I would not advise this. You can also customize the text that is displayed on hover.

 

If you need a cross-reference, then it can also be hyperlinked. Only you can choose whether you want a cross-reference or some other kind of hyperlink.
