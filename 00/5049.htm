a. user say:

Two important points about accessibility:

  • Microsoft's addition of the alt text attribute to the Table object was a mistake. There is no corresponding attribute in HTML, and content entered there is not used by screen readers. You may ignore that error from the Accessibility Checker. (You do need to designate the top row as a header row, if it is one.)
  • Point size is less relevant than many make it out to be. Set your font size with styles and use themes to change the font face. When you do, the user will be able to change the theme or edit the style (or use their own template, in which the style has a larger point size) to make the text legible to themselves.

So the failure of the Accessibility Checker to flag the table as missing alt text is not a bug. It's actually a fix. The failure to question the lack of a header row is a bug.

Little-known hack: If you must use a table to lay out content, add an empty space to the Title field of the alt text of the table. When you create a PDF of the document, the table structure is ignored, as the structure of a layout table should be.

Finally, point size is a measure tailored to metal type: The measure is the size of the block on which the letter rests. It has nothing to do with the size of the letter itself. 12-point type in two different faces will fill the same vertical space when used to set a line of type, even though the letters of one face might be much smaller than the letters of the other. So letters set in 12-point Verdana are larger than letters set in 12-point Times New Roman. (In fact, letters in 11-point Verdana are larger than letters in 12-point Times New Roman.) But in metal type, both 12-point fonts would rest on a block of the same size.

a. user say:

Hi, Lora!

You understand that rolling back to Office 2016 doesn't completely solve the problem, right? Instead of ignoring green flags, you'll have to ignore false positives--that is, items that are correct but are flagged as errors.

Alt Text is not an attribute of tables, but Word includes it among the Table Properties. In Word 2016 the Accessibility Checker flags tables as errors if you don't include alt text. For example, this is the result of checking a short document that includes three tables:

You can satisfy the Accessibility Checker by entering an empty space in the Description field of the alt text property of each table, but that's a waste of your time.

It seems the most vexing problem you're having is getting the table's header row properly identified. If you modify your favorite table style to make the top row a header row and then make it your default style for tables, every new table you create will have the header row identified. In an existing document, you can use the Navigation pane to go from table to table and change its style to your default style to ensure that the header row is identified.

But not every table needs a header row--it might need only a column of row headers. So regardless of which version of Office you have you still have to review each table individually and decide whether "Missing Header Row" actually is an error for that table.

Ok I figured it out. I went into View Header/Footer, made sure my page numbering started at one in the first section with Arabic numbering.
